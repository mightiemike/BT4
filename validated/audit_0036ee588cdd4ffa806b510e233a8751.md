Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the originating user, allowing any caller to bypass the swap allowlist when the router is allowlisted — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist, but `sender` is always `msg.sender` of the `pool.swap` call — the router contract — not the originating user. A pool admin who allowlists the router to enable router-mediated swaps for curated users inadvertently opens the allowlist to every user on the network who routes through `MetricOmmSimpleRouter`.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender`:**

`MetricOmmPool.swap` calls `_beforeSwap` with `msg.sender` as the first argument:

```solidity
// MetricOmmPool.sol lines 230-240
_beforeSwap(
    msg.sender,   // ← always the immediate caller of pool.swap
    recipient,
    ...
);
``` [1](#0-0) 

**Step 2 — ExtensionCalling forwards `sender` unchanged:**

`ExtensionCalling._beforeSwap` encodes `sender` verbatim into the extension call:

```solidity
// ExtensionCalling.sol lines 160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(
        IMetricOmmExtensions.beforeSwap,
        (sender, recipient, ...)
    )
);
``` [2](#0-1) 

**Step 3 — Extension checks `sender` (the router) against the allowlist:**

```solidity
// SwapAllowlistExtension.sol lines 37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap` — the router.

**Step 4 — Router is always `msg.sender` at the pool level:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly:

```solidity
// MetricOmmSimpleRouter.sol lines 72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
``` [4](#0-3) 

At the pool level, `msg.sender` is the router. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][originalUser]`.

**Root cause:** When a pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps for their curated users, `allowedSwapper[pool][router]` becomes `true` for every caller. From that moment, any unprivileged user who routes through `MetricOmmSimpleRouter` passes the allowlist check — including users who were never individually allowlisted. The same path applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

## Impact Explanation

Once the router is allowlisted, `SwapAllowlistExtension` is completely defeated for all router-mediated swaps. Any unprivileged user can trade on a pool intended to be restricted to a curated set of addresses (e.g., institutional-only, KYC-gated, or partner-only liquidity). This allows unauthorized parties to execute swaps at oracle-anchored prices against LP positions that were never intended to be accessible to them, bypassing the pool admin's curation policy entirely. This constitutes a broken core pool functionality (allowlist access control) and an admin-boundary break where the pool admin's intended restriction is circumvented by an unprivileged path.

## Likelihood Explanation

Medium. The bypass requires the pool admin to add the router to the allowlist, which is a natural and expected configuration step for any pool admin who wants to allow their curated users to use the router. The bypass is then permanent and affects all router-mediated swaps for that pool. Any unprivileged user can exploit it by simply calling `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) targeting the restricted pool.

## Recommendation

The extension should check the originating user, not the immediate caller. Two approaches:

1. **Pass the originating user through `extensionData`:** The router encodes `msg.sender` (the originating user) into `extensionData`, and the extension decodes and checks it when `msg.sender` (the pool's caller) is a known router.

2. **Check `tx.origin` as a fallback:** When `sender` is a known router, fall back to `tx.origin` for the allowlist check. This is simpler but has known limitations with smart-contract wallets.

3. **Preferred — router forwards originator:** Add a dedicated field to the swap call or extension data that the router populates with the originating user address, and have the extension verify both that the caller is a trusted router and that the originating user is allowlisted.
