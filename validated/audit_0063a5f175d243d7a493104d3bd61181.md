### Title
Missing AuxPoW-flag-to-data Consistency Check Allows AuxPoW Validation Bypass — (`contract/src/dogecoin.rs`)

### Summary

`submit_block_header` dispatches to `check_aux` only when `aux_data` is `Some`. There is no guard that enforces the inverse: if `block_header.version & BLOCK_VERSION_AUXPOW != 0`, then `aux_data` **must** be `Some`. A caller who submits a header with the AuxPoW version bit set but `aux_data = None` falls silently into the standalone-PoW branch, where only the Scrypt hash of the Dogecoin header itself is checked against the target. The entire AuxPoW validation — parent-block PoW, coinbase merkle proof, chain merkle proof, chain-ID checks — is skipped.

---

### Finding Description

In `submit_block_header` the branching logic is:

```
if let Some(ref aux_data) = aux_data {
    self.check_aux(&block_header, aux_data);   // full AuxPoW path
} else {
    // standalone Scrypt PoW path — no version-flag check
    require!(block_hash_pow() <= target_from_bits(bits));
}
``` [1](#0-0) 

`check_aux` does verify the flag internally:

```rust
require!(
    block_header.version & BLOCK_VERSION_AUXPOW != 0,
    "Aux POW block does not have AuxPoW flag set in version"
);
``` [2](#0-1) 

But that guard is only reached when `aux_data` is already `Some`. The symmetric guard — "if the version flag is set, `aux_data` must be `Some`" — is absent. The real Dogecoin node enforces this at the serialization layer: a block with the AuxPoW bit set is deserialized as an AuxPoW block and rejected if the AuxPoW fields are missing. The contract has no equivalent enforcement.

The `submit_blocks` public entry point is gated by `#[trusted_relayer]`: [3](#0-2) 

A trusted relayer (or any account granted `Role::UnrestrictedSubmitBlocks`) can submit arbitrary `(Header, Option<AuxData>)` tuples. The attacker-controlled data path is therefore reachable without DAO compromise.

---

### Impact Explanation

A block accepted by the light client but rejected by every real Dogecoin node breaks the core invariant of the light client: that its canonical chain is a sub-chain of the real Dogecoin chain. Downstream consumers (bridge contracts, proof verifiers) that rely on `get_last_block_header` or inclusion proofs would operate against a fabricated chain tip. This is a **chain-validity bypass** — the critical impact class for a light client.

---

### Likelihood Explanation

The attacker must:

1. Be a trusted relayer (or hold `Role::UnrestrictedSubmitBlocks`).
2. Mine a Scrypt hash of the crafted 80-byte header below the current Dogecoin target.

Requirement 2 is non-trivial but not impossible: Dogecoin's Scrypt difficulty is orders of magnitude lower than Bitcoin's SHA256d difficulty, and merged-mining pools already possess the necessary Scrypt hardware. An attacker who is also a legitimate merged miner can produce such a header as a side-effect of normal Litecoin mining. The `block_hash_pow` function confirms Scrypt is used when the `scrypt_hash` feature is enabled: [4](#0-3) 

---

### Recommendation

Add an explicit consistency check immediately before the `if let Some` branch in `submit_block_header`:

```rust
const BLOCK_VERSION_AUXPOW: i32 = 0x100;
if block_header.version & BLOCK_VERSION_AUXPOW != 0 {
    require!(
        aux_data.is_some(),
        "AuxPoW version flag is set but no AuxPoW data was provided"
    );
}
```

This mirrors what the real Dogecoin node enforces at the wire-format level and closes the bypass without changing any other validation logic.

---

### Proof of Concept

```rust
// Craft a header with AuxPoW version flag set (chain_id=0x62, AuxPoW bit=0x100)
let crafted_header = Header {
    version: 0x0062_0100,   // AuxPoW flag set, Dogecoin chain_id
    prev_block_hash: <tip hash>,
    merkle_root: H256::default(),
    time: <valid timestamp>,
    bits: <current target bits>,
    nonce: <nonce satisfying Scrypt target>,
};

// Submit with aux_data = None and skip_pow_verification = false
let headers: Vec<(Header, Option<AuxData>)> = vec![(crafted_header, None)];
user_account
    .call(contract.id(), "submit_blocks")
    .args_borsh(headers)
    .deposit(STORAGE_DEPOSIT_PER_BLOCK)
    .transact()
    .await?;

// Result: check_aux is never called; block is accepted if Scrypt hash <= target.
// The real Dogecoin network would reject this block entirely.
```
