### Title
ERC4626 Share Price Inflation Causes Silent Skip in `wrapVaultAsset`, Permanently Locking User Asset Tokens in `DirectDepositV1` — (`ContractOwner.sol`)

---

### Summary

`ContractOwner.wrapVaultAsset` guards its deposit logic with a `previewDeposit != 0` check. An attacker can inflate an ERC4626 vault's share price by donating underlying tokens directly to the vault, causing `previewDeposit(assetBalance)` to return 0. When this happens, the entire deposit block is silently skipped: the user's asset tokens remain stranded in `directDepositV1` with no user-accessible recovery path.

---

### Finding Description

`wrapVaultAsset` is the only on-chain path to convert a user's underlying asset tokens (e.g., USDC) held in `directDepositV1` into the ERC4626 vault shares that the protocol actually tracks as collateral. The function is `external` with no access-control modifier — any caller can invoke it. [1](#0-0) 

The critical guard is at line 525:

```solidity
uint256 assetBalance = IERC20Base(assetTokenAddr).balanceOf(directDepositV1);
if (IERC4626Base(tokenAddr).previewDeposit(assetBalance) != 0) {
    DirectDepositV1(directDepositV1).withdraw(IIERC20Base(assetTokenAddr));
    ...
    IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1);
}
``` [2](#0-1) 

For a standard ERC4626 vault, `previewDeposit(assets)` computes `assets * totalSupply / totalAssets`. An attacker who donates tokens directly to the vault contract (bypassing `deposit`) inflates `totalAssets` without minting new shares. Once `totalAssets` is large enough relative to `assetBalance * totalSupply`, the division truncates to 0 and the `if` branch is never entered.

The consequence is a **silent no-op**: no revert, no event, no error — the function returns successfully while the user's asset tokens remain in `directDepositV1`.

`creditDeposit()` in `DirectDepositV1` iterates only over registered product tokens (vault shares), not underlying asset tokens:

```solidity
function creditDeposit() external {
    uint32[] memory productIds = spotEngine.getProductIds();
    for (uint256 i = 0; i < productIds.length; i++) {
        address tokenAddr = spotEngine.getToken(productId);
        uint256 balance = token.balanceOf(address(this));
        if (balance != 0) { ... endpoint.depositCollateralWithReferral(...); }
    }
}
``` [3](#0-2) 

Since the asset tokens were never converted to vault shares, `creditDeposit` finds zero vault-share balance and skips the subaccount credit entirely.

The only recovery path for the stranded asset tokens is `withdrawFromDirectDepositV1`, which is `onlyOwner`: [4](#0-3) 

`DirectDepositV1.withdraw` is also `onlyOwner` (owner = `ContractOwner`): [5](#0-4) 

The user has no self-service recovery. Their funds are locked until the protocol multisig intervenes.

---

### Impact Explanation

A user who sends underlying asset tokens to their `directDepositV1` address expecting them to be wrapped and credited as collateral will find:

- `wrapVaultAsset` silently no-ops due to the inflated vault.
- `creditDepositV1` / `creditDeposit` finds zero vault-share balance and skips the credit.
- The user's subaccount receives no collateral.
- The asset tokens are locked in `directDepositV1` — recoverable only by the protocol owner via `withdrawFromDirectDepositV1`.

The corrupted state delta: `directDepositV1.assetTokenBalance > 0` persists indefinitely while `subaccount.spotBalance` for the vault-share product remains unchanged (no credit applied).

---

### Likelihood Explanation

- `wrapVaultAsset` is permissionless (`external`, no modifier) — any address can call it, and any address can attempt to grief it.
- The inflation attack is a well-known ERC4626 vector. The attacker donates tokens directly to the vault address; the cost is proportional to `assetBalance * totalSupply` of the vault at the time of attack.
- For a vault with low total supply (e.g., early-stage or low-TVL vault), the donation cost to make `previewDeposit(assetBalance)` truncate to 0 can be very small.
- The attacker can front-run every `wrapVaultAsset` call to sustain the block indefinitely.
- `isWrapVaultAssetReady` also calls `previewDeposit` and returns `false` when it is 0, so the off-chain sequencer/bot would also stop retrying, compounding the lock. [6](#0-5) 

---

### Recommendation

1. **Remove the silent-skip guard.** The `if (previewDeposit(...) != 0)` check was added to avoid the ERC4626 `ZERO_SHARES` revert, but it converts a revert into a silent fund lock — a worse outcome. Instead, revert explicitly with a descriptive error when `previewDeposit` returns 0, so callers know the vault is in an abnormal state.

2. **Enforce a minimum deposit amount.** Before calling `wrapVaultAsset`, verify that `assetBalance` is large enough that `previewDeposit` cannot be truncated to 0 under any realistic share price. Revert if the balance is below this threshold.

3. **Consider virtual shares/assets offset** (ERC4626 inflation mitigation): if the vault itself is under Nado's control, add a virtual offset (e.g., OpenZeppelin's `_decimalsOffset`) to make the share price inflation attack economically infeasible.

---

### Proof of Concept

```
Setup:
  - ERC4626 vault V has totalSupply = 1 share, totalAssets = 1 token (normal state).
  - User sends 100 asset tokens to directDepositV1[subaccount].

Attack:
  1. Attacker calls V.asset().transfer(address(V), 100_000e18)
     → totalAssets = 100_001e18, totalSupply = 1
     → previewDeposit(100) = 100 * 1 / 100_001e18 = 0 (truncated)

  2. Anyone calls ContractOwner.wrapVaultAsset(subaccount, productId).
     → assetBalance = 100
     → previewDeposit(100) == 0  →  if-branch NOT entered
     → function returns silently

  3. creditDepositV1(subaccount) is called.
     → DirectDepositV1.creditDeposit() iterates product tokens
     → vault share balance of directDepositV1 = 0  →  skipped
     → subaccount receives zero collateral credit

  4. User's 100 asset tokens remain in directDepositV1 indefinitely.
     Only ContractOwner (multisig) can recover them via withdrawFromDirectDepositV1.
```
