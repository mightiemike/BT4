### Title
Odd-width Merkle tree phantom-position ambiguity allows the same proof to validate two distinct `tx_index` values — (`merkle-tools/src/lib.rs`)

### Summary

`compute_root_from_merkle_proof` uses only the parity of `transaction_position` at each level to decide left/right placement. In a Bitcoin Merkle tree with an odd number of leaves, the last real leaf is duplicated. This creates a structural symmetry: the proof for the last real leaf at position `N-1` is mathematically identical to a proof for the phantom position `N`. The verifier has no way to distinguish them because it never validates that `tx_index < tree_size`.

### Finding Description

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` iterates over proof elements and uses `current_position % 2` to determine hash ordering, then divides by 2: [1](#0-0) 

For a tree with `N` leaves where `N` is odd, the last leaf `T[N-1]` is duplicated to `T[N]` during tree construction. The proof for `T[N-1]` at position `N-1` contains `T[N-1]` itself as the first sibling (the duplicate). When the same proof is submitted with `tx_index = N` (the phantom position), the parity sequence at every level is identical, producing the same root hash.

`verify_transaction_inclusion` passes the attacker-supplied `tx_index` directly to `compute_root_from_merkle_proof` with no upper-bound check: [2](#0-1) 

The block header stores only the Merkle root — not the transaction count — so the contract has no in-scope mechanism to reject an out-of-range index. [3](#0-2) 

### Impact Explanation

A downstream bridge, unlock, or mint contract that uses `(tx_id, tx_index, block_hash)` as its uniqueness key will accept two distinct proof submissions for the same underlying transaction:

1. Legitimate call: `(T[N-1], N-1, block_hash)` → `verify_transaction_inclusion` returns `true` → bridge unlocks funds.
2. Replay call: `(T[N-1], N, block_hash)` → `verify_transaction_inclusion` also returns `true` → bridge unlocks funds a second time.

The `tx_id` is the same real transaction in both calls, but the differing `tx_index` bypasses any deduplication keyed on the full tuple. This enables cross-chain double-spending.

### Likelihood Explanation

- `verify_transaction_inclusion` is a public, unpermissioned call (only gated by `#[pause]`, not by `#[trusted_relayer]` or `#[private]`).
- The attacker needs only a real block with an odd transaction count (extremely common on Bitcoin mainnet) and the legitimate proof for the last transaction.
- No privileged role, key leak, or social engineering is required.
- The deprecated function remains callable and is the entrypoint for many existing integrations. [4](#0-3) 

### Recommendation

1. **Bounds-check `tx_index`**: The proof length encodes the tree depth (`2^proof_len` is the padded tree width). Reject any `tx_index >= 2^proof_len / 2` that falls in the phantom half, or require `tx_index < 2^(proof_len-1)` when the proof length implies an odd-leaf level.
2. **Alternatively**, enforce that the proof element at the level where duplication occurs is never equal to the current hash (i.e., reject self-sibling proofs unless the tree has exactly one transaction).
3. **Prefer `verify_transaction_inclusion_v2`** for all new integrations; consider removing the deprecated v1 path.
