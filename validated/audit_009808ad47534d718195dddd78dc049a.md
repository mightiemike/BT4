### Title
Zero-Amount Claim Bypasses `claimed` Guard, Enabling Double-Claim — (`core/contracts/Airdrop.sol`)

### Summary
`_verifyProof` uses `claimed[week][sender] == 0` as the "not yet claimed" sentinel. Writing `claimed[week][sender] = 0` (when `totalAmount = 0`) is a no-op, leaving the slot in its default state and allowing a second claim for the same `(week, sender)` pair.

### Finding Description
`claimed` is `mapping(uint32 => mapping(address => uint256))`. The guard at line 48 checks `== 0` and the write at line 62 stores `totalAmount`. If `totalAmount = 0`, the write is indistinguishable from the uninitialized default. Any subsequent call for the same `(week, sender)` passes the guard again.

Attack sequence (requires Merkle tree with two leaves for the same sender/week):
1. `claim([{week: w, totalAmount: 0, proof_0}])` → `claimed[w][sender]` stays `0`
2. `claim([{week: w, totalAmount: X, proof_X}])` → guard passes, `X` tokens transferred

### Impact Explanation
The reward pool is drained by `X` extra tokens per attacker who has both a zero-amount and a non-zero-amount leaf in the same week's Merkle tree. This breaks the invariant that each `(week, sender)` pair can claim at most once, directly destroying protocol-held token value.

### Likelihood Explanation
Requires the registered Merkle tree to contain two leaves for the same `(sender, week)` — one with `totalAmount=0` and one with `totalAmount=X`. While atypical in normal operation, the contract enforces no constraint against it, and the flaw is exploitable whenever such a tree is published (e.g., due to a tooling bug, off-chain data error, or deliberate manipulation of the tree-building pipeline).

### Recommendation
Either:
- Add `require(totalAmount > 0, "Zero amount not claimable");` at the start of `_verifyProof`, or
- Replace the `uint256` sentinel pattern with a separate `mapping(uint32 => mapping(address => bool)) internal hasClaimed` and check/set that boolean independently of the amount.
