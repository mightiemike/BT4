Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates the router address instead of the originating user, enabling allowlist bypass or breaking router-mediated swaps — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` passes `msg.sender` (the direct pool caller) as the `sender` argument to every configured `beforeSwap` hook. When users route through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so `SwapAllowlistExtension` checks the router's address against the per-pool allowlist rather than the originating user's address. This creates two mutually exclusive failure modes: either the router is allowlisted (any user bypasses per-user restrictions) or it is not (allowlisted users cannot use the standard router interface at all).

## Finding Description
`MetricOmmPool.swap` at line 231 passes `msg.sender` as `sender` to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← always the direct caller of pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` at lines 149-177 forwards this verbatim via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`.

`SwapAllowlistExtension.beforeSwap` at line 37 then checks that `sender` against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` at lines 72-80 calls `pool.swap(...)` directly, making the router `msg.sender` to the pool. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` at line 32 ignores `sender` entirely (unnamed first parameter) and checks the explicit `owner` parameter — the economic actor — which `addLiquidity` carries separately from `msg.sender`. The swap path has no equivalent explicit `swapper` parameter; `pool.swap` only exposes `recipient`, not the originating user.

**Mode A — Allowlist bypass:** Pool admin allowlists the router so router-mediated swaps work. Because `allowedSwapper[pool][router] == true`, every user who calls `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` on the router passes the check regardless of individual allowlist status. Unauthorized users trade on a curated pool.

**Mode B — Broken core functionality:** Pool admin allowlists only specific user addresses (not the router). Allowlisted users who call the router receive `NotAllowedToSwap` because the router's address is not in the allowlist. Since `pool.swap` requires a `IMetricOmmSwapCallback` implementation, users cannot call the pool directly without deploying a custom callback contract. The primary user-facing interface is unusable for the very users the admin intended to serve.

## Impact Explanation
Mode A constitutes a direct allowlist bypass reachable by any unprivileged actor through the public `MetricOmmSimpleRouter` entrypoints. Unauthorized users can trade on pools designed to restrict counterparties, exposing LP principal to adversarial or unintended counterparties — a broken core pool invariant with direct fund impact. Mode B renders the standard swap interface completely unusable for allowlisted users on any pool that configures `SwapAllowlistExtension` with per-user restrictions, constituting broken core pool functionality. Both modes are High/Medium severity under Sherlock thresholds.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard, documented user-facing swap interface. Any pool admin who deploys a curated pool with `SwapAllowlistExtension` must decide whether to allowlist the router — either choice produces a broken invariant. No special privileges, malicious setup, or non-standard tokens are required. Any user with a token balance can trigger Mode A; any allowlisted user attempting a router swap triggers Mode B. All four router entrypoints (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) are affected.

## Recommendation
Redesign the swap allowlist to gate the economic actor rather than the direct pool caller. Two approaches:

1. **Preferred — forward the originating user through `extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. The extension can enforce that the decoded address matches a trusted source.

2. **Alternative — add an explicit `swapper` parameter to `pool.swap`:** Mirror the deposit path's `owner` parameter. The pool accepts a `swapper` address (defaulting to `msg.sender` for direct calls), and the extension checks that address. The router passes `msg.sender` as `swapper`.

Until fixed, pools that require per-user swap gating should not rely on `SwapAllowlistExtension` for router-mediated flows.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension configured
  - Pool admin calls setAllowedToSwap(pool, router, true)
    // allowlist the router so users can swap via router
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack (Mode A — bypass):
  1. attacker (not individually allowlisted) calls:
       router.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(recipient, ...) with msg.sender = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] == true  ✓
  5. Swap executes — attacker trades on the curated pool

Broken invariant:
  allowedSwapper[pool][attacker] == false, yet the swap succeeds.
  The per-user allowlist is completely bypassed for any user routing through the router.

Verification (Mode B — broken functionality):
  1. Pool admin calls setAllowedToSwap(pool, alice, true)
     // does NOT allowlist router
  2. alice calls router.exactInputSingle({pool: pool, ...})
  3. Router calls pool.swap(...) with msg.sender = router
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] == false  ✗
  5. Reverts with NotAllowedToSwap — alice cannot use the router despite being allowlisted
```
