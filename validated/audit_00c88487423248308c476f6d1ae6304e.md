### Title
Duplicate-Leaf Path Collision in Odd-Length Merkle Trees Allows False Transaction Inclusion at Nonexistent Index — (`merkle-tools/src/lib.rs`)

### Summary
`compute_root_from_merkle_proof` accepts a caller-supplied `transaction_position` with no bounds check against the actual tree size. For any odd-length transaction tree, the proof generated by `merkle_proof_calculator` for the last leaf (index `N-1`) also verifies correctly when submitted with `tx_index = N` (a nonexistent index), because the duplicate-leaf sibling is identical to the leaf itself, making `H(leaf, sibling)` and `H(sibling, leaf)` produce the same hash.

### Finding Description

`merkle_proof_calculator` handles odd-length trees by appending a duplicate of the last element before computing each level: [1](#0-0) 

This means for the last leaf at index `N-1` (even), the sibling pushed into the proof is `current_hashes[N]` — which is the duplicate of `current_hashes[N-1]`, i.e., the leaf itself.

`compute_root_from_merkle_proof` uses only the parity of `transaction_position` to decide left/right ordering: [2](#0-1) 

When `tx_index = N` (odd), the verifier computes `H(proof[0], leaf)` instead of `H(leaf, proof[0])`. Since `proof[0] == leaf` (the duplicate), both orderings produce `H(leaf, leaf)` — identical to the correct computation. All subsequent levels are also identical because `N/2 == (N-1)/2` for even `N-1`.

`verify_transaction_inclusion` passes the caller-supplied `tx_index` directly to `compute_root_from_merkle_proof` with no upper-bound validation: [3](#0-2) 

`verify_transaction_inclusion_v2` also delegates to `verify_transaction_inclusion` after its coinbase check, so it is equally affected: [4](#0-3) 

### Impact Explanation

An unprivileged NEAR caller can call `verify_transaction_inclusion` (or `verify_transaction_inclusion_v2`) with:
- `tx_id` = the real last transaction hash `T_{N-1}`
- `tx_index` = `N` (nonexistent — the block only has `N` transactions, indexed `0..N-1`)
- `merkle_proof` = the valid proof generated for index `N-1`
- `confirmations` = any valid value

The contract returns `true`, falsely asserting that `T_{N-1}` exists at index `N`. Any downstream NEAR contract or bridge application that relies on this return value to authorize a cross-chain action (e.g., token unlock, proof-of-payment) will be deceived. The invariant that a proof uniquely binds a `tx_id` to a specific `tx_index` is broken.

This affects every odd-length Bitcoin/Dogecoin/Litecoin/Zcash block (roughly half of all blocks).

### Likelihood Explanation

- No privileged role is required. Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are public NEAR view/call methods gated only by `#[pause]`.
- The attacker needs only publicly available block data (the real transaction list and merkle root) to construct the proof.
- The collision is deterministic and requires no brute force.
- The deprecated `verify_transaction_inclusion` is still callable and `verify_transaction_inclusion_v2` is the current recommended path — both are vulnerable.

### Recommendation

In `compute_root_from_merkle_proof`, validate that `transaction_position < (1 << merkle_proof.len())` — i.e., the position must be a valid leaf index for a tree of the given proof depth. Alternatively, pass the known tree size (number of transactions) from the block header context and reject any `tx_index >= tx_count`. The simplest fix is to add a bound check in `verify_transaction_inclusion` before calling `compute_root_from_merkle_proof`:

```rust
require!(
    args.tx_index < header.block_header.tx_count, // or equivalent bound
    "tx_index out of range"
);
```

If `tx_count` is not stored in the header, the proof length alone can bound the maximum valid index: `tx_index` must be `< 2^(proof.len())`, but the tighter fix is to store and check the actual transaction count.

pub fn merkle_proof_calculator(tx_hashes: Vec<H256>, transaction_position: usize) -> Vec<H256> {
    let mut transaction_position = transaction_position;
    let mut merkle_proof = Vec::new();
    let mut current_hashes = tx_hashes;

    while current_hashes.len() > 1 {
        if current_hashes.len() % 2 == 1 {
            current_hashes.push(current_hashes[current_hashes.len() - 1].clone());
        }

        if transaction_position % 2 == 1 {
            merkle_proof.push(current_hashes[transaction_position - 1].clone());
        } else {
            merkle_proof.push(current_hashes[transaction_position + 1].clone());
        }

        let mut new_hashes = Vec::new();

        for i in (0..current_hashes.len() - 1).step_by(2) {
            new_hashes.push(compute_hash(&current_hashes[i], &current_hashes[i + 1]));
        }

        current_hashes = new_hashes;
        transaction_position /= 2;
    }

    merkle_proof
}

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
