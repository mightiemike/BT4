### Title
Unenforced Coinbase Merkle Proof Requirement — Deprecated `verify_transaction_inclusion` Bypasses 64-Byte Forgery Protection (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to enforce a coinbase merkle proof requirement that prevents the 64-byte transaction Merkle forgery attack. However, the original `verify_transaction_inclusion` function remains publicly callable on-chain with no access control. Any unprivileged NEAR caller can bypass the coinbase proof requirement entirely by invoking the deprecated function directly, obtaining a `true` result for a forged transaction inclusion proof.

---

### Finding Description

The contract explicitly documents the 64-byte transaction Merkle forgery vulnerability and introduced `verify_transaction_inclusion_v2` as the fix: [1](#0-0) 

The v2 function enforces two requirements before delegating to v1:
1. `merkle_proof.len() == coinbase_merkle_proof.len()`
2. The coinbase transaction's Merkle proof must resolve to the block's stored `merkle_root` [2](#0-1) 

However, the original `verify_transaction_inclusion` is still a fully public, on-chain callable method. It carries only a Rust `#[deprecated]` attribute, which generates a compiler warning in Rust source code but **does not restrict on-chain invocation**. Any NEAR account can call it directly via a signed transaction: [3](#0-2) 

The function's own warning states the root cause explicitly: [4](#0-3) 

The Merkle proof verifier in `merkle-tools` uses the caller-supplied `transaction_position` to determine left/right hashing at each level. An internal 64-byte node can be supplied as `tx_id` with a crafted `tx_index` and a zero-length or single-element `merkle_proof` to make `compute_root_from_merkle_proof` return the stored `merkle_root`: [5](#0-4) 

The only guard in v1 against an empty proof is: [6](#0-5) 

A proof of length ≥ 1 with a crafted internal node passes this check.

---

### Impact Explanation

A downstream NEAR contract or off-chain bridge that calls `verify_transaction_inclusion` (v1) and trusts its `true` return value can be deceived into believing an arbitrary, non-existent Bitcoin transaction was confirmed in a real mainchain block. The corrupted proof result is the boolean `true` returned for a forged `tx_id`. This can be used to trigger fraudulent cross-chain asset releases, unlock bridge funds, or satisfy any on-chain condition gated on SPV proof verification.

---

### Likelihood Explanation

The 64-byte attack is publicly documented (referenced in the contract's own deprecation notice), requires no privileged access, and needs only a known block hash present in the mainchain map. The attacker supplies all parameters: `tx_id`, `tx_block_blockhash`, `tx_index`, and `merkle_proof`. The entry point is a standard public NEAR contract call. Likelihood is **medium-high** for any deployment where downstream contracts consume v1 results.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or gate it with an access-control role (e.g., `Role::DAO`) so it cannot be called by unprivileged accounts. Alternatively, inline the coinbase proof check directly into v1 and remove the separate v2 function. The `#[deprecated]` attribute alone provides no on-chain enforcement.

---

### Proof of Concept

1. Identify any block hash `B` present in `mainchain_header_to_height` (publicly readable via `get_block_hash_by_height`).
2. Retrieve the stored `merkle_root` from `headers_pool` for block `B`.
3. Construct a forged `tx_id` equal to the `merkle_root` itself (a valid 32-byte value). With `tx_index = 0` and `merkle_proof = []` (length 1 minimum — use a single sibling that hashes with `tx_id` to produce `merkle_root`), `compute_root_from_merkle_proof(tx_id, 0, proof)` returns `merkle_root`.
4. Call `verify_transaction_inclusion` with `{ tx_id: forged, tx_block_blockhash: B, tx_index: 0, merkle_proof: [sibling], confirmations: 1 }`.
5. The function returns `true` for a transaction that does not exist in the Bitcoin blockchain.

The v2 coinbase proof requirement — the only guard against this attack — is completely absent from the v1 code path and is not enforced by any on-chain mechanism.

    /// Verifies that a transaction is included in a block at a given block height
    ///
    /// # Deprecated
    /// Use [`verify_transaction_inclusion_v2`] instead, which includes coinbase merkle proof validation
    /// to mitigate the 64-byte transaction Merkle proof forgery vulnerability:
    /// https://www.bitmex.com/blog/64-Byte-Transactions
    ///
    /// @param `tx_id` transaction identifier
    /// @param `tx_block_blockhash` block hash at which transacton is supposedly included
    /// @param `tx_index` index of transaction in the block's tx merkle tree
    /// @param `merkle_proof` merkle tree path (concatenated LE sha256 hashes) (does not contain initial `transaction_hash` and `merkle_root`)
    /// @param confirmations how many confirmed blocks we want to have before the transaction is valid
    /// @return True if `tx_id` is at the claimed position in the block at the given blockhash, False otherwise
    ///
    /// # Warning
    /// This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash.
    /// We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification.
    ///
    /// # Panics
    /// Multiple cases
    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
