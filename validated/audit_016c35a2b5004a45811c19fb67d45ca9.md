Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Allowing Any User to Bypass the Swap Allowlist — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of the pool's `swap` call. When a user swaps through `MetricOmmSimpleRouter`, the router is `msg.sender` at the pool level, so the extension checks the **router's address** against the allowlist rather than the **end user's address**. If the pool admin allowlists the router (the only way to permit router-based swaps), every unpermissioned user can bypass the curated allowlist by routing through the router.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← router address when called via router
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this value verbatim to the extension:

```solidity
// ExtensionCalling.sol L162-176
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, zeroForOne, ...)
)
```

`SwapAllowlistExtension.beforeSwap` then checks that `sender` is on the allowlist keyed by `msg.sender` (the pool):

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)`. At the pool, `msg.sender` is the router, so `sender` forwarded to the extension is the **router address**. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

The router passes `params.extensionData` through from the user's call parameters without encoding the original caller, and the `SwapAllowlistExtension` does not decode `extensionData` — it only checks `sender`. There is no existing mechanism to propagate the real end-user identity through the router to the extension.

A pool admin who wants allowlisted users to access the standard router UX must call `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, **any** address can call the router and have the extension pass, because the extension only sees the router as the swapper — not the actual end user.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` is intended to restrict trading to a curated set of addresses (e.g., KYC-verified counterparties, whitelisted market makers). Once the pool admin allowlists the router to support standard UX, the allowlist is completely bypassed for all router-mediated swaps. Any unpermissioned user can trade against the pool's liquidity, violating the pool's curation policy and potentially draining LP funds through trades the pool was designed to reject. This is a direct bypass of a configured access-control guard on a production swap path, reachable by any unprivileged user with no special setup.

**Severity: High** — broken core pool access-control functionality causing potential loss of LP funds.

## Likelihood Explanation

The scenario is highly likely in practice:
1. A pool admin deploys a pool with `SwapAllowlistExtension` to restrict trading.
2. The admin allowlists specific users and also allowlists the router so those users can use the standard interface.
3. Any non-allowlisted user discovers they can call the router and bypass the guard.

The router is the canonical swap entrypoint in the periphery; allowlisting it is the natural and expected action for any pool admin who wants their allowlisted users to have a normal UX. The bypass requires no special privileges, no flash loans, and no unusual setup — just calling the public router.

## Recommendation

The `SwapAllowlistExtension` should check the **originating user** rather than the immediate pool caller. Two approaches:

1. **Pass the original caller through the router**: The router should forward the original `msg.sender` in `extensionData`, and the extension should decode and check that address.
2. **Check `recipient` instead of `sender`**: If the pool's design guarantees that `recipient` is always the end user, the extension could check `recipient`. However, this changes the semantics and must be validated against the full call path.

The cleanest fix is approach 1: the router encodes the original caller in `extensionData`, and the extension decodes and verifies it, while still enforcing `onlyPool` on `msg.sender`.

## Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
2. Pool admin calls setAllowedToSwap(pool, alice, true)       // allowlist Alice
3. Pool admin calls setAllowedToSwap(pool, router, true)      // allow router so Alice can use it
4. Bob (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle(...)
   → router calls pool.swap(recipient=Bob, ...)
   → pool calls _beforeSwap(sender=router, ...)
   → extension checks allowedSwapper[pool][router] == true  ✓
   → swap executes — Bob bypasses the allowlist entirely
