### Title
Unauthenticated `SetNodeDynamicConfig` on `RemoteComponentServer` allows any network peer to overwrite live sequencer configuration including `native_classes_whitelist` — (File: `crates/apollo_config_manager/src/config_manager.rs`)

### Summary
The `ConfigManager` component exposes a `set_node_dynamic_config` mutation endpoint through the generic `RemoteComponentServer` (HTTP/2) with zero caller authentication. In any distributed deployment topology where `ConfigManager` runs in `LocalExecutionWithRemoteEnabled` mode, any process that can reach the bound TCP port can push an arbitrary `NodeDynamicConfig` — including a modified `native_classes_whitelist` — and the sequencer will silently adopt it for all subsequent block executions.

### Finding Description
The `RemoteComponentServer` in `crates/apollo_infra/src/component_server/remote_component_server.rs` is a generic HTTP/2 server that deserializes any incoming request body and forwards it to the local component client. Its handler `remote_component_server_handler` performs no authentication, no IP allowlist check, and no shared-secret verification before dispatching the request.

The `ConfigManager` component registers a `ConfigManagerRequest::SetNodeDynamicConfig` variant that, when received, calls `ConfigManager::set_node_dynamic_config`. That method runs only a structural `validate()` check on the incoming `NodeDynamicConfig` and then atomically replaces the live configuration held in `latest_node_dynamic_config`.

`NodeDynamicConfig` carries, among other fields, `batcher_dynamic_config` which contains `native_classes_whitelist`. This whitelist is the sole runtime gate that decides whether a given contract class is dispatched to the native Rust executor or to the CASM/Cairo-VM executor. Changing it mid-operation causes the sequencer to select a different compiled artifact for every subsequent transaction that touches a whitelisted class.

The `ConfigManagerRunner` (the legitimate writer) reads config from the filesystem and calls `set_node_dynamic_config` through the same `SharedConfigManagerClient` interface. Because the `RemoteComponentServer` exposes the identical interface over HTTP with no authentication, any peer on the same network segment can issue the same call.

### Impact Explanation
An attacker who can reach the `ConfigManager`'s HTTP port can:

1. **Wrong compiled class / native artifact selected for execution (Critical):** By adding an arbitrary class hash to `native_classes_whitelist`, the attacker causes the sequencer to route execution of that class to the native Rust path instead of CASM. If the native and CASM paths diverge in output, storage writes, events, or gas consumption, every block that includes a transaction touching that class will produce a wrong state root, wrong receipt, or wrong event log — a direct match to the "Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution" critical impact.

2. **Incorrect fee / gas accounting (Critical):** Removing a class from the whitelist forces it back to CASM execution with different gas metering, silently altering fee charges for all users of that contract.

3. **Consensus parameter tampering (High):** Fields such as `validator_id` inside `consensus_dynamic_config` can be overwritten, potentially causing the node to sign proposals under a wrong identity or to accept proposals it should reject.

### Likelihood Explanation
In the consolidated single-process deployment the `RemoteComponentServer` for `ConfigManager` is never started, so the attack surface is absent. In the hybrid or fully-distributed topology — which is the production configuration for mainnet and Sepolia-alpha — the `ConfigManager` runs as a separate pod/process and its `RemoteComponentServer` is bound to a configurable `bind_ip` (defaulting to `Ipv4Addr::LOCALHOST` in code but overridden to a routable address in cluster deployments). Any process that can reach that address and port — including a compromised co-located component, a misconfigured network policy, or an attacker who has lateral movement within the cluster — can trigger the attack without any credentials.

### Recommendation
**Short term:** Add a shared-secret or HMAC token check inside `remote_component_server_handler` before dispatching any mutating request variant. At minimum, gate `ConfigManagerRequest::SetNodeDynamicConfig` (and `L1GasPriceRequest::AddGasPrice` / `Initialize`) behind an IP allowlist restricted to the single legitimate caller component.

**Long term:** Implement mutual TLS between all `RemoteComponentClient` / `RemoteComponentServer` pairs so that only components holding a valid node certificate can issue requests. This mirrors the recommendation in the original report to use a multi-sig owner rather than an open setter.

### Proof of Concept

```
# 1. Identify the ConfigManager remote server port from the node config
#    (e.g. config_manager_config.remote_server_config.port = 8085)

# 2. Craft a SetNodeDynamicConfig payload that adds an attacker-controlled
#    class hash to native_classes_whitelist
PAYLOAD=$(cat <<'EOF'
{
  "SetNodeDynamicConfig": {
    "batcher_dynamic_config": {
      "native_classes_whitelist": ["0xATTACKER_CLASS_HASH"]
    },
    "consensus_dynamic_config": null,
    ...
  }
}
EOF
)

# 3. Serialize with the same bincode/SerdeWrapper framing the RemoteComponentClient uses
#    and POST to the ConfigManager's HTTP/2 endpoint
curl -s --http2-prior-knowledge \
     -H "Content-Type: application/octet-stream" \
     -H "X-Request-Id: 1" \
     --data-binary @serialized_payload.bin \
     http://<config_manager_host>:8085/

# 4. From this point forward, every transaction that invokes 0xATTACKER_CLASS_HASH
#    is dispatched to the native executor instead of CASM, producing divergent
#    execution results, wrong state roots, and incorrect fee charges.
```

The root cause is in `remote_component_server_handler` (no auth gate before `local_client.send(request)`) and `set_node_dynamic_config` (no caller identity check, only structural validation). The `ConfigManagerClient::set_node_dynamic_config` trait method is the public API surface that any holder of a `RemoteConfigManagerClient` can invoke without restriction.
