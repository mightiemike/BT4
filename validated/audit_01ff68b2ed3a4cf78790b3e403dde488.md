### Title
Difficulty Retarget Uses Mainchain Height Index During Fork Validation, Allowing Incorrect `expected_bits` Acceptance - (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

When a fork block is validated at a difficulty-retarget boundary, `get_next_work_required` calls `get_header_by_height(first_block_height)` to retrieve the timestamp of the first block in the retarget interval. `get_header_by_height` resolves exclusively through `mainchain_height_to_header` — the mainchain height index — which at validation time still reflects the *current mainchain*, not the fork chain. If the fork diverges from the mainchain at or before `first_block_height`, the mainchain block at that height is a different block than the fork's actual ancestor, and carries a different timestamp. The difficulty calculation therefore uses the wrong `first_block_time`, producing an incorrect `expected_bits`. A fork block whose `bits` field matches this wrong target passes `check_pow` even though it does not satisfy the correct retarget difficulty.

---

### Finding Description

`get_header_by_height` is implemented as:

```rust
// contract/src/lib.rs  lines 677-682
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

It always returns the **mainchain** block at the given height.

In `bitcoin.rs` the retarget path is:

```rust
// contract/src/bitcoin.rs  lines 78-86
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);

let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
```

The identical pattern appears in `litecoin.rs` (lines 86-93) and `dogecoin.rs` (lines 286-297). The Dogecoin code even carries an explicit acknowledgement of the problem:

```rust
// contract/src/dogecoin.rs  line 291
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

`check_pow` is called inside `submit_block_header` **before** any reorg is applied and before the fork block is inserted into the mainchain index. At that moment `mainchain_height_to_header` is fully populated with the current mainchain. If the fork diverged at height `F < first_block_height`, the mainchain block at `first_block_height` is a completely different block from the fork's ancestor at the same height, and its `time` field can differ arbitrarily.

---

### Impact Explanation

The corrupted state is `expected_bits` — the difficulty target that the submitted fork block must satisfy. Because `first_block_time` is taken from the wrong block, `actual_time_taken` (the measured interval duration) is wrong, and `calculate_next_work_required` / `zcash_calculate_next_work_required` produces a target that does not correspond to the fork chain's actual history. A fork block whose `bits` field matches this incorrect target passes `check_pow` and is stored. If the fork accumulates enough (incorrectly validated) chain work, `reorg_chain` promotes it to the mainchain, replacing the legitimate tip. Downstream callers of `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` then verify proofs against a fraudulently promoted chain.

---

### Likelihood Explanation

The trigger is reachable by any unprivileged NEAR account via the public `submit_blocks` entry point. No role, key, or privileged access is required. The attacker must:

1. Build a fork that diverges before the retarget boundary (every 2016 blocks for Bitcoin/Litecoin; every block post-145k for Dogecoin; every 17 blocks for Zcash).
2. Craft fork ancestors whose timestamps produce a favorable `first_block_time` when looked up through the mainchain index.
3. Submit the fork headers in a single `submit_blocks` call.

The Dogecoin post-145k case (per-block Digishield) is the most accessible: `blocks_to_go_back = 0`, so `height_first = prev_block_header.block_height`, and `get_header_by_height` is called with the fork's previous block's height. The mainchain block at that height is a different block from the fork's previous block whenever the fork is more than one block deep, making the desynchronization trivially reachable on every single fork block submission.

---

### Recommendation

Replace `get_header_by_height` calls in all three retarget functions with ancestor traversal via `get_prev_header`, walking back from `prev_block_header` the required number of steps. This ensures the timestamp is taken from the fork chain's actual ancestor rather than from the mainchain index. The Zcash implementation in `zcash.rs` already does this correctly (it walks back via `get_prev_header` in a loop), and should serve as the reference pattern for the other three modules.

---

### Proof of Concept

**Bitcoin / Litecoin scenario (retarget at height 2016):**

1. Contract is initialized with genesis at height 0. Mainchain grows to height 2016 (a retarget boundary). Mainchain block at height 0 has `time = T_main`.
2. Attacker builds a fork starting at height 0 whose block at height 0 has `time = T_fork` where `T_fork ≠ T_main`. (The genesis block is shared, but the fork's block at height 1 can have any timestamp, and the fork's ancestor at height 0 is the shared genesis — however, for a fork that diverges at height 1, `first_block_height = 2016 - 2015 = 1`, and the mainchain block at height 1 differs from the fork's block at height 1.)
3. Attacker submits fork blocks up to height 2016. When the fork block at height 2016 is validated, `get_header_by_height(1)` returns the **mainchain** block at height 1 with `time = T_main_1`.
4. The fork's actual ancestor at height 1 has `time = T_fork_1`. The attacker chose `T_fork_1` such that `T_fork_1 - T_main_1` shifts `actual_time_taken` to its minimum (`pow_target_timespan / 4`), producing the lowest possible `expected_bits` (easiest difficulty).
5. The fork block at height 2016 sets `bits` to this lowered target and passes `check_pow` with far less PoW than the legitimate retarget requires.
6. If the fork's cumulative `chain_work` exceeds the mainchain's, `reorg_chain` is triggered and the fraudulent fork becomes the new mainchain tip. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
