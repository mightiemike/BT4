### Title
`starknet_getStateUpdate` returns non-canonical `deployed_contracts` ordering when class replacements are present — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

`convert_thin_state_diff` uses `IndexMap::swap_remove` to excise class-replacement addresses from `deployed_contracts`. `swap_remove` fills the vacated slot with the last entry, destroying the address-sorted invariant of the underlying `IndexMap`. The resulting RPC `ThinStateDiff::deployed_contracts` `Vec` is then built directly from this disrupted map. The `sort()` method that exists on the RPC `ThinStateDiff` is **never called** in the `get_state_update` path, so the non-canonical ordering is served to callers.

---

### Finding Description

**Step 1 — `convert_thin_state_diff` disrupts ordering via `swap_remove`.**

The function iterates over `thin_state_diff.deployed_contracts` (an `IndexMap` whose addresses are strictly increasing per the `starknet_api` invariant), collects every address that `is_deployed` in the previous block into `replaced_classes`, then removes each one with `swap_remove`: [1](#0-0) 

`IndexMap::swap_remove` removes the target entry and moves the **last** entry into the vacated slot. For example, if the map contains addresses `[A, B, C, D]` (sorted) and `B` is a replacement, after `swap_remove(B)` the map holds `[A, D, C]` — no longer sorted.

**Step 2 — `ThinStateDiff::from` preserves the disrupted order.**

`ThinStateDiff::from` converts `diff.deployed_contracts` into a `Vec<DeployedContract>` by a plain `into_iter()` — it does not sort: [2](#0-1) 

**Step 3 — `sort()` exists but is never called in `get_state_update`.**

The RPC `ThinStateDiff` has a `sort()` method that sorts `deployed_contracts` by address: [3](#0-2) 

`get_state_update` calls `convert_thin_state_diff` and returns the result directly with no intervening `sort()`: [4](#0-3) 

---

### Impact Explanation

The **actual block hash** is not wrong — it is computed from the internal `starknet_api::ThinStateDiff` (which preserves the sorted-address invariant) before the RPC conversion ever runs.

The impact is therefore **High, not Critical**: `starknet_getStateUpdate` returns an authoritative-looking `deployed_contracts` list whose ordering diverges from the canonical address-sorted order. Any client that:
- receives the RPC response, and
- attempts to recompute the state diff commitment (e.g., to verify the block hash or cross-check with a proof)

will compute a wrong commitment hash because the input ordering is wrong. This falls squarely under **"High — RPC returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

The bug fires on every accepted block that contains at least two entries in `deployed_contracts` where one or more are class replacements (a previously-deployed contract receiving a new class hash). Class replacement is a standard Starknet operation available to any contract owner; no privileged access is required to produce such a block.

---

### Recommendation

Call `state_diff.sort()` immediately after `convert_thin_state_diff` returns inside `get_state_update`, before constructing the `AcceptedStateUpdate` response. Alternatively, call `sort()` at the end of `ThinStateDiff::from` so the invariant is enforced at construction time.

---

### Proof of Concept

```rust
// Pseudocode — build an IndexMap with three addresses sorted A < B < C,
// mark B as a replacement, call convert_thin_state_diff logic,
// assert the resulting Vec<DeployedContract> is still A < C (not A, C, B or A, C).
let mut map: IndexMap<ContractAddress, ClassHash> = indexmap![
    addr(0x10) => class(1),
    addr(0x20) => class(2),   // <-- replacement
    addr(0x30) => class(3),
];
map.swap_remove(&addr(0x20));
// map is now [addr(0x10), addr(0x30)] — still sorted only by luck here.
// With four entries [0x10, 0x20, 0x30, 0x40] and removing 0x20:
// result is [0x10, 0x40, 0x30] — NOT sorted.
let deployed: Vec<_> = map.into_iter().collect();
assert!(deployed.windows(2).all(|(a, b)| a.0 < b.0),
    "deployed_contracts ordering violated after swap_remove");
```

The assertion fails for any map with ≥ 3 entries where a non-last entry is removed.
