### Title
Unrestricted Public Access to `run_mainchain_gc` Allows Any Caller to Corrupt Canonical Chain State - (File: `contract/src/lib.rs`)

### Summary
`run_mainchain_gc` is a public, state-mutating function on `BtcLightClient` that carries no caller-identity restriction when the contract is unpaused. Any unprivileged NEAR account can invoke it with an arbitrary `batch_size`, causing the contract to permanently delete mainchain block headers from storage, advancing `mainchain_initial_blockhash` forward and breaking SPV proof verification for the purged range.

### Finding Description
The function is declared as:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
``` [1](#0-0) 

The `#[pause(except(roles(Role::UnrestrictedRunGC)))]` attribute only restricts the function when the contract is **paused** — in that state, only holders of `UnrestrictedRunGC` may call it. When the contract is **unpaused** (its normal operating mode), the attribute imposes no restriction whatsoever. There is no `#[trusted_relayer]` guard, no `#[private]`, and no explicit `require!` on `env::predecessor_account_id()`. [2](#0-1) 

The `#[trusted_relayer]` macro is declared on the `impl` block but only activates on individual methods that carry the `#[trusted_relayer]` attribute — `submit_blocks` has it, `run_mainchain_gc` does not. [3](#0-2) 

Inside the function, when `amount_of_headers_we_store > self.gc_threshold`, it computes `selected_amount_to_remove = min(total_amount_to_remove, batch_size)` and then permanently deletes those headers from both `mainchain_height_to_header` and `headers_pool`, then advances `mainchain_initial_blockhash`: [4](#0-3) 

### Impact Explanation
The deleted entries are gone from on-chain storage permanently. After deletion, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` will panic with `"block does not belong to the current main chain"` for any transaction whose containing block was in the purged height range, because `mainchain_header_to_height` no longer contains that block hash. [5](#0-4) 

An attacker who calls `run_mainchain_gc` repeatedly (each call removes up to `total_amount_to_remove` headers) can drain the entire stored mainchain history down to exactly `gc_threshold` blocks, permanently destroying SPV verifiability for all older transactions. Because `mainchain_initial_blockhash` is advanced, the contract's own view of its oldest retained block is corrupted, and the damage cannot be undone without a full re-sync from genesis.

### Likelihood Explanation
The entry point is a standard NEAR contract call requiring no deposit, no role, and no special key — any NEAR account can submit it. The contract is designed to be publicly accessible (relayers and SPV consumers interact with it), so the attack surface is fully reachable. The condition `amount_of_headers_we_store > gc_threshold` is met in normal operation once the chain has been running for longer than the GC window (e.g., >52,704 blocks ≈ 1 year for a production deployment). Likelihood is **high** once the chain matures past the threshold.

### Recommendation
Add a caller-identity guard to `run_mainchain_gc` equivalent to the one on `submit_blocks`. The simplest correct fix is to add `#[trusted_relayer]` (matching the relayer bypass roles already configured on the impl block) or an explicit role check such as:

```rust
require!(
    self.acl_has_role(Role::DAO, &env::predecessor_account_id())
        || self.acl_has_role(Role::UnrestrictedRunGC, &env::predecessor_account_id()),
    "Caller is not authorized to run GC"
);
```

Alternatively, restrict it to `#[private]` if it is only ever meant to be called internally (it is already called from within `submit_blocks`). [6](#0-5) 

### Proof of Concept
1. Deploy the contract with `gc_threshold = 52704` and sync it for >52,704 blocks so that `amount_of_headers_we_store > gc_threshold`.
2. From any unprivileged NEAR account (no role granted), call:
   ```
   near call <contract_id> run_mainchain_gc '{"batch_size": 52704}' --accountId attacker.near
   ```
3. The call succeeds. `selected_amount_to_remove = min(amount_of_headers_we_store - 52704, 52704)` headers are deleted from `mainchain_height_to_header` and `headers_pool`, and `mainchain_initial_blockhash` is advanced.
4. Repeat until the stored window is reduced to exactly `gc_threshold` blocks.
5. Any subsequent call to `verify_transaction_inclusion` for a transaction in a purged block now panics: `"block does not belong to the current main chain"` — SPV verification is permanently broken for that range. [7](#0-6)
