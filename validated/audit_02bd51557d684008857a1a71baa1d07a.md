Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Enabling Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` passes `msg.sender` (the direct caller of `swap`) as `sender` to `_beforeSwap`, which forwards it to `SwapAllowlistExtension.beforeSwap`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`. This either locks out allowlisted EOAs who use the router, or — if the admin adds the router to unblock them — grants every unprivileged user access to a curated pool.

## Finding Description
`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at line 230: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that `sender` value and dispatches it to the configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then gates on `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the pool's `msg.sender`: [4](#0-3) 

The same pattern applies to `exactInput` (line 104), `exactOutputSingle` (line 136), and `exactOutput` (line 165). In the multi-hop `exactInput` path, intermediate hops pass `address(this)` (the router itself) as the payer context, so the pool still sees the router as `msg.sender`: [5](#0-4) 

No existing guard threads the original EOA through the pool's `swap` signature. The `_setNextCallbackContext` call stores `msg.sender` only for payment purposes in the callback, not for extension identity checks.

## Impact Explanation
Two fund-impacting outcomes are directly reachable:

**Scenario A (broken functionality):** A pool admin allowlists specific EOAs. Those EOAs cannot use the router because the extension sees the router address and reverts with `NotAllowedToSwap`. Allowlisted traders are locked out of the standard periphery path, breaking core swap functionality.

**Scenario B (allowlist bypass):** The admin adds the router to the allowlist to unblock legitimate users. Every unprivileged EOA can now call any `exact*` function through the router and pass the extension check, because `allowedSwapper[pool][router] == true`. The per-user curation is completely nullified — any user can trade on a pool intended to be restricted, draining LP value or executing trades the pool admin explicitly prohibited. This constitutes a direct loss of LP assets and broken core pool functionality, meeting Critical/High severity thresholds.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap entrypoint shipped with the protocol. Pool admins configuring `SwapAllowlistExtension` have no reason to expect that routing through the standard periphery contract changes the identity the extension sees. The bypass requires only a normal router call — no flash loans, no reentrancy, no privileged access. Any unprivileged user can execute it.

## Recommendation
Thread the original caller through the pool's `swap` call so extensions gate the economically relevant actor. Add a `sender` parameter to `IMetricOmmPoolActions.swap` that the pool forwards to extensions, and have the router pass `msg.sender` explicitly:

```solidity
// In MetricOmmSimpleRouter.exactInputSingle:
IMetricOmmPoolActions(params.pool).swap(
    msg.sender,      // sender — the actual user
    params.recipient,
    params.zeroForOne,
    ...
);
```

The pool then passes this explicit `sender` to `_beforeSwap` instead of `msg.sender`. Alternatively, the extension can read the payer from the router's transient storage context (analogous to `_getPayer()` used for payment), but threading the real caller through the `swap` signature is the cleanest and most general fix.

## Proof of Concept

```solidity
// Setup: pool deployed with SwapAllowlistExtension.
// Admin allowlists only trustedUser.
swapAllowlist.setAllowedToSwap(pool, trustedUser, true);
// Admin is forced to also allowlist the router so trustedUser can use it.
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Attacker (not allowlisted) calls the router.
// Pool sees msg.sender = router; extension checks allowedSwapper[pool][router] == true → passes.
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: attacker,
    tokenIn: token0,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));

// Attacker received token1 despite not being on the allowlist.
assertGt(token1.balanceOf(attacker), 0);
```
