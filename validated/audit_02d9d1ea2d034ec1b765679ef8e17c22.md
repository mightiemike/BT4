Audit Report

## Title
`lastDecayTs` Not Updated for Zero-Share Bins Causes Stop-Loss Watermark to Over-Decay and Be Bypassed — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`_afterSwapOracleStopLoss` skips `_checkAndUpdateWatermarks` for bins where `totalShares == 0`, leaving `lastDecayTs` stale for the entire zero-share period. When liquidity is later re-added and a swap touches the bin, the elapsed `dt` is computed against the stale timestamp, causing `_decayed` to return 0. A zero watermark makes `_applyWatermark` unconditionally return `(metric, false)`, silently bypassing the stop-loss and allowing value-draining swaps to complete without reversion.

## Finding Description
In `_afterSwapOracleStopLoss` (lines 236–242), the loop skips any bin where `totalShares == 0` via `continue`, never calling `_checkAndUpdateWatermarks`. That function (lines 267–284) is the sole writer of `hwmS.lastDecayTs` during swap execution. When the bin is later re-populated and a swap crosses it, `dt = block.timestamp - hwmS.lastDecayTs` reflects the entire zero-share gap. In `_decayed` (lines 319–324), if `ratePerSecondE8 * dt >= 1e8` (crossed after `1e8 / rate` seconds — ~20 days at rate=58), the function returns 0. `_applyWatermark(metric, 0, floorMultiplier)` (lines 328–336) then evaluates `metric >= 0`, which is always true for `uint256`, returning `(metric, false)` — no breach — and silently ratcheting the watermark up to the current (potentially already-drained) metric value. The `OracleStopLossTriggered` revert is never reached.

## Impact Explanation
The `OracleValueStopLossExtension` is the primary on-chain guard that reverts swaps when per-share bin value falls below the configured `drawdownE6` floor. Bypassing it constitutes a direct loss of LP principal: a swap draining value beyond the configured drawdown threshold completes without reversion, and the watermark is silently reset to the post-drain metric, leaving subsequent swaps equally unprotected. This is a direct loss of user principal above the protocol-configured drawdown threshold, meeting the Critical/High impact gate.

## Likelihood Explanation
All preconditions are reachable by unprivileged actors with no oracle manipulation:
1. Any LP can fully withdraw from a bin, setting `totalShares == 0`.
2. ~20 days elapse — a realistic gap for inactive bins or during market stress.
3. Any allowed depositor re-adds liquidity to the bin.
4. Any swap crossing the bin range triggers `afterSwap` and the bypassed check.

The conditions are independent, composable, and require no coordination with privileged roles.

## Recommendation
Advance `lastDecayTs` for zero-share bins without performing the breach check or watermark ratchet:

```diff
for (uint256 i = 0; i < count; i++) {
    uint256 totalShares = PoolStateLibrary._decodeBinTotalShares(shares[i]);
    if (totalShares == 0) {
+       // Advance the decay clock so stale dt cannot accumulate.
+       highWatermarks[pool_][binIdxs[i]].lastDecayTs = uint32(block.timestamp);
        continue;
    }
    (uint104 t0, uint104 t1,,,) = PoolStateLibrary._decodeBinState(states[i]);
    (uint256 metricT0, uint256 metricT1) = _metrics(t0, t1, totalShares, minShares, midPriceX64);
    _checkAndUpdateWatermarks(pool_, binIdxs[i], metricT0, metricT1, floorMultiplier, decayRate, zeroForOne);

```

## Proof of Concept
1. Deploy pool with `OracleValueStopLossExtension`, `drawdownE6 = 50_000` (5%), `decayPerSecondE8 = 58` (~5%/day), `timelock = 0`.
2. LP Alice adds liquidity to bin 0. Admin sets watermarks: `hwm0 = 1000`, `hwm1 = 1000`, `lastDecayTs = T0`.
3. Alice removes all liquidity. `totalShares[0] = 0`.
4. 20 days pass (`dt = 1_728_000 s`). Swaps on other bins skip bin 0 each time; `lastDecayTs` for bin 0 remains `T0`.
5. Bob adds liquidity to bin 0. `totalShares[0] > 0`.
6. A swap crosses bin 0. `_checkAndUpdateWatermarks` is called:
   - `dt = 1_728_000`
   - `factor = 58 * 1_728_000 = 100_224_000 >= 1e8` → `_decayed(1000, 58, 1_728_000) = 0`
   - `_applyWatermark(metricT0, 0, 950_000)` → `metricT0 >= 0` → returns `(metricT0, false)`
7. No `OracleStopLossTriggered` revert. Swap completes. Bob's deposited value is drained beyond the 5% drawdown floor with no on-chain protection.
