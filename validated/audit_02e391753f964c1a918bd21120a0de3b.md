### Title
Deflationary ERC20 Token Deposits Corrupt Subaccount Accounting via Nominal-Amount Crediting — (File: `core/contracts/EndpointStorage.sol`)

---

### Summary

`EndpointStorage.handleDepositTransfer` pulls `amount` tokens from the user and immediately forwards the same nominal `amount` to the Clearinghouse. The slow-mode transaction queued by `Endpoint.depositCollateralWithReferral` also records the original `amount`. When the sequencer later processes the slow-mode transaction, `Clearinghouse.depositCollateral` credits the full nominal `amount` to the subaccount — never checking what the Clearinghouse actually received. If the deposited token is deflationary (charges a transfer fee), the Clearinghouse receives less than `amount`, but the subaccount is credited the full `amount`, inflating the protocol's internal accounting beyond its real token holdings.

---

### Finding Description

The deposit path is:

1. **`Endpoint.depositCollateralWithReferral`** (Endpoint.sol:123–167) calls `handleDepositTransfer(token, msg.sender, uint256(amount))` and then enqueues a `SlowModeTx` whose payload encodes `DepositCollateral{sender, productId, amount}` — the original caller-supplied `amount`. [1](#0-0) 

2. **`EndpointStorage.handleDepositTransfer`** (EndpointStorage.sol:111–119) performs two sequential transfers using the same nominal `amount`:
   - `safeTransferFrom(token, from, amount)` — pulls `amount` from the user; for a deflationary token the Endpoint actually receives `amount × (1 − fee)`.
   - `safeTransferTo(token, address(clearinghouse), amount)` — forwards the nominal `amount` to the Clearinghouse; for a deflationary token the Clearinghouse actually receives `amount × (1 − fee)²` (fee applied twice) or `amount × (1 − fee)` (fee applied once on `transferFrom` only). [2](#0-1) 

3. **`Clearinghouse.depositCollateral`** (Clearinghouse.sol:193–208) reads `txn.amount` — the original nominal value — computes `amountRealized = int128(txn.amount) * int128(multiplier)`, and calls `spotEngine.updateBalance(txn.productId, txn.sender, amountRealized)`. No balance snapshot is taken before or after the transfer to verify what was actually received. [3](#0-2) 

The `ERC20Helper.safeTransferFrom` helper (ERC20Helper.sol:23–42) simply calls `transferFrom` and checks the boolean return; it does not compare pre/post balances. [4](#0-3) 

**Enabling condition for the push to succeed**: `safeTransferTo(token, clearinghouse, amount)` in step 2 requires the Endpoint to hold at least `amount` tokens. After the pull, the Endpoint holds only `amount × (1 − fee)`. The push therefore succeeds only when the Endpoint has a prior residual balance of the deflationary token of at least `amount × fee`. Such residual balance can accumulate via slow-mode fees charged in the same token (`chargeSlowModeFee`, EndpointStorage.sol:83–93) or from any other mechanism that leaves tokens in the Endpoint. [5](#0-4) 

When the push succeeds, the Clearinghouse receives `amount × (1 − fee)` (or less), but the slow-mode record and the subsequent `depositCollateral` call both use the nominal `amount`. The subaccount is credited `amount × multiplier` while the Clearinghouse holds only `amount × (1 − fee) × multiplier` worth of real tokens.

---

### Impact Explanation

Every successful deposit of a deflationary token inflates the sum of all subaccount balances beyond the Clearinghouse's real token holdings by `amount × fee × multiplier` per deposit. Over time, the Clearinghouse becomes insolvent for that token: later withdrawals or liquidations will fail or drain other users' collateral to cover the shortfall. The broken invariant is:

```
sum(spotEngine balances for productId) == Clearinghouse.balanceOf(token)
```

This invariant is violated silently — no revert, no event — because the accounting update in `depositCollateral` is entirely disconnected from the actual on-chain transfer.

---

### Likelihood Explanation

Two conditions must hold simultaneously:

1. A deflationary ERC20 token is listed as a supported spot product (an admin/owner action, but not necessarily malicious — the admin may not audit every token for fee-on-transfer behavior).
2. The Endpoint contract holds a prior residual balance of that token (from accumulated slow-mode fees or other deposits), so the `safeTransferTo(clearinghouse, amount)` call does not revert.

Condition 1 is realistic in a permissioned but growing DEX that onboards community-requested tokens. Condition 2 is realistic once the token has been live for any period (slow-mode fees accumulate). The combination makes this a low-to-medium likelihood finding with high impact when triggered.

---

### Recommendation

In `handleDepositTransfer`, measure the Clearinghouse's actual token balance before and after the transfer and use the delta — not the nominal `amount` — as the value passed to the slow-mode transaction payload:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount)
    internal returns (uint256 actualReceived)
{
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    actualReceived = token.balanceOf(address(clearinghouse)) - before;
}
```

Then use `actualReceived` (cast to `uint128`) when constructing the `DepositCollateral` slow-mode payload in `depositCollateralWithReferral`. Alternatively, document and enforce at the product-listing level that fee-on-transfer tokens are not supported.

---

### Proof of Concept

Assume token `T` charges a 1% fee on every transfer. Endpoint has a prior balance of 1 T (from slow-mode fees).

1. Alice calls `depositCollateralWithReferral(aliceSubaccount, productId, 100e18, "")`.
2. `handleDepositTransfer` executes:
   - `safeTransferFrom(T, Alice, 100e18)`: Alice pays 100 T; Endpoint receives 99 T. Endpoint balance: 100 T (99 new + 1 prior).
   - `safeTransferTo(T, clearinghouse, 100e18)`: Endpoint sends 100 T; Clearinghouse receives 99 T (1% fee). Endpoint balance: 0 T.
3. Slow-mode tx is queued with `amount = 100e18`.
4. Sequencer processes the tx; `depositCollateral` credits Alice's subaccount with `100e18 × multiplier`.
5. Clearinghouse actually holds `99e18` worth of T.
6. Shortfall: `1e18 × multiplier` — Alice's subaccount is overcredited by 1 T equivalent. The 1 T of prior Endpoint balance (belonging to the protocol/other users) was silently consumed to fund the push, and the Clearinghouse is still short by 1 T.

Repeating this across many deposits compounds the insolvency, and the prior Endpoint balance is exhausted after the first deposit, causing all subsequent deposits to revert (DoS) until more slow-mode fees accumulate. [6](#0-5) [2](#0-1) [7](#0-6)

        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);
