### Title
Inverted Parity Check in `settlePnl` Causes PnL Settlement to Always Return Zero — (`File: core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.settlePnl` uses `if (productId % 2 == 0)` to identify perp products within a packed `productIds` bitmask. In Vertex-derived protocols (which Nado is), perp products are assigned **odd** product IDs (1, 3, 5, …) and spot products are assigned **even** IDs (0, 2, 4, …). The condition is therefore always `false` for every real perp product, causing `totalSettled` to always return `0`. The Clearinghouse then applies a zero delta to the subaccount's quote balance, meaning PnL is never settled on-chain.

---

### Finding Description

`PerpEngine.settlePnl` iterates over a packed `uint256` of product IDs and is supposed to skip spot products:

```solidity
// core/contracts/PerpEngine.sol  lines 84-103
while (productIds != 0) {
    uint32 productId = uint32(productIds & ((1 << 32) - 1));
    // otherwise it means the product is a spot.
    if (productId % 2 == 0) {          // ← BUG: even IDs are SPOTS, not perps
        (
            int128 canSettle,
            State memory state,
            Balance memory balance
        ) = getSettlementState(productId, subaccount);

        state.availableSettle -= canSettle;
        balance.vQuoteBalance -= canSettle;
        totalSettled += canSettle;
        ...
    }
    productIds >>= 32;
}
return totalSettled;
```

The developer comment reads *"otherwise it means the product is a spot"*, indicating the intent was: even → perp, odd → spot. But the actual product-ID convention (inherited from Vertex) is the opposite: `QUOTE_PRODUCT_ID = 0` (even, spot), BTC-PERP = 1 (odd, perp), BTC = 2 (even, spot), ETH-PERP = 3 (odd, perp), etc.

Because every perp product ID is odd, `productId % 2 == 0` is **always false** for perp products. The body of the `if` block is never entered, `totalSettled` remains `0`, and the function returns `0` unconditionally.

The correct guard is `if (productId % 2 != 0)`. [1](#0-0) 

The caller in `Clearinghouse._settlePnl` blindly applies the returned value:

```solidity
int128 amountSettled = perpEngine.settlePnl(subaccount, productIds);
_spotEngine().updateBalance(QUOTE_PRODUCT_ID, subaccount, amountSettled);
```

Because `amountSettled` is always `0`, the quote balance update is a no-op. [2](#0-1) 

---

### Impact Explanation

`getHealth` aggregates spot and perp health contributions and is used for every health-sensitive operation: [3](#0-2) 

Because unsettled positive perp PnL is never converted into quote balance, the subaccount's spot health contribution is permanently understated. This produces four concrete harms:

1. **Unfair liquidation** — `isUnderMaintenance` returns `true` for accounts that are actually healthy, allowing liquidators to liquidate solvent positions.
2. **Reduced borrow capacity** — `_isAboveInitial` gates `transferQuote`; understated health blocks legitimate quote transfers.
3. **Blocked withdrawals** — `withdrawCollateral` requires `getHealth(...) >= 0`; users with large unrealised gains cannot withdraw collateral they are entitled to.
4. **Blocked NLP burns** — `burnNlp` checks maintenance health; users are blocked from burning NLP even when their true health is positive. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

`settlePnl` is a standard sequencer-submitted transaction that any user can trigger by requesting PnL settlement. The broken condition is unconditional — it fires on every call regardless of market conditions, user state, or product configuration. Any account holding a perp position with positive PnL is affected every time settlement is attempted.

---

### Recommendation

Invert the parity check so that odd product IDs (perps) are processed and even product IDs (spots) are skipped:

```solidity
// Before (wrong):
if (productId % 2 == 0) {

// After (correct):
if (productId % 2 != 0) {
```

---

### Proof of Concept

1. User opens a long BTC-PERP position (product ID 1, odd).
2. Price moves in their favour; `calculatePositionPnl` returns a positive value.
3. Sequencer submits a `SettlePnl` transaction with `productIds` containing product ID 1.
4. `PerpEngine.settlePnl` enters the loop: `productId = 1`, `1 % 2 == 0` → **false** → body skipped.
5. Function returns `0`.
6. `Clearinghouse._settlePnl` calls `spotEngine.updateBalance(QUOTE_PRODUCT_ID, subaccount, 0)` — no-op.
7. Subaccount quote balance is unchanged; health score remains below its true value.
8. A liquidator calls `liquidateSubaccountImpl`; `isUnderMaintenance` returns `true` because the unsettled PnL is not reflected in the quote balance; the healthy account is liquidated. [6](#0-5) [7](#0-6)

    function settlePnl(bytes32 subaccount, uint256 productIds)
        external
        returns (int128)
    {
        _assertInternal();
        int128 totalSettled = 0;

        while (productIds != 0) {
            uint32 productId = uint32(productIds & ((1 << 32) - 1));
            // otherwise it means the product is a spot.
            if (productId % 2 == 0) {
                (
                    int128 canSettle,
                    State memory state,
                    Balance memory balance
                ) = getSettlementState(productId, subaccount);

                state.availableSettle -= canSettle;
                balance.vQuoteBalance -= canSettle;

                totalSettled += canSettle;

                _setState(productId, state);
                _setBalanceAndUpdateBitmap(productId, subaccount, balance);
            }
            productIds >>= 32;
        }
        return totalSettled;
    }
