### Title
Insufficient Bonus Pool Blocks All Exit Paths, Permanently Locking User Principal — (`x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

When the bonus rewards pool is empty or insufficient, every exit path from a tier position (`MsgTierUndelegate`, `MsgExitTierWithDelegation`) fails atomically because each handler calls `claimRewards` before executing the delegation transfer. A user whose position has accrued any non-zero bonus cannot recover their staked principal until an external party replenishes the pool.

---

### Finding Description

`MsgTierUndelegate` and `MsgExitTierWithDelegation` both call `claimRewards` as a mandatory first step before modifying the position. [1](#0-0) [2](#0-1) 

`claimRewards` calls `processEventsAndClaimBonus`, which calls `sufficientBonusPoolBalance`. If the pool cannot cover the accrued bonus, it returns `ErrInsufficientBonusPool` and the entire transaction is rolled back. [3](#0-2) [4](#0-3) [5](#0-4) 

The only remaining exit path, `MsgWithdrawFromTier`, requires the position to already be undelegated — validated by `validateWithdrawFromTier` which rejects delegated positions with `ErrPositionDelegated`. [6](#0-5) 

`MsgClearPosition` (cancel exit) and `MsgTierRedelegate` also call `claimRewards` first, so they are equally blocked. [7](#0-6) 

The broken invariant is stated in `ExitWithFullDelegation`: after the exit commitment elapses, a user should always be able to exit and recover their principal. [8](#0-7) 

The ADR itself acknowledges this design choice for user-driven paths but does not address the consequence that principal recovery is blocked: [9](#0-8) 

Contrast with `BeforeRedelegationSlashed`, which explicitly forfeits bonus silently when the pool is insufficient to avoid chain halt — the same logic should apply to exit paths to avoid principal lock: [10](#0-9) 

---

### Impact Explanation

The user's staked principal is held as a live staking delegation at the position's per-position delegator address (`pos.DelegatorAddress`). Neither `MsgTierUndelegate` nor `MsgExitTierWithDelegation` can execute when the pool is empty, so the principal cannot be recovered. The corrupted invariant: after `ExitUnlockAt` elapses, a position owner must be able to exit and recover their principal regardless of pool state. `ErrInsufficientBonusPool` breaks this invariant, locking the principal indefinitely.

---

### Likelihood Explanation

The bonus pool is funded by governance/community and drains continuously as bonus rewards are paid out. The `BeginBlocker` also drains the pool for base-rewards top-up. If governance does not replenish the pool — a realistic scenario for deprecated or sunset tiers (marked `CloseOnly`) — the pool reaches zero. Any position with non-zero accrued bonus is then permanently locked. The `CloseOnly` governance path explicitly blocks new positions while allowing exits, but if the pool is empty at that point, exits are also blocked. [11](#0-10) 

---

### Recommendation

For `MsgTierUndelegate` and `MsgExitTierWithDelegation`, decouple the bonus claim from the exit execution. If `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, log the forfeiture and continue with the exit (identical to the `BeforeRedelegationSlashed` pattern). Base rewards should still be claimed. This ensures principal recovery is never gated on pool solvency.

---

### Proof of Concept

1. User calls `MsgLockTier` — tokens delegated, position created.
2. Time passes — bonus accrues (`LastBonusAccrual` advances, `computeSegmentBonus` returns non-zero).
3. User calls `MsgTriggerExitFromTier` — `ExitUnlockAt` set.
4. Exit commitment elapses (`block_time >= ExitUnlockAt`).
5. Bonus pool is empty (drained by prior claims or `BeginBlocker` top-up).
6. User calls `MsgTierUndelegate`:
   - `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` → `ErrInsufficientBonusPool` → tx reverts.
7. User calls `MsgExitTierWithDelegation` with full position amount:
   - Same failure path — tx reverts.
8. User calls `MsgWithdrawFromTier`:
   - `validateWithdrawFromTier` returns `ErrPositionDelegated` — position is still delegated because step 6/7 failed.
9. User's principal is locked indefinitely until an external party funds the pool. [12](#0-11) [13](#0-12) [14](#0-13)

func (ms msgServer) TierUndelegate(ctx context.Context, msg *types.MsgTierUndelegate) (*types.MsgTierUndelegateResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	pos, err := ms.getPositionState(ctx, msg.PositionId)
	if err != nil {
		return nil, err
	}

	if err := ms.validateUndelegatePosition(ctx, pos, msg.Owner); err != nil {
		return nil, err
	}

	pos, _, _, err = ms.claimRewards(ctx, pos)
	if err != nil {
		return nil, err
	}

	srcValidator := pos.Delegation.ValidatorAddress
	valAddr, err := sdk.ValAddressFromBech32(srcValidator)
	if err != nil {
		return nil, err
	}

	delAddr, err := sdk.AccAddressFromBech32(pos.DelegatorAddress)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")
	}

	completionTime, _, err := ms.undelegate(ctx, delAddr, valAddr, pos.Delegation.Shares)
	if err != nil {
		return nil, err
	}

	pos.ClearBonusCheckpoints()

	if err := ms.setPosition(ctx, pos.Position, &ValidatorTransition{PreviousAddress: srcValidator}); err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventPositionUndelegated{
		PositionId:     pos.Id,
		TierId:         pos.TierId,
		Owner:          pos.Owner,
		Validator:      srcValidator,
		CompletionTime: completionTime,
	}); err != nil {
		return nil, err
	}

	return &types.MsgTierUndelegateResponse{
		CompletionTime: completionTime,
		PositionId:     pos.Id,
	}, nil

func (ms msgServer) WithdrawFromTier(ctx context.Context, msg *types.MsgWithdrawFromTier) (*types.MsgWithdrawFromTierResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	pos, err := ms.getPositionState(ctx, msg.PositionId)
	if err != nil {
		return nil, err
	}

	if err := ms.validateWithdrawFromTier(ctx, pos, msg.Owner); err != nil {
		return nil, err
	}

	ownerAddr, err := sdk.AccAddressFromBech32(pos.Owner)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid owner address")
	}

	delAddr, err := sdk.AccAddressFromBech32(pos.DelegatorAddress)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")
	}

	balances := ms.bankKeeper.SpendableCoins(ctx, delAddr)
	if !balances.IsZero() {
		if err := ms.bankKeeper.SendCoins(ctx, delAddr, ownerAddr, balances); err != nil {
			return nil, err
		}
	}

	if err := ms.deletePosition(ctx, pos.Position, nil); err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventPositionWithdrawn{
		Position: pos.Position,
		Amount:   balances,
	}); err != nil {
		return nil, err
	}

	return &types.MsgWithdrawFromTierResponse{
		Amount: balances,
	}, nil
}

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
