### Title
Slow-Mode Transaction Data Permanently Deleted Before Execution Outcome Is Known — (`core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint::_executeSlowModeTransaction()` deletes the queued slow-mode transaction record **before** attempting to execute it. If execution fails and the failure is silently swallowed by the surrounding `try/catch`, the transaction data is permanently gone. For deposit-type slow-mode transactions, the user's tokens are already held by the contract at that point, leaving them irrecoverable. The removed comment `// try return funds now removed` confirms the team was aware a recovery path once existed and deliberately removed it, leaving no fallback.

---

### Finding Description

In `Endpoint::_executeSlowModeTransaction()`, the slow-mode transaction record is unconditionally deleted and the cursor advanced **before** the call to `processSlowModeTransaction` is attempted:

```solidity
SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
delete slowModeTxs[_slowModeConfig.txUpTo++];   // ← data gone before call
``` [1](#0-0) 

The call is then wrapped in a `try/catch` that silently discards any revert (except out-of-gas):

```solidity
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
        assembly { invalid() }
    }
    // try return funds now removed
}
``` [2](#0-1) 

After the outer function writes `slowModeConfig` back to storage, `txUpTo` has been incremented past the deleted slot. The failed transaction is permanently unrecoverable. [3](#0-2) 

**Deposit path — funds stuck.** When a user calls `depositCollateralWithReferral`, tokens are transferred from the user to the `Endpoint` contract *at submission time*, and a `DepositCollateral` slow-mode transaction is queued: [4](#0-3) 

If the corresponding slow-mode transaction later fails during execution (e.g., because the product was delisted in the intervening 3-day window, causing `_decimals(productId)` to revert inside `clearinghouse.depositCollateral`), the slow-mode record is deleted, `txUpTo` advances, and the user's tokens remain in the `Endpoint` contract with no on-chain path to recover them. [5](#0-4) 

**Withdrawal path — forced re-queue with fee loss.** For `WithdrawCollateral` slow-mode transactions, if a user's health falls below zero between submission and execution (due to normal trading losses), `clearinghouse.withdrawCollateral` reverts on the health check: [6](#0-5) 

The slow-mode record is deleted, the slow-mode fee already paid at submission is not refunded, and the user must re-submit and pay again. This path requires no admin action — any unprivileged actor can call `executeSlowModeTransaction()` to trigger it. [3](#0-2) 

---

### Impact Explanation

| Scenario | Trigger | Asset delta |
|---|---|---|
| Deposit slow-mode tx fails | Product delisted before execution | User tokens permanently locked in `Endpoint` |
| Withdrawal slow-mode tx fails | User health < 0 at execution time | Slow-mode fee lost; user must re-queue and wait another 3-day delay |

The deposit scenario is the most severe: tokens are already in the contract, the slow-mode record is gone, and there is no admin-callable recovery function for this case. The withdrawal scenario is directly reachable by any unprivileged caller who calls `executeSlowModeTransaction()` at the right moment.

---

### Likelihood Explanation

- `executeSlowModeTransaction()` is `external` with no access control — any address can advance the queue.
- The 3-day `SLOW_MODE_TX_DELAY` creates a wide window during which conditions can change (product delisting, health deterioration).
- The withdrawal failure path requires only that a user's health drops below zero between submission and execution, which is a normal market event.

---

### Recommendation

1. Move `delete slowModeTxs[...]` and the `txUpTo` increment to **after** a successful execution, not before.
2. On caught failure, either preserve the slow-mode record for retry or re-implement the removed fund-return path (as the comment `// try return funds now removed` acknowledges was once present).
3. At minimum, emit a failure event so off-chain tooling can detect and surface stuck deposits.

---

### Proof of Concept

```
// Withdrawal path (no admin action required):
// 1. User submits WithdrawCollateral slow-mode tx; slow-mode fee charged.
// 2. User's health drops below 0 due to trading losses.
// 3. Anyone calls Endpoint.executeSlowModeTransaction().
// 4. processSlowModeTransaction → clearinghouse.withdrawCollateral reverts
//    on getHealth(...) >= 0 check.
// 5. try/catch swallows the revert.
// 6. slowModeTxs[txUpTo] is already deleted; txUpTo already incremented.
// 7. User's slow-mode tx is permanently gone; fee is not refunded.
// 8. User must re-submit and pay the slow-mode fee again.

// Deposit path (product delisted between deposit and execution):
// 1. User calls depositCollateralWithReferral(subaccount, productId, amount).
//    → tokens transferred from user to Endpoint (handleDepositTransfer).
//    → DepositCollateral slow-mode tx queued.
// 2. Admin delists productId via DelistProduct slow-mode tx.
// 3. Anyone calls Endpoint.executeSlowModeTransaction() for the deposit tx.
// 4. processSlowModeTransaction → clearinghouse.depositCollateral
//    → _decimals(productId) reverts (token address is now 0).
// 5. try/catch swallows the revert.
// 6. slowModeTxs[txUpTo] already deleted; txUpTo already incremented.
// 7. User tokens remain in Endpoint with no recovery path.
```

            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
