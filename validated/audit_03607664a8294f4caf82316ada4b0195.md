### Title
Unprotected `initialize` in `WithdrawPool` Allows Front-Running to Seize Ownership and Drain Funds — (File: `core/contracts/WithdrawPool.sol`)

---

### Summary

`WithdrawPool.initialize` carries no access control and no `initializer` modifier on the public entry point. Any unprivileged caller who races the deployment transaction can become the contract owner and inject attacker-controlled `clearinghouse` and `verifier` addresses, enabling complete drainage of all ERC-20 tokens held by the pool.

---

### Finding Description

`WithdrawPool.initialize` is declared `external` with no guard:

```solidity
// core/contracts/WithdrawPool.sol
function initialize(address _clearinghouse, address _verifier) external {
    _initialize(_clearinghouse, _verifier);
}
``` [1](#0-0) 

The `initializer` modifier lives only on the internal `_initialize` in `BaseWithdrawPool`, not on the public entry point:

```solidity
// core/contracts/BaseWithdrawPool.sol
function _initialize(address _clearinghouse, address _verifier)
    internal
    initializer
{
    __Ownable_init();
    clearinghouse = _clearinghouse;
    verifier = _verifier;
}
``` [2](#0-1) 

Because `__Ownable_init()` sets `msg.sender` as owner, whoever calls `initialize` first wins ownership and controls both critical address slots. The `_disableInitializers()` in the constructor only protects the implementation contract; the proxy's storage is wide open until the first `initialize` call lands. [3](#0-2) 

Compare this to every other upgradeable contract in the protocol, which either places `initializer` directly on the public function (`Clearinghouse`, `Endpoint`) or adds an explicit caller check (`ContractOwner` requires `_deployer == msg.sender`): [4](#0-3) [5](#0-4) 

`WithdrawPool` is the only contract that exposes a bare `external` wrapper with no such protection.

---

### Impact Explanation

After a successful front-run the attacker holds three independent draining vectors:

1. **`removeLiquidity` (owner-gated)** — attacker is now `owner` and can call `removeLiquidity(productId, amount, sendTo)` to transfer any ERC-20 balance out of the pool immediately. [6](#0-5) 

2. **`submitWithdrawal` (clearinghouse-gated)** — the only guard is `require(msg.sender == clearinghouse)`. With attacker-controlled `clearinghouse`, the attacker can call this directly to mark arbitrary indices and trigger token transfers to any address. [7](#0-6) 

3. **`submitFastWithdrawal` (verifier-gated)** — signature validation is delegated to `Verifier(verifier).requireValidTxSignatures(...)`. With attacker-controlled `verifier`, the attacker deploys a contract that always passes validation and submits fabricated fast-withdrawal transactions to drain tokens. [8](#0-7) 

The corrupted state is: `owner`, `clearinghouse`, and `verifier` storage slots — all set to attacker-controlled values in a single transaction. The legitimate `initialize` call subsequently reverts (already initialized), leaving the pool permanently compromised.

---

### Likelihood Explanation

The attack requires only a standard mempool front-run on a single transaction during deployment — no special privileges, no leaked keys, no governance capture. The window is the gap between proxy deployment and the `initialize` call. On any public EVM chain (including Ink Chain) this is a realistic, low-cost attack. The attacker needs only gas.

---

### Recommendation

Add the `initializer` modifier directly to `WithdrawPool.initialize`, matching the pattern used by every other upgradeable contract in the protocol:

```solidity
// WithdrawPool.sol
function initialize(address _clearinghouse, address _verifier) external initializer {
    _initialize(_clearinghouse, _verifier);
}
```

Alternatively, remove the `initializer` modifier from `_initialize` and keep it only on the public function. Either way, the `initializer` guard must be present on the externally callable entry point, not only on the internal helper.

---

### Proof of Concept

1. Protocol deploys `WithdrawPool` proxy. The constructor calls `_disableInitializers()` on the implementation; the proxy storage is uninitialized.
2. The deployer broadcasts `initialize(realClearinghouse, realVerifier)`.
3. Attacker observes the pending transaction in the mempool and broadcasts `initialize(attackerClearinghouse, attackerVerifier)` with higher gas.
4. Attacker's transaction is mined first. `__Ownable_init()` sets attacker as `owner`; `clearinghouse` and `verifier` are set to attacker-controlled contracts.
5. Deployer's transaction reverts: `Initializable: contract is already initialized`.
6. Attacker calls `removeLiquidity(quoteProductId, poolBalance, attackerAddress)` — all quote tokens are transferred out.
7. Alternatively, attacker deploys a mock `Verifier` that always returns success, then calls `submitFastWithdrawal` with a crafted payload to drain any remaining token balances. [9](#0-8) [2](#0-1)
