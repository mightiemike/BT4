Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Non-Allowlisted Addresses to Bypass the Deposit Guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual depositor transferring tokens) and gates access solely on `owner` (the LP-position recipient). Because `sender` and `owner` are distinct fields forwarded independently by the pool, any non-allowlisted address can bypass the deposit guard by specifying any allowlisted address as `owner`. The pool admin's per-depositor access control is rendered completely ineffective.

## Finding Description

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both `sender` and `owner` as separate arguments to the extension: [1](#0-0) 

The interface explicitly separates the two roles: [2](#0-1) 

However, `DepositAllowlistExtension.beforeAddLiquidity` leaves `sender` unnamed (discarded) and checks only `owner`: [3](#0-2) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual swapper) and discards the recipient: [4](#0-3) 

The admin setter `setAllowedToDeposit` stores permissions keyed by `depositor`, making the semantic intent unambiguous — the guard is meant to restrict who provides funds, not who receives the LP position: [5](#0-4) 

**Exploit path:**
1. Pool is deployed with `DepositAllowlistExtension`.
2. Admin calls `setAllowedToDeposit(pool, alice, true)` — only `alice` is allowlisted.
3. Non-allowlisted `bob` calls `pool.addLiquidity(owner=alice, ...)`.
4. Pool calls `extension.beforeAddLiquidity(sender=bob, owner=alice, ...)`.
5. The check evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
6. Bob's tokens are transferred into the pool; the LP position is minted to `alice`.
7. Bob has deposited into a restricted pool without being allowlisted.

## Impact Explanation

The deposit allowlist extension's core invariant — that only allowlisted addresses may add liquidity — is completely broken. Any non-allowlisted address can deposit tokens into a restricted pool at will, with zero privileged access required. This constitutes a broken core pool functionality (access control bypass enabling unauthorized fund flows into the pool), meeting the contest's allowed impact gate for broken core pool functionality.

## Likelihood Explanation

Exploitation requires only: (1) knowing one allowlisted address, which is publicly readable from the `allowedDepositor` mapping, and (2) calling `addLiquidity` with that address as `owner`. No privileged access, special tokens, flash loans, or complex setup is needed. The attack is repeatable by any address at any time while the pool uses this extension.

## Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositor) instead of `owner`, consistent with `SwapAllowlistExtension.beforeSwap`:

```solidity
// Fix: check sender (actual depositor), discard owner
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

```solidity
// 1. Deploy pool with DepositAllowlistExtension
// 2. Pool admin allowlists only alice
depositAllowlist.setAllowedToDeposit(pool, alice, true);

// 3. Bob (non-allowlisted) calls addLiquidity specifying alice as owner
vm.prank(bob);
pool.addLiquidity(alice /*owner*/, salt, deltas, extensionData);

// 4. Expected: revert NotAllowedToDeposit
// 5. Actual: no revert — check passes because allowedDepositor[pool][alice] == true
//    Bob's tokens enter the pool; alice receives the LP position
```
