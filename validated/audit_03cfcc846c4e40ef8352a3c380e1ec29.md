### Title
Integer Overflow in `op_unknown` Cost Multiplier Produces Zero-Cost Execution — (File: `src/more_ops.rs`)

---

### Summary

`op_unknown` in `src/more_ops.rs` computes a loop-based cost bounded by `max_cost`, then multiplies it by `cost_multiplier + 1` using plain `u64` arithmetic with no overflow guard. When the product wraps past `u64::MAX` (Rust release-mode wrapping semantics), the only post-multiplication check — `cost > u32::MAX` — can pass on the wrapped value, causing the function to return a cost of zero (or any small value) for an operation that should have been astronomically expensive. This is an undercharged-execution vulnerability with consensus impact.

---

### Finding Description

In `src/more_ops.rs`, `op_unknown` is the handler for all unknown opcodes in lenient (consensus) mode:

```
cost_function  = (op[op.len()-1] & 0b11000000) >> 6
cost_multiplier = u32_from_u8(&op[0..op.len()-1])   // attacker-controlled, ≤ u32::MAX
```

The loop (cost_function 1, 2, or 3) accumulates a `loop_cost` and calls `check_cost(…, max_cost)` on every iteration, so `loop_cost ≤ max_cost` is enforced. After the loop:

```rust
check_cost(cost, max_cost)?;          // line 260 – loop cost is within budget
cost *= cost_multiplier + 1;          // line 261 – NO overflow check
if cost > u32::MAX as u64 {           // line 262 – only guard
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`cost_multiplier + 1` can be up to `2^32` (when `cost_multiplier = u32::MAX`). If `loop_cost ≥ 2^32`, the product `loop_cost * (cost_multiplier + 1)` overflows `u64`. In Rust release builds, `u64` overflow wraps deterministically (two's complement). The wrapped value can be ≤ `u32::MAX`, bypassing the only post-multiplication guard and returning a small or zero cost.

The reserved-opcode check (`op[0] == 0xff && op[1] == 0xff`) blocks `cost_multiplier = u32::MAX` directly, but does **not** block `cost_multiplier = 2^31 − 1 = 0x7FFFFFFF` (opcode bytes `[0x7F, 0xFF, 0xFF, 0xFF, 0x40]`). With this multiplier:

- `cost_multiplier + 1 = 2^31`
- For `loop_cost = 2^33 = 8,589,934,592`: `2^33 × 2^31 = 2^64 ≡ 0 (mod 2^64)`
- `0 > u32::MAX` → **false** → returns `Ok(Reduction(0, nil))` — zero cost

`loop_cost = 2^33` is reachable with cost_function 1 (ARITH-like):

```
99 + n × 320 + total_bytes × 3 = 2^33
```

Solved by `n ≈ 26,843,545` zero-length atom arguments plus 31 total bytes. The allocator ceiling is `MAX_NUM_PAIRS = 62,500,000`, so 26.8 M pairs is within bounds. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

`op_unknown` is invoked in consensus mode (block validation) whenever `NO_UNKNOWN_OPS` is **not** set. `MEMPOOL_MODE` sets `NO_UNKNOWN_OPS`, so the mempool rejects unknown ops; block validators do not. [4](#0-3) [5](#0-4) 

A crafted CLVM program can execute an unknown opcode that the cost model intends to price at `2^64` units but which is reported as cost 0. The running total in `run_program` never exceeds `max_cost`, so the block is accepted. This breaks the cost-limit invariant that bounds how much computation a block may contain, enabling an attacker to include far more work than the block budget permits — a consensus-critical undercharged-execution bug. [6](#0-5) 

---

### Likelihood Explanation

**Attacker-controlled inputs:** The opcode bytes (setting `cost_function` and `cost_multiplier`) and the argument list are both fully attacker-controlled CLVM bytes. No privileged access is required.

**Practical constraint:** Reaching `loop_cost = 2^33` with cost_function 1 requires ~26.8 M arguments. The cost of evaluating those arguments as path lookups (~26.8 M × 44 ≈ 1.18 billion) is within Chia's ~11 billion block cost limit. However, the serialized program would be ~27–54 MB, which likely exceeds Chia's block-size limit enforced at the transaction layer. This reduces immediate exploitability but does not eliminate the vulnerability: the overflow is real, deterministic, and reachable in any execution context (simulation, testing infrastructure, future protocol changes with larger block sizes, or alternative callers of `run_program`).

---

### Recommendation

Replace the unchecked multiplication with a checked variant and treat overflow as an invalid opcode:

```rust
// Before (line 261):
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}

// After:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .filter(|&c| c <= u32::MAX as u64)
    .ok_or(EvalErr::Invalid(o))?;
```

This makes overflow indistinguishable from an oversized cost, which is already an error.

### Proof of Concept

**Opcode bytes:** `[0x7F, 0xFF, 0xFF, 0xFF, 0x40]`
- `cost_function = (0x40 & 0xC0) >> 6 = 1` (ARITH-like loop)
- `cost_multiplier = u32_from_u8([0x7F, 0xFF, 0xFF, 0xFF]) = 2,147,483,647 = 2^31 − 1`
- Not reserved: `op[0] = 0x7F ≠ 0xFF`

**Arguments:** 26,843,545 zero-length atoms + a few atoms totalling 31 bytes, giving:
```
loop_cost = 99 + 26,843,545 × 320 + 31 × 3 = 8,589,934,592 = 2^33
```

**Multiplication:**
```
cost = 2^33 × (2^31 − 1 + 1) = 2^33 × 2^31 = 2^64 ≡ 0  (mod 2^64, Rust wrapping)
```

**Guard check:** `0 > u32::MAX` → false → `Ok(Reduction(0, nil))`.

**Total program cost reported:** ~1.18 billion (argument evaluation) + 0 (unknown opcode) = ~1.18 billion, well within the 11 billion block limit. The intended cost was `2^64`. [7](#0-6) [1](#0-0)
