### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Allows Draining USDCe from Direct Deposit Addresses Without Providing USDC - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` performs a raw `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` call without checking its return value. If the USDC token returns `false` instead of reverting on failure, the function continues execution, withdrawing USDCe from the DDA and sending it to the caller â€” without the caller ever providing USDC.

---

### Finding Description

`replaceUsdcEWithUsdc` is an unrestricted `external` function (no `onlyOwner` or equivalent guard) that is intended to atomically swap USDCe held in a Direct Deposit Address (DDA) for USDC provided by the caller:

1. Read USDCe balance of the DDA.
2. Pull USDC from `msg.sender` into the DDA via `transferFrom`.
3. Withdraw USDCe from the DDA to `ContractOwner`.
4. Send USDCe to `msg.sender`.

The critical flaw is at step 2:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
``` [1](#0-0) 

The return value is silently discarded. Every other transfer in the codebase uses `ERC20Helper.safeTransferFrom` / `ERC20Helper.safeTransfer`, which wraps the call and `require`s a truthy result:

```solidity
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [2](#0-1) 

This function is the sole location in the production contracts that calls a raw `.transferFrom` without the safe wrapper. [3](#0-2) 

---

### Impact Explanation

If the USDC token at the hardcoded address (`0x2D270e6886d130D724215A266106e6832161EAEd` on chain 57073) returns `false` on a failed transfer (e.g., caller has zero allowance or zero balance) rather than reverting, the function proceeds to:

- Call `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))`, which transfers all USDCe from the DDA to `ContractOwner`.
- Call `IERC20Base(usdcE).safeTransfer(msg.sender, balance)`, which sends that USDCe to the attacker.

The attacker receives USDCe for free. The DDA is drained of its USDCe balance. No USDC is deposited in exchange. This is a direct asset theft from any DDA that holds USDCe. [4](#0-3) 

---

### Likelihood Explanation

The function is `external` with no access control beyond a `block.chainid == 57073` check, making it callable by any unprivileged address on the Ink chain. The only precondition is that a DDA exists for the target `subaccount` and holds a nonzero USDCe balance. Whether the specific USDC deployment at that address returns `false` or reverts is a token-implementation detail, but the code is structurally broken regardless â€” it is inconsistent with the rest of the codebase and violates the safe-transfer invariant the protocol otherwise enforces everywhere. [5](#0-4) 

---

### Recommendation

Replace the raw `transferFrom` call with the `ERC20Helper.safeTransferFrom` wrapper already used throughout the codebase:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ContractOwner` already imports and uses `ERC20Helper` via `using ERC20Helper for IERC20Base`, so no additional imports are needed. [6](#0-5) 

---

### Proof of Concept

1. A DDA exists for some `subaccount` with nonzero USDCe balance.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` with zero USDC allowance granted to `ContractOwner`.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, balance)` returns `false` (no revert).
4. Execution continues: `DirectDepositV1.withdraw(usdcE)` moves USDCe from DDA â†’ `ContractOwner`.
5. `safeTransfer(attacker, balance)` moves USDCe from `ContractOwner` â†’ attacker.
6. Attacker holds USDCe; DDA balance is zero; no USDC was ever deposited. [4](#0-3)

        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
