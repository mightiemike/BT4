### Title
4-Byte Secp Opcode Path Bypasses `ENABLE_SECP_OPS` Flag, Allowing Secp Verification Without Hard-Fork Activation - (File: src/chia_dialect.rs)

---

### Summary

In `src/chia_dialect.rs`, the `ChiaDialect::op()` function dispatches 4-byte secp opcodes (`0x13d61f00` for `secp256k1_verify`, `0x1c3a8f00` for `secp256r1_verify`) directly to the actual secp verification functions **without checking `ENABLE_SECP_OPS`**. The 1-byte opcode path (opcodes 64 and 65) correctly guards behind `ENABLE_SECP_OPS`, but the 4-byte path has no such guard. This is the direct analog of the external report: the wrong condition (no check at all) is used instead of the correct one (`ENABLE_SECP_OPS`), allowing a restricted operation to execute when it should be blocked.

---

### Finding Description

`ChiaDialect::op()` handles two distinct opcode encodings for secp operations:

**1-byte path (correct — guarded):** [1](#0-0) 

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

**4-byte path (wrong — unguarded):** [2](#0-1) 

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

The 4-byte opcodes `0x13d61f00` and `0x1c3a8f00` are specifically engineered to encode the same cost as the secp ops (1,300,000 and 1,850,000 respectively via the unknown-op cost formula), confirming they were designed as pre-hardfork cost placeholders. Before the hard-fork, they should fall through to `op_unknown` (returning nil with the correct cost). After the hard-fork (when `ENABLE_SECP_OPS` is set), they should dispatch to the real secp functions. The current code skips this state check entirely and always dispatches to the real secp functions.

The flag definition confirms the intended gating role: [3](#0-2) 

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
```

`MEMPOOL_MODE` deliberately omits `ENABLE_SECP_OPS`, confirming there are production execution contexts where secp ops are supposed to be unavailable: [4](#0-3) 

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

In `MEMPOOL_MODE`, the 1-byte opcode 64 falls through to `unknown_operator` and is rejected by `NO_UNKNOWN_OPS`. But the 4-byte opcode `0x13d61f00` is dispatched directly to `op_secp256k1_verify`, bypassing both `ENABLE_SECP_OPS` and `NO_UNKNOWN_OPS`. The two encodings of the same operation produce different results under the same flags — a broken invariant.

---

### Impact Explanation

An attacker-controlled CLVM program using the 4-byte opcode `0x13d61f00` or `0x1c3a8f00` can invoke real secp ECDSA signature verification in any execution context where `ENABLE_SECP_OPS` is absent, including `MEMPOOL_MODE`. This means:

1. **Pre-hardfork secp coin creation**: Coins locked by secp-based puzzles can be created and spent before the hard-fork activates, violating the activation invariant.
2. **Mempool/consensus divergence**: A program using the 4-byte secp opcode succeeds in mempool validation (where `ENABLE_SECP_OPS` is absent) while the semantically identical 1-byte opcode 64 is rejected. This inconsistency can be exploited to craft programs that pass mempool checks but behave differently under consensus rules, or vice versa.
3. **Flag rendered ineffective**: The `ENABLE_SECP_OPS` hard-fork flag provides no protection against the 4-byte opcode path, making the activation mechanism incomplete.

---

### Likelihood Explanation

The 4-byte opcode values `0x13d61f00` and `0x1c3a8f00` are derivable directly from the secp cost constants documented in the source code comments. Any attacker who reads `src/chia_dialect.rs` can construct a CLVM program using these opcodes. The entry path is fully attacker-controlled: arbitrary CLVM bytes submitted to `run_program` with a `ChiaDialect` that lacks `ENABLE_SECP_OPS`. No special privileges are required.

---

### Recommendation

Add the `ENABLE_SECP_OPS` guard to the 4-byte opcode dispatch, mirroring the 1-byte path:

```rust
// in the op_len == 4 branch:
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

This ensures that before the hard-fork (`ENABLE_SECP_OPS` absent), the 4-byte opcodes fall through to `op_unknown`, returning nil with the correct pre-committed cost — the intended pre-hardfork behavior. After the hard-fork, they dispatch to the real secp functions, consistent with the 1-byte aliases.

### Proof of Concept

```
# CLVM program: (secp256k1_verify pubkey msg sig) using 4-byte opcode 0x13d61f00
# Encoded as a 4-byte atom operator in a CLVM expression

program = (0x13d61f00 pubkey msg sig)
flags   = MEMPOOL_MODE  # NO ENABLE_SECP_OPS

# Expected: unknown op error (NO_UNKNOWN_OPS) or "secp not available"
# Actual:   op_secp256k1_verify executes; returns nil if signature valid

# Contrast: using 1-byte opcode 64 (0x40) under same flags
program2 = (0x40 pubkey msg sig)
# Actual: falls to unknown_operator -> EvalErr::Unimplemented (NO_UNKNOWN_OPS)
```

The two programs are semantically identical but produce different outcomes under `MEMPOOL_MODE`, confirming the wrong-condition bypass.
