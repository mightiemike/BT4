### Title
`ValidResourceBounds` Type Collapse in P2P Protobuf Deserialization Produces Wrong Transaction Hash - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf-to-Rust conversion for `ValidResourceBounds` silently collapses an `AllResources` transaction (with `l2_gas = 0` and `l1_data_gas = 0`) into the `L1Gas` variant. Because `get_tip_resource_bounds_hash` includes the L1DataGas element in the Poseidon hash chain only for `AllResources`, the hash computed at submission time (H1, with L1DataGas=0 appended) differs from the hash that would be computed from the stored body after P2P round-trip (H2, without L1DataGas). Any node that syncs via P2P and then serves the transaction via RPC returns a body whose hash does not match the canonical hash the signer committed to.

---

### Finding Description

**Step 1 – Submission (gateway path)**

`RpcInvokeTransactionV3` always carries `resource_bounds: AllResourceBounds`. The gateway converts it to `InternalRpcInvokeTransactionV3` (also always `AllResourceBounds`) and calls `calculate_transaction_hash`, which delegates to `get_invoke_transaction_v3_hash`.

Inside `get_invoke_transaction_v3_hash`, the hash preimage is built by calling `get_tip_resource_bounds_hash`:

```
// crates/starknet_api/src/transaction_hash.rs  lines 188-211
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // ← no L1DataGas element
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // ← L1DataGas appended
    }
});
```

For `AllResources` with `l1_data_gas = 0`, the hash chain is:
`poseidon(tip, concat(L1Gas, l1_bounds), concat(L2Gas, 0), concat(L1DataGas, 0))` → **H1**

**Step 2 – P2P propagation (protobuf path)**

When the transaction is serialised to protobuf and received by a peer, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` runs:

```
// crates/apollo_protobuf/src/converters/transaction.rs  lines 431-435
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← type collapsed
} else {
    ValidResourceBounds::AllResources(...)
})
```

Because `l2_gas = 0` and `l1_data_gas = 0`, the variant is silently changed to `L1Gas`.

**Step 3 – Hash recomputation from stored body**

If the receiving node (or any downstream consumer) recomputes the hash from the stored `InvokeTransactionV3` body, `get_tip_resource_bounds_hash` now takes the `L1Gas` branch and produces:
`poseidon(tip, concat(L1Gas, l1_bounds), concat(L2Gas, 0))` → **H2 ≠ H1**

The L1DataGas element is absent, so the Poseidon output is different.

**Step 4 – RPC serving failure**

The RPC conversion path `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` explicitly rejects `L1Gas`:

```
// crates/starknet_api/src/rpc_transaction.rs  lines 591-597
resource_bounds: match value.resource_bounds {
    ValidResourceBounds::AllResources(bounds) => bounds,
    _ => {
        return Err(StarknetApiError::OutOfRange { string: "resource_bounds".to_string() });
    }
},
```

A node that received the transaction via P2P and stored it as `L1Gas` will return an error when a client queries it via RPC, even though the transaction is valid and was accepted by the originating node.

---

### Impact Explanation

Two concrete impacts:

1. **Wrong hash served by RPC** – A client that queries the transaction from a P2P-synced node receives the canonical hash H1 (stored from submission) alongside a body that, if hashed, produces H2 ≠ H1. The RPC response is internally inconsistent: the hash field does not match the transaction body. Any client that verifies the hash (e.g., a wallet, a block explorer, or a proof verifier) will reject the transaction as invalid.

2. **RPC hard error for valid transactions** – The `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` conversion fails with `OutOfRange` for any transaction stored as `L1Gas` after P2P deserialization. The RPC returns an error instead of the transaction, making the transaction permanently unqueryable on that node.

Both match the allowed impact: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value"* and *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

---

### Likelihood Explanation

The trigger requires a user to submit an `AllResources` V3 invoke transaction with `l2_gas = 0` and `l1_data_gas = 0` (only `l1_gas > 0`). The gateway's `validate_resource_bounds` check only requires at least one non-zero bound, so `l1_gas > 0, l2_gas = 0, l1_data_gas = 0` passes validation. This is a valid, user-constructable transaction. Any node that receives it via P2P (rather than directly from the submitter) is affected.

---

### Recommendation

Remove the heuristic type-collapse in the protobuf converter. Instead of inferring the variant from zero-value fields, always produce `AllResources` when the protobuf message was sent as a V3 (post-0.13.3) transaction. A separate flag or the transaction version field should determine whether `L1Gas` or `AllResources` is the correct variant, not the runtime values of the bounds:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs
// Replace the heuristic collapse with an unconditional AllResources for V3 txs:
Ok(ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }))
```

The `L1Gas` variant should only be produced when deserialising pre-0.13.3 transactions that were originally signed without L1DataGas.

---

### Proof of Concept

1. Construct an `RpcInvokeTransactionV3` with `resource_bounds = AllResourceBounds { l1_gas: ResourceBounds { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: ResourceBounds::default(), l1_data_gas: ResourceBounds::default() }`.

2. Submit it to Node A via the gateway. Node A computes hash H1 using `get_tip_resource_bounds_hash` with the `AllResources` branch (3-element hash chain including L1DataGas=0).

3. Node A propagates the transaction to Node B via P2P protobuf. Node B's `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` sees `l2_gas.is_zero() && l1_data_gas.is_zero()` and produces `ValidResourceBounds::L1Gas(l1_gas)`.

4. Node B stores the transaction with hash H1 (from the protobuf message) but body variant `L1Gas`.

5. Query Node B's RPC for the transaction. Either:
   - The `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` conversion returns `OutOfRange` error, or
   - If served through a different path, the returned body hashes to H2 = `poseidon(tip, concat(L1Gas, l1_bounds), concat(L2Gas, 0))` ≠ H1.

6. Verify: `H1 = poseidon(tip, concat(L1Gas, l1_bounds), concat(L2Gas, 0), concat(L1DataGas, 0))` vs `H2 = poseidon(tip, concat(L1Gas, l1_bounds), concat(L2Gas, 0))`. pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

pub(crate) fn get_invoke_transaction_v3_hash<T: InvokeTransactionV3Trait>(
    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();
    let calldata_hash =
        HashChain::new().chain_iter(transaction.calldata().0.iter()).get_poseidon_hash();
    let mut hash_chain = HashChain::new()
        .chain(&INVOKE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address().0.key())
        .chain(&tip_resource_bounds_hash)
        .chain(&paymaster_data_hash)
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce().0)
        .chain(&data_availability_mode)
        .chain(&account_deployment_data_hash)
        .chain(&calldata_hash);
    if !transaction.proof_facts().0.is_empty() {
        let proof_facts_hash =
            HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
        hash_chain = hash_chain.chain(&proof_facts_hash);
    }
    Ok(TransactionHash(hash_chain.get_poseidon_hash()))
