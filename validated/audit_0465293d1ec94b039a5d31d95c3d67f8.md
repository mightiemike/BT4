### Title
IBC NFT Receiver Not Checked Against Blocked/Module Accounts — NFT Permanently Locked on Sink Chain - (`x/nft-transfer/keeper/packet.go`)

---

### Summary

`processReceivedPacket` parses the attacker-controlled `data.Receiver` field and passes it directly as the NFT `owner` to `k.nftKeeper.MintNFT` with no check against the chain's blocked/module-account set. An IBC counterparty can set `receiver` to any module account address (e.g., `distribution`, `bonded_tokens_pool`). The NFT is minted to that address and is permanently irrecoverable because module accounts have no private key.

---

### Finding Description

The full receive path is:

```
IBCModule.OnRecvPacket          (x/nft-transfer/ibc_module.go:153)
  → keeper.OnRecvPacket         (x/nft-transfer/keeper/relay.go:101)
    → processReceivedPacket     (x/nft-transfer/keeper/packet.go:140)
      → k.nftKeeper.MintNFT(…, escrowAddress, receiver)   (line 182)
        → MintNFTUnverified(…, owner=receiver)             (x/nft/keeper/keeper.go:81)
```

**Step 1 — receiver is parsed but never validated against blocked addresses.** [1](#0-0) 

`sdk.AccAddressFromBech32` succeeds for any valid bech32 address, including every module account address.

**Step 2 — `ValidateBasic` on the packet data only checks bech32 validity.** [2](#0-1) 

No module-account or blocked-address check is performed.

**Step 3 — `MintNFT` is called with `escrowAddress` as `sender` and the attacker-supplied address as `owner`.** [3](#0-2) 

Inside `MintNFT`, the only guard is `IsDenomCreator(sender)`. Because the denom was issued with `escrowAddress` as creator (line 168), this check passes unconditionally. The `owner` argument is forwarded to `MintNFTUnverified` without any further validation. [4](#0-3) 

**Step 4 — `MintNFTUnverified` writes the NFT record with the module account as owner.** [5](#0-4) 

**Step 5 — `BlockedAddrs` is wired only into the bank keeper, not the NFT keeper.** [6](#0-5) 

The `NFTKeeper` interface used by the nft-transfer module contains no `IsBlockedAddr` or `BlockedAddr` method: [7](#0-6) 

---

### Impact Explanation

The NFT is minted to a module account that has no private key. No `MsgTransferNFT`, `MsgBurnNFT`, or any other user-signed message can move it because all such messages require the signer to be the current owner. The cross-chain asset is permanently destroyed on the sink chain while the original NFT remains escrowed (or burned) on the source chain, depending on direction. This is a direct, irreversible fund-loss for the original owner.

---

### Likelihood Explanation

The attack requires only a valid IBC channel between two chains running this module. The attacker controls the `receiver` field of the outgoing `MsgTransfer` on the source chain. No special privilege, governance action, or key compromise is needed. The module account addresses are deterministic and publicly known (`sha256(module_name)[:20]`).

---

### Recommendation

In `processReceivedPacket`, before calling `MintNFT`, add a blocked-address check. The `AccountKeeper` interface should be extended (or a separate `BlockedAddr` map injected into the keeper) to reject module account receivers:

```go
if k.authKeeper.IsModuleAccount(ctx, receiver) {
    return sdkerrors.Wrapf(sdkerrors.ErrUnauthorized,
        "receiver %s is a module account and cannot receive NFTs", receiver)
}
```

Alternatively, pass `app.BlockedAddrs()` into the nft-transfer keeper and check it at receive time, mirroring the pattern already used in the bank keeper.

---

### Proof of Concept

1. On the source chain, call `MsgTransfer` with `receiver = bech32(sha256("distribution")[:20])` (the distribution module account address on the sink chain).
2. The IBC relayer delivers the packet to the sink chain.
3. `OnRecvPacket` → `processReceivedPacket` → `MintNFT` succeeds; the NFT is recorded as owned by the distribution module account.
4. Query `nft owner` on the sink chain: owner is the distribution module address.
5. Attempt `MsgTransferNFT` from any user address: fails with "not the owner".
6. The original NFT on the source chain is escrowed (or burned), and the voucher on the sink chain is permanently inaccessible.
