Audit Report

## Title
Swap Allowlist Checks Router Address Instead of End User, Enabling Full Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which equals `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router's address, not the end user's. A pool admin who allowlists the router to support legitimate router-mediated swaps inadvertently grants unrestricted swap access to every address on-chain, completely defeating the per-user allowlist.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` verbatim as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is allowlisted for the calling pool (`msg.sender` in the extension is the pool): [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` seen by the pool: [3](#0-2) 

Therefore, `sender` arriving at the extension is always the router's address, not the end user's. The allowlist lookup becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. There is no mechanism in the extension or the pool to recover the original end user's address. The `extensionData` field passed by the router is an empty string `""` for `exactInputSingle`, so no out-of-band user identity is forwarded either.

This creates an irresolvable dilemma:
- Router **not** allowlisted → all router-mediated swaps revert, blocking legitimate allowlisted users from using the router.
- Router **allowlisted** → every address on-chain can bypass the individual allowlist by routing through the router.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., KYC'd market makers, whitelisted institutions) cannot enforce that restriction for any user who routes through `MetricOmmSimpleRouter`. If the router is allowlisted — the only way to enable router-mediated swaps — any unprivileged address can execute swaps on the restricted pool. Unauthorized parties can drain LP value through unrestricted trading on a pool designed to be curated, constituting a direct policy bypass with fund-impacting consequences.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary production swap interface. Pool admins who deploy allowlisted pools and want to support standard user flows will naturally allowlist the router, not realizing this collapses the per-user gate. The bypass requires no special privileges — any EOA can call `exactInputSingle` on the router. The precondition (router allowlisted) is the expected operational state for any pool that intends to support router users.

## Recommendation

The extension must gate on the original end user, not the direct caller of `pool.swap()`. Two viable approaches:

1. **Pass the original sender through `extensionData`**: The router should encode `msg.sender` (the end user) in `extensionData`, and the extension should decode and check that value. This requires a convention between the router and the extension.
2. **Dedicated router-aware allowlist**: The extension could accept a secondary "original sender" field in `extensionData` and verify it is signed or attested by a trusted router, preventing spoofing by arbitrary callers.

## Proof of Concept

```
1. Pool admin deploys pool with SwapAllowlistExtension.
2. Pool admin calls setAllowedToSwap(pool, alice, true)  — only Alice is allowed.
3. Pool admin calls setAllowedToSwap(pool, router, true) — router allowlisted so Alice can use it.
4. Bob (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...}).
5. Router calls pool.swap(params.recipient, ...) — msg.sender in pool = router address.
6. _beforeSwap receives sender = router address.
7. allowedSwapper[pool][router] == true → check passes.
8. Bob's swap executes on the restricted pool. Bypass confirmed.
```
