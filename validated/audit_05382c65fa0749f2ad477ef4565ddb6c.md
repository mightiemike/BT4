### Title
`FeeEstimation.overall_fee` Includes L2 Gas Cost Not Reflected in Breakdown Fields, Making Documented Formula Wrong - (`File: crates/apollo_rpc_execution/src/objects.rs`)

### Summary

`tx_execution_output_to_fee_estimation` populates `FeeEstimation.overall_fee` from the actual execution receipt (which includes L1 gas + L1 data gas + L2 gas costs), but only exposes `gas_consumed` (L1 gas) and `data_gas_consumed` (L1 data gas) in the breakdown fields. The `l2_gas_consumed` field is entirely absent. The struct's own doc-comment and the OpenAPI spec both state `overall_fee = gas_consumed * gas_price + data_gas_consumed * data_gas_price`, which is wrong for any V3 transaction with non-zero L2 gas.

### Finding Description

In `tx_execution_output_to_fee_estimation`:

```rust
let gas_vector = tx_execution_output.execution_info.receipt.gas;

Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),       // L1 gas only
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(), // L1 data gas only
    l1_data_gas_price,
    l2_gas_price,                                   // price present, but no l2_gas_consumed!
    overall_fee: tx_execution_output.execution_info.receipt.fee, // includes L2 gas cost
    unit: tx_execution_output.price_unit,
})
```

`receipt.fee` is computed by `GasVector::cost()` which sums all three resources:

```rust
for (gas, price, resource) in [
    (self.l1_gas, gas_prices.l1_gas_price, Resource::L1Gas),
    (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
    (self.l2_gas, tipped_l2_gas_price, Resource::L2Gas),
] { ... }
```

So `overall_fee = l1_gas * l1_gas_price + l1_data_gas * l1_data_gas_price + l2_gas * (l2_gas_price + tip)`, but the response only exposes `gas_consumed` (L1) and `data_gas_consumed` (L1 data). The L2 gas component is silently absorbed into `overall_fee` with no corresponding breakdown field.

The OpenAPI spec reinforces the wrong formula:

```json
"overall_fee": {
    "description": "...equals to gas_consumed*gas_price + data_gas_consumed*data_gas_price"
}
```

The in-code TODO confirms this is a known gap:

```rust
// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
// close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
```

### Impact Explanation

Any caller of `starknet_estimateFee` or `starknet_simulateTransactions` for a V3 transaction with `AllResourceBounds` (L2 gas bounds) receives a response where:

- `overall_fee` is the correct actual fee (includes L2 gas cost)
- `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price` is **lower** than `overall_fee`

A client that reconstructs the fee from the breakdown fields (as the documented formula instructs) will compute a value lower than the actual fee. If that client uses the reconstructed value to set `max_fee` or resource bounds on a subsequent transaction, the transaction will be rejected or reverted for insufficient fee. The response is authoritative-looking but internally inconsistent.

This matches: **High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

### Likelihood Explanation

Every V3 transaction with non-zero L2 gas (i.e., any transaction using `AllResourceBounds`, which is the standard for Starknet v0.13+) triggers this discrepancy. Any client that follows the documented formula to verify or reconstruct `overall_fee` from the breakdown fields will observe the mismatch. The trigger is unprivileged — any user calling `starknet_estimateFee` or `starknet_simulateTransactions`.

### Recommendation

Add `l2_gas_consumed: Felt` to `FeeEstimation` and populate it from `gas_vector.l2_gas.0`:

```rust
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(),
    l1_data_gas_price,
    l2_gas_consumed: gas_vector.l2_gas.0.into(),   // add this
    l2_gas_price,
    overall_fee: tx_execution_output.execution_info.receipt.fee,
    unit: tx_execution_output.price_unit,
})
```

Update the struct doc-comment and the OpenAPI `FEE_ESTIMATE` schema description to reflect the correct formula: `overall_fee = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price + l2_gas_consumed * (l2_gas_price + tip)`.

### Proof of Concept

1. Submit `starknet_estimateFee` for a V3 invoke transaction with `AllResourceBounds` (non-zero L2 gas bounds) against any block where L2 gas price > 0.
2. Receive `FeeEstimation` response with `overall_fee`, `gas_consumed`, `l1_gas_price`, `data_gas_consumed`, `l1_data_gas_price`, `l2_gas_price`.
3. Compute `reconstructed = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`.
4. Observe `reconstructed < overall_fee` — the difference equals `l2_gas * l2_gas_price` (plus tip contribution), which is the L2 gas cost silently absorbed into `overall_fee` but absent from the breakdown.

The existing test fixture confirms real transactions carry non-zero L2 gas: `"l2_gas_consumed": "0xb56b6"` with `"l2_gas_price": "0x1dcd65000"` and `"overall_fee": "0x151eb86f3ed400"` — the L2 gas component `0xb56b6 * 0x1dcd65000 = 0x151eb86f3ed400 - (l1_gas_cost + l1_data_gas_cost)` is non-trivial. pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,

    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),

        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),

        gas_consumed: gas_vector.l1_gas.0.into(),

        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}

    /// Computes the cost (in fee token units) of the gas vector (panicking on overflow).
    pub fn cost(&self, gas_prices: &GasPriceVector, tip: Tip) -> Fee {
        let tipped_l2_gas_price =
            gas_prices.l2_gas_price.checked_add(tip.into()).unwrap_or_else(|| {
                panic!(
                    "Tip overflowed: addition of L2 gas price ({}) and tip ({}) resulted in \
                     overflow.",
                    gas_prices.l2_gas_price, tip
                )
            });

        let mut sum = Fee(0);
        for (gas, price, resource) in [
            (self.l1_gas, gas_prices.l1_gas_price, Resource::L1Gas),
            (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
            (self.l2_gas, tipped_l2_gas_price, Resource::L2Gas),
        ] {
            let cost = gas.checked_mul(price.get()).unwrap_or_else(|| {
                panic!(
                    "{resource} cost overflowed: multiplication of gas amount ({gas}) by price \
                     per unit ({price}) resulted in overflow."
                )
            });
            sum = sum.checked_add(cost).unwrap_or_else(|| {
                panic!(
                    "Total cost overflowed: addition of current sum ({sum}) and cost of \
                     {resource} ({cost}) resulted in overflow."
                )
            });
        }
        sum
    }
