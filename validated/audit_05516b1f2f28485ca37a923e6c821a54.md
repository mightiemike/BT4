### Title
Signature Malleability in `op_secp256k1_verify` Accepts High-S Signatures — (File: src/secp_ops.rs)

---

### Summary

`op_secp256k1_verify` in `src/secp_ops.rs` parses attacker-supplied signature bytes with `K1Signature::from_slice()` and verifies them with `verify_prehash()` without enforcing low-S normalization. The `k256 0.14.0-rc` crate's `verify_prehash` implementation checks only that `s ∈ [1, n-1]`, not that `s ≤ n/2`. This means both `(r, s)` and `(r, n-s)` are accepted as valid for the same message and public key — the direct analog to the `elliptic` signature malleability vulnerability (GHSA-vh7m-p724-62c2).

---

### Finding Description

In `src/secp_ops.rs`, `op_secp256k1_verify` performs the following steps:

1. Parses the 64-byte compact signature with `K1Signature::from_slice(sig.as_ref())` (line 89).
2. Verifies it with `verifier.verify_prehash(msg.as_ref(), &sig)` (line 97).

Neither step enforces low-S normalization. The `k256` crate's `VerifyingKey::verify_prehash` delegates to the `ecdsa` crate's `verify_prehashed` function, which validates only that `r, s ∈ [1, n-1]` — the standard ECDSA range check — but does not additionally require `s ≤ n/2` (the low-S constraint required by Bitcoin BIP-62 and expected by Chia's secp256k1 usage).

The `k256` crate does expose `Signature::normalize_s()` for this purpose, but it is never called in `op_secp256k1_verify`. No `normalize_s()` call appears anywhere in `src/secp_ops.rs`. [1](#0-0) 

The `Cargo.toml` confirms the dependency is `k256 = { version = "0.14.0-rc", features = ["ecdsa"] }`, which does not enable any low-S enforcement by default. [2](#0-1) 

---

### Impact Explanation

For any valid signature `(r, s)` accepted by `op_secp256k1_verify`, the signature `(r, n - s)` — where `n` is the secp256k1 group order — is also accepted for the same message and public key. An attacker who observes a valid on-chain spend bundle using a `secp256k1_verify` puzzle can trivially compute the malleable counterpart signature and broadcast a second spend bundle with a different transaction ID that is equally valid at the CLVM level.

This creates **transaction ID malleability**: two distinct spend bundles spending the same coin both pass CLVM validation. Only one can be included in a block, but off-chain systems tracking by transaction ID (wallets, explorers, payment processors, mempool monitors) can be confused or deceived. This is the same class of impact as the `elliptic` GHSA-vh7m-p724-62c2 finding: a verifier that accepts both `(r, s)` and `(r, n-s)` breaks the uniqueness invariant of signature-to-transaction binding.

**Impact: 3** — Transaction malleability with real off-chain consequences; not double-spending.

---

### Likelihood Explanation

The attacker-controlled entry path is direct: any CLVM program byte sequence that invokes `secp256k1_verify` with a valid public key, message digest, and a high-S signature reaches the vulnerable code. Valid spend bundles are publicly visible on-chain. The malleability transformation `s' = n - s` is a single modular subtraction — trivially computable with no cryptographic secrets required.

**Likelihood: 3** — Requires observing a valid spend bundle (public), computation is trivial.

---

### Recommendation

Before calling `verify_prehash`, normalize the signature to low-S:

```rust
let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| { ... })?;
// Enforce low-S normalization to prevent signature malleability
let sig = sig.normalize_s().unwrap_or(sig);
```

Alternatively, reject signatures where `s > n/2` explicitly. The same fix should be evaluated for `op_secp256r1_verify` depending on whether Chia's P-256 usage requires low-S. [3](#0-2)

The `s` component is the second 32 bytes. Compute `s' = n - s` where `n = 0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141`. Construct the malleable signature `(r || s')`. Submit `(secp256k1_verify pubkey msg (r || s'))` as a CLVM program. The operator returns `0` (success) for both the original and the malleable signature, confirming that two distinct 64-byte byte strings both verify as valid for the same `(pubkey, msg)` pair. ```rust
pub fn op_secp256k1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256K1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256k1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256k1_verify pubkey")?;
    let verifier = K1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256k1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is message
    let msg = atom(a, msg, "secp256k1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256k1_verify sig")?;
    let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
}
