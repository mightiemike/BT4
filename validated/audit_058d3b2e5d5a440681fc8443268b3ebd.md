### Title
`spotEngine` and `perpEngine` Cached as `address(0)` in `Endpoint` When Initialized Before Engines Are Registered â€” (`File: core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.initialize()` populates its `spotEngine` and `perpEngine` storage references by querying `clearinghouse.getEngineByType()` at initialization time. If engines have not yet been registered in the `Clearinghouse` (via `addEngine()`) when `Endpoint.initialize()` is called â€” which is the natural deployment order given the proxy architecture â€” both references are permanently stored as `address(0)`. No setter exists in `Endpoint` or `EndpointStorage` to update them post-initialization. Every subsequent user transaction that calls `chargeFee()` or `_getQuote()` will revert with a low-level call failure.

---

### Finding Description

`Endpoint.initialize()` resolves `spotEngine` and `perpEngine` by calling `clearinghouse.getEngineByType()`: [1](#0-0) 

`Clearinghouse.getEngineByType()` reads from the `engineByType` mapping: [2](#0-1) 

`engineByType` is **never populated in `Clearinghouse.initialize()`**. It is only set by `addEngine()`, which is `onlyOwner` and called as a post-deployment step: [3](#0-2) 

`Clearinghouse.initialize()` sets no engine references: [4](#0-3) 

With ERC1967 proxies, all contract addresses are known before any `initialize()` call. The natural deployment sequence is:

1. Initialize `Clearinghouse` (with `Endpoint` address)
2. Initialize `Endpoint` (with `Clearinghouse` address) â†’ engines not yet registered â†’ `spotEngine = address(0)`, `perpEngine = address(0)`
3. Call `clearinghouse.addEngine()` â†’ engines registered in `Clearinghouse`, but `Endpoint`'s cached references remain `address(0)` forever

`EndpointStorage` declares `spotEngine` and `perpEngine` as plain storage variables with no setter: [5](#0-4) 

`EndpointTx.chargeFee()` calls `spotEngine.updateBalance()` directly on the cached reference: [6](#0-5) 

`EndpointTx._getQuote()` calls `spotEngine.getToken()` directly: [7](#0-6) 

---

### Impact Explanation

Every user-facing transaction type that passes through `processTransactionImpl()` or `submitSlowModeTransactionImpl()` is affected:

- **`WithdrawCollateral` / `WithdrawCollateralV2`**: calls `chargeFee()` â†’ `spotEngine.updateBalance()` â†’ reverts. [8](#0-7) 
- **`LiquidateSubaccount`**: calls `chargeFee(signedTx.tx.sender, LIQUIDATION_FEE)` â†’ reverts. [9](#0-8) 
- **`MintNlp` / `BurnNlp` / `TransferQuote`**: all call `chargeFee(signedTx.tx.sender, HEALTHCHECK_FEE)` â†’ reverts. [10](#0-9) 
- **`SpotTick`**: calls `spotEngine.updateStates(dt)` â†’ reverts. [11](#0-10) 
- **`PerpTick`**: calls `perpEngine.updateStates(dt, txn.avgPriceDiffs)` â†’ reverts. [12](#0-11) 
- **Slow-mode transactions** (non-owner): call `chargeSlowModeFee(_getQuote(), sender)` â†’ `spotEngine.getToken()` â†’ reverts. [13](#0-12) 

The result is a complete protocol freeze: no withdrawals, no liquidations, no order matching ticks, no NLP operations. The corrupted state is permanent because there is no setter to update `spotEngine`/`perpEngine` in `Endpoint` post-initialization.

---

### Likelihood Explanation

**Medium.** The vulnerability is triggered by a specific but natural deployment ordering: `Endpoint.initialize()` called before `clearinghouse.addEngine()`. With ERC1967 proxies, all addresses are known upfront, so there is no technical barrier to calling `addEngine()` before `Endpoint.initialize()`. However, the absence of any guard (e.g., `require(address(spotEngine) != address(0))` in `initialize()`, or a post-initialization setter) means a deployment script that initializes `Endpoint` before registering engines will silently produce a broken system. The error only manifests at runtime when the first user transaction is submitted, not at deployment time.

---

### Recommendation

1. **Add a zero-address guard in `Endpoint.initialize()`**:
   ```solidity
   require(address(spotEngine) != address(0), "SpotEngine not registered");
   require(address(perpEngine) != address(0), "PerpEngine not registered");
   ```
   This forces the deployment to register engines in `Clearinghouse` before `Endpoint` can be initialized.

2. **Alternatively**, add an `onlyOwner` setter in `Endpoint` (analogous to the `setRewardsManager` pattern) to allow post-initialization correction:
   ```solidity
   function setEngines() external onlyOwner {
       spotEngine = ISpotEngine(clearinghouse.getEngineByType(IProductEngine.EngineType.SPOT));
       perpEngine = IPerpEngine(clearinghouse.getEngineByType(IProductEngine.EngineType.PERP));
       require(address(spotEngine) != address(0));
       require(address(perpEngine) != address(0));
   }
   ```

### Proof of Concept

1. Deploy `Clearinghouse` proxy and call `initialize(_endpoint, _quote, _clearinghouseLiq, _spreads, _withdrawPool)`. No engines registered yet.
2. Deploy `Endpoint` proxy and call `initialize(_sanctions, _sequencer, _offchainExchange, _clearinghouse, _verifier, _endpointTx)`.
   - `clearinghouse.getEngineByType(SPOT)` returns `address(0)` â†’ `spotEngine = address(0)`.
   - `clearinghouse.getEngineByType(PERP)` returns `address(0)` â†’ `perpEngine = address(0)`.
3. Call `clearinghouse.addEngine(spotEngineAddr, offchainExchange, SPOT)` and `addEngine(perpEngineAddr, offchainExchange, PERP)`. Engines are now registered in `Clearinghouse`, but `Endpoint.spotEngine` and `Endpoint.perpEngine` remain `address(0)`.
4. Any user calls `endpoint.depositCollateral(...)` â†’ sequencer submits `WithdrawCollateral` â†’ `processTransactionImpl()` â†’ `chargeFee()` â†’ `spotEngine.updateBalance()` â†’ low-level call to `address(0)` â†’ **revert: `[FAIL. Reason: Address: low-level call failed]`**.
