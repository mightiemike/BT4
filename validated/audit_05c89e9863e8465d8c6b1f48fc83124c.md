Audit Report

## Title
`DepositAllowlistExtension` Registered in `beforeRemoveLiquidity` Order Permanently Locks All LP Funds - (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension` only overrides `beforeAddLiquidity`; all other hook slots, including `beforeRemoveLiquidity`, are inherited from `BaseMetricExtension` which unconditionally reverts with `ExtensionNotImplemented()`. The factory's `ValidateExtensionsConfig` validates only structural correctness of extension orders (index bounds, no duplicates) and never verifies that a registered extension actually implements the hook it is wired to. A pool deployed with `DepositAllowlistExtension` in the `BEFORE_REMOVE_LIQUIDITY_ORDER` will have every `removeLiquidity` call revert permanently, locking all LP principal with no recovery path.

## Finding Description
`DepositAllowlistExtension` inherits from `BaseMetricExtension` and overrides only `beforeAddLiquidity`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol (L32-42)
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{ ... }
// beforeRemoveLiquidity is NOT overridden
```

`BaseMetricExtension.beforeRemoveLiquidity` is a `virtual` function that always reverts:

```solidity
// metric-periphery/contracts/extensions/base/BaseMetricExtension.sol (L63-70)
function beforeRemoveLiquidity(address, address, uint80, LiquidityDelta calldata, bytes calldata)
  external virtual onlyPool returns (bytes4)
{
  revert ExtensionNotImplemented();
}
```

`ValidateExtensionsConfig.validateExtensionsConfig` only checks structural validity — that indices are in-bounds and non-duplicate — but never calls or inspects the extension to confirm it implements the hook:

```solidity
// metric-core/contracts/libraries/ValidateExtensionsConfig.sol (L36-41)
_validateExtensionOrder(orders.beforeAddLiquidity, extensions.length);
_validateExtensionOrder(orders.afterAddLiquidity, extensions.length);
_validateExtensionOrder(orders.beforeRemoveLiquidity, extensions.length);  // no semantic check
```

The full exploit call path when `BEFORE_REMOVE_LIQUIDITY_ORDER` includes the `DepositAllowlistExtension` slot:

1. `MetricOmmPool.removeLiquidity` calls `_beforeRemoveLiquidity` (L207)
2. `_callExtensionsInOrder(BEFORE_REMOVE_LIQUIDITY_ORDER, ...)` iterates the packed order (L125-128 of `ExtensionCalling.sol`)
3. `CallExtension.callExtension(extension, data)` is invoked (L83 of `ExtensionCalling.sol`)
4. `DepositAllowlistExtension.beforeRemoveLiquidity` is called; `onlyPool` passes (pool is registered in factory)
5. `revert ExtensionNotImplemented()` is hit (L69 of `BaseMetricExtension.sol`)
6. `CallExtension.callExtension` bubbles the revert up via assembly (L11-14 of `CallExtension.sol`)
7. `removeLiquidity` reverts — permanently, for every caller

All extension orders are stored as immutables set at construction time via `ExtensionCalling`'s constructor, so there is no way to fix this after deployment:

```solidity
// metric-core/contracts/ExtensionCalling.sol (L47)
BEFORE_REMOVE_LIQUIDITY_ORDER = extensionOrders.beforeRemoveLiquidity;
```

## Impact Explanation
All LP principal deposited into the pool is permanently locked. `removeLiquidity` is the sole withdrawal path for LPs; there is no alternative recovery function. Because `BEFORE_REMOVE_LIQUIDITY_ORDER` is an immutable set at construction time, no admin action — not even the factory owner — can unblock withdrawals after deployment. This is a direct, total loss of user principal with no upper bound on the amount at risk, satisfying the "Critical/High direct loss of user principal" and "Broken core pool functionality causing loss of funds or unusable withdraw/liquidity flows" impact gates.

## Likelihood Explanation
`DepositAllowlistExtension` is the primary production allowlist extension. A pool creator intending to gate deposits would naturally register it as `extension1` and set `beforeAddLiquidity = 1`. If they also set `beforeRemoveLiquidity = 1` — believing the extension governs all liquidity operations, or by copy-paste error — the pool is bricked. The factory provides no warning, no revert, and no simulation of the hook call during pool creation. The misconfiguration is silent at deployment and only manifests when the first LP attempts to withdraw. This is reachable by any public pool creator without any privileged access.

## Recommendation
`ValidateExtensionsConfig.validateExtensionsConfig` should perform a dry-run call to each extension for each hook order it is registered in, verifying the extension returns the expected selector without reverting. Alternatively, `BaseMetricExtension` should return the correct selector (pass-through) for unimplemented hooks rather than reverting, making it safe to register in any order. The current "revert on unimplemented" default is appropriate for catching integration bugs during development but is catastrophic when it reaches a production pool with real funds.

## Proof of Concept
```solidity
// Pool created with DepositAllowlistExtension in beforeRemoveLiquidity order:
//   extensions = [address(depositAllowlistExtension)]
//   extensionOrders.beforeAddLiquidity    = 1  (extension1)
//   extensionOrders.beforeRemoveLiquidity = 1  (extension1) ← misconfiguration

// Factory accepts this — ValidateExtensionsConfig only checks index bounds.

// LP deposits successfully:
pool.addLiquidity(owner, salt, deltas, callbackData, "");  // passes: beforeAddLiquidity is implemented

// LP attempts to withdraw:
pool.removeLiquidity(owner, salt, deltas, "");
// → _beforeRemoveLiquidity → _callExtensionsInOrder(1, ...)
// → DepositAllowlistExtension.beforeRemoveLiquidity (inherited from BaseMetricExtension)
// → onlyPool passes (pool is registered)
// → revert ExtensionNotImplemented()
// → CallExtension bubbles revert
// → removeLiquidity reverts — permanently, for all LPs, forever
```
