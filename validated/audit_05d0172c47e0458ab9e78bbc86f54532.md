### Title
Permanent NFT Loss via Duplicate TokenID Race in `refundPacketToken` Sink-Direction Refund — (`x/nft-transfer/keeper/packet.go`)

### Summary

`refundPacketToken` unconditionally calls `MintNFT` for sink-direction refunds without checking whether the tokenID already exists in the voucher class. A malicious IBC counterparty can re-mint the same tokenID via a new incoming packet between the original burn and the error-ack delivery, causing the refund transaction to permanently revert and leaving the original sender with a permanently lost NFT.

### Finding Description

When a chain acts as a sink zone and a user sends a voucher NFT back toward origin, `createOutgoingPacket` calls `BurnNFTUnverified` to destroy the voucher, then commits the packet. If the counterparty later delivers an error acknowledgement (or a timeout), `refundPacketToken` is called to re-mint the voucher back to the original sender.

The refund path in the sink branch is: [1](#0-0) 

It calls `MintNFT` → `MintNFTUnverified`, which enforces uniqueness: [2](#0-1) 

There is **no pre-check** in `refundPacketToken` for whether the tokenID already exists. If the tokenID has been re-minted between the burn and the refund, `MintNFTUnverified` returns `ErrNFTAlreadyExists`, the error propagates out of `refundPacketToken`, and the entire `MsgAcknowledgement` (or `MsgTimeout`) transaction reverts. [3](#0-2) 

The re-mint of the same tokenID is possible because `processReceivedPacket` in the away-from-origin branch mints whatever tokenID the counterparty specifies: [4](#0-3) 

A malicious counterparty fully controls the tokenIDs it sends in packets and can deliberately send a new packet carrying the same tokenID after observing the burn event on-chain.

### Impact Explanation

The refund transaction reverts every time it is retried because the state (re-minted tokenID) does not change between retries. The original sender burned their voucher NFT and can never recover it. The NFT is permanently lost. This is a direct, irreversible fund loss for the user.

### Likelihood Explanation

The attack requires a malicious IBC counterparty — a chain or relayer set that controls what packets it sends. This is a realistic threat model for IBC: any chain connected via a channel can craft arbitrary packet data. The attacker only needs to:
1. Observe the burn event on the victim chain (public on-chain data).
2. Submit a new packet with the same tokenID before delivering the error ack.
3. Deliver the error ack after the re-mint is committed.

No governance, key compromise, or social engineering is required.

### Recommendation

In the sink-direction branch of `refundPacketToken`, check whether the tokenID already exists before calling `MintNFT`. If it does, the refund cannot proceed as a simple mint. Options include:

- Minting with a derived/suffixed tokenID to avoid collision, then recording the mapping.
- Transferring ownership from the current holder if the current holder is the escrow address (which would indicate a race with an in-flight packet).
- Returning a sentinel error that causes the IBC core to write a special "refund failed" state, allowing out-of-band recovery.

The minimal guard would be:

```go
if k.nftKeeper.HasNFT(ctx, voucherClassID, tokenID) {
    // handle collision: cannot blindly mint
    return sdkerrors.Wrapf(types.ErrNFTAlreadyExists,
        "refund collision: tokenID %s already exists in %s", tokenID, voucherClassID)
}
```

Combined with a protocol-level mechanism to retry with a unique tokenID or to escrow the refund claim.

func (k Keeper) refundPacketToken(ctx sdk.Context, packet channeltypes.Packet, data types.NonFungibleTokenPacketData) error {
	sender, err := sdk.AccAddressFromBech32(data.Sender)
	if err != nil {
		return err
	}

	classTrace := types.ParseClassTrace(data.ClassId)
	voucherClassID := classTrace.IBCClassID()

	isAwayFromOrigin := types.IsAwayFromOrigin(packet.GetSourcePort(),
		packet.GetSourceChannel(), data.ClassId)

	escrowAddress := types.GetEscrowAddress(packet.GetSourcePort(), packet.GetSourceChannel())

	if isAwayFromOrigin {
		// unescrow tokens back to the sender
		for _, tokenID := range data.TokenIds {
			if err := k.nftKeeper.TransferOwner(ctx, voucherClassID, tokenID, escrowAddress, sender); err != nil {
				return err
			}
		}
	} else {
		// we are sink chain, mint voucher back to sender
		for i, tokenID := range data.TokenIds {
			if err := k.nftKeeper.MintNFT(ctx, voucherClassID, tokenID, "", data.TokenUris[i], "", escrowAddress, sender); err != nil {
				return err
			}
		}
	}

	return nil
}

// MintNFTUnverified mints an NFT without verifying if the owner is the creator of denom
// Needed during genesis initialization
func (k Keeper) MintNFTUnverified(ctx sdk.Context, denomID, tokenID, tokenNm, tokenURI, tokenData string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	if k.HasNFT(ctx, denomID, tokenID) {
		return sdkerrors.Wrapf(types.ErrNFTAlreadyExists, "NFT %s already exists in collection %s", tokenID, denomID)
	}

	k.setNFT(
		ctx, denomID,
		types.NewBaseNFT(
			tokenID,
			tokenNm,
			owner,
			tokenURI,
			tokenData,
		),
	)
	k.setOwner(ctx, denomID, tokenID, owner)
	k.increaseSupply(ctx, denomID)

	return nil

func (k Keeper) OnAcknowledgementPacket(ctx sdk.Context, channelVersion string, packet channeltypes.Packet, data types.NonFungibleTokenPacketData, ack channeltypes.Acknowledgement) error {
	switch ack.Response.(type) {
	case *channeltypes.Acknowledgement_Error:
		return k.refundPacketToken(ctx, packet, data)
	default:
		// the acknowledgement succeeded on the receiving chain so nothing
		// needs to be executed and no error needs to be returned
		return nil
	}
}
