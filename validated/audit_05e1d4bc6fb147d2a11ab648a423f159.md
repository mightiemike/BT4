Audit Report

## Title
Atomic `collectFees` blocked by USDC-blacklisted `adminFeeDestination`, locking protocol fees and freezing fee-rate updates — (File: `metric-core/contracts/MetricOmmPool.sol`)

## Summary
`MetricOmmPool.collectFees` performs four sequential `safeTransfer` calls in a single atomic block; if any one reverts (e.g., USDC blacklists `adminFeeDestination`), the entire call reverts and no state is mutated. Because `MetricOmmPoolFactory.setPoolAdminFees` and `setPoolProtocolFee` unconditionally invoke `collectFees` before writing new fee rates, a blocked `collectFees` also freezes all fee-rate updates for the affected pool. Accrued protocol fees remain trapped in the pool until the pool admin rotates `adminFeeDestination` via `setPoolAdminFeeDestination`.

## Finding Description
`MetricOmmPool.collectFees` (callable only by FACTORY) executes four conditional `safeTransfer` calls inside a single `unchecked` block:

```solidity
if (totalFee0ToAdmin > 0) transferToken0(adminFeeDestination_, totalFee0ToAdmin);  // (1)
if (totalFee1ToAdmin > 0) transferToken1(adminFeeDestination_, totalFee1ToAdmin);  // (2)
if (totalFee0ToProtocol > 0) transferToken0(FACTORY, totalFee0ToProtocol);         // (3)
if (totalFee1ToProtocol > 0) transferToken1(FACTORY, totalFee1ToProtocol);         // (4)
notionalFeeToken0Scaled = 0;
notionalFeeToken1Scaled = 0;
``` [1](#0-0) 

`transferToken0` and `transferToken1` both use `IERC20.safeTransfer`, which reverts on failure. [2](#0-1) 

If USDC blacklists `adminFeeDestination_`, transfer (1) reverts, the entire call reverts, and `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled` are not zeroed. All three factory entry points route through this same call:

- `collectPoolFees` (open to any caller): [3](#0-2) 
- `setPoolAdminFees` (pool admin): [4](#0-3) 
- `setPoolProtocolFee` (factory owner): [5](#0-4) 

The only recovery path is `setPoolAdminFeeDestination`, which does not call `collectFees` and can rotate the destination to a non-blacklisted address. [6](#0-5) 

## Impact Explanation
Protocol fees (both notional, tracked in `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled`, and spread, computed inline from bin surplus) cannot be distributed to the FACTORY or admin destination for the duration of the blockage. Fee-rate updates via `setPoolAdminFees` and `setPoolProtocolFee` are also frozen. Fees are not permanently lost — they remain in the pool and are recoverable once the pool admin rotates `adminFeeDestination` — but the DoS on `collectPoolFees` (open to any caller) and on fee-rate management constitutes a medium-severity broken protocol-fee-collection flow. USDC/USDT blacklists are explicitly in scope per contest rules.

## Likelihood Explanation
USDC and USDT both implement address blacklists. `adminFeeDestination` is set by the pool admin to any address; an address valid at pool creation may later be blacklisted by regulatory action or key compromise. No privileged escalation is required: the trigger is a standard USDC blacklist event on a valid, non-zero `adminFeeDestination`. The pool admin can resolve the blockage by calling `setPoolAdminFeeDestination` with a clean address, but an unresponsive or adversarial pool admin can sustain the DoS indefinitely.

## Recommendation
Isolate each transfer leg so a failure in one does not block the others:

1. **Wrap each `safeTransfer` in a `try/catch`** and emit a failure event, allowing partial collection and retry of the failed leg.
2. **Separate the fee-collection call per token and per recipient**, adding `token` and `recipient` parameters so callers can collect one leg at a time with independent accounting.

Additionally, track spread-fee amounts in persistent storage per token so a successful token1 collection can zero its counter independently of a failed token0 leg.

## Proof of Concept
1. Deploy pool with `token0 = USDC`, `token1 = WETH`.
2. Pool admin sets `adminFeeDestination` to address `A` (valid at the time).
3. USDC blacklists address `A`.
4. Swaps accrue spread and notional fees.
5. Call `MetricOmmPoolFactory.collectPoolFees(pool)` → factory calls `pool.collectFees(...)` → `transferToken0(A, ...)` → USDC `safeTransfer` reverts → entire call reverts; `notionalFeeToken0Scaled` unchanged.
6. Factory owner calls `setPoolProtocolFee(pool, newFee, newNotional)` → same `collectFees` call → same revert → fee-rate update blocked.
7. Pool admin calls `setPoolAdminFees(pool, ...)` → same revert.
8. Fees accumulate but cannot be distributed; fee-rate management is frozen until pool admin calls `setPoolAdminFeeDestination(pool, cleanAddress)`.
