### Title
Secp256k1/r1 Verify Operators Reachable via 4-Byte Opcode Form Regardless of `ENABLE_SECP_OPS` Flag — (`src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` contains two separate dispatch branches for the secp256k1 and secp256r1 verify operators. The 1-byte opcode branch (opcodes 64 and 65) correctly gates execution behind `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte opcode branch (`0x13d61f00` / `0x1c3a8f00`) calls the same underlying functions **without any flag check**, bypassing the hard-fork activation guard entirely. An attacker who crafts CLVM bytes using the 4-byte opcode encoding can invoke secp verification in any execution context — including `MEMPOOL_MODE` — regardless of whether `ENABLE_SECP_OPS` is set.

---

### Finding Description

In `ChiaDialect::op()`, the 4-byte opcode branch runs first:

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,
        0x1c3a8f00 => op_secp256r1_verify,
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
``` [1](#0-0) 

The 1-byte opcode branch, reached only when `op_len == 1`, correctly gates the same operators:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The flag is defined as a hard-fork activation guard:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
``` [3](#0-2) 

`MEMPOOL_MODE` deliberately omits `ENABLE_SECP_OPS`, meaning secp ops are supposed to be unavailable in mempool validation:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [4](#0-3) 

The two paths share the same settlement point (`op_secp256k1_verify` / `op_secp256r1_verify`) but only one path sets up the required precondition (the flag check). This is structurally identical to the reported LienToken bug: two entry paths into a shared execution endpoint, one of which omits the critical state/guard setup.

---

### Impact Explanation

**Consensus divergence and hard-fork bypass.** Before the secp hard fork activates, all callers are expected to run without `ENABLE_SECP_OPS`. Via the 1-byte opcode path, opcodes 64/65 fall through to `unknown_operator` (a no-op with cost in lenient mode, or a hard rejection in `MEMPOOL_MODE` due to `NO_UNKNOWN_OPS`). Via the 4-byte path, the same secp verification executes unconditionally.

Concrete consequences:

1. **Pre-fork consensus mode** (`ChiaDialect::new(ClvmFlags::empty())`): A program using `0x13d61f00` executes real secp256k1 signature verification instead of being treated as an unknown no-op. A program that should be a no-op becomes a signature verifier, producing a different execution result and cost than the reference behavior.

2. **Mempool mode** (`MEMPOOL_MODE`): `NO_UNKNOWN_OPS` is set, so 1-byte opcodes 64/65 are rejected. But the 4-byte form bypasses `unknown_operator` entirely and executes secp verification. A program that should be rejected by the mempool is instead accepted and executed.

3. **Signature-check bypass framing**: If a coin puzzle uses `(secp256k1_verify pubkey msg sig)` via the 4-byte opcode form, it will execute and succeed (returning nil) or fail (raising an error) — but it will do so in contexts where the operator was supposed to be unavailable. This breaks the invariant that secp-gated puzzles cannot be spent before the hard fork.

The corrupted result is: the `Response` returned by `run_program` for a program containing `0x13d61f00`/`0x1c3a8f00` is `Ok(Reduction(1300000, nil))` or `Err(Secp256Failed)` instead of `Ok(Reduction(unknown_op_cost, nil))` (lenient) or `Err(Unimplemented)` (strict/mempool).

---

### Likelihood Explanation

**Medium.** The 4-byte opcode encoding is documented in the source code comments and is a well-specified part of the CLVM unknown-op cost formula. Any researcher or attacker reading `chia_dialect.rs` can identify the exact byte sequences `\x13\xd6\x1f\x00` and `\x1c\x3a\x8f\x00`. Crafting a CLVM program that uses these opcodes requires only serializing a 4-byte atom as the operator in a CLVM pair. The entry path is fully attacker-controlled: `run_serialized_chia_program` in `wheel/src/api.rs` accepts arbitrary `program: &[u8]` bytes from Python callers. [5](#0-4) 

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode branch, mirroring the 1-byte branch:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

When `ENABLE_SECP_OPS` is not set, the 4-byte opcodes should fall through to `unknown_operator`, producing the same behavior as any other unrecognized 4-byte opcode.

### Proof of Concept

```python
from clvm_rs import run_serialized_chia_program, MEMPOOL_MODE

# Serialize: (0x13d61f00 pubkey msg sig)
# 4-byte opcode atom: ff 04 13 d6 1f 00 ...
# This invokes op_secp256k1_verify via the 4-byte branch,
# bypassing ENABLE_SECP_OPS (0x0800) which is absent from MEMPOOL_MODE.

# Without ENABLE_SECP_OPS, opcode 64 (1-byte) would be rejected by NO_UNKNOWN_OPS.
# With the 4-byte form, secp256k1_verify executes unconditionally.

flags_no_secp = MEMPOOL_MODE  # does not include ENABLE_SECP_OPS

# A program encoding (0x13d61f00 <pubkey> <msg> <sig>) in CLVM serialization
# will reach op_secp256k1_verify and return Ok or Err(Secp256Failed),
# not Err(Unimplemented) as MEMPOOL_MODE + NO_UNKNOWN_OPS requires.
cost, result = run_serialized_chia_program(
    secp_program_bytes,   # uses 4-byte opcode 0x13d61f00
    args_bytes,
    max_cost=10_000_000,
    flags=flags_no_secp,
)
# Expected (correct): raises ValueError("unknown op") due to NO_UNKNOWN_OPS
# Actual (buggy):     executes secp256k1_verify, returns cost=1300000
```

pub fn run_serialized_chia_program(
    py: Python,
    program: &[u8],
    args: &[u8],
    max_cost: Cost,
    flags: u32,
) -> PyResult<(u64, LazyNode)> {
    let flags = ClvmFlags::from_bits_truncate(flags);
    let mut allocator = if flags.contains(ClvmFlags::LIMIT_HEAP) {
        Allocator::new_limited(500000000)
    } else {
        Allocator::new()
    };

    let r: Response = (|| -> PyResult<Response> {
        let program = node_from_bytes(&mut allocator, program).map_err(eval_to_py)?;
        let args = node_from_bytes(&mut allocator, args).map_err(eval_to_py)?;
        let dialect = ChiaDialect::new(flags);

        Ok(py.detach(|| run_program(&mut allocator, &dialect, program, args, max_cost)))
    })()?;
    adapt_response(py, allocator, r)
}
