Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual trader — allowlist fully bypassed for any router user - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the address the pool passes as the swap initiator. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so `sender` received by the extension is the router address — not the actual end-user. Any trader who calls the router bypasses the per-user allowlist entirely, because the router (not the individual trader) is the address being checked.

## Finding Description
**Call path:**

1. Trader calls `MetricOmmSimpleRouter.exactInputSingle(params)` (line 67). The router stores the actual user in transient storage via `_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn)` (line 71) — but this is only used for the payment callback, not for the extension check.

2. The router then calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` (line 72–80). At this point `msg.sender` inside the pool is the **router address**, not the trader.

3. The pool internally calls `_beforeSwap(msg.sender /*= router*/, recipient, ...)` via `ExtensionCalling._beforeSwap` (line 149–177 of `ExtensionCalling.sol`). The `sender` argument forwarded to the extension is therefore the router address.

4. `SwapAllowlistExtension.beforeSwap` (line 37) evaluates:
   ```solidity
   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
       revert NotAllowedToSwap();
   }
   ```
   Here `msg.sender` = pool (correct), `sender` = **router** (wrong actor). The check passes whenever the router is allowlisted, regardless of who the actual trader is.

**Root cause:** The extension receives the pool's direct caller (the router) as `sender`, but the allowlist is semantically intended to gate individual traders. The actual trader's address is never forwarded to or checked by the extension.

**Why existing guards fail:** The only guard is the `allowedSwapper` mapping keyed on `sender`. Because `sender` is always the router for router-mediated swaps, allowlisting the router (which is required for any router-based swap to work) opens the gate for every user of that router.

## Impact Explanation
Any unprivileged trader can execute swaps in a pool that the pool admin intended to restrict to a specific set of allowlisted addresses. The allowlist — a core access-control mechanism of the extension system — is rendered completely ineffective for all router-mediated swaps. This constitutes broken core pool functionality: the pool's configured access policy is silently bypassed, and unauthorized traders can interact with the pool's liquidity, potentially causing adverse-selection losses to LPs in restricted pools.

## Likelihood Explanation
Exploitability requires only that: (a) a pool is configured with `SwapAllowlistExtension`, and (b) the router is allowlisted (which is the normal operational requirement for the router to function at all). Any unprivileged trader can then call `exactInputSingle` or `exactInput` through the router. No special privileges, flash loans, or complex setup are needed. The condition is met in every standard production deployment that uses the router with this extension.

## Recommendation
The pool must forward the original end-user address as `sender` to extensions, not its own `msg.sender`. One approach: require the caller to supply a `sender` address explicitly in the swap call (similar to how Uniswap v4 passes `msg.sender` through the `PoolManager`), or have the router pass the actual user address through `extensionData` and have the extension decode it — with the pool authenticating that the router-supplied sender is trustworthy. Alternatively, `SwapAllowlistExtension` should not rely on `sender` alone when `sender` is a known router; it should require the router to attest the real user via a signed or callback-verified mechanism.
