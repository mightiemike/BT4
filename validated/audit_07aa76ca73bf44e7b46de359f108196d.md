Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing full allowlist bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension` is intended to gate `addLiquidity` by the depositing caller's address. However, its `beforeAddLiquidity` hook silently discards the `sender` parameter and evaluates the allowlist against `owner` — an arbitrary address supplied by any caller. Because `MetricOmmPool.addLiquidity` accepts an arbitrary `owner` from any `msg.sender`, any unprivileged address can bypass the allowlist by supplying an already-allowed address as `owner`. Additionally, the override drops the `onlyPool` modifier present in `BaseMetricExtension`, allowing the hook to be called directly by any external address.

## Finding Description

`MetricOmmPool.addLiquidity` is public and passes `msg.sender` as `sender` and the caller-controlled `owner` argument to the extension hook: [1](#0-0) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first parameter (`sender`) is unnamed and discarded; the guard is evaluated only against `owner`: [2](#0-1) 

The storage mapping is named `allowedDepositor` and keyed by `depositor`, confirming the intent is to restrict the calling depositor, not the LP recipient: [3](#0-2) 

By contrast, the analogous `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual caller) and discards `recipient`: [4](#0-3) 

`BaseMetricExtension.beforeAddLiquidity` carries the `onlyPool` modifier, but `DepositAllowlistExtension` drops it in its override, leaving the hook callable by any external address: [5](#0-4) 

## Impact Explanation

Any unprivileged address can add liquidity to a pool the admin intended to restrict to a whitelist of depositors. The attacker supplies an already-allowed address as `owner`, the allowlist check passes, the attacker pays tokens via the callback, and the allowed address receives an unsolicited LP position. The pool admin's access-control boundary is fully bypassed: unauthorized parties can freely manipulate pool liquidity depth, alter bin state, and violate any compliance or risk-management intent behind the allowlist. This constitutes a broken admin-boundary / broken core pool functionality with direct fund-impacting consequences.

## Likelihood Explanation

`addLiquidity` is public with no other caller restriction. Any address that knows a single allowed `owner` address — discoverable on-chain from `AllowedToDepositSet` events — can execute the bypass in a single transaction. No special privilege, flash loan, or oracle manipulation is required. The attack is repeatable indefinitely.

## Recommendation

Replace the `owner` check with a `sender` check in `beforeAddLiquidity`, mirroring `SwapAllowlistExtension`, and restore the `onlyPool` modifier:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    onlyPool
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][Alice] = true
  Bob is NOT in the allowlist

Attack:
  Bob calls pool.addLiquidity(
      owner        = Alice,   // allowed address used as bypass key
      salt         = 0,
      deltas       = <any valid delta>,
      callbackData = <Bob's callback pays his own tokens>,
      extensionData = ""
  )

Trace:
  MetricOmmPool.addLiquidity:
    _beforeAddLiquidity(sender=Bob, owner=Alice, ...)
    → extension.beforeAddLiquidity(Bob /*unnamed, ignored*/, Alice, ...)
    → allowedDepositor[pool][Alice] == true  → guard passes
    → LiquidityLib.addLiquidity credits shares to (Alice, 0)
    → callback pulls tokens from Bob
    → Alice holds LP position she did not initiate
    → Bob (unauthorized) has successfully added liquidity to a restricted pool
```
