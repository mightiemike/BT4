### Title
Global `lastLiquidationFees` Stale-State Causes Incorrect `canLiquidateMore` Computation Across Subaccounts, Enabling Premature Socialization — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`lastLiquidationFees` is a single global storage variable shared across all subaccounts. When `_finalizeSubaccount` is called for subaccount B, it subtracts the fees recorded from the most recent liquidation step — which may have been for a completely different subaccount A. This causes `v.canLiquidateMore` to be computed as `false` even when the insurance fund is genuinely sufficient to cover subaccount B's deficit, bypassing the spot-balance-zero check and triggering premature socialization that spreads bad debt to all open-interest holders.

---

### Finding Description

`ClearinghouseStorage` declares `lastLiquidationFees` as a single protocol-wide variable: [1](#0-0) 

In `_handleLiquidationPayment`, after every non-finalization liquidation step, this variable is overwritten with the fees from that step: [2](#0-1) 

The design intent (per the inline comment at lines 582–586) is to prevent the fees just added to `insurance` from being immediately re-used to cover the same subaccount's deficit in the subsequent finalization call. However, the variable is **never reset between subaccounts**. It persists in storage across all liquidation transactions.

In `_finalizeSubaccount`, the effective insurance available is computed as: [3](#0-2) 

If the last `_handleLiquidationPayment` call was for subaccount A (not B), then `lastLiquidationFees = X` reflects A's fees, not B's. The subtraction `insurance - X` gives the insurance level *before* A's fees were added — which is incorrect for B's finalization. The full current `insurance` (including A's fees) should be available for B.

The `canLiquidateMore` flag gates the spot-balance-zero check: [4](#0-3) 

When `canLiquidateMore` is incorrectly `false`, this check is skipped. Subaccount B can be finalized while still holding non-zero spot liabilities (negative spot balances). The unconditional check at lines 300–311 only requires `balance.amount <= 0`, so negative balances pass it. [5](#0-4) 

With `v.insurance <= 0` (due to the stale subtraction), the insurance coverage branch is skipped and `spotEngine.socializeSubaccount` is called: [6](#0-5) 

The global `insurance` is restored at the end via `v.insurance += lastLiquidationFees`, but the socialization has already occurred — the damage is irreversible. [7](#0-6) 

The entry path is fully reachable: `Endpoint.submitTransactions` → `Clearinghouse.liquidateSubaccount` → `delegatecall` to `ClearinghouseLiq.liquidateSubaccountImpl`: [8](#0-7) 

---

### Impact Explanation

**Broken invariant:** Insurance funds that are genuinely available to cover subaccount B's spot liabilities are not used. Instead, those liabilities are socialized to all open-interest holders, destroying value for them and violating the protocol's solvency guarantee that insurance is exhausted before socialization.

**Concrete asset delta:**
- Insurance fund retains `X` (A's fees) that should have been used to cover B's deficit.
- Open-interest holders absorb bad debt equal to the amount that insurance could have covered.
- The spot-balance-zero check is bypassed, allowing finalization of a subaccount that still holds spot liabilities — a state that should be unreachable.

---

### Likelihood Explanation

The attack requires two sequential liquidation transactions for different subaccounts where:
1. Subaccount A is liquidated (non-finalization), setting `lastLiquidationFees = X`.
2. Subaccount B is finalized, where `insurance > 0` but `insurance - X <= 0` (i.e., A's fees boosted insurance above B's deficit threshold, but the pre-boost level was below it).

This is a realistic market stress scenario: the insurance fund is low, multiple subaccounts become insolvent simultaneously, and sequential liquidations naturally produce this state. No special privileges are required — any liquidator can submit both transactions. They can even be in the same `submitTransactions` batch, since `lastLiquidationFees` persists in storage between calls within the same batch.

---

### Recommendation

`lastLiquidationFees` must be scoped to the subaccount being finalized, not stored globally. The simplest fix is to reset `lastLiquidationFees` to `0` at the start of `_finalizeSubaccount` (since the finalization path does not go through `_handleLiquidationPayment` for the same subaccount in the same call), or to pass the fees as a local parameter rather than reading from global storage. Alternatively, track `lastLiquidationFees` as a per-subaccount mapping keyed by `txn.liquidatee`.

---

### Proof of Concept

**State setup:**
- `insurance = 50` (global)
- Subaccount A: under-maintenance, has a spot liability
- Subaccount B: insolvent, has spot liabilities, `quoteBalance.amount = -80`

**Step 1:** Liquidator submits a non-finalization liquidation for subaccount A.
- `_handleLiquidationPayment` runs: `insurance += 60` → `insurance = 110`, `lastLiquidationFees = 60`.

**Step 2:** Liquidator submits a finalization (`productId == type(uint32).max`) for subaccount B.
- `_finalizeSubaccount` runs:
  - `v.insurance = 110 - 60 = 50`
  - `v.canLiquidateMore = (-80 + 50) > 0` → `(-30) > 0` → **false**
  - Spot-balance-zero check at lines 372–383 is **skipped**
  - `insuranceCover = min(50, 80) = 50` — **but this branch is not reached** because `v.insurance = 50 > 0`... 

Wait — let me re-examine. `v.insurance = 50 > 0`, so `v.insurance <= 0` is false, meaning `spotEngine.socializeSubaccount` is NOT called in this specific numeric example. Let me adjust:

**Adjusted Step 1:** `insurance = 50`, A's fees = `60` → `insurance = 110`, `lastLiquidationFees = 60`.

**Step 2 (finalization of B):** `quoteBalance.amount = -80`
- `v.insurance = 110 - 60 = 50`
- `v.canLiquidateMore = (-80 + 50) > 0` → false → **spot-balance-zero check skipped** ✓
- `insuranceCover = min(50, 80) = 50` → `v.insurance = 0`, quote balance covered partially
- `v.insurance <= 0` → `spotEngine.socializeSubaccount(B)` called for remaining `-30` ✓
- `v.insurance += 60` → `insurance = 60` restored

**The harm:** Subaccount B had spot liabilities that were not zeroed before finalization (the check was skipped). The remaining `-30` quote deficit is socialized. With the correct `v.insurance = 110`, `insuranceCover = min(110, 80) = 80` would have fully covered the deficit with no socialization needed. The `60` in insurance that belonged to A's fees was available but not applied to B's coverage due to the stale subtraction.

**State-machine test assertion:**
```solidity
// After step 1: assert lastLiquidationFees == 60, insurance == 110
// After step 2: assert spotEngine.socializeSubaccount was called for B
//               assert insurance == 60 (not 30, confirming 30 was incorrectly socialized)
//               assert spot liabilities of B were non-zero at time of finalization
```
