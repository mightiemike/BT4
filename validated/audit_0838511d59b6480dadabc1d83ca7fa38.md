### Title
Difficulty Calculation for Fork Blocks Reads Stale Mainchain State Instead of Fork Ancestor — (`contract/src/dogecoin.rs`)

### Summary

When validating the difficulty of a Dogecoin fork block, `get_next_work_required` calls `blocks_getter.get_header_by_height(height_first)` to obtain the boundary block's timestamp for the Digishield retarget window. `get_header_by_height` unconditionally reads from the `mainchain_height_to_header` index. For any fork block at depth ≥ 3 (height H+2 or beyond), `height_first` falls inside the fork itself, so the function returns the **mainchain** block at that height rather than the fork's actual ancestor. The resulting timespan is computed against the wrong block, producing an incorrect expected difficulty. The code even contains an inline acknowledgement of this: `// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor`.

### Finding Description

**Root cause — stale state read before fork state is reflected**

`get_header_by_height` is implemented as: [1](#0-0) 

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

It always resolves height → hash via `mainchain_height_to_header`, which only contains **mainchain** blocks. Fork blocks are stored only in `headers_pool` (via `store_fork_header`), never in `mainchain_height_to_header`. [2](#0-1) 

The Dogecoin Digishield retarget path calls this function: [3](#0-2) 

For Digishield (post-block 145 000), `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` and `height_first = prev_block_header.block_height - 1`.

**Concrete trigger sequence:**

Let the common ancestor be at height H−1. The fork diverges at height H.

| Fork block submitted | `prev_block_header` height | `height_first` | Mainchain block at `height_first` == fork ancestor? |
|---|---|---|---|
| H (1st fork block) | H−1 (common ancestor) | H−2 | ✅ same block |
| H+1 (2nd fork block) | H (fork) | H−1 (common ancestor) | ✅ same block |
| **H+2 (3rd fork block)** | **H+1 (fork)** | **H (fork block)** | **❌ mainchain block at H ≠ fork block at H** |

From the 3rd fork block onward, `get_header_by_height(H)` returns the **mainchain** block at height H, whose timestamp differs from the fork block at height H. The timespan used in `calculate_next_work_required` is therefore:

```
modulated_timespan = fork_block_H+1.time − mainchain_block_H.time   ← WRONG
```

instead of:

```
modulated_timespan = fork_block_H+1.time − fork_block_H.time         ← CORRECT
```

The same structural defect exists in `bitcoin.rs` and `litecoin.rs`: [4](#0-3) [5](#0-4) 

For Bitcoin/Litecoin the interval is 2016 blocks, making the fork depth required impractical. For Dogecoin the interval is 1 block, so the bug is triggered by any fork ≥ 3 blocks deep — a routine occurrence.

### Impact Explanation

The incorrect `first_block_time` shifts the computed `modulated_timespan`. Digishield clamps the timespan to `[retarget_timespan − retarget_timespan/4, retarget_timespan + retarget_timespan/2]`, i.e., `[45 s, 90 s]` for Dogecoin mainnet. If the mainchain block at H has an **earlier** timestamp than the fork block at H, the computed timespan is inflated, the expected difficulty target is relaxed, and the contract accepts fork blocks that carry less proof-of-work than the protocol requires. An attacker who can choose which mainchain block to fork from (to maximise the timestamp gap) can systematically mine fork blocks at reduced difficulty, accumulate chainwork faster than expected, and trigger a chain reorganisation. This corrupts the canonical chain pointer (`mainchain_tip_blockhash`) and invalidates all SPV proofs issued against the displaced mainchain blocks.

### Likelihood Explanation

Any Dogecoin fork submission of depth ≥ 3 triggers the bug — no special privilege is required. A relayer (or any NEAR caller with `UnrestrictedSubmitBlocks`) can submit fork headers via `submit_blocks`. The Dogecoin network produces forks of this depth regularly. An adversarial relayer can deliberately construct a fork starting from a mainchain block whose timestamp is significantly earlier than the fork block at the same height, maximising the difficulty reduction. The code comment at the exact call site (`// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor`) confirms the developers were already aware of this ambiguity. [6](#0-5) 

### Recommendation

Replace the `get_header_by_height` call with an ancestor walk that follows `prev_block_hash` links through `headers_pool`, starting from `prev_block_header`, until the target height is reached. This mirrors how `zcash.rs` correctly traverses the chain via `get_prev_header` rather than the mainchain height index: [7](#0-6) 

A helper such as `get_ancestor_at_height(start: &ExtendedHeader, target_height: u64) -> ExtendedHeader` that walks `get_prev_header` links should replace every `get_header_by_height` call inside difficulty-calculation paths.

### Proof of Concept

1. Deploy the Dogecoin build of the contract and initialise it with a genesis at height 145 001 (Digishield active).
2. Submit mainchain blocks up to height H, where the mainchain block at H has timestamp `T_main`.
3. Submit a fork block at height H with timestamp `T_fork_H > T_main` (fork diverges from the block at H−1).
4. Submit a fork block at height H+1 with timestamp `T_fork_H1`.
5. Submit a fork block at height H+2. The contract calls `get_header_by_height(H)`, which returns the **mainchain** block with timestamp `T_main`. The computed timespan is `T_fork_H1 − T_main`. If `T_main` is sufficiently earlier than `T_fork_H`, the timespan exceeds the correct value (`T_fork_H1 − T_fork_H`), the expected bits are relaxed, and the contract accepts a block whose actual difficulty is lower than the protocol mandates.
6. Repeat for subsequent fork blocks; each uses the wrong mainchain timestamp, compounding the difficulty distortion.
7. Once the fork's `chain_work` exceeds the mainchain's, `reorg_chain` is triggered, replacing the canonical chain with the attacker-controlled fork.
