Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of End-User, Allowing Full Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender` — the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. A pool admin who allowlists the router address to enable router-mediated swaps for their allowlisted users inadvertently opens the pool to every user, completely bypassing the per-user allowlist.

## Finding Description

**Root cause — `MetricOmmPool.swap` passes `msg.sender` as `sender`:**

`MetricOmmPool.swap` (L230–231) calls `_beforeSwap(msg.sender, ...)`. When `MetricOmmSimpleRouter.exactInputSingle` is the caller, `msg.sender` inside the pool is the router contract address, not the end user.

**Extension receives the router address as `sender`:**

`SwapAllowlistExtension.beforeSwap` (L37) evaluates:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()` — the router. The check becomes `allowedSwapper[pool][router]`.

**Router calls `pool.swap()` directly without forwarding the real user:**

`MetricOmmSimpleRouter.exactInputSingle` (L72–80) calls `IMetricOmmPoolActions(params.pool).swap(...)` with no mechanism to pass the original `msg.sender` (the end user) into `extensionData` or any other field visible to the extension.

**Exploit flow:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to KYC'd addresses.
2. Admin calls `setAllowedToSwap(pool, router, true)` — intending to allow router-mediated swaps for their allowlisted users.
3. Attacker (not in the per-user allowlist) calls `router.exactInputSingle({pool: pool, ..., recipient: attacker})`.
4. Pool calls `_beforeSwap(msg.sender=router, ...)`.
5. Extension checks `allowedSwapper[pool][router] == true` → passes.
6. Attacker receives output tokens; the per-user allowlist was never consulted.

**Existing guards are insufficient:** The extension has no mechanism to distinguish between the router calling on behalf of an allowlisted user versus an unauthorized user. The `extensionData` field is passed through but the extension does not decode it. All existing tests in `FullMetricExtension.t.sol` exercise only direct pool calls (`callers[i]` → `pool.swap`), never the router path, so the misbinding is untested.

The same bypass applies to `exactInput` (L99–112) and `exactOutputSingle` (L136–137) multi-hop paths, where the router is always the direct caller of `pool.swap()`.

## Impact Explanation

A curated pool that restricts swaps to a specific set of counterparties (e.g., KYC'd addresses, protocol-owned bots, or whitelisted market makers) loses that restriction entirely for any user who routes through the public `MetricOmmSimpleRouter`. Unauthorized users can trade in pools designed to be closed to the public, draining LP value at oracle-quoted prices and extracting spread fees. This is a direct loss of LP principal and a broken core pool invariant (access control). Severity: High — direct loss of user principal with no additional preconditions beyond the admin's natural and expected action of allowlisting the router.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary public swap entrypoint. Any pool admin who deploys a `SwapAllowlistExtension`-gated pool and also wants to support router-mediated swaps for their allowlisted users will naturally add the router to the allowlist. The bypass is then immediately reachable by any unprivileged user with no special setup. The trigger is a single standard `exactInputSingle` call. The Smart Audit Pivot for the allowlist path explicitly flags: "cannot be bypassed through router."

## Recommendation

The `sender` forwarded to extensions should represent the economic actor (the end user), not the intermediary contract. Two complementary fixes:

1. **In the router**: encode the original `msg.sender` (the end user) into `extensionData` before calling `pool.swap()`, so extensions can decode and verify it.
2. **In `SwapAllowlistExtension`**: decode a user address from `extensionData` (with a trusted-forwarder pattern or signature verification) and use that for the allowlist check, rather than relying on the raw `sender` argument.

The simplest safe fix is to have the router encode the real user address into `extensionData` and have the extension decode and verify it, rather than relying on the raw `sender` argument.

## Proof of Concept

```
1. Deploy a pool with SwapAllowlistExtension configured.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   — intending to allow router-mediated swaps for their allowlisted users.
3. Attacker (not in the per-user allowlist) calls:
     router.exactInputSingle({pool: pool, ..., recipient: attacker})
4. Router calls pool.swap(...) — pool's msg.sender = router.
5. Pool calls _beforeSwap(msg.sender=router, ...).
6. Extension checks allowedSwapper[pool][router] == true → passes.
7. Attacker receives output tokens; the per-user allowlist was never consulted.
```

Foundry test plan: fork `FullMetricExtensionTest`, deploy `MetricOmmSimpleRouter`, call `setAllowedToSwap(pool, router, true)`, then call `router.exactInputSingle` from an address not in the allowlist and assert the swap succeeds (demonstrating the bypass).
