Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual User, Enabling Full Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument passed by the pool, which is `msg.sender` at the pool level — the router contract, not the end user, when a swap is routed through `MetricOmmSimpleRouter`. A pool admin who allowlists the router to enable router-mediated swaps on a permissioned pool inadvertently grants every address on the network the ability to swap, completely defeating the allowlist.

## Finding Description
In `MetricOmmPool.swap`, `_beforeSwap` is invoked with `msg.sender` as the `sender` argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that `sender` value verbatim to every registered extension via `abi.encodeCall`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then gates on `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool passed — the router address when routing through `MetricOmmSimpleRouter`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, passing `""` as `extensionData` and providing no mechanism to forward the original `msg.sender` (the end user) into the pool's `sender` slot: [4](#0-3) 

The same flaw exists in `exactInput`, `exactOutputSingle`, and `exactOutput`, all of which call `pool.swap` with the router as `msg.sender` and pass empty or opaque `extensionData` that the extension never decodes.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` gates on the explicit `owner` parameter (the position owner), not on `sender` (the caller), so it is not affected: [5](#0-4) 

The existing integration test `FullMetricExtension.t.sol` only exercises direct pool calls via `TestCaller`, never a router-mediated path, so the bypass is untested: [6](#0-5) 

## Impact Explanation
Any non-allowlisted user can call `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`/`exactOutputSingle`/`exactOutput`) targeting a pool that has `SwapAllowlistExtension` configured and the router allowlisted. The extension checks `allowedSwapper[pool][router]` → `true` and permits the swap. The unauthorized user receives pool tokens at oracle-driven prices the pool was not intended to serve them, draining LP assets and undermining the entire permissioned-pool design. This constitutes a direct loss of LP principal and broken core pool functionality (the allowlist invariant is violated).

## Likelihood Explanation
The trigger requires the pool admin to allowlist the router — the natural and expected configuration step for any permissioned pool intended to be accessible through the standard periphery. The admin has no on-chain signal that this action opens the gate to all users; `isAllowedToSwap(pool, router)` returns `true` with no indication of the broader bypass. The router is a public, unprivileged contract callable by any EOA or contract. The condition is therefore met in every realistic permissioned-pool deployment that uses the router.

## Recommendation
`SwapAllowlistExtension.beforeSwap` must gate on the actual end user, not the intermediary caller. The safest fix is to require the router to ABI-encode the original `msg.sender` into `extensionData`, and have the extension verify `msg.sender` (the pool's caller, i.e., the router) is a known trusted router before trusting the decoded address. Concretely:

1. In `MetricOmmSimpleRouter`, replace the `""` / `params.extensionData` passthrough with `abi.encode(msg.sender)` (or prepend it) when calling pools that use `SwapAllowlistExtension`.
2. In `SwapAllowlistExtension.beforeSwap`, if `sender` is a registered trusted router, decode the real user from `extensionData` and check `allowedSwapper[pool][realUser]` instead.

Alternatively, document clearly that allowlisting the router grants access to all users and provide a separate router-aware extension that performs this decoding.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is the only intended swapper
  allowedSwapper[pool][router] = true  // admin adds router so alice can use it

Attack:
  bob (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({
      pool: pool,
      tokenIn: token1,
      zeroForOne: false,
      amountIn: 10_000,
      amountOutMinimum: 0,
      recipient: bob,
      extensionData: ""
    })

  pool.swap() is called with msg.sender = router
  _beforeSwap(sender = router, ...)
  SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  Swap executes; bob receives token0 from the permissioned pool.

Result:
  bob, who was never allowlisted, successfully swaps on a pool
  the admin intended to restrict to alice only.
```

A Foundry reproduction adds a second `TestCaller` representing the router, allowlists it, then calls `pool.swap` from a third address (bob) impersonating the router — or deploys `MetricOmmSimpleRouter` in the test, allowlists it, and calls `exactInputSingle` from an address not in `allowedSwapper`.
