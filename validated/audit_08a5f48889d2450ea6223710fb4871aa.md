### Title
Zero-fill `matchOrders` charges taker minimum fee with no position change and non-advancing `filledAmounts`, enabling indefinite fee drain — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`matchOrders` in `OffchainExchange.sol` rounds `amountDelta` down to the nearest `sizeIncrement` but never checks whether the result is zero. When `amountDelta` rounds to zero, `quoteDelta` is also zero, yet `applyFee` still charges the taker the flat minimum fee (`market.minSize * feeRate`). Because `filledAmounts[digest]` is incremented by zero, the order is never consumed and the same match can be submitted again indefinitely, draining the taker's quote balance without any position change.

---

### Finding Description

In `matchOrders`, after computing the executable fill size, the code snaps it to the nearest `sizeIncrement`:

```solidity
ordersInfo.taker.amountDelta -=
    ordersInfo.taker.amountDelta %
    market.sizeIncrement;
``` [1](#0-0) 

If the raw `amountDelta` is smaller than `sizeIncrement`, this subtraction sets `amountDelta = 0`. There is no guard after this line to abort the match. Execution continues to `applyFee`:

```solidity
applyFee(
    callState.productId,
    ordersInfo.taker,
    market,
    -maker.order.priceX18.mul(filledAmounts[ordersInfo.taker.digest]),
    taker.order.appendix,
    true
);
``` [2](#0-1) 

Inside `applyFee`, when `taker = true` and `alreadyMatched == 0` (a fresh order whose `filledAmounts` is still zero), the flat minimum-fee branch fires unconditionally:

```solidity
if (alreadyMatched == 0) {
    meteredQuote += market.minSize;
    ...
}
``` [3](#0-2) 

With `matchQuote = 0`, the subsequent `feeApplied` calculation yields a negative value and is skipped, so `meteredQuote = market.minSize`. The fee computation then produces:

```
orderInfo.fee = market.minSize * feeRate  (> 0)
orderInfo.quoteDelta = 0 − fee − builderFee  (< 0)
``` [4](#0-3) 

`_updateBalances` is called with `amountDelta = 0` and `quoteDelta < 0`, so the taker loses quote tokens but gains zero position. Critically, `filledAmounts` is then incremented by zero:

```solidity
filledAmounts[ordersInfo.taker.digest] += ordersInfo.taker.amountDelta;
``` [5](#0-4) 

The order is never consumed. The sequencer can submit the same `MatchOrders` transaction repeatedly, each time extracting `market.minSize * feeRate` from the taker.

---

### Impact Explanation

The taker's quote balance is drained by the minimum fee on every zero-fill match. Because `filledAmounts` never advances, the order remains perpetually valid and the drain is unbounded — limited only by the taker's quote balance. The fee is credited to `market.collectedFees` (protocol) or a builder, so the sequencer can also route it to a builder it controls.

---

### Likelihood Explanation

The sequencer is the only entity that can submit `MatchOrders` transactions (via `submitTransactions`, which is `onlyEndpoint`). This is the direct analog of the MATCH_EXECUTOR in M-06. The sequencer can craft a match where the taker's order has `amount < sizeIncrement` — a non-zero value that passes `_validateOrder`'s `order.amount != 0` check — against any maker order. The condition is easy to construct and requires no external cooperation. This is a centralization risk identical in structure to the original finding. [6](#0-5) 

---

### Recommendation

After the `sizeIncrement` rounding, add an explicit guard:

```solidity
ordersInfo.taker.amountDelta -=
    ordersInfo.taker.amountDelta % market.sizeIncrement;

require(ordersInfo.taker.amountDelta != 0, ERR_ORDERS_CANNOT_BE_MATCHED);
```

This ensures that any match that would produce a zero fill is rejected outright, preventing fee extraction without position change and preventing the order from being re-matched indefinitely. [1](#0-0) 

---

### Proof of Concept

1. A product is configured with `sizeIncrement = 1e18` (1 full token in 18-decimal fixed-point).
2. A taker submits a signed order with `amount = 1` (1 raw unit, non-zero — passes `_validateOrder`).
3. A maker order exists with `amount = -1e18` (sufficient to match).
4. The sequencer submits `MatchOrders(taker, maker)`.
5. `amountDelta = min(1, 1e18) = 1`; after rounding: `1 - (1 % 1e18) = 1 - 1 = 0`.
6. `quoteDelta = 0 * priceX18 = 0`.
7. `applyFee` fires with `alreadyMatched = 0`, charges `market.minSize * feeRate` from the taker.
8. `_updateBalances`: taker loses quote, gains zero position.
9. `filledAmounts[takerDigest] += 0` — order unchanged.
10. Sequencer repeats step 4 indefinitely, draining the taker's quote balance. [7](#0-6) [8](#0-7)
