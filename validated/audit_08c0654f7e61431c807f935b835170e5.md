### Title
Missing Expiry Parameter on `MintNlp`/`BurnNlp` Signed Transactions Allows Stale Execution at Unfavorable Oracle Price — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `MintNlp` and `BurnNlp` user-signed transaction structs contain no expiry/deadline field. The oracle price used to settle these exchanges is injected by the sequencer at execution time and is not covered by the user's signature. If the sequencer delays processing, the user's transaction executes at a materially different NLP price than anticipated, causing direct asset loss with no on-chain protection.

---

### Finding Description

The `Order` struct — used for all spot/perp trade matching — carries an explicit `expiration` field that `_validateOrder` enforces via `!_expired(order.expiration)`. [1](#0-0) [2](#0-1) 

By contrast, the `MintNlp` and `BurnNlp` structs contain only `sender`, an asset amount, and a `nonce` — no expiry:

```solidity
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;
    uint64 nonce;
}

struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint64 nonce;
}
``` [3](#0-2) 

The sequencer-facing wrappers append `oraclePriceX18` and `nlpPoolRebalanceX18` **after** the user's signature is formed:

```solidity
struct SignedMintNlp {
    MintNlp tx;
    bytes  signature;
    int128 oraclePriceX18;
    int128[] nlpPoolRebalanceX18;
}
``` [4](#0-3) 

`processTransactionImpl` validates the user's signature over only the `MintNlp` body, then immediately uses the sequencer-supplied `oraclePriceX18` to settle the mint:

```solidity
validateSignedTx(signedTx.tx.sender, signedTx.tx.nonce,
                 transaction, signedTx.signature, true);
chargeFee(signedTx.tx.sender, HEALTHCHECK_FEE);
priceX18[NLP_PRODUCT_ID] = signedTx.oraclePriceX18;
clearinghouse.mintNlp(signedTx.tx, signedTx.oraclePriceX18, ...);
``` [5](#0-4) 

The identical pattern applies to `BurnNlp`. [6](#0-5) 

Because the user's signed payload contains no expiry, there is no on-chain mechanism to reject a `MintNlp`/`BurnNlp` transaction that arrives stale. The sequencer can hold a signed transaction and submit it at any future block with any oracle price it chooses to attach.

---

### Impact Explanation

A user who signs `MintNlp { quoteAmount: 1000 USDC, nonce: N }` intending to mint NLP at the current oracle price has no protection if the transaction is processed hours or days later at a price that is 2× higher. The user receives half the expected NLP tokens while the full `quoteAmount` is debited. The inverse holds for `BurnNlp`: a user burning NLP to recover quote collateral receives far less quote if the oracle price has fallen. The corrupted state is the user's NLP token balance and quote balance — a direct, quantifiable asset delta.

---

### Likelihood Explanation

The sequencer is the sole party that calls `submitTransactionsChecked`. Even without malicious intent, operational delays (sequencer downtime, reorgs, batching backlogs) can cause signed `MintNlp`/`BurnNlp` transactions to execute well after the user's intent window. Because the user's signature is permanently valid (only a nonce increment can invalidate it), any delay window is exploitable. The protocol already demonstrates awareness of this risk class by adding `expiration` to `Order`; the omission from NLP operations is a concrete gap.

---

### Recommendation

Add a `uint64 expiration` field to both `MintNlp` and `BurnNlp` structs (mirroring the `Order` struct). In `processTransactionImpl`, before calling `clearinghouse.mintNlp`/`burnNlp`, assert:

```solidity
require(block.timestamp <= signedTx.tx.expiration, "MintNlp: expired");
```

This ensures the user's intent is time-bounded and stale transactions are rejected on-chain, consistent with the existing `_expired` guard used for trading orders.

---

### Proof of Concept

1. User signs `MintNlp { sender: S, quoteAmount: 1000e18, nonce: 0 }` when NLP oracle price is `1.00 USDC/NLP`, expecting ~1000 NLP tokens.
2. Sequencer queues the transaction but does not submit it immediately.
3. NLP oracle price rises to `2.00 USDC/NLP` (e.g., due to pool rebalancing or market movement).
4. Sequencer submits the transaction with `oraclePriceX18 = 2e18`.
5. `clearinghouse.mintNlp` executes: user's quote balance is debited 1000 USDC, user receives only ~500 NLP tokens.
6. No on-chain check rejects the transaction; the user's signature remains valid because `MintNlp` carries no expiry field.

The corrupted invariant: user's NLP balance is `~500` instead of the `~1000` they committed to at signing time, with no recourse.
