### Title
`burnNlp` Uses Wrong Health Type (`MAINTENANCE` Instead of `INITIAL`), Allowing Subaccounts to Fall Below Initial Collateral Threshold â€” (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

The `burnNlp` function enforces only `MAINTENANCE` health after burning NLP tokens, while every other collateral-reduction path (`withdrawCollateral`) enforces the stricter `INITIAL` health. This inconsistency lets a user deliberately burn NLP and leave their subaccount below the initial collateral threshold without triggering liquidation.

---

### Finding Description

In `Clearinghouse.sol`, `withdrawCollateral` selects the health type as follows:

```solidity
IProductEngine.HealthType healthType = sender == X_ACCOUNT
    ? IProductEngine.HealthType.PNL
    : IProductEngine.HealthType.INITIAL;
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
``` [1](#0-0) 

For any normal subaccount this enforces `INITIAL` health â€” the stricter threshold. `mintNlp` does the same:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [2](#0-1) 

But `burnNlp` â€” which is also a collateral-reduction operation â€” only checks `MAINTENANCE`:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [3](#0-2) 

`RiskHelper._getWeightX18` shows that `INITIAL` and `MAINTENANCE` use different risk weights:

```solidity
weight = healthType == IProductEngine.HealthType.INITIAL
    ? risk.longWeightInitialX18
    : risk.longWeightMaintenanceX18;
``` [4](#0-3) 

`INITIAL` weights are stricter (lower for longs, higher for shorts), so `INITIAL health < MAINTENANCE health` for the same position. A subaccount can satisfy `MAINTENANCE >= 0` while having `INITIAL < 0`. The code comment itself acknowledges the risk: *"Burning NLP can decrease health if the burn fee exceeds the health improvement from the withdrawal."* [5](#0-4) 

The burn fee is always at least `ONE` (1e18):

```solidity
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [6](#0-5) 

This guarantees the user always receives less quote than the NLP was worth, so the net health impact of a burn is always negative or zero. The wrong health type means this negative impact is evaluated against the wrong (lenient) threshold.

---

### Impact Explanation

A user with open perp positions and NLP balance can burn NLP to push their subaccount into the zone where `INITIAL health < 0` but `MAINTENANCE health >= 0`. This:

1. **Bypasses the initial collateral requirement** that `withdrawCollateral` enforces for every other collateral reduction.
2. **Leaves the subaccount in a structurally undercollateralized state** â€” it cannot open new positions but is not yet liquidatable.
3. **Increases bad-debt risk**: a small adverse price move immediately makes the subaccount liquidatable, potentially leaving the protocol with a shortfall that the insurance fund must absorb.

The corrupted state is the subaccount's health value: `INITIAL health < 0` after a `burnNlp` that the protocol should have rejected.

---

### Likelihood Explanation

Any unprivileged user who holds NLP tokens and has open positions can trigger this. The entry path is fully permissionless:

1. User signs a `BurnNlp` transaction (EIP-712).
2. Sequencer submits it via `submitTransactionsChecked` â†’ `EndpointTx` â†’ `Clearinghouse.burnNlp`.
3. No admin access, governance, or special role is required. [7](#0-6) 

---

### Recommendation

Replace `MAINTENANCE` with `INITIAL` in `burnNlp` to match the invariant enforced by `withdrawCollateral` and `mintNlp`:

```solidity
// Before
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);

// After
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
```

---

### Proof of Concept

Assume a product with `longWeightInitialX18 = 0.8e18` and `longWeightMaintenanceX18 = 0.9e18`.

1. User holds NLP worth 1000 quote units and has an open long perp position.
2. Current `INITIAL health = +50`, `MAINTENANCE health = +150`.
3. User burns NLP. After the burn fee, they receive 990 quote. The NLP removal reduces health by ~100 INITIAL units (due to the stricter weight) but only ~90 MAINTENANCE units.
4. Post-burn: `INITIAL health = 50 âˆ’ 100 = âˆ’50`, `MAINTENANCE health = 150 âˆ’ 90 = +60`.
5. The `burnNlp` check (`MAINTENANCE >= 0`) passes. The same operation would have been **rejected** by `withdrawCollateral` (`INITIAL >= 0`).
6. The subaccount now holds a position that is undercollateralized by initial standards, a state the protocol explicitly forbids for all other collateral-reduction paths.
