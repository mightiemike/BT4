### Title
Merkle Proof Length Not Validated Against Tree Depth Enables Internal-Node Forgery in `verify_transaction_inclusion_v2` — (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

Neither `verify_transaction_inclusion` nor `verify_transaction_inclusion_v2` validates the submitted Merkle proof length against the expected tree depth. An unprivileged NEAR caller can supply a proof that is shorter than the actual tree depth, using an **internal Merkle tree node** as the claimed `tx_id`. Because `compute_root_from_merkle_proof` blindly iterates over whatever proof elements are provided, the truncated proof still reconstructs the correct Merkle root, causing the function to return `true` for a transaction that does not exist.

---

### Finding Description

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` accepts a proof of arbitrary length and iterates over it without any bound check:

```rust
for proof_hash in merkle_proof {
    // hash current_hash with proof_hash, shift position
}
return current_hash;
``` [1](#0-0) 

The contract's deprecated `verify_transaction_inclusion` only checks that the proof is non-empty:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [2](#0-1) 

The recommended `verify_transaction_inclusion_v2` adds a coinbase cross-check and requires both proofs to have equal length, but **never validates either length against the actual tree depth**:

```rust
require!(
    args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
    "Coinbase merkle proof and transaction merkle proof should have the same length"
);
``` [3](#0-2) 

The block header stored on-chain contains only the `merkle_root`; the transaction count is never stored, so there is no on-chain reference depth to validate against. [4](#0-3) 

---

### Impact Explanation

An attacker who knows the Merkle tree of any confirmed Bitcoin block (all data is public) can:

1. Pick an **internal node** at depth `k` from the bottom of the tree as their claimed `tx_id`.
2. Construct a proof of length `D − k` (where `D` is the actual tree depth) from that internal node up to the root. This proof is mathematically valid and produces the correct `merkle_root`.
3. For `verify_transaction_inclusion_v2`, simultaneously construct a matching shortened coinbase proof of the same length `D − k`, where `coinbase_tx_id` is the internal node at depth `k` at position 0.
4. Call `verify_transaction_inclusion_v2` with these crafted arguments.

The function passes all checks and returns `true` for a **non-existent transaction**. Any consumer contract (e.g., an atomic swap, bridged asset release, or cross-chain state gate) that relies on this return value as proof of Bitcoin transaction inclusion is deceived into accepting a fraudulent claim. [5](#0-4) 

---

### Likelihood Explanation

- **No privileged access required.** `verify_transaction_inclusion_v2` is a public, unpermissioned NEAR view/call method.
- **All required data is public.** Bitcoin block Merkle trees are fully reconstructible from any block explorer or full node.
- **The coinbase mitigation does not block this attack.** The equal-length check only ensures both proofs are the same depth; it does not anchor either proof to the actual tree depth. An attacker supplies a shortened coinbase proof alongside the shortened target proof, and both validate correctly.
- **The deprecated v1 function is still callable** (it is `#[pause]`-gated, not removed), widening the attack surface further. [6](#0-5) 

---

### Recommendation

Store the transaction count (`tx_count`) in the `ExtendedHeader` or `LightHeader` at block submission time. During proof verification, compute the expected proof depth as `ceil(log2(tx_count))` and reject any proof whose length does not match:

```rust
let expected_depth = (tx_count as f64).log2().ceil() as usize;
require!(
    args.merkle_proof.len() == expected_depth,
    "Merkle proof length does not match tree depth"
);
```

This mirrors the recommendation in the referenced Sablier audit: store the tree depth at creation time and validate the proof length against it during the claim/verification process.

---

### Proof of Concept

Consider a Bitcoin block with **8 transactions** (tree depth = 3). The Merkle tree looks like:

```
Level 3 (root):  R
Level 2:         A              B
Level 1:         C    D         E    F
Level 0 (leaf):  T0 T1  T2 T3   T4 T5  T6 T7
```

**Legitimate proof for T0** (length 3): `[T1, D, B]`

**Attacker's forged proof** (length 2, claiming internal node `C` is a transaction):
- `tx_id` = `C` (= `hash(T0, T1)`, an internal node)
- `tx_index` = 0
- `merkle_proof` = `[D, B]`
- `coinbase_tx_id` = `C` (same internal node, position 0)
- `coinbase_merkle_proof` = `[D, B]`

`compute_root_from_merkle_proof(C, 0, [D, B])`:
- Step 1: `hash(C, D)` = `A`
- Step 2: `hash(A, B)` = `R` ✓ matches `merkle_root`

Both the coinbase check and the transaction check pass. `verify_transaction_inclusion_v2` returns `true` for a transaction that does not exist in the block. [7](#0-6) [8](#0-7)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
