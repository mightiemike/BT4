### Title
Fork Retarget Validation Uses Mainchain Timestamp Instead of Fork Ancestor — (`contract/src/bitcoin.rs` + `contract/src/lib.rs`)

### Summary

`get_next_work_required` calls `get_header_by_height` to fetch the timestamp of the first block in the current difficulty period. `get_header_by_height` always reads from `mainchain_height_to_header`, which only indexes mainchain blocks. When the block being validated is a fork block at a retarget boundary, and the fork diverges at or before `first_block_height`, the contract uses the mainchain's block timestamp instead of the fork's ancestor timestamp. This produces an incorrect expected `bits` value, allowing the contract to accept a fork block whose `bits` field would be rejected by the real Bitcoin network.

### Finding Description

In `contract/src/bitcoin.rs`, `get_next_work_required` computes the retarget difficulty: [1](#0-0) 

`first_block_height` is the block at the start of the 2016-block difficulty window. The timestamp of that block determines the computed timespan and thus the new target. The lookup is delegated to `blocks_getter.get_header_by_height`, whose only implementation is: [2](#0-1) 

This reads exclusively from `mainchain_height_to_header`. Fork headers are stored only in `headers_pool` via `store_fork_header`: [3](#0-2) 

`store_fork_header` never writes to `mainchain_height_to_header`. Therefore, if a fork diverges at or before `first_block_height`, the fork's block at that height is in `headers_pool` but is **not** reachable through `get_header_by_height`. The contract silently falls back to the mainchain block at that height.

By contrast, `get_prev_header` correctly follows the fork chain: [4](#0-3) 

The two traversal methods are inconsistent: `get_prev_header` is fork-aware; `get_header_by_height` is not.

### Impact Explanation

An attacker who mines a Bitcoin testnet fork that:
- diverges at or before `first_block_height = retarget_height - 2015`, and
- places a fork block at `first_block_height` with timestamp `T_fork > T_mainchain` (the mainchain block at that height has an earlier timestamp),

causes the contract to compute a longer apparent timespan (`T_prev_fork - T_mainchain` instead of `T_prev_fork - T_fork`). A longer timespan produces a lower computed target difficulty. The attacker then mines the retarget block with `bits` matching this artificially lowered difficulty. The contract accepts it; the real Bitcoin network would reject it because the correct `bits` (computed from `T_fork`) is higher.

If this fork accumulates more chainwork than the mainchain (feasible on testnet with min-difficulty blocks), `reorg_chain` promotes it: [5](#0-4) 

The contract's mainchain now contains a block with an invalid `bits` value. Any downstream bridge that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` against this fork will treat confirmations on an invalid chain as final — a light client verification bypass.

### Likelihood Explanation

- On Bitcoin testnet, `pow_allow_min_difficulty_blocks = true`. The attacker can mine min-difficulty blocks (20-bit target) to build the 2015-block fork prefix cheaply.
- The attacker does not need to be a trusted relayer. The honest relayer reads from the Bitcoin testnet P2P network and submits whatever fork has the most chainwork. The attacker only needs to mine a heavier fork.
- The `trusted_relayer` gate on `submit_blocks` is not a barrier: the attacker controls the *data* the relayer submits, not the relayer itself.
- The only constraint is mining ~2015 min-difficulty testnet blocks, which is computationally trivial.

### Recommendation

`get_header_by_height` must be made fork-aware. The simplest correct fix is to walk the fork chain backwards from `prev_block_header` by `difficulty_adjustment_interval - 1` steps using `get_prev_header`, exactly as Bitcoin Core does. This avoids any height-indexed lookup and always follows the correct branch:

```rust
// In get_next_work_required, replace the get_header_by_height call with:
let mut cursor = prev_block_header.clone();
for _ in 0..(config.difficulty_adjustment_interval - 1) {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
// cursor is now the first block of the difficulty window on the correct branch
calculate_next_work_required(config, prev_block_header, cursor.block_header.time.into())
```

Alternatively, pass the fork-branch ancestor explicitly through the call chain rather than relying on a height index.

### Proof of Concept

1. Initialize the contract on testnet at a retarget boundary height `N * 2016`.
2. Mine mainchain blocks up to height `N * 2016 + 2015` (the block before the next retarget). Call the mainchain block at height `N * 2016 + 1` block `M` with timestamp `T_M`.
3. Mine a fork starting at height `N * 2016 + 1` with a block `F` whose timestamp `T_F > T_M + X` for some large `X`. Continue the fork with min-difficulty blocks up to height `N * 2016 + 2015`.
4. At height `N * 2016 + 2016` (retarget), the contract calls `get_header_by_height(N * 2016 + 1)` and gets `M` (timestamp `T_M`) instead of `F` (timestamp `T_F`). The computed timespan is `T_prev - T_M > T_prev - T_F`, yielding a lower `bits` value.
5. Mine the fork's retarget block with `bits` = the lowered value. The hash satisfies the easier PoW target.
6. Submit the fork via the relayer. `check_pow` passes because `expected_bits == block_header.bits` (both use the wrong mainchain timestamp). The block is stored.
7. If the fork's total chainwork exceeds the mainchain's, `reorg_chain` promotes it. The contract's canonical chain now contains a block with invalid `bits`.
8. Call `verify_transaction_inclusion` with a transaction in the fork. It returns `true` — a confirmation on a chain the real Bitcoin network rejects.
