### Title
Zero `confirmations` Accepted Without Minimum Validation, Bypassing Finality Guarantee in SPV Proof — (`contract/src/lib.rs`)

---

### Summary

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations: u64` with no lower-bound validation. When `confirmations = 0`, the confirmation-depth check is trivially satisfied for any on-chain block, causing the contract to certify a transaction as "included" with zero finality. Any downstream NEAR contract that passes a user-controlled `confirmations` value to the light client can be tricked into accepting a transaction that has not yet achieved any depth on the Bitcoin chain and may be reorganized away.

---

### Finding Description

`ProofArgs` and `ProofArgsV2` both carry a `confirmations: u64` field with no minimum constraint. [1](#0-0) 

Inside `verify_transaction_inclusion`, the only upper-bound guard is:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [2](#0-1) 

The depth check that follows is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [3](#0-2) 

When `confirmations = 0`:
- The upper-bound guard passes: `0 <= gc_threshold` is always true.
- The depth expression evaluates to `N >= 0` where `N` is a `u64`, which is always true regardless of how recently the block was submitted.

The function then proceeds to compute and return the Merkle proof result — returning `true` for a transaction in a block sitting at the chain tip with no depth at all. [4](#0-3) 

`verify_transaction_inclusion_v2` (the non-deprecated, production-recommended path) delegates directly to `verify_transaction_inclusion` after its coinbase proof check, inheriting the same flaw: [5](#0-4) 

The contract's own integration tests normalize `confirmations: 0` as an accepted call pattern, confirming the contract never rejects it: [6](#0-5) 

---

### Impact Explanation

The `confirmations` parameter is the sole finality control exposed to callers. Setting it to 0 causes the contract to certify a transaction as SPV-proven while the containing block is still at the chain tip — one block deep — and therefore trivially reversible by a Bitcoin reorg.

Any downstream NEAR contract (bridge, token release, escrow) that:
- accepts a user-supplied `confirmations` value and forwards it to the light client, **or**
- calls the light client with `confirmations = 0` as a default

will accept a proof result that carries no finality guarantee. An attacker can:
1. Broadcast a Bitcoin transaction and mine/submit the containing block to the light client.
2. Immediately call `verify_transaction_inclusion_v2` with `confirmations = 0` — the call returns `true`.
3. Trigger the downstream NEAR contract action (e.g., mint wrapped BTC, release escrowed funds).
4. Cause or wait for a Bitcoin reorg that removes the original block, invalidating the transaction.

The corrupted value is the **proof result**: the contract returns `true` for a transaction that has not achieved any confirmation depth, violating the SPV finality invariant the contract is designed to enforce.

---

### Likelihood Explanation

The likelihood is **medium**. The contract is explicitly designed as an infrastructure primitive for downstream NEAR contracts. The `confirmations` parameter is the only knob those contracts have for finality control. Any downstream contract that exposes `confirmations` as a user-settable parameter (a natural design for a general-purpose bridge) is immediately exploitable. The contract's own tests use `confirmations: 0` as a routine call pattern, making it likely that integrators will treat 0 as a valid production value rather than a test-only shortcut.

---

### Recommendation

Add a minimum confirmation bound enforced inside `verify_transaction_inclusion`:

```rust
const MIN_CONFIRMATIONS: u64 = 1; // or a configurable on-chain parameter

require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    "Confirmations must be at least MIN_CONFIRMATIONS"
);
```

Alternatively, expose a `min_confirmations` field in contract state (set at `init`) and enforce it in both proof functions. Document clearly that `confirmations = 0` is not a safe production value and should be rejected by the contract rather than silently accepted.

---

### Proof of Concept

**Attacker-controlled entry path** (no privileged role required):

1. Relayer submits Bitcoin block `B` containing attacker transaction `T` to the light client via `submit_blocks`.
2. Attacker immediately calls (as any unprivileged NEAR account):

```json
{
  "tx_id": "<hash of T>",
  "tx_block_blockhash": "<hash of B>",
  "tx_index": 0,
  "merkle_proof": ["<valid sibling hash>"],
  "coinbase_tx_id": "<coinbase hash>",
  "coinbase_merkle_proof": ["<valid sibling hash>"],
  "confirmations": 0
}
```

on `verify_transaction_inclusion_v2`.

3. The upper-bound check `0 <= gc_threshold` passes. The depth check `(tip_height - B_height + 1) >= 0` evaluates to `1 >= 0` — always true. The Merkle proof is valid. The function returns `true`.
4. Block `B` is at the chain tip with zero depth. A competing Bitcoin chain tip that excludes `T` can be submitted, causing a reorg that removes `B` from the canonical chain.
5. The downstream contract has already acted on the `true` result. [7](#0-6) [8](#0-7)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );
```
