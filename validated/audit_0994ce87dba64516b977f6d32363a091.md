### Title
Protobuf `ValidResourceBounds` Deserialization Collapses `AllResources` to `L1Gas` When `l2_gas`/`l1_data_gas` Are Zero, Producing a Different Transaction Hash Preimage — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic (`l1_data_gas.is_zero() && l2_gas.is_zero()`) to decide which enum variant to reconstruct. A V3 transaction signed with `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` produces a hash preimage that includes a `L1_DATA_GAS(zero)` element. After a protobuf round-trip the same transaction is reconstructed as `L1Gas(X)`, whose hash preimage omits that element entirely. The two Poseidon hashes are therefore distinct, so any node that recomputes the hash from the deserialized form will disagree with the hash the signer committed to.

---

### Finding Description

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` produces structurally different preimages depending on the `ValidResourceBounds` variant:

- `L1Gas(X)` → `poseidon_hash([tip, L1_GAS_packed(X), L2_GAS_packed(0)])` — **3 elements**
- `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` → `poseidon_hash([tip, L1_GAS_packed(X), L2_GAS_packed(0), L1_DATA_GAS_packed(0)])` — **4 elements** [1](#0-0) 

The serializer `From<ValidResourceBounds> for protobuf::ResourceBounds` encodes both variants identically when `l2_gas` and `l1_data_gas` are zero: [2](#0-1) 

The deserializer then reconstructs the variant purely from the wire values:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(...)
})
``` [3](#0-2) 

Because the protobuf wire format carries no tag distinguishing the two variants, the deserializer always produces `L1Gas` for the zero-bounds case, silently discarding the `AllResources` identity that was present when the hash was originally computed.

The RPC and gateway paths always use `AllResourceBounds` (never `ValidResourceBounds::L1Gas`) for V3 transactions: [4](#0-3) 

So the hash stored in `InternalRpcTransaction.tx_hash` is always computed under the `AllResources` branch. The P2P block-sync path, however, uses the `ValidResourceBounds` converter, which can silently switch to `L1Gas`. [5](#0-4) 

---

### Impact Explanation

Any node that receives a block via P2P sync and recomputes the transaction hash from the deserialized `ValidResourceBounds` will obtain a hash that differs from the one the sequencer stored and the user signed. This causes:

1. **Hash validation failure** — the syncing node rejects a valid block, producing a chain split between the sequencer and all syncing peers.
2. **Wrong executable object** — if the wrong variant is stored and later used for re-execution or proving, the fee-computation mode (`GasVectorComputationMode::NoL2Gas` vs `All`) and the balance-check path differ, producing wrong receipts and wrong fee charges.

The impact matches: *High — Transaction conversion or signature/hash logic binds the wrong hash or executable payload.*

---

### Likelihood Explanation

The trigger requires only a standard V3 invoke transaction with `l2_gas = 0` and `l1_data_gas = 0`. The gateway's stateless validator explicitly accepts this (the test suite includes `valid_l1_gas` with only `l1_gas` non-zero). No special privilege is needed; any unprivileged user can submit such a transaction. Once included in a block, every syncing peer is affected.

---

### Recommendation

Preserve the variant identity across the protobuf boundary. Two concrete options:

1. **Add a discriminant field** to `protobuf::ResourceBounds` (e.g., a boolean `is_all_resources`) so the deserializer can reconstruct the exact variant without inspecting values.
2. **Reject the ambiguous case at the gateway** — if a V3 transaction arrives with `l2_gas = 0` and `l1_data_gas = 0`, treat it as `L1Gas` *before* hashing, so the hash is always computed under the variant that survives the protobuf round-trip. This requires the gateway to normalise the variant before calling `calculate_transaction_hash`.

Either way, the invariant that must hold is: `serialize(v)` followed by `deserialize(...)` must return a `ValidResourceBounds` whose `get_tip_resource_bounds_hash` output is identical to the original.

---

### Proof of Concept

```
1. Craft a V3 invoke transaction T with:
     resource_bounds = AllResourceBounds { l1_gas: {max_amount:1, max_price:1},
                                           l2_gas: {0,0},
                                           l1_data_gas: {0,0} }

2. Submit T to the gateway via JSON-RPC.
   Gateway calls convert_rpc_tx_to_internal → tx.resource_bounds is wrapped as
   ValidResourceBounds::AllResources(...) → get_tip_resource_bounds_hash produces
   H1 = poseidon([tip, L1_GAS_packed, L2_GAS_packed(0), L1_DATA_GAS_packed(0)]).
   tx_hash = H1 is stored in InternalRpcTransaction.

3. T is included in block B with stored tx_hash = H1.

4. A syncing peer receives B via P2P.
   The transaction is deserialized through TryFrom<protobuf::ResourceBounds> for ValidResourceBounds.
   Because l2_gas.is_zero() && l1_data_gas.is_zero(), the result is ValidResourceBounds::L1Gas(l1_gas).

5. The syncing peer recomputes the hash:
   get_tip_resource_bounds_hash(L1Gas) → H2 = poseidon([tip, L1_GAS_packed, L2_GAS_packed(0)]).
   H2 ≠ H1 (different number of Poseidon inputs).

6. Hash validation fails → the syncing peer rejects block B.
   The sequencer and all syncing peers diverge on the canonical chain.
```

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
