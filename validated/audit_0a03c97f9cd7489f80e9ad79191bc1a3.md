### Title
`MALACHITE` Flag Fully Implemented in Rust but Never Exposed to Python Callers — Arithmetic Hard-Fork Path Permanently Unreachable via Python API - (File: `wheel/src/api.rs`)

---

### Summary

`ClvmFlags::MALACHITE` (`0x1000`) is defined in `src/chia_dialect.rs`, fully wired into four arithmetic operators in `src/more_ops.rs`, and exercised by fuzz targets — but it is never added to the Python module in `wheel/src/api.rs`. Python callers (including `chia_rs`, the primary Chia node implementation) have no documented or exported path to activate the malachite arithmetic path. When the `MALACHITE` hard fork is intended to activate, Python nodes cannot set the flag through the published API, creating a permanent wiring gap between the Rust implementation and its Python-facing control surface.

---

### Finding Description

`ClvmFlags::MALACHITE` is declared alongside every other hard-fork flag in `src/chia_dialect.rs`:

```rust
/// Use malachite-bigint instead of num-bigint for div, divmod, mod, and modpow.
const MALACHITE = 0x1000;
``` [1](#0-0) 

The flag is fully implemented: `op_div`, `op_divmod`, `op_mod`, and `op_modpow` in `src/more_ops.rs` each branch on `ClvmFlags::MALACHITE` and dispatch to dedicated malachite-bigint variants (`op_div_malachite`, `op_divmod_malachite`, `op_mod_malachite`, `op_modpow_malachite`). [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

The Python module registration in `wheel/src/api.rs` exports every other operationally significant flag — `NO_UNKNOWN_OPS`, `LIMIT_HEAP`, `MEMPOOL_MODE`, `ENABLE_SHA256_TREE`, `ENABLE_SECP_OPS`, `DISABLE_OP`, `CANONICAL_INTS` — but `MALACHITE` is absent:

```rust
m.add("NO_UNKNOWN_OPS", ClvmFlags::NO_UNKNOWN_OPS.bits())?;
m.add("LIMIT_HEAP", ClvmFlags::LIMIT_HEAP.bits())?;
m.add("MEMPOOL_MODE", MEMPOOL_MODE.bits())?;
m.add("ENABLE_SHA256_TREE", ClvmFlags::ENABLE_SHA256_TREE.bits())?;
m.add("ENABLE_SECP_OPS", ClvmFlags::ENABLE_SECP_OPS.bits())?;
m.add("DISABLE_OP", ClvmFlags::DISABLE_OP.bits())?;
m.add("CANONICAL_INTS", ClvmFlags::CANONICAL_INTS.bits())?;
// MALACHITE (0x1000) is never added
``` [6](#0-5) 

The Python stub file confirms the omission — `MALACHITE` does not appear among the exported constants: [7](#0-6) 

The project's own operator-addition checklist explicitly mandates: *"Expose the new flag(s) to python in chia_rs."* [8](#0-7) 

`run_serialized_chia_program` accepts `flags: u32` and calls `ClvmFlags::from_bits_truncate(flags)`, which would honour bit `0x1000` if passed. However, because the constant is not exported, downstream callers have no stable, documented way to set it — they would have to hard-code a magic integer, which is not how any other flag is consumed. [9](#0-8) 

---

### Impact Explanation

**Medium.** The `MALACHITE` flag controls which bignum library (`num-bigint` vs `malachite-bigint`) is used for `div`, `divmod`, `mod`, and `modpow`. The existence of four separate malachite-specific implementations and dedicated fuzz targets that compare both paths against a reference implementation confirms the developers expect the two libraries to be interchangeable — but the need to fuzz-test this equivalence implies edge-case divergence is possible. When the `MALACHITE` hard fork is intended to activate on-chain, Python-based nodes (e.g., `chia_rs`) cannot set the flag through the exported API. If any input produces a different result under the two libraries, Python nodes and Rust nodes that correctly pass `MALACHITE` will disagree on program output, causing **consensus divergence**: a spend bundle accepted by one class of nodes is rejected by another. Even if the two libraries are currently equivalent for all tested inputs, the wiring gap means the activation mechanism is broken by design — the hard fork can never be properly engaged through the Python control surface.

---

### Likelihood Explanation

**Medium.** The flag is clearly structured as a hard-fork activation flag (same pattern as `RELAXED_BLS`, `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`, `ENABLE_SHA256_TREE`, `ENABLE_SECP_OPS`). The project checklist explicitly requires Python exposure of new flags. The omission is not guarded by any comment indicating intentional deferral. Any future attempt to activate the `MALACHITE` hard fork via the Python wheel will silently fail unless the constant is added.

---

### Recommendation

Add the missing constant to the Python module registration in `wheel/src/api.rs`:

```rust
m.add("MALACHITE", ClvmFlags::MALACHITE.bits())?;
```

And add the corresponding entry to `wheel/python/clvm_rs/clvm_rs.pyi`:

```python
MALACHITE: int
```

This mirrors the pattern used for every other hard-fork flag and satisfies the project's own operator-addition checklist requirement.

---

### Proof of Concept

A Python caller attempting to activate the malachite arithmetic path:

```python
from clvm_rs import run_serialized_chia_program, MALACHITE  # ImportError: cannot import name 'MALACHITE'
```

The constant does not exist. A caller who knows the raw bit value can work around it:

```python
MALACHITE_RAW = 0x1000
cost, result = run_serialized_chia_program(program, env, max_cost, MALACHITE_RAW)
```

But this is undocumented, fragile, and not how any other flag is consumed. When the hard fork activates and `chia_rs` is updated to pass `MALACHITE`, the missing export means the update cannot be made through the normal flag-constant mechanism, and any node that fails to pass the raw integer will execute `div`/`divmod`/`mod`/`modpow` under `num-bigint` semantics instead of `malachite-bigint` semantics — a silent arithmetic divergence on every affected opcode invocation.
