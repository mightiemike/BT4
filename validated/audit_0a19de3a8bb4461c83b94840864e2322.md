Audit Report

## Title
SwapAllowlistExtension Checks Router Address as `sender`, Allowing Any User to Bypass the Swap Allowlist via `MetricOmmSimpleRouter` â€” (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`, which forwards it to `SwapAllowlistExtension.beforeSwap`. When a user routes through `MetricOmmSimpleRouter`, `sender` becomes the router's address. If the pool admin adds the router to the allowlist (the natural setup to enable multi-hop routing for legitimate users), any non-allowlisted user can bypass the restriction by calling the public router instead of the pool directly.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` directly as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes and forwards this `sender` to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the direct caller of `pool.swap`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router `msg.sender` of that call: [4](#0-3) 

The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput` â€” all router entry points call `pool.swap` with the router as `msg.sender`: [5](#0-4) 

When the admin adds the router to the allowlist so that allowlisted users can perform multi-hop swaps, `allowedSwapper[pool][router] = true`. Any non-allowlisted user who calls the router will have `sender = router` evaluated by the extension, which passes the check. The originating user's address is never inspected.

## Impact Explanation
A pool admin deploys `SwapAllowlistExtension` to restrict swaps to a curated set of addresses (e.g., KYC'd traders). To enable multi-hop routing for those users, the admin adds `MetricOmmSimpleRouter` to the allowlist. This is the only way to allow allowlisted users to use the router. The consequence is that every non-allowlisted user can now call `MetricOmmSimpleRouter` targeting the restricted pool and execute swaps â€” the allowlist guard is fully bypassed. This is an admin-boundary break: an unprivileged, public path circumvents a pool-level access control intended to restrict who can trade, with direct fund-flow consequences (non-permitted parties execute swaps against the pool's liquidity).

## Likelihood Explanation
`MetricOmmSimpleRouter` is a public, permissionless contract. Any user can call it with any pool address and swap parameters. The only precondition is that the router is on the allowlist, which is the natural and necessary setup any admin would choose to allow their own allowlisted users to use multi-hop routing. No privileged access, no special tokens, no malicious setup is required â€” a standard user interaction triggers the bypass.

## Recommendation
`SwapAllowlistExtension.beforeSwap` must gate on the originating user, not on `sender` (the direct pool caller). Two viable approaches:

1. **Forward the true caller via `extensionData`**: Require the router to encode `msg.sender` (the originating user) into `extensionData` and have the extension verify that address against the allowlist instead of `sender`.
2. **Extend the extension interface**: Add a dedicated field for the originating caller that the pool or router populates, and check that field in the allowlist hook.

Option 3 from the submission (do not allowlist the router) is not viable in practice because it prevents allowlisted users from using multi-hop routing.

## Proof of Concept

1. Pool admin deploys pool with `SwapAllowlistExtension` configured in `BEFORE_SWAP_ORDER`.
2. Admin calls `setAllowedToSwap(pool, routerAddress, true)` and `setAllowedToSwap(pool, aliceAddress, true)`. Bob (non-allowlisted) is excluded.
3. Bob calls `MetricOmmSimpleRouter.exactInputSingle({pool: restrictedPool, ...})`.
4. Router calls `IMetricOmmPoolActions(pool).swap(recipient=Bob, ...)` â€” router is `msg.sender` of `pool.swap`.
5. Pool calls `_beforeSwap(sender=router, ...)` â†’ extension receives `sender=router`.
6. Extension evaluates `allowedSwapper[pool][router]` â†’ `true` â†’ returns success selector.
7. Swap executes. Bob, explicitly excluded from the allowlist, has successfully swapped against the restricted pool.
