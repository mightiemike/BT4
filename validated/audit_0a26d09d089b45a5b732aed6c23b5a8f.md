### Title
Linked Signer Can Permanently Replace Itself With an Attacker-Controlled Address — (`core/contracts/EndpointTx.sol`)

### Summary
The fast-mode `LinkSigner` handler in `processTransactionImpl` calls `validateSignedTx` with `allowLinkedSigner=true`, meaning the currently registered linked signer's signature is accepted as sufficient authorization to overwrite `linkedSigners[sender]`. There is no guard requiring the subaccount owner to authorize signer changes. A linked signer can therefore replace itself with any attacker-controlled address, permanently hijacking signing authority over the victim's subaccount.

### Finding Description

In `processTransactionImpl`, the `LinkSigner` branch: [1](#0-0) 

calls `validateSignedTx(..., true)` — the final `true` is `allowLinkedSigner`. This flag is forwarded through: [2](#0-1) 

into `Verifier.validateSignature`, which accepts the signature if the recovered address equals either the subaccount owner **or** the currently registered linked signer: [3](#0-2) 

`requireSubaccount` only verifies the subaccount is registered — it does not verify the caller is the owner: [4](#0-3) 

The `linkedSigners` mapping is then unconditionally overwritten: [5](#0-4) 

No code anywhere in this path checks that the signer of a `LinkSigner` transaction is the subaccount owner rather than the currently delegated linked signer.

### Impact Explanation
Once `linkedSigners[victim:default]` is set to `attacker`, the attacker can sign any fast-mode transaction on behalf of the victim's subaccount — including `WithdrawCollateral`, further `LinkSigner` changes, `TransferQuote`, and `LiquidateSubaccount`. This is a complete, permanent takeover of the victim's subaccount with no recovery path unless the victim submits a slow-mode `LinkSigner` override (which requires the victim to notice the hijack and act before the attacker drains funds).

### Likelihood Explanation
Any party that has ever been granted linked-signer access (e.g., a trading bot, a compromised API key, a former delegate) can execute this unilaterally. The sequencer submits the transaction without any additional owner-consent check. The precondition — being a registered linked signer — is a normal, intended protocol state.

### Recommendation
Change the `allowLinkedSigner` flag to `false` for the `LinkSigner` transaction type in `processTransactionImpl`:

```solidity
// core/contracts/EndpointTx.sol  ~line 581
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // ← was true; only the subaccount owner may change the linked signer
);
```

This ensures only the subaccount owner (the address embedded in `sender`) can authorize signer changes, matching the invariant that signing authority delegation must originate from the owner.

### Proof of Concept

1. Deploy fixture: victim registers subaccount `victim:default`, deposits collateral, calls slow-mode `LinkSigner` to register `linkedSigner` address.
2. `linkedSigner` constructs and signs `LinkSigner{sender=victim:default, signer=attacker, nonce=N}`.
3. Sequencer calls `submitTransactionsChecked` → `processTransactionImpl`.
4. `validateSignedTx(..., true)` recovers `linkedSigner` from the signature; `Verifier.validateSignature` accepts it because `recovered == linkedSigner`.
5. `linkedSigners[victim:default]` is set to `attacker`.
6. Assert `linkedSigners[victim:default] == attacker`.
7. `attacker` signs `WithdrawCollateral{sender=victim:default, ...}`; sequencer submits; withdrawal succeeds.

        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
