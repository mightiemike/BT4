Audit Report

## Title
`SwapAllowlistExtension` Per-User Gate Bypassed via Router Identity — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to its own `msg.sender` — the router contract when a user routes through `MetricOmmSimpleRouter`. If the router is allowlisted (required for any allowlisted user to use the standard periphery), every unprivileged address can bypass the per-user restriction by calling the public router. The pool admin cannot achieve selective per-user allowlisting for router-mediated swaps.

## Finding Description

**Root cause — wrong identity checked:**

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool (correct) and `sender` is the first argument forwarded by the pool. [1](#0-0) 

**Pool wires its own `msg.sender` as `sender`:**

`MetricOmmPool.swap` passes `msg.sender` of the `swap()` call as the first argument to `_beforeSwap`. When the user routes through `MetricOmmSimpleRouter`, that `msg.sender` is the router contract, not the end user. [2](#0-1) 

**ExtensionCalling forwards it unchanged:**

`_beforeSwap` in `ExtensionCalling` passes `sender` directly into the encoded call to every registered extension with no mechanism to recover the original initiator. [3](#0-2) 

**Router is the direct caller of `pool.swap`:**

`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `IMetricOmmPoolActions(params.pool).swap(...)` directly, making the router the `msg.sender` seen by the pool. [4](#0-3) 

**Contrast with `DepositAllowlistExtension` (correct pattern):**

The deposit extension ignores `sender` and checks `owner` — the actual position owner passed explicitly by the caller — so it correctly gates the economic actor even through `MetricOmmPoolLiquidityAdder`. The swap extension has no equivalent "end-user" identity argument. [5](#0-4) 

**Exploit flow:**
1. Pool admin deploys pool with `SwapAllowlistExtension` as `beforeSwap` hook.
2. Admin calls `setAllowedToSwap(pool, userA, true)` — only `userA` is allowed.
3. Admin calls `setAllowedToSwap(pool, router, true)` — required for `userA` to use the standard periphery.
4. `userB` (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
5. Router calls `pool.swap(recipient, ...)` with `msg.sender = router`.
6. Pool calls `_beforeSwap(sender=router, ...)`.
7. `SwapAllowlistExtension` checks `allowedSwapper[pool][router] == true` → **PASSES**.
8. `userB`'s swap executes against the LP pool despite never being added to the allowlist.

The admin faces a binary choice: allowlist the router (nullifying the per-user gate for all users) or don't (making the router unusable for this pool). There is no configuration that achieves "only allowlisted users may swap through the router."

## Impact Explanation
A pool admin who deploys `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., KYC-verified market makers, institutional partners, or a closed beta) must allowlist the router for those users to access the pool via the standard periphery. Once the router is allowlisted, any unprivileged address can execute swaps against the pool, bypassing the per-user gate entirely. LPs in such a pool are exposed to unrestricted swap flow from actors the pool was explicitly designed to exclude, enabling adverse-selection losses against LP principal. This constitutes a broken core access-control mechanism with direct LP fund impact — an admin-boundary break reachable by any unprivileged EOA or contract.

## Likelihood Explanation
Any pool that (a) configures a swap allowlist for access control and (b) allowlists the router to support standard periphery usage is immediately vulnerable. Both conditions are the natural, expected configuration for a restricted pool that still wants to support the standard router UX. The bypass requires no special privilege — any EOA or contract can call `MetricOmmSimpleRouter`. The vulnerability is repeatable on every swap through the router against any such pool.

## Recommendation
The extension must gate the **economic initiator**, not the intermediary. Two viable approaches:

1. **Pass initiator through `extensionData`**: Require the router to encode the original `msg.sender` in `extensionData`; the extension decodes and checks that address. This requires a convention between router and extension.
2. **Document and enforce direct-call-only**: Document that the allowlist only works for direct pool calls (no router), and have the router expose a separate entry point that pools can gate differently.

The deposit extension's pattern — checking `owner` rather than `sender` — is the correct model. The swap extension needs an equivalent "end-user" identity that survives router intermediation. [6](#0-5) 

## Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
2. Admin: setAllowedToSwap(pool, userA, true)   // only userA is allowed
3. Admin: setAllowedToSwap(pool, router, true)   // router must be allowlisted
                                                  // for userA to use it
4. userB (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
   → router calls pool.swap(recipient, ...) with msg.sender = router
   → pool calls _beforeSwap(sender=router, ...)
   → SwapAllowlistExtension checks allowedSwapper[pool][router] == true → PASSES
5. userB's swap executes successfully against the LP pool,
   despite never being added to the allowlist.
```

Foundry test plan: deploy `SwapAllowlistExtension`, configure a pool with it, allowlist only `userA` and the router, then call `exactInputSingle` from `userB` and assert the swap succeeds (no `NotAllowedToSwap` revert).
