### Title
Difficulty Calculation for Fork Blocks Uses Stale Mainchain Boundary Block Instead of Fork Ancestor - (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

### Summary

When verifying PoW for fork-chain block submissions, `get_next_work_required` fetches the difficulty-period boundary block via `get_header_by_height`, which reads exclusively from the `mainchain_height_to_header` mapping. For any fork that diverges before the boundary block height, this returns the **mainchain's** boundary block rather than the **fork's** actual ancestor, producing a wrong expected `bits` value and breaking the PoW difficulty invariant for fork blocks.

### Finding Description

The vulnerability class from the external report is **cross-module desynchronization**: a stored/cached value is consumed in place of a value that must be computed dynamically from the current chain state. The exact same pattern appears here.

`get_header_by_height` is implemented as: [1](#0-0) 

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

This always reads from `mainchain_height_to_header`. Fork blocks are stored only in `headers_pool` (via `store_fork_header`), never in `mainchain_height_to_header`: [2](#0-1) 

All three chain implementations call `get_header_by_height` to obtain the boundary block timestamp for difficulty recalculation:

**Dogecoin** (the most exploitable case — the codebase itself flags this with a `TODO`): [3](#0-2) 

```rust

    .block_header
    .time;
```

**Bitcoin:** [4](#0-3) 

**Litecoin:** [5](#0-4) 

For Dogecoin post-height 145,000, `difficulty_adjustment_interval` is `1`, so `blocks_to_go_back = 1` for every non-genesis block: [6](#0-5) 

This means `height_first = prev_block_header.block_height - 1` — the boundary block is always the block **two positions behind** the submitted fork block. A fork diverging just two blocks back already has a different block at `height_first`, yet the contract reads the mainchain's block there.

The correct approach is to traverse the fork chain backwards from `prev_block_header` to find the true ancestor at `height_first`, analogous to how the reference Dogecoin implementation walks the chain rather than indexing by height.

### Impact Explanation

`check_pow` enforces: [7](#0-6) 

```rust
require!(
    expected_bits == block_header.bits,
    ...
);
```

When `expected_bits` is computed from the wrong boundary block timestamp, the contract either:

1. **Accepts fork blocks with incorrect difficulty** — if the mainchain boundary block's timestamp produces a lower difficulty (higher target) than the fork's actual chain history requires, the attacker can submit fork blocks that satisfy a weaker PoW target. The subsequent hash check `U256::from_le_bytes(&pow_hash.0) <= target_from_bits(header.bits)` then uses this weaker target, making the block cheaper to mine.

2. **Rejects fork blocks with correct difficulty** — if the mainchain boundary block's timestamp produces a higher difficulty than the fork's actual chain history requires, legitimate fork blocks are rejected.

Scenario 1 is the security-critical path: an attacker who controls the fork's boundary block timestamp can cause the contract to enforce a weaker PoW requirement for subsequent fork blocks, enabling a cheaper chain reorganization. A successful reorg corrupts `mainchain_height_to_header` and `mainchain_tip_blockhash`, directly affecting the output of `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` for all downstream consumers. [8](#0-7) 

### Likelihood Explanation

For **Dogecoin** (post-height 145,000): the boundary block is always `prev_height - 1`. Any fork diverging two or more blocks back — a routine occurrence — triggers the bug. No special privileges are required; `submit_blocks` is a public payable entry point accessible to any NEAR account. [9](#0-8) 

For **Bitcoin/Litecoin**: the boundary block is 2016 blocks back, requiring a deeper fork, but the same code path is reachable by any relayer submitting a fork batch.

### Recommendation

Replace `get_header_by_height(height_first)` with a backwards traversal from `prev_block_header` that follows `prev_block_hash` links through `headers_pool` until reaching `height_first`. This mirrors the reference implementations and ensures the correct fork ancestor is used regardless of what the mainchain mapping contains at that height.

### Proof of Concept

1. Deploy the Dogecoin contract variant on NEAR testnet (height > 145,000).
2. Submit a legitimate mainchain sequence up to height N. The mainchain block at height N-2 has timestamp `T_main`.
3. Craft a fork block at height N-2 with timestamp `T_fork` where `T_fork - T_{N-3}` is much smaller than `T_main - T_{N-3}`, which under the Dogecoin DigiShield formula would produce a higher required difficulty for the next block.
4. Submit the fork block at N-2, then a fork block at N-1.
5. Submit a fork block at height N with `bits` matching the difficulty computed from `T_main` (the mainchain's block at N-2), not from `T_fork`.
6. Observe that `check_pow` passes: `expected_bits` is derived from `get_header_by_height(N-2)` which returns the mainchain block (timestamp `T_main`), not the fork block (timestamp `T_fork`). The fork block at height N is accepted with a weaker difficulty than the fork's own chain history requires. [10](#0-9)

```rust
fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
```
