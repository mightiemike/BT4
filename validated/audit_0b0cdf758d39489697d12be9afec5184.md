Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address as swapper, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which the pool populates with its own `msg.sender` — the router contract address for all router-mediated swaps, not the originating user. A pool admin who allowlists the router to enable standard periphery access inadvertently grants every user on-chain access to the restricted pool, completely defeating the per-user allowlist. Conversely, if only individual users are allowlisted, those users cannot swap through the router at all.

## Finding Description

In `MetricOmmPool.swap`, the pool passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this as `sender` to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the pool's caller: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)`, the pool's `msg.sender` is the router contract, not the originating user: [4](#0-3) 

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. This is structurally asymmetric with `DepositAllowlistExtension.beforeAddLiquidity`, which correctly checks `owner` — the second parameter, explicitly representing the economic actor — rather than the intermediary caller: [5](#0-4) 

The `beforeSwap` interface has no equivalent `owner`/`swapper` parameter separate from `sender`; the actual user's address is only available in the router's transient callback context, invisible to the extension: [6](#0-5) 

## Impact Explanation

Two concrete fund-impacting consequences follow:

**1. Complete allowlist bypass:** A pool admin who allowlists the router address — the natural action to let allowlisted users swap via the standard periphery — inadvertently grants every user on-chain access. Any non-allowlisted user can call `MetricOmmSimpleRouter.exactInputSingle`; the extension sees `sender = router`, checks `allowedSwapper[pool][router] == true`, and passes. Unauthorized swaps execute at oracle-derived prices, draining LP assets.

**2. Broken core swap flow for allowlisted users:** If the admin allowlists only specific user addresses (not the router), those users cannot swap through the router at all — the extension sees `sender = router` and reverts with `NotAllowedToSwap`. The standard periphery path is permanently broken for the intended users.

Both consequences meet the Sherlock threshold for broken core pool functionality causing loss of funds or unusable swap flows.

## Likelihood Explanation

The bypass requires the pool admin to allowlist the router. This is a predictable and natural configuration step: a pool admin deploying a restricted pool who wants allowlisted users to use the standard periphery will allowlist the router. The extension provides no mechanism to gate individual users through the router, so the admin has no correct alternative. Once the router is allowlisted, the bypass requires no special access — any unprivileged user routing through `MetricOmmSimpleRouter` bypasses the allowlist.

## Recommendation

The `beforeSwap` hook must gate the actual initiating user, not the intermediary. Two options:

1. **Pass the real user via `extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated convention between router and extension.

2. **Mirror the deposit pattern:** Add a dedicated `swapper` parameter to the swap hook (analogous to `owner` in the deposit hook) that the pool populates with the economically relevant actor. This is a protocol-level change but closes the gap cleanly and consistently.

Until fixed, pool admins must not allowlist the router address in `SwapAllowlistExtension`; allowlisted users must call `pool.swap()` directly.

## Proof of Concept

```
Setup:
  pool deployed with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)       // alice is the intended user
  admin calls setAllowedToSwap(pool, router, true)      // to let alice use the router

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: bob, ...})

  router calls:
    pool.swap(bob, zeroForOne, amount, priceLimit, "", extensionData)
    // pool's msg.sender = router

  pool calls:
    _beforeSwap(sender=router, ...)
    // ExtensionCalling forwards sender=router to extension

  extension checks:
    allowedSwapper[pool][router] == true  ← passes

  Result:
    bob's swap executes at oracle price, draining LP assets
    per-user allowlist is completely bypassed
```

Foundry test plan: deploy a pool with `SwapAllowlistExtension`, allowlist the router, call `exactInputSingle` from an address not in the allowlist, and assert the swap succeeds (bypass confirmed). Then remove the router from the allowlist, add only a specific user, and assert that user's `exactInputSingle` call reverts with `NotAllowedToSwap` (broken flow confirmed).
