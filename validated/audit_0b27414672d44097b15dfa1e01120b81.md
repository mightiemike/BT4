Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual token payer / caller of `addLiquidity`) and evaluates only `owner` (the LP position recipient). Because `MetricOmmPool.addLiquidity` explicitly supports `msg.sender ≠ owner`, any unprivileged address can bypass the allowlist by naming any allowlisted address as `owner`, paying tokens themselves, and having the LP position credited to the allowlisted address. The pool admin's primary access-control mechanism is silently defeated.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as two distinct arguments to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes both into the extension call: [2](#0-1) 

The extension receives both, but the first parameter (`sender`) is unnamed and unused. Only `owner` is evaluated: [3](#0-2) 

The pool's own NatSpec explicitly documents the `sender ≠ owner` operator pattern as a supported and intended use case: [4](#0-3) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (with explicit `owner`) directly exposes this path to any caller: [5](#0-4) 

`SwapAllowlistExtension` does not share this flaw — it correctly checks `sender`: [6](#0-5) 

**Exploit path:**
1. Pool admin deploys pool with `DepositAllowlistExtension` on `BEFORE_ADD_LIQUIDITY_ORDER`.
2. Admin calls `setAllowedToDeposit(pool, alice, true)` — only `alice` is allowlisted; `bob` is not.
3. `bob` calls `pool.addLiquidity(owner=alice, salt=1, deltas=[...], ...)` directly or via `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, alice, ...)`.
4. The extension receives `(sender=bob, owner=alice, ...)`, ignores `bob`, checks `allowedDepositor[pool][alice] == true` → passes.
5. `bob`'s tokens are transferred to the pool; `alice`'s LP position is credited. The allowlist never evaluated `bob`.

## Impact Explanation

This is a direct admin-boundary break: an unprivileged address (`bob`) bypasses the pool admin's configured deposit allowlist. Concrete impacts include: (1) unauthorized token deposits into KYC-gated or restricted pools; (2) pool state manipulation (shifting `curPosInBin`/`curBinIdx`) harming existing LPs through altered swap pricing; (3) an allowlisted depositor using the legitimate operator pattern (`sender ≠ owner`) to credit a position to a non-allowlisted `owner` is incorrectly rejected, breaking the intended use case. The wrong value is the extension's access-control decision (`allowedDepositor[pool][owner]` instead of `allowedDepositor[pool][sender]`).

## Likelihood Explanation

No special privileges, flash loans, or oracle manipulation are required. The `owner ≠ sender` pattern is publicly documented in the pool interface NatSpec and directly exposed by `MetricOmmPoolLiquidityAdder`. Any attacker who reads the periphery interface or tests discovers the bypass immediately. A single direct call to `pool.addLiquidity` suffices; the attack is repeatable at will.

## Recommendation

Replace the unnamed first parameter with `sender` and check it instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the payer and the position recipient, both `sender` and `owner` should be checked. The `setAllowedToDeposit` setter and `isAllowedToDeposit` view should be reviewed to confirm which actor the pool admin intends to control.

## Proof of Concept

```
Setup:
  - Pool deployed with DepositAllowlistExtension on BEFORE_ADD_LIQUIDITY_ORDER
  - pool.admin calls setAllowedToDeposit(pool, alice, true)
  - bob is NOT allowlisted

Attack (Foundry test):
  vm.prank(bob);
  pool.addLiquidity(
      alice,          // owner = allowlisted address
      1,              // salt
      deltas,         // any valid LiquidityDelta
      callbackData,
      extensionData
  );
  // → _beforeAddLiquidity(sender=bob, owner=alice, ...)
  // → DepositAllowlistExtension.beforeAddLiquidity(_, alice, ...)
  // → allowedDepositor[pool][alice] == true → passes
  // → bob's tokens transferred; alice's LP position credited
  // → bob, not on the allowlist, successfully deposited

Assert: bob's token balances decreased; alice's position shares increased;
        no NotAllowedToDeposit revert was triggered.
```
