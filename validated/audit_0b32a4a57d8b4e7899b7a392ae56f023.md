### Title
Linked Signer Can Hijack Its Own Delegation via `LinkSigner` with `allowLinkedSigner = true` - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction handler passes `allowLinkedSigner = true` to `validateSignedTx`. This means the current linked signer — a delegated key with limited intended authority — can sign a new `LinkSigner` transaction to reassign the linked signer to any arbitrary address, including a malicious one. The account owner's exclusive right to manage their own linked signer is broken.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true   // <-- allowLinkedSigner
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which in turn calls `verifier.validateSignature` passing `getLinkedSigner(sender)` as the accepted signer:

```solidity
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

`verifier.validateSignature` accepts the signature if `recovered == linkedSigner`:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

Therefore, the current linked signer can sign a `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with any address it chooses.

By contrast, the slow-mode `LinkSigner` handler correctly uses `validateSender`, which enforces that only the actual EOA owner of the subaccount can submit the transaction:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    ...
    validateSender(txn.sender, sender);
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The fast-mode path has no equivalent ownership check — it accepts the linked signer's signature as sufficient authority to reassign itself.

---

### Impact Explanation

Once a linked signer reassigns `linkedSigners[subaccount]` to an attacker-controlled address, that address can sign any transaction that accepts `allowLinkedSigner = true`, including:

- `WithdrawCollateral` / `WithdrawCollateralV2` — drain all collateral from the subaccount
- `LiquidateSubaccount` — trigger liquidations against the victim
- `TransferQuote` — transfer quote balance to any recipient
- `MintNlp` / `BurnNlp` — manipulate NLP positions [5](#0-4) 

The account owner loses control of their subaccount. All assets are at risk of theft.

---

### Likelihood Explanation

Any address that has ever been granted linked-signer status for a subaccount can exploit this immediately. No special conditions, governance capture, or admin access is required. The attacker-controlled entry path is: sign a valid `LinkSigner` EIP-712 message as the current linked signer, submit it through the sequencer's fast-mode path (`processTransactionImpl`), and the `linkedSigners` mapping is overwritten.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction handler in `processTransactionImpl`. The linked signer is a delegated key and must not have the authority to reassign delegation — only the subaccount owner (the primary key) should be able to do so:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only the account owner may reassign the linked signer
);
``` [6](#0-5) 

---

### Proof of Concept

1. Alice owns subaccount `alice_subaccount` and sets linked signer to `botKey` (a trading bot).
2. `botKey` constructs a `LinkSigner` EIP-712 message: `{ sender: alice_subaccount, signer: attackerKey, nonce: N }`.
3. `botKey` signs the message and submits it to the sequencer.
4. The sequencer calls `processTransactionImpl` → `validateSignedTx(..., true)` → `verifier.validateSignature(alice_subaccount, botKey, digest, sig)` — passes because `recovered == botKey == linkedSigner`.
5. `linkedSigners[alice_subaccount]` is now set to `attackerKey`.
6. `attackerKey` signs a `WithdrawCollateral` transaction for Alice's full balance. It passes `validateSignature` because `recovered == attackerKey == linkedSigner`.
7. Alice's collateral is drained. [1](#0-0) [7](#0-6)
