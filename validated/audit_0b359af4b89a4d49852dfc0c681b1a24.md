### Title
`L1Gas` V3 Transaction Hash Computed with 2 Resource Bounds in Rust While Cairo OS Unconditionally Asserts 3 — (`crates/starknet_api/src/transaction_hash.rs`)

### Summary

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` produces a 2-element Poseidon hash `(tip, L1_GAS, L2_GAS)` for `ValidResourceBounds::L1Gas` V3 transactions, while the Cairo OS `hash_fee_fields` in `transaction_hash.cairo` unconditionally asserts `n_resource_bounds = 3`. A `L1Gas` V3 transaction submitted through the deprecated gateway is admitted and executed by the blockifier, but when the OS attempts to prove the block it aborts because the resource-bounds count diverges from the expected value of 3. This is the direct sequencer analog of the external M-04 bug: one side of a version boundary declares a shorter/different return shape than the other side expects, causing every transaction that crosses that boundary to fail.

---

### Finding Description

**Rust side — `get_tip_resource_bounds_hash`** [1](#0-0) 

For `ValidResourceBounds::L1Gas`, the function builds a 2-element resource vector `[L1_GAS, L2_GAS]` and omits `L1_DATA_GAS`:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],          // ← only 2 elements total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
```

The resulting hash is `poseidon(tip, L1_GAS_packed, L2_GAS_packed)`.

**Cairo OS side — `hash_fee_fields`** [2](#0-1) 

```cairo
with_attr error_message("Invalid number of resource bounds: {n_resource_bounds}.") {
    assert n_resource_bounds = 3;   // ← hard assertion, no version gate
}
```

The OS always expects exactly 3 resource bounds (L1_GAS, L2_GAS, L1_DATA_GAS). If the hint supplies `n_resource_bounds = 2` for a `L1Gas` transaction, the assertion aborts. If the hint pads to 3 (adding a zero L1_DATA_GAS), the OS computes `poseidon(tip, L1_GAS, L2_GAS, 0)`, which differs from the Rust-computed hash, causing the subsequent `AssertTransactionHash` hint to fail. **Admission path — deprecated gateway accepts `L1Gas` V3 transactions**

`InvokeV3Transaction` in the writer objects uses `DeprecatedResourceBoundsMapping`: [4](#0-3) 

The conversion `ValidResourceBounds::try_from(DeprecatedResourceBoundsMapping)` produces `L1Gas` whenever `L1DataGas` is absent and `L2Gas` is zero: [5](#0-4)

The blockifier executes `L1Gas` transactions without error (fee validation handles both variants). The divergence surfaces only when the OS attempts to prove the block.

### Impact Explanation

This matches **High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload**.

The transaction hash committed by the sequencer for a `L1Gas` V3 transaction is `poseidon(…, tip+L1_GAS+L2_GAS)` (2 resource bounds). The OS hash domain requires `poseidon(…, tip+L1_GAS+L2_GAS+L1_DATA_GAS)` (3 resource bounds). The two values are cryptographically distinct. The OS therefore either aborts on the `n_resource_bounds = 3` assertion or rejects the hash via `AssertTransactionHash`. Either path makes the block unprovable, binding the wrong hash to the accepted transaction.

---

### Likelihood Explanation

Any client that omits `L1_DATA_GAS` from a V3 invoke's resource bounds (the pre-0.13.3 wire format) and submits via the deprecated HTTP gateway will trigger this path. No authentication or privilege is required. The deprecated gateway endpoint remains active for backward compatibility.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
