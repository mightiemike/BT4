### Title
`getTierFeeRateX18` Returns Zero Fees for Any (Tier, Product) Pair Where the Tier Was Activated on a Different Product — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`getTierFeeRateX18` uses a single per-tier bitmask (`nonDefaultFeeTierMask`) to decide whether to return a stored fee rate or the protocol default. Once a tier is activated for **any** product, the mask bit is set and the function returns the raw (zero-initialized) storage value for every other product that was never explicitly configured for that tier. This is a direct structural analog to M-02: an incomplete coverage in a tiered pricing lookup causes certain inputs to silently resolve to zero cost.

---

### Finding Description

`updateTierFeeRates` has two code paths:

```solidity
// core/contracts/OffchainExchange.sol  lines 962–991
if (txn.productId == QUOTE_PRODUCT_ID) {
    // iterates all current spot + perp products and sets feeRates[tier][each]
} else {
    feeRates[txn.tier][txn.productId] = FeeRates(txn.makerRateX18, txn.takerRateX18);
}
nonDefaultFeeTierMask |= uint128(1) << txn.tier;   // ← always sets the bit
``` [1](#0-0) 

Regardless of which path is taken, the tier bit is unconditionally set in `nonDefaultFeeTierMask`. The lookup then is:

```solidity
// lines 933–946
if (nonDefaultFeeTierMask & (1 << tier) != 0) {
    return feeRates[tier][productId];   // returns {0, 0} if never written
}
return FeeRates({ makerRateX18: 0, takerRateX18: 200_000_000_000_000 }); // 2 bps default
``` [2](#0-1) 

Once the bit for tier `T` is set (by configuring it for even a single product), every call to `getTierFeeRateX18(T, productId)` for any product that was **never explicitly written** returns the zero-initialized `FeeRates({0, 0})` — bypassing the 2 bps default entirely.

The two concrete triggers are:

1. **Per-product tier configuration**: Admin calls `updateTierFeeRates` with a specific `productId` (not `QUOTE_PRODUCT_ID`). The tier bit is set, but only one product gets a rate. All other products for that tier return `{0, 0}`.

2. **New product added after tier setup**: Admin uses `QUOTE_PRODUCT_ID` to configure tier `T` for all products at time T₀. A new product is listed at T₁ > T₀. `feeRates[T][newProduct]` is never written, but the bit for tier `T` is already set, so `getTierFeeRateX18(T, newProduct)` returns `{0, 0}`.

The zero fee rate flows directly into `applyFee`:

```solidity
int128 keepRateX18 = ONE - feeInfo.feeRate;   // = ONE when feeRate == 0
int128 newMeteredQuote = meteredQuote.mul(keepRateX18);  // = meteredQuote
orderInfo.fee = meteredQuote - newMeteredQuote;  // = 0
``` [3](#0-2) 

The user pays zero taker fees on the affected product.

---

### Impact Explanation

Any user assigned to a non-default fee tier can trade on any product not explicitly configured for their tier and pay **zero taker fees** instead of the 2 bps default. The protocol's fee revenue for those trades is entirely lost. Because `collectedFees` for the product is never incremented, the shortfall is permanent and unrecoverable. The impact scales with trading volume on affected products.

---

### Likelihood Explanation

High. The protocol is designed to list new products over time. Any product added after a tier has been configured will be uncovered. Scenario 2 (new product after tier setup) requires only normal operational actions — no unusual admin behavior. Scenario 1 is triggered whenever the admin configures a tier for a single product rather than using the `QUOTE_PRODUCT_ID` broadcast path. Users assigned to non-default tiers can observe this by calling the public `getTierFeeRateX18` view function and identifying products that return `{0, 0}`, then directing their order flow there.

---

### Recommendation

The `nonDefaultFeeTierMask` must be indexed per `(tier, productId)`, not per tier alone. Replace the single `uint128` bitmask with a `mapping(uint32 => mapping(uint32 => bool)) internal tierProductConfigured` and gate the lookup on `tierProductConfigured[tier][productId]` rather than the tier-only bit. Alternatively, when the bit for a tier is set but `feeRates[tier][productId]` is zero-initialized (both fields zero), fall back to the protocol default rather than returning `{0, 0}`.

---

### Proof of Concept

1. Admin calls `updateTierFeeRates(tier=1, productId=1, makerRate=0, takerRate=50bps)`. `nonDefaultFeeTierMask` bit 1 is set; `feeRates[1][1]` is written.
2. Admin calls `updateFeeTier(userA, 1)`. User A is now on tier 1.
3. Protocol lists product 2 (new market). `feeRates[1][2]` is never written.
4. User A submits a taker order on product 2. `getTierFeeRateX18(1, 2)` checks `nonDefaultFeeTierMask & 2 != 0` → true → returns `feeRates[1][2]` = `{0, 0}`.
5. `applyFee` computes `fee = 0`. User A pays no taker fee. Protocol receives nothing.
6. User A can repeat this for every newly listed product indefinitely. [4](#0-3) [1](#0-0)
