### Title
`getSlowModeFee` Reverts for Quote Tokens with Fewer Than 6 Decimals Due to Arithmetic Underflow — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.getSlowModeFee()` unconditionally computes `token.decimals() - 6` as a `uint8` subtraction. If the quote token has fewer than 6 decimals, Solidity 0.8+ checked arithmetic causes an underflow panic, reverting every call. This permanently blocks slow-mode transaction submission for any deployment whose quote token has fewer than 6 decimals.

---

### Finding Description

`getSlowModeFee()` is responsible for returning the slow-mode fee denominated in the quote token's native units. The constant `SLOW_MODE_FEE = 1_000_000` is defined as "$1" in 6-decimal precision. The function scales it to the quote token's actual decimals:

```solidity
// Clearinghouse.sol L759-L766
function getSlowModeFee() external view returns (uint256) {
    ISpotEngine spotEngine = _spotEngine();
    IERC20Base token = IERC20Base(
        spotEngine.getConfig(QUOTE_PRODUCT_ID).token
    );
    int256 multiplier = int256(10**(token.decimals() - 6));  // ← underflows if decimals < 6
    return uint256(int256(SLOW_MODE_FEE) * multiplier);
}
```

`token.decimals()` returns `uint8`. In Solidity ≥ 0.8, the expression `uint8(x) - 6` where `x < 6` triggers a checked-arithmetic underflow panic and reverts. There is no guard (e.g., `require(decimals >= 6)`) before this subtraction.

`SLOW_MODE_FEE` is defined as `1000000` (representing $1) in `Constants.sol`:

```solidity
// Constants.sol L23
int128 constant SLOW_MODE_FEE = 1000000; // $1
```

The design assumes the quote token has at least 6 decimals (e.g., USDC = 6). Any quote token with fewer than 6 decimals (e.g., a 2-decimal stablecoin like GUSD) causes `getSlowModeFee()` to always revert.

---

### Impact Explanation

`getSlowModeFee()` is called from `EndpointStorage` as part of the slow-mode transaction submission path — the censorship-resistance mechanism that allows users to bypass the sequencer. If this function reverts, every call to submit a slow-mode transaction reverts, permanently disabling the slow-mode path for the affected deployment. Users lose their only recourse against sequencer censorship.

---

### Likelihood Explanation

The quote token is set once at `Clearinghouse.initialize()`. If a deployment uses a quote token with fewer than 6 decimals, the bug is triggered unconditionally from that point forward. While USDC (6 decimals) is the expected quote token and is not affected (6 − 6 = 0, no underflow), the code contains no enforcement that the quote token must have ≥ 6 decimals. Any deployment or future upgrade that uses a lower-decimal quote token silently breaks the slow-mode path.

---

### Recommendation

Add a guard in `getSlowModeFee()` (or at product registration time) to ensure the quote token has at least 6 decimals, and handle the sub-6-decimal case explicitly:

```solidity
function getSlowModeFee() external view returns (uint256) {
    ISpotEngine spotEngine = _spotEngine();
    IERC20Base token = IERC20Base(
        spotEngine.getConfig(QUOTE_PRODUCT_ID).token
    );
    uint8 decimals = token.decimals();
    if (decimals >= 6) {
        int256 multiplier = int256(10**(decimals - 6));
        return uint256(int256(SLOW_MODE_FEE) * multiplier);
    } else {
        // divide down for sub-6-decimal tokens
        int256 divisor = int256(10**(6 - decimals));
        return uint256(int256(SLOW_MODE_FEE) / divisor);
    }
}
```

Alternatively, enforce `require(decimals >= 6)` at the point the quote token is registered.

---

### Proof of Concept

1. Deploy `Clearinghouse` with a quote token that has 2 decimals (e.g., a GUSD-like token).
2. Call `clearinghouse.getSlowModeFee()`.
3. The call reverts with an arithmetic underflow panic because `uint8(2) - 6` wraps under Solidity 0.8 checked math.
4. Any user attempting to submit a slow-mode transaction (which internally calls `getSlowModeFee()`) will always receive a revert, permanently disabling the censorship-resistance path.
