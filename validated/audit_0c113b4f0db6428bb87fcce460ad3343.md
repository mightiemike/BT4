### Title
Gateway `allow_client_side_proving` config flag accepts proof-bearing invoke transactions that blockifier always rejects when `allowed_proof_versions` is empty — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The gateway's stateless validator exposes a static config flag `allow_client_side_proving` that, when `true` (the production default), passes invoke V3 transactions carrying non-empty `proof_facts` through admission. The blockifier's `validate_proof_facts` independently enforces a protocol-version gate via `os_constants.allowed_proof_versions` from `VersionedConstants`. For Starknet v0.14.0, that list is an empty array. Because the gateway never consults `VersionedConstants` before admitting proof-bearing transactions, any such transaction accepted by the gateway on a v0.14.0 node will be unconditionally rejected by the blockifier at pre-validation — a permanent DoS on the client-side proving feature identical in structure to the `depositFromNotional`/`executeTrade` payable mismatch.

### Finding Description

**Layer 1 — Gateway admission (no version check):**

`StatelessTransactionValidator::validate_client_side_proving_allowed` short-circuits to `Ok(())` the moment `self.config.allow_client_side_proving` is `true`, with no reference to the current Starknet version or `VersionedConstants`:

```rust
// crates/apollo_gateway/src/stateless_transaction_validator.rs:231-247
fn validate_client_side_proving_allowed(
    &self,
    tx: &RpcInvokeTransaction,
) -> StatelessTransactionValidatorResult<()> {
    if self.config.allow_client_side_proving {
        return Ok(());   // ← unconditional pass; no VersionedConstants check
    }
    let RpcInvokeTransaction::V3(tx) = tx;
    let has_proof_data = !tx.proof_facts.is_empty() || !tx.proof.is_empty();
    if has_proof_data {
        return Err(StatelessTransactionValidatorError::ClientSideProvingNotAllowed);
    }
    Ok(())
}
```

The production deployment config sets this flag to `true`:

```json
// crates/apollo_deployments/resources/app_configs/gateway_config.json:21
"gateway_config.static_config.stateless_tx_validator_config.allow_client_side_proving": true
```

**Layer 2 — Blockifier pre-validation (version-gated):**

`AccountTransaction::validate_proof_facts` checks `os_constants.allowed_proof_versions` and `os_constants.allowed_virtual_os_program_hashes` from the block's `VersionedConstants`:

```rust
// crates/blockifier/src/transaction/account_transaction.rs:314-328
if !os_constants.allowed_proof_versions.contains(&snos_proof_facts.proof_version.as_felt())
{
    return Err(TransactionPreValidationError::InvalidProofFacts(format!(
        "Proof version {} is not allowed under this protocol version.",
        snos_proof_facts.proof_version
    )));
}
let allowed = &os_constants.allowed_virtual_os_program_hashes;
if !allowed.contains(&snos_proof_facts.program_hash) {
    return Err(TransactionPreValidationError::InvalidProofFacts(format!(
        "Virtual OS program hash {} is not allowed",
        snos_proof_facts.program_hash
    )));
}
```

**Version-specific constants — the empty gate:**

For Starknet v0.14.0, both lists are empty arrays, meaning every proof-bearing transaction will unconditionally fail the blockifier check:

```json
// crates/blockifier/resources/blockifier_versioned_constants_0_14_0.json:129-130
"allowed_virtual_os_program_hashes": [],
"allowed_proof_versions": [],
```

Client-side proving was only enabled starting from v0.14.2:

```json
// crates/blockifier/resources/blockifier_versioned_constants_0_14_2.json:129-134
"allowed_virtual_os_program_hashes": [
    "0x3e98c2d7703b03a7edb73ed7f075f97f1dcbaa8f717cdf6e1a57bf058265473"
],
"allowed_proof_versions": [
    "0x50524f4f4630"
],
```

**The divergence:**

| Layer | Result for proof-bearing invoke V3 on v0.14.0 |
|---|---|
| Gateway (`allow_client_side_proving: true`) | `Ok(())` — transaction admitted to mempool |
| Blockifier `validate_proof_facts` | `Err(InvalidProofFacts("Proof version … is not allowed under this protocol version."))` |

The transaction is accepted at the admission boundary and unconditionally rejected at execution — the exact structural analog of `depositFromNotional` (payable, accepts ETH) delegating to `executeTrade` (non-payable, always reverts when `msg.value > 0`).

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

Any unprivileged user can submit an invoke V3 transaction with syntactically valid proof facts to a node running Starknet v0.14.0 with the default gateway config. The gateway admits the transaction; the blockifier rejects it at pre-validation. The transaction consumes mempool slots and sequencer resources without ever being sequenced. Client-side proving is permanently broken for all users on that node, regardless of proof correctness.

### Likelihood Explanation

The default `StatelessTransactionValidatorConfig` sets `allow_client_side_proving: true` and the production deployment config mirrors this. Any operator who deploys the node against a Starknet v0.14.0 chain without explicitly disabling the flag will exhibit this behavior. No privileged access is required; any wallet that constructs an invoke V3 transaction with a non-empty `proof_facts` field triggers the divergence.

### Recommendation

`validate_client_side_proving_allowed` must consult the current protocol version's `VersionedConstants` before admitting proof-bearing transactions. Concretely, the gateway should reject any transaction whose `proof_facts` carries a `proof_version` not present in `os_constants.allowed_proof_versions` for the current Starknet version, or whose `program_hash` is not in `os_constants.allowed_virtual_os_program_hashes`. This mirrors the existing pattern used for `nonce_data_availability_mode` and `fee_data_availability_mode`, where the gateway proactively enforces OS-level constraints to prevent transactions from failing downstream.

### Proof of Concept

1. Deploy the sequencer node against a Starknet v0.14.0 chain with the default gateway config (`allow_client_side_proving: true`).
2. Construct an invoke V3 transaction with a non-empty `proof_facts` field (e.g., using the SNOS proof facts format with any `proof_version` felt).
3. Submit the transaction to the gateway's `add_transaction` endpoint.
4. **Gateway response**: `200 OK` — transaction admitted; `validate_client_side_proving_allowed` returns `Ok(())` because `allow_client_side_proving: true`.
5. **Blockifier response** (at block execution): `TransactionPreValidationError::InvalidProofFacts("Proof version 0x50524f4f4630 is not allowed under this protocol version.")` — because `blockifier_versioned_constants_0_14_0.json` has `"allowed_proof_versions": []`.
6. The transaction is permanently stuck: accepted by the gateway, always rejected by the blockifier. Client-side proving is a dead feature on this node.
