### Title
`op_bls_pairing_identity` Accepts Zero Arguments and Returns Success Without Any Pairing Verification — (File: `src/bls_ops.rs`)

---

### Summary

`op_bls_pairing_identity` in `src/bls_ops.rs` has no guard requiring at least one `(G1, G2)` pair. When invoked with zero arguments the iteration loop is skipped entirely, `aggregate_pairing` is called with an empty vector (which returns `true` by the empty-product identity of the pairing group), and the operator returns `nil` — success — without performing any cryptographic check. This is a direct structural analog to the SafEth `stake()` bug: a function that accepts input, iterates over a collection, and when the collection is empty silently succeeds instead of rejecting the call.

---

### Finding Description

In `src/bls_ops.rs`, `op_bls_pairing_identity` (lines 332–358) collects `(G1, G2)` pairs with:

```rust
while !nilp(a, args) {
    cost += BLS_PAIRING_COST_PER_ARG;
    check_cost(cost, max_cost)?;
    let g1 = a.g1(first(a, args)?)?;
    args = rest(a, args)?;
    let g2 = a.g2(first(a, args)?)?;
    args = rest(a, args)?;
    items.push((g1, g2));
}
```

When the argument list is `nil` (zero arguments), `nilp(a, args)` is immediately `true`, the loop body never executes, `items` remains an empty `Vec<(G1Element, G2Element)>`, and the call falls through to:

```rust
if !aggregate_pairing(items) {
    Err(EvalErr::BLSPairingIdentityFailed(input))?
} else {
    Ok(Reduction(cost, a.nil()))
}
```

In standard BLS cryptography the empty product of pairings equals the identity element in the target group GT, so `aggregate_pairing([])` returns `true`. The operator therefore returns `Ok(Reduction(BLS_PAIRING_BASE_COST, nil))` — success — having verified nothing.

There is no guard of the form:

```rust
if items.is_empty() { return Err(...); }
```

anywhere before the `aggregate_pairing` call.

The same structural gap exists in `op_bls_g1_subtract` (lines 52–78) and `op_bls_g2_subtract` (lines 165–191): both silently return the group identity element when called with zero arguments, because `is_first` is never set to `false` and `total` is never assigned from an actual argument. ---

### Impact Explanation

Any CLVM puzzle that uses `bls_pairing_identity` for signature verification and sources its arguments from an attacker-controlled solution can be bypassed. The attacker passes an empty solution list `()`; the operator returns `nil` (success) without verifying any pairing. This allows spending coins protected by such puzzles without supplying any valid BLS signature — a consensus-level signature-verification bypass leading to unauthorized coin spending.

The operator is wired as opcode `58` in `ChiaDialect.op()` and is available unconditionally in all CLVM programs (not gated behind a softfork guard), so the attack surface is the full set of on-chain puzzles. [4](#0-3) 

---

### Likelihood Explanation

Moderate. The vulnerability requires a puzzle that passes arguments to `bls_pairing_identity` from the attacker-controlled solution. Multi-signature puzzles and aggregated-signature puzzles that accept a variable number of `(pk, sig)` pairs from the solution are realistic targets. The attacker-controlled entry path is direct: craft a CLVM solution that is the empty list `()`, causing `bls_pairing_identity` to be invoked with zero arguments via attacker-supplied CLVM bytes.

---

### Recommendation

Add a guard check before calling `aggregate_pairing` to require at least one `(G1, G2)` pair:

```rust
if items.is_empty() {
    return Err(EvalErr::InvalidOpArg(
        input,
        "bls_pairing_identity requires at least one (G1, G2) pair".to_string(),
    ));
}
```

Apply the same fix to `op_bls_g1_subtract` and `op_bls_g2_subtract`: return an error when called with zero arguments rather than silently returning the group identity.

---

### Proof of Concept

The following CLVM program passes its entire solution as the argument list to `bls_pairing_identity`:

```
(bls_pairing_identity . 1)   ; opcode 58; "1" evaluates to the full solution
```

Run with solution `()` (empty list):

1. `bls_pairing_identity` is dispatched to `op_bls_pairing_identity` in `src/bls_ops.rs`.
2. `input` is `nil`; `nilp(a, args)` is immediately `true`.
3. The `while` loop body is never entered; `items` is `[]`.
4. `aggregate_pairing([])` returns `true` (empty-product identity).
5. The operator returns `Ok(Reduction(3_000_000, nil))` — success — with no signature checked.

Any puzzle of the form `(assert (bls_pairing_identity . solution_args))` is therefore satisfiable with the empty solution, bypassing all BLS signature enforcement. [5](#0-4)

pub fn op_bls_pairing_identity(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let mut cost = BLS_PAIRING_BASE_COST;
    check_cost(cost, max_cost)?;
    let mut items = Vec::<(G1Element, G2Element)>::new();

    let mut args = input;
    while !nilp(a, args) {
        cost += BLS_PAIRING_COST_PER_ARG;
        check_cost(cost, max_cost)?;
        let g1 = a.g1(first(a, args)?)?;
        args = rest(a, args)?;
        let g2 = a.g2(first(a, args)?)?;
        args = rest(a, args)?;
        items.push((g1, g2));
    }

    if !aggregate_pairing(items) {
        Err(EvalErr::BLSPairingIdentityFailed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
}
