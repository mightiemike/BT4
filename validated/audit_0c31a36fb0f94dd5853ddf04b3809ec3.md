### Title
`DISABLE_OP` in `MEMPOOL_MODE` Completely Blocks `op_modpow` (Opcode 60) in Mempool While It Remains Valid in Consensus, Creating a Divergence That Permanently Prevents Spending of Coins Using `modpow` Through the Mempool — (`src/chia_dialect.rs`)

---

### Summary

The `MEMPOOL_MODE` constant in `src/chia_dialect.rs` includes `ClvmFlags::DISABLE_OP`, which completely disables `op_modpow` (opcode 60) in mempool mode. However, `op_modpow` is a fully wired, unconditionally available operator in consensus mode — it carries no soft-fork guard and no activation flag. This creates a concrete consensus divergence: a coin whose puzzle uses `modpow` is valid on-chain (consensus accepts it) but its spend transaction is permanently rejected by every mempool node. The coin becomes unspendable through normal propagation paths, directly analogous to the H-01 pattern of a guard blocking the only available operation path with no built-in alternative.

---

### Finding Description

**Root cause — `MEMPOOL_MODE` wires in `DISABLE_OP`:** [1](#0-0) 

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)   // ← blocks modpow entirely
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

**Guard in the opcode dispatch — opcode 60 returns `Unimplemented` when `DISABLE_OP` is set:** [2](#0-1) 

```rust
60 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;
    }
    op_modpow
}
```

**Consensus mode — opcode 60 is unconditionally dispatched to `op_modpow` with no activation flag:** [3](#0-2) 

Every other conditionally-enabled operator (keccak256 at opcode 62, sha256_tree at 63, secp ops at 64–65) is gated by a positive flag (`ENABLE_KECCAK_OPS_OUTSIDE_GUARD`, `ENABLE_SHA256_TREE`, `ENABLE_SECP_OPS`). Opcode 60 is the only operator that is **present by default in consensus** but **negatively gated out in mempool** by `DISABLE_OP`. There is no soft-fork guard, no `OperatorSet` extension, and no alternative opcode path for modular exponentiation.

The `DISABLE_OP` flag also applies a 2048-byte operand cap to `op_div`, `op_divmod`, and `op_mod`, but those operators are merely restricted — they are not eliminated. `op_modpow` is the only operator that is completely removed from the reachable operator set in mempool mode. [4](#0-3) 

---

### Impact Explanation

The divergence has two concrete consequences:

1. **Coin lock-up.** A coin whose puzzle tree evaluates `modpow` (opcode 60) can be created on-chain — puzzle execution does not occur at creation time. When the owner later submits a spend bundle, every full node running in `MEMPOOL_MODE` evaluates the puzzle and hits `EvalErr::Unimplemented` at opcode 60. The spend is rejected by the entire mempool layer. The coin cannot be spent through any normal propagation path. The only escape is a farmer who bypasses the mempool entirely and includes the spend directly in a block — a path unavailable to ordinary users.

2. **Consensus divergence.** A block produced by such a farmer is accepted by all nodes in consensus mode (no `DISABLE_OP`), but the same spend would have been rejected by every mempool. This is a direct mempool/consensus split on a defined, non-soft-forked operator.

The analog to H-01 is exact: the `_ensureNotSystem` guard blocked the only call path to `claimSurplusCollShares`, leaving surplus collateral permanently stuck. Here, `DISABLE_OP` blocks the only execution path for `modpow` in mempool mode, leaving coins with `modpow` puzzles permanently unspendable through normal means.

---

### Likelihood Explanation

The trigger is any attacker-controlled or user-authored CLVM program that contains opcode 60 (`modpow`) anywhere in its puzzle tree. The operator is documented, costed, and fully functional in consensus mode. A developer writing a Chialisp puzzle that uses `modpow` for cryptographic or arithmetic purposes would have no indication from the consensus-mode API that the puzzle is mempool-incompatible. The divergence is reachable by any spend bundle submitted to a standard full node running `MEMPOOL_MODE`.

---

### Recommendation

The asymmetry must be resolved in one of two ways:

1. **If `modpow` is intended to be a live consensus operator:** Remove `DISABLE_OP` from `MEMPOOL_MODE`, or replace the blanket `DISABLE_OP` guard on opcode 60 with a positive activation flag (e.g., `ENABLE_MODPOW`) that is set in both consensus and mempool modes when the operator is active. This mirrors the pattern used for keccak256, sha256_tree, and secp ops.

2. **If `modpow` is not yet activated:** Move it behind a `softfork_extension` guard (like keccak256 at extension 1) so it is unreachable in both mempool and consensus until the fork activates. The current state — reachable in consensus, blocked in mempool — is the worst possible configuration.

The `DISABLE_OP` flag name is also misleading because it has two unrelated effects (complete removal of `modpow` vs. operand-size capping of `div`/`divmod`/`mod`). These should be separated into distinct, self-documenting flags.

### Proof of Concept

```rust
use clvmr::{Allocator, run_program};
use clvmr::chia_dialect::{ChiaDialect, ClvmFlags, MEMPOOL_MODE};

fn main() {
    let mut allocator = Allocator::new();

    // Build (modpow 3 10 7) — opcode 60 with three atom arguments
    // Expected result: 3^10 mod 7 = 4
    let nil = allocator.nil();
    let seven  = allocator.new_small_number(7).unwrap();
    let ten    = allocator.new_small_number(10).unwrap();
    let three  = allocator.new_small_number(3).unwrap();
    let args   = allocator.new_pair(seven, nil).unwrap();
    let args   = allocator.new_pair(ten, args).unwrap();
    let args   = allocator.new_pair(three, args).unwrap();
    let op60   = allocator.new_small_number(60).unwrap(); // modpow opcode
    let program = allocator.new_pair(op60, args).unwrap();
    let env    = allocator.nil();

    // Consensus mode: succeeds, returns 4
    let consensus = ChiaDialect::new(ClvmFlags::empty());
    let result = run_program(&mut allocator, &consensus, program, env, u64::MAX);
    assert!(result.is_ok(), "consensus must accept modpow");

    // Mempool mode: fails with EvalErr::Unimplemented — coin is stuck
    let mempool = ChiaDialect::new(MEMPOOL_MODE);
    let result = run_program(&mut allocator, &mempool, program, env, u64::MAX);
    assert!(result.is_err(), "mempool incorrectly rejects valid modpow");
    // EvalErr::Unimplemented — no alternative path exists
}
```

The divergence is triggered by any CLVM bytes containing opcode `0x3c` (60) in a callable position. The entry path is fully attacker-controlled: any spend bundle submitted to a node running `MEMPOOL_MODE` that evaluates a puzzle containing `modpow` will hit this guard. The corrupted result is `EvalErr::Unimplemented` in mempool vs. a valid `Reduction` in consensus — a concrete, reproducible consensus split on a defined operator.
