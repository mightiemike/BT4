### Title
`ValidResourceBounds` Protobuf Round-Trip Collapses `AllResources` to `L1Gas` Variant, Producing a Different Transaction Hash Preimage — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` uses a zero-value heuristic to decide whether to reconstruct a `ValidResourceBounds::L1Gas` (pre-0.13.3) or `ValidResourceBounds::AllResources` (post-0.13.3) variant. When a post-0.13.3 V3 transaction carries `AllResources` with `l2_gas = 0` and `l1_data_gas = 0`, the serializer emits all three fields as zero, and the deserializer silently collapses the result back to `L1Gas`. Because `get_tip_resource_bounds_hash` hashes a different number of resource felts for each variant, the transaction hash computed after the protobuf round-trip is structurally different from the hash the signer committed to, breaking signature verification for any node that re-derives the hash from the deserialized object.

---

### Finding Description

**Serialization path** (`From<ValidResourceBounds> for protobuf::ResourceBounds`, line 471–490):

Both variants emit all three protobuf fields. For `AllResources` with zero `l2_gas` and `l1_data_gas`, the wire bytes are identical to what `L1Gas` would emit. [1](#0-0) 

**Deserialization path** (`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, line 417–437):

The only discriminant is the runtime zero-check on the decoded values:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // ← wrong variant for post-0.13.3 tx
} else {
    ValidResourceBounds::AllResources(...)
})
``` [2](#0-1) 

**Hash domain divergence** (`get_tip_resource_bounds_hash`, line 188–211):

The function branches on the variant tag, not on the numeric values:

- `L1Gas` → poseidon hash over **2** resource felts: `[tip, L1_GAS_concat, L2_GAS_concat]`
- `AllResources` → poseidon hash over **3** resource felts: `[tip, L1_GAS_concat, L2_GAS_concat, L1_DATA_GAS_concat]` [3](#0-2) 

Even when `l1_data_gas = 0`, `get_concat_resource` encodes the ASCII resource name `"L1_DATA"` into the upper 56 bits of the felt, so `L1_DATA_GAS_concat` is never the zero felt. The two poseidon outputs are therefore always distinct. [4](#0-3) 

**Invariant broken**: `ValidResourceBounds` is defined as a version-discriminated enum, not a value-discriminated one. The comment on `L1Gas` reads *"Pre 0.13.3. Only L1 gas."* A post-0.13.3 transaction with zero-valued `AllResources` is a distinct protocol object from a pre-0.13.3 `L1Gas` transaction, but the protobuf round-trip erases that distinction. [5](#0-4) 

---

### Impact Explanation

A V3 transaction submitted to the gateway with `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is accepted and its hash `H_all` is computed with the three-felt preimage. When this transaction is propagated over P2P (historical block sync uses `Transaction` → protobuf → `Transaction`), the receiving node deserializes it as `L1Gas` and recomputes hash `H_l1 ≠ H_all`. The stored `tx_hash` field in the block no longer matches the hash derivable from the transaction body. Any node that re-derives the hash (e.g., for signature verification, block hash inclusion, or OS re-execution) will obtain the wrong value, causing:

- Valid transactions to fail signature verification on peer nodes (the signature was produced over `H_all`; the peer checks against `H_l1`).
- Wrong transaction hashes committed into block hash calculations and storage, producing divergent state between nodes that received the transaction before vs. after the protobuf round-trip.

This matches the **High** impact: *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

---

### Likelihood Explanation

The trigger requires only a standard V3 transaction where the user sets `l2_gas = 0` and `l1_data_gas = 0` (e.g., a transaction that only consumes L1 gas). This is a valid, gateway-accepted configuration. No privileged access, malformed bytes, or adversarial peer is required. The bug fires on every such transaction that passes through the P2P historical sync path.

---

### Recommendation

Remove the value-based heuristic. The protobuf message must carry an explicit version/variant tag (e.g., a boolean `is_all_resources` field, or a oneof discriminant) so the deserializer can reconstruct the exact variant without inspecting numeric values. Until the wire format is extended, the deserializer should default to `AllResources` for any message that contains all three fields, reserving `L1Gas` only for legacy messages that are missing the `l1_data_gas` field entirely (i.e., `value.l1_data_gas.is_none()`):

```rust
let l1_data_gas_opt = value.l1_data_gas;
// ...
Ok(if l1_data_gas_opt.is_none() {
    ValidResourceBounds::L1Gas(l1_gas)   // truly pre-0.13.3: field absent on wire
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

This preserves backward compatibility with pre-0.13.3 messages (which never set `l1_data_gas`) while correctly reconstructing post-0.13.3 messages regardless of their numeric values.

### Proof of Concept

```rust
use starknet_api::transaction::fields::{
    AllResourceBounds, ResourceBounds, ValidResourceBounds, GasAmount, GasPrice,
};
use apollo_protobuf::protobuf;
use std::convert::TryInto;

fn main() {
    // A post-0.13.3 V3 transaction with AllResources, l2_gas=0, l1_data_gas=0.
    let original = ValidResourceBounds::AllResources(AllResourceBounds {
        l1_gas:      ResourceBounds { max_amount: GasAmount(1000), max_price_per_unit: GasPrice(5) },
        l2_gas:      ResourceBounds::default(), // zero
        l1_data_gas: ResourceBounds::default(), // zero
    });

    // Serialize to protobuf (as done in P2P block sync).
    let proto: protobuf::ResourceBounds = original.into();

    // Deserialize — triggers the heuristic.
    let reconstructed: ValidResourceBounds = proto.try_into().unwrap();

    // Variant has changed: AllResources → L1Gas.
    assert!(matches!(reconstructed, ValidResourceBounds::L1Gas(_)),
        "reconstructed as L1Gas even though original was AllResources");

    // Hash preimage differs: AllResources hashes 3 resource felts, L1Gas hashes 2.
    // The L1_DATA_GAS_concat felt for zero bounds is non-zero (resource name in upper bits).
    let tip = starknet_api::transaction::fields::Tip(0);
    let h_all = starknet_api::transaction_hash::get_tip_resource_bounds_hash(&original, &tip).unwrap();
    let h_l1  = starknet_api::transaction_hash::get_tip_resource_bounds_hash(&reconstructed, &tip).unwrap();

    assert_ne!(h_all, h_l1,
        "hash mismatch: signer used h_all, peer verifies against h_l1 → signature fails");
}
```

fn get_concat_resource(
    resource_bounds: &ResourceBounds,
    resource_name: &ResourceName,
) -> Result<Felt, StarknetApiError> {
    let max_amount = resource_bounds.max_amount.0.to_be_bytes();
    let max_price = resource_bounds.max_price_per_unit.0.to_be_bytes();
    let concat_bytes =
        [[0_u8].as_slice(), resource_name.as_slice(), max_amount.as_slice(), max_price.as_slice()]
            .concat();
    Ok(Felt::from_bytes_be(&concat_bytes.try_into().expect("Expect 32 bytes")))
}
