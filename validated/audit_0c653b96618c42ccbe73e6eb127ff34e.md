### Title
Caller-Controlled `confirmations = 0` Bypasses Finality Check in `verify_transaction_inclusion` — (File: `contract/src/lib.rs`)

---

### Summary

The `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` functions accept a fully caller-controlled `confirmations` field in `ProofArgs`/`ProofArgsV2`. When set to `0`, the confirmation-depth guard is trivially satisfied for any u64 comparison, allowing any unprivileged NEAR caller to obtain a `true` proof-verification result for a transaction in any mainchain block regardless of how recently that block was submitted — directly analogous to passing `receiveAmtMin = 0` to bypass slippage protection.

---

### Finding Description

`ProofArgs.confirmations` is a `u64` supplied entirely by the caller: [1](#0-0) 

The only upper-bound guard is: [2](#0-1) 

The actual finality check is: [3](#0-2) 

When `args.confirmations = 0`, the expression evaluates to:

```
(heaviest_block_height.saturating_sub(target_block_height) + 1) >= 0
```

Because the left-hand side is a `u64`, it is always `>= 0`. The `require!` never fires. The function proceeds to compute and return the Merkle proof result for a block that may have zero depth in the chain.

`verify_transaction_inclusion_v2` inherits the same flaw because it delegates to the deprecated function after its own coinbase-proof check: [4](#0-3) 

No minimum confirmation count is enforced anywhere in the contract.

---

### Impact Explanation

The light client's `verify_transaction_inclusion` API is the trust anchor for downstream dApps that gate fund releases on BTC transaction finality. By passing `confirmations = 0`, any NEAR caller receives `true` for a transaction whose block was submitted one second ago and could still be reorganized away. The corrupted proof result — the exact output the contract is designed to produce — is delivered to the consuming contract or dApp with no indication that the finality guarantee was bypassed. Any dApp that does not independently enforce a minimum confirmation count (a reasonable assumption given the contract exposes the parameter as a caller-supplied safety knob) is exposed to double-spend-style attacks: an attacker submits a BTC transaction, waits for the relayer to post the header, then immediately triggers the dApp with `confirmations = 0`.

---

### Likelihood Explanation

The entry path requires no privileged role: `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are both public, unpaused view-like calls reachable by any NEAR account. The only precondition is that the target block is already in the mainchain (submitted by the trusted relayer), which is the normal operating state of the contract. The attack parameter (`confirmations = 0`) is a single field in a Borsh-serialized struct — trivial to craft. Likelihood is **medium**: it materialises whenever a consuming dApp forwards a user-supplied or insufficiently validated `confirmations` value, which is the natural usage pattern given the contract exposes it as a caller-controlled input.

---

### Recommendation

Enforce a protocol-level minimum confirmation count inside `verify_transaction_inclusion`. For example:

```rust
const MIN_CONFIRMATIONS: u64 = 1;
require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    "confirmations must be at least 1"
);
```

A stricter minimum (e.g., 6 for Bitcoin finality) should be considered based on the security model of consuming dApps. The contract should not delegate this safety invariant entirely to callers.

---

### Proof of Concept

1. Assume the light client has at least one mainchain block (normal operating state).
2. Obtain the `block_hash` of any mainchain block and a valid `tx_id` + `merkle_proof` for a transaction in it (publicly available from Bitcoin RPC).
3. Call `verify_transaction_inclusion` with:
   ```json
   {
     "tx_id": "<valid tx hash>",
     "tx_block_blockhash": "<block hash>",
     "tx_index": <index>,
     "merkle_proof": [<proof nodes>],
     "confirmations": 0
   }
   ```
4. The function returns `true` regardless of how recently the block was submitted or how many blocks have been built on top of it.
5. The same applies to `verify_transaction_inclusion_v2` by additionally supplying a valid `coinbase_tx_id` and `coinbase_merkle_proof`.

The broken invariant: `confirmations = 0` causes `(depth + 1) >= 0` — always `true` for `u64` — so the finality guard at [5](#0-4) 

is never enforced, and the Merkle proof result is returned unconditionally.
