Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` gates on `owner` instead of `sender`, enabling allowlist bypass and blocking legitimate router-based deposits — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter and enforces the allowlist against `owner` (the LP-position beneficiary, freely chosen by the caller). `SwapAllowlistExtension.beforeSwap` correctly gates on `sender`. Because `sender` and `owner` are passed as independent values by `ExtensionCalling._beforeAddLiquidity`, the guard evaluates the wrong identity, producing both an allowlist bypass and a router-mediated deposit DoS.

## Finding Description

`ExtensionCalling._beforeAddLiquidity` encodes both `sender` (the `addLiquidity` caller) and `owner` (the LP-position beneficiary) as distinct arguments: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` unnamed-discards the first argument (`sender`) and binds the second to `owner`, then checks `allowedDepositor[msg.sender][owner]`: [2](#0-1) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly names and checks `sender`, discarding the second argument: [3](#0-2) 

**Bypass path**: An unprivileged address `bob` calls `addLiquidity(owner=alice, ...)` where `alice` is allowlisted. The extension reads `allowedDepositor[pool][alice]` → `true` and returns the success selector. Bob's tokens enter the pool under Alice's position with zero privilege required.

**DoS path**: Pool admin allowlists a periphery router by its address (expecting `sender` to be checked). The router calls `addLiquidity(owner=end_user, ...)`. The extension reads `allowedDepositor[pool][end_user]` → `false` and reverts with `NotAllowedToDeposit`, permanently blocking all router-mediated deposits.

The `allowedDepositor` mapping is public, so any attacker can trivially read which addresses are allowlisted and use one as `owner`. [4](#0-3) 

## Impact Explanation

- **Bypass (admin-boundary break)**: Any unprivileged address can deposit into a pool intended to be restricted by naming an allowlisted address as `owner`. Unauthorized tokens enter the pool's accounting, violating the pool admin's access-control invariant.
- **DoS (broken core pool functionality)**: If the pool admin allowlists a router as the expected `sender`, every deposit routed through it is reverted because the end-user `owner` is not separately allowlisted. Core `addLiquidity` functionality becomes permanently unusable for the intended actor class without full reconfiguration.

Both outcomes satisfy the allowed impact gate: admin-boundary break and broken core pool functionality causing an unusable liquidity flow.

## Likelihood Explanation

- The bypass requires only reading the public `allowedDepositor` mapping to find one allowlisted address, then calling `addLiquidity` with `owner` set to that address. No special privilege is needed and the condition is reachable by any unprivileged actor.
- The DoS is triggered automatically in the standard periphery-router pattern (router allowlisted as `sender`, deposits on behalf of users as `owner`), which is the natural deployment model for this extension.
- Both conditions are repeatable and require no special setup beyond a pool using this extension.

## Recommendation

Change `beforeAddLiquidity` to name and check `sender` (the first parameter), mirroring `SwapAllowlistExtension.beforeSwap`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If gating on `owner` is intentional, the NatSpec, `setAllowedToDeposit` documentation, and admin tooling must be updated to state that every end-user `owner` must be individually allowlisted, and routers cannot be allowlisted by their own address.

## Proof of Concept

1. Deploy a pool with `DepositAllowlistExtension` in the `beforeAddLiquidity` hook order.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)`. Alice is the only allowlisted depositor.
3. Bob (not allowlisted) calls `pool.addLiquidity(owner=alice, salt=0, deltas=..., extensionData=...)` directly.
4. The pool calls `extension.beforeAddLiquidity(sender=bob, owner=alice, ...)`.
5. The extension evaluates `allowedDepositor[pool][alice]` → `true` and returns the success selector.
6. Bob's tokens are deposited into Alice's LP position. The allowlist is bypassed with zero privilege.

For the DoS path: pool admin calls `setAllowedToDeposit(pool, router, true)`. Router calls `pool.addLiquidity(owner=bob, ...)`. Extension evaluates `allowedDepositor[pool][bob]` → `false` and reverts with `NotAllowedToDeposit`, blocking all router-mediated deposits.
