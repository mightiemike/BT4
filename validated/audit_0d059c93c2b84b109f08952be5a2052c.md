### Title
Missing Chain ID in Precommit Vote Message Digest Allows Cross-Network Consensus Signature Replay - (File: crates/apollo_signature_manager/src/signature_manager.rs)

### Summary

The `build_precommit_vote_message_digest` function in `apollo_signature_manager` computes the ECDSA signing preimage for consensus precommit votes using only a fixed domain separator and the block hash bytes. No chain ID is included. This is the direct Sequencer analog of the external "withdraw message replayable on a different chain" bug: a validator's precommit vote signature produced on one Starknet network is cryptographically valid on any other Starknet network that presents the same block hash value.

### Finding Description

`build_precommit_vote_message_digest` constructs the message that validators sign when casting a precommit vote in the Tendermint-style consensus protocol:

```rust
fn build_precommit_vote_message_digest(block_hash: BlockHash) -> MessageDigest {
    let block_hash = block_hash.to_bytes_be();
    let mut message = Vec::with_capacity(PRECOMMIT_VOTE.len() + block_hash.len());
    message.extend_from_slice(PRECOMMIT_VOTE);   // b"PRECOMMIT_VOTE"
    message.extend_from_slice(&block_hash);
    MessageDigest(blake2s_to_felt(&message))
}
``` [1](#0-0) 

The preimage is exactly `b"PRECOMMIT_VOTE" || block_hash_bytes_be`. No chain ID, no network identifier, no height, no round. The `SignatureManager` struct itself holds no chain ID: [2](#0-1) 

The public API `sign_precommit_vote` accepts only a `BlockHash`: [3](#0-2) 

And the verification path `verify_precommit_vote_signature` reconstructs the same chain-ID-free digest: [4](#0-3) 

The `calculate_block_hash` function that produces the `BlockHash` value being signed also does **not** include a chain ID in its preimage — it hashes block number, state root, sequencer address, timestamp, commitments, gas prices, starknet version, and previous block hash: [5](#0-4) 

Therefore the signed object (`block_hash`) is itself chain-ID-free, and the signing domain adds no chain ID either.

### Impact Explanation

A precommit vote signature `(r, s)` over `blake2s("PRECOMMIT_VOTE" || H)` is valid on **every** Starknet network for which the block hash equals `H`. Concrete scenarios:

1. **Chain fork / testnet initialized from mainnet state**: Both forks share identical block hashes for all blocks up to the fork point. A validator's precommit signature for any pre-fork block is valid on both forks simultaneously. An adversary who collects signatures from one fork can inject them into the other fork's consensus, causing validators to appear to have voted for a block they never intended to finalize on that network.

2. **Testnet/staging network with same genesis**: If a staging network is seeded from mainnet state (a common operational practice), block hashes can collide. Precommit signatures from mainnet validators are directly replayable on the staging network and vice versa.

The `SignatureManagerRequest::SignPrecommitVote` RPC path confirms this is the live production signing path used by the consensus manager: [6](#0-5) 

This matches the allowed impact: **"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."** — the signature logic fails to bind the chain ID, so the same signature authorizes a vote on a different network.

### Likelihood Explanation

The trigger requires two Starknet networks to share a block hash value. This is not the common case for fully independent mainnet/testnet deployments (genesis diverges immediately), but it is realistic for:
- Any chain fork (both forks share all pre-fork block hashes)
- Staging/shadow networks initialized from mainnet snapshots
- Testnets that replay mainnet history

An unprivileged observer on the p2p network can collect broadcast precommit vote signatures and replay them on another network without any special access.

### Recommendation

Include the chain ID in the precommit vote message digest. The `SignatureManager` should be constructed with a `chain_id` field, and `build_precommit_vote_message_digest` should incorporate it:

```rust
fn build_precommit_vote_message_digest(
    chain_id: &ChainId,
    block_hash: BlockHash,
) -> MessageDigest {
    let chain_id_bytes = chain_id.as_hex().into_bytes();
    let block_hash = block_hash.to_bytes_be();
    let mut message = Vec::new();
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&chain_id_bytes);
    message.extend_from_slice(&block_hash);
    MessageDigest(blake2s_to_felt(&message))
}
```

Both `sign_precommit_vote` and `verify_precommit_vote_signature` must be updated to pass the chain ID, and the `SignatureManager` struct should store it at construction time. The `SignatureManagerRequest::SignPrecommitVote` variant and its client/server wiring in `apollo_signature_manager_types` must also be updated to carry the chain ID.

### Proof of Concept

1. Run two Starknet sequencer nodes: one on `SN_MAIN` (mainnet) and one on `SN_GOERLI` (testnet), both initialized from the same genesis snapshot so that block 0 has the same hash `H0`.

2. Observe a validator `V` broadcast a precommit vote on mainnet for block 0 with signature `(r, s)` over `blake2s(b"PRECOMMIT_VOTE" || H0)`.

3. On the testnet, inject a `Vote { vote_type: Precommit, proposal_commitment: H0, voter: V, signature: (r, s) }` into the consensus broadcast channel.

4. Call `verify_precommit_vote_signature(BlockHash(H0), (r,s), V.public_key)` — it reconstructs the identical digest `blake2s(b"PRECOMMIT_VOTE" || H0)` and the ECDSA verification passes, because the digest is identical on both networks.

5. The testnet consensus layer accepts `V`'s precommit vote for block 0 even though `V` never signed anything on the testnet, demonstrating cross-network signature replay. [1](#0-0) [4](#0-3)
