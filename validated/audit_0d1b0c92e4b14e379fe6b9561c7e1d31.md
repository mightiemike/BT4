### Title
Wrong `ValidResourceBounds` Type Classification in Protobuf Deserialization Causes Transaction Hash Canonicalization Break - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` implementation conflates two structurally distinct cases — a missing `l1_data_gas` field (old 0.13.2 wire format) and a present-but-zero `l1_data_gas` field (current `AllResources` format) — and silently downgrades the latter to `ValidResourceBounds::L1Gas`. Because `get_tip_resource_bounds_hash` includes the `l1_data_gas` element in the Poseidon preimage for `AllResources` but omits it entirely for `L1Gas`, any V3 transaction whose `l2_gas` and `l1_data_gas` bounds are both zero will produce a different transaction hash after a protobuf round-trip than the hash that was computed and signed at submission time.

### Finding Description

**Root cause — `crates/apollo_protobuf/src/converters/transaction.rs` lines 417–436:**

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else { ... };
        let Some(l2_gas) = value.l2_gas else { ... };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // ← erases None vs Some(0)
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {          // ← wrong discriminant
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}
``` [1](#0-0) 

After `unwrap_or_default()` the information about whether `l1_data_gas` was absent (`None`, old 0.13.2 peer) or present-but-zero (`Some(ResourceLimits{0,0})`, current peer) is permanently lost. The subsequent `is_zero()` guard therefore fires for both cases and returns `L1Gas` for what was originally an `AllResources` transaction.

**Hash divergence — `crates/starknet_api/src/transaction_hash.rs` lines 188–210:**

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],                          // ← l1_data_gas absent
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // ← present
        }
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
``` [2](#0-1) 

For `AllResources` the Poseidon chain is `[tip, l1_gas_felt, l2_gas_felt, l1_data_gas_felt]`; for `L1Gas` it is `[tip, l1_gas_felt, l2_gas_felt]`. Even when `l1_data_gas = 0` the two chains produce different digests because the element count differs.

**Serialization side is correct — `crates/apollo_protobuf/src/converters/transaction.rs` lines 471–489:**

```rust
impl From<ValidResourceBounds> for protobuf::ResourceBounds {
    fn from(value: ValidResourceBounds) -> Self {
        match value {
            ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
                l1_gas: Some(l1_gas.into()),
                l2_gas: Some(value.get_l2_bounds().into()),
                l1_data_gas: Some(ResourceBounds::default().into()),  // ← Some(zero)
            },
            ValidResourceBounds::AllResources(...) => protobuf::ResourceBounds {
                l1_gas: Some(l1_gas.into()),
                l2_gas: Some(l2_gas.into()),
                l1_data_gas: Some(l1_data_gas.into()),                // ← Some(zero)
            },
        }
    }
}
``` [3](#0-2) 

Both variants serialize `l1_data_gas` as `Some(zero)` when the bound is zero, so the wire bytes are identical. The asymmetry is entirely in the deserializer.

**Trigger path:**

1. A user submits a V3 `InvokeTransactionV3` with `AllResourceBounds { l1_gas: non-zero, l2_gas: zero, l1_data_gas: zero }`. The gateway accepts it (fee is non-zero because `l1_gas` is non-zero; the `ZeroResourceBounds` check passes). [4](#0-3) 

2. The hash is computed via `InternalRpcInvokeTransactionV3::calculate_transaction_hash` → `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash(AllResources{…})`. The `l1_data_gas = 0` element is included in the Poseidon chain. Call this hash **H₁**. [5](#0-4) 

3. The transaction is propagated to a peer via P2P protobuf. The serializer emits `l1_data_gas = Some(zero)`.

4. The peer deserializes via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Because `l1_data_gas.is_zero() && l2_gas.is_zero()`, the result is `ValidResourceBounds::L1Gas(l1_gas)`.

5. The peer recomputes the hash via `validate_transaction_hash` → `get_transaction_hash` → `get_tip_resource_bounds_hash(L1Gas{…})`. The `l1_data_gas` element is **absent** from the Poseidon chain. Call this hash **H₂ ≠ H₁**. [6](#0-5) 

6. `validate_transaction_hash` returns `false`; the peer rejects the transaction.

### Impact Explanation

A valid, gateway-accepted V3 transaction whose `l2_gas` and `l1_data_gas` bounds are both zero will be permanently rejected by every peer that receives it over P2P. The originating node's mempool holds the transaction; no other node will sequence it. This is a targeted, unprivileged denial-of-sequencing attack: any user can craft such a transaction, have it accepted by the gateway, and ensure it never reaches consensus.

### Likelihood Explanation

The trigger requires only a standard V3 invoke transaction with `l2_gas = 0` and `l1_data_gas = 0` (but `l1_gas > 0`). No special privileges, no malformed bytes, no out-of-range values. The gateway's `min_gas_price` check applies to `l2_gas.max_price_per_unit`; if that config value is 0 (or the user sets `l2_gas.max_price_per_unit = 0`), the transaction passes all stateless checks. The bug is latent in every P2P sync path that deserializes a `Transaction` containing `ValidResourceBounds`.

### Recommendation

Preserve the `Option` before calling `unwrap_or_default` and use the original optionality — not the post-conversion zero check — as the discriminant:

```rust
let l1_data_gas_opt = value.l1_data_gas;
let l1_data_gas: ResourceBounds = l1_data_gas_opt
    .unwrap_or_default()
    .try_into()?;
Ok(if l1_data_gas_opt.is_none() && l2_gas.is_zero() {
    // Genuine old 0.13.2 wire format: no l1_data_gas field present.
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

This preserves backward compatibility with 0.13.2 peers (which omit `l1_data_gas` entirely) while correctly round-tripping current `AllResources` transactions whose `l1_data_gas` bound happens to be zero.

### Proof of Concept

```
1. Construct InvokeTransactionV3 with:
     resource_bounds = AllResourceBounds {
         l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
         l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
         l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },
     }

2. Submit to gateway → accepted, hash H1 computed as AllResources
   (l1_data_gas element IS in Poseidon chain)

3. Serialize to protobuf::ResourceBounds:
     l1_gas      = Some(ResourceLimits{1000, 1})
     l2_gas      = Some(ResourceLimits{0, 0})
     l1_data_gas = Some(ResourceLimits{0, 0})   ← present but zero

4. Deserialize on peer via TryFrom:
     l1_data_gas.is_zero() && l2_gas.is_zero() → true
     → ValidResourceBounds::L1Gas(l1_gas)       ← wrong type

5. Recompute hash as L1Gas → H2
   (l1_data_gas element NOT in Poseidon chain)
   H2 ≠ H1

6. validate_transaction_hash returns false → transaction rejected
```
