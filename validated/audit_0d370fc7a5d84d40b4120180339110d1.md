Audit Report

## Title
Single `MAX_REF_STALENESS` Applied to Both Legs of Synthetic Quote Allows Stale-Price Execution — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

## Summary
`AnchoredPriceProvider` supports a two-feed synthetic ratio mode (e.g., BTC/USD ÷ ETH/USD = BTC/ETH) but validates both legs against a single immutable `MAX_REF_STALENESS`. When the two Chainlink feeds have different heartbeats, no single staleness value can correctly serve both legs: setting it to the slower feed's heartbeat silently accepts a massively stale price from the faster feed, producing a wrong synthetic ratio that reaches pool swaps; setting it to the faster feed's heartbeat permanently DoS-es the slower feed. Either outcome is locked in at deployment because `MAX_REF_STALENESS` is immutable.

## Finding Description
`_readLeg` applies `MAX_REF_STALENESS` uniformly to whichever `feedId` it is called with (line 283). `_getBidAndAskPrice` calls `_readLeg(baseFeedId)` and then `_readLeg(quoteFeedId)` (lines 259, 264), combining the results into a synthetic ratio `mid = mulDiv(mid1, ORACLE_DECIMALS, mid2)` (line 267). There is no per-leg staleness parameter in the constructor or anywhere else in the contract. The constructor accepts only a single `_maxRefStaleness` bounded to `[0, 7 days]` (line 150) and stores it as the immutable `MAX_REF_STALENESS` (line 151).

For a USDC/ETH synthetic provider:
- `baseFeedId` = USDC/USD — Chainlink heartbeat 86 400 s
- `quoteFeedId` = ETH/USD — Chainlink heartbeat 3 600 s

**Case A (`MAX_REF_STALENESS = 86 400`):** ETH/USD passes `_isStale` up to 23 h 59 m after its last update. The synthetic ratio `USDC/USD ÷ ETH/USD` is computed with a stale ETH denominator, producing a wrong bid/ask that is forwarded to the pool swap. LPs absorb the resulting loss.

**Case B (`MAX_REF_STALENESS = 3 600`):** USDC/USD, which legitimately updates every 24 hours, is rejected as stale on every call between heartbeats, permanently bricking the provider.

No existing guard compensates: `_isStale` is a pure function with no per-feed override path, and `MAX_REF_STALENESS` cannot be changed post-deployment.

## Impact Explanation
In Case A, a trader observing that ETH/USD has not been updated for several hours can swap against the pool at a price that no longer reflects the true ETH value. The pool's LPs absorb the loss. This is a direct loss of LP principal matching the "bad-price execution: stale bid/ask reaches a pool swap" allowed impact. The magnitude scales with ETH price movement during the stale window (up to ~24 h), which can easily exceed Sherlock's medium/high thresholds.

## Likelihood Explanation
Synthetic providers pairing a stablecoin feed (86 400 s heartbeat) with a volatile-asset feed (3 600 s heartbeat) are a natural and explicitly documented use case (the contract NatDoc names BTC/USD ÷ ETH/USD as the canonical example). Any such deployment is structurally forced into one of the two failure modes. No privileged access, malicious setup, or non-standard token behavior is required; normal swap traffic triggers the bad-price path in Case A.

## Recommendation
Replace the single `MAX_REF_STALENESS` immutable with two per-leg immutables:

```solidity
uint256 public immutable MAX_BASE_STALENESS;
uint256 public immutable MAX_QUOTE_STALENESS;
```

Pass both through the constructor (each bounded to `[0, 7 days]`) and apply each only to its respective `_readLeg` call:

```solidity
// in _getBidAndAskPrice, base leg:
if (_isStale(refTime, block.timestamp, MAX_BASE_STALENESS)) ...

// quote leg:
if (_isStale(refTime, block.timestamp, MAX_QUOTE_STALENESS)) ...
```

For single-feed deployments (`quoteFeedId == 0`), `MAX_QUOTE_STALENESS` is unused and can be set to any value (e.g., `type(uint256).max`).

## Proof of Concept
1. Deploy `AnchoredPriceProvider` with:
   - `baseFeedId` = USDC/USD (Chainlink heartbeat 86 400 s)
   - `quoteFeedId` = ETH/USD (Chainlink heartbeat 3 600 s)
   - `MAX_REF_STALENESS = 86400`
2. ETH/USD is last updated at `T`. At `T + 82 800` (23 hours later) ETH has moved 5%.
3. A trader calls `getBidAndAskPrice()` through the pool's swap path.
4. `_readLeg(baseFeedId)` passes — USDC/USD is fresh.
5. `_readLeg(quoteFeedId)` passes — `(now − T) = 82 800 < 86 400 = MAX_REF_STALENESS`.
6. `mid = mulDiv(USDC_mid, ORACLE_DECIMALS, ETH_mid_23h_stale)` — ratio uses the stale ETH price.
7. The resulting bid/ask is ~5% off from the true market price.
8. The trader swaps, extracting value from LPs at the stale price.

A Foundry fork test can reproduce this by mocking the oracle's `price()` return to return a `refTime` of `block.timestamp − 82800` for the ETH/USD feed and asserting that `getBidAndAskPrice()` returns a bid/ask materially different from the true ratio.
