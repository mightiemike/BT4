The vulnerability is real. Let me trace through the exact mechanics.

**Step 1: Pre-v8 state setup is reachable**

Before v8, `validateNewPosition` did not include `validateNonVestingAccount`. The vesting account filter was added only in v8. [1](#0-0) 

The integration test explicitly confirms that a `PermanentLockedAccount` could create both `CommitDelegationToTier` and `LockTier` positions on v7.2.0: [2](#0-1) 

**Step 2: `alignVestingDelegationTracking` calls `TrackDelegation` with `deficit = positionAmount`**

After `transferDelegationFromPosition` returns the delegation to the owner (bypassing the bank-side hook), `alignVestingDelegationTracking` computes `deficit = actualDelegated - tracked` and calls `vacc.TrackDelegation(blockTime, coins(deficit), coins(deficit))`: [3](#0-2) 

**Step 3: `TrackDelegation` DV/DF split for `PermanentLockedAccount`**

The Cosmos SDK `BaseVestingAccount.TrackDelegation` computes:
```
vestingAmt = GetVestingCoins(blockTime) = OriginalVesting  (always, for PermanentLockedAccount)
DV_new = min(vestingAmt, DV_old + amount)
DF_new = DF_old + max(0, amount - (DV_new - DV_old))
```

For `OriginalVesting=100`, `DV_old=0`, `amount=200`:
- `DV_new = min(100, 200) = 100`
- `DF_new = 0 + (200 - 100) = 100`

**Step 4: `LockedCoins = 0` makes bank balance spendable**

`BaseVestingAccount.LockedCoins = max(0, OriginalVesting - DelegatedVesting)`. With `DV = 100 = OriginalVesting`, `LockedCoins = 0`. `SpendableCoins = balance - 0 = balance`.

The owner's bank balance at this point is `original_balance - amount` (after LockTier sent `amount` to the position delegator). If `original_balance > amount` (e.g., `original_balance=300`, `amount=200`), then `balance=100` and `SpendableCoins=100` — the permanently locked coins are now fully spendable.

**Step 5: The Mixed test confirms `DF > 0` is the expected post-migration state**

The integration test and keeper test both explicitly assert `DF = lockAmount` after migration, confirming this is the actual code path: [4](#0-3) [5](#0-4) 

Neither test asserts `SpendableCoins == 0` after migration for the `PermanentLockedAccount` owner.

**Step 6: The exploit path**

The `TestForceFullExitWithDelegation_VestingOwner_LockOrigin` test uses `balance = 2*lockedAmount`, `amount = lockedAmount`. After LockTier, bank balance = `lockedAmount`. After migration, `DV = lockedAmount = OriginalVesting`, `LockedCoins = 0`, `SpendableCoins = lockedAmount`. The test does not assert `SpendableCoins == 0`: [6](#0-5) 

Compare with `TestTransferDelegationFromPosition_VestingOwnerForcedExitWithDelegation`, which does assert `SpendableCoins == 0` — but that test does not go through `alignVestingDelegationTracking`: [7](#0-6)

### Title
`alignVestingDelegationTracking` sets `DelegatedVesting = OriginalVesting` for `PermanentLockedAccount`, zeroing `LockedCoins` and making permanently locked bank balance spendable — (`x/tieredrewards/keeper/force_exit.go`)

### Summary

`alignVestingDelegationTracking` calls `TrackDelegation(deficit, deficit)` where `deficit` equals the LockTier position amount. For a `PermanentLockedAccount`, `TrackDelegation` saturates `DelegatedVesting` at `OriginalVesting` and overflows the remainder into `DelegatedFree`. This causes `LockedCoins = OriginalVesting - DelegatedVesting = 0`, making the owner's entire bank balance spendable — including the permanently locked principal.

### Finding Description

**Root cause** (`x/tieredrewards/keeper/force_exit.go`, lines 163–172):

```go
deficit := actualDelegated.Sub(tracked)
coins := sdk.NewCoins(sdk.NewCoin(bondDenom, deficit))
vacc.TrackDelegation(sdkCtx.BlockTime(), coins, coins)
```

`TrackDelegation` is designed for time-based vesting accounts where `vestingCoins` decreases over time. For `PermanentLockedAccount`, `vestingCoins = OriginalVesting` always. When `deficit >= OriginalVesting`, the SDK sets `DV = OriginalVesting` and `DF = deficit - OriginalVesting`. The `LockedCoins` formula (`OriginalVesting - DV`) then yields zero, making the bank balance fully spendable.

**Concrete numbers** (`OriginalVesting=100`, `balance=300`, `LockTier amount=200`):

| Step | Bank balance | DV | DF | LockedCoins | SpendableCoins |
|------|-------------|----|----|-------------|----------------|
| Initial | 300 | 0 | 0 | 100 | 200 |
| After LockTier(200) | 100 | 0 | 0 | 100 | 0 |
| After migration | 100 | 100 | 100 | **0** | **100** ← BUG |
| Owner spends 100 | 0 | 100 | 100 | 0 | 0 |
| After undelegate 200 | 200 | 0 | 0 | 100 | 100 |

The owner extracted 100 permanently locked coins.

### Impact Explanation

A `PermanentLockedAccount` owner who held a LockTier position (created pre-v8) with `amount >= OriginalVesting` can, after the v8 migration, spend their permanently locked principal. The amount extractable is `min(OriginalVesting, bank_balance_after_LockTier)`. This is a direct loss of locked funds — the vesting schedule is bypassed.

### Likelihood Explanation

The precondition requires a `PermanentLockedAccount` that created a LockTier position pre-v8 with `amount >= OriginalVesting`. This requires the owner to have received external transfers (e.g., staking rewards, airdrops) exceeding `OriginalVesting` before v8. The integration test setup (`_create_vesting_acc_owned_positions`) explicitly demonstrates this pattern is expected and tested. Any such account on mainnet at the time of the v8 upgrade is affected.

### Recommendation

In `alignVestingDelegationTracking`, before calling `TrackDelegation`, check whether the account is a `PermanentLockedAccount` (or more generally, whether `vestingCoins == OriginalVesting` at all times). If so, cap `deficit` at `OriginalVesting - DV_current` to prevent `DV` from reaching `OriginalVesting` via this artificial top-up. Alternatively, skip `TrackDelegation` entirely for `PermanentLockedAccount` and instead directly set `DV = min(OriginalVesting, DV + deficit)` and `DF = 0`, preserving the invariant that `PermanentLockedAccount` never has `DelegatedFree > 0`.

### Proof of Concept

```go
// Keeper test: PermanentLockedAccount, OriginalVesting=100, LockTier amount=200
// After ForceFullExitWithDelegation, assert DelegatedFree == 0 and SpendableCoins == 0.

lockedAmount := sdkmath.NewInt(100)
// balance = 300: 100 locked + 200 extra (external transfer)
owner := s.newVestingOwnerWithBalance(bondDenom, lockedAmount, sdkmath.NewInt(300))
// LockTier with amount=200 > OriginalVesting=100 (uses 200 spendable coins)
pos := s.createLockTierPositionV1(owner, valAddr, sdkmath.NewInt(200))

    "PermanentLockedAccount must never have DelegatedFree > 0")
// BUG: SpendableCoins == 100 (the locked coins), should be 0
s.Require().True(s.app.BankKeeper.SpendableCoins(s.ctx, owner).IsZero(),
    "permanently locked coins must not become spendable after migration")
```
