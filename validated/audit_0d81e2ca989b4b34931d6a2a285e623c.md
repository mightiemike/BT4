### Title
`updateTierFeeRates` Does Not Apply to Future Products, Causing Zero-Fee Trading for Non-Default Tier Users — (`core/contracts/OffchainExchange.sol`)

---

### Summary

When `updateTierFeeRates` is called with `productId == QUOTE_PRODUCT_ID` to configure custom fee rates for a tier, it only populates `feeRates[tier][productId]` for products that exist at that moment. The tier is permanently marked as non-default in `nonDefaultFeeTierMask`. Any product added afterward has a zero-initialized `feeRates[tier][newProduct]` entry, and `getTierFeeRateX18` returns that zero struct — meaning users in non-default tiers pay **zero fees** on every new product, while default-tier users pay the 2 bps standard rate.

---

### Finding Description

`updateTierFeeRates` iterates over current product IDs when `productId == QUOTE_PRODUCT_ID`:

```solidity
// OffchainExchange.sol
function updateTierFeeRates(IEndpoint.UpdateTierFeeRates memory txn)
    external
    onlyEndpoint
{
    if (txn.productId == QUOTE_PRODUCT_ID) {
        uint32[] memory spotProductIds = spotEngine.getProductIds();
        uint32[] memory perpProductIds = perpEngine.getProductIds();
        for (uint32 i = 0; i < spotProductIds.length; i++) { ... }
        for (uint32 i = 0; i < perpProductIds.length; i++) { ... }
    } else { ... }
    nonDefaultFeeTierMask |= uint128(1) << txn.tier;   // tier permanently marked non-default
}
``` [1](#0-0) 

`getTierFeeRateX18` branches on `nonDefaultFeeTierMask` and returns the stored struct for any non-default tier — including the zero-initialized struct for products that did not exist when the tier was configured:

```solidity
function getTierFeeRateX18(uint32 tier, uint32 productId)
    public view returns (FeeRates memory)
{
    if (nonDefaultFeeTierMask & (1 << tier) != 0) {
        return feeRates[tier][productId];   // returns {0, 0} for new products
    }
    return FeeRates({ makerRateX18: 0, takerRateX18: 200_000_000_000_000 }); // 2 bps default
}
``` [2](#0-1) 

The fee is then applied in `applyFee` using the returned `feeRate`:

```solidity
FeeInfo memory feeInfo = getUserFeeRateWithBuilder(...);
int128 keepRateX18 = ONE - feeInfo.feeRate;   // ONE - 0 = ONE → zero fee charged
``` [3](#0-2) 

The root cause is the same class as the FrankenDAO bug: a stored value (`feeRates[tier][productId]`) is computed from the system state at configuration time. When the system state changes (new product added), existing stored entries are not retroactively updated, and the lookup logic (`nonDefaultFeeTierMask` check) cannot distinguish between "explicitly set to zero" and "never set."

---

### Impact Explanation

Users assigned to any non-default fee tier pay **zero taker and maker fees** on every product added after their tier was configured. Default-tier users continue to pay 2 bps taker fees on those same products. This creates a permanent, asymmetric fee advantage for non-default tier users on new products, directly reducing protocol fee revenue for every trade on those products for the lifetime of the exchange.

---

### Likelihood Explanation

The trigger is two sequential, routine admin operations: (1) configure a custom fee tier (expected for VIP/market-maker users), then (2) list a new trading product (expected as the protocol grows). Neither action is unusual or adversarial. Any user who has been assigned a non-default tier — including market makers or high-volume traders — can immediately exploit the zero-fee window on every new product without any further action.

---

### Recommendation

In `getTierFeeRateX18`, fall back to the default rate when the stored entry for a non-default tier is zero-initialized:

```solidity
function getTierFeeRateX18(uint32 tier, uint32 productId)
    public view returns (FeeRates memory)
{
    if (nonDefaultFeeTierMask & (1 << tier) != 0) {
        FeeRates memory stored = feeRates[tier][productId];
        if (stored.takerRateX18 != 0 || stored.makerRateX18 != 0) {
            return stored;
        }
    }
    return FeeRates({ makerRateX18: 0, takerRateX18: 200_000_000_000_000 });
}
```

Alternatively, when a new product is registered, iterate over all non-default tiers and populate `feeRates[tier][newProductId]` with a sensible default.

---

### Proof of Concept

1. Admin calls `updateTierFeeRates(tier=1, productId=QUOTE_PRODUCT_ID, takerRateX18=1bps)`. This sets `feeRates[1][productA] = {0, 1bps}`, `feeRates[1][productB] = {0, 1bps}`, and permanently sets `nonDefaultFeeTierMask |= (1 << 1)`.
2. Admin adds new perp product C via `addOrUpdateProduct`.
3. `feeRates[1][productC]` is zero-initialized (`{0, 0}`).
4. User Alice is in tier 1. She submits a taker order on product C.
5. `getUserFeeRateWithBuilder` calls `getTierFeeRateX18(1, productC)`. Since `nonDefaultFeeTierMask & (1 << 1) != 0`, it returns `feeRates[1][productC]` = `{0, 0}`.
6. `applyFee` computes `keepRateX18 = ONE - 0 = ONE`, so `fee = 0`. Alice pays zero fees.
7. User Bob is in tier 0 (default). He submits a taker order on product C. `getTierFeeRateX18(0, productC)` returns the default `{0, 2bps}`. Bob pays 2 bps.
8. Alice has a permanent zero-fee advantage over Bob on product C, and on every future product added to the exchange.
