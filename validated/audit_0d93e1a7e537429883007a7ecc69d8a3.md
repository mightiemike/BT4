Audit Report

## Title
SwapAllowlistExtension Gates on Router Address Instead of Real User, Allowing Any User to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which `MetricOmmPool.swap()` sets to its own `msg.sender`. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the end user. A pool admin who allowlists the router address to enable standard periphery access inadvertently grants every user on the internet the ability to pass the allowlist check, completely voiding the access control.

## Finding Description
The root cause chain is confirmed by the production code:

1. `MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap()`: [1](#0-0) 

2. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value from step 1 — the router address, not the end user: [2](#0-1) 

3. `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` at the pool level. The original caller's address is stored only in transient callback state via `_setNextCallbackContext` and is never forwarded to the extension: [3](#0-2) 

There are no existing guards that compensate for this mismatch. The `allowedSwapper` mapping is keyed on `[pool][sender]`, and `sender` is structurally always the immediate caller of `pool.swap()` — the router — not the economic actor initiating the trade.

## Impact Explanation
The swap allowlist is the primary access-control boundary for curated pools. Its complete bypass via the canonical router path constitutes a direct admin-boundary break. Any user can execute swaps on pools restricted to specific counterparties (e.g., KYC'd or institutional users) simply by calling `exactInputSingle` through `MetricOmmSimpleRouter` on a pool where the router is allowlisted. This enables unauthorized trading on restricted pools, regulatory/compliance breaches, and potential fund extraction by untrusted actors on pools where the allowlist was the sole guard.

## Likelihood Explanation
Likelihood is high. `MetricOmmSimpleRouter` is the canonical swap entry point. Any pool admin deploying `SwapAllowlistExtension` who also wants allowlisted users to access the pool through the standard router will allowlist the router address — this is the natural and expected operational pattern. No special knowledge or adversarial setup is required; any user calling `exactInputSingle` through the router on such a pool silently bypasses the guard.

## Recommendation
The extension must check the original user's identity, not the intermediary's. The cleanest fix is for the router to encode `msg.sender` into `extensionData` (e.g., as an ABI-encoded prefix), and for `SwapAllowlistExtension.beforeSwap` to decode and check that address when `msg.sender` (the pool) is being called via a known router. Alternatively, document and enforce that `SwapAllowlistExtension` requires direct pool calls and revert when the `sender` is a known router address.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (intending to allow standard periphery access for allowlisted users)
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  - attacker calls MetricOmmSimpleRouter.exactInputSingle(pool, ...)
  - Router calls pool.swap(...) — msg.sender at pool = router
  - Pool calls _beforeSwap(sender=router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] == true → PASSES
  - Attacker's swap executes on the curated pool despite never being allowlisted

Result:
  - SwapAllowlistExtension.beforeSwap() at line 37 evaluates
    allowedSwapper[pool][router] = true, not allowedSwapper[pool][attacker]
  - The allowlist is completely bypassed for any user who routes through the router
``` [4](#0-3) [5](#0-4)
