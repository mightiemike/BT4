Missing `ENABLE_SECP_OPS` Flag Check on 4-Byte Secp Opcode Path Bypasses Operator Gate - (File: src/chia_dialect.rs)

---

### Summary

The `ChiaDialect::op()` function in `src/chia_dialect.rs` dispatches secp256k1 and secp256r1 signature-verification operators through **two separate code paths**: a 4-byte opcode path (opcodes `0x13d61f00` / `0x1c3a8f00`) and a 1-byte opcode path (opcodes 64 / 65). The 1-byte path correctly gates execution behind the `ENABLE_SECP_OPS` flag. The 4-byte path performs **no such check**, unconditionally dispatching to the same underlying operator functions. An attacker who crafts CLVM bytes using the 4-byte opcodes can invoke secp signature verification regardless of whether `ENABLE_SECP_OPS` is set in the dialect, bypassing the flag gate entirely.

---

### Finding Description

In `src/chia_dialect.rs`, the `op()` method of `ChiaDialect` handles operator dispatch. For 1-byte opcodes, the secp operators are correctly gated:

```rust
// lines 248-249
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [1](#0-0) 

However, the 4-byte opcode path (entered when `op_len == 4`) dispatches the same two operator functions with **no flag check**:

```rust
// lines 175-182
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
``` [2](#0-1) 

The `ENABLE_SECP_OPS` flag is defined as the gate for secp operations:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
``` [3](#0-2) 

The `MEMPOOL_MODE` constant — the stricter validation mode used for mempool acceptance — does **not** include `ENABLE_SECP_OPS`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [4](#0-3) 

This means that in `MEMPOOL_MODE`, secp ops via 1-byte opcodes 64/65 are correctly rejected (falling through to `unknown_operator`), but the identical operations invoked via 4-byte opcodes `0x13d61f00` / `0x1c3a8f00` execute unconditionally.

---

### Impact Explanation

The `ENABLE_SECP_OPS` flag gate is rendered ineffective for any caller that does not set it (including `MEMPOOL_MODE`). A CLVM program using the 4-byte secp opcodes will:

1. **Execute secp signature verification** in contexts where it is supposed to be blocked.
2. **Cause consensus divergence**: nodes or validators running with `ENABLE_SECP_OPS` unset will accept programs that should be rejected, while nodes with the flag set will also accept them — but for different reasons. Any node that relies on the flag to distinguish pre- and post-activation behavior will diverge.
3. **Undermine the softfork activation model**: the flag exists precisely to allow controlled, coordinated activation of secp ops. The 4-byte bypass makes the activation boundary meaningless for programs that use those opcodes.

---

### Likelihood Explanation

The 4-byte opcodes are valid, parseable CLVM atoms. Any attacker who can submit a CLVM program to a node (e.g., via a spend bundle in the mempool) can trivially craft a program using `0x13d61f00` or `0x1c3a8f00` as the operator. No special privileges, keys, or social engineering are required. The attacker-controlled entry path is direct: serialize a 4-byte atom with the target opcode value as the operator in a CLVM expression and submit it.

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch path, mirroring the guard already present for the 1-byte aliases:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures that both encoding forms of the secp operators are subject to the same flag gate, and that `MEMPOOL_MODE` and any other flag configuration that omits `ENABLE_SECP_OPS` consistently rejects secp operations regardless of how the opcode is encoded.

### Proof of Concept

1. Construct a CLVM program where the operator atom is the 4-byte big-endian encoding of `0x13d61f00` (secp256k1_verify) with valid pubkey, message-hash, and signature arguments.
2. Run it via `run_program` with a `ChiaDialect` initialized **without** `ENABLE_SECP_OPS` (e.g., using `MEMPOOL_MODE`).
3. Observe that `op_secp256k1_verify` executes and returns a successful `Reduction` — identical to the result when `ENABLE_SECP_OPS` is set — instead of returning `EvalErr::Unimplemented` or being routed to `unknown_operator`.
4. Repeat with `0x1c3a8f00` for secp256r1_verify.

The root cause is at `src/chia_dialect.rs` lines 175–182: the match on 4-byte opcodes dispatches directly to the secp operator functions without consulting `flags.contains(ClvmFlags::ENABLE_SECP_OPS)`, while the parallel 1-byte dispatch at lines 248–249 does perform that check. [5](#0-4)
