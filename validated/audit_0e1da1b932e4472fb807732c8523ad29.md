### Title
`op_unknown` Cost Multiplier Applied After Budget Check, Allowing Returned Cost to Exceed `max_cost` - (File: `src/more_ops.rs`)

---

### Summary

In `op_unknown` (`src/more_ops.rs`), the per-operator budget check (`check_cost`) is applied to the **pre-multiplication** base cost, but the cost returned to the interpreter is `base_cost * (cost_multiplier + 1)`. Because the multiplier is applied **after** the budget check, `op_unknown` can return `Ok` with a cost up to `u32::MAX` even when the remaining execution budget is as small as 1. This is a direct structural analog to the pool fee bug: the bound is enforced on a component, not on the final computed value.

---

### Finding Description

`op_unknown` decodes two fields from the opcode bytes: a `cost_function` (2 bits) that selects a base-cost formula, and a `cost_multiplier` (up to 4 bytes, i.e., up to `u32::MAX` as `u64`) that scales the result. [1](#0-0) 

The base cost is computed for `cost_function` 0–3, with intermediate `check_cost` calls inside each branch using the caller-supplied `max_cost` (the remaining budget): [2](#0-1) 

After the branch, a final `check_cost` is called on the **pre-multiplication** base cost, and only then is the multiplier applied: [3](#0-2) 

The sequence is:
1. `check_cost(cost, max_cost)` — enforces `cost ≤ max_cost` (pre-multiplication)
2. `cost *= cost_multiplier + 1` — scales cost by up to `u32::MAX + 1`
3. `if cost > u32::MAX` → error; else return `Ok(Reduction(cost, nil))`

The only upper bound on the **returned** cost is `u32::MAX` (~4.3 billion), not `max_cost`. The `check_cost` at line 260 is structurally equivalent to checking `base_fee` before adding `fee_adjustment` in the pool bug: the final value is never bounded relative to the budget.

The `max_cost` passed into `op_unknown` is the **remaining** budget (`effective_max_cost - current_cost`), computed in `apply_op`: [4](#0-3) 

So `op_unknown` can return `Ok` with a cost of up to `u32::MAX` when the remaining budget is 1.

---

### Impact Explanation

The broken invariant is: *an operator must not return `Ok` with a cost exceeding the remaining budget.* `op_unknown` violates this invariant whenever `cost_multiplier` is large enough that `base_cost * (cost_multiplier + 1) > max_cost`.

The returned over-budget cost is added to the running total in `run_program`: [5](#0-4) 

The loop-level check catches the excess on the **next** iteration, so the program is ultimately rejected. However:

- The per-operator budget enforcement in `op_unknown` is broken: the function returns `Ok` when it should return `Err(CostExceeded)`.
- Inside a **softfork guard**, `effective_max_cost = guard.expected_cost`. An `op_unknown` with a large multiplier that passes the pre-multiplication check but returns a cost exceeding the guard's budget causes `CostExceeded` to be raised from the loop rather than from the operator itself. This can cause divergence in how cost-mismatch errors are attributed and surfaced between implementations that check cost at different granularities.
- An attacker-controlled CLVM program can craft an unknown opcode byte sequence (attacker-controlled bytes → `op_unknown` dispatch) with a `cost_multiplier` near `u32::MAX` and a tiny base cost, causing the operator to return `Ok` with a cost of `u32::MAX` against a budget of 1. The `check_cost` guard is entirely ineffective for the final cost. [6](#0-5) 

---

### Likelihood Explanation

The entry path is direct and attacker-controlled: any CLVM program byte sequence that encodes an unknown opcode (not `0xffff`-prefixed, not empty) with bytes structured to set a large `cost_multiplier` field triggers this path. The `op_unknown` function is reachable from `run_program` via `dialect.op(...)` whenever the opcode is not a known built-in. No special privileges or configuration are required; any caller of `run_program` with `allow_unknown_ops` enabled (lenient/consensus mode) is affected.

---

### Recommendation

Move `check_cost` to **after** the multiplication, checking the final cost against `max_cost` before returning:

```rust
check_cost(cost, max_cost)?;          // pre-multiplication guard (keep for early exit)
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}
check_cost(cost, max_cost)?;          // ADD: enforce budget on final cost
Ok(Reduction(cost as Cost, allocator.nil()))
```

This mirrors the remediation recommended for the pool fee bug: enforce the bound on the **final** computed value, not on a component.

---

### Proof of Concept

Craft a 5-byte unknown opcode:
- Bytes `[0xFF, 0xFF, 0xFF, 0xFF, 0x00]`: last byte `0x00` → `cost_function = 0` (base cost = 1), prefix bytes `[0xFF, 0xFF, 0xFF, 0xFF]` → `cost_multiplier = u32::MAX = 4294967295`.
- `check_cost(1, max_cost)` passes for any `max_cost ≥ 1`.
- `cost = 1 * (4294967295 + 1) = 4294967296 > u32::MAX` → `EvalErr::Invalid` (caught by the u32 guard).

Use `[0x7F, 0xFF, 0xFF, 0xFF, 0x00]` instead: `cost_multiplier = 0x7FFFFFFF = 2147483647`.
- `cost = 1 * 2147483648 = 2147483648 ≤ u32::MAX` → returns `Ok(Reduction(2147483648, nil))`.
- With `max_cost` (remaining budget) = 2, `check_cost(1, 2)` passes, but the returned cost is 2,147,483,648 — over a billion times the remaining budget.
- `op_unknown` returns `Ok` when it should return `Err(CostExceeded)`. [7](#0-6)
