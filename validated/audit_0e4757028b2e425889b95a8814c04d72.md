### Title
Unsafe Narrowing Cast from `ShardId` (`u64`) to `u16` Silently Corrupts `CongestionInfoV1::allowed_shard` in Chunk Headers — (`File: core/primitives/src/congestion_info.rs`)

---

### Summary

`CongestionInfoV1::allowed_shard` is a `u16` field that is Borsh-serialized into every chunk header as part of the congestion-control protocol. It is populated by calling `allowed_shard.into()` where `allowed_shard` is a `ShardId` (`u64`). The `Into<u16> for ShardId` implementation performs a silent `as u16` truncation with no bounds check. If any live `ShardId` value exceeds `u16::MAX` (65 535), the stored field silently encodes a different shard, breaking the protocol invariant that guarantees forward progress during full congestion.

---

### Finding Description

`ShardId` is a newtype over `u64`: [1](#0-0) 

Its `Into<u16>` implementation is an unchecked narrowing cast: [2](#0-1) 

`CongestionInfoV1` stores the allowed shard as `u16`: [3](#0-2) 

`finalize_allowed_shard` writes a live `ShardId` into that field via the truncating conversion: [4](#0-3) 

`bootstrap_congestion_info` does the same at bootstrap time: [5](#0-4) 

The field is then Borsh-serialized into the chunk header and read back by every validator. The comparison that enforces the "one allowed shard" invariant during full congestion is: [6](#0-5) 

`allowed_shard()` returns the raw `u16`: [7](#0-6) 

`ShardId::from(u16)` zero-extends, so if the stored `u16` is the truncated low 16 bits of a real `ShardId`, the comparison `sender_shard == ShardId::from(self.info.allowed_shard())` will never match the intended shard.

The same `Into<u32>` truncation exists for `ShardUId::new`, which uses the `u32` shard_id as a database key prefix: [8](#0-7) [9](#0-8) 

---

### Impact Explanation

When a shard is fully congested, the protocol guarantees exactly one "allowed shard" can still forward receipts to prevent deadlock. If `allowed_shard` is silently truncated:

- The stored `u16` encodes a different (or non-existent) shard ID.
- `outgoing_gas_limit` returns `Gas::ZERO` for every sender, including the intended allowed shard.
- No shard can make forward progress: the network deadlocks on that shard.
- The corrupted value is Borsh-serialized into the chunk header and propagated to all validators, so the divergence is consensus-wide, not local.

The secondary `ShardUId` truncation (`u64 → u32`) would cause two distinct shards to share the same 8-byte DB key prefix, corrupting trie state lookups for both.

---

### Likelihood Explanation

`ShardLayoutV3` assigns shard IDs as monotonically increasing `u64` values across resharding events. Current mainnet has ~6 shards. Reaching `u16::MAX` (65 535) requires 65 535 cumulative resharding splits; reaching `u32::MAX` requires ~4 billion. At any realistic resharding cadence this is not imminent. However:

- The type system provides **no compile-time or runtime guard** against the truncation.
- The `Into<u16>` impl is unconditional and silently lossy.
- As dynamic resharding (`ShardLayoutV3`) is now in production, shard IDs will grow unboundedly over the protocol's lifetime.
- The bug is structural: it will silently activate the moment any shard ID crosses the threshold, with no warning.

---

### Recommendation

Replace the unchecked `as u16` / `as u32` casts with checked conversions that panic or return an error at the boundary:

```rust
// core/primitives-core/src/types.rs
impl TryInto<u16> for ShardId {
    type Error = TryFromIntError;
    fn try_into(self) -> Result<u16, Self::Error> {
        u16::try_from(self.0)
    }
}
```

In `finalize_allowed_shard` and `bootstrap_congestion_info`, use `u16::try_from(allowed_shard.0).expect("ShardId exceeds u16::MAX")` (or propagate the error). Alternatively, widen `CongestionInfoV1::allowed_shard` to `u64` in a new `CongestionInfo::V2` variant, which is the correct long-term fix given that `ShardId` is already `u64`.

---

### Proof of Concept

```
ShardId(65536_u64).into() as u16
= 65536u64 as u16
= 0u16                          // silent truncation

// stored in CongestionInfoV1::allowed_shard = 0
// real intended allowed shard = ShardId(65536)
// comparison: ShardId::from(0u16) == ShardId(65536)  → false
// → every sender gets Gas::ZERO → full deadlock
```

The Borsh encoding of `CongestionInfoV1` writes `allowed_shard` as a 2-byte little-endian `u16`: [3](#0-2) 

So the divergent bytes are bytes 32–33 of the serialized `CongestionInfoV1` blob embedded in every chunk header once any `ShardId ≥ 65536` is selected as the allowed shard.
