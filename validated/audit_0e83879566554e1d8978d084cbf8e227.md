### Title
Unchecked ERC20 `transferFrom` Return Value Enables USDC.e Drain from DirectDepositV1 - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` uses a raw `IERC20Base.transferFrom` call without checking its return value. If the USDC `transferFrom` returns `false` instead of reverting (a valid ERC20 behavior), the function silently continues, draining USDC.e from the target `DirectDepositV1` address and transferring it to the caller — with no USDC provided in exchange.

---

### Finding Description

The function `replaceUsdcEWithUsdc` in `ContractOwner.sol` is a publicly callable migration helper (gated only by `block.chainid == 57073`) that is intended to swap USDC.e held in a `DirectDepositV1` (DDA) address for native USDC. The intended flow is:

1. Caller provides USDC via `transferFrom` into the DDA.
2. USDC.e is withdrawn from the DDA to `ContractOwner`.
3. USDC.e is forwarded to the caller.

The critical flaw is at step 1:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
```

This is a raw call to `transferFrom` whose return value is **never checked**. The rest of the codebase consistently uses `ERC20Helper.safeTransferFrom`, which wraps the call and requires a truthy return value. This single call deviates from that pattern. [1](#0-0) 

If `transferFrom` returns `false` (e.g., caller has zero USDC allowance or zero balance), execution continues to:

```solidity
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
```

`DirectDepositV1.withdraw` is `onlyOwner`, and `ContractOwner` is the owner of every DDA it deploys, so this call succeeds unconditionally. It transfers all USDC.e from the DDA to `ContractOwner`, which then forwards it to the caller via `safeTransfer`. [2](#0-1) 

The safe transfer helper used everywhere else in the codebase explicitly requires a truthy return: [3](#0-2) 

---

### Impact Explanation

An unprivileged caller on Ink chain (chainid 57073) can call `replaceUsdcEWithUsdc` for any subaccount whose `directDepositV1Address` is non-zero and holds USDC.e. If the USDC token's `transferFrom` returns `false` on failure (rather than reverting), the caller receives the full USDC.e balance of that DDA without providing any USDC. This is a direct, unbounded theft of user collateral assets held in DDA contracts.

---

### Likelihood Explanation

The function has no access control beyond a chain ID check — any address on Ink mainnet can call it. The exploitability depends on whether the specific USDC contract at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink returns `false` or reverts on a failed transfer. Many bridged or non-Circle USDC variants follow the original ERC20 spec and return `false` rather than reverting. Even if the current deployment reverts, the code is structurally broken: it is the only `transferFrom` call in the entire codebase that bypasses `ERC20Helper.safeTransferFrom`, creating a latent vulnerability that activates if the token contract is ever upgraded or replaced. [4](#0-3) 

---

### Recommendation

Replace the raw `transferFrom` call with the `ERC20Helper.safeTransferFrom` wrapper already used throughout the codebase:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [5](#0-4) 

---

### Proof of Concept

1. A subaccount `S` has a deployed `DirectDepositV1` at address `DDA` holding 1000 USDC.e.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(S)` with zero USDC allowance.
3. `IERC20Base(usdc).transferFrom(attacker, DDA, 1000)` returns `false` — no revert.
4. `DirectDepositV1(DDA).withdraw(usdcE)` executes: 1000 USDC.e moves from `DDA` → `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, 1000)` executes: 1000 USDC.e moves from `ContractOwner` → attacker.
6. Attacker receives 1000 USDC.e having provided 0 USDC. The subaccount's DDA is drained. [1](#0-0)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
