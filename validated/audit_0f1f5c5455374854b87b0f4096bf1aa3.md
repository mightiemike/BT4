### Title
`ContractOwner` Lacks `receive()` Function, Making Native ETH Withdrawal from `DirectDepositV1` Always Revert - (File: `core/contracts/ContractOwner.sol`)

### Summary
`ContractOwner` has no `receive()` or `fallback()` function. When `withdrawFromDirectDepositV1` is called with `token == address(0)` to recover native ETH from a `DirectDepositV1` (DDA) contract, it invokes `DirectDepositV1.withdrawNative()`, which attempts to push ETH to `msg.sender` (i.e., `ContractOwner`) via a low-level call. Because `ContractOwner` cannot accept ETH, this call always fails, permanently locking any native ETH held in the DDA.

### Finding Description
`ContractOwner.withdrawFromDirectDepositV1` handles the native-ETH case as follows:

```solidity
// ContractOwner.sol L628-L636
if (token == address(0)) {
    uint256 preBalance = address(this).balance;
    DirectDepositV1(directDepositV1).withdrawNative();   // ← always reverts
    uint256 postBalance = address(this).balance;
    require(postBalance > preBalance, "empty");
    (bool success, ) = msg.sender.call{value: postBalance - preBalance}("");
    require(success, "xfer");
}
```

`DirectDepositV1.withdrawNative()` is:

```solidity
// DirectDepositV1.sol L108-L112
function withdrawNative() external onlyOwner {
    uint256 balance = address(this).balance;
    (bool success, ) = msg.sender.call{value: balance}("");
    require(success, "Failed to transfer native token to owner");
}
```

`msg.sender` inside `withdrawNative()` is `ContractOwner`. A grep over `ContractOwner.sol` returns zero matches for `receive()` or `fallback()`, confirming the contract has no ETH-accepting hook. The low-level call therefore returns `success = false`, and `require(success, "Failed to transfer native token to owner")` reverts the entire transaction.

`ContractOwner` is the deployer-owner of every `DirectDepositV1` instance (created via `new DirectDepositV1{salt: ...}(...)` in `createDirectDepositV1`), so this is the only privileged withdrawal path for native ETH.

### Impact Explanation
Any native ETH that accumulates in a `DirectDepositV1` contract — whether sent directly before the `receive()` wrapping hook fires, via `selfdestruct` from another contract, or as a coinbase reward — is permanently inaccessible. The `withdrawFromDirectDepositV1(subaccount, address(0))` code path is entirely dead: it will always revert at the `withdrawNative()` call, and there is no alternative recovery path for native ETH in the DDA.

### Likelihood Explanation
The `DirectDepositV1.receive()` function wraps incoming ETH to wrapped-native tokens, so under normal operation the DDA's raw ETH balance stays at zero. However, ETH can still accumulate via `selfdestruct` (which bypasses `receive()`), or if the wrapping call to `wrappedNative` fails silently (the constructor path emits `NativeTokenTransferFailed` and continues). The `withdrawNative` function exists precisely to handle these edge cases, but the missing `receive()` on `ContractOwner` makes it permanently non-functional.

### Recommendation
Add a `receive()` function to `ContractOwner` so it can accept ETH forwarded from `DirectDepositV1.withdrawNative()`:

```solidity
receive() external payable {}
```

This mirrors the fix recommended in the referenced report for `AxelarBridgedGovernor`.

### Proof of Concept
1. A `DirectDepositV1` DDA is deployed for some `subaccount` via `ContractOwner.createDirectDepositV1`.
2. ETH enters the DDA's raw balance (e.g., via `selfdestruct` from another contract targeting the DDA address).
3. The multisig owner calls `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))`.
4. Internally, `DirectDepositV1.withdrawNative()` executes `ContractOwner.call{value: balance}("")`.
5. `ContractOwner` has no `receive()` or `fallback()` — the call returns `success = false`.
6. `require(success, "Failed to transfer native token to owner")` reverts.
7. The ETH remains permanently locked in the DDA with no recovery path.
