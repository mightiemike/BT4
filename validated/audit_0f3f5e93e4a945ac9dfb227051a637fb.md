Audit Report

## Title
`SwapAllowlistExtension` Per-User Swap Allowlist Bypassed via Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the allowlist, but `MetricOmmPool.swap` sets `sender` to `msg.sender` — which is the router, not the end user, for router-mediated swaps. Once a pool admin allowlists the router (required for normal router usage), every user who calls through the router passes the check regardless of individual allowlist status, completely neutralising the per-user gate.

## Finding Description
In `MetricOmmPool.swap`, `msg.sender` (the router) is passed as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]` — i.e., `allowedSwapper[pool][router]`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` with `params.recipient` as the recipient but never forwards the original caller's identity; the router itself becomes `msg.sender` to the pool: [4](#0-3) 

The router stores the original user only in transient callback context for payment purposes, not in any field visible to the pool or extension. No existing guard in the extension or pool recovers the original user's address.

## Impact Explanation
A pool admin deploying `SwapAllowlistExtension` intends to restrict swaps to specific addresses (KYC-gated, OTC, protocol-internal). Because the extension checks the router's address rather than the end user's, any unpermitted user can bypass this restriction by routing through `MetricOmmSimpleRouter`. The attacker executes swaps the pool admin explicitly prohibited, extracting tokens at oracle-anchored prices without authorisation. This is a direct admin-boundary break with fund-impacting consequences.

## Likelihood Explanation
The bypass requires the router to be allowlisted on the pool. This is a natural and expected configuration: any pool admin who wants to support standard user flows through the periphery router must allowlist it. The moment they do, the per-user gate collapses for all users. No special privileges are required beyond knowing the pool uses `SwapAllowlistExtension` and that the router is allowlisted.

## Recommendation
The extension must verify the original end user, not the intermediary contract. The most direct fix: the router encodes `msg.sender` into `extensionData` before calling `pool.swap`, and `SwapAllowlistExtension.beforeSwap` decodes and checks it when `sender` is a known router. Alternatively, the pool interface could accept an explicit `originator` parameter that the router populates with `msg.sender`, and the extension checks `originator` rather than `sender`.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true      // alice is the only permitted user
  allowedSwapper[pool][router] = true     // required for normal router-mediated swaps

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, ..., recipient: bob})

  pool.swap(recipient=bob, ...) called with msg.sender = router
  _beforeSwap(sender=router, ...)
  extension.beforeSwap(sender=router, ...)
    → allowedSwapper[pool][router] == true → passes

  bob receives swap output; allowlist is bypassed.
```
