Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Real Trader, Allowlist Fully Bypassed via MetricOmmSimpleRouter - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which equals `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the real trader. If the pool admin allowlists the router — the only practical choice for a live pool using the standard periphery — every non-allowlisted user can bypass the curated gate by calling through the router.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` forwards this value verbatim as `sender` to every configured extension. `SwapAllowlistExtension.beforeSwap` then checks: [2](#0-1) 

where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`.

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly with no identity forwarding: [3](#0-2) 

The real trader's identity (`msg.sender` of `exactInputSingle`) is stored in transient storage only for the payment callback via `_setNextCallbackContext`, but is never passed to the pool's `swap()` call or to any extension. The pool therefore sees `msg.sender = router`, so `sender` delivered to the extension is the router address. The allowlist lookup becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

This creates a binary trap for the pool admin:

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | All legitimate users are blocked from using the standard periphery |
| **Allowlist the router** | Every non-allowlisted user bypasses the gate by routing through the router |

The second branch is the exploitable path: once the router is allowlisted, the allowlist provides zero protection against any public user.

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to known counterparties (e.g., KYC'd traders, whitelisted market makers, or protocol-controlled addresses) is fully open to arbitrary swappers the moment the router is allowlisted. Any user can execute trades the pool admin intended to prevent, draining LP-owned liquidity at oracle prices. This is a direct loss of LP principal and a broken core pool invariant (curation policy), meeting the "Broken core pool functionality causing loss of funds" and "Admin-boundary break bypassed by an unprivileged path" criteria.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface for the protocol. Any pool admin who wants allowlisted users to use the standard periphery must allowlist the router. The bypass requires no special privileges, no flash loans, and no multi-step setup — a single `exactInputSingle` call from any EOA suffices. The condition (router allowlisted) is the expected production state for any live pool using the standard periphery.

## Recommendation

The extension must gate on the real trader identity, not the intermediary. The cleanest fix is for the router to encode `msg.sender` into `extensionData` before calling the pool, and for the extension to decode and check it. The router already uses transient storage for callback context (`_setNextCallbackContext` stores the real payer); the same real-caller identity should be forwarded to extensions via `extensionData`. Alternatively, the extension could be redesigned to check `recipient` instead of `sender`, though this is a weaker identity proxy.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured in beforeSwap order
  - Pool admin calls setAllowedToSwap(pool, router, true)   // to allow legitimate users
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)
  - Pool has liquidity

Attack:
  - attacker (not allowlisted) calls:
      router.exactInputSingle({pool: pool, ..., recipient: attacker, ...})
  - Router calls pool.swap(attacker, ...) with msg.sender = router
  - Pool calls _beforeSwap(sender=router, ...)
  - Extension checks: allowedSwapper[pool][router] == true  → passes
  - Swap executes; attacker receives output tokens
  - Allowlist policy is completely bypassed
``` [4](#0-3) [5](#0-4) [6](#0-5)
