### Title
Dogecoin Fork Difficulty Retarget Uses Mainchain Ancestor Instead of Fork Ancestor — (`contract/src/dogecoin.rs`)

### Summary

`get_next_work_required` in the Dogecoin module calls `blocks_getter.get_header_by_height(height_first)` to obtain the `first_block_time` for the Digishield retarget calculation. The `BlocksGetter` implementation always resolves this height against `mainchain_height_to_header`, not the fork's actual ancestor chain. For any fork block at depth ≥ 2 past the divergence point (and at height ≥ 145,001 where `difficulty_adjustment_interval = 1`), the contract uses the wrong timestamp, producing an incorrect `expected_bits`. An attacker can exploit this to submit fork blocks whose actual difficulty is lower than the correct retarget requires, accumulating `chain_work` faster than legitimately possible and triggering a fraudulent reorg.

The code itself acknowledges this with a `TODO` comment at the exact line in question.

### Finding Description

**Step 1 — `get_header_by_height` is mainchain-only.** [1](#0-0) 

`get_header_by_height` unconditionally reads from `mainchain_height_to_header`. There is no fork-aware path.

**Step 2 — `get_next_work_required` calls it for every fork block at height ≥ 145,001.** [2](#0-1) 

When `prev_block_header.block_height >= 145_000`, `difficulty_adjustment_interval = 1`, so every block is a retarget block and `blocks_to_go_back = 1`. [3](#0-2) 

`height_first = prev_block_header.block_height - 1`. For a fork block at height H+2 (fork diverges at H), `height_first = H`. The contract fetches the **mainchain** block at H, not the fork block at H.

**Step 3 — The TODO comment confirms the known design gap.** [4](#0-3) 

**Step 4 — Incorrect `first_block_time` flows into `calculate_next_work_required`.** [5](#0-4) 

`modulated_timespan = prev_block_header.block_header.time - first_block_time`. If `first_block_time` is the mainchain block's timestamp rather than the fork ancestor's, the timespan—and therefore the computed target—is wrong.

**Step 5 — Fork blocks are accepted before any reorg.** [6](#0-5) 

Fork blocks are stored and their `chain_work` is compared to the mainchain tip. If the fork's `chain_work` exceeds the mainchain's, `reorg_chain` is triggered. The difficulty check (`check_pow` → `get_next_work_required`) runs **before** storage, so an incorrectly computed `expected_bits` directly gates what fork blocks are accepted.

### Impact Explanation

**Concrete attack (Dogecoin mainnet, height ≥ 145,001):**

Let mainchain block at height H have timestamp `T_main`. The attacker creates:
- Fork block at H with timestamp `T_fork_H = T_main + 300s` (5 minutes later; valid under the 2-hour future cap and MTP rules)
- Fork block at H+1 with timestamp `T_fork_H+1 = T_main + 360s`

**Correct retarget for fork block at H+2:**
- `actual_timespan = T_fork_H+1 - T_fork_H = 60s`
- `modulated_timespan = 60 + (60 - 60)/8 = 60s` → normal difficulty

**Buggy retarget (contract uses mainchain block at H):**
- `actual_timespan = T_fork_H+1 - T_main = 360s` → clamped to 300s (max)
- `modulated_timespan = 60 + (300 - 60)/8 = 90s` → **50% easier difficulty** (target × 1.5)

The attacker mines fork blocks at 50% of the correct difficulty per block. Over a sequence of fork blocks, this compounds: the attacker can accumulate `chain_work` exceeding the mainchain with substantially less mining power than required, triggering `reorg_chain` and replacing the canonical chain with a fraudulent fork.

### Likelihood Explanation

- The attack path is a standard public `submit_blocks` call — no privileged role required.
- The attacker only needs to submit fork blocks starting at any height ≥ 145,001 (all of current Dogecoin mainnet).
- Timestamp manipulation is trivial; the only constraint is the 2-hour future cap and MTP check, both of which the attacker can satisfy.
- The per-block difficulty reduction (up to 50%) is large enough to be practically exploitable with modest hash rate.
- The TODO comment confirms the developers identified this gap but left it unresolved.

### Recommendation

Replace the `get_header_by_height` call in `get_next_work_required` with a chain-traversal that walks back through `get_prev_header` from `prev_block_header` by `blocks_to_go_back` steps. This correctly resolves the ancestor on whichever chain (main or fork) the block being validated belongs to, matching the reference Dogecoin implementation's `GetAncestor` behavior.

### Proof of Concept

```
State: Dogecoin mainnet contract, mainchain at height H+10 (H ≥ 145,001).
Mainchain block at H has timestamp T_main.

1. Submit fork_H: prev = mainchain block at H-1,
   timestamp = T_main + 300s, bits = mainchain_bits.
   → Accepted (height_first = H-1, shared ancestor, correct).

2. Submit fork_H+1: prev = fork_H,
   timestamp = T_main + 360s.
   → height_first = H.
   → Contract fetches mainchain block at H (timestamp T_main).
   → actual_timespan = 360s → clamped to 300s → modulated = 90s.
   → expected_bits = bits encoding target × (90/60) = target × 1.5.
   → Attacker mines fork_H+1 satisfying this easier target. Accepted.

3. Repeat for fork_H+2, fork_H+3, …
   Each block uses mainchain timestamp at height_first instead of fork ancestor.
   Attacker mines each block at up to 50% reduced difficulty.

4. Once fork chain_work > mainchain chain_work, reorg_chain() fires.
   Fraudulent fork becomes canonical chain.
```

The invariant broken: **difficulty retarget for a fork block must use the fork's own ancestor at `height_first`, not the mainchain block at that height.**
