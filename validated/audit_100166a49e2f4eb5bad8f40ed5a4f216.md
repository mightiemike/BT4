### Title
`verify_transaction_inclusion` Allows Caller-Controlled `confirmations = 0` to Bypass Confirmation Guard — (`contract/src/lib.rs`)

### Summary

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations` value with no lower-bound enforcement. When a caller passes `confirmations = 0`, the confirmation-depth guard evaluates to `any_u64 >= 0`, which is always `true` in Rust for `u64`. This allows any unprivileged NEAR caller to obtain a `true` proof result for a transaction that has zero confirmations — i.e., one that could still be reorganized out of the chain.

---

### Finding Description

In `verify_transaction_inclusion`, the confirmation check is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [1](#0-0) 

The only other check on `confirmations` is an upper-bound guard:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [2](#0-1) 

There is **no lower-bound check**. When `args.confirmations = 0`, the expression `(any u64 value) >= 0u64` is trivially `true` in Rust because `u64` is unsigned and can never be negative. The guard is completely bypassed.

`verify_transaction_inclusion_v2` delegates to the same function after its coinbase-proof check, so it inherits the same flaw:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

Both functions are callable by any unprivileged NEAR account — they carry only `#[pause]`, not an access-control role restriction.

---

### Impact Explanation

The confirmation depth is the primary security parameter that protects consumer contracts against Bitcoin reorganizations. A consumer contract (e.g., a bridge or cross-chain settlement contract) that calls `verify_transaction_inclusion` to gate fund releases relies on this check to ensure the referenced Bitcoin transaction is buried deep enough to be irreversible. With `confirmations = 0`, the function returns `true` for any transaction present in any main-chain block, including the most recently submitted one. An attacker can:

1. Broadcast a Bitcoin transaction and have the relayer submit its block.
2. Immediately call `verify_transaction_inclusion` with `confirmations = 0`.
3. Receive `true` from the contract.
4. Trigger the consumer contract's release logic before the transaction has any safety margin against reorgs.

The corrupted invariant is the proof result: `verify_transaction_inclusion` returns `true` when the security guarantee ("transaction is confirmed by N blocks") has not been met.

---

### Likelihood Explanation

The entry path is fully unprivileged — any NEAR account can call `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` with `confirmations = 0`. No special role, staked position, or social engineering is required. The `ProofArgs` struct is caller-supplied and serialized via Borsh, so the caller has complete control over the `confirmations` field. Any consumer contract that does not independently enforce a minimum confirmation count is directly exposed.

---

### Recommendation

Add a minimum confirmation lower-bound check at the start of `verify_transaction_inclusion`:

```rust
require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    "Confirmations must be at least MIN_CONFIRMATIONS"
);
```

where `MIN_CONFIRMATIONS` is a protocol constant (e.g., `1` at minimum, or a chain-appropriate value such as `6` for Bitcoin). Alternatively, enforce that `confirmations >= 1` unconditionally, since zero confirmations provides no reorganization safety and should never be a valid input for a security-critical proof function.

---

### Proof of Concept

1. The relayer submits a single Bitcoin block containing a target transaction `tx_id`. The block is accepted and becomes the chain tip at height `H`.
2. An attacker (or the transaction sender) immediately calls `verify_transaction_inclusion` with:
   - `tx_block_blockhash` = the just-submitted block hash
   - `tx_id` = the target transaction hash
   - `merkle_proof` = valid Merkle path for `tx_id`
   - `confirmations = 0`
3. The upper-bound check passes: `0 <= gc_threshold`.
4. The confirmation check evaluates as: `(H - H) + 1 = 1 >= 0` → `true`.
5. The Merkle proof is valid, so the function returns `true`.
6. The consumer contract releases funds, even though the Bitcoin transaction has zero confirmations and could be reorganized away. [4](#0-3)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
