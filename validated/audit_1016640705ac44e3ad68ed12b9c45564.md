Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which resolves to `msg.sender` of `MetricOmmPool.swap` — the router contract, not the end user. When a pool admin allowlists the router to enable router-mediated access for their curated users, every unprivileged user can bypass the per-user allowlist by routing through the same router, because the extension sees only the router's address and cannot distinguish individual callers.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs the following check:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` is the pool (the extension is called by the pool). `sender` is the first argument forwarded by `ExtensionCalling._beforeSwap`:

```solidity
// ExtensionCalling.sol L160-165
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(
        IMetricOmmExtensions.beforeSwap,
        (sender, ...)
    )
);
```

That `sender` originates from `MetricOmmPool.swap`:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // always the immediate caller of pool.swap()
    ...
);
```

All four `MetricOmmSimpleRouter` entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) call `pool.swap(...)` directly without embedding the end user's address anywhere in the call. For example:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData   // end user identity not included
    );
```

The router stores `msg.sender` only in transient storage as the payer for the callback (`_setNextCallbackContext`), but this is never read by the extension. The extension receives only the router's address as `sender` for every user who routes through it.

**Exploit path:**
1. Pool admin deploys pool with `SwapAllowlistExtension`.
2. Admin allowlists `user1`: `allowedSwapper[pool][user1] = true`.
3. Admin allowlists the router so `user1` can use it: `allowedSwapper[pool][router] = true`.
4. `user2` (not individually allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle`.
5. Router calls `pool.swap(...)` → pool calls `_beforeSwap(router, ...)` → extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. `user2` successfully trades in a pool they were never authorized to access.

No existing guard prevents this: the extension has no mechanism to recover the original end user's address from `extensionData` or any other channel, and the router provides no such attestation.

## Impact Explanation
The allowlist invariant — *only individually authorized addresses may swap in a curated pool* — is broken for all router-mediated swaps once the router is allowlisted. Any unprivileged user can trade in a pool that was intended to be access-controlled, violating the pool admin's access policy. This constitutes an admin-boundary break where an unprivileged router caller bypasses a factory/pool-level role check, fitting the contest's allowed impact gate.

## Likelihood Explanation
The pool admin has a direct, legitimate reason to allowlist the router: their authorized users need router UX (multi-hop, slippage protection, deadline checks). The admin allowlists the router believing it is a trusted periphery that will enforce the allowlist downstream — but the router performs no such check. This is a realistic and foreseeable misconfiguration, not a malicious or exotic setup. All four router entry points are affected, and the bypass is repeatable by any unprivileged caller with no special capability required.

## Recommendation
The extension must check the economically relevant actor — the end user — rather than the immediate pool caller. Two concrete approaches:

1. **Router-attested identity via `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling `pool.swap`. The extension decodes and verifies this field. A convention (e.g., a well-known ABI prefix) must be established between the router and the extension.

2. **Transient-storage attestation**: The router writes the real caller into a transient storage slot keyed by pool address before calling `pool.swap`. The extension reads that slot (via a known interface on the router) to recover the true user. This mirrors how the router already stores the payer via `_setNextCallbackContext`.

As a minimum safe documentation fix: explicitly prohibit adding the router address to any pool's swap allowlist, and consider a factory-level guard that reverts if the router is registered as an allowlisted swapper on a curated pool.
