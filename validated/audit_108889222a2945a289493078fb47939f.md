### Title
Missing ClassTrace Validation in `processReceivedPacket` Allows Degenerate ClassID to Permanently Lock NFTs — (`x/nft-transfer/keeper/packet.go`)

---

### Summary

`processReceivedPacket` calls `ParseClassTrace` on the attacker-controlled `prefixedClassID` but never calls `classTrace.Validate()`. A malicious counterparty chain can send a packet whose `ClassId` field is a slash-only string (e.g. `"///"`). This passes `ValidateBasic()`, produces a `ClassTrace` with an empty `BaseClassId`, stores it in the KV store, and mints an NFT under a degenerate `ibc/{hash}` voucher class ID. The original NFT on the source chain is permanently escrowed and unrecoverable.

---

### Finding Description

**Step 1 — `ValidateBasic()` does not reject `"///"`**

`OnRecvPacket` calls `data.ValidateBasic()` before delegating to `processReceivedPacket`: [1](#0-0) 

`ValidateBasic()` only rejects a classID that is blank/whitespace: [2](#0-1) 

`"///"` is not blank, so it passes.

**Step 2 — `ParseClassTrace("nft/channel-0///")` produces empty `BaseClassId`**

In `processReceivedPacket`, the destination port/channel prefix is prepended: [3](#0-2) 

`ParseClassTrace` splits on `"/"` and assigns the last element as `BaseClassId`: [4](#0-3) 

For input `"nft/channel-0///"`:
- `strings.Split("nft/channel-0///", "/")` → `["nft", "channel-0", "", "", ""]`
- `BaseClassId = classSplit[last] = ""`
- `Path = "nft/channel-0//"`

Result: `ClassTrace{Path: "nft/channel-0//", BaseClassId: ""}`.

**Step 3 — `Validate()` is never called in `processReceivedPacket`**

`Validate()` explicitly catches this case: [5](#0-4) 

But `processReceivedPacket` never calls it. After `ParseClassTrace`, the code immediately calls `IBCClassID()`: [6](#0-5) 

**Step 4 — `IBCClassID()` returns a valid-looking `"ibc/{hash}"` for non-empty `Path`** [7](#0-6) 

Since `Path = "nft/channel-0//"` is non-empty, `IBCClassID()` returns `"ibc/{sha256("nft/channel-0///")}"`—a well-formed IBC class ID derived from a degenerate trace.

**Step 5 — Degenerate trace is stored and NFT is minted** [8](#0-7) 

`SetClassTrace` stores `ClassTrace{Path: "nft/channel-0//", BaseClassId: ""}`, `IssueDenom` creates the denom, and `MintNFT` mints the token under `"ibc/{hash}"`.

**Step 6 — Original NFT is permanently locked**

The source chain escrowed the original NFT when the packet was sent. To recover it, the receiving chain must send the voucher back. However, `GetFullClassPath()` for the degenerate trace returns `"nft/channel-0///"` (path + "/" + "" = trailing slash). When the source chain receives this return packet, `RemoveClassPrefix` strips the port/channel prefix, leaving `"//"`. `ParseClassTrace("//")` produces another degenerate trace with empty `BaseClassId`, and `TransferOwner` fails because no NFT with that computed class ID exists in escrow. The original NFT is permanently locked.

---

### Impact Explanation

A malicious counterparty chain sends one IBC NFT packet with `ClassId = "///"`. The receiving chain:
1. Stores a degenerate `ClassTrace{Path: "nft/channel-0//", BaseClassId: ""}` in its KV store.
2. Mints an NFT under `"ibc/{hash}"` that is unresolvable to any legitimate origin class.
3. Permanently locks the original NFT in escrow on the source chain with no recovery path.

This corrupts NFT ownership state and causes direct, permanent fund loss for the NFT holder.

---

### Likelihood Explanation

Requires a malicious or non-standard counterparty chain that can craft IBC packets with slash-only `ClassId` values. Most legitimate chains would reject such a classID at issuance, but IBC does not guarantee counterparty chain behavior. The receiving chain is responsible for independently validating all incoming packet data. The missing `Validate()` call is a concrete, locally reproducible code gap.

---

### Recommendation

In `processReceivedPacket`, call `classTrace.Validate()` immediately after `ParseClassTrace` and return an error if validation fails:

```go
classTrace := types.ParseClassTrace(prefixedClassID)
if err := classTrace.Validate(); err != nil {
    return err
}
```

This mirrors the pattern already used in `ClassHash` gRPC handler: [9](#0-8) 

Additionally, `ValidateBasic()` on `NonFungibleTokenPacketData` should validate the structural integrity of `ClassId` beyond a blank check.

---

### Proof of Concept

```go
// Unit test demonstrating the gap
func TestDegenerateClassTrace(t *testing.T) {
    // "///" passes ValidateBasic (not blank)
    pkt := types.NonFungibleTokenPacketData{
        ClassId:   "///",
        TokenIds:  []string{"token1"},
        TokenUris: []string{"uri1"},
        Sender:    validAddr,
        Receiver:  validAddr,
    }
    require.NoError(t, pkt.ValidateBasic()) // passes — BUG

    // After prefixing by processReceivedPacket:
    prefixed := "nft/channel-0///"
    ct := types.ParseClassTrace(prefixed)
    require.Equal(t, "", ct.BaseClassId)   // empty BaseClassId
    require.NotEmpty(t, ct.IBCClassID())   // returns "ibc/{hash}" — proceeds to mint

    // Validate() would catch it, but is never called in processReceivedPacket:
    require.Error(t, ct.Validate()) // "base class_id cannot be blank"
}
```

```go
func ParseClassTrace(rawClassID string) ClassTrace {
	classSplit := strings.Split(rawClassID, "/")

	if classSplit[0] == rawClassID {
		return ClassTrace{
			Path:        "",
			BaseClassId: rawClassID,
		}
	}

	return ClassTrace{
		Path:        strings.Join(classSplit[:len(classSplit)-1], "/"),
		BaseClassId: classSplit[len(classSplit)-1],
	}
}
```
