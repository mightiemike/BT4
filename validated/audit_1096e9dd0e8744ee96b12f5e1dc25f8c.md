### Title
Deprecated `verify_transaction_inclusion` Accepts Unvalidated Caller-Supplied `tx_id` Enabling Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function accepts a caller-supplied `tx_id` and passes it directly into `compute_root_from_merkle_proof` without any on-chain validation that it represents a real leaf-level transaction hash rather than an internal Merkle tree node. Any unprivileged NEAR caller can exploit this to obtain a `true` proof-of-inclusion result for a Bitcoin transaction that never existed.

---

### Finding Description

In `contract/src/lib.rs`, `verify_transaction_inclusion` is a public, `#[pause]`-gated (but not role-gated) function callable by any NEAR account: [1](#0-0) 

The function receives `ProofArgs` — which includes a caller-controlled `tx_id: H256` — and passes it without any structural validation directly to: [2](#0-1) 

Inside `compute_root_from_merkle_proof`, the hash is treated as a leaf and hashed upward through the tree using only the caller-supplied `tx_index` and `merkle_proof`: [3](#0-2) 

The function performs no check that `tx_id` is a leaf-level transaction hash rather than an internal node hash. The code itself documents this trust-boundary gap explicitly: [4](#0-3) 

The analog to the external report is direct: just as NFT `src` URLs are passed to `<NftThumbnail />` without validation of their origin or content type, `tx_id` is passed to `compute_root_from_merkle_proof` without validation that it is a real transaction hash. In both cases, the rendering/verification component blindly trusts caller-supplied data that crosses a trust boundary.

The `verify_transaction_inclusion_v2` function was introduced to mitigate this by requiring a matching coinbase proof of equal depth, which constrains the tree structure: [5](#0-4) 

However, the deprecated v1 function remains fully callable on-chain. `#[deprecated]` in Rust is a compile-time lint only; it imposes no runtime restriction. Any NEAR account — including a consumer contract or a direct cross-contract call — can invoke `verify_transaction_inclusion` today.

---

### Impact Explanation

**Impact: Medium**

A consumer contract that calls `verify_transaction_inclusion` to gate a cross-chain action (e.g., releasing bridged assets, confirming an atomic swap, or crediting a deposit) can be deceived into accepting a forged proof. The attacker does not need to mine any Bitcoin block; they only need to know the `merkle_root` of any mainchain block and the hash of one of its internal Merkle nodes. They supply that internal node hash as `tx_id`, set `tx_index` to the node's position in the tree, and provide the correct sibling hashes as `merkle_proof`. The function returns `true` for a transaction that does not exist.

The corrupted state is the **proof result** returned to the consumer contract: a `true` value that asserts Bitcoin transaction inclusion where none occurred.

---

### Likelihood Explanation

**Likelihood: Medium**

The function is public and requires no privileged role. Any NEAR account can call it. Consumer contracts that have not yet migrated to `verify_transaction_inclusion_v2` are directly exposed. The attack requires only knowledge of a mainchain block's Merkle tree structure, which is public Bitcoin data. No key compromise, social engineering, or privileged access is needed.

---

### Recommendation

1. **Remove or hard-block the deprecated function**: Add a `require!(false, "use verify_transaction_inclusion_v2")` body, or remove the function entirely in the next upgrade, to eliminate the callable surface.
2. **Alternatively, add a depth guard**: Require that `merkle_proof` is non-empty and that `tx_index < 2^merkle_proof.len()`, and document that callers must independently confirm `tx_id` is a leaf. This does not fully close the gap but raises the bar.
3. **Emit a deprecation panic at runtime**: Since Rust's `#[deprecated]` is compile-time only, add an explicit `env::log_str` warning or a configurable `require!` that rejects calls after a sunset block height.

---

### Proof of Concept

1. Identify any mainchain block `B` with known `merkle_root` and at least two transactions. Obtain the full Merkle tree for `B` from a Bitcoin node.
2. Select an internal node `N` at depth `d` (where `d < tree_depth`). Compute `N`'s hash and its position `idx` in the level-`d` array.
3. Collect the `d` sibling hashes needed to walk from `N` up to `merkle_root`. These are the `merkle_proof` entries.
4. Call `verify_transaction_inclusion` with:
   - `tx_id` = hash of internal node `N`
   - `tx_block_blockhash` = hash of block `B` (which is in the mainchain)
   - `tx_index` = `idx`
   - `merkle_proof` = the `d` sibling hashes
   - `confirmations` = any value ≤ `gc_threshold`
5. `compute_root_from_merkle_proof(N, idx, proof)` reconstructs `merkle_root` exactly, so the function returns `true` — asserting that a non-existent transaction was included in block `B`. [6](#0-5) [7](#0-6)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
