### Title
`DelegateAction` Expiry Off-by-One: Relayer Can Execute at `max_block_height` Despite Spec Requiring Expiry — (`runtime/runtime/src/actions.rs`)

### Summary

`apply_delegate_action` uses a strict `>` comparison to check whether a `DelegateAction` has expired, but the protocol specification and the field's own documentation state the action should expire when `block_height >= max_block_height`. This off-by-one allows an unprivileged relayer to execute a meta-transaction at exactly `max_block_height` — one block after the sender intended the action to be dead.

### Finding Description

`DelegateAction.max_block_height` is documented as:

> "The maximal height of the block in the blockchain **below which** the given DelegateAction is valid."

The runtime spec (`docs/RuntimeSpec/Actions.md`) states explicitly:

> "If the current block is **equal or greater than** `max_block_height`" → `DelegateActionExpired`

The production expiry gate in `apply_delegate_action` is:

```rust
// runtime/runtime/src/actions.rs, line 435
if apply_state.block_height > delegate_action.max_block_height() {
    result.result = Err(ActionErrorKind::DelegateActionExpired.into());
    return Ok(());
}
```

The condition is `>` (strictly greater), not `>=`. When `block_height == max_block_height` the branch is **not taken** and the action executes successfully. The existing unit test `test_delegate_action` at line 1289–1303 explicitly asserts this:

```rust
let apply_state =
    create_apply_state(signed_delegate_action.delegate_action.max_block_height);
// ...
assert!(result.result.is_ok(), ...);   // passes — action executes at max_block_height
```

The error variant's own doc string in `core/primitives/src/errors.rs` says:

> "Delegate action has expired. `max_block_height` is **less than** actual block height."

That wording (`<`) is consistent with the code (`>`), but contradicts the field description ("below which") and the spec ("equal or greater than"). The spec and field description are the authoritative statement of sender intent; the code and error string share the same off-by-one defect.

### Impact Explanation

A sender signs a `DelegateAction` with `max_block_height = N` to express: *"this authorisation is valid only for blocks strictly below N."* A relayer who receives the signed action can withhold it and submit it at exactly block N. The runtime accepts it, executing the inner actions one block after the sender believed the authorisation had expired.

Concrete harm: a sender uses `max_block_height` to bound a time-sensitive operation (e.g., a token swap, a vote, a key rotation). The relayer can front-run the expiry by submitting at the last valid block according to the code (`N`) rather than the last valid block according to the spec (`N-1`). The sender has no on-chain recourse because the signature is valid and the nonce is consumed.

### Likelihood Explanation

`DelegateAction` (meta-transactions / NEP-366) is a live, permissionless feature on mainnet. Any relayer — an unprivileged network participant — controls submission timing. Choosing to submit at exactly `max_block_height` requires no special privilege, no validator collusion, and no race condition beyond normal block production. The 1-block window is deterministic and reproducible.

### Recommendation

Change the expiry check from strict `>` to `>=` to match the specification:

```diff
-    if apply_state.block_height > delegate_action.max_block_height() {
+    if apply_state.block_height >= delegate_action.max_block_height() {
         result.result = Err(ActionErrorKind::DelegateActionExpired.into());
         return Ok(());
     }
```

Update the unit test `test_delegate_action` to use `max_block_height - 1` as the valid block height, and add a test asserting that `block_height == max_block_height` returns `DelegateActionExpired`. Update the error variant doc string in `core/primitives/src/errors.rs` to read "less than **or equal to**" to match the corrected semantics.
