### Title
Protobuf `ValidResourceBounds` Deserialization Heuristic Silently Downgrades `AllResources` to `L1Gas`, Producing a Divergent Transaction Hash - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary
The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` converter uses a value-based heuristic — checking whether `l2_gas` and `l1_data_gas` are both zero — to reconstruct the `ValidResourceBounds` variant after protobuf deserialization. A valid `AllResources` transaction whose `l2_gas` and `l1_data_gas` bounds happen to be zero is silently reconstructed as `ValidResourceBounds::L1Gas`. Because `get_tip_resource_bounds_hash` includes the `L1_DATA_GAS` packed element only for the `AllResources` variant, the hash computed from the deserialized object is structurally different from the hash computed at submission time, breaking the canonicalization invariant across the protobuf serialization boundary.

### Finding Description

**Root cause — protobuf deserializer heuristic:**

```
crates/apollo_protobuf/src/converters/transaction.rs  lines 431-435
```

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

The protobuf wire format carries no tag distinguishing the two variants; the deserializer infers the variant purely from whether the numeric fields are zero. A transaction originally created as `AllResources` with `l2_gas = 0` and `l1_data_gas = 0` is therefore reconstructed as `L1Gas`.

**Divergent hash computation:**

`get_tip_resource_bounds_hash` branches on the variant:

```
crates/starknet_api/src/transaction_hash.rs  lines 203-208
```

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
``` [2](#0-1) 

- `AllResources` path: Poseidon over `[tip, L1_GAS_packed, L2_GAS_packed(0), L1_DATA_GAS_packed(0)]` — **4 elements**
- `L1Gas` path: Poseidon over `[tip, L1_GAS_packed, L2_GAS_packed(0)]` — **3 elements**

These produce distinct field elements. The hash committed at gateway ingestion time (4-element) differs from the hash recomputed after protobuf round-trip (3-element).

**Submission path always uses `AllResources`:**

`RpcInvokeTransactionV3`, `InternalRpcInvokeTransactionV3`, and `InternalRpcDeclareTransactionV3` all store `resource_bounds: AllResourceBounds` (the concrete struct, not the enum). Their `TransactionHasher` implementations wrap it unconditionally in `ValidResourceBounds::AllResources(...)`: [3](#0-2) 

So the hash computed at submission is always the 4-element variant for any transaction that entered through the RPC/gateway path.

**Serialization round-trip loses the variant:**

When `ValidResourceBounds::AllResources` with zero l2/l1_data_gas is serialized to protobuf, all three `ResourceLimits` fields are emitted with `max_amount = 0` and `max_price_per_unit = 0`: [4](#0-3) 

On deserialization the heuristic fires and the variant is changed to `L1Gas`, silently altering the hash preimage.

### Impact Explanation

Any component that recomputes the transaction hash from a `ValidResourceBounds`-bearing object received over protobuf (state-sync block delivery, consensus block replay, RPC `starknet_getTransactionByHash` hash verification) will produce a hash that does not match the hash committed in the block. This constitutes "transaction conversion or signature/hash logic binds the wrong hash type or executable payload" (High impact). Concretely:

- A state-sync peer that validates `validate_transaction_hash` against the deserialized `InvokeTransactionV3` will compute the wrong hash and reject a valid block.
- An RPC node that re-derives the transaction hash from stored block data will return an authoritative-looking wrong value.

### Likelihood Explanation

The trigger is a user submitting a V3 `AllResources` transaction with `l2_gas = 0` and `l1_data_gas = 0` — a configuration the gateway accepts (the stateless validator only requires at least one non-zero bound across all three resources). No privileged access is required. The condition is reachable on mainnet today for any transaction that only consumes L1 gas.

### Recommendation

Remove the value-based heuristic. The protobuf `ResourceBounds` message should carry an explicit boolean or enum field (e.g., `is_all_resources: bool`) so the deserializer can reconstruct the correct variant without inspecting numeric values. Alternatively, always deserialize to `AllResources` when all three `ResourceLimits` fields are present in the message, reserving `L1Gas` only for the legacy case where `l2_gas` is absent (`None`) at the wire level.

### Proof of Concept

1. Craft a V3 invoke transaction with `AllResourceBounds { l1_gas: {max_amount: 1000, max_price: 1}, l2_gas: {0,0}, l1_data_gas: {0,0} }`.
2. Submit via RPC; gateway computes hash H₄ = Poseidon(INVOKE, v3, sender, Poseidon(tip, L1_GAS_packed, L2_GAS_packed(0), L1_DATA_GAS_packed(0)), …).
3. Transaction is included in block B; H₄ is stored in the block body.
4. A peer receives block B via state-sync protobuf; deserializes the `InvokeV3` message.
5. `TryFrom<protobuf::ResourceBounds>`: `l1_data_gas.is_zero() && l2_gas.is_zero()` → `ValidResourceBounds::L1Gas(l1_gas)`.
6. Peer calls `calculate_transaction_hash` → H₃ = Poseidon(INVOKE, v3, sender, Poseidon(tip, L1_GAS_packed, L2_GAS_packed(0)), …).
7. H₃ ≠ H₄; any hash-equality check (block hash verification, `validate_transaction_hash`) fails for a valid transaction.
