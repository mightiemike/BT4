### Title
Unchecked `transferFrom()` Return Value in `replaceUsdcEWithUsdc` Allows Draining USDC.e from DDAs for Free - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the return value and without using the protocol's own `ERC20Helper.safeTransferFrom()`. The function is `external` with no access control. If the USDC token returns `false` instead of reverting on a failed transfer, any caller can drain USDC.e from any Direct Deposit Account (DDA) without providing any USDC in exchange.

---

### Finding Description

The Nado protocol defines `ERC20Helper` — its own safe-transfer library — and applies it uniformly via `using ERC20Helper for IERC20Base` throughout `ContractOwner`, `EndpointStorage`, `Clearinghouse`, and other contracts. Every other token transfer in the codebase goes through `safeTransfer` or `safeTransferFrom`, both of which check the low-level call's success flag and the decoded boolean return value. [1](#0-0) 

The single exception is line 616 of `ContractOwner.replaceUsdcEWithUsdc()`:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
``` [2](#0-1) 

The return value of this call is silently discarded. The function then unconditionally proceeds to:

1. Call `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))`, which pulls all USDC.e out of the DDA and sends it to `ContractOwner` (msg.sender of the `withdraw` call is `ContractOwner`).
2. Call `IERC20Base(usdcE).safeTransfer(msg.sender, balance)`, which sends that USDC.e to the original caller. [3](#0-2) 

The function carries no `onlyOwner` or any other access modifier — it is callable by any address on chain ID 57073. [4](#0-3) 

---

### Impact Explanation

If the hardcoded USDC token (`0x2D270e6886d130D724215A266106e6832161EAEd`) returns `false` on a failed `transferFrom` (e.g., insufficient allowance, insufficient balance, or a non-standard/upgraded implementation), the attacker pays nothing yet receives the full USDC.e balance of the targeted DDA. The corrupted asset delta is: attacker gains `balance` USDC.e; the DDA loses `balance` USDC.e; no USDC is deposited into the DDA. The DDA owner's collateral is permanently stolen.

---

### Likelihood Explanation

The function is permissionlessly callable by any unprivileged address on Ink (chain ID 57073). The only precondition is that a DDA for the target subaccount exists and holds a non-zero USDC.e balance — both are normal operating conditions during the USDC.e → USDC migration this function was designed to facilitate. The USDC address is a proxy contract; any upgrade that changes revert-on-failure behavior to return-false would immediately make this exploitable. The inconsistency with every other transfer in the codebase (all of which use `ERC20Helper`) indicates this was an oversight rather than an intentional design choice.

---

### Recommendation

Replace the bare `transferFrom` call with the protocol's own safe wrapper, consistent with every other transfer site in the codebase:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`safeTransferFrom` is already available on `IERC20Base` via `using ERC20Helper for IERC20Base` declared at the top of `ContractOwner`. [5](#0-4) 

---

### Proof of Concept

1. A DDA exists for `subaccount` and holds `N` USDC.e (normal post-migration state).
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` with zero USDC allowance granted to `ContractOwner`.
3. `balance = IERC20Base(usdcE).balanceOf(directDepositV1)` → `N`.
4. `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, N)` → returns `false` (no revert) due to zero allowance on a non-reverting USDC implementation.
5. Return value is ignored; execution continues.
6. `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))` → transfers `N` USDC.e from DDA to `ContractOwner`. [6](#0-5) 

7. `IERC20Base(usdcE).safeTransfer(msg.sender, N)` → attacker receives `N` USDC.e.
8. Net result: attacker gains `N` USDC.e; DDA loses `N` USDC.e; no USDC was deposited. [7](#0-6)

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
