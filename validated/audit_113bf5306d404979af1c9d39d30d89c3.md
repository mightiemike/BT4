### Title
Missing Heap-Limit Check in `new_substr` SmallAtom Materialization Path — (`src/allocator.rs`)

### Summary

`Allocator::new_substr` contains a branch for `SmallAtom` nodes that writes new bytes into `u8_vec` without first checking the configured `heap_limit`. Every other allocation path that writes to `u8_vec` performs this check. An attacker-controlled CLVM program can call `substr` on a SmallAtom value whose extracted slice does not fit back into a SmallAtom, triggering the unguarded write repeatedly within the allowed cost budget, bypassing the heap limit entirely.

### Finding Description

`Allocator::new_substr` dispatches on the node type of its input. For `ObjectType::Bytes` nodes, the function creates a new `AtomBuf` that points into already-existing data in `u8_vec` — no new bytes are written, so no heap check is needed. For `ObjectType::SmallAtom` nodes, the function reconstructs the atom's bytes from the inline value and then checks whether the extracted slice fits back into a SmallAtom via `fits_in_small_atom`. If it does, a ghost atom is recorded and no heap bytes are consumed. If it does **not** fit (the `else` branch), the function writes the slice directly into `u8_vec` and pushes a new `AtomBuf` — but performs **no heap-limit check** before doing so.


Compare with every other allocation path that writes to `u8_vec`:

- `new_atom` (line 621): `if start as usize + self.ghost_heap + v.len() > self.heap_limit { return Err(EvalErr::OutOfMemory); }`
- `new_small_number` (line 643): same pattern
- `new_concat` (line 882): same pattern

The `SmallAtom` else branch is the only write path that omits this guard.

**Trigger condition.** A SmallAtom stores a non-negative integer up to `NODE_PTR_IDX_MASK = 0x3FFFFFF`. Its canonical byte encoding follows CLVM signed big-endian rules, so value 128 is encoded as `[0x00, 0x80]` (two bytes, leading zero for positive sign). Taking `substr` from byte offset 1 to 2 extracts `[0x80]`. Because `0x80` has its high bit set it is interpreted as a negative CLVM integer and does not fit in a SmallAtom, so `fits_in_small_atom` returns `None` and the unguarded else branch executes, writing one byte to `u8_vec` past the heap limit.

**Reachable entry path.** `op_substr` in `src/more_ops.rs` (lines 854–884) accepts attacker-controlled CLVM bytes, resolves the first argument as any atom node (including SmallAtom), validates indices, and calls `a.new_substr(a0, start as u32, end as u32)` unconditionally. The cost assigned to `op_substr` is a flat `1` (line 882). A CLVM program of the form `(substr (q . 128) 1 2)` embedded in a loop therefore costs approximately 10–20 units per iteration. With a `max_cost` budget of 11 billion (the Chia mainnet limit), an attacker can execute hundreds of millions of iterations, each writing 1 byte past the heap limit, accumulating hundreds of megabytes to gigabytes of unguarded allocation.

### Impact Explanation

The `heap_limit` in `Allocator` is the primary mechanism preventing a single CLVM program from exhausting node memory. Bypassing it allows a crafted puzzle to grow `u8_vec` without bound within the cost budget, causing out-of-memory termination of the full node process. If different nodes enforce different heap limits (e.g., due to configuration or platform differences), the same program may succeed on one node and fail on another, producing a **consensus split**: one node accepts the spend, another rejects it with `OutOfMemory`, breaking chain agreement.

### Likelihood Explanation

The trigger is straightforward: any CLVM atom whose canonical encoding contains a byte with the high bit set (e.g., value 128 = `[0x00, 0x80]`) is a SmallAtom, and `substr` extracting the high-bit byte hits the unguarded path. No special privileges are required; any coin spend submitted to the mempool can carry this program. The cost of 1 per `substr` call makes repeated exploitation cheap relative to the block cost limit.

### Recommendation

Add the same heap-limit guard used by `new_atom` and `new_small_number` at the start of the SmallAtom else branch in `new_substr`:

```rust
} else {
    let start = self.u8_vec.len();
    // ADD: heap-limit check before writing
    if start + self.ghost_heap + substr.len() > self.heap_limit {
        return Err(EvalErr::OutOfMemory);
    }
    let end = start + substr.len();
    self.u8_vec.extend_from_slice(substr);
    ...
}
```

### Proof of Concept

1. Construct a CLVM program that loops and calls `(substr (q . 128) 1 2)` on each iteration. Value 128 is a SmallAtom with bytes `[0x00, 0x80]`; extracting byte 1 yields `[0x80]`, which does not fit in a SmallAtom.
2. Create an `Allocator` with a small `heap_limit`, e.g. `Allocator::new_limited(10)`.
3. Run the program via `run_program`. Each `substr` call writes 1 byte to `u8_vec` without triggering `OutOfMemory`, even after `u8_vec.len() + ghost_heap` exceeds 10.
4. After N iterations, `u8_vec.len()` is N bytes, far exceeding the configured limit of 10, demonstrating the bypass.
