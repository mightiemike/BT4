### Title
`gas_prices_to_hash` omits `l2_gas_consumed` and `next_l2_gas_price` from block hash preimage for Starknet ≥ 0.14.0, breaking fee-market canonicalization — (`crates/starknet_api/src/block_hash/block_hash_calculator.rs`)

---

### Summary

`gas_prices_to_hash` is the sole function that encodes gas-price data into the block hash. For `BlockHashVersion >= V0_13_4` it hashes only the six static price fields (`l1_gas_price_{wei,fri}`, `l1_data_gas_price_{wei,fri}`, `l2_gas_price_{wei,fri}`). The two EIP-1559 fee-market fields introduced in Starknet 0.14.0 — `l2_gas_consumed` (total L2 gas used in the block) and `next_l2_gas_price` (the price for the next block) — are stored in every block header, transmitted over P2P, and served by the RPC, but are **never chained into the block hash**. The `BlockHashVersion` enum has no variant beyond `V0_13_4`, so all 0.14.x blocks silently reuse the 0.13.4 formula. A developer TODO in the source code acknowledges the gap but it has not been resolved despite the protocol advancing to 0.14.4.

---

### Finding Description

`BlockHeaderWithoutHash` and `StorageBlockHeader` both carry `l2_gas_consumed: GasAmount` and `next_l2_gas_price: GasPrice`. [1](#0-0) [2](#0-1) 

`PartialBlockHashComponents`, the struct fed into `calculate_block_hash`, has no slots for these two fields: [3](#0-2) 

`BlockHashInput::to_final_block_hash_components` explicitly drops them when constructing `PartialBlockHashComponents`: [4](#0-3) 

`gas_prices_to_hash`, the only place gas data enters the hash, hashes the six static price felts but carries a developer TODO acknowledging the omission:

```
// TODO(Ayelet): add l2_gas_consumed, next_l2_gas_price after 0.14.0.
pub fn gas_prices_to_hash(...)
``` [5](#0-4) 

`BlockHashVersion` has only two variants (`V0_13_2`, `V0_13_4`); there is no `V0_14_0` branch: [6](#0-5) 

Consequently every block produced at Starknet version 0.14.0–0.14.4 is hashed with the frozen `V0_13_4` formula. The P2P `SignedBlockHeader` protobuf carries both fields (tags 18 and 19) and even warns that `fee_proposal_fri` (a related field) "is currently not part of the block hash, so the value must be trusted" — but no equivalent warning exists for `l2_gas_consumed` / `next_l2_gas_price`: [7](#0-6) 

The consensus orchestrator reads `next_l2_gas_price` from a synced block and directly assigns it as the sequencer's own L2 gas price for the next block: [8](#0-7) 

Because the block hash does not commit to `next_l2_gas_price`, a peer can supply any value for that field in a `SignedBlockHeader` and the receiving node's hash verification will pass. The node then adopts the attacker-supplied price as its own `l2_gas_price` for the next block. Since `l2_gas_price` **is** included in the next block's hash, the resulting block hash diverges from the canonical chain.

The RPC layer (`starknet_getBlockWithTxHashes`, `starknet_getBlockWithTxs`, fee-estimation endpoints) returns `next_l2_gas_price` as an authoritative header field. Because it is not bound to the block hash, any node can serve an arbitrary value for a given block hash without cryptographic detection.

---

### Impact Explanation

**High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

A node serving a manipulated `next_l2_gas_price` for a known block hash passes all hash-based integrity checks. Clients that call `starknet_estimateFee` or read the block header to price their next transaction receive a wrong base fee. A syncing sequencer node that ingests the manipulated value will set its own `l2_gas_price` to the wrong amount, causing every subsequent block it produces to carry a wrong gas price in its hash, diverging from the canonical chain.

---

### Likelihood Explanation

The trigger is reachable by any peer that participates in P2P block-header sync, or by any RPC node operator. No privileged access is required. The gap has existed since Starknet 0.14.0 was deployed and the TODO has not been resolved through 0.14.4. The `BlockHashVersion` enum has no mechanism to automatically pick up new protocol versions, so the omission silently persists for every future 0.14.x release until the enum and `gas_prices_to_hash` are updated.

---

### Recommendation

1. Add a `V0_14_0` (or later) variant to `BlockHashVersion` and update the `TryFrom<StarknetVersion>` conversion.
2. In `gas_prices_to_hash`, add a branch for `block_hash_version >= V0_14_0` that chains `l2_gas_consumed` and `next_l2_gas_price` into the `STARKNET_GAS_PRICES0` Poseidon hash alongside the existing six price felts.
3. Add `l2_gas_consumed` and `next_l2_gas_price` fields to `PartialBlockHashComponents` and populate them in `PartialBlockHashComponents::new` and `BlockHashInput::to_final_block_hash_components`.
4. Add a protobuf-level warning for `l2_gas_consumed` / `next_l2_gas_price` analogous to the existing warning for `fee_proposal_fri` until the fix is deployed.

// [Poseidon (

//     l2_gas_price_wei, l2_gas_price_fri

// [gas_price_wei, gas_price_fri, data_gas_price_wei, data_gas_price_fri].
// TODO(Ayelet): add l2_gas_consumed, next_l2_gas_price after 0.14.0.
pub fn gas_prices_to_hash(
    l1_gas_price: &GasPricePerToken,
    l1_data_gas_price: &GasPricePerToken,
    l2_gas_price: &GasPricePerToken,
    block_hash_version: &BlockHashVersion,
) -> Vec<Felt> {
    if block_hash_version >= &BlockHashVersion::V0_13_4 {
        vec![
            HashChain::new()
                .chain(&STARKNET_GAS_PRICES0)
                .chain(&l1_gas_price.price_in_wei.0.into())
                .chain(&l1_gas_price.price_in_fri.0.into())
                .chain(&l1_data_gas_price.price_in_wei.0.into())
                .chain(&l1_data_gas_price.price_in_fri.0.into())
                .chain(&l2_gas_price.price_in_wei.0.into())
                .chain(&l2_gas_price.price_in_fri.0.into())
                .get_poseidon_hash(),
        ]
    } else {
        vec![
            l1_gas_price.price_in_wei.0.into(),
            l1_gas_price.price_in_fri.0.into(),
            l1_data_gas_price.price_in_wei.0.into(),
            l1_data_gas_price.price_in_fri.0.into(),
        ]
    }
}
