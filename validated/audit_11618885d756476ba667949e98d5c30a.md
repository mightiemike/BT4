Audit Report

## Title
Router-Mediated Swaps Pass Router Address as `sender` to `SwapAllowlistExtension.beforeSwap`, Enabling Allowlist Bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, so when a user routes through `MetricOmmSimpleRouter`, the router's address—not the end user's address—is what `SwapAllowlistExtension.beforeSwap` checks against the allowlist. A pool admin who allowlists the router address (the natural production configuration to permit router-mediated swaps) inadvertently opens the gate to every unprivileged user, completely defeating the per-user allowlist.

## Finding Description

**Call path:**

1. Unprivileged user calls `MetricOmmSimpleRouter.exactInputSingle(params)`.
2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — the router is now `msg.sender` to the pool.
3. `MetricOmmPool.swap` (line 231) calls `_beforeSwap(msg.sender, ...)`, binding `sender = router_address`.
4. `ExtensionCalling._beforeSwap` (lines 160–176) encodes `sender = router_address` and dispatches to the extension.
5. `SwapAllowlistExtension.beforeSwap` (line 37) evaluates `allowedSwapper[msg.sender][sender]` = `allowedSwapper[pool][router_address]`.

**Root cause:** The pool always uses `msg.sender` as the `sender` argument. There is no mechanism to thread the original end-user address through the router into the extension hook. The extension therefore sees the router contract address, not the human trader.

**Why existing guards fail:** `BaseMetricExtension.onlyPool` (called via `msg.sender` in the extension) correctly restricts who may *call* the extension, but does nothing to validate the *identity argument* passed as `sender`. The allowlist mapping is keyed on that argument, so the guard is structurally sound but operates on the wrong value.

**Two concrete exploit paths:**

- **Bypass (primary):** Pool admin calls `setAllowedToSwap(pool, router, true)` to permit router-mediated swaps. Every unprivileged user now passes the check by routing through the router, because `allowedSwapper[pool][router] == true` for all of them.
- **Broken functionality (secondary):** Pool admin allowlists individual user addresses. Those users cannot swap through the router because the router address fails the check, making the standard swap path unusable for allowlisted users.

## Impact Explanation
The allowlist gate is the sole mechanism for restricting who may trade on a gated pool. When the router is allowlisted (the only way to enable router-mediated swaps), any unprivileged trader can execute swaps on a pool the admin intended to restrict. This constitutes broken core pool functionality and an admin-boundary break: the pool admin's intended per-user access control is entirely ineffective against router-routed calls. Traders who should be blocked receive output tokens from the pool, and the pool receives input from unauthorized counterparties.

## Likelihood Explanation
The router is the standard, publicly documented entry point for swaps. Any pool that uses `SwapAllowlistExtension` and also needs to support router-mediated swaps must allowlist the router, triggering the bypass automatically. No special attacker capability is required beyond calling the public `exactInputSingle` or `exactInput` functions. The condition is repeatable on every swap.

## Recommendation
The extension must be able to recover the original end-user address. Two options:

1. **Pass original caller through extensionData:** The router encodes `msg.sender` into `extensionData` before calling the pool; `SwapAllowlistExtension.beforeSwap` decodes and checks that address instead of `sender`.
2. **Check `recipient` instead of `sender`:** If the pool's design guarantees that `recipient` is always the end user, the extension can gate on `recipient`. This is weaker and depends on router conventions.
3. **Dedicated router allowlist entry:** Add a separate `allowedRouter` mapping; when `sender` is an allowlisted router, decode the real user from `extensionData` and check that address.

The cleanest fix is option 1, with the router always appending its `msg.sender` to `extensionData` and the extension verifying it.

## Proof of Concept

```solidity
// Foundry test sketch
function test_allowlistBypass_viaRouter() public {
    // Setup: pool with SwapAllowlistExtension, router is allowlisted
    extension.setAllowedToSwap(address(pool), address(router), true);
    // attacker is NOT individually allowlisted
    assertFalse(extension.isAllowedToSwap(address(pool), attacker));

    // Attacker calls through the router — passes because router address is checked
    vm.prank(attacker);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        recipient: attacker,
        zeroForOne: true,
        amountIn: 1000,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp,
        tokenIn: token0,
        extensionData: ""
    }));
    // swap succeeds — allowlist bypassed
}
```
