Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the Original User, Enabling Complete Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of `pool.swap`. When users route through `MetricOmmSimpleRouter`, the pool sees the router as `msg.sender`, not the original user. If the pool admin adds the router to the allowlist (the only way to let allowlisted users use the standard periphery), every unpermissioned user can bypass the curated-pool gate by routing through the same public router contract.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap`: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router is the direct caller of `pool.swap`, so `sender = router address`: [3](#0-2) 

The same applies to multi-hop `exactInput`, where every hop calls `pool.swap` directly from the router: [4](#0-3) 

The pool admin has no alternative: if the router is not allowlisted, even their own approved users cannot use the periphery. Once `allowedSwapper[pool][router] = true` is set, the mapping entry grants every caller of the router unconditional access to the curated pool. There is no mechanism inside the extension to recover the original user's address from `extensionData`, and `extensionData` is user-controlled so it cannot be trusted for authentication.

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of addresses is fully bypassed. Any unpermissioned user can trade against the pool's oracle-anchored bins by routing through `MetricOmmSimpleRouter`. This breaks the core pool invariant that only allowlisted actors may execute swaps, and directly exposes LP funds to trades from actors the pool admin explicitly excluded. The loss surface is the full pool depth available to any swap direction. This constitutes a broken core pool functionality causing direct loss of LP assets and an admin-boundary break where an unprivileged path bypasses the pool admin's access control.

## Likelihood Explanation
The bypass requires the router to be on the allowlist. Pool admins who want their allowlisted users to use the standard periphery must add the router — there is no alternative supported path. This is the natural, expected configuration. The trap is therefore encountered in every realistic curated-pool deployment that intends to support router-based trading. The attacker requires no special permissions, only the ability to call the public `MetricOmmSimpleRouter` contract.

## Recommendation
The extension must gate the original economic actor, not the immediate caller. Two sound approaches:

1. **Pass the original user through `extensionData` and verify it on-chain via a trusted router**: The router encodes `msg.sender` into `extensionData` before calling the pool; the extension decodes and checks that address. This requires a trusted encoding convention enforced by the router, with the extension verifying that `sender` is a known trusted router before accepting the decoded address.

2. **Add a `trustedRouter` registry to the extension**: When `sender == trustedRouter`, read the real user from a router-side transient slot (e.g., the payer stored by `_setNextCallbackContext`) and check that address against the allowlist instead.

Either way, the invariant must be: the address checked against `allowedSwapper` is the address that controls the economic decision to swap, not the address of the forwarding contract.

## Proof of Concept
```
Setup:
  pool   = MetricOmmPool with SwapAllowlistExtension
  admin  = pool admin
  alice  = allowlisted user
  eve    = NOT on the allowlist
  router = MetricOmmSimpleRouter (public, shared)

Step 1 – admin allows alice and the router so alice can use the periphery:
  extension.setAllowedToSwap(pool, alice,  true)
  extension.setAllowedToSwap(pool, router, true)  // required for alice to use periphery

Step 2 – eve calls the router directly (no special permission):
  router.exactInputSingle(ExactInputSingleParams({
    pool:          pool,
    recipient:     eve,
    zeroForOne:    true,
    amountIn:      X,
    extensionData: ""
  }))

Execution trace:
  router.exactInputSingle  →  pool.swap(recipient=eve, ...)
    msg.sender inside pool.swap = router
    _beforeSwap(sender=router, ...)
    SwapAllowlistExtension.beforeSwap(sender=router, ...)
      allowedSwapper[pool][router] == true  →  PASSES

Result:
  Eve's swap executes. The allowlist guard is completely bypassed.
  Eve receives token output; the pool's LP funds are exposed to an
  actor the admin explicitly did not permit.
```
