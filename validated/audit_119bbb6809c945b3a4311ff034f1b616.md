Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the real user, making per-user allowlists trivially bypassable via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` as seen by the pool — the router contract address, not the end user. Any pool admin who allowlists the router to enable router-mediated access inadvertently opens the pool to every address on the network. There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users.

## Finding Description
**Step 1 — Pool passes `msg.sender` as `sender` to extensions.**
`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, recipient, ...)` at L230–240. When `MetricOmmSimpleRouter` calls `pool.swap(...)`, the pool's `msg.sender` is the router contract address. [1](#0-0) 

**Step 2 — `SwapAllowlistExtension` checks that `sender` value.**
`beforeSwap(address sender, ...)` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. For any router-mediated swap, `sender` is the router address. [2](#0-1) 

**Step 3 — The router always calls `pool.swap` as itself.**
`exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` all call `IMetricOmmPoolActions(params.pool).swap(...)` from the router's own execution context. The pool always sees `msg.sender = router`. [3](#0-2) 

**Step 4 — Contrast with `DepositAllowlistExtension`, which correctly uses `owner`.**
`beforeAddLiquidity(address, address owner, ...)` ignores the `sender` (the liquidity-adder contract) and checks `allowedDepositor[msg.sender][owner]` — the economically relevant actor explicitly threaded through `addLiquidity`. No equivalent "real user" parameter exists on the swap path. [4](#0-3) 

The irreconcilable conflict: if the router is not allowlisted, allowlisted EOAs cannot use the router at all (broken UX). If the router is allowlisted, every address on the network can bypass the per-user allowlist through the router. No configuration achieves both goals simultaneously.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` for institutional or KYC-restricted trading can be accessed by any arbitrary address simply by routing through `MetricOmmSimpleRouter`. The allowlist guard — the only on-chain mechanism preventing unauthorized swaps — is rendered ineffective for the entire router-mediated surface. Unauthorized traders can execute swaps against the pool's oracle-priced bins, extracting value from LPs who deposited under the assumption that only vetted counterparties could trade. This is a direct admin-boundary break: an unprivileged path (the public router) bypasses a configured access-control guard with fund-impacting consequences for LP principal.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing entry point for swaps. Any pool admin who deploys a restricted pool and also wants users to benefit from the router's slippage protection, multi-hop routing, and deadline enforcement will naturally allowlist the router. The bypass requires no special privileges, no flash loans, and no unusual token behavior — a single call to `exactInputSingle` from any EOA suffices. The condition is a natural and expected operational configuration.

## Recommendation
Mirror the `DepositAllowlistExtension` pattern: gate on the economically relevant actor, not the immediate caller. Two viable approaches:

1. **Add a `swapper` parameter to `pool.swap`** (analogous to `owner` in `addLiquidity`). The router would pass `msg.sender` (the real user) as `swapper`; the pool would forward it to extensions. Extensions check `allowedSwapper[pool][swapper]`.
2. **Standardized `extensionData` forwarding.** Define a convention where the router prepends the real user address to `extensionData`. The extension decodes and checks that address. Approach 1 is cleaner as it does not require trusting the encoding.

Until fixed, pool admins should be warned that `SwapAllowlistExtension` provides no meaningful restriction for router-mediated swaps.

## Proof of Concept
```
1. Deploy a pool with SwapAllowlistExtension configured as beforeSwap hook.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   — intending to let allowlisted users access the router.
3. Pool admin calls setAllowedToSwap(pool, alice, true)
   — intending to restrict individual swappers.
4. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, ..., recipient: bob, ...})
   The router calls pool.swap(bob, ...) with msg.sender = router.
   The pool calls extension.beforeSwap(router, bob, ...).
   Extension checks: allowedSwapper[pool][router] == true → PASSES.
5. Bob successfully swaps in a pool he was never authorized to access.
   Alice's allowlist entry is irrelevant; the router entry is the only one checked.
```
