Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Original Swapper, Enabling Allowlist Bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, so when `MetricOmmSimpleRouter` calls `pool.swap(...)`, the extension receives the router's address rather than the original user's address. A pool admin who allowlists the router to support router-mediated swaps for their allowlisted users inadvertently grants every user access to the restricted pool, because the extension evaluates `allowedSwapper[pool][address(router)]` — which is `true` — instead of `allowedSwapper[pool][original_user]`.

## Finding Description

`MetricOmmPool.swap` calls `_beforeSwap` with `msg.sender` as the first argument:

```solidity
// metric-core/contracts/MetricOmmPool.sol:230-240
_beforeSwap(
    msg.sender,   // ← this becomes `sender` in the extension
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol:160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap` checks that value against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)`:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol:72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

`msg.sender` inside `pool.swap` is `address(router)`. The extension therefore evaluates `allowedSwapper[pool][address(router)]`. If the admin has allowlisted the router (the natural step to support router-mediated swaps for their allowlisted users), every user — including non-allowlisted ones — passes the check. The `extensionData` field is forwarded unchanged but `SwapAllowlistExtension` ignores it entirely, so there is no existing mechanism for the router to convey the original caller's identity to the guard.

## Impact Explanation

Any unprivileged user can bypass a pool's swap allowlist by routing through `MetricOmmSimpleRouter` when the router is allowlisted. This breaks the primary access-control mechanism for restricted pools: pools intended for KYC-verified counterparties, trusted market makers, or any other restricted set are silently opened to unrestricted public access. This matches "Broken core pool functionality causing loss of funds" and "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path." LP value can be drained by unauthorized arbitrageurs in pools designed to exclude toxic flow.

## Likelihood Explanation

Medium. The bypass requires the pool admin to allowlist the router, which is the natural and expected operational step for any pool that wants to support router-mediated swaps for its allowlisted users. The system provides no warning that allowlisting the router opens the gate to all users. A pool admin following the obvious pattern — allowlist specific users, then allowlist the router so those users can use it — will unknowingly enable the bypass for every user.

## Recommendation

1. **Router-forwarded identity**: The router should encode the original `msg.sender` into `extensionData` (e.g., as the first 20 bytes), and `SwapAllowlistExtension` should decode and check that identity when `sender` is a known router address.
2. **Extension redesign**: Add a separate `allowedRouter` mapping; when `sender` is an allowlisted router, decode the original user from `extensionData` and check that address against `allowedSwapper`.
3. **Documentation warning**: At minimum, document that allowlisting the router address opens the allowlist to all users, so pool admins can make an informed choice.

## Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension as a beforeSwap hook.
2. Pool admin calls:
       setAllowedToSwap(pool, alice, true)           // Alice is the intended user
       setAllowedToSwap(pool, address(router), true) // to let Alice use the router
3. Charlie (not allowlisted) calls:
       router.exactInputSingle({pool: pool, recipient: charlie, ...})
4. Router calls pool.swap(charlie, ...) with msg.sender = address(router).
5. Pool calls _beforeSwap(address(router), charlie, ...).
6. Extension checks allowedSwapper[pool][address(router)] == true → passes.
7. Charlie's swap executes against the restricted pool.
   Expected: revert NotAllowedToSwap.
   Actual:   swap succeeds.
```
