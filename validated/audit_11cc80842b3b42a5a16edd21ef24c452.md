### Title
Unvalidated `tx_id` Accepted as Internal Merkle Node Enables SPV Proof Forgery — (`File: contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` accepts a caller-supplied `tx_id` and passes it directly to `compute_root_from_merkle_proof` without validating that it represents a genuine leaf-level transaction hash. An unprivileged NEAR caller can supply an internal Merkle tree node hash as `tx_id`, construct a matching proof path, and cause the function to return `true` for a transaction that was never included in the block. This is the direct analog of the reported vulnerability: untrusted input is forwarded to a security-critical function without a sanitization/validation gate.

### Finding Description

`verify_transaction_inclusion` is a public, unguarded contract method (no role check, only `#[pause]`). It accepts `ProofArgs` from any caller and immediately passes the caller-controlled `tx_id` into `compute_root_from_merkle_proof`: [1](#0-0) 

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

`compute_root_from_merkle_proof` treats `tx_id` as the starting hash and walks up the tree: [2](#0-1) 

There is no check that `tx_id` is a leaf-level transaction hash rather than an internal Merkle tree node. The contract's own documentation acknowledges this:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [3](#0-2) 

The `#[deprecated]` Rust attribute produces only a compile-time warning; it does not restrict runtime invocation. The function remains fully callable by any NEAR account. [4](#0-3) 

`verify_transaction_inclusion_v2` was introduced to mitigate this class of attack by requiring a coinbase proof at index 0, but it delegates back to the deprecated function after the coinbase check: [5](#0-4) 

This means the vulnerable path remains reachable directly.

### Impact Explanation

A recipient contract or bridge that calls `verify_transaction_inclusion` and acts on a `true` result (e.g., releasing funds, minting tokens) can be deceived into processing a fabricated transaction. The attacker does not need to mine a block or control any privileged role — they only need to know the Merkle tree structure of any mainchain block already stored in the contract. The corrupted value is the proof result (`bool`) returned to the caller, which is the sole authorization signal for downstream actions.

### Likelihood Explanation

The 64-byte transaction Merkle proof forgery is a well-documented Bitcoin SPV attack (https://www.bitmex.com/blog/64-Byte-Transactions). All mainchain block headers and their Merkle roots are publicly readable from the contract state. An attacker can reconstruct internal node hashes offline from any Bitcoin block explorer, then call `verify_transaction_inclusion` with a crafted `ProofArgs`. No privileged access, leaked keys, or social engineering is required.

### Recommendation

Remove or hard-disable `verify_transaction_inclusion` at the contract level (e.g., always panic with a migration notice) so that callers are forced to use `verify_transaction_inclusion_v2`. Alternatively, add an explicit validation gate inside `verify_transaction_inclusion` that rejects any `tx_id` whose byte length equals 32 and whose value matches a known internal-node pattern — but the cleanest fix is removal, since `v2` already supersedes it. [6](#0-5) 

### Proof of Concept

1. Query the contract for any mainchain block hash via `get_block_hash_by_height`.
2. Fetch the corresponding Bitcoin block from a public explorer; extract the full transaction list and compute the Merkle tree offline.
3. Select any internal node hash `N` at depth `d` in the tree. Construct a `merkle_proof` of length `d` that walks from `N` up to the Merkle root using the sibling hashes at each level.
4. Call `verify_transaction_inclusion` with:
   - `tx_id` = `N` (internal node hash, not a real transaction)
   - `tx_block_blockhash` = the block hash from step 1
   - `tx_index` = the position consistent with the proof path
   - `merkle_proof` = the sibling hashes from step 3
   - `confirmations` = 1
5. The function returns `true`, falsely asserting that `N` is a confirmed transaction in that block. [1](#0-0) [2](#0-1)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root

            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
