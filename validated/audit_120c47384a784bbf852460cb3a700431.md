### Title
Deprecated `verify_transaction_inclusion` Remains Callable, Enabling Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The `verify_transaction_inclusion` function is marked `#[deprecated]` but remains a live, unrestricted public method on the NEAR contract. It lacks the coinbase-proof guard introduced in `verify_transaction_inclusion_v2`, making it vulnerable to the 64-byte transaction Merkle proof forgery attack. Any unprivileged NEAR caller can supply a crafted `tx_id` (an internal Merkle tree node) with a matching proof path to obtain a `true` return value for a Bitcoin transaction that does not exist.

---

### Finding Description

`verify_transaction_inclusion` at `contract/src/lib.rs:288–323` accepts a `ProofArgs` struct containing `tx_id`, `tx_block_blockhash`, `tx_index`, `merkle_proof`, and `confirmations`. It verifies inclusion by computing:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

There is no check that `tx_id` is a leaf node rather than an internal Merkle tree node. The function's own documentation explicitly acknowledges this gap:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [2](#0-1) 

The 64-byte transaction forgery (https://www.bitmex.com/blog/64-Byte-Transactions) exploits this directly. An internal Merkle node — the SHA256d hash of two child nodes — is itself 32 bytes and is structurally indistinguishable from a transaction hash at the verification layer. An attacker can:

1. Select any mainchain block with a known Merkle root.
2. Identify an internal Merkle node `N` at depth `d` in the tree (all data is public on Bitcoin).
3. Construct a `merkle_proof` of length `d` that leads from `N` to the Merkle root.
4. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_index` matching `N`'s position, and the crafted proof.
5. The function returns `true`, falsely confirming a non-existent transaction.

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` performs no leaf-vs-internal-node distinction — it simply hashes pairs up the tree regardless of depth:

```rust
for proof_hash in merkle_proof {
    if current_position % 2 == 0 {
        current_hash = compute_hash(&current_hash, proof_hash);
    } else {
        current_hash = compute_hash(proof_hash, &current_hash);
    }
    current_position /= 2;
}
``` [3](#0-2) 

`verify_transaction_inclusion_v2` mitigates this by requiring a coinbase proof at index 0 that must independently match the Merkle root, making it computationally infeasible to forge a fake leaf at an arbitrary position:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
``` [4](#0-3) 

However, v1 remains fully callable. In Rust, `#[deprecated]` generates only a compiler warning — it does not restrict runtime access. The function is a public NEAR method and can be invoked by any account. [5](#0-4) 

---

### Impact Explanation

Any downstream NEAR contract or application that calls `verify_transaction_inclusion` (v1) to gate fund releases — e.g., a bridge, atomic swap, or cross-chain payment protocol — can be tricked into releasing funds for a Bitcoin transaction that never occurred. The attacker needs only a valid mainchain block hash and the block's Merkle tree structure, both of which are publicly available from Bitcoin's blockchain. The corrupted proof result is the canonical output consumed by downstream logic; the contract itself has no further safeguard once `true` is returned.

---

### Likelihood Explanation

**Medium-High.** The attack requires no privileged access, no leaked keys, and no social engineering. The attacker needs only to:

1. Know a valid mainchain block hash (publicly available).
2. Compute the Merkle tree of that block (all transaction data is public).
3. Call the public `verify_transaction_inclusion` function with crafted arguments.

The forgery technique is well-documented and has been exploited in production systems. The function remains callable on-chain despite being deprecated.

---

### Recommendation

Remove `verify_transaction_inclusion` entirely from the contract, or add a hard revert guard:

```rust
pub fn verify_transaction_inclusion(&self, ...) -> bool {
    env::panic_str("Removed: use verify_transaction_inclusion_v2");
}
```

Do not rely on `#[deprecated]` to prevent on-chain calls. Deprecation is a compile-time hint only and has no effect on NEAR runtime dispatch. [6](#0-5) 

---

### Proof of Concept

Using Bitcoin mainnet block at height 685452 (whose headers are already present in the contract's test data):

1. Obtain the full block from Bitcoin's public RPC: `getblock <hash> 2`.
2. Compute the Merkle tree over all transaction IDs. Let `N` = SHA256d(txid[0] || txid[1]) — the internal node at depth 1 covering the first two transactions.
3. The proof from `N` to the Merkle root has length `tree_depth - 1` and is fully derivable from public data.
4. Call on NEAR:

```json
{
  "tx_id": "<N as hex>",
  "tx_block_blockhash": "<block hash>",
  "tx_index": 0,
  "merkle_proof": ["<sibling of N>", "..."],
  "confirmations": 1
}
```

5. `verify_transaction_inclusion` returns `true`. No such transaction exists at index 0 in that block — `N` is an internal node, not a real transaction hash.

A downstream bridge contract consuming this `true` result would release funds for a Bitcoin deposit that never happened. [7](#0-6) [8](#0-7)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
