Audit Report

## Title
`executeOracleStopLossDecay` retroactively applies new decay rate over full elapsed period, silently zeroing watermarks and disabling stop-loss protection — (File: `metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`executeOracleStopLossDecay` stores a new `decayPerSecondE8` without resetting `lastDecayTs` in any bin's `BinHighWatermarks`. On the next swap, `_checkAndUpdateWatermarks` computes `dt` from the stale `lastDecayTs` and feeds the new (higher) rate into `_decayed` over the entire elapsed window. With `decayPerSecondE8 = 1e8` (the maximum allowed value) and any `dt >= 1 second`, every watermark is zeroed, causing `_applyWatermark` to never report a breach, and the stop-loss fails open for the next trade — allowing a value-leaking swap that should have been blocked to execute against LP principal.

## Finding Description
`executeOracleStopLossDecay` writes only to the pool-level config:

```solidity
// L143-144
uint32 decay = sched.pendingDecayPerSecondE8;
oracleStopLossConfig[pool_].decayPerSecondE8 = decay;
```

`lastDecayTs` lives inside each bin's `BinHighWatermarks` and is only written by `_checkAndUpdateWatermarks` (L284, on every swap) and by `executeOracleStopLossHighWatermarks` (L174, which correctly resets it to `block.timestamp`). `executeOracleStopLossDecay` performs no equivalent reset.

On the next swap after the rate change, `_checkAndUpdateWatermarks` computes:

```solidity
// L267-270
BinHighWatermarks storage hwmS = highWatermarks[pool_][binIdx];
uint256 dt = block.timestamp - hwmS.lastDecayTs;   // spans entire gap since last swap
(uint256 hwm0, bool breach0) = _applyWatermark(metricT0, _decayed(hwmS.token0, decayRate, dt), floorMultiplier);
```

`_decayed` applies the new rate over the full `dt`:

```solidity
// L321-322
uint256 factor = ratePerSecondE8 * dt;
if (factor >= E8) return 0;
```

With `decayPerSecondE8 = 1e8` (permitted by `_validateDecay` at L309-311, which uses a strict `>` check) and `dt >= 1`, `factor = 1e8 * 1 = 1e8 >= E8`, so `_decayed` returns 0 for every watermark.

`_applyWatermark` then receives `hwm = 0`:

```solidity
// L333
if (metric >= hwm) return (metric, false);
```

Since `metric` is `uint256`, `metric >= 0` is always true. `breached = false` is never set, and the stop-loss never fires regardless of how far the pool value has fallen. The value-leaking swap executes, draining LP principal.

The timelock delays when the new rate is stored but does not prevent the retroactive effect. Once stored, any public trader triggering the first post-change swap causes the watermarks to be zeroed and the stop-loss to fail open.

## Impact Explanation
The stop-loss extension's stated guarantee (L27-28) is that value per share at oracle marks cannot fall faster than drawdown + decay × t. Retroactive zeroing of watermarks breaks this guarantee entirely. A swap that should have been blocked — because the pool's per-share value has fallen below the drawdown floor relative to the established watermark — is allowed to proceed. This is a direct loss of LP principal through an unguarded swap, meeting the "direct loss of user principal" impact threshold.

## Likelihood Explanation
The pool admin must propose and execute the decay change through the timelock. However: (1) the timelock itself does not prevent the retroactive effect — it only delays when the new rate is stored; (2) the timelock period can be reduced via `executeOracleStopLossTimelock` (also timelocked, but the initial timelock can be set to zero at pool creation); (3) the maximum allowed decay rate (`1e8`) is reachable through the normal admin flow with no additional cap; (4) any public trader can trigger the first post-change swap, causing the watermarks to be zeroed — the pool admin does not need to execute the swap themselves. The pool admin is semi-trusted, but the retroactive application is an unintended fund-impacting effect that bypasses the stop-loss protection the system is designed to provide.

## Recommendation
In `executeOracleStopLossDecay`, before writing the new rate, apply the old decay rate to every active bin's watermark and reset `lastDecayTs` to `block.timestamp`. This mirrors the pattern already used in `executeOracleStopLossHighWatermarks` (L173-174), which correctly resets `lastDecayTs` when watermarks are manually set. Concretely: iterate over all bins in the pool's configured range, read each `BinHighWatermarks`, apply `_decayed(hwm, oldRate, block.timestamp - lastDecayTs)`, write the decayed value back, and set `lastDecayTs = block.timestamp`. Then store the new `decayPerSecondE8`. This ensures the new rate is only applied to time elapsed after the change, not retroactively.
