### Title
Host-function gas-key fee calculation uses borsh wire length instead of trie-id length for ML-DSA-65 keys, breaking fee-domain parity invariant - (File: `runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

The three gas-key host functions (`promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`, `promise_batch_action_transfer_to_gas_key`) pass the raw borsh wire length of the public key (`public_key_len as usize`) to the fee-calculation helpers. For ML-DSA-65 keys the wire length is **1953 bytes** but the on-trie identifier is only **33 bytes** (`trie_id_len()`). The transaction path correctly uses `public_key.trie_id_len()`. The result is a fee-domain invariant violation: the same logical operation costs ~1920 × `gas_key_byte` × `num_nonces` more gas when invoked via host function than via a signed transaction.

---

### Finding Description

`PublicKey` has two distinct length concepts:

- `len()` — borsh wire length: 33 / 65 / **1953** bytes for ED25519 / SECP256K1 / ML-DSA-65.
- `trie_id_len()` — on-trie identifier length: 33 / 65 / **33** bytes. For ML-DSA-65 the trie stores a SHA3-256 hash, not the full pubkey. [1](#0-0) 

The fee helpers `gas_key_add_key_exec_fee` and `gas_key_transfer_exec_fee` accept a `public_key_len: usize` argument that is supposed to represent the on-trie key length, because they compute the trie-key byte cost: [2](#0-1) 

The **transaction path** in `permission_exec_fees` and `total_send_fees` correctly passes `public_key.trie_id_len()`: [3](#0-2) [4](#0-3) 

The **host-function path** in `promise_batch_action_add_gas_key_with_full_access` passes `public_key_len as usize` — the raw borsh wire length supplied by the calling contract: [5](#0-4) 

Identically in `promise_batch_action_add_gas_key_with_function_call`: [6](#0-5) 

And in `promise_batch_action_transfer_to_gas_key` for both send and exec fees: [7](#0-6) 

The wasmtime runner duplicates the same defect at the corresponding lines: [8](#0-7) [9](#0-8) [10](#0-9) 

For ED25519 and SECP256K1 keys `len() == trie_id_len()`, so the divergence is invisible. For ML-DSA-65 the divergence is **1920 bytes** per key reference.

The existing fee-parity test only exercises ED25519 keys and therefore does not catch this: [11](#0-10) 

---

### Impact Explanation

**Exact divergent value** — for an ML-DSA-65 gas key with `num_nonces = N` and receiver account-id of length `A`:

```
correct exec fee  = gas_key_byte.exec × (access_key_key_len(A, 33)   + sizeof(NonceIndex) + NONCE_VALUE_LEN) × N
actual  exec fee  = gas_key_byte.exec × (access_key_key_len(A, 1953) + sizeof(NonceIndex) + NONCE_VALUE_LEN) × N
overcharge        = gas_key_byte.exec × 1920 × N
```

For `TransferToGasKey` / `WithdrawFromGasKey` via host function, both the send and exec per-byte components are overcharged by `gas_key_byte × 1920`.

Consequences:

1. **Gas exhaustion / contract failure** — a contract that adds or funds an ML-DSA-65 gas key via host function is charged orders-of-magnitude more gas than the equivalent signed transaction. With `num_nonces = 4` the overcharge alone can consume a substantial fraction of the 300 Tgas per-transaction budget, causing the receipt to run out of gas and revert even though the same operation succeeds via a direct transaction.

2. **Broken fee-parity protocol invariant** — the protocol guarantees that adding a gas key via transaction and via host function costs the same (`test_gas_key_fee_parity`). This guarantee is silently violated for ML-DSA-65 keys once `PostQuantumSignatures` activates.

3. **Deterministic divergence** — every node computes the same wrong fee, so consensus is not broken, but the fee charged is provably incorrect relative to the trie work actually performed.

---

### Likelihood Explanation

The `PostQuantumSignatures` protocol feature must be active for ML-DSA-65 `AddKey` actions to pass `validate_add_key_action`. Once it activates, any unprivileged user can deploy or call a contract that invokes `promise_batch_action_add_gas_key_with_full_access` with a 1953-byte ML-DSA-65 borsh-encoded public key. The contract controls `public_key_len` directly (it is the length of the bytes it writes to WASM memory), so the overcharge is deterministically triggered by any ML-DSA-65 gas-key host-function call. No privileged role is required.

---

### Recommendation

After `get_public_key` succeeds, decode the key and use `trie_id_len()` for all fee calculations, mirroring the transaction path:

```rust
// In promise_batch_action_add_gas_key_with_full_access:
let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
let decoded_key = public_key.decode()?;          // decode once
let trie_key_len = decoded_key.trie_id_len();    // use trie length for fees
// ...
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    trie_key_len,            // ← was: public_key_len as usize
    num_nonces,
);
self.ext.append_action_add_gas_key_with_full_access(receipt_idx, decoded_key, num_nonces);
```

Apply the same fix to `promise_batch_action_add_gas_key_with_function_call` and `promise_batch_action_transfer_to_gas_key` in both `logic.rs` and `wasmtime_runner/logic.rs`. Extend `test_gas_key_fee_parity` to cover `KeyType::MLDSA65`.

---

### Proof of Concept

1. Activate `PostQuantumSignatures` on a test network.
2. Generate an ML-DSA-65 key pair; borsh-encode the public key (1953 bytes).
3. Deploy a contract that calls `promise_batch_action_add_gas_key_with_full_access` with the 1953-byte key and `num_nonces = 4`.
4. Submit the same `AddKey(GasKeyFullAccess)` action as a signed transaction.
5. Compare `gas_burnt` on the action-execution receipt:
   - Transaction path: fee computed with `trie_id_len() = 33`.
   - Host-function path: fee computed with `public_key_len = 1953`.
   - Difference: `gas_key_byte.exec × 1920 × 4` — the host-function receipt burns ~7680 × `gas_key_byte` more gas than the transaction receipt, violating the parity invariant asserted by `test_gas_key_fee_parity`. [12](#0-11) [13](#0-12) [14](#0-13) [5](#0-4)

pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =

        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }

fn test_gas_key_fee_parity(mode: GasKeyKind) {
    let mut setup = setup_host_function_test();
    let account = setup.account.clone();

    let num_nonces: NonceIndex = 4;
    let gas_key_a_signer: Signer =
        InMemorySigner::from_seed(account.clone(), KeyType::ED25519, "gas_key_tx").into();
    let gas_key_b_signer: Signer =
        InMemorySigner::from_seed(account.clone(), KeyType::ED25519, "gas_key_host_fn").into();

    let public_key_b_base64 = near_primitives_core::serialize::to_base64(
        &borsh::to_vec(&gas_key_b_signer.public_key()).unwrap(),
    );

    // Add gas key A via transaction, B via host function
    let add_a_outcome = setup.run_actions(vec![Action::AddKey(Box::new(AddKeyAction {
        public_key: gas_key_a_signer.public_key(),
        access_key: mode.access_key(num_nonces, &account),
    }))]);
    let add_b_outcome = setup.run_call_promise(serde_json::json!([
        {"batch_create": {"account_id": account.as_str()}, "id": 0},
        mode.add_action_json(num_nonces, &account, &public_key_b_base64),
    ]));
    assert_eq!(add_a_outcome.gas_burnt, add_b_outcome.gas_burnt);
    assert_eq!(add_a_outcome.tokens_burnt, add_b_outcome.tokens_burnt);
