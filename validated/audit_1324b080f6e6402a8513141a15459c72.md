### Title
Unchecked `transferFrom` Return Value Enables Draining `usdcE` from DirectDepositV1 Without Providing USDC - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(...)` without checking its return value and without using the protocol's own `safeTransferFrom` wrapper. If the USDC token returns `false` instead of reverting on failure, execution continues, and the caller receives `usdcE` from the target DDA without having transferred any USDC.

---

### Finding Description

In `replaceUsdcEWithUsdc`, the function is intended to atomically swap `usdcE` held in a `DirectDepositV1` (DDA) contract for real USDC provided by the caller. The swap logic is:

1. Pull USDC from `msg.sender` into the DDA via `transferFrom` (line 616)
2. Withdraw `usdcE` from the DDA to `ContractOwner` (line 617)
3. Send `usdcE` to `msg.sender` via `safeTransfer` (line 618)

The critical flaw is at step 1:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // return value ignored
```

The return value is silently discarded. If the USDC token at the hardcoded address (`0x2D270e6886d130D724215A266106e6832161EAEd` on chain 57073) returns `false` on failure rather than reverting, steps 2 and 3 still execute unconditionally.

The protocol already has a correct `safeTransferFrom` in `ERC20Helper`, and `ContractOwner` already declares `using ERC20Helper for IERC20Base` at line 24. The same function even uses `safeTransfer` correctly on line 618 for the outbound leg. The inbound leg is the only unchecked call.

The function carries no access control modifier — it is callable by any unprivileged address. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

If the USDC token returns `false` on a failed `transferFrom` (e.g., caller has no allowance or insufficient balance), the caller receives the full `usdcE` balance of the targeted DDA without providing any USDC. This is a direct theft of `usdcE` collateral from the DDA, corrupting the asset balance of the subaccount that owns that DDA.

The corrupted state delta: `usdcE` balance of `directDepositV1[subaccount]` goes to zero; `msg.sender` receives it for free; no USDC enters the DDA.

---

### Likelihood Explanation

The function is permissionless — any address can call it for any subaccount that has a deployed DDA with a nonzero `usdcE` balance. The only precondition is that the USDC token at the hardcoded address returns `false` on failure rather than reverting. Many ERC-20 tokens (including some USDC variants and bridged stablecoins) follow this pattern. The hardcoded address is a bridged/wrapped USDC on Ink (chain 57073), whose exact revert-vs-return-false behavior determines exploitability. Given the permissionless entry point and the direct asset impact, likelihood is medium-to-high if the token does not revert.

---

### Recommendation

Replace the bare `transferFrom` call with the protocol's existing `safeTransferFrom` wrapper:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`safeTransferFrom` is already available on `IERC20Base` via `using ERC20Helper for IERC20Base` declared at line 24 of `ContractOwner.sol`. [4](#0-3) [5](#0-4) 

---

### Proof of Concept

1. A DDA exists for `subaccount` with `usdcE` balance > 0.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` with zero USDC allowance granted to `ContractOwner`.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, balance)` returns `false` (no revert) due to missing allowance.
4. Return value is not checked; execution continues.
5. `DirectDepositV1(directDepositV1).withdraw(usdcE)` transfers the full `usdcE` balance to `ContractOwner`.
6. `IERC20Base(usdcE).safeTransfer(attacker, balance)` sends `usdcE` to the attacker.
7. Attacker receives `usdcE` for free; the DDA's `usdcE` balance is zero; no USDC was deposited. [6](#0-5)

```text
    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
