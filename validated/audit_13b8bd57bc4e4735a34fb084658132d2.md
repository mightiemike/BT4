### Title
Unbounded Loop Over All Spot Products in `claimBuilderFee()` Can OOG, Permanently Locking Builder Fees — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.claimBuilderFee()` iterates over the entire `spotEngine.getProductIds()` array in a single transaction, performing multiple storage reads and writes per product. As the protocol registers more spot products over time, this loop grows without bound. When the product count is large enough, the function will OOG, permanently preventing any builder from claiming their accumulated fees.

---

### Finding Description

`claimBuilderFee()` is callable by any registered builder owner via a `ClaimBuilderFee` slow-mode transaction submitted to `Endpoint`. The function fetches the full spot product list and iterates over every entry:

```solidity
uint32[] memory productIds = spotEngine.getProductIds();
for (uint32 i = 0; i < productIds.length; i++) {
    uint32 productId = productIds[i];
    int128 collectedFee = collectedBuilderFee[productId][builderId];
    if (collectedFee == 0) {
        continue;
    }
    emit ClaimBuilderFee(builderId, productId, sender, collectedFee);
    spotEngine.updateBalance(productId, sender, collectedFee);
    collectedBuilderFee[productId][builderId] = 0;
}
``` [1](#0-0) 

Each non-zero iteration executes `spotEngine.updateBalance()`, which reads and writes multiple storage slots (`balances`, `nonZeroBalances` bitmap, state), plus an SSTORE to zero out `collectedBuilderFee[productId][builderId]`. There is no pagination, no per-call product limit, and no batching mechanism.

The `productIds` array is appended to each time a new spot product is registered via `_addOrUpdateProduct()`: [2](#0-1) 

The entry path for an unprivileged builder is:

1. Builder calls `Endpoint.submitSlowModeTransaction(ClaimBuilderFee)` — `ClaimBuilderFee` is **not** in the owner-only slow-mode gate, so it passes through the standard slow-mode fee path. [3](#0-2) 

2. After the delay, the sequencer processes it via `processSlowModeTransactionImpl()`, which calls `claimBuilderFee(txn.sender, txn.builderId)`. [4](#0-3) 

3. `claimBuilderFee` verifies `builders[builderId].owner == address(uint160(bytes20(sender)))` and then enters the unbounded loop. [5](#0-4) 

---

### Impact Explanation

**Impact: High.** If the loop OOGs, the slow-mode transaction fails and cannot be retried successfully (the product count does not decrease). The builder's accumulated fees across all products remain permanently locked in the contract with no alternative claim path. This is a direct, irreversible loss of earned funds for the builder, not merely a denial-of-service.

---

### Likelihood Explanation

**Likelihood: Low.** The trigger requires the protocol to have registered a large number of spot products. This is a realistic long-term scenario for a growing DEX but requires significant protocol growth before the gas cost exceeds the block limit. The condition is not user-controllable in the short term.

---

### Recommendation

Introduce pagination into `claimBuilderFee()` by accepting `startAt` and `limit` parameters, mirroring the existing `getCustomFeeAddresses()` pattern already present in the same contract: [6](#0-5) 

Alternatively, allow builders to specify an explicit list of `productId`s to claim for in a single call, bounding the loop to caller-supplied input.

---

### Proof of Concept

1. Protocol registers `N` spot products over time (normal protocol growth).
2. Builder accumulates non-zero `collectedBuilderFee` entries across many of those products.
3. Builder submits a `ClaimBuilderFee` slow-mode transaction.
4. Sequencer processes it; `claimBuilderFee()` enters the loop over all `N` product IDs.
5. Each non-zero entry triggers `spotEngine.updateBalance()` (multiple SLOADs + SSTOREs) and a zeroing SSTORE.
6. At sufficient `N`, the cumulative gas exceeds the block gas limit; the transaction reverts with OOG.
7. The builder's fees remain locked in `collectedBuilderFee` with no alternative withdrawal path.
