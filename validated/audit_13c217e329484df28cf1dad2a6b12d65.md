### Title
Stale `digestToSubaccount` Mapping After Isolated Subaccount Closure Enables Margin-Free Position Reopening — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`_tryCloseIsolatedSubaccount` clears the parent/slot/mask state for a closed isolated subaccount but never clears `digestToSubaccount[digest]`. After closure, the original order digest still maps to the now-closed isolated subaccount. When `createIsolatedSubaccount` is called again with the same order, it hits the early-return guard and returns the stale closed subaccount without performing margin transfer, allowing a position to be opened in an isolated subaccount that holds zero margin.

---

### Finding Description

`_tryCloseIsolatedSubaccount` performs the following cleanup when `balance.amount == 0`:

```solidity
isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);
isolatedSubaccounts[parent][id] = bytes32(0);
parentSubaccounts[subaccount] = bytes32(0);
``` [1](#0-0) 

It does **not** clear `digestToSubaccount[digest]` or `digestToMargin[digest]`. [2](#0-1) 

`createIsolatedSubaccount` contains an early-return guard that is the sole idempotency check for re-submissions of the same order:

```solidity
if (digestToSubaccount[digest] != bytes32(0)) {
    return digestToSubaccount[digest];
}
``` [3](#0-2) 

This early return bypasses both the signature check and the margin transfer block that follows:

```solidity
int128 margin = int128(_isolatedMargin(txn.order.appendix));
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
}
``` [4](#0-3) 

Because `CreateIsolatedSubaccount` transaction processing in `EndpointTx` does not call `validateSignedTx` (no transaction-level nonce check), the sequencer can re-submit the same `CreateIsolatedSubaccount` payload after the isolated subaccount has been closed: [5](#0-4) 

Additionally, after closure `parentSubaccounts[subaccount]` is zeroed, so `getLinkedSigner` for the closed isolated subaccount resolves to `linkedSigners[bytes32(0)]` — a desynchronized signer state that persists because `subaccountIds[subaccount]` in `EndpointStorage` is never cleared: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

After an isolated subaccount IS1 is closed (e.g., via liquidation), `digestToSubaccount[D1]` still resolves to IS1. The freed slot can be reused: `createIsolatedSubaccount` for a new order O2 may assign the same `id` and produce the identical `bytes32` value as IS1 (same address, same productId, same slot id). At that point `digestToSubaccount[D1]` silently aliases the new subaccount IS2. More directly, if the sequencer re-submits `CreateIsolatedSubaccount` for the original order O1 (e.g., because the order was only partially filled before liquidation), the early return hands back IS1/IS2 with zero margin transferred. Any subsequent order fill against that subaccount creates a position backed by no collateral, corrupting the protocol's risk accounting and generating bad debt absorbed by the insurance fund.

---

### Likelihood Explanation

The scenario is reachable through normal protocol operation: a user places an isolated order that is partially filled, the isolated subaccount is liquidated (zeroing the position and returning margin to the parent), and the remaining unfilled portion of the order is still live. The sequencer, processing the remaining fill, calls `createIsolatedSubaccount` with the same order digest and receives the stale closed subaccount without margin. No privileged access, governance action, or key compromise is required.

---

### Recommendation

In `_tryCloseIsolatedSubaccount`, after clearing the parent/slot/mask state, also clear the digest mappings. The digest can be recovered from the subaccount's stored order context, or the mapping can be inverted (subaccount → digest) to enable cleanup:

```solidity
// pseudocode addition inside _tryCloseIsolatedSubaccount
bytes32 digest = subaccountToDigest[subaccount];
if (digest != bytes32(0)) {
    digestToSubaccount[digest] = bytes32(0);
    digestToMargin[digest] = 0;
    subaccountToDigest[subaccount] = bytes32(0);
}
```

Alternatively, store the digest at `createIsolatedSubaccount` time in a reverse mapping `subaccount → digest` so that `_tryCloseIsolatedSubaccount` can look it up and clear both entries atomically.

---

### Proof of Concept

1. User signs isolated order O1 (digest D1) for product P, amount 10, margin M.
2. Sequencer submits `CreateIsolatedSubaccount(O1)` → IS1 created, M transferred to IS1, `digestToSubaccount[D1] = IS1`.
3. 5 units of O1 are filled; IS1 holds a 5-unit position.
4. IS1 is liquidated: position zeroed, M returned to parent. `_tryCloseIsolatedSubaccount` clears `parentSubaccounts[IS1]`, mask bit, and slot — but **not** `digestToSubaccount[D1]`.
5. Sequencer re-submits `CreateIsolatedSubaccount(O1)` for the remaining 5 units.
6. `createIsolatedSubaccount` hits `digestToSubaccount[D1] != bytes32(0)` → returns IS1 immediately; no margin is transferred.
7. Sequencer matches the remaining 5 units against IS1 → IS1 now holds a 5-unit position with 0 margin.
8. IS1 is immediately undercollateralized; any adverse price move creates bad debt charged to the insurance fund. [8](#0-7) [3](#0-2) [4](#0-3)
