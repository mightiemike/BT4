### Title
`WithdrawCollateralV2` Missing from Slow-Mode Dispatch Table Permanently Locks User Collateral - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

`WithdrawCollateralV2` is a fully defined transaction type in the `IEndpoint.TransactionType` enum and is handled in the sequencer path (`processTransactionImpl`), but it is absent from `processSlowModeTransactionImpl`. Because `submitSlowModeTransactionImpl` accepts and queues it (charging the slow-mode fee), while `processSlowModeTransactionImpl` silently reverts on it, any user who submits a `WithdrawCollateralV2` via slow mode loses their fee and their withdrawal is permanently dropped — with no on-chain recourse.

---

### Finding Description

`IEndpoint.TransactionType` defines `WithdrawCollateralV2` as enum value 32. [1](#0-0) 

The struct is fully defined with a `sendTo` field enabling withdrawal to a custom recipient address. [2](#0-1) 

`processTransactionImpl` correctly handles `WithdrawCollateralV2` in the sequencer path. [3](#0-2) 

`submitSlowModeTransactionImpl` does **not** include `WithdrawCollateralV2` in the `DepositCollateral`-revert branch or the owner-only branch, so it falls into the `else` branch: the slow-mode fee is charged and the transaction is queued. [4](#0-3) 

`processSlowModeTransactionImpl` has no `WithdrawCollateralV2` case and terminates with `else { revert(); }`. [5](#0-4) 

When `_executeSlowModeTransaction` runs, it wraps `processSlowModeTransaction` in a try/catch. The revert is caught, the gas-OOG heuristic passes (it is a clean revert, not OOG), and execution continues silently. The comment `// try return funds now removed` confirms that the former fund-return-on-failure path was deleted. [6](#0-5) 

The transaction is consumed from the queue (`delete slowModeTxs[_slowModeConfig.txUpTo++]`) with no state change to the user's collateral balance. [7](#0-6) 

---

### Impact Explanation

Two concrete asset deltas occur:

1. **Slow-mode fee loss**: The user's quote-token balance is debited by `SLOW_MODE_FEE` at submission time. This fee is never refunded because the catch block no longer returns funds.
2. **Withdrawal permanently blocked via slow mode**: The user's collateral remains locked in the protocol. Because the queued transaction is silently dropped, the user has no on-chain path to execute a `WithdrawCollateralV2` withdrawal without the sequencer. The `sendTo` field — the distinguishing feature of V2 — is entirely inaccessible in slow mode.

The slow-mode path is the protocol's safety mechanism of last resort for sequencer downtime. Blocking it for `WithdrawCollateralV2` means any user who specifically needs to withdraw to a custom recipient address (`sendTo != address(0)`) during sequencer unavailability cannot do so and loses their fee in the attempt.

---

### Likelihood Explanation

The entry point is `Endpoint.submitSlowModeTransaction`, a public function callable by any user. `WithdrawCollateralV2` is a documented, first-class transaction type with a full struct, a Verifier digest path, and sequencer-path handling. Any user who submits it via slow mode — whether intentionally or by following protocol documentation — triggers the bug. No privileged access, governance action, or external dependency is required.

---

### Recommendation

Add a `WithdrawCollateralV2` branch to `processSlowModeTransactionImpl` in `core/contracts/EndpointTx.sol`, mirroring the logic already present in `processTransactionImpl` (lines 437–465). The slow-mode variant should decode `SignedWithdrawCollateralV2`, validate the sender, and call `clearinghouse.withdrawCollateral` with the `sendTo` field. Additionally, consider whether `WithdrawCollateralV2` should be explicitly reverted in `submitSlowModeTransactionImpl` (like `DepositCollateral`) until the processing branch is added, to prevent fee loss during the interim.

---

### Proof of Concept

1. Sequencer goes offline (or user simply prefers slow mode).
2. User calls `Endpoint.submitSlowModeTransaction(txBytes)` where `txBytes[0] == uint8(IEndpoint.TransactionType.WithdrawCollateralV2)` (value `32`).
3. `submitSlowModeTransactionImpl` evaluates: not `DepositCollateral`, not `DepositInsurance`, not in the owner-only list → falls to `else` branch.
   - `chargeSlowModeFee(_getQuote(), sender)` debits the user's quote balance.
   - `slowModeFees += SLOW_MODE_FEE`.
   - Transaction is pushed to `slowModeTxs`.
4. After `SLOW_MODE_TX_DELAY` (3 days), user or anyone calls `Endpoint.executeSlowModeTransaction()`.
5. `_executeSlowModeTransaction` calls `this.processSlowModeTransaction(txn.sender, txn.tx)` inside a try/catch.
6. `processSlowModeTransactionImpl` reaches `else { revert(); }` — no `WithdrawCollateralV2` branch exists.
7. The catch block fires; gas check passes (clean revert); execution continues silently.
8. The transaction slot is deleted from `slowModeTxs`. The user's collateral is unchanged. The fee is gone.
