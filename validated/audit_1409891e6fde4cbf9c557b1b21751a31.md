Audit Report

## Title
Router-Mediated Swap Passes Router Address as `sender` to `SwapAllowlistExtension`, Enabling Allowlist Bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it to `SwapAllowlistExtension.beforeSwap`. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the originating user. The allowlist therefore gates the router address rather than the actual swapper, creating a wrong-actor binding that allows any unprivileged user to bypass a curated pool's swap allowlist by routing through the public router.

## Finding Description

**Call path:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle(params)` — `msg.sender = user`.
2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — pool's `msg.sender = router`.
3. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` — passes `router` as `sender`.
4. `ExtensionCalling._beforeSwap` encodes and dispatches `IMetricOmmExtensions.beforeSwap(sender=router, ...)` to the configured extension.
5. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` = `allowedSwapper[pool][router]`.

The check never sees the originating user's address. If the pool admin allowlists the router (the only way to permit router-based swaps on a curated pool), every user — including those the allowlist was designed to exclude — can swap freely through the router. Conversely, if individual users are allowlisted but the router is not, those users are incorrectly blocked when using the router.

**Root cause:** `MetricOmmPool.swap` uses `msg.sender` as the `sender` forwarded to extensions. The router is an intermediary, so `msg.sender` at the pool is always the router, not the economic actor.

**Existing guards are insufficient:** `BaseMetricExtension.onlyPool` only verifies the caller is a registered pool; it does not recover the original user. The router stores the original `msg.sender` only in transient storage for its own callback payment logic (`_setNextCallbackContext`), and never passes it to the pool's `swap` call.

## Impact Explanation
A pool admin who deploys a `SwapAllowlistExtension`-gated pool and allowlists the router to support standard periphery usage inadvertently opens the pool to all users. Any unprivileged trader can bypass the allowlist by calling `MetricOmmSimpleRouter.exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput`. This constitutes a direct admin-boundary break and broken core pool functionality (curation policy failure) on allowlisted pools, meeting Sherlock High severity thresholds.

## Likelihood Explanation
The bypass requires no special privileges: any user with token approvals can call the public router. The router is the standard, documented periphery entry point, so the scenario where the router is allowlisted is the expected production configuration for curated pools that still want to support normal UX. The exploit is repeatable on every swap.

## Recommendation
Pass the originating user's address through the swap call rather than relying on `msg.sender` at the pool. One approach: add an explicit `swapper` parameter to `IMetricOmmPoolActions.swap` that the router populates with its own `msg.sender` (the user). Alternatively, the `SwapAllowlistExtension` can check the `recipient` field or require the pool to expose a trusted-forwarder mechanism. The simplest fix consistent with the existing design is for the router to pass the user address in `extensionData` and for the extension to decode it, but this requires a coordinated interface change. The cleanest fix is an explicit `swapper` field in the swap interface that the pool forwards to extensions.

## Proof of Concept

```solidity
// 1. Deploy pool with SwapAllowlistExtension; allowlist only the router.
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// 2. Attacker (not individually allowlisted) calls router directly.
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool),
    recipient: attacker,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    tokenIn: token0,
    extensionData: ""
}));
// Swap succeeds: extension sees sender=router (allowlisted), not attacker.

// 3. Verify: direct pool call from attacker reverts (NotAllowedToSwap),
//    but router call succeeds — demonstrating the bypass.
