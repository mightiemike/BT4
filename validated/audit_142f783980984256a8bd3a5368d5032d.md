Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual User, Enabling Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender` of the `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`. A pool admin who allowlists the router to enable official periphery access inadvertently grants unrestricted swap access to every user who routes through the router, rendering the per-user allowlist a no-op for router-mediated swaps.

## Finding Description
**Root cause:** `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` (lines 230–240), and `ExtensionCalling._beforeSwap` forwards that value as the `sender` argument to every configured extension (lines 160–177). `MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap(params.recipient, ...)` directly, making the router the pool's `msg.sender` (lines 71–86). `SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]` where `sender` is the router address (line 37).

**Exploit flow:**
1. Pool admin deploys a curated pool with `SwapAllowlistExtension` to enforce per-user KYC/institution-only access.
2. Admin calls `setAllowedToSwap(pool, router, true)` to allow allowlisted users to access the pool through the official periphery.
3. Non-allowlisted user Alice calls `MetricOmmSimpleRouter.exactInputSingle({pool, recipient=alice, ...})`.
4. Router calls `pool.swap(alice, ...)` — pool's `msg.sender` is the router.
5. Extension receives `sender = router`, evaluates `allowedSwapper[pool][router] == true`, and permits the swap.
6. Alice receives output tokens despite never being individually allowlisted.

**Why existing checks fail:** There is no secondary check on the originating user. The extension has no access to the real caller's address; `extensionData` is passed through but the extension does not decode it. `DepositAllowlistExtension` correctly checks `owner` (the economic actor passed explicitly by the pool), but `SwapAllowlistExtension` has no equivalent — the swap interface does not pass a separate "economic actor" field distinct from `sender`.

## Impact Explanation
This is an admin-boundary break. A pool admin configures `SwapAllowlistExtension` to enforce per-user access control (e.g., KYC-gated or institution-only pool). By allowlisting the official router — a natural and expected configuration for enabling periphery access — the admin inadvertently opens the pool to every user who routes through `MetricOmmSimpleRouter`. The allowlist becomes a no-op for router-mediated swaps. Non-allowlisted users can trade against the pool, violating the curation policy. Because the pool is oracle-anchored, repeated access by unintended counterparties exposes LPs to adverse-selection risk from actors the pool was specifically designed to exclude.

## Likelihood Explanation
The trigger is a plausible, non-malicious admin configuration: allowlisting the official router so that allowlisted users can access the pool through the standard periphery. The admin has no way to distinguish "allow the router for allowlisted users only" from "allow the router for everyone," because the extension does not expose the originating user's address. Any pool that uses `SwapAllowlistExtension` and also permits router access is affected. No special attacker capability is required — any unprivileged user can call `MetricOmmSimpleRouter` functions.

## Recommendation
The extension must gate the economically relevant actor, not the intermediary. The cleanest fix is to have the router encode the originating `msg.sender` into `extensionData` before forwarding to the pool, and have `SwapAllowlistExtension.beforeSwap` decode and check that address when `sender` is a known router. This requires a coordinated change to the router and the extension. Until implemented, documentation for `SwapAllowlistExtension` must explicitly warn that allowlisting the router grants unrestricted swap access to all users, and pool admins must not allowlist the router on curated pools.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice]  = false   (Alice is NOT allowlisted)
  allowedSwapper[pool][router] = true    (admin allowlists the official router)

Attack:
  Alice calls MetricOmmSimpleRouter.exactInputSingle({pool, recipient=alice, ...})
    → router calls pool.swap(alice, ...)          // router is msg.sender to pool
    → pool calls _beforeSwap(msg.sender=router, ...)
    → extension receives sender=router
    → checks allowedSwapper[pool][router] == true  ✓
    → swap executes; Alice receives output tokens

Direct call (blocked correctly):
  Alice calls pool.swap(alice, ...) directly
    → pool calls _beforeSwap(msg.sender=alice, ...)
    → extension receives sender=alice
    → checks allowedSwapper[pool][alice] == false  → NotAllowedToSwap ✓
```
Reproducible as a Foundry integration test: deploy pool with `SwapAllowlistExtension`, set `allowedSwapper[pool][router] = true`, call `exactInputSingle` from an unallowlisted EOA, assert the swap succeeds and output tokens are received.
