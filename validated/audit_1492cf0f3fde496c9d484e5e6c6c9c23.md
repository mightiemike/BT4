### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The contract exposes two transaction-inclusion verification methods. The v2 method (`verify_transaction_inclusion_v2`) was introduced specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability. However, the v1 method (`verify_transaction_inclusion`) remains a fully public, unrestricted NEAR entry point. Any unprivileged caller can invoke v1 directly, bypassing the coinbase-proof guard entirely and obtaining a `true` result for a forged transaction inclusion claim.

---

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` in Rust, but that attribute is a compiler hint only — it does not restrict on-chain access. The function carries no `#[private]`, no access-control role check, and no `#[trusted_relayer]` guard. [1](#0-0) 

The v2 function adds a coinbase-proof requirement before delegating to v1: [2](#0-1) 

The coinbase proof forces the caller to demonstrate knowledge of the real coinbase transaction at index 0, which defeats the 64-byte internal-node substitution attack. When v1 is called directly, that guard is never evaluated. The only check v1 performs on the proof itself is: [3](#0-2) 

An attacker supplies a `tx_id` that is actually an internal Merkle-tree node (a 64-byte-serialised pair of child hashes), a `merkle_proof` path that reconstructs the block's `merkle_root` from that node, and any `tx_block_blockhash` that is on the current mainchain with sufficient confirmations. Because `compute_root_from_merkle_proof` is purely arithmetic — it does not distinguish leaf nodes from internal nodes — the comparison succeeds and the function returns `true`.

---

### Impact Explanation

Any recipient NEAR contract that calls `verify_transaction_inclusion` (v1) to gate a financial action (e.g., releasing bridged funds, minting tokens, settling a bet) can be deceived into accepting a fabricated transaction. The corrupted value is the boolean proof result returned to the caller. The attacker does not need to submit any invalid Bitcoin header; all required data (a real mainchain block hash, a real Merkle root, and a crafted internal-node path) is publicly available from the Bitcoin blockchain.

---

### Likelihood Explanation

The 64-byte attack is well-documented (referenced in the contract's own deprecation notice and the linked BitMEX post). The v1 entry point is discoverable by inspecting the contract's ABI or source. A motivated attacker targeting a bridge or settlement contract built on top of this light client has a clear, low-cost path: read the target block's Merkle tree, identify a suitable internal node, and call `verify_transaction_inclusion` directly. No privileged key or social engineering is required.

---

### Recommendation

Remove `verify_transaction_inclusion` as a public NEAR method. If backward compatibility must be preserved, add `#[private]` to restrict it to internal calls only, or gate it with an access-control role that no external party holds. All external callers should be directed exclusively to `verify_transaction_inclusion_v2`.

---

### Proof of Concept

1. Identify any mainchain block `B` with `>= confirmations` confirmations whose Merkle tree has depth ≥ 2. Let `root` be its `merkle_root`.
2. Pick any internal node `N` at depth 1 (i.e., `N = SHA256d(left_child || right_child)`). `N` is 32 bytes and is a valid `H256`.
3. Construct `merkle_proof = [sibling_of_N]` and set `tx_index` to the position of `N`'s subtree.
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node, not a real transaction)
   - `tx_block_blockhash = B`
   - `tx_index` and `merkle_proof` as above
   - `confirmations = 1`
5. `compute_root_from_merkle_proof(N, tx_index, [sibling_of_N])` reconstructs `root`, the comparison `== header.block_header.merkle_root` is `true`, and the function returns `true` — falsely asserting that the internal node is an included transaction. [1](#0-0)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
