Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Actual User, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which is the direct `msg.sender` of `pool.swap()` — the router, not the end user. Because the router must be allowlisted for any user to interact with the pool via the periphery, any unprivileged address can bypass the per-user swap gate by routing through the allowlisted `MetricOmmSimpleRouter`. The `DepositAllowlistExtension` correctly checks the beneficiary (`owner`) rather than the router, confirming the asymmetry is a defect, not a design choice.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)`, making the router the `msg.sender` of the pool: [2](#0-1) 

`_beforeSwap` in `ExtensionCalling.sol` forwards `sender` (= router address) and `recipient` (= actual user) to the extension: [3](#0-2) 

`SwapAllowlistExtension.beforeSwap` ignores `recipient` entirely and checks only `sender`: [4](#0-3) 

The check `allowedSwapper[pool][sender]` therefore evaluates `allowedSwapper[pool][router]`. For the router to be usable at all with this pool, the pool admin must allowlist the router. Once the router is allowlisted, the check passes for every user who routes through it, regardless of whether that user is individually allowlisted.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` ignores `sender` and checks `owner` (the LP beneficiary): [5](#0-4) 

The asymmetry is structural: the deposit allowlist gates by the economic beneficiary; the swap allowlist should do the same but gates by the intermediary instead.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a specific set of addresses (e.g., KYC'd counterparties, whitelisted market makers, or protocol-internal actors) provides zero enforcement once the router is allowlisted. Any address can execute swaps against the restricted pool by calling `MetricOmmSimpleRouter`, bypassing the allowlist entirely. This constitutes broken core pool functionality — the access-control invariant the pool admin explicitly configured is violated — and enables unauthorized parties to drain liquidity or execute trades the pool admin intended to block, constituting a direct loss of user/LP principal.

## Likelihood Explanation

Both required conditions — a pool configured with `SwapAllowlistExtension` and the router allowlisted as a swapper — are the expected production configuration for any permissioned pool that still wants to support the standard periphery. The bypass is reachable by any unprivileged user in the normal interaction flow with no special privileges, front-running, or off-chain coordination required.

## Recommendation

Gate by the actual user, not the intermediary. The `beforeSwap` hook receives `recipient` as its second parameter. For a swap allowlist, the meaningful identity to check is the address that initiates and benefits from the swap:

```solidity
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

This mirrors the pattern already used correctly in `DepositAllowlistExtension`. If the intent is to gate both the initiating contract and the beneficiary, require both `sender` and `recipient` to be allowlisted. Document the distinction clearly so pool admins configure the allowlist for the correct address.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` and calls `setAllowedToSwap(pool, router, true)` (required for normal periphery operation) and `setAllowedToSwap(pool, alice, true)`.
2. `bob` (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(pool, ..., recipient = bob)`.
3. The router calls `pool.swap(bob, ...)`. The pool calls `_beforeSwap(sender = router, recipient = bob, ...)`.
4. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router]` → `true` → no revert.
5. Bob's swap executes successfully against the restricted pool.
6. The pool admin's intent — that only `alice` may swap — is violated with no on-chain mechanism to prevent it.
