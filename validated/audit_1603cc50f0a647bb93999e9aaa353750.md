### Title
Gateway Admission Skips `max_l2_gas_amount` Upper-Bound Check for Declare Transactions — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The `StatelessTransactionValidator::validate_resource_bounds` function enforces an upper-bound check (`max_l2_gas_amount`) on the `l2_gas.max_amount` field of incoming transactions, but explicitly skips this check for `Declare` transactions. Any unprivileged user can submit a `Declare` transaction with `l2_gas.max_amount` set to an arbitrarily large value (up to `u64::MAX`) and the gateway will admit it, violating the configured admission-control invariant.

### Finding Description

In `crates/apollo_gateway/src/stateless_transaction_validator.rs`, the `validate_resource_bounds` method reads:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The `else if` branch is never reached for `Declare` transactions. The configured limit `max_l2_gas_amount` (default `1_210_000_000` per `StatelessTransactionValidatorConfig::default()`) is therefore never enforced for Declare transactions. [2](#0-1) 

The same limit is correctly enforced for `Invoke` and `DeployAccount` transactions, and the test suite explicitly confirms that a Declare transaction with `l2_gas.max_amount = 200` passes even when `max_l2_gas_amount = 100`: [3](#0-2) 

### Impact Explanation

A user can submit a `RpcDeclareTransactionV3` with `resource_bounds.l2_gas.max_amount = u64::MAX`. The gateway's `StatelessTransactionValidator` admits it without error. The transaction enters the mempool and is forwarded to the batcher. The batcher's bouncer then evaluates actual resource consumption at block-building time, but the gateway has already violated its own configured admission policy. This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."**

The analog to the external report is direct: just as `_getOraclePairPrice()` assumes `balancerOracleWeight ≤ BALANCER_ORACLE_WEIGHT_PRECISION` but never validates it, `validate_resource_bounds` assumes the `max_l2_gas_amount` invariant holds for all transaction types but never enforces it for `Declare`.

### Likelihood Explanation

Any unprivileged user can craft and submit a `Declare` transaction with an arbitrarily large `l2_gas.max_amount`. No special access, key material, or privileged configuration is required. The code path is reachable through the public RPC endpoint (`starknet_addDeclareTransaction`). The TODO comment in the source confirms this is a known, unresolved gap.

### Recommendation

Apply the same upper-bound check to `Declare` transactions. Remove the early-return branch that skips the check:

```rust
// Remove the special-case skip:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher L2 gas ceiling, introduce a separate `max_l2_gas_amount_declare` config field with its own validated upper bound, mirroring the pattern used for other per-type limits.

### Proof of Concept

1. Construct a valid `RpcDeclareTransactionV3` with `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)` and a valid Sierra contract class.
2. Submit it to the gateway via `starknet_addDeclareTransaction`.
3. Observe that the gateway returns a transaction hash (admission success) rather than `MaxGasAmountTooHigh`.
4. Confirm the transaction appears in the mempool despite `max_l2_gas_amount = 1_210_000_000` in the default config.

The existing test at line 173–201 of `stateless_transaction_validator_test.rs` already documents this behavior as expected, confirming the gap is present in the production code path. [4](#0-3) [5](#0-4)

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        // TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
        if let RpcTransaction::Declare(_) = tx {
        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
            return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())
    }
