### Title
Integer Overflow in `op_unknown` Cost Multiplication Allows Zero-Cost Execution in Consensus Mode - (File: `src/more_ops.rs`)

### Summary

`op_unknown` in `src/more_ops.rs` performs a `check_cost` guard on the base cost **before** multiplying by `cost_multiplier + 1`. In Rust release builds, the multiplication `cost *= cost_multiplier + 1` can silently wrap around to zero (or a tiny value) when the product overflows `u64`. The only post-multiplication guard is `cost > u32::MAX`, which a wrapped-to-zero result trivially passes. The function then returns `Ok(Reduction(0, nil))`, charging zero cost for an operation that should cost billions of units. This is the direct analog of the external report's rounding-down bug: in both cases, an arithmetic operation that should produce a large positive charge instead produces zero, allowing the attacker to escape paying the intended cost.

### Finding Description

In `src/more_ops.rs`, `op_unknown` computes cost in two stages:

```rust
// Stage 1: compute base cost, bounded by max_cost
let mut cost = match cost_function {
    1 => { /* ARITH-like: cost += per_arg + per_byte */ }
    2 => { /* MUL-like:   cost += quadratic terms    */ }
    3 => { /* CONCAT-like: cost += per_arg + per_byte */ }
    _ => 1,
};

// Stage 2: guard (cost <= max_cost here), then multiply
check_cost(cost, max_cost)?;          // line 260 — passes because cost ≤ max_cost
cost *= cost_multiplier + 1;          // line 261 — u64 wraps silently in release builds
if cost > u32::MAX as u64 {           // line 262 — only guard after multiplication
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))  // line 265 — returns wrapped cost
}
```

`cost_multiplier` is extracted from the opcode bytes via `u32_from_u8`, so its maximum value is `u32::MAX = 4,294,967,295`, making `cost_multiplier + 1` at most `2^32`.

For overflow to occur: `cost * (cost_multiplier + 1) > u64::MAX`. With `cost_multiplier = u32::MAX`, this requires `cost > u64::MAX / 2^32 = 2^32 − 1`, i.e., `cost ≥ 2^32 = 4,294,967,296`.

Chia's consensus-mode max cost is approximately 11,000,000,000 (11 billion), which is well above `2^32`. The `check_cost` inside the cost-function loop allows `cost` to grow up to `max_cost` before the outer guard fires, so `cost = 2^32` is reachable.

**Concrete overflow example:**

| Parameter | Value |
|---|---|
| `cost_multiplier` | `u32::MAX = 4,294,967,295` |
| `cost_multiplier + 1` | `2^32 = 4,294,967,296` |
| `cost` (crafted) | `2^32 = 4,294,967,296` |
| `cost * (cost_multiplier + 1)` | `2^64 ≡ 0 (mod 2^64)` |
| Post-wrap check `0 > u32::MAX` | **false** → passes |
| Returned cost | **0** |

The attacker achieves zero-cost execution for an operation whose intended cost is `2^64` (astronomically large).

### Impact Explanation

In consensus mode (`ChiaDialect` without `NO_UNKNOWN_OPS`), unknown opcodes are valid no-ops with a well-defined cost formula. The cost model is the sole mechanism preventing resource exhaustion on full nodes. By returning cost 0, the attacker can include an unknown-opcode call in a puzzle that should be rejected as exceeding the cost limit, but is instead accepted. This breaks the cost-model invariant that every executed operation is charged its full computed cost, and can be used to craft programs that consume unbounded node resources while appearing cheap to the cost checker.

### Likelihood Explanation

The attacker controls all CLVM program bytes submitted in a transaction. Crafting an opcode atom with `cost_multiplier = u32::MAX` (bytes `[0xff, 0xff, 0xff, 0xff, <last_byte_with_cost_function_bits>]`) is trivial. Achieving `cost = 2^32` exactly via cost_function 1 or 3 requires a specific number of argument bytes (approximately 1.43 billion bytes for cost_function=1, or fewer with cost_function=3 using many short arguments), which is large but not impossible given no heap limit in consensus mode. The vulnerability is only reachable in consensus mode (block validation), not mempool mode (which sets `NO_UNKNOWN_OPS`), but consensus-mode exploitation is the higher-severity target.

### Recommendation

Replace the unchecked multiplication with a checked or saturating variant, and move the `check_cost` guard to **after** the multiplication:

```rust
// Use checked_mul to detect overflow
let Some(multiplied) = cost.checked_mul(cost_multiplier + 1) else {
    return Err(EvalErr::Invalid(o))?;
};
cost = multiplied;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    check_cost(cost, max_cost)?;   // guard after multiplication
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

This ensures that any overflow is caught as an error rather than silently wrapping to a small value, and that the final multiplied cost is checked against the budget.

### Proof of Concept

Attacker constructs a CLVM program in consensus mode (no `NO_UNKNOWN_OPS` flag) containing an unknown opcode atom of 5 bytes:

```
opcode bytes: [0xff, 0xff, 0xff, 0xff, 0x40]
              |--- cost_multiplier = u32::MAX ---|  |cost_function=1|
```

The last byte `0x40 = 0b01000000` → `cost_function = (0x40 >> 6) & 0b11 = 1` (arith-like).

Arguments are crafted so that the arith-like cost loop accumulates `cost = 4,294,967,296 = 2^32` (e.g., via `(4,294,967,296 - 99) / 3 ≈ 1,431,655,732` total argument bytes across enough atoms, adjusted to hit the exact multiple of `2^32`).

After the loop, `check_cost(4294967296, max_cost)` passes (since `4294967296 < 11000000000`). Then:

```
cost *= 4294967296   →   4294967296 * 4294967296 = 2^64 ≡ 0 (mod 2^64)
0 > u32::MAX         →   false
return Ok(Reduction(0, nil))
```

The operation is charged **0 cost units** instead of the intended `~1.8 × 10^19`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
