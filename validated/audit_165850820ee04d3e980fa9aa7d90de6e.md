Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, allowing any trader to bypass a curated pool's swap guard via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to every extension hook. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool boundary is the router contract, not the originating user. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]`, so it evaluates the router's allowlist status rather than the real trader's. Any pool admin who allowlists the router as a trusted periphery contract inadvertently grants swap access to every user who routes through it, regardless of individual allowlist status.

## Finding Description

`MetricOmmPool.swap` unconditionally passes `msg.sender` as the first argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← always the direct caller of pool.swap()
  recipient,
  ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that value against the per-pool allowlist, using `msg.sender` (the pool) as the namespace key:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Every public entry point in `MetricOmmSimpleRouter` calls `pool.swap()` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
IMetricOmmPoolActions(params.pool).swap(
  params.recipient, params.zeroForOne, ..., params.extensionData
);
```

At that call boundary, `msg.sender` inside the pool is the **router address**. The extension therefore evaluates `allowedSwapper[pool][router]` and never consults `allowedSwapper[pool][user]`. The router has no mechanism to forward the originating user's identity, so the misbinding is structural.

The allowlist storage and setter confirm the design intent is per-user gating:

```solidity
// SwapAllowlistExtension.sol L12, L17-19
mapping(address pool => mapping(address swapper => bool)) public allowedSwapper;
function setAllowedToSwap(address pool_, address swapper, bool allowed) external onlyPoolAdmin(pool_) {
  allowedSwapper[pool_][swapper] = allowed;
```

A pool admin who allowlists the router as a trusted periphery contract opens the gate for every user who routes through it. Existing guards are insufficient: `allowAllSwappers` is a separate escape hatch, and the `onlyPoolAdmin` setter only controls who can configure the allowlist, not the structural misbinding of `sender`.

## Impact Explanation

**Severity: High.** A non-allowlisted user on a curated pool can execute swaps that the pool admin explicitly intended to block. The pool receives real token input and pays real token output at oracle prices; there is no slippage protection from the allowlist once it is bypassed. LP funds are exposed to traders the pool was designed to exclude, constituting a direct loss of the curation guarantee and potential adverse-selection losses for LPs on pools that rely on the allowlist as their primary access control. This matches the "Allowlist path" audit pivot: swap allowlist checks must cover the exact actor intended and cannot be bypassed through the router.

## Likelihood Explanation

**Likelihood: High.**

- `MetricOmmSimpleRouter` is the canonical public swap entrypoint in the periphery layer, with `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` all calling `pool.swap()` directly.
- A pool admin who deploys a curated pool and wants users to access it via the standard router will naturally allowlist the router address.
- No special privilege or unusual configuration is required; the bypass is reachable by any user who calls the router instead of the pool directly.
- The pool's `swap` function provides no mechanism to forward the originating user's identity, and the router has no way to pass it either.

## Recommendation

The extension must resolve the real user identity rather than trusting the `sender` argument when the caller is a known intermediary. Two sound approaches:

1. **Pass the real user through `extensionData`**: Add a `swapper` field that the router populates with `msg.sender` before calling the pool. The extension decodes it and checks `allowedSwapper[pool][swapper]` only when `sender` is a recognized router, falling back to `allowedSwapper[pool][sender]` for direct callers.

2. **Redesign the extension to ignore `sender` entirely**: Accept an explicit `swapper` field in `extensionData` that all callers (including direct pool callers) must populate. The extension checks only the decoded `swapper`, making the check independent of the call-boundary actor.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, router, true)   // allowlist the router as trusted periphery
  - Pool admin does NOT call setAllowedToSwap(pool, alice, true)

Attack:
  1. Alice (non-allowlisted) calls MetricOmmSimpleRouter.exactInputSingle(...)
  2. Router calls IMetricOmmPoolActions(pool).swap(recipient=alice, ...)
  3. Pool sets sender = msg.sender = router_address
  4. Pool calls extension.beforeSwap(sender=router, ...)
  5. Extension checks: allowedSwapper[pool][router] == true  → passes
  6. Swap executes; Alice receives output tokens

Result:
  Alice successfully swaps on a pool she is not allowlisted for.
  The allowlist guard is completely bypassed via the public router.
```

Foundry test outline:
- Deploy `SwapAllowlistExtension`, configure a pool with it
- `vm.prank(admin); extension.setAllowedToSwap(pool, address(router), true)`
- `vm.prank(alice); router.exactInputSingle(...)` — assert swap succeeds
- `vm.prank(alice); pool.swap(...)` — assert reverts with `NotAllowedToSwap`
