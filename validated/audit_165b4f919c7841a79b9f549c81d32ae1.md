### Title
`FeeEstimation.gas_consumed` Omits L2 Gas, Making `overall_fee` Irreconcilable with Returned Breakdown - (`File: crates/apollo_rpc_execution/src/objects.rs`)

### Summary

`starknet_estimateFee` returns an `overall_fee` that includes L2 gas costs, but the `gas_consumed` field in the `FeeEstimation` response only reflects L1 gas (`gas_vector.l1_gas`). The L2 gas consumed is silently dropped. The OpenRPC spec describes `overall_fee` as `gas_consumed * gas_price + data_gas_consumed * data_gas_price`, but this formula is wrong for any V3 transaction with non-zero L2 gas usage. The returned breakdown is internally inconsistent: `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price < overall_fee`.

### Finding Description

In `tx_execution_output_to_fee_estimation` in `crates/apollo_rpc_execution/src/objects.rs`:

```rust
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),       // only L1 gas
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(), // only L1 data gas
    l1_data_gas_price,
    l2_gas_price,                                   // price present, but consumed is absent
    overall_fee: tx_execution_output.execution_info.receipt.fee, // includes L2 gas cost
    unit: tx_execution_output.price_unit,
})
```

`receipt.fee` is computed by `GasVector::cost()` which sums all three resources:

```rust
for (gas, price, resource) in [
    (self.l1_gas, gas_prices.l1_gas_price, Resource::L1Gas),
    (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
    (self.l2_gas, tipped_l2_gas_price, Resource::L2Gas),   // included in fee
] { ... }
```

But `gas_consumed` is set to `gas_vector.l1_gas.0` only — `gas_vector.l2_gas` is never placed in any returned field. The `FeeEstimation` struct has no `l2_gas_consumed` field at all. A developer comment in the struct acknowledges this:

```rust
// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
// close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
```

The OpenRPC spec at `crates/apollo_rpc/resources/V0_8/starknet_api_openrpc.json` describes `overall_fee` as:

> "equals to gas_consumed\*gas_price + data_gas_consumed\*data_gas_price"

This description is wrong for any V3 transaction with non-zero L2 gas usage. The actual `overall_fee` is:

```
overall_fee = l1_gas * l1_gas_price
            + l1_data_gas * l1_data_gas_price
            + l2_gas * (l2_gas_price + tip)   ← missing from breakdown
```

### Impact Explanation

**High. RPC execution, fee estimation returns an authoritative-looking wrong value.**

Any caller of `starknet_estimateFee` for a V3 transaction with non-zero L2 gas usage receives a response where:

1. `overall_fee` is correct (includes L2 gas cost).
2. `gas_consumed` is understated (L2 gas omitted).
3. The formula `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price` produces a value strictly less than `overall_fee`.

A client that uses `gas_consumed` to set `l2_gas.max_amount` resource bounds (the natural interpretation, since `l2_gas_price` is returned but no `l2_gas_consumed` is) would set it to zero or an incorrect value, causing the transaction to fail at execution with an out-of-gas error or be rejected at pre-validation. The fee estimation response is authoritative-looking but internally inconsistent.

### Likelihood Explanation

**High.** Every V3 transaction (`AllResources` bounds) that executes Cairo code consumes L2 gas. Since Starknet 0.13.3, V3 transactions are the standard. Any client calling `starknet_estimateFee` for a V3 transaction and attempting to reconstruct or verify the fee from the returned breakdown, or using `gas_consumed` to set `l2_gas.max_amount`, will encounter the discrepancy. No special privileges or unusual inputs are required.

### Recommendation

Add `l2_gas_consumed` to `FeeEstimation` and populate it from `gas_vector.l2_gas.0` in `tx_execution_output_to_fee_estimation`. Update the OpenRPC spec description of `overall_fee` to include the L2 gas term. The corrected construction should be:

```rust
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(),
    l1_data_gas_price,
    l2_gas_consumed: gas_vector.l2_gas.0.into(),   // add this
    l2_gas_price,
    overall_fee: tx_execution_output.execution_info.receipt.fee,
    unit: tx_execution_output.price_unit,
})
```

### Proof of Concept

1. Submit any V3 invoke transaction to `starknet_estimateFee`.
2. Observe the returned `FeeEstimation`: `gas_consumed = X`, `l1_gas_price = P1`, `data_gas_consumed = Y`, `l1_data_gas_price = P2`, `l2_gas_price = P3`, `overall_fee = F`.
3. Compute `X * P1 + Y * P2`. For any transaction with non-zero L2 gas usage, this value is strictly less than `F`.
4. The difference `F - (X * P1 + Y * P2)` equals `l2_gas_consumed * (l2_gas_price + tip)`, which is the L2 gas cost silently included in `overall_fee` but absent from the breakdown.

    pub fn cost(&self, gas_prices: &GasPriceVector, tip: Tip) -> Fee {
        let tipped_l2_gas_price =
            gas_prices.l2_gas_price.checked_add(tip.into()).unwrap_or_else(|| {
                panic!(
                    "Tip overflowed: addition of L2 gas price ({}) and tip ({}) resulted in \
                     overflow.",
                    gas_prices.l2_gas_price, tip
                )
            });

        let mut sum = Fee(0);
        for (gas, price, resource) in [
            (self.l1_gas, gas_prices.l1_gas_price, Resource::L1Gas),
            (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
            (self.l2_gas, tipped_l2_gas_price, Resource::L2Gas),
        ] {
            let cost = gas.checked_mul(price.get()).unwrap_or_else(|| {
                panic!(
                    "{resource} cost overflowed: multiplication of gas amount ({gas}) by price \
                     per unit ({price}) resulted in overflow."
                )
            });
            sum = sum.checked_add(cost).unwrap_or_else(|| {
                panic!(
                    "Total cost overflowed: addition of current sum ({sum}) and cost of \
                     {resource} ({cost}) resulted in overflow."
                )
            });
        }
        sum
    }
