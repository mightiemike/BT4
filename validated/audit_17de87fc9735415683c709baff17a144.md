Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual trader — allowlist fully bypassed by any user via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the `msg.sender` of the pool — which is the router contract, not the originating trader. When a pool admin allowlists the `MetricOmmSimpleRouter` to enable router-based swaps, every unprivileged user can bypass the per-user allowlist by routing through it. Conversely, if the router is not allowlisted, individually allowlisted users cannot swap through the router at all, making the extension non-functional with the router.

## Finding Description

**Call chain:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle()` (line 71–80). The router stores `msg.sender` (the actual user) in transient storage as the payer, then calls `pool.swap(params.recipient, ...)`. At this point `msg.sender` inside the pool is the **router address**.

2. `MetricOmmPool.swap()` (line 230–240) calls `_beforeSwap(msg.sender, ...)` — passing the **router address** as `sender`.

3. `ExtensionCalling._beforeSwap()` (line 160–176) encodes `sender` (= router) as the first argument and dispatches to the extension.

4. `SwapAllowlistExtension.beforeSwap()` (line 37–39) executes:
   ```solidity
   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
       revert IMetricOmmPoolActions.NotAllowedToSwap();
   }
   ```
   Here `msg.sender` = pool, `sender` = **router**. The check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

**Contrast with `DepositAllowlistExtension`:** that extension correctly ignores `sender` and checks `owner` (line 38), which is an explicit parameter the caller supplies — so the deposit allowlist is not affected.

**Why existing guards are insufficient:** The pool has no mechanism to forward the original EOA through the hook. The transient storage in the router stores the real payer for payment purposes only; it is never surfaced to the extension. There is no guard in `SwapAllowlistExtension` that recovers the original caller.

**Exploit flow:**
- Pool admin deploys pool with `SwapAllowlistExtension`, intending to restrict swaps to a set of KYC'd addresses.
- Admin must allowlist the router address to allow router-based swaps (or users discover the router is already allowlisted).
- Any non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle()` targeting that pool.
- `beforeSwap` receives `sender` = router address, which is allowlisted → check passes → swap executes.
- The per-user allowlist is completely bypassed.

## Impact Explanation
The swap allowlist — a core access-control extension — is rendered ineffective for any pool that uses `MetricOmmSimpleRouter`. Unprivileged, non-allowlisted traders can execute swaps in pools intended to be restricted. This constitutes broken core pool functionality (access control bypass). While the swap itself is oracle-priced and does not directly steal LP funds, it violates the pool admin's enforced access boundary and enables unauthorized participation in restricted pools, which may have downstream compliance, regulatory, or economic consequences for LPs who deposited under the assumption of a restricted swapper set.

## Likelihood Explanation
Exploitability is high once the router is allowlisted for a given pool. The router is a public, permissionless contract. Any user can call `exactInputSingle` with any pool address. No special privileges, flash loans, or off-chain data are required. The condition (router allowlisted) is a natural operational state for any pool that intends to support router-based trading alongside the allowlist.

## Recommendation
The extension must check the actual originating trader, not the intermediary caller. Two options:

1. **Pass the real trader explicitly:** Add a `trader` field to `extensionData` that the router populates with `msg.sender`, and have `SwapAllowlistExtension` decode and verify it. This requires the pool to enforce that `extensionData` is not forgeable (e.g., via a trusted router registry).

2. **Check `recipient` instead of `sender`:** If the protocol's invariant is that the recipient is always the actual user, check `allowedSwapper[pool][recipient]`. However, this is only safe if the recipient cannot be set to an arbitrary address by an attacker.

3. **Structural fix:** Introduce a dedicated `trader` parameter in the `beforeSwap` hook signature that the pool populates from a verified source (e.g., transient storage set by the router before calling the pool).

```solidity

pragma solidity ^0.8.35;

// Setup:
// 1. Deploy pool with SwapAllowlistExtension.
// 2. Pool admin calls setAllowedToSwap(pool, address(router), true)
//    (necessary for router-based swaps to work).
// 3. Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

// Attack:
// attacker (non-allowlisted) calls:
router.exactInputSingle(ExactInputSingleParams({
    pool: restrictedPool,
    recipient: attacker,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    tokenIn: token0,
    extensionData: "",
    deadline: block.timestamp
}));

// Result:
// pool.swap() is called with msg.sender = router (allowlisted).
// SwapAllowlistExtension.beforeSwap receives sender = router → passes.
// Attacker swaps successfully despite not being on the allowlist.
```
