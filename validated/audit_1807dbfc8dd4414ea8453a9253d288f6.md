### Title
Linked Signer Can Overwrite Itself to Escalate and Persist Unauthorized Signing Authority — (`core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.sol`, the sequencer-path handler for `LinkSigner` transactions calls `validateSignedTx` with `allowLinkedSigner = true`. This means the **currently registered linked signer** — not just the subaccount owner — can sign and submit a `LinkSigner` transaction to replace the linked signer with any address it chooses. A malicious or compromised linked signer can silently rotate signing authority to an attacker-controlled address, and because all high-value operations (including `WithdrawCollateral`) also accept linked-signer signatures, the attacker can immediately drain the subaccount.

---

### Finding Description

`EndpointTx.processTransactionImpl` handles the sequencer-path `LinkSigner` transaction type at: [1](#0-0) 

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
```

`validateSignedTx` with `allowLinkedSigner = true` resolves to `validateSignature`, which passes the current linked signer to the verifier as an acceptable signer: [2](#0-1) 

```solidity
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
```

`getLinkedSigner` returns `linkedSigners[subaccount]` for a normal subaccount: [3](#0-2) 

The `linkedSigners` mapping is stored in `EndpointStorage`: [4](#0-3) 

There is **no check** that the new signer value is different from the current one, that the current linked signer is not the signer, or that the subaccount owner has consented. The write is unconditional: [5](#0-4) 

By contrast, the slow-mode path for `LinkSigner` correctly restricts the caller to the subaccount's own address via `validateSender`: [6](#0-5) 

The asymmetry means the sequencer path is the only path where a non-owner can mutate `linkedSigners`.

---

### Impact Explanation

Once a linked signer rotates itself to an attacker-controlled address, the attacker inherits full signing authority over the subaccount. Every sequencer-path transaction that uses `allowLinkedSigner = true` is now reachable by the attacker, including:

- `WithdrawCollateral` / `WithdrawCollateralV2` — direct collateral drain
- `MintNlp` / `BurnNlp` — NLP position manipulation
- `TransferQuote` — quote balance exfiltration
- Further `LinkSigner` calls — the attacker can keep rotating the signer to prevent the victim from reclaiming control via the sequencer path [7](#0-6) 

The victim's only recourse is the slow-mode `LinkSigner` path, which carries a hardcoded three-day delay (`SLOW_MODE_TX_DELAY`). During that window the attacker can drain all collateral. [8](#0-7) 

---

### Likelihood Explanation

Any user who has ever called `LinkSigner` — a standard operation for trading bots, API keys, and third-party integrations — is exposed. The attack requires only that the linked signer key is compromised or that the linked signer is a malicious contract/service. No admin access, no governance capture, and no sequencer compromise is needed. The attacker-controlled entry path is the normal sequencer transaction pipeline (`Endpoint.submitTransactions` → `processTransactionImpl`), which is the primary execution path for all protocol activity.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This aligns the sequencer path with the slow-mode path, which already enforces `validateSender` (owner-only). A linked signer should be able to act *as* the subaccount owner for trading operations, but it should never be able to *replace* the subaccount owner's designated signer. [9](#0-8) 

---

### Proof of Concept

1. Alice sets `linkedSigners[aliceSubaccount] = botAddress` via a normal `LinkSigner` transaction.
2. `botAddress` is compromised. The attacker constructs a `SignedLinkSigner` where `tx.sender = aliceSubaccount`, `tx.signer = attackerAddress`, `tx.nonce = currentNonce`, signed by `botAddress`.
3. The sequencer includes the transaction; `processTransactionImpl` validates the signature against `botAddress` (the current linked signer) — validation passes.
4. `linkedSigners[aliceSubaccount]` is overwritten with `attackerAddress`.
5. The attacker immediately submits a `SignedWithdrawCollateral` signed by `attackerAddress`, draining Alice's collateral before she can submit a slow-mode revocation.
6. If Alice submits a sequencer-path `LinkSigner` to revoke, the attacker can race it with another rotation (step 2–4 repeated with the next nonce), keeping control until the slow-mode window expires. [1](#0-0) [7](#0-6)
