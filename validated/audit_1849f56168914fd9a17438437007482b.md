Audit Report

## Title
SwapAllowlistExtension Bypassed via Router: `sender` Is Router, Not End-User — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of `pool.swap`. When a user routes through `MetricOmmSimpleRouter`, the pool sees the router as `sender`, not the end-user. A pool admin who allowlists the router to enable router-mediated swaps inadvertently grants every user — including non-allowlisted ones — the ability to bypass the swap gate, since the router is a public, permissionless contract.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` directly as `sender` to `_beforeSwap`: [1](#0-0) 

When `MetricOmmSimpleRouter.exactOutputSingle` calls `pool.swap`, it never forwards the original caller's identity: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates: [3](#0-2) 

Here `msg.sender` is the pool and `sender` is the router. The end-user's address is never consulted. Any user routing through `MetricOmmSimpleRouter` passes the check as long as the router is allowlisted, regardless of whether the user themselves is allowlisted.

## Impact Explanation
The swap allowlist — the primary mechanism for pool curation — is completely ineffective for any user routing through `MetricOmmSimpleRouter`. A curated pool restricting swaps to a specific set of users provides no restriction once the router is allowlisted. Non-allowlisted users can execute swaps and receive output tokens, draining pool liquidity in violation of admin-configured access control. This constitutes broken core pool functionality with direct fund-flow impact.

## Likelihood Explanation
High. `MetricOmmSimpleRouter` is the standard public swap interface. Pool admins configuring a swap allowlist who also want to support router-mediated swaps will naturally allowlist the router. The bypass requires no special knowledge — any user can call `exactOutputSingle` or `exactInputSingle` on the router against the curated pool.

## Recommendation
Forward the original initiator through the router to the pool. The cleanest fix is for `MetricOmmSimpleRouter` to pass `msg.sender` as part of `extensionData` (or a dedicated `initiator` field in the swap call), and have `SwapAllowlistExtension.beforeSwap` decode and verify that address instead of `sender`. Alternatively, redesign the allowlist to gate at the router level. Pool admin documentation must explicitly warn that allowlisting the router grants access to all users until a fix is deployed.

## Proof of Concept
```solidity
// Foundry integration test
function test_nonAllowlistedUserBypassesSwapAllowlistViaRouter() public {
    // Pool admin allowlists only the router (not the user)
    swapExtension.setAllowedToSwap(address(pool), address(router), true);

    address nonAllowlistedUser = makeAddr("attacker");
    deal(address(tokenIn), nonAllowlistedUser, 10_000);
    vm.prank(nonAllowlistedUser);
    tokenIn.approve(address(router), type(uint256).max);

    uint256 balanceBefore = tokenOut.balanceOf(nonAllowlistedUser);

    // Non-allowlisted user swaps through the router — should revert, but doesn't
    vm.prank(nonAllowlistedUser);
    router.exactOutputSingle(IMetricOmmSimpleRouter.ExactOutputSingleParams({
        pool: address(pool),
        tokenIn: address(tokenIn),
        tokenOut: address(tokenOut),
        zeroForOne: true,
        amountOut: 1000,
        amountInMaximum: 10_000,
        recipient: nonAllowlistedUser,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    }));

    // Assert non-allowlisted user received output — allowlist was bypassed
    assertGt(tokenOut.balanceOf(nonAllowlistedUser) - balanceBefore, 0);
}
```
