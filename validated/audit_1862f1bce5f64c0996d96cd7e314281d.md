Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Swap Allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap()` gates swaps by checking the `sender` argument, which the pool sets to its own `msg.sender` â€” the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` is used, the router is `msg.sender` to the pool, so the allowlist checks the router's address rather than the actual end user's address. Any non-allowlisted user can bypass the curated-pool restriction by routing through `MetricOmmSimpleRouter` instead of calling the pool directly.

## Finding Description
**Root cause â€” pool passes its own `msg.sender` as `sender` to the hook:**

In `MetricOmmPool.swap()`, `_beforeSwap` is called with `msg.sender` as the first argument: [1](#0-0) 

**Hook checks `sender` (the pool's `msg.sender`) against the allowlist:**

`SwapAllowlistExtension.beforeSwap()` uses `msg.sender` (the pool) as the mapping key and `sender` (the caller of `pool.swap()`) as the identity being gated: [2](#0-1) 

**Router calls `pool.swap()` directly, making itself `msg.sender`:**

In `exactInputSingle`, the router calls `pool.swap()` directly. The pool's `msg.sender` is therefore the router contract, not the end user. The actual user's address is stored only in the callback context (for payment), never forwarded to the pool as the swap initiator: [3](#0-2) 

The same pattern applies to `exactInput` (L103-112), `exactOutputSingle` (L135-137), and `exactOutput` (L165-181). [4](#0-3) 

**Result:** The allowlist check resolves to `allowedSwapper[pool][router]`. This creates an irreconcilable dilemma:

| Router allowlisted? | Effect |
|---|---|
| Yes | Every user bypasses the allowlist by routing through the router |
| No | Allowlisted users cannot use the router; they must call the pool directly |

No existing guard prevents this. The `extensionData` field is passed through but `SwapAllowlistExtension` does not decode it to recover the real initiator. There is no transient-storage context exposed to extensions that would carry the verified original sender.

## Impact Explanation
A pool admin deploys a curated pool with `SwapAllowlistExtension` to restrict swaps to KYC'd or otherwise vetted addresses. If the router is allowlisted (the only way to enable router-mediated swaps for any user), any non-allowlisted user can call `MetricOmmSimpleRouter.exactInputSingle()` and swap freely against the pool's liquidity. The allowlist guard is completely defeated. LP funds in the curated pool are exposed to unauthorized swappers, which can result in direct loss of LP principal through adverse-selection trades the pool was specifically configured to prevent. This is a broken core pool functionality / admin-boundary break causing potential loss of funds.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap entrypoint. Any user who reads the interface can call it. No privileged access, no special setup, and no malicious initial configuration is required beyond the pool admin having allowlisted the router (a necessary operational step). The attacker only needs to call a public router function with a valid swap path.

## Recommendation
The `beforeSwap` hook signature receives `sender` (the pool's `msg.sender`) but not the original transaction initiator. The correct fix is to have the pool record the original transaction initiator via transient storage â€” analogous to how the router already uses `_setNextCallbackContext` â€” and expose it to extensions. The router should write the real user into a trusted transient slot before calling `pool.swap()`, and the pool should read that slot and pass it as a verified `originator` parameter to extension hooks. Alternatively, the `beforeSwap` hook signature should be extended to carry a verified original sender set by the pool from a trusted periphery context rather than from raw `msg.sender`. Until fixed, `SwapAllowlistExtension` should not be used with `MetricOmmSimpleRouter` on pools where the allowlist is a security boundary.

## Proof of Concept
```
1. Pool admin deploys a pool with SwapAllowlistExtension.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   â€” required so that any router-mediated swap can pass the hook.
3. Alice (allowlisted) and Bob (NOT allowlisted) both want to swap.
4. Bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...}).
5. Router calls pool.swap(...) â€” msg.sender to pool = router.
6. Pool calls _beforeSwap(router, recipient, ...).
7. SwapAllowlistExtension.beforeSwap checks allowedSwapper[pool][router] == true â†’ passes.
8. Bob's swap executes successfully despite not being on the allowlist.
9. The curated pool's access control is completely bypassed.
```
