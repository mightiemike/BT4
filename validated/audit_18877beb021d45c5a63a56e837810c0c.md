All cited code paths are confirmed in the repository. The vulnerability is real and structurally sound:

1. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `sender` is the pool's immediate `msg.sender`. [1](#0-0) 

2. `MetricOmmPool.swap()` passes `msg.sender` (the direct caller) as `sender` to `_beforeSwap`. [2](#0-1) 

3. `ExtensionCalling._beforeSwap` encodes that `sender` and forwards it to the extension. [3](#0-2) 

4. `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the pool's `msg.sender`. [4](#0-3) 

5. The same applies to `exactInput` multi-hop paths. [5](#0-4)

Audit Report

## Title
SwapAllowlistExtension Allowlist Bypass via Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is the immediate `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the original user. A pool admin who allowlists the router to enable router-mediated swaps for their users inadvertently grants every router user access to the pool, bypassing the per-user allowlist entirely.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap` (ExtensionCalling.sol L230–240). `ExtensionCalling._beforeSwap` encodes this `sender` and forwards it to each configured extension (ExtensionCalling.sol L162–176). `SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the forwarded value (SwapAllowlistExtension.sol L37). When `MetricOmmSimpleRouter.exactInputSingle` or `exactInput` calls `pool.swap()`, the pool's `msg.sender` is the router address, so the extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][originalUser]`. If the pool admin has allowlisted the router — a natural action to enable router-mediated swaps — the check passes for every user who routes through it, regardless of individual allowlist status. No existing guard in the extension, pool, or router prevents this.

## Impact Explanation
The `SwapAllowlistExtension` is the sole on-chain mechanism for restricting which addresses may trade in a pool. Bypassing it allows unauthorized users to execute swaps in pools explicitly configured to exclude them. Pools using the allowlist to enforce KYC/regulatory restrictions, gate access to favorable oracle-anchored rates, or limit trading to trusted counterparties are fully exposed once the router is allowlisted. Unauthorized swaps drain liquidity at rates LPs did not consent to serve to arbitrary users, constituting a direct loss of LP-owed assets and a broken core pool access invariant — matching the allowed impact of broken core pool functionality causing loss of funds.

## Likelihood Explanation
The bypass requires only that the pool admin allowlists the router, which is the natural and expected configuration for any pool that wants to support the standard periphery swap path. The `SwapAllowlistExtension` NatSpec (`@notice Gates swap by swapper address, per pool`) does not warn that allowlisting any intermediary grants that intermediary's entire user base access. The misconfiguration is easy to make, the bypass is trivially reachable by any unprivileged user with no special privileges, and no on-chain guard prevents it.

## Recommendation
The extension must check the original user identity, not the immediate pool caller. The router should encode `msg.sender` into `extensionData`; the extension decodes and checks it. Alternatively, at minimum, the `SwapAllowlistExtension` NatSpec must explicitly warn that allowlisting any intermediary contract (router, multicall, aggregator) grants that contract's entire user base access to the pool, and pool admins must be instructed never to allowlist the router if per-user gating is required.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured in `beforeSwap` order.
2. Pool admin calls `setAllowedToSwap(pool, user1, true)` and `setAllowedToSwap(pool, router, true)`.
3. `user2` (not individually allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(...)` — pool's `msg.sender` = router address.
5. Pool calls `_beforeSwap(sender=router, ...)` → extension evaluates `allowedSwapper[pool][router]` → `true` → no revert.
6. `user2`'s swap executes successfully in a pool they are not authorized to trade in.
