### Title
Missing `l2_gas_consumed` and `next_l2_gas_price` Fields in `gas_prices_to_hash` for V0_14.x Blocks Produces Wrong Block Hash — (File: `crates/starknet_api/src/block_hash/block_hash_calculator.rs`)

### Summary

The `gas_prices_to_hash` function, which computes the gas-prices component of the Starknet block hash, omits `l2_gas_consumed` and `next_l2_gas_price` for all Starknet versions ≥ V0_13_4, including every V0_14.x production block. The `BlockHashVersion` enum has no V0_14.x variant; all V0_14.x versions collapse to `BlockHashVersion::V0_13_4` and use the same six-field formula. A TODO comment in the source explicitly acknowledges the gap. The echonet replay/verification system passes both missing fields to its block-hash CLI, creating a canonicalization divergence between the sequencer's stored block hash and the verification system's expected hash.

### Finding Description

`BlockHashVersion` is defined with only two variants:

```rust
pub enum BlockHashVersion {
    V0_13_2,
    V0_13_4,
}
```

The `TryFrom<StarknetVersion>` conversion maps every version ≥ V0_13_4 — including V0_14_0 through V0_14_4 (the current `LATEST`) — to `BlockHashVersion::V0_13_4`:

```rust
} else {
    Ok(Self::V0_13_4)  // ALL versions >= V0_13_4 collapse here
}
```

`gas_prices_to_hash`, called from `calculate_block_hash`, therefore uses the same six-field Poseidon preimage for every V0_14.x block:

```
Poseidon("STARKNET_GAS_PRICES0" || l1_gas_price_wei || l1_gas_price_fri
         || l1_data_gas_price_wei || l1_data_gas_price_fri
         || l2_gas_price_wei || l2_gas_price_fri)
```

The acknowledged-but-unimplemented TODO at line 416 states:

```rust
// TODO(Ayelet): add l2_gas_consumed, next_l2_gas_price after 0.14.0.
```

The echonet production verification system (`_compute_block_hash`) passes both missing fields to the block-hash CLI:

```python
"l2_gas_consumed": fee_market_info["l2_gas_consumed"],
"next_l2_gas_price": fee_market_info["next_l2_gas_price"],
```

`BlockHeaderWithoutHash` stores `l2_gas_consumed` and `next_l2_gas_price` as first-class fields, and `PartialBlockHashComponents` (the struct fed into `calculate_block_hash`) does not carry them, so the Rust hash path structurally cannot include them even if the version gate were added.

A second missing field is `fee_proposal_fri`, which `BlockHeaderWithoutHash` carries with the comment `// TODO(AndrewL): Add this field into the block hash.`

### Impact Explanation

Every block produced at V0_14.x (the current production version) receives a block hash whose gas-prices sub-hash is computed over six fields instead of the required eight. The stored `BlockHash`, the `PartialBlockHash` used as the consensus `ProposalCommitment`, and the value served by the RPC are all wrong. Any external verifier or prover that implements the correct V0_14.x formula will compute a different hash, breaking cross-component hash consistency. This matches the impact category: **High — RPC execution/tracing/simulation returns an authoritative-looking wrong value** (wrong block hash in `starknet_getBlockWithTxHashes`, `starknet_getBlockWithTxs`, etc.) and potentially **Critical — wrong state value** (block hash stored in the DB and used as `previous_block_hash` for the next block's hash chain).

### Likelihood Explanation

The trigger is unconditional: every block at V0_14.x (all current production blocks) exercises the broken path. No special transaction or attacker input is required. The `BlockHashVersion` enum simply has no V0_14.x variant, so the missing fields are silently dropped on every block finalization.

### Recommendation

1. Add a `V0_14_0` (or later) variant to `BlockHashVersion` and extend `TryFrom<StarknetVersion>` to map V0_14.x versions to it.
2. Add `l2_gas_consumed: GasAmount` and `next_l2_gas_price: GasPrice` to `PartialBlockHashComponents`.
3. Update `gas_prices_to_hash` to accept and chain these fields when `block_hash_version >= V0_14_0`.
4. Update `PartialBlockHashComponents::new` to populate the new fields from `BlockInfo` (or the `FeeMarketInfo` blob).
5. Resolve the `fee_proposal_fri` TODO in `BlockHeaderWithoutHash` with the same versioned-gate pattern.
6. Add a regression test that fixes the expected hash for a V0_14.x block and asserts it matches the echonet CLI output.
