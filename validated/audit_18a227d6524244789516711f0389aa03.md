### Title
Stale Cumulative Interest Multipliers in `getBalance()` Cause Understated Borrower Debt in Health Checks — (`core/contracts/SpotEngineState.sol`)

---

### Summary

`SpotEngineState.getBalance()` reads the last-stored `cumulativeBorrowsMultiplierX18` directly from storage without advancing it to the current time. Every health check in the protocol — including those guarding `withdrawCollateral`, `transferQuote`, `mintNlp`, and `liquidateSubaccountImpl` — is computed using this stale multiplier. Borrower debt is systematically understated between `updateStates` calls, enabling excess collateral withdrawal and liquidation bypass.

---

### Finding Description

The `SpotEngine` tracks borrower debt using a normalized balance system. Each borrower's raw stored value (`amountNormalized < 0`) is multiplied by `cumulativeBorrowsMultiplierX18` to produce the actual debt amount. This multiplier grows over time as interest accrues and is only advanced when `updateStates(dt)` is called — a function restricted to `onlyEndpoint`.

`getBalance` in `SpotEngineState.sol` reads `states[productId]` directly from storage:

```solidity
function getBalance(uint32 productId, bytes32 subaccount)
    public view returns (Balance memory)
{
    State memory state = states[productId];                          // stale state
    BalanceNormalized memory balance = balances[productId][subaccount];
    return balanceNormalizedToBalance(state, balance);               // uses stale multiplier
}
```

`balanceNormalizedToBalance` then multiplies `amountNormalized` by the stale `cumulativeBorrowsMultiplierX18`:

```solidity
return Balance(balance.amountNormalized.mul(cumulativeMultiplierX18));
```

Because the true current multiplier is always `≥` the stored one (interest only grows), the returned balance is always `≥` the true balance for borrowers (i.e., the debt magnitude is understated). The gap widens with every second that passes since the last `updateStates` call.

This stale value propagates directly into `getHealth` via the call chain:

`Clearinghouse.getHealth` → `spotEngine.getHealthContribution` → `BaseEngine._calculateProductHealth` → `SpotEngineState._getBalance` → `SpotEngineState.getBalance`

`getHealth` is the sole health oracle for:
- `Clearinghouse.withdrawCollateral` (line 419)
- `Clearinghouse.transferQuote` (line 249)
- `Clearinghouse.mintNlp` (line 480)
- `ClearinghouseLiq.isUnderMaintenance` → `liquidateSubaccountImpl` (line 603)

---

### Impact Explanation

**Excess collateral withdrawal**: A borrower whose true health (with current interest) would be negative after a withdrawal passes the health check because `getHealth` uses the stale, lower debt figure. The borrower extracts collateral the protocol should have blocked.

**Liquidation bypass**: A borrower whose true maintenance health is negative (i.e., should be liquidatable) appears healthy to `isUnderMaintenance` because the debt is understated. The liquidation call reverts with `ERR_NOT_LIQUIDATABLE`, leaving an undercollateralized position open. As interest continues to accrue, the gap between true and reported health widens, increasing the protocol's bad-debt exposure.

Both impacts corrupt the solvency invariant: the protocol's accounting of outstanding obligations is understated relative to the collateral it holds.

---

### Likelihood Explanation

`updateStates` is called by the sequencer on a periodic schedule. Between any two consecutive calls — which can span minutes to hours depending on sequencer cadence — every health check in the system operates on stale multipliers. Additionally, `Endpoint.executeSlowModeTransaction()` carries no access control and can be called by any user; slow-mode withdrawals processed this way are never preceded by a mandatory `updateStates` call, making the stale-read window unbounded for that path. Any borrower with an open spot liability can exploit this window.

---

### Recommendation

`getBalance` (and `getStateAndBalance`) should compute the up-to-date balance by projecting the stored multiplier forward to the current oracle time before multiplying. Concretely, `_updateState` logic should be applied in a read-only manner inside `getBalance` using the elapsed `dt = currentTime - lastUpdateTime`, so that every health check reflects the true current debt without requiring a prior state-mutating call.

---

### Proof of Concept

1. Deploy the protocol. Add a spot product with a non-zero borrow rate.
2. Subaccount A deposits collateral and borrows the maximum allowed amount (health just above zero).
3. Advance block time by several hours. Do **not** call `updateStates`.
4. Call `Clearinghouse.getHealth(A, MAINTENANCE)`. Observe it returns a value above zero (stale multiplier understates debt).
5. Call `Clearinghouse.withdrawCollateral` for subaccount A. Observe the health check passes and the withdrawal succeeds, even though the true health (with accrued interest) is negative.
6. Now call `updateStates` to advance the multiplier. Call `getHealth(A, MAINTENANCE)` again. Observe it now returns a negative value — the subaccount is undercollateralized and should have been blocked at step 5.

The root cause is `getBalance` at [1](#0-0)  reading the stale `cumulativeBorrowsMultiplierX18` from `states[productId]` without advancing it, while `updateStates` — the only function that advances the multiplier — is gated to `onlyEndpoint` at [2](#0-1) . The stale value flows into `getHealth` via `_calculateProductHealth` at [3](#0-2) , which is the sole health oracle for the withdrawal guard at [4](#0-3)  and the liquidation eligibility check at [5](#0-4) .

```text
    function updateStates(uint128 dt) external onlyEndpoint {
        State memory quoteState;
        require(dt < 7 * SECONDS_PER_DAY, ERR_INVALID_TIME);
        for (uint32 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            if (productId == NLP_PRODUCT_ID) {
                continue;
            }
            State memory state = states[productId];
            if (productId == QUOTE_PRODUCT_ID) {
                quoteState = state;
            }
            if (state.totalDepositsNormalized == 0) {
                continue;
            }
            _updateState(productId, state, dt);
            _setState(productId, state);
        }
    }
```
