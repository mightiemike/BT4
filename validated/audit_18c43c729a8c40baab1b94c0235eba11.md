### Title
Insurance Priority Ordering Unfairly Socializes Spot Depositor Losses Before Perp Losses Are Fully Absorbed — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

In `_finalizeSubaccount`, `perpEngine.socializeSubaccount` is called first with the full insurance balance. It can consume all available insurance to cover negative perp `vQuoteBalance`. Only the insurance *remainder* is then applied to the spot USDC deficit. `spotEngine.socializeSubaccount` is triggered when `v.insurance <= 0` — which can occur even when the pre-socialization insurance was sufficient to cover the USDC deficit. This hardcoded ordering systematically transfers losses from perp traders onto spot depositors.

### Impact Explanation

**Concrete numeric proof** (all values in USDC-equivalent):

| State variable | Value |
|---|---|
| `v.insurance` | 150 |
| Perp `vQuoteBalance` (all perp positions, `amount = 0`) | −200 |
| Spot USDC balance (`quoteBalance.amount`) | −100 |
| `v.canLiquidateMore` | `(−100 + 150) = 50 > 0` → **true** |

**Current code (perp-first)**:
1. `perpEngine.socializeSubaccount(liquidatee, 150)`: uses 150 insurance against −200 perp loss → 50 socialized to perp traders; returns `insurance = 0`
2. `insuranceCover = min(0, 100) = 0` → USDC deficit uncovered
3. `v.insurance = 0 ≤ 0` → `spotEngine.socializeSubaccount` called → **spot depositors absorb −100 USDC loss**

**Hypothetical spot-first ordering**:
1. `insuranceCover = min(150, 100) = 100` → USDC deficit fully covered; `insurance = 50`
2. `perpEngine.socializeSubaccount(liquidatee, 50)`: uses 50 insurance against −200 perp loss → 150 socialized to perp traders; returns `insurance = 0`
3. `v.insurance = 0 ≤ 0` → `spotEngine.socializeSubaccount` called → **USDC balance = 0, nothing socialized to spot depositors**

**Delta**: Spot depositors lose 100 USDC under the current ordering that they would not lose under a spot-first or proportional ordering. Perp traders absorb 100 fewer USDC of socialized loss. This is a direct, concrete value transfer from spot depositors to perp traders.

### Likelihood Explanation

This is triggered on every finalization of a subaccount that simultaneously holds:
- Negative perp `vQuoteBalance` (closed perp positions with unsettled negative PnL), and
- Negative spot USDC balance (e.g., from borrowed USDC or prior spot liquidations)

The settlement loop at lines 345–366 only nets negative perp PnL against a *positive* USDC balance. If USDC is already negative, no netting occurs, and both deficits coexist at the point of socialization. This is a normal end-state for an insolvent subaccount with mixed perp and spot exposure. Any liquidator calling `liquidateSubaccount` with `productId == type(uint32).max` on such a subaccount triggers the path. [8](#0-7) 

---

### Recommendation

Apply insurance proportionally across both perp and spot deficits before socializing either, or cover the spot USDC deficit with insurance *before* calling `perpEngine.socializeSubaccount`. The `v.canLiquidateMore` guard already signals the protocol's intent that insurance should cover the USDC deficit — the socialization order must be consistent with that intent.
