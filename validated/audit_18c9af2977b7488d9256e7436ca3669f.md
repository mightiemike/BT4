### Title
Fee-on-Transfer Token Deposit Inflates User Balance Beyond Actual Received Tokens ŌĆö (`File: core/contracts/EndpointStorage.sol`)

---

### Summary

`handleDepositTransfer()` in `EndpointStorage.sol` transfers `amount` from the user to the `Endpoint`, then forwards the same `amount` to the `Clearinghouse`, without accounting for any fee deducted by a fee-on-transfer token. The `DepositCollateral` slow-mode transaction is then queued with the original nominal `amount`, and `Clearinghouse.depositCollateral()` credits the full `amount` to the user's spot balance. The result is that the `Clearinghouse` holds fewer tokens than the sum of all credited user balances, creating a solvency shortfall that causes later withdrawals to fail.

---

### Finding Description

The deposit flow spans three contracts:

**Step 1 ŌĆö `EndpointStorage.handleDepositTransfer` (lines 111ŌĆō119)**

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);          // pulls `amount` from user
    safeTransferTo(token, address(clearinghouse), amount); // forwards `amount` to Clearinghouse
}
```

If the token charges a fee on transfer, the `Endpoint` receives `amount - feeŌéü` from the user. It then attempts to forward the full `amount` to the `Clearinghouse`. This either reverts (if the `Endpoint` has no surplus balance) or silently drains tokens already held by the `Endpoint` (e.g., slow-mode fees), causing the `Clearinghouse` to receive `amount - feeŌéé` at best. [1](#0-0) 

**Step 2 ŌĆö `Endpoint.depositCollateralWithReferral` (lines 144ŌĆō164)**

The slow-mode transaction is queued with the original caller-supplied `amount`, not the actual tokens received:

```solidity
handleDepositTransfer(..., uint256(amount));
// ...
DepositCollateral({
    sender: subaccount,
    productId: productId,
    amount: amount          // nominal, not actual received
})
``` [2](#0-1) 

**Step 3 ŌĆö `Clearinghouse.depositCollateral` (lines 193ŌĆō208)**

When the sequencer processes the slow-mode transaction, the full nominal `txn.amount` is scaled and credited to the user's spot balance:

```solidity
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
```

No check is made against the actual token balance held by the `Clearinghouse`. [3](#0-2) 

---

### Impact Explanation

The `Clearinghouse`'s real token balance becomes less than the sum of all credited user balances. Every deposit with a fee-on-transfer token widens this gap. When users later call `withdrawCollateral`, the `assertUtilization` check only compares `totalDepositsNormalized` against `totalBorrowsNormalized` ŌĆö both of which are derived from the inflated credited amounts ŌĆö so it does not catch the real-token shortfall. [4](#0-3) 

The first depositors can withdraw successfully; later depositors will find the `Clearinghouse` insolvent and their `withdrawCollateral` calls will revert due to insufficient token balance in the `WithdrawPool`. [5](#0-4) 

---

### Likelihood Explanation

Any user can trigger this by depositing a fee-on-transfer ERC-20 token that has been registered as a spot product. The entry point (`depositCollateral` / `depositCollateralWithReferral`) is fully permissionless and requires no special role. The impact scales with the number of deposits and the fee rate of the token. [6](#0-5) 

---

### Recommendation

Measure the actual tokens received by comparing balances before and after each transfer leg, and use the measured delta ŌĆö not the nominal `amount` ŌĆö when queuing the `DepositCollateral` slow-mode transaction and crediting the user's balance. Concretely:

1. In `handleDepositTransfer`, record `balanceBefore` of the `Clearinghouse`, execute both transfers, then compute `actualReceived = balanceAfter - balanceBefore`.
2. Return `actualReceived` from `handleDepositTransfer` and use it (after reversing the decimal multiplier) as the `amount` field in the queued `DepositCollateral` struct.

---

// User calls:

// Sum of credited user balances:      1000e18
// Shortfall: 19.9e18 tokens ŌĆö grows with every such deposit.

//   token.safeTransfer(withdrawPool, 1000e18)
