### Title
EIP-712 Domain Separator Uses `productId` as `verifyingContract` Instead of `address(this)`, Enabling Cross-Deployment Order Replay - (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.getDigest()` constructs a custom EIP-712 domain separator that places `address(uint160(productId))` — a `uint32` market identifier cast to an address — in the `verifyingContract` field instead of `address(this)`. This is a direct analog to the Wormhole chain ID confusion: a wrong identifier type is used in a critical cryptographic binding field, causing the domain separator to not bind to the actual contract. Any signed order is valid on any other `OffchainExchange` deployment that shares the same `productId` and `chainId`, enabling cross-deployment order replay.

---

### Finding Description

`getDigest()` in `OffchainExchange.sol` manually constructs an EIP-712 domain separator:

```solidity
bytes32 domainSeparator = keccak256(
    abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),
        _EIP712VersionHash(),
        block.chainid,
        address(uint160(productId))   // ← BUG: productId cast to address, not address(this)
    )
);
``` [1](#0-0) 

The EIP-712 standard mandates that the `verifyingContract` field be `address(this)` — the address of the contract that will verify the signature. Its sole purpose is to prevent cross-contract replay. Here, `productId` is a `uint32` market identifier (e.g., `1`, `2`, `3`), so `address(uint160(productId))` resolves to a near-zero address like `0x0000000000000000000000000000000000000001`. This is not the `OffchainExchange` contract address.

The contract inherits `EIP712Upgradeable` and calls `__EIP712_init("Nado", "0.0.1")`, which internally stores `address(this)` for use by `_hashTypedDataV4`. However, `getDigest()` bypasses `_hashTypedDataV4` entirely and builds its own domain separator with the wrong `verifyingContract`. [2](#0-1) 

This digest is used directly in `matchOrders()` and `createIsolatedSubaccount()` to validate trader signatures: [3](#0-2) 

`_validateOrder()` then calls `_checkSignature()` with this digest: [4](#0-3) 

The only replay protection per order is `filledAmounts[orderDigest]`, which is storage local to each contract deployment: [5](#0-4) 

---

### Impact Explanation

If the `OffchainExchange` contract is redeployed to a new address (e.g., during a protocol upgrade, migration, or proxy replacement), the `filledAmounts` mapping on the new deployment is empty. Because the domain separator does not include `address(this)`, every order digest computed on the old deployment is identical to the digest computed on the new deployment for the same `productId` and `chainId`. A sequencer or attacker can replay any previously signed order — including orders that were already fully filled on the old contract — against the new deployment. This causes unauthorized trades to execute against user balances, resulting in direct loss of funds.

The same applies if two `OffchainExchange` instances coexist on the same chain (e.g., a staging instance or a parallel deployment): orders signed for one instance are cryptographically valid on the other.

**Impact: 5** — unauthorized trade execution against user collateral, direct asset loss.

---

### Likelihood Explanation

Exploitation requires a redeployment of `OffchainExchange` to a new address on the same chain. The protocol uses upgradeable proxies (`ProxyManager`), but a proxy upgrade that changes the implementation does not change the proxy address, so it would not trigger this. However, a full redeployment (new proxy + new implementation), a migration, or a parallel deployment would. Given that the protocol is actively deployed on Ink Chain mainnet and testnet, and protocol upgrades are routine, this scenario is realistic.

**Likelihood: 1** — requires a specific redeployment event, not triggerable by an unprivileged user alone.

---

### Recommendation

Replace `address(uint160(productId))` with `address(this)` in the domain separator construction inside `getDigest()`:

```solidity
bytes32 domainSeparator = keccak256(
    abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),
        _EIP712VersionHash(),
        block.chainid,
        address(this)   // correct: binds to this contract's address
    )
);
```

Alternatively, use the inherited `_hashTypedDataV4(structHash)` directly, which already uses `address(this)` as `verifyingContract` via the `EIP712Upgradeable` base contract. Note that changing the domain separator will invalidate all currently outstanding signed orders, so a migration plan for active orders is required.

---

### Proof of Concept

1. User signs an order for `productId = 1` on the current `OffchainExchange` deployment at address `0xABCD...`. The domain separator encodes `verifyingContract = address(uint160(1)) = 0x0000...0001`.
2. The order is matched and `filledAmounts[digest]` is set to the full order amount on `0xABCD...`.
3. The protocol redeploys `OffchainExchange` to a new address `0xDEAD...` (e.g., as part of a migration).
4. On `0xDEAD...`, `filledAmounts` is empty. `getDigest(1, order)` produces the **identical digest** as on `0xABCD...` because `verifyingContract = address(uint160(1))` in both cases — neither encodes the actual contract address.
5. The sequencer (or any party with access to the old signed order bytes) submits `matchOrders()` on `0xDEAD...` with the original signature.
6. `_validateOrder()` passes: signature is valid, `filledAmounts[digest] == 0`, order is not expired.
7. The trade executes again, draining the user's collateral a second time without their consent. [6](#0-5)
