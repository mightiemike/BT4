### Title
Unconstrained `confirmations` Parameter Allows Zero-Confirmation Transaction Acceptance — (`contract/src/lib.rs`)

### Summary
The permissionless `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` functions accept a caller-controlled `confirmations` parameter with no minimum floor. Any NEAR caller can pass `confirmations = 0`, causing the confirmation check to be trivially satisfied for any transaction in any mainchain block regardless of reorg risk. This is the direct analog of `amountOutMin = 0`: a security-critical lower bound is left unconstrained, allowing any caller to silently bypass the protection.

### Finding Description
In `verify_transaction_inclusion`, the only validation on `confirmations` is an upper-bound check:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [1](#0-0) 

There is no lower-bound check. The actual confirmation guard is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [2](#0-1) 

With `confirmations = 0`, the right-hand side is `0`, and the left-hand side is always `>= 1` for any valid mainchain block, so the `require!` is unconditionally satisfied. The function returns `true` for any transaction in any mainchain block, regardless of how recently it was added or how likely a reorg is.

`verify_transaction_inclusion_v2` has the identical exposure because it converts its arguments and delegates directly to the deprecated v1 function:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

Neither function carries any access-control decorator beyond `#[pause]`; both are fully permissionless when the contract is unpaused. [4](#0-3) [5](#0-4) 

The `ProofArgs` struct exposes `confirmations` as a plain `u64` with no documented minimum: [6](#0-5) 

### Impact Explanation
Any downstream NEAR contract (bridge, escrow, cross-chain dApp) that calls `verify_transaction_inclusion{_v2}` and receives `true` treats the result as a security guarantee. With `confirmations = 0` the guarantee is hollow: the referenced Bitcoin block may be at the very tip of the chain and subject to reorganization. A bridge that releases funds on a `true` result for a 0-confirmation transaction can be drained if the underlying Bitcoin transaction is later reorganized away. The corrupted proof result is concrete: the function attests inclusion with a security level of zero when the caller supplies `confirmations = 0`.

### Likelihood Explanation
The entry path requires no privilege: any unprivileged NEAR account can call either function directly. A malicious actor who controls a Bitcoin transaction (or who can observe a freshly mined one) submits a valid Merkle proof with `confirmations = 0` to a downstream bridge contract. If that bridge forwards the caller-supplied `confirmations` value (a common pattern for flexibility), the attack is immediately executable. Even if the bridge hard-codes a value, a direct call to the light-client contract with `confirmations = 0` still returns `true`, which any contract reading the result on-chain would accept.

### Recommendation
Enforce a protocol-defined minimum confirmation floor inside `verify_transaction_inclusion`:

```rust
const MIN_CONFIRMATIONS: u64 = 1; // or a higher protocol-defined constant
require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    "confirmations must be at least MIN_CONFIRMATIONS"
);
```

Because the function is permissionless, the minimum must be enforced by the contract itself — callers cannot be trusted to supply a safe value, exactly as the AutoCompounder fix required dynamically computing `amountOutMin` rather than accepting a caller-supplied zero.

### Proof of Concept
1. Attacker broadcasts a Bitcoin transaction `T` and waits for it to appear in the most recent mainchain block (0 confirmations beyond inclusion).
2. Attacker calls `verify_transaction_inclusion_v2` on the NEAR light-client contract with a valid Merkle proof for `T` and `confirmations = 0`.
3. The upper-bound check passes (`0 <= gc_threshold`). The confirmation check evaluates `(tip_height - target_height + 1) >= 0`, which is always `true`.
4. The function returns `true`.
5. A downstream bridge contract that trusts this result releases the corresponding wrapped assets on NEAR.
6. A Bitcoin miner (or natural reorg) reorganizes the block containing `T`; `T` disappears from the canonical chain.
7. The bridge has permanently released assets for a transaction that no longer exists on Bitcoin.
