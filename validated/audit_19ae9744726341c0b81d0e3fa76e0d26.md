Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the originating user, allowing any caller to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the originating EOA. Any pool admin who allowlists the router to enable router-based swaps for their curated users inadvertently opens the pool to every user on the network, completely defeating the per-user allowlist.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to `_beforeSwap`:**

`MetricOmmPool.swap` at lines 230–240 calls `_beforeSwap(msg.sender, ...)`: [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards `sender` verbatim to the extension:**

`ExtensionCalling._beforeSwap` at lines 149–177 encodes `sender` directly into the call to each configured extension: [2](#0-1) 

**Step 3 — `SwapAllowlistExtension.beforeSwap` checks `sender` against the per-pool allowlist:**

The extension checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

**Step 4 — `MetricOmmSimpleRouter.exactInputSingle` is the direct caller of `pool.swap()`:**

The router calls `IMetricOmmPoolActions(params.pool).swap(...)` directly, making `msg.sender` inside the pool equal to `address(router)`, not the originating EOA: [4](#0-3) 

The same structural problem exists in `exactInput` (multi-hop) and `exactOutputSingle`/`exactOutput` paths: [5](#0-4) [6](#0-5) 

**Root cause:** The extension framework surfaces only the immediate pool caller (`msg.sender` of `pool.swap()`) to extensions. There is no mechanism to propagate the originating user's address. The extension has no visibility into who called the router. Once the pool admin calls `setAllowedToSwap(pool, router, true)`, the mapping `allowedSwapper[pool][router]` is `true` for every caller of the router, and the per-user gate is completely bypassed. [7](#0-6) 

## Impact Explanation
The `SwapAllowlistExtension` is the sole mechanism for restricting who may trade in a curated pool. When the router is allowlisted, any unprivileged user can trade in a pool intended to be private or compliance-gated by routing through `MetricOmmSimpleRouter`. Unauthorized users gain full access to oracle-priced liquidity. If the oracle price is even momentarily favorable, unauthorized users can extract value from LP positions in pools they were never meant to access. This constitutes a broken core pool functionality (access control) causing potential loss of funds to LPs in restricted pools.

## Likelihood Explanation
The pool admin allowlisting the router is the natural and expected action to make the pool usable through the standard periphery. There is no documentation warning against it. The `setAllowedToSwap` setter is straightforward and the bypass requires only a single `exactInputSingle` call from any EOA. Any user who discovers the pattern can exploit it immediately and repeatably. [7](#0-6) 

## Recommendation
The extension framework must surface the originating user's address to the extension, not just the immediate pool caller. Two viable approaches:

1. **Router-side**: `MetricOmmSimpleRouter` encodes `msg.sender` into `extensionData` on every swap call, and `SwapAllowlistExtension` reads the originating user from `extensionData` when `sender` is a known router.
2. **Extension-side**: `SwapAllowlistExtension` maintains a separate allowlist for trusted routers and, when `sender` is a trusted router, reads the actual user address from a standardized prefix in `extensionData`.

Either way, the extension must not treat the router address as the actor to be gated.

## Proof of Concept
```
1. Pool admin deploys pool with SwapAllowlistExtension configured.
2. Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is a curated user
3. Pool admin calls setAllowedToSwap(pool, router, true)  // to let alice use the router
4. Charlie (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({
           pool: pool,
           recipient: charlie,
           zeroForOne: true,
           amountIn: X,
           ...
       })
5. Router calls pool.swap(charlie, true, X, ...) with msg.sender = router.
6. Pool calls _beforeSwap(router, charlie, ...).
7. Extension evaluates: allowedSwapper[pool][router] == true → passes.
8. Charlie's swap executes in the curated pool despite never being allowlisted.
```
