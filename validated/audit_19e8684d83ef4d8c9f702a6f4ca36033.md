### Title
`WithdrawCollateralV2` Allows Users to Bypass Withdrawal Fees by Setting `feeX18 = 0` — (`core/contracts/EndpointTx.sol`)

---

### Summary

`WithdrawCollateral` (V1) always charges the full per-product `withdrawFeeX18` on every withdrawal. `WithdrawCollateralV2`, a parallel withdrawal path, allows the user to supply any `feeX18` value in the range `[0, currentFeeX18]`, including zero. A user can sign a `WithdrawCollateralV2` transaction with `feeX18 = 0` and pay no withdrawal fee at all, bypassing the fee that V1 unconditionally enforces.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `WithdrawCollateral` (V1) path always charges the full configured fee:

```solidity
chargeFee(
    signedTx.tx.sender,
    spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18,
    signedTx.tx.productId
);
``` [1](#0-0) 

The `WithdrawCollateralV2` path, however, reads a user-supplied `feeX18` field from the signed transaction and only validates that it is non-negative and does not exceed the current configured fee:

```solidity
int128 currentFeeX18 = spotEngine
    .getConfig(signedTx.tx.productId)
    .withdrawFeeX18;
require(signedTx.feeX18 >= 0);
require(signedTx.feeX18 <= currentFeeX18);
chargeFee(
    signedTx.tx.sender,
    signedTx.feeX18,
    signedTx.tx.productId
);
``` [2](#0-1) 

Because `require(signedTx.feeX18 >= 0)` permits `feeX18 = 0`, a user can sign a `WithdrawCollateralV2` transaction with `feeX18 = 0` and the on-chain code will accept it, charging nothing. The `withdrawFeeX18` is a per-product configuration value stored in `SpotEngine` and is also used as the basis for the minimum fast-withdrawal fee in `BaseWithdrawPool` (`5 * spotEngine().getConfig(productId).withdrawFeeX18`), confirming it is a protocol-intended revenue mechanism. [3](#0-2) 

The desynchronization is structural: V1 and V2 are two transaction types for the same underlying operation (`withdrawCollateral` on the `Clearinghouse`), but only V1 enforces the fee. V2 silently makes the fee optional. [4](#0-3) 

---

### Impact Explanation

Any user who submits a `WithdrawCollateralV2` transaction with `feeX18 = 0` pays zero withdrawal fee. The protocol loses all withdrawal fee revenue for that withdrawal. Since `withdrawFeeX18` is configured per product and is the same value used to floor fast-withdrawal fees, this represents a direct, quantifiable loss of protocol fee income on every V2 withdrawal where the user elects `feeX18 = 0`.

---

### Likelihood Explanation

The user signs the `WithdrawCollateralV2` transaction off-chain (including the `feeX18 = 0` field) and submits it to the sequencer. The sequencer's on-chain processing path (`processTransactionImpl`) enforces no minimum fee for V2 — the only on-chain constraints are `feeX18 >= 0` and `feeX18 <= currentFeeX18`. Any user who is aware of the V2 transaction type can exploit this on every withdrawal. The sequencer has no on-chain obligation to reject `feeX18 = 0` transactions; the protocol's own code accepts them.

---

### Recommendation

Enforce the full configured fee in `WithdrawCollateralV2` the same way V1 does, or at minimum require `signedTx.feeX18 == currentFeeX18`. If the intent of V2 is to allow fee negotiation (e.g., for promotional tiers), the lower bound should be a non-zero protocol-defined minimum, not zero.

---

### Proof of Concept

1. User constructs a `WithdrawCollateralV2` transaction struct with `feeX18 = 0` and a valid `productId`, `amount`, `nonce`, and `sendTo`.
2. User signs the transaction with their subaccount key.
3. Sequencer receives the signed transaction and calls `submitTransactionsChecked`, which routes to `processTransactionImpl`.
4. On-chain: `require(signedTx.feeX18 >= 0)` passes (0 ≥ 0). `require(signedTx.feeX18 <= currentFeeX18)` passes (0 ≤ any non-negative fee). `chargeFee(sender, 0, productId)` is called — zero fee is deducted.
5. `clearinghouse.withdrawCollateral(...)` executes normally, transferring the full `amount` to the user with no fee charged. [5](#0-4)
