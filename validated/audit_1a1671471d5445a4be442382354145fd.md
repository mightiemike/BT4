### Title
`ValidResourceBounds` Protobuf Deserialization Silently Downgrades `AllResources` to `L1Gas`, Producing a Different Transaction Hash - (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` silently converts an `AllResources` variant (with zero `l2_gas` and `l1_data_gas`) into the `L1Gas` variant. Because `get_tip_resource_bounds_hash` produces a **different Poseidon hash** for these two variants — `L1Gas` hashes 2 resource elements while `AllResources` hashes 3 — a V3 invoke transaction whose resource bounds have zero L2 and L1-data gas will carry a different transaction hash before and after a protobuf round-trip through the sync path. This is the direct sequencer analog of the ETH/WETH address-identity mismatch: two representations of the same economic intent are treated as distinct hash preimages.

---

**Origination path always uses `AllResources`:**

`RpcInvokeTransactionV3` and `InternalRpcInvokeTransactionV3` both store `resource_bounds: AllResourceBounds`. Their `InvokeTransactionV3Trait` implementation wraps it as `ValidResourceBounds::AllResources(self.resource_bounds)`, so the gateway always computes the hash with the 3-element preimage. [3](#0-2) 

**Sync path uses the discriminating converter:**

The sync/storage path deserializes `InvokeTransactionV3` (which holds `ValidResourceBounds`) via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. When `l2_gas == 0` and `l1_data_gas == 0`, the result is `L1Gas`, and the recomputed hash omits the `L1_DATA_GAS` element. [4](#0-3) 

**Concrete divergence:**

| Step | Variant | Hash preimage elements |
|---|---|---|
| Gateway computes hash | `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` | `[tip, L1_GAS, L2_GAS(0), L1_DATA_GAS(0)]` → H₁ |
| Sync node deserializes + recomputes | `L1Gas(X)` | `[tip, L1_GAS, L2_GAS(0)]` → H₂ |

H₁ ≠ H₂ because Poseidon over 4 elements ≠ Poseidon over 3 elements.

---

### Impact Explanation

A V3 invoke transaction submitted with `AllResourceBounds { l1_gas: nonzero, l2_gas: 0, l1_data_gas: 0 }` — a valid and common configuration for pre-0.13.3 style transactions expressed in the new format — will have its hash computed as H₁ by the proposer/gateway and as H₂ by any node that re-derives the hash after protobuf deserialization. This causes:

- **Wrong transaction hash stored or served by syncing nodes** — the hash bound to the executable object diverges from the canonical hash, corrupting receipts, events, and any downstream hash-based lookups.
- **Transaction conversion binds the wrong hash** — the `InternalConsensusTransaction` and `ExecutableTransaction` objects on the validator side carry H₂ while the proposer committed H₁, breaking the hash invariant across the consensus boundary.

This matches: *"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

---

### Likelihood Explanation

Any V3 invoke transaction that sets `l2_gas = 0` and `l1_data_gas = 0` (a valid configuration, explicitly supported by the codebase and present in real-world transactions) triggers this path. No special privileges are required; a normal user submitting such a transaction causes the divergence automatically during sync.

---

### Recommendation

Remove the discriminating downgrade from the protobuf deserializer. When the wire format carries all three resource fields, always produce `AllResources`, regardless of whether `l2_gas` and `l1_data_gas` are zero:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // ... parse l1_gas, l2_gas, l1_data_gas ...
-       Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
-           ValidResourceBounds::L1Gas(l1_gas)
-       } else {
-           ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
-       })
+       Ok(ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }))
    }
}
```

The `L1Gas` variant should only be produced when deserializing transactions that were originally signed as pre-0.13.3 (i.e., when the wire format itself signals the absence of `l1_data_gas`), not inferred from zero values.

### Proof of Concept

1. Submit an `RpcInvokeTransactionV3` with `resource_bounds = AllResourceBounds { l1_gas: { max_amount: 100, max_price_per_unit: 1 }, l2_gas: { 0, 0 }, l1_data_gas: { 0, 0 } }`.
2. The gateway computes hash H₁ via `InternalRpcInvokeTransactionV3::calculate_transaction_hash` → `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash(AllResources(...))` → 4-element Poseidon input.
3. The transaction is included in a block and serialized to protobuf `InvokeV3` for sync.
4. A syncing node deserializes via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`: since `l2_gas.is_zero() && l1_data_gas.is_zero()` → produces `ValidResourceBounds::L1Gas(l1_gas)`.
5. The syncing node recomputes the hash → `get_tip_resource_bounds_hash(L1Gas(...))` → 3-element Poseidon input → H₂ ≠ H₁.
6. The syncing node stores or serves H₂ as the transaction hash, diverging from the canonical H₁ committed by the proposer. [2](#0-1) [5](#0-4) [3](#0-2)

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```
