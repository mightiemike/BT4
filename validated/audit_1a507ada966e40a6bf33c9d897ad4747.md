### Title
Silent Opcode Slot Overwrite via Missing Duplicate-Index Check in `f_lookup_for_hashmap` — (`File: src/f_table.rs`)

---

### Summary

`f_lookup_for_hashmap` in `src/f_table.rs` builds the 256-entry operator dispatch table used by `RuntimeDialect` without checking whether a given opcode slot is already occupied. If a caller-supplied `op_map` contains two distinct operator names that both resolve to the same single-byte opcode index, the second entry silently overwrites the first with no error, no assertion, and no diagnostic. Because Rust's `HashMap` iteration order is non-deterministic (randomised per-process), the "winning" operator varies across runs, producing a non-deterministic dispatch table from a deterministic input.

---

### Finding Description

`f_lookup_for_hashmap` (lines 81–91 of `src/f_table.rs`) iterates over a caller-supplied `HashMap<String, Vec<u8>>` and writes each resolved operator function pointer into a fixed 256-slot array:

```rust
pub fn f_lookup_for_hashmap(opcode_lookup_by_name: HashMap<String, Vec<u8>>) -> FLookup {
    let mut f_lookup = [None; 256];
    for (name, idx) in opcode_lookup_by_name.iter() {
        if idx.len() == 1 {
            let index = idx[0];
            let op = opcode_by_name(name);
            assert!(op.is_some(), "can't find native operator {name}");
            f_lookup[index as usize] = op;   // ← unconditional overwrite, no duplicate check
        }
    }
    f_lookup
}
```

There is no guard equivalent to:
```rust
assert!(f_lookup[index as usize].is_none(), "opcode slot {} already occupied", index);
```

The function is called unconditionally from `RuntimeDialect::new` (line 27 of `src/runtime_dialect.rs`):

```rust
f_lookup: f_lookup_for_hashmap(op_map),
```

`RuntimeDialect` is the dialect exposed to Python callers through the `wheel` crate's FFI layer. The `op_map` argument is a Python-supplied `dict[str, bytes]` that maps operator names to their opcode bytes. A caller who passes two entries sharing the same single-byte opcode value (e.g., `{"op_add": b"\x10", "op_multiply": b"\x10"}`) will silently lose one operator with no indication of the collision.

The analog to the `updatePair` bug is exact:
- **Smart contract**: `updatePair` accepts a `pairName` that already exists for another pair, silently creating a duplicate entry.
- **clvm_rs**: `f_lookup_for_hashmap` accepts an opcode index that already has an operator assigned, silently overwriting it.

---

### Impact Explanation

**Corrupted operator dispatch table.** When a CLVM program is subsequently executed under the affected `RuntimeDialect`, any opcode that was silently overwritten dispatches to the wrong operator function. For example, if `op_add` (opcode `0x10`) is overwritten by `op_multiply`, every `+` expression in the program silently computes a product instead of a sum. The result, cost, and error behaviour all change without any indication.

**Non-deterministic outcome.** Rust's `HashMap` uses a randomised hash seed. The iteration order of `opcode_lookup_by_name.iter()` is therefore non-deterministic across process invocations. Given two conflicting entries, the "winning" operator is chosen randomly at runtime, meaning the same `op_map` input can produce different dispatch tables on different runs. This is a stronger form of the state-inconsistency described in the reference report.

**Consensus / API-equivalence divergence.** If two nodes or two Python processes construct the same `op_map` dict and pass it to `RuntimeDialect`, they may end up with different dispatch tables due to HashMap non-determinism. CLVM programs evaluated under these dialects can return different results, breaking consensus equivalence.

---

### Likelihood Explanation

The `RuntimeDialect` and its `op_map` parameter are directly exposed to Python callers via the `wheel` FFI layer. Any Python application that programmatically constructs an `op_map` (rather than using a hardcoded, manually-verified dict) can inadvertently introduce a duplicate opcode entry. The absence of any runtime check means the error is completely silent — no panic, no log, no wrong-value error — making it very difficult to detect during development or testing. The non-determinism from HashMap ordering further obscures the bug, since it may only manifest on some runs.

---

### Recommendation

Add a duplicate-slot assertion inside `f_lookup_for_hashmap` before writing each entry:

```rust
pub fn f_lookup_for_hashmap(opcode_lookup_by_name: HashMap<String, Vec<u8>>) -> FLookup {
    let mut f_lookup = [None; 256];
    for (name, idx) in opcode_lookup_by_name.iter() {
        if idx.len() == 1 {
            let index = idx[0];
            let op = opcode_by_name(name);
            assert!(op.is_some(), "can't find native operator {name}");
            assert!(
                f_lookup[index as usize].is_none(),
                "opcode slot 0x{:02x} already occupied; duplicate mapping for operator '{name}'",
                index
            );
            f_lookup[index as usize] = op;
        }
    }
    f_lookup
}
```

This mirrors the recommended fix in the reference report (adding an explicit uniqueness check before the write) and eliminates the silent-overwrite and non-determinism issues simultaneously.

### Proof of Concept

```python
from clvm_rs import RuntimeDialect, run_program, MEMPOOL_MODE

# Two distinct operator names both claim opcode 0x10 (normally op_add)
op_map = {
    "op_add":      bytes([0x10]),
    "op_multiply": bytes([0x10]),   # duplicate slot — silently overwrites op_add
}

dialect = RuntimeDialect(op_map, quote_kw=bytes([1]), apply_kw=bytes([2]), flags=0)

# Program: (+ 2 3)  — expected result: 5
# With the corrupted table, opcode 0x10 may dispatch op_multiply → result: 6
# And the winner changes non-deterministically across process restarts.
program = bytes.fromhex("ff10ffff0102ffff010380")  # (+ 2 3)
result = run_program(dialect, program, b"\x80")
# result is either 5 or 6 depending on HashMap iteration order — no error raised
print(result)  # non-deterministic: 5 or 6
```
