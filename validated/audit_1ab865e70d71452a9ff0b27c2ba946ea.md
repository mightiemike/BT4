### Title
Unsigned `oraclePriceX18` in `SignedMintNlp`/`SignedBurnNlp` Allows Arbitrary NLP Valuation â€” (`File: core/contracts/Verifier.sol`, `core/contracts/interfaces/IEndpoint.sol`)

---

### Summary

The `oraclePriceX18` field in `SignedMintNlp` and `SignedBurnNlp` is structurally excluded from the EIP-712 digest that the user signs. The sequencer can inject any arbitrary price value when submitting these transactions, and no on-chain validation bounds or reference oracle check exists. This directly controls how many NLP tokens a user receives when minting, or how much quote they receive when burning.

---

### Finding Description

`SignedMintNlp` and `SignedBurnNlp` are wrapper structs that contain an inner signed payload (`MintNlp`/`BurnNlp`) plus sequencer-appended fields:

```solidity
struct SignedMintNlp {
    MintNlp tx;          // { sender, quoteAmount, nonce }
    bytes signature;
    int128 oraclePriceX18;       // NOT signed by user
    int128[] nlpPoolRebalanceX18;
}
``` [1](#0-0) 

In `Verifier.sol`, the EIP-712 digest for `MintNlp` commits only to `sender`, `quoteAmount`, and `nonce`:

```solidity
digest = keccak256(abi.encode(
    keccak256(bytes(MINT_NLP_SIGNATURE)),
    signedTx.tx.sender,
    signedTx.tx.quoteAmount,
    signedTx.tx.nonce   // oraclePriceX18 is absent
));
``` [2](#0-1) 

The same omission applies to `BurnNlp`: [3](#0-2) 

In `EndpointTx.sol`, the unsigned `oraclePriceX18` is passed directly into `clearinghouse.mintNlp`/`burnNlp` without any bounds or reference check: [4](#0-3) [5](#0-4) 

Inside `Clearinghouse.sol`, this price is the sole determinant of the NLP token amount:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);   // mintNlp
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);   // burnNlp
``` [6](#0-5) [7](#0-6) 

There is no on-chain oracle reference, no deviation bound, and no staleness check applied to `oraclePriceX18` in either path. The only guard in the broader `updatePrice` flow is `require(txn.priceX18 > 0)`, which does not apply here. [8](#0-7) 

---

### Impact Explanation

A user who signs a `MintNlp` transaction commits to `{sender, quoteAmount, nonce}` but makes **no on-chain commitment to the price at which they will receive NLP tokens**. The sequencer fills in `oraclePriceX18` freely at submission time:

- **Inflated price on mint**: user receives fewer NLP tokens for the same `quoteAmount` â€” direct loss of NLP value.
- **Deflated price on burn**: user receives less quote for their NLP tokens â€” direct loss of quote.
- **Inflated price on burn**: user receives more quote, draining the NLP pool's quote reserves.

The corrupted state delta is: `spotEngine` NLP balance for the user and `N_ACCOUNT`, and the quote balance of the user and NLP pool subaccounts â€” all computed from the unchecked price. [9](#0-8) [10](#0-9) 

---

### Likelihood Explanation

Every `MintNlp` and `BurnNlp` transaction submitted by any user is subject to this condition. The sequencer has structural, unconditional control over `oraclePriceX18` with no on-chain constraint. The user's signature provides zero protection against price substitution. Any NLP mint or burn operation is affected.

---

### Recommendation

Include `oraclePriceX18` in the EIP-712 signed digest for both `MintNlp` and `BurnNlp` in `Verifier.sol`, so that the user explicitly commits to the price at which their operation will execute. Additionally, add an on-chain sanity bound (e.g., maximum deviation from the last accepted price) to reject grossly incorrect values even if the signed price is present.

---

### Proof of Concept

1. User signs a `MintNlp` transaction: `{sender: alice, quoteAmount: 1000e18, nonce: 1}`. The signature covers only these three fields.
2. Sequencer constructs `SignedMintNlp` with `oraclePriceX18 = 1` (1 wei in fixed-point), which is not covered by the signature.
3. `validateSignedTx` passes â€” the signature is valid over `{alice, 1000e18, 1}`.
4. `EndpointTx.sol` line 547 sets `priceX18[NLP_PRODUCT_ID] = 1`.
5. `Clearinghouse.mintNlp` computes `nlpAmount = 1000e18 / 1 = 1000e18` NLP tokens instead of the fair amount.
6. Alice receives a massively inflated NLP balance; the `N_ACCOUNT` is debited by the same inflated amount, corrupting the NLP pool accounting. [1](#0-0) [11](#0-10) [12](#0-11) [13](#0-12)
