### Title
Missing `isRedelegating` Guard in `validateAddToPosition` Allows Inflated Bonus Payout During Redelegation Slash — (`x/tieredrewards/keeper/msg_validate.go`)

---

### Summary

`MsgAddToTierPosition` can be submitted while a redelegation is active for the same position. Because `validateAddToPosition` does not check `isRedelegating`, the new shares are silently added to the live delegation. When `BeforeRedelegationSlashed` later fires, `slashRedelegationPosition` reads the live delegation (which now includes the post-add shares) and passes it to `processEventsAndClaimBonus`, which computes bonus on the inflated share count. The result is an overpayment from the bonus pool proportional to the newly added shares.

---

### Finding Description

**Step 1 — Redelegation initiated.**
`TierRedelegate` (msg_server.go:245–254) calls `redelegate`, receives a non-zero `unbondingID`, and stores `setRedelegationMapping(ctx, unbondingID, pos.Id)`. The position's delegator address now holds a delegation to `dstVal` and a pending redelegation entry. [1](#0-0) 

**Step 2 — Missing guard in `validateAddToPosition`.**
`validateAddToPosition` checks owner, `IsDelegated`, `HasTriggeredExit`, and tier close-only status. It does **not** call `isRedelegating`. Compare with `validateRedelegatePosition` (lines 94–99) and `validateExitTierWithDelegation` (lines 239–244), which both explicitly reject calls when a redelegation is active. [2](#0-1) [3](#0-2) [4](#0-3) 

**Step 3 — New shares added to live delegation.**
`AddToTierPosition` calls `delegate(ctx, delAddr, valAddr, msg.Amount)`, which calls `stakingKeeper.Delegate` with no redelegation guard. The position's live delegation on `dstVal` now holds `originalShares + newShares`. [5](#0-4) [6](#0-5) 

**Step 4 — `BeforeRedelegationSlashed` fires.**
When `srcVal` is slashed, the SDK calls `BeforeRedelegationSlashed(ctx, unbondingID, sharesToUnbond)`, which routes to `slashRedelegationPosition`. [7](#0-6) 

**Step 5 — Live delegation read with inflated shares.**
`slashRedelegationPosition` calls `getPositionState(ctx, positionId)`, which calls `getDelegation` to fetch the **live** staking delegation. This returns `originalShares + newShares`. [8](#0-7) [9](#0-8) 

**Step 6 — Bonus computed on inflated shares.**
`processEventsAndClaimBonus` is called with the inflated `pos.Delegation.Shares`. Inside `computeSegmentBonus`, the formula is:

```
tokens = pos.Delegation.Shares × tokensPerShare
bonus  = tokens × tier.BonusApy × durationSeconds / SecondsPerYear
```

The `pos.Delegation.Shares` value includes the newly added shares, so the bonus is overpaid by `newShares × tokensPerShare × tier.BonusApy × duration / SecondsPerYear`. [10](#0-9) [11](#0-10) 

The comment in `slash.go` line 53 explicitly states "Settle bonus against PRE-slash shares," but the code actually settles against POST-add shares. [12](#0-11) 

---

### Impact Explanation

The bonus pool (`types.RewardsPoolName`) is drained by more than the position's redelegated shares warrant. The overpayment per slash event is:

```
overpaid = newShares × tokensPerShare × tier.BonusApy × segmentDuration / SecondsPerYear
```

A position owner can amplify this by adding a large amount just before a slash fires. Repeated across multiple positions or slash events, this constitutes a direct, measurable fund loss from the module's bonus pool.

---

### Likelihood Explanation

The attacker controls both the timing of `MsgAddToTierPosition` and the choice of validator to redelegate to. Slash events on the Cronos POS chain are infrequent but not rare (double-sign, downtime). The window between `MsgTierRedelegate` and `AfterRedelegationCompleted` is the full unbonding period (typically 21 days), giving ample time to call `MsgAddToTierPosition`. No privileged access is required — only ownership of a tier position.

---

### Recommendation

Add an `isRedelegating` guard to `validateAddToPosition`, mirroring the pattern already used in `validateRedelegatePosition` and `validateExitTierWithDelegation`:

```go
func (k Keeper) validateAddToPosition(ctx context.Context, pos types.PositionState, owner string) error {
    // ... existing checks ...

    isRedelegating, err := k.isRedelegating(ctx, pos.DelegatorAddress)
    if err != nil {
        return err
    }
    if isRedelegating {
        return errorsmod.Wrapf(types.ErrActiveRedelegation,
            "position %d has an active redelegation", pos.Id)
    }

    return nil
}
``` [2](#0-1)

### Proof of Concept

```
1. Alice owns position P delegated to valA.
2. Alice calls MsgTierRedelegate(P, valA → valB).
   → unbondingID=42 stored; pos.Delegation now on valB with S shares.
3. Alice calls MsgAddToTierPosition(P, amount=X).
   → validateAddToPosition passes (no isRedelegating check).
   → delegate(delAddr, valB, X) → live delegation = S + newShares.
4. valA is slashed (infraction during redelegation window).
   → BeforeRedelegationSlashed(ctx, 42, sharesToUnbond) fires.
   → slashRedelegationPosition reads live delegation: S + newShares.
   → processEventsAndClaimBonus computes bonus on S + newShares.
5. Alice receives bonus proportional to S + newShares,
   but only S shares were part of the redelegation.
   Overpayment = newShares × tokensPerShare × BonusApy × Δt / SecondsPerYear.
```

func (k Keeper) validateAddToPosition(ctx context.Context, pos types.PositionState, owner string) error {
	if !pos.IsOwner(owner) {
		return types.ErrNotPositionOwner
	}

	if !pos.IsDelegated() {
		return types.ErrPositionNotDelegated

		return types.ErrPositionTriggeredExit

	if err != nil {
		return err
	}

	if tier.IsCloseOnly() {
		return types.ErrTierIsCloseOnly
	}

	return nil
}
