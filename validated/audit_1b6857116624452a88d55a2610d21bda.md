### Title
Undocumented `bytes32`→`bytes20` Truncation in `LinkSigner` Silently Misdirects Stored Linked Signer — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction stores the linked signer address by taking only the **leftmost 20 bytes** of the `bytes32 signer` field. The EIP-712 type string declares `signer` as `bytes32` with no encoding convention documented. An off-chain integrator who encodes the signer address using the standard Solidity right-aligned convention (`bytes32(uint256(uint160(addr)))`) will silently store `address(0)` as the linked signer, making the intended signer permanently unauthorized.

---

### Finding Description

In both the fast-mode and slow-mode `LinkSigner` handlers, the signer is stored as:

```solidity
linkedSigners[signedTx.tx.sender] = address(uint160(bytes20(signedTx.tx.signer)));
``` [1](#0-0) [2](#0-1) 

In Solidity, `bytes20(bytes32Value)` extracts the **leftmost** 20 bytes. The protocol therefore expects the signer address to be **left-aligned** inside the `bytes32` field (i.e., `bytes32(bytes20(addr))`).

However, the EIP-712 type string declares the field as plain `bytes32`:

```solidity
string internal constant LINK_SIGNER_SIGNATURE =
    "LinkSigner(bytes32 sender,bytes32 signer,uint64 nonce)";
``` [3](#0-2) 

And the digest commits to the full `bytes32 signer` value without any truncation:

```solidity
digest = keccak256(
    abi.encode(
        keccak256(bytes(LINK_SIGNER_SIGNATURE)),
        signedTx.tx.sender,
        signedTx.tx.signer,   // full bytes32 committed
        signedTx.tx.nonce
    )
);
``` [4](#0-3) 

The standard Solidity idiom for converting an address to `bytes32` is **right-aligned**: `bytes32(uint256(uint160(addr)))`. An integrator following this convention produces a value whose first 20 bytes are all zeros. Applying `bytes20(...)` to that value yields `bytes20(0)`, so `address(uint160(bytes20(signer)))` resolves to `address(0)`.

The `linkedSigners` mapping is keyed by `bytes32 subaccount`:

```solidity
mapping(bytes32 => address) internal linkedSigners;
``` [5](#0-4) 

A stored value of `address(0)` is treated as "no linked signer" by `validateSignature` and `validateCompactSignature`, because `recovered != address(0)` is always required:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [6](#0-5) 

The transaction succeeds with no revert, but the intended linked signer is never registered.

---

### Impact Explanation

Any user or smart-contract wallet that submits a `LinkSigner` transaction using the standard right-aligned `bytes32` encoding for the signer address will silently store `address(0)`. The intended signer is never authorized. If the user's primary key is unavailable (e.g., a smart-contract wallet that delegates all signing to the linked signer), all subsequent signed operations — withdrawals, order placement, liquidation, NLP mint/burn — will fail signature validation. Funds become inaccessible without the primary key.

---

### Likelihood Explanation

Any off-chain SDK, smart-contract integration, or custom client that follows standard Solidity address-encoding conventions will trigger this silently. The reference implementation presumably uses left-aligned encoding, but this is not stated anywhere in the EIP-712 type string or protocol interface. The `LinkSigner` path is reachable by any unprivileged user via both the slow-mode (`submitSlowModeTransaction`) and fast-mode (`processTransactionImpl`) entry points.

---

### Recommendation

Change the `signer` field in the `LinkSigner` struct from `bytes32` to `address`, and update the EIP-712 type string to `"LinkSigner(bytes32 sender,address signer,uint64 nonce)"`. This eliminates the ambiguity entirely. Alternatively, if `bytes32` must be retained for forward-compatibility, explicitly document and enforce that the signer address must occupy the **leftmost** 20 bytes of the field, and add an on-chain check that the lower 12 bytes are zero.

---

### Proof of Concept

1. User wants to link signer `0xAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA`.
2. User (non-reference client) encodes: `signer = bytes32(uint256(uint160(0xAAAA...AAAA)))` → `0x000000000000000000000000AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA`.
3. User signs the EIP-712 digest (which commits to the full `bytes32`) and submits `LinkSigner`.
4. Contract executes: `address(uint160(bytes20(0x000000000000000000000000AAAA...AAAA)))` = `address(uint160(bytes20(0x00000000000000000000000000000000000000000000000000000000)))` = `address(0)`.
5. `linkedSigners[sender]` is set to `address(0)`.
6. All subsequent calls to `validateSignature(..., allowLinkedSigner=true)` pass `address(0)` as `linkedSigner`; the check `recovered == linkedSigner` can never pass (since `recovered != address(0)` is required).
7. The intended signer `0xAAAA...AAAA` is permanently unauthorized for this subaccount.
