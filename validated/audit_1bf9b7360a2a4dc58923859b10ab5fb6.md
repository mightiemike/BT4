### Title
`SLOW_MODE_FEE` Decimal Scaling Mismatch Renders `DepositInsurance` Minimum Check Ineffective and Corrupts `slowModeFees` Accounting — (`File: core/contracts/common/Constants.sol`)

---

### Summary

`SLOW_MODE_FEE` is hardcoded at `1e6` (6-decimal precision) while the protocol's internal accounting system uses 18-decimal precision (`ONE = 1e18`). The protocol's own `getSlowModeFee()` view function explicitly applies a decimal multiplier to correct for this, but the actual state-changing paths — the `DepositInsurance` minimum check and the `slowModeFees` accumulator — use the raw `1e6` constant without adjustment. For any deployment where the quote token has 18 decimals, the minimum deposit check is effectively zero and the fee accounting ledger is corrupted by a factor of `1e12`.

---

### Finding Description

In `Constants.sol`, two fee constants are both annotated as `$1` but use incompatible decimal bases:

```
SLOW_MODE_FEE  = 1_000_000        // 1e6  — 6-decimal precision
LIQUIDATION_FEE = 1e18            // 1e18 — 18-decimal precision (matches ONE)
HEALTHCHECK_FEE = 1e18            // 1e18 — 18-decimal precision (matches ONE)
``` [1](#0-0) 

The protocol's own `Clearinghouse.getSlowModeFee()` acknowledges that `SLOW_MODE_FEE` is in 6-decimal units and applies a correction multiplier before returning the value to callers:

```solidity
int256 multiplier = int256(10**(token.decimals() - 6));
return uint256(int256(SLOW_MODE_FEE) * multiplier);
``` [2](#0-1) 

However, in `EndpointTx.submitSlowModeTransactionImpl`, the raw `SLOW_MODE_FEE` constant is used in two places without any decimal adjustment:

**1. `DepositInsurance` minimum check:**
```solidity
require(txn.amount >= uint128(SLOW_MODE_FEE), ERR_DEPOSIT_TOO_SMALL);
``` [3](#0-2) 

**2. `slowModeFees` accumulator:**
```solidity
slowModeFees += SLOW_MODE_FEE;
``` [4](#0-3) 

For an 18-decimal quote token, `SLOW_MODE_FEE = 1e6` represents `$0.000000000001` — not `$1` — in the internal 18-decimal accounting system. The `getSlowModeFee()` correction is only applied in the view function, never in the state-changing paths.

---

### Impact Explanation

**Impact 1 — `DepositInsurance` minimum check bypassed:**
Any unprivileged caller can submit a `DepositInsurance` slow-mode transaction with a deposit of `1e6` raw token units (effectively `$0.000000000001` for an 18-decimal token), satisfying the `ERR_DEPOSIT_TOO_SMALL` guard. The intended anti-spam minimum of `$1` is rendered ineffective, allowing the insurance fund to be polluted with negligible deposits at near-zero cost.

**Impact 2 — `slowModeFees` accounting corruption:**
Each slow-mode transaction increments `slowModeFees` by `1e6` regardless of the actual fee collected (which, via `chargeSlowModeFee`, is scaled to the token's decimals). For an 18-decimal token, the ledger underreports collected fees by a factor of `1e12`. Any downstream logic that reads `slowModeFees` to determine how much to distribute or verify (e.g., `DumpFees`) will operate on a value `1e12` times smaller than the true collected amount, causing permanent fee accounting desynchronization.

---

### Likelihood Explanation

**Medium.** The protocol is deployed on Ink Chain, an EVM-compatible L2 where 18-decimal ERC20 tokens (WETH, wrapped assets) are standard. The existence of `getSlowModeFee()`'s decimal multiplier confirms the protocol explicitly supports non-6-decimal quote tokens. Any deployment using an 18-decimal quote token triggers both issues immediately upon the first slow-mode transaction. An unprivileged user is the only required actor.

---

### Recommendation

Replace the raw `SLOW_MODE_FEE` constant in both state-changing paths with the decimal-adjusted value from `getSlowModeFee()` (or an equivalent internal helper). Specifically:

1. In `DepositInsurance` minimum check, compare `txn.amount` against the adjusted fee (accounting for `token.decimals()`), not the raw `SLOW_MODE_FEE`.
2. In the `slowModeFees` accumulator, add the decimal-adjusted amount, not the raw `1e6` constant.
3. Align `SLOW_MODE_FEE` to 18-decimal precision (i.e., `1e18`) to match `LIQUIDATION_FEE` and `HEALTHCHECK_FEE`, and remove the ad-hoc multiplier in `getSlowModeFee()`.

---

### Proof of Concept

**Setup:** Deploy Nado with an 18-decimal ERC20 as the quote token (e.g., WETH).

**Step 1:** Attacker calls `Endpoint.submitSlowModeTransaction` with a `DepositInsurance` payload where `txn.amount = 1_000_000` (i.e., `1e6` raw units = `$0.000000000001`).

**Step 2:** The check at `EndpointTx.sol:351` evaluates `1_000_000 >= uint128(SLOW_MODE_FEE)` → `1_000_000 >= 1_000_000` → **passes**, despite the deposit being `$0.000000000001`.

**Step 3:** `slowModeFees += SLOW_MODE_FEE` increments the ledger by `1e6`. The actual ERC20 transfer (via `chargeSlowModeFee`) collects `1e6 * 10^(18-6) = 1e18` raw units from the user. The ledger now records `1e6` but `1e18` was collected — a `1e12` discrepancy per transaction.

**Step 4:** After `N` slow-mode transactions, `slowModeFees = N * 1e6` while actual collected fees = `N * 1e18`.
