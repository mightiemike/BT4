Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks Caller-Supplied `owner` Instead of Actual Depositor, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` receives the actual depositor as its first (unnamed, discarded) parameter and the caller-supplied LP-position `owner` as its second parameter, but gates access using only `owner`. Because `owner` is a free argument any caller can set to any address, the allowlist is completely bypassable: any unprivileged address can deposit into a restricted pool by supplying any already-allowed address as `owner`.

## Finding Description
`ExtensionCalling._beforeAddLiquidity` encodes the hook call as `(sender, owner, salt, deltas, extensionData)`, where `sender` is the actual `msg.sender` of `addLiquidity` and `owner` is the caller-supplied LP-position recipient. [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the first parameter and checks only `owner`: [2](#0-1) 

Since `owner` is freely chosen by the caller, any address can pass the guard by supplying any address already on the allowlist. The actual depositor — the address that implements the callback and transfers tokens — is never verified.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks the first parameter (`sender`) and ignores the second (`recipient`): [3](#0-2) 

The asymmetry confirms the deposit check is bound to the wrong identity.

## Impact Explanation
The `DepositAllowlistExtension` allowlist is rendered completely inoperative. Any unprivileged address can add liquidity to a pool the admin intended to restrict, by setting `owner` to any address already on the allowlist. This constitutes an admin-boundary break: the pool admin's access-control invariant ("only approved depositors may add liquidity") is silently violated on every call from an unauthorized depositor. Additionally, LP shares are minted to the `owner` address (an allowed party who did not provide tokens), while the actual token provider holds no position and cannot recover funds — a direct loss of user principal. [4](#0-3) 

## Likelihood Explanation
Exploitation requires only a single `addLiquidity` call with `owner` set to any known-allowed address (the pool admin is always a candidate, as they are typically allowlisted). No special permissions, flash loans, or oracle manipulation are needed. The bypass is deterministic and repeatable by any unprivileged address.

## Recommendation
Check the actual depositor (first parameter, `sender`) instead of `owner`:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external view override returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

This mirrors the correct pattern already used in `SwapAllowlistExtension.beforeSwap`. [2](#0-1) 

## Proof of Concept
1. Deploy a pool with `DepositAllowlistExtension` configured; set `allowedDepositor[pool][alice] = true`; `allowedDepositor[pool][attacker] = false`.
2. `attacker` calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
3. Pool fires `_beforeAddLiquidity(attacker, alice, ...)` via `ExtensionCalling`.
4. Extension evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
5. `LiquidityLib.addLiquidity` calls back to `attacker` (the actual `msg.sender`) to pull tokens; `attacker` pays.
6. LP shares are minted to `alice`; `attacker` has deposited into a restricted pool without being on the allowlist. `attacker`'s tokens are locked under `alice`'s position — `attacker` holds no shares and cannot withdraw. [5](#0-4)
