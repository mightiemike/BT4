### Title
Silent Cost Undercharge via u64 Overflow in `op_unknown` Cost Multiplication — (File: `src/more_ops.rs`)

---

### Summary

In `op_unknown`, the final cost is computed by multiplying a base cost by `(cost_multiplier + 1)`. Both operands are `u64`. In Rust release builds, integer overflow wraps silently. The post-multiplication guard `if cost > u32::MAX as u64` cannot detect a wrapped result that lands at or below `u32::MAX`, so the function returns a cost that is orders of magnitude smaller than the intended cost — including zero — without signaling any error. This is the direct analog of `safeConcurTransfer` silently delivering fewer tokens than requested.

---

### Finding Description

`op_unknown` in `src/more_ops.rs` handles unknown opcodes in lenient/consensus mode. It computes a base cost from the opcode's cost-function bits and arguments, then scales it by a multiplier encoded in the opcode bytes:

```rust
// src/more_ops.rs lines 258–266
assert!(cost > 0);

check_cost(cost, max_cost)?;          // ← checks PRE-multiplication cost only
cost *= cost_multiplier + 1;          // ← u64 × u64: wraps silently in release mode
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`cost_multiplier` is decoded as a `u64` from up to four bytes of the opcode, so its maximum value is `u32::MAX = 4_294_967_295`. The pre-multiplication `check_cost` only verifies that the base cost does not exceed `max_cost`; it says nothing about the product. When the product overflows `u64`, Rust release mode wraps it modulo 2⁶⁴. If the wrapped value is ≤ `u32::MAX`, the guard on line 262 passes and the function returns the wrapped (incorrect, far-too-small) cost as a successful `Reduction`.

**Concrete trigger (cost_function = 2, MUL-like path):**

```rust
// src/more_ops.rs lines 223–242
2 => {
    let mut cost = MUL_BASE_COST;          // 92
    let mut first_iter: bool = true;
    let mut l0: u64 = 0;
    while let Some((arg, rest)) = allocator.next(args) {
        ...
        cost += MUL_COST_PER_OP;           // 885
        cost += (l0 + l1) * MUL_LINEAR_COST_PER_BYTE;   // ×6
        cost += (l0 * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER; // ÷128
        l0 += l1;
        check_cost(cost, max_cost)?;
    }
    cost
}
```

With two arguments of approximately 740 685 bytes each (~723 KB each, well within allocator limits), the quadratic term drives the base cost to exactly 2³² = 4 294 967 296. Setting `cost_multiplier = u32::MAX` (opcode prefix bytes = `0xff ff ff ff`) makes `cost_multiplier + 1 = 2³²`. The multiplication is then:

```
2³² × 2³² = 2⁶⁴  ≡  0  (mod 2⁶⁴)
```

The wrapped result is 0. The guard `0 > u32::MAX` is false, so the function returns `Reduction(0, nil)` — zero cost — for an operation whose intended cost is 2⁶⁴ ≈ 1.8 × 10¹⁹.

The base cost 2³² ≈ 4.3 × 10⁹ is below the standard Chia max-cost of 11 × 10⁹, so `check_cost` on line 260 passes without error.

---

### Impact Explanation

An attacker submitting a crafted CLVM program in consensus mode (`allow_unknown_ops() = true`) can execute an unknown opcode that should consume billions of cost units while the VM accounts for 0 (or any small wrapped value). This breaks the cost-limit invariant that is the primary DoS protection of the CLVM. Nodes running release builds silently accept the program; nodes running debug builds panic on the overflow. This is a **consensus divergence** and an **undercharged execution** vulnerability simultaneously.

---

### Likelihood Explanation

The attack requires only ~1.5 MB of argument data (two ~740 KB atoms) embedded in a CLVM program, which is well within serialization and allocator limits. The opcode bytes are fully attacker-controlled. The lenient/consensus dialect (`ChiaDialect` with `allow_unknown_ops = true`) is the standard path for block validation. No privileged access is required; any submitted transaction can carry the payload.

---

### Recommendation

Replace the unchecked multiplication with a checked variant that returns an error on overflow:

```rust
// src/more_ops.rs — replace lines 261–265
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

This ensures that any multiplication that would overflow is treated as an invalid opcode rather than silently producing a wrapped, under-charged cost.

### Proof of Concept

**Opcode construction:**
- Last byte: `0b10_000000` → cost_function = 2 (MUL-like), lower 6 bits = 0
- Preceding 4 bytes: `0xff 0xff 0xff 0xff` → cost_multiplier = u32::MAX = 4 294 967 295
- Full opcode atom: `[0xff, 0xff, 0xff, 0xff, 0x80]`

**Arguments:** Two atoms each of length S ≈ 740 685 bytes (content irrelevant).

**Cost trace (release build):**
```
base cost (cost_function=2, two args of size S):
  = MUL_BASE_COST + MUL_COST_PER_OP + (S+S)*6 + (S*S)/128
  ≈ 92 + 885 + 8 888 220 + 4 285 078 099
  ≈ 2^32  (tunable to exact value by adjusting S)

check_cost(2^32, 11_000_000_000) → Ok  (2^32 < 11e9)

cost *= cost_multiplier + 1
  = 2^32 * 2^32 = 2^64
  wraps to 0 in u64 (release mode)

if 0 > u32::MAX  →  false
→ Ok(Reduction(0, nil))   // zero cost returned
```

**Consensus impact:** A full-node running a release build accepts this program with cost 0. A node running a debug build panics. Nodes enforcing the cost limit diverge. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
