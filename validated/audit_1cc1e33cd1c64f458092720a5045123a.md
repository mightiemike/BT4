### Title
Stale Linked Signer Persists After Subaccount Finalization, Enabling Unauthorized Fee Drain via MintNlp/BurnNlp — (`File: core/contracts/EndpointTx.sol`, `core/contracts/ClearinghouseLiq.sol`)

---

### Summary

The `linkedSigners` mapping in `EndpointStorage` is set once via a `LinkSigner` transaction and is **never cleared** when a subaccount is fully liquidated and finalized. After finalization zeroes all balances, if the original owner re-deposits to the same subaccount, the stale linked signer retains signing authority over `MintNlp`, `BurnNlp`, `WithdrawCollateral`, and `LiquidateSubaccount` transactions. The most concrete exploitable path is repeated unauthorized `MintNlp`→`BurnNlp` cycles that drain the re-funded subaccount via the 0.1% burn fee.

---

### Finding Description

`linkedSigners` is declared as a persistent mapping in `EndpointStorage`: [1](#0-0) 

It is written by the `LinkSigner` handler in `EndpointTx.processTransactionImpl`: [2](#0-1) 

And also via the slow-mode path: [3](#0-2) 

When a subaccount is fully liquidated, `_finalizeSubaccount` in `ClearinghouseLiq` zeroes all spot and perp balances, socializes the subaccount, and returns. It **never touches `linkedSigners`**: [4](#0-3) 

After finalization, `subaccountIds[subaccount]` remains non-zero (the record is never deleted): [5](#0-4) 

So when the owner re-deposits to the same `bytes32` subaccount key, `_recordSubaccount` is a no-op and the old linked signer is still active. `getLinkedSigner` will return the stale address: [6](#0-5) 

The `MintNlp` and `BurnNlp` transaction handlers both pass `allowLinkedSigner = true`: [7](#0-6) 

The `BurnNlp` path in `Clearinghouse` charges a burn fee of `max(ONE, quoteAmount / 1000)` (≥ 0.1%) on every burn: [8](#0-7) 

**Scope of the stale signer's authority (post-finalization re-fund):**

| Transaction | `allowLinkedSigner` | Attacker outcome |
|---|---|---|
| `MintNlp` | `true` | Converts victim USDC → NLP |
| `BurnNlp` | `true` | Converts NLP → USDC minus ≥0.1% burn fee |
| `WithdrawCollateral` / `WithdrawCollateralV2` (`sendTo==0`) | `true` | Forced withdrawal to victim's own address (griefing, not theft) |
| `WithdrawCollateralV2` (`sendTo!=0`) | `false` | Blocked |
| `TransferQuote` | `true` (in `EndpointTx`) | **Blocked** by `Clearinghouse.transferQuote` which enforces `bytes20(sender)==bytes20(recipient)` |
| `LiquidateSubaccount` | `true` | Victim pays `LIQUIDATION_FEE` per call | [9](#0-8) 

The `TransferQuote` cross-address check and the `sendTo != address(0)` restriction on `WithdrawCollateralV2` are the mitigations that prevent direct fund theft. The residual exploitable surface is the MintNlp/BurnNlp fee drain and repeated liquidation fee payments.

---

### Impact Explanation

A compromised or malicious linked signer that was set before a subaccount was liquidated retains the ability to:

1. **Drain funds via burn fees**: Repeatedly call `MintNlp` (victim USDC → NLP) then `BurnNlp` (NLP → USDC minus ≥0.1% fee). Each round-trip costs the victim at least `max(1e18, quoteAmount / 1000)` in quote units. Over many cycles this constitutes a meaningful asset loss proportional to the re-deposited balance.
2. **Pay liquidation fees on behalf of the victim**: Sign `LiquidateSubaccount` transactions where the victim's subaccount is the `sender`, causing it to pay `LIQUIDATION_FEE` per call.
3. **Force collateral withdrawals**: Drain collateral to the victim's own wallet address (up to the health threshold), disrupting open positions or trading strategies.

The corrupted state delta is the victim's `QUOTE_PRODUCT_ID` balance in `SpotEngine`, reduced by accumulated burn fees and liquidation fees.

---

### Likelihood Explanation

**Medium.** The preconditions are:
- The victim previously linked a signer (common for API/bot traders).
- The victim's subaccount was fully liquidated and finalized.
- The victim re-deposits to the same subaccount name (the natural behavior — users reuse their subaccount name).
- The linked signer is compromised or was set to a malicious address.

The third condition (re-deposit to same subaccount) is the default user behavior since subaccount names are user-chosen strings embedded in the `bytes32` key. The victim has no on-chain signal that their linked signer was not cleared.

---

### Recommendation

In `_finalizeSubaccount` within `ClearinghouseLiq.sol`, after all balances are zeroed and before returning `true`, emit a call back to the `Endpoint` to clear `linkedSigners[txn.liquidatee]`. Alternatively, add a dedicated `clearLinkedSigner` path that is invoked as part of the finalization flow in `liquidateSubaccountImpl`. The fix must be applied in the `Endpoint`/`EndpointStorage` storage context (since `ClearinghouseLiq` runs via `delegatecall` from `Clearinghouse`, it does not share `Endpoint` storage directly — the clearing must be routed through the `Endpoint` interface).

---

### Proof of Concept

1. Alice creates subaccount `alice||"default"` and links signer `mallory_key` via `LinkSigner`. `linkedSigners[alice_subaccount] = mallory_key`.
2. Alice's subaccount is liquidated and `_finalizeSubaccount` is called. All balances are zeroed. `linkedSigners[alice_subaccount]` is **not cleared**.
3. Alice re-deposits 10,000 USDC to `alice||"default"`. `_recordSubaccount` is a no-op (ID already exists). `linkedSigners[alice_subaccount]` still equals `mallory_key`.
4. Mallory signs a `MintNlp` transaction: `sender = alice_subaccount`, `quoteAmount = 10000e18`. Signature is valid — `getLinkedSigner(alice_subaccount)` returns `mallory_key`. Alice's 10,000 USDC is converted to NLP.
5. Mallory signs a `BurnNlp` transaction: `sender = alice_subaccount`, `nlpAmount = full NLP balance`. Burn fee = `max(1e18, 10000e18 / 1000)` = 10 USDC. Alice receives 9,990 USDC back.
6. Repeat steps 4–5. Each cycle costs Alice ≥10 USDC. After 100 cycles, Alice has lost ≥1,000 USDC with no recourse until she explicitly issues a new `LinkSigner` to overwrite `mallory_key`. [10](#0-9) [6](#0-5) [11](#0-10)

    function _finalizeSubaccount(
        IEndpoint.LiquidateSubaccount calldata txn,
        ISpotEngine spotEngine,
        IPerpEngine perpEngine
    ) internal returns (bool) {
        if (txn.productId != type(uint32).max) {
            return false;
        }
        // check whether the subaccount can be finalized:
        // - all perps positions have closed
        // - all spread positions have closed
        // - all spot assets have closed
        // - all positive pnls have been settled

        FinalizeVars memory v;

        v.spotIds = spotEngine.getProductIds();
        v.perpIds = perpEngine.getProductIds();

        require(v.spotIds[0] == QUOTE_PRODUCT_ID);

        // all spot assets (except USDC) must be closed out
        for (uint32 i = 1; i < v.spotIds.length; ++i) {
            uint32 spotId = v.spotIds[i];
            if (spotEngine.getRisk(spotId).longWeightInitialX18 == 0) {
                continue;
            }
            ISpotEngine.Balance memory balance = spotEngine.getBalance(
                spotId,
                txn.liquidatee
            );
            require(balance.amount <= 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
        }

        for (uint32 i = 0; i < v.perpIds.length; ++i) {
            uint32 perpId = v.perpIds[i];
            IPerpEngine.Balance memory balance = perpEngine.getBalance(
                perpId,
                txn.liquidatee
            );
            require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
        }

        // settle all positive pnl
        for (uint32 i = 0; i < v.perpIds.length; ++i) {
            uint32 perpId = v.perpIds[i];
            IPerpEngine.Balance memory balance = perpEngine.getBalance(
                perpId,
                txn.liquidatee
            );
            if (balance.vQuoteBalance > 0) {
                _settlePnlAgainstLiquidator(
                    txn,
                    perpId,
                    balance.vQuoteBalance,
                    spotEngine,
                    perpEngine
                );
            }
        }

        ISpotEngine.Balance memory quoteBalance = spotEngine.getBalance(
            QUOTE_PRODUCT_ID,
            txn.liquidatee
        );

        // settle all negative pnl until quote balance becomes 0
        for (uint32 i = 0; i < v.perpIds.length; ++i) {
            uint32 perpId = v.perpIds[i];
            IPerpEngine.Balance memory balance = perpEngine.getBalance(
                perpId,
                txn.liquidatee
            );
            if (balance.vQuoteBalance < 0 && quoteBalance.amount > 0) {
                int128 canSettle = MathHelper.max(
                    balance.vQuoteBalance,
                    -quoteBalance.amount
                );
                _settlePnlAgainstLiquidator(
                    txn,
                    perpId,
                    canSettle,
                    spotEngine,
                    perpEngine
                );
                quoteBalance.amount += canSettle;
            }
        }

        v.insurance = insurance;
        v.insurance -= lastLiquidationFees;
        v.canLiquidateMore = (quoteBalance.amount + v.insurance) > 0;

        if (v.canLiquidateMore) {
            for (uint32 i = 1; i < v.spotIds.length; ++i) {
                uint32 spotId = v.spotIds[i];
                ISpotEngine.Balance memory balance = spotEngine.getBalance(
                    spotId,
                    txn.liquidatee
                );
                if (spotEngine.getRisk(spotId).longWeightInitialX18 == 0) {
                    continue;
                }
                require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
            }
        }

        v.insurance = perpEngine.socializeSubaccount(
            txn.liquidatee,
            v.insurance
        );

        // we can assure that quoteBalance must be non positive, because if quoteBalance.amount > 0,
        // there must be 1) no negative pnl in perps, and 2) no liabilities in spot after above actions.
        // however, in this case the liquidatee must be healthy and cannot pass the health check at
        // the beginning.
        int128 insuranceCover = MathHelper.min(
            v.insurance,
            -quoteBalance.amount
        );
        if (insuranceCover > 0) {
            v.insurance -= insuranceCover;
            spotEngine.updateBalance(
                QUOTE_PRODUCT_ID,
                txn.liquidatee,
                insuranceCover
            );
        }
        if (v.insurance <= 0) {
            spotEngine.socializeSubaccount(txn.liquidatee);
        }
        v.insurance += lastLiquidationFees;
        insurance = v.insurance;
        return true;
    }
