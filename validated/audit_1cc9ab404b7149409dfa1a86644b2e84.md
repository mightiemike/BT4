### Title
Unconditional trust of attacker-supplied `_cached_sha256_treehash` attribute enables forged puzzle-hash injection — (File: `wheel/python/clvm_rs/tree_hash.py`)

---

### Summary

`sha256_treehash` in `tree_hash.py` reads `_cached_sha256_treehash` from any Python object and returns it immediately without verifying it matches the actual tree content. An attacker-controlled Python object with a pre-set forged `_cached_sha256_treehash` causes the function to return the forged hash, corrupting puzzle-hash computation throughout the Python API. This is the direct analog of the ERC20Permit domain-separator bug: a cached value is fixed at object-creation time and trusted unconditionally thereafter, with no revalidation against the live data.

---

### Finding Description

**Root cause — `tree_hash.py` lines 57–61:**

```python
r = getattr(obj, "_cached_sha256_treehash", None)
if r is not None:
    self.cache_hits += 1
    hash_stack.append(r)
    return
``` [1](#0-0) 

`sha256_treehash` short-circuits the entire hash computation the moment it finds a non-`None` `_cached_sha256_treehash` attribute on the input object. There is no check that the cached bytes are 32 bytes long, are a valid SHA-256 digest, or correspond to the actual atom/pair content of the node. Any Python object that satisfies the `CLVMStorage` duck-type interface (`atom` / `pair` attributes) and carries a forged `_cached_sha256_treehash` will have that forged value returned verbatim.

**Propagation — `program.py` line 142:**

`Program.wrap()` copies `_cached_sha256_treehash` from any `CLVMStorage`-compatible object unconditionally:

```python
o._cached_sha256_treehash = getattr(v, "_cached_sha256_treehash", None)
``` [2](#0-1) 

So a forged hash on an input object survives wrapping into a `Program`.

**Consumption — `program.py` lines 284–286:**

`Program.tree_hash()` returns the cached value directly without recomputing:

```python
if self._cached_sha256_treehash is None:
    self._cached_sha256_treehash = sha256_treehash(self._unwrapped)
return self._cached_sha256_treehash
``` [3](#0-2) 

**Equality comparison — `program.py` line 188:**

`Program.__eq__` is entirely hash-based:

```python
return self.tree_hash() == other_obj.tree_hash()
``` [4](#0-3) 

A forged hash therefore controls equality semantics for the entire `Program` type.

---

### Impact Explanation

In the Chia blockchain, the SHA-256 tree hash of a puzzle program is the **coin puzzle hash** — the on-chain identifier that locks a coin. Wallet software and validators use `Program.tree_hash()` (or `sha256_treehash`) to:

1. Derive the puzzle hash to create a coin.
2. Verify that a spend bundle's puzzle matches the locked coin.
3. Compare programs for deduplication and caching.

A forged `_cached_sha256_treehash` on an attacker-supplied `CLVMStorage` object causes all three operations to use the wrong hash. Concretely:

- A wallet computing `puzzle.tree_hash()` on a crafted object returns the attacker's chosen hash, causing coins to be locked to the wrong puzzle hash.
- `Program.__eq__` returns `True` for structurally different programs that share a forged hash, or `False` for identical programs with different forged hashes — breaking deduplication and validation logic.
- Any downstream system that caches or indexes by `tree_hash()` output inherits the corruption silently.

The corrupted value is a `bytes` object of arbitrary content — not necessarily 32 bytes — so downstream code that assumes a 32-byte digest may also misbehave.

---

### Likelihood Explanation

The `Program.to()` / `Program.wrap()` API is explicitly designed to accept arbitrary Python objects implementing the `CLVMStorage` duck-type interface. Library consumers routinely pass Python-side CLVM objects (e.g., from `clvm` or custom implementations) into these functions. Any such object can carry a pre-set `_cached_sha256_treehash` attribute — either maliciously or accidentally (e.g., a stale cache from a prior computation on a different tree). The attribute is a plain Python instance attribute with no access control. No special privileges, no compromised nodes, and no social engineering are required; only the ability to pass a Python object to `sha256_treehash`, `Program.to()`, or `Program.wrap()`.

---

### Recommendation

**Short term:** In `sha256_treehash` (`tree_hash.py`), validate the cached value before trusting it — at minimum check `isinstance(r, bytes) and len(r) == 32`. Ideally, remove the cache-read shortcut from `sha256_treehash` entirely and let caching be the caller's responsibility with explicit, validated storage.

**Long term:** For all security-critical puzzle-hash computations (coin creation, spend validation), use the Rust-side tree hash path (`clvmr`'s `treehash` via `ObjectCache`) rather than the Python-side `sha256_treehash`, since the Rust path operates on the `Allocator`'s internal node representation and is not susceptible to attacker-controlled Python attributes.

### Proof of Concept

```python
from clvm_rs.program import Program

# Craft a CLVMStorage-compatible object with a forged tree hash
class ForgeryObject:
    atom = b"\x01"          # atom: the byte 0x01
    pair = None
    _cached_sha256_treehash = b"\xff" * 32   # forged: all 0xff

forged = ForgeryObject()

# Wrap into a Program — the forged hash is copied verbatim
p = Program.wrap(forged)

# tree_hash() returns the forged value, not sha256(b"\x01" || b"\x01")
assert p.tree_hash() == b"\xff" * 32   # passes — hash is wrong

# Legitimate atom 0x01 has a different correct hash
correct = Program.new_atom(b"\x01")
correct_hash = correct.tree_hash()
assert correct_hash != b"\xff" * 32

# __eq__ now treats the forged program as unequal to the identical atom
assert p != correct   # passes — same content, different forged hash
```

The forged hash flows from `ForgeryObject._cached_sha256_treehash` → `Program.wrap()` line 142 → `Program.tree_hash()` line 286 → `Program.__eq__()` line 188, with no recomputation or validation at any step.
