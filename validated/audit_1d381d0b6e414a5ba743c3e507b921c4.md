### Title
Raw `IERC20Base.transferFrom()` Used Instead of `safeTransferFrom()` in Token Migration — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` calls `IERC20Base(usdc).transferFrom(...)` directly on the interface instead of using the project's own `ERC20Helper.safeTransferFrom()` wrapper. For any token that does not return a `bool` from `transferFrom` (e.g., USDT on mainnet), Solidity 0.8.x will attempt to ABI-decode the empty return data as `bool` and revert. The rest of the codebase consistently uses `ERC20Helper` to avoid exactly this class of failure; this one call site is the sole exception.

---

### Finding Description

`ContractOwner` declares `using ERC20Helper for IERC20Base` at the top of the contract, making `safeTransfer` and `safeTransferFrom` available on every `IERC20Base` instance. [1](#0-0) 

`ERC20Helper.safeTransferFrom` uses a low-level `.call()` and explicitly accepts both a zero-length return and a `true` bool, tolerating non-standard ERC20 tokens: [2](#0-1) 

Inside `replaceUsdcEWithUsdc`, the outbound leg of the swap calls the raw interface method:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
``` [3](#0-2) 

`IERC20Base.transferFrom` is declared as returning `bool`: [4](#0-3) 

When Solidity 0.8.x calls a typed interface function that declares a `bool` return and the callee returns zero bytes, the built-in ABI decoder reverts. `ERC20Helper.safeTransferFrom` avoids this by checking `data.length == 0 || abi.decode(data, (bool))`. The raw call does not.

Notably, the very next line in the same `if`-block correctly uses `safeTransfer`:

```solidity
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
``` [5](#0-4) 

This makes the inconsistency clear: the inbound leg is safe, the outbound leg is not.

---

### Impact Explanation

`replaceUsdcEWithUsdc` is the only on-chain path for migrating USDC-E balances held in a user's `DirectDepositV1` address to native USDC on Ink chain (chain ID 57073). If the USDC token at the hardcoded address does not return a `bool` from `transferFrom`, every call to this function reverts unconditionally. Users with USDC-E in their DDA cannot complete the migration; their USDC-E remains locked in the DDA with no alternative on-chain exit path through this contract. [6](#0-5) 

---

### Likelihood Explanation

The function carries no access-control modifier — any user on Ink chain (chain ID 57073) can call it. [7](#0-6) 

The token address is hardcoded, so the caller cannot substitute a different token. Whether the specific USDC deployment at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink chain returns `bool` is deployment-dependent. Circle's bridged USDC variants have historically omitted the return value on some chains. Given that the protocol already ships `ERC20Helper` precisely to guard against this class of failure, and given that the identical pattern (raw `transfer`/`transferFrom` on a non-standard token) was the root cause of the referenced Amphor finding, the risk is concrete and not merely theoretical.

---

### Recommendation

Replace the raw interface call with the `ERC20Helper`-backed safe variant, which is already in scope via the `using` directive:

```diff
- IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
+ IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [3](#0-2) 

`ERC20Helper.safeTransferFrom` accepts both zero-length returns and explicit `true` returns, matching the behaviour already applied to every other token transfer in the codebase. [8](#0-7) 

---

### Proof of Concept

1. Deploy or interact with `ContractOwner` on Ink chain (chain ID 57073).
2. Ensure a `DirectDepositV1` exists for a `subaccount` and holds a non-zero USDC-E balance.
3. Call `replaceUsdcEWithUsdc(subaccount)` with sufficient USDC allowance granted to `ContractOwner`.
4. The call reaches line 616: `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)`.
5. If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` returns zero bytes from `transferFrom`, Solidity's ABI decoder reverts with a decoding error.
6. The migration is permanently blocked; USDC-E remains in the DDA with no alternative on-chain exit. [6](#0-5)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
