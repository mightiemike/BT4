Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual User, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the router contract, not the originating user. If the pool admin allowlists the router (required for router-mediated swaps to function), every user on the network can bypass the swap allowlist by calling the router, completely defeating the access-control invariant the extension was deployed to enforce.

## Finding Description
The root cause is a mismatch between the address the pool passes as `sender` and the address the extension intends to gate.

**Step 1 — Pool passes `msg.sender` (router) as `sender` to the extension:** [1](#0-0) 

**Step 2 — Extension checks `allowedSwapper[pool][sender]`, where `sender` is the router:** [2](#0-1) 

**Step 3 — Router calls `pool.swap()` without encoding the original `msg.sender` into `extensionData`:** [3](#0-2) 

The router stores the original `msg.sender` only in transient storage for the payment callback (`_setNextCallbackContext`), not in `extensionData`. The extension therefore never sees the actual user's address. The pool admin faces an impossible choice: if the router is not allowlisted, router-mediated swaps revert for everyone including allowlisted users; if the router is allowlisted, any unprivileged user can bypass the allowlist by calling the router.

## Impact Explanation
This constitutes broken core pool functionality with direct fund-impact consequences. LPs who deposited into a pool configured with `SwapAllowlistExtension` under the assumption that only KYC'd or permissioned counterparties could trade against them are exposed to unrestricted swap flow from any EOA. The access-control invariant the extension was deployed to enforce is completely bypassed. This falls under "Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows" and "Admin-boundary break: pool admin exceeds caps, bypasses timelocks, or factory/oracle role checks are bypassed by an unprivileged path."

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool using `SwapAllowlistExtension` that also wants to support router-mediated swaps must allowlist the router — this is the expected operational configuration. The bypass requires no special privileges, no flash loans, and no unusual token behavior. Any EOA can call `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` on the public router to execute swaps against the restricted pool.

## Recommendation
The extension must check the original user rather than the immediate caller of `pool.swap()`. The cleanest fix is to have the router encode the original `msg.sender` into `extensionData`, and have `SwapAllowlistExtension.beforeSwap` decode and verify it. The `onlyPool` guard (via `msg.sender` being the pool) ensures the pool — not an attacker — is the caller of `beforeSwap`, so the decoded address can be trusted as long as the router is the one encoding it. Alternatively, document that `SwapAllowlistExtension` is incompatible with `MetricOmmSimpleRouter` unless the router is allowlisted (which defeats the purpose), and provide a router variant that forwards the original `msg.sender` as a verified parameter in `extensionData`.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - allowedSwapper[pool][alice] = true   (alice is the only allowed swapper)
  - allowedSwapper[pool][router] = true  (router allowlisted so alice can use it)
  - Add liquidity to the pool

Attack:
  - bob (not allowlisted) calls router.exactInputSingle({pool: pool, ...})
  - Router calls pool.swap(recipient=bob, ...) — msg.sender to pool = router
  - Pool calls _beforeSwap(sender=router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  - Swap executes successfully for bob despite bob not being allowlisted

Result:
  - bob swaps against a pool intended to be restricted to alice only
  - The swap allowlist guard is completely bypassed
