### Title
`DeleteTier` Does Not Verify Active Positions Are Closed Before Deleting Tier - (File: `x/tieredrewards/keeper/msg_server_auth.go`)

---

### Summary

The `MsgDeleteTier` governance message permanently removes a tier from state without checking whether active positions still reference that tier. Any position whose exit has not yet been triggered becomes permanently unrecoverable: `TriggerExitFromTier` calls `getTier` to read `ExitDuration`, which returns `ErrTierNotFound` after deletion, blocking the entire exit path and leaving user funds stuck in the module.

---

### Finding Description

`DeleteTier` in `msg_server_auth.go` performs only an authority check and a tier-existence check before calling `deleteTier`: [1](#0-0) 

There is no guard that verifies the tier has zero active positions before deletion.

After the tier is removed from state, `TriggerExitFromTier` fails for any position that has not yet triggered its exit: [2](#0-1) 

`getTier` returns `ErrTierNotFound` for the deleted tier ID, so the call to `pos.TriggerExit(...)` is never reached. Without a triggered exit, `validateUndelegatePosition` blocks `TierUndelegate`, and `validateWithdrawFromTier` blocks `WithdrawFromTier` via `ErrExitNotTriggered`: [3](#0-2) 

Additionally, `TierUndelegate` and `ClaimTierRewards` both call `claimRewards`, which computes bonus APY from the tier's `BonusApy` field. If the tier is deleted, that lookup also fails, blocking even positions that had already triggered exit.

The `CloseOnly` flag is the intended graceful-sunset mechanism (it blocks new entries while allowing all exit-path messages to proceed), but `DeleteTier` is a harder deletion with no equivalent safety net. [4](#0-3) 

---

### Impact Explanation

User funds (staked bond-denom tokens held via the position's delegator sub-account) become permanently unrecoverable for every position in the deleted tier that has not yet triggered exit. The delegator sub-account holds the delegation and, after unbonding, the liquid tokens. With no path to call `TriggerExitFromTier` → `TierUndelegate` → `WithdrawFromTier`, those tokens are stranded in the module.

---

### Likelihood Explanation

Governance can legitimately want to remove a tier that was added by mistake or is no longer needed. The existence of `MsgDeleteTier` as a governance message makes this a realistic operational action. Without an explicit check or documentation warning, a governance proposal to delete a tier with live positions is plausible, especially if the tier has `CloseOnly = true` and governance assumes all positions have exited.

---

### Recommendation

Add a pre-deletion guard in `DeleteTier` that rejects the message if any positions still reference the tier:

```go
count, err := ms.GetPositionCountForTier(ctx, msg.Id)
if err != nil {
    return nil, err
}
if count > 0 {
    return nil, types.ErrTierHasActivePositions
}
```

Alternatively, enforce that a tier must be `CloseOnly` with a zero position count before it can be deleted, mirroring the pattern used for graceful sunsetting.

---

### Proof of Concept

1. User calls `MsgLockTier` for Tier 1 (no exit triggered yet). Position is created with `TierId = 1`.
2. Governance passes `MsgDeleteTier{Id: 1}`. `deleteTier` removes Tier 1 from state.
3. User calls `MsgTriggerExitFromTier` for their position. `getTier(ctx, 1)` returns `ErrTierNotFound`. Transaction fails.
4. Without a triggered exit, `validateWithdrawFromTier` returns `ErrExitNotTriggered`. `MsgWithdrawFromTier` is blocked.
5. `MsgTierUndelegate` is also blocked (exit not triggered, and `claimRewards` would fail on tier lookup).
6. User's staked tokens are permanently stuck in the position's delegator sub-account. [1](#0-0) [5](#0-4) [3](#0-2)

func (ms msgServer) DeleteTier(ctx context.Context, msg *types.MsgDeleteTier) (*types.MsgDeleteTierResponse, error) {
	if err := ms.requireAuthority(msg.Authority); err != nil {
		return nil, err
	}

	tier, err := ms.getTier(ctx, msg.Id)
	if err != nil {
		return nil, err
	}

	if err := ms.deleteTier(ctx, msg.Id); err != nil {
		return nil, err
	}

	if err := ms.emitTierChangedEvent(ctx, types.TierChangeAction_TIER_CHANGE_ACTION_DELETE, tier); err != nil {
		return nil, err
	}

	return &types.MsgDeleteTierResponse{}, nil
}

func (ms msgServer) TriggerExitFromTier(ctx context.Context, msg *types.MsgTriggerExitFromTier) (*types.MsgTriggerExitFromTierResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	pos, err := ms.getPositionState(ctx, msg.PositionId)
	if err != nil {
		return nil, err
	}

	if err := ms.validateTriggerExit(pos.Position, msg.Owner); err != nil {
		return nil, err
	}

	tier, err := ms.getTier(ctx, pos.TierId)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	pos.TriggerExit(sdkCtx.BlockTime(), tier.ExitDuration)

	if err := ms.setPosition(ctx, pos.Position, nil); err != nil {
		return nil, err
	}

	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventExitTriggered{
		PositionId:   pos.Id,
		TierId:       pos.TierId,
		Owner:        pos.Owner,
		ExitUnlockAt: pos.ExitUnlockAt,
	}); err != nil {
		return nil, err
	}

	return &types.MsgTriggerExitFromTierResponse{
		ExitUnlockAt: pos.ExitUnlockAt,
		PositionId:   pos.Id,
	}, nil
}

func (k Keeper) validateWithdrawFromTier(ctx context.Context, pos types.PositionState, owner string) error {
	if !pos.IsOwner(owner) {
		return types.ErrNotPositionOwner
	}

	if !pos.HasTriggeredExit() {
		return types.ErrExitNotTriggered
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if !pos.CompletedExitLockDuration(sdkCtx.BlockTime()) {
		return types.ErrExitLockDurationNotReached
	}

	if pos.IsDelegated() {
		return types.ErrPositionDelegated
	}

	isUnbonding, err := k.isUnbonding(ctx, pos.DelegatorAddress)
	if err != nil {
		return err
	}
	if isUnbonding {
		return types.ErrPositionUnbonding
	}

	return nil
}
