### Title
Missing `max_l2_gas_amount` Bound for Declare Transactions Allows Admission of Transactions That Diverge from OS Gas Cap - (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The gateway's stateless validator enforces an upper bound on `l2_gas.max_amount` for `Invoke` and `DeployAccount` transactions but explicitly skips this check for `Declare` transactions. The Starknet OS enforces a hard cap (`execute_max_sierra_gas`) on actual execution gas for all transaction types. A `Declare` transaction with an arbitrarily large `l2_gas.max_amount` passes stateless admission, creating a divergence between the gateway's admission policy and the OS execution boundary — the exact structural analog of the LayerZero source/destination gas mismatch.

### Finding Description

In `validate_resource_bounds`, the stateless validator enforces `max_l2_gas_amount` for all transaction types **except** `Declare`:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
``` [1](#0-0) 

The default `max_l2_gas_amount` is `1_210_000_000`: [2](#0-1) 

The OS enforces `execute_max_sierra_gas = 1_110_000_000` and `validate_max_sierra_gas = 100_000_000` as hard caps on actual execution gas for all transaction types, including `Declare`: [3](#0-2) 

The test `valid_l2_gas_amount_on_declare` explicitly confirms that a `Declare` with `max_amount = 200` passes when `max_l2_gas_amount = 100` — a configuration that would reject an `Invoke` or `DeployAccount`: [4](#0-3) 

The stateful validator's `validate_resource_bounds` also only checks the L2 gas **price** against the previous block's price — it does not check the gas **amount** for any transaction type: [5](#0-4) 

A `Declare` transaction with `l2_gas.max_amount = u64::MAX` and `l2_gas.max_price_per_unit = min_gas_price (8_000_000_000)` therefore passes both stateless and stateful gateway admission checks. The blockifier's fee pre-validation then computes the maximum possible fee as `u64::MAX × 8_000_000_000`, which requires an astronomically large account balance. The transaction fails execution with "insufficient balance" — but only **after** being admitted through the gateway and consuming stateful validation resources (running the account's `__validate_declare__` entry point).

The `VersionedConstants` struct stores the `gateway` limits field but marks it `#[allow(dead_code)]`, confirming the blockifier does not re-enforce the gateway's `max_calldata_length` or `max_l2_gas_amount` limits at execution time: [6](#0-5) 

### Impact Explanation

**High. Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

The gateway's stateless validator admits `Declare` transactions with `l2_gas.max_amount` far exceeding the OS execution cap (`execute_max_sierra_gas = 1_110_000_000`). These transactions pass stateless validation unconditionally, then consume stateful validation resources (account `__validate_declare__` execution) before failing the blockifier's fee pre-check. The admission policy is inconsistent: `Invoke` and `DeployAccount` are bounded at `1_210_000_000`; `Declare` is unbounded. An adversary can craft `Declare` transactions with `max_amount = u64::MAX` that reliably pass the stateless gateway, forcing the node to expend stateful validation work on transactions that will never be sequenced.

### Likelihood Explanation

**Medium.** The check is trivially bypassed — any `Declare` transaction with `l2_gas.max_amount > max_l2_gas_amount` passes stateless validation. The TODO comment in the source confirms the developers are aware of the gap. The attacker needs a valid account and nonce but does not need any special privilege or unusual network condition.

### Recommendation

Apply the same `max_l2_gas_amount` upper-bound check to `Declare` transactions in `validate_resource_bounds`, removing the `if let RpcTransaction::Declare(_) = tx { }` early-return branch. The bound should be consistent with or slightly above `execute_max_sierra_gas` from the latest `VersionedConstants`, matching the policy already applied to `Invoke` and `DeployAccount`.

### Proof of Concept

1. Construct a valid `RpcDeclareTransactionV3` with:
   - `l2_gas.max_amount = GasAmount(u64::MAX)`
   - `l2_gas.max_price_per_unit = GasPrice(8_000_000_000)` (≥ `min_gas_price`)
   - Any non-zero `l1_gas` or `l1_data_gas` bound (to pass `ZeroResourceBounds` check)
   - Valid Sierra contract class, valid signature, valid nonce

2. Submit to the gateway. `StatelessTransactionValidator::validate_resource_bounds` reaches line 79, matches `RpcTransaction::Declare(_)`, and returns `Ok(())` without checking `max_amount`.

3. The stateless validator passes. The stateful validator runs `__validate_declare__` (consuming real compute), then the blockifier's fee pre-check computes `u64::MAX × 8_000_000_000` as the maximum possible fee and fails with `InsufficientResourceBounds` — but only after the gateway has already admitted the transaction.

4. Contrast: submitting the same `max_amount = u64::MAX` on an `Invoke` transaction hits line 80–84 and is immediately rejected with `MaxGasAmountTooHigh` at stateless validation, before any stateful work is done. [7](#0-6) [8](#0-7)

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        // TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
        if let RpcTransaction::Declare(_) = tx {
        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
            return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())
    }
