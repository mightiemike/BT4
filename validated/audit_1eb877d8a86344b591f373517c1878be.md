### Title
Coinbase-Proof Guard Bypassable via Direct Call to Deprecated `verify_transaction_inclusion` — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` adds a coinbase Merkle proof check to block the 64-byte transaction forgery attack. However, the original `verify_transaction_inclusion` (v1) remains a fully public, callable NEAR function. Any unprivileged caller can invoke v1 directly, skipping the coinbase guard entirely and obtaining a `true` proof result for a forged transaction.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces the coinbase proof invariant before delegating to v1: [1](#0-0) 

The guard — `compute_root_from_merkle_proof(coinbase_tx_id, 0, coinbase_merkle_proof) == merkle_root` — is the only thing preventing the 64-byte transaction forgery attack. After it passes, v2 simply calls v1: [2](#0-1) 

v1 itself is still declared `pub` with only a `#[pause]` gate: [3](#0-2) 

v1 performs no coinbase validation — it only checks confirmations, main-chain membership, and the raw Merkle path: [4](#0-3) 

The `#[deprecated]` attribute is a Rust compiler hint; it does not restrict call-site access on NEAR. Any account can submit a NEAR function-call transaction targeting `verify_transaction_inclusion` and receive a result without ever touching the coinbase guard.

The `ProofArgs` struct requires no privileged fields: [5](#0-4) 

---

### Impact Explanation

The 64-byte transaction attack (https://www.bitmex.com/blog/64-Byte-Transactions) allows an adversary to craft a 64-byte payload whose double-SHA256 hash collides with an internal Merkle tree node. By supplying this hash as `tx_id` along with a valid sibling path, `compute_root_from_merkle_proof` reconstructs the correct `merkle_root` and v1 returns `true` — asserting inclusion of a transaction that does not exist on-chain. Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate a cross-chain action (e.g., releasing bridged funds, minting tokens, settling a payment) can be deceived into accepting a fabricated proof.

---

### Likelihood Explanation

The entry path requires no privileged role, no leaked key, and no social engineering. The attacker only needs to identify a suitable Bitcoin block (one whose Merkle tree has an internal node that can be interpreted as a 64-byte transaction), construct the forged `ProofArgs`, and submit a standard NEAR function-call transaction. The function is live on mainnet deployments and is not paused by default. The attack is realistic for any adversary motivated by the value locked behind the cross-chain verification gate.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` or replace it with `pub(crate)` / make it `#[private]` so it is only reachable as an internal helper called by v2. If backward compatibility with existing callers is required, add the same coinbase proof check directly inside v1 rather than only in v2, so the guard cannot be bypassed regardless of which entry point is used.

---

### Proof of Concept

1. Identify a confirmed Bitcoin mainchain block `B` whose Merkle tree contains an internal node `N` whose 32-byte value, when interpreted as a transaction hash, is 64 bytes when serialized (i.e., the raw transaction would be 64 bytes — a known class of valid Bitcoin transactions).
2. Construct `ProofArgs`:
   - `tx_id` = `N` (the internal node hash, treated as a fake tx hash)
   - `tx_block_blockhash` = hash of block `B` (already stored in the contract)
   - `tx_index` = the index corresponding to `N`'s position in the tree
   - `merkle_proof` = the sibling path from `N` to the Merkle root of `B`
   - `confirmations` = any value ≤ `gc_threshold`
3. Call `verify_transaction_inclusion` directly (bypassing v2) with the above args.
4. The contract executes: [6](#0-5) 
   and returns `true`, asserting that the non-existent transaction is included in block `B`.
5. Any downstream contract gating on this result treats the forged transaction as confirmed.
