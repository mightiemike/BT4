### Title
Unvalidated `flags` Parameter Silently Enables Hard-Fork Dialect Features Before Activation — (File: `wheel/src/api.rs`)

### Summary

The Python-facing entry point `run_serialized_chia_program` accepts a raw `u32` flags value from the caller and converts it with `ClvmFlags::from_bits_truncate(flags)`. This silently accepts any bit that corresponds to a defined-but-intentionally-unexposed `ClvmFlags` variant — specifically the hard-fork flags `RELAXED_BLS` (`0x0008`) and `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` (`0x0100`) — without any validation or rejection. A Python caller can pass these raw bit values directly, enabling hard-fork dialect behaviors before the corresponding on-chain activation, causing consensus divergence.

### Finding Description

`run_serialized_chia_program` is the primary Python API for executing CLVM programs:

```rust

    args: &[u8],

Two hard-fork flags are **defined** in `ClvmFlags` but **deliberately not exposed** to Python:

```rust
// src/chia_dialect.rs:39-66
const RELAXED_BLS = 0x0008;          // Hard-fork; enable only when it activates.
const ENABLE_KECCAK_OPS_OUTSIDE_GUARD = 0x0100;  // Hard-fork; enable only when it activates.
``` [3](#0-2) 

Because `from_bits_truncate` retains all bits that match any defined flag, a Python caller passing `flags=0x0100` silently enables `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`, which routes opcode 62 to `op_keccak256` outside the softfork guard:

```rust
// src/chia_dialect.rs:246
62 if flags.contains(ClvmFlags::ENABLE_KECCAK_OPS_OUTSIDE_GUARD) => op_keccak256,
``` [4](#0-3) 

Similarly, `flags=0x0008` enables `RELAXED_BLS`, which causes `op_bls_g1_negate` and `op_bls_g2_negate` to accept invalid curve points. No validation, error, or rejection occurs for either raw bit value.

### Impact Explanation

**Consensus divergence.** A node or wallet using `run_serialized_chia_program` with `flags=0x0100` will accept and execute keccak256 programs outside the softfork guard, while all other nodes (without this flag) will treat opcode 62 as an unknown operator and return nil or reject it. This creates a split in which transactions are valid, breaking the determinism invariant that is the core security property of the CLVM. For `RELAXED_BLS` (`0x0008`), BLS negate operations on invalid curve points succeed instead of failing, which can corrupt signature verification outcomes. Both are classified as hard-fork features with explicit comments "enable only when it activates," meaning premature activation is an intended-to-be-prevented state.

### Likelihood Explanation

The Python API is public and directly callable by any Python code using the `clvm_rs` wheel. The `flags` parameter is a plain `u32` with no documented constraint. A developer testing features, a downstream library, or a malicious caller can pass `0x0100` or `0x0008` without any indication that these are invalid inputs. The bit values are derivable from the published Rust source. No privilege is required.

### Recommendation

Replace `from_bits_truncate` with `from_bits` and reject any `flags` value containing bits not in the set of Python-exposed flags:

```rust
// Allowlist of flags valid for Python callers
const PYTHON_ALLOWED_FLAGS: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK)
    .union(ClvmFlags::ENABLE_SHA256_TREE)
    .union(ClvmFlags::ENABLE_SECP_OPS);

let raw = ClvmFlags::from_bits(flags)
    .ok_or_else(|| PyValueError::new_err("unknown flag bits"))?;
if raw.intersects(!PYTHON_ALLOWED_FLAGS) {
    return Err(PyValueError::new_err("flag not permitted from Python API"));
}
```

This ensures hard-fork flags (`RELAXED_BLS`, `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`, `MALACHITE`) cannot be activated through the Python boundary without an explicit, reviewed code change.

### Proof of Concept

```python
import clvm_rs

# Serialize a trivial program: (keccak256 (q . "foobar"))
# opcode 62 = keccak256
program_hex = bytes.fromhex("ff3eff86666f6f62617280")  # (keccak256 (q . "foobar"))
args_hex = b"\x80"  # nil

# Without the flag: opcode 62 is unknown → returns nil (consensus mode)
cost1, result1 = clvm_rs.run_serialized_chia_program(
    program_hex, args_hex, 10_000_000, 0
)

# With raw bit 0x0100 (ENABLE_KECCAK_OPS_OUTSIDE_GUARD, not exposed in Python API):
cost2, result2 = clvm_rs.run_serialized_chia_program(
    program_hex, args_hex, 10_000_000, 0x0100
)
# result2 is the keccak256 hash of "foobar" — different from result1
# Two nodes running the same program with different flags produce different results
# → consensus divergence
```

The `flags` parameter accepts `0x0100` without error. The dialect constructed from it enables `op_keccak256` at opcode 62 outside the softfork guard, producing a different execution result than a node running with `flags=0`. No validation rejects the undocumented bit.
