### Title
Removal of `used_aux_parent_blocks` Tracking Allows AuxPoW Parent Block Reuse for Chain-Reorg Amplification — (File: `contract/src/lib.rs`, `contract/src/dogecoin.rs`)

---

### Summary

The `used_aux_parent_blocks: LookupSet<H256>` field, which recorded every AuxPoW parent block hash after its first use to prevent the same parent block from being submitted more than once, was silently dropped during the migration from `BtcLightClientV2` to the current `BtcLightClient`. The `check_aux` function contains no replacement guard. A relayer can therefore use a single mined parent block to submit an arbitrary number of distinct Dogecoin blocks, multiplying effective mining power and enabling chain reorganizations that would otherwise require proportionally more hash rate.

---

### Finding Description

**Removed tracking field.** `BtcLightClientV2` carried:

```rust
used_aux_parent_blocks: near_sdk::collections::LookupSet<H256>,
``` [1](#0-0) 

The `migrate()` function constructs the new state from the old one and simply omits this field — no replacement set, no alternative check: [2](#0-1) 

The migration comment acknowledges the drop explicitly ("drops `used_aux_parent_blocks`") and notes that orphaned storage entries are left behind on Dogecoin deployments. [3](#0-2) 

**No replacement in `check_aux`.** The entire AuxPoW validation path in the current code is `check_aux` (lines 49–155 of `dogecoin.rs`). It verifies the chain-merkle proof, the coinbase merkle proof, the merged-mining header position, the tree-size/nonce fields, and the parent block's PoW — but it never records or checks the parent block hash against any used-set: [4](#0-3) 

**How the same parent block commits to multiple Dogecoin blocks.** `AuxData.chain_merkle_proof` is a merkle path of length `k`, meaning the chain merkle tree has `2^k` leaves (the only bound is `k ≤ 30`): [5](#0-4) 

Each leaf can hold a different Dogecoin block hash. `check_aux` computes the chain root from `block_header.block_hash()` and the supplied proof, then verifies that root appears in the parent coinbase — it does not verify that no other leaf in the same tree was already accepted: [6](#0-5) 

**Submission path.** `submit_blocks` → `submit_block_header` (Dogecoin variant) → `check_aux` → `submit_block_header_inner`. The public entry point is: [7](#0-6) 

---

### Impact Explanation

An attacker who controls a trusted-relayer account can:

1. Pre-compute a sequence of `n` Dogecoin block headers `B₁…Bₙ` (each extending the previous), knowing all their hashes in advance because the attacker controls their content.
2. Mine **one** Bitcoin parent block whose coinbase chain-merkle tree has `n` leaves, one per Dogecoin block hash.
3. Call `submit_blocks` `n` times, each time supplying a different `(Bᵢ, AuxData_i)` pair. Every call passes `check_aux` because each has a valid chain-merkle proof against the same parent block.
4. The contract accepts all `n` blocks as having independent PoW, updating `mainchain_tip_blockhash` and the canonical maps `mainchain_height_to_header` / `mainchain_header_to_height` accordingly.

The corrupted state is the canonical chain: the attacker can build a fork of arbitrary length `n` at the cost of mining a single parent block, then trigger `reorg_chain` once the fork's accumulated `chain_work` exceeds the honest tip's: [8](#0-7) 

After a successful reorg, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` will attest to transactions in the attacker's fraudulent chain, directly enabling bridge-level double-spends for any consumer contract that relies on these proofs. [9](#0-8) 

---

### Likelihood Explanation

**Medium.** The attack requires the caller to hold trusted-relayer status (enforced by `#[trusted_relayer]` on `submit_blocks`). However, the `trusted_relayer` macro exposes bypass roles (`Role::DAO`, `Role::UnrestrictedSubmitBlocks`) and the relayer model is designed to allow external participants to stake and join. A well-resourced attacker can acquire relayer status legitimately, then exploit the missing guard. The actual Bitcoin hash power needed is that of a single parent block — a one-time cost — after which the amplification is bounded only by the chain-merkle tree depth (up to 2³⁰ leaves).

---

### Recommendation

Restore the parent-block-reuse guard. After `check_aux` validates a submission, record `aux_data.parent_block.block_hash()` in a persistent set and `require!` it is not already present before proceeding. This is exactly what `used_aux_parent_blocks` did in `BtcLightClientV2`. The set should be re-introduced into the contract state and populated during migration for any parent blocks already accepted.

---

### Proof of Concept

```
Setup (off-chain):
  B1 = Dogecoin header at height N   (prev = mainchain tip, attacker-controlled content)
  B2 = Dogecoin header at height N+1 (prev = hash(B1), attacker-controlled content)

  Mine one Bitcoin parent block P whose coinbase chain-merkle tree has 2 leaves:
    leaf[0] = hash(B1)
    leaf[1] = hash(B2)
  P.PoW is valid (single real mining effort).

Attack (on-chain, two submit_blocks calls):
  Call 1: submit_blocks([(B1, AuxData{chain_id=0, chain_merkle_proof=[sibling_of_B2], parent_block=P, ...})])
    → check_aux: chain_root computed from hash(B1)+proof == root in P's coinbase ✓
    → B1 accepted, mainchain_tip = hash(B1)

  Call 2: submit_blocks([(B2, AuxData{chain_id=1, chain_merkle_proof=[sibling_of_B1], parent_block=P, ...})])
    → check_aux: chain_root computed from hash(B2)+proof == same root in P's coinbase ✓
    → B2 accepted, mainchain_tip = hash(B2)

Result: Two blocks accepted with one parent block's PoW.
        mainchain_height_to_header[N]   = hash(B1)
        mainchain_height_to_header[N+1] = hash(B2)
        verify_transaction_inclusion now attests to transactions in attacker-controlled blocks.
```

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }

        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

            .headers_pool
            .get(&self.mainchain_tip_blockhash)

            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1

        );

            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

            usize::try_from(args.tx_index).unwrap(),

    pub(crate) fn check_aux(&mut self, block_header: &Header, aux_data: &AuxData) {
        // The Dogecoin block must have the AuxPoW flag set (bit 8) when AuxPoW data is present.
        // https://github.com/dogecoin/dogecoin/blob/master/src/auxpow.h
        const BLOCK_VERSION_AUXPOW: i32 = 0x100;

        require!(
            aux_data.chain_merkle_proof.len() <= 30,
            "Aux POW chain merkle branch too long"
        );
        require!(
            block_header.version & BLOCK_VERSION_AUXPOW != 0,
            "Aux POW block does not have AuxPoW flag set in version"
        );

        let chain_id = self.get_config().aux_chain_id;

        require!(
            chain_id == block_header.get_chain_id(),
            format!(
                "block does not have our chain ID (got {}, expected {chain_id})",
                block_header.get_chain_id()
            )
        );

        require!(
            chain_id != aux_data.parent_block.get_chain_id(),
            "Aux POW parent has our chain ID"
        );

        let chain_root = merkle_tools::compute_root_from_merkle_proof(
            block_header.block_hash(),
            aux_data.chain_id,
            &aux_data.chain_merkle_proof,
        );

        let coinbase_tx = aux_data.get_coinbase_tx();
        let coinbase_tx_hash = coinbase_tx.compute_txid();

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                H256::from(coinbase_tx_hash.to_raw_hash().to_byte_array()),
                0,
                &aux_data.merkle_proof,
            ) == aux_data.parent_block.merkle_root
        );

        let script_sig = coinbase_tx
            .input
            .first()
            .unwrap()
            .script_sig
            .to_hex_string();
        let pos_merged_mining_header = script_sig.find(MERGED_MINING_HEADER);
        let mut pos_chain_root = script_sig
            .find(&chain_root.to_string())
            .expect("Aux POW missing chain merkle root in parent coinbase");

        match pos_merged_mining_header {
            Some(pos_merged_mining_header) => {
                if script_sig[pos_merged_mining_header + MERGED_MINING_HEADER.len()..]
                    .contains(MERGED_MINING_HEADER)
                {
                    env::panic_str("Multiple merged mining headers in coinbase");
                }

                require!(
                    pos_merged_mining_header + MERGED_MINING_HEADER.len() == pos_chain_root,
                    "Merged mining header is not just before chain merkle root"
                );
            }
            None => {
                require!(pos_chain_root <= 40, "Aux POW chain merkle root must start in the first 20 bytes of the parent coinbase");
            }
        }

        pos_chain_root += chain_root.to_string().len();
        require!(
            script_sig.len() - pos_chain_root >= 16,
            "Aux POW missing chain merkle tree size and nonce in parent coinbase"
        );

        let bytes = hex::decode(&script_sig[pos_chain_root..pos_chain_root + 8]).unwrap();
        let n_size = u32::from_le_bytes([bytes[0], bytes[1], bytes[2], bytes[3]]);
        require!(
            n_size == (1u32 << aux_data.chain_merkle_proof.len()),
            "Aux POW merkle branch size does not match parent coinbase"
        );

        let bytes = hex::decode(&script_sig[pos_chain_root + 8..pos_chain_root + 16]).unwrap();
        let n_nonce = u32::from_le_bytes([bytes[0], bytes[1], bytes[2], bytes[3]]);

        let chain_id = block_header.get_chain_id();

        let expected_index =
            Self::get_expected_index(n_nonce, chain_id, aux_data.chain_merkle_proof.len());

        require!(
            u32::try_from(aux_data.chain_id).ok() == Some(expected_index),
            "Aux POW wrong index"
        );
        let pow_hash = aux_data.parent_block.block_hash_pow();
        require!(
            self.skip_pow_verification
                || U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
            format!("block should have correct pow")
        );
    }
