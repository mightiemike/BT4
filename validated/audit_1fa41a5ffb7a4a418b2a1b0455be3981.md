### Title
Gas Key Host Functions Use Borsh Wire Length Instead of Trie-ID Length for ML-DSA-65 Exec Fee, Breaking Fee Parity — (`runtime/near-vm-runner/src/logic/logic.rs`, `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`)

---

### Summary

The gas key host functions (`promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`) pass the raw caller-supplied `public_key_len` (the borsh wire length) to `gas_key_transfer_exec_fee` / `gas_key_add_key_exec_fee` when computing the exec-side gas fee. For ML-DSA-65 keys this value is **1953 bytes** (borsh wire), whereas the correct reference for exec-fee pricing is `trie_id_len()` = **33 bytes** (the SHA3-256 hash stored in the trie). The transaction-level path in `runtime/runtime/src/config.rs` already uses `action.public_key.trie_id_len()` correctly. The divergence causes the exec fee charged via host function to be approximately **59× higher** than the equivalent transaction-level action for any ML-DSA-65 gas key, breaking the fee-parity invariant that the codebase explicitly tests for ED25519 keys.

---

### Finding Description

The `PostQuantumSignatures` feature (protocol version 85) introduced ML-DSA-65 access keys stored in the trie **by hash**, not by full pubkey. The trie-key encoding for an ML-DSA-65 access key is `[tag=3] || sha3_256(domain_tag || raw_pubkey)` — 33 bytes total — versus the 1953-byte borsh-encoded full pubkey. Two distinct length functions exist to reflect this:

- `PublicKey::len()` → borsh wire length: 33 / 65 / **1953** for ED25519 / SECP256K1 / ML-DSA-65.
- `PublicKey::trie_id_len()` → on-trie length: 33 / 65 / **33** for ED25519 / SECP256K1 / ML-DSA-65.

The documentation explicitly warns: *"Callers that still use `len()` for trie-storage costing will misprice ML-DSA-65 keys by ~1900 bytes."*

**Transaction path (correct):** `runtime/runtime/src/config.rs` lines 347–354 pass `action.public_key.trie_id_len()` to `gas_key_transfer_exec_fee`:

```rust
TransferToGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
```

**Host function path (incorrect):** `runtime/near-vm-runner/src/logic/logic.rs` lines 3092–3096 pass `public_key_len as usize` — the raw caller-supplied borsh wire length — to the same function:

```rust
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,   // ← borsh wire length, not trie_id_len
);
```

`gas_key_transfer_exec_fee` feeds this value directly into `access_key_key_len(account_id_len, public_key_len)` to compute the trie key length, which is then multiplied by the per-byte exec fee rate. For ML-DSA-65, `public_key_len = 1953` instead of the correct `33`, inflating the exec fee by a factor of ≈59.

The same pattern appears in both host function implementations:

- `promise_batch_action_transfer_to_gas_key` — logic.rs line 3095, wasmtime logic.rs line 3325
- `promise_batch_action_add_gas_key_with_full_access` — logic.rs line 3158, wasmtime logic.rs line 3398
- `promise_batch_action_add_gas_key_with_function_call` — logic.rs line 3229

The `gas_key_add_key_exec_fee` function similarly uses the passed `public_key_len` inside `access_key_key_len` to compute nonce trie key lengths, producing the same overestimation for ML-DSA-65 keys.

---

### Impact Explanation

Any contract that calls a gas key host function with an ML-DSA-65 public key will have its exec-side gas fee computed as if the trie key were 1953 bytes long instead of 33 bytes. The exec fee is charged against the gas budget of the resulting receipt. Concretely:

- The exec fee for `TransferToGasKey` / `WithdrawFromGasKey` via host function is ≈59× higher than the identical action submitted as a transaction.
- The exec fee for `AddKey` (gas key) via host function is ≈59× higher per nonce entry.
- Contracts that do not attach sufficient gas will fail with out-of-gas errors even though the equivalent transaction would succeed.
- Contracts that do attach sufficient gas overpay by ≈59× on the exec component, draining contract balances faster than expected.
- The fee-parity invariant tested in `test_gas_key_fee_parity_full_access` / `test_gas_key_fee_parity_function_call` is violated for ML-DSA-65 keys (those tests only exercise ED25519 keys).

---

### Likelihood Explanation

The `PostQuantumSignatures` feature is stabilized at protocol version 85. Once active, any contract that holds or receives an ML-DSA-65 public key and calls one of the three affected host functions triggers the mispricing. A contract author does not need any special privilege — they simply pass a valid 1953-byte borsh-encoded ML-DSA-65 public key to the host function. The caller-supplied `public_key_len` is validated only for key-format correctness (not for being the trie-id length), so the overcharge is deterministic and unavoidable for every ML-DSA-65 gas key host function call.

---

### Recommendation

Replace `public_key_len as usize` with `public_key.trie_id_len()` in the exec-fee calls inside all three affected host functions, mirroring the transaction path. After `get_public_key` decodes the key, `trie_id_len()` is available on the decoded `PublicKey`:

```rust
// After: let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
let decoded_key = public_key.decode()?;
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    decoded_key.trie_id_len(),   // ← correct reference
);
```

Apply the same fix to `gas_key_add_key_exec_fee` calls in `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call`. Add a fee-parity test that exercises ML-DSA-65 keys alongside the existing ED25519 parity tests.

```rust
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}
```

```rust
    /// Length, in bytes, of the on-trie identifier for an access-key
    /// entry owned by this public key. For ed25519/secp256k1 this matches
    /// `len()`; for ML-DSA-65 the trie stores a SHA3-256 hash (33 bytes
    /// including the type tag), not the 1953-byte borsh-encoded pubkey.
    /// Used by storage-fee calculations on the runtime side; cheap to call
    /// (no hashing) - for ML-DSA-65 this returns the size of the digest
    /// form without actually hashing the pubkey.
    pub fn trie_id_len(&self) -> usize {
        match self {
            Self::ED25519(_) => 1 + ed25519_dalek::PUBLIC_KEY_LENGTH,
            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,
        }
    }
```
