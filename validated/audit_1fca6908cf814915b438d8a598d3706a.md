Audit Report

## Title
SwapAllowlistExtension Bypassed via Router: Any User Swaps on Restricted Pools - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates access using the `sender` argument, which `MetricOmmPool.swap()` sets to `msg.sender` — the immediate caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the router becomes `msg.sender` inside the pool. Any admin who allowlists the router to enable legitimate users to use the standard interface simultaneously grants swap access to every unprivileged user, fully bypassing the allowlist guard.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every registered extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` gates on that `sender` value: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` the pool sees. The original user's address is stored only in transient callback context for payment and is never forwarded to the pool or extension: [4](#0-3) 

The same pattern applies to `exactInput` (L92-125), `exactOutputSingle` (L130-147), and `exactOutput` (L154-188). In all cases, the router is `msg.sender` inside `pool.swap()`, so `sender` passed to the extension is always the router address, never the originating user.

**Bypass path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists only addresses A and B (e.g., KYC'd counterparties).
2. To let A and B use the standard router interface, admin calls `setAllowedToSwap(pool, router, true)`.
3. Attacker C (not allowlisted) calls `router.exactInputSingle(...)`.
4. Router calls `pool.swap(...)` → pool passes `msg.sender = router` as `sender` to `beforeSwap`.
5. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
6. Attacker C swaps successfully on a pool that should have blocked them.

If the admin does **not** allowlist the router, the allowlist works but legitimate users A and B cannot use the router — the extension forces direct `pool.swap()` calls, breaking the standard periphery interface. There is no safe configuration that allows both router usage and effective allowlist enforcement.

## Impact Explanation

Any unprivileged user can swap on a pool whose admin intended to restrict access to a specific set of addresses. This breaks the core access-control invariant of `SwapAllowlistExtension`. For institutional pools or pools with asymmetric oracle-priced trades reserved for trusted counterparties, unauthorized swaps can drain LP principal or extract favorable trades that were reserved for trusted parties — constituting a direct loss of LP principal and broken core pool functionality.

## Likelihood Explanation

The trigger is a natural, expected admin action: allowlisting the router so that legitimate users can use the standard periphery. No malicious setup is required. Any pool using `SwapAllowlistExtension` that wants router compatibility is vulnerable. The attacker needs only to call the public `exactInputSingle` function on the router — no special privileges, no flash loans, no callbacks.

## Recommendation

Pass the original user's address through the swap call so extensions can gate on the true initiator. Two concrete approaches:

1. **Add an `initiator` field to `pool.swap()`** — the router passes `msg.sender` explicitly, and `_beforeSwap` forwards it as a separate argument alongside `sender`.
2. **Check `sender` in the extension against a router registry** — if `sender` is a known router, read the initiator from a standardized field in `extensionData` that the router is required to populate; reject calls where `sender` is a router but no valid initiator is encoded.

Until fixed, pool admins must not allowlist the router on allowlist-restricted pools; users on such pools must call `pool.swap()` directly.

## Proof of Concept

```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension (BEFORE_SWAP_ORDER set)
  allowedSwapper[pool][alice]  = true   // legitimate user
  allowedSwapper[pool][router] = true   // admin adds router so alice can use it
  allowedSwapper[pool][attacker] = false

Attack:
  attacker calls:
    router.exactInputSingle(ExactInputSingleParams{
        pool: pool,
        tokenIn: token0,
        zeroForOne: true,
        amountIn: 1_000e18,
        amountOutMinimum: 0,
        ...
    })

  Execution trace:
    router.exactInputSingle()
      → pool.swap(recipient, zeroForOne, amount, limit, "", extensionData)
          msg.sender inside pool = router
        → _beforeSwap(sender=router, ...)
          → SwapAllowlistExtension.beforeSwap(sender=router, ...)
              allowedSwapper[pool][router] == true  ← passes
        → swap executes, attacker receives token1
        → router.metricOmmSwapCallback() pulls token0 from attacker

Result: attacker swaps on a pool they are not allowlisted for.
```
