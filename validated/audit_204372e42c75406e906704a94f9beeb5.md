### Title
Missing Vote Signature Verification in Consensus Allows Fake Votes to Silence Legitimate Validators — (`File: crates/apollo_consensus/src/single_height_consensus.rs`)

---

### Summary

`handle_vote` in `SingleHeightConsensus` accepts and records peer votes without verifying their cryptographic signatures. A malicious peer can broadcast `Vote` messages claiming to be from any committee member. Because the deduplication guard (`received_vote`) keys on `(round, voter)` alone, the real validator's subsequent legitimate vote is silently dropped as a `Conflict`, preventing it from contributing to quorum. This is the direct Sequencer analog of the Shardus `tryAppendVoteHash` bug.

---

### Finding Description

`handle_vote` performs two checks before accepting a vote:

1. The `vote.voter` field names a current committee member.
2. No prior vote from that `(round, voter)` pair has been recorded.

What it does **not** do is verify that the vote's `signature` field was actually produced by the private key corresponding to `vote.voter`. The code acknowledges this with an explicit TODO:

```rust
pub(crate) fn handle_vote(&mut self, vote: Vote) -> Requests {
    // TODO(Asmaa): verify the signature
``` [1](#0-0) 

The deduplication logic in `received_vote` uses `(round, voter)` as the map key. If a vote for that key already exists with a different `proposal_commitment`, the new vote is classified as `VoteStatus::Conflict` and silently dropped:

```rust
VoteStatus::Conflict(old_vote, new_vote) => {
    warn!("Conflicting votes: old={old_vote:?}, new={new_vote:?}");
    CONSENSUS_CONFLICTING_VOTES.increment(1);
    return VecDeque::new();
}
``` [2](#0-1) 

The deduplication key is built purely from `(vote.round, vote.voter)` with no signature component:

```rust
let key = (vote.round, vote.voter);
let map_entry = match vote.vote_type {
    VoteType::Prevote => self.prevotes.get(&key),
    VoteType::Precommit => self.precommits.get(&key),
};
``` [3](#0-2) 

Compounding the problem, legitimate nodes also produce votes with empty default signatures, so there is no baseline signature to compare against:

```rust
let vote = Vote {
    // TODO(Asmaa): sign the vote
    signature: RawSignature::default(),
    ...
};
``` [4](#0-3) 

The verification primitives (`verify_precommit_vote_signature`, `build_precommit_vote_message_digest`) already exist in `apollo_signature_manager` but are never called from `handle_vote`: [5](#0-4) 

---

### Impact Explanation

An attacker who can broadcast consensus messages (any active validator, or any peer admitted to the consensus gossip topic) can:

1. For every committee member `V`, immediately broadcast a `Vote { voter: V, proposal_commitment: <garbage_hash>, signature: <anything> }` for the current round.
2. Each fake vote passes the committee-membership check and is recorded under key `(round, V)`.
3. When the real validator `V` broadcasts its legitimate vote for the correct `proposal_commitment`, `received_vote` returns `VoteStatus::Conflict` and the real vote is dropped.
4. No validator's real vote is ever counted. Quorum is never reached. Block production halts.

Because the fake votes are stored in the in-memory vote maps for the current height, the damage persists for the entire height even if the attacker disconnects. This matches the Shardus report's observation that the network "stays bricked even after the malicious node leaves."

The impact maps to: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing** (consensus stall prevents any block from being committed, so no transactions are ever confirmed).

---

### Likelihood Explanation

Any node that is part of the active validator set, or any peer that can inject messages into the consensus broadcast channel, can trigger this. The attack requires only the ability to send a well-formed `Vote` protobuf message with an arbitrary `voter` field — no cryptographic material from the impersonated validator is needed. The protobuf converter accepts any bytes for the `signature` field:

```rust
let signature =
    value.signature.map(|hashes| hashes.try_into()).transpose()?.unwrap_or_default();
Ok(Vote { vote_type, height, round, proposal_commitment, voter, signature })
``` [6](#0-5) 

---

### Recommendation

In `handle_vote`, after confirming `vote.voter` is a committee member and before calling `received_vote`, verify the vote signature using the voter's public key. The `verify_precommit_vote_signature` function in `apollo_signature_manager` already implements the correct ECDSA check over `build_precommit_vote_message_digest(block_hash)`. The same domain-separated digest must be used when signing in `make_self_vote` (replacing `RawSignature::default()`).

Both TODOs must be resolved together:
- `// TODO(Asmaa): sign the vote` in `state_machine.rs`
- `// TODO(Asmaa): verify the signature` in `single_height_consensus.rs`

Until both are resolved, any peer can impersonate any validator and permanently silence its votes for a given height.

### Proof of Concept

A malicious validator node, upon receiving the first `ProposalInit` for height H round 0, iterates over all committee members and broadcasts one `Vote` per member with:
- `vote_type`: Prevote (or Precommit)
- `height`: H, `round`: 0
- `voter`: the target committee member's `ValidatorId`
- `proposal_commitment`: any value different from the real proposal's commitment
- `signature`: `RawSignature::default()` (empty bytes — identical to what honest nodes currently produce)

Each fake vote passes the committee-membership check at line 249 and is stored in the prevote/precommit map. When the real committee members later broadcast their legitimate votes, `received_vote` returns `VoteStatus::Conflict` for each one (different `proposal_commitment`), and all real votes are dropped at lines 264–268. Quorum is never reached for height H, and the chain halts. [7](#0-6)
