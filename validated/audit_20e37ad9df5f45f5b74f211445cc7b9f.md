### Title
Missing Chain ID Validation in `executeSlowModeTransaction` Allows Replay of Queued Withdrawals on a Forked Chain — (File: `core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.executeSlowModeTransaction()` is a publicly callable function that drains the slow-mode transaction queue. It performs no `block.chainid` validation. Slow-mode transactions stored in `slowModeTxs` also carry no chain ID binding. On a hard fork of the Ink chain, the entire queue state is replicated, and any pending `WithdrawCollateral` (or other asset-moving) slow-mode transactions can be executed by any caller on the forked chain, enabling double-withdrawal of collateral.

---

### Finding Description

`executeSlowModeTransaction()` is an unrestricted external function:

```solidity
function executeSlowModeTransaction() external {
    SlowModeConfig memory _slowModeConfig = slowModeConfig;
    _executeSlowModeTransaction(_slowModeConfig, false);
    nSubmissions += 1;
    slowModeConfig = _slowModeConfig;
}
``` [1](#0-0) 

`_executeSlowModeTransaction` dequeues and executes the next `SlowModeTx` from `slowModeTxs`. It checks only a timestamp guard (`executableAt <= block.timestamp`) and performs no chain ID validation:

```solidity
function _executeSlowModeTransaction(
    SlowModeConfig memory _slowModeConfig,
    bool fromSequencer
) internal {
    ...
    SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
    delete slowModeTxs[_slowModeConfig.txUpTo++];
    require(
        fromSequencer || (txn.executableAt <= block.timestamp),
        ERR_SLOW_TX_TOO_RECENT
    );
    ...
    try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch { ... }
}
``` [2](#0-1) 

Slow-mode transactions are stored as raw `bytes` with no chain ID field:

```solidity
slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
    sender: sender,
    tx: transaction
});
``` [3](#0-2) 

The `processSlowModeTransactionImpl` handler for `WithdrawCollateral` calls `clearinghouse.withdrawCollateral(...)` directly, transferring real tokens out of the protocol:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(...);
    validateSender(txn.sender, sender);
    clearinghouse.withdrawCollateral(
        txn.sender, txn.productId, txn.amount, address(0), nSubmissions
    );
``` [4](#0-3) 

**Contrast with the protected fast-withdrawal path:** `submitFastWithdrawal` in `BaseWithdrawPool` calls `requireValidTxSignatures`, which explicitly binds the digest to `block.chainid`:

```solidity
bytes32 data = keccak256(
    abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
);
``` [5](#0-4) 

The slow-mode path has no equivalent protection.

---

### Impact Explanation

On a hard fork of the Ink chain, the full contract state — including `slowModeTxs`, `slowModeConfig`, and all collateral balances — is replicated on the forked chain. Any pending `WithdrawCollateral` slow-mode transactions (which have a 3-day delay before execution) will still be present in the queue on the forked chain. Any unprivileged caller can invoke `executeSlowModeTransaction()` on the forked chain to drain those queued withdrawals. Users whose withdrawals were pending at the fork point can receive their collateral on **both** chains, effectively stealing from the forked chain's clearinghouse. The corrupted state is: `clearinghouse` token balances on the forked chain are drained without a corresponding reduction in subaccount balances.

---

### Likelihood Explanation

The Ink chain is an OP Stack L2. Hard forks on L2s are less frequent than on L1s but are not impossible (e.g., protocol upgrades, emergency patches). The slow-mode queue has a mandatory 3-day delay (`SLOW_MODE_TX_DELAY`), meaning at any given time there is a non-trivial backlog of pending withdrawals. The attack requires no special privilege — any EOA can call `executeSlowModeTransaction()`. Likelihood is low-to-medium; impact is high (direct collateral theft).

---

### Recommendation

Add a chain ID check at the top of `executeSlowModeTransaction()` (and/or inside `_executeSlowModeTransaction`) that reverts if `block.chainid` does not match the chain ID recorded at contract initialization:

```solidity
uint256 internal deployedChainId;

// in initialize():
deployedChainId = block.chainid;

// in executeSlowModeTransaction():
function executeSlowModeTransaction() external {
    require(block.chainid == deployedChainId, "wrong chain");
    ...
}
```

Alternatively, bind the chain ID into the slow-mode transaction payload at submission time and verify it at execution time, analogous to how `requireValidTxSignatures` already binds `block.chainid` for fast withdrawals.

---

### Proof of Concept

1. User calls `submitSlowModeTransaction` with a `WithdrawCollateral` payload on the original Ink chain. The transaction is stored in `slowModeTxs[N]` with `executableAt = block.timestamp + 3 days`.
2. The Ink chain undergoes a hard fork at block `B` (before the 3-day delay expires). Both chains share identical state including `slowModeTxs[N]`.
3. On the forked chain, after 3 days pass, the attacker (or the user) calls `executeSlowModeTransaction()`.
4. `_executeSlowModeTransaction` dequeues `slowModeTxs[N]`, passes the timestamp check, and calls `processSlowModeTransaction` → `processSlowModeTransactionImpl` → `clearinghouse.withdrawCollateral(...)`.
5. Tokens are transferred out of the forked chain's clearinghouse to the user's address.
6. The same user also executes the withdrawal on the original chain, receiving tokens on both chains.
7. The forked chain's clearinghouse is now undercollateralized by the withdrawn amount, with no corresponding debit to the subaccount balance. [1](#0-0) [4](#0-3)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
```
