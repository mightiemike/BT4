### Title
Unchecked Integer Overflow in `op_unknown` Cost Multiplier Produces Consensus-Diverging Cost — (`File: src/more_ops.rs`)

---

### Summary

The `op_unknown` function in `src/more_ops.rs` computes the final cost of an unknown opcode by multiplying a base cost (derived from attacker-controlled arguments) by a multiplier (derived from attacker-controlled opcode bytes). This multiplication is performed on `u64` values with no overflow check. When the product exceeds `u64::MAX`, Rust debug builds panic (DoS), and release builds silently wrap, producing a falsely small cost value. If the wrapped value passes the subsequent `u32::MAX` guard, the operation is accepted with a severely undercharged cost, constituting a consensus divergence.

---

### Finding Description

In `src/more_ops.rs`, `op_unknown` handles unknown opcodes in lenient (consensus) mode. The cost is computed in two stages:

**Stage 1** — base cost from arguments (cost_function 0–3): [1](#0-0) 

**Stage 2** — multiply by `(cost_multiplier + 1)`: [2](#0-1) 

The `cost_multiplier` is decoded from the opcode bytes as a `u32` cast to `u64`: [3](#0-2) 

The `check_cost(cost, max_cost)` at line 260 only ensures `cost ≤ max_cost` before the multiplication. It does **not** bound the product. The multiplication at line 261 — `cost *= cost_multiplier + 1` — is an unchecked `u64` multiplication. When `cost > u64::MAX / (cost_multiplier + 1)`, the product overflows.

The post-multiplication guard `if cost > u32::MAX as u64` at line 262 is supposed to reject excessively costly unknown ops, but it operates on the already-wrapped value, so it cannot reliably detect the overflow.

**Overflow threshold example:**
- `cost_multiplier = 0xFFFEFFFF` (opcode `[0xFF, 0xFE, 0xFF, 0xFF, X]`, not reserved since `op[1] != 0xFF`)
- `cost_multiplier + 1 = 4,294,836,224`
- Overflow occurs when `cost > u64::MAX / 4,294,836,224 ≈ 4,295,098,368`
- Chia's `max_cost` is `11,000,000,000`, so `cost` can legally reach `4,295,098,368` before the multiplication

With cost_function=2 (mul-like) and two atom arguments of approximately 724 KB each, the base cost reaches ~4.3 billion, triggering the overflow.

The `op_unknown` path is reachable in consensus mode (block validation) but blocked in mempool mode by `NO_UNKNOWN_OPS`: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

**Debug builds (panic / DoS):** Rust panics on integer overflow in debug mode. An attacker submitting a crafted block containing an unknown opcode with a large multiplier and large atom arguments causes any node running a debug build to panic during block validation. This is a targeted DoS.

**Release builds (cost undercharge / consensus divergence):** The multiplication wraps silently. If the wrapped value is ≤ `u32::MAX`, the function returns `Ok(Reduction(wrapped_cost, nil))` — a falsely small cost. The running cost total in `run_program` does not reflect the true cost, allowing the operation to succeed within budget when it should be rejected. Nodes with different build configurations or implementations that correctly detect the overflow would disagree on block validity, causing a consensus split.

Even when the wrapped value is > `u32::MAX` (causing `EvalErr::Invalid`), the error type differs from `EvalErr::CostExceeded`, which can cause divergence between implementations that distinguish these error codes.

---

### Likelihood Explanation

- The vulnerability is reachable in consensus mode (block validation) via any CLVM program containing an unknown opcode with a 5-byte encoding where `op[0]=0xFF, op[1]!=0xFF`.
- The attacker controls both the opcode bytes (setting `cost_multiplier`) and the arguments (setting the base cost).
- Achieving the overflow requires ~1.4 MB of atom data (two ~724 KB atoms), which is within realistic program sizes for Chia.
- No special privileges are required; any block producer can embed such a program in a coin spend.
- The `NO_UNKNOWN_OPS` flag in mempool mode means this bypasses mempool filtering and reaches consensus validation directly.

---

### Recommendation

Replace the unchecked multiplication with a checked variant and reject on overflow:

```rust
// Before (line 261):
cost *= cost_multiplier + 1;

// After:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
```

This ensures that any overflow is caught and returns a deterministic error regardless of build mode, eliminating both the debug panic and the release-mode silent wrap.

---

### Proof of Concept

Craft a 5-byte unknown opcode `[0xFF, 0xFE, 0xFF, 0xFF, 0x80]`:
- `op[0]=0xFF, op[1]=0xFE` → not reserved (reserved requires `op[0]==0xFF && op[1]==0xFF`)
- `cost_multiplier = u32_from_u8([0xFF, 0xFE, 0xFF, 0xFF]) = 0xFFFEFFFF = 4,294,836,223`
- `cost_function = (0x80 & 0b11000000) >> 6 = 2` (mul-like)

Provide two atom arguments each of 741,620 bytes (≈724 KB):
- Base cost ≈ `92 + 885 + 2×741620×6 + 741620²/128 ≈ 4,295,098,369`
- `check_cost(4,295,098,369, 11,000,000,000)` passes
- `4,295,098,369 × 4,294,836,224` overflows `u64`
- **Debug build**: `thread 'main' panicked at 'attempt to multiply with overflow'`
- **Release build**: wrapped value returned as cost, bypassing the `u32::MAX` guard depending on exact atom sizes [6](#0-5) [3](#0-2) [7](#0-6)
