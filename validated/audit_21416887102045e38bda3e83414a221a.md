Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Actual User, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates on `sender`, which resolves to `msg.sender` at the pool — the router contract, not the originating user. Any pool that allowlists the router address (required to support router-based swaps for curated users) inadvertently grants swap access to every user who calls through the router, nullifying per-user access control entirely.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards `sender` unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly with no mechanism to forward the original caller: [4](#0-3) 

The pool therefore sees `msg.sender = router`. The extension checks whether the **router** is allowlisted, not the actual user. By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` (the second parameter — the economically relevant actor), not `sender` (the adder contract): [5](#0-4) 

The swap allowlist has no equivalent correction. The corrupted value is `allowedSwapper[pool][router] = true`, which causes the extension to pass for every user who routes through the router, regardless of individual allowlist status.

## Impact Explanation

The pool admin's per-user swap restriction is reduced to a per-router check, providing no meaningful access control. Any unprivileged user can trade on a curated pool by routing through `MetricOmmSimpleRouter`. LP funds are exposed to unauthorized traders, including toxic flow the allowlist was designed to exclude. This is a broken core pool functionality / admin-boundary break with direct loss-of-protection impact on LP principal, meeting the allowed impact gate.

## Likelihood Explanation

The vulnerable configuration — allowlisting the router to support router-based swaps for curated users — is the natural and expected setup. No special privileges are required: any user can call `MetricOmmSimpleRouter.exactInputSingle`. The trigger is immediate and repeatable for any pool in this configuration.

## Recommendation

`SwapAllowlistExtension.beforeSwap` must check the true originating user, not the intermediary. The preferred fix mirrors `DepositAllowlistExtension`: the router should forward the original `msg.sender` in a tamper-proof way (e.g., via transient storage written by the router before calling the pool, readable by the extension), and the extension should decode and check that value. Alternatively, document that allowlisted pools must not allowlist the router and require users to call the pool directly — though this is operationally fragile and breaks router composability.

## Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only `allowedUser` is allowlisted.
// Pool admin also allowlists the router to support router-based swaps.
swapExtension.setAllowedToSwap(address(pool), address(router), true);
swapExtension.setAllowedToSwap(address(pool), allowedUser, true);

// Attack: unauthorized user bypasses allowlist via router.
vm.prank(unauthorizedUser);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token0),
        recipient: unauthorizedUser,
        amountIn: 1000,
        amountOutMinimum: 0,
        zeroForOne: true,
        priceLimitX64: 0,
        deadline: block.timestamp + 1,
        extensionData: ""
    })
);
// Swap succeeds — allowlist bypassed.
// allowedSwapper[pool][router] == true, so the check passes for any caller of the router.
```
