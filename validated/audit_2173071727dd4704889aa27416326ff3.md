### Title
`ReshardingManager` inherits parent `ChunkExtra.proposed_split` into child shards without reset, causing `InvalidChunkHeaderShardSplit` after dynamic resharding - (File: `chain/chain/src/resharding/manager.rs`)

---

### Summary

When dynamic resharding executes, `ReshardingManager::split_shard_trie_state()` creates child `ChunkExtra` by cloning the parent's `ChunkExtra` but never resets the `proposed_split` field. When `min_epochs_between_resharding = 1`, the parent shard can propose a new split in epoch N+1 (the epoch before resharding executes), storing a non-`None` `proposed_split` in its `ChunkExtra`. The child shards inherit this value. The first chunk produced for each child shard computes `proposed_split = None` (resharding cooldown prevents immediate re-splitting), causing `validate_chunk_with_chunk_extra_and_receipts_root` to return `InvalidChunkHeaderShardSplit` and halting the chain after any dynamic resharding event.

---

### Finding Description

In `chain/chain/src/resharding/manager.rs`, the resharding manager creates child `ChunkExtra` by cloning the parent's `ChunkExtra` and only updating `state_root` and `congestion_info`: [1](#0-0) 

The `proposed_split` field — added in `ChunkExtraV5` — is never reset to `None` for the child shards: [2](#0-1) 

The `proposed_split` field in `ChunkExtra` records the trie-split proposal computed during the current chunk application. It is the authoritative value against which the **next** chunk header's `proposed_split` is validated: [3](#0-2) 

**Exact divergent values at the protocol boundary:**

When `min_epochs_between_resharding = 1`, the resharding cooldown elapses in epoch N+1. The parent shard P was above the memory threshold in epoch N (that is why it was split); it remains above the threshold in epoch N+1 because the split does not take effect until epoch N+2. Therefore, the parent's last chunk in epoch N+1 computes `proposed_split = Some(TrieSplit { boundary_account: "X", left_memory: M1, right_memory: M2 })` and stores it in `ChunkExtra[P, last_block_of_epoch_N+1]`.

The resharding manager then clones this `ChunkExtra` for each child shard:

```
child_ChunkExtra.proposed_split = Some(TrieSplit { boundary_account: "X", ... })  // inherited
```

The first chunk produced for each child shard L and R computes:

```
child_chunk_header.proposed_split = None  // cooldown prevents immediate re-split
```

`validate_chunk_with_chunk_extra_and_receipts_root` compares these two values and returns `Err(Error::InvalidChunkHeaderShardSplit(...))`.

This is explicitly acknowledged in the codebase: [4](#0-3) 

The `proposed_split` field is declared in `ChunkExtraV5` and is the only field not properly initialized for child shards: [5](#0-4) 

The `ChunkExtra::new()` constructor always produces `ChunkExtra::V5`, so the field is always present and always participates in the validation check: [6](#0-5) 

---

### Impact Explanation

`InvalidChunkHeaderShardSplit` causes the first chunk of each child shard to be rejected by all validators. Since the child shards are new and have no prior valid `ChunkExtra`, there is no recovery path without a protocol fix. The chain halts for all shards that were split in the resharding event. Impact is **Critical** for any network running `ProtocolFeature::DynamicResharding` with `min_epochs_between_resharding = 1`.

---

### Likelihood Explanation

The condition is reachable whenever:
1. `ProtocolFeature::DynamicResharding` is enabled (nightly protocol version).
2. `min_epochs_between_resharding = 1` — the minimum permitted value (the docs explicitly state it must be `> 0`).
3. The parent shard remains above the memory threshold in epoch N+1 — highly likely because the split does not take effect until epoch N+2, so the parent shard's memory is unchanged in epoch N+1.

All three conditions are simultaneously satisfiable in any nightly deployment that triggers a dynamic resharding event.

---

### Recommendation

Reset `proposed_split` to `None` for child shards in `ReshardingManager::split_shard_trie_state()` immediately after cloning the parent `ChunkExtra`:

```rust
// chain/chain/src/resharding/manager.rs
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// Child shards start fresh; they must not inherit the parent's proposed split.
// The first chunk produced for each child shard will compute proposed_split
// independently, and the stored ChunkExtra must agree with that computation.
if let ChunkExtra::V5(ref mut v5) = child_chunk_extra {
    v5.proposed_split = None;
}
```

This mirrors the pattern used in `ChunkExtra::new_with_only_state_root`, which explicitly passes `None` for `proposed_split`: [7](#0-6) 

---

### Proof of Concept

1. Build nearcore with `--features nightly` (enables `ProtocolFeature::DynamicResharding`).
2. Configure `DynamicReshardingConfig` with `min_epochs_between_resharding = 1` and a low `memory_usage_threshold`.
3. Deploy contracts that fill a shard above the threshold.
4. Wait for epoch N: the shard proposes a split; `BlockHeaderInnerRestV6.shard_split` is set.
5. Wait for epoch N+1: the parent shard remains above the threshold; its last chunk stores `ChunkExtra.proposed_split = Some(TrieSplit {...})`.
6. Wait for epoch N+2: resharding executes. `ReshardingManager` clones the parent `ChunkExtra` (with non-`None` `proposed_split`) for each child shard.
7. Observe `InvalidChunkHeaderShardSplit` logged for the first chunk of each child shard: [3](#0-2)
