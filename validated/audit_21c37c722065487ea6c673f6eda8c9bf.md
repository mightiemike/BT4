### Title
Escrowed NFTs Permanently Stuck With No Recovery Path on IBC Channel Close — (`File: x/nft-transfer/ibc_module.go`, `x/nft/keeper/keeper.go`)

### Summary
When a user initiates an IBC NFT transfer, the NFT is escrowed (ownership transferred to the escrow address). If the IBC channel is subsequently closed, `OnChanCloseConfirm` returns `nil` without refunding the escrowed NFTs. Because the `BurnNFT` function enforces a dual restriction — the caller must be both the current owner **and** the denom creator — no party can burn, transfer, or recover the escrowed NFT. The NFT is permanently lost with no on-chain recovery path.

### Finding Description
The `BurnNFT` function in `x/nft/keeper/keeper.go` enforces two simultaneous conditions:

```go
nft, err := k.IsOwner(ctx, denomID, tokenID, owner)   // must be owner
_, err = k.IsDenomCreator(ctx, denomID, owner)         // must also be denom creator
``` [1](#0-0) 

When an NFT is escrowed during an IBC transfer via `createOutgoingPacket`, ownership is transferred to the escrow address:

```go
if err := k.nftKeeper.TransferOwner(ctx, classID, tokenID, sender, escrowAddress); err != nil {
``` [2](#0-1) 

After this transfer:
- The **escrow address** is the owner, but is **not** the denom creator.
- The **denom creator** (original minter) is no longer the owner.

Neither party satisfies both conditions of `BurnNFT`. `TransferOwner` also requires the caller to be the current owner (the escrow address), which has no private key. The only legitimate recovery path is the IBC refund flow (timeout or error acknowledgement).

However, `OnChanCloseConfirm` returns `nil` without triggering any refund:

```go
func (im IBCModule) OnChanCloseConfirm(ctx sdk.Context, portID, channelID string) error {
    return nil
}
``` [3](#0-2) 

When the channel closes (e.g., due to IBC light client expiry), in-flight packets are never acknowledged or timed out through the normal flow, and `refundPacketToken` is never called. The escrowed NFTs remain permanently owned by the escrow address with no mechanism to recover them.

### Impact Explanation
Any NFT escrowed on the source chain at the time of channel closure is permanently lost. The NFT record persists in the store under the escrow address's ownership, but:
- No one can call `BurnNFT` (dual owner+creator restriction is unsatisfiable).
- No one can call `TransferOwner` (requires the escrow address as signer, which has no key).
- No governance mechanism exists to override NFT ownership in this module.

The corrupted invariant is NFT ownership: the original owner permanently loses their NFT asset with no on-chain recourse. [4](#0-3) 

### Likelihood Explanation
IBC channel closure is a realistic production event. IBC light clients expire if not updated within the trusting period (e.g., due to relayer downtime). Once a client expires, the channel cannot be used and is effectively closed. Any NFTs escrowed during that window are permanently lost. This is not a theoretical edge case — relayer downtime and client expiry are documented operational risks in IBC deployments.

### Recommendation
In `OnChanCloseConfirm` (and `OnChanCloseInit` if applicable), iterate over all NFTs currently owned by the escrow address for the closing channel and return them to their last known senders, analogous to how `refundPacketToken` operates. Alternatively, introduce a permissionless recovery message that allows the original sender to reclaim an escrowed NFT when the associated channel is closed, bypassing the `BurnNFT` dual-restriction by using `BurnNFTUnverified` or `TransferOwner` with module-level authority.

### Proof of Concept
1. Alice (denom creator) mints `tokenA` under `denomX` via `MsgMintNFT`.
2. Alice calls `MsgTransfer` (IBC NFT transfer) to send `tokenA` to chain B via `channel-0`.
3. `createOutgoingPacket` calls `TransferOwner(ctx, "denomX", "tokenA", alice, escrowAddr)` — escrow address now owns `tokenA`.
4. The IBC light client for `channel-0` expires before the packet is relayed.
5. `OnChanCloseConfirm` is triggered for `channel-0`; it returns `nil` — no refund occurs.
6. Alice attempts `MsgBurnNFT` for `tokenA`: fails — `IsOwner` check fails (escrow address is owner, not Alice).
7. Alice attempts `MsgTransferNFT` for `tokenA`: fails — `IsOwner` check fails.
8. No other message type in the module can move or destroy `tokenA`.
9. `tokenA` is permanently locked in the escrow address. [3](#0-2) [2](#0-1) [1](#0-0)

// BurnNFT deletes a specified NFT
func (k Keeper) BurnNFT(ctx sdk.Context, denomID, tokenID string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	nft, err := k.IsOwner(ctx, denomID, tokenID, owner)
	if err != nil {
		return err
	}

	_, err = k.IsDenomCreator(ctx, denomID, owner)
	if err != nil {
		return err
	}

	k.deleteNFT(ctx, denomID, nft)
	k.deleteOwner(ctx, denomID, tokenID, owner)
	k.decreaseSupply(ctx, denomID)

	return nil
}
```
