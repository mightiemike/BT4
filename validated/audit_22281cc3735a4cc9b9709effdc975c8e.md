Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Gates the Router Address Instead of the Real User, Allowing Any Caller to Bypass a Curated Pool's Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument supplied by the pool, which is the pool's own `msg.sender` — the router contract — when a swap is routed through `MetricOmmSimpleRouter`. If a pool admin allowlists the router (the only viable configuration for a curated pool that accepts router-mediated trades), every unprivileged user can bypass the per-user allowlist by calling any router entry point. This is a structural identity-binding error: the hook gates the intermediary, not the end user.

## Finding Description
`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← router address when called via router
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension via `abi.encodeCall`:

```solidity
// ExtensionCalling.sol L160-177
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

The router stores the real user's address in transient storage for callback settlement only — it is never passed to the pool or to any extension:

```solidity
// MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
IMetricOmmPoolActions(params.pool).swap(params.recipient, ...);
```

The same identity substitution applies to all router entry points: `exactInputSingle`, `exactInput` (all hops), `exactOutputSingle`, and the recursive `exactOutput` callback path, because in every case the pool's `msg.sender` is the router.

**Contrast with `DepositAllowlistExtension`:** the deposit hook ignores `sender` and checks `owner` — the economically relevant actor — so the deposit allowlist is not affected. The swap allowlist has no equivalent "owner" concept; it was designed to gate `sender`, which is correct only for direct pool calls.

**Two broken states arise:**

| Admin configuration | Result |
|---|---|
| `allowedSwapper[pool][router] = true` | Every user bypasses the allowlist via the router |
| `allowedSwapper[pool][router] = false` | Allowlisted users cannot use the standard periphery |

Neither state correctly enforces the intended per-user curation policy.

## Impact Explanation
A pool admin who deploys a curated pool (KYC-gated, institutional-only, whitelist-restricted) and allowlists the router as the permitted swap entry point inadvertently opens the pool to every user. Any unprivileged address can call `MetricOmmSimpleRouter.exactInputSingle` and trade on the curated pool at the pool's configured oracle price without passing the intended identity gate. This is a direct admin-boundary break: the pool admin's allowlist configuration is rendered ineffective by a publicly reachable periphery path, constituting a broken core pool functionality and access-control bypass with direct fund-access impact.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap interface. A pool admin who wants to allow router-mediated swaps on a curated pool must allowlist the router — there is no other mechanism to permit router access while also enforcing per-user restrictions. The misconfiguration is therefore the only viable configuration for a curated pool that accepts router traffic, making the bypass reachable in any realistic deployment of `SwapAllowlistExtension` with the router. No special privileges are required; any EOA can call the router.

## Recommendation
Pass the real initiator's address through the swap path so extensions can gate on it. Two concrete options:

1. **Add a `payer`/`initiator` parameter to `pool.swap`** that the router populates with `msg.sender` before calling the pool. The pool forwards it to extensions alongside `sender`. Extensions can then check whichever actor is appropriate.

2. **Encode the real user address in `extensionData`** and have `SwapAllowlistExtension` decode and verify it (with a signature or trusted-forwarder pattern). This avoids changing the core pool interface.

Additionally, document clearly that `sender` in `beforeSwap` is the immediate caller of `pool.swap` (the router), not the end user, so extension authors do not repeat this mistake.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Admin calls setAllowedToSwap(pool, router, true)   // allow router-mediated swaps
  - Admin does NOT call setAllowedToSwap(pool, eve, true)  // Eve is not allowlisted

Attack:
  - Eve calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  - Router calls pool.swap(...) with msg.sender = router
  - Pool calls _beforeSwap(sender=router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  - Swap executes; Eve trades on the curated pool without being allowlisted

Expected: revert NotAllowedToSwap
Actual:   swap succeeds
```

Foundry test plan: extend `FullMetricExtension.t.sol` — deploy `MetricOmmSimpleRouter`, allowlist the router via `setAllowedToSwap(pool, router, true)`, do NOT allowlist Eve, have Eve call `exactInputSingle`, and assert the swap succeeds (demonstrating the bypass). Then assert that calling `pool.swap` directly as Eve (not the router) correctly reverts with `NotAllowedToSwap`.
