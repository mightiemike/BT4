### Title
Fee-on-Transfer Token Deposit Credits Full Nominal Amount, Inflating Subaccount Collateral Beyond Actual Holdings - (File: `core/contracts/EndpointStorage.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral` transfers tokens using the caller-supplied `amount` parameter and records that same `amount` in a slow-mode transaction. When the slow-mode transaction is later executed, `Clearinghouse.depositCollateral` credits the subaccount with the full nominal `amount` without ever measuring the actual token balance received. For fee-on-transfer (deflationary) tokens, the clearinghouse receives strictly less than `amount`, yet the subaccount is credited the full value, creating a permanent solvency gap.

---

### Finding Description

The deposit flow proceeds in two phases:

**Phase 1 â€” Token transfer (on-chain, immediate):**

`Endpoint.depositCollateralWithReferral` calls `handleDepositTransfer` with the raw `amount`:

```solidity
// Endpoint.sol:144-148
handleDepositTransfer(
    IERC20Base(spotEngine.getToken(productId)),
    msg.sender,
    uint256(amount)
);
```

`handleDepositTransfer` performs **two sequential transfers** of the same nominal `amount`:

```solidity
// EndpointStorage.sol:111-119
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);          // user â†’ Endpoint
    safeTransferTo(token, address(clearinghouse), amount); // Endpoint â†’ Clearinghouse
}
```

For a fee-on-transfer token with fee rate `f`, the clearinghouse receives approximately `amount Ã— (1âˆ’f)Â²` â€” the fee is charged **twice** (once per hop).

**Phase 2 â€” Balance credit (deferred, via slow-mode):**

The slow-mode transaction is queued with the original `amount`:

```solidity
// Endpoint.sol:152-164
slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
    ...
    tx: abi.encodePacked(
        uint8(TransactionType.DepositCollateral),
        abi.encode(DepositCollateral({ sender: subaccount, productId: productId, amount: amount }))
    )
});
```

When executed, `Clearinghouse.depositCollateral` credits `txn.amount` â€” the original nominal value â€” with no balance-delta check:

```solidity
// Clearinghouse.sol:193-208
function depositCollateral(IEndpoint.DepositCollateral calldata txn) external virtual onlyEndpoint {
    ...
    int128 amountRealized = int128(txn.amount) * int128(multiplier);
    spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
    emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
}
```

The same pattern exists for `DepositInsurance`: `submitSlowModeTransactionImpl` calls `handleDepositTransfer(_getQuote(), sender, uint256(txn.amount))` and later `Clearinghouse.depositInsurance` credits `insurance += amount` from `txn.amount` directly.

---

### Impact Explanation

For every deposit of a fee-on-transfer token, the protocol's internal accounting (`spotEngine` balance for the subaccount) exceeds the actual token holdings of the clearinghouse by the fee amount (doubled due to the two-hop transfer). The sum of all subaccount balances for that product will exceed the clearinghouse's real token balance. This creates an undercollateralization gap that grows with each deposit. A user can exploit this to:

1. Deposit a fee-on-transfer token and receive inflated collateral credit.
2. Use the inflated credit to open leveraged positions or withdraw more than was actually deposited.
3. When the last users attempt to withdraw, the clearinghouse lacks sufficient tokens to honor all withdrawals.

The `manualAssert` invariant check (`Clearinghouse.sol:734-744`) compares internal accounting state but does not compare against on-chain token balances, so the desynchronization is not caught by the protocol's own consistency checks.

---

### Likelihood Explanation

Any product whose `spotEngine` config token is a fee-on-transfer ERC20 is affected. The entry path is fully unprivileged: any user can call `Endpoint.depositCollateral` or `Endpoint.depositCollateralWithReferral`. No admin action, governance capture, or special role is required. The trigger is purely a property of the configured token. Likelihood is medium â€” it depends on whether a fee-on-transfer token is ever registered as a product collateral, which is a realistic deployment scenario.

---

### Recommendation

Measure the actual balance delta of the clearinghouse before and after `handleDepositTransfer`, and use that delta â€” not the nominal `amount` â€” as the value recorded in the slow-mode transaction and credited to the subaccount. For example:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount)
    internal returns (uint256 actualReceived)
{
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    actualReceived = token.balanceOf(address(clearinghouse)) - before;
}
```

Then pass `actualReceived` (cast to `uint128`) into the `DepositCollateral` slow-mode transaction instead of `amount`. Alternatively, document that fee-on-transfer and rebasing tokens are explicitly unsupported as product collateral tokens and enforce this at product registration time.

### Proof of Concept

1. Deploy a fee-on-transfer ERC20 token with a 1% transfer fee and register it as a spot product collateral token.
2. Call `Endpoint.depositCollateral(subaccountName, productId, 1_000_000)`.
3. `handleDepositTransfer` executes:
   - `safeTransferFrom(token, user, 1_000_000)` â†’ Endpoint receives `990_000` (1% fee).
   - `safeTransferTo(token, clearinghouse, 1_000_000)` â†’ Clearinghouse receives `980_100` (another 1% fee on `990_000` forwarded, but the call passes `1_000_000` so it reverts unless Endpoint holds enough â€” or if Endpoint had prior balance, the clearinghouse receives `990_000` of the `1_000_000` sent).
   - In the worst case (Endpoint has no prior balance), the second transfer reverts. In the realistic case where the Endpoint forwards exactly what it received, the clearinghouse gets `â‰ˆ980_100`.
4. The slow-mode tx records `amount = 1_000_000`.
5. When executed, `Clearinghouse.depositCollateral` credits `amountRealized` based on `txn.amount = 1_000_000`, not `980_100`.
