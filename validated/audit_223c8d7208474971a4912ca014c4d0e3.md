### Title
Linked Signer Can Authorize `LinkSigner` Transactions to Permanently Hijack Subaccount Control — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The fast-mode `LinkSigner` transaction handler in `EndpointTx.sol` calls `validateSignedTx` with `allowLinkedSigner = true`. This means the **current linked signer** (a session key) can sign a `LinkSigner` transaction to replace itself with any attacker-controlled address. Once replaced, the new linked signer can drain the subaccount via `WithdrawCollateral` or `TransferQuote`. The subaccount owner's primary key is the only recovery path, but the attacker can act before the owner detects the change.

---

### Finding Description

In `processTransactionImpl`, the `LinkSigner` branch is:

```solidity
// EndpointTx.sol L576-590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
```

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which resolves the authorized signer as:

```solidity
// EndpointTx.sol L172-184
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
```

And `Verifier.validateSignature` accepts the signature if it recovers to **either** the subaccount owner address **or** the linked signer:

```solidity
// Verifier.sol L297-303
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
```

This means the **linked signer itself** can sign a `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with any address. The linked signer is a session key intended for trading operations (order signing, withdrawals), not for mutating the signer registry. Allowing it to authorize `LinkSigner` transactions breaks the invariant that only the subaccount owner's primary key should control who is authorized to act on the subaccount.

By contrast, the slow-mode `LinkSigner` path correctly enforces that only the actual `msg.sender` (the subaccount owner) can submit the transaction:

```solidity
// EndpointTx.sol L232-239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // requires msg.sender == subaccount owner
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

The fast-mode path has no equivalent restriction.

---

### Impact Explanation

A compromised or malicious linked signer can:

1. Sign a `LinkSigner` transaction pointing `signedTx.tx.signer` to an attacker-controlled address C.
2. The sequencer processes it; `linkedSigners[victim_subaccount]` is now address C.
3. Address C signs a `TransferQuote` (allowed with `allowLinkedSigner = true`) to sweep the victim's quote balance to an attacker-controlled subaccount, or signs a `WithdrawCollateral` to withdraw to the subaccount owner's wallet (which the attacker may also control if the primary key is compromised).
4. The victim's collateral is permanently lost.

The `TransferQuote` path is the most direct drain vector since it allows the linked signer to transfer to any registered recipient subaccount:

```solidity
// EndpointTx.sol L599-604
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // allowLinkedSigner = true
);
clearinghouse.transferQuote(signedTx.tx);
```

The corrupted state delta is: `linkedSigners[victim_subaccount]` is overwritten to an attacker address, followed by a full collateral drain via `TransferQuote` or `WithdrawCollateral`.

---

### Likelihood Explanation

Linked signers are session keys typically stored in trading bots, browser local storage, or hot wallets — environments with a materially higher compromise surface than a cold primary key. Any user who has set a linked signer is exposed. The attack requires no admin access, no sequencer compromise, and no governance action. The attacker only needs to control the linked signer address, which is the normal threat model for a compromised session key.

---

### Recommendation

Remove `allowLinkedSigner` for `LinkSigner` transactions in fast mode. The linked signer is a delegated session key and must not be permitted to mutate its own delegation. Change the call to:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner's primary key may change the linked signer
);
```

This aligns fast-mode behavior with the slow-mode path, which already enforces `validateSender` (i.e., `msg.sender == subaccount owner`).

---

### Proof of Concept

1. Alice sets her linked signer to address B (her trading bot) via a `LinkSigner` transaction signed by her primary key.
2. Address B is compromised by an attacker.
3. Attacker (controlling B) constructs a `LinkSigner` transaction: `sender = Alice's subaccount`, `signer = address C` (attacker-controlled), signed by address B.
4. Attacker submits this to the sequencer via `submitTransactions`. The sequencer calls `processTransactionImpl`.
5. `validateSignedTx(..., true)` recovers address B from the signature; `getLinkedSigner(Alice's subaccount)` returns B; validation passes.
6. `linkedSigners[Alice's subaccount]` is overwritten to address C.
7. Attacker (controlling C) signs a `TransferQuote` transaction: `sender = Alice's subaccount`, `recipient = attacker's subaccount`, `amount = full balance`.
8. `validateSignedTx(..., true)` recovers C; `getLinkedSigner(Alice's subaccount)` now returns C; validation passes.
9. `clearinghouse.transferQuote` moves Alice's entire quote balance to the attacker's subaccount.
10. Alice's collateral is fully drained.
