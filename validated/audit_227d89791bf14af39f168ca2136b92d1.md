Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Actual User, Allowing Full Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the direct `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, that `msg.sender` is the router contract, not the actual user. Because EOAs cannot call `pool.swap()` directly (the pool immediately calls back `metricOmmSwapCallback` on `msg.sender`, reverting for non-contracts), any pool admin who wants allowlisted EOA users to trade must allowlist the router — which simultaneously grants swap access to every user reachable through the router, completely defeating the allowlist.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` verbatim as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` forwards this `sender` to every configured extension, including `SwapAllowlistExtension`:

`SwapAllowlistExtension.beforeSwap` then checks this `sender` against the per-pool allowlist: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly: [3](#0-2) 

At this point `msg.sender` of `pool.swap()` is the router address, so the extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. The same misbinding applies to `exactInput` (line 104), `exactOutputSingle` (line 136), `exactOutput` (line 165), and intermediate hops in `_exactOutputIterateCallback` (line 220), where the router always calls `pool.swap()` as `msg.sender`.

The structural trap is enforced by the pool's callback requirement: `pool.swap()` calls `IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(...)` immediately after executing the swap: [4](#0-3) 

This means EOAs cannot be the direct caller of `pool.swap()` — they must go through a router contract. The allowlist check therefore never sees the real user's address in any normal EOA swap flow.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses is fully bypassed. Any unprivileged user calls `MetricOmmSimpleRouter.exactInputSingle()` targeting the restricted pool. The extension sees the router address as `sender`; if the router is allowlisted (the only way to enable normal EOA swap flow), the check passes unconditionally. The attacker receives output tokens and the pool receives input tokens — a direct, fund-impacting trade that the pool admin explicitly intended to block. The corrupted value is the `allowedSwapper[pool][sender]` decision: it evaluates `true` for the router when it should evaluate `false` for the unprivileged user.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the canonical periphery swap contract. EOAs cannot call `pool.swap()` directly due to the mandatory `metricOmmSwapCallback` requirement. Therefore, any pool admin who wants allowlisted users to actually trade must allowlist the router, which simultaneously opens the pool to all users. The bypass requires no special knowledge, no privileged access, and no front-running — any user can call the router with no preconditions beyond the router being allowlisted.

## Recommendation

The pool must receive the original user's address through a trusted channel. The simplest safe fix is to have the router encode `abi.encode(msg.sender)` into `extensionData` before calling `pool.swap()`, and have `SwapAllowlistExtension.beforeSwap` decode and check that address when `msg.sender` (the pool's direct caller) is a known trusted router. The extension should verify both that the encoded address is allowlisted and that the call originated from a trusted router, preventing spoofing via arbitrary `extensionData`. Alternatively, the pool could store the original initiator in a transient slot set by the router before calling `pool.swap()` and pass it as a separate field to extensions, distinct from the direct `msg.sender`.

## Proof of Concept

```
Setup:
  - Pool deployed with SwapAllowlistExtension as beforeSwap hook
  - Admin calls setAllowedToSwap(pool, router, true)   // must allowlist router for EOA swaps
  - Admin does NOT call setAllowedToSwap(pool, attacker, true)
  - LP adds liquidity to the pool

Attack:
  1. attacker calls router.exactInputSingle({pool: pool, ...})
  2. router calls pool.swap(recipient, zeroForOne, amount, priceLimit, "", extensionData)
     → msg.sender of pool.swap() = router
  3. pool calls _beforeSwap(sender=router, ...)
  4. SwapAllowlistExtension.beforeSwap checks allowedSwapper[pool][router] → true
  5. Swap executes; attacker receives output tokens

Result:
  attacker successfully swaps on a pool that was supposed to block them.
  The allowlist is completely ineffective for any user who routes through the router.
```
