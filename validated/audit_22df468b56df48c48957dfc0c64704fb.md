### Title
Deprecated `verify_transaction_inclusion` Remains Callable and Lacks Coinbase Merkle Proof Validation, Enabling 64-Byte Transaction Forgery - (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (v1) is still publicly reachable by any unprivileged NEAR caller despite being deprecated. Unlike `verify_transaction_inclusion_v2`, it omits the coinbase Merkle proof check that guards against the 64-byte transaction forgery attack. An attacker can call v1 directly with a crafted internal Merkle tree node as `tx_id` and receive a `true` result for a transaction that was never included in any Bitcoin block.

### Finding Description

The contract exposes two transaction-inclusion verification entry points:

- `verify_transaction_inclusion` (v1): accepts `ProofArgs` and verifies only that `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == block.merkle_root`. [1](#0-0) 

- `verify_transaction_inclusion_v2`: accepts `ProofArgsV2`, first validates a coinbase Merkle proof against the block's `merkle_root`, then delegates to v1. [2](#0-1) 

The v1 function's own docstring acknowledges the gap:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [3](#0-2) 

The Rust `#[deprecated]` attribute is a **compiler-only hint**. It generates a warning for Rust callers at compile time but has zero effect on the compiled WASM binary. The method remains a fully callable public NEAR contract method after deployment. Any NEAR account — including an adversarial one — can invoke it directly via a cross-contract call or a signed transaction.

The coinbase-proof guard in v2 works because the coinbase transaction is always at index 0 in every Bitcoin block. Proving it correctly constrains the full shape of the Merkle tree, making it impossible to substitute an internal 64-byte node for a leaf. Without this guard, an attacker can craft a 64-byte blob whose `double_sha256` equals an internal Merkle node, supply a shorter proof path, and satisfy the root equality check in v1. [4](#0-3) 

The `ProofArgs` struct accepted by v1 does not carry `coinbase_tx_id` or `coinbase_merkle_proof` fields, so there is no way to retrofit the guard into a v1 call. [5](#0-4) 

### Impact Explanation

The contract is the on-chain source of truth for cross-chain applications — atomic swaps, bridged assets, and cross-chain state verification all depend on `verify_transaction_inclusion` returning `true` only for genuine Bitcoin transactions. A forged `true` result lets an attacker:

- Claim a Bitcoin payment was made when it was not, draining a bridged-asset contract.
- Complete the "Bitcoin side" of an atomic swap without ever broadcasting a real Bitcoin transaction.
- Forge any cross-chain state proof that a downstream NEAR contract gates on this call.

The corrupted value is the boolean proof result returned to the consumer contract, which is the contract's sole security output.

### Likelihood Explanation

The entry point is public, requires no special role, and is reachable by any NEAR account. The `#[deprecated]` marker provides no on-chain access restriction. The 64-byte forgery technique is well-documented (https://www.bitmex.com/blog/64-Byte-Transactions) and the construction of a valid forged proof requires only knowledge of a real Bitcoin block's Merkle tree — information that is publicly available. No privileged access, leaked keys, or social engineering is needed.

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public contract interface entirely, or add a runtime guard that panics unconditionally:

```rust
pub fn verify_transaction_inclusion(&self, ...) -> bool {
    env::panic_str("verify_transaction_inclusion is disabled; use verify_transaction_inclusion_v2");
}
```

Alternatively, gate the method with an access-control role so it cannot be called by unprivileged accounts. Marking it `#[private]` would restrict it to self-calls only, which is sufficient since v2 already calls it internally via `self.verify_transaction_inclusion(args.into())`. [6](#0-5) 

### Proof of Concept

1. Identify a real Bitcoin block on the main chain already accepted by the contract (any block in `mainchain_header_to_height`).
2. Obtain its Merkle tree. Compute an internal node `N` at depth `d` (a 64-byte concatenation of two child hashes). `double_sha256(N)` is a valid internal node hash.
3. Construct `ProofArgs`:
   - `tx_id` = the internal node hash computed in step 2.
   - `tx_index` = the index corresponding to that node's position at depth `d`.
   - `merkle_proof` = the `d`-element sibling path from that node up to the root (shorter than a full leaf proof).
   - `tx_block_blockhash` = the real block hash.
   - `confirmations` = 1.
4. Call `verify_transaction_inclusion(args)` on the NEAR contract.
5. `compute_root_from_merkle_proof` traverses only `d` levels and arrives at `merkle_root`. The function returns `true` for a transaction ID that corresponds to no real Bitcoin transaction. [7](#0-6) [8](#0-7)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
