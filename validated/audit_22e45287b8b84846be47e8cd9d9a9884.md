### Title
Missing `ghost_heap` Increment in `new_substr` for SmallAtom→SmallAtom Path Underaccounts Virtual Heap Usage - (File: `src/allocator.rs`)

### Summary

`Allocator::new_substr` in `src/allocator.rs` handles three node types. When the input is a `SmallAtom` and the result also fits in a `SmallAtom`, the function increments `ghost_atoms` but **omits the corresponding `ghost_heap` increment**. Every other allocation path that produces a `SmallAtom` — `new_atom`, `new_small_number` — increments both counters. This breaks the invariant that `ghost_heap` faithfully tracks the virtual bytes consumed by SmallAtom-optimized allocations, causing the heap-limit guard to be systematically under-enforced for any caller that sets a finite `heap_limit` via `Allocator::new_limited`.

### Finding Description

**Root cause — `new_substr`, SmallAtom branch (lines 845–854):**

```rust
ObjectType::SmallAtom => {
    let val = node.index();
    let len = len_for_value(val) as u32;
    bounds_check(node, start, end, len)?;
    let buf: [u8; 4] = val.to_be_bytes();
    let buf = &buf[4 - len as usize..];
    let substr = &buf[start as usize..end as usize];
    if let Some(new_val) = fits_in_small_atom(substr) {
        self.ghost_atoms += 1;
        // ← ghost_heap NOT incremented here
        Ok(self.mk_node(ObjectType::SmallAtom, new_val as usize))
    } else { … }
}
``` [1](#0-0) 

**Correct pattern in `new_atom` (lines 626–629):**

```rust
if let Some(ret) = fits_in_small_atom(v) {
    self.ghost_atoms += 1;
    self.ghost_heap += v.len();   // ← both counters updated
    Ok(self.mk_node(ObjectType::SmallAtom, ret as usize))
}
``` [2](#0-1) 

**Correct pattern in `new_small_number` (lines 647–648):**

```rust
self.ghost_atoms += 1;
self.ghost_heap += len;           // ← both counters updated
``` [3](#0-2) 

The heap-limit guard used by every subsequent allocation is:

```
u8_vec.len() + ghost_heap + new_bytes > heap_limit
``` [4](#0-3) 

Because `ghost_heap` is not incremented by `new_substr` SmallAtom→SmallAtom calls, the guard sees a smaller virtual heap than actually exists, allowing more real bytes to be allocated before the limit fires.

**Attacker-controlled entry path:**

The CLVM `substr` operator calls `new_substr` directly:

```rust
let r = a.new_substr(a0, start as u32, end as u32)?;
``` [5](#0-4) 

Any atom whose value fits in a `SmallAtom` (e.g., the integer `1`, `2`, any single byte ≤ 0x7f) is stored as a `SmallAtom`. An attacker-supplied CLVM program can call `(substr (q . 1) 0 1)` or similar to trigger the path repeatedly. Each call silently omits `end - start` bytes (up to 4 bytes per call) from `ghost_heap`.

### Impact Explanation

The `ghost_heap` counter exists specifically to enforce the heap limit as if SmallAtom-optimized allocations still consumed real heap bytes — the code comment states: *"We still account for them to not affect the limits of atoms and pairs. Those limits must stay the same for consensus purpose."* [6](#0-5) 

When `Allocator::new_limited` is used (confirmed present in `wheel/src/api.rs` and `src/serde/intern.rs`), a crafted program can accumulate a large `ghost_heap` deficit through repeated `substr` calls on SmallAtoms, then allocate real heap bytes that push the true virtual usage past the limit without triggering `OutOfMemory`. This is a heap-limit bypass reachable from attacker-controlled CLVM bytes.

Additionally, `maybe_restore_with_node` (used during softfork guard restoration) contains a guard that checks `ghost_heap >= len` before decrementing it:

```rust
if self.ghost_heap < len {
    return Err(EvalErr::InternalError(NodePtr::NIL, "ghost heap accounting error".to_string()));
}
self.ghost_heap -= len;
``` [7](#0-6) 

If `ghost_heap` is underaccounted due to the missing increment, this guard can fire spuriously as an `InternalError`, stalling execution — the direct analog to the report's "stall report execution" impact.

### Likelihood Explanation

The `substr` operator is a standard CLVM opcode reachable from any attacker-supplied program bytes. No special privileges are required. The SmallAtom path is triggered whenever the input atom is a small integer (extremely common in real programs). The bug fires on every such call, making it straightforward to accumulate a meaningful `ghost_heap` deficit. The impact is conditional on `new_limited` being used with a finite limit, but the Python wheel API (`wheel/src/api.rs`) and the serde intern path (`src/serde/intern.rs`) both call `new_limited`, making this reachable in production.

### Recommendation

In `new_substr`, inside the `ObjectType::SmallAtom` branch, add the missing `ghost_heap` increment before returning the SmallAtom node:

```rust
if let Some(new_val) = fits_in_small_atom(substr) {
    self.ghost_atoms += 1;
    self.ghost_heap += (end - start) as usize;  // ← add this line
    Ok(self.mk_node(ObjectType::SmallAtom, new_val as usize))
}
``` [8](#0-7) 

This mirrors the accounting already performed by `new_atom` and `new_small_number` for the same SmallAtom outcome.

### Proof of Concept

```rust
// Demonstrate ghost_heap underaccounting via new_substr on SmallAtom
let mut a = Allocator::new_limited(10); // 10-byte heap limit

// Allocate a SmallAtom (value=1, 1 byte). new_atom increments ghost_heap by 1.
let small = a.new_atom(&[1]).unwrap();
// ghost_heap is now 2 (initial 1 + 1 for the atom above)

// Call new_substr on the SmallAtom, producing another SmallAtom.
// BUG: ghost_heap is NOT incremented, but it should be += 1.
let _sub = a.new_substr(small, 0, 1).unwrap();
// ghost_heap is still 2, not 3.

// Repeat many times to accumulate deficit, then allocate real bytes
// that exceed the limit without triggering OutOfMemory.
for _ in 0..7 {
    let _ = a.new_substr(small, 0, 1).unwrap();
    // Each call omits 1 byte from ghost_heap
}
// ghost_heap deficit is now 7 bytes; the limit check is 7 bytes too lenient.
// A subsequent new_atom of 7 bytes that should fail will succeed.
let _bypass = a.new_atom(b"1234567").unwrap(); // should be OutOfMemory, but isn't
```
