### Title
`depositCollateral` Encodes `msg.sender` as Subaccount Owner Without Contract Check, Permanently Locking Deposited Funds for Contracts That Cannot Sign ECDSA - (File: core/contracts/Endpoint.sol)

---

### Summary

`Endpoint.depositCollateral` hard-codes `msg.sender` as the first 20 bytes of the subaccount identifier. When a contract (e.g., a multisig, a vault, or any smart-contract wallet) calls this function, the resulting subaccount is permanently owned by the contract address. Because contracts cannot produce ECDSA signatures, they cannot authorize withdrawals through the sequencer path. The only recovery path is the slow-mode queue, but that path also sends tokens back to the contract address and requires the contract to have been written with explicit `submitSlowModeTransaction` call logic. Contracts that lack this logic have their deposited collateral permanently locked.

---

### Finding Description

In `Endpoint.sol`, `depositCollateral` constructs the subaccount by packing `msg.sender` into the high 20 bytes:

```solidity
// Endpoint.sol lines 108-110
bytes32 subaccount = bytes32(
    abi.encodePacked(msg.sender, subaccountName)
);
``` [1](#0-0) 

The first 20 bytes of every subaccount are treated as the "owner address" throughout the protocol. Signature validation in `Verifier.validateSignature` accepts only the address recovered from the ECDSA signature matching `address(uint160(bytes20(sender)))` or a pre-registered linked signer:

```solidity
// Verifier.sol lines 297-303
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [2](#0-1) 

Contracts cannot produce ECDSA signatures. The sequencer-path `WithdrawCollateral` and `WithdrawCollateralV2` both call `validateSignedTx`, which requires such a signature: [3](#0-2) 

Setting a linked signer via the sequencer path (`LinkSigner` transaction type) also requires a valid ECDSA signature from the subaccount owner: [4](#0-3) 

The only remaining path is the slow-mode queue. A contract can call `submitSlowModeTransaction` with a `WithdrawCollateral` payload; `validateSender` accepts `msg.sender` matching the first 20 bytes of the subaccount: [5](#0-4) 

However, when slow-mode `WithdrawCollateral` is processed, `sendTo` is always `address(0)`, which resolves to the contract address itself:

```solidity
// Clearinghouse.sol lines 404-405
if (sendTo == address(0)) {
    sendTo = address(uint160(bytes20(sender)));
}
``` [6](#0-5) [7](#0-6) 

This means:
1. A contract that calls `depositCollateral` cannot withdraw via the sequencer path (no ECDSA capability).
2. It cannot set a linked signer via the sequencer path for the same reason.
3. It can only use slow-mode withdrawal, which sends tokens back to the contract address and requires the contract to have been written with explicit `submitSlowModeTransaction` call logic.
4. Any contract that lacks this call logic (e.g., a multisig deployed before Nado existed, a simple vault, or any contract whose ABI does not include a path to call `submitSlowModeTransaction`) has its deposited collateral permanently locked.

---

### Impact Explanation

Deposited collateral is permanently inaccessible for any contract caller of `depositCollateral` that:
- Cannot produce ECDSA signatures (all contracts by definition), and
- Does not have explicit logic to call `submitSlowModeTransaction` on the Endpoint.

Even when slow-mode withdrawal is available, the tokens are returned to the contract address. If the contract has no mechanism to forward or use those tokens, the value is effectively blackholed. This is a direct asset-loss scenario for affected users, not merely a gas or fee issue.

---

### Likelihood Explanation

Contracts are a natural and expected caller of `depositCollateral`. Multisig wallets (Gnosis Safe, etc.), DeFi aggregators, and protocol-owned vaults routinely interact with DEX deposit functions. The function signature is identical to what a human user would call; there is no documentation warning against contract use. The `depositCollateralWithReferral` overload accepts an arbitrary `bytes32 subaccount`, so a contract that knows to use it can avoid the issue — but `depositCollateral` provides no such escape hatch and is the simpler, more discoverable entry point.

---

### Recommendation

Add an explicit `subaccountOwner` or `recipient` address parameter to `depositCollateral`, or require that the caller is an EOA (`require(msg.sender == tx.origin)`). The simplest safe fix is to allow the caller to specify the address that will be embedded as the subaccount owner, analogous to how `depositCollateralWithReferral` already accepts an arbitrary `bytes32 subaccount`. Alternatively, document clearly that `depositCollateral` must not be called from contracts, and direct contract integrators to `depositCollateralWithReferral` with an EOA-controlled subaccount.

---

### Proof of Concept

1. Alice controls a Gnosis Safe multisig at address `0xMULTISIG`.
2. Alice's multisig calls `Endpoint.depositCollateral("default\x00\x00\x00\x00", USDC_PRODUCT_ID, 10_000e6)`.
3. The Endpoint constructs `subaccount = bytes32(abi.encodePacked(0xMULTISIG, "default\x00\x00\x00\x00"))` and queues a slow-mode deposit. Tokens are transferred from the multisig to the Endpoint.
4. The sequencer processes the deposit; Alice's subaccount now holds 10,000 USDC.
5. Alice wants to withdraw. She attempts to sign a `WithdrawCollateral` transaction — but the multisig cannot produce a raw ECDSA signature over the EIP-712 digest.
6. She attempts to set a linked signer via the sequencer `LinkSigner` path — same problem: requires ECDSA from `0xMULTISIG`.
7. The slow-mode `LinkSigner` path via `submitSlowModeTransaction` would work, but Alice's Gnosis Safe was deployed years ago and has no transaction template for calling `Endpoint.submitSlowModeTransaction`. The Safe's owners do not know the ABI encoding required.
8. Alice's 10,000 USDC is permanently locked in the protocol, credited to a subaccount that no one can authorize a withdrawal from.
