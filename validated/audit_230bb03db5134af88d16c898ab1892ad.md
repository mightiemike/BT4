### Title
Deprecated `verify_transaction_inclusion` Remains Callable, Bypassing the 64-Byte Merkle Forgery Safeguard Introduced in v2 — (`contract/src/lib.rs`)

---

### Summary

The contract exposes two transaction-inclusion verification paths. `verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase merkle proof. The original `verify_transaction_inclusion` (v1) was deprecated but **never removed or access-gated**. Any unprivileged NEAR caller can invoke v1 directly, bypassing the coinbase-proof safeguard entirely and obtaining a `true` result for a forged transaction.

---

### Finding Description

`verify_transaction_inclusion_v2` adds a mandatory coinbase-proof check before delegating to v1: [1](#0-0) 

The check it adds is: [2](#0-1) 

This coinbase proof check is the **only** thing preventing the 64-byte transaction forgery attack. The deprecated v1 function skips it entirely and goes straight to computing the Merkle root from the attacker-supplied `tx_id`: [3](#0-2) 

The Rust `#[deprecated]` attribute is a **compile-time lint only**. It does not prevent runtime invocation on NEAR. The `#[pause]` attribute means the function *can* be paused by a `PauseManager`, but it is not paused by default and is fully live. Any NEAR account can call it permissionlessly.

The 64-byte forgery works as follows: Bitcoin's Merkle tree does not distinguish between leaf nodes (transactions) and internal nodes. A 64-byte blob can be crafted so that its double-SHA256 hash equals an internal Merkle tree node. An attacker supplies this blob as `tx_id` along with a valid partial Merkle path, and v1 returns `true` — confirming inclusion of a transaction that does not exist.

`verify_transaction_inclusion_v2` explicitly documents this risk and delegates to v1 after the coinbase check: [4](#0-3) 

The `ProofArgs` struct accepted by v1 requires no coinbase fields: [5](#0-4) 

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` (v1) — or that an attacker can route through it — will accept a forged Bitcoin transaction inclusion proof. This corrupts the core security guarantee of the light client: that a `true` result means a real Bitcoin transaction was confirmed on-chain. Contracts that gate asset releases, cross-chain bridges, or settlement logic on this result can be drained or manipulated.

---

### Likelihood Explanation

- The function is a public, permissionless NEAR contract method callable by any account.
- The 64-byte Merkle forgery is a well-documented, published Bitcoin vulnerability (referenced in the contract's own comments: https://www.bitmex.com/blog/64-Byte-Transactions).
- Any integrator who has not migrated to v2, or any attacker who deliberately calls v1 instead of v2, is exposed.
- No privileged access, leaked keys, or social engineering is required.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public contract interface entirely, or add an unconditional `env::panic_str("use verify_transaction_inclusion_v2")` at its entry point so it can never return a result. Marking it `#[deprecated]` is insufficient because the Rust deprecation attribute has no runtime effect on NEAR contract dispatch.

---

### Proof of Concept

1. Identify a real Bitcoin block stored in the contract's `headers_pool`.
2. Craft a 64-byte blob `fake_tx` whose `double_sha256(fake_tx)` equals a known internal Merkle tree node `N` of that block.
3. Build a valid partial Merkle path from `N` up to the block's `merkle_root`.
4. Call `verify_transaction_inclusion` directly (not v2) with:
   - `tx_id = fake_tx_hash` (the hash of the 64-byte blob, which equals `N`)
   - `tx_block_blockhash` = the real block hash
   - `merkle_proof` = the path from `N` to `merkle_root`
   - `tx_index` = the index corresponding to `N`'s position
5. The function returns `true`, confirming inclusion of a transaction that does not exist in the Bitcoin block.

Step 4 is reachable by any unprivileged NEAR caller via a direct contract view/call to `verify_transaction_inclusion`. [3](#0-2)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    /// Verifies that a transaction is included in a block at a given block height,
    /// with an additional coinbase merkle proof validation.
    /// This is needed to mitigate the 64-byte transaction Merkle proof forgery vulnerability:
    /// https://www.bitmex.com/blog/64-Byte-Transactions
    ///
    /// @param tx_id transaction identifier
    /// @param tx_block_blockhash block hash at which transaction is supposedly included
    /// @param tx_index index of transaction in the block's tx merkle tree
    /// @param merkle_proof merkle tree path (concatenated LE sha256 hashes) (does not contain initial transaction_hash and merkle_root)
    /// @param coinbase_tx_id coinbase transaction hash
    /// @param coinbase_merkle_proof merkle proof for the coinbase transaction (must have the same length as merkle_proof)
    /// @param confirmations how many confirmed blocks we want to have before the transaction is valid
    /// @return True if tx_id is at the claimed position in the block at the given blockhash, False otherwise
    ///
    /// # Panics
    /// - If `merkle_proof` and `coinbase_merkle_proof` have different lengths
    /// - If `tx_block_blockhash` is not found in the headers pool
    /// - If coinbase merkle proof does not match the block's merkle root
    /// - If the required number of confirmations exceeds the number of stored blocks
    /// - If the block does not belong to the current main chain
    /// - If there are not enough confirmed blocks
    #[pause]
    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
