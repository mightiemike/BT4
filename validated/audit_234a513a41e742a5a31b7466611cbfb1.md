### Title
Conditional `proof_facts` Inclusion in Transaction Hash Creates Hash-Domain Non-Canonicality Exploitable via Feeder-Gateway Sync Path — (`crates/starknet_api/src/transaction_hash.rs`)

### Summary

`get_invoke_transaction_v3_hash` conditionally appends `proof_facts` to the Poseidon hash chain only when the field is non-empty. Multiple deserialization boundaries (`#[serde(default)]`, `unwrap_or_default()`, and the storage backward-compat shim) silently default `proof_facts` to empty when the field is absent. A syncing node that reads a committed `InvokeV3` transaction from the feeder gateway without the `INCLUDE_PROOF_FACTS` response flag will reconstruct the transaction with `proof_facts = []`, compute a different hash than the one committed to the block, and either reject the block or store an authoritative-looking wrong transaction hash — directly matching the "uninitialized field used in validation logic" pattern from the seed report.

### Finding Description

**Root cause — non-canonical hash preimage:**

`get_invoke_transaction_v3_hash` builds the hash chain and then conditionally extends it:

```rust
// crates/starknet_api/src/transaction_hash.rs  lines 388-404
let mut hash_chain = HashChain::new()
    .chain(&INVOKE)
    ...
    .chain(&calldata_hash);
if !transaction.proof_facts().0.is_empty() {          // ← conditional branch
    let proof_facts_hash = HashChain::new()
        .chain_iter(transaction.proof_facts().0.iter())
        .get_poseidon_hash();
    hash_chain = hash_chain.chain(&proof_facts_hash);
}
Ok(TransactionHash(hash_chain.get_poseidon_hash()))
``` [1](#0-0) 

This means the hash preimage has a **variable structure**: transactions with empty `proof_facts` produce `H(base_fields)`, while transactions with non-empty `proof_facts` produce `H(base_fields || proof_facts_hash)`.

**Deserialization boundaries that silently default `proof_facts` to empty:**

1. **`InvokeTransactionV3` JSON serde** — the field is annotated `#[serde(default, skip_serializing_if = "ProofFacts::is_empty")]`, so it is omitted from serialized JSON when empty and defaults to empty when absent during deserialization: [2](#0-1) 

2. **Feeder-gateway / central-sync reader** — `IntermediateInvokeTransaction` carries `proof_facts: Option<ProofFacts>` and the conversion to `InvokeTransactionV3` uses `unwrap_or_default()`: [3](#0-2) 

3. **Storage backward-compat shim** — the custom `StorageSerde` impl for `InvokeTransactionV3` defaults `proof_facts` to empty when no bytes remain after deserializing all other fields: [4](#0-3) 

**The divergence path:**

The feeder-gateway API only returns `proof_facts` when the caller explicitly requests `response_flags: ["INCLUDE_PROOF_FACTS"]` (per the test comment in `raw_rpc_json_test.rs`): [5](#0-4) 

A syncing node that omits this flag receives the transaction body without `proof_facts`. The `unwrap_or_default()` path silently sets `proof_facts = []`. The node then calls `get_invoke_transaction_v3_hash`, which takes the short branch (no `proof_facts_hash` appended), producing hash **H1**. The sequencer originally computed hash **H2** (with `proof_facts` appended). H1 ≠ H2.

### Impact Explanation

- The syncing node stores the transaction under the wrong hash H1.
- Block hash verification, which depends on committed transaction hashes, fails → the node rejects a valid finalized block.
- If the node serves RPC responses (e.g., `starknet_getTransactionByHash`, `starknet_getTransactionReceipt`) using the locally recomputed hash, it returns an authoritative-looking wrong transaction hash to callers.
- This matches: **High — RPC execution or pending view returns an authoritative-looking wrong value**, and **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing**.

### Likelihood Explanation

- Any `InvokeV3` transaction carrying non-empty `proof_facts` (client-side proving, already supported and tested) triggers the divergence.
- The feeder-gateway reader path is production code used for central sync; the `INCLUDE_PROOF_FACTS` flag is opt-in.
- No privilege is required: any user can submit a valid `InvokeV3` with `proof_facts`.
- The backward-compat shim is explicitly marked temporary but is currently active.

### Recommendation

**Short term:**
- Make `get_invoke_transaction_v3_hash` always include `proof_facts` in the hash preimage, even when empty (hash the empty list). This eliminates the conditional branch and makes the preimage canonical regardless of field presence.
- Alternatively, treat `proof_facts = []` and `proof_facts` absent as strictly equivalent at every deserialization boundary and document the invariant explicitly.
- Ensure the feeder-gateway sync client always requests `INCLUDE_PROOF_FACTS` for `InvokeV3` transactions.

**Long term:**
- Add a cross-boundary hash consistency test: serialize a transaction with non-empty `proof_facts` through every deserialization path (JSON serde, protobuf, storage, feeder-gateway reader) and assert that `calculate_transaction_hash` produces the same value in all cases.
- Remove the storage backward-compat shim once all nodes have re-synced (the existing TODO confirms this is planned).

### Proof of Concept

```
1. Submit InvokeV3 tx with proof_facts = [0xdeadbeef, ...] via gateway.
   → Gateway calls get_invoke_transaction_v3_hash → H2 (includes proof_facts_hash).
   → Tx stored with tx_hash = H2, included in block B.

2. Syncing node fetches block B from feeder gateway WITHOUT INCLUDE_PROOF_FACTS.
   → IntermediateInvokeTransaction.proof_facts = None
   → TryFrom conversion: proof_facts = None.unwrap_or_default() = []
   → Node calls get_invoke_transaction_v3_hash → H1 (no proof_facts_hash appended).
   → H1 ≠ H2.

3. Node stores tx under H1.
   → Block hash verification uses H1 ≠ H2 → block rejected (DoS).
   → starknet_getTransactionByHash(H2) returns not-found; H1 returns wrong tx.
```

The conditional at `transaction_hash.rs:399` is the exact analog of the uninitialized `output_cnt` field: a field that is silently absent/defaulted in one representation is used in a security-critical computation (hash) in another, producing a divergent value. [6](#0-5) [7](#0-6) [8](#0-7)

// TODO(AvivG): Remove this migration shim once all nodes have re-synced.
impl StorageSerde for InvokeTransactionV3 {
    fn serialize_into(&self, res: &mut impl std::io::Write) -> Result<(), StorageSerdeError> {
        let mut to_compress: Vec<u8> = Vec::new();
        self.resource_bounds.serialize_into(&mut to_compress)?;
        self.tip.serialize_into(&mut to_compress)?;
        self.signature.serialize_into(&mut to_compress)?;
        self.nonce.serialize_into(&mut to_compress)?;
        self.sender_address.serialize_into(&mut to_compress)?;
        self.calldata.serialize_into(&mut to_compress)?;
        self.nonce_data_availability_mode.serialize_into(&mut to_compress)?;
        self.fee_data_availability_mode.serialize_into(&mut to_compress)?;
        self.paymaster_data.serialize_into(&mut to_compress)?;
        self.account_deployment_data.serialize_into(&mut to_compress)?;
        self.proof_facts.serialize_into(&mut to_compress)?;
        if to_compress.len() > COMPRESSION_THRESHOLD_BYTES {
            IsCompressed::Yes.serialize_into(res)?;
            if to_compress.len() > crate::compression_utils::MAX_DECOMPRESSED_SIZE {
                warn!(
                    "InvokeTransactionV3 serialization size is too large and will lead to \
                     deserialization error: {}",
                    to_compress.len()
                );
            }
            let compressed = compress(to_compress.as_slice())?;
            compressed.serialize_into(res)?;
        } else {
            IsCompressed::No.serialize_into(res)?;
            to_compress.serialize_into(res)?;
        }
        Ok(())
    }

    fn deserialize_from(bytes: &mut impl std::io::Read) -> Option<Self> {
        let is_compressed = IsCompressed::deserialize_from(bytes)?;
        let maybe_compressed_data = Vec::<u8>::deserialize_from(bytes)?;
        let data = match is_compressed {
            IsCompressed::No => maybe_compressed_data,
            IsCompressed::Yes => decompress(maybe_compressed_data.as_slice())
                .expect("destination buffer should be large enough"),
        };
        let data = &mut data.as_slice();
        let resource_bounds = ValidResourceBounds::deserialize_from(data)?;
        let tip = Tip::deserialize_from(data)?;
        let signature = TransactionSignature::deserialize_from(data)?;
        let nonce = Nonce::deserialize_from(data)?;
        let sender_address = ContractAddress::deserialize_from(data)?;
        let calldata = Calldata::deserialize_from(data)?;
        let nonce_data_availability_mode = DataAvailabilityMode::deserialize_from(data)?;
        let fee_data_availability_mode = DataAvailabilityMode::deserialize_from(data)?;
        let paymaster_data = PaymasterData::deserialize_from(data)?;
        let account_deployment_data = AccountDeploymentData::deserialize_from(data)?;
        // Backward compatibility: proof_facts may not exist in old transactions.
        // If no data remains, default to empty; otherwise, deserialize normally.
        let proof_facts = if data.is_empty() {
            ProofFacts::default()
        } else {
            ProofFacts::deserialize_from(data)?
        };
        Some(Self {
            resource_bounds,
            tip,
            signature,
            nonce,
            sender_address,
            calldata,
            nonce_data_availability_mode,
            fee_data_availability_mode,
            paymaster_data,
            account_deployment_data,
            proof_facts,
        })
    }
