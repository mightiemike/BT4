### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables USDC-E Theft Without Providing USDC ŌĆö (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` uses a raw, unchecked `.transferFrom()` call to pull USDC from the caller before sending USDC-E back. If the USDC token returns `false` on failure instead of reverting, the function continues execution and transfers USDC-E to the caller for free. The function is callable by any unprivileged user.

---

### Finding Description

`replaceUsdcEWithUsdc` is a public, permissionless function (no `onlyOwner` modifier) that is intended to perform a 1:1 swap: the caller provides USDC, and receives USDC-E held in a `DirectDepositV1` contract.

The critical sequence is:

```solidity
// ContractOwner.sol lines 615ŌĆō618
if (balance > 0) {
    IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // ŌåÉ raw, unchecked
    DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
    IERC20Base(usdcE).safeTransfer(msg.sender, balance);                 // ŌåÉ safe
}
``` [1](#0-0) 

Line 616 calls `.transferFrom()` directly on the `IERC20Base` interface and **discards the return value**. If the USDC token at the hardcoded address returns `false` on failure (e.g., insufficient allowance or balance) rather than reverting, execution falls through to:

1. `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))` ŌĆö pulls all USDC-E from the DDA into `ContractOwner`.
2. `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` ŌĆö sends that USDC-E to the caller.

The rest of the codebase consistently uses the custom `ERC20Helper.safeTransferFrom` wrapper (which checks both the call success and the decoded return bool), and `ContractOwner` itself imports and uses `ERC20Helper` via `using ERC20Helper for IERC20Base` at line 24. Line 618 in the same `if`-block correctly uses `.safeTransfer`. The raw `.transferFrom` on line 616 is an isolated deviation from this pattern. [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**Asset delta**: An attacker drains USDC-E from any `DirectDepositV1` contract that holds a non-zero USDC-E balance, without providing any USDC in return. The stolen amount equals the full USDC-E balance of the targeted DDA.

**Broken invariant**: The swap is supposed to be atomic ŌĆö USDC in, USDC-E out. The unchecked `transferFrom` breaks this atomicity: the "USDC in" leg can silently fail while the "USDC-E out" leg executes unconditionally.

---

### Likelihood Explanation

The function is callable by any address on chain ID 57073 (Ink). The `subaccount` parameter is fully attacker-controlled, allowing targeting of any DDA with a USDC-E balance. Likelihood depends on whether the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink returns `false` on failure or reverts. Many non-standard or bridged ERC20 implementations (including historical USDT-style tokens) return `false` rather than reverting. Given the structural vulnerability and the permissionless entry path, likelihood is **medium-high** if the token follows a non-reverting failure pattern. [4](#0-3) 

---

### Recommendation

Replace the raw `.transferFrom()` call on line 616 with the `safeTransferFrom` method already available via `ERC20Helper`, consistent with every other transfer in the codebase:

```solidity
// Before (vulnerable):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [5](#0-4) 

---

### Proof of Concept

1. A `DirectDepositV1` DDA for some `subaccount` holds `N` USDC-E tokens.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` with zero USDC allowance granted to `ContractOwner`.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, N)` returns `false` (no revert) due to zero allowance.
4. Return value is ignored; execution continues.
5. `DirectDepositV1(directDepositV1).withdraw(usdcE)` transfers `N` USDC-E from the DDA to `ContractOwner`.
6. `IERC20Base(usdcE).safeTransfer(attacker, N)` sends `N` USDC-E to the attacker.
7. Attacker receives `N` USDC-E; no USDC was ever transferred. The DDA is drained. [6](#0-5)

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
