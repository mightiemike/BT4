### Title
Unprotected `initialize()` in `WithdrawPool` Can Be Front-Run, Granting Attacker Ownership and Enabling Full Pool Drain — (File: `core/contracts/WithdrawPool.sol`)

---

### Summary

`WithdrawPool.initialize()` is `external` with no access control and no `initializer` modifier. An attacker can front-run the deployer's initialization call, becoming the owner of `WithdrawPool` and setting `clearinghouse` and `verifier` to attacker-controlled addresses before the legitimate deployer can act.

---

### Finding Description

`WithdrawPool.initialize()` delegates entirely to `BaseWithdrawPool._initialize()`, which carries the `initializer` modifier and calls `__Ownable_init()` — setting `msg.sender` as the contract owner. [1](#0-0) 

```solidity
function initialize(address _clearinghouse, address _verifier) external {
    _initialize(_clearinghouse, _verifier);
}
``` [2](#0-1) 

```solidity
function _initialize(address _clearinghouse, address _verifier)
    internal
    initializer
{
    __Ownable_init();
    clearinghouse = _clearinghouse;
    verifier = _verifier;
}
```

Because `WithdrawPool.initialize()` itself has **no `initializer` modifier and no access control**, any external caller can invoke it before the legitimate deployer. The `initializer` modifier on `_initialize` only prevents a second call — it does not restrict who makes the first call.

`BaseWithdrawPool` uses `_disableInitializers()` in its constructor (indicating proxy deployment), meaning the proxy's storage is uninitialized until `initialize` is called in a separate transaction — creating the front-run window. [3](#0-2)
