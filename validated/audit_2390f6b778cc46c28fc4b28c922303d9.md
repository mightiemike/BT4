### Title
Fork Difficulty Desynchronization via Mainchain Height Lookup in Dogecoin DigiShield Retarget — (`contract/src/dogecoin.rs`)

---

### Summary

In the Dogecoin light client, the DigiShield every-block difficulty retarget (active after block 145,000) fetches the "first block" timestamp via `get_header_by_height`, which unconditionally reads from the **mainchain** height-to-hash mapping. When validating fork blocks that diverge more than one step from the mainchain, this returns a mainchain block's timestamp instead of the fork ancestor's timestamp. The difficulty validator and the fork submission path are therefore desynchronized: one reads mainchain state, the other processes fork-chain blocks. An unprivileged relayer can exploit this to submit fork blocks whose difficulty is validated against incorrect timestamps, potentially accepting a lower-than-required target and enabling a false chain reorganization.

---

### Finding Description

**Vulnerability class:** Cross-module desynchronization (direct analog to the reported `getPastTotalSupply` overestimation pattern).

**Root cause — `contract/src/dogecoin.rs`, `get_next_work_required`:**

After Dogecoin block 145,000, difficulty adjusts every block (`difficulty_adjustment_interval = 1`). For each new block, the retarget reads the timestamp of the block two steps back: [1](#0-0) 

```rust
let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
let difficulty_adjustment_interval = if new_difficulty_protocol { 1 } else { ... };
...
let height_first = prev_block_header.block_height.checked_sub(blocks_to_go_back)...;

// TODO: check if it is correct to get block header by height from mainchain
//       without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

The self-acknowledged `TODO` at line 291 flags the exact problem. `get_header_by_height` is implemented as: [2](#0-1) 

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // ← always mainchain
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

**The desynchronization:** When `submit_block_header` processes a fork block, `prev_block_header` is correctly fetched by hash (fork ancestor), but `get_header_by_height(height_first)` returns the **mainchain** block at that height — a different block with a different timestamp. [3](#0-2) 

Consider a fork diverging at height F:

| Height | Mainchain block | Fork block |
|--------|----------------|------------|
| F      | M_F            | M_F (same) |
| F+1    | M_{F+1}        | Fork_{F+1} (different timestamp) |
| F+2    | M_{F+2}        | Fork_{F+2} |

When validating Fork_{F+2}:
- `prev_block_header` = Fork_{F+1} ✓ (fetched by hash)
- `height_first` = F+1
- `get_header_by_height(F+1)` → **M_{F+1}** ✗ (should be Fork_{F+1})

The DigiShield formula computes:

```
modulated_timespan = retarget_timespan + (actual_time - retarget_timespan) / 8
```

where `actual_time = Fork_{F+2}.time - M_{F+1}.time` (wrong) instead of `Fork_{F+2}.time - Fork_{F+1}.time` (correct). [4](#0-3) 

If the attacker sets Fork_{F+1}.time to be much later than M_{F+1}.time, the computed `actual_time` becomes negative or very small, the modulated timespan is clamped to `min_timespan`, and the resulting required target is **easier** (higher) than it should be. The attacker then mines fork blocks at this reduced difficulty.

---

### Impact Explanation

The corrupted state is `ExtendedHeader.chain_work` stored in `headers_pool` for fork blocks. Chain work is computed from `block_header.bits`: [5](#0-4) 

If the difficulty check accepts a lower `bits` value (easier target) than the fork's true ancestor timestamps would require, the attacker mines blocks at that easier difficulty. Each such block contributes `work_from_bits(bits)` to `chain_work`. If the fork's cumulative `chain_work` exceeds the mainchain tip's `chain_work`, `submit_block_header_inner` triggers `reorg_chain`: [6](#0-5) 

This replaces the canonical mainchain with the attacker's fork. Downstream consumers calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` will then verify proofs against the attacker's fork, returning `true` for transactions that were never confirmed on the real Dogecoin chain. [7](#0-6) 

---

### Likelihood Explanation

- Requires no privileged role; `submit_blocks` is callable by any trusted relayer (or any account if the trusted-relayer gate is bypassed via `UnrestrictedSubmitBlocks`).
- Dogecoin's every-block retarget means the desynchronization activates at the **third** fork block (F+2), not after thousands of blocks as in Bitcoin.
- The attacker controls fork block timestamps (within the MTP and future-time constraints), giving direct control over the `actual_time` fed to the DigiShield formula.
- The codebase itself flags this with a `TODO` comment, confirming the developers identified the ambiguity.

---

### Recommendation

Replace the `get_header_by_height` call in `get_next_work_required` with an ancestor walk that follows `prev_block_hash` links from `prev_block_header` backward by `blocks_to_go_back` steps. This ensures fork difficulty is computed against the fork's own ancestor chain, not the mainchain's blocks at the same heights. The `headers_pool` already stores all submitted headers (mainchain and fork) keyed by hash, so the walk is feasible:

```rust
// Walk back blocks_to_go_back steps from prev_block_header via prev_block_hash
let mut cursor = prev_block_header.clone();
for _ in 0..blocks_to_go_back {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
let first_block_time = cursor.block_header.time;
```

This mirrors how `prev_block_header` itself is correctly resolved — by hash, not by mainchain height.

---

### Proof of Concept

1. Deploy Dogecoin light client initialized at height ≥ 145,001 (DigiShield active).
2. Let mainchain have blocks M_{H}, M_{H+1}, M_{H+2} with timestamps T, T+60, T+120.
3. Submit fork block Fork_{H+1} with `prev_block_hash = M_H.hash` and timestamp T+7200 (2 hours later, within `MAX_FUTURE_BLOCK_TIME_LOCAL`).
4. Submit fork block Fork_{H+2} with `prev_block_hash = Fork_{H+1}.hash` and timestamp T+7260.
   - Difficulty check: `actual_time = Fork_{H+2}.time - get_header_by_height(H+1).time = T+7260 - (T+60) = 7200s`
   - Correct value would be: `T+7260 - (T+7200) = 60s`
   - DigiShield with 7200s actual time → `modulated_timespan` clamped to `max_timespan` → **easier target** accepted.
5. Mine Fork_{H+2} at the accepted easier target (lower mining cost).
6. Repeat for subsequent fork blocks; each uses the mainchain timestamp at height-2, not the fork's.
7. Once fork `chain_work` exceeds mainchain `chain_work`, `reorg_chain` is triggered.
8. `verify_transaction_inclusion` now confirms attacker-chosen transactions as valid. [8](#0-7) [9](#0-8)

        &mut self,
        header: (Header, Option<AuxData>),
        skip_pow_verification: bool,
    ) {
        let (block_header, aux_data) = header;

        let prev_block_header = self.get_prev_header(&block_header);
        let current_block_hash = block_header.block_hash();

        if !skip_pow_verification {
            self.check_target(&block_header, &prev_block_header);

            if let Some(ref aux_data) = aux_data {
                self.check_aux(&block_header, aux_data);
            } else {
                let pow_hash = block_header.block_hash_pow();

                    U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),

                );
            }

            .chain_work
            .overflowing_add(work_from_bits(block_header.bits));

            block_hash: current_block_hash,
            chain_work: current_block_computed_chain_work,
            block_height: 1 + prev_block_header.block_height,
        };

        self.submit_block_header_inner(current_header, &prev_block_header);

fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,

    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))

fn calculate_next_work_required(
    config: &DogecoinConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let retarget_timespan = config.pow_target_timespan;
    let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;

    let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;

    let min_timespan = retarget_timespan - (retarget_timespan / 4);
    let max_timespan = retarget_timespan + (retarget_timespan / 2);

    if modulated_timespan < min_timespan {
        modulated_timespan = min_timespan;
    } else if modulated_timespan > max_timespan {
        modulated_timespan = max_timespan;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(modulated_timespan).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target =
        new_target / U256::from(<i64 as TryInto<u64>>::try_into(retarget_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()
