### Title
Linked Signer Can Unilaterally Escalate Signing Authority to an Attacker-Controlled Address — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction type in `EndpointTx.processTransactionImpl` is validated with `allowLinkedSigner = true`. This means the **currently linked signer** — not just the subaccount owner — can sign a new `LinkSigner` transaction to replace the linked signer with any arbitrary address. A malicious or compromised linked signer can therefore silently transfer full signing authority over a victim's subaccount to an attacker-controlled address, enabling collateral theft.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch calls `validateSignedTx` with `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which passes the current linked signer address to `Verifier.validateSignature`:

```solidity
// EndpointTx.sol lines 172–184
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

`Verifier.validateSignature` accepts the signature if the recovered address equals **either** the subaccount owner **or** the linked signer:

```solidity
// Verifier.sol lines 291–304
function validateSignature(...) public pure {
    address recovered = ECDSA.recover(digest, signature);
    require(
        (recovered != address(0)) &&
            ((recovered == address(uint160(bytes20(sender)))) ||
                (recovered == linkedSigner)),
        ERR_INVALID_SIGNATURE
    );
}
``` [3](#0-2) 

Therefore, the current linked signer can produce a valid `SignedLinkSigner` transaction that sets `signedTx.tx.signer` to any attacker-controlled address. After the sequencer processes this transaction, `linkedSigners[victim_subaccount]` is overwritten with the attacker's address.

The slow-mode path for `LinkSigner` does **not** share this flaw — it uses `validateSender`, which enforces that `msg.sender` must be the address embedded in the subaccount bytes32 (i.e., the owner):

```solidity
// EndpointTx.sol lines 232–239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    ...
    validateSender(txn.sender, sender);   // ← owner-only check
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The vulnerability is isolated to the sequencer-path `processTransactionImpl`.

---

### Impact Explanation

Once the attacker's address is installed as the linked signer, it is accepted as a valid signer for every transaction type that passes `allowLinkedSigner = true`, including:

- `WithdrawCollateral` / `WithdrawCollateralV2` — direct collateral theft to any `sendTo` address
- `TransferQuote` — drain quote balance to an attacker-controlled subaccount
- `LiquidateSubaccount` — self-liquidation abuse
- `MintNlp` / `BurnNlp` — NLP position manipulation [5](#0-4) 

The corrupted state is `linkedSigners[victim_subaccount]` in `EndpointStorage`, and the resulting asset delta is the full collateral balance of the victim subaccount. [6](#0-5) 

---

### Likelihood Explanation

Linking a signer is a standard, documented feature used for automated trading (hot wallets, bots). Any user who has ever called `LinkSigner` is exposed. The attack requires only that the linked signer is malicious or compromised — no admin access, no sequencer compromise, and no governance capture is needed. The attacker submits a single signed `LinkSigner` transaction through the normal sequencer path, which is a fully supported, unprivileged entry point.

---

### Recommendation

Change the `LinkSigner` branch in `processTransactionImpl` to pass `allowLinkedSigner = false`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
``` [7](#0-6) 

This aligns the sequencer path with the slow-mode path, which already enforces owner-only authorization for `LinkSigner` via `validateSender`.

---

### Proof of Concept

1. Alice owns subaccount `alice_sub` and has linked Bob's address as a signer: `linkedSigners[alice_sub] = bob`.
2. Bob constructs a `LinkSigner` struct: `{ sender: alice_sub, signer: attacker_address, nonce: current_nonce }`.
3. Bob signs the EIP-712 digest of this struct with his private key.
4. Bob (or anyone) submits the signed transaction to the sequencer.
5. The sequencer calls `submitTransactionsChecked`, which calls `processTransactionImpl`.
6. `validateSignedTx(..., true)` recovers Bob's address, matches it against `linkedSigners[alice_sub] == bob`, and passes.
7. `linkedSigners[alice_sub]` is overwritten with `attacker_address`.
8. The attacker signs a `WithdrawCollateralV2` transaction with `sendTo = attacker_eoa`, draining Alice's collateral. [1](#0-0) [8](#0-7)
