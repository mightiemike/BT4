### Title
Duplicate Position ID in `ClaimTierRewards` Enables Double-Spending of Bonus Rewards from the Rewards Pool — (`File: x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`ClaimTierRewards` loads all requested positions into a slice before any processing begins. If the same position ID appears more than once in `msg.PositionIds`, two copies of the same pre-claim position state are placed in the slice. When `claimRewardsAndUpdatesPositions` iterates over the slice, the second copy retains the original (pre-claim) `LastEventSeq` and `LastBonusAccrual` checkpoints, causing `processEventsAndClaimBonus` to recompute and pay out the same bonus rewards a second time, draining the `RewardsPoolName` module account beyond what is owed.

---

### Finding Description

In `ClaimTierRewards`, positions are loaded into a local slice in a first loop, then processed in a second call:

```go
// x/tieredrewards/keeper/msg_server.go lines 434–448
positions := make([]types.PositionState, 0, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    pos, err := ms.getPositionState(ctx, posId)
    ...
    positions = append(positions, pos)
}
totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
``` [1](#0-0) 

If `msg.PositionIds = [X, X]`, two copies of position `X` — both with the same `LastEventSeq` and `LastBonusAccrual` — are appended to the slice before any processing occurs.

Inside `claimRewardsAndUpdatesPositions`, each element is processed independently:

```go
// x/tieredrewards/keeper/claim_rewards.go lines 110–133
for i := range positions {
    pos := &positions[i]
    ...
    bonus, err := k.processEventsAndClaimBonus(ctx, pos)
    ...
    if err := k.setPosition(ctx, pos.Position, nil); err != nil { ... }
}
``` [2](#0-1) 

On the **first** iteration, `processEventsAndClaimBonus` reads `pos.LastEventSeq` from the in-memory copy, walks all pending validator events, pays out bonus coins, advances the checkpoint, and saves the updated position to the store.

On the **second** iteration, the second in-memory copy still holds the **original** `LastEventSeq` (loaded before the first iteration ran). `getValidatorEventsSince` is called again with the same old sequence number, the same events are replayed, and `bankKeeper.SendCoinsFromModuleToAccount` pays out the same bonus a second time:

```go
// x/tieredrewards/keeper/claim_rewards.go lines 153–198
events, err := k.getValidatorEventsSince(ctx, valAddr, pos.LastEventSeq)
...
for _, entry := range events {
    ...
    bonus := k.computeSegmentBonus(...)
    totalBonus = totalBonus.Add(bonus)
    ...
    k.decrementEventRefCount(ctx, valAddr, entry.Seq)
}
``` [3](#0-2) 

```go
// x/tieredrewards/keeper/claim_rewards.go lines 239–241
if err := k.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.RewardsPoolName, ownerAddr, bonusCoins); err != nil {
    return nil, err
}
``` [4](#0-3) 

A secondary corruption also occurs: `decrementEventRefCount` is called for each event on both iterations, so the reference count for each event is decremented twice. If the count reaches zero prematurely, the event is garbage-collected and other positions that have not yet processed it will silently skip it, permanently losing their bonus entitlement.

---

### Impact Explanation

- **Direct**: An attacker with any tier position can multiply their bonus reward payout by N by submitting the same position ID N times in a single `MsgClaimTierRewards` transaction. The `RewardsPoolName` module account is drained at N× the legitimate rate.
- **Secondary**: Event reference counts are decremented N times per event, causing premature garbage collection of validator events and silent loss of bonus rewards for all other positions that share the same validator.

The corrupted values are: the `RewardsPoolName` module account balance (unauthorized outflow) and the per-validator event reference counts (corrupted accounting).

---

### Likelihood Explanation

The entry path is a standard, unprivileged `MsgClaimTierRewards` transaction signed by any position owner. No special role, leaked key, or social engineering is required. The attacker only needs to own one position with accrued bonus rewards and submit `PositionIds = [id, id, id, ...]`. The attack is repeatable every block.

---

### Recommendation

Deduplicate `msg.PositionIds` before loading positions, or add an explicit check in `msg.Validate()` that rejects duplicate IDs. A simple approach:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    if _, dup := seen[posId]; dup {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest, "duplicate position id: %d", posId)
    }
    seen[posId] = struct{}{}
    ...
}
```

Alternatively, process and persist each position immediately inside the loading loop rather than batching into a slice, so the second occurrence reads the already-updated state from the store.

---

### Proof of Concept

1. Attacker owns position ID `42` delegated to a bonded validator with 100 BASECRO of accrued bonus rewards in the pool.
2. Attacker submits `MsgClaimTierRewards{Owner: attacker, PositionIds: [42, 42, 42]}`.
3. `ClaimTierRewards` loads position `42` three times into the `positions` slice; all three copies share the same `LastEventSeq = S`.
4. First iteration: `processEventsAndClaimBonus` processes events since `S`, pays 100 BASECRO, saves position with `LastEventSeq = S'`.
5. Second iteration: uses the stale copy with `LastEventSeq = S`, replays the same events, pays another 100 BASECRO.
6. Third iteration: same — pays another 100 BASECRO.
7. Attacker receives 300 BASECRO instead of 100 BASECRO; the rewards pool is short by 200 BASECRO.
8. Event reference counts for events `[S+1 … S']` are each decremented 3 times instead of once, potentially triggering premature deletion and silently zeroing out bonus rewards for all other positions on the same validator.
