### Title
Unpermissioned `initialize()` in `BaseProxyManager` Allows Front-Running to Seize Full Protocol Upgrade Authority — (`core/contracts/BaseProxyManager.sol`)

---

### Summary

`BaseProxyManager.initialize()` carries no caller restriction. Any address that calls it before the legitimate deployer becomes the `owner` and `submitter` of the ProxyManager, which is the single contract that controls implementation upgrades for every proxy in the Nado protocol. A successful front-run grants an attacker the ability to upgrade all protocol proxies to arbitrary malicious code.

---

### Finding Description

`BaseProxyManager.initialize()` is declared `external initializer` with no permission guard:

```solidity
function initialize() external initializer {
    __Ownable_init();
    submitter = msg.sender;
    proxyManagerHelper = new ProxyManagerHelper();
}
```

`__Ownable_init()` sets `msg.sender` as `owner`. `submitter = msg.sender` sets the same caller as the address authorized to propose new implementation hashes. The `ProxyManagerHelper` is also constructed with `proxyManager = msg.sender` (its constructor), so the helper's `onlyOwner` modifier also resolves to the attacker. [1](#0-0) 

Compare this with `ContractOwner.initialize()`, which explicitly enforces `require(_deployer == msg.sender, "expected deployed to initialize")` — a pattern that is absent from `BaseProxyManager`. [2](#0-1) 

The constructor correctly calls `_disableInitializers()` to protect the bare implementation, but this does not protect the proxy's storage from a front-run on the first `initialize()` call. [3](#0-2) 

Once an attacker owns the ProxyManager, the following privileged functions become attacker-controlled:

- `migrateAll(NewImpl[])` — upgrades all registered proxies (Clearinghouse, Endpoint, SpotEngine, PerpEngine, OffchainExchange, Verifier) to attacker-supplied implementations.
- `forceMigrateSelf(address)` — upgrades the ProxyManager proxy itself.
- `bootstrapEndpointTx(...)` — the `reinitializer(2)` function, gated by `onlyOwner`, becomes callable by the attacker. [4](#0-3) [5](#0-4) 

The `ProxyManagerHelper` is also constructed under attacker control, meaning `upgradeClearinghouseLiq` and `upgradeEndpointTx` — which are gated by `onlyOwner` on the helper — are also attacker-controlled. [6](#0-5) 

---

### Impact Explanation

An attacker who seizes ProxyManager ownership can call `migrateAll()` to point every protocol proxy at a malicious implementation. This enables:

- Arbitrary modification of `Clearinghouse` logic to redirect `withdrawCollateral` transfers to the attacker.
- Arbitrary modification of `Endpoint` logic to bypass signature verification and drain subaccount balances.
- Arbitrary modification of `SpotEngine`/`PerpEngine` to corrupt all balance accounting.

The corrupted state delta is: **all ERC-20 collateral held by Clearinghouse and WithdrawPool becomes drainable by the attacker**. The `insurance` balance, all spot balances, and all perp vQuote balances are at risk. [7](#0-6) 

---

### Likelihood Explanation

The attack requires only mempool observation and a higher-gas-price transaction submitted before the deployer's `initialize()` call. This is a well-documented, mechanically straightforward attack. The window exists for any block between proxy deployment and the `initialize()` call. On Ink Chain (an EVM-compatible L2), block times are short but the mempool is still observable. No privileged access, leaked keys, or social engineering is required — only a standard front-run.

---

### Recommendation

Add a deployer address check to `BaseProxyManager.initialize()`, mirroring the pattern already used in `ContractOwner`:

```solidity
function initialize(address _expectedDeployer) external initializer {
    require(msg.sender == _expectedDeployer, "unauthorized initializer");
    __Ownable_init();
    submitter = msg.sender;
    proxyManagerHelper = new ProxyManagerHelper();
}
```

Alternatively, ensure the proxy deployment and `initialize()` call are executed atomically in a single transaction (e.g., via a factory or `deployAndInit` script), so no block gap exists between proxy creation and initialization.

---

### Proof of Concept

1. Deployer broadcasts a transaction to deploy the `ProxyManager` transparent proxy and call `initialize()`.
2. Attacker observes the pending deployment in the mempool.
3. Attacker submits `ProxyManager.initialize()` with a higher gas price, executing first.
4. Attacker is now `owner` and `submitter`; `proxyManagerHelper.proxyManager == attacker`.
5. Deployer's `initialize()` reverts (already initialized).
6. Attacker calls `registerRegularProxy(...)` (or waits for the deployer to register proxies, since `registerRegularProxy` is `onlyOwner` — now the attacker).
7. Attacker calls `submitImpl("Clearinghouse", maliciousImpl)` as `submitter`.
8. Attacker calls `migrateAll([{name: "Clearinghouse", impl: maliciousImpl}, ...])` as `owner`.
9. All proxies now delegate to attacker-controlled implementations.
10. Attacker drains all user collateral from Clearinghouse. [1](#0-0) [8](#0-7)
