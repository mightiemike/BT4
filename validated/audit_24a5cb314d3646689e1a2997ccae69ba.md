### Title
Linked Signer Can Self-Perpetuate Authority by Re-Linking to Attacker-Controlled Address — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction in `EndpointTx.sol` is validated with `allowLinkedSigner = true`, meaning the **currently registered linked signer** can sign a `LinkSigner` transaction to overwrite the linked signer mapping for a subaccount. This is directly analogous to the Blex `onlyInitOr` bypass: a delegated authority (linked signer) can use its own delegated power to mutate the very permission that grants it authority — without the subaccount owner's consent.

---

### Finding Description

In `EndpointTx.sol`, the `LinkSigner` transaction type is processed as follows:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

The `validateSignedTx` call with `allowLinkedSigner = true` routes into `validateSignature`, which accepts a signature from either the subaccount owner address or the currently registered linked signer:

```solidity
function validateSignature(...) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

And in `Verifier.sol`, the check accepts either the owner or the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

The result: the **current linked signer** can sign a valid `LinkSigner` transaction that replaces `linkedSigners[subaccount]` with any arbitrary address — including an attacker-controlled one — without the subaccount owner's knowledge or consent.

---

### Impact Explanation

A linked signer has broad authority over a subaccount. It can sign `WithdrawCollateral`, `WithdrawCollateralV2`, `LiquidateSubaccount`, and `TransferQuote` transactions — all of which are validated with `allowLinkedSigner = true`. By re-linking to an attacker-controlled address, the attacker permanently acquires full signing authority over the subaccount and can drain all collateral via `WithdrawCollateral` or `WithdrawCollateralV2`.

The broken invariant: only the subaccount owner (the address embedded in the `sender` bytes32) should be able to change who the linked signer is. The current design allows the linked signer itself to mutate this mapping, which is an unauthorized subaccount mutation with direct asset impact.

---

### Likelihood Explanation

Any scenario where a linked signer key is compromised — leaked API key, compromised trading bot, malicious third-party integration — immediately escalates from "temporary signing access" to "permanent account takeover." The attacker only needs to submit one `LinkSigner` transaction through the sequencer before the owner can revoke. Because the sequencer processes transactions in order, the attacker can also front-run an owner revocation by submitting a re-link first.

---

### Recommendation

Change the `LinkSigner` transaction validation to use `allowLinkedSigner = false`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This ensures that only the subaccount owner (the address whose bytes are embedded in `sender`) can authorize a change to the linked signer, preventing a delegated key from perpetuating or transferring its own authority.

---

### Proof of Concept

1. Alice owns subaccount `S` and sets Bob's address as the linked signer via a legitimate `LinkSigner` transaction.
2. Bob's key is compromised by Mallory.
3. Mallory uses Bob's key to sign a new `LinkSigner` transaction: `{ sender: S, signer: Mallory, nonce: N }`.
4. The sequencer processes this transaction. `validateSignedTx` with `allowLinkedSigner = true` accepts Mallory's signature (recovered as Bob = current `linkedSigners[S]`).
5. `linkedSigners[S]` is now set to Mallory's address.
6. Mallory signs a `WithdrawCollateral` transaction as the new linked signer and drains Alice's collateral.
7. Alice's attempt to revoke Bob by submitting her own `LinkSigner` transaction is either front-run or arrives after step 5, leaving Mallory in control. [1](#0-0) [2](#0-1) [5](#0-4)
