Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks LP Position `owner` Instead of Token-Paying `sender`, Allowing Any Address to Bypass the Deposit Gate — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual token payer, i.e., `msg.sender` of `addLiquidity`) and only validates `owner` (the arbitrary LP position recipient). Because `addLiquidity` accepts a caller-supplied `owner` and pulls tokens from `msg.sender` via callback, any non-allowlisted address can pass the gate by naming an allowlisted address as `owner`. The deposit allowlist — the sole access-control mechanism for permissioned pools — is fully bypassed with no privilege required.

## Finding Description

`MetricOmmPool.addLiquidity` passes two distinct actors into the `_beforeAddLiquidity` hook:

- `msg.sender` (the `sender` argument) — the address that calls `addLiquidity` and pays tokens through `metricOmmModifyLiquidityCallback`
- `owner` — a free, caller-supplied address that receives the LP position [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but leaves it unnamed and never reads it. The allowlist check is performed exclusively against `owner`: [2](#0-1) 

Because `owner` is a free parameter chosen by the caller, any address can pass the check by supplying an allowlisted address as `owner`. The actual token payer (`sender`) is never validated.

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, ...)` makes this trivially accessible: `msg.sender` is always the payer while `owner` is a free parameter: [3](#0-2) 

**Exploit path:**
1. Pool admin deploys a permissioned pool with `DepositAllowlistExtension` and allowlists only `alice`.
2. Non-allowlisted `attacker` calls `pool.addLiquidity(owner=alice, salt, deltas, callbackData, extensionData)`.
3. Pool calls `_beforeAddLiquidity(msg.sender=attacker, owner=alice, ...)` → extension checks `allowedDepositor[pool][alice]` → `true` → hook returns valid selector.
4. Pool proceeds; `attacker` pays tokens through the callback; `alice` receives the LP position.
5. Non-allowlisted capital is now inside the permissioned pool. The `NotAllowedToDeposit` revert path is never reached.

Existing guards are insufficient: the only guard is the `allowedDepositor[msg.sender][owner]` check in the extension, and `msg.sender` there is the pool (correct), but `owner` is the wrong actor to gate — it should be `sender`.

## Impact Explanation

The deposit allowlist is the primary access-control mechanism for permissioned/KYC pools. Bypassing it allows any address to inject liquidity into a pool that is supposed to be restricted. Non-allowlisted capital enters the pool's token reserves (`binTotals.scaledToken0`/`scaledToken1`), permanently corrupting the pool's accounting of authorized participants. This constitutes a broken core pool functionality and an admin-boundary break: the pool admin's configured access control is bypassed by an unprivileged path with no special privilege required.

**Severity: High** — the core invariant of the extension (only allowlisted depositors can add liquidity) is completely broken.

## Likelihood Explanation

**Likelihood: Medium.** The attack requires no special privilege — any EOA or contract can call `addLiquidity` with an arbitrary `owner`. The only cost to the attacker is the tokens deposited (which go to the allowlisted `owner`'s LP position). The `MetricOmmPoolLiquidityAdder` makes the pattern trivially accessible via `addLiquidityExactShares(pool, allowlisted_address, ...)`. The attack is repeatable and requires no setup beyond knowing an allowlisted address (which is on-chain public state via `allowedDepositor`).

## Recommendation

Replace the `owner` check with a `sender` check in `beforeAddLiquidity`. The `sender` is the actual token provider and the entity whose access should be gated:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

Pool admins who intentionally want to gate by position owner (not token provider) should use a separate, explicitly documented extension.

## Proof of Concept

```solidity
// Setup: pool with DepositAllowlistExtension; only alice is allowlisted
depositAllowlist.setAllowedToDeposit(pool, alice, true);

// Attacker (not allowlisted) deposits on behalf of alice
vm.prank(attacker);
pool.addLiquidity(
    alice,        // owner = allowlisted → check passes
    salt,
    deltas,
    callbackData, // attacker pays tokens here via callback
    extensionData
);

// Result: attacker's tokens are in the pool; allowlist was never enforced on the actual payer
// beforeAddLiquidity received sender=attacker but never read it — NotAllowedToDeposit never reverts
assertGt(pool.positionShares(alice, salt, binIdx), 0);
// attacker's token balance decreased — non-allowlisted capital is now inside the permissioned pool
```

The `beforeAddLiquidity` hook receives `sender=attacker` as its first argument but the parameter is unnamed and discarded at line 32, so the revert path `NotAllowedToDeposit` is never reached. [4](#0-3)
