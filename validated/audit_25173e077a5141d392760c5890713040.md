### Title
`Role::UnrestrictedSubmitBlocks` Cannot Bypass Pause on `submit_blocks` Despite Being Documented To Do So - (`contract/src/lib.rs`)

### Summary

The `Role::UnrestrictedSubmitBlocks` role is explicitly documented to allow callers to use the contract API even when the contract is paused. However, `submit_blocks` is decorated with a bare `#[pause]` macro that carries no `except` clause, so the role has zero effect on the pause gate. The parallel role `UnrestrictedRunGC` is correctly wired via `#[pause(except(roles(Role::UnrestrictedRunGC)))]` on `run_mainchain_gc`, proving the pattern is known and intentional — it was simply omitted for `submit_blocks`. This is a direct "function incorrect as to spec" defect that silently voids the documented emergency-bypass capability.

### Finding Description

The `Role` enum documents `UnrestrictedSubmitBlocks` as follows: [1](#0-0) 

```rust
/// Allows to use contract API even after contract is paused
UnrestrictedSubmitBlocks,
```

The `trusted_relayer` macro lists this role under `bypass_roles`: [2](#0-1) 

```rust
#[trusted_relayer(
    bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks),
    ...
)]
```

`bypass_roles` in `trusted_relayer` only bypasses the **relayer whitelist** check — it has no effect on the `#[pause]` gate. The `submit_blocks` function is decorated with a bare `#[pause]`: [3](#0-2) 

```rust
#[payable]
#[pause]
#[trusted_relayer]
pub fn submit_blocks(
    &mut self,
    #[serializer(borsh)] headers: Vec<BlockHeader>,
) -> PromiseOrValue<()> {
```

There is no `except(roles(Role::UnrestrictedSubmitBlocks))` argument on `#[pause]`. Compare this to `run_mainchain_gc`, which correctly implements the bypass for its counterpart role: [4](#0-3) 

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
```

The `near-plugins` `Pausable` macro only exempts callers from the pause gate when the role is listed in the `except(roles(...))` argument of the specific `#[pause]` attribute on that function. Because `submit_blocks` carries `#[pause]` with no `except` clause, any call to `submit_blocks` while the contract is paused will be rejected — regardless of whether the caller holds `UnrestrictedSubmitBlocks`.

### Impact Explanation

The `UnrestrictedSubmitBlocks` role exists precisely to keep the light client synchronized during an emergency pause. If a `PauseManager` pauses the contract (e.g., to halt a malicious relayer), the intended recovery path is:

1. Pause the contract to stop the malicious relayer.
2. Use `UnrestrictedSubmitBlocks`-grantees to continue submitting honest headers, keeping the canonical chain current.
3. Unpause once the threat is resolved.

Because step 2 is silently broken, a pause completely halts all block submission — including from trusted emergency operators. The canonical chain tracked by the contract stops advancing, and any downstream consumer relying on `verify_transaction_inclusion` or `get_last_block_header` will operate on a stale chain for the entire duration of the pause. The longer the pause, the larger the gap between the on-chain tip and the real Bitcoin tip, and the greater the window for stale-proof acceptance by consumers.

### Likelihood Explanation

The `PauseManager` role is a normal operational role. Any legitimate emergency (malicious relayer, bug discovery) that triggers a pause immediately activates this defect. No attacker action is required — the broken invariant is triggered by the protocol's own emergency procedure. The `UnrestrictedSubmitBlocks` role is presumably granted to trusted operators at deployment; those operators will discover the bypass does not work only at the moment it is needed most.

### Recommendation

Change the `#[pause]` attribute on `submit_blocks` to include the exception clause, mirroring the pattern already used on `run_mainchain_gc`:

```diff
-    #[pause]
+    #[pause(except(roles(Role::UnrestrictedSubmitBlocks)))]
     #[trusted_relayer]
     pub fn submit_blocks(
``` [5](#0-4) 

### Proof of Concept

1. Deploy the contract with `skip_pow_verification = true`.
2. Grant account `alice` the `UnrestrictedSubmitBlocks` role via `acl_grant_role`.
3. Grant account `bob` the `PauseManager` role.
4. Call `pa_pause` (or the equivalent `Pausable` pause entry-point) from `bob` to pause the contract.
5. Call `submit_blocks` from `alice` with a valid header.
6. **Observed**: the call panics with the `Pausable` "contract is paused" error, even though `alice` holds `UnrestrictedSubmitBlocks`.
7. **Expected**: the call succeeds, because the role is documented to bypass the pause.

For contrast, repeat steps 4–6 using `run_mainchain_gc` from an account holding `UnrestrictedRunGC` — that call succeeds, confirming the `except` pattern works and the omission on `submit_blocks` is the sole root cause.
