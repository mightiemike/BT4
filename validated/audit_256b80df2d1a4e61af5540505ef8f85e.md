Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Gates on the Direct Pool Caller (Router) Instead of the End-User, Allowing Any Address to Bypass a Curated Pool's Swap Allowlist â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter, which the pool sets to its own `msg.sender` â€” the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the router is that direct caller, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][end-user]`. A pool admin who allowlists the router to enable their vetted users to access multi-hop routing inadvertently grants unrestricted swap access to every address on-chain, completely defeating the allowlist's curation purpose.

## Finding Description

**Root cause â€” wrong identity checked:**

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then uses `msg.sender` (the pool) as the mapping key and the forwarded `sender` (the direct caller of the pool) as the swapper identity: [3](#0-2) 

**Router as direct caller:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the pool's `msg.sender`: [4](#0-3) 

The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

**Consequence of allowlisting the router:**

If a pool admin wants individually allowlisted users (e.g., KYC'd addresses) to use the router for multi-hop swaps, they must call `setAllowedToSwap(pool, router, true)`. Once they do, the check degenerates to `allowedSwapper[pool][router] == true`, which is satisfied for **every** caller regardless of their individual allowlist status. Any non-allowlisted address can call `router.exactInputSingle(...)` and execute swaps against the curated pool.

**Contrast with `DepositAllowlistExtension`:**

`DepositAllowlistExtension.beforeAddLiquidity` correctly gates on `owner` (the economic actor / position owner), ignoring `sender` (the operational caller): [6](#0-5) 

This confirms the inconsistency: the deposit extension correctly identifies the economic actor; the swap extension does not.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to KYC'd or otherwise vetted counterparties is completely bypassed the moment the router is allowlisted. Any unprivileged address can execute swaps against the curated pool by routing through `MetricOmmSimpleRouter`, draining LP value or violating the pool's curation policy. LPs who deposited under the assumption that only vetted counterparties could trade face direct loss of principal and fee revenue from unvetted traders. This is a broken core pool functionality / admin-boundary break with direct fund-impacting consequences.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the canonical user-facing entry point for swaps. Pool admins who configure an allowlist and also want their vetted users to access multi-hop routing or native-ETH wrapping will naturally allowlist the router â€” there is no other mechanism to permit router-mediated swaps for individually allowlisted users. The bypass requires no special privileges, no malicious setup, and no non-standard tokens: only a call through the standard periphery router. The precondition (router allowlisted) is a foreseeable and likely operational choice.

## Recommendation

Gate the allowlist on the **end-user identity** rather than the direct pool caller. Two approaches:

1. **Check `recipient` instead of `sender`** â€” `recipient` is the address that receives swap output and is set by the end-user even when routing through the router. This is the minimal change and mirrors how `DepositAllowlistExtension` uses `owner`.
2. **Pass the end-user through `extensionData`** â€” require the router to encode the originating user in `extensionData` and have the extension decode and verify it. This is more robust for multi-hop paths where `recipient` may be an intermediate address.

Additionally, align `SwapAllowlistExtension` with `DepositAllowlistExtension`, which already correctly gates on the economic actor (`owner`) rather than the operational caller (`sender`).

## Proof of Concept

```
1. Pool admin deploys a pool with SwapAllowlistExtension configured.
2. Admin allowlists alice (a KYC'd user):
       setAllowedToSwap(pool, alice, true)
3. Admin allowlists the router so alice can use multi-hop routing:
       setAllowedToSwap(pool, router, true)
4. Bob (not KYC'd, not individually allowlisted) calls:
       router.exactInputSingle({pool: pool, recipient: bob, ...})
5. The router calls pool.swap(...). The pool passes msg.sender = router as `sender`
   to the extension.
6. The extension evaluates:
       allowedSwapper[pool][router] == true  â†’  passes
7. Bob's swap executes against the curated pool, bypassing the allowlist entirely.
```

A Foundry test can reproduce this by:
- Deploying `SwapAllowlistExtension` and a pool configured to use it
- Allowlisting a mock router address via `setAllowedToSwap`
- Calling `pool.swap(...)` with `msg.sender` spoofed to the router address and an arbitrary non-allowlisted `sender` argument
- Confirming the call succeeds despite the non-allowlisted end-user

The root cause is at `SwapAllowlistExtension.sol` L37, where `sender` is the router address whenever the periphery router is the direct caller of `pool.swap()`. [7](#0-6)
