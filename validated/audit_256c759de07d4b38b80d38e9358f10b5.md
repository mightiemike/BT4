### Title
Any Unprivileged Caller Can Trigger Garbage Collection and Permanently Delete Mainchain Block Headers - (File: `contract/src/lib.rs`)

### Summary
`run_mainchain_gc` is a public, state-mutating function that permanently deletes mainchain block headers from contract storage. It carries only a `#[pause]` guard, which blocks calls only when the contract is paused. When the contract is running normally, any unprivileged NEAR account can call it with an arbitrary `batch_size`, forcing the contract to delete as many historical headers as the GC threshold allows. This is a direct analog to the `extendTime` missing-`onlyOwner` bug: a privileged administrative operation is exposed to any caller with no role check.

### Finding Description
`run_mainchain_gc` is declared `pub` and decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`: [1](#0-0) 

The `#[pause]` macro only rejects calls when the contract is in a paused state. It does not restrict who may call the function when the contract is live. There is no `access_control` role check, no `#[trusted_relayer]` guard, and no `#[private]` attribute on this function.

By contrast, `submit_blocks` — the other state-mutating entry point — carries both `#[pause]` and `#[trusted_relayer]`, which enforces a whitelist of authorized relayers: [2](#0-1) 

`run_mainchain_gc` internally calls `remove_block_header`, which permanently removes entries from both `mainchain_header_to_height` and `headers_pool`: [3](#0-2) 

It also removes entries from `mainchain_height_to_header` and advances `mainchain_initial_blockhash` to the new oldest block: [4](#0-3) 

These deletions are irreversible. Once a header is removed from `headers_pool`, it cannot be recovered without re-submitting the block.

### Impact Explanation
An attacker calls `run_mainchain_gc(u64::MAX)` from any NEAR account. The function caps the removal at `total_amount_to_remove = amount_of_headers_we_store - gc_threshold`, so it will delete every header older than the GC threshold in a single transaction: [5](#0-4) 

Concrete consequences:
1. **Transaction inclusion proofs break**: `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` look up the block header in `headers_pool` by `tx_block_blockhash`. If that header was GC'd, the call panics with `ERR_KEY_NOT_EXIST`, permanently invalidating SPV proofs for all deleted blocks.
2. **Chain reorg breaks**: The `reorg_chain` function walks back through `headers_pool` to find the common ancestor. If GC has deleted those headers, it panics with `PrevBlockNotFound`, as explicitly documented in `CLAUDE.md`: [6](#0-5) 

3. **`mainchain_initial_blockhash` is corrupted**: The attacker advances this pointer forward, permanently losing the ability to reference or verify any block before the new initial hash.

### Likelihood Explanation
The entry point is a standard NEAR contract call requiring no deposit, no special role, and no staking. Any NEAR account can call it at any time the contract is not paused. The attack is a single transaction. The only precondition is that the mainchain must have grown beyond `gc_threshold`, which is the normal operating state of a live light client (the recommended `gc_threshold` is 52,704 blocks, approximately one year of Bitcoin blocks).

### Recommendation
Add a role-based access control check to `run_mainchain_gc` so that only authorized accounts (e.g., `Role::DAO` or a dedicated `GCManager` role) can invoke it directly. The simplest fix mirrors the pattern already used for `submit_blocks`: add an `access_control` attribute or an explicit `acl_is_caller_role` check at the top of the function. The `#[pause(except(roles(Role::UnrestrictedRunGC)))]` decorator should be retained alongside the new role restriction.

```rust
// Recommended fix
#[access_control_any(roles(Role::DAO, Role::UnrestrictedRunGC))]
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) { ... }
```

The internal call from `submit_blocks` (line 181) is a private Rust method call and is unaffected by the NEAR-level access control macro.

### Proof of Concept
```
// Any unprivileged NEAR account executes this single transaction:
let outcome = attacker_account
    .call(contract.id(), "run_mainchain_gc")
    .args_json(json!({ "batch_size": u64::MAX }))
    .transact()
    .await?;
// outcome.is_success() == true

// Now any SPV proof for a GC'd block fails:
let proof_outcome = any_account
    .call(contract.id(), "verify_transaction_inclusion_v2")
    .args_borsh(proof_for_old_block)
    .transact()
    .await?;
// Panics: "cannot find requested transaction block" (ERR_KEY_NOT_EXIST)
// mainchain_initial_blockhash has been advanced; historical state is gone.
```
