### Title
`ValidResourceBounds` Protobuf Round-Trip Silently Mutates `AllResources` to `L1Gas`, Producing a Wrong Transaction Hash on Syncing Nodes - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` implementation uses a value-based heuristic — if both `l2_gas` and `l1_data_gas` are zero, it returns `ValidResourceBounds::L1Gas` — to reconstruct the variant after deserialization. This silently converts `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` into `L1Gas(X)` across a protobuf round-trip. Because `get_tip_resource_bounds_hash` produces a structurally different hash preimage for these two variants (3 resource elements vs. 2), the transaction hash computed at the gateway diverges from the hash recomputed by any component that processes the deserialized form — including SNOS re-execution and P2P-syncing nodes.

---

### Finding Description

**Root cause — protobuf deserialization heuristic:** [1](#0-0) 

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

The variant is chosen by inspecting the *values* of the deserialized fields, not by any explicit tag. A transaction submitted with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` serializes to protobuf as `{ l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` and deserializes back as `L1Gas(X)` — a different variant.

**Hash preimage divergence:**

`get_tip_resource_bounds_hash` branches on the variant: [2](#0-1) 

- `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` → Poseidon over `[tip, L1_GAS(X), L2_GAS(0), L1_DATA_GAS(0)]` — **3 resource elements**
- `L1Gas(X)` → Poseidon over `[tip, L1_GAS(X), L2_GAS(0)]` — **2 resource elements, L1_DATA_GAS absent**

These produce different field elements, and since `tip_resource_bounds_hash` feeds directly into the full transaction hash: [3](#0-2) 

the full transaction hash H₁ (computed at the gateway from `AllResources`) differs from H₂ (recomputed from the deserialized `L1Gas` form).

**Gateway always produces `AllResources` hash:**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds` (not `ValidResourceBounds`), so the gateway always computes H₁ via the `AllResources` path: [4](#0-3) [5](#0-4) 

**P2P sync path uses `InvokeTransactionV3` with `ValidResourceBounds`:**

The P2P state-sync path deserializes blocks using `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3`, which calls the broken `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`: [6](#0-5) 

A syncing node therefore stores `resource_bounds = L1Gas(X)` alongside the block-committed hash H₁.

**JSON storage round-trip is consistent (no confusion there):**

The JSON `Deserialize` for `ValidResourceBounds` uses key *presence* (`L1DataGas` key present → `AllResources`), so the storage path preserves the variant correctly: [7](#0-6) 

This means the sequencer node (which reads from its own DB) and a P2P-syncing node see different `resource_bounds` variants for the same transaction.

---

### Impact Explanation

**High — RPC returns an authoritative-looking wrong value / transaction conversion binds the wrong hash.**

A P2P-syncing node stores `resource_bounds = L1Gas(X)` with `tx_hash = H₁`. When it serves the transaction via RPC, any client that recomputes the hash from the returned fields gets H₂ ≠ H₁ — the hash and the transaction body are inconsistent. This is an authoritative-looking wrong value from the RPC layer.

Additionally, the SNOS re-execution path recomputes the transaction hash from the transaction fields it receives. If the SNOS is fed the deserialized form (e.g., from a syncing node's DB), it computes H₂ and the signature verification against H₁ fails, causing re-execution to abort for a transaction that was validly sequenced.

---

### Likelihood Explanation

Any unprivileged user can trigger this by submitting a V3 invoke transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` (non-zero L1 gas, zero L2 and data gas). The gateway's stateless validator accepts it because `max_possible_fee > 0`: [8](#0-7) 

No special privilege, no malformed bytes, no rejected input — the transaction is valid and will be sequenced.

---

### Recommendation

In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, determine the variant by the *presence* of the `l1_data_gas` field in the protobuf message, not by whether its value is zero. When `l1_data_gas` is present (even if zero), reconstruct `AllResources`; only fall back to `L1Gas` when `l1_data_gas` is absent (for genuine pre-0.13.3 messages). This mirrors the JSON deserialization logic in `TryFrom<DeprecatedResourceBoundsMapping>` which already uses key presence as the discriminant. [1](#0-0) 

---

### Proof of Concept

1. Craft a V3 invoke transaction:
   ```
   resource_bounds = AllResourceBounds {
       l1_gas:      ResourceBounds { max_amount: 1000, max_price_per_unit: 1 },
       l2_gas:      ResourceBounds::default(),   // zero
       l1_data_gas: ResourceBounds::default(),   // zero
   }
   ```
2. Submit to the gateway. The gateway computes H₁ via `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` with `AllResources` → preimage includes `L1_DATA_GAS(0)`. User signs H₁.
3. Transaction is sequenced and included in a block. The block is broadcast via P2P protobuf.
4. A syncing node deserializes the transaction: `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` sees `l2_gas = 0, l1_data_gas = 0` → returns `L1Gas(1000/1)`.
5. The syncing node recomputes the hash from the deserialized form → `get_tip_resource_bounds_hash` with `L1Gas` → preimage omits `L1_DATA_GAS` → produces H₂ ≠ H₁.
6. The syncing node's RPC returns `{ tx_hash: H₁, resource_bounds: L1Gas(...) }`. Any client computing the hash from the returned fields gets H₂ ≠ H₁ — the response is internally inconsistent.
7. If the SNOS re-executes the block using the syncing node's data, it computes H₂, fails to match the signed H₁, and aborts re-execution.

    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));

            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
