Audit Report

## Title
SwapAllowlistExtension Allowlist Bypassed via Router ‚Äî Any EOA Can Swap Through an Allowlisted Router - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool, which is the pool's `msg.sender` ‚Äî the router, not the originating EOA. When `MetricOmmSimpleRouter` is allowlisted (a natural "public router, restricted pool" configuration), every EOA that calls through the router bypasses the per-address allowlist, nullifying the extension's access control entirely.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap` (L230‚Äì240). `ExtensionCalling._beforeSwap` encodes and forwards that value unchanged to `SwapAllowlistExtension.beforeSwap` (L149‚Äì177). The extension checks `!allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()` (L37). `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly (L72‚Äì80), so the router becomes the pool's `msg.sender`. If `allowedSwapper[pool][router] = true`, the check passes for every caller of the router regardless of their identity. No existing guard in the pool, router, or extension checks `tx.origin` or the originating EOA.

## Impact Explanation
Pool admins deploy `SwapAllowlistExtension` to restrict trading to specific addresses (KYC'd users, whitelisted market makers, protocol-owned addresses). Allowlisting the router ‚Äî the canonical public swap entry point ‚Äî grants unrestricted swap access to all EOAs on the network. Non-allowlisted addresses can execute swaps against LP funds at oracle-derived prices, constituting direct LP-fund exposure from disallowed swappers and a complete bypass of the admin-configured access boundary.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical swap entry point. A pool admin who wants to allow normal swaps while restricting direct pool access will naturally allowlist the router. The bypass requires no special privileges ‚Äî any EOA can call `exactInputSingle`. The condition is a standard, expected configuration, not an edge case.

## Recommendation
The extension should decode the originating caller from `extensionData` when present, and the router should encode `msg.sender` into `extensionData` before forwarding to the pool. Alternatively, documentation must explicitly warn that allowlisting any public router grants access to all users of that router, and pool admins must never allowlist a public router if per-user gating is intended. A more robust protocol-level fix is to have the pool pass a separate `initiator` field (e.g., `tx.origin` or a router-supplied value) through the extension hook interface.
