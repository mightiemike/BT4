### Title
`verify_transaction_inclusion` incorrectly panics for valid single-transaction block proofs — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (and `verify_transaction_inclusion_v2` which delegates to it) contains a guard that rejects empty Merkle proofs unconditionally. However, an empty proof is the only correct proof for a transaction in a single-transaction block (where the Merkle root equals the transaction hash directly). This causes the function to always panic for a valid and common Bitcoin block structure.

### Finding Description

In `contract/src/lib.rs`, `verify_transaction_inclusion` checks:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

This guard fires before the actual Merkle computation. The computation itself, `compute_root_from_merkle_proof`, handles an empty proof correctly: it returns `transaction_hash` unchanged. [2](#0-1) 

For a Bitcoin block containing only one transaction (the coinbase), the Merkle tree has a single leaf, so `merkle_root == tx_id`. The valid proof for this case is an empty `merkle_proof` slice. The computation `compute_root_from_merkle_proof(tx_id, 0, &[])` returns `tx_id`, which correctly equals `merkle_root`. The guard, however, panics before this computation is reached.

`verify_transaction_inclusion_v2` inherits the same defect because it calls `verify_transaction_inclusion` internally after its own coinbase-proof check passes: [3](#0-2) 

For a single-transaction block, both `merkle_proof` and `coinbase_merkle_proof` are empty, the length-equality check passes, the coinbase proof check passes (empty proof correctly returns `coinbase_tx_id` which equals `merkle_root`), and then the call to `verify_transaction_inclusion` panics.

The case coverage gap mirrors the external report exactly:

| Case | Expected behavior | Actual behavior |
|---|---|---|
| `merkle_proof` non-empty | verify and return result | correct |
| `merkle_proof` empty, multi-tx block | reject (invalid) | correct |
| **`merkle_proof` empty, single-tx block** | **verify and return `true`** | **panics** |

### Impact Explanation

Any NEAR caller or downstream contract invoking `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` to verify a transaction in a single-transaction Bitcoin block will always receive a panic instead of a `true` result. This breaks the core SPV verification guarantee of the contract. Downstream contracts relying on this function for cross-chain settlement or proof-gating will be permanently unable to process valid proofs for single-transaction blocks.

### Likelihood Explanation

Single-transaction blocks (containing only the coinbase) are a normal and recurring occurrence in Bitcoin, particularly during periods of low mempool activity or immediately after a difficulty adjustment. Any relayer or user submitting a valid proof for such a block will trigger the panic deterministically, with no workaround available at the contract level.

### Recommendation

Remove the blanket empty-proof guard and instead allow `compute_root_from_merkle_proof` to handle the empty case naturally. If a guard is desired, it should only reject empty proofs when the block is known to have more than one transaction (i.e., when `merkle_root != tx_id`):

```rust
// Remove this line:
// require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

// The computation already handles the empty-proof case correctly:
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

### Proof of Concept

1. A Bitcoin block is mined containing only the coinbase transaction with hash `T`.
2. Its `merkle_root` is `T` (single-leaf Merkle tree).
3. The block is submitted to the contract via `submit_blocks` and accepted.
4. A caller invokes `verify_transaction_inclusion` with `tx_id = T`, `tx_index = 0`, `merkle_proof = []`.
5. The contract panics with `"Merkle proof is empty"` before reaching the computation.
6. `compute_root_from_merkle_proof(T, 0, &[])` would have returned `T == merkle_root`, i.e., `true`. [4](#0-3) [5](#0-4)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
