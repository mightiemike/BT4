### Title
Ineffective `transaction_timeout_sec` Guard Due to Missing Network-Level Timeout on `get_tx_status` Blocks Relayer Sync Loop Indefinitely — (File: `relayer/src/near_client.rs`)

---

### Summary

`submit_blocks` in `near_client.rs` contains a `transaction_timeout_sec` guard that is intended to bound how long the relayer waits for a NEAR transaction to be confirmed. However, the guard is rendered dead code when the NEAR RPC endpoint is unresponsive, because the underlying `get_tx_status` call has no network-level timeout and will never return. The sync loop hangs indefinitely, stalling all Bitcoin header submissions to the on-chain contract.

---

### Finding Description

`NearClient::submit_blocks` implements a polling loop with an explicit timeout check:

```rust
// relayer/src/near_client.rs  lines 287-293
loop {
    let response = self.get_tx_status(tx_hash).await;   // ← no network timeout
    let received_at = time::Instant::now();
    let delta = (received_at - sent_at).as_secs();

    if delta > self.transaction_timeout_sec {            // ← never reached if above hangs
        Err("time limit exceeded for the transaction to be recognized")?;
    }
``` [1](#0-0) 

The timeout check at line 292 is only evaluated **after** `get_tx_status` returns. `get_tx_status` is implemented as a bare `self.client.call(...)` with no `tokio::time::timeout` wrapper or any other deadline:

```rust
// relayer/src/near_client.rs  lines 588-601
async fn get_tx_status(...) -> MethodCallResult<...> {
    self.client
        .call(methods::tx::RpcTransactionStatusRequest { ... })
        .await          // ← blocks indefinitely if NEAR RPC is unresponsive
}
``` [2](#0-1) 

The same pattern exists in `sign_tx` (called before every submission) and `submit_view_tx` (called by `get_last_block_header`, `is_block_hash_exists`, `get_last_n_blocks_hashes`), all of which issue `self.client.call(...)` with no timeout: [3](#0-2) [4](#0-3) 

The `transaction_timeout_sec` field is stored in `NearClient` and read from config, signalling clear developer intent to bound wait time — but the implementation does not enforce it at the network call level: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

The relayer's `sync` loop is the sole mechanism that pushes Bitcoin block headers into the NEAR on-chain `BtcLightClient` contract. If the loop hangs indefinitely inside `submit_blocks` (or `sign_tx` / `submit_view_tx`), no new headers are submitted. The contract's canonical chain view becomes permanently stale. Any downstream consumer calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` on blocks mined after the stall point will receive a panic (`block does not belong to the current main chain`), breaking the SPV guarantee the system is built on. [7](#0-6) [8](#0-7) 

---

### Likelihood Explanation

The NEAR JSON-RPC endpoint is an external network dependency. Transient unresponsiveness (overloaded RPC node, network partition, DNS failure) is a realistic operational condition. The `near-jsonrpc-client` crate used here does not impose a default request deadline, so any such event causes a permanent hang rather than a recoverable error. The Bitcoin RPC path, by contrast, explicitly sets a 15-second timeout via `CustomMinreqHttpTransport`, confirming the developers are aware of the pattern but did not apply it to the NEAR side. [9](#0-8) 

---

### Recommendation

Wrap every `self.client.call(...)` invocation in `near_client.rs` with `tokio::time::timeout`, using a configurable deadline (e.g., reusing `transaction_timeout_sec` or a dedicated `rpc_timeout_sec` config field). For example:

```rust
use tokio::time::{timeout, Duration};

let response = timeout(
    Duration::from_secs(self.transaction_timeout_sec),
    self.get_tx_status(tx_hash),
)
.await
.map_err(|_| "NEAR RPC timed out")??;
```

Apply the same pattern to `sign_tx` and `submit_view_tx`.

---

### Proof of Concept

1. Configure the relayer against a NEAR RPC endpoint that accepts the TCP connection but never sends a response (e.g., `iptables -A OUTPUT -p tcp --dport 443 -j DROP` after the connection is established, or a mock server that reads but never writes).
2. Start the relayer. It calls `sign_tx` → `self.client.call(ViewAccessKey)` → hangs indefinitely.
3. Alternatively, let one submission reach `submit_blocks`. The `submit_tx` call succeeds (fire-and-forget broadcast), then `get_tx_status` is called and hangs. The `transaction_timeout_sec` deadline is never evaluated.
4. The `sync` loop never advances `first_block_height_to_submit`; the NEAR contract receives no further headers; `verify_transaction_inclusion` panics for any block past the stall height. [10](#0-9) [7](#0-6)

    pub async fn submit_blocks(
        &self,
        signed_tx: SignedTransaction,
    ) -> Result<Result<SubmitResult, CustomError>, Box<dyn std::error::Error + Send + Sync>> {
        let sent_at = time::Instant::now();
        let tx_hash = self.submit_tx(signed_tx).await?;

        info!("Blocks submitted: tx_hash = {tx_hash:?}");

        loop {
            let response = self.get_tx_status(tx_hash).await;
            let received_at = time::Instant::now();
            let delta = (received_at - sent_at).as_secs();

            if delta > self.transaction_timeout_sec {
                Err("time limit exceeded for the transaction to be recognized")?;
            }

            match response {
                Err(err) => match err.handler_error() {
                    Some(
                        RpcTransactionError::TimeoutError
                        | RpcTransactionError::UnknownTransaction { .. },
                    ) => {
                        time::sleep(time::Duration::from_secs(2)).await;
                    }
                    _ => Err(err)?,
                },
                Ok(response) => {
                    info!("Success response gotten after: {delta}s");
                    return Ok(Self::parse_submit_blocks_response(&response));
                }
            }
        }
    }

        let first_block_height_to_submit = Arc::new(AtomicU64::new(
            self.get_last_correct_block_height().await.unwrap() + 1,
        ));

        'main_loop: loop {
            let (current_fetch_size, current_batch_size) = {
                let sizer = self.batch_sizer.lock().await;
                (sizer.current_fetch_size(), sizer.current_batch_size())
            };

            let latest_height = continue_on_fail!(
                self.bitcoin_client.get_block_count(),
                "Bitcoin Client: Error on get_block_count",
                self.config.sleep_time_on_fail_sec,
                'main_loop
            );

            let start_height =
                first_block_height_to_submit.load(std::sync::atomic::Ordering::Relaxed);
            let end_height = latest_height.min(start_height.saturating_add(current_fetch_size));

            let blocks_to_submit = self.fetch_blocks_to_submit(start_height, end_height).await;

            if blocks_to_submit.is_empty() {
                tokio::time::sleep(std::time::Duration::from_secs(
                    self.config.sleep_time_on_reach_last_block_sec,
                ))
                .await;
                continue;
            }

            let number_of_blocks = blocks_to_submit.len().try_into().unwrap();
            let last_block_hash = blocks_to_submit.last().unwrap().1.block_hash().to_string();

            if let Ok(true) = self
                .check_submission_skipped(
                    &last_block_hash,
                    start_height,
                    number_of_blocks,
                    first_block_height_to_submit.clone(),
                )
                .await
            {
                continue;
            }

            info!(
                target: "relay",
                "Adaptive batch: fetch_size={}, batch_size={}, blocks_fetched={}",
                current_fetch_size, current_batch_size, blocks_to_submit.len(),
            );

            self.clone()
                .prepare_and_submit_batches(
                    blocks_to_submit,
                    first_block_height_to_submit.clone(),
                    usize::try_from(current_batch_size).unwrap(),
                )
                .await;

            tokio::time::sleep(std::time::Duration::from_secs(
                self.config.sleep_time_after_sync_iteration_sec,
            ))
            .await;
        }
