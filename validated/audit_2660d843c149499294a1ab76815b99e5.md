### Title
Precision Loss in `minSize`/`sizeIncrement` Storage-Retrieval Pattern Causes Fee and Trade Amount Miscalculation — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.updateMarket()` compresses `minSize` and `sizeIncrement` into `int64` by dividing by `1e9` on write, then `getMarketInfo()` multiplies by `1e9` on read. The intermediate integer division silently truncates any sub-`1e9` remainder, so the value used in live fee and trade-rounding logic is permanently lower than what was configured. This is the same divide-then-multiply precision-loss class as the reference report.

---

### Finding Description

`updateMarket()` stores both market parameters with a lossy compression:

```solidity
marketInfo[productId].minSize      = int64(minSize      / 1e9);
marketInfo[productId].sizeIncrement = int64(sizeIncrement / 1e9);
```

`getMarketInfo()` reconstructs them:

```solidity
m.minSize       = int128(market.minSize)       * 1e9;
m.sizeIncrement = int128(market.sizeIncrement) * 1e9;
``` [1](#0-0) [2](#0-1) 

Because Solidity integer division truncates toward zero, any `minSize` or `sizeIncrement` that is not an exact multiple of `1e9` loses its fractional part permanently. For example:

| Configured value | Stored (`/ 1e9`) | Retrieved (`* 1e9`) | Lost |
|---|---|---|---|
| `1.5e9` | `1` | `1e9` | `0.5e9` |
| `2.3e9` | `2` | `2e9` | `0.3e9` |

The truncated `minSize` is then consumed directly in `applyFee()`:

```solidity
meteredQuote += market.minSize;                                    // flat fee base
...
int128 feeApplied = MathHelper.abs(alreadyMatched + matchQuote) - market.minSize;
feeApplied = MathHelper.min(feeApplied, matchQuote.abs());
``` [3](#0-2) 

The truncated `sizeIncrement` is consumed in `matchOrders()`:

```solidity
ordersInfo.taker.amountDelta -=
    ordersInfo.taker.amountDelta % market.sizeIncrement;
``` [4](#0-3) 

---

### Impact Explanation

**Fee miscalculation (`minSize`):** The flat minimum fee threshold applied to taker orders is lower than configured. Takers whose `matchQuote` falls in the range `[truncated_minSize, configured_minSize)` are metered for fees on a larger portion of their quote than intended — they pay more in fees than the protocol promised. Conversely, the `feeApplied` subtraction uses the truncated value, shifting the fee band. Both directions represent a contract failing to deliver the promised fee schedule.

**Trade amount rounding (`sizeIncrement`):** Every matched trade is floored to the nearest `sizeIncrement`. A truncated increment means trades are rounded to a coarser grid than intended, causing takers to receive or deliver a slightly different base amount than their order specified.

Both effects are permanent for the lifetime of the market configuration and apply to every trade on the affected product.

---

### Likelihood Explanation

Medium. The `updateMarket()` function accepts arbitrary `int128` inputs from the engine. There is no on-chain enforcement that `minSize` or `sizeIncrement` must be exact multiples of `1e9`. Any market configured with a non-multiple value — which is a natural outcome of expressing sizes in human-readable decimals before scaling — silently truncates. The effect is triggered by every ordinary user trade on that product via `matchOrders()`, requiring no special attacker capability. [5](#0-4) 

---

### Recommendation

Remove the lossy compression. Either:

1. Store `minSize` and `sizeIncrement` as `int128` directly (preferred — eliminates the divide/multiply round-trip entirely), or
2. Add a `require(minSize % 1e9 == 0 && sizeIncrement % 1e9 == 0)` guard in `updateMarket()` so that any non-representable value reverts rather than silently truncating.

---

### Proof of Concept

1. Engine calls `updateMarket(productId, quoteId, sizeIncrement=1_500_000_000, minSize=1_500_000_000)`.
2. Stored: `marketInfo[productId].minSize = int64(1_500_000_000 / 1e9) = 1`.
3. Retrieved in `getMarketInfo()`: `m.minSize = int128(1) * 1e9 = 1_000_000_000`.
4. `applyFee()` uses `market.minSize = 1_000_000_000` instead of `1_500_000_000`.
5. A taker whose `matchQuote = 1_200_000_000` is now metered for fees on `200_000_000` of quote (above the truncated threshold) instead of `0` (below the intended threshold) — paying fees the protocol did not intend to charge.
6. Similarly, `matchOrders()` rounds `amountDelta` to multiples of `1_000_000_000` instead of `1_500_000_000`, silently altering every fill size on the market.
