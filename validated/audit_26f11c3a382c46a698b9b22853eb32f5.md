### Title
Deprecated Non-Production-Ready Proof Verification Function Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery - (File: `contract/src/lib.rs`)

---

### Summary

The `verify_transaction_inclusion` function is explicitly marked `#[deprecated(since = "0.5.0")]` and documented as vulnerable to the 64-byte transaction Merkle proof forgery attack, yet it remains a live, publicly callable NEAR contract method accessible by any unprivileged caller. This is a direct analog to the "Use of Non-Production-Ready Trusted Forwarder" vulnerability class: a component explicitly flagged as unsafe for production use is still reachable and produces exploitable results.

---

### Finding Description

In `contract/src/lib.rs`, the function `verify_transaction_inclusion` carries both a Rust `#[deprecated]` attribute and an explicit `# Warning` in its documentation:

```
/// # Warning
/// This function may return `true` if the provided `tx_id` is a hash of an
/// internal node in the Merkle tree rather than a valid transaction hash.
``` [1](#0-0) 

The `#[deprecated]` attribute in Rust is a **compile-time warning only**. It does not remove the function from the compiled WASM binary, does not restrict on-chain access, and does not prevent any NEAR account from calling it. The function remains a fully operational public contract method. [2](#0-1) 

The root cause of the vulnerability is that `compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` accepts any 32-byte value as the starting `transaction_hash` and walks the proof tree without any structural validation that the input is a leaf-level transaction hash rather than an internal node hash. [3](#0-2) 

The safe replacement, `verify_transaction_inclusion_v2`, mitigates this by first validating a coinbase Merkle proof (coinbase is always at index 0, so it cannot be an internal node), which anchors the tree structure before verifying the target transaction. [4](#0-3) 

The deprecated function is also still present as a callable method in the off-chain relayer client, though that path is less critical. [5](#0-4) 

---

### Impact Explanation

An attacker can call `verify_transaction_inclusion` directly with a crafted `tx_id` that is the hash of an internal Merkle tree node (not a real transaction). The function will compute the Merkle root from this internal node hash and the supplied proof path, match it against the stored block header's `merkle_root`, and return `true`. Any downstream NEAR contract or application that relies on this return value to authorize fund releases, cross-chain bridge operations, or SPV-based settlement will be deceived into accepting a non-existent transaction as confirmed. The corrupted proof result is the exact state value at risk: `verify_transaction_inclusion` → `true` for a fabricated `tx_id`.

---

### Likelihood Explanation

The 64-byte transaction forgery attack is publicly documented (referenced in the contract's own comments: https://www.bitmex.com/blog/64-Byte-Transactions). The function requires no privileged role — it is gated only by `#[pause]`, meaning any non-paused deployment is fully open to any NEAR account. The attacker needs only to: (1) identify a Bitcoin block with a known Merkle tree, (2) compute the hash of any internal node, (3) construct a valid proof path from that internal node to the root, and (4) call `verify_transaction_inclusion` with that crafted input. No mining, no key compromise, and no social engineering is required.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public contract interface entirely, or gate it with an access control role that prevents unprivileged callers from invoking it. The `#[deprecated]` attribute provides no on-chain enforcement. All callers should be migrated to `verify_transaction_inclusion_v2`. If backward compatibility is required during a transition period, the function body should be replaced with an unconditional `env::panic_str("use verify_transaction_inclusion_v2")`.

---

### Proof of Concept

1. Identify any Bitcoin block already stored in the contract (e.g., via `get_block_hash_by_height`).
2. Obtain the full transaction list for that block and compute the hash of any internal Merkle tree node `N` at depth `d`.
3. Construct a valid Merkle proof path of length `d` from `N` to the block's `merkle_root`.
4. Call `verify_transaction_inclusion` with:
   - `tx_id` = hash of internal node `N`
   - `tx_block_blockhash` = the stored block hash
   - `tx_index` = the index corresponding to `N`'s position at depth `d`
   - `merkle_proof` = the constructed proof path
   - `confirmations` = 1
5. The function returns `true`, falsely attesting that a non-existent transaction was confirmed in that block.

    /// Verifies that a transaction is included in a block at a given block height
    ///
    /// # Deprecated
    /// Use [`verify_transaction_inclusion_v2`] instead, which includes coinbase merkle proof validation
    /// to mitigate the 64-byte transaction Merkle proof forgery vulnerability:
    /// https://www.bitmex.com/blog/64-Byte-Transactions
    ///
    /// @param `tx_id` transaction identifier
    /// @param `tx_block_blockhash` block hash at which transacton is supposedly included
    /// @param `tx_index` index of transaction in the block's tx merkle tree
    /// @param `merkle_proof` merkle tree path (concatenated LE sha256 hashes) (does not contain initial `transaction_hash` and `merkle_root`)
    /// @param confirmations how many confirmed blocks we want to have before the transaction is valid
    /// @return True if `tx_id` is at the claimed position in the block at the given blockhash, False otherwise
    ///
    /// # Warning
    /// This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash.
    /// We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification.
    ///
    /// # Panics
    /// Multiple cases
    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
