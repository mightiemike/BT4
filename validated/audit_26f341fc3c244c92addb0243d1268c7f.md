### Title
`IIERC20Base.approve` Reverts for Non-Standard ERC20 Tokens in `creditDeposit`, Permanently Blocking Deposits — (`File: core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` as a high-level Solidity call. For non-standard ERC20 tokens (e.g., USDT) that do not return a `bool` from `approve`, the ABI decoder reverts. The same contract already implements a `safeTransfer` helper using low-level `.call()` to handle this exact class of token, but no equivalent `safeApprove` is used.

---

### Finding Description

In `DirectDepositV1.sol`, the `IIERC20Base` interface declares `approve` as:

```solidity
function approve(address spender, uint256 amount) external returns (bool);
``` [1](#0-0) 

`creditDeposit()` then calls this directly:

```solidity
token.approve(address(endpoint), balance);
``` [2](#0-1) 

This is a high-level Solidity call. When the underlying token does not return a `bool` (e.g., USDT on mainnet), the Solidity ABI decoder attempts to decode a `bool` from an empty return buffer and reverts.

By contrast, the same contract already handles this correctly for `transfer` via a low-level `.call()` pattern:

```solidity
(bool success, bytes memory data) = address(self).call(...);
require(success && (data.length == 0 || abi.decode(data, (bool))), "Transfer failed");
``` [3](#0-2) 

The `ERC20Helper` library used elsewhere in the protocol applies the same safe pattern for `safeTransfer` and `safeTransferFrom`, but provides **no `safeApprove`**: [4](#0-3) 

The same raw `.approve()` pattern also appears in `ContractOwner.wrapVaultAsset()`:

```solidity
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
``` [5](#0-4) 

`wrapVaultAsset()` also has no access control, making it reachable by any caller. [6](#0-5) 

---

### Impact Explanation

If any spot product token registered in `SpotEngine` is a non-standard ERC20 (no `bool` return on `approve`), every call to `creditDeposit()` will revert at the `approve` step. Since `creditDeposit()` iterates over **all** product IDs, a single non-standard token in the product list causes the entire deposit flow to be permanently bricked for every subaccount using `DirectDepositV1`. Funds sent to the `DirectDepositV1` address cannot be forwarded to the endpoint and are effectively locked until an owner manually calls `withdraw()`.

---

### Likelihood Explanation

`creditDeposit()` has no access control — any user or bot can call it. [7](#0-6) 

The Nado protocol is a multi-asset exchange. If USDT or any other non-`bool`-returning token is ever listed as a spot product, this path is immediately and unconditionally broken for all `DirectDepositV1` users. The likelihood is directly tied to the token listing choices of the protocol.

---

### Recommendation

Replace the raw `token.approve(...)` call in `creditDeposit()` with a low-level safe approve pattern analogous to the existing `safeTransfer` helper:

```solidity
function safeApprove(IIERC20Base self, address spender, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IIERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        "Approve failed"
    );
}
```

Apply the same fix to `ContractOwner.wrapVaultAsset()` by adding a `safeApprove` to `ERC20Helper` and using it in place of the raw `.approve()` calls. [4](#0-3) 

---

### Proof of Concept

1. A non-standard ERC20 token (e.g., USDT, which omits the `bool` return) is listed as a spot product in `SpotEngine`.
2. A user sends that token to their `DirectDepositV1` address.
3. Any caller invokes `creditDeposit()`.
4. The loop reaches the non-standard token: `token.approve(address(endpoint), balance)` is executed as a high-level call.
5. The token's `approve` returns no data; Solidity's ABI decoder expects a `bool` and reverts.
6. The entire transaction reverts. The user's tokens remain stuck in the `DirectDepositV1` contract with no way to forward them to the endpoint via the normal deposit path. [8](#0-7)
