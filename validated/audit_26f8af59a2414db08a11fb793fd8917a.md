### Title
`ERC20Helper.safeTransfer`/`safeTransferFrom` Lack Contract Code Check, Enabling Silent Transfer Failures and Deposit Accounting Corruption — (`File: core/contracts/libraries/ERC20Helper.sol`)

---

### Summary

Nado's `ERC20Helper` library performs token transfers via raw `.call()` without verifying that the target token address contains deployed contract code. If a registered token address has no code, the EVM returns `(true, "")`, which silently passes the `require` check. The protocol then records the transfer as successful and updates internal balances, while no tokens actually move. This corrupts deposit accounting in `SpotEngine`.

---

### Finding Description

`ERC20Helper.safeTransfer` and `safeTransferFrom` both use the following pattern:

```solidity
(bool success, bytes memory data) = address(self).call(
    abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
);
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [1](#0-0) [2](#0-1) 

When `address(self)` has no deployed bytecode, the EVM treats the call as a no-op and returns `(true, bytes(""))`. The condition `success && (data.length == 0 || ...)` evaluates to `true && true`, so the `require` passes without reverting. No tokens are moved, but the caller believes the transfer succeeded.

Every guard in the codebase only checks `address(token) != address(0)` — not whether the address has code: [3](#0-2) [4](#0-3) 

The affected call sites are:

| Function | File | Transfer direction |
|---|---|---|
| `handleDepositTransfer` | `EndpointStorage.sol:117–118` | user → endpoint → clearinghouse |
| `chargeSlowModeFee` | `EndpointStorage.sol:88–93` | user → endpoint |

### Impact Explanation

The most severe impact is in the **deposit path**. `handleDepositTransfer` calls `safeTransferFrom` (user → endpoint) and then `safeTransfer` (endpoint → clearinghouse). If both silently succeed on a codeless token address, control returns to `Clearinghouse.depositCollateral`, which unconditionally credits the user's `SpotEngine` balance: [8](#0-7) 

The user's internal spot balance is inflated without any real token receipt. This corrupts the protocol's solvency invariant: the sum of all internal balances exceeds the actual token holdings of the clearinghouse. The inflated balance can then be used to open leveraged positions, withdraw other users' collateral, or pass health checks that should fail.

For the **withdrawal path**, `submitFastWithdrawal` marks `markedIdxs[idx] = true` before calling `handleWithdrawTransfer`. If the transfer silently fails, the withdrawal is permanently consumed — the user loses their funds with no recourse, since the idx can never be resubmitted. [9](#0-8) 

---

### Likelihood Explanation

Token addresses are set by the protocol admin via `SpotEngine.addOrUpdateProduct`, so a randomly chosen non-contract address is not the primary risk vector. However, realistic triggers include:

1. **Pre-deployment product listing**: A product is listed before its token contract is deployed to the target chain (common in multi-chain deployments). Any deposit made in this window silently credits balances with no token receipt.
2. **Token contract self-destruction**: A token using `selfdestruct` removes its code; subsequent transfers via `ERC20Helper` silently succeed.
3. **Cross-chain deployment mismatch**: The same product config is reused across chains where the token address exists on one chain but not another.

The deposit entrypoint (`depositCollateralWithReferral` / `depositCollateral`) is fully user-accessible with no privileged gating beyond a valid subaccount. [10](#0-9) 

---

### Recommendation

Replace `ERC20Helper` with OpenZeppelin's `SafeERC20`, which performs a code-size check before calling transfer functions. Alternatively, add an explicit check inside `ERC20Helper`:

```solidity
function safeTransfer(IERC20Base self, address to, uint256 amount) internal {
    require(address(self).code.length > 0, "ERC20Helper: no code at token address");
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
```

Apply the same guard to `safeTransferFrom`.

---

### Proof of Concept

1. Admin lists a new spot product with `productId = X` whose token address `T` has no deployed code (e.g., pre-deployment).
2. Attacker calls `Endpoint.depositCollateralWithReferral(subaccount, X, 1_000_000, "")`.
3. `EndpointStorage.handleDepositTransfer(T, attacker, 1_000_000)` is called.
4. `T.safeTransferFrom(attacker, endpoint, 1_000_000)` → `address(T).call(...)` returns `(true, "")` → `require` passes. No tokens move.
5. `T.safeTransfer(clearinghouse, 1_000_000)` → same silent success.
6. `Clearinghouse.depositCollateral` executes `spotEngine.updateBalance(X, subaccount, amountRealized)`.
7. Attacker's internal balance for product `X` is now `1_000_000 * multiplier` with zero actual token cost.
8. Attacker uses this inflated balance to pass health checks and withdraw real collateral from other products, draining the clearinghouse. [11](#0-10) [12](#0-11)
