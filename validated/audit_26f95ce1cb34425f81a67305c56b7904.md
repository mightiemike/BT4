### Title
`ValidResourceBounds` Variant Collapse Across Protobuf Boundary Produces Wrong Transaction Hash - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

A `ValidResourceBounds::AllResources` value whose `l2_gas` and `l1_data_gas` fields are both zero is serialized to protobuf with those fields set to zero, then deserialized back as `ValidResourceBounds::L1Gas`. Because `get_tip_resource_bounds_hash` includes the `L1DataGas` element in the Poseidon hash chain only for the `AllResources` variant, the hash computed before serialization differs from the hash computed after deserialization. A user can deliberately submit a V3 transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` to trigger this divergence.

### Finding Description

**Serialization side** (`crates/apollo_protobuf/src/converters/transaction.rs`, lines 471–490):

```rust
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),      // ResourceBounds::default() → zeros
    l1_data_gas: Some(ResourceBounds::default().into()), // zeros
},
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds { l1_gas: Some(l1_gas.into()), l2_gas: Some(l2_gas.into()),
                                l1_data_gas: Some(l1_data_gas.into()) },
```

When `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is serialized, the wire bytes are identical to those produced by `L1Gas(X)`.

**Deserialization side** (`crates/apollo_protobuf/src/converters/transaction.rs`, lines 417–436):

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

Any protobuf message with zero `l2_gas` and zero `l1_data_gas` is unconditionally reconstructed as `L1Gas`, regardless of which variant was originally serialized.

**Hash computation** (`crates/starknet_api/src/transaction_hash.rs`, lines 187–211):

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2-element chain
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3-element chain
    }
});
Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
```

- `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` → Poseidon over `[tip, L1_GAS(X), L2_GAS(0), L1_DATA_GAS(0)]`
- `L1Gas(X)` → Poseidon over `[tip, L1_GAS(X), L2_GAS(0)]`

These are distinct field elements; the hashes are different.

**Trigger path:**

1. User submits `RpcInvokeTransactionV3` with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`. The gateway accepts it (zero l2/l1_data gas is valid per the stateless validator).
2. `convert_rpc_tx_to_internal` computes `tx_hash = H1` using `AllResources` variant (3-element Poseidon chain).
3. The transaction is included in a block and stored.
4. During P2P block sync, the `InvokeTransactionV3` (which carries `ValidResourceBounds`) is serialized to protobuf and transmitted.
5. The receiving node deserializes it as `ValidResourceBounds::L1Gas`.
6. Any hash recomputation on the receiving node (e.g., `validate_transaction_hash`, mempool deduplication, RPC `starknet_getTransactionByHash` serving) produces `H2 ≠ H1`.

### Impact Explanation

The receiving node holds a transaction whose stored hash `H1` (from the originating sequencer) does not match the hash `H2` it would compute from the deserialized object. This causes:

- `validate_transaction_hash` to reject a legitimately included transaction, breaking block sync.
- RPC endpoints (`starknet_getTransactionByHash`, `starknet_getTransactionReceipt`) to serve an authoritative-looking wrong hash or fail to locate the transaction.
- Consensus participants that recompute hashes from deserialized objects to disagree on block content.

This matches **High** impact: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value"* and *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

### Likelihood Explanation

Any user can submit a V3 invoke transaction with zero `l2_gas` and zero `l1_data_gas` (only `l1_gas` non-zero). The gateway's stateless validator explicitly permits this (test case `valid_l1_gas`). No special privilege is required. The divergence is deterministic and reproducible on every node that syncs the block.

### Recommendation

Preserve the `ValidResourceBounds` variant across the protobuf boundary. Two options:

1. **Add a discriminant field** to `protobuf::ResourceBounds` (e.g., a boolean `is_all_resources`) so the deserializer can reconstruct the correct variant without relying on zero-value heuristics.
2. **Reject zero-l2/l1_data_gas `AllResources` at the gateway** by adding a stateless validation rule that requires at least one of `l2_gas` or `l1_data_gas` to be non-zero for `AllResources` transactions, making the two variants non-overlapping in the wire representation.

### Proof of Concept

         l1_data_gas: ResourceBounds { max_amount: 0,    max_price_per_unit: 0 },

2. Submit to gateway → accepted. Gateway computes:
     H1 = Poseidon(INVOKE, version=3, sender, tip_resource_hash_AllResources, ...)

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
