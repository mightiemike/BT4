Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of originating user, enabling allowlist bypass or blocking legitimate users — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`MetricOmmPool.swap` forwards its own `msg.sender` as the `sender` argument to every configured extension. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. This breaks the allowlist in both directions: if the router is allowlisted, any user bypasses the curated pool's access control; if it is not, every allowlisted user is blocked from using the protocol's primary swap interface.

## Finding Description

`MetricOmmPool.swap` unconditionally passes `msg.sender` as the first argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol  lines 230-240
_beforeSwap(
    msg.sender,   // always the direct caller of pool.swap()
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that value against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol  lines 37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is the forwarded value — whoever called `pool.swap()`.

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly without forwarding the originating user:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol  lines 71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

The originating user's address is stored only in transient callback context (for payment purposes) and is never surfaced to the pool or its extensions. The resulting call chain is:

```
user → router.exactInputSingle()
         → pool.swap()          [msg.sender = router]
             → extension.beforeSwap(sender = router, ...)
                 → allowedSwapper[pool][router]   ← wrong actor
```

The same misbinding applies to `exactInput` (line 104), `exactOutputSingle` (line 136), `exactOutput` (line 165), and the recursive `_exactOutputIterateCallback` (line 220), where every `pool.swap()` call originates from the router contract.

No existing guard corrects this: the extension has no access to transient storage, `extensionData` is caller-supplied and unverified, and the pool interface provides no separate `originalSender` field.

## Impact Explanation

**Scenario A — router is allowlisted (bypass):** A pool admin who allowlists the router as a trusted intermediary inadvertently opens the pool to every user. Any EOA can call `router.exactInputSingle()` and the extension passes because `allowedSwapper[pool][router] == true`. The curated pool's access control is fully defeated — this is a broken core invariant with direct fund-flow consequences (unauthorized parties execute swaps against a pool intended to be restricted).

**Scenario B — router is not allowlisted (broken functionality):** A legitimately allowlisted user who routes through the router receives `NotAllowedToSwap` because `allowedSwapper[pool][router] == false`. The only working path is a direct `pool.swap()` call, making `MetricOmmSimpleRouter` — the protocol's documented primary swap interface — unusable for any curated pool. This constitutes broken core swap functionality.

Both outcomes satisfy the allowed impact gate: broken core pool functionality causing loss of funds (Scenario A) or unusable swap flows (Scenario B).

## Likelihood Explanation

Every swap routed through `MetricOmmSimpleRouter` against a pool that has `SwapAllowlistExtension` in its `BEFORE_SWAP_ORDER` triggers this mismatch. No special attacker capability is required beyond calling a public router function. Any curated pool that deploys this extension and expects users to use the router is immediately affected. The condition is repeatable and deterministic.

## Recommendation

The pool must surface the originating user's address to extensions. The preferred fix is a pool-level change: add an `originalSender` field populated from a transient-storage slot that the router writes before calling `pool.swap()` (the router already uses transient storage for callback context via `_setNextCallbackContext`). Pass `originalSender` alongside `sender` in the extension call data so `SwapAllowlistExtension.beforeSwap` can check the real initiating address. Alternatively, require the router to embed and sign the caller's address in `extensionData`, but this shifts burden to every integrator and is fragile.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured in `BEFORE_SWAP_ORDER`.
2. **Scenario A:** Call `setAllowedToSwap(pool, router, true)` to allowlist the router. From any EOA not individually allowlisted, call `router.exactInputSingle(...)`. The swap succeeds because `allowedSwapper[pool][router] == true`, bypassing the intended per-user gate.
3. **Scenario B:** Leave the router un-allowlisted. Call `setAllowedToSwap(pool, user, true)` for a specific EOA. From that EOA, call `router.exactInputSingle(...)`. The swap reverts with `NotAllowedToSwap` because `allowedSwapper[pool][router] == false`, even though the user is individually permitted.

The corrupted value in both cases is the identity evaluated by the guard: `allowedSwapper[pool][router]` is checked where `allowedSwapper[pool][user]` is the invariant-correct check. [1](#0-0) [2](#0-1) [3](#0-2)
