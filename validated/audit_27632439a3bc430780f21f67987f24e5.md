Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Actual End-User, Allowing Any User to Bypass the Per-User Allowlist Gate — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it verbatim to `SwapAllowlistExtension.beforeSwap`. When `MetricOmmSimpleRouter` intermediates a swap, `msg.sender` inside the pool is the router's address, not the end-user's. If the pool admin allowlists the router to enable router-mediated swaps for legitimate users, every non-allowlisted user can bypass the per-user gate by routing through the router.

## Finding Description
`MetricOmmPool.swap` invokes `_beforeSwap(msg.sender, ...)` at line 230–240, passing the immediate caller as `sender`.

`ExtensionCalling._beforeSwap` at lines 149–177 forwards `sender` verbatim to every configured extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`.

`SwapAllowlistExtension.beforeSwap` at lines 37–38 evaluates:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool and `sender` is whoever called `pool.swap`. When `MetricOmmSimpleRouter.exactInputSingle` (lines 72–80) calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)`, the router is `msg.sender` inside the pool, so `sender = router`. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

The router provides no mechanism to forward the original user's identity: it calls `pool.swap` directly with `params.recipient` as the output recipient, not as the identity to be checked. The `extensionData` field is passed through but the router does not encode the originating user into it.

This creates an irreconcilable dilemma:
- Router **not** allowlisted → all router-mediated swaps blocked, even for individually allowlisted users.
- Router **allowlisted** → every user bypasses the individual allowlist by routing through the router.

The `allowedSwapper` and `allowAllSwappers` mappings at lines 12–13 confirm there is no secondary identity field; the only two checks are `allowAllSwappers[pool]` and `allowedSwapper[pool][sender]`.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (KYC'd users, trusted market makers, or to exclude adversarial actors) cannot enforce that restriction when the router is involved. Any non-allowlisted user can execute swaps on the restricted pool by calling `MetricOmmSimpleRouter`, bypassing the intended access control entirely. LP funds are exposed to unauthorized trading on a pool whose entire security model depends on the allowlist being enforced. This matches the allowed impact category: **admin-boundary break where a factory/pool role check is bypassed by an unprivileged path**, and **broken core pool functionality causing loss of funds**.

## Likelihood Explanation
High. `MetricOmmSimpleRouter` is a public, permissionless contract reachable by any user. The bypass requires only that the pool admin has allowlisted the router — a natural and expected configuration step to allow legitimate users to trade via the router. No special privileges, malicious setup, or non-standard tokens are required. The attacker needs only to call `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) targeting the restricted pool.

## Recommendation
The extension must check the economically relevant actor, not the immediate pool caller. The most practical fix without changing the core pool interface:

**Encode the originating user in `extensionData`:** Require the router to encode the original `msg.sender` into `extensionData` before calling `pool.swap`. The extension decodes that address and checks it against the allowlist. Since `extensionData` is already forwarded verbatim through `_beforeSwap` to every extension, no core interface change is needed. The extension should verify that when `sender` is a known router, the decoded originator is allowlisted, and reject calls where the originator field is absent or zero.

Alternatively, document explicitly that the router must never be allowlisted and that allowlisted users must call the pool directly — accepting that router-mediated swaps are incompatible with per-user allowlisting.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
2. Pool admin calls: extension.setAllowedToSwap(pool, router, true)
   (allowlists the router so legitimate users can trade via it)
3. Pool admin does NOT call: extension.setAllowedToSwap(pool, attacker, true)
4. attacker calls: MetricOmmSimpleRouter.exactInputSingle(
       ExactInputSingleParams({pool: pool, recipient: attacker, ...})
   )
5. Router calls: pool.swap(attacker, zeroForOne, amount, ...)
   → msg.sender inside pool = router
6. Pool calls: _beforeSwap(router, attacker, ...)
   → sender = router
7. Extension evaluates: allowedSwapper[pool][router] == true → passes
8. Swap executes. attacker successfully trades on a pool they were
   individually excluded from, bypassing the allowlist gate entirely.
```

Foundry test plan: deploy pool with `SwapAllowlistExtension`, allowlist only the router, attempt `exactInputSingle` from an address not individually allowlisted, assert the swap succeeds (demonstrating the bypass) rather than reverting with `NotAllowedToSwap`.
