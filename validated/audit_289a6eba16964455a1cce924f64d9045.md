### Title
Fee Charged Before Balance Deduction Causes Full-Balance Withdrawals to Revert - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.sol`, `chargeFee` is called before `clearinghouse.withdrawCollateral` for both `WithdrawCollateral` and `WithdrawCollateralV2` transaction types. Because the fee is deducted from the user's spot balance first, a subsequent withdrawal of the original (pre-fee) amount will attempt to debit more than the account holds, causing the final health check inside `withdrawCollateral` to revert. This blocks any user who attempts to withdraw their full balance.

---

### Finding Description

In `EndpointTx.sol`, the sequencer-submitted withdrawal flow for `WithdrawCollateral` executes in this order:

1. `chargeFee(signedTx.tx.sender, withdrawFeeX18, signedTx.tx.productId)` — deducts the protocol fee from the user's spot balance for `productId`.
2. `clearinghouse.withdrawCollateral(signedTx.tx.sender, signedTx.tx.productId, signedTx.tx.amount, ...)` — deducts `signedTx.tx.amount` from the same spot balance and enforces a health check. [1](#0-0) 

The same ordering applies to `WithdrawCollateralV2`: [2](#0-1) 

Inside `Clearinghouse.withdrawCollateral`, the balance delta applied is the full, unmodified `amount`:

```solidity
int128 amountRealized = -int128(amount) * int128(multiplier);
spotEngine.updateBalance(productId, sender, amountRealized);
``` [3](#0-2) 

Followed immediately by a health check:

```solidity
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
``` [4](#0-3) 

**Broken invariant:** When a user submits `amount = full_balance`, the sequence is:

| Step | Balance |
|---|---|
| Before fee | `B` |
| After `chargeFee` | `B − fee` |
| After `withdrawCollateral(amount = B)` | `B − fee − B = −fee` |

A negative balance produces negative health, so the health check reverts. The tokens have already been transferred out of the `Clearinghouse` to the `WithdrawPool` at this point (line 408 / `handleWithdrawTransfer`), but the entire transaction reverts atomically, so no funds are lost — the withdrawal simply fails. [5](#0-4) 

---

### Impact Explanation

Any user whose signed `amount` equals (or is close enough to) their full spot balance for the withdrawn product will have their withdrawal permanently revert as long as `withdrawFeeX18 > 0` for that product. The user cannot complete the withdrawal without re-signing a new transaction with a reduced amount. Because the signed `amount` is embedded in the EIP-712 payload and validated by the sequencer, the user must submit an entirely new withdrawal request. This constitutes a denial-of-service on full-balance withdrawals — a common and expected user action.

---

### Likelihood Explanation

Likelihood is **high**. Users routinely withdraw their entire balance when exiting a position. The `withdrawFeeX18` is a standard protocol configuration that is expected to be non-zero in production. Both the V1 and V2 withdrawal paths are affected. No special conditions or attacker involvement are required — any ordinary user triggering a full-balance withdrawal will encounter this revert. [1](#0-0) [6](#0-5) 

---

### Recommendation

Record the user's balance **after** `chargeFee` and cap the withdrawal `amount` to that post-fee balance, mirroring the fix described in the reference report. Alternatively, subtract the fee from the withdrawal amount before passing it to `withdrawCollateral`, consistent with how `submitFastWithdrawal` already handles this:

```solidity
// BaseWithdrawPool.sol — correct pattern already used for fast withdrawals
transferAmount -= uint128(fee);
``` [7](#0-6) 

For the sequenced path, the fix should be applied in `EndpointTx.sol` before the `clearinghouse.withdrawCollateral` call, for both `WithdrawCollateral` and `WithdrawCollateralV2` branches.

---

### Proof of Concept

**Setup:** User subaccount `S` holds exactly `1000 USDC` (in X18: `1000e18`) in product `productId = 1`. `withdrawFeeX18 = 1e18` (1 USDC fee).

**Attack / Trigger path (unprivileged user):**

1. User signs a `WithdrawCollateral` transaction with `amount = 1000e6` (1000 USDC in native decimals).
2. Sequencer submits the transaction to `Endpoint.sol` → `delegatecall` → `EndpointTx.processTransaction`.
3. `chargeFee(S, 1e18, 1)` executes: spot balance of `S` for product 1 becomes `999e18`.
4. `clearinghouse.withdrawCollateral(S, 1, 1000e6, address(0), idx)` executes:
   - `handleWithdrawTransfer` sends 1000 USDC to `WithdrawPool` (tokens leave `Clearinghouse`).
   - `amountRealized = -1000e18`; `spotEngine.updateBalance(1, S, -1000e18)` → balance = `999e18 − 1000e18 = −1e18`.
   - `getHealth(S, INITIAL) < 0` → `require` reverts with `ERR_SUBACCT_HEALTH`.
5. Entire transaction reverts. Withdrawal fails. User must re-sign with `amount ≤ 999e6`. [8](#0-7) [1](#0-0)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
```
