Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of originating user, enabling full allowlist bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` enforces its per-pool allowlist against the `sender` argument, which is the immediate caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` resolves to the router contract address, not the end user. Any pool admin who allowlists the router to enable legitimate users to trade through the standard periphery path simultaneously grants every unprivileged address the ability to bypass the allowlist by routing through the same router.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap` (L230–231). `ExtensionCalling._beforeSwap` encodes and forwards that value verbatim to every configured extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))` (L160–176). `SwapAllowlistExtension.beforeSwap` then evaluates `!allowedSwapper[msg.sender][sender]` (L37), where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`.

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`), the router itself calls `IMetricOmmPoolActions(params.pool).swap(...)` (L72–80), making the router the `msg.sender` inside the pool. The extension therefore receives `sender = address(router)` and evaluates `allowedSwapper[pool][router]`.

The pool admin faces an irresolvable configuration conflict: allowlisting only individual user addresses breaks the router path for those users (router is not allowlisted); allowlisting the router opens the pool to every address. No configuration simultaneously allows legitimate router-based trading and blocks non-allowlisted users. The existing `allowAllSwappers` escape hatch does not resolve this — it simply disables the allowlist entirely.

## Impact Explanation
A pool deploying `SwapAllowlistExtension` to restrict swaps to a curated set of addresses provides zero access control once the router is allowlisted. Any unprivileged address can call `MetricOmmSimpleRouter.exactInputSingle` and execute swaps on the restricted pool. The allowlist — the sole access-control boundary on the swap path — is silently and completely defeated. This constitutes broken core pool functionality (the allowlist extension's entire purpose is nullified) and an admin-boundary break reachable by an unprivileged path, both within the contest's allowed impact gate.

## Likelihood Explanation
The router is the standard, documented entry point for swaps in the Metric OMM periphery. Any pool admin who wants legitimate users to trade through the router must allowlist it. Once that configuration is in place, the bypass requires no privileged role, no special token behavior, and no additional preconditions — any address with funds can call the router. Likelihood is high for any pool using `SwapAllowlistExtension` alongside the router.

## Recommendation
The extension must gate the economic actor (the end user), not the immediate caller. Two sound approaches:

1. **Pass the originating user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. The extension must also verify the caller is a trusted router (e.g., via a factory registry lookup) before trusting the embedded identity, otherwise any caller can forge the field.

2. **Add a dedicated `swapOriginator` field to the hook signature**: The pool passes both `msg.sender` (the immediate caller) and a separately tracked originator. Extensions can then choose which actor to gate.

Until fixed, pools requiring a genuine swap allowlist must instruct users to call `pool.swap()` directly and must not allowlist the router address.

## Proof of Concept
```
1. Pool admin deploys pool with SwapAllowlistExtension configured on beforeSwap.
2. Admin allowlists the router so legitimate users can trade:
       extension.setAllowedToSwap(pool, address(router), true)
3. Attacker (not individually allowlisted) calls:
       router.exactInputSingle(ExactInputSingleParams{pool: pool, ...})
4. Router calls pool.swap(); msg.sender inside pool = router address.
5. Pool calls _beforeSwap(sender=router, ...) → extension.beforeSwap(router, ...).
6. Extension evaluates: allowedSwapper[pool][router] == true → passes.
7. Swap executes. Attacker receives output tokens.
   The allowlist provided zero protection.
```

Foundry test outline:
- Deploy `SwapAllowlistExtension`, configure it on a pool's `beforeSwap` hook.
- Call `extension.setAllowedToSwap(pool, address(router), true)`.
- From an address not in the allowlist, call `router.exactInputSingle(...)`.
- Assert the swap succeeds (no `NotAllowedToSwap` revert).
- Assert the attacker receives output tokens.
