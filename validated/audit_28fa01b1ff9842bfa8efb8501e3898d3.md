### Title
Missing `WithdrawCollateralV2` Handler in `processSlowModeTransactionImpl` Causes Silent Withdrawal Failure and Fee Loss — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

`EndpointTx.submitSlowModeTransactionImpl` accepts `WithdrawCollateralV2` slow-mode submissions and charges a slow-mode fee, but `processSlowModeTransactionImpl` has no handler for that transaction type. When the queued transaction is later executed, it silently reverts and is discarded. The user loses their slow-mode fee and their withdrawal is permanently dropped with no refund path.

---

### Finding Description

`IEndpoint.TransactionType` defines `WithdrawCollateralV2` as a first-class transaction type that extends `WithdrawCollateral` with a custom `sendTo` address and a forward-compatible `appendix` field. [1](#0-0) 

`submitSlowModeTransactionImpl` does **not** explicitly reject `WithdrawCollateralV2`. It falls into the `else` branch, charges the slow-mode fee, and enqueues the transaction: [2](#0-1) 

However, `processSlowModeTransactionImpl` handles only a fixed set of transaction types. `WithdrawCollateralV2` is absent from every branch and falls through to the terminal `else { revert(); }`: [3](#0-2) 

When `_executeSlowModeTransaction` runs the queued entry, it wraps the call in a `try/catch`. A revert from `processSlowModeTransactionImpl` is silently swallowed. The comment in the catch block explicitly notes that the former fund-return mechanism was removed: [4](#0-3) 

The net result: the slow-mode fee is consumed at submission time, the withdrawal is silently discarded at execution time, and there is no recovery path.

---

### Impact Explanation

1. **Fee loss** — the slow-mode fee (charged in `USDC` from the user's quote balance) is permanently lost.
2. **Silent withdrawal failure** — the user's collateral remains locked in the protocol with no on-chain signal that the withdrawal was dropped.
3. **Censorship-resistance bypass** — the slow-mode queue is the protocol's only censorship-resistance mechanism. A user whose sequencer-path `WithdrawCollateralV2` is being censored cannot fall back to slow mode. The V1 path (`WithdrawCollateral`) is still available via slow mode, but it does not support the `sendTo` field, so a user who needs to withdraw to a different address has no recourse.

The corrupted state delta is: `sequencerFee[productId]` is incremented (fee collected) while the corresponding `spotEngine` balance deduction and `withdrawPool` transfer never occur, leaving the user's balance permanently frozen relative to their intent.

---

### Likelihood Explanation

Any unprivileged user who calls `submitSlowModeTransaction` with a `WithdrawCollateralV2`-encoded payload triggers this path. The scenario is realistic: a user who wants to withdraw to a `sendTo` address different from their subaccount address (the only reason to use V2 over V1) and who is experiencing sequencer censorship or latency would naturally attempt the slow-mode path. The protocol gives no indication at submission time that the transaction cannot be executed.

---

### Recommendation

Either:

1. **Add a handler** for `WithdrawCollateralV2` in `processSlowModeTransactionImpl`, mirroring the existing `WithdrawCollateral` branch and passing `signedTx.tx.sendTo` to `clearinghouse.withdrawCollateral`.
2. **Explicitly reject** `WithdrawCollateralV2` in `submitSlowModeTransactionImpl` (as `DepositCollateral` is rejected with `revert()`), so users receive an immediate on-chain error rather than a silent future failure.

Option 1 is preferred because it preserves the censorship-resistance guarantee for V2 withdrawals.

---

### Proof of Concept

```
1. User calls Endpoint.submitSlowModeTransaction(
       abi.encodePacked(
           uint8(TransactionType.WithdrawCollateralV2),
           abi.encode(SignedWithdrawCollateralV2{...sendTo: 0xAlice...})
       )
   )

2. submitSlowModeTransactionImpl:
   - txType == WithdrawCollateralV2
   - Not DepositCollateral → does not revert
   - Not in owner-only list → falls to else branch
   - chargeSlowModeFee() deducts fee from user's quote balance  ← fee lost here
   - slowModeTxs[txCount++] = SlowModeTx{...}                  ← queued

3. After SLOW_MODE_TX_DELAY, executeSlowModeTransaction() is called.

4. _executeSlowModeTransaction → try this.processSlowModeTransaction(sender, tx)

5. processSlowModeTransactionImpl:
   - txType == WithdrawCollateralV2
   - No matching branch
   - else { revert(); }                                         ← reverts

6. catch block in _executeSlowModeTransaction:
   - gasleft() > 250000 and > gasRemaining/2 → no invalid()
   - "try return funds now removed" → nothing returned
   - Execution continues silently

Result:
  - User's slow-mode fee: GONE
  - User's collateral: still locked in SpotEngine
  - Withdrawal to 0xAlice: never executed
  - On-chain evidence of failure: none
```

                // the amount of gas left for execution is (63/64)**len(stack)

                // here that isn't perfect but covers our cases.

                // before we miss out of gas errors; 1/2 ~= (63/64)**44
