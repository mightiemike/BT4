Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end-user, enabling full allowlist bypass or blocking all router-based swaps â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` of `pool.swap()`. When users swap through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. This produces two mutually exclusive broken states: if the router is allowlisted, every address on the internet bypasses the guard; if it is not, every individually-allowlisted user is blocked from using the standard periphery.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` directly as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` forwards that value unchanged to every configured extension via `_callExtensionsInOrder`. `SwapAllowlistExtension.beforeSwap` then checks: [2](#0-1) 

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When the user goes through `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly with no forwarding of the original caller's address: [3](#0-2) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [4](#0-3) 

The `extensionData` forwarded to the pool is the raw user-supplied `params.extensionData` â€” the router never prepends the original caller's address. Consequently, the allowlist lookup is permanently keyed on the router contract address, not the end-user. The pool admin has no mechanism to distinguish individual users who arrive via the router.

## Impact Explanation

Two fund-impacting outcomes, both reachable by any unprivileged user:

**Outcome A â€” Full allowlist bypass (router is allowlisted):** A pool admin who allowlists the router so that vetted users can use the standard interface sets `allowedSwapper[pool][router] = true`. The `beforeSwap` guard then passes for every caller regardless of identity. Any address can swap on the restricted pool, defeating the entire access-control invariant. This is a direct admin-boundary break.

**Outcome B â€” Core swap functionality broken (router is not allowlisted):** If the admin does not allowlist the router, individually-allowlisted users who call `exactInputSingle` or any multi-hop route receive `NotAllowedToSwap` even though their address is listed. The only usable path is a direct `pool.swap()` call, which requires implementing `IMetricOmmSwapCallback` â€” not feasible for ordinary EOAs. This breaks the core swap flow for the pool's intended user base.

Both outcomes match the allowed impact gate: admin-boundary break and broken core pool functionality causing loss of funds or unusable swap flows.

## Likelihood Explanation

Medium. `SwapAllowlistExtension` is a production periphery contract explicitly designed for access-controlled pools. Any pool that deploys it and expects users to interact through `MetricOmmSimpleRouter` (the standard entry point) will encounter one of the two broken states on every swap. No special timing, mempool monitoring, or front-running is required â€” the bypass is structural and reproducible on every router call.

## Recommendation

The extension must check the original end-user identity, not the direct caller of `pool.swap()`. The cleanest fix is to have the router always prepend the original caller's address to `extensionData` before calling `pool.swap()`, and have `SwapAllowlistExtension.beforeSwap` decode and check that address instead of `sender`. This requires a coordinated change to both `MetricOmmSimpleRouter` and `SwapAllowlistExtension`. A partial alternative â€” checking `recipient` instead of `sender` for single-hop swaps â€” breaks for multi-hop routes where intermediate recipients are the router itself.

## Proof of Concept

**Bypass scenario (Outcome A):**
```solidity
// Pool admin allowlists the router so vetted users can use it
ext.setAllowedToSwap(address(pool), address(router), true);

// Bob (unauthorized) calls the router
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool), recipient: bob, zeroForOne: true, amountIn: 1000e18, ...
}));
// pool.swap() called with msg.sender = router
// _beforeSwap(sender=router) â†’ allowedSwapper[pool][router] = true â†’ PASSES
// Bob's swap executes on the restricted pool â€” allowlist bypassed
```

**Broken functionality scenario (Outcome B):**
```solidity
ext.setAllowedToSwap(address(pool), alice, true); // Alice is vetted, router is NOT allowlisted

router.exactInputSingle(ExactInputSingleParams({pool: address(pool), recipient: alice, ...}));
// pool.swap() called with msg.sender = router
// _beforeSwap(sender=router) â†’ allowedSwapper[pool][router] = false â†’ REVERTS NotAllowedToSwap
// Alice cannot use the router despite being individually allowlisted
```
