### Title
GC Removes Interval-Tail Block Required for Next Retarget, Permanently Bricking Difficulty Validation — (`contract/src/lib.rs` / `contract/src/litecoin.rs`)

---

### Summary

When `gc_threshold < difficulty_adjustment_interval + 2` (concretely `gc_threshold < 2018` for Litecoin), a single `submit_blocks` call whose batch ends exactly one block before a retarget boundary causes `run_mainchain_gc` to delete the interval-tail block that `get_next_work_required` must read for the immediately following retarget. Every subsequent attempt to submit that retarget-boundary header panics with `ERR_KEY_NOT_EXIST`, permanently disabling difficulty validation.

---

### Finding Description

**Entrypoint — `submit_blocks`** [1](#0-0) 

`num_of_headers = N` is captured, all N headers are inserted, then `run_mainchain_gc(N)` is called with `batch_size = N`.

**GC logic — `run_mainchain_gc`** [2](#0-1) 

When the chain is at steady-state (`amount_stored = gc_threshold`), submitting N headers makes `amount_stored = gc_threshold + N`, so `total_amount_to_remove = N` and `selected_amount_to_remove = min(N, N) = N`. The GC removes exactly N blocks from the bottom, advancing `mainchain_initial_blockhash` to height `H_old + N`.

**Retarget lookup — `get_next_work_required`** [3](#0-2) 

At every retarget boundary (when `(prev_height + 1) % 2016 == 0`), `blocks_to_go_back = 2016` and `first_block_height = prev_height − 2016`. The function calls `get_header_by_height(first_block_height)`: [4](#0-3) 

which panics with `ERR_KEY_NOT_EXIST` if the block is absent.

**Arithmetic of the break**

Let the chain be at steady-state: tip at height `T`, oldest at `H_old = T − gc_threshold + 1`. After submitting N headers the new oldest is:

```
H_new = T + N − gc_threshold + 1
```

The next retarget at height `R` needs the block at `R − 2017`. The invariant breaks when:

```
H_new > R − 2017
⟺  R < T + N + 2018 − gc_threshold
```

Because `R > T + N` (the retarget is after the batch), this is satisfiable only when:

```
gc_threshold < 2018
```

With `gc_threshold = 2016` the window collapses to `R = T + N + 1`: the batch must end exactly one block before a retarget boundary. That condition recurs naturally every 2016 blocks.

**Critical caveat — recommended production value is safe**

The recommended `gc_threshold = 52704` makes the condition `R < T + N − 50686`, which is impossible since `R > T + N`. **The vulnerability is only reachable when `gc_threshold < 2018`.** No minimum is enforced in `init`: [5](#0-4) 

A deployer who sets `gc_threshold = 2016` (e.g., to minimise storage costs, matching the retarget interval) exposes the contract to this bug.

---

### Impact Explanation

Once the interval-tail block is evicted, every future `submit_blocks` call that reaches a retarget boundary panics unconditionally. The light client can no longer advance past any retarget point; it is permanently bricked for difficulty validation without a contract upgrade.

---

### Likelihood Explanation

Requires `gc_threshold < 2018` — a non-recommended but valid and unenforced configuration. Given that `difficulty_adjustment_interval = 2016` is a natural "minimum storage" choice, a cost-conscious deployer could select it. Once deployed with that value, the failure condition (`batch ends at R − 1`) recurs every 2016 blocks and can be triggered by any legitimate trusted relayer catching up after downtime — no malicious intent required.

---

### Recommendation

Enforce a minimum `gc_threshold` in `init` and `run_mainchain_gc`:

```rust
require!(
    self.gc_threshold >= config.difficulty_adjustment_interval + 2,
    "gc_threshold must exceed difficulty_adjustment_interval + 1"
);
```

Alternatively, guard the GC so it never evicts the block at `tip_height − difficulty_adjustment_interval`:

```rust
let safe_floor = tip_blockheader.block_height
    .saturating_sub(config.difficulty_adjustment_interval);
let end_removal_height = end_removal_height.min(safe_floor);
```

### Proof of Concept

```
gc_threshold = 2016, difficulty_adjustment_interval = 2016
Genesis at height 2015 ((2015+1) % 2016 == 0), skip_pow_verification = true

State before call:
  tip    = 4030   (heights 2015..=4030 stored, exactly 2016 blocks)
  oldest = 2015

submit_blocks([h_4031, ..., h_6047])   // N = 2017 headers
  → all 2017 headers accepted (block at 2015 still present during submission)
  → run_mainchain_gc(2017):
       amount_stored = 6047 − 2015 + 1 = 4033
       total_to_remove = 4033 − 2016 = 2017
       removes heights 2015..4032  (i.e. 2015 through 4031 inclusive)
       new oldest = 4032

State after call:
  tip    = 6047
  oldest = 4032   ← height 4031 is gone

submit_blocks([h_6048])   // retarget boundary: (6047+1) % 2016 == 0
  → get_next_work_required:
       first_block_height = 6047 − 2016 = 4031
       get_header_by_height(4031) → ERR_KEY_NOT_EXIST → PANIC
```

The contract can never again process a retarget-boundary block.
