The code evidence is conclusive. Let me trace the exact logic:

**`keeper.BurnNFT`** (lines 141–161) requires **both** checks to pass on the **same address**:
1. `IsOwner(ctx, denomID, tokenID, owner)` — sender must own the NFT
2. `IsDenomCreator(ctx, denomID, owner)` — sender must be the denom creator

When the creator mints to Alice (`Recipient ≠ Sender`), no single address can satisfy both simultaneously:
- Alice calls `BurnNFT`: `IsOwner` passes, `IsDenomCreator` fails → `ErrUnauthorized`
- Creator calls `BurnNFT`: `IsDenomCreator` passes, `IsOwner` fails → `ErrUnauthorized`

The existing test at `keeper_test.go:172–180` explicitly confirms this exact scenario fails.

### Title
NFT Permanently Unburnable When Minted to Non-Creator Recipient — (`x/nft/keeper/keeper.go`)

### Summary
`keeper.BurnNFT` requires the caller to simultaneously be both the NFT owner and the denom creator. When a creator mints an NFT to a different recipient, no address can satisfy both conditions, making the token permanently unburnable and the supply counter permanently inflated.

### Finding Description
`keeper.BurnNFT` enforces two sequential guards:

```go
// x/nft/keeper/keeper.go:146–154
nft, err := k.IsOwner(ctx, denomID, tokenID, owner)
if err != nil {
    return err
}

_, err = k.IsDenomCreator(ctx, denomID, owner)
if err != nil {
    return err
}
``` [1](#0-0) 

`MintNFT` allows the creator to mint to an arbitrary `Recipient`:

```go
// x/nft/keeper/keeper.go:72–82
func (k Keeper) MintNFT(..., sender, owner sdk.AccAddress) error {
    _, err := k.IsDenomCreator(ctx, denomID, sender)
    ...
    return k.MintNFTUnverified(ctx, denomID, tokenID, tokenNm, tokenURI, tokenData, owner)
}
``` [2](#0-1) 

After `MintNFT(sender=creator, recipient=alice)`:
- Alice owns the token but is not the denom creator → `BurnNFT(alice, tokenID)` fails at `IsDenomCreator`
- Creator is the denom creator but does not own the token → `BurnNFT(creator, tokenID)` fails at `IsOwner`

The spec at `x/nft/spec/02_messages.md:112` documents this as intentional design ("Sender should be the Owner AND Creator"), but the design creates an unresolvable deadlock when ownership and creatorship diverge. [3](#0-2) 

The existing keeper test at lines 172–180 already demonstrates and accepts this failure as expected behavior, confirming the deadlock is real and reproducible: [4](#0-3) 

### Impact Explanation
The NFT supply counter (`GetTotalSupply`) is permanently inflated by 1 for every NFT minted to a non-creator recipient that is never voluntarily transferred back. The token is unburnable by any party without Alice's cooperation. If Alice loses key access or refuses to cooperate, the supply counter is permanently wrong and the token is permanently locked. `BurnNFTUnverified` (used only by IBC) bypasses the creator check but is not accessible via `MsgBurnNFT`. [5](#0-4) 

### Likelihood Explanation
The scenario is trivially reachable: any denom creator sending `MsgMintNFT` with `Recipient ≠ Sender` produces an unburnable token. This is a normal, documented use case (gifting NFTs to users). No special privileges or chain conditions are required beyond issuing a denom. [6](#0-5) 

### Recommendation
Decouple the burn authorization: allow burn if the caller is **either** the current owner **or** the denom creator (not both simultaneously). Alternatively, allow the denom creator to forcibly reclaim/burn any token in the collection regardless of current ownership, or restrict `MintNFT` so that `Recipient` must equal `Sender` (i.e., the creator must own all tokens before transferring them).

func (k Keeper) BurnNFT(ctx sdk.Context, denomID, tokenID string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	nft, err := k.IsOwner(ctx, denomID, tokenID, owner)
	if err != nil {
		return err
	}

	_, err = k.IsDenomCreator(ctx, denomID, owner)
	if err != nil {
		return err
	}

	k.deleteNFT(ctx, denomID, nft)
	k.deleteOwner(ctx, denomID, tokenID, owner)
	k.decreaseSupply(ctx, denomID)

	return nil
}
