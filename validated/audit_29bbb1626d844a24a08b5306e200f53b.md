Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of End User, Allowing Any User to Bypass Curated-Pool Swap Gate via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against a per-pool allowlist, where `sender` is `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the router contract is `msg.sender` to the pool, so the extension checks the router's address rather than the end user's. A pool admin who allowlists the router to enable router-mediated swaps for legitimate users simultaneously grants every user — including non-allowlisted ones — unconditional bypass of the allowlist.

## Finding Description
`MetricOmmPool.swap()` captures `msg.sender` and passes it as `sender` to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- whoever called pool.swap()
  recipient, ...
);
```

`ExtensionCalling._beforeSwap` encodes that `sender` into the call to each configured extension (L149-177). `SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When `MetricOmmSimpleRouter.exactInputSingle` is used, the router calls `pool.swap()` directly (L72-80), making the router the `sender` the extension checks. The same applies to `exactInput` (L104), `exactOutputSingle` (L136), and all hops in `exactOutput` / `_exactOutputIterateCallback` (L165, L220). The router passes `params.extensionData` through unchanged from the caller — there is no mechanism to inject the original `msg.sender` into `extensionData`. The pool admin therefore faces an impossible choice: not allowlisting the router blocks all router-mediated swaps for everyone; allowlisting the router makes `allowedSwapper[pool][router] == true`, which passes for every caller regardless of whether the actual end user is on the allowlist.

## Impact Explanation
Pools deploying `SwapAllowlistExtension` are curated pools — KYC-gated, institutional, or otherwise restricted. A non-allowlisted user who routes through `MetricOmmSimpleRouter` can execute swaps on such a pool whenever the router is allowlisted, directly exposing LP principal to trades from actors the allowlist was designed to exclude. This constitutes an admin-boundary break: an unprivileged path (standard router call) bypasses a pool-admin-configured access control gate, and broken core pool functionality (the allowlist guard) causes unauthorized access to LP liquidity.

## Likelihood Explanation
The router is the standard, documented periphery entry point for end users. Any pool that wants to support router-mediated swaps for its allowlisted users must allowlist the router. Once the router is allowlisted, the bypass is unconditional and requires no special setup from the attacker — only a standard `exactInputSingle` call through the router. The precondition (router allowlisted) is the natural operational state for any pool that intends to support router usage.

## Recommendation
The extension must gate on the economic actor, not the intermediary. The correct fix is to have the router populate an `originalSender` field in `extensionData` before calling the pool, and have `SwapAllowlistExtension.beforeSwap` decode and check that field instead of (or in addition to) `sender`. The pool's `onlyPool` guard in `BaseMetricExtension` already ensures only a registered pool can call the extension, so the extension can trust the decoded value if the pool is trusted. The router already stores `msg.sender` in transient storage for callback payment (`_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn)` at L71); the same value should be forwarded as part of `extensionData` so guards can key on the true originator.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured as a `beforeSwap` hook.
2. Admin calls `setAllowedToSwap(pool, userA, true)` to allowlist `userA`.
3. Admin calls `setAllowedToSwap(pool, router, true)` so that `userA` can use the router.
4. `userB` (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
5. The router calls `pool.swap(...)` with `msg.sender = router`.
6. The pool calls `extension.beforeSwap(router, ...)`.
7. The extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
8. `userB` successfully swaps on a pool that was supposed to block them.

If the admin does not allowlist the router (step 3 omitted), step 6 checks `allowedSwapper[pool][router]` → `false` → `userA`'s router swap also reverts, making the router unusable for all users including allowlisted ones. A Foundry integration test can confirm both branches by deploying the extension, configuring the allowlist, and asserting that a non-allowlisted address can complete a swap through the router when the router is allowlisted.
