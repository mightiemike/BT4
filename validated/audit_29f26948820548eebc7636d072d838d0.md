### Title
AuxPoW Chain Validation Bypassed via Attacker-Controlled `aux_data` Presence — (`File: contract/src/dogecoin.rs`)

---

### Summary

In the Dogecoin build of the BTC light client, `submit_block_header` selects its PoW verification path based solely on whether the caller-supplied `aux_data` argument is `Some` or `None`. It never cross-checks that selection against the block's own `version` field. A relayer can therefore submit a block whose version has the AuxPoW flag set (`version & 0x100 != 0`) while passing `aux_data = None`, causing the contract to silently skip the entire AuxPoW chain validation and fall through to a plain Scrypt hash check instead. This is the direct analog of the EIP-7702 bug: a runtime property that is supposed to reflect the block's identity (AuxPoW vs. standalone) is instead controlled by the caller, causing the wrong verification path to be taken.

---

### Finding Description

`submit_block_header` in `contract/src/dogecoin.rs` dispatches verification as follows:

```rust
if let Some(ref aux_data) = aux_data {
    self.check_aux(&block_header, aux_data);
} else {
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        format!("block should have correct pow")
    );
}
``` [1](#0-0) 

`check_aux` enforces the AuxPoW flag in the version field:

```rust
require!(
    block_header.version & BLOCK_VERSION_AUXPOW != 0,
    "Aux POW block does not have AuxPoW flag set in version"
);
``` [2](#0-1) 

This one-way guard means:
- Submitting `aux_data = Some(...)` for a non-AuxPoW block → correctly rejected (version flag missing).
- Submitting `aux_data = None` for an AuxPoW-flagged block → **silently accepted via the plain Scrypt path**, bypassing all of `check_aux`.

`check_aux` performs the following checks that are entirely skipped when `aux_data = None`:

1. Chain merkle proof: the Dogecoin block hash must appear in the parent Bitcoin block's coinbase merkle tree. [3](#0-2) 
2. Coinbase script contains the chain merkle root at the correct position. [4](#0-3) 
3. The parent block's chain ID must differ from Dogecoin's own chain ID (`0x0062`). [5](#0-4) 
4. The **parent** block's Scrypt hash (not the Dogecoin block's hash) must satisfy the target. [6](#0-5) 

When `aux_data = None`, only the Dogecoin block's own Scrypt hash is checked against the target — the merge-mining relationship with a parent chain is never verified.

The `BlockHeader` type alias for Dogecoin is `(Header, Option<AuxData>)`, so `aux_data` is a caller-supplied field with no on-chain enforcement of its presence relative to the version flag. [7](#0-6) 

---

### Impact Explanation

An attacker who can submit blocks (any unprivileged relayer path via `submit_blocks`) can inject a Dogecoin block header that:

- Has `version & BLOCK_VERSION_AUXPOW != 0` (claims to be merge-mined)
- Has no valid merge-mining relationship with any parent chain
- Is accepted by the contract because only the Dogecoin block's own Scrypt hash is checked

This corrupts the canonical chain state (`mainchain_tip_blockhash`, `mainchain_height_to_header`, `headers_pool`) stored in the light client. Downstream NEAR contracts calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` will receive results anchored to a fraudulent chain tip, enabling false SPV proofs. [8](#0-7) 

---

### Likelihood Explanation

On **Dogecoin mainnet** the attacker must mine a Scrypt hash below the current Dogecoin difficulty — expensive but not impossible for a well-resourced attacker, and the AuxPoW bypass removes the additional constraint of merge-mining with Bitcoin.

On **Dogecoin testnet** (`pow_allow_min_difficulty_blocks = true`), the minimum-difficulty rule allows blocks with `proof_of_work_limit_bits = 0x1e0fffff` when the timestamp gap exceeds `2 * pow_target_spacing`. At minimum difficulty the Scrypt target is trivially reachable, making the attack essentially free. [9](#0-8) 

The entry path is the public, payable `submit_blocks` call — no privileged role is required. [10](#0-9) 

---

### Recommendation

Add an explicit guard in `submit_block_header` that enforces the invariant: **if the block's version has the AuxPoW flag set, `aux_data` must be `Some`; if the flag is clear, `aux_data` must be `None`.**

```rust
const BLOCK_VERSION_AUXPOW: i32 = 0x100;
let is_auxpow_version = block_header.version & BLOCK_VERSION_AUXPOW != 0;
let has_aux_data = aux_data.is_some();

require!(
    is_auxpow_version == has_aux_data,
    "AuxPoW flag in version must match presence of aux_data"
);
```

This mirrors the existing one-way check inside `check_aux` and closes the asymmetry.

---

### Proof of Concept

1. Deploy the contract in Dogecoin testnet mode (`pow_allow_min_difficulty_blocks = true`).
2. Craft a `Header` with `version = 0x00620100` (chain ID `0x0062`, AuxPoW flag `0x100` set) and a timestamp more than 120 seconds after the previous block (triggering min-difficulty).
3. Mine a Scrypt hash below `0x1e0fffff` (trivial at min-difficulty).
4. Call `submit_blocks` with `[(header, None)]` — `aux_data = None`.
5. Observe: the contract accepts the block, stores it in `headers_pool`, and may promote it to the mainchain tip — without ever verifying a merge-mining relationship with any parent chain. [11](#0-10)

    pub(crate) fn submit_block_header(
        &mut self,
        header: (Header, Option<AuxData>),
        skip_pow_verification: bool,
    ) {
        let (block_header, aux_data) = header;

        let prev_block_header = self.get_prev_header(&block_header);
        let current_block_hash = block_header.block_hash();

        if !skip_pow_verification {
            self.check_target(&block_header, &prev_block_header);

            if let Some(ref aux_data) = aux_data {
                self.check_aux(&block_header, aux_data);
            } else {
                let pow_hash = block_header.block_hash_pow();
                // Check if the block hash is less than or equal to the target
                require!(
                    U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
                    format!("block should have correct pow")
                );
            }
        }

        let (current_block_computed_chain_work, overflow) = prev_block_header
            .chain_work
            .overflowing_add(work_from_bits(block_header.bits));
        require!(!overflow, "Addition of U256 values overflowed");

        let current_header = ExtendedHeader {
            block_header: block_header.clone().into_light(),
            block_hash: current_block_hash,
            chain_work: current_block_computed_chain_work,
            block_height: 1 + prev_block_header.block_height,
        };

        self.submit_block_header_inner(current_header, &prev_block_header);
    }

    #[pause]
    #[trusted_relayer]
    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

            .headers_pool
            .get(&args.tx_block_blockhash)
