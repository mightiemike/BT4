### Title
`ValidResourceBounds::AllResources` with zero L2/L1DataGas silently collapses to `L1Gas` in protobuf round-trip, producing a wrong transaction hash and rejecting valid declare transactions — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserialization of `ValidResourceBounds` uses a value-based heuristic: if both `l2_gas` and `l1_data_gas` are zero, the result is `L1Gas`; otherwise `AllResources`. A V3 transaction submitted with `AllResources { l2_gas: 0, l1_data_gas: 0 }` (a valid post-0.13.3 transaction that happens to set those fields to zero) is serialized to protobuf with all-zero L2/data-gas fields and then deserialized back as the pre-0.13.3 `L1Gas` variant. Because `get_tip_resource_bounds_hash` hashes a different number of resource felts for each variant (2 for `L1Gas`, 3 for `AllResources`), the recomputed hash diverges from the original. For declare transactions the conversion additionally hard-rejects the `L1Gas` variant with `DEPRECATED_RESOURCE_BOUNDS_ERROR`, dropping a valid transaction. The codebase itself acknowledges the collapse in a test comment.

---

### Finding Description

**Serialization (lossless in one direction):**

`From<ValidResourceBounds> for protobuf::ResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` serializes `L1Gas` by writing `l1_data_gas = ResourceBounds::default()` (all zeros): [1](#0-0) 

An `AllResources` transaction with `l2_gas = {0,0}` and `l1_data_gas = {0,0}` produces an **identical** protobuf wire encoding to an `L1Gas` transaction.

**Deserialization (lossy — the bug):**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` reconstructs the variant purely from values: [2](#0-1) 

When both fields are zero the result is `L1Gas`, regardless of whether the sender intended `AllResources`.

**Hash domain divergence:**

`get_tip_resource_bounds_hash` feeds a different number of resource felts into the Poseidon hash depending on the variant: [3](#0-2) 

- `L1Gas` → 2 resource felts: `[L1_GAS_packed, L2_GAS_packed(0)]`
- `AllResources` → 3 resource felts: `[L1_GAS_packed, L2_GAS_packed(0), L1_DATA_GAS_packed(0)]`

The Poseidon hash over 2 elements ≠ the Poseidon hash over 3 elements even when the extra element is zero. The transaction hash computed after the round-trip therefore differs from the hash the sender signed.

**Declare transaction hard-rejection:**

`TryFrom<protobuf::DeclareV3WithClass> for RpcDeclareTransactionV3` explicitly rejects the `L1Gas` variant: [4](#0-3) 

A valid declare transaction with zero L2/data-gas bounds is therefore dropped with `DEPRECATED_RESOURCE_BOUNDS_ERROR` by any node that receives it over P2P.

**Codebase acknowledgement:**

The consensus round-trip test explicitly works around the collapse by injecting a non-zero `l2_gas.max_amount`: [5](#0-4) 

The workaround is applied only in the test; production code has no equivalent guard.

---

### Impact Explanation

**For declare transactions (High — valid transaction rejected before sequencing):**  
Any `RpcDeclareTransactionV3` with `l2_gas = {0,0}` and `l1_data_gas = {0,0}` is accepted by the gateway (which computes the hash correctly using `AllResources`), propagated over P2P, and then hard-rejected by every receiving node with `DEPRECATED_RESOURCE_BOUNDS_ERROR`. The transaction can never be sequenced.

**For invoke / deploy-account transactions (High — wrong hash bound to executable payload):**  
The hash stored by the proposer (computed over 3 resource felts) differs from the hash any validator recomputes after deserializing the protobuf (computed over 2 resource felts). This breaks receipt commitment, block hash, and any hash-based deduplication or signature re-verification performed on the receiving side.

---

### Likelihood Explanation

The trigger is a V3 transaction where the user sets `l2_gas` and `l1_data_gas` both to zero — a natural choice for a transaction that only consumes L1 gas. No privileged access is required; any unprivileged user can submit such a transaction. The gateway accepts it (the hash is computed correctly there), so the failure is silent until P2P propagation.

---

### Recommendation

Remove the value-based heuristic. The protobuf wire format should carry an explicit discriminant (e.g., a boolean `is_all_resources` field, or a `oneof`) so the deserializer can reconstruct the correct variant without inspecting field values. Until the wire format is changed, the deserializer should default to `AllResources` whenever all three resource-limit fields are present in the message, reserving `L1Gas` only for messages that are missing the `l1_data_gas` field entirely (the legacy 0.13.2 case already handled by `unwrap_or_default`).

---

### Proof of Concept

```
1. Construct RpcInvokeTransactionV3 with:
     resource_bounds = AllResourceBounds {
         l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
         l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
         l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },
     }

2. Compute hash_A = get_invoke_transaction_v3_hash(tx, chain_id, version)
   → get_tip_resource_bounds_hash(AllResources{...}, tip)
   → Poseidon([tip, L1_GAS_packed, L2_GAS_packed(0), L1_DATA_GAS_packed(0)])  // 3 felts

3. Serialize tx → protobuf::ResourceBounds {
       l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
       l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },   // zero
       l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },   // zero
   }

4. Deserialize protobuf::ResourceBounds → ValidResourceBounds::L1Gas(l1_gas)
   (because l1_data_gas.is_zero() && l2_gas.is_zero())

5. Compute hash_B = get_invoke_transaction_v3_hash(tx', chain_id, version)
   → get_tip_resource_bounds_hash(L1Gas{...}, tip)
   → Poseidon([tip, L1_GAS_packed, L2_GAS_packed(0)])              // 2 felts

6. Assert hash_A ≠ hash_B  ← divergence confirmed

For declare: step 4 additionally triggers DEPRECATED_RESOURCE_BOUNDS_ERROR,
dropping the transaction entirely.
```

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
