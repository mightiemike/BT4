### Title
Missing Lower-Bound Check on `confirmations` Allows Zero-Confirmation Proof Acceptance — (`contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations: u64` with no lower-bound check. Passing `confirmations = 0` silently bypasses the confirmation-depth guard, causing the contract to attest that a transaction is "verified" even when it sits in the chain tip with zero confirmations and is therefore still subject to reorganization.

### Finding Description
`verify_transaction_inclusion` enforces two guards before returning a Merkle-proof result:

```rust
// guard 1 – upper bound only
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
// guard 2 – depth check
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [1](#0-0) 

When `args.confirmations = 0`:

- Guard 1 evaluates to `0 <= gc_threshold`, which is always `true` for any valid `u64`.
- Guard 2 evaluates to `(heaviest − target) + 1 >= 0`, which is always `true` for `u64` arithmetic.

Both guards pass unconditionally, and the function proceeds directly to Merkle-proof verification. If the proof is structurally valid, the function returns `true` — asserting the transaction is confirmed — regardless of how recently the block was submitted.

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after its coinbase-proof check, so it inherits the same flaw:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [2](#0-1) 

The `ProofArgs` struct places no constraint on `confirmations` at the type level:

```rust
pub struct ProofArgs {
    ...
    pub confirmations: u64,
}
``` [3](#0-2) 

### Impact Explanation
Any NEAR account that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` with `confirmations = 0` receives a `true` result for a transaction whose block is the current chain tip. That block has not been buried under any subsequent work and can be reorganized away. A recipient contract (bridge, payment gateway, cross-chain application) that trusts this return value would accept a payment or unlock funds for a transaction that may later be invalidated by a reorg, leading to direct financial loss.

### Likelihood Explanation
Both functions are public, pausable entry points reachable by any NEAR account without any role or stake requirement. The `confirmations` field is a plain `u64` in a Borsh/JSON-serialized struct — an integrator can trivially supply `0`. The deprecated `verify_transaction_inclusion` is still callable on-chain, and `verify_transaction_inclusion_v2` (the recommended replacement) routes through the same vulnerable path. Any downstream contract that omits its own minimum-confirmation enforcement is immediately exposed.

### Recommendation
Add an explicit lower-bound guard at the top of `verify_transaction_inclusion`:

```rust
require!(args.confirmations >= 1, "confirmations must be at least 1");
```

This mirrors the pattern used for the upper-bound check already present and closes the bypass for both the deprecated and current verification entry points.

### Proof of Concept
1. Deploy the contract with any valid genesis set and `gc_threshold = 1000`.
2. Submit one block header via `submit_blocks` so the chain tip advances to height `H`.
3. Call `verify_transaction_inclusion` with:
   - `tx_block_blockhash` = hash of the tip block (height `H`)
   - a valid Merkle proof for any transaction in that block
   - `confirmations = 0`
4. Guard 1: `0 <= 1000` → passes.
5. Guard 2: `(H − H) + 1 = 1 >= 0` → passes.
6. The function returns `true`, attesting the transaction is confirmed with zero buried blocks.
7. A recipient contract consuming this result would accept the transaction as finalized, even though the tip block carries no confirmation depth and is fully reorg-able. [4](#0-3)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
