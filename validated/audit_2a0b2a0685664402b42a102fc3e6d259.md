### Title
Retroactive `BonusApy` Governance Update Silently Zeroes All Unclaimed Bonus Rewards — (`x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`processEventsAndClaimBonus` fetches the **current** tier's `BonusApy` at claim time and applies it uniformly to every historical time segment since the position's last claim. Because `BonusApy` is never snapshotted at accrual time, a governance proposal that lowers (or zeroes) `BonusApy` via `SetTier` retroactively eliminates all bonus rewards that delegators have already earned but not yet claimed. Those tokens remain permanently stranded in the `RewardsPoolName` module account.

---

### Finding Description

`processEventsAndClaimBonus` in `x/tieredrewards/keeper/claim_rewards.go` opens by fetching the live tier:

```go
tier, err := k.getTier(ctx, pos.TierId)   // line 167 — current state, not historical
```

It then iterates over every validator event since `pos.LastEventSeq` and computes the bonus for each bonded segment using that single, live `tier` value:

```go
bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
```

`computeSegmentBonus` multiplies by `tier.BonusApy` directly:

```go
return tokens.Mul(tier.BonusApy).MulInt64(durationSeconds).QuoInt64(types.SecondsPerYear).TruncateInt()
```

The `tokensPerShare` rate **is** snapshotted per event, but `BonusApy` is **not**. Every past segment is therefore re-priced at whatever APY governance has set at the moment of the claim.

`SetTier` is the write path used by governance messages, genesis, and chain upgrades:

```go
// SetTier writes a tier after validation. Used by governance messages, genesis, and chain upgrades.
func (k Keeper) SetTier(ctx context.Context, tier types.Tier) error { ... }
```

A governance proposal that sets `BonusApy = 0` (or any value lower than the original) causes every subsequent call to `processEventsAndClaimBonus` — whether triggered by `ClaimTierRewards`, `TierUndelegate`, `TierRedelegate`, `AddToTierPosition`, `ExitTierWithDelegation`, or `ClearPosition` — to compute zero (or reduced) bonus for all historical segments, regardless of what rate was in effect when those segments were lived.

---

### Impact Explanation

A delegator who locked tokens at `BonusApy = 10 %` and accrued six months of bonus rewards without claiming will, after a governance reduction to `BonusApy = 0 %`, receive **zero** bonus tokens when they finally claim. The accrued amount is not refunded; it stays in the `RewardsPoolName` module account with no recovery path for the affected delegators. The principal delegation is unaffected, but the entire bonus entitlement for the elapsed period is silently destroyed.

The corrupted value is: **the bonus reward balance owed to every position in the modified tier that has not claimed since the last `BonusApy` change**.

---

### Likelihood Explanation

Governance proposals to tune reward rates are routine in proof-of-stake chains. A proposal framed as "reduce bonus APY going forward" will, due to this bug, also retroactively wipe unclaimed rewards for all existing positions. The proposer need not be malicious; the harm is a side-effect of a legitimate parameter adjustment. Any delegator who claims infrequently (e.g., once per quarter) is exposed every time the tier APY is changed.

---

### Recommendation

Snapshot `BonusApy` at the time each bonded segment begins, not at claim time. The existing validator-event infrastructure already snapshots `tokensPerShare`; the same pattern should be applied to `BonusApy`. Concretely:

1. Add a `BonusApy` field to `ValidatorEvent` (or introduce a dedicated `TierApyChangeEvent`).
2. Record the current `BonusApy` whenever `SetTier` is called.
3. In `processEventsAndClaimBonus`, use the snapshotted APY for each segment rather than the live tier value.

Alternatively, store a per-position `AccruedBonusCheckpoint` that accumulates the monetary value of bonus earned (not just the time/seq checkpoint), so that a later APY change cannot retroactively alter already-computed amounts.

---

### Proof of Concept

1. Governance sets `BonusApy = 10 %` for tier 1 via `SetTier`.
2. Alice calls `LockTier` / `CommitDelegationToTier` and enters tier 1.
3. Six months pass; Alice accrues ≈ 5 % of her staked amount as bonus rewards. She does not call `ClaimTierRewards`.
4. Governance passes a proposal setting `BonusApy = 0 %` for tier 1 via `SetTier`.
5. Alice calls `ClaimTierRewards` (or `TierUndelegate`).
6. `processEventsAndClaimBonus` fetches the current tier (`BonusApy = 0`).
7. `computeSegmentBonus` returns `0` for every historical segment.
8. Alice receives **zero** bonus tokens.
9. The tokens that should have been paid remain in `RewardsPoolName` with no mechanism to credit them to Alice.

```go
func (k Keeper) computeSegmentBonus(pos types.PositionState, tier types.Tier, segmentStart, segmentEnd time.Time, tokensPerShare math.LegacyDec) math.Int {
	if !pos.ExitUnlockAt.IsZero() && segmentEnd.After(pos.ExitUnlockAt) {
		segmentEnd = pos.ExitUnlockAt
	}

	if !segmentEnd.After(segmentStart) {
		return math.ZeroInt()
	}

	durationSeconds := int64(segmentEnd.Sub(segmentStart) / time.Second)
	if durationSeconds <= 0 {
		return math.ZeroInt()
	}

	tokens := pos.Delegation.Shares.Mul(tokensPerShare)

	return tokens.
		Mul(tier.BonusApy).
		MulInt64(durationSeconds).
		QuoInt64(types.SecondsPerYear).
		TruncateInt()
}
```
