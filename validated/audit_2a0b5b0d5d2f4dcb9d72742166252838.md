### Title
Linked Signer Can Unilaterally Replace Itself to Escalate Signing Authority Over Any Subaccount - (File: `core/contracts/EndpointTx.sol`)

### Summary
In `processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This permits the currently registered linked signer ‚Äî a lower-trust delegated key ‚Äî to sign and submit a new `LinkSigner` transaction that overwrites the `linkedSigners` mapping for the subaccount with any attacker-controlled address. Because `WithdrawCollateral` is also validated with `allowLinkedSigner = true`, the attacker's newly installed address can immediately drain the subaccount's collateral without any action or consent from the subaccount owner.

### Finding Description
In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch decodes a `SignedLinkSigner` transaction and calls `validateSignedTx` with the fifth argument set to `true` (allowLinkedSigner):

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // ‚Üê linked signer is accepted as valid signer
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
```

`validateSignedTx` resolves the allowed signer via `getLinkedSigner(sender)` and passes it to `Verifier.validateSignature`, which accepts either the subaccount owner address or the linked signer address as a valid recoverer. Therefore, the existing linked signer's ECDSA signature is sufficient to satisfy the check and overwrite `linkedSigners[subaccount]` with any new address.

The `WithdrawCollateral` branch in the same function is also validated with `allowLinkedSigner = true`, so the newly installed attacker address can immediately sign a withdrawal transaction and move all collateral out of the subaccount.

The `linkedSigners` mapping lives in `EndpointStorage` and is the sole authorization gate for all `allowLinkedSigner = true` operations. Corrupting it is equivalent to full subaccount takeover.

### Impact Explanation
A compromised or malicious linked signer can:
1. Sign a `LinkSigner` transaction naming an attacker-controlled address as the new signer.
2. The sequencer includes this in a `submitTransactionsChecked` batch; `processTransactionImpl` overwrites `linkedSigners[victim_subaccount]`.
3. The attacker's address signs a `WithdrawCollateral` (or `WithdrawCollateralV2` with `sendTo == address(0)`) transaction.
4. All collateral is transferred to the attacker.

The subaccount owner receives no on-chain notification and has no blocking mechanism. The nonce is consumed by the attacker's `LinkSigner` transaction, so the owner cannot race to revoke it via the fast path. The slow-mode `LinkSigner` path (`processSlowModeTransactionImpl`) requires `validateSender` (owner address match), but the fast path has no such restriction.

Asset delta: full balance of any collateral product held by the victim subaccount.

### Likelihood Explanation
Any party that has ever been granted linked-signer status for a subaccount ‚Äî including API keys, trading bots, or third-party integrations ‚Äî can execute this attack unilaterally. The attack requires only a valid ECDSA signature from the current linked signer and sequencer inclusion, both of which are normal protocol operations. No privileged access, governance capture, or key leakage is required beyond the linked signer key itself, which is the exact key the protocol already trusts for routine trading.

### Recommendation
Change the `allowLinkedSigner` flag to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. Changing the linked signer is a critical authorization mutation that must require a signature from the subaccount owner (the address embedded in the high 20 bytes of the `sender` subaccount bytes32), not from the currently delegated key. The slow-mode path already enforces this correctly via `validateSender`.

    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
