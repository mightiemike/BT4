### Title
Deprecated `verify_transaction_inclusion` Accepts Internal Merkle Nodes as Valid Transactions, Enabling Proof Forgery — (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (v1) is still a callable public function on the NEAR contract. It verifies only that a supplied `tx_id` hashes up to the stored `merkle_root`, but performs no check to confirm that `tx_id` is a leaf node (an actual transaction) rather than an internal Merkle tree node. An unprivileged caller can supply an internal node as `tx_id` with a shortened proof path, and the function returns `true`. Any downstream DApp that relies on this function to gate asset release or state changes can be deceived into accepting a fabricated transaction inclusion proof.

### Finding Description

The root cause is in `verify_transaction_inclusion` in `contract/src/lib.rs`: [1](#0-0) 

The function is marked `#[deprecated]` but carries no access restriction — it is decorated only with `#[pause]`, meaning any unprivileged NEAR account can call it whenever the contract is not paused. [2](#0-1) 

The sole verification performed is: [3](#0-2) 

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` simply walks the supplied proof hashes upward, combining them with `double_sha256`, and returns whatever root results: [4](#0-3) 

There is no check that the supplied `tx_id` is a 32-byte leaf (a real transaction TXID). In Bitcoin's Merkle tree every internal node is also a 32-byte `double_sha256` value — the hash of its two 64-byte-concatenated children. An attacker can therefore supply an internal node at depth `d` as `tx_id`, provide a proof of length `(tree_depth − d)` instead of `tree_depth`, and `compute_root_from_merkle_proof` will still arrive at the correct `merkle_root`. The function returns `true`.

The v2 function fixes this by first verifying a coinbase proof at index 0 (which anchors the proof depth to the real tree depth), then calling v1: [5](#0-4) 

But v1 remains independently callable, so the fix in v2 does not protect callers of v1.

The contract's own docstring acknowledges the gap but defers responsibility to callers: [6](#0-5) 

This is the direct analog to the external report: just as the Swap validator performs every Join-Split check except the elliptic-curve pairing (the binding commitment check), `verify_transaction_inclusion` performs every SPV check except the leaf-node binding check (the coinbase depth anchor). In both cases the missing check is the one that cryptographically binds the submitted value to a real object in the protocol.

### Impact Explanation

Any NEAR DApp that calls `verify_transaction_inclusion` to gate an action (e.g., releasing wrapped BTC, crediting a deposit, unlocking a cross-chain swap) can be made to accept a fabricated Bitcoin transaction. The attacker does not need to perform any real Bitcoin transaction. They only need:

1. A Bitcoin block that is already accepted by the light client (publicly available data).
2. The block's Merkle tree structure (derivable from any Bitcoin full node or block explorer).
3. An internal Merkle node from that tree and a shortened proof path.

The result is that the DApp's trust assumption — "this function returned `true`, therefore the Bitcoin transaction occurred" — is broken. Assets or state changes gated on that assumption can be triggered without any real on-chain Bitcoin activity.

### Likelihood Explanation

- `verify_transaction_inclusion` is a public, permissionless NEAR function callable by any account.
- The attack requires only public Bitcoin blockchain data; no privileged keys, no leaked secrets.
- The 64-byte transaction Merkle forgery technique is well-documented (referenced in the contract's own comments: https://www.bitmex.com/blog/64-Byte-Transactions).
- Any integrator that has not yet migrated from v1 to v2 is immediately vulnerable.
- The function has not been removed, only deprecated, so it remains a live attack surface.

### Recommendation

Remove `verify_transaction_inclusion` (v1) entirely from the contract, or add an explicit `env::panic_str` / `require!(false, ...)` body so that any call to it reverts unconditionally. Deprecation alone does not close the attack surface because the function remains callable on-chain. All callers must be migrated to `verify_transaction_inclusion_v2`, which anchors proof depth via the coinbase check.

### Proof of Concept

1. Identify any Bitcoin block already accepted by the light client. Call its Merkle root `R` and suppose it has ≥ 2 transactions with leaf hashes `T0`, `T1`, …
2. Compute the internal node `N = double_sha256(T0 ‖ T1)`. This is a valid 32-byte hash.
3. Construct a Merkle proof for `N` at position 0 in the *reduced* tree (the tree whose leaves are the level-1 internal nodes). This proof has length `tree_depth − 1` instead of `tree_depth`.
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node, not a real TXID)
   - `tx_block_blockhash` = the accepted block's hash
   - `tx_index = 0`
   - `merkle_proof` = the shortened proof
   - `confirmations = 1`
5. `compute_root_from_merkle_proof(N, 0, shortened_proof)` produces `R` (the real Merkle root), so the function returns `true`.
6. Any DApp that trusted this return value now believes a transaction with TXID `N` was confirmed in that block, even though no such transaction exists.

        since = "0.5.0",

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
