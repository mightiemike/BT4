### Title
`verify_transaction_inclusion` Unusable for Single-Transaction Blocks Due to Unconditional Empty-Proof Guard ‚Äî (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (and by extension `verify_transaction_inclusion_v2`) contains a guard that unconditionally rejects an empty `merkle_proof`. For a Bitcoin block containing only one transaction (e.g., a coinbase-only block), the Merkle proof is legitimately empty because the transaction hash *is* the Merkle root. The guard makes this valid case permanently unreachable, causing the contract to panic instead of returning `true`.

---

### Finding Description

Inside `verify_transaction_inclusion`, line 315 unconditionally requires a non-empty Merkle proof:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

The proof is then passed to `compute_root_from_merkle_proof`:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

`compute_root_from_merkle_proof` is implemented as a simple loop over the proof elements:

```rust
for proof_hash in merkle_proof {
    ...
    current_position /= 2;
}
current_hash  // returned unchanged if proof is empty
``` [3](#0-2) 

When `merkle_proof` is empty, the loop body never executes and the function returns `transaction_hash` unchanged. Therefore, for a single-transaction block where `tx_id == merkle_root`, calling `compute_root_from_merkle_proof(tx_id, 0, &[])` would correctly return `tx_id`, and the equality check against `header.block_header.merkle_root` would return `true`. The logic is sound ‚Äî but the `require!` guard at line 315 fires first and panics, making this correct code path permanently unreachable.

`verify_transaction_inclusion_v2` is equally affected: it validates the coinbase proof (which also passes with an empty proof for a single-transaction block), then delegates to the deprecated `verify_transaction_inclusion`, which panics:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [4](#0-3) 

---

### Impact Explanation

Any NEAR caller ‚Äî a smart contract consuming verification results or a direct user ‚Äî that submits a valid Merkle inclusion proof for a transaction in a single-transaction Bitcoin block will always receive a panic (`"Merkle proof is empty"`) instead of `true`. The contract's core verification function is broken for this entire class of valid Bitcoin blocks. Single-transaction blocks (coinbase-only) are a normal part of the Bitcoin protocol and occur regularly on mainnet.

---

### Likelihood Explanation

The entry path is fully unprivileged: `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are public, `#[pause]`-gated (but not role-gated) methods callable by any NEAR account. Any relayer or consumer contract that attempts to verify a coinbase-only block's transaction will trigger the panic deterministically. No special conditions, leaked keys, or social engineering are required.

---

### Recommendation

Add an early-escape check before the `require!` guard. If `merkle_proof` is empty and `tx_id == header.block_header.merkle_root`, return `true` immediately (the transaction is the sole leaf and equals the root). Otherwise, retain the existing non-empty requirement:

```rust
// Early exit: single-transaction block ‚Äî proof is legitimately empty
if args.merkle_proof.is_empty() {
    return args.tx_id == header.block_header.merkle_root;
}
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
```

This mirrors the fix recommended in the external report: add an early-escape for the zero/empty input case rather than unconditionally rejecting it.

---

### Proof of Concept

1. Deploy the contract (Bitcoin feature flag) with any valid genesis.
2. Submit a real Bitcoin block whose `merkle_root` equals a single transaction hash (coinbase-only block).
3. Call `verify_transaction_inclusion` with:
   - `tx_id` = that transaction hash
   - `tx_block_blockhash` = the block hash
   - `tx_index` = 0
   - `merkle_proof` = `[]` (empty ‚Äî correct for a single-tx block)
   - `confirmations` = 1
4. The call panics with `"Merkle proof is empty"` at line 315, even though the proof is mathematically valid and `compute_root_from_merkle_proof(tx_id, 0, &[])` would have returned `tx_id == merkle_root = true`. [5](#0-4) [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
