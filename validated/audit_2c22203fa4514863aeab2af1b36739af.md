### Title
Stale Sentinel `node_map` Entry Not Removed on `restore()`, Corrupting Back-Reference Parent Tracking — (`File: src/serde/tree_cache.rs`)

---

### Summary

`TreeCache::restore()` does not remove the sentinel node's existing `node_map` entry when the saved checkpoint was taken before the sentinel was ever inserted into the map. After a restore, the sentinel retains a stale `NodeEntry` index. The next call to `update()` reads parents from this stale entry and unconditionally transfers them to the new root, silently corrupting the parent graph used by `find_path()` to emit back-reference paths during CLVM serialization.

---

### Finding Description

`TreeCache` maintains a `node_map: HashMap<NodePtr, u32>` that maps each CLVM node to an index in `node_entries`. A special *sentinel* node marks an unknown subtree placeholder. The checkpoint/restore pair (`undo_state` / `restore`) is used by the incremental serializer to roll back speculative serialization state.

**`undo_state()`** (lines 151–161) captures the sentinel's current `node_map` index:

```rust
let sentinel_entry = match self.sentinel_node {
    Some(sentinel) => self.node_map.get(&sentinel).cloned(),
    None => None,
};
```

If the sentinel has not yet been inserted into `node_map` at checkpoint time, `sentinel_entry` is `None`.

**`restore()`** (lines 163–180) conditionally re-inserts the sentinel:

```rust
if let Some(sentinel_entry) = st.sentinel_entry {
    self.node_map
        .insert(self.sentinel_node.unwrap(), sentinel_entry);
}
// ← no `else` branch: stale entry is never removed
```

When `sentinel_entry` is `None`, any sentinel entry that was added to `node_map` *after* the checkpoint is silently left in place.

**`update()`** (lines 182–191) then reads parents from whatever `node_map` currently says about the sentinel:

```rust
if let Some(placement) = self.sentinel_node {
    if let Some(idx) = self.node_map.get(&placement) {
        root_parents.append(&mut self.node_entries[*idx as usize].parents);
    }
};
```

Because `node_entries` is append-only and never trimmed on restore, the stale `NodeEntry` (with its post-checkpoint parent links) is still live. Its parents are unconditionally drained and appended to the new root's parent list.

The unconditional overwrite at line 215 then replaces the stale sentinel entry:

```rust
self.node_map.insert(node, idx);   // overwrites without checking existing entry
```

This mirrors the original bug exactly: just as `acceptBidForName` transfers a name to a destination NFT without checking whether it already holds one, `update()` transfers parents from the sentinel's existing entry to the new root without checking whether that entry is a stale post-checkpoint artifact.

---

### Impact Explanation

`find_path()` uses the parent links in `node_entries` to walk from a target node back to the serialization stack root, producing a back-reference path (an environment lookup integer). Spurious parents injected from the stale sentinel entry cause `find_path()` to explore incorrect traversal branches. It may:

- Return a path that points to the wrong subtree in the serialized output.
- Cause the serializer to emit a back-reference byte sequence that, when deserialized by another node, resolves to a different CLVM subtree.

A serialized CLVM program whose back-references resolve differently on different nodes produces a different tree hash and a different execution result — a **consensus divergence**. In the Chia network, this means a coin spend accepted by one full node may be rejected by another, or two nodes may compute different puzzle hashes for the same spend bundle.

---

### Likelihood Explanation

The trigger requires:
1. A `TreeCache` constructed with a non-`None` sentinel node (normal for incremental serialization).
2. `undo_state()` called before the first `update()` — so the sentinel is not yet in `node_map`.
3. `update()` called, inserting the sentinel into `node_map` with a fresh `NodeEntry`.
4. `restore()` called with the checkpoint from step 2 — leaving the stale entry in place.
5. A second `update()` call — draining the stale parents into the new root.

The incremental serializer (`src/serde/incremental.rs`) uses both `undo_state()` and `restore()` (8 call sites found by grep). Whether the exact ordering above is reachable depends on the incremental serializer's control flow, which could not be fully read in this analysis. The sequence is plausible in any speculative/rollback serialization path where a sentinel-bearing `TreeCache` is checkpointed before its first `update()`.

---

### Recommendation

In `restore()`, when `sentinel_entry` is `None` and `self.sentinel_node` is `Some(s)`, explicitly remove the sentinel's current `node_map` entry if one exists:

```rust
self.serialized_nodes = st.serialized_nodes;
match st.sentinel_entry {
    Some(idx) => {
        self.node_map.insert(self.sentinel_node.unwrap(), idx);
    }
    None => {
        if let Some(s) = self.sentinel_node {
            self.node_map.remove(&s);
        }
    }
}
```

This ensures the sentinel's `node_map` state after restore exactly matches what it was at checkpoint time, preventing stale parents from being transferred to the next root.
