### Title
Bonus Accrual Checkpoint Advanced Before Pool Check in `slashRedelegationPosition` Causes Permanent Bonus Loss — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

### Summary

`processEventsAndClaimBonus` advances `LastBonusAccrual` (and consumes validator event references) **before** checking whether the rewards pool has sufficient balance. When called from `slashRedelegationPosition` with an empty pool, the error `ErrInsufficientBonusPool` is silently swallowed, but the position is then persisted with the already-advanced checkpoint. The pre-slash bonus segment is permanently unrecoverable.

### Finding Description

**Step 1 — Checkpoint advancement precedes pool check.**

In `processEventsAndClaimBonus`:

- Lines 172–198 iterate validator events, accumulate `totalBonus`, advance `pos.UpdateLastEventSeq`, and decrement event reference counts.
- Line 215 calls `applyBonusAccrualCheckpoint(&pos.Position, blockTime)`, which sets `pos.LastBonusAccrual = blockTime`.
- Line 217 calls `pos.UpdateLastKnownBonded(bonded)`.
- Only at line 230 is `sufficientBonusPoolBalance` checked — **after** all state mutations to `pos` have already occurred. [1](#0-0) 

**Step 2 — `slashRedelegationPosition` silently swallows the error but persists the mutated position.**

`slashRedelegationPosition` passes `&pos` (a pointer) to `processEventsAndClaimBonus`. When `ErrInsufficientBonusPool` is returned, the error is caught and logged but not propagated. The function then falls through to `setPositionWithState`, which persists the already-mutated `pos` — including the advanced `LastBonusAccrual`. [2](#0-1) 

**Step 3 — Subsequent `MsgClaimTierRewards` sees no elapsed segment.**

Because `LastBonusAccrual` was advanced to the slash block time, a later call to `processEventsAndClaimBonus` via `MsgClaimTierRewards` computes `segmentStart = pos.LastBonusAccrual` equal to the slash time. The pre-slash accrual window is gone; `computeSegmentBonus` returns zero for it. [3](#0-2) 

### Impact Explanation

The bonus rewards owed for the bonded segment between the position's previous `LastBonusAccrual` and the slash block time are permanently destroyed. The pool is not debited (no payment was made), but the owner can never claim those rewards because the checkpoint has moved past the segment. This is a direct, irreversible loss of bonus rewards the owner is entitled to.

### Likelihood Explanation

The preconditions are realistic and reachable without any privileged access:

1. A user creates a tier position and redelegates it (`MsgTierRedelegate`), creating a redelegation mapping with a non-zero `unbondingID`.
2. The source validator is slashed during the redelegation maturation window — a normal chain event.
3. The rewards pool is transiently empty — possible if the pool has not been refilled recently or was drained by concurrent claims.

All three conditions can co-occur in normal chain operation. No governance or operator compromise is required.

### Recommendation

Move `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` to execute **only after** a successful pool balance check and payment. If the pool is insufficient, return the error without mutating the position's checkpoint fields, so the segment remains claimable once the pool is refilled.

Alternatively, in `slashRedelegationPosition`, reset `pos.LastBonusAccrual` and `pos.LastEventSeq` to their pre-call values before persisting when `ErrInsufficientBonusPool` is caught, so the segment is not lost.
