### Title
Stale `byte_count` in `op_keccak256` Loop Allows Cost-Limit Bypass — (`File: src/keccak256_ops.rs`)

### Summary

`op_keccak256` accumulates byte costs in a separate `byte_count` variable that is updated **after** the per-iteration `check_cost` call. The guard therefore always checks against the byte total from the **previous** iteration, not the current one. This is the direct analog of the Casimir bug: a running total is accumulated across loop iterations but the enforcement check uses a stale snapshot of that total, allowing more work to be done than the budget permits.

### Finding Description

In `src/keccak256_ops.rs`, `op_keccak256` uses two separate accumulators: `cost` (for per-arg cost) and `byte_count` (for per-byte cost). The per-iteration guard checks `cost + byte_count * KECCAK256_COST_PER_BYTE`, but `byte_count` is only incremented **after** the guard:

```rust
// src/keccak256_ops.rs lines 24-35
while let Some((arg, rest)) = a.next(input) {
    input = rest;
    cost += KECCAK256_COST_PER_ARG;
    check_cost(
        cost + byte_count as Cost * KECCAK256_COST_PER_BYTE,  // ← stale byte_count
        max_cost,
    )?;
    let blob = atom(a, arg, "keccak256")?;
    byte_count += blob.as_ref().len();   // ← updated AFTER the guard
    hasher.update(blob);
}
cost += byte_count as Cost * KECCAK256_COST_PER_BYTE;  // ← byte cost added only after loop
``` [1](#0-0) 

In iteration N the guard evaluates `cost + (bytes from iterations 1..N-1) * 2`, missing the bytes of the blob fetched in iteration N itself. The actual byte cost for that blob is only folded into `cost` after the loop exits. This is structurally identical to the Casimir pattern: `withdrawnEffectiveBalance` was adjusted only after the `while` loop, so each iteration's fulfillability check used a stale balance.

Compare with the correctly written `op_sha256`, which adds the byte cost to `cost` **before** calling `check_cost`:

```rust
// src/more_ops.rs lines 400-405
cost += SHA256_COST_PER_ARG;
let blob = atom(a, arg, "sha256")?;
cost += blob.as_ref().len() as Cost * SHA256_COST_PER_BYTE;
check_cost(cost, max_cost)?;
``` [2](#0-1) 

`op_keccak256` does not follow this pattern.

### Impact Explanation

An attacker-controlled CLVM program can call `keccak256` (opcode 62, enabled by `ClvmFlags::ENABLE_KECCAK_OPS_OUTSIDE_GUARD` or inside a softfork guard) with a list of large atoms. Because the per-iteration guard uses the stale `byte_count`, the operator will hash one extra large blob beyond what the remaining `max_cost` budget should permit before the guard fires. The operator then returns the correct (over-budget) cost, and the interpreter catches `CostExceeded` on the next dispatch iteration — but the hashing work has already been performed. This is an **undercharged execution** / cost-enforcement bypass: a node is forced to do more cryptographic work than the cost budget authorises, constituting a targeted denial-of-service against full nodes. [3](#0-2) 

### Likelihood Explanation

`keccak256` is reachable from any attacker-supplied CLVM byte string when the dialect flag `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` is set, or unconditionally inside a softfork guard. The attacker only needs to craft a list of large atoms as arguments. No special privileges or configuration are required beyond submitting a transaction to the mempool. [4](#0-3) 

### Recommendation

Follow the pattern used by `op_sha256`: add the per-byte cost to `cost` immediately after reading the blob, then call `check_cost` on the updated `cost`, and remove the post-loop adjustment:

```rust
while let Some((arg, rest)) = a.next(input) {
    input = rest;
    cost += KECCAK256_COST_PER_ARG;
    let blob = atom(a, arg, "keccak256")?;
    cost += blob.as_ref().len() as Cost * KECCAK256_COST_PER_BYTE;
    check_cost(cost, max_cost)?;
    hasher.update(blob);
}
// remove: cost += byte_count as Cost * KECCAK256_COST_PER_BYTE;
``` [5](#0-4) 

### Proof of Concept

Set `max_cost` to a value that is just above `KECCAK256_BASE_COST + KECCAK256_COST_PER_ARG * 2` but below `KECCAK256_BASE_COST + KECCAK256_COST_PER_ARG * 2 + N * KECCAK256_COST_PER_BYTE` for a large atom of length N. Call `op_keccak256` with two arguments: a zero-length atom followed by an atom of length N. On iteration 1, `byte_count = 0`, so the guard checks only `base + per_arg`, which passes. The blob (length 0) is consumed and `byte_count` stays 0. On iteration 2, `byte_count` is still 0 (stale), so the guard again checks only `base + 2*per_arg`, which passes even though the true cost including the N-byte blob would exceed `max_cost`. The N-byte blob is hashed. After the loop, `cost += N * 2` produces a value exceeding `max_cost`, but the hashing work is already done. The interpreter catches `CostExceeded` only on the next dispatch step. [6](#0-5)
