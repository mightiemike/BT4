### Title
Spot Depositors Can Avoid Socialization Losses via Permissionless `submitFastWithdrawal` — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`submitFastWithdrawal` in `BaseWithdrawPool.sol` is a permissionless on-chain function callable by any holder of a valid protocol-signed withdrawal transaction. When a socialization event reduces a depositor's `SpotEngine` balance below their pending withdrawal amount, the depositor can call `submitFastWithdrawal` directly on-chain to extract tokens from the `WithdrawPool` before the sequencer processes their `WithdrawCollateral` transaction. The sequencer's subsequent processing of that transaction fails the health check and reverts — leaving the `WithdrawPool` drained and the depositor's `SpotEngine` balance intact, effectively transferring the socialization loss onto `WithdrawPool` liquidity providers.

---

### Finding Description

`submitFastWithdrawal` is callable by any unprivileged user who possesses a valid verifier-signed withdrawal transaction: [1](#0-0) 

The only guards are replay prevention (`markedIdxs`) and a lower-bound index check (`idx > minIdx`). There is no check that the originating subaccount's `SpotEngine` balance still covers the withdrawal amount at call time. [2](#0-1) 

When the sequencer later processes the corresponding `WithdrawCollateral` transaction via `Clearinghouse.withdrawCollateral`, it:
1. Transfers tokens from the Clearinghouse to the `WithdrawPool` (`handleWithdrawTransfer`)
2. Calls `submitWithdrawal`, which returns early because `markedIdxs[idx]` is already `true`
3. Reduces the `SpotEngine` balance
4. Performs a health check — which now fails because socialization already reduced the balance [3](#0-2) 

On revert, the Clearinghouse's token transfer to the `WithdrawPool` is also reverted, but the tokens already paid out by `submitFastWithdrawal` are not. The `WithdrawPool` is left short by the full withdrawal amount.

Socialization is triggered inside `_finalizeSubaccount` in `ClearinghouseLiq.sol` when a liquidatee's insurance-adjusted quote balance is exhausted: [4](#0-3) 

This reduces all depositors' effective balances via the global `cumulativeDepositsMultiplierX18` in `SpotEngineState`, making any pre-signed withdrawal for the full balance immediately undercollateralized.

---

### Impact Explanation

A depositor with a pre-signed fast-withdrawal transaction for amount `X` can, after observing a socialization event that reduces their balance to `X' < X`:

1. Call `submitFastWithdrawal` on-chain → receive `X` tokens from the `WithdrawPool`
2. The sequencer's `WithdrawCollateral` processing reverts (health check fails at `X' - X < 0`)
3. The depositor retains `X` tokens in their wallet **and** `X'` in their `SpotEngine` balance (since the balance-reduction step was reverted)
4. The `WithdrawPool` LPs absorb the `X`-token shortfall

The depositor has avoided their proportional socialization loss entirely and can subsequently withdraw `X'` again through a new request, extracting a total of `X + X'` against a post-socialization entitlement of only `X'`.

---

### Likelihood Explanation

**Medium.** The preconditions are:
- The user must hold a valid protocol-signed withdrawal transaction (routine in the fast-withdrawal flow)
- A socialization event must occur between signing and sequencer processing (rare but realistic during large liquidations)
- The user must call `submitFastWithdrawal` on-chain before the sequencer marks the index (requires monitoring and timely reaction, but the function is permissionless)

The `WithdrawPool` is explicitly designed to hold pre-funded LP liquidity for fast withdrawals, so the pool balance precondition is satisfied by design.

---

### Recommendation

Mirror the fix applied in the referenced report: before executing a fast withdrawal, verify that the originating subaccount's current `SpotEngine` balance (post any pending socialization) is sufficient to cover the withdrawal amount. Concretely, add a call from `submitFastWithdrawal` into the `SpotEngine` (or `Clearinghouse`) to assert that the subaccount's balance is `>= transferAmount` at the time of the on-chain call. If the balance is insufficient, revert. This closes the window between socialization and sequencer processing.

---

### Proof of Concept

1. Depositor holds 1 000 USDC in `SpotEngine` balance.
2. Depositor requests a fast withdrawal of 1 000 USDC; the protocol's `Verifier` signs it at `idx = N`.
3. A large liquidation is finalized by the sequencer; `spotEngine.socializeSubaccount` reduces the depositor's balance to 800 USDC.
4. Depositor observes the socialization on-chain and calls `submitFastWithdrawal(N, signedTx, sigs)` directly — `idx > minIdx` passes, `markedIdxs[N]` is `false`. The `WithdrawPool` transfers 1 000 USDC to the depositor.
5. Sequencer processes `WithdrawCollateral` for `idx = N`:
   - `handleWithdrawTransfer` sends 1 000 USDC from Clearinghouse → `WithdrawPool`
   - `submitWithdrawal` sees `markedIdxs[N] = true`, returns early
   - `spotEngine.updateBalance` sets balance to `800 - 1000 = -200 USDC`
   - `require(getHealth(...) >= 0)` → **REVERT**
6. All state changes in step 5 are rolled back; the Clearinghouse retains its 1 000 USDC.
7. **Final state:** Depositor holds 1 000 USDC in wallet + 800 USDC in `SpotEngine`; `WithdrawPool` is drained by 1 000 USDC; depositor can request a second withdrawal for 800 USDC. [1](#0-0) [5](#0-4) [6](#0-5)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
