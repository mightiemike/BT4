### Title
Dogecoin Fork Difficulty Miscalculation via Mainchain-Only `get_header_by_height` Lookup — (`contract/src/dogecoin.rs`)

---

### Summary

`get_next_work_required` in `dogecoin.rs` calls `get_header_by_height(height_first)` to obtain the "first block" timestamp for Digishield retargeting. The implementation of `get_header_by_height` unconditionally reads from `mainchain_height_to_header`, the mainchain-only height index. When validating a fork block whose ancestor at `height_first` diverges from the mainchain, the contract uses the **mainchain** block's timestamp instead of the **fork** ancestor's timestamp. This produces an incorrect `expected_bits`, allowing fork blocks with bits that would fail correct difficulty validation to pass `check_pow` and be stored — and potentially promoted to `mainchain_tip_blockhash` via `reorg_chain`.

The TODO comment at `dogecoin.rs:291` explicitly acknowledges this is unresolved.

---

### Finding Description

**Root cause — `get_header_by_height` is mainchain-only:** [1](#0-0) 

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // ← always mainchain index
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

**The call site with the acknowledged TODO:** [2](#0-1) 

**Dogecoin mainnet config — `difficulty_adjustment_interval = 1`, `pow_target_timespan = 60`:** [3](#0-2) 

Because `difficulty_adjustment_interval = 1` for mainnet Dogecoin (and for heights ≥ 145,000 in `get_next_work_required`), the condition `(prev_block_header.block_height + 1) % 1 != 0` is **always false**. Every single block validation falls through to the `get_header_by_height` call: [4](#0-3) 

With `blocks_to_go_back = 1` (for height > 0), `height_first = prev_block_header.block_height - 1`. For a fork diverging at height H:

| Fork block | `height_first` | `get_header_by_height` returns | Correct ancestor |
|---|---|---|---|
| F[H] | H-2 | M[H-2] | M[H-2] ✓ (same, pre-fork) |
| F[H+1] | H-1 | M[H-1] | M[H-1] ✓ (same, pre-fork) |
| **F[H+2]** | **H** | **M[H]** | **F[H] ✗ (diverges here)** |
| F[H+3] | H+1 | M[H+1] | F[H+1] ✗ |

Starting at F[H+2], the contract uses the mainchain block's timestamp at height H instead of the fork block F[H]'s timestamp. The attacker controls F[H]'s timestamp (within MTP and 2-hour future constraints).

**Digishield calculation using the wrong timestamp:** [5](#0-4) 

If the attacker sets F[H].time > M[H].time (e.g., by `delta` seconds), then:
- Incorrect `modulated_timespan` = F[H+1].time − M[H].time (larger by `delta`)
- Correct `modulated_timespan` = F[H+1].time − F[H].time (smaller)

A larger timespan → lower difficulty (easier target). The Digishield clamping bounds the per-block adjustment to at most `max_timespan / retarget_timespan = 90/60 = 1.5×` the previous target (50% easier per block). Since `difficulty_adjustment_interval = 1`, this applies to every subsequent fork block.

**`check_pow` enforces `expected_bits == block_header.bits`:** [6](#0-5) 

The attacker submits fork blocks with `bits` matching the incorrectly computed (easier) `expected_bits`. The check passes. The PoW hash check in `check_aux` then verifies the hash meets this easier target — which the attacker can satisfy with less work.

**Chain work accumulation and reorg:** [7](#0-6) 

`chain_work` is accumulated from `work_from_bits(block_header.bits)`. Easier `bits` = less work per block. The fork accumulates less work per block but the blocks are accepted. If the fork is long enough, it can still exceed mainchain `chain_work` and trigger `reorg_chain`, promoting the false fork to `mainchain_tip_blockhash`.

---

### Impact Explanation

The contract accepts Dogecoin fork blocks whose `bits` field encodes an easier difficulty than Dogecoin consensus rules require. These blocks pass `check_pow` and are stored in `headers_pool`. If the fork accumulates sufficient `chain_work`, `reorg_chain` promotes it to the canonical chain. Downstream callers of `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` then operate against a canonical chain that violates Dogecoin's difficulty rules — enabling false transaction inclusion proofs for transactions that exist only on the attacker's fork.

---

### Likelihood Explanation

- The entry point is `submit_blocks`, which is accessible to any trusted relayer. The trusted relayer system is staking-based (economic, not permissioned), so an attacker can self-onboard.
- The bug is triggered on every fork block starting at the third block after the divergence point, for all mainnet Dogecoin heights ≥ 145,000 (the entire current chain).
- The attacker controls fork block timestamps within the MTP/future-time window, giving direct control over the magnitude of the difficulty reduction (up to 50% per block).
- The TODO comment at line 291 confirms the developers are aware this is incorrect behavior.
- No privileged role, leaked key, or social engineering is required.

---

### Recommendation

Replace the `get_header_by_height` call with an ancestor walk that follows `prev_block_hash` links through `headers_pool` from `prev_block_header` back `blocks_to_go_back` steps. This correctly resolves the fork ancestor regardless of whether it is on the mainchain index. The `get_prev_header` method already performs this correctly via `headers_pool`: [8](#0-7) 

The fix is to walk `blocks_to_go_back` steps backward from `prev_block_header` using `get_prev_header`, rather than calling `get_header_by_height`. This is O(N) in `blocks_to_go_back` (= 1 for Dogecoin mainnet post-145k), so the gas cost is negligible.

---

### Proof of Concept

**State machine test outline (Dogecoin mainnet, height ≥ 145,001, `skip_pow_verification = false`):**

1. Initialize contract with mainchain blocks M[H-2], M[H-1], M[H] where M[H].time = T.
2. Submit fork block F[H] with `prev_block_hash = M[H-1].hash`, `time = T + 300` (5 minutes later than M[H], within 2-hour future limit). F[H] passes difficulty check (its `height_first = H-2`, which is pre-fork, so no discrepancy yet).
3. Submit fork block F[H+1] with `prev_block_hash = F[H].hash`. F[H+1] passes (its `height_first = H-1`, still pre-fork).
4. For F[H+2] with `prev_block_hash = F[H+1].hash`:
   - Contract computes `height_first = H`, calls `get_header_by_height(H)` → returns M[H] with `time = T`.
   - `modulated_timespan = F[H+1].time − T` (uses mainchain timestamp).
   - Correct computation would use `F[H].time = T + 300`: `modulated_timespan = F[H+1].time − (T+300)` (300 seconds smaller).
   - With `retarget_timespan = 60`, the dampened difference = `(300)/8 = 37.5s` → incorrect timespan is clamped to `max_timespan = 90` vs correct `min_timespan = 45`.
   - Incorrect `expected_bits` encodes a target 2× easier than correct.
5. Submit F[H+2] with `bits` matching the easier incorrect `expected_bits` and a valid PoW hash meeting that easier target. The contract accepts it; the correct Dogecoin node would reject it.
6. Repeat for F[H+3], F[H+4], … until fork `chain_work` > mainchain `chain_work`.
7. `reorg_chain` promotes the fork; `mainchain_tip_blockhash` now points to the attacker's invalid fork.
