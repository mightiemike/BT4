Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the actual user when swaps route through MetricOmmSimpleRouter, enabling full allowlist bypass — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is always `msg.sender` of the pool's `swap` call. When a user swaps through `MetricOmmSimpleRouter`, `sender` equals the router's address. A pool admin who allowlists the router to enable router-mediated swaps for their allowlisted users — the only available mechanism to do so — inadvertently grants swap access to every unprivileged user who routes through the router, completely bypassing the allowlist.

## Finding Description

**Step 1:** `MetricOmmPool.swap` passes `msg.sender` (the direct caller of `swap`) as `sender` to `_beforeSwap`: [1](#0-0) 

**Step 2:** `ExtensionCalling._beforeSwap` forwards `sender` verbatim to every configured extension: [2](#0-1) 

**Step 3:** `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is whoever called `pool.swap`. When the user goes through the router, `sender` = router address: [3](#0-2) 

**Step 4:** `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making itself `msg.sender` in the pool context: [4](#0-3) 

The effective allowlist check becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. There is no mechanism in the router to pass the original `msg.sender` through to the pool's `swap` call — the router only stores the payer in transient storage for the payment callback, not for extension identity checks.

**Contrast with `DepositAllowlistExtension`:** The deposit extension correctly gates `owner` (the second parameter, representing the actual beneficiary) rather than `sender` (the direct caller of `addLiquidity`): [5](#0-4) 

For swaps, no equivalent "actual user" parameter is threaded through the call chain, so the extension has no way to distinguish the real user from the router.

## Impact Explanation

This is a direct admin-boundary break. The swap allowlist is the pool admin's mechanism to restrict which addresses may trade against the pool (e.g., institutional-only liquidity, KYC-gated pools, adversarial-bot exclusion). Once the router is allowlisted, any unprivileged user can execute swaps the admin intended to block by routing through `MetricOmmSimpleRouter`. Unauthorized swaps can extract value from LPs or disrupt the pool's intended operation. The corrupted value is the `allowedSwapper[pool][sender]` extension decision, which evaluates `true` for the router address regardless of who the actual trading user is.

## Likelihood Explanation

The bypass requires the pool admin to have allowlisted the router address. This is the natural and expected action for any admin who wants their allowlisted users to be able to use the router — there is no other mechanism to enable router-mediated swaps for allowlisted users while keeping the allowlist active. Any pool that uses `SwapAllowlistExtension` and also intends to support router-based trading is vulnerable the moment the admin takes the obvious configuration step of allowlisting the router. [6](#0-5) 

## Recommendation

1. **Short term:** Document explicitly that `setAllowedToSwap(pool, router, true)` grants swap access to *all* users routing through that router, not just those individually allowlisted. Admins must allowlist individual user addresses only.
2. **Long term:** Redesign the extension to accept the actual user's address via `extensionData` (encoded by the router at call time and verified against the allowlist), falling back to `sender` for direct pool calls. This mirrors how `DepositAllowlistExtension` correctly gates `owner` rather than `msg.sender` of the adder. [3](#0-2) 

## Proof of Concept

```
1. Pool admin deploys pool with SwapAllowlistExtension configured as beforeSwap extension.
2. Admin allowlists alice:
       swapExtension.setAllowedToSwap(pool, alice, true)
3. Admin allowlists router so alice can use it:
       swapExtension.setAllowedToSwap(pool, router, true)
4. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, recipient: bob, ...})
5. Router calls pool.swap(bob, ...) — msg.sender in pool = router.
6. Pool calls extension.beforeSwap(router, ...) — sender = router.
7. Extension evaluates: allowedSwapper[pool][router] == true → passes.
8. Bob's swap executes despite not being on the allowlist.

Foundry test: deploy pool with SwapAllowlistExtension, allowlist alice and router,
assert that an address other than alice can successfully call router.exactInputSingle
on the pool without reverting with NotAllowedToSwap.
``` [7](#0-6) [8](#0-7)
