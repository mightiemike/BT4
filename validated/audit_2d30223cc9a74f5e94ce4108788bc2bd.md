### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc()` Enables Free Drain of usdcE from DirectDepositV1 Addresses - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` using the raw ERC-20 `transferFrom` without checking its return value. If the call returns `false` instead of reverting, execution continues: the usdcE is withdrawn from the `DirectDepositV1` and sent to `msg.sender` — with no USDC actually transferred in. This is a direct analog to the reported bug class: a token-transfer integration error where the wrong transfer primitive is used, allowing a caller to receive tokens without the corresponding debit.

---

### Finding Description

`replaceUsdcEWithUsdc` is a permissionless migration helper (restricted only by `block.chainid == 57073`) that is supposed to atomically swap usdcE held in a user's `DirectDepositV1` for native USDC:

```solidity
// ContractOwner.sol L608-L620
function replaceUsdcEWithUsdc(bytes32 subaccount) external {
    require(block.chainid == 57073, ERR_UNAUTHORIZED);
    address payable directDepositV1 = directDepositV1Address[subaccount];
    require(directDepositV1 != address(0), "no dda");
    address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
    address usdc  = 0x2D270e6886d130D724215A266106e6832161EAEd;
    uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
    if (balance > 0) {
        IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // ← raw, unchecked
        DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
        IERC20Base(usdcE).safeTransfer(msg.sender, balance);
    }
}
```

The rest of the codebase consistently uses `ERC20Helper.safeTransferFrom` (which checks the return value) for all inbound token pulls:

```solidity
// EndpointStorage.sol L95-L101
function safeTransferFrom(IERC20Base token, address from, uint256 amount) internal virtual {
    token.safeTransferFrom(from, address(this), amount);
}
```

```solidity
// BaseWithdrawPool.sol L192-L198
function safeTransferFrom(IERC20Base token, address from, uint256 amount) internal virtual {
    token.safeTransferFrom(from, address(this), amount);
}
```

`replaceUsdcEWithUsdc` is the only site in the production contracts that calls `transferFrom` directly on an `IERC20Base` without the safe wrapper. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

If `IERC20Base(usdc).transferFrom(...)` returns `false` (silent failure) rather than reverting, the three-step body still executes in full:

1. `DirectDepositV1(directDepositV1).withdraw(usdcE)` — transfers the entire usdcE balance from the victim's deposit address to `ContractOwner`. [4](#0-3) 
2. `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` — sends that usdcE to the attacker. [5](#0-4) 

The attacker receives the victim's pending usdcE deposit for free. The `DirectDepositV1` is now empty, so the victim's subsequent `creditDeposit()` call deposits nothing into their subaccount. The victim's collateral is permanently lost. [6](#0-5) 

---

### Likelihood Explanation

The hardcoded USDC address (`0x2D270e6886d130D724215A266106e6832161EAEd`) on Ink chain (chainid 57073) is expected to revert on failure, which prevents exploitation under normal conditions. However:

- The function carries no access control beyond the chain ID check — any unprivileged caller can invoke it for any subaccount that has a live `DirectDepositV1` with usdcE. [7](#0-6) 
- If the USDC contract on Ink is upgraded, replaced, or behaves non-standardly (e.g., returns `false` on insufficient allowance instead of reverting), the full drain path opens immediately.
- The pattern is a latent integration bug that violates the invariant upheld everywhere else in the codebase.

---

### Recommendation

Replace the raw `transferFrom` call with the project's own `safeTransferFrom` wrapper (or OpenZeppelin `SafeERC20.safeTransferFrom`) to enforce return-value checking, consistent with every other inbound transfer in the codebase:

```diff
- IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
+ IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [8](#0-7) 

---

### Proof of Concept

1. A user's `DirectDepositV1` at address `DDA` holds `N` usdcE tokens, waiting to be credited via `creditDeposit()`.
2. Attacker calls `replaceUsdcEWithUsdc(subaccount)` on Ink chain with **zero USDC allowance** granted to `ContractOwner`.
3. `IERC20Base(usdc).transferFrom(attacker, DDA, N)` returns `false` (no revert, e.g. due to non-standard token behavior or future upgrade).
4. Execution continues: `DirectDepositV1(DDA).withdraw(usdcE)` moves `N` usdcE to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, N)` sends `N` usdcE to the attacker.
6. Attacker holds `N` usdcE; victim's `DDA` is empty; victim's subaccount receives no deposit. [9](#0-8)

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
