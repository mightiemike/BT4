Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any unauthorized actor to bypass the deposit allowlist gate - (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` ignores the `sender` argument (the actual caller of `addLiquidity`) and instead checks the `owner` argument (the beneficiary of the position) against the allowlist. Because `owner` is a caller-controlled parameter, any unauthorized actor can bypass the deposit gate by specifying any allowlisted address as `owner`, defeating the pool admin's access-control configuration entirely.

## Finding Description

**Call path into the extension:**

`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)`, which encodes and dispatches:

```solidity
// ExtensionCalling.sol L97
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

Here `sender` = `msg.sender` of `addLiquidity` (the actual depositor/caller) and `owner` = the caller-supplied beneficiary address.

**The broken guard:**

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The first parameter (`sender`) is unnamed and silently discarded. The allowlist lookup uses `owner` — a value the attacker freely controls — instead of `sender`.

**Contrast with the correct pattern in `SwapAllowlistExtension`:**

```solidity
// SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```

`SwapAllowlistExtension` correctly checks `sender`; `DepositAllowlistExtension` does not.

**Exploit flow:**

1. Pool is deployed with `DepositAllowlistExtension`; pool admin calls `setAllowedToDeposit(pool, alice, true)` and does **not** allowlist the attacker.
2. Attacker calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)` — setting `owner = alice`.
3. Pool calls `DepositAllowlistExtension.beforeAddLiquidity(attacker, alice, ...)`.
4. Guard evaluates `allowedDepositor[pool][alice]` → `true` → passes without checking `attacker`.
5. Attacker's deposit executes; the position is recorded under `alice`. Attacker paid the tokens via the swap callback; `alice` owns the resulting position.
6. The allowlist gate is fully bypassed for any unauthorized actor who knows any allowlisted address.

## Impact Explanation
The deposit allowlist is a core pool access-control feature. Its complete bypass means any unprivileged actor can add liquidity to a pool the admin intended to restrict. This constitutes broken core pool functionality (allowlist-gated liquidity provision is rendered inoperative). Additionally, the attacker can force liquidity positions onto allowlisted addresses without their consent, and the allowlisted address can subsequently withdraw those tokens — enabling a token-extraction vector if the attacker and the allowlisted address collude, or a griefing vector otherwise. Severity: **Medium** (core access-control invariant broken; direct fund loss requires a cooperative allowlisted address or a second-step withdrawal).

## Likelihood Explanation
Exploitable by any unprivileged caller with knowledge of at least one allowlisted address (publicly readable via `allowedDepositor`). No special permissions, flash loans, or oracle manipulation required. Repeatable on every block. Any pool that deploys `DepositAllowlistExtension` with a non-trivial allowlist is affected.

## Recommendation
Replace the `owner` lookup with the `sender` argument, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate on the position owner rather than the caller, the function name, NatSpec, and `setAllowedToDeposit` documentation must be updated to reflect that, and the security implications of the owner-vs-sender distinction must be explicitly acknowledged.

## Proof of Concept

```solidity
// Foundry unit test sketch
function test_depositAllowlistBypass() public {
    // Setup: pool with DepositAllowlistExtension; only alice is allowlisted
    address alice = makeAddr("alice");
    address attacker = makeAddr("attacker");
    extension.setAllowedToDeposit(address(pool), alice, true);
    // alice is allowlisted; attacker is not

    // Attacker funds themselves with token0/token1
    deal(token0, attacker, 1e18);
    deal(token1, attacker, 1e18);

    // Attacker calls addLiquidity with owner = alice
    vm.startPrank(attacker);
    IERC20(token0).approve(address(router), type(uint256).max);
    IERC20(token1).approve(address(router), type(uint256).max);
    // Should revert with NotAllowedToDeposit — but does NOT
    pool.addLiquidity(alice, 0, deltas, callbackData, extensionData);
    vm.stopPrank();

    // Verify: position exists under alice, funded by attacker
    // allowlist gate was bypassed
}
```
