### Title
Missing `maxCash` Slippage Guard in `LiquidateSubaccount` Exposes Liquidator to Unbounded Cash Spend - (File: `core/contracts/ClearinghouseLiq.sol`)

---

### Summary

The `LiquidateSubaccount` signed transaction struct contains no `maxCash` (slippage) parameter. The cash the liquidator actually spends is computed entirely at execution time from the live oracle price. Between when the liquidator signs the transaction and when the sequencer includes it in a batch, the oracle price can change — causing the liquidator to spend materially more quote than they anticipated, with no on-chain mechanism to prevent it.

---

### Finding Description

The `LiquidateSubaccount` struct is defined as: [1](#0-0) 

It contains `sender`, `liquidatee`, `productId`, `isEncodedSpread`, `amount`, and `nonce` — but **no `maxCash` field**. The liquidator signs over a fixed `amount` of units to liquidate, but the quote they will spend is not committed at signing time.

The actual cash deducted from the liquidator is computed inside `_handleLiquidationPayment()` at execution time: [2](#0-1) 

The liquidation price is derived from `getLiqPriceX18()`, which reads the live oracle price `risk.priceX18` at the moment of execution: [3](#0-2) 

The sequencer submits `UpdatePrice` transactions in the same batch pipeline as `LiquidateSubaccount` transactions. The ordering within a batch is sequencer-controlled. If an `UpdatePrice` is included before the liquidation in the same batch, the oracle price seen by `getLiqPriceX18()` will be the updated (potentially higher) price, not the price the liquidator observed when signing.

Additionally, for spot and spread liquidations, `_settlePositivePerpPnl()` is called before `_handleLiquidationPayment()`: [4](#0-3) 

This settles the liquidatee's positive perp PnL against the liquidator's quote balance — an additional cash outflow whose magnitude also depends on live oracle prices at execution time and is not bounded by any parameter in the signed transaction.

There is no `maxCash` check anywhere in `liquidateSubaccountImpl`, `_handleLiquidationPayment`, or the signed struct itself. [5](#0-4) 

---

### Impact Explanation

The liquidator's quote balance is debited by `liquidationPayment + liquidationFees` where:

```
liquidationPayment = getLiqPriceX18(productId, amount) * amount
                   = risk.priceX18 * (1 + penaltyX18) * amount
```

If `risk.priceX18` increases by even 5–10% between signing and execution (common in volatile markets), the liquidator pays 5–10% more quote than they budgeted. For large liquidations this is a meaningful, unrecoverable asset loss from the liquidator's subaccount. The liquidator cannot cancel or revoke the signed transaction once submitted to the sequencer.

---

### Likelihood Explanation

Medium. Oracle prices are updated frequently via `UpdatePrice` transactions submitted by the sequencer in the same batch pipeline. In volatile market conditions — precisely when liquidations are most active — price moves between signing and execution are routine. The liquidator has no on-chain recourse once the transaction is signed and submitted.

---

### Recommendation

Add a `maxCash` field to `LiquidateSubaccount`:

```solidity
struct LiquidateSubaccount {
    bytes32 sender;
    bytes32 liquidatee;
    uint32 productId;
    bool isEncodedSpread;
    int128 amount;
    uint64 nonce;
    int128 maxCash; // maximum quote the liquidator is willing to spend
}
```

In `_handleLiquidationPayment()`, after computing `liquidationPayment + liquidationFees`, add:

```solidity
require(
    liquidationPayment + liquidationFees <= txn.maxCash,
    ERR_EXCEEDS_MAX_CASH
);
```

This mirrors the fix applied in the referenced PR #274 for the Dutch auction analog.

---

### Proof of Concept

1. Oracle price for asset A is `P = 1000e18`. Liquidator observes liquidatee is undercollateralized.
2. Liquidator signs `LiquidateSubaccount { amount = 100, ... }`, expecting to pay approximately `100 * 1000 * (1 + penalty) ≈ 105,000 USDC`.
3. Sequencer includes an `UpdatePrice` transaction setting `risk.priceX18 = 1100e18` in the same batch, ordered before the liquidation.
4. `getLiqPriceX18()` returns `1100 * (1 + penalty)` at execution time.
5. `liquidationPayment = 1100 * 100 * (1 + penalty) ≈ 115,500 USDC` is debited from the liquidator.
6. The liquidator spent ~10,500 USDC more than anticipated with no on-chain protection. The signed transaction contained no `maxCash` bound, so the debit proceeds unconditionally. [6](#0-5)

    function liquidateSubaccountImpl(IEndpoint.LiquidateSubaccount calldata txn)
        external
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.sender != txn.liquidatee, ERR_UNAUTHORIZED);
        require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
        require(
            txn.liquidatee != X_ACCOUNT && txn.liquidatee != N_ACCOUNT,
            ERR_NOT_LIQUIDATABLE
        );
        require(
            txn.productId != QUOTE_PRODUCT_ID,
            ERR_INVALID_LIQUIDATION_PARAMS
        );

        ISpotEngine spotEngine = ISpotEngine(
            address(engineByType[IProductEngine.EngineType.SPOT])
        );
        IPerpEngine perpEngine = IPerpEngine(
            address(engineByType[IProductEngine.EngineType.PERP])
        );

        if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
            if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.liquidatee);
            }
            return;
        }

        if (
            (txn.amount < 0) &&
            (txn.isEncodedSpread ||
                address(productToEngine[txn.productId]) == address(spotEngine))
        ) {
            // when it's spread or spot liquidation, we need to make sure the liquidatee has
            // enough quote to buyback the liquidated amount.
            _assertCanLiquidateLiability(txn, spotEngine, perpEngine);
            _settlePositivePerpPnl(txn, spotEngine, perpEngine);
        }

        _assertLiquidationAmount(txn, spotEngine, perpEngine);

        // beyond this point, we can be sure that we can liquidate the entire
        // liquidation amount knowing that the insurance fund will remain solvent
        // subsequently we can just blast the remainder of the liquidation and
        // cover the quote balance from the insurance fund at the end
        _handleLiquidationPayment(txn, spotEngine, perpEngine);
    }

    function getLiqPriceX18(uint32 productId, int128 amount)
        internal
        returns (int128, int128)
    {
        RiskHelper.Risk memory risk = IProductEngine(productToEngine[productId])
            .getRisk(productId);
        int128 penaltyX18 = (RiskHelper._getWeightX18(
            risk,
            amount,
            IProductEngine.HealthType.MAINTENANCE
        ) - ONE) / 5;
        if (penaltyX18.abs() < MIN_NON_SPREAD_LIQ_PENALTY_X18) {
            if (penaltyX18 < 0) {
                penaltyX18 = -MIN_NON_SPREAD_LIQ_PENALTY_X18;
            } else {
                penaltyX18 = MIN_NON_SPREAD_LIQ_PENALTY_X18;
            }
        }
        return (risk.priceX18.mul(ONE + penaltyX18), risk.priceX18);
    }
