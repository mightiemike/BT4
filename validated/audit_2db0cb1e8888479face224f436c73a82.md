Audit Report

## Title
Swap Allowlist Bypass via Router — `sender` Identity Mismatch in `SwapAllowlistExtension.beforeSwap` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the original user. This means the allowlist checks the router's address rather than the actual swapper's identity, creating an irresolvable dilemma: either allowlisted users cannot use the router at all, or the pool admin must allowlist the router itself — which then allows any unprivileged user to bypass the allowlist entirely.

## Finding Description

**Root cause — wrong identity in `sender`:**

In `MetricOmmPool.swap()`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-240
_beforeSwap(
    msg.sender,   // <-- immediate caller of pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this verbatim to the extension:

```solidity
// ExtensionCalling.sol:160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` = pool, `sender` = whoever called `pool.swap()`.

**Router path — sender becomes the router:**

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly:

```solidity
// MetricOmmSimpleRouter.sol:72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The router is `msg.sender` inside `pool.swap()`, so `sender` forwarded to the extension is the **router address**, not the original EOA.

**The dilemma:**

- If the pool admin does NOT allowlist the router: allowlisted users cannot use the router (their swaps revert because the router's address is not in the allowlist).
- If the pool admin DOES allowlist the router: any unprivileged user can call `router.exactInputSingle()` and bypass the allowlist entirely, because the router's address passes the check regardless of who the actual user is.

The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput` — all router entry points call `pool.swap()` with the router as `msg.sender`.

## Impact Explanation

This is an admin-boundary break: the pool admin's swap allowlist — intended to restrict swaps to specific addresses (e.g., KYC'd counterparties, whitelisted market makers) — is rendered ineffective for any user who routes through the public `MetricOmmSimpleRouter`. An unprivileged, non-allowlisted user can execute swaps on a restricted pool by simply calling the router, draining pool liquidity or trading against the oracle price in ways the pool admin explicitly intended to prevent. This constitutes broken core pool functionality and a direct bypass of an access-control cap.

## Likelihood Explanation

The `MetricOmmSimpleRouter` is a public, permissionless contract. Any user can call it. The bypass requires only that the pool admin has allowlisted the router (a necessary step for any allowlisted user to use the router). Once the router is allowlisted, the bypass is trivially repeatable by any address with no special privileges or capital requirements beyond the swap amount.

## Recommendation

The `sender` forwarded to extensions must represent the original user, not the intermediate router. Two viable fixes:

1. **Pass original user through extensionData**: The router encodes `msg.sender` (the original EOA) into `extensionData` and the extension decodes it. This requires a protocol-level convention.
2. **Dedicated router field in the hook signature**: Add an `originator` parameter to `beforeSwap` that the pool populates from a trusted transient-storage slot set by the router before calling `pool.swap()`, analogous to how the router already stores payer context in `TransientCallbackPool`.

The cleanest fix is option 2: the router writes the original `msg.sender` into a transient slot before calling `pool.swap()`, the pool reads it and passes it as `originator` to `_beforeSwap`, and the extension checks `originator` instead of `sender`.

## Proof of Concept

```solidity
// Foundry test sketch
function test_swapAllowlistBypassViaRouter() public {
    // Setup: pool with SwapAllowlistExtension, router allowlisted, attacker NOT allowlisted
    swapExtension.setAllowedToSwap(address(pool), address(router), true);
    // attacker is NOT in allowedSwapper

    // Direct swap by attacker reverts
    vm.prank(attacker);
    vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
    pool.swap(attacker, true, 1000, type(uint128).max, "", "");

    // Router-mediated swap by attacker succeeds (bypass)
    token0.approve(address(router), type(uint256).max);
    vm.prank(attacker);
    router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token0),
        recipient: attacker,
        deadline: block.timestamp + 1,
        zeroForOne: true,
        amountIn: 1000,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        extensionData: ""
    }));
    // Swap succeeds — allowlist bypassed
}
```
