Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `pool.addLiquidity`) and instead validates `owner` (the LP-position recipient). Because `addLiquidity` imposes no `msg.sender == owner` constraint, any unprivileged address can pass an allowlisted address as `owner` and deposit into a restricted pool, completely bypassing the intended access control.

## Finding Description
`MetricOmmPool.addLiquidity` passes two distinct addresses to the extension hook:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
//                  ^^^^^^^^^^  ^^^^^
//                  sender      owner (LP position recipient, may differ)
``` [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `(sender, owner, …)` but drops `sender` (unnamed first parameter) and gates on `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
``` [2](#0-1) 

`addLiquidity` has no `msg.sender == owner` guard (unlike `removeLiquidity`, which explicitly enforces `if (msg.sender != owner) revert NotPositionOwner()` at L206): [3](#0-2) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` makes `owner` a free caller-supplied parameter while `msg.sender` is the payer, providing a ready-made router path for the attack without requiring the attacker to implement any callback: [4](#0-3) 

`_validateOwner` only checks `owner != address(0)`, imposing no allowlist constraint: [5](#0-4) 

Compare with `SwapAllowlistExtension.beforeSwap`, which correctly names and checks `sender`: [6](#0-5) 

The wrong value is the extension decision: `allowedDepositor[pool][owner]` is evaluated when `allowedDepositor[pool][sender]` is the intended guard, causing the hook to return `beforeAddLiquidity.selector` (allow) for any call where `owner` is allowlisted, regardless of who `sender` is.

## Impact Explanation
A pool configured with `DepositAllowlistExtension` to restrict deposits to a KYC/allowlisted set has no effective access control on who actually calls `addLiquidity`. Any unauthorized address can deposit into a restricted pool by specifying an allowlisted address as `owner`. The deposit allowlist — the sole access-control mechanism for `addLiquidity` — is fully bypassed. Pools intended to be restricted (regulated, permissioned, or institutional) accept deposits from any address. This is a direct break of core pool access-control functionality and constitutes a High severity finding under the Broken core pool functionality / Admin-boundary break impact gate.

## Likelihood Explanation
- The `owner` parameter is freely settable by any caller of `pool.addLiquidity` or `MetricOmmPoolLiquidityAdder.addLiquidityExactShares`.
- No other guard in the pool enforces `msg.sender == owner` for `addLiquidity`.
- `MetricOmmPoolLiquidityAdder` makes the `sender ≠ owner` pattern a first-class, documented use case, providing a callback-handling router so the attacker does not need to implement any interface.
- A pool admin who deploys `DepositAllowlistExtension` believing it gates depositors will be unaware the guard is misbound.
- The attack is repeatable with zero special privileges.

## Recommendation
Name and check `sender` (the first parameter), matching the pattern used by `SwapAllowlistExtension`:

```solidity
// Before (wrong):
function beforeAddLiquidity(address, address owner, …) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {

// After (correct):
function beforeAddLiquidity(address sender, address, …) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
```

## Proof of Concept
```
Setup:
  pool configured with DepositAllowlistExtension
  alice  → allowedDepositor[pool][alice] = true
  bob    → not allowlisted

Attack via MetricOmmPoolLiquidityAdder (no callback implementation needed):
  bob calls liquidityAdder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, "")
    → liquidityAdder calls pool.addLiquidity(alice, salt, deltas, abi.encode(KIND_PAY), "")
    → pool calls _beforeAddLiquidity(liquidityAdder /*sender*/, alice /*owner*/, …)
    → hook checks allowedDepositor[pool][alice] == true  ✓  (passes)
    → pool calls liquidityAdder.metricOmmModifyLiquidityCallback(…)
    → liquidityAdder pulls tokens from bob (payer = msg.sender of original call)
    → LP shares credited to alice

  alice calls pool.removeLiquidity(alice, salt, deltas, "")
    → tokens returned to alice

Result: bob deposited into a restricted pool; allowlist completely bypassed.
```
