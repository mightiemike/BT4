### Title
Deposited Collateral Permanently Locked in Clearinghouse When Slow Mode `DepositCollateral` Transaction Fails Silently — (`core/contracts/Endpoint.sol`)

---

### Summary

When a user calls `depositCollateral`, their tokens are immediately transferred to the `Clearinghouse` (escrow) and a `DepositCollateral` slow mode transaction is queued. If that slow mode transaction later reverts during execution, the error is silently swallowed, the queue entry is consumed, and the user's tokens remain permanently locked in the `Clearinghouse` with no subaccount credit and no recovery path. The codebase itself contains a comment confirming the removal of the fund-return mechanism: `// try return funds now removed`.

---

### Finding Description

The deposit flow in `Endpoint.sol` is a two-step process:

**Step 1 — Escrow:** `depositCollateral` transfers tokens from the user to the `Clearinghouse` and enqueues a `DepositCollateral` slow mode transaction. [1](#0-0) 

`handleDepositTransfer` moves tokens from `msg.sender` directly to `address(clearinghouse)`: [2](#0-1) 

**Step 2 — Credit:** The slow mode transaction must be executed (by the sequencer or by anyone after the 3-day delay) to call `clearinghouse.depositCollateral(txn)` and credit the user's subaccount: [3](#0-2) 

**The broken invariant** is in `_executeSlowModeTransaction`. The slow mode entry is deleted from the queue unconditionally before execution, and if `processSlowModeTransaction` reverts, the catch block silently discards the failure with no fund return: [4](#0-3) 

The comment on line 226 — `// try return funds now removed` — explicitly confirms that a fund-return mechanism existed and was deliberately removed, leaving no recovery path.

A concrete trigger: the sequencer can submit a `DelistProduct` slow mode transaction (via `processSlowModeTransactionImpl` line 242-243 of `EndpointTx.sol`) that removes a product from the `SpotEngine`. If a user's `DepositCollateral` slow mode tx for that product is queued behind the `DelistProduct` tx, when the deposit tx is eventually executed, `clearinghouse.depositCollateral` will revert (product no longer valid), the tx is consumed from the queue, and the user's tokens are permanently locked. [5](#0-4) 

---

### Impact Explanation

User funds (ERC-20 collateral tokens) are permanently locked in the `Clearinghouse` contract. The tokens are held at `address(clearinghouse)` with no corresponding subaccount balance and no mechanism to retrieve them. The slow mode queue entry is consumed, so the deposit cannot be retried. There is no admin rescue function, no escrow accounting, and no refund path.

---

### Likelihood Explanation

The window requires a product to be delisted (or any other condition causing `clearinghouse.depositCollateral` to revert) after a user's deposit transfer but before their slow mode tx is processed. Product delisting is an admin action, but the 3-day slow mode delay creates a meaningful window. Additionally, any future revert condition in `clearinghouse.depositCollateral` (e.g., overflow, invariant violation) would trigger the same permanent lock. The silent catch pattern means this can happen without any on-chain signal.

---

### Recommendation

Restore the fund-return mechanism in the catch block of `_executeSlowModeTransaction`. When a `DepositCollateral` slow mode transaction fails, the `Clearinghouse` should transfer the deposited amount back to the original sender. Alternatively, track pending deposits in a mapping keyed by slow mode index so that a failed deposit can be claimed back by the user. At minimum, emit an event on failure so the state is observable.

---

### Proof of Concept

1. User calls `Endpoint.depositCollateral(subaccountName, productId, amount)`.
2. `handleDepositTransfer` moves `amount` tokens from user → `Clearinghouse`. Slow mode tx at index `N` is queued with `executableAt = now + 3 days`.
3. Admin submits a `DelistProduct` slow mode tx at index `N-1` (or sequencer processes it immediately via `ExecuteSlowMode`). The product is removed from `SpotEngine`.
4. After 3 days, anyone calls `executeSlowModeTransaction()`. The queue entry at index `N` is deleted (`delete slowModeTxs[txUpTo++]`).
5. `processSlowModeTransaction` → `clearinghouse.depositCollateral(txn)` reverts because the product no longer exists.
6. The catch block at `Endpoint.sol:207` fires. `gasleft()` checks pass (normal execution). Execution continues past `// try return funds now removed`.
7. User's `amount` tokens remain in `Clearinghouse`. Subaccount balance is zero. No retry is possible. No refund exists. [6](#0-5) [7](#0-6) [3](#0-2)
