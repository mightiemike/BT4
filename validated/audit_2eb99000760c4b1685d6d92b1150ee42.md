Audit Report

## Title
SwapAllowlistExtension checks router address instead of original trader, allowing allowlist bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, not the original trader. If the router is allowlisted on a curated pool, every user — including non-allowlisted ones — can bypass the allowlist by calling any router entry point.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that `sender` directly to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` = pool and `sender` = router: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` stores the original `msg.sender` only in transient storage for the payment callback — it is never forwarded to the pool as the swapper identity: [4](#0-3) 

The check resolves to `allowedSwapper[pool][router]`. A pool admin who deploys `SwapAllowlistExtension` and wants to support the official router will naturally allowlist the router address. Once the router is allowlisted, the check passes for **any** caller of the router, regardless of whether that caller is individually allowlisted.

## Impact Explanation
The `SwapAllowlistExtension` guard is completely defeated for any pool that allowlists the router. Any non-allowlisted trader can execute swaps on a curated pool by routing through `MetricOmmSimpleRouter`. This breaks the core curation policy of the pool and allows unauthorized trading, which can cause direct loss to LPs if the allowlist was intended to restrict trading to trusted counterparties or enforce risk controls. This meets the "broken core pool functionality causing loss of funds" impact gate.

## Likelihood Explanation
High. The bypass requires no special privileges — any EOA can call `exactInputSingle`. The precondition (router allowlisted on a curated pool) is the natural production configuration: a pool admin who deploys `SwapAllowlistExtension` and also wants to support the official router will allowlist the router. The scenario is fully reachable without any malicious setup.

## Recommendation
The router should forward the original trader's address to the pool. The simplest fix is to have the router encode the original `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check it when `sender` is a known router. Alternatively, the pool could expose a `swapOnBehalfOf(address trader, ...)` entry point, or the extension could accept a router-signed attestation of the original sender.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured in `beforeSwapOrder`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` — allowlisting the router so router-mediated swaps work.
3. Pool admin does **not** call `setAllowedToSwap(pool, attacker, true)`.
4. Attacker calls `router.exactInputSingle(...)` with `pool` as target.
5. Router calls `pool.swap(recipient, ...)` — pool sees `msg.sender = router`.
6. Pool calls `_beforeSwap(sender=router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router]` = `true` → passes.
8. Swap executes successfully for the non-allowlisted attacker.

Assert: swap succeeds and attacker receives output tokens despite not being individually allowlisted.
