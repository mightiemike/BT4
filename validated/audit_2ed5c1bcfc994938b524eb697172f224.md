### Title
Stale `digestToSubaccount` Mapping After Isolated Subaccount Closure Enables Zero-Margin Perp Positions — (`core/contracts/OffchainExchange.sol`)

### Summary

`_tryCloseIsolatedSubaccount` clears `parentSubaccounts`, `isolatedSubaccounts`, and the mask, but never clears `digestToSubaccount[digest]`. `matchOrders` unconditionally overrides `order.sender` with `digestToSubaccount[digest]` when non-zero, and the base `isHealthy` always returns `true`. Together these allow a partially-filled isolated order to be re-matched against a closed, zero-margin isolated subaccount, creating undercollateralized perp positions and bad debt.

---

### Finding Description

**Root cause 1 — `_tryCloseIsolatedSubaccount` does not clear the digest mapping** [1](#0-0) 

Lines 202–204 clear `isolatedSubaccountsMask`, `isolatedSubaccounts[parent][id]`, and `parentSubaccounts[subaccount]`, but `digestToSubaccount[digest]` and `digestToMargin[digest]` are never touched. After closure the mapping still points to the now-dead isolated subaccount.

**Root cause 2 — `matchOrders` blindly overrides `order.sender` from the stale mapping** [2](#0-1) 

Every call to `matchOrders` checks `digestToSubaccount[ordersInfo.taker.digest]` and, if non-zero, replaces `taker.order.sender` with whatever is stored there — including a closed, zero-balance isolated subaccount.

**Root cause 3 — `isHealthy` is a no-op in the base contract** [3](#0-2) 

The post-trade health gate unconditionally returns `true`, so no margin check blocks the fill.

**Root cause 4 — `createIsolatedSubaccount` early-returns on stale digest without transferring margin** [4](#0-3) 

A second `CreateIsolatedSubaccount` with the same order hits the `!= bytes32(0)` guard and returns the stale subaccount immediately, skipping the `spotEngine.updateBalance` margin transfer at lines 1077–1086. [5](#0-4) 

---

### Impact Explanation

The closed isolated subaccount has zero quote balance (returned to parent at lines 190–200). When `matchOrders` re-routes a partially-filled order to it, the perp engine records a position with no backing margin. Because `isHealthy` returns `true`, the fill is accepted. The isolated subaccount is immediately under-margined; any adverse price move creates bad debt absorbed by the insurance fund. [6](#0-5) 

---

### Likelihood Explanation

The precondition is a partially-filled isolated order whose position is subsequently closed by a separate reduce-only order (a normal trading pattern). No admin access, key compromise, or governance action is required. The attacker controls the order parameters and can deliberately leave residual fill capacity before closing the position.

---

### Recommendation

In `_tryCloseIsolatedSubaccount`, after clearing the parent/mask state, iterate over (or store a reverse mapping from subaccount to digest) and clear `digestToSubaccount[digest]` and `digestToMargin[digest]`. Alternatively, add a check in `createIsolatedSubaccount` and `matchOrders` that verifies `parentSubaccounts[digestToSubaccount[digest]] != bytes32(0)` before treating the cached mapping as valid.

        uint32 productId = RiskHelper.getIsolatedProductId(subaccount);

        }

            productId,

        if (balance.amount == 0) {

            address addr = address(uint160(bytes20(subaccount)));
            bytes32 parent = parentSubaccounts[subaccount];
