Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks caller-controlled `owner` instead of actual depositor `sender`, allowing full allowlist bypass — (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards its first argument (`sender`, the actual depositor) and gates on `owner`, which is a free parameter supplied by the caller to `MetricOmmPool.addLiquidity`. Any unprivileged address can bypass the deposit allowlist by naming any allowlisted address as `owner`, rendering the entire access-control mechanism ineffective.

## Finding Description
The contract's NatSpec and mapping name both declare the intent to gate by depositor address:

```solidity
/// @notice Gates `addLiquidity` by depositor address, per pool.
mapping(address pool => mapping(address depositor => bool)) public allowedDepositor;
``` [1](#0-0) 

However, the hook implementation discards `sender` (first parameter, left unnamed) and checks `owner` instead:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
``` [2](#0-1) 

In `MetricOmmPool.addLiquidity`, `owner` is a free parameter supplied by the caller, and `msg.sender` is forwarded as `sender`: [3](#0-2) 

`ExtensionCalling._beforeAddLiquidity` correctly encodes both arguments in order `(sender, owner, ...)` and passes them to the extension: [4](#0-3) 

The extension receives `sender` as its first argument but ignores it entirely. Because `owner` is attacker-controlled, the guard approves any caller who names an allowlisted address as `owner`. The attacker's tokens are pulled via the liquidity callback and the LP position is minted to the named `owner`. The actual depositor is never validated.

## Impact Explanation
The deposit allowlist is rendered completely ineffective. Any unprivileged address can add liquidity to a pool the admin intended to restrict. This constitutes an admin-boundary break: the pool admin's access-control configuration is circumvented without any privilege. Pools using the allowlist for KYC, counterparty-risk, or regulatory reasons are silently open to all depositors. Additionally, the attacker can inject liquidity into arbitrary bins, shifting the pool's price distribution and per-share metrics.

## Likelihood Explanation
Exploitation requires only one publicly observable allowlisted address (e.g., the pool admin's own address, readable from the factory) and a single `addLiquidity` call. No special privileges, flash loans, or multi-step setup are needed. The attack is repeatable by any EOA or contract at any time.

## Recommendation
Use `sender` (the actual depositor) instead of `owner` in the guard:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
1. Pool admin deploys a pool with `DepositAllowlistExtension`; only `allowedLP` is permitted to deposit (`allowedDepositor[pool][allowedLP] = true`).
2. Attacker (any EOA or contract) calls `pool.addLiquidity(allowedLP, salt, deltas, callbackData, "")`.
3. `DepositAllowlistExtension.beforeAddLiquidity` evaluates `allowedDepositor[pool][allowedLP]` → `true` → no revert.
4. `LiquidityLib.addLiquidity` pulls tokens from the attacker via the swap callback and mints the LP position to `allowedLP`.
5. The attacker has deposited into a restricted pool without appearing on the allowlist; the actual depositor (`msg.sender`) was never checked.
