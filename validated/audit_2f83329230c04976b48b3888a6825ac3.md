### Title
Unguarded u64 Underflow in `fetch_rate` Panics Sequencer or Silently Corrupts Exchange-Rate Used for Gas-Price Computation - (File: crates/apollo_l1_gas_price/src/exchange_rate_oracle.rs)

### Summary

`ExchangeRateOracleClient::fetch_rate` performs two bare u64 subtractions without overflow guards. When `timestamp < lag_interval_seconds` (line 223) or when `quantized_timestamp == 0` (line 242), the subtractions underflow. In a debug build this panics the sequencer; in a release build Rust wraps the value, causing the oracle to be queried with a wildly wrong timestamp and returning a corrupted ETH→STRK exchange rate that propagates directly into block gas-price computation.

### Finding Description

In `fetch_rate` at line 223:

```rust
let quantized_timestamp = (timestamp - self.config.lag_interval_seconds)
    .checked_div(self.config.lag_interval_seconds)
    .expect("lag_interval_seconds should be non-zero");
```

`timestamp` and `lag_interval_seconds` are both `u64`. The subtraction is a plain `-` with no `checked_sub`. If `timestamp < lag_interval_seconds`, the subtraction underflows.

At line 242, a second bare subtraction occurs:

```rust
if let Some(rate) = cache.get(&(quantized_timestamp - NUMBER_OF_TIMESTAMPS_BACK)) {
```

`NUMBER_OF_TIMESTAMPS_BACK` is the constant `1`. If `quantized_timestamp == 0` (which happens when `timestamp == lag_interval_seconds` exactly, making `(timestamp - lag) / lag == 0`), this second subtraction also underflows.

`lag_interval_seconds` is a plain `u64` field in `ExchangeRateOracleConfig` with no upper-bound validator:

```rust
pub struct ExchangeRateOracleConfig {
    pub lag_interval_seconds: u64,
    ...
}
```

The only runtime guard is an `assert!(self.config.lag_interval_seconds > 0, ...)` inside `spawn_query`, which fires only after the underflow has already occurred. No `#[validate(range(...))]` attribute is present on `lag_interval_seconds`, unlike `number_of_blocks_for_mean` which carries `#[validate(range(min = 1))]`.

The corrupted `quantized_timestamp` is then multiplied by `lag_interval_seconds` in `spawn_query` to form `adjusted_timestamp`, which is sent as the `?timestamp=` query parameter to the external oracle API. A wrapped value near `u64::MAX` produces a timestamp billions of years in the future; the oracle either returns an error (causing the sequencer to stall on gas-price fetching) or returns a stale/wrong rate that is accepted as authoritative.

### Impact Explanation

`fetch_rate` is called from `L1GasPriceProvider::eth_to_fri_rate`, which is called by the consensus orchestrator during `build_proposal` to obtain the ETH→STRK conversion rate used to set `strk_gas_prices` in the block header. A corrupted rate causes every gas-price field derived from it (`l1_gas_price`, `l1_data_gas_price`, `l2_gas_price` in STRK) to be wrong for the proposed block. Validators that independently compute the expected gas price will reject the proposal or accept a block with economically incorrect fees, satisfying the **"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact"** criterion.

In a debug build the panic crashes the sequencer process entirely, satisfying the **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing"** criterion because no further blocks can be proposed.

### Likelihood Explanation

Production Unix timestamps (~1.7 × 10⁹ s) are far larger than the default `lag_interval_seconds` of 900 s, so the underflow does not trigger under normal operation. However:

1. An operator who misconfigures `lag_interval_seconds` to a value larger than the current block timestamp (e.g., a very large number, or a value exceeding the chain's age in seconds on a fresh devnet) triggers the underflow on every call to `fetch_rate`.
2. The second underflow (`quantized_timestamp - 1`) triggers whenever `timestamp == lag_interval_seconds` exactly, which is a reachable edge case in integration tests and early-chain scenarios.
3. No config-load-time validation rejects an oversized `lag_interval_seconds`, so the misconfiguration silently passes startup checks.

### Recommendation

**Short term**: Replace both bare subtractions with `checked_sub` and return a descriptive error instead of panicking or wrapping:

```rust
// Line 223
let shifted = timestamp.checked_sub(self.config.lag_interval_seconds)
    .ok_or(ExchangeRateOracleClientError::InvalidTimestamp(timestamp))?;
let quantized_timestamp = shifted
    .checked_div(self.config.lag_interval_seconds)
    .expect("lag_interval_seconds should be non-zero");

// Line 242
if let Some(prev_qt) = quantized_timestamp.checked_sub(NUMBER_OF_TIMESTAMPS_BACK) {
    if let Some(rate) = cache.get(&prev_qt) { ... }
}
```

**Long term**: Add a `#[validate(range(min = 1, max = <reasonable_upper_bound>))]` attribute to `lag_interval_seconds` in `ExchangeRateOracleConfig` so that an oversized value is rejected at node startup, mirroring the existing guard on `number_of_blocks_for_mean`.

### Proof of Concept

```rust
// Construct a client with lag_interval_seconds larger than the timestamp.
let config = ExchangeRateOracleConfig {
    lag_interval_seconds: 1_000_000, // larger than any early-chain timestamp
    ..Default::default()
};
let client = ExchangeRateOracleClient::new(config, METRICS);

// In debug mode: panics with "attempt to subtract with overflow"
// In release mode: quantized_timestamp wraps to ~u64::MAX / 1_000_000,
//   oracle is queried with timestamp ~u64::MAX, returns wrong/error rate.
let _ = client.fetch_rate(500_000).await; // timestamp < lag_interval_seconds
```
