### Title
Missing `max_cost` Enforcement in `op_ash` and `op_lsh` Allows Computation Beyond Cost Budget — (File: `src/more_ops.rs`)

---

### Summary

Both `op_ash` and `op_lsh` accept a `max_cost` parameter but structurally ignore it (declared as `_max_cost`). Unlike every other cost-bounded operator in the codebase, these two shift operators perform the full bignum shift computation and heap allocation **before** any cost check occurs. An attacker-controlled CLVM program can force the VM to execute shift work that exceeds the remaining cost budget, consuming CPU and allocator memory beyond what the cost limit should permit.

---

### Finding Description

The external report's vulnerability class is **missing parameter bounds enforcement**: a function accepts a limit parameter but applies no guard, allowing unbounded resource consumption. The analog here is identical in structure: `op_ash` and `op_lsh` accept `max_cost` but never call `check_cost()`, so the cost limit is not enforced before or during the computation.

**`op_ash`** (arithmetic shift, signed):

```rust
pub fn op_ash(a: &mut Allocator, input: NodePtr, _max_cost: Cost, _flags: ClvmFlags) -> Response {
    let [n0, n1] = get_args::<2>(a, input, "ash")?;
    let (i0, l0) = int_atom(a, n0, "ash")?;
    let a1 = i32_atom(a, n1, "ash")?;
    if !(-65535..=65535).contains(&a1) {
        return Err(EvalErr::ShiftTooLarge(n1));
    }
    let v: Number = if a1 > 0 { i0 << a1 } else { i0 >> -a1 };  // ← full computation
    let l1 = limbs_for_int(&v);
    let r = a.new_number(v)?;                                      // ← heap allocation
    let cost = ASHIFT_BASE_COST + ((l0 + l1) as Cost) * ASHIFT_COST_PER_BYTE;
    Ok(malloc_cost(a, cost, r))                                    // ← cost returned after work
}
``` [1](#0-0) 

**`op_lsh`** (logical shift, unsigned) has the identical pattern: [2](#0-1) 

Both functions use `_max_cost` (underscore prefix = intentionally unused). Neither calls `check_cost()` at any point. The cost is computed from the result size (`l1`) and returned to `run_program` only after the bignum shift and `new_number()` allocation have already completed.

**Contrast with operators that enforce the limit correctly:**

- `op_multiply` calls `check_cost(cost, max_cost)?` inside its argument loop before each multiplication step. [3](#0-2) 
- `op_sha256` calls `check_cost(cost, max_cost)?` after each argument. [4](#0-3) 
- `op_concat` calls `check_cost(cost, max_cost)?` inside its argument loop. [5](#0-4) 

The cost constants confirm the shift operators can produce non-trivial costs:

```
ASHIFT_BASE_COST: Cost = 596
ASHIFT_COST_PER_BYTE: Cost = 3
LSHIFT_BASE_COST: Cost = 277
LSHIFT_COST_PER_BYTE: Cost = 3
``` [6](#0-5) 

For `(ash 500 65535)`, `l0 = 2`, `l1 ≈ 8192` bytes, yielding a cost of `596 + (2 + 8192) × 3 = 25,178` — but the ~8 KB bignum result is allocated and the shift is computed before that cost is ever checked.

The `run_program` loop passes `effective_max_cost - current_cost` as `max_cost` to each operator and checks the accumulated cost only at the **top of the next iteration**, after the operator has already returned: [7](#0-6) 

---

### Impact Explanation

An attacker who submits a CLVM program with a low `max_cost` can still force the VM to perform a full bignum shift (up to 65,535-bit shift of an arbitrarily large input atom) and allocate the result on the heap. The program will ultimately fail with `CostExceeded`, but the computation has already consumed CPU time and allocator memory beyond what the cost budget should have permitted. In a block-validation context, this allows a malicious coin spend to consume more validator resources than its declared cost, undermining the cost model's role as a resource-metering guarantee. The shift amount is bounded to ±65,535 bits and the input atom size is bounded by allocator limits, so the excess work per invocation is bounded — but the invariant that "no work beyond the cost budget is performed" is broken for these two operators.

---

### Likelihood Explanation

Any CLVM program submitted to the Chia network can trigger this. The attacker only needs to craft a coin spend whose puzzle uses `ash` or `lsh` with a large first argument and a shift near 65,535, submitted with a `max_cost` below the actual cost of the operation. No special privileges, keys, or social engineering are required. The entry path is fully attacker-controlled CLVM bytes.

---

### Recommendation

Add a pre-computation `check_cost` call in both `op_ash` and `op_lsh`, using an estimated upper-bound cost based on the input size and maximum possible shift, before performing the bignum shift and allocation. At minimum, add a `check_cost` call after computing the input size `l0` and the maximum possible output size (`l0 + shift_amount / 8`), mirroring the pattern used in `op_multiply` and `op_sha256`.

---

### Proof of Concept

```
; CLVM program: (ash 500 65535)
; Submitted with max_cost = 1000 (below actual cost of ~25,178)
;
; Execution trace:
; 1. run_program calls apply_op(current_cost=0, max_cost=1000)
; 2. apply_op calls op_ash(allocator, args, max_cost=1000, flags)
; 3. op_ash ignores max_cost=1000 entirely (_max_cost)
; 4. Bignum shift: 500 << 65535 → ~8 KB result allocated on heap
; 5. op_ash returns Ok(Reduction(25178, result_node))
; 6. run_program: cost = 0 + 25178 = 25178
; 7. Next loop iteration: 25178 > 1000 → CostExceeded
;
; The ~8 KB allocation and bignum shift computation occurred
; before the cost limit was enforced — work beyond the budget
; was performed unconditionally.
;
; The same applies to op_lsh with (lsh 500 65535).
```
