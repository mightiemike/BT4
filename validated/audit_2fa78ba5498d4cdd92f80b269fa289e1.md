### Title
Fork Headers Stored in `headers_pool` Are Never Garbage-Collected, Permanently Locking NEAR Storage Deposits - (`contract/src/lib.rs`)

---

### Summary

Fork block headers submitted via `submit_blocks` are written exclusively into `headers_pool` through `store_fork_header`. The only GC mechanism, `run_mainchain_gc`, exclusively iterates over `mainchain_height_to_header` — a map that fork headers are never inserted into. As a result, fork headers accumulate in `headers_pool` indefinitely with no removal path, and the NEAR storage-staking tokens paid to cover their storage are permanently locked in the contract.

---

### Finding Description

When `submit_block_header_inner` determines a submitted header belongs to a fork (i.e., its parent is not the current mainchain tip), it calls `store_fork_header`:

```rust
// contract/src/lib.rs:560
self.store_fork_header(&current_header);
```

`store_fork_header` inserts only into `headers_pool`:

```rust
// contract/src/lib.rs:665-667
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
```

It does **not** insert into `mainchain_height_to_header` or `mainchain_header_to_height`.

The GC function `run_mainchain_gc` exclusively iterates over `mainchain_height_to_header`:

```rust
// contract/src/lib.rs:401-408
for height in start_removal_height..end_removal_height {
    let blockhash = &self
        .mainchain_height_to_header
        .get(&height)
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
    self.remove_block_header(blockhash);
    self.mainchain_height_to_header.remove(&height);
}
```

Because fork headers are never inserted into `mainchain_height_to_header`, they are invisible to GC. The only other removal path is `reorg_chain`, which only removes **displaced mainchain blocks** — not the fork headers that were stored before the reorg. A fork that never accumulates enough chainwork to trigger a reorg leaves its headers in `headers_pool` forever.

`submit_blocks` charges the caller for net storage growth and refunds the excess:

```rust
// contract/src/lib.rs:182-194
let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());
require!(amount >= required_deposit, ...);
let refund = amount.saturating_sub(required_deposit);
if refund > NearToken::from_near(0) {
    Promise::new(env::predecessor_account_id()).transfer(refund).into()
}
```

The portion of the deposit consumed by fork header storage is kept by the contract. Since the storage is never freed, those NEAR tokens are permanently locked with no admin withdrawal function and no GC path.

---

### Impact Explanation

NEAR storage staking tokens paid to cover fork header storage are permanently locked in the contract. The `headers_pool` map grows without bound as forks accumulate. There is no owner-callable function to remove stale fork entries, no GC path that touches fork-only `headers_pool` entries, and no mechanism to reclaim the associated storage deposit. This is a direct analog to the original report: assets deposited into the system enter a one-way path with no retrieval mechanism, violating the protocol's storage-accounting invariant.

---

### Likelihood Explanation

Bitcoin, Dogecoin, Litecoin, and Zcash all experience natural forks regularly. Any trusted relayer following the real chain will submit fork headers during normal operation whenever the relayer observes a competing tip. No adversarial intent is required — routine chain operation continuously triggers `store_fork_header`. The accumulation is therefore certain over time, not merely theoretical.

---

### Recommendation

Extend `run_mainchain_gc` (or add a separate `run_fork_gc`) to track and remove fork headers that have not been promoted to mainchain within a configurable age window. One approach: record the submission height of each fork header and remove entries from `headers_pool` whose submission height falls below `mainchain_initial_blockhash`'s height. Alternatively, maintain a separate age-indexed set of fork header hashes so GC can sweep them without a full pool scan.

---

### Proof of Concept

1. Relayer submits a batch containing one mainchain header and one fork header (parent = a block below the current tip) via `submit_blocks`.
2. `submit_block_header_inner` routes the fork header to `store_fork_header` → inserted into `headers_pool` only.
3. `run_mainchain_gc` runs with `batch_size = 1`; it reads `mainchain_height_to_header`, finds no entry for the fork header, and does not touch it.
4. The fork header remains in `headers_pool` after GC. The NEAR tokens paid for its storage byte cost are retained by the contract.
5. Repeat across thousands of natural Bitcoin forks over months: `headers_pool` grows monotonically; the locked NEAR deposit grows proportionally; no code path ever frees it.
