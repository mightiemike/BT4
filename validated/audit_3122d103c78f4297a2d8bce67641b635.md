### Title
Missing Transaction Hash Verification in P2P Sync Allows Malicious Peer to Poison Transaction Index - (`crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary

The p2p sync client accepts `transaction_hash` values from untrusted peers verbatim, without recomputing or verifying them against the transaction body. A malicious peer can supply an arbitrary `Felt` as `transaction_hash` in a `TransactionsResponse`, which is stored directly into the local transaction index. The RPC then serves this attacker-controlled hash as authoritative.

### Finding Description

**Step 1 — Converter blindly trusts the wire hash.**

`TryFrom<protobuf::TransactionInBlock> for (Transaction, TransactionHash)` extracts `transaction_hash` directly from the protobuf field and wraps it as `TransactionHash` with no recomputation: [1](#0-0) 

The transaction body is decoded separately and the two are returned as a pair — there is no call to any `calculate_transaction_hash` variant to cross-check them.

**Step 2 — The FullTransaction is assembled with the unverified hash.**

`TryFrom<protobuf::TransactionWithReceipt> for FullTransaction` calls the above converter and places the result directly into `FullTransaction { transaction, transaction_output, transaction_hash }`: [2](#0-1) 

**Step 3 — The p2p sync client stores the hash without validation (acknowledged by a TODO).**

`parse_data_for_block` in `TransactionStreamFactory` pushes the peer-supplied hash into `block_body.transaction_hashes` with an explicit TODO comment admitting the missing check: [3](#0-2) 

`write_to_storage` then commits the entire `BlockBody` (including the poisoned hashes) to the storage DB: [4](#0-3) 

**Step 4 — No downstream guard recomputes or cross-checks the hashes.**

The header sync path stores a `transaction_commitment` (a Patricia root over `H(tx_hash, tx_signature)` leaves) in the block header: [5](#0-4) 

However, the p2p sync client never verifies that the stored transaction hashes reproduce the `transaction_commitment` from the already-stored block header. The header and transaction streams are independent pipelines with no cross-validation step between them.

**Step 5 — RPC serves the attacker-controlled hash as authoritative.**

`get_transaction_by_hash` looks up the transaction by the stored hash index and returns it with that same hash in the response: [6](#0-5) 

### Impact Explanation

A malicious p2p peer can:
1. Send a valid transaction body paired with an arbitrary `transaction_hash` value.
2. The syncing node stores the transaction under the attacker-chosen hash.
3. `starknet_getTransactionByHash` returns a `TransactionWithHash` where the `transaction_hash` field does not match the hash recomputed from the transaction fields — violating the hash-content binding invariant.
4. The transaction index is permanently poisoned: the real hash is not indexed, so legitimate lookups by the correct hash return "not found", while the fake hash returns the wrong transaction.

This falls under **High — Transaction conversion or signature/hash logic binds the wrong hash** and **High — RPC returns an authoritative-looking wrong value**.

### Likelihood Explanation

Any peer that the syncing node connects to for block body sync can exploit this. No special privileges are required — only the ability to respond to a `TransactionsRequest` query. The TODO comment at line 88 confirms the developers are aware the validation is absent.

### Recommendation

In `parse_data_for_block` (`crates/apollo_p2p_sync/src/client/transaction.rs`), after receiving each `FullTransaction` from a peer, recompute the transaction hash using `get_transaction_hash` (from `starknet_api::transaction_hash`) and compare it against the peer-supplied `transaction_hash`. If they differ, treat the peer as bad (return `ParseDataError::BadPeer`) and do not store the data. The chain ID required for hash computation is available from the node configuration.

Additionally, after all transactions for a block are collected, verify that the computed `transaction_commitment` (Patricia root over `H(tx_hash, tx_sig)` leaves) matches the `transaction_commitment` stored in the block header for that block number.

### Proof of Concept

```rust
// Pseudocode — illustrates the missing check
let arbitrary_felt = Felt::from(0xdeadbeef_u64);
let proto_tx = protobuf::TransactionInBlock {
    transaction_hash: Some(arbitrary_felt.into()),
    txn: Some(protobuf::transaction_in_block::Txn::InvokeV3(valid_invoke_v3_body())),
};
let (transaction, transaction_hash) =
    <(Transaction, TransactionHash)>::try_from(proto_tx).unwrap();

// transaction_hash is the attacker value, not the recomputed hash
assert_eq!(transaction_hash.0, arbitrary_felt);

// The real hash differs
let real_hash = get_transaction_hash(&transaction, &chain_id, &TransactionOptions::default()).unwrap();
assert_ne!(real_hash, transaction_hash); // passes — divergence confirmed

// After write_to_storage, starknet_getTransactionByHash(arbitrary_felt)
// returns the transaction with transaction_hash = arbitrary_felt
```
