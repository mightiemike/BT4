### Title
Queued Slow-Mode Transactions Cannot Be Canceled and `executeSlowModeTransaction` Is Unprotected — (`File: core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.sol` implements a slow-mode transaction queue as a sequencer-bypass fallback. Once a transaction is enqueued via `submitSlowModeTransaction` or `depositCollateralWithReferral`, there is no mechanism to cancel it. Additionally, `executeSlowModeTransaction()` carries no access control, meaning any unprivileged caller can force-execute the next queued transaction after the delay window expires. This is a direct structural analog to the Compound `BaseBridgeReceiver` / `Timelock` issue: a privileged queue with no cancel path and an unguarded execution entry point.

---

### Finding Description

`executeSlowModeTransaction()` in `Endpoint.sol` is declared `external` with no `onlyOwner`, `onlySequencer`, or any other guard:

```solidity
function executeSlowModeTransaction() external {          // line 231 — no modifier
    SlowModeConfig memory _slowModeConfig = slowModeConfig;
    _executeSlowModeTransaction(_slowModeConfig, false);
    nSubmissions += 1;
    slowModeConfig = _slowModeConfig;
}
``` [1](#0-0) 

`_executeSlowModeTransaction` deletes the entry from the queue **before** attempting execution:

```solidity
SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
delete slowModeTxs[_slowModeConfig.txUpTo++];
``` [2](#0-1) 

There is no `cancelSlowModeTransaction` function anywhere in the contract suite — confirmed by the absence of any such function across all production Solidity files. [3](#0-2) 

The slow-mode queue accepts owner-privileged transaction types — `WithdrawInsurance`, `DelistProduct`, `DumpFees`, `RebalanceXWithdraw`, `UpdateTierFeeRates`, `AddNlpPool`, `UpdateNlpPool`, `DeleteNlpPool`, `ForceRebalanceNlpPool`, `NlpProfitShare`, `UpdateBuilder` — all gated at submission time by `require(sender == owner())`:

```solidity
} else if (
    txType == IEndpoint.TransactionType.WithdrawInsurance ||
    txType == IEndpoint.TransactionType.DelistProduct ||
    ...
    txType == IEndpoint.TransactionType.UpdateBuilder
) {
    require(sender == owner());
``` [4](#0-3) 

Once any of these are enqueued, the 3-day delay begins. After it expires, **any EOA** can call `executeSlowModeTransaction()` to force-execute the head of the queue. The protocol has no mechanism to stop this.

---

### Impact Explanation

**Concrete state delta corrupted:** Insurance fund balance, product listing state, fee tier configuration, NLP pool configuration, or builder registry — depending on which owner-submitted slow-mode transaction is at the head of the queue.

Specific scenarios:
- Owner queues `WithdrawInsurance` to drain the insurance fund to a recipient address. Owner later discovers the recipient address is wrong or the action is premature. No cancel exists. After 3 days, any user calls `executeSlowModeTransaction()` and the insurance fund is drained to the wrong address. `clearinghouse.withdrawInsurance(transaction, nSubmissions)` executes unconditionally.
- Owner queues `DelistProduct`. Market conditions change and the delist is no longer appropriate. No cancel exists. Any user force-executes `clearinghouse.delistProduct(transaction)` after the delay.
- Owner queues `DeleteNlpPool`. Any user force-executes `clearinghouse.clearNlpPoolPosition(...)` after the delay, permanently removing the pool.

Additionally, each call to `executeSlowModeTransaction()` increments `nSubmissions`. The sequencer's `submitTransactionsChecked` validates `idx == nSubmissions`:

```solidity
function validateSubmissionIdx(uint64 idx) private view {
    require(idx == nSubmissions, ERR_INVALID_SUBMISSION_INDEX);
}
``` [5](#0-4) 

An attacker who force-executes slow-mode transactions desynchronizes the sequencer's expected submission index, causing `submitTransactionsChecked` to revert until the sequencer adjusts its counter.

---

### Likelihood Explanation

**High.** The entry point is public and requires no special role, token balance, or prior state. Any user monitoring the mempool can observe when a slow-mode transaction's `executableAt` timestamp is reached and immediately call `executeSlowModeTransaction()`. The 3-day delay window is publicly visible via `getSlowModeTx()`:

```solidity
function getSlowModeTx(uint64 idx) external view returns (
    SlowModeTx memory, uint64, uint64
) {
    return (slowModeTxs[idx], slowModeConfig.txUpTo, slowModeConfig.txCount);
}
``` [6](#0-5) 

No capital, no privileged access, and no complex setup is required.

---

### Recommendation

1. **Add access control to `executeSlowModeTransaction()`** — restrict it to `onlyOwner` or `onlySequencer`, or at minimum require the caller to be the original transaction submitter.
2. **Implement a `cancelSlowModeTransaction(uint64 idx)` function** — restricted to `onlyOwner`, allowing the protocol to remove a specific entry from `slowModeTxs` before it is executed. For user-submitted transactions (e.g., `WithdrawCollateral`), allow the original sender to cancel their own entry.

---

### Proof of Concept

1. Owner calls `submitSlowModeTransaction(encodedWithdrawInsurance)` — enqueued at index `N` with `executableAt = block.timestamp + 3 days`.
2. Owner realizes the recipient address in the transaction is incorrect.
3. Owner searches for a cancel function — none exists in `Endpoint.sol`, `EndpointTx.sol`, or any other production contract.
4. After 3 days, attacker calls `Endpoint.executeSlowModeTransaction()` with no arguments and no special role.
5. `_executeSlowModeTransaction` deletes the entry and calls `this.processSlowModeTransaction(txn.sender, txn.tx)`.
6. `processSlowModeTransactionImpl` routes to `clearinghouse.withdrawInsurance(transaction, nSubmissions)`.
7. Insurance fund is transferred to the incorrect recipient address. Protocol has no recourse. [1](#0-0) [7](#0-6)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }

        SlowModeConfig memory _slowModeConfig = slowModeConfig;
        _executeSlowModeTransaction(_slowModeConfig, false);
