### Title
Missing Programmatic Enforcement of `receipt_l2_gas` == `max_block_size` Invariant Causes Wrong EIP-1559 Fee Market Calculation — (`crates/blockifier/src/bouncer.rs`, `crates/apollo_versioned_constants/src/lib.rs`)

---

### Summary

`BouncerWeights::receipt_l2_gas` (batcher block-closing cap) and `apollo_versioned_constants::VersionedConstants::max_block_size` (orchestrator EIP-1559 fee-market parameter) are documented with explicit "Must stay in sync" comments but are configured in completely separate subsystems with no programmatic cross-check. When a new Starknet version is deployed that changes `max_block_size` in the orchestrator versioned constants, the batcher's `receipt_l2_gas` is not automatically updated. The two values diverge silently, causing the EIP-1559 gas price calculation to use a wrong block-size denominator, producing systematically incorrect base gas prices for every subsequent block.

---

### Finding Description

Two separate constants govern the same logical quantity — the maximum receipt-based L2 gas per block — but live in different subsystems with no shared source of truth:

**1. Batcher side — `BouncerWeights::receipt_l2_gas`** [1](#0-0) 

The comment explicitly states it must equal `max_block_size`, but the value is a free-standing field in `BouncerConfig`, loaded from the node's static JSON config (`batcher_config.static_config.block_builder_config.bouncer_config.block_max_capacity.receipt_l2_gas`). [2](#0-1) 

**2. Orchestrator side — `VersionedConstants::max_block_size`** [3](#0-2) 

This value is version-gated and changes with each Starknet protocol version:

| Version | `max_block_size` |
|---------|-----------------|
| 0.14.0  | 4,000,000,000   |
| 0.14.1  | 5,000,000,000   |
| 0.14.2+ | 5,800,000,000   | [4](#0-3) [5](#0-4) 

The orchestrator always calls `VersionedConstants::latest_constants()` to obtain `max_block_size` and `gas_target` for the EIP-1559 price update: [6](#0-5) 

**3. No programmatic cross-check exists**

`validate_node_config()` and `cross_member_validations()` check component-presence mismatches but never assert `bouncer_config.block_max_capacity.receipt_l2_gas == orchestrator_versioned_constants.max_block_size`: [7](#0-6) 

The config schema exposes both values as independent, separately-settable fields: [8](#0-7) 

**4. The divergence path**

When a new Starknet version is released with a larger `max_block_size` (as happened from 0.14.1 → 0.14.2, a 16% increase), the orchestrator automatically picks up the new value via `latest_constants()`. The batcher's `receipt_l2_gas` remains at the old value until the operator manually edits the config. The two components now disagree on the block size, and the fee market computes the wrong price adjustment for every block.

---

### Impact Explanation

`max_block_size` and `gas_target` (60% of `max_block_size`) are the denominators in the EIP-1559 price formula. If `receipt_l2_gas` (actual block cap) < `max_block_size` (fee-market denominator), every block appears under-congested to the fee market even when the batcher is closing blocks at capacity, causing the base gas price to be driven down incorrectly. The reverse divergence causes artificial price inflation. Both directions produce wrong `starknet_gasPrice` values returned by the RPC and wrong fee estimates, with direct economic impact on every transaction.

This matches: **Critical — Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact.**

---

### Likelihood Explanation

The invariant has already been broken across historical versions (0.14.0 → 0.14.1 → 0.14.2 each changed `max_block_size`). Any operator who upgrades the Starknet version without simultaneously updating the batcher config will silently diverge. The comment-only documentation is the only guard, directly mirroring the `setPerTokenWalletCap` pattern from the external report.

---

### Recommendation

Add a startup assertion in `validate_node_config()` (or in the batcher/orchestrator initialization) that reads the current orchestrator `VersionedConstants::latest_constants().max_block_size` and asserts it equals `batcher_config.block_builder_config.bouncer_config.block_max_capacity.receipt_l2_gas`. For example:

```rust
// In cross_member_validations() or batcher startup:
let vc_max = apollo_versioned_constants::VersionedConstants::latest_constants().max_block_size;
let bouncer_max = batcher_cfg.block_builder_config.bouncer_config
    .block_max_capacity.receipt_l2_gas;
if vc_max != bouncer_max {
    return Err(ConfigError::InvariantViolation(format!(
        "receipt_l2_gas ({bouncer_max}) must equal orchestrator max_block_size ({vc_max})"
    )));
}
```

This converts the manual "must stay in sync" comment into a hard startup failure, exactly as the external report recommended converting `getMaxCommunityLpPositon()` into a `require` statement.

---

### Proof of Concept

1. Node is running on Starknet v0.14.1; operator sets `receipt_l2_gas = 5_000_000_000` (correct for that version).
2. Protocol upgrades to v0.14.2; `orchestrator_versioned_constants_0_14_2.json` sets `max_block_size = 5_800_000_000`.
3. Orchestrator calls `VersionedConstants::latest_constants()` and now uses `max_block_size = 5_800_000_000`, `gas_target = 1_500_000_000` (as in the diff regression file).
4. Batcher still closes blocks at `receipt_l2_gas = 5_000_000_000`.
5. A fully-packed block (5B gas used) is reported to the fee market. The fee market computes `gas_used / gas_target = 5_000_000_000 / 1_500_000_000 ≈ 3.33×` — far above the target — and raises the base gas price aggressively, even though the batcher is operating normally at its own capacity.
6. Every subsequent block's gas price is wrong; fee estimation via `starknet_estimateFee` returns inflated values with no on-chain signal of the misconfiguration. [9](#0-8) [10](#0-9)
