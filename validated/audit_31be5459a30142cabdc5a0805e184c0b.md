Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the actual caller (`sender`) and gates deposits by checking only the `owner` parameter — the address that receives LP shares. Because `MetricOmmPool.addLiquidity` accepts an arbitrary `owner` with no restriction on who may supply it, any address can bypass the deposit allowlist by naming an allowlisted address as `owner`, while the real depositor is never verified.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as the first argument and the caller-supplied `owner` as the second to the hook: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` discards the first argument entirely (unnamed `address`) and checks only `owner`: [2](#0-1) 

The sibling `SwapAllowlistExtension.beforeSwap` correctly captures and checks the first argument (`sender`): [3](#0-2) 

`removeLiquidity` enforces `msg.sender == owner`, preventing the same pattern there: [4](#0-3) 

But `addLiquidity` has no equivalent guard, so any caller may freely supply an arbitrary `owner`. The exploit path is: Bob (not allowlisted) calls `addLiquidity(alice, salt, deltas, callbackData, extensionData)`. The hook receives `sender = bob` (ignored) and `owner = alice`. The check `allowedDepositor[pool][alice]` returns `true`, no revert occurs, Bob's callback transfers tokens into the pool, and Alice receives LP shares she never requested.

## Impact Explanation

The deposit allowlist is the pool admin's primary mechanism for restricting which addresses may inject liquidity (e.g., KYC/AML gating, curated market-maker whitelists). With this bug, the allowlist is fully bypassed: any address can deposit into a restricted pool, the pool receives tokens from unauthorized sources, and allowlisted addresses can be forced into unwanted LP positions they must actively unwind. This is a direct admin-boundary break where an unprivileged path circumvents an access-control gate, matching the allowed impact category.

## Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with a known allowlisted address as `owner`. The `allowedDepositor` mapping is public, so any observer can identify valid allowlisted addresses. No special privileges, flash loans, or oracle manipulation are needed. The attack is immediately repeatable by any EOA or contract.

## Recommendation

Change `beforeAddLiquidity` to capture and check the first parameter (`sender`) instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
// Fixed:
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
  return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Also update `setAllowedToDeposit`, `isAllowedToDeposit`, and the `allowedDepositor` mapping semantics to reflect that the allowlisted entity is the caller, not the LP share recipient.

## Proof of Concept

1. Pool `P` is deployed with `DepositAllowlistExtension`. `allowAllDepositors[P] = false`.
2. Admin calls `setAllowedToDeposit(P, alice, true)`. Alice is the only allowlisted depositor.
3. Bob (not allowlisted) calls `P.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. `beforeAddLiquidity` is invoked with first arg = `bob` (discarded) and `owner = alice`.
5. Check: `allowedDepositor[P][alice]` → `true` → no revert.
6. Bob's `metricOmmAddLiquidityCallback` transfers tokens into the pool; Alice receives LP shares.
7. Bob has deposited into a restricted pool without being on the allowlist.
