### Title
`Endpoint.depositCollateralWithReferral` silently returns instead of reverting on sub-minimum deposits, breaking composability — (File: `core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral` is a `public` function declared in the `IEndpoint` interface. When the deposit amount is below the protocol minimum, it silently returns (no-op) instead of reverting. Any external protocol or contract that calls this function with a below-minimum amount receives a successful transaction with no tokens transferred, no slow-mode entry queued, and no event emitted — with no way to detect the failure.

---

### Finding Description

`depositCollateral` (the `external` entry point) correctly reverts with `ERR_DEPOSIT_TOO_SMALL` when the amount is below the minimum:

```solidity
// Endpoint.sol L103-121
function depositCollateral(bytes12 subaccountName, uint32 productId, uint128 amount) external {
    bytes32 subaccount = bytes32(abi.encodePacked(msg.sender, subaccountName));
    require(isValidDepositAmount(subaccount, productId, amount), ERR_DEPOSIT_TOO_SMALL);
    depositCollateralWithReferral(subaccount, productId, amount, DEFAULT_REFERRAL_CODE);
}
```

However, `depositCollateralWithReferral` — the `public` function that is part of `IEndpoint` and callable directly by any external contract — takes a different path:

```solidity
// Endpoint.sol L137-142
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked when there are
    // multiple assets awaiting credit but one of them is below the minimum deposit amount.
    // we can just skip the deposit and continue with the next asset.
    return;
}
```

The silent `return` was introduced to accommodate `DirectDepositV1.creditDeposit()`, which iterates over all product IDs and calls `depositCollateralWithReferral` for each one. If one product's balance is below the minimum, the loop must not abort. However, this design decision is now embedded in a `public` interface function, making it the observable behavior for all callers.

`DirectDepositV1.creditDeposit()` demonstrates the concrete trigger path:

```solidity
// DirectDepositV1.sol L91-98
if (balance != 0) {
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
}
```

If `balance` is non-zero but below the minimum deposit threshold, the approval is set, `depositCollateralWithReferral` is called, it silently returns, no tokens move, no slow-mode entry is created, and the subaccount is never credited. The caller observes a successful transaction.

---

### Impact Explanation

Any external protocol integrating Nado via the `IEndpoint.depositCollateralWithReferral` interface cannot distinguish a successful deposit from a silently-skipped one. There is no return value, no event, and no revert. Integrators who assume "no revert = deposit succeeded" will record incorrect internal state (e.g., crediting a user's balance in their own system when no collateral was actually deposited into Nado). This is a direct composability break for the `IEndpoint` interface.

---

### Likelihood Explanation

`depositCollateralWithReferral` is `public`, declared in `IEndpoint`, and is the standard integration point for any contract depositing on behalf of a subaccount (as `DirectDepositV1` does). The silent-return path is triggered whenever `amount` is below the minimum deposit threshold — a realistic condition for dust balances, small token amounts, or tokens with high USD value per unit. Any integrator relying on the standard "revert-on-failure" contract convention will be silently broken.

---

### Recommendation

The silent `return` is a design compromise made for the `DirectDepositV1` multi-asset loop. The fix should decouple the DDA-specific tolerance from the public interface:

1. Add a `bool` return value to `depositCollateralWithReferral` (or emit a `DepositSkipped` event) so callers can detect the no-op case.
2. Alternatively, expose a separate internal function (e.g., `_depositCollateralWithReferralUnchecked`) used only by `creditDeposit`, and have the public `depositCollateralWithReferral` revert on sub-minimum amounts, consistent with `depositCollateral`.

---

### Proof of Concept

1. Deploy Nado with a minimum deposit of, say, 10 USDC.
2. An external protocol calls `endpoint.depositCollateralWithReferral(subaccount, productId, 1e6 /* 1 USDC */, "ref")` directly.
3. The call passes sanction checks, reaches line 137, `isValidDepositAmount` returns `false`, and the function returns at line 141.
4. Transaction succeeds. No tokens transferred. No slow-mode entry queued. No event emitted.
5. The external protocol, observing no revert, incorrectly assumes the deposit succeeded and credits the user.
