### Title
`get_sortable_validator_online_ratio` ignores `endorsement_cutoff_threshold`, allowing below-threshold chunk validators to be exempted from kickout and receive unearned epoch rewards — (`chain/epoch-manager/src/validator_stats.rs`)

---

### Summary

`get_sortable_validator_online_ratio` always passes `None` for the endorsement cutoff threshold, computing a raw fractional endorsement ratio for kickout-exemption sorting. The actual kickout check applies a binary cutoff (`chunk_validator_only_kickout_threshold`). The two code paths evaluate the same endorsement data with divergent semantics in the same epoch-finalization call, causing a chunk-validator-only node with an endorsement ratio just below the threshold to be sorted as having a higher online ratio than a node with a much lower ratio, and thus be exempted from kickout. Exempted validators are not kicked out and continue to receive epoch rewards they should not receive.

---

### Finding Description

At epoch finalization, `compute_validators_to_reward_and_kickout` performs two operations that must be consistent:

**Step 1 — Exemption sorting** (`lib.rs:500–506`): validators are sorted by online ratio to determine which ones are protected from kickout (the highest-ratio validators, up to `validator_max_kickout_stake_perc` of total stake):

```rust
let mut sorted_validators = validator_block_chunk_stats
    .iter()
    .map(|(account, stats)| (get_sortable_validator_online_ratio(stats), account))
    .collect_vec();
``` [1](#0-0) 

`get_sortable_validator_online_ratio` unconditionally calls `get_validator_online_ratio(stats, None)` — the `None` means no cutoff is applied and the raw fractional endorsement ratio is used:

``` [2](#0-1) 

**Step 2 — Kickout check** (`lib.rs:541–555`): for chunk-validator-only nodes, the binary cutoff is applied:

```rust
let chunk_validator_only =
    stats.block_stats.expected == 0 && stats.chunk_stats.expected() == 0;

    && stats.chunk_stats.endorsement_stats().less_than(chunk_validator_only_kickout_threshold)

        ValidatorKickoutReason::NotEnoughChunkEndorsements { ... }

``` [3](#0-2) 

The binary cutoff is also applied correctly in `finalize_epoch` when computing rewards:

```rust

    epoch_config.chunk_validator_only_kickout_threshold,
),
``` [4](#0-3) 

The `get_endorsement_ratio` helper shows the exact divergence: with `None`, the raw ratio `(produced, expected)` is returned; with `Some(threshold)`, it returns `(0,1)` or `(1,1)`:

```rust
fn get_endorsement_ratio(stats: &ValidatorStats, cutoff_threshold: Option<u8>) -> (u64, u64) {
    let (numer, denom) = if stats.expected == 0 {
        (0, 0)
    } else if let Some(threshold) = cutoff_threshold {
        if stats.less_than(threshold) { (0, 1) } else { (1, 1) }
    } else {
        (stats.produced, stats.expected)
    };
``` [5](#0-4) 

**Concrete divergence** (threshold = 70%, mainnet `chunk_validator_only_kickout_threshold`):

| Validator | Endorsement ratio | Sorting ratio (raw, `None`) | Kickout check (binary, `Some(70)`) |
|---|---|---|---|
| A | 69/100 = 0.69 | **0.69** | **0** (below threshold → kicked out) |
| B | 50/100 = 0.50 | **0.50** | **0** (below threshold → kicked out) |

With binary cutoff sorting, A and B have the same ratio (0), so exemption is decided by stake/account_id. With raw ratio sorting, A is sorted higher than B and is more likely to be exempted. If A is exempted, it is not kicked out despite failing the endorsement threshold, and it continues to receive epoch rewards.

---

### Impact Explanation

A chunk-validator-only node with an endorsement ratio just below `chunk_validator_only_kickout_threshold` (e.g., 69% when threshold is 70%) is sorted as having a higher online ratio than a node with a much lower ratio (e.g., 50%). When the `validator_max_kickout_stake_perc` cap is reached, the 69% node is exempted from kickout instead of the 50% node. The exempted node:
1. Is not kicked out despite failing the endorsement threshold.
2. Remains in `validator_block_chunk_stats` passed to `calculate_reward`.
3. Receives a non-zero epoch reward because its raw endorsement ratio (0.69) is above `online_min_threshold` (0.90 × 1 = 0.90 after binary remapping would give 0, but with raw ratio it gives 0.69 which may still be above the min threshold depending on block/chunk production).

The reward pool is fixed per epoch; unearned rewards paid to the exempted validator dilute rewards for all other validators.

---

### Likelihood Explanation

This is reachable on every epoch boundary whenever `validator_max_kickout_stake_perc < 100` (mainnet: 30 since protocol version 56) and multiple chunk-validator-only nodes are simultaneously below the endorsement threshold. No privileged role is required; any validator controls its own endorsement production ratio. A validator can deliberately produce endorsements at just below the threshold to maximize its raw ratio among below-threshold peers and increase its probability of being exempted.

---

### Recommendation

Pass the endorsement cutoff threshold to `get_sortable_validator_online_ratio` so that the sorting metric is consistent with the kickout metric:

```rust
// In compute_validators_to_reward_and_kickout, pass the cutoff:
.map(|(account, stats)| (
    get_sortable_validator_online_ratio(stats, Some(chunk_validator_only_kickout_threshold)),
    account
))
```

And update `get_sortable_validator_online_ratio` to accept and forward the threshold:

```rust
pub(crate) fn get_sortable_validator_online_ratio(
    stats: &BlockChunkValidatorStats,
    endorsement_cutoff_threshold: Option<u8>,

    let ratio = get_validator_online_ratio(stats, endorsement_cutoff_threshold);

### Proof of Concept

Given `chunk_validator_only_kickout_threshold = 70`, `validator_max_kickout_stake_perc = 30`:

- Validators: block producers `test0`, `test1` (stake 1000 each); chunk-validator-only `test2` (69/100 endorsements, stake 500), `test3` (50/100 endorsements, stake 500).
- Both `test2` and `test3` are below the 70% threshold and should be kicked out.
- Total stake = 3000. `exempt_perc = 70`. `min_keep_stake = 2100`.
- Sorting by raw ratio: `test2` (0.69) > `test3` (0.50). Exemption fills from highest: `test0` (1.0, stake 1000), `test1` (1.0, stake 1000), `test2` (0.69, stake 500) → exempted stake = 2500 ≥ 2100. `test2` is exempted.
- Kickout check: `test2` is in `exempted_validators` → skipped. `test3` is kicked out.
- **Result**: `test2` (69% endorsement, below threshold) is not kicked out and receives epoch rewards.
- **With binary cutoff sorting**: `test2` and `test3` both have ratio 0; tie-broken by stake (equal) then account_id. The outcome depends on account_id ordering, not on which validator is "less bad" — the protocol's intended behavior. [7](#0-6) [8](#0-7)

    /// When computing validators to kickout, we exempt some validators first so that
    /// the total stake of exempted validators exceed a threshold. This is to make sure
    /// we don't kick out too many validators in case of network instability.
    /// We also make sure that these exempted validators were not kicked out in the last epoch,
    /// so it is guaranteed that they will stay as validators after this epoch.
    ///
    /// `accounts_sorted_by_online_ratio`: Validator accounts sorted by online ratio in ascending order.
    fn compute_exempted_kickout(
        epoch_info: &EpochInfo,
        accounts_sorted_by_online_ratio: &Vec<AccountId>,
        total_stake: Balance,
        exempt_perc: u8,
        prev_validator_kickout: &HashMap<AccountId, ValidatorKickoutReason>,
    ) -> HashSet<AccountId> {
        // We want to make sure the total stake of validators that will be kicked out in this epoch doesn't exceed
        // config.validator_max_kickout_stake_ratio of total stake.
        // To achieve that, we sort all validators by their average uptime (average of block and chunk
        // uptime) and add validators to `exempted_validators` one by one, from high uptime to low uptime,
        // until the total excepted stake exceeds the ratio of total stake that we need to keep.
        // Later when we perform the check to kick out validators, we don't kick out validators in
        // exempted_validators.
        let mut exempted_validators = HashSet::new();
        let min_keep_stake = Balance::from_yoctonear(
            (U256::from(total_stake.as_yoctonear()) * U256::from(exempt_perc as u128)
                / U256::from(100u128))
            .as_u128(),
        );
        let mut exempted_stake = Balance::ZERO;
        for account_id in accounts_sorted_by_online_ratio.into_iter().rev() {
            if exempted_stake >= min_keep_stake {
                break;
            }
            if !prev_validator_kickout.contains_key(account_id) {
                let validator_stake = epoch_info
                    .get_validator_by_account(account_id)
                    .map(|v| v.stake())
                    .unwrap_or_default();
                exempted_stake = exempted_stake.checked_add(validator_stake).unwrap();
                exempted_validators.insert(account_id.clone());
            }
        }
        exempted_validators
    }

pub(crate) fn get_sortable_validator_online_ratio(stats: &BlockChunkValidatorStats) -> BigRational {
    let ratio = get_validator_online_ratio(stats, None);
    let mut bytes: [u8; size_of::<U256>()] = [0; size_of::<U256>()];
    ratio.numer().to_little_endian(&mut bytes);
    let bignumer = BigUint::from_bytes_le(&bytes);
    ratio.denom().to_little_endian(&mut bytes);
    let bigdenom = BigUint::from_bytes_le(&bytes);
    BigRational::new(bignumer.try_into().unwrap(), bigdenom.try_into().unwrap())
}
