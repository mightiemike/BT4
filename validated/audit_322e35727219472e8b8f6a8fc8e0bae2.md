Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on `owner` Instead of `sender`, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual token-providing caller) and gates access on `owner` (the freely-chosen LP-position recipient). Because `MetricOmmPool.addLiquidity` imposes no constraint that `owner == msg.sender`, any unprivileged caller can pass an allowlisted address as `owner` and satisfy the allowlist check unconditionally, depositing tokens into a pool whose admin intended to restrict deposits to approved addresses only.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the position recipient to the extension hook:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` forwards both values verbatim:

```solidity
// metric-core/contracts/ExtensionCalling.sol L95-98
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

`DepositAllowlistExtension.beforeAddLiquidity` declares the first parameter (the actual depositor/`sender`) as unnamed and discarded, then checks only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`addLiquidity` has no guard requiring `owner == msg.sender`. The only `owner` validation in the periphery router (`MetricOmmPoolLiquidityAdder._validateOwner`) only rejects `address(0)`, and the attacker can bypass the router entirely by calling the pool directly.

**Exploit path:**
1. Pool admin configures `DepositAllowlistExtension` with `allowedDepositor[pool][victim] = true` and `allowedDepositor[pool][attacker] = false`.
2. Attacker calls `pool.addLiquidity(owner=victim, salt, deltas, callbackData, "")` directly.
3. The hook receives `sender=attacker` (discarded) and `owner=victim` (checked). `allowedDepositor[pool][victim] == true` → check passes.
4. Attacker's `IMetricOmmModifyLiquidityCallback` implementation provides the tokens.
5. LP position is minted to `victim`. Attacker's tokens are irrecoverably transferred to `victim`'s position.

The wrong value is the extension's access-control decision: `allowedDepositor[msg.sender][owner]` evaluates to `true` for `owner=victim` when it should be evaluating `allowedDepositor[msg.sender][sender]` for `sender=attacker`, which would be `false`.

## Impact Explanation

The deposit allowlist — the sole curation mechanism of `DepositAllowlistExtension` — is completely defeated by any unprivileged caller. Any pool relying on this extension for KYC, compliance, or depositor curation has that guarantee nullified. This is an admin-boundary break: the access control the pool admin configured is bypassed without any privileged role. Secondary effects include the victim receiving unsolicited LP shares (redeemable via `removeLiquidity` since `msg.sender == owner` is enforced there) and the attacker irrecoverably losing their deposited tokens to the victim's position.

## Likelihood Explanation

The attack requires only a direct call to `pool.addLiquidity` with any known allowlisted address as `owner`. No privileged role, no special token, no oracle manipulation, no flash loan. Any address can execute it at any time. The only cost is the attacker's own tokens, which are transferred to the victim's LP position.

## Recommendation

Check `sender` (the actual depositor/token-provider) instead of `owner` (the LP-position recipient):

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

```solidity
function test_allowlistBypass() public {
    // Setup: victim is allowlisted, attacker is not
    vm.prank(poolAdmin);
    extension.setAllowedToDeposit(address(pool), victim, true);
    // allowedDepositor[pool][attacker] == false

    // Attacker calls pool.addLiquidity directly with owner=victim
    // Attacker implements IMetricOmmModifyLiquidityCallback to supply tokens
