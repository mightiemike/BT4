### Title
Declare Transactions Bypass `max_l2_gas_amount` Gateway Admission Limit — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The `StatelessTransactionValidator::validate_resource_bounds` function enforces the configured `max_l2_gas_amount` ceiling for Invoke and DeployAccount transactions but explicitly skips that check for `RpcTransaction::Declare`. Any user can submit a Declare transaction with `l2_gas.max_amount = u64::MAX` and the gateway will admit it, violating the documented admission invariant.

### Finding Description

In `crates/apollo_gateway/src/stateless_transaction_validator.rs` lines 78–85, the `max_l2_gas_amount` check is guarded by a type branch that silently passes all Declare transactions:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // no check — falls through
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
```

The production config (`crates/apollo_deployments/resources/app_configs/gateway_config.json` line 25 and `crates/apollo_gateway_config/src/config.rs` line 193) sets `max_l2_gas_amount = 1_210_000_000`. This limit is enforced for every transaction type **except** Declare. The test `valid_l2_gas_amount_on_declare` (lines 191–201 of `stateless_transaction_validator_test.rs`) explicitly asserts that a Declare transaction with `max_amount: GasAmount(200)` passes validation even when `max_l2_gas_amount: 100` is configured, confirming the bypass is intentional at the code level but unintended from a protocol-limit perspective.

The `StatelessTransactionValidatorConfig` field description reads: *"Maximum allowed L2 gas amount for transactions."* — no Declare exemption is documented in the config schema (`crates/apollo_node/resources/config_schema.json` line 3172–3175).

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

A user can craft a `RpcDeclareTransactionV3` with `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)` and submit it to the gateway. The stateless validator will accept it. The transaction enters the mempool with a claimed L2 gas ceiling of `2^64 − 1`, far above the 1.21 B unit limit that every other transaction type must respect. This:

1. Breaks the admission-control invariant that `max_l2_gas_amount` is a hard ceiling for all transactions.
2. Allows a Declare transaction to carry a `max_possible_fee` of `u64::MAX × max_price_per_unit`, which can overflow or distort any downstream fee-related admission logic that reads `resource_bounds` before execution.
3. Enables mempool pollution with transactions whose stated resource envelope is orders of magnitude larger than the protocol allows, potentially affecting bouncer accounting or block-building heuristics that rely on the declared bounds.

### Likelihood Explanation

**High.** The bypass requires no privilege — any account can submit a Declare transaction. The code path is unconditional: the `if let RpcTransaction::Declare(_) = tx { }` branch is always taken for Declare transactions regardless of the `l2_gas.max_amount` value. The TODO comment confirms the gap is known but unresolved.

### Recommendation

Remove the Declare exemption and apply the same `max_l2_gas_amount` check uniformly:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher L2 gas ceiling (e.g., for compilation), introduce a separate `max_l2_gas_amount_declare` config field with an explicit, documented value rather than leaving the bound unchecked.

### Proof of Concept

1. Construct a `RpcDeclareTransactionV3` with `resource_bounds.l2_gas = ResourceBounds { max_amount: GasAmount(u64::MAX), max_price_per_unit: GasPrice(min_gas_price) }`.
2. Submit to the gateway endpoint.
3. `StatelessTransactionValidator::validate_resource_bounds` reaches line 79, matches `RpcTransaction::Declare(_)`, and returns `Ok(())` without checking `max_amount`.
4. The transaction is admitted to the mempool with `l2_gas.max_amount = 18_446_744_073_709_551_615`, violating the `max_l2_gas_amount = 1_210_000_000` limit.

The existing test `valid_l2_gas_amount_on_declare` (lines 173–201) already demonstrates this: it passes `max_amount: GasAmount(200)` against `max_l2_gas_amount: 100` for a Declare transaction and asserts `Ok(())`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
