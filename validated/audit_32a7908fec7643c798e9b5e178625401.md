### Title
Gateway `validate_resource_bounds` uses stale previous-block L2 gas price instead of `next_l2_gas_price` from block header, causing false admission and rejection at the mempool boundary - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidator::validate_resource_bounds` reads the **current committed block's** L2 gas price to decide whether to admit or reject a transaction. The actual L2 gas price that will govern execution is the **next block's** price, computed by the EIP-1559-style `calculate_next_base_gas_price` in the consensus orchestrator. Because the two prices diverge by up to ~1% per block, the gateway can reject transactions that would succeed during execution (false rejection) or admit transactions that will fail during execution (false admission). The developer acknowledged this with an explicit TODO comment at the exact line.

---

### Finding Description

In `validate_resource_bounds`, the gateway fetches the gas price from the last committed block:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The threshold applied is `min_gas_price_percentage% × previous_block_l2_gas_price`. A transaction is rejected if `tx_l2_gas_price < threshold`. [2](#0-1) 

The **actual** L2 gas price for the next block is computed by `calculate_next_base_gas_price` in the consensus orchestrator, which adjusts the price up or down based on the previous block's gas usage:

```rust
let adjusted_price_u256 =
    if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };
``` [3](#0-2) 

The snapshot tests confirm the price can move by ~1% per block (e.g., 30 Gwei → 29.6875 Gwei under low congestion, or → 30.3125 Gwei under high congestion): [4](#0-3) 

The `next_l2_gas_price` is already stored in the committed block header (used by the protobuf converter and the echonet replay tool), but `get_block_info()` does not surface it: [5](#0-4) 

The same stale price is also used in `run_validate_entry_point`, where the block context is built from the previous block's `BlockInfo` (only the block number is incremented): [6](#0-5) 

This means both the pre-filter and the blockifier validation inside the gateway use the previous block's gas prices, while the batcher executes transactions against the next block's gas prices supplied by the consensus orchestrator. [7](#0-6) 

---

### Impact Explanation

**False rejection (valid transaction denied):** When the previous block had high congestion, the next block's price is *higher* than the current block's price. A transaction with `tx_l2_gas_price` between `min_gas_price_percentage% × P_prev` and `P_next` passes the execution check (`tx_price ≥ P_next`) but is rejected by the gateway (`tx_price < threshold`). The user's valid transaction is permanently denied at the gateway.

**False admission (invalid transaction accepted):** When the previous block had low congestion, the next block's price is *lower* than the current block's price. A transaction with `tx_l2_gas_price` between `P_next` and `min_gas_price_percentage% × P_prev` passes the gateway threshold but will fail the blockifier's `check_fee_bounds` during batcher execution (`tx_price < P_next`). The transaction enters the mempool and wastes batcher resources.

Both paths match: **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The price shift per block is bounded (~1% per block under normal congestion swings, as shown by the snapshot tests). However:

- The condition is triggered by ordinary network activity (any block that is not exactly at the gas target).
- If `min_gas_price_percentage = 100` (the value used in the test suite), the false-rejection window equals the full price delta between consecutive blocks (~300 million fri at 30 Gwei), which is reachable by any user setting their gas price to the current block's price.
- No privileged access is required; any unprivileged user submitting a V3 `AllResources` transaction is affected.

---

### Recommendation

Replace the read of `previous_block_l2_gas_price` with the `next_l2_gas_price` field already stored in the committed block header. The `GatewayFixedBlockStateReader` trait should be extended to expose `next_l2_gas_price`, and `validate_resource_bounds` should use it as the reference price. This directly resolves the acknowledged TODO:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
``` [8](#0-7) 

The same fix should be applied to `run_validate_entry_point` so the blockifier validation inside the gateway also uses the correct next-block gas price.

---

### Proof of Concept

**False rejection scenario (valid transaction denied):**

1. Previous block gas used = 25% of max block size (low congestion).
2. Previous block L2 gas price `P = 30_000_000_000` fri.
3. `calculate_next_base_gas_price` produces `P' = 29_687_500_000` fri (~1% decrease, matching the `low_congestion` snapshot test).
4. `min_gas_price_percentage = 100`, so threshold = `100% × 30_000_000_000 = 30_000_000_000`.
5. User submits a V3 `AllResources` transaction with `l2_gas.max_price_per_unit = 29_800_000_000`.
6. Gateway check: `29_800_000_000 < 30_000_000_000` → **rejected** with `GAS_PRICE_TOO_LOW`.
7. Batcher execution check: `29_800_000_000 ≥ 29_687_500_000` → would **succeed**.

The user's transaction is denied service despite being economically valid for the next block.

**False admission scenario (invalid transaction accepted into mempool):**

1. Previous block gas used = 75% of max block size (high congestion).
2. Previous block L2 gas price `P = 30_000_000_000` fri.
3. `calculate_next_base_gas_price` produces `P' = 30_312_500_000` fri (~1% increase, matching the `high_congestion` snapshot test).
4. `min_gas_price_percentage = 100`, threshold = `30_000_000_000`.
5. User submits a V3 `AllResources` transaction with `l2_gas.max_price_per_unit = 30_100_000_000`.
6. Gateway check: `30_100_000_000 ≥ 30_000_000_000` → **admitted** into mempool.
7. Batcher execution check: `30_100_000_000 < 30_312_500_000` → **fails** `check_fee_bounds`, transaction rejected by blockifier.

    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
