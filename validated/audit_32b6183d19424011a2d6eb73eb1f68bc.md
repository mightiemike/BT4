### Title
Migration Aborts on Unbonding Validator — `ForceFullExitWithDelegation` Propagates `ErrValidatorNotBonded` from `transferDelegationFromPosition` - (`x/tieredrewards/keeper/force_exit.go`, `x/tieredrewards/keeper/transfer_delegation.go`)

---

### Summary

The v8 upgrade migration calls `ForceFullExitWithDelegation` for every vesting-account-owned position. That function unconditionally delegates through `transferDelegationFromPosition`, which hard-rejects any validator that is not in `Bonded` status. If any such position is delegated to a validator that has entered `Unbonding` or `Unbonded` status by the time the upgrade runs, the migration returns a non-nil error, halting the chain upgrade mid-loop with some positions already exited and others not.

---

### Finding Description

**Step 1 — Migration entry point**

`exitVestedAccountsPositions` iterates all positions owned by vesting accounts and calls `ForceFullExitWithDelegation` for each one. Any returned error is immediately propagated, aborting the entire migration: [1](#0-0) 

**Step 2 — `ForceFullExitWithDelegation` propagates the error**

`ForceFullExitWithDelegation` calls `transferDelegationFromPosition` and wraps/returns any error it receives: [2](#0-1) 

**Step 3 — `transferDelegationFromPosition` hard-rejects non-bonded validators**

At line 132, `transferDelegationFromPosition` checks `validator.IsBonded()` and returns `ErrValidatorNotBonded` if the validator is in `Unbonding` or `Unbonded` status: [3](#0-2) 

This guard is correct for normal user-facing operations (preventing new delegations to non-bonded validators), but it is fatal in the force-exit migration path, where the goal is to unconditionally return the delegation to the owner regardless of validator lifecycle state.

**Step 4 — Realistic precondition**

Validators enter `Unbonding` status through normal chain operation: a validator operator sends `MsgUndelegate` on their self-delegation, or the validator is jailed/tombstoned. This is a routine event. An upgrade window can easily overlap with a validator that started unbonding hours or days earlier. No attacker action is required — a single legitimately unbonding validator with a vesting-account delegator is sufficient.

---

### Impact Explanation

- The upgrade handler returns an error mid-loop.
- Some vesting-account positions have already been force-exited (delegation returned to owner, position record deleted).
- Remaining positions are still live in state with their delegation still held by the position delegator address.
- The chain halts at the upgrade height and cannot proceed until the migration code is patched and a new binary is deployed — a coordinated emergency upgrade is required.
- The inconsistency between exited and non-exited positions means some owners have their tokens back and others do not, violating the migration invariant.

---

### Likelihood Explanation

High. Validators unbond continuously on a live chain. The upgrade window is not instantaneous. Any vesting-account position delegated to a validator that entered `Unbonding` status before the upgrade block triggers the bug. No special attacker capability is needed; the condition arises from normal validator lifecycle events.

---

### Recommendation

In `ForceFullExitWithDelegation`, detect the non-bonded case and handle it differently: instead of calling `transferDelegationFromPosition` (which re-delegates to the owner), directly call `stakingKeeper.Unbond` on the position's delegator address and then send the resulting tokens to the owner's account, bypassing the `IsBonded()` guard that is only appropriate for normal delegation transfers. Alternatively, add a dedicated `forceTransferDelegationFromPosition` path that skips the `IsBonded()` check and handles the `Unbonding`/`Unbonded` validator case by creating a standard unbonding delegation entry in the owner's name.

---

### Proof of Concept

```
1. Create a vesting account V.
2. V opens a tieredrewards position P delegated to validator X.
3. Validator X begins unbonding (operator sends MsgUndelegate on self-delegation,
   or validator is jailed). X.Status == Unbonding.
4. Trigger the v8 upgrade.
5. Migrate() → exitVestedAccountsPositions() → ForceFullExitWithDelegation(P)
   → transferDelegationFromPosition(P, X, amount)
   → validator.IsBonded() == false
   → return ErrValidatorNotBonded
   → ForceFullExitWithDelegation returns error
   → exitVestedAccountsPositions returns error
   → Migrate() returns error
   → upgrade handler returns error
   → chain halts at upgrade height.
```
