### Title
Linked Signer Can Unilaterally Redirect Signing Authority to an Attacker-Controlled Address — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The fast-path `LinkSigner` transaction is validated with `allowLinkedSigner = true`, meaning the **current linked signer** — not just the subaccount owner — can sign and submit a `LinkSigner` transaction that overwrites `linkedSigners[subaccount]` with any address. A malicious linked signer can silently redirect signing authority to a second address they control, without the subaccount owner's knowledge or consent, and then use that second address to drain collateral.

---

### Finding Description

In `EndpointTx.sol`, the fast-path handler for `TransactionType.LinkSigner` calls `validateSignedTx` with `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ← allowLinkedSigner
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` passes the current linked signer to `validateCompactSignature` / `validateSignature`, which accepts either the subaccount owner **or** the linked signer as a valid signer:

```solidity
// Verifier.sol lines 306–319
address recovered = ECDSA.recover(digest, signature.r, signature.vs);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),   // ← linked signer accepted
    ERR_INVALID_SIGNATURE
);
``` [2](#0-1) 

Because `linkedSigners` is a flat mapping that is **overwritten unconditionally** with no check on the current value:

```solidity
// EndpointStorage.sol line 50
mapping(bytes32 => address) internal linkedSigners;
``` [3](#0-2) 

…the current linked signer can craft a valid `LinkSigner` transaction that sets `linkedSigners[victim_subaccount]` to any address they choose.

**Contrast with the slow-mode path**, which correctly restricts `LinkSigner` to the subaccount owner via `validateSender`:

```solidity
// EndpointTx.sol lines 232–239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // ← msg.sender must be owner
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The fast path has no equivalent owner-only guard.

---

### Impact Explanation

The `linkedSigners` mapping is the sole authorization gate for all fast-path signed transactions (`WithdrawCollateral`, `WithdrawCollateralV2`, `TransferQuote`, `LiquidateSubaccount`, `MintNlp`, `BurnNlp`, `TransferQuote`, etc.). [5](#0-4) 

Once a malicious linked signer redirects `linkedSigners[victim]` to an address they control, that address can sign any of those transactions on behalf of the victim. The concrete corrupted state delta is:

- **Before:** `linkedSigners[Alice] = Bob` (Alice authorized)
- **After (attacker-induced):** `linkedSigners[Alice] = Bob2` (Alice never authorized)
- **Consequence:** Bob2 can call `WithdrawCollateral` / `WithdrawCollateralV2` on Alice's subaccount, draining her collateral to an arbitrary `sendTo` address.

The `IEndpoint.LinkSigner` struct contains `sender`, `signer`, and `nonce` — there is no field that restricts who may sign the transaction, and no on-chain check that the signer of a `LinkSigner` message is the subaccount owner. [6](#0-5) 

---

### Likelihood Explanation

Linked signers are used for automated trading bots and NLP pool operators — a common, documented use case. Any user who has ever linked a signer is exposed. The attacker only needs to be the current linked signer (a role the victim explicitly granted). No admin access, no sequencer compromise, and no governance capture is required. The sequencer processes the malicious `LinkSigner` transaction as a normal fast-path transaction indistinguishable from a legitimate one. Likelihood is **medium** (requires a malicious or compromised linked signer, which is a realistic threat model for bots and third-party integrations).

---

### Recommendation

1. **Restrict `LinkSigner` signing to the subaccount owner only.** Change `allowLinkedSigner` to `false` for the fast-path `LinkSigner` handler, mirroring the slow-mode path's `validateSender` check:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only owner may change linked signer
);
```

2. **Alternatively**, add an explicit check that the recovered signer equals `address(uint160(bytes20(signedTx.tx.sender)))` before updating `linkedSigners`.

3. **Consider a two-step revocation pattern**: require the new `LinkSigner` to be signed by the subaccount owner, and add a time-lock or confirmation step before the change takes effect, so that a compromised linked signer cannot immediately redirect authority.

---

### Proof of Concept

1. Alice deposits collateral and links Bob as her signer: `linkedSigners[Alice_subaccount] = Bob`.
2. Bob constructs a `LinkSigner` transaction: `{ sender: Alice_subaccount, signer: Bob2, nonce: currentNonce }`.
3. Bob signs this transaction with his own key (Bob's address). The signature is valid because `allowLinkedSigner = true` and `Bob == linkedSigners[Alice_subaccount]`.
4. Bob submits the transaction to the sequencer. The sequencer processes it as a normal fast-path transaction.
5. `linkedSigners[Alice_subaccount]` is now `Bob2`. Alice was never notified and never consented.
6. Bob2 signs a `WithdrawCollateralV2` transaction: `{ sender: Alice_subaccount, productId: QUOTE, amount: fullBalance, sendTo: Bob_wallet, nonce: currentNonce+1 }`.
7. The sequencer processes it. `validateCompactSignature` accepts Bob2 as the linked signer. Alice's collateral is transferred to Bob's wallet.

Alice's entire collateral balance is extracted. The only on-chain evidence is two sequencer-submitted transactions that appear structurally valid.
