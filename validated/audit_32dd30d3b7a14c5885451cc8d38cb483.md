### Title
Deprecated `verify_transaction_inclusion` Remains Callable and Vulnerable to 64-Byte Merkle Proof Forgery - (`contract/src/lib.rs`)

### Summary

The `verify_transaction_inclusion` function is marked `#[deprecated]` since v0.5.0 but remains a live, callable public entry point on the NEAR contract. Unlike its replacement `verify_transaction_inclusion_v2`, it performs no coinbase Merkle proof validation, making it exploitable via the 64-byte transaction Merkle proof forgery attack. Any unprivileged NEAR caller can supply an internal Merkle tree node hash as `tx_id` and receive a `true` proof result for a transaction that was never included in the block.

---

### Finding Description

`verify_transaction_inclusion` is still exposed as a `#[pause]`-gated public method: [1](#0-0) 

Its own inline documentation warns of the exact attack surface: [2](#0-1) 

The function's verification logic reduces to a single Merkle root comparison with no coinbase anchor: [3](#0-2) 

The replacement `verify_transaction_inclusion_v2` was introduced specifically to close this gap by requiring a coinbase Merkle proof at index 0, which anchors the tree to a known-valid leaf: [4](#0-3) 

Because Bitcoin's Merkle tree does not distinguish between leaf nodes (transactions) and internal nodes at the hash level, an attacker can treat any 64-byte internal node as a `tx_id`. A valid Merkle path from that internal node to the block's `merkle_root` exists by construction, so `compute_root_from_merkle_proof(tx_id, tx_index, proof) == header.merkle_root` evaluates to `true` without any real transaction being present.

The `merkle_proof.is_empty()` guard does not prevent this: [5](#0-4) 

---

### Impact Explanation

Any dApp or bridge contract that calls `verify_transaction_inclusion` to gate fund releases, cross-chain messages, or state transitions can be deceived into accepting a forged proof. The attacker receives a `true` return value for a transaction that does not exist, enabling theft or unauthorized state changes in the consuming protocol. The corrupted value is the **proof result** itself — the contract's primary security guarantee.

---

### Likelihood Explanation

The function is callable by any unprivileged NEAR account with no special role or stake required. The 64-byte forgery technique is publicly documented (https://www.bitmex.com/blog/64-Byte-Transactions) and requires only knowledge of a target block's Merkle tree structure, which is public on-chain Bitcoin data. The deprecated function is not removed or access-controlled, so any integrator that has not migrated to `v2` is immediately exposed.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or gate it behind a role (e.g., `Role::DAO`) so it cannot be called by unprivileged accounts. Do not rely on the Rust `#[deprecated]` attribute as a security control — it produces a compiler warning for the contract authors but has no effect on external callers. All callers must be migrated to `verify_transaction_inclusion_v2`.

---

### Proof of Concept

1. Identify any confirmed Bitcoin block stored in the contract's `headers_pool`.
2. Obtain the block's full Merkle tree. Select any internal node `N` at depth `d` whose serialized form is 64 bytes (or construct a synthetic one).
3. Compute the Merkle proof path from `N` up to `merkle_root` — this path is valid by the tree's own construction.
4. Call `verify_transaction_inclusion` with:
   - `tx_id = H256(N)`
   - `tx_block_blockhash` = the target block hash
   - `tx_index` = the index corresponding to `N`'s position at depth `d`
   - `merkle_proof` = the path from `N` to `merkle_root`
   - `confirmations = 1`
5. The function returns `true`. No such transaction exists in the block. [6](#0-5)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
