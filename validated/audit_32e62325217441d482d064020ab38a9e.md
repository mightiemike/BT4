### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Skipping Coinbase Merkle Proof Validation Required by v2 — (File: `contract/src/lib.rs`)

---

### Summary

The contract exposes two public entry points for SPV proof verification. The secure path (`verify_transaction_inclusion_v2`) validates a coinbase merkle proof before checking the target transaction. The insecure path (`verify_transaction_inclusion` v1) skips that validation entirely and remains fully callable by any unprivileged NEAR account. An attacker can route directly through v1 to obtain a `true` verification result for a forged 64-byte transaction inclusion proof, bypassing the only guard that prevents internal Merkle-tree nodes from being accepted as valid transaction hashes.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability (documented at https://www.bitmex.com/blog/64-Byte-Transactions). It enforces two invariants before delegating to v1:

1. `merkle_proof.len() == coinbase_merkle_proof.len()` — proof depth parity.
2. The coinbase transaction's merkle proof must reconstruct the block's `merkle_root` — proving the proof depth is correct and the target `tx_id` is a leaf, not an internal node. [1](#0-0) 

`verify_transaction_inclusion` (v1) performs none of these checks. It only requires a non-empty `merkle_proof` and then directly computes `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == header.merkle_root`. [2](#0-1) 

The `#[deprecated]` attribute is a Rust compiler hint only — it emits a warning to Rust callers at compile time but imposes **no runtime restriction**. The function is `pub`, carries only `#[pause]` (no role-based access control), and is therefore callable by any NEAR account at any time. [3](#0-2) 

The `compute_root_from_merkle_proof` function in `merkle-tools` is a pure hash-chain computation with no awareness of whether the starting hash is a leaf or an internal node: [4](#0-3) 

This is structurally identical to the reported Swivel bug: two callable paths exist for the same operation, and the alternative path (v1 / the `if(e)` branch) omits a required security step (coinbase validation / fee extraction).

---

### Impact Explanation

Any NEAR contract (bridge, atomic swap, cross-chain lending protocol) that consumes `verify_transaction_inclusion` — or that an attacker can cause to call it — will accept a forged proof as valid. Concretely:

- An attacker identifies an internal Merkle-tree node `N = SHA256d(L || R)` in any confirmed Bitcoin block.
- They construct a 64-byte "transaction" whose bytes are exactly `L || R`; its double-SHA256 equals `N`.
- They call `verify_transaction_inclusion` with `tx_id = N`, a valid `tx_index` for that internal node's position, and the sibling hashes above it as `merkle_proof`.
- The function returns `true` because the computed root matches `header.merkle_root`.
- A consumer contract is deceived into believing a non-existent transaction was confirmed, potentially releasing funds or minting tokens.

The corrupted proof result is: `verify_transaction_inclusion` returns `true` for a `tx_id` that is not a valid transaction hash.

---

### Likelihood Explanation

The attack requires no privileged role, no leaked key, and no social engineering. Any NEAR account can call `verify_transaction_inclusion` directly. The 64-byte forgery technique is publicly documented and well understood. The only prerequisite is knowledge of the internal Merkle-tree structure of any confirmed Bitcoin block, which is fully public data. Likelihood is **high** for any deployment where a consumer contract does not independently enforce use of v2.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or restrict it with an access-control role so it cannot be called by unprivileged accounts. All external callers must be forced through `verify_transaction_inclusion_v2`. Alternatively, inline the coinbase-proof check directly into v1 and remove the separate v2 function, eliminating the two-path divergence.

---

### Proof of Concept

1. Deploy the contract on NEAR testnet with `skip_pow_verification = true` and a known set of headers.
2. Select any confirmed block in the contract's mainchain. Obtain its full transaction list and compute the Merkle tree to identify an internal node `N` at depth `d` with sibling path `[s_1, …, s_d]`.
3. Construct `ProofArgs { tx_id: N, tx_block_blockhash: <that block>, tx_index: <index of N at depth d>, merkle_proof: [s_1, …, s_d], confirmations: 1 }`.
4. Call `verify_transaction_inclusion` with this `ProofArgs` (Borsh-serialized).
5. Observe the function returns `true`, despite `N` not being a real transaction hash.
6. Call `verify_transaction_inclusion_v2` with the same `tx_id` and any `coinbase_merkle_proof` of length `d`; observe it panics with `"Incorrect coinbase merkle proof"` — confirming v2 blocks the attack while v1 does not. [5](#0-4) [6](#0-5)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
