### Title
Missing `WithdrawCollateralV2` Handler in `processSlowModeTransactionImpl` Permanently Locks User Withdrawals — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

`processSlowModeTransactionImpl` handles `WithdrawCollateral` (V1) but has no branch for `WithdrawCollateralV2`. Because `submitSlowModeTransactionImpl` does not block `WithdrawCollateralV2` submissions, a user can pay the slow-mode fee and queue a V2 withdrawal. When the sequencer later calls the process function, execution falls through to the unconditional `else { revert(); }`, permanently preventing the withdrawal from completing and locking the user's collateral.

---

### Finding Description

`submitSlowModeTransactionImpl` accepts any transaction type that is not explicitly blocked or owner-gated. `DepositCollateral` is explicitly reverted at line 344. A set of admin-only types requires `sender == owner()` at lines 356–368. Everything else — including `WithdrawCollateralV2` — falls into the `else` branch, which charges the 1 USDC slow-mode fee and enqueues the transaction: [1](#0-0) 

When the sequencer later calls `processSlowModeTransactionImpl` for that queued entry, the function dispatches on `txType`. `WithdrawCollateral` (V1) is handled at lines 217–229: [2](#0-1) 

`WithdrawCollateralV2` has no corresponding branch. The function reaches the terminal `else { revert(); }` at line 328: [3](#0-2) 

By contrast, `processTransactionImpl` (the fast path) fully handles `WithdrawCollateralV2` at lines 437–465: [4](#0-3) 

The missing slow-mode branch is the direct analog of the reference bug: a function that is valid and fully implemented in one code path (`processTransactionImpl`) is silently absent in a parallel path (`processSlowModeTransactionImpl`), causing an unconditional revert when reached.

---

### Impact Explanation

The slow-mode queue is the protocol's censorship-resistance fallback. A user who is being censored by the sequencer and submits a `WithdrawCollateralV2` slow-mode transaction will:

1. Lose the 1 USDC slow-mode fee (charged at submission time, non-refundable).
2. Have their withdrawal permanently stuck — the sequencer's processing call reverts every time it reaches that entry.
3. Because `_delegatecallEndpointTx` propagates all reverts to the caller (lines 74–83), the sequencer's outer transaction fails, which can stall processing of all subsequent slow-mode entries queued after the stuck one. [5](#0-4) 

The concrete corrupted state: the user's collateral balance remains locked inside the protocol with no on-chain path to recover it via slow mode. The `WithdrawCollateralV2` format is the only way to direct a withdrawal to an address other than the subaccount owner (`sendTo` field), so affected users also lose the ability to route funds to a different recipient.

---

### Likelihood Explanation

`WithdrawCollateralV2` is the current, actively-used withdrawal transaction type (it supersedes V1 by adding the `sendTo` parameter). Any user or integration that constructs a slow-mode withdrawal using the V2 struct — a realistic action when the sequencer is unresponsive — triggers this path. No special privileges, no admin access, and no unusual conditions are required. The attacker-controlled entry point is the public `submitSlowModeTransaction` function on `Endpoint`, callable by any non-sanctioned address.

---

### Recommendation

Add a `WithdrawCollateralV2` branch to `processSlowModeTransactionImpl` mirroring the existing `WithdrawCollateral` branch, decoding the V2 struct and calling `clearinghouse.withdrawCollateral` with the resolved `sendTo` address (falling back to the sender's address when `sendTo == address(0)`, consistent with how `BaseWithdrawPool.resolveFastWithdrawal` handles V2): [6](#0-5) 

---

### Proof of Concept

1. User calls `Endpoint.submitSlowModeTransaction` with a `WithdrawCollateralV2` payload.
2. `submitSlowModeTransactionImpl` does not match any explicit branch for `WithdrawCollateralV2`; the `else` branch charges 1 USDC and enqueues the transaction with `executableAt = block.timestamp + 3 days`.
3. After 3 days, the sequencer calls `Endpoint.processSlowModeTransaction` for that entry.
4. `processSlowModeTransactionImpl` evaluates `txType == WithdrawCollateralV2` — no branch matches — and executes `revert()` at line 328.
5. `_delegatecallEndpointTx` propagates the revert; the sequencer's transaction reverts.
6. The slow-mode entry remains unprocessed indefinitely; the user's collateral is locked with no on-chain remedy.
