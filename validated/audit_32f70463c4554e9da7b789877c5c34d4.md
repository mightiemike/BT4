### Title
Stale Mainchain Block Timestamp Used for Fork Difficulty Calculation - (`contract/src/dogecoin.rs`)

### Summary

In the Dogecoin difficulty adjustment logic, `get_header_by_height` always returns the **mainchain** block at a given height. When validating a fork block that is 2+ blocks deep into a fork, the correct ancestor at `height_first` is a fork block — not the mainchain block. Using the mainchain block's timestamp as `first_block_time` produces an incorrect `modulated_timespan`, which corrupts the `expected_bits` check and can allow fork blocks with wrong difficulty to be accepted.

### Finding Description

In `contract/src/dogecoin.rs`, `get_next_work_required` computes the difficulty for the next block by measuring the elapsed time between the previous block and the block at `height_first`:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

`blocks_getter.get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [2](#0-1) 

This always resolves through `mainchain_height_to_header`, returning the **mainchain** block at that height regardless of whether the block being validated is on a fork.

For Dogecoin, `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` for all non-genesis blocks:

```rust
let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
    blocks_to_go_back = difficulty_adjustment_interval;
}
``` [3](#0-2) 

This means `height_first = prev_block_header.block_height - 1`. For a fork block F(H+2) whose parent is F(H+1) and grandparent is F(H) (the fork divergence point), `height_first = H`. The code fetches the **mainchain** block M(H) instead of the fork's actual ancestor F(H). Since F(H) and M(H) are different blocks with different timestamps, `first_block_time` is wrong.

The incorrect `first_block_time` flows directly into `calculate_next_work_required`:

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
``` [4](#0-3) 

This produces a wrong `modulated_timespan`, which yields a wrong `new_target`, which yields a wrong `expected_bits`. The `check_pow` function then enforces that the submitted block's `bits` field equals this wrong `expected_bits`:

```rust
require!(
    expected_bits == block_header.bits,
    ...
);
``` [5](#0-4) 

The developers themselves flagged this with a `TODO` comment at the exact line, confirming awareness of the unresolved correctness concern.

The same structural bug exists in `bitcoin.rs` and `litecoin.rs` at the equivalent `get_header_by_height` call: [6](#0-5) [7](#0-6) 

For Bitcoin/Litecoin the 2016-block interval makes exploitation extremely unlikely in practice. For Dogecoin the 1-block interval makes it structurally guaranteed for any fork 2+ blocks deep.

### Impact Explanation

An attacker who submits a Dogecoin fork chain at least 2 blocks deep can craft fork ancestor timestamps that differ from the mainchain block timestamps at the same heights. The difficulty calculation for the fork's third-and-beyond blocks will use the mainchain's stale timestamp as `first_block_time` instead of the fork's actual ancestor's timestamp. If the mainchain block at `height_first` has an older timestamp than the fork's ancestor at that height, `modulated_timespan` is inflated, the target is raised (difficulty lowered), and the attacker can mine fork blocks with less PoW than the protocol requires. This corrupts the canonical-chain promotion logic: a fork with less total chainwork than required could be promoted to mainchain via `reorg_chain`, causing the contract to report incorrect block hashes and heights to downstream consumers of `verify_transaction_inclusion` and `get_block_hash_by_height`.

### Likelihood Explanation

The entry point is `submit_blocks`, which is callable by any trusted relayer — an unprivileged role that any staked participant can hold. The attacker only needs to submit a 3-block fork sequence. The Dogecoin 1-block difficulty interval guarantees the stale-value path is exercised on every fork block beyond the first. The timestamp manipulation is constrained by MTP and the 2-hour future-time cap, and the difficulty shift is further dampened by the `(modulated_timespan - retarget_timespan) / 8` smoothing and the `[45s, 90s]` clamp, so the per-block difficulty deviation is bounded. However, the deviation compounds across a multi-block fork, and the attacker controls how many fork blocks they submit.

### Recommendation

Replace the `get_header_by_height` lookup with an ancestor traversal that walks the `prev_block_hash` chain backwards from `prev_block_header` until reaching `height_first`. This mirrors the correct behavior in Bitcoin Core's `GetAncestor`. The fix applies to `dogecoin.rs`, `bitcoin.rs`, and `litecoin.rs`.

### Proof of Concept

1. Mainchain contains blocks M(H-1), M(H), M(H+1). M(H).time = T.
2. Attacker submits fork block F(H) with `prev_block_hash = M(H-1).hash` and `F(H).time = T + 55` (within MTP and future-time bounds).
3. Attacker submits fork block F(H+1) with `prev_block_hash = F(H).hash`.
4. Attacker submits fork block F(H+2) with `prev_block_hash = F(H+1).hash`.
5. During validation of F(H+2): `height_first = H`, `get_header_by_height(H)` returns M(H) with `time = T`.
6. `modulated_timespan = F(H+1).time - T`. Correct value would be `F(H+1).time - (T+55)`, which is 55 seconds smaller.
7. The inflated `modulated_timespan` produces a higher target (easier difficulty) for F(H+2).
8. Attacker mines F(H+2) against this easier target and submits it; the contract accepts it because `expected_bits` matches the attacker's crafted `bits` field.
9. If F(H+2).chain_work exceeds the mainchain tip's chain_work, `reorg_chain` promotes the fork, corrupting the canonical chain state.
