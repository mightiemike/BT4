### Title
Linked Signer Can Overwrite Its Own Authorization — Unauthorized Subaccount Signer Hijacking - (`File: core/contracts/EndpointTx.sol`)

---

### Summary

`processTransactionImpl` in `EndpointTx.sol` processes `LinkSigner` transactions with `allowLinkedSigner = true`. This means the **current linked signer** — not just the subaccount owner — can sign and submit a `LinkSigner` transaction that replaces the linked signer with any arbitrary address. The linked signer (a delegated key) can therefore permanently transfer signing authority over a victim's subaccount to an attacker-controlled address, after which the attacker can drain collateral via `WithdrawCollateral`.

---

### Finding Description

The `LinkSigner` transaction type is intended to let a subaccount owner delegate signing authority to a secondary address. The critical flaw is in how the sequencer path validates this transaction type.

In `processTransactionImpl`:

```solidity
// EndpointTx.sol:576-590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:], (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true   // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
```

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which passes `getLinkedSigner(sender)` as the accepted signer:

```solidity
// EndpointTx.sol:172-184
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
```

`Verifier.validateSignature` then accepts a signature from **either** the subaccount owner **or** the linked signer:

```solidity
// Verifier.sol:297-303
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
    ((recovered == address(uint160(bytes20(sender)))) ||
     (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
```

This means the current `linkedSigner` can sign a `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with any address it chooses — including an attacker-controlled address. The `linkedSigner` is decoupled from the subaccount owner and can mutate the very mapping that grants it authority.

**Contrast with the slow mode path**, which correctly restricts `LinkSigner` to the subaccount owner only:

```solidity
// EndpointTx.sol:232-239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);  // msg.sender must == subaccount owner address
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
}
```

The slow mode path uses `validateSender`, which enforces `msg.sender == address(uint160(bytes20(txn.sender)))` — only the actual owner address can submit. The sequencer path has no such restriction.

---

### Impact Explanation

Once the attacker controls the `linkedSigner` slot for a victim subaccount, they can sign any sequencer-path transaction on behalf of that subaccount with `allowLinkedSigner = true`, including:

- `WithdrawCollateral` (line 418–436) — drain all spot collateral
- `WithdrawCollateralV2` (line 442–465) — drain to an arbitrary `sendTo` address
- `TransferQuote` (line 599–614) — transfer quote balance to attacker's subaccount
- `LiquidateSubaccount` (line 397–412) — liquidate the victim's positions at a loss

The corrupted state is `linkedSigners[victim_subaccount]` → `attacker_address`, which directly enables collateral theft from the victim's subaccount.

---

### Likelihood Explanation

The attack requires the attacker to first be the linked signer for a target subaccount. This is a realistic precondition: linked signers are commonly used by trading bots, API keys, and third-party integrators. Any such party — once granted linked signer status — can immediately execute this attack without any further cooperation from the victim. The sequencer processes the malicious `LinkSigner` transaction identically to a legitimate one; there is no on-chain guard that distinguishes owner-signed from linked-signer-signed `LinkSigner` transactions in the sequencer path.

---

### Recommendation

Change the `allowLinkedSigner` flag to `false` for `LinkSigner` transaction processing in `processTransactionImpl`. Only the subaccount owner (the address embedded in `signedTx.tx.sender`) should be permitted to authorize a change to the linked signer mapping. The slow mode path already enforces this correctly via `validateSender`.

```solidity
// EndpointTx.sol:581-587 — change true to false
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only subaccount owner may change linked signer
);
```

---

### Proof of Concept

1. Alice owns subaccount `alice_sub` and grants Bob (`bob_addr`) as linked signer: `linkedSigners[alice_sub] = bob_addr`.
2. Bob constructs a `SignedLinkSigner` transaction: `{ sender: alice_sub, signer: attacker_addr, nonce: current_nonce }`, signed with Bob's private key.
3. Bob (or anyone) submits this to the sequencer, which calls `processTransactionImpl`.
4. `validateSignedTx(..., true)` resolves `getLinkedSigner(alice_sub)` → `bob_addr`, and `ECDSA.recover` returns `bob_addr`, satisfying `Verifier.validateSignature`.
5. `linkedSigners[alice_sub]` is overwritten with `attacker_addr`.
6. Attacker constructs a `SignedWithdrawCollateral` transaction: `{ sender: alice_sub, productId: USDC, amount: full_balance, nonce: next_nonce }`, signed with `attacker_addr`'s private key.
7. `validateSignedTx(..., true)` resolves `getLinkedSigner(alice_sub)` → `attacker_addr`, signature validates, and `clearinghouse.withdrawCollateral` transfers Alice's full collateral balance to the attacker.

**Exact corrupted state:** `linkedSigners[alice_sub]` → `attacker_addr` (line 588–590), followed by `clearinghouse.withdrawCollateral` draining Alice's balance (line 430–436). [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
