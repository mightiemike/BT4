### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables usdcE Drain from Direct Deposit Accounts — (`File: core/contracts/ContractOwner.sol`)

---

### Summary
`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(...)` directly without using `safeTransferFrom` and without checking the boolean return value. If the USDC token at the hardcoded address returns `false` on failure instead of reverting, the function continues executing, withdrawing usdcE from the Direct Deposit Account (DDA) and sending it to the caller — without the caller having paid any USDC.

---

### Finding Description
In `ContractOwner.replaceUsdcEWithUsdc`, the function is intended to atomically swap usdcE held in a DDA for USDC provided by the caller. The critical step is:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
``` [1](#0-0) 

This is a raw `transferFrom` call — the return value is never checked and `safeTransferFrom` (defined in `ERC20Helper`) is not used. The rest of the protocol consistently uses `safeTransferFrom` via `ERC20Helper`: [2](#0-1) [3](#0-2) 

If the `transferFrom` call returns `false` (silent failure), execution continues:

1. `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))` — transfers all usdcE from the DDA to `ContractOwner` (via `safeTransfer` to `msg.sender` of the `withdraw` call, which is `ContractOwner`).
2. `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` — sends the full usdcE balance to the attacker. [4](#0-3) 

The attacker receives `balance` usdcE without having transferred any USDC into the DDA.

---

### Impact Explanation
Any usdcE held in a DDA on chain 57073 (Ink) can be drained by an unprivileged caller. The function has no `onlyOwner` modifier — only a `block.chainid == 57073` guard. The corrupted asset delta is the full usdcE balance of the targeted DDA, transferred to the attacker at zero cost.

---

### Likelihood Explanation
The function is permissionlessly callable by any address on chain 57073. The trigger requires the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` to return `false` rather than revert on a failed transfer (e.g., insufficient allowance). Standard USDC implementations revert, which reduces immediate exploitability. However, the code pattern is definitively wrong: the rest of the codebase uniformly uses `safeTransferFrom`, and this single deviation creates a latent vulnerability that activates if the token contract behaves non-standardly (e.g., after an upgrade, or if the address resolves to a non-standard ERC20 on Ink).

---

### Recommendation
Replace the raw `transferFrom` call with `safeTransferFrom` from `ERC20Helper`, consistent with every other transfer in the protocol:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [2](#0-1) 

---

### Proof of Concept
1. A DDA for `subaccount` holds `N` usdcE on chain 57073.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` with zero USDC allowance granted to `ContractOwner`.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, N)` returns `false` (no revert).
4. Return value is ignored; execution continues.
5. `DirectDepositV1(directDepositV1).withdraw(usdcE)` moves `N` usdcE to `ContractOwner`.
6. `IERC20Base(usdcE).safeTransfer(attacker, N)` sends `N` usdcE to the attacker.
7. Attacker receives `N` usdcE; DDA is emptied; no USDC was paid. [4](#0-3)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
