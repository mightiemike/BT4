### Title
`ClvmFlags::LIMIT_HEAP` Is Stored in `ChiaDialect` but Never Enforced by the Core Rust Engine - (File: `src/chia_dialect.rs`)

### Summary

`ClvmFlags::LIMIT_HEAP` is documented as a flag that "limits the number of atom-bytes allowed to be allocated, as well as the number of pairs." It is included in the `MEMPOOL_MODE` constant used for stricter mempool validation. However, the core Rust engine (`src/run_program.rs`, all operator files) never reads this flag to enforce any heap limit. The only enforcement exists in the Python wheel's `run_serialized_chia_program` function. Any Rust caller using `MEMPOOL_MODE` or `LIMIT_HEAP` directly via the `clvmr` crate receives an unlimited 4 GiB allocator, not the intended 500 MB cap.

### Finding Description

`ClvmFlags::LIMIT_HEAP` is defined in `src/chia_dialect.rs` and included in `MEMPOOL_MODE`: [1](#0-0) [2](#0-1) 

The flag is stored in `ChiaDialect.flags` and propagated through `Dialect::flags()`: [3](#0-2) [4](#0-3) 

However, a grep of all `src/**/*.rs` files for `LIMIT_HEAP` returns **only the two definition/composition sites** in `src/chia_dialect.rs`. The flag is never read inside `src/run_program.rs`, `src/allocator.rs`, or any operator file to actually restrict allocator capacity.

The only place the flag is acted upon is in the Python wheel: [5](#0-4) 

When `LIMIT_HEAP` is set, `wheel/src/api.rs` creates `Allocator::new_limited(500_000_000)` (500 MB cap). When it is not set, `Allocator::new()` is used, which sets `heap_limit = u32::MAX` (~4 GiB): [6](#0-5) 

Any Rust caller that constructs a `ChiaDialect::new(MEMPOOL_MODE)` and calls `run_program` directly gets an `Allocator::new()` (4 GiB) regardless of the `LIMIT_HEAP` flag, because the flag is never read by the engine to enforce a tighter limit.

### Impact Explanation

A Rust consumer of the `clvmr` crate who passes `MEMPOOL_MODE` or `ClvmFlags::LIMIT_HEAP` to `ChiaDialect::new()` and then calls `run_program` directly expects heap allocation to be capped at ~500 MB (matching the Python wheel's behavior). Instead, the allocator is unlimited up to 4 GiB. An attacker-controlled CLVM program can allocate up to 8× more heap than the flag implies, causing memory exhaustion or consensus divergence between Rust-native and Python-wheel callers running the same program under the same flags.

### Likelihood Explanation

Any downstream Rust crate that uses `MEMPOOL_MODE` or `LIMIT_HEAP` directly (e.g., `chia_rs`, block generators, mempool validators) is affected. The flag is exported as a public constant and its documentation explicitly promises heap limiting behavior. The broken invariant is invisible at the call site.

### Recommendation

Move the heap-limit enforcement into the core engine. In `run_program` (or in a factory function for `RunProgramContext`), check `dialect.flags().contains(ClvmFlags::LIMIT_HEAP)` and create the `Allocator` with `Allocator::new_limited(500_000_000)` accordingly, or require callers to pass a pre-configured allocator and document that `LIMIT_HEAP` alone is insufficient without a matching `Allocator::new_limited(...)`.

### Proof of Concept

```rust
use clvmr::allocator::Allocator;
use clvmr::chia_dialect::{ChiaDialect, MEMPOOL_MODE};
use clvmr::run_program::run_program;
use clvmr::serde::node_from_bytes;

fn main() {
    // MEMPOOL_MODE includes LIMIT_HEAP — caller expects 500 MB cap
    let dialect = ChiaDialect::new(MEMPOOL_MODE);

    // But the Rust API always creates an unlimited allocator:
    let mut allocator = Allocator::new(); // heap_limit = u32::MAX (~4 GiB)

    // A crafted program that allocates large atoms in a loop can consume
    // far more than 500 MB before hitting the u32::MAX allocator limit,
    // violating the invariant promised by LIMIT_HEAP / MEMPOOL_MODE.
    let program = node_from_bytes(&mut allocator, &[/* large concat program */]).unwrap();
    let args = allocator.nil();
    let _ = run_program(&mut allocator, &dialect, program, args, u64::MAX);
    // allocator.heap_size() can reach ~4 GiB; LIMIT_HEAP had no effect.
}
```
