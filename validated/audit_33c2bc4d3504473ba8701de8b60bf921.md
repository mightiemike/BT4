### Title
Unexported Hard-Fork `ClvmFlags` Bits Accepted Without Validation via `run_serialized_chia_program` тАФ (`wheel/src/api.rs`)

---

### Summary

`run_serialized_chia_program` accepts a raw `u32` flags parameter from Python callers and converts it with `ClvmFlags::from_bits_truncate`, which silently accepts every known flag bit тАФ including hard-fork flags (`RELAXED_BLS`, `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`, `MALACHITE`) that are **intentionally not exported** to Python callers. A caller that supplies these bits enables hard-fork operators prematurely, causing the node to evaluate programs differently from the rest of the network and producing consensus divergence.

---

### Finding Description

`run_serialized_chia_program` in `wheel/src/api.rs` is the primary Python entry point for CLVM execution:

```rust
pub fn run_serialized_chia_program(
    py: Python,
    program: &[u8],
    args: &[u8],
    max_cost: Cost,
    flags: u32,          // тЖР raw caller-controlled integer
) -> PyResult<(u64, LazyNode)> {
    let flags = ClvmFlags::from_bits_truncate(flags);   // тЖР silently accepts all known bits
``` [1](#0-0) 

`from_bits_truncate` drops bits that correspond to no defined flag, but **accepts every bit that maps to a known `ClvmFlags` variant**, regardless of whether that variant was ever exposed to the caller.

The module exports only a curated subset of flags to Python:

```rust
m.add("NO_UNKNOWN_OPS",    ClvmFlags::NO_UNKNOWN_OPS.bits())?;
m.add("LIMIT_HEAP",        ClvmFlags::LIMIT_HEAP.bits())?;
m.add("MEMPOOL_MODE",      MEMPOOL_MODE.bits())?;
m.add("ENABLE_SHA256_TREE",ClvmFlags::ENABLE_SHA256_TREE.bits())?;
m.add("ENABLE_SECP_OPS",   ClvmFlags::ENABLE_SECP_OPS.bits())?;
m.add("DISABLE_OP",        ClvmFlags::DISABLE_OP.bits())?;
m.add("CANONICAL_INTS",    ClvmFlags::CANONICAL_INTS.bits())?;
``` [2](#0-1) 

Three hard-fork flags are **defined** in `ClvmFlags` but **never exported**:

| Flag | Bits | Effect |
|---|---|---|
| `RELAXED_BLS` | `0x0008` | `bls_g1_negate` / `bls_g2_negate` accept invalid-length points |
| `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` | `0x0100` | op 62 (`keccak256`) becomes available globally, outside the softfork guard |
| `MALACHITE` | `0x1000` | Switches `div`/`divmod`/`mod`/`modpow` to the malachite-bigint backend | [3](#0-2) 

`RELAXED_BLS` carries an explicit comment: *"Hard-fork; enable only when it activates."* `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` is the internal flag set by the softfork-guard machinery when extension 1 (Keccak) is active тАФ it is not meant to be set globally by callers. [4](#0-3) 

In the operator dispatch, these flags gate real execution paths:

```rust
62 if flags.contains(ClvmFlags::ENABLE_KECCAK_OPS_OUTSIDE_GUARD) => op_keccak256,
``` [5](#0-4) 

A Python caller that passes `flags=0x0100` makes keccak256 available to every program, not just those inside a softfork guard. A caller that passes `flags=0x0008` enables `RELAXED_BLS` before the hard-fork activates. Neither is rejected; neither produces a warning.

---

### Impact Explanation

**Consensus divergence.** A Chia full node that evaluates a spend bundle with an unexported hard-fork flag enabled will accept programs that every other node rejects (or vice versa). Concretely:

- With `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` set, a program invoking op 62 outside a softfork guard succeeds on the misconfigured node and fails on all others тЖТ the node forks off the canonical chain.
- With `RELAXED_BLS` set before the hard-fork height, a program passing an invalid-length point to `bls_g1_negate` succeeds locally and fails everywhere else тЖТ same fork.
- With `MALACHITE` set, arithmetic edge cases (large-integer division, modpow) may produce different byte-level results тЖТ silent consensus split.

---

### Likelihood Explanation

**Medium.** The `flags` parameter is a raw `u32` with no allowlist check. Any Python code path that derives the flags value from user-controlled data (configuration, RPC input, plugin API) is a direct trigger. The chia-blockchain node software constructs these flags; if any integration layer forwards an attacker-supplied integer as the flags argument, the unexported bits are silently honoured. The bug requires no special CLVM program тАФ the program itself can be benign; only the flags need to be wrong.

---

### Recommendation

1. **Allowlist validation**: After `from_bits_truncate`, assert that the result contains only the bits that are exported to Python callers. Reject (return `PyErr`) if any unexported bit is set.
2. **Typed API**: Replace the raw `u32` parameter with a validated newtype or an explicit set of boolean keyword arguments, so callers cannot accidentally or maliciously supply hard-fork bits.
3. **Do not expose `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` as a caller flag at all**: it is an internal flag set by the softfork-guard machinery and should never be accepted from outside.

---

### Proof of Concept

```python
from clvm_rs import run_serialized_chia_program

# ENABLE_KECCAK_OPS_OUTSIDE_GUARD = 0x0100 тАФ not exported, but accepted
# A program that calls keccak256 (op 62) directly, outside any softfork guard.
# Serialised CLVM: (keccak256 (q . b"hello"))
program = bytes.fromhex("ff3eff0568656c6c6f80")
args    = bytes.fromhex("80")

# On a correctly configured node this raises "unknown op" (op 62 not available).
# With the unexported flag it succeeds and returns the keccak256 digest.
cost, result = run_serialized_chia_program(program, args, 10_000_000, 0x0100)
# result is the keccak256 of b"hello" тАФ accepted only on this node,
# rejected by every peer тЖТ consensus divergence.
```

The root cause is `from_bits_truncate` at `wheel/src/api.rs:47` accepting `0x0100` without checking it against the exported flag set at lines 317тАУ323. The `ChiaDialect` constructed at line 57 then gates op 62 on the flag at `src/chia_dialect.rs:246`, making keccak256 globally reachable. [6](#0-5) [7](#0-6)
