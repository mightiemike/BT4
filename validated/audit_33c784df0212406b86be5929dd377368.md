### Title
Stale `ShardedRpcNode.tracked_shards` After Shard Layout Change Causes Permanent `UNAVAILABLE_SHARD` Errors - (File: `chain/jsonrpc/src/sharded_rpc.rs`)

---

### Summary

`ShardedRpcNode.tracked_shards` is a `Vec<ShardId>` populated once at pool construction from the static config file and never updated when the shard layout changes via dynamic resharding. After a shard split, child shard IDs are absent from every remote node's `tracked_shards`, so the pool's routing logic permanently fails to find any node for queries targeting child shards, returning `UNAVAILABLE_SHARD` to all callers.

---

### Finding Description

`ShardedRpcNode` stores the set of shards a remote pool node tracks as a plain `Vec<ShardId>`: [1](#0-0) 

This vector is populated exactly once, at `ShardedRpcPool::new`, by cloning the static config value: [2](#0-1) 

There is no mechanism to update `tracked_shards` when the epoch manager's shard layout changes. After dynamic resharding splits a parent shard into two child shards, the child `ShardId` values are new identifiers that do not appear in any node's `tracked_shards` list.

Every routing path in the pool checks remote nodes using `node.tracked_shards.contains(&shard_id)`:

- `nodes_for_account_in_epochs` (account-based routing): [3](#0-2) 

- `nodes_for_shard_in_epochs` (shard-id-based routing), with the comment explicitly acknowledging the static nature: [4](#0-3) 

- `pick_node_for_shard` (scatter-gather routing): [5](#0-4) 

Because child shard IDs are never inserted into any `ShardedRpcNode.tracked_shards`, all three paths return an empty candidate list for any child shard, causing the pool to fall through to the emergency local-node fallback or return an error.

The codebase itself acknowledges this defect with an `#[ignore]`-marked integration test and an explicit TODO: [6](#0-5) 

The test harness configures `rpc0` with `TrackedShardsConfig::Shards(vec![rpc0_shard])` and `rpc1` with `rpc1_shard`, then triggers a dynamic shard split. After resharding, queries for accounts in child shards fail with `UNAVAILABLE_SHARD` because neither node's static `tracked_shards` contains the new child shard IDs: [7](#0-6) 

---

### Impact Explanation

After any dynamic resharding event, the `ShardedRpcPool` routing table becomes permanently stale. All JSON-RPC queries (`query`, `view_account`, `view_state`, etc.) that target accounts or shard IDs belonging to child shards will fail to be routed to any capable remote node. The local node's `ShardTracker` correctly resolves child shards via `check_if_descendant_of_tracked_shard`, but remote nodes are checked only against the static `tracked_shards` list. The result is systematic `UNAVAILABLE_SHARD` errors for the entire post-resharding shard space, affecting all users of any sharded RPC pool deployment.

**Severity: High** — complete loss of RPC availability for all child shards after a protocol-level resharding event.

---

### Likelihood Explanation

Dynamic resharding is an active production feature. The `DynamicReshardingConfig` and `force_split_shards` mechanism are wired into the epoch manager and are exercised by multiple integration tests. Any deployment using `ShardedRpcConfig` (the sharded RPC pool) will encounter this failure at the first resharding epoch boundary. The bug requires no special privileges: any user querying an account in a child shard triggers the broken routing path. [8](#0-7) 

---

### Recommendation

`ShardedRpcNode.tracked_shards` must be updated (or re-derived) whenever the shard layout changes. Concretely:

1. Replace the static `Vec<ShardId>` with a dynamic structure that, given a `ShardId` and an `EpochId`, can determine whether the remote node tracks that shard — mirroring the logic already used for the local node via `ShardTracker::rpc_tracks_shard_at_epoch`.
2. Alternatively, at each epoch boundary, recompute each remote node's effective shard set by expanding parent shard IDs to their children using the epoch manager's shard layout history, analogous to `check_if_descendant_of_tracked_shard_impl`.
3. Remove the `#[ignore]` from `test_rpc_query_after_resharding` once the fix is in place and use it as a regression guard. [9](#0-8) 

---

### Proof of Concept

The existing `test_rpc_query_after_resharding` test in `test-loop-tests/src/tests/sharded_rpc_resharding.rs` is a direct proof of concept. It:

1. Creates a 2-shard layout; `rpc0` tracks shard 0, `rpc1` tracks shard 1 (both as static `TrackedShardsConfig::Shards`).
2. Triggers dynamic resharding that splits shard 0 into two child shards.
3. Queries `alice`'s account (which lands in a child shard) from both RPC nodes.

The test is marked `#[ignore]` precisely because it fails with `UNAVAILABLE_SHARD` — the pool cannot find any remote node whose static `tracked_shards` contains the new child shard ID: [10](#0-9) 

Removing `#[ignore]` and running the test reproduces the failure deterministically. The root divergence is the child `ShardId` value (e.g., a new numeric ID assigned by `ShardLayout::derive_v3`) that is absent from every `ShardedRpcNode.tracked_shards` vector populated before the resharding epoch.
