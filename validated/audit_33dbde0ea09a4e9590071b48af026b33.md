Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Curated Pool Allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which resolves to `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actualUser]`. A pool admin who allowlists the router to enable router-mediated swaps for permitted users inadvertently grants swap access to every user on the internet.

## Finding Description
`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension caller) and `sender` is whoever called `pool.swap`. [1](#0-0) 

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly, making the router itself `msg.sender` to the pool: [3](#0-2) 

The full call chain is:
```
User → router.exactInputSingle(params)
     → pool.swap(recipient=user, ...) [msg.sender = router]
     → _beforeSwap(sender=router, recipient=user, ...)
     → extension.beforeSwap(sender=router, ...)
     → checks allowedSwapper[pool][router]   ← wrong actor
```

The pool admin faces an impossible choice:
1. **Do not allowlist the router** → even legitimately permitted users cannot swap through the router.
2. **Allowlist the router** → every user can bypass the allowlist by routing through `MetricOmmSimpleRouter`.

No existing guard in the extension, pool, or router resolves this ambiguity. The `recipient` argument is available as the second parameter to `beforeSwap` but is currently unnamed and ignored. [1](#0-0) 

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of counterparties (e.g., KYC'd addresses, institutional partners, or whitelisted market makers) loses that protection entirely for any user who routes through `MetricOmmSimpleRouter`. Since the router is the primary user-facing swap interface, the allowlist is effectively nullified for the most common access path. Any disallowed user can execute swaps against the pool's LP assets, directly contradicting the pool's curation policy and causing direct loss of LP funds if the pool was designed to trade only with trusted counterparties. This constitutes a broken core pool functionality and an admin-boundary break where an unprivileged path bypasses a configured access control.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard supported periphery path for swaps. Any user who discovers a curated pool and attempts to swap through the router will succeed if the router is allowlisted. The router address is public and fixed, so no special knowledge or privilege is required. The trigger is a normal `exactInputSingle` or `exactInput` call from any EOA. The precondition — the pool admin allowlisting the router — is a natural and expected configuration step for any admin who wants permitted users to use the router.

## Recommendation
The `beforeSwap` hook must gate the actual end-user, not the intermediary. The most robust fix is to check `recipient` instead of `sender`, since the recipient is the address that economically benefits from the swap and cannot be spoofed by an intermediary:

```solidity
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

Note that for multi-hop `exactInput`, intermediate hops use `address(this)` (the router) as recipient; the extension should be documented as incompatible with intermediate hops, or the router should encode the original caller in `extensionData` for the extension to decode.

## Proof of Concept
**Setup:**
- Deploy a pool with `SwapAllowlistExtension` configured.
- Pool admin allowlists `alice`: `extension.setAllowedToSwap(pool, alice, true)`.
- Pool admin allowlists the router so `alice` can use it: `extension.setAllowedToSwap(pool, router, true)`.

**Attack:**
- `bob` (not allowlisted) calls `router.exactInputSingle({pool: pool, recipient: bob, ...})`.
- Router calls `pool.swap(recipient=bob, ...)` with `msg.sender = router`.
- Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
- `bob` successfully swaps against the curated pool despite never being allowlisted.

**Verification:**
```solidity
// Extension sees sender = router, not bob
function beforeSwap(address sender, ...) external view {
    // sender == address(router), not address(bob)
    // allowedSwapper[pool][router] == true  ← passes
    // allowedSwapper[pool][bob]    == false ← would have blocked
}
