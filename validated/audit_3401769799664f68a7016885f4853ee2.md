### Title
`ValidResourceBounds` Protobuf Round-Trip Canonicalization Failure Produces Divergent Transaction Hash for Zero-L2/Data-Gas Transactions - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` silently converts `AllResources` with zero `l2_gas` and `l1_data_gas` to `L1Gas`. Because `get_tip_resource_bounds_hash` includes `L1DataGas` in the Poseidon preimage only for the `AllResources` variant, the same transaction produces two distinct hashes: one computed at RPC ingestion (always `AllResources`) and one recomputed after P2P protobuf deserialization (`L1Gas`). Any node that recomputes the hash after sync will diverge from the sequencer-assigned hash stored in the block.

### Finding Description

`InternalRpcInvokeTransactionV3` and `InternalRpcDeclareTransactionV3` store `resource_bounds: AllResourceBounds` and always return `ValidResourceBounds::AllResources(self.resource_bounds)` from their `resource_bounds()` trait method: [1](#0-0) 

The hash function `get_invoke_transaction_v3_hash` calls `get_tip_resource_bounds_hash`, which appends `L1DataGas` to the Poseidon preimage **only** when the variant is `AllResources`: [2](#0-1) 

So a transaction with zero `l2_gas` and `l1_data_gas` submitted via RPC is hashed as:
```
poseidon(tip, l1_gas_packed, l2_gas_packed_zero, l1_data_gas_packed_zero)
```

But the protobuf deserializer converts the same wire representation back to `L1Gas`: [3](#0-2) 

After deserialization, the hash is recomputed as:
```
poseidon(tip, l1_gas_packed, l2_gas_packed_zero)
```

The two hashes differ because the `AllResources` path includes an extra zero element (`l1_data_gas_packed_zero`) in the Poseidon preimage. The serialization path confirms the round-trip is lossy: `From<ValidResourceBounds> for protobuf::ResourceBounds` emits identical wire bytes for `AllResources{l1_gas, l2_gas=0, l1_data_gas=0}` and `L1Gas{l1_gas}` (both produce `l2_gas=0, l1_data_gas=0` on the wire), but deserialization collapses both to `L1Gas`: [4](#0-3) 

The hash computation path for `InvokeTransactionV3` (the storage type) uses `self.resource_bounds` directly: [5](#0-4) 

### Impact Explanation

Any node that recomputes the transaction hash after deserializing from protobuf (e.g., during P2P sync, block validation, or receipt verification via `validate_transaction_hash`) will compute a hash H′ that differs from the sequencer-assigned hash H stored in the block. Consequences:

1. **Wrong receipt/event data**: RPC calls that look up a transaction by hash H will fail on a synced node that stored it under H′.
2. **Block validation failure**: A syncing node calling `validate_transaction_hash` on a deserialized transaction with zero l2/data gas will always fail, causing it to reject valid committed blocks.
3. **Inconsistent state across nodes**: The sequencer and syncing nodes disagree on the canonical hash of the same transaction, breaking the invariant that `hash(tx) == hash(deserialize(serialize(tx)))`.

This matches the **"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input"** critical impact, and the **"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value"** high impact.

### Likelihood Explanation

A transaction with zero `l2_gas` and `l1_data_gas` (non-zero `l1_gas`) is valid and accepted by the gateway. The stateless validator only requires at least one non-zero resource bound: [6](#0-5) 

Any pre-0.13.3 style V3 transaction (L1-gas-only) submitted through the current RPC path triggers this. The TODO comment in the protobuf converter confirms this is a known backward-compatibility gap, not an intentional design: [7](#0-6) 

### Recommendation

Fix the protobuf deserialization to preserve the `AllResources` variant even when `l2_gas` and `l1_data_gas` are zero. The condition `if l1_data_gas.is_zero() && l2_gas.is_zero()` should not silently downgrade to `L1Gas` for V3 transactions that were originally submitted as `AllResources`. Alternatively, `get_tip_resource_bounds_hash` should always include `L1DataGas` in the hash preimage for all V3 transactions, regardless of the `ValidResourceBounds` variant, to make the hash invariant to the variant choice.

### Proof of Concept

1. Submit a V3 invoke transaction: `l1_gas = {max_amount: 1000, max_price_per_unit: 1}`, `l2_gas = {0, 0}`, `l1_data_gas = {0, 0}`.
2. Gateway converts to `InternalRpcInvokeTransactionV3` with `AllResourceBounds`; computes hash **H** = `poseidon(tip=0, l1_gas_packed, l2_gas_packed_zero, l1_data_gas_packed_zero)`.
3. Transaction is included in a block with stored hash **H**.
4. Syncing node receives the block via P2P; deserializes `protobuf::ResourceBounds` → `ValidResourceBounds::L1Gas(l1_gas)` (because `l2_gas.is_zero() && l1_data_gas.is_zero()`).
5. Syncing node recomputes hash **H′** = `poseidon(tip=0, l1_gas_packed, l2_gas_packed_zero)` — no `L1DataGas` element.
6. **H ≠ H′**: `validate_transaction_hash` fails; RPC `starknet_getTransactionByHash(H)` returns not-found on the synced node; receipts and events referencing **H** are unreachable.

This is the direct sequencer analog of the external rounding bug: just as `convert_amount` loses information by truncating `1 token → 0 notes`, the protobuf round-trip loses information by collapsing `AllResources{zero, zero}` → `L1Gas`, silently changing the hash domain and producing a divergent canonical identifier for the same transaction.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

pub(crate) fn get_invoke_transaction_v3_hash<T: InvokeTransactionV3Trait>(
    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();
    let calldata_hash =
        HashChain::new().chain_iter(transaction.calldata().0.iter()).get_poseidon_hash();
    let mut hash_chain = HashChain::new()
        .chain(&INVOKE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address().0.key())
        .chain(&tip_resource_bounds_hash)
        .chain(&paymaster_data_hash)
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce().0)
        .chain(&data_availability_mode)
        .chain(&account_deployment_data_hash)
        .chain(&calldata_hash);
    if !transaction.proof_facts().0.is_empty() {
        let proof_facts_hash =
            HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
        hash_chain = hash_chain.chain(&proof_facts_hash);
    }
    Ok(TransactionHash(hash_chain.get_poseidon_hash()))
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
