Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of real swapper, allowing any user to bypass per-pool allowlist via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract address, not the end user. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][real_user]`, making the allowlist trivially bypassable by any user who calls the public router. This constitutes a complete nullification of the pool admin's configured access-control policy.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` (the immediate caller) as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks that exact value against its per-pool allowlist, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly — the router is `msg.sender` at the pool, not the end user: [3](#0-2) 

The same pattern applies to `exactInput` (L104), `exactOutputSingle` (L136), and `exactOutput` (L165), as well as the recursive callback path in `_exactOutputIterateCallback` (L220). [4](#0-3) 

The result is a complete identity mismatch. Two failure modes follow:

1. **Allowlist bypass** — if the pool admin allowlists the router (the natural operational choice so that normal users can swap), every address in the world can swap through the router and pass the guard, defeating the allowlist entirely.
2. **Allowlist lock-out** — if the pool admin allowlists individual user addresses (not the router), those users cannot swap through the router at all, breaking the normal swap UX.

Existing guards are insufficient: `BaseMetricExtension` only validates that `msg.sender` is a known pool (via factory), but does not validate the identity of the economic actor. There is no mechanism in the current extension interface to distinguish the real user from the router.

## Impact Explanation
`SwapAllowlistExtension` is a production access-control extension whose sole purpose is to restrict which addresses may execute swaps on a given pool. A bypass of this guard means an unpermissioned address can execute swaps against a pool explicitly configured to reject it. This constitutes a broken core pool functionality and a direct admin-boundary break: the pool admin's configured access policy is silently nullified by any user who routes through the public router. Depending on the pool's purpose (institutional-only, KYC-gated, whitelist-only), this is a High severity finding.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing entry point for swaps. Any user who wants to swap on an allowlist-protected pool will naturally use the router. The bypass requires no special privileges, no malicious setup, and no non-standard tokens — only a call to a public function that is already the standard swap path. The condition is trivially reachable by any unprivileged trader.

## Recommendation
The extension must check the economically relevant actor, not the immediate `msg.sender` of the pool. Two sound approaches:

1. **Preferred** — Redesign the router to pass the real user's address (`msg.sender` of the router call) as the `sender` argument to `pool.swap()` rather than letting the pool use the router's address. This requires a protocol-level convention where the router is a trusted forwarder and the pool/extension interface is updated to accept an explicit `originator` field.
2. **Alternative** — Require that any allowlisted router also passes the real user identity in `extensionData`, and update `SwapAllowlistExtension.beforeSwap` to decode and check that value when `sender` is a known trusted forwarder.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension registered
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (natural operational setup so users can swap via router)
  - Alice (address NOT in allowlist) calls:
      router.exactInputSingle({pool: pool, ...})

Execution trace:
  Alice → MetricOmmSimpleRouter.exactInputSingle()
    → IMetricOmmPoolActions(pool).swap(recipient, ...)
      [msg.sender at pool = router address]
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓  (passes)
      → swap executes, Alice receives output tokens

Result:
  Alice, who is NOT in the allowlist, successfully swaps on an
  allowlist-protected pool. The guard checked allowedSwapper[pool][router]
  instead of allowedSwapper[pool][Alice].
```

A Foundry test can confirm this by: (1) deploying a pool with `SwapAllowlistExtension`, (2) calling `setAllowedToSwap(pool, address(router), true)` from the pool admin, (3) calling `router.exactInputSingle(...)` from an address not in the allowlist, and asserting the swap succeeds without revert.
