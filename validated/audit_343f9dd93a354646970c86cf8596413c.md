### Title
Gateway Performs Expensive Proof Verification Before Any Fee Commitment, Enabling Free CPU Exhaustion via Invalid Client-Side Proofs — (`crates/apollo_transaction_converter/src/transaction_converter.rs`, `crates/apollo_gateway/src/gateway.rs`)

---

### Summary

The sequencer's gateway unconditionally runs an expensive STARK proof verification circuit for every Invoke V3 transaction that carries `proof_facts` and a `proof` field, before any fee is committed or charged. Because the transaction is rejected (and no fee is deducted) when the proof is invalid, an attacker can repeatedly submit structurally valid but cryptographically invalid proofs at near-zero cost, exhausting the gateway's CPU and blocking thread pool.

---

### Finding Description

**Background — client-side proving path**

Invoke V3 transactions may carry two extra fields: `proof_facts` (a vector of `Felt` values encoding a SNOS proof statement) and `proof` (the compressed STARK proof bytes). When `allow_client_side_proving: true` (the production default), the gateway accepts these transactions and verifies the proof before admitting the transaction.

**Validation order in the gateway**

The gateway processes an incoming `RpcTransaction` in this order:

1. **Stateless validation** (`StatelessTransactionValidator::validate`) — cheap structural checks: non-zero resource bounds, proof/proof_facts consistency (both empty or both non-empty), size limits.
2. **Proof verification** — `TransactionConverter::convert_rpc_tx_to_internal_rpc_tx` spawns a blocking task that calls `starknet_proof_verifier::verify_proof`, which runs `privacy_circuit_verify_v1::verify_recursive_circuit`. The gateway **awaits** this task synchronously before proceeding.
3. **Stateful validation** — balance checks, nonce checks, `__validate__` entry point execution. [1](#0-0) [2](#0-1) [3](#0-2) 

**The broken invariant**

Step 2 (expensive proof verification) executes before step 3 (fee/balance enforcement). If the proof is invalid, the transaction is rejected at step 2 and **no fee is ever charged**. The attacker's only cost is the network bandwidth to send the HTTP request.

**What the attacker submits**

A structurally valid Invoke V3 transaction that passes all stateless checks:

- `proof_facts`: correct layout — valid `ProofVersion::V1` marker, `VIRTUAL_SNOS` variant marker, a program hash from `allowed_virtual_os_program_hashes`, a recent-enough block number, and the correct stored block hash (all publicly readable from the chain).
- `proof`: any non-empty byte string up to `max_proof_size` (480 000 bytes in production) — does not need to be a valid proof.
- `resource_bounds`: non-zero L2 gas amount and price ≥ `min_gas_price` (8 000 000 000) to pass the stateless resource-bounds check. No funds are required because the transaction is rejected before execution. [4](#0-3) [5](#0-4) 

**Why existing gates do not prevent this**

- `validate_proof_facts_and_proof_consistency` only checks that both fields are simultaneously empty or non-empty — it does not verify the proof cryptographically.
- `validate_client_side_proving_allowed` only checks the feature flag.
- `max_proof_size: 480000` limits proof size but does not limit the number of concurrent verification requests.
- There is no concurrency cap on proof-verification blocking tasks (unlike `max_concurrent_declare_compilations: 40` for Sierra compilation). Tokio's default blocking thread pool (512 threads) can be saturated.
- The stateful validator's balance check (`verify_can_pay_committed_bounds`) runs **after** proof verification and is never reached for invalid proofs. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

An unprivileged attacker with no on-chain funds can flood the gateway with Invoke V3 transactions carrying valid proof_facts metadata but invalid proof bytes. Each request forces the gateway to run the full STARK recursive circuit verifier before rejecting the transaction. Because no fee is charged on rejection, the attack is economically free. Saturating the blocking thread pool stalls all concurrent gateway operations (including legitimate transaction admission), matching the "High — Mempool/gateway/RPC admission" impact category.

---

### Likelihood Explanation

- `allow_client_side_proving: true` is the production default.
- Valid proof_facts metadata (program hash, block hash) is publicly derivable from on-chain data.
- No funds, no deployed account, and no valid signature are required — the transaction is rejected before `__validate__` runs.
- No rate-limiting or concurrency cap on proof verification is present in the codebase.

---

### Recommendation

Apply one or more of the following mitigations:

1. **Concurrency cap**: Add a semaphore (analogous to `max_concurrent_declare_compilations`) that limits the number of simultaneously running proof-verification blocking tasks.
2. **Proof-of-work / fee commitment before verification**: Require a small fee commitment (e.g., a signed nonce or a pre-paid deposit) that can be seized if the proof is invalid, before running the expensive verifier.
3. **Cheap pre-filter**: Add a fast structural pre-check on the proof bytes (e.g., version prefix, minimum length, decompression header) before dispatching to the full circuit verifier, to reject obviously malformed proofs cheaply.
4. **Per-sender rate limiting**: Throttle proof-verification requests per sender address at the HTTP layer before they reach the converter.

    /// Runs proof verification: checks if the proof already exists, and if not, verifies it.
    /// Returns `true` if verification was performed, `false` if skipped (proof already stored).
    /// This is the shared verification logic used by both gateway and consensus flows.
    async fn run_proof_verification(
        proof_facts: ProofFacts,
        proof: Proof,
        proof_manager_client: SharedProofManagerClient,
    ) -> Result<bool, TransactionConverterError> {
        let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;

        if contains_proof {
            return Ok(false);
        }

        let proof_facts_hash = proof_facts.hash();
        let verify_start = Instant::now();
        tokio::task::spawn_blocking(move || {
            starknet_proof_verifier::verify_proof(proof_facts, proof)
        })
        .await
        .expect("proof verification task panicked")?;
        let verify_duration = verify_start.elapsed();
        PROOF_VERIFICATION_LATENCY.record(verify_duration.as_secs_f64());
        info!(
            "Proof verification took: {verify_duration:?} for proof facts hash: \
             {proof_facts_hash:?}"
        );

        Ok(true)
    }

/// Verifies a submitted proof against the proof facts using the circuit verifier.
///
/// The first element of `proof_facts` must be V1, verified via `privacy-circuit-verify-v1`.
pub fn verify_proof(proof_facts: ProofFacts, proof: Proof) -> Result<(), VerifyProofError> {
    // Reject empty proof payloads before running the verifier.
    if proof.is_empty() {
        return Err(VerifyProofError::EmptyProof);
    }

    let proof_version_felt = proof_facts.0.first().copied().unwrap_or_default();
    let proof_version = ProofVersion::try_from(proof_version_felt)
        .map_err(|()| VerifyProofError::InvalidProofVersion { actual: proof_version_felt })?;

    let output_preimage = reconstruct_output_preimage(&proof_facts)?;
    // TODO(Avi): Avoid cloning the proof.
    let proof_bytes = proof.0.to_vec();

    match proof_version {
        // V0 proofs are no longer verifiable: the v0 circuit was removed. V0 proof facts are only
        // tolerated by the blockifier (gated per protocol version) for replaying historical blocks.
        ProofVersion::V0 => {
            return Err(VerifyProofError::InvalidProofVersion { actual: proof_version_felt });
        }
        ProofVersion::V1 => {
            let proof_output = privacy_circuit_verify_v1::PrivacyProofOutput {
                proof: proof_bytes,
                output_preimage,
            };
            privacy_circuit_verify_v1::verify_recursive_circuit(&proof_output)
                .map_err(|e| VerifyProofError::Verification(e.to_string()))?;
        }
    }

    Ok(())
}
