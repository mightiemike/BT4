### Title
`get_sortable_validator_online_ratio` Ignores `endorsement_cutoff_threshold`, Causing Divergent Kickout-Exemption Ordering vs. Reward Calculation - (`chain/epoch-manager/src/validator_stats.rs`)

### Summary

`get_sortable_validator_online_ratio` always calls `get_validator_online_ratio` with `endorsement_cutoff_threshold = None`, while `calculate_reward` calls the same function with the epoch-configured cutoff threshold. This means the sort order used to determine which validators are **exempted from kickout** is computed with a different endorsement-ratio semantics than the sort order implied by the **reward calculation**. Validators near the cutoff boundary are ranked inconsistently between the two paths, causing the wrong validators to be protected from kickout.

### Finding Description

`get_validator_online_ratio` accepts an `endorsement_cutoff_threshold: Option<u8>`. When a threshold is provided, the endorsement ratio is binarized: below the threshold it becomes `0/1`, at or above it becomes `1/1`. When `None` is passed, the raw ratio `produced/expected` is used.

Two call sites diverge:

**Reward path** — `calculate_reward` in `reward_calculator.rs` passes the epoch-configured cutoff:

```rust
let production_ratio =
    get_validator_online_ratio(&stats, online_thresholds.endorsement_cutoff_threshold);
```

The cutoff is set from `epoch_config.chunk_validator_only_kickout_threshold` in `finalize_epoch`.

**Kickout-exemption sort path** — `compute_validators_to_reward_and_kickout` in `lib.rs` calls the sortable wrapper, which **hard-codes `None`**:

```rust
pub(crate) fn get_sortable_validator_online_ratio(stats: &BlockChunkValidatorStats) -> BigRational {
    let ratio = get_validator_online_ratio(stats, None);   // cutoff always ignored
    ...
}
```

This wrapper is used to sort all validators and determine who is exempted from kickout:

```rust
let mut sorted_validators = validator_block_chunk_stats
    .iter()
    .map(|(account, stats)| (get_sortable_validator_online_ratio(stats), account))
    .collect_vec();
sorted_validators.sort_by(validator_comparator);
...
let exempted_validators = Self::compute_exempted_kickout(..., &accounts_sorted_by_online_ratio, ...);
```

**Concrete divergence example** (cutoff = 50%, `online_min` = 90%, `online_max` = 99%):

| Validator | blocks | chunks | endorsements | Sortable ratio (None) | Reward ratio (cutoff=50%) |
|---|---|---|---|---|---|
| A | 90/100 | 90/100 | 51/100 | (90+90+51)/300 = **231/300** | (90+90+100)/300 = **280/300** |
| B | 95/100 | 95/100 | 49/100 | (95+95+49)/300 = **239/300** | (95+95+0)/300 = **190/300** |

- **Exemption sort**: B > A → B is protected from kickout, A may be kicked out.
- **Reward calculation**: A > B → A has higher reward-relevant uptime.

The exemption mechanism is supposed to protect the validators with the highest actual uptime. Here it protects B (lower reward-relevant uptime) and exposes A (higher reward-relevant uptime) to kickout.

### Impact Explanation

The `compute_exempted_kickout` safety mechanism is designed to prevent too many high-uptime validators from being simultaneously removed. When the sort metric diverges from the reward metric, the wrong validators are protected. A validator whose endorsement ratio is just below the cutoff (e.g., 49%) is ranked higher in the exemption sort than a validator whose endorsement ratio is just above the cutoff (e.g., 51%) but whose block/chunk production is slightly lower. This can cause:

1. A validator with higher reward-relevant uptime to be kicked out while a lower-uptime validator is protected.
2. Repeated across epochs, this degrades the validator set quality in a way that is invisible to the reward signal.
3. In adversarial conditions, a validator can deliberately keep endorsement ratio just below the cutoff to maximize their exemption-sort rank while paying zero endorsement-component reward cost.

### Likelihood Explanation

The condition requires validators near the endorsement cutoff boundary and `validator_max_kickout_stake_perc` configured such that not all violating validators can be kicked out simultaneously. Both conditions are normal in production: the cutoff is set to `chunk_validator_only_kickout_threshold` (a standard config value), and the max-kickout-stake cap is a standard safety parameter. The divergence is always present in the code; it activates whenever the exemption mechanism is triggered.

### Recommendation

Pass the same `endorsement_cutoff_threshold` to `get_sortable_validator_online_ratio` that is used in `calculate_reward`. Concretely, add the threshold parameter to `get_sortable_validator_online_ratio` and forward it to `get_validator_online_ratio`:

```rust
pub(crate) fn get_sortable_validator_online_ratio(
    stats: &BlockChunkValidatorStats,
    endorsement_cutoff_threshold: Option<u8>,  // add parameter
) -> BigRational {
    let ratio = get_validator_online_ratio(stats, endorsement_cutoff_threshold);
    ...
}
```

Then pass `Some(config.chunk_validator_only_kickout_threshold)` at the call site in `compute_validators_to_reward_and_kickout`, consistent with how `finalize_epoch` constructs `online_thresholds`.

```rust
fn get_endorsement_ratio(stats: &ValidatorStats, cutoff_threshold: Option<u8>) -> (u64, u64) {
    let (numer, denom) = if stats.expected == 0 {
        debug_assert_eq!(stats.produced, 0);
        (0, 0)
    } else if let Some(threshold) = cutoff_threshold {
        if stats.less_than(threshold) { (0, 1) } else { (1, 1) }
    } else {
        (stats.produced, stats.expected)
    };
    (numer, denom)
}
```
