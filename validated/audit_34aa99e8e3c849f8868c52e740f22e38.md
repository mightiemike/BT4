### Title
`PauseManager` Role Never Granted During Initialization Renders Pause Mechanism Permanently Inoperable - (`contract/src/lib.rs`)

### Summary

`BtcLightClient` derives `Pausable` from `near_plugins`, applies `#[pause]` guards to all critical public methods, and defines a `PauseManager` role to control those guards. However, the `init` function never grants the `PauseManager` role to any account. Because the only super admin set during initialization is the contract account itself — not a human-controlled account — no external actor can invoke `pa_pause` or `pa_unpause` after deployment without a separate, explicit, and easily-omitted post-deployment step. The pause mechanism is declared and wired in, but is effectively dead on arrival.

### Finding Description

The contract configures the `Pausable` plugin and applies `#[pause]` to every externally-reachable state-changing and verification method: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

The `near_plugins` `Pausable` derive macro generates `pa_pause` and `pa_unpause` methods that are exclusively gated by `manager_roles(Role::PauseManager)`. Only an account holding `PauseManager` can call them.

The `init` function sets the contract account itself as the sole super admin: [6](#0-5) 

The comment explicitly acknowledges this is done so roles can be granted in the constructor — yet no `acl_grant_role` call for `PauseManager` (or any other role) follows. The constructor returns without ever assigning `PauseManager` to any account: [7](#0-6) 

The grep over all production source files confirms `PauseManager` appears only in the role enum definition and the `#[pausable]` attribute — never in a role-grant call within `contract/src/`: [8](#0-7) 

The super admin is `env::current_account_id()` — the contract's own account ID, not the deployer's account. In NEAR, calling `acl_grant_role` from the contract account requires either a full-access key on that account (which is typically deleted after deployment for security) or a cross-contract call. Neither is guaranteed, and neither is enforced by the contract code.

### Impact Explanation

All four `#[pause]`-guarded entry points — `submit_blocks`, `verify_transaction_inclusion`, `verify_transaction_inclusion_v2`, and `run_mainchain_gc` — cannot be halted by any account after deployment unless `PauseManager` is explicitly granted in a separate post-deployment transaction. If a critical vulnerability is discovered (e.g., a PoW verification bypass, a chain-reorg manipulation, or a Merkle proof forgery), the protocol cannot be paused to stop ongoing exploitation. Downstream consumers calling `verify_transaction_inclusion_v2` would continue to receive potentially corrupted proof results with no ability to halt the contract. The `#[pause]` guards create a false sense of security: they are syntactically present but operationally unreachable.

### Likelihood Explanation

The `init` function's own comment states the super admin is set "so we can grant any role in the constructor," yet no role is granted. This is a straightforward omission that will manifest the first time an operator attempts to pause the contract in an emergency and discovers no account holds `PauseManager`. The likelihood of the omission existing in a live deployment is high; the likelihood of it being discovered only under emergency conditions makes the impact severe.

### Recommendation

Grant `PauseManager` (and other critical operational roles such as `DAO`) to a designated governance account inside `init`, before returning:

```rust
#[init]
#[private]
#[must_use]
pub fn init(args: InitArgs) -> Self {
    let mut contract = Self { ... };

    near_sdk::require!(
        contract.acl_init_super_admin(env::current_account_id()),
        "Failed to initialize super admin",
    );

    // Grant PauseManager to a designated operator account passed via InitArgs
    near_sdk::require!(
        contract.acl_grant_role_unchecked(
            &"PauseManager".parse().unwrap(),
            &args.pause_manager,
        ),
        "Failed to grant PauseManager role",
    );

    contract.init_genesis(...);
    contract
}
```

Alternatively, accept a `Vec<(Role, AccountId)>` in `InitArgs` so all operational roles are assigned atomically at deployment time, eliminating the post-deployment window where the pause mechanism is inoperable.

### Proof of Concept

1. Deploy the contract and call `init` with any valid `InitArgs`.
2. Attempt to call `pa_pause` from any account (including the deployer's account).
3. The call panics with an access-control error because no account holds `PauseManager`.
4. Simultaneously, call `submit_blocks` from a trusted relayer — it succeeds, confirming the contract is live and accepting headers.
5. There is no on-chain path to halt `submit_blocks` or `verify_transaction_inclusion_v2` without first executing a separate `acl_grant_role` transaction, which itself requires the contract account to be the predecessor — a condition that may be impossible if the contract account's full-access key has been deleted.
