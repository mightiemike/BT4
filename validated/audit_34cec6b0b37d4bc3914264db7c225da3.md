Audit Report

## Title
Permissionless `register()` Clears ADMIN-Set Pool Blacklist, Bypassing Emergency Oracle Access Revocation — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

## Summary
`OracleBase.register()` is a permissionless, fee-gated function that unconditionally clears `blacklisted[pool]` whenever a pool re-registers for a feed. Because `setBlacklist()` is the only ADMIN mechanism to revoke a pool's oracle-price access, any actor (including the pool operator) can pay 1 wei to immediately undo an ADMIN blacklist, restoring the pool's ability to call `price()` and execute swaps against live oracle data. This is a direct admin-boundary break: an ADMIN-level security action is overridden by a permissionless, near-zero-cost call.

## Finding Description
`OracleBase` defines two asymmetric paths for the pool blacklist:

**Setting the blacklist — ADMIN only** (`setBlacklist`, L271–276): requires `ADMIN_ROLE`.

**Clearing the blacklist — permissionless via `register()`** (L201–214): requires only `msg.value >= registrationFee` (default 1 wei), an approved factory, and `factory.isPool(pool) == true`. Lines 207–210 silently clear the blacklist:

```solidity
if (blacklisted[pool]) {
    blacklisted[pool] = false;
    emit BlacklistUpdated(pool, false);
}
```

The `price()` function enforces the blacklist at call time (L167: `require(!blacklisted[pool], Blacklisted(pool))`). Once `register()` clears `blacklisted[pool]`, this check passes and the pool resumes reading live oracle prices. The NatSpec comment at L198 even acknowledges this behavior: *"Paying also clears any blacklist on the pool."*

The exploit path:
1. ADMIN calls `setBlacklist(pool, true)` — pool's `price()` calls revert.
2. Anyone calls `register{value: 1 wei}(feedId, pool, approvedFactory)` — lines 207–210 set `blacklisted[pool] = false`.
3. Pool calls `price(feedId, pool)` — L167 check passes, live oracle data returned, swaps resume.

No privileged access, no timelock, no special role is required for step 2.

## Impact Explanation
The blacklist is the emergency mechanism by which ADMIN revokes a pool's oracle access — e.g., when a pool is found to be malicious, compromised, or draining LP funds through bad-price execution. Once cleared via `register()`, the pool resumes reading live oracle bid/ask and executing swaps, directly exposing LP principal to the condition that triggered the blacklist. This is an admin-boundary break (unprivileged path bypasses an oracle role check) with direct potential for loss of LP funds — meeting the Critical/High allowed impact gate.

## Likelihood Explanation
The trigger requires only: (1) a pool still recognized by an approved factory (`isPool(pool) == true`, the normal state for any legitimately deployed pool), (2) 1 wei, and (3) a single public transaction. The pool operator whose pool was blacklisted has both the motive and trivial means to call `register()` immediately after the ADMIN blacklist. The attack is repeatable: ADMIN blacklists, operator re-registers, indefinitely.

## Recommendation
Remove the blacklist-clearing logic from `register()`. Blacklisted pools should be blocked from re-registering, not silently unblocked:

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
    require(pool != address(0));
    require(approvedFactories.contains(factory), FactoryNotApproved(factory));
    require(IPoolFactory(factory).isPool(pool), NotAPool(pool));
-   if (blacklisted[pool]) {
-       blacklisted[pool] = false;
-       emit BlacklistUpdated(pool, false);
-   }
+   require(!blacklisted[pool], Blacklisted(pool));
    registeredPool[feedId][pool] = true;
    emit PoolRegistered(feedId, pool, msg.sender, msg.value);
}
```

Only `ADMIN_ROLE` via `setBlacklist(pool, false)` should be able to clear the blacklist.

## Proof of Concept

```solidity
// 1. ADMIN blacklists the pool (emergency response)
oracle.setBlacklist(address(pool), true);
// pool.price() now reverts with Blacklisted(pool)

// 2. Anyone (including the pool operator) calls register() with 1 wei
oracle.register{value: 1 wei}(feedId, address(pool), address(approvedFactory));
// Lines 207-210: blacklisted[pool] = false

// 3. Pool reads oracle price again — blacklist fully bypassed
(uint256 mid, uint256 spread,,) = oracle.price(feedId, address(pool));
// Succeeds. Pool executes swaps against live oracle data.
```

Preconditions: `approvedFactory.isPool(pool)` returns `true` (pool has not been removed from the factory) — trivially satisfied for any legitimately deployed pool.
