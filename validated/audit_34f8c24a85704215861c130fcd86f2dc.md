Audit Report

## Title
`SwapAllowlistExtension` gates on routing intermediary instead of end-user, allowing allowlist bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool binds to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end-user. If the pool admin allowlists the router to enable router-mediated swaps for legitimate users, every other user can simultaneously bypass the per-user allowlist by routing through the same router.

## Finding Description
**Root cause — wrong identity checked by the guard**

`SwapAllowlistExtension.beforeSwap` checks:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the pool calls the extension) and `sender` is the first argument passed by `ExtensionCalling._beforeSwap`.

**How `sender` is bound**

`ExtensionCalling._beforeSwap` encodes the `sender` argument directly from what `MetricOmmPool.swap` passes: [2](#0-1) 

`MetricOmmPool.swap` passes its own `msg.sender` as `sender`: [3](#0-2) 

**How the router calls the pool**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the `msg.sender` of that call: [4](#0-3) 

So the extension receives `sender = router_address`, not the actual end-user.

**The bypass**
1. Admin deploys pool with `SwapAllowlistExtension` and calls `setAllowedToSwap(pool, alice, true)`.
2. Admin also calls `setAllowedToSwap(pool, router, true)` so alice can use the standard router UX.
3. Bob (not allowlisted) calls `router.exactInputSingle({pool: pool, recipient: bob, ...})`.
4. Router calls `pool.swap(recipient=bob, ...)` → pool calls `extension.beforeSwap(sender=router, ...)`.
5. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds for bob.

**Contrast with `DepositAllowlistExtension`**

The deposit allowlist correctly ignores `sender` and checks `owner` (the economic beneficiary), which is stable regardless of routing intermediary: [5](#0-4) 

The swap allowlist has no equivalent stable identity to fall back on because the `beforeSwap` interface only exposes `sender` and `recipient` — and `sender` is the intermediary.

## Impact Explanation
Any unprivileged user can bypass a pool's per-user swap allowlist by routing through `MetricOmmSimpleRouter`. The pool admin cannot simultaneously restrict swaps to specific counterparties and support router-mediated swaps. Allowlisting the router to satisfy the latter nullifies the former entirely. Unauthorized users gain unrestricted swap access to a pool intended to be private, exposing LPs to swap flow they explicitly opted out of. In an oracle-priced pool this translates directly to LP value leakage from uninvited counterparties trading against the pool's liquidity — a direct loss of LP principal meeting the Critical/High threshold.

## Likelihood Explanation
The scenario is highly likely in practice. Pools using `SwapAllowlistExtension` are explicitly designed for restricted counterparty access. The router is the standard user-facing entry point; pool admins who want allowlisted users to have a normal UX will allowlist the router. No NatSpec on `SwapAllowlistExtension` warns that allowlisting the router opens the gate to all users. The bypass requires no special privileges — any EOA can call the router.

## Recommendation
Gate on an identity that is stable across routing intermediaries. The simplest fix consistent with how `DepositAllowlistExtension` handles the analogous case is to check `recipient` (the economic beneficiary of the swap output) instead of `sender`:

```solidity
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

This mirrors `DepositAllowlistExtension`'s use of `owner` and is robust to any routing intermediary because `recipient` is set by the end-user and cannot be spoofed by the router.

## Proof of Concept
```
Setup:
  pool = deploy pool with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)
  admin calls setAllowedToSwap(pool, router, true)   // to let alice use the router

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({
      pool: pool,
      recipient: bob,
      zeroForOne: true,
      amountIn: X,
      ...
    })

  → router calls pool.swap(recipient=bob, ...)          // router is msg.sender
  → pool calls _beforeSwap(sender=router, ...)
  → ExtensionCalling encodes sender=router and calls extension
  → extension checks allowedSwapper[pool][router] == true  ✓
  → swap executes; bob receives output tokens

Result:
  bob swapped in a pool he was explicitly excluded from.
  The allowlist invariant is broken.

Foundry test plan:
  1. Deploy SwapAllowlistExtension and a pool configured with it.
  2. setAllowedToSwap(pool, alice, true) and setAllowedToSwap(pool, router, true).
  3. vm.prank(bob); router.exactInputSingle({pool: pool, recipient: bob, ...}).
  4. Assert the swap succeeds (no NotAllowedToSwap revert) and bob receives tokens.
```
