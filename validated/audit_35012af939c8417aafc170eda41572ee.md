### Title
Deprecated `verify_transaction_inclusion` Accepts Internal Merkle Tree Nodes as Valid Transaction Proofs, Enabling Proof Forgery - (File: `contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

The deprecated but still publicly callable `verify_transaction_inclusion` function does not distinguish between leaf nodes (real transactions) and internal nodes in the Bitcoin Merkle tree. Any unprivileged NEAR caller can supply an internal Merkle tree node hash as `tx_id` with a valid sibling-path proof, causing the function to return `true` for a transaction that does not exist.

---

### Finding Description

`verify_transaction_inclusion` delegates proof checking entirely to `merkle_tools::compute_root_from_merkle_proof`, which treats any 32-byte `transaction_hash` as a starting point and walks up the tree using the supplied sibling hashes: [1](#0-0) 

The function has no concept of "leaf vs. internal node." If the caller supplies an internal node hash (e.g., `H(T0, T1)`) as `tx_id` and the correct sibling path from that node to the root as `merkle_proof`, the computed root will equal the block's stored `merkle_root`, and the function returns `true`.

The caller-facing function enforces only that the proof is non-empty and that the block is on the mainchain: [2](#0-1) 

The function carries a documented warning acknowledging this exact flaw: [3](#0-2) 

Despite the `#[deprecated]` attribute and the warning, the function remains `pub` with only a `#[pause]` guard (not paused by default), making it callable by any NEAR account: [4](#0-3) 

The fixed version, `verify_transaction_inclusion_v2`, mitigates this by requiring a coinbase proof of equal length, which pins the tree depth and forces the target to be a leaf: [5](#0-4) 

The deprecated function is not gated behind `verify_transaction_inclusion_v2`'s coinbase check, so both code paths coexist and the vulnerable one remains reachable.

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate a security-critical action (e.g., releasing bridged funds upon proof of a Bitcoin deposit) can be deceived. An attacker supplies:

- `tx_id` = hash of an internal Merkle node from any mainchain block
- `tx_index` = the position of that internal node in the tree
- `merkle_proof` = the sibling hashes from that node to the root
- `confirmations` = 1 (minimum)

The function returns `true`, falsely certifying that a non-existent transaction was included in a confirmed Bitcoin block. The corrupted value is the **proof result** returned to the caller — the exact invariant the contract is designed to protect.

---

### Likelihood Explanation

Likelihood is **high**. The attack requires no privileged role, no private key, and no special network position. Every real Bitcoin mainchain block with more than one transaction contains internal Merkle nodes. Their hashes and sibling paths are publicly derivable from block data. The attacker needs only to:

1. Pick any mainchain block stored in the contract.
2. Compute the internal node hashes from the block's transaction list (public data).
3. Call `verify_transaction_inclusion` with the internal node hash as `tx_id`.

No mining, no staking, no social engineering is required.

---

### Recommendation

Remove `verify_transaction_inclusion` entirely or add a hard runtime guard that panics unconditionally (not just a deprecation attribute). All callers must be migrated to `verify_transaction_inclusion_v2`. If backward compatibility is required during a transition period, the function body should be replaced with:

```rust
env::panic_str("verify_transaction_inclusion is disabled; use verify_transaction_inclusion_v2");
```

This eliminates the reachable vulnerable code path while preserving the ABI entry point for any tooling that references it.

---

### Proof of Concept

Given a mainchain block with transactions `[T0, T1, T2, T3]` and Merkle tree:

```
Root = H(H(T0,T1), H(T2,T3))
         /              \
   H(T0,T1)         H(T2,T3)
    /    \            /    \
   T0    T1          T2    T3
```

The attacker calls `verify_transaction_inclusion` with:

```
tx_id             = H(T0, T1)          // internal node, not a real transaction
tx_block_blockhash = <any mainchain block hash>
tx_index          = 0
merkle_proof      = [ H(T2, T3) ]      // single sibling to reach root
confirmations     = 1
```

Inside `compute_root_from_merkle_proof`:
- `current_hash = H(T0,T1)`, `current_position = 0` (even)
- Step 1: `current_hash = H(H(T0,T1), H(T2,T3))` = Root ✓

The computed root equals `header.block_header.merkle_root`, so the function returns `true` — certifying a transaction that does not exist. [6](#0-5) [7](#0-6)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
