### Title
Linked Signer Can Unilaterally Replace Itself, Transferring Subaccount Control to an Arbitrary Address — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction in `EndpointTx.sol` is validated with `allowLinkedSigner = true`. This means the **current linked signer** of a subaccount — a delegated role, not the account owner — can sign and submit a `LinkSigner` transaction to replace itself with any arbitrary address. This allows a linked signer to permanently transfer full signing authority over a victim's subaccount to an attacker-controlled address, without the subaccount owner's knowledge or consent.

---

### Finding Description

In `EndpointTx.sol`, the `LinkSigner` fast-path handler calls `validateSignedTx` with `allowLinkedSigner = true`:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` calls `validateSignature`, which passes `getLinkedSigner(sender)` as the accepted signer when `allowLinkedSigner` is `true`:

```solidity
function validateSignature(...) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

The `Verifier` then accepts a signature from **either** the subaccount owner address **or** the linked signer:

```solidity
(recovered == address(uint160(bytes20(sender)))) ||
    (recovered == linkedSigner)
``` [3](#0-2) 

The `linkedSigners` mapping stores the current linked signer per subaccount: [4](#0-3) 

**Root cause:** The `LinkSigner` transaction — which *sets* the linked signer — accepts a signature from the *existing* linked signer. A linked signer is a delegated role (e.g., a hot wallet or API key), not the account owner. Allowing it to re-assign itself to any address is a privilege escalation: the linked signer can transfer its own authority to an attacker-controlled address without the owner's involvement.

This is the direct analog to the Chakra `ckr_btc.cairo` flaw: just as any manager could call `add_manager` to install a new manager, any linked signer can call `LinkSigner` to install a new linked signer — bypassing the account owner entirely.

---

### Impact Explanation

A linked signer that is compromised, malicious, or acting outside its intended scope can:

1. Craft a `SignedLinkSigner` transaction with `sender = victim subaccount`, `signer = attacker address`, signed with the linked signer's key.
2. Submit it to the sequencer. `validateSignedTx` accepts the linked signer's signature.
3. `linkedSigners[victim subaccount]` is overwritten with the attacker's address.
4. The attacker's address now has full signing authority and can: withdraw all collateral (`WithdrawCollateral`/`WithdrawCollateralV2`), liquidate positions, transfer quote balances, and mint/burn NLP — all signed as the victim subaccount.

The victim's original owner key is effectively locked out until they submit a corrective `LinkSigner` transaction themselves, but by then assets may already be drained.

---

### Likelihood Explanation

Linked signers are a standard operational pattern in the Nado protocol — they are used by traders running automated strategies or API bots. Any subaccount that has set a linked signer is exposed. The attack requires only that the linked signer key be compromised (e.g., leaked API key, compromised hot wallet), which is a realistic and common threat model for exchange users. No privileged protocol access is needed.

---

### Recommendation

Change `allowLinkedSigner` to `false` for `LinkSigner` transactions so that only the subaccount owner's key can change the linked signer:

```diff
// EndpointTx.sol — fast-path LinkSigner handler
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
-   true
+   false
);
```

Apply the same fix to the slow-mode `LinkSigner` path in `processSlowModeTransactionImpl`, which already uses `validateSender` (owner-only check) and is not affected, but the fast-path must be hardened.

---

### Proof of Concept

1. Alice (subaccount `0xAlice000...`) sets Bob (`0xBob`) as her linked signer via a valid `LinkSigner` transaction signed by Alice.
2. Bob crafts `SignedLinkSigner { tx: { sender: 0xAlice000..., signer: 0xMallory, nonce: N }, signature: Bob_sig }`.
3. Bob signs the EIP-712 digest with his own key.
4. The sequencer calls `processTransactionImpl` → `LinkSigner` branch → `validateSignedTx(..., true)`.
5. `validateSignature` calls `verifier.validateSignature(0xAlice000..., 0xBob, digest, Bob_sig)` — passes, because `recovered == linkedSigner`.
6. `linkedSigners[0xAlice000...] = 0xMallory`.
7. Mallory signs a `WithdrawCollateral` transaction as `0xAlice000...` and drains all of Alice's collateral.
