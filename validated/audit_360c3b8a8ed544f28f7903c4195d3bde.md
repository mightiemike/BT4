### Title
Upgradeable Implementation Contracts Missing `_disableInitializers()` Allow Unauthorized Initialization — (`Endpoint.sol`, `Clearinghouse.sol`, `SpotEngine.sol`, `PerpEngine.sol`, `OffchainExchange.sol`)

---

### Summary

Five core Nado upgradeable implementation contracts do not call `_disableInitializers()` in a constructor. Any unprivileged caller can directly initialize the implementation contracts (not the proxies), take ownership, and set critical protocol addresses to attacker-controlled values. This mirrors the exact vulnerability class described in the external report.

---

### Finding Description

The following contracts are deployed behind `TransparentUpgradeableProxy` instances managed by `BaseProxyManager`, yet their implementation contracts have no constructor calling `_disableInitializers()`:

| Contract | `initialize` modifier | `_disableInitializers()` in constructor |
|---|---|---|
| `Endpoint.sol` | `initializer` | ❌ absent |
| `Clearinghouse.sol` | `initializer` | ❌ absent |
| `SpotEngine.sol` | calls `_initialize()` with `initializer` | ❌ absent |
| `PerpEngine.sol` | calls `_initialize()` with `initializer` | ❌ absent |
| `OffchainExchange.sol` | `initializer` | ❌ absent |

By contrast, the contracts that were already fixed follow the correct pattern:

- `BaseProxyManager` — `constructor() { _disableInitializers(); }` [1](#0-0) 
- `ContractOwner` — `constructor() { _disableInitializers(); }` [2](#0-1) 
- `Verifier` — `constructor() { _disableInitializers(); }` [3](#0-2) 
- `BaseWithdrawPool` — `constructor() { _disableInitializers(); }` [4](#0-3) 

The unprotected contracts have no constructor at all:

`Endpoint.sol` — no constructor, `initialize` uses `initializer` modifier directly: [5](#0-4) 

`Clearinghouse.sol` — no constructor, `initialize` uses `initializer` modifier directly: [6](#0-5) 

`SpotEngine.sol` — no constructor, `initialize` delegates to `BaseEngine._initialize()` which carries `initializer`: [7](#0-6) 

`PerpEngine.sol` — no constructor, same pattern: [8](#0-7) 

`OffchainExchange.sol` — no constructor, `initialize` uses `initializer` modifier directly: [9](#0-8) 

`BaseEngine._initialize()` carries the `initializer` modifier and sets ownership and `canApplyDeltas`: [10](#0-9) 

---

### Impact Explanation

An attacker who calls `initialize` on the `Endpoint` implementation becomes its owner and sets `sequencer`, `clearinghouse`, `offchainExchange`, `verifier`, and `endpointTx` to attacker-controlled addresses on the implementation's own storage. [11](#0-10) 

For `SpotEngine` / `PerpEngine`, the attacker becomes owner and populates `canApplyDeltas` with attacker-controlled addresses on the implementation's storage. [12](#0-11) 

For `OffchainExchange`, the attacker becomes owner and sets `clearinghouse`, `spotEngine`, `perpEngine` to attacker-controlled contracts on the implementation's storage. [13](#0-12) 

While the proxy's storage is unaffected (no direct asset loss), ownership of the implementation contracts enables social engineering and phishing attacks: an attacker can present the initialized implementation as a legitimate Nado contract, forge misleading on-chain state, or confuse integrators and tooling that inspect implementation addresses directly. This is the exact impact class described in the external report.

---

### Likelihood Explanation

The attack requires no privileges. Any EOA can call `initialize` on the deployed implementation address with arbitrary parameters. The implementation addresses are publicly discoverable from the proxy's EIP-1967 storage slot. The attack is a single transaction with zero cost beyond gas.

---

### Recommendation

Add a constructor calling `_disableInitializers()` to each affected contract, annotated with `/// @custom:oz-upgrades-unsafe-allow constructor`:

```solidity
/// @custom:oz-upgrades-unsafe-allow constructor
constructor() {
    _disableInitializers();
}
```

Apply this to: `Endpoint.sol`, `Clearinghouse.sol`, `SpotEngine.sol` (or `BaseEngine.sol`), `PerpEngine.sol`, and `OffchainExchange.sol`.

---

### Proof of Concept

```solidity
// Attacker script (pseudocode)
address endpointImpl = ITransparentUpgradeableProxy(endpointProxy).implementation();

// Any unprivileged caller can do this:
Endpoint(endpointImpl).initialize(
    address(0),          // _sanctions
    attacker,            // _sequencer  <- attacker-controlled
    attackerExchange,    // _offchainExchange
    IClearinghouse(attackerClearinghouse),
    attackerVerifier,    // _verifier
    attackerEndpointTx   // _endpointTx
);

// Attacker is now owner of the Endpoint implementation
// and all critical addresses on the implementation point to attacker contracts.
```

The same pattern applies to `Clearinghouse`, `SpotEngine`, `PerpEngine`, and `OffchainExchange` implementations. [14](#0-13)
