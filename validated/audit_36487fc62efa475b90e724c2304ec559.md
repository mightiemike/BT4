### Title
Withdrawal Permanently Reverts and User Funds Locked Due to Missing Balance-Difference Check in `handleWithdrawTransfer` — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.handleWithdrawTransfer` transfers a pre-specified `amount` to the `WithdrawPool`, then instructs the pool to forward that same `amount` to the user. No balance-difference check is performed after the transfer to the pool. For any fee-on-transfer collateral token, the pool receives `amount - fee` but is instructed to send `amount`, causing every withdrawal to revert permanently and locking user funds inside the Clearinghouse.

---

### Finding Description

`Clearinghouse.handleWithdrawTransfer` is the internal dispatch used by every withdrawal path in the protocol:

```solidity
// Clearinghouse.sol L377-L385
function handleWithdrawTransfer(
    IERC20Base token,
    address to,
    uint128 amount,
    uint64 idx
) internal virtual {
    token.safeTransfer(withdrawPool, uint256(amount));                    // (1)
    BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx); // (2)
}
```

Step (1) pushes the pre-specified `amount` from the Clearinghouse to the `WithdrawPool`. Step (2) calls `submitWithdrawal` with the same `amount`, which in turn calls `BaseWithdrawPool.handleWithdrawTransfer`:

```solidity
// BaseWithdrawPool.sol L184-L190
function handleWithdrawTransfer(
    IERC20Base token,
    address to,
    uint128 amount
) internal virtual {
    token.safeTransfer(to, uint256(amount));   // tries to send full `amount`
}
```

The actual balance received by the `WithdrawPool` after step (1) is never measured. The pre-specified `amount` is forwarded verbatim to the user in step (2). For a fee-on-transfer token, the pool receives `amount - fee` in step (1) but is told to send `amount` in step (2). Because the pool's balance is insufficient, `safeTransfer` reverts.

The same structural gap exists in `EndpointStorage.handleDepositTransfer`:

```solidity
// EndpointStorage.sol L111-L119
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal {
    safeTransferFrom(token, from, amount);                    // receives amount - fee
    safeTransferTo(token, address(clearinghouse), amount);    // tries to forward full amount → revert
}
```

Here the Endpoint receives `amount - fee` from the user but immediately tries to forward the full `amount` to the Clearinghouse, causing the deposit to revert as well.

---

### Impact Explanation

For any fee-on-transfer collateral token listed as a spot product:

- **Withdrawal path**: `Clearinghouse.withdrawCollateral` (L391–L421) calls `handleWithdrawTransfer` before debiting the user's SpotEngine balance (L412). Because the transfer to the user reverts, the entire transaction reverts, the balance debit never executes, and the user's funds remain credited in the SpotEngine but are permanently unwithdrawable — effectively locked inside the Clearinghouse with no recovery path.
- **Deposit path**: `Endpoint.depositCollateralWithReferral` (L144–L148) calls `handleDepositTransfer`, which reverts for the same reason, permanently blocking deposits of that token.

The corrupted invariant is: `Clearinghouse token balance ≥ Σ user SpotEngine balances`. After a successful deposit (possible if the Endpoint holds a residual balance) followed by a failed withdrawal, the Clearinghouse holds fewer tokens than the sum of credited balances, breaking solvency.

---

### Likelihood Explanation

The protocol supports multiple spot products with arbitrary ERC-20 tokens (`SpotEngine.addOrUpdateProduct`). Any token that implements a transfer fee — deflationary tokens, tokens with protocol-fee mechanics, or tokens whose contracts are later upgraded to add fees — triggers this path. The entry point (`depositCollateral` / `WithdrawCollateral` transaction type) is reachable by any unprivileged user who holds a balance in the affected product. Likelihood is **medium**: it is contingent on a fee-bearing token being listed, but the code provides no guard against it.

---

### Recommendation

Replace the fixed-`amount` forwarding in both functions with a balance-difference check:

**`Clearinghouse.handleWithdrawTransfer`**:
```solidity
function handleWithdrawTransfer(
    IERC20Base token,
    address to,
    uint128 amount,
    uint64 idx
) internal virtual {
    uint256 before = token.balanceOf(withdrawPool);
    token.safeTransfer(withdrawPool, uint256(amount));
    uint128 actualReceived = uint128(token.balanceOf(withdrawPool) - before);
    BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, actualReceived, idx);
}
```

**`EndpointStorage.handleDepositTransfer`**:
```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    // use (token.balanceOf(clearinghouse) - before) as the credited amount
}
```

Alternatively, restrict listed tokens to non-fee-bearing ERC-20s via an explicit allowlist check in `addOrUpdateProduct`.

---

### Proof of Concept

1. A fee-on-transfer token (1% fee per transfer) is listed as a spot product via `SpotEngine.addOrUpdateProduct`.
2. User acquires a balance in that product (e.g., via an off-chain matched trade that credits their SpotEngine balance directly, bypassing the deposit path).
3. User submits a `WithdrawCollateral` transaction for `amount = 1000` tokens.
4. Sequencer calls `submitTransactionsChecked` → `processTransaction` → `Clearinghouse.withdrawCollateral`.
5. `handleWithdrawTransfer` executes `token.safeTransfer(withdrawPool, 1000)` → WithdrawPool receives **990** tokens.
6. `submitWithdrawal(token, user, 1000, idx)` → `BaseWithdrawPool.handleWithdrawTransfer` calls `token.safeTransfer(user, 1000)` → WithdrawPool only holds 990 → **REVERT**.
7. Entire `withdrawCollateral` reverts; SpotEngine balance debit at L412 never executes.
8. User's 1000-token balance remains in SpotEngine; no withdrawal path succeeds. Funds are permanently locked. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
