### Title
`require!(!merkle_proof.is_empty())` in Shared Verification Path Blocks Legitimate Single-Transaction Block Proofs — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) contains a blanket `require!(!args.merkle_proof.is_empty())` guard that is also exercised when `verify_transaction_inclusion_v2` delegates to it. For a Bitcoin block containing exactly one transaction, the Merkle proof is legitimately empty — the transaction hash *is* the Merkle root — yet this shared guard unconditionally panics, permanently blocking any consumer contract from verifying such a transaction.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced to close the 64-byte Merkle-proof forgery window by requiring a separate coinbase proof. Its implementation correctly validates the coinbase path and then delegates the target-transaction check to the deprecated v1 function: [1](#0-0) 

The delegation converts `ProofArgsV2` → `ProofArgs` (preserving `merkle_proof` unchanged) and calls v1: [2](#0-1) 

Inside v1, before the actual Merkle computation, there is an unconditional guard: [3](#0-2) 

`compute_root_from_merkle_proof` is defined to return `transaction_hash` unchanged when the proof slice is empty: [4](#0-3) 

For a block with exactly one transaction (the coinbase), the Merkle root equals the coinbase hash directly; no sibling hashes exist, so `merkle_proof = []` is the *correct* and *only* valid proof. The guard at line 315 fires before `compute_root_from_merkle_proof` is ever reached, causing an unconditional panic.

The v2 coinbase-proof check at lines 358–365 already handles the empty-proof case correctly (it passes when `coinbase_merkle_proof` is empty and `coinbase_tx_id == merkle_root`), but the shared v1 guard then undoes that by rejecting the equally-empty `merkle_proof`. [5](#0-4) 

---

### Impact Explanation

Any NEAR consumer contract that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` to confirm a transaction residing in a single-transaction Bitcoin block will receive a contract panic instead of a boolean result. If the consumer uses the verification result to gate fund release (e.g., an atomic-swap or bridge contract), those funds become permanently inaccessible for that class of Bitcoin block. The corrupted value is the proof result: the contract returns "invalid" for a cryptographically valid inclusion proof.

---

### Likelihood Explanation

Single-transaction blocks occur on mainnet (blocks mined with no mempool transactions, containing only the coinbase). They are uncommon but real and unambiguously valid. Any relayer or user who submits such a block header and then attempts to prove the coinbase transaction's inclusion hits this path deterministically. The entry point (`verify_transaction_inclusion_v2`) is a public, unpermissioned NEAR call reachable by any account.

---

### Recommendation

Remove the `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard from `verify_transaction_inclusion`. The downstream call to `compute_root_from_merkle_proof` already handles an empty proof correctly (returning the transaction hash unchanged), so the guard provides no additional safety and actively rejects valid single-transaction proofs. If a non-empty proof is required for some other reason, the check must be moved to a branch that is *not* exercised when the block contains exactly one transaction.

---

### Proof of Concept

1. A Bitcoin block is mined containing only the coinbase transaction `T`. Its Merkle root equals `hash(T)`.
2. The relayer submits the block header via `submit_blocks`; the contract stores it.
3. A consumer contract calls `verify_transaction_inclusion_v2` with:
   - `tx_id = hash(T)`
   - `tx_index = 0`
   - `merkle_proof = []`
   - `coinbase_tx_id = hash(T)`
   - `coinbase_merkle_proof = []`
   - `confirmations = 1`
4. v2 length check: `0 == 0` ✓
5. v2 coinbase check: `compute_root_from_merkle_proof(hash(T), 0, &[])` returns `hash(T)` == `merkle_root` ✓
6. v2 delegates to v1 via `self.verify_transaction_inclusion(args.into())`.
7. v1 reaches `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` — `merkle_proof` is `[]` → **panic**.
8. The consumer contract's cross-contract call fails; any fund-release logic gated on a `true` result never executes. [6](#0-5)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }
