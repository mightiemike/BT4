Audit Report

## Title
`SwapAllowlistExtension` allowlist bypassed when router is allowlisted — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool, which is always `msg.sender` of the `pool.swap()` call. When swaps are routed through `MetricOmmSimpleRouter`, the router is `msg.sender` at the pool level. A pool admin who allowlists the router to support router-mediated swaps on a curated pool inadvertently opens the gate to every user who routes through the router, completely bypassing the per-address allowlist.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router contract `msg.sender` at the pool: [4](#0-3) 

The same pattern applies to `exactInput` (line 104), `exactOutputSingle` (line 136), and `exactOutput` (line 165). In every router-mediated path the pool sees the router address as `sender`.

**Root cause:** The allowlist check identifies the intermediary contract (router) as the economic actor, not the originating user. Once `allowedSwapper[pool][router] = true`, the check passes for every caller who routes through the router.

The existing test `test_allowedSwapSucceeds` allowlists `callers[0]` (a `TestCaller` contract) directly rather than the router, so this bypass path is never exercised: [5](#0-4) 

## Impact Explanation
Curated pools deploy `SwapAllowlistExtension` to restrict trading to a defined set of addresses (e.g., KYC-verified counterparties). A complete bypass allows any unpermissioned address to execute swaps against the pool, violating the pool's access-control invariant. LPs who deposited under the assumption that only allowlisted counterparties could trade are exposed to unauthorized counterparties. This constitutes broken core pool functionality with direct fund-impact consequences.

## Likelihood Explanation
Likelihood is high. Supporting the router is the standard user-facing integration path. A pool admin configuring a curated pool who also wants users to access it through the official router will naturally call `setAllowedToSwap(pool, router, true)`. There is no documentation or on-chain signal warning that this single entry opens the gate to all users. The admin action is valid, follows the expected integration pattern, and the bypass requires no special privileges from the attacker — only routing through the public router.

## Recommendation
The `sender` forwarded to extension hooks must represent the economic actor (end user), not the intermediary. Two complementary fixes:

1. **Router-side**: `MetricOmmSimpleRouter` should pass the originating caller's address in `extensionData` so extensions can recover it. The router already stores `msg.sender` in transient storage via `_setNextCallbackContext`; the same address should be forwarded to extensions.

2. **Extension-side**: `SwapAllowlistExtension.beforeSwap` should parse an optional `extensionData` payload carrying the verified end-user address when `sender` is a known router, falling back to `sender` for direct calls.

The cleanest fix is for `pool.swap()` to accept an explicit `onBehalfOf` parameter that extensions receive as the authoritative actor, with `msg.sender` still used for callback settlement.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Admin allowlists the router: setAllowedToSwap(pool, router, true)
  - Alice (address not in allowlist) wants to swap

Attack:
  1. Alice calls MetricOmmSimpleRouter.exactInputSingle(pool, ...)
  2. Router calls pool.swap(recipient, ...) — msg.sender at pool = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  5. Swap executes; Alice receives output tokens

Result:
  Alice, who is not individually allowlisted, successfully swaps on a
  curated pool. The allowlist guard is completely bypassed.
```

A Foundry test can reproduce this by: (1) deploying a pool with `SwapAllowlistExtension`, (2) deploying `MetricOmmSimpleRouter`, (3) calling `setAllowedToSwap(pool, router, true)`, (4) having an un-allowlisted address call `exactInputSingle` through the router, and asserting the swap succeeds rather than reverting with `NotAllowedToSwap`.
