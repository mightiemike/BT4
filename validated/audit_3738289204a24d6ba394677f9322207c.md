### Title
Duplicate-Leaf Position Bypass in `verify_transaction_inclusion_v2` Allows False Inclusion for Nonexistent Index — (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

The coinbase-proof length-equality check in `verify_transaction_inclusion_v2` is insufficient to prevent a false-inclusion claim for a nonexistent duplicate-leaf position in an odd-length block. An unprivileged NEAR caller can submit a real transaction hash at a phantom index (e.g., index 3 in a 3-tx block) using the same Merkle proof as the real index (2), and the function returns `true`.

---

### Finding Description

`verify_transaction_inclusion_v2` applies two guards before delegating to `verify_transaction_inclusion`:

1. **Length equality**: `merkle_proof.len() == coinbase_merkle_proof.len()`
2. **Coinbase root check**: the coinbase proof must compute to the stored `merkle_root` [1](#0-0) 

After both pass, it calls `verify_transaction_inclusion`, which simply computes `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == header.merkle_root` with no bounds check on `tx_index`. [2](#0-1) 

`compute_root_from_merkle_proof` is a pure positional hash traversal — it uses `current_position % 2` to decide left/right at each level, with no awareness of the actual transaction count in the block. [3](#0-2) 

**The duplicate-leaf arithmetic collision:**

Bitcoin's Merkle tree duplicates the last leaf when the count is odd. For a 3-tx block `[T0, T1, T2]`, the tree is built over `[T0, T1, T2, T2]`:

```
Level 0 (leaves): T0   T1   T2   T2
Level 1:        H(T0,T1)   H(T2,T2)
Root:         H(H(T0,T1), H(T2,T2))
```

`merkle_proof_calculator` reflects this — it pushes a duplicate before processing an odd-length level: [4](#0-3) 

The valid proof for `T2` at **index 2** is `[T2, H(T0,T1)]`:
- Step 1: pos=2 (even) → `H(T2, T2)`, pos→1
- Step 2: pos=1 (odd) → `H(H(T0,T1), H(T2,T2))` = root ✓

Now submit `T2` at **index 3** (nonexistent) with the **same proof** `[T2, H(T0,T1)]`:
- Step 1: pos=3 (odd) → `H(T2, T2)`, pos→1  ← same intermediate hash, different parity
- Step 2: pos=1 (odd) → `H(H(T0,T1), H(T2,T2))` = root ✓

Both index 2 and index 3 produce the identical root because `H(T2, T2) == H(T2, T2)` regardless of which side `T2` is placed on. The length-equality guard is satisfied because the coinbase proof for `T0` at index 0 in a 3-tx block also has length 2: `[T1, H(T2,T2)]`.

---

### Impact Explanation

Any consumer of `verify_transaction_inclusion_v2` that treats a `true` return as proof of transaction inclusion will accept a false claim. A transaction at a phantom index (one that does not exist in the actual block) is reported as included. This directly violates the stated invariant of the function and the purpose of the v2 mitigation.

---

### Likelihood Explanation

The attack requires only:
- Knowledge of a real block with an odd transaction count (extremely common on Bitcoin mainnet)
- The real coinbase tx hash and its valid Merkle proof (publicly available from any Bitcoin node or block explorer)
- The real last-tx hash and its valid Merkle proof (same source)

No privileged role, no key compromise, no special configuration. The function is `pub` and callable by any NEAR account. [5](#0-4) 

---

### Recommendation

Add an explicit upper-bound check on `tx_index` against the number of transactions implied by the proof length. The maximum valid index for a proof of length `L` is `2^L - 1`, but the actual transaction count must be stored or passed and validated. Concretely:

- Reject any `tx_index >= (1 << merkle_proof.len())` as trivially out of range.
- More precisely, require callers to supply the block's transaction count and validate `tx_index < tx_count`.
- Alternatively, check that `tx_index` and `coinbase_index` (always 0) are not both reachable by the same proof path — i.e., that the proof for `tx_index` is not identical to the proof for `tx_index XOR 1` when the sibling is the same hash.

---

### Proof of Concept

Concrete call that returns `true` for a nonexistent transaction in a 3-tx block:

```
Block B has transactions: [T0 (coinbase), T1, T2]
Merkle root = H(H(T0,T1), H(T2,T2))

verify_transaction_inclusion_v2({
    tx_id:                 T2,
    tx_block_blockhash:    B,
    tx_index:              3,          // ← nonexistent; only indices 0,1,2 exist
    merkle_proof:          [T2, H(T0,T1)],   // ← valid proof for index 2, reused for 3
    coinbase_tx_id:        T0,
    coinbase_merkle_proof: [T1, H(T2,T2)],   // ← genuinely valid coinbase proof, len=2
    confirmations:         1,
})
→ true
```

Step-by-step root reconstruction for `tx_index=3`:
1. pos=3 (odd): `current_hash = H(proof[0]=T2, T2) = H(T2,T2)`; pos→1
2. pos=1 (odd): `current_hash = H(proof[1]=H(T0,T1), H(T2,T2))` = stored `merkle_root`

Both guards pass (length=2 on both sides; coinbase proof valid), and the final comparison succeeds, returning `true` for a transaction that does not exist at index 3. [6](#0-5) [7](#0-6)

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
