Audit Report

## Title
Missing `whenNotPaused` Guard in `addLiquidity` Allows Deposits Into a Paused Pool — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary
`MetricOmmPool.addLiquidity` lacks the `whenNotPaused` modifier that `swap` carries, meaning any caller can deposit real tokens into a pool whose state has been administratively frozen. Tokens deposited during a pause are immediately at risk of being extracted at a manipulated price once the pool is unpaused.

## Finding Description
`pauseLevel` is stored at [1](#0-0)  and the `whenNotPaused` modifier enforces it via `_checkNotPaused` at [2](#0-1) . `swap` correctly applies this guard: [3](#0-2) . `addLiquidity` does not — its signature carries only `nonReentrant`: [4](#0-3) . The function proceeds through `_beforeAddLiquidity`, `LiquidityLib.addLiquidity` (which pulls tokens from the caller via callback), and `_afterAddLiquidity` without ever consulting `pauseLevel`. [5](#0-4) 

## Impact Explanation
A pool is paused precisely because its state is unsafe. Tokens deposited via `addLiquidity` during a pause are priced against whatever broken oracle or manipulated bin state triggered the pause. Once `setPause(0)` is called, the first `swap` can drain those tokens at the bad price, constituting a direct loss of deposited principal for the LP. This meets the "Critical/High direct loss of user principal" impact gate.

## Likelihood Explanation
`setPause` is callable by the factory: [6](#0-5) . The protocol explicitly documents two pause levels for different emergency scenarios. Any time either pause level is active, the missing guard is reachable by any address with no special role or precondition beyond the pool being paused.

## Recommendation
Add `whenNotPaused` to `addLiquidity`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
```

`removeLiquidity` intentionally omits the guard (LPs must be able to exit), but new deposits must be blocked while `pauseLevel != 0`.

## Proof of Concept
1. Deploy pool with a mutable price provider; oracle begins returning a manipulated price.
2. Factory calls `pool.setPause(1)` — swaps revert, but `addLiquidity` does not.
3. Attacker (or unsuspecting LP) calls `pool.addLiquidity(...)` — no revert; tokens are transferred in via the callback.
4. Factory calls `pool.setPause(0)` after believing the incident is resolved.
5. Attacker immediately calls `pool.swap(...)` against the now-unpaused pool, pricing the newly deposited liquidity at the manipulated oracle price and extracting the deposited tokens at a loss to the LP.
