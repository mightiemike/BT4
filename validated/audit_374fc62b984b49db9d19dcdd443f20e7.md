Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Allowlist - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps using `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the pool's direct caller. When `MetricOmmSimpleRouter` intermediates a swap, the pool's `msg.sender` is the router contract, so the extension checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actual_user]`. If the router is allowlisted — a natural admin action to enable router-mediated swaps — every on-chain user bypasses the allowlist gate regardless of their individual allowlist status. Additionally, `SwapAllowlistExtension.beforeSwap` drops the `onlyPool` modifier present in `BaseMetricExtension.beforeSwap`, allowing direct external calls to the extension.

## Finding Description

**Root cause — wrong identity checked:**

`SwapAllowlistExtension.beforeSwap` performs:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (correct key) and `sender` is the first argument forwarded by the pool — which is the pool's own `msg.sender` at call time.

`MetricOmmPool.swap` passes `msg.sender` (the direct pool caller) as `sender` to `_beforeSwap`: [2](#0-1) 

`ExtensionCalling._beforeSwap` forwards that value unchanged into the extension call: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the pool's `msg.sender`: [4](#0-3) 

The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

So the effective check becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`. The actual user's identity is never consulted.

**Secondary issue — missing `onlyPool` modifier:**

`BaseMetricExtension.beforeSwap` carries `onlyPool`: [6](#0-5) 

`SwapAllowlistExtension.beforeSwap` overrides it without `onlyPool`, making it `external view`: [7](#0-6) 

Without `onlyPool`, any address can call `beforeSwap` directly on the extension. While direct calls from non-pool addresses would use `allowedSwapper[caller][sender]` (which defaults to false and reverts), the absence of the guard is a defense-in-depth failure and violates the intended access model.

## Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` (e.g., for KYC-gated or partner-only trading) and allowlists `MetricOmmSimpleRouter` — the canonical periphery swap path, expected for normal usage — exposes the pool to swaps from every on-chain user. Any address explicitly denied in `allowedSwapper` can bypass the gate by routing through the router. The allowlist invariant is broken: the extension decision (`allowedSwapper[pool][user]`) is the exact wrong value consulted, and LP funds in the pool are exposed to swaps from actors the pool admin explicitly intended to exclude. This constitutes broken core pool functionality (the allowlist extension) causing potential loss of funds and an admin-boundary break where the pool admin's access control is bypassed by an unprivileged path.

## Likelihood Explanation

The trigger requires only three conditions, all reachable by unprivileged actors using the documented public interface:
1. A pool configured with `SwapAllowlistExtension` (a supported, documented extension).
2. The pool admin calling `setAllowedToSwap(pool, router, true)` — the natural action to permit router-mediated swaps for allowlisted users.
3. Any disallowed user calling `router.exactInputSingle` (or any `exact*` variant).

No special tokens, malicious setup, privileged access, or flash loans are required. The exploit is repeatable on every swap attempt.

## Recommendation

The pool must forward the original user's address through the call chain so the extension can gate on the economically relevant actor:

1. **Pool-level fix**: Add a separate `msgSender` parameter to `swap()` that the pool populates with `msg.sender` and passes to `_beforeSwap` alongside the existing `sender` argument (or use transient storage to record the original caller before the extension call).

2. **Extension-level fix**: `SwapAllowlistExtension.beforeSwap` must re-add the `onlyPool` modifier (inherited from `BaseMetricExtension`) and check a separately forwarded original-caller field rather than the `sender` argument, which is the direct pool caller (the router).

3. **Router-level mitigation** (insufficient alone): The router could forward `msg.sender` via `extensionData`, but this is opt-in and forgeable; the correct fix is to thread the original EOA through the core call path.

## Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension.
2. Pool admin calls setAllowedToSwap(pool, router, true)   // allowlist the router
3. Pool admin does NOT call setAllowedToSwap(pool, alice, true)  // alice is denied

4. alice calls router.exactInputSingle({pool: pool, ...})
   → router calls pool.swap(recipient, ...) with msg.sender = router
   → pool calls _beforeSwap(sender=router, ...)
   → extension checks allowedSwapper[pool][router] == true  ✓
   → swap executes — alice's trade goes through despite being denied

5. alice directly calls pool.swap(...)
   → pool calls _beforeSwap(sender=alice, ...)
   → extension checks allowedSwapper[pool][alice] == false  ✗
   → revert NotAllowedToSwap — correctly blocked
```

The allowlist is enforced only on direct pool calls; the router path silently substitutes the router's address for the user's address, making the guard ineffective for any router-mediated swap.
