### Title
Linked Signer Can Self-Escalate Signing Authority via `LinkSigner` with `allowLinkedSigner = true` — (File: `core/contracts/EndpointTx.sol`)

---

### Summary
In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`, meaning the currently registered linked signer can sign a `LinkSigner` transaction to replace itself with any attacker-controlled address. This is structurally analogous to the reported bug: a lower-level authorization (linked signer) is permitted to override a higher-level control (owner-only signing authority for sensitive operations), because the validation gate does not enforce that `LinkSigner` requires the subaccount owner exclusively.

---

### Finding Description

In `processTransactionImpl` (the sequencer-dispatched path), `LinkSigner` is processed as:

```solidity
// EndpointTx.sol L581-590
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // allowLinkedSigner = true
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
```

`allowLinkedSigner = true` causes `validateSignature` to pass `getLinkedSigner(sender)` as the accepted signer to `Verifier.validateSignature`:

```solidity
// EndpointTx.sol L177-183
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
```

`Verifier.validateSignature` then accepts the signature if `recovered == linkedSigner`:

```solidity
// Verifier.sol L298-303
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
```

This means the existing linked signer can sign a `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with any address.

The slow-mode path (`processSlowModeTransactionImpl`) for the same operation is correctly restricted:

```solidity
// EndpointTx.sol L232-239
validateSender(txn.sender, sender);   // msg.sender must be the subaccount owner
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

`validateSender` enforces `address(uint160(bytes20(txn.sender))) == msg.sender`, so only the wallet that owns the subaccount can change the linked signer via slow mode. The sequencer path has no equivalent restriction, creating a direct inconsistency.

The analog to the reported bug is exact: the global-level control (owner-only authorization for `LinkSigner`) is bypassed by the strategy-level control (linked signer authorization), because the validation gate uses `allowLinkedSigner = true` instead of `false` for this sensitive operation.

---

### Impact Explanation

A compromised or malicious linked signer can:

1. Sign a `LinkSigner` transaction pointing `signedTx.tx.signer` to an attacker-controlled address.
2. `linkedSigners[subaccount]` is overwritten with the attacker's address.
3. The attacker's address now passes `_checkSignature` for all order validation in `OffchainExchange._validateOrder` (line 459-465), since `signer == linkedSigner` is accepted.
4. The attacker can submit orders on behalf of the victim subaccount — including adversarial trades that drain collateral to counterparties the attacker controls.
5. The attacker can sign another `LinkSigner` to rotate the linked signer again, preventing the owner from recovering control through the sequencer path (the owner would need to wait for the slow-mode delay).

For isolated subaccounts, `getLinkedSigner` inherits the parent's linked signer:

```solidity
// EndpointTx.sol L149-156
return
    RiskHelper.isIsolatedSubaccount(subaccount)
        ? linkedSigners[
            IOffchainExchange(offchainExchange).getParentSubaccount(subaccount)
          ]
        : linkedSigners[subaccount];
```

A single `LinkSigner` attack on the parent subaccount therefore propagates signing authority loss to all of its isolated subaccounts simultaneously.

The corrupted state is: `linkedSigners[subaccount]` — a persistent mapping that controls who can authorize trades, withdrawals, and further `LinkSigner` mutations for the subaccount.

---

### Likelihood Explanation

Linked signers are explicitly designed to be hot wallets or automated trading bots — entities with elevated exposure to key compromise. A user who links a trading bot to their subaccount does not expect that bot to be able to permanently reassign signing authority. The attack requires only that the linked signer key be compromised (leaked API key, compromised bot server, malicious SDK, etc.), which is a realistic threat for any active trader using the linked signer feature. No admin access, governance capture, or sequencer compromise is required.

---

### Recommendation

Change `allowLinkedSigner` to `false` for `LinkSigner` in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This aligns the sequencer path with the slow-mode path, which already enforces owner-only authorization for `LinkSigner` via `validateSender`.

---

### Proof of Concept

1. User A owns subaccount `S` and links hot-wallet `B` as the linked signer (`linkedSigners[S] = B`).
2. Attacker compromises `B`'s private key.
3. Attacker constructs a `SignedLinkSigner` transaction: `sender = S`, `signer = C` (attacker-controlled address), signed by `B`.
4. Sequencer dispatches the transaction to `processTransactionImpl`.
5. `validateSignedTx(..., true)` calls `validateSignature(S, digest, sig, true)`, which passes `getLinkedSigner(S) = B` to `Verifier.validateSignature`. Since `recovered == B == linkedSigner`, the require passes.
6. `linkedSigners[S] = C` is written.
7. Attacker now controls `C` and uses it to sign `MatchOrders` transactions via `OffchainExchange._validateOrder`, where `_checkSignature(S, digest, C, sig)` returns `true` because `signer == linkedSigner`.
8. Attacker submits adversarial orders that transfer value out of `S` to attacker-controlled counterparty subaccounts.
9. Attacker signs a second `LinkSigner` with `C` to rotate to `D`, locking the owner out of the sequencer path until the slow-mode delay expires. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
