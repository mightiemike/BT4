### Title
`ParsedTriple` Atom/Pair Discriminant Erased in `deserialize_as_tree` Python Binding — (`wheel/src/api.rs`)

### Summary

The `deserialize_as_tree` Python binding serializes both `ParsedTriple::Atom` and `ParsedTriple::Pair` Rust enum variants as identical-shaped `(u64, u64, u64)` tuples, erasing the variant discriminant. The third field carries `atom_offset` for atoms and `right_index` for pairs — semantically incompatible values — but Python callers receive no signal to distinguish which variant they are handling. Any Python consumer of `deserialize_as_tree` that processes attacker-supplied CLVM bytes will silently misinterpret the third field for at least one variant class.

### Finding Description

In `wheel/src/api.rs`, the helper `tuple_for_parsed_triple` converts the Rust `ParsedTriple` enum to a Python object:

```rust
// wheel/src/api.rs  lines 64-78
fn tuple_for_parsed_triple(py: Python<'_>, p: &ParsedTriple) -> PyResult<Py<PyAny>> {
    let tuple = match p {
        ParsedTriple::Atom { start, end, atom_offset } =>
            PyTuple::new(py, [*start, *end, *atom_offset as u64])?,   // third = byte offset into blob
        ParsedTriple::Pair { start, end, right_index } =>
            PyTuple::new(py, [*start, *end, *right_index as u64])?,   // third = index into triples list
    };
    Ok(tuple.unbind().into_any())
}
```

Both arms produce `(u64, u64, u64)`. The Rust type system's discriminant — the enum variant itself — is completely dropped. `deserialize_as_tree` then returns a flat `Vec` of these indistinguishable tuples:

```rust
// wheel/src/api.rs  lines 80-99
fn deserialize_as_tree(
    py: Python,
    blob: &[u8],
    calculate_tree_hashes: bool,
) -> PyResult<(Vec<Py<PyAny>>, Option<Vec<Py<PyAny>>>)> {
    ...
    let r = r.iter()
        .map(|pt| tuple_for_parsed_triple(py, pt))
        .collect::<PyResult<Vec<_>>>()?;
    Ok((r, s))
}
```

The third element of an Atom triple is `atom_offset` — a small integer (0–6) representing the byte offset within the blob where the atom payload begins. The third element of a Pair triple is `right_index` — an index into the same triples list pointing to the right child. These are semantically incompatible: treating a `right_index` as an `atom_offset` (or vice versa) produces a structurally wrong tree reconstruction or an out-of-bounds blob slice.

This is a direct analog to the reported Solidity mismatch: the Rust implementation's type contract (`ParsedTriple` with two distinct variants) diverges from what the Python-facing API actually delivers (two indistinguishable 3-tuples).

### Impact Explanation

Any Python caller that uses `deserialize_as_tree` to reconstruct or analyse a CLVM tree from attacker-controlled bytes will misinterpret the third field for whichever variant it cannot identify. Concretely:

- A caller that assumes all triples are Atoms will use a Pair's `right_index` as a blob byte offset, producing a wrong atom slice or an out-of-range read.
- A caller that assumes all triples are Pairs will use an Atom's `atom_offset` (0–6) as a child index, pointing to the wrong node in the triples list.

Either path yields a silently corrupted tree representation. If the caller uses this representation to make security-relevant decisions (e.g., static analysis of a coin puzzle before spending, mempool pre-screening, or tree-hash verification), the attacker can craft CLVM bytes whose parsed structure appears safe under the caller's analysis but executes differently.

### Likelihood Explanation

`deserialize_as_tree` is the only Python-accessible path for low-level CLVM tree introspection without full execution. Any downstream Python tool that calls it and iterates the returned list must handle both variants. Because the API provides no discriminant, every such caller is forced to infer the variant from external knowledge (e.g., ordering conventions in `parse_triples`) or silently apply the wrong interpretation. The function accepts arbitrary `blob: &[u8]`, so the attacker-controlled entry path is direct.

### Recommendation

Add a discriminant to the returned tuple so Python callers can branch correctly. The minimal fix is to prepend a tag byte:

```rust
ParsedTriple::Atom { start, end, atom_offset } =>
    PyTuple::new(py, (0u8, *start, *end, *atom_offset as u64))?,
ParsedTriple::Pair { start, end, right_index } =>
    PyTuple::new(py, (1u8, *start, *end, *right_index as u64))?,
```

Alternatively, return a Python `dict` or a typed `PyClass` with named fields so the variant is unambiguous. Update the `.pyi` stub (`wheel/python/clvm_rs/clvm_rs.pyi`) to reflect the corrected return type.

### Proof of Concept

```python
from clvm_rs.clvm_rs import deserialize_as_tree

# A minimal CLVM pair: (cons 1 2) serialised as ff 01 02
blob = bytes.fromhex("ff0102")
triples, _ = deserialize_as_tree(blob, False)

# triples[0] is a Pair, triples[1] and triples[2] are Atoms.
# All three look identical: (u64, u64, u64).
# A caller that iterates and treats every triple as an Atom will
# use triples[0][2] (which is right_index=2, the index of the right child)
# as atom_offset, slicing blob[2:2] — an empty atom — instead of
# recognising it as a pair node.  The tree is silently mis-reconstructed.
for t in triples:
    print(t)   # (start, end, ambiguous_third_field) — no variant tag
```

#[pyfunction]
#[allow(clippy::type_complexity)]
fn deserialize_as_tree(
    py: Python,
    blob: &[u8],
    calculate_tree_hashes: bool,
) -> PyResult<(Vec<Py<PyAny>>, Option<Vec<Py<PyAny>>>)> {
    let mut cursor = io::Cursor::new(blob);
    let (r, tree_hashes) = parse_triples(&mut cursor, calculate_tree_hashes).map_err(eval_to_py)?;
    let r = r
        .iter()
        .map(|pt| tuple_for_parsed_triple(py, pt))
        .collect::<PyResult<Vec<_>>>()?;
    let s = tree_hashes.map(|ths| {
        ths.iter()
            .map(|b| PyBytes::new(py, b).unbind().into_any())
            .collect()
    });
    Ok((r, s))
}
