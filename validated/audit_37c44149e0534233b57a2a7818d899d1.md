### Title
Zero-Confirmation Transaction Verification Accepted by Contract — (`File: contract/src/lib.rs`)

### Summary
Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations` value with no enforced minimum. Passing `confirmations = 0` satisfies every on-chain guard and causes the contract to return `true` for a transaction in the chain tip block — a block that has not yet been buried under any subsequent work and can still be reorganized away.

### Finding Description
In `verify_transaction_inclusion`, the only bounds check on `args.confirmations` is an upper-bound guard:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [1](#0-0) 

The depth check that follows is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [2](#0-1) 

When `args.confirmations = 0`:
- The upper-bound check `0 <= gc_threshold` trivially passes.
- The depth check `(heaviest - target + 1) >= 0` is always true for `u64`.

The function then proceeds to verify the Merkle proof and returns `true` for a transaction in any mainchain block — including the current tip — with zero confirmation depth. `verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after its coinbase proof check, so it inherits the same flaw. [3](#0-2) 

The `confirmations` field is a plain `u64` in `ProofArgs` and `ProofArgsV2` with no minimum constraint at the type or contract level. [4](#0-3) 

### Impact Explanation
A consumer contract (e.g., a bridge, atomic swap, or token minting contract) that calls `verify_transaction_inclusion_v2` and relies on the returned `bool` to release funds or mint assets can be made to act on a zero-confirmation Bitcoin transaction. That transaction can subsequently be evicted by a chain reorganization. The consumer contract has no way to distinguish a `true` returned for `confirmations = 0` from one returned for `confirmations = 6`; the contract itself is the trusted source of truth. The corrupted invariant is the proof result: `true` is returned for a transaction that has not achieved any finality depth.

### Likelihood Explanation
Any unprivileged NEAR account can call `verify_transaction_inclusion_v2` directly. The `confirmations` field is entirely caller-controlled. No privileged role, leaked key, or social engineering is required. Consumer contracts that do not independently re-validate the `confirmations` argument before calling the light client — a reasonable assumption given the contract is the trusted oracle — are immediately exploitable. The attack requires only that the attacker submit a Bitcoin transaction, wait for it to appear in one block, and immediately call the verification endpoint with `confirmations = 0` before any consumer-side timeout.

### Recommendation
Enforce a protocol-level minimum confirmation depth inside the contract. Add a constant (e.g., `MIN_CONFIRMATIONS: u64 = 1`) and `require!(args.confirmations >= MIN_CONFIRMATIONS, ...)` at the top of `verify_transaction_inclusion`. For production bridge use cases, the recommended Bitcoin finality threshold is 6 confirmations; the contract should document and enforce a sensible floor rather than delegating that decision entirely to callers.

### Proof of Concept
1. Deploy the contract with any supported chain feature (e.g., `bitcoin`).
2. Submit enough headers so the mainchain tip is at height `H`.
3. Call `verify_transaction_inclusion_v2` with:
   - `tx_block_blockhash` = hash of the tip block (height `H`)
   - `confirmations = 0`
   - a valid Merkle proof for any transaction in that block
4. The contract returns `true`.
5. The tip block is then reorganized away by a competing fork with more chainwork.
6. The consumer contract has already acted on the now-invalid `true` result.

The depth check `(H - H + 1) >= 0` evaluates to `1 >= 0` which is `true`, so the guard is bypassed entirely when `confirmations = 0`. [2](#0-1)
