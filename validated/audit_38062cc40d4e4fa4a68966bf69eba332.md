### Title
Fee-on-Transfer Token Deposit Inflates User Collateral Balance Beyond Actual Holdings — (`File: core/contracts/EndpointStorage.sol`)

### Summary
`handleDepositTransfer` transfers a caller-specified `amount` to the Clearinghouse and queues a slow-mode transaction crediting that same `amount` to the user's spot balance. If the deposited token charges a transfer fee, the Clearinghouse receives less than `amount`, but the protocol's internal accounting is inflated by the full `amount`. Over time this creates a shortfall: the sum of all credited balances exceeds the actual token holdings, making the protocol insolvent for that asset.

### Finding Description
`EndpointStorage.handleDepositTransfer` performs two sequential transfers using the user-supplied `amount`:

1. `safeTransferFrom(token, from, address(this), amount)` — Endpoint receives `amount − fee`
2. `safeTransferTo(token, address(clearinghouse), amount)` — Clearinghouse receives `amount − fee` (fee deducted again) [1](#0-0) 

Neither transfer checks the actual post-transfer balance via `balanceOf`. The slow-mode transaction is then queued with the original `amount`: [2](#0-1) 

When the sequencer executes the slow-mode transaction, `Clearinghouse.depositCollateral` credits `txn.amount` (the original, undeducted value) to the user's spot balance: [3](#0-2) 

No `balanceOf` snapshot is taken before or after either transfer leg to reconcile the actual received amount against the credited amount.

The `ERC20Helper.safeTransferFrom` correctly checks the boolean return value, so the "silent failure" half of the reference bug is mitigated. The unmitigated half is the fee-on-transfer accounting gap. [4](#0-3) 

### Impact Explanation
For every deposit of a fee-bearing token, the Clearinghouse's internal spot balance for that user is inflated by `2 × fee` (one fee per transfer leg). A user can then withdraw the full credited amount, receiving more tokens than they deposited. Repeated deposits drain the Clearinghouse's real token reserves, eventually making it impossible for honest depositors to withdraw, constituting direct funds insolvency for that collateral asset.

### Likelihood Explanation
Medium. The protocol is designed to support multiple ERC-20 spot collateral tokens (any token registered via `SpotEngine`). If an operator lists a token that charges a transfer fee (e.g., tokens with built-in tax mechanics), every deposit through `depositCollateralWithReferral` or `depositCollateral` triggers the accounting inflation. No special attacker capability is required beyond being a normal depositor.

### Recommendation
1. Measure the actual received amount using a `balanceOf` snapshot before and after each transfer leg in `handleDepositTransfer`, and pass the measured delta — not the user-supplied `amount` — into the slow-mode transaction payload.
2. Alternatively, enforce at product-listing time that only tokens without transfer fees are accepted (e.g., via an invariant check in `addEngine` or product registration).

### Proof of Concept
1. A fee-on-transfer token (1% fee per transfer) is listed as a supported spot product.
2. Attacker calls `Endpoint.depositCollateralWithReferral` with `amount = 1000`.
3. `handleDepositTransfer` executes:
   - `safeTransferFrom(token, attacker, endpoint, 1000)` → Endpoint receives 990.
   - `safeTransferTo(token, clearinghouse, 1000)` → Clearinghouse receives 990 (Endpoint's balance is now −10, reverting unless Endpoint holds residual balance; if Endpoint holds ≥10 tokens from slow-mode fees, the call succeeds and drains that reserve).
4. Slow-mode tx is queued with `amount = 1000`.
5. Sequencer executes: `Clearinghouse.depositCollateral` calls `spotEngine.updateBalance(productId, attacker, 1000 × multiplier)`.
6. Attacker's credited balance is 1000 (scaled), but Clearinghouse holds only 990.
7. Attacker withdraws 1000; Clearinghouse transfers 1000 to the withdraw pool, drawing 10 tokens from other depositors' reserves.
8. Repeated across many deposits, the Clearinghouse becomes insolvent for that token. [5](#0-4) [6](#0-5)

        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));

        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {

            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
