### Title
Missing Non-Negative Guard on `canSettle` Inflates `availableSettle` and `vQuoteBalance` — (`core/contracts/PerpEngine.sol`)

### Summary

`PerpEngine.settlePnl` applies `state.availableSettle -= canSettle` and `balance.vQuoteBalance -= canSettle` without verifying that `canSettle ≥ 0`. When a subaccount holds an underwater perp position, `getSettlementState` returns a **negative** `canSettle` via `MathHelper.min(negativePositionPnl, positiveAvailableSettle)`. Subtracting a negative value inflates both `availableSettle` and `vQuoteBalance`, creating unbacked value in the settlement pool that subsequent legitimate settlers can drain.

### Finding Description

**Step 1 — `calculatePositionPnl` can be negative.**

`positionPnl = priceX18.mul(balance.amount) + balance.vQuoteBalance` [5](#0-4) 

For any long position whose mark price has fallen below entry, or any short position whose mark price has risen above entry, this expression is negative. This is a routine market condition.

**Step 2 — `getSettlementState` propagates the negative value without clamping.**

```solidity
availableSettle = MathHelper.min(
    calculatePositionPnl(balance, productId),  // e.g. -100
    state.availableSettle                       // e.g. +200
);
// returns -100
``` [6](#0-5) 

`MathHelper.min` is a plain signed comparison with no floor at zero. [4](#0-3) 

**Step 3 — `settlePnl` applies the negative `canSettle` with no guard.**

```solidity
state.availableSettle -= canSettle;   // 200 - (-100) = 300  ← INFLATED
balance.vQuoteBalance -= canSettle;   // vQ  - (-100) = vQ+100 ← INFLATED
totalSettled += canSettle;            // = -100  (returned to Clearinghouse)
``` [7](#0-6) 

There is no `if (canSettle <= 0) continue;` or `require(canSettle >= 0)` anywhere in the loop. [8](#0-7) 

**Step 4 — Clearinghouse receives a negative `totalSettled`.**

The Clearinghouse's `_settlePnl` receives the negative return value and credits it to the subaccount's spot quote balance (decreasing it). The subaccount's net value is unchanged (spot −100, perp vQuote +100), but `availableSettle` has been inflated by 100 with no backing. Every subsequent legitimate settler can now drain 100 more units from the pool than was ever deposited.

### Impact Explanation

- `state.availableSettle` is permanently inflated by `|canSettle|` per exploit call. Repeated calls (one per underwater product per subaccount) compound the inflation.
- Legitimate subaccounts with positive PnL call `settlePnl` and receive spot quote funded by the inflated pool — unbacked spot quote is minted.
- This directly violates the invariant: `sum(vQuoteBalance) + availableSettle` is not conserved across `settlePnl`.
- Impact: **unbacked creation of collateral / quote balances** — Critical scope.

### Likelihood Explanation

- The preconditions (negative position PnL + positive `availableSettle`) are normal operating conditions on any active perp market.
- The entry path (`Clearinghouse.settlePnl` → `PerpEngine.settlePnl`) is a standard, externally reachable production transaction.
- No special privileges, admin keys, or sequencer compromise are required.
- Any user with an underwater perp position can trigger this.

### Recommendation

In `PerpEngine.settlePnl`, skip (or clamp) any product where `canSettle ≤ 0`:

```solidity
if (canSettle <= 0) {
    productIds >>= 32;
    continue;
}
```

Alternatively, clamp inside `getSettlementState`:

```solidity
availableSettle = MathHelper.max(
    0,
    MathHelper.min(
        calculatePositionPnl(balance, productId),
        state.availableSettle
    )
);
``` [6](#0-5) 

### Proof of Concept

```
Setup:
  productId = 2 (even, perp)
  state.availableSettle = 200
  balance.amount = 10, balance.vQuoteBalance = -1500
  priceX18 = 100  →  positionPnl = 100*10 + (-1500) = -500

Execution:
  canSettle = min(-500, 200) = -500
  state.availableSettle -= -500  →  200 + 500 = 700  (was 200)
  balance.vQuoteBalance -= -500  →  -1500 + 500 = -1000
  totalSettled = -500  (Clearinghouse debits spot by 500)

Invariant check (before): sum(vQuoteBalance) + availableSettle = -1500 + 200 = -1300
Invariant check (after):  sum(vQuoteBalance) + availableSettle = -1000 + 700 = -300
Delta: +1000 units created from nothing.

Other subaccounts with positive PnL can now settle against the inflated
availableSettle=700, draining 500 extra unbacked units as spot quote.
``` [9](#0-8)

    function settlePnl(bytes32 subaccount, uint256 productIds)
        external
        returns (int128)
    {
        _assertInternal();
        int128 totalSettled = 0;

        while (productIds != 0) {
            uint32 productId = uint32(productIds & ((1 << 32) - 1));
            // otherwise it means the product is a spot.
            if (productId % 2 == 0) {
                (
                    int128 canSettle,
                    State memory state,
                    Balance memory balance
                ) = getSettlementState(productId, subaccount);

                state.availableSettle -= canSettle;
                balance.vQuoteBalance -= canSettle;

                totalSettled += canSettle;

                _setState(productId, state);
                _setBalanceAndUpdateBitmap(productId, subaccount, balance);
            }
            productIds >>= 32;
        }
        return totalSettled;
    }
```
