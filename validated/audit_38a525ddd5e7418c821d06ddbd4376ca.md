Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of actual swapper, allowing allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` parameter, which resolves to `msg.sender` of the pool's `swap()` call — the router, not the end user. When a pool admin allowlists `MetricOmmSimpleRouter` to enable periphery-based trading (a natural and expected admin action), every non-allowlisted address can bypass the curated pool's swap gate by routing through the router. The allowlist is silently nullified.

## Finding Description
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `sender` is the first argument passed by the pool: [1](#0-0) 

`MetricOmmPool.swap` passes `msg.sender` (the immediate caller) as `sender` to `_beforeSwap`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router `msg.sender` to the pool: [3](#0-2) 

Therefore, when a user routes through the router, the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. If the router is allowlisted, the check passes for any caller.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly ignores `sender` and checks `owner` — the actual economic actor: [4](#0-3) 

The two extensions are architecturally inconsistent: the deposit guard keys on the economic actor (`owner`); the swap guard keys on the immediate caller (`sender`/router). No existing guard in `SwapAllowlistExtension` checks the actual user identity when a router intermediates.

## Impact Explanation
Any pool deploying `SwapAllowlistExtension` for access control (KYC enforcement, LP-protection boundaries, regulatory compliance) has its entire swap allowlist nullified once the router is allowlisted. Non-allowlisted users can execute swaps in pools explicitly configured to exclude them, constituting a broken core pool access-control invariant. In pools where the allowlist enforces LP-protection or fee-tier boundaries, this directly exposes LP principal to unauthorized counterparties.

## Likelihood Explanation
The trigger is a standard, expected admin action: allowlisting the official periphery router so users can trade through the supported UI. Any pool that (a) configures `SwapAllowlistExtension` and (b) allowlists the router — a combination that is both natural and the documented supported flow — is fully exposed. No attacker privilege is required beyond calling the public router. The bypass is repeatable by any address on every such pool.

## Recommendation
Replace the `sender` check with the actual user identity, mirroring `DepositAllowlistExtension`'s pattern:

1. **Pass the real user through `extensionData`**: have the router encode `msg.sender` into `extensionData`; the extension decodes and checks it. This is the robust solution and mirrors how `owner` is threaded through `addLiquidity`.
2. **Check `recipient`** as a simpler fallback, if pool semantics guarantee `recipient` is the economic beneficiary (not safe for multi-hop routes where intermediate recipients are contracts).

The deposit allowlist's pattern — ignoring `sender` and checking the explicit economic-actor parameter — should be the model for the swap allowlist.

## Proof of Concept
1. Pool admin deploys pool with `SwapAllowlistExtension`; calls `setAllowedToSwap(pool, alice, true)` (only `alice` is allowlisted).
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable periphery access.
3. `bob` (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: bob, ...})`.
4. Router calls `pool.swap(recipient=bob, ...)` with `msg.sender = router`.
5. Pool calls `_beforeSwap(sender=router, recipient=bob, ...)`.
6. Extension evaluates `allowedSwapper[pool][router]` → `true` → no revert.
7. `bob`'s swap executes in the curated pool despite never being allowlisted.

Verification: `bob` calling `pool.swap()` directly correctly reverts (`allowedSwapper[pool][bob]` is `false`), confirming the bypass is router-specific and not a misconfiguration.
