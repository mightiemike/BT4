Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Enabling Complete Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which is the immediate `msg.sender` of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` resolves to the router contract address, not the originating user. A pool admin who allowlists the router to enable router-mediated swaps for their curated users simultaneously grants unrestricted swap access to every address that calls through the same router.

## Finding Description
`MetricOmmPool.swap` passes its own `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged into the encoded call to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

Every entry point in `MetricOmmSimpleRouter` calls `pool.swap(...)` directly, making the router the `msg.sender` in every case: [4](#0-3) [5](#0-4) 

The mapping resolves to `allowedSwapper[pool][router]`. Once the router is allowlisted, `allowedSwapper[pool][userB]` is never consulted for any user routing through the router. No existing guard in the extension, pool, or router corrects for this: the router does not encode `msg.sender` into `extensionData`, and the extension has no fallback mechanism to recover the originating user.

## Impact Explanation
A pool admin who deploys a curated pool with `SwapAllowlistExtension` and allowlists the router — the only way to let their allowlisted users benefit from slippage protection, multi-hop routing, or deadline enforcement — simultaneously opens the pool to every address on the network. Any non-allowlisted trader can call `router.exactInputSingle({pool: restrictedPool, ...})` and the extension will pass because `allowedSwapper[pool][router] == true`. This breaks the pool's curation invariant and exposes restricted liquidity to open-market trading at prices the pool admin did not intend to make public, constituting broken core pool functionality and potential unauthorized drain of LP assets.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical user-facing swap interface. Any pool admin who configures a swap allowlist and also wants their allowlisted users to use the router will naturally call `setAllowedToSwap(pool, router, true)`. The misconfiguration is non-obvious: the admin believes they are granting router access to their allowlisted users, not to the entire public. Exploitation requires only a standard `router.exactInputSingle` call from any address — no privileged access, no special tokens, no unusual setup. The trigger is repeatable and requires no front-running or timing.

## Recommendation
The extension framework must propagate the original end-user identity through hook arguments rather than the immediate `msg.sender` of `pool.swap()`. Two viable approaches:

1. **Router-side:** Have `MetricOmmSimpleRouter` encode the original `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check that address when present, falling back to `sender` when absent.
2. **Pool-side:** Add an explicit `originator` field to the swap call that the pool passes through to extensions, distinct from `sender` (the immediate caller).

Until fixed, pool admins must not allowlist the router address; allowlisted users must call `pool.swap()` directly.

## Proof of Concept
```
1. Pool admin deploys pool with SwapAllowlistExtension configured on beforeSwap.
2. Pool admin allowlists userA:
       swapExt.setAllowedToSwap(pool, userA, true)
3. Pool admin allowlists the router so userA can use it:
       swapExt.setAllowedToSwap(pool, router, true)
4. userB (not allowlisted) calls:
       router.exactInputSingle({pool: pool, recipient: userB, ...})
5. Router calls pool.swap(...) with msg.sender = router.
6. Pool calls _beforeSwap(sender=router, ...).
7. SwapAllowlistExtension checks allowedSwapper[pool][router] → true → no revert.
8. userB's swap executes on the restricted pool.

allowedSwapper[pool][userB] is never consulted. The allowlist is bypassed in full.
```
