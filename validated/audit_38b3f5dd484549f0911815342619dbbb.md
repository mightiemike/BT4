Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` gates on `owner` instead of `sender`, allowing allowlist bypass via `MetricOmmPoolLiquidityAdder` — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` discards the `sender` parameter and checks `owner` against the allowlist. Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts a fully caller-controlled `owner` with no restriction beyond `!= address(0)`, any unprivileged address can bypass the deposit allowlist by supplying an allowlisted address as `owner`. The actual depositor (Bob) pays the tokens while the allowlisted address (Alice) receives the LP position, breaking the pool admin's access-control invariant entirely.

## Finding Description
**Root cause — wrong identity checked in the hook:**

`DepositAllowlistExtension.beforeAddLiquidity` (L32–42) silently discards the first parameter (`sender`) and keys the allowlist lookup on `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

**How `sender` and `owner` are bound in the pool:**

`MetricOmmPool.addLiquidity` (L191) passes `msg.sender` as `sender` and the caller-supplied `owner` argument as `owner`:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

**How `owner` is caller-controlled through the LiquidityAdder:**

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (L56–68) accepts an explicit `owner` and passes it directly to the pool. The only validation is `_validateOwner` (L247–249), which only checks `owner != address(0)`:

```solidity
function addLiquidityExactShares(address pool, address owner, ...) external payable override {
    _validateOwner(owner);  // only checks != address(0)
    _validateDeltas(deltas);
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
}
```

**The bypass path:**

1. Bob (not allowlisted) calls `addLiquidityExactShares(pool, alice, ...)` with `owner = alice` (allowlisted).
2. LiquidityAdder calls `pool.addLiquidity(owner=alice, callbackData=KIND_PAY, ...)`.
3. Pool calls `_beforeAddLiquidity(sender=LiquidityAdder, owner=alice, ...)`.
4. Extension evaluates `allowedDepositor[pool][alice]` → `true` → hook passes.
5. Pool mints LP shares to Alice's position key.
6. Pool calls `LiquidityAdder.metricOmmModifyLiquidityCallback(...)`, which pulls tokens from `payer = msg.sender = Bob` (L162–177).

Bob pays the tokens; Alice receives the LP position. The deposit allowlist is fully bypassed.

The same pattern applies to `addLiquidityWeighted(address pool, address owner, ...)` (L88–116), which also accepts a caller-controlled `owner` with only `_validateOwner` validation.

## Impact Explanation
The deposit allowlist is the pool admin's mechanism to create restricted-access pools (e.g., private LP vaults, KYC-gated pools). With this bypass, any unprivileged address can deposit into a restricted pool by naming any allowlisted address as `owner`. This is a direct admin-boundary break: a factory-configured guard is bypassed by an unprivileged public path through the periphery. Unauthorized depositors can manipulate the pool's bin liquidity distribution, affecting oracle-anchored swap prices and LP share dilution for legitimate LPs. The pool admin's access-control invariant — that only allowlisted addresses can add liquidity — is completely broken.

## Likelihood Explanation
**High.** `MetricOmmPoolLiquidityAdder` is a public, permissionless periphery contract. Any address can call `addLiquidityExactShares` with an arbitrary `owner`. The only prerequisite is knowing one allowlisted address on the target pool, which is observable on-chain from `AllowedToDepositSet` events or by reading `allowedDepositor` storage directly. No special role, privileged access, or unusual token behavior is required beyond having sufficient token balance and approval.

## Recommendation
The `beforeAddLiquidity` hook must gate on the actual depositor identity. Two options:

1. **Check `sender` instead of `owner`** — gates the direct caller of `pool.addLiquidity`. When users go through the LiquidityAdder, `sender` is the LiquidityAdder address; the pool admin would allowlist the LiquidityAdder and rely on it to enforce its own caller checks:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

2. **Require `sender == owner`** in the extension, preventing delegated deposits entirely on allowlisted pools.

## Proof of Concept
```
Setup:
  - Pool deployed with DepositAllowlistExtension
  - Pool admin calls setAllowedToDeposit(pool, alice, true)
  - Bob is NOT allowlisted

Attack:
  1. Bob calls:
       LiquidityAdder.addLiquidityExactShares(
           pool,
           alice,   // allowlisted address, not Bob
           salt, deltas, max0, max1, extensionData
       )

  2. LiquidityAdder → pool.addLiquidity(owner=alice, callbackData=KIND_PAY, ...)

  3. Pool → _beforeAddLiquidity(sender=LiquidityAdder, owner=alice, ...)
       Extension: allowedDepositor[pool][alice] == true → PASSES

  4. Pool mints LP shares to positionKey(alice, salt, binIdx)

  5. Pool → LiquidityAdder.metricOmmModifyLiquidityCallback(amount0, amount1, KIND_PAY)
       LiquidityAdder pulls tokens from payer=Bob

Result:
  - Bob's tokens transferred to pool ✓
  - Alice holds LP shares she did not authorize ✓
  - Deposit allowlist bypassed: Bob deposited into a pool he is not allowlisted for ✓
```
