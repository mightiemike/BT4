### Title
Incorrect `LastKnownBonded=true` Initialization on Unbonded Validator Causes Illegitimate Bonus Reward Payout — (`x/tieredrewards/keeper/position.go`)

---

### Summary

`createDelegatedPosition` hardcodes `LastKnownBonded = true` for every new position regardless of the validator's actual bonded state. Because `validateNewPosition` imposes no bonded-status guard, an attacker can create a position on an unbonded validator. When that validator later re-bonds, `processEventsAndClaimBonus` seeds its segment walk with `bonded = true`, treats the entire unbonded gap as a bonded segment, and pays out bonus rewards from `RewardsPoolName` for time the validator was never bonded.

---

### Finding Description

**Root cause — hardcoded `true` in `createDelegatedPosition`:** [1](#0-0) 

```go
pos := types.NewPosition(id, owner, tier.Id, delAddr.String(), blockHeight, lastEventSeq, blockTime, true, blockTime)
```

The eighth argument (`lastKnownBonded`) is always `true`, independent of whether the target validator is currently bonded, unbonded, or unbonding.

**No bonded-status guard in `validateNewPosition`:** [2](#0-1) 

The function checks only vesting-account status, tier close-only flag, and minimum lock amount. There is no call to `stakingKeeper.GetValidator` and no rejection of unbonded validators.

**Segment walk seeds from `LastKnownBonded`:** [3](#0-2) 

```go
// Use the persisted bonded state from the last replay, not a hardcoded default.
bonded := pos.LastKnownBonded
segmentStart := pos.LastBonusAccrual
```

If `LastKnownBonded` is `true` but the validator was unbonded at position creation, the first segment is treated as bonded.

**Bonus computed before the bonded-state flip:** [4](#0-3) 

For each event the loop first pays bonus (if `bonded == true`), then updates `bonded` based on the event type. A BOND event therefore triggers payment for the preceding segment — which ran from position creation through the BOND event — entirely while the validator was unbonded.

**Defensive check only covers the tail segment:** [5](#0-4) 

```go
if bonded && val.IsBonded() { … }
```

This guard applies only to the segment after the last event. It does not protect segments inside the event loop.

---

### Impact Explanation

The attacker receives `bankKeeper.SendCoinsFromModuleToAccount(RewardsPoolName, owner, bonusCoins)` for a time window during which the validator was provably unbonded, directly draining `RewardsPoolName`. The amount scales with delegation size, tier `BonusApy`, and the length of the unbonded gap before re-bonding. Repeated across multiple positions or multiple unbond/rebond cycles the pool can be drained to zero, denying legitimate bonus rewards to all other position holders. [6](#0-5) 

---

### Likelihood Explanation

The attack requires only a standard `MsgLockTier` transaction targeting a validator that is currently in the unbonded set — a normal on-chain condition that occurs whenever a validator falls out of the active set. No privileged role, governance action, or key compromise is needed. The attacker simply waits for the validator to re-bond (or redelegates to a validator about to re-bond) and then claims rewards.

---

### Recommendation

In `createDelegatedPosition`, query the validator's actual bonded state and use it to initialize `LastKnownBonded`:

```go
val, err := k.stakingKeeper.GetValidator(ctx, valAddr)
if err != nil {
    return types.Position{}, err
}
lastKnownBonded := val.IsBonded()
pos := types.NewPosition(id, owner, tier.Id, delAddr.String(), blockHeight, lastEventSeq, blockTime, lastKnownBonded, blockTime)
```

Optionally, add a guard in `validateNewPosition` that rejects positions on unbonded validators entirely, which is the simpler and safer invariant. [7](#0-6) 

---

### Proof of Concept

```
1. Validator V is unbonded (UNBOND event at seq 5 already recorded).
2. Attacker calls MsgLockTier { validator: V, amount: X }.
   → validateNewPosition passes (no bonded check).
   → delegate(V, X) succeeds (Cosmos SDK allows delegation to unbonded validators).
   → createDelegatedPosition: lastEventSeq=5, LastKnownBonded=true.
   → setPosition increments PositionCountByValidator[V] to 1.
3. V re-bonds → AfterValidatorBonded fires → BOND event appended at seq 6
   with ReferenceCount=1 (attacker's position).
4. Attacker calls MsgClaimTierRewards.
   → processEventsAndClaimBonus:
       bonded = true  (from LastKnownBonded)
       segmentStart = position creation time T0
       events = [BOND @ seq 6, timestamp T1]
       Loop iteration (BOND event):
           bonded==true → computeSegmentBonus(T0, T1)  ← UNBONDED PERIOD
           totalBonus += bonus_for_unbonded_gap
           bonded = true (no change)
   → bankKeeper.SendCoinsFromModuleToAccount(RewardsPoolName, attacker, bonusCoins)
5. Attacker receives bonus rewards for the period [T0, T1] when V was unbonded.
```

func (k Keeper) validateNewPosition(ctx context.Context, owner string, amount math.Int, tier types.Tier) error {
	if err := k.validateNonVestingAccount(ctx, owner); err != nil {
		return err
	}

	if tier.IsCloseOnly() {
		return types.ErrTierIsCloseOnly
	}

	if !tier.MeetsMinLockRequirement(amount) {
		return types.ErrMinLockAmountNotMet

	return nil
