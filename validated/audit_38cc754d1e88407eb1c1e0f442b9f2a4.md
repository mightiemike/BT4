Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the Originating User, Allowing Any User to Bypass the Swap Allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `sender` is the pool's `msg.sender` — the direct caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the allowlist check resolves to `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. A pool admin who allowlists the router to enable router-mediated swaps inadvertently grants every user — including explicitly blocked addresses — the ability to bypass the allowlist.

## Finding Description

In `MetricOmmPool.swap()`, the pool calls `_beforeSwap(msg.sender, ...)`, passing its own `msg.sender` as the `sender` argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` then ABI-encodes and forwards this `sender` value directly to the extension hook: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` receives this value as its first argument and checks it against the allowlist: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) calls `IMetricOmmPoolActions(params.pool).swap(...)`, the router is the pool's `msg.sender`: [4](#0-3) 

The allowlist check therefore becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. Two broken outcomes follow:

1. **Allowlist bypass**: If the pool admin allowlists the router (required for any router-mediated swap to work), every user — including explicitly blocked addresses — can bypass the allowlist by routing through `MetricOmmSimpleRouter`.
2. **Broken functionality**: If the pool admin does not allowlist the router, legitimate allowlisted users cannot use the router at all.

The same structural flaw exists in `DepositAllowlistExtension.beforeAddLiquidity`, which checks `owner` (the position beneficiary) rather than `msg.sender` (the payer/operator), though the liquidity path's operator pattern makes the impact scenario different: [5](#0-4) 

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of counterparties (e.g., KYC'd addresses, protocol-owned bots, or whitelisted market makers) loses that protection entirely for any user who routes through `MetricOmmSimpleRouter`. The disallowed user can execute swaps at oracle-derived prices, extracting value in ways the pool admin explicitly intended to prevent. This constitutes a direct bypass of an admin-configured access control boundary, resulting in potential loss of LP principal and a curation failure on a production pool. This meets the "Admin-boundary break" and "direct loss of user principal" impact criteria.

## Likelihood Explanation

- `MetricOmmSimpleRouter` is the primary supported periphery entrypoint for swaps; most users will route through it.
- The pool admin must allowlist the router to make router-mediated swaps work at all, which is the expected operational setup.
- No special privileges or malicious setup are required — any user can call the router.
- The bypass is reachable on every production pool that uses `SwapAllowlistExtension` with the router allowlisted.

## Recommendation

The pool must pass the **originating user** as `sender` to the extension, not the intermediate router. Two approaches:

1. **Pool-level fix**: In `MetricOmmPool.swap`, accept an explicit `originatingSender` parameter from the caller and pass it through to `_beforeSwap`. The router would forward `msg.sender` (the actual user) as this parameter.
2. **Extension-level fix**: The extension should not trust the `sender` argument when the pool's `msg.sender` is a known router. The pool should always pass the original transaction initiator (`tx.origin` is not recommended; prefer an explicit parameter).

The `DepositAllowlistExtension` should be audited for the same pattern — verifying that `owner` (not `sender`/payer) is the economically relevant actor the allowlist is intended to gate.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured
  - Pool admin calls setAllowedToSwap(pool, router, true)   // to enable router swaps
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  1. attacker calls MetricOmmSimpleRouter.exactInputSingle(pool, tokenIn, amountIn, ...)
  2. Router calls pool.swap(recipient, zeroForOne, amount, ...)  [router is msg.sender]
  3. Pool calls _beforeSwap(msg.sender=router, ...)
  4. ExtensionCalling encodes sender=router and calls extension.beforeSwap(sender=router, ...)
  5. Extension checks: allowedSwapper[pool][router] == true  → passes
  6. Swap executes at oracle price for attacker
  7. Attacker receives tokenOut despite being explicitly excluded from the allowlist

Expected: revert NotAllowedToSwap
Actual:   swap succeeds
```
