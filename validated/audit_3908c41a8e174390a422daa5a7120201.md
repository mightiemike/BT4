Audit Report

## Title
SwapAllowlistExtension Bypass via Router: `sender` Is Router Address, Not Actual Swapper - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap` passes `msg.sender` (the immediate caller) as the `sender` argument to both `_beforeSwap` and `_afterSwap` extension hooks. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the router contract, not the original user. `SwapAllowlistExtension.beforeSwap` gates access by checking `allowedSwapper[pool][sender]`, so it checks the router's allowlist status rather than the actual swapper's. Any unprivileged user can bypass the per-user allowlist by routing through the public router.

## Finding Description

**Root cause ‚Äî pool passes `msg.sender` as `sender`:**

In `MetricOmmPool.swap` (lines 230‚Äì240 and 281‚Äì295), the pool calls:
```solidity
_beforeSwap(msg.sender, recipient, ...);
// ...
_afterSwap(msg.sender, recipient, ...);
```
`ExtensionCalling._beforeSwap` (lines 160‚Äì176) encodes this value directly into the `IMetricOmmExtensions.beforeSwap` call as the `sender` argument.

**Allowlist checks `sender`, which is the router:**

`SwapAllowlistExtension.beforeSwap` (lines 31‚Äì41):
```solidity
function beforeSwap(address sender, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```
Here `msg.sender` is the pool and `sender` is the address the pool forwarded ‚Äî the router.

**Router call path:**

`MetricOmmSimpleRouter.exactInputSingle` (lines 71‚Äì80) calls `pool.swap(params.recipient, ...)` directly. The pool sees `msg.sender = router`. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

**Exploit flow:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists only specific users (e.g., KYC'd addresses).
2. Pool admin also allowlists the router address so that allowlisted users can swap through it (or the router is allowlisted by default for usability).
3. An unprivileged, non-allowlisted user calls `router.exactInputSingle(...)`.
4. The router calls `pool.swap(...)` ‚Äî pool sees `msg.sender = router`.
5. Extension checks `allowedSwapper[pool][router]` ‚Üí `true` ‚Üí swap proceeds.
6. Non-allowlisted user successfully swaps, bypassing the per-user gate entirely.

**Existing guards are insufficient:** The `onlyPool` modifier on the extension only verifies the caller is a known pool; it does not verify that the `sender` argument reflects the true originating user. There is no mechanism in the pool or router to propagate the original `msg.sender` through to the extension.

## Impact Explanation
The `SwapAllowlistExtension` is a production periphery extension whose sole purpose is to restrict swap access to approved addresses per pool. The bypass completely nullifies this control: any unprivileged user can swap on an allowlisted pool by routing through the public `MetricOmmSimpleRouter`. This constitutes broken core pool functionality (the allowlist gate) and an admin-boundary break where an unprivileged path circumvents a pool-admin-configured access control. Pools relying on this extension for regulatory compliance or risk management are fully exposed.

## Likelihood Explanation
The `MetricOmmSimpleRouter` is a public, permissionless contract. Any user can call it. The only precondition is that the router address is allowlisted on the target pool (which is necessary for any router-mediated swap to work at all). This is a trivially repeatable, zero-cost bypass requiring no special privileges, no flash loans, and no oracle manipulation.

## Recommendation
The pool must propagate the true originating user to the extension. Two options:

1. **Pass the original caller through the router:** Add an optional `address swapper` parameter to `pool.swap` that the router populates with `msg.sender`, and use that value (instead of `msg.sender`) as the `sender` forwarded to extensions.
2. **Check `tx.origin` in the extension (not recommended for general use):** Only viable in narrow, non-contract-caller contexts and introduces its own risks.

The cleanest fix is option 1: the pool's `swap` signature accepts an explicit `sender` address, the router passes `msg.sender` (the true user), and the pool validates that `msg.sender` is either the declared sender or a factory-registered router authorized to act on behalf of users.

## Proof of Concept

```solidity
// Foundry test sketch
function test_swapAllowlist_bypass_via_router() public {
    // Setup: pool with SwapAllowlistExtension, only `allowedUser` is allowlisted
    // Router is allowlisted so allowedUser can use it
    allowlistExtension.setAllowedToSwap(address(pool), address(router), true);
    // allowedUser is NOT individually allowlisted

    address attacker = makeAddr("attacker");
    deal(address(token0), attacker, 10_000);
    vm.startPrank(attacker);
    token0.approve(address(router), type(uint256).max);

    // Direct pool swap reverts ‚Äî attacker is not allowlisted
    vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
    pool.swap(attacker, true, 1000, 0, "", "");

    // Router swap succeeds ‚Äî router is allowlisted, sender=router bypasses user check
    uint256 out = router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool), tokenIn: address(token0), tokenOut: address(token1),
        zeroForOne: true, amountIn: 1000, amountOutMinimum: 0,
        recipient: attacker, deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
    }));
    assertGt(out, 0); // attacker successfully swapped despite not being allowlisted
}
```
