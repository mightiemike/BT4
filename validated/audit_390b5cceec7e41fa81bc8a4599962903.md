### Title
Deprecated `verify_transaction_inclusion` Remains Callable and Accepts Internal Merkle Node as Valid Transaction, Enabling Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function is still reachable by any unprivileged NEAR caller. It does not distinguish between leaf-node transaction hashes and internal Merkle tree node hashes. An attacker can supply an internal node hash as `tx_id` with a crafted proof path, causing the function to return `true` for a transaction that does not exist on-chain. Any recipient contract consuming this result to authorize fund releases or cross-chain actions is deceived.

---

### Finding Description

`verify_transaction_inclusion` at `contract/src/lib.rs:288` delegates Merkle root computation to `compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs:34`. That function treats any 32-byte value as a valid leaf — it does not enforce that the input is a leaf-level transaction hash:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    ...
    for proof_hash in merkle_proof {
        ...
        current_hash = compute_hash(...);
        ...
    }
    current_hash
}
``` [1](#0-0) 

This is the well-known 64-byte transaction Merkle proof forgery (https://www.bitmex.com/blog/64-Byte-Transactions). An internal node at depth D in the Merkle tree is itself a valid 32-byte hash. An attacker can present it as `tx_id` with a D-step proof path to the root, and `compute_root_from_merkle_proof` will produce the correct block Merkle root, causing `verify_transaction_inclusion` to return `true`.

The function's own docstring acknowledges this:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [2](#0-1) 

The `#[deprecated]` Rust attribute only emits a compiler warning. It does **not** prevent the function from being invoked via a NEAR contract call. The function carries only `#[pause]` — no `#[trusted_relayer]` guard, no access-control role check — so any unprivileged NEAR account can call it while the contract is unpaused:

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
``` [3](#0-2) 

`verify_transaction_inclusion_v2` fixes this by first verifying a coinbase Merkle proof (anchoring the tree structure), then calling v1 internally. But because v1 is still directly callable, an attacker can bypass the coinbase anchor check entirely by calling v1 directly. [4](#0-3) 

The `ProofArgs` struct accepted by v1 is fully attacker-controlled — `tx_id`, `tx_block_blockhash`, `tx_index`, `merkle_proof`, and `confirmations` are all caller-supplied with no server-side binding: [5](#0-4) 

---

### Impact Explanation

Any recipient contract or bridge application that calls `verify_transaction_inclusion` (v1) to authorize fund releases, unlock cross-chain assets, or perform other privileged actions will accept a forged proof as genuine. The attacker does not need to mine a block or submit any headers — they only need to know the Merkle tree structure of any block already accepted by the light client (publicly available from Bitcoin's blockchain). The corrupted canonical value is the verification result itself: a `true` return for a transaction that was never broadcast or confirmed on Bitcoin.

This is the direct analog to the oracle price manipulation class: the verification function is the trusted data source, and an unprivileged actor can manipulate its output by supplying adversarial inputs, causing downstream consumers to act on a false result.

---

### Likelihood Explanation

- No privileged access is required.
- The attacker only needs to call a public NEAR contract function.
- The Merkle tree structure of any accepted Bitcoin block is publicly available.
- The attack requires no mining, no relayer compromise, and no social engineering.
- The deprecated function is still fully operational.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the contract entirely, or replace its body with an unconditional `env::panic_str("use verify_transaction_inclusion_v2")` to prevent any caller from reaching the vulnerable code path. The `#[deprecated]` attribute alone is insufficient because it does not block on-chain invocation.

---

### Proof of Concept

1. Let block `B` be any block accepted by the light client with Merkle root `R` and at least two transactions `T0`, `T1`.
2. Compute internal node `N = SHA256d(T0 || T1)` — this is the parent of `T0` and `T1` at depth 1.
3. Identify the sibling of `N` at depth 1, call it `S`.
4. Construct the remaining proof path `[S, P2, P3, ...]` from depth 1 up to the root (length = tree depth − 1).
5. Call `verify_transaction_inclusion` with:
   - `tx_id` = `N` (an internal node, not a real transaction)
   - `tx_block_blockhash` = `B`
   - `tx_index` = `0` (even, so `N` is treated as the left child)
   - `merkle_proof` = `[S, P2, P3, ...]`
   - `confirmations` = `1`
6. `compute_root_from_merkle_proof(N, 0, [S, P2, ...])` computes `SHA256d(N || S)` = the actual depth-0 root = `R`.
7. The comparison `computed_root == header.block_header.merkle_root` passes.
8. The function returns `true` — falsely confirming `N` as a valid transaction in block `B`. [6](#0-5) [1](#0-0)

    let mut current_hash = transaction_hash;

            current_hash = compute_hash(&current_hash, proof_hash);

            current_hash = compute_hash(proof_hash, &current_hash);
        }

    }
