### Title
Unconditional Trust of Attacker-Controlled `_cached_sha256_treehash` Attribute Enables Puzzle Hash Forgery - (File: `wheel/python/clvm_rs/tree_hash.py`)

### Summary
`Treehasher.sha256_treehash()` reads `_cached_sha256_treehash` from any Python object implementing `CLVMStorage` and uses it as the authoritative tree hash without any verification against the object's actual content. An attacker who supplies a Python `CLVMStorage`-conforming object with a forged `_cached_sha256_treehash` attribute causes the function to return an incorrect hash, corrupting puzzle hash computation and downstream coin-ID derivation.

### Finding Description

In `wheel/python/clvm_rs/tree_hash.py`, the inner function `handle_obj` inside `Treehasher.sha256_treehash()` performs the following check at the very start of processing each node:

```python
r = getattr(obj, "_cached_sha256_treehash", None)
if r is not None:
    self.cache_hits += 1
    hash_stack.append(r)
    return
``` [1](#0-0) 

If the attribute is present and non-`None`, the function immediately appends it to the hash stack and returns — **no verification that the cached value matches the actual content of the atom or pair is performed**. The `CLVMStorage` protocol is a structural Python protocol: any object with `.atom` and `.pair` attributes qualifies. There is no enforcement that `_cached_sha256_treehash` was ever computed by the `Treehasher` itself.

This is the direct analog to reading `globalState()` spot price: the code reads an unverified, externally settable attribute and treats it as ground truth, bypassing the actual computation (`shatree_atom` / `shatree_pair`).

The corrupted hash then propagates upward through `handle_pair`:

```python
def handle_pair(...):
    p0 = hash_stack.pop()
    p1 = hash_stack.pop()
    r = shatree_pair(p0, p1)
    hash_stack.append(r)
    obj = obj_stack.pop()
    try:
        setattr(obj, "_cached_sha256_treehash", r)
    except AttributeError:
        pass
``` [2](#0-1) 

A forged leaf hash propagates into every ancestor pair hash, corrupting the entire subtree hash.

`sha256_treehash` is the function called by `Program.tree_hash()`:

```python
def tree_hash(self) -> bytes:
    if self._cached_sha256_treehash is None:
        self._cached_sha256_treehash = sha256_treehash(self._unwrapped)
    return self._cached_sha256_treehash
``` [3](#0-2) 

`self._unwrapped` is the underlying `CLVMStorage` object. If that object carries a forged `_cached_sha256_treehash`, the wrong hash is stored and returned.

### Impact Explanation

**Impact: High.**

Puzzle hashes are the primary identity of Chia coins (coin ID = `sha256(parent_coin_id || puzzle_hash || amount)`). A forged puzzle hash returned by `Program.tree_hash()` causes:

1. **Wrong coin ID derivation** — a wallet computing the address for a puzzle receives an incorrect coin ID, potentially sending funds to an unspendable or attacker-controlled coin.
2. **Consensus divergence** — a validator or mempool that accepts a `CLVMStorage` object from an external source (e.g., a library that wraps user-supplied puzzle bytes in a Python object before hashing) will compute a different puzzle hash than the on-chain canonical hash, breaking spend validation.
3. **Curry/puzzle-hash mismatch** — `Program.curry_hash()` calls `self.tree_hash()` internally; a forged hash silently corrupts curried puzzle hashes used in coin announcements and spend bundles. [4](#0-3) 

### Likelihood Explanation

**Likelihood: Medium.**

The `CLVMStorage` protocol is a public Python structural protocol — any object with `.atom` and `.pair` is accepted. The attack entry points are:

- `sha256_treehash(obj)` called directly with an attacker-supplied object.
- `Program.to(obj)` followed by `.tree_hash()`, where `obj` is a `CLVMStorage`-conforming Python object with a pre-set `_cached_sha256_treehash`.
- Any library function that accepts a `CLVMStorage` object from an external caller (e.g., a wallet SDK accepting a user-provided puzzle object).

The `CLVMStorage` protocol is explicitly designed to be implemented by third-party Python objects (the `SimpleStorage` and `Uncachable` test classes demonstrate this). The `_cached_sha256_treehash` attribute is a plain Python instance attribute with no access control. No deserialization from CLVM bytes is required — the attacker only needs to supply a conforming Python object. [5](#0-4) 

### Recommendation

Remove the unconditional trust of `_cached_sha256_treehash` from externally supplied objects. Options:

1. **Only trust the cache on objects created internally** — check that the object is an instance of a trusted class (e.g., `LazyNode`, `CLVMTree`, or `Program`) before reading the cached attribute.
2. **Never read `_cached_sha256_treehash` from the root object** — only write it after computing the hash, never read it as a shortcut for externally supplied objects.
3. **Validate the cached value** — after reading `_cached_sha256_treehash`, verify it matches the computed hash of the atom content (for atoms) before trusting it.

### Proof of Concept

```python
from clvm_rs.tree_hash import sha256_treehash, shatree_atom

# Correct hash of b"legitimate_puzzle"
correct_hash = shatree_atom(b"legitimate_puzzle")

# Attacker-controlled CLVMStorage object with forged cached hash
class ForgedStorage:
    atom = b"legitimate_puzzle"
    pair = None
    _cached_sha256_treehash = b"\x00" * 32  # forged

result = sha256_treehash(ForgedStorage())

# result is b"\x00" * 32, NOT correct_hash
assert result == b"\x00" * 32          # forged hash returned
assert result != correct_hash           # correct hash bypassed

# Downstream: Program.tree_hash() propagates the forgery
from clvm_rs.program import Program
p = Program.to(ForgedStorage())
# p.tree_hash() returns b"\x00" * 32 — wrong coin ID will be derived
```
