### Title
Fork Difficulty Calculated Using Mainchain Ancestor Instead of Fork-Chain Ancestor at Adjustment Boundary — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

When validating a fork block at a difficulty adjustment boundary, `get_next_work_required` calls `blocks_getter.get_header_by_height(first_block_height)` to retrieve the timestamp of the block at the start of the difficulty period. However, `get_header_by_height` always reads from the **mainchain** index (`mainchain_height_to_header`), not from the fork chain's ancestry. If the fork diverged before the start of the difficulty period, the mainchain block at that height may carry a different timestamp than the fork's actual ancestor, producing an incorrect difficulty target for the fork block. The Dogecoin implementation even contains a self-acknowledged TODO noting this exact concern.

---

### Finding Description

`BlocksGetter::get_header_by_height` is implemented in `lib.rs` as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

This unconditionally resolves a height to the **mainchain** block at that height. It is correct for mainchain submissions, but wrong when called during fork-block difficulty validation.

In all three chain implementations, when a fork block lands on a difficulty adjustment boundary, the code computes `first_block_height` and calls `get_header_by_height` to obtain the timestamp of the period's opening block:

**Bitcoin** (`bitcoin.rs` line 81):
```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [2](#0-1) 

**Litecoin** (`litecoin.rs` line 88):
```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [3](#0-2) 

**Dogecoin** (`dogecoin.rs` lines 292–295), with the developers' own TODO:
```rust
// TODO: check if it is correct to get block header by height from mainchain
//       without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [4](#0-3) 

In all three cases, if the fork diverged **before** `first_block_height`, the mainchain block at that height is a different block than the fork's ancestor at the same height. The timestamp difference corrupts `actual_time_taken` (or `modulated_timespan` for Dogecoin), which directly controls the computed difficulty target.

The `get_prev_header` path used for all other ancestor lookups correctly follows `prev_block_hash` links through `headers_pool` and is unaffected; only the height-indexed lookup is broken. [5](#0-4) 

---

### Impact Explanation

If the mainchain block at `first_block_height` has an **earlier** timestamp than the fork's true ancestor at that height, `actual_time_taken` is inflated, the computed target is easier (larger), and the required `bits` value for the fork block is lower than it should be. The attacker:

1. Observes the mainchain block at `first_block_height` and pre-computes the incorrectly easy `expected_bits`.
2. Mines fork blocks satisfying only the easier target, passing both the `expected_bits == block_header.bits` check and the PoW hash check.
3. Accumulates chainwork on the fork. Because the attacker mines with less real work per block than the mainchain, they can build a longer fork chain with less total computation.
4. Once `current_header.chain_work > total_main_chain_chainwork`, `reorg_chain` is triggered, replacing the canonical chain with the attacker's fork. [6](#0-5) 

The corrupted canonical chain then serves as the ground truth for `verify_transaction_inclusion` and `verify_transaction_inclusion_v2`, meaning any downstream NEAR contract relying on SPV proofs can be fed false inclusion results.

---

### Likelihood Explanation

**Dogecoin (highest likelihood):** With `new_difficulty_protocol` active (height ≥ 145 000), `difficulty_adjustment_interval = 1`, so every block is a retarget and `height_first = prev_block_height - 1`. A fork only needs to diverge **3 blocks** before the current tip for the mainchain block at `height_first` to differ from the fork's ancestor. This is trivially achievable by any relayer submitting adversarial headers. [7](#0-6) 

**Bitcoin / Litecoin (lower but non-negligible likelihood):** The fork must have diverged before the start of the current difficulty period (up to 2016 blocks back for Bitcoin, 2016 for Litecoin). A fork that has been building for two full difficulty periods (≥ 4032 blocks) can exploit this. This requires sustained effort but is the exact threat model a light client is designed to resist. [8](#0-7) 

---

### Recommendation

Replace the height-indexed lookup with an ancestor traversal that follows `prev_block_hash` links from the fork tip backward to `first_block_height`. This mirrors how `get_prev_header` already correctly walks fork ancestry. A helper that walks `N` steps back from a known `ExtendedHeader` using `get_prev_header` in a loop would resolve the issue for all three chain implementations without requiring any new state.

---

### Proof of Concept

**Dogecoin scenario (3-block fork, new difficulty protocol):**

1. Mainchain tip is at height H (H ≥ 145 000). Mainchain block at height H−2 has timestamp `T_main`.
2. Attacker submits a fork block at height H−2 with timestamp `T_fork < T_main` (earlier), branching from the mainchain at height H−3. This block is stored in `headers_pool` but not in `mainchain_height_to_header`.
3. Attacker submits a fork block at height H−1 (prev = fork block at H−2). `get_prev_header` correctly returns the fork block at H−2.
4. Attacker submits a fork block at height H (the retarget block). `get_next_work_required` is called:
   - `height_first = H − 2`
   - `get_header_by_height(H−2)` returns the **mainchain** block at H−2 with timestamp `T_main`
   - `modulated_timespan` is computed using `T_main` instead of `T_fork`
   - Since `T_main > T_fork`, the computed timespan is larger → target is easier → `expected_bits` is lower than correct
5. Attacker's fork block at height H carries the easy `bits` value and a PoW hash satisfying only the easy target. Both checks pass.
6. If the fork's cumulative `chain_work` exceeds the mainchain's, `reorg_chain` executes and the attacker's fork becomes canonical.
7. All subsequent `verify_transaction_inclusion` calls resolve against the attacker-controlled chain. [9](#0-8) [10](#0-9)

```rust
fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
