### Title
Second-Preimage (Internal-Node) Merkle Forgery in `verify_transaction_inclusion` — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` applies no leaf-vs-internal-node domain separation. Any NEAR caller can pass an internal Merkle tree node hash as `tx_id` together with a crafted shorter proof that still reconstructs the stored `merkle_root`, causing `verify_transaction_inclusion` to return `true` for a transaction that was never broadcast or mined.

---

### Finding Description

`compute_root_from_merkle_proof` iterates over the supplied proof elements and hashes them pairwise with the running hash, starting from whatever value is passed as `transaction_hash`: [1](#0-0) 

There is no tag, prefix, or counter that distinguishes a leaf hash from an internal node hash. The function is purely positional.

`verify_transaction_inclusion` (v1) passes the caller-supplied `args.tx_id` directly into this function and compares the result to the block header's stored `merkle_root`: [2](#0-1) 

No check is performed to confirm that `tx_id` is a leaf-level transaction hash. The function is public, carries only a `#[pause]` gate (not paused by default), and has no role restriction: [3](#0-2) 

**Concrete attack for a 4-tx block (depth-2 tree):**

```
Leaves:  tx0  tx1  tx2  tx3
Level 1: N01=hash(tx0,tx1)   N23=hash(tx2,tx3)
Root:    R  =hash(N01,N23)
```

Attacker submits:
- `tx_id     = N01`   ← internal node, not a real transaction
- `tx_index  = 0`
- `merkle_proof = [N23]`   ← one element instead of two

`compute_root_from_merkle_proof(N01, 0, [N23])` computes `hash(N01, N23) = R`, which equals the block's stored `merkle_root`. The function returns `true`.

The codebase itself acknowledges this in the developer documentation: [4](#0-3) 

and in the function's own doc comment: [5](#0-4) 

**The `verify_transaction_inclusion_v2` mitigation is also broken.**

`verify_transaction_inclusion_v2` adds a coinbase proof check intended to enforce proof depth: [6](#0-5) 

The check only requires `compute_root_from_merkle_proof(coinbase_tx_id, 0, coinbase_merkle_proof) == merkle_root`. It does **not** verify that `coinbase_tx_id` is the actual coinbase transaction hash. An attacker can pass the same internal node as both `tx_id` and `coinbase_tx_id` with the same 1-element proof:

- `coinbase_tx_id = N01`, `coinbase_merkle_proof = [N23]` → check passes
- `tx_id = N01`, `merkle_proof = [N23]` → inner v1 call passes
- Length equality `1 == 1` → passes [7](#0-6) 

Both v1 and v2 are exploitable.

---

### Impact Explanation

Any bridge, settlement, or cross-chain contract that calls `verify_transaction_inclusion` (or `verify_transaction_inclusion_v2`) and trusts the boolean result can be deceived into accepting a fabricated transaction as mined. An attacker who knows the internal node hashes of any canonical block (publicly derivable from the block's transaction list) can forge an inclusion proof for a transaction that does not exist, enabling false payment claims, double-spend proofs, or fraudulent bridge withdrawals.

---

### Likelihood Explanation

- No privileged role is required; any NEAR account can call `verify_transaction_inclusion` directly.
- Internal node hashes are fully public: they are derivable from the block's transaction list, which is available from any Bitcoin full node or block explorer.
- The block must be canonical with sufficient confirmations (already satisfied by the precondition), but the attacker does not need to mine or modify any block.
- The attack requires only constructing a valid `ProofArgs` struct with a crafted `tx_id` and a one-level-shorter `merkle_proof`.

---

### Recommendation

1. **Remove `verify_transaction_inclusion` (v1) entirely** from the deployed contract. It is already deprecated; leaving it callable is unnecessary risk.

2. **Fix `verify_transaction_inclusion_v2`** by verifying that `coinbase_tx_id` is the actual coinbase transaction hash. This requires the caller to supply the raw coinbase transaction bytes and computing `double_sha256(raw_coinbase)` on-chain, then comparing to `coinbase_tx_id` before using it in the proof check. Without this, the coinbase-depth mitigation is trivially bypassed.

3. **Enforce proof depth** by requiring `merkle_proof.len() == ceil(log2(tx_count))`. This requires storing the transaction count in the block header state, which Bitcoin headers do not include natively — but it can be supplied by the relayer and validated against the coinbase commitment.

### Proof of Concept

```rust
// In merkle-tools/src/lib.rs tests (pure unit test, no NEAR runtime needed)
#[test]
fn test_internal_node_forgery() {
    // Build a 4-tx tree
    let tx0 = H256([0x01; 32]);
    let tx1 = H256([0x02; 32]);
    let tx2 = H256([0x03; 32]);
    let tx3 = H256([0x04; 32]);

    // Level-1 internal nodes
    let n01 = compute_hash(&tx0, &tx1);  // hash(tx0 || tx1)
    let n23 = compute_hash(&tx2, &tx3);  // hash(tx2 || tx3)

    // Real root
    let real_root = compute_hash(&n01, &n23);

    // Attacker passes internal node n01 as "tx_id" with a 1-element proof [n23]
    let forged_root = compute_root_from_merkle_proof(
        n01.clone(),
        0,
        &vec![n23.clone()],
    );

    // Invariant violated: forged_root == real_root, but n01 is not a leaf tx
    assert_eq!(forged_root, real_root,
        "Internal node forgery succeeds: verify_transaction_inclusion returns true for a non-existent tx");
}
```

This test passes on the unmodified codebase, directly proving the invariant violation. [8](#0-7) [9](#0-8)

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```

```rust
fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
```

```rust
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
```
