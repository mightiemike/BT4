Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Originating User, Allowing Any Caller to Bypass the Swap Allowlist - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded by the pool, which equals `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. A pool admin who allowlists the router to enable standard periphery UX inadvertently opens the gate to every unpermissioned user, because the router is a public, permissionless contract with no caller restrictions.

## Finding Description

**Root cause:** `MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
  msg.sender,   // router address when called via router
  ...
```

`ExtensionCalling._beforeSwap` forwards this verbatim to the extension:

```solidity
// ExtensionCalling.sol L162-165
abi.encodeCall(
  IMetricOmmExtensions.beforeSwap,
  (sender, ...)   // sender = router
)
```

`SwapAllowlistExtension.beforeSwap` then checks the router address against the allowlist:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
// msg.sender = pool, sender = router ŌĆö the actual user is never checked
```

`MetricOmmSimpleRouter` is a public, permissionless contract. Any EOA can call `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` with no caller restriction:

```solidity
// MetricOmmSimpleRouter.sol L67
function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
```

**Exploit path:**
1. Pool admin deploys pool with `SwapAllowlistExtension` as `beforeSwap` hook.
2. Admin calls `swapExtension.setAllowedToSwap(pool, address(router), true)` ŌĆö the necessary step to allow legitimate users to trade via the router.
3. Attacker (any EOA not in `allowedSwapper[pool]`) calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(...)` with `msg.sender = router`.
5. Extension checks `allowedSwapper[pool][router] == true` ŌåÆ passes.
6. Swap executes; attacker receives output tokens.

The originating user's address is never checked at any point in the call chain. The router stores `msg.sender` only in transient storage for the payment callback (`_setNextCallbackContext`), but this is never exposed to or read by the extension.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., KYC-verified addresses, whitelisted market makers, or institutional participants) is completely unprotected once the router is allowlisted. Any address can execute swaps of arbitrary size against the pool's liquidity at oracle-anchored prices without the pool admin's consent. This constitutes a direct bypass of an admin-configured access control boundary by an unprivileged caller, resulting in unauthorized access to pool liquidity and potential direct loss of LP value. The severity is High.

## Likelihood Explanation

The scenario is highly likely in practice. Pool admins who want to support the standard periphery UX must allowlist the router ŌĆö there is no other supported path for router-mediated swaps. The bypass requires no special privileges, no custom contract, and no unusual token behavior. Any EOA can call `exactInputSingle` directly. The mismatch is invisible from the admin's perspective: `isAllowedToSwap(pool, router)` returns `true` as intended, but the invariant "only allowlisted users can swap" is silently broken for all router-mediated swaps.

## Recommendation

The extension must check the originating user, not the immediate pool caller. The correct fix is to have `MetricOmmSimpleRouter` encode `msg.sender` into `extensionData` before calling `pool.swap`, and have `SwapAllowlistExtension.beforeSwap` decode and check that address when `sender` is a known intermediary. A more robust fix is to add an `originalSender` slot to the router's transient context (analogous to the existing payer slot set via `_setNextCallbackContext`) and expose it via a view function that the extension can call back into during `beforeSwap`. Alternatively, the pool could expose a separate `originalSender` field that the router sets in transient storage before calling `swap`.
