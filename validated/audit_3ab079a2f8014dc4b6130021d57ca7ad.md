### Title
`run_serialized_chia_program` Uses `node_from_bytes` Instead of `node_from_bytes_backrefs`, Silently Rejecting Valid Backreference-Compressed Programs — (File: wheel/src/api.rs)

---

### Summary

The primary Python-callable execution entry point `run_serialized_chia_program` in `wheel/src/api.rs` deserializes both the `program` and `args` blobs using `node_from_bytes` — the legacy classic-format-only deserializer. This means any caller supplying a backreference-compressed CLVM blob (a format the network already supports) receives a hard deserialization error. The sibling function `deser_auto` in the same file correctly uses `node_from_bytes_backrefs` as its fallback for all non-2026 blobs, demonstrating the correct approach. The structural mismatch is a direct analog of the external report: a "from" variant that imposes an extra precondition (classic-only format) is used where a more general variant (`node_from_bytes_backrefs`, which is a strict superset) would succeed universally.

---

### Finding Description

In `wheel/src/api.rs`, `run_serialized_chia_program` (the sole `#[pyfunction]` that actually executes a CLVM program) deserializes its inputs as follows:

```rust
// wheel/src/api.rs  lines 55-56
let program = node_from_bytes(&mut allocator, program).map_err(eval_to_py)?;
let args    = node_from_bytes(&mut allocator, args).map_err(eval_to_py)?;
```

`node_from_bytes` (exported from `src/serde/de.rs`) only understands the classic CLVM wire format. It has no back-reference decoder; feeding it a blob that uses the back-reference compression scheme causes an immediate parse error.

`node_from_bytes_backrefs` (exported from `src/serde/de_br.rs`) is a strict superset: it handles both the classic format and the back-reference-compressed format. The same file already imports both:

```rust
// wheel/src/api.rs  lines 15-17
use clvmr::serde::{
    ParsedTriple, node_from_bytes, node_from_bytes_backrefs, …
};
```

The `deser_auto` convenience function in the same file (lines 148–157) correctly uses `node_from_bytes_backrefs` as the fallback for any blob that does not carry the `serde_2026` magic prefix:

```rust
// wheel/src/api.rs  lines 153-154
node_from_bytes_backrefs(&mut a, blob).map_err(eval_to_py)?
```

The comment on `deser_auto` explicitly states it is intended for "consensus-aware callers", confirming that back-reference-compressed blobs are a valid, network-accepted format. `run_serialized_chia_program` is the only function that actually runs programs, yet it uses the narrower deserializer.

---

### Impact Explanation

An attacker (or any ordinary Chia node) can submit a CLVM spend bundle whose generator or solution is serialized with back-references. When the Python layer calls `run_serialized_chia_program` with that blob, `node_from_bytes` returns a parse error and the program is rejected outright. If the reference Python implementation (or another node using `deser_auto` / `node_from_bytes_backrefs`) accepts the same blob, the two implementations diverge on validity — a consensus split. Even without a deliberate attacker, any legitimate compressed spend bundle produced by Chia's own tooling would be silently rejected by this code path, breaking normal operation.

The exact corrupted result: `node_from_bytes` returns `Err(EvalErr)` for a valid back-reference-compressed blob, propagated to Python as a `PyValueError`, causing the program to be treated as invalid when it is not.

---

### Likelihood Explanation

Back-reference compression (`node_to_bytes_backrefs` / `node_from_bytes_backrefs`) is an established, production-used serialization path in this codebase — it is exported from `src/serde/mod.rs`, used in `deser_auto`, and has its own dedicated source file `src/serde/de_br.rs`. Any Chia client or tool that serializes programs with `node_to_bytes_backrefs` and then passes the result to `run_serialized_chia_program` will trigger the failure. The entry path is fully attacker-controlled: a crafted spend bundle with a back-reference-compressed generator is sufficient.

---

### Recommendation

Replace `node_from_bytes` with `node_from_bytes_backrefs` for both the program and args inputs in `run_serialized_chia_program`:

```diff
// wheel/src/api.rs
- let program = node_from_bytes(&mut allocator, program).map_err(eval_to_py)?;
- let args    = node_from_bytes(&mut allocator, args).map_err(eval_to_py)?;
+ let program = node_from_bytes_backrefs(&mut allocator, program).map_err(eval_to_py)?;
+ let args    = node_from_bytes_backrefs(&mut allocator, args).map_err(eval_to_py)?;
```

`node_from_bytes_backrefs` is a strict superset of `node_from_bytes` for classic blobs, so this change is backward-compatible.

---

### Proof of Concept

```python
import clvm_rs
from clvmr.serde import node_to_bytes_backrefs
from clvmr import Allocator

# Build a trivial program: (q . 1)
a = Allocator()
node = a.new_pair(a.new_atom(b'\x01'), a.new_atom(b'\x01'))

# Serialize with back-references (valid, network-accepted format)
compressed = node_to_bytes_backrefs(a, node)

# run_serialized_chia_program uses node_from_bytes internally —
# it will raise PyValueError on the compressed blob even though
# deser_backrefs / deser_auto would accept it without error.
clvm_rs.run_serialized_chia_program(
    compressed,   # <-- valid backrefs blob, rejected by node_from_bytes
    b'\x80',      # NIL args
    10_000_000,
    0,
)
# Raises: ValueError: ... (parse error from node_from_bytes)
# But: clvm_rs.deser_backrefs(compressed) succeeds without error.
```
