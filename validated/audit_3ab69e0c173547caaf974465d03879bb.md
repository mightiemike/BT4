### Title
Protobuf Round-Trip Silently Coerces `AllResources` to `L1Gas` Variant, Causing Consensus-Breaking Transaction Rejection - (File: `crates/apollo_protobuf/src/converters/transaction.rs`, `crates/apollo_protobuf/src/converters/rpc_transaction.rs`)

### Summary

The protobuf deserializer for `InvokeTransactionV3` uses `ValidResourceBounds::try_from(protobuf::ResourceBounds)`, which silently downgrades any transaction whose `l2_gas` and `l1_data_gas` are both zero from `ValidResourceBounds::AllResources` to `ValidResourceBounds::L1Gas`. A subsequent conversion step (`TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3`) explicitly rejects the `L1Gas` variant with `DEPRECATED_RESOURCE_BOUNDS_ERROR`. A user can submit a valid V3 invoke transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` that the gateway accepts and the batcher includes in a block proposal, but every validator rejects when deserializing the consensus P2P message, permanently stalling consensus on that block.

### Finding Description

**Step 1 – Serialization preserves `AllResources`.**

`InternalRpcInvokeTransactionV3` always stores `AllResourceBounds` and its `InvokeTransactionV3Trait::resource_bounds()` always returns `ValidResourceBounds::AllResources(self.resource_bounds)`. [1](#0-0) 

When serialized to protobuf via `From<InvokeTransactionV3> for protobuf::InvokeV3`, the `AllResources` variant emits all three fields including `l1_data_gas: Some(zero)`: [2](#0-1) 

**Step 2 – Deserialization silently downgrades to `L1Gas`.**

`TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` calls `ValidResourceBounds::try_from(...)`: [3](#0-2) 

That converter applies the following rule: if `l1_data_gas.is_zero() && l2_gas.is_zero()`, the result is `ValidResourceBounds::L1Gas(l1_gas)`, discarding the `AllResources` variant identity: [4](#0-3) 

**Step 3 – Conversion back to `RpcInvokeTransactionV3` fails.**

`TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` calls `snapi_invoke.try_into()` which is `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3`. That conversion explicitly rejects any variant other than `AllResources`: [5](#0-4) 

The comment in the protobuf converter acknowledges this failure mode: [6](#0-5) 

**Step 4 – Hash domain divergence (secondary effect).**

Even if the conversion did not hard-fail, `get_tip_resource_bounds_hash` produces structurally different Poseidon hashes for the two variants. `L1Gas` hashes 2 resource felts; `AllResources` hashes 3: [7](#0-6) 

The gateway computes hash H1 (3-felt path, `AllResources`). Any node that reconstructs the transaction from protobuf would compute H2 (2-felt path, `L1Gas`), binding the wrong hash to the transaction.

**Analog to the external report.** The SolvBTCMultiAssetPool bug assumed a 1:1 exchange rate between two distinct asset types. Here, the protobuf converter assumes `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` is interchangeable with `L1Gas(X)`. They are not: they produce different hashes and different conversion outcomes. The "depeg" is the zero-valued `l2_gas`/`l1_data_gas` fields that trigger the silent type coercion.

### Impact Explanation

A single unprivileged user submitting one valid V3 invoke transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` causes the following chain:

1. Gateway accepts the transaction (stateless and stateful validators pass).
2. Batcher includes it in a block proposal.
3. Proposer serializes the block to protobuf and broadcasts it.
4. Every validator calls `convert_consensus_tx_to_internal_consensus_tx`, which internally calls `TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3`, which returns `DEPRECATED_RESOURCE_BOUNDS_ERROR`.
5. Validators reject the block proposal.
6. Consensus cannot commit the block; the sequencer stalls on that height.

This matches the impact scope: **High – Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload** and **High – Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing**.

### Likelihood Explanation

The trigger condition (`l2_gas = 0` and `l1_data_gas = 0` in an `AllResourceBounds` transaction) is a normal, valid configuration for a pre-0.13.3-style V3 transaction that only specifies L1 gas. Any user who submits such a transaction—intentionally or by following older SDK examples—triggers the bug. No privileged access is required.

### Recommendation

1. **Fix the protobuf deserializer for `InvokeTransactionV3`** to use `AllResourceBounds::try_from` (which requires `l1_data_gas` to be present and always produces `AllResources`) instead of `ValidResourceBounds::try_from` (which silently downgrades to `L1Gas`). The `AllResourceBounds::try_from` implementation already exists in `crates/apollo_protobuf/src/converters/rpc_transaction.rs` lines 212–223 and is used for the declare path. [8](#0-7) 

2. **Remove the `L1Gas` branch from `ValidResourceBounds::try_from(protobuf::ResourceBounds)`** or gate it behind an explicit version check so that post-0.13.3 transactions are never silently downgraded.

3. **Add a round-trip test** that serializes an `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` transaction to protobuf and back, asserting that the deserialized variant is still `AllResources` and that the transaction hash is preserved.

### Proof of Concept

        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));

        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));

            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }

impl TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3 {
    type Error = ProtobufConversionError;
    fn try_from(mut value: protobuf::InvokeV3WithProof) -> Result<Self, Self::Error> {
        // Extract proof first, since `starknet_api::transaction::InvokeTransactionV3` does not
        // carry a `proof` field.
        let proof = Proof::from(std::mem::take(&mut value.proof));

        let snapi_invoke: InvokeTransactionV3 = value
            .invoke
            .ok_or(ProtobufConversionError::MissingField {
                field_description: "InvokeV3WithProof::invoke",
            })?
            .try_into()?;

        // This conversion can fail only if the resource_bounds are not AllResources.
        Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
    }

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
