### Title
`op_modpow` (opcode 60) is explicitly disabled in `MEMPOOL_MODE`, creating a consensus/mempool divergence — (File: `src/chia_dialect.rs`)

### Summary

`MEMPOOL_MODE` includes the undocumented `DISABLE_OP` flag (0x200), which causes opcode 60 (`op_modpow`) to unconditionally return `EvalErr::Unimplemented` in mempool validation. Because `op_modpow` is a hardforked operator that succeeds in consensus mode (without `DISABLE_OP`), any CLVM program using it is silently rejected by the mempool while remaining valid on-chain — a direct consensus/mempool divergence.

---

### Finding Description

In `src/chia_dialect.rs`, the `ClvmFlags` bitfield defines `DISABLE_OP = 0x200` with **no documentation comment**, unlike every other flag in the set: [1](#0-0) 

`DISABLE_OP` is silently bundled into the `MEMPOOL_MODE` constant, which is the standard flag set used for mempool validation: [2](#0-1) 

Inside the operator dispatch table, opcode 60 (`op_modpow`) is the **only** hardforked operator that checks this flag and hard-fails: [3](#0-2) 

All adjacent hardforked operators — `op_mod` (61), `op_coinid` (48), all BLS ops (49–59) — are dispatched unconditionally with no such guard. The asymmetry is confirmed by the `gc_candidate()` function, which lists opcode 60 as a fully supported GC-eligible operator alongside the others: [4](#0-3) 

The Python API exports `DISABLE_OP` as a public constant and `MEMPOOL_MODE` as the recommended mempool flag, making this reachable by any caller: [5](#0-4) 

---

### Impact Explanation

Any CLVM program that invokes `op_modpow` (opcode 60) will:

- **Succeed** in consensus mode (`ChiaDialect::new(ClvmFlags::empty())`) — the operator is hardforked and fully implemented.
- **Fail** with `EvalErr::Unimplemented` in mempool mode (`ChiaDialect::new(MEMPOOL_MODE)`) — the `DISABLE_OP` guard fires unconditionally.

This is a **consensus/mempool divergence**: a farmer/full node can include a block containing a spend that uses `op_modpow` (valid in consensus), but that spend would have been rejected by every mempool node. Legitimate smart-coin authors who use `op_modpow` in their puzzle solutions cannot submit transactions through normal channels — their spends are permanently blocked at the mempool layer despite being on-chain valid.

---

### Likelihood Explanation

**High.** `MEMPOOL_MODE` is the standard, documented constant for mempool validation. Any node operator using `run_serialized_chia_program` with `MEMPOOL_MODE` (or any flags that include `DISABLE_OP`) will trigger this. The `DISABLE_OP` flag carries no documentation explaining its purpose or intended scope, making it easy to overlook and impossible to reason about correctly.

---

### Recommendation

1. **Document `DISABLE_OP`**: Add a doc comment explaining exactly which operator it disables and why it belongs in `MEMPOOL_MODE`.
2. **Remove `DISABLE_OP` from `MEMPOOL_MODE`** if `op_modpow` is a fully activated hardforked operator (as its presence in `gc_candidate()` and the dispatch table implies). The guard at opcode 60 should be removed or replaced with a proper soft-fork activation flag following the pattern of `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` and `ENABLE_SHA256_TREE`.
3. If `op_modpow` is intentionally restricted in mempool mode (e.g., for DoS reasons), the flag should be renamed (e.g., `DISABLE_MODPOW`), documented, and the `gc_candidate()` list updated to exclude opcode 60.

---

### Proof of Concept

```python
# Using the Python API
from clvm_rs import run_serialized_chia_program, MEMPOOL_MODE
from clvm_rs import NO_UNKNOWN_OPS

# A minimal CLVM program: (modpow 2 10 1000) => 24
# Serialized: opcode 60 = 0x3c, args are atoms
program_bytes = bytes.fromhex("ff3cff02ff0aff8203e880")  # (modpow 2 10 1000)
args_bytes = bytes.fromhex("80")  # ()

# Consensus mode: succeeds
result = run_serialized_chia_program(program_bytes, args_bytes, 10_000_000, 0)
# => (cost, LazyNode(24))

# Mempool mode: raises EvalErr::Unimplemented — same program, same chain state
result = run_serialized_chia_program(program_bytes, args_bytes, 10_000_000, MEMPOOL_MODE)
# => PyValueError: unimplemented operator
```

The root cause is the unconditional `DISABLE_OP` check at `src/chia_dialect.rs:240–241`, activated by `MEMPOOL_MODE` at `src/chia_dialect.rs:72–76`. No attacker input is required — any program byte sequence encoding opcode `0x3c` (60) as the operator triggers the divergence. [3](#0-2) [2](#0-1)
