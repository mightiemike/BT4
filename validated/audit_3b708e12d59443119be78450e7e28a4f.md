Audit Report

## Title
SwapAllowlistExtension Bypassed via MetricOmmSimpleRouter — Any User Can Swap on Allowlist-Restricted Pools - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against a per-pool allowlist, but `MetricOmmPool.swap()` passes `msg.sender` (the immediate caller) as `sender`. When swaps are routed through `MetricOmmSimpleRouter`, `msg.sender` is the router contract, not the end user. Any pool admin who allowlists the router to support standard periphery access inadvertently grants every unprivileged user the ability to bypass the per-user allowlist entirely.

## Finding Description
In `MetricOmmPool.swap()`, the pool passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value forwarded from the pool — i.e., the router address: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly, making itself `msg.sender`: [3](#0-2) 

The same substitution occurs in `exactInput` (multi-hop hops after the first use `address(this)` as payer): [4](#0-3) 

And in `exactOutputSingle`: [5](#0-4) 

This creates a binary trap: if the router is **not** allowlisted, no user can swap through it (even legitimately allowlisted users). If the router **is** allowlisted, the check `allowedSwapper[pool][router] == true` passes for every caller regardless of their individual allowlist status. The user's identity is completely lost before the extension check.

The existing test suite only validates the direct-call path and contains no test for the router-mediated bypass: [6](#0-5) 

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to specific addresses (e.g., KYC-verified users, whitelisted market makers, institutional counterparties) is completely open to any user who routes through `MetricOmmSimpleRouter`. An attacker can trade at oracle-anchored prices on a pool never intended to be publicly accessible, extracting value from LP positions sized and priced for a restricted counterparty set. This constitutes a broken core pool access-control invariant with direct loss of LP principal and protocol fee revenue.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary supported user-facing swap interface. Any pool admin who deploys a curated pool and wants allowlisted users to use the standard router **must** allowlist the router address — this is the natural and expected operational setup. The bypass is therefore reachable in every realistic curated-pool deployment that supports router access, triggered by any unprivileged user with no special preconditions and no privileged access required.

## Recommendation
The extension must gate on the economic actor (the end user), not the immediate `pool.swap()` caller. Viable fixes:

1. **Pass the original user through `extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. Requires a protocol-level convention enforced by the router.
2. **Add a dedicated `originator` field to the extension interface:** A separate `originator` parameter (the true end user) passed alongside `sender` would be the cleanest fix.
3. **Require the router to forward user identity and be the sole allowlisted address:** The extension decodes the true user from `extensionData` that the router is required to populate with `msg.sender`.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin allowlists router: setAllowedToSwap(pool, router, true)
  - Pool admin does NOT allowlist attacker: allowedSwapper[pool][attacker] = false

Attack:
  - attacker calls router.exactInputSingle({pool: pool, ...})
  - router calls pool.swap(recipient, ...) with msg.sender = router
  - pool calls extension.beforeSwap(router, ...)
  - extension checks allowedSwapper[pool][router] == true  ✓ PASSES
  - attacker's swap executes on the restricted pool

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds — allowlist completely bypassed
```

A Foundry integration test can confirm this by deploying the pool with `SwapAllowlistExtension`, allowlisting only the router, and verifying that an unallowlisted EOA successfully swaps via `MetricOmmSimpleRouter.exactInputSingle()` without revert.
