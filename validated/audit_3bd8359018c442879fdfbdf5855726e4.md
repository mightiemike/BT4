### Title
Difficulty Calculation for Fork Blocks Reads Mainchain Ancestor Instead of Fork Ancestor, Enabling Difficulty Manipulation - (File: `contract/src/dogecoin.rs`)

### Summary

`get_next_work_required` in `dogecoin.rs` (and `bitcoin.rs`) calls `blocks_getter.get_header_by_height(height_first)` to obtain the timestamp of the first block in the difficulty adjustment window. This helper always reads from `mainchain_height_to_header`, returning the **mainchain** block at that height. When the block being validated is on a fork that diverged from the mainchain at or before `height_first`, the wrong block's timestamp is used, producing an incorrect expected difficulty. A trusted relayer supplying adversarial fork data can exploit this to submit fork blocks with artificially lowered difficulty that pass `check_pow`, potentially triggering a chain reorg with less hashpower than the protocol requires.

### Finding Description

In `dogecoin.rs`, `get_next_work_required` computes the expected difficulty for the next Dogecoin block. After height 145,000, `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` for almost every block, and `height_first = prev_block_header.block_height - 1`. [1](#0-0) 

The function then fetches the boundary block's timestamp:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

The `TODO` comment at line 291 explicitly flags the unresolved concern. `get_header_by_height` is implemented as: [2](#0-1) 

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

This always returns the **mainchain** block at `height_first`. When the block under validation is on a fork that diverged at or before `height_first`, the mainchain block at that height is a **different block** from the fork's true ancestor at that height. The difficulty calculation therefore uses the wrong timestamp.

The resulting `modulated_timespan` in `calculate_next_work_required` is: [3](#0-2) 

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
```

If the mainchain block at `height_first` has a **later** timestamp than the fork's true ancestor at that height, `modulated_timespan` is smaller than it should be, and the computed target is easier (lower difficulty). The attacker then sets the fork block's `bits` field to this lower value, which passes: [4](#0-3) 

```rust
require!(
    expected_bits == block_header.bits,
    ...
);
```

The same structural bug exists in `bitcoin.rs` at line 81, but the 2016-block adjustment interval makes it far harder to reach in practice. [5](#0-4) 

### Impact Explanation

An attacker controlling a trusted relayer can:

1. Submit a fork chain that diverges from the mainchain at height `H-3` or earlier.
2. Craft the fork's ancestor at height `H-2` to carry an **earlier** timestamp than the mainchain block at `H-2`.
3. When the contract validates the fork block at height `H`, `get_header_by_height(H-2)` returns the mainchain block (later timestamp), making `modulated_timespan` smaller and the computed expected difficulty lower.
4. Submit fork blocks with `bits` matching this artificially lowered difficulty; `check_pow` accepts them.
5. Mine AuxPoW parent blocks at the reduced difficulty (less hashpower required).
6. Once the fork's cumulative chainwork exceeds the mainchain's, `submit_block_header_inner` triggers `reorg_chain`. [6](#0-5) 

After the reorg, `mainchain_height_to_header` and `mainchain_header_to_height` reflect the attacker's fork. Downstream callers of `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` receive results based on the corrupted canonical chain, breaking the SPV guarantee for all consumers. [7](#0-6) 

### Likelihood Explanation

For Dogecoin after height 145,000, difficulty adjusts every block, so **every fork block** triggers this code path. Any party that can act as a trusted relayer (the staking mechanism may be permissionless) can submit adversarial fork data. The attacker only needs to mine AuxPoW parent blocks at the reduced difficulty, which is lower than the protocol-correct target. The timestamp manipulation is bounded by MTP and the two-hour future-time rule, but within those bounds there is sufficient room to meaningfully reduce the computed difficulty.

### Recommendation

Replace the `get_header_by_height` call in `get_next_work_required` with a walk along the fork's ancestor chain via `get_prev_header`, traversing `prev_block_hash` links from `prev_block_header` until reaching `height_first`. This mirrors how `reorg_chain` correctly traverses fork ancestry rather than consulting the mainchain index. [8](#0-7) 

### Proof of Concept

```
Mainchain:  [A@H-3] → [B@H-2, time=T_main] → [C@H-1] → [D@H]
                                                           ↑ tip

Fork:       [A@H-3] → [F2@H-2, time=T_fork < T_main] → [F3@H-1] → [F4@H]
```

1. Attacker submits fork blocks F2, F3 (accepted as fork headers into `headers_pool`).
2. When submitting F4 at height H, `check_pow` calls `get_next_work_required`.
3. `height_first = H - 2`. `get_header_by_height(H-2)` returns mainchain block B with `time = T_main`.
4. `modulated_timespan = time(F3) - T_main`. Because `T_main > T_fork`, this is smaller than the correct `time(F3) - T_fork`, yielding a lower expected difficulty.
5. Attacker sets F4's `bits` to the lower value; `expected_bits == block_header.bits` passes.
6. Attacker mines an AuxPoW parent block meeting the lower target.
7. F4's chainwork exceeds the mainchain tip's chainwork; `reorg_chain` is called, corrupting the canonical chain. [9](#0-8) [10](#0-9)

    fn reorg_chain(&mut self, fork_tip_header: ExtendedHeader, last_main_chain_block_height: u64) {
        let fork_tip_height = fork_tip_header.block_height;
        if last_main_chain_block_height > fork_tip_height {
            // If we see that main chain is longer than fork we first garbage collect
            // outstanding main chain blocks:
            //
            //      [m1] - [m2] - [m3] - [m4] <- We should remove [m4]
            //     /
            // [m0]
            //     \
            //      [f1] - [f2] - [f3]
            for height in (fork_tip_height + 1)..=last_main_chain_block_height {
                let current_main_chain_blockhash = self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str("cannot get a block"));
                self.remove_block_header(&current_main_chain_blockhash);
                self.mainchain_height_to_header.remove(&height);
            }
        }

        // Now we are in a situation where mainchain is equivalent to fork size:
        //
        //      [m1] - [m2] - [m3] - [m4] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip
        //
        //
        // Or in a situation where it is shorter:
        //
        //      [m1] - [m2] - [m3] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip

        let fork_tip_hash = fork_tip_header.block_hash.clone();
        let mut fork_header_cursor = fork_tip_header;

        while !self
            .mainchain_header_to_height
            .contains_key(&fork_header_cursor.block_hash)
        {
            let prev_block_hash = fork_header_cursor.block_header.prev_block_hash;
            let current_block_hash = fork_header_cursor.block_hash;
            let current_height = fork_header_cursor.block_height;

            // Inserting the fork block into the main chain, if some mainchain block is occupying
            // this height let's save its hashcode
            let main_chain_block = self
                .mainchain_height_to_header
                .insert(&current_height, &current_block_hash);
            self.mainchain_header_to_height
                .insert(&current_block_hash, &current_height);

            // If we found a mainchain block at the current height than remove this block from the
            // header pool and from the header -> height map
            if let Some(current_main_chain_blockhash) = main_chain_block {
                self.remove_block_header(&current_main_chain_blockhash);
            }

            // Switch iterator cursor to the previous block in fork
            fork_header_cursor = self
                .headers_pool
                .get(&prev_block_hash)
                .unwrap_or_else(|| env::panic_str("previous fork block should be there"));
        }

        // Updating tip of the new main chain
        self.mainchain_tip_blockhash = fork_tip_hash;
    }
