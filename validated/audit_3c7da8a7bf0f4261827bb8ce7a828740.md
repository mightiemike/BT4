Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any address to bypass the deposit allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual token payer) and instead checks `owner` (the position recipient). Because `MetricOmmPool.addLiquidity` explicitly supports an operator pattern where `msg.sender ≠ owner` with no equality check enforced, any non-allowlisted address can bypass the deposit gate by supplying an allowlisted address as `owner`.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as the actual depositor and `owner` as the position recipient to the extension hook: [1](#0-0) 

Unlike `removeLiquidity` which enforces `msg.sender == owner`, `addLiquidity` imposes no such constraint: [2](#0-1) 

`ExtensionCalling._beforeAddLiquidity` correctly encodes both `sender` and `owner` and forwards them to the extension: [3](#0-2) 

However, `DepositAllowlistExtension.beforeAddLiquidity` leaves the first parameter (`sender`) unnamed and checks `owner` instead: [4](#0-3) 

The sibling `SwapAllowlistExtension.beforeSwap` correctly names and checks `sender`: [5](#0-4) 

The operator pattern is explicitly exercised by `MetricOmmPoolLiquidityAdder.addLiquidityExactShares`, which accepts a caller-supplied `owner` distinct from `msg.sender`: [6](#0-5) 

**Exploit flow:**
1. Pool admin deploys pool with `DepositAllowlistExtension` and calls `setAllowedToDeposit(pool, alice, true)`.
2. Non-allowlisted Eve calls `pool.addLiquidity(owner=alice, salt, deltas, callbackData, "")`.
3. The extension receives `sender=Eve` (discarded) and `owner=alice` (checked): `allowedDepositor[pool][alice] == true` → passes.
4. Eve's tokens are pulled via the modify-liquidity callback; Alice's position is credited.

The wrong value checked is `allowedDepositor[msg.sender][owner]` at line 38 — `owner` is the position recipient, not the token payer. [7](#0-6) 

## Impact Explanation
The deposit allowlist is the pool admin's primary mechanism to enforce a permissioned LP set (KYC/compliance gates, private pools). The broken check renders the gate completely ineffective: any unprivileged address can deposit into a restricted pool by naming any allowlisted address as `owner`. This constitutes an admin-boundary break where an unprivileged path bypasses a pool admin access control, and enables unsolicited position griefing against the named `owner`. This matches the "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" and "Broken core pool functionality" impact categories.

## Likelihood Explanation
No special privilege is required — any EOA or contract can call `addLiquidity` with an arbitrary `owner`. The operator pattern (`msg.sender ≠ owner`) is a documented, first-class feature of `addLiquidity` with no on-chain restriction. `MetricOmmPoolLiquidityAdder` explicitly exercises this pattern with a caller-supplied `owner`. The bypass is trivially reachable with zero preconditions.

## Recommendation
Bind `sender` (the actual depositor) instead of `owner` in `beforeAddLiquidity`:

```solidity
// AFTER (correct actor)
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Add a unit test calling `addLiquidity` with `msg.sender ≠ owner` against a pool configured with `DepositAllowlistExtension`, asserting the non-allowlisted sender is rejected even when `owner` is allowlisted.

## Proof of Concept

```
Setup:
  pool admin deploys pool with DepositAllowlistExtension
  pool admin calls setAllowedToDeposit(pool, alice, true)
  Eve is NOT in the allowlist

Attack:
  Eve calls pool.addLiquidity(
      owner = alice,   // allowlisted — passes the broken check
      salt  = 0,
      deltas = <valid bins>,
      callbackData = ...,
      extensionData = ""
  )

Extension check (beforeAddLiquidity):
  sender = Eve   ← unnamed, discarded (DepositAllowlistExtension.sol:32)
  owner  = alice ← checked at line 38: allowedDepositor[pool][alice] == true → PASS

Result:
  Eve's tokens are pulled via metricOmmModifyLiquidityCallback
  Alice's position is credited with Eve's deposit
  Eve has successfully deposited into a pool she is not allowlisted for
```
