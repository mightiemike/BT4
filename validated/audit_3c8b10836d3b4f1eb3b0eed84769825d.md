### Title
Missing Caller Validation in `run_mainchain_gc` Allows Unauthorized Premature Pruning of Canonical Chain State - (File: `contract/src/lib.rs`)

### Summary

`run_mainchain_gc` is a public, state-mutating function that carries no caller identity check. When the contract is not paused, any unprivileged NEAR account can invoke it with an attacker-controlled `batch_size`, causing the contract to immediately prune all excess mainchain headers in a single call. This corrupts `mainchain_initial_blockhash` and permanently removes headers from `headers_pool` and `mainchain_height_to_header`, breaking SPV proof verification and chain-reorg resolution for any block in the pruned range.

### Finding Description

`run_mainchain_gc` is decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`. [1](#0-0) 

The `#[pause]` macro enforces a caller-role check **only when the contract is paused** — the `except(roles(...))` clause is the bypass for the paused state. When the contract is running normally (unpaused), the macro imposes no restriction on who may call the function. There is no `#[trusted_relayer]` guard, no `#[private]` guard, and no explicit `env::predecessor_account_id()` check anywhere in the function body. [2](#0-1) 

The attacker-controlled `batch_size` parameter is used directly in:

```rust
let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);
```

Passing `u64::MAX` causes `selected_amount_to_remove` to equal `total_amount_to_remove` — the full number of excess blocks — so the entire prunable window is deleted in one call. The loop then removes every header in that range from `mainchain_height_to_header` and `headers_pool`, and advances `mainchain_initial_blockhash` to the new oldest block. [3](#0-2) 

By contrast, `submit_blocks` — the only intended caller of `run_mainchain_gc` — is protected by `#[trusted_relayer]` and passes only `num_of_headers` (the count of headers in the current batch, typically a small number) as `batch_size`, deliberately bounding how many blocks are pruned per call. [4](#0-3) 

### Impact Explanation

Two concrete state invariants are broken:

**1. SPV proof verification is permanently invalidated for pruned blocks.**
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` both look up the target block in `mainchain_header_to_height` and `headers_pool`. After an attacker-triggered GC run, those entries no longer exist. Any downstream NEAR contract that calls either verification function for a transaction in a pruned block will receive a panic ("block does not belong to the current main chain" / "cannot find requested transaction block") instead of a valid result. [5](#0-4) 

**2. Chain reorg resolution panics for forks whose ancestor is in the pruned range.**
`reorg_chain` walks the fork chain backward until it finds a block present in `mainchain_header_to_height`. If the common ancestor was pruned, the walk reaches a block whose `prev_block_hash` is no longer in `headers_pool`, causing `env::panic_str("previous fork block should be there")`. This permanently breaks the contract's ability to process any reorg rooted in the pruned window. [6](#0-5) 

The CLAUDE.md explicitly acknowledges this reorg-failure mode but treats it as a consequence of normal GC, not of unauthorized early GC: [7](#0-6) 

An attacker can trigger this failure window at will, at any time the mainchain has grown beyond `gc_threshold`, without holding any role.

### Likelihood Explanation

The preconditions are minimal: the contract must be unpaused (its normal operating state) and the mainchain must have grown beyond `gc_threshold` (inevitable in production; the recommended value is 52,704 blocks ≈ 1 year). No privileged key, no staked relayer role, and no special knowledge is required. The call costs only standard NEAR gas fees. Any NEAR account can execute this.

### Recommendation

Add a caller-identity guard to `run_mainchain_gc` equivalent to the one on `submit_blocks`. The simplest fix is to add `#[trusted_relayer]` (or a role check such as `Role::DAO | Role::RelayerManager`) so that only authorized relayers or governance accounts can trigger GC directly. Alternatively, make the function `pub(crate)` and remove it from the public ABI entirely, since its only legitimate external caller is `submit_blocks`.

### Proof of Concept

1. Deploy the contract with `gc_threshold = 52704` and allow the mainchain to grow to `52705` blocks (one block beyond threshold).
2. From any unprivileged NEAR account (no roles granted), call:
   ```
   run_mainchain_gc(18446744073709551615)  // u64::MAX
   ```
3. The contract removes block at height `genesis_height` from `mainchain_height_to_header` and `headers_pool`, and advances `mainchain_initial_blockhash`.
4. Any subsequent call to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a transaction in that pruned block panics with "block does not belong to the current main chain."
5. Any fork whose common ancestor is the pruned block causes `reorg_chain` to panic with "previous fork block should be there" when a relayer later submits a competing chain tip.
