Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Unauthorized Address to Bypass the Deposit Allowlist - (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual token payer) and gates only on `owner` (the position recipient). Because `MetricOmmPool.addLiquidity` supports an operator pattern where `msg.sender ≠ owner` with no equality constraint, any address not on the allowlist can deposit tokens into a curated pool by nominating any allowlisted address as `owner`. The allowlist check passes on the allowlisted `owner` while the actual depositor (`sender`) is never verified.

## Finding Description
`MetricOmmPool.addLiquidity` passes two distinct actors to the extension hook:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`msg.sender` is the actual caller who pays tokens via the callback; `owner` is the address that receives minted LP shares. There is no check enforcing `msg.sender == owner`. [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but leaves it unnamed and never reads it. The guard resolves to `allowedDepositor[pool][owner]`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

Because `owner` is a caller-supplied argument with no constraint tying it to `msg.sender`, an unauthorized caller sets `owner` to any allowlisted address and the check passes unconditionally.

`MetricOmmPoolLiquidityAdder` amplifies the surface: `_validateOwner` only rejects `address(0)`, and `addLiquidityExactShares` explicitly supports depositing on behalf of a different `owner`: [3](#0-2) [4](#0-3) 

The `addLiquidityWeighted` path calls `pool.addLiquidity(owner, ...)` with `sender = address(LiquidityAdder)`, so the extension sees the adder contract as sender and the caller-supplied `owner` as the gated identity — the same bypass applies. [5](#0-4) 

## Impact Explanation
A pool deployer configures `DepositAllowlistExtension` to restrict deposits to a curated set of addresses (e.g., KYC-verified LPs). The bypass allows any unpermissioned address to: (1) deposit tokens into the pool without appearing on the allowlist — the curation policy is fully defeated; (2) force LP shares onto an allowlisted address that did not initiate the deposit — a griefing vector creating unwanted positions for the victim who must spend gas to remove them and may be exposed to adverse price movement; (3) introduce tokens from non-allowlisted sources into a pool whose compliance posture depends on knowing the origin of every deposit. This constitutes a broken core pool functionality causing curation failure and potential fund-impacting consequences. [6](#0-5) 

## Likelihood Explanation
The exploit requires no special privileges, no flash loan, and no oracle manipulation. Any EOA or contract can call `pool.addLiquidity(allowlisted_address, ...)` directly or route through `MetricOmmPoolLiquidityAdder`. The only prerequisite is knowing one allowlisted address, which is publicly readable from the `allowedDepositor` mapping on-chain. Likelihood is **High**. [7](#0-6) 

## Recommendation
Gate on `sender` (the actual token payer) rather than — or in addition to — `owner`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intended policy is that both the payer and the position owner must be allowlisted (to prevent griefing of allowlisted owners), check both:

```solidity
if (!allowAllDepositors[msg.sender]
    && (!allowedDepositor[msg.sender][sender] || !allowedDepositor[msg.sender][owner])) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
}
``` [2](#0-1) 

## Proof of Concept
**Setup:**
- Pool deployed with `DepositAllowlistExtension` as `extension1`, `beforeAddLiquidity` order set to `1`.
- `alice` is allowlisted: `allowedDepositor[pool][alice] = true`.
- `bob` is **not** allowlisted.

**Direct pool call:**
```solidity
// Bob calls addLiquidity directly, nominating Alice as owner.
pool.addLiquidity(
    alice,        // owner — passes allowlist check
    salt,
    deltas,
    callbackData, // Bob's callback pays the tokens
    extensionData
);
// Extension sees: allowedDepositor[pool][alice] == true → passes.
// Bob's callback fires; Bob transfers tokens to the pool.
// Alice receives LP shares she never requested.
```

**Via LiquidityAdder:**
```solidity
liquidityAdder.addLiquidityExactShares(
    address(pool),
    alice,         // owner — passes allowlist check; _validateOwner only checks != address(0)
    salt,
    deltas,
    maxAmount0,
    maxAmount1,
    extensionData
);
// pool.addLiquidity(alice, ...) called with sender = address(liquidityAdder).
// Extension checks allowedDepositor[pool][alice] → true → passes.
// Bob (msg.sender of adder) pays tokens; Alice receives shares.
``` [8](#0-7) [9](#0-8)
