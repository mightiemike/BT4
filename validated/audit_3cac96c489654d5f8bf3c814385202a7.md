### Title
Unknown Operator Cost Computed Correctly But Silently Discarded via Wrong Threshold — (`File: src/more_ops.rs`)

### Summary
`op_unknown` in `src/more_ops.rs` correctly computes the cost of an unknown operator (including the multiplier scaling), but then rejects the operator with `EvalErr::Invalid` whenever the final cost exceeds `u32::MAX` (~4.29 billion). Because `Cost` is `u64` and the network's actual cost ceiling is ~11 billion, any unknown operator whose scaled cost falls in the range `(u32::MAX, max_cost]` is incorrectly rejected in consensus mode. The cost is computed correctly but never applied — it is discarded in favour of a hard error, exactly mirroring H-02's pattern of a value being calculated correctly but never routed to its intended destination.

### Finding Description

In `src/more_ops.rs`, `op_unknown` computes a base cost according to the operator's embedded cost-function bits, then scales it by `cost_multiplier + 1`:

```rust
// src/more_ops.rs  lines 258-266
check_cost(cost, max_cost)?;          // ← guards pre-multiplication cost only
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {           // ← WRONG threshold; Cost is u64
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`Cost` is `u64` (the network enforces an ~11 billion cost ceiling). The guard `check_cost(cost, max_cost)` is applied **before** the multiplication, so it only bounds the pre-scaled value. After multiplication the result can legitimately land anywhere in `(u32::MAX, max_cost]` — a range of roughly 6.7 billion units — yet the code treats every value in that range as a hard protocol error (`EvalErr::Invalid`) rather than a valid, expensive operator.

The correct post-multiplication check is `cost > max_cost` (returning `EvalErr::CostExceeded`), not `cost > u32::MAX as u64` (returning `EvalErr::Invalid`).

**Concrete trigger path:**

A 4-byte unknown opcode encodes a 3-byte multiplier (max `0xFFFFFF = 16,777,215`) and a 1-byte cost-function selector. With cost-function `1` (ARITH-like) and a single argument of ~52 bytes:

```
pre_mult_cost = ARITH_BASE_COST(99)
              + 1 × ARITH_COST_PER_ARG(320)
              + 52 × ARITH_COST_PER_BYTE(3)
              = 575
```

With multiplier `= 16,777,215`:

```
final_cost = 575 × 16,777,216 = 9,646,899,200
```

`9,646,899,200 > u32::MAX (4,294,967,295)` → rejected with `EvalErr::Invalid`.  
`9,646,899,200 < 11,000,000,000 (max_cost)` → should be accepted.

The attacker controls the opcode bytes and the argument, both of which arrive as raw CLVM bytes from an untrusted transaction.

### Impact Explanation

**Consensus divergence.** In consensus mode (`allow_unknown_ops() == true`, i.e. `NO_UNKNOWN_OPS` not set), unknown operators are specified to be no-ops with well-defined costs. A node running this code rejects any such operator whose scaled cost exceeds `u32::MAX` with a hard `Invalid` error, while a correctly-implemented node accepts it (subject to `max_cost`). This causes a fork: the buggy node marks a valid spend bundle as invalid, diverging from the rest of the network.

**Impact: Medium** — affects consensus validity of spend bundles containing unknown operators with large cost multipliers.

### Likelihood Explanation

**Likelihood: High** — the trigger requires only a crafted 4-byte opcode with a large multiplier and a small argument list. No privileged access, no social engineering, and no dependency on local configuration is needed. Any party submitting a transaction to the network can exercise this path. The parameter space that triggers the bug (multiplier ≥ ~7,470, argument ~52 bytes for cost-function 1) is large and easy to hit deliberately.

### Recommendation

Replace the post-multiplication threshold with the actual cost ceiling:

```rust
// After: cost *= cost_multiplier + 1;
if cost > max_cost {
    Err(EvalErr::CostExceeded)?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

This correctly accepts all unknown operators whose scaled cost is within the network limit and rejects only those that genuinely exceed it, with the right error type.

pub fn op_unknown(
    allocator: &mut Allocator,
    o: NodePtr,
    mut args: NodePtr,
    max_cost: Cost,
) -> Response {
    // unknown opcode in lenient mode
    // unknown ops are reserved if they start with 0xffff
    // otherwise, unknown ops are no-ops, but they have costs. The cost is computed
    // like this:

    // byte index (reverse):
    // | 4 | 3 | 2 | 1 | 0          |
    // +---+---+---+---+------------+
    // | multiplier    |XX | XXXXXX |
    // +---+---+---+---+---+--------+
    //  ^               ^    ^
    //  |               |    + 6 bits ignored when computing cost
    // cost_multiplier  |
    // (up to 4 bytes)  + 2 bits
    //                    cost_function

    // 1 is always added to the multiplier before using it to multiply the cost, this
    // is since cost may not be 0.

    // cost_function is 2 bits and defines how cost is computed based on arguments:
    // 0: constant, cost is 1 * (multiplier + 1)
    // 1: computed like operator add, multiplied by (multiplier + 1)
    // 2: computed like operator mul, multiplied by (multiplier + 1)
    // 3: computed like operator concat, multiplied by (multiplier + 1)

    // this means that unknown ops where cost_function is 1, 2, or 3, may still be
    // fatal errors if the arguments passed are not atoms.

    let op_atom = allocator.atom(o);
    let op = op_atom.as_ref();

    if op.is_empty() || (op.len() >= 2 && op[0] == 0xff && op[1] == 0xff) {
        Err(EvalErr::Reserved(o))?;
    }

    let cost_function = (op[op.len() - 1] & 0b11000000) >> 6;
    let cost_multiplier: u64 = match u32_from_u8(&op[0..op.len() - 1]) {
        Some(v) => v as u64,
        None => {
            return Err(EvalErr::Invalid(o))?;
        }
    };
