Now let me check the protobuf schema definition and how block hash computation uses execution resources.

### Title
P2P Sync Protobuf Serialization Silently Drops AddMod/MulMod/RangeCheck96 Builtin Counts in Transaction Receipts — (`crates/apollo_protobuf/src/converters/receipt.rs`)

---

### Summary

The `From<HashMap<Builtin, u64>> for ProtobufBuiltinCounter` implementation omits `AddMod`, `MulMod`, and `RangeCheck96` from the protobuf wire encoding. Any syncing node that receives a block via P2P will store and serve zero for these builtin counts, even when the actual execution used them. The protobuf schema itself acknowledges the gap with `// TODO(alonl): add the missing builtins`.

---

### Finding Description

`ProtobufBuiltinCounter` (type alias for `protobuf::receipt::execution_resources::BuiltinCounter`) is defined in the `.proto` file with only eight fields: [1](#0-0) 

`AddMod`, `MulMod`, and `RangeCheck96` are absent. The Rust serializer matches this gap exactly: [2](#0-1) 

The deserializer also never reads them back: [3](#0-2) 

Yet `starknet_api::execution_resources::Builtin` fully defines all three variants: [4](#0-3) 

And the blockifier actively tracks and emits them (e.g., native entry-point execution): [5](#0-4) 

The P2P sync server serializes stored `TransactionOutput` (which contains `ExecutionResources`) into protobuf for every syncing peer: [6](#0-5) 

The syncing client deserializes it and writes it to storage: [7](#0-6) 

---

### Impact Explanation

Every syncing node that receives a block containing transactions that used `AddMod`, `MulMod`, or `RangeCheck96` builtins will persist `0` for those counts. Any RPC call (`starknet_getTransactionReceipt`, `starknet_simulateTransactions`, trace APIs) served from that node's storage will return authoritative-looking but incorrect `execution_resources`.

**Regarding the block hash claim in the question**: the `receipt_commitment` that enters the block hash is computed by the proposer directly from the correct in-memory `ExecutionResources` via `calculate_block_commitments`: [8](#0-7) 

The syncing node receives this pre-computed commitment in the block header and stores it verbatim — it does not recompute it from the deserialized transaction outputs. Therefore the block hash stored by syncing nodes is **not** corrupted by this bug in the current implementation. The Critical impact (wrong block hash) claimed in the question is **not confirmed**.

The confirmed impact is **High**: syncing nodes serve wrong `execution_resources` values via RPC for any transaction that used these builtins.

---

### Likelihood Explanation

`AddMod` and `MulMod` are used by Cairo's circuit evaluation (`EvalCircuit` hint) and `RangeCheck96` is used by range-check-heavy arithmetic. These are present in real mainnet transactions. Any syncing node (i.e., every full node that is not the original proposer) is affected for all such transactions.

---

### Recommendation

1. Add `add_mod`, `mul_mod`, and `range_check96` fields to `BuiltinCounter` in `crates/apollo_protobuf/src/proto/p2p/proto/sync/receipt.proto` (resolving the existing `TODO(alonl)` comment).
2. Update `From<HashMap<Builtin, u64>> for ProtobufBuiltinCounter` to write those three fields.
3. Update `TryFrom<ProtobufBuiltinCounter> for HashMap<Builtin, u64>` to read them back.
4. Add a round-trip test asserting non-zero `AddMod`/`MulMod`/`RangeCheck96` counts survive serialization.

---

fn add_mod_mul_mod_range_check96_survive_protobuf_round_trip() {
    let mut counts = HashMap::new();
    counts.insert(Builtin::AddMod, 7_u64);
    counts.insert(Builtin::MulMod, 3_u64);
    counts.insert(Builtin::RangeCheck96, 42_u64);

    // Serialize
    let proto = ProtobufBuiltinCounter::from(counts.clone());

    // Deserialize
    let recovered = HashMap::<Builtin, u64>::try_from(proto).unwrap();

    // These assertions FAIL on the current code — all three are 0
    assert_eq!(recovered[&Builtin::AddMod], 7);
    assert_eq!(recovered[&Builtin::MulMod], 3);
    assert_eq!(recovered[&Builtin::RangeCheck96], 42);
}

            .zip(transaction_outputs)
            .zip(transaction_hashes)
            .map(|((a, b), c)| (a, b, c))
        {
            result.push(FullTransaction { transaction, transaction_output, transaction_hash });
        }
        Ok(result)

                    }),
                )?;
                let Some(FullTransaction { transaction, transaction_output, transaction_hash }) =
                    maybe_transaction?.0
                else {
                    if current_transaction_len == 0 {
                        return Ok(None);
                    } else {
                        return Err(ParseDataError::BadPeer(BadPeerError::NotEnoughTransactions {
                            expected: target_transaction_len,
                            actual: current_transaction_len,
                            block_number: block_number.0,
                        }));
                    }
                };
                block_body.transactions.push(transaction);
                block_body.transaction_outputs.push(transaction_output);
                // TODO(eitan): Validate transaction hash from untrusted sources
                block_body.transaction_hashes.push(transaction_hash);
                current_transaction_len += 1;
            }
