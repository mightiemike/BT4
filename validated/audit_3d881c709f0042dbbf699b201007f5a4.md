### Title
`gas_prices_to_hash` omits `l2_gas_consumed` and `next_l2_gas_price` for Starknet versions > 0.14.0, producing a canonically wrong block hash — (File: `crates/starknet_api/src/block_hash/block_hash_calculator.rs`)

---

### Summary

The `gas_prices_to_hash` function that feeds the `gas_prices_hash` slot of every block hash carries an explicit TODO acknowledging that `l2_gas_consumed` and `next_l2_gas_price` must be added for Starknet versions after 0.14.0. Those fields are present in every live block header struct (`BlockHeaderWithoutHash`, `StorageBlockHeader`) and are already forwarded to the block-hash CLI by the echonet verification tool, but `PartialBlockHashComponents` — the sole input to `calculate_block_hash` — never captures them, and `BlockHashVersion` has no variant that would gate their inclusion. Every block produced on Starknet ≥ 0.14.1 therefore carries a hash computed over an incomplete preimage.

---

### Finding Description

`gas_prices_to_hash` is the function that produces the `gas_prices_hash` field chained into `calculate_block_hash`:

```
// TODO(Ayelet): add l2_gas_consumed, next_l2_gas_price after 0.14.0.
pub fn gas_prices_to_hash(
    l1_gas_price: &GasPricePerToken,
    l1_data_gas_price: &GasPricePerToken,
    l2_gas_price: &GasPricePerToken,
    block_hash_version: &BlockHashVersion,
) -> Vec<Felt> {
    if block_hash_version >= &BlockHashVersion::V0_13_4 {
        vec![HashChain::new()
            .chain(&STARKNET_GAS_PRICES0)
            .chain(&l1_gas_price.price_in_wei.0.into())
            ...
            .chain(&l2_gas_price.price_in_fri.0.into())
            .get_poseidon_hash()]
    } ...
}
``` [1](#0-0) 

The `BlockHashVersion` enum has exactly two variants — `V0_13_2` and `V0_13_4` — and every Starknet version ≥ 0.13.4 (including 0.14.0 through 0.14.4) resolves to `V0_13_4`: [2](#0-1) 

No new variant exists for versions > 0.14.0, so there is no code path that would ever include `l2_gas_consumed` or `next_l2_gas_price`.

`PartialBlockHashComponents` — the only struct accepted by `calculate_block_hash` — carries `l1_gas_price`, `l1_data_gas_price`, and `l2_gas_price`, but **not** `l2_gas_consumed` or `next_l2_gas_price`: [3](#0-2) 

Both fields are present in the canonical block header types used everywhere else in the system: [4](#0-3) [5](#0-4) 

The echonet OS-verification tool already passes both fields to the block-hash CLI when computing the authoritative hash for a blob: [6](#0-5) 

The sequencer itself populates `l2_gas_consumed` and `next_l2_gas_price` in the `BlockHeaderWithoutHash` it hands to state-sync: [7](#0-6) 

The divergence is therefore: the sequencer stores the correct values in the header, but `PartialBlockHashComponents::new` silently drops them, so `calculate_block_hash` hashes an incomplete preimage for every block on Starknet > 0.14.0. [8](#0-7) 

---

### Impact Explanation

The block hash is the canonical commitment to a block's content. It is:

1. **Stored as authoritative state** in `StorageBlockHeader.block_hash` and served by the RPC layer.
2. **Verified by the echonet tool** before L1 finalization; a mismatch causes the tool to reject the block.
3. **Embedded in proof facts** (`SnosProofFacts.block_hash`) and validated by `validate_proof_block_hash` during transaction pre-validation — a wrong stored hash causes all client-side-proving invoke transactions referencing that block to be rejected.

A sequencer that omits `l2_gas_consumed` and `next_l2_gas_price` from the hash preimage produces a hash that diverges from the value the echonet CLI computes and from what any external verifier or prover expects. This maps to **Critical — wrong state / storage value** (the stored block hash is wrong for every block on Starknet ≥ 0.14.1) and **Critical — wrong receipt / revert result** (proof-facts-bearing transactions are incorrectly rejected or accepted against the wrong hash).

---

### Likelihood Explanation

The current production Starknet version is 0.14.4 (confirmed by test fixtures such as `block_post_0_14_2.json` and `block_post_0_14_3.json`). Every block produced since 0.14.1 is affected. No configuration flag or feature gate suppresses the omission; it is unconditional for all `BlockHashVersion::V0_13_4` blocks.

---

### Recommendation

1. Add a new `BlockHashVersion::V0_14_1` (or equivalent) variant to `BlockHashVersion` that activates for Starknet versions > 0.14.0.
2. Extend `PartialBlockHashComponents` with `l2_gas_consumed: GasAmount` and `next_l2_gas_price: GasPrice`.
3. Update `PartialBlockHashComponents::new` to populate these fields from `BlockInfo` (or the equivalent source).
4. In `gas_prices_to_hash`, chain `l2_gas_consumed` and `next_l2_gas_price` into the Poseidon hash when `block_hash_version >= BlockHashVersion::V0_14_1`, matching the preimage already used by the echonet block-hash CLI.

---

### Proof of Concept

The omission is self-documenting in the source:

```rust
// TODO(Ayelet): add l2_gas_consumed, next_l2_gas_price after 0.14.0.
pub fn gas_prices_to_hash(
    l1_gas_price: &GasPricePerToken,
    l1_data_gas_price: &GasPricePerToken,
    l2_gas_price: &GasPricePerToken,   // ← only 3 price inputs; consumed/next absent
    block_hash_version: &BlockHashVersion,
) -> Vec<Felt> { ... }
```

The echonet tool already supplies the missing fields to the CLI:

```python
"l2_gas_consumed": fee_market_info["l2_gas_consumed"],   # present in CLI call
"next_l2_gas_price": fee_market_info["next_l2_gas_price"], # present in CLI call
```

`PartialBlockHashComponents` never receives them:

```rust
pub struct PartialBlockHashComponents {
    pub l1_gas_price: GasPricePerToken,
    pub l1_data_gas_price: GasPricePerToken,
    pub l2_gas_price: GasPricePerToken,
    // l2_gas_consumed: absent
    // next_l2_gas_price: absent
    ...
}
```

Any block produced on Starknet 0.14.1–0.14.4 will have a hash computed without these two fields, diverging from the canonical value expected by the echonet verifier and any external prover that follows the full specification.

impl From<BlockHashVersion> for StarknetVersion {
    fn from(value: BlockHashVersion) -> Self {
        match value {
            BlockHashVersion::V0_13_2 => StarknetVersion::V0_13_2,
            BlockHashVersion::V0_13_4 => StarknetVersion::V0_13_4,
        }
    }
}

impl TryFrom<StarknetVersion> for BlockHashVersion {
    type Error = StarknetApiError;

    fn try_from(value: StarknetVersion) -> StarknetApiResult<Self> {
        if value < Self::V0_13_2.into() {
            Err(StarknetApiError::BlockHashVersion { version: value.to_string() })
        } else if value < Self::V0_13_4.into() {
            // Starknet versions 0.13.2 and 0.13.3 both have the same block hash mechanism.
            Ok(Self::V0_13_2)
        } else {
            Ok(Self::V0_13_4)
        }
    }
}
