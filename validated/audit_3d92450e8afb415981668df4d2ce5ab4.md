### Title
Unprivileged Caller Can Front-Run `verify_transaction_inclusion` by Triggering GC to Permanently Prune the Target Block - (`contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` carries no role restriction and is callable by any unprivileged NEAR account when the contract is not paused. An attacker can call it immediately before a victim's `verify_transaction_inclusion` (or `_v2`) call to prune the target block from both `mainchain_header_to_height` and `headers_pool`, causing the verification to panic permanently. Because GC also removes the predecessor chain, the pruned block cannot be re-submitted, making the failure unrecoverable for any bridge or protocol that depends on that proof.

---

### Finding Description

**Root cause — no access control on `run_mainchain_gc`:**

`run_mainchain_gc` is decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`. When the contract is not paused, any NEAR account — with zero privileges — can call it with an arbitrary `batch_size`. [1](#0-0) 

**What GC does to state:**

For every height in `start_removal_height..end_removal_height`, GC calls `remove_block_header`, which deletes the entry from both `mainchain_header_to_height` and `headers_pool`: [2](#0-1) [3](#0-2) 

**Why the guard in `verify_transaction_inclusion` does not protect against this:**

The only safety check is:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [4](#0-3) 

This validates the *requested confirmation count* against the threshold, but it does **not** verify that the specific target block has not already been pruned. Immediately after this check, the function performs an unconditional lookup: [5](#0-4) 

If the block was removed by a concurrent or front-running `run_mainchain_gc` call, this lookup panics with `"block does not belong to the current main chain"`.

**End-to-end exploit path:**

1. The chain has grown to `gc_threshold + N` stored blocks (e.g., 52705 when `gc_threshold = 52704`). This is normal after ~1 year of production operation.
2. A user (or bridge contract) submits `verify_transaction_inclusion` for block at height `H` (the oldest stored block, which satisfies `confirmations ≤ gc_threshold`).
3. The attacker observes the pending call and front-runs it by submitting `run_mainchain_gc(N)`.
4. GC removes block `H` from `mainchain_header_to_height` and `headers_pool`.
5. The user's call reaches the `mainchain_header_to_height.get(...)` lookup and panics.
6. Because `headers_pool` no longer contains block `H` or its predecessors, `submit_blocks` cannot re-anchor the chain at that height (`get_prev_header` would panic with `"PrevBlockNotFound"`), making the failure permanent. [6](#0-5) 

The same attack applies to `verify_transaction_inclusion_v2`, which calls `verify_transaction_inclusion` internally after its own `headers_pool.get` (which would also panic on a pruned block): [7](#0-6) 

---

### Impact Explanation

Any NEAR contract (e.g., a BTC bridge) that calls `verify_transaction_inclusion` to gate asset release will receive a permanent panic for the targeted block. Because the block and its predecessor chain are both erased from `headers_pool`, the block cannot be re-submitted. The bridge's release transaction fails unrecoverably, permanently locking the user's funds. The attacker needs no privileged keys and spends only the NEAR gas cost of one `run_mainchain_gc` call.

---

### Likelihood Explanation

**Preconditions:**
- The chain must have grown beyond `gc_threshold` blocks. With the recommended production value of `gc_threshold = 52704` (~1 year of Bitcoin blocks), this is a certainty in any long-running deployment.
- The target block must be at or near the GC boundary. Any transaction in a block that is `≥ gc_threshold` blocks old satisfies this.

**Attacker capability:** Zero privileges required. The attacker only needs a funded NEAR account and the ability to submit a transaction before the victim's call is finalized. NEAR processes transactions in receipt order within a shard, so submitting `run_mainchain_gc` first is straightforwardly achievable by any party monitoring the chain.

**Motivation:** A malicious competitor, a censorship actor, or anyone wishing to block a specific Bitcoin-to-NEAR bridge withdrawal has a clear financial incentive.

---

### Recommendation

**Option A (preferred):** Restrict `run_mainchain_gc` to trusted roles (e.g., `Role::DAO` or a new `Role::GCManager`), so unprivileged accounts cannot trigger pruning on demand. The automatic GC already runs inside `submit_blocks` (called by trusted relayers), so external public access to this function is unnecessary.

```rust
// Before
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) { ... }

// After
#[pause(except(roles(Role::UnrestrictedRunGC)))]
#[access_control_any(roles(Role::DAO, Role::RelayerManager))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) { ... }
```

**Option B:** Add a safety margin check inside `verify_transaction_inclusion` that ensures the target block is not within the next `batch_size` blocks of the GC boundary, giving callers a guaranteed stable window.

**Option C:** Remove the public `run_mainchain_gc` entry point entirely and rely solely on the automatic GC invoked within `submit_blocks`. [8](#0-7) 

---

### Proof of Concept

```
Setup:
  gc_threshold = 10
  Submit 12 blocks (heights 0–11) → amount_of_headers_we_store = 12 > 10
  Block at height 0 is now eligible for GC (total_amount_to_remove = 2)

Victim action (pending):
  verify_transaction_inclusion({
      tx_block_blockhash: hash_of_block_0,
      confirmations: 6,   // ≤ gc_threshold=10, passes the guard
      ...
  })

Attacker action (submitted first):
  run_mainchain_gc(batch_size = 2)
  → removes blocks at heights 0 and 1 from mainchain_header_to_height and headers_pool

Result:
  Victim's call reaches:
    mainchain_header_to_height.get(&hash_of_block_0)
    → None → panic("block does not belong to the current main chain")

  Block 0 cannot be re-submitted:
    submit_blocks([block_0]) → get_prev_header(block_0.prev) → panic("PrevBlockNotFound")
    (genesis predecessor was also pruned)

  Verification is permanently impossible for this transaction.
```
