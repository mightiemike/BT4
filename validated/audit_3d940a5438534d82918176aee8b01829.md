### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables Theft of usdcE from Direct Deposit Accounts — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` is a publicly callable function (no `onlyOwner` guard) that is supposed to atomically swap USDC from the caller into a Direct Deposit Account (DDA) in exchange for releasing usdcE back to the caller. The USDC pull at line 616 uses a raw `IERC20Base.transferFrom` call whose boolean return value is never checked. If the USDC token returns `false` on failure instead of reverting, the function continues past the failed pull, withdraws usdcE from the DDA, and transfers it to the caller — who provided nothing.

---

### Finding Description

`replaceUsdcEWithUsdc` executes the following sequence:

```
1. balance = usdcE.balanceOf(directDepositV1)
2. usdc.transferFrom(msg.sender, directDepositV1, balance)   ← raw call, return ignored
3. DirectDepositV1(directDepositV1).withdraw(usdcE)          ← drains usdcE to ContractOwner
4. usdcE.safeTransfer(msg.sender, balance)                   ← sends usdcE to caller
```

Step 2 uses the raw interface method `IERC20Base.transferFrom` directly, not the `ERC20Helper.safeTransferFrom` wrapper that the rest of the codebase uses consistently. The return value of `transferFrom` is silently discarded. If the call returns `false` (i.e., the caller has no USDC balance or allowance but the token does not revert), steps 3 and 4 still execute, draining usdcE from the DDA and sending it to the attacker.

The function has no `onlyOwner` or `onlyDeployer` modifier — only a chain-ID gate (`require(block.chainid == 57073)`), making it reachable by any unprivileged caller on Ink Chain. [1](#0-0) 

Compare with the safe wrapper used everywhere else in the same file and in `EndpointStorage`: [2](#0-1) [3](#0-2) 

---

### Impact Explanation

An attacker who calls `replaceUsdcEWithUsdc(subaccount)` for any DDA that holds usdcE, without holding or approving any USDC, can receive the full usdcE balance of that DDA at zero cost if the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` returns `false` on a failed `transferFrom` rather than reverting. The corrupted asset delta is: usdcE balance of the targeted DDA is reduced to zero; the attacker's wallet gains that usdcE; the DDA receives no USDC in exchange. This is a direct, irreversible token theft from user-owned DDAs. [4](#0-3) 

---

### Likelihood Explanation

The function is unconditionally reachable by any unprivileged caller on Ink Chain (chain ID 57073). Exploitability is gated on whether the hardcoded USDC implementation at `0x2D270e6886d130D724215A266106e6832161EAEd` returns `false` on failure or reverts. Standard Circle USDC reverts; however, bridged or wrapped USDC variants on L2s sometimes follow the older ERC-20 pattern of returning `false`. The code defect is unconditional — the missing return-value check is present regardless of token behavior — and any future token upgrade or redeployment that changes revert behavior would immediately make this exploitable. Likelihood is **medium**: the code path is fully open to unprivileged callers, and the only barrier is the specific runtime behavior of one external token contract. [5](#0-4) 

---

### Recommendation

Replace the raw `transferFrom` call at line 616 with the `ERC20Helper.safeTransferFrom` wrapper already imported and used throughout the contract:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ERC20Helper.safeTransferFrom` wraps the call in a low-level `call`, checks `success`, and decodes the return bool, reverting on any failure — consistent with how every other token transfer in the protocol is handled. [2](#0-1) 

---

### Proof of Concept

1. Identify a DDA (`directDepositV1Address[subaccount]`) that holds a non-zero usdcE balance.
2. As an unprivileged EOA with zero USDC balance and zero USDC allowance to `ContractOwner`, call:
   ```solidity
   ContractOwner(contractOwner).replaceUsdcEWithUsdc(subaccount);
   ```
3. If the USDC token returns `false` on the failed `transferFrom` (rather than reverting):
   - `DirectDepositV1.withdraw(usdcE)` executes, pulling all usdcE from the DDA into `ContractOwner`.
   - `usdcE.safeTransfer(msg.sender, balance)` executes, sending the full usdcE balance to the attacker.
4. The attacker receives `balance` usdcE tokens; the DDA's usdcE balance is zero; no USDC was ever transferred. [6](#0-5) [7](#0-6)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
