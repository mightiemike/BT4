### Title
Unbounded `NlpLockedBalanceQueue` Growth Can Permanently Lock User NLP Tokens via Out-of-Gas in `tryUnlockNlpBalance` — (`core/contracts/SpotEngineState.sol`)

---

### Summary

The `NlpLockedBalanceQueue` stored per-subaccount in `SpotEngineState.sol` grows by one entry for every `MintNlp` transaction executed at a distinct oracle timestamp. The `tryUnlockNlpBalance` function processes all unlocked entries in a single unbounded `while` loop with no gas cap. A user who accumulates a sufficiently large queue will find that once all entries pass the 4-day lock period, any call to `burnNlp` (which requires `tryUnlockNlpBalance` to succeed) runs out of gas, permanently locking their NLP tokens in the protocol.

---

### Finding Description

**Queue growth — `SpotEngine.sol` `handleNlpLockedBalance`:**

Each `MintNlp` transaction routes through `Clearinghouse.mintNlp` → `spotEngine.updateBalance(NLP_PRODUCT_ID, ...)` → `handleNlpLockedBalance`. Inside that function, a new `NlpLockedBalance` entry is appended to the queue whenever the last entry's `unlockedAt` timestamp differs from `getOracleTime() + NLP_LOCK_PERIOD`: [1](#0-0) 

The merging guard only collapses two mints into one entry if they share the exact same oracle second. Any two mints separated by even one oracle tick produce two distinct entries. `NLP_LOCK_PERIOD` is 4 days and `HEALTHCHECK_FEE` is $1 per mint. [2](#0-1) 

There is **no cap** on `balanceCount`: [3](#0-2) 

**Unbounded iteration — `SpotEngineState.sol` `tryUnlockNlpBalance`:**

The function iterates from `unlockedUpTo` to the first still-locked entry in a single `while` loop. If N entries all share the same `unlockedAt` (or all become unlocked simultaneously after 4 days), the loop executes N storage reads and N `delete` operations in one transaction: [4](#0-3) 

There is no partial-processing path; the loop cannot be resumed across transactions because `unlockedUpTo` is only written on success.

**Critical call site — `Clearinghouse.burnNlp`:**

`burnNlp` calls `spotEngine.getNlpUnlockedBalance(txn.sender)`, which calls `tryUnlockNlpBalance` internally, before checking whether the user holds enough unlocked NLP to burn: [5](#0-4) [6](#0-5) 

If `tryUnlockNlpBalance` reverts due to out-of-gas, `burnNlp` reverts, and the user's NLP balance is permanently inaccessible.

---

### Impact Explanation

A user who has submitted many `MintNlp` transactions at distinct oracle timestamps accumulates a large `NlpLockedBalanceQueue`. After the 4-day lock period expires, all entries become unlocked simultaneously. The next call to `burnNlp` (or any NLP balance update) triggers `tryUnlockNlpBalance`, which must iterate through every unlocked entry in a single transaction. If the entry count exceeds what fits within the block gas limit, the transaction reverts unconditionally. Because there is no mechanism to process a subset of entries, the user's NLP tokens are permanently locked — a concrete collateral-lock impact.

---

### Likelihood Explanation

Each `MintNlp` costs `HEALTHCHECK_FEE = $1`. Creating ~10,000 entries costs ~$10,000. An active liquidity provider who mints NLP daily over years naturally accumulates entries without any malicious intent. A motivated attacker targeting a specific subaccount (e.g., a whale LP whose liquidation would benefit the attacker) could deliberately inflate the queue before the victim's position becomes unhealthy, preventing liquidation and causing bad debt. The cost is bounded and the effect is permanent.

---

### Recommendation

1. **Cap queue size**: Enforce a maximum `balanceCount` per subaccount (e.g., 256 entries). Reject or merge mints that would exceed the cap.
2. **Batch processing**: Modify `tryUnlockNlpBalance` to accept a `maxEntries` parameter so callers can process the queue incrementally across multiple transactions.
3. **Separate unlock from burn**: Do not require a full queue drain inside `burnNlp`; instead, track `unlockedBalanceSum` lazily and only drain entries up to a gas-safe limit per call.

---

### Proof of Concept

1. Attacker (or victim through normal usage) submits 10,000 `MintNlp` transactions, each at a different oracle timestamp (sequencer advances time between each). Each creates one new `NlpLockedBalance` entry; `balanceCount` reaches 10,000.
2. All entries share `unlockedAt ≈ mintTime + 4 days`. After 4 days, all 10,000 entries satisfy `unlockedAt <= getOracleTime()`.
3. User submits `BurnNlp`.
4. `burnNlp` → `getNlpUnlockedBalance` → `tryUnlockNlpBalance`.
5. The `while` loop attempts 10,000 iterations, each performing a storage read and a `delete`. This exceeds the block gas limit.
6. Transaction reverts. `unlockedUpTo` is not updated (revert rolls back storage).
7. Every subsequent `BurnNlp` attempt hits the same loop and reverts. The user's NLP tokens are permanently locked with no recovery path. [7](#0-6)

    function tryUnlockNlpBalance(bytes32 subaccount)
        public
        returns (Balance memory)
    {
        NlpLockedBalanceQueue storage queue = nlpLockedBalanceQueues[
            subaccount
        ];
        while (
            queue.unlockedUpTo < queue.balanceCount &&
            queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
        ) {
            // we can unlock this balance
            queue.unlockedBalanceSum.amount += queue
                .balances[queue.unlockedUpTo]
                .balance
                .amount;
            delete queue.balances[queue.unlockedUpTo];
            queue.unlockedUpTo++;
        }

        return queue.unlockedBalanceSum;
    }
