### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Rendering Coinbase Proof Forgery Protection Useless — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still a live, publicly accessible NEAR contract method despite being deprecated. `verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase Merkle proof. Because v1 remains callable by any unprivileged NEAR account, the entire protection added in v2 can be bypassed: an attacker supplies a forged `tx_id` (an internal Merkle tree node hash) directly to v1, receives `true`, and uses that result to deceive any downstream NEAR contract that consumes the verification output.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced to mitigate the well-documented 64-byte transaction Merkle proof forgery attack (referenced in the contract's own NatSpec at line 267–268). The mitigation works by first verifying a coinbase Merkle proof at index 0 before delegating to v1: [1](#0-0) 

However, v1 itself is still declared `pub` and carries only a Rust `#[deprecated]` attribute, which is a compile-time lint — it has **no runtime enforcement**: [2](#0-1) 

The contract's own inline warning acknowledges the danger: [3](#0-2) 

v1 accepts `ProofArgs`, which contains no `coinbase_tx_id` or `coinbase_merkle_proof` fields: [4](#0-3) 

The entire verification in v1 reduces to a single Merkle root comparison with no coinbase anchor: [5](#0-4) 

This is structurally identical to the sNOTE bug: a protection mechanism (coinbase proof gate in v2) is rendered useless because the unprotected predecessor path (v1) remains open to any caller.

---

### Impact Explanation

The 64-byte forgery attack allows an attacker to supply a `tx_id` that is actually the hash of an **internal Merkle tree node** (not a real transaction). Because Bitcoin's Merkle tree uses the same double-SHA256 hash for both leaf and internal nodes, a crafted 64-byte blob can be interpreted as either. v1 computes the Merkle root from the attacker-supplied `tx_id` and proof path and returns `true` if it matches the stored block's `merkle_root` — which it will for a correctly crafted forgery.

Any downstream NEAR contract that calls `verify_transaction_inclusion` (v1) to gate a financial action (e.g., releasing bridged BTC, minting wrapped tokens, settling a cross-chain swap) will accept the forged proof and execute the action for a Bitcoin transaction that never existed.

---

### Likelihood Explanation

- No privileged role is required. `verify_transaction_inclusion` is `pub` with only `#[pause]`, meaning any NEAR account can call it when the contract is unpaused.
- The 64-byte forgery technique is publicly documented and has known tooling.
- Downstream bridge or DeFi contracts built on this light client are the natural targets; they are the exact production consumers of the verification result.
- The attacker only needs a real Bitcoin block header already stored in the contract's `headers_pool` — they do not need to submit any new headers.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or change its visibility to `pub(crate)` / make it `#[private]`. All external callers must be forced through `verify_transaction_inclusion_v2`. If backward compatibility is required, have v1 internally delegate to v2 (requiring the caller to supply coinbase proof fields), rather than the current arrangement where v2 delegates to v1.

---

### Proof of Concept

1. A real Bitcoin block at height H is stored in the contract's `headers_pool` with a known `merkle_root`.
2. Attacker identifies an internal Merkle tree node hash `N` from that block's transaction tree (derivable from public Bitcoin block data).
3. Attacker constructs a `ProofArgs` with `tx_id = N`, `tx_block_blockhash = <block at H>`, `tx_index` and `merkle_proof` chosen so that `compute_root_from_merkle_proof(N, tx_index, proof) == merkle_root`.
4. Attacker calls `verify_transaction_inclusion(proof_args)` directly on the NEAR contract.
5. v1 performs no coinbase check; it computes the root, finds it matches, and returns `true`.
6. A downstream NEAR contract that calls `verify_transaction_inclusion` to authorize a bridge withdrawal receives `true` and releases funds for a Bitcoin transaction that was never broadcast or confirmed. [6](#0-5) [7](#0-6)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,

            "Incorrect coinbase merkle proof"
        );
