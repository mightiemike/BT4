Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end user, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`MetricOmmPool.swap` passes `msg.sender` (the immediate caller) as the `sender` argument to `_beforeSwap`, which `ExtensionCalling` forwards unchanged to `SwapAllowlistExtension.beforeSwap`. When swaps are routed through `MetricOmmSimpleRouter`, `msg.sender` inside the pool is the router contract, not the end user. Any pool admin who allowlists the router to enable router-mediated access for curated users inadvertently opens the allowlist gate to every unprivileged user who calls through the router.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender`:**
`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, recipient, ...)` at L230–231. When the router calls `pool.swap(...)`, `msg.sender` inside the pool is the router address.

**Step 2 — Extension layer forwards `sender` unchanged:**
`ExtensionCalling._beforeSwap` (L149–177) encodes `sender` directly into the `abi.encodeCall` payload and dispatches it to every configured extension without modification.

**Step 3 — Allowlist checks the wrong address:**
`SwapAllowlistExtension.beforeSwap` (L37) evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router. The check becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Step 4 — Router does not forward the originating user:**
`MetricOmmSimpleRouter.exactInputSingle` (L71–80) stores the originating `msg.sender` in transient storage via `_setNextCallbackContext` solely for payment purposes, but passes no user-identity argument to `pool.swap(...)`. The same applies to `exactInput` (L103–112), `exactOutputSingle` (L135–137), and `exactOutput` (L165–181).

**Why existing guards fail:**
The `allowAllSwappers` flag is a separate escape hatch and is irrelevant here. There is no mechanism in the pool or extension layer to recover the true originating user from the router's transient storage. The allowlist check is structurally bound to the immediate `pool.swap()` caller.

## Impact Explanation

A curated pool's swap allowlist is completely bypassed for any user who routes through `MetricOmmSimpleRouter` once the router is allowlisted. The pool's core access-control invariant — that only explicitly allowlisted addresses may swap — fails open. Non-allowlisted users can execute swaps against the pool, causing direct loss of LP principal under conditions the pool admin explicitly intended to prevent (adverse-selection windows, restricted-participant pools, compliance-gated venues). This constitutes broken core pool functionality causing loss of funds and an admin-boundary break via an unprivileged path.

## Likelihood Explanation

Likelihood is high. `MetricOmmSimpleRouter` is the canonical periphery swap path. Any pool admin deploying a curated pool who wants allowlisted users to access standard UX or multi-hop routing will naturally allowlist the router. There is no on-chain signal that doing so voids the allowlist for all other users. The bypass requires no special privilege, no flash loan, and no contract deployment — a plain `exactInputSingle` call from any EOA suffices and is repeatable indefinitely.

## Recommendation

The `sender` forwarded to `beforeSwap` must represent the economic actor (the end user), not the immediate caller of `pool.swap()`.

1. **In `MetricOmmSimpleRouter`**: expose the originating `msg.sender` stored in transient storage (already tracked by `_setNextCallbackContext` / `_getPayer()`) via a public getter so the pool or extension can read it.
2. **In `SwapAllowlistExtension.beforeSwap`**: if the immediate `sender` is a known router, query the router's stored originator and check that address against `allowedSwapper[pool][originator]` instead.

Alternatively, require the router to populate `extensionData` with the originating user's address, and have the extension decode and verify it — but this equally requires a router-side change and must be authenticated to prevent spoofing.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured.
  - Admin: setAllowedToSwap(pool, alice, true)   // alice is the only allowed swapper
  - Admin: setAllowedToSwap(pool, router, true)  // allowlist router so alice can use it

Attack (executed by bob, who is NOT allowlisted):
  1. bob calls MetricOmmSimpleRouter.exactInputSingle({
         pool: pool, zeroForOne: true, amountIn: X, recipient: bob, ...
     })
  2. Router calls IMetricOmmPoolActions(pool).swap(bob, true, X, ...) — msg.sender = router.
  3. MetricOmmPool.swap calls _beforeSwap(msg.sender=router, recipient=bob, ...).
  4. ExtensionCalling._beforeSwap encodes sender=router and dispatches to SwapAllowlistExtension.
  5. SwapAllowlistExtension.beforeSwap checks allowedSwapper[pool][router] == true → passes.
  6. Swap executes. bob receives output tokens.

Result: bob, a non-allowlisted user, successfully swaps against a pool
        configured to allow only alice.
```
