### Title
Protobuf `ValidResourceBounds` Deserialization Heuristic Produces Wrong Transaction Hash for `AllResources` Transactions with Zero L2/L1DataGas — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic to decide which enum variant to reconstruct. When a V3 transaction is submitted via RPC with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`, the gateway computes a 4-element Poseidon hash (including a zero `L1_DATA_GAS` slot). After the transaction is serialized to protobuf and deserialized on a syncing node, the heuristic silently downgrades the bounds to `ValidResourceBounds::L1Gas(X)`, which produces a 3-element Poseidon hash. The two hashes are permanently different, causing block-sync hash validation to fail or wrong transaction hashes to be stored.

### Finding Description

**Two incompatible converters exist for the same protobuf type:**

`crates/apollo_protobuf/src/converters/rpc_transaction.rs` (used by the gateway/mempool P2P path) converts `protobuf::ResourceBounds` directly to `AllResourceBounds`, always preserving the `AllResources` variant: [1](#0-0) 

`crates/apollo_protobuf/src/converters/transaction.rs` (used by the canonical `InvokeTransactionV3` / block-sync path) converts `protobuf::ResourceBounds` to `ValidResourceBounds` using a value-based heuristic: [2](#0-1) 

The heuristic at line 431 maps any protobuf message where `l1_data_gas.is_zero() && l2_gas.is_zero()` to `ValidResourceBounds::L1Gas(l1_gas)`, regardless of whether the original transaction was created as `AllResources`.

**The serialization direction is also lossy.** `From<ValidResourceBounds> for protobuf::ResourceBounds` emits identical bytes for both `L1Gas(X)` and `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`: [3](#0-2) 

**The hash function branches on the variant.** `get_tip_resource_bounds_hash` includes a `L1_DATA_GAS` element only for `AllResources`: [4](#0-3) 

- `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` → `poseidon(tip, concat(X, L1_GAS), concat(0, L2_GAS), concat(0, L1_DATA_GAS))` — **4 elements**
- `L1Gas(X)` → `poseidon(tip, concat(X, L1_GAS), concat(0, L2_GAS))` — **3 elements**

These are cryptographically distinct values.

**The gateway always uses `AllResources`.** `InternalRpcInvokeTransactionV3` stores `resource_bounds: AllResourceBounds` and its `InvokeTransactionV3Trait` impl unconditionally wraps it: [5](#0-4) 

The hash is computed from this `AllResources` variant at gateway ingestion time: [6](#0-5) 

**The block-sync path uses `InvokeTransactionV3` with `ValidResourceBounds`.** When a syncing node deserializes a block transaction from protobuf, it goes through `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, which applies the heuristic and produces `L1Gas(X)`. The hash recomputed from this object differs from the hash committed by the proposer.

### Impact Explanation

A syncing node that receives a block containing a V3 invoke transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` will compute a transaction hash that does not match the hash committed by the proposer. This causes:

1. **Wrong transaction hash stored in the syncing node's state** — the stored hash H2 differs from the canonical H1 committed in the block, corrupting receipts, event queries, and any downstream hash-based lookups. This matches the Critical impact: *Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.*
2. **Block-sync hash validation failure** — if the syncing node verifies the transaction commitment against recomputed hashes, it will reject valid blocks, matching the High impact: *Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.*

### Likelihood Explanation

The trigger is an unprivileged user submitting a V3 invoke transaction with only `l1_gas` set (l2_gas and l1_data_gas both zero). The gateway stateless validator explicitly accepts this pattern: [7](#0-6) 

No special permissions or malicious peer are required. Any ordinary user transaction of this form will trigger the divergence once the block is synced.

### Recommendation

Remove the value-based heuristic from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. The protobuf schema should carry an explicit discriminant (e.g., a boolean `is_all_resources` field, or a `oneof` wrapper) so the deserializer can reconstruct the exact variant without inspecting field values. Alternatively, align the block-sync path to use `AllResourceBounds` (the same type used by the gateway and mempool P2P path) so that the hash computation is always the 4-element form for V3 transactions.

### Proof of Concept

1. Submit a V3 invoke transaction via RPC with `resource_bounds = AllResourceBounds { l1_gas: ResourceBounds { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: ResourceBounds::default(), l1_data_gas: ResourceBounds::default() }`.
2. The gateway converts it to `InternalRpcInvokeTransactionV3` with `resource_bounds: AllResourceBounds { ... }`. The `InvokeTransactionV3Trait::resource_bounds()` returns `ValidResourceBounds::AllResources(...)`. `get_tip_resource_bounds_hash` hashes 4 elements: `[tip, concat(1000/1, L1_GAS), concat(0/0, L2_GAS), concat(0/0, L1_DATA_GAS)]`. Call this **H1**.
3. The transaction is included in a block and committed with H1.
4. A syncing node receives the block via P2P. The protobuf `ResourceBounds` message has `l1_gas = {1000, 1}`, `l2_gas = {0, 0}`, `l1_data_gas = {0, 0}`.
5. `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` fires: `l1_data_gas.is_zero() && l2_gas.is_zero()` → produces `ValidResourceBounds::L1Gas(ResourceBounds { max_amount: 1000, max_price_per_unit: 1 })`.
6. `get_tip_resource_bounds_hash` hashes 3 elements: `[tip, concat(1000/1, L1_GAS), concat(0/0, L2_GAS)]`. Call this **H2**.
7. **H1 ≠ H2** — the syncing node stores or validates against the wrong hash.

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.

            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
```

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```
