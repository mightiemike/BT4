### Title
Fork Retarget Uses Mainchain Historical Block Instead of Fork Branch Block — (`contract/src/lib.rs`, `contract/src/litecoin.rs`)

---

### Summary

When validating a Litecoin fork block at a difficulty-retarget boundary, `get_next_work_required` fetches the historical interval-start block via `get_header_by_height`. That function unconditionally reads from `mainchain_height_to_header` — the canonical chain index — rather than walking the fork branch. If the fork diverges before the retarget-interval start height, the timestamp used in the difficulty calculation belongs to the mainchain block at that height, not the fork block. An attacker who controls the fork's timestamps can therefore pass retarget validation with a difficulty that the real Litecoin consensus would reject, and if the crafted fork accumulates more chainwork it is promoted to mainchain, causing a downstream bridge to treat invalid confirmations as final.

---

### Finding Description

**Root cause — `get_header_by_height` is mainchain-only:** [1](#0-0) 

The implementation resolves a height exclusively through `mainchain_height_to_header`. There is no mechanism to walk the fork branch by height; the only fork-aware traversal available is `get_prev_header`, which follows `prev_block_hash` links through `headers_pool`.

**Retarget path calls `get_header_by_height`:** [2](#0-1) 

At every retarget boundary `(prev_height + 1) % difficulty_adjustment_interval == 0`, `get_next_work_required` computes `first_block_height = prev_block_header.block_height - blocks_to_go_back` and calls `blocks_getter.get_header_by_height(first_block_height)`. The returned header's `.time` is fed directly into `calculate_next_work_required` as `first_block_time`.

**Fork submission path:** [3](#0-2) 

`submit_block_header` calls `check_target` → `check_pow` → `get_next_work_required` for every block, including fork blocks. At this point `mainchain_height_to_header` still maps `first_block_height` to the canonical-chain block, not the fork block at that height.

**Contrast with the correct traversal used elsewhere:** [4](#0-3) 

`get_prev_header` correctly resolves headers through `headers_pool` keyed by `prev_block_hash`, so it follows the fork branch. The retarget path is the only place that bypasses this and uses the height index instead.

---

### Impact Explanation

An attacker submits a Litecoin testnet fork that diverges at or before `first_block_height`. The fork block at that height carries a timestamp chosen by the attacker. Because `get_header_by_height` returns the mainchain block's timestamp instead, the retarget calculation uses the wrong `first_block_time`. By choosing the fork's divergence point and timestamps carefully, the attacker can make `actual_time_taken` in `calculate_next_work_required` appear larger than it really is, yielding a higher (easier) target. The fork blocks then satisfy PoW at lower difficulty, accumulate chainwork faster, trigger `reorg_chain`, and become the new mainchain. A downstream bridge that counts confirmations against `mainchain_tip_blockhash` will treat those confirmations as final even though the source chain would never accept this fork as canonical.

---

### Likelihood Explanation

The attacker only needs to be a trusted relayer (the `#[trusted_relayer]` gate on `submit_blocks`) and to produce scrypt-valid headers — both are within reach for a well-resourced attacker on Litecoin testnet. No privileged role beyond trusted-relayer status is required. The condition is deterministic: any fork whose divergence point is at or before the retarget-interval start height triggers the wrong lookup.

---

### Recommendation

Replace the height-based lookup in `get_next_work_required` with a backward walk along the fork branch using `get_prev_header`. Starting from `prev_block_header`, walk back `blocks_to_go_back` steps through `prev_block_hash` links. This mirrors how Bitcoin Core and the Litecoin reference implementation traverse the chain and ensures the timestamp comes from the correct branch regardless of what the mainchain index contains.

---

### Proof of Concept

1. Initialize the contract on Litecoin testnet with genesis at height `N` where `(N+1) % 2016 == 0` (retarget boundary).
2. Submit a mainchain sequence up to height `N + 2016` (one full retarget interval). The mainchain block at height `N` has timestamp `T_main`.
3. Construct a fork that diverges at height `N`: the fork block at height `N` has timestamp `T_fork = T_main + large_delta`, making the fork's apparent interval much longer.
4. Extend the fork to height `N + 2016 + 1` (the next retarget boundary). When `get_next_work_required` is called for this fork block, `get_header_by_height(N)` returns the mainchain block with `T_main`, not the fork block with `T_fork`.
5. The retarget calculation uses `T_main`, producing a difficulty that does not match what the real Litecoin node would compute for this fork. The fork passes `check_target` with an incorrect (lower) difficulty.
6. Mine the remaining fork blocks at the lower difficulty, accumulate more chainwork than the mainchain, and trigger `reorg_chain`.
7. Verify that `mainchain_tip_blockhash` now points to the attacker's fork tip and that `verify_transaction_inclusion` returns `true` for transactions in the attacker's fork blocks.
