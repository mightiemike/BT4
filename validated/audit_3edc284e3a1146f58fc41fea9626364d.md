### Title
`verify_transaction_inclusion_v2` Coinbase Leaf-vs-Node Confusion Allows Internal Merkle Node to Pass as Valid Coinbase, Returning `true` for Non-Existent Transactions — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` is designed to mitigate the 64-byte Merkle proof forgery vulnerability by requiring a valid coinbase proof at position 0. However, the function never verifies that `coinbase_tx_id` is an actual leaf node (real transaction hash). `compute_root_from_merkle_proof` is a pure hash-chain function that treats any 32-byte input identically, whether it is a leaf or an internal node. An attacker can supply an internal Merkle node hash as `coinbase_tx_id` with a shortened proof that still reconstructs the correct `merkle_root`, satisfy the length-equality guard, and then supply the same internal node hash as `tx_id` with the same proof, causing the function to return `true` for a hash that is not a real transaction in the block.

---

### Finding Description

**Entry point**: `verify_transaction_inclusion_v2` is a public, permissionless view function (only gated by `#[pause]`). [1](#0-0) 

The function performs three checks:

1. **Length equality**: `merkle_proof.len() == coinbase_merkle_proof.len()` [2](#0-1) 

2. **Coinbase Merkle check**: `compute_root_from_merkle_proof(coinbase_tx_id, 0, &coinbase_merkle_proof) == merkle_root` [3](#0-2) 

3. **Tx Merkle check** (delegated to `verify_transaction_inclusion`): `compute_root_from_merkle_proof(tx_id, tx_index, &merkle_proof) == merkle_root` [4](#0-3) 

`compute_root_from_merkle_proof` is a pure iterative hash combiner: [5](#0-4) 

It applies no constraint whatsoever on whether its first argument is a leaf or an internal node. Any 32-byte value is treated identically.

**Concrete attack for a 4-transaction block `[T0, T1, T2, T3]`:**

```
         root
        /    \
      h01    h23
     /  \   /  \
    T0  T1 T2  T3
```

Where `h01 = double_sha256(T0 || T1)` and `h23 = double_sha256(T2 || T3)` and `root = double_sha256(h01 || h23)`.

The attacker submits:
- `coinbase_tx_id = h01` (internal node, **not** the real coinbase `T0`)
- `coinbase_merkle_proof = [h23]` (length 1)
- `tx_id = h01`
- `tx_index = 0`
- `merkle_proof = [h23]` (length 1)

**Check 1** (length): `1 == 1` ✓

**Check 2** (coinbase): `compute_root_from_merkle_proof(h01, 0, [h23])` → position 0 is even → `hash(h01, h23) = root` ✓

**Check 3** (tx): `compute_root_from_merkle_proof(h01, 0, [h23])` → same computation → `root` ✓

`verify_transaction_inclusion_v2` returns **`true`** for `tx_id = h01`, which is an internal Merkle node and not a real transaction in the block.

The attacker can also prove `tx_id = h23` at `tx_index = 1` with `merkle_proof = [h01]` using `coinbase_tx_id = h01` and `coinbase_merkle_proof = [h23]` — both proofs length 1, both reconstructing `root`.

The `verify_transaction_inclusion` deprecation warning even acknowledges this class of issue for v1: [6](#0-5) 

But v2 does not close it for the coinbase input itself.

---

### Impact Explanation

Any application (bridge, escrow, oracle) that calls `verify_transaction_inclusion_v2` and acts on a `true` result without independently verifying the transaction can be deceived into treating an internal Merkle node hash as a confirmed transaction. The attacker needs only publicly available block data (all internal node hashes are computable from the block's transaction list) and a single NEAR view call. No privileged role, no key compromise, no social engineering is required.

---

### Likelihood Explanation

The block's internal node hashes are deterministically computable by anyone with access to the block's transaction list (available from any Bitcoin node or block explorer). The call is permissionless. The only precondition is that the target block is in the canonical chain stored by the light client, which is the normal operating state.

---

### Recommendation

Add an explicit check that `coinbase_tx_id` is at tree depth equal to `log2(proof_length + 1)` — i.e., that the coinbase proof length corresponds to a leaf at the bottom of the tree, not a shortened proof starting from an internal node. Concretely, the coinbase proof length must equal `ceil(log2(num_txs))` for the block, or alternatively, enforce that `coinbase_tx_id` cannot equal any value derivable from a shorter proof path. The simplest robust fix is to require the caller to supply the raw coinbase transaction bytes and compute `coinbase_tx_id = double_sha256(coinbase_tx_bytes)` on-chain, preventing substitution of internal node hashes.

---

### Proof of Concept

```rust
// Given a real canonical block with 4 transactions [T0, T1, T2, T3]:
// h01 = double_sha256(T0 || T1)
// h23 = double_sha256(T2 || T3)
// root = double_sha256(h01 || h23)  -- stored in block header

// Attacker call (no special permissions needed):
verify_transaction_inclusion_v2(ProofArgsV2 {
    tx_id:                  h01,   // internal node, NOT a real transaction
    tx_block_blockhash:     B,     // any canonical block hash
    tx_index:               0,
    merkle_proof:           vec![h23],
    coinbase_tx_id:         h01,   // same internal node as fake coinbase
    coinbase_merkle_proof:  vec![h23],
    confirmations:          0,
})
// Returns: true
// Invariant broken: h01 is not a transaction in block B
```

Both `compute_root_from_merkle_proof` calls receive identical arguments and produce `root`, satisfying both the coinbase guard and the tx-inclusion check. [5](#0-4) [7](#0-6)

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"

            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,

            "Incorrect coinbase merkle proof"

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
