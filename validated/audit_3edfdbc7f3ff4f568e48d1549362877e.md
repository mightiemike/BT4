### Title
Interest Fee Overcharging Due to `FEES_ACCOUNT` Balance Included in `totalDeposits` Fee Base - (File: `core/contracts/SpotEngineState.sol`)

---

### Summary

In `SpotEngineState._updateState()`, the protocol fee amount (`feesAmt`) is computed using `totalDeposits`, which is derived from `state.totalDepositsNormalized`. Because each prior fee accrual credits `FEES_ACCOUNT` and thereby increases `totalDepositsNormalized`, subsequent fee calculations are applied to a base that already includes previously accumulated protocol fees. This is a structural compounding overcharge: the protocol charges interest fees on its own prior fee balance, at the expense of depositors.

---

### Finding Description

In `_updateState()`, `totalDeposits` is computed as:

```solidity
int128 totalDeposits = state.totalDepositsNormalized.mul(
    state.cumulativeDepositsMultiplierX18
);
```

The protocol fee is then:

```solidity
int128 feesAmt = totalDeposits.mul(
    totalDepositRateX18 - realizedDepositRateX18
);
```

Which simplifies to:

```
feesAmt = totalDeposits * totalDepositRateX18 * INTEREST_FEE_FRACTION
```

After computing `feesAmt`, it is credited to `FEES_ACCOUNT` via `_updateBalanceNormalized`:

```solidity
_updateBalanceNormalized(state, feesAccBalance, feesAmt);
```

Inside `_updateBalanceNormalized`, when the updated `balance.amountNormalized > 0`, the following line executes:

```solidity
state.totalDepositsNormalized += balance.amountNormalized;
```

This state is then persisted via `_setState`. On the **next** call to `_updateState`, `state.totalDepositsNormalized` — and therefore `totalDeposits` — includes the `FEES_ACCOUNT` balance accumulated from all prior fee accruals. The fee calculation is therefore applied to a base that is inflated by the protocol's own previously collected fees.

`FEES_ACCOUNT` is defined as `bytes32(0)` and `INTEREST_FEE_FRACTION` is `20%`. There is no mechanism to exclude the `FEES_ACCOUNT` balance from `totalDepositsNormalized` before computing `feesAmt`.

---

### Impact Explanation

The protocol systematically overcharges interest fees. Each `updateStates` call computes `feesAmt` on a `totalDeposits` that includes the `FEES_ACCOUNT`'s accumulated balance. The overcharge per period is:

```
overcharge = FEES_ACCOUNT_balance * totalDepositRateX18 * INTEREST_FEE_FRACTION
```

This excess is extracted from depositors (who receive a slightly lower `realizedDepositRateX18` than they are entitled to) and credited to `FEES_ACCOUNT`. The overcharge compounds over time as `FEES_ACCOUNT` grows. With `INTEREST_FEE_FRACTION = 20%` and a large, active market, the accumulated deviation becomes material. Depositors receive less interest than the protocol's stated fee model guarantees.

---

### Likelihood Explanation

This occurs automatically on every `updateStates(dt)` call after the first fee accrual. `updateStates` is a routine sequencer-submitted operation with no special preconditions. No attacker action is required — the bug is structural and self-reinforcing. Likelihood is high; the only question is magnitude, which grows monotonically with protocol usage and time.

---

### Recommendation

Before computing `feesAmt`, subtract the `FEES_ACCOUNT`'s current balance from `totalDeposits` to obtain the user-owned deposit base:

```solidity
BalanceNormalized memory feesAccBalance = balances[productId][FEES_ACCOUNT];
int128 feesAccAmount = feesAccBalance.amountNormalized > 0
    ? feesAccBalance.amountNormalized.mul(state.cumulativeDepositsMultiplierX18)
    : int128(0);

int128 userTotalDeposits = totalDeposits - feesAccAmount;

int128 feesAmt = userTotalDeposits.mul(
    totalDepositRateX18 - realizedDepositRateX18
);
```

This mirrors the mitigation suggested in the original Basket.sol report: track the fee-owned portion and exclude it from the fee base.

---

### Proof of Concept

1. At `t=0`: `totalDepositsNormalized = D` (all user deposits). `FEES_ACCOUNT` balance = 0.
2. `updateStates(dt)` is called. `feesAmt = D * rate * 0.20` is credited to `FEES_ACCOUNT`. `totalDepositsNormalized` becomes `D + feesAmt/cdm`.
3. At `t=1`: `updateStates(dt)` is called again. `totalDeposits` now includes the `FEES_ACCOUNT` balance. `feesAmt` is computed on `D + feesAmt_prev`, not just `D`.
4. The excess fee charged = `feesAmt_prev * rate * 0.20` — the protocol charges 20% of the interest rate on its own prior fee balance, taken from depositors.
5. This repeats every epoch, compounding the overcharge.
