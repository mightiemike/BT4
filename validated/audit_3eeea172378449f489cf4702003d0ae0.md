Audit Report

## Title
SwapAllowlistExtension checks router address instead of original user, allowing allowlist bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
When a user swaps through `MetricOmmSimpleRouter`, the `sender` argument delivered to `SwapAllowlistExtension.beforeSwap` is the **router's address**, not the original user's address. If the pool admin allowlists the router (a natural step to let allowlisted users access the router), every unprivileged user can bypass the per-user swap restriction by routing through the public router contract.

## Finding Description

**Exact call path:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle()` — `msg.sender = user`.
2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — here `msg.sender` to the pool is the **router address**.
3. `MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, recipient, ...)` — `msg.sender` is the router.
4. `ExtensionCalling._beforeSwap` encodes `sender = msg.sender` (router) and dispatches to the extension.
5. `SwapAllowlistExtension.beforeSwap(address sender, ...)` receives `sender = router address`.
6. The guard checks: `allowedSwapper[msg.sender][sender]` → `allowedSwapper[pool][router]`.

The extension never sees the original user. The check is entirely against the router's address.

**Root cause in `SwapAllowlistExtension.beforeSwap`:**
```solidity
function beforeSwap(address sender, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```
`sender` here is the router, not the original user.

**Why existing guards are insufficient:**
- `BaseMetricExtension.onlyPool` only verifies that `msg.sender` is a registered pool — it does not recover the original user identity.
- The router stores the original `msg.sender` only in transient storage for the payment callback (`_setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn)`), but this is never forwarded to the extension.
- There is no mechanism in the pool or extension system to propagate the original caller through the router hop.

**Exploit flow:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists only specific KYC'd addresses.
2. To let those users access the router, the admin also calls `setAllowedToSwap(pool, router, true)`.
3. Any unprivileged user (not on the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle()`.
4. The pool receives `msg.sender = router`, the extension checks `allowedSwapper[pool][router] = true` → passes.
5. The non-allowlisted user executes a swap that the allowlist was designed to block.

## Impact Explanation
The swap allowlist — a core access-control mechanism — fails to enforce per-user restrictions for all router-mediated swaps. Any unprivileged user can bypass the allowlist gate by routing through the public `MetricOmmSimpleRouter`. This breaks the admin-boundary invariant: the pool admin's configured allowlist is rendered ineffective for the primary public swap entry point. Pools relying on the allowlist for compliance or risk-management purposes (e.g., restricting swaps to KYC'd counterparties) are fully exposed to unrestricted swapping by any address.

## Likelihood Explanation
The bypass requires the router to be allowlisted, which is a natural and expected admin action whenever the pool admin wants allowlisted users to be able to use the standard router. Any unprivileged user can then exploit this by simply calling the public router. The attack is trivially repeatable with no special capability required beyond a standard EOA or contract call.

## Recommendation
Forward the original user identity through the extension call chain. One approach: the router should pass the original `msg.sender` as `callbackData` or `extensionData` in a standardized format, and the pool should forward a verified "originator" field to extensions. Alternatively, `SwapAllowlistExtension` should be documented to only be used with direct pool calls (not router-mediated), and the router should expose a separate allowlist-aware entry point that verifies the caller before invoking the pool.
