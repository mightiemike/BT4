Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Original User, Allowing Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` as seen by the pool. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. The extension therefore checks whether the router is allowlisted rather than the actual swapper, completely defeating the per-address access control the extension is designed to enforce.

## Finding Description
**Root cause:** `MetricOmmPool.swap` passes `msg.sender` directly as the `sender` argument to `_beforeSwap` (L230–240). `ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension (L149–177). `SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the router (L37). `MetricOmmSimpleRouter.exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` all call `pool.swap()` directly with no mechanism to forward the original caller's address as `sender` (L71–80, L99–112, L136–137, L165–181).

**Exploit flow (Scenario A — bypass):**
1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to approved addresses.
2. Admin calls `setAllowedToSwap(pool, address(router), true)` so that allowlisted users can use the standard periphery.
3. Unprivileged attacker (not on the allowlist) calls `router.exactInputSingle(...)` targeting the curated pool.
4. Pool calls `_beforeSwap(sender=router, ...)` → extension checks `allowedSwapper[pool][router] == true` → passes.
5. Attacker's swap executes despite not being allowlisted.

**Exploit flow (Scenario B — broken functionality):**
1. Admin does not allowlist the router.
2. An individually allowlisted user calls `router.exactInputSingle(...)`.
3. Extension checks `allowedSwapper[pool][router] == false` → reverts with `NotAllowedToSwap`.
4. Allowlisted users cannot use the documented standard periphery path.

**Existing guards insufficient:** The only guard is `allowedSwapper[msg.sender][sender]` in the extension. There is no secondary check on the original transaction initiator, no `tx.origin` guard, and no originator field in the swap hook arguments. The router stores the original `msg.sender` only in transient storage for the payment callback (`_setNextCallbackContext`), but this value is never surfaced to the pool or extension.

## Impact Explanation
Scenario A constitutes a broken core pool access-control invariant: an unprivileged trader can swap in a pool explicitly configured to restrict swaps to approved addresses. Depending on the pool's purpose (institutional-only liquidity, rate-limited access, KYC-gated pools), this allows unauthorized parties to extract favorable oracle-anchored prices or drain LP value from a curated pool. This is a direct loss of LP principal and a bypass of an admin-configured access boundary by an unprivileged path. Scenario B renders the core swap flow unusable for allowlisted users via the standard periphery, constituting broken core pool functionality.

## Likelihood Explanation
`MetricOmmSimpleRouter` is a public, permissionless contract callable by any address. A pool admin who wants allowlisted users to use the router has no alternative but to allowlist the router address, which is the natural and expected admin action. No privileged access, special token behavior, or off-chain oracle manipulation is required. The attacker only needs to call `exactInputSingle` on the router targeting the curated pool. The condition is trivially reachable and repeatable.

## Recommendation
**Short term:** In `SwapAllowlistExtension.beforeSwap`, also check the `recipient` parameter, or require pools using this extension to pass the original initiator via `extensionData` and validate it there. Alternatively, document and enforce at the factory level that `SwapAllowlistExtension` is incompatible with router-mediated swaps (e.g., reject pool configurations that combine a swap allowlist extension with any known router address as a potential caller).

**Long term:** Introduce an `originator` field in the swap hook arguments (analogous to Uniswap v4's `hookData` originator pattern) so extensions can gate on the true economic actor rather than the immediate `msg.sender` of the pool. The router would populate this field with `msg.sender` before calling `pool.swap`.
