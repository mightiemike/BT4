### Title
Protobuf `ValidResourceBounds` Deserialization Collapses `AllResources(l1_gas, 0, 0)` to `L1Gas`, Producing a Different Transaction Hash Preimage - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf converter for `ValidResourceBounds` uses the condition `l1_data_gas.is_zero() && l2_gas.is_zero()` to decide whether to produce a `L1Gas` or `AllResources` variant. An `AllResources` transaction with both `l2_gas = 0` and `l1_data_gas = 0` is silently collapsed to `L1Gas` after a protobuf round-trip. Because `get_tip_resource_bounds_hash` hashes a **different number of resource elements** for each variant (2 for `L1Gas`, 3 for `AllResources`), the transaction hash preimage changes across the protobuf boundary, breaking the hash/signature domain invariant.

### Finding Description

**Root cause — protobuf deserialization (`crates/apollo_protobuf/src/converters/transaction.rs`, lines 417–436):**

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)          // ← variant collapses here
        } else {
            ValidResourceBounds::AllResources(...)
        })
    }
}
``` [1](#0-0) 

The condition `l1_data_gas.is_zero() && l2_gas.is_zero()` is evaluated on the **deserialized values**, not on the presence/absence of the protobuf field. When a sender transmits an `AllResources` transaction with both bounds set to zero, the receiving node silently reclassifies it as `L1Gas`.

**Hash preimage divergence — `get_tip_resource_bounds_hash` (`crates/starknet_api/src/transaction_hash.rs`, lines 188–211):**

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];
    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],                          // ← 2 elements
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // ← 3 elements
        }
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
``` [2](#0-1) 

The two variants produce structurally different Poseidon preimages:

| Variant | Preimage elements |
|---|---|
| `L1Gas` | `[tip, L1_GAS_packed, L2_GAS_packed]` |
| `AllResources` | `[tip, L1_GAS_packed, L2_GAS_packed, L1_DATA_GAS_packed]` |

**The divergence path:**

1. A user submits a V3 transaction via RPC. The `RpcTransaction` type always carries `AllResourceBounds` (never `ValidResourceBounds`), so the gateway always creates `AllResources`: [3](#0-2) 

2. `convert_rpc_tx_to_internal` computes `tx_hash` from the `AllResources` body — a 3-element Poseidon hash: [4](#0-3) 

3. The `From<ValidResourceBounds> for protobuf::ResourceBounds` serializer faithfully emits all three fields (including zero-valued `l1_data_gas`): [5](#0-4) 

4. The receiving node deserializes: `l1_data_gas.is_zero() && l2_gas.is_zero()` → `L1Gas`. The hash recomputed from this body is a 2-element Poseidon hash — **different from the stored `tx_hash`**.

5. The blockifier's `calculate_resource_bounds` and pre-validation logic branch on the variant: [6](#0-5) 

In `L1Gas` mode only L1 gas is checked; L2 gas and L1 data gas bounds are ignored. In `AllResources` mode all three are enforced independently.

### Impact Explanation

**Hash/signature domain broken (Critical/High):** The transaction was signed over the `AllResources` hash (3-element preimage). After protobuf deserialization the body carries `L1Gas`, so any component that recomputes the hash from the body — including consensus validators verifying block proposals — will derive a different hash. This causes a valid, properly-signed transaction to appear invalid, or binds the wrong hash to the executable payload.

**Wrong execution mode (Critical):** The blockifier uses the `ValidResourceBounds` variant to select `GasVectorComputationMode`. A transaction deserialized as `L1Gas` is executed in `NoL2Gas` mode: L1 data
