### Title
Division-by-Zero in `PerpEngine.socializeSubaccount` When `openInterest` Is Zero Blocks Liquidation Finalization - (File: `core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when a subaccount's `vQuoteBalance` remains negative after insurance coverage. If `openInterest` is zero at that moment — a reachable state when the subaccount being finalized is the last participant in a given perp market — the call reverts with `"DBZ"`, permanently blocking the finalization of an insolvent subaccount.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, after attempting to cover a negative `vQuoteBalance` with insurance, the protocol tries to spread the remaining loss across all open-interest holders:

```solidity
// actually socialize if still not enough
if (balance.vQuoteBalance < 0) {
    int128 fundingPerShare = -balance.vQuoteBalance.div(
        state.openInterest   // ← can be 0
    );
```

`MathSD21x18.div` explicitly reverts on a zero denominator (`require(y != 0, ERR_DIV_BY_ZERO)`).

The call site is `_finalizeSubaccount` in `ClearinghouseLiq.sol`, which:
1. Requires `balance.amount == 0` for every perp product before proceeding.
2. Calls `perpEngine.socializeSubaccount(txn.liquidatee, v.insurance)` unconditionally.

Because `_finalizeSubaccount` enforces `amount == 0` for the liquidatee across all perps, the liquidatee's own contribution to `openInterest` is already zero. If no other account holds an open position in that perp (i.e., the liquidatee was the last participant), `state.openInterest == 0`. The subaccount can still carry a negative `vQuoteBalance` from a previously closed position (e.g., opened at a loss and closed at `amount = 0`). When insurance is insufficient to cover it, the division fires and reverts.

---

### Impact Explanation

The revert in `liquidateSubaccountImpl` → `_finalizeSubaccount` → `perpEngine.socializeSubaccount` means the insolvent subaccount **can never be finalized**. Bad debt remains permanently unresolved in the protocol. The insurance fund cannot be correctly updated, and the socialization mechanism — the last line of defense against protocol insolvency — is rendered inoperable for this subaccount. This is a solvency/accounting corruption impact, not merely a denial of service.

---

### Likelihood Explanation

The condition is reachable in any low-liquidity perp market where the liquidatee was the sole or last remaining participant. A subaccount can legitimately reach `amount == 0` with `vQuoteBalance < 0` by opening a position, incurring a loss, and closing the position before being finalized. If no other account opens a position in that perp in the interim, `openInterest` drops to zero. The liquidator (an unprivileged user) triggers `liquidateSubaccountImpl` with `productId = type(uint32).max` to initiate finalization, which is the standard protocol path.

---

### Recommendation

Guard the division with a zero-check on `openInterest` before attempting to socialize:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No participants to socialize against; absorb into insurance or skip
        state.availableSettle += balance.vQuoteBalance; // adjust as appropriate
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

Alternatively, route any residual loss to the insurance fund or mark it as protocol bad debt rather than attempting to divide by zero open interest.

---

### Proof of Concept

1. Deploy a perp market with product ID `P`.
2. Account A opens a long position of size `S` at price `X`, giving `vQuoteBalance = -S*X` and `openInterest = S`.
3. Account A closes the position at price `X - delta` (a loss), giving `amount = 0`, `vQuoteBalance = -S*delta`, `openInterest = 0`.
4. Account A's quote balance falls below maintenance health (e.g., due to the loss).
5. Liquidator calls `liquidateSubaccountImpl` with `productId = type(uint32).max` (finalization path).
6. `_finalizeSubaccount` passes the `amount == 0` check, then calls `perpEngine.socializeSubaccount`.
7. Inside `socializeSubaccount`: `balance.vQuoteBalance < 0` is true, insurance is insufficient, so the inner `if (balance.vQuoteBalance < 0)` branch executes and calls `(-balance.vQuoteBalance).div(state.openInterest)` where `openInterest == 0`.
8. `MathSD21x18.div` reverts with `"DBZ"`. Finalization is permanently blocked. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
