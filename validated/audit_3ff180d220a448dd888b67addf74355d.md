### Title
`ValidResourceBounds` Variant Conflation in Protobuf Conversion Produces Wrong Transaction Hash and Rejects Valid `AllResources` Invoke V3 Transactions — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ResourceBounds` uses a zero-value heuristic to decide between `ValidResourceBounds::L1Gas` and `ValidResourceBounds::AllResources`. An `AllResources` Invoke V3 transaction whose `l2_gas` and `l1_data_gas` bounds are both zero is silently downgraded to `L1Gas` after any protobuf round-trip. Because `get_tip_resource_bounds_hash` hashes a different number of elements for the two variants (3 vs 4 Poseidon inputs), the transaction hash computed on the receiving node diverges from the hash computed on the originating node. Additionally, the downstream conversion `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` hard-rejects the `L1Gas` variant, so the transaction is dropped by every P2P peer that receives it.

---

### Finding Description

**Root cause — lossy protobuf conversion**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` lines 417–436:

```rust
let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // absent ≡ {0,0}
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)                      // ← heuristic

    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })

``` [1](#0-0) 

Protobuf 3 cannot distinguish "field absent" from "field present with default (zero) value". A pre-0.13.3 transaction that never set `l1_data_gas` and a post-0.13.3 `AllResources` transaction that explicitly set `l1_data_gas = {0, 0}` produce identical wire bytes. The heuristic therefore irreversibly converts the latter to `L1Gas`.

---

### Impact Explanation

Two distinct impacts:

1. **Wrong transaction hash (Critical-adjacent)**: The originating node stores the transaction under hash H1 (AllResources, 4-element Poseidon preimage). Any node that receives the transaction via protobuf and successfully processes it would compute hash H2 (L1Gas, 3-element preimage). H1 ≠ H2. This binds the wrong hash to the executable payload — matching the "Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload" High impact.

2. **Valid transaction rejected (High)**: Because `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` hard-rejects `L1Gas`, every P2P peer drops the transaction. The originating node accepted it; all other nodes reject it. This matches "Mempool/gateway/RPC admission rejects valid transactions before sequencing."

---

### Likelihood Explanation

The trigger requires an `AllResources` Invoke V3 transaction where both `l2_gas` and `l1_data_gas` have `max_amount = 0` and `max_price_per_unit = 0`. This is an unusual but protocol-valid configuration (e.g., a client-side-proving transaction that sets zero fee bounds). The RPC gateway accepts it without restriction. Once accepted, every subsequent P2P hop silently corrupts or drops it.

---

### Recommendation

Replace the zero-value heuristic with an explicit wire-level discriminant. Options:

1. Add a boolean `is_all_resources` field to the `ResourceBounds` protobuf message so the variant is unambiguous regardless of field values.
2. Alternatively, always serialize as `AllResources` on the wire (setting zero values for absent fields) and always deserialize as `AllResources`, letting the hash function handle the zero case uniformly.
3. At minimum, add a gateway-level check that rejects `AllResources` transactions where both `l2_gas` and `l1_data_gas` are fully zero, so the ambiguous case never enters the system.

---

### Proof of Concept

```
1. Craft an AllResources InvokeV3 transaction:
   resource_bounds = AllResourceBounds {
       l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
       l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },  // zero
       l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },  // zero
   }

2. Submit via RPC to node A.
   - Gateway accepts it (AllResources, valid).
   - Hash H1 = Poseidon([INVOKE, ver, sender, tip_resource_hash_4_elements, ...])
     where tip_resource_hash_4_elements = Poseidon([tip, L1_GAS, L2_GAS(0), L1_DATA_GAS(0)])
   - Stored in mempool under H1.

3. Node A propagates via P2P (MempoolTransaction or ConsensusTransaction protobuf).
   - Serialized: l1_data_gas field = {max_amount:0, max_price:0} (indistinguishable from absent).

4. Node B receives protobuf, calls TryFrom<protobuf::ResourceBounds>:
   - l1_data_gas.is_zero() && l2_gas.is_zero() → true
   - Produces ValidResourceBounds::L1Gas(l1_gas)  ← wrong variant

5. Node B calls TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3:
   - Hits the `_ => Err(OutOfRange)` arm → transaction dropped.

6. Alternatively, if node B computes the hash before rejection:
   - Hash H2 = Poseidon([INVOKE, ver, sender, tip_resource_hash_3_elements, ...])
     where tip_resource_hash_3_elements = Poseidon([tip, L1_GAS, L2_GAS(0)])
   - H1 ≠ H2 → hash mismatch, wrong executable payload bound to wrong hash.
```

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
