### Title
Builder Fee Charged on Full Matched Quote Instead of Fee-Exempt-Adjusted `meteredQuote` — (`core/contracts/OffchainExchange.sol`)

---

### Summary

In `OffchainExchange.applyFee()`, the protocol trading fee is correctly computed on `meteredQuote` — which excludes the fee-exempt portion of a taker fill below `market.minSize`. However, the builder fee is computed on `matchQuote.abs()` (the full matched quote), bypassing the same exemption. When a taker order is partially filled across multiple matches and the cumulative fill crosses the `minSize` threshold mid-fill, the builder fee is charged on the full current fill including the fee-exempt portion, causing the taker to overpay builder fees.

---

### Finding Description

`applyFee()` computes `meteredQuote` to represent the portion of the taker's fill that is subject to fees. For a taker, the first `minSize` of cumulative quote is fee-exempt (only a flat `minSize` fee applies on the first fill). On subsequent fills, the portion of the current fill that brings the cumulative total from below `minSize` to above `minSize` is also fee-exempt:

```solidity
// OffchainExchange.sol lines 536–543
int128 feeApplied = MathHelper.abs(alreadyMatched + matchQuote) -
    market.minSize;
feeApplied = MathHelper.min(feeApplied, matchQuote.abs());
if (feeApplied > 0) {
    ...
    meteredQuote += feeApplied;
}
``` [1](#0-0) 

The protocol fee is correctly applied to `meteredQuote`:

```solidity
// line 560
orderInfo.fee = meteredQuote - newMeteredQuote;
``` [2](#0-1) 

But the builder fee ignores `meteredQuote` entirely and uses the full `matchQuote.abs()`:

```solidity
// line 561
orderInfo.builderFee = matchQuote.abs().mul(feeInfo.builderFeeRate);
``` [3](#0-2) 

Both fees are then deducted from the taker's quote balance:

```solidity
// lines 562–565
orderInfo.quoteDelta =
    orderInfo.quoteDelta -
    orderInfo.fee -
    orderInfo.builderFee;
``` [4](#0-3) 

**Concrete trigger scenario:**

- `minSize = 100`, `builderFeeRate = 50 bps`
- Fill 1: `alreadyMatched = 0`, `matchQuote = 60` → `filledAmounts[digest] = 60`
- Fill 2: `alreadyMatched = -60`, `matchQuote = 80`
  - `meteredQuote = abs(-60 + -80) - 100 = 40` (only 40 of 80 is above the minSize threshold)
  - Protocol fee base: **40**
  - Builder fee base: `matchQuote.abs()` = **80** ← incorrect; should be 40
  - Excess builder fee charged: `(80 - 40) * 0.005 = 0.2` quote units per fill

The excess builder fee equals `(minSize - abs(alreadyMatched)) * builderFeeRate` for any fill that crosses the `minSize` threshold with `0 < abs(alreadyMatched) < minSize`.

---

### Impact Explanation

The taker's `quoteDelta` is reduced by an excess builder fee equal to `(minSize - abs(alreadyMatched)) * builderFeeRate`. This excess is accumulated in `collectedBuilderFee` and claimable by the builder via `claimBuilderFee()`. [5](#0-4) 

The taker suffers a direct, measurable financial loss on every partial fill that crosses the `minSize` threshold. The builder receives more fees than the protocol intends to grant them. The corrupted state is `collectedBuilderFee[quoteId][builderId]` (inflated) and the taker's quote balance (deflated).

---

### Likelihood Explanation

This is triggered in normal trading whenever:
1. A taker order is matched across multiple fills (partial fills are common in order-book DEXes).
2. The cumulative fill crosses `minSize` in a fill where `alreadyMatched > 0`.
3. The order uses a builder with a non-zero `builderFeeRate`.

Any builder can set `builderFeeRate` up to `builder.highestFeeRate` via the `appendix` field of the order. A builder can maximize the excess by setting a high fee rate. The taker has no way to avoid this because the fee is applied by the protocol during `matchOrders`, which is called by the sequencer. [6](#0-5) 

---

### Recommendation

Replace `matchQuote.abs()` with `meteredQuote.abs()` in the builder fee calculation so that the same fee-exempt portion applies to both the protocol fee and the builder fee:

```solidity
// Before (incorrect):
orderInfo.builderFee = matchQuote.abs().mul(feeInfo.builderFeeRate);

// After (correct):
orderInfo.builderFee = meteredQuote.abs().mul(feeInfo.builderFeeRate);
``` [3](#0-2) 

---

### Proof of Concept

**Setup:**
- `minSize = 100e18`, `builderFeeRate = 100 bps (0.01)`
- Taker places an order to buy base, total `amount = 200` base units at price `1.0`

**Fill 1** (`alreadyMatched = 0`, `matchQuote = -60`):
- `meteredQuote = -100` (flat minSize fee, since `abs(matchQuote) < minSize`)
- `builderFee = 60 * 0.01 = 0.6` ← charged on actual 60
- Protocol fee base: 100 (flat, intentional)

**Fill 2** (`filledAmounts[digest] = 60`, `alreadyMatched = -60`, `matchQuote = -80`):
- `feeApplied = abs(-60 + -80) - 100 = 40`
- `meteredQuote = 40` (only 40 of 80 is above minSize)
- Protocol fee base: **40**
- `builderFee = 80 * 0.01 = 0.8` ← charged on full 80
- Correct builder fee should be: `40 * 0.01 = 0.4`
- **Excess builder fee: 0.4 quote units** extracted from taker, credited to builder

The taker loses `0.4` extra quote units per such fill. Over many fills and with a high `builderFeeRate`, this constitutes a meaningful and systematic overcharge.
