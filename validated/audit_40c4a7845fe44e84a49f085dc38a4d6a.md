### Title
`paid_fee_on_l1` Silently Hardcoded to `Fee(1)` in Consensus L1 Handler Conversion, Bypassing Fee Enforcement - (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary

`convert_consensus_l1_handler_to_internal_l1_handler` converts a `ConsensusTransaction::L1Handler` into an `InternalConsensusTransaction::L1Handler` (executable). During this conversion the actual fee paid on L1 is not forwarded — it is replaced with the hardcoded constant `Fee(1)`. This is the direct Sequencer analog of the external bug: a conversion function that receives a value from one representation but never routes it to the final executable object, leaving the enforcement logic permanently bypassed.

### Finding Description

`ConsensusTransaction::L1Handler` wraps `transaction::L1HandlerTransaction`, which carries no `paid_fee_on_l1` field. When a validator receives a block proposal containing an L1 handler transaction via the consensus path, `convert_consensus_tx_to_internal_consensus_tx` dispatches to `convert_consensus_l1_handler_to_internal_l1_handler`:

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
``` [1](#0-0) 

The `executable_transaction::L1HandlerTransaction` that reaches the blockifier always carries `paid_fee_on_l1 = Fee(1)`, regardless of what was actually paid on L1. The blockifier's fee check is:

```rust
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(...InsufficientFee...));
}
``` [2](#0-1) 

Because `Fee(1) != Fee(0)`, this check always passes. The `ConsensusTransaction` type itself does not carry `paid_fee_on_l1` for L1 handler transactions:

```rust
pub enum ConsensusTransaction {
    RpcTransaction(RpcTransaction),
    L1Handler(transaction::L1HandlerTransaction),  // no paid_fee_on_l1
}
``` [3](#0-2) 

So the conversion path has no source from which to read the real fee — the field is structurally absent from the consensus wire type, and the conversion silently substitutes a placeholder.

### Impact Explanation

Every L1 handler transaction processed through the consensus path (i.e., received by a validator as part of a block proposal) is executed with `paid_fee_on_l1 = Fee(1)`. The sequencer's own fee-sufficiency check is permanently bypassed for this entire class of transactions. A proposer can include L1 handler transactions whose actual on-chain fee is zero or below the required threshold; the sequencer will execute them, produce receipts, and commit state changes as if the fee were valid. The resulting receipts and state diffs diverge from what the L1 core contract would consider legitimate, producing wrong execution results for accepted input.

This matches: **High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload** — the conversion binds the wrong fee value to the executable object — and **Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input** — the blockifier executes with a fabricated fee and produces receipts that do not reflect the true L1 payment.

### Likelihood Explanation

This is triggered unconditionally for every L1 handler transaction that passes through the consensus conversion path. No special attacker capability is required beyond being a block proposer (or having a proposer include an L1 handler transaction). The code path is reachable in normal operation.

### Recommendation

Add `paid_fee_on_l1: Fee` to `ConsensusTransaction::L1Handler` (or to a wrapper type) so the actual fee is carried through the consensus wire format. Update `convert_consensus_l1_handler_to_internal_l1_handler` to read the fee from the incoming transaction rather than substituting `Fee(1)`. The existing TODO comment acknowledges this gap; it should be treated as a correctness requirement, not a deferred enhancement.

### Proof of Concept

1. A proposer constructs a block proposal containing a `ConsensusTransaction::L1Handler` whose corresponding L1 message paid zero fee.
2. A validator calls `convert_consensus_tx_to_internal_consensus_tx`, which dispatches to `convert_consensus_l1_handler_to_internal_l1_handler`.
3. The conversion produces `executable_transaction::L1HandlerTransaction { paid_fee_on_l1: Fee(1), … }`.
4. The blockifier executes the transaction; the check `if paid_fee == Fee(0)` evaluates to `false` (since `Fee(1) != Fee(0)`), so execution proceeds and state is committed.
5. The resulting receipt records a successful execution with `paid_fee_on_l1 = Fee(1)`, diverging from the actual L1 payment of zero. [4](#0-3) [1](#0-0) [5](#0-4)
