Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Original User, Enabling Allowlist Bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the original user. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. Any user can bypass a curated pool's individual allowlist by routing through the public router if the router address is allowlisted, or allowlisted users are silently blocked from using the router if it is not.

## Finding Description

**Call path:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle()` (`msg.sender = user`).
2. Router calls `IMetricOmmPoolActions(pool).swap(recipient, zeroForOne, amount, priceLimit, "", extensionData)` — at this point `msg.sender` inside the pool is the **router address**.
3. `MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)` passing the router address as `sender`.
4. `ExtensionCalling._beforeSwap(sender=router, ...)` encodes and dispatches `IMetricOmmExtensions.beforeSwap(sender=router, ...)` to every configured extension.
5. `SwapAllowlistExtension.beforeSwap(address sender, ...)` executes:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct) and `sender` is the **router address** (wrong — should be the original user). The check resolves to `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Root cause:** The pool has no mechanism to propagate the original EOA/caller through the router into the extension. The router never passes the original `msg.sender` to the pool, and the pool passes its own `msg.sender` (the router) as `sender` to extensions.

**Existing guards are insufficient:** The `onlyPool` modifier in `BaseMetricExtension` only verifies the caller is a registered pool — it does not verify the identity of the economic actor. The pool-level `NotAllowedToSwap` error path is only reached after the extension already evaluated the wrong address.

**Bypass scenario:** A pool admin allowlists specific users and also allowlists the router (a natural production setup to allow router-mediated swaps). Any non-allowlisted user can then call `exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` on the router targeting that pool and the extension will pass because `allowedSwapper[pool][router] == true`.

**Broken-functionality scenario:** If the router is not allowlisted, allowlisted users cannot swap through the router at all, defeating the purpose of the periphery for curated pools.

## Impact Explanation
Direct policy bypass on curated pools: unauthorized traders can execute swaps that the pool's allowlist was designed to prevent. This constitutes a broken core swap flow and an admin-boundary break (the pool admin's curation policy is bypassed by an unprivileged path). Severity: **High**.

## Likelihood Explanation
Any unprivileged user can call the public router. No special preconditions beyond the pool having `SwapAllowlistExtension` configured and the router being allowlisted (a necessary production condition for the router to function at all). The bypass is deterministic and repeatable with zero cost beyond gas.

## Recommendation
Pass the original caller's identity through the swap path so the extension can gate the economically relevant actor. Options:
- Add an `originator` field to the pool's `swap()` calldata that the router populates with `msg.sender`, and have `ExtensionCalling._beforeSwap` forward it to extensions alongside `sender`.
- Alternatively, have `SwapAllowlistExtension` decode an allowlisted address from `extensionData` and verify it matches a signature from the original user — but this is complex and error-prone.
- The cleanest fix is to extend the `beforeSwap` hook signature with an explicit `originator` address that the pool sets to the router-supplied value, and update `SwapAllowlistExtension` to check `originator` instead of `sender`.

## Proof of Concept

```solidity
// 1. Deploy pool with SwapAllowlistExtension
// 2. Pool admin allowlists the router: extension.setAllowedToSwap(pool, address(router), true)
// 3. Pool admin does NOT allowlist attacker: allowedSwapper[pool][attacker] == false
// 4. Attacker calls:
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: token0,
    tokenOut: token1,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    recipient: attacker,
    deadline: block.timestamp + 1,
    priceLimitX64: 0,
    extensionData: ""
}));
// Extension checks allowedSwapper[pool][router] == true → passes
// Attacker receives token1 despite not being individually allowlisted
```
