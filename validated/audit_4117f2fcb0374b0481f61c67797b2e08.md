### Title
Unbounded Fork-Length Iteration in `reorg_chain` Causes Gas Exhaustion, Permanently Corrupting Canonical Chain — (File: `contract/src/lib.rs`)

---

### Summary

`reorg_chain` contains two unbounded loops whose iteration count equals the fork depth. Because no on-chain cap exists on fork length, a relayer can accumulate an arbitrarily long fork across many `submit_blocks` calls. When the fork's accumulated chain work finally exceeds the main chain, `reorg_chain` is triggered inside a single NEAR transaction. If the fork is deep enough, the 300 TGas budget is exhausted, the transaction reverts, and the contract is permanently stuck reporting the wrong canonical chain tip — corrupting every downstream `verify_transaction_inclusion` result.

---

### Finding Description

`reorg_chain` is called from `submit_block_header_inner` whenever a fork's `chain_work` exceeds the main chain's: [1](#0-0) 

Inside `reorg_chain`, two separate unbounded loops execute:

**Loop 1** — removes every main-chain block above the fork tip height: [2](#0-1) 

**Loop 2** — walks the fork chain backwards, one storage read + two storage writes + one storage remove per iteration, until it reaches the common ancestor: [3](#0-2) 

Each iteration of Loop 2 performs at minimum five storage operations (`contains_key`, `insert` × 2, `remove_block_header` which itself does two removes, and `headers_pool.get`). At NEAR's storage-operation gas costs, a fork of ~50–100 blocks is sufficient to exhaust the 300 TGas budget.

The contract imposes **no on-chain cap** on fork length. The only limit (`max_fork_len = 500`) lives in the off-chain relayer config: [4](#0-3) 

This is not enforced by the contract. Fork blocks are stored in `headers_pool` across many separate `submit_blocks` transactions, each individually within gas budget. The reorg is only triggered — and the full iteration must complete atomically — when the final block tips the chain-work comparison: [5](#0-4) 

When the transaction reverts due to gas exhaustion, the fork blocks remain in `headers_pool` (written in prior committed transactions), but `mainchain_tip_blockhash` is never updated. Every subsequent block extending the fork re-triggers the same failing reorg. The contract is permanently stuck.

---

### Impact Explanation

`mainchain_tip_blockhash` is never updated to the heavier fork. All downstream consumers are affected:

- `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` check `mainchain_header_to_height` to confirm a block belongs to the canonical chain. Blocks on the stale chain pass; blocks on the true canonical chain fail. [6](#0-5) 

- `get_last_block_header` / `get_last_block_height` return the stale tip, misleading any bridge or cross-chain application that relies on the contract's view of the Bitcoin chain. [7](#0-6) 

The corrupted state is **permanent**: no recovery path exists without a privileged migration, because the fork blocks are already committed to storage and every new block extending the fork re-triggers the same out-of-gas reorg.

---

### Likelihood Explanation

The attack is staged across many ordinary `submit_blocks` calls, each individually within gas budget. The relayer role is the explicit production entry point for adversarial chain data per the audit scope. On Bitcoin mainnet a deep fork requires significant hash power, but on testnet, Dogecoin, Litecoin, or Zcash (all supported by this contract) the PoW threshold is far lower. Additionally, `skip_pow_verification` can be set to `true` at init time for testing deployments, removing even that barrier. The off-chain `max_fork_len` config is not enforced on-chain and can be bypassed by any caller who submits directly to the contract.

---

### Recommendation

Enforce an on-chain cap on the fork depth before entering `reorg_chain`. Introduce a constant (e.g., `MAX_FORK_LEN`) and assert the fork depth does not exceed it:

```rust
// In submit_block_header_inner, before calling reorg_chain:
let fork_depth = current_header.block_height
    .saturating_sub(common_ancestor_height);
require!(
    fork_depth <= MAX_FORK_LEN,
    "Fork depth exceeds maximum allowed reorg length"
);
```

Alternatively, split the reorg into a paginated, multi-transaction operation so no single call iterates over an unbounded number of blocks.

---

### Proof of Concept

1. Deploy the contract with `skip_pow_verification = true` (or use a low-difficulty testnet).
2. Submit N main-chain blocks via `submit_blocks` to establish a main chain of height N.
3. Submit N+1 fork blocks (each extending the same ancestor, with slightly higher cumulative work) across N+1 separate `submit_blocks` calls. Each call stores one fork block in `headers_pool` and returns successfully.
4. Submit the final fork block whose `chain_work` exceeds the main chain tip. This triggers `reorg_chain`, which must iterate over all N+1 fork blocks in a single transaction.
5. For N ≥ ~60 (exact threshold depends on chain variant and storage costs), the transaction exhausts 300 TGas and reverts.
6. Observe that `get_last_block_header()` still returns the stale main-chain tip, and `verify_transaction_inclusion` accepts proofs for stale-chain blocks while rejecting proofs for the true canonical chain. [8](#0-7)

    fn reorg_chain(&mut self, fork_tip_header: ExtendedHeader, last_main_chain_block_height: u64) {
        let fork_tip_height = fork_tip_header.block_height;
        if last_main_chain_block_height > fork_tip_height {
            // If we see that main chain is longer than fork we first garbage collect
            // outstanding main chain blocks:
            //
            //      [m1] - [m2] - [m3] - [m4] <- We should remove [m4]
            //     /
            // [m0]
            //     \
            //      [f1] - [f2] - [f3]
            for height in (fork_tip_height + 1)..=last_main_chain_block_height {
                let current_main_chain_blockhash = self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str("cannot get a block"));
                self.remove_block_header(&current_main_chain_blockhash);
                self.mainchain_height_to_header.remove(&height);
            }
        }

        // Now we are in a situation where mainchain is equivalent to fork size:
        //
        //      [m1] - [m2] - [m3] - [m4] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip
        //
        //
        // Or in a situation where it is shorter:
        //
        //      [m1] - [m2] - [m3] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip

        let fork_tip_hash = fork_tip_header.block_hash.clone();
        let mut fork_header_cursor = fork_tip_header;

        while !self
            .mainchain_header_to_height
            .contains_key(&fork_header_cursor.block_hash)
        {
            let prev_block_hash = fork_header_cursor.block_header.prev_block_hash;
            let current_block_hash = fork_header_cursor.block_hash;
            let current_height = fork_header_cursor.block_height;

            // Inserting the fork block into the main chain, if some mainchain block is occupying
            // this height let's save its hashcode
            let main_chain_block = self
                .mainchain_height_to_header
                .insert(&current_height, &current_block_hash);
            self.mainchain_header_to_height
                .insert(&current_block_hash, &current_height);

            // If we found a mainchain block at the current height than remove this block from the
            // header pool and from the header -> height map
            if let Some(current_main_chain_blockhash) = main_chain_block {
                self.remove_block_header(&current_main_chain_blockhash);
            }

            // Switch iterator cursor to the previous block in fork
            fork_header_cursor = self
                .headers_pool
                .get(&prev_block_hash)
                .unwrap_or_else(|| env::panic_str("previous fork block should be there"));
        }

        // Updating tip of the new main chain
        self.mainchain_tip_blockhash = fork_tip_hash;
    }
