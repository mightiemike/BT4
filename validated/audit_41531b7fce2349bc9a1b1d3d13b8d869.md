### Title
Division by Zero in `PerpEngine.socializeSubaccount` Blocks Liquidation Finalization When `openInterest` Is Zero — (File: `core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when a subaccount has residual negative `vQuoteBalance` after insurance coverage. If `openInterest` is zero at the time of finalization — a reachable state when the liquidatee held the last open position in a perp market — the division reverts, permanently blocking liquidation finalization for that subaccount.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, when a subaccount's `vQuoteBalance` remains negative after the insurance fund is applied, the protocol attempts to socialize the loss across all remaining participants by computing a per-share funding adjustment:

```solidity
int128 fundingPerShare = -balance.vQuoteBalance.div(
    state.openInterest
);
```

There is no guard requiring `state.openInterest != 0` before this division. [1](#0-0) 

The finalization path in `ClearinghouseLiq._finalizeSubaccount` first requires that all of the liquidatee's perp position amounts are exactly zero before calling `perpEngine.socializeSubaccount`: [2](#0-1) 

Once the liquidatee's position amount is zeroed out, their contribution to `openInterest` is removed. If they were the last participant in that perp market, `openInterest` becomes zero. The subsequent call to `socializeSubaccount` then hits the division by zero. [3](#0-2) 

---

### Impact Explanation

When the revert occurs, the entire `liquidateSubaccountImpl` call reverts. The liquidatee's subaccount cannot be finalized: the insurance fund cannot be applied, the negative `vQuoteBalance` cannot be cleared, and the subaccount remains permanently stuck in an under-maintenance state. No liquidator — including `N_ACCOUNT` — can complete the finalization. The insurance fund accounting is also corrupted because `lastLiquidationFees` and `insurance` are never updated for this subaccount. [4](#0-3) 

---

### Likelihood Explanation

This is reachable in any perp market where the liquidatee held the last open position. Concretely: a low-activity or recently-delisted perp product where only one subaccount remains with a position. After prior liquidation rounds reduce their `amount` to zero (required by the finalization check), `openInterest` drops to zero. The subaccount still carries negative `vQuoteBalance` (unrealized loss exceeding insurance), and the finalization call reverts. No privileged access is required — any liquidator can submit the finalization transaction with `productId = type(uint32).max`. [5](#0-4) 

---

### Recommendation

Add a zero-check for `openInterest` before the division in `socializeSubaccount`. If `openInterest` is zero, there are no other participants to socialize the loss against; the remaining negative `vQuoteBalance` should be absorbed entirely by the insurance fund or written off:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No participants to socialize against; absorb or write off
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
``` [6](#0-5) 

---

### Proof of Concept

1. A perp market (e.g., `productId = 2`) has exactly one participant: subaccount `A` with `amount = 100`, `vQuoteBalance = -500e18`, `openInterest = 100`.
2. The market price drops; subaccount `A` falls below maintenance health.
3. A liquidator calls `liquidateSubaccountImpl` with `productId = 2`, reducing `A`'s `amount` to `0`. `openInterest` is now `0`.
4. The liquidator calls `liquidateSubaccountImpl` with `productId = type(uint32).max` (finalization).
5. `_finalizeSubaccount` passes the `balance.amount == 0` check (line 319).
6. `perpEngine.socializeSubaccount` is called. `balance.vQuoteBalance = -500e18 < 0`. Insurance is insufficient. The code reaches `fundingPerShare = -balance.vQuoteBalance.div(state.openInterest)` → division by zero → revert.
7. Subaccount `A` is permanently stuck; no finalization is possible. [7](#0-6) [8](#0-7)

        external

        uint32[] memory _productIds = getProductIds();

            uint32 productId = _productIds[i];
            (State memory state, Balance memory balance) = getStateAndBalance(
                productId,
                subaccount
            );
            if (balance.vQuoteBalance < 0) {
                int128 insuranceCover = MathHelper.min(
                    insurance,
                    -balance.vQuoteBalance
                );
                insurance -= insuranceCover;
                balance.vQuoteBalance += insuranceCover;
                state.availableSettle += insuranceCover;

                // actually socialize if still not enough
                if (balance.vQuoteBalance < 0) {

                    balance.vQuoteBalance = 0;
                }
                _setState(productId, state);
                _setBalanceAndUpdateBitmap(productId, subaccount, balance);
            }
        }
        return insurance;
    }
