### Title
Missing `address(0)` Check Before `delegatecall` to `clearinghouseLiq` Allows Silent Liquidation Bypass - ([File: core/contracts/Clearinghouse.sol])

---

### Summary

`Clearinghouse.liquidateSubaccount()` performs a `delegatecall` to `clearinghouseLiq` without first asserting that `clearinghouseLiq != address(0)`. If `clearinghouseLiq` is zero (due to misconfiguration at initialization or an unchecked upgrade), the `delegatecall` to `address(0)` returns `(true, "")`, the `require(success, ...)` guard passes, and the entire liquidation is silently skipped. In contrast, `Endpoint._delegatecallEndpointTx()` explicitly guards its analogous delegatecall with `require(endpointTx != address(0), "Endpoint Tx not set")`.

---

### Finding Description

In `Clearinghouse.sol`, `liquidateSubaccount()` delegates all liquidation logic to `clearinghouseLiq`:

```solidity
(bool success, bytes memory result) = clearinghouseLiq.delegatecall(
    liquidateSubaccountCall
);
require(success, string(result));
``` [1](#0-0) 

There is no guard of the form `require(clearinghouseLiq != address(0), ...)` before this call. In the EVM, a `delegatecall` to `address(0)` (an address with no code) returns `success = true` with empty return data. The subsequent `require(success, ...)` therefore passes unconditionally, and no liquidation logic executes.

`clearinghouseLiq` is set without a zero-address check in two places:

1. `Clearinghouse.initialize()` assigns `clearinghouseLiq = _clearinghouseLiq` with no validation: [2](#0-1) 

2. `Clearinghouse.upgradeClearinghouseLiq()` assigns `clearinghouseLiq = _clearinghouseLiq` with no zero-check: [3](#0-2) 

The `ProxyManagerHelper.upgradeClearinghouseLiq()` wrapper also performs no zero-check before forwarding: [4](#0-3) 

Compare with `Endpoint._delegatecallEndpointTx()`, which correctly guards its delegatecall:

```solidity
require(endpointTx != address(0), "Endpoint Tx not set");
(bool success, bytes memory result) = endpointTx.delegatecall(callData);
``` [5](#0-4) 

Also compare with `bootstrapEndpointTx()` in `BaseProxyManager`, which explicitly checks `require(_endpointTx != address(0), "EndpointTx is zero")` before setting the analogous `endpointTx` address: [6](#0-5) 

No equivalent protection exists for `clearinghouseLiq`.

---

### Impact Explanation

If `clearinghouseLiq` is `address(0)`:

- Every call to `liquidateSubaccount()` silently succeeds without executing `liquidateSubaccountImpl`. Underwater subaccounts are never actually liquidated.
- The protocol's primary solvency defense — forced liquidation of unhealthy positions — is completely bypassed.
- Liquidators submitting `LiquidateSubaccount` transactions still pay `LIQUIDATION_FEE` (charged in `EndpointTx.processTransactionImpl`) but receive no liquidation proceeds, as no state change occurs.
- Insolvent subaccounts accumulate, threatening the insurance fund and all counterparties. [7](#0-6) 

---

### Likelihood Explanation

`clearinghouseLiq` can be zero in two realistic scenarios:

1. **Deployment misconfiguration**: `initialize()` accepts `address(0)` for `_clearinghouseLiq` without reverting. A deployment script error silently produces a broken state.
2. **Upgrade misconfiguration**: `upgradeClearinghouseLiq(address(0))` is accepted without reversion. A botched upgrade (e.g., passing the wrong argument) clears the address.

Neither scenario requires a malicious actor — both are plausible operational errors. Once triggered, any unprivileged liquidator interacting through the standard `LiquidateSubaccount` transaction path hits the broken delegatecall.

---

### Recommendation

Add a zero-address guard immediately before the `delegatecall` in `Clearinghouse.liquidateSubaccount()`, mirroring the pattern already used in `Endpoint._delegatecallEndpointTx()`:

```solidity
require(clearinghouseLiq != address(0), "ClearinghouseLiq not set");
(bool success, bytes memory result) = clearinghouseLiq.delegatecall(
    liquidateSubaccountCall
);
require(success, string(result));
```

Additionally, add a zero-address check in both `Clearinghouse.initialize()` and `Clearinghouse.upgradeClearinghouseLiq()`:

```solidity
require(_clearinghouseLiq != address(0), "ClearinghouseLiq is zero");
```

---

### Proof of Concept

1. Deploy `Clearinghouse` with `_clearinghouseLiq = address(0)` (no revert occurs).
2. Any subaccount becomes underwater (maintenance health < 0).
3. A liquidator constructs and submits a valid `LiquidateSubaccount` transaction via `Endpoint.submitTransactionsChecked()` or `submitSlowModeTransaction()`.
4. `EndpointTx.processTransactionImpl()` charges `LIQUIDATION_FEE` from the liquidator and calls `clearinghouse.liquidateSubaccount(signedTx.tx)`.
5. Inside `Clearinghouse.liquidateSubaccount()`, `clearinghouseLiq.delegatecall(...)` is called with `clearinghouseLiq = address(0)`.
6. EVM returns `(true, "")`. `require(success, ...)` passes.
7. No liquidation state change occurs. The underwater subaccount remains open. The liquidator loses the fee with no benefit. The protocol's solvency mechanism is silently disabled. [8](#0-7)
