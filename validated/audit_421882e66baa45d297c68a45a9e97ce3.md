### Title
4-byte opcode `0x13d61f00` bypasses `ENABLE_SECP_OPS` guard, reaching `op_secp256k1_verify` unconditionally — (`src/chia_dialect.rs`)

### Summary
The `ChiaDialect::op` dispatch function has two separate branches for opcode routing. The 1-byte branch correctly gates opcodes 64 and 65 behind `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte branch, which handles `0x13d61f00` (secp256k1) and `0x1c3a8f00` (secp256r1), contains **no such guard** and unconditionally calls `op_secp256k1_verify` / `op_secp256r1_verify` regardless of whether `ENABLE_SECP_OPS` is set.

### Finding Description
In `ChiaDialect::op`:

```rust
// 4-byte branch — NO flag check
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,   // dispatched unconditionally
    0x1c3a8f00 => op_secp256r1_verify,
    _ => { return unknown_operator(...); }
};
return f(allocator, argument_list, max_cost, flags);  // flags ignored inside

// 1-byte branch — correctly gated
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

An attacker-controlled CLVM program that uses the 4-byte atom `0x13d61f00` as its operator will reach `op_secp256k1_verify` and execute a full secp256k1 signature verification even on a node running with `ClvmFlags::empty()` (no `ENABLE_SECP_OPS`). [4](#0-3) 

### Impact Explanation
This creates a **consensus split**. The intended invariant is that secp ops are only available after `ENABLE_SECP_OPS` activates (a hard-fork flag). Before activation:

- A node running the current code will **accept and execute** a CLVM program using opcode `0x13d61f00` with a valid secp256k1 signature, returning `Ok(Reduction(1300000, nil))`.
- A node running older code (or one where this 4-byte path was not present) would treat `0x13d61f00` as an unknown operator and either reject it (if `NO_UNKNOWN_OPS`) or treat it as a no-op with computed cost.

Any spend bundle that uses `0x13d61f00` with a valid secp256k1 signature will be accepted by some nodes and rejected (or treated differently) by others, splitting consensus.

### Likelihood Explanation
The path is fully attacker-controlled: the opcode atom is part of the CLVM program, which is attacker-supplied data. No special privileges, compromised nodes, or social engineering are required. The 4-byte atom `0x13d61f00` is a valid, deserializable CLVM atom. The call to `op_secp256k1_verify` will succeed given a valid pubkey/msg/sig triple.

### Recommendation
Add the `ENABLE_SECP_OPS` guard to the 4-byte dispatch branch, mirroring the 1-byte branch:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => { return unknown_operator(allocator, o, argument_list, flags, max_cost); }
};
```

### Proof of Concept
A Rust unit test calling `ChiaDialect::op` with:
- `o` = 4-byte atom `[0x13, 0xd6, 0x1f, 0x00]`
- `flags` = `ClvmFlags::empty()` (no `ENABLE_SECP_OPS`)
- valid `(pubkey, msg, sig)` arguments

will return `Ok(Reduction(1300000, nil))`, proving `op_secp256k1_verify` executed successfully. The same test with opcode `64` (1-byte) under `ClvmFlags::empty()` falls through to `unknown_operator`, confirming the asymmetry. [5](#0-4) [6](#0-5)
