### Title
Kickout-Exemption Sorting Uses Cutoff-Blind Online Ratio While Reward Calculation Uses Cutoff-Adjusted Ratio — (`chain/epoch-manager/src/validator_stats.rs`)

---

### Summary

`get_sortable_validator_online_ratio` always passes `None` for `endorsement_cutoff_threshold` when computing the online ratio used to rank validators for kickout exemption. The reward calculator, however, applies the actual `endorsement_cutoff_threshold` from the epoch config. When that threshold is set, the two computations produce materially different orderings, causing the wrong validators to be shielded from kickout and the wrong validators to be ejected — a broken protocol invariant with direct stake-loss consequences.

---

### Finding Description

**Two divergent online-ratio computations for the same epoch:**

**Path 1 — kickout-exemption sorting** (`chain/epoch-manager/src/lib.rs`, line 502):

```rust
let mut sorted_validators = validator_block_chunk_stats
    .iter()
    .map(|(account, stats)| (get_sortable_validator_online_ratio(stats), account))
    .collect_vec();
```

`get_sortable_validator_online_ratio` is defined as:

```rust
pub(crate) fn get_sortable_validator_online_ratio(stats: &BlockChunkValidatorStats) -> BigRational {
    let ratio = get_validator_online_ratio(stats, None);   // ← always None
    ...
}
``` [1](#0-0) 

**Path 2 — reward calculation** (`chain/epoch-manager/src/reward_calculator.rs`, line 96):

```rust
let production_ratio =
    get_validator_online_ratio(&stats, online_thresholds.endorsement_cutoff_threshold);
``` [2](#0-1) 

`get_validator_online_ratio` with a `Some(threshold)` binarises the endorsement component: if the validator's endorsement ratio is below the threshold it contributes **0**, otherwise it contributes **1**. With `None` it uses the raw ratio. [3](#0-2) 

The sorted list is consumed by `compute_exempted_kickout`, which walks from highest to lowest online ratio and accumulates stake until the exemption budget is exhausted: [4](#0-3)

### Impact Explanation

When `endorsement_cutoff_threshold` is set (e.g., `Some(50)`), the cutoff creates a cliff in the reward-side ratio that is invisible to the sorting side.

**Concrete divergence:**

| Validator | blocks | chunks | endorsements | Sort ratio (None) | Reward ratio (cutoff=50%) |
|-----------|--------|--------|--------------|-------------------|--------------------------|
| A | 95/100 | 95/100 | 49/100 | ≈ 79.7% | ≈ 63.3% → **0 reward** (below `online_min`) |
| B | 85/100 | 85/100 | 51/100 | ≈ 73.7% | ≈ 90.3% → **full reward** |

The exemption logic ranks A above B (79.7 > 73.7) and shields A from kickout. But A earns zero reward while B earns full reward. The protocol ejects the productive validator and retains the unproductive one.

This violates the invariant that kickout exemption should protect validators who are genuinely contributing to the network. A validator who will receive zero reward (because their endorsement ratio is below the cutoff) can be exempted over a validator who will receive full reward.

**Stake impact:** The exempted-but-zero-reward validator retains its locked stake and validator seat for the next two epochs, while the productive validator loses its seat and must re-stake. The productive validator's stake is not slashed, but it loses two epochs of rewards and its seat.

---

### Likelihood Explanation

The divergence is latent whenever `endorsement_cutoff_threshold` is `Some(_)` in the epoch config. The field exists in `ValidatorOnlineThresholds` and is explicitly tested with `Some(50)` in `test_reward_stateless_validation_with_endorsement_cutoff`. [5](#0-4) 

The divergence is triggered automatically at epoch finalization — no privileged action is required. Any validator whose endorsement ratio straddles the cutoff boundary is affected. The effect is most pronounced when `validator_max_kickout_stake_perc` is low (tight exemption budget), forcing the algorithm to make fine-grained distinctions between validators.

---

### Recommendation

Pass the epoch config's `endorsement_cutoff_threshold` into `get_sortable_validator_online_ratio` so that the sorting used for kickout exemption is computed with the same formula as the reward calculation:

```rust
// In compute_validators_to_reward_and_kickout, after reading config:
let endorsement_cutoff = config.endorsement_cutoff_threshold; // read from EpochConfig

let mut sorted_validators = validator_block_chunk_stats
    .iter()
    .map(|(account, stats)| {
        (get_sortable_validator_online_ratio(stats, endorsement_cutoff), account)
    })
    .collect_vec();
```

Update `get_sortable_validator_online_ratio` to accept and forward the threshold parameter to `get_validator_online_ratio`. [6](#0-5)

### Proof of Concept

Given epoch config with `endorsement_cutoff_threshold = Some(50)`, `online_min_threshold = 9/10`, `validator_max_kickout_stake_perc = 60` (exemption budget = 40% of stake):

**Validators (equal stake):**
- `alice`: blocks=95/100, chunks=95/100, endorsements=49/100
- `bob`: blocks=85/100, chunks=85/100, endorsements=51/100
- `carol`: blocks=100/100, chunks=100/100, endorsements=100/100 (clearly best)

**Sort ratios (cutoff ignored):**
- carol: 100% → exempted first
- alice: (95+95+49)/300 ≈ 79.7% → exempted second (budget exhausted)
- bob: (85+85+51)/300 ≈ 73.7% → **kicked out**

**Reward ratios (cutoff applied, endorsement binarised):**
- carol: (100+100+100)/300 = 100% → full reward
- alice: (95+95+0)/200 = 95% → full reward (endorsement treated as 0, but only 2 components averaged since endorsement=0)

Wait — let me re-check. With cutoff=50 and alice's endorsement=49/100 < 50%, `get_endorsement_ratio` returns `(0, 1)`. So alice's ratio = (95*1 + 95*1 + 0*100*100) / (3 * 100 * 100 * 1) ... actually looking at the match arm for `(expected_blocks, expected_chunks, expected_endorsements)` where all three are nonzero:

```
numer = produced_blocks * expected_chunks * expected_endorsements
      + produced_chunks * expected_blocks * expected_endorsements
      + produced_endorsements * expected_blocks * expected_chunks
denom = 3 * expected_chunks * expected_blocks * expected_endorsements
```

With cutoff applied: `produced_endorsements=0`, `expected_endorsements=1`:
- numer = 95*100*1 + 95*100*1 + 0*100*100 = 9500 + 9500 + 0 = 19000
- denom = 3 * 100 * 100 * 1 = 30000
- ratio = 19000/30000 = 63.3% < 90% online_min → **alice gets 0 reward**

**bob** with cutoff applied: `produced_endorsements=1`, `expected_endorsements=1`:
- numer = 85*100*1 + 85*100*1 + 1*100*100 = 8500 + 8500 + 10000 = 27000
- denom = 3 * 100 * 100 * 1 = 30000
- ratio = 27000/30000 = 90% ≥ 90% online_min → **bob gets reward**

**Result:** alice is exempted from kickout (gets to stay as validator) but earns zero reward. Bob is kicked out but would have earned reward. The exemption logic protected the wrong validator. [4](#0-3) [7](#0-6)

}
