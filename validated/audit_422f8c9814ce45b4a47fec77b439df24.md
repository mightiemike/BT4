### Title
Raw `approve` on Non-Standard ERC20 Tokens Permanently Blocks All DirectDepositV1 Collateral Deposits - (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` directly through the `IIERC20Base` interface, which expects a `bool` return value. Non-standard tokens like USDT do not return a value from `approve`, causing the ABI decoder to revert. Because `creditDeposit()` iterates over all spot products, a single USDT-like token with a non-zero balance in the DDA causes the entire function to revert, blocking deposits of all tokens for that subaccount.

---

### Finding Description

`DirectDepositV1.creditDeposit()` is the core function for crediting tokens held in a per-subaccount Direct Deposit Address (DDA) into the Nado protocol. It iterates over every spot product registered in `SpotEngine`, checks the DDA's balance, and for any non-zero balance calls:

```solidity
token.approve(address(endpoint), balance);
``` [1](#0-0) 

The `IIERC20Base` interface declares `approve` as returning `bool`:

```solidity
function approve(address spender, uint256 amount) external returns (bool);
``` [2](#0-1) 

USDT (and similar non-standard tokens) do not return any value from `approve`. In Solidity 0.8.x, calling a function through an interface that declares a `bool` return when the callee returns nothing causes the ABI decoder to revert. There is no `SafeERC20.forceApprove` or low-level call wrapper used here — the call is a direct interface dispatch.

The `ERC20Helper` library, which is the project's safe-transfer utility, only provides `safeTransfer` and `safeTransferFrom` — it has no `safeApprove` equivalent: [3](#0-2) 

A secondary instance of the same root cause exists in `ContractOwner.wrapVaultAsset()`, which calls `assetToken.approve(tokenAddr, 0)` and `assetToken.approve(tokenAddr, assetBalance)` via the same raw `IERC20Base` interface, with no access modifier: [4](#0-3) 

---

### Impact Explanation

**Impact: Medium**

`creditDeposit()` carries no access modifier — it is callable by any external address: [5](#0-4) 

The loop iterates over all spot products. If any one of them is a USDT-like token and holds a non-zero balance in the DDA, the entire `creditDeposit()` call reverts — blocking deposits of every other token in that DDA as well. The corrupted state is: the DDA holds token balances that can never be credited to the subaccount via the normal deposit path, effectively locking user funds in the DDA contract.

---

### Likelihood Explanation

**Likelihood: Medium**

USDT is one of the most widely used stablecoins and a natural candidate for listing as a spot collateral product. If it is listed, every DDA that receives USDT will have its `creditDeposit()` permanently broken. The function is permissionlessly callable, so no privileged actor is needed to trigger the revert.

---

### Recommendation

Replace the raw `token.approve(...)` call in `DirectDepositV1.creditDeposit()` with a low-level call pattern that tolerates missing return data, analogous to the existing `safeTransfer` helper already implemented in `DirectDepositV1`: [6](#0-5) 

Apply the same pattern for `approve`:

```solidity
function safeApprove(IIERC20Base self, address spender, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IIERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        "Approve failed"
    );
}
```

Then replace `token.approve(address(endpoint), balance)` with `safeApprove(token, address(endpoint), balance)`.

Apply the same fix to `ContractOwner.wrapVaultAsset()` by using `SafeERC20.forceApprove` (OpenZeppelin) or an equivalent low-level wrapper for the `assetToken.approve` calls. [4](#0-3) 

---

### Proof of Concept

1. USDT is listed as a spot product in `SpotEngine` (a realistic protocol configuration).
2. A user sends USDT to their DDA (`DirectDepositV1` instance).
3. Any caller invokes `DirectDepositV1.creditDeposit()`.
4. The loop reaches the USDT product entry; `token.balanceOf(address(this))` returns a non-zero value.
5. `token.approve(address(endpoint), balance)` is dispatched through `IIERC20Base`, which expects a `bool` return.
6. USDT's `approve` executes but returns no data; Solidity's ABI decoder reverts.
7. The entire `creditDeposit()` transaction reverts — no token (including non-USDT tokens with non-zero balances) is deposited.
8. The user's funds remain stranded in the DDA with no standard recovery path. [7](#0-6)

        for (uint256 i = 0; i < productIds.length; i++) {

            address tokenAddr = spotEngine.getToken(productId);
