### Title
Missing `ENABLE_SECP_OPS` Flag Guard on 4-Byte Secp Opcode Dispatch Path — (`File: src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` in `src/chia_dialect.rs` contains two separate dispatch paths for secp signature-verification operators. The 1-byte opcode aliases (opcodes 64 and 65) are correctly gated behind the `ENABLE_SECP_OPS` flag. The 4-byte opcode path (`0x13d61f00` → `op_secp256k1_verify`, `0x1c3a8f00` → `op_secp256r1_verify`) dispatches to the same functions with no flag check at all. An attacker who controls CLVM bytes can invoke live secp signature verification using the 4-byte form regardless of whether `ENABLE_SECP_OPS` is set, bypassing the intended activation gate.

---

### Finding Description

`ChiaDialect::op()` branches first on opcode length. When `op_len == 4`, it matches two specific 4-byte opcodes and calls the secp functions directly:

```rust
// src/chia_dialect.rs lines 175-182
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
``` [1](#0-0) 

When `op_len == 1`, the same two operations are gated:

```rust
// src/chia_dialect.rs lines 248-249
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The `ENABLE_SECP_OPS` flag is defined as the activation gate for these operators: [3](#0-2) 

`MEMPOOL_MODE` does not include `ENABLE_SECP_OPS`: [4](#0-3) 

**Exploit path**: An attacker submits a CLVM program whose operator atom is the 4-byte encoding `\x13\xd6\x1f\x00` (secp256k1) or `\x1c\x3a\x8f\x00` (secp256r1). The `op_len == 4` branch fires before the 1-byte branch is ever reached. No flag check exists in that branch. `op_secp256k1_verify` or `op_secp256r1_verify` executes unconditionally. [5](#0-4) 

---

### Impact Explanation

**Mempool bypass (highest impact):** In mempool mode (`NO_UNKNOWN_OPS` set, `ENABLE_SECP_OPS` absent), a program using 1-byte opcode 64 is rejected with `Unimplemented`. The identical secp operation encoded as the 4-byte opcode `0x13d61f00` is accepted and executes live signature verification. An attacker can craft transactions that pass mempool validation using the 4-byte form while the same operation would be rejected in the 1-byte form. This breaks the mempool's pre-softfork enforcement boundary.

**Consensus semantic divergence:** In consensus mode without `ENABLE_SECP_OPS`, 1-byte opcode 64 falls through to `unknown_operator` and returns nil (no-op). The 4-byte form executes real signature verification and raises on failure. Two programs encoding the same secp check with different opcode widths produce different results on the same inputs, creating a consensus-observable divergence exploitable by a crafted coin puzzle.

---

### Likelihood Explanation

The attacker only needs to control the CLVM bytes of a program — the normal capability of any coin spender on Chia. The 4-byte opcode values are derivable from the cost formula documented in the source comments and are already present in the fuzzing harness (`clvm-fuzzing/src/make_tree.rs`) and the benchmark tool (`tools/src/bin/benchmark-clvm-cost.rs`), confirming they are reachable and exercised. No elevated privilege is required. [6](#0-5) 

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag guard to the 4-byte opcode dispatch path, mirroring the guard already present on the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures that secp operations are uniformly gated by `ENABLE_SECP_OPS` regardless of which opcode encoding the caller uses, eliminating the mempool bypass and the consensus semantic divergence.

### Proof of Concept

```
; CLVM program using 4-byte secp256k1_verify opcode (0x13d61f00)
; bypasses ENABLE_SECP_OPS gate in mempool mode

; Dialect: ChiaDialect with MEMPOOL_MODE (NO_UNKNOWN_OPS set, ENABLE_SECP_OPS absent)
; Opcode atom: 4 bytes = 0x13 0xd6 0x1f 0x00

(0x13d61f00 pubkey_atom msg_atom sig_atom)
```

- With 1-byte opcode `(64 pubkey msg sig)`: `run_program` returns `EvalErr::Unimplemented` in mempool mode — correctly rejected.
- With 4-byte opcode `(0x13d61f00 pubkey msg sig)`: `op_secp256k1_verify` is called directly at `src/chia_dialect.rs:176,182` — no flag check, executes and returns `Ok(Reduction(1300000, nil))` on a valid signature.

The broken invariant is at `src/chia_dialect.rs` lines 175–182: the `ENABLE_SECP_OPS` guard present at lines 248–249 is absent from the structurally parallel 4-byte branch. [1](#0-0) [7](#0-6)
