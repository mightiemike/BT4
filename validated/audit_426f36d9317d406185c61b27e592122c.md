Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Original User, Allowing Any User to Bypass Curated-Pool Swap Gate via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` always sets to `msg.sender` — the immediate caller of the pool. When `MetricOmmSimpleRouter` is that caller, `sender` is the router's address, not the original EOA. A pool admin who allowlists the router to enable curated users to reach the pool through the standard periphery inadvertently opens the gate to every user on-chain, because the extension never inspects the original user's identity.

## Finding Description

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool: [1](#0-0) 

`MetricOmmPool.swap` always passes `msg.sender` — the immediate caller of the pool — as the first argument to `_beforeSwap`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly. The router is therefore `msg.sender` inside the pool, so `sender` forwarded to the extension is the router's address. The original EOA (`msg.sender` of the router call) is stored only in transient storage via `_setNextCallbackContext` for payment purposes and is never forwarded to the pool or extension: [3](#0-2) 

The same pattern applies to `exactInput` (L92-125), `exactOutputSingle` (L130-147), and `exactOutput` (L154-188). [4](#0-3) 

The `extensionData` bytes field is passed through the call chain but the extension does not decode it to recover the original user — it is ignored in `beforeSwap`. [1](#0-0) 

## Impact Explanation

The swap allowlist is the primary access-control mechanism for curated pools. Once bypassed, any user can execute swaps against a pool configured to restrict trading to a specific set of addresses (e.g., KYC'd counterparties, whitelisted market makers, or protocol-internal actors). This breaks the pool's curation invariant and allows unauthorized users to extract value from the pool at oracle-anchored prices, constituting a direct loss of LP principal relative to the pool's intended operating model. This is broken core pool functionality causing loss of funds — a qualifying impact under the allowed impact gate.

## Likelihood Explanation

The bypass requires the pool admin to call `setAllowedToSwap(pool, router, true)`. This is a natural and expected administrative action: without it, allowlisted users cannot use the standard periphery at all, making the pool effectively unusable through the supported interface. A pool admin who reads the allowlist docs and wants to support router-mediated swaps for their curated users will perform exactly this step, unknowingly opening the gate to everyone. The trigger is a single bounded admin call, and once performed, the bypass is available to every on-chain user at zero additional cost. [5](#0-4) 

## Recommendation

The extension must check the original user's identity, not the immediate pool caller. Two approaches:

1. **Preferred — pass original user through `extensionData`:** Have the router encode `msg.sender` into the `extensionData` it forwards to the pool. The extension decodes and checks that address. Pool admins allowlist individual users, not the router.

2. **Alternative — separate `originator` field in the hook interface:** Add an `originator` parameter to `beforeSwap` that the pool populates from a transient-storage context set by the router before calling `pool.swap`. The extension checks `originator` instead of `sender`.

Either approach ensures the allowlist gates the economically relevant actor regardless of which supported entrypoint is used.

## Proof of Concept

```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension configured
  alice = allowlisted user (allowedSwapper[pool][alice] = true)
  bob   = non-allowlisted user
  admin calls: setAllowedToSwap(pool, router, true)
    (so that alice can use the standard router interface)

Attack:
  bob calls MetricOmmSimpleRouter.exactInputSingle({
    pool: pool,
    recipient: bob,
    ...
  })

  Router calls pool.swap(bob, ...) — msg.sender inside pool = router
  Pool calls _beforeSwap(sender=router, ...)
  Extension checks allowedSwapper[pool][router] → true
  Swap executes for bob despite bob not being allowlisted

Result:
  bob bypasses the curated-pool swap gate at zero additional cost.
  The allowlist provides no protection for router-mediated swaps.

    _checkDeadline(params.deadline);
    _validatePath(params.tokens, params.pools, params.extensionDatas);

    uint256 last = params.pools.length - 1;
    int128 amount = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn);

    for (uint256 i = 0; i <= last; i++) {
      address pool = params.pools[i];
      bool zeroForOne = MetricOmmSwapPath.resolveZeroForOneBitmap(params.zeroForOneBitMap, i);

      _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
      (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
        .swap(
          i == last ? params.recipient : address(this),
          zeroForOne,
          amount,
          MetricOmmSwapPath.openLimit(zeroForOne),
          "",
          params.extensionDatas[i]
        );

      int128 amountInActual = MetricOmmSwapResults.extractAmountIn(zeroForOne, amount0Delta, amount1Delta);
      if (amountInActual < amount) revert InvalidInputAmountAtHop(uint8(i), amountInActual, amount);

      amount = MetricOmmSwapResults.extractAmountOut(zeroForOne, amount0Delta, amount1Delta);
    }

    if (amount <= 0) revert InvalidSwapDeltas();
    amountOut = MetricOmmSwapInputs.int128ToUint128(amount);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
