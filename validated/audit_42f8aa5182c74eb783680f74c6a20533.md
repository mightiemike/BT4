### Title
Stale Fill Price Used in `alreadyMatched` Computation Causes Systematic Taker Fee Overcharge Across Multi-Fill Orders — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

In `OffchainExchange.matchOrders`, the `alreadyMatched` argument passed to `applyFee` for the taker is computed as `-maker.order.priceX18 * filledAmounts[taker.digest]`. This uses the **current fill's maker price** to retroactively value all previously filled base amounts, rather than the actual cumulative quote amounts from those historical fills. When a taker order is filled across multiple sequential fills at different prices, this stale price assumption corrupts the fee metering boundary, causing the taker to be systematically overcharged when later fills execute at a higher price than earlier fills.

---

### Finding Description

`matchOrders` calls `applyFee` for the taker with:

```solidity
applyFee(
    callState.productId,
    ordersInfo.taker,
    market,
    -maker.order.priceX18.mul(filledAmounts[ordersInfo.taker.digest]),
    taker.order.appendix,
    true
);
``` [1](#0-0) 

`filledAmounts[ordersInfo.taker.digest]` is the cumulative **base** amount filled across all prior fills of this order digest. The `alreadyMatched` parameter is supposed to represent the total **quote** amount already matched, so that `applyFee` can correctly determine how much of the current fill falls above the `minSize` fee-free threshold.

Inside `applyFee`, the fee-bearing portion is computed as:

```solidity
int128 feeApplied = MathHelper.abs(alreadyMatched + matchQuote) - market.minSize;
feeApplied = MathHelper.min(feeApplied, matchQuote.abs());
``` [2](#0-1) 

The problem: each prior fill may have executed at a different maker price. The actual cumulative quote matched is `Σ(price_i × base_i)` over all prior fills. But the code computes it as `current_price × Σ(base_i)`, substituting the current fill's maker price for all historical fill prices. These two quantities are equal only when all fills execute at the same price.

`filledAmounts` is updated **after** the fee is applied:

```solidity
if (taker.order.sender != X_ACCOUNT) {
    filledAmounts[ordersInfo.taker.digest] += ordersInfo.taker.amountDelta;
}
``` [3](#0-2) 

So the stale-price error accumulates with every fill after the first.

---

### Impact Explanation

When a taker's large order is filled across multiple fills in a rising market (later fills at higher prices than earlier fills), `alreadyMatched` is **overestimated** in absolute terms. This makes `feeApplied` larger than the correct value, causing the taker to be charged fees on a larger portion of the current fill's quote than they should be. The taker's quote balance is debited more than the protocol's fee schedule entitles.

Concretely, the overcharge per fill is bounded by:

```
overcharge ≈ feeRate × (current_price − avg_historical_price) × historical_base_fills
```

For a 2 bps taker fee, a 10% price appreciation between fills, and $10 M in prior fills, the overcharge per fill is ~$2,000 in quote. This is a direct, non-recoverable loss of user funds silently applied during normal order execution.

The reverse direction (current price lower than historical prices) undercharges the taker, representing a loss of protocol fee revenue.

---

### Likelihood Explanation

No malicious actor is required. This occurs in any volatile market where a large taker order is filled across multiple sequential `matchOrders` calls at different maker prices — a routine occurrence on any active perpetual or spot market. The sequencer fills orders as they arrive; price movement between fills is normal. The taker has no mechanism to detect or prevent the overcharge.

---

### Recommendation

Track the cumulative **quote** amount filled per order digest alongside the existing base-amount tracker. Replace the stale-price reconstruction with the stored quote total:

```solidity
// New storage
mapping(bytes32 => int128) public filledQuoteAmounts;

// In matchOrders, pass the stored quote total:
applyFee(
    callState.productId,
    ordersInfo.taker,
    market,
    filledQuoteAmounts[ordersInfo.taker.digest],   // actual cumulative quote
    taker.order.appendix,
    true
);

// After balances are updated:
filledQuoteAmounts[ordersInfo.taker.digest] += ordersInfo.taker.quoteDelta;
```

This ensures `alreadyMatched` always reflects the true historical quote cost of the order, regardless of price movement between fills.

---

### Proof of Concept

Setup: `minSize = 1000` quote units, taker fee = 2 bps, taker places a buy order for 100 base.

**Fill 1** — maker price = 10, base = 50, quote = 500:
- `filledAmounts[digest] = 0` → `alreadyMatched = 0` (correct, first fill)
- Flat minimum fee applied on `minSize = 1000`; `feeApplied = abs(0 − 500) − 1000 = −500` → 0
- `meteredQuote = −1000`; fee = `1000 × 0.0002 = 0.2`
- `filledAmounts[digest] += 50` → 50

**Fill 2** — maker price = 20, base = 50, quote = 1000:
- Computed: `alreadyMatched = −20 × 50 = −1000`
- Actual: `alreadyMatched = −500` (the real quote from Fill 1)
- `feeApplied` (computed): `abs(−1000 + (−1000)) − 1000 = 1000` → `meteredQuote = −1000`; fee = `1000 × 0.0002 = 0.2`
- `feeApplied` (correct): `abs(−500 + (−1000)) − 1000 = 500` → `meteredQuote = −500`; fee = `500 × 0.0002 = 0.1`

**Overcharge on Fill 2 = 0.1 quote units.** Scales linearly with order size and price movement. For a $10 M order with 10% price appreciation between fills, the overcharge is ~$2,000 per fill, directly debited from the taker's quote balance with no recourse.
