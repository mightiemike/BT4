### Title
`TryFrom<protobuf::ResourceBounds>` collapses `AllResources(l1_data_gas=0, l2_gas=0)` to `L1Gas`, producing a divergent transaction hash after protobuf round-trip — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserialization of `ValidResourceBounds` uses a value-based heuristic to decide between the `L1Gas` and `AllResources` variants. Any transaction whose `l2_gas` and `l1_data_gas` are both zero is silently collapsed to `L1Gas`, regardless of which variant was used when the hash was originally computed. Because `get_tip_resource_bounds_hash` produces structurally different Poseidon preimages for the two variants, the transaction hash computed from the deserialized protobuf object diverges from the hash that was signed and stored at ingestion time.

---

### Finding Description

**Step 1 – Two variants, two different hash preimages.**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` builds the resource-bounds sub-hash differently depending on the variant:

```
L1Gas(X):
  poseidon(tip, concat(L1_GAS, X.amount, X.price), concat(L2_GAS, 0, 0))
  → 3-element input

AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0):
  poseidon(tip, concat(L1_GAS, X.amount, X.price), concat(L2_GAS, 0, 0), concat(L1_DATA, 0, 0))
  → 4-element input
```

The two produce **different Poseidon digests** even when every numeric field is identical.

**Step 2 – Gateway always ingests as `AllResources`.**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds` (not `ValidResourceBounds`). The conversion to `InternalRpcInvokeTransactionV3` wraps it as `ValidResourceBounds::AllResources`. The hash stored in `InternalRpcTransaction.tx_hash` is therefore always computed via the 4-element path for any V3 transaction, including those where `l2_gas = l1_data_gas = 0`.

**Step 3 – Protobuf round-trip silently changes the variant.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` applies:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← collapses AllResources
} else {
    ValidResourceBounds::AllResources(...)
})
```

A transaction with `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` serialised to protobuf (which always emits `l1_data_gas: Some(0)`) is deserialised back as `L1Gas(X)`. The variant has changed.

**Step 4 – Hash re-computation diverges.**

Any code path that re-derives the transaction hash from the deserialized `InvokeTransactionV3` fields — including `validate_transaction_hash`, the OS hint `AssertTransactionHash`, and the SNOS re-execution path — will compute the 3-element hash `H2`, while the stored/signed hash is the 4-element `H1 ≠ H2`.

The analog to H-05 is exact:
- H-05: `reserves = balance − fees`, so `amount_in = balance − reserves = fees` — the stored value carries an implicit offset that the delta computation inherits.
- Here: `AllResources` hash carries an extra `L1_DATA_GAS_0` element (the "offset"). After protobuf round-trip the variant is `L1Gas`, which omits that element, so any re-derived hash is off by exactly that term.

---

### Impact Explanation

**High — Transaction conversion or signature/hash logic binds the wrong hash.**

1. **State-sync / P2P validation**: A syncing node that receives a `FullTransaction` via protobuf and calls `validate_transaction_hash` on the reconstructed `InvokeTransactionV3` will compute `H2` and reject the transaction as having an invalid hash, even though it was correctly signed and sequenced.

2. **SNOS re-execution**: The OS hint `AssertTransactionHash` re-computes the hash inside the Cairo VM. If the transaction was stored with `H1` but the hint processor feeds the OS a `ValidResourceBounds::L1Gas` object (after protobuf round-trip), the assertion fails, breaking proof generation for any block containing such a transaction.

3. **`allocate_tx_resource_bounds_segment`** in `crates/blockifier/src/execution/syscalls/hint_processor.rs` calls `valid_resource_bounds_as_felts`, which emits 2 resource entries for `L1Gas` and 3 for `AllResources`. An account contract whose `__validate__` reads `n_resource_bounds` from `TxInfo` will see a different count depending on which variant is present, potentially causing validation to revert or pass incorrectly.

---

### Likelihood Explanation

Any V3 invoke transaction submitted via the gateway with `l2_gas = 0` and `l1_data_gas = 0` (a common pattern for transactions that only need L1 gas) triggers this path. The condition is reachable by any unprivileged user. The divergence is deterministic and reproducible.

---

### Recommendation

Replace the value-based heuristic in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` with an explicit discriminant. The simplest fix is to treat the presence of a non-`None` `l1_data_gas` field in the protobuf message as the signal for `AllResources`, regardless of its numeric value:

```rust
// In crates/apollo_protobuf/src/converters/transaction.rs
Ok(if value.l1_data_gas.is_none() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

Alternatively, add an explicit `variant` field to the protobuf `ResourceBounds` message so the serialiser can preserve the original discriminant without relying on numeric values.

---

### Proof of Concept

```
1. User submits InvokeV3 via gateway:
     resource_bounds = AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }

2. Gateway computes hash H1 using AllResources path (4-element Poseidon):
     H1 = poseidon(INVOKE, version, sender,
                   poseidon(tip, L1_GAS||X, L2_GAS||0, L1_DATA||0),  ← 4 elements
                   paymaster_hash, chain_id, nonce, da_mode,
                   account_deployment_data_hash, calldata_hash)

3. Transaction stored with tx_hash = H1, propagated via P2P as protobuf::InvokeV3.

4. Receiving node deserialises protobuf::ResourceBounds:
     l1_data_gas = Some(0), l2_gas = Some(0)
     → l1_data_gas.is_zero() && l2_gas.is_zero() == true
     → ValidResourceBounds::L1Gas(X)          ← variant changed

5. Receiving node calls validate_transaction_hash / AssertTransactionHash:
     H2 = poseidon(INVOKE, version, sender,
                   poseidon(tip, L1_GAS||X, L2_GAS||0),               ← 3 elements
                   paymaster_hash, chain_id, nonce, da_mode,
                   account_deployment_data_hash, calldata_hash)

6. H2 ≠ H1 → hash validation fails; valid transaction rejected / proof generation aborted.
```

```rust
    }
```

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```
