### Title
ML-DSA-65 `TransferToGasKey` exec-fee uses wire-format key length instead of trie-id length, causing ~59× overcharge — (`runtime/near-vm-runner/src/logic/logic.rs`)

### Summary

`promise_batch_action_transfer_to_gas_key` (and its wasmtime twin) passes the raw guest-memory wire-format length of the public key (`public_key_len as usize` = 1953 bytes for ML-DSA-65) to `gas_key_transfer_exec_fee`, which is documented to compute the fee based on the **on-trie key length**. For ML-DSA-65 the trie stores a SHA3-256 hash (33 bytes including the type tag), not the 1953-byte borsh-encoded pubkey. The `exec_fee` path in `runtime/runtime/src/config.rs` correctly calls `action.public_key.trie_id_len()` (= 33), but the host-function path uses the raw wire length, producing a ~59× overcharge in the per-byte exec-fee component for every `TransferToGasKey` / `WithdrawFromGasKey` action that carries an ML-DSA-65 gas key.

### Finding Description

`gas_key_transfer_exec_fee` in `core/parameters/src/cost.rs` is documented as computing the exec fee "based on the access key trie key length + estimated value length (what the receiver needs to read/write in the trie)". [1](#0-0) 

For ML-DSA-65 keys the trie key uses a 32-byte SHA3-256 hash (tag 3 + 32 bytes = 33 bytes total), not the 1952-byte raw pubkey. `PublicKey::trie_id_len()` returns 33 for ML-DSA-65 and 1953 for `PublicKey::len()`. [2](#0-1) 

The `exec_fee` dispatch in `runtime/runtime/src/config.rs` correctly uses `action.public_key.trie_id_len()`:

```rust
TransferToGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
``` [3](#0-2) 

But the host-function path in `logic.rs` passes `public_key_len as usize` — the raw byte count of the borsh-encoded key as supplied by the contract — to the same function:

```rust
let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
// ...
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,   // ← wire-format length, not trie_id_len()
);
``` [4](#0-3) 

The wasmtime path has the identical defect: [5](#0-4) 

Inside `gas_key_transfer_exec_fee`, `public_key_len` is fed directly into `access_key_key_len(account_id_len, public_key_len)`, which computes the trie key length as `1 + account_id_len + 1 + public_key_len`. For ML-DSA-65 this yields `2 + account_id_len + 1953` instead of the correct `2 + account_id_len + 33` — a 1920-byte overstatement. [6](#0-5) 

The `access_key_storage_usage` function in `runtime/runtime/src/access_keys.rs` and `permission_exec_fees` in `config.rs` both correctly call `public_key.trie_id_len()`, confirming the intended invariant. [7](#0-6) 

### Impact Explanation

Any contract that calls `promise_batch_action_transfer_to_gas_key` (or the withdraw variant) with an ML-DSA-65 gas key is charged an exec fee computed against 1953 bytes instead of 33 bytes for the trie-key portion. The per-byte exec-fee component is overcharged by `gas_key_byte.exec_fee() × 1920`. This is a protocol-level fee-domain invariant violation: the exec fee is supposed to reflect the actual trie I/O cost, but for ML-DSA-65 keys it reflects the wire-format cost instead — a ~59× divergence. Contracts that budget gas based on the documented trie-cost model will fail with out-of-gas errors or pay far more than expected.

### Likelihood Explanation

Both `GasKeys` and `PostQuantumSignatures` stabilized at protocol version 85; the current stable protocol version is 86, so both features are live on mainnet. Any unprivileged contract can call `promise_batch_action_transfer_to_gas_key` with an ML-DSA-65 public key. The divergent value (1953 vs 33) is exact and deterministic. [8](#0-7) 

### Recommendation

In `promise_batch_action_transfer_to_gas_key` (both the `logic.rs` and `wasmtime_runner/logic.rs` paths), replace `public_key_len as usize` with the decoded key's `trie_id_len()` when calling `gas_key_transfer_exec_fee`:

```rust
let decoded_pk = public_key.decode()?;
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    decoded_pk.trie_id_len(),   // ← correct: 33 for ML-DSA-65
);
```

This matches the pattern already used in `runtime/runtime/src/config.rs` and `runtime/runtime/src/access_keys.rs`.

### Proof of Concept

Exact divergent values for an ML-DSA-65 key with a 10-byte account id:

| Path | `public_key_len` passed | `trie_key_len` computed | Correct value |
|---|---|---|---|
| `config.rs` exec_fee | `trie_id_len()` = 33 | `2 + 10 + 33` = 45 | 45 |
| `logic.rs` host fn | `public_key_len` = 1953 | `2 + 10 + 1953` = 1965 | 45 |

The per-byte exec-fee overcharge is `gas_key_byte.exec_fee() × (1965 − 45)` = `gas_key_byte.exec_fee() × 1920` per `TransferToGasKey` / `WithdrawFromGasKey` action carrying an ML-DSA-65 key. [9](#0-8)

/// Exec fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the access key trie key length + estimated value length (what the
/// receiver needs to read/write in the trie).
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
```

    /// Length, in bytes, of the on-trie identifier for an access-key
    /// entry owned by this public key. For ed25519/secp256k1 this matches
    /// `len()`; for ML-DSA-65 the trie stores a SHA3-256 hash (33 bytes
    /// including the type tag), not the 1953-byte borsh-encoded pubkey.

    /// form without actually hashing the pubkey.

            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,
        }
    }
