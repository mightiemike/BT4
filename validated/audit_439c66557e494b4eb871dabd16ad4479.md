Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Swap Allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap()` gates swaps by checking the `sender` argument, which the pool sets to its own `msg.sender` — the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` intermediates a swap, the router becomes `msg.sender` to the pool, so the allowlist checks the router's address rather than the real end user's address. Any non-allowlisted user can bypass a curated pool's access control by routing through `MetricOmmSimpleRouter`.

## Finding Description
**Root cause — pool passes its own `msg.sender` as `sender`:**

In `MetricOmmPool.swap()`, `_beforeSwap` is called with `msg.sender` as the first argument: [1](#0-0) 

**Allowlist checks `sender` (the pool's `msg.sender`), not the originating user:**

`SwapAllowlistExtension.beforeSwap()` uses `msg.sender` (the pool) as the mapping key and `sender` (whoever called `pool.swap()`) as the identity being gated: [2](#0-1) 

**Router calls `pool.swap()` directly, making itself the `sender`:**

In `exactInputSingle`, the router calls `pool.swap()` directly. The pool's `msg.sender` is therefore the router contract, not the end user: [3](#0-2) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [4](#0-3) 

**Result:** The allowlist check resolves to `allowedSwapper[pool][router]`. For router-mediated swaps to work at all, the pool admin must allowlist the router address. Once the router is allowlisted, every user — regardless of their own allowlist status — can call any router function (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) and swap freely against the pool. The allowlist guard is completely defeated.

Existing guards are insufficient: the `_setNextCallbackContext` mechanism in the router only controls callback payment authorization, not the identity forwarded to extensions. [5](#0-4) 

## Impact Explanation
A pool admin deploys a curated pool with `SwapAllowlistExtension` to restrict swaps to KYC'd or otherwise vetted addresses. If the router is allowlisted (the only way to enable router-mediated swaps), any non-allowlisted user can call `MetricOmmSimpleRouter.exactInputSingle()` and swap freely against the pool's liquidity. The allowlist guard is completely defeated. LP funds in the curated pool are exposed to unauthorized swappers, which can result in direct loss of LP principal through adverse-selection trades the pool was specifically configured to prevent. This is a **High** severity broken core pool functionality / admin-boundary break: an unprivileged user bypasses an explicitly configured access control boundary, exposing LP assets to unauthorized counterparties.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap entrypoint. No privileged access, special setup, or malicious initial configuration is required. Any user who can read the interface can call it. The only precondition is that the pool admin has allowlisted the router (which is required for any router-mediated swap to function), a configuration the admin is expected to make. The bypass is repeatable by any address on every swap.

## Recommendation
The `beforeSwap` hook signature receives `sender` (the pool's `msg.sender`) but not the original transaction initiator. The correct fix is to have the pool record the original transaction initiator via transient storage — analogous to how the router already uses `_setNextCallbackContext` — and expose it to extensions. The router should write the real user's address into a trusted transient slot before calling `pool.swap()`, and the pool should read that slot and pass it as the verified `sender` to extensions. Alternatively, the `beforeSwap` hook signature should be extended to carry a verified original sender set by the pool from a trusted periphery context rather than from raw `msg.sender`. Until fixed, `SwapAllowlistExtension` must not be used with `MetricOmmSimpleRouter` on pools where the allowlist is a security boundary.

## Proof of Concept
```
1. Pool admin deploys a pool with SwapAllowlistExtension configured.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   — required so that any router-mediated swap can pass the hook.
3. Alice (allowlisted directly) and Bob (NOT allowlisted) both want to swap.
4. Bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...}).
5. Router calls pool.swap(...) — msg.sender to pool = router address.
6. Pool calls _beforeSwap(router, recipient, ...).
7. SwapAllowlistExtension.beforeSwap checks allowedSwapper[pool][router] == true → passes.
8. Bob's swap executes successfully despite not being on the allowlist.
9. The curated pool's access control is completely bypassed.

Foundry test outline:
- Deploy pool with SwapAllowlistExtension.
- setAllowedToSwap(pool, router, true); do NOT allowlist Bob.
- vm.prank(Bob); router.exactInputSingle(...) → assert swap succeeds.
- vm.prank(Bob); pool.swap(...) directly → assert reverts with NotAllowedToSwap.
```
