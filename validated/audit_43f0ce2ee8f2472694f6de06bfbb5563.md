### Title
Valid SPV Proofs for Single-Transaction Blocks Are Unconditionally Rejected — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` contain a hard guard that rejects any empty merkle proof. For a Bitcoin block containing exactly one transaction (the coinbase), the merkle root equals the transaction hash and the correct merkle proof is an empty vector. The underlying `compute_root_from_merkle_proof` handles this case correctly, but the preceding `require!` check panics before the computation is reached, making it impossible for any NEAR caller to verify a transaction in a single-transaction block.

---

### Finding Description

In `verify_transaction_inclusion`, line 315 of `contract/src/lib.rs`:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
```

This check is unconditional. It fires before the merkle root computation and causes the call to panic for any caller supplying an empty proof.

In Bitcoin, a block with a single transaction has a merkle tree of depth zero: the merkle root is exactly the transaction hash. The valid SPV proof for that transaction is therefore an empty vector. `compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` already handles this correctly — with an empty `merkle_proof` the `for` loop does not execute and the function returns `transaction_hash` unchanged:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {   // ← loop body never runs for empty proof
        ...
    }
    current_hash   // returns transaction_hash unchanged
}
```

If `transaction_hash == header.block_header.merkle_root`, the proof is valid. The `require!` guard at line 315 prevents this correct path from ever being reached.

`verify_transaction_inclusion_v2` inherits the same defect because it delegates to `verify_transaction_inclusion` after its own coinbase check:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
```

For a single-transaction block both `merkle_proof` and `coinbase_merkle_proof` are empty, so the length-equality check at line 349 passes, the coinbase root check at lines 358–365 passes (empty proof returns `coinbase_tx_id` which equals `merkle_root`), and then the inner call panics on the empty-proof guard.

---

### Impact Explanation

The core value proposition of this contract is SPV verification. Any NEAR caller — a bridge, a payment verifier, or a dApp — that attempts to verify a transaction in a single-transaction Bitcoin block will have its call panic unconditionally, even when supplying a cryptographically correct proof. The verification result is permanently wrong for this class of blocks: the contract returns a false rejection instead of `true`. Downstream systems that rely on the contract's `verify_transaction_inclusion_v2` return value to gate fund releases or state transitions will be unable to process legitimate Bitcoin transactions from single-transaction blocks.

---

### Likelihood Explanation

Single-transaction blocks (containing only the coinbase) occur in Bitcoin during periods of low mempool activity and are common in early chain history and on testnets. Any relayer that submits such a block to the contract creates a permanently unverifiable block: no caller can ever produce a proof that passes the guard. The entry path requires no privilege — any unprivileged NEAR account can call `verify_transaction_inclusion` or `verify_transaction_inclusion_v2`.

---

### Recommendation

Remove the unconditional empty-proof guard. The correctness of the proof is already determined by whether the computed root matches the stored merkle root; an empty proof is valid precisely when the transaction is the sole transaction in the block. If a non-empty proof is desired as a sanity check for the common case, the check should be conditioned on the block having more than one transaction, which requires storing or supplying the transaction count — or the check should simply be dropped and the mathematical comparison trusted.

---

### Proof of Concept

1. A Bitcoin block is mined containing only the coinbase transaction. Its `merkle_root` equals `coinbase_txid`.
2. The relayer submits this block via `submit_blocks`; it is stored in `headers_pool` and promoted to the mainchain.
3. A NEAR caller invokes `verify_transaction_inclusion_v2` with:
   - `tx_id = coinbase_txid`
   - `tx_block_blockhash = <hash of the single-tx block>`
   - `tx_index = 0`
   - `merkle_proof = []`
   - `coinbase_tx_id = coinbase_txid`
   - `coinbase_merkle_proof = []`
   - `confirmations = 1`
4. The length check at line 349 passes (`0 == 0`).
5. The coinbase root check at lines 358–365 passes: `compute_root_from_merkle_proof(coinbase_txid, 0, &[])` returns `coinbase_txid` which equals `merkle_root`.
6. `verify_transaction_inclusion` is called. At line 315, `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` panics.
7. The call aborts. The valid proof is rejected.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
