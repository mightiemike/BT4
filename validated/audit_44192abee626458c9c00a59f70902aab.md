### Title
Division by Zero in `PerpEngine.socializeSubaccount` When `openInterest` Is Zero ‚Äî (`File: core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` divides by `state.openInterest` without a zero guard. When a product has no remaining open positions at the time of socialization ‚Äî a reachable state during liquidation finalization ‚Äî the transaction reverts, permanently blocking the finalization of insolvent subaccounts.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, when insurance is insufficient to cover a subaccount's negative `vQuoteBalance`, the protocol attempts to spread the loss across all open-interest holders:

```solidity
int128 fundingPerShare = -balance.vQuoteBalance.div(
    state.openInterest   // ‚Üê no zero check
);
```

`MathSD21x18.div` explicitly enforces `require(y != 0, ERR_DIV_BY_ZERO)`, so if `state.openInterest == 0`, the call reverts. [1](#0-0) 

`state.openInterest` is the sum of absolute position sizes across all participants in a product. It is maintained in `_updateBalance`:

```solidity
state.openInterest -= balance.amount.abs();   // pre-update
...
state.openInterest += balance.amount;          // post-update (long)
state.openInterest -= balance.amount;          // post-update (short)
``` [2](#0-1) 

`socializeSubaccount` is called from `_finalizeSubaccount` in `ClearinghouseLiq`, which is only reached when `txn.productId == type(uint32).max`. Before calling `socializeSubaccount`, `_finalizeSubaccount` enforces that **all perp positions of the liquidatee are zero**:

```solidity
require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
``` [3](#0-2) 

Then it calls:

```solidity
v.insurance = perpEngine.socializeSubaccount(
    txn.liquidatee,
    v.insurance
);
``` [4](#0-3) 

Inside `socializeSubaccount`, `getStateAndBalance` is called, which internally calls `_updateBalance(state, balance, 0, 0)`. When `balance.amount == 0`, the pre-update subtracts 0 and the post-update adds 0 ‚Äî `openInterest` is unchanged. If the liquidatee was the **only participant** in a given perp product, `state.openInterest == 0` at this point. [5](#0-4) 

The condition `balance.vQuoteBalance < 0` can be satisfied even with `balance.amount == 0`: a trader can close their position (amount ‚Üí 0) while retaining a negative `vQuoteBalance` from accumulated losses or funding payments. The division at line 166 then hits `state.openInterest == 0` and reverts. [6](#0-5) 

---

### Impact Explanation

The revert propagates through `_finalizeSubaccount` ‚Üí `liquidateSubaccountImpl` ‚Üí the `delegatecall` in `Clearinghouse.liquidateSubaccount`. The entire liquidation transaction reverts. The insolvent subaccount cannot be finalized: insurance cannot absorb its losses, losses cannot be socialized, and the subaccount remains in an unresolvable insolvent state. This is a **liquidation/settlement bypass** ‚Äî a broken protocol invariant, not merely a gas issue. [7](#0-6) 

---

### Likelihood Explanation

The trigger requires a perp product where the liquidatee is the sole remaining participant (or all other participants have zero positions), and the liquidatee has `balance.amount == 0` with `balance.vQuoteBalance < 0`. This is a realistic end-of-lifecycle scenario for low-liquidity or newly listed perp products. Any liquidator submitting a finalization transaction (`productId == type(uint32).max`) against such a subaccount will be permanently blocked.

---

### Recommendation

Add a zero guard before the division, mirroring the pattern already used in `PerpEngineState.updateStates` (line 111):

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; absorb loss silently or skip
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(
            state.openInterest
        );
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
``` [8](#0-7) 

---

### Proof of Concept

1. A perp product (e.g., `productId = 2`) is listed. Only one subaccount (`victim`) ever trades it.
2. `victim` opens a long position, accumulates losses, then closes the position (`balance.amount = 0`, `balance.vQuoteBalance < 0`). `state.openInterest = 0`.
3. `victim`'s maintenance health drops below zero (e.g., due to spot liabilities).
4. A liquidator calls `Endpoint ‚Üí Clearinghouse.liquidateSubaccount` with `txn.productId = type(uint32).max` to finalize `victim`.
5. `_finalizeSubaccount` passes all pre-checks (all perp amounts are 0), then calls `perpEngine.socializeSubaccount(victim, insurance)`.
6. Inside `socializeSubaccount`, for `productId = 2`: `balance.vQuoteBalance < 0` and `state.openInterest == 0`.
7. `MathSD21x18.div` reverts with `"DBZ"`. The entire liquidation transaction reverts.
8. `victim` cannot be finalized; insurance and socialization are permanently blocked for this subaccount. [9](#0-8)
