Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Unauthorized Address to Bypass the Deposit Allowlist — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual token-providing caller, always `msg.sender` of the pool call) and performs its allowlist check on `owner` (the LP-share recipient, a free parameter chosen by the caller). Because `owner` is attacker-controlled, any unprivileged address can bypass the deposit restriction by supplying any allowlisted address as `owner`. The deposit allowlist — the sole access-control gate for `addLiquidity` on permissioned pools — is fully defeated.

## Finding Description

In `MetricOmmPool.addLiquidity`, the pool invokes the extension hook as:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`msg.sender` is the actual depositor who will provide tokens via the `IMetricOmmSwapCallback`; `owner` is an arbitrary address supplied by the caller that receives LP shares. `ExtensionCalling._beforeAddLiquidity` encodes both and forwards them to the extension:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first positional argument (`sender`) is unnamed and ignored. The guard reads only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Since `owner` is freely chosen by the caller, any address can pass the guard by naming any allowlisted address as `owner`. The actual depositor (`sender`) is never validated. By contrast, `SwapAllowlistExtension.beforeSwap` correctly names and checks `sender` (first parameter) while ignoring the second (`recipient`), demonstrating the intended pattern that `DepositAllowlistExtension` fails to follow.

## Impact Explanation

The deposit allowlist is the sole access-control gate for `addLiquidity` on pools that configure `DepositAllowlistExtension`. Bypassing it allows any unprivileged address to add liquidity to a permissioned pool, directly altering `binTotals.scaledToken0` / `binTotals.scaledToken1` and `_binTotalShares` / `_positionBinShares` without admin consent. This constitutes a broken core pool functionality (permissioned liquidity provision is rendered inoperative) and an admin-boundary break (the pool admin's cap on depositors is bypassed by an unprivileged path). Existing LPs in the restricted pool suffer dilution of their positions, and downstream extensions such as `OracleValueStopLossExtension` that derive per-share metrics from live bin balances can be manipulated as a consequence.

## Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with `owner` set to any address that appears in `allowedDepositor`. Allowlisted addresses are observable on-chain via `AllowedToDepositSet` events or direct storage reads. No special role, flash loan, or multi-step setup is needed. Any unprivileged EOA or contract can trigger this immediately after pool deployment with the extension configured.

## Recommendation

Name and check `sender` (the actual token provider) instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

```
Setup:
  - Pool deployed with DepositAllowlistExtension configured.
  - Admin calls setAllowedToDeposit(pool, alice, true).
  - bob is NOT in allowedDepositor.

Attack:
  1. bob calls pool.addLiquidity(
         owner         = alice,   // allowlisted address — attacker-chosen
         salt          = 0,
         deltas        = <valid deltas>,
         callbackData  = ...,
         extensionData = ""
     );

  2. Pool calls _beforeAddLiquidity(msg.sender=bob, owner=alice, ...).
     Extension receives: beforeAddLiquidity(sender=bob [unnamed/ignored], owner=alice [checked]).
     Guard evaluates: allowedDepositor[pool][alice] == true → passes.

  3. bob's metricOmmSwapCallback provides tokens.
     LiquidityLib.addLiquidity credits LP shares to alice.
     binTotals.scaledToken0 / scaledToken1 are updated.

Result:
  bob (unauthorized) has deposited into the restricted pool.
  Pool bin balances are altered without admin consent.
  The deposit allowlist guard is fully bypassed.
