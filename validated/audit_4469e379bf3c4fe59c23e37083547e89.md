### Title
Deposited Tokens Permanently Stuck in `Endpoint` When Slow-Mode `DepositCollateral` Transaction Fails Silently — (`File: core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral` transfers user tokens into the contract **before** queuing the slow-mode transaction. When that queued transaction is later executed via `_executeSlowModeTransaction`, a `try/catch` silently swallows any revert from `processSlowModeTransaction`. The catch block contains no refund logic — a developer comment explicitly reads `// try return funds now removed`. If the inner call fails (e.g., because the product was delisted between deposit and execution), the user's tokens are permanently locked in the `Endpoint` contract with no recovery path.

---

### Finding Description

**Step 1 — Tokens transferred before slow-mode tx is queued**

In `depositCollateralWithReferral`, `handleDepositTransfer` pulls tokens from the caller into the `Endpoint` contract at line 144, and only *after* that does the function enqueue the `DepositCollateral` slow-mode transaction at lines 152–165. [1](#0-0) 

**Step 2 — Slow-mode tx deleted before the try/catch**

In `_executeSlowModeTransaction`, the queued transaction is deleted from storage at line 194 (`delete slowModeTxs[_slowModeConfig.txUpTo++]`) *before* the `try` block fires. There is no way to re-queue it on failure. [2](#0-1) 

**Step 3 — Silent failure with no refund**

The `try/catch` at lines 207–227 catches any revert from `processSlowModeTransaction` and does nothing. The developer comment on line 226 — `// try return funds now removed` — explicitly confirms that a refund mechanism existed and was deliberately removed, leaving no compensation for the depositor. [3](#0-2) 

**Step 4 — Failure path in `processSlowModeTransactionImpl`**

When the slow-mode tx type is `DepositCollateral`, `processSlowModeTransactionImpl` calls `clearinghouse.depositCollateral(txn)` at line 216. This call can revert if the product has been delisted between the time of deposit and the time of execution (the slow-mode delay is hardcoded to three days per line 153/377). A `DelistProduct` slow-mode transaction processed by the sequencer in that window is sufficient to trigger the failure. [4](#0-3) 

---

### Impact Explanation

Any user who deposits collateral for a product that is subsequently delisted before their slow-mode transaction executes will have their tokens permanently locked inside the `Endpoint` contract. There is no admin rescue function, no re-queue mechanism, and no refund path. The asset delta is the full deposited amount for every affected user.

---

### Likelihood Explanation

The slow-mode delay is hardcoded to three days. Product delisting is an admin-controlled slow-mode transaction that can be submitted and executed by the sequencer within that window. While this requires an admin action, it is a routine protocol operation (e.g., deprecating a market), not an exotic attack. Any user who deposits into a product that is being wound down faces this risk without any warning or protection. The `DirectDepositV1.creditDeposit` path also calls `depositCollateralWithReferral` in a loop, amplifying exposure. [5](#0-4) 

---

### Recommendation

Move `handleDepositTransfer` to *after* the slow-mode transaction is successfully queued, or — more robustly — restore the removed refund logic inside the `catch` block of `_executeSlowModeTransaction`. When `processSlowModeTransaction` fails for a `DepositCollateral` transaction, the contract should decode the original deposit amount and token from the deleted `SlowModeTx` and return the funds to the original depositor.

```diff
// In _executeSlowModeTransaction catch block:
} catch {
    if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
        assembly { invalid() }
    }
-   // try return funds now removed
+   // Refund deposited collateral if the deposit slow-mode tx failed
+   if (IEndpoint.TransactionType(uint8(txn.tx[0])) ==
+           IEndpoint.TransactionType.DepositCollateral) {
+       IEndpoint.DepositCollateral memory dep = abi.decode(
+           txn.tx[1:], (IEndpoint.DepositCollateral));

```

### Proof of Concept

1. Product `P` exists; user calls `depositCollateralWithReferral(subaccount, P, 1000e6, "")`.
2. `handleDepositTransfer` moves 1000 USDC from user → `Endpoint`. Slow-mode tx queued at index `N`.
3. Sequencer submits and executes a `DelistProduct` slow-mode tx for product `P` within the three-day window.
4. After three days, user (or anyone) calls `executeSlowModeTransaction()`.
5. `_executeSlowModeTransaction` deletes the tx at index `N`, then calls `processSlowModeTransaction` in a `try`.
6. `clearinghouse.depositCollateral` reverts because product `P` is delisted.
7. The `catch` block executes with `// try return funds now removed` — no refund issued.
8. User's 1000 USDC is permanently locked in `Endpoint` with no recovery path.
