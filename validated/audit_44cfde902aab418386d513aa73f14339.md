### Title
`verify_transaction_inclusion` Accepts Internal Merkle Tree Nodes as `tx_id`, Enabling Proof Forgery for Non-Existent Transactions — (File: `contract/src/lib.rs`)

---

### Summary

The still-callable `verify_transaction_inclusion` function does not validate that the caller-supplied `tx_id` is a leaf node (an actual transaction) in the block's Merkle tree. An unprivileged NEAR caller can supply an internal node hash as `tx_id` together with a valid proof path to the root, causing the function to return `true` for a transaction that does not exist in the block.

---

### Finding Description

`verify_transaction_inclusion` is a public NEAR contract method that accepts a `ProofArgs` struct containing a caller-controlled `tx_id`. It verifies the proof by computing the Merkle root from `tx_id`, `tx_index`, and `merkle_proof`, then comparing the result to the stored block header's `merkle_root`: [1](#0-0) 

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

The function performs no check that `tx_id` is a leaf node (i.e., an actual transaction hash) rather than an internal node of the Merkle tree. The contract itself documents this gap explicitly: [2](#0-1) 

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash. We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification."

Despite being marked `#[deprecated]`, the function carries `pub` visibility and the `#[pause]` attribute (not `#[private]`), meaning any unprivileged NEAR account can call it directly at any time the contract is unpaused: [3](#0-2) 

The `compute_root_from_merkle_proof` function in `merkle-tools` is a pure path-traversal computation — it has no concept of tree depth, leaf count, or whether the starting hash is a leaf or an internal node: [4](#0-3) 

This is the well-known 64-byte transaction Merkle proof forgery vulnerability (https://www.bitmex.com/blog/64-Byte-Transactions). The `verify_transaction_inclusion_v2` function was introduced to mitigate it by requiring a coinbase proof at position 0 with the same proof length: [5](#0-4) 

However, `verify_transaction_inclusion` (v1) was never removed and remains reachable.

**Analog mapping to C-01:** In C-01, `approve()` calls are permitted but the spender address parameter is not validated, allowing the session key to approve an arbitrary address. Here, `verify_transaction_inclusion` calls are permitted but the `tx_id` parameter is not validated to be a leaf node, allowing an attacker to supply an arbitrary internal node hash and receive a `true` proof result.

---

### Impact Explanation

Any downstream NEAR contract or application that calls `verify_transaction_inclusion` to gate cross-chain logic (e.g., unlocking NEAR tokens upon Bitcoin payment confirmation, minting wrapped assets, releasing escrow) can be deceived into accepting a forged proof. The attacker does not need to send any real Bitcoin transaction — they only need to know the Merkle tree structure of any mainchain block, which is fully public. The function returns `true` for a transaction that was never broadcast or mined.

---

### Likelihood Explanation

The function is publicly callable by any NEAR account with no role restriction. The required inputs — a real mainchain block hash and its Merkle tree structure — are freely available from any Bitcoin block explorer. The 64-byte Merkle forgery technique is well-documented with known construction methods. No privileged access, leaked keys, or social engineering is required.

---

### Recommendation

Remove `verify_transaction_inclusion` from the contract entirely, or gate it with `#[private]` to prevent external calls. All external callers should be directed exclusively to `verify_transaction_inclusion_v2`. If v1 must remain for backward compatibility, add an explicit check that `tx_index` is strictly less than the number of transactions in the block (which requires storing or passing the transaction count), or reject calls where `merkle_proof.len()` would allow an internal node to be treated as a leaf.

---

### Proof of Concept

1. Attacker identifies any real Bitcoin block `B` currently in the contract's mainchain (e.g., via `get_block_hash_by_height`).
2. Attacker fetches block `B`'s full transaction list from a public Bitcoin node and reconstructs its Merkle tree.
3. Attacker selects an internal node `I` at depth `D` (position `P` at that depth) in the tree. The proof from `I` to the root has length `total_depth - D`.
4. Attacker calls `verify_transaction_inclusion` with:
   - `tx_id` = `I` (internal node hash, not a real txid)
   - `tx_block_blockhash` = `B`'s hash
   - `tx_index` = `P`
   - `merkle_proof` = the `total_depth - D` sibling hashes from `I` up to the root
   - `confirmations` = 1
5. `compute_root_from_merkle_proof(I, P, proof)` traverses from `I` upward and correctly reconstructs `merkle_root`.
6. The function returns `true`.
7. Any downstream contract consuming this result treats the forged `tx_id` as a confirmed Bitcoin transaction. [6](#0-5) [7](#0-6)

```rust
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
```

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```
