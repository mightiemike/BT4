### Title
Hardcoded `lastKnownBonded=true` in `TierRedelegate` Enables Bonus Accrual During Unbonded Period — (`x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`TierRedelegate` unconditionally calls `pos.UpdateBonusCheckpoints(latestSeq, blockTime, true)` with a hardcoded `true` for `lastKnownBonded`, regardless of the destination validator's actual bond status. When the destination is currently unbonded (its latest recorded event is an UNBOND), the position's `LastKnownBonded` is incorrectly set to `true`. The next time `processEventsAndClaimBonus` runs after the validator re-bonds, it initializes `bonded=true` and computes bonus for the segment spanning the entire unbonded period, draining the rewards pool.

---

### Finding Description

**Step 1 — No destination bond-status check in `validateRedelegatePosition`.**

`validateRedelegatePosition` (msg_validate.go:67–103) checks owner, delegation status, same-validator guard, exit-lock elapsed, tier close-only, and active-redelegation. It never queries the destination validator's bond status. [1](#0-0) 

**Step 2 — Hardcoded `true` written to `LastKnownBonded`.**

After the SDK redelegate succeeds, `TierRedelegate` reads the destination validator's latest event sequence and calls:

```go
pos.UpdateBonusCheckpoints(latestSeq, sdkCtx.BlockTime(), true)
``` [2](#0-1) 

`UpdateBonusCheckpoints` blindly stores the supplied boolean into `LastKnownBonded`:

```go
func (p *Position) UpdateBonusCheckpoints(lastEventSeq uint64, t time.Time, lastKnownBonded bool) {
    p.LastEventSeq = lastEventSeq
    p.LastBonusAccrual = t
    p.LastKnownBonded = lastKnownBonded
}
``` [3](#0-2) 

If the destination validator's latest event is an UNBOND (seq=N), the position is stored with `LastEventSeq=N` and `LastKnownBonded=true` — the opposite of the correct state.

**Step 3 — `processEventsAndClaimBonus` trusts `LastKnownBonded` to seed the loop.**

```go
bonded := pos.LastKnownBonded   // incorrectly true
segmentStart := pos.LastBonusAccrual
...
for _, entry := range events {
    if bonded {
        bonus := k.computeSegmentBonus(...)  // fires for the unbonded segment
        totalBonus = totalBonus.Add(bonus)
    }
    switch evt.EventType {
    case UNBOND: bonded = false
    case BOND:   bonded = true
    }
    segmentStart = evt.Timestamp
}
``` [4](#0-3) 

When the validator later re-bonds (BOND event at seq=N+1), the loop fires with `bonded=true` and computes bonus for `[LastBonusAccrual, bondEventTimestamp]` — the entire period the validator was unbonded.

**Step 4 — The defensive check at line 206 does NOT protect intermediate segments.**

```go
if bonded && val.IsBonded() {   // only guards the tail segment
    ...
}
``` [5](#0-4) 

This guard only prevents computing bonus for the segment *after* all events are processed. The bonus already computed inside the loop for the unbonded segment is unaffected.

---

### Impact Explanation

The position owner receives bonus rewards for the period during which the destination validator was unbonded. The bonus is paid from the `RewardsPoolName` module account via `bankKeeper.SendCoinsFromModuleToAccount`. The magnitude equals `shares × tokensPerShare × BonusAPY × unbondedDuration / SecondsPerYear`, which can be material for large positions or long unbonded periods.

---

### Likelihood Explanation

The attack is reachable via a standard `MsgTierRedelegate` transaction — no governance, no privileged role, no leaked keys. A validator can be jailed (e.g., for downtime), transition to unbonded after the unbonding period, and remain a valid redelegate target in the SDK. The attacker simply redelegates to it, waits for it to re-bond (or unjails it themselves if they control the validator), then claims rewards.

---

### Recommendation

Replace the hardcoded `true` with the actual current bond status of the destination validator at the time of redelegate:

```go
dstVal, err := ms.stakingKeeper.GetValidator(ctx, dstValAddr)
if err != nil {
    return nil, err
}
pos.UpdateBonusCheckpoints(latestSeq, sdkCtx.BlockTime(), dstVal.IsBonded())
```

This mirrors the invariant that `LastKnownBonded` must reflect the validator's state *after* the last processed event, not an assumed default.

### Proof of Concept

```
1. Setup: bonded validator V1 (src), unbonded validator V2 (dst, was jailed → unbonded).
   V2 has UNBOND event at seq=5 as its latest event.

2. Owner calls MsgTierRedelegate{src=V1, dst=V2}.
   → validateRedelegatePosition: passes (no bond-status check on dst).
   → SDK redelegate: succeeds (SDK allows delegating to unbonded validators).
   → latestSeq = 5 (the UNBOND event seq for V2).
   → UpdateBonusCheckpoints(5, T_redelegate, true)
   → Stored: LastEventSeq=5, LastBonusAccrual=T_redelegate, LastKnownBonded=true.

3. Advance time by D (e.g., 30 days). V2 is still unbonded.

4. V2 is unjailed and becomes bonded → AfterValidatorBonded hook fires →
   BOND event appended at seq=6 with timestamp T_bond.

5. Owner calls MsgClaimTierRewards.
   → processEventsAndClaimBonus:
       bonded = true  (from LastKnownBonded — WRONG)
       segmentStart = T_redelegate
       events = [seq=6: BOND at T_bond]
       Loop iteration (BOND event):
           bonded=true → computeSegmentBonus(T_redelegate, T_bond, ...)
           → bonus paid for the 30-day unbonded period  ← INCORRECT
           bonded = true (BOND event)
           segmentStart = T_bond
       After loop: bonded && val.IsBonded() → bonus for [T_bond, now] ← also paid

6. Assert: owner received bonus for the 30-day unbonded period.
   Expected: zero bonus for that period.
```
