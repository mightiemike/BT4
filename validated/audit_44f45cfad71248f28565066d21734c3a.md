Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Original User, Allowing Any User to Bypass a Curated Pool's Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender` — the direct caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` at the pool, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. Any pool admin who allowlists the router to permit router-mediated swaps on a curated pool silently opens the allowlist to every user who routes through it.

## Finding Description

**Root cause — `MetricOmmPool.swap` passes `msg.sender` as `sender`:**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at line 231. When the caller is `MetricOmmSimpleRouter`, `msg.sender` is the router address.

**`ExtensionCalling._beforeSwap` forwards `sender` unchanged:**

`ExtensionCalling._beforeSwap` (lines 149–177) encodes the `sender` argument directly into the `abi.encodeCall` payload dispatched to every configured extension. No transformation or substitution occurs.

**`SwapAllowlistExtension.beforeSwap` checks the wrong identity:**

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When the user routes through `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)` directly (lines 72–80 of `MetricOmmSimpleRouter.sol`), so `sender` is the router, not the original user.

**Multi-hop path compounds the issue:**

In `exactInput`, hops after the first set the payer to `address(this)` (the router) at line 103. Every hop's `pool.swap` call therefore presents the router as `sender` to the extension.

**Impossible admin choice:**

- Do not allowlist the router → all router-mediated swaps revert, including those from legitimately allowlisted users.
- Allowlist the router → `allowedSwapper[pool][router] == true`, so the guard passes for *any* caller who routes through the router, regardless of whether that caller is on the allowlist.

There is no mechanism in the router to forward the original `msg.sender` to the pool as the `sender` identity.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, whitelisted market makers, or protocol-internal actors) can be fully bypassed by any unprivileged user routing through `MetricOmmSimpleRouter`. The bypassing user executes swaps at oracle-derived prices, extracting value from LP positions deposited under the assumption that only allowlisted counterparties would trade against them. This constitutes direct loss of LP principal and fee revenue attributable to trades that should have been blocked — a High-severity impact under Sherlock's direct loss of user principal threshold.

## Likelihood Explanation

The router is the primary user-facing swap entrypoint documented and deployed by the protocol. Any pool admin who enables a swap allowlist and also wants to support normal router-mediated swaps must allowlist the router — the exact configuration that opens the bypass. The trigger requires no special privilege: any EOA can call `router.exactInputSingle` with a valid pool address. The condition is reachable on every curated pool that has the router allowlisted, and the attack is repeatable at zero marginal cost per swap.

## Recommendation

The `sender` identity forwarded to extensions must represent the original economic actor, not the intermediary contract. Two complementary fixes:

1. **In the router:** store the original `msg.sender` in a dedicated transient storage slot (analogous to the existing `T_PAYER_SLOT` in `TransientCallbackPool`) before calling `pool.swap`, and have `SwapAllowlistExtension.beforeSwap` decode and verify it from that well-known slot rather than trusting the `sender` argument.
2. **In the extension:** document explicitly that `sender` is the direct caller of `pool.swap`, not the end user, and require pool admins to allowlist routers only when the allowlist is not intended to gate individual users.

The cleaner fix is option 1: the router writes the original caller to a known transient slot, and the extension reads it from there, making the check spoof-resistant because only the router (a factory-validated contract) can write to its own transient context.

## Proof of Concept

```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension as extension1
  admin calls extension.setAllowedToSwap(pool, router, true)   // to enable router swaps
  alice = allowlisted user (allowedSwapper[pool][alice] == true)
  bob   = non-allowlisted user (allowedSwapper[pool][bob] == false)

Attack:
  bob calls router.exactInputSingle({pool: pool, ...})
    → MetricOmmSimpleRouter calls pool.swap(recipient=bob, ...)   // msg.sender at pool = router
    → MetricOmmPool._beforeSwap(sender=router, ...)
    → SwapAllowlistExtension.beforeSwap checks allowedSwapper[pool][router] == true  ✓
    → swap executes — bob receives output tokens

Expected:
  SwapAllowlistExtension should check allowedSwapper[pool][bob] == false → revert NotAllowedToSwap

Actual:
  The check passes because the router is allowlisted; bob's swap settles at oracle price,
  extracting value from LP positions deposited under a curated-counterparty assumption.

Foundry test outline:
  1. Deploy pool with SwapAllowlistExtension.
  2. allowlist router, not bob.
  3. Call router.exactInputSingle from bob's address.
  4. Assert swap succeeds (no revert) — demonstrating the bypass.
  5. Assert allowedSwapper[pool][bob] == false to confirm bob was never individually authorized.
```
