Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the actual user when swaps route through `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension` gates swaps per-user by checking the `sender` argument passed from the pool, but `sender` is always `msg.sender` of the `pool.swap()` call. When users route through `MetricOmmSimpleRouter`, the router is the direct caller of `pool.swap()`, so the extension checks the router's address against the allowlist instead of the actual user's address. This produces two fund-impacting outcomes: either the router being allowlisted nullifies per-user access control entirely (allowlist bypass), or allowlisted users are permanently locked out of the canonical router path.

## Finding Description
**Root cause — actor binding:**

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that value as the first positional argument forwarded to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` reads that first argument as `sender` and checks it against `allowedSwapper[pool][sender]`: [3](#0-2) 

**What the router actually sends:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly. The router is therefore `msg.sender` inside the pool, so `sender` delivered to the extension is the router's address, not the user who called the router: [4](#0-3) 

The same mismatch applies to `exactInput` (router is `msg.sender` for every hop), `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

**Why existing guards fail:**

There is no mechanism in the extension, the pool, or the router to forward the original `msg.sender` of the router call. The extension has no awareness of intermediary contracts and no fallback identity resolution. The `allowedSwapper` mapping is keyed purely on the address received as `sender`, which is structurally always the router when the router is used.

## Impact Explanation
**Scenario A — Allowlist bypass (high impact):** A pool admin allowlists the router address to enable router-based swaps. Because the extension checks `allowedSwapper[pool][router]`, every user who calls the router passes the guard regardless of whether they are individually allowlisted. The curated pool's per-user access control is completely nullified for the router path — any unprivileged user can swap.

**Scenario B — Allowlisted users locked out (medium impact):** If the admin does not allowlist the router, every call through `MetricOmmSimpleRouter` reverts with `NotAllowedToSwap`, even for users who are individually allowlisted. The canonical periphery swap path is permanently broken for the pool.

Both outcomes break the invariant that a curated pool must enforce the same allowlist policy regardless of which supported public entrypoint reaches it, and directly affect user funds and pool usability.

## Likelihood Explanation
No special privileges, malicious setup, non-standard tokens, or off-chain components are required. Any pool operator who deploys both `SwapAllowlistExtension` and `MetricOmmSimpleRouter` together triggers one of the two scenarios immediately upon normal operation. The router is the canonical swap entrypoint for end users, making this a near-certain collision in any production deployment of a curated pool.

## Recommendation
The extension must identify the true economic actor. The preferred fix is router-aware actor forwarding: have `MetricOmmSimpleRouter` encode the original `msg.sender` as the first word of `extensionData`, and have `SwapAllowlistExtension.beforeSwap` decode and use that value when `sender` is a known/trusted router. The extension should validate that the forwarded address is consistent with the callback payer stored in transient storage to prevent spoofing. Alternatively, pass the user through the router by encoding `msg.sender` in `extensionData` and updating the extension interface to decode it, requiring a coordinated interface change across router and extension.

## Proof of Concept
```
Setup:
  pool deployed with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)   // alice is the intended allowlisted user
  admin calls setAllowedToSwap(pool, router, true)  // admin enables router-based swaps

Attack (Scenario A — bypass):
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: bob, ...})

  Execution trace:
    router → pool.swap(recipient=bob, ...)          // msg.sender = router
    pool   → _beforeSwap(sender=router, ...)
    pool   → extension.beforeSwap(sender=router, recipient=bob, ...)
    extension checks: allowedSwapper[pool][router]  // TRUE — router is allowlisted
    → swap proceeds; bob receives tokens despite never being allowlisted

Broken functionality (Scenario B):
  alice (allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: alice, ...})

  Execution trace:
    router → pool.swap(recipient=alice, ...)        // msg.sender = router
    pool   → extension.beforeSwap(sender=router, ...)
    extension checks: allowedSwapper[pool][router]  // FALSE — router not allowlisted
    → revert NotAllowedToSwap; alice cannot use the router
```
