### Title
ETH Permanently Stuck in `DirectDepositV1` When `creditDeposit` Is Called With No Registered Products — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1` accepts native ETH via `receive()`, immediately wrapping it to WETH. A separate call to `creditDeposit()` is required to route that WETH into the protocol. If `creditDeposit()` is called when `SpotEngine` has no registered products (i.e., `getProductIds()` returns an empty array), the deposit loop is silently skipped, the WETH remains in the contract, and the user's subaccount receives no credit. The only recovery path is `withdraw()`, which is `onlyOwner`.

---

### Finding Description

The two-step deposit flow in `DirectDepositV1` is:

1. **Step 1 — `receive()`** (lines 64–67): Any caller sends native ETH. The contract immediately wraps it by calling `wrappedNative.call{value: msg.value}("")`. This call succeeds unconditionally (it reverts only on wrap failure). WETH is now held by the contract.

2. **Step 2 — `creditDeposit()`** (lines 83–101): Queries `spotEngine.getProductIds()` and iterates over the result. For each product, if the contract holds a balance of the corresponding token, it approves and calls `endpoint.depositCollateralWithReferral(...)`.

The critical flaw is in `creditDeposit()`:

```solidity
// core/contracts/DirectDepositV1.sol, lines 83–101
function creditDeposit() external {
    uint32[] memory productIds = spotEngine.getProductIds();
    for (uint256 i = 0; i < productIds.length; i++) {   // silently skipped if empty
        ...
    }
}
```

There is **no guard** requiring `productIds.length > 0`. If `getProductIds()` returns an empty array, the loop body never executes. The WETH balance accumulated from prior `receive()` calls remains in the contract indefinitely. The user's subaccount is never credited. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

A user who sends ETH to `DirectDepositV1` before any products are registered in `SpotEngine` (or after all products are removed) will have their ETH wrapped to WETH and permanently locked in the contract. Their protocol subaccount receives zero credit. The only recovery mechanism is `withdraw(token)`, which is restricted to `onlyOwner`. [3](#0-2) 

This is a direct, permanent asset loss for the depositing user with no self-service recovery path.

---

### Likelihood Explanation

The scenario is realistic in two concrete situations:

1. **Protocol initialization window**: `DirectDepositV1` is deployed and users send ETH before the operator has registered any products in `SpotEngine`. The `receive()` wraps ETH successfully, giving no indication of failure, and a subsequent `creditDeposit()` silently does nothing.
2. **Product removal**: If all products are removed from `SpotEngine` after the contract is live, any ETH sent during that window is similarly stranded.

The `receive()` function gives no warning — it succeeds and emits no event indicating that the WETH cannot yet be routed. A user has no on-chain signal that calling `creditDeposit()` will be a no-op. [1](#0-0) 

---

### Recommendation

Add a guard in `creditDeposit()` requiring at least one registered product before proceeding:

```solidity
function creditDeposit() external {
    uint32[] memory productIds = spotEngine.getProductIds();
+   require(productIds.length > 0, "No products registered");
    for (uint256 i = 0; i < productIds.length; i++) {
        ...
    }
}
```

This mirrors the recommended fix from M-10 (`require(derivativeCount > 0, ...)`). Alternatively, the `receive()` function could also guard against wrapping when no products are registered, preventing funds from entering the contract in an unroutable state. [4](#0-3) 

---

### Proof of Concept

1. Deploy `DirectDepositV1` pointing to a `SpotEngine` with zero registered products (`getProductIds()` returns `[]`).
2. User sends `1 ETH` to `DirectDepositV1`. `receive()` executes: `wrappedNative.call{value: 1 ether}("")` succeeds. Contract now holds `1 WETH`.
3. User (or anyone) calls `creditDeposit()`. `spotEngine.getProductIds()` returns `[]`. The `for` loop at line 85 iterates zero times. Function returns without depositing anything.
4. `WETH.balanceOf(address(directDepositV1)) == 1 ether` — funds are stuck.
5. User's subaccount balance in the protocol is unchanged (zero credit).
6. User cannot call `withdraw()` — it is `onlyOwner`. Funds are permanently inaccessible to the user. [5](#0-4)

        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            address tokenAddr = spotEngine.getToken(productId);

            IIERC20Base token = IIERC20Base(tokenAddr);
            uint256 balance = token.balanceOf(address(this));

                token.approve(address(endpoint), balance);

                    productId,
                    uint128(balance),
                    "-1"
                );

        }
    }
