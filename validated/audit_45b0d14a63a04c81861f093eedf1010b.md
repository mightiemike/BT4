### Title
Incomplete Coinbase Anchor in `verify_transaction_inclusion_v2` Allows Merkle Proof Forgery for Non-Existent Transactions — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to fix the 64-byte transaction Merkle proof forgery vulnerability by anchoring proof depth to the coinbase transaction. The mitigation is incomplete: the contract never verifies that `coinbase_tx_id` is the actual coinbase transaction hash. An unprivileged NEAR caller can supply two internal Merkle tree nodes — one as `coinbase_tx_id`, one as `tx_id` — with equal-length proofs that both compute to the block's real Merkle root. The function returns `true` for a transaction that does not exist in the block.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces two guards before delegating to the deprecated v1 function:

1. `coinbase_merkle_proof.len() == merkle_proof.len()`
2. `compute_root_from_merkle_proof(coinbase_tx_id, 0, coinbase_merkle_proof) == merkle_root` [1](#0-0) 

The intended invariant is: the real coinbase transaction lives at index 0 at the maximum leaf depth of the Merkle tree, so its proof has length `tree_depth`. Requiring `merkle_proof` to match that length forces the target transaction to also be a leaf, preventing an attacker from substituting a shorter-proof internal node.

The invariant breaks because `coinbase_tx_id` is **fully attacker-controlled** and is never checked against the actual coinbase transaction stored anywhere in the contract. The contract stores only the Merkle root in `block_header.merkle_root`; it has no record of the real coinbase hash. [2](#0-1) 

`compute_root_from_merkle_proof` is a pure positional hash-chain function with no leaf/internal-node distinction: [3](#0-2) 

Consider a block whose four transactions produce this tree:

```
        R = hash(AB, CD)
       / \
      AB   CD
     / \ / \
    T0 T1 T2 T3        (T0 = real coinbase)
```

An attacker computes `AB = double_sha256(T0 || T1)` and `CD = double_sha256(T2 || T3)` from public block data, then calls `verify_transaction_inclusion_v2` with:

| Field | Attacker-supplied value |
|---|---|
| `coinbase_tx_id` | `AB` (internal node, **not** T0) |
| `coinbase_merkle_proof` | `[CD]` (length 1) |
| `tx_id` | `CD` (internal node, **not** a real tx) |
| `tx_index` | `1` |
| `merkle_proof` | `[AB]` (length 1) |

Trace through `compute_root_from_merkle_proof`:

- Coinbase check: `hash(AB, CD) = R` ✓ (position 0 is even)
- Length check: `1 == 1` ✓
- Transaction check (v1): `hash(AB, CD) = R` ✓ (position 1 is odd → `hash(AB, CD)`)

All three guards pass. The function returns `true` for `tx_id = CD`, which is an internal Merkle node and corresponds to no real Bitcoin transaction. [4](#0-3) 

The deprecated `verify_transaction_inclusion` (v1) carries an explicit warning that it may accept internal nodes as transaction hashes: [5](#0-4) 

`verify_transaction_inclusion_v2` was supposed to close that gap but does not.

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion_v2` to gate fund releases, cross-chain actions, or other security-critical operations can be deceived into accepting a forged proof. An attacker can assert that an arbitrary 32-byte internal Merkle node is a confirmed Bitcoin transaction in any block that has been submitted to the light client, without performing any hash collision computation. This enables theft of funds from any bridge or protocol that relies on this verification function as a trust anchor.

---

### Likelihood Explanation

The attack requires zero brute-force computation. Every Bitcoin block with two or more transactions (i.e., virtually every block ever mined) exposes at least one pair of internal nodes at the same depth. All required values (`AB`, `CD`) are derived from publicly available block data. The attacker only needs to:

1. Pick any block already accepted by the light client.
2. Download its transaction list (public).
3. Compute the first level of the Merkle tree (two SHA-256d calls).
4. Submit the crafted `ProofArgsV2` to `verify_transaction_inclusion_v2`.

No privileged role, no key material, and no timing dependency is required.

---

### Recommendation

The root cause is that `coinbase_tx_id` is accepted as an opaque 32-byte value with no binding to the actual coinbase transaction. Two concrete fixes:

1. **Require the caller to supply the raw coinbase transaction bytes.** Hash them on-chain (`double_sha256`) and use the result as `coinbase_tx_id`. This eliminates the ability to substitute an internal node.

2. **Enforce that the coinbase proof length equals the full tree depth.** Store the Merkle tree depth (or the transaction count) in `ExtendedHeader` at submission time, and require `coinbase_merkle_proof.len() == stored_depth`. An internal node at depth `d < tree_depth` produces a proof of length `tree_depth - d`, which would then fail the length check even if `coinbase_tx_id` is fake.

Option 1 is the stronger fix because it makes the coinbase anchor cryptographically binding rather than structurally inferred.

---

### Proof of Concept

Using Bitcoin mainnet block 685452 (already used in the contract's own test suite) as an example target:

1. Fetch the block's transaction list (public RPC).
2. Compute `AB = double_sha256(T0_bytes || T1_bytes)` and `CD = double_sha256(T2_bytes || T3_bytes)`.
3. Confirm `double_sha256(AB || CD) == block.merkle_root`.
4. Call `verify_transaction_inclusion_v2` with:
   - `tx_block_blockhash` = block 685452 hash (already in the contract after `init`)
   - `coinbase_tx_id` = `AB`
   - `coinbase_merkle_proof` = `[CD]`
   - `tx_id` = `CD`
   - `tx_index` = `1`
   - `merkle_proof` = `[AB]`
   - `confirmations` = `1`
5. The function returns `true`, falsely certifying `CD` as a confirmed Bitcoin transaction. [1](#0-0) [3](#0-2) [6](#0-5)

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
