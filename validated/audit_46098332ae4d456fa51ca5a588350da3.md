Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of originating user, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of `MetricOmmPool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. Any pool admin who allowlists the router (the necessary step to let allowlisted users use the standard periphery) simultaneously opens the pool to every user, completely defeating the allowlist invariant.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap()`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // whoever called pool.swap() — the router when routed
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap()` forwards this `sender` value as the first argument to each configured extension's `beforeSwap` selector (lines 162-176). `SwapAllowlistExtension.beforeSwap` then checks it against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is the router address when the swap is routed. `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly without forwarding the originating user:

```solidity
// MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ...);
```

The originating user's address (`msg.sender`) is stored only in transient callback context for payment purposes — it is never forwarded to `pool.swap()` as an argument. The same applies to `exactInput` (all hops call `pool.swap()` with `msg.sender = router`) and `exactOutput` (the recursive `_exactOutputIterateCallback` also calls subsequent pools with `msg.sender = router`).

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly ignores the `sender` argument (marked `address` with no name) and checks `owner` instead — the position owner explicitly threaded through the call stack — making it immune to this class of bypass.

**Attack path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a curated set.
2. Admin allowlists `alice`: `allowedSwapper[pool][alice] = true`.
3. Admin allowlists the router so `alice` can use the standard periphery: `allowedSwapper[pool][router] = true`.
4. `bob` (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(...)` — pool sees `msg.sender = router`.
6. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
7. `bob` swaps on the curated pool without being individually allowlisted.

## Impact Explanation
A pool admin who intends to restrict swaps to a curated set of users cannot enforce that restriction if those users need to use the canonical router. Allowlisting the router — the only way to let allowlisted users use the periphery — simultaneously opens the pool to every user. Any user can execute swaps on a pool designed to be private, draining LP value at oracle-anchored prices that the pool admin assumed only trusted counterparties would access. This is a direct loss of LP principal and protocol fees on curated pools, and constitutes broken core pool functionality (the allowlist invariant is violated).

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical periphery entry point. Pool admins who deploy `SwapAllowlistExtension` and want their allowlisted users to use the router will naturally allowlist the router — this is the expected operational pattern. No privileged access beyond the admin's own intended setup is required by the attacker. The bypass is reachable by any user with no unusual preconditions.

## Recommendation
The pool's `swap()` interface carries no originator address, so the fix must be applied in the extension or the router:

1. **Preferred — router forwards originator in `extensionData`:** The router prepends `abi.encode(msg.sender)` to `extensionData` before calling `pool.swap()`. `SwapAllowlistExtension.beforeSwap` decodes and checks this address instead of `sender`. The extension must also verify that `msg.sender` (the pool's caller, i.e., `sender`) is a trusted router; otherwise any caller can forge the originator field.

2. **Minimal guard:** Document that `SwapAllowlistExtension` is incompatible with the router and revert in `beforeSwap` if `sender` is a known router address, forcing direct pool interaction only.

## Proof of Concept
```solidity
// Setup: pool with SwapAllowlistExtension
// Admin allowlists alice and the router
swapExtension.setAllowedToSwap(address(pool), alice, true);
swapExtension.setAllowedToSwap(address(pool), address(router), true);
// bob is NOT allowlisted

// bob bypasses the allowlist via the router
vm.prank(bob);
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool:             address(pool),
    tokenIn:          address(token0),
    recipient:        bob,
    deadline:         block.timestamp + 1,
    zeroForOne:       true,
    amountIn:         1000,
    amountOutMinimum: 0,
    priceLimitX64:    0,
    extensionData:    ""
}));
// Succeeds: pool.swap() sees msg.sender = router, which is allowlisted.
// bob swaps on the curated pool without being individually allowlisted.
```
