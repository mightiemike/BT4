### Title
`op_lsh` Ignores `max_cost` Entirely, Allowing Execution Beyond Budget — (`File: src/more_ops.rs`)

### Summary

`op_lsh` (logical shift) accepts a `max_cost` parameter but explicitly ignores it (underscore-prefixed `_max_cost`). The shift operation and its heap allocation execute unconditionally, and the cost is only computed and returned *after* the work is done — never checked against the budget inside the operator. Every comparable operator in the same file calls `check_cost()` during or before its expensive work. This is a direct analog to the "Gas Fee Not Shown" class: the fee/cost is not enforced at the point of execution, allowing one operation to exceed the declared budget.

### Finding Description

`op_lsh` is declared as:

```rust
pub fn op_lsh(a: &mut Allocator, input: NodePtr, _max_cost: Cost, _flags: ClvmFlags) -> Response {
``` [1](#0-0) 

The underscore prefix on `_max_cost` is not cosmetic — it is never read inside the function body. The operator:

1. Parses both arguments unconditionally.
2. Performs the full `BigUint` shift (`i0 << a1` or `i0 >> -a1`).
3. Allocates the result atom via `a.new_number(v)`.
4. **Only then** computes the cost and returns it — with no `check_cost()` call anywhere.

```rust
    let v: Number = if a1 > 0 { i0 << a1 } else { i0 >> -a1 };
    let l1 = limbs_for_int(&v);
    let r = a.new_number(v)?;
    let cost = LSHIFT_BASE_COST + ((l0 + l1) as Cost) * LSHIFT_COST_PER_BYTE;
    Ok(malloc_cost(a, cost, r))
``` [2](#0-1) 

Contrast with every comparable operator. `op_sha256` calls `check_cost()` inside its argument loop: [3](#0-2) 

`op_add` and `op_subtract` both call `check_cost()` per argument in both fast and slow paths: [4](#0-3) 

The main evaluator loop does check cost after each operation returns:

```rust
if cost > effective_max_cost {
    return Err(EvalErr::CostExceeded);
}
``` [5](#0-4) 

But this check fires *after* `op_lsh` has already allocated the result. The allocation is not rolled back.

### Impact Explanation

An attacker-controlled CLVM program can call `lsh` with a large input atom and a shift value up to ±65535 bits. The shift limit is enforced: [6](#0-5) 

A 65535-bit left shift on an 8-byte input produces an output of roughly 8200 bytes. The cost for this single call is `LSHIFT_BASE_COST + (8 + 8200) * LSHIFT_COST_PER_BYTE = 277 + 8208 * 3 ≈ 24,901`. If the remaining budget is 1 when `op_lsh` is invoked, the operation still completes, the heap atom is allocated, and only then does the main loop detect the overrun and abort. The heap allocation is not reclaimed (no checkpoint/restore wraps individual operator calls). This allows a program to force one heap allocation beyond the declared budget, bypassing the cost model's enforcement boundary.

### Likelihood Explanation

Any caller that can submit attacker-controlled CLVM bytes to `run_program` — including the Chia full node evaluating spend bundles — is exposed. The trigger requires only a valid two-argument `lsh` call placed at the point where the remaining budget is small. This is straightforwardly constructible by an attacker who knows the cost model. Likelihood is **medium**: the heap overrun is bounded (≤ ~8 KB per call), but it is a reliable, reproducible bypass of the cost enforcement invariant.

### Recommendation

Add a `check_cost` call inside `op_lsh` before performing the shift and allocation, mirroring the pattern used by every other operator in the file. The `_max_cost` parameter should be renamed to `max_cost` and used:

```rust
pub fn op_lsh(a: &mut Allocator, input: NodePtr, max_cost: Cost, _flags: ClvmFlags) -> Response {
    // ... parse args ...
    let cost = LSHIFT_BASE_COST + ((l0 + l1_estimate) as Cost) * LSHIFT_COST_PER_BYTE;
    check_cost(cost, max_cost)?;
    // ... perform shift and allocate ...
}
```

Alternatively, compute a conservative upper-bound cost before the shift (using `l0` and the shift amount to bound `l1`) and check it before allocating.

### Proof of Concept

```

; Attacker submits this as a spend bundle condition program
