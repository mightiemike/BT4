Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual Swapper, Allowing Any User to Bypass the Curated-Pool Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument forwarded by the pool, which is the pool's own `msg.sender` at the time `swap` is called. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so `sender` resolves to the router address — not the end user. A pool admin who allowlists the router to enable router-mediated swaps inadvertently opens the gate to every user, because the allowlist check never sees the actual trader's address. This breaks the core allowlist invariant and exposes curated-pool LP liquidity to any unprivileged caller.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // <-- whoever called pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct namespace) and `sender` is whoever called `pool.swap(...)`. When the call path is `User → MetricOmmSimpleRouter → pool.swap(...)`, the pool's `msg.sender` is the router, so `sender = address(router)`. The check evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

`MetricOmmSimpleRouter.exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` all call `IMetricOmmPoolActions(pool).swap(...)` directly — none of them encode the originating user address into `extensionData` or any other field the extension could inspect. There is no mechanism in the current code for the extension to recover the actual trader identity when the router is the intermediary.

The `DepositAllowlistExtension` does not share this flaw because `beforeAddLiquidity` checks the `owner` parameter (second argument), which the pool and `MetricOmmPoolLiquidityAdder` both set to the actual LP owner, not the adder contract itself.

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of addresses (e.g., KYC-verified counterparties, protocol-owned bots, or whitelisted market makers) provides no real restriction when the router is allowlisted. Any arbitrary user can call `MetricOmmSimpleRouter.exactInput` / `exactOutput` and execute swaps against the pool, draining LP liquidity at oracle-derived prices. This constitutes a direct loss of LP principal and breaks the core pool functionality the allowlist was designed to protect. The exact corrupted value is `allowedSwapper[pool][sender]`: the extension reads the router's allowlist entry instead of the actual trader's entry, causing the gate to pass for all users when the router is allowlisted.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary production entry point for swaps. Any pool admin who deploys `SwapAllowlistExtension` and also wants users to be able to swap via the router must allowlist the router — the natural and expected operational step — which immediately opens the bypass to all users. The trigger requires no special privilege: any unprivileged address can call the public router functions. The scenario is directly reachable and repeatable.

## Recommendation

The pool should forward the original transaction initiator or accept an explicit sender override from trusted periphery contracts via `extensionData`. The cleanest fix is to have the router encode the actual user address in `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check that address when the caller (`msg.sender` in the extension, i.e., the pool) is reached via a known trusted router. Alternatively, the pool's `swap` entrypoint can accept an explicit `onBehalfOf` parameter that the extension checks instead of the raw `msg.sender`-derived `sender`. A registry of trusted routers in the extension, combined with `extensionData`-decoded user addresses, would allow correct enforcement without breaking direct swaps.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps, and does **not** allowlist `attacker`.
3. `attacker` calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the pool.
4. The router calls `pool.swap(...)` — pool's `msg.sender` is the router.
5. Pool calls `extension.beforeSwap(router, ...)` — `sender = address(router)`.
6. `allowedSwapper[pool][router] == true` → check passes.
7. The attacker's swap executes against the curated pool, bypassing the intended allowlist gate.

Foundry test outline:
```solidity
function test_routerBypassesAllowlist() public {
    // Setup: pool with SwapAllowlistExtension, only router allowlisted
    extension.setAllowedToSwap(pool, address(router), true);
    // attacker is NOT allowlisted
    
    // Attacker calls router directly
    vm.prank(attacker);
    router.exactInputSingle(ExactInputSingleParams({pool: pool, ...}));
    // Swap succeeds — allowlist invariant broken
}
```
