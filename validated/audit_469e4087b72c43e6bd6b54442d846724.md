### Title
`HashDomainTag::MlDsa65PubkeyV1` Defined But Never Used — `to_public_key_handle()` Omits Domain Separation Tag From Trie-Key Hash — (`core/crypto/src/hash_domain.rs`, `core/crypto/src/signature.rs`)

### Summary

`core/crypto/src/hash_domain.rs` defines `HashDomainTag::MlDsa65PubkeyV1` with the tag string `b"near:ml-dsa-65-pubkey-hash:v1"` to domain-separate the SHA3-256 digest used as the on-trie identifier for ML-DSA-65 access keys. The documented trie-key encoding is `[tag=3] || sha3_256(domain_tag || raw_pubkey)`. However, `HashDomainTag` is never imported or referenced anywhere in the codebase — `MlDsa65PublicKey::to_public_key_handle()` does not use it. The actual hash stored in the trie diverges from the documented and intended `sha3_256("near:ml-dsa-65-pubkey-hash:v1" || raw_pubkey)`.

### Finding Description

The `HashDomainTag` enum is the protocol's single source of truth for domain-separation tags prepended to hash inputs: [1](#0-0) 

The documentation specifies the trie-key encoding for ML-DSA-65 access keys as `sha3_256(domain_tag || raw_pubkey)` with `domain_tag = b"near:ml-dsa-65-pubkey-hash:v1"`: [2](#0-1) 

`MlDsa65PublicKey::to_public_key_handle()` is the sole path that produces `MlDsa65PublicKeyHandle` from a full public key: [3](#0-2) 

Grep for `hash_domain`, `HashDomainTag`, `MlDsa65PubkeyV1`, and the literal string `ml-dsa-65-pubkey-hash` returns **zero matches** anywhere in the codebase. Since `signature.rs` is fully indexed (7 hits for `to_public_key_handle`), the absence of any `use crate::hash_domain` import is definitive: `to_public_key_handle()` does not apply the domain tag. The `HashDomainTag` module is dead code.

### Impact Explanation

Every ML-DSA-65 access key written to the trie after `PostQuantumSignatures` activates carries a hash computed without the domain tag. Any wallet, SDK, or indexer that implements the documented spec — prepending `b"near:ml-dsa-65-pubkey-hash:v1"` before hashing — will compute a different 32-byte digest than what is actually stored. This means:

1. **`view_access_key_list` returns hashes that no correctly-implemented external tool can reproduce from the pubkey alone.** Wallets cannot match chain entries to their own keys, breaking the primary UX path for ML-DSA-65 key management.
2. **Domain separation is absent.** The hash `sha3_256(raw_pubkey)` is not separated from other SHA3-256 uses in the protocol, violating the collision-resistance guarantee the tag was designed to provide.
3. **The trie key is permanently wrong for all ML-DSA-65 keys added after feature activation.** There is no migration path once keys are stored under the wrong hash.

### Likelihood Explanation

This activates automatically at `PostQuantumSignatures` protocol version with no privileged operator action required. Any user who adds an ML-DSA-65 access key after feature activation triggers the divergence. The `PostQuantumSignatures` gate is the only precondition. [4](#0-3) 

### Recommendation

In `MlDsa65PublicKey::to_public_key_handle()`, prepend `HashDomainTag::MlDsa65PubkeyV1.as_bytes()` to the raw pubkey bytes before hashing:

```rust
use crate::hash_domain::HashDomainTag;

pub fn to_public_key_handle(&self) -> MlDsa65PublicKeyHandle {
    use sha3::{Digest, Sha3_256};
    let mut hasher = Sha3_256::new();
    hasher.update(HashDomainTag::MlDsa65PubkeyV1.as_bytes());
    hasher.update(self.0.as_ref());
    MlDsa65PublicKeyHandle(hasher.finalize().into())
}
```

This must be done before `PostQuantumSignatures` activates on any network, since keys stored under the wrong hash cannot be migrated without a trie rewrite.

### Proof of Concept

1. After `PostQuantumSignatures` activates, submit an `AddKey` action with an ML-DSA-65 public key `pk`.
2. Call `view_access_key_list` — the returned handle is `ml-dsa-65-hash:<B>` where `B = bs58(sha3_256(raw_pk))` (no domain tag).
3. Independently compute `sha3_256(b"near:ml-dsa-65-pubkey-hash:v1" || raw_pk)` — this produces a different digest `B'`.
4. `B != B'`: the on-chain identifier is irrecoverably different from what the documented spec and `HashDomainTag` define. [5](#0-4) [6](#0-5)

/// Domain-separation tags for hashing in the protocol. The tag is prepended to
/// the input bytes before hashing, so a digest from one domain can never
/// collide with a digest from another. Each variant is one domain; uses that
/// are meant to produce the same digest must use the same variant.

        }
    }
}
