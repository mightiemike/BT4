## Vulnerability Analysis

Let me trace through the exact code to verify the mathematical claim.

### Step 1: Trace `merkle_proof_calculator` for a 3-tx block [A, B, C]

**For position 2 (real tx C):** [1](#0-0) 

- `len=3`, odd → push duplicate: `[A, B, C, C]`
- `pos=2`, even → sibling = `current_hashes[3]` = `C`; proof = `[C]`
- new_hashes = `[hash(A,B), hash(C,C)]`; `pos = 1`
- `pos=1`, odd → sibling = `current_hashes[0]` = `hash(A,B)`; proof = `[C, hash(A,B)]`

**For position 3 (phantom tx, does not exist):**
- `len=3`, odd → push duplicate: `[A, B, C, C]`
- `pos=3`, odd → sibling = `current_hashes[2]` = `C`; proof = `[C]`
- new_hashes = `[hash(A,B), hash(C,C)]`; `pos = 1`
- `pos=1`, odd → sibling = `current_hashes[0]` = `hash(A,B)`; proof = `[C, hash(A,B)]`

**Result: `merkle_proof_calculator(txs, 2) == merkle_proof_calculator(txs, 3)` — identical proofs.**

### Step 2: Trace `compute_root_from_merkle_proof(C, 3, [C, hash(A,B)])` [2](#0-1) 

- `pos=3`, odd → `current_hash = hash(C, C)`; `pos=1`
- `pos=1`, odd → `current_hash = hash(hash(A,B), hash(C,C))` = **merkle_root**

The phantom index 3 produces the correct root. The invariant is broken.

### Step 3: Check `verify_transaction_inclusion` for guards [3](#0-2) 

The function checks:
1. Block is in canonical chain (`mainchain_header_to_height` lookup)
2. Sufficient confirmations
3. Merkle proof is non-empty
4. `compute_root_from_merkle_proof(...) == header.block_header.merkle_root`

**There is no check that `tx_index < number_of_transactions_in_block`.** The contract stores only the block header (80 bytes), not the transaction list, so it has no way to enforce this bound.

### Step 4: Access control [4](#0-3) 

The function is `#[pause]` (not paused by default) with no `#[trusted_relayer]` or `#[private]` restriction. Any unprivileged NEAR account can call it.

### Step 5: Does `verify_transaction_inclusion_v2` fix this? [5](#0-4) 

No. The v2 function adds a coinbase proof check at index 0, but then calls `verify_transaction_inclusion` internally. The coinbase check does not constrain the `tx_index` of the target transaction, so the phantom-index attack works identically through v2.

### Title
False Merkle Inclusion Acceptance for Nonexistent Transaction Index via Duplicate-Leaf Padding — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` (and `verify_transaction_inclusion_v2`) accept a Merkle proof for a phantom transaction at index N in an N-transaction block with odd N, because Bitcoin's duplicate-leaf padding causes the proof for the real last transaction (index N-1) to be mathematically identical to a proof for the nonexistent index N.

### Finding Description
In `merkle_proof_calculator`, when the transaction list has an odd length, the last element is duplicated before computing sibling hashes. [1](#0-0) 

For a 3-tx block [A, B, C], both `merkle_proof_calculator(txs, 2)` and `merkle_proof_calculator(txs, 3)` return `[C, hash(A,B)]`. In `compute_root_from_merkle_proof`, position 3 is odd at the first level, so it hashes `(C, C)` — exactly what the tree does for the padded duplicate — and produces the correct root. [6](#0-5) 

`verify_transaction_inclusion` performs no bounds check on `tx_index` against the actual number of transactions in the block. The contract stores only the 80-byte block header, not the transaction list, so no such check is possible without additional data. [7](#0-6) 

### Impact Explanation
Any downstream NEAR contract or off-chain system that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` to gate an action (e.g., token minting, bridge unlock, settlement) can be tricked into accepting a proof for a transaction that does not exist at the claimed index. The attacker only needs a canonical block with an odd transaction count and knowledge of the last real transaction hash.

### Likelihood Explanation
- The attacker needs no special role or key — the function is publicly callable.
- Odd-length transaction blocks are common on Bitcoin mainnet.
- The proof construction is deterministic and requires only public blockchain data.
- The deprecated `verify_transaction_inclusion` is still live and callable; `verify_transaction_inclusion_v2` is equally affected.

### Recommendation
Add an explicit `tx_count` parameter to the proof arguments and require `tx_index < tx_count`. Then verify that `tx_count` is consistent with the proof length (i.e., `proof.len() == ceil(log2(tx_count))`). Alternatively, require the caller to supply the full sibling path in a way that makes the phantom-index case structurally distinguishable (e.g., by requiring the proof to include the actual leaf count and rejecting proofs where `tx_index >= tx_count`).

fn phantom_index_attack() {
    fn h(s: &str) -> H256 { s.parse().unwrap() }
    let txs = vec![
        h("aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa"),
        h("bbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb"),
        h("cccccccccccccccccccccccccccccccccccccccccccccccccccccccccccccccc"),

    let proof_real   = merkle_proof_calculator(txs.clone(), 2); // real last tx
    let proof_phantom = merkle_proof_calculator(txs.clone(), 3); // phantom tx
    assert_eq!(proof_real, proof_phantom); // identical proofs

    let root_real = compute_root_from_merkle_proof(txs[2].clone(), 2, &proof_real);
    let root_phantom = compute_root_from_merkle_proof(txs[2].clone(), 3, &proof_phantom);
    assert_eq!(root_real, root_phantom); // phantom index produces correct root
    // verify_transaction_inclusion with tx_index=3 returns true for a 3-tx block
}

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
