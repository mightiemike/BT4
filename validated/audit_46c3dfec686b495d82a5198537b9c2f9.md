### Title
Storage Deposits Freed by GC Are Permanently Trapped in the Contract — (File: `contract/src/lib.rs`)

---

### Summary
`submit_blocks` requires callers to attach NEAR tokens to cover on-chain storage costs for new block headers. When `run_mainchain_gc` later deletes those headers, the storage is freed and the corresponding NEAR is released back into the contract's own balance. Because the contract has no withdrawal function, no admin drain path, and no per-depositor accounting, every NEAR token ever paid for storage that GC subsequently frees is permanently locked inside the contract.

---

### Finding Description

`submit_blocks` is `#[payable]` and enforces a deposit equal to the **net** storage increase of the current call:

```
diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
required_deposit   = env::storage_byte_cost() * diff_storage_usage;
refund             = amount - required_deposit;   // only excess for this call
``` [1](#0-0) 

The refund is correct for the *current* transaction. However, `run_mainchain_gc` — called automatically inside `submit_blocks` and also callable by anyone externally — deletes old block headers from all three storage maps:

```rust
self.remove_block_header(blockhash);          // removes from headers_pool + mainchain_header_to_height
self.mainchain_height_to_header.remove(&height);
``` [2](#0-1) 

On NEAR Protocol, deleting storage entries releases the previously locked NEAR back into the contract account's free balance. The contract never tracks which depositor paid for which storage slot, and there is no `withdraw`, `storage_withdraw`, or admin-drain function anywhere in the contract source:



The freed NEAR accumulates silently in the contract's balance with no path to recovery.

---

### Impact Explanation

Every NEAR token paid by a relayer for a block header that is subsequently garbage-collected is permanently lost. With the recommended `gc_threshold = 52704` (≈ one year of Bitcoin blocks), GC begins removing headers after the first year and runs continuously thereafter. Each `ExtendedHeader` occupies storage across three `LookupMap` entries (`headers_pool`, `mainchain_height_to_header`, `mainchain_header_to_height`), totalling roughly 200–300 bytes per block. At NEAR's storage cost of ~10 yoctoNEAR/byte, each GC'd block frees ~0.002–0.003 NEAR. Over the operational lifetime of the contract, the trapped balance grows unboundedly and is irrecoverable without a contract upgrade.

---

### Likelihood Explanation

This is a certainty, not a probability. GC is triggered automatically on every `submit_blocks` call once the chain exceeds `gc_threshold` headers. [3](#0-2)  Additionally, `run_mainchain_gc` is a public, permissionless entry point (only gated by the pause flag) that any external account can call with an arbitrary `batch_size`, accelerating the accumulation. [4](#0-3)  No special role, leaked key, or social engineering is required.

---

### Recommendation

Add a privileged withdrawal function (callable by `Role::DAO` or a dedicated treasurer role) that transfers the contract's free balance — i.e., `env::account_balance() - env::storage_byte_cost() * env::storage_usage()` — to a configurable beneficiary address. Alternatively, track freed storage per depositor and issue refunds at GC time, analogous to how the original H-01 fix redirected fees to a configurable admin address instead of the contract itself.

---

### Proof of Concept

1. Relayer calls `submit_blocks` with 100 headers and attaches the required deposit (e.g., 0.2 NEAR). The deposit is accepted; excess is refunded. The 0.2 NEAR is now locked as storage cost.
2. After `gc_threshold` more blocks are submitted, `run_mainchain_gc` removes those 100 headers. NEAR Protocol releases the 0.2 NEAR back into the contract's free balance.
3. The 0.2 NEAR sits in the contract's balance. No function exists to move it out.
4. Any external account calls `run_mainchain_gc(1000)` directly — no role required — accelerating the accumulation of freed deposits. [5](#0-4) 
5. Repeat indefinitely. The trapped balance grows with every GC cycle and is irrecoverable without a contract upgrade.

    #[pause(except(roles(Role::UnrestrictedRunGC)))]
    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
