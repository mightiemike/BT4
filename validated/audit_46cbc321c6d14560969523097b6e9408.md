### Title
`total_stake` Inflated by Already-Kicked-Out Validators Skews `validator_max_kickout_stake_perc` Exemption Threshold — (`File: chain/epoch-manager/src/lib.rs`)

---

### Summary

In `EpochManager::compute_validators_to_reward_and_kickout`, the `total_stake` accumulator includes the stake of every validator in the current epoch's `validators_iter()`, including those already present in `prev_validator_kickout` (validators determined to be kicked out by the previous epoch's computation). This inflated `total_stake` is then used as the denominator to compute `min_keep_stake` inside `compute_exempted_kickout`. Because those same already-kicked-out validators are explicitly excluded from the exemption set (they cannot be added to `exempted_validators`), the algorithm attempts to reach an unachievable threshold, exempting more validators from kickout than `validator_max_kickout_stake_perc` intends.

---

### Finding Description

At the end of epoch T, `collect_blocks_info` retrieves `prev_validator_kickout` from `next_epoch_info.validator_kickout()` — the kickout set that was computed at the end of epoch T-1. [1](#0-0) 

Validators in this set are still present in epoch T's `epoch_info.validators_iter()` (because epoch T's validator set was determined at the end of T-2, before the T-1 kickout was computed). `compute_validators_to_reward_and_kickout` then unconditionally accumulates their stake into `total_stake`: [2](#0-1) 

This inflated `total_stake` is passed directly to `compute_exempted_kickout`: [3](#0-2) 

Inside `compute_exempted_kickout`, `min_keep_stake` is derived from this inflated value: [4](#0-3) 

But the exemption loop explicitly skips validators in `prev_validator_kickout`: [5](#0-4) 

The result is a structural divergence: `min_keep_stake` is computed over the full validator set (including already-kicked-out stake), but `exempted_stake` can only accumulate over the subset that is not already kicked out. The algorithm therefore exempts more validators than `validator_max_kickout_stake_perc` intends, because it is trying to satisfy a threshold that is inflated by stake that can never contribute to the exemption pool.

---

### Impact Explanation

`validator_max_kickout_stake_perc` is the protocol's safety valve against mass validator ejection during network instability. It is active on mainnet at 30% since protocol version 78. [6](#0-5) 

When `prev_validator_kickout` contains validators with significant stake, `min_keep_stake` is inflated proportionally. The exemption loop then over-exempts validators from the current epoch's kickout computation — validators who genuinely underperformed are shielded from ejection. This directly violates the protocol invariant that "at most `validator_max_kickout_stake_perc`% of total active stake may be kicked out in one epoch," because the denominator used to compute that limit is not the active-and-exemptable stake but the full epoch stake including already-removed validators.

Concretely: with 5 validators each staking 1000 and one already in `prev_validator_kickout`, `total_stake = 5000`, `min_keep_stake = 3500` (at `exempt_perc = 70`). Only 4 validators can be exempted (max 4000 stake). The algorithm exempts all 4, kicking out nobody. The correct `total_stake` should be 4000, giving `min_keep_stake = 2800`, requiring only 3 exemptions and allowing 1 kickout.

---

### Likelihood Explanation

`prev_validator_kickout` is non-empty in every epoch where at least one validator was kicked out or unstaked in the previous epoch — a routine occurrence on mainnet. The bug is therefore triggered in every such epoch transition. The magnitude of the distortion scales with the fraction of total stake held by already-kicked-out validators. No privileged action is required; normal epoch progression triggers it deterministically.

---

### Recommendation

Exclude validators present in `prev_validator_kickout` from the `total_stake` accumulation used to compute `min_keep_stake`. Specifically, in the loop at line 445–479 of `chain/epoch-manager/src/lib.rs`, skip adding `v.stake()` to `total_stake` when `prev_validator_kickout.contains_key(account_id)` is true. This ensures `min_keep_stake` is computed over only the stake that is actually eligible for exemption, restoring the intended semantics of `validator_max_kickout_stake_perc`.

---

### Proof of Concept

**Setup**: 5 validators, each with 1000 yoctoNEAR stake. `validator_max_kickout_stake_perc = 30` (mainnet v78+). One validator (`V5`) is in `prev_validator_kickout` (kicked out at end of T-1 for `NotEnoughBlocks`). All 5 are still in epoch T's `validators_iter()`. Validators V1–V4 all underperformed below the kickout threshold in epoch T.

**Current (buggy) behavior**:
- `total_stake = 5000` (includes V5's 1000)
- `exempt_perc = 70`, `min_keep_stake = 3500`
- Exemption loop skips V5; accumulates V1–V4 (4000 stake total)
- `exempted_stake (4000) >= min_keep_stake (3500)` → all 4 remaining validators exempted
- **Result**: zero validators kicked out this epoch despite all underperforming

**Correct behavior**:
- `total_stake = 4000` (excludes V5)
- `min_keep_stake = 2800`
- Exemption loop accumulates top-3 performers (3000 stake)
- `exempted_stake (3000) >= min_keep_stake (2800)` → 3 exempted, 1 kicked out
- **Result**: 1 underperforming validator correctly ejected

The divergent value is the `total_stake` passed to `compute_exempted_kickout` at `chain/epoch-manager/src/lib.rs:513`, which is inflated by the stake of validators in `prev_validator_kickout`. [7](#0-6) [8](#0-7) [4](#0-3) [9](#0-8)
