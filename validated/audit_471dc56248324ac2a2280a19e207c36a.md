### Title
Hardcoded 6-Decimal Assumption in `getSlowModeFee()` Produces Incorrect Slow-Mode Fee for Non-6-Decimal Quote Tokens — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.getSlowModeFee()` hardcodes the assumption that the quote token has exactly 6 decimals. For quote tokens with fewer than 6 decimals the function reverts via Solidity 0.8.x checked-arithmetic underflow, permanently breaking slow-mode submission. For quote tokens with more than 6 decimals (e.g. 18-decimal WETH/DAI) the returned fee is inflated by `10^(decimals−6)`, making slow-mode transactions economically infeasible and eliminating the protocol's censorship-resistance guarantee.

---

### Finding Description

`SLOW_MODE_FEE` is defined as the integer `1000000` with the comment `// $1`, calibrated for a 6-decimal quote token (USDC). [1](#0-0) 

`getSlowModeFee()` then scales this constant by `10**(token.decimals() − 6)`: [2](#0-1) 

This formula embeds two broken assumptions:

1. **`token.decimals() >= 6` is required.** In Solidity 0.8.x, `uint8(d) − 6` reverts with an underflow panic for any `d < 6`. There is no guard enforcing this lower bound.

2. **The fee is expressed in raw token units, not in a price-normalised amount.** For an 18-decimal quote token the multiplier is `10^12`, so the returned fee is `1,000,000 × 10^12 = 10^18` — i.e. exactly 1 full token unit. If the quote token is WETH at $3 000, the slow-mode fee becomes $3 000 instead of $1.

By contrast, every other decimal-sensitive path in the same contract correctly uses `MAX_DECIMALS − decimals` normalisation: [3](#0-2) [4](#0-3) 

`getSlowModeFee()` is the only path that inverts this pattern and introduces a hard dependency on a 6-decimal quote token. The `MAX_DECIMALS = 18` constant exists precisely to avoid such assumptions: [5](#0-4) 

The function is consumed by `EndpointTx.sol` (confirmed by grep: 6 matches for `SLOW_MODE_FEE`/`getSlowModeFee` in that file) as part of `submitSlowModeTransactionImpl`, which is the implementation target of the public `submitSlowModeTransaction` entry point: [6](#0-5) 

---

### Impact Explanation

**Quote token with < 6 decimals (e.g. a 2-decimal stablecoin):**
`getSlowModeFee()` always reverts. Every call to `submitSlowModeTransaction` propagates this revert, permanently disabling the slow-mode path. Users have no on-chain mechanism to bypass a censoring sequencer.

**Quote token with > 6 decimals (e.g. 18-decimal WETH or DAI):**
The fee is `10^(decimals−6)` times larger than intended. For WETH the slow-mode fee is 1 WETH (≈ $3 000+). Ordinary users cannot afford to submit slow-mode transactions, achieving the same effective censorship-resistance failure without a revert. The corrupted state delta is the fee amount charged to every user who attempts to use the slow-mode path.

**Impact:** High — the censorship-resistance guarantee of the protocol is broken for any deployment that does not use a 6-decimal quote token.

---

### Likelihood Explanation

**Low.** The quote token is set once at deployment by the owner via `Clearinghouse.initialize(_quote)`. The protocol targets Ink Chain and is expected to use USDC (6 decimals) in production. However, the contract imposes no on-chain enforcement of a 6-decimal quote token, and the `initialize` function accepts any address. A deployment with WETH, DAI, or any non-6-decimal token — whether intentional or accidental — silently activates the bug. The analog to the external report is exact: the contract is designed to accept any ERC20 as the quote token, but the implementation assumes 6 decimals.

---

### Recommendation

Replace the inverted `decimals − 6` formula with the same `MAX_DECIMALS − decimals` normalisation used everywhere else in the contract:

```solidity
function getSlowModeFee() external view returns (uint256) {
    ISpotEngine spotEngine = _spotEngine();
    IERC20Base token = IERC20Base(
        spotEngine.getConfig(QUOTE_PRODUCT_ID).token
    );
    uint8 decimals = token.decimals();
    require(decimals <= MAX_DECIMALS);
    // SLOW_MODE_FEE is in 18-decimal internal representation ($1 = 1e18 units)
    // Convert to native token decimals by dividing out the normalisation factor
    int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
    return uint256(int256(SLOW_MODE_FEE) / multiplier);
}
```

This mirrors the pattern already used in `depositCollateral` and `withdrawCollateral` and removes the implicit 6-decimal dependency.

### Proof of Concept

**Scenario A — 18-decimal quote token (WETH):**

1. Deploy Nado with WETH (18 decimals) as the quote token.
2. Call `clearinghouse.getSlowModeFee()`.
   - `multiplier = 10**(18 − 6) = 10^12`
   - Return value = `1,000,000 × 10^12 = 10^18` (1 WETH ≈ $3 000+).
3. Any user calling `submitSlowModeTransaction` must pay 1 WETH as the slow-mode fee.
4. The fee is ~3 000× the intended $1, making slow-mode economically inaccessible and eliminating censorship resistance.

**Scenario B — sub-6-decimal quote token (e.g. 2-decimal token):**

1. Deploy Nado with a 2-decimal quote token.
2. Call `clearinghouse.getSlowModeFee()`.
   - `uint8(2) − 6` underflows → Solidity 0.8.x panic revert.
3. `submitSlowModeTransaction` always reverts.
4. Slow-mode is permanently broken; users cannot bypass a censoring sequencer.
