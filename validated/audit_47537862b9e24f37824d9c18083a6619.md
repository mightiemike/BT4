### Title
Upgrade Halt via `ErrInsufficientBonusPool` in v8 Migration's `ForceFullExitWithDelegation` — (`x/tieredrewards/keeper/force_exit.go`)

### Summary

The v8 upgrade migration calls `ForceFullExitWithDelegation` for every vesting-owned tier position. That function unconditionally calls `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance`. If the rewards pool balance is insufficient to cover accrued bonus for any such position, `ErrInsufficientBonusPool` propagates all the way up through the upgrade handler, halting the chain at the upgrade height with no recovery path.

### Finding Description

**Call chain (all errors propagate without suppression):**

`RunMigrations` → `Migrate` → `exitVestedAccountsPositions` → `ForceFullExitWithDelegation` → `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` → returns `ErrInsufficientBonusPool`

**Step 3 — `exitVestedAccountsPositions` calls `ForceFullExitWithDelegation` and propagates error:** [3](#0-2) 

**Step 4 — `ForceFullExitWithDelegation` calls `claimRewards` and propagates error:** [4](#0-3) 

**Step 5 — `claimRewards` calls `processEventsAndClaimBonus` and propagates error:** [5](#0-4)

**The contrast with the slash hook is decisive.** `slashRedelegationPosition` explicitly catches `ErrInsufficientBonusPool` and swallows it with a log message, specifically to avoid chain halt: [8](#0-7) 

`ForceFullExitWithDelegation` has no equivalent guard. The ADR-006 security invariants section explicitly documents this asymmetry — the slash hook forfeits bonus silently to avoid chain halt, but the migration path has no such protection: [9](#0-8) 

### Impact Explanation

If any vesting-owned position has accrued bonus rewards exceeding the current `rewards_pool` balance at upgrade height, the upgrade handler returns an error. In Cosmos SDK, a failed upgrade handler causes the node to panic and halt. Since all validators hit the same state at the same height, the entire chain halts. Some positions may have already been force-exited (those processed before the failing one), leaving the chain in a partially-migrated state with no in-protocol recovery path.

### Likelihood Explanation

The preconditions are realistic:
- Pre-v8 code allowed vesting accounts to create tier positions (the restriction `ErrVestingAccountNotAllowed` is a v8 addition). The migration exists precisely because such positions exist on-chain.
- The rewards pool can be legitimately drained by normal user claims before the upgrade block.
- The BeginBlocker only tops up base rewards from the fee collector; it does not replenish the bonus pool.
- The bonus pool is a separate module account (`RewardsPoolName`). Its balance at upgrade time is not guaranteed to cover all accrued bonus for all vesting-owned positions.

### Recommendation

Mirror the pattern used in `slashRedelegationPosition`: catch `ErrInsufficientBonusPool` inside `ForceFullExitWithDelegation` (or inside `exitVestedAccountsPositions`) and skip bonus payment with a log warning rather than returning the error. The delegation transfer and position deletion must still complete. Alternatively, skip `claimRewards` entirely during migration and only perform the delegation transfer and position deletion, since the migration's purpose is structural (fix vesting tracking), not reward settlement.

```go
func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
```

```go
// TestProcessEvents_InsufficientPool_Error verifies that claiming without

func (s *KeeperSuite) TestProcessEvents_InsufficientPool_Error() {
	lockAmount := sdkmath.NewInt(sdk.DefaultPowerReduction.Int64())
	pos := s.setupNewTierPosition(lockAmount, false)

	// Advance time so bonus would be non-zero. Do NOT fund the pool.
	s.ctx = s.ctx.WithBlockTime(s.ctx.BlockTime().Add(30 * 24 * time.Hour))

	_, err := s.keeper.ProcessEventsAndClaimBonus(s.ctx, &pos)
	s.Require().Error(err, "should fail when bonus pool is insufficient")
	s.Require().ErrorContains(err, "insufficient bonus pool",
		"error should mention insufficient bonus pool")
```
