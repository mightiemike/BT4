### Title
Precommit Vote Signature Digest Omits Round and Height, Enabling Cross-Round Replay — (`crates/apollo_signature_manager/src/signature_manager.rs`)

### Summary

`build_precommit_vote_message_digest` constructs the signed digest as `blake2s(b"PRECOMMIT_VOTE" || block_hash_bytes)`. Neither the consensus round, the block height, nor the voter address is included. A signature produced by validator V for block_hash B at round R is cryptographically identical to a valid signature for block_hash B at any other round R' or height H'. An attacker who observes a precommit signature on the network can replay it verbatim at a different round (or height) for the same block hash, and `verify_precommit_vote_signature` will return `true`.

### Finding Description

`sign_precommit_vote` accepts only `block_hash: BlockHash`: [1](#0-0) 

`build_precommit_vote_message_digest` builds the preimage as:

```
PRECOMMIT_VOTE  ||  block_hash.to_bytes_be()
``` [2](#0-1) 

`verify_precommit_vote_signature` reconstructs the same digest from `block_hash` alone and verifies against it: [3](#0-2) 

The `Vote` wire struct carries `height`, `round`, `proposal_commitment`, `voter`, and `signature`: [4](#0-3) 

Of those five fields, only `proposal_commitment` (the block hash) is covered by the signature. `round`, `height`, and `voter` are unsigned and can be freely mutated by any network peer after the signature is produced.

### Impact Explanation

The allowed impact "High — signature/hash logic binds the wrong signer, hash, type, or executable payload" is directly satisfied. The concrete divergent value is the `round` field of a `Vote` message: a signature produced for `(block_hash=B, round=R)` is indistinguishable from a signature for `(block_hash=B, round=R')`. An attacker who collects a quorum of precommit signatures from round R can re-broadcast those same `Vote` messages with the `round` field changed to R', and every call to `verify_precommit_vote_signature` will return `true` for each one. This allows constructing a spurious quorum at an arbitrary round for any block hash that was previously committed, potentially driving the consensus state machine to a decision at the wrong round or replaying a commit across heights.

### Likelihood Explanation

The attack requires only passive observation of consensus gossip (precommit votes are broadcast to all peers) and the ability to re-send `Vote` messages with a modified `round` field. No validator key material, operator access, or privileged position is needed. The precondition — the same block hash appearing at two rounds — is a normal occurrence whenever a round times out and the same block is re-proposed.

### Recommendation

Include `height`, `round`, and `voter` (contract address) in the preimage before hashing:

```rust
fn build_precommit_vote_message_digest(
    block_hash: BlockHash,
    height: BlockNumber,
    round: Round,
    voter: ContractAddress,
) -> MessageDigest {
    let mut message = Vec::new();
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&block_hash.to_bytes_be());
    message.extend_from_slice(&height.0.to_be_bytes());
    message.extend_from_slice(&round.to_be_bytes());
    message.extend_from_slice(voter.0.key().to_bytes_be().as_ref());
    MessageDigest(blake2s_to_felt(&message))
}
```

`sign_precommit_vote` and `verify_precommit_vote_signature` must be updated to accept and pass these fields. The existing TODO comment at line 122–124 already flags a related delimiter-ambiguity concern: [5](#0-4) 

### Proof of Concept

```rust
#[tokio::test]
async fn test_cross_round_replay() {
    let key_store = LocalKeyStore::new_for_testing();
    let sm = SignatureManager::new(key_store);

    let block_hash = BlockHash(felt!("0xdeadbeef"));

    // Validator signs a precommit at round 0.
    let sig_round_0 = sm.sign_precommit_vote(block_hash).await.unwrap();

    // Attacker replays the same signature claiming it is for round 1.
    // verify_precommit_vote_signature has no round parameter, so it
    // reconstructs the same digest and returns true.
    let result = verify_precommit_vote_signature(
        block_hash,
        sig_round_0,
        key_store.public_key,
    ).unwrap();

    // This assertion PASSES, demonstrating the replay succeeds.
    assert!(result, "cross-round replay accepted — round is not in the digest");
}
```

The test passes because `verify_precommit_vote_signature` accepts only `block_hash` and has no way to distinguish a round-0 signature from a round-1 signature for the same block hash. [2](#0-1)
