### Title
`only_query` Hardcoded to `false` in `BroadcastedTransaction` → `ExecutableTransactionInput` Conversion, Causing Wrong Version Reported During Fee Estimation and Simulation - (`File: crates/apollo_rpc/src/v0_8/api/mod.rs`)

### Summary

The `TryFrom<BroadcastedTransaction> for ExecutableTransactionInput` conversion in the RPC layer hardcodes `only_query = false` for `Invoke` and `DeployAccount` transactions. This conversion is used by both `starknet_estimateFee` and `starknet_simulateTransactions`. Per the Starknet protocol, simulation and fee-estimation calls must set bit 128 of the transaction version (the "query bit") so that contracts can distinguish a dry-run from a real execution. Because `only_query` is never set to `true`, the version reported to the contract via the `get_execution_info` syscall is the real execution version (e.g., `0x3`) instead of the query version (e.g., `2^128 + 3`). Any contract that branches on the query bit will execute the wrong code path, producing an authoritative-looking but incorrect simulation trace and fee estimate.

### Finding Description

**Root cause — conversion boundary** [1](#0-0) 

The `only_query` field is hardcoded to `false` for both `Invoke` and `DeployAccount` variants. The developer-left TODO at line 332 explicitly acknowledges the omission: `// TODO(yair): pass the right value for only_query field.`

**How `only_query` propagates to the version seen by the contract**

`only_query` is stored in `ExecutionFlags` and passed to `BlockifierTransaction::from_api`: [2](#0-1) 

Inside the blockifier, `tx_version_for_get_execution_info` calls `signed_version`, which sets bit 128 when `only_query` is `true`: [3](#0-2) [4](#0-3) 

**Call path from the public RPC endpoints**

Both `estimate_fee` and `simulate_transactions` convert the caller-supplied `BroadcastedTransaction` list through the broken `TryFrom` impl before dispatching to the blockifier: [5](#0-4) [6](#0-5) 

**The `ExecutableTransactionInput` type carries `OnlyQuery` as a tuple field** [7](#0-6) 

### Impact Explanation

Any contract whose `__validate__` or `__execute__` function reads `tx.version` via `get_execution_info` and checks whether bit 128 is set (the standard way to detect a simulation) will observe version `0x3` instead of `2^128 + 3`. This causes:

1. **Wrong simulation trace** — the contract executes the "real execution" branch instead of the "query" branch, producing an incorrect `starknet_simulateTransactions` response.
2. **Wrong fee estimate** — `starknet_estimateFee` returns a gas figure derived from the wrong code path, which can be either an over- or under-estimate.

Both results are returned as authoritative RPC responses with no indication that the query bit was missing.

### Likelihood Explanation

Any unprivileged caller can trigger this by submitting any `Invoke` or `DeployAccount` transaction to `starknet_estimateFee` or `starknet_simulateTransactions`. No special privileges, malformed bytes, or peer-level access are required. The affected endpoints are part of the standard public JSON-RPC surface.

### Recommendation

In `TryFrom<BroadcastedTransaction> for ExecutableTransactionInput`, set `only_query = true` for the `Invoke` and `DeployAccount` arms (and for `Declare` variants once the same TODO is resolved there). The correct value is `true` because these endpoints are explicitly dry-run paths:

```rust
BroadcastedTransaction::DeployAccount(tx) => Ok(Self::DeployAccount(tx.into(), true)),
BroadcastedTransaction::Invoke(tx) => Ok(Self::Invoke(tx.into(), true)),
```

### Proof of Concept

1. Deploy a contract whose `__execute__` checks `get_execution_info().tx_info.version` and branches on whether bit 128 is set (query path vs. execution path), with different gas costs on each branch.
2. Call `starknet_estimateFee` with an `Invoke` transaction targeting that contract.
3. Observe that the returned fee corresponds to the execution-path gas cost, not the query-path gas cost — confirming that `only_query` was `false` and the wrong version was reported to the contract.

The existing test `simulate_with_query_bit_outputs_same_as_no_query_bit` in `crates/apollo_rpc_execution/src/execution_test.rs` (line 750) only covers a deprecated V1 contract that does not inspect the query bit, so it does not catch this divergence for V3 contracts that do. pub fn signed_tx_version(
    tx_version: &TransactionVersion,
    transaction_options: &TransactionOptions,
) -> TransactionVersion {
    // If only_query is true, set the 128-th bit.
    let query_only_bit = *QUERY_VERSION_BASE;
    assert_eq!(
        tx_version.0.to_biguint() & query_only_bit.to_biguint(),
        BigUint::from(0_u8),
        "Requested signed tx version with version that already has query bit set: {tx_version:?}."
    );
    if transaction_options.only_query {
        TransactionVersion(tx_version.0 + query_only_bit)
    } else {
        *tx_version
    }
}
