Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates on router address instead of originating user, allowing full allowlist bypass for router-mediated swaps â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` at the time `pool.swap()` is called. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` is the router contract, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. Any pool admin who allowlists the router to enable router access for legitimate users simultaneously grants swap access to every address that calls through the same router, completely defeating the allowlist.

## Finding Description

**Root cause â€” `MetricOmmPool.swap` passes `msg.sender` as `sender`:**

`MetricOmmPool.sol` L230â€“231 passes `msg.sender` directly as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
  msg.sender,   // router address when called via router
  recipient,
  ...
);
``` [1](#0-0) 

**`ExtensionCalling._beforeSwap` forwards `sender` unchanged:**

`ExtensionCalling.sol` L149â€“176 encodes and dispatches `sender` to every configured extension without modification:

```solidity
abi.encodeCall(
  IMetricOmmExtensions.beforeSwap,
  (sender, recipient, ...)   // sender = router
)
``` [2](#0-1) 

**`SwapAllowlistExtension.beforeSwap` checks the wrong identity:**

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool (correct) and `sender` is the router (wrong). The resolved check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**`MetricOmmSimpleRouter.exactInputSingle` does not forward the originating user:**

```solidity
IMetricOmmPoolActions(params.pool).swap(
  params.recipient,
  params.zeroForOne,
  ...
  params.extensionData
);
``` [4](#0-3) 

`msg.sender` (the actual user) is stored only in the callback context for payment purposes and is never surfaced to the pool or extensions as the swap initiator.

**Contrast with `DepositAllowlistExtension` (correct design):**

`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` â€” the position owner explicitly passed by the caller â€” which correctly identifies the economic actor regardless of intermediaries:

```solidity
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
``` [5](#0-4) 

The swap path has no equivalent mechanism to carry the originating user's identity through the router.

**Full exploit call chain:**

```

  â†’ MetricOmmSimpleRouter.exactInputSingle()
      â†’ pool.swap(recipient=bob, ...)   [msg.sender = router]
          â†’ _beforeSwap(sender=router, ...)
              â†’ SwapAllowlistExtension.beforeSwap(sender=router)
                  â†’ allowedSwapper[pool][router] == true  âś“  (bypass)

## Impact Explanation

A pool admin deploying a curated pool with `SwapAllowlistExtension` intends to restrict trading to a known set of addresses. Once the admin allowlists the router (the natural step to let legitimate users trade via the canonical entrypoint), `allowedSwapper[pool][router] == true`, and the guard passes for every caller who routes through it. Non-allowlisted users can trade on a pool designed to be curated, causing direct loss of LP assets and breaking the pool's intended access control policy. This constitutes broken core pool functionality and an admin-boundary break reachable by any unprivileged user.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the canonical, publicly documented swap entrypoint. Any pool admin who wants allowlisted users to trade via the router must allowlist the router address â€” this is the expected operational path, not an edge case. The bypass is therefore reachable by any user on any curated pool whose admin has taken the natural step of enabling router access. No special privileges or unusual conditions are required beyond calling the public router function.

## Recommendation

The extension must gate on the originating user, not the intermediary. The cleanest fix matching the deposit-side design:

1. **Protocol-level `originalSender` field:** Add a dedicated `originalSender` field to the `beforeSwap` extension interface. The router passes `msg.sender` (the actual user) as `originalSender` in `extensionData` or a new parameter, and `SwapAllowlistExtension` reads and checks it instead of `sender`.

2. **Router-level enforcement:** Require the router to pass `msg.sender` as a verified field in `extensionData`, and have `SwapAllowlistExtension` decode and check it â€” analogous to how `addLiquidity` passes `owner` directly as the identity to gate on.

3. **Minimal interim fix:** Gate on `recipient` and require `recipient == msg.sender` at the router level, though this is weaker since `recipient` is caller-controlled.
