Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Original User, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the immediate `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` becomes the router's address rather than the original user. A pool admin who allowlists the router to support normal periphery flows inadvertently grants every user the ability to bypass the per-user allowlist gate entirely.

## Finding Description

`SwapAllowlistExtension.beforeSwap` performs the check at line 37:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (used as the namespace key) and `sender` is the first argument forwarded by the pool — which is `msg.sender` of the pool's own `swap()` call, as seen in `MetricOmmPool.sol` lines 230–231:

```solidity
_beforeSwap(
    msg.sender,   // ← immediate caller of pool.swap(), not the original user
    ...
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly (lines 72–80), making the router the pool's `msg.sender`. The extension therefore evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][originalUser]`. The same substitution occurs in `exactInput` (line 104) and `exactOutputSingle` (line 136). In the multi-hop `exactOutput` path, intermediate hops are called from within `_exactOutputIterateCallback` (line 220), where `msg.sender` is the pool of the prior hop — also not the original user.

This creates an irreconcilable dilemma for the pool admin:
- **Do not allowlist the router**: Router-mediated swaps fail for all users, including allowlisted ones.
- **Allowlist the router**: Every user — including those not on the allowlist — can bypass the gate via the router.

There is no configuration that simultaneously supports the router and enforces per-user restrictions.

## Impact Explanation

Complete bypass of the swap allowlist on curated pools. Any user can execute swaps on a pool intended to be restricted to specific addresses by routing through `MetricOmmSimpleRouter`. This is a broken core pool functionality and a direct admin-boundary break: the pool admin's access-control policy is silently nullified by a supported periphery path. LP funds in the pool are exposed to trades from actors the pool admin explicitly intended to exclude.

## Likelihood Explanation

High. `MetricOmmSimpleRouter` is the standard periphery entry point for swaps. A pool admin who deploys a `SwapAllowlistExtension` and wants to support normal user flows through the router must allowlist the router address — this is the natural and expected configuration. Any user who knows the router address (a public, deployed contract) can exploit this immediately without any privileged access or special setup.

## Recommendation

The allowlist must gate the original economic actor, not the immediate pool caller. The most robust fix is to have the router encode `msg.sender` into `extensionData` for each hop, and have the extension decode and check it against the allowlist. This requires a coordinated change to the router and extension. Alternatively, define a standard interface where the router attests the original user in a verifiable way (e.g., via a transient-storage slot the pool reads), and the extension verifies that attestation. Using `tx.origin` is generally fragile and should be avoided.

## Proof of Concept

```
1. Pool admin deploys pool with SwapAllowlistExtension.
2. Admin allowlists userA (intended trader):
       setAllowedToSwap(pool, userA, true)
3. Admin also allowlists the router to support normal periphery flows:
       setAllowedToSwap(pool, router, true)
4. userB (NOT on the allowlist) calls:
       router.exactInputSingle({pool: pool, recipient: userB, ...})
5. Router calls pool.swap(...) — pool sees msg.sender = router.
6. Extension evaluates: allowedSwapper[pool][router] → true → PASSES.
7. userB's swap executes successfully on the restricted pool.
```
