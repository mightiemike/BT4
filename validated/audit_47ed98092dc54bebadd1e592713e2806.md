### Title
Phantom-Index Merkle Proof Forgery via Odd-Level Duplication Asymmetry — (`merkle-tools/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` has no knowledge of the actual tree size and performs no bounds check on `transaction_position`. Because `merkle_proof_calculator` duplicates the last leaf at **every** odd-length level (not just the root), the proof generated for the last real leaf (index `n-1`) is mathematically identical to the proof that would be generated for the phantom index `n`. An attacker can therefore call `verify_transaction_inclusion` (or `verify_transaction_inclusion_v2`) with a real `tx_id`, a phantom `tx_index = n`, and the proof for index `n-1`, and the function returns `true`.

---

### Finding Description

**Root cause — `merkle_proof_calculator`** duplicates the last element at every odd-length level: [1](#0-0) 

This means for a 5-tx tree `[A, B, C, D, E]`:

| Level | Nodes before padding | After padding | Proof element for idx 4 | Proof element for idx 5 |
|---|---|---|---|---|
| 0 | `[A,B,C,D,E]` | `[A,B,C,D,E,E]` | `E` (sibling at 5) | `E` (sibling at 4) |
| 1 | `[AB,CD,EE]` | `[AB,CD,EE,EE]` | `EE` (sibling at 3) | `EE` (sibling at 3) |
| 2 | `[ABCD, EEEE]` | (even, no pad) | `ABCD` (sibling at 0) | `ABCD` (sibling at 0) |

Both index 4 and phantom index 5 produce the **identical proof**: `[E, hash(E,E), hash(hash(A,B),hash(C,D))]`.

**Root cause — `compute_root_from_merkle_proof`** only uses `position % 2` to decide left/right at each level, with no bounds check: [2](#0-1) 

Tracing `compute_root_from_merkle_proof(E, 5, [E, hash(E,E), ABCD])`:
- Step 1: `5 % 2 == 1` → `hash(E, E)`, position → 2
- Step 2: `2 % 2 == 0` → `hash(hash(E,E), hash(E,E))`, position → 1
- Step 3: `1 % 2 == 1` → `hash(ABCD, hash(hash(E,E),hash(E,E)))` = **root** ✓

Tracing `compute_root_from_merkle_proof(E, 4, [E, hash(E,E), ABCD])`:
- Step 1: `4 % 2 == 0` → `hash(E, E)`, position → 2
- Step 2: `2 % 2 == 0` → `hash(hash(E,E), hash(E,E))`, position → 1
- Step 3: `1 % 2 == 1` → `hash(ABCD, hash(hash(E,E),hash(E,E)))` = **root** ✓

Both produce the canonical merkle root. The contract's verification path passes the result directly to equality with `header.block_header.merkle_root`: [3](#0-2) 

**`verify_transaction_inclusion_v2` does not fix this.** Its only additional guard is:
1. `merkle_proof.len() == coinbase_merkle_proof.len()` — both proofs have depth 3 for a 5-tx tree, so this passes.
2. Coinbase proof at index 0 must verify to the root — the attacker provides a legitimate coinbase proof of the same length 3, which passes. [4](#0-3) 

After both checks pass, v2 delegates to `verify_transaction_inclusion`, which returns `true` for the phantom index.

---

### Impact Explanation

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` return `true` for a claim that transaction `E` is at index `n` (a phantom position that does not exist in the block). The stated contract of the function — "True if `tx_id` is at the **claimed position** in the block" — is violated. Any downstream consumer that relies on the `(tx_id, tx_index)` pair being jointly valid (e.g., UTXO identification, ordered-transaction logic, cross-chain bridge settlement that keys on tx position) can be deceived. The attacker does not need to forge a nonexistent transaction; they only need to shift a real transaction's claimed index by 1 to a phantom slot, which the verifier accepts as valid.

---

### Likelihood Explanation

Both entry points are public, permissionless NEAR view/call methods. No privileged role, relayer compromise, or key leak is required. The attacker only needs to know the block's transaction list (publicly available on-chain Bitcoin data) and construct the proof for the last real leaf. The attack is deterministic and requires no brute force. It applies to any block where any subtree at any level has an odd number of children — a common occurrence in real Bitcoin blocks.

---

### Recommendation

`compute_root_from_merkle_proof` must validate that `transaction_position < 2^(merkle_proof.len())` (i.e., the position is within the bounds of the tree implied by the proof depth). More precisely, the verifier should reject any `tx_index >= num_transactions`, but since the contract does not store the transaction count, the minimum fix is to reject positions that exceed the maximum valid index for a tree of the given proof depth. Alternatively, encode the tree size in the proof or require the caller to supply it and validate it against the stored block header.

### Proof of Concept

```rust
#[test]
fn test_phantom_index_forgery_5tx() {
    // 5-tx tree: [A, B, C, D, E]
    let a = H256([1u8; 32]);
    let b = H256([2u8; 32]);
    let c = H256([3u8; 32]);
    let d = H256([4u8; 32]);
    let e = H256([5u8; 32]);

    let txs = vec![a.clone(), b.clone(), c.clone(), d.clone(), e.clone()];

    // Compute proof for the real last tx (index 4)
    let proof_for_4 = merkle_proof_calculator(txs.clone(), 4);

    // Compute the canonical root
    let root_via_4 = compute_root_from_merkle_proof(e.clone(), 4, &proof_for_4);

    // Now use the SAME proof with phantom index 5
    let root_via_5 = compute_root_from_merkle_proof(e.clone(), 5, &proof_for_4);

    // Both produce the same root — phantom index 5 passes verification
    assert_eq!(root_via_4, root_via_5,
        "Phantom index 5 produces the same root as real index 4");

    // Confirm the proof for index 5 is also identical to the proof for index 4
    let proof_for_5 = merkle_proof_calculator(txs.clone(), 5);
    assert_eq!(proof_for_4, proof_for_5,
        "Proofs for index 4 and phantom index 5 are identical");
}
```

This test passes on the unmodified codebase, confirming that `verify_transaction_inclusion` returns `true` for `tx_id = E, tx_index = 5` using the proof for `tx_index = 4`, violating the inclusion invariant.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
