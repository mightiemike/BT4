### Title
Duplicate Position ID in `MsgClaimTierRewards` Enables Double-Claim of Bonus Rewards — (`File: x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`ClaimTierRewards` loads all requested positions into an in-memory slice before processing any of them. If the same position ID appears more than once in `msg.PositionIds`, the stale pre-update copy is processed a second time, causing the current-segment bonus to be computed and paid twice from the `RewardsPoolName` module account.

---

### Finding Description

In `ClaimTierRewards` (`x/tieredrewards/keeper/msg_server.go`), all positions are fetched from state and appended to a slice before any reward processing begins: [1](#0-0) 

The slice is then passed to `claimRewardsAndUpdatesPositions`, which iterates over it and, for each entry, calls `processEventsAndClaimBonus` and then `setPosition` to persist the updated checkpoints: [2](#0-1) 

Inside `processEventsAndClaimBonus`, the current-segment bonus is computed using `pos.LastBonusAccrual` as the segment start time, and `applyBonusAccrualCheckpoint` advances it to `blockTime` only on the **in-memory copy**: [3](#0-2) 

When the same position ID `X` appears twice in `msg.PositionIds`:

1. Both `positions[0]` and `positions[1]` are loaded from state with the same stale `LastBonusAccrual = T0` before any processing.
2. First iteration: bonus for segment `[T0, blockTime]` is computed and paid; `setPosition` writes `LastBonusAccrual = blockTime` to the store.
3. Second iteration: `positions[1]` still carries `LastBonusAccrual = T0` (loaded before the first iteration ran). `processEventsAndClaimBonus` recomputes the identical segment `[T0, blockTime]` and pays the same bonus again.
4. `setPosition` is called a second time, silently overwriting the store with the same final value, leaving no trace of the double payment.

The `sufficientBonusPoolBalance` guard only prevents over-drawing the pool in a single call; it does not prevent paying the same entitlement twice within one transaction: [4](#0-3) 

---

### Impact Explanation

The attacker receives `2×` (or `N×` for `N` duplicates) the bonus rewards they are entitled to. Each duplicate drains `bonusCoins` from the `RewardsPoolName` module account and transfers them to the position owner. The corrupted value is the `RewardsPoolName` module-account balance and the per-position `LastBonusAccrual` checkpoint.

---

### Likelihood Explanation

The entry path is a standard, unprivileged `MsgClaimTierRewards` transaction signed by the position owner. No special role, leaked key, or social engineering is required. The attacker only needs to own at least one position with accrued bonus rewards and submit a transaction with the position ID repeated in `PositionIds`.

---

### Recommendation

Deduplicate `msg.PositionIds` before loading positions, or re-fetch each position from state immediately before processing it (instead of using the pre-loaded slice). The simplest fix is to add a duplicate-ID check inside `msg.Validate()` or at the top of `ClaimTierRewards`:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, id := range msg.PositionIds {
    if _, ok := seen[id]; ok {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest, "duplicate position id: %d", id)
    }
    seen[id] = struct{}{}
}
```

---

### Proof of Concept

1. Owner creates a position (ID = `42`) and accrues bonus rewards over several blocks.
2. Owner submits `MsgClaimTierRewards{Owner: owner, PositionIds: [42, 42]}`.
3. `ClaimTierRewards` loads `pos42` from state twice into `positions[0]` and `positions[1]`, both with `LastBonusAccrual = T0`.
4. First loop iteration: `processEventsAndClaimBonus` computes segment `[T0, blockTime]`, sends `bonusCoins` to owner, calls `setPosition` (store now has `LastBonusAccrual = blockTime`).
5. Second loop iteration: `positions[1].LastBonusAccrual` is still `T0`; `processEventsAndClaimBonus` recomputes the same segment and sends another `bonusCoins` to owner.
6. Owner's balance increases by `2 × bonusCoins`; `RewardsPoolName` is drained by twice the legitimate amount.

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}
