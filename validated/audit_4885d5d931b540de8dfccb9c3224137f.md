### Title
Unauthorized Slow Mode Transaction Execution via Missing Access Control on `processSlowModeTransactionImpl` â€” (File: `core/contracts/EndpointTx.sol`)

---

### Summary

`processSlowModeTransactionImpl` in `EndpointTx.sol` is declared `public` with no access control modifier. It is intended to be invoked only by the Endpoint contract as part of slow mode queue processing. Because Endpoint uses `delegatecall` to `EndpointTx` (the standard upgradeable proxy pattern described in the wiki), any external caller can invoke `processSlowModeTransactionImpl` directly on the Endpoint contract, bypassing the slow mode queue entirely and executing privileged transactions with an arbitrary `sender` parameter.

---

### Finding Description

The function at `EndpointTx.sol` line 203:

```solidity
function processSlowModeTransactionImpl(
    address sender,
    bytes calldata transaction
) public {
```

accepts a caller-controlled `sender` address and a raw transaction payload. It processes multiple transaction types including `LinkSigner`, `WithdrawCollateral`, `DepositInsurance`, `DelistProduct`, `DumpFees`, and `UpdateTierFeeRates`. [1](#0-0) 

For the `LinkSigner` branch (slow mode path):

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [2](#0-1) 

`validateSender` checks that `txn.sender` (the subaccount bytes32) is derived from `sender` (the address parameter passed by the caller). It does **not** check that `msg.sender` is the Endpoint, and it does **not** check that `sender` is the actual submitter of a queued slow mode transaction. An attacker can pass any victim's address as `sender`, craft a `LinkSigner` payload pointing to the victim's subaccount, and set themselves as the linked signer â€” all without any queued slow mode entry or signature from the victim.

The `WithdrawCollateral` branch has the same structural flaw:

```solidity
validateSender(txn.sender, sender);
clearinghouse.withdrawCollateral(
    txn.sender,
    txn.productId,
    txn.amount,
    address(0),
    nSubmissions
);
``` [3](#0-2) 

Transaction types `DepositInsurance`, `DelistProduct`, `DumpFees`, and `UpdateTierFeeRates` have no `validateSender` check at all, making them unconditionally executable by any caller. [4](#0-3) 

The root cause is identical in class to the Comet `set_freeze_status` bug: a function intended to be callable only by a designated privileged role (the Endpoint contract) carries no enforcement of that restriction.

---

### Impact Explanation

A linked signer has full authority to sign transactions on behalf of the subaccount, including order placement, cancellation, and collateral withdrawal. By calling `processSlowModeTransactionImpl` with a victim's address as `sender` and a crafted `LinkSigner` payload, an attacker sets themselves as the linked signer for the victim's subaccount. From that point the attacker can drain the victim's collateral via signed `WithdrawCollateral` transactions submitted through the normal sequencer path. The corrupted state is `linkedSigners[victim_subaccount]` in Endpoint's storage.

---

### Likelihood Explanation

The function is `public` with zero preconditions beyond the victim having an existing subaccount (`requireSubaccount`). Any on-chain address can execute the attack in a single transaction. No privileged key, leaked secret, or governance capture is required.

---

### Recommendation

Change `processSlowModeTransactionImpl` visibility from `public` to `internal`, or add an explicit caller check:

```solidity
require(msg.sender == address(this), "only endpoint");
```

This mirrors the fix applied to the Comet `set_freeze_status` bug: enforce that only the designated privileged caller can invoke the function.

---

### Proof of Concept

1. Victim holds collateral in subaccount `victim_sub` (bytes32 whose first 20 bytes equal `victim_addr`).
2. Attacker constructs `transaction = abi.encodePacked(uint8(TransactionType.LinkSigner), abi.encode(LinkSigner({sender: victim_sub, signer: bytes32(uint256(uint160(attacker_addr)))})))`.
3. Attacker calls `Endpoint.processSlowModeTransactionImpl(victim_addr, transaction)`.
4. Endpoint's delegatecall fallback forwards execution to `EndpointTx.processSlowModeTransactionImpl` in Endpoint's storage context.
5. `validateSender(victim_sub, victim_addr)` passes â€” `victim_sub`'s address prefix matches `victim_addr`.
6. `requireSubaccount(victim_sub)` passes â€” victim has an existing subaccount.
7. `linkedSigners[victim_sub] = attacker_addr` is written to Endpoint's storage.
8. Attacker now signs a `WithdrawCollateral` transaction for `victim_sub` through the normal sequencer path and drains the victim's collateral.
