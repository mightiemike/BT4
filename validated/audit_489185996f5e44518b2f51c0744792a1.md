Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` gates on `owner` instead of `sender`, allowing non-allowlisted payers to bypass the deposit guard via `MetricOmmPoolLiquidityAdder` - (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument and checks only `owner` (the position holder) against the allowlist. Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` allows any `msg.sender` to specify an arbitrary `owner`, a non-allowlisted payer can name an allowlisted `owner` to pass the guard and inject tokens into the pool without authorization.

## Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` discards the first parameter (`sender`) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to `_beforeAddLiquidity`:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [2](#0-1) 

`ExtensionCalling._beforeAddLiquidity` forwards both arguments faithfully to the extension:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [3](#0-2) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (the owner-specifying overload) allows any `msg.sender` to supply an arbitrary `owner`, validated only for non-zero:

```solidity
function addLiquidityExactShares(address pool, address owner, ...) external payable override {
    _validateOwner(owner);   // only checks owner != address(0)
    _validateDeltas(deltas);
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
}
``` [4](#0-3) 

The call chain is: `bob` calls `liquidityAdder.addLiquidityExactShares(pool, alice, ...)` → pool's `addLiquidity` is called with `msg.sender = liquidityAdder` → `_beforeAddLiquidity(liquidityAdder, alice, ...)` → extension checks `allowedDepositor[pool][alice]` → passes because `alice` is allowlisted → `bob` pays tokens via callback, `alice` receives the LP position.

The `sender` argument (which is `liquidityAdder`, representing `bob`'s economic action) is never checked. The `IMetricOmmPoolActions` NatSpec even acknowledges the operator pattern explicitly: "msg.sender pays but need not equal owner (operator pattern)." [5](#0-4) 

The existing test suite confirms the extension gates on `owner` (the `depositor` argument passed as the second parameter), not `sender`: [6](#0-5) 

The integration test in `FullMetricExtensionTest` allowlists `_getCallerAddress(0)` (the `TestCaller` contract, which is `msg.sender` to the pool), confirming the intended guard is the direct pool caller — but the extension checks `owner` instead. [7](#0-6) 

## Impact Explanation

This is a direct admin-boundary break. The `DepositAllowlistExtension` is the pool admin's primary mechanism to restrict who can economically participate (inject tokens, grow LP positions). A non-allowlisted actor can bypass this guard entirely by routing through `MetricOmmPoolLiquidityAdder` with an allowlisted `owner`. Consequences include: (1) unrestricted token injection into a pool intended to be curated/KYC-gated; (2) forced LP positions on allowlisted addresses they did not initiate, usable as a griefing vector; (3) complete nullification of any compliance or curated-LP invariant the pool admin configured. This meets the "admin-boundary break: pool admin exceeds caps, bypasses timelocks, or factory/oracle role checks are bypassed by an unprivileged path" impact criterion.

## Likelihood Explanation

- `MetricOmmPoolLiquidityAdder` is a public, permissionless periphery contract with no access control on `addLiquidityExactShares`.
- The attacker needs only one allowlisted address, trivially discoverable from `AllowedToDepositSet` events or direct `allowedDepositor` storage reads.
- No flash loans, oracle manipulation, or special privileges are required.
- The attack is repeatable for any pool using `DepositAllowlistExtension` with a non-open allowlist.

## Recommendation

Change `beforeAddLiquidity` to gate on `sender` (the actual payer / pool's `msg.sender`) rather than `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the payer and the position holder, check both `sender` and `owner`. Update `setAllowedToDeposit` documentation to clarify which identity pool admins should allowlist.

## Proof of Concept

```solidity
// Pool deployed with DepositAllowlistExtension.
// Admin allowlists only alice (as owner/depositor).
extension.setAllowedToDeposit(pool, alice, true);

// Bob (not allowlisted) calls the public liquidity adder,
// naming alice as owner. Bob is the actual payer (msg.sender).
vm.startPrank(bob);
token0.approve(address(liquidityAdder), type(uint256).max);
token1.approve(address(liquidityAdder), type(uint256).max);

// Call path:
// liquidityAdder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, "")
//   -> pool.addLiquidity(alice, salt, deltas, callbackData, "")
//      -> _beforeAddLiquidity(liquidityAdder, alice, ...)
//         -> extension.beforeAddLiquidity(liquidityAdder, alice, ...)
//            -> checks allowedDepositor[pool][alice] == true -> PASSES
//      -> callback pulls tokens from bob (payer stored in transient context)
//      -> alice receives LP position
liquidityAdder.addLiquidityExactShares(
    pool, alice, salt, deltas, type(uint256).max, type(uint256).max, ""
);
vm.stopPrank();

// Bob (non-allowlisted) has injected tokens; alice holds the LP position.
// DepositAllowlistExtension guard was bypassed.
assertGt(stateView.positionBinShares(pool, alice, salt, bin), 0);
```
