Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the Original User, Allowing Any User to Bypass the Swap Allowlist via the Router - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `sender` is the pool's `msg.sender` â€” the router contract â€” not the original end-user. When swaps are routed through `MetricOmmSimpleRouter`, the extension checks whether the router is allowlisted rather than the actual trader, completely defeating the per-user access control the pool admin configured.

## Finding Description

**Root cause:** In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]` â€” `msg.sender` is the pool (correct), but `sender` is whoever called the pool, which is the router: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly with no original-user context passed to the pool: [4](#0-3) 

The same applies to `exactInput` (L103-112), `exactOutputSingle` (L130-147), and `exactOutput` (L154-188): [5](#0-4) 

**Why existing checks fail:** There is no mechanism in the pool's `swap` interface or in `_beforeSwap` to carry the original caller's identity through the router. The extension has no way to distinguish the router from the end-user.

**Contrast with `DepositAllowlistExtension`:** The deposit path avoids this bug because `addLiquidity` accepts an explicit `owner` parameter that the caller sets to the actual position owner, and the extension checks `owner` rather than `sender`: [6](#0-5) 

No analogous explicit-user parameter exists in the `swap` interface.

## Impact Explanation

Two concrete failure modes arise, both breaking the allowlist invariant:

**Mode 1 â€” Full allowlist bypass:** The pool admin must allowlist the router address (`setAllowedToSwap(pool, router, true)`) for any router-mediated swap to succeed. Once the router is allowlisted, every user on the network can swap through the router regardless of whether their own address is individually blocked. The per-user allowlist is completely defeated for the router path.

**Mode 2 â€” Allowlisted users locked out:** If the admin does not allowlist the router (attempting to enforce per-user control), individually allowlisted users cannot use the router at all, even though they are supposed to be permitted. Core swap functionality is broken for the intended user set.

In both cases `SwapAllowlistExtension` fails to enforce the policy the pool admin configured. This constitutes broken core pool functionality â€” the allowlist extension is the primary mechanism for curated pools to restrict trading to known counterparties.

## Likelihood Explanation

- `MetricOmmSimpleRouter` is the primary public swap interface; most users will route through it rather than calling the pool directly.
- Any pool that deploys `SwapAllowlistExtension` and expects to support router-mediated swaps is immediately affected upon deployment.
- No special privilege or setup is required: any user simply calls `exactInputSingle` on the router pointing at the curated pool.
- The bug is deterministic and reproducible on every router-mediated swap to an allowlist-gated pool.

## Recommendation

Add an explicit `originalSender` parameter to the pool's `swap` function and thread it through `_beforeSwap` and `IMetricOmmExtensions.beforeSwap`, so `SwapAllowlistExtension` can check the actual end-user rather than the intermediary router. The router would pass `msg.sender` as `originalSender` when calling the pool. This mirrors how `addLiquidity` separates `msg.sender` (the caller/router) from `owner` (the actual position owner), which is why `DepositAllowlistExtension` does not share this bug.

Alternatively, document `SwapAllowlistExtension` as only safe for direct pool calls and implement a trusted-forwarder pattern in the router to propagate original caller identity.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured on `beforeSwap`.
2. Admin allowlists `alice` via `setAllowedToSwap(pool, alice, true)`.
3. Admin also allowlists the router via `setAllowedToSwap(pool, router, true)` (required for any router swap to work).
4. `bob` (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
5. The pool calls `_beforeSwap(msg.sender=router, ...)`.
6. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][router]` â†’ `true`.
7. `bob`'s swap succeeds despite never being allowlisted â€” allowlist fully bypassed.

Conversely, if step 3 is omitted (router not allowlisted), `alice`'s router swap reverts at step 6 even though she is individually permitted â€” allowlisted users locked out of the router path.
