### Title
`ENABLE_SECP_OPS` Flag Check Bypassed via 4-Byte Opcode Path for secp256k1/r1 Verify - (File: `src/chia_dialect.rs`)

### Summary
In `ChiaDialect::op()`, the 1-byte opcodes 64 and 65 for `op_secp256k1_verify` and `op_secp256r1_verify` are gated behind `ClvmFlags::ENABLE_SECP_OPS`. However, the parallel 4-byte opcode path for the same two operations (`0x13d61f00` and `0x1c3a8f00`) dispatches directly to the same functions without checking `ENABLE_SECP_OPS`. An attacker-controlled CLVM program can use the 4-byte opcode encoding to invoke secp signature verification in any execution context, regardless of whether the flag is set.

### Finding Description

In `src/chia_dialect.rs`, the `op()` method of `ChiaDialect` has two separate dispatch paths for secp operations.

**Path 1 — 1-byte opcodes (flag-gated):** [1](#0-0) 

Opcodes 64 and 65 are only dispatched to `op_secp256k1_verify` / `op_secp256r1_verify` when `flags.contains(ClvmFlags::ENABLE_SECP_OPS)` is true. Without the flag, they fall through to `unknown_operator`, which in `MEMPOOL_MODE` (which sets `NO_UNKNOWN_OPS`) returns an error.

**Path 2 — 4-byte opcodes (no flag check):** [2](#0-1) 

When the operator atom is 4 bytes long, the code matches `0x13d61f00` → `op_secp256k1_verify` and `0x1c3a8f00` → `op_secp256r1_verify` and calls them directly — with no check for `ENABLE_SECP_OPS`. The `flags` variable passed to the function is the same combined flags value, but the gate itself is absent.

The flag is defined as: [3](#0-2) 

`MEMPOOL_MODE` does not include `ENABLE_SECP_OPS`: [4](#0-3) 

So in mempool mode, a program using 1-byte opcode 64 is rejected, but the same secp verification using 4-byte opcode `0x13d61f00` executes successfully.

### Impact Explanation

**Impact: Low–Medium**

Any caller that relies on `ENABLE_SECP_OPS` being absent to prevent secp operations (e.g., a node running before a soft-fork activation, or a mempool validator) can be bypassed. The concrete corrupted result is: `op_secp256k1_verify` or `op_secp256r1_verify` executes and returns a result (or raises) in a context where it should be unconditionally rejected. This can cause:

1. **Consensus divergence**: Nodes that enforce the flag via 1-byte opcodes reject the program; nodes that receive the 4-byte form accept it, producing a split.
2. **Flag-gate bypass**: The `ENABLE_SECP_OPS` flag, exported to Python callers via `wheel/src/api.rs`, provides a false security guarantee — callers believe setting it absent blocks secp ops, but the 4-byte path is always open. [5](#0-4) 

### Likelihood Explanation

**Likelihood: High**

The attacker-controlled entry path is direct: submit a CLVM program blob to `run_serialized_chia_program` (or any `run_program` caller) with the operator atom set to the 4-byte value `0x13d61f00` or `0x1c3a8f00`. No privileged access is required. The 4-byte opcode encoding is part of the documented "unknown operators with assigned cost" scheme and is a valid, parseable CLVM atom. Any external caller submitting crafted CLVM bytes can trigger this.

### Recommendation

Add the `ENABLE_SECP_OPS` check to the 4-byte opcode dispatch path, mirroring the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures both encoding paths enforce the same flag gate, eliminating the bypass.

### Proof of Concept

1. Call `run_serialized_chia_program` with `flags = MEMPOOL_MODE` (which does **not** include `ENABLE_SECP_OPS`).
2. Construct a CLVM program that applies operator atom `\x13\xd6\x1f\x00` (4 bytes) to three arguments (public key, message, signature).
3. **Expected (correct) behavior**: rejected, because secp ops are not enabled.
4. **Actual behavior**: `op_secp256k1_verify` executes and returns a result, bypassing the flag gate entirely.

The same program using 1-byte opcode `\x40` (64) would be rejected by `unknown_operator` under `NO_UNKNOWN_OPS`, confirming the asymmetry. [6](#0-5)
