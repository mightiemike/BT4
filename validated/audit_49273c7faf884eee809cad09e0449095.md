Audit Report

## Title
SwapAllowlistExtension checks router address instead of original swapper, enabling allowlist bypass via MetricOmmSimpleRouter ŌĆö (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. If a pool admin allowlists the router address to enable router-mediated swaps for approved users, every unpermissioned address can bypass the allowlist by routing through the same router contract.

## Finding Description
**Root cause ŌĆö wrong identity checked:**

`SwapAllowlistExtension.beforeSwap` checks `sender` against `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`:

```solidity
// SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmPool.swap` passes its own `msg.sender` as `sender` to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-231
_beforeSwap(
    msg.sender,   // ŌåÉ whoever called pool.swap()
    ...
```

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension:

```solidity
// ExtensionCalling.sol:160-165
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

**Exploit path:**

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `IMetricOmmPoolActions(params.pool).swap(...)` directly:

```solidity
// MetricOmmSimpleRouter.sol:72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
    );
```

The pool receives `msg.sender = router address`. The extension then checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][original_user]`.

**Bypass scenario:**
1. Pool admin deploys pool with `SwapAllowlistExtension` to restrict swaps to approved addresses.
2. Admin allowlists the router address (`setAllowedToSwap(pool, router, true)`) so that approved users can swap via the router.
3. Any unpermissioned address calls `MetricOmmSimpleRouter.exactInputSingle` with that pool.
4. The pool sees `msg.sender = router`, the extension checks `allowedSwapper[pool][router] == true`, and the swap proceeds ŌĆö bypassing the per-user allowlist entirely.

**Existing guards are insufficient:** The extension has no mechanism to recover the original EOA/caller from the router context. There is no `tx.origin` check, no callback-context forwarding of the original caller, and no separate router-aware allowlist path.

## Impact Explanation
The `SwapAllowlistExtension` is the sole on-chain mechanism for pools to restrict swap access to approved addresses. A complete bypass allows any unpermissioned address to swap in a pool intended to be restricted, violating the admin-boundary invariant. Depending on pool configuration, this can result in unauthorized trading, extraction of liquidity at controlled prices, or circumvention of KYC/compliance-gated pools ŌĆö constituting a broken core access-control flow with direct fund impact.

## Likelihood Explanation
The attack requires only that a pool admin has allowlisted the router address (a natural and expected configuration for any pool that wants to support router-mediated swaps for approved users). Once that condition holds, any unpermissioned address can exploit it permissionlessly and repeatably with a single `exactInputSingle` call. No special privileges, flash loans, or complex setup are required.

## Recommendation
The extension must check the true originating swapper, not the intermediary router. Options:
1. **Pass original caller through router:** Have `MetricOmmSimpleRouter` store `msg.sender` in transient storage and expose it via a standardized interface (e.g., `ICallerContext`), then have `SwapAllowlistExtension.beforeSwap` query the pool's registered router for the original caller.
2. **Reject router-mediated swaps entirely:** Document that allowlisted pools must not allowlist router addresses and instead require users to call `pool.swap()` directly.
3. **Use `tx.origin` with caution:** Only viable if the pool is not intended to be called from other contracts; generally not recommended.

The cleanest fix is option 1: the router stores `msg.sender` in transient storage before calling the pool, and the extension reads it back via a known interface on the router.
