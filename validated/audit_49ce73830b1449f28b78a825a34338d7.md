### Title
Fee-on-Transfer Token Accounting Inflation in `depositCollateralWithReferral()` Allows Subaccount Balance to Exceed Actual Protocol Holdings — (File: `core/contracts/Endpoint.sol` / `core/contracts/Clearinghouse.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral()` transfers a caller-supplied `amount` of tokens into the protocol via `handleDepositTransfer`, then enqueues a slow-mode `DepositCollateral` transaction carrying that same nominal `amount`. When the sequencer later executes the slow-mode transaction, `Clearinghouse.depositCollateral()` credits the full nominal `amount` to the subaccount without ever measuring what was actually received. For any fee-on-transfer token listed as a supported product, the protocol's internal SpotEngine balance for the depositor is inflated above the real token holdings, creating a permanent solvency shortfall.

---

### Finding Description

**Step 1 — Deposit entry point records nominal amount, not received amount.**

`Endpoint.depositCollateralWithReferral()` calls `handleDepositTransfer` to pull tokens, then immediately encodes the caller-supplied `amount` into a slow-mode transaction: [1](#0-0) 

The `DepositCollateral` struct stored in `slowModeTxs` carries `amount: amount` — the value the caller passed in, not the value the contract actually received.

**Step 2 — Clearinghouse credits the nominal amount unconditionally.**

When the sequencer executes the slow-mode transaction, `Clearinghouse.depositCollateral()` scales and credits `txn.amount` directly: [2](#0-1) 

There is no pre/post balance measurement, no comparison against actual received tokens, and no revert path if the two values diverge.

**Step 3 — `DirectDepositV1.creditDeposit()` is an unpermissioned trigger.**

Any external caller can invoke `creditDeposit()`, which reads the DDA's token balance and passes it as the deposit amount: [3](#0-2) 

For a fee-on-transfer token, `handleDepositTransfer` pulls `balance` from the DDA but the Endpoint contract receives only `balance − fee`. The slow-mode queue records `balance`. The subaccount is later credited `balance`, not `balance − fee`.

---

### Impact Explanation

For every deposit of a fee-on-transfer token, the SpotEngine records a balance larger than the actual token holdings of the protocol. The gap accumulates with each deposit. When users withdraw, `withdrawCollateral` calls `spotEngine.assertUtilization` and transfers the full credited amount out. The last withdrawers will find the protocol insolvent — the token balance held on-chain is less than the sum of all credited subaccount balances. This is a direct, permanent solvency/accounting corruption affecting all depositors of the affected token. [4](#0-3) 

---

### Likelihood Explanation

Medium. The trigger requires a fee-on-transfer token to be registered as a supported product via `spotEngine.getConfig(productId).token`. If any such token is listed (e.g., tokens with built-in transfer taxes, rebasing tokens, or tokens that deduct fees on transfer), any unprivileged user can call `depositCollateralWithReferral` or `DirectDepositV1.creditDeposit()` to inflate their subaccount balance. The entry points are fully public and require no special role. [5](#0-4) 

---

### Recommendation

Measure the actual received amount using a pre/post balance check inside `handleDepositTransfer` (or at the `depositCollateralWithReferral` call site), and encode only the measured delta into the slow-mode `DepositCollateral` transaction. For example:

```solidity
uint256 pre = token.balanceOf(address(this));
handleDepositTransfer(token, msg.sender, uint256(amount));
uint256 post = token.balanceOf(address(this));
uint128 received = uint128(post - pre);
// encode `received` into the slow-mode tx, not `amount`
```

This mirrors the standard "received-amount" pattern and eliminates the inflation vector for fee-on-transfer tokens.

---

### Proof of Concept

1. A fee-on-transfer token (e.g., 1% fee on every transfer) is registered as product `P` in SpotEngine.
2. Attacker holds 1000 tokens. They call `Endpoint.depositCollateralWithReferral(subaccount, P, 1000, "")`.
3. `handleDepositTransfer` pulls 1000 tokens from the attacker; the Endpoint contract receives 990 (1% fee deducted by the token).
4. A slow-mode `DepositCollateral { amount: 1000 }` is enqueued.
5. Sequencer executes the slow-mode tx; `Clearinghouse.depositCollateral` credits `1000 * multiplier` to the subaccount.
6. Attacker's subaccount shows 1000 units of collateral; the protocol only holds 990 tokens.
7. Attacker withdraws 1000 tokens. The protocol transfers 1000 tokens out, leaving a 10-token deficit that is socialized across remaining depositors.
8. Repeated deposits compound the shortfall until the protocol is insolvent for the affected token. [6](#0-5) [7](#0-6)

        bytes32 subaccount,
        uint32 productId,
        uint128 amount,

        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

            // we can just skip the deposit and continue with the next asset.

    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
