### Title
`CongestionInfoV1::allowed_shard` stores a raw `ShardId` value cast to `u16`, but is compared against `ShardId` after resharding when `ShardId ≠ ShardIndex` — (`core/primitives/src/congestion_info.rs`)

### Summary

`CongestionInfoV1.allowed_shard` is a `u16` field that is written by casting a `ShardId` directly via `.into()`, and read back by casting the `u16` back to a `ShardId` via `ShardId::from(self.info.allowed_shard())`. Under `ShardLayoutV0`/`V1` this is harmless because `ShardId == ShardIndex` (both are contiguous from 0). Under `ShardLayoutV2`/`V3` (post-resharding), `ShardId` values are **non-contiguous** (e.g. after one split the children get ids `max+1`, `max+2`). The `u16` field can only hold values up to 65535, and more importantly the round-trip `ShardId → u16 → ShardId` is only correct when the shard id fits in a `u16` and is equal to the shard index. After resharding the stored `allowed_shard` value is a raw `ShardId` integer, but the congestion-control comparison in `outgoing_gas_limit` and `outgoing_size_limit` reconstructs a `ShardId` from that integer and compares it against the actual `ShardId` of the sender — which is correct only if the `ShardId` value itself was stored. The real bug is the **semantic ambiguity**: the field is documented as "only this shard can forward receipts" but the type (`u16`) is too narrow for large post-resharding `ShardId` values, and the conversion path `ShardId → u16` silently truncates any `ShardId > 65535`.

### Finding Description

`CongestionInfoV1` stores the allowed shard as a `u16`:

```rust
pub struct CongestionInfoV1 {
    ...
    pub allowed_shard: u16,
}
```

`finalize_allowed_shard` selects a `ShardId` from `all_shards` and stores it via:

```rust
self.set_allowed_shard(allowed_shard.into());
```

where `Into<u16> for ShardId` is implemented as `self.0 as u16` — a **truncating cast**.

The read-back path in `outgoing_gas_limit` and `outgoing_size_limit` reconstructs the `ShardId` as:

```rust
if sender_shard == ShardId::from(self.info.allowed_shard()) { ... }
```

Under `ShardLayoutV2`/`V3`, after each resharding event the new child shard ids are assigned as `max_shard_id + 1`, `max_shard_id + 2`. After enough resharding rounds (or starting from a large initial shard id), the `ShardId` value exceeds `u16::MAX = 65535`. At that point `allowed_shard.into()` silently truncates the high bits, storing a wrong value. When the congestion control logic reads it back and reconstructs a `ShardId`, it gets a different shard than was intended. The result is that the wrong shard is granted the exclusive right to send receipts to a fully-congested shard, while the correct shard is blocked — or vice versa.

Additionally, in `run_bandwidth_scheduler`, the `allowed_shard` is read back and converted to a `ShardId`, then looked up via `shard_layout.get_shard_index(allowed_sender_shard_id).ok()`. If the truncated value happens to match a valid but wrong `ShardId`, the bandwidth scheduler grants the wrong shard exclusive access. If it matches no valid shard, `get_shard_index` returns `Err` and `.ok()` yields `None`, causing `allowed_sender_shard_index` to be `None` — meaning **no shard** is allowed to send to the fully-congested receiver, permanently stalling progress (liveness failure).

The analog to the external bug is exact: just as `LiquidityPool.deposit` returns shares (not token amounts) and the caller treats them as amounts, `finalize_allowed_shard` produces a `ShardId` (not a `u16`-range index) and the storage field treats it as a `u16`, silently truncating the identity of the allowed shard.

### Impact Explanation

**Liveness / correctness failure under resharding.** When a shard is fully congested and its `allowed_shard` field has been truncated:

1. The wrong shard may be granted exclusive forwarding rights, causing the correct shard to be permanently blocked from sending receipts to the congested shard.
2. If the truncated value matches no valid shard, `allowed_sender_shard_index` becomes `None` in the bandwidth scheduler, and `calculate_is_link_allowed` returns `false` for all senders — no shard can send to the congested receiver, stalling the network.

This is a **protocol-level liveness invariant break**: the congestion control guarantee that "at least one shard can always make progress" is violated.

### Likelihood Explanation

The current mainnet shard layout uses `ShardLayoutV2` with shard ids assigned as `max+1`, `max+2` per split. Starting from 4 shards (ids 0–3), after ~16 resharding rounds the maximum shard id exceeds 65535. This is not imminent on mainnet today, but the code path is reachable in principle as resharding continues. The truncation is silent — no panic, no error — so it would be very hard to detect before it causes a liveness failure.

### Recommendation

Replace the `u16 allowed_shard` field in `CongestionInfoV1` with a type that can faithfully represent the full `ShardId` range (e.g. `u64`), or introduce a new `CongestionInfoV2` with `allowed_shard: u64`. The conversion in `finalize_allowed_shard` should store the raw `u64` value of the `ShardId`, and the read-back in `outgoing_gas_limit`/`outgoing_size_limit`/`run_bandwidth_scheduler` should reconstruct the `ShardId` from a `u64`. Since `CongestionInfo` is part of the `ChunkHeader` and is Borsh-serialized on-chain, this requires a protocol version bump and a new versioned variant.

### Proof of Concept

The truncation path:

```
finalize_allowed_shard(own_shard, all_shards, seed)
  → get_new_allowed_shard returns ShardId(N)   // N > 65535 after many reshards
  → set_allowed_shard(ShardId(N).into())        // Into<u16>: N as u16 → truncated!
  → CongestionInfoV1.allowed_shard = (N & 0xFFFF) as u16
```

The read-back path:

```
outgoing_gas_limit(sender_shard: ShardId(M))
  → ShardId::from(self.info.allowed_shard())   // = ShardId((N & 0xFFFF) as u64)
  → sender_shard == ShardId((N & 0xFFFF))      // false if M == N but N > 65535
  → returns Gas::ZERO                           // correct shard is blocked
```
