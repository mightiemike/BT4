Audit Report

## Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Allowlist Bypass — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` is documented as gating `addLiquidity` by depositor address, but checks the caller-supplied `owner` parameter (LP-position recipient) instead of the `sender` parameter (the actual transaction initiator who pays tokens via callback). Any unprivileged address can bypass the allowlist by passing an allowlisted address as `owner`, depositing tokens into a restricted pool without being allowlisted.

## Finding Description
`MetricOmmPool.addLiquidity` accepts `owner` as a caller-supplied argument and forwards `msg.sender` as `sender` to extensions:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` passes both to the extension via `abi.encodeCall`: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the first argument (`sender`) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

The contract NatSpec and setter naming confirm the intent is to gate the actual depositing party: [4](#0-3) 

Token payment is pulled from `msg.sender` (the `sender`) via `IMetricOmmModifyLiquidityCallback(msg.sender).metricOmmModifyLiquidityCallback(...)` in `LiquidityLib.addLiquidity`, not from `owner`: [5](#0-4) 

Because `owner` is a freely chosen parameter with no on-chain binding to the token source, any address can pass an allowlisted address as `owner` and the guard succeeds. The pool then pulls tokens from the unauthorized `sender` via callback and credits LP shares to the supplied `owner`.

## Impact Explanation
The pool admin's deposit allowlist is completely ineffective. Any address — regardless of allowlist status — can deposit tokens into a restricted pool by supplying an allowlisted address as `owner`. This breaks the core access-control invariant for pools configured as private, institutional, or KYC-gated liquidity venues. The unauthorized depositor pays tokens (pulled via callback), the allowlisted address receives LP shares it never requested, and the pool admin's restriction is violated. This constitutes a broken core pool functionality and admin-boundary break where an unprivileged path bypasses a configured access control.

## Likelihood Explanation
The bypass requires only a single `addLiquidity` call with a publicly observable allowlisted address as `owner`. The `allowedDepositor` mapping is public, so any actor can read allowlist state on-chain. No special privileges, flash loans, or multi-step setup are needed. Any actor who can call `addLiquidity` and implement `IMetricOmmModifyLiquidityCallback` can execute the bypass immediately and repeatedly.

## Recommendation
Replace the `owner` check with a `sender` check to gate the actual depositing party:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
1. Pool admin deploys pool with `DepositAllowlistExtension` and calls `setAllowedToDeposit(pool, Alice, true)`. Bob is **not** allowlisted.
2. Bob (implementing `IMetricOmmModifyLiquidityCallback`) calls `pool.addLiquidity(Alice, salt, deltas, callbackData, extensionData)`.
3. Pool calls `_beforeAddLiquidity(Bob /*msg.sender*/, Alice /*owner*/, ...)` → extension receives `(Bob, Alice, ...)`.
4. Extension discards `Bob` (first arg) and evaluates `allowedDepositor[pool][Alice]` → `true` → **no revert**.
5. `LiquidityLib.addLiquidity` executes: LP shares credited to `Alice`'s position key; callback pulls tokens from `Bob`.
6. Bob has deposited into a pool he is not allowlisted for; the allowlist is fully bypassed.
