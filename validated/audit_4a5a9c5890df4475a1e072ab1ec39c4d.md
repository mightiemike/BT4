### Title
Gateway L2 Gas Price Admission Check Uses Stale Previous-Block Price Instead of Next-Block EIP-1559 Price, Causing Incorrect Transaction Admission/Rejection - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `validate_resource_bounds` check computes its admission threshold from the **previous committed block's** L2 gas price. The actual L2 gas price for the next block is computed by the proposer using an EIP-1559-style fee market algorithm (`calculate_next_base_gas_price`). These two values diverge every block. Transactions whose `max_l2_gas_price` falls in the gap between the stale threshold and the actual next-block price are admitted by the gateway but will fail at execution. Conversely, during price decreases, transactions above the actual next-block price but below the stale threshold are incorrectly rejected. The code itself contains a developer TODO acknowledging the root cause.

---

### Finding Description

In `validate_resource_bounds`, the gateway reads the L2 gas price from the last committed block via `gateway_fixed_block_state_reader.get_block_info()` and uses it as the reference for the admission threshold:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The threshold is then computed as `min_gas_price_percentage% * previous_block_l2_gas_price`:

```rust
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();
if tx_l2_gas_price.0 < threshold {
    return Err(...GAS_PRICE_TOO_LOW...);
}
``` [2](#0-1) 

However, the actual L2 gas price for the next block is computed by the consensus orchestrator using an EIP-1559 formula in `calculate_next_base_gas_price`:

```rust
let price_change = (price_u256 * gas_delta) / denominator;
let adjusted_price_u256 =
    if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };
``` [3](#0-2) 

This price is stored in `next_l2_gas_price` in the block header and propagated to the next block's `BlockInfo` via `update_state_sync_with_new_block`: [4](#0-3) 

The `GatewayFixedBlockSyncStateClient` reads the **current block's** `l2_gas_price` (not `next_l2_gas_price`) from the block header: [5](#0-4) 

The two values are structurally different: `l2_gas_price` is the price *used in* the committed block, while `next_l2_gas_price` is the price *computed for* the upcoming block by the EIP-1559 algorithm. The gateway uses the former; execution uses the latter.

---

### Impact Explanation

**High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

Two concrete failure modes:

1. **False admission (invalid tx accepted):** During high congestion, `next_l2_gas_price > previous_block_l2_gas_price`. A transaction with `max_l2_gas_price` satisfying `threshold ≤ max_l2_gas_price < next_l2_gas_price` passes gateway admission but fails at execution because the block's actual L2 gas price exceeds the transaction's declared maximum. The transaction is admitted to the mempool, consumes sequencer resources, and then reverts or is dropped.

2. **False rejection (valid tx rejected):** During low congestion, `next_l2_gas_price < previous_block_l2_gas_price`. A transaction with `max_l2_gas_price` satisfying `next_l2_gas_price ≤ max_l2_gas_price < threshold` is rejected by the gateway even though it would succeed at execution. This is a liveness issue for users.

The EIP-1559 formula can move the price by up to `price / gas_price_max_change_denominator` per block in each direction. [6](#0-5) 

With the test values showing `GasPrice(30_000_000_000)` → `GasPrice(30_312_500_000)` in a single high-congestion block (~1.04% increase), the gap between the stale threshold and the actual next-block price is non-trivial and grows with sustained congestion. [7](#0-6) 

---

### Likelihood Explanation

**High.** The divergence occurs every block whenever gas usage deviates from the target. The EIP-1559 mechanism is designed to continuously adjust the price, so the stale-vs-actual gap is the normal operating condition, not an edge case. Any user submitting a transaction with `max_l2_gas_price` near the threshold during a period of changing gas prices will be affected. No special privileges or adversarial behavior are required.

The developer TODO comment `// TODO(Arni): getnext_l2_gas_price from the block header.` at line 229 confirms the team is aware the current value is wrong and the correct value is `next_l2_gas_price` from the block header. [8](#0-7) 

---

### Recommendation

Replace the `l2_gas_price` read in `validate_resource_bounds` with `next_l2_gas_price` from the committed block header. The `GatewayFixedBlockStateReader::get_block_info` interface should expose `next_l2_gas_price` (already present in `BlockHeaderWithoutHash`) so the gateway can compute the threshold against the price that will actually govern the next block's execution.

Concretely, in `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`, populate a `next_l2_gas_price` field from `block_header.next_l2_gas_price` and use that value in `validate_resource_bounds` instead of `block_header.l2_gas_price`.

---

### Proof of Concept

1. Suppose the last committed block has `l2_gas_price = 30_000_000_000` fri and was at 75% capacity (above the 60% gas target).
2. The EIP-1559 formula computes `next_l2_gas_price ≈ 30_312_500_000` fri (~1.04% increase).
3. The gateway's threshold with `min_gas_price_percentage = 100` is `30_000_000_000`.
4. A user submits a transaction with `max_l2_gas_price = 30_100_000_000`.
5. Gateway check: `30_100_000_000 >= 30_000_000_000` → **admitted**.
6. Actual block execution: block L2 gas price is `30_312_500_000 > 30_100_000_000` → **fee check fails**, transaction reverts or is excluded.

The transaction was admitted by the gateway based on a stale price reference, consuming mempool and sequencer resources, but cannot be successfully executed in the next block.

pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }
