### Title
Duplicate Position IDs in `ClaimTierRewards` Enable Double-Claiming of Bonus Rewards - (File: x/tieredrewards/keeper/msg_server.go)

---

### Summary

`MsgClaimTierRewards` accepts a list of position IDs, loads all positions into memory before processing any of them, and then iterates over the in-memory slice to claim and update each position. Because there is no deduplication of `PositionIds`, an attacker who submits the same position ID twice causes `processEventsAndClaimBonus` to execute twice against the same stale in-memory checkpoint, resulting in double-payment of bonus rewards from the `RewardsPoolName` module account.

---

### Finding Description

`ClaimTierRewards` in `msg_server.go` follows a **read-all-then-act** pattern: [1](#0-0) 

All positions are loaded and ownership-validated in a first pass. Only then does `claimRewardsAndUpdatesPositions` iterate over the collected slice and process each entry: [2](#0-1) 

If `msg.PositionIds` contains `[X, X]`, the slice holds **two independent in-memory copies** of position X, both carrying the same stale `LastBonusAccrual` and `LastEventSeq` values. The first iteration claims the bonus, advances the checkpoints, and persists the updated position to the store via `setPosition`. The second iteration operates on the second copy, which still holds the original (pre-claim) `LastBonusAccrual` as `segmentStart`: [3](#0-2) 

Because `segmentStart` is stale, `computeSegmentBonus` produces the same non-zero result as the first iteration. The pool-balance guard and `SendCoinsFromModuleToAccount` are then executed a second time: [4](#0-3) 

`claimBaseRewards` is safe from doubling because the Cosmos SDK distribution module tracks rewards per delegator-validator pair and returns zero on the second `WithdrawDelegationRewards` call. The bonus path has no equivalent on-chain guard.

---

### Impact Explanation

The `types.RewardsPoolName` module account balance is the corrupted value. An attacker who owns a position with accrued bonus rewards can drain the pool at a rate proportional to the number of duplicate IDs they include in a single `MsgClaimTierRewards` transaction. Each duplicate causes one additional full bonus payment for the same accrual window. With N duplicates the attacker receives N × their legitimate bonus in a single transaction.

---

### Likelihood Explanation

The entry path is a standard, unprivileged `MsgClaimTierRewards` transaction signed by any position owner. No special role, governance action, or IBC relay is required. The only prerequisite is owning at least one position with a non-zero accrued bonus. The transaction is valid from the perspective of basic message validation because the position IDs are syntactically correct and the caller is the legitimate owner.

---

### Recommendation

Add a deduplication check in `ClaimTierRewards` before appending to the `positions` slice, or enforce uniqueness inside `msg.Validate()` on `MsgClaimTierRewards`. A simple approach:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    if _, dup := seen[posId]; dup {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest, "duplicate position id: %d", posId)
    }
    seen[posId] = struct{}{}
    // ... existing load and validate logic
}
```

Alternatively, enforce the invariant at the `msg.Validate()` level so it is rejected before reaching the keeper.

---

### Proof of Concept

1. Attacker creates a tiered-rewards position (position ID = `42`) and waits for bonus rewards to accrue.
2. Attacker submits `MsgClaimTierRewards{Owner: attacker, PositionIds: [42, 42]}`.
3. The first pass (lines 435–446) loads position 42 twice into `positions[0]` and `positions[1]`, both with `LastBonusAccrual = T_old`.
4. `claimRewardsAndUpdatesPositions` processes `positions[0]`: computes bonus for `[T_old, T_now]`, sends coins from `RewardsPoolName`, saves updated position with `LastBonusAccrual = T_now`.
5. `claimRewardsAndUpdatesPositions` processes `positions[1]` (stale copy): `segmentStart = T_old` again, computes the same bonus for `[T_old, T_now]`, passes `sufficientBonusPoolBalance`, sends coins from `RewardsPoolName` a second time.
6. Attacker receives 2× the legitimate bonus; the rewards pool is debited twice for the same accrual window. [5](#0-4) [2](#0-1) [6](#0-5)

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}
