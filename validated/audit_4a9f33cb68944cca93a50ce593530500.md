Audit Report

## Title
SwapAllowlistExtension checks the router address instead of the originating user, allowing any caller to bypass the curated-pool swap allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the direct caller of `pool.swap()` (the `sender` argument, which is `msg.sender` inside the pool), not the originating end user. When `MetricOmmSimpleRouter` is allowlisted — the only way to enable router-mediated swaps on an allowlisted pool — any unprivileged user can bypass the curated-pool swap gate by calling the public, permissionless router. The swap allowlist, the pool's primary curation mechanism, is rendered entirely ineffective.

## Finding Description
**Root cause — wrong actor checked by the allowlist hook**

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← direct caller of pool.swap(), not the originating EOA
  recipient,
  ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the direct caller of `pool.swap()`:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

**The bypass path**

`MetricOmmSimpleRouter.exactInputSingle` is a fully public, permissionless function — no access control of any kind:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L67
function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
```

When the router calls `pool.swap(...)`, the pool receives `msg.sender = router`. The extension therefore checks `allowedSwapper[pool][router]`, not the originating EOA. If the pool admin has allowlisted the router (the natural operational step to enable router-mediated swaps for allowlisted users), the check passes for every caller on the network. There is no way to simultaneously allow router-mediated swaps for allowlisted users and block non-allowlisted users from using the same router, because the router is a public contract with no caller validation.

**Existing guards are insufficient**

The `allowedSwapper` mapping is keyed by `(pool, sender)` where `sender` is always the router when swapping through `MetricOmmSimpleRouter`. The `allowAllSwappers` flag is a separate escape hatch. Neither mechanism captures the originating EOA. The `_requireExpectedCallbackCaller` check in the router's callback only validates that the callback comes from the expected pool — it does not restrict who can initiate a swap through the router.

## Impact Explanation
Any user — regardless of whether they appear in the allowlist — can execute swaps on a curated pool by routing through `MetricOmmSimpleRouter`, provided the pool admin has allowlisted the router. The swap allowlist, which is the pool's primary curation mechanism, is rendered ineffective. For pools intended to be KYC-gated, institutional-only, or rate-limited, this constitutes a direct policy bypass: unauthorized parties can drain arbitrage value, front-run allowlisted LPs, or interact with pools they were explicitly excluded from. This matches the "admin-boundary break bypassed by an unprivileged path" and "broken core pool functionality" impact categories.

## Likelihood Explanation
Medium-to-high. Any pool admin who deploys a `SwapAllowlistExtension`-protected pool and also wants to support the standard router path will allowlist the router. The design gives no indication that doing so opens the gate to all users. The trap is structural — it requires no malicious intent from the admin, only the natural operational step of enabling router-mediated swaps. The attack is repeatable by any unprivileged address with no special capability required.

## Recommendation
The allowlist hook must check the **originating user**, not the intermediary. Two viable approaches:

1. **Pass the real caller through `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling `pool.swap`; the extension decodes and checks it. This requires the router to be trusted to supply honest data, which is acceptable for a canonical periphery contract.

2. **Trusted-forwarder registry**: The extension maintains a registry of trusted forwarders; when `sender` is a forwarder, it reads the real caller from `extensionData`; otherwise it checks `sender` directly.

Additionally, document clearly that allowlisting the router address is equivalent to `allowAllSwappers = true` for any user who can reach the router.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls setAllowedToSwap(pool, address(router), true)
    → intent: allow router-mediated swaps for allowlisted users
    → actual effect: any user can now swap

Attack:
  attacker = address not in allowedSwapper[pool]
  attacker calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(recipient, ...) with msg.sender = router
    → pool calls _beforeSwap(router, ...)
    → SwapAllowlistExtension.beforeSwap receives sender = router
    → checks allowedSwapper[pool][router] → true → passes
    → swap executes; attacker receives output tokens

The allowlist check never sees the attacker's address. The guard is fully bypassed.
