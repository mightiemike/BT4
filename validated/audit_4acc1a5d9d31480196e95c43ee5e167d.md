Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of end-user, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, that value is the router contract address, not the actual end-user. If the pool admin allowlists the router — the necessary configuration for any allowlisted user to use the standard periphery — the guard is completely neutralized and any unprivileged address can swap freely against the restricted pool.

## Finding Description
`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
  msg.sender,   // ← router address when called via MetricOmmSimpleRouter
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged via `abi.encodeCall` to every configured extension. `SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

where `msg.sender` = pool and `sender` = router. The check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

`MetricOmmSimpleRouter.exactInputSingle`, `exactInput`, and `exactOutputSingle` all call `pool.swap()` directly with no mechanism to encode or forward the original `msg.sender` into the `sender` slot. The `onlyPool` modifier on `BaseMetricExtension` confirms that `msg.sender` to the extension is always the pool, making `sender` the sole identity the extension can inspect — and it is the router, not the user.

Exploit flow:
1. Pool admin deploys pool with `SwapAllowlistExtension` in `BEFORE_SWAP_ORDER`.
2. Admin calls `setAllowedToSwap(pool, address(router), true)` so that allowlisted users can use the standard router.
3. Attacker (not individually allowlisted) calls `router.exactInputSingle(...)`.
4. Router calls `pool.swap(...)` with `msg.sender = router`.
5. Extension checks `allowedSwapper[pool][router]` → `true` → passes.
6. Attacker receives output tokens; allowlist guard is fully bypassed.

## Impact Explanation
The `SwapAllowlistExtension` is designed to restrict swapping to a curated set of addresses (e.g., KYC'd counterparties, whitelisted market makers). Once the router is allowlisted — which is required for any allowlisted user to use the standard periphery — the guard is completely neutralized. Any address can swap against the pool's LP reserves at oracle-derived prices. LPs who deposited under the assumption that access was restricted suffer unauthorized extraction of value from their positions. This constitutes an admin-boundary break (allowlist bypassed by an unprivileged path) and broken core pool functionality causing potential loss of LP assets, meeting Sherlock Medium/High thresholds.

## Likelihood Explanation
The two required conditions — a pool configured with `SwapAllowlistExtension` and the router being allowlisted — are both part of the intended production deployment pattern. No privileged attacker role is required; any EOA can call the public router. The scenario is repeatable and requires no special timing or state manipulation. The `generate_scanned_questions.py` pivot target explicitly identifies this call path as a validation focus, confirming it is an expected production configuration.

## Recommendation
The pool must convey the original caller's identity to the extension, not the intermediary's address. Two viable approaches:

1. **Router-forwarded identity via `extensionData`**: Define a convention where the router encodes `msg.sender` into `extensionData`, and `SwapAllowlistExtension.beforeSwap` decodes and checks that value when `sender` is a known trusted router. The extension must verify the router is trusted before accepting the encoded identity.

2. **Dedicated `originator` parameter**: Add an explicit `originator` field to the pool's `swap` signature that the router populates with its own `msg.sender` before calling the pool. The extension checks `originator` instead of `sender`. This is the cleanest fix and mirrors how Uniswap v4 separates `sender` from `hookData`-encoded originators.

```solidity
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool:             address(pool),
        recipient:        attacker,
        tokenIn:          token0,
        zeroForOne:       true,
        amountIn:         1_000e18,
        amountOutMinimum: 0,
        priceLimitX64:    0,
        deadline:         block.timestamp,
        extensionData:    ""
    })
);
// pool.swap is called with msg.sender = router.
// SwapAllowlistExtension checks allowedSwapper[pool][router] → true → passes.
// Attacker receives token1 output; allowlist guard is fully bypassed.
```

Foundry test plan: deploy pool with `SwapAllowlistExtension`, allowlist only the router, assert that a non-allowlisted EOA calling `router.exactInputSingle` succeeds, and assert that the same EOA calling `pool.swap` directly reverts with `NotAllowedToSwap`.
