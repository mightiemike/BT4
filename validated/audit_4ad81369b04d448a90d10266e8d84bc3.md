Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the actual swapper, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the direct `msg.sender` of `pool.swap()`. When any user routes through `MetricOmmSimpleRouter`, `sender` resolves to the router contract address. If the router is allowlisted (required for any allowlisted user to use the router), every unprivileged user can bypass the allowlist by routing through the router. This is a complete admin-boundary break: the pool admin's configured access control is rendered ineffective for all router-mediated swaps.

## Finding Description

`SwapAllowlistExtension.beforeSwap` performs the allowlist check against the `sender` parameter:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension's caller) and `sender` is whatever address the pool received as `msg.sender` when `swap()` was called. In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← this is the router, not the end-user
    ...
``` [2](#0-1) 

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()`, the pool's `msg.sender` is the router contract:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
``` [3](#0-2) 

So the allowlist check resolves to `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`. The router passes `params.extensionData` through verbatim from the caller with no encoding of the originating user's identity, so there is no existing mechanism to recover the real caller.

The asymmetry with `DepositAllowlistExtension` confirms the bug. The deposit allowlist correctly checks `owner` — an explicit parameter that is invariant to router indirection — which remains the real user even when `MetricOmmPoolLiquidityAdder` is used:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L38
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
``` [4](#0-3) 

The swap allowlist has no equivalent "actual user" field — it only sees `sender = router`. The `addLiquidity` function passes `owner` explicitly as a separate parameter through the call chain, while `swap` has no such separate originating-user field. [5](#0-4) 

## Impact Explanation

This is an admin-boundary break. The pool admin configures `SwapAllowlistExtension` to restrict swap access to a specific set of addresses. The bypass requires no special privileges — any address can call `MetricOmmSimpleRouter`. The pool admin is forced into an impossible choice: if the router is not allowlisted, all router-mediated swaps revert (allowlisted users cannot use the router at all); if the router is allowlisted (the only way to let any allowlisted user use the router), every user — allowlisted or not — can bypass the gate. The allowlist is completely ineffective for router-mediated swaps, allowing unauthorized parties to execute swaps on pools intended to be restricted.

## Likelihood Explanation

This surfaces in any deployment where a pool is configured with `SwapAllowlistExtension` to restrict swap access and the pool admin allowlists the router so that legitimate allowlisted users can trade via the standard periphery. Both conditions are expected in normal production use of the extension. The bypass requires no special privileges, no front-running, and no special token behavior — any address can call `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) on the restricted pool.

## Recommendation

`SwapAllowlistExtension` must gate on the actual end-user identity, not the direct caller. Since the pool's `beforeSwap` hook does not carry a separate originating-user field, the recommended fix is to require the actual user's identity to be passed in `extensionData`. The router should be modified to encode the original `msg.sender` into `extensionData` before calling `pool.swap()`, and the extension should decode and verify that address against the allowlist instead of `sender`. Alternatively, adopt the same pattern as `DepositAllowlistExtension` by adding an explicit originating-user parameter to the swap call chain, analogous to how `owner` is threaded through `addLiquidity`.

## Proof of Concept

```solidity
// Pool admin sets up SwapAllowlistExtension: only Alice is allowed to swap.
allowlistExt.setAllowedToSwap(pool, alice, true);

// Pool admin also allowlists the router so Alice can use it.
allowlistExt.setAllowedToSwap(pool, address(router), true);

// Mallory (not allowlisted) calls the router directly.
// The router calls pool.swap() → pool's msg.sender = router.
// beforeSwap receives sender = router.
// allowedSwapper[pool][router] == true → swap succeeds.
vm.prank(mallory);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: mallory,
    tokenIn: token0,
    zeroForOne: true,
    amountIn: 1_000e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// Mallory's swap succeeds despite not being on the allowlist.
```
