Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Allowlist Bypass — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the address that pays tokens via callback) and validates the caller-controlled `owner` parameter against the allowlist instead. Any unprivileged actor can bypass the deposit allowlist by calling `pool.addLiquidity(owner = allowlistedAddress, …)`, injecting liquidity into a restricted pool while assigning LP shares to an allowlisted address.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the LP-position recipient to the extension hook:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first parameter (`sender`) is unnamed and ignored. The guard checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [2](#0-1) 

`owner` is a free parameter supplied by the caller of `addLiquidity` — it is never required to equal `msg.sender` (unlike `removeLiquidity`, which enforces `msg.sender == owner`). The actual token payment flows through the `IMetricOmmModifyLiquidityCallback` invoked on `msg.sender` (the `sender`), not on `owner`. Therefore the allowlist check is performed on the wrong address.

`SwapAllowlistExtension.beforeSwap` correctly checks `sender`:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

The asymmetry between the two allowlist extensions confirms that checking `owner` in `DepositAllowlistExtension` is unintended.

## Impact Explanation
The deposit allowlist is the primary access-control mechanism for restricted pools. Its bypass constitutes an admin-boundary break: an unprivileged actor circumvents an admin-set guard with direct LP-asset consequences. Any EOA or contract can inject liquidity into a pool the admin intended to restrict (e.g., for compliance, LP-composition, or anti-dilution reasons). The unauthorized `sender` pays the tokens; the allowlisted `owner` receives unsolicited LP shares they did not request. The admin-configured guard is rendered entirely ineffective.

## Likelihood Explanation
Exploitation requires no special privilege, no flash loan, and no oracle manipulation. Any EOA or contract can call `pool.addLiquidity(owner = knownAllowlistedAddress, …)`. Allowlisted addresses are discoverable on-chain via the `AllowedToDepositSet` event or by reading `allowedDepositor`. The attack is repeatable at will.

## Recommendation
Replace the `owner` check with a `sender` check, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

## Proof of Concept
1. Pool admin deploys a pool with `DepositAllowlistExtension` configured in `beforeAddLiquidity`.
2. Admin calls `setAllowedToDeposit(pool, Alice, true)` — Alice is the only allowlisted address.
3. Bob (not allowlisted) calls `pool.addLiquidity(owner = Alice, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(Bob, Alice, …)` → extension receives `sender = Bob` (ignored), `owner = Alice`.
5. Extension evaluates `allowedDepositor[pool][Alice]` → `true` → no revert.
6. Pool processes the liquidity addition; Bob's callback is invoked and Bob pays the tokens.
7. Alice's LP position is credited with the new shares.
8. Bob has successfully deposited into a pool he is not allowlisted for; the allowlist is fully bypassed. [4](#0-3) [5](#0-4)
