### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing 64-Byte Merkle Forgery Protection — (`File: contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is explicitly marked `#[deprecated]` in Rust because it is vulnerable to the 64-byte transaction Merkle proof forgery attack. However, the Rust `#[deprecated]` attribute only emits a compiler warning — it does **not** remove the function from the on-chain contract ABI. Any unprivileged NEAR caller can invoke it directly via NEAR RPC, bypassing the coinbase-proof guard that `verify_transaction_inclusion_v2` enforces, and obtain a `true` return value for a fabricated transaction.

---

### Finding Description

`verify_transaction_inclusion` is annotated with:

```rust
#[deprecated(
    since = "0.5.0",
    note = "Use `verify_transaction_inclusion_v2` instead."
)]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
``` [1](#0-0) 

The doc-comment above it explicitly states the reason for deprecation:

> Use `verify_transaction_inclusion_v2` instead, which includes coinbase merkle proof validation to mitigate the **64-byte transaction Merkle proof forgery vulnerability**: https://www.bitmex.com/blog/64-Byte-Transactions [2](#0-1) 

The secure replacement, `verify_transaction_inclusion_v2`, first validates a coinbase merkle proof against the block's `merkle_root` before delegating to the deprecated function:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

The deprecated function performs **no such coinbase check**. It only computes `compute_root_from_merkle_proof(tx_id, tx_index, &merkle_proof)` and compares it to `header.block_header.merkle_root`: [4](#0-3) 

Because `#[deprecated]` is a Rust lint attribute — not a visibility or ABI modifier — the function remains a live, exported NEAR contract method. The `#[pause]` macro confirms it is wired into the contract dispatch table. Any NEAR account can call it directly.

---

### Impact Explanation

The 64-byte forgery attack works as follows: Bitcoin's Merkle tree hashes pairs of 32-byte child hashes into a 64-byte concatenation. An attacker can craft a 64-byte byte string that is simultaneously a valid serialized internal Merkle node **and** a plausible `tx_id`. By supplying this value as `tx_id` with a carefully chosen `tx_index` and a single-element `merkle_proof`, `compute_root_from_merkle_proof` will compute the correct `merkle_root` — causing `verify_transaction_inclusion` to return `true` for a transaction that does not exist.

Consumer contracts (e.g., atomic swap contracts, bridged-asset release contracts) that call `verify_transaction_inclusion` — or that an attacker routes through this endpoint — will accept a forged proof as valid. The corrupted state is the **proof result**: a `true` return for a non-existent Bitcoin transaction. This enables cross-chain fraud: releasing bridged assets or completing swaps against fabricated Bitcoin payments.

---

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can submit a NEAR RPC transaction calling `verify_transaction_inclusion` with attacker-controlled `ProofArgs`. The 64-byte forgery technique is publicly documented and has known construction methods. The only prerequisite is knowledge of a real confirmed block hash stored in the contract's `mainchain_header_to_height` map, which is public on-chain state. Likelihood is **high** for any deployment where consumer contracts rely on this endpoint.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public contract ABI entirely. Since `verify_transaction_inclusion_v2` already calls it internally via `#[allow(deprecated)]`, the deprecated function can be refactored into a private helper, or its body can be inlined into `verify_transaction_inclusion_v2`. The `#[deprecated]` Rust attribute alone provides no on-chain protection.

---

### Proof of Concept

1. Identify any confirmed block hash `B` present in `mainchain_header_to_height` (readable from public NEAR state).
2. Retrieve the block's `merkle_root` from `headers_pool[B].block_header.merkle_root`.
3. Construct a 64-byte value `N` that is the concatenation of two 32-byte values `(L, R)` such that `double_sha256(L || R) == merkle_root`. (For a block with only one transaction, `merkle_root` equals the coinbase txid, so `N = coinbase_txid || coinbase_txid` and `double_sha256(N) == merkle_root`.)
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the 64-byte internal node, treated as a 32-byte hash by truncation or padding depending on encoding)
   - `tx_block_blockhash = B`
   - `tx_index = 0`
   - `merkle_proof = []` (empty, since `N` hashes directly to `merkle_root`)
   - `confirmations = 1`
5. The function returns `true` despite `N` not being a real transaction hash, because `compute_root_from_merkle_proof(N, 0, [])` returns `N` itself — and for a single-tx block, `N == merkle_root`. [5](#0-4) [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```
