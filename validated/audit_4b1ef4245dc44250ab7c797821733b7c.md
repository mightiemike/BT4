Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks position recipient (`owner`) instead of actual depositor, allowing allowlist bypass — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument and gates on `owner` (the position recipient). Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts any non-zero `owner` from the caller, an unprivileged user can bypass the deposit allowlist by naming an allowlisted address as `owner`, depositing tokens into a curated pool without being on the allowlist themselves.

## Finding Description

The extension's `beforeAddLiquidity` ignores the first positional argument (`sender`) and checks only `owner`: [1](#0-0) 

The check `allowedDepositor[msg.sender][owner]` uses `msg.sender` as the pool address and `owner` as the key — meaning it asks "is the position recipient allowlisted?" not "is the depositor allowlisted?".

`MetricOmmPool.addLiquidity` passes `msg.sender` (the liquidity adder contract) as `sender` and the caller-supplied `owner` as `owner`: [2](#0-1) 

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both arguments to the extension: [3](#0-2) 

`addLiquidityExactShares` accepts any non-zero `owner` from the caller — `_validateOwner` only checks `owner != address(0)`: [4](#0-3) 

**Full attack path:**
1. Pool admin deploys a curated pool with `DepositAllowlistExtension` and allowlists `userA`.
2. `userB` (not allowlisted) calls `liquidityAdder.addLiquidityExactShares(pool, owner=userA, ...)`.
3. The adder calls `pool.addLiquidity(owner=userA, ...)` — `msg.sender` to the pool is the adder contract.
4. The pool calls `_beforeAddLiquidity(sender=liquidityAdder, owner=userA, ...)`.
5. The extension evaluates `allowedDepositor[pool][userA]` → `true` → does not revert.
6. `userB`'s tokens are pulled in the callback via the transient `payer` context; `userA` receives LP shares.
7. The pool has accepted a deposit from a non-allowlisted actor.

The same path applies to `addLiquidityWeighted` with an explicit `owner` argument: [5](#0-4) 

The probe call also passes `owner` to the extension, so the allowlist check passes during the probe, and the subsequent paying call passes identically.

## Impact Explanation
The deposit allowlist is rendered ineffective. Any unprivileged user can deposit into a curated pool by naming an allowlisted address as `owner`. The pool receives tokens from a non-allowlisted actor, violating the pool admin's access-control policy. For KYC-gated or regulated pools this constitutes a direct admin-boundary break: the restriction the pool admin configured is bypassed by a standard periphery call with no elevated privilege. This matches the allowed impact gate: **admin-boundary break where pool admin's access control is bypassed by an unprivileged path**.

## Likelihood Explanation
Medium-High. The bypass requires no admin cooperation, no special privilege, and no unusual conditions — only the ability to call `addLiquidityExactShares` with an allowlisted address as `owner`. The attack is repeatable, requires no front-running, and works on any pool using `DepositAllowlistExtension` without `allowAllDepositors` set. Off-chain coordination between `userB` (payer) and `userA` (owner/recipient) makes the bypass fully exploitable for economic gain.

## Recommendation
The extension must check the actual depositor, not the position recipient. Since `sender` in the extension call is the liquidity adder (not the original EOA), the correct fix depends on the intended trust model:

1. **Check `sender` instead of `owner`**: Change the allowlist key from `owner` to the first argument (`sender`). This gates on the direct caller of the pool (the router/adder), which is appropriate if the adder is trusted to represent the real user.
2. **Pass the real payer via `extensionData`**: Have the liquidity adder encode the real payer in `extensionData` and have the extension decode and check it (requires the extension to trust the adder).
3. **Restrict direct pool access**: Require that only allowlisted addresses call `pool.addLiquidity` directly (bypassing the adder entirely for curated pools).

The minimal fix is to replace `owner` with the first positional argument in the allowlist check:
```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

```solidity
// Foundry test sketch
function test_allowlistBypass() public {
    // Setup: pool with DepositAllowlistExtension, userA allowlisted, userB not
    extension.setAllowedToDeposit(pool, userA, true);

    // userB has tokens and has approved the liquidity adder
    vm.startPrank(userB);
    token0.approve(address(adder), type(uint256).max);
    token1.approve(address(adder), type(uint256).max);

    // userB calls addLiquidityExactShares with owner=userA
    // Extension checks allowedDepositor[pool][userA] == true → passes
    // userB's tokens are pulled, userA gets LP shares
    adder.addLiquidityExactShares(pool, userA, salt, deltas, max0, max1, "");
    vm.stopPrank();

    // Assert: pool received tokens from userB (non-allowlisted)
    // userA holds the LP position
    // No revert occurred despite userB not being allowlisted
}
```
