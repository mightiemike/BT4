Audit Report

## Title
`SwapAllowlistExtension` binds the allowlist check to the router address instead of the originating user, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the pool's own `msg.sender` — the router contract when a user swaps through `MetricOmmSimpleRouter`. The extension therefore evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actualUser]`. Any pool admin who allowlists the router to support standard periphery UX inadvertently grants every public user unrestricted access to the curated pool, defeating the entire purpose of the extension.

## Finding Description

**Root cause — wrong actor bound to `sender`:**

`MetricOmmPool.swap` passes its own `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged into the encoded call to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When the call originates from `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly: [4](#0-3) 

The router does **not** encode the originating user's address into `extensionData` or any other field. The pool therefore sees `msg.sender = router`, passes `sender = router` to the extension, and the check resolves to `allowedSwapper[pool][router]` — never touching the actual user's address.

**Bypass exploit path:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict trading to KYC-verified addresses.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` so that permitted users can swap through the standard periphery path.
3. Non-allowlisted attacker calls `MetricOmmSimpleRouter.exactInputSingle({pool: restrictedPool, ...})`.
4. Router calls `pool.swap(recipient=attacker, ...)` — pool's `msg.sender` is the router.
5. Pool calls `_beforeSwap(sender=router, ...)` → extension evaluates `allowedSwapper[pool][router]` → `true`.
6. Swap executes; attacker receives output tokens from a pool they are explicitly barred from.

**Alternate broken-functionality scenario:** If the pool admin does *not* allowlist the router, then even explicitly permitted users cannot swap through the router, because the extension sees `sender = router` and rejects it. The allowlist becomes entirely incompatible with the supported periphery path.

## Impact Explanation

A non-allowlisted user can execute swaps on a curated pool (e.g., KYC-gated, institutional, or regulatory-restricted) by routing through the public `MetricOmmSimpleRouter`. This is a direct, fund-impacting bypass of the pool's access-control policy: the attacker receives swap output they are not entitled to, and LP providers are exposed to counterparties the pool admin explicitly excluded. This matches the "admin-boundary break" impact class — an unprivileged path bypasses a pool admin's access control — and constitutes a broken core pool functionality causing loss of policy enforcement with direct fund-flow consequences on every curated pool that allowlists the router.

## Likelihood Explanation

The trigger requires only that the pool admin allowlists the router — the natural and expected configuration for any curated pool that intends to support the standard periphery UX. No privileged access, no malicious setup, and no non-standard tokens are needed. Any public user can execute the bypass with a single `exactInputSingle` call. The condition is not hypothetical; it is the only way to make the extension compatible with the router at all.

## Recommendation

The extension must check the **originating user**, not the intermediary router. Two complementary fixes:

1. **Router-attested identity via `extensionData`:** Have `MetricOmmSimpleRouter` encode `msg.sender` (the real user) into `extensionData` before calling `pool.swap`. `SwapAllowlistExtension.beforeSwap` decodes this value when `sender` is a recognized router and checks it against the allowlist instead of `sender`.

2. **Protocol-level convention:** Define a standard encoding for "real initiator" in `extensionData` that all periphery contracts must populate, and have the extension always prefer that value over `sender` when present.

The simplest safe fix is option 1: in `MetricOmmSimpleRouter.exactInputSingle` (and all other swap entry points), prepend `abi.encode(msg.sender)` to `params.extensionData`, and in `SwapAllowlistExtension.beforeSwap`, decode and verify that address when `sender` is a known router.

## Proof of Concept

```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension
  pool admin: setAllowedToSwap(pool, router, true)   // to support router UX
  pool admin: setAllowedToSwap(pool, alice, true)    // alice is KYC'd
  bob is NOT allowlisted

Attack:
  bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(recipient=bob, ...)          [msg.sender = router]
    → pool calls _beforeSwap(sender=router, ...)
    → ExtensionCalling forwards sender=router to extension
    → SwapAllowlistExtension checks allowedSwapper[pool][router] → true
    → swap executes; bob receives output tokens

Result:
  bob, who is explicitly excluded from the curated pool,
  successfully swaps by routing through the public router.
  allowedSwapper[pool][bob] is never consulted.
```
