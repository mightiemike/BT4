Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates on the router address instead of the originating user, defeating per-user swap allowlist enforcement â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` which is `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is the direct caller of `pool.swap()`, so `sender = router`. The extension then evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actualUser]`. This produces two fund-impacting outcomes: (1) if the pool admin allowlists the router to enable periphery-based swaps, every non-allowlisted user can bypass the per-user gate by routing through the router; (2) if the admin allowlists individual users but not the router, those users are blocked from using the standard periphery path even though they are allowlisted.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- direct caller of pool.swap()
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap()` forwards that value verbatim to every configured extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap` gates on that `sender`:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly without forwarding the originating user:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
```

Since `msg.sender` of `pool.swap()` is the router, `sender` arriving at the extension is the router address. The check `allowedSwapper[pool][router]` is evaluated instead of `allowedSwapper[pool][user]`. The same mismatch applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates on `owner` (the actual position owner explicitly passed through the call stack), not on `sender`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L38
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
```

The swap path has no analogous explicit `swapper` parameter â€” the pool's `swap()` signature carries no originating-user field, so the extension has no way to recover the true economic actor.

## Impact Explanation
Two concrete fund-impacting outcomes arise:

1. **Allowlist bypass (primary):** A pool admin who calls `setAllowedToSwap(pool, router, true)` â€” the natural step to enable periphery-based swaps on a curated pool â€” inadvertently opens the gate to every user. Any non-allowlisted address calls `router.exactInputSingle(...)`, the extension sees `sender = router` (allowlisted), and the swap executes. The per-user curation is completely defeated for all router-routed swaps. This constitutes broken core pool functionality: the allowlist invariant that only approved addresses may swap is violated.

2. **Broken core functionality (secondary):** A pool admin who allowlists individual user addresses (the intended usage) blocks those same users from swapping through the router, because the router address is not in the allowlist. The standard periphery path becomes unusable for allowlisted users, constituting an unusable swap flow.

Both outcomes violate the invariant that a curated pool must enforce the same allowlist policy regardless of which supported public entrypoint reaches it.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary supported swap entrypoint in the periphery layer. Any pool that deploys `SwapAllowlistExtension` and expects users to interact via the router will encounter this bug. The bypass path requires no special privilege â€” calling the router is the normal user flow. Any pool admin who enables the extension and allowlists the router (or individual users) triggers one of the two broken outcomes immediately.

## Recommendation
The `beforeSwap` hook must gate on the actual economic actor, not the intermediary. The preferred fix mirrors the `addLiquidity` pattern: add an explicit `swapper` parameter to `pool.swap()` (analogous to `owner` in `addLiquidity`) so the router can forward the originating user address (`msg.sender` at the router level). The extension then checks `allowedSwapper[pool][swapper]`. As a short-term mitigation, document that `SwapAllowlistExtension` is incompatible with router-based flows and require direct `pool.swap()` calls only.
