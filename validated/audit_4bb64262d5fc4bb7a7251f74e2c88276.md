### Title
Fee-on-Transfer / Deflation Token Deposit Overcredits Subaccount Balance — (File: `core/contracts/EndpointStorage.sol`, `core/contracts/Endpoint.sol`, `core/contracts/Clearinghouse.sol`)

### Summary
`handleDepositTransfer` performs two sequential ERC-20 transfers (user → Endpoint, Endpoint → Clearinghouse) using the caller-supplied `amount`, then queues a slow-mode transaction that credits the same `amount` to the subaccount. For fee-on-transfer / deflation tokens, each hop burns a fee, so the Clearinghouse receives strictly less than `amount`, yet the subaccount is credited the full `amount`. This creates a permanent, compounding solvency deficit in the Clearinghouse.

### Finding Description

The deposit flow spans three files:

**Step 1 — `EndpointStorage.handleDepositTransfer`** performs two transfers with the raw `amount`:

```solidity
// core/contracts/EndpointStorage.sol  lines 111-119
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);          // hop 1: user → Endpoint
    safeTransferTo(token, address(clearinghouse), amount); // hop 2: Endpoint → Clearinghouse
}
``` [1](#0-0) 

**Step 2 — `Endpoint.depositCollateralWithReferral`** queues a slow-mode transaction encoding the original `amount`:

```solidity
// core/contracts/Endpoint.sol  lines 144-165

// ...

``` [2](#0-1) 

**Step 3 — `Clearinghouse.depositCollateral`** credits `txn.amount` (the original value) unconditionally:

```solidity
// core/contracts/Clearinghouse.sol  lines 205-208
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
``` [3](#0-2) 

No balance-before/after check is performed at any step. The protocol never measures what the Clearinghouse actually received.

### Impact Explanation

For a deflation token with per-transfer fee `f`:
- Hop 1 (`user → Endpoint`): Endpoint receives `amount − f`
- Hop 2 (`Endpoint → Clearinghouse`): Clearinghouse receives `amount − 2f` (assuming Endpoint has enough residual balance to cover the shortfall; otherwise the tx reverts, but if Endpoint holds any prior balance the transfer succeeds)
- Slow-mode tx credits `amount` to the subaccount

Each deposit creates a deficit of `2f` tokens in the Clearinghouse. Over many deposits the Clearinghouse's real token balance falls below the sum of all credited subaccount balances. Later withdrawals drain the pool until it is empty, leaving the last withdrawers unable to redeem their credited collateral. This is a direct, quantifiable solvency / accounting corruption. [4](#0-3) 

### Likelihood Explanation

Any deflation or fee-on-transfer ERC-20 token listed as a supported spot product triggers this path. The entry point (`depositCollateral` / `depositCollateralWithReferral`) is fully permissionless — any user can call it. No privileged role is required. The bug fires on every single deposit of such a token, making exploitation automatic and cumulative. [5](#0-4) 

### Recommendation

Measure the Clearinghouse's actual token balance before and after the transfer and use the delta as the credited amount:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    uint256 actualReceived = token.balanceOf(address(clearinghouse)) - before;
    // pass actualReceived (not amount) into the slow-mode DepositCollateral struct
}
```

Alternatively, explicitly document and enforce that deflation / fee-on-transfer tokens are not supported as collateral assets, and add a token-registration guard that rejects such tokens. [1](#0-0) 

### Proof of Concept

1. A deflation token `DFT` (1% burn per transfer) is registered as a spot product.
2. Alice calls `depositCollateral("alice", productId, 1000e18)`.
3. `handleDepositTransfer` executes:
   - `safeTransferFrom(DFT, Alice, 1000e18)` → Endpoint receives `990e18` (1% burned).
   - `safeTransferTo(DFT, clearinghouse, 1000e18)` → if Endpoint has ≥ `10e18` residual DFT, Clearinghouse receives `990e18` (another 1% burned); Endpoint's residual is drained by `10e18`.
4. Slow-mode tx is queued with `amount = 1000e18`.
5. After the 3-day delay, `Clearinghouse.depositCollateral` runs: `amountRealized = 1000e18 * multiplier`; `spotEngine.updateBalance(productId, alice, 1000e18_scaled)`.
6. Clearinghouse holds `990e18` DFT but has credited `1000e18` to Alice — a `10e18` deficit per deposit.
7. Repeated deposits compound the deficit until the Clearinghouse cannot honor all withdrawals. [6](#0-5) [7](#0-6)

    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
