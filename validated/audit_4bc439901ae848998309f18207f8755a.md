Audit Report

## Title
`SwapAllowlistExtension` Per-User Allowlist Bypassed via `MetricOmmSimpleRouter` — Router Address Substitutes for Actual User Identity - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, the pool receives `msg.sender = router`, so the extension checks `allowedSwapper[pool][router]` — not the actual user's address. Any pool admin who allowlists the router (required for legitimate users to use it) simultaneously grants every user on the network the ability to bypass the per-user allowlist, completely defeating the access-control invariant of the extension.

## Finding Description

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool and uses `msg.sender` (the calling pool) as the mapping key:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

`sender` is populated by the pool as its own `msg.sender` at the time `swap` is called:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,  // ← whoever called pool.swap()
    ...
);
``` [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly with no user identity forwarded in `extensionData` (it passes `params.extensionData` which is user-supplied and not validated to contain the real caller):

```solidity
// MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData  // ← user-controlled, not enforced to contain real caller
    );
``` [3](#0-2) 

Inside the pool, `msg.sender` is the router. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The actual initiating user's address is never consulted. The pool admin faces an inescapable dilemma: not allowlisting the router blocks all legitimate users from using it, while allowlisting the router grants every address on the network the ability to bypass the per-user allowlist. There is no configuration that simultaneously allows legitimate users to use the router and blocks non-allowlisted users.

The `onlyPool` modifier on `beforeSwap` (via `BaseMetricExtension`) only verifies that the caller is a registered pool — it does not validate the identity of the original swap initiator. [4](#0-3) 

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict swaps to specific counterparties (e.g., KYC-verified addresses, institutional partners, or whitelisted market makers) is completely ineffective once the router is allowlisted. Any unprivileged user can call `MetricOmmSimpleRouter.exactInputSingle` or `exactInput` targeting the restricted pool and execute swaps that the allowlist was designed to block. This breaks the core pool access-control invariant and allows unauthorized parties to interact with restricted liquidity pools, constituting a broken core pool functionality with direct fund-impact potential (unauthorized drain or manipulation of restricted liquidity).

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface for the protocol. Any pool admin who deploys a `SwapAllowlistExtension` and also wants legitimate allowlisted users to be able to use the standard router must allowlist the router address — this is the expected operational configuration. The bypass is therefore reachable in every realistic deployment of the extension with the router, requires no special privileges, and is repeatable by any address.

## Recommendation

The extension must resolve the actual initiating user's identity rather than relying solely on `sender` (which collapses to the router for all router-mediated swaps). Two viable approaches:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData` in a standardized, authenticated way; the extension decodes and checks it. This requires a coordinated change to the router and extension, and the extension must reject calls where `sender` is a known router but `extensionData` does not contain a valid user identity.

2. **Router registry with `extensionData` fallback**: If `sender` is a known router, the extension reads the real user from `extensionData`; otherwise it checks `sender` directly. The extension should revert if `sender` is a known router and `extensionData` is empty or malformed.

## Proof of Concept

```solidity
// Pool admin sets up allowlist: only Alice may swap
swapAllowlist.setAllowedToSwap(pool, alice, true);
// Admin also allowlists the router so Alice can use it
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Charlie (not allowlisted) bypasses the guard via the router
vm.prank(charlie);
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool:             address(pool),
    tokenIn:          address(token0),
    tokenOut:         address(token1),
    zeroForOne:       true,
    amountIn:         1000,
    amountOutMinimum: 0,
    recipient:        charlie,
    deadline:         block.timestamp + 1,
    priceLimitX64:    0,
    extensionData:    ""
}));
// ✓ swap succeeds — allowlist check passed because msg.sender in pool == router (allowlisted)
// Charlie swapped in a pool he was explicitly barred from
```
