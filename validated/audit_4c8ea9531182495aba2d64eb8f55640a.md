### Title
Duplicate-last-leaf position aliasing in `compute_root_from_merkle_proof` allows phantom `tx_index` to verify — (`merkle-tools/src/lib.rs`)

### Summary
`compute_root_from_merkle_proof` uses only the parity of `transaction_position` at each level to decide left/right placement. It has no knowledge of the actual transaction count in the block. For any odd-width Bitcoin Merkle tree, the last real leaf T_N and the phantom duplicate position N+1 produce identical intermediate hashes (`H(T_N, T_N)` vs `H(T_N, T_N)`), so the same proof bytes verify against the same root for two distinct `tx_index` values. `verify_transaction_inclusion` passes the caller-supplied `tx_index` directly into this function with no bounds check.

### Finding Description

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` computes the Merkle root purely from `(transaction_hash, transaction_position, merkle_proof)`: [1](#0-0) 

At each level it branches on `current_position % 2`:
- Even → `H(current_hash, proof_hash)`
- Odd → `H(proof_hash, current_hash)`

For the last leaf T_N in an odd-width tree, the sibling in the proof is T_N itself (the Bitcoin duplicate). At the real index N (even), the step computes `H(T_N, T_N)`. At the phantom index N+1 (odd), the step computes `H(T_N, T_N)` — identical, because both arguments are T_N. All subsequent levels are therefore identical, and the function returns the same root R for both positions.

`verify_transaction_inclusion` passes the caller-supplied `args.tx_index` directly to this function: [2](#0-1) 

There is no check that `tx_index` is within the actual transaction count of the block. The contract stores only the 80-byte header (which contains the Merkle root) and has no record of how many transactions a block contains.

`verify_transaction_inclusion_v2` adds a coinbase-proof check but ultimately delegates to the same vulnerable function: [3](#0-2) 

### Impact Explanation

A downstream bridge, unlock, mint, or withdrawal contract that:
1. Calls `verify_transaction_inclusion` (or v2) to confirm a BTC deposit/event, and
2. Uses `(tx_id, tx_index)` as a replay-protection key (a common and reasonable pattern)

…can be double-spent. The attacker submits the real proof for T_N at index N to claim the first payout, then submits the same proof with index N+1 to claim a second payout. Both calls return `true` against the same on-chain Merkle root.

The deprecated `verify_transaction_inclusion` is still a live public `#[pause]`-gated entry point callable by any unprivileged account. [4](#0-3) 

### Likelihood Explanation

- Requires only a BTC block with an odd transaction count (the majority of Bitcoin blocks have an odd number of transactions).
- The attacker needs only the real Merkle proof for the last transaction, which is publicly available from any Bitcoin node.
- No privileged role, no relayer compromise, no leaked keys.
- The call is permissionless; any NEAR account can invoke `verify_transaction_inclusion`.

### Recommendation

1. **Bounds-check `tx_index`**: The contract must reject any `tx_index` that equals or exceeds the actual transaction count of the block. This requires storing or passing the transaction count alongside the proof (e.g., as part of `ProofArgs`).
2. **Alternatively, enforce that the sibling proof element at the last level of an odd-width tree is never equal to `transaction_hash`**: if `proof[0] == tx_id` and `tx_index` is even, the position is the last real leaf; reject `tx_index + 1` as a valid claim for the same `tx_id`.
3. **Deprecate `verify_transaction_inclusion` more aggressively**: gate it behind a role or remove it entirely, since `verify_transaction_inclusion_v2` is the intended replacement and still inherits this bug.
