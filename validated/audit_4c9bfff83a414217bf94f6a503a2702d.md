### Title
`verify_transaction_inclusion_v2` Always Panics for Single-Transaction Blocks Due to Downstream Empty-Proof Rejection - (`File: contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion_v2` (the recommended, non-deprecated SPV path) delegates to `verify_transaction_inclusion` (v1) after performing its coinbase-anchor check. The v1 function contains a hard `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard. For any block containing exactly one transaction (the coinbase), the merkle proof is legitimately empty — the coinbase hash *is* the merkle root. The v2 function never validates that the proof is non-empty before forwarding it to v1, so every call to `verify_transaction_inclusion_v2` for a single-transaction block unconditionally panics. No alternative non-deprecated path exists.

### Finding Description

`verify_transaction_inclusion_v2` performs two steps before delegating to v1:

1. It checks `args.merkle_proof.len() == args.coinbase_merkle_proof.len()` — satisfied trivially when both are empty.
2. It checks that `compute_root_from_merkle_proof(coinbase_tx_id, 0, &coinbase_merkle_proof) == header.block_header.merkle_root`. For a single-tx block, `merkle_root == coinbase_tx_id`, and `compute_root_from_merkle_proof` with an empty proof returns its first argument unchanged, so this check also passes. [1](#0-0)

Because the proof is empty, this `require!` always panics. The v2 function never inserts a non-empty guard of its own before the delegation. [3](#0-2) 

The empty-proof return path in `compute_root_from_merkle_proof` (loop body never executes, returns `transaction_hash` unchanged) is what makes the coinbase anchor check silently pass, masking the downstream incompatibility.

### Impact Explanation

Any NEAR caller or downstream contract invoking `verify_transaction_inclusion_v2` for a transaction in a single-transaction block receives an unconditional panic rather than a boolean result. Because v1 is marked `#[deprecated]` and v2 is the documented replacement, there is no supported non-panicking path for this class of proof. Downstream contracts that gate fund releases or cross-chain actions on a `true` result from this function are permanently blocked for single-tx blocks. [4](#0-3) 

### Likelihood Explanation

Single-transaction blocks (containing only the coinbase) occur on all supported chains during periods of low mempool activity and are valid by consensus rules. A relayer or downstream application that submits such a block and then attempts to verify the coinbase transaction — a pattern used in mining-reward verification and certain bridge designs — will trigger this path deterministically. No privileged access is required; any NEAR account can call `verify_transaction_inclusion_v2`.

### Recommendation

Add an explicit non-empty proof guard inside `verify_transaction_inclusion_v2` before delegating to v1, or handle the empty-proof case directly in v2 (returning `tx_id == header.block_header.merkle_root` when both proofs are empty). Alternatively, lift the `require!(!args.merkle_proof.is_empty())` guard out of v1 and into a shared validation layer that both entry points use, so the constraint is applied consistently and only where it is semantically required.

### Proof of Concept

1. Submit a block containing exactly one transaction (the coinbase) so that `merkle_root == coinbase_tx_id`.
2. Call `verify_transaction_inclusion_v2` with:
   - `tx_id = coinbase_tx_id`
   - `tx_block_blockhash` = hash of that block
   - `tx_index = 0`
   - `merkle_proof = []`
   - `coinbase_tx_id = coinbase_tx_id`
   - `coinbase_merkle_proof = []`
   - `confirmations = 1`
3. Execution path:
   - `merkle_proof.len() == coinbase_merkle_proof.len()` → `0 == 0` ✓
   - `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id == merkle_root` ✓
   - `verify_transaction_inclusion` is called; `require!(!args.merkle_proof.is_empty())` fires → **panic: "Merkle proof is empty"** [1](#0-0) [5](#0-4)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
