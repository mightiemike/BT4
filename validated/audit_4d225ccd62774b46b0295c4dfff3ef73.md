### Title
`DelegateAction` Signed Hash Lacks Chain-ID Domain Separation, Enabling Cross-Network Replay — (File: `core/primitives/src/action/delegate.rs`)

---

### Summary

`DelegateAction::get_nep461_hash()` produces a signing payload of `sha256(borsh(MessageDiscriminant_366 || DelegateAction))`. The `DelegateAction` struct contains `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key` — but **no chain identifier**. A signature produced on testnet is byte-for-byte identical to a valid signature on mainnet for the same account, key, and nonce value. A malicious relayer can replay a testnet-signed `DelegateAction` on mainnet.

---

### Finding Description

`DelegateAction::get_nep461_hash()` constructs a `SignableMessage` whose only domain tag is a `MessageDiscriminant` encoding the NEP number (366 for V1, 611 for V2):

```rust
// core/primitives/src/action/delegate.rs:353-357
pub fn get_nep461_hash(&self) -> CryptoHash {
    let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
    let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
    hash(&bytes)
}
```

`SignableMessage` is `borsh(discriminant: u32 || msg: &DelegateAction)`:

```rust
// core/primitives/src/signable_message.rs:61-65
pub struct SignableMessage<'a, T> {
    pub discriminant: MessageDiscriminant,
    pub msg: &'a T,
}
```

The `MessageDiscriminant` is computed as `MIN_ON_CHAIN_DISCRIMINANT + 366 = 0x4000_016E`:

```rust
// core/primitives/src/signable_message.rs:221-223
SignableMessageType::DelegateAction => {
    MessageDiscriminant::new_on_chain(NEP_366_META_TRANSACTIONS).unwrap()
}
```

The `DelegateAction` struct signed over is:

```rust
// core/primitives/src/action/delegate.rs:46-64
pub struct DelegateAction {
    pub sender_id: AccountId,
    pub receiver_id: AccountId,
    pub actions: Vec<NonDelegateAction>,
    pub nonce: Nonce,
    pub max_block_height: BlockHeight,
    pub public_key: PublicKey,
}
```

**No `chain_id`, genesis hash, or network identifier appears anywhere in the signed bytes.** The discriminant `0x4000_016E` is a global constant identical on mainnet, testnet, and any other NEAR network. `SignedDelegateAction::verify()` checks only the signature against this hash:

```rust
// core/primitives/src/action/delegate.rs:84-90
pub fn verify(&self) -> bool {
    let delegate_action = &self.delegate_action;
    let hash = delegate_action.get_nep461_hash();
    let public_key = &delegate_action.public_key;
    self.signature.verify(hash.as_ref(), public_key)
}
```

The same absence applies to `DelegateActionV2` / `VersionedSignedDelegateAction` (NEP-611, discriminant `0x4000_0263`).

Contrast this with the validator-layer signatures, which are protected by chain-specific data: `PartialEncodedStateWitnessInner` includes `epoch_id` (chain-specific), `ChunkEndorsementInnerV1` includes `chunk_hash` (chain-specific). The `DelegateAction` path has no equivalent binding.

---

### Impact Explanation

A malicious relayer who receives a signed `DelegateAction` from a user on testnet can submit the identical `(DelegateAction, signature)` pair on mainnet. If the user's mainnet access-key nonce satisfies `nonce_in_action > current_mainnet_nonce` and `block_height < max_block_height`, the runtime accepts and executes the action. Possible consequences include:

- Unintended token transfers (draining mainnet funds)
- Unauthorized `AddKey` / `DeleteKey` (account takeover)
- Arbitrary contract calls on mainnet

The runtime's `DelegateAction` processing path performs no chain-binding check beyond signature validity and nonce ordering.

---

### Likelihood Explanation

The conditions for exploitation are realistic:

1. **Same account on both networks** — extremely common; NEAR account names are global strings, and users routinely create the same account on testnet and mainnet.
2. **Same key on both networks** — common; many wallets and SDKs reuse the same ED25519 key pair.
3. **Nonce alignment** — the action's `nonce` must exceed the current mainnet access-key nonce. For a new mainnet account (nonce = 0), any testnet `DelegateAction` with `nonce ≥ 1` qualifies. For accounts with few transactions, alignment is achievable.
4. **`max_block_height` window** — the relayer must act before the block height passes on mainnet; with a typical `max_block_height` set days in the future, this is not a constraint.

---

### Recommendation

Include a chain-binding identifier in the signed payload. The simplest approach is to add the genesis block hash or `chain_id` string to `DelegateAction` (or to `SignableMessage`), making the signed bytes network-specific:

```rust
pub struct DelegateAction {
    pub sender_id: AccountId,
    pub receiver_id: AccountId,
    pub actions: Vec<NonDelegateAction>,
    pub nonce: Nonce,
    pub max_block_height: BlockHeight,
    pub public_key: PublicKey,
    pub chain_id: String,  // e.g. "mainnet", "testnet", or genesis hash
}
```

Alternatively, incorporate the genesis block hash into `SignableMessage` as a second discriminant field, keeping `DelegateAction` unchanged for backward compatibility while making the hash network-specific.

---

### Proof of Concept

1. Alice has `alice.near` on both testnet and mainnet, same ED25519 key, mainnet access-key nonce currently at 4.
2. Alice signs on testnet: `DelegateAction { sender_id: "alice.near", receiver_id: "bob.near", actions: [Transfer(100 NEAR)], nonce: 5, max_block_height: 9_000_000, public_key: alice_key }`.
3. Signed hash = `sha256(borsh(0x4000_016E_u32 || DelegateAction))` — **identical bytes on mainnet**.
4. Malicious relayer wraps this in a mainnet transaction and submits it.
5. Runtime calls `SignedDelegateAction::verify()` → signature valid (same hash, same key).
6. Nonce check: `5 > 4` → passes; nonce advanced to 5 on mainnet.
7. `max_block_height` check: `9_000_000 > current_mainnet_height` → passes.
8. Transfer of 100 NEAR executes on mainnet without Alice's intent.
