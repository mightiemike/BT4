**Step 1 — `processEventsAndClaimBonus` mutates `pos` before the pool check**

Inside `processEventsAndClaimBonus`, the in-memory `pos` pointer is mutated in this order:

- Line 193: `pos.UpdateLastEventSeq(entry.Seq)` — advances `LastEventSeq` for every event processed
- Line 196: `k.decrementEventRefCount(...)` — decrements event reference counts (side-effect on store)
- Line 215: `applyBonusAccrualCheckpoint(&pos.Position, blockTime)` — advances `LastBonusAccrual` to `blockTime`
- Line 217: `pos.UpdateLastKnownBonded(bonded)` — updates bonded state

- Line 230: `k.sufficientBonusPoolBalance(ctx, bonusCoins)` — returns `ErrInsufficientBonusPool` here [1](#0-0)

### Title
Checkpoint Advancement Without Bonus Payment in `slashRedelegationPosition` Causes Permanent Loss of Accrued Bonus Rewards — (`x/tieredrewards/keeper/slash.go`, `claim_rewards.go`)

### Summary
When `BeforeRedelegationSlashed` fires and the `RewardsPoolName` balance is insufficient to cover the owed bonus, `processEventsAndClaimBonus` advances all three bonus checkpoints (`LastBonusAccrual`, `LastEventSeq`, `LastKnownBonded`) in memory before returning `ErrInsufficientBonusPool`. `slashRedelegationPosition` swallows that error and then calls `setPositionWithState` with the already-mutated position, persisting the advanced checkpoints. The bonus is never paid, and the accrual window is permanently closed.

### Finding Description
`processEventsAndClaimBonus` performs all checkpoint mutations on the `pos` pointer before reaching the pool-sufficiency guard:

```
line 193  pos.UpdateLastEventSeq(entry.Seq)          // inside event loop
line 196  k.decrementEventRefCount(...)               // store side-effect
line 215  applyBonusAccrualCheckpoint(&pos.Position, blockTime)
line 217  pos.UpdateLastKnownBonded(bonded)
...
line 230  k.sufficientBonusPoolBalance(ctx, bonusCoins) // ← error returned here
```

When the pool check fails, the function returns `ErrInsufficientBonusPool` with `pos` already mutated. In `slashRedelegationPosition`:

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error(...)   // swallowed
    } else {
        return err
    }
}
// pos.LastBonusAccrual == blockTime, LastEventSeq == latest — bonus never paid
return k.setPositionWithState(ctx, pos, nil)  // persists advanced checkpoints
```

The position is saved with `LastBonusAccrual = blockTime`. All future bonus computations start from that timestamp, so the entire pre-slash accrual segment is permanently lost.

Additionally, `decrementEventRefCount` is called inside the loop (line 196) before the pool check, so event reference counts are decremented even though the position never received the bonus for those events. If the count reaches zero, the event is garbage-collected, making the segment unrecoverable even if the pool is later refilled.

### Impact Explanation
Position owners permanently lose accrued bonus rewards proportional to their stake size, tier APY, and the duration since their last claim. The loss is irreversible: once `LastBonusAccrual` is advanced and events are garbage-collected, no retry or pool refill can recover the owed amount. This is a direct, quantifiable economic loss to users.

### Likelihood Explanation
The `RewardsPoolName` pool is a shared module account drained by all legitimate bonus claims. It can reach zero through normal operation (high claim volume, delayed governance top-up). Redelegation slashes are triggered by validator double-signing, which is an externally observable on-chain event. The combination — pool empty at the moment of a redelegation slash — is realistic and requires no privileged access or attacker coordination. The position owner is the victim; no attacker profit is required for the loss to occur.

### Recommendation
Move `applyBonusAccrualCheckpoint`, `pos.UpdateLastKnownBonded`, and `pos.UpdateLastEventSeq` to execute **only after** a successful `SendCoinsFromModuleToAccount`, or restructure `processEventsAndClaimBonus` to return the computed bonus amount without mutating `pos`, letting the caller apply checkpoints only on confirmed payment. In `slashRedelegationPosition`, if the pool is insufficient, either skip the checkpoint advancement entirely (preserving the accrual window for a future retry) or record the owed-but-unpaid amount in a separate debt store so it can be settled when the pool is replenished.

### Proof of Concept
1. Create a tier position with a redelegation in progress (position mapped via `RedelegationMappings`).
2. Advance block time by 30 days so bonus accrues.
3. Drain `RewardsPoolName` to zero (e.g., via legitimate claims from other positions).
4. Trigger `BeforeRedelegationSlashed` with a partial `sharesToUnbond`.
5. Observe: no bonus is paid to the owner; `pos.LastBonusAccrual` is advanced to `blockTime`; event ref counts are decremented.
6. Refill `RewardsPoolName` with sufficient funds.
7. Call `MsgClaimTierRewards` for the position.
8. Assert: the pre-slash bonus segment (steps 1–4 duration) yields zero — it is permanently unrecoverable. [5](#0-4) [2](#0-1) [6](#0-5)
