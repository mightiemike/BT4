### Title
Pending Slow-Mode Collateral Deposit Not Reflected in Health Check Creates Liquidation Race Condition — (`core/contracts/Endpoint.sol`)

---

### Summary

`depositCollateralWithReferral` in `Endpoint.sol` immediately transfers a user's tokens to the clearinghouse but queues the subaccount balance credit as a slow-mode transaction with a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`). During this window the user's on-chain health is computed without the deposited collateral. A liquidator can submit a `LiquidateSubaccount` transaction that the sequencer processes before the pending deposit is executed, liquidating a user who has already committed the rescue collateral.

---

### Finding Description

In `Endpoint.sol`, `depositCollateralWithReferral` (lines 123–167) calls `handleDepositTransfer` at line 144, which immediately moves tokens from the user into the clearinghouse: [1](#0-0) 

Immediately after, the actual balance update is enqueued as a slow-mode transaction with a 3-day execution delay: [2](#0-1) 

The constant is: [3](#0-2) 

Until the sequencer includes an `ExecuteSlowMode` transaction in a batch (or 3 days elapse and anyone calls `executeSlowModeTransaction`), `Clearinghouse.depositCollateral` is never called, so `spotEngine.updateBalance` is never invoked and the subaccount's balance remains unchanged. [4](#0-3) 

Meanwhile, `liquidateSubaccountImpl` in `ClearinghouseLiq.sol` checks health against the current on-chain balance: [5](#0-4) 

`isUnderMaintenance` queries `getHealth` which reads only credited balances: [6](#0-5) 

Because the pending deposit has not yet been credited, the health calculation does not include the deposited collateral. The subaccount appears liquidatable even though the user has already sent the funds.

The sequencer processes liquidations via `submitTransactionsChecked` → `processTransactionImpl` → `clearinghouse.liquidateSubaccount`. There is no on-chain ordering constraint that forces pending `ExecuteSlowMode` deposit transactions to be processed before `LiquidateSubaccount` transactions. [7](#0-6) 

---

### Impact Explanation

A user who calls `depositCollateralWithReferral` to rescue a near-liquidation position has their tokens locked in the clearinghouse immediately, but their subaccount health does not improve until the slow-mode transaction is executed. If the market moves further against them during this window, a liquidator can submit a valid `LiquidateSubaccount` transaction. The sequencer processes the liquidation before the deposit credit, the user's position is seized at a discount, and the liquidation fee is charged — all while the user's funds are already held by the protocol. The user suffers a concrete asset loss (liquidation penalty + position loss) despite having acted in good faith.

---

### Likelihood Explanation

Every call to `depositCollateralWithReferral` opens this window. In normal sequencer operation the window is short (seconds to minutes), but during sequencer downtime, high load, or upgrade windows the window extends to the full 3-day timeout. The scenario is directly analogous to the external report: the protocol's "disabled" period is replaced by the sequencer's processing delay. The liquidator is an unprivileged on-chain actor who only needs to submit a signed `LiquidateSubaccount` transaction; no privileged access is required.

---

### Recommendation

Credit the deposit to the subaccount balance immediately inside `depositCollateralWithReferral` (as `Clearinghouse.depositCollateral` does) rather than deferring it through the slow-mode queue. If the slow-mode queue is required for other reasons (e.g., replay protection), add a per-subaccount "pending deposit" accumulator that is included in health calculations so that the deposited-but-not-yet-credited amount is counted when evaluating liquidatability.

---

### Proof of Concept

1. User A holds a perp position; maintenance health is `+5 USD`.
2. User A calls `depositCollateralWithReferral(name, QUOTE_PRODUCT_ID, 100e6)`. Tokens are transferred to the clearinghouse immediately; a slow-mode tx is queued with `executableAt = now + 3 days`. Subaccount balance is unchanged; health is still `+5 USD`.
3. Oracle price moves; maintenance health drops to `-3 USD`.
4. Liquidator B signs a `LiquidateSubaccount` struct targeting User A and submits it to the sequencer.
5. Sequencer calls `submitTransactionsChecked([..., liquidateTx, ...])`. `processTransactionImpl` dispatches to `clearinghouse.liquidateSubaccount`.
6. `isUnderMaintenance(userA)` returns `true` (pending deposit not credited). Liquidation proceeds; User A's position is seized and a liquidation fee is charged.
7. Three days later, `executeSlowModeTransaction` is called; `depositCollateral` credits `100e6` to User A's now-empty subaccount — the rescue collateral arrives after the position is already gone. [8](#0-7) [9](#0-8)

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
        // copy from submitSlowModeTransaction
        SlowModeConfig memory _slowModeConfig = slowModeConfig;

        slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: abi.encodePacked(
                uint8(TransactionType.DepositCollateral),
                abi.encode(
                    DepositCollateral({
                        sender: subaccount,
                        productId: productId,
                        amount: amount
                    })
                )
            )
        });
        slowModeConfig = _slowModeConfig;
    }

    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
