### Title
`ValidResourceBounds` Protobuf Round-Trip Is Non-Injective: `AllResources` with Zero L2/L1DataGas Deserializes to `L1Gas`, Producing a Different Transaction Hash - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` silently collapses `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` into `L1Gas(X)`. Because `get_tip_resource_bounds_hash` hashes a **different number of resource elements** depending on the variant (2 for `L1Gas`, 3 for `AllResources`), the transaction hash computed before serialization differs from the hash computed after deserialization. This is the sequencer-native analog of the "allocate and remove are not exact inverses" bug: two operations that should be inverses (serialize → deserialize) silently change the canonical representation, causing a divergent hash value.

---

### Finding Description

**Step 1 – Non-injective deserialization.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` applies the following rule:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← collapses AllResources
} else {
    ValidResourceBounds::AllResources(...)
})
``` [1](#0-0) 

The serializer (`From<ValidResourceBounds> for protobuf::ResourceBounds`) emits identical wire bytes for both `L1Gas(X)` and `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` — both produce `l2_gas=0, l1_data_gas=0` in the protobuf message. [2](#0-1) 

**Step 2 – Hash divergence.**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` branches on the variant:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 elements total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 elements total
    }
});
``` [3](#0-2) 

For the same numeric values `{ l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`:

| Variant | Elements hashed | Result |
|---|---|---|
| `L1Gas(X)` | `[tip, pack(X, L1_GAS), pack(0, L2_GAS)]` | H₁ |
| `AllResources { X, 0, 0 }` | `[tip, pack(X, L1_GAS), pack(0, L2_GAS), pack(0, L1_DATA_GAS)]` | H₂ ≠ H₁ |

The extra `pack(0, L1_DATA_GAS)` element makes H₂ ≠ H₁ for any non-trivial poseidon input.

**Step 3 – Reachable trigger path.**

1. A user submits `RpcInvokeTransactionV3` with `resource_bounds: AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` via the HTTP gateway. This is valid — the stateless validator only requires at least one non-zero bound, and `l1_gas` is non-zero.
2. The gateway converts it to `InternalRpcInvokeTransactionV3` (which always stores `AllResourceBounds`). Its `InvokeTransactionV3Trait::resource_bounds()` returns `ValidResourceBounds::AllResources(...)`.
3. `convert_rpc_tx_to_internal` computes `tx_hash = H₂` (AllResources path, 3 elements). [4](#0-3) 

4. The transaction is propagated over P2P as `InvokeTransactionV3` (storage type, `ValidResourceBounds::AllResources { X, 0, 0 }`), serialized to `protobuf::InvokeV3`. [5](#0-4) 

5. The receiving node deserializes via `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3`, which calls `ValidResourceBounds::try_from(...)` and produces `ValidResourceBounds::L1Gas(X)`. [6](#0-5) 

6. Any subsequent hash computation on the receiving node (e.g., block hash, receipt, re-execution) calls `get_invoke_transaction_v3_hash` on the deserialized `InvokeTransactionV3` and produces H₁ ≠ H₂. [7](#0-6) 

The P2P sync test confirms that transaction hashes are stored and compared after round-trip: [8](#0-7) 

---

### Impact Explanation

A transaction submitted via RPC with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is assigned hash H₂ at the originating node. After P2P propagation and protobuf deserialization, the receiving node holds the same transaction bytes but computes hash H₁ ≠ H₂. This causes:

- **Wrong transaction hash stored in block receipts and events** on the receiving node — the stored hash H₁ does not match the canonical hash H₂ signed by the user.
- **Block hash divergence** — if the block hash commits to transaction hashes, nodes will compute different block hashes for the same block, breaking consensus.
- **Wrong state/receipt/event** from any RPC call that re-derives the hash from the stored `InvokeTransactionV3` (e.g., `starknet_getTransactionReceipt`, `starknet_getTransactionByHash`).

This matches the allowed impact: *"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input"* and *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

---

### Likelihood Explanation

The trigger is an unprivileged user submitting a valid V3 invoke transaction with `l2_gas=0` and `l1_data_gas=0` but non-zero `l1_gas`. This is a common pattern for pre-0.13.3 style transactions submitted through the new RPC. No special permissions or malformed bytes are required. The condition is reachable on every production network.

---

### Recommendation

**Short term:** In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, do not collapse `AllResources` to `L1Gas` based on zero values. Instead, preserve the variant by encoding it explicitly in the wire format (e.g., add a discriminant field to `protobuf::ResourceBounds`). Alternatively, always deserialize as `AllResources` when all three fields are present in the protobuf message, regardless of their values.

**Long term:** Add a round-trip property test asserting that for all `ValidResourceBounds` values `v`, `ValidResourceBounds::try_from(protobuf::ResourceBounds::from(v)) == Ok(v)`. This would have caught the non-injectivity immediately.

### Proof of Concept

```rust
use starknet_api::transaction::fields::{
    AllResourceBounds, ResourceBounds, ValidResourceBounds, GasAmount, GasPrice
};
use starknet_api::transaction_hash::get_tip_resource_bounds_hash;
use starknet_api::transaction::fields::Tip;
use apollo_protobuf::converters::transaction::*;

let l1_gas = ResourceBounds { max_amount: GasAmount(100), max_price_per_unit: GasPrice(50) };
let tip = Tip(0);

// AllResources with zero l2_gas and l1_data_gas
let all_resources = ValidResourceBounds::AllResources(AllResourceBounds {
    l1_gas,
    l2_gas: ResourceBounds::default(),    // zero
    l1_data_gas: ResourceBounds::default(), // zero
});

// L1Gas with same l1_gas
let l1_gas_only = ValidResourceBounds::L1Gas(l1_gas);

// Hashes differ
let h_all = get_tip_resource_bounds_hash(&all_resources, &tip).unwrap();
let h_l1  = get_tip_resource_bounds_hash(&l1_gas_only,  &tip).unwrap();
assert_ne!(h_all, h_l1);  // ← PASSES: different hashes for same numeric values

// Protobuf round-trip collapses AllResources → L1Gas
let proto: protobuf::ResourceBounds = all_resources.into();
let recovered = ValidResourceBounds::try_from(proto).unwrap();
assert!(matches!(recovered, ValidResourceBounds::L1Gas(_)));  // ← PASSES: variant changed

// Hash after round-trip equals L1Gas hash, not AllResources hash
let h_recovered = get_tip_resource_bounds_hash(&recovered, &tip).unwrap();
assert_eq!(h_recovered, h_l1);   // ← PASSES
assert_ne!(h_recovered, h_all);  // ← PASSES: hash changed after round-trip
```

The originating node stores the transaction under `h_all`; the receiving node computes `h_l1` ≠ `h_all`, producing a wrong transaction hash in every receipt, event, and block hash commitment derived from the deserialized transaction.

pub(crate) fn get_invoke_transaction_v3_hash<T: InvokeTransactionV3Trait>(
    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();
    let calldata_hash =
        HashChain::new().chain_iter(transaction.calldata().0.iter()).get_poseidon_hash();
    let mut hash_chain = HashChain::new()
        .chain(&INVOKE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address().0.key())
        .chain(&tip_resource_bounds_hash)
        .chain(&paymaster_data_hash)
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce().0)
        .chain(&data_availability_mode)
        .chain(&account_deployment_data_hash)
        .chain(&calldata_hash);
    if !transaction.proof_facts().0.is_empty() {
        let proof_facts_hash =
            HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
        hash_chain = hash_chain.chain(&proof_facts_hash);
    }
    Ok(TransactionHash(hash_chain.get_poseidon_hash()))
