Audit Report

## Title
`DepositAllowlistExtension` Validates LP Recipient (`owner`) Instead of Token Payer (`sender`), Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual token payer) and validates `owner` (the arbitrary LP position recipient) instead. Because `MetricOmmPool.addLiquidity` accepts any `owner` address and collects tokens from `msg.sender` via callback, any non-allowlisted caller can bypass the deposit allowlist by supplying an already-allowlisted address as `owner`. The pool admin's access control is completely defeated.

## Finding Description
`MetricOmmPool.addLiquidity` passes both the real caller and the position recipient to the extension hook:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

The first argument (`msg.sender`) is the actual token payer; the second (`owner`) is the LP position recipient. `DepositAllowlistExtension.beforeAddLiquidity` ignores the first parameter entirely and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

Tokens are collected from `msg.sender` (the caller) via callback in `LiquidityLib`: [3](#0-2) 

LP shares are credited to `owner` via `positionBinShares[posKey]` where `posKey = keccak256(owner, salt, bin)`: [4](#0-3) 

`removeLiquidity` enforces `msg.sender == owner`, so only the allowlisted `owner` can withdraw — but the non-allowlisted caller already paid the tokens in: [5](#0-4) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the first parameter, the actual caller): [6](#0-5) 

The asymmetry is the root cause: the swap guard checks the right actor; the deposit guard does not.

## Impact Explanation
This is a direct admin-boundary break. The pool admin configures `allowedDepositor` to restrict which addresses may deposit (e.g., for KYC/AML compliance or pool composition control). Any unprivileged address can route a deposit through any allowlisted `owner`, injecting arbitrary liquidity into the restricted pool. The allowlist provides zero protection: the wrong value checked is `allowedDepositor[pool][owner]` when it must be `allowedDepositor[pool][sender]`. This matches the allowed impact: *admin-boundary break — factory/oracle role checks are bypassed by an unprivileged path*.

## Likelihood Explanation
Likelihood is high. The bypass requires no special permissions, no flash loan, and no oracle manipulation. Any EOA or contract implementing `IMetricOmmModifyLiquidityCallback` can call `addLiquidity` with an arbitrary `owner`. Allowlisted addresses are discoverable on-chain via the `AllowedToDepositSet` event or the `allowedDepositor` mapping. The attack is a single transaction, repeatable indefinitely.

## Recommendation
Check `sender` (the actual token payer) instead of `owner` (the LP position recipient), mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```
Setup:
  - Pool P has DepositAllowlistExtension configured.
  - allowedDepositor[P][Alice] = true (Alice is allowlisted)
  - Bob is NOT allowlisted.

Attack:
  1. Bob deploys a contract implementing IMetricOmmModifyLiquidityCallback
     that transfers the required tokens to the pool in the callback.
  2. Bob calls pool.addLiquidity(owner=Alice, salt=0, deltas=..., callbackData=..., extensionData=...).
  3. Pool calls _beforeAddLiquidity(sender=Bob, owner=Alice, ...).
  4. DepositAllowlistExtension checks allowedDepositor[P][Alice] → true → no revert.
  5. LiquidityLib credits LP shares to Alice (positionBinShares[keccak256(Alice,0,bin)] += shares).
  6. Callback fires to Bob's contract; Bob's contract transfers tokens to the pool.
  7. Alice now holds LP shares she did not request; Bob's tokens are in the pool.
  8. Alice calls removeLiquidity(owner=Alice, ...) and recovers the tokens.

Result: Bob deposited into a restricted pool without being allowlisted.
        The deposit allowlist is completely bypassed.
```
