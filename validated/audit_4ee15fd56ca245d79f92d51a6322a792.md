### Title
Unbounded storage loop in `reorg_chain()` can exceed NEAR gas limit for deep forks, permanently stranding the mainchain tip on the wrong chain - (`contract/src/lib.rs`)

### Summary
`reorg_chain()` contains two unbounded loops that each perform multiple on-chain storage operations per iteration. The number of iterations scales with fork depth, which is uncapped. For a fork deep enough to exhaust NEAR's 300 TGas per-transaction limit, the reorg transaction fails and is fully reverted, leaving the contract's `mainchain_tip_blockhash` permanently pointing to the lower-chainwork chain. All subsequent SPV proofs are evaluated against the wrong canonical chain.

### Finding Description

`reorg_chain()` is called automatically from `submit_block_header_inner()` whenever a fork's accumulated `chain_work` exceeds the current mainchain tip's `chain_work`. It contains two sequential unbounded loops:

**Loop 1** — demotes excess mainchain blocks (lines 586–593):

```rust
for height in (fork_tip_height + 1)..=last_main_chain_block_height {
    let current_main_chain_blockhash = self
        .mainchain_height_to_header
        .get(&height)
        .unwrap_or_else(|| env::panic_str("cannot get a block"));
    self.remove_block_header(&current_main_chain_blockhash);
    self.mainchain_height_to_header.remove(&height);
}
```

Each iteration performs 3 storage mutations (1 read + 2 removes). The iteration count equals `last_main_chain_block_height − fork_tip_height`, which is unbounded.

**Loop 2** — promotes fork blocks to mainchain (lines 616–643):

```rust
while !self.mainchain_header_to_height.contains_key(&fork_header_cursor.block_hash) {
    // ...
    let main_chain_block = self.mainchain_height_to_header.insert(...);
    self.mainchain_header_to_height.insert(...);
    if let Some(current_main_chain_blockhash) = main_chain_block {
        self.remove_block_header(&current_main_chain_blockhash);
    }
    fork_header_cursor = self.headers_pool.get(&prev_block_hash)...;
}
```

Each iteration performs 4–6 storage operations (2 reads + 2 writes + up to 2 removes). The iteration count equals the distance from the fork tip back to the common ancestor — the entire fork depth.

Neither loop has a `batch_size` guard, a gas check, or any upper bound. The `run_mainchain_gc()` function, by contrast, correctly accepts a `batch_size` parameter and uses `std::cmp::min(total_amount_to_remove, batch_size)` to bound its loop. No equivalent protection exists in `reorg_chain()`. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

When the reorg transaction runs out of gas, NEAR reverts all state changes for that transaction. The fork blocks submitted in prior transactions remain in `headers_pool` (they were stored via `store_fork_header()` in earlier, successful transactions). The contract's `mainchain_tip_blockhash` is left pointing to the old, lower-chainwork chain. The fork tip — which has more accumulated work and is therefore the correct canonical chain — can never be promoted because every attempt to submit the triggering block re-enters `reorg_chain()` and OOGs again.

Consequences:
- `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` evaluate proofs against the wrong mainchain, returning `false` for valid transactions on the correct chain and potentially `true` for transactions on the abandoned chain.
- The corrupted canonical-chain pointer (`mainchain_tip_blockhash`) and the height-to-hash index (`mainchain_height_to_header`) cannot be repaired without a privileged migration. [4](#0-3) [5](#0-4) [6](#0-5) 

### Likelihood Explanation

The entry path is `submit_blocks()`, which is gated by `#[trusted_relayer]`. Any account that stakes as a trusted relayer can call it — this is an open economic mechanism, not a privileged administrative role. The test suite confirms that an unregistered account receives `"Relayer is not active"`, while any staked relayer passes the guard. [7](#0-6) [8](#0-7) 

A trusted relayer can build a deep fork by:
1. Submitting many valid fork-chain headers one by one (each in a separate, gas-bounded transaction). Each call stores the fork block via `store_fork_header()` without triggering a reorg because the fork's chainwork has not yet exceeded the mainchain.
2. Submitting the final fork block whose cumulative `chain_work` exceeds the mainchain tip. This triggers `reorg_chain()` in a single transaction that must walk back through all previously stored fork blocks.

For Dogecoin (1-minute blocks, higher fork frequency) and for any chain where the contract is initialized with a stale tip, forks of dozens to hundreds of blocks are realistic. Even for Bitcoin, the `gc_threshold` default of 52,704 blocks means the contract stores up to a year of headers, and a fork of even ~50 blocks performing 5 storage operations each at NEAR's storage-write costs can approach the 300 TGas limit. [9](#0-8) 

### Recommendation

Apply the same batching pattern already used in `run_mainchain_gc()`: add a `reorg_batch_size` parameter (or derive it from the number of submitted headers) and limit each call to at most that many loop iterations. If the reorg cannot complete in one transaction, store intermediate cursor state (e.g., the current fork cursor hash and the demote-phase progress) so subsequent calls can resume. Alternatively, enforce a protocol-level cap on the maximum fork depth the contract will process in a single transaction, rejecting reorgs that exceed it. [10](#0-9) 

### Proof of Concept

1. Deploy the contract with `gc_threshold = 52704` (the recommended production value).
2. As a staked trusted relayer, submit N mainchain blocks normally (e.g., N = 200).
3. Submit N fork blocks one by one, each extending from the same ancestor as the mainchain. Each call succeeds individually because the fork's chainwork is still below the mainchain.
4. Submit the (N+1)-th fork block whose cumulative `chain_work` exceeds the mainchain tip. This triggers `reorg_chain()`, which must:
   - Execute Loop 1 for up to N iterations (demoting mainchain blocks), and
   - Execute Loop 2 for N+1 iterations (promoting fork blocks).
5. The transaction runs out of gas and is reverted. `mainchain_tip_blockhash` remains on the lower-chainwork chain.
6. Call `verify_transaction_inclusion_v2` for a transaction in a fork block: it returns `false` (block not in mainchain index) even though the fork has more work and is the correct canonical chain. [11](#0-10) [1](#0-0) [2](#0-1)

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.mainchain_height_to_header.remove(&height);
            }

                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
