### Title
Missing `tx_index` Range Validation Enables Proof Replay with Manipulated Position — (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `tx_index` (`u64`) but never validate that it falls within the valid leaf range `[0, 2^merkle_proof.len() - 1]`. Because `compute_root_from_merkle_proof` only consumes the lower `merkle_proof.len()` bits of the position, any `tx_index` value that shares those lower bits with the real position produces an identical root computation and causes the function to return `true`. An unprivileged NEAR caller can therefore present a single valid on-chain proof and have it accepted at multiple distinct `tx_index` values.

---

### Finding Description

`verify_transaction_inclusion` is a public, unpermissioned view function: [1](#0-0) 

It receives `ProofArgs`, which carries `tx_index: u64`: [2](#0-1) 

The only guards applied before delegating to `compute_root_from_merkle_proof` are:
- `confirmations <= gc_threshold`
- block is on the mainchain
- enough confirmations exist
- `merkle_proof` is non-empty

There is **no check** that `tx_index < 2^merkle_proof.len()`.

`compute_root_from_merkle_proof` uses `transaction_position` only to decide left/right ordering at each level: [3](#0-2) 

The loop body is:
```rust
if current_position % 2 == 0 { ... } else { ... }
current_position /= 2;
```

After `N` iterations (where `N = merkle_proof.len()`), only the lowest `N` bits of the original `transaction_position` have been consumed. Bits above position `N` are never read. Therefore, for any real position `X`, the positions `X`, `X + 2^N`, `X + 2·2^N`, … all produce **identical** root computations with the same proof vector.

`verify_transaction_inclusion_v2` inherits the same flaw because it calls the deprecated function after its own coinbase check: [4](#0-3) 

The coinbase position is hardcoded to `0usize` and is unaffected, but the transaction `tx_index` is passed through without any range guard.

---

### Impact Explanation

The contract is consumed by downstream NEAR contracts that call `verify_transaction_inclusion_v2` to gate actions (e.g., minting wrapped BTC, releasing funds, recording a processed payment). Such contracts commonly use the tuple `(tx_block_blockhash, tx_index)` as a unique identifier to prevent replay of the same Bitcoin transaction. Because the light client returns `true` for `tx_index = X` **and** for `tx_index = X + 2^N` using the same proof, an attacker can:

1. Obtain a valid proof for a real Bitcoin transaction at position `X` in block `B`.
2. Call `verify_transaction_inclusion_v2` with `tx_index = X` → returns `true`; downstream contract records `(B, X)` as spent.
3. Call again with `tx_index = X + 2^N` (same proof, same `tx_id`) → also returns `true`; downstream contract records `(B, X + 2^N)` as a distinct, unspent entry and releases funds a second time.

The corrupted value is the **proof result** (`true`) for a position that does not correspond to any real leaf in the block's Merkle tree, breaking the uniqueness invariant that downstream contracts rely on.

---

### Likelihood Explanation

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are public, carry no access-control attribute beyond `#[pause]`, and are callable by any NEAR account with no deposit requirement. The attacker needs only one legitimately mined Bitcoin transaction and its standard SPV proof. The manipulation of `tx_index` requires no cryptographic capability — it is pure arithmetic. Any downstream contract that uses `(tx_block_blockhash, tx_index)` as a replay guard is directly exploitable.

---

### Recommendation

Add a range check in `verify_transaction_inclusion` immediately after the `merkle_proof.is_empty()` guard:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

// tx_index must be a valid leaf index for a tree of this proof depth
let max_valid_index: u64 = (1u64)
    .checked_shl(args.merkle_proof.len() as u32)
    .map(|v| v - 1)
    .unwrap_or(u64::MAX);
require!(
    args.tx_index <= max_valid_index,
    "tx_index exceeds the maximum leaf index for the given proof depth"
);
```

Alternatively, enforce the constraint inside `compute_root_from_merkle_proof` by asserting `transaction_position < (1 << merkle_proof.len())` before the loop.

---

### Proof of Concept

Given a real Bitcoin block with `N = 3` proof levels and a transaction at position `X = 0`:

```
merkle_proof = [sibling_0, sibling_1, sibling_2]   // valid proof for position 0
tx_id        = <real transaction hash>
tx_block_blockhash = <real block hash on mainchain>
confirmations = 1
```

Call 1: `tx_index = 0` → `compute_root_from_merkle_proof(tx_id, 0, proof)` traverses path L→L→L → returns correct root → **`true`**.

Call 2: `tx_index = 8` (= 0 + 2^3) → `compute_root_from_merkle_proof(tx_id, 8, proof)`:
- level 0: `8 % 2 == 0` → L (same as position 0)
- level 1: `4 % 2 == 0` → L (same)
- level 2: `2 % 2 == 0` → L (same)

Returns **identical root** → **`true`**.

The downstream contract sees `(block_hash, tx_index=8)` as a new, unprocessed proof and releases funds a second time. [5](#0-4) [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
