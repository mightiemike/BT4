Audit Report

## Title
`PriceVelocityGuardExtension` Uses `block.number` Instead of `block.timestamp` for Velocity Window, Making the Guard Inaccurate on Chains with Non-Constant Block Time - (File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol)

## Summary
`PriceVelocityGuardExtension` stores `lastUpdateBlock` as `uint64(block.number)` and computes the allowed price-change window using `blockDiff = block.number - prevBlock`. On Berachain, where block time is non-constant, faster-than-expected block production causes `blockDiff` to grow larger for the same real-world elapsed time, making `allowedSq` larger and the guard more permissive than the admin intended. This allows a manipulated oracle mid-price to pass the `beforeSwap` velocity check and reach pool execution.

## Finding Description
The contract stores and computes the velocity window entirely in block units:

- `s.lastUpdateBlock = uint64(block.number)` is written in both `setLastMidPrice` (L32) and `beforeSwap` (L58).
- `blockDiff = block.number - prevBlock` (L63) feeds directly into `allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff)` (L70).
- The check `if (actualSq > allowedSq) revert PriceVelocityExceeded(...)` (L72–74) is the sole guard before the swap proceeds.

When Berachain produces blocks faster than the rate the admin assumed when setting `maxChangePerBlockE18`, `blockDiff` accumulates more quickly per real-world second. For example, if the admin calibrated at 2 s/block and blocks arrive at 0.5 s/block, four blocks elapse in 2 real-world seconds, yielding `allowedSq = maxChange² × 5` instead of the intended `maxChange² × 2` — a 2.5× expansion of the permitted price-change window. A price move that should be rejected passes the check, and the swap executes at the manipulated bid/ask.

No existing secondary guard catches this: the `beforeSwap` hook is the only velocity enforcement path, and once `actualSq ≤ allowedSq` the function returns the selector unconditionally (L78).

## Impact Explanation
The permissive miscalibration directly enables bad-price execution: a mid-price derived from a manipulated oracle feed that exceeds the admin's intended per-second rate passes the `beforeSwap` hook. The pool then executes the swap at the inflated bid/ask, causing the trader to receive more output tokens than the true oracle price permits or the pool to receive fewer input tokens than owed — a direct loss of LP principal. This maps to the "bad-price execution" and "swap conservation failure" allowed impacts.

## Likelihood Explanation
Berachain's block time is documented as variable (congestion-dependent). No privileged access is required: the miscalibration is passive and worsens whenever the network produces blocks faster than the admin's assumed rate. An attacker need only time a swap during a period of elevated block throughput while the oracle mid-price has moved beyond the intended per-second threshold. The oracle mid-price is derived from the caller-supplied `bidPriceX64`/`askPriceX64` passed into `beforeSwap` (L43–44, L48), so a public oracle pusher or router caller can supply a price at the boundary of the expanded window.

## Recommendation
Replace `block.number` with `block.timestamp` throughout the contract. Rename `lastUpdateBlock` → `lastUpdateTimestamp` and `maxChangePerBlockE18` → `maxChangePerSecondE18`. Update the velocity formula to:

```solidity
uint256 timeDiff = block.timestamp - prevTimestamp; // seconds
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + timeDiff);
```

This makes the guard invariant to block-time fluctuations and consistent with the oracle layer, which already uses unix timestamps for staleness and future-drift checks.

## Proof of Concept
1. Deploy pool with `PriceVelocityGuardExtension`; admin calls `setMaxChangePerBlock` with `maxChangePerBlockE18 = 0.005e18` (0.5%/block), calibrated at 2 s/block (intended rate: 0.25%/s).
2. Berachain enters a fast-block period: 0.5 s/block. In 2 real-world seconds, 4 blocks elapse.
3. Attacker submits a swap with `bidPriceX64`/`askPriceX64` encoding a mid-price 1.118% above `lastMidPriceX64`.
4. `beforeSwap` computes `blockDiff = 4`, `allowedSq = (0.005e18)² × 5`; `actualSq = (0.01118e18)²` ≤ `allowedSq` → check passes.
5. Admin intended only 0.25%/s × 2 s = 0.5% to be allowed; 1.118% should have been rejected.
6. Swap executes at the manipulated price; LPs absorb the loss.

Reproduce with a Foundry fork test on Berachain: mock `block.number` advancing 4 times while `block.timestamp` advances only 2 seconds, then call `beforeSwap` with a 1.118% price delta and assert no revert.
