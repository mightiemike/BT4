### Title
Deprecated `verify_transaction_inclusion` Bypasses 64-Byte Merkle Forgery Protection Present in `verify_transaction_inclusion_v2` — (`contract/src/lib.rs`)

---

### Summary

The contract exposes two public SPV verification entry points. `verify_transaction_inclusion_v2` enforces a coinbase Merkle proof check specifically to block the 64-byte transaction Merkle forgery attack. The older `verify_transaction_inclusion` function, though marked deprecated, remains fully public and callable by any unprivileged NEAR account. An attacker can call the deprecated function directly, bypassing the coinbase proof guard entirely, and obtain a `true` result for a forged transaction inclusion proof.

---

### Finding Description

`verify_transaction_inclusion_v2` adds a coinbase proof requirement on top of the base verification logic:

```rust
// lib.rs:358-365
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
```

This coinbase check is the sole mitigation for the [64-byte Merkle leaf forgery](https://www.bitmex.com/blog/64-Byte-Transactions): because the coinbase transaction is always at index 0 with a known proof path, verifying it authenticates the entire Merkle tree structure and prevents an attacker from substituting a crafted 64-byte internal node as a "transaction."

The deprecated `verify_transaction_inclusion` is still declared `pub` with no access restriction:

```rust
// lib.rs:283-288
#[deprecated(
    since = "0.5.0",
    note = "Use `verify_transaction_inclusion_v2` instead."
)]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
```

It performs only the basic Merkle root comparison:

```rust
// lib.rs:318-323
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

No coinbase proof is required. Any caller can invoke this function directly, skipping the protection that `verify_transaction_inclusion_v2` enforces.

---

### Impact Explanation

An attacker who crafts a 64-byte payload that hashes to an internal Merkle tree node can construct a Merkle proof that makes `verify_transaction_inclusion` return `true` for a transaction that was never included in the block. Consumer contracts that call `verify_transaction_inclusion` — or that accept a cross-contract call result from it — will receive a false positive and may release funds, update state, or take other irreversible actions based on a fabricated Bitcoin transaction.

The protection asymmetry is structural: the same state-reading outcome (a `true` SPV result) is reachable via two public entry points, but only one of them enforces the forgery guard.

---

### Likelihood Explanation

The entry point is fully public and requires no special role or stake. Any NEAR account can call `verify_transaction_inclusion` with a crafted `ProofArgs`. The 64-byte forgery technique is well-documented and has known tooling. Consumer contracts that integrated before `v0.5.0` or that call the deprecated function by mistake are directly exposed.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or gate it behind an access-control role so it cannot be called by arbitrary accounts. The coinbase proof check should not be optional for any externally reachable verification path.

---

### Proof of Concept

1. Identify a Bitcoin block whose Merkle tree has an internal node whose 32-byte left-child hash concatenated with its 32-byte right-child hash forms a 64-byte value that can be interpreted as a valid-looking transaction hash `F`.
2. Construct `merkle_proof` such that `compute_root_from_merkle_proof(F, index, proof) == block.merkle_root`.
3. Call `verify_transaction_inclusion` with `tx_id = F`, the crafted `merkle_proof`, and the real `tx_block_blockhash`.
4. The function returns `true` despite `F` never existing as a real transaction.
5. Calling `verify_transaction_inclusion_v2` with the same `tx_id` and no valid coinbase proof would panic at the `"Incorrect coinbase merkle proof"` require, demonstrating the asymmetry. [1](#0-0) [2](#0-1)

    #[deprecated(

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
