Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks position recipient (`owner`) instead of actual depositor (`sender`), allowing non-allowlisted callers to bypass the deposit gate — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`, i.e., the address supplying tokens) and instead validates the caller-supplied `owner` (the position recipient). Any unprivileged address can bypass the allowlist by calling `pool.addLiquidity(allowlisted_address, …)`, passing an allowlisted address as `owner` while they themselves supply the tokens via the liquidity callback. The allowlist invariant — that only approved depositors may interact with the curated pool — is completely broken.

## Finding Description

**Root cause — wrong argument checked:**

In `DepositAllowlistExtension.beforeAddLiquidity` the first parameter (`sender`) is unnamed and discarded; the check is performed on `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

**Call path — `sender` is `msg.sender`, `owner` is attacker-controlled:**

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the position recipient:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [2](#0-1) 

`ExtensionCalling._beforeAddLiquidity` forwards both arguments faithfully to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L95-98
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [3](#0-2) 

**Exploit flow:**

1. Pool is configured with `DepositAllowlistExtension`; only `allowedDepositor[pool][alice] = true`.
2. Attacker (not allowlisted) calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
3. Pool calls `_beforeAddLiquidity(msg.sender=attacker, owner=alice, …)`.
4. Extension checks `allowedDepositor[pool][alice]` → `true` → passes.
5. `LiquidityLib.addLiquidity` mints the position attributed to `alice`, with tokens supplied by the attacker via the liquidity callback.
6. The attacker has successfully deposited into a pool that was supposed to be restricted to allowlisted addresses only.

**Existing guards are insufficient:** The only guard is the allowlist check itself, which is the broken component. There is no secondary check on `sender` anywhere in the extension or the core pool path.

## Impact Explanation
The `DepositAllowlistExtension` is the sole mechanism for curated pools to restrict who may provide liquidity. Because it checks the wrong address, the allowlist provides zero protection: any unprivileged address can deposit into any pool gated by this extension. This breaks a core pool access-control invariant — the pool admin's ability to restrict depositors — and constitutes broken core pool functionality. Depending on deployment context (e.g., compliance/KYC-gated pools), this can also expose the protocol to regulatory or financial risk from unauthorized participants interacting with restricted pools.

## Likelihood Explanation
The attack requires no special privileges, no capital at risk beyond the deposited tokens (which are attributed to the `owner`), and no complex setup. Any EOA or contract can execute it in a single transaction. It is repeatable indefinitely and requires only knowledge of one allowlisted address (which is publicly readable from `allowedDepositor`). The precondition — a pool using `DepositAllowlistExtension` with at least one allowlisted address — is the normal operating state of any curated pool.

## Recommendation
Replace the unnamed discarded first parameter with `sender` and check it instead of (or in addition to) `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This ensures the address actually supplying tokens (the real depositor) is the one validated against the allowlist, consistent with the NatSpec comment "Gates `addLiquidity` by depositor address."

## Proof of Concept

```solidity
// Foundry unit test sketch
function test_allowlistBypass() public {
    // Setup: pool with DepositAllowlistExtension, only alice is allowlisted
    address alice = makeAddr("alice");
    extension.setAllowedToDeposit(address(pool), alice, true);

    // Attacker (not allowlisted) calls addLiquidity with alice as owner
    address attacker = makeAddr("attacker");
    vm.startPrank(attacker);
    // Fund attacker with tokens, approve pool callback
    token0.mint(attacker, 1e18);
    token1.mint(attacker, 1e18);
    // This should revert if allowlist works, but it succeeds:
    pool.addLiquidity(alice, 0, deltas, callbackData, "");
    vm.stopPrank();

    // Verify: position minted for alice, tokens came from attacker
    // allowlist was bypassed — attacker deposited into restricted pool
}
```
