### Title
Missing Slippage Protection in `mintNlp()` and `burnNlp()` — User Cannot Constrain Received Amount Against Oracle Price Deviation - (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`mintNlp()` and `burnNlp()` in `Clearinghouse.sol` compute the amount of NLP tokens minted (or quote tokens returned) using an `oraclePriceX18` that is **not included in the user's EIP-712 signed digest**. The user signs only their input amount (`quoteAmount` or `nlpAmount`) with no ability to specify a minimum output. Because the oracle price can deviate from real market prices, users can receive materially fewer tokens than expected with no on-chain protection.

---

### Finding Description

**`mintNlp` path:**

The user signs a `MintNlp` struct containing only `{sender, quoteAmount, nonce}`. The NLP tokens credited are computed as:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [1](#0-0) 

The `oraclePriceX18` is a sequencer-supplied field in the outer `SignedMintNlp` wrapper struct, **outside** the signed portion. The `Verifier.computeDigest` for `MintNlp` hashes only `{MINT_NLP_SIGNATURE, sender, quoteAmount, nonce}`:

```solidity
digest = keccak256(
    abi.encode(
        keccak256(bytes(MINT_NLP_SIGNATURE)),
        signedTx.tx.sender,
        signedTx.tx.quoteAmount,
        signedTx.tx.nonce
    )
);
``` [2](#0-1) 

`oraclePriceX18` is never committed to by the user's signature. The `MintNlp` struct has no `minNlpAmount` field:

```solidity
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;
    uint64 nonce;
}
``` [3](#0-2) 

**`burnNlp` path:**

The user signs a `BurnNlp` struct containing only `{sender, nlpAmount, nonce}`. The quote returned is:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [4](#0-3) 

Again, `oraclePriceX18` is not in the user's signed digest:

```solidity
digest = keccak256(
    abi.encode(
        keccak256(bytes(BURN_NLP_SIGNATURE)),
        signedTx.tx.sender,
        signedTx.tx.nlpAmount,
        signedTx.tx.nonce
    )
);
``` [5](#0-4) 

The `BurnNlp` struct has no `minQuoteAmount` field:

```solidity
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint64 nonce;
}
``` [6](#0-5) 

The sequencer constructs the full `SignedMintNlp`/`SignedBurnNlp` and supplies `oraclePriceX18` at execution time. The user's signed intent is decoupled from the price used to compute their output. There is no on-chain check that the received amount meets any user-specified minimum.

---

### Impact Explanation

**Impact: Medium**

- A user minting NLP by spending `quoteAmount` of quote tokens receives `quoteAmount / oraclePriceX18` NLP tokens. If `oraclePriceX18` is inflated relative to the real market price (e.g., stale oracle, volatile market), the user receives fewer NLP tokens than they would at the real price, with no recourse.
- A user burning `nlpAmount` of NLP tokens receives `nlpAmount * oraclePriceX18 - burnFee` quote tokens. If `oraclePriceX18` is deflated, the user receives fewer quote tokens than expected.
- In both cases the user's signed commitment does not constrain the output amount. The asset delta is a direct loss of value to the user — fewer NLP tokens credited or fewer quote tokens returned — with no minimum-output guard. [7](#0-6) 

---

### Likelihood Explanation

**Likelihood: Medium**

Oracle prices naturally lag real market prices, especially during volatile periods. The `oraclePriceX18` is provided by the sequencer at the time the batch is submitted, which can be materially different from the price at the time the user signed their intent. No malicious actor is required — normal market movement between signing and execution is sufficient to cause the user to receive a worse-than-expected output. The absence of any `minNlpAmount` or `minQuoteAmount` field in the signed structs means there is no protocol-level mechanism to protect users. [8](#0-7) 

---

### Recommendation

Add a `minNlpAmount` field to `MintNlp` and a `minQuoteAmount` field to `BurnNlp`, include these fields in the EIP-712 signed digest in `Verifier.computeDigest`, and enforce them in `Clearinghouse.mintNlp` / `Clearinghouse.burnNlp`:

```solidity
// In mintNlp:
require(nlpAmount >= txn.minNlpAmount, ERR_SLIPPAGE_TOO_HIGH);

// In burnNlp:
require(quoteAmount >= txn.minQuoteAmount, ERR_SLIPPAGE_TOO_HIGH);
```

This mirrors the existing `ERR_SLIPPAGE_TOO_HIGH` error already defined in the codebase. [9](#0-8) 

---

### Proof of Concept

1. User signs `MintNlp{sender=alice, quoteAmount=1000e18, nonce=5}` when NLP oracle price is `1.00`.
2. Before the sequencer includes the transaction, the NLP oracle price moves to `1.10` (10% increase).
3. Sequencer submits the batch with `oraclePriceX18 = 1.10e18`.
4. `mintNlp` executes: `nlpAmount = 1000e18 / 1.10e18 ≈ 909e18` NLP tokens credited.
5. Alice expected ~1000 NLP tokens but received ~909 — a ~9% loss — with no on-chain protection.
6. Alice's signature was valid; no check on minimum NLP output exists anywhere in `mintNlp`. [10](#0-9)

    function mintNlp(
        IEndpoint.MintNlp calldata txn,
        int128 oraclePriceX18,
