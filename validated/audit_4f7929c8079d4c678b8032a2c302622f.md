### Title
`slowModeFees` Accumulates Real ERC20 Tokens With No Withdrawal or Distribution Path — (`File: core/contracts/EndpointStorage.sol` / `core/contracts/EndpointTx.sol`)

---

### Summary

Every slow-mode transaction charges a real ERC20 fee that is transferred into the `Endpoint` contract and tracked in `slowModeFees`. No function in the protocol ever reads, decrements, or distributes `slowModeFees`. The collected tokens are permanently locked in the `Endpoint` contract.

---

### Finding Description

In `EndpointStorage.sol`, `slowModeFees` is declared as an `int128` accounting variable: [1](#0-0) 

In `EndpointTx.submitSlowModeTransactionImpl`, every non-privileged slow-mode submission charges the caller a real ERC20 fee via `chargeSlowModeFee`, which calls `token.safeTransferFrom(from, address(this), clearinghouse.getSlowModeFee())`, transferring actual tokens into the `Endpoint` contract. Immediately after, `slowModeFees` is incremented: [2](#0-1) 

The `chargeSlowModeFee` function that moves the real tokens: [3](#0-2) 

The `DumpFees` slow-mode transaction type — the only fee-claiming path in the protocol — handles only `sequencerFee[productId]` (the per-product mapping) and calls `clearinghouse.claimSequencerFees`. It never touches `slowModeFees`: [4](#0-3) 

`slowModeFees` is never decremented, read, or passed to any distribution function anywhere in the codebase. The ERC20 tokens transferred to `address(this)` (the `Endpoint` proxy) accumulate with no recovery path.

---

### Impact Explanation

Every slow-mode fee paid by any user — for `WithdrawCollateral`, `LinkSigner`, `ClaimBuilderFee`, and other non-privileged slow-mode transactions — is permanently locked inside the `Endpoint` contract. The protocol has no function to sweep, distribute, or otherwise use these tokens. The `slowModeFees` accounting variable is a dead accumulator. The asset delta is: all slow-mode fees ever paid, denominated in the quote token (USDC), are irrecoverably stranded.

---

### Likelihood Explanation

High. Every non-privileged slow-mode transaction triggers this path. The slow-mode path is the censorship-resistance mechanism available to any user when the sequencer is unresponsive. Any user who has ever submitted a slow-mode transaction has contributed to the locked balance. The condition is already active on mainnet from the first slow-mode submission.

---

### Recommendation

Implement a privileged sweep function (callable by the owner or via a sequencer-submitted `DumpFees`-style transaction) that transfers the accumulated `slowModeFees` balance to the `X_ACCOUNT` or the protocol treasury, mirroring how `sequencerFee` is handled in `claimSequencerFees`. Alternatively, route the slow-mode fee directly into `sequencerFee[QUOTE_PRODUCT_ID]` at collection time so it is automatically swept by the existing `DumpFees` path, and remove the now-redundant `slowModeFees` variable.

---

### Proof of Concept

1. User calls `Endpoint.submitSlowModeTransaction` with a `WithdrawCollateral` transaction.
2. `submitSlowModeTransactionImpl` reaches the `else` branch (line 370), calls `chargeSlowModeFee(_getQuote(), sender)`, which executes `token.safeTransferFrom(sender, address(this), clearinghouse.getSlowModeFee())` — real USDC moves into `Endpoint`.
3. `slowModeFees += SLOW_MODE_FEE` is executed (line 371).
4. The sequencer later submits a `DumpFees` transaction. Inspection of `processSlowModeTransactionImpl` lines 244–253 shows it only processes `sequencerFee[spotIds[i]]` and calls `clearinghouse.claimSequencerFees(fees)`. `slowModeFees` is untouched.
5. Repeat for any number of users. The USDC balance of the `Endpoint` contract grows monotonically with no outflow path for the slow-mode fee portion.
