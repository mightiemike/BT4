### Title
Gateway Admission Validates Gas Price Against Previous Block While Execution Enforces Current Block Price, Causing Admitted Transactions to Fail During Sequencing — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's stateful validator admits transactions by checking `max_price_per_unit` against the **previous** block's L2 gas price. The blockifier's pre-validation inside the batcher enforces the **current** block's gas price. When EIP-1559 raises the gas price between blocks, transactions admitted at the boundary price pass gateway admission but are rejected during sequencing with `InsufficientResourceBounds`, matching the external report's pattern of a "max" check that omits a downstream cost.

### Finding Description

`StatefulTransactionValidator::run_validate_entry_point` builds its `BlockContext` from `gateway_fixed_block_state_reader.get_block_info()`, which returns the latest **committed** block's `BlockInfo`. The blockifier's `perform_pre_validation_stage` then enforces:

```rust
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(ResourceBoundsError::MaxGasPriceTooLow { … });
}
```

using that stale price. A transaction with `l2_gas.max_price_per_unit = previous_block_l2_gas_price` therefore passes gateway validation. [1](#0-0) 

The batcher builds the **next** block's `BlockContext` with a freshly computed EIP-1559 gas price. The consensus `VersionedConstants` caps the per-block price change at `previous_price / gas_price_max_change_denominator`, so the price can legitimately rise above the previous block's value whenever gas usage exceeds `gas_target`. [2](#0-1) 

When the batcher calls `perform_pre_validation_stage` with the new, higher price, the same transaction fails: [3](#0-2) 

The stateful validator's earlier `validate_resource_bounds` call only checks L2 gas price and explicitly skips L1 and L1-data gas prices (marked with a TODO), widening the gap further: [4](#0-3) 

This is the direct sequencer analog of the external report: `maxWithdraw` delegates to `adapter.maxWithdraw` (previous-block price), while `withdraw` adds fee shares (current-block price enforcement), causing the downstream call to revert.

### Impact Explanation

**High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

Any user can craft a transaction with `l2_gas.max_price_per_unit` equal to the previous block's L2 gas price. The gateway admits it. During any block where EIP-1559 raises the price (congestion), the batcher's pre-validation rejects it with `MaxGasPriceTooLow`. The transaction occupies a mempool slot and wastes a sequencing attempt without ever executing. At scale this degrades throughput and misleads users who relied on the gateway's acceptance as a correctness signal.

### Likelihood Explanation

**Medium.** The condition triggers whenever the network is congested enough for EIP-1559 to raise the L2 gas price between consecutive blocks. With the default `min_gas_price_percentage = 100`, there is zero buffer: any upward price tick invalidates boundary-priced transactions. The TODO comment in the source confirms the developers are aware the check is incomplete. [5](#0-4) 

### Recommendation

1. Compute the **expected next-block** L2 gas price (applying the EIP-1559 formula once to the previous block's price) and use that as the admission threshold, matching what the batcher will enforce.
2. Extend `validate_tx_l2_gas_price_within_threshold` to cover L1 gas and L1-data gas prices (resolve the existing TODO).
3. Raise the default `min_gas_price_percentage` to at least `100 + ceil(100 / gas_price_max_change_denominator)` to absorb one full EIP-1559 step.

### Proof of Concept

```
Previous block L2 gas price:  P  (e.g. 1 000 000)
gas_price_max_change_denominator: 8   → max increase per block = P/8 = 125 000

User submits:  l2_gas.max_price_per_unit = P = 1 000 000
Gateway check: P >= 100% * P  → ADMITTED

EIP-1559 raises next-block price to P' = 1 125 000  (block is full)

Batcher pre-validation: 1 000 000 < 1 125 000  → MaxGasPriceTooLow → REJECTED
```

The transaction passes `run_validate_entry_point` at the gateway (which uses the old `BlockInfo`) but is dropped by `perform_pre_validation_stage` in the batcher (which uses the new `BlockContext`). [6](#0-5) [7](#0-6)

    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
