Audit Report

## Title
`SwapAllowlistExtension` checks the router's address as `sender` instead of the actual end-user, enabling any caller to bypass per-user swap restrictions when the router is allowlisted — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension` is documented as gating swaps "by swapper address," but the `sender` it receives and checks is `msg.sender` of the pool call — the direct pool caller. When `MetricOmmSimpleRouter` intermediates, `sender` is the router contract, not the actual end-user. A pool admin who allowlists the router (to permit router-based swaps for approved users) inadvertently grants every user the ability to bypass the per-user restriction by routing through it.

## Finding Description
**Root cause — wrong identity propagated to the extension:**

`MetricOmmPool.swap` passes `msg.sender` (the direct pool caller) as `sender` to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
  msg.sender,   // direct pool caller, not the end-user
  ...
);
```

`ExtensionCalling._beforeSwap` forwards that value unchanged into the encoded extension call:

```solidity
// ExtensionCalling.sol L160-176
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the direct pool caller. When `MetricOmmSimpleRouter.exactInputSingle` is used, the router calls `pool.swap()` directly:

```solidity
// MetricOmmSimpleRouter.sol L72-80
IMetricOmmPoolActions(params.pool).swap(
    params.recipient, params.zeroForOne, ...
);
```

So the pool sees `msg.sender = router`, and the extension evaluates `allowedSwapper[pool][router]` — not `allowedSwapper[pool][end_user]`.

**Exploit flow:**
1. Pool is deployed with `SwapAllowlistExtension` configured.
2. Pool admin allowlists specific user addresses (e.g., KYC'd traders) and also allowlists the router address so those users can swap via the router.
3. Any unprivileged user calls `MetricOmmSimpleRouter.exactInputSingle` targeting that pool.
4. The pool sees `sender = router`; the extension finds `allowedSwapper[pool][router] == true` and passes.
5. The unprivileged user's swap executes despite not being individually allowlisted.

**Why existing guards fail:**
The extension has no mechanism to distinguish the actual end-user from the router. The `allowAllSwappers` flag and `allowedSwapper` mapping both key on `sender`, which collapses to the router address for all router-mediated swaps. There is no secondary check on `recipient` or any user-supplied identity field.

**Contrast with `DepositAllowlistExtension`:**
`DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` — a parameter explicitly separate from `sender` (the direct caller) — so the actual position owner is always verified regardless of who calls `addLiquidity`. The swap path has no equivalent separate identity parameter.

## Impact Explanation
Broken core pool functionality: the `SwapAllowlistExtension` access control is fully bypassed for any user who routes through `MetricOmmSimpleRouter` when the router is allowlisted. A pool intended to restrict swaps to a specific set of addresses (e.g., permissioned/institutional pools) becomes open to all callers via the router, violating the pool's access policy and potentially exposing LP funds to unintended counterparties.

## Likelihood Explanation
Any unprivileged user can exploit this by calling `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`). The only precondition is that the pool admin has allowlisted the router — a natural and expected configuration for any pool that wants to support router-based swaps alongside a user allowlist. The router is a public, permissionless contract, so no special access is required by the attacker.

## Recommendation
Pass the actual end-user identity through the swap call chain. One approach: add an explicit `swapper` (or `originator`) address parameter to `IMetricOmmExtensions.beforeSwap` that the pool populates from a trusted source (e.g., a callback-context field set by the router before calling `pool.swap`, analogous to how `_setNextCallbackContext` already stores the payer). `SwapAllowlistExtension` should then check `allowedSwapper[pool][swapper]` against that field rather than `sender`. Alternatively, mirror the `addLiquidity` pattern: require callers to pass an explicit `swapper` address and enforce `msg.sender == swapper || isApprovedRouter(msg.sender)` at the pool level before forwarding to extensions.
