Audit Report

## Title
Swap Allowlist Bypassed via Router: `beforeSwap` Gates on Router Address Instead of Actual Trader - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument — which the pool sets to `msg.sender` of `pool.swap()` — against the per-pool allowlist. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router's address, not the user's. A pool admin who allowlists the router to permit router-based swaps for approved users inadvertently grants every user who calls through the router the ability to bypass the allowlist, defeating the access control entirely.

## Finding Description

**Call chain:**

1. Any user calls `MetricOmmSimpleRouter.exactInputSingle` (or any `exact*` variant). The router calls `IMetricOmmPoolActions(params.pool).swap(...)` directly, so inside `MetricOmmPool.swap`, `msg.sender` = router address.

2. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` passing the router address as `sender`:
   ```solidity
   // metric-core/contracts/MetricOmmPool.sol:230-240
   _beforeSwap(
       msg.sender,   // <-- router address
       recipient,
       ...
   );
   ```

3. `ExtensionCalling._beforeSwap` forwards `sender` unchanged to the extension:
   ```solidity
   // metric-core/contracts/ExtensionCalling.sol:160-176
   _callExtensionsInOrder(
       BEFORE_SWAP_ORDER,
       abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
   );
   ```

4. `SwapAllowlistExtension.beforeSwap` receives `sender` = router address and checks:
   ```solidity
   // metric-periphery/contracts/extensions/SwapAllowlistExtension.sol:37
   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
       revert IMetricOmmPoolActions.NotAllowedToSwap();
   }
   ```
   Here `msg.sender` = pool, `sender` = router. The check resolves to `allowedSwapper[pool][router]`.

**Exploit flow:**
- Pool admin deploys a restricted pool with `SwapAllowlistExtension` and allowlists only specific user addresses.
- Pool admin also calls `setAllowedToSwap(pool, routerAddress, true)` to allow router-mediated swaps for those approved users.
- Any unprivileged user (not in the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle` targeting the restricted pool.
- `beforeSwap` receives `sender = routerAddress`, finds `allowedSwapper[pool][routerAddress] == true`, and passes — the unapproved user swaps successfully.

**Existing guards are insufficient:** The `onlyPool` modifier on `BaseMetricExtension.beforeSwap` only ensures the caller is a registered pool; it does not verify the identity of the original trader. There is no mechanism in the extension to recover the actual end-user address from the router call.

## Impact Explanation
The swap allowlist is a core access-control mechanism for pool operators who need to restrict trading to approved counterparties (e.g., KYC'd users, whitelisted market makers). The bypass allows any unprivileged user to trade in a pool that should be restricted, breaking the intended access control invariant. This constitutes broken core pool functionality and an admin-boundary break reachable via an unprivileged public path (the router).

## Likelihood Explanation
The condition is trivially reachable: any user with access to the public `MetricOmmSimpleRouter` can exploit this whenever the router address is allowlisted on a restricted pool. No special privileges, flash loans, or complex setup are required. The attacker only needs to know the pool address and call the router's standard swap functions. The scenario where a pool admin allowlists the router is the natural and expected configuration for router-based pools.

## Recommendation
The extension must check the actual end-user identity, not the intermediary router. Two approaches:

1. **Pass the original caller through `extensionData`:** Have the router encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and verify it. This requires the router to be trusted to populate this field honestly, which is acceptable since the router is a known periphery contract.

2. **Check `recipient` instead of `sender`:** If the pool's design guarantees that `recipient` is always the actual beneficiary, the extension could gate on `recipient`. However, `recipient` can be set to any address by the caller, so this is not a reliable identity check.

3. **Preferred fix — two-level check:** When `sender` is a known router, require an additional field in `extensionData` that carries the verified original caller (set by the router and validated by the extension). Alternatively, redesign the allowlist to gate on `recipient` only when `sender` is a trusted router, and gate on `sender` otherwise.

## Proof of Concept

```solidity
// Foundry test sketch
function test_routerBypassesSwapAllowlist() public {
    // Setup: pool with SwapAllowlistExtension, only routerAddress allowlisted
    extension.setAllowedToSwap(pool, address(router), true);
    // attacker is NOT allowlisted
    assertFalse(extension.isAllowedToSwap(pool, attacker));

    // Attacker calls router — sender seen by extension = router address
    vm.prank(attacker);
    router.exactInputSingle(ExactInputSingleParams({
        pool: pool,
        recipient: attacker,
        zeroForOne: true,
        amountIn: 1000,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    }));
    // Swap succeeds — allowlist bypassed
}
