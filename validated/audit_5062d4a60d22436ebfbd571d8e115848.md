### Title
Fee-on-Transfer Token Deposit Overcrediting Breaks Solvency Invariant — (`core/contracts/Clearinghouse.sol`)

---

### Summary
`Clearinghouse.depositCollateral` credits the caller-specified `txn.amount` to a subaccount's internal balance without measuring the actual token amount received by the contract. For fee-on-transfer tokens, the clearinghouse receives `amount − fee` but records the full `amount`, permanently inflating `totalDepositsNormalized`. The `assertUtilization` guard only compares internal accounting figures and never checks the real token balance, so the discrepancy is invisible to the protocol until withdrawals fail.

---

### Finding Description

**Root cause — `Clearinghouse.depositCollateral`:**

```solidity
// Clearinghouse.sol lines 205-208
int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
```

`txn.amount` is the value encoded in the sequencer-submitted transaction. The actual ERC-20 `transferFrom` that moves tokens into the clearinghouse happens upstream (in `Endpoint.depositCollateralWithReferral` / `handleDepositTransfer`). For a standard ERC-20 the two values are equal. For a fee-on-transfer token they diverge: the clearinghouse receives `amount − fee` but `spotEngine.updateBalance` is called with the full `amount`, inflating `totalDepositsNormalized` by the fee amount on every deposit. [1](#0-0) 

**Why the guard does not catch it — `SpotEngine.assertUtilization`:**

```solidity
// SpotEngine.sol lines 232-241
function assertUtilization(uint32 productId) external view {
    ...
    int128 totalDeposits = _state.totalDepositsNormalized.mul(
        _state.cumulativeDepositsMultiplierX18
    );
    int128 totalBorrows = _state.totalBorrowsNormalized.mul(
        _state.cumulativeBorrowsMultiplierX18
    );
    require(totalDeposits >= totalBorrows, ERR_MAX_UTILIZATION);
}
```

This check is called on every `withdrawCollateral`. It compares two internal accounting figures; it never calls `token.balanceOf(address(this))`. The inflated `totalDepositsNormalized` satisfies the inequality even though the clearinghouse physically holds fewer tokens than the sum of all credited balances. [2](#0-1) 

**`DirectDepositV1.creditDeposit` is not affected** because it reads `token.balanceOf(address(this))` before calling `depositCollateralWithReferral`, so the amount passed already reflects any fee deduction. The vulnerability is confined to the standard deposit path where `txn.amount` is set by the sequencer to match the user's intent, not the actual received amount. [3](#0-2) 

---

### Impact Explanation
Every deposit of a fee-on-transfer token inflates `totalDepositsNormalized` by the fee amount. Over time, the sum of all credited balances exceeds the real token balance held by the clearinghouse. Withdrawals are processed in FIFO order; early withdrawers receive their full credited amount while later withdrawers find the clearinghouse insolvent. The shortfall is socialised across remaining depositors via `socializeSubaccount`, causing permanent loss of principal for honest depositors who did not deposit a fee-on-transfer token. [4](#0-3) 

---

### Likelihood Explanation
The `addOrUpdateProduct` function in `SpotEngine` contains no on-chain check that rejects fee-on-transfer tokens. Any ERC-20 address can be registered as a product. The protocol is deployed on Ink Chain and is designed to support arbitrary spot tokens; the absence of a token-type guard means a fee-on-transfer token can be added inadvertently or deliberately. Once added, every deposit by any user through the standard path triggers the accounting inflation — no special attacker capability is required beyond making a normal deposit call. [5](#0-4) 

---

### Recommendation
1. In `Clearinghouse.depositCollateral`, replace the use of `txn.amount` with a pre/post balance measurement:
   ```solidity
   uint256 pre = token.balanceOf(address(this));
   // transfer already happened upstream
   uint256 post = token.balanceOf(address(this));
   uint128 actualReceived = uint128(post - pre);
   ```
   Credit `actualReceived` instead of `txn.amount`.
2. Alternatively, add an explicit on-chain check in `addOrUpdateProduct` that rejects tokens whose `transferFrom` delivers a different amount than requested (a standard fee-on-transfer detection pattern).
3. Document that fee-on-transfer and rebasing tokens are unsupported and enforce this at the product-registration layer.

---

### Proof of Concept

1. Admin registers a fee-on-transfer token `FTT` (1% fee) as product ID `P`.
2. Alice calls `Endpoint.depositCollateralWithReferral(aliceSubaccount, P, 1000e18, "")`.
   - `handleDepositTransfer` calls `FTT.transferFrom(Alice, clearinghouse, 1000e18)`.
   - Clearinghouse receives `990e18` (1% fee deducted).
3. Sequencer submits `DepositCollateral{sender: aliceSubaccount, productId: P, amount: 1000e18}`.
   - `Clearinghouse.depositCollateral` calls `spotEngine.updateBalance(P, aliceSubaccount, 1000e18 * multiplier)`.
   - `totalDepositsNormalized` is inflated by `10e18 * multiplier` relative to actual holdings.
4. Alice immediately withdraws `1000e18`. `assertUtilization` passes (internal accounting is consistent). Clearinghouse transfers `990e18` it holds plus `10e18` from other depositors' funds.
5. Bob, who deposited `1000e18` of `FTT` earlier (clearinghouse received `990e18`), now finds only `980e18` remaining. His withdrawal of `1000e18` fails or is partially socialised. [6](#0-5) [7](#0-6)

    function _updateBalanceNormalized(
        State memory state,
        BalanceNormalized memory balance,
        int128 balanceDelta
    ) internal pure {
        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized -= balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized += balance.amountNormalized;
        }

        int128 cumulativeMultiplierX18;
        if (balance.amountNormalized > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        int128 newAmount = balance.amountNormalized.mul(
            cumulativeMultiplierX18
        ) + balanceDelta;

        if (newAmount > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        balance.amountNormalized = newAmount.div(cumulativeMultiplierX18);

        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized += balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized -= balance.amountNormalized;
        }
    }
