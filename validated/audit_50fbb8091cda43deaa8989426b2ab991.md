### Title
WithdrawPool Liquidity Drained via Health-Check-Free `submitFastWithdrawal` Front-Running Liquidation — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.submitFastWithdrawal` is a public, permissionless function that transfers tokens to a user immediately upon presentation of valid sequencer-signed withdrawal bytes, with **no account health check**. The normal on-chain withdrawal path (`Clearinghouse.withdrawCollateral`) enforces a strict `getHealth >= 0` guard. A user whose account is about to be liquidated can call `submitFastWithdrawal` directly on the `WithdrawPool` contract before the sequencer processes the liquidation on-chain, extracting collateral that the health check would have blocked. When the sequencer later attempts to settle the withdrawal on-chain, the health check in `Clearinghouse.withdrawCollateral` reverts the entire call, preventing the `WithdrawPool` from being replenished. The pool suffers a direct, unrecoverable token loss.

---

### Finding Description

**Normal withdrawal path** (`EndpointTx.processTransactionImpl` → `Clearinghouse.withdrawCollateral`): [1](#0-0) 

The critical guard is at line 419:

```solidity
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
```

This prevents any withdrawal that would leave the account below initial margin.

**Fast withdrawal path** (`BaseWithdrawPool.submitFastWithdrawal`): [2](#0-1) 

The function performs only three checks:

```solidity
require(!markedIdxs[idx], "Withdrawal already submitted");
require(idx > minIdx, "idx too small");
// ... requireValidTxSignatures ...
``` [3](#0-2) 

There is **no call to `getHealth`**, no check that the subaccount is solvent, and no check that the account has not been liquidated. The function immediately transfers tokens from the pool's own balance to the user.

**Settlement reconciliation** (`Clearinghouse.handleWithdrawTransfer` → `BaseWithdrawPool.submitWithdrawal`): [4](#0-3) [5](#0-4) 

When the sequencer later processes the withdrawal on-chain, `Clearinghouse.withdrawCollateral` calls `handleWithdrawTransfer`, which transfers tokens to the `WithdrawPool` and calls `submitWithdrawal`. Because `markedIdxs[idx]` is already `true` (set by the earlier `submitFastWithdrawal`), `submitWithdrawal` returns early — the pool is supposed to keep those tokens as replenishment. However, `withdrawCollateral` then deducts the balance from the subaccount and runs the health check. If the account was liquidated in the interim, the health check at line 419 **reverts the entire transaction**, including the token transfer to the pool. The pool is never replenished.

---

### Impact Explanation

The `WithdrawPool` suffers a direct, permanent loss of tokens equal to the full withdrawal amount. The pool's liquidity providers bear this loss. Because the pool is designed to front multiple withdrawals simultaneously, a single coordinated exploit can drain a significant portion of pool liquidity. The corrupted state is the `WithdrawPool`'s ERC-20 token balance: it decreases by `transferAmount - fee` with no corresponding on-chain replenishment.

---

### Likelihood Explanation

The precondition is that the attacker must possess a sequencer-signed withdrawal transaction (i.e., the sequencer's multi-sig signatures for a specific `idx`). This is a normal artifact of the fast-withdrawal service: any user who has requested a fast withdrawal and received signed bytes from the sequencer holds this capability. The window of opportunity exists whenever the sequencer has queued a liquidation for the same account. On a live chain with observable mempool or known sequencer batch ordering, this is a realistic, repeatable attack requiring no privileged access beyond a normal user interaction.

---

### Recommendation

- **Short term:** Add a health check inside `submitFastWithdrawal` before transferring funds. Call `IClearinghouse(clearinghouse).getHealth(sender, IProductEngine.HealthType.INITIAL) >= 0` and revert if the account is unhealthy. Alternatively, implement a two-step fast-withdrawal: mark the intent, enforce a delay, then settle — mirroring the slow-mode queue pattern already present in `Endpoint`.
- **Long term:** Audit all paths that bypass the sequencer's transaction ordering (i.e., any public function on `WithdrawPool` or `Endpoint` that can be called directly) and verify that each one enforces the same invariants as the sequencer-gated path.

---

### Proof of Concept

1. Eve's account holds 10,000 USDC as collateral and has an open perp position. The account is healthy.
2. Eve submits a `WithdrawCollateralV2` request to the sequencer for 9,000 USDC. The sequencer signs it and assigns `idx = 500`.
3. A price drop causes Eve's account to fall below maintenance margin.
4. The sequencer queues on-chain: `[LiquidateSubaccount(Eve), WithdrawCollateral(Eve, idx=500)]`.
5. Eve observes the pending liquidation and immediately calls `WithdrawPool.submitFastWithdrawal(500, withdrawalTx, signatures)`.
6. `submitFastWithdrawal` passes all three checks (not marked, `500 > minIdx`, valid sigs) and transfers 9,000 USDC (minus fee) from the pool to Eve. `markedIdxs[500] = true`.
7. The sequencer processes `LiquidateSubaccount(Eve)` on-chain — Eve's collateral is seized.
8. The sequencer processes `WithdrawCollateral(Eve, idx=500)` on-chain. `Clearinghouse.withdrawCollateral` calls `handleWithdrawTransfer`, sending 9,000 USDC to the pool and calling `submitWithdrawal(idx=500)` — which returns early because `markedIdxs[500]` is true.
9. `withdrawCollateral` then deducts 9,000 USDC from Eve's (now near-zero) balance and calls `getHealth` — health is deeply negative → **REVERT**.
10. The entire transaction reverts, including the 9,000 USDC transfer to the pool. The pool has paid Eve 9,000 USDC and received nothing back. [2](#0-1) [6](#0-5) [4](#0-3) [7](#0-6)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
