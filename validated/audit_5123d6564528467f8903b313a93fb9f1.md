### Title
Unrestricted `run_mainchain_gc` Allows Any Caller to Aggressively Prune Mainchain Headers — (File: `contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` is a public, state-mutating function with no caller restriction when the contract is not paused. Any unprivileged NEAR account can call it with an arbitrarily large `batch_size`, removing the maximum possible number of mainchain block headers in a single transaction. This permanently corrupts the canonical chain state by destroying headers that are required for fork resolution and SPV proof verification.

---

### Finding Description

`run_mainchain_gc` is decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`. That macro enforces a role check exclusively when the contract is **paused**; when the contract is running normally, the decorator is a no-op and the function is callable by any NEAR account with no restriction. [1](#0-0) 

By contrast, `submit_blocks` — the only other function that calls `run_mainchain_gc` internally — carries the `#[trusted_relayer]` attribute, which enforces that only staked/approved relayers or accounts holding `Role::UnrestrictedSubmitBlocks` / `Role::DAO` may invoke it. [2](#0-1) 

`run_mainchain_gc` removes the oldest mainchain entries from both `mainchain_height_to_header` and `headers_pool`, then advances `mainchain_initial_blockhash` forward. The `batch_size` parameter is fully attacker-controlled and is only capped by `total_amount_to_remove` (the number of blocks currently exceeding `gc_threshold`). [3](#0-2) 

---

### Impact Explanation

**1. Permanent reorg failure.** When a fork accumulates more chainwork than the main chain, `reorg_chain` walks backward through `headers_pool` from the fork tip to the common ancestor. If the attacker has already GC'd the common ancestor block, the walk panics with `"previous fork block should be there"` and the reorg can never complete. The fork is permanently stuck, and the canonical chain tip is frozen at a lower-work chain. [4](#0-3) 

**2. SPV proof invalidation.** `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` look up the target block in `mainchain_header_to_height`. If the attacker has GC'd that block, the call panics with `"block does not belong to the current main chain"`, permanently breaking any downstream contract that relies on the light client to confirm a Bitcoin transaction. [5](#0-4) 

Both effects are **irreversible**: deleted entries in `LookupMap` cannot be recovered without a full contract re-initialization.

---

### Likelihood Explanation

The entry point is a standard NEAR contract call requiring no stake, no role, and no deposit. Any account can call `run_mainchain_gc({"batch_size": 18446744073709551615})` at any time the contract is not paused. The attacker does not need to know anything about the chain state beyond the contract address. The attack is cheap (single transaction, gas only) and can be repeated after every new batch of blocks is submitted by the relayer.

---

### Recommendation

Add a caller restriction to `run_mainchain_gc` equivalent to the one on `submit_blocks`. The simplest correct fix is to add `#[trusted_relayer]` (or an explicit `require!(env::predecessor_account_id() == env::current_account_id() || self.acl_has_role(...))` check) so that only authorized relayers and DAO accounts can invoke it directly. The internal call from `submit_blocks` is already protected by `submit_blocks`'s own `#[trusted_relayer]` guard and would remain unaffected.

---

### Proof of Concept

1. Relayer submits enough blocks that `mainchain_size > gc_threshold` (e.g., `gc_threshold = 52704`, chain has 53000 blocks stored).
2. A fork exists branching from block at height 52500 (within the "excess" window of 296 blocks).
3. Attacker calls:
   ```
   run_mainchain_gc({"batch_size": 18446744073709551615})
   ```
   This removes all 296 excess blocks (heights 0–295 relative to `mainchain_initial_blockhash`), including the common ancestor at height 52500.
4. The fork's accumulated chainwork now exceeds the main chain. The relayer submits the next fork block, triggering `reorg_chain`.
5. `reorg_chain` walks back through `headers_pool` looking for the common ancestor at height 52500 — it has been deleted. The contract panics with `"previous fork block should be there"`.
6. The canonical chain tip is permanently frozen; no further reorgs can complete; all SPV proofs for blocks below height 52500 fail with `"block does not belong to the current main chain"`. [6](#0-5)
