### Title
Accrued Bonus Rewards Permanently Forfeited When Rewards Pool Is Empty During Redelegation Slash — (File: `x/tieredrewards/keeper/slash.go`)

---

### Summary

In `slashRedelegationPosition`, when `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, the error is silently swallowed. However, by the time the error is returned, the position's bonus checkpoints (`LastBonusAccrual`, `LastEventSeq`, `LastKnownBonded`) have already been advanced in-memory, and event reference counts have already been decremented in the store. The caller then persists the modified position via `setPositionWithState`. The result is that the user's accrued bonus for the entire period up to the slash is permanently forfeited — the pool balance is not guaranteed to cover all users' accrued rewards, and when it cannot, the reward entitlement is silently destroyed rather than preserved for a later retry.

---

### Finding Description

`processEventsAndClaimBonus` advances all bonus checkpoints **before** checking whether the pool can cover the computed bonus: [1](#0-0) 

Inside the event loop, `pos.UpdateLastEventSeq(entry.Seq)` and `k.decrementEventRefCount` (a store write) are called for every event before the pool check: [2](#0-1) 

After the loop, `applyBonusAccrualCheckpoint` and `UpdateLastKnownBonded` are called before `sufficientBonusPoolBalance`: [3](#0-2) 

In `slashRedelegationPosition`, the `ErrInsufficientBonusPool` error is caught and execution continues with the already-mutated `pos`: [4](#0-3) 

The modified `pos` — with `LastBonusAccrual` advanced to `blockTime`, `LastEventSeq` advanced past all pending events, and `LastKnownBonded` updated — is then persisted via `setPositionWithState`. The accrued bonus for the entire elapsed period is gone: the checkpoints will never replay those events again, and the pool may be replenished later but the user's entitlement for that window is destroyed.

---

### Impact Explanation

A position owner whose position is in a redelegating state loses all accrued bonus rewards for the period from `LastBonusAccrual` to the slash block time. The corrupted state is:

- `pos.LastBonusAccrual` — advanced to `blockTime` without payment
- `pos.LastEventSeq` — advanced past all pending validator events without payment
- Event reference counts in the store — decremented (events garbage-collected) without the corresponding bonus being paid

This is permanent: no retry path exists. User-driven paths (`MsgClaimTierRewards`, `MsgTierRedelegate`, etc.) fail atomically and allow retry, but the `BeforeRedelegationSlashed` hook silently advances the checkpoints and persists them, closing the window forever. [5](#0-4) 

---

### Likelihood Explanation

Three conditions must coincide:

1. A tier position is in a redelegating state (the owner has called `MsgTierRedelegate` and the redelegation has not yet matured).
2. The destination validator is slashed (double-sign evidence or downtime — a normal protocol event).
3. The `RewardsPoolName` module account balance is zero or below the computed bonus at slash time.

Condition 3 is realistic: the `BeginBlocker` drains the pool every block to top up base rewards when the fee collector is short, and the pool can reach zero naturally. [6](#0-5) 

The pool draining to zero is explicitly tested and expected: [7](#0-6) 

---

### Recommendation

Do not advance bonus checkpoints before confirming the pool can cover the bonus. Two concrete fixes:

1. **Snapshot and restore**: Save `pos.LastBonusAccrual`, `pos.LastEventSeq`, and `pos.LastKnownBonded` before calling `processEventsAndClaimBonus`. On `ErrInsufficientBonusPool`, restore the snapshot and persist the un-advanced position so the user can claim later.

2. **Separate checkpoint advancement from payment**: Restructure `processEventsAndClaimBonus` so that the pool check and payment occur before any checkpoint mutation. If the pool check fails, return without modifying `pos` or decrementing event reference counts.

The current design documents this as intentional ("chain-halt avoidance"), but the trade-off silently destroys earned user rewards rather than deferring them. A safer alternative is to skip the bonus settlement entirely in the hook (leave checkpoints unchanged) and let the user claim the pre-slash bonus via `MsgClaimTierRewards` after the pool is replenished.

---

### Proof of Concept

1. User calls `MsgLockTier` to create a position on validator V1.
2. User calls `MsgTierRedelegate` to move the delegation to validator V2. A `RedelegationMapping` entry is created.
3. The `RewardsPoolName` balance drains to zero (via BeginBlocker top-ups over several blocks, or via other users claiming bonus rewards).
4. V2 is slashed (double-sign evidence submitted). The staking module calls `BeforeRedelegationSlashed` with the position's `unbondingId`.
5. `slashRedelegationPosition` calls `processEventsAndClaimBonus`:
   - Event reference counts are decremented in the store.
   - `pos.LastBonusAccrual` is advanced to `blockTime`.
   - `pos.LastEventSeq` is advanced past all pending events.
   - `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`.
6. The error is caught at `slash.go:56`. Execution continues.
7. `setPositionWithState` persists the modified `pos` with advanced checkpoints.
8. The user later calls `MsgClaimTierRewards`. `processEventsAndClaimBonus` finds no pending events (all consumed) and `LastBonusAccrual` is already at the slash time. Zero bonus is returned. The accrued bonus for the entire pre-slash period is permanently lost. [8](#0-7) [9](#0-8) [10](#0-9)

func (k Keeper) slashRedelegationPosition(ctx context.Context, unbondingId uint64, sharesToUnbond math.LegacyDec) error {
	positionId, err := k.getRedelegationMapping(ctx, unbondingId)
	if errors.Is(err, collections.ErrNotFound) {
		return nil
	}
	if err != nil {
		return err
	}

	pos, err := k.getPositionState(ctx, positionId)
	if errors.Is(err, types.ErrPositionNotFound) {
		k.logger(ctx).Error("position not found during redelegation slash",
			"position_id", positionId,
			"unbonding_id", unbondingId,
			"error", err.Error(),
		)
		return nil
	}
	if err != nil {
		return err
	}

	if !pos.IsDelegated() {
		// Defensive
		k.logger(ctx).Error("delegation missing during BeforeRedelegationSlashed",
			"position_id", positionId,
			"unbonding_id", unbondingId,
			"shares_to_unbond", sharesToUnbond.String(),
		)
		return nil
	}

	dstValStr := pos.Delegation.ValidatorAddress

	// Settle bonus against PRE-slash shares.
	if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
		// Deliberately forgo bonus rewards if pool is insufficient to prevent chain halt.
		if errors.Is(err, types.ErrInsufficientBonusPool) {
			k.logger(ctx).Error("insufficient bonus pool during redelegation slash",
				"position_id", pos.Id,
				"error", err.Error(),
			)
		} else {
			return err
		}
	}

	fullSlash := sharesToUnbond.GTE(pos.Delegation.Shares)

	if fullSlash {
		pos.Delegation = nil
		pos.ClearBonusCheckpoints()
		return k.setPositionWithState(ctx, pos, &ValidatorTransition{PreviousAddress: dstValStr})
	}
	// In-memory only: the persisted Position carries no share count, and the
	// live delegation will reflect the post-Unbond shares on the next read.
	// Update the local copy so any follow-up logic in this call sees consistent state.
	pos.Delegation.Shares = pos.Delegation.Shares.Sub(sharesToUnbond)
	return k.setPositionWithState(ctx, pos, nil)

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
