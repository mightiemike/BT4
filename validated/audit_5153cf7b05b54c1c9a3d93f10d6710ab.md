Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Original Caller, Allowing Allowlist Bypass via Router â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension::beforeSwap` receives `sender = router_address` (the pool's immediate `msg.sender`) when a swap is routed through `MetricOmmSimpleRouter`. The hook checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][original_user]`. Any unprivileged user can bypass the per-pool swap allowlist by routing through the router if the router is allowlisted, completely nullifying the access control the extension was designed to enforce.

## Finding Description
When a user calls `MetricOmmSimpleRouter::exactInputSingle` (or any `exact*` variant), the router calls `IMetricOmmPoolActions(params.pool).swap(...)` directly. At this point, `msg.sender` inside `MetricOmmPool::swap` is the router address. The pool then calls `_beforeSwap(msg.sender, ...)`, passing the router address as `sender`. `ExtensionCalling::_beforeSwap` encodes that router address and calls the extension via `_callExtensionsInOrder`. `SwapAllowlistExtension::beforeSwap` evaluates `allowedSwapper[msg.sender][sender]` â€” i.e., `allowedSwapper[pool][router]`. The original user's identity is stored only in transient callback context (`_setNextCallbackContext`) for payment purposes and never reaches the extension. There is no mechanism in the hook, pool, or router to propagate the original caller's identity through the pool boundary. This creates a binary failure: if the router is not allowlisted, all allowlisted users are blocked from using the router; if the router is allowlisted (the only way to enable router-mediated swaps), every user on the network can bypass the allowlist by routing through the router.

## Impact Explanation
The `SwapAllowlistExtension` is the protocol's mechanism for restricting swaps to a specific set of addresses per pool (e.g., institutional traders, whitelisted market makers). This access control is completely nullified for router-mediated swaps. Any unprivileged user can execute swaps in a restricted pool by calling `MetricOmmSimpleRouter::exactInputSingle` (or any `exact*` variant), provided the router is allowlisted. This breaks the core functionality the extension was designed to enforce â€” an admin-boundary break where an unprivileged path bypasses a pool-level access control check.

## Likelihood Explanation
The router is the standard, documented entrypoint for swaps. Pool admins who want their allowlisted users to use the router must allowlist the router, which opens the pool to all users. This is a near-certain misconfiguration path with no in-protocol warning or mitigation. The attacker requires no special privileges, no capital beyond swap amounts, and the attack is repeatable on every block.

## Recommendation
The extension must verify the original initiator, not the immediate caller. Options:
- Require the router to forward the original caller's address in `extensionData`, and have the extension decode and verify it â€” with the pool or factory enforcing that only trusted routers can set this field.
- Alternatively, document explicitly that pools using `SwapAllowlistExtension` must allowlist individual users and must NOT allowlist the router, and that router-mediated swaps are incompatible with this extension.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router usage for allowlisted users.
3. Unprivileged attacker (not in the allowlist) calls `MetricOmmSimpleRouter::exactInputSingle` targeting the restricted pool.
4. The pool calls `_beforeSwap(router, ...)`. The extension checks `allowedSwapper[pool][router] == true`. The check passes.
5. The attacker's swap executes successfully despite not being allowlisted.
