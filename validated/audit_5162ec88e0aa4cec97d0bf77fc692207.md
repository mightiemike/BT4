### Title
Unrestricted `run_mainchain_gc` Allows Any Caller to Prematurely Purge Mainchain Block Headers - (File: contract/src/lib.rs)

### Summary
`run_mainchain_gc` is a public, state-mutating NEAR contract method with no caller restriction. Any unprivileged account can invoke it with an arbitrarily large `batch_size`, immediately purging all GC-eligible block headers from the mainchain storage maps and advancing `mainchain_initial_blockhash`. This permanently destroys on-chain header records that downstream callers depend on for SPV proof verification.

### Finding Description
`run_mainchain_gc` is declared `pub fn` under the `#[near]` impl block and carries only a `#[pause(except(roles(Role::UnrestrictedRunGC)))]` attribute. [1](#0-0) 

The `#[pause]` decorator controls whether the function is callable when the contract is in a *paused* state; it imposes no restriction on *who* may call the function when the contract is running normally. There is no `#[trusted_relayer]`, no `#[private]`, and no role-gate on the caller identity. Compare with `submit_blocks`, which carries both `#[pause]` and `#[trusted_relayer]`: [2](#0-1) 

Inside `run_mainchain_gc`, when `amount_of_headers_we_store > gc_threshold`, the function removes up to `min(total_amount_to_remove, batch_size)` headers from `mainchain_height_to_header`, `mainchain_header_to_height`, and `headers_pool`, then advances `mainchain_initial_blockhash`: [3](#0-2) 

An attacker passes `batch_size = u64::MAX` to consume the entire eligible removal budget in a single call.

### Impact Explanation
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` look up the target block in `mainchain_header_to_height` and `headers_pool`: [4](#0-3) 

Once a block header is purged by the attacker-triggered GC, both lookup paths panic with `"block does not belong to the current main chain"` or `"cannot find requested transaction block"`. Any cross-chain bridge, DeFi protocol, or user contract that relies on SPV proof verification for those heights will have its verification permanently rejected. The `mainchain_initial_blockhash` pointer is also permanently advanced, so the purge cannot be reversed without a full contract migration.

### Likelihood Explanation
The entry point is a plain NEAR transaction callable by any account with no deposit, no role, and no staking requirement. The only precondition is that the contract holds more headers than `gc_threshold`, which is the normal operating state of a live relay (the recommended `gc_threshold` is 52,704 blocks). An attacker monitoring the chain can front-run any pending SPV verification by submitting `run_mainchain_gc(u64::MAX)` in the same or prior block.

### Recommendation
Add a caller restriction to `run_mainchain_gc` matching the pattern used for `submit_blocks`. The simplest correct fix is to require the caller to hold a designated role (e.g., `Role::DAO` or a new `GCManager` role) before the function body executes, or to mark the function `#[private]` so only the contract itself can invoke it (keeping the internal call from `submit_blocks` intact). [5](#0-4) 

### Proof of Concept
1. Deploy the contract with `gc_threshold = 100` and relay 110 Bitcoin headers via the trusted relayer path (`submit_blocks`). The contract now stores 110 headers; 10 are eligible for GC.
2. From any unprivileged NEAR account (no roles granted), call:
   ```
   run_mainchain_gc(batch_size: 18446744073709551615)
   ```
3. The function computes `total_amount_to_remove = 110 - 100 = 10`, removes all 10 oldest headers from `mainchain_height_to_header`, `mainchain_header_to_height`, and `headers_pool`, and advances `mainchain_initial_blockhash` to height 10.
4. Any subsequent call to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` referencing a block at heights 0–9 now panics, permanently breaking SPV verification for those heights. [6](#0-5)
