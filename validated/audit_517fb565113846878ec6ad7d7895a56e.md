### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (v1) is marked `#[deprecated]` but remains a fully callable public NEAR contract method. Any unprivileged NEAR caller can invoke it directly, bypassing the coinbase Merkle proof check that `verify_transaction_inclusion_v2` adds to prevent the 64-byte transaction Merkle proof forgery attack. A crafted `ProofArgs` payload with a fake 64-byte "transaction" that maps to an internal Merkle tree node will pass the incomplete v1 validation and return `true`, forging transaction inclusion.

### Finding Description

The contract exposes two SPV proof verification methods:

- `verify_transaction_inclusion` (v1): validates only that `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == header.merkle_root`. [1](#0-0) 
- `verify_transaction_inclusion_v2`: adds a mandatory coinbase proof check — it first verifies `compute_root_from_merkle_proof(coinbase_tx_id, 0, coinbase_merkle_proof) == header.merkle_root` before delegating to v1. [2](#0-1) 

The v1 function is annotated `#[deprecated]` to signal that callers should migrate to v2:

```rust
#[deprecated(
    since = "0.5.0",
    note = "Use `verify_transaction_inclusion_v2` instead."
)]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
``` [3](#0-2) 

In Rust, `#[deprecated]` is a **compiler lint only** — it emits a warning when the function is called from Rust source, but it does **not** remove the function from the compiled WASM binary or prevent it from being registered as a NEAR contract endpoint. Because the function is `pub` and carries `#[pause]` (not `#[private]`), it is fully exposed on-chain and callable by any NEAR account.

The 64-byte transaction forgery works as follows: Bitcoin's Merkle tree hashes 64-byte concatenations of child hashes at every internal node. A 64-byte "transaction" is indistinguishable from an internal node at the serialization level. An attacker who knows the Merkle tree of a real block can craft a `tx_id` equal to an internal node hash and supply a shorter proof path that still reconstructs the correct `merkle_root`. The v1 function has no mechanism to reject this — it only checks `!args.merkle_proof.is_empty()` and then computes the root:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

The `compute_root_from_merkle_proof` function in `merkle-tools` performs no depth or leaf-vs-node distinction — it simply iterates the proof array and hashes pairs: [4](#0-3) 

The `ProofArgs` struct accepted by v1 contains only `tx_id`, `tx_block_blockhash`, `tx_index`, `merkle_proof`, and `confirmations` — no coinbase fields: [5](#0-4) 

The analog to the reported vulnerability is direct: the original report shows that a signing function checks for required fields but does not reject extra fields, allowing a crafted payload to pass validation and be reused in a different malicious context. Here, the verification function checks required proof fields but does not enforce the coinbase-depth constraint, allowing a crafted payload (a 64-byte fake transaction mapped to an internal node) to pass validation and forge a `true` inclusion result.

### Impact Explanation

Any bridge, atomic swap, or lending protocol on NEAR that calls `verify_transaction_inclusion` (v1) to confirm a Bitcoin payment will receive a forged `true` for a transaction that was never broadcast or confirmed on Bitcoin. This enables:

- **Direct loss of funds**: a bridge contract releases wrapped BTC or other assets in exchange for a fake BTC deposit proof
- **Permanent freezing of funds**: a lending protocol marks a loan as repaid based on a forged payment proof

The impact matches the target scope (direct loss of funds, permanent freezing of funds).

### Likelihood Explanation

The entry path requires no privileged role, no leaked keys, and no social engineering. Any NEAR account can call `verify_transaction_inclusion` directly. The attacker needs only:
1. A real Bitcoin block that is already accepted by the light client (publicly observable on-chain)
2. Knowledge of that block's Merkle tree (publicly available from any Bitcoin node)
3. Ability to craft a 64-byte value whose double-SHA256 equals an internal Merkle node (this is the known 64-byte forgery technique, not a new cryptographic break — it exploits the structural ambiguity in Bitcoin's Merkle tree)

The technique is documented publicly at https://www.bitmex.com/blog/64-Byte-Transactions and is the exact reason v2 was introduced.

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public contract interface entirely, or gate it with `#[private]` to prevent external calls. Rust's `#[deprecated]` attribute provides zero on-chain access control. Downstream consumers must be migrated to `verify_transaction_inclusion_v2` before v1 is removed.

### Proof of Concept

1. Identify a Bitcoin block already accepted by the light client (e.g., via `get_last_block_header()`).
2. Obtain the full transaction list for that block from a Bitcoin node.
3. Compute the Merkle tree. Select any internal node hash `N` at depth `d` from the root.
4. Construct `ProofArgs`:
   - `tx_id` = `N` (the internal node hash, treated as a fake 64-byte transaction)
   - `tx_block_blockhash` = the real block hash
   - `tx_index` = the index corresponding to `N`'s position at depth `d`
   - `merkle_proof` = the `d`-element sibling path from `N` up to the root
   - `confirmations` = 1
5. Call `verify_transaction_inclusion(args)` directly on the NEAR contract.
6. The function returns `true` for a transaction that does not exist.

The v1 function's only guard against an empty proof (`require!(!args.merkle_proof.is_empty()`) is satisfied by the `d`-element path. [6](#0-5)  The coinbase depth check that would catch this forgery exists only in v2 and is never invoked. [7](#0-6)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
