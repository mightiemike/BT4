Audit Report

## Title
PriceVelocityGuardExtension rolling-baseline bypass via multicall batching allows N×M cumulative same-block price move — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary
`PriceVelocityGuardExtension.beforeSwap` writes `s.lastMidPriceX64` and `s.lastUpdateBlock` to storage **before** performing the velocity check. When `MetricOmmSimpleRouter.multicall` batches N `exactInputSingle` calls in one transaction, each sub-swap is checked only against the previous sub-swap's mid price rather than the block-start mid. N swaps each moving the price by M−ε all pass individually while the total cumulative move reaches N×M, defeating the per-block cap.

## Finding Description
In `beforeSwap`, the storage baseline is advanced unconditionally before the guard check: [1](#0-0) 

Lines 57–58 write `s.lastMidPriceX64 = midPrice` and `s.lastUpdateBlock = uint64(block.number)` before the `if (prevMid != 0)` check at line 60. For swap 1 (cross-block), `blockDiff ≥ 1` and the allowed move is `M * sqrt(1 + blockDiff)`. For swaps 2…N within the same block, `prevBlock` was already set to `block.number` by swap 1, so `blockDiff = block.number - block.number = 0` and `allowedSq = maxChange² * 1`. Critically, `prevMid` is now the **previous sub-swap's mid**, not the block-start mid, so each sub-swap is independently bounded by M while the cumulative move is N×M.

`multicall` enables this in one transaction: [2](#0-1) 

Each `delegatecall` to `exactInputSingle` calls `pool.swap()` → `extension.beforeSwap()` as a normal external call, so `priceVelocityState[pool]` in the extension's own storage is mutated after every hop. The rolling baseline advances with each sub-swap, and no existing guard prevents the accumulation.

## Impact Explanation
The velocity guard exists to bound how far the pool's oracle-derived mid price can move in a single block, protecting LPs from price manipulation. Bypassing it allows an attacker to move the pool price by N×M in one block. LPs are forced to trade at a price N times further from fair value than the guard was designed to permit, resulting in direct loss of LP principal above the intended threshold. This constitutes a bad-price execution impact and direct loss of LP assets above Sherlock thresholds.

## Likelihood Explanation
Any unprivileged user can call `multicall` with an arbitrary array of `exactInputSingle` calls. No special role, factory ownership, or malicious pool setup is required. The attacker only needs to supply the input tokens for each sub-swap. The attack is repeatable every block and requires no off-chain coordination.

## Recommendation
Only advance `lastMidPriceX64` when crossing into a new block. Within the same block, keep `prevMid` as the block-start mid and compare all sub-swaps against that fixed baseline:

```solidity
if (block.number > prevBlock) {
    s.lastMidPriceX64 = midPrice;
    s.lastUpdateBlock = uint64(block.number);
}
// else: do NOT advance the baseline mid-block
```

The velocity check should then always compare `midPrice` against the block-start `prevMid`, regardless of how many swaps have occurred in the current block.

## Proof of Concept
```solidity
// Foundry integration test sketch
function test_velocityGuardBypassViaMulticall() external {
    // Pool configured with PriceVelocityGuardExtension, maxChangePerBlockE18 = M
    // lastMidPriceX64 = P (set in a prior block)

    bytes[] memory calls = new bytes[](3);
    // Each exactInputSingle moves price by M - epsilon
    calls[0] = abi.encodeCall(router.exactInputSingle, (paramsMovingByMEpsilon));
    calls[1] = abi.encodeCall(router.exactInputSingle, (paramsMovingByMEpsilon));
    calls[2] = abi.encodeCall(router.exactInputSingle, (paramsMovingByMEpsilon));

    // All three pass — no PriceVelocityExceeded revert
    router.multicall(calls);

    // Assert cumulative token1 outflow > what a single M-bounded swap would permit
    assertGt(token1BalanceDelta, singleSwapMaxOutflow);
}
```

Each of the three `beforeSwap` calls sees `blockDiff = 0` (after the first) and compares only against the immediately preceding sub-swap's mid. All three pass. The cumulative price move is ≈3M, draining LP token1 balance by the excess beyond the single-swap M cap.
