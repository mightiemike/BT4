### Title
Merkle Leaf Missing Week and Token Binding Enables Cross-Deployment Proof Replay — (File: `core/contracts/Airdrop.sol`)

---

### Summary
The Merkle leaf in `Airdrop._verifyProof()` is computed over only `(sender, totalAmount)`, omitting the `week`, the `token` address, and any contract/chain identifier. A valid proof generated for one `Airdrop` deployment (or one chain) is structurally valid against any other deployment that registers the same Merkle root, allowing a user to claim the same airdrop allocation multiple times across deployments or chains.

---

### Finding Description
In `Airdrop._verifyProof()`, the leaf is constructed as:

```solidity
bytes32 leaf = keccak256(
    bytes.concat(keccak256(abi.encode(sender, totalAmount)))
);
``` [1](#0-0) 

The leaf encodes only `(sender, totalAmount)`. It does not bind to:
- The `week` number (which selects which Merkle root to verify against)
- The `token` address stored in the contract
- The contract address or chain ID

The `token` address is set at initialization and is the sole determinant of which ERC-20 is transferred on a successful claim:

```solidity
SafeERC20.safeTransfer(IERC20(token), msg.sender, totalAmount);
``` [2](#0-1) 

Because the leaf is context-free, any proof valid for deployment A is equally valid for deployment B if B has the same Merkle root registered for any week. The `claimed[week][sender]` guard only prevents re-use within a single deployment:

```solidity
require(claimed[week][sender] == 0, "Already claimed.");
``` [3](#0-2) 

It provides no protection across separate contract instances.

---

### Impact Explanation
If the Nado protocol deploys `Airdrop` on multiple chains (e.g., Ethereum mainnet and an L2) and registers the same Merkle root on both — a standard operational pattern for multi-chain token distributions — every eligible user can claim their full `totalAmount` on each chain using a single proof. The airdrop contract on each chain holds its own token balance, so the attacker receives `N × totalAmount` tokens across `N` deployments, directly draining the airdrop reserves. The corrupted asset delta is the full token balance of every secondary deployment.

---

### Likelihood Explanation
Multi-chain deployment of the same airdrop campaign with a shared Merkle root is a routine practice. The `Airdrop` contract is an upgradeable proxy (`OwnableUpgradeable`, `_disableInitializers`) designed for repeated use across protocol versions. [4](#0-3)  The attack requires no privileged access: any airdrop recipient can call `claim()` on each deployment independently. The only precondition is that the same root is registered on more than one deployment, which is the expected behavior for a multi-chain airdrop.

---

### Recommendation
Bind the leaf to the deployment context by including the `token` address, the `week`, and/or the contract address:

```solidity
bytes32 leaf = keccak256(
    bytes.concat(keccak256(abi.encode(sender, totalAmount, token, week)))
);
```

This ensures a proof generated for one token/week/deployment cannot be replayed against any other. The off-chain Merkle tree generation must be updated to match.

---

### Proof of Concept
1. Protocol deploys `Airdrop` on Chain A (token = `USDC`) and Chain B (token = `USDC` or a different ERC-20). Both deployments call `registerMerkleRoot(1, ROOT)` with the same root `ROOT`.
2. Alice is entitled to 1,000 tokens. Her leaf is `keccak256(bytes.concat(keccak256(abi.encode(alice, 1000))))` — identical on both chains.
3. Alice calls `claim([{week: 1, totalAmount: 1000, proof: P}])` on Chain A → receives 1,000 tokens. `claimed[1][alice] = 1000` on Chain A only.
4. Alice submits the exact same calldata on Chain B. `_verifyProof` on Chain B checks `claimed[1][alice] == 0` (true, independent state), recomputes the same leaf, verifies against the same root `ROOT` with the same proof `P` → passes. Alice receives another 1,000 tokens.
5. Alice has claimed 2,000 tokens against an entitlement of 1,000, draining Chain B's airdrop reserve. [5](#0-4)

    function _verifyProof(
        uint32 week,
        address sender,
        uint256 totalAmount,
        bytes32[] calldata proof
    ) internal {
        require(claimed[week][sender] == 0, "Already claimed.");
        require(
            merkleRoots[week] != bytes32(0),
            "Week hasn't been registered."
        );
        require(
            !ISanctionsList(sanctions).isSanctioned(sender),
            "address is sanctioned."
        );
        bytes32 leaf = keccak256(
            bytes.concat(keccak256(abi.encode(sender, totalAmount)))
        );
        bool isValidLeaf = MerkleProof.verify(proof, merkleRoots[week], leaf);
        require(isValidLeaf, "Invalid proof.");
        claimed[week][sender] = totalAmount;
    }
```
