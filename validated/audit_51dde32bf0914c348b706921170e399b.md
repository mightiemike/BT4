### Title
Missing Zero-Check on `val.DelegatorShares` Before Division in First-Pass Tally — (`File: x/tieredrewards/keeper/gov_tally.go`)

### Summary

`NewCustomTallyTierVotesFn` in `gov_tally.go` divides by `val.DelegatorShares` in the first-pass delegator loop without guarding against a zero denominator. The second pass in the same function and the sibling helper `positionVotingPower` both carry an explicit `IsZero()` guard, making the omission in the first pass an inconsistent and latent bug. A `math.LegacyDec.Quo()` call with a zero denominator panics in Go, which would abort the governance `EndBlocker` and halt the chain.

### Finding Description

`NewCustomTallyTierVotesFn` implements a custom governance tally that adds tier-delegated voting power on top of standard staking power. During the first pass it iterates every voter's delegations:

```go
// gov_tally.go line 86 — no zero guard
votingPower := delegation.GetShares().MulInt(val.BondedTokens).Quo(val.DelegatorShares)
``` [1](#0-0) 

The second pass of the same function, which attributes remaining validator shares to the validator's own vote, explicitly guards the identical expression:

```go
// gov_tally.go lines 141-149 — zero guard present
if val.DelegatorShares.IsZero() {
    continue
}
...
votingPower := sharesAfterDeductions.MulInt(val.BondedTokens).Quo(val.DelegatorShares)
``` [2](#0-1) 

The sibling helper `positionVotingPower` in `voting_power.go` also guards the same pattern:

```go
if !ok || val.DelegatorShares.IsZero() {
    return math.LegacyZeroDec()
}
return pos.Delegation.Shares.MulInt(val.BondedTokens).Quo(val.DelegatorShares)
``` [3](#0-2) 

The developer clearly understood the zero-denominator risk (they added the guard in two other places) but omitted it in the first-pass delegator loop.

### Impact Explanation

`math.LegacyDec.Quo()` panics on a zero denominator. This tally function is invoked from the governance `EndBlocker` when a proposal's voting period expires. A panic there propagates up and halts the chain. All subsequent blocks are blocked until the node is patched and restarted, constituting a chain halt.

### Likelihood Explanation

In normal Cosmos SDK operation a bonded validator always has non-zero `DelegatorShares` because the staking module transitions a validator to unbonded when its last delegation is removed. However, the inconsistency is a latent defect: any future state-transition edge case (e.g., a migration, a slash-to-zero scenario, or a governance-triggered parameter change) that leaves a bonded validator with zero shares would trigger the panic. The fact that the developer added the guard in both the second pass and `positionVotingPower` confirms the scenario was considered a real risk.

### Recommendation

Add the same `IsZero()` guard to the first-pass delegator loop, mirroring the pattern already used in the second pass:

```go
if val, ok := validators[valAddrStr]; ok {
    if val.DelegatorShares.IsZero() {
        // skip — cannot compute voting power without shares
    } else {
        val.DelegatorDeductions = val.DelegatorDeductions.Add(delegation.GetShares())
        validators[valAddrStr] = val
        votingPower := delegation.GetShares().MulInt(val.BondedTokens).Quo(val.DelegatorShares)
        ...
    }
}
```

### Proof of Concept

1. A governance proposal is submitted and enters its voting period.
2. A delegator votes on the proposal.
3. At the moment the proposal's voting period ends, the `EndBlocker` calls `NewCustomTallyTierVotesFn`.
4. The tally function iterates the voter's delegations via `sk.IterateDelegations`.
5. For a delegation whose validator is present in the `validators` map with `DelegatorShares == 0`, execution reaches line 86: `delegation.GetShares().MulInt(val.BondedTokens).Quo(val.DelegatorShares)`.
6. `math.LegacyDec.Quo(zeroDec)` panics.
7. The `EndBlocker` aborts; the chain halts. [4](#0-3)
