### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables `usdcE` Drain Without USDC Payment — (`ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` uses a raw `IERC20Base.transferFrom()` call at line 616 without checking its return value. If the USDC `transferFrom` returns `false` (rather than reverting), execution continues: the function withdraws `usdcE` from the `directDepositV1` contract and sends it to `msg.sender`, even though no USDC was actually received. The contract already imports and uses `ERC20Helper.safeTransferFrom` everywhere else, making this an inconsistent and dangerous omission.

---

### Finding Description

`replaceUsdcEWithUsdc` is an `external` function with no access control beyond a chain ID check. Its intended flow is:

1. Read the `usdcE` balance held by a `directDepositV1` contract.
2. Pull `usdc` from `msg.sender` into `directDepositV1` to replace the `usdcE`.
3. Withdraw `usdcE` from `directDepositV1` to `ContractOwner`.
4. Forward `usdcE` to `msg.sender`.

Step 2 is the critical transfer:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
``` [1](#0-0) 

The return value of `transferFrom` is silently discarded. If the USDC token at the hardcoded address returns `false` on failure (e.g., insufficient allowance) instead of reverting, the subsequent lines still execute:

```solidity
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
```

The caller receives `usdcE` without having paid any USDC.

By contrast, every other token transfer in the codebase uses `ERC20Helper.safeTransferFrom`, which performs a low-level call and explicitly checks both the call success flag and the decoded boolean return:

```solidity
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [2](#0-1) 

`ContractOwner` imports `ERC20Helper` and declares `using ERC20Helper for IERC20Base` at line 24, so `safeTransferFrom` is directly available on the `IERC20Base` type at the call site. [3](#0-2) 

The same safe pattern is consistently applied in `EndpointStorage.chargeSlowModeFee` and `EndpointStorage.safeTransferFrom`: [4](#0-3) 

And in `BaseWithdrawPool.safeTransferFrom`: [5](#0-4) 

---

### Impact Explanation

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` (Ink chain, chainid 57073) returns `false` on a failed `transferFrom` rather than reverting, any caller can:

- Call `replaceUsdcEWithUsdc(subaccount)` for any subaccount whose `directDepositV1` holds `usdcE`.
- Receive the full `usdcE` balance without transferring any USDC.
- Repeat across all subaccounts, draining all `usdcE` held in `directDepositV1` contracts.

The corrupted asset delta is: `usdcE` balance of `directDepositV1[subaccount]` → zero, with no corresponding USDC credit. [6](#0-5) 

---

### Likelihood Explanation

- The function is `external` with no `onlyOwner` or `onlyDeployer` modifier — any EOA or contract on chain 57073 can call it.
- The only precondition is that a `directDepositV1` address exists for the target subaccount and holds a nonzero `usdcE` balance.
- Exploitability is conditional on the specific USDC token implementation: if it reverts on failure (as Circle's canonical USDC does), the bug is latent but not immediately exploitable. If it returns `false` (as some bridged or non-standard ERC20s do), it is directly exploitable.
- The inconsistency with every other transfer site in the codebase strongly suggests this is an unintentional omission. [7](#0-6) 

---

### Recommendation

Replace the raw `transferFrom` call with the safe wrapper already available on the type:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

This matches the pattern used at every other transfer site in the protocol. [8](#0-7) 

---

### Proof of Concept

1. Identify a `subaccount` whose `directDepositV1` holds a nonzero `usdcE` balance.
2. Call `replaceUsdcEWithUsdc(subaccount)` from an EOA with **zero USDC allowance** granted to `ContractOwner`.
3. If USDC's `transferFrom` returns `false` instead of reverting:
   - The `transferFrom` at line 616 silently fails.
   - `DirectDepositV1(directDepositV1).withdraw(usdcE)` executes, moving `usdcE` to `ContractOwner`.
   - `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` executes, sending `usdcE` to the caller.
4. Caller receives `usdcE` at zero cost; `directDepositV1` is drained. [9](#0-8)

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
