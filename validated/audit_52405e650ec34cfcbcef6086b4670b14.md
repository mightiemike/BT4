### Title
Drastically Underpriced Cost Constants for `g1_negate` and `g2_negate` Enable CPU Exhaustion via Undercharged Execution - (File: `src/bls_ops.rs`)

---

### Summary

`BLS_G1_NEGATE_BASE_COST` and `BLS_G2_NEGATE_BASE_COST` in `src/bls_ops.rs` are set to `916` and `1,204` respectively, but the repository's own benchmark data shows the actual CPU cost of these operations is approximately `850,000` and `2,376,000` cost units. The constants are off by a factor of ~928x and ~1,974x — a direct analog to the "missing scaling factor" class of bug in the report. An attacker can craft CLVM programs that call these opcodes within a legitimate cost budget while consuming orders of magnitude more CPU time than the budget implies.

---

### Finding Description

In `src/bls_ops.rs`, the cost constants for the two negate operators are defined as:

```rust
// this is the same cost as XORing the top bit (minus the heap allocation of the
// return value, which the operator is adding back)
const BLS_G1_NEGATE_BASE_COST: Cost = 1396 - 480;   // = 916
const BLS_G2_NEGATE_BASE_COST: Cost = 2164 - 960;   // = 1204
```

The comment reveals the developer's intent: model the cost as a simple bit-flip (XOR of the top bit of the compressed point). However, the actual implementation in `op_bls_g1_negate` and `op_bls_g2_negate` does significantly more work in strict mode (the default for Chia):

```rust
if strict {
    a.validate_g1(point, blob)?;   // full BLS12-381 point validation
}
blob[0] ^= 0x20;
if strict {
    a.add_validated_g1(blob);
}
```

The `validate_g1` / `validate_g2` calls perform full elliptic curve point validation on BLS12-381, which is the dominant CPU cost. The repository's own `benchmarks.txt` records the true measured cost:

| Opcode | Charged cost (constant) | Measured cost (benchmark) | Ratio |
|---|---|---|---|
| `g1_negate` (51) | 916 | ~850,233 | ~928x underpriced |
| `g2_negate` (55) | 1,204 | ~2,376,799 | ~1,974x underpriced |

Additionally, both functions declare `_max_cost: Cost` (prefixed with `_`), meaning no `check_cost` guard is called during execution — there is no early-exit safety net even if the cost were correct.

---

### Impact Explanation

An attacker submits a spend bundle whose CLVM puzzle calls `g1_negate` (opcode 51) or `g2_negate` (opcode 55) in a loop. With a standard Chia block cost limit of 11,000,000,000 (11 billion), the attacker can invoke `g1_negate` approximately `11,000,000,000 / 916 ≈ 12,000,000` times per program execution. Each call takes ~132,000 ns of real CPU time, yielding ~1,584 seconds (~26 minutes) of CPU consumption per node per block evaluation — while the cost model implies the program should complete in a fraction of a second. Every validating full node must execute this program, making this a network-wide denial-of-service via consensus-layer undercharged execution.

---

### Likelihood Explanation

The entry path requires only crafting valid CLVM bytecode that calls opcode 51 or 55 in a loop with a valid 48-byte or 96-byte G1/G2 point argument. This is trivially achievable by any attacker who can submit a spend bundle to the Chia mempool. No privileged access, social engineering, or dependency compromise is required. The `ChiaDialect` default enables strict mode, so `validate_g1`/`validate_g2` is called on every invocation.

---

### Recommendation

Replace the cost constants with values calibrated to the actual measured CPU cost. Based on `benchmarks.txt` (using the cost scale of `6.425771579565472` nanoseconds-per-cost-unit):

- `BLS_G1_NEGATE_BASE_COST` should be approximately `850,000` (matching the ~132,316 ns measurement × cost scale).
- `BLS_G2_NEGATE_BASE_COST` should be approximately `2,376,000` (matching the ~369,885 ns measurement × cost scale).

The comment claiming "same cost as XORing the top bit" must be removed or corrected to reflect that point validation dominates the cost in strict mode.

```rust
pub fn op_bls_g1_negate(
    a: &mut Allocator,
    input: NodePtr,
    _max_cost: Cost,
    flags: ClvmFlags,
) -> Response {
    let strict = !flags.contains(ClvmFlags::RELAXED_BLS);
    let [point] = get_args::<1>(a, input, "g1_negate")?;

    let mut blob: [u8; 48] = atom(a, point, "G1 atom").and_then(|blob| {
        blob.as_ref().try_into().map_err(|_| {
            EvalErr::InvalidOpArg(point, "atom is not a G1 size, 48 bytes".to_string())
        })
    })?;
    if strict {
        a.validate_g1(point, blob)?;
    }

    if (blob[0] & 0xe0) == 0xc0 {
        // This is compressed infinity. negating it is a no-op
        // we can just pass through the same atom as we received. We'll charge
        // the allocation cost anyway, for consistency
        Ok(Reduction(
            BLS_G1_NEGATE_BASE_COST + 48 * MALLOC_COST_PER_BYTE,
            point,
        ))
    } else {
        blob[0] ^= 0x20;
        if strict {
            a.add_validated_g1(blob);
        }
        new_atom_and_cost(a, BLS_G1_NEGATE_BASE_COST, &blob)
    }
}
```
