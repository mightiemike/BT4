### Title
Approve Without Prior Zero-Reset in `DirectDepositV1.creditDeposit()` Permanently Bricks USDT-Like Token Deposits — (`core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without first resetting the allowance to zero. For tokens like USDT that revert when `approve()` is called with a non-zero existing allowance, a residual allowance left by a prior sub-minimum deposit attempt permanently prevents any future `creditDeposit()` call from succeeding, locking deposited funds in the DDA contract.

---

### Finding Description

`DirectDepositV1.creditDeposit()` iterates over all spot product IDs and, for each product with a non-zero balance, calls:

```solidity
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
``` [1](#0-0) 

The critical issue is that `Endpoint.depositCollateralWithReferral()` contains an **early return path** that skips the actual token transfer when the deposit amount is below the minimum:

```solidity
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked when there are
    // multiple assets awaiting credit but one of them is below the minimum deposit amount.
    // we can just skip the deposit and continue with the next asset.
    return;
}
``` [2](#0-1) 

When this early return fires, `handleDepositTransfer` is never called, so the allowance set by `token.approve(...)` is **not consumed** and remains non-zero. On the next call to `creditDeposit()`, the bare `token.approve(address(endpoint), newBalance)` call reverts for USDT-like tokens because their `approve()` implementation requires the current allowance to be zero before setting a new value.

`creditDeposit()` carries no access control modifier — it is callable by any address. [3](#0-2) 

---

### Impact Explanation

Once a non-zero residual allowance is established for a USDT-like token, every subsequent call to `creditDeposit()` reverts at the `token.approve(...)` line for that token. Because the function iterates over all product IDs in a single loop without isolation, the revert propagates and blocks deposits for **all** products in that call. Any USDT (or equivalent) tokens already sitting in the DDA contract cannot be forwarded to the endpoint and are effectively stuck until the allowance is manually cleared — which is not possible through any existing function in `DirectDepositV1`.

---

### Likelihood Explanation

The trigger is straightforward and requires no special privilege:

1. A sub-minimum amount of USDT is sent to the DDA (accidentally or deliberately by a griever).
2. Anyone calls `creditDeposit()`. The endpoint's early-return path fires, leaving a non-zero allowance.
3. Any subsequent `creditDeposit()` call with a valid deposit amount reverts permanently.

`creditDeposit()` is a public, permissionless function. The sub-minimum deposit condition is explicitly anticipated by the Endpoint codebase itself (the comment in the early-return block confirms this is a known operational scenario). This makes the likelihood realistic. [2](#0-1) 

---

### Recommendation

Reset the allowance to zero before each `approve` call inside `creditDeposit()`:

```solidity
if (balance != 0) {
    token.approve(address(endpoint), 0);       // reset first
    token.approve(address(endpoint), balance); // then set
    endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
}
```

This mirrors the fix applied in the referenced Connext commit and the pattern already correctly used in `ContractOwner.wrapVaultAsset()`:

```solidity
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
``` [4](#0-3) 

---

### Proof of Concept

1. Deploy the system with USDT as a supported spot collateral token.
2. Send `1 wei` of USDT to the DDA contract (below `MIN_DEPOSIT_AMOUNT`).
3. Call `DirectDepositV1.creditDeposit()`. Inside the loop for the USDT product:
   - `balance = 1`, so `token.approve(address(endpoint), 1)` executes — allowance is now `1`.
   - `endpoint.depositCollateralWithReferral(...)` is called; `isValidDepositAmount` returns `false`; function returns early without transferring tokens.
   - Allowance remains `1`.
4. Send `10_000e6` USDT (a valid deposit amount) to the DDA.
5. Call `DirectDepositV1.creditDeposit()` again. For the USDT product:
   - `balance = 10_000e6 + 1`, so `token.approve(address(endpoint), 10_000e6 + 1)` is called.
   - USDT's `approve()` reverts because the current allowance (`1`) is non-zero.
   - The entire `creditDeposit()` call reverts.
6. The `10_000e6` USDT is permanently stuck in the DDA with no recovery path. [3](#0-2) [5](#0-4)

    function creditDeposit() external {
        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {

            address tokenAddr = spotEngine.getToken(productId);
