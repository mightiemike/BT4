### Title
Valid SPV Proofs for Single-Transaction Blocks Permanently Rejected — (`contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` (and by extension `verify_transaction_inclusion_v2`) contains a guard `require!(!args.merkle_proof.is_empty())` that unconditionally rejects any proof with an empty merkle proof list. For a Bitcoin block containing exactly one transaction (the coinbase), the valid merkle proof is the empty list — the merkle root *is* the transaction hash. This check permanently prevents any caller from verifying a legitimately included transaction in a single-transaction block, even though the contract's own `compute_root_from_merkle_proof` would correctly accept it.

---

### Finding Description

In `verify_transaction_inclusion`, after confirming the block is on the main chain and has sufficient confirmations, the contract executes:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

For a block with a single transaction, the Bitcoin protocol defines `merkle_root = txid`. The correct merkle proof for that transaction is the empty list. `compute_root_from_merkle_proof` handles this correctly — when `merkle_proof` is empty, the loop body never executes and `current_hash` (initialized to `transaction_hash`) is returned directly:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    // ...
    for proof_hash in merkle_proof { /* skipped when empty */ }
    current_hash  // returns transaction_hash unchanged
}
``` [2](#0-1) 

So `compute_root_from_merkle_proof(coinbase_txid, 0, &[]) == merkle_root` would evaluate to `true` — but the `require!` guard fires first and permanently panics, making the call unreachable.

`verify_transaction_inclusion_v2` is equally affected: it validates the coinbase proof independently, then delegates to `verify_transaction_inclusion` via `self.verify_transaction_inclusion(args.into())`, hitting the same guard: [3](#0-2) 

---

### Impact Explanation

Any downstream NEAR contract that gates fund release on a successful `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` call for a transaction in a single-transaction Bitcoin block will be permanently blocked. The block header is validly stored on-chain, the proof is cryptographically correct, but the contract will always panic before evaluating it. There is no alternative code path or workaround available to the caller — the state is permanently unresolvable through the contract's public API.

---

### Likelihood Explanation

Single-transaction blocks (containing only the coinbase) occur in Bitcoin, particularly during periods of low mempool activity. They are a normal, protocol-valid block type. Any relayer that submits such a block header to the contract creates a permanently unverifiable block for SPV consumers. An adversarial relayer could deliberately submit only single-transaction blocks to selectively deny SPV verification for specific block heights.

---

### Recommendation

Remove the blanket `require!(!args.merkle_proof.is_empty())` guard. The subsequent `compute_root_from_merkle_proof` call already handles the empty-proof case correctly and will return the transaction hash itself, which is then compared against `header.block_header.merkle_root`. If a non-empty proof is desired as a policy constraint, the check should be conditioned on the block having more than one transaction — but since the contract does not store the transaction count, the simplest correct fix is to remove the guard entirely and rely on the merkle root comparison as the sole validity check.

---

### Proof of Concept

1. A Bitcoin block `B` containing exactly one transaction (coinbase, `txid = T`) is relayed and accepted by `submit_blocks`. The contract stores `B` with `merkle_root = T`.
2. A caller invokes `verify_transaction_inclusion` with `tx_id = T`, `tx_block_blockhash = hash(B)`, `tx_index = 0`, `merkle_proof = []`, `confirmations = 1`.
3. The block is found in `mainchain_header_to_height` ✓, confirmations are satisfied ✓.
4. Execution reaches line 315: `require!(!args.merkle_proof.is_empty())` → **panics** with `"Merkle proof is empty"`.
5. The call reverts. No state change occurs. The same call will always revert — the transaction can never be verified despite being genuinely included in the block.
6. Any downstream contract awaiting this verification result is permanently blocked. [4](#0-3)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
