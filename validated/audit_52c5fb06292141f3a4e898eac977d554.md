### Title
Single Reverting Withdrawal Blocks Entire Sequencer Batch in `submitTransactionsChecked` — (File: `core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.submitTransactionsChecked` processes a sequencer-submitted batch of transactions in a plain loop with no per-transaction error isolation. A single `WithdrawCollateralV2` transaction whose user-controlled `sendTo` address is blacklisted by USDC/USDT causes `BaseWithdrawPool.handleWithdrawTransfer` to revert, which propagates through `_delegatecallEndpointTx` and reverts the entire batch. Every other transaction in that batch — trades, liquidations, other users' withdrawals — is also rolled back.

---

### Finding Description

**Root cause chain:**

1. `submitTransactionsChecked` iterates over all transactions and calls `processTransaction` for each with no try/catch: [1](#0-0) 

2. `processTransaction` delegates to `EndpointTx` via `_delegatecallEndpointTx`, which explicitly re-reverts on any inner failure: [2](#0-1) 

3. `processTransactionImpl` handles `WithdrawCollateralV2` and passes the user-supplied `sendTo` directly to `clearinghouse.withdrawCollateral`: [3](#0-2) 

4. `Clearinghouse.handleWithdrawTransfer` first moves tokens to the `withdrawPool`, then calls `submitWithdrawal`: [4](#0-3) 

5. `BaseWithdrawPool.submitWithdrawal` calls `handleWithdrawTransfer`, which executes `token.safeTransfer(to, amount)` — the final transfer to the user-controlled `sendTo`: [5](#0-4) 

If `sendTo` is on the USDC/USDT compliance blacklist, `safeTransfer` reverts. Because `_delegatecallEndpointTx` re-reverts unconditionally and `submitTransactionsChecked` has no isolation, the entire batch is rolled back.

**Contrast with slow-mode path:** `_executeSlowModeTransaction` wraps `processSlowModeTransaction` in a `try/catch`, so slow-mode failures are silently skipped. The sequencer batch path has no equivalent protection. [6](#0-5) 

---

### Impact Explanation

Every transaction co-batched with the failing withdrawal is reverted: order matches, liquidations, price updates, and other users' withdrawals. The sequencer must identify the offending transaction, reconstruct the batch without it, re-sign, and resubmit. A malicious actor who repeatedly submits `WithdrawCollateralV2` transactions targeting known blacklisted addresses can force continuous batch reconstruction, causing sustained settlement delays for all protocol users.

---

### Likelihood Explanation

USDC and USDT maintain public or semi-public blacklists. An attacker with a funded subaccount can craft a `WithdrawCollateralV2` pointing `sendTo` at any known blacklisted address. The attacker does not need to be blacklisted themselves; they only need to know one blacklisted address. The sequencer has no on-chain mechanism to pre-screen `sendTo` addresses before batch inclusion, so the failure only surfaces at execution time.

---

### Recommendation

- Wrap each `processTransaction` call inside `submitTransactionsChecked` in a `try/catch` (mirroring the slow-mode path), recording failed indices for sequencer-side handling.
- Alternatively, adopt a pull-based withdrawal model: record the withdrawal claim on-chain and let users call a separate `claimWithdrawal` function, removing the transfer from the sequencer batch entirely.
- As a short-term measure, the sequencer should simulate each batch transaction off-chain before submission and exclude any that would revert.

---

### Proof of Concept

1. Attacker identifies a known USDC-blacklisted address `0xBAD...`.
2. Attacker holds a funded subaccount and signs a `WithdrawCollateralV2` with `sendTo = 0xBAD...` and a valid nonce.
3. Sequencer includes this transaction in a batch alongside legitimate trades and liquidations.
4. `submitTransactionsChecked` processes the batch; when it reaches the attacker's withdrawal, execution reaches `BaseWithdrawPool.handleWithdrawTransfer` → `token.safeTransfer(0xBAD..., amount)`.
5. USDC reverts the transfer (blacklist check).
6. `_delegatecallEndpointTx` re-reverts with the USDC error.
7. The entire `submitTransactionsChecked` call reverts; `nSubmissions` is not incremented for any transaction in the batch.
8. All co-batched transactions (other users' trades, liquidations) are rolled back and must be resubmitted.
9. Attacker repeats with a new nonce to continuously disrupt batch processing.

            assembly {
                revert(add(result, 0x20), mload(result))
            }
