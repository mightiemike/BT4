### Title
Stale Mainchain State Used in Fork-Block Difficulty Calculation Enables Incorrect PoW Acceptance - (File: `contract/src/dogecoin.rs`)

---

### Summary

The `get_next_work_required` function in `dogecoin.rs` (and analogously in `bitcoin.rs` and `litecoin.rs`) resolves the reference block for difficulty retargeting via `blocks_getter.get_header_by_height(height_first)`. That helper reads exclusively from `mainchain_height_to_header`, so when a fork block is being validated, the function silently uses the **mainchain** ancestor's timestamp instead of the **fork's** actual ancestor at the same height. The difficulty target computed from this stale value is wrong, and the bits check `expected_bits == block_header.bits` is evaluated against it, allowing fork blocks with incorrect difficulty to be accepted.

A developer-written `TODO` comment in the code explicitly flags this concern:

```rust
// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
```

---

### Finding Description

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // ← mainchain-only map
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

Fork blocks are stored only in `headers_pool` via `store_fork_header`; they are never inserted into `mainchain_height_to_header`. [2](#0-1) 

In the Dogecoin path (post-block 145 000, where `difficulty_adjustment_interval = 1`), every block triggers a retarget. The reference block height is:

```rust
let height_first = prev_block_header
    .block_height
    .checked_sub(blocks_to_go_back)   // blocks_to_go_back = 1
    ...;

let first_block_time = blocks_getter
    .get_header_by_height(height_first)   // ← reads mainchain, not fork ancestor
    .block_header
    .time;
``` [3](#0-2) 

For a fork that diverges at height H, the fork block at H+2 has `height_first = H`. `get_header_by_height(H)` returns the **mainchain** block at H, whose timestamp may differ arbitrarily from the fork's block at H. The resulting `first_block_time` is stale relative to the fork's actual chain history.

The same structural defect exists in the Bitcoin and Litecoin retarget paths: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

`calculate_next_work_required` computes:

```rust
let mut actual_time_taken: i64 = prev_block_time - first_block_time;
``` [6](#0-5) 

If `first_block_time` (mainchain block at H) is earlier than the fork's actual ancestor at H, `actual_time_taken` is inflated, producing a lower (easier) `expected_bits`. The check:

```rust
require!(
    expected_bits == block_header.bits,
    "Error: Incorrect target. ..."
);
``` [7](#0-6) 

passes when the attacker sets `block_header.bits` to match the incorrectly computed `expected_bits`. The subsequent PoW hash check then uses this easier target, so the attacker mines the block against a lower difficulty than the protocol requires. Accepted fork blocks accumulate chain work and can trigger `reorg_chain`, corrupting the canonical chain tip. [8](#0-7) 

---

### Likelihood Explanation

- Dogecoin retargets every block post-145 000, so the stale-state path is exercised for **every fork block after the first** in any multi-block fork submission.
- A relayer (the explicit production entry point for `submit_blocks`) can craft fork headers with chosen timestamps within the MTP lower bound and the `MAX_FUTURE_BLOCK_TIME_LOCAL` upper bound, giving meaningful control over the timestamp delta.
- The `trusted_relayer` gate is the only barrier; the audit scope explicitly includes "relayer-path user supplying adversarial chain data." [9](#0-8) 

---

### Recommendation

Replace the mainchain-only `get_header_by_height` lookup with an ancestor walk that follows `prev_block_hash` links through `headers_pool` from the known fork ancestor. This mirrors the reference Dogecoin implementation's intent and is what the existing TODO comment already calls for. For Bitcoin and Litecoin the same fix applies at their respective difficulty-boundary lookups.

---

### Proof of Concept

1. Contract is initialized with a Dogecoin mainchain at height H (post-145 000). Mainchain block at H has `time = T_main`.
2. Relayer submits fork block F_H (diverges at H) with `time = T_main + Δ` where `Δ` is large (e.g., 600 s, within MTP/future-time limits).
3. Relayer submits fork block F_{H+1} (prev = F_H). Difficulty check for F_{H+1} uses `height_first = H-1`; mainchain and fork agree at H-1, so `first_block_time` is correct here.
4. Relayer submits fork block F_{H+2} (prev = F_{H+1}). `height_first = H`. `get_header_by_height(H)` returns the **mainchain** block at H with `time = T_main`, not F_H with `time = T_main + Δ`.
5. `actual_time_taken = F_{H+1}.time - T_main` is `Δ` seconds larger than the correct value `F_{H+1}.time - (T_main + Δ)`.
6. The inflated timespan produces a lower `expected_bits` (easier target). Attacker sets `F_{H+2}.bits` to this easier value, mines a valid PoW hash against it, and the block is accepted.
7. Repeating steps 4–6 for subsequent fork blocks allows the attacker to build a fork with systematically under-difficulty blocks, potentially accumulating enough chain work to trigger `reorg_chain` and replace the canonical tip. [10](#0-9) [11](#0-10)

```rust
fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
```
