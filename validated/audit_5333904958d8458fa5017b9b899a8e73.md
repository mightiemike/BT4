Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Originating User, Allowing Any User to Bypass Swap Allowlist ŌĆö (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to its own `msg.sender` ŌĆö the direct caller of `pool.swap`. When `MetricOmmSimpleRouter` intermediates, `sender` becomes the router's address, not the end user. Any pool admin who allowlists the router to support standard periphery UX inadvertently grants every unprivileged user the ability to bypass the per-user swap allowlist.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs its check as:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (used as the namespace key) and `sender` is the first argument the pool passes into the hook. `MetricOmmPool.swap` passes its own `msg.sender` as that argument:

```solidity
_beforeSwap(
    msg.sender,   // ŌåÉ becomes `sender` in the extension
    recipient,
    ...
```

When `MetricOmmSimpleRouter.exactInput` (or `exactInputSingle`, `exactOutput`, `exactOutputSingle`) calls `pool.swap(...)`, the pool's `msg.sender` is the router contract. The extension therefore checks `allowedSwapper[pool][router_address]`, not `allowedSwapper[pool][actual_user]`. The router provides no mechanism to forward the originating user's address ŌĆö it calls `IMetricOmmPoolActions(pool).swap(...)` directly. If the pool admin allowlists the router (required for any allowlisted user to use the standard periphery), the check passes for every caller of the router, regardless of whether they are individually allowlisted.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd users, whitelisted market makers) can be fully bypassed by any unprivileged user calling `MetricOmmSimpleRouter`. The bypassing user executes swaps against the pool's liquidity, defeating the access control the pool admin explicitly configured. This constitutes broken core pool functionality and direct loss-of-policy-enforcement on LP assets.

## Likelihood Explanation
Any pool that (1) deploys `SwapAllowlistExtension` to restrict swaps and (2) allowlists the router so that legitimate users can use the standard periphery is immediately and completely vulnerable. The router is a public, permissionless contract; no special privilege is required to call it. The attacker only needs to know the pool address.

## Recommendation
The pool should propagate the originating user to the extension rather than the immediate `msg.sender`. The cleanest fix is for `MetricOmmPool.swap` to accept an explicit `sender` parameter verified against `msg.sender` or a trusted-forwarder registry, so the extension always sees the economically relevant actor. Alternatively, `SwapAllowlistExtension` can decode the real user from `extensionData` when the direct caller is a known router, but this requires a trusted router registry and coordinated encoding by the router.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Pool admin allowlists `alice`: `setAllowedToSwap(pool, alice, true)`.
3. Pool admin allowlists the router so `alice` can use the standard periphery: `setAllowedToSwap(pool, router, true)`.
4. `bob` (not allowlisted) calls `MetricOmmSimpleRouter.exactInput(...)` targeting the pool.
5. The pool calls `_beforeSwap(msg.sender=router, ...)`. The extension checks `allowedSwapper[pool][router]` ŌåÆ `true`. The swap proceeds.
6. `bob` successfully swaps on a pool he was explicitly barred from, bypassing the curated allowlist entirely.
