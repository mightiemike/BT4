Audit Report

## Title
SwapAllowlistExtension Checks Immediate Caller (`sender`) Instead of End User, Enabling Full Allowlist Bypass via Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which resolves to `msg.sender` of `pool.swap()` — the immediate caller, not the end user. When `MetricOmmSimpleRouter` is allowlisted (a natural configuration for giving allowlisted users access to multi-hop or exact-output routing), any unpermissioned user can bypass the swap allowlist entirely by routing through the router. The sibling `DepositAllowlistExtension` correctly ignores `sender` and gates on `owner`, making the asymmetry non-obvious to pool admins.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the first argument (`sender`) to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // sender = whoever called pool.swap()
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that value:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller), so the effective check is `allowedSwapper[pool][sender]` where `sender` = whoever called `pool.swap()`.

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The pool receives `msg.sender = router`. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

**Exploit path:**
1. Pool admin deploys pool with `SwapAllowlistExtension`.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — alice is KYC'd.
3. Admin calls `setAllowedToSwap(pool, router, true)` — to let alice use multi-hop routing.
4. Bob (not allowlisted) calls `router.exactInputSingle({pool: pool, recipient: bob, ...})`.
5. Router calls `pool.swap(bob, ...)` with `msg.sender = router`.
6. Extension checks `allowedSwapper[pool][router]` → `true` → swap succeeds.
7. Bob receives output tokens despite never being allowlisted.

The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput` — all router entry points call `pool.swap()` directly with `msg.sender = router`.

`DepositAllowlistExtension.beforeAddLiquidity` correctly ignores `sender` and gates on `owner`:

```solidity
// DepositAllowlistExtension.sol L32-38
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    ...
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
```

The asymmetry between the two extensions makes the flaw non-obvious to pool admins.

## Impact Explanation

A pool whose admin allowlists the router loses all swap-access control. Any address — including those explicitly excluded from the allowlist — can execute swaps by routing through `MetricOmmSimpleRouter`. This breaks the core invariant of the extension: only approved addresses may trade on the pool. This is a direct allowlist path bypass through the router, explicitly identified as a required audit pivot. Depending on the pool's purpose (institutional-only, regulatory-compliant, pre-launch restricted), unauthorized swaps cause unauthorized price impact and extraction of LP value by disallowed parties.

## Likelihood Explanation

The trigger requires the router to be allowlisted. This is a natural and expected configuration: a pool admin who wants allowlisted users to access multi-hop or exact-output routing must allowlist the router. The admin is unlikely to realize this opens the gate to all users, because the sibling `DepositAllowlistExtension` correctly gates on `owner` and does not exhibit this problem. The asymmetry makes the flaw non-obvious. Once the router is allowlisted, the bypass is trivially repeatable by any address with no special privileges.

## Recommendation

`SwapAllowlistExtension.beforeSwap` should not check `sender` (the immediate caller of `pool.swap()`). The simplest safe fix is to gate on `recipient` — the address that actually receives the output tokens — mirroring how `DepositAllowlistExtension` gates on `owner`:

```solidity
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

Alternatively, document explicitly that the router must never be allowlisted and that router-based swaps are incompatible with per-user swap gating.

## Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension configured.
2. Admin: swapExtension.setAllowedToSwap(pool, alice, true)   // alice is KYC'd
3. Admin: swapExtension.setAllowedToSwap(pool, router, true)  // enable router for alice
4. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, recipient: bob, ...})
5. Router calls pool.swap(bob, ...) with msg.sender = router.
6. Extension checks allowedSwapper[pool][router] → true → swap succeeds.
7. Bob receives output tokens despite never being allowlisted.
