### Title
Unbounded `NlpLockedBalanceQueue` Growth Causes Permanent OOG Lock on `burnNlp` - (File: `core/contracts/SpotEngine.sol`)

---

### Summary

`SpotEngine.handleNlpLockedBalance` appends a new entry to a per-subaccount queue every time a user mints NLP at a distinct oracle timestamp, with no cap on queue depth. The `tryUnlockNlpBalance` helper — called on every NLP balance mutation and on `getNlpUnlockedBalance` — must iterate over the entire queue. A user who mints NLP across many distinct timestamps will grow this queue without bound. Once the queue is large enough, any subsequent `burnNlp` call will exceed the block gas limit, permanently locking the user's NLP tokens with no emergency-exit path.

---

### Finding Description

`SpotEngine.handleNlpLockedBalance` manages a per-subaccount FIFO queue of locked NLP mints:

```solidity
// core/contracts/SpotEngine.sol  lines 148-173
if (
    queue.balanceCount > 0 &&
    queue.balances[queue.balanceCount - 1].unlockedAt ==
    getOracleTime() + NLP_LOCK_PERIOD
) {
    queue.balances[queue.balanceCount - 1].balance.amount += amountDelta;
} else {
    queue.balances[queue.balanceCount] = NlpLockedBalance({
        balance: Balance({amount: amountDelta}),
        unlockedAt: getOracleTime() + NLP_LOCK_PERIOD
    });
    queue.balanceCount++;   // ← grows without bound
}
``` [1](#0-0) 

Entries are merged only when two consecutive mints share the exact same `getOracleTime()`. Any mint at a different oracle timestamp appends a new entry. There is no maximum on `queue.balanceCount`.

`tryUnlockNlpBalance` is called unconditionally at the top of `handleNlpLockedBalance` (and inside `getNlpUnlockedBalance`):

```solidity
// core/contracts/SpotEngine.sol  lines 133-137
function getNlpUnlockedBalance(bytes32 subaccount)
    external
    returns (Balance memory)
{
    tryUnlockNlpBalance(subaccount);
    ...
}
``` [2](#0-1) 

`tryUnlockNlpBalance` (defined in `SpotEngineState`, not fully indexed) must scan the queue from `startIndex` to `balanceCount` to move matured entries into `unlockedBalanceSum`. As `balanceCount` grows, this scan grows proportionally.

`Clearinghouse.burnNlp` calls `getNlpUnlockedBalance` before any burn can proceed:

```solidity
// core/contracts/Clearinghouse.sol  lines 498-501
require(
    spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
    ERR_UNLOCKED_NLP_INSUFFICIENT
);
``` [3](#0-2) 

`burnNlp` is only reachable via `processTransactionImpl` (sequencer path):

```solidity
// core/contracts/EndpointTx.sol  lines 554-573
} else if (txType == IEndpoint.TransactionType.BurnNlp) {
    ...
    clearinghouse.burnNlp(...);
}
``` [4](#0-3) 

There is no slow-mode or emergency path for `BurnNlp` in `processSlowModeTransactionImpl`. [5](#0-4) 

---

### Impact Explanation

Once `queue.balanceCount` is large enough that `tryUnlockNlpBalance`'s scan exceeds the block gas limit, every `burnNlp` call for that subaccount reverts with OOG. Because there is no slow-mode `BurnNlp` handler and no emergency-exit mechanism for NLP, the user's NLP tokens are permanently locked in the protocol. This is a direct, irreversible loss of user funds — not merely a performance degradation.

---

### Likelihood Explanation

NLP is a long-term liquidity product. A user who mints NLP once per oracle tick (e.g., once per day) accumulates one queue entry per day. After one year that is ~365 entries; after two years ~730. The gas cost of scanning 500–1000 entries plus the surrounding `burnNlp` logic (health checks, balance updates, `_applyNlpRebalance` loop over `nlpPools`) can realistically exceed the block gas limit on Ink Chain. No special privilege is required — any NLP holder who mints regularly over a long period reaches this state organically.

---

### Recommendation

1. **Cap `balanceCount`**: Enforce a hard maximum (e.g., 50–100 entries) on `nlpLockedBalanceQueues[subaccount].balanceCount`. Reject or merge mints that would exceed the cap.
2. **Lazy / paginated unlock**: Allow `tryUnlockNlpBalance` to process only a fixed number of entries per call, storing a `startIndex` cursor so subsequent calls continue where the last left off.
3. **Accumulator pattern**: Replace the per-entry queue with a cumulative accounting scheme (analogous to the sushi-masterchef suggestion in the external report) that tracks total locked and unlocked amounts without per-mint entries.

---

### Proof of Concept

1. User calls `MintNlp` (signed, via sequencer) at oracle time `T₀` → `queue.balanceCount = 1`.
2. User calls `MintNlp` at oracle time `T₁ ≠ T₀` → `queue.balanceCount = 2`.
3. Repeat N times at distinct oracle timestamps → `queue.balanceCount = N`.
4. User calls `BurnNlp`. Sequencer processes it → `clearinghouse.burnNlp` → `spotEngine.getNlpUnlockedBalance` → `tryUnlockNlpBalance` iterates over N entries.
5. For sufficiently large N (empirically determinable per chain gas limit), the transaction reverts OOG.
6. No alternative path exists to recover the locked NLP balance.

The exact OOG threshold depends on the gas cost of each queue-entry scan in `tryUnlockNlpBalance` (not fully indexed), but the structural unboundedness of `queue.balanceCount` at [6](#0-5)  is the concrete root cause.
