### Title
Unbounded Repeated Margin Transfer in `createIsolatedSubaccount` — Missing Parent Health Check Allows Unlimited Collateral Drain — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`createIsolatedSubaccount` deduplicates on `digest` (one per unique order), but the margin transfer block executes unconditionally for every new digest, even when the isolated subaccount for that `productId` already exists. Because there is no post-transfer health check on the parent, N distinct isolated orders for the same `productId` each drain `margin` from the parent into the same isolated subaccount, pushing the parent's quote balance arbitrarily negative.

---

### Finding Description

The replay guard in `createIsolatedSubaccount` is:

```solidity
if (digestToSubaccount[digest] != bytes32(0)) {
    return digestToSubaccount[digest];   // early-exit only for identical digest
}
``` [1](#0-0) 

After that guard, the function loops over `isolatedSubaccountsMask` to find an existing isolated subaccount for the same `productId`. When one is found it sets `newIsolatedSubaccount` to the existing subaccount and breaks — but does **not** skip the margin transfer:

```solidity
if (productId == txn.productId) {
    newIsolatedSubaccount = subaccount;
    break;
}
``` [2](#0-1) 

The margin transfer block is **outside** the `if (newIsolatedSubaccount == bytes32(0))` creation branch and runs unconditionally for every new digest:

```solidity
int128 margin = int128(_isolatedMargin(txn.order.appendix));
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
}
``` [3](#0-2) 

`spotEngine.updateBalance` applies the delta with no floor check — the parent's balance can go negative (into protocol debt): [4](#0-3) 

Contrast this with `transferQuote` in `Clearinghouse.sol`, which enforces a health check after every quote movement:

```solidity
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -toTransfer);
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.recipient, toTransfer);
require(_isAboveInitial(txn.sender), ERR_SUBACCT_HEALTH);
``` [5](#0-4) 

No equivalent `_isAboveInitial` guard exists anywhere in `createIsolatedSubaccount`.

Additionally, the `CreateIsolatedSubaccount` dispatch in `EndpointTx.sol` does **not** call `validateSignedTx` (no endpoint-level nonce check), so the only deduplication is the per-digest map: [6](#0-5) 

---

### Impact Explanation

An attacker who controls a parent subaccount can:

1. Pre-sign N isolated orders for the same `productId`, each with a distinct `(nonce, price, amount)` tuple → N distinct EIP-712 digests.
2. Submit all N as `CreateIsolatedSubaccount` transactions to the sequencer (a normal user-initiated flow).
3. Call 1 creates the isolated subaccount and transfers `margin` from parent → isolated.
4. Calls 2…N each find the existing isolated subaccount, skip creation, but still execute the margin transfer: parent loses `margin` again each time.
5. After N calls: parent balance = `initial − N×margin` (can be deeply negative); isolated subaccount balance = `N×margin`.

The isolated subaccount now holds far more collateral than the parent ever legitimately allocated. The parent is left insolvent (negative health), immediately liquidatable. The excess margin in the isolated subaccount can be used to open outsized positions or extracted via `TransferQuote` back to the parent after the parent's debt is socialized.

---

### Likelihood Explanation

The path is fully user-initiated: a user signs multiple isolated orders (different nonce/price/amount) for the same product and submits them. The sequencer processes each as a `CreateIsolatedSubaccount` transaction. No privileged access, no sequencer compromise, and no impossible token behavior is required. The only prerequisite is that the parent subaccount has a non-zero initial balance (to sign the first order credibly); subsequent transfers drive it negative without any on-chain gate.

---

### Recommendation

Add a parent health check immediately after the margin transfer block, mirroring the pattern used in `transferQuote`:

```solidity
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
    // ADD: enforce parent remains solvent
    require(_isAboveInitial(txn.order.sender), ERR_SUBACCT_HEALTH);
}
```

Additionally, consider enforcing that only one `digestToSubaccount` entry (and therefore one margin reservation) is permitted per `(sender, productId)` pair, so that the total margin allocated to an isolated subaccount is bounded by a single signed order.

---

### Proof of Concept

```solidity
// 5 isolated orders, same productId, different (nonce, price, amount) → 5 distinct digests
for (uint i = 0; i < 5; i++) {
    IEndpoint.Order memory order = IEndpoint.Order({
        sender:   parentSubaccount,
        priceX18: 1000e18 + int128(int(i)),   // distinct price → distinct digest
        amount:   1e18,
        expiration: ...,
        nonce:    uint64(i),
        appendix: isolatedAppendixWithMargin(1000e6)  // margin = 1000 USDC
    });
    endpoint.submitSlowModeTransaction(
        abi.encodePacked(
            uint8(IEndpoint.TransactionType.CreateIsolatedSubaccount),
            abi.encode(IEndpoint.CreateIsolatedSubaccount({
                order: order,
                productId: TARGET_PRODUCT_ID,
                signature: sign(order)
            }))
        )
    );
}
// Assert: spotEngine.getBalance(QUOTE_PRODUCT_ID, parentSubaccount).amount
//         == initialBalance - 5000e6
// Assert: spotEngine.getBalance(QUOTE_PRODUCT_ID, isolatedSubaccount).amount
//         == 5000e6
```

Each of the 5 calls passes the `digestToSubaccount` guard (new digest each time), finds the existing isolated subaccount on calls 2–5, and unconditionally transfers 1000 USDC from parent to isolated, draining the parent by 5000 USDC total.
