### Title
`PendingTransactionQueue` Does Not Track `WithdrawFromGasKey` Nested Inside `DelegateV2` Inner Actions, Breaking Gas-Key Balance Enforcement - (`chain/client/src/pending_transaction_queue.rs`)

### Summary

The `PendingTransactionQueue` (PTQ) is designed to enforce that gas key transactions are only admitted into chunks when the gas key has sufficient balance. It explicitly tracks `WithdrawFromGasKey` actions in top-level transaction actions. However, when a `WithdrawFromGasKey` is nested as an inner action inside a `DelegateV2` (meta-transaction) action, the PTQ's scanning logic does not recurse into the delegate's inner actions, leaving the withdrawal invisible to the PTQ. This breaks the stated invariant that "gas key balance only changes through transactions that PTQ explicitly tracks," allowing an attacker to cause the PTQ to admit gas key transactions that will fail at execution time.

### Finding Description

The `PendingTransactionQueue::add_chunk_transactions` method scans each transaction's top-level actions for `WithdrawFromGasKey` to track pending gas key balance reductions:

```rust
// Scan actions for WithdrawFromGasKey (affects gas key balance).
for action in tx.actions() {
    if let Action::WithdrawFromGasKey(withdraw) = action {
        let gas_key_entry = chunk_data
            .gas_key_costs
            .entry((signer_id.clone(), withdraw.public_key.clone()))
            .or_insert(Balance::ZERO);
        *gas_key_entry = gas_key_entry.saturating_add(withdraw.amount);
    }
}
```

Similarly, `PendingTxSession::check_pending` scans only top-level actions:

```rust
for action in tx.transaction.actions() {
    if let Action::WithdrawFromGasKey(withdraw) = action {
        ...
    }
}
```

Neither path recurses into the inner actions of an `Action::DelegateV2(...)`. A `WithdrawFromGasKey` is a valid `NonDelegateAction` (it is not excluded by the delegate action validation in `validate_action_with_mode`), so it can legally appear as an inner action of a `DelegateV2`. The `check_actor_permissions` function permits `WithdrawFromGasKey` when `actor_id == account_id`, which holds when the delegate sender is the gas key owner.

The design intent is explicit in the codebase — the comment in `imports.rs` states:

> "There are intentionally no promise batch actions for `WithdrawFromGasKey`. Actions that reduce gas key balance must only be initiated via transactions, not by contracts. Otherwise, they will not be visible to the pending transaction queue."

This guard covers contract host functions but does not cover `WithdrawFromGasKey` nested inside a `DelegateV2` inner action list, which is also invisible to the PTQ.

**Attack path:**

1. Alice holds a gas key with balance `B` (sufficient for `N` gas key transactions).
2. Alice constructs a `DelegateV2` whose inner actions include `WithdrawFromGasKey { public_key: gas_key_pub, amount: B }`, draining the entire balance.
3. A relayer wraps this in a transaction and submits it; it is included in block `N`.
4. `add_chunk_transactions` is called for block `N`. It iterates `tx.actions()`, sees `Action::DelegateV2(...)`, and does **not** recurse into the inner `WithdrawFromGasKey`. The PTQ's `pending_gas_key_costs` for Alice's gas key remains `0`.
5. Before block `N` is certified, Alice submits `N` gas key transactions.
6. `check_pending` is called for each: `paid_from_gas_key = 0`, so `available_gas_key_balance = B`. All `N` gas key txs are admitted.
7. Block `N` executes: the `WithdrawFromGasKey` reduces the gas key balance to `0`.
8. Block `N+1` executes: all `N` gas key transactions fail with `NotEnoughGasKeyBalance`, wasting block space.

### Impact Explanation

The PTQ's gas key balance enforcement invariant is broken. An unprivileged user can cause the PTQ to admit an unbounded number of gas key transactions that will deterministically fail at execution time. This enables a block-space DoS: an attacker can fill chunks with provably-invalid gas key transactions at no cost (since the gas key balance is drained before the gas key txs execute, and failed txs with zero balance cannot be charged). The invariant stated in the code — that all gas key balance reductions are visible to the PTQ — is violated.

### Likelihood Explanation

Any user who can create a gas key and access a relayer (or act as their own relayer) can trigger this. The `DelegateV2` action type is a standard protocol feature. No privileged role is required. The attack requires only constructing a valid `DelegateV2` with a `WithdrawFromGasKey` inner action, which passes all existing validation checks.

### Recommendation

In both `PendingTransactionQueue::add_chunk_transactions` and `PendingTxSession::check_pending`, recurse into the inner actions of `Action::DelegateV2` (and `Action::Delegate`) when scanning for `WithdrawFromGasKey`:

```rust
for action in tx.actions() {
    // Recurse into DelegateV2 inner actions
    let inner_actions: &[Action] = match action {
        Action::DelegateV2(d) => d.delegate_action.get_actions_ref(),
        _ => std::slice::from_ref(action),
    };
    for inner in inner_actions {
        if let Action::WithdrawFromGasKey(withdraw) = inner {
            // track withdrawal
        }
    }
}
```

Alternatively, add a validation rule in `validate_delegate_action` that explicitly rejects `WithdrawFromGasKey` as an inner action of any delegate action, mirroring the existing prohibition on contract host functions.
