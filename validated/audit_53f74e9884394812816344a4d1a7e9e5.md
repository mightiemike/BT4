### Title
Precommit Vote Signature Preimage Omits Chain ID, Enabling Cross-Chain/Cross-Fork Replay — (File: `crates/apollo_signature_manager/src/signature_manager.rs`)

---

### Summary

`build_precommit_vote_message_digest` constructs the ECDSA preimage as `PRECOMMIT_VOTE || block_hash` with no chain ID. Independently, `calculate_block_hash` (both the Rust implementation and the Cairo OS program) also omits chain ID from the block hash. The combination means a precommit vote signature produced on one Starknet chain is cryptographically valid on any other chain that arrives at the same block hash — which is structurally possible whenever two chains share a common ancestor and produce identical block content after the fork point.

---

### Finding Description

**Root cause — missing chain ID in the signing preimage:**

`build_precommit_vote_message_digest` at lines 138–145 of `crates/apollo_signature_manager/src/signature_manager.rs` constructs:

```
message = PRECOMMIT_VOTE || block_hash.to_bytes_be()
digest  = blake2s_to_felt(message)
``` [1](#0-0) 

No chain ID is present. The `SignatureManager` struct holds only a `keystore`; it has no `chain_id` field and the `SignatureManagerRequest::SignPrecommitVote(block_hash)` variant carries only the `BlockHash`. [2](#0-1) [3](#0-2) 

`verify_precommit_vote_signature` reconstructs the same chain-ID-free digest and accepts the signature: [4](#0-3) 

**Compounding factor — chain ID is also absent from the block hash:**

`calculate_block_hash` in `crates/starknet_api/src/block_hash/block_hash_calculator.rs` hashes: `BLOCK_HASH_VERSION`, `block_number`, `state_root`, `sequencer_address`, `block_timestamp`, commitment fields, gas prices, `starknet_version`, a reserved zero, and `previous_block_hash`. Chain ID is not a field. [5](#0-4) 

The Cairo OS program `block_hash.cairo` confirms the same structure: [6](#0-5) 

Because `previous_block_hash` is chained, two networks that share a common ancestor and then produce blocks with identical content (same transactions, same sequencer, same gas prices, same timestamps) will arrive at the same `BlockHash` value. The precommit vote signature over that hash is therefore identical and interchangeable between the two chains.

---

### Impact Explanation

An attacker who can observe precommit vote signatures broadcast on chain A (e.g., via the P2P gossip layer) and who can arrange for chain B to produce a block with the same hash can inject those signatures into chain B's consensus round. Because `verify_precommit_vote_signature` accepts any `(block_hash, signature, public_key)` triple without a chain-ID check, the replayed signatures pass verification. This allows the attacker to:

- Artificially inflate the apparent precommit quorum on chain B using votes that were never cast for chain B.
- Potentially push chain B past the 2/3 quorum threshold using validators who never participated in chain B's consensus, causing an unauthorized block commitment.

This matches the allowed impact scope: **High — transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload** (here: the wrong chain).

---

### Likelihood Explanation

The attack requires two conditions to hold simultaneously:

1. Two chains share a common ancestor (fork scenario, testnet mirroring mainnet, or two nodes misconfigured with different chain IDs but identical genesis).
2. After the fork point, both chains produce a block with identical content so that `calculate_block_hash` returns the same value.

Condition 2 is non-trivial but not impossible: immediately after a fork, if both chains process the same pending mempool transactions with the same sequencer address, timestamp, and gas prices, the block hashes will collide. The `previous_block_hash` field chains history, so the collision window is limited to the first diverging block. Likelihood is **Low-Medium**.

---

### Recommendation

Include the chain ID in the precommit vote signing preimage. The `SignatureManager` should be constructed with a `chain_id` field (analogous to how `TransactionConverter` holds `chain_id`), and `build_precommit_vote_message_digest` should incorporate it:

```rust
// Proposed fix
fn build_precommit_vote_message_digest(
    block_hash: BlockHash,
    chain_id: &ChainId,
) -> MessageDigest {
    let block_hash = block_hash.to_bytes_be();
    let chain_id_bytes = chain_id.as_hex(); // or canonical encoding
    let mut message = Vec::new();
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&block_hash);
    message.extend_from_slice(chain_id_bytes.as_bytes());
    MessageDigest(blake2s_to_felt(&message))
}
```

`SignatureManagerRequest::SignPrecommitVote` and `verify_precommit_vote_signature` must be updated to pass the chain ID through. The existing TODO comment in the file already flags a related concern about delimiter ambiguity: [7](#0-6) 

---

### Proof of Concept

1. Chain A (mainnet) and chain B (a fork of mainnet at block N) are both running Apollo sequencers.
2. Both chains have the same `previous_block_hash` at block N (identical history up to the fork).
3. Both chains include the same set of transactions in block N+1, with the same sequencer address, timestamp, and gas prices.
4. `calculate_block_hash` produces the same `BlockHash X` on both chains for block N+1.
5. Validator V on chain A signs a precommit vote: `sig = ecdsa_sign(key_V, blake2s_to_felt(PRECOMMIT_VOTE || X))`.
6. The attacker observes `sig` from chain A's P2P network.
7. The attacker injects a `Vote { vote_type: Precommit, height: N+1, round: R, proposal_commitment: Some(X), voter: V, signature: sig }` into chain B's consensus.
8. Chain B calls `verify_precommit_vote_signature(X, sig, pubkey_V)` → reconstructs `blake2s_to_felt(PRECOMMIT_VOTE || X)` → verifies successfully.
9. Chain B counts V's precommit toward quorum even though V never participated in chain B's consensus round. [8](#0-7) [1](#0-0) [4](#0-3)

func calculate_block_hash{poseidon_ptr: PoseidonBuiltin*}(
    block_info: BlockInfo*,
    header_commitments: BlockHeaderCommitments*,
    gas_prices_hash: felt,
    state_root: felt,
    previous_block_hash: felt,
    starknet_version: felt,
) -> felt {
    static_assert BlockInfo.SIZE == 3;
    static_assert BlockHeaderCommitments.SIZE == 5;

    let hash_state = hash_init();
    with hash_state {
        hash_update_single(BLOCK_HASH_VERSION);
        hash_update_single(block_info.block_number);
        hash_update_single(state_root);
        hash_update_single(block_info.sequencer_address);
        hash_update_single(block_info.block_timestamp);
        hash_update_single(header_commitments.packed_lengths);
        hash_update_single(header_commitments.state_diff_commitment);
        hash_update_single(header_commitments.transaction_commitment);
        hash_update_single(header_commitments.event_commitment);
        hash_update_single(header_commitments.receipt_commitment);
        hash_update_single(gas_prices_hash);
        hash_update_single(starknet_version);
        hash_update_single(0);
        hash_update_single(previous_block_hash);
    }

    let block_hash = hash_finalize(hash_state=hash_state);
    return block_hash;
}
