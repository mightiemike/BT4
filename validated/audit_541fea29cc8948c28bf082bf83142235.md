### Title
`Declare` Transactions Bypass `max_l2_gas_amount` Upper-Bound Admission Check in Gateway - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` applies the `max_l2_gas_amount` upper-bound check to `Invoke` and `DeployAccount` transactions but silently skips it for `Declare` transactions. An unprivileged user can submit a `Declare` transaction with `l2_gas.max_amount = u64::MAX`, which the gateway admits unconditionally while an identical `Invoke` or `DeployAccount` would be rejected.

### Finding Description

In `validate_resource_bounds`, the check for the L2 gas amount ceiling is written as:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The `if let RpcTransaction::Declare(_) = tx { }` branch is an empty arm that falls through without any check. The `else if` branch that enforces the ceiling is therefore unreachable for `Declare` transactions. The codebase even has a test that explicitly asserts this bypass is the current behavior:

```rust
fn valid_l2_gas_amount_on_declare(…) {
    // config: max_l2_gas_amount: 100, tx: max_amount: GasAmount(200)
    assert_matches!(tx_validator.validate(&tx), Ok(()));
}
``` [2](#0-1) 

The `max_l2_gas_amount` ceiling exists to prevent oversized transactions from entering the mempool. For `AllResources` bounds, `initial_sierra_gas` returns `l2_gas.max_amount` directly as the Sierra VM gas limit for the transaction:

```rust
TransactionInfo::Current(CurrentTransactionInfo {
    resource_bounds: ValidResourceBounds::AllResources(AllResourceBounds { l2_gas, .. }),
    ..
}) => { l2_gas.max_amount }
``` [3](#0-2) 

A `Declare` transaction with `l2_gas.max_amount = u64::MAX` therefore passes gateway admission and enters the mempool with an effectively unbounded Sierra gas limit, while the same value in an `Invoke` or `DeployAccount` transaction is rejected at the gateway.

### Impact Explanation

The gateway's `max_l2_gas_amount` admission control is bypassed for the entire `Declare` transaction type. Any unprivileged user can submit a `Declare` transaction with `l2_gas.max_amount = u64::MAX` (or any value exceeding `config.max_l2_gas_amount`), which:

1. Is admitted to the mempool unconditionally.
2. Reaches the blockifier with `initial_sierra_gas = GasAmount(u64::MAX)`, giving the transaction an unbounded Sierra VM gas limit.
3. Allows the Declare transaction to consume far more L2 gas per execution than the gateway's configured ceiling was intended to permit.

This matches the required impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

Likelihood is high. The trigger requires no privilege — any account can broadcast a `Declare` transaction. The bypass is structural (an empty `if` arm), not conditional on any runtime state. The TODO comment confirms the gap is known but unresolved.

### Recommendation

Remove the type-based branch and apply the same upper-bound check uniformly:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If `Declare` transactions legitimately require a higher ceiling, introduce a separate `max_l2_gas_amount_declare` config field and enforce it explicitly rather than skipping the check entirely.

### Proof of Concept

1. Construct an `RpcDeclareTransactionV3` with `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)` and `resource_bounds.l2_gas.max_price_per_unit >= config.min_gas_price`.
2. Submit it to the gateway's `add_tx` endpoint.
3. Observe that `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never raised.
4. Submit an identical `RpcInvokeTransactionV3` with the same `l2_gas.max_amount`; observe it is rejected with `MaxGasAmountTooHigh`.

The divergence is confirmed by the existing test `valid_l2_gas_amount_on_declare` which asserts `Ok(())` for a Declare with `max_amount: GasAmount(200)` when `max_l2_gas_amount: 100`. [4](#0-3) [5](#0-4)
