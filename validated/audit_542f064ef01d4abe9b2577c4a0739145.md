Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual user — allowlist bypass via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
The `SwapAllowlistExtension` is designed to gate swap access per individual swapper address. However, when a user routes through `MetricOmmSimpleRouter`, the pool's `swap()` function passes `msg.sender` (the router contract) as the `sender` argument to the extension's `beforeSwap` hook. The extension then checks whether the **router** is allowlisted, not the actual end user. Any user can bypass the per-user allowlist by calling through the router if the router address is allowlisted.

## Finding Description
**Root cause — sender identity mismatch:**

In `MetricOmmPool.swap()` (line 230–240), the pool passes `msg.sender` as the `sender` to `_beforeSwap`:
```solidity
_beforeSwap(
    msg.sender,   // <-- this is the router, not the end user
    recipient,
    ...
);
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle()` (line 72–80), the router calls `pool.swap(...)` directly. From the pool's perspective, `msg.sender` is the **router address**, not the originating user.

In `SwapAllowlistExtension.beforeSwap()` (line 37–39), the check is:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool (calling the extension), and `sender` is the router address. So the check resolves to `allowedSwapper[pool][router]` — it checks whether the **router** is allowlisted, not the actual end user.

**Exploit flow:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a specific set of users.
2. Pool admin allowlists the `MetricOmmSimpleRouter` address (or it is allowlisted by default for usability).
3. Any unprivileged user — not on the per-user allowlist — calls `MetricOmmSimpleRouter.exactInputSingle()` targeting the restricted pool.
4. The pool calls `_beforeSwap(msg.sender=router, ...)`, the extension checks `allowedSwapper[pool][router]` which is `true`, and the swap proceeds.
5. The allowlist restriction is completely bypassed.

**Existing guards are insufficient:** The extension has no mechanism to recover the original `tx.origin` or the router's caller. The `sender` parameter it receives is structurally bound to `msg.sender` of the pool's `swap()` call, which is always the router when the router is used.

## Impact Explanation
The `SwapAllowlistExtension` is a core production extension whose stated purpose is "Gates `swap` by swapper address, per pool." This bypass allows any unprivileged user to execute swaps on a pool that is intended to be restricted, breaking the allowlist's access-control invariant. Depending on the pool's purpose (e.g., institutional-only pools, KYC-gated pools), this can result in unauthorized fund flows and broken core pool functionality for the pool operator's intended use case. This constitutes broken core pool functionality and an admin-boundary break reachable by an unprivileged trader.

## Likelihood Explanation
The attack requires only that the `MetricOmmSimpleRouter` is allowlisted for the target pool — a likely operational condition since the router is the primary user-facing entry point. Any unprivileged user can execute this with a standard router call. No special privileges, flash loans, or complex setup are needed. The attack is repeatable on every swap.

## Recommendation
The extension should not rely on the `sender` parameter (which reflects the immediate pool caller) to identify the end user. Instead, one of the following fixes should be applied:
1. **Pass `tx.origin` or a user-supplied identity through `extensionData`** and verify it in the extension, with the router explicitly encoding `msg.sender` (the user) into `extensionData` before calling the pool.
2. **Require direct pool interaction** for allowlisted pools (document that the router must not be allowlisted and that users must call the pool directly).
3. **Have the router forward the original caller** via a dedicated field in `extensionData`, and update `SwapAllowlistExtension.beforeSwap` to decode and check that field when present.

function test_allowlistBypassViaRouter() public {
    // Setup: pool with SwapAllowlistExtension, router is allowlisted
    allowlistExt.setAllowedToSwap(pool, address(router), true);
    // attacker is NOT allowlisted
    assertFalse(allowlistExt.isAllowedToSwap(pool, attacker));

    // Attacker calls through router — swap succeeds despite not being allowlisted
    vm.prank(attacker);
    router.exactInputSingle(ExactInputSingleParams({
        pool: pool,
        recipient: attacker,
        tokenIn: token0,
        zeroForOne: true,
        amountIn: 1e18,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    }));
    // No revert — allowlist bypassed
}
