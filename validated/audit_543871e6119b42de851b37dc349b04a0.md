### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the 64-Byte Transaction Forgery Guard Added in v2 - (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase merkle proof. However, the original `verify_transaction_inclusion` (v1) is still a fully public, unguarded NEAR entry point. Any unprivileged caller can invoke v1 directly, skipping the coinbase validation entirely and obtaining a `true` result for a forged transaction inclusion proof.

### Finding Description

`verify_transaction_inclusion_v2` enforces a coinbase merkle proof check before delegating to v1:

```rust
// lib.rs lines 358-365
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
``` [1](#0-0) 

After passing that guard, v2 calls v1 internally:

```rust
// lib.rs line 368
self.verify_transaction_inclusion(args.into())
``` [2](#0-1) 

But v1 itself is still a public `#[pause]`-decorated function with no access control, callable by any NEAR account:

```rust
// lib.rs lines 287-288
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
``` [3](#0-2) 

The `#[deprecated]` Rust attribute is a compiler lint only; it does not restrict on-chain invocation. The coinbase guard in v2 is therefore a dead letter: any caller who routes through v1 directly receives the same `bool` result without the forgery protection.

The 64-byte transaction attack works because Bitcoin's Merkle tree hashes pairs of 32-byte child hashes into a 64-byte input. A crafted 64-byte "transaction" can be made to collide with an internal Merkle node, causing `compute_root_from_merkle_proof` to return the correct root for a transaction that does not exist. The coinbase proof in v2 prevents this by anchoring the tree structure at index 0, but v1 has no such anchor. [4](#0-3) 

### Impact Explanation

A consumer contract (bridge, atomic swap, cross-chain lending) that calls `verify_transaction_inclusion` (v1) — whether by mistake, by following outdated documentation, or because an attacker supplies a crafted call — will receive `true` for a Bitcoin transaction that was never broadcast. This allows an attacker to claim funds on NEAR for a Bitcoin payment that never occurred, directly corrupting the cross-chain trust assumption the light client is designed to provide.

### Likelihood Explanation

The entry point is reachable by any unprivileged NEAR account with no special role or staking requirement. The `#[trusted_relayer]` macro is not applied to either verification function. The 64-byte transaction forgery technique is publicly documented (referenced in the contract's own deprecation notice at line 268). Any consumer contract that has not been updated to call v2, or that is deliberately targeted by an attacker who supplies a crafted `ProofArgs`, is vulnerable. [5](#0-4) 

### Recommendation

Remove the public exposure of `verify_transaction_inclusion` (v1) entirely, or gate it with an access-control role so it cannot be called by unprivileged accounts. The safest fix is to make v1 `pub(crate)` or `private`, forcing all external callers through `verify_transaction_inclusion_v2`.

```rust
// Change:
pub fn verify_transaction_inclusion(...) -> bool {
// To:
fn verify_transaction_inclusion(...) -> bool {
```

This mirrors the fix applied in the JOJO report: the bypassed guard must be enforced on every externally reachable code path that produces the same security-critical result.

### Proof of Concept

1. Attacker identifies a block whose Merkle tree has an internal node `N` whose 64-byte serialization is a valid-looking `tx_id`.
2. Attacker constructs `ProofArgs { tx_id: N, tx_block_blockhash: <valid block>, tx_index: <index of N>, merkle_proof: <path to root>, confirmations: 1 }`.
3. Attacker calls `verify_transaction_inclusion(args)` directly on the NEAR contract.
4. `compute_root_from_merkle_proof(N, index, proof)` returns the block's real `merkle_root` because `N` is a genuine internal node.
5. The function returns `true`.
6. A consumer bridge contract, observing this `true` result, releases funds on NEAR for a Bitcoin transaction that never existed.

`verify_transaction_inclusion_v2` would have rejected this at step 3 because the coinbase proof would not validate consistently with the forged `tx_id` path, but v1 has no such check. [6](#0-5)

    /// Verifies that a transaction is included in a block at a given block height
    ///
    /// # Deprecated
    /// Use [`verify_transaction_inclusion_v2`] instead, which includes coinbase merkle proof validation
    /// to mitigate the 64-byte transaction Merkle proof forgery vulnerability:
    /// https://www.bitmex.com/blog/64-Byte-Transactions
    ///
    /// @param `tx_id` transaction identifier
    /// @param `tx_block_blockhash` block hash at which transacton is supposedly included
    /// @param `tx_index` index of transaction in the block's tx merkle tree
    /// @param `merkle_proof` merkle tree path (concatenated LE sha256 hashes) (does not contain initial `transaction_hash` and `merkle_root`)
    /// @param confirmations how many confirmed blocks we want to have before the transaction is valid
    /// @return True if `tx_id` is at the claimed position in the block at the given blockhash, False otherwise
    ///
    /// # Warning
    /// This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash.
    /// We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification.
    ///
    /// # Panics
    /// Multiple cases
    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
