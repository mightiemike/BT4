### Title
Odd-Width Merkle Tree Duplicate-Leaf Index Ambiguity Allows Ghost-Transaction Proof — (`merkle-tools/src/lib.rs::compute_root_from_merkle_proof`, `contract/src/lib.rs::verify_transaction_inclusion`)

---

### Summary

`compute_root_from_merkle_proof` performs no bounds check on `transaction_position` and has no awareness of the actual leaf count. In Bitcoin's Merkle tree, a block with an odd number of transactions duplicates the last leaf. This makes the same `(tx_id, merkle_proof)` tuple produce the correct Merkle root for **two distinct index values** — the real last index `N-1` and the nonexistent ghost index `N`. `verify_transaction_inclusion` returns `true` for the ghost index, certifying inclusion of a transaction that does not exist in the block.

---

### Finding Description

`compute_root_from_merkle_proof` iterates over the proof array, choosing left/right placement purely from `current_position % 2`, then halves the position: [1](#0-0) 

There is no check that `transaction_position` is less than the number of leaves, and no detection that a sibling hash equals the current hash (the fingerprint of odd-width duplication).

`verify_transaction_inclusion` passes the caller-supplied `tx_index` directly into this function without any upper-bound validation: [2](#0-1) 

**Concrete arithmetic for a 3-transaction block `[T0, T1, T2]`:**

Bitcoin constructs the tree by padding to even width: `[T0, T1, T2, T2]`.

| Level | Nodes |
|-------|-------|
| 0 | T0, T1, T2, T2 |
| 1 | H(T0,T1), H(T2,T2) |
| 2 (root) | H(H(T0,T1), H(T2,T2)) |

The legitimate proof for T2 at index 2 is `proof = [T2, H(T0,T1)]` (sibling at level 0 is the duplicate T2).

**Legitimate call** — `compute_root_from_merkle_proof(T2, 2, [T2, H(T0,T1)])`:
- pos=2 (even) → `H(T2, T2)`; pos→1
- pos=1 (odd) → `H(H(T0,T1), H(T2,T2))` = root ✓

**Attacker's ghost call** — `compute_root_from_merkle_proof(T2, 3, [T2, H(T0,T1)])`:
- pos=3 (odd) → `H(T2, T2)`; pos→1
- pos=1 (odd) → `H(H(T0,T1), H(T2,T2))` = root ✓

Both calls return the same root. Index 3 does not correspond to any real transaction; the contract nonetheless returns `true`.

The same ambiguity propagates into `verify_transaction_inclusion_v2`, which calls `verify_transaction_inclusion` after its coinbase check: [3](#0-2) 

The coinbase guard addresses the 64-byte forgery attack, not the odd-width duplication attack.

---

### Impact Explanation

A downstream bridge, unlock, or mint contract that uses `(tx_id, tx_index)` as its replay-protection key — a natural choice when a single TXID can appear at multiple logical positions — will process the same economic event twice. The attacker:

1. Observes a real deposit transaction T2 at index N-1 in any odd-transaction-count block.
2. Submits the legitimate proof once to unlock/mint funds (index N-1).
3. Resubmits the identical `(tx_id=T2, merkle_proof=[T2, …])` with `tx_index=N` (ghost position).
4. The light client returns `true`; the bridge sees a new `(T2, N)` key and releases funds a second time.

The contract stores no record of which proofs have been verified; replay tracking is entirely the caller's responsibility.

---

### Likelihood Explanation

- Odd transaction counts are common in real Bitcoin blocks.
- `verify_transaction_inclusion` is a public, unpermissioned NEAR call (only gated by `#[pause]`, not by any role or access-key restriction). [4](#0-3) 

- The attacker needs only public blockchain data (the block's transaction list) and the ability to call the NEAR contract.
- The `confirmations <= gc_threshold` guard only ensures the block is still stored; it does not constrain `tx_index`. [5](#0-4) 

---

### Recommendation

1. **Bounds-check `tx_index`**: The block header does not encode the transaction count, so the light client cannot enforce this alone. The recommended fix is to require callers to supply the total transaction count and validate `tx_index < tx_count` before calling `compute_root_from_merkle_proof`.

2. **Detect self-sibling duplication**: Inside `compute_root_from_merkle_proof`, reject (or at minimum flag) any proof step where `proof_hash == current_hash`, which is the exact signature of odd-width padding.

3. **Deprecate v1 more aggressively**: `verify_transaction_inclusion` is marked deprecated but remains fully callable. Consider adding a runtime `require!(false, "use v2")` guard or removing it entirely, since `verify_transaction_inclusion_v2` also inherits this bug via its internal call. [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
