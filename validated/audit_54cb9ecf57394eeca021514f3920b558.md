### Title
Unbounded `ValidatorEvents` array in `processEventsAndClaimBonus` can permanently lock position funds — (`x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

The `processEventsAndClaimBonus` function in the tieredrewards module iterates over every validator event accumulated since a position's `LastEventSeq` in a single transaction. Because there is no cap on how many events can be appended to a validator's event log, a position with a large event backlog will eventually fail with gas exhaustion. Since `claimRewards` (which calls `processEventsAndClaimBonus`) is a mandatory prerequisite in `TierUndelegate`, `TierRedelegate`, `AddToTierPosition`, `ClearPosition`, and `ExitTierWithDelegation`, all of these operations will revert, permanently locking the delegator's staked tokens inside the position.

---

### Finding Description

**Root cause — unbounded event append:**

`appendValidatorEvent` in `validator_events.go` stores a new `ValidatorEvent` for a validator on every bond, unbond, or slash transition. There is no limit on how many events can accumulate. [1](#0-0) 

**Root cause — unbounded event fetch:**

`getValidatorEventsSince` walks the entire `ValidatorEvents` collection from `startSeq+1` to the end of the validator's prefix, collecting every entry into a slice with no size cap. [2](#0-1) 

**Critical iteration path:**

`processEventsAndClaimBonus` calls `getValidatorEventsSince` and then iterates over the returned slice in a single transaction. Each iteration performs storage reads (`decrementEventRefCount`) and arithmetic, making the gas cost linear in the number of accumulated events. [3](#0-2) 

**Mandatory call sites — all critical position operations:**

`claimRewards` (which calls `processEventsAndClaimBonus`) is invoked unconditionally before the state-changing logic in:

- `TierUndelegate` (line 166)
- `TierRedelegate` (line 229)
- `AddToTierPosition` (line 314)
- `ClearPosition` (line 406)
- `ExitTierWithDelegation` (line 532) [4](#0-3) [5](#0-4) [6](#0-5) 

**Reference-count cleanup does not prevent accumulation:**

Events are only deleted when their `ReferenceCount` reaches zero, i.e., when every position that was alive at event creation has processed it. A position that never calls `claimRewards` keeps the reference count above zero indefinitely, so events accumulate without bound for that position. [7](#0-6) 

**Governance tally also iterates all positions per owner:**

`GetPositionStatesByOwner`, called during every governance tally, iterates all positions for a voter. If a single voter has many positions, this path is also affected. [8](#0-7) 

---

### Impact Explanation

Once the event backlog for a validator grows large enough that processing it in a single transaction exceeds the per-transaction gas limit, every call to `TierUndelegate`, `TierRedelegate`, `AddToTierPosition`, `ClearPosition`, and `ExitTierWithDelegation` will revert for any position delegated to that validator whose `LastEventSeq` lags behind. The delegator's staked tokens remain locked inside the position with no reachable exit path. The exact corrupted value is the delegator's bonded token balance held by the position's derived delegator account, which becomes permanently inaccessible.

---

### Likelihood Explanation

Validator bond/unbond/slash events are produced by normal Cosmos SDK staking operations: a validator being jailed for downtime produces an unbond event; unjailing produces a bond event; each slashing incident produces a slash event. A validator that is repeatedly jailed and unjailed (e.g., due to intermittent downtime) will accumulate events at a rate proportional to the number of jail/unjail cycles. A position holder who infrequently claims rewards (or never does) will accumulate a backlog equal to all events since position creation. The likelihood is low under normal conditions but rises to medium for long-lived positions on validators with a history of jailing, or in adversarial scenarios where a validator operator deliberately cycles jail/unjail state.

---

### Recommendation

1. **Hard cap on events processed per call:** In `processEventsAndClaimBonus`, process at most `N` events per transaction (e.g., 100) and persist the updated `LastEventSeq` so subsequent calls continue from where the previous one stopped. This makes reward claiming a multi-step operation but prevents gas exhaustion.

2. **Hard cap on total events per validator:** In `appendValidatorEvent`, enforce a maximum number of stored events per validator (e.g., reject or overwrite the oldest when the cap is reached).

3. **Decouple `claimRewards` from exit paths:** Allow `TierUndelegate` and `ExitTierWithDelegation` to proceed without requiring a full reward claim, so that a user can always exit their position even if the event backlog is too large to process in one transaction.

---

### Proof of Concept

1. User calls `LockTier` or `CommitDelegationToTier`, creating a position delegated to validator `V`. The position's `LastEventSeq` is set to the current latest sequence for `V`.
2. Validator `V` is jailed for downtime → `appendValidatorEvent` stores an `UNBOND` event (seq N+1, refcount ≥ 1).
3. Validator operator calls `MsgUnjail` → `appendValidatorEvent` stores a `BOND` event (seq N+2, refcount ≥ 1).
4. Steps 2–3 repeat many times (e.g., 5,000 cycles). The user never calls `ClaimTierRewards`, so `LastEventSeq` remains at N and `decrementEventRefCount` is never called for this position. All 10,000 events remain in storage.
5. User attempts `TierUndelegate`. The call reaches `claimRewards` → `processEventsAndClaimBonus` → `getValidatorEventsSince` returns 10,000 entries. The loop at line 172 of `claim_rewards.go` performs 10,000 iterations, each with storage reads and arithmetic, exhausting the transaction gas limit.
6. The transaction reverts. The same outcome occurs for `TierRedelegate`, `AddToTierPosition`, `ClearPosition`, and `ExitTierWithDelegation`.
7. The user's staked tokens are permanently locked inside the position's derived delegator account with no reachable exit. [9](#0-8) [1](#0-0)

func (k Keeper) getValidatorEventsSince(ctx context.Context, valAddr sdk.ValAddress, startSeq uint64) ([]EventEntry, error) {
	// Range from (valAddr, startSeq+1) to end of valAddr prefix.
	rng := collections.NewPrefixedPairRange[sdk.ValAddress, uint64](valAddr).
		StartExclusive(startSeq)

	var entries []EventEntry
	err := k.ValidatorEvents.Walk(ctx, rng, func(key collections.Pair[sdk.ValAddress, uint64], event types.ValidatorEvent) (bool, error) {
		entries = append(entries, EventEntry{Seq: key.K2(), Event: event})
		return false, nil
	})
	if err != nil {
		return nil, err
	}
	return entries, nil
}

func (k Keeper) decrementEventRefCount(ctx context.Context, valAddr sdk.ValAddress, seq uint64) error {
	key := collections.Join(valAddr, seq)
	event, err := k.ValidatorEvents.Get(ctx, key)
	if errors.Is(err, collections.ErrNotFound) {
		return nil // already cleaned up
	}
	if err != nil {
		return err
	}

	if event.ReferenceCount <= 1 {
		return k.ValidatorEvents.Remove(ctx, key)
	}

	event.ReferenceCount--
	return k.ValidatorEvents.Set(ctx, key, event)
}

func (k Keeper) processEventsAndClaimBonus(ctx context.Context, pos *types.PositionState) (sdk.Coins, error) {
	// Rewards should have been claimed before undelegation
	if !pos.IsDelegated() {
		return sdk.NewCoins(), nil
	}

	valAddr, err := sdk.ValAddressFromBech32(pos.Delegation.ValidatorAddress)
	if err != nil {
		return nil, err
	}

	events, err := k.getValidatorEventsSince(ctx, valAddr, pos.LastEventSeq)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	blockTime := sdkCtx.BlockTime()

	totalBonus := math.ZeroInt()
	// Use the persisted bonded state from the last replay, not a hardcoded default.
	// This prevents overpaying bonus for unbonded gaps between claims.
	bonded := pos.LastKnownBonded
	segmentStart := pos.LastBonusAccrual

	tier, err := k.getTier(ctx, pos.TierId)
	if err != nil {
		return nil, err
	}

	for _, entry := range events {
		evt := entry.Event

		if bonded {
			// Compute bonus for the bonded segment [segmentStart, eventTime]
			// using the snapshot rate at the event.
			bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
			totalBonus = totalBonus.Add(bonus)
		}

		// Update bonded state based on event type.
		switch evt.EventType {
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_UNBOND:
			bonded = false
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_BOND:
			bonded = true
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_SLASH:
			// Slash doesn't change bonded state.
		}

		segmentStart = evt.Timestamp
		pos.UpdateLastEventSeq(entry.Seq)

		// Decrement reference count.
		if err := k.decrementEventRefCount(ctx, valAddr, entry.Seq); err != nil {
			return nil, err
		}
	}

func (k Keeper) GetPositionStatesByOwner(ctx context.Context, owner sdk.AccAddress) ([]types.PositionState, error) {
	ids, err := k.getPositionsIdsByOwner(ctx, owner)
	if err != nil {
		return nil, err
	}
	states := make([]types.PositionState, 0, len(ids))

			continue

		}
