Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks position `owner` instead of actual depositor `sender`, allowing unprivileged allowlist bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter and gates access solely on `owner`. Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` allows any caller to supply an arbitrary `owner` address, an unprivileged address can bypass the deposit allowlist by nominating an allowlisted address as the position owner while remaining the actual token payer. The admin-configured access-control boundary is fully circumvented.

## Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` (the direct caller of `pool.addLiquidity`) as its first argument but discards it with an unnamed parameter, checking only `owner`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` correctly passes both `sender` and `owner` to the extension: [2](#0-1) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (the explicit-owner overload) accepts any caller-supplied `owner` and records `msg.sender` as the token payer: [3](#0-2) 

`_validateOwner` only rejects `address(0)`, imposing no allowlist check: [4](#0-3) 

`_addLiquidity` calls `pool.addLiquidity(positionOwner, ...)`, making the pool's `msg.sender` the `LiquidityAdder` contract. The pool passes `sender = LiquidityAdder` and `owner = positionOwner` to the extension. The extension evaluates `allowedDepositor[pool][positionOwner]` — the actual token payer is never examined: [5](#0-4) 

## Impact Explanation

An address not on the deposit allowlist can deposit tokens into any pool configured with `DepositAllowlistExtension` as a `beforeAddLiquidity` hook. The admin-configured access-control boundary is bypassed by any unprivileged caller. This is a direct admin-boundary break: a pool admin-configured guard is circumvented through an unprivileged periphery path. Secondary impact: the allowlisted address receives an unsolicited LP position it did not authorize; if that address is a contract without `removeLiquidity` logic, the deposited tokens become permanently locked.

## Likelihood Explanation

No special role or privilege is required. Any EOA or contract can call `MetricOmmPoolLiquidityAdder.addLiquidityExactShares`. The allowlisted addresses to nominate as `owner` are publicly readable from the `allowedDepositor` mapping. The bypass is deterministic and requires no oracle manipulation, flash loan, or timing dependency. [6](#0-5) 

## Recommendation

Change `beforeAddLiquidity` to check the `sender` parameter (the direct caller of `pool.addLiquidity`) rather than `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Note: when `MetricOmmPoolLiquidityAdder` is the intermediary, `sender` will be the adder's address, so the adder itself must be allowlisted for router-mediated deposits. If per-user gating through the adder is required, the adder must forward the original `msg.sender` via `extensionData` and the extension must decode it — or the pool admin must allowlist the adder and rely on the adder's own access controls.

## Proof of Concept

1. Pool `P` is deployed with `DepositAllowlistExtension` as a `beforeAddLiquidity` hook.
2. Pool admin calls `setAllowedToDeposit(P, Alice, true)`. Alice is the only allowlisted depositor.
3. Bob (not allowlisted) calls `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(P, owner=Alice, salt, deltas, max0, max1, "")`.
4. `_addLiquidity` stores `payer = Bob` in transient context and calls `P.addLiquidity(Alice, salt, deltas, abi.encode(KIND_PAY), "")`.
5. Pool calls `DepositAllowlistExtension.beforeAddLiquidity(sender=LiquidityAdder, owner=Alice, ...)`.
6. Extension evaluates `allowedDepositor[P][Alice]` → `true` → no revert.
7. Pool proceeds; callback fires; Bob's tokens are pulled via `pay(token, Bob, P, amount)`.
8. Position is recorded under `(Alice, salt)`. Bob's tokens are now in the pool. The deposit allowlist did not block Bob. [7](#0-6)
