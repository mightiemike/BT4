Audit Report

## Title
Router-Mediated Swaps Bypass `SwapAllowlistExtension` by Substituting Router Address for Actual Swapper Identity - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against a per-pool allowlist. The pool always passes its own `msg.sender` as `sender`, so when `MetricOmmSimpleRouter` calls `pool.swap()`, the extension receives the router address instead of the originating user. This makes the identity check structurally wrong for every router-mediated swap, producing either a full allowlist bypass or a complete block of router-based swaps for allowlisted users.

## Finding Description
**Root cause — pool unconditionally passes `msg.sender` as `sender`:**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool passed: [3](#0-2) 

**Router never forwards the original user to the pool:**

`MetricOmmSimpleRouter.exactInputSingle` stores `msg.sender` only in transient storage for the payment callback, then calls `pool.swap()` with no user identity argument — making the router itself `msg.sender` at the pool: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

**Result:** `allowedSwapper[pool][router]` is evaluated instead of `allowedSwapper[pool][user]`. The actual user identity is permanently lost before the extension check.

## Impact Explanation
Two concrete failure modes arise from the same root cause:

**Mode A — Allowlist bypass (High):** A pool admin who wants to permit router-based swaps must add the router address to the allowlist (`setAllowedToSwap(pool, router, true)`). Once done, every user — including those explicitly excluded — can bypass the allowlist by routing through `MetricOmmSimpleRouter`. The curated pool becomes effectively public, breaking the admin-configured access boundary.

**Mode B — Broken core functionality (High):** If the admin does not add the router, every allowlisted user is silently blocked from using the supported periphery path. `SwapAllowlistExtension` is a first-party periphery contract; its structural incompatibility with the first-party router breaks the core swap flow for any pool that deploys it.

Both outcomes satisfy the required impact gate: broken core pool functionality causing unusable swap flows, and admin-boundary break where an unprivileged path bypasses a configured access control.

## Likelihood Explanation
No special preconditions, oracle manipulation, or non-standard tokens are required. Any pool that deploys `SwapAllowlistExtension` and expects to support router-based trading encounters this immediately. Mode A requires only that the admin adds the router to the allowlist (the only way to make the router work with the extension at all), after which any unprivileged user can call `MetricOmmSimpleRouter.exactInputSingle` to bypass the allowlist. The attack is repeatable with zero cost beyond gas.

## Recommendation
The extension must receive the true originating user, not the immediate caller. Two options:

1. **Pass the original user through the pool's `swap` signature** — add a `swapper` parameter distinct from `msg.sender` so the router can supply `msg.sender` (the end user) explicitly. The pool still uses `msg.sender` for the callback, but passes the explicit `swapper` to extensions.

2. **Encode the original user in `extensionData`** — the router encodes `msg.sender` into `extensionData`; the extension decodes and verifies it. This is only safe if the pool validates the caller is a known router, introducing a trust assumption.

Option 1 is cleaner and avoids trust assumptions.

## Proof of Concept
```solidity
// Setup: pool with SwapAllowlistExtension; only `allowedUser` is allowlisted.
swapExtension.setAllowedToSwap(pool, allowedUser, true);
// Admin also adds router so router-based swaps work at all:
swapExtension.setAllowedToSwap(pool, address(router), true);

// Attack: disallowedUser routes through the router.
vm.startPrank(disallowedUser);
token1.approve(address(router), type(uint256).max);
// Succeeds — the hook sees sender=router, which is allowlisted.
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: disallowedUser,
    zeroForOne: false,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: type(uint128).max,
    tokenIn: address(token1),
    deadline: block.timestamp,
    extensionData: ""
}));
// disallowedUser successfully swapped on a pool they were explicitly excluded from.
```

The `beforeSwap` check resolves to `allowedSwapper[pool][router] == true`, so it passes regardless of who the actual user is. [6](#0-5)
