### Title
Deprecated `verify_transaction_inclusion` Accepts Arbitrary `tx_id` Enabling Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function remains publicly callable on-chain and accepts a caller-supplied `tx_id` without validating it is a leaf node (real transaction). Any unprivileged NEAR caller can supply a `tx_id` equal to an internal Merkle tree node hash, causing the function to return `true` for a Bitcoin transaction that does not exist.

---

### Finding Description

`verify_transaction_inclusion` in `contract/src/lib.rs` accepts `ProofArgs` containing a `tx_id` and verifies it by computing the Merkle root from the supplied proof and comparing it to the stored block header's `merkle_root`. The function performs no validation that `tx_id` is a leaf node rather than an internal node of the Merkle tree.

The contract's own inline documentation acknowledges this broken invariant: [1](#0-0) 

Despite being marked `#[deprecated]` since v0.5.0, the function carries no runtime enforcement — `#[deprecated]` in Rust is a compiler-only lint. The function remains `pub` and decorated with `#[pause]` (not paused by default), making it callable by any NEAR account: [2](#0-1) 

The core verification step that is exploitable: [3](#0-2) 

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` treats the supplied `transaction_hash` as an opaque 32-byte value and hashes it upward through the tree without any leaf-vs-internal-node distinction: [4](#0-3) 

Bitcoin's Merkle tree does not tag leaf nodes differently from internal nodes. An internal node at depth D, position P in a real block's tree has a valid sibling path to the root. An attacker who knows the block's full transaction set can compute that path and present the internal node as a `tx_id`, satisfying the root comparison.

The only guards in the function — confirmations check, mainchain membership check, and non-empty proof check — are all satisfiable with a real mainchain block: [5](#0-4) 

`verify_transaction_inclusion_v2` was introduced specifically to close this gap by requiring a coinbase proof at position 0, which anchors the tree structure: [6](#0-5) 

But v1 remains reachable and unguarded.

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate a fund release (e.g., a Bitcoin bridge, a payment-verification contract, or a cross-chain settlement layer) will accept a forged proof as valid. An attacker can claim a Bitcoin payment was made when it was not, causing the downstream contract to release funds. This is a direct analog to the external report: an unvalidated caller-supplied parameter bypasses the critical proof check, enabling theft of funds held by contracts that trust the return value.

The corrupted proof result is concrete and exact: `verify_transaction_inclusion` returns `true` for a `tx_id` that does not correspond to any transaction in the block.

---

### Likelihood Explanation

High. The entry path requires no privileged role, no leaked key, and no social engineering. Any NEAR account can call `verify_transaction_inclusion` directly. The required inputs — a real mainchain block hash and the block's Merkle tree structure — are publicly available from any Bitcoin block explorer. The 64-byte Merkle forgery technique is well-documented and has working public implementations.

---

### Recommendation

Remove `verify_transaction_inclusion` from the contract entirely. If a deprecation period is required, replace the function body with `env::panic_str("use verify_transaction_inclusion_v2")` so the runtime rejects all calls rather than silently returning a potentially forged result. All callers must migrate to `verify_transaction_inclusion_v2`, which anchors the tree structure via the coinbase proof and closes the forgery path.

---

### Proof of Concept

1. Identify a Bitcoin block at height H that is present in the contract's mainchain (i.e., `mainchain_header_to_height` contains its hash).
2. Obtain the block's full transaction list from a public Bitcoin node or explorer and reconstruct the Merkle tree.
3. Select any internal node N at depth D, position P in the tree (e.g., the parent of the first two transactions).
4. Compute the sibling path from N up to the Merkle root — this is a valid proof for N treated as a "transaction."
5. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node hash, 32 bytes)
   - `tx_block_blockhash = H` (the real block hash)
   - `tx_index = P`
   - `merkle_proof = [sibling hashes from N to root]`
   - `confirmations = 1`
6. The function returns `true`. No transaction with hash N exists in block H.

Any downstream contract checking `verify_transaction_inclusion(...) == true` before releasing funds will be deceived into treating this forged proof as a valid Bitcoin payment confirmation.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
