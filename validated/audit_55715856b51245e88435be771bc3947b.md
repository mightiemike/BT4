### Title
ML-DSA-65 Gas Key Host Functions Use Wire Length Instead of Trie ID Length for Fee Calculation — (File: `runtime/near-vm-runner/src/logic/logic.rs`)

### Summary

The `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, and `promise_batch_action_add_gas_key_with_function_call` host functions compute gas fees using the raw borsh wire length of the caller-supplied public key (`public_key_len as usize`). For an ML-DSA-65 key this is **1953 bytes**. The transaction path for the same actions uses `action.public_key.trie_id_len()`, which for ML-DSA-65 is **33 bytes** (the SHA3-256 hash stored in the trie). The divergence is ~59×, breaking the fee-parity invariant between the transaction path and the host-function path, and causing any contract that issues a `TransferToGasKey` or `AddGasKey` action for an ML-DSA-65 key via a host function to be overcharged by ~1920 × `gas_key_byte` gas units per call.

### Finding Description

**Transaction path** — `total_send_fees` and `exec_fee` in `runtime/runtime/src/config.rs` both call the fee helpers with `action.public_key.trie_id_len()`:

```rust
// runtime/runtime/src/config.rs  line 114-116
TransferToGasKey(action) => {
    gas_key_transfer_send_fee(fees, sender_is_receiver, action.public_key.trie_id_len())
        .total()
}
// line 347-350
TransferToGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
```

For `PublicKey::MLDSA65`, `trie_id_len()` returns `1 + ML_DSA_65_HASH_LENGTH = 33`.

**Host-function path** — `promise_batch_action_transfer_to_gas_key` in `runtime/near-vm-runner/src/logic/logic.rs` passes the raw caller-supplied byte count:

```rust
// logic.rs  line 3091-3096
let send = gas_key_transfer_send_fee(&self.fees_config, sir, public_key_len as usize);
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,   // ← wire length, not trie_id_len
);
```

When the contract passes a borsh-encoded ML-DSA-65 public key, `public_key_len` is 1953. The same mismatch exists in `promise_batch_action_add_gas_key_with_full_access` (line 3155–3160) and `promise_batch_action_add_gas_key_with_function_call` (line 3226–3231), which pass `public_key_len as usize` to `gas_key_add_key_exec_fee`.

`gas_key_transfer_exec_fee` is documented as modelling "the access key trie key length + estimated value length (what the receiver needs to read/write in the trie)". The trie key for an ML-DSA-65 access key is 33 bytes (hash form), not 1953 bytes (full pubkey). Using 1953 bytes inflates the computed trie-key length by 1920 bytes, producing a per-call overcharge of `1920 × gas_key_byte_exec_fee` gas units.

The existing fee-parity integration test (`test_gas_key_fee_parity_full_access` / `test_gas_key_fee_parity_function_call`) only exercises `KeyType::ED25519` keys, for which `len()` and `trie_id_len()` are identical (both 33 bytes), so the divergence is invisible in the test suite.

### Impact Explanation

Once both `GasKeys` and `PostQuantumSignatures` protocol features are active, any smart contract that calls `promise_batch_action_transfer_to_gas_key` or `promise_batch_action_add_gas_key_with_*` with an ML-DSA-65 public key is charged ~59× more gas than the equivalent signed transaction. The overcharge is deterministic and reproducible. Contracts designed around the expected fee budget will run out of gas, making ML-DSA-65 gas keys effectively unusable via host functions. The fee-parity invariant — that the same action costs the same gas whether submitted as a transaction or emitted by a contract — is broken at the protocol level for this key type.

### Likelihood Explanation

The bug is reachable by any unprivileged user once both `GasKeys` and `PostQuantumSignatures` are stabilised. No privileged role is required: a user deploys a contract, funds it, and calls the host function with an ML-DSA-65 public key. The root cause is a missing `trie_id_len()` call in three host-function implementations, and the existing test suite does not exercise the ML-DSA-65 code path.

### Recommendation

Replace `public_key_len as usize` with `public_key.trie_id_len()` (after decoding the key) in all three host-function implementations:

```rust
// After: let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
let key_trie_len = public_key.decode()?.trie_id_len();

let send = gas_key_transfer_send_fee(&self.fees_config, sir, key_trie_len);
let exec = gas_key_transfer_exec_fee(&self.fees_config, receiver_id.len(), key_trie_len);
```

Apply the same fix to `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call`. Extend `test_gas_key_fee_parity` to cover `KeyType::MLDSA65` to prevent regression.

```rust
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
```

```rust
pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =
        access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }
```

```rust
fn test_gas_key_fee_parity(mode: GasKeyKind) {
    let mut setup = setup_host_function_test();
    let account = setup.account.clone();

    let num_nonces: NonceIndex = 4;
    let gas_key_a_signer: Signer =
        InMemorySigner::from_seed(account.clone(), KeyType::ED25519, "gas_key_tx").into();
    let gas_key_b_signer: Signer =
        InMemorySigner::from_seed(account.clone(), KeyType::ED25519, "gas_key_host_fn").into();

    let public_key_b_base64 = near_primitives_core::serialize::to_base64(
        &borsh::to_vec(&gas_key_b_signer.public_key()).unwrap(),
    );

    // Add gas key A via transaction, B via host function
    let add_a_outcome = setup.run_actions(vec![Action::AddKey(Box::new(AddKeyAction {
        public_key: gas_key_a_signer.public_key(),
        access_key: mode.access_key(num_nonces, &account),
    }))]);
    let add_b_outcome = setup.run_call_promise(serde_json::json!([
        {"batch_create": {"account_id": account.as_str()}, "id": 0},
        mode.add_action_json(num_nonces, &account, &public_key_b_base64),
    ]));
    assert_eq!(add_a_outcome.gas_burnt, add_b_outcome.gas_burnt);
    assert_eq!(add_a_outcome.tokens_burnt, add_b_outcome.tokens_burnt);

    // Fund gas key A via transaction, B via host function
    let fund_amount = Balance::from_millinear(10);
    let fund_a_outcome =
        setup.run_actions(vec![Action::TransferToGasKey(Box::new(TransferToGasKeyAction {
            public_key: gas_key_a_signer.public_key(),
            deposit: fund_amount,
        }))]);
    let fund_b_outcome = setup.run_call_promise(serde_json::json!([
        {"batch_create": {"account_id": account.as_str()}, "id": 0},
        {"action_transfer_to_gas_key": {
            "promise_index": 0,
            "public_key": public_key_b_base64,
            "amount": fund_amount.as_yoctonear().to_string(),
        }, "id": 0},
    ]));
    assert_eq!(fund_a_outcome.gas_burnt, fund_b_outcome.gas_burnt);
    assert_eq!(fund_a_outcome.tokens_burnt, fund_b_outcome.tokens_burnt);
```
