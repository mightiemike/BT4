### Title
Hardcoded `protocol_reward_rate` Silently Overrides Configured Value on Mainnet/Testnet, Creating RPC Compatibility Divergence - (File: `chain/epoch-manager/src/reward_calculator.rs`)

### Summary

In `RewardCalculator::calculate_reward`, the protocol treasury reward rate (`protocol_reward_rate`) is unconditionally overridden with the hardcoded literal `Rational32::new_raw(1, 10)` (10%) whenever `genesis_protocol_version == PROD_GENESIS_PROTOCOL_VERSION` (29). This is the genesis version for both mainnet and testnet. The configured value from `GenesisConfig::protocol_reward_rate` is silently discarded. Meanwhile, the public `EXPERIMENTAL_protocol_config` RPC endpoint returns the configured (discarded) value, not the hardcoded value actually used in computation. The `_protocol_version` parameter accepted by `calculate_reward` is intentionally unused, meaning no future protocol upgrade can alter this rate for mainnet/testnet through the normal governance path.

### Finding Description

In `chain/epoch-manager/src/reward_calculator.rs`:

```rust
let use_hardcoded_value = self.genesis_protocol_version == PROD_GENESIS_PROTOCOL_VERSION;
let protocol_reward_rate = if use_hardcoded_value {
    Rational32::new_raw(1, 10)   // hardcoded 10%
} else {
    self.protocol_reward_rate    // configured value, only used for non-prod chains
};
``` [1](#0-0) 

`PROD_GENESIS_PROTOCOL_VERSION` is the constant `29`, which is the genesis protocol version for both mainnet and testnet. [2](#0-1) 

`RewardCalculator` is constructed from `GenesisConfig`, reading `config.protocol_reward_rate` into `self.protocol_reward_rate`: [3](#0-2) 

But `GenesisConfig` defines `protocol_reward_rate` as a fully configurable field: [4](#0-3) 

The `_protocol_version` parameter in `calculate_reward` is intentionally unused (underscore prefix), confirming no protocol-version-gated path can override the hardcoded value: [5](#0-4) 

The `EXPERIMENTAL_protocol_config` RPC handler returns `ProtocolConfigView`, which is built directly from `genesis_config.protocol_reward_rate` — the configured value that is silently discarded in the actual computation: [6](#0-5) 

The test in `reward_calculator.rs` explicitly confirms the divergence: it sets `protocol_reward_rate: Ratio::new(0, 1)` (0%) with `genesis_protocol_version: PROD_GENESIS_PROTOCOL_VERSION`, and the comment reads `// Unused, would only be used for genesis_protocol_version`: [7](#0-6) 

The test then asserts the protocol reward is `expected_total / 10` (10%), confirming the hardcoded value is used regardless of the configured value: [8](#0-7) 

### Impact Explanation

Every epoch on mainnet and testnet, `epoch_protocol_treasury` is computed using the hardcoded `1/10` rate, not the value in `GenesisConfig`: [9](#0-8) 

This produces two concrete broken invariants:

1. **RPC compatibility divergence**: Any caller of `EXPERIMENTAL_protocol_config` reads `protocol_reward_rate` from the response and assumes it reflects the actual treasury split. The actual split is always `1/10` on mainnet/testnet regardless of what the RPC reports. Tools, wallets, and analytics that use this field to compute expected treasury or validator rewards will produce wrong results if the genesis config value differs from `1/10`.

2. **Upgrade-boundary freeze**: The `_protocol_version` argument is unused. No protocol upgrade can change the treasury rate for mainnet/testnet through the standard governance path (80% stake vote → new protocol version → new parameter value). Any NEP or governance proposal that modifies `protocol_reward_rate` will be silently ignored for the two production networks.

### Likelihood Explanation

The condition `genesis_protocol_version == PROD_GENESIS_PROTOCOL_VERSION` is permanently true for every mainnet and testnet node. The code path is exercised every epoch boundary. The divergence between the RPC-reported value and the computed value is observable by any user querying `EXPERIMENTAL_protocol_config` and comparing it against on-chain reward distributions. No special privileges or attacker position are required to observe the discrepancy.

### Recommendation

Remove the hardcoded override. If backward compatibility with the historical 10% rate must be preserved, encode it as a versioned parameter in the `EpochConfig`/`RuntimeConfig` store (keyed by protocol version), so it is both visible through the standard config RPC and upgradeable through the normal governance path. At minimum, the `EXPERIMENTAL_protocol_config` response should reflect the value actually used in reward computation, not the value stored in genesis config.

### Proof of Concept

1. Query `EXPERIMENTAL_protocol_config` on mainnet. Observe `protocol_reward_rate` in the response (e.g., `[1, 10]` or any other configured value).
2. Independently compute expected treasury reward for any epoch: `treasury = total_epoch_reward × protocol_reward_rate_from_rpc`.
3. Compare against the actual treasury account balance delta at the epoch boundary.
4. For any genesis config where `protocol_reward_rate ≠ 1/10`, the RPC-derived computation will diverge from the on-chain result, because `calculate_reward` always uses `Rational32::new_raw(1, 10)` for `genesis_protocol_version == 29`.
5. Confirm in code: set `genesis_protocol_version = PROD_GENESIS_PROTOCOL_VERSION` and `protocol_reward_rate = Ratio::new(0, 1)` in a `RewardCalculator`, call `calculate_reward`, and observe the treasury receives 10% of epoch rewards, not 0%. [1](#0-0) [10](#0-9)
