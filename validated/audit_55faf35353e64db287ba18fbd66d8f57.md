Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Non-Allowlisted Callers to Bypass the Deposit Gate — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter and validates `owner` (the caller-supplied position beneficiary) instead. Because `addLiquidity` accepts an arbitrary `owner` address from `msg.sender`, any non-allowlisted address can bypass the gate by nominating an allowlisted address as `owner`, while the actual depositor is never checked. This breaks the core access-control invariant of permissioned pools using this extension.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as the first argument to `_beforeAddLiquidity`:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

The `DepositAllowlistExtension` override receives `(address sender, address owner, ...)` but discards `sender` entirely (unnamed first parameter), binding only `owner` to a named variable and checking it:

```solidity
// DepositAllowlistExtension.sol lines 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

The admin-facing setter names its parameter `depositor`, making the intended gating target unambiguous — the actual caller, not the position beneficiary: [3](#0-2) 

By contrast, the sibling `SwapAllowlistExtension.beforeSwap` correctly binds and checks `sender`: [4](#0-3) 

The exploit path is enabled by `removeLiquidity` enforcing `msg.sender == owner`, meaning only the allowlisted `owner` can withdraw — completing the collusion loop: [5](#0-4) 

## Impact Explanation
An unprivileged, non-allowlisted address (BOB) can deposit into a permissioned pool (e.g., KYC-gated or regulated) by nominating any known allowlisted address (ALICE) as `owner`. The extension checks `allowedDepositor[pool][ALICE]` → `true` and passes. BOB's tokens enter the pool, LP shares are credited to ALICE, and ALICE can call `removeLiquidity` to return proceeds to BOB off-chain. The pool admin's invariant — that only allowlisted addresses may supply liquidity — is broken. This constitutes a broken core pool functionality (access-control bypass) with direct fund-level consequences: unauthorized actors alter pool liquidity depth, bin state, and LP share dilution for legitimately allowlisted LPs. The inverse failure also exists: an allowlisted `sender` specifying a non-allowlisted `owner` is incorrectly blocked, causing a DoS for legitimate depositors.

## Likelihood Explanation
- No special privilege is required; any EOA or contract can call `addLiquidity` with an arbitrary `owner`.
- Finding a single allowlisted address is trivial — the pool admin is always publicly known and is typically allowlisted.
- The collusion step (owner returning funds off-chain) is a simple off-chain arrangement requiring no on-chain privilege.
- The bug is present in every pool that deploys `DepositAllowlistExtension` with a non-empty allowlist and `allowAllDepositors` set to `false`.

## Recommendation
Bind and check `sender` (the actual caller) instead of `owner`, mirroring the `SwapAllowlistExtension` pattern:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```
Setup:
  pool deployed with DepositAllowlistExtension
  allowedDepositor[pool][ALICE] = true   // ALICE is allowlisted
  BOB is NOT allowlisted

Attack:
  BOB calls pool.addLiquidity(
      owner    = ALICE,   // allowlisted → check passes
      salt     = 0,
      deltas   = <desired bins/shares>,
      callbackData = <BOB provides tokens via callback>
  )

Execution:
  1. addLiquidity calls _beforeAddLiquidity(msg.sender=BOB, owner=ALICE, ...)
  2. Extension checks allowedDepositor[pool][ALICE] → true → no revert
  3. Pool executes LiquidityLib.addLiquidity crediting position (ALICE, 0) with shares
  4. Pool calls BOB's callback → BOB transfers tokens to pool

Extraction:
  5. ALICE calls removeLiquidity → receives tokens → returns to BOB off-chain
  6. BOB has effectively deposited into a permissioned pool without being allowlisted

Repeated indefinitely with any known allowlisted address as owner.
```
