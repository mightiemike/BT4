### Title
Duplicate Bonus Reward Drain via Repeated Position IDs in `MsgClaimTierRewards` — (`x/tieredrewards/keeper/msg_server.go`)

---

### Summary

The `ClaimTierRewards` message handler in `x/tieredrewards` loads all requested positions from the store **upfront** into a slice before processing them. If the same position ID appears more than once in `msg.PositionIds`, each duplicate entry carries the same stale `LastBonusAccrual` checkpoint. The processing loop then pays out bonus rewards for each duplicate independently, draining the `RewardsPool` by N× the legitimate amount for a single position.

---

### Finding Description

In `msg_server.go`, `ClaimTierRewards` first loads every requested position into a local slice: [1](#0-0) 

All positions are fetched from the store before any reward processing begins. The slice is then passed to `claimRewardsAndUpdatesPositions`: [2](#0-1) 

Inside `claimRewardsAndUpdatesPositions`, each entry in the slice is processed independently: [3](#0-2) 

For each entry, `processEventsAndClaimBonus` is called. It reads `pos.LastBonusAccrual` from the **in-memory** copy, computes the bonus for the segment `[LastBonusAccrual, blockTime]`, transfers coins from `RewardsPool` to the owner, and then advances the checkpoint to `blockTime` via `applyBonusAccrualCheckpoint`: [4](#0-3) [5](#0-4) 

After the first duplicate is processed, `setPosition` writes the updated position (with `LastBonusAccrual = blockTime`) back to the store. However, the **second duplicate entry in the slice** was loaded before this write and still holds the old `LastBonusAccrual`. When `processEventsAndClaimBonus` runs on it, it recomputes the same bonus segment and issues a second payment from `RewardsPool`. This repeats for every additional duplicate.

Base rewards (`claimBaseRewards`) are not affected because `distributionKeeper.WithdrawDelegationRewards` is idempotent — after the first withdrawal there is nothing left to withdraw. The vulnerability is exclusive to the **bonus reward** path, which reads only the in-memory position state. [6](#0-5) 

---

### Impact Explanation

An attacker who owns a position with accrued bonus rewards submits a single `MsgClaimTierRewards` transaction with the same position ID repeated N times. The `RewardsPool` module account is drained by N× the legitimate bonus amount. With a large enough N (bounded only by the transaction gas limit), the attacker can empty the entire `RewardsPool` in one transaction, stealing bonus rewards that belong to all other participants.

The corrupted value is the `RewardsPool` balance and the `pos.LastBonusAccrual` checkpoint, which is written correctly only after the last duplicate is processed — but by then the funds are already gone.

---

### Likelihood Explanation

The entry path is a standard, unprivileged Cosmos SDK transaction (`MsgClaimTierRewards`) available to any position owner. No special role, governance action, or social engineering is required. The attacker only needs one position with any non-zero accrued bonus. The attack is executable in a single transaction and is trivially automatable.

---

### Recommendation

Deduplicate `msg.PositionIds` before processing, either in `MsgClaimTierRewards.Validate()` or at the start of the handler. A simple approach:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, id := range msg.PositionIds {
    if _, dup := seen[id]; dup {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest, "duplicate position id %d", id)
    }
    seen[id] = struct{}{}
}
```

Alternatively, re-fetch each position from the store inside `claimRewardsAndUpdatesPositions` rather than relying on the pre-loaded slice, so that the updated `LastBonusAccrual` is always used.

---

### Proof of Concept

1. Alice locks tokens and creates position ID `42` in a tier with a non-zero `BonusApy`.
2. Alice waits for bonus to accrue (any non-zero duration).
3. Alice submits `MsgClaimTierRewards{Owner: alice, PositionIds: [42, 42, 42, ..., 42]}` (N repetitions).
4. `ClaimTierRewards` loads N identical copies of position `42` (all with the same `LastBonusAccrual = T0`) into the `positions` slice.
5. The loop calls `processEventsAndClaimBonus` N times, each time computing `bonus = shares × tokensPerShare × BonusApy × (blockTime − T0) / SecondsPerYear` and transferring it from `RewardsPool` to Alice.
6. Alice receives N× the legitimate bonus. The `RewardsPool` is drained by N× the correct amount. [7](#0-6) [8](#0-7)

func applyBonusAccrualCheckpoint(pos *types.Position, blockTime time.Time) {
	accrualEnd := blockTime
	if pos.CompletedExitLockDuration(blockTime) {
		accrualEnd = pos.ExitUnlockAt
	}
	pos.UpdateLastBonusAccrual(accrualEnd)
}

func (k Keeper) computeSegmentBonus(pos types.PositionState, tier types.Tier, segmentStart, segmentEnd time.Time, tokensPerShare math.LegacyDec) math.Int {
	if !pos.ExitUnlockAt.IsZero() && segmentEnd.After(pos.ExitUnlockAt) {
		segmentEnd = pos.ExitUnlockAt
	}

	if !segmentEnd.After(segmentStart) {
		return math.ZeroInt()
	}

	durationSeconds := int64(segmentEnd.Sub(segmentStart) / time.Second)
	if durationSeconds <= 0 {
		return math.ZeroInt()
	}

	tokens := pos.Delegation.Shares.Mul(tokensPerShare)

	return tokens.
		Mul(tier.BonusApy).
		MulInt64(durationSeconds).
		QuoInt64(types.SecondsPerYear).
		TruncateInt()
}
