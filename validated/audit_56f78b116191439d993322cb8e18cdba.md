### Title
`ValidResourceBounds::AllResources` with zero `l2_gas`/`l1_data_gas` deserializes as `L1Gas` via protobuf, producing a divergent transaction hash — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic to reconstruct the enum variant. When an `AllResources` transaction carries zero `l2_gas` and zero `l1_data_gas` bounds, the deserializer silently reconstructs it as `L1Gas`. Because `get_tip_resource_bounds_hash` hashes a different number of resource-bound elements for the two variants, the transaction hash computed after a protobuf round-trip differs from the hash computed at submission time. Any node that receives such a transaction through the P2P sync layer will compute a wrong hash, breaking hash validation and causing state divergence.

### Finding Description

**Serialization side** (`crates/apollo_protobuf/src/converters/transaction.rs`, lines 471–489):

```rust
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),   // always ResourceBounds::default() → zeros
    l1_data_gas: Some(ResourceBounds::default().into()), // zeros
},
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(l2_gas.into()),
        l1_data_gas: Some(l1_data_gas.into()),
    },
```

Both `L1Gas` and `AllResources(l2_gas=0, l1_data_gas=0)` produce an identical protobuf wire encoding: `l2_gas = 0`, `l1_data_gas = 0`.

**Deserialization side** (lines 417–436):

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

Any protobuf message with both fields zero is unconditionally reconstructed as `L1Gas`, even when the original was `AllResources`.

**Hash divergence** (`crates/starknet_api/src/transaction_hash.rs`, lines 188–211):

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],                          // 2-element input
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3-element input
        }
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```

`L1Gas` feeds `poseidon([tip, l1_packed, l2_packed])` (3 elements).  
`AllResources` feeds `poseidon([tip, l1_packed, l2_packed, l1_data_packed])` (4 elements).

Even when `l2_packed` and `l1_data_packed` are both zero, the Poseidon hash over 3 elements ≠ the Poseidon hash over 4 elements. The transaction hash is therefore different before and after a protobuf round-trip.

The codebase itself acknowledges the ambiguity in the consensus test helper (`crates/apollo_protobuf/src/converters/consensus_test.rs`, lines 26–44):

```rust
// If all the fields of `AllResources` are 0 upon serialization,
// then the deserialized value will be interpreted as the `L1Gas` variant.
fn add_gas_values_to_transaction(transactions: &mut [ConsensusTransaction]) {
    ...
    resource_bounds.l2_gas.max_amount = GasAmount(1);  // workaround
}
```

The workaround is applied only in tests; no equivalent guard exists in the production P2P sync path.

### Impact Explanation

A V3 transaction with `AllResources(l1_gas = X, l2_gas = 0, l1_data_gas = 0)` is valid at the gateway (the `RpcTransaction` path uses `AllResourceBounds` directly and imposes no lower bound on `l2_gas` or `l1_data_gas`). Once included in a block, the transaction is serialized for P2P sync using the `ValidResourceBounds` protobuf converter. Every receiving node deserializes it as `L1Gas`, computes a hash over 3 Poseidon elements, and obtains a value that differs from the 4-element hash stored in the block. This causes `validate_transaction_hash` to return `false`, breaking block validation on syncing nodes and producing divergent state between the proposing node and all peers.

The impact matches: **High — Transaction conversion or signature/hash logic binds the wrong hash/type for an accepted payload**, and potentially **Critical — Wrong state from blockifier/execution logic for accepted input** if the hash mismatch propagates to committed storage.

### Likelihood Explanation

Any user can submit a V3 invoke, declare, or deploy-account transaction with `l2_gas.max_amount = 0`, `l2_gas.max_price_per_unit = 0`, `l1_data_gas.max_amount = 0`, `l1_data_gas.max_price_per_unit = 0`, and a non-zero `l1_gas` bound. The gateway accepts it (fee is enforced via `l1_gas`). The blockifier executes it. The block proposer includes it. Every syncing peer then fails to validate the block. No special privilege is required; a single such transaction is sufficient to trigger the divergence.

### Recommendation

Replace the value-based heuristic with an explicit discriminant field in the protobuf schema, or add a dedicated boolean/enum field (e.g., `is_all_resources`) to the `ResourceBounds` message so the variant can be reconstructed unambiguously regardless of the numeric values of `l2_gas` and `l1_data_gas`. Until the schema is updated, the deserializer should default to `AllResources` when `l1_data_gas` is present in the message (even if zero), reserving `L1Gas` only for messages where `l1_data_gas` is absent (`None`), which is the pre-0.13.3 wire format.

### Proof of Concept

1. Submit a V3 invoke transaction with:
   - `l1_gas = { max_amount: 1000, max_price_per_unit: 1 }`
   - `l2_gas = { max_amount: 0, max_price_per_unit: 0 }`
   - `l1_data_gas = { max_amount: 0, max_price_per_unit: 0 }`

2. The gateway converts it via `RpcInvokeTransactionV3 → InternalRpcInvokeTransactionV3` (using `AllResourceBounds` directly). The hash `H_orig` is computed by `get_invoke_transaction_v3_hash` over 4 Poseidon elements (including `l1_data_gas_packed`).

3. The transaction is included in a block. The block is serialized for P2P sync. `ValidResourceBounds::AllResources(l2_gas=0, l1_data_gas=0)` is encoded as `protobuf::ResourceBounds { l2_gas: 0, l1_data_gas: 0 }`.

4. A syncing peer deserializes the message. The condition `l1_data_gas.is_zero() && l2_gas.is_zero()` is true, so the peer reconstructs `ValidResourceBounds::L1Gas(l1_gas)`.

5. The peer calls `get_invoke_transaction_v3_hash` over 3 Poseidon elements (no `l1_data_gas_packed`), obtaining `H_peer ≠ H_orig`.

6. `validate_transaction_hash` returns `false`; the peer rejects the block. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
