### Title
Unprivileged Caller Can Prematurely Trigger Irreversible Mainchain GC, Permanently Breaking Transaction Verification — (File: `contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` is exposed as a public NEAR function with no caller access control beyond the pause gate. Any unprivileged NEAR account can call it at any time (when the contract is not paused) with an attacker-controlled `batch_size`. The function permanently and irreversibly deletes mainchain block headers from all three storage maps. Once deleted, those headers cannot be re-submitted. Any subsequent call to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a GC'd block will permanently panic, breaking downstream SPV consumers.

---

### Finding Description

`submit_blocks` — the only legitimate path for advancing the chain — is gated behind both `#[pause]` and `#[trusted_relayer]`, meaning only a staked, approved relayer can submit headers and trigger GC as a side-effect. [1](#0-0) 

`run_mainchain_gc`, however, carries only `#[pause(except(roles(Role::UnrestrictedRunGC)))]` — no caller identity check whatsoever: [2](#0-1) 

Any unprivileged NEAR account can call it directly. The `batch_size` argument is fully attacker-supplied. The function computes the maximum eligible removal and caps only by `batch_size`: [3](#0-2) 

Passing `batch_size = u64::MAX` removes every block currently eligible for pruning in a single transaction. Each removed block is erased from all three storage structures via `remove_block_header`: [4](#0-3) 

and its height entry is also dropped: [5](#0-4) 

After deletion, `verify_transaction_inclusion` panics at the `mainchain_header_to_height` lookup: [6](#0-5) 

and `verify_transaction_inclusion_v2` panics at the `headers_pool` lookup: [7](#0-6) 

Neither function can recover; the deleted headers are gone permanently.

The design intent documented in `contract/CLAUDE.md` is that GC is paced — one block removed per submitted header — so that the `gc_threshold` window is consumed gradually as the chain advances: [8](#0-7) 

The public exposure of `run_mainchain_gc` breaks this invariant: an attacker can drain the entire eligible window in one call, without submitting any new block.

---

### Impact Explanation

A downstream NEAR contract that calls `verify_transaction_inclusion_v2` to gate a cross-chain action (e.g., releasing locked tokens after a Bitcoin deposit) will receive a permanent panic for any block that has been prematurely GC'd. The verification result is not "false" — it is an unrecoverable contract panic. The downstream contract cannot retry with a different proof; the block header data is gone. The Bitcoin transaction is effectively unverifiable on-chain forever, and any value gated behind that proof is permanently inaccessible.

---

### Likelihood Explanation

The attack requires the chain to have grown at least one block beyond `gc_threshold` (so GC has eligible targets). At the recommended `gc_threshold = 52704` (~1 year of Bitcoin blocks), this condition is met continuously in production. The attacker needs no stake, no role, and no special knowledge — only the ability to send a NEAR transaction. The call is cheap (bounded gas, no deposit required). A griefing attacker can call `run_mainchain_gc(u64::MAX)` at any time to remove all currently eligible headers in one shot, racing any pending `verify_transaction_inclusion` calls from legitimate users or downstream contracts.

---

### Recommendation

Restrict `run_mainchain_gc` to privileged callers. The simplest fix is to add an access-control role check (e.g., require `Role::UnrestrictedRunGC` or `Role::DAO`) so that the function cannot be invoked by arbitrary accounts. The internal call from `submit_blocks` already bypasses the pause gate via the `except(roles(...))` mechanism and would continue to work. Alternatively, make `run_mainchain_gc` a private/internal function and remove its public NEAR method exposure entirely, since `submit_blocks` already calls it automatically.

---

### Proof of Concept

1. Deploy the contract with `gc_threshold = 100`.
2. Trusted relayer submits 101 blocks. The chain now has 101 headers stored; block at height 0 is eligible for GC.
3. A downstream contract is about to call `verify_transaction_inclusion_v2` for the block at height 0 with `confirmations = 100` (valid: `100 <= gc_threshold`).
4. **Attacker** (any unprivileged NEAR account) calls `run_mainchain_gc(u64::MAX)` — no deposit, no role required.
5. The GC removes the block at height 0 from `mainchain_height_to_header`, `mainchain_header_to_height`, and `headers_pool`.
6. The downstream contract's `verify_transaction_inclusion_v2` call now panics: `"cannot find requested transaction block"` (line 356) or `"block does not belong to the current main chain"` (line 301).
7. The verification is permanently broken. No re-submission of the header is possible because `init_genesis` only runs at initialization.
