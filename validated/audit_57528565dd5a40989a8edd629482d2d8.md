### Title
Order Expiration Check Rejects Orders With `expiration == getOracleTime()` - (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange._expired()` uses a `<=` comparison against `getOracleTime()`, causing any order whose `expiration` equals the current oracle time to be incorrectly treated as expired and rejected. This is the direct analog of the reported `FrontendManager` deadline bug: a strict boundary condition that silently invalidates orders that should be accepted.

---

### Finding Description

In `OffchainExchange.sol`, the `_expired` helper is:

```solidity
function _expired(uint64 expiration) internal view returns (bool) {
    return expiration <= getOracleTime();
}
``` [1](#0-0) 

`getOracleTime()` delegates to `IEndpoint(endpoint).getTime()`, which returns the current sequencer-reported oracle timestamp — the protocol's equivalent of `block.timestamp`. [2](#0-1) 

`_expired` is consumed inside `_validateOrder`, which returns `false` (invalid) when the order is expired:

```solidity
!_expired(order.expiration);
``` [3](#0-2) 

`matchOrders` then enforces validity with a hard revert:

```solidity
require(
    _validateOrder(..., taker, ..., txn.takerLinkedSigner),
    ERR_INVALID_TAKER
);
require(
    _validateOrder(..., maker, ..., txn.makerLinkedSigner),
    ERR_INVALID_MAKER
);
``` [4](#0-3) 

Because `_expired` returns `true` when `expiration == getOracleTime()`, any order signed with `expiration` set to the exact current oracle time is treated as already expired and the match reverts. The correct semantic — consistent with every major DeFi protocol — is that an order is valid *up to and including* its expiration timestamp, i.e., the check should be `expiration < getOracleTime()`.

---

### Impact Explanation

Any taker or maker order whose `expiration` field equals the current oracle time is silently rejected with `ERR_INVALID_TAKER` / `ERR_INVALID_MAKER`. This corrupts the order-matching flow: a legitimately signed, in-time order is treated as stale, causing the entire `matchOrders` call to revert. The sequencer must re-submit or drop the match, and the user's trade does not execute despite the order being within its stated validity window. For time-sensitive strategies (e.g., TWAP orders, liquidation-adjacent fills) this can cause missed fills at the intended price.

---

### Likelihood Explanation

Any trader or integrated system that sets `expiration` to the current oracle second — a standard and widely-used practice — will trigger this. The sequencer batches matches and submits them; if the oracle time advances to exactly equal a pending order's expiration between signing and sequencer submission, the match reverts. This is a realistic, unprivileged, user-controlled trigger requiring no special access.

---

### Recommendation

Change the comparison in `_expired` from `<=` to `<`:

```solidity
function _expired(uint64 expiration) internal view returns (bool) {
    return expiration < getOracleTime();
}
```

This makes an order valid when `expiration == getOracleTime()`, consistent with the standard DeFi convention that the deadline/expiration is inclusive of the current time.

---

### Proof of Concept

1. User signs an order with `expiration = T`, where `T` is the current oracle time at signing.
2. Sequencer calls `matchOrders` at oracle time `T` (same second).
3. `_expired(T)` evaluates `T <= T` → `true` → order is expired.
4. `_validateOrder` returns `false`.
5. `matchOrders` reverts with `ERR_INVALID_TAKER` or `ERR_INVALID_MAKER`.
6. The trade fails despite the order being within its stated validity window.
