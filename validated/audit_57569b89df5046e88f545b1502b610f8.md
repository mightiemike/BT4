Audit Report

## Title
`OracleValueStopLossExtension` uses arithmetic mid-price instead of geometric mid-price, causing stop-loss guard to under-trigger for `!zeroForOne` swaps — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid-price as an arithmetic mean `(bid + ask) / 2`, while the canonical protocol formula — used by `SwapMath.midAndSpreadFeeX64FromBidAsk` and `PriceVelocityGuardExtension` — uses the geometric mean `sqrt(bid * ask)`. By AM-GM, the arithmetic mean is always ≥ the geometric mean, inflating `midPriceX64` and therefore `metricToken1`. This causes the `!zeroForOne` stop-loss branch to fail to trigger when it should, allowing value-leaking swaps to drain LP token0 balances beyond the configured drawdown floor.

## Finding Description
`_afterSwapOracleStopLoss` computes the mid-price at line 218:

```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
``` [1](#0-0) 

This `midPriceX64` is passed directly to `_metrics`, which computes:

```solidity
metricT1 = _clampMetric(Math.mulDiv(Math.mulDiv(uint256(t0), midPriceX64, Q64), METRIC_SCALE, shares) + t1ps);
``` [2](#0-1) 

The canonical protocol formula in `SwapMath` uses the geometric mean:

```solidity
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);
``` [3](#0-2) 

`PriceVelocityGuardExtension` correctly calls this canonical formula:

```solidity
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
``` [4](#0-3) 

`OracleValueStopLossExtension` does not import `SwapMath` at all and uses the arithmetic mean instead. Since `metricToken1 ∝ t0 * midPriceX64`, an inflated mid inflates `metricToken1`, making the pool appear healthier than it is. The breach check for `!zeroForOne`:

```solidity
(uint256 hwm1, bool breach1) = _applyWatermark(metricT1, _decayed(hwmS.token1, decayRate, dt), floorMultiplier);
if (breach1 && !zeroForOne) {
    revert OracleStopLossTriggered(binIdx, false, metricT1, (hwm1 * floorMultiplier) / E6);
}
``` [5](#0-4) 

…never fires because the inflated `metricT1` stays above the floor even when the true geometric-mid metric has breached it. No existing guard compensates for this discrepancy.

## Impact Explanation
The `OracleValueStopLossExtension` is the sole on-chain mechanism protecting LP positions from sustained value leakage after adverse price moves. When `metricToken1` is inflated by the arithmetic-vs-geometric discrepancy, `!zeroForOne` swaps that should be blocked are permitted to continue, draining LP token0 balances beyond the configured `drawdownE6` floor. This constitutes a direct loss of LP principal. The magnitude scales with oracle spread width: a 50% spread (bid=0.75, ask=1.25) produces ~3.3% inflation, effectively relaxing a 5% drawdown floor to ~8.3% — nearly double the configured threshold. This meets the threshold for a medium-to-high direct loss of user principal.

## Likelihood Explanation
The trigger requires only a public `swap` call on a pool with `OracleValueStopLossExtension` configured with a non-zero `drawdownE6` and a non-trivial oracle spread. No privileged access is required. Any unprivileged trader can execute `!zeroForOne` swaps at a moment when the pool's true geometric-mid `metricToken1` has fallen to the configured floor, knowing the guard will not fire because it uses the inflated arithmetic mid. The wider the oracle spread, the larger the effective bypass. This is repeatable on any such pool.

## Recommendation
Replace the arithmetic mean in `_afterSwapOracleStopLoss` with the canonical geometric mean formula, matching `PriceVelocityGuardExtension` and the pool's own swap math. Import `SwapMath` and replace line 218:

```solidity
// Before (incorrect):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (correct — matches pool swap math):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(
    uint256(bidPriceX64),
    uint256(askPriceX64)
);
```

## Proof of Concept
```solidity
function test_stopLoss_arithmeticMidBypass() public {
    // 50% oracle spread: bid=0.75, ask=1.25 (geometric mid ≈ 0.9683)
    uint128 bid = uint128(3 * Q64 / 4);   // 0.75 in Q64
    uint128 ask = uint128(5 * Q64 / 4);   // 1.25 in Q64

    uint256 geoMid   = Math.sqrt(uint256(bid) * uint256(ask)); // ≈ 0.9683 * Q64
    uint256 arithMid = (uint256(bid) + uint256(ask)) / 2;      // = 1.0 * Q64

    // With drawdownE6 = 50_000 (5%), a pool whose true metricToken1
    // (at geometric mid) has fallen to 96% of the watermark should trigger.
    // The stop-loss sees it at 96% * (arithMid/geoMid) ≈ 99.2% and does NOT revert.
    // Execute !zeroForOne swaps; assert OracleStopLossTriggered is NOT emitted
    // even though the geometric-mid metric is below the 95% floor.
}
```

The discrepancy is confirmed by comparing `OracleValueStopLossExtension` line 218 (arithmetic mean) against `SwapMath.midAndSpreadFeeX64FromBidAsk` line 70 (geometric mean via `Math.sqrt`) and `PriceVelocityGuardExtension` line 48 (which correctly calls `SwapMath.midAndSpreadFeeX64FromBidAsk`).
