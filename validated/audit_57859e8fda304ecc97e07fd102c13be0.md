### Title
`burnNlp` Uses `MAINTENANCE` Health Check Instead of `INITIAL`, Allowing Collateral Extraction Below Initial Margin — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`burnNlp` enforces only a `MAINTENANCE` health check after execution, while the functionally equivalent `withdrawCollateral` and `mintNlp` both enforce the stricter `INITIAL` health check. This is the same bug class as the reference report: the wrong variant of a check is used in a parallel operation path, allowing a user to extract quote value from the protocol when they should be blocked.

---

### Finding Description

In `Clearinghouse.sol`, three operations affect a subaccount's collateral balance and each performs a post-operation health check:

**`withdrawCollateral`** (line 415–419) — uses `INITIAL`:
```solidity
IProductEngine.HealthType healthType = sender == X_ACCOUNT
    ? IProductEngine.HealthType.PNL
    : IProductEngine.HealthType.INITIAL;
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
```

**`mintNlp`** (line 479–482) — uses `INITIAL`:
```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
```

**`burnNlp`** (line 526–529) — uses `MAINTENANCE`:
```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
```

`burnNlp` removes NLP tokens from the sender and credits quote tokens in return — this is economically equivalent to a collateral withdrawal. Yet it only checks `MAINTENANCE` health, which is the weaker of the two checks. `INITIAL` health uses stricter weights (`longWeightInitialX18` / `shortWeightInitialX18`) than `MAINTENANCE` (`longWeightMaintenanceX18` / `shortWeightMaintenanceX18`), so a subaccount can be above `MAINTENANCE` health while below `INITIAL` health. ---

### Impact Explanation

A user whose subaccount is below `INITIAL` health (blocked from `withdrawCollateral`) but above `MAINTENANCE` health can call `burnNlp` to convert NLP tokens into quote tokens. Because NLP tokens and quote tokens carry different risk weights in the health calculation, this conversion can shift the health profile of the subaccount. Concretely:

- If NLP weight < quote weight (e.g., NLP at 0.9 initial weight vs quote at 1.0), burning NLP and receiving quote increases `INITIAL` health, potentially lifting the subaccount above the `INITIAL` threshold and enabling a subsequent `withdrawCollateral` call.
- Even without that secondary step, the user has extracted quote value from the protocol in a state where the protocol's risk model intended to block all collateral outflows.

This corrupts the protocol's solvency invariant: the `INITIAL` health gate exists to ensure a buffer above `MAINTENANCE` before any value leaves the system. `burnNlp` bypasses this gate entirely. [4](#0-3) 

---

### Likelihood Explanation

Any unprivileged user holding NLP tokens whose subaccount falls into the band `MAINTENANCE health > 0 > INITIAL health` can trigger this. This band is a normal operating state for leveraged accounts that have taken on risk since their last deposit. The entry path is a standard signed transaction (`BurnNlp`) routed through `EndpointTx`, requiring no special privileges. [5](#0-4) 

---

### Recommendation

Change the health check in `burnNlp` from `MAINTENANCE` to `INITIAL`, consistent with `withdrawCollateral` and `mintNlp`:

```solidity
// Before
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);

// After
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [6](#0-5) 

---

### Proof of Concept

1. User opens a leveraged perp short. Subaccount state: `INITIAL health = -50`, `MAINTENANCE health = +100`.
2. User holds 1000 USDC worth of NLP tokens (NLP `longWeightInitialX18 = 0.9`, quote weight = 1.0).
3. `withdrawCollateral` reverts: `INITIAL health < 0`.
4. User calls `burnNlp` for all NLP tokens. Receives ~999 USDC quote (after 0.1% burn fee).
5. `burnNlp` checks only `MAINTENANCE` health — passes.
6. Post-burn: NLP balance removed (−900 USDC initial health contribution), quote balance added (+999 USDC initial health contribution). Net `INITIAL` health delta ≈ +99. Subaccount is now above `INITIAL` health.
7. User calls `withdrawCollateral` — now succeeds, extracting quote tokens that the protocol's risk model should have locked. [7](#0-6) [8](#0-7)

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
```

    function _getWeightX18(
        Risk memory risk,
        int128 amount,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }

        int128 weight;
        if (amount >= 0) {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.longWeightInitialX18
                : risk.longWeightMaintenanceX18;
        } else {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.shortWeightInitialX18
                : risk.shortWeightMaintenanceX18;
        }

        return weight;
    }
