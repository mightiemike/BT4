### Title
Fork Block Difficulty Validation Uses Mainchain Ancestor Instead of Fork Ancestor, Enabling Difficulty Bypass on Fork Submissions — (`contract/src/dogecoin.rs`, `contract/src/litecoin.rs`, `contract/src/bitcoin.rs`)

---

### Summary

When validating proof-of-work difficulty for fork blocks, `get_next_work_required` fetches the difficulty-interval boundary block via `get_header_by_height`, which unconditionally reads from `mainchain_height_to_header`. For fork blocks whose chain diverged before the relevant boundary height, this returns the **mainchain** block at that height rather than the fork's actual ancestor. The resulting difficulty calculation is based on wrong state, allowing an attacker to submit fork blocks whose `bits` field satisfies the corrupted (mainchain-derived) difficulty check rather than the correct fork-lineage difficulty. A developer TODO comment in the code explicitly acknowledges this unresolved issue.

---

### Finding Description

**Vulnerability class (analog mapping):** Cross-module desynchronization / stale accumulated state used in a downstream calculation.

In the external report, `PSM.balanceOf(address(this))` returns the entire contract balance (including an unexpected refund) instead of the expected amount, inflating the value passed to `_addLiquidity()`. Here, `get_header_by_height` returns the entire mainchain's block at a given height instead of the fork's actual ancestor at that height, inflating (or deflating) the timestamp delta passed to `calculate_next_work_required`.

**Root cause — `get_header_by_height`:** [1](#0-0) 

This function exclusively reads `mainchain_height_to_header`, which only maps heights to mainchain blocks. It has no awareness of which chain lineage is being validated.

**Call site in Dogecoin (with explicit TODO acknowledgment):** [2](#0-1) 

**Call site in Litecoin:** [3](#0-2) 

**Call site in Bitcoin:** [4](#0-3) 

**Concrete trigger for Dogecoin (most severe):**

After block 145,000, Dogecoin adjusts difficulty every block (`difficulty_adjustment_interval = 1`): [5](#0-4) 

With `difficulty_adjustment_interval = 1`, `blocks_to_go_back = 1`, so `height_first = prev_block_header.block_height - 1`. For a fork that diverged at height F, the fork block at height F+3 has `prev_block_header` = fork block at F+2, and `height_first = F+1`. `get_header_by_height(F+1)` returns the **mainchain** block at F+1, but the fork's actual ancestor at F+1 is the fork block at F+1 — a different block with a potentially different timestamp. The attacker controls the fork block at F+1's timestamp (within the MTP and future-time bounds), so they can craft it to produce a more lenient difficulty for subsequent fork blocks.

For Bitcoin and Litecoin the discrepancy requires a fork longer than 2016 blocks, making it harder but not impossible in a long-running attack.

**Broken invariant:** The difficulty check for a fork block must use the fork's own ancestor at the boundary height, not the mainchain's block at that height. The contract enforces `expected_bits == block_header.bits` but computes `expected_bits` from wrong state. [6](#0-5) 

---

### Impact Explanation

An attacker who has submitted a fork diverging at height F can, for fork blocks at height ≥ F+3 (Dogecoin) or ≥ F+2017 (Bitcoin/Litecoin), pass a `bits` value that satisfies the mainchain-derived difficulty check rather than the correct fork-lineage difficulty. If the mainchain block at the boundary height has a timestamp that yields a lower difficulty (higher target) than the fork's ancestor would, the attacker can submit fork blocks with less work than the protocol requires. If the fork's cumulative `chain_work` then exceeds the mainchain tip's `chain_work`, `reorg_chain` is triggered, corrupting the canonical chain mapping stored in `mainchain_height_to_header` and `mainchain_tip_blockhash`. Downstream consumers of `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` would then verify proofs against a fraudulent chain. [7](#0-6) 

---

### Likelihood Explanation

For Dogecoin (post-145,000), the discrepancy activates for any fork block at depth ≥ 3 from the divergence point. The `submit_blocks` entry point is open to any trusted relayer (or DAO/`UnrestrictedSubmitBlocks` role holder), and the relayer role can be obtained through the staking mechanism. The attacker needs only to submit a short fork (3+ blocks) with a crafted timestamp in the first fork block to shift the difficulty calculation for subsequent fork blocks. The TODO comment confirms the developers identified but did not fix this path. [8](#0-7) 

---

### Recommendation

Replace `get_header_by_height` (mainchain lookup) with an ancestor traversal that walks the fork's own `prev_block_hash` chain back to the required boundary height. Concretely, starting from `prev_block_header`, follow `get_prev_header` links for `blocks_to_go_back` steps instead of calling `get_header_by_height`. This mirrors how Bitcoin Core's `GetAncestor` works and ensures the difficulty boundary block is always the correct lineage ancestor regardless of whether the block being validated is on the mainchain or a fork.

---

### Proof of Concept

1. Mainchain has blocks at heights 0…F+10 (Dogecoin, post-145,000). Mainchain block at F+1 has `time = T_main`.
2. Attacker submits fork block at F+1 (diverges from mainchain at F) with `time = T_fork` where `T_fork - T_prev` is large, causing `calculate_next_work_required` to produce a higher target (lower difficulty) for the next block.
3. Attacker submits fork block at F+2 (child of fork F+1).
4. Attacker submits fork block at F+3. `get_next_work_required` calls `get_header_by_height(F+1)`, which returns the **mainchain** block at F+1 with `time = T_main`, not the fork block with `time = T_fork`. The expected `bits` is computed from `T_main`, not `T_fork`.
5. Attacker sets `bits` in fork block F+3 to match the mainchain-derived expected bits. The check `expected_bits == block_header.bits` passes even though the correct fork-lineage difficulty would require different bits.
6. Attacker repeats, accumulating fork chainwork with under-required difficulty until `fork.chain_work > mainchain.chain_work`, triggering `reorg_chain` and corrupting the canonical chain. [9](#0-8) [10](#0-9)
