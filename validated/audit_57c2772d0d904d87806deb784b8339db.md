Audit Report

## Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees, Bypassing Factory Fee Cap System - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

## Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6` and `addFeeSellE6` directly to the pool with no upper-bound validation, while the analogous `setPoolAdminFees` enforces `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8` caps. A pool admin can set per-bin additional fees to `type(uint16).max` (65535 in E6 = 6.5535%) on any bin, bypassing the factory's cap system and extracting excess fees from swappers in that bin.

## Finding Description
`setPoolAdminFees` enforces caps before writing:

```solidity
// MetricOmmPoolFactory.sol lines 414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` has no equivalent check and forwards raw caller-supplied values straight to the pool:

```solidity
// MetricOmmPoolFactory.sol lines 450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool-side handler performs only a bin-index range check, writing fee values unconditionally:

```solidity
// MetricOmmPool.sol lines 464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

These per-bin fees are then added directly on top of the base spread fee in swap math:

```solidity
// MetricOmmPool.sol lines 540-541
uint256 buyFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [4](#0-3) 

The same pattern appears in the bin-walk swap loop: [5](#0-4) 

The factory's `maxAdminSpreadFeeE6` defaults to `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) and can be lowered by the factory owner, but this cap is entirely bypassed for the per-bin additional fee path. [6](#0-5) 

## Impact Explanation
Swappers whose trades route through the targeted bin pay the base spread fee **plus** the uncapped per-bin additional fee. At `uint16.max` the additional fee is 6.5535%, which can be applied to any bin at any time by the pool admin. The excess fee accrues to `adminFeeDestination`, constituting a direct loss of user principal. This is a confirmed admin-boundary break: the pool admin exceeds the cap system the factory is designed to enforce. The contest rules explicitly state: "If they can exceed caps or bypass timelocks and it leads to Medium or higher severity issue, then it can be valid."

## Likelihood Explanation
Requires a malicious or compromised pool admin. However, the inconsistency is structural: `setPoolAdminFees` is capped while `setPoolBinAdditionalFees` is not, so the cap system the factory exposes as a safety guarantee does not actually cover all fee-setting paths. The pool admin can call this at any time with no timelock.

## Recommendation
Add a cap check inside `setPoolBinAdditionalFees` before forwarding to the pool, mirroring the pattern used in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` cap that can be tuned independently.

## Proof of Concept
1. Pool is deployed with `spreadFeeE6 = 10_000` (1%) and `maxAdminSpreadFeeE6 = 200_000` (20%).
2. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
3. Factory forwards the call; pool writes `addFeeBuyE6 = 65535`, `addFeeSellE6 = 65535` for bin 0 with no revert (only bin-index range is checked).
4. A victim's swap routing through bin 0 now pays 1% (base) + 6.5535% (per-bin) = 7.5535% total spread fee.
5. The excess 6.5535% flows to `adminFeeDestination` — a direct loss of principal to the swapper that the factory's published cap system was supposed to prevent.

Foundry test plan: deploy factory + pool, `vm.prank(admin)` call `setPoolBinAdditionalFees(pool, 0, 65535, 65535)`, assert no revert, then perform a swap through bin 0 and assert the effective fee exceeds `maxAdminSpreadFeeE6`.
