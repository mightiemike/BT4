Audit Report

## Title
Unbounded `newTimelock` in `proposeOracleStopLossTimelock` Enables uint32 Wrap, Bypassing the Stop-Loss Timelock Guard - (File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol)

## Summary

`proposeOracleStopLossTimelock` accepts any `uint32 newTimelock` with no upper-bound check. Setting it to `type(uint32).max` causes `_afterTimelock` to silently truncate the uint256 sum to a timestamp one second in the past, making every subsequent proposal immediately executable. The pool admin can then atomically propose and execute `drawdownE6 = E6`, collapsing `floorMultiplier` to zero and permanently disabling the stop-loss guard for all LP positions in every monitored bin.

## Finding Description

`proposeOracleStopLossTimelock` stores any `uint32 newTimelock` without validation:

```solidity
// L78-84
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);   // uses CURRENT timelock, not newTimelock
    sched.pendingTimelock = newTimelock;
    sched.pendingTimelockExecuteAfter = executeAfter;
    ...
}
```

`_afterTimelock` adds the stored `uint32` timelock to `block.timestamp` in `uint256`, then truncates to `uint32`:

```solidity
// L297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

With `block.timestamp ≈ 1_753_000_000` (July 2026) and `timelock = type(uint32).max`:

```
uint32(1_753_000_000 + 4_294_967_295)
= uint32(6_047_967_295)
= 6_047_967_295 % 4_294_967_296
= 1_752_999_999   ← one second in the past
```

`_requireElapsed` then passes immediately for every subsequent proposal:

```solidity
// L301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert ...;
    // block.timestamp (1_753_000_000) < 1_752_999_999 → false → passes
}
```

By contrast, `_validateDrawdown` and `_validateDecay` both enforce hard ceilings; no `_validateTimelock` exists. The same absence of validation exists in `initialize` at L56-58, where `timelock` is decoded and stored without any check.

**Exploit path:**

1. Pool is initialized with `timelock = 0` (permitted — no validation in `initialize`; confirmed in test setUp at L49).
2. Admin calls `proposeOracleStopLossTimelock(pool, type(uint32).max)`. Because the current timelock is 0, `executeAfter = uint32(block.timestamp + 0) = block.timestamp`. `_requireElapsed(block.timestamp)` evaluates `block.timestamp < block.timestamp` → false → passes immediately.
3. Admin calls `executeOracleStopLossTimelock(pool)`. `oracleStopLossConfig[pool_].timelock` is now `type(uint32).max`.
4. Admin calls `proposeOracleStopLossDrawdown(pool, 1e6)`. `_validateDrawdown(1e6)` passes (`1e6 > 1e6` is false). `_afterTimelock` wraps to a past timestamp. `executeAfter` is in the past.
5. Admin calls `executeOracleStopLossDrawdown(pool)`. `_requireElapsed(past_timestamp)` passes immediately. `oracleStopLossConfig[pool_].drawdownE6 = 1e6`.
6. In `_afterSwapOracleStopLoss`, `floorMultiplier = E6 - drawdown = 1e6 - 1e6 = 0`.
7. In `_applyWatermark`: `breached = metric < (hwm * 0) / E6` → always false. Stop-loss is permanently silenced.

## Impact Explanation

Once `floorMultiplier = 0`, `_applyWatermark` can never set `breached = true`, so `afterSwap` never reverts regardless of how far the per-share bin value drops. All LP principal in every bin the extension monitors is exposed to unchecked value extraction through swaps. This is a direct admin-boundary break: the pool admin has bypassed the only on-chain constraint (the timelock) that bounds their power over LP-adverse parameter changes, matching the allowed impact gate item "Admin-boundary break: pool admin exceeds caps, bypasses timelocks."

## Likelihood Explanation

The trigger requires the pool admin, a semi-trusted role whose power is explicitly bounded by the timelock. A `timelock = 0` initial configuration is valid (no code prevents it, and the test harness uses it by default). With an initial timelock of 0, the entire attack executes in a single block with no prior waiting. Even with a non-zero initial timelock, the admin can wait out that period, set `timelock = type(uint32).max`, and then bypass all future timelocks permanently. The attack is deterministic, requires no external conditions, and is repeatable.

## Recommendation

Add a `_validateTimelock` function mirroring the existing validators and call it in both `initialize` and `proposeOracleStopLossTimelock`:

```solidity
uint32 private constant MAX_TIMELOCK = 365 days;

function _validateTimelock(uint256 timelock) private pure {
    if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);
}
```

Apply it in `initialize` (after L58):

```solidity
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
_validateTimelock(timelock);   // add
```

And at the top of `proposeOracleStopLossTimelock` (before L80):

```solidity
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    _validateTimelock(newTimelock);   // add
    ...
}
```

The ceiling must be chosen so that `block.timestamp + MAX_TIMELOCK` never overflows `uint32` for any realistic timestamp (365 days ≈ 31,536,000 seconds; `1_753_000_000 + 31_536_000 = 1_784_536_000`, well within uint32 range).
