### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Causing Consensus Peers to Reject Valid V3 Transactions - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf-to-Rust conversion for `ValidResourceBounds` silently maps any `AllResources` transaction whose `l2_gas` and `l1_data_gas` are both zero to the `L1Gas` variant. A subsequent conversion step in the consensus path then hard-rejects any `L1Gas` variant as a deprecated resource-bounds error. A user can submit a valid V3 invoke transaction with only L1 gas bounds (l2_gas = 0, l1_data_gas = 0) through the gateway, which accepts it, but every consensus peer will fail to deserialize it from protobuf and reject the block proposal that contains it.

### Finding Description

**Step 1 – Downgrade in protobuf deserialization**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` applies the following heuristic:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← forced downgrade
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

This is called from `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3`: [2](#0-1) 

**Step 2 – Hard rejection in the consensus path**

The consensus protobuf path converts `protobuf::InvokeV3WithProof` → `InvokeTransactionV3` (via the above) → `RpcInvokeTransactionV3`. The final step is `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3`, which hard-rejects any `L1Gas` variant:

```rust
resource_bounds: match value.resource_bounds {
    ValidResourceBounds::AllResources(bounds) => bounds,
    _ => {
        return Err(StarknetApiError::OutOfRange { string: "resource_bounds".to_string() });
    }
},
``` [3](#0-2) 

The consensus transaction deserialization calls this chain: [4](#0-3) 

And the consensus transaction dispatcher: [5](#0-4) 

**Step 3 – Gateway accepts the transaction**

`RpcInvokeTransactionV3` always carries `AllResourceBounds` (never `ValidResourceBounds`), so the gateway never applies the downgrade heuristic. A transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` passes stateless validation (the test suite explicitly marks `valid_l1_gas` as a passing case) and is accepted into the mempool. [6](#0-5) 

**Step 4 – Hash domain divergence (secondary)**

`get_tip_resource_bounds_hash` produces structurally different Poseidon inputs for `L1Gas` (2 resource felts) vs `AllResources` (3 resource felts), even when l2_gas and l1_data_gas are zero. Any node that recomputes the hash after the downgrade will produce a hash that does not match the one computed at submission time. [7](#0-6) 

### Impact Explanation

A proposer that includes a valid V3 invoke transaction with `l2_gas = 0` and `l1_data_gas = 0` in a block proposal will have that proposal rejected by every validator peer. The validators fail at `TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` with `DEPRECATED_RESOURCE_BOUNDS_ERROR`, causing them to treat the proposal as malformed. This maps to:

- **High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload** — the conversion produces the wrong `ValidResourceBounds` variant, causing the transaction to be treated as a deprecated pre-0.13.3 transaction.
- **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing** — the gateway accepts the transaction while the consensus layer rejects it.

### Likelihood Explanation

The trigger is a standard V3 invoke transaction where the user sets only `l1_gas` bounds and leaves `l2_gas` and `l1_data_gas` at zero. The gateway's stateless validator explicitly permits this (test case `valid_l1_gas`). Any user or automated tool that submits such a transaction can trigger the failure. No privileged access is required.

### Recommendation

Remove the value-based heuristic from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. The `L1Gas` vs `AllResources` distinction is a protocol-version property of the transaction, not a property of the resource-bound values. The serializer already encodes `L1Gas` with an explicit zero `l1_data_gas` field:

```rust
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),
    l1_data_gas: Some(ResourceBounds::default().into()),  // explicit zero
},
``` [8](#0-7) 

The deserializer should use an explicit discriminant field (e.g., a boolean `is_all_resources` flag in the protobuf schema) rather than inferring the variant from whether the values happen to be zero. Until the schema is updated, the deserializer should default to `AllResources` when all three fields are present, reserving `L1Gas` only for messages that are missing the `l1_data_gas` field entirely (the pre-0.13.2 wire format).

### Proof of Concept

1. Submit a V3 invoke transaction to the gateway with `resource_bounds = AllResourceBounds { l1_gas: { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: { max_amount: 0, max_price_per_unit: 0 }, l1_data_gas: { max_amount: 0, max_price_per_unit: 0 } }`.
2. The gateway accepts it; `InternalRpcInvokeTransactionV3.resource_bounds` stores `AllResourceBounds` with zero l2/l1_data fields.
3. The batcher includes it in a block proposal and serializes it via `From<ValidResourceBounds> for protobuf::ResourceBounds` → all three fields are present and zero for l2/l1_data.
4. A validator peer calls `TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction` → `TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` → `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` → `ValidResourceBounds::try_from(...)` returns `ValidResourceBounds::L1Gas(...)` because `l1_data_gas.is_zero() && l2_gas.is_zero()`.
5. The next step `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` returns `Err(DEPRECATED_RESOURCE_BOUNDS_ERROR)`.
6. The validator rejects the block proposal.

impl TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3 {
    type Error = ProtobufConversionError;
    fn try_from(mut value: protobuf::InvokeV3WithProof) -> Result<Self, Self::Error> {
        // Extract proof first, since `starknet_api::transaction::InvokeTransactionV3` does not
        // carry a `proof` field.
        let proof = Proof::from(std::mem::take(&mut value.proof));

        let snapi_invoke: InvokeTransactionV3 = value
            .invoke
            .ok_or(ProtobufConversionError::MissingField {
                field_description: "InvokeV3WithProof::invoke",
            })?
            .try_into()?;

        // This conversion can fail only if the resource_bounds are not AllResources.
        Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
    }
