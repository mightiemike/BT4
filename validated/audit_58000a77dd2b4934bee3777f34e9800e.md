### Title
Linked Signer Can Overwrite Its Own Authorization — Unauthorized `linkedSigners` Mutation via `LinkSigner` Fast Path - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` fast-path handler calls `validateSignedTx` with `allowLinkedSigner = true`. This permits the currently-registered linked signer — a delegated, non-owner entity — to sign and submit a `LinkSigner` transaction that overwrites `linkedSigners[subaccount]` with any arbitrary address. Only the subaccount owner (the wallet address embedded in the `bytes32` subaccount key) should be permitted to change who the linked signer is.

---

### Finding Description

The `linkedSigners` mapping in `EndpointStorage` records, for each subaccount, a single delegated address that may sign sequencer-submitted transactions on the subaccount owner's behalf. [1](#0-0) 

In the **slow-mode path** (`processSlowModeTransactionImpl`), `LinkSigner` is correctly restricted to the subaccount owner via `validateSender`, which checks that `msg.sender == address(uint160(bytes20(txn.sender)))`: [2](#0-1) 

In the **fast path** (`processTransactionImpl`), however, the same `LinkSigner` transaction type is handled with `allowLinkedSigner = true` passed to `validateSignedTx`: [3](#0-2) 

`validateSignedTx` with `allowLinkedSigner = true` resolves the permitted signer as either the subaccount owner **or** the currently-registered linked signer: [4](#0-3) 

Because the linked signer's signature satisfies the check, it can craft and submit (via the sequencer) a `SignedLinkSigner` transaction that sets `linkedSigners[subaccount]` to any address it chooses — including a fresh attacker-controlled address.

The `LinkSigner` struct confirms the fields involved: `sender` (the subaccount), `signer` (the new linked signer address), and `nonce`: [5](#0-4) 

---

### Impact Explanation

A compromised or malicious linked signer can:

1. **Permanently hijack the subaccount** — replace itself with an attacker-controlled address, granting that address full signing authority over the subaccount (withdrawals, transfers, order placement, NLP mint/burn).
2. **Prevent owner recovery** — because the attacker-controlled linked signer can keep re-issuing `LinkSigner` transactions faster than the owner can submit a slow-mode revocation, the owner may be unable to reclaim control.
3. **Drain collateral** — the new linked signer can immediately sign `WithdrawCollateral` or `TransferQuote` transactions, which also accept `allowLinkedSigner = true`, to move all collateral out of the subaccount.

The corrupted state is `linkedSigners[subaccount]`, which directly controls who may authorize every sensitive sequencer-submitted operation for that subaccount.

---

### Likelihood Explanation

Any party that has ever been granted linked-signer access — a trading bot, an API key, a third-party integration — can exploit this. The attack requires only that the linked signer construct a valid `SignedLinkSigner` payload and have it included in a sequencer batch. No admin access, governance capture, or leaked owner key is required. The linked signer already has the signing key needed to produce the required signature.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` handler in `processTransactionImpl`:

```solidity
// EndpointTx.sol — processTransactionImpl, LinkSigner branch
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // ← was true; only the subaccount owner may change the linked signer
);
```

This aligns the fast path with the slow-mode path, which already enforces that only the subaccount owner (`msg.sender == address(uint160(bytes20(txn.sender)))`) may issue a `LinkSigner` transaction. [6](#0-5) 

---

### Proof of Concept

1. Alice owns subaccount `alice_subaccount` and has registered `linkedSigner = bot_address`.
2. `bot_address` (the linked signer) constructs a `SignedLinkSigner` payload:
   - `tx.sender = alice_subaccount`
   - `tx.signer = attacker_address` (bytes32-encoded)
   - `tx.nonce = current_nonce`
   - `signature` = ECDSA signature over the EIP-712 digest, signed by `bot_address`
3. The sequencer includes this transaction in a batch submitted to `Endpoint.submitTransactionsChecked`.
4. `processTransactionImpl` reaches the `LinkSigner` branch and calls `validateSignedTx(..., true)`.
5. `validateSignature` resolves the allowed signer as `getLinkedSigner(alice_subaccount)` = `bot_address`. The signature is valid.
6. `linkedSigners[alice_subaccount]` is overwritten with `attacker_address`.
7. `attacker_address` now signs a `WithdrawCollateral` transaction for `alice_subaccount`, draining all collateral. [3](#0-2) [7](#0-6)
