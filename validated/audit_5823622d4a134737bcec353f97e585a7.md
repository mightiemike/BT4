### Title
`deser_auto` Silently Ignores `max_atom_len` and `strict` Parameters for Legacy-Format Blobs — (`File: wheel/src/api.rs`)

---

### Summary

`deser_auto` accepts caller-supplied `max_atom_len` and `strict` guard parameters but silently drops both when the input blob is in legacy/backrefs format. An attacker who controls the serialized bytes can bypass the intended atom-size cap by presenting a legacy-format blob, causing unbounded atom allocation regardless of the limit the caller specified.

---

### Finding Description

`deser_auto` is the Python-facing auto-detecting deserializer. Its signature promises two safety knobs:

```rust
// wheel/src/api.rs  lines 147-157
#[pyfunction]
#[pyo3(signature = (blob, *, max_atom_len=PY_DEFAULT_MAX_ATOM_LEN, strict=true))]
fn deser_auto(blob: &[u8], max_atom_len: usize, strict: bool) -> PyResult<LazyNode> {
    let mut a = Allocator::new();
    let node = if let Some(body) = blob.strip_prefix(SERDE_2026_MAGIC_PREFIX.as_slice()) {
        deserialize_2026_body_from_stream(&mut a, &mut Cursor::new(body), max_atom_len, strict)
            .map_err(eval_to_py)?
    } else {
        node_from_bytes_backrefs(&mut a, blob).map_err(eval_to_py)?   // ← max_atom_len and strict are NOT forwarded
    };
    Ok(LazyNode::new(Rc::new(a), node))
}
``` [1](#0-0) 

When the blob starts with `SERDE_2026_MAGIC_PREFIX`, both `max_atom_len` and `strict` are forwarded to `deserialize_2026_body_from_stream`. When the blob is in any other format (classic or backrefs), `node_from_bytes_backrefs` is called with **no size cap and no strictness check**. The two parameters accepted by the function signature become completely inert for the non-2026 path.

The analog to the external report is direct:

| External report | clvm_rs |
|---|---|
| `depositRewards(uint256 _amount)` accepts `_amount` but spends `msg.value` | `deser_auto(blob, max_atom_len, strict)` accepts `max_atom_len`/`strict` but ignores them for legacy blobs |
| Caller believes `_amount` controls what is spent | Caller believes `max_atom_len` caps atom size |
| Parameter is decorative; actual behavior is unconstrained | Parameter is decorative for non-2026 blobs; actual behavior is unconstrained | [2](#0-1) 

---

### Impact Explanation

Any Python caller that invokes `deser_auto(untrusted_blob, max_atom_len=N)` to protect against oversized atoms receives **no protection** when the attacker presents a legacy-format blob. The `Allocator` has no per-atom byte limit of its own; it only caps the total number of atoms (`MAX_NUM_ATOMS = 62_500_000`). A single atom of, say, 500 MB is perfectly legal under the allocator's rules and will be allocated in full before any limit is checked. [3](#0-2) [4](#0-3) 

Downstream wrappers (e.g. `chia_rs`) that supply their own caps via `max_atom_len` and rely on `deser_auto` for format-agnostic parsing are silently unprotected against legacy-format blobs. The result is unbounded heap growth, leading to OOM termination of the Python process or node.

---

### Likelihood Explanation

The attacker-controlled entry path is straightforward: supply a legacy-format CLVM blob (no magic prefix required) containing a single atom whose encoded length exceeds `max_atom_len`. The blob is valid CLVM serialization, requires no special privileges, and is accepted by `deser_auto` without any format restriction. Any API surface that calls `deser_auto` on externally-supplied bytes is reachable.

---

### Recommendation

Forward `max_atom_len` and `strict` to the legacy path as well. Since `node_from_bytes_backrefs` does not currently accept these parameters, either:

1. Add `max_atom_len` (and optionally `strict`) parameters to `node_from_bytes_backrefs` / `node_from_bytes`, or
2. Add a post-parse atom-size walk before returning the `LazyNode`, or
3. Document explicitly that `max_atom_len` and `strict` are **only** enforced for serde_2026 blobs, and rename the parameters or add a warning so callers are not misled.

Option 1 is the correct fix. The parameter contract implied by the function signature must be honoured uniformly across all format branches.

---

### Proof of Concept

```python
import clvm_rs

# Craft a legacy-format atom of 2 MB (well above the 1-byte cap below).
# Legacy encoding: 0x8E followed by 0x00 0x20 0x00 00 (length = 2097152), then 2MB of zeros.
# Simpler: use the Python clvm library to serialise a large atom, or hand-craft:
atom_len = 2 * 1024 * 1024          # 2 MB
# Legacy header for atom of length > 0x3FFFF uses 3-byte length prefix: 0xFE <hi> <lo>
# For simplicity, build via node_to_bytes on a known-good path, then pass to deser_auto.

# Caller intends to cap atoms at 1 byte:
node = clvm_rs.deser_auto(
    b"\x8e" + b"\x00" * (2 * 1024 * 1024),   # legacy-format 2 MB atom (approximate)
    max_atom_len=1,   # caller expects this to be enforced
    strict=True,
)
# No error is raised; a 2 MB atom is returned, bypassing max_atom_len=1.
```

The `max_atom_len=1` guard is silently ignored because the blob lacks the `SERDE_2026_MAGIC_PREFIX`, causing `node_from_bytes_backrefs` to be called without any size constraint. [5](#0-4)
