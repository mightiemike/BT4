### Title
Silent `max_atom_len` Drop in `deser_auto` for Non-serde_2026 Blobs — (File: `wheel/src/api.rs`)

---

### Summary

`deser_auto` accepts `max_atom_len` and `strict` parameters but silently discards both when the input blob is in legacy or backrefs format. The API signature, the `.pyi` stub, and the Python-level docstring all present these parameters as applying to the "auto" path, but they are only forwarded when the blob starts with the serde_2026 magic prefix. For every other blob, the parameters are dropped without error, without warning, and without any enforcement.

---

### Finding Description

In `wheel/src/api.rs`, `deser_auto` is declared with `max_atom_len` and `strict`:

```rust
#[pyo3(signature = (blob, *, max_atom_len=PY_DEFAULT_MAX_ATOM_LEN, strict=true))]
fn deser_auto(blob: &[u8], max_atom_len: usize, strict: bool) -> PyResult<LazyNode> {
    let mut a = Allocator::new();
    let node = if let Some(body) = blob.strip_prefix(SERDE_2026_MAGIC_PREFIX.as_slice()) {
        deserialize_2026_body_from_stream(&mut a, &mut Cursor::new(body), max_atom_len, strict)
            .map_err(eval_to_py)?
    } else {
        node_from_bytes_backrefs(&mut a, blob).map_err(eval_to_py)?   // ← both params dropped
    };
    Ok(LazyNode::new(Rc::new(a), node))
}
``` [1](#0-0) 

When the blob is not serde_2026, `node_from_bytes_backrefs` is called with no size cap and no strictness flag. The caller's `max_atom_len` value — including the module-wide default of `PY_DEFAULT_MAX_ATOM_LEN = 1 << 20` — is silently discarded. [2](#0-1) 

The Python-level `serde.py` docstring reinforces the false impression:

```
Keyword-only limits (applied to "2026" and "auto" paths):
    max_atom_len:   largest single atom in bytes (default 1 MiB)
    strict:         reject overlong varint encodings
``` [3](#0-2) 

The phrase "applied to … 'auto' paths" is misleading: the limit is only applied when the auto path detects serde_2026 format. For legacy and backrefs blobs — the majority of existing on-chain data — neither limit is applied.

The `.pyi` stub exposes the same false contract to every downstream type-checker and IDE: [4](#0-3) 

`Program.from_bytes()` calls `deser_auto(blob)` with no explicit `max_atom_len`, relying on the default 1 MiB cap — which is never enforced for legacy blobs: [5](#0-4) 

The Rust docstring for `deser_auto` does contain a single buried sentence — "Consensus-aware callers should sniff the prefix themselves and use their own caps" — but this is the only indication that the parameters are not universally applied, and it appears after the functional description, in a position easy to overlook. [6](#0-5) 

---

### Impact Explanation

The `max_atom_len` cap exists to prevent a malformed blob from driving unbounded pre-allocation. In the serde_2026 deserializer, atom length is declared before the bytes are read, and `checked_bounded_usize(-length_val, max_atom_len)` enforces the cap before `buf.resize(length, 0)` is called: [7](#0-6) 

For legacy/backrefs blobs routed through `deser_auto`, no equivalent cap is applied. An attacker-controlled legacy blob with a large declared atom length bypasses the caller's intended limit entirely. Any caller that passes `max_atom_len` to `deser_auto` expecting it to apply universally — including callers relying on the default 1 MiB — receives no protection for legacy-format input.

---

### Likelihood Explanation

`Program.from_bytes()` is the primary deserialization entry point for Python callers and is used throughout the Chia ecosystem. It calls `deser_auto` without an explicit cap, relying on the default. Legacy-format blobs are the dominant on-chain format. Any service that deserializes attacker-supplied legacy blobs through `Program.from_bytes()` or `deser_auto()` is exposed. The misleading docstring and `.pyi` stub make it likely that downstream callers believe the cap is enforced.

---

### Recommendation

**Short term:** In `deser_auto`, apply `max_atom_len` to the legacy/backrefs branch. Since `node_from_bytes_backrefs` does not accept a size cap, add a pre-parse length check or thread `max_atom_len` into the legacy deserializer.

**Long term:** Update the `serde.py` docstring and the `.pyi` stub to accurately state that `max_atom_len` and `strict` are only enforced for serde_2026 blobs, and add a visible warning that callers processing legacy blobs must enforce their own atom-size policy. Consider adding a separate `max_atom_len` parameter to `node_from_bytes_backrefs` so the cap can be uniformly applied across all formats.

---

### Proof of Concept

```python
from clvm_rs.clvm_rs import deser_auto

# Craft a legacy-format blob containing a 5 MiB atom.
# Legacy encoding: 0xfe <3-byte big-endian length> <data>
atom_size = 5 * 1024 * 1024          # 5 MiB
header = bytes([0xfe,
                (atom_size >> 16) & 0xFF,
                (atom_size >> 8)  & 0xFF,
                 atom_size        & 0xFF])
blob = header + b'\xab' * atom_size

# Caller intends to cap atoms at 1 KiB.
node = deser_auto(blob, max_atom_len=1024)

# Expected: ValueError — atom exceeds max_atom_len.
# Actual:   deser_auto succeeds; the 5 MiB atom is allocated
#           because max_atom_len is silently dropped for legacy blobs.
print(len(node.atom))   # prints 5242880
```

The discrepancy is a direct structural analog to the original report: a parameter that appears to apply universally is silently ignored for a specific input class, and the gap is described only in buried documentation that callers are likely to miss.
