### Title
Rebasing Token Balance Desynchronization Causes Stuck Funds in Clearinghouse — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

Nado's internal accounting in `SpotEngine` tracks deposited token amounts based on values recorded at deposit time. The solvency guard `assertUtilization` only validates internal accounting invariants and never reconciles against actual on-chain token balances. If a rebasing or balance-modifying ERC20 token is listed as a collateral product, an external rebase event can silently desynchronize internal accounting from actual holdings, causing withdrawals to revert and funds to become permanently stuck in the `Clearinghouse`.

---

### Finding Description

The deposit flow in `Endpoint.depositCollateralWithReferral` performs two independent operations:

1. `handleDepositTransfer` immediately transfers `amount` tokens from the caller to the `Clearinghouse`.
2. The same `amount` value is encoded into a `SlowModeTx` queued for sequencer processing (up to `SLOW_MODE_TX_DELAY` = 3 days later). [1](#0-0) 

When the sequencer eventually processes the slow-mode transaction, `Clearinghouse.depositCollateral` credits the user with `amountRealized = txn.amount * multiplier`, derived entirely from the cached `amount` value — not from a live `balanceOf` check. [2](#0-1) 

The only solvency guard called on withdrawal is `SpotEngine.assertUtilization`, which checks `totalDepositsNormalized * cumulativeMultiplier >= totalBorrowsNormalized * cumulativeMultiplier`. This is a purely internal accounting invariant. [3](#0-2) 

It never queries `token.balanceOf(address(clearinghouse))`. If a rebasing token reduces the actual balance held by the `Clearinghouse` between deposit and withdrawal, the internal accounting still reflects the original deposited amount. The invariant passes, but the actual `safeTransfer` in `handleWithdrawTransfer` will revert because the contract holds fewer tokens than the internal ledger claims. [4](#0-3) 

The same desynchronization applies in the upward-rebase direction: excess tokens accrue in the `Clearinghouse` with no on-chain rescue path for them.

---

### Impact Explanation

When a negative rebase reduces the `Clearinghouse`'s actual token balance below the sum of internal user balances, any user attempting to withdraw will trigger `token.safeTransfer(withdrawPool, amount)` with an `amount` that exceeds the contract's real holdings. This call reverts, permanently locking the affected users' funds. The last users to withdraw bear the full loss. Additionally, a positive rebase leaves excess tokens permanently stranded in the `Clearinghouse` with no recovery mechanism.

---

### Likelihood Explanation

Medium. The trigger requires a rebasing or balance-modifying ERC20 token to be configured as a collateral product via `SpotEngine.addOrUpdateProduct`. The protocol does not restrict or document exclusion of such tokens. The `IERC20Base` interface is a minimal ERC20 interface with no rebasing guard. [5](#0-4) 

The 3-day `SLOW_MODE_TX_DELAY` window between token receipt and credit processing creates an additional exposure window during which a rebase can occur before the user's balance is even credited. [6](#0-5) 

---

### Recommendation

1. **Document and enforce token restrictions**: Explicitly prohibit rebasing, fee-on-transfer, and other balance-modifying tokens from being listed as collateral products. Add a check or comment in `SpotEngine.addOrUpdateProduct`.
2. **Balance reconciliation on withdrawal**: Optionally, `assertUtilization` could be extended to compare `totalDeposits` against `token.balanceOf(clearinghouse)` for the relevant product, rejecting withdrawals that would overdraw actual holdings.
3. **Rescue function**: Add an owner-callable function to recover excess tokens that accumulate in the `Clearinghouse` due to positive rebases or direct transfers.

---

### Proof of Concept

1. Owner lists AMPL (Ampleforth rebasing token) as a collateral product via `SpotEngine.addOrUpdateProduct`.
2. Alice calls `Endpoint.depositCollateralWithReferral` with `amount = 1000e18`. `handleDepositTransfer` transfers 1000 AMPL to the `Clearinghouse`. A `SlowModeTx` is queued encoding `amount = 1000e18`.
3. A negative AMPL rebase occurs. The `Clearinghouse` now holds 800 AMPL.
4. The sequencer processes the slow-mode tx. `Clearinghouse.depositCollateral` credits Alice with `amountRealized = 1000e18 * multiplier` via `spotEngine.updateBalance`. `assertUtilization` passes (internal accounting is self-consistent).
5. Alice calls `withdrawCollateral` for 1000 AMPL. `handleWithdrawTransfer` calls `token.safeTransfer(withdrawPool, 1000e18)`. The `Clearinghouse` only holds 800 AMPL — the call reverts. Alice's 1000 AMPL credit is permanently stuck. [7](#0-6)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
