Audit Report

## Title
SwapAllowlistExtension checks router address instead of actual user, enabling allowlist bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the value passed by the pool — which is `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. The extension therefore checks whether the router is allowlisted, not the real trader. If the pool admin allowlists the router (the only way to enable router-mediated swaps on a curated pool), every unprivileged user can bypass the allowlist by routing through the public router.

## Finding Description

**Call path:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle()`.
2. Router calls `IMetricOmmPoolActions(pool).swap(recipient, ...)` — at this point `msg.sender` inside the pool is the **router address**.
3. Pool calls `_beforeSwap(msg.sender, ...)` — passing the router address as `sender`.
4. `ExtensionCalling._beforeSwap` encodes and dispatches to `SwapAllowlistExtension.beforeSwap(sender=router, ...)`.
5. The extension evaluates:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct) and `sender` is the **router** (wrong — should be the original user).

**Root cause:** The pool passes its own `msg.sender` as the `sender` argument to the extension:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
  msg.sender,   // <-- router address when called via router
  recipient,
  ...
)
```

The extension was designed to gate the economically relevant actor, but when the router intermediates, the checked identity is the router, not the user.

**Why existing guards fail:** `BaseMetricExtension.beforeSwap` carries an `onlyPool` modifier that ensures `msg.sender` is a registered pool — this is correct and prevents spoofing the pool key. However, it does nothing to validate that the `sender` argument reflects the true end-user. The router is a public, permissionless contract; any user can call it.

**Exploit path:**
- Pool admin configures a curated pool with `SwapAllowlistExtension` and allowlists specific addresses.
- To allow any router-mediated swap to work at all, the admin must also call `setAllowedToSwap(pool, router, true)`.
- Once the router is allowlisted, a non-allowlisted user calls `router.exactInputSingle(pool, ...)`. The extension sees `sender = router`, finds `allowedSwapper[pool][router] = true`, and passes — the disallowed user trades successfully.
- If the admin does NOT allowlist the router, allowlisted users are broken out of the router path entirely (DoS on core periphery functionality).

## Impact Explanation
A curated pool's swap allowlist — the primary access-control mechanism for restricting who may trade — is fully bypassed by any unprivileged user routing through the public `MetricOmmSimpleRouter`. Disallowed users can execute swaps on pools that were explicitly configured to exclude them, violating the admin-boundary invariant and enabling unauthorized fund flows through the pool. This constitutes a broken core pool functionality / admin-boundary break with direct fund impact on curated pools.

## Likelihood Explanation
The router is a public, deployed periphery contract. Any user can call `exactInputSingle` or `exactInput` without any precondition beyond holding the input token. The only required condition is that the pool admin has allowlisted the router (which is necessary for any legitimate router user to function). The attack is repeatable every block, requires no special privilege, and is trivially executable.

## Recommendation
Pass the original caller's identity through the router to the pool, or have the extension resolve the true user from the router's transient payer context. The simplest correct fix is to have the pool expose a `swapFrom(address realSender, ...)` entry point that only the router may call, or to have the extension check `sender` against a registry that maps router → original caller. Alternatively, the `SwapAllowlistExtension` should be documented as incompatible with the router and the router should revert when used with allowlisted pools, but the cleanest fix is to thread the real user address through the call stack so the extension can gate the correct actor.

## Proof of Concept

```solidity
// Foundry test sketch
function test_swapAllowlistBypassViaRouter() public {
    // 1. Deploy pool with SwapAllowlistExtension
    // 2. Admin allowlists alice but NOT bob
    extension.setAllowedToSwap(address(pool), alice, true);
    // 3. Admin must allowlist router for alice to use it
    extension.setAllowedToSwap(address(pool), address(router), true);
    // 4. Bob (not allowlisted) swaps via router — should revert, but passes
    vm.prank(bob);
    router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: token0,
        tokenOut: token1,
        zeroForOne: true,
        amountIn: 1000,
        amountOutMinimum: 0,
        recipient: bob,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    }));
    // Bob successfully swaps despite not being on the allowlist
}
