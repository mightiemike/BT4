### Title
Missing Minimum Confirmation Validation Allows Zero-Confirmation Proof Acceptance — (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations = 0` without enforcing any minimum. An unprivileged NEAR caller can supply `confirmations = 0` alongside a valid Merkle proof for a transaction in the chain tip block and receive a `true` result, bypassing the finality guarantee the confirmation parameter is designed to enforce. Any recipient contract consuming this result is exposed to Bitcoin reorg-based double-spend.

### Finding Description
In `verify_transaction_inclusion` the only two guards on `confirmations` are:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [1](#0-0) 

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [2](#0-1) 

When `confirmations = 0`:
- Guard 1: `0 <= gc_threshold` — always passes.
- Guard 2: `(any_u64_value) >= 0` — always passes because the left-hand side is `u64`.

The function then proceeds directly to Merkle root comparison and returns `true` for any structurally valid proof, regardless of how recently the block was submitted. [3](#0-2) 

`verify_transaction_inclusion_v2` delegates to the same function after its coinbase check, so it inherits the identical flaw. [4](#0-3) 

Neither function carries `#[trusted_relayer]` or any other role guard — both are callable by any unprivileged NEAR account while the contract is unpaused. [5](#0-4) 

The `confirmations` field is part of the caller-supplied `ProofArgs` / `ProofArgsV2` structs with no server-side floor. [6](#0-5) 

### Impact Explanation
A recipient contract that calls `verify_transaction_inclusion[_v2]` and trusts the `true` return value to release assets can be exploited as follows: the attacker submits a Bitcoin transaction, waits for it to land in the chain tip, then immediately calls the verification function with `confirmations = 0`. The function returns `true`. The downstream contract releases assets. If the tip block is subsequently reorged (naturally or via a 51 % attack on lower-hashrate chains such as Dogecoin or Litecoin, both supported by this contract), the transaction no longer exists on-chain while the downstream contract has already acted. The corrupted value is the proof result (`true`) returned for a block with zero finality depth.

### Likelihood Explanation
The trigger requires only a valid Merkle proof for a transaction in any mainchain block — public data available to any observer. No privileged key or role is needed. The reorg step is the hardest part, but Dogecoin and Litecoin have materially lower hashrates than Bitcoin mainnet, making short reorgs realistic. Even without a reorg, the absence of a minimum forces downstream contracts to implement their own floor, which many will not.

### Recommendation
Enforce a non-zero minimum inside `verify_transaction_inclusion` before any other logic:

```rust
require!(args.confirmations >= 1, "confirmations must be at least 1");
```

Consider raising the floor to a protocol-appropriate constant (e.g., 6 for Bitcoin, higher for Dogecoin/Litecoin given their reorg history) and documenting it as a security invariant so downstream integrators cannot accidentally pass 0.

### Proof of Concept
1. Deploy the contract (Bitcoin or Dogecoin feature flag).
2. Submit a batch of headers so the chain tip is at height H.
3. Construct `ProofArgs { tx_id: <valid tx hash>, tx_block_blockhash: <tip block hash>, tx_index: 0, merkle_proof: <valid single-element proof>, confirmations: 0 }`.
4. Call `verify_transaction_inclusion` with the above args.
5. Observe `true` returned immediately — the tip block has 0 additional confirmations and is not final.
6. A downstream contract acting on this `true` result is now exposed to any reorg of the tip block.
