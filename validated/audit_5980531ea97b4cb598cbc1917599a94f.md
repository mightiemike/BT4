Audit Report

## Title
Router-mediated swaps pass the router address as `sender` to `SwapAllowlistExtension.beforeSwap()`, enabling allowlist bypass by any unprivileged user - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
When a swap is routed through `MetricOmmSimpleRouter`, the pool's `swap()` function receives the router as `msg.sender` and forwards it as `sender` to `SwapAllowlistExtension.beforeSwap()`. The extension then checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actualUser]`. If the pool admin allowlists the router address to enable router-based swaps, every unprivileged user can bypass the per-user allowlist by routing through the router.

## Finding Description

**Exact call path:**

1. Unprivileged user calls `MetricOmmSimpleRouter.exactInputSingle()` (or `exactInput`, `exactOutputSingle`, `exactOutput`).
2. The router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — at this point `msg.sender` seen by the pool is the **router contract address**.
3. `MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`, passing the router address as `sender`.
4. `ExtensionCalling._beforeSwap` encodes `sender = router` and dispatches to `SwapAllowlistExtension.beforeSwap(sender=router, ...)`.
5. `SwapAllowlistExtension.beforeSwap` evaluates:
   ```solidity
   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
       revert IMetricOmmPoolActions.NotAllowedToSwap();
   }
   ```
   Here `msg.sender` = pool, `sender` = **router address** — the actual user identity is lost.

**Root cause:** The pool correctly uses `msg.sender` (the immediate caller) as `sender`, but the router is the immediate caller, not the end user. The extension has no mechanism to recover the original user identity.

**Why existing guards fail:** The `onlyPool` guard in `BaseMetricExtension` only verifies that the extension is called by a registered pool — it does not validate that `sender` represents the actual economic actor. The router's callback-context transient storage (`_setNextCallbackContext`) records the real payer for payment purposes but never surfaces it to the extension.

**Exploit flow:**
- Pool admin deploys a curated pool with `SwapAllowlistExtension` and allowlists the router address (required for any router swap to work).
- Non-allowlisted attacker calls `exactInputSingle` on the router targeting the curated pool.
- The pool receives `sender = router`, which is allowlisted → `beforeSwap` passes → swap executes.
- The attacker trades on a pool they were explicitly excluded from.

## Impact Explanation
This is a direct allowlist bypass on curated pools. The `SwapAllowlistExtension` is the sole on-chain mechanism to restrict which addresses may trade on a given pool. Once bypassed, any unprivileged user can execute swaps on pools intended for a restricted set of counterparties (e.g., KYC'd users, institutional LPs). This constitutes broken core pool functionality and a direct admin-boundary break reachable by an unprivileged trader through a supported public periphery path. Severity: **High**.

## Likelihood Explanation
The condition is trivially reachable: the pool admin must allowlist the router to support router-based swaps (the expected production usage pattern). Any user who can call `exactInputSingle` on the router — i.e., any Ethereum account — can exploit this. No special privileges, flash loans, or multi-transaction setup are required. The attack is repeatable on every block.

## Recommendation
`SwapAllowlistExtension.beforeSwap` must check the **original user**, not the intermediate router. Two sound approaches:

1. **Pass the real initiator through `extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension decodes and verifies it. This requires a trusted router registry or a signed payload.
2. **Check `sender` against the allowlist and require direct pool calls for curated pools:** Document that router-mediated swaps are incompatible with `SwapAllowlistExtension` and revert if `sender` is a known router.
3. **Preferred — transient-storage originator:** Store the original `msg.sender` in transient storage at the router entry point and expose it via a standard interface that the extension reads from the pool's callback context, similar to how `_getPayer()` works for payment.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Foundry test sketch
// 1. Deploy pool with SwapAllowlistExtension
// 2. Allowlist the router (not the attacker)
// 3. Attacker calls router.exactInputSingle() — expect success (bypass)

function test_allowlistBypassViaRouter() public {
    // Setup: allowlist router, NOT attacker
    swapExtension.setAllowedToSwap(address(pool), address(router), true);
    // attacker is NOT allowlisted

    // Fund attacker and approve router
    token0.mint(attacker, 1e18);
    vm.prank(attacker);
    token0.approve(address(router), type(uint256).max);

    // Attacker swaps through router — should revert but does NOT
    vm.prank(attacker);
    router.exactInputSingle(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(token0),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: 1000,
            amountOutMinimum: 0,
            recipient: attacker,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""
        })
    );
    // Swap succeeds — allowlist bypassed
}
```
