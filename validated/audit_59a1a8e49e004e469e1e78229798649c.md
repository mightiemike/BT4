### Title
`ENABLE_SECP_OPS` Flag Guard Bypassed via 4-Byte Opcode Dispatch Path — (`File: src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` dispatches `op_secp256k1_verify` and `op_secp256r1_verify` through two independent code paths. The 1-byte opcode path (opcodes 64 and 65) correctly checks `ClvmFlags::ENABLE_SECP_OPS` before executing the secp operations. The 4-byte opcode path (opcodes `0x13d61f00` and `0x1c3a8f00`) dispatches to the same secp implementations **without any flag check**, allowing an attacker to invoke the secp operations even when `ENABLE_SECP_OPS` is absent from the active `ClvmFlags` — including in `MEMPOOL_MODE`.

---

### Finding Description

In `src/chia_dialect.rs`, the `Dialect::op()` implementation for `ChiaDialect` handles opcodes in two branches based on atom length.

**4-byte path (lines 157–183) — no flag check:**

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,
        0x1c3a8f00 => op_secp256r1_verify,
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
``` [1](#0-0) 

**1-byte path (lines 248–249) — correctly gated:**

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The `ENABLE_SECP_OPS` flag (bit `0x0800`) is the designated gate for the secp operations: [3](#0-2) 

`MEMPOOL_MODE` — the strict validation mode used before block inclusion — does **not** include `ENABLE_SECP_OPS`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [4](#0-3) 

This means that in `MEMPOOL_MODE`, a program using 1-byte opcode `64` or `65` is correctly rejected (flag absent → falls through to `unknown_operator` → `EvalErr::Unimplemented` because `NO_UNKNOWN_OPS` is set). But a program using the 4-byte form `0x13d61f00` or `0x1c3a8f00` bypasses the flag check entirely and executes the secp cryptographic verification.

The `ENABLE_SECP_OPS` flag is exposed to Python callers as a named constant: [5](#0-4) 

The `run_serialized_chia_program` Python entry point accepts a raw `u32` flags argument and constructs `ChiaDialect` from it, making the flag-bypass reachable from any Python caller that passes attacker-controlled CLVM bytes: [6](#0-5) 

---

### Impact Explanation

**Broken invariant:** `ENABLE_SECP_OPS` is supposed to be the sole gate controlling whether `op_secp256k1_verify` and `op_secp256r1_verify` execute. The 4-byte opcode path silently bypasses this gate.

**Concrete corrupted result:** A CLVM program using `0x13d61f00 pubkey msg sig` or `0x1c3a8f00 pubkey msg sig` executes a full secp256k1/r1 signature verification and returns `nil` on success (or raises `EvalErr::Secp256Failed`) even when the caller explicitly did not set `ENABLE_SECP_OPS`. In `MEMPOOL_MODE`, the same program using 1-byte opcode `64`/`65` is rejected, but the 4-byte form is accepted — a direct behavioral divergence between the two opcode encodings of the same operation.

**Consensus / mempool divergence:** Any node or validator that gates secp ops via `ENABLE_SECP_OPS` (e.g., before a hard fork activates) will accept 4-byte-encoded secp programs while rejecting 1-byte-encoded ones. This creates an inconsistency in which programs are valid, which is a consensus-relevant flag/operator wiring error.

---

### Likelihood Explanation

The 4-byte opcodes `0x13d61f00` and `0x1c3a8f00` are documented in the source code comments as the secp operator encodings. Any attacker who reads the source or the Chia protocol documentation can craft a CLVM program using these 4-byte forms. The entry path is fully attacker-controlled: the program bytes are supplied externally to `run_serialized_chia_program` or `run_program`. No special privileges are required.

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch path, mirroring the guard already present on the 1-byte path:

```diff
 let f = match opcode {
-    0x13d61f00 => op_secp256k1_verify,
-    0x1c3a8f00 => op_secp256r1_verify,
+    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
+    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
     _ => {
         return unknown_operator(allocator, o, argument_list, flags, max_cost);
     }
 };
```

This ensures that both opcode encodings of the secp operations are subject to the same flag guard, eliminating the bypass.

### Proof of Concept

1. Construct a CLVM program that calls `secp256k1_verify` using the 4-byte opcode form:
   - Opcode atom: `\x13\xd6\x1f\x00` (4 bytes)
   - Arguments: a valid secp256k1 pubkey, 32-byte message digest, and signature

2. Run it via `run_serialized_chia_program` with `flags = MEMPOOL_MODE` (which does **not** include `ENABLE_SECP_OPS`).

3. **Expected (correct) behavior:** The program should be rejected, because `ENABLE_SECP_OPS` is not set.

4. **Actual behavior:** The 4-byte dispatch path at `src/chia_dialect.rs` lines 175–182 matches `0x13d61f00`, skips any flag check, and calls `op_secp256k1_verify` directly. The secp verification executes and returns `nil` on a valid signature.

5. For comparison, run the same verification using 1-byte opcode `64` under the same `MEMPOOL_MODE` flags — it falls through to `unknown_operator`, which raises `EvalErr::Unimplemented` because `NO_UNKNOWN_OPS` is set. The two encodings of the same operation produce opposite outcomes under identical flags. [7](#0-6) [2](#0-1)

    py: Python,
    program: &[u8],
    args: &[u8],
    max_cost: Cost,
    flags: u32,

    let flags = ClvmFlags::from_bits_truncate(flags);
    let mut allocator = if flags.contains(ClvmFlags::LIMIT_HEAP) {
        Allocator::new_limited(500000000)
    } else {
        Allocator::new()
    };

    let r: Response = (|| -> PyResult<Response> {
        let program = node_from_bytes(&mut allocator, program).map_err(eval_to_py)?;
        let args = node_from_bytes(&mut allocator, args).map_err(eval_to_py)?;
        let dialect = ChiaDialect::new(flags);

        Ok(py.detach(|| run_program(&mut allocator, &dialect, program, args, max_cost)))
    })()?;
    adapt_response(py, allocator, r)
}
