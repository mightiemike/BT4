### Title
Missing `ClassTrace.Validate()` in `processReceivedPacket` Allows Malicious Counterparty to Mint Voucher NFTs with Empty `BaseClassId` — (`x/nft-transfer/keeper/packet.go`)

---

### Summary

`processReceivedPacket` calls `ParseClassTrace` on the attacker-controlled `data.ClassId` (after prepending the dest port/channel prefix) but never calls `classTrace.Validate()` before storing the trace and minting NFTs. A malicious IBC counterparty can send a packet whose `ClassId` ends in `/`, causing `ParseClassTrace` to produce a `ClassTrace` with an empty `BaseClassId`. The receiving chain stores this invalid trace and mints voucher NFTs under it — with no legitimate backing on the counterparty chain.

---

### Finding Description

**Step 1 — Entry point: `OnRecvPacket` / `ValidateBasic`**

`OnRecvPacket` in `relay.go` calls `data.ValidateBasic()` before processing: [1](#0-0) 

`ValidateBasic` in `packet.go` only checks that `ClassId` is not blank: [2](#0-1) 

A `ClassId` of `"someClass/"` passes this check — it is not blank.

---

**Step 2 — `ParseClassTrace` splits on `/` and takes the last segment as `BaseClassId`** [3](#0-2) 

For input `"nft/channel-1/someClass/"` (dest prefix prepended to `"someClass/"`):
- `classSplit = ["nft", "channel-1", "someClass", ""]`
- `BaseClassId = ""` (last element)
- `Path = "nft/channel-1/someClass"`

---

**Step 3 — `processReceivedPacket` stores the invalid trace and mints NFTs without calling `Validate()`** [4](#0-3) 

There is no call to `classTrace.Validate()` anywhere in this path. The trace with empty `BaseClassId` is stored via `k.SetClassTrace`, a denom is issued, and NFTs are minted to the receiver.

---

**Step 4 — `Validate()` exists and would catch this, but is never called here**

`ClassTrace.Validate()` explicitly rejects an empty `BaseClassId`: [5](#0-4) 

It is only called in the `ClassHash` gRPC query handler: [6](#0-5) 

It is **never** called in `processReceivedPacket`, `OnRecvPacket`, or `refundPacketToken`.

---

### Impact Explanation

A malicious counterparty that has established a legitimate IBC channel can send a packet with `data.ClassId = "someClass/"`. The victim chain will:
1. Store a `ClassTrace{Path: "nft/channel-1/someClass", BaseClassId: ""}` — violating the invariant that `BaseClassId` must be non-empty for any IBC class trace.
2. Issue a denom and mint voucher NFTs under `"ibc/{hash}"` with no corresponding escrowed NFTs on the counterparty chain.
3. Corrupt class trace accounting: `GetFullClassPath()` returns `"nft/channel-1/someClass/"` (trailing slash), making any subsequent transfer of these NFTs produce incorrect path parsing.

This is **unauthorized voucher minting** — NFTs are created on the victim chain with no legitimate backing.

---

### Likelihood Explanation

Any chain that has opened an IBC channel with the victim chain can trigger this. No governance, privileged key, or social engineering is required — only the ability to submit an IBC packet, which is the normal operation of a counterparty chain. The receiving chain is responsible for validating packet data, and this validation gap is exploitable by any counterparty acting in bad faith.

---

### Recommendation

Add a `classTrace.Validate()` call in `processReceivedPacket` immediately after `ParseClassTrace`, before `SetClassTrace` and `IBCClassID()`:

```go
classTrace := types.ParseClassTrace(prefixedClassID)
if err := classTrace.Validate(); err != nil {
    return err
}
if !k.HasClassTrace(ctx, classTrace.Hash()) {
    k.SetClassTrace(ctx, classTrace)
}
```

The same guard should be added in `refundPacketToken` for the `else` (mint-back) branch.

func ParseClassTrace(rawClassID string) ClassTrace {
	classSplit := strings.Split(rawClassID, "/")

	if classSplit[0] == rawClassID {
		return ClassTrace{
			Path:        "",
			BaseClassId: rawClassID,
		}
	}

	return ClassTrace{
		Path:        strings.Join(classSplit[:len(classSplit)-1], "/"),
		BaseClassId: classSplit[len(classSplit)-1],
	}
}
