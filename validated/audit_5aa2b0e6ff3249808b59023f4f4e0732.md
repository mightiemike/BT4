### Title
Stale Non-Zero Approval in `DirectDepositV1.creditDeposit()` Permanently Bricks USDT Deposits — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` before invoking `endpoint.depositCollateralWithReferral(...)`. The endpoint silently returns early — without consuming the approval — when the deposit amount is below the minimum threshold. This leaves a non-zero allowance on the endpoint. Any subsequent call to `creditDeposit()` will attempt to set a new non-zero approval over the existing one, which reverts for USDT and any other token that enforces the `allowance == 0` precondition before a non-zero `approve`.

---

### Finding Description

In `DirectDepositV1.creditDeposit()`:

```solidity
uint256 balance = token.balanceOf(address(this));
if (balance != 0) {
    token.approve(address(endpoint), balance);          // [1] sets non-zero approval
    endpoint.depositCollateralWithReferral(
        subaccount,
        productId,
        uint128(balance),
        "-1"
    );                                                  // [2] may NOT consume the approval
}
``` [1](#0-0) 

Inside `Endpoint.depositCollateralWithReferral`, there is an explicit early-return path that skips `handleDepositTransfer` entirely when the deposit amount is below the minimum:

```solidity
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked...
    // we can just skip the deposit and continue with the next asset.
    return;
}
handleDepositTransfer(...);   // ← never reached; approval is NOT consumed
``` [2](#0-1) 

After this silent return, the approval set at `[1]` remains non-zero on the endpoint. The next call to `creditDeposit()` — whether triggered by the owner, by `ContractOwner.creditDepositV1`, or by any unprivileged caller (the function has no access control) — will attempt `token.approve(address(endpoint), balance)` again with a non-zero value. USDT enforces:

```solidity
require(!((_value != 0) && (allowed[msg.sender][_spender] != 0)));
```

This causes the `approve` call to revert, permanently breaking `creditDeposit()` for that token on that `DirectDepositV1` instance.

Note that `ContractOwner.wrapVaultAsset` correctly handles this by resetting to zero first (`assetToken.approve(tokenAddr, 0)` then `assetToken.approve(tokenAddr, assetBalance)`), but `DirectDepositV1.creditDeposit()` does not. [3](#0-2) 

---

### Impact Explanation

The `DirectDepositV1` contract for a given subaccount is permanently bricked for USDT deposits. Any USDT sent to the contract after the stale approval is set cannot be deposited via `creditDeposit()` — every call reverts at the `approve` step. Funds accumulate in the contract and are only recoverable by the owner via `withdraw()`, but the deposit-to-subaccount functionality is irreversibly broken for that DDA instance. Because each subaccount maps to exactly one `DirectDepositV1` address (created once via `CREATE2` with a fixed salt), there is no way to redeploy a fresh instance for the same subaccount. [4](#0-3) 

---

### Likelihood Explanation

The trigger is realistic and attacker-controllable. `creditDeposit()` is an unrestricted `external` function — any address can call it. [5](#0-4) 

An attacker can:
1. Send a dust amount of USDT (below `MIN_DEPOSIT_AMOUNT` or `MIN_FIRST_DEPOSIT_AMOUNT`) to a target `DirectDepositV1` address.
2. Call `creditDeposit()` — the approval is set but the deposit is silently skipped.
3. The stale approval is now locked in. All future `creditDeposit()` calls for USDT on this DDA revert.

This can also occur non-maliciously: any user who accidentally sends a sub-minimum USDT amount to their DDA and triggers `creditDeposit()` (e.g., via `ContractOwner.creditDepositV1`) will permanently break their own deposit address for USDT. [6](#0-5) 

---

### Recommendation

Before setting a new non-zero approval, check whether an existing non-zero allowance is present and reset it to zero first:

```solidity
uint256 balance = token.balanceOf(address(this));
if (balance != 0) {
    // Reset stale approval before setting a new one (required for USDT-like tokens)
    uint256 currentAllowance = token.allowance(address(this), address(endpoint));
    if (currentAllowance != 0) {
        token.approve(address(endpoint), 0);
    }
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(
        subaccount,
        productId,
        uint128(balance),
        "-1"
    );
}
```

Alternatively, unconditionally reset the approval to zero after the deposit call to handle any partial or skipped consumption:

```solidity
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(...);
token.approve(address(endpoint), 0); // clear residual allowance
```

The `IIERC20Base` interface already exposes `approve`, so no interface changes are required. [7](#0-6) 

---

### Proof of Concept

1. Deploy or locate an existing `DirectDepositV1` instance for a subaccount.
2. Send 1 wei of USDT to the DDA address (below `MIN_FIRST_DEPOSIT_AMOUNT`).
3. Call `creditDeposit()`. The call succeeds: `token.approve(endpoint, 1)` is set, but `depositCollateralWithReferral` returns early without consuming it.
4. Send any further amount of USDT to the DDA.
5. Call `creditDeposit()` again. The call reverts at `token.approve(endpoint, newBalance)` because USDT rejects a non-zero approval over an existing non-zero allowance.
6. The DDA is now permanently bricked for USDT deposits. All future `creditDeposit()` calls for USDT revert at step 5.
