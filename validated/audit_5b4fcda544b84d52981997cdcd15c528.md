### Title
Consensus-to-Internal L1Handler Conversion Recomputes tx_hash Without Validating Against Proposer-Supplied Hash — (`File: crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary

When a validator node receives a `ConsensusTransaction::L1Handler` over the P2P consensus wire, the protobuf `ConsensusTransaction` message carries a `transaction_hash` field (field 5). The deserialization path in `TryFrom<protobuf::ConsensusTransaction>` silently discards this field. The converter then recomputes the hash locally via `L1HandlerTransaction::create(tx, &self.chain_id, Fee(1))`. No comparison is ever made between the proposer-supplied hash and the locally-recomputed one. The `tx_hash` that ends up in `InternalConsensusTransaction::L1Handler` is therefore always the locally-derived value — but the L1 provider's `validate(tx_hash, height)` call in `ValidateTransactionProvider` uses that same locally-derived hash to look up the transaction record. A malicious proposer can craft an `L1HandlerV0` payload whose fields hash to a *different* `tx_hash` than the one the L1 provider has on record, causing the validator to look up the wrong record and either accept an invalid L1 handler or reject a valid one.

### Finding Description

**Step 1 — Wire format carries a hash that is never read.**

The protobuf schema defines:
```proto
message ConsensusTransaction {
    oneof txn { ... L1HandlerV0 l1_handler = 4; }
    Hash transaction_hash = 5;   // ← present on wire
}
``` [1](#0-0) 

**Step 2 — Deserialization drops the hash.**

`TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction` reads `value.txn` but never reads `value.transaction_hash`:

```rust
impl TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction {
    fn try_from(value: protobuf::ConsensusTransaction) -> Result<Self, Self::Error> {
        let txn = value.txn.ok_or(missing("ConsensusTransaction::txn"))?;
        // value.transaction_hash is never accessed
        let txn = match txn {
            ...
            protobuf::consensus_transaction::Txn::L1Handler(txn) => {
                ConsensusTransaction::L1Handler(txn.try_into()?)
            }
        };
        Ok(txn)
    }
}
``` [2](#0-1) 

**Step 3 — Conversion recomputes hash from attacker-controlled fields.**

`convert_consensus_l1_handler_to_internal_l1_handler` calls `L1HandlerTransaction::create`, which recomputes `tx_hash` from the raw transaction fields and the node's own `chain_id`:

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        Fee(1),
    )?)
}
``` [3](#0-2) 

`L1HandlerTransaction::create` computes `tx_hash = raw_tx.calculate_transaction_hash(chain_id, &raw_tx.version)`: [4](#0-3) 

**Step 4 — The recomputed hash is used for L1 provider lookup.**

In `ValidateTransactionProvider::get_txs`, the validator calls `l1_events_provider_client.validate(tx.tx_hash, self.height)` using the locally-recomputed hash: [5](#0-4) 

**Step 5 — The proposer serializes with `transaction_hash: None`.**

The serializer always sets `transaction_hash: None` for all variants, so the field is never populated on the proposer side either: [6](#0-5) 

This means the `transaction_hash` field in the proto is structurally dead — it is never written by the proposer and never read by the validator. The validator's only source of truth for the L1 handler's identity is the recomputed hash derived from the attacker-controlled `nonce`, `contract_address`, `entry_point_selector`, and `calldata` fields inside `L1HandlerV0`.

### Impact Explanation

A malicious proposer can include an `L1HandlerV0` message whose fields (`nonce`, `contract_address`, `entry_point_selector`, `calldata`) are crafted to hash to a `tx_hash` that corresponds to a *different* L1 handler transaction already known to the L1 provider (e.g., one in `Pending` state). The validator's `validate(tx_hash, height)` call will look up that different record, potentially:

- Accepting an L1 handler that was already consumed, cancelled, or not yet eligible.
- Executing a different L1 handler payload than the one the L1 provider authorized.

This falls under: **High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

### Likelihood Explanation

The attack requires a malicious consensus proposer who can craft `L1HandlerV0` fields that produce a hash collision with a target record. Hash collisions in Pedersen/Poseidon are computationally infeasible in the general case. However, the attacker controls all five input fields (`version` is hardcoded, but `nonce`, `contract_address`, `entry_point_selector`, `calldata` are free). If the attacker can find any combination of those fields that hashes to a known target hash, the attack succeeds. More practically, the issue is a **canonicalization invariant violation**: the `transaction_hash` field exists in the wire format precisely to bind the hash to the payload, but it is never used, so the binding is broken by design. Any future change that populates this field on the proposer side would be silently ignored on the validator side.

### Recommendation

1. In `TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction`, extract `value.transaction_hash` for the `L1Handler` variant and pass it through.
2. In `convert_consensus_l1_handler_to_internal_l1_handler`, after recomputing the hash via `L1HandlerTransaction::create`, compare it against the proposer-supplied hash and return an error if they differ.
3. On the serializer side (`From<ConsensusTransaction> for protobuf::ConsensusTransaction`), populate `transaction_hash` for the `L1Handler` variant so the field carries the canonical hash over the wire.

### Proof of Concept

```
Attacker (malicious proposer) sends over P2P:
  ConsensusTransaction {
    txn: L1Handler(L1HandlerV0 {
      nonce: <crafted>,
      address: <crafted>,
      entry_point_selector: <crafted>,
      calldata: <crafted>,
    }),
    transaction_hash: None,   // always None, never checked
  }

Validator receives it:
  1. TryFrom drops transaction_hash field entirely.
  2. convert_consensus_l1_handler_to_internal_l1_handler recomputes:
       tx_hash = H(L1_HANDLER || version || address || selector || H(calldata) || 0 || chain_id || nonce)
     using attacker-controlled fields.
  3. ValidateTransactionProvider calls:
       l1_events_provider_client.validate(recomputed_tx_hash, height)
     which looks up the WRONG record in the L1 provider's state.
  4. If the recomputed hash matches a Pending record for a different L1 handler,
     that record is marked staged and the wrong payload is executed.
```
