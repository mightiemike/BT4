### Title
CEI Violation in `Clearinghouse.withdrawCollateral` Enables Balance Inflation via ERC-777 Callback Reentrancy — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` transfers tokens to the recipient **before** decrementing the sender's `SpotEngine` balance and performing the health check. No reentrancy guard exists anywhere in the protocol. An attacker using an ERC-777 token can exploit the `tokensReceived` callback window to re-deposit the withdrawn tokens via `Endpoint.depositCollateralWithReferral`, restoring the Clearinghouse's token balance while the slow-mode deposit queues a future balance credit. After the outer `withdrawCollateral` decrements the balance and the sequencer later processes the slow-mode deposit, the attacker's balance is fully restored — yet they have permanently extracted the tokens.

---

### Finding Description

`Clearinghouse.withdrawCollateral` follows the order: **Interaction → Effect → Check**, a direct violation of the Checks-Effects-Interactions pattern. [1](#0-0) 

The interaction is `handleWithdrawTransfer`, which chains two token transfers: [2](#0-1) 

The second transfer, inside `BaseWithdrawPool.handleWithdrawTransfer`, calls `token.safeTransfer(to, amount)` where `to` is the user-controlled recipient address: [3](#0-2) 

At the moment this `safeTransfer` fires, the following state is still true inside `Clearinghouse`:
- `spotEngine.updateBalance` has **not** been called — the sender's balance is still at the pre-withdrawal level.
- The health check `require(getHealth(...) >= 0)` has **not** been evaluated. [4](#0-3) 

If the token is ERC-777, the `tokensReceived` hook fires on the recipient contract during this window. The attacker's contract can call `Endpoint.depositCollateralWithReferral` from within the hook: [5](#0-4) 

`depositCollateralWithReferral` calls `handleDepositTransfer`, which transfers the tokens from the attacker back to the Clearinghouse and queues a slow-mode deposit transaction: [6](#0-5) 

The slow-mode deposit is not immediately credited — it is queued and processed later by the sequencer via `processSlowModeTransactionImpl` → `clearinghouse.depositCollateral` → `spotEngine.updateBalance(+amount)`. [7](#0-6) 

There is **no reentrancy guard** anywhere in the protocol — confirmed by a full-codebase search returning zero matches for `nonReentrant`, `ReentrancyGuard`, or `reentrancy`.

---

### Impact Explanation

**Critical — collateral theft / solvency accounting corruption.**

The attacker executes the following cycle:

| Step | Action | SpotEngine Balance | Clearinghouse Token Balance |
|---|---|---|---|
| 0 | Initial state | X | X |
| 1 | Sequencer processes `WithdrawCollateral` | X (not yet decremented) | X |
| 2 | `handleWithdrawTransfer` sends tokens to attacker | X (not yet decremented) | 0 |
| 3 | ERC-777 callback: attacker calls `depositCollateralWithReferral(X)` | X (not yet decremented) | X (restored) |
| 4 | Callback returns; `spotEngine.updateBalance(-X)` | 0 | X |
| 5 | Health check passes (balance = 0, no open positions) | 0 | X |
| 6 | Sequencer processes slow-mode deposit | X (restored) | X |

After one cycle: the attacker holds X tokens externally **and** their on-chain balance is fully restored to X. The cycle is repeatable. The Clearinghouse's token reserves are drained while the SpotEngine's accounting remains internally consistent, masking the theft from `manualAssert` checks. [8](#0-7) 

---

### Likelihood Explanation

**Medium.** The attack requires:
1. A supported collateral token that implements ERC-777 (or any token with a recipient callback on `transfer`).
2. The attacker to have pre-approved the `Endpoint` as an ERC-777 operator (or via standard `approve`) before the attack — a one-time setup step.
3. The attacker to deploy a contract implementing `tokensReceived` that calls `depositCollateralWithReferral`.

ERC-777 tokens are backward-compatible with ERC-20 and are accepted by the `IERC20Base` interface used throughout the protocol. The `ERC20Helper.safeTransfer` wrapper uses a low-level `call`, which forwards all gas and does not suppress callbacks. [9](#0-8) 

No privileged access, sequencer compromise, or governance capture is required. The attacker only needs a valid signed `WithdrawCollateral` transaction (their own) processed by the sequencer.

---

### Recommendation

1. **Apply the CEI pattern** in `Clearinghouse.withdrawCollateral`: decrement the SpotEngine balance and perform the health check **before** calling `handleWithdrawTransfer`.
2. **Add a reentrancy guard** (e.g., OpenZeppelin `ReentrancyGuardUpgradeable`) to `Endpoint.depositCollateralWithReferral` and `Clearinghouse.withdrawCollateral`. Since these are separate contracts, a shared reentrancy lock (stored in a common base or checked cross-contract) is needed to guard the multi-contract system.
3. **Document token compatibility**: explicitly state that tokens with transfer callbacks (ERC-777, fee-on-transfer, etc.) are not supported as collateral, and enforce this at the `addOrUpdateProduct` level.

### Proof of Concept

```
Attacker contract (implements ERC-777 tokensReceived):

tokensReceived(operator, from, to, amount, ...) {
    // Called during BaseWithdrawPool.handleWithdrawTransfer → token.safeTransfer(attacker, X)
    // At this point: SpotEngine balance = X (not yet decremented)
    //                Clearinghouse token balance = 0 (tokens are here in attacker)
    
    // Re-deposit the received tokens back
    token.approve(address(endpoint), amount);  // or use ERC-777 operator
    endpoint.depositCollateralWithReferral(subaccount, productId, amount, "");
    // Clearinghouse token balance = X (restored)
    // Slow-mode deposit queued at slowModeTxs[txCount]
}

Attack sequence:
1. Attacker submits signed WithdrawCollateral(amount=X) → sequencer processes it
2. withdrawCollateral → handleWithdrawTransfer → token.safeTransfer(attacker, X)
3. tokensReceived fires → depositCollateralWithReferral(X) → tokens back to Clearinghouse
4. withdrawCollateral resumes → spotEngine.updateBalance(-X) → balance = 0
5. getHealth(sender) >= 0 → passes (balance = 0)
6. Sequencer processes slow-mode deposit → spotEngine.updateBalance(+X) → balance = X
7. Attacker holds X tokens externally; balance restored to X
8. Repeat from step 1
```

The corrupted invariant: `sum(SpotEngine balances for productId) == Clearinghouse.balanceOf(token)` is violated after each cycle — the SpotEngine sum grows relative to the actual token reserve — until the Clearinghouse is insolvent. [1](#0-0) [10](#0-9) [11](#0-10)

        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        requireUnsanctioned(sender);

            // we cannot revert here, otherwise direct deposit could be blocked when there are

            return;

            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,

                        sender: subaccount,
                        productId: productId,
                        amount: amount
                    })
