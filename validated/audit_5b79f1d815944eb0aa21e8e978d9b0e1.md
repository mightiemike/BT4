### Title
Unprivileged Caller Can Prematurely Garbage-Collect Mainchain Block Headers via `run_mainchain_gc` - (File: contract/src/lib.rs)

### Summary

`run_mainchain_gc` is a public, state-mutating function that permanently deletes mainchain block headers from contract storage. It carries no caller access control: any unprivileged NEAR account can invoke it at any time the contract is not paused. An attacker can call it with `batch_size = u64::MAX` to immediately prune all headers beyond `gc_threshold`, permanently corrupting the light client's historical chain state and breaking `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` for every deleted block.

### Finding Description

`submit_blocks` is correctly gated behind `#[trusted_relayer]`, which enforces that only staked, role-bearing relayers may submit headers. However, `run_mainchain_gc` â€” which performs the destructive half of that same operation â€” carries only a `#[pause]` attribute: [1](#0-0) 

The `#[pause(except(roles(Role::UnrestrictedRunGC)))]` decorator restricts calls **only when the contract is paused**. When the contract is running normally (the common case), the function is fully open to any NEAR account. There is no `#[trusted_relayer]`, no role check, and no caller validation of any kind.

The function permanently removes entries from `mainchain_height_to_header` and `headers_pool`, then advances `mainchain_initial_blockhash`: [2](#0-1) 

The `batch_size` parameter is only bounded by `total_amount_to_remove` (the number of headers currently exceeding `gc_threshold`), not by any per-caller limit: [3](#0-2) 

The existence of the `UnrestrictedRunGC` role â€” whose sole documented purpose is to allow calling `run_mainchain_gc` on a **paused** contract â€” confirms the designers intended some form of access restriction, but they applied it only to the paused path and left the unpaused path completely open.

**Exploit path:**
1. Attacker waits until the relayer has submitted enough blocks that `amount_of_headers_we_store > gc_threshold` (i.e., the chain has grown past the retention window â€” a normal operational condition).
2. Attacker calls `run_mainchain_gc(u64::MAX)` from any NEAR account with no deposit or role.
3. The contract removes all mainchain headers from `initial_height` up to `tip_height - gc_threshold`, advances `mainchain_initial_blockhash` to the new oldest block, and permanently discards the deleted entries.
4. Any subsequent call to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a transaction in a deleted block panics with `"block does not belong to the current main chain"`. [4](#0-3) 

The deleted headers cannot be re-inserted: `submit_block_header` requires the previous block to already be in `headers_pool`, and the genesis anchor has been advanced past the deleted range.

### Impact Explanation

- **Permanent corruption of canonical chain state**: `mainchain_height_to_header`, `mainchain_header_to_height`, and `headers_pool` entries for the pruned range are irrecoverably deleted.
- **SPV proof verification broken**: Every downstream contract or user that calls `verify_transaction_inclusion_v2` for a Bitcoin transaction confirmed in a pruned block will receive a panic, not a `false` â€” making the light client actively harmful rather than merely unavailable.
- **Chain reorg blocked**: As noted in `CLAUDE.md`, if GC has advanced past a fork point, a legitimate reorg will panic with `PrevBlockNotFound`, permanently preventing the contract from tracking the heaviest chain. [5](#0-4) 

### Likelihood Explanation

The precondition â€” `amount_of_headers_we_store > gc_threshold` â€” is the **normal operating state** of a live deployment. The recommended `gc_threshold` is 52,704 blocks (~1 year). Once the chain has been running for a year, any anonymous NEAR account can trigger maximum GC at any time with a single free transaction. No staking, no role, no deposit is required.

### Recommendation

Add `#[trusted_relayer]` (or an equivalent role check such as `Role::UnrestrictedRunGC` or `Role::DAO`) to `run_mainchain_gc` so that only authorized accounts can invoke it directly. The internal call from `submit_blocks` already bypasses the guard correctly since it is an in-contract call. Alternatively, make `run_mainchain_gc` a private helper and expose a separate public entry point that enforces the same role gate as `submit_blocks`.

### Proof of Concept

```rust
// Any unprivileged NEAR account can do this:
let attacker = sandbox.dev_create_account().await?;
// No role grant, no deposit needed.
let outcome = attacker
    .call(contract.id(), "run_mainchain_gc")
    .args_json(json!({"batch_size": u64::MAX}))
    .max_gas()
    .transact()
    .await?;
assert!(outcome.is_success()); // succeeds with no authorization

// Now verify_transaction_inclusion panics for any pruned block:
let proof_outcome = attacker
    .call(contract.id(), "verify_transaction_inclusion_v2")
    .args_borsh(proof_for_pruned_block)
    .transact()
    .await?;
assert!(!proof_outcome.is_success()); // "block does not belong to the current main chain"
```

The test suite itself demonstrates that `run_mainchain_gc` is callable by a `user_account` that only holds `UnrestrictedSubmitBlocks` â€” not `UnrestrictedRunGC` â€” confirming no role check is enforced on the unpaused path: [6](#0-5)
