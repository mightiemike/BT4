### Title
Gas-key host functions pass borsh wire length instead of trie storage length to `gas_key_add_key_exec_fee` — (File: `runtime/near-vm-runner/src/logic/logic.rs`)

### Summary

Both `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call` forward the raw WASM-memory parameter `public_key_len` (the borsh-encoded byte count) directly to `gas_key_add_key_exec_fee`. That function expects the **trie storage length** (`trie_id_len()`). For ML-DSA-65 keys the two values diverge by a factor of ~59 (1953 borsh bytes vs. 33 trie bytes), producing a massively wrong exec-fee and a deterministic fee discrepancy between the transaction path and the host-function path.

### Finding Description

`gas_key_add_key_exec_fee` computes the per-nonce exec fee as:

```
nonce_key_len = access_key_key_len(account_id_len, public_key_len) + sizeof(NonceIndex)
per_byte      = gas_key_byte.exec_fee() × (nonce_key_len + NONCE_VALUE_LEN) × num_nonces
```

`access_key_key_len` treats `public_key_len` as the on-trie key contribution of the public key. For ML-DSA-65 the trie stores a 32-byte SHA3-256 hash (tag + hash = 33 bytes), not the 1952-byte raw key. The correct value to pass is therefore `public_key.trie_id_len()` = 33.

The **transaction path** in `runtime/runtime/src/config.rs` does this correctly:

```rust
let nonce_fee = gas_key_add_key_exec_fee(
    fees,
    account_id.len(),
    public_key.trie_id_len(),   // ← 33 for ML-DSA-65
    gas_key_info.num_nonces,
);

### Impact Explanation

For an ML-DSA-65 gas key with `num_nonces = N`:

- Correct `nonce_key_len` contribution from pubkey: 33 bytes  
- Actual `nonce_key_len` contribution passed: 1953 bytes  
- Excess bytes per nonce: ~1920  
- Excess exec gas: `1920 × gas_key_byte.exec_fee() × N`

With the production `gas_key_byte` exec fee and even a small `num_nonces`, the overcharge exceeds hundreds of Ggas per call. At large `num_nonces` values the overcharge surpasses the per-transaction gas limit, causing the host-function call to abort with `GasExceeded` even though the same operation succeeds via the transaction path. This creates a protocol-level fee-domain inconsistency: identical semantic operations (adding an ML-DSA-65 gas key) produce divergent gas costs depending on whether they are submitted as a signed transaction or issued from a contract via a host function.

### Likelihood Explanation

The bug is reachable by any unprivileged contract once both `ProtocolFeature::GasKeys` and `ProtocolFeature::PostQuantumSignatures` are active. No privileged role is required; any contract that calls `promise_batch_action_add_gas_key_with_full_access` or `promise_batch_action_add_gas_key_with_function_call` with an ML-DSA-65 public key triggers the wrong fee path. The features are present in the production codebase and gated only by protocol version.

### Recommendation

Replace `public_key_len as usize` with `public_key.decode()?.trie_id_len()` (or decode the key first and call `trie_id_len()` on the result) in both host-function call sites:

```rust
// After: let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
let decoded_pk = public_key.decode()?;
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    decoded_pk.trie_id_len(),   // ← correct: 33 for ML-DSA-65
    num_nonces,
);
```

Add a variant of `test_gas_key_fee_parity` that exercises `KeyType::MLDSA65` to prevent regression.

### Proof of Concept

1. Enable `GasKeys` + `PostQuantumSignatures` on a test network.  
2. Deploy a contract that calls `promise_batch_action_add_gas_key_with_full_access` with an ML-DSA-65 public key and `num_nonces = 1`.  
3. Observe that `gas_burnt` on the host-function receipt is ~59× larger than `gas_burnt` on an equivalent `AddKey` transaction for the same key.  
4. Increase `num_nonces` until the host-function call fails with `GasExceeded` while the transaction path succeeds.  
5. Confirm the discrepancy disappears when the same test is run with an ED25519 key (where `len()` == `trie_id_len()` == 33).
