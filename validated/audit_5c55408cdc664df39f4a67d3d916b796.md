### Title
`chain_gas_consumed` Hardcodes L2 Gas as `Felt::ZERO` in Receipt Hash, Producing Wrong `receipt_commitment` and Block Hash - (File: `crates/starknet_api/src/block_hash/receipt_commitment.rs`)

---

### Summary

The `chain_gas_consumed` helper in `receipt_commitment.rs` always chains a hardcoded `Felt::ZERO` for the L2-gas slot of every transaction's receipt hash, silently discarding the actual `gas_consumed.l2_gas` value that the blockifier populates. Starting with Starknet 0.14.0, L2 gas is non-zero for ordinary invoke transactions. Every such transaction therefore produces a wrong receipt leaf hash, the aggregated `ReceiptCommitment` is wrong, and the final block hash that commits to it is wrong.

---

### Finding Description

`chain_gas_consumed` is the sole function that serialises gas consumption into the per-transaction receipt hash:

```rust
// crates/starknet_api/src/block_hash/receipt_commitment.rs  lines 86-91
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&Felt::ZERO)                    // ← hardcoded, ignores gas_consumed.l2_gas
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
```

The comment "In the current RPC: always 0" was accurate before L2 gas was introduced, but `GasVector` already carries a live `l2_gas` field:

```rust
// crates/starknet_api/src/execution_resources.rs  lines 106-111
pub struct GasVector {
    pub l1_gas: GasAmount,
    pub l1_data_gas: GasAmount,
    pub l2_gas: GasAmount,   // populated from total_gas_consumed.l2_gas
}
```

The feeder-gateway client and the blockifier both populate `gas_consumed.l2_gas` from `total_gas_consumed.l2_gas` (e.g. `76014735` in the regression fixture `block_post_0_14_2.json`). That value flows into `TransactionOutputForHash.gas_consumed`, which is passed directly to `chain_gas_consumed` — but the function discards it.

The call chain is:

```
calculate_receipt_hash(receipt_element)          // receipt_commitment.rs:45
  └─ chain_gas_consumed(hash_chain, gas_consumed) // uses Felt::ZERO, not gas_consumed.l2_gas
       └─ calculate_receipt_commitment            // receipt_commitment.rs:33
            └─ calculate_block_commitments        // block_hash_calculator.rs:285
                 └─ calculate_block_hash          // block_hash_calculator.rs:245
                      └─ .chain(&block_commitments.receipt_commitment.0)
```

`receipt_commitment` is unconditionally chained into the block hash for all versions ≥ `V0_13_2`. The wrong leaf hash propagates to a wrong `ReceiptCommitment`, which propagates to a wrong `BlockHash`.

---

### Impact Explanation

**Critical — Wrong receipt result and wrong block hash for accepted input.**

For every block containing at least one transaction with non-zero L2 gas (all post-0.14.0 blocks), the sequencer commits a `receipt_commitment` that does not match the canonical Starknet specification. The block hash is therefore wrong. Any verifier, prover, or sync peer that independently recomputes the receipt commitment will observe a mismatch. RPC endpoints that serve `receipt_commitment` or `block_hash` return authoritative-looking wrong values.

---

### Likelihood Explanation

**High.** Starknet 0.14.0 made L2 gas non-zero for all standard invoke transactions. The regression fixtures (`block_post_0_14_2.json`, `block_post_0_14_3.json`) already contain transactions with `total_gas_consumed.l2_gas = 76014735`. No special attacker action is required; any ordinary V3 invoke transaction triggers the wrong hash.

---

### Recommendation

Replace the hardcoded `Felt::ZERO` with the actual `l2_gas` field:

```rust
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&gas_consumed.l2_gas.into())   // was: Felt::ZERO
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
```

---

### Proof of Concept

1. Submit any V3 invoke transaction on a post-0.14.0 network. The blockifier records `gas_consumed.l2_gas > 0` in `TransactionOutputForHash`.

2. `calculate_block_commitments` calls `calculate_receipt_commitment`, which calls `calculate_receipt_hash` for each transaction.

3. Inside `calculate_receipt_hash`, `chain_gas_consumed` is called with the populated `GasVector`. It chains `Felt::ZERO` instead of `gas_consumed.l2_gas` (e.g. `76014735`).

4. The resulting `ReceiptCommitment` differs from the canonical value. Concretely, the regression test in `receipt_commitment_test.rs` uses `get_transaction_output()` which sets `gas_consumed.l2_gas`; if that value were non-zero the expected hash `0x6276abf21e7c68b2eecfdc8a845b11b44401901f5f040efe10c60d625049646` would change, but the test passes today only because the test fixture happens to use zero L2 gas.

5. `calculate_block_hash` chains this wrong `receipt_commitment` at line 264, producing a wrong `BlockHash` for every block with non-zero L2 gas transactions. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
