### Title
Protobuf `ValidResourceBounds` Deserialization Heuristic Misclassifies `AllResources` Transactions as `L1Gas`, Producing Wrong Transaction Hash - (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic — checking whether `l2_gas` and `l1_data_gas` are both zero — to decide which enum variant to reconstruct. An `AllResources` transaction (Starknet ≥ 0.13.3) whose L2 and L1_DATA gas bounds happen to be zero is silently downgraded to `L1Gas`. Because `get_tip_resource_bounds_hash` branches on the enum variant to decide whether to include the `L1_DATA_GAS` field in the Poseidon hash, the receiving node computes a different transaction hash than the originating node, causing valid transactions to be rejected or stored under a wrong hash.

---

### Finding Description

**Root cause — wrong variant selected during protobuf deserialization**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` (lines 417–436):

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← chosen for ANY tx with zero L2/L1_DATA
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

The protobuf wire format carries no tag distinguishing `L1Gas` from `AllResources`; both variants serialize to the same three fields (`l1_gas`, `l2_gas`, `l1_data_gas`). For `L1Gas` the serializer writes zero for `l2_gas` and `l1_data_gas`; for `AllResources` it writes the actual values. When an `AllResources` transaction carries zero L2 and zero L1_DATA gas, the serialized bytes are identical to an `L1Gas` transaction, and the deserializer always picks `L1Gas`.

**Divergent hash — `get_tip_resource_bounds_hash` branches on the variant**

`crates/starknet_api/src/transaction_hash.rs` (lines 188–211):

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    // Always hashes L1_GAS and L2_GAS.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];
    // Only hashes L1_DATA_GAS for AllResources.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],                          // ← 2 elements
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // ← 3 elements
        }
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```

`get_concat_resource` encodes the resource name into the felt:

```
[0x00 | resource_name (56 bit) | max_amount (64 bit) | max_price_per_unit (128 bit)]
```

For `L1_DATA_GAS` with zero bounds the result is `0x004c315f44415441_0000000000000000_00000000000000000000000000000000` — a **non-zero felt**. Including vs. omitting it produces a different Poseidon digest.

**Concrete divergence**

| Originating node | Receiving node (after protobuf round-trip) |
|---|---|
| `AllResources { l1_gas, l2_gas=0, l1_data_gas=0 }` | `L1Gas(l1_gas)` |
| Hash = `poseidon(tip, L1_GAS_felt, L2_GAS_felt, L1_DATA_felt)` | Hash = `poseidon(tip, L1_GAS_felt, L2_GAS_felt)` |
| Canonical hash H₁ | Computed hash H₂ ≠ H₁ |

---

### Impact Explanation

The transaction hash is the canonical identifier used for mempool deduplication, receipt storage, and signature verification. A receiving node that recomputes H₂ ≠ H₁ will either:

1. **Reject a valid transaction** — if it validates the received hash against its own computation (mempool/gateway admission rejects a valid transaction before sequencing), or
2. **Store the transaction under the wrong hash** — producing a wrong receipt, wrong event, or wrong state entry for any downstream consumer that queries by hash.

This matches the **High** impact: "Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing" and the **Critical** impact: "Wrong state, receipt, event … or revert result from blockifier/syscall/execution logic for accepted input."

---

### Likelihood Explanation

- The trigger is an `AllResources` V3 transaction (Starknet ≥ 0.13.3) with `l2_gas = 0` and `l1_data_gas = 0`. This is a valid configuration — a sender may set both to zero if they do not intend to pay for L2 execution gas or data-availability gas.
- The path is reachable by any unprivileged user submitting such a transaction to the gateway; it is then propagated via P2P protobuf sync to all peers.
- No special privileges or malformed bytes are required; the transaction is well-formed and accepted at the gateway.

---

### Recommendation

Preserve the variant tag explicitly in the protobuf encoding. The simplest fix is to add a boolean or enum field to the protobuf `ResourceBounds` message that records whether the transaction is `L1Gas` or `AllResources`, and use that field — not the zero-value heuristic — during deserialization. Alternatively, always reconstruct `AllResources` when the protobuf message was produced by a ≥ 0.13.3 node (e.g., by checking the transaction version field before calling the converter).

---

### Proof of Concept

1. Construct an `AllResources` V3 invoke transaction with `l1_gas = {max_amount: 1000, max_price: 1}`, `l2_gas = {0, 0}`, `l1_data_gas = {0, 0}`.
2. Compute the canonical hash H₁ using `get_tip_resource_bounds_hash` on the `AllResources` variant — this includes the `L1_DATA_GAS` felt in the Poseidon input.
3. Serialize the transaction to protobuf via `From<ValidResourceBounds> for protobuf::ResourceBounds` — both `l2_gas` and `l1_data_gas` fields are written as zero.
4. Deserialize via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` — the heuristic at line 431 fires (`l1_data_gas.is_zero() && l2_gas.is_zero()`), returning `L1Gas(l1_gas)`.
5. Recompute the hash H₂ using `get_tip_resource_bounds_hash` on the `L1Gas` variant — the `L1_DATA_GAS` felt (`0x004c315f44415441…`) is **not** included.
6. Assert H₁ ≠ H₂. The receiving node stores or validates against H₂, diverging from the canonical H₁. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
