### Title
`traceTransaction` Always Returns `internal_server_error` for Pending Declare V0/V1 Transactions — (`crates/apollo_rpc/src/v0_8/api/mod.rs`)

### Summary

The `trace_transaction` pending path converts each pending transaction via `stored_txn_to_executable_txn`, which for Declare V0 and V1 calls `get_deprecated_class_for_re_execution`. That helper only reads from committed storage; pending declares have not yet been committed. The lookup always returns `None`, producing `internal_server_error("Missing deprecated class definition of {class_hash}.")` for every pending Declare V0/V1 transaction instead of a valid trace.

---

### Finding Description

**Pending path in `trace_transaction`** (api_impl.rs:1167–1181):

`state_number` is set to `StateNumber::unchecked_right_after_block(latest_committed_block)`. Each pending transaction is then converted:

```rust
stored_txn_to_executable_txn(
    starknet_api_transaction,
    &storage_txn,
    state_number,   // ← state of latest committed block
)
``` [1](#0-0) 

**`stored_txn_to_executable_txn` for Declare V0/V1** (mod.rs:347–367) calls `get_deprecated_class_for_re_execution` with that `state_number`: [2](#0-1) 

**`get_deprecated_class_for_re_execution`** (mod.rs:436–451) advances the state number one block further and queries committed storage:

```rust
let state_number_after_block =
    StateNumber::unchecked_right_after_block(state_number.block_after());
storage_txn
    .get_state_reader()?
    .get_deprecated_class_definition_at(state_number_after_block, &class_hash)
    ...
    .ok_or_else(|| internal_server_error(
        format!("Missing deprecated class definition of {class_hash}.")
    ))
``` [3](#0-2) 

The code comment at line 433–435 explicitly states this function is designed for **re-execution of already-committed transactions** ("They are stored in the state after the block in which they appeared"). For a pending Declare V0/V1, the class has never been committed to storage, so `get_deprecated_class_definition_at` always returns `None`. [4](#0-3) 

Additionally, the `ExecutionPendingData` constructed for the pending path explicitly zeroes out all class-related fields (`old_declared_contracts`, `classes`), so the execution engine also has no access to the pending class: [5](#0-4) 

---

### Impact Explanation

Any call to `traceTransaction` for a pending Declare V0 or V1 transaction returns `internal_server_error("Missing deprecated class definition of …")` — an authoritative-looking server error — instead of a valid `TransactionTrace`. This fits the allowed impact: **High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

Declare V1 (Cairo 0 contract declarations) is still a valid transaction type on Starknet. Any user who submits a Declare V1 transaction that is included in the pending block and then calls `traceTransaction` on the resulting pending hash will deterministically hit this error. No special privileges are required beyond submitting a standard transaction.

---

### Recommendation

`stored_txn_to_executable_txn` must not be used for pending Declare V0/V1 transactions without first supplying the class definition from the pending block's own class data. The fix should:

1. Pass the pending block's deprecated class map alongside the transaction list in the pending path.
2. For Declare V0/V1 in the pending path, look up the class from that map rather than calling `get_deprecated_class_for_re_execution` (which is only correct for committed blocks).

---

### Proof of Concept

1. Build a node with an empty committed chain (or any committed chain).
2. Add a Declare V1 transaction to the pending block (via the feeder gateway or a test harness) **without** committing the block.
3. Call `starknet_traceTransaction` with the pending transaction hash.
4. Observe the response is `internal_server_error("Missing deprecated class definition of <class_hash>.")` rather than a valid `DECLARE` transaction trace.

The divergent value is concrete: `internal_server_error` with the exact message above, instead of a `TransactionTrace` with a `DeclareTransactionTrace` variant.

```rust
fn get_deprecated_class_for_re_execution(
    storage_txn: &StorageTxn<'_, RO>,
    state_number: StateNumber,
    class_hash: ClassHash,
) -> Result<starknet_api::deprecated_contract_class::ContractClass, ErrorObjectOwned> {
    let state_number_after_block =
        StateNumber::unchecked_right_after_block(state_number.block_after());
    storage_txn
        .get_state_reader()
        .map_err(internal_server_error)?
        .get_deprecated_class_definition_at(state_number_after_block, &class_hash)
        .map_err(internal_server_error)?
        .ok_or_else(|| {
            internal_server_error(format!("Missing deprecated class definition of {class_hash}."))
        })
}
