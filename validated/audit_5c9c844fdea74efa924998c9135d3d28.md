### Title
Immediate Maintenance Weight Update via `updateRisk()` Enables Forced Liquidations Without User Action ‚Äî (`core/contracts/BaseEngine.sol`)

---

### Summary

`BaseEngine.updateRisk()` allows the owner to immediately overwrite `longWeightMaintenance` and `shortWeightMaintenance` for any product. Because maintenance health is computed live from these weights, any position that was above the maintenance health threshold before the update can fall below it the instant the update is applied. Any liquidator can then immediately liquidate those positions with no market movement or user action required.

---

### Finding Description

`BaseEngine.updateRisk()` stores the new `RiskStore` directly into `_risk().value[productId]` with no delay, staging period, or timelock: [1](#0-0) 

The owner-facing entry points in `ContractOwner` call this function directly for both spot and perp engines: [2](#0-1) 

Maintenance health for every subaccount is computed on-demand by `_calculateProductHealth()`, which reads `_risk(productId)` live at call time: [3](#0-2) 

`RiskHelper._getWeightX18()` selects `longWeightMaintenanceX18` for long positions and `shortWeightMaintenanceX18` for short positions when `healthType == MAINTENANCE`: [4](#0-3) 

A subaccount is liquidatable when its maintenance health is negative: [5](#0-4) 

There is no timelock, grace period, or staged rollout anywhere in the codebase. A grep for `timelock`, `gracePeriod`, and `delay` across all `.sol` files returns zero matches. [1](#0-0) 

---

### Impact Explanation

For a long position of size `A` at oracle price `P`, the maintenance health contribution is:

```
health_contribution = A * longWeightMaintenanceX18 * P
```

If the owner decreases `longWeightMaintenance` (e.g., from `0.95e9` to `0.80e9`), every long position's maintenance health contribution drops by `(0.95 - 0.80) * A * P` instantaneously. A subaccount whose total maintenance health was slightly positive ‚Äî say `+Œµ` ‚Äî can immediately become negative, making it liquidatable. The liquidator then captures the position at a discount equal to the liquidation penalty derived from the new (lower) maintenance weight, causing direct asset loss to the user.

The same applies symmetrically to short positions via `shortWeightMaintenance`: increasing it (toward 2) reduces the health contribution of short positions.

**Corrupted state:** subaccount maintenance health (`IProductEngine.HealthType.MAINTENANCE`), which gates liquidation eligibility. The asset delta is the liquidation penalty paid by the liquidatee. [6](#0-5) 

---

### Likelihood Explanation

The owner key is a single EOA or multisig. The function `perpUpdateRisk` / `spotUpdateRisk` is callable at any time with no on-chain announcement. A malicious or compromised owner, or a governance capture, can execute this atomically. Even a well-intentioned owner making a routine risk parameter tightening (e.g., reducing leverage limits) will trigger immediate liquidations for all positions that were near the maintenance boundary. This is not a theoretical edge case ‚Äî it is the normal operational path for adjusting product risk parameters. [7](#0-6) 

---

### Recommendation

1. **Timelock for weight reductions:** Apply a mandatory delay (e.g., 24‚Äì72 hours) before a decrease in `longWeightMaintenance` or an increase in `shortWeightMaintenance` takes effect. Increases in `longWeightMaintenance` (which improve user health) can remain immediate.
2. **Staged rollout:** Interpolate the weight change over multiple blocks/epochs so users have time to add margin or close positions.
3. **Snapshot-based liquidation guard:** At the time of a liquidation check, use the weight that was in effect when the position was opened, or the maximum of old and new weight, for a grace window.
4. **Emit an event with a future effective timestamp** so users and keepers can react before the change takes effect.

---

### Proof of Concept

**Setup:**
- Product: perp ETH, `longWeightMaintenance = 0.95e9` (i.e., `0.95 * 1e9`)
- User has a long position: `amount = 10 ETH`, `quoteAmount = -9,000 USDC`, oracle price = `1,000 USDC`
- Maintenance health = `10 * 0.95 * 1000 + (-9000) = 9500 - 9000 = +500` ‚Üí **safe**

**Attack:**
1. Owner calls `ContractOwner.perpUpdateRisk(ethPerpId, RiskStore{longWeightMaintenance: 0.89e9, ...})`.
2. `BaseEngine.updateRisk()` executes `_risk().value[productId] = riskStore` immediately. [8](#0-7) 
3. User's maintenance health is now: `10 * 0.89 * 1000 - 9000 = 8900 - 9000 = -100` ‚Üí **liquidatable**.
4. Liquidator submits `LiquidateSubaccount` transaction through the Endpoint.
5. `isUnderMaintenance()` returns `true`. [5](#0-4) 
6. Liquidation proceeds; user loses position and pays liquidation penalty. The penalty is derived from the new (lower) maintenance weight via `getLiqPriceX18`. [6](#0-5) 

No market price movement occurred. No user action triggered the liquidation. The sole cause was the immediate weight update.

```text
    function _getWeightX18(
        Risk memory risk,
        int128 amount,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }

        int128 weight;
        if (amount >= 0) {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.longWeightInitialX18
                : risk.longWeightMaintenanceX18;
        } else {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.shortWeightInitialX18
                : risk.shortWeightMaintenanceX18;
        }

        return weight;
    }
```
