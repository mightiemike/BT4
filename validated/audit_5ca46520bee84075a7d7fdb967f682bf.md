### Title
Linked Signer Can Self-Replace via `LinkSigner` Fast Path — (`core/contracts/EndpointTx.sol`)

---

### Summary

In `processTransactionImpl`, the `LinkSigner` transaction type calls `validateSignedTx` with `allowLinkedSigner = true`. This means the currently linked signer — who is supposed to have only delegated trading authority — can sign a `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with any attacker-controlled address, without the subaccount owner's involvement.

---

### Finding Description

**Root cause — `EndpointTx.sol` lines 576–590:**

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(...);
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx(..., true)` resolves to `validateSignature(sender, digest, sig, true)`, which passes `getLinkedSigner(sender)` as the accepted alternate signer:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` then accepts the signature if `recovered == subaccount_owner` **OR** `recovered == linkedSigner`:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

So `signerA` (the current linked signer) satisfies `recovered == linkedSigner`, and the `LinkSigner` transaction is accepted, writing `attackerKey` into `linkedSigners[subaccount]`.

**Contrast with the slow-mode path**, which correctly enforces owner-only control:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    ...
    validateSender(txn.sender, sender);   // msg.sender must be subaccount owner
    linkedSigners[txn.sender] = ...;
}
``` [4](#0-3) 

The fast path has no equivalent owner-only guard.

---

### Impact Explanation

Once `linkedSigners[subaccount] = attackerKey`, the attacker controls the linked signer slot. `WithdrawCollateral` in the fast path also passes `allowLinkedSigner = true`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // ← linked signer accepted for withdrawals
);
``` [5](#0-4) 

The attacker can then sign `WithdrawCollateral` transactions draining the victim's collateral, and sign any other transaction type that accepts `allowLinkedSigner = true` (liquidations, transfers, perp orders, etc.), achieving full unauthorized control over the victim's subaccount.

---

### Likelihood Explanation

Any address that has ever been granted linked-signer status on a subaccount can execute this unilaterally. The only external dependency is sequencer inclusion of the crafted `LinkSigner` transaction, which is the normal production path. No admin keys, governance, or leaked secrets are required.

---

### Recommendation

Pass `false` for `allowLinkedSigner` when processing `LinkSigner` transactions in `processTransactionImpl`, mirroring the slow-mode path's owner-only enforcement:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only subaccount owner may change linkedSigners
);
``` [6](#0-5) 

---

### Proof of Concept

1. Owner of `subaccount` sets `linkedSigners[subaccount] = signerA` (normal delegation).
2. Attacker controls `signerA`. Attacker constructs a `SignedLinkSigner` transaction:
   - `tx.sender = subaccount`
   - `tx.signer = bytes32(uint256(uint160(attackerKey)))`
   - `tx.nonce = nonces[address(uint160(bytes20(subaccount)))]`
   - Signs the EIP-712 digest with `signerA`'s private key.
3. Attacker submits the transaction to the sequencer via the normal fast-path endpoint.
4. `validateSignedTx` accepts the signature (`recovered == signerA == getLinkedSigner(subaccount)`).
5. `linkedSigners[subaccount]` is overwritten with `attackerKey`.
6. Attacker signs a `WithdrawCollateral` transaction with `attackerKey`, draining the subaccount.
7. Assert: `linkedSigners[subaccount] == attackerKey` and victim balance = 0.
