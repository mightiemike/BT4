Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address as the swapper identity, allowing any user to bypass per-user swap restrictions via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `sender` equals the router contract address, not the end user. A pool admin who allowlists the router to enable router-based swaps inadvertently grants every user — including those not individually allowlisted — the ability to bypass the per-user restriction by calling any of the router's `exact*` entry points.

## Finding Description

**Root cause — `sender` is the router, not the end user:**

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // direct caller of pool.swap() — the router when routed
    recipient,
    ...
    extensionData
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks whether `sender` is allowlisted for the calling pool (`msg.sender` = pool):

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

**Exploit path:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a specific set of users (e.g., KYC-verified addresses).
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to allow router-mediated swaps (a natural and expected action for any pool that wants to support the public periphery).
3. A non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle(...)`. The router calls `pool.swap(...)` directly, making `msg.sender` in the pool context equal to the router address.
4. `SwapAllowlistExtension.beforeSwap` receives `sender = router`, finds `allowedSwapper[pool][router] == true`, and passes — the non-allowlisted end user's swap succeeds.

**Why existing guards fail:**

The extension has no mechanism to recover the original end user. The `extensionData` field is user-controlled and unverified, so it cannot be trusted to carry the real caller. The `recipient` argument is also user-controlled. There is no on-chain path from the extension back to the router's `msg.sender`.

## Impact Explanation

The `SwapAllowlistExtension` is the sole on-chain mechanism for pools to restrict which addresses may swap. When the router is allowlisted (required for any pool that wants to support the standard periphery), the allowlist is completely ineffective — every public user can swap regardless of their individual allowlist status. This constitutes broken core pool functionality: the access-control extension does not gate the actor it is designed to gate, and the pool admin has no way to simultaneously support router-based swaps and enforce per-user swap restrictions.

## Likelihood Explanation

The precondition — pool admin allowlisting the router — is a routine and expected configuration for any pool that wants to support the public `MetricOmmSimpleRouter`. The bypass requires no special privileges, no flash loans, and no complex setup: any user with tokens simply calls a standard router entry point. The attack is repeatable on every block and requires no coordination.

## Recommendation

The extension must gate the actual end user, not the intermediary. Two sound approaches:

1. **Recover the end user from `extensionData` with router-side signing:** Have the router encode and sign the original `msg.sender` into `extensionData`; the extension verifies the signature against a trusted router address. This requires the extension to know the router address.

2. **Check `tx.origin` as a fallback for known router callers:** When `sender` is a known/trusted router, fall back to `tx.origin` as the identity to check. This is safe in this context because `tx.origin` cannot be spoofed and the router is a non-upgradeable contract.

3. **Structural fix:** Add a second `address realSwapper` parameter to the `beforeSwap` interface that the pool populates from a verified source, or require the router to pass the end user's address through a verified `extensionData` field that the extension decodes and checks.

## Proof of Concept

```solidity
// Foundry integration test sketch
function test_swapAllowlistBypassViaRouter() public {
    // 1. Deploy pool with SwapAllowlistExtension
    // 2. Admin allowlists only `trustedUser`
    extension.setAllowedToSwap(address(pool), trustedUser, true);
    // 3. Admin also allowlists the router (to support router swaps)
    extension.setAllowedToSwap(address(pool), address(router), true);

    // 4. `attacker` is NOT allowlisted
    vm.startPrank(attacker);
    token0.approve(address(router), type(uint256).max);

    // 5. Attacker routes through the public router — swap succeeds
    router.exactInputSingle(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            recipient: attacker,
            tokenIn: address(token0),
            zeroForOne: true,
            amountIn: 1000,
            amountOutMinimum: 0,
            priceLimitX64: 0,
            deadline: block.timestamp + 1,
            extensionData: ""
        })
    );
    // Swap succeeds — attacker bypassed the per-user allowlist
}
```
