Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual swapper, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against `allowedSwapper[msg.sender][sender]`. The pool passes `msg.sender` of its own `swap()` call as `sender` to the extension. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the user. If the router is allowlisted, every user on the network can bypass the per-user swap allowlist by routing through the public router. If the router is not allowlisted, individually allowlisted users cannot use the router at all, breaking the intended UX.

## Finding Description
**Root cause — identity mismatch in the swap hook:**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, recipient, ...)` (line 231), forwarding its own `msg.sender` as the `sender` argument to every configured extension.

`SwapAllowlistExtension.beforeSwap` (line 37) then checks:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`.

`MetricOmmSimpleRouter.exactInputSingle` (line 72-80) calls `pool.swap(...)` directly with no mechanism to forward the original `msg.sender`. From the pool's perspective, `msg.sender = address(router)`. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Exploit flow:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists only specific addresses (e.g., KYC'd traders).
2. A non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting that pool.
3. The router calls `pool.swap(...)` — pool sees `msg.sender = router`.
4. `beforeSwap` checks `allowedSwapper[pool][router]`. If the router was ever allowlisted (e.g., to let any allowlisted user use the router), this check passes for everyone.
5. The non-allowlisted user's swap executes, bypassing the allowlist entirely.

Alternatively, if the router is not allowlisted, all router-mediated swaps revert even for individually allowlisted users, making the router unusable with allowlisted pools.

**Existing guards are insufficient:** The `onlyPool` modifier in `BaseMetricExtension` only verifies the caller is a registered pool — it does not verify the identity of the original economic actor. There is no mechanism in the pool or router to thread the original `msg.sender` through to the extension.

## Impact Explanation
The swap allowlist is a core access-control mechanism. Its bypass allows unprivileged users to trade in pools that pool admins intended to restrict (e.g., permissioned/KYC pools, institutional pools). This constitutes an admin-boundary break: the pool admin's configured restriction is circumvented by any user routing through the public `MetricOmmSimpleRouter`. Depending on pool design, this can result in unauthorized parties executing swaps and extracting value from restricted liquidity pools.

## Likelihood Explanation
The `MetricOmmSimpleRouter` is the primary public entry point for swaps. Any user who knows the pool address can call it. No special privileges, capital, or setup are required beyond having the input token. The bypass is deterministic and repeatable on every swap.

## Recommendation
Pass the original economic actor's address through the call chain. One approach: add a `swapper` field to the swap parameters or use transient storage (already used by the router for callback context) to record `msg.sender` at router entry and expose it to extensions. The pool's `_beforeSwap` call should pass the true originating user, not `msg.sender` of the pool call. Alternatively, the `SwapAllowlistExtension` could accept an `extensionData`-encoded address as the identity to check, with the router injecting `msg.sender` into `extensionData` before forwarding.

## Proof of Concept
```solidity
// 1. Deploy pool with SwapAllowlistExtension
// 2. Allowlist only `allowedUser`
swapExtension.setAllowedToSwap(address(pool), allowedUser, true);

// 3. Non-allowlisted user routes through the public router
// Router calls pool.swap() — pool sees msg.sender = router
// Extension checks allowedSwapper[pool][router] — if router is allowlisted, passes
vm.prank(nonAllowedUser);
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool),
    recipient: nonAllowedUser,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    tokenIn: token0,
    extensionData: ""
}));
// Swap succeeds despite nonAllowedUser not being on the allowlist
```
