### Title
Parallel `spotEngine` and `perpEngine` Addresses Stored Independently in `Endpoint` and `OffchainExchange` May Cause Cross-Contract Desynchronization - (File: `core/contracts/EndpointStorage.sol`, `core/contracts/OffchainExchange.sol`)

---

### Summary

Both `Endpoint` (via `EndpointStorage`) and `OffchainExchange` independently cache the `spotEngine` and `perpEngine` addresses at initialization time. Since these addresses are set separately in each contract with no atomic update mechanism, a migration or engine replacement scenario could leave the two contracts pointing to different engine instances, corrupting balance accounting.

---

### Finding Description

`EndpointStorage.sol` declares and caches `spotEngine` and `perpEngine` as independent storage variables: [1](#0-0) 

These are populated in `Endpoint.initialize()` by querying the clearinghouse: [2](#0-1) 

`OffchainExchange.sol` independently declares and caches the same two addresses: [3](#0-2) 

And populates them in its own `initialize()` with the same clearinghouse query: [4](#0-3) 

Neither contract exposes a post-initialization setter for these addresses. If the protocol ever migrates to new engine contracts (deploying new proxies rather than upgrading implementations), an admin would need to update both `Endpoint` and `OffchainExchange` separately. There is no atomic mechanism to do so.

---

### Impact Explanation

`OffchainExchange` uses its locally cached `spotEngine` and `perpEngine` for all balance mutations in `matchOrders`, `dumpFees`, and `claimBuilderFee`: [5](#0-4) 

`Endpoint` uses its own cached copies for routing and health checks. If the two sets of cached addresses diverge after a migration, `OffchainExchange` would apply balance deltas to stale engine contracts while `Endpoint` routes queries to the new ones. This produces silent accounting corruption: trades settle against one engine while health and collateral checks read from another, potentially allowing undercollateralized positions to pass health checks or blocking valid withdrawals.

---

### Likelihood Explanation

The desynchronization requires an engine migration (new proxy deployment rather than an in-place implementation upgrade). The current upgrade path via `ProxyManager` upgrades implementations in-place, keeping proxy addresses stable, which mitigates the risk under normal operations. However, the structural duplication is a latent hazard for any future migration and is directly analogous to the Isle governor desynchronization pattern. Severity is **Low**, consistent with the reference report.

---

### Recommendation

Remove the `spotEngine` and `perpEngine` storage variables from `EndpointStorage`. Instead, have `Endpoint` and `OffchainExchange` derive these addresses dynamically from the single source of truth — `clearinghouse.getEngineByType()` — at call time, or expose a single authoritative getter on `Clearinghouse` that both contracts call. This mirrors the Isle recommendation of removing the duplicated `Governable` inheritance and reading the governor from a single contract.

---

### Proof of Concept

1. Protocol deploys new `SpotEngine` proxy at address `0xNEW`.
2. Admin calls clearinghouse to register `0xNEW` as the SPOT engine.
3. Admin updates `Endpoint`'s cached `spotEngine` (requires a new setter or re-initialization).
4. Admin forgets to update `OffchainExchange`'s cached `spotEngine`.
5. `OffchainExchange.matchOrders()` calls `spotEngine.updateBalance(...)` on the old engine at `0xOLD`.
6. `Endpoint` reads balances from `0xNEW` for health checks.
7. Balances in `0xOLD` and `0xNEW` are now permanently diverged; users' on-chain balances as seen by the health checker differ from those actually modified by trade settlement. [6](#0-5) [2](#0-1)
