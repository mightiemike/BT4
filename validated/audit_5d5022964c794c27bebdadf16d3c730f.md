Audit Report

## Title
`SwapAllowlistExtension` checks the router address as `sender` instead of the originating EOA, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of `pool.swap()`. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` of `pool.swap()` is the router contract address, not the originating EOA. A pool admin who allowlists the router to support router-based swaps for their curated users inadvertently opens the allowlist to every caller of the router, breaking the curation invariant entirely.

## Finding Description
In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // router address when called via MetricOmmSimpleRouter
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` (L149-177) forwards this value verbatim to `SwapAllowlistExtension.beforeSwap`. The extension then evaluates:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct) and `sender` is the router (wrong). The check resolves to `allowedSwapper[pool][router]`.

In `MetricOmmSimpleRouter.exactInputSingle` (L71-80), the router calls `pool.swap(params.recipient, ...)` with no forwarding of the original caller. The same applies to `exactInput` (L104-112), `exactOutputSingle` (L136-137), and `exactOutput` (L165-181) — all router entry points call `pool.swap()` with `msg.sender = router`.

The exploit path is:
1. Pool admin deploys pool with `SwapAllowlistExtension`.
2. Admin calls `setAllowedToSwap(pool, router, true)` to enable router-based swaps for their allowlisted users.
3. Non-allowlisted EOA `attacker` calls `router.exactInputSingle({pool: pool, recipient: attacker, ...})`.
4. Router calls `pool.swap(attacker, ...)` — `msg.sender` of `pool.swap()` is the router.
5. Pool calls `_beforeSwap(router, ...)`.
6. Extension evaluates `allowedSwapper[pool][router]` → `true` → passes.
7. Swap executes; attacker receives tokens from the curated pool without ever being allowlisted.

No existing guard prevents this: `SwapAllowlistExtension` has no check on `tx.origin`, no router-awareness, and no mechanism to decode the true initiator from `extensionData`.

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict trading to a specific set of addresses loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. The attacker pays no special cost and calls the public router with the target pool address. The pool executes the swap at oracle-derived prices, transferring tokens out of the pool to the attacker's chosen recipient. This constitutes unauthorized access to pool liquidity — a direct loss of user principal and a broken core pool functionality (swap access control).

## Likelihood Explanation
The bypass requires the router to be allowlisted, which is the natural operational state for any curated pool that wants its allowlisted users to use the standard periphery router. The moment the admin adds the router to the allowlist, the allowlist silently becomes open to all users. There is no warning in the extension or the router that this is the consequence. The likelihood is high for any curated pool that supports router-based swaps.

## Recommendation
The extension must check the economically relevant actor — the EOA initiating the transaction — not the intermediary. The cleanest fix is to have `MetricOmmSimpleRouter` encode `msg.sender` (the originating EOA) in `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check it when `sender` is a known intermediary contract (e.g., `sender.code.length > 0`). Alternatively, add a check in the extension that reverts if `sender` is not an EOA (`sender.code.length != 0`) and document that `SwapAllowlistExtension` only enforces correctly on direct `pool.swap()` calls.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `swapExtension.setAllowedToSwap(pool, address(router), true)` to enable router-based swaps.
3. Non-allowlisted EOA `attacker` calls `router.exactInputSingle({pool: pool, recipient: attacker, amountIn: X, ...})`.
4. Router calls `pool.swap(attacker, zeroForOne, amountIn, ...)` — `msg.sender` of `pool.swap()` is the router.
5. Pool calls `_beforeSwap(router, attacker, ...)` → `ExtensionCalling._beforeSwap` forwards `sender=router` to extension.
6. `SwapAllowlistExtension.beforeSwap(sender=router, ...)` evaluates `allowedSwapper[pool][router]` → `true` → no revert.
7. Swap executes; attacker receives tokens from the curated pool despite never being allowlisted.
