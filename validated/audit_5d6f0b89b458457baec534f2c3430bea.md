### Title
`ENABLE_SECP_OPS` Flag Check Bypassed via 4-Byte Opcode Aliases for secp256k1/r1 Verify — (`File: src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` has two independent dispatch paths for secp signature-verification operations. The 1-byte opcode path (opcodes 64 and 65) correctly gates execution behind `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte opcode path (opcodes `0x13d61f00` and `0x1c3a8f00`) dispatches to the identical underlying functions **without any flag check**, allowing an attacker-controlled CLVM program to execute secp operations even when the flag is absent.

---

### Finding Description

`ChiaDialect::op()` in `src/chia_dialect.rs` contains two separate branches that both resolve to `op_secp256k1_verify` and `op_secp256r1_verify`.

**Branch 1 — 1-byte opcodes (lines 248–249): flag-gated correctly**

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

When `ENABLE_SECP_OPS` is absent, opcodes 64 and 65 fall through to `unknown_operator`, which either errors (if `NO_UNKNOWN_OPS` is set) or charges a deterministic unknown-op cost.

**Branch 2 — 4-byte opcodes (lines 175–182): no flag check**

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

No `ENABLE_SECP_OPS` check exists here. An attacker who encodes the operator atom as the 4-byte value `0x13d61f00` or `0x1c3a8f00` bypasses the flag gate entirely and reaches the live secp implementation.

The flag's own documentation states it "Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify)" — it makes no mention of the 4-byte aliases, yet those aliases invoke the same functions.

The structural parallel to the external report is exact: a security gate (`ENABLE_SECP_OPS` / blacklist) is enforced in one code path (1-byte dispatch / `create`) and silently absent in a parallel code path (4-byte dispatch / `migrate`). [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**Consensus divergence / flag-bypass.** The `ENABLE_SECP_OPS` flag is a hard-fork gate. Before the hard-fork activates, nodes run without this flag. A CLVM program submitted with operator atom `0x13d61f00` (4 bytes) will:

- Execute `op_secp256k1_verify` and return `nil` (success) or raise `Secp256Failed` — a real cryptographic result.
- Do so regardless of whether `ENABLE_SECP_OPS` is set.

Meanwhile, the same program written with 1-byte opcode 64 would be rejected as an unknown op (or charged unknown-op cost) under the same flag configuration.

In `MEMPOOL_MODE` — which sets `NO_UNKNOWN_OPS` but **not** `ENABLE_SECP_OPS` — opcode 64 raises `EvalErr::Unimplemented`, but `0x13d61f00` silently executes the full secp verification. This creates a divergence between mempool validation and consensus execution, and between nodes at different hard-fork heights. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

The 4-byte opcode values are documented in the source code itself (the comment at lines 172–174 names the exact byte values and their cost derivation). Any party reading the source can craft a CLVM atom of `\x13\xd6\x1f\x00` or `\x1c\x3a\x8f\x00` and submit it as an operator. No privileged access, no social engineering, and no dependency compromise is required — only attacker-controlled CLVM bytes fed to `run_program`. [6](#0-5) 

---

### Recommendation

Add the `ENABLE_SECP_OPS` guard to the 4-byte dispatch branch, mirroring the 1-byte branch:

```rust
// in the op_len == 4 branch
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures both dispatch paths enforce the same security gate, eliminating the bypass. [7](#0-6)

```rust
pub fn op_secp256k1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256K1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256k1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256k1_verify pubkey")?;
    let verifier = K1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256k1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is message
    let msg = atom(a, msg, "secp256k1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256k1_verify sig")?;
    let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
```
