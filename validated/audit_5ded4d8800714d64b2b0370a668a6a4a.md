Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the originating user, allowing any user to bypass the swap allowlist via the router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed from the pool. However, the pool always passes `msg.sender` (the immediate caller) as `sender`, which is the router contract when a user swaps through `MetricOmmSimpleRouter`. If the router is allowlisted — which is required for any router-mediated swap to succeed — every unprivileged user can bypass the per-user allowlist simply by routing through the public router.

## Finding Description

**Root cause — wrong actor propagated as `sender`:**

In `MetricOmmPool.swap`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
    msg.sender,   // ← always the immediate caller (router, not the user)
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension:

```solidity
// ExtensionCalling.sol line 160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(
        IMetricOmmExtensions.beforeSwap,
        (sender, recipient, ...)  // sender = router address
    )
);
```

`SwapAllowlistExtension.beforeSwap` then checks this `sender` against the per-pool allowlist, using `msg.sender` (the pool) as the pool key:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly:

```solidity
// MetricOmmSimpleRouter.sol line 72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The pool sees `msg.sender = router`. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Exploit flow:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a specific set of addresses.
2. Pool admin allowlists the router (`setAllowedToSwap(pool, router, true)`) so that allowlisted users can swap through the router.
3. Any non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the restricted pool.
4. The router calls `pool.swap(...)` — pool sees `msg.sender = router`.
5. Extension checks `allowedSwapper[pool][router]` → `true`.
6. Swap succeeds; the non-allowlisted user has bypassed the allowlist entirely.

**Existing guards are insufficient:** The only guard is the `allowedSwapper` mapping keyed by `sender`. Since `sender` is the router (not the user), the check is structurally wrong for any router-mediated path. There is no secondary check on the originating user anywhere in the pool or extension call chain.

The multi-hop `exactOutput` path compounds this: inside `_exactOutputIterateCallback`, intermediate hops call `pool.swap(msg.sender, ...)` where `msg.sender` is the previous pool, so `sender` seen by the extension is a pool address — making the bypass even more trivially achievable.

## Impact Explanation
The swap allowlist is a core access-control mechanism. Its bypass allows any unprivileged user to execute swaps in pools that pool admins intended to restrict (e.g., KYC-gated, institutional, or compliance-restricted pools). This constitutes a broken core pool functionality and an admin-boundary break: the pool admin's allowlist configuration is rendered ineffective by any user routing through the public router. Unauthorized swaps can drain pool liquidity or violate regulatory/compliance constraints, causing direct loss of user principal or protocol integrity.

## Likelihood Explanation
The exploit requires only that the router be allowlisted for the target pool — a necessary condition for any router-mediated swap to work for legitimate users. Any unprivileged user can trigger it by calling the public `exactInputSingle` or `exactInput` functions on `MetricOmmSimpleRouter`. No special privileges, flash loans, or complex setup are needed. The condition is repeatable on every block.

## Recommendation
The pool should pass the originating user's address — not `msg.sender` — as `sender` to extensions. One approach: add an explicit `sender` parameter to `MetricOmmPool.swap` (analogous to how `addLiquidity` separates `msg.sender` payer from `owner`), and have the router pass `msg.sender` (the user) in that field. Alternatively, `SwapAllowlistExtension` could accept an encoded user address from `extensionData`, but this is weaker because it is caller-supplied. The cleanest fix is to thread the true originating address through the pool's `swap` signature so extensions always see the economically relevant actor.

## Proof of Concept

```solidity
// Foundry test sketch
function test_routerBypassesSwapAllowlist() public {
    // 1. Deploy pool with SwapAllowlistExtension
    // 2. Allowlist the router (required for router swaps to work)
    swapExtension.setAllowedToSwap(address(pool), address(router), true);
    // 3. Do NOT allowlist the attacker
    address attacker = makeAddr("attacker");
    assertFalse(swapExtension.isAllowedToSwap(address(pool), attacker));

    // 4. Attacker swaps through the router — should revert, but succeeds
    vm.startPrank(attacker);
    token0.approve(address(router), type(uint256).max);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token0),
        recipient: attacker,
        amountIn: 1000,
        amountOutMinimum: 0,
        zeroForOne: true,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    }));
    vm.stopPrank();
    // Swap succeeds — allowlist bypassed
}
```
