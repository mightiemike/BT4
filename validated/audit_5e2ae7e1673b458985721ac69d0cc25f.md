### Title
Bonus Reward Checkpoints and Event Reference Counts Are Permanently Advanced When `ErrInsufficientBonusPool` Is Swallowed in `slashRedelegationPosition` — (File: `x/tieredrewards/keeper/slash.go`)

---

### Summary

In `processEventsAndClaimBonus`, event reference counts are decremented and position checkpoints (`LastEventSeq`, `LastBonusAccrual`, `LastKnownBonded`) are advanced **before** the bonus pool sufficiency check. When `ErrInsufficientBonusPool` is returned, those store writes are already committed. In `slashRedelegationPosition`, this error is explicitly swallowed and execution continues, persisting the advanced checkpoints via `setPositionWithState`. The position owner permanently loses all accrued bonus rewards for the consumed events.

---

### Finding Description

**Root cause — premature checkpoint advancement before payment confirmation:**

In `processEventsAndClaimBonus` (`x/tieredrewards/keeper/claim_rewards.go`), the event-processing loop decrements each event's reference count and advances `pos.LastEventSeq` on every iteration, **before** the pool balance check: [1](#0-0) 

After the loop, `applyBonusAccrualCheckpoint` and `UpdateLastKnownBonded` advance the remaining in-memory checkpoints: [2](#0-1) 

Only then does the pool sufficiency check run. If it fails, the function returns `ErrInsufficientBonusPool` — but all reference-count decrements and checkpoint updates have already been applied: [3](#0-2) 

**Root cause — error swallowed in the slash hook path:**

`slashRedelegationPosition` explicitly catches `ErrInsufficientBonusPool` and continues execution with the already-mutated `pos`: [4](#0-3) 

It then calls `setPositionWithState` with the modified position, persisting the advanced checkpoints: [5](#0-4) 

**Broken invariant:** Event reference counts must only be decremented when the corresponding bonus is actually paid. If payment fails, the events must not be consumed. Here, events are consumed (refcounts decremented, events potentially deleted when refcount reaches zero) and checkpoints are advanced, but no bonus is paid. The user's accrued rewards for those events are permanently unrecoverable.

---

### Impact Explanation

The exact corrupted values are:
- **`pos.LastEventSeq`** — advanced past events that were never paid for; future claims skip those events entirely.
- **`pos.LastBonusAccrual`** — advanced to current block time; the time window for the unpaid rewards is permanently closed.
- **`ValidatorEvent.ReferenceCount`** — decremented (and events deleted at zero) without corresponding payment; the on-chain record of the reward-bearing segment is destroyed.

A position owner whose validator is slashed while the bonus pool is empty or insufficient permanently loses all bonus rewards accrued since their last claim. There is no recovery path: the events are gone, the checkpoints are advanced, and no retry is possible.

---

### Likelihood Explanation

The trigger requires two concurrent conditions:
1. A validator with tier positions is slashed (a normal protocol event).
2. The `RewardsPoolName` module account has insufficient balance to cover the accrued bonus at the moment of the slash.

The bonus pool is a shared resource drained by all position owners claiming rewards. It can legitimately reach zero or near-zero through normal claiming activity, especially in periods of high claim volume or low pool funding. Neither condition requires attacker privilege: any delegator can drain the pool by claiming their own rewards, and validator slashing is a standard protocol event.

---

### Recommendation

Move all persistent state mutations (reference count decrements, checkpoint updates) to **after** the payment is confirmed. Specifically:

1. Collect the list of events to process and compute `totalBonus` without touching the store.
2. Check `sufficientBonusPoolBalance` and send coins.
3. Only if payment succeeds, decrement reference counts and persist the updated checkpoints.

Alternatively, perform the pool check before entering the event loop, and only proceed with mutations if the pool is sufficient.

---

### Proof of Concept

1. Alice locks tokens in a tier position on Validator V. Bonus accrues over time.
2. The `RewardsPoolName` module account balance drops to zero (e.g., other users claim all available bonus).
3. Validator V is slashed. The Cosmos SDK staking module fires `BeforeRedelegationSlashed`, which calls `slashRedelegationPosition`.
4. Inside `slashRedelegationPosition`, `processEventsAndClaimBonus` is called. The event loop runs: for each validator event since Alice's `LastEventSeq`, `decrementEventRefCount` is called (store write committed), and `pos.LastEventSeq` is updated. After the loop, `applyBonusAccrualCheckpoint` and `UpdateLastKnownBonded` are applied.
5. `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`. No coins are sent.
6. Back in `slashRedelegationPosition`, the error is caught and swallowed. `setPositionWithState` is called with the mutated `pos`.
7. Alice's position now has `LastEventSeq` pointing past all consumed events, `LastBonusAccrual` set to the current block time, and the validator events deleted (refcount hit zero). Her accrued bonus is permanently lost.
8. When the pool is later refunded and Alice calls `ClaimTierRewards`, `processEventsAndClaimBonus` finds no events to process (they were deleted) and the time window is closed — zero bonus is paid.
