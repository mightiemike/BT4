### Title
`ValidResourceBounds` Protobuf Deserialization Misclassifies `AllResources` as `L1Gas`, Producing a Divergent Transaction Hash - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

When a V3 invoke transaction is submitted with `AllResources` bounds where both `l2_gas` and `l1_data_gas` are zero (but `l1_gas` is non-zero), the gateway computes its hash using the `AllResources` variant — which includes a third `l1_data_gas` felt in the Poseidon preimage. After the transaction is included in a block and propagated via P2P protobuf, the receiving node's protobuf deserializer reclassifies the same bounds as `ValidResourceBounds::L1Gas`, causing `get_tip_resource_bounds_hash` to omit the `l1_data_gas` felt entirely. The two hash computations diverge, so the receiving node rejects the block as containing an invalid transaction hash.

---

### Finding Description

**Step 1 — Attacker-controlled input that passes gateway admission**

`RpcInvokeTransactionV3.resource_bounds` is typed as `AllResourceBounds` (always three fields). A user submits:

```
AllResourceBounds { l1_gas: { max_amount: N, max_price_per_unit: P }, l2_gas: zero, l1_data_gas: zero }
```

The gateway's `validate_resource_bounds` only checks that `max_possible_fee(Tip::ZERO) != 0`. With non-zero `l1_gas`, this passes.

**Step 2 — Hash computed at the gateway uses `AllResources`**

`InternalRpcInvokeTransactionV3::resource_bounds()` unconditionally returns `ValidResourceBounds::AllResources(self.resource_bounds)`:

```rust
// crates/starknet_api/src/rpc_transaction.rs:637-638
fn resource_bounds(&self) -> ValidResourceBounds {
    ValidResourceBounds::AllResources(self.resource_bounds)
}
```

`get_tip_resource_bounds_hash` is then called with `AllResources`, which appends a third felt for `l1_data_gas` (value zero) to the Poseidon preimage:

```rust
// crates/starknet_api/src/transaction_hash.rs:203-208
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 felts total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 felts total
    }
});
```

The resulting hash **H1** is stored in `InternalRpcTransaction.tx_hash` and committed to the block.

**Step 3 — Protobuf serialization preserves the zero fields**

`From<ValidResourceBounds> for protobuf::ResourceBounds` serializes `AllResources` with explicit zero `l1_data_gas`:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs:479-487
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(l2_gas.into()),          // zero
        l1_data_gas: Some(l1_data_gas.into()), // zero
    },
```

**Step 4 — Protobuf deserialization on the receiving node misclassifies the variant**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies a heuristic: if both `l2_gas` and `l1_data_gas` are zero, it returns `L1Gas`:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs:431-435
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong variant
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

**Step 5 — Hash recomputed on the receiving node uses `L1Gas` → different preimage**

With `ValidResourceBounds::L1Gas`, `get_tip_resource_bounds_hash` produces only two resource felts (no `l1_data_gas` felt). The resulting hash **H2 ≠ H1**. Any hash validation against the stored H1 fails, and the block is rejected.

---

### Impact Explanation

The transaction hash is the canonical identifier used for signature verification, block commitment, and receipt lookup. When the hash computed at the sequencer (gateway path, always `AllResources`) diverges from the hash recomputed by a syncing peer (P2P path, misclassified as `L1Gas`), the syncing node rejects a legitimately sequenced block. This constitutes **wrong hash logic that binds the wrong executable payload** and causes valid blocks to be rejected by syncing nodes — matching the "High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload" impact.

---

### Likelihood Explanation

The trigger requires only a standard V3 invoke transaction with `l2_gas = 0` and `l1_data_gas = 0` (only `l1_gas` non-zero). This is a valid pre-0.13.3 style transaction that passes all gateway checks. Any user can craft this transaction without special privileges. The divergence is deterministic and reproducible for every such transaction included in a block.

---

### Recommendation

Remove the heuristic in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. The variant should be determined by an explicit tag in the protobuf message, not inferred from whether the zero-valued fields are present. Until a tag is added, the deserializer should always produce `AllResources` when all three fields are present (even if zero), reserving `L1Gas` only for messages that are missing `l1_data_gas` entirely (the legacy 0.13.2 case, already handled by `unwrap_or_default`). Concretely:

```rust
// Proposed fix: only fall back to L1Gas when l1_data_gas was absent (legacy path)
let l1_data_gas_opt = value.l1_data_gas;
let l1_data_gas: ResourceBounds = l1_data_gas_opt.unwrap_or_default().try_into()?;
Ok(if l1_data_gas_opt.is_none() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

### Proof of Concept

1. Submit a V3 invoke transaction to the gateway with:
   ```json
   "resource_bounds": {
     "l1_gas": { "max_amount": "0x100", "max_price_per_unit": "0x1" },
     "l2_gas": { "max_amount": "0x0",   "max_price_per_unit": "0x0" },
     "l1_data_gas": { "max_amount": "0x0", "max_price_per_unit": "0x0" }
   }
   ```
2. Observe that the gateway accepts the transaction and computes hash H1 using `AllResources` (3-felt preimage including `l1_data_gas = 0`).
3. Let the transaction be included in a block and propagated via P2P protobuf.
4. On the receiving node, add a log in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` to confirm it returns `L1Gas`.
5. Observe that the receiving node recomputes hash H2 using `L1Gas` (2-felt preimage, no `l1_data_gas`).
6. Confirm H1 ≠ H2 by comparing the two Poseidon hashes:
   - H1 = `poseidon([tip, pack(l1_gas), pack(l2_gas=0), pack(l1_data_gas=0)])`
   - H2 = `poseidon([tip, pack(l1_gas), pack(l2_gas=0)])`
7. The receiving node rejects the block due to hash mismatch.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
