### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the 64-Byte Transaction Forgery Protection Added in v2 — (`contract/src/lib.rs`)

### Summary

The contract introduced `verify_transaction_inclusion_v2` specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability (CVE-2012-2459) by requiring a coinbase Merkle proof of equal length. However, the original `verify_transaction_inclusion` function was only marked `#[deprecated]` — it was not removed or access-restricted. In Rust, `#[deprecated]` is a compiler lint warning only; it does not prevent on-chain callers from invoking the method. Any unprivileged NEAR caller can bypass the coinbase-proof protection entirely by calling the v1 endpoint directly, forging transaction inclusion for a transaction that never existed in a Bitcoin block.

### Finding Description

The protection mechanism lives exclusively in `verify_transaction_inclusion_v2`:

```rust
// contract/src/lib.rs:347-368
pub fn verify_transaction_inclusion_v2(&self, args: ProofArgsV2) -> bool {
    require!(
        args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
        "Coinbase merkle proof and transaction merkle proof should have the same length"
    );
    // ... coinbase root check ...
    require!(
        merkle_tools::compute_root_from_merkle_proof(
            args.coinbase_tx_id.clone(), 0usize, &args.coinbase_merkle_proof,
        ) == header.block_header.merkle_root,
        "Incorrect coinbase merkle proof"
    );
    self.verify_transaction_inclusion(args.into())   // delegates to v1 after guard
}
```

The v1 function, however, remains `pub` and `#[pause]`-gated (not access-controlled beyond the global pause):

```rust
// contract/src/lib.rs:283-323
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, args: ProofArgs) -> bool {
    // ... confirmations check, mainchain membership check ...
    require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
    merkle_tools::compute_root_from_merkle_proof(
        args.tx_id,
        usize::try_from(args.tx_index).unwrap(),
        &args.merkle_proof,
    ) == header.block_header.merkle_root
    // NO coinbase proof check whatsoever
}
```

The `#[deprecated]` attribute emits a Rust compiler warning to crate-internal callers only. It has zero effect on external NEAR contract calls. The NEAR runtime dispatches calls by method name string; any caller can invoke `verify_transaction_inclusion` directly, skipping the coinbase guard entirely.

The 64-byte forgery works because `compute_hash` in `merkle-tools` concatenates two 32-byte child hashes into a 64-byte buffer and double-SHA256s them — the same operation performed on a raw 64-byte transaction:

```rust
// merkle-tools/src/lib.rs:54-60
fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);
    double_sha256(&concat_inputs)
}
```

There is no domain separation between leaf nodes and internal nodes. An attacker who supplies an internal node hash as `tx_id` with a shortened Merkle path that terminates one level above the true leaf level will produce a root that matches the stored `merkle_root`, causing `verify_transaction_inclusion` to return `true`.

### Impact Explanation

The fundamental invariant of the light client — that `verify_transaction_inclusion` returns `true` only for transactions genuinely included in a Bitcoin block — is broken. Any downstream NEAR DApp that calls `verify_transaction_inclusion` (rather than v2) to gate asset releases, cross-chain bridge settlements, or payment confirmations can be deceived into accepting a forged proof. The attacker does not need to mine a Bitcoin block or control any privileged key; they only need to identify a suitable internal node in any existing block's Merkle tree and craft the corresponding 64-byte raw transaction preimage.

### Likelihood Explanation

The entry path is fully unprivileged: any NEAR account can call `verify_transaction_inclusion` by name. The 64-byte transaction forgery technique is publicly documented (bitmex.com/blog/64-Byte-Transactions, referenced in the contract's own doc comment at line 267). The contract's own warning at line 278 acknowledges the risk. Downstream DApps that integrated before v2 was introduced, or that call the v1 method for any reason, are immediately exploitable. The attacker's only cost is identifying a suitable internal node in a real Bitcoin block's Merkle tree, which is trivially achievable offline.

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or restrict it to an internal (`fn`, not `pub fn`) helper. Since `verify_transaction_inclusion_v2` already delegates to it internally via `#[allow(deprecated)]`, the v1 logic can remain as a private function. The public-facing method name must be eliminated from the NEAR contract interface so the NEAR runtime cannot dispatch calls to it.

### Proof of Concept

1. Select any confirmed Bitcoin block whose Merkle tree has depth ≥ 2 (i.e., more than one transaction).
2. Identify two adjacent leaf hashes `L` and `R` at depth 1. Their parent internal node hash `P = double_sha256(L || R)` is a valid internal node.
3. Construct a 64-byte raw transaction whose `txid = P` (the attacker crafts a transaction whose serialized form is exactly `L || R`).
4. Build a Merkle proof path from `P` up to the Merkle root (one level shorter than a real leaf proof).
5. Call `verify_transaction_inclusion` (v1) on the NEAR contract with:
   - `tx_id = P`
   - `tx_block_blockhash` = the real block hash (already in the contract's mainchain)
   - `tx_index` = the index corresponding to the internal node's position
   - `merkle_proof` = the shortened path from `P` to the root
   - `confirmations = 1`
6. The contract computes `compute_root_from_merkle_proof(P, index, proof)` which equals the real `merkle_root` stored in the block header, and returns `true` — despite the transaction never existing on Bitcoin.

The v2 guard is never reached because the attacker called v1 directly, exactly as the DYAD attacker bypassed `idToBlockOfLastDeposit` by using liquidations instead of deposits.

---

**Structural analog to H-10:** In DYAD, the flash-loan protection (`idToBlockOfLastDeposit`) was bypassed by using the `liquidate → move` code path, which transferred funds without updating the guard variable. Here, the 64-byte forgery protection (coinbase proof check in `verify_transaction_inclusion_v2`) is bypassed by calling `verify_transaction_inclusion` directly — an alternative code path that transfers the same verification responsibility without enforcing the guard. In both cases, a security control added to a "v2" path is rendered ineffective because the unguarded "v1" path remains reachable. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
