Audit Report

## Title
Extension registered in `extensions` array but not wired to any hook order silently bypasses configured guard — (`metric-core/contracts/libraries/ValidateExtensionsConfig.sol`)

## Summary

`validateExtensionsConfig` enforces that at least one order field is non-zero and that each order field only references valid indices, but it never verifies that every registered extension index (1 to `extensionCount`) appears in at least one order field. An extension can be registered, initialized by the factory, and bound to the pool while remaining permanently unreachable by any hook dispatch, silently voiding the access control it was meant to enforce.

## Finding Description

`validateExtensionsConfig` performs the following checks on the `orders` struct:

- `_ordersAreZero(orders)` must be false — at least one order field is non-zero [1](#0-0) 
- `_validateExtensionOrder` on each of the six fields — each referenced index must be in `[1, extensionCount]` with no intra-field duplicates [2](#0-1) 

`_validateExtensionOrder` returns immediately when `order == 0`, so an order field of zero is always accepted: [3](#0-2) 

There is no cross-check that every index from 1 to `extensions.length` appears in at least one of the six order fields. The following config passes validation without revert:

```
extensions        = [depositAllowlist, swapAllowlist]   // indices 1 and 2
orders.beforeSwap = encode(2)                           // only index 2 wired
// orders.beforeAddLiquidity == 0 — index 1 never wired
```

`_ordersAreZero` returns `false` (one field is non-zero), and `_validateExtensionOrder(encode(2), 2)` passes (index 2 ≤ extensionCount). Validation succeeds.

After validation, the factory calls `initialize` on **every** extension in the array: [4](#0-3) 

So `depositAllowlist.initialize(pool, ...)` is called and the extension is bound to the pool. However, `BEFORE_ADD_LIQUIDITY_ORDER` is stored as `0`, and `_callExtensionsInOrder` returns immediately when `order == 0`: [5](#0-4) 

`_beforeAddLiquidity` dispatches through `BEFORE_ADD_LIQUIDITY_ORDER`, which is `0`, so `depositAllowlist.beforeAddLiquidity` is never invoked on any pool action: [6](#0-5) 

The pool's on-chain state shows the extension initialized and bound, but the guard is permanently inert.

## Impact Explanation

For a `DepositAllowlistExtension` left unwired from `beforeAddLiquidity`: any address can call `addLiquidity`, minting LP shares in a pool intended to be curated. Unauthorized parties earn fees and dilute the principal of existing curated LPs — a direct loss of LP assets above Sherlock thresholds.

For a `SwapAllowlistExtension` left unwired from `beforeSwap`: any address can swap against LP positions without restriction, directly extracting value from LP principal through trades the pool was configured to block.

Both outcomes are direct fund-impacting consequences matching the allowed impact gate (broken core pool functionality causing loss of funds; direct loss of LP assets).

## Likelihood Explanation

`createPool` is fully permissionless — any caller supplies the `extensions` array and `extensionOrders` struct. A pool creator who intends to gate deposits but omits `orders.beforeAddLiquidity` (or wires the extension only to an unrelated hook slot) deploys a pool with no effective deposit guard. The validation emits no signal that the registered extension is inert. Because the factory initializes every extension regardless of wiring, the pool's on-chain state appears fully configured. This mistake is easy to make and undetectable from on-chain state alone.

## Recommendation

Add a cross-check inside `validateExtensionsConfig` that every extension index from 1 to `extensionCount` appears in at least one order field:

```solidity
uint256 referenced;
referenced |= _collectBits(orders.beforeAddLiquidity);
referenced |= _collectBits(orders.afterAddLiquidity);
referenced |= _collectBits(orders.beforeRemoveLiquidity);
referenced |= _collectBits(orders.afterRemoveLiquidity);
referenced |= _collectBits(orders.beforeSwap);
referenced |= _collectBits(orders.afterSwap);

for (uint256 i = 1; i <= extensions.length; i++) {
    if (referenced & (1 << i) == 0) revert InvalidExtensionsConfig();
}

function _collectBits(uint256 order) private pure returns (uint256 bits) {
    while (order != 0) {
        bits |= (1 << (order & 0x7));
        order >>= 3;
    }
}
```

## Proof of Concept

```solidity
function test_extensionRegisteredButNotWiredToAnyHook() public {
    address[] memory extensions = new address[](2);
    extensions[0] = address(depositAllowlist);   // index 1
    extensions[1] = address(swapAllowlist);      // index 2
    bytes[] memory initData = new bytes[](2);

    ExtensionOrders memory orders;
    // Only beforeSwap references index 2; beforeAddLiquidity is zero.
    orders.beforeSwap = ExtensionOrderTestLib.encodeExtensionOrder(2, 0, 0, 0, 0, 0, 0);

    // Passes validation — no revert despite depositAllowlist being unwired.
    harness.validateExtensionsConfig(extensions, orders, initData);

    // Deploy pool; depositAllowlist.initialize() is called but BEFORE_ADD_LIQUIDITY_ORDER == 0.
    address pool = factory.createPool(buildParams(extensions, orders, initData));

    // Any address can addLiquidity without restriction.
    vm.prank(unauthorizedDepositor);
    pool.addLiquidity(unauthorizedDepositor, 0, delta, "");
    // succeeds — DepositAllowlistExtension.beforeAddLiquidity never ran
}
```
