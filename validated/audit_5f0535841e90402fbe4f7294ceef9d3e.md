### Title
Secp256k1/r1 Verify Operators Bypass `ENABLE_SECP_OPS` Flag via 4-Byte Opcode Form — (File: `src/chia_dialect.rs`)

---

### Summary

The 4-byte opcode forms of `secp256k1_verify` (`0x13d61f00`) and `secp256r1_verify` (`0x1c3a8f00`) are dispatched directly to the actual cryptographic verification functions without checking `ENABLE_SECP_OPS`. The 1-byte opcode forms (64 and 65) correctly gate on `ENABLE_SECP_OPS`. This dual-path wiring error allows attacker-controlled CLVM bytes to invoke live secp signature verification in any execution context — including `MEMPOOL_MODE` — where secp ops are supposed to be disabled.

---

### Finding Description

In `src/chia_dialect.rs`, the `op()` method of `ChiaDialect` has two separate dispatch paths for secp operations:

**Path 1 — 4-byte opcodes (lines 157–183):** The code enters the `op_len == 4` branch, which is commented as handling *"unknown operators with assigned cost."* Two specific 4-byte values are matched and dispatched to the actual secp functions with no flag check:

```rust
// lines 175–182
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
``` [1](#0-0) 

**Path 2 — 1-byte opcodes (lines 248–249):** The same functions are dispatched only when `ENABLE_SECP_OPS` is set:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The `ENABLE_SECP_OPS` flag is defined to gate secp opcodes:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
``` [3](#0-2) 

`MEMPOOL_MODE` deliberately omits `ENABLE_SECP_OPS`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [4](#0-3) 

The comment on the 4-byte branch says these are *"unknown operators with assigned cost,"* meaning the design intent is that they should behave like `op_unknown` — a no-op returning nil with a fixed cost. Instead, they are wired to the live cryptographic functions `op_secp256k1_verify` and `op_secp256r1_verify` in `src/secp_ops.rs`, which perform real ECDSA verification and raise `EvalErr::Secp256Failed` on invalid signatures. [5](#0-4) [6](#0-5) 

The costs were engineered to match the unknown-op formula (`multiplier + 1` with cost_function 0): `0x13d61f + 1 = 1300000 = SECP256K1_VERIFY_COST` and `0x1c3a8f + 1 = 1850000 = SECP256R1_VERIFY_COST`. This confirms the 4-byte opcodes were intended to be cost-equivalent no-ops, not live verifiers. [7](#0-6) 

---

### Impact Explanation

**Behavioral divergence between the two opcode forms:**

| Context | 1-byte opcode 64/65 (no `ENABLE_SECP_OPS`) | 4-byte opcode `0x13d61f00`/`0x1c3a8f00` |
|---|---|---|
| Consensus mode | `op_unknown` → nil, fixed cost (no-op) | Live secp verify → nil or `Secp256Failed` |
| Mempool mode (`NO_UNKNOWN_OPS`) | `unknown_operator` → `EvalErr::Unimplemented` (FAIL) | Live secp verify → nil or `Secp256Failed` |

In **mempool mode**, a program using the 4-byte form with a valid secp signature **succeeds** (returns nil), while the same logical operation using the 1-byte form **fails** with an unimplemented-op error. This is a direct mempool validation bypass: an attacker can craft CLVM bytes that pass mempool evaluation using secp signature authorization when secp ops are supposed to be entirely disabled.

In **consensus mode** without `ENABLE_SECP_OPS`, the 4-byte form raises `Secp256Failed` on an invalid signature, while the 1-byte form silently returns nil. A coin puzzle using the 4-byte secp opcode with an invalid/attacker-chosen signature would be rejected by nodes running this code but accepted as a no-op by nodes treating it as a true unknown op — a consensus split.

The corrupted result is the `Response` returned by `ChiaDialect::op()`: it is `Ok(Reduction(cost, nil))` or `Err(Secp256Failed)` instead of the correct `Ok(Reduction(cost, nil))` (no-op) that the flag model requires.

---

### Likelihood Explanation

The entry path is direct: any caller submitting CLVM bytes with a 4-byte atom opcode `0x13d61f00` or `0x1c3a8f00` as the operator triggers this path. No special privileges are required. The attacker only needs to serialize a CLVM program using the 4-byte opcode form, which is straightforward given the public serialization format. The divergence is deterministic and reproducible.

---

### Recommendation

Either:
1. Apply the same `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch, mirroring the 1-byte form:
   ```rust
   0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
   0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
   ```
   And fall through to `unknown_operator` otherwise.

2. Or, if the 4-byte opcodes are intended to always be no-ops with assigned cost, remove them from the named dispatch table entirely and let them fall through to `unknown_operator`, which will compute the correct cost from the opcode bytes.

---

### Proof of Concept

Craft a CLVM program whose operator is the 4-byte atom `\x13\xd6\x1f\x00` with three arguments: a valid secp256k1 pubkey, a 32-byte message digest, and a valid signature. Run it under `MEMPOOL_MODE` (which sets `NO_UNKNOWN_OPS` and omits `ENABLE_SECP_OPS`).

**Expected** (per flag model): `EvalErr::Unimplemented` — secp ops are not enabled, and unknown ops are forbidden.

**Actual**: `Ok(Reduction(1300000, nil))` — the 4-byte dispatch bypasses the flag check and executes live secp256k1 verification, returning nil because the signature is valid.

The same program with the 1-byte opcode `\x40` (64) as operator under the same flags returns `EvalErr::Unimplemented`, confirming the divergence between the two dispatch paths for the same underlying operation. [8](#0-7) [9](#0-8)

pub fn op_secp256r1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256R1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256r1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256r1_verify pubkey")?;
    let verifier = P1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256r1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is sha256 hash of message
    let msg = atom(a, msg, "secp256r1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256r1_verify sig")?;
    let sig = P1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
