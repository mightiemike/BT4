Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` keys on router address instead of end-user, enabling allowlist bypass or blocking authorized router users - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps using the `sender` parameter, which is `msg.sender` of the pool's `swap()` call. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the actual end-user. This creates two mutually exclusive broken states: allowlisting the router bypasses per-user restrictions for all callers, while allowlisting only individual users blocks them from using the standard router path entirely.

## Finding Description

In `MetricOmmPool.sol::swap()`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

This `sender` is forwarded through `ExtensionCalling._beforeSwap` directly to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle()` is used, the router calls `pool.swap()` directly, making the router the `msg.sender` inside the pool: [4](#0-3) 

The same wrong-actor binding applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

The extension therefore evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`. By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates on `owner` — a separate parameter explicitly distinct from `sender` — which is stable regardless of the intermediary: [6](#0-5) 

The swap interface has no equivalent stable identity parameter for the true end-user; `sender` is the only actor identity available, and it changes based on whether the user calls the pool directly or through the router.

## Impact Explanation

A pool admin deploying a curated pool with `SwapAllowlistExtension` (e.g., for KYC enforcement) faces two fund-impacting broken states:

1. **Allowlist bypass (High):** If the admin allowlists the router address to enable router-based swaps, every user — including those never individually authorized — can trade in the curated pool via `MetricOmmSimpleRouter`. Unauthorized parties receive token output from the pool, directly violating the curation policy with fund-impacting consequences.

2. **Router path blocked (Medium):** If the admin allowlists only individual users, those users cannot swap through `MetricOmmSimpleRouter` at all despite being authorized. They must implement `IMetricOmmSwapCallback` themselves and call `pool.swap()` directly, which is not the standard supported flow.

Both states constitute broken core pool functionality. Case (1) is a direct policy bypass where unauthorized users extract tokens from a pool configured to exclude them.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary supported periphery entry point for swaps. Any user of a curated pool who uses the router — the expected and documented path — triggers this misbinding. The pool admin has no configuration option that correctly enforces per-user restrictions while also permitting router-based swaps, because the extension has no access to the true originating user address. The condition is reachable by any unprivileged trader using the standard periphery path.

## Recommendation

`SwapAllowlistExtension` must identify the true end-user rather than the intermediary router. Two viable approaches:

1. **`extensionData` forwarding:** Have `MetricOmmSimpleRouter` encode `msg.sender` (the end-user) into `extensionData` for each hop. `SwapAllowlistExtension.beforeSwap` decodes and checks that address instead of `sender`. When `extensionData` is empty (direct pool call), fall back to checking `sender`.

2. **Dual-check:** Check both `sender` and a user address decoded from `extensionData`, accepting the swap if either is allowlisted and `extensionData` is non-empty.

## Proof of Concept

**Scenario: Allowlist bypass via router**

```
Setup:
  - Pool deployed with SwapAllowlistExtension
  - Admin calls: swapExtension.setAllowedToSwap(pool, router, true)
    (intending to allow router-based swaps for allowlisted users)

Attack:
  - Mallory (not individually allowlisted) calls:
      router.exactInputSingle({pool: pool, recipient: mallory, ...})
  - Router calls: pool.swap(mallory, ...)  [router is msg.sender]
  - Pool calls: extension.beforeSwap(router, mallory, ...)
  - Extension checks: allowedSwapper[pool][router] == true  ✓
  - Swap executes — Mallory receives tokens from the curated pool
    despite never being individually authorized.

Result:
  - Per-user allowlist completely bypassed for all router users.
```

**Scenario: Authorized user blocked via router**

```
Setup:
  - Pool deployed with SwapAllowlistExtension
  - Admin calls: swapExtension.setAllowedToSwap(pool, alice, true)

Alice attempts to swap via router:
  - router.exactInputSingle({pool: pool, recipient: alice, ...})
  - Router calls: pool.swap(alice, ...)  [router is msg.sender]
  - Pool calls: extension.beforeSwap(router, alice, ...)
  - Extension checks: allowedSwapper[pool][router] == false  ✗
  - Revert: NotAllowedToSwap()

Result:
  - Alice cannot use the standard periphery path despite being allowlisted.
```

A Foundry integration test deploying a pool with `SwapAllowlistExtension`, allowlisting the router, and asserting that an unallowlisted EOA successfully swaps via `exactInputSingle` reproduces case (1) directly.
