### Title
Fee-on-Transfer Token Deposit Overcredits User Balances, Causing Last Withdrawers to Lose Funds — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.depositCollateral` credits users with the caller-specified `txn.amount` without verifying the actual token amount received by the contract. For fee-on-transfer (deflationary) tokens, this creates a growing accounting discrepancy where the protocol's internal deposit ledger exceeds the actual token balance held. The first withdrawers receive their full credited amount; later withdrawers cannot, because the real token balance is exhausted before the accounting says it should be.

---

### Finding Description

In `Clearinghouse.depositCollateral`, the credited amount is derived directly from the transaction parameter:

```solidity
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
``` [1](#0-0) 

`txn.amount` is the value the user signed in their off-chain transaction. For standard ERC20 tokens this equals the actual received amount. For a fee-on-transfer token, the Clearinghouse physically receives `txn.amount × (1 − fee)`, but the user is credited the full `txn.amount`. The gap is never reconciled.

The only on-chain solvency guard is `assertUtilization`, called after every withdrawal:

```solidity
int128 totalDeposits = _state.totalDepositsNormalized.mul(
    _state.cumulativeDepositsMultiplierX18
);
int128 totalBorrows = _state.totalBorrowsNormalized.mul(
    _state.cumulativeBorrowsMultiplierX18
);
require(totalDeposits >= totalBorrows, ERR_MAX_UTILIZATION);
``` [2](#0-1) 

This check only verifies internal accounting consistency (`deposits ≥ borrows`). It does **not** compare `totalDeposits − totalBorrows` against `token.balanceOf(clearinghouse)`. For a product with no borrows the check is trivially `totalDeposits ≥ 0` and always passes, regardless of how far the actual balance has drifted below the credited total.

The withdrawal path transfers tokens out of the Clearinghouse to the `WithdrawPool`:

```solidity
token.safeTransfer(withdrawPool, uint256(amount));
BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx);
``` [3](#0-2) 

When the real token balance is exhausted, `safeTransfer` reverts, permanently blocking later withdrawers even though their internal balance is non-zero.

The normalized-balance accounting in `SpotEngineState._updateBalanceNormalized` faithfully tracks the user-credited amounts through the `cumulativeDepositsMultiplierX18` multiplier:

```solidity
int128 newAmount = balance.amountNormalized.mul(
    cumulativeMultiplierX18
) + balanceDelta;
``` [4](#0-3) 

Because the overcredit is baked in at deposit time, every subsequent interest accrual compounds the discrepancy further.

---

### Impact Explanation

- **Concrete asset loss**: users who deposited identical amounts receive different net withdrawals depending solely on withdrawal order — the first withdrawers are made whole; later withdrawers are not.
- **Withdrawal DoS**: once the real token balance is exhausted, all remaining withdrawal calls revert, permanently locking credited balances.
- **Solvency corruption**: `totalDepositsNormalized × cumulativeDepositsMultiplierX18` permanently exceeds the actual token balance; the invariant the protocol relies on for solvency is silently broken.

---

### Likelihood Explanation

A fee-on-transfer token must be listed as a collateral product (owner action). However, once listed — even inadvertently — the vulnerability is triggered by ordinary, unprivileged user deposits. No malicious intent is required from any party after listing. The code contains no guard (e.g., a balance-before/after check) that would catch the discrepancy at deposit time or prevent the token from being listed.

---

### Recommendation

Measure the actual received amount by snapshotting the token balance before and after the transfer in the deposit flow, and pass the delta — not `txn.amount` — to `depositCollateral`. Alternatively, add an explicit check in `assertUtilization` that `token.balanceOf(clearinghouse) ≥ totalDeposits − totalBorrows`, or document and enforce (via a revert) that fee-on-transfer tokens are unsupported.

---

### Proof of Concept

1. Owner lists a fee-on-transfer token (1 % burn fee) as collateral for product `P`.
2. Alice deposits 100 → Clearinghouse receives 99, credits Alice 100.
3. Bob deposits 100 → Clearinghouse receives 99, credits Bob 100.
4. Eve deposits 100 → Clearinghouse receives 99, credits Eve 100.
5. **Actual balance: 297. Internal accounting: 300.**
6. Alice withdraws 100 → `safeTransfer(100)` succeeds; balance drops to 197.
7. Bob withdraws 100 → `safeTransfer(100)` succeeds; balance drops to 97.
8. Eve withdraws 100 → `safeTransfer(100)` **reverts** (only 97 tokens remain).
9. Eve's 100-unit internal balance is permanently unwithdrawable; she has lost 3 tokens relative to Alice and Bob despite depositing the same amount at the same time.

`assertUtilization` passes at every step because `totalDeposits (300 → 200 → 100) ≥ totalBorrows (0)` throughout, masking the real-balance shortfall entirely.
