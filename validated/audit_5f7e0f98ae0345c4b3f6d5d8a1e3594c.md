### Title
Permanent Loss of Earned Bonus Rewards via Checkpoint Advance Before Pool Check in `BeforeRedelegationSlashed` — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

When `BeforeRedelegationSlashed` fires and the bonus pool is empty, `processEventsAndClaimBonus` advances `pos.LastBonusAccrual` to `blockTime` **before** the pool balance check. The error is then swallowed in `slashRedelegationPosition`, and the position is persisted with the advanced checkpoint. The accrued-but-unpaid bonus window is permanently erased. After pool replenishment, the user cannot recover it.

---

### Impact Explanation

A position owner who legitimately accrued bonus rewards over a time window `[LastBonusAccrual_old, blockTime_slash]` permanently loses those rewards if the pool is empty when the redelegation slash fires. The pool retains the funds, but the user's claim is erased. After pool replenishment, `MsgClaimTierRewards` computes zero bonus for the lost window because `LastBonusAccrual` was already advanced past it.

This is a direct, permanent, unrecoverable loss of earned rewards — not a temporary delay.

---

### Likelihood Explanation

The pool can be empty through normal operation: the `BeginBlocker` top-up mechanism drains the pool to fund base rewards, and many concurrent bonus claims can exhaust it. A redelegation slash is a standard staking event triggered by validator downtime or double-signing. No privileged access, governance action, or attacker control is required — the condition arises from ordinary chain state.

The ADR explicitly acknowledges this behavior as a deliberate trade-off:

> "Bonus forfeits silently if the pool is short, to avoid chain halt." [5](#0-4) 

However, the trade-off is implemented incorrectly: the checkpoint is advanced even though no payment was made, making the loss permanent rather than deferred.

---

### Recommendation

Move `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` to **after** the successful `SendCoinsFromModuleToAccount` call, so checkpoints are only advanced when the bonus is actually paid. If the pool is insufficient, return the error without mutating `pos`, leaving `LastBonusAccrual` at its original value so the user can retry after replenishment — consistent with how all user-driven paths handle this case.

Additionally, `decrementEventRefCount` (a store write) inside the event loop should not be called before the pool check is confirmed to succeed, or the decrement should be rolled back on failure.

### Proof of Concept

```go
func (s *KeeperSuite) TestBonusPermanentlyLostOnRedelegSlashWithEmptyPool() {
    lockAmount := sdkmath.NewInt(sdk.DefaultPowerReduction.Int64())
    _, bondDenom := s.getStakingData()
    // Do NOT fund the pool.

    pos, _, unbondingID := s.setupRedelegatingPosition(lockAmount)
    owner := sdk.MustAccAddressFromBech32(pos.Owner)

    // Advance time so bonus accrues.
    s.ctx = s.ctx.WithBlockTime(s.ctx.BlockTime().Add(30 * 24 * time.Hour))

    balBefore := s.app.BankKeeper.GetBalance(s.ctx, owner, bondDenom)

    // Fire redelegation slash with empty pool — hook swallows ErrInsufficientBonusPool.
    sharesToUnbond := pos.Delegation.Shares.Quo(sdkmath.LegacyNewDec(10))
    err := s.keeper.Hooks().BeforeRedelegationSlashed(s.ctx, unbondingID, sharesToUnbond)
    s.Require().NoError(err) // hook does not return error

    // Replenish the pool.
    s.fundRewardsPool(sdkmath.NewInt(1_000_000_000), bondDenom)

    // Attempt to claim — should recover the forfeited bonus, but cannot.
    msgServer := keeper.NewMsgServerImpl(s.keeper)
    _, err = msgServer.ClaimTierRewards(s.ctx, &types.MsgClaimTierRewards{
        Owner:       pos.Owner,
        PositionIds: []uint64{pos.Id},
    })
    s.Require().NoError(err)

    balAfter := s.app.BankKeeper.GetBalance(s.ctx, owner, bondDenom)
    // This assertion FAILS — the bonus is permanently lost.
    s.Require().True(balAfter.Amount.GT(balBefore.Amount),
        "owner should have recovered forfeited bonus after pool replenishment")
}
```

The test demonstrates that after pool replenishment, `MsgClaimTierRewards` pays zero bonus for the forfeited window, because `LastBonusAccrual` was already advanced to `blockTime` by the hook. [6](#0-5) [1](#0-0)
