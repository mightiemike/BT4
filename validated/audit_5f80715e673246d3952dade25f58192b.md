Audit Report

## Title
SwapAllowlistExtension Bypass via Router: Any User Can Swap on Allowlisted Pools When Router Is Allowlisted - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap()` passes `msg.sender` (the direct caller) as the `sender` argument to `_beforeSwap`, which forwards it to `SwapAllowlistExtension.beforeSwap`. When `MetricOmmSimpleRouter` calls `pool.swap()`, the router becomes `sender`, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actual_user]`. Any pool admin who allowlists the router to enable router-mediated swaps for vetted users inadvertently grants unrestricted swap access to every caller of the public, permissionless router.

## Finding Description
`MetricOmmPool.swap()` invokes `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as `sender` to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly without encoding or forwarding the originating user's address — `params.extensionData` is passed through as-is from the caller, with no enforced identity field: [4](#0-3) 

The same applies to `exactInput` (multi-hop) and the `exactOutput` recursive callback path: [5](#0-4) [6](#0-5) 

In every router entry point, the pool receives `msg.sender = router`, so the extension evaluates `allowedSwapper[pool][router]`. Once the router is allowlisted, the check passes for every caller of the router regardless of whether that caller is individually allowlisted. The `allowAllSwappers` short-circuit and the `onlyPool` modifier on the extension do not mitigate this; they are orthogonal guards. [7](#0-6) 

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to KYC-verified counterparties, institutional LPs, or whitelisted market makers loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. An unprivileged attacker receives real token output from the pool's LP reserves without being on the allowlist, directly harming LPs who deposited under the assumption that only vetted counterparties could trade against them. This constitutes a direct loss of LP principal protection and broken core pool access-control functionality, meeting the "Admin-boundary break" and "Broken core pool functionality causing loss of funds" impact criteria.

## Likelihood Explanation
The bypass requires the pool admin to have added the router to the allowlist. This is the natural and expected configuration for any pool that uses `SwapAllowlistExtension` to restrict direct swaps while still wanting allowlisted users to access the standard router UI. The design provides no mechanism for the admin to simultaneously permit router-mediated swaps for allowlisted users and block them for non-allowlisted users. The bypass is therefore reachable in any production deployment that combines the allowlist extension with the router, requires no special privileges, and is repeatable by any unprivileged address.

## Recommendation
The extension must distinguish the economic actor from the routing intermediary. Two complementary approaches:

1. **Pass originating user through `extensionData`**: The router should ABI-encode `msg.sender` (the actual user) into `extensionData` for each hop. `SwapAllowlistExtension.beforeSwap` can decode and check that address when `sender` is a known router address. This requires a convention between the router and the extension, and the extension must validate that the `extensionData` field cannot be spoofed by an arbitrary caller.

2. **Explicit originator field in pool interface**: Extend the pool's `swap` interface with a separate `originator` parameter that the pool forwards to extensions alongside `sender`, allowing extensions to check the economic actor independently of the routing intermediary.

Until fixed, pool admins must not add the router to the allowlist; allowlisted users must call `pool.swap()` directly.

## Proof of Concept
```
Setup:
  1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
  2. Pool admin calls setAllowedToSwap(pool, router, true)
     (intended to let allowlisted users reach the pool via the router).
  3. Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

Attack:
  1. attacker (not on allowlist) calls
       router.exactInputSingle({ pool: pool, ... })
  2. Router calls pool.swap(recipient, ...) — msg.sender = router.
  3. Pool calls _beforeSwap(sender=router, ...).
  4. Extension evaluates allowedSwapper[pool][router] == true → passes.
  5. Swap executes; attacker receives token output from LP reserves.

Expected: revert NotAllowedToSwap (attacker not on allowlist).
Actual:   swap succeeds; allowlist guard is fully bypassed.

Foundry test sketch:
  - Deploy SwapAllowlistExtension, pool, router.
  - vm.prank(poolAdmin); extension.setAllowedToSwap(pool, router, true);
  - vm.prank(attacker); router.exactInputSingle(params);
  - Assert swap succeeds and attacker receives output tokens.
```
