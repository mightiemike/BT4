### Title
`ENABLE_SECP_OPS` Flag Enforced Only for 1-Byte Opcodes; 4-Byte Secp Opcodes Bypass Flag Entirely - (File: `src/chia_dialect.rs`)

---

### Summary

`ClvmFlags::ENABLE_SECP_OPS` is defined to gate secp256k1/secp256r1 verification. It is checked for 1-byte opcodes 64 and 65, but the same secp verification functions are dispatched unconditionally via 4-byte opcodes `0x13d61f00` and `0x1c3a8f00` with no flag check at all. Any caller or dialect configuration that omits `ENABLE_SECP_OPS` expecting secp ops to be unavailable will find them fully reachable through the 4-byte opcode path.

---

### Finding Description

In `src/chia_dialect.rs`, `ChiaDialect::op()` handles two separate dispatch paths for secp operators.

**Path 1 — 4-byte opcodes (lines 157–183):** The dispatcher matches `op_len == 4` and directly dispatches `op_secp256k1_verify` and `op_secp256r1_verify` with no flag check:

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
``` [1](#0-0) 

**Path 2 — 1-byte opcodes (lines 248–249):** The same functions are gated by `ENABLE_SECP_OPS`:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The flag is defined as:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
``` [3](#0-2) 

`ENABLE_SECP_OPS` is **not** included in `MEMPOOL_MODE`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [4](#0-3) 

In mempool mode, 1-byte opcodes 64 and 65 fall through to `unknown_operator` and fail (because `NO_UNKNOWN_OPS` is set). But the 4-byte opcodes `0x13d61f00` and `0x1c3a8f00` are dispatched before the unknown-operator path and succeed unconditionally. Secp verification is therefore fully reachable in mempool mode via the 4-byte form.

---

### Impact Explanation

Any attacker-controlled CLVM program submitted to a node running in mempool mode (or any dialect without `ENABLE_SECP_OPS`) can invoke secp256k1 or secp256r1 signature verification by using the 4-byte opcode form. The concrete corrupted result is: `op_secp256k1_verify` / `op_secp256r1_verify` returns `Reduction(1300000/1850000, nil)` — a successful execution — in a context where the caller's flag configuration was intended to make these operators unavailable. If a higher-level policy layer relies on `ENABLE_SECP_OPS` absence to block secp ops (e.g., for pre-activation softfork enforcement), that policy is silently bypassed. This can cause consensus divergence between nodes that enforce the flag strictly and those that accept the 4-byte form.

---

### Likelihood Explanation

The 4-byte opcode values are documented in comments and used in benchmark tooling: [5](#0-4) 

They are also referenced in `clvm-fuzzing/src/make_tree.rs`, meaning they are known, stable, and reachable by any CLVM program author. An attacker who knows the 4-byte opcode values (publicly documented) can trivially craft a program that invokes secp ops regardless of the dialect flag.

---

### Recommendation

Add an `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch path, mirroring the check applied to 1-byte opcodes 64 and 65:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

Alternatively, if the 4-byte opcodes are intentionally always available (legacy behavior), the `ENABLE_SECP_OPS` flag documentation should be updated to clarify it only gates the 1-byte aliases, and any policy layer relying on the flag to fully block secp ops must be corrected.

---

### Proof of Concept

A CLVM program using the 4-byte opcode `0x13d61f00` for secp256k1_verify will execute successfully even when `ChiaDialect` is constructed without `ENABLE_SECP_OPS`:

```rust
// dialect without ENABLE_SECP_OPS — secp ops should be unavailable
let dialect = ChiaDialect::new(MEMPOOL_MODE); // ENABLE_SECP_OPS not set

// program: (0x13d61f00 pubkey msg sig)  — uses 4-byte opcode directly
// This executes op_secp256k1_verify and returns nil (success)
// instead of failing with Unimplemented/unknown-op error
let result = run_program(&mut allocator, &dialect, program, env, max_cost);
// result is Ok(...) — flag restriction bypassed
```

The attacker-controlled entry path is any CLVM byte sequence submitted to `run_program` that encodes a 4-byte operator atom equal to `0x13d61f00` or `0x1c3a8f00`. No special privileges are required.
