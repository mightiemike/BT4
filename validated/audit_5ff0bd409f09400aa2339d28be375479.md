### Title
Zero-Signer Quorum Bypass in `requireValidTxSignatures` Allows Unauthorized Fast Withdrawal — (`core/contracts/Verifier.sol`)

### Summary

When `nSigner == 0`, the quorum check `require(nSignatures == nSigner)` trivially passes with an empty signatures array, allowing any unprivileged EOA to execute `submitFastWithdrawal` without any cryptographic authorization and drain the `WithdrawPool`.

### Finding Description

`requireValidTxSignatures` counts valid signatures from the supplied array and then enforces:

```solidity
require(nSignatures == nSigner, "not enough signatures");
``` [1](#0-0) 

`nSigner` is a plain storage counter incremented by `_assignPubkey` and decremented by `deletePubkey`:

```solidity
uint256 internal nSigner;
``` [2](#0-1) 

When no pubkeys have been assigned (fresh deployment before initialization completes) or all pubkeys have been deleted (key rotation window), `nSigner == 0`. Supplying `signatures = []` yields `nSignatures = 0`, so `require(0 == 0)` passes unconditionally.

`submitFastWithdrawal` has no additional guard beyond this call:

```solidity
v.requireValidTxSignatures(transaction, idx, signatures);
// ... proceeds directly to resolveFastWithdrawal and handleWithdrawTransfer
``` [3](#0-2) 

The only other guards (`markedIdxs[idx]` and `idx > minIdx`) are trivially satisfied with a fresh `idx`.

Note the contrast with `requireValidSignature` (the Schnorr path), which uses `checkQuorum`:

```solidity
return nSigned * 2 > nSigner;
``` [4](#0-3) 

When `nSigner == 0` and `nSigned == 0`, this evaluates to `0 > 0 = false`, correctly rejecting. The ECDSA fast-withdrawal path lacks this protection.

### Impact Explanation

An attacker can craft any withdrawal transaction (arbitrary `productId`, `sendTo`, `amount`) and call `submitFastWithdrawal` with an empty signatures array. The call will succeed, transferring tokens from the `WithdrawPool` to an attacker-controlled address with no cryptographic authorization. This is a complete bypass of the fast-withdrawal quorum mechanism.

### Likelihood Explanation

The vulnerable state (`nSigner == 0`) occurs:
1. **Deployment window**: Between proxy deployment and the first `assignPubKey` call.
2. **Key rotation**: If all existing pubkeys are deleted before new ones are assigned.

Both are realistic operational states. The attack requires no special privileges — any EOA can call `submitFastWithdrawal`.

### Recommendation

Add an explicit non-zero signer guard at the start of `requireValidTxSignatures`:

```solidity
require(nSigner > 0, "no signers registered");
require(nSignatures == nSigner, "not enough signatures");
```

This mirrors the implicit protection already present in `checkQuorum` via `nSigned * 2 > nSigner`.

### Proof of Concept

1. Deploy `Verifier` via proxy; call `initialize` with all-zero `Point[8]` so `nSigner` remains `0`.
2. Deploy `WithdrawPool` pointing to this `Verifier`; fund it with USDC.
3. Craft a `WithdrawCollateral` transaction encoding `sendTo = attacker`, `amount = poolBalance`.
4. Call `submitFastWithdrawal(idx=1, transaction=crafted, signatures=[])`.
5. `requireValidTxSignatures` executes: loop body never runs (`signatures.length == 0`), `nSignatures = 0`, `require(0 == 0)` passes.
6. `handleWithdrawTransfer` sends full pool balance to attacker.
7. Assert the call did **not** revert — confirming the bypass.
