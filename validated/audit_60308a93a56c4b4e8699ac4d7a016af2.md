### Title
`burnNlp` Lacks Minimum Quote Output Protection, Exposing Users to Unbound NLP Price Slippage — (`core/contracts/Clearinghouse.sol`)

---

### Summary

The `burnNlp` function in `Clearinghouse.sol` allows users to redeem NLP tokens for quote, but the user's EIP-712 signature covers only `nlpAmount` and `nonce` — never the `oraclePriceX18` that determines the actual quote payout. There is no `minQuoteAmount` field in the `BurnNlp` struct and no on-chain check enforcing a user-specified minimum. A user who signs a burn request has no protection against receiving substantially less quote than expected if the NLP oracle price moves unfavorably between signing and sequencer execution.

---

### Finding Description

The `BurnNlp` transaction struct contains only three fields:

```solidity
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint64 nonce;
}
``` [1](#0-0) 

The EIP-712 type string used to build the user's digest is:

```
"BurnNlp(bytes32 sender,uint128 nlpAmount,uint64 nonce)"
``` [2](#0-1) 

The `oraclePriceX18` that determines the quote payout lives in `SignedBurnNlp` but is **outside** the signed struct:

```solidity
struct SignedBurnNlp {
    BurnNlp tx;
    bytes signature;
    int128 oraclePriceX18;        // appended by sequencer, not signed by user
    int128[] nlpPoolRebalanceX18;
}
``` [3](#0-2) 

`computeDigest` for `BurnNlp` confirms only `sender`, `nlpAmount`, and `nonce` are hashed:

```solidity
digest = keccak256(abi.encode(
    keccak256(bytes(BURN_NLP_SIGNATURE)),
    signedTx.tx.sender,
    signedTx.tx.nlpAmount,
    signedTx.tx.nonce
));
``` [4](#0-3) 

Inside `burnNlp`, the quote amount is computed directly from the sequencer-supplied price with no floor check against any user intent:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [5](#0-4) 

No `require(quoteAmount >= txn.minQuoteAmount)` or equivalent guard exists anywhere in the execution path. [6](#0-5) 

---

### Impact Explanation

A user who signs a `BurnNlp` transaction commits to burning a fixed `nlpAmount` but has no on-chain commitment to the price at which that burn executes. If the NLP oracle price declines between the moment the user signs and the moment the sequencer submits the transaction, the user receives fewer quote tokens than anticipated with no recourse. Because `oraclePriceX18` is not part of the signed digest, the user cannot even detect post-hoc whether the price used was the market price at submission time. The burn fee (`max($1, 0.1% of quoteAmount)`) compounds the loss at lower prices. [7](#0-6) 

---

### Likelihood Explanation

Every user who calls `burnNlp` (i.e., redeems NLP for quote) is exposed. NLP price is a function of the protocol's aggregate PnL and pool composition, which can shift meaningfully during periods of high market volatility. The sequencer processes transactions in batch; a user's signed burn may sit in the queue while the NLP price moves. No privileged access is required to trigger the loss — it is an inherent property of the missing field. [8](#0-7) 

---

### Recommendation

Add a `minQuoteAmount` field to the `BurnNlp` struct and include it in the EIP-712 type string so it is part of the user's signed commitment:

```solidity
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint128 minQuoteAmount;   // new: user-specified floor
    uint64 nonce;
}
```

In `burnNlp`, enforce the guard after computing `quoteAmount`:

```solidity
require(quoteAmount >= int128(txn.minQuoteAmount), "STH"); // ERR_SLIPPAGE_TOO_HIGH
```

Users who want unconditional execution can set `minQuoteAmount = 0`. [9](#0-8) 

---

### Proof of Concept

1. User observes NLP oracle price = $1.00 and signs `BurnNlp { nlpAmount: 1000e18, nonce: N }`, expecting ~$1000 quote minus fee.
2. Before the sequencer executes, NLP price drops to $0.80 (e.g., due to a large perp loss in the pool).
3. Sequencer submits `SignedBurnNlp` with `oraclePriceX18 = 0.80e18`.
4. `burnNlp` computes `quoteAmount = 1000e18 * 0.80e18 / 1e18 = 800e18`, deducts fee, credits user ~$799.
5. User receives ~$201 less than expected. The signed digest — covering only `sender`, `nlpAmount`, `nonce` — was valid for any price, so the transaction cannot be rejected. [10](#0-9) [11](#0-10)

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
```
