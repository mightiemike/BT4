### Title
AuxPoW Parent Block PoW Checked Against Child Dogecoin Difficulty Instead of Parent Chain Difficulty — (`contract/src/dogecoin.rs`)

### Summary

In `check_aux`, the parent block's proof-of-work hash is validated against `target_from_bits(block_header.bits)` — the **child Dogecoin block's** difficulty target — rather than `target_from_bits(aux_data.parent_block.bits)` — the **parent block's own** difficulty target. A trusted relayer can submit a Dogecoin AuxPoW block backed by a fake parent block mined only to Dogecoin's (easier) difficulty, bypassing the requirement that the parent chain (e.g. Litecoin) actually performed the merged mining work.

### Finding Description

In `contract/src/dogecoin.rs`, the final PoW check in `check_aux` reads:

```rust
let pow_hash = aux_data.parent_block.block_hash_pow();
require!(
    self.skip_pow_verification
        || U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
    format!("block should have correct pow")
);
``` [1](#0-0) 

`block_header` is the **child Dogecoin block**. Its `bits` field encodes Dogecoin's difficulty target. The parent block (a Litecoin block in real merged mining) has its own `bits` field (`aux_data.parent_block.bits`) encoding Litecoin's difficulty target. The correct check must use `target_from_bits(aux_data.parent_block.bits)`.

The call chain is:

```
submit_blocks (public, #[trusted_relayer])
  → submit_block_header (dogecoin feature)
      → check_target (validates child block.bits matches expected Dogecoin difficulty)
      → check_aux (validates AuxPoW structure — BUG HERE)
``` [2](#0-1) 

`check_target` enforces that `block_header.bits` matches the expected Dogecoin retarget value — the attacker cannot freely set it to minimum difficulty. However, the bug does not require that. The invariant being violated is independent: the **parent block's PoW** must satisfy the **parent chain's own difficulty**, not the child chain's difficulty.

Real-world data from the test suite confirms the asymmetry: Litecoin mainnet bits `0x19309265` encodes a **harder** (smaller) target than Dogecoin mainnet bits `0x1a00a52e`. The test comment explicitly notes: *"Litecoin bits 0x19309265 satisfies the easier DOGE target 0x1a00a52e"*. [3](#0-2) 

This means the current code accepts a parent block whose PoW only satisfies Dogecoin's easier target, even though it should require the parent block to satisfy Litecoin's harder target.

### Impact Explanation

A trusted relayer can submit a Dogecoin block with a fully crafted fake AuxPoW structure:

1. Construct a Dogecoin child block with valid `bits` (passes `check_target`).
2. Construct a fake "Litecoin" parent block with:
   - A coinbase transaction embedding the Dogecoin block hash via the `fabe6d6d` merged mining commitment (attacker-controlled).
   - A valid Merkle proof from the coinbase txid to the parent block's `merkle_root` (attacker-controlled, since the attacker builds the parent block's transaction tree).
   - A valid chain Merkle proof from the Dogecoin block hash to the chain root in the coinbase (attacker-controlled).
   - A `nonce` chosen so that `block_hash_pow()` satisfies Dogecoin's difficulty — orders of magnitude easier than Litecoin's actual difficulty.
3. All other `check_aux` guards (chain ID checks, coinbase script position, `n_size`/`n_nonce` index check) are structural and can be satisfied by the attacker who controls the entire parent block construction.

The result is that `check_aux` returns `Ok`, `submit_block_header_inner` stores the header, and the contract's canonical chain can be extended or reorganized using blocks that have no real merged mining backing. [4](#0-3) 

### Likelihood Explanation

The entry point `submit_blocks` is gated by `#[trusted_relayer]` with `bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks)`. [5](#0-4) 

The `trusted_relayer` macro from near-plugins implements a staking mechanism: any account that stakes the required NEAR can become a trusted relayer without requiring DAO or RelayerManager approval. This is the intended permissionless relayer path. An attacker who stakes the required amount can call `submit_blocks` with attacker-controlled AuxPoW data. This is explicitly listed as a valid entry point in the scope rules ("trusted-relayer submission path with attacker-relevant data").

### Recommendation

Replace `block_header.bits` with `aux_data.parent_block.bits` in the PoW check inside `check_aux`:

```rust
// BEFORE (buggy):
|| U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),

// AFTER (correct):
|| U256::from_le_bytes(&pow_hash.0) <= target_from_bits(aux_data.parent_block.bits),
``` [6](#0-5) 

This matches the reference Dogecoin implementation, which checks the parent block's PoW against the parent block's own `nBits` field.

### Proof of Concept

```rust
#[test]
fn test_auxpow_parent_pow_checked_against_child_bits() {
    // Child Dogecoin block bits: minimum difficulty (0x207fffff)
    // target_from_bits(0x207fffff) is near U256::MAX — trivially satisfiable
    let child_bits: u32 = 0x207fffff;

    // Parent block bits: real Litecoin mainnet difficulty (0x19309265)
    // target_from_bits(0x19309265) is a very small number — hard to satisfy
    let parent_bits: u32 = 0x19309265;

    // Construct a parent block whose block_hash_pow() satisfies child_bits
    // but NOT parent_bits. Since child_bits = 0x207fffff, almost any hash works.
    // The attacker mines the parent block's nonce until hash <= target_from_bits(child_bits).
    // This requires ~1 hash attempt on average.
    let parent_block = Header {
        version: 0x20000000,  // no chain ID (Litecoin-like)
        prev_block_hash: H256::default(),
        merkle_root: /* coinbase with DOGE commitment */,
        time: current_time,
        bits: parent_bits,  // claims Litecoin difficulty
        nonce: /* found by trivial mining to child_bits target */,
    };

    let pow_hash = parent_block.block_hash_pow();
    let child_target = target_from_bits(child_bits);
    let parent_target = target_from_bits(parent_bits);

    // The hash satisfies the child (easy) target:
    assert!(U256::from_le_bytes(&pow_hash.0) <= child_target);
    // But does NOT satisfy the parent's own (hard) target:
    assert!(U256::from_le_bytes(&pow_hash.0) > parent_target);

    // check_aux uses block_header.bits (child_bits), so it passes — BUG
    // It should use aux_data.parent_block.bits (parent_bits), which would reject it
}
```
