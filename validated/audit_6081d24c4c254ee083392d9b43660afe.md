Audit Report

## Title
`SwapAllowlistExtension` checks the immediate pool caller (router) instead of the end user, enabling allowlist bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the immediate caller of `pool.swap()`. When `MetricOmmSimpleRouter` is used, `sender` is the router contract, not the end user. Any user who routes through an allowlisted router address bypasses the per-user allowlist entirely.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` (the immediate caller) as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called the pool: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()`, the pool's `msg.sender` is the router: [4](#0-3) 

The extension therefore evaluates `allowedSwapper[pool][router]` — the router's allowlist status — rather than the end user's. A pool admin who allowlists the router (a natural action to enable standard UI/SDK access) inadvertently opens the gate to every user who calls the router, regardless of whether that user is individually allowlisted. There is no existing guard in the extension, pool, or router that checks the originating user's allowlist status.

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of addresses (e.g., KYC'd counterparties, whitelisted market makers) loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. The non-allowlisted user receives the full swap output at the oracle-anchored price; the pool's LP assets are exposed to unrestricted trading. This is a direct, fund-impacting policy bypass: the allowlist invariant — "only approved addresses may swap" — is broken for the router-mediated path.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical periphery entry point for swaps. Pool admins are expected to allowlist it so that their users can access the pool through the standard UI/SDK. The `setAllowedToSwap` and `setAllowAllSwappers` interfaces give no indication that allowlisting the router grants blanket access to all downstream callers. The mistake is a natural and foreseeable admin action, not an exotic edge case.

## Recommendation
Pass the economically relevant actor — the end user — to the extension rather than the immediate pool caller. Two options:

1. **Router-side**: Have `MetricOmmSimpleRouter` encode the original `msg.sender` in `extensionData` and have the extension decode it.
2. **Extension-side (preferred)**: Change `SwapAllowlistExtension.beforeSwap` to require the pool to pass both the immediate caller and the originating user, or check `tx.origin` as a fallback when `sender` is a known router. The cleanest fix is for the pool to pass both the immediate caller and the originating user so extensions can choose which to gate.

## Proof of Concept
```
Setup:
  1. Deploy pool with SwapAllowlistExtension.
  2. Admin calls setAllowedToSwap(pool, alice, true)   // alice is allowed
  3. Admin calls setAllowedToSwap(pool, router, true)  // router is allowed (natural admin action)
  4. bob is NOT individually allowlisted.

Attack:
  5. bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: bob, ...})
  6. Router calls pool.swap(bob, ...) — pool's msg.sender = router.
  7. _beforeSwap passes sender = router to SwapAllowlistExtension.
  8. Extension checks allowedSwapper[pool][router] == true → passes.
  9. bob's swap executes at oracle price; allowlist is bypassed.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds, bob receives token output.
```
