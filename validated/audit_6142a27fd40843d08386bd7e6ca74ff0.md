Audit Report

## Title
Arithmetic-Mean Mid Price in `_afterSwapOracleStopLoss` Understates Token0 Watermarks, Allowing LP Funds to Drain Beyond Configured Drawdown — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid price as an arithmetic mean of bid and ask at line 218, while the pool's own oracle mid-price derivation (`SwapMath.midAndSpreadFeeX64FromBidAsk`) and `PriceVelocityGuardExtension` both use the geometric mean. By AM-GM, the arithmetic mean is always ≥ the geometric mean when bid ≠ ask, causing `metricT0` to be systematically understated. The resulting high watermark is set too low, so the drawdown floor is lower than the admin configured, and LP funds can drain further than the intended threshold before the stop-loss reverts the swap.

## Finding Description

**Root cause — arithmetic mean at line 218:**
```solidity
// OracleValueStopLossExtension.sol L218
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
```

**Correct formula used everywhere else — geometric mean:**
```solidity
// SwapMath.sol L70
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);

// PriceVelocityGuardExtension.sol L48
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
```

By AM-GM: `(bid + ask)/2 ≥ sqrt(bid × ask)`, with equality only when `bid == ask`. In normal operation `bid < ask`, so `mid_arith > mid_geo`.

This inflated `midPriceX64` is fed into `_metrics` at lines 254–255:
```solidity
metricT0 = _clampMetric(t0ps + Math.mulDiv(Math.mulDiv(uint256(t1), Q64, midPriceX64), METRIC_SCALE, shares));
metricT1 = _clampMetric(Math.mulDiv(Math.mulDiv(uint256(t0), midPriceX64, Q64), METRIC_SCALE, shares) + t1ps);
```

Because `mid_arith > mid_geo`, dividing by a larger price in the `t1 * Q64 / midPriceX64` term produces a smaller token0-equivalent value for the token1 holdings, understating `metricT0`.

The understated `metricT0` is then used to ratchet up the high watermark at lines 270–273:
```solidity
(uint256 hwm0, bool breach0) = _applyWatermark(metricT0, _decayed(hwmS.token0, decayRate, dt), floorMultiplier);
if (breach0 && zeroForOne) {
  revert OracleStopLossTriggered(binIdx, true, metricT0, (hwm0 * floorMultiplier) / E6);
}
```

`_applyWatermark` sets `newHwm = metric` when `metric >= hwm` (ratchet up). Since `metricT0` is understated, `hwm0` is set lower than the true geometric-mid value. The breach floor is `hwm0 * (E6 - drawdown) / E6`. A lower `hwm0` produces a lower floor, so a subsequent genuine value loss must fall further before the stop-loss triggers.

No existing guard corrects this: the arithmetic mean is computed unconditionally before any bin iteration, and there is no fallback to `SwapMath.midAndSpreadFeeX64FromBidAsk`.

## Impact Explanation
The stop-loss guard's token0 drawdown floor is set systematically lower than the admin's configured `drawdownE6`. For a 5% drawdown limit and a 10% oracle spread, the arithmetic mean exceeds the geometric mean by approximately `(spread/2)² / (2 × mid) ≈ 0.125%`, making the effective protection ~5.125% instead of 5%. The error grows quadratically with spread width and becomes material during volatile markets when spreads widen. LP principal drains beyond the configured threshold before the guard reverts the swap — a direct loss of LP funds above the intended cap.

## Likelihood Explanation
Every public swap through a pool with `OracleValueStopLossExtension` configured and a non-zero `drawdownE6` triggers `_afterSwapOracleStopLoss`. The miscalibration is present on every call whenever `bid < ask`, which is always true in normal operation. No privileged access, special setup, or unusual token behavior is required; any unprivileged trader executing a swap causes the watermark to be set at the wrong value.

## Recommendation
Replace the arithmetic mean with the geometric mean, consistent with `SwapMath.midAndSpreadFeeX64FromBidAsk` and `PriceVelocityGuardExtension`:

```solidity
// Before (wrong):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (correct):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
```

This ensures the stop-loss guard evaluates LP value at the same oracle mid price the pool uses for fee computation, making the drawdown floor match the admin's configured intent.

## Proof of Concept
1. Deploy a pool with `OracleValueStopLossExtension`, `drawdownE6 = 50_000` (5%), and a price provider returning asymmetric quotes: `bid = 0.92 × P`, `ask = 1.05 × P` for some reference price `P`.
2. Add liquidity to bin 0 with `t0 = 1000`, `t1 = 1000`.
3. Execute a swap (`zeroForOne = true`) to initialize the watermark. The guard computes `mid_arith = (0.92 + 1.05)/2 × P = 0.985 × P`, while the correct `mid_geo = sqrt(0.92 × 1.05) × P ≈ 0.9827 × P`. The arithmetic mean is ~0.23% higher, understating `metricT0` by ~0.23%, setting `hwm0` ~0.23% too low.
4. Drain the bin by 5.23% of token0 value. The stop-loss should trigger at 5% but does not, because the floor was set 0.23% too low.
5. Confirm the stop-loss triggers only after the additional 0.23% drain, demonstrating LP funds leaked beyond the configured threshold. A Foundry unit test can assert `OracleStopLossTriggered` is not emitted at exactly 5% drawdown but is emitted at 5.23%, reproducing the miscalibration.
