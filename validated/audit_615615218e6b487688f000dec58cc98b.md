### Title
Difficulty Adjustment Uses Mainchain Ancestor Instead of Fork Ancestor, Enabling Incorrect PoW Acceptance on Fork Blocks - (`contract/src/dogecoin.rs`)

### Summary

The Dogecoin (and Litecoin/Bitcoin) difficulty recalculation functions call `get_header_by_height()`, which always resolves blocks from the **mainchain** height-to-header map. When a fork block is being validated, this returns the mainchain block at the boundary height rather than the fork's actual ancestor at that height. For Dogecoin post-block 145,000 (where difficulty adjusts every single block), this means every fork block at height `H+2` or beyond (where `H` is the fork point) has its difficulty validated against the wrong ancestor's timestamp, producing an incorrect expected target.

### Finding Description

In `contract/src/dogecoin.rs`, `get_next_work_required` computes the difficulty for the next block using the timestamp of the block at `height_first`: [1](#0-0) 

The `get_header_by_height` implementation in `lib.rs` exclusively reads from `mainchain_height_to_header`: [2](#0-1) 

This map only contains blocks on the current canonical chain. Fork blocks are stored only in `headers_pool` (keyed by hash), not in `mainchain_height_to_header`: [3](#0-2) 

The codebase itself acknowledges this problem with a `TODO` comment directly above the call: [4](#0-3) 

**Concrete scenario for Dogecoin (height ≥ 145,000, where `difficulty_adjustment_interval = 1`):**

- Fork diverges at mainchain height `H`. The fork block at `H` is stored in `headers_pool` only.
- Attacker submits fork block at height `H+1`. `prev_block_header` is the fork block at `H` (correct, fetched by hash). `height_first = H - 1`. `get_header_by_height(H-1)` returns the mainchain block at `H-1`, which equals the fork's ancestor at `H-1` (since the fork diverged at `H`). **This call is correct.**
- Attacker submits fork block at height `H+2`. `prev_block_header` is the fork block at `H+1` (correct). `height_first = H`. `get_header_by_height(H)` returns the **mainchain** block at `H`, **not** the fork block at `H`. These are different blocks with different timestamps. **This call is wrong.**

The incorrect `first_block_time` flows directly into `calculate_next_work_required`: [5](#0-4) 

The `modulated_timespan` is computed as `prev_block_header.time - first_block_time`. If the mainchain block at `H` has a later timestamp than the fork block at `H`, `modulated_timespan` is smaller, the computed target is lower (easier), and the contract accepts a fork block whose actual `bits` field encodes an easier difficulty than the protocol requires.

The same structural bug exists in `litecoin.rs` and `bitcoin.rs`, though it triggers less frequently (only at 2016-block boundaries): [6](#0-5) [7](#0-6) 

### Impact Explanation

An attacker can submit a multi-block fork where blocks at height `H+2` and beyond pass the difficulty check with an incorrect (too-easy) target. This allows the fork to accumulate chainwork with less actual PoW than the mainchain requires. If the fork's cumulative chainwork exceeds the mainchain tip's chainwork (checked in `submit_block_header_inner`), a chain reorganization is triggered, corrupting the canonical chain stored in the light client. Downstream contracts calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` would then verify transactions against a fraudulent chain. [8](#0-7) 

### Likelihood Explanation

For Dogecoin (the most affected chain), every fork block at height `H+2` or beyond triggers the bug — no special timing or privileged access is required. Any unprivileged NEAR account can call `submit_blocks` with adversarially crafted Dogecoin fork headers. The attacker controls the fork block timestamps and `bits` field, and can choose values that exploit the timestamp mismatch to produce an easier-than-correct target.

### Recommendation

Replace the `get_header_by_height` call in all three difficulty functions with an ancestor traversal that walks the `headers_pool` chain by `prev_block_hash` from `prev_block_header` back `blocks_to_go_back` steps. This ensures the correct fork ancestor is used regardless of whether the block being validated is on the mainchain or a fork. The `TODO` comment in `dogecoin.rs` already identifies this as the correct fix direction.

### Proof of Concept

1. Deploy the Dogecoin light client contract initialized at height ≥ 145,000.
2. Submit a valid mainchain block at height `H` (fork point). The mainchain block at `H` has timestamp `T_main`.
3. Craft a fork block at height `H` with timestamp `T_fork < T_main` (same `prev_block_hash` as the mainchain block at `H-1`, different content). Submit it — it is stored as a fork block in `headers_pool` only.
4. Submit a fork block at height `H+1` building on the fork block at `H`. Difficulty check uses `get_header_by_height(H-1)` — correct, same block on both chains.
5. Submit a fork block at height `H+2` building on the fork block at `H+1`. Difficulty check calls `get_header_by_height(H)`, which returns the **mainchain** block at `H` with timestamp `T_main`. The computed `modulated_timespan = T_fork_H1 - T_main` is smaller than the correct value `T_fork_H1 - T_fork`, producing an easier target. Set the fork block's `bits` to match this easier target. The `require!(expected_bits == block_header.bits, ...)` check passes with an under-difficulty block.
6. Continue submitting fork blocks with under-difficulty `bits`. Each accumulates chainwork. Once the fork's `chain_work` exceeds the mainchain tip's `chain_work`, `reorg_chain` is triggered, replacing the canonical chain with the attacker's fraudulent fork. [9](#0-8)

fn calculate_next_work_required(
    config: &DogecoinConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let retarget_timespan = config.pow_target_timespan;
    let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;

    let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;

    let min_timespan = retarget_timespan - (retarget_timespan / 4);
    let max_timespan = retarget_timespan + (retarget_timespan / 2);

    if modulated_timespan < min_timespan {
        modulated_timespan = min_timespan;
    } else if modulated_timespan > max_timespan {
        modulated_timespan = max_timespan;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(modulated_timespan).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target =
        new_target / U256::from(<i64 as TryInto<u64>>::try_into(retarget_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()
