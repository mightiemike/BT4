### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing 64-Byte Transaction Merkle Proof Forgery Protection — (File: `contract/src/lib.rs`)

---

### Summary

The contract exposes two public versions of its SPV proof function. The v2 version (`verify_transaction_inclusion_v2`) adds a coinbase merkle proof check that mitigates the documented 64-byte transaction Merkle proof forgery attack. The v1 version (`verify_transaction_inclusion`) is marked `#[deprecated]` but remains a fully reachable on-chain entry point. Any unprivileged NEAR caller can invoke v1 directly, bypassing the coinbase check entirely and obtaining a forged `true` result.

---

### Finding Description

`verify_transaction_inclusion` is annotated with Rust's `#[deprecated]` attribute, which is a **compile-time lint only**. It imposes no runtime restriction on NEAR contract calls. The function remains a public `#[pause]`-gated method callable by any account. [1](#0-0)

This is the exact structural analog to the original report: two versions of a verification function exist, the less complete version (v1, no coinbase guard) is still reachable, and callers who use it bypass the critical security parameter added in the more complete version (v2).

### Impact Explanation

An attacker can forge a transaction inclusion proof against any Bitcoin block stored in the contract. By supplying a 64-byte value whose double-SHA256 hash equals an internal Merkle tree node, and constructing a matching proof path, the attacker causes `verify_transaction_inclusion` to return `true` for a non-existent transaction. Any recipient NEAR contract consuming this result — e.g., a bridge, escrow, or settlement contract — is deceived into believing a Bitcoin payment or event occurred when it did not. This breaks the core security invariant of the SPV system.

---

### Likelihood Explanation

The 64-byte transaction attack is documented and publicly known (referenced in the contract's own deprecation notice and at https://www.bitmex.com/blog/64-Byte-Transactions). The v1 entry point is visible on-chain. An attacker does not need any privileged role; they only need to call `verify_transaction_inclusion` instead of `verify_transaction_inclusion_v2`. The crafting of a valid 64-byte forgery requires knowledge of the target block's Merkle tree but is feasible for a motivated attacker.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public contract interface, or restrict it with `#[private]` so it cannot be called externally. `verify_transaction_inclusion_v2` should be the sole public entry point for SPV proof verification. The internal delegation from v2 to v1 (`self.verify_transaction_inclusion(args.into())`) can be refactored into a private helper to avoid exposing the unchecked path.

---

### Proof of Concept

1. Attacker selects a Bitcoin block already stored in the contract (any block in `mainchain_header_to_height`).
2. Attacker identifies an internal Merkle tree node `N` in that block's transaction Merkle tree.
3. Attacker constructs `ProofArgs` with `tx_id = N`, a valid `tx_block_blockhash`, an `tx_index` and `merkle_proof` such that `compute_root_from_merkle_proof(N, tx_index, merkle_proof) == block.merkle_root`.
4. Attacker calls `verify_transaction_inclusion` (v1) directly on the NEAR contract with these args.
5. The function passes all checks (block is on mainchain, confirmations satisfied, merkle root matches) and returns `true`.
6. No coinbase proof is required; the forgery succeeds.
7. Any downstream NEAR contract that trusted this result treats the forged transaction as confirmed. [7](#0-6)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
