### Title
Protobuf `ValidResourceBounds` Deserialization Silently Collapses `AllResources` to `L1Gas`, Producing a Different Transaction Hash Domain - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf converter for `ValidResourceBounds` silently maps an `AllResources` variant (with zero `l2_gas` and zero `l1_data_gas`) to the `L1Gas` variant after deserialization. Because `get_tip_resource_bounds_hash` hashes a different number of resource-bound felts depending on the variant, the transaction hash computed at the gateway (using `AllResources`, 3 resource felts) diverges from the hash recomputed by any node that deserializes the same transaction from protobuf (using `L1Gas`, 2 resource felts). This is a hash-domain boundary violation analogous to the SEDA duplicate-validator issue: a structural invariant that should be preserved across a serialization boundary is silently broken, causing downstream consumers to operate on a different canonical value.

### Finding Description

**Step 1 – Hash domain is variant-sensitive.**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` produces a different Poseidon digest depending on the `ValidResourceBounds` variant:

```
L1Gas      → hash([tip, l1_gas_packed, l2_gas_zero_packed])          // 2 resource felts
AllResources → hash([tip, l1_gas_packed, l2_gas_packed, l1_data_gas_packed])  // 3 resource felts
``` [1](#0-0) 

**Step 2 – Gateway always produces `AllResources`.**

`RpcInvokeTransactionV3` and `InternalRpcInvokeTransactionV3` both carry `resource_bounds: AllResourceBounds`. The `InvokeTransactionV3Trait` implementation for `InternalRpcInvokeTransactionV3` wraps this in `ValidResourceBounds::AllResources(...)`, so the hash computed at the gateway always uses the 3-felt path. [2](#0-1) 

**Step 3 – Protobuf round-trip silently collapses the variant.**

The P2P sync path deserializes committed blocks using `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3`, which calls `ValidResourceBounds::try_from(...)`. That converter applies the following lossy rule:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // ← variant changed
} else {
    ValidResourceBounds::AllResources(...)
})
``` [3](#0-2) 

A transaction submitted with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is serialized to protobuf with all three fields present but zero for l2/l1_data. On deserialization the converter produces `L1Gas`, not `AllResources`.

**Step 4 – Divergent hash.**

After deserialization the syncing node calls `get_invoke_transaction_v3_hash` on the reconstructed `InvokeTransactionV3`. Because the variant is now `L1Gas`, `get_tip_resource_bounds_hash` hashes only 2 resource felts, yielding hash H2 ≠ H1 (the hash stored in the block by the sequencer). [4](#0-3) 

### Impact Explanation

Any node that re-derives the transaction hash from a protobuf-deserialized `InvokeTransactionV3` with zero l2/l1_data gas will compute a hash that differs from the canonical hash stored in the block. This breaks:

1. **Block hash validation during P2P sync** – if the block hash is derived from transaction hashes, syncing nodes will compute a wrong block hash and reject the block, causing a chain split.
2. **Transaction hash binding** – the hash stored in `InternalRpcTransaction.tx_hash` at the sequencer is authoritative, but any peer that re-hashes the deserialized transaction gets a different value, breaking deduplication, receipt lookup, and any downstream hash-equality check.

This matches the allowed impact: *"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

### Likelihood Explanation

The trigger is an unprivileged user submitting a V3 invoke transaction with `AllResourceBounds` where `l2_gas.max_amount = 0` and `l1_data_gas.max_amount = 0`. The gateway accepts `AllResourceBounds` unconditionally (it only validates the l2 gas *price* threshold, not the amount). Such a transaction would revert during execution (zero initial Sierra gas) but would still be included in the block and propagated. No special privilege is required.

### Recommendation

Remove the lossy `L1Gas` / `AllResources` disambiguation from the protobuf deserializer. The protobuf wire format already carries all three resource-bound fields; the deserializer should always produce `AllResources` when deserializing a V3 transaction, preserving the exact variant that was serialized. The `L1Gas` variant should only be produced when deserializing pre-0.13.3 transactions that genuinely lack the l2/l1_data gas fields (i.e., when `l2_gas` is `None` in the protobuf, not merely zero).

```rust
// Proposed fix: only fall back to L1Gas when l2_gas is absent, not when it is zero.
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else { return Err(missing("l1_gas")); };
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        match (value.l2_gas, value.l1_data_gas) {
            (None, _) => Ok(ValidResourceBounds::L1Gas(l1_gas)),  // pre-0.13.3
            (Some(l2_gas), l1_data_gas) => {
                let l2_gas: ResourceBounds = l2_gas.try_into()?;
                let l1_data_gas: ResourceBounds =
                    l1_data_gas.unwrap_or_default().try_into()?;
                Ok(ValidResourceBounds::AllResources(
                    AllResourceBounds { l1_gas, l2_gas, l1_data_gas }
                ))
            }
        }
    }
}
```

### Proof of Concept

1. Submit an invoke V3 transaction with `AllResourceBounds { l1_gas: { max_amount: 1000, max_price: 1 }, l2_gas: { max_amount: 0, max_price: 0 }, l1_data_gas: { max_amount: 0, max_price: 0 } }`.
2. The gateway computes hash H1 via `get_tip_resource_bounds_hash(AllResources(...))` → 3-felt Poseidon input.
3. The transaction is included in a block (as a reverted transaction due to zero L2 gas).
4. A syncing peer receives the block over P2P and calls `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`.
5. Because `l2_gas.is_zero() && l1_data_gas.is_zero()`, the converter returns `L1Gas(l1_gas)`.
6. The peer computes hash H2 via `get_tip_resource_bounds_hash(L1Gas(...))` → 2-felt Poseidon input.
7. H1 ≠ H2. Any hash-equality check (block hash derivation, receipt lookup, deduplication) fails for this transaction on the syncing peer. [5](#0-4) [6](#0-5)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
