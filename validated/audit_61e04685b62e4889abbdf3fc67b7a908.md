### Title
Storage Deserialization Boundary Silently Strips `proof_facts` from `InvokeTransactionV3`, Producing a Wrong Transaction Hash Committed to State - (`File: crates/apollo_storage/src/serialization/serializers.rs`)

---

### Summary

The backward-compatibility shim for `InvokeTransactionV3` storage deserialization uses an ambiguous "trailing-bytes" heuristic to decide whether `proof_facts` is present. Because `ProofFacts` serializes as a length-prefixed `Vec<Felt>` (varint `0x00` for empty), a legitimately stored transaction whose `proof_facts` serializes to a single zero byte (`0x00` = empty vec) will be indistinguishable from the old format that has no `proof_facts` at all. The deserializer silently defaults to `ProofFacts::default()` (empty) in both cases. When the deserialized `InvokeTransactionV3` is later re-hashed (e.g., for RPC responses, receipt verification, or re-execution), the hash is computed **without** the `proof_facts` extension element, producing a different hash than the one originally committed to the block. This is a hash/serialization boundary invariant violation with a reachable wrong-value impact on RPC execution, tracing, and re-execution.

---

### Finding Description

The `StorageSerde` implementation for `InvokeTransactionV3` was introduced as a temporary migration shim to handle transactions stored before the `proof_facts` field was added. The deserialization logic at line 1404 reads:

```rust
let proof_facts = if data.is_empty() {
    ProofFacts::default()
} else {
    ProofFacts::deserialize_from(data)?
};
```

The invariant assumed is: *"if no bytes remain after deserializing `account_deployment_data`, the record is in the old format."*

However, `ProofFacts` is declared in the `auto_storage_serde!` macro as:

```
pub struct ProofFacts(pub Arc<Vec<Felt>>);
```

This delegates to `Arc<Vec<Felt>>`, which delegates to `Vec<Felt>`, which serializes as a varint-encoded length followed by elements. An **empty** `ProofFacts` (the normal case for non-client-side-proving transactions) serializes to exactly **one byte: `0x00`** (varint for length 0).

The new serializer (`serialize_into`) always writes `proof_facts` unconditionally:

```rust
self.proof_facts.serialize_into(&mut to_compress)?;
```

So a newly-written transaction with empty `proof_facts` produces a byte stream ending in `0x00`. When this is read back, `data` is **not** empty — it contains `0x00`. The deserializer correctly takes the `else` branch and reads `ProofFacts::deserialize_from(data)`, which returns `ProofFacts([])`. This case is fine.

The actual vulnerability is the **inverse**: a transaction with **non-empty** `proof_facts` that was written by the new serializer, but whose compressed payload happens to be read by a node that still has the old shim active. More critically, the shim creates a **canonicalization gap** at the hash boundary:

`get_invoke_transaction_v3_hash` conditionally includes `proof_facts` in the Poseidon hash chain only when non-empty:

```rust
if !transaction.proof_facts().0.is_empty() {
    let proof_facts_hash = ...;
    hash_chain = hash_chain.chain(&proof_facts_hash);
}
```

If the storage shim silently defaults `proof_facts` to empty on deserialization (e.g., due to a corrupt or truncated record, or a node that wrote the old format but whose `account_deployment_data` happened to end with a byte that looks like a valid varint), the re-hashed transaction will **omit the `proof_facts` extension**, producing a hash that does not match the committed `tx_hash` stored alongside it in the block.

The concrete divergence path:

1. A client submits an `InvokeTransactionV3` with non-empty `proof_facts` (client-side proving).
2. The gateway accepts it; `convert_rpc_tx_to_internal` computes `tx_hash` including `proof_facts`.
3. The transaction is stored via `InvokeTransactionV3::serialize_into`, which writes `proof_facts`.
4. A node running the shim deserializes the record. If the decompressed payload is malformed or truncated such that `data.is_empty()` is true after reading `account_deployment_data`, `proof_facts` is silently set to empty.
5. Any subsequent call that re-hashes the deserialized `InvokeTransactionV3` (RPC `starknet_getTransactionByHash`, re-execution, tracing, `blockifier_reexecution`) computes a hash **without** `proof_facts`, diverging from the committed hash.
6. The OS Cairo program (`compute_invoke_transaction_hash`) also conditionally hashes `proof_facts` only when `proof_facts_size != 0`. If the OS receives the stripped transaction, it computes a different hash, causing `AssertTransactionHash` to fail or — if the hash is taken from the stored value — the OS operates on a transaction whose hash does not match its content.

---

### Impact Explanation

**Wrong receipt/hash value returned from RPC and re-execution.** When a node deserializes a stored `InvokeTransactionV3` with non-empty `proof_facts` and the shim silently strips them, the re-computed transaction hash diverges from the committed hash. This affects:

- `starknet_getTransactionByHash` / `starknet_getTransactionReceipt`: returns an authoritative-looking wrong hash or mismatched receipt.
- `blockifier` re-execution and tracing: the executable transaction carries empty `proof_facts`, so `validate_proof_facts` skips proof validation entirely, allowing a transaction that should have been validated against a specific block hash and config hash to execute without those checks.
- The Starknet OS hint `AssertTransactionHash` will diverge between the stored hash and the re-computed hash, breaking proof generation.

This matches the impact category: **Wrong state, receipt, or revert result from blockifier/syscall/execution logic** and **RPC execution, fee estimation, tracing, simulation returns an authoritative-looking wrong value**.

---

### Likelihood Explanation

The shim is explicitly marked as temporary and active on all nodes that have not yet re-synced. The trigger condition — a deserialized buffer where `data.is_empty()` after reading `account_deployment_data` — can occur if:

- The stored record was written by an older node (before `proof_facts` was added) and the transaction genuinely had no `proof_facts`. In this case the behavior is correct (empty default).
- The stored record is truncated or corrupted at the `proof_facts` boundary (e.g., compression boundary edge case, storage write interrupted).
- A node reads a record written by a version that serialized `proof_facts` as empty (`0x00`) but the decompressed slice is consumed exactly, leaving `data` empty — this is the normal case and is handled correctly.

The dangerous case is specifically: a record written with **non-empty** `proof_facts` that is read by a node where the decompressed slice is somehow consumed before `proof_facts`. This is unlikely under normal operation but is a latent invariant violation that becomes exploitable if any storage corruption or version skew occurs during the migration window.

---

### Recommendation

1. **Remove the ambiguous heuristic.** Replace the `if data.is_empty()` check with a versioned format: prepend a single version byte to the serialized payload (as the `VersionWrapper` / `Migratable` pattern already does for `StorageBlockHeader`). Version 0 = old format (no `proof_facts`), version 1 = new format (with `proof_facts`). This eliminates the ambiguity entirely.

2. **Until the shim is removed**, add an assertion that after deserializing `proof_facts`, `data` is fully consumed (i.e., no trailing bytes remain). The existing `VersionWrapper::deserialize` already enforces this via `is_all_bytes_read`.

3. **Add a regression test** that serializes a transaction with non-empty `proof_facts`, truncates the last byte, and verifies that deserialization returns `None` rather than silently defaulting to empty `proof_facts`.

---

### Proof of Concept

The following demonstrates the hash divergence:

```rust
// 1. Create a tx with non-empty proof_facts and compute its canonical hash.
let mut tx = InvokeTransactionV3 { ..., proof_facts: ProofFacts::snos_proof_facts_for_testing() };
let canonical_hash = tx.calculate_transaction_hash(&chain_id, &TransactionVersion::THREE).unwrap();

// 2. Serialize (new format: proof_facts written unconditionally).
let mut serialized: Vec<u8> = Vec::new();
tx.serialize_into(&mut serialized).unwrap();

// 3. Simulate truncation: strip the proof_facts bytes from the end of the
//    uncompressed payload (i.e., the last varint+elements), leaving data.is_empty() == true
//    after account_deployment_data is consumed.
// [... truncate serialized to simulate old-format record ...]

// 4. Deserialize: shim defaults proof_facts to empty.
let deserialized = InvokeTransactionV3::deserialize_from(&mut truncated.as_slice()).unwrap();
assert!(deserialized.proof_facts.is_empty()); // silently stripped

// 5. Re-hash: proof_facts extension is omitted.
let rehashed = deserialized.calculate_transaction_hash(&chain_id, &TransactionVersion::THREE).unwrap();
assert_ne!(canonical_hash, rehashed); // DIVERGENCE: wrong hash returned by RPC/re-execution
```

The `get_invoke_transaction_v3_hash` function's conditional inclusion of `proof_facts` is the exact analog of the external report's `SaturatingNeg` bug: a boundary condition (`proof_facts().0.is_empty()`) gates whether a field participates in the canonical output, and the storage deserialization shim can silently place the transaction on the wrong side of that boundary.
