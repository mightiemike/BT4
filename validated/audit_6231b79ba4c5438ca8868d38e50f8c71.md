Audit Report

## Title
`uint32` overflow in `_afterTimelock` allows pool admin to bypass all parameter timelocks instantly ‚Äî (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`_afterTimelock` computes `uint32(block.timestamp + oracleStopLossConfig[pool_].timelock)`. When `timelock` is set to `type(uint32).max`, the `uint256` sum exceeds `type(uint32).max` and the explicit cast silently truncates (Solidity 0.8+ explicit casts do not revert), producing an `executeAfter` value already in the past. `_requireElapsed` then passes immediately, allowing the pool admin to bypass the timelock on every subsequent parameter change and instantly disable LP stop-loss protection.

## Finding Description

`_afterTimelock` at line 297‚Äì299 performs the overflow:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` (~1,753,000,000 in 2026) + `type(uint32).max` (4,294,967,295) = 6,047,967,295. Cast to `uint32`: `6,047,967,295 mod 4,294,967,296 ‚âà 1,752,999,999`, which is approximately `block.timestamp ‚àí 1` ‚Äî already elapsed.

`_requireElapsed` at line 301‚Äì303 then trivially passes:

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
```

`proposeOracleStopLossTimelock` at line 78‚Äì84 has no validation on `newTimelock`, unlike `_validateDrawdown` and `_validateDecay` which enforce upper bounds at lines 305‚Äì310. `initialize` at line 56‚Äì58 also calls `_validateDrawdown` and `_validateDecay` but has no `_validateTimelock`.

**Attack path:**
1. Pool initialized with `timelock = 0` (no validation; confirmed by test at `OracleValueStopLossSubExtension.t.sol:49`).
2. Admin calls `proposeOracleStopLossTimelock(pool, type(uint32).max)`. With current `timelock = 0`, `executeAfter = uint32(block.timestamp)` ‚Äî immediately executable.
3. Admin calls `executeOracleStopLossTimelock(pool)`. `block.timestamp >= block.timestamp` passes. `timelock` is now `type(uint32).max`.
4. Admin calls `proposeOracleStopLossDrawdown(pool, 0)`. `_afterTimelock` wraps to a past value.
5. Admin immediately calls `executeOracleStopLossDrawdown(pool)`. `_requireElapsed` passes. `drawdownE6 = 0`.
6. `_afterSwapOracleStopLoss` at line 216‚Äì217 short-circuits: `if (drawdown == 0) return;` ‚Äî stop-loss permanently disabled.

The same bypass applies to `proposeOracleStopLossDecay` and `proposeOracleStopLossHighWatermarks`, both of which call `_afterTimelock` at lines 133 and 162.

## Impact Explanation

The NatDoc at lines 13‚Äì28 explicitly states: *"Drawdown and decay changes are timelocked so LPs can react."* The timelock is the sole mechanism protecting LPs from sudden stop-loss parameter changes. By bypassing it, the pool admin can immediately set `drawdownE6 = 0` (disabling stop-loss entirely), set watermarks to arbitrarily high values (bricking the pool), or maximize decay to collapse all watermarks within one second. This constitutes an admin-boundary break (pool admin bypasses timelocks) and broken core pool functionality causing loss of funds (LP principal unprotected from oracle-driven value loss).

## Likelihood Explanation

Requires a malicious or compromised pool admin. The pool admin is explicitly semi-trusted ‚Äî constrained by caps and timelocks ‚Äî and the timelock bypass is the exact boundary violation the contest rules flag as in-scope. The initial `timelock = 0` (as used in tests) makes step 2 executable in the same block with no waiting. The overflow arithmetic is deterministic and requires no special on-chain conditions.

## Recommendation

Add a `_validateTimelock` function with a reasonable upper bound (e.g., 30 days = 2,592,000 seconds, well within `uint32` range and safe from overflow for any realistic `block.timestamp`) and call it in both `initialize` and `proposeOracleStopLossTimelock`:

```solidity
uint32 private constant MAX_TIMELOCK = 30 days;

function _validateTimelock(uint32 timelock) private pure {
    if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);
}
```

Call alongside `_validateDrawdown`/`_validateDecay` in `initialize` (line 57‚Äì58) and at the top of `proposeOracleStopLossTimelock` (line 78).
