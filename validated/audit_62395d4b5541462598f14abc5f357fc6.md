### Title
Migration Halt via Vesting-Owned Position with Active Redelegation — (`x/tieredrewards/migrations/v2/migrate.go`, `x/tieredrewards/keeper/force_exit.go`, `x/tieredrewards/keeper/transfer_delegation.go`)

---

### Summary

A vesting account owner who holds a tier position and calls `TierRedelegate` before the v8 upgrade can cause the `Migrate1to2` upgrade migration to return a non-nil error, aborting the upgrade handler and permanently halting the chain at the upgrade height.

---

### Finding Description

The v8 upgrade migration calls `exitVestedAccountsPositions`, which iterates all positions owned by vesting accounts and calls `ForceFullExitWithDelegation` on each. That function calls `transferDelegationFromPosition`, which contains a "Defensive" guard:

```go
// x/tieredrewards/keeper/transfer_delegation.go:117-123
isRedelegating, err := k.isRedelegating(ctx, pos.DelegatorAddress)
if err != nil {
    return ..., err
}
if isRedelegating {
    return ..., errorsmod.Wrapf(types.ErrActiveRedelegation, "position %d has an active redelegation", pos.Id)
}
```

This error propagates without recovery through the entire migration call stack:

- `transferDelegationFromPosition` → `ErrActiveRedelegation`
- `ForceFullExitWithDelegation` line 62-63: `return fmt.Errorf("transfer delegation back to owner for position %d: %w", posID, err)`
- `exitVestedAccountsPositions` line 99-100: `return fmt.Errorf("force-exit position %d: %w", posID, err)`
- `Migrate` line 37-38: `return fmt.Errorf("exit vested accounts positions: %w", err)`
- `Migrate1to2` line 18: returns the error
- `RunMigrations` propagates it to the upgrade handler
- `registerV8UpgradeHandler` line 58-60: `return map[string]uint64{}, err`

In Cosmos SDK, an upgrade handler returning a non-nil error causes the node to panic and halt at the upgrade height.

**The attacker-controlled precondition is reachable because `TierRedelegate` does not check for vesting account ownership.** `validateRedelegatePosition` (called by `TierRedelegate`) only checks ownership, delegation state, exit state, tier close-only status, and transitive redelegation — it does not call `validateNonVestingAccount`. By contrast, `validateNewPosition` (used by `LockTier` and `CommitDelegationToTier`) does call `validateNonVestingAccount`. This means a vesting account that created a position before v8 can freely call `TierRedelegate` on it.

**Concrete attack steps:**
1. Before v8: vesting account owner creates a tier position (allowed on v7.x).
2. Before v8: owner calls `TierRedelegate` on the position. `validateRedelegatePosition` passes — no vesting check. The staking module creates a redelegation entry for `pos.DelegatorAddress` with a completion time of `blockTime + unbondingTime` (typically 21 days).
3. Attacker ensures the upgrade height falls within the 21-day redelegation window (upgrade heights are announced via governance, giving the attacker precise timing).
4. At the upgrade height, `Migrate1to2` runs, hits `isRedelegating == true` for the position's delegator address, returns `ErrActiveRedelegation`, and the chain halts.

---

### Impact Explanation

The upgrade handler aborts with a non-nil error, causing a chain-wide panic and halt at the upgrade height. All user funds — staking delegations, tier positions, bank balances — are permanently inaccessible until validators coordinate an emergency patch, rebuild, and restart. This matches the Critical scope: an unprivileged on-chain action causes permanent lock of all user funds.

---

### Likelihood Explanation

- The attacker only needs to own a vesting account with a tier position (a state that existed on v7.x) and submit one `TierRedelegate` transaction within 21 days of the announced upgrade height.
- Upgrade heights are published via on-chain governance proposals, giving the attacker precise timing information.
- No special privileges, leaked keys, or social engineering are required.
- The redelegation window (21 days) is long relative to the typical upgrade announcement window, making it straightforward to keep the redelegation active at the upgrade height.

---

### Recommendation

In `exitVestedAccountsPositions`, catch and log `ErrActiveRedelegation` from `ForceFullExitWithDelegation` instead of propagating it, and skip the position (or defer it). Alternatively, remove the `isRedelegating` guard from `transferDelegationFromPosition` when called from the migration path, since the migration is a privileged, non-user-driven path where the slash-escape concern that motivated the guard does not apply.

---

### Proof of Concept

**Exact call chain with file and line references:**

1. Attacker submits `MsgTierRedelegate` for a vesting-owned position. [1](#0-0) 
   `validateRedelegatePosition` is called — no vesting account check present. [2](#0-1) 

2. `BeginRedelegation` creates a staking redelegation entry for `pos.DelegatorAddress` with a 21-day completion time. [3](#0-2) 

3. At upgrade height, `Migrate1to2` calls `exitVestedAccountsPositions`, which calls `ForceFullExitWithDelegation` for each vesting-owned position. [4](#0-3) 

4. `ForceFullExitWithDelegation` calls `transferDelegationFromPosition`. [5](#0-4) 

5. `transferDelegationFromPosition` calls `isRedelegating` and returns `ErrActiveRedelegation`. [6](#0-5) 

6. The error propagates through `Migrate` back to the upgrade handler, which returns it. [7](#0-6) [8](#0-7) 

The existing test `TestTransferDelegationFromPosition_ActiveRedelegation` already proves that `transferDelegationFromPosition` returns `ErrActiveRedelegation` when a redelegation is active — the migration path has no special handling to bypass this guard. [9](#0-8)

func (k Keeper) validateRedelegatePosition(ctx context.Context, pos types.PositionState, owner, dstValidator string) error {
	if !pos.IsOwner(owner) {
		return types.ErrNotPositionOwner
	}

	if !pos.IsDelegated() {
		return types.ErrPositionNotDelegated
	}

	if pos.Delegation.ValidatorAddress == dstValidator {
		return types.ErrRedelegationToSameValidator
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if pos.HasTriggeredExit() && pos.CompletedExitLockDuration(sdkCtx.BlockTime()) {
		return types.ErrExitLockDurationElapsed
	}

	tier, err := k.getTier(ctx, pos.TierId)
	if err != nil {
		return err
	}

	if tier.IsCloseOnly() {
		return types.ErrTierIsCloseOnly
	}

	isRedelegating, err := k.isRedelegating(ctx, pos.DelegatorAddress)
	if err != nil {
		return err
	}
	if isRedelegating {
		return errorsmod.Wrapf(types.ErrActiveRedelegation, "position %d has an active redelegation", pos.Id)
	}

	return nil
}

func (s *KeeperSuite) TestTransferDelegationFromPosition_ActiveRedelegation() {
	lockAmount := sdkmath.NewInt(10000)
	pos := s.setupNewTierPosition(lockAmount, true)
	_, bondDenom := s.getStakingData()
	s.fundRewardsPool(sdkmath.NewInt(1_000_000), bondDenom)

	// Redelegate BEFORE exit elapses (redelegate is blocked after exit elapsed).
	dstValAddr, _ := s.createSecondValidator()
	msgServer := keeper.NewMsgServerImpl(s.keeper)
	_, err := msgServer.TierRedelegate(s.ctx, &types.MsgTierRedelegate{
		Owner:        pos.Owner,
		PositionId:   pos.Id,
		DstValidator: dstValAddr.String(),
	})
	s.Require().NoError(err)

	// Now advance past exit duration.
	s.advancePastExitDuration()

	// Re-fetch position after redelegate (validator changed).
	pos, err = s.keeper.GetPositionState(s.ctx, pos.Id)
	s.Require().NoError(err)

	newValAddr := sdk.MustValAddressFromBech32(pos.Delegation.ValidatorAddress)
	_, _, _, err = s.keeper.TransferDelegationFromPosition(s.ctx, pos, newValAddr, s.getPositionAmount(pos))
	s.Require().ErrorIs(err, types.ErrActiveRedelegation)
