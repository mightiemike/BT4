Audit Report

## Title
Stale `lastDecayTs` After `executeOracleStopLossDecay` Retroactively Misapplies New Rate, Silently Disabling the Stop-Loss Guard — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`executeOracleStopLossDecay()` writes the new `decayPerSecondE8` into `oracleStopLossConfig` but never resets `highWatermarks[pool][binIdx].lastDecayTs` for any bin. On the next swap touching an idle bin, `_checkAndUpdateWatermarks` computes `dt = block.timestamp - hwmS.lastDecayTs` spanning the entire pre-change idle period and applies the new (higher) rate over that full `dt`. When `newRate × dt ≥ 1e8`, `_decayed()` returns `0`, zeroing the watermark and causing `_applyWatermark` to never report a breach, silently disabling the stop-loss guard for those bins.

## Finding Description

**Root cause — `executeOracleStopLossDecay` (L139–147):** [1](#0-0) 

Only `oracleStopLossConfig[pool_].decayPerSecondE8` is updated. No bin's `lastDecayTs` is touched.

**Lazy `dt` computation — `_checkAndUpdateWatermarks` (L267–268):** [2](#0-1) 

`dt` spans from the last swap in that bin to `block.timestamp`, which includes the entire idle period before the rate change.

**Zeroing condition — `_decayed` (L319–324):** [3](#0-2) 

When `newRate × dt ≥ 1e8`, the function returns `0`.

**Guard bypass — `_applyWatermark` (L328–336):** [4](#0-3) 

With `hwm == 0`, `metric >= 0` is always true for `uint256`, so `_applyWatermark` returns `(metric, false)` — `breached` is never set, and the stop-loss revert is never triggered.

**Exploit path:**
1. Bin X has `lastDecayTs = T0`, watermark `W > 0`, old rate `r_old`.
2. No swaps occur in bin X for `Δt` seconds.
3. Pool admin proposes and (after timelock) executes a rate increase to `r_new` where `r_new × Δt ≥ 1e8`.
4. Any unprivileged caller triggers a swap touching bin X.
5. `_decayed(W, r_new, Δt)` returns `0`; watermark is zeroed; stop-loss guard is permanently disabled for bin X until a new watermark is manually set.

The `_validateDecay` guard only enforces `decayPerSecondE8 ≤ 1e8` (≤ 100%/second), which does not prevent the retroactive zeroing. [5](#0-4) 

## Impact Explanation

Once a bin's watermark is zeroed, `_applyWatermark` never detects a breach regardless of how far the per-share value falls. Value-draining swaps proceed without triggering `OracleStopLossTriggered`, exposing LP principal to losses beyond the configured `drawdownE6` threshold with no on-chain protection. This is a direct loss of LP principal exceeding the protocol's own configured safety bound.

## Likelihood Explanation

The pool admin is semi-trusted and legitimately adjusts the decay rate over the pool's lifetime. The timelock delays execution but does not prevent the retroactive misapplication. Any bin idle since before the rate change is vulnerable. In low-activity pools or during quiet market periods, bins can be idle for days; a modest rate increase (e.g., from 58 to 580 per second) combined with a 3-day idle period satisfies `580 × 259200 = 150,336,000 ≥ 1e8`. The triggering swap can be executed by any unprivileged caller immediately after the rate change takes effect.

## Recommendation

Before writing the new `decayPerSecondE8` in `executeOracleStopLossDecay`, checkpoint every active bin's watermark by applying the current (old) rate up to `block.timestamp` and resetting `lastDecayTs`. Since active bins are not enumerated on-chain, the minimal safe fix is one of:

1. **Piecewise decay:** Store the rate-change timestamp alongside the rate and apply rate₁ for `[lastDecayTs, changeTs]` and rate₂ for `[changeTs, now]` inside `_decayed`.
2. **Checkpoint function:** Add `checkpointBinWatermark(address pool_, int8 binIdx)` that applies the current rate and resets `lastDecayTs`, and require all active bins to be checkpointed before `executeOracleStopLossDecay` is callable.
3. **Conservative reset:** Inside `executeOracleStopLossDecay`, reset `lastDecayTs` to `block.timestamp` for all bins (treating the pre-change period as zero-decay), preventing retroactive application of the new rate.

## Proof of Concept

```
Setup:
  decayPerSecondE8 = 58   (~5%/day)
  drawdownE6       = 100_000 (10%)
  highWatermarks[pool][X] = {token0: 1_000_000, token1: 1_000_000, lastDecayTs: T0}

Step 1 (T0 → T0+3days): No swaps in bin X. lastDecayTs[X] = T0.

Step 2 (T0+3days): Admin proposes decayPerSecondE8 = 580 (~50%/day).
  Timelock elapses; executeOracleStopLossDecay() called.
  oracleStopLossConfig[pool].decayPerSecondE8 = 580
  lastDecayTs[X] still = T0.

Step 3 (T0+3days+1s): Unprivileged caller triggers swap in bin X.
  dt     = 259_201 s
  factor = 580 × 259_201 = 150_336_580 ≥ 1e8
  _decayed(1_000_000, 580, 259_201) → 0

Step 4: _applyWatermark(currentMetric, 0, floorMultiplier):
  currentMetric >= 0 always → returns (currentMetric, false)
  breached = false → no revert

Result: Stop-loss guard disabled for bin X. Value-draining swaps
  proceed unchecked; LP principal drains beyond 10% drawdown threshold.
```
