Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address as `sender` instead of actual end user, allowing full allowlist bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` passes `msg.sender` (the immediate caller) as the `sender` argument to `_beforeSwap`, which forwards it verbatim to `SwapAllowlistExtension.beforeSwap`. When users route through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the extension checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actual_user]`. Any pool admin who allowlists the router to enable router-based trading simultaneously grants every unprivileged user on-chain the ability to bypass the per-user swap allowlist entirely.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension via `_callExtensionsInOrder`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the extension's caller) and `sender` is whoever called `pool.swap`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap` directly: [4](#0-3) 

The router is therefore `msg.sender` to the pool. The pool passes the router address as `sender` to `_beforeSwap`, and the extension evaluates `allowedSwapper[pool][router]`. If the pool admin has allowlisted the router (the natural operational action to enable router-based trading), this check passes for every caller regardless of whether they are individually allowlisted.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` checks the `owner` parameter (the position owner, passed explicitly by the caller), not the `sender` (the `msg.sender` of `addLiquidity`), so it is not affected: [5](#0-4) 

The swap path has no equivalent forwarding of the true caller identity. There is no existing guard that recovers the originating user address before the allowlist check.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a specific set of addresses (e.g., KYC'd counterparties, whitelisted market makers) can be fully bypassed by any unprivileged user routing through the public `MetricOmmSimpleRouter`. The bypassing user executes swaps against the pool's liquidity, extracting tokens from LPs who deposited under the assumption that only allowlisted parties could trade. This constitutes direct loss of LP principal through unauthorized swap execution — a broken core pool access-control invariant with High/Medium impact depending on pool TVL and allowlist strictness. This matches the "Broken core pool functionality causing loss of funds" and "Admin-boundary break" allowed impact criteria.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing entry point for swaps. Any pool admin who enables `SwapAllowlistExtension` and also wants users to trade via the router must allowlist the router — at which point the bypass is immediately active for all users. No special privileges are required: any EOA can call the router. The only precondition is that the pool admin has allowlisted the router, which is the expected and natural operational action. The bypass is repeatable and requires no special on-chain state beyond this standard configuration.

## Recommendation

The pool must forward the true originating user identity to extensions, not just `msg.sender`. Two approaches:

1. **Router passes the real sender via `extensionData`**: The router encodes the actual user address (`msg.sender` at the router level) into `extensionData`, and `SwapAllowlistExtension.beforeSwap` decodes and checks it. This requires a convention between router and extension, and must ensure untrusted callers cannot spoof the encoded address (e.g., by only trusting the decoded value when `sender` is a factory-registered trusted router).

2. **Pool exposes a `senderForExtensions` parameter**: Add an optional argument to `pool.swap` that trusted routers can populate with the real user address. The pool validates that if `senderForExtensions != address(0)`, `msg.sender` is a factory-registered trusted router before forwarding it to extensions.

Either approach must ensure the override path cannot be spoofed by an untrusted caller.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension as beforeSwap hook
  - Pool admin calls swapExtension.setAllowedToSwap(pool, router, true)
    (allowlisting the router so users can trade via it)
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  1. Attacker (not individually allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle(pool, ...)
  2. Router calls pool.swap(recipient=attacker, ...)
       → pool passes msg.sender (= router) as `sender` to _beforeSwap
  3. SwapAllowlistExtension.beforeSwap checks:
       allowedSwapper[pool][router] == true  ✓ (router is allowlisted)
       → hook passes, swap executes
  4. Attacker receives output tokens; pool LPs bear the trade

Result: Attacker swapped in a pool intended to be restricted to specific users.
        The per-user allowlist provided zero protection.

Foundry test outline:
  - deployPool(extensions=[swapAllowlist])
  - swapAllowlist.setAllowedToSwap(pool, router, true)
  - vm.prank(attacker); router.exactInputSingle(pool, ...)
  - assertGt(token1.balanceOf(attacker), 0)  // swap succeeded despite no individual allowlist entry
```
