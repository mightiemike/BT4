### Title
Integer Overflow in `op_unknown` Final Cost Multiplication Bypasses Cost Limit — (`File: src/more_ops.rs`)

### Summary

`op_unknown` in `src/more_ops.rs` uses a plain `*=` multiplication for its final cost calculation (`cost *= cost_multiplier + 1`), while every other arithmetic-sensitive path in the codebase uses `checked_add`, `checked_mul`, or `saturating_add/mul`. When the product overflows `u64`, the wrapped result can fall below the `u32::MAX` guard that follows, causing the function to return `Ok` with a near-zero cost for an operation whose true cost is enormous. This is the direct analog of the SafeERC20 pattern: a safe arithmetic API is consistently used everywhere else, but the raw/unchecked operator is used in this one critical path.

---

### Finding Description

In `op_unknown`, after computing the base cost through one of four cost functions (each guarded by `check_cost`), the code applies the opcode's cost multiplier:

```rust
// src/more_ops.rs lines 260-266
check_cost(cost, max_cost)?;
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`cost_multiplier` is a `u64` derived from `u32_from_u8`, so its maximum value is `u32::MAX = 4_294_967_295`, making `cost_multiplier + 1` at most `2^32 = 4_294_967_296`. `cost` before the multiplication is bounded by `max_cost` (via `check_cost`). Chia's mempool cost limit is ~11 billion (`1.1 × 10^10`). The maximum product is therefore:

```
1.1 × 10^10  ×  4.3 × 10^9  ≈  4.7 × 10^19  >  u64::MAX ≈ 1.84 × 10^19
```

In a release build, Rust wraps silently. The wrapped value can be ≤ `u32::MAX`, passing the guard and returning `Ok(Reduction(wrapped_cost, nil))` with a cost that is orders of magnitude smaller than the true cost.

**Concrete trigger**: set `cost_multiplier = u32::MAX` (opcode bytes `[0xff, 0xff, 0xff, 0xff, XX]` where `XX` encodes cost_function) and drive `cost` to any multiple of `2^32` via the argument list. The product is `k × 2^32 × 2^32 = k × 2^64 ≡ 0 (mod 2^64)`. The guard `0 > u32::MAX` is false, so the function returns `Ok(Reduction(0, nil))` — zero cost for an arbitrarily expensive operation.

**Contrast with the rest of the codebase** — the same file and its dependencies consistently use safe arithmetic:

- `op_add` fast path: `total.checked_add(val as u64)` [1](#0-0) 
- `op_subtract` fast path: `total.checked_sub(val as i64)` [2](#0-1) 
- `serde_2026/de.rs` index arithmetic: `n.checked_neg().and_then(|x| x.checked_sub(2))` [3](#0-2) 
- `object_cache.rs` serialized-length accumulation: `1_u64.saturating_add(*left_value).saturating_add(*right_value)` [4](#0-3) 
- `tree_cache.rs` serialized-length accumulation: `1_u64.saturating_add(left.serialized_length.saturating_add(right.serialized_length))` [5](#0-4) 

The vulnerable line is the sole exception: [6](#0-5) 

---

### Impact Explanation

An attacker who controls CLVM bytes can submit a program containing a crafted unknown opcode. The program executes with a reported cost of 0 (or any small value ≤ `u32::MAX`) while the true computational work corresponds to a cost far exceeding the network's limit. This enables:

1. **Undercharged execution / cost-limit bypass**: the mempool and block validators accept the transaction as within budget.
2. **Consensus divergence**: any validator using a Python reference implementation or a future corrected Rust build computes the correct (enormous) cost and rejects the block, splitting consensus.
3. **DoS amplification**: an attacker can pack many such operations into a single block, forcing validators to perform unbounded work at zero declared cost.

---

### Likelihood Explanation

`op_unknown` is reachable in lenient/mempool mode via any attacker-supplied CLVM blob. The opcode byte layout is fully documented in the function's own comments, making the cost_multiplier field trivially controllable. Driving `cost` to a specific value before multiplication requires iterating over the cost formula (which is also documented), but is straightforward for an attacker with knowledge of the cost constants. No privileged access, social engineering, or dependency compromise is required — only the ability to submit a transaction.

---

### Recommendation

Replace the plain multiplication with a checked variant and treat overflow as an invalid opcode:

```rust
// src/more_ops.rs — op_unknown, after check_cost(cost, max_cost)?
let cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

This is consistent with every other arithmetic-sensitive path in the codebase.

---

### Proof of Concept

Craft an unknown opcode atom with:
- bytes `[0xff, 0xff, 0xff, 0xff, 0x00]`: last byte `0x00` → `cost_function = 0`, cost before multiplication = `1`; first four bytes → `cost_multiplier = 0xffffffff = u32::MAX`
- `cost_multiplier + 1 = 2^32`
- `cost * (cost_multiplier + 1) = 1 * 2^32 = 2^32`
- `2^32 > u32::MAX` → returns `EvalErr::Invalid` (this specific case is caught)

Adjust to cost_function=1 and supply `N` empty-atom arguments until `cost` reaches a value `v` such that `v * 2^32 mod 2^64 ≤ u32::MAX` (i.e., `v mod 2^32 = 0`). The smallest such `v` reachable under Chia's 11-billion cost limit is `v = 2^32 ≈ 4.3 × 10^9`, achievable with ~13.4 million single-byte arguments (each costing `ARITH_COST_PER_ARG + ARITH_COST_PER_BYTE = 323` units). The returned cost is `0`. [7](#0-6)
