### Title
Stale Chain Tip Used Without Freshness Check in SPV Proof Verification — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` serve SPV proof results to any unprivileged NEAR caller using the contract's stored chain tip, but neither function validates that the stored chain tip's timestamp is recent relative to the current NEAR block time. If the relayer stops submitting headers — even temporarily — the contract's view of the Bitcoin chain becomes stale, and the functions will continue returning `true` for transactions that may have been invalidated by a subsequent Bitcoin reorg.

---

### Finding Description

Both public verification entry points read the chain tip from `self.mainchain_tip_blockhash` and use it to count confirmations:

```rust
// contract/src/lib.rs  lines 294-308
let heaviest_block_header = self
    .headers_pool
    .get(&self.mainchain_tip_blockhash)
    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
let target_block_height = self
    .mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    ...;

require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
```

The `heaviest_block_header.block_header.time` field is a `u32` Unix timestamp stored in the `LightHeader`. Neither `verify_transaction_inclusion` nor `verify_transaction_inclusion_v2` compares this timestamp against `env::block_timestamp_ms() / 1000` to detect a stale tip.

By contrast, `check_pow` in every chain module does perform exactly this kind of wall-clock comparison for *incoming* headers:

```rust
// contract/src/bitcoin.rs  lines 35-39
let current_timestamp = u32::try_from(env::block_timestamp_ms() / 1000).unwrap();
require!(
    block_header.time <= current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL,
    "time-too-new: block timestamp too far in the future"
);
```

The same guard is present in `litecoin.rs` (lines 36–39), `dogecoin.rs` (lines 42–46), and `zcash.rs` (lines 48–52). It is entirely absent from the verification path.

---

### Impact Explanation

A downstream contract (bridge, escrow, cross-chain application) that calls `verify_transaction_inclusion_v2` and acts on a `true` result can be deceived into releasing funds for a Bitcoin transaction that was later invalidated by a reorg. The contract's confirmation count is computed relative to a stale tip, so the required depth (e.g., 6 confirmations) is satisfied against an outdated chain view while the real Bitcoin chain has already reorganized past that block.

---

### Likelihood Explanation

The relayer is an off-chain service that can experience downtime due to bugs, infrastructure failures, or network partitions. Bitcoin reorgs of 1–3 blocks occur naturally several times per year. If the relayer is down for even a few hours during such a reorg, the contract's tip will not reflect the reorg, and any caller can exploit the stale state by submitting a proof for the now-invalid transaction before the relayer catches up.

---

### Recommendation

Add a freshness check inside `verify_transaction_inclusion` (and propagated through `verify_transaction_inclusion_v2`) that compares the stored chain tip's timestamp against the current NEAR block time:

```rust
let current_timestamp = u32::try_from(env::block_timestamp_ms() / 1000).unwrap();
require!(
    heaviest_block_header.block_header.time + ACCEPTABLE_TIP_STALENESS >= current_timestamp,
    "chain tip is too stale to serve SPV proofs"
);
```

`ACCEPTABLE_TIP_STALENESS` should be a configurable parameter (e.g., 2–4 hours for Bitcoin, proportionally shorter for Dogecoin/Litecoin). This mirrors the existing `MAX_FUTURE_BLOCK_TIME_LOCAL` guard already applied to incoming headers.

---

### Proof of Concept

1. Contract is initialized with Bitcoin mainnet headers up to height N (tip timestamp T).
2. Relayer goes offline. Real Bitcoin chain advances to height N+10, including a 2-block reorg that removes the block at height N−1.
3. Attacker calls `verify_transaction_inclusion_v2` with a valid Merkle proof for a transaction in the now-reorged block at height N−1, requesting `confirmations = 2`.
4. Contract evaluates: `(N − (N−1)) + 1 = 2 >= 2` → passes the confirmation check.
5. No freshness check exists; `heaviest_block_header.block_header.time` is never compared to `env::block_timestamp_ms()`.
6. Function returns `true`. A downstream bridge contract releases funds for a Bitcoin transaction that no longer exists on the canonical chain.
