### Title
`MAX_FUTURE_BLOCK_TIME_MTP` Hardcoded 24× Too Small, Causing Legitimate Zcash Block Rejection — (`btc-types/src/network.rs`, `contract/src/zcash.rs`)

---

### Summary

The constant `MAX_FUTURE_BLOCK_TIME_MTP` is set to `90 * 60 = 5400` seconds in `btc-types/src/network.rs`, but the Zcash protocol's actual value for this parameter is `129600` seconds (36 hours). The code comment in `zcash.rs` even acknowledges the correct value. As a result, the Zcash `check_pow` function will reject valid Zcash block headers whose timestamp is more than 90 minutes ahead of the median-time-past (MTP) of the previous block, even though Zcash permits up to 36 hours. This is a direct structural analog to the `StrategyFloorFromChainlink` bug: a hardcoded ceiling is set to a value calibrated for a different context (or simply wrong), causing legitimate on-chain operations to revert.

---

### Finding Description

In `btc-types/src/network.rs`, two future-block-time constants are defined:

```rust
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 90 * 60;   // 5400 seconds
pub const MAX_FUTURE_BLOCK_TIME_LOCAL: u32 = 2 * 60 * 60; // 7200 seconds
``` [1](#0-0) 

`MAX_FUTURE_BLOCK_TIME_MTP` is consumed exclusively in the Zcash `check_pow` path:

```rust
// MAX_FUTURE_BLOCK_TIME_MTP is typically 129600 seconds (36 hours) in Zcash
require!(
    block_header.time
        <= next_work_result.prev_block_median_time_past + MAX_FUTURE_BLOCK_TIME_MTP,
    "time-too-far-ahead-of-mtp: block timestamp is too far ahead of median-time-past"
);
``` [2](#0-1) 

The inline comment explicitly states the correct Zcash protocol value is `129600` seconds (36 hours), yet the constant is `5400` seconds — a factor of **24× too small**. The Zcash protocol's `nMaxFutureBlockTime` for the MTP-based soft-fork rule is 36 hours, not 90 minutes.

`check_pow` is called from `submit_block_header` (the non-dogecoin path in `lib.rs`) whenever `skip_pow_verification` is `false`, which is the production setting:

```rust
if !skip_pow_verification {
    self.check_target(&header, &prev_block_header);
    ...
}
``` [3](#0-2) 

`check_target` delegates to `check_pow`:

```rust
fn check_target(&self, block_header: &Header, prev_block_header: &ExtendedHeader) {
    self.check_pow(block_header, prev_block_header);
}
``` [4](#0-3) 

---

### Impact Explanation

Any valid Zcash block header whose `time` field exceeds `prev_block_median_time_past + 5400` will be unconditionally rejected by the contract with `"time-too-far-ahead-of-mtp"`, even though the Zcash protocol permits up to `prev_block_median_time_past + 129600`. This corrupts the contract's ability to advance its canonical Zcash chain tip: the relayer cannot submit the block, the mainchain tip stalls, and any downstream consumer calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for blocks beyond the stall point will also fail. The broken invariant is: **the contract's accepted canonical chain tip diverges from the real Zcash mainchain**.

---

### Likelihood Explanation

Zcash targets a 75-second block interval. Under normal conditions the MTP (median of the last 11 block timestamps) lags the chain tip by roughly 5–6 blocks (~375–450 s), so a new block's timestamp is typically only a few hundred seconds ahead of MTP — well within 5400 s. However, the condition triggers whenever the Zcash network experiences a sustained hashrate drop (e.g., a mining pool going offline), causing blocks to arrive slowly. If 11 consecutive blocks each take ~8–10 minutes, the MTP falls ~55–90 minutes behind the current wall-clock time. A legitimately timestamped new block would then be rejected. This is a realistic operational scenario, not a theoretical one, and it requires no attacker: any honest relayer submitting a valid Zcash block header during a slow-block period will hit the revert.

---

### Recommendation

Change the constant to match the actual Zcash protocol value:

```rust
// btc-types/src/network.rs
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 129_600; // 36 hours, per Zcash protocol
```

If the constant is intended to be chain-agnostic and reused across chains, it should be moved into the per-chain config struct (e.g., `ZcashConfig`) so each chain can carry its own correct value, mirroring how `pow_target_spacing` and `pow_averaging_window` are already chain-specific.

---

### Proof of Concept

1. Deploy the Zcash-feature contract with `skip_pow_verification = false`.
2. Pre-populate the chain with 11 + `pow_averaging_window` (= 17) = 28 Zcash headers whose timestamps are spaced ~10 minutes apart (simulating a slow-hashrate period).
3. After 11 such blocks, the MTP is the median of those 11 timestamps, lagging the latest block by ~55 minutes.
4. Submit the next valid Zcash block header with `time = MTP + 5401` (one second beyond the contract's limit, but still within Zcash's 36-hour allowance).
5. The contract panics with `"time-too-far-ahead-of-mtp"` even though the block is fully valid per the Zcash protocol.
