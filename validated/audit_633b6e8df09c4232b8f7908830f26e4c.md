### Title
Unremoved Deprecated `verify_transaction_inclusion` Accepts Internal Merkle Node as Valid Transaction — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` remains a live, publicly callable NEAR contract method despite being marked `#[deprecated]`. It performs no check that the supplied `tx_id` is a leaf-level transaction hash. Any unprivileged caller can supply a serialized internal Merkle tree node as `tx_id` with a crafted one-element proof and receive `true`, falsely proving inclusion of a transaction that does not exist.

---

### Finding Description

`verify_transaction_inclusion` is declared `pub fn` inside a `#[near]` impl block with only a `#[pause]` guard and a Rust `#[deprecated]` attribute. [1](#0-0) 

`#[deprecated]` in Rust is a **compile-time lint only**. It emits a warning when the function is called from Rust source, but it does not remove the method from the compiled WASM binary or from the contract's public ABI. Any NEAR account can invoke it via RPC regardless of the deprecation annotation.

The `#[pause]` attribute blocks calls only when the contract has been administratively paused; in normal operation it imposes no caller restriction. [2](#0-1) 

The function body delegates entirely to `merkle_tools::compute_root_from_merkle_proof`, passing `args.tx_id` verbatim as the starting hash: [3](#0-2) 

`compute_root_from_merkle_proof` performs no validation that its `transaction_hash` argument is a leaf node. It simply iterates the proof path and hashes upward: [4](#0-3) 

The code's own doc comment acknowledges the flaw explicitly: [5](#0-4) 

---

### Impact Explanation

For any canonical block B containing transactions `[tx0, tx1, tx2, tx3]`:

```
node_A = double_sha256(tx0 || tx1)   // level-1 internal node
node_B = double_sha256(tx2 || tx3)   // level-1 internal node
root   = double_sha256(node_A || node_B)
```

An attacker calls:

```
verify_transaction_inclusion({
    tx_id:              node_A,
    tx_block_blockhash: B.hash,
    tx_index:           0,
    merkle_proof:       [node_B],
    confirmations:      1,
})
```

`compute_root_from_merkle_proof(node_A, 0, [node_B])` computes `double_sha256(node_A || node_B)` = `root`, which equals `header.block_header.merkle_root`. The function returns `true`.

`node_A` is not a transaction. It is a 32-byte hash of two concatenated 32-byte hashes — a 64-byte preimage — which is the exact forgery vector described at https://www.bitmex.com/blog/64-Byte-Transactions. Any downstream bridge, payment gate, or NEAR contract that calls `verify_transaction_inclusion` to authorize an action based on a claimed BTC payment will accept this forged proof. [6](#0-5) 

---

### Likelihood Explanation

- No privileged role is required. Any NEAR account can call the method.
- The only precondition is that `tx_block_blockhash` belongs to the canonical chain, which is trivially satisfied by any recently relayed Bitcoin block.
- The internal node values are fully deterministic and publicly computable from the block's transaction list (available from any Bitcoin node or block explorer).
- The `#[deprecated]` annotation provides zero runtime protection.

---

### Recommendation

**Remove `verify_transaction_inclusion` from the contract entirely.** Because `#[deprecated]` does not prevent on-chain invocation, the only safe remediation is to delete the method or replace its body with an unconditional `panic!("use verify_transaction_inclusion_v2")`. Keeping it as a deprecated-but-callable method leaves the forgery surface permanently open.

If removal is not immediately possible, add an explicit runtime guard at the top of the function body:

```rust
env::panic_str("verify_transaction_inclusion is disabled; use verify_transaction_inclusion_v2");
```

The fixed v2 path already exists and correctly mitigates the attack by requiring a valid coinbase proof at the same tree depth: [7](#0-6) 

---

### Proof of Concept

```rust
#[test]
fn internal_node_forgery_returns_true() {
    use merkle_tools::{compute_root_from_merkle_proof, double_sha256, H256};

    // Build a 4-tx Merkle tree
    let tx = |b: u8| H256([b; 32]);
    let tx0 = tx(0x01); let tx1 = tx(0x02);
    let tx2 = tx(0x03); let tx3 = tx(0x04);

    let concat = |a: &H256, b: &H256| -> H256 {
        let mut v = Vec::with_capacity(64);
        v.extend(a.0); v.extend(b.0);
        double_sha256(&v)
    };

    let node_a = concat(&tx0, &tx1); // internal node at level 1
    let node_b = concat(&tx2, &tx3); // internal node at level 1
    let root   = concat(&node_a, &node_b);

    // Submit internal node as tx_id with a 1-element proof
    let computed = compute_root_from_merkle_proof(node_a.clone(), 0, &vec![node_b]);

    // This assertion passes — internal node accepted as a valid transaction
    assert_eq!(computed, root);
}
```

This test passes against the unmodified `merkle-tools/src/lib.rs` code. [8](#0-7)

```rust
#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```
