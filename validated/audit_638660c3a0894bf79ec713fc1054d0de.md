### Title
Unprotected `initialize()` in `SpotEngine` and `PerpEngine` Allows Frontrunning to Block `Clearinghouse.addEngine()` - (File: `core/contracts/SpotEngine.sol`, `core/contracts/PerpEngine.sol`)

---

### Summary

`SpotEngine.initialize()` and `PerpEngine.initialize()` are `external` functions with no caller access control. `Clearinghouse.addEngine()` deploys these engines separately and then calls `productEngine.initialize()` to configure them. An attacker can observe the engine deployment transaction in the mempool and frontrun the subsequent `addEngine()` call by directly invoking `initialize()` on the freshly deployed engine contract. Because the `initializer` modifier (inherited via `BaseEngine._initialize()`) prevents re-initialization, the legitimate call from `Clearinghouse.addEngine()` will revert, forcing the deployer to redeploy the engine. This process can be repeated indefinitely at lower cost to the attacker than to the protocol team.

---

### Finding Description

`Clearinghouse.addEngine()` is the sole legitimate caller of engine initialization: [1](#0-0) 

It calls `productEngine.initialize(...)` after registering the engine. The engine contracts are deployed separately before this call, creating a window for frontrunning.

`SpotEngine.initialize()` is declared `external` with no access control modifier: [2](#0-1) 

`PerpEngine.initialize()` has the same pattern: [3](#0-2) 

Both delegate to `BaseEngine._initialize()`, which carries the `initializer` modifier: [4](#0-3) 

Neither `SpotEngine` nor `PerpEngine` defines a constructor with `_disableInitializers()`. Contrast this with contracts that are properly protected — `Verifier`, `BaseWithdrawPool`, `BaseProxyManager`, `ContractOwner`, and `Airdrop` all call `_disableInitializers()` in their constructors: [5](#0-4) [6](#0-5) 

Furthermore, `SpotEngine` and `PerpEngine` are not listed among the proxy-managed contracts in `BaseProxyManager` (which only tracks `Clearinghouse`, `ClearinghouseLiq`, `Endpoint`, `EndpointTx`): [7](#0-6) 

This confirms the engines are deployed as standalone contracts, making the uninitialized window directly exploitable.

---

### Impact Explanation

An attacker who frontruns `SpotEngine.initialize()` or `PerpEngine.initialize()` with attacker-controlled parameters (e.g., setting themselves as `_admin`) causes the `initializer` guard to mark the contract as already initialized. When `Clearinghouse.addEngine()` subsequently calls `productEngine.initialize(...)`, it reverts unconditionally. The deployer must redeploy the engine contract and retry. The attacker can repeat this at a fraction of the deployer's gas cost, potentially draining protocol deployment funds and indefinitely delaying the protocol launch. Additionally, if the attacker sets themselves as `_admin` (owner) of the engine, the compromised contract could be used to manipulate product risk parameters if it were ever mistakenly registered.

---

### Likelihood Explanation

The attack requires only mempool monitoring and a standard frontrunning transaction. No special privileges, leaked keys, or governance access are needed. The deployment sequence (engine deployed → `addEngine()` called) is predictable and observable on-chain. On EVM-compatible chains like Ink Chain, transaction ordering can be influenced by gas price, making this straightforward to execute.

---

### Recommendation

Add `_disableInitializers()` to constructors in `SpotEngine` and `PerpEngine` (and their base `BaseEngine`) to prevent direct initialization of implementation contracts:

```solidity
/// @custom:oz-upgrades-unsafe-allow constructor
constructor() {
    _disableInitializers();
}
```

Alternatively, add an access control check inside `initialize()` to restrict the caller to the expected deployer or factory address, consistent with the pattern used in `ContractOwner.initialize()`: [8](#0-7) 

---

### Proof of Concept

1. Deployer broadcasts a transaction deploying `SpotEngine` at address `0xSPOT`.
2. Attacker observes the pending deployment in the mempool.
3. Attacker submits `SpotEngine(0xSPOT).initialize(attacker, attacker, attacker, attacker, attacker)` with higher gas price.
4. Attacker's transaction is mined first; `BaseEngine._initialize()` runs with attacker parameters, setting `initialized = true`.
5. Deployer's `Clearinghouse.addEngine(0xSPOT, ...)` is mined next; it calls `SpotEngine(0xSPOT).initialize(...)` which reverts: `"Initializable: contract is already initialized"`.
6. Deployer must redeploy `SpotEngine` and repeat from step 1. Attacker repeats step 3 each time. [9](#0-8)
