### Title
Deprecated `verify_transaction_inclusion` Lacks Coinbase Proof Validation Present in v2, Enabling Merkle Proof Forgery — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (v1) remains publicly callable by any unprivileged NEAR account despite being deprecated. Unlike `verify_transaction_inclusion_v2`, it performs no coinbase Merkle proof check, leaving it fully exposed to the 64-byte transaction Merkle proof forgery attack. An attacker can supply an internal Merkle tree node as `tx_id` and receive a `true` verification result for a transaction that does not exist, corrupting any downstream NEAR contract that trusts the result.

### Finding Description

The contract exposes two public verification entry points:

`verify_transaction_inclusion` (v1) — still `pub`, gated only by `#[pause]` (active by default): [1](#0-0) 

It computes the Merkle root directly from the caller-supplied `tx_id` and `merkle_proof` with no additional constraint: [2](#0-1) 

The codebase's own docstring acknowledges the gap:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [3](#0-2) 

`verify_transaction_inclusion_v2` — the fixed version — closes the gap by requiring a coinbase Merkle proof that must independently reconstruct the same Merkle root before delegating to v1: [4](#0-3) 

The `#[deprecated]` Rust attribute is a compile-time lint only; it imposes no runtime restriction. The v1 function remains fully reachable at the NEAR RPC level by any account.

The underlying Merkle computation in `merkle-tools` is position-based and accepts any 32-byte value as the leaf: [5](#0-4) 

Because `compute_root_from_merkle_proof` treats `transaction_hash` as an opaque 32-byte value, an internal node hash (which is itself the double-SHA256 of two concatenated 32-byte children) is indistinguishable from a real transaction hash. An attacker who knows the Merkle tree of any confirmed Bitcoin block can pick any internal node, construct a valid sibling path to the root, and pass the check.

### Impact Explanation

Any NEAR DApp that calls `verify_transaction_inclusion` (v1) — for example, to gate an asset release or a state transition — can be made to accept a forged Bitcoin transaction inclusion proof. The function returns `true` for a `tx_id` that corresponds to no real Bitcoin transaction. The corrupted boolean result propagates directly to the caller's authorization logic, enabling unauthorized asset movement or false state transitions in downstream contracts.

Severity: **Medium** (matches the reference report's rating). The contract itself holds no user funds, but the verification result is the trust anchor for external NEAR DApps that do.

### Likelihood Explanation

- No privileged access is required; any NEAR account can call `verify_transaction_inclusion`.
- The attacker only needs knowledge of a confirmed Bitcoin block's Merkle tree, which is public.
- The 64-byte transaction forgery technique is well-documented (referenced in the codebase itself via the BitMEX link).
- The v1 function is still the default entry point for any integrator that has not explicitly migrated to v2.

### Recommendation

Remove or hard-disable `verify_transaction_inclusion` (v1) at the contract level — either by deleting the public method or by making it unconditionally panic with a migration message. Rust's `#[deprecated]` attribute does not prevent runtime calls. Keeping the unprotected path callable alongside the protected v2 path mirrors exactly the `Executor.sol` / `SwapperV2.sol` split identified in the reference report. [6](#0-5) 

### Proof of Concept

1. Identify any confirmed Bitcoin block `B` whose Merkle tree has depth ≥ 1 (i.e., more than one transaction).
2. Select any internal node `N` at depth 1: `N = double_sha256(tx0 || tx1)`.
3. Construct `merkle_proof` as the sibling path from `N` up to the Merkle root (standard SPV path, but starting one level above the leaves).
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node hash, not a real transaction ID)
   - `tx_block_blockhash` = hash of block `B` (must be in the mainchain)
   - `tx_index` = the position of `N` in the level-1 layer
   - `merkle_proof` = the sibling path constructed in step 3
   - `confirmations` = any value ≤ `gc_threshold`
5. The function returns `true`, confirming a transaction that does not exist. [7](#0-6)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
