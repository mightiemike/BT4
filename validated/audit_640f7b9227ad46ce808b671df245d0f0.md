### Title
Unprotected `initialize()` in `SpotEngine` and `PerpEngine` Allows Anyone to Seize Engine Ownership — (`File: core/contracts/SpotEngine.sol`, `core/contracts/PerpEngine.sol`)

---

### Summary

`SpotEngine.initialize()` and `PerpEngine.initialize()` are declared `external` with no access-control modifier and no OpenZeppelin `initializer` guard. Neither contract calls `_disableInitializers()` in its constructor. Any unprivileged caller can invoke `initialize()` before the legitimate deployer, seizing ownership of the engine and supplying attacker-controlled configuration.

---

### Finding Description

`SpotEngine.initialize()` is declared as:

```solidity
function initialize(
    address _clearinghouse,
    address _offchainExchange,
    address _quote,
    address _endpoint,
    address _admin
) external {
    _initialize(_clearinghouse, _offchainExchange, _endpoint, _admin);
    ...
}
``` [1](#0-0) 

`PerpEngine.initialize()` is identical in structure:

```solidity
function initialize(
    address _clearinghouse,
    address _offchainExchange,
    address,
    address _endpoint,
    address _admin
) external {
    _initialize(_clearinghouse, _offchainExchange, _endpoint, _admin);
}
``` [2](#0-1) 

Neither function carries an `initializer` modifier, an `onlyOwner` modifier, or any `require(msg.sender == ...)` guard. The grep results confirm that `_disableInitializers()` is absent from both `SpotEngine` and `PerpEngine` (it appears only in `BaseProxyManager`, `ContractOwner`, `Verifier`, `Airdrop`, and `BaseWithdrawPool`). [3](#0-2) 

Contrast this with contracts that are correctly protected: `Verifier` calls `_disableInitializers()` in its constructor and uses the `initializer` modifier, and `ContractOwner.initialize()` enforces `require(_deployer == msg.sender, "expected deployed to initialize")`. [4](#0-3) 

The `_admin` argument passed to `_initialize` becomes the `owner` of the engine via `__Ownable_init()` in `EndpointGated` / `OwnableUpgradeable`. [5](#0-4) 

---

### Impact Explanation

An attacker who seizes ownership of `SpotEngine` can call `addOrUpdateProduct` (guarded only by `onlyOwner`) to overwrite risk parameters — `longWeightInitial`, `shortWeightInitial`, `longWeightMaintenance`, `shortWeightMaintenance`, `priceX18` — for any product, including the quote product (product 0). [6](#0-5) 

Setting extreme risk weights (e.g., `longWeightInitial = 0` or `priceX18 = 0`) causes `getHealthContribution` to return corrupted health values for all subaccounts, enabling:

- Immediate liquidation of all solvent positions (health reported as negative when it is not).
- Prevention of legitimate liquidations (health reported as positive when it is not).
- Theft of collateral via artificially inflated borrow capacity.

The `SpotEngine` also holds the quote-token configuration used by `BaseWithdrawPool.fastWithdrawalFeeAmount` and `getToken`, so corrupting `configs[QUOTE_PRODUCT_ID]` breaks all withdrawal paths. [7](#0-6) 

For `PerpEngine`, ownership seizure allows manipulation of `addOrUpdateProduct` for all perp markets, corrupting `cumulativeFundingLongX18`/`cumulativeFundingShortX18` accounting and open-interest tracking. [8](#0-7) 

---

### Likelihood Explanation

The attack requires only a standard EOA and a single transaction submitted before the legitimate deployer's initialization transaction. On any public mempool chain this is trivially achievable via front-running. No privileged access, leaked keys, or social engineering is required. The window exists from the moment the proxy or implementation contract is deployed until `initialize()` is called by the deployer.

---

### Recommendation

1. Add `_disableInitializers()` to the constructors of `SpotEngine` and `PerpEngine` (as is already done in `Verifier`, `Airdrop`, `BaseWithdrawPool`, etc.) to prevent initialization of the implementation contract itself.
2. Add the OpenZeppelin `initializer` modifier to both `SpotEngine.initialize()` and `PerpEngine.initialize()` so the function can only execute once per proxy instance.
3. Optionally, add a `require(msg.sender == _admin, ...)` guard (mirroring `ContractOwner`'s pattern) so only the intended deployer can trigger initialization.

---
