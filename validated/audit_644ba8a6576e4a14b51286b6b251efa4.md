### Title
Unprivileged MsgSend to BurnedAddress Bypasses MaxSupply Cap, Enabling Minting Beyond Hard Limit — (`x/inflation/abci.go`)

### Summary

An unprivileged attacker holding bond-denom tokens can send them to any address listed in `params.BurnedAddresses` via a standard `MsgSend`. Because `BeginBlocker` subtracts the live balance of every `BurnedAddress` from `totalsupply` before comparing against `MaxSupply`, the attacker's deposit artificially deflates the effective circulating supply figure. This suppresses the MaxSupply panic, allowing the `x/mint` module (which runs **before** `x/inflation` in `BeginBlockers`) to mint tokens beyond the intended hard cap every subsequent block.

---

### Finding Description

**Execution order** is confirmed in `app/app.go`:

```
SetOrderBeginBlockers:
  minttypes.ModuleName,          // line 617 — mints first
  ...
  inflationtypes.ModuleName,     // line 637 — checks cap after minting
``` [1](#0-0) 

**BeginBlocker supply-cap logic** (`x/inflation/abci.go`):

```go
totalsupply, denom, err := k.GetSupplyAndDenom(ctx)   // bank.GetSupply — ALL tokens
burned := math.NewInt(0)
for _, ba := range params.BurnedAddresses {
    balance, _ := k.GetAddressBalance(ctx, ba, denom)  // live balance
    burned = burned.Add(balance)
}
totalsupply = totalsupply.Sub(burned)                  // effective = total - burned

if maxsupply.IsPositive() && totalsupply.GT(maxsupply) {
    panic(...)
}
``` [2](#0-1) 

**`BurnedAddresses` are plain bech32 accounts.** Validation only checks that they are valid bech32 and unique — they are never added to `BlockedAddrs` (which only contains module accounts): [3](#0-2) 

`BlockedAddrs` is wired to `BankKeeper` at construction time and does not include `BurnedAddresses`: [4](#0-3) 

**Attack path:**

1. Attacker holds X bond-denom tokens.
2. Attacker sends X tokens to a `BurnedAddress` via `MsgSend` (no special permission required).
3. `burned` increases by X; `effective = totalsupply − burned` decreases by X.
4. Next block: `x/mint` BeginBlocker mints new tokens (total supply grows).
5. `x/inflation` BeginBlocker computes `effective = (MaxSupply + minted) − (original_burned + X)`. If `X ≥ minted`, the check passes silently.
6. The attacker can repeat step 2 indefinitely, keeping `effective` below `MaxSupply` while true total supply grows without bound.

---

### Impact Explanation

- The `MaxSupply` hard cap — the only enforcement mechanism preventing unbounded minting — is defeated without any privileged access.
- True `bank.GetSupply()` exceeds `MaxSupply` by an amount proportional to the attacker's cumulative deposits.
- Newly minted tokens beyond the cap are real, spendable tokens distributed to stakers/validators, constituting unbacked supply inflation.
- ADR-007 explicitly states the chain halt is intentional to prevent further minting beyond the cap; this path silently bypasses it. [5](#0-4) 

---

### Likelihood Explanation

- Requires only a standard `MsgSend` transaction — no governance, no special role, no leaked keys.
- Any token holder can execute it. The cost is the deposited tokens (which are sacrificed to the burn address), but the attacker recovers value indirectly if they are a staker receiving the extra minted rewards.
- The attack is repeatable every block with arbitrarily small deposits.

---

### Recommendation

`BurnedAddresses` should be module accounts (or addresses explicitly added to `BlockedAddrs`) so that `MsgSend` to them is rejected at the bank layer. Alternatively, the supply-cap check should compare `bank.GetSupply()` directly against `MaxSupply` without subtracting burned-address balances, and a separate, privileged-only mechanism should handle the "circulating supply" concept. At minimum, `BurnedAddresses` must be immutable after genesis or require governance with a supply-consistency invariant check on update.

// Integration test sketch (unmodified Go/Cosmos test setup)
func TestMaxSupplyBypassViaBurnedAddress(t *testing.T) {
    app := testutil.Setup(false, nil)
    ctx := app.BaseApp.NewContext(false)

    burnAddr := sdk.AccAddress([]byte("burn_address_______"))
    currentSupply := app.BankKeeper.GetSupply(ctx, "basecro").Amount

    // Set MaxSupply = currentSupply (at the cap)
    params := types.NewParams(currentSupply, []string{burnAddr.String()}, sdk.ZeroDec())
    app.InflationKeeper.SetParams(ctx, params)

    // Fund attacker and send 1 token to burned address
    attacker := sdk.AccAddress([]byte("attacker___________"))
    banktestutil.FundAccount(app.BankKeeper, ctx, attacker, sdk.NewCoins(sdk.NewInt64Coin("basecro", 1)))
    app.BankKeeper.SendCoins(ctx, attacker, burnAddr, sdk.NewCoins(sdk.NewInt64Coin("basecro", 1)))
    // total supply is now currentSupply+1, burned=1, effective=currentSupply → no panic

    // Simulate mint module minting 1 token (total supply = currentSupply+1)
    // inflation BeginBlocker: effective = (currentSupply+1) - 1 = currentSupply ≤ MaxSupply → NO PANIC
    require.NotPanics(t, func() {
        inflation.BeginBlocker(ctx, app.InflationKeeper)
    })
    // True total supply now exceeds MaxSupply by 1, but no halt occurred.
}

func validateBurnedAddresses(v any) error {
	burnedAddresses, ok := v.([]string)
	if !ok {
		return fmt.Errorf("invalid parameter type: %T", v)
	}

	addressMap := make(map[string]bool)

	for i, addr := range burnedAddresses {
		if _, err := sdk.AccAddressFromBech32(addr); err != nil {
			return fmt.Errorf("invalid burned address at index %d: %s, error: %w", i, addr, err)
		}

		if addressMap[addr] {
			return fmt.Errorf("duplicate burned address found: %s", addr)
		}
		addressMap[addr] = true

	}

	return nil
}

## 3. Supply Cap (BeginBlocker)

Runs every block **after** the Mint module (which mints new tokens):

```
circulating_supply = bank.TotalSupply(bond_denom) - sum(bank.Balance(addr, bond_denom) for addr in BurnedAddresses)

```

The chain halt is intentional -- it prevents further minting beyond the cap. Recovery requires a governance upgrade or parameter change.
