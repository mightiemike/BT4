### Title
Reentrancy via CEI Violation in `withdrawCollateral` Enables Collateral Double-Spend — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs the token transfer (external call) **before** updating the subaccount balance in `SpotEngine`. No `nonReentrant` modifier exists anywhere in the codebase. If a supported collateral token implements ERC777-like transfer hooks, an attacker can reenter `Endpoint.depositCollateralWithReferral` during the transfer, queue a slow-mode deposit for the same amount, and after the delay recover their full balance — while also retaining the withdrawn tokens.

---

### Finding Description

`Clearinghouse.withdrawCollateral` violates the Checks-Effects-Interactions pattern:

```
handleWithdrawTransfer(token, sendTo, amount, idx);   // line 408 — external call FIRST
...
spotEngine.updateBalance(productId, sender, amountRealized);  // line 412 — state update AFTER
spotEngine.assertUtilization(productId);              // line 413
require(getHealth(sender, healthType) >= 0, ...);     // line 419
``` [1](#0-0) 

`handleWithdrawTransfer` in `Clearinghouse` first calls `token.safeTransfer(withdrawPool, amount)`, then calls `BaseWithdrawPool.submitWithdrawal`, which in turn calls `token.safeTransfer(sendTo, amount)` — delivering tokens to the recipient before any balance mutation occurs. [2](#0-1) [3](#0-2) 

`Endpoint.depositCollateralWithReferral` is `public` with no reentrancy guard, and immediately pulls tokens from `msg.sender` while queuing a slow-mode `DepositCollateral` credit: [4](#0-3) 

A grep across all production Solidity files confirms **zero uses** of `nonReentrant` or `ReentrancyGuard` in the entire codebase.

---

### Impact Explanation

An attacker who controls the `sendTo` address (their own contract) and whose collateral token fires a recipient hook on `transfer` can:

1. Withdraw X tokens — tokens land in their contract before balance is decremented.
2. During the hook, call `depositCollateralWithReferral` — X tokens are pulled back into `Clearinghouse` and a slow-mode deposit is queued.
3. Control returns to `withdrawCollateral` — balance is decremented to 0; health check passes (0 ≥ 0).
4. After `SLOW_MODE_TX_DELAY` (~3 days), the queued deposit is processed — balance is restored to X.
5. Attacker holds X tokens in their wallet **and** X balance in `SpotEngine`.
6. Repeat indefinitely.

Net result: unbounded collateral theft from the protocol's `Clearinghouse`.

---

### Likelihood Explanation

**Medium.** The attack requires a supported collateral token with ERC777-style transfer hooks. The `IERC20Base` interface imposes no restriction on token type, and the protocol is designed to support multiple assets. Any future listing of an ERC777-compatible token (or a token with a `tokensReceived`-style callback) immediately activates this path. The attacker needs only a valid withdrawal transaction (achievable via slow mode, no sequencer cooperation required) and a pre-approved allowance on `Endpoint`. [5](#0-4) 

---

### Recommendation

1. **Add `nonReentrant`** (OpenZeppelin `ReentrancyGuardUpgradeable`) to `Clearinghouse.withdrawCollateral` and `Endpoint.depositCollateralWithReferral` at minimum; audit all other public/external entry points.
2. **Follow CEI strictly** in `withdrawCollateral`: call `spotEngine.updateBalance` and perform the health check *before* calling `handleWithdrawTransfer`.
3. **Add `nonReentrant`** to `BaseWithdrawPool.submitFastWithdrawal`, which is also `public` with no guard and makes external token calls before completing its `fees` accounting update. [6](#0-5) 

---

### Proof of Concept

```
Setup:
  - Product P uses an ERC777-like token T.
  - Attacker deploys contract A (implements tokensReceived hook).
  - Attacker deposits X tokens into subaccount (A || name) via Endpoint.
  - Attacker pre-approves Endpoint to spend T from contract A.
  - Attacker submits slow-mode WithdrawCollateral(sender=(A||name), productId=P, amount=X).

After SLOW_MODE_TX_DELAY:

  [1] Anyone calls Endpoint._executeSlowModeTransaction (or sequencer processes it).
  [2] Endpoint → Clearinghouse.withdrawCollateral(sender, P, X, sendTo=A, idx)
  [3]   handleWithdrawTransfer(T, A, X, idx)
  [4]     T.safeTransfer(withdrawPool, X)
  [5]     WithdrawPool.submitWithdrawal(T, A, X, idx)
  [6]       T.safeTransfer(A, X)          ← ERC777 hook fires on A
  [7]         A.tokensReceived() {
  [8]           Endpoint.depositCollateralWithReferral(A||name, P, X, "")
  [9]             T.safeTransferFrom(A, Clearinghouse, X)   ← tokens returned
  [10]            slowModeTxs[n] = DepositCollateral(A||name, P, X)  ← queued
  [11]        }
  [12]  spotEngine.updateBalance(P, A||name, -X)   ← balance = 0
  [13]  getHealth(A||name, INITIAL) >= 0            ← passes (0 >= 0)

After another SLOW_MODE_TX_DELAY:

  [14] slowModeTxs[n] processed → spotEngine.updateBalance(P, A||name, +X)
       balance = X again

Result: A holds X tokens in wallet AND X balance in SpotEngine.
        Repeat from step [1].
```

        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
        // copy from submitSlowModeTransaction
        SlowModeConfig memory _slowModeConfig = slowModeConfig;

        slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: abi.encodePacked(
                uint8(TransactionType.DepositCollateral),
                abi.encode(
                    DepositCollateral({
                        sender: subaccount,
                        productId: productId,
                        amount: amount
                    })
                )
            )
        });
        slowModeConfig = _slowModeConfig;
    }
