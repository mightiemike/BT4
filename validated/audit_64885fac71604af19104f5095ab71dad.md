### Title
Insufficient Bonus Pool Causes Permanent Liveness Failure for Critical Exit/Undelegate Operations - (`x/tieredrewards/keeper/bonus_rewards.go`, `x/tieredrewards/keeper/msg_server.go`)

---

### Summary

When the tiered-rewards bonus pool is empty or insufficient, `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`. This error propagates unhandled through `claimRewards` into every critical user-facing message handler — `TierUndelegate`, `TierRedelegate`, `ExitTierWithDelegation`, `AddToTierPosition`, and `ClearPosition` — permanently blocking users from exiting or moving their locked positions. The codebase already demonstrates the correct fix in `slash.go`, where the same error is explicitly swallowed to prevent a chain halt, but that pattern was not applied to the user-facing paths.

---

---

### Impact Explanation

When the `RewardsPoolName` module account is depleted — a normal operational condition as bonus rewards are paid out over time — every call to `TierUndelegate`, `TierRedelegate`, `ExitTierWithDelegation`, `AddToTierPosition`, and `ClearPosition` reverts with `ErrInsufficientBonusPool`. Users cannot exit their locked positions, cannot undelegate their staked tokens, and cannot redelegate to a different validator. Their bonded tokens are frozen inside the tiered-rewards module until the pool is replenished by a governance action. The corrupted invariant is: **a user's right to exit a tier position and recover their delegation is unconditionally blocked by the state of an unrelated pool account**.

---

### Likelihood Explanation

The bonus pool is a finite, depletable module account. As positions accrue bonus rewards and claim them, the pool drains. No automatic refill mechanism exists in the BeginBlocker or message handlers. Once drained, every position that has unclaimed bonus accrual (i.e., every active delegated position) will fail all exit/undelegate/redelegate operations. This is a reachable steady-state condition requiring no privileged access — it occurs through normal protocol operation.

---

### Recommendation

In `processEventsAndClaimBonus`, or in each critical message handler, catch `ErrInsufficientBonusPool` and proceed without paying the bonus (log the shortfall), mirroring the pattern already used in `slash.go`. Bonus rewards are a supplemental incentive; their absence must never block the fundamental right to exit a position or recover a delegation.

---

### Proof of Concept

1. Deploy chain with a small `RewardsPoolName` balance.
2. Create several tier positions via `LockTier` or `CommitDelegationToTier`.
3. Wait for bonus accrual to exceed the pool balance.
4. Submit `MsgTierUndelegate` for any position.
5. `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`.
6. `TierUndelegate` returns the error; the user's delegation remains locked.
7. Repeat for `ExitTierWithDelegation`, `TierRedelegate`, `AddToTierPosition`, `ClearPosition` — all fail identically.
8. The user has no path to recover their staked tokens until governance replenishes the pool.

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
