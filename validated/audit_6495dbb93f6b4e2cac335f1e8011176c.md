### Title
Fast Withdrawal Fee Borne by Provider Instead of Withdrawer in Third-Party Calls — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.submitFastWithdrawal()` has two structurally inconsistent fee-handling paths depending on whether the caller is the withdrawer or a third-party provider. When a third-party provider calls the function (`sendTo != msg.sender`), the fast withdrawal fee is charged to the provider rather than deducted from the user's received amount. The user receives the full signed `transferAmount` without bearing any fee cost. This is the direct analog to the reported bug class: the fee is supposed to be paid by the withdrawer but is actually borne by the provider.

---

### Finding Description

`submitFastWithdrawal` branches on `sendTo == msg.sender`:

```solidity
int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

if (sendTo == msg.sender) {
    require(transferAmount > uint128(fee), "Fee larger than balance");
    transferAmount -= uint128(fee);          // user bears fee
} else {
    safeTransferFrom(token, msg.sender, uint128(fee)); // provider bears fee
}

fees[productId] += fee;
handleWithdrawTransfer(token, sendTo, transferAmount); // user gets full amount in Case 2
``` [1](#0-0) 

**Case 1 (`sendTo == msg.sender`):** The fee is subtracted from `transferAmount` before the transfer. The user receives `transferAmount − fee`. The fee stays in the contract.

**Case 2 (`sendTo != msg.sender`):** The fee is pulled from `msg.sender` (the provider) via `safeTransferFrom`. The user receives the full `transferAmount`. The provider pays the fee out of pocket.

The invariant that the withdrawer always bears the fast withdrawal fee is broken in Case 2. The user's signed transaction specifies `transferAmount` as the amount they want to withdraw. The fee is a protocol cost of the fast withdrawal service and should be deducted from what the user receives, regardless of who submits the transaction. Instead, the fee burden shifts entirely to the provider when a third party submits.

This is structurally identical to the reported bug: in the reference finding, commission was supposed to be paid by the buyer but was subtracted from the seller's proceeds. Here, the fast withdrawal fee is supposed to be paid by the withdrawer but is instead charged to the provider.

---

### Impact Explanation

A user whose withdrawal is submitted by a third-party provider receives `fee` more tokens than they should. The fast withdrawal fee rate is `FAST_WITHDRAWAL_FEE_RATE = 0.1%` with a floor of `5 × withdrawFeeX18`. [2](#0-1) [3](#0-2) 

For a 100,000 USDC withdrawal, the fee is 100 USDC. The user receives 100,000 USDC instead of 99,900 USDC. The provider pays 100 USDC to the contract. The protocol collects the fee correctly (`fees[productId] += fee`), but the fee burden is misallocated: the withdrawer is enriched at the provider's expense. Any user who can arrange for a third party to submit their fast withdrawal avoids the fee entirely on-chain.

---

### Likelihood Explanation

The third-party provider path (`sendTo != msg.sender`) is the primary operational mode of the fast withdrawal system. Fast withdrawal providers are market participants who submit signed user withdrawal transactions to earn a spread. Every such submission triggers Case 2. The entry path is fully permissionless — any address can call `submitFastWithdrawal` with a valid signed transaction. [4](#0-3) 

---

### Recommendation

In Case 2, deduct the fee from `transferAmount` before transferring to `sendTo`, consistent with Case 1:

```solidity
} else {
    require(transferAmount > uint128(fee), "Fee larger than balance");
    transferAmount -= uint128(fee);
    // provider does not pay the fee; user receives less
}
```

If the design intent is that the provider absorbs the fee as a cost of service, this should be explicitly documented and the `fees[productId]` accounting should reflect that the fee source is the provider, not the withdrawer.

---

### Proof of Concept

1. User signs a `WithdrawCollateral` transaction for `transferAmount = 100,000e6` USDC.
2. Fast withdrawal provider calls `submitFastWithdrawal(idx, transaction, signatures)`.
3. `sendTo = address(uint160(bytes20(signedTx.tx.sender)))` = user's address.
4. `msg.sender` = provider's address. `sendTo != msg.sender` → Case 2.
5. `fee = fastWithdrawalFeeAmount(...)` = 100e6 USDC (0.1%).
6. `safeTransferFrom(token, provider, 100e6)` — provider pays 100 USDC.
7. `handleWithdrawTransfer(token, user, 100_000e6)` — user receives full 100,000 USDC.
8. **Expected**: user receives 99,900 USDC; provider receives nothing extra.
9. **Actual**: user receives 100,000 USDC; provider is out 100 USDC. [5](#0-4)
