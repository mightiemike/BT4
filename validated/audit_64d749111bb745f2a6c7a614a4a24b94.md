### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Allowing Any Caller to Bypass Coinbase Proof Validation and Forge SPV Proofs - (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase Merkle proof before delegating to the original `verify_transaction_inclusion`. However, `verify_transaction_inclusion` is still declared `pub` with no access restriction beyond `#[pause]`, meaning any unprivileged NEAR caller can invoke it directly, completely bypassing the coinbase proof guard. This is structurally identical to M-02: a security-critical check lives in the "outer" entry point, but the "inner" function remains reachable by anyone.

### Finding Description

The intended call chain is:

```
caller → verify_transaction_inclusion_v2
           ├─ require: coinbase_merkle_proof length == merkle_proof length
           ├─ require: coinbase proof recomputes to block's merkle_root   ← forgery guard
           └─ self.verify_transaction_inclusion(args.into())              ← inner logic
```

`verify_transaction_inclusion_v2` enforces the coinbase proof check at lines 348–365 before calling the inner function at line 368. [1](#0-0) 

The inner function `verify_transaction_inclusion` is still `pub` with only a `#[pause]` attribute and no caller restriction: [2](#0-1) 

Rust's `#[deprecated]` is a compile-time lint, not a runtime guard. On NEAR, the function is exported as a normal contract method callable by any account. The `#[pause]` attribute only blocks calls when the contract is paused by a `PauseManager`; it imposes no caller identity check.

The 64-byte transaction forgery attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions) works because Bitcoin's Merkle tree uses `SHA256d(left || right)` for internal nodes. An internal node at depth `d` can be presented as a leaf at depth `d+1` with a crafted proof. Without the coinbase proof, the contract cannot verify the proof depth and therefore cannot distinguish a real transaction hash from an internal node hash. The contract's own warning acknowledges this: [3](#0-2) 

An attacker constructs `ProofArgs` where `tx_id` is an internal Merkle node hash and `merkle_proof` is a valid path from that node to the block's `merkle_root`. The call to `merkle_tools::compute_root_from_merkle_proof` at lines 318–322 will recompute the correct root and the function returns `true` for a transaction that does not exist. [4](#0-3) 

The `ProofArgs` struct requires only `tx_id`, `tx_block_blockhash`, `tx_index`, `merkle_proof`, and `confirmations` — all attacker-controlled: [5](#0-4) 

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` to authorize a cross-chain action (e.g., releasing bridged assets, minting tokens, settling a payment) can be deceived into accepting a forged Bitcoin transaction inclusion proof. The attacker does not need to mine a Bitcoin block or control any privileged key — they only need to identify an internal Merkle node in any already-confirmed block and construct the corresponding proof path. The result is unauthorized state changes in any consumer contract that trusts the light client's `true` return value.

### Likelihood Explanation

The attack requires no privileged access. Any NEAR account can call `verify_transaction_inclusion` directly. The attacker needs only a confirmed Bitcoin block (publicly available) and the ability to compute a Merkle path from an internal node to the root (standard Merkle tree arithmetic). The 64-byte forgery technique is well-documented and tooling exists. The only precondition is that a downstream contract uses the deprecated v1 endpoint or that the attacker can influence which endpoint a consumer calls.

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` or restrict it to `pub(crate)` / internal use only. Since `verify_transaction_inclusion_v2` already calls it internally via `#[allow(deprecated)]`, the inner function does not need to be a public contract method. Alternatively, add an `#[private]` attribute (NEAR SDK) so that only the contract itself can call it as a cross-contract callback, preventing any external account from invoking it directly.

```rust
// Before
#[deprecated(...)]
#[pause]
pub fn verify_transaction_inclusion(&self, ...) -> bool { ... }

// After: restrict to internal use only
#[deprecated(...)]
pub(crate) fn verify_transaction_inclusion(&self, ...) -> bool { ... }
```

### Proof of Concept

1. Identify any confirmed Bitcoin block `B` with at least 2 transactions. Let its `merkle_root` be `R`.
2. Compute the Merkle tree for `B`. Pick any internal node `N` at depth `d` (where `d < tree_height`).
3. Construct a Merkle proof path from `N` to `R` (this is a standard Merkle path of length `tree_height - d`).
4. Call `verify_transaction_inclusion` directly on the NEAR contract with:
   - `tx_id` = hash of internal node `N`
   - `tx_block_blockhash` = hash of block `B` (must be in the mainchain)
   - `tx_index` = the index corresponding to `N`'s position at depth `d`
   - `merkle_proof` = the path from `N` to `R`
   - `confirmations` = 0
5. The contract computes `compute_root_from_merkle_proof(N, index, proof)` which equals `R` (the stored `merkle_root`), and returns `true`.
6. The transaction `N` does not exist on Bitcoin. The proof is forged.

This call succeeds because `verify_transaction_inclusion` is `pub` with no caller restriction, and the coinbase proof guard present in `verify_transaction_inclusion_v2` (lines 358–365) is never executed. [6](#0-5)

        since = "0.5.0",

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
