### Title
`SpotEngine.updateBalance` Uses Stale Interest Multipliers Before Normalizing Balances, Corrupting Deposit and Withdrawal Accounting — (File: `core/contracts/SpotEngineState.sol`)

---

### Summary

`SpotEngineState._updateBalanceNormalized()` normalizes user balances using `cumulativeDepositsMultiplierX18` / `cumulativeBorrowsMultiplierX18` without first accruing pending interest. These multipliers are only updated via `updateStates(dt)`, which is triggered exclusively by a sequencer-submitted `SpotTick` transaction. When a deposit or withdrawal is processed without a preceding `SpotTick`, the normalization uses stale multipliers, causing incorrect accounting: depositors receive unearned interest and withdrawers lose earned interest.

---

### Finding Description

`SpotEngineState._updateBalanceNormalized()` converts a user's stored normalized balance to an actual balance and back using the current `cumulativeDepositsMultiplierX18`:

```solidity
int128 newAmount = balance.amountNormalized.mul(cumulativeMultiplierX18) + balanceDelta;
// ...
balance.amountNormalized = newAmount.div(cumulativeMultiplierX18);
``` [1](#0-0) 

This multiplier is only updated inside `_updateState()`, which is called exclusively from `updateStates(dt)`:

```solidity
function updateStates(uint128 dt) external onlyEndpoint {
    // ...
    _updateState(productId, state, dt);
    _setState(productId, state);
}
``` [2](#0-1) 

`updateStates` is triggered only when the sequencer submits a `SpotTick` transaction:

```solidity
} else if (txType == IEndpoint.TransactionType.SpotTick) {
    // ...
    spotEngine.updateStates(dt);
    t.spotTime = txn.time;
    times = t;
``` [3](#0-2) 

However, `SpotEngine.updateBalance()` — called during both `depositCollateral` and `withdrawCollateral` — reads the multiplier directly from storage without calling `_updateState()` first:

```solidity
function updateBalance(uint32 productId, bytes32 subaccount, int128 amountDelta) external {
    _assertInternal();
    State memory state = states[productId];  // stale multiplier
    // ...
    _updateBalanceNormalized(state, balance, amountDelta);
``` [4](#0-3) 

`depositCollateral` and `withdrawCollateral` both call `spotEngine.updateBalance()` without any prior interest accrual: [5](#0-4) [6](#0-5) 

**Concrete accounting corruption:**

Let `M` = stale multiplier (current stored value), `M'` = correct multiplier after accruing elapsed interest (`M' > M`).

- **Deposit of `X` tokens at stale `M`**: new `amountNormalized = amountNormalized + X/M`. After `SpotTick` updates to `M'`, the depositor's actual balance includes `X * M'/M` instead of `X` — they receive `X * (M'/M − 1)` in unearned interest.
- **Withdrawal of `X` tokens at stale `M`**: new `amountNormalized = amountNormalized − X/M`. After `SpotTick` updates to `M'`, the withdrawer's remaining balance is reduced by `X * M'/M` instead of `X` — they lose `X * (M'/M − 1)` in earned interest.

---

### Impact Explanation

Direct asset accounting corruption in the spot lending system. Depositors who deposit when the multiplier is stale receive unearned interest at the expense of existing depositors. Withdrawers who withdraw when the multiplier is stale lose interest they had already earned. The magnitude scales with the elapsed time since the last `SpotTick` and the borrow utilization rate (which drives the interest rate). This is a solvency/accounting corruption class impact.

---

### Likelihood Explanation

The sequencer controls `SpotTick` submission. In normal operation, `SpotTick` is submitted periodically and precedes balance-changing transactions. However, the slow mode path provides a realistic unprivileged trigger: users submit slow mode `WithdrawCollateral` or `DepositCollateral` transactions directly on-chain, which are queued with a fixed delay (`SLOW_MODE_TX_DELAY`). When processed via `processSlowModeTransactionImpl`, there is no guarantee that a `SpotTick` has been submitted recently:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    // ...
    clearinghouse.withdrawCollateral(txn.sender, txn.productId, txn.amount, address(0), nSubmissions);
``` [7](#0-6) 

No `updateStates` call precedes this path. Any sequencer delay or liveness gap during the slow mode processing window is sufficient to trigger the stale-multiplier condition without requiring sequencer compromise.

---

### Recommendation

In `SpotEngine.updateBalance()`, accrue pending interest by calling `_updateState(productId, state, dt)` with the elapsed time before invoking `_updateBalanceNormalized()`. The elapsed time `dt` can be derived from the stored `times.spotTime` and the current oracle time, mirroring the pattern already used in `updateStates`. This ensures the multiplier is always current before any balance normalization, regardless of whether a `SpotTick` has been recently submitted.

---

### Proof of Concept

1. The sequencer experiences a liveness gap; no `SpotTick` is submitted for a period `Δt`.
2. During this gap, a user submits a slow mode `DepositCollateral` transaction for product `P` with amount `X`.
3. The slow mode transaction is processed via `processSlowModeTransactionImpl` → `clearinghouse.depositCollateral()` → `spotEngine.updateBalance(P, sender, X)`.
4. Inside `_updateBalanceNormalized`, `state.cumulativeDepositsMultiplierX18 = M` (stale; correct value is `M' > M`).
5. The user's `amountNormalized` is set to `X / M` instead of `X / M'`.
6. The sequencer resumes and submits `SpotTick`, calling `updateStates(Δt)`, which updates the multiplier to `M'`.
7. The user's actual balance is now `(X / M) * M' = X * M'/M > X`.
8. The user has received `X * (M'/M − 1)` in interest they did not earn, at the expense of existing depositors whose share of the pool is diluted.
