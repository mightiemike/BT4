### Title
`deserialize_as_tree` silently accepts trailing bytes that `is_canonical_serialization` rejects, creating a Python/Rust acceptance split — (`wheel/src/api.rs`)

### Summary
`deserialize_as_tree` calls `parse_triples` over a `Cursor` wrapping the full input blob but never checks whether the cursor reached the end of the buffer. `is_canonical_serialization` performs the identical parse loop and then enforces `f.position() == f.get_ref().len()`. Any blob consisting of valid CLVM followed by one or more trailing bytes passes `deserialize_as_tree` and fails `is_canonical_serialization`.

### Finding Description
`parse_triples` is a stream-oriented parser: it pops `ParseOpRef::ParseObj` operations until the stack is empty, then returns `Ok((triples, hashes))` with no EOF assertion. [7](#0-6) 

`deserialize_as_tree` constructs a `Cursor::new(blob)`, calls `parse_triples`, and immediately converts the result to Python objects without inspecting `cursor.position()`. [8](#0-7) 

`is_canonical_serialization` runs the same parse loop and then gates success on `f.get_ref().len() as u64 == f.position()`. [9](#0-8) 

Because `is_canonical_serialization` is not registered in the Python module, Python callers have no in-wheel equivalent of this check. [6](#0-5) 

### Impact Explanation
A Python wallet or mempool implementation that uses `deserialize_as_tree` (via `deserialize_as_tuples`) to parse and implicitly validate a CLVM spend bundle will accept a blob with trailing garbage. A Rust implementation that calls `is_canonical_serialization` on the same blob will reject it. This is a concrete API-equivalence split: two implementations of the same validation step reach opposite conclusions on the same attacker-supplied input.

### Likelihood Explanation
The attacker input is trivially constructed: any valid serialized CLVM object with one appended byte. No special privileges, no internal state, no timing dependency. The Python path is the default path for Python CLVM consumers (`deserialize_as_tuples` in `de.py` calls `deserialize_as_tree` first). [10](#0-9) 

### Recommendation
Add an EOF check in `deserialize_as_tree` after `parse_triples` returns:

```rust
if cursor.position() != blob.len() as u64 {
    return Err(PyValueError::new_err("trailing bytes after CLVM object"));
}
```

Alternatively, expose `is_canonical_serialization` as a Python function and document that callers must invoke it when canonical validation is required.

### Proof of Concept

```python
from clvm_rs.clvm_rs import deserialize_as_tree

# 0x80 = NIL (valid, complete CLVM object), 0xAB = trailing garbage
blob = bytes([0x80, 0xAB])

triples, _ = deserialize_as_tree(blob, False)
assert len(triples) == 1          # succeeds — trailing byte silently ignored

# Rust side (called directly):
# is_canonical_serialization(&[0x80, 0xAB]) → false
# The two APIs disagree on the same input.
```

pub fn is_canonical_serialization(b: &[u8]) -> bool {
    let mut f = Cursor::new(b);
    let mut counter = 1;
    let mut b = [0; 1];
    while counter > 0 {
        counter -= 1;
        if f.read_exact(&mut b).is_err() {
            return false;
        }
        if b[0] == CONS_BOX_MARKER {
            // we expect to parse two more items from the stream
            // the left and right sub tree
            counter += 2;
        } else if b[0] == BACK_REFERENCE {
            // This is a back-ref. We don't actually need to resolve it, just
            // parse the path and move on
            if f.read_exact(&mut b).is_err() {
                return false;
            }
            if !is_canonical_atom(&mut f, b[0]) {
                return false;
            }
        } else if !is_canonical_atom(&mut f, b[0]) {
            return false;
        }
        if (f.get_ref().len() as u64) < f.position() {
            return false;
        }
    }
    f.get_ref().len() as u64 == f.position()
}
