### Title
Secp256k1/r1 Verify Executes Unconditionally via 4-Byte Opcode Encoding, Bypassing `ENABLE_SECP_OPS` Gate — (`src/chia_dialect.rs`)

---

### Summary

The `ChiaDialect::op` implementation in `src/chia_dialect.rs` contains two separate dispatch paths for secp operations. The 1-byte opcode path (opcodes 64 and 65) correctly gates execution behind `ENABLE_SECP_OPS`. The 4-byte opcode path (`0x13d61f00`, `0x1c3a8f00`) dispatches to the identical underlying functions with **no flag check at all**, creating a complete bypass of the soft-fork gate.

---

### Finding Description

In `ChiaDialect::op`, when `op_len == 4`, the code matches two specific opcodes and dispatches them directly: [1](#0-0) 

No `ENABLE_SECP_OPS` check is present in this branch. By contrast, the 1-byte opcode path enforces the flag: [2](#0-1) 

Both paths call the same underlying functions `op_secp256k1_verify` and `op_secp256r1_verify` from `src/secp_ops.rs`. The functions themselves accept `_flags: ClvmFlags` but ignore it entirely: [3](#0-2) [4](#0-3) 

There is no secondary guard inside the operator implementations. The only guard is the dispatch-time flag check in `ChiaDialect::op`, and the 4-byte path omits it entirely.

---

### Impact Explanation

The `ENABLE_SECP_OPS` flag is defined as the gate for secp opcodes: [5](#0-4) 

`MEMPOOL_MODE` does not include `ENABLE_SECP_OPS`: [6](#0-5) 

This means in mempool mode (the stricter validation path), a CLVM program using 1-byte opcode `0x40` (64) for `secp256k1_verify` is rejected as an unknown operator, while the semantically identical program using 4-byte opcode `[0x13, 0xd6, 0x1f, 0x00]` executes successfully. Any caller that sets `NO_UNKNOWN_OPS` but not `ENABLE_SECP_OPS` — which is exactly `MEMPOOL_MODE` — is exposed to this split. The operator-availability invariant ("secp ops require `ENABLE_SECP_OPS`") is violated for the 4-byte encoding form.

---

### Likelihood Explanation

The path is fully attacker-controlled: an attacker submits a CLVM program with the 4-byte opcode atom. No special privileges are required. The bypass is unconditional — it does not depend on any runtime state other than the opcode bytes themselves. The proof of concept is a straightforward Rust unit test with `ChiaDialect::new(ClvmFlags::empty())`.

---

### Recommendation

Add an `ENABLE_SECP_OPS` guard in the 4-byte opcode branch, mirroring the 1-byte branch:

```rust
// in the op_len == 4 branch, lines 175-182
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This makes both encoding forms subject to the same flag gate.

---

### Proof of Concept

```rust
#[test]
fn secp_4byte_bypasses_enable_secp_ops_flag() {
    use crate::allocator::Allocator;
    use crate::chia_dialect::{ChiaDialect, ClvmFlags};
    use crate::dialect::{Dialect, OperatorSet};

    let mut allocator = Allocator::new();
    let dialect = ChiaDialect::new(ClvmFlags::empty()); // ENABLE_SECP_OPS NOT set

    // 4-byte opcode 0x13d61f00 — should be gated but is not
    let op_4byte = allocator.new_atom(&[0x13, 0xd6, 0x1f, 0x00]).unwrap();
    let args = allocator.nil();
    let result = dialect.op(&mut allocator, op_4byte, args, u64::MAX, OperatorSet::Default);
    // result is NOT EvalErr::Unimplemented — it dispatches to op_secp256k1_verify
    // (fails on arg count, not on flag check — proving dispatch occurred)

    // 1-byte opcode 64 — correctly gated
    let op_1byte = allocator.new_atom(&[0x40]).unwrap();
    let result2 = dialect.op(&mut allocator, op_1byte, args, u64::MAX, OperatorSet::Default);
    // result2 IS EvalErr::Unimplemented — falls through to unknown_operator
}
```

The two results differ: the 4-byte form reaches `op_secp256k1_verify` (failing on argument validation, not on a flag check), while the 1-byte form is rejected as an unknown operator. This confirms the bypass.
