### Title
NFT Denom ID Is Unauthenticated, Enabling Front-Running of Denom Creation - (File: x/nft/keeper/denom.go, x/nft/keeper/msg_server.go)

### Summary
The `IssueDenom` message handler in the NFT module creates a denom using a caller-supplied, free-form `Id` string with no binding between the denom ID and the sender's address. Any observer of the mempool can copy a pending `MsgIssueDenom` transaction, substitute their own address as `Sender`, and submit it with a higher gas price. The attacker's transaction is included first, making them the recorded `Creator` of the denom. The legitimate user's transaction then fails with "denomID already exists." The attacker, now the denom creator, can mint arbitrary NFTs under that denom ID.

### Finding Description
`MsgIssueDenom` carries a user-chosen `Id` field. In `ValidateBasic`, only the format of the ID is checked: [1](#0-0) 

The keeper's `SetDenom` function enforces only uniqueness — it rejects a duplicate ID but does not verify that the caller has any ownership claim over the chosen ID string: [2](#0-1) 

The message server records `sender` as the denom `Creator` without any prior authentication of the ID to that sender: [3](#0-2) 

Because the denom ID is a plain string with no cryptographic binding to the submitting address, the first transaction to be included wins ownership of that ID. A mempool observer can race the legitimate creator by reusing the exact same `Id` with their own `Sender`.

### Impact Explanation
The attacker becomes the on-chain `Creator` of the denom. The `IsDenomCreator` check gates all privileged denom operations (minting NFTs) on the stored `Creator` field: [4](#0-3) 

Consequently the attacker can mint NFTs under the hijacked denom ID to any recipient, including themselves. The legitimate creator's transaction fails permanently for that ID. If downstream users or contracts rely on a specific `(denomID, creator)` pair — e.g., a marketplace or IBC NFT-transfer flow that trusts a known denom ID — they will interact with attacker-controlled NFTs, enabling counterfeit NFT issuance and potential token theft through deceptive sales.

### Likelihood Explanation
The attack requires only a standard Cosmos SDK transaction submitted with a marginally higher gas price before the victim's transaction is included in a block. No privileged access, leaked keys, or social engineering is needed. The Cosmos mempool is public; any node operator or mempool observer can execute this. The attack is repeatable: every time the victim attempts a new denom ID, the attacker can front-run again, permanently denying the victim the ability to create any denom they publicly announce.

### Recommendation
Bind the denom ID to the creator's address at the protocol level. Two approaches:

1. **Derive the denom ID deterministically** from the creator's address (e.g., `hash(creator_address || user_nonce)`), so no other address can produce the same ID.
2. **Require the denom ID to contain the creator's address as a prefix** and validate this in `ValidateBasic` / `SetDenom`, similar to how IBC class traces are prefixed with port/channel identifiers.

Either approach ensures that only the address that signs the transaction can create a denom with that specific ID, eliminating the front-running surface.

### Proof of Concept
1. Alice broadcasts `MsgIssueDenom{Id: "alice-art", Name: "Alice Art", Sender: alice_addr}`.
2. Eve observes the pending transaction in the mempool.
3. Eve broadcasts `MsgIssueDenom{Id: "alice-art", Name: "Alice Art", Sender: eve_addr}` with a higher gas price.
4. Eve's transaction is included first. `SetDenom` stores `Creator = eve_addr` for denom `"alice-art"`.
5. Alice's transaction fails: `"denomID alice-art has already exists"`.
6. Eve calls `MintNFT` with `DenomId: "alice-art"`. `IsDenomCreator` passes because `eve_addr == denom.Creator`. Eve mints NFTs to herself.
7. Eve lists the NFTs on a marketplace under the well-known `"alice-art"` denom ID. Buyers expecting Alice's collection purchase Eve's counterfeit NFTs. [5](#0-4) [6](#0-5)

func (k Keeper) SetDenom(ctx sdk.Context, denom types.Denom) error {
	if k.HasDenomID(ctx, denom.Id) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denomID %s has already exists", denom.Id)
	}

	if k.HasDenomNm(ctx, denom.Name) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denomName %s has already exists", denom.Name)
	}

	store := ctx.KVStore(k.storeKey)
	bz := k.cdc.MustMarshal(&denom)
	store.Set(types.KeyDenomID(denom.Id), bz)
	store.Set(types.KeyDenomName(denom.Name), []byte(denom.Id))
	return nil
```

// Return the Denom if true, an error otherwise
func (k Keeper) IsDenomCreator(ctx sdk.Context, denomID string, address sdk.AccAddress) (types.Denom, error) {
	denom, err := k.GetDenom(ctx, denomID)
	if err != nil {
		return types.Denom{}, err
	}

	creator, err := sdk.AccAddressFromBech32(denom.Creator)
	if err != nil {
		panic(err)
	}

	if !creator.Equals(address) {
		return types.Denom{}, sdkerrors.Wrapf(types.ErrUnauthorized, "%s is not the creator of %s", address, denomID)
	}

	return denom, nil
```

func (m msgServer) IssueDenom(goCtx context.Context, msg *types.MsgIssueDenom) (*types.MsgIssueDenomResponse, error) {
	sender, err := sdk.AccAddressFromBech32(msg.Sender)
	if err != nil {
		return nil, err
	}

	ctx := sdk.UnwrapSDKContext(goCtx)
	if err := m.Keeper.IssueDenom(ctx, msg.Id, msg.Name, msg.Schema, msg.Uri, sender); err != nil {
		return nil, err
	}
