### Title
`HashDomainTag::MlDsa65PubkeyV1` is immutably bound to both the access-key trie identifier and the planned ML-DSA-65 implicit account ID, enabling front-running of implicit account creation — (File: `core/crypto/src/hash_domain.rs`)

---

### Summary

The domain tag `b"near:ml-dsa-65-pubkey-hash:v1"` is hardcoded in `HashDomainTag::MlDsa65PubkeyV1` and used to compute the on-trie identifier for every ML-DSA-65 access key. The codebase explicitly plans to reuse this same domain tag for ML-DSA-65 implicit account ID derivation. Because the tag is immutably set at compile time and already written into the trie for every ML-DSA-65 access key, the access-key trie identifier and the future implicit account ID will be the same 32-byte SHA3-256 digest for any given pubkey. An unprivileged user who observes a target's ML-DSA-65 public key on-chain can pre-create an implicit account whose ID equals the target's access-key hash, permanently blocking the target from ever creating their own ML-DSA-65 implicit account — and, because the implicit account ID is derived from the pubkey, the target cannot simply "re-register" under a different name; they must generate an entirely new keypair.

---

### Finding Description

`core/crypto/src/hash_domain.rs` defines the only hash-domain tag in the protocol:

```rust
pub enum HashDomainTag {
    MlDsa65PubkeyV1,
}
impl HashDomainTag {
    pub const fn as_bytes(self) -> &'static [u8] {
        match self {
            HashDomainTag::MlDsa65PubkeyV1 => b"near:ml-dsa-65-pubkey-hash:v1",
        }
    }
}
```

`MlDsa65PublicKey::to_public_key_handle()` in `core/crypto/src/signature.rs` uses this tag to compute the on-trie identifier for every ML-DSA-65 access key:

```rust
pub fn to_public_key_handle(&self) -> MlDsa65PublicKeyHandle {
    let mut hasher = Sha3_256::new();
    hasher.update(HashDomainTag::MlDsa65PubkeyV1.as_bytes());
    hasher.update(&self.0[..]);
    ...
}
```

The trie key for an ML-DSA-65 access key is therefore:

```
[tag=3] || SHA3-256("near:ml-dsa-65-pubkey-hash:v1" || raw_pubkey)
```

The comment immediately above `ML_DSA_65_HASH_LENGTH` in `core/crypto/src/signature.rs` states:

> "The same digest is expected to be reused as the account-id payload for ML-DSA-65 implicit accounts when that feature lands; update this comment then."

`docs/architecture/how/post_quantum_signatures.md` §3 (Remaining follow-ups, item 3) confirms:

> "the access-key hashing here has set the precedent (SHA3-256, domain-separated) that implicit-account derivation should probably follow."

If the ML-DSA-65 implicit account feature lands using the same `MlDsa65PubkeyV1` tag, then for any pubkey P:

| Use | Value |
|-----|-------|
| Access-key trie identifier | `SHA3-256("near:ml-dsa-65-pubkey-hash:v1" \|\| P)` |
| Implicit account ID | `SHA3-256("near:ml-dsa-65-pubkey-hash:v1" \|\| P)` |

These are identical. The domain tag is hardcoded at compile time and already written into the trie for every ML-DSA-65 access key added since protocol version 85. It cannot be changed without migrating all existing access-key trie entries — exactly the "must redeploy" severity of the external bug.

---

### Impact Explanation

**High.** An attacker who observes Alice's ML-DSA-65 public key P (visible in any transaction or `AddKey` action Alice has ever submitted) can:

1. Compute `H = SHA3-256("near:ml-dsa-65-pubkey-hash:v1" || P)`.
2. Send a `Transfer` to account ID `H` before the implicit account feature activates, creating an implicit account with that ID.
3. When the ML-DSA-65 implicit account feature activates, Alice's implicit account ID derived from P is also `H`, but the account already exists and is owned by the attacker.
4. Alice cannot create her ML-DSA-65 implicit account. Because the implicit account ID is deterministically derived from the pubkey, she cannot "re-register" under a different name — she must generate a new keypair entirely, losing any pre-distributed pubkey material.

Unlike the ed25519 case (where the implicit account ID is the hex of the raw pubkey and the access-key trie identifier is also the raw pubkey but in a different trie column with no semantic overlap), the ML-DSA-65 design explicitly intends the same 32-byte digest to serve both roles, making the collision structural rather than incidental.

---

### Likelihood Explanation

**Medium.** The ML-DSA-65 implicit account feature is explicitly planned (documented as a follow-up in `docs/architecture/how/post_quantum_signatures.md`). ML-DSA-65 public keys are 1952 bytes and appear in plaintext in `AddKey` actions and signed transactions, so any observer of the chain can collect them. The attack requires no special privilege: any account can send a `Transfer` to an arbitrary account ID to create an implicit account. The window is the entire period between protocol version 85 (when ML-DSA-65 access keys became possible) and the activation of the implicit account feature.

---

### Recommendation

Introduce a **separate** domain tag for ML-DSA-65 implicit account ID derivation, distinct from the access-key trie-identifier tag:

```rust
pub enum HashDomainTag {
    /// On-trie access-key identifier (current, stable since protocol v85).
    MlDsa65PubkeyV1,
    /// Implicit account ID derivation (new, for the upcoming PQ implicit account feature).
    MlDsa65ImplicitAccountV1,
}
impl HashDomainTag {
    pub const fn as_bytes(self) -> &'static [u8] {
        match self {
            HashDomainTag::MlDsa65PubkeyV1 =>
                b"near:ml-dsa-65-pubkey-hash:v1",
            HashDomainTag::MlDsa65ImplicitAccountV1 =>
                b"near:ml-dsa-65-implicit-account:v1",
        }
    }
}
```

This ensures that for any pubkey P:
- Access-key trie identifier = `SHA3-256("near:ml-dsa-65-pubkey-hash:v1" || P)`
- Implicit account ID = `SHA3-256("near:ml-dsa-65-implicit-account:v1" || P)`

These are cryptographically independent 32-byte values, eliminating the domain collision. The existing `MlDsa65PubkeyV1` tag and all trie entries written under it remain unchanged.

### Proof of Concept

```
// Protocol version >= 85 (PostQuantumSignatures active)

// 1. Alice generates an ML-DSA-65 keypair and adds it as an access key.
//    The AddKey action is visible on-chain; attacker reads pubkey P.

// 2. Attacker computes the collision target:
H = SHA3-256(b"near:ml-dsa-65-pubkey-hash:v1" || P)
// H is both Alice's access-key trie identifier AND her future implicit account ID.

// 3. Attacker sends a Transfer to account ID H (base58(H)):
//    near send attacker.near <base58(H)> 1yN
// This creates an implicit account owned by no one (or by the attacker
// if they also control the corresponding key under the future scheme).

// 4. ML-DSA-65 implicit account feature activates.
//    Alice attempts to create her implicit account by sending a Transfer to H.
//    The runtime finds account H already exists → creation fails.
//    Alice's pubkey P is permanently blocked from having an implicit account.
//    She must generate a new keypair; any pre-distributed P is useless.
```

The root cause is the single-variant `HashDomainTag` enum in `core/crypto/src/hash_domain.rs` whose sole tag `MlDsa65PubkeyV1` = `b"near:ml-dsa-65-pubkey-hash:v1"` is hardcoded at compile time and immutably linked to the access-key trie encoding in `core/crypto/src/signature.rs`, while the codebase simultaneously plans to reuse it for implicit account derivation — the exact "must redeploy because the name is hardcoded" pattern from the external report, translated to a hash-domain invariant at a protocol upgrade boundary. impl MlDsa65PublicKey {
    /// Compute the on-trie [`MlDsa65PublicKeyHandle`] for this public key -
    /// the SHA3-256 of (domain-separation tag || raw pubkey bytes).
    ///
    /// This is the form an ML-DSA-65 access key takes inside the trie: the
    /// full pubkey lives only on the wire (in transactions and actions),
    /// never in state. The full pubkey can be derived from the handle only
    /// by brute force.
    pub fn to_public_key_handle(&self) -> MlDsa65PublicKeyHandle {
        use sha3::{Digest, Sha3_256};
        let mut hasher = Sha3_256::new();
        hasher.update(HashDomainTag::MlDsa65PubkeyV1.as_bytes());
        hasher.update(&self.0[..]);
        let mut out = [0u8; ML_DSA_65_HASH_LENGTH];
        out.copy_from_slice(&hasher.finalize());
        MlDsa65PublicKeyHandle(out)
    }
