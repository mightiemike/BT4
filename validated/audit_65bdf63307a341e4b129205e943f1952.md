### Title
`verify_transaction_inclusion_v2` Always Panics for Single-Transaction Blocks Due to Inherited Empty-Proof Guard in `verify_transaction_inclusion` — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` is the designated safe replacement for the deprecated `verify_transaction_inclusion`. It performs its own coinbase-proof validation and then delegates to the deprecated function. However, the deprecated function contains a hard `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard that was never designed to accommodate the single-transaction block case. For any Bitcoin block containing exactly one transaction (the coinbase), the merkle proof is legitimately empty, yet the inherited guard causes an unconditional panic, permanently blocking proof verification for that block.

---

### Finding Description

`verify_transaction_inclusion_v2` validates the coinbase merkle proof independently, then calls the deprecated `verify_transaction_inclusion` via `self.verify_transaction_inclusion(args.into())`: [1](#0-0) 

Inside `verify_transaction_inclusion`, after all other checks pass, the following guard fires unconditionally: [2](#0-1) 

For a single-transaction block the merkle root **is** the coinbase transaction hash. The correct proof for that transaction is an empty vector — no siblings exist. `verify_transaction_inclusion_v2` explicitly accepts this case: its coinbase-proof check calls `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])`, which returns `coinbase_tx_id` unchanged (the loop body never executes): [3](#0-2) 

That result equals `header.block_header.merkle_root` for a single-transaction block, so the coinbase check passes. The length-equality guard also passes because both `merkle_proof` and `coinbase_merkle_proof` are empty. Execution then enters `verify_transaction_inclusion`, which immediately panics on the empty-proof guard — a check that was placed there to block the 64-byte Merkle forgery attack but was never conditioned on the single-transaction case.

The structural parallel to H-07 is exact:

---

### Impact Explanation

Any downstream NEAR contract or user that calls `verify_transaction_inclusion_v2` with a valid proof for a single-transaction Bitcoin block receives an unconditional panic instead of `true`. If the verification result gates fund releases or cross-chain actions, those actions are permanently blocked for every single-transaction block — not because the proof is wrong, but because the inherited guard was not designed for this call path. The contract state is not corrupted, but the proof-verification result is permanently wrong (panic instead of `true`) for a class of valid inputs.

---

### Likelihood Explanation

Single-transaction blocks are real and common in Bitcoin history (early mainnet blocks, blocks containing only a coinbase). Any relayer that has submitted such a block to the light client creates a permanently unverifiable block for downstream consumers. The trigger requires no privilege: any unprivileged NEAR account can call `verify_transaction_inclusion_v2` with a valid single-transaction proof and reproduce the panic deterministically.

---

### Recommendation

In `verify_transaction_inclusion_v2`, do not delegate to `verify_transaction_inclusion`. Instead, inline the merkle-root comparison directly, skipping the empty-proof guard (which is already superseded by the coinbase-proof validation performed in v2):

```rust
pub fn verify_transaction_inclusion_v2(&self, args: ProofArgsV2) -> bool {
    // ... existing coinbase proof check ...

    // Inline the final check instead of calling the deprecated function
    let header = self.headers_pool.get(&args.tx_block_blockhash)...;
    // confirmations check ...
    // For single-tx blocks merkle_proof is empty and tx_id == merkle_root; allow it.
    merkle_tools::compute_root_from_merkle_proof(
        args.tx_id,
        usize::try_from(args.tx_index).unwrap(),
        &args.merkle_proof,
    ) == header.block_header.merkle_root
}
```

Alternatively, add a guard in `verify_transaction_inclusion` that skips the empty-proof check when the proof length is zero and `tx_id == merkle_root`.

---

### Proof of Concept

1. A Bitcoin block exists with exactly one transaction (coinbase only). Its `merkle_root` equals `coinbase_tx_id`.
2. The relayer submits this block header via `submit_blocks`; it is accepted and stored.
3. A downstream caller invokes `verify_transaction_inclusion_v2` with:
   - `tx_id = coinbase_tx_id`
   - `tx_index = 0`
   - `merkle_proof = []`
   - `coinbase_tx_id = coinbase_tx_id`
   - `coinbase_merkle_proof = []`
   - `confirmations = 1`
4. The coinbase-proof check passes: `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` = `coinbase_tx_id` = `merkle_root`. [5](#0-4) 
5. `verify_transaction_inclusion(args.into())` is called. [6](#0-5) 
6. `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` fires → panic. [2](#0-1) 
7. The call reverts. The valid proof is permanently rejected. Any fund-release logic gated on this call is permanently blocked for this block.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
