### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables Theft of usdcE From DirectDepositV1 ŌĆö (File: `core/contracts/ContractOwner.sol`)

---

### Summary
`ContractOwner.replaceUsdcEWithUsdc` performs a raw `IERC20Base.transferFrom` call whose return value is never checked. If the USDC token on Ink chain silently returns `false` instead of reverting on a failed transfer, the function continues to drain usdcE from the target DirectDepositV1 address and deliver it to the caller ŌĆö without the caller having provided any USDC. The function has no access control, making it reachable by any unprivileged caller.

---

### Finding Description

`replaceUsdcEWithUsdc` is designed as a 1:1 swap: the caller supplies USDC and receives usdcE in return. The inbound USDC leg at line 616 uses a raw `transferFrom`:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
```

The return value is silently discarded. The rest of the Nado codebase consistently uses `ERC20Helper.safeTransferFrom`, which wraps the low-level call and enforces:

```solidity
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [1](#0-0) 

The raw call at line 616 skips this check entirely. [2](#0-1) 

If `transferFrom` returns `false` (e.g., caller has zero USDC allowance or balance, and the token does not revert), execution falls through to:

1. `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))` ŌĆö pulls all usdcE from the DDA to `ContractOwner` (ContractOwner is the owner of every DDA it deploys).
2. `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` ŌĆö forwards that usdcE to the caller. [3](#0-2) 

The function carries no `onlyOwner` or any other access-control modifier, so any external address can invoke it. [4](#0-3) 

The `DirectDepositV1.withdraw` function transfers the full token balance of the DDA to its caller (`ContractOwner`), which then forwards it to `msg.sender`. [5](#0-4) 

---

### Impact Explanation

**High asset impact.** An attacker who calls `replaceUsdcEWithUsdc` for any subaccount whose DDA holds usdcE, while providing zero USDC allowance, can steal the entire usdcE balance of that DDA in a single transaction. The corrupted asset delta is: attacker gains `balance` usdcE; DDA loses `balance` usdcE; no USDC is transferred in exchange.

---

### Likelihood Explanation

**Medium.** The function is publicly callable with no access control. The trigger depends on whether the specific USDC deployment at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink chain (chainid 57073) returns `false` rather than reverting on a failed `transferFrom`. As the reference report notes, USDC implementations differ across chains. The inconsistency with the rest of the codebase ŌĆö which uniformly uses `ERC20Helper.safeTransferFrom` ŌĆö indicates this is an unintentional omission rather than a deliberate design choice, raising the realistic probability of the vulnerable path being exercisable.

---

### Recommendation

Replace the raw `transferFrom` call with the `ERC20Helper.safeTransferFrom` wrapper already used throughout the codebase:

```solidity
// Before (line 616):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After:
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ContractOwner` already imports and uses `ERC20Helper` via `using ERC20Helper for IERC20Base`, so no additional imports are needed. [6](#0-5) 

---

### Proof of Concept

1. A DDA for `subaccount` holds `N` usdcE tokens (deposited by a legitimate user).
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` with **zero** USDC allowance granted to `ContractOwner`.
3. `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` returns `false` ŌĆö no revert, no USDC moves.
4. `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))` executes: `N` usdcE is transferred from the DDA to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` executes: `N` usdcE is transferred from `ContractOwner` to the attacker.
6. Attacker receives `N` usdcE having provided zero USDC. The DDA is drained. [7](#0-6)

        require(block.chainid == 57073, ERR_UNAUTHORIZED);

        require(directDepositV1 != address(0), "no dda");

            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
