### Title
Secp Operator Dual-Wiring Bypasses `ENABLE_SECP_OPS` Flag via 4-Byte Opcodes — (File: `src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op` wires `op_secp256k1_verify` and `op_secp256r1_verify` in **two independent dispatch paths**: once via 4-byte opcodes (`0x13d61f00`, `0x1c3a8f00`) with **no flag check**, and once via 1-byte opcodes (`64`, `65`) gated behind `ENABLE_SECP_OPS`. The 4-byte path is a permanent bypass of the flag, making `ENABLE_SECP_OPS` ineffective as an access gate for secp verification.

---

### Finding Description

In `ChiaDialect::op`, the first dispatch branch handles 4-byte opcodes:

```rust
// src/chia_dialect.rs lines 157–182
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,   // no flag check
        0x1c3a8f00 => op_secp256r1_verify,   // no flag check
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
``` [1](#0-0) 

The second dispatch branch handles 1-byte opcodes and **does** enforce the flag:

```rust
// src/chia_dialect.rs lines 248–249
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The same two operator functions are thus reachable via two different opcode encodings, but only one encoding is gated by `ENABLE_SECP_OPS`. The 4-byte opcodes were crafted to match the secp cost formula (multiplier `0x13d61f` + 1 = 1,300,000 for k1; `0x1c3a8f` + 1 = 1,850,000 for r1), but instead of falling through to `unknown_operator` (a no-op), they call the live verification functions unconditionally. [3](#0-2) 

The `MEMPOOL_MODE` constant — the production mempool configuration — does **not** include `ENABLE_SECP_OPS`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [4](#0-3) 

In mempool mode with `NO_UNKNOWN_OPS` set, a program using 1-byte opcode `64` falls through to `unknown_operator`, which returns `Err(EvalErr::Unimplemented(o))`. But a program using 4-byte opcode `0x13d61f00` bypasses that path entirely and calls `op_secp256k1_verify` directly — performing live ECDSA verification. [5](#0-4) 

---

### Impact Explanation

The broken invariant is: **`ENABLE_SECP_OPS` does not actually gate all access to secp verification**. The 4-byte opcode path is a permanent bypass.

Concrete divergence:

| Opcode encoding | `ENABLE_SECP_OPS` absent, valid secp args | `ENABLE_SECP_OPS` absent, invalid secp args |
|---|---|---|
| 1-byte `0x40` (opcode 64) | `Err(Unimplemented)` | `Err(Unimplemented)` |
| 4-byte `0x13d61f00` | **`Ok(nil, cost=1300000)`** | `Err(Secp256Failed)` |

A program that should be uniformly rejected (secp not enabled) instead succeeds when encoded with the 4-byte opcode and valid arguments. This is a **flag/operator wiring error** producing a corrupted accept/reject result — the exact analog of the YetiToken `transfer`/`transferFrom` state divergence, where one path writes to a different mapping than the other.

If `ENABLE_SECP_OPS` is used as a hard-fork activation gate (enabling secp only after a specific block height), the 4-byte path allows secp verification to execute on any node regardless of fork activation state, causing **consensus divergence** between nodes that have activated the fork (and thus accept both encodings) and nodes that have not (which should reject both but silently accept the 4-byte form).

---

### Likelihood Explanation

The entry path is fully attacker-controlled: an adversary submits CLVM bytes where the operator atom is the 4-byte sequence `[0x13, 0xd6, 0x1f, 0x00]` followed by a valid secp256k1 pubkey, message digest, and signature. No special privileges are required. The 4-byte opcodes are valid CLVM atoms accepted by the deserializer. The bypass is unconditional — it fires on every invocation regardless of dialect flags.

---

### Recommendation

Either:
1. Add an `ENABLE_SECP_OPS` flag check to the 4-byte dispatch branch before calling `op_secp256k1_verify` / `op_secp256r1_verify`, mirroring the 1-byte branch guard; or
2. Remove the 4-byte secp dispatch entirely and rely solely on the 1-byte opcodes (64, 65) gated by `ENABLE_SECP_OPS`, letting the 4-byte opcodes fall through to `unknown_operator` as the comment ("these are unknown operators with assigned cost") implies they should.

---

### Proof of Concept

Construct a CLVM program whose operator atom is the 4-byte sequence `\x13\xd6\x1f\x00` with three arguments: a valid secp256k1 compressed public key (33 bytes), a 32-byte message digest, and a valid 64-byte DER/compact signature. Run it under `ChiaDialect::new(MEMPOOL_MODE)` (which excludes `ENABLE_SECP_OPS`). The program returns `Ok(Reduction(1300000, nil))` — secp verification executed and succeeded — even though `ENABLE_SECP_OPS` is absent. The same program encoded with 1-byte opcode `\x40` returns `Err(EvalErr::Unimplemented)` under identical flags, demonstrating the inconsistency. [6](#0-5) [7](#0-6)

pub fn op_secp256k1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256K1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256k1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256k1_verify pubkey")?;
    let verifier = K1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256k1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is message
    let msg = atom(a, msg, "secp256k1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256k1_verify sig")?;
    let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
