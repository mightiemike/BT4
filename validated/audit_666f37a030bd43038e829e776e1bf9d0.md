### Title
NFT Transfer to Zero Address Allowed, Causing Permanent Asset Loss and Supply Accounting Corruption — (File: x/nft/keeper/keeper.go)

---

### Summary

`TransferOwner` in `x/nft/keeper/keeper.go` and `MsgTransferNFT.ValidateBasic()` in `x/nft/types/msgs.go` do not reject a bech32-encoded zero address as the destination owner. Any NFT owner can submit a `MsgTransferNFT` transaction with a zero address recipient, permanently locking the NFT in an inaccessible state while leaving the denom's supply count inflated and the owner index corrupted.

---

### Finding Description

`TransferOwner` accepts any `sdk.AccAddress` as `dstOwner` without checking `dstOwner.Empty()` or whether the address is the all-zeros address:

```go
// x/nft/keeper/keeper.go:121-138
func (k Keeper) TransferOwner(
    ctx sdk.Context, denomID, tokenID string, srcOwner, dstOwner sdk.AccAddress,
) error {
    ...
    nft.Owner = dstOwner.String()
    k.setNFT(ctx, denomID, nft)
    k.swapOwner(ctx, denomID, tokenID, srcOwner, dstOwner)
    return nil
}
```

`MsgTransferNFT.ValidateBasic()` only calls `sdk.AccAddressFromBech32(msg.Recipient)`, which accepts a valid bech32-encoded zero address (e.g., `cosmos1qqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqnrql8a`) without error:

```go
// x/nft/types/msgs.go:94-96
if _, err := sdk.AccAddressFromBech32(msg.Recipient); err != nil {
    return newsdkerrors.Wrapf(sdkerrors.ErrInvalidAddress, "invalid recipient address (%s)", err)
}
```

No additional guard exists in `msgServer.TransferNFT` either:

```go
// x/nft/keeper/msg_server.go:129-146
func (m msgServer) TransferNFT(...) {
    recipient, err := sdk.AccAddressFromBech32(msg.Recipient)
    ...
    if err := m.TransferOwner(ctx, msg.DenomId, msg.Id, sender, recipient); err != nil { ... }
}
```

The same gap exists in the IBC path: `processReceivedPacket` decodes `data.Receiver` via `sdk.AccAddressFromBech32` and passes it directly to `MintNFT` or `TransferOwner` without a zero-address guard.

```go
// x/nft-transfer/keeper/packet.go:143-146
receiver, err := sdk.AccAddressFromBech32(data.Receiver)
if err != nil {
    return err
}
```

---

### Impact Explanation

1. **Permanent NFT loss**: After `TransferOwner` completes, `nft.Owner` is set to the zero address string. No private key exists for the zero address, so the NFT can never be transferred, edited, or burned again. The underlying asset is permanently inaccessible.

2. **Supply count inflation**: `TransferOwner` does not call `decreaseSupply`. The denom's total supply counter remains unchanged even though the NFT is effectively destroyed. Every NFT sent to the zero address permanently inflates the reported supply by 1. Any consumer of `GetTotalSupply` (e.g., external governance or cross-chain contracts querying collection size) receives a stale, inflated value.

3. **Owner index corruption**: `swapOwner` adds the tokenID to the zero address's owner list. Queries over `ownerToNFTokenIdList[zero_address]` will return these orphaned tokens, corrupting collection enumeration.

4. **IBC escrow drain (back-to-origin path)**: In `processReceivedPacket`, when a packet returns an NFT to its origin chain, `TransferOwner` moves the NFT from the escrow account to `receiver`. If `receiver` is the zero address, the escrowed NFT is permanently removed from the escrow account and assigned to an uncontrollable address, making it irrecoverable.

---

### Likelihood Explanation

The trigger is a standard, unprivileged `MsgTransferNFT` transaction signed by the current NFT owner. No special role, governance approval, or chain-level permission is required. Any holder of an NFT can execute this in a single transaction. The IBC variant requires the sender on the source chain to specify a zero-address receiver, which is also an unprivileged action controlled entirely by the packet initiator.

---

### Recommendation

1. Add an explicit zero-address guard in `TransferOwner`:
   ```go
   if dstOwner.Empty() {
       return sdkerrors.Wrap(types.ErrInvalidAddress, "transfer to zero address not allowed")
   }
   ```
2. Add the same guard in `MintNFTUnverified` for the `owner` parameter.
3. Add the same guard in `processReceivedPacket` after decoding `receiver`.
4. Add the check in `MsgTransferNFT.ValidateBasic()` and `MsgMintNFT.ValidateBasic()` to reject zero-address recipients at the message-validation layer.

---

### Proof of Concept

1. Alice owns NFT `(denomID="mycollection", tokenID="token1")`.
2. Alice submits `MsgTransferNFT{Id:"token1", DenomId:"mycollection", Sender:alice, Recipient:"cosmos1qqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqnrql8a"}`.
3. `ValidateBasic` passes because `sdk.AccAddressFromBech32("cosmos1qqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqnrql8a")` succeeds.
4. `TransferOwner` sets `nft.Owner = "cosmos1qqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqnrql8a"`, calls `swapOwner`, and returns nil.
5. `GetTotalSupply("mycollection")` still returns the pre-transfer count — inflated by 1 relative to accessible tokens.
6. `GetNFTsByOwner(zero_address)` now lists `token1`.
7. No transaction can ever move `token1` again; it is permanently locked.

func (k Keeper) TransferOwner(

	return nil
}
