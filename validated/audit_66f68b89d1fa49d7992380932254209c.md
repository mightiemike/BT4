Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of the `addLiquidity` call) and gates only on the caller-supplied `owner` address. Because `MetricOmmPool.addLiquidity` imposes no `msg.sender == owner` constraint (unlike `removeLiquidity`), any unprivileged caller can pass an allowlisted address as `owner`, satisfy the extension check, and deposit liquidity into a permissioned pool — breaking the invariant that only allowlisted addresses may hold positions.

## Finding Description
`MetricOmmPool.addLiquidity` passes both `msg.sender` (as `sender`) and the free-form `owner` parameter to the extension hook:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`DepositAllowlistExtension.beforeAddLiquidity` discards the first argument and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`removeLiquidity` enforces `if (msg.sender != owner) revert NotPositionOwner()` at L206, but `addLiquidity` has no equivalent guard. An attacker observes an allowlisted address via `AllowedToDepositSet` events, calls `pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, extensionData)`, and the extension evaluates `allowedDepositor[pool][allowlistedAddress]` — which is `true` — and passes. The attacker's contract pays tokens in the `metricOmmModifyLiquidityCallback`. Liquidity is minted under `allowlistedAddress`; the attacker cannot withdraw it (`msg.sender != owner` in `removeLiquidity`), but the pool's access-control invariant is broken and bin-level liquidity state is modified by an unauthorized actor.

The `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, ...)` path (L56-68) also exposes this: it only validates `owner != address(0)` via `_validateOwner`, then calls `pool.addLiquidity(positionOwner, ...)` with `msg.sender = LiquidityAdder`, so the extension again sees only `owner`.

## Impact Explanation
The deposit allowlist is the primary access-control mechanism for permissioned pools. Bypassing it allows an unprivileged actor to deposit into a pool they are not authorized to interact with, breaking the admin-configured invariant that only allowlisted addresses may hold positions. The attacker can add liquidity to specific bins, altering the pool's liquidity distribution and affecting subsequent swap execution. This constitutes an admin-boundary break — an unprivileged path bypasses a factory/pool admin-configured access control — and can affect swap conservation by changing bin liquidity depth available to traders.

## Likelihood Explanation
No special privilege is required; any EOA or contract can call `pool.addLiquidity` directly. The only precondition is knowing one allowlisted address, which is observable on-chain via `AllowedToDepositSet` events. The attacker must implement a trivial `metricOmmModifyLiquidityCallback` and fund the deposit. The attack is repeatable. The attacker sacrifices deposited capital, making it economically rational when bin-state manipulation yields an off-pool profit exceeding the deposit cost.

## Recommendation
Gate on `sender` (the actual caller) rather than `owner`, consistent with the contract's stated purpose of gating `addLiquidity` by depositor:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is instead to restrict which addresses may *own* positions, `MetricOmmPool.addLiquidity` must enforce `msg.sender == owner` (mirroring the `removeLiquidity` guard at L206), so that `sender` and `owner` are always the same address.

## Proof of Concept

```solidity
contract BypassDeposit {
    IMetricOmmPool pool;
    address allowlistedLP; // observed from AllowedToDepositSet event

    function attack(LiquidityDelta calldata deltas) external {
        // owner = allowlistedLP → extension check passes
        // sender = address(this) → extension check is never evaluated
        pool.addLiquidity(allowlistedLP, 0, deltas, abi.encode(uint8(1)), "");
    }

    function metricOmmModifyLiquidityCallback(
        uint256 amount0, uint256 amount1, bytes calldata
    ) external {
        IERC20(token0).transfer(msg.sender, amount0);
        IERC20(token1).transfer(msg.sender, amount1);
    }
}
```

Steps:
1. Deploy `BypassDeposit`, fund with token0/token1.
2. Call `attack(deltas)` targeting any bin.
3. `DepositAllowlistExtension.beforeAddLiquidity` receives `sender = BypassDeposit`, `owner = allowlistedLP`; checks only `allowedDepositor[pool][allowlistedLP]` → `true` → no revert.
4. Liquidity is minted under `allowlistedLP`; pool bin state is modified by the unauthorized actor.
