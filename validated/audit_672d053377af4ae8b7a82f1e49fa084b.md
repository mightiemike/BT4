### Title
Division Before Multiplication Causes Systematic Target Precision Loss in Zcash Difficulty Adjustment — (`File: contract/src/zcash.rs`)

### Summary
In `zcash_calculate_next_work_required`, the new mining target is computed by first dividing `average_target` by `averaging_window_timespan` and then multiplying by `actual_timespan`. Integer division truncates before the multiplication, producing a systematically lower (harder) target than the true Zcash protocol value. This can cause the contract to reject valid Zcash block headers or accept an incorrect difficulty, corrupting the canonical chain state.

### Finding Description
The retarget formula in `zcash_calculate_next_work_required` is:

```rust
// contract/src/zcash.rs lines 184-187
let new_target = average_target
    / U256::from(<i64 as TryInto<u64>>::try_into(averaging_window_timespan).unwrap());
let (mut new_target, new_target_overflow) =
    new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(actual_timespan).unwrap());
```

The mathematically correct formula is:

```
new_target = average_target × actual_timespan / averaging_window_timespan
```

But the code evaluates it as:

```
new_target = (average_target / averaging_window_timespan) × actual_timespan
```

Because `U256` division truncates toward zero, the intermediate result `average_target / averaging_window_timespan` loses up to `averaging_window_timespan − 1` units of precision. That truncation error is then amplified by `actual_timespan` in the subsequent multiplication.

For Zcash mainnet, `averaging_window_timespan = pow_averaging_window × pow_target_spacing = 17 × 75 = 1275`. [1](#0-0) 

The maximum `actual_timespan` is `max_actual_timespan = (1275 × 132) / 100 = 1683`. [2](#0-1) 

The maximum absolute error in `new_target` is therefore up to `1274 × 1683 ≈ 2,144,142` units of the raw U256 target value. When this error crosses a boundary in `target_to_bits` (which encodes only 3 bytes of mantissa), the returned `expected_bits` value differs from the true protocol value. [3](#0-2)

### Impact Explanation
The computed `expected_bits` is compared directly against the submitted `block_header.bits` in `check_pow`: [7](#0-6) 

When the precision loss shifts `expected_bits` to a different compact value, the contract enforces a difficulty that diverges from the true Zcash protocol. Concretely:

- A **valid Zcash block** (satisfying the true protocol difficulty) can be **rejected** because the contract computes a harder (lower) target.
- This breaks the invariant that every block accepted by the Zcash network is accepted by the light client, making the contract's canonical chain diverge from the real Zcash chain.

### Likelihood Explanation
The bug is triggered on every Zcash difficulty adjustment evaluation — every block submission for the Zcash build. The precision loss is deterministic and depends on the specific `average_target` and `actual_timespan` values at each retarget. Near retarget boundaries where `target_to_bits` precision is coarse, the erroneous `expected_bits` will differ from the correct value, causing rejection of valid headers. Any unprivileged NEAR caller submitting a valid Zcash block header via `submit_blocks` can trigger this path. [8](#0-7) 

### Recommendation
Swap the order of operations to multiply before dividing:

```rust
// Correct order: multiply first, then divide
let (scaled_target, overflow) =
    average_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(actual_timespan).unwrap());
require!(!overflow, "new target overflow");
let new_target = scaled_target
    / U256::from(<i64 as TryInto<u64>>::try_into(averaging_window_timespan).unwrap());
```

This matches the pattern already used correctly in `bitcoin.rs` and `litecoin.rs`.

### Proof of Concept
Consider `average_target = U256::from(1275u64 * K + 637)` for any integer `K`, and `actual_timespan = 1683` (the maximum allowed):

- **Correct**: `(1275K + 637) × 1683 / 1275 = 1683K + (637 × 1683 / 1275) = 1683K + 840` (floor)
- **Buggy**: `((1275K + 637) / 1275) × 1683 = K × 1683`

The error is `840` units of the raw target. If this shifts the mantissa across a byte boundary in `target_to_bits`, the returned `bits` value changes, and a valid block header with the correct `bits` is rejected by the `require!(expected_bits == block_header.bits, ...)` check. [9](#0-8)

    pub fn target_to_bits(&self) -> u32 {
        let mut n_size = self.bits().div_ceil(8);
        let mut n_compact: u32;

        if n_size <= 3 {
            n_compact = u32::try_from(self.1 << (8 * (3 - n_size))).unwrap();
        } else {
            let target = *self >> (8 * (n_size - 3));
            n_compact = u32::try_from(target.1 & 0x00ff_ffff).unwrap();
        }

        if n_compact & 0x00800000 != 0 {
            n_compact >>= 8;
            n_size += 1;
        }

        n_compact |= n_size << 24;
        n_compact
    }
