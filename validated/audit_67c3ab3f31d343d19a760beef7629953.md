### Title
Fee-on-Transfer Token Deposit Inflates Subaccount Balance Beyond Actual Collateral Received — (`File: core/contracts/EndpointStorage.sol`)

---

### Summary

`EndpointStorage.handleDepositTransfer` performs two sequential ERC-20 transfers using the caller-supplied `amount` parameter without measuring the actual tokens received at each hop. When a fee-on-transfer token is used as collateral, the Clearinghouse receives fewer tokens than `amount`, but the subaccount is credited the full `amount`. This permanently inflates the on-chain accounting relative to real token holdings, making the protocol insolvent for that product.

---

### Finding Description

The deposit path is:

1. `Endpoint.depositCollateralWithReferral` → calls `handleDepositTransfer(token, msg.sender, uint256(amount))`
2. `handleDepositTransfer` executes two transfers in sequence using the same `amount`:

```solidity
// EndpointStorage.sol L111-L119
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);       // hop 1: user → Endpoint
    safeTransferTo(token, address(clearinghouse), amount); // hop 2: Endpoint → Clearinghouse
}
```

3. The slow-mode queue entry encodes the original `amount`:

```solidity
// Endpoint.sol L155-L163
DepositCollateral({
    sender: subaccount,
    productId: productId,
    amount: amount          // original caller-supplied value
})
```

4. When the sequencer processes the queue, `Clearinghouse.depositCollateral` credits `txn.amount` (the original value) to the subaccount:

```solidity
// Clearinghouse.sol L205-L208
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
```

With a fee-on-transfer token:

- **Hop 1** (`safeTransferFrom`): Endpoint receives `amount − fee₁`.
- **Hop 2** (`safeTransferTo`): Endpoint attempts to forward `amount`. If the Endpoint holds a pre-existing balance of the token (e.g., accumulated slow-mode fees in the quote token via `chargeSlowModeFee`), this succeeds by consuming `fee₁` from that reserve. Clearinghouse receives `amount − fee₂`.
- **Accounting**: subaccount is credited `amount`, but Clearinghouse only holds `amount − fee₂`.

The invariant `Σ(subaccount balances for productId) == Clearinghouse token balance` is permanently broken upward.

---

### Impact Explanation

Every deposit of a fee-on-transfer token inflates the protocol's internal accounting by `fee₁ + fee₂` relative to actual holdings. Concretely:

- The depositor's subaccount balance exceeds the real collateral contributed.
- The depositor can withdraw the inflated balance, draining tokens that belong to other users.
- Repeated deposits drain the Endpoint's slow-mode fee reserve and then cause subsequent deposits to revert (Endpoint cannot forward `amount` when it only holds `amount − fee₁`), but the accounting damage from earlier deposits persists.
- Protocol solvency is corrupted: the last users to withdraw cannot be paid out in full.

---

### Likelihood Explanation

The Nado protocol is deployed on Ink Chain (EVM-compatible L2) and explicitly supports multiple ERC-20 collateral tokens via `SpotEngine`. USDT has a fee switch that can be enabled by its owner. If USDT or any other fee-on-transfer token is listed as a supported collateral product, any unprivileged user calling `depositCollateralWithReferral` triggers the root cause. No special role or governance action is required beyond the token being listed. The Endpoint accumulates quote-token slow-mode fees continuously, providing the pre-existing balance that allows hop 2 to succeed for the quote product.

---

### Recommendation

Measure the actual received amount at each transfer boundary using a before/after balance check, and credit only the net received amount:

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), token.balanceOf(address(this)));
    uint256 actualReceived = token.balanceOf(address(clearinghouse)) - before;
    // pass actualReceived (not amount) into the slow-mode queue entry
}
```

Alternatively, explicitly disallow fee-on-transfer tokens at the product listing level and document this restriction.

### Proof of Concept

**Setup**: USDT is listed as a supported collateral product. USDT fee is enabled at 1%. Endpoint holds 10 USDT in accumulated slow-mode fees.

**Steps**:

1. Attacker calls `Endpoint.depositCollateralWithReferral(subaccount, usdtProductId, 1000e6, "")`.
2. `handleDepositTransfer(usdt, attacker, 1000e6)` executes:
   - `safeTransferFrom(usdt, attacker, 1000e6)` → Endpoint receives **990e6** USDT (10e6 fee to USDT owner).
   - `safeTransferTo(usdt, clearinghouse, 1000e6)` → Endpoint sends 1000e6 (990e6 from this deposit + 10e6 from existing slow-mode reserve). Clearinghouse receives **990e6** USDT (another 10e6 fee).
3. Slow-mode queue records `amount = 1000e6`.
4. Sequencer processes the queue; `Clearinghouse.depositCollateral` credits `1000e6 * multiplier` to attacker's subaccount.
5. Attacker's subaccount shows **1000e6** USDT; Clearinghouse holds only **990e6** USDT for this deposit.
6. Attacker withdraws 1000e6 USDT. The 10e6 shortfall is silently taken from other users' collateral.

**Corrupted state delta**: `SpotEngine` balance for attacker's subaccount = `+1000e6 * multiplier`; actual Clearinghouse token increment = `+990e6`. The 10e6 gap is a permanent solvency deficit per deposit. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
