### Title
Unbounded Tier Position Growth Permanently Blocks `MsgUpdateTier` BonusApy Governance Execution - (`x/tieredrewards/keeper/claim_rewards.go`)

### Summary

`claimRewardsAndUpdateTierPositions` iterates every position in a tier without any bound. It is called unconditionally inside `UpdateTier` whenever `BonusApy` changes. Because any unprivileged user can create positions in any open tier via `LockTier` or `CommitDelegationToTier`, an attacker can flood a tier with positions until the governance proposal that executes `MsgUpdateTier` exceeds the block gas limit and fails. Once the tier is large enough, governance permanently loses the ability to adjust that tier's `BonusApy`, preventing the chain from reducing an overly generous rate or correcting a misconfigured one.

### Finding Description

`UpdateTier` in `msg_server_auth.go` checks whether `BonusApy` has changed and, if so, calls `claimRewardsAndUpdateTierPositions` before persisting the new tier:

```go
if !oldTier.BonusApy.Equal(msg.Tier.BonusApy) {
    if err := ms.claimRewardsAndUpdateTierPositions(ctx, msg.Tier.Id); err != nil {
        return nil, err
    }
}
```

`claimRewardsAndUpdateTierPositions` fetches **all** position IDs for the tier and iterates every one of them:

```go
func (k Keeper) claimRewardsAndUpdateTierPositions(ctx context.Context, tierId uint32) error {
    ids, err := k.getPositionsIdsByTier(ctx, tierId)
    ...
    for _, id := range ids {
        pos, err := k.getPositionState(ctx, id)
        ...
        if _, err := k.claimBaseRewards(ctx, pos); err != nil { return err }
        if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil { return err }
        if err := k.setPosition(ctx, pos.Position, nil); err != nil { return err }
    }
    return nil
}
```

Each iteration performs multiple KV reads and writes (position state, delegation lookup, validator event walk, bank send, distribution `WithdrawDelegationRewards`). The cost is strictly proportional to the number of positions in the tier.

There is no cap on the number of positions per tier. `LockTier` and `CommitDelegationToTier` are permissionless messages available to any account. `PositionCountByTier` is tracked but never used to enforce an upper bound.

The governance proposal execution path in Cosmos SDK runs the embedded `sdk.Msg` handlers inside a cached context. If the handler returns an error (or panics), the proposal is marked failed and the tier update never takes effect. Once the tier's position count is large enough to exhaust the block gas budget, every future `MsgUpdateTier` that touches `BonusApy` will fail, permanently locking the governance module out of that parameter.

### Impact Explanation

Governance permanently loses the ability to change `BonusApy` for any tier that has been flooded with positions. The concrete consequences are:

1. **Rewards pool drain**: If `BonusApy` is set too high and cannot be reduced, the rewards pool is depleted at the inflated rate. Once the pool is empty, every user-driven path that settles bonus rewards (`ClaimTierRewards`, `AddToPosition`, `TierUndelegate`, `TierRedelegate`, `ClearPosition`) fails atomically, freezing all position lifecycle operations for every user in the affected tier.
2. **Permanent governance lock on tier economics**: The chain cannot correct a misconfigured APY, cannot respond to market conditions, and cannot close a tier cleanly (since `close_only` is a separate field but `BonusApy` adjustments are blocked).

The corrupted invariant is: *governance must always be able to update tier parameters*. The corrupted value is the `BonusApy` field of the tier, which becomes immutable once the position count crosses the gas threshold.

### Likelihood Explanation

The attack requires creating many positions in a single tier. Each position requires locking the tier's `MinLockAmount` of the bond denom. For a tier with a low `MinLockAmount`, the cost per position is small. An attacker (or a coalition of users acting in concert) can distribute the cost across many addresses. The number of positions needed to exhaust block gas depends on the chain's gas limit and the per-position cost, but there is no protocol-level defense preventing accumulation. The attack is permanent once executed: positions are not automatically removed, and the attacker does not need to maintain them.

### Recommendation

1. **Enforce a per-tier position cap** checked inside `validateNewPosition`. Store the cap in the `Tier` proto and reject `LockTier`/`CommitDelegationToTier` when `PositionCountByTier[tierId] >= tier.MaxPositions`.
2. **Decouple the tier-sweep from `UpdateTier`**. Instead of settling all positions atomically at APY-change time, record the old APY alongside a `LastApyChangeSeq` on the tier and settle lazily per-position at claim time (similar to how validator events are already handled lazily). This eliminates the O(N) sweep entirely.
3. **Add a governance-level circuit breaker**: if the sweep is kept, cap the number of positions processed per `UpdateTier` call and emit an event when the cap is hit, so a follow-up governance proposal can process the remainder.

### Proof of Concept

1. Governance creates Tier 1 with `MinLockAmount = 1` and `BonusApy = 5%`.
2. Attacker (or many colluding users) calls `LockTier` with `amount = 1` from N distinct addresses, creating N positions in Tier 1. N is chosen so that iterating N positions in `claimRewardsAndUpdateTierPositions` exceeds the block gas limit.
3. Governance submits `MsgUpdateTier` with `BonusApy = 3%` to reduce the drain on the rewards pool.
4. The governance proposal passes voting but fails execution: `claimRewardsAndUpdateTierPositions` iterates all N positions, exhausts gas, and the handler returns an error. The proposal is marked failed; `BonusApy` remains at 5%.
5. Every subsequent governance proposal to change `BonusApy` for Tier 1 fails identically. The rewards pool drains at 5% until empty, after which all position lifecycle messages for Tier 1 fail with `ErrInsufficientBonusPool`.

```go
func (k Keeper) claimRewardsAndUpdateTierPositions(ctx context.Context, tierId uint32) error {
	ids, err := k.getPositionsIdsByTier(ctx, tierId)
	if err != nil {
		return err
	}
	if len(ids) == 0 {
		return nil
	}

	for _, id := range ids {
		pos, err := k.getPositionState(ctx, id)
		if err != nil {
			return err
		}
		if !pos.IsDelegated() {
			continue
		}

		if _, err := k.claimBaseRewards(ctx, pos); err != nil {
			return err
		}
		if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
			return err
		}
		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return err
		}
	}

	return nil
}
```
