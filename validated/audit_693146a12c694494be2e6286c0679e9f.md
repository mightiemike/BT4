### Title
Asymmetric Fee Formula in `applyFee` Overcharges Takers on Buy-Side Trades - (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.applyFee` applies two different fee formulas depending on the sign of `meteredQuote`. When `meteredQuote > 0` (seller receiving quote), the fee is correctly computed as `meteredQuote * feeRate`. When `meteredQuote < 0` (buyer paying quote), the code uses division by `keepRateX18` instead of multiplication, computing the fee as `|meteredQuote| * feeRate / (1 - feeRate)` — a strictly larger value than the stated rate. Every taker placing a buy order is systematically overcharged.

---

### Finding Description

In `OffchainExchange.sol`, `applyFee` computes the fee via a conditional branch on the sign of `meteredQuote`:

```solidity
int128 keepRateX18 = ONE - feeInfo.feeRate;
int128 newMeteredQuote = (meteredQuote > 0)
    ? meteredQuote.mul(keepRateX18)   // seller path
    : meteredQuote.div(keepRateX18);  // buyer path
orderInfo.fee = meteredQuote - newMeteredQuote;
```

`MathSD21x18.mul(x, y)` computes `x * y / 1e18` and `MathSD21x18.div(x, y)` computes `x * 1e18 / y`.

**Seller path** (`meteredQuote > 0`, taker selling or maker selling):
- `newMeteredQuote = meteredQuote * keepRate`
- `fee = meteredQuote - meteredQuote * keepRate = meteredQuote * feeRate` ✓

**Buyer path** (`meteredQuote < 0`, taker buying):
- `newMeteredQuote = meteredQuote / keepRate` (more negative, since `keepRate < 1`)
- `fee = meteredQuote - meteredQuote / keepRate = meteredQuote * (1 - 1/keepRate) = |meteredQuote| * feeRate / (1 - feeRate)` ✗

The buyer pays `feeRate / (1 - feeRate)` of trade value instead of `feeRate`. The overcharge per trade is:

```
overcharge = |meteredQuote| * feeRate² / (1 - feeRate)
```

For the default 2 bps taker fee (`feeRate = 0.0002`): overcharge ≈ `0.000004%` of trade value per fill. For a 100 bps custom tier: overcharge ≈ `0.0101%` per fill. The delta is small per trade but is systematic — it applies to every buy-side taker fill across all markets.

The fee is then applied to `quoteDelta`:

```solidity
orderInfo.quoteDelta = orderInfo.quoteDelta - orderInfo.fee - orderInfo.builderFee;
```

The excess fee flows into `market.collectedFees` and ultimately to the protocol's fee account (`X_ACCOUNT`), meaning the overcharge is a direct, permanent asset transfer away from the taker.

---

### Impact Explanation

Every taker placing a buy order (paying quote, `quoteDelta < 0`) pays a higher effective fee rate than the stated `feeRate`. The excess is `feeRate² / (1 - feeRate)` of the metered quote per fill. This is a direct, irreversible accounting corruption: the taker's `quoteDelta` is reduced by more than the protocol's stated fee schedule, and the surplus accrues to the protocol fee account. The invariant broken is: **effective taker fee rate == stated `feeRate`**. Takers selling (receiving quote) pay the correct rate; takers buying pay a higher rate, creating an asymmetry that is not disclosed or documented anywhere in the code.

---

### Likelihood Explanation

This triggers on every matched buy-side taker order submitted through `matchOrders()`. No special conditions, permissions, or adversarial setup are required. Any user who places a taker buy order through the normal sequencer flow hits this path. The entry point is `OffchainExchange.matchOrders` → internal `_matchOrderAMM` / `_matchOrders` → `applyFee`. The sequencer submits these on behalf of users; the user's signed order is the only prerequisite.

---

### Recommendation

Replace the conditional `mul`/`div` with a uniform fee computation that applies `feeRate` to the absolute trade value regardless of direction:

```solidity
// Before (buggy):
int128 newMeteredQuote = (meteredQuote > 0)
    ? meteredQuote.mul(keepRateX18)
    : meteredQuote.div(keepRateX18);
orderInfo.fee = meteredQuote - newMeteredQuote;

// After (correct):
// fee = |meteredQuote| * feeRate, preserving sign semantics
orderInfo.fee = meteredQuote.abs().mul(feeInfo.feeRate);
```

This ensures both buyers and sellers pay exactly `feeRate` of the metered quote value, consistent with the stated fee schedule. For negative `feeRate` (maker rebate), `abs().mul(feeRate)` correctly yields a negative fee (rebate), which when subtracted from `quoteDelta` increases the maker's receipt.

---

### Proof of Concept

Assume `feeRate = 200_000_000_000_000` (2 bps, the default), `keepRate = ONE - feeRate`.

**Taker selling** (receiving 10,000 USDC, `meteredQuote = 10_000e18`):
- `newMeteredQuote = 10_000e18 * 0.9998 = 9_998e18`
- `fee = 10_000e18 - 9_998e18 = 2e18` → **2 USDC** (= 2 bps of 10,000) ✓

**Taker buying** (paying 10,000 USDC, `meteredQuote = -10_000e18`):
- `newMeteredQuote = -10_000e18 / 0.9998 = -10_002.0004e18`
- `fee = -10_000e18 - (-10_002.0004e18) = 2.0004e18` → **2.0004 USDC** (= 2.0004 bps of 10,000) ✗

The buyer pays **0.0004 USDC more** than the seller for an identical 10,000 USDC trade. At $1B daily buy-side volume, this is ~$400/day in systematic overcharge. The overcharge scales quadratically with `feeRate`: at a 100 bps custom tier, the overcharge per $1M buy trade is ~$101. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
