Audit Report

## Title
`DepositAllowlistExtension#beforeAddLiquidity` checks `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension` is documented as gating `addLiquidity` by depositor address, but its `beforeAddLiquidity` hook silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and validates only `owner` (the caller-supplied LP-share recipient). Because `owner` is a free parameter, any non-allowlisted address can pass the check by naming any allowlisted address as `owner`, fully bypassing the intended access control on token provision.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` to the extension hook:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` correctly forwards both values:

```solidity
// ExtensionCalling.sol L95-98
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [2](#0-1) 

However, `DepositAllowlistExtension.beforeAddLiquidity` declares the first parameter (the actual sender) as unnamed and discards it, then checks only `owner`:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

`SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual swap initiator), confirming the asymmetry is a defect, not a design choice: [4](#0-3) 

No other guard in the `addLiquidity` path checks the caller's allowlist status. The `removeLiquidity` function enforces `msg.sender == owner`, meaning LP shares assigned to `owner` can only be withdrawn by `owner` — the attacker's tokens are locked unless `owner` cooperates. [5](#0-4) 

## Impact Explanation
Any pool deploying `DepositAllowlistExtension` to restrict liquidity providers (permissioned pools, KYC-gated pools) receives no protection against the actual token provider. A non-allowlisted address can:

1. Add liquidity to a pool that was supposed to exclude them, shifting `curBinIdx` and `curPosInBin` without restriction — directly corrupting pool bin state.
2. Exploit the shifted bin state to swap against the pool at an advantageous price, extracting value from existing LPs (direct loss of LP principal).
3. With a cooperating allowlisted address as `owner`, recover the deposited tokens via `removeLiquidity`, completing a round-trip that makes the attacker an effective LP in a pool designed to exclude them.

This constitutes a broken core pool access-control primitive causing potential loss of LP funds and pool-state manipulation, meeting the contest's High/Critical impact threshold.

## Likelihood Explanation
- Requires only knowledge of one allowlisted address, which is publicly readable from the `allowedDepositor` mapping or on-chain events.
- No privileged role, special token, oracle manipulation, or flash loan needed.
- Triggerable on any pool that has deployed `DepositAllowlistExtension` without an additional sender-level guard.
- The `owner` parameter is a standard, documented field of `addLiquidity`; any integrator or attacker reading the ABI can exploit this immediately.

## Recommendation
Check `sender` (the actual caller) instead of `owner` (the LP-share recipient), mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to restrict LP-share ownership rather than token provision, rename the extension and its setter to reflect that, and document the `owner`-check semantics explicitly so integrators are not misled.

## Proof of Concept

```
Setup:
  Pool P with DepositAllowlistExtension E configured in BEFORE_ADD_LIQUIDITY_ORDER
  allowedDepositor[P][Alice] = true
  allowedDepositor[P][Bob]   = false  // Bob is NOT allowlisted

Attack:
  Bob calls pool.addLiquidity(
      owner        = Alice,   // allowlisted — passes the hook
      salt         = 0,
      deltas       = <liquidity delta>,
      callbackData = "",
      extensionData = ""
  )

  Inside beforeAddLiquidity:
    first param (sender = Bob) → unnamed, discarded
    owner = Alice → allowedDepositor[P][Alice] == true → hook passes

  LiquidityLib calls IMetricOmmModifyLiquidityCallback(Bob).metricOmmModifyLiquidityCallback(...)
    Bob transfers tokens to pool.

  Alice receives LP shares.

Result:
  Bob has added liquidity to a pool that was supposed to exclude him.
  Pool's curBinIdx / curPosInBin are shifted by Bob's deposit.
  Bob can now swap against the pool at the shifted price to extract value from existing LPs.
  With Alice's cooperation, Bob recovers deposited tokens via removeLiquidity,
  completing a full allowlist bypass round-trip.
```
