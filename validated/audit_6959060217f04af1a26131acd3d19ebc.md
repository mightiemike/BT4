### Title
`only_query` Flag Hardcoded to `false` in `BroadcastedTransaction`→`ExecutableTransactionInput` Conversion Causes Wrong Transaction Hash and Execution Context in `estimate_fee` and `simulate_transactions` - (File: `crates/apollo_rpc/src/v0_8/api/mod.rs`)

---

### Summary

The Starknet protocol mandates that `starknet_estimateFee` and `starknet_simulateTransactions` execute transactions in **query mode** (`only_query = true`), which sets the 128th bit of the transaction version (e.g., `0x100000000000000000000000000000003` for V3). This bit is included in the transaction hash preimage and is exposed to contracts via the `get_execution_info` syscall. In the Apollo sequencer, the conversion from `BroadcastedTransaction` to `ExecutableTransactionInput` hardcodes `only_query = false` for all transaction types, causing fee estimation and simulation to execute with the wrong version, wrong hash, and wrong execution context.

---

### Finding Description

`ExecutableTransactionInput` carries an `OnlyQuery` boolean as a tuple field for every variant:

```rust

    Invoke(InvokeTransaction, OnlyQuery),
    DeclareV1(DeclareTransactionV0V1, DeprecatedContractClass, AbiSize, OnlyQuery),
    DeclareV2(DeclareTransactionV2, CasmContractClass, SierraSize, AbiSize, OnlyQuery, SierraVersion),
    DeclareV3(DeclareTransactionV3, CasmContractClass, SierraSize, AbiSize, OnlyQuery, SierraVersion),
    DeployAccount(DeployAccountTransaction, OnlyQuery),
    L1Handler(L1HandlerTransaction, Fee, OnlyQuery),
}
``` [1](#0-0) 

The `TryFrom<BroadcastedDeclareTransaction>` implementation hardcodes `false` with an explicit TODO:

```rust

``` [2](#0-1) 

Similarly, `stored_txn_to_executable_txn` (used for re-execution/tracing) hardcodes `false` for every transaction type: [3](#0-2) 

The `to_blockifier_tx` function in `apollo_rpc_execution` also carries a TODO acknowledging the issue:

``` [4](#0-3) 

The `only_query` flag flows into `calc_tx_hash`, which calls `get_transaction_hash` with `TransactionOptions { only_query }`: [5](#0-4) 

`get_transaction_hash` calls `signed_tx_version_from_tx`, which sets the 128th bit when `only_query = true`:

```rust
pub fn signed_tx_version(...) -> TransactionVersion {
    // If only_query is true, set the 128-th bit.
    let query_only_bit = *QUERY_VERSION_BASE;
    if transaction_options.only_query {
        TransactionVersion(tx_version.0 + query_only_bit)
    } else {
        *tx_version
    }
}
``` [6](#0-5) 

The version is chained directly into the hash preimage for all V3 transaction types: [7](#0-6) 

The `only_query` flag is also forwarded into `ExecutionFlags` and from there into `create_tx_info`, which exposes it to contracts via the `get_execution_info` syscall: [8](#0-7) 

The RPC `estimate_fee` handler converts `BroadcastedTransaction` directly via `try_into()` without overriding `only_query`:

```rust
let executable_txns =
    transactions.into_iter().map(|tx| tx.try_into()).collect::<Result<_, _>>()?;
``` [9](#0-8) 

The same pattern applies to `simulate_transactions`: [10](#0-9) 

---

### Impact Explanation

Two concrete wrong values are produced for every `starknet_estimateFee` and `starknet_simulateTransactions` call:

1. **Wrong transaction hash in execution context.** The hash is computed with version `3` instead of `0x100000000000000000000000000000003`. Any contract that reads `get_execution_info().tx_info.transaction_hash` during fee estimation receives a hash that does not match the query-mode hash the Starknet spec mandates. This is an authoritative-looking wrong value returned by the RPC.

2. **Wrong `is_query` / version exposed to contracts.** Account contracts that inspect `get_execution_info().tx_info.version` to detect query mode (e.g., to skip state-mutating side effects or to short-circuit validation during simulation) will see `version = 3` instead of the query version. This causes them to execute the full non-query code path, producing wrong gas consumption figures and thus wrong fee estimates.

Impact category: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

Every call to `starknet_estimateFee` or `starknet_simulateTransactions` that involves a contract which reads `get_execution_info` is affected. This is a structural, unconditional misconfiguration — no special input is required. Any user submitting a fee estimation request triggers the wrong behavior.

---

### Recommendation

In the `TryFrom<BroadcastedTransaction>` (and `TryFrom<BroadcastedDeclareTransaction>`) implementations for `ExecutableTransactionInput`, set `only_query = true` unconditionally, since these conversions are only used for query operations (`estimate_fee`, `simulate_transactions`). The existing TODO comments at `crates/apollo_rpc/src/v0_8/api/mod.rs:506` and `crates/apollo_rpc_execution/src/lib.rs:811` already identify this gap. The fix is to replace the hardcoded `false` with `true` in all `BroadcastedTransaction`→`ExecutableTransactionInput` conversion arms used by the RPC query path.

---

### Proof of Concept

1. Deploy an account contract whose `__validate__` entry point reads `get_execution_info().tx_info.version` and branches on whether the 128th bit is set (query mode). In query mode it returns immediately; in non-query mode it performs an expensive computation.

2. Call `starknet_estimateFee` with an invoke transaction from this account.

3. Observe that the fee estimate reflects the expensive non-query path, because `only_query = false` is passed to the blockifier. The contract sees `version = 3` instead of `0x100000000000000000000000000000003`.

4. Confirm by calling `starknet_simulateTransactions` and inspecting the `validate_invocation` trace — it will show the full non-query execution path, not the short-circuit path, and the `tx_info.version` field in the trace will be `3` rather than the query version.

pub fn signed_tx_version(
    tx_version: &TransactionVersion,
    transaction_options: &TransactionOptions,
) -> TransactionVersion {
    // If only_query is true, set the 128-th bit.
    let query_only_bit = *QUERY_VERSION_BASE;
    assert_eq!(
        tx_version.0.to_biguint() & query_only_bit.to_biguint(),
        BigUint::from(0_u8),
        "Requested signed tx version with version that already has query bit set: {tx_version:?}."
    );
    if transaction_options.only_query {
        TransactionVersion(tx_version.0 + query_only_bit)
    } else {
        *tx_version
    }
}
