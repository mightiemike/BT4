Audit Report

## Title
`SwapAllowlistExtension` Gates the Router Address Instead of the Actual End User, Enabling Full Allowlist Bypass via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument supplied by the pool, which is the pool's `msg.sender` — the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, that direct caller is the router contract, not the end user. If the pool admin allowlists the router address (a natural step to enable router-mediated swaps for curated pools), every unprivileged user can bypass the per-user allowlist by routing through the router, as the actual end user's address is never inspected.

## Finding Description

**Root cause — wrong actor checked in `beforeSwap`**

`SwapAllowlistExtension.beforeSwap` evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the pool calls the extension) and `sender` is the first argument forwarded by the pool. [1](#0-0) 

**How the pool populates `sender`**

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap(), NOT the end user
    recipient,
    ...
    extensionData
);
``` [2](#0-1) 

`ExtensionCalling._beforeSwap` faithfully forwards this value as the first argument to every configured extension: [3](#0-2) 

**How the router calls the pool**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly. The actual end user's address (`msg.sender` from the router's perspective) is stored only in transient storage for the payment callback — it is never forwarded to the pool as a named argument:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
``` [4](#0-3) 

The same pattern applies to `exactInput` (multi-hop) and `exactOutputSingle`: [5](#0-4) 

**The bypass chain**

When any user calls `router.exactInputSingle(pool, ...)`:
1. Router → `pool.swap()` with `msg.sender = router`
2. Pool → `extension.beforeSwap(sender = router, ...)`
3. Extension evaluates `allowedSwapper[pool][router]`

If the pool admin has set `allowedSwapper[pool][router] = true` (to permit router-mediated swaps for their curated pool), the check passes for every caller regardless of whether that caller is individually allowlisted. The actual end user's address is never inspected.

The admin setter that creates the precondition: [6](#0-5) 

**Secondary broken-functionality consequence**

Even without the bypass, if the admin only allowlists individual user addresses (not the router), those allowlisted users cannot swap through the router at all — the extension sees the router address and reverts. Since `MetricOmmSimpleRouter` is the protocol's primary user-facing swap entrypoint, this silently breaks the core swap flow for every curated pool.

## Impact Explanation

LP providers on a curated pool expect to trade only against a vetted counterparty set. Once the router is allowlisted, any unprivileged user can execute swaps against those LPs. The allowlist guard — the sole access-control mechanism on the swap path — fails completely open, enabling non-allowlisted actors (arbitrageurs, MEV bots) to extract value from LP positions at will, causing direct loss of LP principal and accrued fees. This constitutes broken core pool functionality causing loss of funds, matching the required impact gate.

## Likelihood Explanation

The precondition is that the pool admin allowlists the router address. This is a natural, non-malicious administrative action: a pool operator who wants their curated users to enjoy the router's UX (slippage protection, multi-hop, deadline checks) must allowlist the router. The design gives the admin no way to simultaneously allow router-mediated swaps for approved users and block router-mediated swaps for unapproved users. The precondition is reachable through ordinary, good-faith pool administration and is repeatable by any unprivileged user indefinitely.

## Recommendation

The extension must identify the economic actor (the end user who approved the token spend and receives the output), not the intermediary contract. Viable approaches:

1. **Pass the originating user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. Requires a coordinated convention between router and extension.
2. **Read the payer from transient storage**: Expose a view on the router's transient payer slot so the extension can retrieve the true originator during the hook call.
3. **Two-tier allowlist**: Gate the router address at one level and the per-user identity (passed via `extensionData`) at a second level, so both can be enforced independently.

## Proof of Concept

```
Setup
─────
1. Pool admin deploys a pool with SwapAllowlistExtension wired into beforeSwap.
2. Admin calls setAllowedToSwap(pool, router, true)
   — intending to allow router-mediated swaps for their curated users.
3. Admin calls setAllowedToSwap(pool, alice, true)
   — alice is the only individually approved trader.

Attack
──────
4. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, ..., extensionData: ""})

5. Router calls pool.swap(recipient=bob, ...) with msg.sender = router.

6. Pool calls extension.beforeSwap(sender=router, ...).

7. Extension evaluates:
       allowAllSwappers[pool]          → false
       allowedSwapper[pool][router]    → true   ← passes

8. Swap executes. Bob receives output tokens.
   The allowlist guard never inspected Bob's address.

Result: Bob, an unprivileged user, trades on a curated pool whose LP
providers expected only alice as a counterparty. Bob can repeat this
arbitrarily, draining LP value.

Foundry test sketch
───────────────────
- Deploy pool with SwapAllowlistExtension as beforeSwap hook.
- Call setAllowedToSwap(pool, address(router), true).
- Prank as an address NOT in the allowlist.
- Call router.exactInputSingle(...) targeting the pool.
- Assert the swap succeeds (no NotAllowedToSwap revert).
- Assert the non-allowlisted address received output tokens.
```
