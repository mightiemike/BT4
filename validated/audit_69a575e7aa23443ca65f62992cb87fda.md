### Title
Native ETH Permanently Locked in `DirectDepositV1` Due to Missing `receive()` in `ContractOwner` — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.withdrawFromDirectDepositV1()` is the sole privileged recovery path for native ETH held in a `DirectDepositV1` (DDA) contract. It calls `DirectDepositV1.withdrawNative()`, which attempts to push ETH to `ContractOwner` via a low-level call. Because `ContractOwner` declares no `receive()` or `fallback()` function, this ETH transfer always fails and reverts. Any native ETH that reaches a DDA — whether through a `selfdestruct` force-send, a CREATE2 front-run, or a constructor-time wrapping failure — is permanently locked with no on-chain recovery path.

---

### Finding Description

`DirectDepositV1` exposes a `withdrawNative()` function restricted to its owner:

```solidity
function withdrawNative() external onlyOwner {
    uint256 balance = address(this).balance;
    (bool success, ) = msg.sender.call{value: balance}("");
    require(success, "Failed to transfer native token to owner");
}
``` [1](#0-0) 

The owner of every DDA is `ContractOwner`, set implicitly because `ContractOwner.createDirectDepositV1()` is the only factory that deploys DDAs:

```solidity
DirectDepositV1 directDepositV1 = new DirectDepositV1{
    salt: bytes32(uint256(1))
}(address(endpoint), address(spotEngine), subaccount, wrappedNative);
``` [2](#0-1) 

`ContractOwner` exposes `withdrawFromDirectDepositV1()` as the intended recovery path for native ETH (`token == address(0)`):

```solidity
uint256 preBalance = address(this).balance;
DirectDepositV1(directDepositV1).withdrawNative();
uint256 postBalance = address(this).balance;
require(postBalance > preBalance, "empty");
(bool success, ) = msg.sender.call{value: postBalance - preBalance}("");
require(success, "xfer");
``` [3](#0-2) 

The call chain is: `ContractOwner.withdrawFromDirectDepositV1` → `DirectDepositV1.withdrawNative()` → `msg.sender.call{value: balance}("")` where `msg.sender` is `ContractOwner`. `ContractOwner` has no `receive()` or `fallback()` function anywhere in its body. The low-level ETH push therefore fails unconditionally, causing `withdrawNative()` to revert with `"Failed to transfer native token to owner"`, which bubbles up and reverts `withdrawFromDirectDepositV1` as well. [4](#0-3) 

The `receive()` function in `DirectDepositV1` wraps ETH to WETH immediately and `require`s success, so ETH does not accumulate through normal deposits. However, two realistic paths bypass it:

1. **`selfdestruct` force-send**: Any contract can call `selfdestruct(dda_address)`, pushing ETH directly to the DDA without invoking `receive()`. This is always available to any unprivileged caller.
2. **CREATE2 front-run + constructor wrapping failure**: Because `createDirectDepositV1` uses a fixed `salt: bytes32(uint256(1))`, the DDA address for any subaccount is deterministic and pre-computable. An attacker can send ETH to that address before deployment. The constructor attempts to wrap it but only emits `NativeTokenTransferFailed` on failure — it does not revert — leaving ETH stranded:

```solidity
if (!success) {
    emit NativeTokenTransferFailed(balance);
}
``` [5](#0-4) 

In both cases, the ETH is in the DDA, `withdrawNative()` is the only exit, and `ContractOwner` cannot receive the ETH.

---

### Impact Explanation

Native ETH that reaches any `DirectDepositV1` contract is permanently locked. There is no alternative withdrawal path: `withdrawNative()` is `onlyOwner`, the owner is `ContractOwner`, and `ContractOwner` cannot receive ETH. The asset delta is a permanent loss of all native ETH held in any DDA, with no recovery possible by the protocol, the owner, or any administrator.

---

### Likelihood Explanation

The `selfdestruct` vector is unconditionally available to any unprivileged caller at zero protocol permission. It requires only that the attacker knows a deployed DDA address (all are stored in the public mapping `directDepositV1Address`) and is willing to sacrifice ETH to grief the protocol. The CREATE2 front-run vector requires predicting the DDA address before deployment and a coincident wrapping failure, making it lower likelihood but still reachable without any privileged access.

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept ETH forwarded from `DirectDepositV1.withdrawNative()`:

```solidity
receive() external payable {}
```

Alternatively, refactor `withdrawFromDirectDepositV1` to avoid routing ETH through `ContractOwner` by having `withdrawNative()` accept a `sendTo` address parameter and push directly to `msg.sender` (the multisig owner), bypassing `ContractOwner` entirely.

---

### Proof of Concept

```solidity
// Attacker contract
contract Attacker {
    constructor(address payable dda) payable {
        selfdestruct(dda); // force-sends ETH to DDA, bypasses receive()
    }

// 1. ETH is now in the DDA, not wrapped as WETH
// 2. Owner calls ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))

            uint256 postBalance = address(this).balance;
            require(postBalance > preBalance, "empty");

            require(success, "xfer");

            uint256 preBalance = IERC20Base(token).balanceOf(address(this));
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(token));
            uint256 postBalance = IERC20Base(token).balanceOf(address(this));
            require(postBalance > preBalance, "empty");
            IERC20Base(token).safeTransfer(
                msg.sender,
                postBalance - preBalance
