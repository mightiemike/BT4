### Title
Bonus Reward Checkpoints Advanced Without Payment on Insufficient Pool During Redelegation Slash - (`x/tieredrewards/keeper/slash.go`)

---

### Summary

In `slashRedelegationPosition`, when `processEventsAndClaimBonus` fails with `ErrInsufficientBonusPool`, the function catches and silences the error, then persists the position with its already-advanced reward checkpoints (`LastBonusAccrual`, `LastEventSeq`, `LastKnownBonded`) and decremented event reference counts. The bonus is never paid, but the accounting state is permanently advanced past the accrual period. The affected position owner can never recover the lost bonus.

---

### Finding Description

`slashRedelegationPosition` in `x/tieredrewards/keeper/slash.go` calls `processEventsAndClaimBonus` to settle bonus rewards against pre-slash shares before the staking `Unbond` runs: [1](#0-0) 

When `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, the error is caught and execution continues. The function then calls `setPositionWithState` to persist the position: [2](#0-1) 

The problem is what `processEventsAndClaimBonus` has already done before returning the error. Inside the event-replay loop, it:

1. Advances `pos.LastEventSeq` for each processed event (line 193)
2. Calls `k.decrementEventRefCount` — a direct store write that is **not rolled back** (line 196)

After the loop, it advances `pos.LastBonusAccrual` and `pos.LastKnownBonded` unconditionally (lines 215–217), then checks pool balance and returns the error: [3](#0-2) [4](#0-3) 

Because `slashRedelegationPosition` ignores the error and calls `setPositionWithState` with the mutated `pos`, the position is saved with:
- `LastBonusAccrual` advanced to `blockTime` (or `ExitUnlockAt`)
- `LastEventSeq` advanced past all replayed events
- `LastKnownBonded` updated

But the bonus coins were never transferred. On any future claim, `getValidatorEventsSince(ctx, valAddr, pos.LastEventSeq)` returns only events **after** the advanced sequence, so the accrual period that was processed but not paid is permanently skipped. The event reference counts were also already decremented, potentially garbage-collecting those events from the store.

---

### Impact Explanation

A position owner whose redelegation is slashed while the bonus pool is empty permanently loses all accrued bonus rewards up to the slash block. The accounting checkpoints are advanced as if payment occurred, but no coins are transferred. There is no retry path: the position's `LastBonusAccrual` and `LastEventSeq` are already past the lost period, and the relevant validator events may have been garbage-collected.

The corrupted value is: `position.LastBonusAccrual` / `position.LastEventSeq` — they are advanced to reflect a payment that was never made, permanently destroying the user's claim to the accrued bonus.

---

### Likelihood Explanation

The trigger requires two concurrent conditions:
1. A validator is slashed while a tier position has an active redelegation entry to that validator (tracked via `RedelegationMappings`).
2. The `tieredrewards` rewards pool (`types.RewardsPoolName`) has insufficient balance to cover the accrued bonus at that moment.

Both conditions are realistic in production. The pool can be empty if governance has not funded it, or if `BeginBlocker` has drained it via base-reward top-ups. Validator slashing is a normal protocol event. No privileged access or attacker action is required — the loss is triggered by ordinary protocol mechanics.

---

### Recommendation

In `slashRedelegationPosition`, when `ErrInsufficientBonusPool` is caught, the position's in-memory checkpoints must **not** be persisted with the advanced values. Either:

1. Use a cache-wrapped context for the `processEventsAndClaimBonus` call so that store writes (ref count decrements) are discarded on error, and do not call `setPositionWithState` with the mutated position when the error is caught; or
2. Only advance `LastBonusAccrual` / `LastEventSeq` / `LastKnownBonded` after a successful payment, and skip the checkpoint update entirely when the pool is insufficient (accepting that the bonus will be re-computed on the next claim once the pool is replenished).

Option 2 is simpler: move `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` to after the `sufficientBonusPoolBalance` check, and similarly guard `pos.UpdateLastEventSeq` and `decrementEventRefCount` so they only execute when payment will succeed.

---

### Proof of Concept

1. A position owner redelegates from validator A to validator B. A `RedelegationMappings` entry is created.
2. The `tieredrewards` rewards pool is empty (e.g., drained by `BeginBlocker` top-ups or simply unfunded).
3. Validator B is slashed. The staking module calls `BeforeRedelegationSlashed` → `slashRedelegationPosition`.
4. `processEventsAndClaimBonus` runs, advances `LastEventSeq` and `LastBonusAccrual` in memory, decrements event ref counts in the store, then returns `ErrInsufficientBonusPool`.
5. `slashRedelegationPosition` logs the error and calls `setPositionWithState` with the mutated position.
6. The position is now stored with advanced checkpoints. No bonus was paid.
7. The pool is later replenished. The owner calls `MsgClaimTierRewards`. `getValidatorEventsSince` returns only events after the advanced `LastEventSeq`; the accrual period before the slash is permanently gone. The owner receives zero bonus for that period. [5](#0-4) [6](#0-5)
