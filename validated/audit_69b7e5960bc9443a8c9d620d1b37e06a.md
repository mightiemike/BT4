### Title
`max_l2_gas_amount` Admission Bound Not Enforced for Declare Transactions — (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The gateway's stateless validator enforces a `max_l2_gas_amount` ceiling on `l2_gas.max_amount` for Invoke and DeployAccount transactions, but explicitly skips this check for Declare transactions. A user can submit a Declare transaction with an arbitrarily large `l2_gas.max_amount` (e.g., `u64::MAX`) and it will pass gateway admission, entering the mempool unchecked.

### Finding Description

In `validate_resource_bounds`, the `max_l2_gas_amount` upper-bound check is guarded by a transaction-type branch that silently no-ops for Declare:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← no check performed; any l2_gas.max_amount is accepted
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
``` [1](#0-0) 

The config value `max_l2_gas_amount` is set to `1_210_000_000` in production: [2](#0-1) 

The stateful validator's `validate_resource_bounds` only checks the L2 gas *price* threshold against the previous block's price — it does not re-enforce the `max_l2_gas_amount` ceiling: [3](#0-2) 

There is no downstream enforcement of `max_l2_gas_amount` for Declare transactions anywhere in the admission pipeline. The blockifier's pre-validation only checks that `max_amount >= minimal_gas_amount`, so a huge `max_amount` always passes that check: [4](#0-3) 

The test for `test_invalid_max_l2_gas_amount` explicitly excludes `TransactionType::Declare` from its test cases, confirming the gap is known but untested: [5](#0-4) 

### Impact Explanation

**High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

A Declare transaction with `l2_gas.max_amount = u64::MAX` (or any value above `max_l2_gas_amount`) passes all gateway stateless and stateful checks and is admitted to the mempool. The `max_l2_gas_amount` limit exists specifically to bound the resource footprint of admitted transactions; bypassing it for Declare transactions undermines this admission control invariant. An attacker can flood the mempool with Declare transactions carrying unbounded gas amounts, which the gateway is supposed to reject.

### Likelihood Explanation

**High.** The trigger requires only a well-formed Declare V3 transaction with `l2_gas.max_amount > 1_210_000_000`. No special privilege is required. The TODO comment at line 78 confirms the developers are aware of the gap but have not closed it.

### Recommendation

Apply the same `max_l2_gas_amount` check to Declare transactions, removing the empty `if let RpcTransaction::Declare(_) = tx {}` branch:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

Add a corresponding test case for `TransactionType::Declare` in `test_invalid_max_l2_gas_amount`.

### Proof of Concept

1. Construct a valid `RpcDeclareTransaction::V3` with:
   - `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)` (far above `max_l2_gas_amount = 1_210_000_000`)
   - `resource_bounds.l2_gas.max_price_per_unit` set to the configured `min_gas_price`
   - All other fields valid (valid Sierra class, valid sender address, etc.)
2. Submit to the gateway's `add_tx` endpoint.
3. Observe: `stateless_tx_validator.validate(&tx)` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never raised for Declare transactions.
4. The transaction proceeds through stateful validation and enters the mempool with an unchecked `l2_gas.max_amount`, bypassing the admission bound that applies to all other transaction types.
