Audit Report

## Title
`PriceVelocityGuardExtension.beforeSwap` resets `lastMidPriceX64` before the guard check, allowing cumulative intra-block price movement to exceed the configured per-block cap — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary

`PriceVelocityGuardExtension.beforeSwap` unconditionally writes `lastMidPriceX64 = midPrice` and `lastUpdateBlock = block.number` at lines 57–58, before the velocity check executes. Every subsequent swap in the same block therefore compares against the prior swap's mid price, not the start-of-block mid price. An attacker who chains N swaps within one block — each individually within the `maxChangePerBlockE18` envelope — accumulates a total intra-block price movement of up to N × `maxChangePerBlockE18`, directly contradicting the NatSpec invariant that the guard "caps how fast the provided price can move between blocks."

## Finding Description

In `beforeSwap` (lines 53–75), the storage state is updated unconditionally before the guard:

```solidity
uint128 prevMid = s.lastMidPriceX64;   // read old value
uint64 prevBlock = s.lastUpdateBlock;

s.lastMidPriceX64 = midPrice;          // ← committed before check (line 57)
s.lastUpdateBlock = uint64(block.number); // ← (line 58)

if (prevMid != 0) {
    uint256 blockDiff = block.number - prevBlock; // 0 for same-block swaps
    uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
    // allowedSq = maxChange² when blockDiff == 0
    if (actualSq > allowedSq) revert PriceVelocityExceeded(...);
}
```

For two swaps in the same block with `blockDiff = 0`, `allowedSq = maxChange²`. Swap 1 compares current mid against P₀ (start-of-block); swap 2 compares against P₁ (written by swap 1). The guard never sees the cumulative delta from P₀. After N swaps the oracle mid has moved from P₀ to P₀ × (1 + maxChange)^N while every individual check passed. The NatSpec at line 10 states the guard "caps how fast the provided price can move **between blocks**," but the implementation enforces only a per-swap cap. [1](#0-0) [2](#0-1) 

## Impact Explanation

The velocity guard is the primary on-chain protection against oracle-price manipulation for pools that configure it. Bypassing it allows the pool's bid/ask quotes — anchored to the oracle mid — to be driven arbitrarily far from the true start-of-block price within a single block. LP bins priced at the stale start-of-block mid are then drained at a discount equal to the excess cumulative movement, constituting a direct loss of LP principal. This matches the allowed impact gate: **bad-price execution** (unclamped oracle-derived bid/ask quote reaches a pool swap) and **direct loss of LP principal**.

## Likelihood Explanation

Pyth Network price feeds are updated permissionlessly: any party holding a valid VAA (freely available from the Pyth off-chain API) can call `updatePriceFeeds`. On chains with sub-second block times or via Flashbots bundles on Ethereum, an attacker can interleave multiple `updatePriceFeeds` + `pool.swap` calls within a single block. No privileged role is required beyond being an allowed swapper (or the pool having no swap allowlist). The `PythOracle.sol` oracle layer is in scope and supports this update pattern.

## Recommendation

Only update `lastMidPriceX64` when a new block has started. Within the same block, compare the current mid against the value stored at the beginning of the block without overwriting it:

```solidity
if (block.number > prevBlock) {
    s.lastMidPriceX64 = midPrice;
    s.lastUpdateBlock = uint64(block.number);
} else {
    // same block: do NOT update lastMidPriceX64; compare against block-start reference
}
```

Alternatively, store a separate `blockStartMidPriceX64` that is written only once per block and always used as the comparison baseline.

## Proof of Concept

1. Pool configured with `PriceVelocityGuardExtension`, `maxChangePerBlockE18 = 0.01e18` (1% per block).
2. Oracle mid at block N−1: P₀ = 1.00.
3. Attacker bundles in block N:
   - `updatePriceFeeds(VAA: P₁ = 1.01)` → `pool.swap(...)` → Δ = 1%, allowed = 1%, passes; `lastMidPriceX64 = P₁`.
   - `updatePriceFeeds(VAA: P₂ = 1.0201)` → `pool.swap(...)` → Δ = 1% from P₁, passes; `lastMidPriceX64 = P₂`.
   - Repeat 10 times → final mid ≈ 1.1046 (10.46% above P₀).
4. Each individual swap passed the 1% cap; cumulative intra-block movement is ~10.46%, 10× the intended limit.
5. LP bins priced at P₀ are drained at a ~10% discount, extracting LP principal.
