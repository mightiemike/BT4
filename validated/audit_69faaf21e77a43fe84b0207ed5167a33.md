### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling Merkle Proof Forgery via Internal-Node Substitution — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still a live, unrestricted public NEAR method despite being deprecated. Any unprivileged NEAR caller can invoke it with an internal Merkle-tree node hash as `tx_id` and a shortened sibling path as `merkle_proof`, causing the function to return `true` for a transaction that never existed. The `verify_transaction_inclusion_v2` coinbase-anchor fix is entirely bypassed because v1 remains independently reachable.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte Merkle-proof forgery path by requiring a coinbase proof of equal depth, forcing the claimed leaf to sit at the true leaf level of the tree. The fix is sound when v2 is the only callable path. However, v1 is still a public contract method:

```rust

`#[deprecated]` in Rust emits a compiler warning; it does **not** gate runtime access. Any NEAR account can call `verify_transaction_inclusion` directly, skipping the coinbase-anchor check in v2 entirely.

The sole proof check inside v1 is:

```rust

`compute_root_from_merkle_proof` is a pure hash-chain walk:

```rust
// merkle-tools/src/lib.rs  lines 34-52
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;
    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }
    current_hash
}
```

If `tx_id` is an **internal node** `N` at depth `D` in a real block's Merkle tree, the attacker supplies the `tree_depth − D` sibling hashes above `N`. The walk produces the correct `merkle_root`, so the function returns `true` — for a hash that is not a transaction at all.

The structural parallel to the reported Astaria bug is exact:

| Astaria | BTC light client |
|---|---|
| `commitToLien` callable by anyone | `verify_transaction_inclusion` callable by anyone |
| `!ROUTER().isValidVault(receiver)` skips the holder/operator check when receiver is a registered vault | Merkle-root equality check passes when `tx_id` is an internal node with a shortened proof |
| Attacker registers a vault (trivial) to satisfy the bypass condition | Attacker reads any mainchain block's Merkle tree (trivial) to obtain a valid internal node |
| Funds redirected to attacker-controlled vault | Proof result forged; consumer contract deceived into releasing funds |

---

### Impact Explanation

Any NEAR contract that calls `verify_transaction_inclusion` to gate a fund-release (bridge payout, atomic-swap settlement, cross-chain claim) can be made to return `true` for a fabricated transaction hash. The attacker does not need to produce a real Bitcoin transaction; they only need to identify an internal Merkle node in any mainchain block and supply the corresponding shortened sibling path. The contract's canonical-chain state is not corrupted, but the **proof result** — the only output consumer contracts consume — is forged.

---

### Likelihood Explanation

The attack requires zero privileges and zero mining power:

1. Query any Bitcoin block explorer for a mainchain block already stored by the contract.
2. Read the block's full transaction list and compute the Merkle tree (all data is public).
3. Pick any internal node `N` at depth `D ≥ 1`.
4. Derive the `tree_depth − D` sibling hashes above `N`.
5. Call `verify_transaction_inclusion` with `tx_id = N`, the correct `tx_block_blockhash`, the matching `tx_index`, and the shortened `merkle_proof`.

The function returns `true`. The only prerequisite is that the target block is in the contract's mainchain (guaranteed for any recently submitted block).

---

### Recommendation

1. **Remove `verify_transaction_inclusion` entirely**, or gate it with `#[private]` so only the contract itself can call it (which would restrict it to the internal call from v2).
2. If backward compatibility is required, add a runtime guard at the top of v1 that panics unconditionally, forcing all callers to migrate to v2.
3. Alternatively, enforce the coinbase-anchor constraint inside v1 itself so the two code paths are equivalent.

---

### Proof of Concept

```
Block B is in the mainchain. Its Merkle tree (depth = 3, 8 leaves) has:
  root R
  ├── node A = SHA256d(tx0 || tx1)
  │   ├── tx0  (coinbase, index 0)
  │   └── tx1  (index 1)
  └── node B_node = SHA256d(tx2 || tx3)
      ├── tx2  (index 2)
      └── tx3  (index 3)

Attacker picks internal node A (depth 1, position 0).
Shortened proof = [B_node, SHA256d(C_node || D_node)]  (length 2, not 3)

    confirmations:     1,

  step 1: pos=0 (even) → SHA256d(A || B_node) = root_left
