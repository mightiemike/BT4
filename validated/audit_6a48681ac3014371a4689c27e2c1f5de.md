### Title
Sanctioned Address Can Withdraw Funds via Slow Mode Queue Due to Missing Sanctions Re-Check at Execution Time - (File: `core/contracts/Endpoint.sol`, `core/contracts/EndpointTx.sol`)

---

### Summary

The Nado protocol enforces a sanctions check (`requireUnsanctioned`) when a user submits a slow mode transaction, but does **not** re-check sanctions status when the queued transaction is executed 3 days later. This is structurally identical to the reported bug: a deferred/asynchronous execution path bypasses a critical state guard that was only checked at enqueue time.

---

### Finding Description

The slow mode system is a two-phase mechanism:

**Phase 1 – Submission** (`submitSlowModeTransactionImpl` in `EndpointTx.sol`):

```solidity
IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
requireUnsanctioned(sender);                          // ← sanctions checked HERE
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
    sender: sender,
    tx: transaction
});
slowModeConfig = _slowModeConfig;
``` [1](#0-0) 

**Phase 2 – Execution** (`executeSlowModeTransaction` → `_executeSlowModeTransaction` → `processSlowModeTransactionImpl` in `Endpoint.sol` / `EndpointTx.sol`):

```solidity
function executeSlowModeTransaction() external {
    SlowModeConfig memory _slowModeConfig = slowModeConfig;
    _executeSlowModeTransaction(_slowModeConfig, false);   // ← no sanctions check
    nSubmissions += 1;
    slowModeConfig = _slowModeConfig;
}
``` [2](#0-1) 

Inside `processSlowModeTransactionImpl`, the `WithdrawCollateral` branch directly calls `clearinghouse.withdrawCollateral` with no sanctions re-check:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.WithdrawCollateral)
    );
    validateSender(txn.sender, sender);
    clearinghouse.withdrawCollateral(
        txn.sender, txn.productId, txn.amount, address(0), nSubmissions
    );
``` [3](#0-2) 

The `SLOW_MODE_TX_DELAY` is hardcoded to 3 days, creating a window during which a user's sanctions status can change. [4](#0-3) 

`executeSlowModeTransaction` is a permissionless `external` function — any caller can trigger execution of queued transactions. [5](#0-4) 

---

### Impact Explanation

A user who is added to the OFAC/sanctions list **after** submitting a `WithdrawCollateral` slow mode transaction can still have their withdrawal executed 3 days later. The actual asset transfer out of the protocol occurs at execution time, not submission time. This allows a sanctioned entity to successfully extract funds from the protocol, violating the protocol's compliance invariant that sanctioned addresses must not be able to move funds.

The `requireUnsanctioned` check in `EndpointStorage` is the protocol's sole enforcement mechanism for this invariant: [6](#0-5) 

Since it is only applied at submission and not at execution, the invariant is broken for the deferred path.

---

### Likelihood Explanation

OFAC sanctions lists are updated regularly. The 3-day mandatory delay (`SLOW_MODE_TX_DELAY`) between submission and execution creates a concrete, realistic window for a user's status to change. The execution function is permissionless, so the sanctioned user (or any third party) can trigger it. No special privileges or external compromise are required.

---

### Recommendation

Add a `requireUnsanctioned` check on the stored `txn.sender` (or the `sender` parameter) inside `processSlowModeTransactionImpl` at the start of execution, before any state-changing operation is performed. At minimum, it must be applied to the `WithdrawCollateral` and `LinkSigner` branches, which have direct asset or signer-state impact.

```solidity
function processSlowModeTransactionImpl(
    address sender,
    bytes calldata transaction
) public {
    requireUnsanctioned(sender); // ← add this
    ...
}
```

---

### Proof of Concept

1. User `A` submits a `WithdrawCollateral` slow mode transaction via `submitSlowModeTransaction`. `requireUnsanctioned(A)` passes at this point.
2. `A` is added to the OFAC sanctions list the next day.
3. After 3 days, anyone calls `executeSlowModeTransaction()`.
4. `_executeSlowModeTransaction` dequeues the transaction and calls `processSlowModeTransaction(A, tx)`.
5. `processSlowModeTransactionImpl` decodes the `WithdrawCollateral` payload and calls `clearinghouse.withdrawCollateral(A.subaccount, ...)` — no sanctions check occurs.
6. Funds are transferred out of the protocol to the sanctioned address, violating the protocol's compliance invariant.
