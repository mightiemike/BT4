Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the originating user, allowing any unprivileged user to bypass the per-pool swap allowlist — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` passes `msg.sender` (the direct caller) as the `sender` argument to `_beforeSwap`, which forwards it to `SwapAllowlistExtension.beforeSwap`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` inside the pool is the router, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. Any pool admin who allowlists the router to enable router-mediated swaps inadvertently grants every unprivileged user the ability to bypass the allowlist entirely.

## Finding Description
`MetricOmmPool.swap` at lines 230–240 passes `msg.sender` as `sender` to `_beforeSwap`:

```solidity
_beforeSwap(
  msg.sender,   // ← direct caller, not originating user
  recipient,
  ...
  extensionData
);
```

`SwapAllowlistExtension.beforeSwap` (lines 37–39) checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap`. When `MetricOmmSimpleRouter.exactInputSingle` (lines 72–80) calls `pool.swap(...)`, the pool sees `msg.sender = router`, so `sender = router`. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][alice]`.

The `extensionData` field is passed through the router unchanged from the caller, but `SwapAllowlistExtension.beforeSwap` ignores it entirely (the last `bytes calldata` parameter is unnamed and unused). There is no existing mechanism for the router to communicate the originating user's identity to the extension in a way the extension would verify.

The pool admin faces an impossible choice: allowlist the router (every user bypasses the allowlist) or do not allowlist the router (all router-mediated swaps revert). No configuration simultaneously enforces per-user restrictions and allows router-mediated swaps.

## Impact Explanation
The swap allowlist is the sole access-control mechanism preventing unauthorized swaps in restricted pools (e.g., KYC-gated, institutional-only). Its complete bypass by any unprivileged user constitutes broken core pool functionality and unauthorized access to pool liquidity. LPs who deposited into a restricted pool lose the protection they relied upon; any excluded user can drain liquidity at oracle-quoted prices, constituting direct loss of the protection guarantee and potential fund-impacting unauthorized access to LP assets.

## Likelihood Explanation
- `MetricOmmSimpleRouter` is a public, permissionless contract callable by any EOA or contract.
- Pool admins who want router-mediated swaps to function must allowlist the router — this is the expected operational configuration.
- No special knowledge, privileged access, or unusual conditions are required; any user aware of the allowlist restriction can exploit it by simply using the router instead of calling the pool directly.
- The bypass is unconditional and repeatable once the router is allowlisted.

## Recommendation
The extension must verify the originating user, not the direct pool caller. The cleanest fix is to have the router encode `msg.sender` (the real user) into `extensionData`, and have `SwapAllowlistExtension.beforeSwap` decode and check that identity when `sender` is a recognized router address. Alternatively, add an `originSender` field to the `beforeSwap` hook arguments so the pool can forward it explicitly. Using `tx.origin` as a fallback is fragile and not recommended for production.

## Proof of Concept
```
Setup:
  1. Deploy MetricOmmPool with SwapAllowlistExtension in BEFORE_SWAP_ORDER.
  2. Pool admin calls setAllowedToSwap(pool, router, true)   // allowlist the router
  3. Pool admin does NOT call setAllowedToSwap(pool, alice, true)

Attack:
  4. Alice (not allowlisted) calls:
       router.exactInputSingle({
           pool:        pool,
           recipient:   alice,
           zeroForOne:  true,
           amountIn:    X,
           ...
       })

  5. Router calls pool.swap(alice_recipient, true, X, ..., extensionData)
     → pool passes msg.sender = router as `sender` to _beforeSwap
     → SwapAllowlistExtension checks allowedSwapper[pool][router] → true
     → swap proceeds; Alice receives output tokens

  6. Assert: alice's swap succeeded despite not being in the allowlist.
     Direct call pool.swap(...) from alice would revert with NotAllowedToSwap.
```
