Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual user — swap allowlist fully bypassed via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` of the `pool.swap()` call. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the end user. The extension therefore checks whether the **router** is allowlisted, not the actual trader. Any pool admin who allowlists the router to support router-based swaps simultaneously grants every unprivileged user the ability to bypass the per-user allowlist gate.

## Finding Description

**Call path:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle(params)`.
2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — at this point `msg.sender` inside the pool is the router address.
3. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, passing the router address as `sender`.
4. `ExtensionCalling._beforeSwap` encodes `sender = router_address` and dispatches to the configured extension.
5. `SwapAllowlistExtension.beforeSwap(address sender, ...)` executes the check:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct key) and `sender` is the **router address**, not the end user. The mapping lookup is `allowedSwapper[pool][router]`.

**Root cause:** The pool passes its own `msg.sender` (the immediate caller) as `sender` to the extension. The extension was written assuming `sender` is the economic actor, but through the router it is always the router contract.

**Why existing guards fail:** `BaseMetricExtension.onlyPool` correctly restricts who can call the extension (only registered pools), but does nothing to validate that `sender` represents the actual end user. There is no mechanism in the extension or the pool to forward the original `tx.origin` or the router's `msg.sender` to the extension.

**Exploit flow:**
- Pool admin deploys a curated pool with `SwapAllowlistExtension` and allowlists specific user addresses (e.g., KYC'd wallets).
- To support router-based swaps for those users, the admin also calls `setAllowedToSwap(pool, router, true)`.
- Any unprivileged user (not individually allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle` targeting that pool.
- The extension sees `sender = router`, finds `allowedSwapper[pool][router] = true`, and allows the swap.
- The allowlist is completely bypassed.

**Exact wrong value:** `allowedSwapper[pool][sender]` where `sender` is the router address instead of the actual end-user address.

## Impact Explanation
The swap allowlist is the sole on-chain mechanism for curated pools to restrict trading to authorized participants. When bypassed, any unprivileged user can trade against the pool's LP positions, enabling unauthorized value extraction, adverse selection against LPs, and violation of the pool's intended access policy. This constitutes a broken core pool functionality and an admin-boundary break where an unprivileged path circumvents a configured protection, with direct fund-impacting consequences for LPs in curated pools.

## Likelihood Explanation
The bypass is reachable by any unprivileged user with a single public call to `MetricOmmSimpleRouter`. The only precondition is that the pool admin has allowlisted the router — a natural and expected action for any pool admin who wants to support standard periphery-based swaps for their allowlisted users. The router is a public, permissionless contract. The exploit is repeatable every block at zero cost beyond gas.

## Recommendation
The extension must check the actual end-user identity, not the immediate pool caller. Two approaches:

1. **Pass original caller through extensionData:** The router encodes `msg.sender` into `extensionData`; the extension decodes and verifies it. This requires a protocol-level convention.
2. **Check `recipient` instead of `sender`:** If the pool's `recipient` parameter reliably identifies the economic beneficiary, the extension can gate on `recipient`. However, `recipient` can be set to any address by the router caller, so this is only safe if the pool enforces `recipient == sender`.
3. **Preferred fix:** Add a `originalCaller` field to the swap hook arguments at the pool level, populated from `tx.origin` or a router-forwarded context, so extensions can always gate on the true end user.

## Proof of Concept

```solidity
// Foundry test sketch
function test_swapAllowlistBypassViaRouter() public {
    // Setup: pool with SwapAllowlistExtension, router allowlisted
    swapExtension.setAllowedToSwap(address(pool), address(router), true);
    // user1 is NOT individually allowlisted

    // user1 swaps directly → should revert
    vm.prank(user1);
    vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
    pool.swap(user1, false, 1000, type(uint128).max, "", "");

    // user1 swaps via router → succeeds (bypass)
    vm.prank(user1);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        recipient: user1,
        tokenIn: address(token1),
        zeroForOne: false,
        amountIn: 1000,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp + 1,
        extensionData: ""
    }));
    // swap succeeds — allowlist bypassed
}
```
