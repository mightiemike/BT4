### Title
Silent Narrowing Cast in `Into<u16> for ShardId` Corrupts `CongestionInfoV1::allowed_shard` at Shard-Layout Upgrade Boundary — (`core/primitives-core/src/types.rs`)

---

### Summary

`ShardId` is a `u64` newtype. `CongestionInfoV1::allowed_shard` is a Borsh-serialized `u16` field embedded in every chunk header. The `Into<u16> for ShardId` implementation silently truncates the inner `u64` to `u16` with no bounds check. `CongestionInfo::finalize_allowed_shard` calls `.into()` on a `ShardId` to write this field. When a shard-layout upgrade assigns any shard an ID ≥ 65536, the stored `allowed_shard` is a silently wrong value, causing the congestion-control "allowed shard" invariant to be broken across the entire network.

---

### Finding Description

**Root cause — `Into<u16> for ShardId`:** [1](#0-0) 

```rust
impl Into<u16> for ShardId {
    fn into(self) -> u16 {
        self.0 as u16   // silent truncation, no bounds check
    }
}
```

`ShardId` is explicitly declared as a `u64` wrapper: [2](#0-1) 

The protocol documentation for `ShardId` states it "may be an arbitrary number — it does not need to be a number in the range 0..NUM_SHARDS": [3](#0-2) 

**Sink — `CongestionInfoV1::allowed_shard` is a `u16` Borsh field in the chunk header:** [4](#0-3) 

**Call site — `finalize_allowed_shard` writes the truncated value:** [5](#0-4) 

`allowed_shard` is a `ShardId` returned by `get_new_allowed_shard`, then converted to `u16` via `.into()` and stored. No range assertion exists anywhere in this path.

**Read site — the stored `u16` is widened back to `ShardId` for comparison:** [6](#0-5) 

```rust
if sender_shard == ShardId::from(self.info.allowed_shard()) {
```

`ShardId::from(u16)` widens losslessly. So the round-trip is:

```
ShardId(X)  →  (X as u16)  →  ShardId(X & 0xFFFF)
```

If `X ≥ 65536`, the stored and compared shard ID is permanently wrong.

---

### Impact Explanation

`CongestionInfoV1` is Borsh-serialized into every chunk header and is part of the consensus-critical protocol schema: [7](#0-6) 

When the network is fully congested, `outgoing_gas_limit` returns `Gas::ZERO` for every shard **except** the one whose ID matches `allowed_shard`. If `allowed_shard` is truncated:

1. The **intended** shard (with ID ≥ 65536) receives `Gas::ZERO` — it cannot forward any receipts.
2. A **different** shard (whose ID happens to equal `X & 0xFFFF`) is incorrectly granted the forwarding slot.

This breaks the liveness guarantee of the congestion-control protocol: the shard that is supposed to make progress is silently blocked, while an unintended shard is unblocked. Because all nodes run the same truncating code, the divergence is not a consensus split — it is a uniform, silent semantic corruption of the congestion-control invariant that is baked into every chunk header produced after the shard-layout upgrade.

**Severity: High** — protocol-level liveness failure under full congestion after any shard-layout upgrade that assigns IDs ≥ 65536.

---

### Likelihood Explanation

`ShardId` is explicitly designed to be an arbitrary `u64` (not a sequential index). The shard-layout upgrade path is a normal, public protocol-upgrade boundary. The current shard IDs are small (< 10), so the bug is latent today. However:

- The `Into<u32> for ShardId` implementation has the same silent-truncation pattern (used in `ShardUId` DB keys), confirming this is a systemic pattern, not an isolated oversight. [8](#0-7) 

- No existing gate, assertion, or schema validator prevents a future shard layout from assigning IDs ≥ 65536.
- The `CongestionInfoV1` struct is versioned and immutable by design; a new version would be required to fix the field width, making this a migration-boundary issue.

---

### Recommendation

1. **Add a bounds check in `finalize_allowed_shard`** before calling `.into()`:

```rust
pub fn finalize_allowed_shard(
    &mut self,
    own_shard: ShardId,
    all_shards: &[ShardId],
    congestion_seed: u64,
) {
    let allowed_shard = Self::get_new_allowed_shard(own_shard, all_shards, congestion_seed);
    let allowed_shard_u16: u64 = allowed_shard.into();
    assert!(
        allowed_shard_u16 <= u16::MAX as u64,
        "ShardId {} exceeds u16::MAX; CongestionInfoV1::allowed_shard cannot represent it",
        allowed_shard_u16
    );
    self.set_allowed_shard(allowed_shard_u16 as u16);
}
```

2. **Widen `CongestionInfoV1::allowed_shard` to `u64`** in a new `CongestionInfoV2` before any shard-layout upgrade that could assign IDs ≥ 65536.

3. **Remove or gate the `Into<u16> for ShardId` and `Into<u32> for ShardId` implementations** — both perform silent narrowing casts that violate the `ShardId`-as-arbitrary-u64 contract.

---

### Proof of Concept

```
Precondition: a shard-layout upgrade assigns ShardId(65537) to a shard.

1. Network reaches full congestion on that shard.
2. `finalize_allowed_shard` selects ShardId(65537) as the allowed shard.
3. `self.set_allowed_shard(ShardId(65537).into())` stores `(65537u64 as u16) = 1u16`.
4. The chunk header is Borsh-serialized with `allowed_shard = 1`.
5. `outgoing_gas_limit(ShardId(65537))` evaluates:
       ShardId(65537) == ShardId::from(1u16)  →  ShardId(65537) == ShardId(1)  →  false
   → returns Gas::ZERO  (intended shard is blocked).
6. `outgoing_gas_limit(ShardId(1))` evaluates:
       ShardId(1) == ShardId(1)  →  true
   → returns `allowed_shard_outgoing_gas`  (wrong shard is unblocked).

Expected: ShardId(65537) can forward receipts.
Actual:   ShardId(65537) is silently blocked; ShardId(1) is incorrectly unblocked.
```
