### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Producing a Different Transaction Hash Than the RPC Ingestion Path — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf converter for `ValidResourceBounds` classifies a transaction as `ValidResourceBounds::L1Gas` whenever `l2_gas` and `l1_data_gas` are both zero, even when the original transaction was `ValidResourceBounds::AllResources`. The transaction hash function `get_tip_resource_bounds_hash` produces structurally different hash preimages for these two variants (2 vs 3 resource elements). A V3 transaction with zero `l2_gas` and `l1_data_gas` is accepted by the gateway. The RPC ingestion path always computes the hash under `AllResources` (H1). When the same block is received by a syncing node via P2P protobuf, the transaction is deserialized as `L1Gas` and the hash recomputed as H2 ≠ H1.

---

### Finding Description

**Step 1 — RPC ingestion always forces `AllResources`.**

`RpcInvokeTransactionV3` carries `AllResourceBounds` (not `ValidResourceBounds`). The conversion to `InternalRpcInvokeTransactionV3` preserves this, and the hash is computed via `get_invoke_transaction_v3_hash` where `resource_bounds()` returns `ValidResourceBounds::AllResources(self.resource_bounds)`. [1](#0-0) 

The hash calculation for `AllResources` with zero `l2_gas` and `l1_data_gas` includes **three** resource elements: L1_GAS, L2_GAS (zero), and L1_DATA_GAS (zero): [2](#0-1) 

**Step 2 — Protobuf deserialization silently downgrades to `L1Gas`.**

The protobuf converter for `ValidResourceBounds` applies a value-based heuristic: if `l1_data_gas` and `l2_gas` are both zero (regardless of whether they were explicitly present), it returns `ValidResourceBounds::L1Gas`: [3](#0-2) 

**Step 3 — `get_tip_resource_bounds_hash` produces a different preimage for `L1Gas`.**

For `ValidResourceBounds::L1Gas`, the hash chain contains only **two** resource elements (L1_GAS and L2_GAS); the L1_DATA_GAS element is absent: [4](#0-3) 

This is structurally different from the `AllResources` path, which always appends the L1_DATA_GAS element. The resulting Poseidon hash is therefore different.

**Step 4 — The JSON/storage serialization path does NOT have this bug.**

The JSON `Serialize` impl for `ValidResourceBounds::AllResources` always emits the `L1_DATA_GAS` key, and the `Deserialize` impl promotes to `AllResources` whenever that key is present (even if zero). The protobuf path uses a value-only check and loses the type information: [5](#0-4) 

**Step 5 — A V3 transaction with zero `l2_gas` and `l1_data_gas` is valid.**

The stateless validator accepts a transaction with only `l1_gas` set: [6](#0-5) 

---

### Impact Explanation

A V3 invoke transaction with `l2_gas = 0` and `l1_data_gas = 0` is submitted via RPC. The sequencer computes hash **H1** (under `AllResources`, 3-element preimage) and includes it in the block. When a peer node receives the block via P2P protobuf sync, it deserializes the transaction as `ValidResourceBounds::L1Gas` and recomputes hash **H2** (2-element preimage). H1 ≠ H2.

Consequences:
- Any call to `validate_transaction_hash` on the syncing node's deserialized `Transaction` will fail, causing valid blocks to be rejected.
- If the syncing node uses H2 for the transaction commitment (part of the block hash), the block hash diverges from the proposer's, breaking consensus.
- RPC endpoints (`starknet_getTransactionByHash`) on syncing nodes return `L1Gas` resource bounds for a transaction whose stored hash was computed under `AllResources`, producing an authoritative-looking but incorrect response.

This matches: **Critical — Wrong state/receipt/revert result from blockifier/execution logic for accepted input** and **High — Transaction conversion or signature/hash logic binds the wrong hash or executable payload**.

---

### Likelihood Explanation

Any user can submit a V3 transaction with `l2_gas = 0` and `l1_data_gas = 0` (only `l1_gas` set). This is a valid, accepted transaction type. No special privilege is required. The divergence is triggered automatically whenever such a transaction is propagated via P2P protobuf sync.

---

### Recommendation

Replace the value-based heuristic in the protobuf `ValidResourceBounds` converter with an explicit presence-based check that mirrors the JSON deserializer: if `l1_data_gas` is present in the protobuf message (even if zero), produce `AllResources`; only produce `L1Gas` when `l1_data_gas` is absent (`None`). The existing `TODO` comment acknowledges this:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
```

The fix is to branch on `value.l1_data_gas.is_none()` rather than on the zero-value of the deserialized bounds:

```rust
Ok(match value.l1_data_gas {
    None => ValidResourceBounds::L1Gas(l1_gas),
    Some(raw) => {
        let l1_data_gas: ResourceBounds = raw.try_into()?;
        ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
    }
})
```

This aligns the protobuf path with the JSON path and ensures that `AllResources` with zero values round-trips to the same variant and the same hash.

---

### Proof of Concept

1. Submit a V3 invoke transaction with `l1_gas = {max_amount: 1000, max_price_per_unit: 1}`, `l2_gas = {0, 0}`, `l1_data_gas = {0, 0}` via the RPC gateway.
2. The sequencer computes hash H1 = `poseidon(tip, concat(L1_GAS, 1000, 1), concat(L2_GAS, 0, 0), concat(L1_DATA_GAS, 0, 0))` (3 resource elements).
3. The transaction is included in a block and propagated via P2P protobuf.
4. A syncing node deserializes the protobuf: `l1_data_gas.is_zero() && l2_gas.is_zero()` → `ValidResourceBounds::L1Gas`.
5. The syncing node computes hash H2 = `poseidon(tip, concat(L1_GAS, 1000, 1), concat(L2_GAS, 0, 0))` (2 resource elements).
6. H1 ≠ H2. `validate_transaction_hash` on the syncing node returns `false` for the stored hash H1, causing the block to be rejected or the transaction commitment to diverge. [7](#0-6) [8](#0-7)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
