Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on Caller-Supplied `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of the `addLiquidity` call) and instead checks the caller-supplied `owner` address against the allowlist. Because `owner` is a free argument to `MetricOmmPool.addLiquidity`, any unpermissioned address can pass the guard by naming any allowlisted address as `owner`, completely nullifying the admin-configured deposit restriction.

## Finding Description
`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` and passes `msg.sender` as `sender` alongside `owner` to the extension hook:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` encodes both actors and forwards them to the extension:

```solidity
// ExtensionCalling.sol L95-98
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

`DepositAllowlistExtension.beforeAddLiquidity` receives `(sender, owner, …)` but the first parameter is unnamed (discarded) and the check is performed only on `owner`:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The asymmetry is confirmed by `SwapAllowlistExtension.beforeSwap`, which correctly names and gates on `sender`:

```solidity
// SwapAllowlistExtension.sol L31-37
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
```

**Exploit path:**
1. Pool is deployed with `DepositAllowlistExtension`; admin allowlists only `alice`.
2. `bob` (not allowlisted) reads `allowedDepositor[pool][alice] == true` on-chain.
3. `bob` calls `pool.addLiquidity(alice /*owner*/, salt, deltas, callbackData, extensionData)`.
4. The extension evaluates `allowedDepositor[pool][alice]` → `true`; no revert.
5. `bob`'s tokens are pulled via the swap callback; LP shares are credited to `alice`.
6. The allowlist guard is fully bypassed.

No existing guard prevents this: `removeLiquidity` does enforce `msg.sender == owner`, but `addLiquidity` has no such check, making the `owner` parameter freely settable by any caller.

## Impact Explanation
The pool admin deploys `DepositAllowlistExtension` to enforce KYC/regulatory compliance or pool privacy. The bypass allows any unpermissioned address to inject liquidity into the restricted pool, breaking the admin-configured invariant. Concretely, pool state (`binTotals`, `_binTotalShares`, `_positionBinShares`, `curPosInBin`) is modified by actors the admin explicitly excluded. This constitutes an admin-boundary break: an unprivileged path bypasses a factory/pool role check that the admin intended to be enforced.

## Likelihood Explanation
Exploitation requires only a single `addLiquidity` call with any allowlisted address as `owner`. No special role, flash loan, or multi-step setup is needed. The allowlist mappings (`allowedDepositor`) are public and readable by any on-chain observer. The attack is trivially repeatable by any address.

## Recommendation
Change `beforeAddLiquidity` to gate on the first parameter (`sender`) instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
1. Deploy pool with `DepositAllowlistExtension`; call `setAllowedToDeposit(pool, alice, true)`.
2. From `bob` (not allowlisted): call `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
3. Observe: extension checks `allowedDepositor[pool][alice]` → `true`, no revert.
4. `bob`'s tokens are transferred via callback; LP shares credited to `alice`.
