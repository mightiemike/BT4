### Title
Zero-Value Approval Reverts Break `wrapVaultAsset` for Tokens Like BNB — (`core/contracts/ContractOwner.sol`)

### Summary

`ContractOwner.wrapVaultAsset()` unconditionally calls `assetToken.approve(tokenAddr, 0)` before setting the real allowance. For tokens that revert on zero-value approvals (e.g., BNB on BSC), this line always reverts, permanently breaking the vault-wrapping deposit path for those tokens. The function is permissionless (`external`, no `onlyOwner`), so any caller can attempt it and observe the failure.

### Finding Description

`wrapVaultAsset` is designed to pull an underlying asset token from a user's `DirectDepositV1` contract into `ContractOwner`, approve the ERC4626 vault contract to spend it, deposit into the vault, and return the wrapped shares to the `DirectDepositV1` address. The critical sequence is:

```solidity
// ContractOwner.sol lines 526–532
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(assetTokenAddr));
IERC20Base assetToken = IERC20Base(assetTokenAddr);
assetToken.approve(tokenAddr, 0);          // ← reverts for BNB-like tokens
assetToken.approve(tokenAddr, assetBalance);
IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1);
``` [1](#0-0) 

The `approve(tokenAddr, 0)` call at line 530 is intended to reset any residual allowance before granting a fresh one. However, tokens such as BNB that implement the [Revert on Zero Value Approvals](https://github.com/d-xo/weird-erc20?tab=readme-ov-file#revert-on-zero-value-approvals) behaviour will revert here unconditionally, causing the entire transaction to revert. Because the `withdraw()` call at line 526 is part of the same transaction, it also reverts, so no tokens are permanently lost in a single call — but the function can **never succeed** for these tokens, permanently blocking the vault-wrapping path.

The function signature carries no access-control modifier:

```solidity
function wrapVaultAsset(bytes32 subaccount, uint32 productId) external {
``` [2](#0-1) 

Any caller — including the affected user — can invoke it and will always receive a revert.

The `ERC20Helper` library used elsewhere in the codebase wraps transfers safely but provides no safe-approve helper, and no equivalent of a `safeResetAndApprove` guard is applied here. [3](#0-2) 

### Impact Explanation

For any ERC4626 vault product whose underlying `asset()` is a token that reverts on zero-value approvals:

- `wrapVaultAsset` is permanently and completely broken.
- User assets that arrive at the `DirectDepositV1` contract as the raw underlying token cannot be wrapped into vault shares and credited to the subaccount via this path.
- The deposit flow for those vault products is entirely non-functional, constituting a permanent denial of service for the affected token/product pairs.

### Likelihood Explanation

The protocol targets BSC deployment (where BNB is the canonical native/wrapped token and a common collateral asset). BNB is the canonical example of a token that reverts on zero-value approvals. Any vault product backed by BNB or a similarly behaving token triggers this path. The function is permissionless, so no special role is needed to observe or confirm the failure.

### Recommendation

Replace the hard `approve(tokenAddr, 0)` call with a try/catch pattern that silently ignores a revert on the zero-value approval, mirroring the `_safeResetAndApprove` approach described in the external report:

```solidity
// try to reset; ignore revert for tokens that disallow zero-value approvals
(bool ok, ) = address(assetToken).call(
    abi.encodeWithSelector(assetToken.approve.selector, tokenAddr, 0)
);
// ok is intentionally unused — failure here is expected for some tokens
assetToken.approve(tokenAddr, assetBalance);
```

Alternatively, check the current allowance first and skip the reset if it is already zero.

### Proof of Concept

1. Deploy the protocol on BSC with a vault product whose `asset()` is BNB (or any token that reverts on `approve(spender, 0)`).
2. Send some BNB-equivalent tokens to the `DirectDepositV1` address for any subaccount.
3. Call `ContractOwner.wrapVaultAsset(subaccount, productId)` from any EOA.
4. The call reverts at `assetToken.approve(tokenAddr, 0)` (line 530) because BNB does not permit zero-value approvals.
5. The vault-wrapping deposit is permanently blocked; the user's assets remain in the `DirectDepositV1` contract and cannot be credited to their subaccount through this path. [4](#0-3)

library ERC20Helper {
    function safeTransfer(
        IERC20Base self,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
        );
        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
}
