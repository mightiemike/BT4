Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates on the router address instead of the actual user, allowing any unprivileged caller to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter, which is `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` resolves to the router contract address, not the actual user. If the pool admin allowlists the router to enable router-mediated swaps for their allowlisted users, every unprivileged address on the network can bypass the individual allowlist by calling any router entry-point, permanently breaking the pool's access-control invariant.

## Finding Description

`MetricOmmPool.swap` passes its own `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` forwards this value verbatim as `sender` to the extension. `SwapAllowlistExtension.beforeSwap` then checks that `sender` against the per-pool allowlist: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` of that call. The actual user's address is stored only in transient storage for the payment callback — it is never passed to `pool.swap()`: [3](#0-2) 

So the extension sees `sender = address(router)`, not the actual user. The allowlist check becomes `allowedSwapper[pool][router]`. Once the router is allowlisted — the natural configuration for a pool that wants to support the standard periphery — every address on the network can swap by calling the router, regardless of whether they are individually allowlisted.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates on `owner` (the actual beneficiary, the second parameter), not `sender` (the intermediary, the first parameter): [4](#0-3) 

The swap allowlist is architecturally inconsistent with the deposit allowlist and cannot enforce per-user restrictions when the router is in the call path. The `isAllowedToSwap` view function also returns `true` for the router address, giving the admin no on-chain signal that allowlisting the router opens the pool to all users. [5](#0-4) 

## Impact Explanation

Any address can swap in a pool the admin intended to restrict to a specific set of counterparties (e.g., KYC-verified addresses, whitelisted protocols, or institutional partners). The bypass is unconditional once the router is allowlisted: the attacker supplies any valid `amountIn`, the pool executes the swap at the oracle price, and the attacker receives output tokens. The pool's access-control invariant — that only allowlisted addresses may trade — is permanently broken for all router-mediated paths. This constitutes an admin-boundary break with direct fund-flow consequences: unauthorized parties extract value from a restricted pool. This matches the "Admin-boundary break" allowed impact criterion.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface. A pool admin who wants to allow router-mediated swaps for their allowlisted users will naturally add the router to `allowedSwapper`. The admin has no on-chain signal that doing so opens the pool to all users. The trigger requires only a standard router call with no special permissions, making this reachable by any unprivileged user with no preconditions beyond the router being allowlisted — a configuration the admin is incentivized to set up.

## Recommendation

Gate `SwapAllowlistExtension.beforeSwap` on the economically relevant actor — the address that pays tokens in — rather than the immediate pool caller. Two concrete approaches:

1. **Decode the payer from `extensionData`**: Require the router to encode `msg.sender` (the actual user) into `extensionData` and have the extension decode and check that address. This is consistent with how `extensionData` is already forwarded through the full hook chain.

2. **Mirror the deposit pattern**: Introduce a `payer` field analogous to `owner` in the liquidity hooks, passed explicitly by the pool from the callback context, so the extension always sees the address that funds the swap.

Until fixed, document that allowlisting the router in `SwapAllowlistExtension` is equivalent to calling `setAllowAllSwappers(pool, true)`.

## Proof of Concept

```
Setup:
  1. Pool admin deploys MetricOmmPool with SwapAllowlistExtension as beforeSwap hook.
  2. Admin calls setAllowedToSwap(pool, alice, true)       // alice is the only allowed swapper
  3. Admin calls setAllowedToSwap(pool, router, true)      // enable router-mediated swaps for alice

Attack:
  4. Bob (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({
         pool:             <restricted pool>,
         tokenIn:          token1,
         zeroForOne:       false,
         amountIn:         X,
         amountOutMinimum: 0,
         recipient:        bob,
         ...
       })

  5. Router calls pool.swap(bob, false, X, limit, "", "")
     → pool calls _beforeSwap(msg.sender=router, ...)
     → SwapAllowlistExtension.beforeSwap(sender=router, ...)
     → allowedSwapper[pool][router] == true  → check passes
     → swap executes at oracle price
     → Bob receives token0 output

Result: Bob, who is not individually allowlisted, successfully swaps in a pool
        the admin intended to restrict to alice only.
        The allowlist guard is fully bypassed via the public router.
```
