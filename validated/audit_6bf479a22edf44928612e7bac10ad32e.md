### Title
Deprecated `verify_transaction_inclusion` Remains Callable and Explicitly Offloads Proof-Forgery Defense to Uncontrolled Third-Party Contracts — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still reachable by any unprivileged NEAR caller despite being deprecated. The function explicitly states that validation of whether `tx_id` is a real transaction (not an internal Merkle node) is "performed at a higher level of verification." This design pushes a critical security invariant onto downstream third-party NEAR contracts that are outside the light client's control — an exact structural analog to the DSR locking pattern, where the protocol design forces users through uncontrolled external code to obtain a security property the core contract should enforce itself.

---

### Finding Description

`verify_transaction_inclusion` carries an explicit in-code warning:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash. We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification." [1](#0-0) 

Despite this, the function is not removed, not access-controlled, and not gated behind any on-chain guard. It is callable by any NEAR account: [2](#0-1) 

`verify_transaction_inclusion_v2` was introduced specifically to close this gap by requiring a coinbase Merkle proof, which prevents the 64-byte internal-node forgery. But v2 does not replace v1 — it calls v1 internally after its own coinbase check: [3](#0-2) 

The Merkle verification itself (`compute_root_from_merkle_proof`) is mathematically correct but makes no distinction between a leaf (real transaction) and an internal node: [4](#0-3) 

The only check on `merkle_proof` in v1 is that it is non-empty: [5](#0-4) 

An attacker who supplies an internal Merkle node as `tx_id` with a valid proof path to the root will receive `true` from v1, because the Merkle arithmetic is satisfied.

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` (v1) and acts on its `true` result — releasing funds, minting tokens, unlocking state — can be deceived into accepting a proof for a Bitcoin transaction that does not exist. The corrupted proof result is the exact state value that downstream contracts rely on. Because the light client explicitly says "validation is at a higher level," every third-party integration is individually responsible for implementing the coinbase-proof check — a check that is easy to omit, misunderstand, or implement incorrectly. This is the same systemic risk as DSR: the protocol design forces security-critical logic into uncontrolled external code.

---

### Likelihood Explanation

The 64-byte Merkle proof forgery is a well-documented Bitcoin attack (referenced in the code itself via the BitmexResearch link). The entry path requires only: a real block hash present in the light client's mainchain, knowledge of that block's Merkle tree structure (publicly available from any Bitcoin node), and a direct NEAR call to `verify_transaction_inclusion`. No privileged role, no leaked key, no social engineering is needed. Any downstream contract that has not independently implemented the coinbase-proof mitigation is exploitable.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public API entirely, or gate it with a hard `require!(false, "use verify_transaction_inclusion_v2")` panic so it cannot be called at runtime. The security invariant — that `tx_id` must be a leaf, not an internal node — must be enforced inside the light client itself, not delegated to downstream contracts. Keeping a callable but insecure verification path in the contract, even with a deprecation annotation, replicates the DSR anti-pattern: it creates a class of third-party integrations that are silently insecure because they rely on the protocol's own public API.

---

### Proof of Concept

1. A real Bitcoin block `B` at height `H` has been submitted to the light client and is part of the mainchain (`mainchain_header_to_height` contains `B`).
2. The attacker retrieves block `B`'s full transaction list from a public Bitcoin node and identifies an internal Merkle node `N` at depth `d` (e.g., the hash of two real transactions). `N` is 32 bytes and is a valid `H256`.
3. The attacker constructs a Merkle proof path `P` of length `d` from `N` upward to the block's `merkle_root`. This proof is mathematically valid: `compute_root_from_merkle_proof(N, index, P) == merkle_root`.
4. The attacker calls:
   ```
   verify_transaction_inclusion({
     tx_id: N,                  // internal node, not a real txid
     tx_block_blockhash: B,     // real mainchain block
     tx_index: <index of N>,
     merkle_proof: P,           // valid path from N to root
     confirmations: 1,
   })
   ```
5. The function looks up `B` in `mainchain_header_to_height` — succeeds. It checks confirmations — passes. It calls `compute_root_from_merkle_proof(N, index, P)` — returns `merkle_root`. It compares to `header.block_header.merkle_root` — equal. **Returns `true`.**
6. A downstream NEAR contract that called v1 and received `true` now believes the non-existent transaction `N` was confirmed in block `B`, and acts accordingly (e.g., releases bridged funds). [6](#0-5) [4](#0-3)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
