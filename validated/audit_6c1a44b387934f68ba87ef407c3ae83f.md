### Title
Incorrect Difficulty Ancestor Lookup for Fork Blocks in Dogecoin Per-Block Retarget — (File: `contract/src/dogecoin.rs`)

### Summary

In the Dogecoin build of the BTC light client, `get_next_work_required` fetches the difficulty-interval boundary block by mainchain height rather than by walking the fork's own ancestor chain. For the post-145000 per-block retarget protocol this means the `first_block_time` used in every fork-block difficulty calculation is taken from the **mainchain** block at that height, not from the fork's actual ancestor. An attacker-controlled relayer can exploit the resulting timestamp delta to cause the contract to accept fork blocks whose `bits` field encodes a lower difficulty than the protocol requires, enabling a low-work chain reorganization.

### Finding Description

`get_next_work_required` in `contract/src/dogecoin.rs` computes the expected difficulty for a new block. For the new-protocol path (height ≥ 145 000, per-block adjustment), `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 0` and `height_first = prev_block_header.block_height`. The boundary block is then fetched with:

```rust
// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [2](#0-1) 

This always resolves through `mainchain_height_to_header`, which maps heights to **mainchain** block hashes. When a fork block is being submitted, `prev_block_header` is the fork's own previous block (fetched via `get_prev_header` from `headers_pool`), but `get_header_by_height(prev_block_header.block_height)` returns the **mainchain** block at that height — a different block with a potentially different timestamp.

The difficulty calculation then becomes:

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
``` [3](#0-2) 

where `prev_block_header.block_header.time` is the fork ancestor's timestamp and `first_block_time` is the **mainchain** block's timestamp. The correct value for the per-block retarget would be `fork_prev_time − fork_prev_time = 0` (same block), but the actual value is `fork_prev_time − mainchain_block_time`, which can be arbitrarily large if the attacker controls the fork block's timestamp.

The `expected_bits` produced by this calculation is then compared to `block_header.bits`:

```rust
require!(
    expected_bits == block_header.bits,
    format!(
        "Error: Incorrect target. Expected bits: {:?}, Actual bits: {:?}",
        expected_bits, block_header.bits
    )
);
``` [4](#0-3) 

Because the contract computes `expected_bits` from the wrong timestamp, an attacker who knows the mainchain block's timestamp can predict exactly what `expected_bits` will be and submit a fork block whose `bits` field matches that (lower-difficulty) value. The `require!` passes silently — there is no panic, no fault, no rejection — mirroring the fault-tolerant `rebalance` call in the original report.

The TODO comment in the code itself acknowledges the unresolved correctness question: [5](#0-4) 

### Impact Explanation

A successful exploit corrupts the canonical chain state stored in the contract. Specifically:

- The attacker submits a fork chain whose blocks carry a lower `bits` value (higher target, less work per block) than the protocol requires.
- Because `expected_bits` is computed from the wrong timestamp, the difficulty check passes.
- If the fork's cumulative `chain_work` (computed from the attacker-chosen `bits`) exceeds the mainchain's `chain_work`, `reorg_chain` is triggered and the contract's `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height` are rewritten to point to the attacker's fork. [6](#0-5) 

Any downstream NEAR contract calling `verify_transaction_inclusion_v2` after the reorg will verify proofs against the attacker's fork, not the real Dogecoin mainchain, producing false-positive SPV proofs for transactions that were never confirmed on the real chain.

### Likelihood Explanation

The attacker entry point is `submit_blocks`, which is callable by any account that satisfies the `#[trusted_relayer]` staking requirement or holds `Role::UnrestrictedSubmitBlocks`. [7](#0-6) 

The attack requires:
1. Submitting a fork that diverges from the mainchain at any height ≥ 145 000.
2. Setting the fork's prev-block timestamp to a value significantly later than the mainchain block at the same height (within the MTP and future-time constraints).
3. Computing the `expected_bits` the contract will derive and submitting a block with that `bits` value.

All three steps are fully attacker-controlled through the public `submit_blocks` API. No privileged key or social engineering is required.

### Recommendation

Replace the `get_header_by_height` call with an ancestor walk that follows `prev_block_hash` links through `headers_pool` until the block at `height_first` is reached, regardless of whether it is on the mainchain or on the fork. This is exactly what the TODO comment anticipates. The corrected logic should mirror how `get_prev_header` is used elsewhere in the difficulty-calculation loops (e.g., in `zcash.rs` and the `pow_allow_min_difficulty_blocks` path in `dogecoin.rs`). [8](#0-7) 

### Proof of Concept

1. Contract is deployed in Dogecoin mode with mainchain at height 200 000. Mainchain block at height 199 999 has timestamp `T_main = 1_700_000_000`.
2. Attacker submits a fork block at height 199 999 whose timestamp is `T_fork = 1_700_010_000` (10 000 seconds later, within the 2-hour future-time limit relative to MTP).
3. Attacker submits a fork block at height 200 000 with `prev_block_hash` pointing to the fork block at 199 999.
4. Contract calls `get_next_work_required`. `height_first = 199_999`. `get_header_by_height(199_999)` returns the **mainchain** block with `T_main = 1_700_000_000`.
5. `modulated_timespan = T_fork − T_main = 10_000`. After Digishield damping: `modulated_timespan = retarget_timespan + (10_000 − retarget_timespan) / 8`. For Dogecoin mainnet `retarget_timespan = 60`, this yields a large positive timespan, producing a significantly higher target (lower difficulty) `expected_bits`.
6. Attacker submits the height-200 000 fork block with `bits = expected_bits` (the lower-difficulty value). The `require!` passes.
7. Attacker repeats, accumulating chainwork. Because each block requires less PoW, the fork's `chain_work` can overtake the mainchain's with far fewer real hashes, triggering `reorg_chain` and rewriting the contract's canonical chain to the attacker's fork. [9](#0-8)
