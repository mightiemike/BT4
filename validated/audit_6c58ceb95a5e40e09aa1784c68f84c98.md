Audit Report

## Title
Swap Allowlist Bypass via Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to its own `msg.sender` — the router contract — rather than the actual end-user. When a pool admin allowlists the router to permit allowlisted users to swap through it, every user, including non-allowlisted ones, can bypass the curated-pool gate by routing through `MetricOmmSimpleRouter`. There is no configuration that allows allowlisted users to use the router while blocking non-allowlisted ones.

## Finding Description

**Root cause — wrong actor bound in `beforeSwap`**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (correct key for the mapping), and `sender` is whatever the pool forwarded. The pool always forwards its own `msg.sender`: [2](#0-1) 

`ExtensionCalling._beforeSwap` passes that value straight through to the extension: [3](#0-2) 

**Router path**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly without encoding the real end-user anywhere: [4](#0-3) 

When the router is the caller, `msg.sender` to the pool is the router address. The pool therefore passes `sender = router` to the extension. The extension evaluates `allowedSwapper[pool][router]` — the real user is never checked.

**The dilemma for the pool admin**

| Router allowlisted? | Allowlisted user via router | Non-allowlisted user via router |
|---|---|---|
| No | Blocked (broken UX) | Blocked |
| Yes | Passes | **Also passes — bypass** |

There is no configuration in which the pool admin can allow allowlisted users to use the router while blocking non-allowlisted users.

**Contrast with `DepositAllowlistExtension`**

The deposit extension does not share this flaw — it ignores `sender` and checks `owner`, which is the actual position owner passed explicitly: [5](#0-4) 

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, institutional LPs, or protocol-controlled accounts) can be fully bypassed by any user routing through `MetricOmmSimpleRouter`. The attacker receives the same oracle-anchored execution as an allowlisted user, draining LP value or front-running in a pool the admin intended to gate. This breaks the core curation invariant and constitutes unauthorized access to restricted pool liquidity — a direct loss of LP principal and a broken core pool functionality impact.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary public swap entrypoint for end-users. Any pool admin who enables the router for allowlisted users simultaneously opens the gate to all users. The bypass requires no special privileges, no flash loans, and no unusual token behavior — only a standard router call. The attack is repeatable on every block.

## Recommendation

The extension must resolve the actual end-user, not the intermediary. The cleanest fix is to require the router to forward the real user: add a `swapper` field to `extensionData` that the router populates with `msg.sender`, and have `SwapAllowlistExtension.beforeSwap` decode and check that field instead of the `sender` argument when `sender` is a known router address. Alternatively, the pool could pass `tx.origin` as `sender`, though this is generally discouraged.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is the only allowed swapper
  allowedSwapper[pool][router] = true  // admin must set this for alice to use the router

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: bob, ...})

  router calls:
    pool.swap(bob, zeroForOne, amount, priceLimit, "", extensionData)
    // msg.sender to pool = router

  pool calls:
    _beforeSwap(sender=router, ...)

  extension checks:
    allowedSwapper[pool][router] == true  ✓  → swap proceeds

Result:
  bob swaps successfully in a pool he is not allowlisted for.
  The allowlist is completely bypassed.
```

A Foundry integration test can reproduce this by deploying a pool with `SwapAllowlistExtension`, allowlisting only `alice` and the router, then calling `router.exactInputSingle` from `bob` and asserting the swap succeeds rather than reverting with `NotAllowedToSwap`.
