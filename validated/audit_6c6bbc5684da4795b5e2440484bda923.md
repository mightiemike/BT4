### Title
NLP Redemption Uses Oracle Price Without Pool Solvency Check, Enabling First-Come-First-Serve Drain тАФ (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`burnNlp` in `Clearinghouse.sol` redeems NLP tokens at the sequencer-supplied oracle price without verifying that the NLP pool subaccounts remain solvent after the deduction. When the pools have a quote deficit (actual backing < oracle-implied value), early redeemers receive full oracle-price-based payouts while later redeemers drain the pools further into negative balances. This eventually causes `assertUtilization` to block all quote withdrawals for remaining users тАФ a first-come-first-serve loss distribution identical in structure to the Bancor finding.

---

### Finding Description

`burnNlp` computes the user's redemption amount as:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
``` [1](#0-0) 

The `oraclePriceX18` is a parameter supplied by the sequencer and reflects an external oracle price for NLP, not the actual per-token backing of the pool subaccounts. If the pools have suffered losses (e.g., from bad debt socialization via `socializeSubaccount`, or from trading losses credited to pool subaccounts), the oracle price can exceed the real backing.

`_validateNlpRebalance` only enforces that the sum of pool deductions equals `quoteAmount`:

```solidity
require(deltaQuoteAmount == rebalanceAmount, ERR_INVALID_NLP_REBALANCE);
``` [2](#0-1) 

It does **not** check that any individual pool subaccount remains non-negative after the deduction. `_applyNlpRebalance` then unconditionally applies the deductions:

```solidity
spotEngine.updateBalance(QUOTE_PRODUCT_ID, nlpPools[i].subaccount, nlpPoolRebalanceX18[i]);
``` [3](#0-2) 

`SpotEngine.updateBalance` permits negative balances тАФ a subaccount going negative is treated as a borrow, increasing `totalBorrowsNormalized` in the quote product state: [4](#0-3) 

After `burnNlp`, the only health checks performed are on `txn.sender` (the user burning NLP):

```solidity
require(spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0, ...);
require(getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0, ...);
``` [5](#0-4) 

There is **no** health or solvency check on the NLP pool subaccounts themselves. Their quote balances can go arbitrarily negative.

When users subsequently call `withdrawCollateral` to redeem their credited quote, `assertUtilization` is enforced:

```solidity
require(totalDeposits >= totalBorrows, ERR_MAX_UTILIZATION);
``` [6](#0-5) 

As pool subaccounts accumulate negative quote balances, `totalBorrows` for the quote product grows. Once `totalBorrows >= totalDeposits`, every subsequent `withdrawCollateral` call for quote reverts. Users who withdrew early succeed; users who withdraw later are permanently blocked.

---

### Impact Explanation

All NLP holders should be able to redeem their proportional share of pool assets. Instead, when a pool deficit exists, early redeemers receive full oracle-price-based payouts, draining the pools. Later redeemers receive the same oracle-price credit in the SpotEngine but cannot convert it to actual tokens because `assertUtilization` blocks quote withdrawals once `totalBorrows` exceeds `totalDeposits`. The loss is not distributed proportionally тАФ it falls entirely on the last users to withdraw, who may lose their entire credited quote balance.

---

### Likelihood Explanation

NLP pool subaccounts can accumulate quote deficits through normal protocol operation: bad debt socialization (`socializeSubaccount` reduces `cumulativeDepositsMultiplierX18`, effectively reducing all depositors' balances including pool subaccounts), or through trading losses credited to pool subaccounts. The oracle price for NLP is set by the sequencer and may lag behind actual pool backing. Any NLP holder can submit a `BurnNlp` transaction via `Endpoint.submitSlowModeTransaction` тАФ no special privileges are required. The trigger is realistic whenever the NLP pools are undercollateralized relative to the oracle price.

---

### Recommendation

Add a per-pool solvency check inside `_applyNlpRebalance` or `burnNlp` to ensure no pool subaccount's quote balance goes below zero after the deduction. Alternatively, compute the redemption amount based on the actual pro-rata share of pool assets rather than the oracle price, so that any existing deficit is distributed proportionally across all NLP holders at the time of redemption rather than falling entirely on the last to exit.

---

### Proof of Concept

1. NLP pools collectively hold 900 USDC in quote balances (e.g., after trading losses), but the oracle price implies 1000 USDC total backing for outstanding NLP supply.
2. User A holds 50% of NLP supply and calls `burnNlp`. The sequencer processes it with `oraclePriceX18` reflecting 1000 USDC total backing. User A receives 500 USDC in quote credit. `_applyNlpRebalance` deducts 500 USDC from pool subaccounts тАФ pools now hold 400 USDC.
3. User A calls `withdrawCollateral` for 500 USDC. `assertUtilization` passes (400 USDC deposits, 0 borrows from pools). Transfer succeeds.
4. User B holds the other 50% of NLP supply and calls `burnNlp`. Sequencer processes at same oracle price. User B receives 500 USDC in quote credit. `_applyNlpRebalance` deducts 500 USDC from pool subaccounts тАФ pools now hold -100 USDC (borrow of 100 USDC recorded in `totalBorrowsNormalized`).
5. User B calls `withdrawCollateral` for 500 USDC. `assertUtilization` checks: `totalDeposits` (remaining depositors) vs `totalBorrows` (100 USDC from pool deficit). If the deficit pushes `totalBorrows > totalDeposits`, the call reverts with `ERR_MAX_UTILIZATION`.
6. User B cannot withdraw their 500 USDC quote credit. User A received 500 USDC; User B receives 0. The 100 USDC loss falls entirely on User B rather than being shared 50/50. [7](#0-6) [8](#0-7) [6](#0-5)

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
