Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` silently discards `sender` and only validates `owner`, allowing any unprivileged address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` receives the actual payer (`sender`) as its first argument but discards it (unnamed parameter), checking only the caller-supplied `owner` address against `allowedDepositor`. Because `addLiquidity` allows `msg.sender` and `owner` to differ (operator pattern), any address not on the allowlist can deposit tokens into a restricted pool by nominating any allowlisted address as `owner`, paying the tokens themselves via the modify-liquidity callback.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` (the actual on-chain caller who will pay tokens) as the first argument and the caller-supplied `owner` as the second to the extension hook:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives both but discards `sender` (first parameter is unnamed `address,`) and only validates `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

Token payment is collected from `msg.sender` (the actual caller, Bob) via the callback invoked in `LiquidityLib.addLiquidity`:

```solidity
IMetricOmmModifyLiquidityCallback(msg.sender)
    .metricOmmModifyLiquidityCallback(amount0Added, amount1Added, callbackData);
``` [3](#0-2) 

So the allowlist check passes (Alice is allowlisted), but Bob pays the tokens and Alice receives the LP position. The guard is applied to the wrong address.

`removeLiquidity` enforces `msg.sender == owner`, so only Alice can remove the position: [4](#0-3) 

This means the deposit bypass is unconditional (no collusion needed for the deposit itself); full fund recovery requires Alice's cooperation.

Note: the `IMetricOmmPoolActions` NatSpec states "If `DEPOSIT_ALLOWLIST_PROVIDER` is set, `owner` must pass allowlist" — but this contradicts the extension's own naming (`allowedDepositor`, `isAllowedToDeposit`, `AllowedToDepositSet`) which consistently implies the intent is to gate the actual depositor/payer, not the LP position holder. [5](#0-4) 

## Impact Explanation
This is an admin-boundary break: a pool-admin-configured access-control restriction (`DepositAllowlistExtension`) is bypassed by an unprivileged path. Any address not on the allowlist can deposit tokens into a pool configured as a closed LP set (e.g., institutional pools, permissioned vaults, KYC-gated pools) by nominating any allowlisted address as `owner`. The pool receives liquidity from an unauthorized source, violating the invariant the pool admin intended to enforce. With a cooperating allowlisted address, the bypass is end-to-end (deposit and withdrawal), rendering the allowlist entirely ineffective.

## Likelihood Explanation
No special privilege is required. Any EOA or contract can call `addLiquidity` with an arbitrary `owner`. Allowlisted addresses are discoverable on-chain via `AllowedToDepositSet` events emitted by `setAllowedToDeposit`. [6](#0-5) 
The deposit bypass itself (unauthorized address depositing tokens) requires no cooperation — only full fund recovery requires a cooperating allowlisted address. Likelihood: **Medium**.

## Recommendation
Check `sender` (the actual payer) instead of, or in addition to, `owner` in `beforeAddLiquidity`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    address pool_ = msg.sender;
    if (!allowAllDepositors[pool_]
        && !allowedDepositor[pool_][sender]
        && !allowedDepositor[pool_][owner])
    {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is strictly to gate by LP-position holder (`owner`), the NatSpec, mapping name (`allowedDepositor`), event name (`AllowedToDepositSet`), and view function (`isAllowedToDeposit`) should be updated to reflect that, and the security implication (any payer can deposit) must be explicitly documented.

## Proof of Concept
```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][Alice] = true   (set via setAllowedToDeposit)
  allowedDepositor[pool][Bob]   = false  // Bob is NOT authorized

Step 1 — expected revert:
  Bob calls pool.addLiquidity(owner=Bob, salt, deltas, callbackData, extensionData)
  → extension checks allowedDepositor[pool][Bob] → false → reverts ✓

Step 2 — bypass:
  Bob calls pool.addLiquidity(owner=Alice, salt, deltas, callbackData, extensionData)
  → extension checks allowedDepositor[pool][Alice] → true → passes
  → LiquidityLib calls Bob's metricOmmModifyLiquidityCallback → Bob transfers token0/token1
  → Alice receives the LP position
  → Bob has deposited into the restricted pool without being on the allowlist ✓

Step 3 — full bypass with collusion:
  Alice calls pool.removeLiquidity(owner=Alice, salt, deltas, extensionData)
  → Alice receives token0/token1
  → Alice transfers proceeds to Bob off-chain
  → Net effect: Bob deposited and withdrew from a pool he was explicitly barred from ✓
```
