### Title
Fork Difficulty Validation Uses Stale Mainchain Ancestor Instead of Fork Ancestor, Enabling Invalid Block Acceptance - (File: `contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

---

### Summary

The `get_header_by_height` helper always reads from `mainchain_height_to_header` — the current canonical chain. When a fork block is being validated, the difficulty-adjustment calculation calls `get_header_by_height(height_first)` to obtain the reference block's timestamp. If the fork diverges before `height_first`, the mainchain block at that height is a **different block** from the fork's actual ancestor, producing an incorrect `expected_bits`. The fork block is then stored (and potentially triggers a reorg) based on a difficulty check performed against stale, wrong state. A self-acknowledged `TODO` comment in `dogecoin.rs` confirms the developers were aware of this desynchronization.

---

### Finding Description

`get_header_by_height` is implemented in `contract/src/lib.rs` as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

This unconditionally reads from the **mainchain** height-to-hash mapping. There is no mechanism to walk the fork's ancestor chain by height.

In `dogecoin.rs`, the difficulty calculation for the post-145000 protocol (where `difficulty_adjustment_interval = 1`) calls:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [2](#0-1) 

With `difficulty_adjustment_interval = 1`, `blocks_to_go_back = 1` for any block after genesis, so `height_first = prev_block_header.block_height - 1`. [3](#0-2) 

When a fork block at height N is being validated:
- `prev_block_header` is the fork's block at height N−1 (correctly fetched via `get_prev_header`, which follows `prev_block_hash` links through `headers_pool`)
- `height_first = N − 2`
- `get_header_by_height(N − 2)` returns the **mainchain** block at height N−2

If the fork diverges at height H and N−2 ≥ H, the mainchain block at N−2 is a **different block** from the fork's actual ancestor at N−2. The `first_block_time` used in `calculate_next_work_required` is therefore stale/wrong, producing an incorrect `expected_bits`.

The same structural bug exists in `bitcoin.rs` and `litecoin.rs`: [4](#0-3) [5](#0-4) 

The desynchronization is: `get_prev_header` correctly traverses the fork's ancestor chain (via hash links), but `get_header_by_height` always reads the mainchain (via height index). These two data sources diverge as soon as the fork's ancestor chain differs from the mainchain at the queried height.

---

### Impact Explanation

An attacker can submit a fork that diverges at height H. For any fork block at height N ≥ H+2 (Dogecoin post-145000), the difficulty check uses the mainchain block at height N−2 as the reference timestamp instead of the fork's actual ancestor. The attacker controls the fork block's timestamp (within MTP and future-time constraints) and can craft it so that `prev_block_time − first_block_time_mainchain` produces a `modulated_timespan` at the maximum allowed value, yielding the easiest permissible `expected_bits`. The attacker then sets `block_header.bits` to match this easy target and mines the block against it. The block passes both the `expected_bits == block_header.bits` check and the PoW hash check. If the fork accumulates sufficient chainwork, `reorg_chain` is triggered, corrupting the canonical chain mapping (`mainchain_height_to_header`, `mainchain_header_to_height`, `mainchain_tip_blockhash`). [6](#0-5) 

For Bitcoin/Litecoin (interval = 2016), the attack requires the fork to diverge before the difficulty-adjustment boundary, making it harder but structurally identical.

---

### Likelihood Explanation

For Dogecoin mainnet (height ≥ 145,000, `difficulty_adjustment_interval = 1`): **High**. Any fork block submitted at height ≥ H+2 triggers the stale-state path. The attacker only needs to submit two fork blocks before the vulnerability activates. The `TODO` comment in the production code confirms the developers identified this as an open question. The entry path is the public `submit_blocks` call, reachable by any unprivileged NEAR caller. [7](#0-6) 

For Bitcoin/Litecoin: **Low** in practice due to the 2016-block interval, but the root cause is identical.

---

### Recommendation

Replace `get_header_by_height` calls inside difficulty-adjustment logic with an ancestor-walking loop that follows `prev_block_hash` links from `prev_block_header` backward by `blocks_to_go_back` steps. This mirrors how `get_prev_header` already correctly traverses the fork chain. The `get_header_by_height` shortcut is only safe when the block being validated is on the mainchain; for fork blocks it must not be used.

---

### Proof of Concept

1. Contract is deployed for Dogecoin mainnet at height ≥ 145,000 (`difficulty_adjustment_interval = 1`).
2. Attacker calls `submit_blocks` with fork block F_{H+1} (diverges from mainchain at height H). This is stored via `store_fork_header`.
3. Attacker calls `submit_blocks` with fork block F_{H+2}. `check_pow` → `get_next_work_required` is called. `height_first = H`. `get_header_by_height(H)` returns the **mainchain** block at height H (not F_{H+1}). The attacker has set F_{H+2}'s timestamp so that `F_{H+2}.time − mainchain[H].time` maximizes `modulated_timespan`, yielding the easiest allowed `expected_bits`.
4. Attacker sets `F_{H+2}.bits` to match this easy `expected_bits` and mines F_{H+2} against it. Both `expected_bits == block_header.bits` and the PoW hash check pass.
5. Attacker repeats for F_{H+3}, F_{H+4}, … each time using the wrong mainchain reference block, accumulating chainwork at artificially low difficulty.
6. Once `fork_tip.chain_work > mainchain_tip.chain_work`, `reorg_chain` executes, replacing the canonical chain with the attacker's fork. [8](#0-7) [9](#0-8)
