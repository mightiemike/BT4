### Title
Proof Forgery via Unchecked `tx_id` Field Accepted as Internal Merkle Node in `verify_transaction_inclusion` — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` accepts a caller-supplied `tx_id` (H256) and a caller-supplied `merkle_proof` without validating that `tx_id` is a leaf node. An unprivileged NEAR caller can supply an internal Merkle tree node hash as `tx_id` together with a proof path from that internal node to the root, causing the function to return `true` for a transaction that does not exist. Any downstream contract that consumes this result will treat a fabricated Bitcoin transaction as verified.

---

### Finding Description

`verify_transaction_inclusion` is a public, non-privileged NEAR method that accepts `ProofArgs` containing `tx_id`, `tx_index`, `merkle_proof`, and `confirmations` — all fully attacker-controlled. [1](#0-0) 

The function's only structural guard on the proof is: [2](#0-1) 

After that, it passes `tx_id` and `tx_index` directly into `compute_root_from_merkle_proof`: [3](#0-2) 

`compute_root_from_merkle_proof` iterates over the proof elements, hashing left or right based on `current_position % 2`, and returns the final accumulated hash: [4](#0-3) 

There is **no check** that `tx_id` is a leaf-level hash. Bitcoin's Merkle tree internal nodes are computed as `double_sha256(left_child || right_child)` — the same hash function applied to leaves. An internal node at depth `k` with position `p` has a valid proof path of length `k` to the root. An attacker who supplies that internal node hash as `tx_id`, `p` as `tx_index`, and the `k`-element path as `merkle_proof` will cause `compute_root_from_merkle_proof` to reproduce the correct Merkle root, making the function return `true`.

The function's own documentation acknowledges this broken invariant: [5](#0-4) 

Despite being marked `#[deprecated]`, the function remains a live public entry point with `#[pause]` (pausable, not removed): [6](#0-5) 

The `ProofArgs` struct confirms all fields are fully caller-supplied with no server-side constraints: [7](#0-6) 

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` and acts on a `true` result — for example, to release funds, mint tokens, or update cross-chain state — can be deceived into accepting a fabricated Bitcoin transaction as proven. The corrupted value is the boolean proof result, which is the sole output of the SPV verification path. This is a **proof-verification forgery** with direct cross-chain financial impact.

---

### Likelihood Explanation

The attack requires no privileged role, no leaked key, and no social engineering. All inputs (`tx_id`, `tx_index`, `merkle_proof`) are freely supplied by the caller. The internal Merkle tree node hashes of any Bitcoin block are publicly computable from the block's transaction list, which is available from any Bitcoin full node or block explorer. The attacker only needs to:

1. Pick any confirmed Bitcoin block already tracked by the contract.
2. Compute its Merkle tree and select any internal node `N` at depth `k`, position `p`.
3. Compute the `k`-element proof path from `N` to the root.
4. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_index = p`, `merkle_proof = [path]`.

No cryptographic work is required. Likelihood is **high** for any deployment where a downstream contract consumes the v1 result.

---

### Recommendation

1. **Remove or gate `verify_transaction_inclusion` (v1).** The function should be made inaccessible to unprivileged callers (e.g., restricted to `Role::DAO` or removed entirely), not merely deprecated. A Rust `#[deprecated]` attribute produces only a compiler warning and does not prevent runtime calls.
2. **Enforce leaf-depth validation in `verify_transaction_inclusion_v2`.** The v2 coinbase-length check correctly prevents the 64-byte attack for callers using v2, but v1 remains reachable independently.
3. **Add a guard** that `tx_index < (1u64 << merkle_proof.len())` to reject structurally inconsistent inputs in both v1 and v2.

---

### Proof of Concept

Given a Bitcoin block `B` already tracked by the contract with Merkle root `R` and transactions `[T0, T1, T2, T3]`:

```
Merkle tree:
        R
       / \
     N01  N23
     / \  / \
    T0 T1 T2 T3
```

Internal node `N01 = double_sha256(T0 || T1)` is at depth 1, position 0.

The proof from `N01` to `R` is `[N23]` (length 1).

**Attack call** (any unprivileged NEAR account):

```json
{
  "tx_id":            "<N01 hash>",
  "tx_block_blockhash": "<hash of B>",
  "tx_index":         0,
  "merkle_proof":     ["<N23 hash>"],
  "confirmations":    1
}
```

`compute_root_from_merkle_proof(N01, 0, [N23])`:
- position 0 is even → `hash(N01, N23)` = `R`
- Returns `R`

`R == header.block_header.merkle_root` → **returns `true`**.

`N01` is not a transaction. No transaction with hash `N01` exists on Bitcoin. The contract has certified a fabricated inclusion proof. [8](#0-7) [9](#0-8)

        // Check requested confirmations. No need to compute proof if insufficient confirmations.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
