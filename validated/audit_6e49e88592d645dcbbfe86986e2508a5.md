Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Actual User — Any Unprivileged User Bypasses the Swap Allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is the immediate `msg.sender` of `MetricOmmPool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the actual end user. If the pool admin allowlists the router — required for any router-based swap to succeed — every unprivileged address on the network can bypass the swap allowlist entirely by calling the router.

## Finding Description

The root cause is a mismatch between the identity the extension checks and the identity the pool admin intends to gate.

**Call chain:**

1. `MetricOmmSimpleRouter.exactInputSingle` (public, no access control) calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — the router is `msg.sender` to the pool. [1](#0-0) 

2. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, passing the router address as `sender`. [2](#0-1) 

3. `ExtensionCalling._beforeSwap` forwards `sender` (the router) verbatim to the extension via `abi.encodeCall`. [3](#0-2) 

4. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router. The actual end user's address is never inspected. [4](#0-3) 

**Bypass path:** Once the pool admin executes `setAllowedToSwap(pool, router, true)` to permit legitimate users to swap through the router, the mapping entry `allowedSwapper[pool][router] = true` is set. From that point, any caller of `MetricOmmSimpleRouter.exactInputSingle` passes the guard unconditionally, because the extension only sees the router's address. [5](#0-4) 

No existing guard compensates for this: the `beforeSwap` interface exposes only `sender` (immediate pool caller) and `recipient` (output receiver); neither is guaranteed to be the economic actor the pool admin intended to gate, and the extension performs no additional checks.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict swaps to a curated set of counterparties (e.g., KYC-gated market makers, institutional participants) is fully open to any address the moment the pool admin allowlists the router. The allowlist guard is rendered a no-op for all router-mediated swaps. Unauthorized users can execute swaps at oracle-anchored prices, drain the pool's liquidity, extract fees, and manipulate the pool's bin cursor — all consequences the allowlist was deployed to prevent. This constitutes a direct loss of user principal and broken core pool functionality (the access-control invariant of the extension is violated), meeting the Critical/High threshold under Sherlock contest rules.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard user-facing entry point for swaps. Any pool that wants to support normal UX must allowlist the router. The bypass is therefore reachable in every realistic deployment of `SwapAllowlistExtension` that also permits router usage. No special privileges, flash loans, or unusual token behavior are required — a single call to `exactInputSingle` suffices. The attacker needs only to know the pool address and have the input token.

## Recommendation

The extension must gate the **economic actor**, not the immediate caller. Two viable approaches:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a convention between the router and the extension and must be enforced so that a malicious caller cannot forge the encoded address.
2. **Structural fix**: Add a `realSender` field to the `beforeSwap` hook that the pool populates from a trusted transient-storage slot set by the router, similar to how the liquidity adder stores payer context via `_setNextCallbackContext`. The extension then checks `realSender` instead of `sender`.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin allowlists only trustedUser:
      setAllowedToSwap(pool, trustedUser, true)
  - Pool admin also allowlists the router (required for trustedUser to swap via router):
      setAllowedToSwap(pool, router, true)

Attack:
  - attacker (not in allowlist) calls:
      MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: attacker, ...})
  - Router calls pool.swap(attacker, ...)  →  sender = router
  - SwapAllowlistExtension checks allowedSwapper[pool][router] == true  →  passes
  - Attacker's swap executes; NotAllowedToSwap is never triggered

Result:
  - attacker receives output tokens from a pool that was supposed to be restricted
  - The allowlist guard is completely bypassed
```

A Foundry integration test can reproduce this by deploying the pool with `SwapAllowlistExtension`, calling `setAllowedToSwap` for both `trustedUser` and the router, then calling `exactInputSingle` from an unprivileged `attacker` address and asserting the swap succeeds without revert.
