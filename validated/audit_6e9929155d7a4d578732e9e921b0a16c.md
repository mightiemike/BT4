### Title
Storage Deposits Permanently Locked After GC Frees Block Headers — (`contract/src/lib.rs`)

---

### Summary

NEAR tokens deposited by relayers to pay for block-header storage are permanently locked in the contract whenever `run_mainchain_gc` later frees that storage. There is no withdrawal or storage-refund function anywhere in the contract, so the freed deposit value accumulates in the contract's balance indefinitely with no recovery path for the original depositors.

---

### Finding Description

`submit_blocks` is `#[payable]` and requires callers to attach enough NEAR to cover the net storage growth of the call. The deposit accounting is computed as the **net** storage delta measured **after** GC has already run:

```rust
// contract/src/lib.rs  lines 173-197
let amount = env::attached_deposit();
let initial_storage = env::storage_usage();

for header in headers {
    self.submit_block_header(header, self.skip_pow_verification);
}

self.run_mainchain_gc(num_of_headers);          // ← GC frees old blocks HERE
let diff_storage_usage = env::storage_usage()
    .saturating_sub(initial_storage);           // ← net delta, may be 0
let required_deposit =
    env::storage_byte_cost()
        .saturating_mul(diff_storage_usage.into());

let refund = amount.saturating_sub(required_deposit);
if refund > NearToken::from_near(0) {
    Promise::new(env::predecessor_account_id()).transfer(refund).into()
} else {
    PromiseOrValue::Value(())
}
```

Only the **current call's** excess deposit is refunded. When GC removes blocks that were paid for in a **previous** call, the storage bytes are freed (reducing `env::storage_usage()`), but the NEAR tokens from that earlier call are never returned. They become part of the contract's available balance with no mechanism to send them back.

`run_mainchain_gc` removes the oldest mainchain blocks unconditionally:

```rust
// contract/src/lib.rs  lines 401-408
for height in start_removal_height..end_removal_height {
    let blockhash = &self.mainchain_height_to_header
        .get(&height)
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
    self.remove_block_header(blockhash);
    self.mainchain_height_to_header.remove(&height);
}
```

No refund is issued to whoever originally paid for those blocks. A search for `withdraw`, `reclaim`, `refund_storage`, or `storage_refund` across the entire repository returns zero matches — there is no recovery function.

**Concrete scenario:**

| Step | Action | Storage delta | Deposit consumed |
|------|--------|--------------|-----------------|
| Call 1 | Relayer A submits 100 blocks; GC threshold not yet exceeded | +100 blocks | 100 × `storage_byte_cost` locked |
| Call 2 | Relayer B submits 100 blocks; GC removes the 100 from Call 1 | net = 0 | 0 required; Relayer B gets full refund |
| — | Relayer A's deposit | — | **permanently locked, never returned** |

This repeats every GC cycle. The recommended `gc_threshold` is 52 704 blocks (≈ one year of Bitcoin blocks), meaning every year's worth of deposits from the first year of operation are locked forever.

---

### Impact Explanation

NEAR tokens deposited by relayers for storage costs are permanently locked in the contract's balance whenever GC frees the corresponding storage. Over the operational lifetime of the contract the locked amount grows proportionally to the total number of blocks ever GC'd. There is no function to withdraw or redistribute these tokens, so they are irrecoverable without a contract upgrade.

---

### Likelihood Explanation

This is **certain** to occur in normal operation. GC is designed to run automatically on every `submit_blocks` call once the mainchain exceeds `gc_threshold`. Every relayer who submits blocks before GC triggers will have their deposit permanently locked when GC eventually removes those blocks. No adversarial action is required — routine chain growth is sufficient.

---

### Recommendation

Track the cumulative NEAR freed by each GC cycle and either:

1. **Return freed storage value to a designated beneficiary** (e.g., the DAO treasury or a per-relayer accounting map) when `run_mainchain_gc` removes blocks; or
2. **Measure storage before and after GC separately** and issue a refund for the freed portion to the contract's designated treasury address within `submit_blocks`.

At minimum, add a privileged `withdraw_freed_storage(amount, recipient)` function callable by the DAO so that accumulated freed deposits can be recovered.

---

### Proof of Concept

The following pseudocode demonstrates the locked-deposit scenario using the existing test infrastructure:

```
// 1. Relayer A submits 100 blocks; gc_threshold not yet exceeded.
//    Storage grows by 100 blocks worth of bytes.
//    Relayer A pays: 100 * storage_byte_cost NEAR.
submit_blocks(100_headers).deposit(100 * STORAGE_DEPOSIT_PER_BLOCK)

// 2. Relayer B submits 100 more blocks.
//    GC fires, removes the 100 blocks Relayer A paid for.
//    Net storage delta = +100 - 100 = 0  →  required_deposit = 0.
//    Relayer B receives a full refund of their attached deposit.
submit_blocks(100_headers).deposit(100 * STORAGE_DEPOSIT_PER_BLOCK)
// Relayer B gets back: 100 * STORAGE_DEPOSIT_PER_BLOCK  ✓

// 3. Relayer A's 100 * STORAGE_DEPOSIT_PER_BLOCK is now in the
//    contract's available balance with no withdrawal path.
//    grep for "withdraw" across the entire repo → 0 matches.
assert!(contract_balance_increased_by(100 * STORAGE_DEPOSIT_PER_BLOCK));
assert!(no_withdraw_function_exists());
```

The root cause is at [1](#0-0)  — GC runs before the storage delta is measured, so freed storage from prior calls silently offsets the current call's growth, and the prior depositors receive nothing. The GC itself at [2](#0-1)  issues no refund when removing blocks. The complete absence of any withdrawal function is confirmed by the full public API at [3](#0-2) .

    /// Public call to run GC on a mainchain.
    /// `batch_size` is how many block headers should be removed in the execution
    ///
    /// # Panics
    /// If initial blockheader or tip blockheader are not in a header pool
    #[pause(except(roles(Role::UnrestrictedRunGC)))]
    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
