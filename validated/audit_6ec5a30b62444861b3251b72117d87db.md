Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual User, Enabling Full Allowlist Bypass via Router — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` resolves to the router's address, not the actual user's address. If the pool admin allowlists the router — the only mechanism to let allowlisted users access the router — every user, including those the admin intended to block, can bypass the allowlist by routing through the router.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- this is the router when called via MetricOmmSimpleRouter
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this `sender` verbatim to the extension. `SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

Where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When the call originates from `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    ...
    params.extensionData
  );
```

So `sender` inside the extension is always the router address for all router-mediated swaps. The router stores the real user's address only in transient storage for callback payment purposes (`_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn)`) and never injects it into `extensionData`. The extension has no mechanism to recover the real caller.

**Exploit path:**
1. Admin deploys pool with `SwapAllowlistExtension`.
2. Admin calls `setAllowedToSwap(pool, userA, true)` to allowlist userA.
3. Admin calls `setAllowedToSwap(pool, router, true)` to let userA use the router — the only available mechanism.
4. UserB (not allowlisted) calls `router.exactInputSingle({pool: pool, ..., recipient: userB})`.
5. Router calls `pool.swap(recipient=userB, ...)` with `msg.sender = router`.
6. Extension checks `allowedSwapper[pool][router] == true` → passes.
7. Swap executes. UserB receives token output. Allowlist is fully bypassed.

Existing guards are insufficient: the `allowAllSwappers` flag is a separate escape hatch and does not help here. The `DepositAllowlistExtension` avoids this problem by checking `owner` (the position owner passed explicitly by the pool), but the swap extension has no equivalent — it only has `sender`, which collapses to the router for all router-mediated swaps.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict swaps to a curated set of addresses (KYC-gated, institutional-only, whitelist) loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter` once the admin allowlists the router. Unauthorized swappers receive real token output from the pool; LP providers bear the other side of every unauthorized swap. This constitutes broken core pool functionality (access-controlled swap flow) and direct loss of user principal for LP providers, meeting the contest's allowed impact criteria.

## Likelihood Explanation
The `SwapAllowlistExtension` is a production extension explicitly designed for access-controlled pools. The `MetricOmmSimpleRouter` is the canonical user-facing entry point. The only way to enable router access for allowlisted users is to allowlist the router itself — the exact action that opens the bypass. No special privilege, malicious setup, or non-standard token behavior is required. Any unprivileged user can exploit this by simply calling the router after the admin performs the natural configuration step.

## Recommendation
The extension must gate the actual user, not the intermediary. Two viable approaches:

1. **Router-injected identity:** Have `MetricOmmSimpleRouter` prepend `msg.sender` (the real user) to `extensionData` before forwarding to the pool. `SwapAllowlistExtension.beforeSwap` decodes and checks that address. This requires the allowlist to trust the router as an honest forwarder, acceptable for a factory-deployed periphery contract.

2. **Signed credential in `extensionData`:** Provide a companion extension that decodes a signed user credential from `extensionData` for router paths, and document clearly that allowlisting the router grants access to all router users without this mechanism.

Either way, the current behavior — where `sender = router` for all router-mediated swaps — must be fixed before the extension is used in access-controlled production pools.
