Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address as `sender` instead of actual end-user, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the pool's `msg.sender` — the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actual_user]`. Any pool admin who allowlists the router (required to support router-mediated swaps for legitimate users) inadvertently grants swap access to every user on the public internet, completely defeating the per-user gate.

## Finding Description

**Root cause:** `MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) calls `pool.swap(...)`, the pool's `msg.sender` is the router: [4](#0-3) 

So the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

**Why existing guards fail:** The only guard is the `allowedSwapper` / `allowAllSwappers` mapping. There is no secondary check on the true originator. The `DepositAllowlistExtension` avoids this problem by checking `owner` (the LP position owner explicitly passed by the caller) rather than `sender`: [5](#0-4) 

No analogous explicit-owner parameter exists in the swap path.

**Exploit flow:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` in `beforeSwap` order.
2. Admin calls `setAllowedToSwap(pool, router, true)` — necessary because without this, even allowlisted users cannot swap via the router (the extension would see `sender = router`, find it not allowlisted, and revert).
3. Admin does **not** call `setAllowedToSwap(pool, attacker, true)`.
4. Attacker calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
5. Router calls `pool.swap(recipient, ...)` with `msg.sender = router`.
6. Pool calls `_beforeSwap(router, recipient, ...)`.
7. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
8. Attacker successfully swaps against the restricted pool.

## Impact Explanation

The `SwapAllowlistExtension` is designed to restrict pool swaps to a curated set of addresses (e.g., KYC'd counterparties, whitelisted market makers). The bypass allows unauthorized users to execute swaps against pools intended to be restricted, potentially draining liquidity or extracting value at oracle-anchored prices the pool admin intended to offer only to specific counterparties. This constitutes broken core pool functionality causing loss of funds and a broken access-control invariant — a direct loss of user/LP principal above Sherlock thresholds when the pool holds meaningful liquidity.

## Likelihood Explanation

- `MetricOmmSimpleRouter` is a public, permissionless contract any user can call.
- The bypass condition (router allowlisted) is created automatically the moment the pool admin tries to support router-mediated swaps for legitimate allowlisted users — there is no safe configuration that supports both router access for allowlisted users and exclusion of non-allowlisted users.
- No privileged access, no special setup, and no malicious initial configuration is required. Any EOA can trigger this by calling the router.

## Recommendation

The extension must check the true originator, not the intermediate router. Options:

1. **Explicit sender in `extensionData`:** The router encodes `msg.sender` (the end-user) into `extensionData`; the extension decodes and checks it. This requires the extension to trust that the pool's `msg.sender` is a known, non-spoofable router.
2. **Adopt the `DepositAllowlistExtension` pattern:** Add an explicit `swapper` parameter to the swap call (analogous to `owner` in `addLiquidity`) that the router passes as `msg.sender`, and gate on that value instead of `sender`.
3. **Allowlist individual users, not the router:** Document that the router must never be allowlisted and that allowlisted users must call `pool.swap()` directly — but this removes router usability entirely.

Option 2 is the cleanest architectural fix.
