### Title
Unconditional Trust of Attacker-Supplied `_cached_sha256_treehash` Attribute Yields Wrong Puzzle Hash — (`File: wheel/python/clvm_rs/tree_hash.py`)

---

### Summary

`sha256_treehash()` reads `_cached_sha256_treehash` from any caller-supplied Python object and returns it immediately, without any correctness check against the object's actual content. Because `Program.wrap()` propagates this attribute from arbitrary `CLVMStorage` objects, an attacker who controls a Python object passed into the hashing pipeline can force any 32-byte value to be returned as the puzzle hash of an arbitrary CLVM tree. This is a direct structural analog to the oracle-freshness bug: a cached value is consumed without validating that it still reflects the underlying data.

---

### Finding Description

`Treehasher.sha256_treehash()` in `wheel/python/clvm_rs/tree_hash.py` implements a non-recursive tree-hash walk. At the very first step of processing each node it does:

```python
r = getattr(obj, "_cached_sha256_treehash", None)
if r is not None:
    self.cache_hits += 1
    hash_stack.append(r)
    return          # ← traversal stops; children are never visited
``` [1](#0-0) 

If `r` is not `None`, the function treats it as the authoritative hash of the entire subtree rooted at `obj` and never inspects `obj.atom` or `obj.pair`. There is no check that `r` is 32 bytes, that it is a valid SHA-256 digest, or that it matches the actual content of the node.

`Program.wrap()` propagates this attribute unconditionally from any `CLVMStorage` object:

```python
o._cached_sha256_treehash = getattr(v, "_cached_sha256_treehash", None)
``` [2](#0-1) 

`Program.tree_hash()` then short-circuits on the already-poisoned field:

```python
if self._cached_sha256_treehash is None:
    self._cached_sha256_treehash = sha256_treehash(self._unwrapped)
return self._cached_sha256_treehash
``` [3](#0-2) 

The `CLVMStorage` protocol explicitly documents `_cached_sha256_treehash` as an optional speed hint, not a trusted field:

```python
# optional fields used to speed implementations:
# `_cached_sha256_treehash: Optional[bytes]` is used by `sha256_treehash`
``` [4](#0-3) 

Because the protocol is structural (duck-typed), any Python object with `.atom`, `.pair`, and `._cached_sha256_treehash` attributes satisfies it. No validation is performed on the attribute's value before it is used as the hash.

---

### Impact Explanation

The corrupted hash propagates into every downstream consumer of `tree_hash()`:

1. **`Program.__eq__`** compares programs by `tree_hash()`:
   ```python
   return self.tree_hash() == other_obj.tree_hash()
   ``` [5](#0-4) 
   Two structurally different programs can be made to compare as equal, or two identical programs as different.

2. **`Program.curry_hash`** feeds `tree_hash()` into puzzle-hash derivation:
   ```python
   quoted_mod_hash = curry_treehasher.calculate_hash_of_quoted_mod_hash(self.tree_hash())
   return curry_treehasher.curry_and_treehash(quoted_mod_hash, *args)
   ``` [6](#0-5) 
   A poisoned `tree_hash()` produces a wrong curried puzzle hash, which in Chia's consensus model is the coin's identity. A wallet or validator computing a coin ID from a poisoned puzzle hash will derive the wrong coin ID.

3. **`sha256_treehash` used as a sub-tree hash** — because the walk short-circuits at the poisoned node, all ancestor nodes in the same call also receive wrong hashes, silently corrupting the entire tree hash.

The corrupted result is a concrete, attacker-chosen 32-byte value returned in place of the correct SHA-256 tree hash.

---

### Likelihood Explanation

The attack requires the adversary to supply a Python object implementing `CLVMStorage` with a pre-set `_cached_sha256_treehash`. This is realistic in any application that:

- Accepts CLVM objects from an untrusted source (e.g., a wallet library that deserializes spend bundles and then passes the resulting Python objects through `Program.to()` or `Program.wrap()`), or
- Allows a caller to pass an arbitrary `CLVMStorage`-compatible object to `sha256_treehash()` directly.

The `CLVMStorage` protocol is intentionally open (duck-typed), so no special subclassing is required. The attacker only needs to set one attribute on an otherwise valid-looking object. No memory corruption, no Rust-level access, and no privileged position is needed.

---

### Recommendation

Validate the cached value before trusting it. At minimum, enforce that `_cached_sha256_treehash` is exactly 32 bytes before short-circuiting:

```python
r = getattr(obj, "_cached_sha256_treehash", None)
if isinstance(r, bytes) and len(r) == 32:
    self.cache_hits += 1
    hash_stack.append(r)
    return
```

For stronger guarantees in security-sensitive paths (e.g., puzzle-hash derivation used for coin-ID computation), consider ignoring the cached attribute entirely and always recomputing from the node's actual content, or restricting the cache-hit path to objects whose type is known to set the attribute correctly (e.g., `LazyNode` or `Program` instances produced by the library itself).

Similarly, `Program.wrap()` should not blindly copy `_cached_sha256_treehash` from arbitrary `CLVMStorage` objects unless the source type is trusted.

---

### Proof of Concept

```python
from clvm_rs.tree_hash import sha256_treehash
from clvm_rs.program import Program

POISON = b"\xde\xad" * 16  # 32 bytes, clearly wrong

class PoisonedAtom:
    atom = b"hello"          # actual content: atom "hello"
    pair = None
    _cached_sha256_treehash = POISON   # attacker-controlled stale value

obj = PoisonedAtom()

# Direct call: returns POISON instead of sha256(b"\x01" + b"hello")
result = sha256_treehash(obj)
assert result == POISON, f"expected poison, got {result.hex()}"

# Via Program.wrap + tree_hash: same wrong result
p = Program.wrap(obj)
assert p.tree_hash() == POISON

# Equality check is now broken:
p_real = Program.new_atom(b"hello")
# p and p_real have the same content but different tree_hash() values
assert p.tree_hash() != p_real.tree_hash()   # should be equal, but isn't
print("Puzzle hash poisoned:", result.hex())
```

The correct hash of atom `b"hello"` is `sha256(b"\x01" + b"hello")`. The poisoned object causes `sha256_treehash` to return `POISON` instead, and `Program.wrap` propagates this into `tree_hash()`, `__eq__`, and `curry_hash` without any recomputation or validation.
