### Title
Module-level `shatree_atom`/`shatree_pair` captured in closures instead of `self` methods, silently ignoring custom prefixes — (`wheel/python/clvm_rs/tree_hash.py`)

### Summary
`Treehasher.sha256_treehash` defines two inner closures (`handle_obj`, `handle_pair`) that resolve `shatree_atom` and `shatree_pair` via Python's LEGB scoping rules. Because those names are not local to the closures and not qualified with `self.`, Python resolves them to the module-level bindings — which are permanently bound to `CHIA_TREEHASHER`'s methods (prefix `0x01`/`0x02`). Any `Treehasher` instance with different prefixes silently produces Chia-prefixed hashes.

### Finding Description
The `Treehasher` class is designed to support pluggable `atom_prefix` and `pair_prefix` (constructor lines 32–35). The instance methods `shatree_atom` (line 37) and `shatree_pair` (line 43) correctly use `self.atom_prefix` and `self.pair_prefix`. However, the closures inside `sha256_treehash` (lines 63 and 86) call the bare names `shatree_atom` and `shatree_pair`, which Python resolves to the module-level aliases at lines 108–109 — permanently bound to `CHIA_TREEHASHER`. The fix is trivially `self.shatree_atom(obj.atom)` and `self.shatree_pair(p0, p1)`.

**Proof of concept** (no attacker-controlled input needed — any atom suffices):

```python
from clvm_rs.tree_hash import Treehasher
from hashlib import sha256

class FakeAtom:
    atom = b"hello"
    pair = None

h = Treehasher(b'\x03', b'\x04')
result = h.sha256_treehash(FakeAtom())

expected_custom = sha256(b'\x03' + b'hello').digest()
expected_chia   = sha256(b'\x01' + b'hello').digest()

assert result == expected_chia,   "FAILS: got Chia hash, not custom"
assert result == expected_custom, "PASSES: would be correct if bug fixed"
```

The first assert passes (demonstrating the bug); the second fails.

### Impact Explanation
Any caller that instantiates `Treehasher` with non-Chia prefixes to compute an alternative-dialect tree hash silently receives Chia-prefixed hashes. This produces wrong puzzle hashes for that dialect — a silent semantic error with no exception raised. Wallet or consensus code relying on alternative-dialect hashing would compute incorrect puzzle hashes, potentially causing fund loss or consensus failures in those dialects.

### Likelihood Explanation
The `Treehasher` class is a public, documented API with explicit prefix parameters. Any downstream code using it for a non-Chia dialect (e.g., a custom softfork dialect with different prefixes) is affected. The bug is 100% reproducible with a single-atom input and requires no special attacker capability — the wrong behavior is triggered by ordinary API use.

### Recommendation
Replace the bare name references in the closures with `self`-qualified calls:

- Line 63: `r = shatree_atom(obj.atom)` → `r = self.shatree_atom(obj.atom)`
- Line 86: `r = shatree_pair(p0, p1)` → `r = self.shatree_pair(p0, p1)`
