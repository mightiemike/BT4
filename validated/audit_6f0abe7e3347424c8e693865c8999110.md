### Title
Division by Zero in `socializeSubaccount` When `openInterest` Is Zero - (File: `core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` divides by `state.openInterest` without a zero-guard. When the liquidatee is the sole remaining participant in a perp market (so `openInterest == 0`) and still carries a negative `vQuoteBalance`, the call reverts with "DBZ", permanently blocking finalization of that insolvent subaccount.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, after exhausting insurance coverage, the protocol attempts to spread the remaining loss across all open-interest holders:

```solidity
// actually socialize if still not enough
if (balance.vQuoteBalance < 0) {
    // socialize across all other participants
    int128 fundingPerShare = -balance.vQuoteBalance.div(
        state.openInterest   // ŌåÉ divides by zero if openInterest == 0
    );
```

`state.openInterest` is the sum of absolute position sizes across all participants in that product. If the liquidatee's `amount` is already `0` (a hard requirement imposed by `_finalizeSubaccount` before it calls `socializeSubaccount`) and no other account holds a position in that market, `openInterest == 0`. The `MathSD21x18.div` helper enforces `require(y != 0, "DBZ")`, so the entire call reverts.

The call chain that reaches this code is:

```
Endpoint (onlyEndpoint) 
  ŌåÆ ClearinghouseLiq.liquidateSubaccountImpl
      ŌåÆ _finalizeSubaccount  (triggered when txn.productId == type(uint32).max)
          ŌåÆ perpEngine.socializeSubaccount   ŌåÉ revert here
```

`_finalizeSubaccount` explicitly requires every perp `balance.amount == 0` before proceeding, which is precisely the condition that can drive `openInterest` to zero for a single-participant market.

---

### Impact Explanation

When the revert fires:

1. The insolvent subaccount cannot be finalized ŌĆö it remains in the system with unresolved negative `vQuoteBalance`.
2. The insurance fund cannot be applied to cover the deficit (the insurance deduction happens inside `socializeSubaccount` before the division, but the whole call reverts, so no state is committed).
3. The liquidator's transaction always reverts for this subaccount, making the finalization path permanently inaccessible for that market/subaccount combination.
4. Protocol accounting is corrupted: the bad debt is never written off, and the subaccount is never closed.

---

### Likelihood Explanation

The condition is reachable without any privileged access:

- A perp market with a single active participant is realistic at launch or for low-liquidity markets.
- The liquidatee's `amount` reaches `0` through normal liquidation steps before finalization is attempted.
- Any liquidator can submit a `LiquidateSubaccount` transaction with `productId == type(uint32).max` to trigger the path.
- No admin action, key compromise, or governance capture is required.

---

### Recommendation

Add a zero-check before the division. If `openInterest == 0` there are no other participants to socialize the loss to; the remaining deficit should be absorbed entirely by the insurance fund or written off:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No participants to socialize to; write off remaining deficit
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(
            state.openInterest
        );
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

---

### Proof of Concept

1. Deploy a perp market with product ID `P`.
2. Account A opens a long position, then the position is fully liquidated to `amount == 0` while `vQuoteBalance < 0` (e.g., closed at a loss with no quote to settle against).
3. No other account holds a position in market `P` ŌåÆ `states[P].openInterest == 0`.
4. A liquidator submits `LiquidateSubaccount { liquidatee: A, productId: type(uint32).max, ... }`.
5. `_finalizeSubaccount` passes all `amount == 0` checks, then calls `perpEngine.socializeSubaccount(A, insurance)`.
6. Inside `socializeSubaccount`, `balance.vQuoteBalance < 0` and `state.openInterest == 0` ŌåÆ `MathSD21x18.div` reverts with `"DBZ"`.
7. The entire liquidation transaction reverts; the insolvent subaccount is never finalized. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
