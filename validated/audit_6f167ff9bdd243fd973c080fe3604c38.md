### Title
Bonus Pool Exhaustion Permanently Locks Position Lifecycle Operations — (`x/tieredrewards/keeper/bonus_rewards.go`, `claim_rewards.go`, `msg_server.go`)

---

### Summary

When the `rewards_pool` module account balance is insufficient to cover accrued bonus rewards, `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`. Because every position lifecycle mutation — `TierUndelegate`, `TierRedelegate`, `AddToTierPosition`, `ClearPosition`, and `ExitTierWithDelegation` — calls `claimRewards` and propagates this error without any fallback, users with non-zero accrued bonus are permanently unable to exit or modify their positions until the pool is externally refilled.

---

### Finding Description

**Root cause — `sufficientBonusPoolBalance`:** [1](#0-0) 

The function returns `ErrInsufficientBonusPool` whenever `poolBalance.IsAllGTE(bonus)` is false and `bonus` is non-zero. There is no cap-to-available or skip-bonus path.

**Propagation — `processEventsAndClaimBonus`:** [2](#0-1) 

The early-return guard at line 219 only fires when `totalBonus.IsZero()`. If any bonus has accrued (even 1 uatom), execution reaches line 230 and the pool-balance check can fail.

**`claimRewards` propagates the error unconditionally:** [3](#0-2) 

**Every lifecycle message calls `claimRewards` and propagates the error:**

| Message | Line |
|---|---|
| `TierUndelegate` | [4](#0-3)  |
| `TierRedelegate` | [5](#0-4)  |
| `AddToTierPosition` | [6](#0-5)  |
| `ClearPosition` | [7](#0-6)  |
| `ExitTierWithDelegation` | [8](#0-7)  |

None of these callers catch `ErrInsufficientBonusPool` and continue; they all return the error directly to the transaction handler.

---

### Impact Explanation

Any user whose position has accrued at least 1 unit of bonus reward is completely unable to undelegate, redelegate, add funds, clear an exit, or transfer their delegation out of the tier system while the `rewards_pool` balance is insufficient. Their staked tokens are effectively frozen with no user-accessible escape hatch. The pool can reach this state through normal operation (many users claiming bonuses over time) without any attacker action.

---

### Likelihood Explanation

The `rewards_pool` is a finite module account funded externally (governance/admin top-up). As the protocol matures and more bonus is paid out, the pool will eventually be exhausted. At that point, every active position with any accrued bonus — which is the normal state for any position that has been delegated for more than a few seconds — becomes permanently locked. No special attacker capability is required; the condition arises from ordinary usage.

---

### Recommendation

Decouple bonus payment from position lifecycle operations. Two concrete options:

1. **Cap-to-available**: In `processEventsAndClaimBonus`, replace the hard error with a cap: pay `min(totalBonus, poolBalance)` and record the unpaid remainder as a debt. Lifecycle operations succeed; the debt is paid when the pool is refilled.
2. **Skip-on-insufficient**: When called from a lifecycle operation (not `ClaimTierRewards`), pass a flag or use a separate helper that skips bonus payment rather than erroring when the pool is insufficient, and records the owed amount for later collection.

Either approach preserves the invariant that position lifecycle operations must not be blocked by bonus pool exhaustion.

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
