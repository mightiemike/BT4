### Title
Zero `confirmations` Bypasses Confirmation-Depth Check in SPV Verification — (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (and by extension `verify_transaction_inclusion_v2`) accepts a caller-supplied `confirmations: u64` field with no lower-bound validation. Passing `confirmations = 0` makes the confirmation-depth `require!` trivially true for every block on the main chain, defeating the only security guarantee the function provides to downstream consumers.

### Finding Description

`verify_transaction_inclusion` performs two checks on `args.confirmations`:

```rust
// upper-bound only
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [1](#0-0) 

```rust
// depth check
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [2](#0-1) 

`confirmations` is a `u64`. When the caller supplies `0`, the depth expression evaluates to `<some u64> >= 0`, which is unconditionally `true` in Rust's type system. The `require!` never fires. No lower-bound guard exists anywhere in the function or in `ProofArgs` / `ProofArgsV2`. [3](#0-2) 

`verify_transaction_inclusion_v2` delegates to the deprecated `verify_transaction_inclusion` after its coinbase-proof check, so it inherits the same flaw. [4](#0-3) 

### Impact Explanation

The `confirmations` parameter is the sole mechanism by which a downstream NEAR contract enforces Bitcoin finality before releasing funds or updating state. With `confirmations = 0`, the function returns `true` for any transaction whose block is the current main-chain tip — a block that has received zero confirmations and can still be reorganized away. An attacker who controls a Bitcoin transaction (e.g., a double-spend attempt) can submit the block, immediately call `verify_transaction_inclusion` with `confirmations = 0`, receive `true`, and trigger whatever action the consuming contract guards behind this check, before the honest chain has time to reorg the block out.

### Likelihood Explanation

The entry point is fully public and unprivileged — any NEAR account can call `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` with an arbitrary `ProofArgs` / `ProofArgsV2` struct. No role, stake, or special permission is required. The only prerequisite is that the attacker's target block is already on the main chain, which the attacker can arrange by submitting it via `submit_blocks`. The exploit requires no leaked keys, no social engineering, and no privileged access.

### Recommendation

Add a lower-bound guard at the start of `verify_transaction_inclusion`:

```rust
require!(args.confirmations > 0, "confirmations must be at least 1");
```

Alternatively, enforce the constraint in `ProofArgs` / `ProofArgsV2` at deserialization time, or document and enforce a minimum at the `verify_transaction_inclusion_v2` layer so the deprecated path cannot be called with zero confirmations either.

### Proof of Concept

```rust
// Attacker submits a block they control (e.g., containing a double-spend tx).
// The block is now the main-chain tip with 0 confirmations.
contract.submit_blocks(vec![attacker_block_header]);

// Attacker immediately calls verify with confirmations = 0.
let result = contract.verify_transaction_inclusion(ProofArgs {
    tx_id: double_spend_tx_id,
    tx_block_blockhash: attacker_block_hash,
    tx_index: 0,
    merkle_proof: valid_merkle_proof,
    confirmations: 0,   // <-- no lower-bound check; depth require! is 1 >= 0 → true
});

assert!(result); // returns true despite zero confirmations
// Downstream contract releases funds before any reorg can occur.
```

The same exploit works through `verify_transaction_inclusion_v2` by additionally supplying a valid `coinbase_merkle_proof`.
