Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks Caller-Supplied `owner` Instead of Actual `sender`, Allowing Deposit Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the real transaction initiator) and enforces the allowlist against the caller-supplied `owner` parameter instead. Because `owner` is a free parameter of `MetricOmmPool.addLiquidity`, any unprivileged address can pass an allowlisted address as `owner`, satisfy the check, and inject liquidity into a restricted pool. The deposit allowlist — the primary access-control gate for permissioned pools — is entirely nullified.

## Finding Description
`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)`, correctly forwarding both the real caller (`msg.sender`) as `sender` and the position-owner parameter as `owner`. [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes both addresses and dispatches them to the extension: [2](#0-1) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first parameter (`sender`) is unnamed and discarded. The allowlist check is performed on `owner`: [3](#0-2) 

Since `owner` is a free caller-supplied argument to `addLiquidity`, any address can pass an already-allowlisted address as `owner`, causing `allowedDepositor[pool][owner]` to return `true` and the guard to pass without checking the actual depositor.

The sibling `SwapAllowlistExtension.beforeSwap` correctly names and checks `sender` (the first argument), confirming the deposit extension is checking the wrong field: [4](#0-3) 

`removeLiquidity` enforces `msg.sender == owner`, so the attacker cannot reclaim the deposited tokens — the position accrues to the allowlisted address — but the pool's liquidity composition is altered without authorization. [5](#0-4) 

## Impact Explanation
The deposit allowlist is the sole on-chain mechanism restricting who may provide liquidity to a permissioned pool. Bypassing it breaks the core pool invariant that only authorized depositors can alter bin balances. Unauthorized liquidity injection can shift bin token ratios, move `curPosInBin`, and interact with stop-loss or velocity extension watermarks in ways the pool admin did not authorize. Pools deployed for compliance-gated or institutional environments have their access control silently nullified. This constitutes broken core pool functionality as defined by the allowed impact gate.

## Likelihood Explanation
Exploitation requires no privilege. Any EOA or contract can call `pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, "")`. The allowlisted addresses are readable from public `allowedDepositor` mapping state. No flash loan, oracle manipulation, or admin cooperation is needed. The attacker forfeits the deposited tokens (credited to the allowlisted `owner`), making this a griefing/unauthorized-injection vector rather than a theft vector, but the bypass itself is unconditional and repeatable.

## Recommendation
Name the first parameter `sender` and enforce the allowlist against it, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
1. Pool `P` is deployed with `DepositAllowlistExtension` configured as a `beforeAddLiquidity` hook.
2. Admin calls `setAllowedToDeposit(P, alice, true)`. Alice is the only allowlisted depositor.
3. Bob (not allowlisted) deploys `BobRouter` implementing `IMetricOmmSwapCallback`.
4. `BobRouter` calls `P.addLiquidity(alice, 0, deltas, callbackData, "")`.
5. The extension evaluates `allowedDepositor[P][alice]` → `true` → no revert. Bob's actual address is never checked.
6. `LiquidityLib.addLiquidity` records the position under key `(alice, 0)`.
7. The pool calls `BobRouter.metricOmmSwapCallback(...)`, which transfers the required tokens from Bob.
8. Bob has deposited into the restricted pool; the allowlist was never applied to Bob's address.
