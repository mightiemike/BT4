### Title
Unbounded `NlpLockedBalanceQueue` Growth Causes Permanent NLP Token Lock for Frequent Minters — (File: `core/contracts/SpotEngine.sol`)

---

### Summary

`SpotEngine.handleNlpLockedBalance` appends a new entry to a per-subaccount `NlpLockedBalanceQueue` on every NLP mint that occurs at a distinct oracle timestamp. `tryUnlockNlpBalance`, which iterates over this queue, is called on every subsequent NLP balance update. A user who mints NLP repeatedly across many distinct timestamps can grow their personal queue without bound. Once the queue is large enough, `tryUnlockNlpBalance` exhausts the block gas limit, causing every future NLP balance update — including burns — for that subaccount to revert. The user's NLP tokens become permanently unredeemable.

---

### Finding Description

`SpotEngine.handleNlpLockedBalance` (lines 139–174) manages a per-subaccount queue of locked NLP balances. When `amountDelta > 0` (a mint), the function either merges the new amount into the last queue entry if the `unlockedAt` timestamp matches, or appends a brand-new entry and increments `queue.balanceCount`:

```solidity
queue.balances[queue.balanceCount] = NlpLockedBalance({
    balance: Balance({amount: amountDelta}),
    unlockedAt: getOracleTime() + NLP_LOCK_PERIOD
});
queue.balanceCount++;
```

Before appending, the function calls `tryUnlockNlpBalance(subaccount)` (line 147), which iterates over `queue.balances[0..balanceCount-1]` to move matured entries into `unlockedBalanceSum`. This same call is made on every NLP balance update — including burns — via `updateBalance` → `handleNlpLockedBalance`.

Because the merge condition requires an exact timestamp match (`queue.balances[queue.balanceCount - 1].unlockedAt == getOracleTime() + NLP_LOCK_PERIOD`), any mint at a new oracle timestamp creates a distinct entry. On a chain like Arbitrum where block timestamps advance every ~0.25 s, a user minting NLP once per block accumulates one new queue entry per block. Over the `NLP_LOCK_PERIOD` window, the queue can grow to hundreds of thousands of entries. At that scale, `tryUnlockNlpBalance`'s iteration exhausts the block gas limit, causing every subsequent call to `updateBalance` for that subaccount to revert — including the burn path.

There is no administrative escape hatch: the queue is keyed by subaccount, NLP balances are not transferable as raw ERC-20 tokens outside the engine's balance system, and no function exists to prune or reset the queue.

---

### Impact Explanation

A user whose `NlpLockedBalanceQueue` has grown beyond the gas-safe iteration threshold can no longer:

- Burn NLP to redeem underlying collateral.
- Receive any further NLP mints (the update itself reverts).
- Have their NLP balance updated by any internal protocol path.

The NLP tokens held by that subaccount are permanently locked with no recovery path. The exact corrupted state is `nlpLockedBalanceQueues[subaccount]` — the queue becomes unprocessable, and the subaccount's NLP balance is frozen at its last recorded value. This constitutes a direct, irreversible loss of user funds.

---

### Likelihood Explanation

The trigger is entirely within the user's own transaction history. On Arbitrum, oracle timestamps advance frequently; a user running an automated strategy that mints NLP on each rebalance (e.g., a yield-optimizing bot) will accumulate one queue entry per distinct timestamp. Over weeks of operation this reaches a gas-unsafe size without any unusual behavior. The user need not be adversarial — the lock is self-inflicted through normal high-frequency usage. No admin action, sequencer compromise, or external dependency is required.

---

### Recommendation

1. **Cap queue depth**: Enforce a maximum `balanceCount` (e.g., 256 entries) and revert or merge aggressively when the cap is reached.
2. **Bounded iteration in `tryUnlockNlpBalance`**: Process at most `N` entries per call and track a `processedUpTo` cursor so the work is amortized across transactions rather than done all at once.
3. **Merge on approximate timestamp**: Widen the merge window (e.g., same epoch rather than exact timestamp) to reduce entry creation rate.

---

### Proof of Concept

1. User calls `MintNlp` (via sequencer) once per distinct oracle timestamp over a long period, accumulating `K` entries in `nlpLockedBalanceQueues[subaccount]`.
2. Each mint at a new timestamp hits the `else` branch at line 162 of `SpotEngine.sol`, incrementing `queue.balanceCount`.
3. When `K` is large enough (empirically testable; on Arbitrum likely in the range of tens of thousands), the next call to `updateBalance(NLP_PRODUCT_ID, subaccount, ...)` invokes `handleNlpLockedBalance` → `tryUnlockNlpBalance`, which iterates over all `K` entries and runs out of gas.
4. The `BurnNlp` transaction (sequencer-submitted, user-signed) calls `clearinghouse.burnNlp` → `spotEngine.updateBalance(NLP_PRODUCT_ID, subaccount, -amount)` → `handleNlpLockedBalance` → `tryUnlockNlpBalance` → OOG revert.
