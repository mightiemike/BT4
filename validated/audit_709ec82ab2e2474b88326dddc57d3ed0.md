Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual swapper, enabling allowlist bypass via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to all before-swap extensions. When swaps are routed through `MetricOmmSimpleRouter`, `msg.sender` inside the pool is the router contract, not the end user. `SwapAllowlistExtension.beforeSwap` checks this `sender` value against the per-pool allowlist, so any pool that allowlists the router to support approved-user routing inadvertently grants swap access to every user who calls the router.

## Finding Description
`MetricOmmPool.swap` captures `msg.sender` at the call site and forwards it as `sender` to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ŌåÉ always the direct caller of pool.swap
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` encodes that value as the first positional argument of the `IMetricOmmExtensions.beforeSwap` ABI call (lines 162-176). `SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap`. When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)` directly (lines 72-80 of `MetricOmmSimpleRouter.sol`) without forwarding the originating user's address. The pool therefore receives `msg.sender = router`, and the extension evaluates `allowedSwapper[pool][router]`.

A pool admin who wants allowlisted users to swap via the router must add the router to the allowlist. Once `allowedSwapper[pool][router] = true`, every user ŌĆö including those never individually allowlisted ŌĆö can bypass the gate by routing through `MetricOmmSimpleRouter`. There is no mechanism in the extension to distinguish the router from the actual economic actor.

## Impact Explanation
Any pool deploying `SwapAllowlistExtension` as a before-swap hook to create a permissioned trading venue loses its access-control guarantee the moment the router is added to the allowlist. Non-allowlisted users can trade against the pool at oracle-anchored prices. If those users hold informational advantages (e.g., stale oracle, imbalanced pool), they can extract value from LP positions. The allowlist ŌĆö the pool's primary defense against adverse selection ŌĆö fails open on the most common periphery path. This constitutes broken core pool functionality with potential LP principal loss.

## Likelihood Explanation
Medium. The bypass is only reachable after the pool admin adds the router to the allowlist. However, this is the natural, expected action for any operator who wants approved users to use the standard periphery. The admin has no indication from the contract or its NatSpec that doing so opens the gate to all users. The scenario is a foreseeable misconfiguration rather than an exotic edge case.

## Recommendation
The extension must check the identity of the economic actor, not the calling contract. Two viable approaches:

1. **Pass the originating user through `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling the pool; `SwapAllowlistExtension` decodes and checks that address. Pool admins allowlist individual users, not the router.

2. **Two-layer check**: Introduce a router allowlist and a user allowlist. First verify the calling contract is an approved router, then verify the user address embedded in `extensionData` is on the per-user allowlist.

Either approach ensures the guard checks the same actor the pool admin intended to gate.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured as a before-swap hook.
2. Pool admin calls `setAllowedToSwap(pool, userA, true)` to allowlist user A.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` to allow router-mediated swaps for user A.
4. Non-allowlisted user B calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
5. The router calls `pool.swap(recipient, ...)` ŌĆö `msg.sender` inside the pool is the router address.
6. `_beforeSwap` encodes `sender = router` and calls the extension.
7. The extension evaluates `allowedSwapper[pool][router] == true` ŌåÆ no revert.
8. User B's swap executes against the curated pool, bypassing the per-user allowlist entirely.
