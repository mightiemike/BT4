### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Allowing Any Caller to Bypass Coinbase Merkle Proof Validation and Forge SPV Proof Results — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is marked `#[deprecated]` in Rust but remains a fully accessible `pub` NEAR contract method. Any unprivileged NEAR account can call it directly, bypassing the coinbase Merkle proof check that `verify_transaction_inclusion_v2` enforces to mitigate the 64-byte transaction Merkle proof forgery vulnerability. The Rust `#[deprecated]` attribute only emits compiler warnings for Rust callers — it has no effect on on-chain method dispatch.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced to fix a known proof-forgery vulnerability: without a coinbase proof, an attacker can supply an internal Merkle tree node hash as `tx_id` and construct a valid Merkle path to the block's `merkle_root`, causing the contract to return `true` for a transaction that does not exist. The fix requires the caller to also supply a coinbase Merkle proof, which anchors the tree depth and prevents internal-node substitution.

However, `verify_transaction_inclusion` — the insecure predecessor — was not removed or access-restricted. It is still declared `pub` and carries only a Rust-level `#[deprecated]` annotation:

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
    ...
    merkle_tools::compute_root_from_merkle_proof(
        args.tx_id,
        usize::try_from(args.tx_index).unwrap(),
        &args.merkle_proof,
    ) == header.block_header.merkle_root
}
```

The `#[deprecated]` attribute is a Rust compiler hint. NEAR's runtime dispatches methods by name string — it has no concept of deprecation. Any caller can invoke `"verify_transaction_inclusion"` directly via a NEAR transaction or view call, completely skipping the coinbase validation gate in `verify_transaction_inclusion_v2`.

The contract's own documentation acknowledges the vulnerability in the v1 function:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash."

The analog to ENS M-01 is direct: just as wrapper approval in ENS unintentionally extended to unwrapped domains (a context it was not meant to cover), the verification permission here unintentionally extends to the insecure deprecated path — any caller who should be routed through the secure v2 gate can instead call v1 and bypass it entirely.

---

### Impact Explanation

An attacker can obtain a `true` SPV proof result from the on-chain contract for a Bitcoin transaction that was never included in any block. Any downstream NEAR contract or off-chain system that calls `verify_transaction_inclusion` and trusts its boolean result can be deceived into releasing funds, minting tokens, or taking other irreversible actions predicated on a forged Bitcoin transaction.

The exact corrupted value is the **proof result**: `verify_transaction_inclusion` returns `true` for a crafted `(tx_id, tx_index, merkle_proof)` tuple where `tx_id` is an internal Merkle tree node, not a real transaction hash.

---

### Likelihood Explanation

The entry path requires no privileges — any NEAR account can call `verify_transaction_inclusion` as a view call (free) or a signed transaction. The 64-byte transaction forgery technique is publicly documented (https://www.bitmex.com/blog/64-Byte-Transactions) and the contract itself cites it. Any integrator who reads the contract ABI (which lists both methods) may call v1 by mistake or by intent. The likelihood is high wherever downstream contracts consume the v1 result.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public contract interface entirely. Since `verify_transaction_inclusion_v2` already calls `verify_transaction_inclusion` internally via `#[allow(deprecated)]`, the shared logic can be extracted into a private helper and the public v1 method deleted. If removal is not immediately feasible, the method body should be replaced with an unconditional `env::panic_str("use verify_transaction_inclusion_v2")` to prevent any on-chain use.

---

### Proof of Concept

1. A block with `merkle_root = R` is on the mainchain.
2. Attacker identifies two sibling hashes `A` and `B` such that `SHA256d(A || B) = R` (or any internal node reachable from `R`).
3. Attacker calls `verify_transaction_inclusion` with:
   - `tx_id = A` (an internal node, not a real transaction)
   - `tx_index = 0`
   - `merkle_proof = [B]`
   - `tx_block_blockhash` = the block's hash
4. `compute_root_from_merkle_proof(A, 0, [B])` computes `SHA256d(A || B) = R`, which equals `header.block_header.merkle_root`.
5. The function returns `true`.
6. A downstream contract that called `verify_transaction_inclusion` now believes the fake transaction was confirmed.

This call path is entirely unprivileged and costs only standard NEAR gas. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
