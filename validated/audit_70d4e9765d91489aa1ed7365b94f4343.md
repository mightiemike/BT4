Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Originating User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `sender`, which is the pool's `msg.sender` at swap time. When a swap is routed through `MetricOmmSimpleRouter`, `sender` is the router's address, not the original user's address. If the pool admin allowlists the router (the only way to permit allowlisted users to trade via the standard periphery), every unprivileged user can bypass the allowlist by calling the router, rendering the access control completely ineffective.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs the check at line 37:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is the first argument forwarded by the pool — the pool's own `msg.sender` at the time `swap` was called.

In `MetricOmmPool.swap` (lines 230–240), `_beforeSwap(msg.sender, ...)` is called, so `sender` in the extension equals whoever called `pool.swap(...)`.

In `MetricOmmSimpleRouter.exactInputSingle` (lines 72–80), the router calls `pool.swap(...)` directly — making the pool's `msg.sender` the **router contract address**. The original user's address is stored only in transient callback context (`_setNextCallbackContext`) for payment purposes and is never forwarded to the extension via `extensionData` or any other parameter. The `params.extensionData` passed to the pool is user-supplied and not router-injected.

The allowlist check therefore evaluates `allowedSwapper[pool][router]`. The original user's identity is entirely absent from the extension's decision. There is no existing guard that recovers the original initiator.

**Exploit flow:**
1. Pool admin deploys pool with `SwapAllowlistExtension` in `BEFORE_SWAP_ORDER`.
2. Admin allowlists `user1`: `allowedSwapper[pool][user1] = true`.
3. Admin allowlists the router so `user1` can trade via `MetricOmmSimpleRouter`: `allowedSwapper[pool][router] = true`.
4. Attacker (`user2`, not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(...)` — `msg.sender` to pool = router address.
6. Pool calls `_beforeSwap(sender=router, ...)`.
7. Extension evaluates: `allowedSwapper[pool][router] == true` → passes.
8. Swap executes. `user2` has bypassed the allowlist with no privileged access.

## Impact Explanation
`SwapAllowlistExtension` is the protocol's primary mechanism for restricting swap access to specific addresses (e.g., KYC'd users, institutional counterparties, whitelisted market makers). Once the router is allowlisted, the guard is fully bypassed: any address can execute swaps against a pool intended to be access-controlled. LPs who deposited under the assumption that only vetted counterparties would trade against them are exposed to unrestricted swap flow, which can drain their positions at oracle-derived prices. This constitutes broken core pool functionality and an admin-boundary break via an unprivileged path — both qualifying impacts under the contest rules.

## Likelihood Explanation
The trigger condition — the router being allowlisted — is not a misconfiguration; it is the only valid operational state for a pool that uses `SwapAllowlistExtension` and also wants allowlisted users to trade through `MetricOmmSimpleRouter`. Any pool operator who deploys this extension and expects users to interact via the standard periphery router will inevitably reach this state. The bypass is then reachable by any unprivileged user with no special access, no admin role, and no malicious setup beyond calling the public router function.

## Recommendation
Two remediation paths exist:

1. **Protocol-level fix (preferred):** Have `MetricOmmSimpleRouter` encode `msg.sender` into `extensionData` before calling the pool (e.g., ABI-encode it as a prefix). `SwapAllowlistExtension.beforeSwap` would then decode the original initiator from `extensionData` and check it against the allowlist instead of `sender`. This ensures the checked identity is always the economic initiator, not the intermediary.

2. **Extension-level workaround:** Document that `SwapAllowlistExtension` is incompatible with any intermediary router and must only be used with direct pool calls. Enforce this by reverting if `sender` is a known router address, or by requiring a dedicated non-public router that enforces its own identity checks before calling the pool.
