Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which equals the pool's `msg.sender`. When `MetricOmmSimpleRouter` calls the pool, the pool's `msg.sender` is the router contract, not the end user. Any pool admin who allowlists the router to enable router-mediated swaps for legitimate users simultaneously grants every non-allowlisted user the ability to bypass the allowlist by routing through the same router contract.

## Finding Description
`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at line 230–240, passing the pool's own `msg.sender` as `sender`. `ExtensionCalling._beforeSwap` (L149–177) encodes that value as the first argument in the `abi.encodeCall` to the extension. `SwapAllowlistExtension.beforeSwap` (L31–41) then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

where `msg.sender` is the pool and `sender` is the router. When `MetricOmmSimpleRouter.exactInputSingle` (L67–86) calls `IMetricOmmPoolActions(params.pool).swap(...)`, the router is the pool's `msg.sender`. The router does not encode the original caller into `extensionData`, and the extension does not inspect `extensionData` for an alternate identity. Therefore the check resolves to `allowedSwapper[pool][router]`.

**Call path for bypass:**
1. Admin deploys pool with `SwapAllowlistExtension`.
2. Admin allowlists `userA`: `setAllowedToSwap(pool, userA, true)`.
3. Admin allowlists the router so `userA` can use it: `setAllowedToSwap(pool, address(router), true)`.
4. Disallowed `userB` calls `router.exactInputSingle(...)` targeting the same pool.
5. Pool receives `msg.sender = router`; extension checks `allowedSwapper[pool][router]` → `true`; swap proceeds.

No existing guard prevents this: the extension has no router registry, no transient-storage identity propagation, and no fallback to `extensionData`.

## Impact Explanation
Once the router is allowlisted — a necessary step for any admin who wants allowlisted users to use the standard periphery — the bypass is unlimited. Any disallowed user can execute an arbitrary number of swaps on the curated pool. LP principal is exposed to unauthorized trading activity on a pool explicitly configured to restrict access (e.g., KYC, institutional-only, compliance-gated). The pool's curation policy is silently voided with no on-chain signal. This matches the "admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" impact gate and the "broken core pool functionality" gate, as the allowlist is a core configured protection.

## Likelihood Explanation
The trigger is a routine admin action: allowlisting `MetricOmmSimpleRouter` so that allowlisted users can use the standard periphery. The admin has no on-chain signal that doing so opens the allowlist to all users. The bypass requires no special privileges, no flash loan, and no oracle manipulation — only a standard router call. Any user who discovers the allowlisted router can exploit it immediately and repeatedly.

## Recommendation
**Short-term:** Add an explicit NatSpec warning to both `SwapAllowlistExtension` and `MetricOmmSimpleRouter` that the router cannot be allowlisted on a restricted pool without voiding the allowlist for all users.

**Long-term:** Have `MetricOmmSimpleRouter` encode the original `msg.sender` into `extensionData` (or a dedicated transient slot readable by extensions), and update `SwapAllowlistExtension.beforeSwap` to extract and check that identity when the direct `sender` is a known router. Alternatively, maintain a registry of trusted routers and require routers to attest the real initiator via a transient-storage mechanism.
