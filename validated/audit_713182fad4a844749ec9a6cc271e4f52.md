### Title
`ValidResourceBounds::L1Gas` / `AllResources` Misclassification in P2P Sync Protobuf Deserialization Produces Wrong Transaction Hash Preimage and Wrong RPC Resource Bounds - (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` silently collapses `AllResources(l2_gas=0, l1_data_gas=0)` into `ValidResourceBounds::L1Gas`. Because `get_tip_resource_bounds_hash` hashes a **different number of resource-bound felts** depending on the variant (2 for `L1Gas`, 3 for `AllResources`), the hash preimage changes. Any V3 transaction that was signed and committed with `AllResources` but has both `l2_gas=0` and `l1_data_gas=0` is stored by a P2P-syncing node with the wrong variant. This causes the RPC to return wrong resource bounds for that transaction and causes re-execution to use a different gas-computation mode (`NoL2Gas` instead of `All`), potentially producing a different execution result than the original.

---

### Finding Description

**Step 1 – The variant-collapsing conversion**

`crates/apollo_protobuf/src/converters/transaction.rs` lines 417–436:

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // ...
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        // ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)          // ← collapses AllResources
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}

**Step 2 – The hash preimage divergence**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` produces a **different number of felts** depending on the variant:

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],                          // 2 bounds
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 bounds
        }
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
``` [3](#0-2) 

`poseidon([tip, L1_GAS_packed, L2_GAS_packed])` ≠ `poseidon([tip, L1_GAS_packed, L2_GAS_packed, L1_DATA_GAS_packed])` even when `L2_GAS_packed = 0` and `L1_DATA_GAS_packed = 0`, because Poseidon is sensitive to the number of inputs.

**Step 3 – The gas-computation-mode divergence**

```rust
pub fn get_gas_vector_computation_mode(&self) -> GasVectorComputationMode {
    match self {
        Self::AllResources(_) => GasVectorComputationMode::All,
        Self::L1Gas(_)        => GasVectorComputationMode::NoL2Gas,
    }
}
``` [4](#0-3) 

A transaction originally executed under `All` mode (which enforces `l2_gas.max_amount=0` as a hard cap) is re-executed under `NoL2Gas` mode (which does not track L2 gas separately), potentially producing a different revert/success outcome.

**Step 4 – The P2P sync path trusts the wire hash**

`TryFrom<protobuf::TransactionInBlock> for (Transaction, TransactionHash)` takes the hash directly from the wire:

```rust
let tx_hash = value
    .transaction_hash
    .clone()
    .ok_or(missing("Transaction::transaction_hash"))?
    .try_into()
    .map(TransactionHash)?;
``` [5](#0-4) 

The stored hash is H₁ (computed with 3 bounds by the proposer), but the stored `ValidResourceBounds` variant is `L1Gas`. Any code that recomputes the hash from the stored transaction fields produces H₂ ≠ H₁.

**Step 5 – The consensus path is unaffected (but the sync path is)**

The consensus/gateway path always uses `AllResourceBounds` (never `ValidResourceBounds::L1Gas`) and recalculates the hash:

```rust
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
``` [6](#0-5) 

The vulnerability is confined to the P2P state-sync path, which stores the transaction with the wrong variant and the wire hash without re-validation.

---

### Impact Explanation

**High – RPC returns an authoritative-looking wrong value.**

A node that synced a block via P2P will serve `starknet_getTransactionByHash` with `resource_bounds` missing the `L1_DATA_GAS` key for any V3 transaction that had `l2_gas=0` and `l1_data_gas=0`. The `ValidResourceBounds::L1Gas` serializer emits only `{L1_GAS, L2_GAS}`:

```rust
ValidResourceBounds::L1Gas(l1_gas) => BTreeMap::from([
    (Resource::L1Gas, *l1_gas),
    (Resource::L2Gas, ResourceBounds::default()),
]),
``` [7](#0-6) 

Additionally, blockifier re-execution on the syncing node uses `NoL2Gas` mode instead of `All`, which can produce a different revert/success result for the same transaction, breaking state-root consistency between the proposing node and syncing nodes.

---

### Likelihood Explanation

Any V3 Invoke or DeployAccount transaction with `l2_gas.max_amount = 0` and `l1_data_gas.max_amount = 0` (a valid and common configuration for pre-0.13.3-style V3 transactions submitted after the upgrade) triggers this path on every P2P-syncing node. No special attacker capability is required beyond submitting a normal transaction.

---

### Recommendation

1. **In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`**: do not collapse `AllResources` to `L1Gas` based on zero values. The `L1Gas` variant should only be produced when the wire message explicitly signals a pre-0.13.3 transaction (e.g., by the absence of the `l1_data_gas` field, not by its zero value). Use a separate sentinel or a version tag to distinguish the two cases.

2. **In the P2P sync path**: after deserializing a `TransactionInBlock`, recompute the transaction hash from the deserialized fields and verify it matches the wire hash before storing. This would catch any variant-collapse that changes the hash preimage.

3. **Consider making `ValidResourceBounds` non-collapsible** in the protobuf layer by using `AllResourceBounds` (the always-three-bounds type) for all V3 transactions in the sync path, mirroring the consensus path.

---

### Proof of Concept

1. Submit a V3 Invoke transaction to the gateway with `resource_bounds = AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`. The gateway computes hash H₁ using `get_tip_resource_bounds_hash` with 3 bounds: `poseidon([tip, pack(L1_GAS,X), pack(L2_GAS,0), pack(L1_DATA_GAS,0)])`.

2. The transaction is included in a block. A second node syncs the block via P2P. The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion fires: `l1_data_gas.is_zero() && l2_gas.is_zero()` → stores `ValidResourceBounds::L1Gas(X)`.

3. Call `starknet_getTransactionByHash` on the syncing node. The returned `resource_bounds` map contains only `{L1_GAS: X, L2_GAS: 0}`, missing `L1_DATA_GAS`. This differs from the original submission and from the proposing node's response.

4. Recompute the hash from the stored transaction on the syncing node: `get_tip_resource_bounds_hash` now uses 2 bounds → `poseidon([tip, pack(L1_GAS,X), pack(L2_GAS,0)])` = H₂ ≠ H₁. The stored hash is H₁ but the transaction fields produce H₂, breaking any hash-consistency check. [1](#0-0) [3](#0-2) [8](#0-7) [9](#0-8)

    tip: &Tip,

    let l1_resource_bounds = resource_bounds.get_l1_bounds();

    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],

            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())

#[derive(Clone, Copy, Debug, Eq, PartialEq, Hash, Ord, PartialOrd)]
pub enum ValidResourceBounds {
    L1Gas(ResourceBounds), // Pre 0.13.3. Only L1 gas. L2 bounds are signed but never used.
    AllResources(AllResourceBounds),
}

impl From<AllResourceBounds> for ValidResourceBounds {
    fn from(value: AllResourceBounds) -> Self {
        Self::AllResources(value)
    }
}

impl ValidResourceBounds {
    pub fn get_l1_bounds(&self) -> ResourceBounds {
        match self {
            Self::L1Gas(l1_bounds) => *l1_bounds,
            Self::AllResources(AllResourceBounds { l1_gas, .. }) => *l1_gas,
        }
    }

    pub fn get_l2_bounds(&self) -> ResourceBounds {
        match self {
            Self::L1Gas(_) => ResourceBounds::default(),
            Self::AllResources(AllResourceBounds { l2_gas, .. }) => *l2_gas,
        }
    }

    /// Returns the maximum possible fee that can be charged for the transaction.
    /// The computation is saturating, meaning that if the result is larger than the maximum
    /// possible fee, the maximum possible fee is returned.
    pub fn max_possible_fee(&self, tip: Tip) -> Fee {
        match self {
            ValidResourceBounds::L1Gas(l1_bounds) => {
                l1_bounds.max_amount.saturating_mul(l1_bounds.max_price_per_unit)
            }
            ValidResourceBounds::AllResources(AllResourceBounds {
                l1_gas,
                l2_gas,
                l1_data_gas,
            }) => l1_gas
                .max_amount
                .saturating_mul(l1_gas.max_price_per_unit)
                .saturating_add(
                    l2_gas
                        .max_amount
                        .saturating_mul(l2_gas.max_price_per_unit.saturating_add(tip.into())),
                )
                .saturating_add(
                    l1_data_gas.max_amount.saturating_mul(l1_data_gas.max_price_per_unit),
                ),
        }
    }

    pub fn get_gas_vector_computation_mode(&self) -> GasVectorComputationMode {
        match self {
            Self::AllResources(_) => GasVectorComputationMode::All,
            Self::L1Gas(_) => GasVectorComputationMode::NoL2Gas,
        }
    }
