### Title
NLP pool owner (`nlpSigner`) excluded from `validateSignature` authorization check, blocking collateral withdrawal and other signed operations on NLP pool subaccounts - (`File: core/contracts/EndpointTx.sol`)

### Summary

The Nado protocol maintains two distinct authorized-signer mappings for subaccounts: `linkedSigners` (set by users via `LinkSigner`) and `nlpSigners` (set by the protocol when an NLP pool is created). The `getLinkedSignerOrNlpSigner` helper correctly consults both, and is used for `MatchOrders` / `MatchOrdersWithAmount`. However, `validateSignature` and `validateCompactSignature` — which gate every other signed transaction type — only call `getLinkedSigner`, silently omitting `nlpSigners`. As a result, an NLP pool owner is a recognized authorized signer for order-matching but is completely unrecognized for `WithdrawCollateral`, `WithdrawCollateralV2`, `LinkSigner`, `TransferQuote`, and `LiquidateSubaccount` when the sender is an NLP pool subaccount.

### Finding Description

**Root cause — `validateSignature` uses `getLinkedSigner` instead of `getLinkedSignerOrNlpSigner`:**

`EndpointTx.validateSignature` (and its compact variant) resolve the permitted delegate as:

```solidity
// EndpointTx.sol L178-183
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
```

`getLinkedSigner` only reads `linkedSigners[subaccount]`:

```solidity
// EndpointTx.sol L143-157
function getLinkedSigner(bytes32 subaccount) public view virtual returns (address) {
    return RiskHelper.isIsolatedSubaccount(subaccount)
        ? linkedSigners[IOffchainExchange(offchainExchange).getParentSubaccount(subaccount)]
        : linkedSigners[subaccount];
}
```

The broader helper that also checks `nlpSigners` exists but is never called from `validateSignature`:

```solidity
// EndpointTx.sol L159-170
function getLinkedSignerOrNlpSigner(bytes32 subaccount) internal view virtual returns (address) {
    address linkedSigner = getLinkedSigner(subaccount);
    if (linkedSigner != address(0)) {
        return linkedSigner;
    }
    return nlpSigners[subaccount];
}
```

`getLinkedSignerOrNlpSigner` is used only for `MatchOrders` and `MatchOrdersWithAmount`:

```solidity
// EndpointTx.sol L506-510
takerLinkedSigner: getLinkedSignerOrNlpSigner(txn.taker.order.sender),
makerLinkedSigner: getLinkedSignerOrNlpSigner(txn.maker.order.sender),
```

**NLP pool subaccount structure makes the gap concrete:**

When an NLP pool is created, its subaccount is set to `NLP_POOL_ACCOUNT_START` (or an increment thereof) — a protocol-controlled sentinel value, not the owner's address. The owner is stored only in `nlpSigners[subaccount]`:

```solidity
// EndpointTx.sol L32-48
bytes32 subaccount = NLP_POOL_ACCOUNT_START;
...
nlpSigners[subaccount] = owner;
```

`linkedSigners[nlpPoolSubaccount]` is never set. Therefore, for any signed transaction routed through `validateSignedTx` → `validateSignature`:

- `address(uint160(bytes20(sender)))` = protocol sentinel address (not the owner)
- `getLinkedSigner(sender)` = `address(0)` (not set)
- `nlpSigners[sender]` = owner address (correct, but never consulted)

`Verifier.validateSignature` then requires the recovered address to equal either the embedded address or the linked signer:

```solidity
// Verifier.sol L297-303
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
```

The NLP pool owner's signature will always fail this check for `WithdrawCollateral`, `WithdrawCollateralV2`, `LinkSigner`, `TransferQuote`, and `LiquidateSubaccount` when the sender is an NLP pool subaccount.

### Impact Explanation

**Collateral lock in NLP pool subaccounts.** If an NLP pool subaccount accumulates collateral (e.g., from trading activity or residual balances after `deleteNlpPool` calls `clearNlpPoolPosition` but does not drain all assets), the NLP pool owner cannot sign a `WithdrawCollateral` or `WithdrawCollateralV2` transaction for that subaccount. The `NlpProfitShare` mechanism is a separate, sequencer-gated path and does not substitute for a direct withdrawal. Collateral can become permanently inaccessible to the owner.

**Blocked `LinkSigner` for NLP pool subaccounts.** The NLP pool owner cannot delegate signing authority to another address for their pool subaccount, preventing operational key rotation.

**Blocked `TransferQuote`.** The NLP pool owner cannot transfer quote tokens out of the NLP pool subaccount via the signed path.

**Blocked `LiquidateSubaccount`.** The NLP pool subaccount cannot act as a liquidator (earning liquidation fees) because the owner's signature is rejected.

Severity: **Medium** — direct asset impact (collateral lock) is realistic when pools are deleted or accumulate residual balances; the broken invariant is proven by the existence of `getLinkedSignerOrNlpSigner` and its correct use in `MatchOrders`.

### Likelihood Explanation

NLP pools are created and deleted by the protocol owner via slow-mode transactions. Any NLP pool that is deleted or that accumulates collateral outside the `NlpProfitShare` flow will leave the pool owner unable to recover assets. The sequencer must include the signed transaction, but the owner's inability to produce a valid signature means the sequencer cannot include it even if it wanted to. The trigger is reachable whenever an NLP pool subaccount holds a non-zero balance and the owner attempts a direct withdrawal.

### Recommendation

Replace `getLinkedSigner` with `getLinkedSignerOrNlpSigner` inside `validateSignature` and `validateCompactSignature`:

```solidity
// EndpointTx.sol L178-183 — proposed fix
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSignerOrNlpSigner(sender) : address(0),
    digest,
    signature
);

// EndpointTx.sol L192-197 — proposed fix
verifier.validateCompactSignature(
    sender,
    allowLinkedSigner ? getLinkedSignerOrNlpSigner(sender) : address(0),
    digest,
    signature
);
```

This mirrors the existing correct behavior already applied to `MatchOrders` and `MatchOrdersWithAmount`, and ensures NLP pool owners are recognized as authorized signers across all signed transaction types.

### Proof of Concept

1. Protocol owner creates NLP pool via slow-mode `AddNlpPool` with `owner = ALICE`. `nlpSigners[NLP_POOL_ACCOUNT_START] = ALICE` is set; `linkedSigners[NLP_POOL_ACCOUNT_START]` remains `address(0)`.
2. The NLP pool subaccount accumulates collateral through trading.
3. Protocol owner later calls `DeleteNlpPool` via slow mode. `clearNlpPoolPosition` is called but residual collateral remains.
4. ALICE constructs a valid EIP-712 `WithdrawCollateral` transaction with `sender = NLP_POOL_ACCOUNT_START` and signs it with her key.
5. Sequencer includes the transaction. `processTransactionImpl` routes to `validateSignedTx(NLP_POOL_ACCOUNT_START, ...)`.
6. `validateSignature` calls `getLinkedSigner(NLP_POOL_ACCOUNT_START)` → returns `address(0)`.
7. `Verifier.validateSignature` checks: `recovered == address(uint160(bytes20(NLP_POOL_ACCOUNT_START)))` (false — sentinel address) OR `recovered == address(0)` (false). Reverts with `ERR_INVALID_SIGNATURE`.
8. ALICE's collateral is permanently locked; no alternative withdrawal path exists.
