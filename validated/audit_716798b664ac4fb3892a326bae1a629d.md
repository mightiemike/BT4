Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of end-user, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` passes `msg.sender` (the router contract) as the `sender` argument to `_beforeSwap`, which forwards it unchanged to `SwapAllowlistExtension.beforeSwap`. The extension gates on `allowedSwapper[pool][sender]`, which resolves to the router address rather than the actual end-user. Any pool admin who allowlists the router to enable router-mediated swaps for their permitted users simultaneously opens the pool to every address on the network.

## Finding Description
`MetricOmmPool.swap` at line 230–240 calls `_beforeSwap(msg.sender, ...)`, where `msg.sender` is the router when a user routes through `MetricOmmSimpleRouter`. [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that `sender` value unchanged to every configured extension via `_callExtensionsInOrder`. [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool (correct) and `sender` is the address the pool forwarded — the router, not the end-user. [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly with no mechanism to encode the original `msg.sender` into the call — the router becomes `msg.sender` to the pool. [4](#0-3) 

The same misbinding applies to `exactInput` (line 104), and any other router path that calls `pool.swap` directly. The only way to make router-mediated swaps work for allowlisted users is to allowlist the router itself, which then grants every user on the network unrestricted swap access to the pool.

## Impact Explanation
`SwapAllowlistExtension` is the sole on-chain mechanism for restricting swap access (e.g., KYC/AML-gated or institutional-only pools). When the router is allowlisted, the per-user gate is entirely bypassed: any unpermissioned user can execute swaps against a pool configured to reject them. This breaks the core access-control invariant the extension enforces and allows unauthorized parties to interact with pools designed to serve only vetted counterparties, constituting a direct breach of the pool's intended access boundary and potential loss of LP assets.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard publicly deployed periphery. Any user wanting to swap will naturally use it. A pool admin deploying `SwapAllowlistExtension` who wants router-mediated swaps to work for their allowlisted users **must** allowlist the router — there is no other option in the current architecture. Once the router is allowlisted, the bypass is trivially reachable by any address with no special privileges, no malicious setup, and no non-standard tokens.

## Recommendation
The pool must forward the original end-user identity to the extension, not the intermediary's address. The simplest correct fix is for the router to encode the real user address in `extensionData` (analogous to how the payer is stored via `_setNextCallbackContext` for the callback), and for `SwapAllowlistExtension.beforeSwap` to decode and check it when the caller is a known trusted router, falling back to `sender` for direct pool calls.

## Proof of Concept
1. Deploy pool P with `SwapAllowlistExtension` E in `BEFORE_SWAP_ORDER`.
2. Admin calls `E.setAllowedToSwap(P, alice, true)` — only Alice is permitted.
3. Admin calls `E.setAllowedToSwap(P, router, true)` — required for Alice to use the router.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: P, ...})`.
5. Router calls `P.swap(recipient, ...)` with `msg.sender = router`.
6. Pool calls `E.beforeSwap(router, recipient, ...)`.
7. Extension evaluates `allowedSwapper[P][router]` → `true`.
8. Bob's swap executes successfully despite not being on the allowlist.
