Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating caller, allowing any address to bypass per-pool swap restrictions via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of `MetricOmmPool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. Allowlisting the router — the only way to make router-mediated swaps functional on a restricted pool — silently grants every caller of the router unconditional access, completely defeating the allowlist.

## Finding Description
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool and checks it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (the extension caller). `sender` is whatever `MetricOmmPool.swap()` received as its own `msg.sender`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // pool's msg.sender = whoever called pool.swap()
    ...
```

Every public entry point in `MetricOmmSimpleRouter` calls `pool.swap()` directly from the router contract, so the pool always sees `msg.sender = router`:

- `exactInputSingle` (L72): `IMetricOmmPoolActions(params.pool).swap(...)` — router is caller
- `exactInput` (L104): `IMetricOmmPoolActions(pool).swap(...)` — router is caller
- `exactOutputSingle` (L136): `IMetricOmmPoolActions(params.pool).swap(...)` — router is caller
- `exactOutput` (L165): `IMetricOmmPoolActions(pool).swap(...)` — router is caller
- `_exactOutputIterateCallback` (L220): `IMetricOmmPoolActions(pool).swap(...)` — router is caller

The router stores the original `msg.sender` only in transient storage for payment purposes (`_setNextCallbackContext`), but never forwards it to the pool or extension. There is no mechanism to pass the originating caller's identity through `extensionData` or any other channel.

For a restricted pool to be usable via the router at all, the admin must call `setAllowedToSwap(pool, address(router), true)`. Once done, `allowedSwapper[pool][router] == true`, and every call through the router — regardless of who the original `msg.sender` is — passes the check unconditionally.

## Impact Explanation
`SwapAllowlistExtension` is the protocol's primary mechanism for restricting swap access (KYC-gated pools, institutional-only pools, restricted counterparty pools). When the router is allowlisted, the access-control invariant is completely broken for all router callers. Any address can execute swaps on a nominally restricted pool, enabling unauthorized trades and potential fund loss for LPs who deposited under the assumption that only vetted counterparties could trade. This is a direct, fund-impacting bypass of a core access-control mechanism.

## Likelihood Explanation
The scenario is highly likely in practice. Pool admins who configure `SwapAllowlistExtension` and also want allowlisted users to benefit from the router (slippage protection, multi-hop routing, deadline enforcement) will naturally allowlist the router alongside individual users. No privileged attacker capability is required — any EOA can call the public router functions. The bypass is silent and leaves no on-chain signal that the allowlist has been circumvented.

## Recommendation
The extension must gate the **original user**, not the intermediary. Two complementary approaches:

1. **Pass the original caller through the router**: Add an `originSender` field to `extensionData` that the router populates with `msg.sender` before calling the pool. The extension reads and verifies this field only when `msg.sender` (the pool's caller, i.e., the router) is a registered trusted forwarder. The extension must validate `allowedSwapper[pool][trustedRouter]` AND `allowedSwapper[pool][originSender]` before granting access.

2. **Trusted-router registry in the extension**: The extension maintains a set of trusted routers. When `sender` is a trusted router, the extension reads the actual user identity from `extensionData`; otherwise it checks `sender` directly. This avoids granting blanket access to the router address.

Either approach must ensure the extension never grants access based solely on the router's address without also verifying the end user's identity.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension.
  - setAllowedToSwap(pool, alice, true)   // alice is the intended allowlisted user
  - setAllowedToSwap(pool, router, true)  // router allowlisted so alice can use it

Attack:
  - bob (not allowlisted) calls router.exactInputSingle({pool: pool, ...})
  - Router calls pool.swap(...); pool's msg.sender = router
  - _beforeSwap receives sender = router
  - allowedSwapper[pool][router] == true → check passes
  - Bob's swap executes on the restricted pool

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds
```
