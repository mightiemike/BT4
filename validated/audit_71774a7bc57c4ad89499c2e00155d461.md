### Title
Order Digest Domain Separator Binds to `productId` Instead of `address(this)`, Enabling Cross-Deployment Signature Replay — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.getDigest()` manually constructs an EIP-712 domain separator that uses `address(uint160(productId))` as the `verifyingContract` field instead of `address(this)`. Because the verifying contract address is a tiny pseudo-address derived from the product ID (e.g., `0x0000…0001`), any two Nado deployments on the same chain sharing the same product IDs will produce identical order digests for identical order structs. A signed order from one deployment is therefore cryptographically valid on any other deployment, enabling cross-deployment replay.

---

### Finding Description

`getDigest` in `OffchainExchange.sol` manually builds the EIP-712 domain separator:

```solidity
bytes32 domainSeparator = keccak256(
    abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),
        _EIP712VersionHash(),
        block.chainid,
        address(uint160(productId))   // ← productId cast to address, NOT address(this)
    )
);
``` [1](#0-0) 

The `_TYPE_HASH` is declared as the standard EIP-712 domain type that includes a `verifyingContract` field:

```solidity
bytes32 private constant _TYPE_HASH =
    keccak256(
        "EIP712Domain(string name,string version,uint256 chainId,address verifyingContract)"
    );
``` [2](#0-1) 

By substituting `address(uint160(productId))` for `verifyingContract`, the domain separator is bound to a deterministic tiny address (e.g., `0x0000…0002` for productId 2) rather than to the actual `OffchainExchange` contract instance. Every Nado deployment that lists the same productId will compute the same domain separator and therefore the same final digest for any given order struct.

This contrasts with how `EndpointTx` handles user transaction signatures — it correctly calls `_hashTypedDataV4(computeDigest(...))`, which uses the inherited `EIP712Upgradeable._hashTypedDataV4` that embeds `address(this)` (the proxy address) in the domain separator: [3](#0-2) 

Order digests are the only replay-protection mechanism for `OffchainExchange` orders; the `filledAmounts` mapping is keyed by digest and is deployment-local: [4](#0-3) 

---

### Impact Explanation

**Impact: High**

If a second Nado deployment exists on the same chain (e.g., a v2 upgrade deployed at a new proxy address, a parallel pool deployment, or a shadow instance), a sequencer or attacker controlling transaction submission to that second deployment can replay any previously signed order from the first deployment verbatim. The replayed order passes `_checkSignature` because the digest is identical, and `filledAmounts` on the new deployment starts at zero, so the full order amount can be filled again. This results in unauthorized trades executed against the victim's subaccount on the second deployment — draining collateral or creating unwanted positions without any new user signature. [5](#0-4) 

---

### Likelihood Explanation

**Likelihood: Low**

Exploitation requires two Nado deployments on the same chain sharing overlapping product IDs. Given the protocol's upgradeable proxy architecture, a single canonical deployment is the norm. However, the protocol explicitly supports Ink Sepolia and Ink Mainnet, and a v2 proxy migration or a parallel NLP pool deployment would create the necessary conditions. The signed order bytes are public once submitted on-chain, so no privileged access is needed beyond the ability to submit transactions to the second deployment.

---

### Recommendation

Replace `address(uint160(productId))` with `address(this)` in the domain separator construction inside `getDigest`, binding each order signature to the specific `OffchainExchange` contract instance:

```solidity
bytes32 domainSeparator = keccak256(
    abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),
        _EIP712VersionHash(),
        block.chainid,
        address(this)   // bind to this contract instance
    )
);
```

Alternatively, use the inherited `_domainSeparatorV4()` from `EIP712Upgradeable` directly, which already encodes `address(this)` correctly, consistent with how `EndpointTx` handles all other signed transaction types.

---

### Proof of Concept

1. User signs an `Order` for `productId = 2` on Nado deployment A (proxy at `0xAAAA…`). The domain separator's `verifyingContract` field resolves to `address(uint160(2))` = `0x0000…0002`.
2. The order is matched and fully filled on deployment A; `filledAmounts[digest]` is set to the full amount.
3. Nado deploys a v2 proxy at `0xBBBB…` (new address, same product IDs).
4. Attacker submits the identical signed order bytes to deployment B's sequencer pipeline.
5. Deployment B calls `getDigest(2, order)` — it computes the same domain separator (`verifyingContract = 0x0000…0002`) and the same final digest.
6. `_checkSignature` passes (same digest, same user signature). `filledAmounts[digest]` on deployment B is zero.
7. The order is filled in full on deployment B, executing an unauthorized trade against the user's subaccount. [6](#0-5)

    function getDigest(uint32 productId, IEndpoint.Order memory order)
        public
        view
        returns (bytes32)
    {
        string
            memory structType = "Order(bytes32 sender,int128 priceX18,int128 amount,uint64 expiration,uint64 nonce,uint128 appendix)";

        bytes32 structHash = keccak256(
            abi.encode(
                keccak256(bytes(structType)),
                order.sender,
                order.priceX18,
                order.amount,
                order.expiration,
                order.nonce,
                order.appendix
            )
        );

        bytes32 domainSeparator = keccak256(
            abi.encode(
                _TYPE_HASH,
                _EIP712NameHash(),
                _EIP712VersionHash(),
                block.chainid,
                address(uint160(productId))
            )
        );

        return ECDSAUpgradeable.toTypedDataHash(domainSeparator, structHash);
    }
```

    function _checkSignature(
        bytes32 subaccount,
        bytes32 digest,
        address linkedSigner,
        bytes memory signature
    ) internal view virtual returns (bool) {
        address signer = ECDSA.recover(digest, signature);
        return
            (signer != address(0)) &&
            (signer == address(uint160(bytes20(subaccount))) ||
                signer == linkedSigner);
    }
