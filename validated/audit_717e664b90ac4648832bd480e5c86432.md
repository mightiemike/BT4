### Title
ETH Permanently Locked in `DirectDepositV1` — `withdrawNative()` Always Reverts When Called via `ContractOwner` (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.withdrawNative()` sends ETH to `msg.sender` via `call{value:}`. When invoked through `ContractOwner.withdrawFromDirectDepositV1()`, `msg.sender` is the `ContractOwner` contract, which has no `receive()` or `fallback()` function. The ETH transfer therefore always fails, permanently locking any native ETH that accumulates in a `DirectDepositV1` instance.

---

### Finding Description

`DirectDepositV1.withdrawNative()` is the sole recovery path for native ETH held in a `DirectDepositV1` (DDA) contract: [1](#0-0) 

```solidity
function withdrawNative() external onlyOwner {
    uint256 balance = address(this).balance;
    (bool success, ) = msg.sender.call{value: balance}("");
    require(success, "Failed to transfer native token to owner");
}
```

The only caller of this function in production is `ContractOwner.withdrawFromDirectDepositV1()`: [2](#0-1) 

```solidity
if (token == address(0)) {
    uint256 preBalance = address(this).balance;
    DirectDepositV1(directDepositV1).withdrawNative();   // <-- msg.sender = ContractOwner
    uint256 postBalance = address(this).balance;
    require(postBalance > preBalance, "empty");
    (bool success, ) = msg.sender.call{value: postBalance - preBalance}("");
    require(success, "xfer");
}
```

`ContractOwner` inherits only from `EIP712Upgradeable` and `OwnableUpgradeable` and defines **no `receive()` or `fallback()` function**: [3](#0-2) 

When `withdrawNative()` executes `msg.sender.call{value: balance}("")`, the callee (`ContractOwner`) cannot accept ETH, so `success = false`, and the `require` reverts. The entire `withdrawFromDirectDepositV1` call reverts with it.

ETH can accumulate in a DDA in the constructor when the initial wrapping attempt fails silently: [4](#0-3) 

```solidity
uint256 balance = address(this).balance;
if (balance != 0) {
    // shouldn't revert even if the transfer fails, otherwise the funds
    // will be stuck in the DDA forever.
    (bool success, ) = wrappedNative.call{value: balance}("");
    if (!success) {
        emit NativeTokenTransferFailed(balance);
    }
}
```

The developer comment explicitly acknowledges the risk of ETH being stuck. However, the intended recovery path (`withdrawNative` → `ContractOwner`) is itself broken for the reason above.

---

### Impact Explanation

Any native ETH that ends up in a `DirectDepositV1` contract — whether from a failed constructor wrap or any other path — is **permanently unrecoverable**. The only withdrawal function (`withdrawNative`) always reverts when called through `ContractOwner` because `ContractOwner` cannot receive ETH. The ETH is locked with no alternative exit.

---

### Likelihood Explanation

`ContractOwner.creditDepositV1()` is callable by any unprivileged user and will deploy a new `DirectDepositV1` if one does not exist for the given subaccount: [5](#0-4) 

If the `wrappedNative` contract is temporarily paused, rate-limited, or otherwise unavailable at the moment of DDA construction, the constructor's wrapping call fails silently and ETH remains in the DDA. The developer comment confirms this scenario was anticipated. Once ETH is stuck, the broken `withdrawNative` path means it can never be recovered.

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept ETH forwarded from `withdrawNative()`:

```solidity
receive() external payable {}
```

Alternatively, refactor `withdrawNative()` to accept an explicit `recipient` parameter rather than using `msg.sender`, so the ETH can be sent directly to the intended destination without routing through `ContractOwner`:

```solidity
function withdrawNative(address payable recipient) external onlyOwner {
    uint256 balance = address(this).balance;
    (bool success, ) = recipient.call{value: balance}("");
    require(success, "Failed to transfer native token");
}
```

---

### Proof of Concept

1. Any user calls `ContractOwner.creditDepositV1(subaccount)`.
2. A new `DirectDepositV1` is deployed. If `wrappedNative.call{value:}` fails in the constructor, ETH remains in the DDA and `NativeTokenTransferFailed` is emitted.
3. The multisig (owner) calls `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))` to recover the ETH.
4. Internally, `DirectDepositV1.withdrawNative()` executes `ContractOwner.call{value: balance}("")`.
5. `ContractOwner` has no `receive()` function → `success = false` → `require(success, "Failed to transfer native token to owner")` reverts.
6. The ETH is permanently locked in the `DirectDepositV1` contract with no recovery path. [1](#0-0) [6](#0-5)

    function withdrawFromDirectDepositV1(bytes32 subaccount, address token)
        external
        onlyOwner
    {
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        if (token == address(0)) {
            uint256 preBalance = address(this).balance;
            DirectDepositV1(directDepositV1).withdrawNative();
            uint256 postBalance = address(this).balance;
            require(postBalance > preBalance, "empty");
            (bool success, ) = msg.sender.call{value: postBalance - preBalance}(
                ""
            );
            require(success, "xfer");
        } else {
            uint256 preBalance = IERC20Base(token).balanceOf(address(this));
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(token));
            uint256 postBalance = IERC20Base(token).balanceOf(address(this));
            require(postBalance > preBalance, "empty");
            IERC20Base(token).safeTransfer(
                msg.sender,
                postBalance - preBalance
            );
        }
    }
