Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the Original User, Allowing Allowlist Bypass via Router - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the direct caller of `pool.swap()`. When a user swaps through `MetricOmmSimpleRouter`, the pool receives `msg.sender = router`, so the extension checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. A pool admin who allowlists the router to enable periphery-mediated swaps for their curated pool inadvertently grants every unprivileged user a universal bypass key.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as `sender` in the ABI-encoded extension call: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks that exact value against the per-pool allowlist: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly with `msg.sender = router`, forwarding no user identity: [4](#0-3) 

The same applies to `exactInput` (L99â€“118), `exactOutputSingle` (L130â€“147), and the recursive `_exactOutputIterateCallback` path (L201â€“233), all of which call `pool.swap()` with `msg.sender = router`. [5](#0-4) 

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates on the `owner` argument â€” the actual position owner passed through unchanged regardless of who calls `addLiquidity` â€” not on the raw caller: [6](#0-5) 

No existing guard in the extension, pool, or router recovers the original user identity. The `extensionData` field is passed through but the extension does not read it.

## Impact Explanation
Any user can bypass the swap allowlist on a curated pool by routing through `MetricOmmSimpleRouter` whenever the router address is allowlisted. The pool admin's intent â€” restricting swaps to a specific set of addresses (e.g., KYC'd counterparties, whitelisted market participants) â€” is completely defeated. Non-allowlisted users can execute swaps at oracle-anchored prices, draining LP liquidity at the pool's bid/ask spread, constituting a direct loss of LP principal and protocol fees on a pool explicitly configured to prevent open access. This is a direct loss of user principal meeting Critical/High Sherlock thresholds.

## Likelihood Explanation
The scenario requires the pool admin to allowlist the router. This is a natural and expected administrative action: a pool admin who deploys a curated pool and wants their allowlisted users to use the standard periphery router will add the router to `allowedSwapper`. The admin's mental model is that the router is a trusted intermediary that passes through user identity; the actual behavior is that the router becomes a universal bypass key. The router is a public, permissionless contract, so once it is allowlisted, any user can exploit the bypass without any further privileged action.

## Recommendation
Gate `SwapAllowlistExtension` on the economically relevant actor â€” the original user â€” not the direct caller of `pool.swap()`. Two complementary fixes:

1. **Pass the original user through the router**: `MetricOmmSimpleRouter` should forward `msg.sender` as a verified field in `extensionData`, and the extension should read that value after verifying it was set by a trusted router (e.g., via a registry of trusted routers held in the extension or factory).

2. **Mirror the `DepositAllowlistExtension` pattern**: Introduce a per-swap `owner`-equivalent (e.g., a router-attested user address in `extensionData`) and gate on that value rather than on the raw `sender`.
