### Title
No Slippage Protection on NLP Mint and Burn — (`core/contracts/Clearinghouse.sol`, `core/contracts/interfaces/IEndpoint.sol`, `core/contracts/Verifier.sol`)

---

### Summary

The `MintNlp` and `BurnNlp` user-signed structs contain no minimum output field. The `oraclePriceX18` that determines the actual NLP-to-quote conversion rate is provided by the sequencer at execution time and is **not covered by the user's EIP-712 signature**. In volatile conditions, a user can receive significantly fewer NLP tokens (on mint) or significantly less quote (on burn) than they expected when they signed the transaction, with no on-chain protection.

---

### Finding Description

When a user signs a `MintNlp` transaction, the EIP-712 digest is computed over only three fields: `sender`, `quoteAmount`, and `nonce`. [1](#0-0) 

The `oraclePriceX18` field present in `SignedMintNlp` is **outside** the signed struct and is not committed to by the user: [2](#0-1) 

The same pattern applies to `BurnNlp` — the user signs only `sender`, `nlpAmount`, `nonce`, while `oraclePriceX18` is sequencer-supplied: [3](#0-2) [4](#0-3) 

At execution time in `Clearinghouse.mintNlp`, the NLP amount minted is computed as:

```
nlpAmount = quoteAmount / oraclePriceX18
``` [5](#0-4) 

And in `Clearinghouse.burnNlp`, the quote returned is:

```
quoteAmount = nlpAmount * oraclePriceX18 - burnFee
``` [6](#0-5) 

There is no `minNlpAmount` check in `mintNlp` and no `minQuoteAmount` check in `burnNlp`. The sequencer provides `oraclePriceX18` at batch submission time, which reflects the NLP price at that moment — not the price at the time the user signed.

---

### Impact Explanation

**MintNlp**: A user signs a transaction to spend `quoteAmount` of quote tokens expecting to receive NLP at the price they observed. If the NLP oracle price rises between signing and sequencer execution, `nlpAmount = quoteAmount / oraclePriceX18` yields fewer NLP tokens than expected. The user has no on-chain recourse.

**BurnNlp**: A user signs a transaction to burn `nlpAmount` of NLP tokens expecting to receive quote at the price they observed. If the NLP oracle price falls between signing and execution, `quoteAmount = nlpAmount * oraclePriceX18 - burnFee` yields less quote than expected. The user has no on-chain recourse.

In both cases the user's balance is permanently reduced relative to their expectation, with no slippage guard to revert the transaction.

---

### Likelihood Explanation

NLP price is derived from the aggregate value of the multi-asset liquidity pool. During periods of market volatility — large perp PnL swings, rapid collateral price moves, or large NLP pool rebalances — the NLP oracle price can shift materially between the moment a user signs a transaction and the moment the sequencer includes it in a batch. This is a normal operating condition, not an attack scenario. The sequencer is honest but the price moves.

---

### Recommendation

Add a `minNlpAmount` field to the `MintNlp` struct and a `minQuoteAmount` field to the `BurnNlp` struct. Include these fields in the EIP-712 signed digest so users commit to a minimum acceptable output. In `Clearinghouse.mintNlp` and `Clearinghouse.burnNlp`, revert if the computed output falls below the user-specified minimum.

```diff
// IEndpoint.sol
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;
+   uint128 minNlpAmount;
    uint64 nonce;
}

struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
+   uint128 minQuoteAmount;
    uint64 nonce;
}
```

```diff
// Clearinghouse.sol — mintNlp
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
+ require(nlpAmount >= int128(txn.minNlpAmount), ERR_SLIPPAGE_TOO_HIGH);

// Clearinghouse.sol — burnNlp
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
+ require(quoteAmount >= int128(txn.minQuoteAmount), ERR_SLIPPAGE_TOO_HIGH);
```

The `Verifier.sol` EIP-712 type strings and digest computation must also be updated to include the new fields.

---

### Proof of Concept

1. NLP oracle price is `1.00 USDC/NLP` at time T₀. User signs `MintNlp { quoteAmount: 1000e18, nonce: N }` expecting ~1000 NLP.
2. Between T₀ and T₁ (sequencer batch submission), a large trader's perp position closes profitably, increasing the NLP pool value. NLP oracle price updates to `1.10 USDC/NLP`.
3. Sequencer submits the batch with `oraclePriceX18 = 1.10e18`.
4. `Clearinghouse.mintNlp` computes `nlpAmount = 1000e18 / 1.10e18 ≈ 909 NLP`.
5. User receives ~909 NLP instead of ~1000 NLP — a ~9% loss — with no revert or protection. [7](#0-6) [8](#0-7)
