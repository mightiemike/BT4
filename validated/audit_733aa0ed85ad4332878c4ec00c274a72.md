### Title
Forged Internal Merkle Node Accepted as Coinbase in `verify_transaction_inclusion_v2`, Nullifying 64-Byte Attack Mitigation — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` is supposed to prevent the 64-byte Merkle proof forgery attack by requiring a coinbase proof at position 0. However, the coinbase check only verifies that `compute_root_from_merkle_proof(coinbase_tx_id, 0, coinbase_merkle_proof) == merkle_root`. It does **not** verify that `coinbase_tx_id` is the actual coinbase transaction (the real leaf at position 0). An attacker can supply an internal Merkle node as `coinbase_tx_id` with a proof one level shorter than the real tree depth, satisfying the coinbase check. Because `merkle_proof` must equal `coinbase_merkle_proof` in length, the attacker also provides a same-length `merkle_proof` for `tx_id` set to the same internal node. Both checks pass and the function returns `true` for a `tx_id` that is an internal node — not a real transaction — completely defeating the mitigation.

---

### Finding Description

The coinbase guard in `verify_transaction_inclusion_v2` is:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
``` [1](#0-0) 

`compute_root_from_merkle_proof` is a pure hash-chain computation with no constraint on what the starting hash represents:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;
    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }
    current_hash
}
``` [2](#0-1) 

For a 4-tx block with leaves `tx[0..3]`, define:
- `N01 = hash(tx[0], tx[1])` (level-1 internal node)
- `N23 = hash(tx[2], tx[3])` (level-1 internal node)
- `R   = hash(N01, N23)` (merkle root)

The attacker calls `verify_transaction_inclusion_v2` with:

| Field | Value |
|---|---|
| `coinbase_tx_id` | `N01` (internal node, **not** the real coinbase `tx[0]`) |
| `coinbase_merkle_proof` | `[N23]` (length 1) |
| `tx_id` | `N01` |
| `tx_index` | `0` |
| `merkle_proof` | `[N23]` (length 1) |

**Step 1** — length check: `1 == 1` ✓ [3](#0-2) 

**Step 2** — coinbase check: `compute_root_from_merkle_proof(N01, 0, [N23]) = hash(N01, N23) = R == merkle_root` ✓ [1](#0-0) 

**Step 3** — `verify_transaction_inclusion` with `tx_id=N01, tx_index=0, merkle_proof=[N23]`: `compute_root_from_merkle_proof(N01, 0, [N23]) = R == merkle_root` → returns `true` ✓ [4](#0-3) 

The function returns `true` for `tx_id = N01`, an internal node. The real coinbase `tx[0]` is never verified.

---

### Impact Explanation

The 64-byte attack mitigation is entirely nullified. An attacker who knows the Merkle tree of any canonical block (all data is public on-chain) can claim that any internal node `N` is a "transaction" included in that block, and `verify_transaction_inclusion_v2` will confirm it. Any downstream contract or application that calls this function to gate value transfers, bridge withdrawals, or cross-chain state transitions can be deceived with a forged inclusion proof for a non-existent transaction.

---

### Likelihood Explanation

The attack requires no privileges, no private keys, and no special chain state. The Merkle tree structure of every block is public. Any unprivileged NEAR account can call `verify_transaction_inclusion_v2` directly. The construction is deterministic and requires only arithmetic on known public values.

---

### Recommendation

The coinbase check must verify that `coinbase_tx_id` is the **actual** coinbase transaction at position 0, not merely that some hash chain from it reaches the root. The correct fix is to enforce that the proof length matches the expected tree depth. Concretely:

1. Require `coinbase_merkle_proof.len() >= 1` (already done implicitly via the non-empty check in `verify_transaction_inclusion`).
2. Require that `coinbase_merkle_proof.len() == merkle_proof.len()` **and** that the proof length equals `ceil(log2(tx_count))` for the block — i.e., the full depth of the tree. This prevents using a shorter proof rooted at an internal node.
3. Alternatively, store the coinbase txid in the block header metadata and compare `coinbase_tx_id` directly against the stored value, rather than re-deriving it from a caller-supplied proof.

---

### Proof of Concept

For a canonical 4-tx block with known `tx[0..3]`:

```rust
// Precompute internal nodes (all public from block data)
let n01 = double_sha256([tx[0], tx[1]].concat());  // hash(tx[0], tx[1])
let n23 = double_sha256([tx[2], tx[3]].concat());  // hash(tx[2], tx[3])
// root = double_sha256([n01, n23].concat()) == block.merkle_root

verify_transaction_inclusion_v2(ProofArgsV2 {
    tx_id:                 n01,    // internal node, not a real tx
    tx_block_blockhash:    block_hash,
    tx_index:              0,
    merkle_proof:          vec![n23],   // length 1, not the real depth-2 proof
    coinbase_tx_id:        n01,    // same internal node
    coinbase_merkle_proof: vec![n23],   // length 1
    confirmations:         0,
})
// → returns true
// Invariant violated: coinbase_tx_id is not the actual coinbase transaction tx[0]
```

The coinbase check at lines 358–364 passes because `hash(n01, n23) == merkle_root`. The tx inclusion check also passes for the same reason. The real coinbase `tx[0]` is never touched. [5](#0-4) [2](#0-1)
