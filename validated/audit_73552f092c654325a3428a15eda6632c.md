Audit Report

## Title
SwapAllowlistExtension checks router identity instead of original user when swaps are routed through MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by the `sender` argument, which `MetricOmmPool.swap` sets to its own `msg.sender`. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the originating user. The extension therefore checks whether the **router** is allowlisted, not whether the **user** is allowlisted. If a pool admin allowlists the router — a natural step to support router-based swaps for curated users — any non-allowlisted address can trade on the restricted pool by routing through the router.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← direct caller of pool.swap(), not the end-user
  recipient, ...
);
```

`ExtensionCalling._beforeSwap` forwards that `sender` verbatim to the extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))` (L162-175).

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When `MetricOmmSimpleRouter.exactInputSingle` is used, it calls `pool.swap()` directly (L72-80) without forwarding the original `msg.sender`, so `sender = router`. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][original_user]`.

**Bypass table:**

| Router allowlisted? | Non-allowlisted user via router | Allowlisted user via router |
|---|---|---|
| No | Blocked | Also blocked |
| **Yes** | **Passes — bypass** | Passes |

A pool admin who wants their curated users to be able to use the standard periphery router will naturally call `setAllowedToSwap(pool, router, true)`. Doing so silently opens the pool to every address that routes through the router, defeating per-user curation entirely. There is no indication in the extension's interface or documentation that this configuration grants universal access.

## Impact Explanation
The per-user swap allowlist — the sole access-control mechanism for curated pools (e.g., KYC-gated, institutional-only) — is completely bypassed for any user who routes through `MetricOmmSimpleRouter` when the router is allowlisted. Any non-allowlisted address can execute swaps on a pool that was intended to be restricted. This is a broken core pool functionality: the access-control invariant enforced by `SwapAllowlistExtension` does not hold on the router path, matching the "Allowlist path" smart audit pivot (bypass through router).

## Likelihood Explanation
Likelihood is **medium**. The bypass requires the pool admin to have allowlisted the router. This is a natural and expected configuration for any pool that wants to support both direct and router-based swaps for its allowlisted users. The admin has no indication from the extension's interface or documentation that doing so opens the pool to all users. The condition is realistic and repeatable for any pool using this extension with router support.

## Recommendation
`SwapAllowlistExtension` must not rely solely on the immediate `sender` (the direct caller of `pool.swap()`). Options:
1. Have the router write the original `msg.sender` into a trusted transient storage slot before calling `pool.swap()`, and have the extension read that slot to verify the end-user identity.
2. Require callers to pass the original user address through `extensionData` and verify it in the hook (requires router cooperation and trust in the data).
3. Document explicitly that allowlisting the router grants access to all users, and provide a separate `RouterSwapAllowlistExtension` that reads the payer from transient storage set by the router.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured as a `beforeSwap` extension.
2. Admin calls `setAllowedToSwap(pool, alice, true)` (alice is the intended curated user) and `setAllowedToSwap(pool, router, true)` (to support router-based swaps for alice).
3. `bob` (not individually allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient, ...)` — pool's `msg.sender` is the router.
5. Pool calls `_beforeSwap(sender=router, ...)` → extension receives `sender=router`.
6. Extension checks `allowedSwapper[pool][router] == true` → passes.
7. Bob's swap executes on the allowlist-gated pool despite not being individually allowlisted.
