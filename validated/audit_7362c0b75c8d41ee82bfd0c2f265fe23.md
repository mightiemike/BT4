### Title
Precommit Vote Signature Preimage Omits `chain_id`, `height`, and `round`, Enabling Cross-Context Replay — (File: `crates/apollo_signature_manager/src/signature_manager.rs`)

---

### Summary

`build_precommit_vote_message_digest` constructs the ECDSA signing preimage as `PRECOMMIT_VOTE || block_hash` only. It omits `chain_id`, consensus `height`, and `round`. A valid precommit signature produced for `block_hash X` at height `H`, round `R` on chain `SN_MAIN` is byte-for-byte identical to a valid precommit for `block_hash X` at any other height, round, or chain. This is the direct Sequencer analog of the KeyRegistry finding: just as the removal signature omitted `fid` and could be replayed against a different `fid`, the precommit signature omits the consensus context and can be replayed in a different context.

---

### Finding Description

`SignatureManager::sign_precommit_vote` calls `build_precommit_vote_message_digest(block_hash)`:

```rust
fn build_precommit_vote_message_digest(block_hash: BlockHash) -> MessageDigest {
    let block_hash = block_hash.to_bytes_be();
    let mut message = Vec::with_capacity(PRECOMMIT_VOTE.len() + block_hash.len());
    message.extend_from_slice(PRECOMMIT_VOTE);   // b"PRECOMMIT_VOTE"
    message.extend_from_slice(&block_hash);       // 32-byte block hash only
    MessageDigest(blake2s_to_felt(&message))
}
```

The `Vote` struct that carries this signature over the network contains `height`, `round`, `proposal_commitment` (= `block_hash`), `voter`, and `signature`. None of `height`, `round`, `chain_id`, or `voter` are committed into the signature preimage. The signed digest is purely a function of the `block_hash`.

The Tendermint-like consensus protocol explicitly supports reproposing the same locked value in later rounds (`SMRequest::Repropose`). When a validator reproposed `block_hash X` from round 0 in round 1, the `block_hash` is identical. A precommit signature produced for `(height=H, round=0, block_hash=X)` is therefore cryptographically valid as a precommit for `(height=H, round=1, block_hash=X)`. An adversary who observed the round-0 signature can inject it as a round-1 precommit without the validator's participation.

Additionally, the same `block_hash` can appear on a different chain (e.g., `SN_SEPOLIA` vs `SN_MAIN`) if the same block content is proposed. The signature is valid on both chains because `chain_id` is absent from the preimage.

---

### Impact Explanation

**Matching impact**: *High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.*

A precommit signature is the authoritative attestation that a validator has committed to a specific block at a specific consensus position. By omitting `height`, `round`, and `chain_id` from the preimage, the signature domain is under-specified. An attacker who captures a legitimate precommit signature for `block_hash X` can replay it:

1. **Cross-round replay (same chain, same height)**: In Tendermint, the same proposal can be reproposed in a later round. A precommit from round 0 is valid as a precommit for round 1 if the `block_hash` matches. This can contribute to a false quorum at a round the validator never actually voted in.

2. **Cross-chain replay**: A precommit signed on `SN_MAIN` is valid on `SN_SEPOLIA` (or any other chain) for the same `block_hash`. A validator's key used on mainnet can be exploited to forge consensus participation on testnet or a fork.

Both scenarios cause the consensus layer to accept a precommit attributed to a validator who did not actually cast that vote in that context, directly corrupting the consensus safety guarantee.

---

### Likelihood Explanation

**Medium.** Cross-round replay requires the same `block_hash` to be reproposed in a later round, which is a normal Tendermint protocol event (the `Repropose` path is implemented and tested). An adversary monitoring the P2P network can collect round-0 precommit signatures and replay them in round-1 without any privileged access. Cross-chain replay requires a matching `block_hash` on a second chain, which is less common but not impossible for testnets running the same software.

---

### Recommendation

Include `chain_id`, `height`, and `round` in the precommit vote signing preimage:

```rust
fn build_precommit_vote_message_digest(
    block_hash: BlockHash,
    chain_id: &ChainId,
    height: BlockNumber,
    round: Round,
) -> MessageDigest {
    let block_hash = block_hash.to_bytes_be();
    let chain_id_felt: Felt = chain_id.try_into().expect("chain_id to felt");
    let height_bytes = height.0.to_be_bytes();
    let round_bytes = round.to_be_bytes();
    let mut message = Vec::new();
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&block_hash);
    message.extend_from_slice(&chain_id_felt.to_bytes_be());
    message.extend_from_slice(&height_bytes);
    message.extend_from_slice(&round_bytes);
    MessageDigest(blake2s_to_felt(&message))
}
```

`sign_precommit_vote` and `verify_precommit_vote_signature` must be updated to accept and pass these additional fields. The `SignatureManagerRequest::SignPrecommitVote` variant must carry `chain_id`, `height`, and `round` alongside `block_hash`.

### Proof of Concept

Given the existing public API:

```rust
// Validator signs a precommit at height=5, round=0 for block_hash X
let sig = sign_precommit_vote(block_hash_X).await;

// Current digest: blake2s("PRECOMMIT_VOTE" || block_hash_X)
// This is identical regardless of height or round.

// Attacker replays the same sig as a precommit at height=5, round=1
// (after the same block_hash_X is reproposed in round 1)
let valid = verify_precommit_vote_signature(block_hash_X, sig, validator_pubkey);
// Returns true — the signature verifies correctly for the wrong round.
```

The existing test `test_sign_precommit_vote` in `signature_manager_test.rs` confirms the API accepts only `block_hash` with no context, and `verify_precommit_vote_signature` accepts only `block_hash` — no height, round, or chain_id parameter exists to prevent this replay.
