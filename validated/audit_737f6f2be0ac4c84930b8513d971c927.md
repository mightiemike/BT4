Audit Report

## Title
Swap Allowlist Bypassed via Router: `sender` Identity Is Router Address, Not End User - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against a per-pool allowlist. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` — and therefore the `sender` forwarded to the extension — is the router contract address, not the end user. If the router address is added to the allowlist (the only way to permit router-mediated swaps on a curated pool), every unprivileged user can bypass the allowlist by calling the router, completely defeating the curation policy.

## Finding Description

**Call path:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle(params)` (or `exactInput` / `exactOutputSingle` / `exactOutput`).
2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — the pool's `msg.sender` is the router address.
3. Inside `MetricOmmPool.swap`, the pool calls `_beforeSwap(msg.sender, recipient, ...)`, forwarding the router address as `sender`.
4. `ExtensionCalling._beforeSwap` dispatches to `SwapAllowlistExtension.beforeSwap(sender=router, ...)`.
5. The extension evaluates:
   ```solidity
   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
       revert IMetricOmmPoolActions.NotAllowedToSwap();
   }
   ```
   where `msg.sender` is the pool and `sender` is the **router address**.

**Root cause:** `MetricOmmPool.swap` passes `msg.sender` (the immediate caller) as the `sender` to `_beforeSwap`. When the immediate caller is the router, the extension never sees the actual end user's address.

**Bypass condition:** A pool admin who wants allowlisted users to be able to trade via the router must add the router address to `allowedSwapper[pool][router]`. Once the router is allowlisted, `allowedSwapper[pool][router] == true` for every call, so the check passes for **any** end user who routes through the router — including users who are explicitly not in the allowlist.

**Existing guards are insufficient:** The `onlyPool` modifier on the extension ensures only the pool can call `beforeSwap`, but it does not recover the original end-user identity. There is no mechanism in the pool, the router, or the extension to thread the actual `tx.origin` or a user-supplied identity through the hook in a trustworthy way.

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict trading to KYC'd or otherwise approved addresses is rendered ineffective for all router-mediated swaps. Any unprivileged user can execute swaps against the pool by calling `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point), receiving output tokens and draining pool liquidity in violation of the pool's curation policy. This constitutes a broken core pool functionality / admin-boundary break allowing unauthorized fund flows from the pool.

## Likelihood Explanation
The bypass requires only that the pool admin has allowlisted the router address — a natural and expected action for any pool that intends to support the official periphery. The attacker needs no special privileges, no flash loan, and no multi-transaction setup: a single `exactInputSingle` call suffices. The condition is repeatable across every block.

## Recommendation
The extension must gate on the actual end user, not the immediate pool caller. Two viable approaches:

1. **Pass the original caller through `extensionData`:** Have the router encode `msg.sender` (the end user) into `extensionData` and have the extension decode and verify it. This requires the pool to also enforce that the encoded address matches the callback payer, or the extension must trust the router — which reintroduces the same trust problem.

2. **Check `tx.origin` as a fallback (not recommended for general use):** Only acceptable in narrow, well-understood contexts.

3. **Preferred — router-aware allowlist:** Add a separate mapping `allowedRouterSwapper[pool][endUser]` and require the router to pass the end user's address as a verified field in `extensionData`, with the extension decoding and checking it. The pool admin would allowlist end users under this key rather than the router address itself.

Alternatively, document clearly that `SwapAllowlistExtension` is incompatible with router-mediated swaps and enforce this at the extension or factory level.

## Proof of Concept

```solidity
// Pool configured with SwapAllowlistExtension.
// Pool admin allowlists the router so that approved users can trade via router:
swapAllowlistExtension.setAllowedToSwap(pool, address(router), true);

// allowedUser is explicitly allowlisted:
swapAllowlistExtension.setAllowedToSwap(pool, allowedUser, true);

// bannedUser is NOT allowlisted.
// bannedUser calls the router directly:
vm.prank(bannedUser);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: token0,
    recipient: bannedUser,
    amountIn: 1000e18,
    amountOutMinimum: 0,
    zeroForOne: true,
    priceLimitX64: 0,
    deadline: block.timestamp + 1,
    extensionData: ""
}));
// Swap succeeds — bannedUser receives token1 despite not being allowlisted.
```

The pool's `_beforeSwap` receives `sender = address(router)`, which is allowlisted, so the check passes for `bannedUser`.
