I need to carefully trace the math through `compute_root_from_merkle_proof` for an odd-width tree to verify whether the duplicate-position claim holds.

### Title
Duplicate-Position Proof Acceptance in Odd-Width Merkle Trees — (`merkle-tools/src/lib.rs::compute_root_from_merkle_proof`)

---

### Summary

`compute_root_from_merkle_proof` performs no bounds check on `transaction_position`. For any block whose transaction count N is odd, Bitcoin's Merkle construction duplicates the last leaf. The verifier's position-parity logic then accepts the same `(tx_id, proof)` tuple for **both** the real index N-1 and the phantom index N, causing `verify_transaction_inclusion` to return `true` for a position that does not exist in the actual tree.

---

### Finding Description

**Root cause — `compute_root_from_merkle_proof`** [1](#0-0) 

The function iterates over the proof elements using only `current_position % 2` (left/right) and `current_position /= 2` (parent). It never checks that `transaction_position < tree_leaf_count`. No such count is passed in.

**Entrypoint — `verify_transaction_inclusion`** [2](#0-1) 

This public, unpermissioned (only `#[pause]`) function passes the attacker-supplied `args.tx_index` directly to `compute_root_from_merkle_proof` with no upper-bound validation. [3](#0-2) 

**Mathematical proof with a 3-leaf tree [T0, T1, T2]:**

Bitcoin duplicates the last leaf → working set becomes [T0, T1, T2, T2].

```
Level 0: [T0,  T1,  T2,  T2]
Level 1: [H(T0,T1), H(T2,T2)]
Root:    H(H(T0,T1), H(T2,T2))
```

Legitimate proof for index 2 (T2):
- pos=2 even → sibling = T2 → `H(T2,T2)`, pos→1
- pos=1 odd  → sibling = `H(T0,T1)` → Root, pos→0
- **proof = [T2, H(T0,T1)]**

Now call `compute_root_from_merkle_proof(T2, 3, [T2, H(T0,T1)])`:
- pos=3 odd  → `H(T2, T2)`, pos→1
- pos=1 odd  → `H(H(T0,T1), H(T2,T2))` = **Root** ✓

The function returns the correct root for the phantom index 3, so `verify_transaction_inclusion` returns `true` for `tx_index=3` even though only indices 0–2 exist.

**`ProofArgs.tx_index` is a caller-supplied `u64`** with no server-side ceiling. [4](#0-3) 

---

### Impact Explanation

Any downstream bridge, unlock, or mint contract that:
1. calls the deprecated but still-live `verify_transaction_inclusion`, and
2. uses `(tx_id, tx_index)` as a replay nonce

can be triggered twice for the same on-chain Bitcoin transaction: once with the real index N-1 and once with the phantom index N. This enables economic replay / double-spend of the bridged asset.

The function is deprecated but **not removed and not access-controlled** — any NEAR account can call it. [5](#0-4) 

---

### Likelihood Explanation

- The entrypoint is public with no role gate.
- Odd transaction counts are the norm in real Bitcoin blocks (coinbase + any odd number of user txs).
- The attacker needs only a valid block already in the mainchain and the legitimate proof for the last transaction — both are public information from any Bitcoin node.
- No privileged key, relayer compromise, or social engineering is required.

---

### Recommendation

1. **In `compute_root_from_merkle_proof`**: accept an explicit `tree_size: usize` parameter and `require!(transaction_position < tree_size)` before the loop.
2. **In `verify_transaction_inclusion`**: derive `tree_size` from `merkle_proof.len()` (i.e., `1usize << merkle_proof.len()`) and enforce the bound, or remove the deprecated function entirely.
3. Prefer `verify_transaction_inclusion_v2` for all new integrations; consider removing v1 in the next breaking release.

---

### Proof of Concept

```rust
// 3-tx block: [T0, T1, T2]
// Legitimate proof for T2 at index 2:
let proof = vec![T2.clone(), hash(T0, T1)];

// Real call — returns true (correct)
verify_transaction_inclusion(T2.clone(), block_hash, 2, proof.clone(), 1);

// Exploit call — also returns true (phantom index 3 does not exist)
verify_transaction_inclusion(T2.clone(), block_hash, 3, proof.clone(), 1);
// → compute_root_from_merkle_proof(T2, 3, [T2, H(T0,T1)])
//   step1: pos=3 odd  → H(T2,T2),       pos→1
//   step2: pos=1 odd  → H(H(T0,T1),H(T2,T2)) == merkle_root  ✓
```

A downstream consumer keying on `(tx_id=T2, tx_index=3)` would treat this as a distinct, unspent proof and release funds a second time.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

            .get(&self.mainchain_tip_blockhash)

            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root

            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,

    }
