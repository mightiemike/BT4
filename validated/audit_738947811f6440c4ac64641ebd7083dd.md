### Title
Uncleared ERC20 Approval in `creditDeposit()` After Silently Skipped Deposit Causes Permanent DoS for USDT-like Tokens — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` sets an ERC20 approval on the `Endpoint` contract before calling `depositCollateralWithReferral`. However, `depositCollateralWithReferral` contains an explicit silent early-return path that skips the token transfer when the deposit amount is below the minimum threshold. This leaves a non-zero residual allowance on the token. For tokens like USDT that require the allowance to be zero before a new approval can be set, any subsequent call to `creditDeposit()` for that token will permanently revert, bricking the DDA's deposit functionality for that asset.

---

### Finding Description

In `DirectDepositV1.creditDeposit()`, for each product token the contract:

1. Reads the current balance: `uint256 balance = token.balanceOf(address(this))`
2. Sets approval unconditionally: `token.approve(address(endpoint), balance)`
3. Calls: `endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1")` [1](#0-0) 

In `Endpoint.depositCollateralWithReferral()`, there is an explicit silent early-return path:

```solidity
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked when there are
    // multiple assets awaiting credit but one of them is below the minimum deposit amount.
    // we can just skip the deposit and continue with the next asset.
    return;
}
``` [2](#0-1) 

When this early return fires, `handleDepositTransfer` is never reached, meaning no `transferFrom` is executed. The approval set in step 2 is never consumed and persists as a non-zero allowance on the token contract.

On the next call to `creditDeposit()` for the same token, `token.approve(address(endpoint), balance)` is called again. For USDT-like tokens that revert when setting a non-zero approval over an existing non-zero allowance, this call reverts — permanently blocking `creditDeposit()` for that token on that DDA.

Note that `creditDeposit()` has no access control — it is callable by any external address. [3](#0-2) 

The protocol itself already demonstrates awareness of the correct pattern: `ContractOwner.wrapVaultAsset()` explicitly resets the approval to 0 before setting a new one:

```solidity
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
``` [4](#0-3) 

This safe pattern is absent in `DirectDepositV1.creditDeposit()`.

---

### Impact Explanation

For USDT-like tokens (non-zero-to-non-zero approval reverts): after a single below-minimum deposit attempt, `creditDeposit()` permanently reverts for that token on the affected DDA. The DDA cannot be re-initialized — it is deployed at a deterministic address with `salt: bytes32(uint256(1))` and the `directDepositV1Address` mapping is fixed per subaccount. [5](#0-4) 

For all tokens: residual non-zero approvals accumulate on the `Endpoint` contract, representing an unnecessary and growing attack surface.

---

### Likelihood Explanation

The early-return path in `depositCollateralWithReferral` is explicitly designed and commented as an expected code path. Any actor can trigger it by:

- Sending a dust amount of tokens to a DDA (e.g., 1 wei of a high-value token whose USD value falls below `MIN_DEPOSIT_AMOUNT`)
- Calling the permissionless `creditDeposit()` function

The `creditDeposit()` function has no access control, so any unprivileged caller can trigger this state. The minimum deposit check (`isValidDepositAmount`) depends on the oracle price, meaning a token price drop can retroactively cause a previously valid balance to fall below the threshold on the next call. [6](#0-5) 

---

### Recommendation

In `DirectDepositV1.creditDeposit()`, reset the approval to 0 after the `depositCollateralWithReferral` call to clear any unconsumed allowance:

```solidity
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
token.approve(address(endpoint), 0); // reset residual approval
```

Alternatively, mirror the safe pattern already used in `ContractOwner.wrapVaultAsset()` by resetting to 0 before each new approval.

---

### Proof of Concept

1. A DDA exists for subaccount `X` (or is created via `createDirectDepositV1`).
2. An attacker (or any user) sends 1 wei of a USDT-like token to the DDA address.
3. Anyone calls `DirectDepositV1(dda).creditDeposit()`.
4. Inside the loop, `token.approve(address(endpoint), 1)` is called — allowance is now `1`.
5. `endpoint.depositCollateralWithReferral(X, productId, 1, "-1")` is called; `isValidDepositAmount` returns `false` (1 wei is below minimum); the function returns early without calling `handleDepositTransfer`.
6. The allowance on the USDT-like token remains `1` (non-zero).
7. Later, the legitimate owner sends a valid deposit amount (e.g., 100 USDT) to the DDA.
8. Anyone calls `creditDeposit()` again.
9. `token.approve(address(endpoint), 100e6)` is called — **REVERTS** because USDT requires the current allowance to be 0 before setting a new non-zero value.
10. The DDA is permanently bricked for that token; no further deposits can be credited via this mechanism.
