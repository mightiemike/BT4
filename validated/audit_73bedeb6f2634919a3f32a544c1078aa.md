Audit Report

## Title
`SwapAllowlistExtension` checks router address as `sender` instead of end-user, enabling allowlist bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap()` receives `sender` as the immediate caller of `MetricOmmPool.swap()`, which is the router contract when swaps are routed through `MetricOmmSimpleRouter`. A pool admin who allowlists the router to enable standard periphery usage inadvertently grants every user access, completely defeating the per-user allowlist. Any unprivileged user can bypass the access control by routing through the standard periphery.

## Finding Description
**Step 1 — Pool passes `msg.sender` as `sender` to the hook:**

In `MetricOmmPool.swap()`, `_beforeSwap` is called with `msg.sender` as the first argument unconditionally: [1](#0-0) 

**Step 2 — Router is `msg.sender` when calling `pool.swap()`:**

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly, making the router the `msg.sender` inside the pool: [2](#0-1) 

The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [3](#0-2) 

**Step 3 — `SwapAllowlistExtension.beforeSwap` checks `sender` (the router) against the allowlist:**

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [4](#0-3) 

Here `msg.sender` is the pool (correct) and `sender` is the router — not the end-user. The actual user's address is never inspected.

**Contrast with `DepositAllowlistExtension`:** That extension checks `owner` (the second parameter), which is explicitly supplied by the caller and can be set to the real user. `SwapAllowlistExtension` has no equivalent — it relies solely on `sender`, which is structurally bound to the immediate caller of `pool.swap()`. [5](#0-4) 

**No existing guard prevents this:** The router does not encode the real user address into `extensionData`, and `SwapAllowlistExtension` does not decode any user address from `extensionData`. The `setAllowedToSwap` setter only stores a flat `allowedSwapper[pool][swapper]` mapping with no router-awareness. [6](#0-5) 

## Impact Explanation
A pool admin who deploys a pool with `SwapAllowlistExtension` to restrict swaps to a specific set of counterparties (e.g., KYC'd addresses, whitelisted market makers) must also call `setAllowedToSwap(pool, router, true)` so that permitted users can use the standard periphery path. Once the router is allowlisted, **every user** who routes through it passes the `beforeSwap` check, regardless of whether they are individually permitted. The per-user guard is completely bypassed. Unauthorized users can execute swaps against LP funds in a pool that was intended to be restricted, breaking the core access-control invariant the extension is designed to enforce.

## Likelihood Explanation
This triggers under a natural and expected configuration: pool admin enables `SwapAllowlistExtension` to restrict access, then allowlists the router so that permitted users can use the standard periphery. Any user who discovers the router bypass can exploit it immediately with no special privileges. The router is a standard, documented periphery contract (`MetricOmmSimpleRouter`), so this path is reachable by any user. No privileged role, special token, or unusual condition is required beyond the admin having performed the natural setup.

## Recommendation
The actual end-user address must be conveyed to the extension through a channel the router cannot spoof:

1. **Pass the real user via `extensionData`**: The router encodes the end-user (`msg.sender`) into `extensionData`; the extension decodes and checks it. This requires the router to be trusted to supply the correct address.
2. **Introduce a `swapper` parameter distinct from `sender`**: Add an explicit `swapper` field to the `beforeSwap` hook signature that the pool populates from a verified source (e.g., transient-storage callback context set during the swap callback, analogous to how the reentrancy guard uses transient storage in `MetricReentrancyGuardTransient`).

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Admin calls `setAllowedToSwap(pool, router, true)` — allowlisting the router so that permitted users can swap via periphery.
3. Admin does **not** call `setAllowedToSwap(pool, userC, true)` — `userC` is not a permitted counterparty.
4. `userC` calls `MetricOmmSimpleRouter.exactInputSingle(...)`, which internally calls `pool.swap(recipient, ...)`.
5. Inside `pool.swap()`, `_beforeSwap(msg.sender=router, ...)` is called.
6. `SwapAllowlistExtension.beforeSwap(sender=router, ...)` evaluates `allowedSwapper[pool][router] == true` → passes.
7. `userC`'s swap executes against LP funds. The allowlist guard was never applied to `userC`.
