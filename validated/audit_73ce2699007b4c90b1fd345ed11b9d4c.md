### Title
No Minimum Confirmation Enforcement in `verify_transaction_inclusion` Allows Zero-Confirmation Proof Acceptance — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations: u64` value with no lower-bound enforcement. A caller passing `confirmations = 0` causes the confirmation-depth check to be trivially satisfied for any on-chain block, returning `true` for a transaction with zero subsequent confirmations. Any downstream NEAR contract that consumes this result and releases funds or state transitions based on it is exposed to a double-spend via chain reorganization.

### Finding Description

Both public verification entry points accept `ProofArgs` / `ProofArgsV2`, each containing a `confirmations: u64` field. [1](#0-0) 

Inside `verify_transaction_inclusion`, the only guard on `confirmations` is an **upper-bound** check: [2](#0-1) 

The confirmation-depth check that follows is: [3](#0-2) 

When `args.confirmations == 0`, the right-hand side of `>=` is `0`. Because the left-hand side is a `u64` expression (`saturating_sub(...) + 1`), it is always `>= 0`, so the `require!` is unconditionally satisfied. The function then proceeds to Merkle-proof verification and returns `true` for a transaction sitting in the chain tip with no subsequent blocks confirming it.

`verify_transaction_inclusion_v2` delegates to the deprecated function after its coinbase-proof check, so it inherits the same flaw: [4](#0-3) 

There is no floor on `confirmations` anywhere in the call path. The contract enforces `confirmations <= gc_threshold` (upper bound) but never `confirmations >= 1` (lower bound).

### Impact Explanation

A downstream NEAR contract that calls `verify_transaction_inclusion{_v2}` with `confirmations = 0` (or is tricked into doing so) will receive `true` for a Bitcoin transaction that has zero subsequent confirmations. That transaction can be reversed by a chain reorganization. Any value released by the downstream contract in response — bridged tokens, unlocked collateral, minted assets — can be stolen: the attacker submits the Bitcoin transaction, immediately triggers the NEAR-side release with a 0-confirmation proof, then double-spends the Bitcoin transaction on the Bitcoin side.

### Likelihood Explanation

The entry point is fully public and callable by any unprivileged NEAR account. The `confirmations` field is a plain integer with no documented minimum; integrators who omit a minimum or set it to `0` for testing and forget to change it in production will silently expose their protocol. The pattern is realistic: the analogous GMX bug was exactly a hardcoded `0` in a field that was supposed to carry a user-protection threshold.

### Recommendation

Add a lower-bound `require!` immediately alongside the existing upper-bound check in `verify_transaction_inclusion`:

```rust
require!(args.confirmations >= 1, "Confirmations must be at least 1");
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
```

Consider also documenting a recommended minimum (e.g., 6 for Bitcoin mainnet) in the public API so integrators are not left to guess.

### Proof of Concept

1. Deploy the contract with any valid genesis and a small `gc_threshold`.
2. Submit one block beyond genesis so the chain tip is at height 1.
3. Call `verify_transaction_inclusion` with:
   - `tx_block_blockhash` = the genesis block hash (height 0, on the main chain)
   - a valid Merkle proof for a transaction in that block
   - `confirmations = 0`
4. Observe the function returns `true` despite the transaction having only 1 subsequent confirmation (or 0 if the target block is the tip itself).
5. A downstream contract releasing funds on this `true` result is exposed to a reorg-based double-spend. [5](#0-4)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
