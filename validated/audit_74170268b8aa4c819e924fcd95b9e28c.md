### Title
Incorrect Weight Used in Spread Health Calculation Overestimates Subaccount Health — (`core/contracts/BaseEngine.sol`)

### Summary

`BaseEngine.getCoreRisk` always passes the hardcoded value `1` to `RiskHelper._getWeightX18`, which unconditionally returns the **long** weight regardless of the actual position direction. When `Clearinghouse.getHealth` uses this to compute `existingWeight` for spread positions, it underestimates the existing weight for the short leg of a long-spot / short-perp spread. This inflates the spread health bonus, causing the subaccount's health to be materially overestimated. A user holding such a spread can open positions beyond their true collateral capacity or avoid liquidation when they should be liquidatable.

---

### Finding Description

`BaseEngine.getCoreRisk` returns a `CoreRisk` struct whose `longWeight` field is populated by calling `_getWeightX18(risk, 1, healthType)`:

```solidity
// core/contracts/BaseEngine.sol:179-192
function getCoreRisk(
    bytes32 subaccount,
    uint32 productId,
    IProductEngine.HealthType healthType
) external returns (IProductEngine.CoreRisk memory) {
    RiskHelper.Risk memory risk = _risk(productId);
    (int128 amount, ) = _getBalance(productId, subaccount);
    return
        IProductEngine.CoreRisk(
            amount,
            risk.priceX18,
            RiskHelper._getWeightX18(risk, 1, healthType)  // ← always 1, never `amount`
        );
}
```

Because `1 >= 0`, `_getWeightX18` always branches into the long-weight path:

```solidity
// core/contracts/libraries/RiskHelper.sol:44-47
if (amount >= 0) {
    weight = healthType == IProductEngine.HealthType.INITIAL
        ? risk.longWeightInitialX18
        : risk.longWeightMaintenanceX18;
}
```

`Clearinghouse.getHealth` then uses these `CoreRisk` values to compute `existingWeight`:

```solidity
// core/contracts/Clearinghouse.sol:125-135
int128 existingWeight = (spotCoreRisk.longWeight +
    perpCoreRisk.longWeight) / 2;
int128 spreadWeight = RiskHelper._getSpreadWeightX18(
    perpCoreRisk,
    spotCoreRisk,
    healthType
);
health += basisAmount
    .mul(spotCoreRisk.price + perpCoreRisk.price)
    .mul(spreadWeight - existingWeight);
```

`existingWeight` is supposed to represent the average risk weight of the two legs as standalone positions. For a **long spot / short perp** spread:

- The spot leg is long → `spotLongWeight` is correct.
- The perp leg is **short** → the correct weight is `perpShortWeightInitialX18`, which by the invariant enforced in `_addOrUpdateProduct` satisfies `perpShortWeight >= 1e18 > perpLongWeight`.

Because `getCoreRisk` returns `perpLongWeight` instead of `perpShortWeight`, `existingWeight` is **underestimated**. This makes `spreadWeight − existingWeight` larger than it should be, inflating the health delta.

**Concrete example (INITIAL health, illustrative values):**

| Parameter | Value |
|---|---|
| `perpLongWeightInitialX18` | 0.90 × 10¹⁸ |
| `perpShortWeightInitialX18` | 1.10 × 10¹⁸ |
| `spotLongWeightInitialX18` | 0.90 × 10¹⁸ |
| `spreadWeight` (capped at 0.99) | 0.98 × 10¹⁸ |

With the bug:
- `existingWeight = (0.90 + 0.90) / 2 = 0.90`
- `spreadWeight − existingWeight = 0.98 − 0.90 = +0.08`
- Health bonus = `basisAmount × price × 0.08` (**positive**)

Correct calculation:
- `existingWeight = (0.90 + 1.10) / 2 = 1.00`
- `spreadWeight − existingWeight = 0.98 − 1.00 = −0.02`
- Health adjustment = `basisAmount × price × (−0.02)` (**negative penalty**)

The sign flips entirely. The bug converts a health penalty into a health bonus for long-spot / short-perp spreads.

---

### Impact Explanation

A user holding a long spot / short perp spread has their `INITIAL` and `MAINTENANCE` health overestimated by `basisAmount × (spotPrice + perpPrice) × (correctExistingWeight − buggyExistingWeight)`. This allows them to:

1. **Open additional positions** beyond their true collateral capacity, because `getHealth(sender, INITIAL) >= 0` passes when it should not (checked in `withdrawCollateral`, `transferQuote`, `mintNlp`, `nlpProfitShare`).
2. **Avoid liquidation** when their true maintenance health is negative, because `isUnderMaintenance` returns `false` when it should return `true` (checked in `liquidateSubaccountImpl`).

The overestimation scales with position size and price, so large spread positions produce large health inflation. The insurance fund and other subaccounts bear the resulting insolvency risk.

---

### Likelihood Explanation

Any unprivileged user can open a long spot / short perp spread through the normal `Endpoint` transaction flow. The spread recognition logic in `Clearinghouse.getHealth` is triggered automatically whenever both legs are non-zero and opposing. No special permissions, governance actions, or oracle manipulation are required. The only prerequisite is that the product pair is registered in `spreads`.

---

### Recommendation

In `BaseEngine.getCoreRisk`, pass the actual `amount` to `_getWeightX18` instead of the hardcoded `1`:

```solidity
// core/contracts/BaseEngine.sol:190
- RiskHelper._getWeightX18(risk, 1, healthType)
+ RiskHelper._getWeightX18(risk, amount, healthType)
```

This ensures that short positions return `shortWeightInitialX18` / `shortWeightMaintenanceX18`, making `existingWeight` in `Clearinghouse.getHealth` correctly reflect the actual per-leg risk weights and eliminating the health inflation.

---

### Proof of Concept

1. User deposits sufficient quote collateral.
2. User opens a long spot position on product `spotId` (e.g., 100 units at price 1.0).
3. User opens a short perp position on product `perpId` (e.g., −100 units).
4. `Clearinghouse.getHealth` is called (e.g., during a subsequent `withdrawCollateral`).
5. `getCoreRisk(subaccount, perpId, INITIAL)` returns `longWeight = 0.90` even though the perp position is short (correct weight = 1.10).
6. `existingWeight = (0.90 + 0.90) / 2 = 0.90`; `spreadWeight = 0.98`.
7. Health bonus = `100 × 2.0 × 0.08 = +16` units of quote.
8. Correct health would be `100 × 2.0 × (−0.02) = −4` units of quote.
9. The user's health appears `+20` units higher than reality, allowing them to withdraw collateral or open further positions that would otherwise be blocked by the `ERR_SUBACCT_HEALTH` check. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    function _getWeightX18(
        Risk memory risk,
        int128 amount,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }

        int128 weight;
        if (amount >= 0) {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.longWeightInitialX18
                : risk.longWeightMaintenanceX18;
        } else {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.shortWeightInitialX18
                : risk.shortWeightMaintenanceX18;
        }

        return weight;
    }
```

    function _getSpreadWeightX18(
        IProductEngine.CoreRisk memory perpCoreRisk,
        IProductEngine.CoreRisk memory spotCoreRisk,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }
        int128 spreadWeight;
        if (spotCoreRisk.amount > 0) {
            spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
        } else {
            spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;
        }
        int128 maxSpreadWeight;
        if (healthType == IProductEngine.HealthType.INITIAL) {
            maxSpreadWeight = ONE - ONE / 100; // 0.99
        } else {
            maxSpreadWeight = ONE - (ONE * 6) / 1000; // 0.994
        }
        if (spreadWeight > maxSpreadWeight) {
            spreadWeight = maxSpreadWeight;
        }
        return spreadWeight;
    }
```
