### Title
`replace()` Propagates Non-Canonical `_cached_serialization` from `CLVMTree` Subtrees, Breaking `sexp_to_bytes` Canonicality Invariant — (`wheel/python/clvm_rs/replace.py`, `ser.py`, `clvm_tree.py`)

---

### Summary

When `replace()` is called with a path that modifies only one child of a pair, the unmodified child — a `CLVMTree` node — is returned as-is. If that `CLVMTree` was deserialized from a non-canonical blob, its `_cached_serialization` property returns the raw non-canonical bytes. `Program.wrap()` blindly copies this value, and `sexp_to_byte_iterator` short-circuits on it, yielding non-canonical bytes directly. The invariant that `sexp_to_bytes` always produces canonical output is broken.

---

### Finding Description

**Step 1 — Non-canonical deserialization is accepted.**

`CLVMTree.from_bytes` calls `deserialize_as_tuples`, whose Python fallback `_atom_size_from_cursor` parses atom size prefixes without checking minimality: [1](#0-0) 

The byte sequence `\x81\x01` is parsed as a valid 1-byte atom with value `\x01` (size prefix `0x81` → size=1, then one data byte). The canonical encoding is the single byte `\x01`. No rejection occurs.

**Step 2 — `replace()` returns the unmodified `CLVMTree` child verbatim.** [2](#0-1) [3](#0-2) 

When called as `replace(tree, f=new_val)`, `args_by_prefix["r"]` is `{}`. The recursive call `replace(pair[1], **{})` hits the `len(kwargs) == 0` guard and returns `pair[1]` — the original `CLVMTree` — unchanged.

**Step 3 — `CLVMTree._cached_serialization` is a property returning raw blob bytes.** [4](#0-3) 

This always returns `self.blob[start:end]` — the original, potentially non-canonical bytes.

**Step 4 — `Program.wrap()` copies the non-canonical value into `_cached_serialization`.** [5](#0-4) 

`getattr(v, "_cached_serialization", None)` triggers the `CLVMTree` property, returning the non-canonical slice. This is stored as the `Program`'s `_cached_serialization`.

**Step 5 — `sexp_to_byte_iterator` short-circuits on `_cached_serialization`, yielding non-canonical bytes.** [6](#0-5) 

The check `if r is not None: yield r; continue` bypasses all canonical re-encoding logic. The non-canonical bytes are emitted directly into the output stream.

---

### Impact Explanation

- `sexp_to_bytes(program.replace(f=new_val))` returns non-canonical bytes for the right subtree.
- Two programs with identical tree hashes (same semantic content) produce different serialized byte strings and different `serialized_length` values depending on whether they passed through `replace()` on a `CLVMTree` built from non-canonical input.
- `Program.__bytes__` caches this non-canonical result, so all subsequent uses of the program's byte representation are tainted.
- Any downstream system using the serialized bytes for deduplication, fee calculation (cost is proportional to serialized size), or wire transmission receives incorrect data.

---

### Likelihood Explanation

The attack requires constructing a `CLVMTree` from a non-canonical blob, which the Python deserializer accepts without error. `CLVMTree.from_bytes` and `Program.from_bytes_with_cursor` are both public API entry points. The `replace()` method is a documented, production-facing convenience API on `Program`. The full path is reachable with attacker-controlled CLVM data passed through supported public APIs.

---

### Recommendation

In `sexp_to_byte_iterator`, do not trust `_cached_serialization` on nodes that are not the root of a freshly-deserialized, validated-canonical tree. Either:

1. Strip `_cached_serialization` from `CLVMTree` nodes before they enter `replace()`, or
2. In `Program.wrap()`, do not copy `_cached_serialization` from `CLVMTree` instances (only from `Program` instances whose serialization was already produced canonically), or
3. In `replace()`, when `len(kwargs) == 0` and the returned node is a `CLVMTree`, re-serialize it canonically before returning.

The safest fix is option 2: in `Program.wrap()`, only propagate `_cached_serialization` when `isinstance(v, Program)`, not from arbitrary `CLVMStorage` implementations.

### Proof of Concept

```python
from clvm_rs.clvm_tree import CLVMTree
from clvm_rs.program import Program
from clvm_rs.ser import sexp_to_bytes

# Non-canonical blob: pair of (atom \x01 encoded as \x81\x01, nil \x80)
# Canonical would be: b"\xff\x01\x80"
non_canonical_blob = b"\xff\x80\x81\x01"
# pair(nil, atom \x01) — right child uses 2-byte prefix instead of 1-byte

tree = CLVMTree.from_bytes(non_canonical_blob)

# Replace only the left (f) child; right child is returned unchanged
result = tree.replace(f=Program.null())

# Serialize the result
result_bytes = bytes(result)
print("result_bytes:", result_bytes.hex())
# Actual output:   ff 80 81 01  (non-canonical — right subtree kept raw bytes)
# Expected output: ff 80 01     (canonical)

# Confirm tree hashes match (same semantic content)
canonical = Program.to((Program.null(), Program.to(b"\x01")))
print("tree hashes equal:", result.tree_hash() == canonical.tree_hash())  # True
print("bytes equal:", bytes(result) == bytes(canonical))  # False — BUG
```

The `tree_hash()` values are identical (correct), but `bytes()` diverges, proving that two semantically equivalent programs produce different serialized bytes after `replace()`.

        raise ValueError("end of stream")
