### Title
Gateway `allowed_proof_versions` / `allowed_virtual_os_program_hashes` version-boundary mismatch causes valid proof-bearing transactions to be rejected and invalid ones admitted - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary
The gateway's stateful validator always builds its `BlockContext` from `VersionedConstants::latest_constants()`, while the batcher builds its `BlockContext` from the versioned constants that match the block's on-chain `StarknetVersion`. The `allowed_proof_versions` and `allowed_virtual_os_program_hashes` lists inside `OsConstants` changed materially between protocol versions. During any window where the running binary is ahead of the network's committed Starknet version, the two components enforce different allowlists for the same proof-facts fields, causing the gateway to reject transactions the batcher would accept, and to admit transactions the batcher will reject.

### Finding Description

**Root cause â€“ gateway always uses latest VC**

`run_validate_entry_point` in `StatefulTransactionValidator` constructs the block context with:

```rust
let mut versioned_constants = VersionedConstants::get_versioned_constants(
    self.config.versioned_constants_overrides.clone(),
);
versioned_constants.disable_casm_hash_migration();
``` [1](#0-0) 

`get_versioned_constants` always starts from `Self::latest_constants()`:

```rust
pub fn get_versioned_constants(...) -> Self {
    let latest_constants = Self::latest_constants().clone();
    ...
}
``` [2](#0-1) 

**Root cause â€“ batcher uses block-version VC**

The virtual-block executor (and the batcher's `BlockContext`) selects versioned constants by the block's `StarknetVersion`:

```rust
let mut versioned_constants = if use_latest_versioned_constants {
    VersionedConstants::latest_constants().clone()
} else {
    VersionedConstants::get(&block_info.starknet_version)...
};
``` [3](#0-2) 

For the sequencer's own batcher the block context is built from the committed block's version, not from `latest_constants()`.

**The divergent values â€“ `allowed_proof_versions` and `allowed_virtual_os_program_hashes`**

| Version | `allowed_proof_versions` | `allowed_virtual_os_program_hashes` |
|---|---|---|
| 0.14.0 | `[]` | `[]` |
| 0.14.2 | `["0x50524f4f4630"]` (PROOF0) | `["0x3e98c2d7..."]` |
| 0.14.3 | `["0x50524f4f4631"]` (PROOF1) | `["0x53f6c9fc..."]` |
| 0.14.4 | `["0x50524f4f4631"]` (PROOF1) | `["0x53f6c9fc...", "0x1c7be322..."]` | [4](#0-3) [5](#0-4) 

**The enforcement point â€“ `validate_proof_facts`**

During `perform_pre_validation_stage`, the blockifier checks:

```rust
if !os_constants.allowed_proof_versions.contains(&snos_proof_facts.proof_version.as_felt())
{
    return Err(TransactionPreValidationError::InvalidProofFacts(...));
}
...
if !allowed.contains(&snos_proof_facts.program_hash) {
    return Err(TransactionPreValidationError::InvalidProofFacts(...));
}
``` [6](#0-5) 

The gateway's `StatefulValidator` is created with the latest-VC block context and calls `validate`, which invokes `perform_pre_validation_stage` before running the account's `__validate__` entry point. The batcher's executor uses the block-version VC for the same check.

**Concrete divergence scenario (binary at 0.14.4, network at 0.14.2)**

- Gateway VC: `allowed_proof_versions = ["0x50524f4f4631"]` (PROOF1 only)  
- Batcher VC: `allowed_proof_versions = ["0x50524f4f4630"]` (PROOF0 only)

*Case A â€“ valid transaction rejected:* User submits Invoke V3 with PROOF0 proof facts. Gateway rejects it (`PROOF0 âˆ‰ {PROOF1}`). Batcher would have accepted it (`PROOF0 âˆˆ {PROOF0}`).

*Case B â€“ invalid transaction admitted:* User submits Invoke V3 with PROOF1 proof facts. Gateway accepts it (`PROOF1 âˆˆ {PROOF1}`). Batcher rejects it during pre-validation (`PROOF1 âˆ‰ {PROOF0}`). The transaction enters the mempool and is wasted.

The stateless validator's `allow_client_side_proving` flag must be `true` for proof-bearing transactions to reach the stateful validator, but this flag is explicitly designed to be enabled when client-side proving is live. [7](#0-6) 

### Impact Explanation

**High â€“ Mempool/gateway admission rejects valid transactions or accepts invalid ones before sequencing.**

In Case A, legitimate users with PROOF0 proof facts are permanently blocked from submitting transactions during the upgrade window, even though those transactions are protocol-valid for the current network version. In Case B, PROOF1 transactions pass gateway admission, consume mempool slots, and are then silently dropped by the batcher, wasting user fees and sequencer resources. Neither case requires any privileged access.

### Likelihood Explanation

The window exists whenever the sequencer binary is deployed at a version ahead of the network's committed Starknet version â€” a normal condition during any protocol upgrade. The `allowed_proof_versions` list changed between 0.14.2 and 0.14.3, and the `allowed_virtual_os_program_hashes` list changed between 0.14.2, 0.14.3, and 0.14.4. Any operator running a 0.14.3+ binary against a 0.14.2 network with client-side proving enabled hits this divergence immediately.

### Recommendation

The gateway's `run_validate_entry_point` should derive its versioned constants from the **current committed block's Starknet version** (available from `gateway_fixed_block_state_reader.get_block_info()`) rather than from `latest_constants()`. Specifically, replace:

```rust
let mut versioned_constants = VersionedConstants::get_versioned_constants(
    self.config.versioned_constants_overrides.clone(),
);
```

with a lookup keyed on `block_info.starknet_version`, applying the same overrides on top. This ensures the gateway and batcher enforce identical `allowed_proof_versions` and `allowed_virtual_os_program_hashes` allowlists for the same block height.

### Proof of Concept

1. Deploy a sequencer binary built at Starknet version 0.14.4 against a network whose latest committed block carries `starknet_version = "0.14.2"`.
2. Enable `allow_client_side_proving = true` in the gateway config.
3. Construct an Invoke V3 transaction whose `proof_facts` field begins with the PROOF0 version marker (`0x50524f4f4630`).
4. Submit via `starknet_addInvokeTransaction`.
5. **Observed:** Gateway's `validate_proof_facts` uses 0.14.4 VC (`allowed_proof_versions = ["0x50524f4f4631"]`); PROOF0 is absent â†’ `InvalidProofFacts` error returned to the user.
6. **Expected:** Gateway should use 0.14.2 VC (`allowed_proof_versions = ["0x50524f4f4630"]`); PROOF0 is present â†’ transaction accepted and forwarded to the mempool.

The reverse (PROOF1 admitted by gateway, rejected by batcher) is demonstrated by substituting `0x50524f4f4631` as the proof version marker in step 3.
