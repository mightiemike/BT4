### Title
Bonus Accrual Checkpoint Advanced Without Coin Transfer on Insufficient Pool During Redelegation Slash — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/bonus_rewards.go`)

---

### Summary

In `slashRedelegationPosition`, when `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, the error is intentionally swallowed to prevent a chain halt. However, `processEventsAndClaimBonus` mutates `pos.LastBonusAccrual` via `applyBonusAccrualCheckpoint` **before** the pool balance check. The mutated position is then persisted by `setPositionWithState`. The result: the accrual checkpoint advances to the current block time, but no coins are ever sent from `RewardsPoolName` to the owner. The bonus earned for the entire bonded segment prior to the slash is permanently unrecoverable.

---

### Finding Description

**Exact call chain:**

1. `BeforeRedelegationSlashed` (hooks.go:128–130) → `slashRedelegationPosition` (slash.go:19)
2. `slashRedelegationPosition` calls `processEventsAndClaimBonus(ctx, &pos)` at slash.go:54 — passing `pos` **by pointer**.
3. Inside `processEventsAndClaimBonus` (claim_rewards.go:142–251):
   - **Line 215**: `applyBonusAccrualCheckpoint(&pos.Position, blockTime)` — unconditionally mutates `pos.LastBonusAccrual` to `blockTime` in-place, before any pool check.
   - **Line 217**: `pos.UpdateLastKnownBonded(bonded)` — also mutates in-place.
   - **Line 230**: `sufficientBonusPoolBalance(ctx, bonusCoins)` — if pool balance is zero and `totalBonus > 0`, returns `ErrInsufficientBonusPool`.
   - The function returns the error **without** executing `SendCoinsFromModuleToAccount` (line 239).
4. Back in `slashRedelegationPosition` (slash.go:54–64): `ErrInsufficientBonusPool` is caught and **silently ignored** (just logged).
5. At this point, `pos.LastBonusAccrual` is already `blockTime` in the local struct.
6. `setPositionWithState(ctx, pos, ...)` is called at slash.go:71 (full slash) or slash.go:77 (partial slash) — **persisting the advanced checkpoint** with no corresponding coin transfer. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

The ordering bug is clear: `applyBonusAccrualCheckpoint` runs at line 215, but `sufficientBonusPoolBalance` runs at line 230. Any error path between those two lines leaves the checkpoint advanced with no payment. [5](#0-4) 

---

### Impact Explanation

The invariant that `pos.LastBonusAccrual` must only advance when the corresponding bonus coins are actually transferred from `RewardsPoolName` to the owner is violated. After the slash:

- `pos.LastBonusAccrual` = current block time (persisted).
- `RewardsPoolName` balance = still zero (no transfer occurred).
- Owner's balance = unchanged.

On any subsequent claim, the accrual window starts from the advanced checkpoint, so the bonus for the entire bonded segment `[old_LastBonusAccrual, slash_block_time]` is **permanently lost**. There is no mechanism to retroactively recover it. This is a direct, irreversible loss of legitimately earned bonus rewards from the module's accounting.

---

### Likelihood Explanation

The `RewardsPoolName` pool can be at zero (or below the owed bonus amount) whenever:
- The pool has been recently drained by other position claims.
- The pool has not been replenished by governance/inflation.

A redelegation slash is a normal protocol event (double-sign or downtime on a validator that has active redelegations). No attacker action is required — the position owner is the victim. The combination of a non-empty pool being temporarily depleted and a concurrent redelegation slash is realistic in production.

---

### Recommendation

Move `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` to **after** the successful `SendCoinsFromModuleToAccount` call, so the checkpoint only advances when coins are actually transferred:

```go
// claim_rewards.go — reorder so checkpoint advances only on success
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err
}
if err := k.bankKeeper.SendCoinsFromModuleToAccount(...); err != nil {
    return nil, err
}
// Only advance checkpoint after confirmed transfer
applyBonusAccrualCheckpoint(&pos.Position, blockTime)
pos.UpdateLastKnownBonded(bonded)
```

Alternatively, in `slashRedelegationPosition`, when `ErrInsufficientBonusPool` is caught, explicitly reset `pos.LastBonusAccrual` and `pos.LastKnownBonded` to their pre-call values before calling `setPositionWithState`. [6](#0-5) [7](#0-6)
