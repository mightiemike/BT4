### Title
Proposer-Supplied `starknet_version` in `ProposalInit` Bypasses `fee_proposal_fri` Bounds Enforcement — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The validator's version gate for `fee_proposal_fri` reads the `starknet_version` field directly from the proposer-supplied `ProposalInit` message rather than from the validator's own locally-determined expected version. A malicious proposer can set `starknet_version` to a pre-V0_14_3 value, omit `fee_proposal_fri`, and bypass the bounds check entirely. This injects a `None` entry into the `fee_proposals_window`, which causes `compute_fee_actual` to return `None` for the next `window_size` blocks, disabling bounds enforcement for all proposals during that window and enabling unconstrained manipulation of the L2 gas price.

---

### Finding Description

In `is_proposal_init_valid` the version gate is:

```rust
let fee_proposal_required = init_proposed.starknet_version >= StarknetVersion::V0_14_3;
match (init_proposed.fee_proposal_fri, fee_proposal_required) {
    (Some(_), false) => { return Err(...) }
    (None, true)     => { return Err(...) }
    _ => {}
}
``` [1](#0-0) 

`init_proposed.starknet_version` is the value the proposer wrote into the network message. The `ProposalInitValidation` struct carries a `starknet_version` field that is always set to `StarknetVersion::LATEST` by the validator: [2](#0-1) 

but this local field is **never compared** against `init_proposed.starknet_version` in the validation logic. The only other structural check is `version_constant_commitment`, which is a fixed sentinel value (`expected_version_constant_commitment()`) that is independent of `starknet_version`: [3](#0-2) 

A malicious proposer therefore constructs a `ProposalInit` with:
- `starknet_version = V0_14_2` (any pre-V0_14_3 value)
- `fee_proposal_fri = None`
- `version_constant_commitment = expected_version_constant_commitment()` (the fixed sentinel, unchanged)

The validator evaluates `V0_14_2 >= V0_14_3 → false`, so `fee_proposal_required = false`, the `(None, false)` arm is a no-op, and the bounds check:

```rust
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{ ... }
``` [4](#0-3) 

is also skipped because `init_proposed.fee_proposal_fri` is `None`. The proposal is accepted.

The accepted block is stored with `fee_proposal_fri = None` in `BlockHeaderWithoutHash`. When the orchestrator later calls `compute_fee_actual` for any height whose sliding window includes this block:

```rust
Some(None) | None => {
    warn!("... fee_proposals_window has no recorded fee_proposal for height {source_height}");
    return None;
}
``` [5](#0-4) 

`compute_fee_actual` returns `None` for the entire window period. When `fee_actual = None`, the bounds check is unconditionally skipped for every proposal during that window, allowing any proposer to set `fee_proposal_fri` to an arbitrary value.

The `proposal_commitment_from` function also produces a different hash for `None` vs `Some(x)`:

```rust
let Some(fee_proposal) = fee_proposal else {
    return ProposalCommitment(partial.0);   // pre-V0_14_3 path
};
ProposalCommitment(Poseidon::hash_array(&[partial.0, Felt::from(fee_proposal.0)]))
``` [6](#0-5) 

so the commitment the validator computes and signs over is `partial.0` (the pre-V0_14_3 form), which is consistent with the proposer's own computation — consensus reaches agreement on a commitment that omits the fee binding entirely.

---

### Impact Explanation

`fee_proposal_fri` is the per-block oracle signal that feeds the SNIP-35 sliding-window median (`fee_actual`), which in turn bounds every future proposer's `fee_proposal_fri` via `fee_proposal_bounds`. By injecting a single `None` into the window, a malicious proposer disables this bound for `window_size` consecutive blocks. During that window any proposer can freely set `fee_proposal_fri` to an extreme value (e.g., orders of magnitude above or below the true market rate). After the window heals, `fee_actual` is computed from a window that contains those extreme values, permanently shifting the L2 gas price. This directly affects the fee every user pays for every transaction — matching the "Incorrect fee, gas, bouncer, resource accounting … with economic impact" critical impact category.

---

### Likelihood Explanation

Any consensus participant that is scheduled to propose a block can execute this attack with a single crafted `ProposalInit` message. No special privilege, no off-chain coordination, and no invalid bytes are required — the message is structurally valid and passes all other checks. The attack is therefore reachable by any validator node acting as proposer.

---

### Recommendation

Replace the proposer-supplied version gate with the validator's locally-determined expected version:

```rust
// Use the validator's own starknet_version, not the proposer-supplied one.
let fee_proposal_required =
    proposal_init_validation.starknet_version >= StarknetVersion::V0_14_3;
```

Additionally, add an explicit check that `init_proposed.starknet_version` equals `proposal_init_validation.starknet_version` (or at minimum is not lower), so a proposer cannot claim an older protocol version to escape any version-gated validation rule.

---

### Proof of Concept

1. Malicious proposer builds `ProposalInit`:
   - `starknet_version = StarknetVersion::V0_14_2`
   - `fee_proposal_fri = None`
   - `version_constant_commitment = expected_version_constant_commitment()` (unchanged sentinel)
   - All gas-price fields within the L1 margin (normal values)

2. Validator runs `is_proposal_init_valid`:
   - L1 gas price checks pass (fields are normal).
   - `fee_proposal_required = V0_14_2 >= V0_14_3 = false`.
   - `(None, false)` → no error, no bounds check.
   - `version_constant_commitment` matches sentinel → no error.
   - Returns `Ok(())`.

3. Validator computes `proposal_commitment_from(partial, None) = ProposalCommitment(partial.0)` — identical to what the proposer computed — so consensus votes agree.

4. Block is committed and stored with `fee_proposal_fri = None` in `BlockHeaderWithoutHash`.

5. For the next `window_size` blocks, `compute_fee_actual` hits the `Some(None)` arm and returns `None`.

6. During those blocks, the bounds check `if let (Some(fee_actual), Some(fee_proposal)) = ...` is never entered; any proposer may set `fee_proposal_fri` to any value.

7. After the window heals, `fee_actual` is the median of a window that includes the attacker-chosen extreme values, permanently shifting the L2 gas price.
