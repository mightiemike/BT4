### Title
Sanctions Validation Exists But Is Never Applied to `sendTo` in `WithdrawCollateralV2` — (`EndpointTx.sol`)

---

### Summary

The Nado protocol has a `requireUnsanctioned` function that is explicitly called on both the depositor and the subaccount owner during deposits. However, in the `WithdrawCollateralV2` execution path, this same validation is never applied to the user-controlled `sendTo` address. A user can sign a `WithdrawCollateralV2` transaction directing their funds to any address — including an OFAC-sanctioned one — and the on-chain contract will process it without any sanctions check on the destination.

---

### Finding Description

`EndpointStorage.requireUnsanctioned` is the protocol's on-chain sanctions gate: [1](#0-0) 

It is explicitly invoked for both parties during deposits in `Endpoint.depositCollateralWithReferral`: [2](#0-1) 

The `WithdrawCollateralV2` struct includes a user-signed `sendTo` field that redirects the withdrawal to an arbitrary address: [3](#0-2) 

This field is included in the EIP-712 digest, so the user explicitly signs over it: [4](#0-3) 

In `EndpointTx.processTransactionImpl`, the `WithdrawCollateralV2` branch validates the signature and charges a fee, but never calls `requireUnsanctioned` on `signedTx.tx.sendTo`: [5](#0-4) 

The call chain continues into `Clearinghouse.withdrawCollateral`, which also performs no sanctions check on `sendTo` before transferring funds: [6](#0-5) 

The `sendTo` address flows directly into `handleWithdrawTransfer` with no interception: [7](#0-6) 

By contrast, the slow-mode submission path does call `requireUnsanctioned(sender)` before queuing any transaction: [8](#0-7) 

This creates a direct structural parallel to the external report: a validation function (`requireUnsanctioned`) exists and is used in some paths, but is silently absent from the `WithdrawCollateralV2` execution path for the destination address.

---

### Impact Explanation

A user can withdraw their own collateral directly to an OFAC-sanctioned address by setting `sendTo` to that address in a signed `WithdrawCollateralV2` transaction. The on-chain contract will execute the transfer without any sanctions check on the destination. This bypasses the protocol's stated compliance posture (evidenced by the sanctions checks on deposits) and results in a real, on-chain asset transfer to a prohibited address. The corrupted state is the token balance delta at the sanctioned destination address.

---

### Likelihood Explanation

The entry path is fully user-controlled: the user signs the `sendTo` field as part of the EIP-712 digest and submits the transaction to the sequencer. The sequencer may perform off-chain filtering, but the on-chain contract provides no enforcement. Any user with a funded subaccount can construct and sign such a transaction. The sequencer's off-chain filtering is not a protocol-level guarantee and can be circumvented (e.g., by a sequencer that does not implement filtering, or by a future upgrade). Likelihood is **medium** — it requires deliberate intent, but the on-chain path is fully open.

---

### Recommendation

Add `requireUnsanctioned(signedTx.tx.sendTo)` in the `WithdrawCollateralV2` branch of `EndpointTx.processTransactionImpl`, immediately after signature validation and before calling `clearinghouse.withdrawCollateral`. This mirrors the pattern already established for deposits and closes the gap between the protocol's intended compliance posture and its on-chain enforcement.

---

### Proof of Concept

1. Alice has collateral in the protocol.
2. Alice constructs a `WithdrawCollateralV2` transaction with `sendTo` = `0xSANCTIONED_ADDRESS` and signs it with her EIP-712 key.
3. The sequencer submits the batch via `submitTransactionsChecked`.
4. `processTransactionImpl` decodes the transaction, validates Alice's signature via `validateSignedTx`, and calls `clearinghouse.withdrawCollateral(..., 0xSANCTIONED_ADDRESS, ...)`.
5. `Clearinghouse.withdrawCollateral` calls `handleWithdrawTransfer(token, 0xSANCTIONED_ADDRESS, amount, idx)`.
6. Funds are transferred on-chain to the sanctioned address. `requireUnsanctioned` was never called on `0xSANCTIONED_ADDRESS` at any point in the execution path.

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
