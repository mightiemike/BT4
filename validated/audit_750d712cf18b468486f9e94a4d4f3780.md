### Title
Incorrect Ancestor Block Timestamp Used in Dogecoin Fork Difficulty Calculation — (File: `contract/src/dogecoin.rs`)

---

### Summary

In `get_next_work_required` inside `contract/src/dogecoin.rs`, the difficulty target for fork blocks is computed using the **mainchain** block's timestamp at `height_first` instead of the fork's actual ancestor block's timestamp. This is a direct analog to the Perennial M-7 bug class: a wrong value (stale/incorrect data from a parallel module) is silently substituted into a critical calculation, producing an incorrect result. The developers themselves flagged this with a `TODO` comment at the exact line.

---

### Finding Description

`get_next_work_required` in `contract/src/dogecoin.rs` computes the next difficulty target by fetching the block at `height_first` via:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

`get_header_by_height` is implemented in `contract/src/lib.rs` as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [2](#0-1) 

This lookup is **unconditionally against the mainchain** (`mainchain_height_to_header`). It has no awareness of which chain the `prev_block_header` belongs to.

For Dogecoin after block 145,000, `difficulty_adjustment_interval` is overridden to `1`:

```rust
let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
let difficulty_adjustment_interval = if new_difficulty_protocol { 1 } else { ... };
``` [3](#0-2) 

With `difficulty_adjustment_interval = 1`, `blocks_to_go_back` resolves to `1` for any non-genesis block, so `height_first = prev_block_header.block_height - 1`. This means the function fetches the **mainchain block at `prev_block_height - 1`** and uses its timestamp as `first_block_time`. [4](#0-3) 

When a fork block is being validated, `prev_block_header` is a fork block. If the fork diverged at or before `height_first`, the mainchain block at `height_first` is a **different block** from the fork's ancestor at that height. Their timestamps can differ arbitrarily (within MTP and future-time constraints). The wrong `first_block_time` is then fed into `calculate_next_work_required`:

```rust
calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
``` [5](#0-4) 

Inside `calculate_next_work_required`, `first_block_time` directly controls `modulated_timespan`, which directly controls the output difficulty target:

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;
``` [6](#0-5) 

The same structural bug exists in `contract/src/bitcoin.rs` at line 81, but with a 2016-block adjustment interval, making it far less reachable in practice. [7](#0-6) 

---

### Impact Explanation

**Scenario A — Easier target (security-critical):** If the mainchain block at `height_first` has an **earlier** timestamp than the fork's ancestor at that height, `modulated_timespan` is inflated, yielding a **lower (easier) difficulty target** than the fork's actual ancestor chain requires. Fork blocks with insufficient PoW pass `check_pow` and are stored. If such a fork accumulates more chainwork than the mainchain, `reorg_chain` promotes it as the canonical chain, corrupting the on-chain Bitcoin state that downstream NEAR contracts rely on for SPV proof verification.

**Scenario B — Harder target (liveness):** If the mainchain block at `height_first` has a **later** timestamp, the computed target is harder than required. Valid fork blocks are incorrectly rejected, preventing legitimate chain reorganizations from being processed.

The canonical chain state (`mainchain_tip_blockhash`, `mainchain_height_to_header`, `mainchain_header_to_height`) is the corrupted artifact. Any call to `verify_transaction_inclusion_v2` after a corrupt reorg will verify proofs against the wrong chain. [8](#0-7) 

---

### Likelihood Explanation

For Dogecoin (the primary affected chain), `difficulty_adjustment_interval = 1` after block 145,000 means **every fork block beyond the first** triggers this code path. A relayer submitting a fork chain that diverges more than one block from the mainchain will always hit this bug. The attacker controls fork block timestamps (bounded only by MTP and the 2-hour future-time limit), making it feasible to craft a fork where the mainchain ancestor has an earlier timestamp than the fork ancestor, yielding an easier target. This is a realistic, low-barrier attack for any unprivileged proof submitter on the Dogecoin deployment.

---

### Recommendation

Replace the `get_header_by_height` call with an ancestor walk that follows `prev_block_hash` links through the fork chain to find the actual ancestor at `height_first`. This is exactly what the `TODO` comment at line 291 suggests. The correct approach mirrors how `get_median_time_past` works — it walks backwards through `get_prev_header` rather than looking up by height in the mainchain map. [9](#0-8) 

---

### Proof of Concept

1. Mainchain has blocks at heights `H-1` (timestamp `T_main`) and `H` (timestamp `T_main + 60`). Both are in `mainchain_height_to_header`.
2. Attacker submits a fork block at height `H-1` with timestamp `T_fork = T_main - 30` (earlier than mainchain). This fork block is stored in `headers_pool` only (not mainchain).
3. Attacker submits a fork block at height `H` with timestamp `T_fork + 60`, building on the fork block at `H-1`.
4. Attacker submits a fork block at height `H+1`. `prev_block_header` is the fork block at `H`. `height_first = H - 1`. `get_header_by_height(H-1)` returns the **mainchain** block with timestamp `T_main`.
5. `modulated_timespan` is computed as `(T_fork + 60) - T_main = T_fork + 60 - T_main`. Since `T_fork = T_main - 30`, this equals `30`. The correct value using the fork ancestor would be `(T_fork + 60) - T_fork = 60`.
6. With `retarget_timespan = 60` and `modulated_timespan = 30`, the Digishield formula produces a harder target — but the attacker can tune `T_fork` to be earlier than `T_main` to produce an easier target instead (e.g., `T_fork = T_main + 30` gives `modulated_timespan = 90 > 60`, yielding an easier target).
7. The fork block at `H+1` passes `check_pow` with lower PoW than the fork's actual ancestor chain requires.
8. If the fork's cumulative `chain_work` exceeds the mainchain's, `reorg_chain` is triggered, promoting the under-PoW fork as the canonical chain. [10](#0-9) [8](#0-7)
