Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks LP Position `owner` Instead of Actual Caller `sender`, Allowing Unauthorized Liquidity Deposits — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the first parameter (`sender`, the actual `msg.sender` of `addLiquidity`) and instead checks the second parameter (`owner`, the LP-position recipient) against `allowedDepositor`. Any address not on the allowlist can bypass the gate by naming any allowlisted address as `owner`, completely defeating the deposit access control.

## Finding Description

`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)`, encoding the actual caller as the first argument and the LP-position recipient as the second, matching the `IMetricOmmExtensions.beforeAddLiquidity(address sender, address owner, ...)` interface. [1](#0-0) [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` discards the first parameter and evaluates `allowedDepositor[msg.sender][owner]`: [3](#0-2) 

The NatSpec and mapping name both confirm the intended subject is the depositor (caller), not the owner: [4](#0-3) 

`SwapAllowlistExtension.beforeSwap` correctly checks `sender` (first parameter): [5](#0-4) 

`ExtensionCalling._beforeAddLiquidity` confirms the ABI encoding order — `sender` first, `owner` second: [6](#0-5) 

## Impact Explanation

The deposit allowlist is the pool admin's primary mechanism for restricting who can supply liquidity to a restricted pool. Because the guard checks `owner` instead of `sender`, any unauthorized address can inject liquidity into the pool by naming any allowlisted address as the LP-position recipient. This is an admin-boundary break: an unprivileged path circumvents a configured pool-level access control, allowing unauthorized parties to affect bin-level balances, pool dynamics, and LP dilution. The attacker's tokens enter the pool and the LP position is credited to the named `owner`; the attacker loses the deposited tokens but successfully bypasses the restriction.

## Likelihood Explanation

Exploitation requires no special privilege. Any external address can call `pool.addLiquidity(owner=<any_allowlisted_address>, ...)`. The allowlisted `owner` need not cooperate. The bypass is trivially reachable by any external actor on any pool using this extension with `allowAllDepositors[pool] = false`.

## Recommendation

Replace `owner` with `sender` in the allowlist check, mirroring `SwapAllowlistExtension`:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external view override returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

## Proof of Concept

1. Deploy pool with `DepositAllowlistExtension`; `allowAllDepositors[pool] = false`.
2. Admin calls `setAllowedToDeposit(pool, bob, true)`. Alice is **not** allowlisted.
3. Alice calls `pool.addLiquidity(owner=bob, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(alice, bob, ...)` → extension receives `(alice, bob, ...)`.
5. Extension evaluates `allowedDepositor[pool][bob]` → `true` → no revert.
6. Alice's tokens are deposited; Bob receives the LP position.
7. Alice has successfully added liquidity to a restricted pool she is not authorized to touch.
