### Title
Unbounded `while` loop in `reorg_chain` can exceed NEAR gas limit, permanently corrupting canonical chain state — (`contract/src/lib.rs`)

---

### Summary

The `reorg_chain` function contains an unbounded `while` loop that walks the entire fork chain back to the common ancestor with the main chain. Because fork headers are accumulated across many cheap transactions and the reorg work is concentrated in a single final transaction, a sufficiently long fork causes that transaction to exceed NEAR's 300 TGas per-transaction gas limit. The transaction reverts, the reorg never completes, and the light client is permanently stuck with a canonical chain that has less cumulative proof-of-work than the fork — corrupting every subsequent SPV verification result.

---

### Finding Description

`reorg_chain` is called from `submit_block_header_inner` whenever a newly submitted fork header's `chain_work` exceeds the current main-chain tip's `chain_work`. [1](#0-0) 

Inside `reorg_chain`, after optionally trimming extra main-chain blocks, the function enters an unbounded `while` loop: [2](#0-1) 

The loop terminates only when `fork_header_cursor.block_hash` is found in `mainchain_header_to_height` — i.e., when the common ancestor is reached. There is no iteration cap. Each loop body performs:

- one `LookupMap::contains_key` (storage read) [3](#0-2) 
- one `LookupMap::insert` into `mainchain_height_to_header` (storage write) [4](#0-3) 
- one `LookupMap::insert` into `mainchain_header_to_height` (storage write) [5](#0-4) 
- conditionally, `remove_block_header` which does two more storage removals [6](#0-5) 
- one `LookupMap::get` to advance the cursor [7](#0-6) 

That is 4–5 storage operations per iteration. NEAR storage writes cost ~100 GGas each; at 300 TGas per transaction, the loop exhausts the budget at roughly **600–750 iterations** (fork headers).

Fork headers are accumulated cheaply: each call to `submit_blocks` that delivers a fork header only calls `store_fork_header`, which is a single `headers_pool.insert`. [8](#0-7) 

There is no cap on how many fork headers can be stored, and no cap on the fork depth that `reorg_chain` must traverse. The entire reorg cost is concentrated in the single transaction that submits the tip header whose `chain_work` finally exceeds the main chain.

A secondary unbounded loop exists in `submit_blocks` itself — `headers: Vec<BlockHeader>` has no length limit, so a single call with thousands of headers also exhausts gas: [9](#0-8) 

---

### Impact Explanation

When the reorg transaction reverts, NEAR rolls back all state changes. The fork headers remain in `headers_pool` but are not promoted to `mainchain_height_to_header`. The main-chain tip pointer (`mainchain_tip_blockhash`) stays on the lower-chainwork chain. Every subsequent call to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` checks confirmations and Merkle roots against this stale, lower-chainwork chain. [10](#0-9) 

Concretely:
- Transactions confirmed only on the real (higher-chainwork) chain are rejected as unconfirmed.
- Transactions confirmed only on the now-orphaned chain are accepted as confirmed.
- The light client cannot self-correct: every retry of the reorg-triggering transaction fails identically.

---

### Likelihood Explanation

The entry path is `submit_blocks`, which is gated by `#[trusted_relayer]`. [11](#0-10) 

However, the bypass roles `Role::DAO` and `Role::UnrestrictedSubmitBlocks` allow any account holding those roles to call `submit_blocks` without the trusted-relayer stake requirement. [12](#0-11) 

Additionally, the `skip_pow_verification` flag (set at init time) removes the only computational cost that would otherwise make building a long fork expensive. With `skip_pow_verification = true`, any relayer-path actor can submit thousands of structurally valid fork headers at negligible cost, then submit the chainwork-exceeding tip to trigger the unbounded reorg loop. [13](#0-12) 

Even with PoW verification enabled, a real Bitcoin fork of a few hundred blocks (well within historical precedent for deep reorgs on test networks or adversarial scenarios) is sufficient to exceed the gas limit.

---

### Recommendation

1. **Cap fork depth before reorg**: Before entering the `while` loop in `reorg_chain`, compute the fork depth and panic (or return an error) if it exceeds a safe maximum (e.g., 200 headers). This mirrors the fix applied in the referenced report (reducing max bid count).

2. **Batch the reorg**: Split `reorg_chain` into a resumable, multi-transaction operation. Store a "pending reorg" cursor in contract state and allow the relayer to call a `continue_reorg` method until the common ancestor is reached.

3. **Cap `submit_blocks` batch size**: Add a `require!(headers.len() <= MAX_BATCH, ...)` guard at the top of `submit_blocks` to prevent the secondary unbounded loop from also exhausting gas.

---

### Proof of Concept

With `skip_pow_verification = true` and a fork branching from the genesis block:

1. Call `submit_blocks` with N fork headers (e.g., N = 800), each building on the previous fork header and carrying `bits` set to give slightly more cumulative work than the main chain tip. Each call is cheap (one `headers_pool.insert` per header).

2. Call `submit_blocks` with the final fork tip header whose `chain_work` exceeds the main chain. This triggers `reorg_chain`.

3. `reorg_chain` enters the `while` loop and must iterate 800 times, performing ~4,000 storage operations. At ~100 GGas per write, this totals ~400 TGas — exceeding NEAR's 300 TGas limit.

4. The transaction panics with out-of-gas. The main-chain tip remains on the lower-chainwork chain. All subsequent `verify_transaction_inclusion` calls return results based on the wrong canonical chain.
