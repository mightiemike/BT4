### Title
`only_query` Flag Hardcoded to `false` in `BroadcastedTransaction` Conversion Causes Wrong Transaction Hash and Incorrect Fee Estimation for Query-Version Transactions — (`crates/apollo_rpc/src/v0_8/api/mod.rs`)

### Summary

The `TryFrom<BroadcastedTransaction> for ExecutableTransactionInput` conversion unconditionally sets `only_query = false` for every incoming broadcasted transaction, discarding the query-version bit (bit 128) that clients legitimately set when calling `starknet_estimateFee` or `starknet_simulateTransactions`. Because `only_query` controls whether `QUERY_VERSION_BASE` is added to the version before hashing, the transaction hash computed during fee estimation is the *regular* hash rather than the *query* hash. Any account contract whose `__validate__` entry point verifies a signature over the transaction hash will fail validation during estimation, causing the RPC to return an authoritative-looking `TRANSACTION_EXECUTION_ERROR` instead of a valid fee estimate.

### Finding Description

In `crates/apollo_rpc/src/v0_8/api/mod.rs`, the conversion from `BroadcastedTransaction` to `ExecutableTransactionInput` is:

```rust
impl TryFrom<BroadcastedTransaction> for ExecutableTransactionInput {
    type Error = ErrorObjectOwned;
    fn try_from(value: BroadcastedTransaction) -> Result<Self, Self::Error> {
        // TODO(yair): pass the right value for only_query field.
        match value {
            BroadcastedTransaction::Declare(tx) => Ok(tx.try_into()?),
            BroadcastedTransaction::DeployAccount(tx) => Ok(Self::DeployAccount(tx.into(), false)),
            BroadcastedTransaction::Invoke(tx) => Ok(Self::Invoke(tx.into(), false)),
        }
    }
}
```

The `false` literal is hardcoded for every variant. The same pattern appears in `TryFrom<BroadcastedDeclareTransaction>` at line 506–507.

The `only_query` value flows directly into `calc_tx_hash`:

```rust
fn calc_tx_hash(self, chain_id: &ChainId) -> ExecutionResult<(Self, TransactionHash)> {
    match self.apply_on_transaction(|tx, only_query| {
        get_transaction_hash(tx, chain_id, &TransactionOptions { only_query })
    }) { … }
}
```

`get_transaction_hash` calls `signed_tx_version_from_tx`, which adds `QUERY_VERSION_BASE` (2^128) to the version only when `only_query = true`. With `only_query = false` hardcoded, the hash is always the non-query hash regardless of the version the client sent.

The same `only_query` value is forwarded to `ExecutionFlags` in `to_blockifier_tx`, so the version exposed to the contract's `__validate__` entry point via `get_execution_info` is also wrong (regular version instead of query version).

The Starknet RPC spec defines `TransactionVersion1OnlyQuery = 0x100000000000000000000000000000001` (and analogous values for v2/v3) precisely so that clients can signal "this is a query/estimation call." The sequencer's own RPC transaction types already model these variants:

```rust
pub enum TransactionVersion1 {
    #[serde(rename = "0x1")]
    Version1,
    #[serde(rename = "0x100000000000000000000000000000001")]
    Version1OnlyQuery,
}
```

But the conversion silently drops the distinction.

### Impact Explanation

When a client submits a fee estimation request with a query-version transaction (as the Starknet spec recommends), the sequencer:

1. Computes the transaction hash without the query bit → produces the *regular* hash.
2. Passes `only_query = false` to `ExecutionFlags` → the contract's `__validate__` sees the regular version, not the query version.
3. If the account contract's `__validate__` verifies a signature over the transaction hash (standard for all OpenZeppelin-style accounts), and the client signed over the *query* hash, the signature check fails.
4. `estimate_fee` returns `TRANSACTION_EXECUTION_ERROR` instead of a valid `FeeEstimation`.

This is a wrong authoritative value from the RPC fee-estimation endpoint. Clients relying on `starknet_estimateFee` with query-version transactions receive an error for transactions that are actually valid, or receive a fee estimate computed under the wrong execution context (wrong version visible to `__validate__`), leading to incorrect gas figures.

### Likelihood Explanation

Any client that follows the Starknet spec and sends query-version transactions to `starknet_estimateFee` or `starknet_simulateTransactions` triggers this path. The TODO comment in the source confirms the issue is known but unresolved. The `SIMULATION_FLAG_FOR_ESTIMATE_FEE` schema only exposes `SKIP_VALIDATE`; clients that do not skip validation (the default) will hit the wrong-hash path on every query-version estimation call.

### Recommendation

Extract the `only_query` bit from the incoming transaction version before constructing `ExecutableTransactionInput`. The RPC `TransactionVersion*` enums already encode this information. The conversion should be:

```rust
BroadcastedTransaction::Invoke(tx) => {
    let only_query = tx.is_only_query(); // check Version*OnlyQuery variant
    Ok(Self::Invoke(tx.into_base_version(), only_query))
}
```

where `into_base_version()` strips the query bit from the version field before passing it to the `starknet_api` layer, and `only_query` is set to `true` when the query bit was present. This mirrors the existing `signed_tx_version` / `QUERY_VERSION_BASE` logic already used in hash computation.

### Proof of Concept

1. Deploy a standard account contract whose `__validate__` verifies `ecdsa_verify(pubkey, tx_hash, sig_r, sig_s)`.
2. Construct an Invoke v1 transaction with version `0x100000000000000000000000000000001` (query version).
3. Compute the *query* hash: `H(... version=0x100000000000000000000000000000001 ...)` and sign it.
4. Call `starknet_estimateFee` with this transaction and no `SKIP_VALIDATE` flag.
5. The sequencer converts with `only_query = false`, computes the *regular* hash `H(... version=0x1 ...)`, and passes it to `__validate__`.
6. `ecdsa_verify` fails (signature was over the query hash, not the regular hash).
7. `estimate_fee` returns `TRANSACTION_EXECUTION_ERROR` with a validation failure, even though the transaction is valid and would succeed on-chain.

The divergent value is: the RPC returns an error where it should return a valid `FeeEstimation` struct, and the transaction version/hash seen by `__validate__` during estimation differs from what the contract would see during actual execution. pub fn signed_tx_version(
    tx_version: &TransactionVersion,
    transaction_options: &TransactionOptions,
) -> TransactionVersion {
    // If only_query is true, set the 128-th bit.
    let query_only_bit = *QUERY_VERSION_BASE;
    assert_eq!(
        tx_version.0.to_biguint() & query_only_bit.to_biguint(),
        BigUint::from(0_u8),
        "Requested signed tx version with version that already has query bit set: {tx_version:?}."
    );
    if transaction_options.only_query {
        TransactionVersion(tx_version.0 + query_only_bit)
    } else {
        *tx_version
    }
}
