### Title
Stale `lastLiquidationFees` Reduces Effective Insurance During Subaccount Finalization, Causing Premature Socialization — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`lastLiquidationFees` is a global storage variable set during every liquidation step across all subaccounts. When `_finalizeSubaccount` is called, it unconditionally subtracts `lastLiquidationFees` from the effective insurance to determine coverage capacity. If `lastLiquidationFees` was set by a *different* subaccount's liquidation step (not the one being finalized), the effective insurance is artificially deflated, causing the protocol to socialize losses to other depositors/traders even though the insurance fund has sufficient capacity to cover the bad debt.

---

### Finding Description

In `ClearinghouseLiq.sol`, `_finalizeSubaccount` computes the effective insurance as:

```solidity
v.insurance = insurance;
v.insurance -= lastLiquidationFees;
v.canLiquidateMore = (quoteBalance.amount + v.insurance) > 0;
``` [1](#0-0) 

The intent (per the inline comment) is to exclude the fees earned during the *last liquidation step of the current subaccount*, because those fees were just added to `insurance` and would otherwise prevent the socialization path from triggering. The value is added back at the end:

```solidity
v.insurance += lastLiquidationFees;
insurance = v.insurance;
``` [2](#0-1) 

However, `lastLiquidationFees` is a global storage variable set in `_handleLiquidationPayment` for every liquidation step across every subaccount:

```solidity
insurance += v.liquidationFees;
...
lastLiquidationFees = v.liquidationFees;
``` [3](#0-2) 

It is **never reset** between different subaccounts. Finalization (`productId == type(uint32).max`) and the preceding liquidation steps are separate transactions. Between those transactions, any other subaccount can be liquidated, overwriting `lastLiquidationFees` with an unrelated value.

When `_finalizeSubaccount` is subsequently called for the original subaccount, `v.insurance` is computed using the stale `lastLiquidationFees` from the unrelated liquidation. This causes:

1. **`v.canLiquidateMore` is incorrectly `false`** — the guard that requires all spot assets to be zero before finalization is skipped.
2. **`perpEngine.socializeSubaccount` receives a deflated insurance value** — less insurance is applied to cover negative `vQuoteBalance`, causing more perp PnL to be socialized across all open-interest holders.
3. **`insuranceCover` is reduced** — less insurance covers the liquidatee's negative quote balance.
4. **`spotEngine.socializeSubaccount` is triggered prematurely** — depositors absorb losses that the insurance fund could have covered. [4](#0-3) 

---

### Impact Explanation

Other depositors and perp traders take socialized losses that should have been absorbed by the insurance fund. The insurance fund balance is ultimately restored (because `lastLiquidationFees` is added back at the end), but the intermediate accounting is wrong: the wrong amount of insurance is applied to cover bad debt, and the socialization path is triggered when it should not be. This is a direct, concrete loss to innocent participants — analogous to the Ajna finding where deposits were written off while reserve funds existed.

---

### Likelihood Explanation

Moderate-to-high. In any volatile market period, multiple subaccounts are simultaneously under liquidation. A liquidator (or any market participant) can liquidate subaccount A (setting `lastLiquidationFees` to a large value) and then immediately call finalization on subaccount B. No special privileges are required — `liquidateSubaccountImpl` is reachable by any caller through the `Endpoint` via `liquidateSubaccount`. [5](#0-4) 

---

### Recommendation

Scope `lastLiquidationFees` to the subaccount being liquidated, not as a global variable. One approach: pass the liquidatee's identifier alongside `lastLiquidationFees` and only apply the deduction in `_finalizeSubaccount` if the stored liquidatee matches `txn.liquidatee`. Alternatively, reset `lastLiquidationFees` to zero at the start of each `liquidateSubaccountImpl` call before any liquidation step is taken for a new subaccount.

---

### Proof of Concept

1. Subaccount A is partially liquidated. `lastLiquidationFees` is set to `F_A` (a large value from a large position). `insurance` is updated to include `F_A`.
2. In a separate transaction, a liquidator calls `liquidateSubaccountImpl` with `productId = type(uint32).max` for subaccount B (which has a negative quote balance that `insurance` could cover).
3. Inside `_finalizeSubaccount` for B:
   - `v.insurance = insurance - F_A` — artificially low, possibly negative.
   - `v.canLiquidateMore = (quoteBalance_B + v.insurance) > 0` — evaluates to `false` even if `insurance` alone could cover B's debt.
   - `perpEngine.socializeSubaccount(B, v.insurance)` — called with deflated or negative insurance; perp traders absorb losses.
   - `insuranceCover = min(v.insurance, -quoteBalance_B)` — zero or minimal; insurance does not cover B's negative quote.
   - `spotEngine.socializeSubaccount(B)` — triggered; spot depositors absorb losses.
4. At the end, `insurance = v.insurance + F_A` — the insurance balance is numerically restored, but the socialization damage to other participants is irreversible. [6](#0-5)
