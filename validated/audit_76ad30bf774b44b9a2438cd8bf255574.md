### Title
Regular Subaccount Spoofs Isolated Subaccount Type via `isIsolatedSubaccount` Sentinel Collision - (File: `core/contracts/libraries/RiskHelper.sol`)

---

### Summary

`RiskHelper.isIsolatedSubaccount` identifies isolated subaccounts by checking whether the lowest 3 bytes of a `bytes32` subaccount identifier equal the magic sentinel `0x696F73` ("ios", decimal `6910831`). Because regular subaccounts are formed as `bytes32(abi.encodePacked(msg.sender, subaccountName))` where `subaccountName` is a user-controlled `bytes12`, any user can craft a regular subaccount whose last 3 bytes match the sentinel. This causes the regular subaccount to be misclassified as isolated throughout the protocol, corrupting fee accounting in `TransferQuote` and breaking linked-signer resolution.

---

### Finding Description

**Isolated subaccount format** (constructed in `OffchainExchange.createIsolatedSubaccount`):

```
| address (20 bytes) | reserved (6 bytes) | productId (2 bytes) | id (1 byte) | 0x696F73 (3 bytes) |
```

The sentinel `6910831 = 0x696F73` occupies the lowest 3 bytes of the `bytes32`.

**Regular subaccount format** (constructed in `Endpoint.depositCollateral`):

```
bytes32(abi.encodePacked(msg.sender, subaccountName))  // subaccountName is bytes12
```

The lowest 3 bytes of the `bytes32` are the last 3 bytes of `subaccountName`.

**Detection predicate** in `RiskHelper.isIsolatedSubaccount`:

```solidity
return uint256(subaccount) & 0xFFFFFF == 6910831;
```

This check is purely positional — it does not verify any structural property that only a genuine isolated subaccount can satisfy. A user who chooses `subaccountName = bytes12(0x000000000000000000696F73)` (last 3 bytes = `\x69\x6f\x73`) produces a regular subaccount that passes the predicate identically to a real isolated subaccount.

---

### Impact Explanation

**1. `TransferQuote` fee bypass (accounting corruption)**

In `EndpointTx.processTransactionImpl`, the fee charged for a `TransferQuote` depends on `isIsolatedSubaccount`:

```solidity
if (
    RiskHelper.isIsolatedSubaccount(signedTx.tx.recipient) ||
    RiskHelper.isIsolatedSubaccount(signedTx.tx.sender)
) {
    chargeFee(signedTx.tx.sender, HEALTHCHECK_FEE / 10);   // $0.10
} else {
    chargeFee(signedTx.tx.sender, HEALTHCHECK_FEE);         // $1.00
}
```

A user whose regular subaccount has the sentinel suffix pays `HEALTHCHECK_FEE / 10` ($0.10) instead of `HEALTHCHECK_FEE` ($1.00) — a 10× fee reduction — for every `TransferQuote` operation. The sequencer fee accounting (`sequencerFee`) is correspondingly under-credited by $0.90 per call.

**2. Linked-signer resolution broken**

`EndpointTx.getLinkedSigner` routes isolated subaccounts through the parent-subaccount lookup:

```solidity
return
    RiskHelper.isIsolatedSubaccount(subaccount)
        ? linkedSigners[
            IOffchainExchange(offchainExchange).getParentSubaccount(subaccount)
          ]
        : linkedSigners[subaccount];
```

For a crafted regular subaccount, `parentSubaccounts[subaccount]` is `bytes32(0)`, so the lookup returns `linkedSigners[bytes32(0)]` instead of the user's actual linked signer. Any transaction that relies on the linked signer for signature validation (`validateSignedTx` with `allowLinkedSigner = true`) will fail or resolve to an unintended address.

**3. Deposit path blocked**

`Endpoint.depositCollateralWithReferral` rejects deposits to any subaccount that passes `isIsolatedSubaccount`:

```solidity
require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);
```

A crafted regular subaccount cannot receive collateral via the standard deposit path, permanently locking it out of the normal funding flow.

---

### Likelihood Explanation

The attack requires only choosing a `bytes12` subaccount name whose last 3 bytes are `\x69\x6f\x73`. This is trivially achievable by any user interacting with `Endpoint.depositCollateral` or by submitting a `TransferQuote` with a crafted recipient (which calls `_recordSubaccount` without the isolation guard). No privileged access, oracle manipulation, or sequencer compromise is required. The fee bypass is directly profitable and repeatable.

---

### Recommendation

Replace the 3-byte magic-number sentinel with a structural check that cannot be replicated by a user-controlled `bytes12` name. One approach: reserve a dedicated bit in the `bytes32` layout that is never reachable from `abi.encodePacked(address, bytes12)` — for example, use a bit in the address-adjacent reserved region that is always zero for regular subaccounts. Alternatively, maintain an explicit on-chain registry (`mapping(bytes32 => bool) isIsolated`) populated only by `createIsolatedSubaccount`, and replace all `isIsolatedSubaccount` call sites with a registry lookup.

---

### Proof of Concept

1. User controls address `0xAAAA...AAAA`.
2. User chooses `subaccountName = bytes12(0x000000000000000000696F73)`.
3. Regular subaccount = `bytes32(abi.encodePacked(0xAAAA...AAAA, 0x000000000000000000696F73))`.
4. `uint256(subaccount) & 0xFFFFFF` = `0x696F73` = `6910831` → `isIsolatedSubaccount` returns `true`.
5. User gets this subaccount registered as a recipient via a `TransferQuote` (the `_recordSubaccount` call precedes the isolation check).
6. User submits subsequent `TransferQuote` transactions with this subaccount as `sender`; the sequencer charges `HEALTHCHECK_FEE / 10` ($0.10) instead of `HEALTHCHECK_FEE` ($1.00).
7. `getLinkedSigner` for this subaccount returns `linkedSigners[bytes32(0)]` rather than the user's registered linked signer, breaking any linked-signer-authenticated operations.
