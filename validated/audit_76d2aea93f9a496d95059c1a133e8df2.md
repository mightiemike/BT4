### Title
Difficulty Retarget Uses Mainchain Block Instead of Fork Ancestor — (`contract/src/bitcoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

Both `get_next_work_required` implementations look up the retarget interval's first block by **mainchain height** (`get_header_by_height`) rather than by walking back through the **actual chain ancestry** of the block being validated. When a fork diverges before a difficulty-adjustment boundary, the mainchain block at that height is a different block with a different timestamp than the fork's true ancestor. This produces an incorrect `actual_time_taken`, which yields a wrong expected difficulty target. An unprivileged relayer can exploit this to submit fork blocks that pass `check_pow` with a lower-than-correct PoW requirement.

---

### Finding Description

**Bitcoin path — `contract/src/bitcoin.rs`, `get_next_work_required` (lines 78–86):**

```rust
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);

let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
```

`get_header_by_height` is implemented in `contract/src/lib.rs` (lines 677–682):

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

This always returns the **mainchain** block at `first_block_height`. When a fork diverges at or before that height, the mainchain block there is a completely different block from the fork's ancestor at the same height. The two blocks can have arbitrarily different timestamps (within the per-block timestamp validity window accumulated over many blocks).

**Dogecoin path — `contract/src/dogecoin.rs`, `get_next_work_required` (lines 286–295):**

```rust
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

The codebase itself acknowledges this with a `TODO` comment directly above (line 291):

> `// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor`

In the post-145,000 Dogecoin era, `difficulty_adjustment_interval = 1`, so **every single block** triggers this lookup, making the surface much wider than Bitcoin's 2016-block boundary.

**Root cause (analog to external report):**
The external report uses *total LP supply* (a global aggregate) instead of *active LP positions* (the specific subset relevant to the calculation). Here, the code uses the *mainchain block at height H* (a global/canonical value) instead of the *fork's actual ancestor at height H* (the specific value relevant to the fork being validated). Both bugs substitute a broader, incorrect reference value for the narrower, correct one.

---

### Impact Explanation

The `calculate_next_work_required` function computes:

```
actual_time_taken = prev_block_time - first_block_time
new_target = prev_target * actual_time_taken / pow_target_timespan
```

If the mainchain's block at `first_block_height` has an **earlier** timestamp than the fork's true ancestor at that height, then `actual_time_taken` is inflated, `new_target` is inflated (easier difficulty), and the `expected_bits` returned to `check_pow` encodes a lower PoW requirement than the protocol mandates. The attacker's fork block, carrying `bits` matching this easier target, passes:

```rust
require!(
    expected_bits == block_header.bits,
    "bad-diffbits: incorrect proof of work"
);
// ...
require!(
    U256::from_le_bytes(&pow_hash.0) <= target_from_bits(header.bits),
    "block should have correct pow"
);
```

A fork built with under-difficulty blocks can still accumulate `chain_work` (computed from `bits`) and, if it surpasses the mainchain's `chain_work`, trigger `reorg_chain`. This corrupts the canonical chain stored in `mainchain_height_to_header` / `mainchain_header_to_height`, which is the authoritative source for `verify_transaction_inclusion` and `verify_transaction_inclusion_v2`. SPV proofs verified against the corrupted chain return incorrect results.

---

### Likelihood Explanation

- `submit_blocks` is callable by any unprivileged NEAR account (the `#[trusted_relayer]` macro only gates staking/slashing, not block submission itself for accounts that pass the relayer check, and the `UnrestrictedSubmitBlocks` role bypasses even that).
- Fork submission is an explicitly supported code path (`store_fork_header`, `reorg_chain`).
- The attacker only needs to pre-build a fork chain that diverges before a difficulty-adjustment boundary. For Dogecoin post-145,000, every block is a boundary, so the attack surface is continuous.
- Timestamp manipulation within the MTP-lower / local-time-upper window is feasible over a long fork chain; the cumulative drift can be significant.
- No privileged key, social engineering, or external dependency is required.

---

### Recommendation

Replace the `get_header_by_height` call with an ancestor walk that follows `prev_block_hash` links from `prev_block_header` back by `difficulty_adjustment_interval - 1` steps. This ensures the retarget calculation always uses the fork's own ancestor, not the mainchain block at the same height. The Dogecoin `TODO` comment already identifies this as a known concern.

---

### Proof of Concept

1. Deploy the contract (Bitcoin feature) with genesis at height 0, `difficulty_adjustment_interval = 2016`.
2. Submit mainchain blocks 0–2016 normally; mainchain block at height 0 has timestamp `T_main`.
3. Submit a fork block at height 1 whose timestamp is `T_fork > T_main` (valid: > MTP, ≤ local time + 2h). Continue the fork to height 2015.
4. At height 2016, `get_next_work_required` fires. It fetches `get_header_by_height(0)` → mainchain genesis with timestamp `T_main`. The fork's actual ancestor at height 0 is the same genesis block here, but for a fork diverging at height 1, the ancestor at height 1 differs.
5. For a fork diverging at height 1: `first_block_height = 2016 - 2015 = 1`. `get_header_by_height(1)` returns the **mainchain** block at height 1 (timestamp `T_main_1`). The fork's ancestor at height 1 has timestamp `T_fork_1`. If `T_main_1 < T_fork_1`, `actual_time_taken` is larger, target is easier.
6. Craft the fork's block at height 2016 with `bits` encoding the easier target. `check_pow` accepts it. Continue mining the fork at this easier difficulty until `chain_work` exceeds the mainchain tip, triggering `reorg_chain`.
7. `verify_transaction_inclusion` now resolves against the attacker-controlled fork. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```rust
fn calculate_next_work_required(
    config: &NetworkConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let prev_block_time: i64 = prev_block_header.block_header.time.into();

    let mut actual_time_taken: i64 = prev_block_time - first_block_time;
    if actual_time_taken < config.pow_target_timespan / 4 {
        actual_time_taken = config.pow_target_timespan / 4;
    }
    if actual_time_taken > config.pow_target_timespan * 4 {
        actual_time_taken = config.pow_target_timespan * 4;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(actual_time_taken).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target = new_target
        / U256::from(<i64 as TryInto<u64>>::try_into(config.pow_target_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()
```
