### Title
GC-Before-Measurement Order Allows Zero-Cost Storage Deposit in `submit_blocks` - (File: `contract/src/lib.rs`)

### Summary

`submit_blocks` records `initial_storage` before processing headers, then calls `run_mainchain_gc(num_of_headers)` — which removes old headers from storage — **before** computing `diff_storage_usage`. Because GC runs first and frees storage, `env::storage_usage().saturating_sub(initial_storage)` can evaluate to `0`, making `required_deposit = 0`. A submitter can add N new headers to on-chain storage while paying nothing, with the storage cost silently absorbed by the freed deposits of previous submitters.

### Finding Description

In `submit_blocks`, the storage accounting follows this sequence:

```
initial_storage = env::storage_usage()          // snapshot BEFORE
submit N headers                                 // storage grows by ~N * header_size
run_mainchain_gc(N)                              // removes up to N old headers; storage shrinks
diff_storage_usage = env::storage_usage()
                     .saturating_sub(initial_storage)  // can be 0
required_deposit = storage_byte_cost * diff_storage_usage  // can be 0
require!(amount >= required_deposit)             // 0 >= 0 always passes
``` [1](#0-0) 

Inside `run_mainchain_gc`, when `amount_of_headers_we_store > gc_threshold`, it removes `selected_amount_to_remove = min(total_amount_to_remove, batch_size)` headers. Since `batch_size = num_of_headers = N`, whenever the backlog exceeds `gc_threshold` by at least N, GC removes exactly N headers. [2](#0-1) 

Because each header occupies roughly the same storage, removing N old headers and adding N new headers produces a net storage delta of ≈ 0. `saturating_sub` then returns `0`, `required_deposit = 0`, and the deposit check `amount >= 0` is trivially satisfied with zero attached NEAR. [3](#0-2) 

The structural analog to the external report is exact: the external report's `_deposit()` measures a "before" balance, then an external call (ERC777 callback) corrupts the "after" balance before it is read. Here, `initial_storage` is the "before" snapshot, and `run_mainchain_gc` is the intermediate state-mutating operation that corrupts the "after" snapshot before `diff_storage_usage` is computed.

### Impact Explanation

Every new header stored on-chain consumes NEAR storage that must be paid for. When GC is active (i.e., the chain has grown past `gc_threshold`), the storage freed by removing old headers — which was funded by previous submitters' deposits — offsets the cost of new headers in the accounting formula. The current submitter pays nothing for the new storage they occupy. Over time this drains the contract's storage reserve, which was funded by earlier depositors, and can leave the contract unable to cover its own storage costs.

### Likelihood Explanation

The condition `amount_of_headers_we_store > gc_threshold` is permanently satisfied once the contract has been live long enough to accumulate `gc_threshold` headers (e.g., the recommended value of 52,704 ≈ one year of Bitcoin blocks). After that point, **every** call to `submit_blocks` triggers GC, and the zero-cost path is always reachable. The `#[trusted_relayer]` attribute narrows the caller set, but the `bypass_roles(Role::UnrestrictedSubmitBlocks)` escape hatch means any account granted that role can also reach this path. Trusted relayers themselves are the primary submitters and are directly exposed. [4](#0-3) 

### Recommendation

Move `run_mainchain_gc` to **after** the storage cost check and refund, so that the deposit accounting reflects only the net new storage added by the submitted headers:

```rust
// 1. snapshot
let initial_storage = env::storage_usage();

// 2. submit headers
for header in headers { self.submit_block_header(...); }

// 3. measure and charge BEFORE GC mutates storage
let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());
require!(amount >= required_deposit, ...);

// 4. refund
let refund = amount.saturating_sub(required_deposit);
if refund > NearToken::from_near(0) { ... }

// 5. GC runs after accounting is settled
self.run_mainchain_gc(num_of_headers);
```

### Proof of Concept

**Setup**: Contract initialized with `gc_threshold = 52_704`. After one year of operation, `amount_of_headers_we_store = 52_705` (one over threshold). `total_amount_to_remove = 1`.

**Attack** (attacker submits 1 header with 0 attached NEAR):

1. `initial_storage = S` (before any changes).
2. 1 new header is stored → `storage_usage = S + Δ`.
3. `run_mainchain_gc(1)` fires: `selected_amount_to_remove = min(1, 1) = 1`. One old header is deleted → `storage_usage = S + Δ - Δ_old ≈ S`.
4. `diff_storage_usage = S.saturating_sub(S) = 0`.
5. `required_deposit = 0`.
6. `require!(0 >= 0)` passes.
7. Attacker's 0-NEAR deposit is "refunded" (no-op). New header is permanently stored at zero cost.

**Scaled attack**: Submit a batch of N headers. GC removes N old headers. Net storage delta = 0. N headers stored for free. The freed storage (and its associated deposit) belonged to previous submitters.
