### Title
Fork Difficulty Miscalculation Uses Mainchain Ancestor Instead of Fork Ancestor - (`contract/src/dogecoin.rs`)

### Summary

When validating a fork block's proof-of-work, the difficulty retarget calculation fetches the reference block's timestamp from the **mainchain** by height rather than walking the fork's own ancestor chain. For Dogecoin (retarget interval = 1 block), this produces a wrong `first_block_time` for any fork block that is two or more blocks deep, yielding an incorrect expected `bits` value. An attacker who controls fork block timestamps can exploit the discrepancy to have the contract accept fork blocks at an easier difficulty than the protocol requires.

### Finding Description

`get_next_work_required` in `contract/src/dogecoin.rs` computes `height_first = prev_block_height - blocks_to_go_back` and then calls `blocks_getter.get_header_by_height(height_first)` to obtain the reference timestamp for the retarget formula. The implementation of `get_header_by_height` always reads from `mainchain_height_to_header`, the canonical chain index:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

The developers themselves flagged this with a `TODO` comment directly above the call site:

```rust
// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

For Dogecoin after block 145 000, `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` and `height_first = prev_block_height − 1`. Consider a fork that diverges at mainchain height H (the fork's first new block is at H+1):

| Fork depth | Block being validated | `height_first` | Mainchain block at that height | Fork ancestor at that height | Same? |
|---|---|---|---|---|---|
| 1 | H+1 | H | mainchain[H] | fork[H] = mainchain[H] | ✓ |
| 2 | H+2 | H+1 | mainchain[H+1] | fork[H+1] ≠ mainchain[H+1] | **✗** |
| 3 | H+3 | H+2 | mainchain[H+2] | fork[H+2] ≠ mainchain[H+2] | **✗** |

From depth 2 onward, `first_block_time` is taken from the mainchain block at `height_first`, not from the fork's own ancestor. The retarget formula in `calculate_next_work_required` is:

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
```

If the mainchain block at `height_first` has an earlier timestamp than the fork's ancestor at the same height, `modulated_timespan` is inflated, the computed target rises (easier difficulty), and the contract accepts the fork block at a lower work threshold than the protocol mandates.

### Impact Explanation

The contract's core security guarantee is that `mainchain_tip_blockhash` always points to the chain with the greatest cumulative proof-of-work. If fork blocks are accepted at an artificially easy difficulty, their `chain_work` is computed from their declared `bits` (which is validated against the wrong expected value), allowing a fork to accumulate apparent chainwork faster than honest miners. A successful reorg promotes the attacker's chain to the mainchain, corrupting every subsequent `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` result returned to downstream consumers.

### Likelihood Explanation

Dogecoin retargets every block, so the bug is reachable for any fork two or more blocks deep — a routine occurrence during normal relayer operation. The attacker only needs to be a NEAR caller with access to `submit_blocks` (no privileged role required; the function is `#[trusted_relayer]` but the relayer role is open to staked participants). The attacker observes the mainchain timestamps, computes the expected difficulty deviation, and crafts fork block timestamps within the valid MTP/future-time window to maximise the discrepancy.

### Recommendation

Replace the `get_header_by_height` call with an ancestor walk that follows `prev_block_hash` links through `headers_pool` until the target height is reached. This ensures the reference timestamp always belongs to the block that is actually an ancestor of the block being validated, regardless of whether that block is on the mainchain or a fork. The same fix should be applied to the analogous calls in `contract/src/bitcoin.rs` and `contract/src/litecoin.rs` for consistency, even though their 2016-block retarget interval makes exploitation far less practical.

### Proof of Concept

1. Mainchain has blocks at heights H, H+1, H+2 with timestamps T_H, T_H1, T_H2.
2. Attacker submits fork block F1 at height H+1 (prev = mainchain[H]) with timestamp T_F1 < T_H1.
3. Attacker submits fork block F2 at height H+2 (prev = F1). The contract calls `get_next_work_required` with `prev = F1`, computes `height_first = H+1`, and fetches `mainchain[H+1].time = T_H1` instead of `F1.time = T_F1`.
4. `modulated_timespan = T_F1 − T_H1`. Since T_F1 < T_H1, this is negative before clamping, so it is clamped to `min_timespan`, yielding the minimum possible difficulty increase — or, if T_H1 is sufficiently large relative to T_F1, the clamped value produces a higher target (easier mining) than the correct fork-based calculation would.
5. The contract enforces `require!(expected_bits == block_header.bits)` against this wrong expected value, so F2 with the attacker-chosen `bits` passes validation.
6. Repeated for F3, F4, … the fork accumulates chainwork at reduced cost and eventually triggers `reorg_chain`, replacing the honest mainchain. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)
