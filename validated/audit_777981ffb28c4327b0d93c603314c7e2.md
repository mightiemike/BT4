### Title
`GasPrice::wei_to_fri` silently returns zero on a zero conversion rate, bypassing the oracle fallback and causing block-production DoS - (File: crates/starknet_api/src/block.rs)

### Summary

`GasPrice::wei_to_fri` does not check whether `eth_to_fri_rate` is zero. When the oracle returns `0`, the function silently returns `Ok(GasPrice(0))` instead of an error. This causes `get_l1_prices_in_fri_and_wei_and_conversion_rate` to treat the zero-rate oracle response as a successful conversion, skip the fallback to previous-block prices, and propagate zero FRI prices into block info — ultimately crashing block production.

### Finding Description

`GasPrice::fri_to_wei` correctly guards against a zero rate:

```rust
.checked_div(eth_to_fri_rate)
.ok_or_else(|| StarknetApiError::GasPriceConversionError(
    "FRI to ETH rate must be non-zero".to_string()))
``` [1](#0-0) 

`GasPrice::wei_to_fri` has no such guard. When `eth_to_fri_rate = 0`:

1. `self.checked_mul_u128(0)` → `Some(GasPrice(0))` (zero times anything is zero, no overflow).
2. `GasPrice(0).checked_div(WEI_PER_ETH)` → `Some(GasPrice(0))` (zero divided by a constant is zero, no error).
3. The function returns `Ok(GasPrice(0))` — silently. [2](#0-1) 

The test suite confirms the asymmetry: `fri_to_wei_errors_on_conversion_rate_zero` exists, but there is no `wei_to_fri_errors_on_conversion_rate_zero`. [3](#0-2) 

In `get_l1_prices_in_fri_and_wei_and_conversion_rate`, the oracle result is matched as `Ok(eth_to_fri_rate)`. A zero value satisfies this match. `L1PricesInFri::convert_from_wei` then calls `wei_to_fri(0)`, which returns `Ok(GasPrice(0))` for both L1 gas and L1 data gas prices. Because the conversion returns `Ok`, the code takes the early-return path and **skips the fallback to previous-block prices entirely**:

```rust
if let Ok(prices_in_fri) = l1_gas_prices_fri_result {
    return (prices_in_fri, prices_in_wei, eth_to_fri_rate); // returns (zeros, ..., 0)
}
``` [4](#0-3) 

The zero FRI prices and zero rate are then forwarded to `convert_to_sn_api_block_info`, which calls `NonzeroGasPrice::new(GasPrice(0))` and returns an error, causing block building to fail. [5](#0-4) 

Additionally, if `override_l1_gas_price_fri` or `override_l1_data_gas_price_fri` is configured, the code calls `fri_to_wei(values.2)` where `values.2 = 0`. `fri_to_wei(0)` returns an error, which is handled with `unwrap_or_else(|err| panic!(...))` — a hard crash. [6](#0-5) 

### Impact Explanation

When the oracle returns `eth_to_fri_rate = 0` (transient bug, misconfiguration, or external manipulation), the three-tier fallback chain (live data → previous block → defaults) is short-circuited at tier 1. The proposer cannot build a valid block. Block production halts until the oracle recovers. If price overrides are active, the node panics.

### Likelihood Explanation

The oracle is an external HTTP endpoint. A transient zero response (parsing error, API change, empty response interpreted as 0) is a realistic failure mode. The original bug report's analog is a protocol state where the denominator is zero; here the denominator is an externally-supplied rate that can be zero without any privileged access.

### Recommendation

Add an explicit zero-check in `wei_to_fri`, mirroring `fri_to_wei`:

```rust
pub fn wei_to_fri(self, eth_to_fri_rate: u128) -> Result<GasPrice, StarknetApiError> {
    if eth_to_fri_rate == 0 {
        return Err(StarknetApiError::GasPriceConversionError(
            "ETH to FRI rate must be non-zero".to_string(),
        ));
    }
    Ok(self
        .checked_mul_u128(eth_to_fri_rate)
        .ok_or_else(|| { ... })?
        .checked_div(WEI_PER_ETH)
        .expect("WEI_PER_ETH must be non-zero"))
}
```

This makes `wei_to_fri(0)` return `Err(...)`, so `convert_from_wei` returns `Err`, the `if let Ok` branch is skipped, and the fallback to previous-block prices is correctly triggered.

### Proof of Concept

```rust
use starknet_api::block::GasPrice;

fn main() {
    // fri_to_wei(0) correctly returns Err
    let r1 = GasPrice(5).fri_to_wei(0);
    assert!(r1.is_err(), "fri_to_wei(0) correctly errors");

    // wei_to_fri(0) silently returns Ok(GasPrice(0)) — the bug
    let r2 = GasPrice(5).wei_to_fri(0);
    assert_eq!(r2, Ok(GasPrice(0)), "wei_to_fri(0) silently returns zero!");

    // Consequence: convert_from_wei with rate=0 returns Ok with zero prices,
    // bypassing the fallback in get_l1_prices_in_fri_and_wei_and_conversion_rate.
}
```
