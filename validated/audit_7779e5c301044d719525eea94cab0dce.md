### Title
Lossy Protobuf Round-Trip of `ValidResourceBounds` Corrupts Transaction Hash Domain for `AllResources` Transactions with Zero L2/L1DataGas Bounds — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf serialization of `ValidResourceBounds` is lossy: an `AllResources` transaction with zero `l2_gas` and `l1_data_gas` serializes to an identical protobuf wire encoding as an `L1Gas` (pre-0.13.3) transaction. On deserialization, a zero-check heuristic silently downgrades the variant from `AllResources` to `L1Gas`. Because `get_tip_resource_bounds_hash` includes `L1_DATA_GAS` in the Poseidon preimage only for `AllResources`, the two variants produce different transaction hashes over the same numeric field values. A transaction signed under the `AllResources` hash domain will fail signature verification after a protobuf round-trip, and any re-execution of the stored transaction will use the wrong gas computation mode.

---

### Finding Description

**Serializer** (`crates/apollo_protobuf/src/converters/transaction.rs`, lines 471–489):

```rust
impl From<ValidResourceBounds> for protobuf::ResourceBounds {
    fn from(value: ValidResourceBounds) -> Self {
        match value {
            ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
                l1_gas: Some(l1_gas.into()),
                l2_gas: Some(value.get_l2_bounds().into()),
                l1_data_gas: Some(ResourceBounds::default().into()),  // ← zero
            },
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
                protobuf::ResourceBounds {
                    l1_gas: Some(l1_gas.into()),
                    l2_gas: Some(l2_gas.into()),
                    l1_data_gas: Some(l1_data_gas.into()),
                },
        }
    }
}
```

When `l2_gas = 0` and `l1_data_gas = 0`, both arms produce an identical wire encoding: `{l1_gas: X, l2_gas: 0, l1_data_gas: 0}`. The variant discriminant is not preserved.

**Deserializer** (`crates/apollo_protobuf/src/converters/transaction.rs`, lines 417–436):

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong for AllResources with zero bounds
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })

```

The `unwrap_or_default()` backward-compat shim (for 0.13.2 peers that omit `l1_data_gas`) combined with the zero-check heuristic means any `AllResources` transaction whose `l2_gas` and `l1_data_gas` are both zero is permanently misclassified as `L1Gas` after a protobuf round-trip.

**Hash domain divergence** (`crates/starknet_api/src/transaction_hash.rs`, lines 188–211):

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    ...
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],                          // ← 2 elements
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // ← 3 elements
        }
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```

`Poseidon([tip, L1_GAS, L2_GAS])` ≠ `Poseidon([tip, L1_GAS, L2_GAS, L1_DATA_GAS(zero)])` even when the numeric values of `L2_GAS` and `L1_DATA_GAS` are both zero. The signer commits to the `AllResources` hash (3-element preimage); after protobuf round-trip the verifier recomputes the `L1Gas` hash (2-element preimage).

---

### Impact Explanation

**Hash/signature domain mismatch (Critical — wrong transaction accepted/rejected through account validation):** The transaction was signed over `H_AllResources`. After protobuf round-trip the sequencer or verifier recomputes `H_L1Gas ≠ H_AllResources`. Signature verification fails, causing a valid transaction to be rejected, or — if the stored hash is trusted without re-verification — the wrong hash is bound to the transaction in the receiving node's state.

**Wrong gas computation mode (Critical — incorrect fee/gas/bouncer accounting):** `ValidResourceBounds::L1Gas` → `GasVectorComputationMode::NoL2Gas`; `ValidResourceBounds::AllResources` → `GasVectorComputationMode::All`. Re-execution on the syncing node (e.g., for proof generation via `starknet_transaction_prover`) uses `NoL2Gas` mode, producing different `estimate_minimal_gas_vector`, fee-check, and bouncer-weight results than the originating sequencer, causing state divergence and invalid proofs.

---

### Likelihood Explanation

The triggering condition — a V3 transaction with `l2_gas = 0` and `l1_data_gas = 0` — is explicitly tested and accepted by the gateway. Any user who submits a V3 invoke with only `l1_gas` set (a natural pattern when migrating from V1 transactions) produces this condition. No privileged access or malformed bytes are required; the transaction is valid and accepted end-to-end.

---

### Recommendation

1. **Add a variant discriminant to the protobuf `ResourceBounds` message** (e.g., a boolean `is_all_resources` field or an enum tag) so the serializer can preserve the `L1Gas` vs `AllResources` distinction without relying on value-level heuristics.

2. **Alternatively, derive the variant from the transaction version**: all transactions with version ≥ 0.13.3 must use `AllResources`; the zero-check heuristic should only apply to transactions with version < 0.13.3.

3. **Remove the silent `unwrap_or_default()`** for `l1_data_gas` once 0.13.2 peer support is dropped (as the existing TODO notes), and replace it with an explicit error for missing `l1_data_gas` on post-0.13.3 transactions.
