### Title
Silent NFT Lock on Nil-Response Acknowledgement in `OnAcknowledgementPacket` — (`x/nft-transfer/keeper/relay.go`)

---

### Summary

`keeper.OnAcknowledgementPacket` uses a type-switch on `ack.Response` that treats every non-`Acknowledgement_Error` variant — including a **nil** `Response` — as a successful delivery and returns `nil` without refunding. Because `channeltypes.Acknowledgement` is a protobuf oneof, an empty JSON object `{}` unmarshals successfully with `Response = nil`. A malicious counterparty chain can deliberately write `{}` as the packet acknowledgement, causing the sending chain to silently accept it as success, permanently locking the escrowed (or burning the burned) NFT with no recovery path.

---

### Finding Description

**Entrypoint — `ibc_module.go` `OnAcknowledgementPacket`:** [1](#0-0) 

The only guard on the raw acknowledgement bytes is `UnmarshalJSON`. If that succeeds, the decoded `ack` is passed directly to the keeper. There is no call to `ack.ValidateBasic()` or any check that `ack.Response != nil`.

**Root cause — `keeper/relay.go` `OnAcknowledgementPacket`:** [2](#0-1) 

The switch has exactly two arms:
- `*channeltypes.Acknowledgement_Error` → calls `refundPacketToken`
- `default` → returns `nil` (no refund, treated as success)

`*channeltypes.Acknowledgement_Result` is never explicitly matched; it falls into `default`. So does `nil`. Both are treated identically as "success, do nothing."

**Why `{}` unmarshals to `Response = nil`:**

`channeltypes.Acknowledgement` is a protobuf `oneof response { bytes result = 21; string error = 22; }`. The protobuf JSON codec does not require oneof fields to be present; absent fields default to nil. `{}` is valid JSON and unmarshals without error, producing `Acknowledgement{Response: nil}`.

**IBC core does not block this:**

The IBC core in ibc-go v10 verifies only the commitment hash of the acknowledgement bytes against what was stored on the receiving chain. It does not inspect or validate the internal structure of the bytes before calling the module's `OnAcknowledgementPacket`.

**What happens to the NFT:**

When `SendTransfer` is called, the NFT is either escrowed (source chain) or burned (sink chain): [3](#0-2) 

`refundPacketToken` is the only recovery path — it either unescrows or re-mints: [4](#0-3) 

If `refundPacketToken` is never called (because `default:` returns `nil`), the NFT is permanently lost.

---

### Impact Explanation

A malicious counterparty chain that receives an NFT packet can write `{}` as the IBC acknowledgement. The sending chain's module processes this as a successful delivery, commits the acknowledgement, and never refunds the sender. The NFT is permanently locked in the escrow account (or permanently burned if the sending chain was a sink). There is no retry mechanism — once the acknowledgement is committed, the packet lifecycle is complete.

---

### Likelihood Explanation

This requires a malicious or compromised counterparty chain that deliberately deviates from the ICS-721 protocol by writing a non-standard acknowledgement. This is a realistic IBC threat model: chains are supposed to be isolated such that a malicious counterparty cannot cause permanent asset loss on the sending chain. The attack requires no special privileges on the sending chain — only the ability to control what acknowledgement bytes are written on the receiving chain.

---

### Recommendation

In `keeper.OnAcknowledgementPacket`, explicitly handle `*channeltypes.Acknowledgement_Result` and return an error for any other case (including nil):

```go
func (k Keeper) OnAcknowledgementPacket(..., ack channeltypes.Acknowledgement) error {
    switch ack.Response.(type) {
    case *channeltypes.Acknowledgement_Error:
        return k.refundPacketToken(ctx, packet, data)
    case *channeltypes.Acknowledgement_Result:
        return nil
    default:
        return sdkerrors.Wrapf(sdkerrors.ErrUnknownRequest,
            "invalid acknowledgement response type: %T", ack.Response)
    }
}
```

Returning an error from `OnAcknowledgementPacket` causes the IBC core to revert the transaction, preventing the acknowledgement from being committed and allowing the relayer to retry with a valid acknowledgement (or the timeout path to eventually trigger a refund).

---

### Proof of Concept

1. Chain A sends an NFT via `SendTransfer`. The NFT is escrowed at `GetEscrowAddress(sourcePort, sourceChannel)`.
2. Chain B (malicious) receives the packet. Instead of writing `{"result":"AQ=="}`, it writes `{}` as the acknowledgement.
3. A relayer submits `MsgAcknowledgement` to Chain A with `acknowledgement = []byte("{}")`.
4. IBC core verifies the commitment hash of `{}` matches what Chain B stored — it does.
5. `ibc_module.OnAcknowledgementPacket` is called. `UnmarshalJSON("{}", &ack)` succeeds; `ack.Response == nil`.
6. `keeper.OnAcknowledgementPacket` switch hits `default:` → returns `nil`.
7. Transaction succeeds. Acknowledgement is committed. No refund is issued.
8. Assert: NFT owner is the escrow address (not the original sender). The sender has lost the NFT permanently.

func (k Keeper) refundPacketToken(ctx sdk.Context, packet channeltypes.Packet, data types.NonFungibleTokenPacketData) error {
	sender, err := sdk.AccAddressFromBech32(data.Sender)
	if err != nil {
		return err
	}

	classTrace := types.ParseClassTrace(data.ClassId)
	voucherClassID := classTrace.IBCClassID()

	isAwayFromOrigin := types.IsAwayFromOrigin(packet.GetSourcePort(),
		packet.GetSourceChannel(), data.ClassId)

	escrowAddress := types.GetEscrowAddress(packet.GetSourcePort(), packet.GetSourceChannel())

	if isAwayFromOrigin {
		// unescrow tokens back to the sender
		for _, tokenID := range data.TokenIds {
			if err := k.nftKeeper.TransferOwner(ctx, voucherClassID, tokenID, escrowAddress, sender); err != nil {
				return err
			}
		}
	} else {
		// we are sink chain, mint voucher back to sender
		for i, tokenID := range data.TokenIds {
			if err := k.nftKeeper.MintNFT(ctx, voucherClassID, tokenID, "", data.TokenUris[i], "", escrowAddress, sender); err != nil {
				return err
			}
		}
	}

	return nil
