### Title
`blocking_check_fail_open` Defaults to `true`, Allowing Bypass of the Transaction Blocking Gate When the External Check Service Is Unavailable — (File: `crates/starknet_transaction_prover/src/config.rs`)

---

### Summary

`ProverConfig` and `RawServiceConfig` both hard-code `blocking_check_fail_open: true` as their default. When an operator enables the external compliance/security gate by setting `blocking_check_url`, any network failure, timeout, or non-10000 error from that service causes the prover to proceed with generating a proof (fail-open). An attacker who can make the blocking check service unreachable or slow enough to exceed the 2-second default timeout can bypass the intended gate and obtain a cryptographic proof for a transaction that the operator intended to block.

---

### Finding Description

`ProverConfig::default()` in `crates/starknet_transaction_prover/src/config.rs` sets:

```rust
blocking_check_fail_open: true,   // line 48
```

`RawServiceConfig::default()` in `crates/starknet_transaction_prover/src/server/config.rs` mirrors this:

```rust
blocking_check_fail_open: true,   // line 120
```

Both configs also default `blocking_check_url: None`, so the blocking check is entirely disabled until an operator explicitly sets a URL. The moment an operator sets `blocking_check_url` to enable the gate, `fail_open: true` silently remains in effect unless the operator also explicitly overrides it.

The decision logic in `prove_with_blocking_check` (`crates/starknet_transaction_prover/src/proving/virtual_snos_prover.rs`, lines 256–263) is:

```rust
Ok(BlockingCheckResult::Inconclusive) => {
    info!(fail_open = client.fail_open, "Blocking check inconclusive");
    (client.fail_open, None)          // true → allow
}
Err(_) => {
    info!(fail_open = client.fail_open, "Blocking check timed out");
    (client.fail_open, None)          // true → allow
}
```

`BlockingCheckResult::Inconclusive` is returned for every non-10000 error code, every network failure, and every deserialization failure (lines 116, 126, 133, 141, 152 of `blocking_check.rs`). With `fail_open = true`, all of these outcomes silently allow the transaction to proceed to proving.

The `BlockingCheckClient` is constructed with the config value passed directly:

```rust
BlockingCheckClient::new(
    url,
    prover_config.blocking_check_timeout_millis,
    prover_config.blocking_check_fail_open,   // true by default
)
```

The default timeout is 2000 ms. An attacker who can delay or drop the blocking check service's responses for 2 seconds per request will cause every check to time out and every transaction to be proved.

---

### Impact Explanation

**High — RPC service returns an authoritative-looking wrong value.**

The `starknet_proveTransaction` RPC endpoint returns a cryptographic proof and `proof_facts` for a transaction that the operator's compliance gate was supposed to block. The proof is indistinguishable from a legitimately screened proof. Any downstream consumer (L1 verifier, bridge, compliance auditor) that relies on the prover having enforced the blocking check will accept the proof as valid authorization.

The `additional_data` field (the screening signature relayed from the blocking check's allow response) is absent when the gate is bypassed via fail-open, but the proof itself is present and cryptographically valid, making the bypass invisible to consumers who do not inspect `additional_data`.

---

### Likelihood Explanation

**Medium.** The precondition is that the operator has configured `blocking_check_url` but has not explicitly set `blocking_check_fail_open: false`. Given that `true` is the documented default and the CLI flag description says "default: true", operators who read the documentation may intentionally leave it as-is, not realizing that fail-open defeats the purpose of the gate. The attacker's required capability — making the blocking check service unreachable or slow for 2 seconds — is achievable via network-level interference (DDoS of the check endpoint, BGP manipulation, or exploiting any availability weakness in the check service itself).

---

### Recommendation

Change the default for `blocking_check_fail_open` to `false` in both `ProverConfig::default()` and `RawServiceConfig::default()`. Fail-close is the safe default for a security gate: if the gate cannot render a verdict, the transaction should not be proved. Operators who explicitly want fail-open behavior for availability reasons can opt in by setting the flag to `true`.

```rust
// crates/starknet_transaction_prover/src/config.rs
impl Default for ProverConfig {
    fn default() -> Self {
        Self {
            // ...
            blocking_check_fail_open: false, // Fail-close by default; opt in to fail-open explicitly.
        }
    }
}
```

```rust
// crates/starknet_transaction_prover/src/server/config.rs
impl Default for RawServiceConfig {
    fn default() -> Self {
        Self {
            // ...
            blocking_check_fail_open: false,
        }
    }
}
```

Update the CLI argument description accordingly:

```
/// Fail-open when blocking check is inconclusive (default: false). Set to true for fail-open.
```

### Proof of Concept

1. Operator deploys the prover with `blocking_check_url` set to a compliance screening service and does not set `blocking_check_fail_open` (defaults to `true`).
2. Attacker identifies a transaction `T` that the compliance service would block (returns error code 10000).
3. Attacker floods the compliance service endpoint with requests, causing it to be slow or unreachable.
4. Attacker submits `T` to `starknet_proveTransaction`.
5. `prove_with_blocking_check` spawns the proof task and calls `check_transaction` with a 2000 ms timeout.
6. The compliance service does not respond within 2000 ms (due to the flood).
7. The `tokio::time::timeout` arm fires → `(client.fail_open, None)` = `(true, None)` → `allow = true`.
8. The background proof task completes; the prover returns a valid `ProveTransactionResult` with `proof` and `proof_facts` populated and `additional_data: None`.
9. The attacker receives a cryptographic proof for a transaction the operator intended to block.
