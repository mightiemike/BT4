### Title
Permanent Pause Lock with No Time-Based Fallback Disables Proof Verification — (`contract/src/lib.rs`)

---

### Summary

The `BtcLightClient` contract's pause mechanism is a single point of failure. If the contract is paused and all accounts holding `Role::PauseManager` lose access to their keys, the contract remains permanently paused. Unlike `submit_blocks` (which has a `trusted_relayer` bypass for `Role::DAO` / `Role::UnrestrictedSubmitBlocks`) and `run_mainchain_gc` (which has `except(roles(Role::UnrestrictedRunGC))`), the proof-verification endpoints `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` carry a bare `#[pause]` with **no bypass roles and no time-based escape hatch**. There is no mechanism for any unprivileged caller to restore service after a prolonged pause.

---

### Finding Description

The contract declares its pause manager as a single role:

```rust
#[pausable(manager_roles(Role::PauseManager))]
``` [1](#0-0) 

Only an account holding `Role::PauseManager` can call `pa_unpause()`. The two proof-verification functions are gated by a bare `#[pause]` with no exception roles:

```rust
#[pause]
pub fn verify_transaction_inclusion(&self, ...) -> bool { ... }

#[pause]
pub fn verify_transaction_inclusion_v2(&self, ...) -> bool { ... }
``` [2](#0-1) [3](#0-2) 

Contrast this with `run_mainchain_gc`, which explicitly provides a bypass:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) { ... }
``` [4](#0-3) 

And `submit_blocks`, which gains a bypass through the `trusted_relayer` macro's `bypass_roles`:

```rust
#[trusted_relayer(
    bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks),
    ...
)]
``` [5](#0-4) 

No equivalent bypass exists for the proof-verification functions. The super-admin set during `init()` is the **contract account itself**, not a human-controlled account:

```rust
contract.acl_init_super_admin(env::current_account_id())
``` [6](#0-5) 

If the contract account's full-access keys are deleted or inaccessible, no one can grant a new `PauseManager` role, and no time-based fallback exists to allow any caller to unpause after a delay.

---

### Impact Explanation

`verify_transaction_inclusion_v2` is the primary production endpoint for Bitcoin transaction inclusion proofs. Any downstream NEAR contract that calls it to gate asset releases, bridge withdrawals, or cross-chain actions will receive a permanent panic/rejection while the contract is paused. The broken invariant is: **a valid Merkle proof for a confirmed Bitcoin transaction must always be verifiable by an unprivileged caller**. A permanent pause violates this invariant with no recovery path available to any non-privileged party.

---

### Likelihood Explanation

The scenario requires: (1) a legitimate or emergency pause by `PauseManager`, followed by (2) incapacitation or key loss of all `PauseManager` (and `DAO`) accounts. This is low-probability but non-zero — key loss, organizational failure, or a compromised multisig are realistic over a long operational horizon. The Fantom audit treated the analogous scenario as a reportable finding at the same likelihood level.

---

### Recommendation

Add a time-based escape hatch to `pa_unpause` (or a dedicated `force_unpause` function) that allows any caller to unpause after a sufficiently long period has elapsed since the pause was initiated — analogous to the Fantom recommendation:

```rust
require!(
    self.acl_has_role(Role::PauseManager, &env::predecessor_account_id())
        || env::block_timestamp() > pause_initiated_at + PAUSE_TIMEOUT_NS,
    "Not authorized to unpause"
);
```

Alternatively, ensure `Role::DAO` is explicitly listed in `manager_roles` for the `Pausable` plugin so that the DAO multisig can always recover from a lost `PauseManager` key.

---

### Proof of Concept

1. Deploy `BtcLightClient` on NEAR testnet; grant `Role::PauseManager` to account `alice.testnet`.
2. `alice.testnet` calls `pa_pause()` — contract is now paused.
3. `alice.testnet` loses her private key (or the account is deleted).
4. Any caller invokes `verify_transaction_inclusion_v2(...)` — the `#[pause]` guard panics unconditionally because no bypass role is configured and no time-based fallback exists.
5. All downstream contracts relying on this endpoint for Bitcoin proof verification are permanently broken with no on-chain recovery path available to any unprivileged party.
