Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the real user when swaps route through `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool populates with `msg.sender` of the `pool.swap()` call. When `MetricOmmSimpleRouter` is used, `msg.sender` to the pool is the router contract, not the end user. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`, breaking the intended per-user curation policy in both directions: allowlist bypass when the router is allowlisted, and complete router unusability when it is not.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` directly as the `sender` argument to `_beforeSwap()`: [1](#0-0) 

`ExtensionCalling._beforeSwap()` forwards that value unchanged to the extension hook: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]` — where `msg.sender` is the pool (correct) and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly with no explicit swapper parameter, so `msg.sender` to the pool is the router: [4](#0-3) 

The same applies to `exactInput` (intermediate hops use `address(this)` = router): [5](#0-4) 

And to `exactOutputSingle` and `exactOutput` recursive hops: [6](#0-5) [7](#0-6) 

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` checks the explicit `owner` parameter — a separate argument that `addLiquidity` accepts independently of `msg.sender`, preserving the real depositor identity through intermediaries: [8](#0-7) 

No equivalent explicit-swapper parameter exists on `IMetricOmmPoolActions.swap`: [9](#0-8) 

The real user identity is irrecoverably lost at the pool boundary when the router intermediates the call.

## Impact Explanation

**Path A — Allowlist bypass (higher severity):** A pool admin allowlists the router address to enable router-based trading on a curated pool (the natural operational step). Any non-allowlisted user — including those excluded for KYC, institutional, or rate-limiting reasons — can call `exactInputSingle`/`exactInput`/`exactOutput` through the router. The extension evaluates `allowedSwapper[pool][router] == true` and passes. The entire swap curation policy is nullified for all router-routed trades, granting unrestricted access to a pool intended to be restricted.

**Path B — Broken core functionality (medium severity):** If the router is not allowlisted, every individually-allowlisted user who routes through `MetricOmmSimpleRouter` receives `NotAllowedToSwap`. The router becomes completely unusable on any allowlist-protected pool, stripping users of slippage protection, multi-hop routing, and deadline enforcement.

Both outcomes represent a broken invariant: a curated pool must enforce the same allowlist policy regardless of which supported public entrypoint reaches it.

## Likelihood Explanation

- `MetricOmmSimpleRouter` is the primary documented user-facing swap entrypoint; router-mediated swaps are the expected production path.
- Pool admins who want to allow router-based swaps on a curated pool have no documented alternative to allowlisting the router address, making Path A a natural operational mistake requiring no malicious intent.
- No special privileges, malicious tokens, or off-chain manipulation are required — a standard `exactInputSingle` call from any EOA suffices.
- `DepositAllowlistExtension` works correctly (checks `owner`), so admins have no reason to suspect the swap extension behaves differently.
- The bug is repeatable on every router-mediated swap to any allowlist-protected pool.

## Recommendation

Add an explicit `address swapper` parameter to `IMetricOmmPoolActions.swap` (analogous to `owner` in `addLiquidity`) that the router populates with `msg.sender` (the real user) before calling the pool. The pool forwards this value to `_beforeSwap`, and `SwapAllowlistExtension` checks it instead of the forwarded `msg.sender`. This mirrors the liquidity operator pattern already present in `DepositAllowlistExtension` and preserves the real initiator identity through any intermediary.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured on `beforeSwap`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-based swaps (natural operational step).
3. Non-allowlisted `attacker` calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient, ...)` — pool's `msg.sender` is the router.
5. Pool calls `_beforeSwap(router, recipient, ...)` → extension receives `sender = router`.
6. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
7. Attacker's swap executes on a pool that was supposed to reject them.

Conversely, if the router is not allowlisted:

3. Allowlisted `user` calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
4–5. Same as above; extension evaluates `allowedSwapper[pool][router] == false` → reverts `NotAllowedToSwap`.
6. Allowlisted user cannot use the router at all on this pool.
