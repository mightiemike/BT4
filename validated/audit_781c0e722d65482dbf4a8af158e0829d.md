Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address as `sender` instead of actual user, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the pool's `msg.sender` — the router contract address when a user routes through `MetricOmmSimpleRouter`. For any allowlisted user to swap through the router, the pool admin must allowlist the router address. Once the router is allowlisted, every unprivileged user can bypass the allowlist with a single `exactInputSingle` call, exposing LP funds to unauthorized trading.

## Finding Description

**Root cause:** `MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← router address when called via MetricOmmSimpleRouter
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to the extension. The extension then checks:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

Here `msg.sender` is the pool and `sender` is the router — not the EOA initiating the trade.

**Broken invariant:** For any allowlisted user (e.g., Alice) to swap through `MetricOmmSimpleRouter`, the pool admin must allowlist the router address, because `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly, making the pool see `msg.sender = router`. Once the router is allowlisted, `allowedSwapper[pool][router] = true` satisfies the check for every caller of the router, regardless of who they are.

**Two broken states:**

| Pool admin action | Effect |
|---|---|
| Does **not** allowlist the router | Allowlisted users cannot swap through the router — core swap path broken |
| **Allowlists the router** | Every user bypasses the allowlist via the router |

**Existing guards are insufficient:** There is no mechanism in `SwapAllowlistExtension` or `MetricOmmSimpleRouter` to propagate the real caller's identity. The extension has no way to distinguish a router call on behalf of an authorized user from one on behalf of an unauthorized user.

## Impact Explanation

A pool with `SwapAllowlistExtension` is a curated pool whose admin intends to restrict trading to specific counterparties (KYC'd users, institutional traders, whitelisted market makers). Once the router is allowlisted — the only way to enable router-based swaps for any user — the allowlist provides zero protection. Any unprivileged actor can execute swaps against the pool's oracle-anchored liquidity, extract value from LP positions through arbitrage or directional trading, and drain LP principal in pools where the allowlist was the sole access control. This is a direct loss of LP principal and a broken core pool invariant (allowlist bypass enabling unauthorized swap execution).

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard documented periphery swap path. No special privileges, flash loans, or complex setup are required — a single `exactInputSingle` call suffices. Any user who observes that the router is allowlisted (readable on-chain via `allowedSwapper`) can immediately exploit this. The condition (router allowlisted) is a necessary operational state for any pool that wants to support router-based swaps for its allowlisted users, making exploitation highly likely in production.

## Recommendation

The extension must check the actual user, not the intermediary. Two options:

1. **Pass the original `msg.sender` through the router via `extensionData`.** The router encodes the real caller in `extensionData`; the extension decodes and checks it, but only after verifying `msg.sender` (the pool's caller) is a trusted router. This requires a trusted-router registry in the extension.

2. **Check `recipient` instead of `sender`.** The pool already passes `recipient` as the second argument to `beforeSwap`. For swap allowlists, the economically relevant actor is the recipient of output tokens. The extension should check `allowedSwapper[pool][recipient]` rather than `allowedSwapper[pool][sender]`. This is simpler and requires no router trust, but the pool admin must understand the allowlist gates the output recipient, not the payer.

## Proof of Concept

```
Setup:
  - Pool P has SwapAllowlistExtension configured.
  - Pool admin calls setAllowedToSwap(P, router, true)
    so that allowlisted users can swap through the router.
  - Bob (address B) is NOT allowlisted.

Attack:
  1. Bob calls MetricOmmSimpleRouter.exactInputSingle({pool: P, recipient: Bob, ...}).
  2. Router calls P.swap(recipient=Bob, ...) — pool sees msg.sender = router (R).
     [MetricOmmSimpleRouter.sol L72-80]
  3. Pool calls _beforeSwap(sender=R, recipient=Bob, ...).
     [MetricOmmPool.sol L230-240]
  4. SwapAllowlistExtension checks allowedSwapper[P][R] → true (router is allowlisted).
     [SwapAllowlistExtension.sol L37]
  5. Swap executes. Bob receives output tokens. Allowlist bypassed.

Result:
  Bob, an unauthorized user, successfully swaps against the curated pool.
  LP funds are exposed to any unprivileged actor who routes through the router.
```

Foundry test outline:
- Deploy pool with `SwapAllowlistExtension`.
- Call `setAllowedToSwap(pool, router, true)`.
- Prank as an un-allowlisted address and call `MetricOmmSimpleRouter.exactInputSingle`.
- Assert the swap succeeds (no `NotAllowedToSwap` revert).
