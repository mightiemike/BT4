### Title
`GlobalContractDistributionReceipt` with stale `target_shard` causes node panic in `receipt_filter_fn` under static-resharding shard-layout incompatibility — (`File: runtime/runtime/src/congestion_control.rs`)

### Summary

`DelayedReceiptQueueWrapper::receipt_filter_fn` calls `receiver_shard_id(&shard_layout).unwrap()` on every receipt popped from the delayed queue. For `GlobalContractDistribution` receipts whose `target_shard` is from a shard layout that is two or more generations old, `receiver_shard_id` returns `Err(EpochError::ShardingError(...))` when the current layout is V1 or V2 (static resharding), because `ShardLayout::resolve_to_current_shard` for V0/V1/V2 only looks at the immediate children of the most recent split — it cannot walk a multi-generation split history. The `.unwrap()` then panics, crashing the node during chunk application.

### Finding Description

**Root cause — `receipt_filter_fn` panics on `Err`:**

```rust
// runtime/runtime/src/congestion_control.rs  line 874-878
fn receipt_filter_fn(&self, receipt: &ReceiptOrStateStoredReceipt) -> bool {
    let shard_layout = self.epoch_info_provider.shard_layout(&self.epoch_id).unwrap();
    let receipt_shard_id = receipt.get_receipt().receiver_shard_id(&shard_layout).unwrap(); // ← panics
    receipt_shard_id == self.shard_id
}
```

**`receiver_shard_id` returns `Err` when `resolve_to_current_shard` returns `None`:**

```rust
// core/primitives/src/receipt.rs  line 447-462
ReceiptEnum::GlobalContractDistribution(receipt) => {
    let target_shard = receipt.target_shard();
    if shard_layout.shard_ids().contains(&target_shard) {
        target_shard
    } else {
        let Some(current_shard) = shard_layout.resolve_to_current_shard(target_shard)
        else {
            return Err(EpochError::ShardingError(format!(
                "Shard {target_shard} does not exist in the shard layout or its split history",
            )));
        };
        current_shard
    }
}
```

**`resolve_to_current_shard` for V0/V1/V2 only resolves one generation:**

```rust
// core/primitives/src/shard_layout/mod.rs  line 230-237
pub fn resolve_to_current_shard(&self, shard_id: ShardId) -> Option<ShardId> {
    match self {
        Self::V0(_) | Self::V1(_) | Self::V2(_) => {
            self.get_children_shards_ids(shard_id).map(|c| c[0])  // single-generation only
        }
        Self::V3(v3) => v3.resolve_to_current_shard(shard_id),    // full history walk
    }
}
```

`ShardLayoutV2::get_children_shards_ids` only stores the most recent split map — it has no knowledge of grandparent shards. If a `GlobalContractDistributionReceipt` was created with `target_shard = S`, shard `S` was split into `S1/S2` (first resharding), and then `S1` was split into `S3/S4` (second resharding), the current V2 layout's `shards_split_map` only contains `{S1 → [S3, S4]}`. Querying `resolve_to_current_shard(S)` returns `None` because `S` is absent from both `shard_ids` and the current `shards_split_map`.

**Call path to the panic:**

`Runtime::apply` → `DelayedReceiptQueueWrapper::pop` / `peek_iter` → `receipt_filter_fn` → `.unwrap()` → **panic** → node crash.

The `apply` function is called from `NightshadeRuntime::apply_chunk` → `apply_new_chunk` → block processing pipeline. A panic here crashes the node and halts chain progress on the affected shard.

**Attacker-controlled entry point:**

Any unprivileged user can call `DeployGlobalContract` to create a `GlobalContractDistributionReceipt`. By submitting heavy transactions to saturate the shard's compute budget, the user can force the distribution receipt into the delayed queue. The receipt persists across epoch boundaries. If two static-resharding protocol upgrades occur while the receipt is delayed, the `target_shard` becomes two generations stale, and the next time the delayed queue is drained the node panics.

The test `test_stale_global_contract_distribution_after_double_resharding` explicitly documents this scenario and notes: *"The fix only works with V3 shard layouts (dynamic resharding). With static resharding, the shard layout doesn't maintain a full split history."*

### Impact Explanation

A panic inside `receipt_filter_fn` during `Runtime::apply` crashes the node process. Because every validator tracking the affected shard must apply the same chunk, all such validators crash simultaneously. The shard stops producing chunks, halting the chain. Recovery requires a coordinated restart and potentially a hotfix. This is a **Critical** availability impact — equivalent to the MarketMaker DOS where a single cheap user action permanently breaks a core protocol function.

### Likelihood Explanation

**Moderate preconditions, low current risk, non-zero future risk:**

1. A user deploys a global contract — trivially unprivileged.
2. The receipt is pushed to the delayed queue — achievable via compute saturation.
3. Two static-resharding protocol upgrades each change the shard layout while the receipt is queued.

Condition 3 is the gating factor. On mainnet, `GlobalContractDistribution` was introduced at protocol version 83; the current stable version is 86 with no shard-layout changes in that range. However, future protocol upgrades (e.g., SimpleNightshadeV4, V5) could trigger this. The window between a global-contract deployment and the second resharding can span many epochs, making the receipt easy to keep alive via congestion. The fix is already applied for V3/dynamic-resharding layouts but the V1/V2 static-resharding path remains unguarded.

### Recommendation

Replace the `.unwrap()` in `receipt_filter_fn` with proper error propagation, changing the return type to `Result<bool, RuntimeError>` and propagating the error up through `pop` and `peek_iter`. Alternatively, extend `resolve_to_current_shard` for V1/V2 layouts to walk the full layout history (analogous to the V3 fix), or add a guard that converts an unresolvable `GlobalContractDistribution` receipt into a no-op (skip and log) rather than panicking.

### Proof of Concept

1. At protocol version N (V1/V2 shard layout with shards `{0,1,2,3}`), user calls `DeployGlobalContract`, creating a `GlobalContractDistributionReceipt` with `target_shard = 3`.
2. User saturates shard 3's compute budget each block, forcing the receipt into the delayed queue.
3. Protocol upgrades to version N+1: shard 3 splits into shards `{4,5}` (V2 layout, `shards_split_map = {3 → [4,5]}`).
4. Protocol upgrades to version N+2: shard 4 splits into shards `{6,7}` (new V2 layout, `shards_split_map = {4 → [6,7]}`). The new layout has no entry for shard 3.
5. User stops saturating; the delayed queue drains.
6. `receipt_filter_fn` is called with the stale receipt (`target_shard = 3`). `shard_layout.shard_ids()` does not contain `3`. `resolve_to_current_shard(3)` calls `get_children_shards_ids(3)` on the new V2 layout — returns `None` because `3` is absent from the new `shards_split_map`. `receiver_shard_id` returns `Err`. `.unwrap()` panics.
