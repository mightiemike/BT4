Audit Report

## Title
SwapAllowlistExtension gates router address instead of actual swapper, breaking per-user allowlist enforcement - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the calling pool and `sender` is the first parameter passed by the pool — which is the `msg.sender` of the pool's `swap()` call. When `MetricOmmSimpleRouter` executes a swap, the router is `msg.sender` in the pool, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actualUser]`. This renders the per-user allowlist either completely ineffective (if the router is allowlisted) or blocks all router-based swaps for legitimately allowlisted users.

## Finding Description

**Root cause — `SwapAllowlistExtension.beforeSwap` (lines 31–41):**
```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```
`msg.sender` = the pool (confirmed by the `onlyPool` pattern in `BaseMetricExtension`). `sender` = the first argument the pool passes to `beforeSwap`, which is the `msg.sender` of the pool's `swap()` invocation.

**Router call path — `MetricOmmSimpleRouter.exactInputSingle` (lines 67–86):**
```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
IMetricOmmPoolActions(params.pool).swap(
    params.recipient, params.zeroForOne, ..., params.extensionData
);
```
The router calls `pool.swap()` directly. Inside the pool, `msg.sender` is the router contract. The pool then calls `extension.beforeSwap(router_address, ...)`. The allowlist lookup becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Two broken scenarios:**

1. **Allowlist bypass**: Pool admin allowlists the router (a natural action to enable router-based swaps for their users). Any unprivileged user can now call `exactInputSingle` through the router and pass the check, because `allowedSwapper[pool][router] == true` for all callers regardless of their individual allowlist status.

2. **Allowlist DoS**: Pool admin allowlists specific EOAs but not the router. Those EOAs cannot swap through the router at all — their swaps revert with `NotAllowedToSwap()` even though they are individually authorized.

**Existing guards are insufficient**: The `onlyPool` modifier on `BaseMetricExtension.beforeSwap` only ensures the caller is a valid pool; it does not fix the identity of the `sender` argument. No guard in the extension or router reconstructs the original EOA from transient storage before the extension check.

## Impact Explanation
Broken core pool functionality: the `SwapAllowlistExtension` fails to enforce per-user swap gating when the standard periphery router is used. In scenario 1, unauthorized users gain unrestricted swap access to a pool intended to be restricted, potentially draining liquidity at oracle-derived prices from a pool whose LP depositors expected access controls. In scenario 2, legitimately allowlisted users are locked out of the primary swap interface. Both outcomes directly impair the swap flow and LP asset safety of any pool relying on this extension.

## Likelihood Explanation
Any unprivileged user can trigger scenario 1 by calling `MetricOmmSimpleRouter.exactInputSingle` or `exactInput` against a pool whose admin has allowlisted the router. This is a zero-precondition exploit once the router is allowlisted — no special role, no flash loan, no oracle manipulation required. Scenario 2 is triggered automatically the first time any allowlisted user attempts a router swap. Both paths are fully reachable in production with standard user behavior.

## Recommendation
The pool should pass the original transaction initiator (or a verified user identity) as `sender` to `beforeSwap`, not its own `msg.sender`. One approach: the router stores the original `msg.sender` in transient storage (it already does this for the payer via `_setNextCallbackContext`) and the pool reads it from the callback context to forward as `sender` to the extension. Alternatively, `SwapAllowlistExtension` should check the `recipient` parameter or require the pool to forward the payer address from transient storage, ensuring the allowlist gates the actual economic actor rather than the intermediary router.

## Proof of Concept
```solidity
// 1. Deploy pool with SwapAllowlistExtension
// 2. Pool admin calls setAllowedToSwap(pool, router, true)  // allowlist the router
// 3. Unauthorized EOA (not individually allowlisted) calls:
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: attacker,
    tokenIn: token0,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// 4. Pool calls extension.beforeSwap(router, ...)
// 5. allowedSwapper[pool][router] == true → check passes
// 6. Unauthorized user successfully swaps in restricted pool
```
A Foundry integration test deploying the pool with `SwapAllowlistExtension`, allowlisting only the router, and asserting that a non-allowlisted EOA can complete a swap via the router reproduces the bypass.
