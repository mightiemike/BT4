### Title
Dogecoin (and Litecoin) `get_next_work_required` Reads Difficulty Anchor Timestamp from Mainchain Instead of Fork Ancestor Chain — (`contract/src/dogecoin.rs`, `contract/src/litecoin.rs`)

---

### Summary

`get_header_by_height` is implemented to always read from `mainchain_height_to_header`. When `get_next_work_required` calls it to fetch the "first block" timestamp for a difficulty retarget calculation, it retrieves the **mainchain** block at that height rather than the fork's actual ancestor. An attacker who mines a fork diverging at height F can place a fork block at F with an attacker-controlled timestamp; when the fork's block at F+2 is submitted, the difficulty is computed using the mainchain block at F's timestamp instead of the fork block at F's timestamp. This produces a false `expected_bits` that can be lower than the correct value, allowing a block with artificially reduced difficulty to pass `check_pow` and be accepted into the canonical chain after a reorg.

The developers explicitly flagged this with a TODO comment at the exact line in question.

---

**Dogecoin mainnet config: `difficulty_adjustment_interval = 1`, `pow_target_timespan = 60` seconds:** [4](#0-3) 

With `difficulty_adjustment_interval = 1`, every block is a retarget block. The `blocks_to_go_back` calculation: [5](#0-4) 

For a fork diverging at height F, when submitting fork block at height F+2:
- `prev_block_header` = fork block at F+1, `height_first = F`
- `get_header_by_height(F)` returns the **mainchain** block at F (hash stored in `mainchain_height_to_header`)
- The fork block at F is stored only in `headers_pool`, not in `mainchain_height_to_header`
- The fork block at F has an attacker-controlled timestamp (up to `current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL = +2 hours`) [6](#0-5) 

**Difficulty manipulation via `calculate_next_work_required`:** [7](#0-6) 

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;
// clamped to [retarget_timespan - retarget_timespan/4, retarget_timespan + retarget_timespan/2]
// = [45s, 90s] for Dogecoin mainnet
```

If the mainchain block at F has timestamp T_main and the fork block at F has timestamp T_fork > T_main, then `get_header_by_height(F)` returns T_main (earlier), making `modulated_timespan` larger than correct, yielding a larger target (lower difficulty). The Digishield clamp caps the maximum target increase at 50% (`retarget_timespan + retarget_timespan/2 = 90s` vs `60s` baseline).

**`check_pow` is called before `submit_block_header_inner`**, so the difficulty check uses the pre-reorg mainchain state: [8](#0-7) 

---

### Impact Explanation

An attacker who can submit fork blocks (via the trusted-relayer submission path) can:
1. Mine a fork diverging at height F with a fork block at F carrying a timestamp up to 2 hours ahead of the mainchain block at F.
2. Submit fork blocks F+1, F+2, … where each retarget uses the mainchain block's earlier timestamp, yielding up to a **50% increase in target** (33% reduction in required work) per block.
3. Accumulate enough chainwork to trigger a reorg (`reorg_chain`), promoting the fork to canonical.
4. After the reorg, `mainchain_tip_blockhash` points to the attacker's chain, and `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` will confirm transactions in the attacker's blocks.

This breaks the invariant that difficulty must be computed from the actual ancestor chain of the submitted block, enabling false transaction confirmations on a chain built with less work than the protocol requires.

---

### Likelihood Explanation

- The bug is **self-acknowledged** by the developers (TODO comment at the exact line).
- For Dogecoin mainnet (`difficulty_adjustment_interval = 1`), the bug triggers on **every fork block at height F+2 and beyond** — no special retarget boundary alignment is needed.
- The same bug exists in `litecoin.rs` but requires the fork to straddle a 2016-block retarget boundary.
- Entry requires the trusted-relayer submission path; an honest relayer naturally relays any blocks it observes on the Dogecoin network, so the attacker only needs to mine fork blocks on-chain (no relayer compromise needed).
- The 50% difficulty reduction is bounded by Digishield clamping, so the attacker still needs substantial hash power to outpace the mainchain — this limits likelihood to well-resourced attackers.

---

### Recommendation

Replace the `get_header_by_height` call with an ancestor-chain walk. Starting from `prev_block_header`, follow `get_prev_header` links `blocks_to_go_back` times to reach the correct ancestor. This is O(interval) but correct. The TODO comment already identifies this as the intended fix direction.

For Dogecoin (`interval = 1`), `blocks_to_go_back = 1`, so the fix is a single `get_prev_header` call on `prev_block_header` — trivially cheap.

---

### Proof of Concept

```
State setup (skip_pow = true, Dogecoin mainnet, height >= 145_001):

Mainchain: ... [M_{F-1}, ts=T0] → [M_F, ts=T1] → [M_{F+1}, ts=T2] → tip

Fork submission sequence:
1. Submit fork_F:   prev=M_{F-1}, ts = T1 + 7000  (attacker sets timestamp ~2h ahead of T1)
                    → stored in headers_pool only (fork)
2. Submit fork_{F+1}: prev=fork_F, ts = T1 + 7060
                    → check_pow: height_first = F-1, get_header_by_height(F-1) = M_{F-1} ✓ (correct, fork diverges at F)
                    → stored in headers_pool only
3. Submit fork_{F+2}: prev=fork_{F+1}, ts = T1 + 7120
                    → check_pow: height_first = F, get_header_by_height(F) = M_F (ts=T1) ← BUG
                    → correct would use fork_F (ts=T1+7000)
                    → modulated_timespan uses T1 instead of T1+7000
                    → first_block_time is 7000s earlier → modulated_timespan hits max clamp (90s)
                    → expected_bits computed from 90s timespan instead of correct ~60s
                    → target is 1.5× larger (difficulty 33% lower)
                    → fork_{F+2} with bits matching this lower difficulty passes check_pow

4. Continue submitting fork blocks with accumulated chainwork > mainchain chainwork
5. reorg_chain() fires → mainchain_tip_blockhash = fork tip
6. verify_transaction_inclusion on fork_{F+2} returns true
``` [9](#0-8) [10](#0-9)

```rust
fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
}
