### Title
Missing Validation of `gc_threshold` in `init` Permanently Breaks Confirmation Security Model — (`contract/src/lib.rs`)

---

### Summary

The `init` function stores `gc_threshold` from `InitArgs` directly into contract state without any validation. If a deployer accidentally sets `gc_threshold = 0`, the value is permanently locked (no setter exists), and every subsequent call to `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` will only accept `confirmations = 0`. With zero confirmations, the depth check is trivially satisfied for any mainchain block, completely bypassing the confirmation security guarantee that downstream bridge or SPV consumers depend on.

---

### Finding Description

`InitArgs.gc_threshold` flows directly into `BtcLightClient.gc_threshold` with no lower-bound check: [1](#0-0) 

```rust
pub fn init(args: InitArgs) -> Self {
    let mut contract = Self {
        ...
        gc_threshold: args.gc_threshold,   // ← no validation
        ...
    };
``` [2](#0-1) 

`gc_threshold` is never written again after `init`; there is no setter anywhere in the contract. It is effectively immutable post-deployment.

`gc_threshold` is used in two critical places:

**1. Confirmation upper-bound gate in `verify_transaction_inclusion`:** [3](#0-2) 

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
...
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
```

If `gc_threshold = 0`, the first `require!` forces `args.confirmations` to be `0`. With `confirmations = 0`, the second `require!` becomes `N >= 0`, which is always `true` for any `u64`. Any block that exists in the mainchain is therefore "verified" regardless of how recently it was submitted.

**2. GC threshold in `run_mainchain_gc` (called inside every `submit_blocks`):** [4](#0-3) 

```rust
if amount_of_headers_we_store > self.gc_threshold {
    let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
    ...
    self.mainchain_initial_blockhash = self
        .mainchain_height_to_header
        .get(&end_removal_height)
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
}
```

With `gc_threshold = 0`, `total_amount_to_remove` equals the entire stored chain length. GC will attempt to remove all headers on every `submit_blocks` call. Once all headers at heights below `end_removal_height` are removed, the `get(&end_removal_height)` lookup panics with `ERR_KEY_NOT_EXIST`, making `submit_blocks` permanently reverting.

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` and is equally affected: [5](#0-4) 

---

### Impact Explanation

Any unprivileged NEAR caller can invoke `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` (both are public, only gated by `#[pause]`). With `gc_threshold = 0` set at deployment:

- Passing `confirmations = 0` satisfies both `require!` checks for **any block present in the mainchain**, regardless of actual depth.
- A downstream bridge or SPV consumer that calls this function to gate fund releases or cross-chain actions receives `true` for a block with zero confirmations, making double-spend or reorg attacks trivially feasible.
- Simultaneously, `submit_blocks` panics on every call once GC exhausts stored headers, rendering the light client permanently non-functional.

---

### Likelihood Explanation

`gc_threshold = 0` is the natural Rust/JSON default for an unset `u64`. A deployer who omits or zeroes the field in the deployment JSON produces a permanently broken contract. The `init` function is `#[private]` (called once at deployment), so there is no recovery path — the contract must be redeployed. This is the exact accidental-misconfiguration scenario described in M-02 for `amountToConvert`.

---

### Recommendation

Add an explicit lower-bound check in `init` before storing `gc_threshold`:

```rust
require!(
    args.gc_threshold >= MIN_GC_THRESHOLD,
    "gc_threshold must be at least MIN_GC_THRESHOLD blocks"
);
```

A sensible minimum is `144` (≈ 1 day of Bitcoin blocks), matching the comment already present in the `init` doc-comment: [6](#0-5) 

---

### Proof of Concept

1. Deploy the contract with `InitArgs { gc_threshold: 0, ... }`.
2. Submit a genesis block and a few subsequent headers via `submit_blocks`.
3. Call `verify_transaction_inclusion` with `{ confirmations: 0, tx_block_blockhash: <any mainchain block>, ... }`.
4. The first `require!` passes (`0 <= 0`). The second `require!` passes (`N >= 0`). The function returns `true` for any block with a structurally valid merkle proof, with zero confirmation depth enforced.
5. Simultaneously, each `submit_blocks` call triggers `run_mainchain_gc` which removes all stored headers and eventually panics on `ERR_KEY_NOT_EXIST`, bricking the relayer path.
