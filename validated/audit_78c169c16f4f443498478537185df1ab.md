Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of originating user, allowing any caller to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument forwarded by the pool, which equals `msg.sender` of the `pool.swap()` call — the router contract address when swaps are routed through `MetricOmmSimpleRouter`. Because the only way to allow approved users to swap through the public router is to allowlist the router address itself, any unapproved user can bypass the allowlist entirely by calling the same public router. The allowlist invariant is completely defeated for all router-mediated swaps.

## Finding Description
`MetricOmmPool.swap` captures `msg.sender` and passes it as `sender` to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← router address when called via MetricOmmSimpleRouter
    recipient, zeroForOne, amountSpecified, priceLimitX64,
    packedSlot0Initial, bidPriceX64, askPriceX64, extensionData
);
```

`ExtensionCalling._beforeSwap` forwards that value verbatim as `sender` to every configured extension via `_callExtensionsInOrder`:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value forwarded above:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` (and all other swap entry points) calls `pool.swap(...)` directly, making itself `msg.sender` of that call. The originating user is stored only in transient callback context for payment settlement and is never surfaced to the pool or extension:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The extension therefore sees `sender = router_address` for every router-mediated swap, regardless of who called the router. If the pool admin allowlists the router (the only mechanism to let approved users swap through it), the check `allowedSwapper[pool][router] == true` passes for every caller of the router, approved or not.

`DepositAllowlistExtension` does not share this flaw: it ignores `sender` and checks `owner` (the position owner explicitly supplied by the caller), which is correctly preserved through the `MetricOmmPoolLiquidityAdder` path.

## Impact Explanation
A pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to approved addresses (e.g., KYC-verified counterparties). To allow those approved users to swap through `MetricOmmSimpleRouter` — the primary documented swap interface — the admin must call `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, every unprivileged address can call any of `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` on the router and the extension will pass the check, because it sees `sender = router`. The allowlist is completely defeated for all router-mediated swaps. Unauthorized parties can execute swaps at oracle prices in a pool designed to be permissioned, constituting a direct admin-boundary break with potential for fund loss to LPs and protocol.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool admin who wants allowlisted users to use the router must allowlist the router address — there is no other mechanism. This is the natural, expected configuration. The bypass is reachable by any unprivileged user on any pool that has both `SwapAllowlistExtension` enabled and the router allowlisted, with no special preconditions beyond calling the public router.

## Recommendation
The extension must gate on the identity of the economic actor, not the immediate `msg.sender` of the pool call. Two sound approaches:

1. **Pass the originating user through `extensionData`**: The router encodes `msg.sender` (the real user) into `extensionData` before forwarding to the pool. The extension decodes and checks that address. This requires a convention between router and extension but avoids `tx.origin`.

2. **Check `tx.origin` as the swapper identity**: Replace `sender` with `tx.origin` inside `beforeSwap`. This is safe in this context because the extension is only checking access control (not using it for payment), and `tx.origin` correctly identifies the EOA initiating the transaction regardless of intermediary contracts.

Either way, the `allowedSwapper` mapping should be keyed on the originating user, not the router contract.

## Proof of Concept
```
Setup
─────
1. Pool admin deploys pool with SwapAllowlistExtension.
2. Admin calls setAllowedToSwap(pool, alice, true)   // alice is approved
3. Admin calls setAllowedToSwap(pool, router, true)  // router allowlisted so alice can use it
4. bob is NOT in the allowlist.

Attack
──────
5. bob calls MetricOmmSimpleRouter.exactInputSingle({
       pool: pool, zeroForOne: true, amountIn: X, ...
   });

6. Router calls pool.swap(recipient=bob, ...) with msg.sender = router.

7. Pool calls _beforeSwap(sender=router, ...).

8. SwapAllowlistExtension.beforeSwap receives sender=router.
   Checks: allowedSwapper[pool][router] == true  ✓  → no revert.

9. Swap executes. bob receives output tokens.
   bob's address was never checked against the allowlist.

Result: bob, an unprivileged address, swaps successfully in a pool
        configured to restrict swaps to approved addresses only.
```
