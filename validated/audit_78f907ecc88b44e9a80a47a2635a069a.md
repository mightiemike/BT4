Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Allowing Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`, which is the router contract address when a swap originates from `MetricOmmSimpleRouter`. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]`, so it validates the router's permission rather than the end user's. Any unpermitted user who calls the public router can bypass a pool's swap allowlist if the router address is allowlisted.

## Finding Description
In `MetricOmmPool.swap` (L230–240), the pool calls `_beforeSwap(msg.sender, ...)`. When the call originates from `MetricOmmSimpleRouter.exactInputSingle` (L72–80), `msg.sender` is the router contract, not the end user.

`ExtensionCalling._beforeSwap` (L149–177) encodes this value as `sender` and dispatches it to all configured extensions via `_callExtensionsInOrder`.

`SwapAllowlistExtension.beforeSwap` (L37) then evaluates:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool (correct) and `sender` is the router address (wrong). The check `allowedSwapper[pool][router]` passes for any caller of the router, not just allowlisted users.

`MetricOmmSimpleRouter` stores no end-user identity in a field the pool can forward to extensions — the end user's address is only used internally as the `payer` in transient storage for the callback, never passed as `sender` to `pool.swap`.

**Exploit path:**
1. Pool admin deploys pool with `SwapAllowlistExtension`, intending to restrict swaps to specific addresses.
2. Pool admin calls `ext.setAllowedToSwap(pool, address(router), true)` so that allowlisted users can route through the public router.
3. Unpermitted attacker calls `router.exactInputSingle(...)` targeting the restricted pool.
4. Router calls `pool.swap(recipient, ...)` — pool sees `msg.sender = router`.
5. `_beforeSwap(router, ...)` dispatched; extension checks `allowedSwapper[pool][router]` → `true` → passes.
6. Attacker's swap executes against the restricted pool's liquidity.

There is no existing guard that recovers the originating user identity from the router call.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` for KYC-gating, institutional-only access, or market-maker restriction is fully bypassed by any user routing through `MetricOmmSimpleRouter`. The attacker executes swaps at oracle-anchored prices against liquidity the pool admin intended to restrict, causing direct loss of LP assets and breaking the core access-control invariant of the extension system. This meets the "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" impact criteria.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary public-facing swap interface. Any pool admin who allowlists the router to support router-mediated swaps — a natural and expected configuration — inadvertently opens the pool to all users. The bypass requires no special privileges: any user with a standard ERC-20 approval can call `exactInputSingle`. The condition is trivially reachable and repeatable.

## Recommendation
The `sender` forwarded to extensions must represent the originating end user, not the intermediary router. Two complementary fixes:

1. **Router passes originating user identity**: `MetricOmmSimpleRouter` should encode `msg.sender` (the end user) into `callbackData` or a dedicated field, and the pool should forward it as `sender` to extensions instead of `msg.sender`. This is analogous to EIP-2771 meta-transaction forwarding.
2. **Extension checks recipient**: As a weaker alternative, `SwapAllowlistExtension` could check the `recipient` field (which the router sets to the end user's address). However, `recipient` is the token destination and may differ from the authorizing party in multi-hop or aggregator scenarios.

The root fix is architectural: the pool's extension dispatch must carry the originating user identity through the router hop.
