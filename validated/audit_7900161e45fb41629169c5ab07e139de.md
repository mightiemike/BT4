### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the 64-Byte Forgery Protection Added in v2 - (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase transaction proof. However, the original `verify_transaction_inclusion` function remains `pub` on the NEAR contract and is directly callable by any unprivileged account. The `#[deprecated]` Rust attribute is a compiler-level hint only — it does not restrict external NEAR callers from invoking the function via a signed transaction. Any caller can bypass the coinbase proof check entirely by calling the old function directly.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces two security properties that `verify_transaction_inclusion` does not:

1. It requires `coinbase_tx_id` and `coinbase_merkle_proof` of equal length to the transaction proof.
2. It verifies that the coinbase proof resolves to the block's `merkle_root` before delegating to the inner function. [1](#0-0) 

The inner function `verify_transaction_inclusion` is decorated only with `#[deprecated]` and `#[pause]`. Neither attribute restricts external NEAR callers:

- `#[deprecated]` emits a Rust compiler warning for Rust-language callers. It has zero effect on NEAR RPC or cross-contract callers who invoke the method by name.
- `#[pause]` only blocks calls when the contract is administratively paused — it is not a permanent access restriction. [2](#0-1) 

There is no `#[private]` attribute, no role check, and no other guard on `verify_transaction_inclusion`. It is a fully reachable NEAR contract method.

The function's own doc comment acknowledges the risk explicitly:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [3](#0-2) 

---

### Impact Explanation

The 64-byte transaction forgery attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions) allows an attacker to construct a crafted 64-byte blob whose double-SHA256 hash matches an internal Merkle tree node. By supplying this internal node hash as `tx_id` and a valid Merkle path to the root, the attacker can make `verify_transaction_inclusion` return `true` for a transaction that was never included in any Bitcoin block.

Any downstream NEAR contract or application that calls `verify_transaction_inclusion` — or that accepts a proof verified by it — can be deceived into treating a fabricated transaction as confirmed. This directly corrupts the canonical verification result that the light client is designed to provide. [4](#0-3) 

---

### Likelihood Explanation

The entry path requires no privilege. Any NEAR account can submit a transaction calling `verify_transaction_inclusion` by name with a crafted `ProofArgs` payload. The `ProofArgs` struct is fully public and serializable. [5](#0-4) 

The attacker needs only a valid block hash that exists in the `headers_pool` and a crafted 64-byte internal node that hashes to a value satisfying the Merkle path. This is a known, documented attack with published tooling. The contract is live and the method is callable today.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` or add `#[private]` to make it a NEAR-internal-only method. Since `verify_transaction_inclusion_v2` already calls it internally via `self.verify_transaction_inclusion(args.into())`, the function must remain accessible within the contract — but it must not be reachable as an external NEAR contract method.

Concretely, change:

```rust
pub fn verify_transaction_inclusion(&self, ...) -> bool {
```

to:

```rust
fn verify_transaction_inclusion(&self, ...) -> bool {
```

This mirrors the recommended fix in M-15: make the inner function `internal` (private) so that the security guard on the wrapper cannot be bypassed by direct invocation. [6](#0-5) 

---

### Proof of Concept

1. Attacker identifies a block hash `B` present in `mainchain_header_to_height` (i.e., on the canonical chain) with sufficient confirmations.
2. Attacker retrieves the block's Merkle root from `headers_pool`.
3. Attacker constructs a 64-byte blob whose double-SHA256 equals an internal Merkle tree node `N` at depth `d`, and computes a valid Merkle path from `N` to the Merkle root.
4. Attacker submits a NEAR transaction calling `verify_transaction_inclusion` directly with:
   - `tx_id = N` (the internal node hash, not a real transaction)
   - `tx_block_blockhash = B`
   - `tx_index` = index corresponding to the chosen path
   - `merkle_proof` = the crafted path
   - `confirmations = 1`
5. `verify_transaction_inclusion` computes `compute_root_from_merkle_proof(N, tx_index, merkle_proof)` and compares it to the stored Merkle root. The comparison succeeds and the function returns `true`.
6. Any downstream contract that trusts this result treats the fabricated transaction as confirmed on Bitcoin.

The coinbase proof check in `verify_transaction_inclusion_v2` — which would have rejected this input — is never executed. [7](#0-6) [8](#0-7)

    #[pause]
    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
