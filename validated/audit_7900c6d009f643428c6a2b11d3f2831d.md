### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Merkle Proof Validation — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still exposed as a `pub` NEAR contract method despite being deprecated. Any unprivileged NEAR caller can invoke it directly, completely bypassing the coinbase Merkle proof check that `verify_transaction_inclusion_v2` was introduced to enforce. This is the direct analog of the external report's pattern: a method that should only be reachable through a guarded path remains callable without the required precondition check.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability (documented at https://www.bitmex.com/blog/64-Byte-Transactions). Its security guarantee rests on first verifying a coinbase Merkle proof before delegating to v1: [1](#0-0) 

However, v1 itself is still declared `pub` and carries no runtime guard that prevents direct invocation: [2](#0-1) 

The `#[deprecated]` attribute is a Rust compiler hint only — it emits a warning at the call site in Rust source code but imposes **no restriction at the WASM ABI level**. Any NEAR account can call `verify_transaction_inclusion` directly via a cross-contract call or a signed transaction, skipping the coinbase check entirely.

The function's own doc comment acknowledges the consequence: [3](#0-2) 

The v1 proof logic reduces to a single Merkle root comparison with no coinbase anchor: [4](#0-3) 

---

### Impact Explanation

The 64-byte transaction forgery attack allows an attacker to supply a 64-byte blob that is a valid internal Merkle tree node as the `tx_id`. Because the Merkle tree is built by hashing pairs of child nodes, an internal node hash is indistinguishable from a leaf (transaction) hash at the root-comparison step. Without a coinbase anchor proof (which v2 enforces), `verify_transaction_inclusion` returns `true` for a fabricated "transaction" that was never broadcast or mined.

Any recipient contract that calls `verify_transaction_inclusion` directly — or that is pointed at v1 by an integrator who did not migrate — will accept a forged proof as valid. This corrupts the proof-verification result that is the sole security output of the light client, enabling false claims of Bitcoin transaction inclusion on NEAR.

---

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can submit a signed transaction calling `verify_transaction_inclusion`. The 64-byte forgery technique is publicly documented and has known tooling. Integrators who deployed against v1 before v2 was released, or who copy-paste older example code, will continue to call v1 without realising the coinbase check is absent. The method name gives no indication that it is unsafe.

---

### Recommendation

Remove `pub` from `verify_transaction_inclusion` (make it `pub(crate)` or `fn`) so it is only reachable as an internal helper called by `verify_transaction_inclusion_v2`. Alternatively, add the coinbase Merkle proof check directly inside v1 and remove the delegation pattern. Either change closes the direct-call bypass at the ABI boundary rather than relying on a compiler-only deprecation warning.

---

### Proof of Concept

1. Identify any block `B` stored in the light client with known Merkle root `R`.
2. Construct a 64-byte value `N` that is a valid internal Merkle node of `B`'s transaction tree (i.e., the concatenation of two child hashes at some level). By construction, a Merkle proof path exists from `N` to `R`.
3. Call `verify_transaction_inclusion` directly (not v2) with:
   - `tx_id = N`
   - `tx_block_blockhash = B`
   - `merkle_proof` = the sibling path from `N` to `R`
   - `tx_index` = the index corresponding to `N`'s position
   - `confirmations` = any value ≤ `gc_threshold`
4. The function computes `compute_root_from_merkle_proof(N, index, proof) == R` — which is `true` — and returns `true`, certifying that the fabricated `N` is an included transaction. [5](#0-4) 

No privileged role, no leaked key, and no special chain state is required. The call is reachable by any unprivileged NEAR account at any time after contract initialization.

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
