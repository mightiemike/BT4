Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the end user, allowing any user to bypass per-user swap allowlists via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which is the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, that caller is the router contract, not the end user. Any pool admin who allowlists the router to support router-mediated swaps inadvertently grants swap access to every user on-chain, completely defeating the per-user allowlist invariant the extension is designed to enforce.

## Finding Description
**Root cause:** `MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // whoever called pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this value verbatim into the extension hook:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-177
abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
```

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router.

**Exploit flow:**
1. Pool admin deploys a curated pool with `SwapAllowlistExtension` and calls `setAllowedToSwap(pool, router, true)` so that legitimate users can reach the pool through `MetricOmmSimpleRouter`.
2. Any unprivileged user (not on the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting that pool.
3. The router calls `pool.swap(...)` — the pool's `msg.sender` is the router.
4. `_beforeSwap` is invoked with `sender = router`.
5. `allowedSwapper[pool][router] == true` → the check passes for every caller regardless of identity.
6. The swap executes; the allowlist is completely bypassed.

**Why existing guards fail:** There is no secondary check on the end user's identity anywhere in the swap path. The extension has no access to the original `tx.origin` or any forwarded user context. The router stores the original `msg.sender` only in transient storage for the payment callback (`_setNextCallbackContext`), but this context is never surfaced to extensions.

## Impact Explanation
The `SwapAllowlistExtension` is a core access-control primitive for curated pools. When the router is allowlisted, the extension's invariant — "only approved addresses may swap" — is broken for every user on-chain. Any unprivileged trader can swap in a pool the admin intended to restrict, draining liquidity reserved for curated counterparties and executing swaps the admin explicitly prohibited. This constitutes broken core pool functionality (the allowlist gate is rendered inoperative) and an admin-boundary break (an unprivileged path bypasses the access control the pool admin configured).

## Likelihood Explanation
The condition is trivially reachable: any user who can call `MetricOmmSimpleRouter.exactInputSingle` can exploit this. The only prerequisite is that the pool admin has allowlisted the router — a natural and expected configuration for any pool that wants to support standard router-mediated swaps. No special privileges, flash loans, or complex setup are required. The attack is repeatable on every block.

## Recommendation
Pass the original end-user address through the extension hook rather than the immediate `msg.sender`. One approach: have the router encode the end user's address in `extensionData` and have the extension decode and verify it. A more robust fix is to add a dedicated `originalSender` field to the swap hook signature that the pool populates from a trusted, router-supplied context (e.g., transient storage written by the router before calling `pool.swap`), so the extension can check the actual initiating user. Alternatively, document that allowlisting the router is equivalent to `allowAllSwappers = true` and enforce that constraint at the admin setter level.
