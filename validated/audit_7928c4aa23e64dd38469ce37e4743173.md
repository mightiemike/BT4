### Title
`WithdrawCollateralV2` Silently Fails in Slow Mode, Causing Fee Loss and Blocked Withdrawal — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

`submitSlowModeTransactionImpl` accepts `WithdrawCollateralV2` transactions (charges the slow mode fee and queues them), but `processSlowModeTransactionImpl` has no handler for `WithdrawCollateralV2` and falls through to `else { revert(); }`. The revert is silently swallowed by the try/catch in `_executeSlowModeTransaction`. The result is that any user who submits a V2 withdrawal via slow mode loses their fee and their withdrawal is permanently dropped with no on-chain signal.

---

### Finding Description

`submitSlowModeTransactionImpl` has three branches for incoming transaction types:

1. `DepositCollateral` → explicit `revert()`
2. A hardcoded owner-only list → `require(sender == owner())`
3. Everything else → charge slow mode fee and queue

`WithdrawCollateralV2` is not in the owner-only list, so it falls into branch 3: the fee is charged and the transaction is enqueued. [1](#0-0) 

When the queued transaction is later executed, `processSlowModeTransactionImpl` handles `WithdrawCollateral` (V1) at line 217 but has no case for `WithdrawCollateralV2`. It falls through to the terminal `else { revert(); }`: [2](#0-1) [3](#0-2) 

The revert is silently caught by `_executeSlowModeTransaction` in `Endpoint.sol`: [4](#0-3) 

By contrast, `processTransactionImpl` (the sequencer path) and `resolveFastWithdrawal` (the fast-withdrawal path) both handle `WithdrawCollateralV2` correctly: [5](#0-4) [6](#0-5) 

The slow mode path is the only path where V2 is accepted (fee charged) but never executed.

---

### Impact Explanation

**Slow mode fee loss**: Every user who submits `WithdrawCollateralV2` via `submitSlowModeTransaction` pays `SLOW_MODE_FEE` in quote tokens. The fee is non-refundable; the transaction is silently dropped.

**Censorship-resistance bypass blocked for V2**: The slow mode queue exists specifically so users can bypass sequencer censorship. `WithdrawCollateralV2` adds a `sendTo` field that allows withdrawing to an address other than the subaccount owner. If the sequencer censors a V2 withdrawal (e.g., to a specific recipient address), the user has no on-chain recourse: slow mode accepts the transaction, charges the fee, and silently discards it. The user cannot achieve the same result with V1 because V1 always sends to `address(0)` (resolved to the sender's own address): [2](#0-1) 

---

### Likelihood Explanation

`WithdrawCollateralV2` is the current production withdrawal path — it is handled in `processTransactionImpl`, `resolveFastWithdrawal`, and `Verifier.computeDigest`. Any user who constructs a V2 withdrawal and submits it via `submitSlowModeTransaction` (a public, permissionless function) triggers the bug. No special privileges are required. The user receives no revert at submission time, making silent failure likely to go unnoticed.

---

### Recommendation

Add a `WithdrawCollateralV2` branch to `processSlowModeTransactionImpl` mirroring the V1 branch, decoding `SignedWithdrawCollateralV2`, validating the sender, and calling `clearinghouse.withdrawCollateral` with the resolved `sendTo` address (defaulting to the sender's address when `sendTo == address(0)`), consistent with how `resolveFastWithdrawal` handles V2.

---

### Proof of Concept

1. User calls `submitSlowModeTransaction(abi.encodePacked(uint8(TransactionType.WithdrawCollateralV2), abi.encode(signedV2Tx)))`.
2. `submitSlowModeTransactionImpl` does not match any explicit branch for `WithdrawCollateralV2`; falls into `else` → charges `SLOW_MODE_FEE`, increments `txCount`, stores the transaction.
3. After `SLOW_MODE_TX_DELAY`, anyone calls `executeSlowModeTransaction()`.
4. `_executeSlowModeTransaction` calls `this.processSlowModeTransaction(sender, tx)` inside a try/catch.
5. `processSlowModeTransactionImpl` reaches `else { revert(); }` — no V2 handler exists.
6. The revert is caught silently; `nSubmissions` is incremented, `txUpTo` advances.
7. User's withdrawal is permanently dropped; slow mode fee is permanently lost; no event or error is emitted.

                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily

                // before we miss out of gas errors; 1/2 ~= (63/64)**44
