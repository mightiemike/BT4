### Title
Liquidator with sub-zero initial health is blocked from liquidating even when the liquidation improves their health — (`File: core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`ClearinghouseLiq._handleLiquidationPayment` enforces a post-liquidation initial-health check on the liquidator (`txn.sender`). When a liquidator's initial health is already below zero before the call, and the liquidation improves but does not fully restore their health to zero, the transaction reverts with `ERR_SUBACCT_HEALTH`. This mirrors the DSCEngine pattern exactly: a health gate on the liquidator blocks a beneficial action that the protocol should permit.

---

### Finding Description

After all position and quote transfers are applied inside `_handleLiquidationPayment`, the function asserts:

```solidity
require(
    txn.sender == N_ACCOUNT || !isUnderInitial(txn.sender),
    ERR_SUBACCT_HEALTH
);
``` [1](#0-0) 

`isUnderInitial` returns `true` when the liquidator's INITIAL health (the stricter threshold used for opening positions) is negative:

```solidity
function isUnderInitial(bytes32 subaccount) public returns (bool) {
    return getHealthFromClearinghouse(subaccount, IProductEngine.HealthType.INITIAL) < 0;
}
``` [2](#0-1) 

For **spot-asset liquidations** (positive `txn.amount`, liquidatee holds a long spot position) and **perp liquidations** (liquidator absorbs a discounted perp position), the liquidation payment logic transfers assets to the liquidator at a haircut price. The liquidator receives value greater than what they pay, so their health contribution increases. However, if their initial health was, say, −100 before the call and the liquidation adds +50, their post-liquidation health is −50 — still negative — and the `require` reverts.

The concrete state delta that is corrupted: the liquidatee's under-maintenance subaccount remains unliquidated, accumulating bad debt, while the liquidator is denied a health-improving action.

The error constant `ERR_NOT_LIQUIDATABLE_INITIAL` ("Liquidator health too low") is defined in `Errors.sol` but is **never referenced anywhere in the codebase**, suggesting the current `ERR_SUBACCT_HEALTH` check was not originally intended to gate the liquidator's pre-existing health state. [3](#0-2) 

---

### Impact Explanation

A liquidator whose initial health is below zero — but whose maintenance health is still non-negative (i.e., they are not themselves liquidatable) — is completely blocked from liquidating any spot-asset or perp position. Because spot-asset and perp liquidations always transfer discounted value to the liquidator, the action would improve the liquidator's health and reduce systemic risk. The check prevents this, leaving under-maintenance subaccounts open longer than necessary and increasing the probability of bad-debt socialization.

---

### Likelihood Explanation

Realistic. A market participant who holds a large perp position that has moved against them can have initial health < 0 while maintenance health ≥ 0. This is a normal intermediate state during volatile markets — exactly when liquidations are most needed. Such a participant has both the economic incentive (discounted assets) and the technical ability to liquidate others, but is blocked by the post-liquidation health gate.

---

### Recommendation

Replace the blanket post-liquidation initial-health check on the sender with a check that only applies when the liquidation itself worsens the liquidator's health. Concretely, capture the liquidator's initial health before the payment step and only revert if the post-liquidation health is strictly lower than the pre-liquidation health and is negative. Alternatively, mirror the DSCEngine recommendation directly: remove the `!isUnderInitial(txn.sender)` gate from `_handleLiquidationPayment` and instead enforce only that the liquidatee is not over-liquidated (`ERR_LIQUIDATED_TOO_MUCH`), which is already checked on line 573. [4](#0-3) 

---

### Proof of Concept

1. Liquidator subaccount `L` holds a large long perp position. Market moves against them; their initial health drops to −80, maintenance health = +5 (not liquidatable).
2. Liquidatee subaccount `V` holds a long spot position; their maintenance health drops below 0 (liquidatable).
3. `L` calls `liquidateSubaccount` targeting `V`'s spot position with a positive `txn.amount`.
4. Inside `_handleLiquidationPayment`, the spot transfer executes: `V` loses spot assets, `L` gains them at the liquidation price (below oracle). `L`'s initial health improves from −80 to −30.
5. The check `!isUnderInitial(txn.sender)` evaluates `isUnderInitial(L)` → `true` (health is −30 < 0).
6. `require(... || !isUnderInitial(txn.sender), ERR_SUBACCT_HEALTH)` reverts.
7. `V` remains under-maintenance. `L` is denied a health-improving liquidation. The system accumulates bad debt. [5](#0-4) [6](#0-5)

    function _handleLiquidationPayment(
        IEndpoint.LiquidateSubaccount calldata txn,
        ISpotEngine spotEngine,
        IPerpEngine perpEngine
    ) internal {
        LiquidationVars memory v;
        address engine = txn.isEncodedSpread
            ? address(0)
            : address(productToEngine[txn.productId]);

        if (txn.isEncodedSpread) {
            uint32 spotId = txn.productId & 0xFFFF;
            uint32 perpId = txn.productId >> 16;
            (
                v.liquidationPriceX18,
                v.oraclePriceX18,
                v.oraclePriceX18Perp
            ) = getSpreadLiqPriceX18(spotId, perpId, txn.amount);

            v.liquidationPayment = v.liquidationPriceX18.mul(txn.amount);

            v.liquidationFees = (v.oraclePriceX18 - v.liquidationPriceX18)
                .mul(LIQUIDATION_FEE_FRACTION)
                .mul(txn.amount);

            // transfer spot at the calculated liquidation price
            spotEngine.updateBalance(spotId, txn.liquidatee, -txn.amount);
            spotEngine.updateBalance(
                QUOTE_PRODUCT_ID,
                txn.liquidatee,
                v.liquidationPayment
            );
            spotEngine.updateBalance(spotId, txn.sender, txn.amount);
            spotEngine.updateBalance(
                QUOTE_PRODUCT_ID,
                txn.sender,
                -v.liquidationPayment - v.liquidationFees
            );

            v.liquidationPayment = v.oraclePriceX18Perp.mul(txn.amount);
            perpEngine.updateBalance(
                perpId,
                txn.liquidatee,
                txn.amount,
                -v.liquidationPayment
            );

            perpEngine.updateBalance(
                perpId,
                txn.sender,
                -txn.amount,
                v.liquidationPayment
            );

            if (txn.amount < 0) {
                insurance = spotEngine.updateQuoteFromInsurance(
                    txn.liquidatee,
                    insurance
                );
            }
        } else if (engine == address(spotEngine)) {
            (v.liquidationPriceX18, v.oraclePriceX18) = getLiqPriceX18(
                txn.productId,
                txn.amount
            );

            v.liquidationPayment = v.liquidationPriceX18.mul(txn.amount);
            v.liquidationFees = (v.oraclePriceX18 - v.liquidationPriceX18)
                .mul(LIQUIDATION_FEE_FRACTION)
                .mul(txn.amount);

            spotEngine.updateBalance(
                txn.productId,
                txn.liquidatee,
                -txn.amount
            );

            spotEngine.updateBalance(
                QUOTE_PRODUCT_ID,
                txn.liquidatee,
                v.liquidationPayment
            );

            spotEngine.updateBalance(txn.productId, txn.sender, txn.amount);

            spotEngine.updateBalance(
                QUOTE_PRODUCT_ID,
                txn.sender,
                -v.liquidationPayment - v.liquidationFees
            );

            if (txn.amount < 0) {
                insurance = spotEngine.updateQuoteFromInsurance(
                    txn.liquidatee,
                    insurance
                );
            }
        } else {
            (v.liquidationPriceX18, v.oraclePriceX18) = getLiqPriceX18(
                txn.productId,
                txn.amount
            );
            v.liquidationPayment = v.liquidationPriceX18.mul(txn.amount);
            v.liquidationFees = (v.oraclePriceX18 - v.liquidationPriceX18)

                QUOTE_PRODUCT_ID,

            txn.sender == N_ACCOUNT || !isUnderInitial(txn.sender),

    function liquidateSubaccountImpl(IEndpoint.LiquidateSubaccount calldata txn)
        external
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.sender != txn.liquidatee, ERR_UNAUTHORIZED);
        require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
        require(
            txn.liquidatee != X_ACCOUNT && txn.liquidatee != N_ACCOUNT,
            ERR_NOT_LIQUIDATABLE
        );
        require(
            txn.productId != QUOTE_PRODUCT_ID,
            ERR_INVALID_LIQUIDATION_PARAMS
        );

        ISpotEngine spotEngine = ISpotEngine(
            address(engineByType[IProductEngine.EngineType.SPOT])
        );
        IPerpEngine perpEngine = IPerpEngine(
            address(engineByType[IProductEngine.EngineType.PERP])
        );

        if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
            if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.liquidatee);
            }
            return;
        }

        if (
            (txn.amount < 0) &&
            (txn.isEncodedSpread ||
                address(productToEngine[txn.productId]) == address(spotEngine))
        ) {
            // when it's spread or spot liquidation, we need to make sure the liquidatee has
            // enough quote to buyback the liquidated amount.
            _assertCanLiquidateLiability(txn, spotEngine, perpEngine);
            _settlePositivePerpPnl(txn, spotEngine, perpEngine);
        }

        _assertLiquidationAmount(txn, spotEngine, perpEngine);

        // beyond this point, we can be sure that we can liquidate the entire
        // liquidation amount knowing that the insurance fund will remain solvent
        // subsequently we can just blast the remainder of the liquidation and
        // cover the quote balance from the insurance fund at the end
        _handleLiquidationPayment(txn, spotEngine, perpEngine);
    }
