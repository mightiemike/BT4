### Title
`Program.__hash__` Uses Object Identity While `__eq__` Uses Content Equality, Breaking Dict/Set Invariants — (File: `wheel/python/clvm_rs/program.py`)

---

### Summary

The `Program` class defines `__hash__` using `id(self)` (Python object identity) and `__eq__` using `tree_hash()` (cryptographic content equality). These two metrics can diverge for any two `Program` objects constructed from the same CLVM bytes, violating Python's mandatory invariant that `a == b` implies `hash(a) == hash(b)`. Any caller that uses `Program` objects as dict keys or set members — a natural and common pattern — will silently receive wrong lookup results.

---

### Finding Description

In `wheel/python/clvm_rs/program.py`:

```python
def __hash__(self):          # line 100-101
    return id(self)          # object identity — changes every construction

def __eq__(self, other):     # line 182-188
    ...
    return self.tree_hash() == other_obj.tree_hash()  # content equality
``` [1](#0-0) [2](#0-1) 

Python's data model requires: if `a == b` then `hash(a) == hash(b)`. Here, two `Program` objects built from identical bytes satisfy `p1 == p2` (same `tree_hash()`) but `hash(p1) != hash(p2)` (different `id()`). A grep across all Python files in the repository confirms `__hash__` is defined only in this one location and is never overridden in the `CLVMStorage` base class. [3](#0-2) 

The `wrap()` classmethod copies `_cached_sha256_treehash` from the wrapped object, so the equality side is consistent — but `id()` is always fresh per Python object, so the hash side is not. [4](#0-3) 

---

### Impact Explanation

The broken invariant corrupts every Python `dict` and `set` that uses `Program` objects as keys:

- `{p1: solution}[p2]` raises `KeyError` even when `p1 == p2` (same puzzle bytes).
- `p2 in {p1}` returns `False` even when the programs are identical.

In the Chia ecosystem the `Program` API is the primary Python surface for puzzle/solution handling. Wallet code, mempool validators, and puzzle-hash caches that store `Program` objects as dict keys will silently fail to find entries they already hold. A concrete consequence is that a puzzle-hash cache keyed by `Program` will never hit, forcing re-evaluation on every lookup — or, worse, a security check that expects a cached result will fall through to an incorrect default.

---

### Likelihood Explanation

The `Program` class is the central Python type exported by the `clvm_rs` wheel and is used throughout Chia tooling. Any caller that naturally writes `cache[program] = result` or `program in seen_programs` — both idiomatic Python — will trigger the bug. The entry path is fully attacker-controlled: an adversary submits CLVM bytes that are deserialized into a `Program` object; if that object is then used as a dict key against a previously stored equal object, the lookup silently fails. No special privileges are required.

---

### Recommendation

Replace the `id(self)`-based hash with a content-based hash consistent with `__eq__`:

```python
def __hash__(self) -> int:
    return int.from_bytes(self.tree_hash()[:8], "big")
```

This makes `hash` and `eq` agree on the same content identity, restoring the Python invariant and making `Program` objects safe as dict keys and set members.

---

### Proof of Concept

```python
from clvm_rs.program import Program

# Two Program objects from identical bytes
p1 = Program.from_bytes(bytes.fromhex("ff0102"))   # (1 . 2)
p2 = Program.from_bytes(bytes.fromhex("ff0102"))   # same bytes, new object

# __eq__ says they are equal (tree_hash comparison)
assert p1 == p2, "programs are equal by content"

# __hash__ says they are different (id() comparison)
assert hash(p1) != hash(p2), "but hashes differ — invariant broken"

# Dict lookup silently fails
cache = {p1: "puzzle_solution"}
result = cache.get(p2)          # returns None
assert result is None           # lookup missed even though p1 == p2

# Set membership silently wrong
seen = {p1}
assert p2 not in seen           # False negative — p2 is "not seen"
```

The root cause is exclusively at `wheel/python/clvm_rs/program.py` lines 100–101 (`__hash__`) diverging from lines 182–188 (`__eq__`). No other Python file in the repository defines `__hash__` for this class. [1](#0-0) [2](#0-1)
