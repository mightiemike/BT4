### Title
Smart Contract Accounts Cannot Sign Protocol Transactions Due to ECDSA-Only Verification — (`core/contracts/Verifier.sol`, `core/contracts/OffchainExchange.sol`)

---

### Summary

`Verifier.validateSignature`, `Verifier.validateCompactSignature`, and `OffchainExchange._checkSignature` all use `ECDSA.recover` exclusively. Because `Endpoint.depositCollateral` constructs subaccounts directly from `msg.sender`, any smart contract (Safe multisig, DAO treasury, ERC-4337 account) can deposit collateral and create a subaccount — but can never authorize any fast-path signed operation, including trading, fast withdrawal, or quote transfer. There is no slow-mode trading path, so deposited collateral is permanently frozen from trading.

---

### Finding Description

`Endpoint.depositCollateral` constructs the subaccount identifier as:

```solidity
bytes32 subaccount = bytes32(abi.encodePacked(msg.sender, subaccountName));
``` [1](#0-0) 

There is no restriction on `msg.sender` being an EOA — any smart contract can call this and create a valid subaccount. The deposited funds are credited to a subaccount whose first 20 bytes are the contract's address.

All fast-path signed operations (sequencer-processed) route through `EndpointTx.validateSignedTx` → `EndpointTx.validateSignature` → `verifier.validateSignature`:

```solidity
function validateSignature(
    bytes32 sender,
    address linkedSigner,
    bytes32 digest,
    bytes memory signature
) public pure {
    address recovered = ECDSA.recover(digest, signature);
    require(
        (recovered != address(0)) &&
            ((recovered == address(uint160(bytes20(sender)))) ||
                (recovered == linkedSigner)),
        ERR_INVALID_SIGNATURE
    );
}
``` [2](#0-1) 

The compact variant has the same flaw:

```solidity
address recovered = ECDSA.recover(digest, signature.r, signature.vs);
``` [3](#0-2) 

`ECDSA.recover` calls the `ecrecover` precompile, which only works for EOA signatures. A smart contract account cannot produce a raw ECDSA signature — it implements EIP-1271 `isValidSignature` instead. The recovered address will never match the smart contract's address, so every fast-path transaction from a smart contract subaccount reverts with `ERR_INVALID_SIGNATURE`.

The same flaw exists in `OffchainExchange._checkSignature`, which gates all order matching:

```solidity
function _checkSignature(
    bytes32 subaccount,
    bytes32 digest,
    address linkedSigner,
    bytes memory signature
) internal view virtual returns (bool) {
    address signer = ECDSA.recover(digest, signature);
    return
        (signer != address(0)) &&
        (signer == address(uint160(bytes20(subaccount))) ||
            signer == linkedSigner);
}
``` [4](#0-3) 

There is no slow-mode trading path — order matching is exclusively a fast-path sequencer operation. A smart contract subaccount owner therefore cannot trade at all.

The slow-mode path (`processSlowModeTransactionImpl`) uses `validateSender` (a `msg.sender == address(bytes20(subaccount))` check, not a signature check), so slow-mode `WithdrawCollateral` and `LinkSigner` remain accessible to smart contracts. However:

- Slow-mode has a hardcoded 3-day delay. [5](#0-4) 
- There is no slow-mode path for trading, so collateral deposited by a smart contract is permanently frozen from any trading activity.
- If the smart contract sets a linked signer that is also a smart contract, the linked signer path in `_checkSignature` and `validateSignature` is equally broken.

---

### Impact Explanation

A smart contract account (Safe multisig, DAO treasury, ERC-4337 wallet) that deposits collateral into Nado:

1. **Cannot trade** — `OffchainExchange._checkSignature` uses `ECDSA.recover`; no slow-mode trading path exists.
2. **Cannot perform fast withdrawals** — `validateSignature` / `validateCompactSignature` block all fast-path `WithdrawCollateral` and `WithdrawCollateralV2`.
3. **Cannot perform fast `LinkSigner`, `TransferQuote`, `MintNlp`, `BurnNlp`** — all route through the same ECDSA-only verifier.

The deposited collateral is locked from trading indefinitely. The only escape is a slow-mode withdrawal after a 3-day delay, which requires the smart contract to remain operational and unsanctioned.

---

### Likelihood Explanation

Smart contract wallets (Safe/Gnosis multisig, ERC-4337 accounts) are standard for institutional users, DeFi protocols, and DAOs. The protocol places no restriction on `msg.sender` in `depositCollateral` or `depositCollateralWithReferral`. Any such user who deposits collateral — a normal, expected action — immediately encounters this lockout. Likelihood is **medium-high** given the prevalence of smart contract wallets among the institutional and DeFi-native user base that a derivatives protocol targets.

---

### Recommendation

Replace `ECDSA.recover` with OpenZeppelin's `SignatureChecker.isValidSignatureNow` in both `Verifier.validateSignature` / `validateCompactSignature` and `OffchainExchange._checkSignature`. `SignatureChecker` first attempts `ecrecover` for EOAs and falls back to EIP-1271 `isValidSignature` for smart contract accounts, preserving backward compatibility.

```solidity
// Instead of:
address recovered = ECDSA.recover(digest, signature);
require(recovered != address(0) && (recovered == owner || recovered == linkedSigner), ...);

// Use:
bool valid = SignatureChecker.isValidSignatureNow(address(uint160(bytes20(sender))), digest, signature)
          || (linkedSigner != address(0) && SignatureChecker.isValidSignatureNow(linkedSigner, digest, signature));
require(valid, ERR_INVALID_SIGNATURE);
```

---

### Proof of Concept

1. Deploy a Safe multisig at address `0xSafe`.
2. Call `Endpoint.depositCollateral(bytes12("default"), USDC_ID, 1000e6)` from `0xSafe`. Subaccount = `bytes32(abi.encodePacked(0xSafe, "default"))`.
3. The sequencer credits the deposit. The subaccount now holds 1000 USDC.
4. The Safe attempts to submit a signed `WithdrawCollateral` fast-path transaction. The sequencer calls `processTransactionImpl` → `validateSignedTx` → `verifier.validateSignature`. [6](#0-5) 
5. `ECDSA.recover(digest, safeSignature)` returns some garbage EOA address (or reverts), never `0xSafe`. The require fails with `ERR_INVALID_SIGNATURE`. [7](#0-6) 
6. The Safe also cannot place any orders — `OffchainExchange._checkSignature` returns `false` for the same reason. [8](#0-7) 
7. The 1000 USDC is frozen from trading. The only recovery path is a slow-mode `WithdrawCollateral` submitted directly by `0xSafe` (3-day delay, fee required).
