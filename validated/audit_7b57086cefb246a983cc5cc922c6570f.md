### Title
`topUpBaseRewards` Uses `totalBonded` for Shortfall Calculation but Distributes Only to Voting Validators — (`x/tieredrewards/keeper/abci.go`)

### Summary

`topUpBaseRewards` in the tiered-rewards `BeginBlocker` computes the per-block shortfall using `TotalBondedTokens` (all bonded validators), but distributes the resulting top-up only to validators present in `VoteInfos` (validators that signed the previous block). When any bonded validator misses a block, the shortfall is over-estimated by that validator's proportional target, the rewards pool is over-drained by that excess, and the voting validators receive more than their fair share of the top-up.

### Finding Description

In `topUpBaseRewards`, the target per-block staker reward is:

```go
targetStakersRewardPerBlock := math.LegacyNewDecFromInt(totalBonded).
    Mul(targetBaseRewardsRate).
    Quo(math.LegacyNewDec(int64(blocksPerYear)))
```

`totalBonded` is fetched from `stakingKeeper.TotalBondedTokens`, which sums tokens across **all** bonded validators, including those that did not sign the previous block. [1](#0-0) [2](#0-1) 

The shortfall is then computed against the fee-collector balance:

```go
shortFallAmount := targetStakersRewardPerBlock.Sub(defaultStakersRewardPerBlock).TruncateInt()
``` [3](#0-2) 

The full `topUpAmount` (derived from `shortFallAmount`) is transferred from the rewards pool to the distribution module and then allocated **only** to validators in `bondedVotes` — i.e., validators that appeared in `VoteInfos` for the previous block:

```go
bondedVotes := sdkCtx.VoteInfos()
// ...
for i, vote := range bondedVotes {
    // distribute reward to each voting validator
}
``` [4](#0-3) [5](#0-4) 

**The mismatch**: `totalBonded` (metric used for the shortfall) includes non-voting validators' tokens, but the distribution (metric used for payout) covers only voting validators. The shortfall is therefore inflated by:

```
nonVotingBonded × targetBaseRewardsRate / blocksPerYear
```

per block where any validator misses. The full inflated amount is transferred from the pool and allocated to voting validators, which receive more than their proportional share.

### Impact Explanation

The rewards pool (`RewardsPoolName` module account) is over-drained every block in which at least one bonded validator does not appear in `VoteInfos`. The excess tokens are permanently transferred to voting validators via `AllocateTokensToValidator` — they cannot be reclaimed. Over time this accelerates pool exhaustion, shortening the period during which the `targetBaseRewardsRate` guarantee can be honoured for all stakers. Once the pool is empty, the top-up mechanism silently stops and all stakers receive below-target base rewards indefinitely.

The corrupted value is the **rewards pool balance** (`RewardsPoolName` module account), which is drained faster than the protocol intends.

### Likelihood Explanation

Validators missing individual blocks is a routine occurrence on any live proof-of-stake network (network jitter, restarts, upgrades). No special conditions or privileged access are required. The `BeginBlocker` runs every block, so the over-drain accumulates continuously. A large validator (high bonded-token share) that misses even a small fraction of blocks produces a proportionally large over-drain.

### Recommendation

Replace `totalBonded` in the shortfall calculation with the sum of bonded tokens belonging only to the validators that appear in `VoteInfos`. Concretely, derive the voting-validator token total from `previousTotalPower` (already computed) scaled by the bond denom's power-reduction factor, or iterate `bondedVotes` and sum `validator.Tokens` for each entry. This ensures the shortfall and the distribution use the same population of validators.

### Proof of Concept

1. Two validators: V1 (1 000 000 bonded tokens, voted) and V2 (19 000 000 bonded tokens, **missed** the block). `totalBonded = 20 000 000`.
2. `targetBaseRewardsRate = 0.10`, `blocksPerYear = 6 311 520`.
3. `targetStakersRewardPerBlock = 20 000 000 × 0.10 / 6 311 520 ≈ 317` (all validators).
4. Fee collector is empty → `defaultStakersRewardPerBlock = 0`.
5. `shortFallAmount = 317`. Pool is drained by 317.
6. All 317 tokens are allocated to V1 (the only voter).
7. V1's delegators receive the target reward for **all 20 000 000** bonded tokens, not just their own 1 000 000.
8. The correct top-up for V1 alone would be `1 000 000 × 0.10 / 6 311 520 ≈ 16`.
9. The pool is over-drained by `317 − 16 = 301` tokens per such block. Repeated every block V2 misses, the pool is exhausted ~19× faster than intended.
