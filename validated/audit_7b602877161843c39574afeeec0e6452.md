### Title
`ResourceBoundsMapping → ValidResourceBounds` Version-Boundary Misclassification Produces Wrong Transaction Hash in RPC Simulation/Estimation — (`crates/apollo_rpc/src/v0_8/transaction.rs`)

### Summary

The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion in the RPC layer uses a zero-value heuristic (`l1_data_gas == 0 && l2_gas == 0`) to decide whether to produce `ValidResourceBounds::L1Gas` (pre-0.13.3 hash domain, omits `l1_data_gas` from the fee-fields hash) or `ValidResourceBounds::AllResources` (post-0.13.3 hash domain, includes `l1_data_gas`). A V3 transaction that legitimately sets both fields to zero is silently downgraded to the pre-0.13.3 hash domain, producing a transaction hash that diverges from the hash the user signed and from the hash the gateway computes.

### Finding Description

**Root cause — `crates/apollo_rpc/src/v0_8/transaction.rs` lines 188–199:**

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)          // ← pre-0.13.3 domain
        } else {
            Self::AllResources(AllResourceBounds {
                l1_gas: value.l1_gas,
                l1_data_gas: value.l1_data_gas,
                l2_gas: value.l2_gas,
            })
        }
    }
}
