### Title
`verify_transaction_inclusion_v2` Always Reverts When `verify_transaction_inclusion` Is Paused Due to Nested `#[pause]` Guard - (`File: contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` delegates its core Merkle proof logic to `verify_transaction_inclusion` via a direct internal Rust call. Both functions carry the `#[pause]` attribute from `near-plugins`. The `#[pause]` macro injects a runtime pause-state guard at the entry of every decorated function, and this guard fires on every call — including internal (non-cross-contract) calls within the same contract. When a `PauseManager` pauses the deprecated `verify_transaction_inclusion` method (a natural operational action to enforce migration to v2), the injected guard inside `verify_transaction_inclusion` panics on every invocation, including the one made by `verify_transaction_inclusion_v2`. The result is that `verify_transaction_inclusion_v2` — the production, non-deprecated endpoint — permanently reverts for all callers, even though it is not itself paused.

---

### Finding Description

`verify_transaction_inclusion_v2` is the current, recommended proof-verification endpoint. After performing its own coinbase Merkle proof validation, it delegates to `verify_transaction_inclusion` for the remaining logic: [1](#0-0) 

`verify_transaction_inclusion` is marked both `#[deprecated]` and `#[pause]`: [2](#0-1) 

The `near-plugins` `#[pause]` macro (version `0.2.0`, tag `v0.4.1`) transforms the decorated function by prepending a guard that checks whether the method name is in the contract's paused-methods set. This guard is part of the compiled function body, not a cross-contract call interceptor. Therefore, when `verify_transaction_inclusion_v2` calls `self.verify_transaction_inclusion(args.into())` at line 368, the guard inside `verify_transaction_inclusion` executes and panics if that method is paused — regardless of the caller being the contract itself. [3](#0-2) 

The `PauseManager` role is the only entity authorized to pause methods: [4](#0-3) 

There is no exception role defined on `verify_transaction_inclusion` (unlike `run_mainchain_gc` which has `except(roles(Role::UnrestrictedRunGC))`): [5](#0-4) 

---

### Impact Explanation

Once `verify_transaction_inclusion` is paused, every call to `verify_transaction_inclusion_v2` panics at line 368 with the near-plugins pause message. The entire proof-verification surface of the light client is silently broken: any downstream NEAR contract that calls `verify_transaction_inclusion_v2` to confirm a Bitcoin transaction receives a failed cross-contract call and cannot distinguish a "proof invalid" result from a "contract deadlocked" result. The canonical proof-verification result is corrupted from the perspective of all consumers.

**Impact: 5** — The primary production endpoint for transaction inclusion proofs is rendered permanently non-functional.

---

### Likelihood Explanation

The `#[deprecated]` annotation on `verify_transaction_inclusion` explicitly instructs users to migrate to v2. A `PauseManager` enforcing that deprecation by pausing the v1 method is a routine, well-motivated operational action. The `PauseManager` role is a normal operational role (not a DAO-level role), making this scenario realistic without requiring any malicious intent. The `#[allow(deprecated)]` suppressor at line 367 confirms the internal call is intentional and not a mistake, meaning the coupling is load-bearing and will not be removed by a routine refactor.

**Likelihood: 4** — Realistic operational trigger; no adversarial action required.

---

### Recommendation

Extract the shared Merkle proof logic from `verify_transaction_inclusion` into a private, unpause-guarded helper function (e.g., `verify_transaction_inclusion_inner`). Have both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` call this helper. This mirrors the fix described in the reference report: the inner logic must not carry the access-restricting guard that the outer public entry points carry.

```rust
// Private helper — no #[pause]
fn verify_transaction_inclusion_inner(&self, args: ProofArgs) -> bool {
    // ... shared logic ...
}

#[deprecated] #[pause]
pub fn verify_transaction_inclusion(&self, args: ProofArgs) -> bool {
    self.verify_transaction_inclusion_inner(args)
}

#[pause]
pub fn verify_transaction_inclusion_v2(&self, args: ProofArgsV2) -> bool {
    // coinbase proof check ...
    self.verify_transaction_inclusion_inner(args.into())
}
```

### Proof of Concept

1. Deploy the contract with any supported chain feature.
2. Grant account `pause_mgr` the `PauseManager` role.
3. As `pause_mgr`, call `pa_pause_feature("verify_transaction_inclusion")` to pause the deprecated v1 method.
4. As any unprivileged account, call `verify_transaction_inclusion_v2` with a valid proof.
5. Observe: the call panics (reverts) inside `verify_transaction_inclusion` at the `#[pause]` guard, even though `verify_transaction_inclusion_v2` is not paused and the proof is valid.
6. The caller receives a failed transaction; the proof result is lost. [6](#0-5)
