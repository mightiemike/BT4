Audit Report

## Title
`SwapAllowlistExtension` allowlist bypass via `MetricOmmSimpleRouter` — router address replaces real user as `sender` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed from the pool, which is always `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, the router contract becomes `msg.sender` of the pool, so the extension checks whether the **router** is allowlisted rather than the actual user. If the pool admin allowlists the router to permit router-mediated swaps, every unprivileged user can bypass the per-user allowlist by calling through the public router.

## Finding Description

**Call path:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle()` (or `exactInput`, `exactOutputSingle`, `exactOutput`).
2. The router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — so inside `MetricOmmPool.swap`, `msg.sender` is the **router address**.
3. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, passing the router as `sender`.
4. `ExtensionCalling._beforeSwap` encodes and dispatches to `SwapAllowlistExtension.beforeSwap(sender=router, ...)`.
5. `SwapAllowlistExtension.beforeSwap` checks:
   ```solidity
   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
       revert IMetricOmmPoolActions.NotAllowedToSwap();
   }
   ```
   Here `msg.sender` = pool, `sender` = router. The check is: *is the router allowlisted for this pool?*

**Root cause:** The pool correctly passes `msg.sender` as `sender` to extensions, but the extension interprets `sender` as the economically relevant user. When an intermediary public contract (the router) sits between the user and the pool, `sender` is the intermediary, not the user.

**Exploit flow:**
- Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a whitelist of addresses (e.g., KYC'd users).
- To allow those users to also use the router, the admin allowlists the router address: `setAllowedToSwap(pool, router, true)`.
- Any non-allowlisted user now calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting that pool.
- The extension sees `sender = router`, which is allowlisted, and permits the swap.
- The non-allowlisted user successfully swaps, bypassing the intended access control.

**Existing guards are insufficient:** The `onlyPool` modifier in `BaseMetricExtension` only verifies the caller is a registered pool — it does not verify the identity of the original user. There is no mechanism in the pool or extension framework to propagate the original EOA through the router.

## Impact Explanation
The `SwapAllowlistExtension` is a core access-control primitive. Its bypass allows unprivileged users to trade on pools intended to be restricted (e.g., permissioned liquidity pools, regulatory-compliant venues). This constitutes broken core pool functionality — the allowlist gate is rendered ineffective for any pool that also permits router-mediated swaps. Depending on pool configuration, this can expose restricted LP positions to unrestricted trading, directly impacting LP assets and protocol integrity.

## Likelihood Explanation
The condition is reachable by any unprivileged user with no special setup: they only need to call the public `MetricOmmSimpleRouter`. The bypass is triggered whenever a pool admin allowlists the router (a natural and expected operational step to support normal UX). The router is a public, permissionless contract. The attack is repeatable on every swap.

## Recommendation
The extension must gate on the **original user**, not the intermediary. Options:
1. Have the router pass the original `msg.sender` as `extensionData`, and update `SwapAllowlistExtension` to decode and check that address when `sender` is a known router.
2. Introduce a trusted-forwarder pattern: the router stores the original caller in transient storage, and the extension reads it via a standardized interface.
3. Alternatively, document that the router must never be allowlisted and that allowlisted users must call the pool directly — but this breaks normal UX and is not a code-level fix.
