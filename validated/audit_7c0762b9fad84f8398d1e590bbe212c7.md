### Title
Unchecked `transferFrom` Return Value Enables USDC-E Drain Without Providing USDC — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`replaceUsdcEWithUsdc` in `ContractOwner.sol` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the return value or using the protocol's own `ERC20Helper.safeTransferFrom` wrapper. If the USDC token on Ink chain returns `false` on a failed transfer instead of reverting, any unprivileged caller can drain USDC-E from any DDA subaccount without depositing any USDC.

---

### Finding Description

`replaceUsdcEWithUsdc` is an `external` function with no access control beyond a chain ID check (`block.chainid == 57073`). Its intended logic is a three-step atomic swap:

1. Pull `balance` worth of USDC from `msg.sender` into the DDA via `transferFrom`.
2. Withdraw USDC-E from the DDA to `ContractOwner`.
3. Send USDC-E to `msg.sender` via `safeTransfer`.

The critical flaw is at step 1:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // return value ignored
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
```

The return value of `transferFrom` is never captured or checked. Steps 2 and 3 execute unconditionally regardless of whether step 1 succeeded. The protocol already has `ERC20Helper.safeTransferFrom` — which validates the return value — and even uses `safeTransfer` on line 618 in the same function body, making the omission on line 616 a direct inconsistency. The safe wrapper that should have been used is defined in the protocol's own library: [2](#0-1)

### Impact Explanation

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink chain (chainid 57073) returns `false` on a failed transfer (e.g., caller has zero allowance or zero balance) rather than reverting, an attacker can:

- Call `replaceUsdcEWithUsdc(subaccount)` for any subaccount whose DDA holds USDC-E.
- The `transferFrom` silently fails (returns `false`, ignored).
- `withdraw(usdcE)` pulls all USDC-E from the DDA into `ContractOwner`.
- `safeTransfer(msg.sender, balance)` sends all USDC-E to the attacker.

**Corrupted asset delta**: The DDA loses its entire USDC-E balance; the attacker gains it; no USDC is deposited. The protocol's collateral accounting is corrupted because the DDA's token balance is drained without a corresponding deposit.

---

### Likelihood Explanation

The function is `external` with no `onlyOwner` or similar guard — any address on Ink chain can call it. The exploitability depends on whether the specific USDC deployment at the hardcoded address returns `false` vs. reverts on failure. Many USDC deployments (including Circle's canonical USDC) do revert, but the absence of a return-value check means the safety guarantee is entirely delegated to the token's behavior rather than enforced by the protocol. Given that the protocol already has and uses `safeTransfer` in the same function, this is a straightforward integration bug with a realistic path to exploitation if the token's behavior differs from the assumption.

---

### Recommendation

Replace the raw `transferFrom` call with the protocol's existing `ERC20Helper.safeTransferFrom`:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ContractOwner` already imports and uses `ERC20Helper` via `using ERC20Helper for IERC20Base`, so this is a one-word fix consistent with the rest of the codebase. [3](#0-2) 

---

### Proof of Concept

1. Identify a DDA subaccount on Ink chain (chainid 57073) holding a non-zero USDC-E balance.
2. Call `ContractOwner.replaceUsdcEWithUsdc(subaccount)` from an EOA with zero USDC allowance/balance.
3. If USDC's `transferFrom` returns `false` instead of reverting: the return value is silently discarded at line 616.
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` executes, pulling all USDC-E to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` executes, sending all USDC-E to the attacker.
6. Attacker receives `balance` USDC-E; DDA balance is zero; no USDC was deposited. [4](#0-3)

```text
    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
