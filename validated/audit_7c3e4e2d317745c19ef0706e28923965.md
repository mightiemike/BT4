Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the actual user, making the allowlist either universally bypassable or incompatible with the router — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `sender` equals the router contract address, not the actual user. This creates an irreconcilable conflict: allowlisting the router grants every on-chain address swap access (nullifying the allowlist), while not allowlisting it silently breaks all router-based swaps for every legitimately allowlisted user.

## Finding Description
The full call path is confirmed by the production code:

1. `MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` directly — making the router `msg.sender` to the pool. [1](#0-0) 

2. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, recipient, ...)` — passing the router as `sender`. [2](#0-1) 

3. `ExtensionCalling._beforeSwap` encodes and dispatches `IMetricOmmExtensions.beforeSwap(sender, recipient, ...)` — the router address propagates as `sender`. [3](#0-2) 

4. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]` — resolving to `allowedSwapper[pool][router]`, never `allowedSwapper[pool][user]`. [4](#0-3) 

The `recipient` (the actual user) is available as the second unnamed parameter in `beforeSwap` but is discarded entirely. [5](#0-4) 

No existing guard corrects this: the extension has no mechanism to unwrap the intermediary and identify the originating account.

## Impact Explanation
Two fund-impacting outcomes result from normal usage:

- **Allowlist bypass (LP principal loss):** A pool admin deploying a private/KYC-gated pool must set `allowedSwapper[pool][router] = true` to let allowlisted users trade via the router. This immediately grants every address on-chain the ability to swap against the pool. Because the pool is oracle-anchored and the oracle price is public, adversarial informed traders can systematically extract value from LPs whenever the oracle lags the true market price — a direct loss of LP principal that the extension was explicitly configured to prevent.

- **Broken core functionality:** If the admin does not allowlist the router, every allowlisted user's swap through `MetricOmmSimpleRouter` reverts with `NotAllowedToSwap`, rendering the pool's liquidity inaccessible via the standard periphery interface.

Both outcomes meet the allowed impact gate: the first is direct loss of LP principal through unrestricted adverse selection; the second is broken core swap functionality causing loss of access to funds.

## Likelihood Explanation
`SwapAllowlistExtension` is a production-ready extension explicitly intended for "private or permissioned liquidity pools." Any pool admin who configures per-user allowlists and expects users to trade through `MetricOmmSimpleRouter` — the primary and recommended swap interface — will encounter this conflict on the first swap attempt. No special attacker capability is required; the conflict is triggered by ordinary, documented usage.

## Recommendation
The extension must identify the actual initiating account, not the intermediary. Two concrete approaches:

1. **Check `recipient` instead of `sender`:** The `recipient` (second parameter in `beforeSwap`) is already available and equals the user in `exactInputSingle`. Change the check to `allowedSwapper[msg.sender][recipient]` for single-hop swaps. This requires no router changes. [6](#0-5) 

2. **Pass the real initiator through `extensionData`:** The router encodes the original `msg.sender` into `extensionData` before calling `pool.swap`. The extension decodes and checks that address. This requires a convention between the router and the extension but supports multi-hop paths where `recipient` may be an intermediate contract.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — Alice is the only intended swapper.
3. Alice calls `MetricOmmSimpleRouter.exactInputSingle({..., recipient: alice})`.
4. Router calls `pool.swap(alice, zeroForOne, amount, priceLimit, "", extensionData)` — `msg.sender` to the pool is the router.
5. Pool calls `_beforeSwap(router, alice, ...)`.
6. `SwapAllowlistExtension.beforeSwap(sender=router, ...)` evaluates `allowedSwapper[pool][router]` → `false`.
7. Transaction reverts with `NotAllowedToSwap` even though Alice is explicitly allowlisted.

**Bypass path:** Admin calls `setAllowedToSwap(pool, router, true)`.

8. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
9. `allowedSwapper[pool][router]` → `true` → Bob's swap succeeds.
10. The allowlist is fully bypassed; any user can swap against the pool.
