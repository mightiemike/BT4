### Title
Linked Signer Can Unilaterally Rotate Its Own Authorization Without Subaccount Owner Confirmation — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

In `processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`, meaning the currently registered linked signer can sign a `LinkSigner` transaction to replace itself with any arbitrary address. The subaccount owner is never required to confirm this change. A compromised or malicious linked signer can therefore permanently transfer signing authority over a subaccount to an attacker-controlled address, enabling persistent unauthorized access and full collateral theft.

---

### Finding Description

The `LinkSigner` transaction type is handled in two code paths:

**Slow-mode path** (`processSlowModeTransactionImpl`, lines 232–239): uses `validateSender`, which enforces that `msg.sender` matches the address embedded in the subaccount `bytes32`. Only the subaccount owner can submit this path. [1](#0-0) 

**Fast path** (`processTransactionImpl`, lines 576–590): calls `validateSignedTx` with `allowLinkedSigner = true`. This passes the currently registered linked signer as an accepted signer to `verifier.validateSignature`. [2](#0-1) 

`validateSignature` in `Verifier.sol` accepts a signature from either the subaccount owner address or the `linkedSigner` address: [3](#0-2) 

Because `allowLinkedSigner = true` is passed for `LinkSigner` in the fast path, the linked signer can sign a `LinkSigner` transaction that writes a new address into `linkedSigners[subaccount]`. There is no restriction preventing the linked signer from modifying the authorization state it was granted access through. [4](#0-3) 

The `validateSignedTx` function only checks nonce validity and that the subaccount exists (`requireSubaccount`). Neither check prevents the linked signer from issuing a `LinkSigner` mutation. [5](#0-4) 

---

### Impact Explanation

A linked signer that is compromised (e.g., a hot wallet or API key) can:

1. Sign a `LinkSigner` transaction pointing `signedTx.tx.signer` to an attacker-controlled address.
2. The sequencer processes this, writing `linkedSigners[victim_subaccount] = attacker_address`.
3. The attacker's address is now the registered linked signer and can sign `WithdrawCollateral` or `WithdrawCollateralV2` transactions, draining all collateral from the subaccount. [6](#0-5) 

The corrupted state is `linkedSigners[subaccount]` — a persistent mapping that controls who can authorize all future signed transactions for that subaccount. The asset delta is the full collateral balance of the victim subaccount. Even if the owner attempts to revoke the linked signer via the slow-mode path (3-day delay), the attacker can front-run by rotating to a fresh address before the revocation executes.

---

### Likelihood Explanation

Linked signers are the standard mechanism for API-key-style delegation in Nado. Any user operating via an automated trading system or third-party integration will have a linked signer set. A single compromised hot wallet key is sufficient to trigger the attack. No privileged access, governance capture, or admin keys are required. The attacker only needs to know the current nonce for the subaccount owner's address, which is publicly readable via `getNonce`. [7](#0-6) 

---

### Recommendation

`LinkSigner` transactions must only be accepted from the subaccount owner — the address embedded in the first 20 bytes of the `sender` `bytes32`. In `processTransactionImpl`, change the `allowLinkedSigner` argument for the `LinkSigner` case from `true` to `false`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only the subaccount owner may rotate the linked signer
);
```

This aligns the fast path with the slow-mode path, which already enforces `validateSender` (owner-only) for `LinkSigner`. [1](#0-0) 

---

### Proof of Concept

1. Alice has subaccount `alice_subaccount` with `linkedSigners[alice_subaccount] = hot_wallet`.
2. Attacker compromises `hot_wallet` private key.
3. Attacker reads `nonces[address(alice_subaccount)]` from `Endpoint.getNonce`.
4. Attacker signs a `SignedLinkSigner` transaction using `hot_wallet`:
   - `tx.sender = alice_subaccount`
   - `tx.signer = attacker_address`
   - `tx.nonce = current_nonce`
5. Attacker submits this to the sequencer (standard API call, no privilege required).
6. Sequencer calls `processTransactionImpl`; `validateSignedTx` passes because `hot_wallet == getLinkedSigner(alice_subaccount)` and `allowLinkedSigner = true`.
7. `linkedSigners[alice_subaccount]` is overwritten with `attacker_address`.
8. Attacker signs a `WithdrawCollateral` transaction with `attacker_address`, draining Alice's collateral.
9. Alice's slow-mode revocation (3-day delay) arrives too late; the attacker has already rotated to a fresh key and withdrawn funds. [8](#0-7)

        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),

        );
