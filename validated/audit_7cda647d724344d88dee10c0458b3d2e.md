Audit Report

## Title
Immutable Extension Configuration With No Error Isolation Permanently Bricks `removeLiquidity` When Extension Reverts ŌĆö (File: `metric-core/contracts/ExtensionCalling.sol`)

## Summary
`ExtensionCalling` stores all seven extension addresses and six hook-order bitmasks as `immutable` values set once at construction with no post-deployment update path. The `_callExtensionsInOrder` dispatcher calls each extension via `CallExtension.callExtension` with no try/catch, propagating any revert directly to the caller. If any extension registered for `BEFORE_REMOVE_LIQUIDITY_ORDER` or `AFTER_REMOVE_LIQUIDITY_ORDER` permanently reverts, every LP's call to `removeLiquidity` is permanently bricked with no admin recovery path, because `removeLiquidity` carries no `whenNotPaused` guard and no factory setter can update extension slots.

## Finding Description

**Immutable extension slots:** All seven extension addresses and six hook-order bitmasks are set once in the constructor and cannot be changed:

```solidity
// ExtensionCalling.sol L17-35
address internal immutable EXTENSION_1;
...
uint256 internal immutable BEFORE_REMOVE_LIQUIDITY_ORDER;
uint256 internal immutable AFTER_REMOVE_LIQUIDITY_ORDER;
```

**No error isolation in dispatcher:** `_callExtensionsInOrder` iterates the packed bitmask and calls each extension with no try/catch:

```solidity
// ExtensionCalling.sol L75-86
function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    ...
    CallExtension.callExtension(extension, data);
    ...
}
```

**`CallExtension.callExtension` propagates reverts:** In `CallExtension.sol` L8-32, if `success == false`, the function either re-reverts with the returned revert data or reverts with `ExtensionCallFailed()`. There is no swallowing of errors.

**`removeLiquidity` calls both hooks without `whenNotPaused`:** At `MetricOmmPool.sol` L199-212, `removeLiquidity` is guarded only by `nonReentrant` ŌĆö not `whenNotPaused`. It unconditionally calls `_beforeRemoveLiquidity` (L207) and `_afterRemoveLiquidity` (L211). By contrast, `swap` at L224 carries `whenNotPaused`.

**No admin recovery path:** The only factory-callable setters on the pool are `collectFees`, `setPoolFees`, `setPause`, `setBinAdditionalFees`, and `setPriceProvider`. None can update extension addresses or hook-order bitmasks. `setPause` only gates `swap` via `whenNotPaused`; it does not help LPs calling `removeLiquidity`.

**Periphery extensions confirm the risk surface:** `OracleValueStopLossExtension` reads oracle prices and reverts on stop-loss breach (`afterSwap`). `PriceVelocityGuardExtension` reads prior price observations and reverts if velocity is exceeded (`beforeSwap`). `DepositAllowlistExtension` and `SwapAllowlistExtension` gate by external allowlist state. Any custom extension implementing `beforeRemoveLiquidity` or `afterRemoveLiquidity` that depends on external state inherits this risk. Additionally, if any of the shipped extensions is registered for a remove-liquidity hook it does not implement, `BaseMetricExtension`'s default handler reverts with `ExtensionNotImplemented()` immediately and permanently.

## Impact Explanation

If any extension registered under `BEFORE_REMOVE_LIQUIDITY_ORDER` or `AFTER_REMOVE_LIQUIDITY_ORDER` permanently reverts ŌĆö whether due to an external oracle/feed deprecation, an allowlist registry failure, or a stop-loss threshold permanently breached by a market event ŌĆö every call to `removeLiquidity` reverts. LP positions are frozen with no recovery path. This is direct, permanent loss of user principal, satisfying the allowed impact gate: *"Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows."*

The secondary case ŌĆö a permanently reverting `beforeSwap` or `afterSwap` extension ŌĆö bricks all swaps. `setPause` can also gate swaps, but cannot fix the extension failure for `removeLiquidity`.

## Likelihood Explanation

The trigger is an external operational event, not a privileged malicious action. Concrete scenarios:
1. A custom extension implementing `beforeRemoveLiquidity` reads an oracle feed that is deprecated or returns a permanently stale/invalid answer, causing the extension to always revert.
2. An extension registered for a remove-liquidity hook does not implement that hook (e.g., any of the shipped periphery extensions), causing `ExtensionNotImplemented()` on every `removeLiquidity` call from deployment.
3. An allowlist registry used by a remove-liquidity guard loses its admin key, permanently locking or unlocking access in a way that causes the extension to always revert.

None of these require a privileged actor to act maliciously. The immutability of extension slots means there is zero tolerance for any post-deployment failure in any registered extension.

## Recommendation

1. **Wrap extension calls in try/catch** inside `_callExtensionsInOrder`, with a configurable per-slot fail-open/fail-closed flag, so a broken extension emits a failure event rather than aborting the pool operation.
2. **Make extension slots mutable** via a factory-gated setter (analogous to `setPriceProvider`) so the factory admin can remove or replace a broken extension without redeploying the pool.
3. **Exempt `removeLiquidity` from extension hooks** or provide a separate emergency withdrawal path that bypasses extension calls, ensuring LPs can always recover principal even if guard extensions fail.

  function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;

    while (true) {
      uint256 extensionIndex = order & 0x7;
      if (extensionIndex == 0) break;
      address extension = _extensionAddress(extensionIndex);
      if (extension == address(0)) revert PanicEmptyExtension();
      CallExtension.callExtension(extension, data);
      order >>= 3;
    }
  }
