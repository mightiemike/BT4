### Title
`ReceiptView → Receipt` Roundtrip Silently Downgrades `ActionReceiptV2` to `ActionReceipt` (V1) When `refund_to` Is `None` — (`File: core/primitives/src/views.rs`)

---

### Summary

The `TryFrom<ReceiptView> for Receipt` conversion in `core/primitives/src/views.rs` reconstructs an `ActionReceipt` (Borsh discriminant `0`) instead of `ActionReceiptV2` (Borsh discriminant `5`) whenever `refund_to` is `None`, even though the runtime **always** emits `ActionReceiptV2` for every new cross-contract receipt. This is an acknowledged, unfixed inconsistency (tracked as issue #14709). Any code path that serialises a receipt to its view representation and then deserialises it back — including the `EXPERIMENTAL_receipt` RPC endpoint, sandbox/archival replay, and any tooling that round-trips receipts through JSON — will silently produce a structurally different receipt with a different Borsh encoding and a different discriminant byte.

---

### Finding Description

The runtime's `function_call.rs` always creates outgoing receipts as `ReceiptEnum::ActionV2` (discriminant `5`):

```rust
// runtime/runtime/src/function_call.rs:172-184
let new_action_receipt = ActionReceiptV2 {
    signer_id: ...,
    refund_to: receipt.refund_to,   // None for ordinary cross-contract calls
    ...
};
let new_receipt = if receipt.is_promise_yield {
    ReceiptEnum::PromiseYieldV2(new_action_receipt)
} else {
    ReceiptEnum::ActionV2(new_action_receipt)   // discriminant 5
};
```

The view conversion in `views.rs` serialises both `ActionReceipt` and `ActionReceiptV2` into the same flat `ReceiptEnumView::Action { ..., refund_to: Option<AccountId> }` struct. On the way back (`TryFrom<ReceiptView> for Receipt`), the reconstruction branches solely on whether `refund_to.is_some()`:

```rust
// core/primitives/src/views.rs:2518-2551
// Note that this is not consistent with how `new_receipts` are
// created by the runtime - there we always create ActionReceiptV2.
// ActionReceiptV2 without refund_to becomes V1 after a roundtrip
// through views. This will be fixed with #14709.
if refund_to.is_some() {
    ReceiptEnum::ActionV2(...)   // discriminant 5
} else {
    ReceiptEnum::Action(...)     // discriminant 0  ← wrong for V2 receipts
}
```

Because ordinary cross-contract calls have `refund_to = None`, every such receipt is reconstructed as `ReceiptEnum::Action` (discriminant `0`) after a view roundtrip, even though it was originally `ReceiptEnum::ActionV2` (discriminant `5`).

The two variants have **different Borsh encodings**: `ActionReceipt` does not carry the `refund_to` field at all, while `ActionReceiptV2` carries it as an `Option`. Their discriminant bytes differ (`0` vs `5`). Any consumer that Borsh-serialises the reconstructed receipt — for hashing, state-witness construction, or re-submission — will produce bytes that do not match the original on-chain receipt.

---

### Impact Explanation

**Divergent Borsh bytes / receipt-hash mismatch.** A receipt that was originally stored on-chain as `ReceiptEnum::ActionV2` (discriminant `5`) is reconstructed as `ReceiptEnum::Action` (discriminant `0`) after a view roundtrip. Borsh-serialising the reconstructed receipt produces a different byte sequence, so any hash computed over it will differ from the canonical on-chain hash.

**Broken receipt re-submission / sandbox replay.** Tools or contracts that fetch a receipt via the RPC view layer and attempt to re-submit or replay it will inject a structurally different receipt. Downstream validation that checks the receipt's Borsh hash against the on-chain value will fail or silently accept a wrong receipt.

**State-witness / chunk-hash divergence.** If a stateless-validation path ever reconstructs receipts from their view representation (e.g., during archival replay or cross-shard proof verification), the reconstructed receipt will have a different discriminant and therefore a different hash, breaking the chunk state-root invariant.

---

### Likelihood Explanation

The bug is reachable by any unprivileged user or tool that:
1. Queries a receipt via the `EXPERIMENTAL_receipt` or `changes` RPC endpoint (which returns `ReceiptView`), and
2. Converts the result back to a `Receipt` via `TryFrom<ReceiptView>`.

This is the normal path for archival nodes, sandbox environments, and any indexer that processes receipts. The condition (`refund_to == None`) is satisfied by **every** ordinary cross-contract call receipt produced after the `ActionReceiptV2` migration, making this the common case rather than an edge case.

---

### Recommendation

In `TryFrom<ReceiptView> for Receipt` (`core/primitives/src/views.rs`), preserve the original receipt version by adding a discriminant field to `ReceiptEnumView::Action` (e.g., `is_v2: bool`) that is set during the `Receipt → ReceiptView` conversion based on whether the source was `ReceiptEnum::ActionV2` / `ReceiptEnum::PromiseYieldV2`. The reconstruction should then use `ActionReceiptV2` whenever the source was V2, regardless of whether `refund_to` is `Some` or `None`. This is the fix already planned under issue #14709.

Alternatively, always reconstruct as `ActionReceiptV2` (since V2 is a strict superset of V1 with `refund_to = None` being semantically equivalent to the V1 behaviour), and update all callers that pattern-match on `ReceiptEnum::Action` to also handle `ReceiptEnum::ActionV2`.

---

### Proof of Concept

1. Submit a cross-contract function call (no `refund_to`). The runtime creates `ReceiptEnum::ActionV2 { refund_to: None, ... }` (discriminant `5`).
2. Query the receipt via `EXPERIMENTAL_receipt` RPC → receive `ReceiptEnumView::Action { refund_to: None, ... }`.
3. Call `Receipt::try_from(receipt_view)`. The branch at line 2522 evaluates `refund_to.is_some()` → `false`, so it constructs `ReceiptEnum::Action(ActionReceipt { ... })` (discriminant `0`).
4. Borsh-serialise both the original and the reconstructed receipt. The original begins with discriminant byte `5`; the reconstructed begins with discriminant byte `0`. The byte sequences differ, so `borsh::to_vec(&original) != borsh::to_vec(&reconstructed)`.
