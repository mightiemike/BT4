## Analysis

Let me trace the full exploit path carefully.

**Step 1: Is `ForceFullExitWithDelegation` user-callable?**

The function is a keeper method, not a `MsgServer` handler. It is only invoked by the v8 upgrade migration handler. However, the v8 upgrade is explicitly listed as a production path ("genesis/migration/upgrade path with attacker-relevant state").

**Step 2: Can a vesting account have a LockTier position with amount > originalVesting before v8?**

Yes. Before v8, `validateNewPosition` did not include the vesting filter. The filter (`validateNonVestingAccount`) was added as part of v8. A `PermanentLockedAccount` with `originalVesting = X` and `balance = 3X` (X locked + 2X spendable) could call `MsgLockTier` with `amount = 2X` using the spendable portion.

**Step 3: What does `alignVestingDelegationTracking` do for this case?**

After `transferDelegationFromPosition` returns 2X delegation to the owner (with `subtractAccount=false`, skipping `TrackDelegation`):
- `actualDelegated = 2X`, `tracked = DV + DF = 0`, `deficit = 2X`
- `TrackDelegation(2X, 2X)` is called at line 171

The SDK's `BaseVestingAccount.TrackDelegation` logic:
- `vestingAmt = originalVesting = X`
- `delVestingAmt = DV = 0`
- `x = min(X, max(0, X - 0)) = X` → `DV += X` → `DV = X` (saturated)
- `y = 2X - X = X` → `DF += X` → `DF = X`

**Step 4: Does DV saturation unlock the bank balance?**

For `PermanentLockedAccount`:
```
LockedCoins = vestingCoins - min(vestingCoins, DV)
            = X - min(X, X) = 0
SpendableCoins = balance - 0 = balance = X (the originally locked coins)
```

After `alignVestingDelegationTracking` sets `DV = X`, the owner's bank balance (containing the permanently locked X coins) becomes fully spendable. This is the unlock.

**Step 5: Full exploit trace**

| State | bank balance | DV | DF | LockedCoins | SpendableCoins |
|---|---|---|---|---|---|
| Initial | 3X | 0 | 0 | X | 2X |
| After LockTier(2X) | X | 0 | 0 | X | 0 |
| After v8 ForceFullExit | X | X | X | 0 | **X ← UNLOCKED** |
| Owner undelegates X (DF portion) | 2X | X | 0 | 0 | **2X ← STILL UNLOCKED** |
| Owner spends 2X | 0 | X | 0 | 0 | 0 |
| Owner undelegates remaining X | X | 0 | 0 | X | 0 |

Net: owner started with X locked + 2X spendable = 3X total. Ended with X locked + 2X transferred out. The permanently locked X coins have been unlocked and moved.

**Step 6: Is the test suite aware of this?**

`TestForceFullExitWithDelegation_VestingOwner_LockOrigin` and `TestForceFullExitWithDelegation_VestingOwner_Mixed` do not assert `SpendableCoins == 0` after the force exit. The mixed test only checks `spendable.LTE(bal)` (line 309), which is trivially true and does not catch the unlock.

`TestTransferDelegationFromPosition_VestingOwnerForcedExitWithDelegation` (line 547) does assert `SpendableCoins.IsZero()`, but only *after* undelegation completes — at which point `TrackUndelegation` has already reduced DV back to 0, re-locking the coins. The window between force exit and undelegation is not tested.

**Step 7: Guards that do NOT prevent this**

- `validateNonVestingAccount`: only blocks new position creation post-v8, not the migration
- `TrackDelegation` invariant check (`amount <= balance`): passes because `balance == amount` is passed explicitly at line 171
- No guard prevents spending bank balance when `DV == originalVesting`

### Title
`alignVestingDelegationTracking` saturates `DelegatedVesting` to `originalVesting` for LockTier-origin positions, making permanently locked bank coins temporarily spendable during v8 migration — (`x/tieredrewards/keeper/force_exit.go`)

### Summary

During the v8 upgrade migration, `ForceFullExitWithDelegation` calls `alignVestingDelegationTracking` to fix stale-low `DV+DF` tracking for `LockTier`-origin positions. For a `PermanentLockedAccount` owner whose position amount ≥ `originalVesting`, this saturates `DelegatedVesting` at `originalVesting`. Because `LockedCoins = vestingCoins - min(vestingCoins, DV)`, saturation drives `LockedCoins` to zero, making the owner's entire bank balance — including the permanently locked principal — spendable until they undelegate.

### Finding Description

`alignVestingDelegationTracking` computes `deficit = actualDelegated - (DV + DF)` and calls `vacc.TrackDelegation(blockTime, coins(deficit), coins(deficit))` at line 171. [1](#0-0) 

For a `PermanentLockedAccount` with `originalVesting = X` and a LockTier-origin position of amount `≥ X`, the deficit fills DV to `X` (saturated) and assigns the remainder to DF. Once `DV == originalVesting`, the SDK's `LockedCoins` formula returns zero, making the owner's bank balance fully spendable. [2](#0-1) 

The `lockFunds` path for `MsgLockTier` sends coins from the owner's bank to the position delegator via `bankKeeper.SendCoins`, leaving the owner's remaining bank balance (the locked principal) intact but locked. [3](#0-2) 

After `alignVestingDelegationTracking` saturates DV, that remaining bank balance (the locked principal) becomes spendable. The owner can transfer it out before undelegating. When they later undelegate, `TrackUndelegation` reduces DV back to zero, re-locking the returned coins — but the original principal has already left the account.

### Impact Explanation

Permanently locked coins (intended to be irrevocably locked in a `PermanentLockedAccount`) can be transferred out during the window between the v8 migration and the owner's subsequent undelegation. This is a direct loss of locked assets: value leaves the intended account boundary in violation of the vesting invariant.

### Likelihood Explanation

The exploit requires:
1. A `PermanentLockedAccount` that created a `LockTier` position before v8 (allowed pre-v8, since `validateNonVestingAccount` was not yet enforced)
2. The v8 upgrade to fire (a scheduled, one-time production event)
3. The owner to submit a bank send transaction after the migration block

The integration test `_create_vesting_acc_owned_positions` explicitly creates this exact pre-condition (a `PermanentLockedAccount` with both `CommitDelegationToTier` and `LockTier` positions), confirming the scenario was considered reachable. [4](#0-3) 

The post-migration assertion `assert_v8_vesting_acc_owned_positions_exited` explicitly expects `DF == lock_amount` (overflow into `DelegatedFree`) but does not assert `SpendableCoins == 0`. [5](#0-4) 

The unit test `TestForceFullExitWithDelegation_VestingOwner_LockOrigin` similarly omits a `SpendableCoins` check after the force exit. [6](#0-5) 

### Recommendation

After calling `alignVestingDelegationTracking`, assert (or enforce) that `SpendableCoins` for the vesting owner has not increased beyond what it was before the force exit. Alternatively, instead of calling `TrackDelegation` with the full deficit, cap the top-up at `originalVesting - DV` (i.e., only fill the vesting bucket, never overflow into `DelegatedFree` for a `PermanentLockedAccount`). A `PermanentLockedAccount` should never have `DelegatedFree > 0` because no coins ever vest free.

func (k Keeper) alignVestingDelegationTracking(ctx context.Context, ownerAddr sdk.AccAddress) error {
	logger := k.logger(ctx)
	acc := k.accountKeeper.GetAccount(ctx, ownerAddr)
	if acc == nil {
		logger.Error("vesting-alignment: owner account not found, skipping",
			"owner", ownerAddr.String(),
		)
		return nil
	}
	vacc, ok := acc.(sdkvesting.VestingAccount)
	if !ok {
		logger.Error("vesting-alignment: owner is not a vesting account, skipping",
			"owner", ownerAddr.String(),
		)
		return nil
	}

	bondDenom, err := k.stakingKeeper.BondDenom(ctx)
	if err != nil {
		return fmt.Errorf("bond denom: %w", err)
	}

	delegations, err := k.stakingKeeper.GetDelegatorDelegations(ctx, ownerAddr, 1000)
	if err != nil {
		return fmt.Errorf("get delegator delegations: %w", err)
	}
	actualDelegated := sdkmath.ZeroInt()
	for _, d := range delegations {
		valAddr, err := sdk.ValAddressFromBech32(d.GetValidatorAddr())
		if err != nil {
			return fmt.Errorf("parse validator address %q: %w", d.GetValidatorAddr(), err)
		}
		val, err := k.stakingKeeper.GetValidator(ctx, valAddr)
		if err != nil {
			return fmt.Errorf("get validator %s: %w", valAddr, err)
		}
		actualDelegated = actualDelegated.Add(val.TokensFromShares(d.Shares).TruncateInt())
	}

	dv := vacc.GetDelegatedVesting().AmountOf(bondDenom)
	df := vacc.GetDelegatedFree().AmountOf(bondDenom)
	tracked := dv.Add(df)
	logger.Info("vesting-alignment: pre-state",
		"owner", ownerAddr.String(),
		"delegated_vesting", dv.String(),
		"delegated_free", df.String(),
		"tracked_total", tracked.String(),
		"actual_delegations", actualDelegated.String(),
	)

	if !actualDelegated.GT(tracked) {
		logger.Info("vesting-alignment: no top-up needed",
			"owner", ownerAddr.String(),
			"gap", actualDelegated.Sub(tracked).String(),
		)
		return nil
	}
	deficit := actualDelegated.Sub(tracked)
	coins := sdk.NewCoins(sdk.NewCoin(bondDenom, deficit))
	// Pass balance == amount: TrackDelegation only uses balance for an
	// invariant check (amount <= balance); the DV/DF split is computed from
	// vestingCoins(blockTime) and DelegatedVesting. The owner's actual bank
	// balance is irrelevant here because the delegation came from the position
	// pool, not from the owner's balance.
	sdkCtx := sdk.UnwrapSDKContext(ctx)
	vacc.TrackDelegation(sdkCtx.BlockTime(), coins, coins)
	k.accountKeeper.SetAccount(ctx, vacc)

def assert_v8_vesting_acc_owned_positions_exited(cluster, ctx):
    """Verify the v8 migration:
    - both vesting-owned positions are deleted;
    - the vesting owner's staking delegation equals commit + lock amounts;
    - DelegatedVesting saturates at OriginalVesting (=commit_amount), and
      the LockTier amount overflows into DelegatedFree, so DV+DF == Σ
      delegations and the vesting bookkeeping invariant holds.
    """
    owner_addr = ctx["owner_addr"]
    val_addr = ctx["val_addr"]
    commit_amount = ctx["commit_amount"]
    lock_amount = ctx["lock_amount"]
    total_amount = commit_amount + lock_amount

    # Both positions deleted.
    positions_after = query_positions_by_owner(cluster, owner_addr).get("positions", [])
    assert (
        positions_after == []
    ), f"expected zero positions post-upgrade, got {positions_after}"

    # Vesting metadata still intact.
    post_acct = unwrap_account(cluster.cosmos_cli().account(owner_addr))
    assert post_acct["@type"] in (
        "cosmos-sdk/PermanentLockedAccount",
        "/cosmos.vesting.v1beta1.PermanentLockedAccount",
    ), f"vesting metadata must survive, got {post_acct}"

    # Owner has staking delegation restored, equal to commit + lock.
    deleg = query_staking_delegation(cluster, owner_addr, val_addr)
    deleg_amount = int(deleg["balance"]["amount"])
    assert deleg_amount == total_amount, (
        f"owner delegation should be {total_amount} "
        f"(commit={commit_amount} + lock={lock_amount}), got {deleg_amount}"
    )

    # DV saturates at OriginalVesting (=commit_amount); LockTier amount
    # overflows into DF. DV+DF must equal Σ delegations.
    dv, df = _vesting_delegated_amounts(post_acct)
    assert dv == commit_amount, (
        f"DelegatedVesting should saturate at OriginalVesting={commit_amount}; "
        f"got DV={dv}"
    )
    assert df == lock_amount, (
        f"DelegatedFree should equal the LockTier amount={lock_amount}; " f"got DF={df}"
    )
    assert dv + df == total_amount, (
        f"DV+DF must equal Σ delegations: "
        f"DV({dv}) + DF({df}) = {dv + df}, expected {total_amount}"
    )

func (s *KeeperSuite) TestForceFullExitWithDelegation_VestingOwner_LockOrigin() {
	s.setupTier(1)
	vals, bondDenom := s.getStakingData()
	val := vals[0]
	valAddr := sdk.MustValAddressFromBech32(val.GetOperator())
	s.setValidatorCommission(valAddr, sdkmath.LegacyZeroDec())

	lockedAmount := sdkmath.NewInt(sdk.DefaultPowerReduction.Int64())
	// OriginalVesting equals the locked amount; the account is funded with
	// 2*lockedAmount so that lockedAmount of it is spendable (balance −
	// LockedCoins). LockTier consumes the spendable portion via bank send,
	// without touching DelegatedVesting/DelegatedFree.
	owner := s.newVestingOwnerWithBalance(bondDenom, lockedAmount, lockedAmount.MulRaw(2))

	pos := s.createLockTierPositionV1(owner, valAddr, lockedAmount)

	// Pre-migration: DV/DF are zero (LockTier didn't touch them); owner has
	// no on-chain delegation.
	s.Require().True(s.delegatedVesting(owner).AmountOf(bondDenom).IsZero(),
		"DV must be zero for a LockTier-origin position before force exit")
	s.Require().True(s.delegatedFree(owner).AmountOf(bondDenom).IsZero(),
		"DF must be zero for a LockTier-origin position before force exit")
	s.Require().True(s.totalDelegated(owner).IsZero(),
		"owner has no on-chain delegation pre force-exit; position holds it")

	s.advanceForRewards(valAddr, bondDenom)

	s.Require().NoError(s.keeper.ForceFullExitWithDelegation(s.ctx, pos.Id))

	_, err := s.keeper.GetPosition(s.ctx, pos.Id)
	s.Require().Error(err)
	ownerDel := s.totalDelegated(owner)
	s.Require().Equal(lockedAmount, ownerDel,
		"owner must hold the returned delegation post force-exit")

	// Alignment must have topped up DV+DF by lockedAmount; otherwise a later
	// normal Undelegate would underflow vesting accounting.
	s.Require().Equal(lockedAmount, s.delegatedVesting(owner).AmountOf(bondDenom),
		"DV must saturate at OriginalVesting (= lockedAmount)")
	s.Require().True(s.delegatedFree(owner).AmountOf(bondDenom).IsZero(),
		"DF must be zero — the deficit fits entirely within OriginalVesting")
	s.Require().Equal(ownerDel, s.trackedTotal(owner, bondDenom),
		"alignment must satisfy DV + DF == Σ delegations for LockTier-origin positions")
}
