### Title
Low-Level Call to Non-Existent Token Contract Returns Success, Enabling Silent Deposit Crediting Without Token Transfer - (File: `core/contracts/libraries/ERC20Helper.sol`)

---

### Summary

`ERC20Helper.safeTransfer` and `ERC20Helper.safeTransferFrom` use a raw low-level `.call()` on the token address without first verifying that the target address contains deployed bytecode. Per EVM semantics, a `.call()` to an address with no code returns `(true, "")`. The post-call check `success && (data.length == 0 || abi.decode(data, (bool)))` evaluates to `true` in this case, silently passing. This is the exact same root cause as the external report. In Nado, this affects the deposit and withdrawal paths, with the deposit path being the most severe: a user can receive on-chain balance credit in the clearinghouse without transferring any tokens.

---

### Finding Description

`ERC20Helper.safeTransfer` and `ERC20Helper.safeTransferFrom` are implemented as:

```solidity
(bool success, bytes memory data) = address(self).call(
    abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
);
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [1](#0-0) [2](#0-1) 

When `address(self)` has no deployed code (e.g., a token that was self-destructed, or a product registered with a CREATE2 address before the token is deployed), the EVM returns `(true, "")`. The guard condition `success && (data.length == 0 || ...)` becomes `true && true`, so `require` passes without reverting. No tokens are moved.

This library is consumed in two critical paths:

**1. Deposit path** — `EndpointStorage.handleDepositTransfer`:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
}
``` [3](#0-2) 

The only guard is `address(token) != address(0)`. A non-zero address with no code passes this check. Both `safeTransferFrom` and `safeTransferTo` silently succeed. The slow-mode transaction is then queued and eventually processed by `clearinghouse.depositCollateral`, which credits the user's balance. [4](#0-3) 

**2. Withdrawal path** — `BaseWithdrawPool.handleWithdrawTransfer`:

```solidity
function handleWithdrawTransfer(IERC20Base token, address to, uint128 amount) internal virtual {
    token.safeTransfer(to, uint256(amount));
}
``` [5](#0-4) 

Called from both `submitWithdrawal` (clearinghouse-gated) and `submitFastWithdrawal` (public). If the token has no code, the transfer silently succeeds, the withdrawal is marked as processed (`markedIdxs[idx] = true`), but no tokens reach the recipient. [6](#0-5) 

---

### Impact Explanation

**Deposit path (critical accounting corruption):** If a registered product's token address has no code, any user calling `depositCollateralWithReferral` with that `productId` will have their balance credited in the clearinghouse without transferring any real tokens. The clearinghouse's internal accounting diverges from its actual token holdings. The user can then trade, withdraw, or liquidate against phantom collateral, draining real funds from other users.

**Withdrawal path (permanent fund loss):** If the token has no code at withdrawal time, `submitWithdrawal` marks the withdrawal index as processed and irreversibly consumed (`markedIdxs[idx] = true`, `minIdx = idx`), but no tokens are sent. The user's on-chain balance is debited with no recourse. [7](#0-6) 

---

### Likelihood Explanation

Trigger conditions:
- A token registered in a spot product is self-destructed after registration (possible on any EVM chain that still supports `selfdestruct`).
- A product is registered using a CREATE2-predetermined address before the token bytecode is deployed (a deployment ordering mistake).
- A chain migration or redeployment scenario where the token address is reused but not yet live.

The deposit path is reachable by any unprivileged user via the public `depositCollateralWithReferral` function. The fast-withdrawal path is reachable by any caller via the public `submitFastWithdrawal`. No privileged access is required once the token address has no code. [8](#0-7) [9](#0-8) 

---

### Recommendation

Add a code-existence check before the low-level call in both `safeTransfer` and `safeTransferFrom` in `ERC20Helper`:

```solidity
require(address(self).code.length > 0, ERR_TRANSFER_FAILED);
```

Alternatively, replace the low-level call pattern with a typed interface call (e.g., `IERC20Base(self).transfer(to, amount)`) which will revert on a non-contract address in Solidity ≥0.8.0 when the call returns no data and the return type is non-void. OpenZeppelin's `SafeERC20` with `Address.functionCall` also performs this check. [10](#0-9) 

---

### Proof of Concept

1. Admin registers a spot product with token address `T` (valid at registration time).
2. Token `T` is self-destructed (or was a CREATE2 address never deployed).
3. Attacker calls `Endpoint.depositCollateralWithReferral(attackerSubaccount, productId, 1_000_000e6, "")`.
4. `handleDepositTransfer` is entered; `address(T) != address(0)` passes.
5. `ERC20Helper.safeTransferFrom(T, attacker, 1_000_000e6)` → `T.call(...)` → `(true, "")` → `require` passes. No tokens moved.
6. `ERC20Helper.safeTransferTo(T, clearinghouse, 1_000_000e6)` → same, passes.
7. Slow-mode transaction is queued. Sequencer processes it; `clearinghouse.depositCollateral` credits attacker's subaccount with `1_000_000e6` (normalized) of the phantom token.
8. Attacker uses this phantom collateral to borrow real quote tokens or withdraw real assets, draining the protocol. [11](#0-10) [12](#0-11)

        (

        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);

            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
