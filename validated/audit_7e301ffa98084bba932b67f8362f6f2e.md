### Title
`nSubmissions` Unconditional Increment on Failed Slow-Mode Transactions Enables Fast-Withdrawal Double-Spend — (`core/contracts/Endpoint.sol`)

---

### Summary

`executeSlowModeTransaction` increments `nSubmissions` unconditionally after `_executeSlowModeTransaction` returns, even when the inner `processSlowModeTransaction` call reverts and is silently swallowed on non-hardhat chains. Because `nSubmissions` is used verbatim as the `idx` parameter passed to `clearinghouse.withdrawCollateral → BaseWithdrawPool.submitWithdrawal`, an attacker who pre-positions a failing slow-mode transaction can shift the `idx` used for a subsequent sequencer-processed withdrawal by +1, causing the withdrawal to land on an `idx` slot that was never marked by the corresponding fast-withdrawal, resulting in a second token transfer to the attacker.

---

### Finding Description

**Root cause — unconditional increment:** [1](#0-0) 

```solidity
function executeSlowModeTransaction() external {
    SlowModeConfig memory _slowModeConfig = slowModeConfig;
    _executeSlowModeTransaction(_slowModeConfig, false);
    nSubmissions += 1;          // ← always runs, even after silent revert
    slowModeConfig = _slowModeConfig;
}
```

**Silent catch on non-hardhat chains:** [2](#0-1) 

On `chainid != 31337`, `processSlowModeTransaction` is wrapped in `try … {} catch {}`. A revert that does not trigger the OOG heuristic is swallowed, yet control returns normally to `executeSlowModeTransaction`, which then increments `nSubmissions`.

**`nSubmissions` used as withdrawal `idx`:**

In `processTransactionImpl` (sequencer path): [3](#0-2) 

In `processSlowModeTransactionImpl` (slow-mode path): [4](#0-3) 

Both pass the live `nSubmissions` storage value as `idx` to `clearinghouse.withdrawCollateral`.

**`submitWithdrawal` skips transfer when `markedIdxs[idx]` is already set:** [5](#0-4) 

```solidity
function submitWithdrawal(IERC20Base token, address to, uint128 amount, uint64 idx) public {
    require(msg.sender == clearinghouse);
    if (markedIdxs[idx]) { return; }   // ← silent skip
    markedIdxs[idx] = true;
    minIdx = idx;
    handleWithdrawTransfer(token, to, amount);
}
```

**`submitFastWithdrawal` marks `markedIdxs[idx]` for a caller-supplied `idx`:** [6](#0-5) 

The `idx` is included in the signed digest: [7](#0-6) 

```solidity
bytes32 data = keccak256(abi.encodePacked(uint256(block.chainid), uint256(idx), txn));
```

So fast-withdrawal signatures are bound to a specific `idx`.

**Token transfer happens before the early-return check:** [8](#0-7) 

```solidity
function handleWithdrawTransfer(...) internal virtual {
    token.safeTransfer(withdrawPool, uint256(amount));          // ← tokens leave clearinghouse
    BaseWithdrawPool(withdrawPool).submitWithdrawal(..., idx);  // ← may return early
}
```

---

### Impact Explanation

**Double-spend via fast-withdrawal + `nSubmissions` inflation:**

| Step | Action | State |
|------|--------|-------|
| 0 | `nSubmissions = N` | — |
| 1 | Attacker requests fast-withdrawal; protocol signers sign `(chainid, N, txn)` | — |
| 2 | Attacker calls `submitFastWithdrawal(N, txn, sigs)` | `markedIdxs[N]=true`; attacker receives tokens from pool |
| 3 | Attacker calls `executeSlowModeTransaction` with a pre-staged failing tx | `nSubmissions = N+1` |
| 4 | Sequencer submits next batch (now with `idx = N+1`); withdrawal processed | `nSubmissions` read as `N+1` inside `processTransactionImpl` |
| 5 | `clearinghouse.withdrawCollateral(…, N+1)` → `handleWithdrawTransfer` sends tokens to pool, then `submitWithdrawal(…, N+1)` | `markedIdxs[N+1]` is **false** → tokens transferred to attacker **again** |
| 6 | Attacker's balance debited once; attacker received tokens twice | Clearinghouse loses `amount` tokens |

The clearinghouse transfers `amount` tokens to the withdrawPool (step 5), and the pool forwards them to the attacker because `markedIdxs[N+1]` was never set. The attacker's on-chain balance is debited only once (in step 5's `spotEngine.updateBalance`), but they hold twice the tokens.

---

### Likelihood Explanation

- **Precondition 1**: Attacker must have a failing slow-mode transaction executable (requires 3-day `SLOW_MODE_TX_DELAY`). This is trivially satisfied by submitting a malformed or over-limit withdrawal 3 days in advance.
- **Precondition 2**: Attacker must obtain a fast-withdrawal signature from the protocol signers for `idx = N`. This is a normal user-facing operation.
- **Timing window**: The attacker must call `executeSlowModeTransaction` after submitting the fast withdrawal but before the sequencer includes the withdrawal in a batch. Given typical sequencer latency, this window is realistic.
- No admin keys, governance capture, or off-chain compromise required.

---

### Recommendation

Move `nSubmissions += 1` inside the success branch of `_executeSlowModeTransaction`, or track whether the inner call succeeded and only increment on success:

```solidity
function executeSlowModeTransaction() external {
    SlowModeConfig memory _slowModeConfig = slowModeConfig;
    bool success = _executeSlowModeTransaction(_slowModeConfig, false);
    if (success) nSubmissions += 1;
    slowModeConfig = _slowModeConfig;
}
```

Alternatively, return a boolean from `_executeSlowModeTransaction` indicating whether `processSlowModeTransaction` completed without reverting, and gate the increment on that value.

---

### Proof of Concept

```
Setup (3 days before attack):
  attacker.submitSlowModeTransaction(malformed_tx)   // will revert when processed

Attack day:
  1. nSubmissions = N  (read on-chain)
  2. attacker requests fast-withdrawal for amount=X, idx=N from protocol signers
  3. attacker.submitFastWithdrawal(N, txn, sigs)
     → markedIdxs[N] = true
     → withdrawPool transfers X tokens to attacker
  4. attacker.executeSlowModeTransaction()
     → _executeSlowModeTransaction: try processSlowModeTransaction() → reverts, caught
     → nSubmissions += 1  →  nSubmissions = N+1
  5. sequencer.submitTransactionsChecked(N+1, [WithdrawCollateral(attacker, X)], …)
     → processTransactionImpl: clearinghouse.withdrawCollateral(attacker, pid, X, 0, N+1)
     → handleWithdrawTransfer: token.safeTransfer(withdrawPool, X)
     → submitWithdrawal(token, attacker, X, N+1): markedIdxs[N+1] == false
     → withdrawPool transfers X tokens to attacker AGAIN
     → spotEngine.updateBalance debits attacker once

Result: attacker holds 2X tokens; clearinghouse lost X tokens.
``` [9](#0-8) [10](#0-9) [11](#0-10) [7](#0-6)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }

    function executeSlowModeTransaction() external {
        SlowModeConfig memory _slowModeConfig = slowModeConfig;
        _executeSlowModeTransaction(_slowModeConfig, false);
        nSubmissions += 1;
        slowModeConfig = _slowModeConfig;
    }

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }

    function submitWithdrawal(
        IERC20Base token,
        address sendTo,
        uint128 amount,
        uint64 idx
    ) public {
        require(msg.sender == clearinghouse);

        if (markedIdxs[idx]) {
            return;
        }
        markedIdxs[idx] = true;
        // set minIdx to most recent withdrawal submitted by sequencer
        minIdx = idx;

        handleWithdrawTransfer(token, sendTo, amount);
    }

    function requireValidTxSignatures(
        bytes calldata txn,
        uint64 idx,
        bytes[] calldata signatures
    ) public view {
        require(signatures.length <= 256, "too many signatures");
        bytes32 data = keccak256(
            abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
        );
        bytes32 hashedMsg = keccak256(
            abi.encodePacked("\x19Ethereum Signed Message:\n32", data)
        );

        uint256 nSignatures = 0;
        for (uint256 i = 0; i < signatures.length; i++) {
            if (signatures[i].length > 0) {
                nSignatures += 1;
                require(
                    checkIndividualSignature(
                        hashedMsg,
                        signatures[i],
                        uint8(i)
                    ),
                    "invalid signature"
                );
            }
        }
        require(nSignatures == nSigner, "not enough signatures");
    }

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
