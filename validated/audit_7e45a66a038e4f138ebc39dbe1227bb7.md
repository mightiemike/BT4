### Title
`RuntimeDialect::op()` Silently Ignores `_extensions: OperatorSet`, Breaking Softfork-Guard Operator Activation — (`File: src/runtime_dialect.rs`)

---

### Summary

`RuntimeDialect` implements the `Dialect` trait but silently discards the `OperatorSet` extension value that the interpreter passes when executing operators inside a softfork guard. Simultaneously, `RuntimeDialect::softfork_extension()` always returns `OperatorSet::Default` regardless of the extension number encoded in the CLVM program. Together these two defects mean that any operator whose activation depends on a softfork extension (e.g., keccak256 via extension 1) is permanently unreachable inside a softfork guard when `RuntimeDialect` is the active dialect. The expected control flow is silently broken in the same structural way as the proxy-admin wiring bug in the reference report: a parameter is accepted by the interface but never routed to the component that actually controls the behavior.

---

### Finding Description

**Broken invariant 1 — `softfork_extension()` always returns `Default`**

`src/runtime_dialect.rs` lines 73–75:
```rust
fn softfork_extension(&self, _ext: u32) -> OperatorSet {
    OperatorSet::Default
}
```

`ChiaDialect::softfork_extension()` (`src/chia_dialect.rs` lines 269–283) maps extension `1` → `OperatorSet::Keccak`. `RuntimeDialect` maps every extension number to `OperatorSet::Default`.

**Broken invariant 2 — `op()` ignores the `_extensions` argument**

`src/runtime_dialect.rs` lines 40–61:
```rust
fn op(
    &self,
    allocator: &mut Allocator,
    o: NodePtr,
    argument_list: NodePtr,
    max_cost: Cost,
    _extensions: OperatorSet,   // ← silently discarded
) -> Response {
    let atom = allocator.atom(o);
    let b = atom.as_ref();
    if b.len() == 1 && let Some(f) = self.f_lookup[b[0] as usize] {
        return f(allocator, argument_list, max_cost, self.flags);
    }
    ...
}
```

`ChiaDialect::op()` (`src/chia_dialect.rs` lines 144–154) uses the `extension` argument to augment the active flags:
```rust
let flags = self.flags
    | match extension {
        OperatorSet::Keccak => ClvmFlags::ENABLE_KECCAK_OPS_OUTSIDE_GUARD,
        _ => ClvmFlags::empty(),
    };
```

`RuntimeDialect::op()` never performs this augmentation. Even if `softfork_extension()` were fixed to return `OperatorSet::Keccak`, the `op()` function would still not enable keccak256 because `_extensions` is never read.

**Execution path through the interpreter**

`src/run_program.rs` lines 400–427 show the interpreter calling `self.dialect.softfork_extension(ext)` to obtain the `OperatorSet`, storing it in `SoftforkGuard::operator_set`, and then passing it as `current_extensions` to every `self.dialect.op()` call made inside the guard (lines 435–447). Both links in this chain are broken for `RuntimeDialect`.

---

### Impact Explanation

Any CLVM program that uses `(softfork cost 1 <program-using-keccak256>)` will behave differently under `RuntimeDialect` versus `ChiaDialect`:

- Under `ChiaDialect`: keccak256 (opcode 62) is enabled inside the guard via `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`; the program executes correctly.
- Under `RuntimeDialect`: opcode 62 falls through to `unknown_operator`. In consensus mode (`allow_unknown_ops = true`) it returns nil; in mempool mode (`NO_UNKNOWN_OPS`) it raises `Unimplemented`.

This is a **consensus/API divergence**: any caller that uses `RuntimeDialect` (e.g., the Python wheel's `run_program` path) will accept or reject transactions differently from a node running `ChiaDialect`, or will compute wrong intermediate values that downstream signature/hash checks depend on.

---

### Likelihood Explanation

`RuntimeDialect` is a public, exported type (`src/lib.rs`) consumed by the Python wheel and any downstream integrator that builds a custom operator map. An attacker who can submit CLVM programs to a validator using `RuntimeDialect` can craft a `softfork`-guarded keccak256 call that produces nil instead of the correct hash. If that hash feeds a comparison or a coin-ID derivation, the attacker can cause the validator to accept an invalid spend or reject a valid one, diverging from nodes running `ChiaDialect`.

---

### Recommendation

1. In `RuntimeDialect::softfork_extension()`, implement the same extension-to-`OperatorSet` mapping as `ChiaDialect`, or delegate to a shared helper.
2. In `RuntimeDialect::op()`, rename `_extensions` to `extensions` and apply the same flag-augmentation logic as `ChiaDialect::op()` before dispatching to `f_lookup`.

```rust
// src/runtime_dialect.rs
fn softfork_extension(&self, ext: u32) -> OperatorSet {
    match ext {
        0 => OperatorSet::Bls,
        1 => OperatorSet::Keccak,
        _ => OperatorSet::Default,
    }
}

fn op(&self, allocator: &mut Allocator, o: NodePtr,
      argument_list: NodePtr, max_cost: Cost,
      extensions: OperatorSet) -> Response {
    let effective_flags = self.flags | match extensions {
        OperatorSet::Keccak => ClvmFlags::ENABLE_KECCAK_OPS_OUTSIDE_GUARD,
        _ => ClvmFlags::empty(),
    };
    let atom = allocator.atom(o);
    let b = atom.as_ref();
    if b.len() == 1 && let Some(f) = self.f_lookup[b[0] as usize] {
        return f(allocator, argument_list, max_cost, effective_flags);
    }
    ...
}
```

### Proof of Concept

```rust
// Demonstrates that keccak256 inside softfork extension 1 returns nil
// under RuntimeDialect but the correct hash under ChiaDialect.
use clvmr::{Allocator, ChiaDialect, ClvmFlags, run_program};
use clvmr::runtime_dialect::RuntimeDialect;
use clvmr::serde::node_from_bytes;
use std::collections::HashMap;

// Program: (softfork (q . 100) (q . 1) (keccak256 (q . "hello")))
// Extension 1 should enable keccak256 inside the guard.
let program_hex = /* ... crafted softfork+keccak256 program bytes ... */;

let mut a = Allocator::new();
let prg = node_from_bytes(&mut a, &program_hex).unwrap();
let env = a.nil();

// ChiaDialect: keccak256 executes correctly, returns hash
let chia = ChiaDialect::new(ClvmFlags::empty());
let r1 = run_program(&mut a, &chia, prg, env, 1_000_000);
assert!(r1.is_ok());

// RuntimeDialect: keccak256 treated as unknown op, returns nil → divergence
let runtime = RuntimeDialect::new(HashMap::new(), vec![1], vec![2], ClvmFlags::empty());
let r2 = run_program(&mut a, &runtime, prg, env, 1_000_000);
// r2 produces nil instead of the keccak hash — consensus divergence
assert_ne!(r1.unwrap().1, r2.unwrap().1);
