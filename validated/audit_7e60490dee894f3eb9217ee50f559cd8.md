### Title
Bonus Accrual Checkpoint Unconditionally Advanced on Zero-Truncated Bonus, Causing Permanent Loss of Accrued Rewards — (`x/tieredrewards/keeper/claim_rewards.go`, `x/tieredrewards/keeper/bonus_rewards.go`)

---

### Summary

In `processEventsAndClaimBonus`, `applyBonusAccrualCheckpoint` unconditionally advances `pos.LastBonusAccrual` to the current block time even when every call to `computeSegmentBonus` returns zero due to integer truncation. The position is then persisted with the advanced checkpoint. The accrued time window is permanently discarded without the user receiving any bonus tokens.

---

### Finding Description

**Root cause — `computeSegmentBonus` truncates to zero**

`computeSegmentBonus` computes:

```
shares × tokensPerShare × bonusApy × durationSeconds / SecondsPerYear
```

and finalises with `.TruncateInt()`:

```go
// x/tieredrewards/keeper/bonus_rewards.go:41-45
return tokens.
    Mul(tier.BonusApy).
    MulInt64(durationSeconds).
    QuoInt64(types.SecondsPerYear).
    TruncateInt()
```

For any combination of small shares, short elapsed time, or low APY where the product is less than 1 (in the smallest denomination), the result is `0`.

Concrete example with 1 CRO (`10^8` uatom) locked, 5 % APY, 1-second segment:

```
10^8 × 1.0 × 0.05 × 1 / 31_557_600 ≈ 0.158  →  TruncateInt() = 0
```

**Root cause — checkpoint always advanced regardless of zero result**

After summing all segment bonuses, `processEventsAndClaimBonus` unconditionally calls `applyBonusAccrualCheckpoint` at line 215, then returns early with empty coins at line 219–221:

```go
// x/tieredrewards/keeper/claim_rewards.go:215-221
applyBonusAccrualCheckpoint(&pos.Position, blockTime)   // ← always runs
pos.UpdateLastKnownBonded(bonded)

if totalBonus.IsZero() {
    return sdk.NewCoins(), nil   // ← returns zero, but checkpoint already advanced
}
```

`applyBonusAccrualCheckpoint` sets `pos.LastBonusAccrual = blockTime`:

```go
// x/tieredrewards/keeper/bonus_rewards.go:15-21
func applyBonusAccrualCheckpoint(pos *types.Position, blockTime time.Time) {
    accrualEnd := blockTime
    if pos.CompletedExitLockDuration(blockTime) {
        accrualEnd = pos.ExitUnlockAt
    }
    pos.UpdateLastBonusAccrual(accrualEnd)
}
```

The caller then persists the mutated position:

```go
// x/tieredrewards/keeper/claim_rewards.go:129
if err := k.setPosition(ctx, pos.Position, nil); err != nil {
```

The time window `[old LastBonusAccrual, blockTime)` is permanently erased. The user cannot reclaim it on the next call because `segmentStart` will be the new, advanced `LastBonusAccrual`.

**Affected call sites**

Every path that settles rewards before a mutation shares this flaw:

| Message | Settling call |
|---|---|
| `MsgClaimTierRewards` | `claimRewardsAndUpdatesPositions` → `processEventsAndClaimBonus` |
| `MsgTierRedelegate` | `claimRewards` → `processEventsAndClaimBonus` |
| `MsgAddToTierPosition` | `claimRewards` → `processEventsAndClaimBonus` |
| `MsgTierUndelegate` | `claimRewards` → `processEventsAndClaimBonus` |
| `MsgClearPosition` | `claimRewards` → `processEventsAndClaimBonus` |
| `MsgExitTierWithDelegation` | `claimRewards` → `processEventsAndClaimBonus` |
| Tier governance (update/delete) | `claimRewardsAndUpdateTierPositions` → `processEventsAndClaimBonus` |

---

### Impact Explanation

A user with a small locked position (at or near `MinLockAmount`) who claims frequently — or who is forced through a reward-settling path by their own mutation messages — will have their bonus accrual checkpoint advanced to the current block time while receiving zero bonus tokens. The sub-unit bonus that had accumulated since the last claim is silently discarded. Over many such calls the user permanently loses a material fraction of their entitled bonus rewards. The lost tokens remain in the rewards pool and are effectively redistributed to other claimants.

---

### Likelihood Explanation

`MinLockAmount` is a governance parameter with no enforced lower bound in the keeper. Any position whose `shares × tokensPerShare × bonusApy × durationSeconds / SecondsPerYear < 1` will trigger the bug. This is reachable by:

1. A user with a small position calling `MsgClaimTierRewards` or any mutation message (redelegate, add-to-position, etc.) more frequently than the bonus accumulation rate requires.
2. A governance action (tier parameter update) that triggers `claimRewardsAndUpdateTierPositions` across all positions in a tier, including small ones.

Both are normal, unprivileged production entry points.

---

### Recommendation

Move `applyBonusAccrualCheckpoint` inside the `totalBonus > 0` branch, so the checkpoint is only advanced when the user actually receives tokens:

```go
// x/tieredrewards/keeper/claim_rewards.go
if totalBonus.IsZero() {
    // Do NOT advance LastBonusAccrual — preserve the accrued window.
    pos.UpdateLastKnownBonded(bonded)
    return sdk.NewCoins(), nil
}

applyBonusAccrualCheckpoint(&pos.Position, blockTime)
pos.UpdateLastKnownBonded(bonded)
```

Alternatively, carry the sub-unit remainder in the position state (a `PendingBonusRemainder` field) so that truncated fractions accumulate across claims rather than being discarded.

---

### Proof of Concept

1. Governance sets `MinLockAmount = 1_000_000 uatom` (1 CRO), `BonusApy = 0.05`.
2. Alice calls `MsgLockTier` with `amount = 1_000_000 uatom` on a bonded validator. `tokensPerShare ≈ 1.0`.
3. Alice calls `MsgClaimTierRewards` 10 seconds later.
   - `computeSegmentBonus`: `1_000_000 × 1.0 × 0.05 × 10 / 31_557_600 ≈ 1.58 → TruncateInt() = 1` (just above zero here).
4. Alice calls `MsgClaimTierRewards` again 1 second later.
   - `computeSegmentBonus`: `1_000_000 × 1.0 × 0.05 × 1 / 31_557_600 ≈ 0.158 → TruncateInt() = 0`.
   - `applyBonusAccrualCheckpoint` advances `LastBonusAccrual` by 1 second.
   - Alice receives 0 bonus tokens; the 1-second accrual window is permanently lost.
5. Repeating step 4 every second, Alice loses all sub-unit bonus fractions. Over a year, the cumulative loss approaches the full entitled bonus for those discarded windows.

The entry path is `MsgClaimTierRewards` (a standard signed transaction by the position owner), which calls `claimRewardsAndUpdatesPositions` [1](#0-0) 
→ `processEventsAndClaimBonus` [2](#0-1) 
where `computeSegmentBonus` truncates to zero [3](#0-2) 
but `applyBonusAccrualCheckpoint` still advances the checkpoint unconditionally. [4](#0-3) [5](#0-4)

```go
func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}
```

```go
func (k Keeper) processEventsAndClaimBonus(ctx context.Context, pos *types.PositionState) (sdk.Coins, error) {
	// Rewards should have been claimed before undelegation
	if !pos.IsDelegated() {
		return sdk.NewCoins(), nil
	}

	valAddr, err := sdk.ValAddressFromBech32(pos.Delegation.ValidatorAddress)
	if err != nil {
		return nil, err
	}

	events, err := k.getValidatorEventsSince(ctx, valAddr, pos.LastEventSeq)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	blockTime := sdkCtx.BlockTime()

	totalBonus := math.ZeroInt()
	// Use the persisted bonded state from the last replay, not a hardcoded default.
	// This prevents overpaying bonus for unbonded gaps between claims.
	bonded := pos.LastKnownBonded
	segmentStart := pos.LastBonusAccrual

	tier, err := k.getTier(ctx, pos.TierId)
	if err != nil {
		return nil, err
	}

	for _, entry := range events {
		evt := entry.Event

		if bonded {
			// Compute bonus for the bonded segment [segmentStart, eventTime]
			// using the snapshot rate at the event.
			bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
			totalBonus = totalBonus.Add(bonus)
		}

		// Update bonded state based on event type.
		switch evt.EventType {
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_UNBOND:
			bonded = false
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_BOND:
			bonded = true
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_SLASH:
			// Slash doesn't change bonded state.
		}

		segmentStart = evt.Timestamp
		pos.UpdateLastEventSeq(entry.Seq)

		// Decrement reference count.
		if err := k.decrementEventRefCount(ctx, valAddr, entry.Seq); err != nil {
			return nil, err
		}
	}

	val, err := k.stakingKeeper.GetValidator(ctx, valAddr)
	if err != nil {
		return nil, err
	}
	// Defensive: validator bond status check
	if bonded && val.IsBonded() {
		currentRate, err := k.getTokensPerShare(ctx, valAddr)
		if err != nil {
			return nil, err
		}
		bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
		totalBonus = totalBonus.Add(bonus)
	}

	applyBonusAccrualCheckpoint(&pos.Position, blockTime)
	// Persist the bonded state so the next replay starts correctly.
	pos.UpdateLastKnownBonded(bonded)

	if totalBonus.IsZero() {
		return sdk.NewCoins(), nil
	}
```
