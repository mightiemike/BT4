### Title
`is_proposal_init_valid` timestamp lower-bound check silently passes when `previous_proposal_init` is `None`, mirroring the LibDiamond zero-acceptance-time bypass — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` derives the lower-bound for the proposed block timestamp from `previous_proposal_init.as_ref().map_or(0, ...)`. When `previous_proposal_init` is `None` — which is the initial state after every node restart, not only at genesis — the sentinel value `0` is used. Because the timestamp field is `u64`, the guard `init_proposed.timestamp < last_block_timestamp` (i.e., `timestamp < 0`) is structurally impossible to trigger, so **any** timestamp in `[0, now + window]` passes unconditionally. This is the direct sequencer analog of the LibDiamond M-04 finding: a value that should be treated as "not yet set / invalid" is instead treated as a valid lower bound of zero, silently bypassing the intended ordering constraint.

---

### Finding Description

In `is_proposal_init_valid`:

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs  L261-263
let last_block_timestamp =
    proposal_init_validation.previous_proposal_init.as_ref().map_or(0, |info| info.timestamp);
if init_proposed.timestamp < last_block_timestamp {
    return Err(ValidateProposalError::InvalidProposalInit(...));
}
```

`previous_proposal_init` is an `Option<PreviousProposalInitInfo>` that starts as `None` in `SequencerConsensusContext::new()` and is only written in two places:

1. `decision_reached()` — line 1039 of `sequencer_consensus_context.rs`:
   ```rust
   self.previous_proposal_init = Some(PreviousProposalInitInfo::from(&init));
   ```
2. `try_sync()` — lines 1083-1084 of `sequencer_consensus_context.rs`:
   ```rust
   self.previous_proposal_init =
       Some(previous_proposal_init_from_block_header(&sync_block.block_header_without_hash));
   ```

`set_height_and_round` — the entry point called when consensus begins at a new height — does **not** set `previous_proposal_init`. Therefore, after any restart at height H > 0, a validator enters `validate_proposal` with `previous_proposal_init = None`, causing `last_block_timestamp = 0` and making the lower-bound check a no-op.

The identical `map_or(0, ...)` pattern also appears in `try_sync` at line 1064-1065, where the same bypass applies to sync-path timestamp validation.

---

### Impact Explanation

A malicious proposer can broadcast a `ProposalInit` carrying `timestamp = 1` (or any value far below the true previous block timestamp) to validators that have just restarted. Those validators will pass `is_proposal_init_valid` because `1 < 0` is false. If a quorum of validators restart simultaneously (e.g., during a coordinated upgrade), the block can be finalized with a fabricated timestamp. This produces:

- A wrong `BlockTimestamp` committed to storage, affecting every contract that reads `get_block_timestamp` via syscall — wrong execution results.
- Wrong L1 gas price calculations: `get_l1_prices_in_fri_and_wei` uses the proposed timestamp to look up historical oracle prices; a timestamp of 1 (Unix epoch + 1 s) maps to a price point that does not correspond to the actual L1 state at block time, producing incorrect fee/gas accounting.
- A wrong block hash, since the timestamp is a direct input to block hash computation.

These map to the **Critical** impact category: "Wrong state, receipt, event … or revert result from blockifier/syscall/execution logic for accepted input" and "Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."

---

### Likelihood Explanation

The precondition — `previous_proposal_init == None` at a non-genesis height — is reached on every node restart before the first `decision_reached` or successful `try_sync` call. Coordinated restarts (software upgrades, crash recovery) are routine operational events. The malicious proposer only needs to be the elected leader for one round during the restart window. No special privileges are required beyond being a valid consensus participant.

---

### Recommendation

Add an explicit guard that treats `previous_proposal_init == None` as valid **only** at genesis (height 0). For all other heights, either reject the proposal or require that `previous_proposal_init` be populated before validation proceeds:

```rust
// In is_proposal_init_valid:
match &proposal_init_validation.previous_proposal_init {
    None if proposal_init_validation.height == BlockNumber(0) => {
        // Genesis: no lower bound to enforce.
    }
    None => {
        // Non-genesis with no previous block info: cannot validate timestamp ordering.
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            "previous_proposal_init is None at non-genesis height; \
             cannot enforce timestamp ordering".to_string(),
        ));
    }
    Some(prev) => {
        if init_proposed.timestamp < prev.timestamp {
            return Err(...);
        }
    }
}
```

Apply the same fix to the `try_sync` timestamp check in `sequencer_consensus_context.rs` lines 1064-1069.

### Proof of Concept

1. Start a validator node at height H = 5 (after a restart). `previous_proposal_init` is `None`.
2. `set_height_and_round(BlockNumber(5), 0)` is called — `previous_proposal_init` remains `None`.
3. A malicious proposer sends `ProposalInit { height: 5, timestamp: 1, ... }`. The true previous block had `timestamp = 1_700_000_000`.
4. Inside `is_proposal_init_valid`:
   - `last_block_timestamp = None.map_or(0, ...) = 0`
   - Check: `1 < 0` → **false** → no error returned
5. The validator proceeds to `initiate_validation` and votes for the proposal.
6. If a quorum of restarted validators do the same, the block is finalized with `timestamp = 1`, producing wrong execution results for all timestamp-dependent contracts and wrong L1 gas price accounting for the block. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
