### Title
ICS-721 `ValidateBasic` Enforces Bech32 on Sender Address, Breaking `OnRecvPacket` for Non-Cosmos Counterparty Chains — (`x/nft-transfer/types/packet.go`)

### Summary

`NonFungibleTokenPacketData.ValidateBasic()` in `x/nft-transfer/types/packet.go` validates both the `Sender` and `Receiver` fields using `sdk.AccAddressFromBech32`, despite a code comment that explicitly states these fields must **not** be validated this way. This causes `OnRecvPacket` to always return an error acknowledgement whenever the sending chain uses a non-bech32 address format, permanently breaking ICS-721 NFT reception from any such counterparty.

### Finding Description

The comment at the top of `ValidateBasic` reads:

> NOTE: The addresses formats are not validated as the sender and recipient can have different formats defined by their corresponding chains that are not known to IBC.

Yet the implementation immediately contradicts this:

```go
// decode the sender address
if _, err := sdk.AccAddressFromBech32(nftpd.Sender); err != nil {
    return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid sender address")
}
// decode the receiver address
if _, err := sdk.AccAddressFromBech32(nftpd.Receiver); err != nil {
    return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid receiver address")
}
``` [1](#0-0) 

`ValidateBasic` is called unconditionally inside `OnRecvPacket` before any application logic runs:

```go
if err := data.ValidateBasic(); err != nil {
    return err
}
``` [2](#0-1) 

`OnRecvPacket` is the IBC module entry point for incoming NFT packets: [3](#0-2) 

When the source chain uses non-bech32 addresses (e.g., hex Ethereum-style addresses from an EVM chain implementing ICS-721), `sdk.AccAddressFromBech32(nftpd.Sender)` will always return an error. The result is an error acknowledgement written back to the sender — every single incoming NFT packet from such a chain is rejected at validation, before any escrow or minting logic is reached.

### Impact Explanation

The corrupted invariant is **NFT ownership on the destination chain**: NFTs sent from a non-bech32 counterparty chain are never minted to the receiver. The NFT remains locked in escrow on the source chain until the packet times out, at which point the source chain refunds the sender. No funds are permanently lost in the timeout path, but the ICS-721 receive path is completely non-functional for any non-Cosmos counterparty, rendering the `x/nft-transfer` module useless in that configuration.

### Likelihood Explanation

ICS-721 is an open standard. Any chain implementing it — including EVM chains with hex addresses — can open a channel to Cronos POS Chain. The channel handshake succeeds (version negotiation does not touch address format), so the misconfiguration is invisible until the first `OnRecvPacket` call. A user on such a chain who initiates an NFT transfer will have their NFT locked in escrow with no recourse until timeout.

### Recommendation

Remove the `sdk.AccAddressFromBech32` calls from `ValidateBasic`. The ICS-721 spec requires only that `Sender` and `Receiver` are non-empty strings. Address format validation belongs in `processReceivedPacket` (where the receiver **is** known to be a local bech32 address) and in `refundPacketToken` (where the sender **is** known to be a local bech32 address), both of which already perform that check independently. [4](#0-3) [5](#0-4) 

### Proof of Concept

1. A non-Cosmos chain implementing ICS-721 opens a channel to Cronos POS Chain (`nft` port). Channel handshake succeeds normally.
2. A user on that chain calls the ICS-721 transfer message with their hex address (e.g., `0xAbCd...`) as `Sender` and a valid bech32 address on Cronos POS Chain as `Receiver`.
3. The IBC relayer picks up the packet and submits `MsgRecvPacket` to Cronos POS Chain.
4. `IBCModule.OnRecvPacket` unmarshals the packet data and calls `data.ValidateBasic()`.
5. `sdk.AccAddressFromBech32("0xAbCd...")` returns an error; `ValidateBasic` returns `ErrInvalidAddress`.
6. `OnRecvPacket` writes an error acknowledgement. The NFT is never minted. The source chain's escrow holds the NFT until timeout.
7. Every subsequent attempt from this counterparty chain produces the same result — the receive path always reverts. [6](#0-5) [7](#0-6)

// ValidateBasic is used for validating the nft transfer.
// NOTE: The addresses formats are not validated as the sender and recipient can have different
// formats defined by their corresponding chains that are not known to IBC.
func (nftpd NonFungibleTokenPacketData) ValidateBasic() error {
	if strings.TrimSpace(nftpd.ClassId) == "" {
		return newsdkerrors.Wrap(ErrInvalidClassID, "classId cannot be blank")
	}

	if len(nftpd.TokenIds) == 0 {
		return newsdkerrors.Wrap(ErrInvalidTokenID, "tokenId cannot be blank")
	}

	if len(nftpd.TokenIds) != len(nftpd.TokenUris) {
		return newsdkerrors.Wrap(ErrInvalidPacket, "tokenIds and tokenUris lengths do not match")
	}

	if strings.TrimSpace(nftpd.Sender) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "sender address cannot be blank")
	}

	// decode the sender address
	if _, err := sdk.AccAddressFromBech32(nftpd.Sender); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid sender address")
	}

	if strings.TrimSpace(nftpd.Receiver) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "receiver address cannot be blank")
	}

	// decode the receiver address
	if _, err := sdk.AccAddressFromBech32(nftpd.Receiver); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid receiver address")
	}
	return nil

func (im IBCModule) OnRecvPacket(
	ctx sdk.Context,
	channelVersion string,
	packet channeltypes.Packet,
	relayer sdk.AccAddress,
) ibcexported.Acknowledgement {
	ack := channeltypes.NewResultAcknowledgement([]byte{byte(1)})

	var data types.NonFungibleTokenPacketData
	if err := types.ModuleCdc.UnmarshalJSON(packet.GetData(), &data); err != nil {
		ack = channeltypes.NewErrorAcknowledgement(fmt.Errorf("cannot unmarshal ICS-721 nft-transfer packet data"))
	}

	// only attempt the application logic if the packet data
	// was successfully decoded
	if ack.Success() {
		if err := im.keeper.OnRecvPacket(ctx, channelVersion, packet, data); err != nil {
			ack = types.NewErrorAcknowledgement(err)
		}
	}
	sdkCtx := sdk.UnwrapSDKContext(ctx)
	sdkCtx.EventManager().EmitEvent(
		sdk.NewEvent(
			types.EventTypePacket,
			sdk.NewAttribute(sdk.AttributeKeyModule, types.ModuleName),
			sdk.NewAttribute(sdk.AttributeKeySender, data.Sender),
			sdk.NewAttribute(types.AttributeKeyReceiver, data.Receiver),
			sdk.NewAttribute(types.AttributeKeyClassID, data.ClassId),
			sdk.NewAttribute(types.AttributeKeyTokenIDs, strings.Join(data.TokenIds, ",")),
			sdk.NewAttribute(types.AttributeKeyAckSuccess, fmt.Sprintf("%t", ack.Success())),
		),
	)

	// NOTE: acknowledgement will be written synchronously during IBC handler execution.
	return ack
}
