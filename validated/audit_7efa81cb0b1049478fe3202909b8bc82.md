### Title
Token Transfer Revert to Blacklisted Recipient Permanently Locks User Collateral — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.handleWithdrawTransfer` performs a direct `safeTransfer` to the user-supplied `sendTo` address with no fallback. If the recipient is USDC-blacklisted, the transfer reverts, propagating up through the entire call stack. For `WithdrawCollateral` (V1), `sendTo` is always derived from the sender's own address, so a blacklisted depositor can never recover their collateral.

---

### Finding Description

`BaseWithdrawPool.handleWithdrawTransfer` unconditionally calls `token.safeTransfer(to, uint256(amount))`: [1](#0-0) 

`ERC20Helper.safeTransfer` wraps the call with a hard `require(success && ...)`: [2](#0-1) 

Any failure — including a USDC blacklist rejection — causes an unconditional revert.

For `WithdrawCollateral` (V1), `sendTo` is always resolved to the sender's own Ethereum address inside `Clearinghouse.withdrawCollateral`: [3](#0-2) 

This means a user whose address is USDC-blacklisted after depositing has no alternative destination — every V1 withdrawal attempt will revert.

The full call chain is:

```
Endpoint.submitTransactionsChecked
  → processTransaction (no try/catch)
    → processTransactionImpl [WithdrawCollateral]
      → clearinghouse.withdrawCollateral
        → Clearinghouse.handleWithdrawTransfer
          → token.safeTransfer(withdrawPool, amount)   // moves funds to pool
          → BaseWithdrawPool.submitWithdrawal
            → BaseWithdrawPool.handleWithdrawTransfer
              → token.safeTransfer(blacklistedAddr, amount)  // REVERTS
``` [4](#0-3) [5](#0-4) 

`submitTransactionsChecked` has no try/catch around `processTransaction`: [6](#0-5) 

The revert propagates through the entire batch, undoing all state changes including the `token.safeTransfer(withdrawPool, amount)` that already moved funds into the pool.

---

### Impact Explanation

A user whose address is USDC-blacklisted after depositing collateral has their funds permanently locked:

- The V1 withdrawal path always sends to the sender's own address; there is no override.
- Every sequencer batch containing this withdrawal reverts entirely.
- The sequencer must permanently exclude the transaction, leaving the user's `SpotEngine` balance credited but unreachable.
- No pull-pattern or claim mechanism exists to recover the funds.

The corrupted state is: `SpotEngine` balance remains positive for the user, but the corresponding on-chain USDC can never be transferred out. [7](#0-6) 

---

### Likelihood Explanation

USDC is the quote token for all collateral in Nado. Circle actively maintains a USDC blacklist for regulatory compliance and has blacklisted hundreds of addresses. A user depositing USDC collateral before their address is blacklisted creates an irrecoverable state. No attacker action is required — the trigger is an external regulatory event on a token the protocol explicitly supports.

---

### Recommendation

Adopt the pull pattern for withdrawals. Instead of pushing tokens directly to `sendTo` inside `handleWithdrawTransfer`, record the pending amount in a `mapping(address => mapping(address => uint256)) pendingWithdrawals` and expose a `claimWithdrawal(address token)` function. This decouples the critical accounting step from the token transfer, so a blacklisted recipient cannot block sequencer batches or lock their own funds permanently.

As a minimal mitigation, wrap the transfer in a try/catch and store failed transfers for later retry:

```solidity
function handleWithdrawTransfer(IERC20Base token, address to, uint128 amount) internal virtual {
    try token.safeTransfer(to, uint256(amount)) {}
    catch {
        pendingWithdrawals[address(token)][to] += amount;
    }
}
```

---

### Proof of Concept

1. User deposits USDC collateral via `Endpoint.depositCollateral`. Their `SpotEngine` balance is credited.
2. Circle blacklists the user's address (e.g., for regulatory reasons).
3. User submits a signed `WithdrawCollateral` transaction.
4. Sequencer includes it in a batch via `submitTransactionsChecked`.
5. `Clearinghouse.withdrawCollateral` resolves `sendTo = address(uint160(bytes20(sender)))` — the blacklisted address.
6. `Clearinghouse.handleWithdrawTransfer` first transfers tokens to `withdrawPool` (succeeds), then calls `BaseWithdrawPool.submitWithdrawal`.
7. `BaseWithdrawPool.handleWithdrawTransfer` calls `token.safeTransfer(blacklistedAddress, amount)`.
8. USDC reverts the transfer. `ERC20Helper.safeTransfer` propagates the revert with `ERR_TRANSFER_FAILED`.
9. The entire batch reverts. The user's `SpotEngine` balance remains positive, but the funds can never be transferred out. No recovery path exists. [1](#0-0) [8](#0-7)
