### Title
Coinbase Merkle Proof Check Bypassed via Direct Call to Deprecated `verify_transaction_inclusion` — (`File: contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle proof forgery vulnerability by adding a mandatory coinbase merkle proof check. However, the original `verify_transaction_inclusion` (v1) remains a fully public, callable NEAR contract function. Any unprivileged caller can invoke v1 directly, bypassing the coinbase check entirely and forging a transaction inclusion proof.

### Finding Description

`verify_transaction_inclusion_v2` adds a coinbase merkle proof validation step before delegating to v1: [1](#0-0) 

The coinbase check is placed only in the v2 wrapper. The underlying v1 function is still `pub` and decorated with `#[pause]` (callable by anyone when the contract is not paused): [2](#0-1) 

The `#[deprecated]` attribute in Rust is a **compile-time lint only**. It emits a compiler warning to Rust callers but does not restrict runtime invocation. Any NEAR account can call `verify_transaction_inclusion` directly via a cross-contract call or a direct transaction, completely skipping the coinbase proof check added in v2.

The contract's own documentation acknowledges the danger of v1: [3](#0-2) 

The underlying `compute_root_from_merkle_proof` in `merkle-tools` accepts any 32-byte value as `transaction_hash` with no validation that it is a real transaction hash: [4](#0-3) 

### Impact Explanation

The 64-byte transaction Merkle proof forgery attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions) allows an attacker to craft a 64-byte value that is a valid internal Merkle tree node and pass it as `tx_id` to `verify_transaction_inclusion`. Because the Merkle tree is built by concatenating two 32-byte hashes and double-SHA256-hashing them, an internal node hash is structurally indistinguishable from a leaf (transaction) hash at the API level. The function returns `true` for a Bitcoin transaction that was never included in any block.

This corrupts the core proof result of the light client: the canonical guarantee that `verify_transaction_inclusion(...) == true` means the transaction was included in a confirmed Bitcoin block is broken. Any NEAR contract consuming this result (e.g., a bridge, a payment verifier, a settlement contract) can be deceived into releasing funds or accepting state transitions based on a forged Bitcoin transaction.

### Likelihood Explanation

- No privileged role is required. Any NEAR account can call `verify_transaction_inclusion` directly.
- The attack is well-documented and the exploit technique is publicly known.
- The contract itself references the vulnerability in the v1 deprecation notice, confirming the developers are aware of the risk but left the entry point open.
- The attacker only needs a real Bitcoin block header already accepted by the light client and knowledge of its Merkle tree structure to craft a valid forgery.

### Recommendation

Move the coinbase merkle proof check into `verify_transaction_inclusion` itself (the v1 function), or make v1 `pub(crate)` / private so it is no longer reachable as a NEAR contract entry point. The structural fix mirrors the original report's recommendation exactly: the security check must be placed on the function that is actually exposed as the public entry point, not only on a wrapper that callers can bypass.

### Proof of Concept

1. Attacker identifies a real Bitcoin block already accepted by the light client (e.g., block at height H with known Merkle root R).
2. Attacker finds an internal Merkle node `N` in that block's transaction tree such that `compute_root_from_merkle_proof(N, index, proof) == R`.
3. Attacker calls `verify_transaction_inclusion` directly on the NEAR contract with `tx_id = N`, `tx_block_blockhash = <block hash at H>`, `tx_index = index`, `merkle_proof = proof`, `confirmations = 1`.
4. The function computes `compute_root_from_merkle_proof(N, index, proof)` at line 318–322, which equals `R` (the real Merkle root stored in the block header), and returns `true`.
5. `N` is not a real transaction hash — it is an internal node — but the contract returns `true`, forging proof of inclusion for a non-existent transaction. [5](#0-4)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
