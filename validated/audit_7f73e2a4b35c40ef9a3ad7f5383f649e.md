### Title
`paid_fee_on_l1` Hardcoded to `Fee(1)` at Consensus-to-Executable L1Handler Conversion Boundary Bypasses Fee Adequacy Check — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

The `ConsensusTransaction::L1Handler` variant carries a `transaction::L1HandlerTransaction` that has **no `paid_fee_on_l1` field**. When `convert_consensus_l1_handler_to_internal_l1_handler` reconstructs the executable form, it hardcodes `Fee(1)` for that field. The only fee guard in `execute_raw` is `if paid_fee == Fee(0)`, so `Fee(1)` unconditionally passes. Every L1Handler transaction that enters the sequencer through the consensus path is therefore executed with a fabricated fee value, not the fee actually paid on L1.

---

### Finding Description

**Conversion boundary — hardcoded sentinel value**

`ConsensusTransaction` is defined as:

```rust
pub enum ConsensusTransaction {
    RpcTransaction(RpcTransaction),
    L1Handler(transaction::L1HandlerTransaction),   // no paid_fee_on_l1
}
``` [1](#0-0) 

The executable form carries the real fee:

```rust
pub struct L1HandlerTransaction {
    pub tx: crate::transaction::L1HandlerTransaction,
    pub tx_hash: TransactionHash,
    pub paid_fee_on_l1: Fee,          // actual fee from L1 event
}
``` [2](#0-1) 

The converter that reconstructs the executable object from the consensus object hardcodes the field:

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
``` [3](#0-2) 

**Fee guard that the hardcoded value trivially satisfies**

The only check in `execute_raw` is:

```rust
let paid_fee = self.paid_fee_on_l1;
// For now, assert only that any amount of fee was paid.
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee { paid_fee, actual_fee: receipt.fee },
    )));
}
``` [4](#0-3) 

`Fee(1) != Fee(0)` is always true, so the guard is permanently bypassed for every L1Handler that arrives through the consensus path.

**Same pattern in the RPC simulation path**

`stored_txn_to_executable_txn`, used by `simulate_transactions` and `trace_block_transactions`, applies the identical workaround:

```rust
starknet_api::transaction::Transaction::L1Handler(value) => {
    // todo(yair): This is a temporary solution until we have a better way to get the l1 fee.
    let paid_fee_on_l1 = Fee(1);
    Ok(ExecutableTransactionInput::L1Handler(value, paid_fee_on_l1, false))
}
``` [5](#0-4) 

---

### Impact Explanation

**Conversion binds the wrong executable payload.** Every validator node that reconstructs an L1Handler from a consensus message executes it with `paid_fee_on_l1 = Fee(1)`, not the value that was actually locked on L1. The fee adequacy invariant — that the sequencer must verify the L1-locked fee covers the actual execution cost — is silently dropped at this boundary.

If the actual `paid_fee_on_l1` from the L1 event is `Fee(0)` (e.g., due to a bug or future change in the L1 contract), the proposer would reject the transaction (`Fee(0) == Fee(0)` triggers `InsufficientFee`), but every validator would accept it (`Fee(1) != Fee(0)` passes), producing a **proposer/validator state divergence** that breaks consensus.

For the RPC path, `simulate_transactions` and `trace_block_transactions` return authoritative-looking results computed against `Fee(1)` rather than the real on-chain fee, making fee estimation for L1Handler messages unreliable.

---

### Likelihood Explanation

The trigger is any L1→L2 message processed through the consensus path, which is the normal production flow. The L1 Starknet core contract currently enforces a non-zero fee, so the divergence scenario requires `paid_fee_on_l1 = 0` to manifest as a consensus split. However, the wrong value is bound on every single L1Handler execution through consensus today, and the acknowledged TODO confirms this is a known incomplete state.

---

### Recommendation

Add `paid_fee_on_l1` to `ConsensusTransaction::L1Handler` (or to the protobuf `L1HandlerV0` message) so the actual fee survives the consensus wire format. Until then, the conversion should at minimum read the fee from the L1 provider's stored transaction rather than substituting a sentinel. The RPC path should similarly retrieve the stored `paid_fee_on_l1` from the transaction receipt table instead of hardcoding `Fee(1)`.

---

### Proof of Concept

1. The L1 scraper receives a `MessageSent` event with `paid_fee = 0` (or any value that would fail the `paid_fee == Fee(0)` check on the proposer).
2. The proposer creates `executable_transaction::L1HandlerTransaction { paid_fee_on_l1: Fee(0), … }` and calls `execute_raw`; the guard fires and the transaction is **rejected**.
3. The same transaction is broadcast to validators as `ConsensusTransaction::L1Handler(transaction::L1HandlerTransaction { … })` — `paid_fee_on_l1` is absent from this type.
4. Each validator calls `convert_consensus_l1_handler_to_internal_l1_handler`, which produces `paid_fee_on_l1: Fee(1)`.
5. The validator calls `execute_raw`; `Fee(1) != Fee(0)` passes; the transaction is **accepted and committed**.
6. Proposer and validators hold divergent states; consensus halts or the block is invalid. [6](#0-5) [7](#0-6)
