Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, allowing any caller to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is the `msg.sender` of `MetricOmmPool.swap`. When a user routes through the public `MetricOmmSimpleRouter`, `sender` becomes the router's address. If the pool admin allowlists the router to support router-mediated swaps, every user — including those explicitly excluded — can bypass the individual allowlist by calling the router instead of the pool directly.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the hook:**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at line 231: [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards `sender` unchanged:** [2](#0-1) 

**Step 3 — `SwapAllowlistExtension.beforeSwap` checks `sender` (the immediate pool caller) against the allowlist:**

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

**Step 4 — `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router the `msg.sender`:** [4](#0-3) 

**Exploit path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists only specific addresses (e.g., KYC'd users).
2. To support router-mediated swaps for those users, the admin calls `setAllowedToSwap(pool, router, true)`.
3. Any non-allowlisted user calls `router.exactInputSingle(...)`. The router calls `pool.swap(...)`, so `msg.sender` inside the pool is the router.
4. `_beforeSwap` receives `sender = router`, which is allowlisted → check passes → swap executes.
5. The non-allowlisted user's swap succeeds, bypassing the intended access control entirely.

**Existing guards are insufficient:** The `allowAllSwappers` flag is a separate escape hatch. The per-address `allowedSwapper` mapping is the precise control that fails here — it is keyed on the intermediate router address, not the originating user.

## Impact Explanation
The swap allowlist is a core pool access-control mechanism. Its bypass allows any unprivileged address to execute swaps on pools that are explicitly restricted (e.g., permissioned/KYC pools, institutional pools). This breaks the intended pool functionality and constitutes an admin-boundary break via an unprivileged public path. Depending on pool design, this can expose restricted liquidity to unauthorized counterparties, causing direct economic harm to LPs who deposited under the assumption of a restricted swap set.

## Likelihood Explanation
The condition is reachable by any unprivileged user with no special setup beyond knowing the router address. It requires only that the pool admin has allowlisted the router (a natural and expected configuration to support router-mediated swaps). The attack is repeatable, requires no privileged access, and is trivially executable on every affected pool.

## Recommendation
The extension should check the originating user, not the intermediate router. Two approaches:
1. **Pass `recipient` or a separate `originator` field** through the hook interface so the extension can gate on the true economic actor.
2. **Check `recipient` instead of `sender`** in `beforeSwap` if the pool's design intent is that the recipient is the beneficiary being gated.
3. **Alternatively**, require the router to forward the original `msg.sender` as part of `extensionData`, and have the extension decode and verify it — though this requires router cooperation and is weaker.

The cleanest fix is to add an `originator` (the outermost `msg.sender`) to the hook call path so extensions can gate on the true initiating address.

## Proof of Concept

```solidity
// Foundry test sketch
function test_bypassSwapAllowlistViaRouter() public {
    // Admin allowlists only `allowedUser` and the router
    swapExtension.setAllowedToSwap(address(pool), allowedUser, true);
    swapExtension.setAllowedToSwap(address(pool), address(router), true);

    // bannedUser is NOT allowlisted
    // Direct pool call reverts
    vm.prank(bannedUser);
    vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
    pool.swap(bannedUser, true, 1000, 0, "", "");

    // Router call succeeds — bypass achieved
    vm.prank(bannedUser);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        tokenIn: token0,
        ...
        amountIn: 1000,
        ...
    }));
    // bannedUser successfully swapped despite being excluded from the allowlist
}
```
