Audit Report

## Title
Router-Mediated Swaps Pass Router Address as `sender` to `SwapAllowlistExtension::beforeSwap`, Breaking Per-User Allowlist Gating ŌĆö (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension::beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the original user. This makes per-user allowlisting incoherent for any router-mediated swap, producing two mutually exclusive failure modes: allowlisted users cannot use the router, or allowlisting the router bypasses all per-user gating.

## Finding Description

**Step 1 ŌĆö Pool passes `msg.sender` as `sender` to the hook.**

`MetricOmmPool::swap` calls `_beforeSwap(msg.sender, ...)` at line 230ŌĆō240: [1](#0-0) 

`ExtensionCalling::_beforeSwap` forwards that value unchanged as the first argument to every configured extension: [2](#0-1) 

**Step 2 ŌĆö The hook checks `allowedSwapper[pool][sender]`.**

`SwapAllowlistExtension::beforeSwap` uses `msg.sender` (the pool) as the pool key and the passed `sender` argument as the swapper identity: [3](#0-2) 

**Step 3 ŌĆö The router is the direct caller of `pool.swap()`.**

`MetricOmmSimpleRouter::exactInputSingle` calls `pool.swap(...)` directly. At that call site, `msg.sender` inside the pool is the **router**, not the end user: [4](#0-3) 

The router stores the original `msg.sender` only in transient callback context for payment purposes ŌĆö it is never forwarded to the pool or the hook: [5](#0-4) 

The same pattern applies to `exactOutputSingle` (line 135ŌĆō137) and `exactInput` (line 103ŌĆō112), and for multi-hop `exactOutput` the recursive callback calls `pool.swap()` from within the callback, making `msg.sender` the prior pool address. [6](#0-5) [7](#0-6) 

## Impact Explanation

Two mutually exclusive failure modes exist for any pool configured with `SwapAllowlistExtension`:

1. **Allowlisted users cannot use the router.** If the pool admin allowlists specific user addresses, every router-mediated swap by those users reverts with `NotAllowedToSwap` because the hook sees the router's address instead of the user's.

2. **Allowlisting the router opens the pool to all users.** If the pool admin allowlists the router address to restore router functionality, any unprivileged user can call `exactInputSingle` and swap in a pool intended to be restricted. The per-user gating is entirely nullified.

This constitutes broken core pool functionality: the `SwapAllowlistExtension` is a first-party production extension whose primary purpose ŌĆö per-user swap gating ŌĆö is rendered inoperable for the protocol's primary swap interface. This meets the "Broken core pool functionality causing loss of funds or unusable swap flows" impact criterion. [8](#0-7) 

## Likelihood Explanation

No special attacker capability is required. Any normal user calling `exactInputSingle` on a pool configured with `SwapAllowlistExtension` triggers the mismatch. The router is the primary public swap interface. Any pool deploying this extension and expecting users to interact via the router will immediately encounter one of the two failure modes. The condition is always present and requires no setup beyond normal protocol usage. [9](#0-8) 

## Recommendation

Two concrete options:

1. **Pass the original initiator as a dedicated hook argument.** Add an `initiator` field alongside `sender` in the `beforeSwap` signature. For direct calls `initiator == sender`; for router calls the router encodes the original `msg.sender` into `extensionData` and the pool/hook reads it.

2. **Extension-level router awareness.** `SwapAllowlistExtension::beforeSwap` checks: if `sender` is a known trusted router, decode the real initiator from `extensionData` and check that address instead. This requires the extension to maintain a trusted-router registry. [3](#0-2) 

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is allowlisted
  allowedSwapper[pool][router] = false // router is not

Broken functionality path:
  alice calls router.exactInputSingle({pool: pool, ...})
  ŌåÆ router calls pool.swap(recipient, ...)   // msg.sender inside pool = router
  ŌåÆ pool calls _beforeSwap(msg.sender=router, ...)
  ŌåÆ SwapAllowlistExtension.beforeSwap(sender=router, ...)
  ŌåÆ allowedSwapper[pool][router] == false ŌåÆ revert NotAllowedToSwap
  Alice is allowlisted but cannot use the router.

Bypass path (if admin allowlists router to fix above):
  allowedSwapper[pool][router] = true

  mallory (not allowlisted) calls router.exactInputSingle({pool: pool, ...})
  ŌåÆ router calls pool.swap(...)   // msg.sender = router
  ŌåÆ allowedSwapper[pool][router] == true ŌåÆ swap succeeds
  Mallory bypasses the per-user allowlist entirely.
```
