### Title
Equihash Solution Validity Return Value Ignored, Allowing Invalid PoW Block Acceptance - (File: `contract/src/zcash.rs`)

### Summary
In the Zcash PoW verification path, `equihash::is_valid_solution` returns `Result<bool, Error>` where `Ok(true)` means valid and `Ok(false)` means invalid. The contract only handles the `Err` variant via `.unwrap_or_else`, silently discarding the `bool` result. An unprivileged caller can submit a Zcash block header with a structurally well-formed but cryptographically invalid Equihash solution; the contract will accept it, corrupting the canonical chain.

### Finding Description
In `contract/src/zcash.rs`, the `check_pow` function validates the Equihash proof-of-work solution as follows:

```rust
equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
```

`equihash::is_valid_solution` returns `Result<bool, Error>`:
- `Ok(true)` — solution is valid
- `Ok(false)` — solution is **invalid** (wrong XOR/collision structure)
- `Err(e)` — parameter error (e.g., bad `n`/`k`)

`.unwrap_or_else(|e| ...)` only intercepts the `Err` arm. The `Ok(bool)` value is **completely discarded**. When the solution is invalid and the library returns `Ok(false)`, execution continues without panicking, and the block is stored as a valid mainchain or fork header.

This is structurally identical to the ERC20WrapperBundler bug: a function returns a boolean success indicator, the caller handles only the error/exception path, and silently ignores the `false`/invalid result.

### Impact Explanation
An attacker submitting a Zcash block header through `submit_blocks` with a fabricated or zeroed `solution` field (but otherwise valid `bits`, `time`, `prev_block_hash`) will pass all other checks (timestamp, difficulty bits) and have the block accepted into `headers_pool`. If the fabricated block's computed `chain_work` exceeds the current tip, `reorg_chain` is triggered, replacing the canonical chain with attacker-controlled headers. This breaks the core invariant of the light client: that every stored mainchain header has valid proof-of-work. Downstream consumers calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` against these headers will receive fraudulent confirmations.

### Likelihood Explanation
The `submit_blocks` entry point is reachable by any NEAR account (subject to the `trusted_relayer` gate, but the relayer role is grantable and the bypass roles exist). An attacker who can submit blocks — including a malicious relayer or anyone who obtains the role — can trivially craft a header with an invalid Equihash solution. The `solution` field is attacker-controlled input with no structural pre-filter before `check_pow` is called.

### Recommendation
Replace the discarded `.unwrap_or_else` pattern with an explicit check of the boolean result:

```rust
let valid = equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
require!(valid, "Invalid Equihash solution");
```

### Proof of Concept
1. Construct a Zcash `Header` with valid `bits`, `time`, `prev_block_hash`, and `version >= 4`, but with `solution` set to an all-zero or random byte vector that does not satisfy the Equihash(200,9) constraints.
2. Call `submit_blocks` with this header.
3. `check_pow` calls `equihash::is_valid_solution(200, 9, &input, &nonce, &solution)`.
4. The library returns `Ok(false)` (invalid solution, no parameter error).
5. `.unwrap_or_else` does not trigger (no `Err`); the `false` is dropped.
6. `check_pow` returns normally; `submit_block_header_inner` stores the header.
7. The block is now in `headers_pool` and may become the mainchain tip if its `chain_work` is highest. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    fn submit_block_header_inner(
        &mut self,
        current_header: ExtendedHeader,
        prev_block_header: &ExtendedHeader,
    ) {
        // Main chain submission
        if prev_block_header.block_hash == self.mainchain_tip_blockhash {
            // Probably we should check if it is not in a mainchain?
            // chainwork > highScore
            log!("Block {}: saving to mainchain", current_header.block_hash);
            // Validate chain
            assert_eq!(
                self.mainchain_tip_blockhash,
                current_header.block_header.prev_block_hash
            );

            self.store_block_header(&current_header);
            self.mainchain_tip_blockhash = current_header.block_hash;
        } else {
            log!("Block {}: saving to fork", current_header.block_hash);
            // Fork submission
            let main_chain_tip_header = self
                .headers_pool
                .get(&self.mainchain_tip_blockhash)
                .unwrap_or_else(|| env::panic_str("tip should be in a header pool"));

            let last_main_chain_block_height = main_chain_tip_header.block_height;
            let total_main_chain_chainwork = main_chain_tip_header.chain_work;

            self.store_fork_header(&current_header);

            // Current chainwork is higher than on a current mainchain, let's promote the fork
            if current_header.chain_work > total_main_chain_chainwork {
                log!("Chain reorg");
                self.reorg_chain(current_header, last_main_chain_block_height);
            }
        }
    }
