### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Merkle Proof Forgery Protection — (File: `contract/src/lib.rs`)

---

### Summary

The contract introduced `verify_transaction_inclusion_v2()` specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability. However, the original `verify_transaction_inclusion()` function, despite being marked `#[deprecated]`, remains a fully reachable public NEAR contract method with no runtime guard. Any unprivileged NEAR caller can invoke it directly, bypassing the coinbase Merkle proof check entirely and obtaining a `true` result for a forged transaction inclusion proof.

---

### Finding Description

The contract's own documentation acknowledges the 64-byte transaction Merkle proof forgery risk and directs callers to use `verify_transaction_inclusion_v2()`: [1](#0-0) 

`verify_transaction_inclusion_v2()` enforces the coinbase Merkle proof check before delegating to the deprecated function: [2](#0-1) 

The deprecated function itself has no runtime guard — `#[deprecated]` is a Rust compile-time lint only. It is compiled into the WASM binary and exposed in the NEAR contract ABI identically to any other `pub fn`. The only runtime attribute it carries is `#[pause]`, which is inactive by default: [3](#0-2) 

Critically, `verify_transaction_inclusion()` calls `compute_root_from_merkle_proof()` with no coinbase anchor check: [4](#0-3) 

`compute_root_from_merkle_proof()` in `merkle-tools` is a pure hash-chain computation with no structural validation of the leaf: [5](#0-4) 

The `ProofArgs` struct accepted by the deprecated function has no field for a coinbase proof and no mechanism to enforce one: [6](#0-5) 

---

### Impact Explanation

The 64-byte transaction attack (https://www.bitmex.com/blog/64-Byte-Transactions) allows an attacker to craft a 64-byte blob that is simultaneously a valid serialized internal Merkle tree node and a plausible "transaction ID." By supplying this blob as `tx_id` with a crafted `merkle_proof`, an attacker can make `compute_root_from_merkle_proof()` return the genuine Merkle root of a real Bitcoin block, causing `verify_transaction_inclusion()` to return `true` for a transaction that does not exist in that block.

Any downstream NEAR contract or off-chain application that calls `verify_transaction_inclusion()` and acts on a `true` result — e.g., releasing funds, minting tokens, or updating state contingent on Bitcoin transaction confirmation — can be deceived. The security fix in `verify_transaction_inclusion_v2()` is rendered optional rather than mandatory, because the unprotected entry point remains open.

---

### Likelihood Explanation

The deprecated function is a named public method in the deployed WASM ABI. Any NEAR account can call it without any role, stake, or permission. The function name is visible in the contract source and in any ABI introspection tool. The 64-byte transaction forgery technique is publicly documented and has known tooling. Likelihood is **Medium**: the attack requires knowledge of a real block's Merkle root and construction of a valid-looking 64-byte preimage, but no privileged access whatsoever.

---

### Recommendation

Remove `verify_transaction_inclusion()` from the public ABI entirely, or add a `panic!` / `env::panic_str` at its entry point to make it unconditionally revert when called directly. The `#[deprecated]` Rust attribute provides zero on-chain enforcement. If backward compatibility is required, replicate the coinbase Merkle proof check from `verify_transaction_inclusion_v2()` inside `verify_transaction_inclusion()` before the proof computation.

---

### Proof of Concept

1. Attacker identifies a real Bitcoin block `B` with known Merkle root `R` and at least one real transaction `T_real` at index `i`.
2. Attacker constructs a 64-byte blob `T_fake` such that `double_sha256(T_fake)` equals an internal Merkle tree node `N` that, combined with a crafted sibling path, produces `R`.
3. Attacker calls `verify_transaction_inclusion()` directly on the NEAR contract with:
   - `tx_id = T_fake`
   - `tx_block_blockhash = hash(B)` (a real block already in the contract's `headers_pool`)
   - `tx_index` and `merkle_proof` chosen so that `compute_root_from_merkle_proof(T_fake, tx_index, merkle_proof) == R`
   - `confirmations = 1`
4. The contract performs no coinbase anchor check, computes the Merkle root, finds it matches `header.block_header.merkle_root`, and returns `true`.
5. Any consumer of this result treats the non-existent transaction as confirmed on Bitcoin.

The `verify_transaction_inclusion_v2()` path would have rejected this at step 3 by requiring a valid coinbase proof anchoring the tree, but the deprecated entry point has no such gate. [7](#0-6) [5](#0-4)

    /// Verifies that a transaction is included in a block at a given block height
    ///
    /// # Deprecated
    /// Use [`verify_transaction_inclusion_v2`] instead, which includes coinbase merkle proof validation
    /// to mitigate the 64-byte transaction Merkle proof forgery vulnerability:
    /// https://www.bitmex.com/blog/64-Byte-Transactions
    ///
    /// @param `tx_id` transaction identifier
    /// @param `tx_block_blockhash` block hash at which transacton is supposedly included
    /// @param `tx_index` index of transaction in the block's tx merkle tree
    /// @param `merkle_proof` merkle tree path (concatenated LE sha256 hashes) (does not contain initial `transaction_hash` and `merkle_root`)
    /// @param confirmations how many confirmed blocks we want to have before the transaction is valid
    /// @return True if `tx_id` is at the claimed position in the block at the given blockhash, False otherwise
    ///
    /// # Warning
    /// This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash.
    /// We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification.
    ///
    /// # Panics
    /// Multiple cases
    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
