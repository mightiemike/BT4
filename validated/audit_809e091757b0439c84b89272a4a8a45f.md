### Title
`ERC20Helper.safeTransfer` and `safeTransferFrom` Do Not Check Token Contract Existence, Enabling Phantom Collateral Deposit or Silent Withdrawal Failure — (File: `core/contracts/libraries/ERC20Helper.sol`)

---

### Summary

`ERC20Helper.safeTransfer` and `ERC20Helper.safeTransferFrom` perform low-level `call` operations on token addresses without first verifying that a contract exists at that address. Per EVM specification, a low-level `call` to a non-existent (e.g., self-destructed) address returns `success = true` with empty return data. This passes the `require(success && (data.length == 0 || abi.decode(data, (bool))))` guard silently, completing a "transfer" that moved no tokens. The same pattern exists independently in `DirectDepositV1.safeTransfer`.

---

### Finding Description

`ERC20Helper` is the protocol's sole token transfer library, adapted from Uniswap v3's `TransferHelper`: [1](#0-0) [2](#0-1) 

Neither function checks `address(self).code.length > 0` before the call. The `require` on lines 17–20 and 38–41 only validates the boolean return value — which is trivially satisfied when calling a non-existent contract (EVM returns `success=true`, `data=""`, so `data.length == 0` is true).

`BaseWithdrawPool` imports and uses this library via `using ERC20Helper for IERC20Base`: [3](#0-2) 

`EndpointStorage.handleDepositTransfer` calls both `safeTransferFrom` and `safeTransferTo` (which delegates to `safeTransfer`) in the deposit path: [4](#0-3) 

`Clearinghouse.handleWithdrawTransfer` calls `token.safeTransfer` in the withdrawal path: [5](#0-4) 

`BaseWithdrawPool.handleWithdrawTransfer` calls `token.safeTransfer` when paying out to the end recipient: [6](#0-5) 

The same unchecked low-level call pattern also exists in `DirectDepositV1.safeTransfer`: [7](#0-6) 

---

### Impact Explanation

Two distinct asset-corrupting outcomes are reachable:

**Path A — Phantom Collateral (deposit of destroyed token):**
If a token registered as a spot product is self-destructed, any caller invoking `depositCollateral` or `depositCollateralWithReferral` on `Endpoint` will have `ERC20Helper.safeTransferFrom` silently succeed (no actual token movement). The slow-mode transaction is queued and later processed by the sequencer, causing `clearinghouse.depositCollateral` to credit the subaccount with a real positive balance: [8](#0-7) 

The attacker now holds phantom collateral in the spot engine. They can use it to satisfy health checks and withdraw real quote tokens (USDC) or other legitimate collateral from the protocol, draining real assets.

**Path B — Silent withdrawal failure (withdrawal of destroyed token):**
If a user holds a legitimate balance of a token that is subsequently self-destructed, their `withdrawCollateral` call will invoke `safeTransfer` which silently succeeds. The spot engine balance is decremented: [9](#0-8) 

The user's on-chain balance is permanently zeroed, but they receive nothing. The real tokens previously deposited remain locked in the clearinghouse or withdraw pool forever.

---

### Likelihood Explanation

Requires a token registered as a Nado spot product to be self-destructible (non-standard). This is a low-frequency precondition, but:
- The protocol is permissionless for depositors and withdrawers once a product is listed.
- Path A is attacker-controlled end-to-end: an attacker can deploy a self-destructible token, get it listed, self-destruct it, then deposit phantom amounts.
- Path B affects innocent users with no ability to protect themselves once the token is destroyed.

Likelihood: **Low-Medium** (requires non-standard token, but the exploit is fully permissionless once the precondition is met).

---

### Recommendation

Add a contract existence check at the top of both `ERC20Helper.safeTransfer` and `ERC20Helper.safeTransferFrom`, and in `DirectDepositV1.safeTransfer`:

```solidity
function safeTransfer(
    IERC20Base self,
    address to,
    uint256 amount
) internal {
    require(address(self).code.length > 0, ERR_TRANSFER_FAILED); // <-- add this
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
```

Apply the same guard to `safeTransferFrom` and to `DirectDepositV1.safeTransfer`.

---

### Proof of Concept

1. Attacker deploys `MaliciousToken` with a `selfdestruct` owner function and gets it listed as spot `productId = N` in Nado.
2. Attacker calls `MaliciousToken.destroy()` — contract is self-destructed; `address(MaliciousToken).code.length == 0`.
3. Attacker calls `Endpoint.depositCollateral(subaccountName, N, 1_000_000e18)`.
4. Inside `handleDepositTransfer`, `ERC20Helper.safeTransferFrom(token, attacker, 1_000_000e18)` is called. The low-level call to the destroyed address returns `(true, "")`. `data.length == 0` → `require` passes. No tokens move.
5. A `DepositCollateral` slow-mode transaction is queued for the attacker's subaccount with `amount = 1_000_000e18`.
6. After the 3-day delay, the sequencer executes it: `clearinghouse.depositCollateral` credits the attacker's subaccount with `1_000_000e18` of phantom `MaliciousToken` balance.
7. Attacker calls `withdrawCollateral` for the USDC quote product, using the phantom balance to satisfy the health check (`getHealth >= 0`).
8. Real USDC is transferred out of the clearinghouse to the attacker. Protocol is drained. [10](#0-9) [4](#0-3) [8](#0-7)
