Audit Report

## Title
`OracleValueStopLossExtension` Uses Arithmetic Mean Instead of Geometric Mean for Mid-Price, Causing Stop-Loss Guard to Systematically Underprotect the `zeroForOne == false` Direction - (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes `midPriceX64` as the arithmetic mean of `bidPriceX64` and `askPriceX64` at line 218, while `MetricOmmPool.swap` derives the settlement mid price via `SwapMath.midAndSpreadFeeX64FromBidAsk`, which uses the geometric mean (`Math.sqrt(bid * ask)`). Because AM ≥ GM always holds, the stop-loss extension evaluates LP value at a systematically higher mid price than the pool used for actual swap settlement, inflating `metricT1` and causing the stop-loss guard for the `zeroForOne == false` direction to fail to trigger when it should, allowing LP value to drain beyond the configured drawdown floor.

## Finding Description

**Root cause — two different mid-price formulas for the same oracle quotes:**

`MetricOmmPool.swap` captures `bidPriceX64` and `askPriceX64` once and passes them to both the before/after hooks and to `SwapMath.midAndSpreadFeeX64FromBidAsk` for settlement: [1](#0-0) 

`SwapMath.midAndSpreadFeeX64FromBidAsk` uses the **geometric mean**: [2](#0-1) 

`OracleValueStopLossExtension._afterSwapOracleStopLoss` receives the same `bidPriceX64`/`askPriceX64` but computes the **arithmetic mean**: [3](#0-2) 

This `midPriceX64` is then passed to `_metrics`, which computes `metricT1` by multiplying `t0` by `midPriceX64`: [4](#0-3) 

Since AM > GM whenever `bid ≠ ask`, `metricT1` is inflated relative to the pool's actual settlement price. The watermark ratchet records this inflated value, and the breach check `metric < hwm * floorMultiplier / E6` is evaluated against an inflated baseline: [5](#0-4) 

**Exploit flow:**
1. Pool is configured with `OracleValueStopLossExtension` and a `drawdownE6` floor (e.g., 5%).
2. Oracle spread widens (e.g., 50%: `bid = 0.75 * P`, `ask = 1.25 * P`). AM = `P`; GM = `sqrt(0.9375) * P ≈ 0.9683 * P`.
3. Attacker executes `zeroForOne == false` swaps, draining token0. After each swap, `_afterSwapOracleStopLoss` runs with the arithmetic mean (~3.2% higher than GM).
4. Actual LP value at GM has dropped 7.5% below watermark (exceeds 5% floor → should trigger). But `metricT1` computed with AM shows only ~4.3% drop → no breach → stop-loss does not trigger.
5. Attacker continues draining token0 for an additional ~3.2% of LP value beyond the configured floor.

No existing guard caps the spread width: `_getBidAndAskPriceX64` only validates `bid > 0` and `bid < ask`.

## Impact Explanation

Direct loss of LP principal beyond the configured drawdown floor. For a $1M pool with a 50% oracle spread, the stop-loss fails to protect up to ~$31,250 beyond the configured floor. This is a fund-loss path triggered by any public swapper with no privileged access, meeting the Critical/High direct loss threshold under the allowed impact gate.

## Likelihood Explanation

Oracle spreads widen during high-volatility events — precisely when the stop-loss guard is most needed. The `confidenceParam` in `PriceProvider`/`ProtectedPriceProvider` can scale the spread further. Any unprivileged swapper can execute `zeroForOne == false` swaps during a wide-spread window. No special setup or privileged role is required; the path is reachable through the standard public `swap` entrypoint.

## Recommendation

Replace the arithmetic mean in `_afterSwapOracleStopLoss` with the geometric mean to match the pool's settlement formula:

```solidity
// Before (line 218):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After:
uint256 midPriceX64 = Math.sqrt(uint256(bidPriceX64) * uint256(askPriceX64));
```

This ensures the stop-loss evaluates LP value at exactly the same mid price used for swap settlement, eliminating the systematic bias in both directions.

## Proof of Concept

1. Deploy pool with `OracleValueStopLossExtension`, `drawdownE6 = 50_000` (5% floor).
2. Set oracle to `bid = 0.75e18`, `ask = 1.25e18` (50% spread). AM = `1.0e18`; GM = `~0.9683e18`.
3. Execute `zeroForOne == false` swaps as an unprivileged attacker until actual LP value (at GM) drops 7.5% below watermark.
4. Observe: `_afterSwapOracleStopLoss` computes `metricT1` with AM, showing only ~4.3% drop → `breach1 == false` → no revert.
5. Confirm stop-loss only triggers after an additional ~3.2% loss, i.e., at ~8.2% total drawdown instead of the configured 5%.
6. Foundry test: assert `OracleStopLossTriggered` is not emitted at 7.5% drawdown with AM, then assert it is emitted at the same drawdown when the fix (GM) is applied.
