### Title
`RewardsPoolName` Module Account Can Receive Any Token But Only `bondDenom` Is Ever Drained — Non-Bond-Denom Tokens Are Permanently Stuck - (File: `app/app.go`, `x/tieredrewards/keeper/abci.go`)

---

### Summary

The `rewards_pool` module account is explicitly opened to receive any token via `MsgSend` from unprivileged accounts, but the only code that ever drains it (`topUpBaseRewards` in `BeginBlocker`) reads and transfers only `bondDenom` (basecro). Any other denom sent to the pool is permanently stuck with no governance message, admin function, or protocol path to recover it.

---

### Finding Description

In `app/app.go`, the `moduleAccsAllowedToReceiveExternalFunds` map explicitly whitelists `tieredrewardstypes.RewardsPoolName` (`"rewards_pool"`), which causes `BlockedAddrs()` to return `false` for that address. This means the Cosmos SDK bank module will accept a standard `MsgSend` from any account to the rewards pool address. [1](#0-0) 

The `BeginBlocker` for the tieredrewards module calls `topUpBaseRewards`, which queries the pool balance exclusively for `bondDenom` and transfers only that denom to the distribution module: [2](#0-1) 

The entire function is `bondDenom`-scoped. It calls `k.bankKeeper.GetBalance(ctx, poolAddr, bondDenom)` and `SendCoinsFromModuleToModule(..., sdk.NewCoins(sdk.NewCoin(bondDenom, topUpAmount)))`. No code path ever reads or moves any other denom from the pool. [3](#0-2) 

The `RewardsPoolName` module account has `nil` permissions (no Minter, Burner, or Staking), so the module itself cannot burn or programmatically sweep non-bond-denom balances. [4](#0-3) 

Searching all messages in the tieredrewards module (`MsgLockTier`, `MsgCommitDelegationToTier`, `MsgAddToTierPosition`, `MsgTierRedelegate`, `MsgTierUndelegate`, `MsgTriggerExitFromTier`, `MsgClearPosition`, `MsgWithdrawFromTier`, `MsgClaimTierRewards`, `MsgExitTierWithDelegation`, `MsgUpdateParams`, `MsgAddTier`, `MsgUpdateTier`, `MsgDeleteTier`) — none of them withdraw arbitrary denoms from the `RewardsPoolName` module account. [5](#0-4) 

---

### Impact Explanation

**Impact: Medium.** Any non-`basecro` token sent to the `rewards_pool` module account address is permanently irrecoverable. There is no governance proposal type, no admin message, and no protocol-level sweep that touches non-bond-denom balances in this account. The value at risk is bounded by whatever tokens are sent, but the loss is total and irreversible for those tokens.

---

### Likelihood Explanation

**Likelihood: Medium.** The `rewards_pool` address is a well-known module account address (derivable from the module name via SHA-256, as shown in the integration test helpers). The `moduleAccsAllowedToReceiveExternalFunds` whitelist is intentional and documented — it is the designed funding mechanism for the pool. Any user who mistakenly sends an IBC voucher, a governance token, or any non-`basecro` denom to this address (e.g., intending to fund the pool with a different asset, or making a copy-paste error) will permanently lose those funds. The integration tests themselves demonstrate this funding pattern via `MsgSend`. [6](#0-5) 

---

### Recommendation

Add a governance-gated message (e.g., `MsgSweepRewardsPool` with `authority = gov`) that calls `SendCoinsFromModuleToAccount` or `SendCoinsFromModuleToModule` for all balances held by `RewardsPoolName`, not just `bondDenom`. Alternatively, restrict `moduleAccsAllowedToReceiveExternalFunds` for `RewardsPoolName` to only accept `bondDenom` by adding a denom-level check in the ante handler or a custom `MsgSend` restriction.

---

### Proof of Concept

1. Derive the `rewards_pool` module account address (SHA-256 of `"rewards_pool"`, first 20 bytes, bech32-encoded with `cro` prefix — as done in `integration_tests/utils.py`).
2. Submit a `MsgSend` from any account sending, e.g., `1000ibc/XXXX` (an IBC voucher denom) to that address. The transaction succeeds because `BlockedAddrs()` returns `false` for this address.
3. Observe that `BeginBlocker` runs every block but only queries and transfers `bondDenom` from the pool — the IBC voucher balance is never touched.
4. Confirm no message in the tieredrewards module can recover the IBC voucher. The tokens are permanently stuck. [7](#0-6) [8](#0-7)
