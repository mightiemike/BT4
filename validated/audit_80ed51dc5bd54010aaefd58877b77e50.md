### Title
Unchecked `u64` Multiplication Overflow in `op_unknown` Cost Scaling Produces Zero-Cost Execution — (`File: src/more_ops.rs`)

---

### Summary

`op_unknown` in `src/more_ops.rs` computes a base cost from attacker-controlled argument sizes, then scales it by `cost_multiplier + 1` (derived from attacker-controlled opcode bytes) using an unchecked `u64` multiplication. In Rust release builds, this multiplication wraps silently on overflow. A crafted opcode can cause the wrapped product to fall below `u32::MAX`, bypassing the only post-multiplication guard and returning a `Reduction` with cost 0 — making a resource-consuming no-op appear free to the consensus engine.

---

### Finding Description

The vulnerable sequence is at lines 258–266 of `src/more_ops.rs`:

```rust
assert!(cost > 0);

check_cost(cost, max_cost)?;          // ← guards only the pre-scale cost
cost *= cost_multiplier + 1;          // ← unchecked u64 × u64, wraps in release
if cost > u32::MAX as u64 {           // ← overflow guard, bypassed when wrap < u32::MAX
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`cost_multiplier` is decoded from the leading bytes of the opcode atom:

```rust
let cost_multiplier: u64 = match u32_from_u8(&op[0..op.len() - 1]) {
    Some(v) => v as u64,
    ...
};
```

`u32_from_u8` returns at most `u32::MAX = 4 294 967 295`, so `cost_multiplier + 1` reaches `2^32 = 4 294 967 296`.

`cost` is a `u64` (`type Cost = u64` in `src/cost.rs`). `check_cost` only verifies `cost ≤ max_cost` before the multiplication; it does not prevent the product from overflowing `u64`.

For `cost_function = 2` (mul-like, selected by bits 7–6 of the last opcode byte), the base cost is:

```
cost = MUL_BASE_COST(92) + MUL_COST_PER_OP(885)
     + (l0 + l1) × MUL_LINEAR_COST_PER_BYTE(6)
     + (l0 × l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER(128)
```

With two arguments of size `l0 = l1 ≈ 740 682` bytes, the quadratic term drives `cost` to exactly `2^32`. Then:

```
cost × (cost_multiplier + 1)
= 2^32 × 2^32
= 2^64
≡ 0  (mod 2^64)   ← wraps to 0 in u64 release mode
```

`0 > u32::MAX` is `false`, so the guard is bypassed and `Reduction(0, nil)` is returned — the opcode costs **zero**.

---

### Impact Explanation

- **Undercharged execution (release builds):** An attacker submits a CLVM program containing a crafted unknown opcode with ~1.48 MB of argument atoms. The program is accepted with cost 0 instead of the expected ~4.3 billion. The attacker can pack many such programs into a block, consuming full validator CPU time while paying no cost against the block cost limit.
- **Node crash (debug builds):** The same input causes a Rust integer-overflow panic, crashing any node compiled in debug mode.
- **Consensus divergence:** Debug-mode and release-mode nodes disagree on whether the program is valid and what its cost is, which is a consensus-splitting condition in the Chia network.

---

### Likelihood Explanation

The trigger requires:
1. An opcode atom whose leading bytes encode `cost_multiplier = u32::MAX` (`0xFF 0xFF 0xFF 0xFF`) and whose last byte has bits 7–6 = `10` (cost_function 2).
2. Two argument atoms of ~740 KB each (total ~1.48 MB), which is within Chia's block size envelope.
3. `max_cost ≥ 2^32 ≈ 4.3 × 10^9`; the Chia block cost limit is `11 × 10^9`, so this is satisfied whenever the opcode is evaluated early in a block.
4. The opcode must be an unknown opcode accepted in lenient/softfork mode — a normal execution path in Chia.

All four conditions are fully attacker-controlled through the CLVM program bytes and are reachable without any privileged access.

---

### Recommendation

Replace the bare multiplication with an explicit overflow check before the guard:

```rust
// Replace:
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 { ... }

// With:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .filter(|&c| c <= u32::MAX as u64)
    .ok_or(EvalErr::Invalid(o))?;
```

Alternatively, use `saturating_mul` and then apply the existing guard, which is safe because any saturated value exceeds `u32::MAX`.

### Proof of Concept

**Opcode bytes (5 bytes):**
- Bytes 0–3: `0xFF 0xFF 0xFF 0xFF` → `cost_multiplier = u32::MAX = 4 294 967 295`
- Byte 4: `0x80` → bits 7–6 = `10` → `cost_function = 2`; bits 5–0 ignored

**Arguments:** two atoms each of length `L ≈ 740 682` bytes (any content).

**Cost computation inside `op_unknown`:**
```
cost = 977 + 12 × 740682 + (740682²) / 128
     ≈ 977 + 8 888 184 + 4 286 078 135
     ≈ 4 294 967 296  =  2^32
```

**Scaling step (release mode):**
```
cost *= 4 294 967 296   // cost_multiplier + 1 = 2^32
2^32 × 2^32 = 2^64 ≡ 0 (mod 2^64)
```

**Guard check:** `0 > u32::MAX` → `false` → returns `Reduction(0, nil)`.
