### Title
`AllResources` Transaction with Zero L2/L1-Data Gas Silently Reclassified as `L1Gas` During Protobuf Deserialization, Producing a Different Transaction Hash - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf-to-`ValidResourceBounds` conversion in the P2P sync path silently reclassifies a post-0.13.3 `AllResources` transaction as a pre-0.13.3 `L1Gas` transaction whenever both `l2_gas` and `l1_data_gas` are zero. Because `get_tip_resource_bounds_hash` includes `L1_DATA_GAS` in the hash preimage for `AllResources` but omits it for `L1Gas`, the hash recomputed from the deserialized transaction diverges from the hash that was computed and stored when the transaction was originally sequenced. A syncing node that recomputes the hash will reject a valid block, causing a chain split.

### Finding Description

**Root cause — `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`** in `crates/apollo_protobuf/src/converters/transaction.rs`:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← pre-0.13.3 variant
} else {
    ValidResourceBounds::AllResources(...)       // ← post-0.13.3 variant
})
``` [1](#0-0) 

Any transaction whose `l2_gas` and `l1_data_gas` fields are both zero is silently downgraded to `ValidResourceBounds::L1Gas`, regardless of whether it was originally created as an `AllResources` transaction.

**Hash divergence — `get_tip_resource_bounds_hash`** in `crates/starknet_api/src/transaction_hash.rs`:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // L1_DATA_GAS omitted
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // included
    }
});
``` [2](#0-1) 

For `AllResources`: `poseidon(tip, L1_GAS, L2_GAS, L1_DATA_GAS)` — three resource felts.
For `L1Gas`: `poseidon(tip, L1_GAS, L2_GAS)` — two resource felts.

These produce different `tip_resource_bounds_hash` values, which propagate into the full transaction hash via `get_invoke_transaction_v3_hash`. [3](#0-2) 

**Gateway path always uses `AllResources`**: `RpcInvokeTransactionV3` stores `resource_bounds: AllResourceBounds` (never `ValidResourceBounds`), so the hash computed at ingress always uses `AllResources` semantics — including `L1_DATA_GAS = 0` in the preimage. [4](#0-3) 

**Sync path uses `ValidResourceBounds`**: The canonical `InvokeTransactionV3` stores `resource_bounds: ValidResourceBounds`. When a syncing node deserializes this transaction from protobuf, the conversion above reclassifies it as `L1Gas`, and the recomputed hash omits `L1_DATA_GAS`. [5](#0-4) 

The two deserialization paths for `AllResourceBounds` (RPC/mempool) vs `ValidResourceBounds` (sync) are inconsistent: the RPC path (`rpc_transaction.rs`) requires `l1_data_gas` to be present and always produces `AllResources`, while the sync path (`transaction.rs`) silently defaults it to zero and may produce `L1Gas`. [6](#0-5) 

### Impact Explanation

A syncing node that calls `validate_transaction_hash` on a block containing such a transaction will compute a hash that does not match the stored hash, causing the block to be rejected. This diverges the syncing node's view of the chain from the sequencer's view — a chain split. The impact maps to **Critical: Wrong state from blockifier/execution logic for accepted input** (the block is valid but rejected) and **High: RPC/sync returns an authoritative-looking wrong value** (the syncing node's chain tip is wrong).

### Likelihood Explanation

The trigger requires a post-0.13.3 `AllResources` invoke transaction with `l2_gas.max_amount = 0`, `l2_gas.max_price_per_unit = 0`, `l1_data_gas.max_amount = 0`, `l1_data_gas.max_price_per_unit = 0`, and non-zero `l1_gas`. Whether the gateway's pre-validation (`estimate_minimal_gas_vector` check) rejects such a transaction depends on whether the minimal L2 gas amount is zero for the given transaction type. If the gateway accepts it (e.g., for transactions where fee enforcement is disabled), the sequencer will include it in a block and the chain split follows deterministically. The TODO comment acknowledges this is a known transitional issue.

### Recommendation

Replace the heuristic zero-check with an explicit version/type tag in the protobuf schema to distinguish pre-0.13.3 (`L1Gas`) from post-0.13.3 (`AllResources`) transactions. Until then, the `unwrap_or_default` fallback should produce `AllResources` (not `L1Gas`) when `l1_data_gas` is absent but `l2_gas` is present, matching the behavior of the RPC deserialization path. The TODO comment already tracks this:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
```

Once 0.13.2 support is removed, assert that `l1_data_gas` is always present and always produce `AllResources`, eliminating the ambiguity.

### Proof of Concept

1. Submit an invoke V3 transaction with `resource_bounds = AllResources { l1_gas: {max_amount: 1000, max_price_per_unit: 1}, l2_gas: {0, 0}, l1_data_gas: {0, 0} }` through the gateway.
2. The gateway computes `tx_hash_A = poseidon(INVOKE, 3, sender, poseidon(tip, L1_GAS_concat, L2_GAS_concat_zero, L1_DATA_GAS_concat_zero), ...)` — `AllResources` path, 3 resource felts.
3. The sequencer includes the transaction in a block with `tx_hash_A`.
4. A syncing node receives the block and deserializes the transaction via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Since `l2_gas.is_zero() && l1_data_gas.is_zero()`, it produces `ValidResourceBounds::L1Gas(l1_gas)`.
5. The syncing node recomputes `tx_hash_B = poseidon(INVOKE, 3, sender, poseidon(tip, L1_GAS_concat, L2_GAS_concat_zero), ...)` — `L1Gas` path, 2 resource felts.
6. `tx_hash_A ≠ tx_hash_B` → `validate_transaction_hash` returns `false` → block rejected → chain split.

impl TryFrom<protobuf::ResourceBounds> for AllResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        Ok(Self {
            l1_gas: value.l1_gas.ok_or(missing("ResourceBounds::l1_gas"))?.try_into()?,
            l2_gas: value.l2_gas.ok_or(missing("ResourceBounds::l2_gas"))?.try_into()?,
            l1_data_gas: value
                .l1_data_gas
                .ok_or(missing("ResourceBounds::l1_data_gas"))?
                .try_into()?,
        })
    }
}
