Audit Report

## Title
`OracleValueStopLossExtension` Uses Arithmetic Mean Instead of Geometric Mean for Oracle Mid-Price, Weakening Stop-Loss Guard — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid price as an arithmetic mean of bid and ask (`(bid + ask) / 2`), while the rest of the protocol — `SwapMath.midAndSpreadFeeX64FromBidAsk` and `PriceVelocityGuardExtension` — uses the geometric mean (`sqrt(bid * ask)`). By the AM-GM inequality, AM ≥ GM for any non-zero spread, causing `metricToken0` to be systematically underestimated and its high-watermark set too low, so the stop-loss guard for `zeroForOne = true` swaps requires a larger real value drop before it fires than the pool admin configured.

## Finding Description
In `_afterSwapOracleStopLoss`, the mid price is computed as:

```solidity
// OracleValueStopLossExtension.sol L218
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
``` [1](#0-0) 

The rest of the protocol uses the geometric mean. `SwapMath.midAndSpreadFeeX64FromBidAsk` computes:

```solidity
// SwapMath.sol L70
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);
``` [2](#0-1) 

`PriceVelocityGuardExtension` explicitly calls this function:

```solidity
// PriceVelocityGuardExtension.sol L48
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
``` [3](#0-2) 

The per-bin metrics are:

```solidity
metricT0 = t0ps + mulDiv(mulDiv(t1, Q64, midPriceX64), METRIC_SCALE, shares);
metricT1 = mulDiv(mulDiv(t0, midPriceX64, Q64), METRIC_SCALE, shares) + t1ps;
``` [4](#0-3) 

Since AM ≥ GM for any non-zero spread:
- `1/AM < 1/GM` → `metricT0` is **underestimated** (the t1 component is divided by a larger-than-correct mid)
- `AM > GM` → `metricT1` is **overestimated** (the t0 component is multiplied by a larger-than-correct mid)

The high-watermark ratchets up to the maximum observed metric. Because `metricT0` is underestimated, `hwm0` is set to a lower value than it would be with the correct geometric mean. The stop-loss floor is `hwm0 * floorMultiplier / E6`:

```solidity
(uint256 hwm0, bool breach0) = _applyWatermark(metricT0, _decayed(hwmS.token0, decayRate, dt), floorMultiplier);
if (breach0 && zeroForOne) {
  revert OracleStopLossTriggered(binIdx, true, metricT0, (hwm0 * floorMultiplier) / E6);
}
``` [5](#0-4) 

A lower `hwm0` produces a lower `floor0`, meaning the guard requires a **larger actual value drop** before it fires for `zeroForOne = true` swaps. No special privilege is required to trigger this path — any swap through a pool with this extension configured calls `afterSwap` and the biased metric computation.

## Impact Explanation
The stop-loss guard for `zeroForOne = true` swaps is less sensitive than the pool admin configured. The invariant "value per share at oracle marks cannot fall faster than drawdown" is violated by the AM/GM gap. LPs who rely on the configured `drawdownE6` threshold for protection are exposed to larger-than-intended value losses before the guard fires. For a 10% oracle spread, the effective drawdown threshold is approximately 0.5% wider than configured; for a 50% spread (common during high-volatility periods when protection is most needed), the gap reaches ~13.4% of the t1 component. On a $1M pool with significant token1 inventory and a 10% oracle spread, this represents ~$5,000 of additional extractable value before the stop-loss fires. This constitutes a medium-severity direct loss of LP principal beyond the configured protection threshold.

## Likelihood Explanation
The bias is present on every swap through any pool with `OracleValueStopLossExtension` configured, whenever `bid ≠ ask` (i.e., whenever the oracle provides a spread, which is the normal operating mode). No special privilege is required — any unprivileged trader can execute a swap. The effect is larger during high-volatility periods when oracle spreads widen, which is precisely when the stop-loss protection is most needed.

## Recommendation
Replace the arithmetic mean with the geometric mean, consistent with the rest of the protocol:

```solidity
// Before (incorrect):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (correct, matches pool swap engine and PriceVelocityGuardExtension):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(
    uint256(bidPriceX64),
    uint256(askPriceX64)
);
```

This aligns the stop-loss metric with the price the pool actually uses for swap settlement, ensuring the guard fires at the threshold the pool admin intended.

## Proof of Concept
1. Deploy a pool with `OracleValueStopLossExtension`, `drawdownE6 = 100_000` (10%), oracle bid = 0.9 Q64, ask = 1.1 Q64.
2. Add liquidity: bin holds `t0 = 0`, `t1 = 1000`, shares = 1000.
3. First swap (`zeroForOne = false`) sets the watermark:
   - AM mid = 1.0 Q64 → `metricToken0 = 0 + 1000/1.0 = 1000`
   - GM mid = 0.99499 Q64 → correct `metricToken0 = 0 + 1000/0.99499 ≈ 1005`
   - `hwm0` is set to 1000 (AM) instead of 1005 (GM).
4. Floor with AM: `1000 * 0.9 = 900`.
5. Floor with GM: `1005 * 0.9 = 904.5`.
6. A subsequent `zeroForOne = true` swap that reduces `metricToken0` to 901:
   - **Passes** the AM-based guard (901 > 900) — stop-loss does not fire.
   - **Would fail** the GM-based guard (901 < 904.5) — stop-loss should fire.
7. The pool admin configured a 10% drawdown but the guard effectively allows ~10.4% drawdown due to the formula mismatch.
