Audit Report

## Title
Missing `owner != address(0)` Validation in `addLiquidity` Permanently Locks Deposited Funds - (File: metric-core/contracts/MetricOmmPool.sol)

## Summary

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` address with no zero-address guard. Because `removeLiquidity` enforces `msg.sender == owner` and the EVM guarantees `msg.sender` is never `address(0)`, any position minted to `address(0)` is permanently irremovable. The depositor's tokens are locked in the pool forever, and `binTotals` is permanently inflated, distorting all subsequent swap and LP operations.

## Finding Description

`MetricOmmPool.addLiquidity` (lines 182–196) accepts `owner` without any validation:

```solidity
function addLiquidity(
    address owner,   // ← never checked against address(0)
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
    if (deltas.binIdxs.length == 0) return (0, 0);
    if (deltas.binIdxs.length != deltas.shares.length) revert LiquidityDeltaLengthMismatch();
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
    (amount0Added, amount1Added) = LiquidityLib.addLiquidity(
      _liquidityContext(), owner, salt, deltas, callbackData, binTotals, _binStates, _binTotalShares, _positionBinShares
    );
    ...
}
```

`owner` is written directly into the position key via `_positionBinKey(owner, salt, int8(binIdx))` in `LiquidityLib.addLiquidity`. The sole authorization gate in `removeLiquidity` (lines 199–212) is:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
```

Since the EVM guarantees `msg.sender != address(0)` for every external call, a position keyed to `address(0)` can never satisfy this check. The position and its underlying token balances are permanently frozen.

**Existing guards are insufficient:**

- The periphery contract `MetricOmmPoolLiquidityAdder` does implement `_validateOwner(owner)` (line 247–249), which reverts on `address(0)`. However, `MetricOmmPool.addLiquidity` is `external` with no access control — any caller can bypass the periphery and call the pool directly.
- The `_beforeAddLiquidity` hook dispatches to configured extensions, but extensions are optional. Pools deployed with no extensions (or with `allowAllDepositors = true` on `DepositAllowlistExtension`) have no protection.
- No core-level guard exists anywhere in the `addLiquidity` → `LiquidityLib.addLiquidity` call path.

## Impact Explanation

**Direct loss of user principal**: every token deposited into an `owner = address(0)` position is permanently locked; the depositor cannot recover them. `binTotals.scaledToken0` / `scaledToken1` grow by the locked amount, inflating apparent liquidity and skewing `curPosInBin` for all subsequent swaps and LP operations on the pool. This constitutes broken core pool functionality causing direct loss of funds and unusable withdraw flows — matching the allowed impact gate for Critical/High severity.

## Likelihood Explanation

`MetricOmmPool.addLiquidity` is `external` with no privileged-caller restriction. Any EOA, router, multicall aggregator, or contract that implements the swap callback can trigger this by passing `owner = address(0)`. No privileged role is required. A malicious actor can deliberately grief a pool by locking a meaningful amount of liquidity, permanently distorting pricing for all other LPs. A non-malicious caller who zero-initialises a struct or passes a default address value can trigger this accidentally.

## Recommendation

Add an explicit zero-address guard at the top of `addLiquidity` in `MetricOmmPool.sol`, before any state-mutating logic:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
    if (owner == address(0)) revert InvalidOwner();   // ← add this
    if (deltas.binIdxs.length == 0) return (0, 0);
    ...
}
```

This mirrors the guard already present in the periphery `MetricOmmPoolLiquidityAdder._validateOwner` and closes the bypass path for direct pool callers.

## Proof of Concept

1. Deploy a `MetricOmmPool` with no extensions (or with `allowAllDepositors = true`).
2. Deploy a contract that implements `metricOmmModifyLiquidityCallback` to supply tokens.
3. Call `pool.addLiquidity(address(0), salt, deltas, callbackData, "")` — the callback transfers `amount0Added` / `amount1Added` tokens from the caller; `_positionBinShares[keccak256(abi.encode(address(0), salt, binIdx))]` is incremented and `binTotals` is updated.
4. Attempt `pool.removeLiquidity(address(0), salt, deltas, "")` from any address → reverts with `NotPositionOwner` because `msg.sender != address(0)`.
5. The deposited tokens are permanently locked; `binTotals` reflects the inflated balance, distorting all future swap prices for the pool.
