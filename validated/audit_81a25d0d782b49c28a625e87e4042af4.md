### Title
`ValidResourceBounds::AllResources` with all-zero gas values silently deserializes as `ValidResourceBounds::L1Gas` across the protobuf p2p boundary, producing a different transaction hash than the one committed on-chain - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

A V3 transaction whose `AllResourceBounds` fields are all zero (e.g. `l1_gas=0, l2_gas=0, l1_data_gas=0`) is serialized over the p2p wire as a `protobuf::ResourceBounds` with three zero-valued fields. Upon deserialization, the converter checks `l1_data_gas.is_zero() && l2_gas.is_zero()` and silently downgrades the result to `ValidResourceBounds::L1Gas`. The `get_tip_resource_bounds_hash` function hashes a different number of resource-bound felts depending on the variant (`L1Gas` → 2 felts; `AllResources` → 3 felts), so the reconstructed transaction hash diverges from the hash that was computed and stored at ingestion time. A peer node that receives such a transaction via p2p sync will compute a wrong hash, accept or reject it incorrectly, and may commit a wrong receipt or state.

---

### Finding Description

**Serialization side (`From<ValidResourceBounds> for protobuf::ResourceBounds`):**

When `ValidResourceBounds::AllResources` is serialized, all three resource-limit fields are emitted, even when they are all zero. [1](#0-0) 

**Deserialization side (`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`):**

The deserializer applies a value-based heuristic: if both `l2_gas` and `l1_data_gas` are zero after decoding, it returns `ValidResourceBounds::L1Gas` regardless of whether the sender intended `AllResources`. [2](#0-1) 

The codebase itself documents this hazard in a comment in the consensus test helper:

> "If all the fields of `AllResources` are 0 upon serialization, then the deserialized value will be interpreted as the `L1Gas` variant." [3](#0-2) 

**Hash divergence:**

`get_tip_resource_bounds_hash` branches on the variant tag, not on the numeric values. `L1Gas` hashes only 2 resource felts (L1_GAS + L2_GAS); `AllResources` hashes 3 (adding L1_DATA_GAS). An all-zero `AllResources` transaction hashed at ingestion produces a 3-felt hash, but after p2p round-trip the reconstructed `L1Gas` variant produces a 2-felt hash. [4](#0-3) 

**Ingestion path always uses `AllResources`:**

`InternalRpcInvokeTransactionV3` stores `resource_bounds: AllResourceBounds` (never `L1Gas`). Its `InvokeTransactionV3Trait` implementation unconditionally wraps it in `ValidResourceBounds::AllResources`, so the hash stored in `tx_hash` always uses the 3-felt path. [5](#0-4) 

After p2p deserialization the same transaction is reconstructed as `ValidResourceBounds::L1Gas`, so any subsequent hash recomputation (e.g. in `validate_transaction_hash`, block commitment, or RPC tracing) produces a different value.

---

### Impact Explanation

**High – RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

A syncing node that receives a V3 transaction with all-zero resource bounds over p2p will reconstruct it with the wrong `ValidResourceBounds` variant. Any hash recomputation, receipt lookup, or RPC trace call on that node will return a hash that does not match the on-chain committed hash, silently serving wrong data to callers.

Additionally, the `GasVectorComputationMode` derived from the variant (`NoL2Gas` for `L1Gas`, `All` for `AllResources`) affects fee validation and gas accounting paths, meaning the syncing node may apply incorrect fee logic to the transaction.

---

### Likelihood Explanation

**Medium.** All-zero resource bounds are a valid and common pattern for fee-free or test transactions (e.g. `AllResourceBounds::new_unlimited_gas_no_fee_enforcement` sets `l1_gas` and `l1_data_gas` to zero amounts, and `AllResourceBounds::create_for_testing` sets all amounts to zero). Any such transaction propagated over the p2p sync path triggers the downgrade. The condition is reachable without any privileged access — any user submitting a V3 transaction with zero L2 and L1-data gas bounds triggers it. [6](#0-5) 

---

### Recommendation

Replace the value-based heuristic in the protobuf deserializer with an explicit discriminant. Options:

1. **Add a boolean/enum field to `protobuf::ResourceBounds`** that encodes whether the sender intended `L1Gas` or `AllResources`, and use that field on deserialization instead of inspecting numeric values.
2. **Treat the presence of a non-`None` `l1_data_gas` field as the `AllResources` discriminant** (the field is already `Option`; a sender of `L1Gas` would omit it, a sender of `AllResources` would include it even when zero).
3. **Remove the downgrade entirely** for the p2p sync path, since all modern transactions are `AllResources` and the `L1Gas` variant is only needed for pre-0.13.3 historical transactions that are not propagated via this path. [7](#0-6) 

---

### Proof of Concept

```
1. Construct a V3 InvokeTransaction with AllResourceBounds { l1_gas: 0, l2_gas: 0, l1_data_gas: 0 }.
2. Compute its hash at ingestion via InternalRpcInvokeTransactionV3::calculate_transaction_hash
   → calls get_invoke_transaction_v3_hash
   → calls get_tip_resource_bounds_hash(AllResources(...))
   → hashes [tip, L1_GAS_felt, L2_GAS_felt, L1_DATA_GAS_felt]  (3 resource felts)
   → produces hash H_all.

3. Serialize the transaction to protobuf::ResourceBounds via From<ValidResourceBounds>:
   → emits l1_gas=Some(0), l2_gas=Some(0), l1_data_gas=Some(0).

4. Deserialize via TryFrom<protobuf::ResourceBounds> for ValidResourceBounds:
   → l1_data_gas.is_zero() && l2_gas.is_zero() == true
   → returns ValidResourceBounds::L1Gas(l1_gas=0).

5. Recompute hash on the receiving node:
   → calls get_tip_resource_bounds_hash(L1Gas(...))
   → hashes [tip, L1_GAS_felt, L2_GAS_felt]  (only 2 resource felts, L1_DATA_GAS omitted)
   → produces hash H_l1 ≠ H_all.

6. The receiving node now holds a transaction whose stored tx_hash (H_all) does not match
   the hash it would compute from the deserialized fields (H_l1).
   RPC trace/receipt lookups keyed by H_all will fail or return wrong data.
```

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}
