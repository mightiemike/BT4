Audit Report

## Title
SwapAllowlistExtension Gates Router Address Instead of End-User, Allowing Complete Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist, where `sender` is `msg.sender` of `pool.swap()` — the immediate caller. When `MetricOmmSimpleRouter` is used, `sender` is the router's address, not the end-user. Allowlisting the router to enable router-based swaps grants every user on the network unconditional access, rendering the allowlist ineffective. Conversely, not allowlisting the router blocks all legitimate users from swapping through the standard periphery entry point.

## Finding Description

**Pool passes `msg.sender` (the router) as `sender` to extensions:**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, recipient, ...)`, hardcoding the immediate caller as `sender`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

**Extension checks the router address, not the actual user:**

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router: [3](#0-2) 

**Router calls `pool.swap()` directly with no identity forwarding:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` — the router becomes `msg.sender` to the pool, and the original caller's identity is lost: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

**Contrast with the correctly designed deposit path:**

`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` — the position owner explicitly passed through the call chain — not `sender`. This correctly identifies the economic actor regardless of intermediaries: [5](#0-4) 

The swap path has no equivalent explicit "actual swapper" parameter. Only `sender` (immediate caller) and `recipient` (output destination) are available, and neither reliably identifies the end-user through a router.

## Impact Explanation

Two fund-impacting failure modes exist:

**Mode 1 — Complete allowlist bypass (critical):** A pool admin who wants to support router-based swaps must allowlist the router. Once `allowedSwapper[pool][router] = true`, every user on the network can call `router.exactInputSingle(...)` and the extension passes unconditionally. Any excluded address — sanctioned parties, non-KYC'd users, competitors — can freely swap against the pool. The allowlist provides zero access control.

**Mode 2 — Broken core swap flow for legitimate users (high):** If the admin does not allowlist the router, every allowlisted user who attempts to swap through the router receives `NotAllowedToSwap`. The only working path is calling `pool.swap()` directly, which requires implementing `IMetricOmmSwapCallback` for token settlement — not a realistic expectation for end-users. The pool's swap functionality is effectively unusable through the standard periphery.

Both modes represent broken core pool functionality with direct fund impact.

## Likelihood Explanation

High. Every pool deploying `SwapAllowlistExtension` that expects users to interact via `MetricOmmSimpleRouter` deterministically hits one of the two failure modes. The router is the standard user-facing entry point. A pool admin who configures an allowlist and also wants router support will naturally allowlist the router, triggering Mode 1. No special attacker capability is required — any EOA calling the router suffices. The unit tests confirm the check is on `sender` (the pool's `msg.sender`), and the integration tests only test direct pool calls, leaving the router-mediated bypass path untested: [6](#0-5) [7](#0-6) 

## Recommendation

The swap path needs an explicit "actual swapper" identity that survives router intermediation. The structurally cleanest fix — consistent with how `DepositAllowlistExtension` correctly gates `owner` — is to extend the swap signature with an explicit `swapper` parameter (analogous to `owner` in `addLiquidity`). The pool passes it to extensions as the gated identity, and the router forwards `msg.sender` as `swapper`. Alternatively, encode the actual user in `extensionData` and have the extension decode it, with the router injecting `msg.sender` before forwarding — though this introduces trust assumptions on the router's encoding.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls: extension.setAllowedToSwap(pool, router, true)
    → intent: allow router-mediated swaps for allowlisted users
    → actual effect: allowedSwapper[pool][router] = true

Attack:
  attacker (not individually allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: attacker, ...})

  router calls:
    pool.swap(attacker, zeroForOne, amount, limit, "", extensionData)
    // msg.sender to pool = router

  pool calls:
    _beforeSwap(router, attacker, ...)

  extension evaluates:
    allowedSwapper[pool][router] → true  ✓ passes

  result:
    attacker swaps successfully against the allowlisted pool
    allowlist provides zero protection
```
