### Title
Missing Block Timestamp Validation (MTP Enforcement) Allows Timestamp Manipulation — (`contract/src/utils.rs`, `contract/src/lib.rs`)

---

### Summary

The contract defines the Median Time Past (MTP) computation and the associated timestamp-bound constants but never enforces them during block submission. Any unprivileged proof submitter can call `submit_blocks()` with headers whose `time` field is arbitrarily set — below the MTP of the previous 11 blocks, or far in the future — and the contract will accept them. This corrupts the difficulty-adjustment calculation and allows the contract to track a chain that the real Bitcoin network would have rejected outright.

---

### Finding Description

Bitcoin's consensus rules impose two timestamp constraints on every block header:

1. **Lower bound (MTP rule):** `header.time > median_time_past(prev 11 blocks)`
2. **Upper bound:** `header.time < network_time + 2h` (or MTP + 90 min)

The codebase acknowledges both rules. The constants are declared in `btc-types/src/network.rs`:

```rust
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 90 * 60;
pub const MAX_FUTURE_BLOCK_TIME_LOCAL: u32 = 2 * 60 * 60;
``` [1](#0-0) 

The MTP computation itself is implemented in `contract/src/utils.rs`:

```rust
#[allow(unused)]
pub fn get_median_time_past(
    block_header: ExtendedHeader,
    prev_block_getter: &impl BlocksGetter,
) -> u32 {
``` [2](#0-1) 

The `#[allow(unused)]` attribute is the direct indicator that this function is **never called**. The `submit_block_header` path in `contract/src/lib.rs` only performs two checks when `skip_pow_verification` is false:

1. `check_target` — validates the difficulty target value
2. PoW hash ≤ target — validates proof-of-work

```rust
if !skip_pow_verification {
    self.check_target(&header, &prev_block_header);
    let pow_hash = header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(header.bits),
        format!("block should have correct pow")
    );
}
``` [3](#0-2) 

No call to `get_median_time_past` or any timestamp comparison against `MAX_FUTURE_BLOCK_TIME_MTP` / `MAX_FUTURE_BLOCK_TIME_LOCAL` appears anywhere in the block-submission path. The `header.time` field is stored verbatim without validation.

---

### Impact Explanation

**Difficulty retarget corruption:** Bitcoin's difficulty adjustment (every 2016 blocks) computes `actual_timespan = last_block.time - first_block.time`. An attacker who submits headers with manipulated timestamps can make `actual_timespan` appear arbitrarily small (inflating difficulty) or arbitrarily large (deflating difficulty). A deflated difficulty means subsequent blocks require less real work, enabling a low-cost chain reorg attack against the light client.

**Acceptance of consensus-invalid blocks:** The contract will accept and permanently store headers that the real Bitcoin network would reject (e.g., `time < MTP`). Downstream NEAR contracts that call `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` will receive `true` for transactions in blocks that are not part of the canonical Bitcoin chain, breaking the SPV guarantee entirely. [4](#0-3) 

---

### Likelihood Explanation

The entry point is the public, unprivileged (after the `trusted_relayer` check) `submit_blocks()` function. Any registered relayer — or any account if the trusted-relayer gate is bypassed via `Role::UnrestrictedSubmitBlocks` — can submit crafted headers. Constructing a header with a manipulated `time` field requires no special knowledge beyond the previous block hash and a valid PoW solution at the current target. The attack is fully self-contained on-chain. [5](#0-4) 

---

### Recommendation

Enforce both timestamp bounds inside `submit_block_header` before storing the header:

1. Compute `mtp = get_median_time_past(prev_block_header, self)` using the already-implemented function.
2. `require!(header.time > mtp, "block time must exceed MTP")`.
3. For the upper bound, use `env::block_timestamp()` (NEAR wall-clock, in nanoseconds) converted to seconds as a proxy for network time, and `require!(header.time < near_time_secs + MAX_FUTURE_BLOCK_TIME_LOCAL)`.

Remove the `#[allow(unused)]` suppression once the function is wired in.

---

### Proof of Concept

1. Deploy the contract with `skip_pow_verification = false` and a valid genesis chain.
2. Craft a `Header` whose `time` field is set to `0` (or any value below the MTP of the last 11 stored blocks). Set `prev_block_hash` to the current tip, compute a valid PoW solution at the current target.
3. Call `submit_blocks([crafted_header])`.
4. Observe: the contract accepts the header and updates `mainchain_tip_blockhash` to the new block.
5. After 2016 such blocks, the difficulty retarget computes `actual_timespan = 0 - genesis_time`, which is clamped or wraps, producing a corrupted target that diverges from the real Bitcoin network's target.
6. Any subsequent `verify_transaction_inclusion_v2` call for a transaction in one of these timestamp-manipulated blocks returns `true`, despite the block being consensus-invalid on the real Bitcoin network. [2](#0-1) [6](#0-5) [7](#0-6)

#[allow(unused)]
pub fn get_median_time_past(
    block_header: ExtendedHeader,
    prev_block_getter: &impl BlocksGetter,
) -> u32 {
    use btc_types::network::MEDIAN_TIME_SPAN;

    let mut median_time = [0u32; MEDIAN_TIME_SPAN];
    let mut current_header = block_header;

    for slot in &mut median_time {
        *slot = current_header.block_header.time;
        current_header = prev_block_getter.get_prev_header(&current_header.block_header);
    }

    median_time.sort_unstable();
    median_time[median_time.len() / 2]
}
```

    pub fn submit_blocks(

        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }

    #[cfg(not(feature = "dogecoin"))]
    #[allow(clippy::needless_pass_by_value)]
    fn submit_block_header(&mut self, header: Header, skip_pow_verification: bool) {
        // We do not have a previous block in the headers_pool, there is a high probability
        // it means we are starting to receive a new fork,
        // so what we do now is we are returning the error code
        // to ask the relay to deploy the previous block.
        //
        // Offchain relay now, should submit blocks one by one in decreasing height order
        // 80 -> 79 -> 78 -> ...
        // And do it until we can accept the block.
        // It means we found an initial fork position.
        // We are starting to gather new fork from this initial position.
        #[allow(clippy::useless_conversion)]
        let prev_block_header = self.get_prev_header(&header.clone().into());
        let current_block_hash = header.block_hash();

        let (current_block_computed_chain_work, overflow) = prev_block_header
            .chain_work
            .overflowing_add(work_from_bits(header.bits));
        require!(!overflow, "Addition of U256 values overflowed");

        let current_header = ExtendedHeader {
            block_header: header.clone().into_light(),
            block_hash: current_block_hash,
            chain_work: current_block_computed_chain_work,
            block_height: 1 + prev_block_header.block_height,
        };

        if !skip_pow_verification {
            self.check_target(&header, &prev_block_header);

            let pow_hash = header.block_hash_pow();
            // Check if the block hash is less than or equal to the target
            require!(
                U256::from_le_bytes(&pow_hash.0) <= target_from_bits(header.bits),
                format!("block should have correct pow")
            );
        }

        self.submit_block_header_inner(current_header, &prev_block_header);
    }
