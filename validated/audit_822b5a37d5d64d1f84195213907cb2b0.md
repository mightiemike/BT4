Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of real swapper, allowing any user to bypass per-pool swap allowlist via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][real_user]`, making the allowlist trivially bypassable by any user who calls the public router — the primary swap entry point.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` directly as `sender` to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol:230-240
_beforeSwap(
    msg.sender,   // whoever called pool.swap()
    recipient, ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to every registered extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol:160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap` checks that exact value against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly without forwarding the original caller:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol:72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The same pattern applies to `exactInput` (L104), `exactOutputSingle` (L136), and `exactOutput` (L165). In all cases, the pool sees `msg.sender = router`, so the extension checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][real_user]`.

Two concrete failure modes result:
1. **Allowlist bypass**: If the pool admin allowlists the router (the natural operational choice so normal users can swap), every address in the world passes the guard.
2. **Allowlist lock-out**: If the pool admin allowlists individual user addresses instead of the router, those users cannot swap through the router at all, breaking the standard swap UX.

The `DepositAllowlistExtension` is not affected because it checks `owner` (line 38: `allowedDepositor[msg.sender][owner]`), which is an explicit parameter representing the LP position owner — not derived from `msg.sender` of the pool call.

## Impact Explanation

`SwapAllowlistExtension` is a production access-control extension whose sole purpose is to restrict which addresses may execute swaps on a given pool. A complete bypass of this guard means an unpermissioned address can execute swaps against a pool explicitly configured to reject it. This constitutes both a broken core pool functionality (the allowlist guard is rendered inoperative) and an admin-boundary break: the pool admin's configured access policy is silently nullified by any user routing through the public router. Pools intended for institutional-only, KYC-gated, or whitelist-only access are fully exposed.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing entry point for swaps. Any user who wants to swap on an allowlist-protected pool will naturally use the router. The bypass requires no special privileges, no malicious setup, no non-standard tokens — only a call to a public function that is already the standard swap path. The bypass is repeatable by any address unconditionally.

## Recommendation

The extension must check the economically relevant actor, not the immediate `msg.sender` of the pool. Two sound approaches:

1. **Preferred**: Redesign `MetricOmmSimpleRouter` to pass `msg.sender` (the end user) as the `sender` argument to `pool.swap()` rather than relying on the pool to use `msg.sender`. This requires a protocol-level change to the pool's `swap()` interface to accept an explicit `sender` parameter, or the router must encode the real caller in `extensionData` and the extension must unwrap it from trusted forwarders.
2. **Alternative**: Introduce a trusted-forwarder registry in `SwapAllowlistExtension` so that when `sender` is a known router, the extension reads the real user identity from `extensionData` (which the router must populate with `msg.sender`).

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (natural setup: allowlist the router so "permitted" users can swap)
  - Alice (address NOT in allowlist) calls:
      router.exactInputSingle({pool: pool, recipient: alice, ...})

Execution trace:
  Alice → router.exactInputSingle()
    → pool.swap(alice_as_recipient, ...)   [msg.sender at pool = router]
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓  (passes)
      → swap executes, Alice receives output tokens

Result:
  Alice, who is NOT in the allowlist, successfully swaps on an allowlist-protected pool.
  The guard checked allowedSwapper[pool][router] instead of allowedSwapper[pool][Alice].

Foundry test sketch:
  1. Deploy pool + SwapAllowlistExtension, configure extension on pool.
  2. setAllowedToSwap(pool, address(router), true).
  3. vm.prank(alice); router.exactInputSingle(...) — assert no revert.
  4. vm.prank(alice); pool.swap(...) directly — assert revert NotAllowedToSwap.
  Steps 3 and 4 together prove the bypass is router-specific.
```
