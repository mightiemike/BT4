### Title
Asymmetric Fee Formula Systematically Overcharges Quote-Paying (Buy) Traders vs Quote-Receiving (Sell) Traders — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

The `applyFee` function in `OffchainExchange.sol` branches on the **sign** of `meteredQuote` to compute fees, using multiplication for positive values (sellers) and division for negative values (buyers). This produces two structurally different fee formulas: sellers pay `feeRate × Q` while buyers pay `feeRate/(1−feeRate) × Q`. Buyers are therefore always overcharged relative to sellers at the same nominal fee rate, and the gap grows as the fee rate increases.

---

### Finding Description

In `applyFee`, after computing `meteredQuote`, the fee is calculated as:

```solidity
int128 keepRateX18 = ONE - feeInfo.feeRate;
int128 newMeteredQuote = (meteredQuote > 0)
    ? meteredQuote.mul(keepRateX18)
    : meteredQuote.div(keepRateX18);
orderInfo.fee = meteredQuote - newMeteredQuote;
``` [1](#0-0) 

The two branches produce mathematically distinct fee amounts for the same nominal `feeRate`:

**Branch A — `meteredQuote > 0` (seller, receiving quote):**
- `newMeteredQuote = Q × (1 − feeRate)`
- `fee = Q × feeRate` ← **FeeIn style**

**Branch B — `meteredQuote < 0` (buyer, paying quote):**
- `newMeteredQuote = −Q / (1 − feeRate)` (more negative)
- `fee = Q × feeRate / (1 − feeRate)` ← **FeeOut style** (larger)

The sign of `meteredQuote` is set directly from `matchQuote = orderInfo.quoteDelta`, which is assigned as `−ordersInfo.maker.quoteDelta` for the taker: [2](#0-1) 

So a taker **buying** base always has `quoteDelta < 0` → `meteredQuote < 0` → FeeOut path. A taker **selling** base always has `quoteDelta > 0` → `meteredQuote > 0` → FeeIn path.

The same asymmetry applies to maker orders via the `else` branch at line 547, where `meteredQuote = matchQuote` directly. [3](#0-2) 

---

### Impact Explanation

**Impact: Low**

For a matched quote amount `Q` and fee rate `r`:
- Seller fee: `r × Q`
- Buyer fee: `r/(1−r) × Q`
- Excess charged to buyer: `r² / (1−r) × Q`

At the default taker rate of 2 bps (`r = 0.0002`):
- Excess per unit = `0.0002² / 0.9998 ≈ 0.00000004` (0.004 bps)

The absolute difference is very small for typical fee rates. However, it is a **systematic, one-directional bias** that always disadvantages buyers and always benefits the protocol's fee collection. The excess is not returned to the counterparty — it accrues to `market.collectedFees`. [4](#0-3) 

---

### Likelihood Explanation

**Likelihood: High**

Every buy-side trade (taker buying base, or maker buying base) triggers the `div` branch. This is a structural property of the formula, not a conditional edge case. Any unprivileged trader submitting a buy order through `matchOrders` is affected on every fill. [5](#0-4) 

---

### Recommendation

Standardize the fee formula to use the same mathematical operation regardless of trade direction. The simplest symmetric fix is to always apply multiplication (FeeIn style):

```solidity
// Symmetric: fee = |meteredQuote| * feeRate, regardless of sign
int128 keepRateX18 = ONE - feeInfo.feeRate;
int128 newMeteredQuote = meteredQuote.mul(keepRateX18);
orderInfo.fee = meteredQuote - newMeteredQuote;
```

This ensures `fee = meteredQuote × feeRate` for both buyers and sellers, eliminating the directional asymmetry. Alternatively, document the intentional asymmetry explicitly if it is a deliberate design choice.

---

### Proof of Concept

Consider a taker buying base with matched quote `Q = 1000e18` and taker fee rate `r = 0.0002` (2 bps):

**Current behavior (FeeOut for buyer):**
- `keepRateX18 = 1e18 − 0.0002e18 = 0.9998e18`
- `meteredQuote = −1000e18` (negative, buyer pays quote)
- `newMeteredQuote = −1000e18 / 0.9998e18 = −1000.2e18`
- `fee = −1000e18 − (−1000.2e18) = 0.2004e18`
- Effective rate: `0.2004 / 1000 = 0.02004%` (2.004 bps)

**Symmetric behavior (FeeIn for buyer):**
- `newMeteredQuote = −1000e18 × 0.9998e18 = −999.8e18`
- `fee = −1000e18 − (−999.8e18) = 0.2e18`
- Effective rate: `0.2 / 1000 = 0.02%` (2.000 bps)

**Excess charged to buyer:** `0.0004e18` per `1000e18` quote traded — a 0.004 bps overcharge that accrues to `collectedFees` on every buy-side fill. [6](#0-5)
