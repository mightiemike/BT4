### Title
Builder Self-Referral Enables Trading Fee Cashback — (`core/contracts/OffchainExchange.sol`)

---

### Summary

The Nado builder fee system contains no check preventing a registered builder from embedding their own `builderId` in their own signed orders. A builder who also trades can route a portion of their own trading fees into `collectedBuilderFee` and then claim it back, achieving a net fee reduction. This is the direct structural analog to M-05: a user controls both sides of a fee-sharing relationship to extract a cashback.

---

### Finding Description

The `appendix` field of every signed `Order` encodes an optional `builderId` and `builderFeeRate` (bits 48–63 and 38–47 respectively). [1](#0-0) 

During order matching, `applyFee()` calls `getUserFeeRateWithBuilder()`, which validates the builder but performs **no check** that the order sender's address differs from the builder's `owner` address: [2](#0-1) 

The only validations are: the builder exists (`owner != address(0)`) and the requested `builderFeeRate` is within the builder's configured bounds. There is no `require(address(uint160(bytes20(sender))) != builder.owner)` guard.

When the fee is applied, the builder fee portion is credited to `collectedBuilderFee`: [3](#0-2) 

The builder owner later claims this via `claimBuilderFee()`, which credits the accumulated amount directly to their subaccount balance: [4](#0-3) 

The claim check only verifies that the sender's embedded address matches the builder's `owner` — it does not prevent the same address from having been the original order sender: [5](#0-4) 

Builder registration is gated to the contract owner via `ContractOwner.updateBuilders()`: [6](#0-5) 

However, once registered, the builder's owner address faces no on-chain restriction against also being a trader who self-references their own `builderId` in signed orders.

---

### Impact Explanation

A registered builder who also trades can reduce their effective taker fee. For example, with a default 2 bps taker rate and a self-specified 1 bps builder fee rate, the builder-trader pays 2 bps gross but recovers 1 bps via `claimBuilderFee`, netting only 1 bps. The 1 bps difference is diverted away from `market.collectedFees` (protocol revenue) and into the builder-trader's own subaccount. At scale, this constitutes measurable value leakage from the protocol's fee pool.

---

### Likelihood Explanation

Builder registration requires the contract owner's approval, which is a prerequisite. However, builders are legitimate, expected participants (frontend providers, integrators). Once registered — a normal operational event — any builder can self-reference in every order they sign without any further admin action. The `appendix` is user-controlled and signed by the trader, not the sequencer. The sequencer submits matched orders but has no on-chain obligation to filter self-referential builder fees, and no event or revert signals this condition. Rational builder-traders will exploit this as soon as they understand the mechanic.

---

### Recommendation

Add a check in `getUserFeeRateWithBuilder()` (or `applyFee()`) that rejects orders where the order sender's address equals the builder's owner address:

```solidity
require(
    address(uint160(bytes20(sender))) != builder.owner,
    ERR_INVALID_BUILDER
);
```

This mirrors the spirit of the M-05 mitigation: prevent the fee recipient from being the same economic party as the fee payer.

---

### Proof of Concept

1. Alice is registered as a builder: `builderId = 7`, `owner = alice_address`, `lowestFeeRate = 0`, `highestFeeRate = 1e14` (1 bps).
2. Alice signs a taker order for product `P` with `appendix` encoding `builderId = 7`, `builderFeeRate = 1e14`.
3. The sequencer matches Alice's order. `applyFee()` is called:
   - `getUserFeeRateWithBuilder` reads `builderId = 7`, finds `builders[7].owner = alice_address`, checks fee rate bounds — **passes with no self-referral check**.
   - `orderInfo.builderFee = matchQuote.abs().mul(1e14)` — 1 bps of Alice's trade value.
   - `collectedBuilderFee[quoteId][7] += orderInfo.builderFee`.
   - Alice's `quoteDelta` is reduced by both the protocol fee and the builder fee.
4. Alice submits a `ClaimBuilderFee` slow-mode transaction with `sender = alice_subaccount`, `builderId = 7`.
5. `claimBuilderFee` verifies `builders[7].owner == alice_address` — **passes**. Credits `collectedBuilderFee[quoteId][7]` back to Alice's subaccount.
6. Net result: Alice paid 2 bps gross, recovered 1 bps, effectively trading at 1 bps — half the protocol's intended taker fee. The 1 bps difference is permanently diverted from `market.collectedFees`.

    function getUserFeeRateWithBuilder(
        bytes32 sender,
        uint32 productId,
        uint128 appendix,
        bool taker
    ) internal view returns (FeeInfo memory) {
        (uint32 builderId, int128 builderFeeRate) = _builderInfo(appendix);
        Builder memory builder;
        if (builderId != 0) {
            builder = builders[builderId];
            if (
                builder.owner == address(0) ||
                builderFeeRate > builder.highestFeeRate ||
                builderFeeRate < builder.lowestFeeRate
            ) {
                revert(ERR_INVALID_BUILDER);
            }
        } else if (builderFeeRate != 0) {
            revert(ERR_INVALID_BUILDER);
        }

        uint32 feeTier = feeTiers[address(uint160(bytes20(sender)))];
        if (feeTier < builder.defaultFeeTier) {
            feeTier = builder.defaultFeeTier;
        }
        FeeRates memory userFeeRates = getTierFeeRateX18(feeTier, productId);
        int128 feeRate = taker
            ? userFeeRates.takerRateX18
            : userFeeRates.makerRateX18;
        return FeeInfo(feeRate, builderId, builderFeeRate);
    }
