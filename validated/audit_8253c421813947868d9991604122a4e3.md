Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual Swapper, Allowing Any User to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap()` checks the `sender` argument forwarded by the pool, which is `msg.sender` of the `pool.swap()` call — the router contract, not the originating user. A pool admin who allowlists the router so that legitimate users can swap via `MetricOmmSimpleRouter` inadvertently opens the gate to every unpermissioned address, since any caller can route through the same public router and have the extension see only the allowlisted router address.

## Finding Description
`MetricOmmPool.swap()` unconditionally passes `msg.sender` as the `sender` argument to `_beforeSwap()`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // router address when called via router
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap()` encodes that value verbatim and dispatches it to every configured extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap()` then evaluates:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

where `msg.sender` is the pool and `sender` is the router. The check resolves to `allowedSwapper[pool][router]`, never `allowedSwapper[pool][actualUser]`.

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly, making itself the `msg.sender` to the pool:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

This creates an inescapable dilemma: if the router is not allowlisted, no allowlisted user can swap via the router; if the router is allowlisted (the only way to enable router-mediated swaps for legitimate users), every unpermissioned address can bypass the gate with a single `exactInputSingle` call. There are no additional guards in the extension, the pool, or the router that recover the originating user identity.

## Impact Explanation
`SwapAllowlistExtension` is the sole on-chain mechanism for restricting swap access on a pool. Bypassing it allows any unpermissioned address to execute swaps on pools the admin intended to restrict (e.g., KYC-gated, whitelist-only, or partner-only pools). LPs who deposited under the assumption that only vetted counterparties could trade against them are exposed to the full universe of public traders, with direct adverse-selection and fund-impacting consequences on LP principal. This constitutes broken core pool functionality causing loss of funds for LPs.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing entry point for swaps. The bypass requires no special privileges, no flash loans, and no multi-transaction setup — a single `exactInputSingle` call suffices. Pool admins who want allowlisted users to be able to use the router will naturally add the router to the allowlist, unknowingly opening the gate to everyone. The mismatch is invisible in normal unit testing because extension tests call it directly from a mock pool with a controlled `sender`, never through the router.

## Recommendation
The extension must gate the economic actor, not the intermediate dispatcher. Two complementary approaches:

1. **Pass the originating user through `extensionData`:** Have the router encode `msg.sender` into `extensionData` before forwarding to the pool. The extension decodes and verifies that address against the allowlist. This must be paired with a check that the encoded address matches the actual router caller to prevent spoofing.

2. **Add a dedicated `originalSender` field via transient storage:** Extend the interface with an `originalSender` parameter that the pool populates from a trusted transient-storage slot set by the router (similar to how the router already stores payer context in transient storage for the swap callback). The extension then checks `allowedSwapper[pool][originalSender]`.

## Proof of Concept
```solidity
// Setup:
// 1. Deploy pool with SwapAllowlistExtension
// 2. Admin allowlists Alice and the router so Alice can use the router
//    extension.setAllowedToSwap(pool, alice, true);
//    extension.setAllowedToSwap(pool, address(router), true);
// 3. Bob (not allowlisted) calls the router directly

function testBypassViaRouter() public {
    assertFalse(extension.isAllowedToSwap(address(pool), bob));
    assertTrue(extension.isAllowedToSwap(address(pool), address(router)));

    vm.prank(bob);
    router.exactInputSingle(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: token0,
            recipient: bob,
            zeroForOne: true,
            amountIn: 1000,
            amountOutMinimum: 0,
            priceLimitX64: type(uint128).max,
            deadline: block.timestamp,
            extensionData: ""
        })
    );
    // Swap succeeds — extension saw sender=router, not bob
}
