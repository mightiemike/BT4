Audit Report

## Title
SwapAllowlistExtension Allowlist Bypassed via Router: Any User Can Swap on Curated Pools When Router Is Allowlisted — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` is used, the router is `msg.sender` to the pool, not the original EOA. If the pool admin allowlists the router (the only way to support router-mediated swaps for legitimate users), every unprivileged user bypasses the allowlist gate entirely by routing through the router. LP principal is directly exposed to unauthorized swappers trading at oracle-derived prices.

## Finding Description

**Root cause — wrong identity checked in `beforeSwap`:**

`SwapAllowlistExtension.beforeSwap` (L37) checks `allowedSwapper[msg.sender][sender]`, where `sender` is the first argument forwarded by the pool:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmPool.swap` sets that argument to `msg.sender` of `pool.swap()` (L231):

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly (L72-80), making the router `msg.sender` to the pool:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][original_EOA]`.

**The binary trap for pool admins:**

| Admin action | Result |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router — broken UX |
| **Allowlist the router** | Every user bypasses the allowlist via the router |

The same structural problem applies to multi-hop `exactInput` (L103-112) and `exactOutput` paths: every hop calls `pool.swap()` with `msg.sender = router`, so every curated pool in the path is bypassed identically.

**Existing guards are insufficient:** There is no mechanism in the extension, pool, or router to propagate the original EOA identity to `beforeSwap`. The `extensionData` field is caller-controlled and not authenticated, so it cannot be used as-is to pass the originator.

## Impact Explanation

A curated pool protected by `SwapAllowlistExtension` is intended to restrict who can trade against its liquidity. When the router is allowlisted (the only way to support router-mediated swaps for legitimate users), any unprivileged EOA can call `router.exactInputSingle()` targeting the curated pool. The extension check passes because `allowedSwapper[pool][router] == true`. The unauthorized user executes a swap at oracle-derived bid/ask prices against LP capital. This is a direct loss of LP principal — the exact loss vector the allowlist was configured to prevent. This satisfies the "admin-boundary break bypassed by an unprivileged path" and "direct loss of user principal" impact gates.

## Likelihood Explanation

- `MetricOmmSimpleRouter` is the canonical, documented periphery for swaps.
- Any pool admin who wants allowlisted users to interact normally (without implementing `IMetricOmmSwapCallback` themselves) must allowlist the router.
- No special conditions, malicious setup, or non-standard tokens are required. Any EOA can call the router.
- The bypass is reachable on every curated pool that allowlists the router, which is the expected production configuration.

## Recommendation

The extension must identify the original economic actor, not the direct caller of `pool.swap()`. Two options:

1. **Require the router to encode the original user's address in `extensionData`** and have the extension decode and authenticate it. Pool admins would allowlist users, not the router. The router must be the only trusted encoder (e.g., via a registry or signature scheme).
2. **Introduce a first-class "originator" field** in the swap call path so extensions always receive the identity of the economic actor regardless of how many intermediary contracts are in the call stack.

Option 1 is the shorter-term fix; option 2 is the architecturally correct long-term solution.

## Proof of Concept

```
Setup:
  pool = curated pool with SwapAllowlistExtension
  pool admin: setAllowedToSwap(pool, alice, true)
  pool admin: setAllowedToSwap(pool, router, true)   ← required for router UX

Attack (Bob, not allowlisted):
  Bob calls router.exactInputSingle({
      pool:       pool,
      recipient:  bob,
      zeroForOne: true,
      amountIn:   X,
      ...
  })

  router → pool.swap(bob, true, X, ...)
           msg.sender to pool = router

  pool → extension.beforeSwap(router, bob, ...)
         msg.sender in extension = pool  ✓
         sender in extension     = router

  Check: allowedSwapper[pool][router] == true  → PASSES

  Bob's swap executes on the curated pool.
  LP funds are exposed to an unauthorized swapper.
```

Foundry test plan: deploy `SwapAllowlistExtension`, configure a pool with it, allowlist `alice` and the router, then call `router.exactInputSingle` from `bob` (not allowlisted) and assert the swap succeeds — demonstrating the bypass.
