Audit Report

## Title
`beforeSwap` Writes Storage Before Velocity Check, Causing Permanent Swap Deadlock on Rapid Price Moves — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary
`PriceVelocityGuardExtension.beforeSwap` unconditionally writes `lastMidPriceX64` and `lastUpdateBlock` to storage before performing the velocity check. When the check reverts, the EVM rolls back those writes, leaving the stored reference price anchored at the pre-move value. Every subsequent swap reads the same stale reference, computes the same excess velocity, and reverts identically, permanently freezing all swaps in both directions until the pool admin manually calls `setLastMidPrice`.

## Finding Description
In `beforeSwap` (lines 57–58), storage is written unconditionally before the guard check:

```solidity
s.lastMidPriceX64 = midPrice;       // line 57 — written before check
s.lastUpdateBlock = uint64(block.number); // line 58 — written before check
```

The velocity check follows at lines 60–74. When `actualSq > allowedSq`, `revert PriceVelocityExceeded` fires at line 73. The EVM rolls back the writes at lines 57–58, restoring `lastMidPriceX64` to `prevMid`. On the next block, the oracle still reports the new price `P₁`; the extension again reads `prevMid = P₀`, computes the same delta, and reverts. This cycle repeats indefinitely. The only escape is a privileged call to `setLastMidPrice` (lines 29–34), which is restricted to the pool admin via `onlyPoolAdmin`.

The guard is also direction-agnostic (the `bool zeroForOne` parameter at line 39 is unnamed/ignored), so a rapid price move blocks swaps in both directions equally — including swaps that would restore the pool toward equilibrium.

## Impact Explanation
All swaps through any pool with this extension active are completely blocked in both directions for the duration of the rapid price move. This satisfies the "broken core pool functionality causing loss of funds or unusable swap flows" impact gate. Traders cannot execute any swap; arbitrageurs cannot rebalance the pool; LPs who wish to exit via swap are blocked. If `OracleValueStopLossExtension` is also active, its `afterSwap` hook never fires, silently bypassing stop-loss protection while LP balances remain exposed to the new oracle price on the next withdrawal.

## Likelihood Explanation
Any genuine rapid market move — a flash crash, a large liquidation cascade, or a high-volatility news event — that moves the oracle mid-price beyond `maxChangePerBlockE18` in a single block triggers the deadlock. No privileged access or special setup is required: any public swap call against a pool with this extension active is sufficient. On chains with fast block times (e.g., HyperEVM), even moderate sustained moves can exceed a tightly configured cap across consecutive blocks.

## Recommendation
Move the storage writes **after** the velocity check so that a failed check leaves `lastMidPriceX64` unchanged:

```solidity
if (prevMid != 0) {
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        uint256 blockDiff = block.number - prevBlock;
        uint256 delta = midPrice > prevMid ? uint256(midPrice - prevMid) : uint256(prevMid - midPrice);
        uint256 changeE18 = (delta * 1e18) / uint256(prevMid);
        uint256 actualSq = changeE18 * changeE18;
        uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
        if (actualSq > allowedSq) revert PriceVelocityExceeded(actualSq, allowedSq);
    }
}
// Write only after passing the check
s.lastMidPriceX64 = midPrice;
s.lastUpdateBlock = uint64(block.number);
```

Additionally, consider making the guard direction-aware so that a rapid price move in one direction does not block swaps in the opposite direction that would restore the pool toward equilibrium.

## Proof of Concept
1. Deploy a pool with `PriceVelocityGuardExtension` configured with `maxChangePerBlockE18 = 0.05e18` (5% per block).
2. Execute one swap at oracle mid-price `P₀`. Extension writes `lastMidPriceX64 = P₀`, `lastUpdateBlock = N`.
3. In block `N+1`, the oracle reports mid-price `P₁ = 0.80 * P₀` (20% crash — legitimate market move).
4. Any swap attempt: reads `P₁` from oracle; writes `lastMidPriceX64 = P₁` (temporary); computes `changeE18 = 0.20e18`, `actualSq = 0.04e36`, `allowedSq = 0.05² * 2 * e36 = 0.005e36`; `0.04e36 > 0.005e36` → `revert PriceVelocityExceeded`; storage rolls back; `lastMidPriceX64` stays at `P₀`.
5. Repeat step 4 in block `N+2`, `N+3`, … — every attempt reverts identically.
6. Pool is frozen. Only `setLastMidPrice(pool, P₁)` by the pool admin unblocks it.
