### Title
Minimum Liquidation Penalty Floor Causes Counterproductive Liquidations for High-Weight Spot Assets — (`core/contracts/ClearinghouseStorage.sol`)

---

### Summary

`getLiqPriceX18` enforces a hard minimum liquidation penalty of 0.5% (`MIN_NON_SPREAD_LIQ_PENALTY_X18`). When a product's `longWeightMaintenanceX18` exceeds `0.995` (i.e., `> 1 - MIN_NON_SPREAD_LIQ_PENALTY_X18`), the minimum penalty floor is insufficient to offset the health weight, causing each liquidation to **decrease** the liquidatee's maintenance health rather than improve it — the exact analog of the Compound counterproductive-incentive class.

---

### Finding Description

In `getLiqPriceX18`, the per-unit liquidation price for a long spot position is:

```
penaltyX18 = (longWeightMaintenanceX18 - ONE) / 5
```

Since `longWeightMaintenanceX18 < ONE` for a long position, `penaltyX18` is negative (a discount). The absolute value is then clamped upward:

```solidity
if (penaltyX18.abs() < MIN_NON_SPREAD_LIQ_PENALTY_X18) {
    penaltyX18 = -MIN_NON_SPREAD_LIQ_PENALTY_X18;   // floor at -0.5%
}
return (risk.priceX18.mul(ONE + penaltyX18), risk.priceX18);
``` [1](#0-0) 

The minimum penalty constant is:

```solidity
int128 constant MIN_NON_SPREAD_LIQ_PENALTY_X18 = ONE / 200; // 0.5%
``` [2](#0-1) 

When the floor applies, the liquidatee receives `price × 0.995` quote per unit of spot surrendered. The maintenance health change from liquidating `x` units is:

```
Δhealth = x × price × (liquidationPrice/price − longWeightMaintenanceX18)
        = x × price × (0.995 − longWeightMaintenanceX18)
```

**When `longWeightMaintenanceX18 > 0.995`, `Δhealth < 0`** — every liquidation makes the subaccount less healthy.

The floor activates whenever `longWeightMaintenanceX18 > 0.975` (computed penalty `< 0.5%`), but the health-worsening threshold is `> 0.995`. For any product configured in that band (e.g., a stablecoin-like asset with weight `0.998`), the invariant is broken.

The `_handleLiquidationPayment` path for a spot long confirms the liquidatee receives exactly `liquidationPrice × amount` quote and loses `amount` spot:

```solidity
spotEngine.updateBalance(txn.productId, txn.liquidatee, -txn.amount);
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.liquidatee, v.liquidationPayment);
``` [3](#0-2) 

No post-liquidation check verifies that maintenance health improved. The only guard is:

```solidity
require(!isAboveInitial(txn.liquidatee), ERR_LIQUIDATED_TOO_MUCH);
``` [4](#0-3) 

This prevents over-liquidation (health going too positive) but does **not** prevent health from becoming more negative.

The same structural flaw exists in `getSpreadLiqPriceX18` with `MIN_SPREAD_LIQ_PENALTY_X18 = 0.25%`:

```solidity
int128 constant MIN_SPREAD_LIQ_PENALTY_X18 = ONE / 400; // 0.25%
``` [5](#0-4) 

For spread positions, the threshold is `spreadWeight > 0.9975`, reachable when the underlying perp's `longWeightMaintenanceX18 > 0.9875`.

---

### Impact Explanation

A liquidator can submit repeated liquidations against a subaccount holding a high-weight spot asset. Each call:

1. Passes the `isUnderMaintenance` check (health is already negative).
2. Executes the transfer at the minimum-penalty price.
3. Leaves the subaccount with **lower** maintenance health than before.
4. Repeats until the subaccount is fully insolvent.

The insurance fund is then forced to cover the resulting bad debt via `spotEngine.socializeSubaccount` / `perpEngine.socializeSubaccount` in `_finalizeSubaccount`, socializing losses across all NLP holders. The protocol's core invariant — that liquidations restore solvency — is violated. [6](#0-5) 

---

### Likelihood Explanation

The trigger requires a product whose `longWeightMaintenanceX18 > 0.995`. The `RiskStore` stores weights as `int32` values scaled by `1e9`; a value of `998_000_000` (weight `0.998`) is syntactically valid and semantically reasonable for a stablecoin-collateral or low-volatility asset. No on-chain enforcement prevents this configuration. [7](#0-6) 

Once such a product exists, any liquidator whose request is sequenced can trigger the counterproductive path. The sequencer submits the transaction, but the broken math executes unconditionally inside `liquidateSubaccountImpl`. [8](#0-7) 

---

### Recommendation

Enforce at product-registration time (or in `getLiqPriceX18`) that:

```
longWeightMaintenanceX18 ≤ ONE − MIN_NON_SPREAD_LIQ_PENALTY_X18
```

i.e., `longWeightMaintenanceX18 ≤ 0.995`. This guarantees `Δhealth > 0` for every non-spread liquidation. Alternatively, replace the hard floor with a dynamic minimum penalty derived from the weight itself:

```
effectivePenalty = max(|(weight − 1) / 5|, weight − (1 − ε))
```

so the discount always exceeds the health weight by at least `ε`. Apply the same fix to `getSpreadLiqPriceX18` with `MIN_SPREAD_LIQ_PENALTY_X18`.

---

### Proof of Concept

**Setup:** product with `longWeightMaintenanceX18 = 0.998`, oracle price `$1.00`.

**Subaccount state:**
- Spot balance: `1000` units → health contribution `1000 × 1.00 × 0.998 = 998`
- Quote balance: `−999` → health contribution `−999`
- Maintenance health: `998 − 999 = −1` → **liquidatable** ✓

**Liquidation of `x = 100` units:**
- `penaltyX18 = (0.998 − 1) / 5 = −0.0004` → clamped to `−0.005`
- `liquidationPrice = 1.00 × 0.995 = $0.995`
- `liquidationPayment = 100 × 0.995 = 99.5`

**Post-liquidation state:**
- Spot balance: `900` → health `900 × 0.998 = 898.2`
- Quote balance: `−999 + 99.5 = −899.5`
- Maintenance health: `898.2 − 899.5 = −1.3`

Health moved from `−1` to `−1.3`. The liquidation made the subaccount **more insolvent**. Repeating this process drives the subaccount to full insolvency, forcing the insurance fund to socialize the deficit. [9](#0-8)

    function getLiqPriceX18(uint32 productId, int128 amount)
        internal
        returns (int128, int128)
    {
        RiskHelper.Risk memory risk = IProductEngine(productToEngine[productId])
            .getRisk(productId);
        int128 penaltyX18 = (RiskHelper._getWeightX18(
            risk,
            amount,
            IProductEngine.HealthType.MAINTENANCE
        ) - ONE) / 5;
        if (penaltyX18.abs() < MIN_NON_SPREAD_LIQ_PENALTY_X18) {
            if (penaltyX18 < 0) {
                penaltyX18 = -MIN_NON_SPREAD_LIQ_PENALTY_X18;
            } else {
                penaltyX18 = MIN_NON_SPREAD_LIQ_PENALTY_X18;
            }
        }
        return (risk.priceX18.mul(ONE + penaltyX18), risk.priceX18);
    }

    function liquidateSubaccountImpl(IEndpoint.LiquidateSubaccount calldata txn)
        external
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.sender != txn.liquidatee, ERR_UNAUTHORIZED);
        require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
        require(
            txn.liquidatee != X_ACCOUNT && txn.liquidatee != N_ACCOUNT,
            ERR_NOT_LIQUIDATABLE
        );
        require(
            txn.productId != QUOTE_PRODUCT_ID,
            ERR_INVALID_LIQUIDATION_PARAMS
        );

        ISpotEngine spotEngine = ISpotEngine(
            address(engineByType[IProductEngine.EngineType.SPOT])
        );
        IPerpEngine perpEngine = IPerpEngine(
            address(engineByType[IProductEngine.EngineType.PERP])
        );

        if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
            if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.liquidatee);
            }
            return;
        }

        if (
            (txn.amount < 0) &&
            (txn.isEncodedSpread ||
                address(productToEngine[txn.productId]) == address(spotEngine))
        ) {
            // when it's spread or spot liquidation, we need to make sure the liquidatee has
            // enough quote to buyback the liquidated amount.
            _assertCanLiquidateLiability(txn, spotEngine, perpEngine);
            _settlePositivePerpPnl(txn, spotEngine, perpEngine);
        }

        _assertLiquidationAmount(txn, spotEngine, perpEngine);

        // beyond this point, we can be sure that we can liquidate the entire
        // liquidation amount knowing that the insurance fund will remain solvent
        // subsequently we can just blast the remainder of the liquidation and
        // cover the quote balance from the insurance fund at the end
        _handleLiquidationPayment(txn, spotEngine, perpEngine);
    }
