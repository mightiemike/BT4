### Title
Missing `tx_index` Upper-Bound Check Enables Phantom-Position SPV Proof Acceptance - (File: `merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `tx_index` without verifying it is less than `2^merkle_proof.len()`. Bitcoin's Merkle tree duplicates the last hash on odd-length levels. For any block with an odd transaction count, the last transaction's proof verifies correctly at both its real position and the phantom duplicate position (real + 1), allowing any unprivileged NEAR caller to obtain a `true` SPV result for a position that does not exist in the block.

### Finding Description

`compute_root_from_merkle_proof` uses `transaction_position` only for left/right ordering (`current_position % 2`) and then halves it each round. There is no check that `transaction_position < 2^merkle_proof.len()`. [1](#0-0) 

The contract passes the raw caller-supplied `args.tx_index` directly into this function: [2](#0-1) 

`tx_index` is a plain `u64` in `ProofArgs` with no upper-bound validation anywhere in the call chain: [3](#0-2) 

**Why the phantom position verifies correctly:**

Bitcoin's Merkle tree pads odd-length levels by duplicating the last hash. For a 3-transaction block `[A, B, C]`:

```
Level 0: [A, B, C, C]          ← C duplicated
Level 1: [H(A,B), H(C,C)]
Root:     H(H(A,B), H(C,C))
```

Proof for C at real position 2 → `proof = [C, H(A,B)]`:
- pos=2 (even): `hash = H(C, C)`, pos → 1
- pos=1 (odd):  `hash = H(H(A,B), H(C,C))` = root ✓

Same proof with phantom position 3 (does not exist in the block):
- pos=3 (odd):  `hash = H(C, C)`, pos → 1
- pos=1 (odd):  `hash = H(H(A,B), H(C,C))` = root ✓

Both positions produce the correct root. The function returns `true` for position 3 even though only positions 0, 1, 2 are real.

`verify_transaction_inclusion_v2` is equally affected. Its coinbase proof check only anchors the coinbase at position 0; it does not constrain `tx_index` for the target transaction: [4](#0-3) 

### Impact Explanation

Any NEAR contract consuming `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` as a trust gate (e.g., a cross-chain bridge releasing funds upon SPV proof) receives a `true` result for a phantom duplicate position. The attacker cannot forge an arbitrary transaction (the `tx_id` must be a real transaction in the block), but they can present a real transaction as existing at a non-existent index. Consuming contracts that track processed `(tx_id, tx_index)` pairs rather than just `tx_id` are directly exploitable for double-action: the same transaction can be "proven" at both its real position and its phantom duplicate position, triggering the guarded action twice.

### Likelihood Explanation

Medium. Blocks with an odd transaction count are common on Bitcoin (and all supported chains). The required inputs — the last transaction's `tx_id` and its Merkle proof — are publicly available from any Bitcoin node or block explorer. No privileged access is needed; any NEAR account can call the public `verify_transaction_inclusion_v2` endpoint.

### Recommendation

Add an explicit upper-bound check on `tx_index` before invoking `compute_root_from_merkle_proof`:

```rust
require!(
    args.tx_index < (1u64 << args.merkle_proof.len()),
    "tx_index exceeds the maximum valid position for the given proof depth"
);
```

This mirrors the invariant that the CurveV1 report recommends enforcing (`i < N_COINS`) and closes the phantom-position acceptance path.

### Proof of Concept

1. Identify a Bitcoin block `B` with exactly 3 transactions `[C0, T1, T2]` (odd count).
2. Obtain `T2`'s Merkle proof from any block explorer: `proof = [T2, H(C0,T1)]`.
3. Call `verify_transaction_inclusion_v2` on the NEAR contract:
   - `tx_id` = hash of `T2`
   - `tx_index` = **3** (phantom duplicate position; only 0–2 are real)
   - `merkle_proof` = `[T2, H(C0,T1)]`
   - `coinbase_tx_id` = hash of `C0`
   - `coinbase_merkle_proof` = valid proof for `C0` at position 0
   - `tx_block_blockhash` = hash of `B`
4. The coinbase check passes (C0 at position 0 is valid). `verify_transaction_inclusion` is then called with `tx_index=3`. `compute_root_from_merkle_proof` computes `H(T2,T2)` then `H(H(C0,T1), H(T2,T2))` = stored root. The function returns **`true`** for a position that does not exist in the block. [5](#0-4)

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```
