### Title
Wrong `l1_da_mode` in Pending-Block Fee Estimation When Pending Data Is Stale — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary

When the in-memory pending data is stale (its `parent_block_hash` does not match the latest committed block hash), `read_pending_data` constructs a synthetic fallback using `DeprecatedPendingBlock`. That type's `l1_da_mode()` unconditionally returns `L1DataAvailabilityMode::Calldata`. If the latest committed block uses `L1DataAvailabilityMode::Blob`, the fallback silently drops the Blob DA mode, causing `create_block_context` to set `use_kzg_da = false` and fee estimation to compute DA costs using calldata pricing instead of blob pricing.

### Finding Description

**Step 1 — Fallback construction omits `l1_da_mode`.**

`read_pending_data` checks whether the cached pending block's `parent_block_hash` matches the latest committed block hash. On mismatch it builds:

```rust
PendingBlockOrDeprecated::Deprecated(DeprecatedPendingBlock {
    parent_block_hash: latest_header.block_hash,
    eth_l1_gas_price: latest_header.block_header_without_hash.l1_gas_price.price_in_wei,
    strk_l1_gas_price: ...,
    timestamp: ...,
    sequencer_address: ...,
    starknet_version: ...,
    ..Default::default()
})
``` [1](#0-0) 

`latest_header.block_header_without_hash.l1_da_mode` is available but is never copied into the fallback.

**Step 2 — `l1_da_mode()` on `Deprecated` variant is hardcoded to `Calldata`.**

```rust
pub fn l1_da_mode(&self) -> L1DataAvailabilityMode {
    match self {
        // In older versions, all blocks were using calldata.
        PendingBlockOrDeprecated::Deprecated(_) => L1DataAvailabilityMode::Calldata,
        PendingBlockOrDeprecated::Current(block) => block.l1_da_mode,
    }
}
``` [2](#0-1) 

**Step 3 — Conversion to `ExecutionPendingData` propagates the wrong value.**

`client_pending_data_to_execution_pending_data` calls `client_pending_data.block.l1_da_mode()` directly, so the `Calldata` value flows into `ExecutionPendingData.l1_da_mode`. [3](#0-2) 

**Step 4 — `create_block_context` converts `l1_da_mode` to `use_kzg_da`.**

```rust
let use_kzg_da = if override_kzg_da_to_false { false } else { l1_da_mode.is_use_kzg_da() };
```

With `l1_da_mode = Calldata`, `use_kzg_da = false` even when the committed block used Blob DA. [4](#0-3) 

**Step 5 — `estimate_fee` for `block_id = Tag::Pending` uses this path.** [5](#0-4) 

**Step 6 — DA gas cost computation diverges.**

With `use_kzg_da = false`, `get_da_gas_cost` charges calldata gas (`SHARP_GAS_PER_DA_WORD` per field element) instead of blob gas (`DATA_GAS_PER_FIELD_ELEMENT`), producing a materially different fee estimate. [6](#0-5) 

### Impact Explanation

Any unprivileged caller invoking `starknet_estimateFee` (or `starknet_simulateTransactions`, `starknet_traceBlockTransactions`) with `block_id = "pending"` during the window when pending data is stale receives a fee estimate computed under the wrong DA mode. On a Blob-DA network this means blob-DA costs are replaced with calldata-DA costs, which are structurally different (blob gas vs L1 gas). The returned `FeeEstimation` is authoritative-looking but wrong.

This matches the allowed impact: **High — RPC fee estimation returns an authoritative-looking wrong value.**

### Likelihood Explanation

The staleness window is a normal operational condition: it occurs on every block transition between the moment a new block is committed to storage and the moment the pending-data sync task updates the in-memory `PendingData`. On a Blob-DA network (Starknet mainnet post-0.13.2) this window is always present and the bug fires on every block boundary for any caller using `block_id = "pending"`.

### Recommendation

In the fallback branch of `read_pending_data`, copy `l1_da_mode` from the latest committed block header instead of relying on `DeprecatedPendingBlock`'s hardcoded `Calldata` default:

```rust
// In the fallback DeprecatedPendingBlock construction, add:
// (or switch to PendingBlock which carries l1_da_mode explicitly)
```

The cleanest fix is to use `PendingBlock` (the `Current` variant) for the fallback, which carries an explicit `l1_da_mode` field, and populate it from `latest_header.block_header_without_hash.l1_da_mode`.

### Proof of Concept

1. Commit a block with `l1_da_mode = L1DataAvailabilityMode::Blob` to storage.
2. Set the in-memory `PendingData.block.parent_block_hash` to a stale/wrong value (simulating the transition window).
3. Call `estimate_fee` with `block_id = Tag::Pending`.
4. Observe that `create_block_context` receives `l1_da_mode = Calldata` → `use_kzg_da = false`, and the returned fee uses calldata DA pricing instead of blob DA pricing.

The existing test at `execution_test.rs:440-453` already demonstrates that `l1_da_mode` materially changes the fee for committed blocks; the same divergence occurs for the pending fallback path but is not currently tested. [7](#0-6)
