### Title
Missing version gate on `fee_proposal_fri` in `TryFrom<protobuf::SignedBlockHeader>` allows malicious peer to poison `fee_proposals_window` — (`crates/apollo_protobuf/src/converters/header.rs`)

---

### Summary

The `TryFrom<protobuf::SignedBlockHeader> for SignedBlockHeader` converter unconditionally maps the optional `fee_proposal_fri` protobuf field into the internal `BlockHeaderWithoutHash` struct without checking whether the block's `starknet_version` is `>= V0_14_3`. A malicious p2p peer can send a crafted `SignedBlockHeader` with `protocol_version = "0.14.2"` and `fee_proposal_fri = Some(arbitrary_u128)`. The value passes deserialization, is written to storage, and is later loaded into the consensus orchestrator's `fee_proposals_window`, poisoning `compute_fee_actual` and skewing fee proposal validation for future blocks.

---

### Finding Description

**Deserialization — no version gate:**

In `crates/apollo_protobuf/src/converters/header.rs` line 179, the converter reads `starknet_version` from `protocol_version` (line 126–134) but never cross-checks it against `fee_proposal_fri`:

```rust
let fee_proposal_fri = value.fee_proposal_fri.map(|v| GasPrice(u128::from(v)));
``` [1](#0-0) [2](#0-1) 

The proto definition itself warns that the field must be absent for pre-V0_14_3 blocks and is not part of the block hash:

```
// Proposer's oracle-derived recommended fee. Absent for pre-V0_14_3 blocks.
// WARNING: this field is currently not part of the block hash, so the value must be trusted.
optional Uint128 fee_proposal_fri = 22;
``` [3](#0-2) 

The same absence-of-version-check is confirmed in the generated Rust struct: [4](#0-3) 

**P2P sync client — no additional validation:**

`parse_data_for_block` in `crates/apollo_p2p_sync/src/client/header.rs` only checks block number ordering and signature count. There is no block-hash verification and no version-gating of `fee_proposal_fri`: [5](#0-4) 

`write_to_storage` then calls `append_header` directly, persisting the injected value: [6](#0-5) 

**Storage — injected value persisted:**

`StorageBlockHeader` stores `fee_proposal_fri: Option<GasPrice>` with no version invariant enforced at write time: [7](#0-6) 

**`fee_proposals_window` poisoning:**

`initialize_fee_proposals_window` reads `fee_proposal_fri` directly from storage and inserts it into the in-memory window: [8](#0-7) 

`try_sync` does the same for blocks synced during consensus: [9](#0-8) 

**`compute_fee_actual` skewed:**

`compute_fee_actual` treats any `Some(price)` entry as a valid fee proposal and includes it in the median. A pre-0.14.3 block that should contribute `None` (causing the function to return `None` and fall back to `l2_gas_price`) instead contributes an attacker-chosen value, shifting the median: [10](#0-9) 

**Contrast with the guarded consensus path:**

The version check exists only for `ProposalInit` in the consensus validator, not for `SignedBlockHeader` in the p2p sync path: [11](#0-10) 

---

### Impact Explanation

A malicious p2p peer can inject an arbitrary `fee_proposal_fri` value into any pre-0.14.3 `SignedBlockHeader`. Because `fee_proposal_fri` is explicitly excluded from the block hash, block-hash verification cannot detect the injection. The poisoned value propagates from storage into `fee_proposals_window`, corrupting `compute_fee_actual`. This causes the sequencer to compute a wrong `fee_actual`, which is then used to validate or reject fee proposals from other validators — a concrete, attacker-controlled divergence in fee estimation logic.

The RPC does not expose `fee_proposal_fri` (no matches in `crates/apollo_rpc/src/**/*.rs`), so the specific RPC-exposure sub-claim in the question is not realized. The actual impact is on the consensus fee-estimation path.

---

### Likelihood Explanation

Any unprivileged p2p peer the node syncs from can craft this message. The p2p sync client performs no cryptographic verification of `fee_proposal_fri` and no version-gating. The attack requires only that the attacker be a reachable p2p peer during historical sync, which is a standard network-level capability.

---

### Recommendation

Add a version gate in `TryFrom<protobuf::SignedBlockHeader> for SignedBlockHeader` immediately after parsing `starknet_version`:

```rust
if value.fee_proposal_fri.is_some()
    && starknet_version < StarknetVersion::V0_14_3
{
    return Err(ProtobufConversionError::OutOfRangeValue {
        type_description: "fee_proposal_fri",
        value_as_str: format!(
            "fee_proposal_fri must be absent for version {starknet_version}"
        ),
    });
}
let fee_proposal_fri = value.fee_proposal_fri.map(|v| GasPrice(u128::from(v)));
```

The same guard should be applied to `TryFrom<protobuf::ProposalInit> for ProposalInit` at line 229 of `crates/apollo_protobuf/src/converters/consensus.rs`, which has the same structural gap (though the downstream consensus validator catches it there). [12](#0-11)

### Proof of Concept

```rust
// In a Rust integration test (production code path, no mocks):
use apollo_protobuf::protobuf::{self, Uint128};
use apollo_protobuf::sync::SignedBlockHeader;
use std::convert::TryFrom;

let mut proto_header = protobuf::SignedBlockHeader::default();
// Fill required fields with valid values ...
proto_header.protocol_version = "0.14.2".to_string();
proto_header.fee_proposal_fri = Some(Uint128 { low: 999, high: 0 });

let result = SignedBlockHeader::try_from(proto_header).unwrap();
// Asserts: fee_proposal_fri is Some(GasPrice(999)) despite protocol_version < 0.14.3
assert_eq!(
    result.block_header.block_header_without_hash.fee_proposal_fri,
    Some(starknet_api::block::GasPrice(999))
);
// The value is then written to storage and loaded into fee_proposals_window,
// corrupting compute_fee_actual for future blocks.
```
