### Title
Division-by-Zero in `socializeSubaccount` When `openInterest == 0` — (`core/contracts/PerpEngine.sol`)

---

### Summary

`socializeSubaccount` unconditionally divides by `state.openInterest` when insurance is insufficient to cover a negative `vQuoteBalance`. If `openInterest` is zero at the time of the call — a reachable state when all positions in a market are closed — the transaction reverts with a division-by-zero panic, permanently blocking the liquidation/socialization path for that subaccount.

---

### Finding Description

The vulnerable division is at lines 166–168 of `PerpEngine.sol`:

```solidity
if (balance.vQuoteBalance < 0) {
    // socialize across all other participants
    int128 fundingPerShare = -balance.vQuoteBalance.div(
        state.openInterest   // ← can be 0
    );
``` [1](#0-0) 

There is no guard checking `state.openInterest > 0` before this division.

**How `openInterest == 0` with `vQuoteBalance < 0` is reached:**

`_updateBalance` tracks `openInterest` as the sum of absolute position sizes across all participants:

```solidity
state.openInterest -= balance.amount.abs();   // pre-update
...
state.openInterest += balance.amount;          // post-update (long)
// or
state.openInterest -= balance.amount;          // post-update (short)
``` [2](#0-1) 

When a subaccount closes its position (`amount → 0`), `vQuoteBalance` retains any accumulated realized loss. If every other participant also closes their positions, `openInterest` reaches zero. The closed subaccount still carries a negative `vQuoteBalance` representing its realized loss.

**Concrete path:**

1. Subaccount A opens a perp position → `amount != 0`, `vQuoteBalance` goes negative (loss).
2. Subaccount A closes the position → `amount = 0`, `vQuoteBalance` remains negative.
3. All other participants close their positions → `state.openInterest = 0`.
4. Subaccount A's health drops below maintenance (negative `vQuoteBalance` reduces spot health via `settlePnl`).
5. Sequencer submits `liquidateSubaccount` → `Clearinghouse.liquidateSubaccount` → `ClearinghouseLiq.liquidateSubaccountImpl` → `PerpEngine.socializeSubaccount`.
6. Insurance is zero or insufficient → `balance.vQuoteBalance` remains `< 0` after the insurance cover step.
7. Division by zero at line 166 → revert → liquidation is permanently blocked.

The `updateStates` function already handles this case correctly by skipping products with zero open interest:

```solidity
if (state.openInterest == 0) {
    continue;
}
``` [3](#0-2) 

`socializeSubaccount` lacks the equivalent guard.

---

### Impact Explanation

The liquidation transaction reverts every time it is attempted for this subaccount. The insolvent account cannot be socialized, its bad debt cannot be resolved, and the protocol's last-resort insolvency mechanism is permanently broken for that market state. This is not a transient revert — it persists as long as `openInterest` remains zero, which is a stable state once all participants have exited.

---

### Likelihood Explanation

Requires: (a) a subaccount with `amount == 0` and `vQuoteBalance < 0`, (b) all other participants having closed their positions in that product, and (c) insurance being insufficient. This is a low-frequency but entirely reachable edge case, especially in low-liquidity or newly-launched perp markets.

---

### Recommendation

Add a zero-check before the division, mirroring the pattern already used in `updateStates`:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; absorb loss or skip
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(
            state.openInterest
        );
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

The exact handling of the zero-`openInterest` branch (absorb into insurance, write off, or emit an event) is a protocol design decision, but the division must be guarded.

---

### Proof of Concept

```solidity
// Hardhat pseudocode
// 1. Open position for accountA (only participant)
await perpEngine.updateBalance(productId, accountA, +1e18, -100e18); // long 1, vQuote -100

// 2. Close position for accountA
await perpEngine.updateBalance(productId, accountA, -1e18, +90e18);  // close, vQuote = -10 (loss)
// state.openInterest == 0 now

// 3. Verify state
const state = await perpEngine.states(productId);
assert(state.openInterest == 0);
const bal = await perpEngine.getBalance(productId, accountA);
assert(bal.vQuoteBalance < 0);

// 4. Call socializeSubaccount with zero insurance → division by zero revert
await expect(
    perpEngine.connect(clearinghouse).socializeSubaccount(accountA, 0)
).to.be.revertedWithPanic(0x12); // division by zero
``` [4](#0-3)
