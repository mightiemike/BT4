### Title
`FeeEstimation` Omits `l2_gas_consumed` While Including `l2_gas_price`, Making the RPC Fee Breakdown Structurally Inconsistent and Misleading - (`File: crates/apollo_rpc_execution/src/objects.rs`)

### Summary

The `FeeEstimation` struct returned by `starknet_estimateFee` and `starknet_simulateTransactions` exposes `l2_gas_price` but never populates or exposes `l2_gas_consumed`. The `overall_fee` field is taken directly from `receipt.fee` (which correctly includes the l2_gas cost), but the breakdown fields (`gas_consumed`, `data_gas_consumed`) only cover l1_gas and l1_data_gas. For any v3 transaction with non-zero l2_gas usage, the invariant stated in both the struct doc-comment and the OpenRPC spec — `overall_fee = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price` — is provably false, and the caller has no way to reconstruct the correct fee or derive the l2_gas resource bound needed to submit the transaction.

### Finding Description

In `tx_execution_output_to_fee_estimation` the full `GasVector` is read from the receipt, but only `l1_gas` and `l1_data_gas` are placed into the response:

```rust
// crates/apollo_rpc_execution/src/objects.rs  lines 172-182
let gas_vector = tx_execution_output.execution_info.receipt.gas;

Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),      // l1_gas only
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(), // l1_data_gas only
    l1_data_gas_price,
    l2_gas_price,                                  // price present …
    overall_fee: tx_execution_output.execution_info.receipt.fee, // includes l2_gas cost
    unit: tx_execution_output.price_unit,
})
// gas_vector.l2_gas is silently dropped
``` [1](#0-0) 

The struct definition acknowledges the gap with a TODO:

```rust
// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
// close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
pub l2_gas_price: GasPrice,
``` [2](#0-1) 

The OpenRPC spec for `FEE_ESTIMATE` states the invariant:

> `overall_fee` equals `gas_consumed*gas_price + data_gas_consumed*data_gas_price` [3](#0-2) 

For any v3 transaction where `gas_vector.l2_gas > 0`, the actual `overall_fee` (which includes `l2_gas * l2_gas_price`) is strictly greater than the value the spec formula produces from the exposed fields. The discrepancy is not a rounding error; it is a structurally missing term.

### Impact Explanation

**High — RPC fee estimation returns an authoritative-looking wrong value.**

1. **Broken fee breakdown**: A caller that reconstructs the fee from the returned fields (`gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`) will compute a value lower than `overall_fee` for every v3 transaction with l2_gas. The response is internally inconsistent.

2. **Unusable for resource-bound derivation**: The primary use of `starknet_estimateFee` is to derive the `resource_bounds` to attach to a v3 transaction. Without `l2_gas_consumed`, a caller cannot determine the correct `l2_gas.max_amount` bound. Setting it to zero causes the transaction to fail at pre-validation (`MaxGasAmountExceeded`); setting it to an arbitrary large value forces the pre-validation balance check against the full worst-case fee (`max_amount * max_price_per_unit` for all three resources), potentially blocking submission.

3. **`l2_gas_price` present without `l2_gas_consumed`**: The presence of `l2_gas_price` in the response implies the caller can use it, but without the paired consumed amount the field is structurally orphaned and misleading.

### Likelihood Explanation

Every v3 (`AllResources`) transaction executed on a network running Starknet ≥ 0.13.3 will have non-zero `l2_gas` in its receipt. The `starknet_estimateFee` and `starknet_simulateTransactions` endpoints are the canonical way for wallets and SDKs to size resource bounds before submission, so this path is exercised on every transaction submission flow.

### Recommendation

Add `l2_gas_consumed` to `FeeEstimation` and populate it from `gas_vector.l2_gas`:

```rust
pub struct FeeEstimation {
    pub gas_consumed: Felt,
    pub l1_gas_price: GasPrice,
    pub data_gas_consumed: Felt,
    pub l1_data_gas_price: GasPrice,
    pub l2_gas_consumed: Felt,   // add this
    pub l2_gas_price: GasPrice,
    pub overall_fee: Fee,
    pub unit: PriceUnit,
}

// in tx_execution_output_to_fee_estimation:
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(),
    l1_data_gas_price,
    l2_gas_consumed: gas_vector.l2_gas.0.into(),  // add this
    l2_gas_price,
    overall_fee: tx_execution_output.execution_info.receipt.fee,
    unit: tx_execution_output.price_unit,
})
```

Update the OpenRPC spec description of `overall_fee` to reflect the three-term formula, and update the `FEE_ESTIMATE` schema to include `l2_gas_consumed` as a required field.

### Proof of Concept

1. Submit a v3 invoke transaction to `starknet_estimateFee` on a network where l2_gas is priced non-zero (any post-0.13.3 network).
2. Observe the returned `FeeEstimation`: `l2_gas_price` is non-zero, but there is no `l2_gas_consumed` field.
3. Compute `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`. The result is strictly less than `overall_fee`.
4. Attempt to submit the transaction with `l2_gas.max_amount = 0` (the only value derivable from the response): the blockifier rejects it with `MaxGasAmountExceeded` for `L2Gas`.

The `gas_vector.l2_gas` value is read at line 172 but never written into the returned struct, confirming the omission is structural rather than conditional. [4](#0-3)

pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}
