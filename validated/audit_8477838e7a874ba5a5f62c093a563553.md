### Title
Migration `Migrate1to2` halts chain when `RewardsPool` is insufficient to cover vesting-account bonus rewards — (`x/tieredrewards/migrations/v2/migrate.go`, `x/tieredrewards/keeper/force_exit.go`)

### Summary

`ForceFullExitWithDelegation` propagates `ErrInsufficientBonusPool` unhandled through the entire migration call stack. If the `RewardsPool` module account cannot cover accrued bonus rewards for any vesting-owned position at upgrade time, `Migrate1to2` returns an error, the upgrade handler aborts, and the chain halts permanently at the upgrade height — locking all user funds.

### Finding Description

**Full call chain with file/line references:**

1. `module.go:106` registers `migrator.Migrate1to2` as the v1→v2 migration handler. [1](#0-0) 

2. `Migrate1to2` delegates to `v2.Migrate`, which calls `exitVestedAccountsPositions`. [2](#0-1) 

3. `exitVestedAccountsPositions` iterates all positions, identifies vesting-account owners, and calls `ForceFullExitWithDelegation` for each — returning any error immediately. [3](#0-2) 

4. `ForceFullExitWithDelegation` calls `claimRewards` and returns its error unwrapped. [4](#0-3) 

5. `claimRewards` calls `processEventsAndClaimBonus` and returns its error. [5](#0-4) 

6. `processEventsAndClaimBonus` calls `sufficientBonusPoolBalance`, which returns `ErrInsufficientBonusPool` when the pool balance is less than the computed bonus. [6](#0-5) [7](#0-6) 

**The critical contrast:** `slashRedelegationPosition` in `slash.go` explicitly catches `ErrInsufficientBonusPool` and swallows it with a log message, precisely to prevent a chain halt: [8](#0-7) 

`ForceFullExitWithDelegation` has no equivalent guard. The error propagates all the way back to the Cosmos SDK module migration runner, which causes the upgrade handler to fail and the chain to halt.

### Impact Explanation

When the upgrade block is reached, the chain executes `RunMigrations`. If any module migration returns a non-nil error, the upgrade handler panics/returns an error and the chain halts at that height. No further blocks are produced. All user funds — staked tokens, positions, balances — are permanently inaccessible until a coordinated emergency patch and restart, which requires social consensus and is not guaranteed.

### Likelihood Explanation

The precondition is that the `RewardsPool` balance is less than the total accrued bonus for at least one vesting-owned position at upgrade time. This is realistic because:

- The pool is funded externally and can be depleted by normal bonus claims over time.
- A single large vesting-account position with a long accrual history (many validator events, high `BonusApy`, large shares) can accumulate a bonus that exceeds the remaining pool balance.
- An adversary holding such a position can time their own bonus claims to drain the pool to near-zero before the upgrade height, ensuring the migration fails.

No privileged access is required. The attacker only needs a vesting account with a tiered position — a normal user action.

### Recommendation

In `ForceFullExitWithDelegation`, handle `ErrInsufficientBonusPool` the same way `slashRedelegationPosition` does: catch the error, log it, skip the bonus payout, and continue the exit. The migration must complete regardless of pool state. Conceptually:

```go
posState, baseRewards, bonusRewards, err := k.claimRewards(ctx, posState)
if err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        logger.Error("force-exit: insufficient bonus pool, skipping bonus", "position_id", posID, "error", err)
        bonusRewards = sdk.NewCoins()
        // re-fetch posState without bonus claim, or proceed with zero bonus
    } else {
        return fmt.Errorf("claim rewards for position %d: %w", posID, err)
    }
}
```

Alternatively, refactor `claimRewards` to accept a flag that skips bonus when called from migration context.

### Proof of Concept

1. Set up an integration test with a vesting account owning a tiered position with large shares and a long bonded history (many validator events).
2. Drain the `RewardsPool` module account to zero (e.g., by having other positions claim all available bonus).
3. Trigger the v1→v2 migration via `Migrate1to2`.
4. Assert: without the fix, `Migrate1to2` returns a non-nil error wrapping `ErrInsufficientBonusPool`, and the upgrade handler aborts. With the fix, the migration completes, the position is exited (without bonus), and the chain continues.

```go
func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
```
