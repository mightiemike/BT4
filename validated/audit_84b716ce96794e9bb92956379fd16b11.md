### Title
`action_transfer_to_gas_key` Lacks Cumulative Cap Check, Allowing Gas Key Balance to Exceed `MAX_BALANCE_TO_BURN` and Permanently Block `DeleteKey`/`DeleteAccount` - (File: `runtime/runtime/src/access_keys.rs`)

### Summary

`action_transfer_to_gas_key` unconditionally accumulates deposits into a gas key's balance with no upper-bound check. The protocol cap `GasKeyInfo::MAX_BALANCE_TO_BURN = 1 NEAR` is only enforced at deletion time in `delete_gas_key` and `action_delete_account`. Any number of `TransferToGasKey` calls can push the balance above 1 NEAR, after which `DeleteKey` and `DeleteAccount` both return `GasKeyBalanceTooHigh` and are permanently blocked for that key/account until the balance is manually reduced via `WithdrawFromGasKey`. In the worst case — an account that holds only `GasKeyFunctionCall` keys (no full-access key, no `GasKeyFullAccess` key) — `WithdrawFromGasKey` is also unavailable, making the excess balance permanently unrecoverable.

### Impact Explanation

**Immediate (Medium):** Any account that accumulates gas key balance > 1 NEAR through multiple `TransferToGasKey` calls cannot delete the gas key (`GasKeyBalanceTooHigh`) and cannot delete the account if the aggregate gas key balance exceeds 1 NEAR. The user must first issue `WithdrawFromGasKey` to reduce the balance.

**Worst case (High):** An account that holds only `GasKeyFunctionCall` keys — no full-access key and no `GasKeyFullAccess` key — cannot issue `WithdrawFromGasKey` (which requires full-access or gas-key-full-access permission). If such an account's gas key balance exceeds 1 NEAR through incremental `TransferToGasKey` calls, the excess balance is permanently unrecoverable and the key/account can never be deleted. The `check_actor_permissions` function does not restrict `WithdrawFromGasKey` by actor, but the transaction signing key must have sufficient permission to issue it. [6](#0-5) 

### Likelihood Explanation

`TransferToGasKey` is callable both via direct transactions and via the `promise_batch_action_transfer_to_gas_key` host function from contract execution. A contract can create a receipt targeting any account and include a `TransferToGasKey` action, funding that account's gas key beyond 1 NEAR. An account owner who issues multiple funding transactions without tracking the cumulative balance will silently cross the threshold. The protocol provides no warning at deposit time.

### Recommendation

Add a cumulative cap check inside `action_transfer_to_gas_key` before updating the balance:

```rust
// runtime/runtime/src/access_keys.rs
let new_balance = gas_key_info.balance.checked_add(action.deposit).ok_or_else(|| ...)?;
if new_balance > GasKeyInfo::MAX_BALANCE_TO_BURN {
    result.result = Err(ActionErrorKind::GasKeyBalanceTooHigh {
        account_id: account_id.clone(),
        public_key: Some(Box::new(action.public_key.clone())),
        balance: new_balance,
    }.into());
    return Ok(());
}
gas_key_info.balance = new_balance;
```

This mirrors the pattern used in `delete_gas_key` and ensures the invariant is enforced at the point of deposit rather than only at deletion.

### Proof of Concept

1. Create an account with a `GasKeyFunctionCall` key only (no full-access key).
2. Issue two `TransferToGasKey` transactions, each depositing 0.6 NEAR → cumulative balance = 1.2 NEAR > `MAX_BALANCE_TO_BURN`.
3. Attempt `DeleteKey` → returns `GasKeyBalanceTooHigh { balance: 1.2 NEAR }`.
4. Attempt `DeleteAccount` → returns `GasKeyBalanceTooHigh { balance: 1.2 NEAR }`.
5. Attempt `WithdrawFromGasKey` → rejected at transaction verification because the only signing key is `GasKeyFunctionCall`, which lacks permission to issue `WithdrawFromGasKey`.
6. The 1.2 NEAR in the gas key is permanently locked; the key and account cannot be removed from state.

The exact divergent value is `GasKeyInfo::MAX_BALANCE_TO_BURN = Balance::from_near(1)` at `core/primitives-core/src/account.rs:554`, which is enforced only at `runtime/runtime/src/access_keys.rs:103` and `runtime/runtime/src/actions.rs:340`, but never at `runtime/runtime/src/access_keys.rs:281`. pub(crate) fn action_transfer_to_gas_key(
    state_update: &mut TrieUpdate,
    result: &mut ActionResult,
    account_id: &AccountId,
    action: &TransferToGasKeyAction,
) -> Result<(), RuntimeError> {
    let Some(mut access_key) = get_access_key(state_update, account_id, &action.public_key)? else {
        result.result = Err(ActionErrorKind::GasKeyDoesNotExist {
            account_id: account_id.clone(),
            public_key: Box::new(action.public_key.clone()),
        }
        .into());
        return Ok(());
    };
    let Some(gas_key_info) = access_key.gas_key_info_mut() else {
        // Key exists but is not a gas key
        result.result = Err(ActionErrorKind::GasKeyDoesNotExist {
            account_id: account_id.clone(),
            public_key: Box::new(action.public_key.clone()),
        }
        .into());
        return Ok(());
    };

    gas_key_info.balance = gas_key_info.balance.checked_add(action.deposit).ok_or_else(|| {
        RuntimeError::StorageError(StorageError::StorageInconsistentState(
            "gas key balance integer overflow".to_string(),
        ))
    })?;
    set_access_key(state_update, account_id.clone(), action.public_key.clone(), &access_key);
    Ok(())
}
