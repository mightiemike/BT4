### Title
SHA-1 Used for Subtree Identity in `TreeCache` Back-Reference Serialization — (`File: src/serde/tree_cache.rs`)

---

### Summary

`TreeCache` uses SHA-1 — a cryptographically broken hash function — to identify and deduplicate CLVM subtrees during back-reference compressed serialization. A SHA-1 collision between two distinct atoms causes the serializer to treat them as identical, emitting a back-reference (`0xfe` + path) that points to the wrong subtree. When deserialized, the resulting CLVM program differs from the original, producing a consensus divergence.

---

### Finding Description

In `src/serde/tree_cache.rs`, the internal `hash_atom` function computes a 20-byte SHA-1 digest of each atom:

```rust
fn hash_atom(salt: &[u8], blob: &[u8]) -> Bytes20 {
    let mut ctx = Sha1::default();
    ctx.update(salt);
    ctx.update(blob);
    ctx.finalize().into()
}
``` [1](#0-0) 

These 20-byte digests are stored in `atom_lookup`, a `HashMap<Bytes20, u32, RandomState>`, and used to merge identical atoms into a single `NodeEntry`: [2](#0-1) 

When `update()` traverses a new atom, it computes its SHA-1 hash and checks `atom_lookup`. If a matching entry exists, the new atom is silently aliased to the existing `NodeEntry`: [3](#0-2) 

If two distinct atoms A and B produce the same SHA-1 hash (a collision), they share a `NodeEntry`. Because `pair_lookup` keys are built from `NodeEntry` indices (`(left_idx << 32) | right_idx`), the collision propagates upward: any pair containing A and any pair containing B at the same position also share a `NodeEntry`. The `find_path()` search then returns a path to the wrong subtree, and the serializer emits a back-reference that decodes to a different atom than intended. [4](#0-3) 

The code itself acknowledges the weakness:

> "We compute hash-trees using SHA-1 in order to determine whether the trees are identical or not. To mitigate malicious SHA-1 hash collisions, we salt the hashes" [5](#0-4) 

The salt is 8 bytes of random data generated per `TreeCache::new()` call: [6](#0-5) 

The rest of the codebase exclusively uses SHA-256 for all cryptographic hashing (tree hashing, serialization, operator implementations): [7](#0-6) [8](#0-7) 

The `Serializer` struct (the public API that wraps `TreeCache`) is the entry point for incremental back-reference serialization of CLVM trees, including block generators: [9](#0-8) 

---

### Impact Explanation

If a SHA-1 collision occurs between two distinct atoms in a CLVM tree being serialized by `Serializer`, the output byte stream contains a `0xfe` back-reference that resolves to the wrong atom upon deserialization. The deserialized CLVM program is structurally different from the original. In the context of block generators, this is a **consensus divergence**: nodes that serialize and re-deserialize the generator would execute a different program, potentially disagreeing on transaction validity. The collision also propagates to all ancestor pairs via `pair_lookup`, amplifying the corruption.

**Impact: 3**

---

### Likelihood Explanation

The 8-byte random salt substantially raises the bar: an attacker cannot use known SHA-1 chosen-prefix collision techniques (e.g., SHAttered) because the salt is unknown at collision-crafting time. A birthday attack against SHA-1's 160-bit output requires approximately 2^80 operations, which is computationally infeasible today. However, SHA-1 is a formally broken algorithm (NIST deprecated it in 2011; practical chosen-prefix collisions were demonstrated in 2020), and the rest of the codebase uses SHA-256 for all equivalent purposes. The risk is low but non-zero, and the design relies on a weak primitive with a partial mitigation rather than a sound one.

**Likelihood: 2**

---

### Recommendation

Replace SHA-1 with SHA-256 in `hash_atom` and update `Bytes20` / `atom_lookup` to use 32-byte digests. This eliminates the need for the salt mitigation entirely, aligns with the rest of the codebase, and removes reliance on a broken primitive:

```rust
use chia_sha2::Sha256;
type Bytes32 = [u8; 32];

fn hash_atom(blob: &[u8]) -> Bytes32 {
    let mut ctx = Sha256::new();
    ctx.update(blob);
    ctx.finalize()
}
```

The `salt` field and `RandomState` hasher on `atom_lookup` can then be removed, and `atom_lookup` can use a standard `HashMap<Bytes32, u32>`.

---

### Proof of Concept

1. Attacker crafts two distinct atoms `A` and `B` such that `SHA1(salt || A) == SHA1(salt || B)` for the session salt (requires ~2^80 work, or a future SHA-1 break).
2. Both atoms appear in a CLVM tree submitted for serialization via `Serializer::add()`.
3. In `TreeCache::update()`, atom `B` is processed after `A`. `atom_lookup.entry(hash)` finds `A`'s `NodeEntry` and aliases `B` to it.
4. `find_path(node_B)` returns a path to `node_A`'s position in the parse stack.
5. The serializer emits `0xfe <path_to_A>` where `B` should appear.
6. Any deserializer following the Chia back-reference spec reconstructs a program containing `A` at `B`'s position — a different program than was serialized.
7. Nodes that independently serialize and deserialize the block generator execute different CLVM programs, causing consensus divergence. [10](#0-9) [11](#0-10) [12](#0-11)
