Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` (Position Recipient) Instead of `sender` (Actual Payer), Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual token payer / `msg.sender` of `addLiquidity`) and checks only `owner` (the position recipient). Because `MetricOmmPool.addLiquidity` explicitly supports an operator pattern where `msg.sender ≠ owner`, any address not on the allowlist can bypass the gate by calling `pool.addLiquidity(allowedAddress, ...)`, routing the LP position to an allowed owner while paying tokens themselves. The pool admin's curation policy is completely defeated.

## Finding Description
`MetricOmmPool.addLiquidity` passes both actors to the extension hook:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
//                  ^^^^^^^^^^  ^^^^^
//                  sender      owner (position recipient)
``` [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` correctly encodes both `sender` and `owner` into the call payload forwarded to the extension:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [2](#0-1) 

However, `DepositAllowlistExtension.beforeAddLiquidity` discards `sender` (unnamed `address,`) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

The interface NatDoc for `addLiquidity` explicitly states the operator pattern is supported (`msg.sender` pays but need not equal `owner`): [4](#0-3) 

The contract's own NatDoc states it *"Gates `addLiquidity` by depositor address"* — the depositor is the payer (`sender`), not the position recipient (`owner`): [5](#0-4) 

The `allowedDepositor` mapping is keyed by `depositor` address, and the admin-facing setter `setAllowedToDeposit` takes a `depositor` parameter — semantically the payer, not the recipient: [6](#0-5) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, ...)` explicitly separates payer (`msg.sender`) from position recipient (`owner`), confirming the operator pattern is a first-class supported flow: [7](#0-6) 

The `allowedDepositor` mapping is public, so any attacker can enumerate allowed addresses on-chain at zero cost: [8](#0-7) 

## Impact Explanation
A pool admin deploys a curated pool with `DepositAllowlistExtension` to restrict which addresses can deposit tokens. Because the check is on `owner` rather than `sender`, any address not on the allowlist can:

1. Call `pool.addLiquidity(allowedAddress, salt, deltas, callbackData, extensionData)` directly.
2. The extension evaluates `allowedDepositor[pool][allowedAddress]` → `true` → check passes.
3. The unauthorized caller (`sender`) pays the tokens and alters pool state (bin totals, cursor position, price impact).
4. LP shares are credited to `allowedAddress`.

The pool admin's curation policy — preventing specific addresses from depositing — is completely defeated. The unauthorized depositor changes pool composition and can influence oracle-anchored pricing for subsequent swaps. This constitutes broken core pool functionality (deposit allowlist) with direct fund-impact potential on LP positions, meeting the "broken core pool functionality causing loss of funds" impact criterion.

## Likelihood Explanation
- The operator pattern (`msg.sender ≠ owner`) is explicitly documented and supported by the pool interface.
- `MetricOmmPoolLiquidityAdder` uses this pattern in `addLiquidityExactShares(pool, owner, ...)`, passing `msg.sender` as payer and a caller-supplied `owner` as position recipient.
- Any address can call `pool.addLiquidity` directly with an arbitrary `owner`.
- No special privilege or setup is required beyond knowing one allowed address, which is publicly readable from the `allowedDepositor` mapping.
- The bypass is deterministic and repeatable with zero preconditions beyond having tokens to pay.

## Recommendation
Change `beforeAddLiquidity` to check `sender` (the actual payer/depositor) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This aligns the check with the actor who actually pays tokens and drives pool state changes, consistent with the analogous `SwapAllowlistExtension.beforeSwap` which correctly checks `sender`: [9](#0-8) 

## Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][alice] = true
  bob is NOT on the allowlist

Attack (direct pool call):
  bob calls pool.addLiquidity(
      owner        = alice,   // allowed address
      salt         = 0,
      deltas       = <bob's desired bins/shares>,
      callbackData = <bob's payment callback>,
      extensionData = ""
  )

Extension evaluation:
  sender (arg 1, ignored) = bob
  owner  (arg 2, checked) = alice
  allowedDepositor[pool][alice] == true  →  check passes

Result:
  bob pays tokens into the pool (pool state changes: bin totals, cursor)
  alice receives LP shares she did not request
  bob has bypassed the deposit allowlist entirely

Attack (via MetricOmmPoolLiquidityAdder):
  bob calls adder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, "")
  adder calls pool.addLiquidity(alice, salt, deltas, abi.encode(KIND_PAY), "")
  pool calls _beforeAddLiquidity(adder_address, alice, ...)
  extension checks allowedDepositor[pool][alice] == true  →  passes
  adder callback pulls tokens from bob (msg.sender of adder call)
  alice receives LP shares; bob pays; allowlist bypassed
```
