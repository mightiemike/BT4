### Title
Missing Cross-Field Invariant Validation in `StatelessTransactionValidatorConfig` Sierra Version Bounds Causes Permanent Rejection of All Declare Transactions - (File: `crates/apollo_gateway_config/src/config.rs`)

### Summary
`StatelessTransactionValidatorConfig` holds two independently settable fields, `min_sierra_version` and `max_sierra_version`, but no cross-field validation enforces the invariant `min_sierra_version <= max_sierra_version`. If an operator misconfigures these fields in inverted order, `validate_sierra_version` constructs an impossible range and permanently rejects every declare transaction submitted to the gateway.

### Finding Description
`StatelessTransactionValidatorConfig` derives the `Validate` trait but carries no `#[validate(custom = ...)]` or any other cross-field constraint on its two Sierra version bound fields:

```rust
// crates/apollo_gateway_config/src/config.rs, lines 166-186
#[derive(Clone, Debug, Deserialize, PartialEq, Serialize, Validate)]
pub struct StatelessTransactionValidatorConfig {
    ...
    pub min_sierra_version: VersionId,
    pub max_sierra_version: VersionId,
    ...
}
``` [1](#0-0) 

At runtime, `validate_sierra_version` enforces the range with a single compound condition:

```rust
if self.config.min_sierra_version <= sierra_version && sierra_version <= max_sierra_version {
    return Ok(());
}
``` [2](#0-1) 

When `min_sierra_version > max_sierra_version`, no value of `sierra_version` can simultaneously satisfy both halves of the conjunction. Every declare transaction therefore hits the error path:

```rust
Err(StatelessTransactionValidatorError::UnsupportedSierraVersion { ... })
``` [3](#0-2) 

The `validate_declare_tx` path calls `validate_sierra_version` unconditionally for every `RpcDeclareTransaction::V3`, so the broken range silently blocks the entire declare surface. [4](#0-3) 

The node-level config validation (`validate_node_config`) delegates to the `Validate` derive, which only checks per-field constraints; no cross-member check for the sierra version pair exists anywhere in the call chain. [5](#0-4) 

### Impact Explanation
**High.** A misconfigured `min_sierra_version > max_sierra_version` causes the gateway to reject every valid declare transaction before sequencing. No contract class can be declared on the network for as long as the misconfiguration persists. This maps directly to the allowed impact: *"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation
**Low.** Exploiting this requires an operator fat-finger or a large configuration error when setting the two version fields — exactly the same likelihood characterization as the original CrowdSale report. The default values (`min = 1.1.0`, `max = 1.9.MAX`) are correct; the risk arises only on misconfiguration. [6](#0-5) 

### Recommendation
Add a cross-field validation method to `StatelessTransactionValidatorConfig` (via `#[validate(custom = "validate_sierra_version_range")]` or an equivalent `Validate` impl) that asserts `min_sierra_version <= max_sierra_version` and returns a `ValidationError` if violated. This check should be exercised by the existing `validate_node_config` / `config_validate` call at node startup so that a misconfigured node fails fast rather than silently rejecting all declares.

### Proof of Concept
1. Set the gateway config with inverted Sierra version bounds, e.g.:
   ```json
   "gateway_config.static_config.stateless_tx_validator_config.min_sierra_version.minor": 9,
   "gateway_config.static_config.stateless_tx_validator_config.max_sierra_version.minor": 1
   ``` [7](#0-6) 

2. Start the node. `validate_node_config` succeeds because no cross-field check exists. [1](#0-0) 

3. Submit any `RpcDeclareTransaction::V3` with a Sierra program whose embedded version is, e.g., `1.5.0` (well within the intended range).

4. `validate_sierra_version` evaluates `1.9.0 <= 1.5.0` → `false`; the transaction is rejected with `UnsupportedSierraVersion { version: 1.5.0, min: 1.9.0, max: 1.1.MAX }`. [8](#0-7) 

5. Every declare transaction, regardless of its actual Sierra version, is rejected for the lifetime of the misconfigured node.
