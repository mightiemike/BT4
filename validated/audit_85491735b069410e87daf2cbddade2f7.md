Audit Report

## Title
SwapAllowlistExtension Gates on Router Address Instead of End-User, Allowing Any User to Bypass the Allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is the pool's `msg.sender`. When `MetricOmmSimpleRouter` mediates a swap, it calls `pool.swap()` directly, making the router the pool's `msg.sender`. Any pool that allowlists the router to support normal UX inadvertently grants swap access to every user on the network, completely voiding per-user access control.

## Finding Description
The full call chain is confirmed in production code:

**Step 1 — Extension check** (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`, L37):
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool and `sender` is the value forwarded from the pool.

**Step 2 — Pool passes its own `msg.sender` as `sender`** (`metric-core/contracts/MetricOmmPool.sol`, L230–231):
```solidity
_beforeSwap(
    msg.sender,   // becomes `sender` in the extension
    ...
);
```
Confirmed via `ExtensionCalling._beforeSwap` (`metric-core/contracts/ExtensionCalling.sol`, L149–176), which encodes `sender` as the first argument to `IMetricOmmExtensions.beforeSwap`.

**Step 3 — Router calls `pool.swap()` directly** (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`, L71–80):
```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```
The router is the `msg.sender` of the `pool.swap()` call. The same pattern applies to `exactInput` (L103–112), `exactOutputSingle` (L135–137), and `exactOutput` (L165–181).

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`. The `extensionData` field is caller-controlled and is not used by the extension for identity checks, so it cannot close this gap. No existing guard in the extension or pool recovers the original end-user identity.

## Impact Explanation
A curated pool (KYC-only, institutional-only, or regulatory-restricted) deploying `SwapAllowlistExtension` must allowlist the router to support normal UX. Once the router is allowlisted, any non-allowlisted EOA can call `router.exactInputSingle()` targeting that pool and execute a swap the allowlist was intended to block. The allowlist guard fails open for all router-mediated flows, which is the dominant swap path for end users. This constitutes broken core pool functionality with direct fund-flow and regulatory consequences — High severity.

## Likelihood Explanation
The router is a standard, publicly deployed periphery contract. Any pool supporting normal UX must allowlist it. The bypass requires zero privilege: any EOA calls `exactInputSingle` with the target pool address. The pool admin has no on-chain signal that the allowlist is being bypassed; the extension simply sees an allowlisted address (the router). The condition is trivially reachable the moment the router is allowlisted, which is the expected operational state.

## Recommendation
The extension must gate on the economic actor, not the immediate caller. Two viable options:

1. **Trusted forwarder / signed attestation**: The router includes the original `msg.sender` in `extensionData`; the extension verifies a signature or checks a trusted-forwarder registry before accepting the claimed identity.
2. **Router registry with attestation**: The extension maintains a registry of trusted routers; when `msg.sender` (the pool's caller) is a known router, the extension requires the router to attest the real user via a separate trusted channel.

As a safe interim measure, document and enforce that `SwapAllowlistExtension` is incompatible with router-mediated swaps by reverting if the pool's caller is not a direct EOA — though this breaks composability and is not the recommended long-term fix.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, router, true)   // required for router UX
  - Pool admin does NOT allowlist attacker EOA

Attack:
  - attacker (non-allowlisted EOA) calls:
      router.exactInputSingle({
          pool: curated_pool,
          recipient: attacker,
          zeroForOne: true,
          amountIn: X,
          ...
      })

  - router calls pool.swap(...) with msg.sender = router
  - pool calls _beforeSwap(sender=router, ...)
  - ExtensionCalling encodes sender=router and calls extension.beforeSwap(router, ...)
  - extension checks: allowedSwapper[pool][router] == true  ✓
  - swap executes; attacker receives output tokens

Result:
  - Non-allowlisted attacker successfully swaps on a curated pool.
  - The allowlist guard is completely bypassed for all router-mediated flows.

Foundry test plan:
  1. Deploy pool + SwapAllowlistExtension; configure extension on pool.
  2. Admin calls setAllowedToSwap(pool, router, true); does NOT allowlist attacker.
  3. vm.prank(attacker); router.exactInputSingle({pool: pool, ...});
  4. Assert swap succeeds (no NotAllowedToSwap revert) and attacker receives tokens.
```
