### Title
Deprecated `verify_transaction_inclusion` (v1) Remains Publicly Callable Without Coinbase Proof Check, Enabling 64-Byte Merkle Forgery — (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (v1) is still a live, unpermissioned public NEAR contract method. It omits the coinbase Merkle proof check that `verify_transaction_inclusion_v2` introduced to close the 64-byte transaction Merkle forgery attack. Any unprivileged NEAR caller can invoke v1 directly, bypassing the security property that v2 was designed to enforce, and obtain a `true` SPV verification result for a fabricated transaction.

### Finding Description

`verify_transaction_inclusion` (v1) computes a Merkle root from a caller-supplied `tx_id` and `merkle_proof` and compares it to the stored block's `merkle_root`: [1](#0-0) 

`compute_root_from_merkle_proof` is a pure path-traversal function with no constraint that `tx_id` must be a leaf node: [2](#0-1) 

Because any internal Merkle tree node also has a valid proof path to the root, an attacker can supply an internal node as `tx_id` and a matching sub-path as `merkle_proof`, and the comparison will succeed. The function's own doc comment acknowledges this: [3](#0-2) 

`verify_transaction_inclusion_v2` closes this gap by first verifying a coinbase proof at index 0 with the same proof length, which anchors the tree depth and forces `tx_id` to be a genuine leaf: [4](#0-3) 

However, v1 carries only a Rust `#[deprecated]` attribute, which is a compile-time hint only. It does **not** gate or remove the NEAR method. The function is `pub`, carries no `#[private]` or `#[trusted_relayer]` guard, and is protected only by `#[pause]` (off by default): [5](#0-4) 

Any NEAR account can call `verify_transaction_inclusion` directly, completely bypassing the coinbase-proof requirement that v2 enforces.

### Impact Explanation

A recipient dApp that calls `verify_transaction_inclusion` (v1) to gate fund releases or cross-chain actions receives a `true` result for a fabricated transaction. The attacker does not need to mine a block or control any privileged role — they only need a real block already in the contract's mainchain and knowledge of its Merkle tree structure (publicly available from any Bitcoin node). This constitutes **proof-verification forgery**: the contract's canonical SPV result is corrupted for any caller that uses v1.

### Likelihood Explanation

The 64-byte transaction attack is a well-documented Bitcoin Merkle tree property (referenced in the contract's own deprecation notice). Constructing a valid internal-node proof requires only reading the public Merkle tree of any confirmed block. No privileged access, key material, or social engineering is needed. Any NEAR account is a valid entry point.

### Recommendation

Remove `verify_transaction_inclusion` (v1) as a callable NEAR method entirely, or add `#[private]` to prevent external invocation. The `#[deprecated]` attribute provides no runtime enforcement. All callers must be migrated to `verify_transaction_inclusion_v2`.

### Proof of Concept

1. Identify any block `B` in the contract's mainchain (e.g., via `get_last_block_header`).
2. Obtain `B`'s full transaction list from a Bitcoin node and compute its Merkle tree.
3. Select any internal node `N` at depth `d` (e.g., the left child of the root). `N` is a 32-byte hash, indistinguishable from a transaction hash.
4. Construct `merkle_proof` of length `d` — the sibling path from `N` up to the Merkle root. This path is valid by construction.
5. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_block_blockhash = B.hash`, `tx_index = position_of_N`, `merkle_proof = proof`, `confirmations = 1`.
6. `compute_root_from_merkle_proof(N, position, proof)` returns `B.merkle_root`; the function returns `true`.
7. Any dApp gating on this result treats the fabricated transaction as confirmed. [6](#0-5) [2](#0-1)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
