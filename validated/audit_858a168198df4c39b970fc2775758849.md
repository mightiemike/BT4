### Title
Health Checks in `withdrawCollateral` and `liquidateSubaccountImpl` Use Stale Spot Interest Accumulators — (File: `core/contracts/Clearinghouse.sol`, `core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` and `ClearinghouseLiq.liquidateSubaccountImpl` invoke `getHealth` / `isUnderMaintenance` without first calling `spotEngine.updateStates`. The health computation reads `state.cumulativeBorrowsMultiplierX18` directly from `states[productId]` storage, which is only refreshed by `SpotTick` transactions. If `SpotTick` has not been submitted recently, the multiplier is stale, causing borrow balances to be understated and accounts to appear healthier than they actually are.

---

### Finding Description

Spot balances in `SpotEngineState` are stored as normalized values (`amountNormalized`) and converted to real balances by multiplying by `cumulativeBorrowsMultiplierX18` (for negative/borrow positions) or `cumulativeDepositsMultiplierX18` (for positive/deposit positions). These multipliers compound interest over time and are updated exclusively inside `updateStates`, which is only reachable via a `SpotTick` transaction submitted by the sequencer. [1](#0-0) 

The conversion from normalized to real balance is: [2](#0-1) 

`_getBalance`, called during every health contribution calculation, reads `states[productId]` directly from storage without triggering any update: [3](#0-2) 

`_calculateProductHealth` in `BaseEngine` calls `_getBalance` for every product in the subaccount's bitmap: [4](#0-3) 

`Clearinghouse.withdrawCollateral` applies the balance delta for the withdrawn product and then immediately calls `getHealth` — with no call to `spotEngine.updateStates` anywhere in the function: [5](#0-4) 

`ClearinghouseLiq.liquidateSubaccountImpl` calls `isUnderMaintenance` as its first eligibility gate, again without any preceding `updateStates`: [6](#0-5) 

`isUnderMaintenance` delegates to `getHealth` via `getHealthFromClearinghouse`: [7](#0-6) 

`getHealth` aggregates contributions from all spot products using the stale multipliers: [8](#0-7) 

The `SpotTick` handler in `EndpointTx` is the only code path that calls `spotEngine.updateStates`, and it is a separate, independently sequenced transaction with no on-chain ordering guarantee relative to `WithdrawCollateral` or `LiquidateSubaccount`: [9](#0-8) 

The slow-mode `WithdrawCollateral` handler in `processSlowModeTransactionImpl` calls `clearinghouse.withdrawCollateral` directly with no `updateStates` call: [10](#0-9) 

---

### Impact Explanation

**Withdrawal over-collateral extraction:** A user holding a borrow position (negative `amountNormalized`) whose `cumulativeBorrowsMultiplierX18` has not been refreshed will have their liability understated. The health check in `withdrawCollateral` passes at a lower collateral level than the protocol intends, allowing the user to extract collateral that should be locked, leaving the protocol undercollateralized.

**Liquidation blocking:** `isUnderMaintenance` in `liquidateSubaccountImpl` uses the same stale multiplier. A borrower whose true maintenance health (with accrued interest) is negative will appear healthy if `SpotTick` has not been called recently. The `require(isUnderMaintenance(...), ERR_NOT_LIQUIDATABLE)` check reverts, blocking the liquidation and allowing bad debt to accumulate — the exact impact class described in the reference report.

---

### Likelihood Explanation

The sequencer is expected to submit `SpotTick` periodically, but there is no on-chain enforcement that `SpotTick` precedes `WithdrawCollateral` or `LiquidateSubaccount` in the same batch. Two concrete trigger paths exist:

1. **Slow-mode self-execution:** `processSlowModeTransactionImpl` is `public`. After the 3-day slow-mode delay, a user can execute their own `WithdrawCollateral` transaction directly, without the sequencer having submitted a `SpotTick` in the same call. The user controls the exact moment of execution and can choose a window where the last `SpotTick` is maximally stale.

2. **Sequencer downtime / batch gap:** During any period of sequencer delay or downtime, interest continues to accrue on-chain but `cumulativeBorrowsMultiplierX18` is not updated. When the sequencer resumes and processes a backlog, if `WithdrawCollateral` or `LiquidateSubaccount` transactions are processed before the next `SpotTick`, the stale-state window is exploitable.

---

### Recommendation

Call `spotEngine.updateStates(0)` (or with the elapsed `dt`) at the top of `withdrawCollateral` and at the top of `liquidateSubaccountImpl` before any health check is performed. This mirrors the remediation in the reference patch (`c3dbdd3`): update all asset states before conducting the health check.

Alternatively, enforce at the sequencer level that a `SpotTick` transaction always immediately precedes any `WithdrawCollateral` or `LiquidateSubaccount` transaction in the same submission batch, and add an on-chain staleness guard (e.g., `require(block.timestamp - times.spotTime < MAX_STALENESS)`) inside `getHealth`.

---

### Proof of Concept

**Over-withdrawal scenario:**

1. Alice borrows 10,000 USDC (product A) — `balances[A][alice].amountNormalized` is negative.
2. Alice deposits 15 ETH (product B) as collateral.
3. The sequencer stops submitting `SpotTick` for 6 hours. `states[A].cumulativeBorrowsMultiplierX18` remains at its last-updated value `M_stale`, while the true value should be `M_actual > M_stale` due to accrued interest.
4. Alice submits a slow-mode `WithdrawCollateral` for ETH.
5. After 3 days, Alice calls `processSlowModeTransactionImpl` herself (no `SpotTick` in the same call).
6. `withdrawCollateral` calls `getHealth(alice, INITIAL)`.
7. `_getBalance(A, alice)` returns `amountNormalized * M_stale`, understating Alice's USDC liability by the accrued interest delta `amountNormalized * (M_actual - M_stale)`.
8. Alice's health appears positive; the `require(getHealth(...) >= 0)` passes.
9. Alice withdraws ETH that should have been locked as collateral against the true (larger) borrow balance.

**Liquidation-blocking scenario:**

1. Bob borrows USDC to the edge of maintenance health.
2. `SpotTick` is not called for several hours; accrued interest pushes Bob's true maintenance health below zero.
3. A liquidator submits `LiquidateSubaccount` targeting Bob.
4. `liquidateSubaccountImpl` calls `isUnderMaintenance(bob)` → `getHealth(bob, MAINTENANCE)`.
5. `_getBalance` uses stale `cumulativeBorrowsMultiplierX18`, reporting Bob's borrow as smaller than it is.
6. `getHealth` returns a non-negative value; `isUnderMaintenance` returns `false`.
7. `require(isUnderMaintenance(...), ERR_NOT_LIQUIDATABLE)` reverts, blocking the liquidation.
8. Bob's bad debt grows until `SpotTick` is eventually called, by which point the protocol may be further exposed.

    function updateStates(uint128 dt) external onlyEndpoint {
        State memory quoteState;
        require(dt < 7 * SECONDS_PER_DAY, ERR_INVALID_TIME);
        for (uint32 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            if (productId == NLP_PRODUCT_ID) {
                continue;

            if (productId == QUOTE_PRODUCT_ID) {

            }
            if (state.totalDepositsNormalized == 0) {
                continue;
            }
            _updateState(productId, state, dt);
            _setState(productId, state);
        }
    }

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {

        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
