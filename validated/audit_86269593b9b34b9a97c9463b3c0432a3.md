### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery Bypass — (`contract/src/lib.rs`)

### Summary
The deprecated `verify_transaction_inclusion` function is still exposed as a live public contract method. Any unprivileged NEAR caller can invoke it directly, bypassing the coinbase-proof guard introduced in `verify_transaction_inclusion_v2`, and obtain a `true` proof-verification result for a transaction that was never included in a Bitcoin block.

### Finding Description
`verify_transaction_inclusion` was deprecated at version 0.5.0 and superseded by `verify_transaction_inclusion_v2`, which adds a coinbase Merkle proof check to close the 64-byte transaction Merkle proof forgery vulnerability (https://www.bitmex.com/blog/64-Byte-Transactions). Despite the deprecation, the function retains its `pub` visibility and its `#[pause]` attribute, making it a fully live, externally callable NEAR contract method.

The function's own documentation warns:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [1](#0-0) 

The security fix in `verify_transaction_inclusion_v2` works by first verifying that the coinbase transaction (at index 0) produces the correct Merkle root, which prevents an attacker from substituting an internal node hash as a leaf `tx_id`. [2](#0-1) 

Because `verify_transaction_inclusion` is still public, a caller can skip `verify_transaction_inclusion_v2` entirely and call the deprecated endpoint directly with a crafted `tx_id` that is an internal Merkle tree node, receiving `true` without any coinbase validation. [3](#0-2) 

### Impact Explanation
The corrupted value is the **proof result** returned by `verify_transaction_inclusion`. A downstream NEAR contract that calls this method to gate a cross-chain action (e.g., releasing funds, minting tokens, recording a Bitcoin payment) will accept a forged proof as valid. The attacker can claim that an arbitrary 32-byte value is a confirmed Bitcoin transaction without ever broadcasting or mining such a transaction.

### Likelihood Explanation
The entry point is a public, permissionless NEAR contract call. No privileged role, leaked key, or social engineering is required. The 64-byte Merkle forgery technique is well-documented and the required inputs (a real block hash in the mainchain and its Merkle tree structure) are publicly observable on-chain. Any caller who knows the block's Merkle tree can construct a valid forgery.

### Recommendation
Remove the `pub` visibility from `verify_transaction_inclusion` or gate it with a role that prevents external calls. Since `verify_transaction_inclusion_v2` calls it internally via `#[allow(deprecated)]`, the simplest fix is to make it `pub(crate)` or `fn` (private), so it can only be called from within the contract module itself and not as an externally dispatched NEAR method. [4](#0-3) 

### Proof of Concept

1. Identify any confirmed Bitcoin block whose hash is stored in `mainchain_header_to_height` (publicly queryable via `get_block_hash_by_height`).
2. Obtain the block's full transaction list and compute its Merkle tree. Pick any internal node `N` at depth `d` with known sibling path.
3. Construct `ProofArgs`:
   - `tx_id` = `N` (an internal node, not a real transaction hash)
   - `tx_block_blockhash` = the confirmed block hash
   - `tx_index` = the position of `N` at depth `d` (treated as a leaf index)
   - `merkle_proof` = the sibling path from `N` up to the Merkle root (length `d`)
   - `confirmations` = any value ≤ `gc_threshold`
4. Call `verify_transaction_inclusion(args)` directly on the contract.
5. The call returns `true` because `compute_root_from_merkle_proof(N, index, proof) == merkle_root` holds by construction, with no coinbase check performed. [5](#0-4)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
