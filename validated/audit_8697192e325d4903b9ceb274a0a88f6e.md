### Title
Stale Chain Tip Accepted Without Freshness Check Allows False SPV Proof Confirmation — (`contract/src/lib.rs`)

---

### Summary

The `BtcLightClient` contract stores no timestamp for when its chain tip was last updated, and neither `verify_transaction_inclusion` nor `verify_transaction_inclusion_v2` checks whether the stored `mainchain_tip_blockhash` is fresh. If the relayer stops submitting headers, the contract's view of the Bitcoin chain becomes arbitrarily stale. An unprivileged NEAR caller can then submit a valid SPV proof for a transaction that was in a block which has since been reorganized out of the real Bitcoin chain, and the contract will return `true`, causing downstream bridge or settlement contracts to release funds against a non-existent Bitcoin transaction.

---

### Finding Description

The `BtcLightClient` struct contains no field recording when the chain tip was last updated:

```rust
pub struct BtcLightClient {
    mainchain_height_to_header: LookupMap<u64, H256>,
    mainchain_header_to_height: LookupMap<H256, u64>,
    mainchain_tip_blockhash: H256,          // no last_updated_at
    mainchain_initial_blockhash: H256,
    headers_pool: LookupMap<H256, ExtendedHeader>,
    skip_pow_verification: bool,
    gc_threshold: u64,
    network: Network,
}
``` [1](#0-0) 

Both verification entry points compute confirmation depth purely against the stored tip, with no staleness guard:

```rust
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
    let heaviest_block_header = self
        .headers_pool
        .get(&self.mainchain_tip_blockhash)   // stale tip used directly
        ...
    require!(
        (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
            >= args.confirmations,
        "Not enough blocks confirmed"
    );
    ...
    merkle_tools::compute_root_from_merkle_proof(...) == header.block_header.merkle_root
}
``` [2](#0-1) 

`verify_transaction_inclusion_v2` delegates to the same function after the coinbase check, inheriting the same flaw: [3](#0-2) 

The only validation performed is:
1. The block hash exists in `mainchain_header_to_height` (i.e., it is on the contract's current main chain).
2. The confirmation depth relative to the stored tip meets `args.confirmations`.
3. The Merkle proof recomputes to the stored `merkle_root`.

None of these checks detect that the contract's chain tip may be hours or days behind the real Bitcoin chain.

---

### Impact Explanation

If the relayer stops submitting headers (network partition, liveness failure, or targeted censorship), the contract's `mainchain_tip_blockhash` freezes. During this window:

- A Bitcoin block that was on the main chain at the time the relayer stopped may subsequently be reorganized out of the real Bitcoin chain.
- The contract still lists that block in `mainchain_header_to_height` and `headers_pool`.
- An attacker who had a transaction in that now-orphaned block can call `verify_transaction_inclusion_v2` with a valid Merkle proof (which they can construct from the original block data).
- The contract returns `true` with the full requested confirmation count.
- Any downstream bridge, atomic swap, or settlement contract that trusts this result will release funds (e.g., wrapped BTC, stablecoins) against a Bitcoin transaction that no longer exists on the canonical chain.

The corrupted state is: `verify_transaction_inclusion` returns `true` for a transaction whose containing block has been reorganized out of the real Bitcoin chain, because the contract's `mainchain_tip_blockhash` is stale and the reorg is invisible to it.

---

### Likelihood Explanation

Bitcoin reorganizations of 1–2 blocks occur regularly (several times per year). A relayer outage of even a few hours is realistic given network conditions, NEAR RPC failures, or targeted denial-of-service against the relayer. The combination — relayer stops, a shallow reorg occurs, attacker submits proof before the relayer recovers — is a realistic attack window. The attacker needs no privileged role; `verify_transaction_inclusion_v2` is a public view-like call accessible to any NEAR account.

---

### Recommendation

1. **Record a last-update timestamp**: Add a `last_submitted_block_timestamp: u64` field to `BtcLightClient`, updated in `submit_blocks` to `env::block_timestamp_ms()`.

2. **Add a freshness guard in both verification functions**: Require that `env::block_timestamp_ms() - self.last_submitted_block_timestamp` is below a configurable `max_staleness_ms` threshold (e.g., 2 hours for Bitcoin's ~10-minute block time). Reject the proof if the light client is stale.

3. **Make the threshold configurable**: Expose a setter restricted to `Role::DAO` or `Role::PauseManager` so the staleness window can be adjusted without a contract upgrade.

---

### Proof of Concept

1. Deploy the contract and initialize with a genesis block. The relayer submits headers up to Bitcoin height H, including block B at height H-6 containing attacker transaction T.
2. The relayer goes offline. The real Bitcoin chain reorganizes: block B is orphaned, replaced by block B' at height H-6.
3. The contract's `mainchain_tip_blockhash` remains frozen at height H. Block B is still in `mainchain_header_to_height` and `headers_pool`.
4. The attacker calls `verify_transaction_inclusion_v2` with:
   - `tx_id` = hash of T
   - `tx_block_blockhash` = hash of B (still in the contract's main chain index)
   - `merkle_proof` = valid Merkle path for T in B (publicly available from the original block)
   - `coinbase_tx_id` / `coinbase_merkle_proof` = valid coinbase proof for B
   - `confirmations` = 6
5. The contract computes: `(H - (H-6)) + 1 = 7 >= 6` → confirmation check passes. Merkle proof verifies against B's stored `merkle_root` → returns `true`.
6. The downstream bridge contract, trusting this result, mints wrapped BTC for the attacker. Transaction T no longer exists on the real Bitcoin chain. [4](#0-3)

```rust
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
