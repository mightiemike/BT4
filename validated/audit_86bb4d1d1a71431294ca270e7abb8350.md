### Title
Stale `digestToSubaccount` mapping not cleared on isolated subaccount closure enables order matching against zero-margin subaccount — (`OffchainExchange.sol`)

---

### Summary

When `_tryCloseIsolatedSubaccount` is called in `OffchainExchange.sol`, it clears `parentSubaccounts`, `isolatedSubaccounts`, and `isolatedSubaccountsMask` for the closed subaccount, but does **not** clear `digestToSubaccount[digest]` or `digestToMargin[digest]`. This stale mapping causes `matchOrders` to continue redirecting the original order's sender to the now-closed, zero-margin isolated subaccount, enabling trades to execute against an undercollateralized account.

---

### Finding Description

**Vulnerability class**: Cross-contract desynchronization — a state transition (isolated subaccount closure) removes some state but leaves a stale pointer that re-activates the old state in a different execution path.

**Root cause — `_tryCloseIsolatedSubaccount`**: [1](#0-0) 

The function sweeps balances back to the parent and clears three mappings: [2](#0-1) 

However, it never clears `digestToSubaccount[digest]` or `digestToMargin[digest]`, which were set during `createIsolatedSubaccount`: [3](#0-2) 

**Stale pointer consumed in `matchOrders`**:

`matchOrders` unconditionally redirects `taker.order.sender` (and `maker.order.sender`) to whatever `digestToSubaccount[digest]` holds: [4](#0-3) 

After closure, this redirects the order to the closed isolated subaccount, which holds **zero margin** (all funds were swept back to the parent during closure).

**Re-creation of the same isolated subaccount address**:

Isolated subaccount addresses are deterministic — they encode `(ownerAddress, productId, id)`: [5](#0-4) 

When the subaccount is closed, the bitmask bit for `id` is cleared, making that `id` available again. A subsequent `createIsolatedSubaccount` for the same parent and product reuses the same `id`, producing the **identical subaccount address**. The new subaccount's margin is now reachable via the old stale `digestToSubaccount[oldDigest]` in addition to the new `digestToSubaccount[newDigest]`.

**`createIsolatedSubaccount` early-return on stale digest**:

If `createIsolatedSubaccount` is called again with the original order (same digest), it returns the closed subaccount immediately without transferring margin: [6](#0-5) 

This means the parent retains the margin that was swept back during closure, while the isolated subaccount has zero margin — yet the order can still be matched against it.

---

### Impact Explanation

**Concrete state delta corrupted**: `digestToSubaccount[digest]` persists after closure, causing `matchOrders` to route fills to a zero-margin isolated subaccount.

**Asset impact**:
1. A partially-filled isolated order whose subaccount was subsequently closed can have its remaining unfilled amount matched against the closed (zero-margin) subaccount.
2. The user effectively trades the remaining order amount **without margin** — the margin was returned to the parent during closure, but the order remains matchable.
3. The closed isolated subaccount goes negative, corrupting the protocol's internal accounting.
4. If the same isolated subaccount address is reused for a new order, the new subaccount's margin is exposed to the old order's fills, draining it unexpectedly.

The base `OffchainExchange` health check is a no-op: [7](#0-6) 

So no on-chain guard prevents the fill from executing against the zero-margin subaccount.

---

### Likelihood Explanation

The sequencer submits `matchOrders` transactions and controls which orders are matched. However:

- The sequencer operates off-chain and may not track the on-chain closure of isolated subaccounts in real time, especially under high load or after a liquidation-triggered closure via `clearinghouse`.
- `tryCloseIsolatedSubaccount` can be triggered by the clearinghouse (e.g., during liquidation), which is an automated path the sequencer may not synchronize with before submitting the next batch.
- The `CloseIsolatedSubaccount` transaction type is processed via the sequencer path (`processTransactionImpl`), meaning a closure and a subsequent match can appear in the same batch, with the match executing against the already-closed subaccount. [8](#0-7) 

---

### Recommendation

In `_tryCloseIsolatedSubaccount`, after clearing the three existing mappings, also delete the `digestToSubaccount` and `digestToMargin` entries for all digests associated with the closing subaccount. Since the subaccount address is deterministic and unique per `(owner, productId, id)`, the protocol should maintain a reverse mapping from subaccount to its associated digest(s), or clear these entries at closure time.

At minimum, add:

```solidity
// In _tryCloseIsolatedSubaccount, after line 204:
// delete digestToSubaccount entries pointing to this subaccount
// delete digestToMargin entries for those digests
```

A reverse mapping `subaccountToDigest` populated in `createIsolatedSubaccount` and consumed in `_tryCloseIsolatedSubaccount` would be the cleanest fix.

---

### Proof of Concept

1. User A submits `CreateIsolatedSubaccount` for product P with margin M and order O1 (digest D1). Isolated subaccount SA is created; M is transferred from parent to SA. `digestToSubaccount[D1] = SA`.
2. O1 is partially filled. SA holds a position and reduced margin.
3. The position is closed (SA's perp balance amount = 0). `_tryCloseIsolatedSubaccount(SA)` is called — remaining margin swept back to parent. `parentSubaccounts[SA]`, `isolatedSubaccounts[parent][0]`, and `isolatedSubaccountsMask` are cleared. **`digestToSubaccount[D1]` is NOT cleared.**
4. O1 still has unfilled amount. The sequencer submits `MatchOrders` with O1 as taker. `matchOrders` reads `digestToSubaccount[D1] = SA` and sets `taker.order.sender = SA`. SA has zero margin. The fill executes. SA goes negative.
5. Alternatively: User A creates a new isolated order O2 for the same product P. `createIsolatedSubaccount` reuses `id=0`, producing the same SA address. `digestToSubaccount[D2] = SA`, new margin M2 transferred to SA. Now both D1 and D2 map to SA. The sequencer can match O1 (stale) against SA, draining M2 — the margin deposited for O2. [9](#0-8) [10](#0-9)

        uint32 productId = RiskHelper.getIsolatedProductId(subaccount);

        }
