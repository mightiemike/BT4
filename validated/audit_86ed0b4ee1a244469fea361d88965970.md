### Title
Silent Discard of `max_atom_len` and `strict` Safety Parameters in `deser_auto` Legacy Path — (`File: wheel/src/api.rs`)

### Summary

`deser_auto` accepts `max_atom_len` and `strict` keyword arguments that callers pass to enforce atom-size and canonicality constraints. When the input blob is in legacy or backrefs format (the `else` branch), both parameters are silently discarded: `node_from_bytes_backrefs` is called with no size cap and no strictness check. An attacker who controls the serialized bytes can force the legacy path and bypass both limits entirely.

### Finding Description

`deser_auto` is the Python-facing auto-detecting deserializer. Its signature advertises two safety knobs: [1](#0-0) 

```python
deser_auto(blob, *, max_atom_len=PY_DEFAULT_MAX_ATOM_LEN, strict=True)
```

The function dispatches on the magic prefix:

- **serde_2026 path** (lines 151–153): both `max_atom_len` and `strict` are forwarded to `deserialize_2026_body_from_stream`.
- **legacy/backrefs path** (lines 153–155): `node_from_bytes_backrefs` is called with **neither** parameter. [2](#0-1) 

`node_from_bytes_backrefs` has no atom-length or strictness parameter at all: [3](#0-2) 

`parse_atom` (called from `node_from_stream_backrefs`) imposes no per-atom size limit; the only guard is the allocator-level heap cap, which is a coarser, separate control: [4](#0-3) 

The `PY_DEFAULT_MAX_ATOM_LEN` constant is explicitly documented as a "don't OOM the parser" default: [5](#0-4) 

### Impact Explanation

**Memory exhaustion via oversized atoms.** A caller that passes `max_atom_len=N` (e.g., `N = 4096` for a constrained validator) expects every atom in the deserialized tree to be at most `N` bytes. By supplying a legacy-format blob (no magic prefix) containing a multi-megabyte atom, an attacker bypasses this cap entirely. The allocator will happily allocate the full atom, potentially exhausting process memory or triggering OOM in the host process.

**Canonicality bypass via `strict=True`.** Whatever non-canonical encodings the `strict` flag rejects in the 2026 path are accepted without question in the legacy path, because `strict` is never consulted. A caller relying on `deser_auto(..., strict=True)` to reject malformed legacy blobs receives no such guarantee.

Both parameters are accepted by the function signature, giving callers a false sense of security that the constraints are uniformly enforced regardless of format.

### Likelihood Explanation

`deser_auto` is the convenience entry point designed for callers who do not want to sniff the format themselves. Any downstream Python code that calls `deser_auto(untrusted_blob, max_atom_len=small_value)` is vulnerable. An attacker who controls the serialized bytes simply omits the serde_2026 magic prefix (`fd ff 32 30 32 36`) to force the legacy branch. This requires no special privileges — only the ability to supply bytes to the API.

### Recommendation

Either:

1. **Enforce `max_atom_len` in the legacy path** by adding a size-checking wrapper around `node_from_bytes_backrefs` (or threading the limit into `parse_atom`), and enforce `strict` canonicality checks for legacy atoms as well; or
2. **Remove `max_atom_len` and `strict` from `deser_auto`'s signature** if they cannot be uniformly applied, and document that the legacy path has no per-atom size cap, so callers can make an informed choice.

### Proof of Concept

```python
import clvm_rs

# Craft a legacy-format blob with a 10 MB atom.
# Legacy atom encoding: 0b10_xxxxxx header for length.
# For simplicity, use a 1-byte atom to illustrate the bypass:
atom_size = 10 * 1024 * 1024          # 10 MB
# Legacy encoding: length prefix for large atoms uses multi-byte header.
# A single 0x8A prefix encodes a 10-byte atom; scale accordingly.
# The caller believes max_atom_len=64 will protect them:
large_legacy_blob = b'\x8a' + b'A' * 10  # small demo; scale to trigger OOM

node = clvm_rs.deser_auto(large_legacy_blob, max_atom_len=4)
# No error is raised. The 10-byte atom is accepted despite max_atom_len=4,
# because node_from_bytes_backrefs ignores the limit entirely.
# Scale atom_size to process memory to trigger OOM.
```

The root cause is in `wheel/src/api.rs` at the `else` branch of `deser_auto` (line 154), where `node_from_bytes_backrefs` is called without forwarding `max_atom_len` or `strict`. [6](#0-5)

fn deser_auto(blob: &[u8], max_atom_len: usize, strict: bool) -> PyResult<LazyNode> {
    let mut a = Allocator::new();
    let node = if let Some(body) = blob.strip_prefix(SERDE_2026_MAGIC_PREFIX.as_slice()) {
        deserialize_2026_body_from_stream(&mut a, &mut Cursor::new(body), max_atom_len, strict)
            .map_err(eval_to_py)?
    } else {
        node_from_bytes_backrefs(&mut a, blob).map_err(eval_to_py)?
    };
    Ok(LazyNode::new(Rc::new(a), node))
}
