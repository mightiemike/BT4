Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` gates on `owner` instead of `sender`, allowing any caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the direct caller of `pool.addLiquidity`) and gates only on `owner` (the freely chosen position beneficiary). Because `owner` is an arbitrary caller-supplied parameter with no allowlist restriction, any unprivileged address can bypass the deposit allowlist by supplying an already-allowlisted address as `owner` while paying for the deposit themselves via `MetricOmmPoolLiquidityAdder`. This fully nullifies the pool admin's intent to restrict depositors.

## Finding Description
`ExtensionCalling._beforeAddLiquidity` passes both `sender` (the direct caller of `pool.addLiquidity`, i.e., `msg.sender` at the pool level) and `owner` (the position beneficiary chosen by the caller) to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L88-98
function _beforeAddLiquidity(address sender, address owner, ...) internal {
    _callExtensionsInOrder(
        BEFORE_ADD_LIQUIDITY_ORDER,
        abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
    );
}
```

`DepositAllowlistExtension.beforeAddLiquidity` receives both but discards `sender` (unnamed first parameter) and checks only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
```

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts an arbitrary `owner` from the caller, validates only `owner != address(0)`, and stores `msg.sender` (the actual payer) in transient storage:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol L56-67
function addLiquidityExactShares(address pool, address owner, ...) external payable override {
    _validateOwner(owner);  // only checks != address(0)
    _validateDeltas(deltas);
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
    //                                              ^^^^^^^^^^^ payer stored in transient storage
}
```

The exploit path: Bob (unauthorized) calls `LiquidityAdder.addLiquidityExactShares(pool, Alice, ...)` where Alice is allowlisted. LiquidityAdder calls `pool.addLiquidity(Alice, ...)`. The pool calls `_beforeAddLiquidity(LiquidityAdder, Alice, ...)`. The hook checks `allowedDepositor[pool][Alice]` → true → passes. Tokens are pulled from Bob (the transient payer), LP shares minted to Alice. Bob (the actual depositor) is never checked.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly gates on `sender`:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```

The two guards are architecturally inconsistent: the swap guard checks the correct identity; the deposit guard does not.

## Impact Explanation
The deposit allowlist is fully nullified: any unprivileged caller with capital can add liquidity to any pool protected by `DepositAllowlistExtension` by supplying one allowlisted address as `owner`. Consequences include unauthorized liquidity concentration or dilution in specific bins (shifting the pool cursor and altering effective bid/ask spread for all subsequent swaps), and direct reduction of existing LPs' owed fee share and notional assets through share dilution. This constitutes a broken admin-boundary (pool admin's deposit restriction bypassed by an unprivileged path) and direct reduction of owed LP assets, meeting the contest's Critical/High impact threshold.

## Likelihood Explanation
The `MetricOmmPoolLiquidityAdder` is a public, permissionless periphery contract. The attacker only needs to identify one allowlisted address, which is observable on-chain from `AllowedToDepositSet` events or direct `allowedDepositor` reads. No privileged access, special token behavior, or malicious pool setup is required. The attack is executable in a single transaction and is repeatable across any bin configuration.

## Recommendation
Gate on `sender` (the actual depositor / direct caller of `pool.addLiquidity`) rather than `owner`:

```solidity
// Before (wrong):
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {

// After (correct):
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
```

This aligns the deposit guard with `SwapAllowlistExtension.beforeSwap`'s correct pattern and ensures the economically relevant actor (the one triggering the state change) is checked against the allowlist.

## Proof of Concept
```
Setup:
  - Pool P deployed with DepositAllowlistExtension E.
  - Pool admin allowlists Alice: allowedDepositor[P][Alice] = true.
  - Bob (unauthorized) holds token0/token1 and has approved LiquidityAdder.

Attack (single tx, Bob as msg.sender):
  1. Bob calls:
       LiquidityAdder.addLiquidityExactShares(
           pool  = P,
           owner = Alice,   // allowlisted address observed on-chain
           salt  = 0,
           deltas = { binIdxs: [0], shares: [1_000_000] },
           maxAmountToken0 = X,
           maxAmountToken1 = Y,
           extensionData = ""
       )
  2. LiquidityAdder calls pool.addLiquidity(Alice, 0, deltas, KIND_PAY, "").
  3. Pool calls E.beforeAddLiquidity(LiquidityAdder, Alice, 0, deltas, "").
     Hook checks allowedDepositor[P][Alice] → true → passes.
  4. Pool mints LP shares to Alice; callback pulls tokens from Bob (payer).

Result:
  - Bob (unauthorized) has deposited into the restricted pool.
  - Alice receives LP shares she did not request.
  - The deposit allowlist provided zero protection against Bob.
  - Bob can repeat with different bins to manipulate pool liquidity distribution.
```
