### Title
Linked Signer Can Overwrite Itself With an Unauthorized Address, Corrupting Subaccount Authorization State — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction in `EndpointTx.processTransactionImpl` is validated with `allowLinkedSigner = true`, meaning the **current linked signer** — not just the subaccount owner — can sign and submit a new `LinkSigner` transaction. This allows a linked signer to silently replace itself with any arbitrary address without the subaccount owner's consent, corrupting the authorization state of the subaccount.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the fast-path `LinkSigner` handler is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which passes the current linked signer as an accepted signer:

```solidity
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

This means the signature on a `LinkSigner` transaction is accepted from **either** the subaccount owner's address **or** the currently registered linked signer. Because the transaction writes `linkedSigners[signedTx.tx.sender] = <new signer>`, the linked signer can unilaterally replace itself with any address.

The `linkedSigners` mapping is the sole authorization record for delegated signing authority over a subaccount:

```solidity
function getLinkedSigner(bytes32 subaccount) public view virtual returns (address) {
    return
        RiskHelper.isIsolatedSubaccount(subaccount)
            ? linkedSigners[IOffchainExchange(offchainExchange).getParentSubaccount(subaccount)]
            : linkedSigners[subaccount];
}
``` [3](#0-2) 

The linked signer is accepted for high-impact operations including `WithdrawCollateral` (also `allowLinkedSigner = true`), `MintNlp`, `BurnNlp`, `TransferQuote`, and `LiquidateSubaccount`. [4](#0-3) 

---

### Impact Explanation

A compromised or malicious linked signer can:

1. Sign a `LinkSigner` transaction naming an attacker-controlled address as the new linked signer.
2. The sequencer processes this, writing the attacker's address into `linkedSigners[subaccount]`.
3. The attacker's address can now sign `WithdrawCollateral` transactions on behalf of the victim's subaccount, draining collateral.
4. Even if the real owner detects the intrusion and attempts to revoke by submitting their own `LinkSigner` transaction, the attacker can front-run or race the revocation by immediately submitting a withdrawal.

The corrupted state is `linkedSigners[subaccount]` — the exact field that gates all delegated signing authority. This is a direct asset-loss path, not merely an observability issue.

---

### Likelihood Explanation

Linked signers are commonly implemented as hot wallets, API keys, or automated trading bots — all of which are higher-risk targets for key compromise than the cold-wallet subaccount owner. Once a linked signer key is leaked or compromised, the attacker can silently escalate persistence before the owner can react. The nonce required is publicly readable via `getNonce`, removing any practical barrier. [5](#0-4) 

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. Only the subaccount owner (the address embedded in `sender`) should be permitted to change the linked signer for their own subaccount. A linked signer is a delegated role and should not be able to re-delegate that role to a third party.

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only owner may change linked signer
);
```

---

### Proof of Concept

1. Alice owns subaccount `alice_subaccount` and has set Bob (`0xBob`) as her linked signer via a prior `LinkSigner` transaction.
2. Bob's private key is compromised by an attacker.
3. The attacker uses Bob's key to sign a new `LinkSigner` transaction: `{ sender: alice_subaccount, signer: 0xAttacker, nonce: current_nonce }`.
4. The sequencer calls `processTransactionImpl` → `validateSignedTx(..., true)` → signature from `0xBob` is accepted because `getLinkedSigner(alice_subaccount) == 0xBob`.
5. `linkedSigners[alice_subaccount]` is overwritten with `0xAttacker`.
6. The attacker immediately signs a `WithdrawCollateral` transaction from `0xAttacker`, draining Alice's collateral before she can revoke.
7. Alice's attempt to revoke by submitting her own `LinkSigner` transaction via slow mode (3-day delay) arrives too late. [1](#0-0) [2](#0-1)
