### Title
`ValidResourceBounds` Protobuf Deserialization Heuristic Produces Wrong Transaction Hash for AllResources Transactions with Zero L2/L1_DATA Bounds — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a zero-value heuristic to decide between the `L1Gas` and `AllResources` variants. A V3 transaction submitted with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is hashed with the `AllResources` preimage (3 resource felts), but after P2P sync deserialization it is reconstructed as `ValidResourceBounds::L1Gas(X)` (2 resource felts). The two variants produce structurally different Poseidon hash inputs in `get_tip_resource_bounds_hash`, so the hash computed from the deserialized transaction diverges from the hash the user signed and the sequencer committed.

---

### Finding Description

**Step 1 — Sender side (gateway/sequencer)**

`RpcInvokeTransactionV3` carries `AllResourceBounds` directly. When converted to `InternalRpcInvokeTransactionV3` and then to `InvokeTransactionV3`, the conversion always wraps in `ValidResourceBounds::AllResources`:

```rust
// crates/starknet_api/src/rpc_transaction.rs:679-694
impl From<InternalRpcInvokeTransactionV3> for InvokeTransactionV3 {
    fn from(tx: InternalRpcInvokeTransactionV3) -> Self {
        Self {
            resource_bounds: ValidResourceBounds::AllResources(tx.resource_bounds),
            ...
        }
    }
}
```

The transaction hash H1 is computed via `get_tip_resource_bounds_hash` with the `AllResources` branch, which hashes **three** resource felts: L1_GAS, L2_GAS, and L1_DATA_GAS:

```rust
// crates/starknet_api/src/transaction_hash.rs:203-208
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 felts total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 felts total
    }
});
```

For `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`, H1 = `poseidon(tip, L1_GAS_concat(X), L2_GAS_concat(0), L1_DATA_GAS_concat(0))`.

**Step 2 — P2P sync deserialization (receiver)**

When the block is received over P2P, `InvokeTransactionV3` is reconstructed from protobuf. The `ValidResourceBounds` deserializer applies a zero-value heuristic:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs:417-436
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)          // ← chosen when l2=0, l1_data=0
        } else {
            ValidResourceBounds::AllResources(...)
        })
    }
}
```

For the transaction above, `l2_gas = 0` and `l1_data_gas = 0`, so the deserializer produces `ValidResourceBounds::L1Gas(X)`.

**Step 3 — Hash divergence**

Any subsequent hash computation on the deserialized transaction uses the `L1Gas` branch, which hashes only **two** resource felts:

H2 = `poseidon(tip, L1_GAS_concat(X), L2_GAS_concat(0))`

H1 ≠ H2 because the Poseidon hash over 4 elements (tip + 3 resource felts) differs from the hash over 3 elements (tip + 2 resource felts), even when the extra element is zero.

**Step 4 — Reachable verification sites**

- `validate_transaction_hash` in `crates/starknet_api/src/transaction_hash.rs:170-185` recomputes the hash from the stored transaction and compares it to the expected hash. Called with the deserialized `L1Gas` variant and expected hash H1, it returns `false`, causing valid transactions to be rejected.
- `calc_tx_hash` in `crates/apollo_rpc_execution/src/lib.rs:469-476` recomputes the hash for RPC execution/simulation. It would produce H2 ≠ H1, causing `starknet_simulateTransactions` and `starknet_estimateFee` to bind the wrong hash to the executable transaction.

---

### Impact Explanation

A V3 invoke transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` — a valid, gateway-accepted transaction — has its `ValidResourceBounds` variant silently changed from `AllResources` to `L1Gas` during P2P sync deserialization. This changes the transaction hash preimage, producing a hash that differs from the one the user signed and the sequencer committed. Any node that syncs the block and then recomputes the hash (for RPC simulation, fee estimation, or hash verification) will compute the wrong hash, returning an authoritative-looking wrong value or rejecting a valid transaction.

---

### Likelihood Explanation

Any user submitting a V3 transaction with only L1 gas bounds (zero L2 gas and zero L1 data gas) triggers this path. This is a common pattern for pre-0.13.3 style V3 transactions. The protobuf serialization round-trip is exercised on every block received via P2P sync. No special privileges or malicious behavior are required.

---

### Recommendation

Remove the zero-value heuristic from the `ValidResourceBounds` protobuf deserializer. Instead, use an explicit discriminant field in the protobuf schema to distinguish `L1Gas` from `AllResources`, or always deserialize as `AllResources` when all three resource fields are present (regardless of their values):

```rust
// crates/apollo_protobuf/src/converters/transaction.rs
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let l1_gas: ResourceBounds = value.l1_gas.ok_or(missing("ResourceBounds::l1_gas"))?.try_into()?;
        let l2_gas: ResourceBounds = value.l2_gas.ok_or(missing("ResourceBounds::l2_gas"))?.try_into()?;
        match value.l1_data_gas {
            // l1_data_gas absent → legacy L1Gas variant (pre-0.13.3)
            None => Ok(ValidResourceBounds::L1Gas(l1_gas)),
            // l1_data_gas present (even if zero) → AllResources variant
            Some(l1_data_gas) => Ok(ValidResourceBounds::AllResources(AllResourceBounds {
                l1_gas,
                l2_gas,
                l1_data_gas: l1_data_gas.try_into()?,
            })),
        }
    }
}
```

This preserves the original variant semantics: `L1Gas` only when the field is absent (true legacy format), `AllResources` whenever all three fields are present.

### Proof of Concept

```rust
use starknet_api::transaction::fields::{
    AllResourceBounds, ResourceBounds, ValidResourceBounds, GasAmount, GasPrice,
};
use starknet_api::transaction_hash::get_tip_resource_bounds_hash;
use starknet_api::transaction::fields::Tip;

// Transaction submitted with AllResources, l2_gas=0, l1_data_gas=0
let all_resources = ValidResourceBounds::AllResources(AllResourceBounds {
    l1_gas: ResourceBounds { max_amount: GasAmount(100), max_price_per_unit: GasPrice(1) },
    l2_gas: ResourceBounds::default(),      // zero
    l1_data_gas: ResourceBounds::default(), // zero
});

// Same l1_gas value, but L1Gas variant (what protobuf deserialization produces)
let l1_gas_only = ValidResourceBounds::L1Gas(
    ResourceBounds { max_amount: GasAmount(100), max_price_per_unit: GasPrice(1) },
);

let tip = Tip(0);
let h1 = get_tip_resource_bounds_hash(&all_resources, &tip).unwrap();
let h2 = get_tip_resource_bounds_hash(&l1_gas_only, &tip).unwrap();

// H1 ≠ H2: AllResources hashes 3 resource felts, L1Gas hashes 2
assert_ne!(h1, h2, "Same resource values, different variants → different hashes");

// Simulate protobuf round-trip: AllResources with zero l2/l1_data → L1Gas
use apollo_protobuf::protobuf;
let proto: protobuf::ResourceBounds = all_resources.into();
let deserialized = ValidResourceBounds::try_from(proto).unwrap();
// deserialized is now L1Gas(100), not AllResources
assert!(matches!(deserialized, ValidResourceBounds::L1Gas(_)));

// Hash after round-trip differs from original hash
let h_after_sync = get_tip_resource_bounds_hash(&deserialized, &tip).unwrap();
assert_ne!(h1, h_after_sync); // ← hash mismatch: signed hash ≠ post-sync hash

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

    transaction: &T,

    let mut hash_chain = HashChain::new()
