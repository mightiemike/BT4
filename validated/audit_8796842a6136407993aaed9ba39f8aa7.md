### Title
Linked Signer Can Redirect Fast Withdrawals to Attacker-Controlled `sendTo` Address, Bypassing Sequencer-Path Guard — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`WithdrawCollateralV2` introduces a `sendTo` field that redirects the on-chain token transfer to an arbitrary address. The sequencer path explicitly blocks a linked signer from signing such a transaction when `sendTo != address(0)`. The fast-withdrawal path (`BaseWithdrawPool.submitFastWithdrawal`) performs a structurally different signature check that does not carry this guard, creating an asymmetry through which a linked signer can sign a `WithdrawCollateralV2` with an attacker-controlled `sendTo` and have the `WithdrawPool` transfer the user's collateral to that address.

---

### Finding Description

**Sequencer path — guard present.**
In `EndpointTx.processTransactionImpl`, the `WithdrawCollateralV2` branch passes `allowLinkedSigner` as `signedTx.tx.sendTo == address(0)`: [1](#0-0) 

When `sendTo` is non-zero, `allowLinkedSigner` is `false`, so `validateSignedTx` → `validateCompactSignature` → `verifier.validateCompactSignature` is called with `linkedSigner = address(0)`, meaning only the subaccount owner's key is accepted. [2](#0-1) 

**Fast-withdrawal path — guard absent.**
`BaseWithdrawPool.submitFastWithdrawal` calls `v.requireValidTxSignatures(transaction, idx, signatures)` and then delegates destination resolution to `resolveFastWithdrawal`: [3](#0-2) 

`resolveFastWithdrawal` decodes the embedded `SignedWithdrawCollateralV2` and returns `signedTx.tx.sendTo` verbatim when it is non-zero, with no check on whether the signer was the owner or a linked signer: [4](#0-3) 

`requireValidTxSignatures` validates that a signature over `(transaction, idx)` is present, but it does not re-implement the `allowLinkedSigner = (sendTo == address(0))` gate that the sequencer path enforces. A linked signer's compact signature embedded inside the `SignedWithdrawCollateralV2.signature` field will therefore pass this check even when `sendTo` points to an arbitrary address.

The resolved `sendTo` is then passed directly to `handleWithdrawTransfer`: [5](#0-4) 

which calls `token.safeTransfer(to, amount)` — an irreversible ERC-20 transfer to the attacker-controlled address.

The `WithdrawCollateralV2` struct confirms that `sendTo` is a first-class, user-supplied field: [6](#0-5) 

---

### Impact Explanation

A linked signer (a key the subaccount owner has delegated via `LinkSigner`) can construct and sign a `WithdrawCollateralV2` transaction with `sendTo = <attacker address>` and submit it through `submitFastWithdrawal`. The `WithdrawPool` will immediately transfer the full collateral amount to the attacker address. The subaccount owner's on-chain balance is debited (`spotEngine.updateBalance` in `Clearinghouse.withdrawCollateral`), and the tokens leave the protocol to an address the owner never authorized. The loss is permanent and equal to the withdrawn collateral amount.

---

### Likelihood Explanation

Any party that holds a linked signer key for a subaccount can execute this unilaterally. Linked signers are commonly held by trading bots, API keys, or third-party integrations — all of which represent realistic compromise surfaces. No governance action, sequencer cooperation, or owner interaction is required beyond the linked signer key being available to the attacker. The fast-withdrawal path is a public, permissionless entry point.

---

### Recommendation

**Short term:** In `resolveFastWithdrawal` (or in `submitFastWithdrawal` before calling `handleWithdrawTransfer`), extract the signer address from the embedded `CompactSignature`, resolve whether it is a linked signer for the subaccount, and revert if `sendTo != address(0)` and the signer is not the subaccount owner — mirroring the `allowLinkedSigner = (sendTo == address(0))` logic already present in the sequencer path.

**Long term:** Centralise the `allowLinkedSigner` policy into a single shared helper (e.g., inside `Verifier` or `EndpointTx`) so that every withdrawal verification path — sequencer, slow-mode, and fast-withdrawal — enforces the same rule without duplication.

---

### Proof of Concept

1. Owner of subaccount `S` has previously called `LinkSigner` to register `linkedKey` as a linked signer.
2. Attacker (who controls `linkedKey`) constructs:
   ```
   WithdrawCollateralV2 {
       sender:    S,
       productId: USDC_PRODUCT_ID,
       amount:    <full balance>,
       nonce:     current_nonce,
       sendTo:    attacker_EOA,   // non-zero → sequencer path would reject linked signer
       appendix:  0
   }
   ```
3. Attacker signs this struct with `linkedKey`, producing a `CompactSignature`, and wraps it in `SignedWithdrawCollateralV2`.
4. Attacker (or any fast-withdrawal provider) calls:
   ```solidity
   withdrawPool.submitFastWithdrawal(idx, encodedSignedTx, providerSigs);
   ```
5. `requireValidTxSignatures` accepts the linked signer's compact signature.
6. `resolveFastWithdrawal` returns `sendTo = attacker_EOA`.
7. `handleWithdrawTransfer` executes `token.safeTransfer(attacker_EOA, amount)`.
8. Subaccount `S` is debited; funds are irreversibly transferred to `attacker_EOA`.

```text
    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
```
