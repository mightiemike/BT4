### Title
Merkle Second-Preimage Proof Forgery via Unvalidated `tx_id` in `verify_transaction_inclusion` — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` accepts a caller-supplied `tx_id` and a caller-supplied `merkle_proof` with no validation that `tx_id` is a leaf node (real transaction hash) rather than an internal Merkle tree node. Any unprivileged NEAR caller can supply an internal node hash as `tx_id`, pair it with a shortened proof path, and receive `true` — a forged SPV inclusion proof — for a transaction that was never included in the block.

---

### Finding Description

`verify_transaction_inclusion` at `contract/src/lib.rs:288` is a public, unrestricted view function. It accepts `ProofArgs` containing attacker-controlled fields: `tx_id`, `tx_index`, `merkle_proof`, and `tx_block_blockhash`. The function's sole verification step is:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

There is no check that `args.tx_id` is a leaf-level transaction hash. Bitcoin block headers store only the Merkle root — not the transaction count — so the contract cannot independently bound the proof depth. An attacker who knows any real block in `headers_pool` can compute an internal Merkle node at depth `d` and supply a proof of length `proof_length - d`, causing `compute_root_from_merkle_proof` to reconstruct the same root from a non-leaf starting point.

The function carries a `#[deprecated]` marker and a warning comment, but it remains fully callable by any NEAR account with no role guard, no `#[private]`, and no `#[trusted_relayer]` restriction. [2](#0-1) 

The project's own internal documentation explicitly acknowledges this root cause:

> "This function is vulnerable to the standard Bitcoin merkle tree second-preimage attack — it may return `true` for an internal node hash rather than a real transaction hash. Block headers do not contain the transaction count, so proof depth cannot be validated on-chain." [3](#0-2) 

`verify_transaction_inclusion_v2` (`contract/src/lib.rs:347`) was introduced to mitigate the 64-byte transaction forgery by requiring a coinbase Merkle proof of equal length, but it internally delegates to the same vulnerable v1 function via `self.verify_transaction_inclusion(args.into())`. [4](#0-3) 

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate a privileged action (e.g., releasing bridged funds, minting wrapped tokens, confirming a cross-chain event) can be deceived into accepting a forged proof. The attacker does not need to submit any new Bitcoin block or forge any PoW — they only need to know the Merkle tree structure of a block already accepted by the contract. The return value `true` is indistinguishable from a legitimate inclusion proof to the caller.

---

### Likelihood Explanation

The function is publicly callable by any NEAR account with no deposit, no role, and no staking requirement. The Merkle tree structure of any Bitcoin block is public information. An attacker targeting a specific downstream application only needs to identify a block in the contract's `headers_pool` and compute the appropriate internal node. This is a well-known Bitcoin SPV attack with published tooling.

---

### Recommendation

1. **Remove or gate `verify_transaction_inclusion` (v1)**: Add `#[private]` or a role guard so it cannot be called externally, or remove it entirely since it is already marked `#[deprecated]`.
2. **Enforce proof depth equality in v2**: `verify_transaction_inclusion_v2` enforces `merkle_proof.len() == coinbase_merkle_proof.len()`, which bounds the depth relative to the coinbase proof. However, since v2 still calls v1 internally, removing v1's external accessibility is the critical fix.
3. **Document the trust boundary**: Any external contract consuming the result of either function must independently verify that `tx_id` corresponds to a valid raw transaction before acting on the result.

---

### Proof of Concept

1. Identify any block `B` already stored in `headers_pool` with at least 4 transactions (so the Merkle tree has at least one internal node above the leaf level).
2. Compute the internal Merkle node `N` at depth 1 (the parent of transactions 0 and 1): `N = SHA256d(tx0 || tx1)`.
3. Construct a `merkle_proof` of length `proof_depth - 1` starting from `N`'s sibling at depth 1 upward to the root.
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node, not a real transaction)
   - `tx_index = 0` (adjusted for the subtree position)
   - `tx_block_blockhash = B.block_hash`
   - `merkle_proof = [sibling_at_depth_1, ..., sibling_at_root]`
   - `confirmations = 0`
5. The function returns `true` — a forged proof that `N` (a non-existent transaction) is included in block `B`. [5](#0-4)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root

            usize::try_from(args.tx_index).unwrap(),
