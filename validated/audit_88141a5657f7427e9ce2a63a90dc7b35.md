Audit Report

## Title
`SwapAllowlistExtension` checks router address as `sender` instead of original user, enabling allowlist bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `msg.sender` of `pool.swap()` is the router contract, not the original user. If the pool admin allowlists the router address to enable router-mediated swaps for legitimate users, every user — including non-allowlisted ones — can bypass the curated allowlist by routing through the router.

## Finding Description
The call chain is confirmed by the production code:

1. `MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, recipient, ...)` — when the router calls `pool.swap()`, `msg.sender` is the router contract. [1](#0-0) 

2. `ExtensionCalling._beforeSwap` forwards `sender` (the router address) unchanged to the extension. [2](#0-1) 

3. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` — here `msg.sender` is the pool and `sender` is the router, so the check passes if the router is allowlisted, regardless of who the original user is. [3](#0-2) 

4. `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making itself `msg.sender` of the pool. The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [4](#0-3) 

The root cause is that `sender` in the extension context represents the immediate caller of `pool.swap()`, not the economically relevant actor initiating the trade. No existing guard in the extension or pool corrects for this indirection.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates on `owner` (the second parameter), which is the explicit position owner passed by the caller and preserved correctly through the `MetricOmmPoolLiquidityAdder` path. [5](#0-4) 

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of counterparties loses that protection entirely once the router is allowlisted. Any unprivileged user can execute swaps against the pool's liquidity at oracle-anchored prices intended only for vetted participants. This constitutes a direct loss of LP principal and a complete failure of the pool's access-control invariant — matching the "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" allowed impact categories.

## Likelihood Explanation
The pool admin must allowlist the router to enable router-mediated swaps for legitimate users — a natural and expected operational step for any curated pool that supports the standard periphery. The bypass requires no special privileges, no malicious setup, and no non-standard tokens: any user can call `MetricOmmSimpleRouter.exactInputSingle()` targeting the pool. The only precondition is that the admin has allowlisted the router, which is a routine configuration step, not an adversarial one. Likelihood is **Medium**.

## Recommendation
The extension should check the original user rather than the immediate caller of `pool.swap()`. The cleanest fix is to check `recipient` and enforce `recipient == sender` at the extension level, then gate on that address. Alternatively, the router can encode the original `msg.sender` in `extensionData`, and the extension decodes and checks it — but this requires a coordinated encoding convention between the router and extension. The `recipient` approach is simpler and does not require protocol-wide coordination.

## Proof of Concept
```
Setup:
  1. Deploy pool with SwapAllowlistExtension.
  2. Pool admin calls setAllowedToSwap(pool, router, true)
     — intending to let allowlisted users trade via the router.
  3. Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

Attack:
  4. attacker (non-allowlisted EOA) calls:
       MetricOmmSimpleRouter.exactInputSingle({
           pool: pool,
           recipient: attacker,
           ...
       })
  5. Router calls pool.swap(attacker, ...) with msg.sender = router.
  6. Pool calls _beforeSwap(sender=router, ...).
  7. SwapAllowlistExtension checks allowedSwapper[pool][router] → true.
  8. Swap executes. Attacker receives pool output tokens.

Result:
  Non-allowlisted attacker successfully swaps against the curated pool,
  bypassing the SwapAllowlistExtension guard entirely.
```
