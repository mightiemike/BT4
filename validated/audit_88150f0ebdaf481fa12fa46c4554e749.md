### Title
Gateway Stateless Validator Unconditionally Applies L2 Gas Price Floor to Transactions With Zero L2 Gas Amount, Blocking Valid Admission - (File: crates/apollo_gateway/src/stateless_transaction_validator.rs)

### Summary

The `validate_resource_bounds` function in `StatelessTransactionValidator` applies a `min_gas_price` floor check against `l2_gas.max_price_per_unit` unconditionally, even when `l2_gas.max_amount` is zero. A transaction that sets only L1 gas (or only L1 data gas) with zero L2 gas amount and zero L2 gas price passes the non-zero fee check but is then incorrectly rejected by the price floor check. In production, `min_gas_price = 8_000_000_000`, so any such transaction is permanently blocked at the gateway.

### Finding Description

In `crates/apollo_gateway/src/stateless_transaction_validator.rs`, `validate_resource_bounds` performs two sequential checks:

```rust
// Check 1 – at least one resource bound is non-zero
if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0) {
    return Err(ZeroResourceBounds { resource_bounds });
}

// Check 2 – L2 gas price meets the floor
if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
    return Err(MaxGasPriceTooLow { ... });
}
```

Check 1 passes for a transaction with `l1_gas.max_amount > 0` and `l2_gas = {max_amount: 0, max_price_per_unit: 0}` because `max_possible_fee` sums all three resources and the L1 gas term is non-zero. [1](#0-0) 

Check 2 then fires unconditionally on `l2_gas.max_price_per_unit`, which is `0` for any transaction that does not use L2 gas. With the production default `min_gas_price = 8_000_000_000`, the comparison `0 < 8_000_000_000` is always true, and the transaction is rejected. [2](#0-1) 

The production value is set in the node config schema: [3](#0-2) 

The test suite for the positive flow (`valid_l1_gas`) uses `DEFAULT_VALIDATOR_CONFIG_FOR_TESTING` which sets `min_gas_price: 0`, so the comparison `0 < 0` is false and the test passes silently, masking the production defect. [4](#0-3) 

The check should be guarded by `l2_gas.max_amount > 0`: when the L2 gas amount is zero, the price per unit is irrelevant to the fee (`0 × price = 0`), and imposing a price floor on it is semantically incorrect. The analogous `max_l2_gas_amount` upper-bound check on the very next line already applies this guard correctly: [5](#0-4) 

### Impact Explanation

Any unprivileged user who submits a V3 `AllResourceBounds` transaction with a non-zero L1 gas (or L1 data gas) bound but zero L2 gas amount and zero L2 gas price will have their transaction permanently rejected by the gateway with `MaxGasPriceTooLow`, even though the transaction is economically valid and would execute correctly. This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The trigger is fully unprivileged and requires only a well-formed V3 transaction. The `AllResourceBounds` type is the only resource-bounds variant accepted by the current gateway (all `RpcTransaction` variants carry `AllResourceBounds`). Any client that sets only L1 gas or only L1 data gas — a natural pattern for transactions whose execution cost is dominated by calldata — will hit this rejection. The production `min_gas_price` of 8 Gwei makes the condition deterministic.

### Recommendation

Guard the L2 gas price floor check with a non-zero amount condition:

```rust
if resource_bounds.l2_gas.max_amount.0 > 0
    && resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price
{
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
        gas_price: resource_bounds.l2_gas.max_price_per_unit,
        min_gas_price: self.config.min_gas_price,
    });
}
```

Add a test case that exercises `validate_resource_bounds: true` with a non-zero `min_gas_price` and a transaction carrying only L1 gas, asserting `Ok(())`.

### Proof of Concept

1. Configure the gateway with the production default `min_gas_price = 8_000_000_000`.
2. Submit a V3 invoke transaction with:
   ```
   resource_bounds = AllResourceBounds {
       l1_gas:      { max_amount: 1000, max_price_per_unit: 10_000_000_000 },
       l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
       l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },
   }
   ```
3. `max_possible_fee(Tip::ZERO) = 1000 × 10_000_000_000 = 10^13 > 0` → Check 1 passes.
4. `l2_gas.max_price_per_unit.0 = 0 < 8_000_000_000` → Check 2 fires, returning `MaxGasPriceTooLow { gas_price: 0, min_gas_price: 8_000_000_000 }`.
5. The transaction is rejected despite being economically valid and having sufficient L1 gas to cover execution.
