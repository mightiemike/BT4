Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address as Swapper Identity, Enabling Full Allowlist Bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the pool — the direct caller. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end-user. If the pool admin allowlists the router to enable legitimate users to swap, any unprivileged user can bypass the allowlist entirely with a single router call. If the admin does not allowlist the router, allowlisted users cannot use the router at all.

## Finding Description

**Exact call chain:**

1. Attacker calls `MetricOmmSimpleRouter.exactInputSingle(params)`. [1](#0-0) 

2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — the pool's `msg.sender` is now the router address. [2](#0-1) 

3. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, recipient, ...)` — `msg.sender` here is the router. [3](#0-2) 

4. `ExtensionCalling._beforeSwap` forwards `sender = router_address` to every configured extension. [4](#0-3) 

5. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]` = `allowedSwapper[pool][router_address]`. [5](#0-4) 

The extension never sees the original user's address. The `recipient` field is available in the function signature but is ignored by the check. There is no mechanism in the pool interface to propagate end-user identity through the router.

**Root cause:** `SwapAllowlistExtension.beforeSwap` uses `sender` (the pool's direct caller) as the identity to gate, but the router is the direct caller for all router-mediated swaps. The `allowedSwapper` mapping is keyed on `(pool, swapper)` where `swapper` must be the direct pool caller. [6](#0-5) 

**Existing guards are insufficient:** `isAllowedToSwap` is a view helper only; it does not affect the `beforeSwap` enforcement path. `allowAllSwappers` is a separate escape hatch that, if set, bypasses the check entirely — but this is not the scenario under analysis. [7](#0-6) 

## Impact Explanation

For any pool deploying `SwapAllowlistExtension` (KYC-gated, institutional, or restricted-access pools), the allowlist is rendered ineffective when the router is involved. If the admin allowlists the router to allow legitimate users to swap, any unprivileged address can call `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` on `MetricOmmSimpleRouter` and execute swaps against the restricted pool. This is a direct admin-boundary break: the pool admin's explicit access control policy is bypassed by an unprivileged caller using the standard router entry point. Unauthorized swaps extract value from LP positions under conditions the pool admin did not intend to permit, constituting a broken core pool functionality and admin-boundary bypass.

## Likelihood Explanation

The router is the primary user-facing entry point for swaps. No special privileges, flash loans, or multi-transaction setup are required. A single `exactInputSingle` call suffices. Any user who observes that a pool has `SwapAllowlistExtension` configured and that the router is allowlisted (or can probe by attempting a swap) can trivially exploit this. The condition is reachable on any restricted pool where the admin has allowlisted the router to support normal user flows.

## Recommendation

The `sender` identity passed to extensions must reflect the economic actor, not the intermediary. The preferred fix is to extend the pool's `swap` interface with an explicit `originator` field that the router populates with `msg.sender` (the end-user), and have `SwapAllowlistExtension.beforeSwap` check `originator` instead of `sender`. Alternatively, `SwapAllowlistExtension` can decode an authenticated user address from `extensionData`, but this requires the router to sign or commit to the user identity in a non-forgeable way. Checking `recipient` instead of `sender` is a partial mitigation only if the pool's design intent is to gate who receives output, not who initiates the swap.

## Proof of Concept

```solidity
// Pool deployed with SwapAllowlistExtension.
// Admin allowlists only `trustedUser`:
// extension.setAllowedToSwap(pool, trustedUser, true);

// Admin also allowlists the router so trustedUser can use it:
// extension.setAllowedToSwap(pool, router, true);

// Attacker (not allowlisted) calls the router:
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: attacker,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    tokenIn: token0,
    extensionData: ""
}));
// Pool receives msg.sender = router.
// _beforeSwap(router, attacker, ...) is called.
// SwapAllowlistExtension checks allowedSwapper[pool][router] → true (router was allowlisted).
// Swap succeeds. Allowlist fully bypassed.
```
