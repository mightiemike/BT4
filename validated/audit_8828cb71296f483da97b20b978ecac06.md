### Title
Airdrop tokens permanently locked when beneficiary is a smart contract unable to invoke `claim()` — (`File: core/contracts/Airdrop.sol`)

---

### Summary

The `Airdrop` contract enforces that only `msg.sender` can claim their own allocation. The Merkle leaf is constructed using `msg.sender` as the `sender`, and the transfer is sent to `msg.sender`. There is no mechanism for a third party to claim on behalf of a beneficiary, and no admin recovery path for unclaimed tokens. If a Merkle-tree beneficiary is a smart contract that cannot call `claim()` (e.g., a DAO treasury, a vesting contract, a multisig with no generic call capability, or a self-destructed contract), its entire allocation is permanently locked in the `Airdrop` contract.

---

### Finding Description

In `_claim()`, the proof is verified against `msg.sender` and the transfer is sent to `msg.sender`: [1](#0-0) 

Inside `_verifyProof`, the Merkle leaf is computed as:

```solidity
bytes32 leaf = keccak256(
    bytes.concat(keccak256(abi.encode(sender, totalAmount)))
);
``` [2](#0-1) 

Because `sender` is always `msg.sender`, no external account can submit a valid proof on behalf of a different address — the leaf would not match. The public `claim()` entrypoint provides no `recipient` or `onBehalfOf` parameter: [3](#0-2) 

The contract also has no `sweep`, `recoverUnclaimed`, or equivalent admin function to retrieve tokens allocated to addresses that never claim.

---

### Impact Explanation

Any airdrop allocation whose Merkle-tree leaf encodes a smart contract address that cannot itself call `claim()` is permanently frozen inside the `Airdrop` contract. The tokens cannot be recovered by the owner, by the intended beneficiary through a proxy, or by any other party. The corrupted state is: `claimed[week][beneficiary]` remains `0` forever while the corresponding token balance sits irrecoverably in the contract.

---

### Likelihood Explanation

Protocol airdrops routinely include smart contract addresses: DAO treasuries, Gnosis Safe multisigs, vesting contracts, and protocol-owned liquidity contracts. Any such address that lacks a generic external-call mechanism (e.g., a minimal proxy, a simple vault, or a self-destructed contract) cannot invoke `claim()`. This is a realistic and common deployment scenario, making the likelihood **medium-high**.

---

### Recommendation

Allow an optional `recipient` parameter so that any caller can claim on behalf of a beneficiary whose address is encoded in the Merkle leaf:

```solidity
function claim(ClaimProof[] calldata claimProofs, address recipient) external {
    address beneficiary = recipient == address(0) ? msg.sender : recipient;
    for (uint32 i = 0; i < claimProofs.length; i++) {
        _claim(beneficiary, claimProofs[i]);
    }
}
```

The `_verifyProof` and `_claim` internals should then use `beneficiary` instead of `msg.sender` for both leaf construction and token transfer. Additionally, consider adding an owner-callable `recoverUnclaimed` function to handle tokens that remain unclaimed after a defined expiry window.

---

### Proof of Concept

1. The protocol owner registers a Merkle root that includes an allocation for address `0xDeadBeef...` — a deployed contract (e.g., a simple vault) with no ability to make arbitrary external calls.
2. `0xDeadBeef...` cannot call `Airdrop.claim()`.
3. No other account can call `claim()` and produce a valid proof for `0xDeadBeef...` because the leaf is `keccak256(keccak256(abi.encode(msg.sender, totalAmount)))` and `msg.sender` would differ from `0xDeadBeef...`.
4. The allocation for `0xDeadBeef...` is permanently locked. `claimed[week][0xDeadBeef...]` stays `0`, and the tokens remain in the `Airdrop` contract with no recovery path. [4](#0-3)

    function _verifyProof(
        uint32 week,
        address sender,
        uint256 totalAmount,
        bytes32[] calldata proof
    ) internal {
        require(claimed[week][sender] == 0, "Already claimed.");
        require(
            merkleRoots[week] != bytes32(0),
            "Week hasn't been registered."
        );
        require(
            !ISanctionsList(sanctions).isSanctioned(sender),
            "address is sanctioned."
        );
        bytes32 leaf = keccak256(
            bytes.concat(keccak256(abi.encode(sender, totalAmount)))
        );
        bool isValidLeaf = MerkleProof.verify(proof, merkleRoots[week], leaf);
        require(isValidLeaf, "Invalid proof.");
        claimed[week][sender] = totalAmount;
    }
