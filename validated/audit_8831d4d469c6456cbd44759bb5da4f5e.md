Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of End-User Identity, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool always sets to its own `msg.sender` — the immediate caller of `pool.swap()`. When `MetricOmmSimpleRouter` calls `pool.swap()`, the pool's `msg.sender` is the router contract, not the end user. A pool admin who allowlists the router address (a natural step to support router-mediated swaps for curated users) inadvertently opens the gate to every user, because the extension sees only the allowlisted router address regardless of who initiated the call.

## Finding Description

**Root cause — `sender` binding in the extension:**

In `SwapAllowlistExtension.beforeSwap` (L31–41), the check is:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
`msg.sender` is the pool (correct for pool identity). `sender` is the first argument passed by the pool.

**What the pool passes as `sender`:**

`MetricOmmPool.swap()` at L230–231 calls:
```solidity
_beforeSwap(
    msg.sender,  // always the immediate caller of pool.swap()
    ...
);
```

**What the router passes:**

`MetricOmmSimpleRouter.exactInputSingle()` at L71–80 calls `pool.swap()` directly with no originator encoding:
```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
IMetricOmmPoolActions(params.pool).swap(
    params.recipient, params.zeroForOne, ..., "", params.extensionData
);
```

The router's `msg.sender` (the end user) is stored only in transient callback context for payment purposes — it is never forwarded to the pool as a swap parameter. The pool's `msg.sender` is therefore the **router contract address**, and that is what the extension receives as `sender`.

**Exploit flow:**

| Call path | `sender` seen by extension | Check result |
|---|---|---|
| user → pool directly | user address | `allowedSwapper[pool][user]` |
| user → router → pool | **router address** | `allowedSwapper[pool][router]` |

Once `allowedSwapper[pool][router] = true`, the check evaluates to `true` for **every** router-mediated swap, regardless of who the end user is. All four router entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) exhibit this behavior, as each calls `pool.swap()` directly.

**Existing guards are insufficient:** The extension has no access to the original `msg.sender` of the router call. The `extensionData` field is passed through unchanged from the router caller (`params.extensionData`), so a malicious user can supply arbitrary `extensionData` — there is no authenticated originator field.

## Impact Explanation

Any non-allowlisted user can bypass `SwapAllowlistExtension` on a curated pool by routing through `MetricOmmSimpleRouter`. The pool receives and settles the swap normally; the extension's `beforeSwap` hook passes because it sees the allowlisted router address. This is a direct, complete bypass of curated-pool access control: non-curated users trade against LP capital deposited under the assumption of a restricted swapper set, constituting a medium-severity loss of LP asset integrity and broken core pool access-control functionality.

## Likelihood Explanation

Medium. The trigger requires the pool admin to allowlist the router — a natural and expected configuration for any curated pool whose allowlisted users are expected to interact through the standard periphery. The `setAllowedToSwap` setter accepts any address without warning or documentation that allowlisting the router opens the gate to all users. The condition is reachable by any unprivileged trader once the admin has taken this step.

## Recommendation

The `beforeSwap` hook must gate the economic actor (end user), not the immediate caller. Two options:

1. **Authenticated originator via `extensionData`:** The router encodes `msg.sender` (ABI-encoded and authenticated by the router's address) into `extensionData` for each hop; the extension decodes and verifies the originator only when `msg.sender` (the pool's caller) is a known router. This requires a coordinated convention between router and extension.
2. **Separate originator field in the hook interface:** Add an `originator` parameter to `IMetricOmmExtensions.beforeSwap` that the pool populates from a router-supplied transient slot set before calling `pool.swap()`. The extension then checks `allowedSwapper[pool][originator]`.

Until fixed, pool admins must not allowlist the router address; allowlisted users must call `pool.swap()` directly.

## Proof of Concept

```
Setup
─────
1. Pool deployed with SwapAllowlistExtension as beforeSwap hook.
2. Pool admin calls setAllowedToSwap(pool, user1, true)
   → user1 is the only allowlisted swapper.
3. Pool admin calls setAllowedToSwap(pool, router, true)
   → intended to let user1 use the router; actually opens the gate.

Attack (user2, not allowlisted)
────────────────────────────────
4. user2 calls router.exactInputSingle({pool: pool, ...})
5. Router calls pool.swap(...) — pool's msg.sender = router
6. Pool calls _beforeSwap(router, ...)
7. ExtensionCalling encodes and calls extension.beforeSwap(router, ...)
8. Extension evaluates: allowedSwapper[pool][router] == true → passes
9. Swap executes; user2 receives output tokens from the curated pool.

Result: user2 bypasses SwapAllowlistExtension entirely.
```
