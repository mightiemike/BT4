Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Originating User, Allowing Any User to Bypass a Curated Pool's Allowlist - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` to the pool — the router contract — not the originating EOA. When a pool admin allowlists the router address to enable router-based swaps, every user on the network can bypass the allowlist by routing through `MetricOmmSimpleRouter`. The allowlist is completely neutralised for any call path that goes through the router.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
  msg.sender,   // ← router address when called via MetricOmmSimpleRouter
  ...
);
```

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension via `abi.encodeCall`:

```solidity
// ExtensionCalling.sol L162-165
abi.encodeCall(
  IMetricOmmExtensions.beforeSwap,
  (sender, ...)   // sender = router, not the originating EOA
)
```

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is on the allowlist:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct) and `sender` is the router (wrong actor). The allowlist is keyed on `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

A pool admin who wants allowlisted users to be able to use the router must add the router to `allowedSwapper`. Once the router is allowlisted, the guard passes for every call that arrives through the router, regardless of who the originating user is. There is no mechanism inside `MetricOmmSimpleRouter` that re-checks the originating `msg.sender` against the pool's allowlist before forwarding the call.

The existing test suite (`FullMetricExtensionTest.test_allowedSwapSucceeds`) allowlists `callers[0]` (a `TestCaller` contract that acts as the direct `msg.sender` to the pool), which correctly models the direct-call path but does not exercise the router path where the router itself becomes `sender`.

## Impact Explanation

Any user can call `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) targeting a pool whose `SwapAllowlistExtension` has the router address in `allowedSwapper`. The extension sees `sender = router`, the check passes, and the swap executes. The pool admin's intent to restrict swaps to a curated set of addresses is silently defeated. Unauthorized users can execute repeated swaps on a pool designed to be access-controlled, draining LP value at oracle-fair prices from a pool whose LPs expected only curated counterparties.

This constitutes broken core pool functionality (the allowlist access control) causing potential loss of LP assets, meeting the contest's Medium/High threshold.

## Likelihood Explanation

The scenario is realistic:
1. A pool admin deploys a curated pool with `SwapAllowlistExtension`.
2. They want allowlisted users to be able to use the standard periphery router (the expected UX path).
3. They call `setAllowedToSwap(pool, router, true)`.
4. Any user can now swap on the pool through the router, bypassing the allowlist entirely.

The router is a public, permissionless contract. The attacker requires no special privileges — only knowledge that the router is allowlisted, which is publicly observable on-chain via `allowedSwapper(pool, router)`.

## Recommendation

The allowlist must gate the economically relevant actor — the originating user — not the intermediate router. Options:

1. **Preferred — enforce at the router level**: Have the router revert if `msg.sender` is not on the pool's allowlist before forwarding the call. This keeps the check at the entry point where the real user identity is known.
2. **Pass originating user through `extensionData`**: The router already stores the payer in transient storage (`_getPayer()`). It could encode the originating `msg.sender` into `extensionData` so the extension can decode and check it. However, this is forgeable by any caller who constructs `extensionData` manually, so it requires a trusted router registry in the extension.
3. **Document the limitation**: At minimum, document that adding the router to the allowlist opens the pool to all router users, and that per-user allowlisting is only enforceable for direct pool calls.

## Proof of Concept

```
Setup:
  - Pool deployed with SwapAllowlistExtension.
  - Pool admin calls setAllowedToSwap(pool, router, true)
    to enable router-based swaps for allowlisted users.
  - Alice (address not in allowedSwapper) is NOT individually allowlisted.

Attack:
  1. Alice calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...}).
  2. Router calls pool.swap(recipient, zeroForOne, amount, ...).
     pool.swap sees msg.sender = router.
  3. Pool calls _beforeSwap(sender=router, ...).
  4. ExtensionCalling encodes beforeSwap(sender=router, ...) and calls SwapAllowlistExtension.
  5. Extension evaluates: allowedSwapper[pool][router] == true → check passes.
  6. Swap executes. Alice receives output tokens.

Result:
  Alice, a non-allowlisted user, successfully swaps on a curated pool.
  The allowlist is completely bypassed for any user who routes through MetricOmmSimpleRouter.

Foundry test plan:
  - Deploy pool with SwapAllowlistExtension as beforeSwap hook.
  - Deploy MetricOmmSimpleRouter.
  - Pool admin calls setAllowedToSwap(pool, address(router), true).
  - Prank as Alice (not individually allowlisted).
  - Call router.exactInputSingle targeting the pool.
  - Assert swap succeeds (no NotAllowedToSwap revert).
  - Assert Alice received output tokens despite not being on the allowlist.
```
