### Title
Unchecked Return Value of `transferFrom` in `replaceUsdcEWithUsdc()` Enables usdcE Theft Without USDC Deposit — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the return value. If the USDC token at the hardcoded address returns `false` on failure instead of reverting, the function continues execution, withdraws usdcE from the `DirectDepositV1` contract, and transfers it to the caller — all without the caller having deposited any USDC. This is a direct token-theft path callable by any unprivileged user on chain 57073.

---

### Finding Description

`replaceUsdcEWithUsdc()` is an `external` function with no access-control modifier beyond a chain-ID check. Its logic is:

1. Read the usdcE balance held by the `directDepositV1` contract for a given subaccount.
2. Pull USDC from `msg.sender` into `directDepositV1` via `transferFrom`.
3. Withdraw usdcE from `directDepositV1` into `ContractOwner`.
4. Send usdcE to `msg.sender`.

Step 2 uses a raw, unchecked `transferFrom`:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
``` [1](#0-0) 

The return value (`bool`) is silently discarded. If the call returns `false` (e.g., insufficient allowance, insufficient balance, or a non-reverting token implementation), execution continues uninterrupted into steps 3 and 4, which unconditionally drain usdcE and send it to the caller.

This is directly inconsistent with the rest of the codebase. The project defines a `safeTransferFrom` wrapper in `ERC20Helper` that validates the return value:

```solidity
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [2](#0-1) 

And the very next line in the same function correctly uses `safeTransfer`:

```solidity
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
``` [3](#0-2) 

The asymmetry — `safeTransfer` for the outbound leg, raw `transferFrom` for the inbound leg — is the root cause.

---

### Impact Explanation

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` (chain 57073, Ink mainnet) returns `false` on a failed transfer rather than reverting, an attacker with zero USDC allowance can:

- Call `replaceUsdcEWithUsdc(subaccount)` for any subaccount whose `directDepositV1` holds usdcE.
- Receive the full usdcE balance of that contract for free.
- Repeat for every subaccount with a non-zero usdcE balance.

The corrupted asset delta is: usdcE drained from `directDepositV1` contracts with no corresponding USDC deposited. The invariant broken is: *usdcE is only released after equivalent USDC is received*.

---

### Likelihood Explanation

The function is callable by any address on chain 57073 with no further restriction. The USDC address is hardcoded and specific to Ink mainnet — a relatively new chain where the token implementation may differ from canonical Circle USDC. Even if the current deployment reverts on failure, the pattern is one token upgrade or redeployment away from being exploitable. The attacker needs only to call the function with zero allowance and observe whether usdcE is received.

---

### Recommendation

Replace the raw `transferFrom` call with the project's own `safeTransferFrom` wrapper from `ERC20Helper`, which is already imported and used throughout `ContractOwner`:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [4](#0-3) 

This mirrors the pattern used everywhere else in the codebase and ensures a failed transfer reverts the entire transaction.

---

### Proof of Concept

1. Identify a subaccount on chain 57073 whose `directDepositV1Address[subaccount]` is non-zero and holds a usdcE balance > 0.
2. Call `ContractOwner.replaceUsdcEWithUsdc(subaccount)` from an EOA with **zero USDC allowance** to `ContractOwner`.
3. If USDC's `transferFrom` returns `false` instead of reverting:
   - `DirectDepositV1(directDepositV1).withdraw(usdcE)` executes, pulling usdcE into `ContractOwner`.
   - `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` executes, sending usdcE to the attacker.
4. Attacker receives usdcE tokens without depositing any USDC. [5](#0-4)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

**File:** core/contracts/libraries/ERC20Helper.sol (L23-42)
```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
