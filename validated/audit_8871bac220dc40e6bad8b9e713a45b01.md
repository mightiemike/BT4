### Title
Incorrect Ancestor Lookup in Difficulty Calculation Allows Fork Blocks with Insufficient PoW — (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

---

### Summary

When computing the expected difficulty for a fork block at a difficulty-adjustment boundary, `get_next_work_required` retrieves the interval's first block via `get_header_by_height`, which unconditionally returns the **mainchain** block at that height. During fork processing, the fork's actual ancestor at that height may carry a different timestamp. The difficulty calculation therefore uses a mismatched reference timestamp, producing an incorrect target. An unprivileged relayer can exploit this to submit a fork block whose `bits` field encodes an easier target than the protocol requires, and the contract will accept it.

---

### Finding Description

**Root cause — `get_header_by_height` always reads the mainchain map:** [1](#0-0) 

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

This function has no concept of "which chain am I on." It always resolves the height through `mainchain_height_to_header`.

**Bitcoin — called at every 2016-block retarget boundary:** [2](#0-1) 

```rust

let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);

    config,

```

The developers themselves flagged this with a `TODO`, confirming awareness of the ambiguity.

**Litecoin — same pattern at every 2016-block boundary:** [4](#0-3) 

**How the wrong timestamp corrupts the target:**

`calculate_next_work_required` (Bitcoin/Litecoin) and `calculate_next_work_required` (Dogecoin) both compute:

```
actual_time_taken = prev_fork_block.time − first_block_time   ← first_block_time is MAINCHAIN
``` [5](#0-4) [6](#0-5) 

If the mainchain block at `first_block_height` has an **earlier** timestamp than the fork's true ancestor at the same height, `actual_time_taken` is inflated, the computed target is raised (easier PoW), and the fork block's `bits` field is validated against this looser threshold. The contract then accepts a block that does not satisfy the protocol's actual difficulty requirement.

The broken invariant is:

> `expected_bits == block_header.bits` — where `expected_bits` must be derived from the fork's own ancestor chain, not the mainchain. [7](#0-6) 

---

### Impact Explanation

**Impact: Medium**

- A fork block carrying an under-difficulty `bits` value passes `check_pow` and is stored in `headers_pool`.
- If the fork accumulates enough chainwork it triggers `reorg_chain`, replacing the canonical chain.
- All downstream callers of `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` then operate against a chain that contains at least one block that did not satisfy the correct PoW requirement — the core security guarantee of the light client is broken.
- For Dogecoin (post-height 145 000, interval = 1), the wrong-ancestor lookup occurs for **every** fork block, not just at periodic boundaries, widening the attack surface significantly. [8](#0-7) 

---

### Likelihood Explanation

**Likelihood: Medium**

- No privileged role is required; `submit_blocks` is callable by any trusted relayer, and the trusted-relayer check can be bypassed by accounts holding `Role::UnrestrictedSubmitBlocks` or `Role::DAO`.
- The attacker must first mine and submit at least one valid fork block (to populate `headers_pool`) before the boundary block. This is a real cost but not a theoretical barrier.
- For Dogecoin the window is every single block after height 145 000; for Bitcoin/Litecoin it is every 2016-block epoch boundary — both are regularly occurring events.
- The attacker controls the timestamps of the fork blocks they submit, so engineering the timestamp divergence between the fork ancestor and the mainchain block at `first_block_height` is straightforward.

---

### Recommendation

Replace the `get_header_by_height` call inside `get_next_work_required` with a function that **walks the ancestor chain** from `prev_block_header` backwards through `get_prev_header` until it reaches `first_block_height`. This ensures the reference timestamp always belongs to the fork's own lineage, regardless of what the mainchain holds at that height.

```rust
// Correct approach: walk the fork's ancestor chain
let mut cursor = prev_block_header.clone();
while cursor.block_height > first_block_height {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
let first_block_time: i64 = cursor.block_header.time.into();
```

This is the approach used by Bitcoin Core and is the only way to guarantee that difficulty retargets are computed over the correct interval for any chain, including forks.

---

### Proof of Concept

**Dogecoin scenario (post-height 145 000, interval = 1):**

1. Mainchain has blocks at heights `h-2`, `h-1` with timestamps `T_main(h-2)` and `T_main(h-1)`.
2. Attacker submits a fork block at height `h-1` (diverging from the mainchain at `h-2`) with timestamp `T_fork(h-1)` and a valid PoW hash. This block is stored in `headers_pool` as a fork header.
3. Attacker crafts fork block at height `h`. The contract calls `get_next_work_required`:
   - `height_first = (h-1) - 1 = h-2`
   - `get_header_by_height(h-2)` → returns **mainchain** block with `T_main(h-2)`
   - `modulated_timespan = T_fork(h-1) − T_main(h-2)`
4. If `T_main(h-2) < T_fork(h-2)` (the fork's ancestor at `h-2` has a later timestamp than the mainchain block), then `T_fork(h-1) − T_main(h-2) > T_fork(h-1) − T_fork(h-2)`, inflating `modulated_timespan`.
5. `calculate_next_work_required` produces a higher (easier) target; `target_to_bits()` encodes it into `expected_bits`. [9](#0-8) 

6. The attacker's fork block at height `h` sets `bits` to this easier value and provides a PoW hash that satisfies the easier target but **not** the correct harder target.
7. `check_pow` passes (`expected_bits == block_header.bits` ✓, hash ≤ easier target ✓).
8. The block is accepted into `headers_pool`; if its chainwork exceeds the mainchain tip, `reorg_chain` is triggered and the canonical chain is corrupted. [10](#0-9)

// source https://github.com/dogecoin/dogecoin/blob/2c513d0172e8bc86fe9a337693b26f2fdf68a013/src/dogecoin.cpp#L41
fn calculate_next_work_required(
    config: &DogecoinConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let retarget_timespan = config.pow_target_timespan;
    let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;

    let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;

    let min_timespan = retarget_timespan - (retarget_timespan / 4);
    let max_timespan = retarget_timespan + (retarget_timespan / 2);

    if modulated_timespan < min_timespan {
        modulated_timespan = min_timespan;
    } else if modulated_timespan > max_timespan {
        modulated_timespan = max_timespan;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(modulated_timespan).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target =
        new_target / U256::from(<i64 as TryInto<u64>>::try_into(retarget_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()
