### Title
Insurance-Depletion Threshold in `_finalizeSubaccount` Allows Spot Liabilities to Be Socialized Instead of Liquidated — (File: `core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`ClearinghouseLiq._finalizeSubaccount` uses a boolean threshold `canLiquidateMore` to decide whether spot liabilities must be fully liquidated to zero before a subaccount is finalized. When the liquidatee's USDC debt exceeds the available insurance fund, this check is skipped and remaining negative spot balances are socialized across all depositors. An attacker who observes the insurance fund being depleted can time their finalization to exploit this threshold, having their spot liabilities written off while depositors absorb the loss — directly analogous to M-03's mass-slashing threshold bypass.

---

### Finding Description

In `_finalizeSubaccount`, the protocol computes:

```solidity
v.insurance = insurance;
v.insurance -= lastLiquidationFees;
v.canLiquidateMore = (quoteBalance.amount + v.insurance) > 0;

if (v.canLiquidateMore) {
    for (uint32 i = 1; i < v.spotIds.length; ++i) {
        ...
        require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
    }
}
``` [1](#0-0) 

When `canLiquidateMore = true`, every non-quote spot product balance must be exactly zero before finalization — meaning all spot liabilities must have been liquidated and paid back. When `canLiquidateMore = false` (the liquidatee's USDC debt exceeds the insurance fund), this `balance.amount == 0` check is entirely skipped.

The function then unconditionally calls `perpEngine.socializeSubaccount`, exhausts the insurance covering the quote debt, and — because `v.insurance` is always driven to zero when `canLiquidateMore` was false — calls `spotEngine.socializeSubaccount`:

```solidity
if (v.insurance <= 0) {
    spotEngine.socializeSubaccount(txn.liquidatee);
}
``` [2](#0-1) 

`SpotEngine.socializeSubaccount` handles each negative balance by reducing `cumulativeDepositsMultiplierX18` — permanently diluting every depositor's balance — and zeroing the subaccount's liability:

```solidity
state.cumulativeDepositsMultiplierX18 = (totalDeposited +
    balance.amount).div(state.totalDepositsNormalized);
...
_setBalanceAndUpdateBitmap(productId, subaccount,
    BalanceNormalized({amountNormalized: 0}));
``` [3](#0-2) 

The invariant broken is: **spot liabilities that should be liquidated (paid back by a liquidator at market price) are instead socialized (absorbed by all depositors) when the insurance threshold is crossed.** This is the direct Nado analog of M-03's "withdrawal classified as rewards" misclassification.

---

### Impact Explanation

An attacker who successfully triggers `canLiquidateMore = false` at finalization time has their negative spot balances written off at zero cost. The loss is redistributed to all depositors through a permanently reduced `cumulativeDepositsMultiplierX18`. The magnitude of the theft equals the market value of the socialized spot liabilities. Because `spotEngine.socializeSubaccount` iterates over all spot products, multiple borrowed assets can be written off in a single finalization call. [4](#0-3) 

---

### Likelihood Explanation

The condition `quoteBalance.amount + (insurance - lastLiquidationFees) <= 0` requires the liquidatee's USDC debt to exceed the available insurance. This mirrors M-03's requirement that >16.1% of validators be slashed. The attacker can:

1. Observe `insurance` on-chain (it is a public storage variable in `ClearinghouseStorage`).
2. Build a position where the USDC debt is large relative to the insurance fund.
3. Wait for a market stress event that depletes the insurance fund through other liquidations.
4. Use a separate account (self-liquidation is blocked by `require(txn.sender != txn.liquidatee)`) to call `liquidateSubaccountImpl` with `productId = type(uint32).max` at the precise moment `canLiquidateMore` flips to false.

    function socializeSubaccount(bytes32 subaccount) external {
        require(msg.sender == address(_clearinghouse), ERR_UNAUTHORIZED);

        uint32[] memory _productIds = getProductIds();
        for (uint128 i = 0; i < _productIds.length; ++i) {
            uint32 productId = _productIds[i];

            State memory state = states[productId];
            Balance memory balance = balanceNormalizedToBalance(
                state,
                balances[productId][subaccount]
            );
            if (balance.amount < 0) {
                int128 totalDeposited = state.totalDepositsNormalized.mul(
                    state.cumulativeDepositsMultiplierX18
                );

                state.cumulativeDepositsMultiplierX18 = (totalDeposited +
                    balance.amount).div(state.totalDepositsNormalized);

                require(state.cumulativeDepositsMultiplierX18 > 0);

                state.totalBorrowsNormalized += balance.amount.div(
                    state.cumulativeBorrowsMultiplierX18
                );

                _setBalanceAndUpdateBitmap(
                    productId,
                    subaccount,
                    BalanceNormalized({amountNormalized: 0})
                );
                _setState(productId, state);
            }
        }
    }
