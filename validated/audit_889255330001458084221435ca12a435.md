### Title
Missing Upper-Bound Validation on `protocol_reward_rate` Causes Panic in `calculate_reward` at Epoch Boundary — (`chain/epoch-manager/src/reward_calculator.rs`)

---

### Summary

`validate_genesis()` enforces upper-bound invariants on `online_max_threshold` (≤ 1) and `gas_price_adjustment_rate` (< 1), but applies **no upper-bound check** on `protocol_reward_rate`. A genesis config with `protocol_reward_rate > 1` (e.g., `[2, 1]`) passes full validation, then causes an unconditional `.unwrap()` panic inside `calculate_reward()` at the first epoch boundary, halting the chain.

---

### Finding Description

`GenesisConfig` stores `protocol_reward_rate` as a `Rational32` fraction with no type-level constraint. [1](#0-0) 

`validate_processed_records()` in `genesis_validate.rs` explicitly checks that `online_max_threshold ≤ 1` and `gas_price_adjustment_rate < 1`, but contains **no analogous check** for `protocol_reward_rate`: [2](#0-1) 

At every epoch boundary, `RewardCalculator::calculate_reward()` computes:

```rust
let epoch_protocol_treasury = epoch_total_reward * protocol_reward_rate;
let epoch_validator_reward =
    epoch_total_reward.checked_sub(epoch_protocol_treasury).unwrap();
``` [3](#0-2) 

When `protocol_reward_rate > 1`, `epoch_protocol_treasury > epoch_total_reward`, so `checked_sub` returns `None` and `.unwrap()` panics. There is no guard before this subtraction.

---

### Impact Explanation

A genesis (or testnet/localnet genesis) with `protocol_reward_rate: [2, 1]` passes `validate_genesis()` without error. At the first epoch boundary every node calls `calculate_reward()`, hits the `.unwrap()` on `None`, and crashes. Because all nodes process the same epoch boundary deterministically, the entire network halts. This is a **Critical** chain-halt: no block can be produced past the first epoch.

---

### Likelihood Explanation

The `protocol_reward_rate` field is a plain JSON array `[numerator, denominator]` with no schema-level maximum. Any operator configuring a new chain, forknet, or testnet who sets `protocol_reward_rate` to a value > 1 (e.g., `[1, 0]` or `[2, 1]`) will trigger the halt. The existing validation code already demonstrates the pattern for `online_max_threshold` and `gas_price_adjustment_rate`, making the omission for `protocol_reward_rate` a clear gap. The forknet template configs confirm `protocol_reward_rate` is a user-supplied field: [4](#0-3) 

---

### Recommendation

Add the following check inside `validate_processed_records()` in `core/chain-configs/src/genesis_validate.rs`, mirroring the existing `online_max_threshold` guard:

```rust
if self.genesis_config.protocol_reward_rate > Rational32::from_integer(1) {
    let error_message = format!(
        "Protocol reward rate must be less than or equal to 1, but current value is {}",
        self.genesis_config.protocol_reward_rate
    );
    self.validation_errors.push_genesis_semantics_error(error_message)
}
```

Similarly add a guard for `max_inflation_rate` to prevent economically irrational inflation values. [5](#0-4) 

---

### Proof of Concept

1. Construct a genesis JSON with `"protocol_reward_rate": [2, 1]` (200%).
2. Call `validate_genesis()` — it returns `Ok(())` with no error.
3. Run the chain to the first epoch boundary.
4. `RewardCalculator::calculate_reward()` is invoked; `epoch_protocol_treasury` = `2 * epoch_total_reward`.
5. `epoch_total_reward.checked_sub(epoch_protocol_treasury)` returns `None`.
6. `.unwrap()` panics — every node crashes simultaneously, chain halts.
