Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of End User, Allowing Any Caller to Bypass the Swap Allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of `pool.swap`. When `MetricOmmSimpleRouter` is used, `msg.sender` of `pool.swap` is the router contract, not the end user. Any pool admin who allowlists the router (the only way to support router-based trading for legitimate users) simultaneously grants every address — including explicitly blocked ones — the ability to bypass the allowlist.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // direct caller of pool.swap
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` uses that argument to enforce the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly without embedding the original `msg.sender` into `extensionData` or any other argument:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData   // no end-user identity embedded
    );
```

The `msg.sender` stored via `_setNextCallbackContext` is used only for the payment callback, not forwarded to the pool's extension hook. When this call reaches the pool, `msg.sender = router`. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end-user]`.

The same flaw applies to `exactOutputSingle` (L135-137) and all multi-hop paths (`exactInput` L103-112, `exactOutput` L165-181, and the recursive callback at L220-228).

## Impact Explanation
A curated pool's swap allowlist is completely ineffective for any user who routes through `MetricOmmSimpleRouter`. A non-allowlisted user calls `router.exactInputSingle(pool, ...)`. The router calls `pool.swap(...)`. The pool passes `sender = router` to the extension. If the router is allowlisted (the only way to support router-based trading for legitimate users), the check passes and the non-allowlisted user executes the swap. The pool's access-control policy is silently voided for all router-mediated swaps. For compliance-gated or institutional pools, unauthorized counterparties can trade against LP funds intended to be restricted to vetted participants, directly exposing LP principal to excluded counterparties.

## Likelihood Explanation
The router is the primary user-facing interface for the protocol. Pool admins who deploy curated pools and want their allowlisted users to trade through the router must allowlist the router. This is the natural and expected configuration. Once the router is allowlisted, the bypass is available to any address with no special privileges or setup required. The precondition (router allowlisted) is the normal operating state for any pool that supports router-based trading.

## Recommendation
The extension must check the identity of the economic actor, not the contract that called `pool.swap`. Two viable approaches:

1. **Router-forwarded identity in `extensionData`**: The router encodes `msg.sender` into `extensionData` for each hop. The extension decodes and checks that address. This requires a coordinated encoding convention between the router and the extension.

2. **Direct-pool-call requirement**: Document that `SwapAllowlistExtension` is incompatible with router-mediated swaps. The extension should enforce this by reverting when `sender` is a known router address, or pool admins must not allowlist the router and require users to call `pool.swap` directly.

## Proof of Concept

```
Setup:
  pool   = MetricOmmPool with SwapAllowlistExtension in BEFORE_SWAP_ORDER
  alice  = allowlisted user
  bob    = non-allowlisted user
  router = MetricOmmSimpleRouter

Admin actions:
  extension.setAllowedToSwap(pool, alice,  true)   // alice may swap
  extension.setAllowedToSwap(pool, router, true)   // router allowlisted so alice can use it

Attack (bob, no special privileges):
  router.exactInputSingle({
      pool:            pool,
      zeroForOne:      true,
      amountIn:        X,
      recipient:       bob,
      extensionData:   ""
  })

Call chain:
  router.exactInputSingle()
    → pool.swap(recipient=bob, ...) with msg.sender=router
      → _beforeSwap(sender=router, ...)
        → extension.beforeSwap(sender=router, ...)
            allowedSwapper[pool][router] == true   // ← router is allowlisted
            → returns selector (no revert)
      → swap executes, bob receives output tokens

Result: bob swaps on a curated pool despite being explicitly excluded from the allowlist.
```
