### Title
Fast Withdrawal Fee Truncates to Zero for Low-Decimal Tokens, Enabling Complete Fee Bypass — (`File: core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`fastWithdrawalFeeAmount` in `BaseWithdrawPool` computes the fast withdrawal fee in X18 fixed-point and then converts it back to native token units via integer division. For tokens with zero or very few decimals, the divisor (`multiplier`) is so large that the division always truncates to zero, causing the protocol to collect no fee and the user to receive the full withdrawal amount without paying anything.

---

### Finding Description

`fastWithdrawalFeeAmount` scales the withdrawal amount up to X18 precision, computes the fee, then scales back down:

```solidity
// BaseWithdrawPool.sol lines 139–148
uint8 decimals = token.decimals();
require(decimals <= MAX_DECIMALS);
int256 multiplier = int256(10**(MAX_DECIMALS - uint8(decimals)));
int128 amountX18 = int128(amount) * int128(multiplier);

int128 proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18);
int128 minFeeX18 = 5 * spotEngine().getConfig(productId).withdrawFeeX18;

int128 feeX18 = MathHelper.max(proportionalFeeX18, minFeeX18);
return feeX18 / int128(multiplier);   // <-- integer division truncates toward zero
```

`FAST_WITHDRAWAL_FEE_RATE = 1_000_000_000_000_000` (0.1%, i.e. `10^15` in X18).
`MAX_DECIMALS = 18`.

For a **0-decimal token** (`multiplier = 10^18`):

| `amount` (raw) | `proportionalFeeX18` | `feeX18 / 10^18` |
|---|---|---|
| 1 | `10^15` | **0** |
| 999 | `999 × 10^15` | **0** |
| 1000 | `10^18` | **1** |

The proportional fee rounds to zero for any withdrawal of fewer than 1,000 raw units. The `minFeeX18` path suffers identically: if `withdrawFeeX18 = 10^16` ($0.01), then `5 × 10^16 / 10^18 = 0`.

The same truncation applies to 1-decimal tokens (threshold: 100 whole tokens) and 2-decimal tokens (threshold: 10 whole tokens).

In `submitFastWithdrawal`, when `fee = 0`:

```solidity
// lines 104–111
if (sendTo == msg.sender) {
    require(transferAmount > uint128(fee), "Fee larger than balance");
    transferAmount -= uint128(fee);   // subtracts 0 — user keeps full amount
} else {
    safeTransferFrom(token, msg.sender, uint128(fee));  // transfers 0
}
fees[productId] += fee;   // increments by 0 — accounting corrupted
```

The user receives the full `transferAmount` and `fees[productId]` is never incremented.

---

### Impact Explanation

- **Corrupted state**: `fees[productId]` (the protocol's fee ledger) is never credited for fast withdrawals of low-decimal tokens below the rounding threshold.
- **Asset delta**: The user retains the fee amount that should have been deducted. For a 0-decimal token worth $1,000 each, a withdrawal of 999 tokens ($999,000) should incur a fee of ~0.999 tokens (~$999) but pays $0.
- **Systematic**: Every fast withdrawal of a low-decimal token below the threshold is fee-free. This is not occasional drift — it is a complete, repeatable bypass of the fast withdrawal fee mechanism.

---

### Likelihood Explanation

The protocol accepts any ERC20 token with `decimals <= MAX_DECIMALS` (18). There is no lower-bound check on decimals anywhere in the deposit or product registration path. Tokens with 0–2 decimals are uncommon but not hypothetical (some wrapped assets, synthetic tokens, or governance tokens use 0 decimals). Any user holding a valid signed withdrawal transaction for such a token can trigger this path permissionlessly via `submitFastWithdrawal`. No admin access, sequencer compromise, or social engineering is required.

**Likelihood: Medium** — depends on a low-decimal token being listed; the entry path is fully unprivileged once such a token exists.

---

### Recommendation

Enforce a minimum decimal count (e.g., `require(decimals >= 2)`) when registering a spot product, mirroring the Niftyswap resolution. Alternatively, round the fee **up** (ceiling division) instead of truncating, so that any non-zero X18 fee always results in at least 1 native unit being charged:

```solidity
// ceiling division: (feeX18 + multiplier - 1) / multiplier
int128 multiplierI = int128(multiplier);
return (feeX18 + multiplierI - 1) / multiplierI;
```

Document the minimum-decimal requirement in the product registration interface and code comments.

---

### Proof of Concept

**Setup**: Deploy a 0-decimal ERC20 token and register it as a spot product. A user holds 500 units and submits a valid signed `WithdrawCollateral` transaction for `amount = 500`.

**Execution**:
1. Caller invokes `submitFastWithdrawal(idx, transaction, signatures)`.
2. `fastWithdrawalFeeAmount` is called:
   - `decimals = 0`, `multiplier = 10^18`
   - `amountX18 = 500 × 10^18`
   - `proportionalFeeX18 = (10^15 × 500 × 10^18) / 10^18 = 500 × 10^15`
   - `feeX18 / 10^18 = 500 × 10^15 / 10^18 = 0` ← truncated
3. `fee = 0`; `transferAmount -= 0`; `fees[productId] += 0`.
4. User receives all 500 tokens. Protocol collects nothing.

**Expected**: User should pay `floor(500 × 0.001) = 0` tokens (still 0 at this scale, but for `amount = 1000` the fee should be 1 token and is correctly computed). The systematic issue is that the entire range `[1, 999]` raw units — which for a high-value 0-decimal token can represent hundreds of thousands of dollars — is permanently fee-exempt.
