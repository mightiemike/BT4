### Title
IBC NFT Receive Path Mints NFTs with Unvalidated TokenIDs, Permanently Locking Bridged Assets - (File: x/nft-transfer/keeper/packet.go)

### Summary
The `processReceivedPacket` function mints NFTs using tokenIDs taken verbatim from IBC packet data without validating them against the local `x/nft` module's strict format requirements. NFTs minted with non-conforming tokenIDs (e.g., numeric ERC-721 IDs like `1`, `2`, `3`) are permanently unusable through all standard Cosmos SDK messages. NFTs with empty tokenIDs are permanently locked with no recovery path at all.

### Finding Description

The `processReceivedPacket` function in `x/nft-transfer/keeper/packet.go` calls `k.nftKeeper.MintNFT(ctx, voucherClassID, tokenID, "", data.TokenUris[i], "", escrowAddress, receiver)` where `tokenID` is taken directly from `data.TokenIds[i]` in the incoming IBC packet. [1](#0-0) 

The `NonFungibleTokenPacketData.ValidateBasic()` only checks that the `TokenIds` slice is non-empty and that its length matches `TokenUris`. It does **not** validate individual tokenID values against any format constraint. [2](#0-1) 

Neither `MintNFT` nor `MintNFTUnverified` in the `x/nft` keeper performs any tokenID format validation — they only check denom existence and NFT uniqueness before storing. [3](#0-2) [4](#0-3) 

However, the `x/nft` module's `ValidateTokenID` enforces strict constraints: lowercase alphanumeric only (`[a-z0-9]+`), length 3–64, must begin with a letter. [5](#0-4) 

Every standard Cosmos SDK message that operates on an NFT calls `ValidateTokenID` at `ValidateBasic` time:
- `MsgTransferNFT.ValidateBasic()` → `ValidateTokenID(msg.Id)`
- `MsgBurnNFT.ValidateBasic()` → `ValidateTokenID(msg.Id)`
- `MsgEditNFT.ValidateBasic()` → `ValidateTokenID(msg.Id)` [6](#0-5) [7](#0-6) [8](#0-7) 

ICS-721 imposes no such restrictions on tokenIDs. Counterparty chains — especially Ethereum-based ICS-721 bridges — routinely use numeric tokenIDs (`1`, `2`, `3`), uppercase IDs, or IDs with hyphens/special characters. These all fail `ValidateTokenID`.

The `refundPacketToken` function (called on timeout/ack-error) has the identical flaw: it also calls `MintNFT` with unvalidated tokenIDs from packet data. [9](#0-8) 

**Empty tokenID sub-case**: `NonFungibleTokenPacketData.ValidateBasic()` does not check individual tokenID values for emptiness — only that the slice is non-empty. A packet with `TokenIds: [""]` passes validation. The resulting NFT with `tokenID = ""` is permanently stuck because `MsgTransfer.ValidateBasic()` also rejects empty tokenIDs (`strings.TrimSpace(tokenID) == ""`). [10](#0-9) 

### Impact Explanation

The corrupted invariant is: an NFT ownership record is written into the `x/nft` module store (supply incremented, owner set, NFT stored) but the NFT cannot be operated on through any standard Cosmos SDK message. The receiver's asset is locked.

- **Non-empty non-conforming tokenIDs** (e.g., `1`, `Token-1`, `MYTOKEN`): NFT is minted and owned by the receiver, but `MsgTransferNFT`, `MsgBurnNFT`, and `MsgEditNFT` all reject it. The receiver can only attempt an IBC transfer back via `MsgTransfer` (which only checks non-empty), but this requires the counterparty chain to still hold the escrowed original — if the counterparty state has changed, recovery fails.

- **Empty tokenIDs**: NFT is permanently locked. No message path — including IBC transfer — can recover it. The NFT supply counter is incremented with no way to decrement it.

### Likelihood Explanation

High. ERC-721 tokens universally use numeric tokenIDs (`1`, `2`, `3`, ...). Any ICS-721 bridge connecting an Ethereum-based chain to Cronos POS Chain will produce packets with numeric tokenIDs. The trigger is a normal, unprivileged IBC NFT transfer transaction. No special access or configuration is required. The `ValidateMsgTransferDecorator` ante handler only limits field lengths and only applies at `CheckTx` time, not at the IBC receive path. [11](#0-10) 

### Recommendation

1. Add per-tokenID format validation inside `NonFungibleTokenPacketData.ValidateBasic()` — or at minimum inside `processReceivedPacket` before calling `MintNFT` — to reject packets whose tokenIDs cannot be stored and used on this chain.
2. Alternatively, relax `ValidateTokenID` to accept any non-empty string (matching ICS-721 semantics), since the strict alphanumeric constraint is incompatible with cross-chain NFT interoperability.
3. Apply the same fix to `refundPacketToken`.

### Proof of Concept

1. On a counterparty ICS-721 chain, own an NFT with `tokenID = "1"` (standard ERC-721 numeric ID).
2. Submit `MsgTransfer` (IBC) to send it to Cronos POS Chain.
3. Counterparty chain escrows the NFT; IBC packet is relayed.
4. `OnRecvPacket` → `data.ValidateBasic()` passes: `TokenIds = ["1"]` is non-empty, lengths match. [12](#0-11) 
5. `processReceivedPacket` → `IssueDenom(voucherClassID, ...)` succeeds → `MintNFT(ctx, voucherClassID, "1", ...)` → `MintNFTUnverified` stores NFT with `tokenID = "1"`. No error returned. [13](#0-12) 
6. Receiver on Cronos POS Chain submits `MsgTransferNFT{Id: "1", DenomId: voucherClassID, ...}`.
7. `ValidateTokenID("1")` fails: length 1 < `MinDenomLen` 3, and `"1"` does not begin with a letter. [14](#0-13) 
8. All of `MsgTransferNFT`, `MsgBurnNFT`, `MsgEditNFT` are rejected. The NFT is locked in the receiver's account with no standard recovery path.

```go
// MintNFTUnverified mints an NFT without verifying if the owner is the creator of denom
// Needed during genesis initialization
func (k Keeper) MintNFTUnverified(ctx sdk.Context, denomID, tokenID, tokenNm, tokenURI, tokenData string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	if k.HasNFT(ctx, denomID, tokenID) {
		return sdkerrors.Wrapf(types.ErrNFTAlreadyExists, "NFT %s already exists in collection %s", tokenID, denomID)
	}

	k.setNFT(
		ctx, denomID,
		types.NewBaseNFT(
			tokenID,
			tokenNm,
			owner,
			tokenURI,
			tokenData,
		),
	)
	k.setOwner(ctx, denomID, tokenID, owner)
	k.increaseSupply(ctx, denomID)

	return nil
```

```go
// ValidateTokenID verify that the tokenID is legal
func ValidateTokenID(tokenID string) error {
	if len(tokenID) < MinDenomLen || len(tokenID) > MaxDenomLen {
		return sdkerrors.Wrapf(ErrInvalidTokenID, "the length of nft id(%s) only accepts value [%d, %d]", tokenID, MinDenomLen, MaxDenomLen)
	}
	if !IsBeginWithAlpha(tokenID) || !IsAlphaNumeric(tokenID) {
		return sdkerrors.Wrapf(ErrInvalidTokenID, "nft id(%s) only accepts lowercase alphanumeric characters, and begin with an english letter", tokenID)
	}
	return nil
```

```go
func (vtd ValidateMsgTransferDecorator) AnteHandle(ctx sdk.Context, tx sdk.Tx, simulate bool, next sdk.AnteHandler) (sdk.Context, error) {
	// avoid breaking consensus
	if !ctx.IsCheckTx() {
		return next(ctx, tx, simulate)
	}

	msgs := tx.GetMsgs()
	for _, msg := range msgs {
		transfer, ok := msg.(*nfttypes.MsgTransfer)
		if !ok {
			continue
		}

		if len(transfer.ClassId) > MaxClassIDLength {
			return ctx, newsdkerrors.Wrapf(sdkerrors.ErrInvalidRequest, "class id length must be less than %d", MaxClassIDLength)
		}

		if len(transfer.TokenIds) > MaxTokenIds {
			return ctx, newsdkerrors.Wrapf(sdkerrors.ErrInvalidRequest, "token id length must be less than %d", MaxTokenIds)
		}

		for _, tokenID := range transfer.TokenIds {
			if len(tokenID) > MaxTokenIDLength {
				return ctx, newsdkerrors.Wrapf(sdkerrors.ErrInvalidRequest, "token id length must be less than %d", MaxTokenIDLength)
			}
		}

		if len(transfer.Receiver) > MaximumReceiverLength {
			return ctx, newsdkerrors.Wrapf(sdkerrors.ErrInvalidRequest, "receiver length must be less than %d", MaximumReceiverLength)
		}
	}

	return next(ctx, tx, simulate)
}
```
