### Title
Unrestricted `run_mainchain_gc` Allows Any Caller to Prematurely Truncate the Canonical Chain — (`File: contract/src/lib.rs`)

### Summary
`run_mainchain_gc` is a public, state-mutating function that any unprivileged NEAR account can call with an arbitrary `batch_size`. There is no access-control guard beyond the `#[pause]` decorator, which only restricts calls when the contract is already paused. An attacker can invoke this function with a maximally large `batch_size` to prematurely evict valid mainchain headers from on-chain storage, permanently corrupting the canonical chain's accessible history and causing all SPV proof verifications for the affected blocks to fail.

### Finding Description
`run_mainchain_gc` is declared `pub` and is exposed as a NEAR contract method with only a `#[pause(except(roles(Role::UnrestrictedRunGC)))]` attribute: [1](#0-0) 

The `#[pause]` decorator only gates calls when the contract is in a paused state; it imposes no restriction on who may call the function during normal operation. The function accepts a caller-controlled `batch_size: u64` and uses it to drive a loop that:

1. Removes entries from `mainchain_height_to_header` and `mainchain_header_to_height` for every height in `[start_removal_height, end_removal_height)`.
2. Removes the corresponding `ExtendedHeader` from `headers_pool`.
3. Advances `mainchain_initial_blockhash` to the new oldest retained block. [2](#0-1) 

By contrast, `submit_blocks` — the only other caller of `run_mainchain_gc` — is protected by `#[trusted_relayer]`, which enforces that only staked, approved relayers may submit headers: [3](#0-2) 

`run_mainchain_gc` carries no equivalent guard, so the access-control boundary that protects header submission is entirely absent for GC.

### Impact Explanation
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` both require the target block to be present in `mainchain_header_to_height`: [4](#0-3) 

Once `run_mainchain_gc` removes a block's entries, any call to verify a transaction in that block panics with `"block does not belong to the current main chain"`. Because the removal is permanent (NEAR storage writes are committed), the corruption cannot be reversed without a contract migration. Consumer contracts that rely on SPV proofs — bridges, atomic-swap protocols, cross-chain lending — will receive hard failures for any transaction whose block was evicted.

### Likelihood Explanation
The entry path requires no special privilege: a single NEAR transaction calling `run_mainchain_gc(u64::MAX)` from any account suffices. The function is publicly listed in the contract ABI. The attack is cheap (one cross-contract call, minimal gas beyond storage refunds) and irreversible, making it attractive for a griefing or denial-of-proof attack against downstream consumers.

### Recommendation
Add an access-control guard to `run_mainchain_gc` that restricts direct external calls to authorized roles (e.g., `Role::DAO` or a dedicated `GCManager` role), mirroring the `#[trusted_relayer]` protection on `submit_blocks`. The internal invocation from `submit_blocks` (line 181) should remain unrestricted since it is already gated by `#[trusted_relayer]`. [5](#0-4) 

### Proof of Concept
1. Deploy the contract with a standard `InitArgs` (e.g., `gc_threshold = 1000`, 12 genesis blocks).
2. Have a legitimate relayer submit 1 000+ blocks so the mainchain has headers at heights 0–1 000.
3. From any unprivileged NEAR account, call:
   ```
   run_mainchain_gc(batch_size: 18446744073709551615)  // u64::MAX
   ```
4. The loop runs from `start_removal_height` up to `min(total_to_remove, u64::MAX)` = `total_to_remove`, evicting all headers below the `gc_threshold` window in a single call.
5. `mainchain_initial_blockhash` is now advanced to the newest retained block.
6. Any subsequent call to `verify_transaction_inclusion` for a block at an evicted height panics: `"block does not belong to the current main chain"`.
7. All downstream consumers of SPV proofs for those blocks receive permanent hard failures.
