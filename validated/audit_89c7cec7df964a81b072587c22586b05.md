### Title
Fee-on-Transfer Token Deposit Credits Inflated Collateral Balance - (File: core/contracts/EndpointStorage.sol)

---

### Summary

`EndpointStorage.handleDepositTransfer` performs two sequential token transfers using the caller-supplied `amount` for both legs, without measuring the actual tokens received. For fee-on-transfer tokens (e.g., USDT with an active fee), the clearinghouse receives fewer tokens than `amount`, but the slow-mode deposit transaction queued in `Endpoint.depositCollateralWithReferral` records the original `amount`. When that slow-mode transaction is later executed, `Clearinghouse.depositCollateral` credits the full `amount * multiplier` to the user's subaccount balance, creating a permanent over-credit relative to actual tokens held.

---

### Finding Description

The deposit entry point is `Endpoint.depositCollateralWithReferral`: [1](#0-0) 

It calls `handleDepositTransfer` with the raw user-supplied `amount`, then immediately enqueues a `DepositCollateral` slow-mode transaction that also encodes the same `amount`: [2](#0-1) 

`handleDepositTransfer` in `EndpointStorage` performs two transfers, both using the original `amount`: [3](#0-2) 

For a fee-on-transfer token:
- **Leg 1** (`safeTransferFrom`): the endpoint receives `amount − fee₁`, not `amount`.
- **Leg 2** (`safeTransferTo`): the endpoint attempts to forward the full `amount` to the clearinghouse. This succeeds only when the endpoint holds a pre-existing balance ≥ `fee₁`. When it does succeed, the clearinghouse receives `amount − fee₂` (fee charged again on the outbound transfer).

The endpoint accumulates the quote token over time through `chargeSlowModeFee`, which transfers slow-mode fees directly into `address(this)` and never forwards them: [4](#0-3) 

This pre-existing balance is the enabling condition that allows Leg 2 to succeed despite the shortfall from Leg 1.

When the queued slow-mode transaction is eventually executed, `Clearinghouse.depositCollateral` credits the full original `amount`: [5](#0-4) 

The `amountRealized` is computed directly from `txn.amount` (the original user-supplied value), not from the tokens actually received by the clearinghouse.

The same pattern is present in `DirectDepositV1.creditDeposit`, which reads the DDA's token balance and passes it verbatim as `amount` to `depositCollateralWithReferral`: [6](#0-5) 

---

### Impact Explanation

Each successful deposit of a fee-on-transfer token inflates the depositor's on-chain collateral balance by `fee₂ × multiplier` relative to the tokens actually held by the clearinghouse. Repeated deposits accumulate this shortfall. The `assertUtilization` check in `SpotEngine` guards withdrawals against over-utilization of deposits vs. borrows, but it does not compare the engine's recorded `totalDeposits` against the clearinghouse's actual ERC-20 balance: [7](#0-6) 

As a result, the clearinghouse's real token balance silently falls below the sum of all credited subaccount balances. Users who deposited legitimately may be unable to withdraw their full balance once the shortfall is large enough, constituting a solvency/accounting corruption.

---

### Likelihood Explanation

USDT currently has a 0% transfer fee, but the fee mechanism is live in its contract and can be activated by the USDT issuer at any time. If the Nado quote token is USDT (or any other fee-on-transfer token), the endpoint will accumulate that token from slow-mode fees charged via `chargeSlowModeFee`. Even a small accumulated balance (≥ `amount × fee_rate`) is sufficient to allow Leg 2 to succeed. Any unprivileged user calling `depositCollateral` or `depositCollateralWithReferral` is the attacker-controlled entry path; no special role is required.

---

### Recommendation

In `EndpointStorage.handleDepositTransfer`, measure the clearinghouse's token balance before and after the transfer and use the observed delta as the credited amount. Pass this measured amount into the queued `DepositCollateral` slow-mode transaction instead of the caller-supplied `amount`:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    uint256 actualReceived = token.balanceOf(address(clearinghouse)) - before;
    // return actualReceived to the caller for use in the DepositCollateral encoding
}
```

The returned `actualReceived` value should replace `amount` in the `DepositCollateral` struct encoded into the slow-mode transaction in `depositCollateralWithReferral`.

---

### Proof of Concept

1. USDT activates its transfer fee (e.g., 10 bps = 0.1%).
2. The endpoint has accumulated ≥ 1 USDT of pre-existing balance from slow-mode fees.
3. Alice calls `depositCollateral` with `amount = 1000 USDT`.
4. `handleDepositTransfer` executes:
   - Leg 1: endpoint receives `999 USDT` (1 USDT fee deducted).
   - Leg 2: endpoint sends `1000 USDT` to clearinghouse (uses 1 USDT from pre-existing balance); clearinghouse receives `999 USDT` (another 1 USDT fee deducted).
5. Slow-mode tx is queued with `amount = 1000 USDT`.
6. After the delay, the slow-mode tx is executed: `Clearinghouse.depositCollateral` credits Alice with `1000 × multiplier` units of collateral.
7. Alice's credited balance is `1000 × multiplier`, but the clearinghouse only holds `999 USDT` on Alice's behalf — a `1 USDT` shortfall per deposit.
8. Repeated deposits compound the shortfall, eventually making the clearinghouse insolvent for that token.
