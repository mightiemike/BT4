### Title
`op_unknown` cost-limit check performed on pre-multiplied value; u64 overflow in release mode returns undercharged cost — (`src/more_ops.rs`)

---

### Summary

`op_unknown` in `src/more_ops.rs` calls `check_cost(cost, max_cost)` on the **pre-multiplied** base cost, then multiplies `cost` by `(cost_multiplier + 1)`. In a release build (no `overflow-checks = true` in `Cargo.toml`), this multiplication can silently wrap to a value ≤ `u32::MAX`, which then passes the overflow guard and is returned as the operation's cost. The program is therefore charged far less than the true cost, consuming its remaining execution budget at a fraction of the intended rate.

---

### Finding Description

In `src/more_ops.rs`, `op_unknown` computes a base cost from the opcode's arguments (lines 209–256), then:

```
check_cost(cost, max_cost)?;          // line 260 — checks pre-multiplied cost
cost *= cost_multiplier + 1;          // line 261 — multiplies; can overflow u64
if cost > u32::MAX as u64 {           // line 262 — overflow guard
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))   // line 265 — returns wrapped cost
}
```

`cost_multiplier` is parsed from the opcode bytes as a `u32` (line 202–207), so `cost_multiplier + 1` reaches at most `2^32 = 4,294,967,296`. The pre-multiplied cost is bounded only by `max_cost` (Chia's block limit is ~11 billion, well above `u32::MAX = 4,294,967,295`). When the pre-multiplied cost exceeds `u32::MAX`, the multiplication at line 261 overflows `u64`.

Rust release builds wrap on overflow by default; `Cargo.toml` sets only `lto = "thin"` for the release profile — no `overflow-checks = true`. A wrapped result of 0 (or any value ≤ `u32::MAX`) passes the guard at line 262 and is returned as the operation's cost.

**Concrete example:**

| Variable | Value |
|---|---|
| `cost_multiplier` | `u32::MAX` = 4,294,967,295 |
| `cost_multiplier + 1` | 2^32 |
| Pre-multiplied cost T | 2^32 (achievable with cost_function=2 and ~1 MB arguments) |
| `check_cost(T, 11e9)` | passes |
| `T * 2^32 mod 2^64` | **0** |
| Overflow guard `0 > u32::MAX` | false |
| Returned cost | **0** |

The program is charged **0** for an operation whose true cost is 2^64.

---

### Impact Explanation

The cost mechanism is the sole DoS protection for CLVM execution on the Chia blockchain. Undercharging allows a crafted program to execute far more operations than `max_cost` permits. After the zero-cost unknown op, the remaining budget is unchanged, so the program can continue executing arbitrary additional operators. This is a consensus-critical undercharging vulnerability: a block containing such a program would be accepted by nodes running release builds while being correctly rejected (or differently costed) by any implementation that does not share the same overflow behavior, creating a potential consensus split.

---

### Likelihood Explanation

**Conditions required:**

1. Unknown ops must be permitted — true in consensus mode (`ChiaDialect` without `NO_UNKNOWN_OPS`), which is the mode used for block validation.
2. The attacker crafts opcode bytes to set `cost_multiplier = u32::MAX` — trivial, the bytes are attacker-controlled.
3. The pre-multiplied cost must be a multiple of `2^32` and ≤ `max_cost`. For `cost_function = 2` (MUL-like), two arguments of ~1 MB each produce a cost near `2^33 ≈ 8.6 billion < 11 billion`. The exact multiple can be hit by adjusting argument sizes.
4. The program must be included in a block — requires a valid transaction, but no privileged access.

The main constraint is the ~2 MB of argument data needed for `cost_function = 2`. This is large but not necessarily beyond Chia's block size limits. Likelihood is **Low-Medium** for a targeted attack, but the impact is severe enough to warrant a fix.

---

### Recommendation

Move the `check_cost` call to **after** the multiplication, so it validates the actual cost that will be returned:

```rust
// src/more_ops.rs, op_unknown
assert!(cost > 0);
// Remove: check_cost(cost, max_cost)?;
cost = cost.checked_mul(cost_multiplier + 1).ok_or(EvalErr::Invalid(o))?;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    check_cost(cost, max_cost)?;   // check the final, post-multiplied cost
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

Using `checked_mul` eliminates the overflow entirely and makes the intent explicit. Alternatively, add `overflow-checks = true` to `[profile.release]` in `Cargo.toml` as a defense-in-depth measure (this would panic rather than silently wrap, which is safer).

### Proof of Concept

**Attacker-controlled CLVM bytes:**

- Opcode: 5 bytes, last byte encodes `cost_function = 2` in bits 7–6 (i.e., `0x80`). First 4 bytes encode `cost_multiplier = u32::MAX = 0xFFFFFFFF`. Opcode = `[0xFF, 0xFF, 0xFF, 0xFF, 0x80]`.
- Arguments: two atoms each of size `L` bytes, chosen so that `MUL_BASE_COST + MUL_COST_PER_OP + (2L)*MUL_LINEAR_COST_PER_BYTE + L²/MUL_SQUARE_COST_PER_BYTE_DIVIDER = k * 2^32` for some integer `k` with `k * 2^32 ≤ max_cost`.

**Execution trace:**

1. `op_unknown` is called with the crafted opcode and two large atom arguments.
2. `cost_function = 2`, `cost_multiplier = u32::MAX`.
3. The MUL-like loop computes pre-multiplied cost T = k·2^32.
4. `check_cost(T, max_cost)` passes (T ≤ 11 billion).
5. `cost *= (u32::MAX + 1)` → `k·2^32 * 2^32 = k·2^64 ≡ 0 (mod 2^64)` in release mode.
6. `0 > u32::MAX` is false → returns `Ok(Reduction(0, nil))`.
7. Running cost in `run_program` is unchanged; program continues with full budget. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

        self.val_stack = vec![];
        self.op_stack = vec![];

        // max_cost is always in effect, and necessary to prevent wrap-around of
        // the cost integer.
        let max_cost = if max_cost == 0 { Cost::MAX } else { max_cost };
        // We would previously allocate an atom to hold the max cost for the program.
        // Since we don't anymore we need to increment the ghost atom counter to remain
        // backwards compatible with the atom count limit
        self.allocator.add_ghost_atom(1)?;
        let mut cost: Cost = 0;

        cost += self.eval_pair(program, env)?;

        loop {
            // if we are in a softfork guard, temporarily use the guard's
            // expected cost as the upper limit. This lets us fail early in case
            // it's wrong. It's guaranteed to be <= max_cost, because we check
            // that when entering the softfork guard
            let effective_max_cost = if let Some(sf) = self.softfork_stack.last() {
                sf.expected_cost
            } else {
                max_cost
            };

            if cost > effective_max_cost {
                return Err(EvalErr::CostExceeded);
            }
            let top = self.op_stack.pop();
            let op = match top {
                Some(f) => f,
                None => break,
            };
            cost += match op {
                Operation::Apply => self.apply_op(cost, effective_max_cost - cost)?,
                Operation::ExitGuard => self.exit_guard(cost)?,
                Operation::Cons => self.cons_op()?,
                Operation::SwapEval => self.swap_eval_op()?,
                Operation::RestoreAllocator => {
                    let Some(checkpoint) = self.allocator_stack.pop() else {
                        return Err(EvalErr::InternalError(
                            NodePtr::NIL,
                            "allocator checkpoint stack empty".to_string(),
                        ));
                    };
                    let Some(&top) = self.val_stack.last() else {
                        return Err(EvalErr::InternalError(
                            NodePtr::NIL,
                            "value stack empty".to_string(),
                        ));
                    };
                    match self.allocator.maybe_restore_with_node(&checkpoint, top)? {
                        MaybeRestore::NoReplace => {}
                        MaybeRestore::Replace(new_node) => {
                            self.val_stack.pop().unwrap();
                            self.val_stack.push(new_node);
                        }
                        MaybeRestore::Aborted => {}
                    }
                    0
                }
                #[cfg(feature = "pre-eval")]
                Operation::PostEval => {
                    let f = self.posteval_stack.pop().unwrap();
                    let peek: Option<NodePtr> = self.val_stack.last().copied();
                    f(self.allocator, peek);
                    0
                }
            };
        }
        self.allocator.clear_validation_caches();
        Ok(Reduction(cost, self.pop()?))
