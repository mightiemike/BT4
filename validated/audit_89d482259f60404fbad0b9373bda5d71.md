### Title
Checks-Effects-Interactions Violation in `withdrawCollateral()` Enables Reentrancy Drain with Callback Tokens ŌĆö (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral()` transfers tokens to the recipient **before** decrementing the subaccount balance in the `SpotEngine`. No reentrancy guard exists anywhere in the codebase. A malicious recipient contract using a callback-capable token (e.g., ERC777 `tokensReceived`) can re-enter `withdrawCollateral()` while the subaccount balance is still at its pre-withdrawal level, passing the health check repeatedly and withdrawing multiples of the actual balance.

---

### Finding Description

In `Clearinghouse.withdrawCollateral()`, the token transfer to the recipient occurs at line 408 via `handleWithdrawTransfer()`, **before** the subaccount balance is decremented at line 412 via `spotEngine.updateBalance()`, and before the health check at line 419. [1](#0-0) 

The transfer chain is:

```
Clearinghouse.handleWithdrawTransfer()
  ŌåÆ token.safeTransfer(withdrawPool, amount)          // Clearinghouse ŌåÆ WithdrawPool
  ŌåÆ BaseWithdrawPool.submitWithdrawal(token, sendTo, amount, idx)
      ŌåÆ BaseWithdrawPool.handleWithdrawTransfer()
          ŌåÆ token.safeTransfer(sendTo, amount)         // WithdrawPool ŌåÆ recipient ŌåÉ callback fires here
``` [2](#0-1) [3](#0-2) 

The `token.safeTransfer(sendTo, amount)` call in `BaseWithdrawPool.handleWithdrawTransfer()` uses a low-level `.call()` via `ERC20Helper.safeTransfer`: [4](#0-3) 

For an ERC777 token, this triggers the recipient's `tokensReceived` hook. At that moment:
- `spotEngine.updateBalance()` has **not** been called ŌĆö the subaccount balance is still the full pre-withdrawal amount.
- The health check has **not** been executed.
- No reentrancy guard exists anywhere in the codebase (confirmed: zero matches for `nonReentrant`, `ReentrancyGuard`, or any lock variable across all `.sol` files).

The attacker can pre-sign N withdrawal transactions with sequential nonces. Nonces are incremented in `validateNonce()` **before** the transfer occurs, so nonce N+1 is valid for the re-entrant call: [5](#0-4) 

The re-entrant call enters `withdrawCollateral()` with the full pre-withdrawal balance still in state, passes the health check, and transfers tokens again. This recurses until all pre-signed withdrawals are exhausted.

The `sendTo` address is attacker-controlled: for `WithdrawCollateral` (V1) it is derived from the subaccount address (attacker deploys a malicious contract there); for `WithdrawCollateralV2` it is explicitly set to any address: [6](#0-5) 

---

### Impact Explanation

An attacker with sufficient cross-margin collateral (e.g., ETH) to cover the resulting negative USDC balance after N re-entrant withdrawals can drain the `WithdrawPool` and `Clearinghouse` of the withdrawal token. Each re-entrant call sees the full pre-withdrawal balance, passes the health check, and transfers `amount` tokens. The corrupted state is the `SpotEngine` balance for the attacker's subaccount, which ends at `-(N-1) * amount` after all re-entrant calls complete ŌĆö covered by the attacker's cross-margin ETH collateral, so the final health check on the outermost call also passes.

---

### Likelihood Explanation

The attack requires the withdrawal token to support a transfer callback (ERC777 `tokensReceived`, or a custom hook). Standard USDC does not have this. However, the structural violation is unconditional: if any listed spot product token supports callbacks, the attack is immediately reachable by any unprivileged user with a subaccount. The `WithdrawCollateralV2` path makes `sendTo` fully attacker-controlled, removing any dependency on the attacker's own address.

---

### Recommendation

1. **Reorder operations in `withdrawCollateral()`**: call `spotEngine.updateBalance()` and the health check **before** `handleWithdrawTransfer()`, following the checks-effects-interactions pattern.
2. **Add a reentrancy guard**: apply OpenZeppelin's `ReentrancyGuard` (or equivalent) to `withdrawCollateral()` and `submitFastWithdrawal()`.

---

### Proof of Concept

1. Deploy a malicious ERC777 receiver contract. Its `tokensReceived` hook calls `Endpoint.submitTransactions()` with the next pre-signed withdrawal
