### Title
Quadratic Cost Computed Before Size Guard in `op_modpow` Enables Integer Overflow — (`File: src/more_ops.rs`)

---

### Summary

`op_modpow` (and its `op_modpow_malachite` variant) compute the quadratic cost expressions `(esize * esize)` and `(msize * msize)` — both unchecked `usize` multiplications — **before** the 256-byte size guard is applied. On 32-bit platforms, an attacker-supplied atom of ≥ 65 536 bytes causes `esize * esize` to wrap to zero in release builds and to panic in debug builds. This is a direct structural analog to the Umee `StandardDeviation` bug: arithmetic on attacker-controlled values is performed before the bounds check that would have rejected them.

---

### Finding Description

In `op_modpow` the execution order is:

1. Parse `exponent` → `esize` (`usize`, unchecked)
2. **Compute** `cost += (esize * esize) as Cost * MODPOW_COST_PER_BYTE_EXPONENT` — unchecked `usize` multiplication
3. Call `check_cost(cost, max_cost)?`
4. Parse `modulus` → `msize`
5. **Compute** `cost += (msize * msize) as Cost * MODPOW_COST_PER_BYTE_MOD` — unchecked `usize` multiplication
6. Call `check_cost(cost, max_cost)?`
7. **Only now**: `if bsize > 256 || esize > 256 || msize > 256 { return Err(...) }` [1](#0-0) 

The identical pattern is present in `op_modpow_malachite`: [2](#0-1) 

`int_atom` returns the raw `atom_len` as `usize` with no upper-bound enforcement: [3](#0-2) 

`Cost` is `u64`: [4](#0-3) 

On a **32-bit** target (`usize` = 32 bits), an atom of exactly 65 536 bytes gives:

```
esize = 65536
esize * esize = 65536 × 65536 = 4 294 967 296  →  wraps to 0  (mod 2³²)
```

In a **Rust debug build** (any platform) this is a guaranteed `panic!("attempt to multiply with overflow")`, crashing the evaluator process. In a release build on 32-bit the product silently wraps to `0`, the cost is undercharged, `check_cost` passes, and only then does the size guard fire — returning `InvalidOpArg` rather than the correct `CostExceeded`.

The quadratic cost constants confirm the intent: [5](#0-4) 

---

### Impact Explanation

An attacker who can submit a CLVM program containing a `modpow` call (opcode 60, part of the BLS/standard operator set) with a ≥ 65 536-byte exponent or modulus atom can:

- **Debug-build nodes (any architecture):** trigger an unconditional `panic` in the CLVM evaluator, crashing the full node or validator process. Because `modpow` is evaluated inside `EndBlocker`-equivalent block-validation logic, every node that attempts to validate the block crashes simultaneously — matching the Umee scenario exactly.
- **32-bit release-build nodes:** cause the quadratic cost to silently wrap to zero, bypassing the cost gate for that sub-expression. The size guard still fires, so the call is ultimately rejected, but the cost accounting is corrupted for the duration of the call frame. [6](#0-5) 

---

### Likelihood Explanation

`modpow` is a standard operator reachable by any CLVM program submitted to the network. Constructing a 65 536-byte atom is trivial (it is a valid serialized atom). Debug builds are routinely used by node operators, developers, and CI pipelines. The overflow condition requires no special privileges, no social engineering, and no dependency compromise — only a crafted CLVM byte string.
