### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Merkle Proof Forgery Mitigation — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase Merkle proof. However, the original `verify_transaction_inclusion` is still declared `pub` and is reachable by any unprivileged NEAR caller. An attacker can call the deprecated function directly, skipping the coinbase proof check entirely, and obtain a `true` inclusion result for a transaction that does not exist in the Bitcoin blockchain.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces two checks before delegating to the deprecated path:

1. `merkle_proof.len() == coinbase_merkle_proof.len()`
2. The coinbase transaction (at index `0`) must produce the block's `merkle_root` via `compute_root_from_merkle_proof` [1](#0-0) 

Only after both checks pass does it call the inner function:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [2](#0-1) 

The deprecated `verify_transaction_inclusion` is itself `pub` and decorated only with `#[pause]` — no access-control role restriction, no `#[private]`, no caller check: [3](#0-2) 

The `#[deprecated]` attribute is a Rust compiler hint; it has no effect on NEAR's on-chain dispatch. Any NEAR account can call `verify_transaction_inclusion` directly via a cross-contract call or a direct transaction, completely bypassing the coinbase proof gate.

The function's own documentation acknowledges the risk:

> **Warning**: This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash. [4](#0-3) 

The 64-byte forgery technique (referenced in the v2 docstring at line 327) allows an attacker to craft a 64-byte blob whose double-SHA256 hash equals an internal Merkle tree node. Because Bitcoin's Merkle tree does not distinguish leaf nodes from internal nodes by type, this crafted hash can be supplied as `tx_id` at the correct `tx_index` with a valid `merkle_proof`, and `compute_root_from_merkle_proof` will return the real block's `merkle_root`, causing the function to return `true`. [5](#0-4) 

---

### Impact Explanation

**High.** Any downstream consumer — a bridge, a DeFi protocol, or an oracle — that calls `verify_transaction_inclusion` and acts on a `true` result can be deceived into accepting a fabricated Bitcoin transaction as confirmed. This directly corrupts the contract's core security guarantee: that a `true` result means the supplied `tx_id` is a real, confirmed Bitcoin transaction in the canonical chain. Funds could be released or state transitions triggered for transactions that never occurred on Bitcoin.

---

### Likelihood Explanation

**Medium.** The entry path is completely open — no role, no stake, no privileged key is required. The 64-byte forgery technique is publicly documented (the bitmex blog post is cited in the contract's own source). An attacker needs only to identify a mainchain block, compute a valid internal-node forgery for that block's Merkle tree, and submit a direct NEAR transaction calling `verify_transaction_inclusion`. No off-chain infrastructure or social engineering is required.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public API. The simplest fix is to change its visibility from `pub` to `pub(crate)` so it is only callable internally by `verify_transaction_inclusion_v2`. Alternatively, add an explicit `env::panic_str` at the top of the function body directing callers to use v2, or gate it with an access-control role that no external account holds.

---

### Proof of Concept

```
Attacker (any NEAR account)
  │
  │  direct NEAR tx: verify_transaction_inclusion({
  │      tx_id:              <crafted 64-byte internal node hash>,
  │      tx_block_blockhash: <real mainchain block hash>,
  │      tx_index:           <position that makes the forged hash compute to merkle_root>,
  │      merkle_proof:       <valid sibling hashes from the real block>,
  │      confirmations:      1,
  │  })
  │
  ▼
BtcLightClient::verify_transaction_inclusion()
  - skips coinbase proof check (not present in v1)
  - calls compute_root_from_merkle_proof(crafted_hash, index, proof)
  - result == real block's merkle_root  ✓
  - returns true  ← fabricated transaction accepted as confirmed
  │
  ▼
Downstream consumer (bridge / DeFi protocol)
  - receives `true`
  - releases funds / updates state for a non-existent Bitcoin transaction
```

Steps:
1. Pick any block stored in `mainchain_header_to_height` (publicly readable via `get_block_hash_by_height`).
2. Retrieve its `merkle_root` from `headers_pool` (via `get_last_block_header` or equivalent).
3. Using the known 64-byte forgery technique, construct a `tx_id` and `tx_index` such that `compute_root_from_merkle_proof(tx_id, tx_index, proof) == merkle_root`.
4. Call `verify_transaction_inclusion` directly — not `verify_transaction_inclusion_v2` — with the crafted arguments.
5. The function returns `true`; the coinbase proof gate in v2 was never reached. [6](#0-5) [7](#0-6)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
