### Title
No Minimum Output Slippage Guard on `mintNlp` / `burnNlp` — `oraclePriceX18` Excluded from User Signature (`Clearinghouse.sol`)

---

### Summary

The `mintNlp` and `burnNlp` functions in `Clearinghouse.sol` convert between quote tokens and NLP tokens using an `oraclePriceX18` value that is **appended by the sequencer** and is **not covered by the user's EIP-712 signature**. There is no minimum output check. A user who signs a `MintNlp` or `BurnNlp` transaction has no on-chain guarantee about the conversion rate that will be applied, and no way to enforce a minimum amount received.

---

### Finding Description

The `SignedMintNlp` and `SignedBurnNlp` structs each carry an `oraclePriceX18` field that is **outside** the signed payload:

```solidity
// IEndpoint.sol
struct SignedMintNlp {
    MintNlp tx;          // ← only this is signed
    bytes signature;
    int128 oraclePriceX18;       // ← NOT signed
    int128[] nlpPoolRebalanceX18; // ← NOT signed
}

struct SignedBurnNlp {
    BurnNlp tx;          // ← only this is signed
    bytes signature;
    int128 oraclePriceX18;       // ← NOT signed
    int128[] nlpPoolRebalanceX18; // ← NOT signed
}
``` [1](#0-0) 

The EIP-712 digest in `Verifier.sol` confirms that `oraclePriceX18` is excluded from both signatures:

```solidity
// MINT_NLP_SIGNATURE = "MintNlp(bytes32 sender,uint128 quoteAmount,uint64 nonce)"
digest = keccak256(abi.encode(
    keccak256(bytes(MINT_NLP_SIGNATURE)),
    signedTx.tx.sender,
    signedTx.tx.quoteAmount,
    signedTx.tx.nonce          // oraclePriceX18 absent
));

// BURN_NLP_SIGNATURE = "BurnNlp(bytes32 sender,uint128 nlpAmount,uint64 nonce)"
digest = keccak256(abi.encode(
    keccak256(bytes(BURN_NLP_SIGNATURE)),
    signedTx.tx.sender,
    signedTx.tx.nlpAmount,
    signedTx.tx.nonce          // oraclePriceX18 absent
));
``` [2](#0-1) 

The sequencer then passes this unsigned price directly into `Clearinghouse.mintNlp` / `burnNlp`, where it is used as the sole conversion rate with no floor check:

```solidity
// mintNlp — user pays quoteAmount, receives nlpAmount
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, nlpAmount);
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -quoteAmount);
``` [3](#0-2) 

```solidity
// burnNlp — user burns nlpAmount, receives quoteAmount
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
``` [4](#0-3) 

There is no `minNlpOut` or `minQuoteOut` parameter anywhere in the call chain. The sequencer supplies the price; the user's signature commits only to the input amount and nonce.

---

### Impact Explanation

**`mintNlp`**: A user signs to pay `quoteAmount`. If the sequencer submits an inflated `oraclePriceX18`, `nlpAmount = quoteAmount / oraclePriceX18` is reduced. The user's quote balance is debited in full while they receive fewer NLP tokens than the prevailing market rate would imply — a direct loss of value.

**`burnNlp`**: A user signs to burn `nlpAmount`. If the sequencer submits a deflated `oraclePriceX18`, `quoteAmount = nlpAmount × oraclePriceX18` is reduced. The user's NLP balance is burned in full while they receive fewer quote tokens — a direct loss of assets.

In both cases the corrupted state delta is the user's `QUOTE_PRODUCT_ID` or `NLP_PRODUCT_ID` balance in `SpotEngine`, and the loss is bounded only by the magnitude of the price deviation.

---

### Likelihood Explanation

The sequencer is the only party that can supply `oraclePriceX18`. Exploitation therefore requires a malicious or negligent sequencer. However, the design flaw is structural: the user's signature provides **zero on-chain commitment** to the conversion rate. There is no mechanism — not a deadline, not a minimum output, not a price tolerance — by which a user can protect themselves. Any future sequencer key compromise, operator error, or protocol upgrade that relaxes sequencer trust assumptions immediately exposes every pending `MintNlp`/`BurnNlp` transaction to this attack.

---

### Recommendation

Add a `minAmountOut` field to both `MintNlp` and `BurnNlp` structs and include it in the EIP-712 signed digest:

```solidity
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;
    uint128 minNlpAmount;   // ← new: minimum NLP to receive
    uint64 nonce;
}

struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint128 minQuoteAmount; // ← new: minimum quote to receive
    uint64 nonce;
}
```

Then enforce the check inside `Clearinghouse.mintNlp` / `burnNlp`:

```solidity
require(nlpAmount >= int128(txn.minNlpAmount), "slippage");
require(quoteAmount >= int128(txn.minQuoteAmount), "slippage");
```

---

### Proof of Concept

1. User signs `MintNlp { sender, quoteAmount = 10_000e18, nonce = 5 }` expecting the current NLP price of `1e18` (1:1), so they expect `10_000` NLP tokens.
2. Sequencer constructs `SignedMintNlp` with `oraclePriceX18 = 2e18` (double the fair price).
3. `Clearinghouse.mintNlp` computes `nlpAmount = 10_000e18 / 2e18 = 5_000`.
4. User's quote balance is debited `10_000e18`; their NLP balance is credited only `5_000` — a 50 % loss of value with no revert, no warning, and no recourse.

The same attack applies symmetrically to `burnNlp` by deflating `oraclePriceX18`.
