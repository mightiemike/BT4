### Title
Wrong Receiver Used for `DeterministicStateInit` Domain Check Inside Delegated Actions — (File: `runtime/runtime/src/action_validation.rs`)

### Summary

`validate_delegate_action` validates inner `DeterministicStateInitAction` against the **outer transaction's receiver** (`receiver` parameter — the relayer's target, i.e. the meta-tx sender's account) instead of the **delegate action's own receiver** (`delegate_action.receiver_id()` — the actual deterministic account to initialize). This is the direct nearcore analog of the external bug: the wrong identity is used for a domain/authorization check, causing a legitimate operation to be permanently blocked.

### Finding Description

In `validate_delegate_action` the variable `inner_receiver` is selected as follows before `FixDelegatedDeterministicStateInit` is active:

```rust
// runtime/runtime/src/action_validation.rs  lines 188-199
let inner_receiver =
    if ProtocolFeature::FixDelegatedDeterministicStateInit.enabled(current_protocol_version) {
        delegate_action.receiver_id()   // correct
    } else {
        receiver                        // BUG: outer tx receiver, not delegate receiver
    };
validate_actions_with_mode(limit_config, &actions, inner_receiver, ...)?;
``` [1](#0-0) 

`validate_actions_with_mode` calls `validate_deterministic_state_init`, which enforces the invariant `derive_near_deterministic_account_id(state_init) == receiver_id`:

```rust
// runtime/runtime/src/action_validation.rs  lines 420-427
let derived_id = derive_near_deterministic_account_id(&action.state_init);
if derived_id != *receiver_id {
    return Err(ActionsValidationError::InvalidDeterministicStateInitReceiver { ... });
}
``` [2](#0-1) 

In a meta-transaction the outer tx receiver is the **sender** of the delegate action (e.g. `det_account_b`), while the delegate action's receiver is the **target deterministic account** (e.g. `det_account_c`). The pre-fix code checks `derive(state_init_c) == det_account_b`, which always fails for any legitimate cross-account meta-tx carrying `DeterministicStateInit`. The result is that every such transaction is rejected at the tx-validation stage with `InvalidDeterministicStateInitReceiver`, making the feature entirely unusable via meta-transactions.

The divergent values are exact:

| Field | Pre-fix value used | Correct value |
|---|---|---|
| `inner_receiver` | `receiver` (outer tx `receiver_id` = meta-tx sender) | `delegate_action.receiver_id()` (deterministic target) |

The protocol-feature boundary where the wrong branch is taken is `ProtocolFeature::FixDelegatedDeterministicStateInit`: [3](#0-2) 

### Impact Explanation

**High.** Any user who attempts to initialize a deterministic account via a meta-transaction (a fully supported and documented flow) is permanently blocked. The `DeterministicStateInit` action is designed to be composable with `DelegateAction` (NEP-616 + NEP-366), but the wrong-receiver check makes the combination impossible before the fix is activated. Funds attached to such transactions are not lost (the tx is rejected before execution), but the intended operation cannot be completed, mirroring the "funds locked in vault" pattern of the external report.

### Likelihood Explanation

**High.** The condition is deterministic: every meta-transaction carrying `DeterministicStateInit` targeting a different account than the outer tx receiver triggers the wrong check. No special attacker capability is required; any unprivileged user constructing a valid meta-tx hits this path. The integration test `test_deterministic_state_init_meta_tx_receiver_check_pre_fix` confirms the failure is reproducible: [4](#0-3) 

### Recommendation

Use `delegate_action.receiver_id()` as `inner_receiver` unconditionally (the fix already applied under `FixDelegatedDeterministicStateInit`):

```diff
- receiver
+ delegate_action.receiver_id()
```

The legacy branch (`receiver`) should be removed once the protocol version floor is raised past `FixDelegatedDeterministicStateInit`.

### Proof of Concept

The exploit scenario is fully exercised by `try_meta_tx_deterministic_receiver_exploit`: [5](#0-4) 

Concretely:
1. Deploy global contract; derive `det_account_b = derive(state_init_b)`.
2. Add a full-access key to `det_account_b` so it can act as meta-tx sender.
3. Craft a `DelegateAction` with `sender_id = det_account_b`, `receiver_id = det_account_a` (a different deterministic account), wrapping `DeterministicStateInit { state_init: state_init_b }`.
4. Wrap in an outer tx with `receiver_id = det_account_b`.
5. Pre-fix: outer tx validation checks `derive(state_init_b) == det_account_b` → passes (wrong identity used). The receipt validation then catches the mismatch and fails with `InvalidDeterministicStateInitReceiver`.
6. For the **legitimate** direction (user wants to initialize `det_account_c` from `det_account_b`): outer tx validation checks `derive(state_init_c) == det_account_b` → always fails, blocking the operation entirely.

fn try_meta_tx_deterministic_receiver_exploit(
    protocol_version: ProtocolVersion,
) -> Result<FinalExecutionOutcomeView, InvalidTxError> {
    let mut env = TestEnv::setup_with_version(Balance::from_near(100), protocol_version);
    env.deploy_global_contract(GlobalContractDeployMode::AccountId);

    let (_state_init_a, det_account_a) = env.new_deterministic_account_with_data(small());
    let (state_init_b, det_account_b) = env.new_deterministic_account_with_data(big());
    assert_ne!(det_account_a, det_account_b);

    // Deploy det_account_b and add a full-access key so it can act as meta_tx_sender.
    let user_signer = create_user_test_signer(&env.user_account());
    let storage_balance = env.balance_for_storage(state_init_b.clone());
    let deploy_tx = SignedTransaction::deterministic_state_init(
        env.next_nonce(),
        env.user_account(),
        det_account_b.clone(),
        &user_signer,
        env.get_tx_block_hash(),
        state_init_b.clone(),
        storage_balance,
    );
    env.run_tx(deploy_tx);

    let meta_tx_sender_signer = create_user_test_signer(&det_account_b);
    let pk_base64 = near_primitives_core::serialize::to_base64(
        &borsh::to_vec(&meta_tx_sender_signer.public_key()).unwrap(),
    );
    let add_key_args = serde_json::json!([
        { "batch_create": { "account_id": det_account_b.as_str() }, "id": 0 },
        {
            "action_add_key_with_full_access": {
                "promise_index": 0,
                "public_key": pk_base64,
                "nonce": 0
            },
            "id": 0,
            "return": true
        }
    ]);
    let add_key_tx = SignedTransaction::call(
        env.next_nonce(),
        env.user_account(),
        det_account_b.clone(),
        &user_signer,
        Balance::from_near(2),
        "call_promise".to_owned(),
        serde_json::to_vec(&add_key_args).unwrap(),
        Gas::from_teragas(300),
        env.get_tx_block_hash(),
    );
    env.run_tx(add_key_tx);

    // Craft the exploit: outer_tx.receiver = det_account_b = derive(state_init_b).
    // Old check: det_account_b == derive(state_init_b) passes.
    // The delegate action targets det_account_a, which is the wrong account.
    // In no protocol version can this ever be allowed to be executed successfully.
    let relayer = env.independent_account();
    let relayer_signer = create_user_test_signer(&relayer);
    let inner_action = Action::DeterministicStateInit(Box::new(DeterministicStateInitAction {
        state_init: state_init_b,
        deposit: Balance::ZERO,
    }));
    let delegate_nonce = env.next_nonce_for(&det_account_b);
    let delegate_action = DelegateAction {
        sender_id: det_account_b.clone(),
        receiver_id: det_account_a,
        actions: vec![NonDelegateAction::try_from(inner_action).unwrap()],
        nonce: delegate_nonce,
        max_block_height: 1_000_000,
        public_key: meta_tx_sender_signer.public_key(),
    };
    let signed_delegate_action =
        SignedDelegateAction::sign(&meta_tx_sender_signer, delegate_action);
    let tx = SignedTransaction::from_actions(
        env.next_nonce(),
        relayer,
        det_account_b,
        &relayer_signer,
        vec![Action::Delegate(Box::new(signed_delegate_action))],
        env.get_tx_block_hash(),
    );
    env.try_execute_tx(tx)
}
