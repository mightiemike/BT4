### Title
ERC4626 Share-Price Inflation Causes Silent Collateral Lock in `wrapVaultAsset` — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.wrapVaultAsset()` gates the entire vault-wrapping deposit on `previewDeposit(assetBalance) != 0`. An attacker who inflates the ERC4626 vault's share price (classic first-depositor inflation attack) causes this check to return `0` for any victim whose underlying asset balance in their `DirectDepositV1` contract is below the inflated threshold. The deposit is silently skipped, permanently locking the victim's underlying assets in the `DirectDepositV1` contract with no automated recovery path.

---

### Finding Description

`ContractOwner.wrapVaultAsset()` is the on-chain entry point for wrapping an ERC4626 underlying asset (held in a user's `DirectDepositV1` contract) into vault shares, which are then credited to the user's Nado subaccount.

The critical guard is at line 525:

```solidity
if (IERC4626Base(tokenAddr).previewDeposit(assetBalance) != 0) {
    DirectDepositV1(directDepositV1).withdraw(IIERC20Base(assetTokenAddr));
    ...
    IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1);
}
``` [1](#0-0) 

If `previewDeposit(assetBalance)` returns `0`, the entire body is skipped — no withdrawal from the DDA, no vault deposit, no credit to the subaccount. The function returns silently with the user's assets still sitting in the `DirectDepositV1` contract.

`previewDeposit` returns `0` when the ERC4626 vault's share price has been inflated such that `shares = totalSupply * assetBalance / totalAssets` rounds down to zero. This is the exact condition produced by the classic inflation attack:

1. Attacker calls `vault.deposit(1, attacker)` → `totalSupply = 1`, `totalAssets = 1`
2. Attacker directly transfers `N - 1` underlying tokens to the vault address → `totalAssets = N`, `totalSupply = 1`
3. For any victim with `assetBalance < N`, `previewDeposit(assetBalance) = 1 * assetBalance / N = 0`

`wrapVaultAsset` has **no access control** — it is `external` with no `onlyOwner` or similar modifier, so any caller can invoke it for any subaccount/productId pair. [2](#0-1) 

The `DirectDepositV1.withdraw()` function that could rescue the assets is `onlyOwner`, meaning only the protocol multisig can recover them. [3](#0-2) 

---

### Impact Explanation

A victim user who sends underlying assets to their `DirectDepositV1` address and calls `wrapVaultAsset` (or has it called on their behalf) will have their assets permanently stuck in the DDA contract. The assets are never wrapped into vault shares, never credited to their Nado subaccount, and cannot be used as collateral. The only recovery path is `withdrawFromDirectDepositV1`, which is `onlyOwner` and requires admin intervention. [4](#0-3) 

The corrupted state delta: `directDepositV1.balanceOf(assetToken) > 0` indefinitely, while `spotEngine.balances[productId][subaccount].amountNormalized == 0` — the user's collateral is unaccounted for in the protocol.

---

### Likelihood Explanation

Medium. The attack requires the ERC4626 vault integrated as a Nado spot product to lack virtual-shares inflation protection (e.g., no `_decimalsOffset` as in OpenZeppelin's updated ERC4626). Many deployed ERC4626 vaults (especially older or simpler implementations) are susceptible. The attacker needs capital equal to the inflation amount but can target any user whose deposit is below that threshold. The `wrapVaultAsset` function is permissionless, so the attacker can also front-run the victim's deposit to ensure the inflation is in place before `wrapVaultAsset` is called.

---

### Recommendation

1. **Remove the `previewDeposit != 0` guard as a deposit gate.** Instead, attempt the deposit unconditionally and revert (or handle) if the vault itself rejects it. Alternatively, revert with a descriptive error if `previewDeposit` returns 0, so the caller is informed rather than silently losing their deposit opportunity.
2. **Require that any ERC4626 vault integrated as a Nado product implements virtual-shares inflation protection** (e.g., OpenZeppelin's `_decimalsOffset` mechanism or equivalent dead-shares seeding at deployment).
3. **Add a minimum-shares-out check** after the vault deposit call to ensure the user received a non-zero share amount.

---

### Proof of Concept

**Setup:** ERC4626 vault `V` with underlying asset `U` is registered as a Nado spot product. Victim `Alice` sends `1000e6` units of `U` to her `DirectDepositV1` address.

**Attack:**
1. Attacker calls `V.deposit(1, attacker)` → `V.totalSupply = 1`, `V.totalAssets = 1`
2. Attacker transfers `10_000e6 - 1` units of `U` directly to `V` → `V.totalAssets = 10_000e6`, `V.totalSupply = 1`
3. Anyone calls `ContractOwner.wrapVaultAsset(alice_subaccount, productId)`
4. `assetBalance = 1000e6` (Alice's balance in her DDA)
5. `previewDeposit(1000e6) = 1 * 1000e6 / 10_000e6 = 0` (integer division)
6. The `if` block at line 525 is not entered — Alice's `1000e6` U tokens remain in her DDA, unprocessed
7. Alice's Nado subaccount receives zero collateral credit [5](#0-4) [6](#0-5)
