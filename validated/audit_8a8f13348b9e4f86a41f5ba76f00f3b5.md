### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables Draining of usdcE from DirectDepositV1 Accounts — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` makes a raw, unchecked `transferFrom` call to pull USDC from the caller before withdrawing usdcE from a `DirectDepositV1` (DDA) contract. If the USDC token returns `false` instead of reverting on failure (ERC20-compliant behavior), execution continues and the caller receives usdcE from the DDA without having provided any USDC. The function has no access control beyond a chain ID check, making it callable by any unprivileged user on chain 57073.

---

### Finding Description

`ContractOwner.replaceUsdcEWithUsdc()` is a migration helper intended to swap usdcE held in a DDA for USDC. Its logic is:

1. Read `balance = usdcE.balanceOf(directDepositV1)`
2. Pull `balance` USDC from `msg.sender` into the DDA
3. Withdraw usdcE from the DDA to `ContractOwner`
4. Forward the usdcE to `msg.sender`

Step 2 is implemented as a bare `transferFrom` call:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
``` [1](#0-0) 

The return value is never checked. The rest of the codebase exclusively uses `ERC20Helper.safeTransferFrom()` for this purpose, which wraps the call and enforces a `require` on the return value:

```solidity
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [2](#0-1) 

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` returns `false` on a failed transfer (e.g., caller has no allowance or insufficient balance) rather than reverting — which is valid per EIP-20 — execution proceeds to steps 3 and 4 regardless. `ContractOwner` is the `Ownable` owner of every DDA it deploys, so `DirectDepositV1.withdraw()` succeeds unconditionally from this call path. [3](#0-2) 

The function carries no `onlyOwner` or `onlyDeployer` modifier. The sole gate is:

```solidity
require(block.chainid == 57073, ERR_UNAUTHORIZED);
``` [4](#0-3) 

Any unprivileged caller on chain 57073 (Ink) can invoke it for any `subaccount` that has a registered DDA with a non-zero usdcE balance.

---

### Impact Explanation

An unprivileged attacker on chain 57073 can drain the entire usdcE balance from any `DirectDepositV1` contract by calling `replaceUsdcEWithUsdc(subaccount)` without holding or approving any USDC. The usdcE drained represents user collateral pending deposit into the Nado protocol — its loss is a direct, irreversible asset theft from depositors. The `safeTransfer` at line 618 uses the safe wrapper and will succeed as long as `ContractOwner` holds the usdcE (which it will, after the unchecked withdraw), so the attacker receives the tokens cleanly. [5](#0-4) 

---

### Likelihood Explanation

The USDC token address is hardcoded to a specific deployment on chain 57073. Whether that deployment returns `false` or reverts on failure determines exploitability. The ERC20 standard explicitly permits returning `false`, and the ENS sponsor confirmed the same class of risk: *"to be ERC20 compliant we must check the return value."* Additionally, the function is entirely permissionless for any caller on the target chain, maximizing the attack surface. The inconsistency with the rest of the codebase — which uniformly uses `safeTransferFrom` — indicates this was an oversight rather than an intentional design choice. [6](#0-5) 

---

### Recommendation

Replace the bare `transferFrom` call with `ERC20Helper.safeTransferFrom`, consistent with every other token transfer in the codebase:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`IERC20Base` is already imported and `ERC20Helper` is already applied via `using ERC20Helper for IERC20Base` at the top of `ContractOwner`. [7](#0-6) 

---

### Proof of Concept

**Setup:** Deploy a mock USDC on chain 57073 that returns `false` instead of reverting when `transferFrom` is called with insufficient allowance (ERC20-compliant).

**Steps:**

1. A DDA exists for `subaccount` with `usdcE.balanceOf(dda) = X > 0`.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` with zero USDC allowance.
3. `balance = X` is read from the DDA.
4. `usdc.transferFrom(attacker, dda, X)` returns `false` — no USDC moves. Return value is ignored.
5. `DirectDepositV1(dda).withdraw(usdcE)` executes — `ContractOwner` is the DDA owner, so this succeeds. `X` usdcE is transferred to `ContractOwner`.
6. `usdcE.safeTransfer(attacker, X)` executes — `X` usdcE is transferred to the attacker.

**Result:** Attacker receives `X` usdcE from the DDA at zero cost. The DDA is drained. The user whose collateral was held in the DDA loses their funds before they are credited to the protocol. [8](#0-7) [9](#0-8)

        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
