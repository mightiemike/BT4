### Title
`action_delete_account` Omits Global-Contract Identifier from Storage-Usage Check, Incorrectly Blocking Account Deletion — (`File: runtime/runtime/src/actions.rs`)

### Summary

`action_delete_account` computes the "non-contract" storage usage to compare against `MAX_ACCOUNT_DELETION_STORAGE_USAGE`. In the legacy code path it subtracts only the **local** contract's code length. For accounts whose contract field is `AccountContract::Global` or `AccountContract::GlobalByAccount`, the identifier's storage overhead (32 bytes for a hash, or `account_id.len()` bytes for a by-account reference) is never subtracted. The overstated usage causes a spurious `DeleteAccountWithLargeState` error for any account whose total storage sits in the narrow band `(MAX, MAX + identifier_size]`, permanently blocking deletion through the protocol until the `FixDeleteAccountGlobalContractStorageUsage` feature activates at protocol version 85.

### Finding Description

`action_delete_account` branches on the protocol version to compute `account_storage_usage`:

```rust
// runtime/runtime/src/actions.rs  ~line 311
let account_storage_usage = if ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage
    .enabled(current_protocol_version)
{
    // FIXED path: subtracts ALL contract storage (local code OR global identifier)
    let contract_storage = get_contract_storage_usage(state_update, account_id, account_ref)?;
    account_ref.storage_usage().saturating_sub(contract_storage)
} else {
    // LEGACY path: only subtracts local contract code length
    let account_storage_usage = account_ref.storage_usage();
    let code_len = get_code_len_or_default(
        state_update,
        account_id.clone(),
        account_ref.local_contract_hash().unwrap_or_default(),  // returns 0 for global contracts
    )?;
    account_storage_usage.saturating_sub(code_len)   // <-- identifier never subtracted
};
if account_storage_usage > Account::MAX_ACCOUNT_DELETION_STORAGE_USAGE {
    result.result = Err(ActionErrorKind::DeleteAccountWithLargeState { … }.into());
    return Ok(());
}
```

`get_contract_storage_usage` (the fixed helper) correctly handles all variants:

```rust
// runtime/runtime/src/actions.rs  ~line 395
fn get_contract_storage_usage(…) -> Result<StorageUsage, StorageError> {
    Ok(match account.contract().as_ref() {
        AccountContract::None => 0,
        AccountContract::Local(code_hash) => get_code_len_or_default(…)?,
        AccountContract::Global(_) | AccountContract::GlobalByAccount(_) => {
            account.contract().identifier_storage_usage()   // 32 bytes or account_id.len()
        }
    })
}
```

In the legacy path, `local_contract_hash()` returns `CryptoHash::default()` for global-contract accounts, so `get_code_len_or_default` returns `0`, and the full `account.storage_usage()` (which already includes the identifier overhead charged at `UseGlobalContract` time) is compared against the limit without any deduction.

The exact divergent value is the identifier size: **32 bytes** for `AccountContract::Global(CryptoHash)` and **`account_id.len()` bytes** for `AccountContract::GlobalByAccount(AccountId)`.

### Impact Explanation

Any account that:
1. Used `UseGlobalContract` (available since protocol version 77), and
2. Has `storage_usage` in the range `(10_000, 10_032]` bytes (for a hash-keyed global contract) or `(10_000, 10_000 + account_id_len]` bytes (for an account-keyed one)

will receive `DeleteAccountWithLargeState` on every `DeleteAccount` action until the network upgrades to protocol version 85. The account's balance cannot be swept to a beneficiary via the deletion path; the account remains on-chain and its storage-staking deposit stays locked. Because the error is deterministic and protocol-version-gated, no workaround exists at the application layer before the upgrade.

### Likelihood Explanation

Global contracts were introduced at protocol version 77 (mainnet). Any user who called `UseGlobalContract` and whose account's total storage happens to fall in the narrow window above `MAX_ACCOUNT_DELETION_STORAGE_USAGE` triggers the bug. The window is small (≤ 32 bytes for hash-keyed contracts) but the condition is fully user-reachable without any privileged role: a normal account holder can deploy a global contract, use it, and then attempt deletion.

### Recommendation

The fix is already present in the codebase behind `ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage` (protocol version 85). The corrected branch calls `get_contract_storage_usage`, which dispatches on the actual `AccountContract` variant and returns the identifier's storage overhead for global contracts. No additional change is needed beyond activating the feature at the next protocol upgrade.

### Proof of Concept

The existing unit test `test_delete_account_global_contract_protocol_transition` in `runtime/runtime/src/actions.rs` demonstrates the divergence:

```rust
// storage = MAX + 32; identifier = 32 bytes
// Before fix: MAX + 32 - 0 = MAX + 32 > MAX  → DeleteAccountWithLargeState (wrong)
let before = test_delete_account_in_empty_trie(
    &account_id,
    AccountContract::Global(CryptoHash::default()),
    Account::MAX_ACCOUNT_DELETION_STORAGE_USAGE + 32,
    ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage.protocol_version() - 1,
);
expect_delete_account_too_large(&before);   // spurious failure

// After fix: MAX + 32 - 32 = MAX, not > MAX → Ok (correct)
let after = test_delete_account_in_empty_trie(
    &account_id,
    AccountContract::Global(CryptoHash::default()),
    Account::MAX_ACCOUNT_DELETION_STORAGE_USAGE + 32,
    ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage.protocol_version(),
);
assert!(after.result.is_ok());
