### Title
Unbound Dual-Signature in `ChunkEndorsementV2::verify` Enables Cache-Slot Poisoning via Cross-Endorsement Component Mixing — (`core/primitives/src/stateless_validation/chunk_endorsement.rs`)

---

### Summary

`ChunkEndorsementV2` signs `inner` (containing `chunk_hash`) and `metadata` (containing `shard_id`, `epoch_id`, `height_created`, `account_id`) as two **independent** signatures with no cryptographic binding between them. A network-level attacker who observes two legitimate endorsements from the same validator V — one for chunk_X and one for chunk_Y — can splice their components into a mixed endorsement that passes all validation in `process_chunk_endorsement`, occupies the cache slot for chunk_Y under validator V's account with chunk_X's hash, and causes `collect_chunk_endorsements` to silently discard V's contribution to chunk_Y's endorsement threshold.

---

### Finding Description

**Structural flaw — `ChunkEndorsementV2::verify`:**

```rust
// core/primitives/src/stateless_validation/chunk_endorsement.rs, lines 112-117
fn verify(&self, public_key: &PublicKey) -> bool {
    let inner = borsh::to_vec(&self.inner).unwrap();
    let metadata = borsh::to_vec(&self.metadata).unwrap();
    self.signature.verify(&inner, public_key)
        && self.metadata_signature.verify(&metadata, public_key)
}
```

`self.signature` covers only `ChunkEndorsementInnerV1 { chunk_hash, signature_differentiator }`. `self.metadata_signature` covers only `ChunkEndorsementMetadata { account_id, shard_id, epoch_id, height_created }`. No single signature covers both fields together. There is no nonce, no cross-field commitment, and no domain separator that binds the two halves. [1](#0-0) 

**Attack construction:**

Given two legitimately observed endorsements from the same validator V:

| Field | Endorsement_X (for chunk_X) | Endorsement_Y (for chunk_Y) |
|---|---|---|
| `inner.chunk_hash` | `X_hash` | `Y_hash` |
| `signature` | `sign_V(inner_X)` | `sign_V(inner_Y)` |
| `metadata` | `{V, X_shard, X_epoch, X_height}` | `{V, Y_shard, Y_epoch, Y_height}` |
| `metadata_signature` | `sign_V(meta_X)` | `sign_V(meta_Y)` |

Mixed endorsement sent to target node:

```
inner.chunk_hash      = X_hash           ← from Endorsement_X
signature             = sign_V(inner_X)  ← from Endorsement_X  (valid over X_hash)
metadata              = {V, Y_shard, Y_epoch, Y_height}  ← from Endorsement_Y
metadata_signature    = sign_V(meta_Y)   ← from Endorsement_Y  (valid over Y metadata)
```

`verify()` returns `true` because both halves are independently valid under V's public key.

**Validation path — all checks pass:**

1. `validate_chunk_relevant_as_validator`: `chunk_production_key()` is derived from `metadata` → chunk_Y's key; V is a valid chunk validator for chunk_Y. ✓
2. `validate_chunk_endorsement_signature` → `endorsement.verify(V.public_key())` → `true`. ✓ [2](#0-1) 

**Cache insertion — wrong hash stored:**

```rust
// chain/client/src/stateless_validation/chunk_endorsement.rs, lines 59-63
cache.get_or_insert_mut(key, || HashMap::new()).insert(
    account_id.clone(),
    (endorsement.chunk_hash(), endorsement.signature()),
    //  ^^^ returns X_hash, not Y_hash
);
```

`key` = chunk_Y's `ChunkProductionKey`; stored value = `(X_hash, sign_V(inner_X))`. [3](#0-2) 

**Deduplication guard — legitimate endorsement silently dropped:**

When the real Endorsement_Y subsequently arrives:

```rust
// lines 49-53
if cache.peek(&key).is_some_and(|entry| entry.contains_key(account_id)) {
    tracing::debug!(target: "client", ?endorsement, "already received chunk endorsement");
    return Ok(());
}
```

The slot `(chunk_Y_key, V)` is already occupied by the mixed endorsement, so the legitimate endorsement is discarded. [4](#0-3) 

**Collection — poisoned entry filtered out:**

```rust
// lines 107-111
let validator_signatures = entry
    .into_iter()
    .filter(|(_, (chunk_hash, _))| chunk_hash == chunk_header.chunk_hash())
    //  ^^^ X_hash != Y_hash → V's entry is excluded
    .map(|(account_id, (_, signature))| (account_id, signature.clone()))
    .collect();
```

Validator V's stake is not counted toward chunk_Y's endorsement threshold. [5](#0-4) 

---

### Impact Explanation

A malicious peer node (no validator privileges required) can selectively nullify any targeted validator's endorsement for any specific chunk. If the targeted validator(s) hold sufficient stake, the block producer cannot collect enough endorsements to include the chunk, stalling chunk production for that slot. The cache slot remains poisoned for the lifetime of the LRU entry — no legitimate endorsement from V for chunk_Y can displace it.

---

### Likelihood Explanation

The attacker needs only to:
1. Observe two P2P-routed endorsements from the same validator V (one for chunk_X, one for chunk_Y).
2. Deliver the mixed endorsement to the target block-producer node before the legitimate chunk_Y endorsement arrives.

Both conditions are achievable by a malicious peer connected to the NEAR P2P network. No cryptographic secret is required; only message interception and replay with component substitution.

---

### Recommendation

Bind `inner` and `metadata` under a single signature. Replace the two independent signatures with one signature over the concatenation (or a committed hash) of both fields:

```rust
// Conceptual fix
fn sign(chunk_hash: ChunkHash, metadata: &ChunkEndorsementMetadata, signer: &ValidatorSigner) -> Signature {
    let payload = borsh::to_vec(&(inner, metadata)).unwrap();
    signer.sign_bytes(&payload)
}

fn verify(&self, public_key: &PublicKey) -> bool {
    let payload = borsh::to_vec(&(&self.inner, &self.metadata)).unwrap();
    self.signature.verify(&payload, public_key)
}
```

This eliminates the ability to mix `inner` from one endorsement with `metadata` from another, since any such splice invalidates the single combined signature.

### Proof of Concept

```rust
// Pseudocode Rust test (production path only, no test infrastructure)
fn test_mixed_endorsement_poisons_cache() {
    // Validator V signs endorsement for chunk_X
    let endorsement_x = ChunkEndorsement::new(epoch_id, &chunk_x_header, &validator_v_signer);
    // Validator V signs endorsement for chunk_Y (same epoch, different shard/height)
    let endorsement_y = ChunkEndorsement::new(epoch_id, &chunk_y_header, &validator_v_signer);

    // Attacker splices: inner+signature from X, metadata+metadata_signature from Y
    let ChunkEndorsement::V2(ex) = endorsement_x;
    let ChunkEndorsement::V2(ey) = endorsement_y;
    let mixed = ChunkEndorsement::V2(ChunkEndorsementV2 {
        inner: ex.inner,               // chunk_X_hash
        signature: ex.signature,       // valid over chunk_X_hash
        metadata: ey.metadata,         // identifies chunk_Y
        metadata_signature: ey.metadata_signature, // valid over chunk_Y metadata
    });

    // Mixed endorsement passes all validation and is stored under chunk_Y's key
    tracker.process_chunk_endorsement(&mixed).unwrap();

    // Legitimate chunk_Y endorsement is now rejected as duplicate
    tracker.process_chunk_endorsement(&endorsement_y).unwrap(); // silently dropped

    // collect_chunk_endorsements for chunk_Y returns zero valid signatures
    let state = tracker.collect_chunk_endorsements(&chunk_y_header).unwrap();
    assert_eq!(state.endorsed_stake, 0); // V's stake not counted despite cache entry existing
}
```
