Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end-user, enabling allowlist bypass or DoS — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`MetricOmmPool.swap` passes `msg.sender` (the immediate caller) as `sender` to `_beforeSwap`, which forwards it to `SwapAllowlistExtension.beforeSwap`. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end-user. The extension therefore evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`, completely defeating the per-user gate and producing either an allowlist bypass or a DoS for allowlisted users depending on admin configuration.

## Finding Description

**Root cause — pool passes `msg.sender` as `sender`:**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at line 230:

```solidity
_beforeSwap(
  msg.sender,   // ← pool's immediate caller, not the end-user
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` encodes that value and forwards it verbatim to every registered extension via `_callExtensionsInOrder`.

**Extension checks the wrong address:**

`SwapAllowlistExtension.beforeSwap` receives the forwarded `sender` and gates on it:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct for the mapping key) and `sender` is whoever called `pool.swap()`.

**Router is the pool's `msg.sender`:**

`MetricOmmSimpleRouter.exactInputSingle` calls the pool directly:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
```

The router does not forward the original `msg.sender` to the pool in any way. The pool therefore sees `sender = address(router)`.

**Two mutually exclusive failure modes:**

- **Mode A (bypass):** Admin allowlists the router address so router-mediated swaps work. Every user who calls through the router passes the check regardless of individual allowlist status. The per-user gate is completely defeated.
- **Mode B (DoS):** Admin allowlists specific user EOAs. Those users call through the router. The extension checks the router address (not allowlisted) and reverts `NotAllowedToSwap`. Allowlisted users cannot use the standard periphery router.

**Contrast with `DepositAllowlistExtension`:**

`DepositAllowlistExtension.beforeAddLiquidity` correctly ignores `sender` (the payer/intermediary) and gates on `owner` (the position owner — the economically relevant actor):

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
  ...
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
```

The swap hook has no equivalent "owner" field — the only user-identifying field available is `sender`, which is the intermediary when a router is used.

## Impact Explanation

**Mode A** is a direct admin-boundary break: a pool configured to restrict trading to a curated set of addresses can be bypassed by any unprivileged user simply by routing through the public `MetricOmmSimpleRouter`. The allowlist provides zero protection for router-mediated swaps.

**Mode B** breaks core swap functionality: allowlisted users on a restricted pool cannot use the standard periphery router, making the pool effectively unusable through the intended user-facing interface.

Both impacts satisfy the allowed impact gate: admin-boundary bypass and broken core pool swap functionality.

## Likelihood Explanation

`SwapAllowlistExtension` is a production periphery extension explicitly designed for deployment on live pools. `MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool that configures both will exhibit one of the two failure modes. No special permissions, malicious setup, or non-standard tokens are required — a normal pool admin action (`setAllowedToSwap`) combined with a normal user action (`exactInputSingle`) is sufficient to trigger either mode.

## Recommendation

The extension must receive the original end-user address, not the intermediary's address. Two approaches:

1. **Extension-data forwarding:** The router encodes the original `msg.sender` into `extensionData` and the extension decodes it. The extension should also verify `msg.sender` (the pool) is a known router-aware pool to prevent spoofing.
2. **Dedicated originator field:** Change the pool's `_beforeSwap` call to pass both the pool's `msg.sender` (the router) and a separately tracked "originator" stored in transient storage by the router before calling the pool, so the extension can gate on the originator.

The `DepositAllowlistExtension` pattern — gating on `owner` rather than `sender` — is the correct model.

## Proof of Concept

```
// Mode A — Allowlist bypass
1. Deploy pool with SwapAllowlistExtension
2. swapExtension.setAllowedToSwap(pool, address(router), true)
   swapExtension.setAllowedToSwap(pool, alice, false)  // alice NOT allowlisted

3. Alice calls router.exactInputSingle({pool: pool, ...})
   → Router calls pool.swap(recipient, ...)
   → pool.msg.sender = router
   → _beforeSwap(sender=router, ...)
   → Extension checks allowedSwapper[pool][router] → true
   → Swap succeeds for Alice despite her not being allowlisted ✓ bypass

// Mode B — DoS for allowlisted user
1. Deploy pool with SwapAllowlistExtension
2. swapExtension.setAllowedToSwap(pool, alice, true)   // alice IS allowlisted
   // router is NOT allowlisted

3. Alice calls router.exactInputSingle({pool: pool, ...})
   → Router calls pool.swap(recipient, ...)
   → pool.msg.sender = router
   → _beforeSwap(sender=router, ...)
   → Extension checks allowedSwapper[pool][router] → false
   → Reverts NotAllowedToSwap — Alice cannot use the router ✓ DoS
