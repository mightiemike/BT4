Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, allowing any caller to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` as seen by the pool. When `MetricOmmSimpleRouter` calls `pool.swap()`, the pool's `msg.sender` is the router contract, not the originating user. The extension therefore evaluates the router's allowlist status rather than the real swapper's identity. If the router is allowlisted, every user who routes through it bypasses the gate entirely; if it is not, every router-mediated swap reverts even for individually allowlisted users.

## Finding Description

**Root cause — three code sites chain together:**

1. `MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

2. `SwapAllowlistExtension.beforeSwap` uses that `sender` argument to look up the per-pool allowlist: [2](#0-1) 

3. `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly without forwarding the originating user's address — the pool's `msg.sender` is therefore the router: [3](#0-2) 

**Full call chain:**
```
User → MetricOmmSimpleRouter.exactInputSingle()
         → pool.swap(recipient, ...)          // msg.sender inside pool = router
              → _beforeSwap(sender=router, ...)
                   → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                        checks: allowedSwapper[pool][router]   ← wrong actor
```

The same wrong-actor binding applies to `exactInput`, `exactOutputSingle`, and `exactOutput` — all four router entry points call `pool.swap()` with the router as `msg.sender`. [4](#0-3) 

No existing guard corrects this: the extension has no knowledge of the router, no callback to retrieve the real user, and no mechanism to distinguish a forwarded call from a direct one.

## Impact Explanation

A pool protected by `SwapAllowlistExtension` is designed to restrict trading to a specific set of addresses (e.g., approved market makers or KYC'd counterparties). Two concrete fund-impacting failure modes exist:

1. **Allowlist bypass (High):** A pool admin who allowlists the router address (the natural configuration to support router-mediated swaps for approved users) inadvertently grants every user on the public internet the ability to trade on the pool. LP positions are exposed to trades from actors the admin explicitly intended to exclude, with direct principal-loss consequences for LPs.

2. **Broken core functionality:** If the router is not allowlisted, every router-mediated swap reverts with `NotAllowedToSwap`, making the primary periphery entry point unusable for all users regardless of their individual allowlist status.

Both outcomes constitute broken core pool functionality causing loss of funds or unusable swap flows, matching the contest's allowed impact gate.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary supported periphery entry point. No special privileges, flash loans, or multi-transaction setups are required. A single `exactInputSingle` call from any EOA suffices to trigger the bypass. Any user who observes that a pool uses `SwapAllowlistExtension` and that the router is allowlisted can immediately exploit it. The condition is trivially discoverable on-chain.

## Recommendation

The extension must gate on the economically relevant actor, not the immediate pool caller. The cleanest fix is to add an explicit `originator` field to the pool's swap call (defaulting to `msg.sender` for direct calls) and propagate it through `_beforeSwap` to all extension hooks. `SwapAllowlistExtension.beforeSwap` would then check `allowedSwapper[pool][originator]` instead of `allowedSwapper[pool][sender]`. `MetricOmmSimpleRouter` would pass `msg.sender` as the originator, ensuring the allowlist always sees the true economic actor regardless of which intermediary initiated the pool call.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps (believing the router is a trusted forwarder for allowlisted users).
3. `attacker` — an address **not** in `allowedSwapper[pool]` — calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the pool.
4. Inside the pool, `msg.sender` is the router. `_beforeSwap(sender=router, ...)` is invoked.
5. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router]` → `true`.
6. The swap executes successfully for `attacker`, bypassing the intended allowlist gate. [5](#0-4) [6](#0-5)

    _checkDeadline(params.deadline);
    _validatePath(params.tokens, params.pools, params.extensionDatas);

    uint256 last = params.pools.length - 1;
    int128 amount = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn);

    for (uint256 i = 0; i <= last; i++) {
      address pool = params.pools[i];
      bool zeroForOne = MetricOmmSwapPath.resolveZeroForOneBitmap(params.zeroForOneBitMap, i);

      _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
      (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
        .swap(
          i == last ? params.recipient : address(this),
          zeroForOne,
          amount,
          MetricOmmSwapPath.openLimit(zeroForOne),
          "",
          params.extensionDatas[i]
        );

      int128 amountInActual = MetricOmmSwapResults.extractAmountIn(zeroForOne, amount0Delta, amount1Delta);
      if (amountInActual < amount) revert InvalidInputAmountAtHop(uint8(i), amountInActual, amount);

      amount = MetricOmmSwapResults.extractAmountOut(zeroForOne, amount0Delta, amount1Delta);
    }

    if (amount <= 0) revert InvalidSwapDeltas();
    amountOut = MetricOmmSwapInputs.int128ToUint128(amount);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
