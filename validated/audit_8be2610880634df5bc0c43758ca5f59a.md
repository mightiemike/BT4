### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the 64-Byte Merkle Forgery Mitigation — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase proof. However, the deprecated `verify_transaction_inclusion` (v1) remains a fully public, unpermissioned NEAR entry point. Any caller can invoke v1 directly, bypassing the coinbase proof check entirely and forging a transaction inclusion proof against a live block header.

---

### Finding Description

The contract exposes two SPV verification entry points:

**v1 — deprecated but still public:** [1](#0-0) 

The function's own docstring acknowledges the flaw:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [2](#0-1) 

**v2 — the intended mitigation:** [3](#0-2) 

`verify_transaction_inclusion_v2` adds two guards before delegating to v1:
1. Equal-length check between `merkle_proof` and `coinbase_merkle_proof`.
2. Coinbase Merkle proof validation against the block's `merkle_root`. [4](#0-3) 

After both guards pass, it calls v1 via `self.verify_transaction_inclusion(args.into())`. [5](#0-4) 

**The bypass:** v1 carries no `#[private]`, no role guard, and no access-control attribute beyond `#[pause]`. Any unprivileged NEAR account can call `verify_transaction_inclusion` directly, skipping both coinbase guards entirely.

The 64-byte attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions): Bitcoin's Merkle tree hashes pairs of 32-byte child hashes into a 64-byte input. An attacker can craft a 64-byte blob whose double-SHA256 hash equals an internal Merkle node. Submitting this blob as `tx_id` to v1 with a suitably crafted `merkle_proof` causes `compute_root_from_merkle_proof` to reconstruct the real `merkle_root`, making the function return `true` for a transaction that does not exist. [6](#0-5)

---

### Impact Explanation

Any downstream NEAR contract or off-chain service that calls `verify_transaction_inclusion` (v1) to authorize a Bitcoin-side action — e.g., releasing bridged tokens, confirming a deposit, or settling a cross-chain swap — can be deceived into accepting a forged proof. The contract returns `true` for a transaction that was never included in any Bitcoin block. This is a proof-verification forgery with direct financial impact on consumers of the SPV API.

---

### Likelihood Explanation

The attack requires no privileged role, no leaked key, and no social engineering. Any NEAR account can call `verify_transaction_inclusion` at any time the contract is not paused. The 64-byte forgery technique is publicly documented, and the necessary internal-node preimage can be computed offline from any block whose Merkle tree is known. The deprecated annotation is a Rust compiler hint only; it does not restrict on-chain callability.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` or add `#[private]` to make it callable only by the contract itself (i.e., only via `verify_transaction_inclusion_v2`). Alternatively, inline the coinbase proof check directly into v1 and remove the separate v2 entry point. The current architecture where the security-critical guard lives only in the wrapper while the vulnerable inner function remains independently callable is the root cause.

---

### Proof of Concept

1. Attacker selects any confirmed Bitcoin block whose Merkle tree is publicly known (e.g., from a block explorer).
2. Attacker identifies an internal Merkle node `N` at depth `d` in the tree.
3. Attacker constructs a 64-byte blob `fake_tx` such that `double_sha256(fake_tx) == N` (this is the standard 64-byte preimage attack).
4. Attacker builds a `merkle_proof` of length `d-1` that walks from `N` up to the Merkle root using real sibling hashes from the block.
5. Attacker calls `verify_transaction_inclusion` directly (not v2) with `tx_id = fake_tx`, `tx_index` matching the position of `N`, and the crafted `merkle_proof`.
6. `compute_root_from_merkle_proof` reconstructs the real `merkle_root`; the function returns `true`.
7. Any contract consuming this result treats the non-existent transaction as confirmed.

The coinbase proof check in `verify_transaction_inclusion_v2` is never reached because the attacker bypasses v2 entirely. [8](#0-7) [9](#0-8)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
