Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Originating User, Enabling Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender` — the direct caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the router becomes `msg.sender`, so the extension checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actualUser]`. Any pool admin who allowlists the router so that legitimate users can reach the pool through it inadvertently grants every unprivileged address the ability to bypass the allowlist entirely.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to every extension hook:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // direct caller of pool.swap() — the router, not the originating user
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that value against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol L37-38
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct for pool-identity gating) and `sender` is whoever called `pool.swap()`. When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The pool therefore sees `msg.sender = router`, and the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. The same applies to multi-hop `exactInput` (L103-112) and `exactOutput` paths, where the router is always the direct caller of each pool's `swap()`.

This creates an inescapable dilemma for any pool admin who deploys a restricted pool and wants allowlisted users to be able to use the router:

| Router allowlisted? | Effect |
|---|---|
| No | Allowlisted users cannot use the router; they must call the pool directly |
| Yes | **Every** user can bypass the allowlist by routing through the router |

## Impact Explanation
Any non-allowlisted address can swap in a pool the admin intended to restrict to a specific set of addresses (e.g., KYC-verified counterparties, whitelisted market makers, or protocol-internal addresses). The allowlist guard is completely ineffective once the router is allowlisted. Swaps execute at the oracle-derived bid/ask price, so an unauthorized swapper can extract value from LP positions at the same rate as any authorized swapper, causing direct loss of LP principal and fees. This meets the "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" impact criteria.

## Likelihood Explanation
The router is a public, permissionless contract. Any pool admin who wants allowlisted users to interact via the router must allowlist the router address — this is the expected operational pattern for any production pool using the periphery. The bypass requires no special privileges, no flash loans, and no unusual token behavior; a single standard router call suffices. The condition is therefore likely to be triggered in any real deployment that combines `SwapAllowlistExtension` with `MetricOmmSimpleRouter`.

## Recommendation
**Short term:** In `SwapAllowlistExtension.beforeSwap`, do not rely solely on the `sender` argument. Instead, require the router to supply the real user identity via `extensionData`, then verify that identity against the allowlist. Alternatively, document explicitly that the extension is incompatible with the router and that pools using it must be accessed directly (no router allowlisting).

**Long term:** Introduce a canonical "originating user" field that the router populates in `extensionData` and that the extension can verify (e.g., signed by the router or validated against a transient-storage context). This ensures the identity of the actual actor — not the intermediate dispatcher — is checked.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` as the `beforeSwap` extension.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is intended to be allowed.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` — necessary so Alice can use the router.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: restrictedPool, recipient: bob, ...})`.
5. The router calls `pool.swap(bob, ...)` with `msg.sender = router` (confirmed at `MetricOmmPool.sol` L231).
6. The pool calls `extension.beforeSwap(sender=router, ...)`.
7. The extension evaluates `allowedSwapper[pool][router]` → `true` → no revert (confirmed at `SwapAllowlistExtension.sol` L37).
8. Bob's swap executes successfully, bypassing the allowlist entirely.
