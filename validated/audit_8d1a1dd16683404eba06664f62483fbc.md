### Title
`ED25519SecretKey::Debug` Leaks Raw Private Key Bytes While `MlDsa65SecretKey::Debug` Is Explicitly Redacted — (File: `core/crypto/src/signature.rs`)

### Summary

`ED25519SecretKey` implements `Debug` by emitting the raw 32-byte secret key in base58. The newer `MlDsa65SecretKey` type added alongside the post-quantum key scheme explicitly redacts its `Debug` output with the comment *"Avoid printing key material; show only the type tag."* Because `SecretKey` derives `Debug` and `Signer` also derives `Debug`, any production code path that formats either type with `{:?}` — in a `tracing::debug!`, an error message, or a structured log field — will emit the validator's raw signing key into logs.

---

### Finding Description

`ED25519SecretKey::Debug` at `core/crypto/src/signature.rs` lines 741–745 prints the actual secret key bytes: [1](#0-0) 

The `MlDsa65SecretKey::Debug` impl added at lines 751–756 explicitly avoids this: [2](#0-1) 

`SecretKey` derives `Debug` at line 759, so `SecretKey::ED25519(k)` formatted with `{:?}` delegates directly to the leaking impl: [3](#0-2) 

`Signer` also derives `Debug` at line 10 of `core/crypto/src/signer.rs`, and `Signer::InMemory` wraps `InMemorySigner` which holds a public `secret_key: SecretKey` field: [4](#0-3) [5](#0-4) 

Any `tracing::debug!("{:?}", signer)` or error path that includes a `Signer::InMemory` or a bare `SecretKey::ED25519` will emit the raw private key. Grep matches for `?signer` patterns were found in production files `chain/client/src/client.rs`, `chain/client/src/rpc_handler.rs`, and `chain/jsonrpc/src/lib.rs`.

The `SecretKey::Display` impl at lines 895–904 also serializes the full key for all variants (intentionally, because `serde::Serialize` calls `collect_str(self)`), but `Display` is a deliberate serialization channel. The `Debug` impl is not — and the developers' own comment on `MlDsa65SecretKey::Debug` confirms the intended invariant: *Debug must not print key material*. [6](#0-5) 

---

### Impact Explanation

A validator's ED25519 signing key emitted into structured logs (CloudWatch, Datadog, any SIEM) gives any reader of those logs the ability to:

1. Sign arbitrary blocks and chunks as the validator — enabling equivocation / double-signing.
2. Trigger slashing of the validator's stake.
3. Participate in consensus under the validator's identity, disrupting liveness or safety.

The key is the full 32-byte secret scalar; base58-decoding the logged string yields a directly usable `ed25519_dalek` signing key.

---

### Likelihood Explanation

The `Signer` and `SecretKey` types are used throughout the client and RPC layers. The inconsistency is self-documenting: the developers added an explicit redaction comment for `MlDsa65SecretKey` but left `ED25519SecretKey::Debug` unchanged. Any new `tracing::debug!` or `tracing::error!` call that includes a `Signer` or `SecretKey` field with `?` (the `{:?}` shorthand in `tracing` structured fields) will silently leak the key. The risk grows with every new log statement added to the client.

---

### Recommendation

Replace `ED25519SecretKey::Debug` with a redacted implementation consistent with `MlDsa65SecretKey::Debug`:

```rust
impl std::fmt::Debug for ED25519SecretKey {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> Result<(), std::fmt::Error> {
        f.write_str("ED25519SecretKey(<redacted>)")
    }
}
```

Also audit `secp256k1::SecretKey`'s `Debug` output (external crate) and verify that any manual `Debug` impl for `InMemorySigner` does not delegate to `secret_key`'s `Debug`.

---

### Proof of Concept

```rust
use near_crypto::{KeyType, SecretKey};

let sk_ed = SecretKey::from_seed(KeyType::ED25519, "validator-key");
println!("{:?}", sk_ed);
// Output: ED25519(3HexuCB7GF8V4EhbtY4kYnfwFE5kT1YmyA4rAcVe7Wu...)
// ^^^ raw base58 secret key — directly usable to sign blocks

let sk_pq = SecretKey::from_seed(KeyType::MLDSA65, "validator-key");
println!("{:?}", sk_pq);
// Output: MLDSA65(MlDsa65SecretKey(<redacted>))
// ^^^ correctly redacted
```

The divergence is exact: `ED25519SecretKey::Debug` at line 743 calls `Bs58(&self.0[..SECRET_KEY_LENGTH])` — the raw 32-byte scalar — while `MlDsa65SecretKey::Debug` at line 754 writes a static redaction string. [1](#0-0) [2](#0-1)
