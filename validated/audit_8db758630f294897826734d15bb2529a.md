Audit Report

## Title
Pool admin can zero-out the timelock via unvalidated `proposeOracleStopLossTimelock`, then immediately disable the stop-loss guard - (File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol)

## Summary
`proposeOracleStopLossTimelock` accepts any `uint32 newTimelock` value with no lower-bound validation, unlike `proposeOracleStopLossDrawdown` and `proposeOracleStopLossDecay` which both call dedicated validators. A pool admin can propose `newTimelock = 0`, wait the current timelock period, execute it, and then in the same block propose and execute `drawdownE6 = 0`, which causes `_afterSwapOracleStopLoss` to return early unconditionally, permanently disabling the stop-loss for all swaps without giving LPs any reaction window.

## Finding Description

**Root cause — missing validation on `newTimelock`:**

`proposeOracleStopLossDrawdown` (line 104) calls `_validateDrawdown(newMaxDrawdownE6)` and `proposeOracleStopLossDecay` (line 131) calls `_validateDecay(newDecayPerSecondE8)`, but `proposeOracleStopLossTimelock` (line 78) stores `newTimelock` directly with no check:

```solidity
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);
    sched.pendingTimelock = newTimelock;          // ← no validation
    sched.pendingTimelockExecuteAfter = executeAfter;
    ...
}
```

**How `timelock = 0` collapses the guard:**

`_afterTimelock` returns `block.timestamp + oracleStopLossConfig[pool_].timelock`. When `timelock = 0`, `executeAfter = block.timestamp`. `_requireElapsed` checks `block.timestamp < executeAfter`, which is `block.timestamp < block.timestamp` = false, so it passes immediately. Every subsequent `propose*` + `execute*` pair can be completed atomically in one block.

**Exploit path (pool admin, two phases):**

1. Pool admin calls `proposeOracleStopLossTimelock(pool, 0)` — no revert.
2. Waits the current timelock period (e.g. 7 days).
3. Calls `executeOracleStopLossTimelock(pool)` — `oracleStopLossConfig[pool].timelock` is now `0`.
4. In the same block: calls `proposeOracleStopLossDrawdown(pool, 0)` then `executeOracleStopLossDrawdown(pool)` — both succeed immediately.
5. `oracleStopLossConfig[pool].drawdownE6 = 0`.
6. Every subsequent swap hits `_afterSwapOracleStopLoss` → line 217: `if (drawdown == 0) return;` — all watermark checks are skipped permanently.

**Existing guards are insufficient:**

`_validateDrawdown` only rejects values `> E6`; `0` passes. The timelock proposal path has no analogous floor check. The `onlyPoolAdmin` modifier is the only gate, and the rules explicitly require proving a timelock bypass for pool-admin-sourced issues.

## Impact Explanation

Once the stop-loss is disabled (`drawdownE6 = 0`), the per-bin value-per-share invariant is no longer enforced on any swap. The pool admin (or any actor who can influence oracle prices within the now-unguarded pool) can drain LP value through repeated swaps at unfavorable oracle marks without the extension ever reverting. This is a direct loss of LP principal — the exact corrupted value is `oracleStopLossConfig[pool].drawdownE6` set to `0`, causing the guard to silently no-op on every `afterSwap` call. Severity: **Medium** (requires pool admin action across two timelock windows, but the LP protection mechanism is fully and permanently bypassed).

## Likelihood Explanation

Any pool admin can execute this in two steps separated by the current timelock period. No external oracle manipulation or special capability is required beyond the pool admin role. The attack is repeatable and deterministic. The test suite itself demonstrates that `timelock = 0` allows immediate execution (see `test_decayTimelockZeroExecutesImmediately` and the `_proposeAndExecuteTimelock` helper which proposes and executes in the same call when the current timelock is zero).

## Recommendation

Add a minimum timelock floor in `proposeOracleStopLossTimelock` (and symmetrically in `initialize`), analogous to the existing validators for drawdown and decay:

```solidity
uint32 private constant MIN_TIMELOCK = 1 days;

function _validateTimelock(uint32 timelock) private pure {
    if (timelock < MIN_TIMELOCK) revert OracleStopLossTimelockTooShort(timelock);
}
```

Call `_validateTimelock(newTimelock)` at the top of `proposeOracleStopLossTimelock` and `_validateTimelock(timelock)` inside `initialize` before storing the config.

## Proof of Concept

```solidity
function test_adminCanBypassTimelockAndDisableStopLoss() public {
    // Pool initialized with 7-day timelock and 50% drawdown
    OracleValueStopLossExtension ext = new OracleValueStopLossExtension(address(factoryStub));
    MockExtensionExtsloadPool pool = new MockExtensionExtsloadPool(address(factoryStub), MIN_SHARES);
    factoryStub.setPoolAdmin(address(pool), admin);
    vm.prank(address(factoryStub));
    ext.initialize(address(pool), abi.encode(uint32(500_000), uint32(58), uint32(7 days)));

    // Phase 1: admin proposes timelock = 0, waits 7 days, executes
    vm.startPrank(admin);
    ext.proposeOracleStopLossTimelock(address(pool), 0);
    vm.warp(block.timestamp + 7 days);
    ext.executeOracleStopLossTimelock(address(pool));

    // Phase 2: same block — propose and execute drawdown = 0 (no delay now)
    ext.proposeOracleStopLossDrawdown(address(pool), 0);
    ext.executeOracleStopLossDrawdown(address(pool));
    vm.stopPrank();

    (uint32 dd,,,) = ext.oracleStopLossConfig(address(pool));
    assertEq(dd, 0); // stop-loss permanently disabled

    // Confirm afterSwap no longer enforces any watermark check
    vm.prank(address(pool));
    // Would have reverted with OracleStopLossTriggered before; now silently passes
    ext.afterSwap(address(0), address(0), true, 0, 0, 0, 0, uint128(Q64), uint128(Q64), 0, 0, 0, "");
}
```
