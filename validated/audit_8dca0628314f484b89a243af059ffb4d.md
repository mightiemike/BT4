### Title
Unbounded `reorg_chain` While Loop Scales with Fork Depth, Causing OOG and Permanent Fork-Choice Corruption — (File: `contract/src/lib.rs`)

---

### Summary

`reorg_chain` contains an unbounded `while` loop that walks from the fork tip back to the common ancestor, performing multiple storage writes per iteration. Fork headers accumulate in `headers_pool` across many `submit_blocks` calls. When the final header tips the fork's chainwork above the mainchain, `reorg_chain` must traverse the entire accumulated fork depth in a single transaction. With a sufficiently deep fork, this exceeds NEAR's per-transaction gas budget, the call reverts, and the contract is permanently stuck tracking the lower-chainwork chain — the true canonical chain can never be applied.

---

### Finding Description

`reorg_chain` is called from `submit_block_header_inner` whenever a fork's cumulative chainwork exceeds the mainchain tip: [1](#0-0) 

Inside `reorg_chain`, after an optional bounded `for` loop that trims excess mainchain blocks, there is an unbounded `while` loop: [2](#0-1) 

Each iteration of this loop performs:
- `mainchain_header_to_height.contains_key` — storage read
- `mainchain_height_to_header.insert` — storage write
- `mainchain_header_to_height.insert` — storage write
- Conditionally `remove_block_header` (two more storage operations)
- `headers_pool.get` — storage read

That is roughly **5 storage operations per fork block traversed**. The loop terminates only when it reaches a block already recorded in `mainchain_header_to_height` — i.e., the common ancestor. The iteration count equals the fork depth.

Fork headers are stored in `headers_pool` across many separate `submit_blocks` calls via `store_fork_header`: [3](#0-2) 

These headers persist across transactions. When the final header tips the chainwork, `reorg_chain` must traverse the entire accumulated fork in one atomic transaction. If the fork is deep enough to exhaust the NEAR gas budget, the transaction panics and reverts. The fork headers already stored in `headers_pool` from prior transactions are **not** rolled back — they remain. Every subsequent attempt to submit the same final header triggers the same OOG revert. The contract is permanently stuck.

---

### Impact Explanation

The contract's canonical chain state — `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height` — is frozen on the lower-chainwork chain: [4](#0-3) 

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` resolve the target block exclusively through `mainchain_header_to_height`: [5](#0-4) 

With the canonical maps frozen on the stale chain, the contract silently returns incorrect SPV proof results. Transactions on the true (higher-chainwork) chain are rejected as "block does not belong to the current main chain," and transactions on the orphaned chain may be falsely accepted. No upgrade or migration path exists within the contract's current logic.

---

### Likelihood Explanation

The entry path is `submit_blocks` → `submit_block_header` → `submit_block_header_inner` → `reorg_chain`. `submit_blocks` is gated by `#[trusted_relayer]`, which the prompt explicitly lists as an in-scope entry path ("relayer-path user supplying adversarial chain data"). The attack does not require a malicious relayer: a legitimate deep fork on the underlying chain (e.g., a 51% attack on Dogecoin or Litecoin, both of which are supported by this contract) would cause the same OOG. The relayer would be submitting valid chain data that the contract cannot process. For Bitcoin mainnet the required fork depth is harder to achieve, but for Dogecoin and Litecoin the hashrate is low enough that deep forks are a realistic threat.

---

### Recommendation

1. **Cap reorg depth**: Reject any fork whose depth from the common ancestor exceeds a configurable `max_reorg_depth`. This bounds the while loop to a safe constant.
2. **Incremental reorg**: Store a "pending reorg" cursor in contract state and process the walk in fixed-size batches across multiple transactions, similar to how `run_mainchain_gc` uses `batch_size`.
3. **Track common ancestor at fork-submission time**: Record the fork's branch point when each fork header is stored, so `reorg_chain` can start from the known common ancestor rather than walking back from the tip.

---

### Proof of Concept

```
1. Trusted relayer submits fork headers f1, f2, …, f(N-1) across multiple submit_blocks calls.
   Each header is stored in headers_pool via store_fork_header.
   None of these calls trigger reorg_chain because fork chainwork < mainchain chainwork.

2. Trusted relayer submits f(N), whose cumulative chainwork finally exceeds the mainchain tip.

3. submit_block_header_inner calls reorg_chain(f(N), last_main_chain_block_height).

4. reorg_chain enters the while loop and must traverse f(N) → f(N-1) → … → f1 → common ancestor.
   Each step: ~5 NEAR storage operations.
   At sufficient N, total gas > 300 TGas (NEAR per-transaction limit) → panic/revert.

5. Transaction reverts. f1…f(N-1) remain in headers_pool. mainchain maps are unchanged.

6. Any retry of step 2 hits the same OOG. The contract is permanently stuck on the
   lower-chainwork mainchain.

7. verify_transaction_inclusion now resolves blocks against the stale canonical maps,
   returning incorrect SPV proof results indefinitely.
``` [6](#0-5)

    fn reorg_chain(&mut self, fork_tip_header: ExtendedHeader, last_main_chain_block_height: u64) {
        let fork_tip_height = fork_tip_header.block_height;
        if last_main_chain_block_height > fork_tip_height {
            // If we see that main chain is longer than fork we first garbage collect
            // outstanding main chain blocks:
            //
            //      [m1] - [m2] - [m3] - [m4] <- We should remove [m4]
            //     /
            // [m0]
            //     \
            //      [f1] - [f2] - [f3]
            for height in (fork_tip_height + 1)..=last_main_chain_block_height {
                let current_main_chain_blockhash = self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str("cannot get a block"));
                self.remove_block_header(&current_main_chain_blockhash);
                self.mainchain_height_to_header.remove(&height);
            }
        }

        // Now we are in a situation where mainchain is equivalent to fork size:
        //
        //      [m1] - [m2] - [m3] - [m4] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip
        //
        //
        // Or in a situation where it is shorter:
        //
        //      [m1] - [m2] - [m3] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip

        let fork_tip_hash = fork_tip_header.block_hash.clone();
        let mut fork_header_cursor = fork_tip_header;

        while !self
            .mainchain_header_to_height
            .contains_key(&fork_header_cursor.block_hash)
        {
            let prev_block_hash = fork_header_cursor.block_header.prev_block_hash;
            let current_block_hash = fork_header_cursor.block_hash;
            let current_height = fork_header_cursor.block_height;

            // Inserting the fork block into the main chain, if some mainchain block is occupying
            // this height let's save its hashcode
            let main_chain_block = self
                .mainchain_height_to_header
                .insert(&current_height, &current_block_hash);
            self.mainchain_header_to_height
                .insert(&current_block_hash, &current_height);

            // If we found a mainchain block at the current height than remove this block from the
            // header pool and from the header -> height map
            if let Some(current_main_chain_blockhash) = main_chain_block {
                self.remove_block_header(&current_main_chain_blockhash);
            }

            // Switch iterator cursor to the previous block in fork
            fork_header_cursor = self
                .headers_pool
                .get(&prev_block_hash)
                .unwrap_or_else(|| env::panic_str("previous fork block should be there"));
        }

        // Updating tip of the new main chain
        self.mainchain_tip_blockhash = fork_tip_hash;
    }
