### Title
Gas-key exec-fee in host functions uses wire-length instead of trie-id-length for ML-DSA-65 keys, creating a deterministic fee-domain inconsistency - (File: `runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

The `PostQuantumSignatures` protocol feature introduced `PublicKey::trie_id_len()` as the correct measurement for storage-fee accounting of ML-DSA-65 keys (33 bytes on-trie vs. 1953 bytes on-wire). The transaction-level fee path was updated to call `trie_id_len()`, but the host-function path for `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call` still passes the raw `public_key_len` parameter (the borsh wire length) directly to `gas_key_add_key_exec_fee()`. This creates a deterministic, protocol-level fee inconsistency: the same `AddKey` operation costs a different amount of exec gas depending on whether it is submitted as a transaction or issued from a contract via a host function.

---

### Finding Description

`gas_key_add_key_exec_fee()` in `core/parameters/src/cost.rs` computes the per-nonce exec fee by calling `access_key_key_len(account_id_len, public_key_len)` to derive the trie key length, then multiplying by `gas_key_byte` exec fee per nonce: [1](#0-0) 

The `public_key_len` argument is supposed to be the **on-trie identifier length** (what the receiver actually reads/writes in the trie). For ML-DSA-65, that is 33 bytes (`[tag=3] || sha3_256(domain || raw_pubkey)`), not the 1953-byte borsh-encoded full pubkey.

**Transaction path (correct):** `runtime/runtime/src/config.rs` calls `gas_key_add_key_exec_fee` with `public_key.trie_id_len()`: [2](#0-1) 

**Host-function path (incorrect):** `runtime/near-vm-runner/src/logic/logic.rs` passes `public_key_len as usize` — the raw byte count of the borsh-encoded key as supplied by the contract — directly to `gas_key_add_key_exec_fee`: [3](#0-2) 

The same defect appears in the Wasmtime backend: [4](#0-3) 

And in `promise_batch_action_add_gas_key_with_function_call` (both backends): [5](#0-4) 

The same inconsistency exists for `promise_batch_action_transfer_to_gas_key`, where `gas_key_transfer_exec_fee` is called with `public_key_len as usize` in the host-function path but with `action.public_key.trie_id_len()` in the transaction path: [6](#0-5) [7](#0-6) 

The design document explicitly flags this as a new contract that all storage-cost code must respect: [8](#0-7) 

---

### Impact Explanation

**Exact divergent value:** For `PublicKey::MLDSA65`, `len()` = 1953 bytes (wire), `trie_id_len()` = 33 bytes (on-trie). The host-function path inflates `public_key_len` by **1920 bytes** per key.

The per-nonce exec fee overcharge is:

```
overcharge_per_nonce = gas_key_byte_exec_fee × 1920
total_overcharge     = gas_key_byte_exec_fee × 1920 × num_nonces
```

With `num_nonces` up to 1024 (the maximum for a gas key), the total overcharge can be substantial. Any contract that calls `promise_batch_action_add_gas_key_with_full_access` or `promise_batch_action_add_gas_key_with_function_call` with an ML-DSA-65 public key will:

1. Be charged ~1920 × `gas_key_byte` exec gas × `num_nonces` more than the protocol intends.
2. Potentially exhaust its gas budget and fail with `GasExceeded` on operations that would succeed if submitted as a transaction.
3. Produce a different gas-burnt outcome than the equivalent transaction, breaking the protocol invariant that the same state transition costs the same gas regardless of invocation path.

This is a **High** impact: it is deterministic, protocol-level, and reachable by any unprivileged contract that uses ML-DSA-65 gas keys via host functions after `PostQuantumSignatures` activates (protocol version 85).

---

### Likelihood Explanation

`PostQuantumSignatures` is stabilized at protocol version 85 and `GasKeys` at the same release. Any contract that programmatically manages ML-DSA-65 gas keys (e.g., a key-management contract, a wallet contract, or a DeFi protocol using gas keys for sponsored transactions) will trigger this path. The overcharge is deterministic and reproducible on every invocation. No special privileges are required.

---

### Recommendation

Replace `public_key_len as usize` with `public_key.trie_id_len()` in all four host-function call sites that feed `gas_key_add_key_exec_fee` and `gas_key_transfer_exec_fee`:

1. `runtime/near-vm-runner/src/logic/logic.rs` — `promise_batch_action_add_gas_key_with_full_access` (line 3158)
2. `runtime/near-vm-runner/src/logic/logic.rs` — `promise_batch_action_add_gas_key_with_function_call` (line 3229)
3. `runtime/near-vm-runner/src/wasmtime_runner/logic.rs` — same two functions (lines 3398, 3502)
4. Both backends for `promise_batch_action_transfer_to_gas_key` — `gas_key_transfer_exec_fee` call (line 3095 / 3325)

The decoded `PublicKey` is already available at those call sites (via `public_key.decode()?`), so `trie_id_len()` can be called on it directly. The send-fee calls (`gas_key_transfer_send_fee`) intentionally use the wire length ("what the sender sees") and should remain unchanged.

Add a unit test analogous to `test_ml_dsa_65_access_key_storage_scales` that asserts the exec fee charged by the host function equals the exec fee charged by the transaction path for an ML-DSA-65 key.

### Proof of Concept

**Transaction path** — `permission_exec_fees` in `runtime/runtime/src/config.rs`:
```rust
let nonce_fee = gas_key_add_key_exec_fee(
    fees,
    account_id.len(),
    public_key.trie_id_len(),   // 33 for ML-DSA-65
    gas_key_info.num_nonces,
);
```

**Host-function path** — `promise_batch_action_add_gas_key_with_full_access` in `runtime/near-vm-runner/src/logic/logic.rs`:
```rust
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,    // 1953 for ML-DSA-65 — WRONG
    num_nonces,
);
```

Inside `gas_key_add_key_exec_fee`:
```rust
let nonce_key_len =
    access_key_key_len(account_id_len, public_key_len)  // inflated by 1920 bytes
    + size_of::<NonceIndex>();
let per_byte = gas_key_byte_exec_fee
    .checked_mul((nonce_key_len + NONCE_VALUE_LEN) as u64)
    .checked_mul(num_nonces);
```

For `num_nonces = 1024` and a representative `gas_key_byte` exec fee, the overcharge is `1920 × gas_key_byte_exec_fee × 1024` gas — a deterministic, protocol-level divergence introduced at the `PostQuantumSignatures` upgrade boundary. [9](#0-8) [1](#0-0) [3](#0-2) [2](#0-1)

```rust
    /// Length, in bytes, of the on-trie identifier for an access-key
    /// entry owned by this public key. For ed25519/secp256k1 this matches
    /// `len()`; for ML-DSA-65 the trie stores a SHA3-256 hash (33 bytes
    /// including the type tag), not the 1953-byte borsh-encoded pubkey.
    /// Used by storage-fee calculations on the runtime side; cheap to call
    /// (no hashing) - for ML-DSA-65 this returns the size of the digest
    /// form without actually hashing the pubkey.
    pub fn trie_id_len(&self) -> usize {
        match self {
            Self::ED25519(_) => 1 + ed25519_dalek::PUBLIC_KEY_LENGTH,
            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,
        }
    }
```
