### Title
`TierUndelegate` Permanently Locks Funds When Bonus Pool Is Empty and Pending Events Exist — (`x/tieredrewards/keeper/msg_server.go`, `claim_rewards.go`)

---

### Summary

`TierUndelegate` unconditionally calls `claimRewards`, which calls `processEventsAndClaimBonus`. When the bonus pool is empty and pending validator events (including SLASH events) would generate non-zero bonus, `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`. `TierUndelegate` propagates this error without any fallback, causing the transaction to revert. Since the bonus pool may remain empty indefinitely, the position owner's funds are permanently locked with no recovery path.

---

### Finding Description

**Entrypoint:** `MsgTierUndelegate` → `msgServer.TierUndelegate`

**Exact call chain:**

`TierUndelegate` calls `claimRewards` unconditionally before undelegating: [1](#0-0) 

`claimRewards` calls `processEventsAndClaimBonus` and propagates any error: [2](#0-1) 

Inside `processEventsAndClaimBonus`, a SLASH event on a bonded validator closes a bonded segment and accumulates non-zero `totalBonus`: [3](#0-2) 

Once `totalBonus > 0`, `sufficientBonusPoolBalance` is called. If the pool is empty, it returns `ErrInsufficientBonusPool`, which is returned directly to the caller: [4](#0-3) 

**The asymmetry:** `slashRedelegationPosition` explicitly catches and swallows `ErrInsufficientBonusPool` to prevent a chain halt, with a comment explaining the deliberate design: [5](#0-4) 

No equivalent guard exists in `TierUndelegate` or `claimRewards`. The error propagates all the way up, reverting the entire transaction.

---

### Impact Explanation

The position owner is permanently unable to call `TierUndelegate` (and therefore unable to withdraw funds via `WithdrawFromTier`, which requires the undelegation to complete first) as long as:

1. The bonus pool balance is zero (or below the computed bonus amount), **and**
2. There is at least one pending validator event (SLASH, BOND, or UNBOND) that closes a bonded segment with non-zero duration.

The bonus pool is a module account funded by governance/inflation decisions. It can be empty for extended or indefinite periods. There is no alternative undelegation path in the production code — `TierUndelegate` is the only `MsgServer` method that initiates undelegation for a delegated position.

**Exact state delta:** The position's delegation shares remain locked in the staking module. The position owner's principal is inaccessible. This is a direct fund-loss impact.

---

### Likelihood Explanation

The preconditions are realistic and reachable on mainnet:

- **Bonus pool drains to zero:** Any period where bonus rewards are claimed faster than the pool is replenished (e.g., many positions claiming simultaneously, or governance failing to top up the pool) produces this state.
- **Pending SLASH event exists:** Any validator slash that occurs after the position's `LastEventSeq` was last updated creates a pending event. Slashes are a normal chain operation.
- **Bonded segment with non-zero duration:** Any position that has been delegated for more than one second to a bonded validator satisfies this.

All three conditions can co-occur without any attacker action — they arise from normal chain operation. A malicious actor could also deliberately drain the bonus pool (by claiming rewards from their own positions) before a slash event is processed, then wait for the victim's `TierUndelegate` to fail.

---

### Recommendation

Mirror the pattern already used in `slashRedelegationPosition`: catch `ErrInsufficientBonusPool` in `claimRewards` (or in `TierUndelegate` directly) and skip the bonus payment rather than reverting. The position's event sequence checkpoint (`LastEventSeq`) should still be advanced so the pending events are consumed and the position is not permanently stuck.

```go
// In claimRewards or TierUndelegate:
bonus, err := k.processEventsAndClaimBonus(ctx, &pos)
if err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during undelegate, skipping bonus", ...)
        // bonus forfeited; position checkpoints already advanced inside processEventsAndClaimBonus
    } else {
        return types.PositionState{}, nil, nil, err
    }
}
```

Note: `processEventsAndClaimBonus` already advances `pos.LastEventSeq` and decrements event reference counts inside the loop before reaching the pool check, so the checkpoint state is partially updated on error. The fix must ensure the checkpoint is fully persisted even when the bonus payment is skipped.
