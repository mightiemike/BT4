### Title
Unchecked `approve()` Return Value in Loop Causes Silent Deposit Failure and Token Lockup — (`File: core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` iterates over all registered spot product IDs and calls `token.approve(address(endpoint), balance)` in a loop without checking the boolean return value. For any ERC20 token that signals failure by returning `false` rather than reverting, the approval silently fails, the subsequent `depositCollateralWithReferral` call proceeds without the required allowance, and the deposited tokens remain locked in the DDA contract with no automated recovery path for the user.

---

### Finding Description

`creditDeposit()` is a permissionless function that loops over every product ID returned by `spotEngine.getProductIds()`. For each product with a non-zero balance held by the DDA contract, it executes two sequential external calls:

```solidity
token.approve(address(endpoint), balance);          // line 92 — return value discarded
endpoint.depositCollateralWithReferral(             // line 93-98
    subaccount, productId, uint128(balance), "-1"
);
```

The `IIERC20Base` interface declares `approve` as returning `bool`, but the return value is never captured or checked. [1](#0-0) 

If `approve()` returns `false` without reverting (a pattern present in several deployed ERC20 tokens, including older USDT variants and some rebasing tokens), the allowance is never set. The endpoint's deposit path then calls `transferFrom` on the token to pull funds from the DDA; without the allowance, this either reverts (rolling back the entire `creditDeposit` call and blocking all products in the loop) or silently returns `false` (leaving tokens stranded in the DDA). [2](#0-1) 

The only recovery mechanism for stranded tokens is `withdraw(token)`, which is `onlyOwner` — inaccessible to the user whose funds are stuck. [3](#0-2) 

---

### Impact Explanation

- **Token lockup**: Tokens sent to the DDA contract by a user are not forwarded to the protocol and cannot be self-recovered. Only the contract owner can call `withdraw()` to retrieve them.
- **Loop-wide revert**: If the failing token's `transferFrom` reverts, the entire `creditDeposit()` call reverts, blocking deposits for every product in the same batch — including products whose `approve()` succeeded.
- **Accounting desynchronization**: The on-chain subaccount balance in `SpotEngine` is never updated for the affected product, while the real token balance sits idle in the DDA, creating a permanent divergence between protocol-recorded collateral and actual token custody.

---

### Likelihood Explanation

`creditDeposit()` carries no access control and is callable by any address. [4](#0-3)  The Nado spot engine supports multiple collateral tokens; if any one of them is a non-reverting-on-failure ERC20 (a well-documented class of deployed tokens), the condition is met. The `ContractOwner.creditDepositV1()` wrapper also calls this function permissionlessly, widening the trigger surface. [5](#0-4)  Likelihood is **medium**: it depends on the specific token set registered in the spot engine, but the code provides no defense against this class of token.

---

### Recommendation

Replace the bare `approve()` call with a checked wrapper analogous to the `safeTransfer` / `safeTransferFrom` pattern already used elsewhere in the codebase:

```solidity
(bool success, bytes memory data) = address(token).call(
    abi.encodeWithSelector(IIERC20Base.approve.selector, address(endpoint), balance)
);
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    "Approve failed"
);
```

Alternatively, reset the allowance to zero before setting it (guarding against tokens that reject non-zero-to-non-zero approvals) and use OpenZeppelin's `SafeERC20.safeApprove` or `forceApprove`. [6](#0-5) 

---

### Proof of Concept

1. A spot product is registered in `SpotEngine` whose collateral token implements `approve()` returning `false` on failure (e.g., a token that blocks approvals when paused).
2. A user sends tokens directly to the DDA contract address.
3. Any caller invokes `DirectDepositV1.creditDeposit()`.
4. The loop reaches the affected product; `token.approve(address(endpoint), balance)` returns `false` — no revert, no error.
5. `endpoint.depositCollateralWithReferral(...)` is called; the endpoint attempts `transferFrom(dda, clearinghouse, amount)` which fails because allowance is 0.
6. Depending on the token: the call reverts (entire `creditDeposit` reverts, all products blocked) or returns `false` silently (tokens remain in DDA, subaccount balance never updated).
7. The user's tokens are locked in the DDA; `withdraw()` is `onlyOwner` and unavailable to the user. [7](#0-6)

```text
    function creditDeposit() external {
        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            address tokenAddr = spotEngine.getToken(productId);
            require(tokenAddr != address(0), "Invalid productId.");
            IIERC20Base token = IIERC20Base(tokenAddr);
            uint256 balance = token.balanceOf(address(this));
            if (balance != 0) {
                token.approve(address(endpoint), balance);
                endpoint.depositCollateralWithReferral(
                    subaccount,
                    productId,
                    uint128(balance),
                    "-1"
                );
            }
        }
    }
```
