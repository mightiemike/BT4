### Title
Merkle Index Aliasing: `tx_index = k + 2^N` Passes Proof Verification for Non-Existent Block Position — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` only uses `current_position % 2` at each level and never validates that the supplied index is within the actual leaf range of the tree. This means any index of the form `k + 2^N` (where `N = merkle_proof.len()`) produces an identical traversal path — and therefore an identical computed root — as the valid index `k`. An unprivileged NEAR caller can supply an out-of-range `tx_index` to `verify_transaction_inclusion` and receive `true` for a position that does not exist in the block.

---

### Finding Description

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` iterates over the proof, using only the parity of `current_position` at each level: [1](#0-0) 

At step `i`, the parity check is `(current_position / 2^i) % 2`. For any valid index `k` and proof of length `N`, substituting `k + 2^N`:

```
(k + 2^N) / 2^i  % 2
= (k/2^i + 2^(N-i)) % 2
= k/2^i % 2          // because 2^(N-i) is even for all i < N
```

The parity is identical at every level, so the same proof hashes are combined in the same order, producing the same root. There is no check that `tx_index < 2^N` or that `tx_index` is within the actual transaction count of the block.

`verify_transaction_inclusion` passes the caller-supplied `tx_index` directly to `compute_root_from_merkle_proof` with no bounds validation: [2](#0-1) 

The only guards present are confirmation depth and main-chain membership — neither constrains the index value: [3](#0-2) 

`verify_transaction_inclusion_v2` adds a coinbase proof check but then delegates unconditionally to `verify_transaction_inclusion`, inheriting the same flaw: [4](#0-3) 

Both functions are `pub fn` with only a `#[pause]` gate — no `#[private]`, no role restriction — so any NEAR account can call them.

---

### Impact Explanation

An attacker who knows a real transaction at index `k` in a canonical block (with a proof of depth `N`) can call `verify_transaction_inclusion` with `tx_index = k + 2^N` (or `k + 2*2^N`, `k + 3*2^N`, …) and receive `true`. The function asserts that the transaction exists at a leaf position that is entirely outside the block's actual transaction set (e.g., index 4 in a 4-tx block, index 8 in an 8-tx block).

The concrete downstream harm: any recipient contract that uses `(tx_block_blockhash, tx_index)` as a unique key to prevent replay or double-claiming can be bypassed. The same real transaction can be presented multiple times under distinct aliased indices, each passing the light-client check, allowing repeated redemption of a single on-chain event.

---

### Likelihood Explanation

The exploit requires only knowledge of a real transaction and its valid Merkle proof — both are publicly available from any Bitcoin full node or block explorer. No privileged role, leaked key, or social engineering is needed. The arithmetic is deterministic and requires no brute force.

---

### Recommendation

Add an upper-bound check on `tx_index` before calling `compute_root_from_merkle_proof`. The valid leaf range for a proof of depth `N` is `[0, 2^N)`:

```rust
require!(
    args.tx_index < (1u64 << args.merkle_proof.len()),
    "tx_index out of range for the given proof depth"
);
```

Alternatively, enforce the check inside `compute_root_from_merkle_proof` itself:

```rust
assert!(
    transaction_position < (1 << merkle_proof.len()),
    "transaction_position exceeds tree capacity"
);
```

### Proof of Concept

```rust
#[test]
fn test_index_aliasing() {
    // 4-tx block, proof depth N=2, valid index k=0
    let tx0 = H256([1u8; 32]);
    let tx1 = H256([2u8; 32]);
    let tx2 = H256([3u8; 32]);
    let tx3 = H256([4u8; 32]);

    let proof_for_0 = merkle_proof_calculator(vec![tx0, tx1, tx2, tx3], 0);
    // proof_for_0.len() == 2, so 2^N == 4

    let root_valid   = compute_root_from_merkle_proof(tx0, 0, &proof_for_0);
    let root_aliased = compute_root_from_merkle_proof(tx0, 4, &proof_for_0); // k + 2^N

    // Both equal the real merkle root — index 4 does not exist in a 4-tx block
    assert_eq!(root_valid, root_aliased);
}
```

Running this test against the unmodified `merkle-tools/src/lib.rs` will pass, confirming that `verify_transaction_inclusion` returns `true` for `tx_index = 4` in a 4-transaction block.
