### Title
Missing Interest Accrual Prior to Spot Product Config Update Leads to Incorrect Interest Distribution — (File: `core/contracts/SpotEngine.sol`)

---

### Summary

`SpotEngine.addOrUpdateProduct` overwrites `configs[productId]` — which contains all interest-rate curve parameters — without first accruing pending interest via `_updateState`. Because `_updateState` reads the live config at accrual time and applies it to the full elapsed `dt`, any config change retroactively distorts the interest computed for the entire unaccrued window.

---

### Finding Description

`SpotEngineState._updateState` computes the borrower rate and minimum deposit rate entirely from the live `configs[productId]` at the moment it is called:

- Lines 70–92 derive `borrowerRateX18` from `config.interestFloorX18`, `config.interestSmallCapX18`, `config.interestLargeCapX18`, and `config.interestInflectionUtilX18`.
- Lines 148–169 apply `config.minDepositRateX18` as a subsidy multiplier to both `cumulativeDepositsMultiplierX18` and `cumulativeBorrowsMultiplierX18`. [1](#0-0) 

The entire interest computation is then applied to the full `dt` window passed in from the endpoint: [2](#0-1) 

`SpotEngine.addOrUpdateProduct` updates `configs[productId]` directly, with no prior call to `_updateState`:

```solidity
// SpotEngine.sol line 83
configs[productId] = config;
``` [3](#0-2) 

If `addOrUpdateProduct` is called between two `updateStates` invocations, the next `updateStates` call will apply the **new** config parameters to the **entire** elapsed `dt`, including the portion of time that accrued under the old parameters. There is no checkpoint or partial-accrual mechanism.

---

### Impact Explanation

Depending on the direction of the change:

- **Rate increase** (e.g., `interestFloorX18` raised, or `minDepositRateX18` raised): depositors receive more interest than they earned; borrowers are charged more than they owe for the pre-change window.
- **Rate decrease** (e.g., `interestFloorX18` lowered, or `minDepositRateX18` lowered): depositors receive less interest than they earned; borrowers are undercharged.

The `INTEREST_FEE_FRACTION` split between depositors and the `FEES_ACCOUNT` is also applied to the miscalculated base, so the fees balance is corrupted in the same direction. [4](#0-3) 

The magnitude scales with both the size of the rate change and the length of the unaccrued window. Products with low trading activity (infrequent `updateStates` calls) are most exposed, exactly mirroring the original report's observation.

---

### Likelihood Explanation

`addOrUpdateProduct` is a routine admin operation used to tune interest-rate curves as market conditions change. Any such update between two `updateStates` epochs silently corrupts the accrual. The admin need not be malicious; the bug fires on every legitimate config update that occurs while interest is pending. Low-activity products, where `updateStates` may not be called for hours or days, amplify the window of exposure. [5](#0-4) 

---

### Recommendation

Before writing the new config, force an interest accrual for the product using the old parameters. Concretely, inside `addOrUpdateProduct`, before `configs[productId] = config`, call `_updateState` with the current state and the elapsed `dt` (as computed by the endpoint), then persist the updated state. This ensures the old curve is applied to the period it was active for, and the new curve only takes effect from the moment of the update onward.

---

### Proof of Concept

1. A spot product has `interestFloorX18 = 1e16` (1 % annualised floor). `updateStates` was last called 12 hours ago; no accrual has occurred since.
2. The admin calls `addOrUpdateProduct` with `interestFloorX18 = 5e16` (5 % floor). `configs[productId]` is overwritten immediately; no interest is accrued for the preceding 12-hour window.
3. 12 hours later, `updateStates` is called with `dt = 86400` (24 hours total). `_updateState` reads the new 5 % floor and applies it to the full 24-hour `dt`.
4. Depositors receive interest computed at 5 % for 24 hours, when they should have received 1 % for the first 12 hours and 5 % for the second 12 hours. The `FEES_ACCOUNT` balance and `cumulativeDepositsMultiplierX18` are both inflated relative to the correct values. [6](#0-5) [7](#0-6)

    function _updateState(
        uint32 productId,
        State memory state,
        uint128 dt
    ) internal {
        int128 borrowRateMultiplierX18;
        int128 totalDeposits = state.totalDepositsNormalized.mul(
            state.cumulativeDepositsMultiplierX18
        );
        int128 totalBorrows = state.totalBorrowsNormalized.mul(
            state.cumulativeBorrowsMultiplierX18
        );
        int128 utilizationRatioX18 = totalBorrows.div(totalDeposits);
        int128 minDepositRateX18;
        {
            Config memory config = configs[productId];

            // annualized borrower rate
            int128 borrowerRateX18 = config.interestFloorX18;
            if (utilizationRatioX18 == 0) {
                // setting borrowerRateX18 to 0 here has the property that
                // adding a product at the beginning of time and not using it until time T
                // results in the same state as adding the product at time T
                borrowerRateX18 = 0;
            } else if (utilizationRatioX18 < config.interestInflectionUtilX18) {
                borrowerRateX18 += config
                    .interestSmallCapX18
                    .mul(utilizationRatioX18)
                    .div(config.interestInflectionUtilX18);
            } else {
                borrowerRateX18 +=
                    config.interestSmallCapX18 +
                    config.interestLargeCapX18.mul(
                        (
                            (utilizationRatioX18 -
                                config.interestInflectionUtilX18).div(
                                    ONE - config.interestInflectionUtilX18
                                )
                        )
                    );
            }

            // convert to per second
            borrowerRateX18 = borrowerRateX18.div(
                MathSD21x18.fromInt(31536000)
            );
            borrowRateMultiplierX18 = (ONE + borrowerRateX18).pow(int128(dt));
            minDepositRateX18 = config.minDepositRateX18;
        }

    function updateStates(uint128 dt) external onlyEndpoint {
        State memory quoteState;
        require(dt < 7 * SECONDS_PER_DAY, ERR_INVALID_TIME);
        for (uint32 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            if (productId == NLP_PRODUCT_ID) {
                continue;
            }
            State memory state = states[productId];
            if (productId == QUOTE_PRODUCT_ID) {
                quoteState = state;
            }
            if (state.totalDepositsNormalized == 0) {
                continue;
            }
            _updateState(productId, state, dt);
            _setState(productId, state);
        }
    }

    function addOrUpdateProduct(
        uint32 productId,
        uint32 quoteId,
        int128 sizeIncrement,
        int128 minSize,
        Config calldata config,
        RiskHelper.RiskStore calldata riskStore
    ) public onlyOwner {
        bool isNewProduct = _addOrUpdateProduct(
            productId,
            quoteId,
            sizeIncrement,
            minSize,
            riskStore
        );
        configs[productId] = config;

        if (isNewProduct) {
            require(productId != QUOTE_PRODUCT_ID);
            _setState(
                productId,
                State({
                    cumulativeDepositsMultiplierX18: ONE,
                    cumulativeBorrowsMultiplierX18: ONE,
                    totalDepositsNormalized: 0,
                    totalBorrowsNormalized: 0
                })
            );
        }
    }
