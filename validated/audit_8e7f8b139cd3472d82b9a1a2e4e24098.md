Audit Report

## Title
L2 Price Providers Lack Sequencer Uptime Feed Check, Allowing Stale Oracle Prices to Reach Pool Swaps — (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

## Summary
Both `PriceProviderL2` and `ProtectedPriceProviderL2` guard oracle freshness solely via a timestamp delta comparison in `_isStale`, with no Chainlink L2 Sequencer Uptime Feed check. When the Arbitrum sequencer is down, push-based oracle data stops updating while `block.timestamp` continues to advance via L1 forced-inclusion. If `MAX_TIME_DELTA` is configured at or above the forced-inclusion delay (~24 h), the staleness check passes and a stale bid/ask price is delivered to the pool swap, enabling direct LP fund loss.

## Finding Description
`_isStale` in both contracts is a pure timestamp comparison with no sequencer liveness signal:

```solidity
// PriceProviderL2.sol L135-150 / ProtectedPriceProviderL2.sol L138-153
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta, uint256 futureTol)
    internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return (refTime - nowTs) > futureTol;
    return (nowTs - refTime) > maxDelta;
}
```

The constructor enforces `MAX_TIME_DELTA <= 7 days` but imposes no lower bound relative to the Arbitrum forced-inclusion delay:

```solidity
// PriceProviderL2.sol L92 / ProtectedPriceProviderL2.sol L96
if (_maxTimeDelta == 0 || _maxTimeDelta > 7 days) revert MaxTimeDeltaOutOfBounds();
```

`_getBidAndAskPrice` (PriceProviderL2, L208-248) and `_computeBidAsk` (ProtectedPriceProviderL2, L203-238) pass the oracle price directly into bid/ask computation whenever `_isStale` returns `false`. No other sequencer health guard exists anywhere in either contract.

**Exploit path:**
1. Sequencer goes offline at time `T₀`; last oracle push sets `refTime = T₀`, price = `P₀`.
2. Market price moves significantly during outage; no oracle update reaches the chain.
3. At `T₀ + 24h`, attacker submits a swap via the L1 delayed inbox (permissionless forced inclusion). `block.timestamp ≈ T₀ + 24h`.
4. `_isStale` evaluates: `(T₀ + 24h) − T₀ = 24h`. If `MAX_TIME_DELTA ≥ 24h` → **not stale**.
5. `_getBidAndAskPrice` / `_computeBidAsk` return bid/ask derived from stale `P₀`.
6. Pool executes swap at stale price; attacker profits at LP expense.

## Impact Explanation
Stale bid/ask price reaches a live pool swap — matching the **bad-price execution** and **direct loss of LP principal** impact categories. LPs suffer direct loss: the attacker buys the underpriced asset or sells the overpriced asset against LP liquidity at the frozen pre-downtime rate. Severity: **Medium** (requires sequencer downtime + `MAX_TIME_DELTA ≥ 24h`, both realistic).

## Likelihood Explanation
- Arbitrum sequencer outages have occurred historically (multiple incidents 2023–2024).
- `MAX_TIME_DELTA` up to 7 days is permitted by the constructor; any value ≥ 24 h makes the attack viable.
- The forced-inclusion path is permissionless — any address can submit after the delay.
- No sequencer uptime feed or circuit-breaker exists anywhere in either L2 provider.

## Recommendation
Add a Chainlink L2 Sequencer Uptime Feed check at the top of `_getBidAndAskPrice` / `_computeBidAsk` in both contracts. If the sequencer is reported down or the uptime feed answer timestamp is itself stale, return `(0, type(uint128).max)` to halt swaps:

```solidity
(, int256 answer, uint256 startedAt, ,) = sequencerUptimeFeed.latestRoundData();
if (answer != 0) return (0, type(uint128).max);
if (block.timestamp - startedAt < GRACE_PERIOD) return (0, type(uint128).max);
```

Additionally, cap `MAX_TIME_DELTA` to a value well below the forced-inclusion delay (e.g., ≤ 1 hour) to reduce the exploitable window even without the uptime feed.

## Proof of Concept
1. Deploy `PriceProviderL2` on Arbitrum fork with `MAX_TIME_DELTA = 2 days`. Configure a pool using this provider.
2. Simulate sequencer downtime: freeze oracle state at `refTime = T₀`, price = `P₀`.
3. Warp `block.timestamp` to `T₀ + 24h` (simulating forced-inclusion execution).
4. Call `getBidAndAskPrice()` — `_isStale` returns `false` (24h < 2 days).
5. Confirm returned bid/ask is derived from stale `P₀`, not current market price.
6. Execute swap at stale price; verify LP balance decreases by the price-discrepancy amount.
