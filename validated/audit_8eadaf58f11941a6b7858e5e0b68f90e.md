Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the end-user, allowing any caller to bypass a curated pool's per-user allowlist - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which is forwarded to `SwapAllowlistExtension.beforeSwap`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the end-user. If the pool admin allowlists the router address to enable standard periphery UX, the extension's per-user gate is silently voided: every caller of the router passes the check regardless of whether they are individually approved.

## Finding Description

`MetricOmmPool.swap` invokes `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

That value arrives at the extension as `sender`. `SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [2](#0-1) 

Here `msg.sender` is the pool (the extension's caller) and `sender` is whoever called `pool.swap()`. When a user calls `MetricOmmSimpleRouter.exactInputSingle` or `exactInput`, the router calls `pool.swap()` directly: [3](#0-2) 

The pool's `msg.sender` is the router address. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. A pool admin who adds the router to the allowlist so that approved users can access the standard UX inadvertently grants every caller of the router — approved or not — a passing check. The lookup `allowedSwapper[pool][charlie]` for an unapproved user is never evaluated.

`DepositAllowlistExtension` avoids this problem by checking `owner` (the position beneficiary passed explicitly), not `sender`: [4](#0-3) 

No analogous trusted-forwarder field exists for swaps, so `SwapAllowlistExtension` has no way to recover the true end-user identity once the router is in the call chain.

## Impact Explanation

Any user not on the allowlist can swap on a curated pool by routing through `MetricOmmSimpleRouter` once the router is allowlisted. The pool's curation policy — which may restrict access to institutional counterparties, comply with regulatory requirements, or protect LP capital from adversarial flow — is completely bypassed. Unrestricted swappers can extract value from the pool's oracle-priced bins at will, causing direct LP principal loss. This constitutes a broken core pool invariant (the allowlist extension's gating function) and direct loss of user/LP principal, meeting the High severity threshold.

## Likelihood Explanation

The trigger is a routine, semi-trusted admin action: allowlisting the router so that approved users can access the standard periphery UX. This is the expected operational path for any production pool that uses `SwapAllowlistExtension` alongside `MetricOmmSimpleRouter`. Once the router is allowlisted, the bypass is available to every unprivileged user with no further preconditions, no special tokens, and no privileged access. The condition is both likely to occur in practice and trivially exploitable once it does.

## Recommendation

Pass the originating end-user's address through the call chain rather than the direct pool caller. Concrete options:

1. **`extensionData` forwarding**: Have the router encode `msg.sender` (the true user) into `extensionData`; have the extension decode and verify it, optionally requiring that the encoding came from a trusted router (e.g., via a factory-registered router registry).
2. **Explicit `swapper` parameter**: Add a `swapper` field to `pool.swap()` distinct from `msg.sender`, populated by the router with the originating user.
3. **Router-aware extension**: Deploy a separate extension that, when `sender` is a known router, reads the user from a trusted field in `extensionData` and falls back to `sender` otherwise.

At minimum, document prominently that allowlisting the router disables per-user gating.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is intended to swap.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` — to let Alice use the router UX.
4. Charlie (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the pool.
5. The router calls `pool.swap(...)` — pool's `msg.sender` = router address.
6. `_beforeSwap(router, ...)` is called; extension evaluates `allowedSwapper[pool][router]` → `true`.
7. Charlie's swap executes successfully despite not being on the allowlist.
8. The check `allowedSwapper[pool][charlie]` is never evaluated; LP principal is reduced.
