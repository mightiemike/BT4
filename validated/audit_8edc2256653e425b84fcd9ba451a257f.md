Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks LP-share recipient (`owner`) instead of token provider (`sender`), allowing full allowlist bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` to the pool who provides tokens via callback) and enforces the allowlist only against `owner` (the LP-share recipient). Because `MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` with no ownership constraint, any non-allowlisted address can pass the guard by supplying any allowlisted address as `owner`, depositing tokens into a restricted pool while the allowlisted address receives unwanted LP shares.

## Finding Description
`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` with no `msg.sender == owner` check and forwards both to the extension hook: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` passes `sender` (= `msg.sender` of the pool call) and `owner` verbatim: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` names the first parameter (sender) with `address` and no identifier — it is silently discarded — and checks only `owner`: [3](#0-2) 

The parallel guard `SwapAllowlistExtension.beforeSwap` correctly checks `sender`: [4](#0-3) 

The asymmetry is the root cause. `sender` is the entity that triggers the call and must satisfy the `metricOmmAddLiquidityCallback` to transfer tokens into the pool. `owner` is merely the address credited with LP shares and is a free, unconstrained parameter. The guard therefore checks the wrong actor.

## Impact Explanation
The deposit allowlist is the pool admin's mechanism to restrict which addresses may inject liquidity (e.g., KYC-gated, institutional, or sanction-screened pools). With this bug, a non-allowlisted address can deposit tokens into any restricted pool in a single transaction, violating the access-control invariant the admin configured. The non-allowlisted address controls which bins receive liquidity, enabling targeted manipulation of the pool's bin distribution followed by value extraction through swaps at prices the manipulated bin state permits — a direct LP-principal loss path. Even absent active exploitation, the compliance guarantee is silently void. This constitutes an admin-boundary break by an unprivileged path and broken core pool functionality with fund-flow consequence.

## Likelihood Explanation
Exploitation requires only: (1) a pool deployed with `DepositAllowlistExtension` and at least one allowlisted address (trivially true for any live restricted pool), and (2) the attacker calls `addLiquidity` with `owner` set to that allowlisted address. No privileged access, special token, oracle manipulation, or multi-step setup is needed. Any EOA or contract can execute this in a single transaction, and it is repeatable indefinitely.

## Recommendation
Change `beforeAddLiquidity` to check `sender` (the actual token provider) instead of `owner`, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If gating on the LP-share recipient (`owner`) is intentional for some use case, a separate `sender` check must also be added so the token-provider path is gated, and the storage mapping and NatSpec must be updated to make the dual intent explicit.

## Proof of Concept
```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][alice] = true
  bob is NOT on the allowlist

Attack (single transaction):
  bob calls pool.addLiquidity(owner = alice, salt, deltas, callbackData, extensionData)

Execution path:
  MetricOmmPool.addLiquidity:
    _beforeAddLiquidity(msg.sender=bob, owner=alice, ...)
      → DepositAllowlistExtension.beforeAddLiquidity(sender=bob [discarded], owner=alice)
      → checks allowedDepositor[pool][alice] == true  ✓  (guard passes)
  LiquidityLib.addLiquidity credits LP shares to alice
  pool calls bob.metricOmmAddLiquidityCallback(...)
  bob transfers tokens into the pool

Result:
  bob (non-allowlisted) has deposited tokens into the restricted pool
  alice holds LP shares she did not request
  the deposit allowlist invariant is broken
```

Foundry test plan: deploy pool with `DepositAllowlistExtension`, allowlist `alice`, call `addLiquidity` from `bob` with `owner = alice`, assert the call succeeds and `bob`'s tokens are transferred into the pool.
