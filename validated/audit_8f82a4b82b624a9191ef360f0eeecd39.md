### Title
Linked Signer Can Silently Re-Delegate Signing Authority to an Arbitrary Address — (`File: core/contracts/EndpointTx.sol`)

### Summary
When a user authorizes a linked signer for their subaccount, the scope of that authorization is broader than what the user signs. The linked signer can itself sign a new `LinkSigner` transaction (because `allowLinkedSigner = true` in the fast path) to silently replace itself with any attacker-controlled address. The subaccount owner never signed for this re-delegation, and the scope of what they authorized is never disclosed to them. This is a direct analog to the `mina_getState` credential-scope disclosure bug: the user consents to a narrow authorization but the protocol silently permits a broader one.

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ← allowLinkedSigner
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` calls `validateSignature`, which passes the current linked signer as an accepted signer when `allowLinkedSigner` is `true`:

```solidity
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

This means the current linked signer's signature is accepted as valid authorization for a `LinkSigner` transaction that replaces the linked signer with a new address. The subaccount owner signed only to authorize signer A; they never signed anything that disclosed signer A could further delegate to signer B, C, etc.

The slow-mode `LinkSigner` path does **not** have this problem — it uses `validateSender`, which enforces `msg.sender == address(bytes20(txn.sender))`, so only the subaccount owner's EOA can submit it:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    ...
    validateSender(txn.sender, sender);   // msg.sender must be owner
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [3](#0-2) 

The fast path (sequencer-submitted) has no such restriction, so the linked signer can re-delegate through the sequencer at any time.

Additionally, `getLinkedSigner` propagates the parent subaccount's linked signer to **all** isolated subaccounts automatically:

```solidity
function getLinkedSigner(bytes32 subaccount) public view virtual returns (address) {
    return
        RiskHelper.isIsolatedSubaccount(subaccount)
            ? linkedSigners[
                IOffchainExchange(offchainExchange).getParentSubaccount(subaccount)
              ]
            : linkedSigners[subaccount];
}
``` [4](#0-3) 

The user who links signer A to their main subaccount is never shown that signer A automatically inherits authority over every isolated subaccount they create. This is the second dimension of undisclosed scope.

### Impact Explanation

A malicious or compromised linked signer can:

1. **Persistently re-delegate authority**: Sign a `LinkSigner` to replace itself with attacker address B. If the owner revokes, attacker B signs another `LinkSigner` to set attacker C. The owner can never permanently revoke without winning a continuous nonce race against the attacker.
2. **Sign orders on behalf of the subaccount**: The linked signer can sign limit/market orders. A colluding sequencer or a counterparty willing to trade at unfavorable prices can drain the subaccount through matched trades.
3. **Sign liquidations**: The linked signer can initiate liquidations on behalf of the subaccount, paying `LIQUIDATION_FEE` and acquiring liquidated assets — potentially at terms unfavorable to the subaccount.
4. **Scope extends to all isolated subaccounts**: All isolated subaccounts of the parent inherit the same linked signer, so the attacker's authority is broader than the user authorized.

Direct fund theft via withdrawal is blocked (withdrawals go to the owner's address), but persistent signing authority combined with order placement creates a realistic path to subaccount drain.

### Likelihood Explanation

Any user who sets a linked signer — a common operation for automated trading bots, API integrations, or institutional setups — is exposed. The attacker precondition is only that the linked signer key is compromised or was malicious from the start (e.g., a third-party trading service). No admin keys, sequencer compromise, or governance capture is required. The re-delegation attack is executable by submitting a single valid signed transaction to the sequencer.

### Recommendation

Change `allowLinkedSigner` to `false` for `LinkSigner` transactions in `processTransactionImpl`. Only the subaccount owner (whose address is embedded in the `sender` bytes32) should be permitted to change the linked signer. The slow-mode path already enforces this correctly via `validateSender`; the fast path should match it:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only owner may re-link
);
```

Additionally, consider surfacing to users (in documentation or UI) that linking a signer to a main subaccount automatically grants that signer authority over all current and future isolated subaccounts.

### Proof of Concept

1. User submits slow-mode `LinkSigner{sender: S, signer: A, nonce: 0}` — sets `linkedSigners[S] = A`.
2. Attacker (holding key A) constructs `SignedLinkSigner{tx: LinkSigner{sender: S, signer: B, nonce: 1}, signature: sig_A}` and submits it to the sequencer via the fast path.
3. Sequencer calls `submitTransactionsChecked`, which calls `processTransactionImpl`. `validateSignedTx` accepts `sig_A` because `allowLinkedSigner = true` and `getLinkedSigner(S) == A`. `linkedSigners[S]` is updated to `B`.
4. User detects the change and submits slow-mode `LinkSigner{sender: S, signer: 0, nonce: 2}` to revoke. This takes 3 days (`SLOW_MODE_TX_DELAY`) to execute.
5. Before the slow-mode revocation executes, attacker (holding key B) submits fast-path `LinkSigner{sender: S, signer: C, nonce: 3}` signed by B. Sequencer processes it immediately. `linkedSigners[S] = C`.
6. The 3-day slow-mode revocation executes and sets `linkedSigners[S] = 0`, but attacker (holding key C) immediately re-links to D with nonce 4.
7. The attacker maintains persistent signing authority indefinitely, with the user never having authorized any address beyond A.
