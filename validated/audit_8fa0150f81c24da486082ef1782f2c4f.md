Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on `owner` Instead of `sender`, Allowing Any Unauthorized Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` checks the caller-supplied `owner` parameter (the position recipient) rather than `sender` (the actual depositor who provides tokens via callback). Because `MetricOmmPool.addLiquidity` imposes no `msg.sender == owner` constraint, any address not on the allowlist can bypass the deposit gate by setting `owner` to any address already on the allowlist, completely defeating the extension's access-control purpose.

## Finding Description
`MetricOmmPool.addLiquidity` accepts a free `owner` argument and forwards it alongside `msg.sender` to the extension hook:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

No guard enforces `msg.sender == owner` for `addLiquidity` (only `removeLiquidity` has this check at L206). The extension hook signature is:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
```

The first parameter (`sender` = `msg.sender`) is silently discarded. The check reads:

```solidity
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
}
```

`msg.sender` here is the pool (the extension is called by the pool), so the lookup is `allowedDepositor[pool][owner]`. An unauthorized caller `bob` sets `owner = alice` where `allowedDepositor[pool][alice] == true`; the check passes. The liquidity callback is then invoked on `bob` (`msg.sender` in the pool context), pulling tokens from `bob`. A position is minted for `alice`. Since `removeLiquidity` only requires `msg.sender == owner`, `alice` can then withdraw the tokens. By contrast, `SwapAllowlistExtension.beforeSwap` correctly gates on `sender`.

## Impact Explanation
The deposit allowlist is the sole on-chain mechanism for pool admins to restrict liquidity provision to KYC'd or trusted addresses. This bug renders it completely ineffective: any unauthorized address can inject liquidity into the pool by naming any known allowed address as `owner`. This constitutes an admin-boundary break — an unprivileged path bypasses a pool-admin-configured access control — and breaks core pool functionality (curated pool deposit restriction). If the unauthorized depositor and the allowed address collude, the allowed address withdraws the tokens, laundering unauthorized liquidity through the curated pool.

## Likelihood Explanation
The bypass requires a single direct call to `pool.addLiquidity` with `owner` set to any address in the public `allowedDepositor` mapping. No flash loans, special privileges, or complex setup are needed. The allowed address list is fully public on-chain. Any motivated actor can trivially identify an allowed address and execute the bypass repeatedly.

## Recommendation
Change `beforeAddLiquidity` to gate on `sender` (the actual depositor) rather than `owner`, consistent with `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Also update `isAllowedToDeposit`, NatDoc, and the `allowedDepositor` mapping key semantics to reflect that the controlled actor is the depositor/payer, not the position owner.

## Proof of Concept
```
Setup:
  - Pool deployed with DepositAllowlistExtension
  - allowedDepositor[pool][alice] = true
  - bob is NOT on the allowlist and holds token0/token1

Attack:
  1. bob calls pool.addLiquidity(alice, salt, deltas, callbackData, "")
  2. Extension checks allowedDepositor[pool][alice] → true → passes
  3. Pool invokes liquidity callback on bob; bob transfers tokens to pool
  4. Position minted for alice at (alice, salt)
  5. alice calls pool.removeLiquidity(alice, salt, deltas, "") → msg.sender == owner check passes
  6. alice receives tokens; alice and bob split off-chain

Result: bob bypassed the deposit allowlist entirely; the curated pool accepted liquidity from an unauthorized address.
```
