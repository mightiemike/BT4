### Title
`max_receipt_size` Invariant Bypassed via Post-Validation `output_data_receivers` Mutation and `DataReceipt` Envelope Overflow — (`runtime/runtime/src/lib.rs`, `runtime/runtime/src/verifier.rs`)

---

### Summary

The `max_receipt_size` limit (4,194,304 bytes) is enforced in `validate_receipt` only at the instant a new receipt object is constructed. Two separate code paths in `apply_action_receipt` then mutate receipts **after** that validation has already passed, producing receipts whose Borsh-serialized size exceeds `max_receipt_size`. Any unprivileged user can trigger either path by deploying a contract that uses `promise_return` or `value_return`. The resulting oversized receipts enter the chain, are tolerated by `ValidateReceiptMode::ExistingReceipt`, and are explicitly acknowledged as an open bug (nearcore issue #12606).

---

### Finding Description

**Invariant being violated**

`max_receipt_size` (= 4 MiB, introduced in protocol version 69) is the hard upper bound on the Borsh-serialized size of any receipt that enters the chain. It is enforced in `validate_receipt`:

```rust
// runtime/runtime/src/verifier.rs:533-542
if mode == ValidateReceiptMode::NewReceipt {
    let receipt_size: u64 =
        borsh::object_length(receipt).unwrap()...;
    if receipt_size > limit_config.max_receipt_size {
        return Err(ReceiptValidationError::ReceiptSizeExceeded { ... });
    }
}
``` [1](#0-0) 

This check is called from `apply_action_receipt` immediately after each action produces new receipts:

```rust
// runtime/runtime/src/lib.rs:856-865
if new_result.result.is_ok() {
    if let Err(e) = new_result.new_receipts.iter().try_for_each(|receipt| {
        validate_receipt(..., ValidateReceiptMode::NewReceipt)
    }) {
        new_result.result = Err(ActionErrorKind::NewReceiptValidationError(e).into());
    }
}
``` [2](#0-1) 

**Bypass path 1 — `promise_return` (`output_data_receivers` appended post-validation)**

After the per-action validation loop completes, `apply_action_receipt` checks whether the parent receipt had `output_data_receivers`. If the contract returned `ReturnData::ReceiptIndex` (i.e., `promise_return`), the runtime **mutates** the already-validated child receipt by appending the parent's `output_data_receivers` to it:

```rust
// runtime/runtime/src/lib.rs:1019-1037
if !action_receipt.output_data_receivers().is_empty() {
    if let Ok(ReturnData::ReceiptIndex(receipt_index)) = result.result {
        match result.new_receipts.get_mut(receipt_index as usize)...receipt_mut() {
            ReceiptEnum::Action(new_action_receipt) | ... =>
                new_action_receipt.output_data_receivers
                    .extend_from_slice(&action_receipt.output_data_receivers()),
            ...
        }
    }
``` [3](#0-2) 

A contract can craft receipt C with Borsh size exactly equal to `max_receipt_size` (with empty `output_data_receivers`). It passes `validate_receipt`. Then the runtime appends one or more `DataReceiver` entries from the parent receipt, pushing the Borsh size above `max_receipt_size`. No re-validation occurs.

**Bypass path 2 — `value_return` (`DataReceipt` envelope overhead)**

When a contract returns `ReturnData::Value`, the runtime wraps the raw bytes in a `DataReceipt` inside a `Receipt::V0` envelope:

```rust
// runtime/runtime/src/lib.rs:1044-1056
result.new_receipts.extend(action_receipt.output_data_receivers().iter().map(
    |data_receiver| {
        Receipt::V0(ReceiptV0 {
            ...
            receipt: ReceiptEnum::Data(DataReceipt {
                data_id: data_receiver.data_id,
                data: data.clone(),
            }),
        })
    },
));
``` [4](#0-3) 

The only size check on the return value is in `value_return` (VM host function), which checks `max_length_returned_data` (= 4 MiB) against the **raw data length**:

```rust
// runtime/near-vm-runner/src/logic/logic.rs:3882-3888
if num_bytes > self.config.limit_config.max_length_returned_data {
    return Err(HostError::ReturnedValueLengthExceeded { ... }.into());
}
``` [5](#0-4) 

`validate_data_receipt` also only checks the raw data length, not the total Borsh size of the enclosing `Receipt`:

```rust
// runtime/runtime/src/verifier.rs:619-631
fn validate_data_receipt(...) {
    let data_len = receipt.data.as_ref().map(|data| data.len()).unwrap_or(0);
    if data_len as u64 > limit_config.max_length_returned_data { ... }
}
``` [6](#0-5) 

A return value of exactly `max_length_returned_data` bytes passes both checks, but the resulting `Receipt::V0 { ReceiptEnum::Data(DataReceipt { data_id, data }) }` Borsh envelope adds ~40+ bytes of overhead, making the total receipt size exceed `max_receipt_size`.

---

### Impact Explanation

Oversized receipts (> 4 MiB) that enter the chain violate the `max_receipt_size` protocol invariant. The state witness size budget targets ≤ 17 MiB total; a single oversized receipt can consume more than its allocated share. Chunk validators independently re-execute chunk application and enforce size limits; if a chunk producer includes an oversized receipt that validators reject, the chunk will not be endorsed, causing liveness degradation. Additionally, the bandwidth scheduler's receipt-group metadata assumes all receipts are ≤ `max_receipt_size`, and the workaround in `try_forward` (clamping the reported size) means the oversized receipt consumes more bandwidth than accounted for, corrupting bandwidth accounting.

---

### Likelihood Explanation

Any user who can deploy a contract can trigger either bypass. The `promise_return` path requires a two-step promise DAG (A → B, where A does `promise_return(C)`) with C sized near `max_receipt_size`. The `value_return` path requires a contract that returns exactly `max_length_returned_data` bytes inside a `promise_then` chain. Both patterns are standard NEAR contract idioms. The bug is confirmed reachable by the integration tests `test_max_receipt_size_promise_return` and `test_max_receipt_size_value_return` in `test-loop-tests/src/tests/max_receipt_size.rs`. [9](#0-8) [10](#0-9) 

---

### Recommendation

**Path 1 fix**: After appending `output_data_receivers` to the child receipt at `lib.rs:1029-1035`, re-validate the mutated receipt's Borsh size against `max_receipt_size` before forwarding it.

**Path 2 fix**: In the `DataReceipt` creation block at `lib.rs:1044-1056`, compute the Borsh size of the full `Receipt::V0` envelope (not just `data.len()`) and reject it if it exceeds `max_receipt_size`. Alternatively, tighten `max_length_returned_data` to `max_receipt_size - RECEIPT_ENVELOPE_OVERHEAD` so the envelope can never push the total over the limit.

Both fixes should be gated behind a new `ProtocolFeature` to ensure deterministic activation across all validators simultaneously.

---

### Proof of Concept

**Path 1 (`promise_return`)**:

1. Deploy a contract with method `method1` that:
   - Creates promise A calling `method2` with args sized to make receipt A's Borsh size = `max_receipt_size - sizeof(DataReceiver)` (with empty `output_data_receivers`)
   - Creates promise B calling `noop`
   - Calls `promise_then(A, self, "noop", ...)` so A has one `output_data_receiver` pointing to B
   - Calls `promise_return(A)`

2. Call `method1` from a transaction that itself has `output_data_receivers` (i.e., is part of a `promise_then` chain).

3. When `method1` executes, receipt A is validated at size = `max_receipt_size - sizeof(DataReceiver)` → passes. Then the runtime appends the parent's `DataReceiver` to A's `output_data_receivers`, making A's Borsh size = `max_receipt_size + sizeof(DataReceiver)` > `max_receipt_size`. No re-validation. The oversized receipt A enters the chain.

The exact `args_size` calculation is demonstrated in the test at: [11](#0-10) 

**Path 2 (`value_return`)**:

1. Deploy a contract with method `return_large_value` that calls `value_return` with exactly `max_receipt_size` (= 4,194,304) bytes.
2. Call it inside a `promise_then` chain: `promise_then(promise_create(self, "return_large_value", ...), self, "noop", ...)`.
3. When `return_large_value` executes, `value_return` checks `4194304 <= max_length_returned_data` → passes. The runtime wraps the data in `Receipt::V0 { ReceiptEnum::Data(DataReceipt { data_id, data: Some(vec![0u8; 4194304]) }) }`. The Borsh size of this receipt is 4,194,304 + ~40 bytes of envelope > `max_receipt_size`. The oversized data receipt enters the chain. [12](#0-11)
