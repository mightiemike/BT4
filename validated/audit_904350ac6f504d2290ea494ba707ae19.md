### Title
`DirectDepositV1::creditDeposit` Reverts for Tokens That Do Not Return a Bool on `approve` — (`core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1::creditDeposit` calls `token.approve(address(endpoint), balance)` directly via the `IIERC20Base` interface, which declares `approve` as returning `bool`. For tokens that do not return a bool (e.g., USDT), Solidity's ABI decoder will revert, permanently blocking the DDA deposit flow for those tokens.

---

### Finding Description

In `DirectDepositV1.sol`, the `creditDeposit()` function iterates over all spot product IDs, checks each token balance held by the DDA contract, and attempts to approve the endpoint before depositing:

```solidity
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(...);
```

The `IIERC20Base` interface declares `approve` as:

```solidity
function approve(address spender, uint256 amount) external returns (bool);
```

When Solidity calls a function declared to return `bool`, it performs a return-data length check. If the token's `approve` implementation returns no data (as USDT does on mainnet), the ABI decoder reverts. The entire `creditDeposit()` call fails, and no collateral is deposited for any product in that iteration.

Notably, the codebase already recognizes this pattern: `ERC20Helper.sol` implements `safeTransfer` and `safeTransferFrom` using low-level `.call()` with `data.length == 0 || abi.decode(data, (bool))` to handle non-standard tokens. The same safe pattern was not applied to `approve`.

The same raw `.approve()` pattern also appears in `ContractOwner::wrapVaultAsset` (lines 530–531), which is also callable by any external account with no access restriction.

---

### Impact Explanation

Any user who sends a non-bool-returning token (e.g., USDT) to their DDA address and calls `creditDeposit()` (or triggers it via `ContractOwner::creditDepositV1`) will have the transaction revert. The tokens remain stuck in the DDA contract and cannot be deposited as collateral through the normal flow. This breaks the core deposit mechanism for any supported token that does not return a bool on `approve`.

---

### Likelihood Explanation

`creditDeposit()` has no access control — it is callable by any external account. Any user interacting with the DDA deposit flow for a non-standard token will trigger this revert. USDT is a widely used token and a realistic candidate for a supported spot product.

---

### Recommendation

Replace the direct `token.approve(...)` call with a low-level safe approve pattern, analogous to the existing `ERC20Helper.safeTransfer`:

```solidity
function safeApprove(IIERC20Base self, address spender, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IIERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        "Approve failed"
    );
}
```

Apply the same fix to `ContractOwner::wrapVaultAsset` at lines 530–531.

---

### Proof of Concept

1. A spot product is configured with USDT as its token.
2. A user sends USDT to their DDA address.
3. The user (or anyone) calls `ContractOwner::creditDepositV1(subaccount)`, which calls `DirectDepositV1::creditDeposit()`.
4. Inside `creditDeposit()`, `token.approve(address(endpoint), balance)` is called on USDT.
5. USDT's `approve` returns no data; Solidity's ABI decoder expects a `bool` and reverts.
6. The deposit fails; USDT remains stuck in the DDA contract.

        for (uint256 i = 0; i < productIds.length; i++) {

            address tokenAddr = spotEngine.getToken(productId);

                    productId,
                    uint128(balance),
