Audit Report

## Title
`OracleValueStopLossExtension` Uses Arithmetic Mean Mid-Price While Pool Uses Geometric Mean, Causing Stop-Loss to Systematically Underprotect LPs - (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes `midPriceX64` as the arithmetic mean of `bidPriceX64` and `askPriceX64`, while `MetricOmmPool.swap` derives the same quantity via `SwapMath.midAndSpreadFeeX64FromBidAsk` as the geometric mean of the same oracle snapshot. By AM-GM inequality, the arithmetic mean is always ≥ the geometric mean, so `metricT1` (the token1-denominated per-share value) is systematically overstated every swap that touches the extension. This inflates the stored high-watermark for token1, causing the stop-loss to require a larger actual drawdown than the admin-configured `drawdownE6` before it fires on `!zeroForOne` swaps — the direction that drains token1 from the pool.

## Finding Description
**Root cause — formula mismatch on the same oracle snapshot:**

In `MetricOmmPool.swap` (`metric-core/contracts/MetricOmmPool.sol`, L228 and L242–243), the pool fetches `bidPriceX64`/`askPriceX64` once and passes them to both the swap math and the extension's `afterSwap` hook:
```solidity
(uint128 bidPriceX64, uint128 askPriceX64) = _getBidAndAskPriceX64();
// ... passed to _beforeSwap and _afterSwap with the same values
(uint256 midPriceX64, uint256 baseFeeX64) =
    SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
```

`SwapMath.midAndSpreadFeeX64FromBidAsk` (`metric-core/contracts/libraries/SwapMath.sol`, L70):
```solidity
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);  // geometric mean
```

`OracleValueStopLossExtension._afterSwapOracleStopLoss` (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`, L218):
```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;  // arithmetic mean
```

This inflated `midPriceX64` feeds into `_metrics` (L255):
```solidity
metricT1 = _clampMetric(Math.mulDiv(Math.mulDiv(uint256(t0), midPriceX64, Q64), METRIC_SCALE, shares) + t1ps);
```

The token0 component of `metricT1` is `t0 * midPriceX64 / Q64`. Because `arithmeticMid > geometricMid` whenever `bidPriceX64 ≠ askPriceX64`, this component is always larger than the value the pool itself assigns to the same token0 balance.

`_applyWatermark` (L333) ratchets the stored watermark up to the current metric whenever `metric >= hwm`. Every swap that sets a new `metricT1` high stores an inflated watermark. The stop-loss for `!zeroForOne` swaps (L275–277) fires only when:
```solidity
(uint256 hwm1, bool breach1) = _applyWatermark(metricT1, _decayed(hwmS.token1, decayRate, dt), floorMultiplier);
if (breach1 && !zeroForOne) {
    revert OracleStopLossTriggered(binIdx, false, metricT1, (hwm1 * floorMultiplier) / E6);
}
```

Because `hwm1` is inflated, the floor `(hwm1 * floorMultiplier) / E6` is also inflated, meaning the actual drawdown required to breach it is larger than `drawdownE6`. The guard is permanently less protective than configured for the direction that drains token1.

**Exploit path:**
1. Any pool deploys `OracleValueStopLossExtension` with non-zero `drawdownE6` and a non-zero oracle spread.
2. Ordinary trading activity causes swaps that set new `metricT1` highs — each one stores an inflated watermark.
3. An attacker (or market movement) drains token1 via `!zeroForOne` swaps.
4. The stop-loss fires at a larger actual drawdown than the admin configured, allowing more LP principal to be extracted before the revert.

**Existing guards are insufficient:** The `_validateDrawdown` check only enforces `drawdownE6 <= E6`; it does not compensate for the systematic watermark inflation. The timelock on watermark resets means the admin cannot quickly correct an inflated watermark after it is set.

## Impact Explanation
The stop-loss is the primary on-chain mechanism protecting LP principal from rapid value extraction. When the watermark is inflated by the AM-GM gap, the effective protection floor is shifted downward by that gap relative to the configured `drawdownE6`. For a 10% oracle spread, the gap is ~0.125%; for a 20% spread, ~0.503%. On a $1M pool with a 5% drawdown floor and 10% spread, this represents ~$1,250 of unprotected LP principal per trigger event. The bias is permanent and cumulative — every swap that sets a new watermark high embeds the inflated value, and the gap never closes without an admin-initiated watermark reset (which is itself timelocked). This constitutes a direct, measurable loss of LP principal above the admin-configured protection threshold, meeting the Medium/High impact threshold for a broken protective invariant.

## Likelihood Explanation
Every pool that deploys `OracleValueStopLossExtension` with a non-zero `drawdownE6` and a non-zero oracle spread is affected. No special attacker capability is required — ordinary trading activity continuously inflates the watermarks. The extension is a reference implementation explicitly in scope. The condition `bidPriceX64 ≠ askPriceX64` (non-zero spread) is the normal operating state for any oracle-quoted pool.

## Recommendation
Replace the arithmetic mean with the geometric mean in `_afterSwapOracleStopLoss`, matching the formula used by `SwapMath.midAndSpreadFeeX64FromBidAsk`:

```solidity
// Before (arithmetic mean — incorrect):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (geometric mean — matches SwapMath.midAndSpreadFeeX64FromBidAsk):
uint256 midPriceX64 = Math.sqrt(uint256(bidPriceX64) * uint256(askPriceX64));
```

This ensures the value metrics used by the stop-loss are computed at the same price the pool uses for swap settlement, eliminating the systematic watermark inflation.

## Proof of Concept
**Setup:** Pool with `bidPriceX64 = 0.9 * M`, `askPriceX64 = 1.1 * M` (10% spread, M = some mid). Admin sets `drawdownE6 = 50_000` (5% floor). Bin holds `t0 = 1000`, `t1 = 0`, `shares = 1000`.

**Step 1 — First swap sets watermark:**
- Arithmetic mid = `(0.9M + 1.1M) / 2 = M`
- Geometric mid = `sqrt(0.9M * 1.1M) = sqrt(0.99) * M ≈ 0.99875M`
- `metricT1` (arithmetic) = `1000 * M / Q64 * SCALE / 1000 = M * SCALE / Q64`
- `metricT1` (geometric) = `1000 * 0.99875M / Q64 * SCALE / 1000 ≈ 0.99875 * M * SCALE / Q64`
- Watermark stored: `hwm1 = M * SCALE / Q64` (inflated by ~0.125%)

**Step 2 — Token1 drain via `!zeroForOne` swaps:**
- After drain, `metricT1` falls to `0.95 * M * SCALE / Q64` (5% actual drawdown)
- Floor = `hwm1 * 950000 / 1e6 = 0.95 * M * SCALE / Q64`
- Check: `metricT1 (0.95x) < floor (0.95x)` → **not breached** (equal, not strictly less)
- With geometric mean: `hwm1 = 0.99875x`, floor = `0.94881x`, and `0.95 > 0.94881` — also not breached at exactly 5%, but the effective protection threshold is shifted by the full 0.125% gap, meaning the attacker can extract an additional ~0.125% of pool value before the stop-loss fires.

A Foundry test can confirm this by deploying the extension with a 10% spread oracle, performing a swap to set the watermark, then performing `!zeroForOne` swaps to drain token1 to exactly the configured floor and observing that `OracleStopLossTriggered` is not emitted when it should be.
