### Title
O(n²) Iteration in `traverse_path_with_vec` During Back-Reference Deserialization - (File: src/serde/de_br.rs)

### Summary
The `traverse_path_with_vec` function in `src/serde/de_br.rs` always iterates from index 0 to `arg_index` when materializing the parse-stack Vec as a linked list, even when all entries are already cached. A crafted compressed CLVM program with N back-references each pointing to the whole parse stack (path `0x01`) causes O(N²) total CPU iterations during deserialization, with no CLVM cost limit applying to this phase.

### Finding Description

`node_from_stream_backrefs` maintains a `Vec<(NodePtr, Option<NodePtr>)>` called `values` as the parse stack. Each parsed token (atom, back-reference, or cons result) is pushed onto it. [1](#0-0) 

When a back-reference byte `0xfe` is encountered, `traverse_path_with_vec` is called with the current `values` slice and the parsed path bytes. [2](#0-1) 

For path `0x01` (`0b1`), the bit-traversal loop in `traverse_path_with_vec` does not execute (the sentinel bit is the only bit), leaving `parsing_sexp = false` and `arg_index = args.len() - 1`. The function then falls into the tail loop: [3](#0-2) 

This loop iterates from index 0 to `arg_index` (= `values.len() - 1`). A per-entry cache (`x.1`) avoids redundant `new_pair` allocations, but **the iteration itself is not eliminated**. Every call with `arg_index = k` performs exactly k+1 loop iterations regardless of cache state.

Because each back-reference pushes its result onto `values`, the k-th back-reference (0-indexed) finds `values` with k+1 entries and iterates k+1 times. For N back-references:

```
Total iterations = 1 + 2 + 3 + ... + (N+1) = O(N²)
```

The pair limit (`MAX_NUM_PAIRS = 62_500_000`) bounds N, but even at N = 10,000 (20 KB of input: `0x01` atom followed by 10,000 × `0xfe 0x01` bytes) the iteration count reaches ~5 × 10⁷. [4](#0-3) 

### Impact Explanation

Deserialization via `node_from_bytes_backrefs` runs **before** any CLVM execution cost limit is applied. An attacker submitting a compressed CLVM program to a Chia full node triggers this path unconditionally. The O(N²) iteration is pure CPU spin with no cost gate:

- N = 10,000 → ~5 × 10⁷ iterations (20 KB input)
- N = 100,000 → ~5 × 10⁹ iterations (200 KB input)
- N = 500,000 → ~1.25 × 10¹¹ iterations (1 MB input)

This is a deserialization-phase denial-of-service. The corrupted result is not a wrong hash or wrong NodePtr — it is unbounded CPU consumption before any execution guard fires, directly analogous to the `_removeFromQueue` pattern of rewriting all unchanged elements on every operation.

### Likelihood Explanation

The attack requires only crafting a byte string: one atom byte followed by N repetitions of `0xfe 0x01`. No cryptographic material, no valid puzzle solution, and no special privilege is needed. The compressed serialization format is accepted on the Chia network since the hard fork at block 5,496,000. [5](#0-4) 

### Recommendation

Replace the O(N)-per-call tail loop with an O(1) incremental update. Maintain a single "whole-stack list" node that is updated each time a new item is pushed onto `values`: prepend the new item to the existing cached list. A back-reference with path `0x01` then returns this pre-built node in O(1). This mirrors the recommendation in the original report: use a structure with two monotonically increasing indexes (or an incrementally maintained head pointer) so that unchanged elements are never reprocessed.

### Proof of Concept

Construct the following byte sequence and pass it to `node_from_bytes_backrefs`:

```
01                  # atom: value 1
fe 01               # back-reference: path 0x01 (whole stack) — iteration 1
fe 01               # back-reference: path 0x01 — iteration 2
fe 01               # back-reference: path 0x01 — iteration 3
...                 # repeat N times
```

Each `fe 01` pair causes `traverse_path_with_vec` to iterate through all k+1 current `values` entries. With N = 100,000 repetitions (200 KB input), the total iteration count exceeds 5 × 10⁹, consuming several seconds of CPU on a modern machine with no CLVM cost limit applied.
