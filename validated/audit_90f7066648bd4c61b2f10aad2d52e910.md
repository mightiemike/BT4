### Title
Stale `proposed_split` in child `ChunkExtra` after shard split causes `InvalidChunkHeaderShardSplit` on first child-shard chunk — (File: chain/chain/src/resharding/manager.rs)

---

### Summary

`process_memtrie_resharding_storage_update` creates each child shard's initial `ChunkExtra` by cloning the parent shard's `ChunkExtra` and updating only `state_root` and `congestion_info`. The `proposed_split` field — a derived, trie-memory-based split proposal — is never reset to `None`. The parent's last chunk always carries a non-`None` `proposed_split` (that is precisely what triggered the resharding). Every child shard therefore starts epoch N+2 with a stale, parent-inherited `proposed_split`. The first chunk header produced for the child shard carries `proposed_split = None` (the child is new, not near its epoch end). `validate_chunk_with_chunk_extra_and_receipts_root` compares the two and returns `InvalidChunkHeaderShardSplit`, permanently rejecting the child shard's first chunk and halting it.

---

### Finding Description

In `process_memtrie_resharding_storage_update`:

```rust
// TODO(resharding): set all fields of `ChunkExtra`. Consider stronger
// typing. Clarify where it should happen when `State` and
// `FlatState` update is implemented.
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
``` [1](#0-0) 

The parent shard's `ChunkExtra` at the epoch-N boundary block has `proposed_split = Some(TrieSplit{...})` because `compute_proposed_split` ran during chunk application and found the shard large enough to split — that is the exact signal that caused the resharding to be scheduled. The child `ChunkExtra` inherits this value verbatim.

`ChunkExtra::proposed_split()` for a V5 record returns the inner `Option<TrieSplit>` directly:

```rust
pub fn proposed_split(&self) -> Option<&TrieSplit> {
    match self {
        Self::V1(_) | Self::V2(_) | Self::V3(_) | Self::V4(_) => None,
        ChunkExtra::V5(v5) => v5.proposed_split.as_ref(),
    }
}
``` [2](#0-1) 

When epoch N+2 begins and the first chunk for the child shard is produced, `compute_proposed_split` returns `None`: the child shard is brand-new and `is_next_block_possibly_last_in_epoch` is false for block 1 of the epoch. [3](#0-2) 

The chunk header therefore carries `proposed_split = None`. Validation then fires:

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    DYNAMIC_RESHARDING_VALIDATION_FAILURES.with_label_values(&["chunk_header"]).inc();
    return Err(Error::InvalidChunkHeaderShardSplit(...));
}
``` [4](#0-3) 

`prev_chunk_extra.proposed_split()` is `Some(TrieSplit{...})` (stale from parent); `chunk_header.proposed_split()` is `None`. They are not equal → `InvalidChunkHeaderShardSplit` → the chunk is rejected.

The codebase itself documents this gap in two places:

1. The inline TODO at the exact site: `// TODO(resharding): set all fields of ChunkExtra.`
2. The architecture doc explicitly lists it as open item #10: *"The resharding manager doesn't set all `ChunkExtra` fields (notably the new `proposed_split` field)."*



The doc also notes that `min_epochs_between_resharding > 0` is required *because* of this inheritance, but the cooldown only prevents the child from scheduling a further split — it does not clear the stale value that causes the first-chunk rejection. [5](#0-4) 

---

### Impact Explanation

Every dynamic resharding event produces two child shards whose initial `ChunkExtra` carries a non-`None` `proposed_split`. The very first chunk header for each child shard will be rejected with `InvalidChunkHeaderShardSplit`. No chunk producer can ever produce a valid first chunk for the child shard; the child shard is permanently halted from the moment it comes into existence at epoch N+2. All accounts whose state was migrated to the child shard become inaccessible. This is a **Critical** protocol-level chain halt scoped to every child shard created by dynamic resharding.

---

### Likelihood Explanation

The condition is deterministic and unavoidable: a resharding is triggered precisely because the parent shard's last chunk computed a non-`None` `proposed_split`. That value is always cloned into the child `ChunkExtra`. The bug fires on 100% of dynamic resharding events once `ProtocolFeature::DynamicResharding` is enabled. No attacker action is required; normal network growth that crosses the memory threshold is sufficient.

---

### Recommendation

After cloning the parent `ChunkExtra`, explicitly reset `proposed_split` to `None` for each child:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// Child shards are new; they carry no pending split proposal.
if let ChunkExtra::V5(ref mut v5) = child_chunk_extra {
    v5.proposed_split = None;
}
```

This mirrors the fix pattern from the external report (`change_edge_px` after `perp_withdraw`): the derived field stored in the "tree" (here `ChunkExtra`, the per-shard derived state record) must be updated whenever the underlying state changes (here, a shard split that creates a new identity).

---

### Proof of Concept

1. Enable `ProtocolFeature::DynamicResharding` (protocol version ≥ its activation version).
2. Drive trie memory usage on one shard above `memory_split_threshold` in `DynamicReshardingConfig`.
3. At the last block of epoch N, `compute_proposed_split` returns `Some(TrieSplit{boundary, left, right})` for the overloaded shard; this is stored in the parent's `ChunkExtra` and propagated into the block header's `shard_split`.
4. `finalize_epoch` derives a new `ShardLayoutV3` for epoch N+2 with two child shards.
5. `process_memtrie_resharding_storage_update` clones the parent `ChunkExtra` into both child `ChunkExtra` records — `proposed_split` is `Some(TrieSplit{...})` in both.
6. Epoch N+2, block 1: chunk producers for both child shards call `compute_proposed_split`; `is_next_block_possibly_last_in_epoch` returns `false` → `proposed_split = None` → chunk headers carry `None`.
7. `validate_chunk_with_chunk_extra_and_receipts_root` compares child `ChunkExtra.proposed_split` (`Some(...)`) with chunk header `proposed_split` (`None`) → mismatch → `InvalidChunkHeaderShardSplit`.
8. Both child shards' first chunks are rejected; neither shard can ever produce a valid chunk. All state on both child shards is permanently inaccessible.
