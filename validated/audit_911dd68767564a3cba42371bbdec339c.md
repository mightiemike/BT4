### Title
Deprecated `verify_transaction_inclusion` Remains Callable by Any Unprivileged Caller, Enabling 64-Byte Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The `BtcLightClient` contract exposes a deprecated proof-verification function (`verify_transaction_inclusion`, v1) that remains callable by any unprivileged NEAR account with no access control. It omits the coinbase Merkle proof validation that was introduced in v2 specifically to block the 64-byte transaction Merkle proof forgery. Because no enforcement prevents callers from invoking v1 instead of v2, the secure path can be silently bypassed — directly analogous to how a missing Android Network Security Configuration leaves cleartext HTTP open alongside HTTPS.

---

### Finding Description

The contract ships two proof-verification entry points:

- `verify_transaction_inclusion` (v1, `#[deprecated since = "0.5.0"]`): computes a Merkle root from the caller-supplied `tx_id` and `merkle_proof` and compares it to the stored block Merkle root. No coinbase proof is required.
- `verify_transaction_inclusion_v2`: additionally requires a `coinbase_tx_id` and `coinbase_merkle_proof` that must independently reconstruct the same Merkle root, closing the 64-byte internal-node forgery path.

Despite the deprecation annotation, v1 is a fully public `#[pause]`-gated function with no role restriction. The contract's own inline warning acknowledges the risk:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [1](#0-0) 

The `compute_root_from_merkle_proof` implementation in `merkle-tools` is position-driven and purely hash-based; it has no mechanism to distinguish a leaf (real transaction) from an internal node: [2](#0-1) 

v1 passes the caller-supplied `tx_id` directly into this function and returns the boolean comparison result: [3](#0-2) 

v2 closes the gap by first asserting that a coinbase proof independently reconstructs the same Merkle root before delegating to v1: [4](#0-3) 

Because v1 is never removed, blocked, or redirected to v2, any caller can choose the insecure path at will. There is no contract-level policy — no role gate, no `require!`, no removal — that enforces use of the secure function.

---

### Impact Explanation

An attacker who supplies a crafted 64-byte value as `tx_id` — one that is actually an internal Merkle tree node — together with a matching `merkle_proof` can cause `verify_transaction_inclusion` to return `true` for a Bitcoin transaction that does not exist in the claimed block. Any downstream NEAR contract that consumes this `true` result (e.g., to release funds, mint tokens, or record a cross-chain event) will accept a forged Bitcoin transaction inclusion proof. This breaks the core security invariant of the light client: that a `true` result certifies a real, confirmed Bitcoin transaction.

---

### Likelihood Explanation

Medium. The 64-byte transaction Merkle proof forgery is publicly documented (https://www.bitmex.com/blog/64-Byte-Transactions) and the deprecated v1 function is visible in the public contract ABI. No privileged access, leaked keys, or social engineering is required. Any NEAR account can call `verify_transaction_inclusion` directly.

---

### Recommendation

Remove or permanently gate the deprecated v1 function. Concrete options:

1. **Delete `verify_transaction_inclusion`** and require all callers to migrate to v2.
2. **Redirect v1 to panic unconditionally**, forcing callers to use v2.
3. **Add a role gate** (e.g., `Role::DAO` only) so unprivileged callers cannot invoke the insecure path.

The analogy to the Android Network Security Configuration fix is exact: just as `cleartextTrafficPermitted="false"` closes the insecure HTTP path system-wide, removing or blocking v1 closes the insecure proof-verification path contract-wide.

---

### Proof of Concept

1. Identify any confirmed mainchain block `B` with known Merkle root `R` stored in the contract.
2. Construct a 64-byte value `N` such that `double_sha256(N || sibling)` or `double_sha256(sibling || N)` equals `R` for some chosen `sibling` — i.e., `N` is a valid internal Merkle node at depth 1.
3. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the forged internal node, presented as a transaction hash)
   - `tx_block_blockhash = B`
   - `tx_index = 0` (or whichever position matches the crafted proof)
   - `merkle_proof = [sibling]`
   - `confirmations = 1`
4. `compute_root_from_merkle_proof(N, 0, [sibling])` computes `double_sha256(N || sibling) == R`.
5. The function returns `true`.
6. A downstream contract that calls `verify_transaction_inclusion` and acts on the `true` result accepts the forged proof, believing a non-existent Bitcoin transaction was confirmed in block `B`. [5](#0-4) [6](#0-5)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
