### Title
Detached `WithdrawPool` Retains Token Balance and Accepts Replayed Withdrawal Signatures — (File: `core/contracts/BaseWithdrawPool.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

When an admin replaces the active `WithdrawPool` via `setWithdrawPool`, the old pool is detached from the clearinghouse but retains any accumulated ERC-20 token balance. Because `submitFastWithdrawal` is a public function with no access control and the old pool's `markedIdxs` and `minIdx` are frozen at the moment of detachment, an attacker can replay any valid sequencer-signed withdrawal transaction (with `idx > minIdx_old`) against the old pool and drain its token balance — a double-spend of the same withdrawal.

---

### Finding Description

**How the pool accumulates tokens:**

`Clearinghouse.handleWithdrawTransfer` pushes ERC-20 tokens directly into the active `withdrawPool` address and then calls `submitWithdrawal` on it: [1](#0-0) 

When a fast withdrawal is processed first via `submitFastWithdrawal`, `markedIdxs[idx]` is set to `true` on the pool. When the clearinghouse later calls `submitWithdrawal` for the same `idx`, it finds the flag set and returns early — but the tokens transferred from the clearinghouse **remain in the pool**, replenishing the liquidity provider's pre-funded balance: [2](#0-1) 

This means the pool accumulates a real ERC-20 balance over time.

**The detachment:**

`setWithdrawPool` atomically swaps the pool pointer in the clearinghouse with no requirement to drain the old pool first: [3](#0-2) 

After this call, the clearinghouse routes all future `handleWithdrawTransfer` calls to the new pool. The old pool's `minIdx` and `markedIdxs` are permanently frozen at their last values.

**The exploit — `submitFastWithdrawal` on the old pool:**

`submitFastWithdrawal` is a public, permissionless function. Its only guards are:

1. `!markedIdxs[idx]` — not already submitted on *this* pool
2. `idx > minIdx` — greater than the frozen `minIdx` on *this* pool
3. `v.requireValidTxSignatures(transaction, idx, signatures)` — valid sequencer BLS signatures [4](#0-3) 

After pool replacement:

- The new pool processes withdrawal `idx = N+1` via `submitFastWithdrawal` (or `submitWithdrawal`). The signed transaction data and BLS signatures are public on-chain.
- The old pool has `minIdx = N` and `markedIdxs[N+1] = false`.
- An attacker calls `submitFastWithdrawal` on the **old** pool with the same `(transaction, idx=N+1, signatures)`.
- All three checks pass: `markedIdxs[N+1]` is false on the old pool, `N+1 > N`, and the signatures are valid (the verifier is shared and does not bind signatures to a specific pool address).
- The old pool transfers tokens to the withdrawal recipient from its accumulated balance.

The same withdrawal is paid out twice: once from the new pool and once from the old pool.

**No enforcement to drain the old pool:**

`removeWithdrawPoolLiquidity` exists but is admin-only and operates on the *current* pool pointer — after `setWithdrawPool` is called, it points to the new pool, leaving the old pool's balance unreachable by the admin through this path: [5](#0-4) 

---

### Impact Explanation

An attacker can drain the entire ERC-20 token balance of the old `WithdrawPool` by replaying any sequencer-signed withdrawal transaction with `idx > minIdx_old`. Each replayed withdrawal transfers real tokens to the withdrawal recipient (or to the attacker if they are the `sendTo` address, or they pay the fee to redirect). The loss is bounded by the token balance held in the old pool at the time of detachment, which can be substantial if the pool was actively used for fast withdrawals. This is a direct, irreversible asset loss.

---

### Likelihood Explanation

Pool replacement is an admin action, but it is a realistic operational event (e.g., upgrading pool logic, migrating liquidity). The attack requires no special privilege — only the ability to observe on-chain calldata from `submitFastWithdrawal` calls on the new pool and replay them. BLS signatures are fully public once submitted. The window of vulnerability persists indefinitely after pool replacement as long as the old pool holds a balance.

---

### Recommendation

1. **Drain before detach:** Require that the old pool's token balance is zero (or explicitly drain it) before `setWithdrawPool` accepts a new address.
2. **Invalidate the old pool:** Add a `pause()` or `deactivate()` function to `BaseWithdrawPool` that blocks `submitFastWithdrawal` after detachment, callable by the clearinghouse or owner.
3. **Bind signatures to pool address:** Include the pool contract address in the verifier's signed digest so that signatures issued for one pool cannot be replayed on another.

---

### Proof of Concept

```
State before attack:
  oldPool.minIdx = 100
  oldPool.markedIdxs[101] = false
  oldPool.tokenBalance(USDC) = 500,000e6  (accumulated from clearinghouse replenishments)

Admin action:
  ContractOwner.setWithdrawPool(newPool)
  → clearinghouse.withdrawPool = newPool

Normal operation on new pool:
  user calls newPool.submitFastWithdrawal(101, withdrawTx, sigs)
  → newPool.markedIdxs[101] = true
  → newPool transfers 10,000 USDC to user
  (sigs and withdrawTx are now public on-chain)

Attacker action:
  attacker calls oldPool.submitFastWithdrawal(101, withdrawTx, sigs)
  → require(!oldPool.markedIdxs[101])  ✓  (false on old pool)
  → require(101 > oldPool.minIdx=100)  ✓
  → verifier.requireValidTxSignatures(withdrawTx, 101, sigs)  ✓  (same verifier, no pool binding)
  → oldPool transfers 10,000 USDC to user (or attacker pays fee to redirect sendTo)

Result: 10,000 USDC double-spent; attacker repeats for all idx > 100 until oldPool is drained.
```

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }

    function submitWithdrawal(
        IERC20Base token,
        address sendTo,
        uint128 amount,
        uint64 idx
    ) public {
        require(msg.sender == clearinghouse);

        if (markedIdxs[idx]) {
            return;
        }
        markedIdxs[idx] = true;
        // set minIdx to most recent withdrawal submitted by sequencer
        minIdx = idx;

        handleWithdrawTransfer(token, sendTo, amount);
    }
