Audit Report

## Title
`SwapAllowlistExtension.beforeSwap()` Checks Router Address as Swapper, Enabling Full Per-User Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap()` gates swaps by checking the `sender` parameter, which the pool sets to `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `sender` is the router's address, not the end user. If the pool admin allowlists the router (the necessary step to enable router-based swaps for permitted users), every unprivileged address can bypass the per-user restriction by calling the router, rendering the allowlist ineffective.

## Finding Description

**Root cause — the guard checks the wrong address:**

`SwapAllowlistExtension.beforeSwap()` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the first argument forwarded by the pool:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

**What the pool passes as `sender`:**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`, so `sender` in the extension is whoever called `pool.swap()`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← becomes `sender` in the extension
```

`ExtensionCalling._beforeSwap()` forwards this value unchanged to the extension.

**What the router passes to the pool:**

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly, making `msg.sender` inside `pool.swap()` equal to the router's address:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The actual end user (`msg.sender` of `exactInputSingle`) is stored only in transient callback context for payment purposes — it is never forwarded to the pool as `sender`.

**Exploit flow:**

1. Pool deployed with `SwapAllowlistExtension` in `beforeSwap` order.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is permitted.
3. Admin calls `setAllowedToSwap(pool, router, true)` — router allowlisted so Alice can use it.
4. Bob (not allowlisted) calls `router.exactInputSingle({pool: restrictedPool, ...})`.
5. Pool receives `msg.sender = router`, calls `_beforeSwap(router, ...)`.
6. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
7. Bob's swap executes in the restricted pool.

**Contrast with `DepositAllowlistExtension`**, which correctly checks `owner` (the position owner explicitly passed by the caller and forwarded unchanged by the pool), not `sender`. The deposit extension is not affected because `owner` is the actual user identity, not the intermediary caller.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` (e.g., KYC-gated, institutional-only, or whitelist-only pool) and with the router allowlisted loses its per-user access control entirely for router-based swaps. Any unprivileged address can call `router.exactInputSingle()` and trade in the restricted pool. This is an admin-boundary break via an unprivileged path: the pool admin's intent to restrict swaps to specific users is silently nullified by the structural mismatch between the allowlist key (intended: end user) and the value checked (actual: router address). The only configuration that makes the router usable for permitted users simultaneously removes the per-user gate for all users.

## Likelihood Explanation
The scenario is realistic and likely. A pool admin deploying `SwapAllowlistExtension` who wants permitted users to access the standard router will naturally allowlist the router address — this is the only way to make router-based swaps work for any user. The bypass requires no special privileges, no flash loans, no unusual token behavior, and no contract deployment — only a standard `exactInputSingle` call. The same issue applies to `exactInput`, `exactOutputSingle`, and `exactOutput` router entry points.

## Recommendation
The extension must identify the actual end user, not the intermediary. The most practical fix is to require that any router wishing to be allowlisted encodes the actual swapper in `extensionData`, and have the extension decode and check that address instead of `sender`. The router already accepts `params.extensionData` and passes it through to the pool unchanged, so a trusted encoding convention can be established. Alternatively, a dedicated "on-behalf-of" field could be added to the swap call that the pool forwards to extensions, distinct from the callback payer.

## Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension as extension1 in beforeSwap order.
2. extension.setAllowedToSwap(pool, alice, true)
3. extension.setAllowedToSwap(pool, router, true)  // necessary for Alice to use router
4. Bob (EOA, not allowlisted) calls:
       router.exactInputSingle(ExactInputSingleParams({
           pool: restrictedPool,
           recipient: bob,
           tokenIn: token0,
           amountIn: 1e18,
           ...
       }))
5. pool.swap() receives msg.sender = router
6. _beforeSwap(router, ...) is called
7. SwapAllowlistExtension checks allowedSwapper[pool][router] == true → no revert
8. Bob's swap executes successfully in the restricted pool
```

Foundry test: deploy the pool with the extension, configure allowlist as above, impersonate an unlisted address, call `router.exactInputSingle`, assert the swap succeeds and no `NotAllowedToSwap` revert occurs.
