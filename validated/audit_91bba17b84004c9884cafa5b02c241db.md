### Title
Stale Cumulative Interest Multipliers in `getHealth()` Enable Liquidation Bypass and Over-Withdrawal — (File: `core/contracts/SpotEngineState.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.getHealth()` computes subaccount health using spot balances derived from `states[productId].cumulativeBorrowsMultiplierX18` and `states[productId].cumulativeDepositsMultiplierX18`. These multipliers are only updated when `SpotEngineState.updateStates()` is called by the endpoint (sequencer). Between sequencer updates, the stored multipliers are stale, causing borrow balances to be understated. Any health check that executes before a fresh `updateStates()` call will overestimate subaccount health, blocking legitimate liquidations and permitting over-withdrawal.

---

### Finding Description

In `SpotEngineState.sol`, the function `_updateState()` is the protocol's analog to `accrueInterest()`. It computes the current utilization ratio and advances `cumulativeBorrowsMultiplierX18` and `cumulativeDepositsMultiplierX18` by the elapsed time `dt`: [1](#0-0) 

These multipliers are stored in `states[productId]` and are only written back to storage when `_setState()` is called at the end of `_updateState()`. The function is gated to `onlyEndpoint`, meaning it is invoked exclusively by the sequencer: [2](#0-1) 

When `getStateAndBalance()` is called for a health check, it reads `states[productId]` directly from storage — the stale, not-yet-accrued values: [3](#0-2) 

`balanceNormalizedToBalance()` then converts a user's normalized borrow balance to an actual balance using the stale multiplier: [4](#0-3) 

Because `cumulativeBorrowsMultiplierX18` has not been advanced to the current time, the actual borrow balance (`amountNormalized × staleBorrowMultiplier`) is lower than the true balance (`amountNormalized × currentBorrowMultiplier`). This understated liability flows directly into `_calculateProductHealth()`: [5](#0-4) 

Which is aggregated in `getHealthContribution()` and ultimately in `Clearinghouse.getHealth()`: [6](#0-5) 

---

### Impact Explanation

Two concrete asset-level impacts:

**1. Liquidation Bypass.** `ClearinghouseLiq.isUnderMaintenance()` calls `getHealth()` with stale multipliers: [7](#0-6) 

`liquidateSubaccountImpl()` gates the entire liquidation on this check: [8](#0-7) 

A subaccount whose true maintenance health is negative (liquidatable) will appear healthy if the borrow multiplier is stale. The liquidator's call reverts with `ERR_NOT_LIQUIDATABLE`, leaving an insolvent position open and exposing the protocol to bad debt.

**2. Over-Withdrawal.** `withdrawCollateral()` enforces health only via `getHealth()`: [9](#0-8) 

With a stale (lower) borrow balance, a borrower's health appears higher than reality. The user can withdraw collateral that would make their true health negative, leaving an undercollateralized position.

---

### Likelihood Explanation

The sequencer batches transactions and calls `updateStates()` periodically, not necessarily before every health-sensitive operation. The contracts themselves impose no ordering constraint: nothing in `getHealth()`, `withdrawCollateral()`, or `liquidateSubaccountImpl()` calls `updateStates()` first. In high-utilization periods or when the sequencer batch interval is long, the gap between the last `updateStates()` and a health check can be significant. A borrower who monitors sequencer batch timing can submit a withdrawal request immediately after a batch (before the next `updateStates()`) to exploit the stale state window. The liquidation bypass is passive — it occurs automatically whenever a liquidation is attempted in a batch that does not include a preceding `updateStates()` call.

---

### Recommendation

Before any health-sensitive computation, apply the current time delta to the in-memory state before reading balances. Specifically, `getHealth()` (or the underlying `getStateAndBalance()` calls it depends on) should compute the up-to-date multipliers using the current timestamp and the stored state, without requiring a prior `updateStates()` call. A read-only version of `_updateState()` that returns the projected multipliers without writing to storage would allow `balanceNormalizedToBalance()` to use fresh values in all health checks, mirroring the recommendation in the original report to call `accrueInterest()` before rate queries.

---

### Proof of Concept

1. User A deposits 1000 USDC and borrows 800 USDC worth of a spot asset (high utilization, e.g., 80%).
2. The sequencer calls `updateStates()` at time T, advancing `cumulativeBorrowsMultiplierX18`.
3. 24 hours pass. At the current borrow rate, the true borrow balance has grown to ~800.8 USDC, pushing maintenance health below zero.
4. At time T+24h, before the sequencer's next `updateStates()` call, a liquidator submits a liquidation transaction.
5. The sequencer includes the liquidation in a batch without a preceding `updateStates()`.
6. `liquidateSubaccountImpl()` calls `isUnderMaintenance()` → `getHealth()` → `getStateAndBalance()` reads the stale `cumulativeBorrowsMultiplierX18` from time T.
7. The borrow balance is computed as 800 USDC (stale), not 800.8 USDC (true). Health appears non-negative.
8. The call reverts with `ERR_NOT_LIQUIDATABLE`. The insolvent position remains open.
9. If the borrow rate is high enough and the sequencer delay is long enough, the position accumulates bad debt that must be socialized across all depositors via `socializeSubaccount()`. [1](#0-0) [4](#0-3) [6](#0-5) [8](#0-7)
