Audit Report

## Title
`SwapAllowlistExtension.beforeSwap()` Checks Router Address as Swapper, Enabling Full Per-User Allowlist Bypass via Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap()` gates swaps by checking the `sender` parameter, which the pool sets to `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `sender` becomes the router's address, not the end user's. Allowlisting the router — the only way to enable router-based swaps for permitted users — silently removes per-user access control, allowing any unprivileged address to swap in the restricted pool via the router.

## Finding Description
`SwapAllowlistExtension.beforeSwap()` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the first argument forwarded by the pool:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmPool.swap()` passes `msg.sender` as `sender` to `_beforeSwap()`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // whoever called pool.swap()
    ...
```

`ExtensionCalling._beforeSwap()` forwards this value unchanged to every extension in the `BEFORE_SWAP_ORDER`. When `MetricOmmSimpleRouter.exactInputSingle()` is used, it calls `pool.swap()` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

So `msg.sender` inside `pool.swap()` is the router, not the end user. The extension receives `sender = router`. The only configuration that makes the router usable — allowlisting the router address — causes `allowedSwapper[pool][router] == true` to pass for every caller regardless of identity.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity()` correctly checks `owner` (the position owner, explicitly passed by the caller and forwarded unchanged), not `sender` (the pool caller), so it is not affected by this pattern.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` for KYC, institutional-only, or whitelist-only access loses its per-user access control entirely for router-based swaps. Any unprivileged address can call `router.exactInputSingle()` and trade in the restricted pool. This is an admin-boundary break via an unprivileged path: the pool admin's intent to restrict swaps to specific addresses is fully defeated without any special privileges, flash loans, or unusual token behavior.

## Likelihood Explanation
The scenario is realistic and likely. A pool admin deploying `SwapAllowlistExtension` who wants permitted users to access the standard router will naturally allowlist the router address — this is the only way to make the router functional for the pool. The bypass requires no special privileges: any address can call `router.exactInputSingle()` with the restricted pool as target. The attack is repeatable and requires no setup beyond the admin's natural configuration step.

## Recommendation
The extension must identify the actual end user, not the intermediary. The most reliable fix is to require that any router wishing to be allowlisted encodes the actual swapper in `extensionData`, and have the extension decode and verify that address instead of `sender`. Concretely:
- The router encodes `msg.sender` into `extensionData` before calling `pool.swap()`.
- `SwapAllowlistExtension.beforeSwap()` decodes the swapper from `extensionData` when `sender` is a known/trusted router, and checks `allowedSwapper[pool][decodedSwapper]`.
- Alternatively, add a dedicated "on-behalf-of" field to the swap call that the pool forwards to extensions, distinct from the callback payer, which the router populates with `msg.sender`.

## Proof of Concept
1. Pool is deployed with `SwapAllowlistExtension` configured in `beforeSwap` order.
2. Admin calls `extension.setAllowedToSwap(pool, alice, true)` — only Alice is permitted.
3. Admin calls `extension.setAllowedToSwap(pool, router, true)` — router is allowlisted so Alice can use it.
4. Bob (not allowlisted) calls `router.exactInputSingle(ExactInputSingleParams({ pool: restrictedPool, ... }))`.
5. Router calls `pool.swap(...)` → `msg.sender` inside `pool.swap()` is the router.
6. Pool calls `_beforeSwap(router, ...)` → extension receives `sender = router`.
7. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
8. Bob's swap executes in the restricted pool despite not being on the allowlist.
