### Title
Linked Signer Can Unauthorized Overwrite Its Own Delegation Without Subaccount Owner Consent - (`core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction handler inside `processTransactionImpl` calls `validateSignedTx` with `allowLinkedSigner = true`. This means the currently-set linked signer — a delegated key, not the subaccount owner — can unilaterally sign a new `LinkSigner` transaction to replace the linked signer with any arbitrary address, without the subaccount owner's knowledge or consent. This is a direct analog to the reported missing opt-in check: a privileged actor (linked signer) can mutate a critical authorization state (the linked signer mapping) on behalf of a user who never authorized that specific action.

---

### Finding Description

In `EndpointTx.sol`, the sequencer-path handler for `LinkSigner` transactions is:

```solidity
// EndpointTx.sol L576-590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true   // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` resolves to `validateCompactSignature`, which passes the current linked signer to `verifier.validateCompactSignature`:

```solidity
// Verifier.sol L306-318
function validateCompactSignature(...) public pure {
    address recovered = ECDSA.recover(digest, signature.r, signature.vs);
    require(
        (recovered != address(0)) &&
            ((recovered == address(uint160(bytes20(sender)))) ||
                (recovered == linkedSigner)),   // linked signer accepted
        ERR_INVALID_SIGNATURE
    );
}
``` [2](#0-1) 

The linked signer is stored in `EndpointStorage.linkedSigners`: [3](#0-2) 

Because the signature check accepts either the subaccount owner **or** the linked signer, the linked signer can craft and sign a valid `LinkSigner` transaction that writes a new address into `linkedSigners[subaccount]`. The subaccount owner's private key is never consulted.

The slow-mode path for `LinkSigner` correctly restricts this to the subaccount owner via `validateSender`:

```solidity
// EndpointTx.sol L232-239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // msg.sender must be owner
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
}
``` [4](#0-3) 

The sequencer path lacks this restriction, creating an asymmetry: the slow-mode path enforces owner-only, but the fast sequencer path does not.

---

### Impact Explanation

A linked signer that becomes malicious (compromised API key, rogue third-party trading service, etc.) can:

1. Sign a `LinkSigner` transaction naming an attacker-controlled address as the new linked signer.
2. The sequencer processes it; `linkedSigners[victim_subaccount]` is overwritten.
3. The attacker's new address now satisfies the `allowLinkedSigner = true` check on **all** subsequent sequencer-path transactions: `WithdrawCollateral`, `WithdrawCollateralV2`, `TransferQuote`, `LiquidateSubaccount`, `MintNlp`, `BurnNlp`, etc.
4. The attacker drains the subaccount via a signed `WithdrawCollateral` or `TransferQuote` transaction.

The subaccount owner can attempt to reset the linked signer via slow mode, but slow-mode transactions have a `SLOW_MODE_TX_DELAY` (hardcoded to three days per the comment at line 377), giving the attacker a multi-day window to act. [5](#0-4) 

**Corrupted state**: `linkedSigners[subaccount]` is set to an attacker-controlled address without owner consent, enabling full signing authority theft and direct asset loss.

---

### Likelihood Explanation

- Linked signers are commonly set to API keys or third-party bots. Any compromise of that key (phishing, key leak, malicious service) immediately enables this escalation.
- No privileged protocol access is required beyond holding the current linked signer key.
- The sequencer processes the transaction without any additional owner-consent check.
- The three-day slow-mode delay means the owner cannot quickly revoke access through the censorship-resistance path.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. Only the subaccount owner's key should be permitted to change the linked signer, consistent with the slow-mode path's `validateSender` enforcement:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only subaccount owner may change linked signer
);
```

---

### Proof of Concept

1. Alice sets `linkedSigners[alice_subaccount] = bot_address` via a normal `LinkSigner` slow-mode transaction.
2. `bot_address` private key is compromised by an attacker.
3. Attacker constructs `SignedLinkSigner { tx: { sender: alice_subaccount, signer: attacker_address, nonce: N }, signature: sig_by_bot }`.
4. Attacker submits this to the sequencer off-chain; sequencer calls `submitTransactionsChecked` → `processTransaction` → `processTransactionImpl`.
5. `validateSignedTx(..., true)` recovers `bot_address` from the signature, which equals `linkedSigners[alice_subaccount]` — check passes.
6. `linkedSigners[alice_subaccount]` is overwritten with `attacker_address`.
7. Attacker immediately signs a `WithdrawCollateralV2` transaction as `attacker_address` (now the linked signer), specifying `sendTo = attacker_wallet`.
8. Sequencer processes it; Alice's collateral is transferred to the attacker.
9. Alice's only recourse is slow-mode reset, but the three-day delay has already allowed the theft. [6](#0-5)
