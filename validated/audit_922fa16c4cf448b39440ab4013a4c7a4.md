### Title
Unprivileged Caller Can Aggressively Prune Mainchain Headers via Unguarded `run_mainchain_gc` - (File: `contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` is a public, state-mutating function on `BtcLightClient` that permanently removes block headers from the contract's three canonical-chain maps. It carries only a `#[pause]` decorator — which governs behavior when the contract is paused, not who may call it when the contract is live. Any unprivileged NEAR account can invoke it with an arbitrarily large `batch_size`, immediately pruning the mainchain down to the `gc_threshold` floor in a single transaction. This destroys headers that are still required for in-flight SPV proof verifications, causing `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` to panic or return `false` for legitimate cross-chain transactions.

---

### Finding Description

`submit_blocks` — the intended and access-controlled entry point — calls `run_mainchain_gc` internally after inserting new headers: [1](#0-0) 

`submit_blocks` is protected by `#[trusted_relayer]`, which enforces a staking/allowlist check on the caller. `run_mainchain_gc` itself, however, is separately exposed as a top-level public method: [2](#0-1) 

The `#[pause(except(roles(Role::UnrestrictedRunGC)))]` attribute only controls whether the function is callable while the contract is in a paused state. It does **not** restrict the caller identity when the contract is running normally. There is no `#[private]`, no `#[trusted_relayer]`, and no `acl_role` check on `run_mainchain_gc`.

Inside the function, the attacker-supplied `batch_size` directly controls how many headers are deleted: [3](#0-2) 

With `batch_size = u64::MAX`, `selected_amount_to_remove` equals `total_amount_to_remove` (the maximum possible), so the entire excess above `gc_threshold` is deleted in one call. For each removed height, `remove_block_header` erases the entry from both `mainchain_header_to_height` and `headers_pool`, and `mainchain_height_to_header` is also cleared: [4](#0-3) 

After the call, `mainchain_initial_blockhash` is advanced to the new oldest surviving block: [5](#0-4) 

Any subsequent call to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a block whose height was pruned will panic at:

```
.unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"))
```

because `mainchain_header_to_height` no longer contains the pruned block hash. [6](#0-5) 

---

### Impact Explanation

The canonical-chain maps (`mainchain_height_to_header`, `mainchain_header_to_height`, `headers_pool`) are the sole source of truth for SPV proof verification. Premature deletion of headers permanently breaks verification for any transaction whose containing block was pruned. Cross-chain bridge contracts or users who submitted a Bitcoin transaction and are waiting for sufficient confirmations before calling `verify_transaction_inclusion` will find the required block gone, causing their verification call to panic. Funds gated behind that verification are effectively locked or lost. The deletion is irreversible — there is no re-insertion path once a header is removed.

---

### Likelihood Explanation

The attack requires no privileged keys, no staking, and no special role. Any NEAR account can call `run_mainchain_gc` directly with a single RPC transaction. The function is publicly documented in the contract ABI. An attacker monitoring the mempool for pending bridge withdrawals can front-run the victim's `verify_transaction_inclusion` call by submitting `run_mainchain_gc(u64::MAX)` first, pruning the target block and causing the victim's verification to fail.

---

### Recommendation

Remove `run_mainchain_gc` from the public ABI or add an explicit caller restriction matching the protection on `submit_blocks`. The simplest fix is to add `#[private]` so only the contract itself can call it (preserving the internal call from `submit_blocks`), or add a role check (e.g., `Role::UnrestrictedRunGC` or `Role::DAO`) using the existing `near-plugins` access-control infrastructure already present in the contract. The `#[pause]` decorator alone is not a substitute for caller authorization.

---

### Proof of Concept

**Preconditions:**
- Contract is deployed and live (not paused).
- The mainchain contains more than `gc_threshold` headers (i.e., `amount_of_headers_we_store > gc_threshold`), which is the normal operating state.
- A legitimate user has submitted a Bitcoin transaction in block at height `H` (within the prunable range) and is waiting for confirmations before calling `verify_transaction_inclusion`.

**Attack steps:**

1. Attacker (any NEAR account, no staking required) calls:
   ```
   run_mainchain_gc(batch_size: 18446744073709551615)  // u64::MAX
   ```
2. The contract computes `total_amount_to_remove = amount_of_headers_we_store - gc_threshold` and deletes all excess headers, including the block at height `H`.
3. `mainchain_initial_blockhash` is advanced past `H`.
4. Victim calls `verify_transaction_inclusion` with `tx_block_blockhash` pointing to the pruned block.
5. The contract panics: `"block does not belong to the current main chain"`.
6. The victim's cross-chain verification fails; any funds gated on this result are locked.

**Expected outcome:** The victim's SPV proof is permanently unverifiable for the pruned block, with no recovery path.
