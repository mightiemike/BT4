### Title
`traceTransaction` Always Returns `internal_server_error` for Pending Declare V0/V1 Transactions — (`crates/apollo_rpc/src/v0_8/api/mod.rs`)

### Summary

The `trace_transaction` pending path converts each pending transaction via `stored_txn_to_executable_txn`, which for Declare V0/V1 calls `get_deprecated_class_for_re_execution`. That function looks up the deprecated class definition exclusively from committed storage. For a pending Declare V0/V1, the class has never been committed to storage, and the `ExecutionPendingData` built for the pending path explicitly zeroes out `old_declared_contracts` and `classes`. The lookup therefore always returns `None`, producing `internal_server_error("Missing deprecated class definition of {class_hash}.")` instead of a valid trace.

---

### Finding Description

**Step 1 — Pending path in `trace_transaction`**

When the target transaction hash is found in the pending block, `state_number` is set to the state right after the latest *committed* block, and `ExecutionPendingData` is constructed with all class-related fields explicitly defaulted to empty: [1](#0-0) 

The comment at line 1194–1196 confirms the intent: "The pending state diff should be empty since we look at the state in the start of the pending block." This means `old_declared_contracts` and `classes` are both empty for the execution context.

**Step 2 — `stored_txn_to_executable_txn` for DeclareV0/V1**

For both `DeclareTransaction::V0` and `DeclareTransaction::V1`, the function immediately calls `get_deprecated_class_for_re_execution` with the committed-storage transaction and the `state_number` from Step 1: [2](#0-1) 

**Step 3 — `get_deprecated_class_for_re_execution` only reads committed storage**

The function advances `state_number` by one block and queries `get_deprecated_class_definition_at`. If the result is `None`, it returns `internal_server_error`: [3](#0-2) 

The comment at lines 433–435 explains the design for *committed* blocks: the class is stored in the state *after* the block it appeared in. For a pending Declare V0/V1, the class has never been committed, so `get_deprecated_class_definition_at` returns `None` unconditionally.

**Contrast with the normal pending-data path**

`client_pending_data_to_execution_pending_data` correctly propagates `old_declared_contracts` and `classes` (from `PendingClasses`) into `ExecutionPendingData`: [4](#0-3) 

But the `trace_transaction` pending path never calls this helper — it constructs `ExecutionPendingData` inline with all class fields zeroed.

---

### Impact Explanation

Any caller of `starknet_traceTransaction` with the hash of a pending Declare V0 or V1 transaction receives `internal_server_error("Missing deprecated class definition of 0x…")` instead of a valid `DeclareTxnTrace`. This is an authoritative-looking server error that misrepresents the state of the network. It falls squarely within: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

Declare V1 transactions are still valid on current Starknet. Any user who submits a Declare V1 transaction and immediately calls `traceTransaction` before the block is committed will reliably trigger this error. No special privileges are required.

---

### Recommendation

In the `trace_transaction` pending path, replace the inline `ExecutionPendingData` construction (with zeroed class fields) with a call to `client_pending_data_to_execution_pending_data`, passing the live `PendingClasses` that already contains the pending deprecated class definitions. Alternatively, `stored_txn_to_executable_txn` should accept an optional `PendingClasses` argument and fall back to it when the committed-storage lookup returns `None`.

---

### Proof of Concept

1. Start a node with an empty storage (no committed blocks, or at least one committed block for `get_latest_block_number` to succeed).
2. Add a Declare V1 transaction to the pending block (class hash `0xABC`, class not in committed storage).
3. Call `starknet_traceTransaction` with the pending transaction hash.
4. Observe: response is `internal_server_error("Missing deprecated class definition of 0xabc.")` rather than a `DeclareTxnTrace` object.

The exact divergent value is the `internal_server_error` JSON-RPC error object in place of a valid `DeclareTxnTrace`.

```rust
fn get_deprecated_class_for_re_execution(
    storage_txn: &StorageTxn<'_, RO>,
    state_number: StateNumber,
    class_hash: ClassHash,
) -> Result<starknet_api::deprecated_contract_class::ContractClass, ErrorObjectOwned> {
    let state_number_after_block =
        StateNumber::unchecked_right_after_block(state_number.block_after());
    storage_txn
        .get_state_reader()
        .map_err(internal_server_error)?
        .get_deprecated_class_definition_at(state_number_after_block, &class_hash)
        .map_err(internal_server_error)?
        .ok_or_else(|| {
            internal_server_error(format!("Missing deprecated class definition of {class_hash}."))
        })
}
```
