Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Enabling Allowlist Bypass — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps using the `sender` argument, which is `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` is used, `msg.sender` to the pool is the router contract, not the end user. If the router is allowlisted to enable legitimate router-mediated trading, every user — including explicitly blocked ones — can bypass the per-user allowlist by routing through the public router.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` directly as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards `sender` unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the immediate caller of `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly with no encoding of the original user into `extensionData`: [4](#0-3) 

This means the allowlist lookup becomes `allowedSwapper[pool][router]`. If the router is allowlisted (necessary for any legitimate router-mediated swap), the check passes for every caller regardless of their individual allowlist status. The router passes `params.extensionData` verbatim from user input with no automatic user-identity encoding, so there is no existing mechanism for the extension to recover the real initiator. [5](#0-4) 

## Impact Explanation
A pool admin deploying a permissioned pool (KYC-only, institutional-only) via `SwapAllowlistExtension` faces an irresolvable configuration dilemma: not allowlisting the router blocks all router-mediated swaps including for legitimate users; allowlisting the router grants every user — including explicitly blocked ones — the ability to execute arbitrary swaps against LP positions. This directly breaks the access-control invariant the admin intended to enforce, enabling unauthorized fund flows against LP principal. This qualifies as an admin-boundary break and broken core pool functionality causing loss of funds.

## Likelihood Explanation
`MetricOmmSimpleRouter` is a public, permissionless periphery contract. Any user blocked from direct `pool.swap()` will naturally attempt the router as an alternative path. The admin allowlisting the router is a predictable and necessary operational action to support normal trading. No privileged access, special tokens, or unusual setup is required.

## Recommendation
The extension must verify the end user's identity, not the immediate caller of `pool.swap()`. The preferred fix is to redesign `beforeSwap` to accept the original initiator separately, or require the router to encode the real `msg.sender` into `extensionData` and have the extension decode and verify it. Alternatively, expose `isAllowedToSwap(pool, user)` as a view that the router calls before forwarding, and enforce that the router only proceeds if the originating user passes the check.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Admin allowlists router: swapExtension.setAllowedToSwap(pool, address(router), true)
  - blockedUser is NOT in the allowlist

Step 1: blockedUser calls pool.swap() directly
  → sender = blockedUser
  → allowedSwapper[pool][blockedUser] = false → REVERT ✓ (guard works)

Step 2: blockedUser calls router.exactInputSingle(pool, ...)
  → router calls pool.swap()
  → msg.sender to pool = router address
  → sender passed to extension = router address
  → allowedSwapper[pool][router] = true → PASSES ✗ (guard bypassed)
  → swap executes, blockedUser receives output tokens
```
