### Title
Stale `linkedSigner` Not Cleared After Liquidation Enables Persistent Unauthorized Subaccount Control — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `linkedSigners` mapping in `EndpointStorage.sol` is never cleared when a subaccount is liquidated or fully drained. If a user re-deposits to the same subaccount after liquidation, the previously authorized linked signer retains full signing authority — including the ability to sign new `LinkSigner` transactions to perpetuate their own access and `MintNlp`/`BurnNlp` transactions to cause direct financial losses.

---

### Finding Description

`linkedSigners[subaccount]` is written in two places in `EndpointTx.sol`: the slow-mode `LinkSigner` path and the fast-mode `LinkSigner` path. [1](#0-0) [2](#0-1) 

It is stored in `EndpointStorage` as a persistent mapping: [3](#0-2) 

Neither `withdrawCollateral` nor `liquidateSubaccount` in `Clearinghouse.sol` touches this mapping. `withdrawCollateral` only adjusts the spot balance and checks health: [4](#0-3) 

`liquidateSubaccount` delegates to `clearinghouseLiq` via `delegatecall`, which operates on `Clearinghouse` storage — entirely separate from `EndpointStorage` where `linkedSigners` lives: [5](#0-4) 

This means after liquidation empties a subaccount, `linkedSigners[subaccount]` retains the old authorized address.

The critical amplifier is that `LinkSigner` fast-mode transactions themselves have `allowLinkedSigner = true`: [2](#0-1) 

This means the stale linked signer B can sign a new `LinkSigner` transaction for subaccount S to re-set `linkedSigners[S]` to any address B controls — even after the legitimate owner tries to revoke B. B can always re-link in fast mode before the owner's revocation is sequenced.

Additionally, `MintNlp` and `BurnNlp` both have `allowLinkedSigner = true`: [6](#0-5) 

The `BurnNlp` path in `Clearinghouse.sol` charges a mandatory burn fee of `max(1, quoteAmount/1000)`: [7](#0-6) 

A stale linked signer can cycle the user's quote through `MintNlp` → `BurnNlp` to extract this fee as a direct financial loss.

---

### Impact Explanation

A stale linked signer on a re-funded subaccount can:

1. **Perpetuate their own access**: Sign a `LinkSigner` transaction (allowed for linked signers) to re-set `linkedSigners[S]` to themselves or a new address, defeating any revocation attempt by the owner.
2. **Cause direct financial loss**: Sign `MintNlp` followed by `BurnNlp` to force the subaccount to pay the 0.1% burn fee on the full quote balance.
3. **Disrupt positions**: Sign `WithdrawCollateral` to force withdrawals (funds go to the owner's address, but positions are disrupted) or sign `TransferQuote` to move quote between the user's own subaccounts.

The `LinkSigner` self-perpetuation is the most severe impact: it makes revocation impossible through normal protocol means once a stale linked signer is active on a re-funded subaccount.

---

### Likelihood Explanation

Liquidation is an involuntary event. A user whose subaccount is liquidated will commonly re-deposit to the same subaccount to resume trading. The subaccount identifier (`bytes32` of address + name) is fixed and reused. The user may not remember that a linked signer was set before liquidation, especially if the linked signer was configured for automated trading. The stale linked signer is not visible in any on-chain event emitted during liquidation, making it easy to miss.

---

### Recommendation

Clear `linkedSigners[subaccount]` when a subaccount is fully liquidated. Since `liquidateSubaccount` in `Clearinghouse.sol` uses `delegatecall` to `clearinghouseLiq`, the clearing must happen in `Endpoint` storage. One approach: after `clearinghouse.liquidateSubaccount(signedTx.tx)` returns in `EndpointTx.processTransactionImpl`, check if the liquidated subaccount's health is terminal (finalization path, `productId == type(uint32).max`) and clear `linkedSigners[signedTx.tx.liquidatee]`.

Additionally, consider disallowing linked signers from signing `LinkSigner` transactions (`allowLinkedSigner = false` for that transaction type), so only the subaccount owner can modify signing authority.

---

### Proof of Concept

1. User A links signer B to subaccount `S`: sequencer processes `LinkSigner` → `linkedSigners[S] = B`.
2. Subaccount `S` becomes unhealthy; a liquidator finalizes it (`productId = type(uint32).max`). All balances are zeroed. `linkedSigners[S]` is **not** cleared.
3. User A re-deposits collateral to subaccount `S` via `DepositCollateral`. `_recordSubaccount(S)` is a no-op since `S` is already registered.
4. B observes the re-deposit on-chain and immediately submits a fast-mode `LinkSigner` transaction signed by B, setting `linkedSigners[S] = B` (or a fresh address C controlled by B). This is valid because `allowLinkedSigner = true` for `LinkSigner`.
5. User A submits a `LinkSigner` revocation (`signer = address(0)`). The sequencer processes B's re-link first (or B re-links again after A's revocation), maintaining persistent access.
6. B signs `MintNlp` for subaccount `S` converting User A's full quote balance to NLP tokens.
7. B signs `BurnNlp` for subaccount `S`, converting NLP back to quote minus the burn fee (`max(1, quoteAmount/1000)`). User A suffers a direct loss of ≥0.1% of their deposited collateral per cycle. [3](#0-2) [2](#0-1) [7](#0-6)
