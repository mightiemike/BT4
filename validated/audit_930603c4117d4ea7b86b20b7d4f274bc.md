Audit Report

## Title
SwapAllowlistExtension Bypass via Router Address Allowlisting ŌĆö (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the pool receives the router's address as `sender`, not the originating user. A pool admin who allowlists the router address to enable router-mediated swaps for permitted users inadvertently grants every caller of the public router the ability to bypass the per-user allowlist.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` directly as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]` ŌĆö i.e., `allowedSwapper[pool][router]` ŌĆö when the call originates from the router: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` the pool and extension observe: [4](#0-3) 

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` checks the explicit `owner` parameter ŌĆö the actual position owner ŌĆö not the immediate caller, so it is not susceptible to this identity mismatch: [5](#0-4) 

The swap path has no equivalent explicit-user parameter; the only identity available to the extension is `msg.sender` of `pool.swap()`. When a pool admin sets `allowedSwapper[pool][router] = true` to allow permitted users to reach the pool through the standard router, the extension passes every call whose `msg.sender` is the router, regardless of who initiated the router call.

## Impact Explanation
`SwapAllowlistExtension` is the sole on-chain mechanism for restricting who may trade against a pool. Once bypassed, any unpermitted address can execute swaps against a pool whose admin believed it was access-controlled. This constitutes a direct broken core pool functionality causing loss of funds: adversarial swappers can extract LP value through repeated one-sided trades at oracle prices, drain a bin, or front-run permitted users while the allowlist appears correctly configured. This meets the Sherlock Critical/High threshold for broken access control with direct loss of LP principal.

## Likelihood Explanation
The bypass requires the pool admin to allowlist the router address. This is a natural and expected operational step: without it, permitted users cannot use the standard router at all. The admin has no in-code warning that allowlisting the router is equivalent to disabling the per-user allowlist for all router callers. The condition is therefore likely to occur in any production deployment that uses `SwapAllowlistExtension` alongside `MetricOmmSimpleRouter`. The exploit is repeatable by any unpermitted address with no special privileges.

## Recommendation
1. **Pass the originating user through `extensionData`**: The router should encode `msg.sender` (the actual user) into `extensionData` before calling `pool.swap()`. `SwapAllowlistExtension.beforeSwap` should decode and check that address when `extensionData` is non-empty, falling back to `sender` for direct pool calls.
2. **Add an explicit `swapper` parameter**: Align with the deposit allowlist pattern by adding an explicit `swapper` parameter to `pool.swap()` (analogous to `owner` in `addLiquidity`) so the extension gates on the economically relevant actor rather than the immediate caller.
3. **Document the identity semantics**: Clearly state in `SwapAllowlistExtension` that `sender` is `msg.sender` of `pool.swap()`, so pool admins understand that allowlisting any intermediary contract grants that contract's entire caller set access.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  allowAllSwappers[pool] = false
  allowedSwapper[pool][permittedUser] = true
  allowedSwapper[pool][router]        = true   ŌåÉ admin adds this to enable router access

Attack:
  unpermittedUser calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  ŌåÆ router calls pool.swap(...)          msg.sender = router
  ŌåÆ pool calls _beforeSwap(router, ...)
  ŌåÆ extension checks allowedSwapper[pool][router] == true  Ō£ō
  ŌåÆ swap executes; unpermittedUser receives output tokens

Result:
  unpermittedUser bypasses the per-user allowlist and trades against the restricted pool.
  The pool admin's access-control invariant is silently broken.

Foundry test plan:
  1. Deploy pool with SwapAllowlistExtension.
  2. Set allowAllSwappers[pool] = false.
  3. setAllowedToSwap(pool, permittedUser, true).
  4. setAllowedToSwap(pool, router, true).
  5. Call router.exactInputSingle from unpermittedUser ŌĆö assert swap succeeds.
  6. Call pool.swap directly from unpermittedUser ŌĆö assert NotAllowedToSwap reverts.
  Step 5 passing and step 6 reverting confirms the bypass.
```
