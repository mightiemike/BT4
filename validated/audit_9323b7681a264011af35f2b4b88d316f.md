### Title
Stale `digestToSubaccount` Entry After Isolated Subaccount Closure Routes Trades to Orphaned Subaccount â€” (`File: core/contracts/OffchainExchange.sol`)

---

### Summary

When an isolated subaccount is closed via `_tryCloseIsolatedSubaccount`, the `digestToSubaccount` mapping is never cleared. The stale entry persists and causes `matchOrders` to silently redirect any subsequent fill of the same order digest to the now-orphaned (closed) isolated subaccount, where resulting balance changes cannot be recovered by the parent.

---

### Finding Description

`createIsolatedSubaccount` records the mapping `digestToSubaccount[digest] = newIsolatedSubaccount` at line 1072 and optionally `digestToMargin[digest] = margin` at line 1076. [1](#0-0) 

`_tryCloseIsolatedSubaccount` (lines 160â€“208) performs the following cleanup when a position is fully closed:

- Clears `isolatedSubaccountsMask[addr]` bit (line 202)
- Clears `isolatedSubaccounts[parent][id]` (line 203)
- Clears `parentSubaccounts[subaccount]` (line 204) [2](#0-1) 

It does **not** clear `digestToSubaccount[digest]` or `digestToMargin[digest]`.

In `matchOrders`, before executing a fill, the sender is unconditionally overwritten if `digestToSubaccount` is non-zero:

```solidity
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    taker.order.sender = digestToSubaccount[ordersInfo.taker.digest];
}
if (digestToSubaccount[ordersInfo.maker.digest] != bytes32(0)) {
    maker.order.sender = digestToSubaccount[ordersInfo.maker.digest];
}
``` [3](#0-2) 

There is no check that the subaccount returned by `digestToSubaccount` is still live. After closure, `parentSubaccounts[subaccount]` is `bytes32(0)`, making the subaccount fully orphaned.

---

### Impact Explanation

Any subsequent fill of a partially-filled isolated order (whose isolated subaccount has since been closed) is routed to the orphaned subaccount. Balance changes â€” losses or gains â€” are applied to a subaccount whose `parentSubaccounts` entry is `bytes32(0)`. Any future call to `_tryCloseIsolatedSubaccount` on this orphaned subaccount would transfer residual balances to `bytes32(0)` (the zero-address subaccount), permanently destroying them. The parent subaccount neither absorbs the loss nor recovers the gain. This is a concrete, unrecoverable accounting corruption.

---

### Likelihood Explanation

An isolated order is partially filled, then the isolated subaccount is closed (e.g., via liquidation or a normal zero-position close triggered by the sequencer or clearinghouse). The `filledAmounts[digest]` for the order is not fully consumed, so the sequencer can legitimately submit another `matchOrders` call for the remaining amount. This is a normal operational sequence â€” partial fills followed by position closure are expected in a perpetuals DEX. No special attacker privilege is required beyond submitting a valid isolated order.

---

### Recommendation

In `_tryCloseIsolatedSubaccount`, after clearing `parentSubaccounts[subaccount]`, also delete the `digestToSubaccount` and `digestToMargin` entries associated with the closed subaccount. This requires either storing the digest on the subaccount at creation time (a reverse mapping `subaccountToDigest`) or iterating the known digests. The simplest fix is to add a `mapping(bytes32 => bytes32) internal subaccountToDigest` populated in `createIsolatedSubaccount`, and use it in `_tryCloseIsolatedSubaccount` to clear both `digestToSubaccount` and `digestToMargin`.

---

### Proof of Concept

1. User signs an isolated order for product P with digest D. Sequencer calls `createIsolatedSubaccount` â†’ `digestToSubaccount[D] = ISO_1`, `parentSubaccounts[ISO_1] = PARENT`. [4](#0-3) 

2. Order is partially filled. `filledAmounts[D] < order.amount`.

3. ISO_1's perp position reaches zero (e.g., via liquidation). Sequencer or clearinghouse calls `tryCloseIsolatedSubaccount(ISO_1)`. Balances are swept to PARENT. `parentSubaccounts[ISO_1] = bytes32(0)`. `isolatedSubaccounts[PARENT][id] = bytes32(0)`. **`digestToSubaccount[D]` is not cleared.** [2](#0-1) 

4. Sequencer submits `matchOrders` for the remaining fill of digest D. `matchOrders` reads `digestToSubaccount[D] = ISO_1` (stale) and overwrites `taker.order.sender = ISO_1`. [5](#0-4) 

5. The fill executes against ISO_1. ISO_1 now holds a non-zero balance with `parentSubaccounts[ISO_1] = bytes32(0)`. Any subsequent close attempt transfers to `bytes32(0)`, permanently destroying the funds. The parent subaccount is unaffected and does not absorb the loss.
