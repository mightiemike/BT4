Looking at the gateway flow vs. consensus path, I've identified a clear analog to the `_notSameBlock()` bypass.

The `check_declare_permissions()` check in `GenericGateway` enforces `authorized_declarer_accounts` and `block_declare` only on the RPC/gateway path. The consensus/P2P path (`validate_proposal` → `convert_consensus_tx_to_internal_consensus_tx` → `send_txs_for_proposal`) never calls this check, so a proposer can include a declare transaction from an unauthorized sender and every validator will execute it.

---

### Title
`check_declare_permissions()` enforced only on the RPC gateway path, bypassed via the consensus proposal path — (`File: crates/apollo_gateway/src/gateway.rs`)

### Summary

`GenericGateway::check_declare_permissions()` enforces the operator-configured `authorized_declarer_accounts` allowlist and the `block_declare` kill-switch exclusively on the RPC `add_tx` path. The consensus validation path (`validate_proposal` → `handle_proposal_part` → `convert_consensus_tx_to_internal_consensus_tx` → `send_txs_for_proposal`) never invokes this check. Any proposer node that does not share the same restriction can include a declare transaction from an unauthorized sender, and every validator node — even one with `authorized_declarer_accounts` set — will convert, execute, and commit it.

### Finding Description

**Gateway path (enforced):**

`GenericGateway::add_tx` calls `check_declare_permissions` before any conversion or execution:

```rust
// crates/apollo_gateway/src/gateway.rs  lines 407-433
fn check_declare_permissions(&self, declare_tx: &RpcDeclareTransaction) -> Result<(), StarknetError> {
    if self.config.static_config.block_declare {
        return Err(...);
    }
    let RpcDeclareTransaction::V3(declare_v3_tx) = declare_tx;
    if !self.config.is_authorized_declarer(&declare_v3_tx.sender_address) {
        return Err(StarknetError { code: UnauthorizedDeclare, ... });
    }
    Ok(())
}
```

`is_authorized_declarer` returns `false` when `authorized_declarer_accounts` is `Some(list)` and the sender is not in the list:

```rust
// crates/apollo_gateway_config/src/config.rs  lines 141-146
pub fn is_authorized_declarer(&self, declarer_address: &ContractAddress) -> bool {
    match &self.static_config.authorized_declarer_accounts {
        Some(allowed_accounts) => allowed_accounts.contains(declarer_address),
        None => true,
    }
}
```

**Consensus path (unenforced):**

When a validator receives a proposal, `handle_proposal_part` converts each `ConsensusTransaction` directly to `InternalConsensusTransaction` and forwards it to the batcher:

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs  lines 602-635

    futures::future::join_all(txs.into_iter().map(|tx| {
        transaction_converter.convert_consensus_tx_to_internal_consensus_tx(tx)
    })).await ...;
...
let input = SendTxsForProposalInput { proposal_id, txs };
batcher.send_txs_for_proposal(input).await ...;
```

`convert_consensus_tx_to_internal_consensus_tx` calls only `convert_rpc_tx_to_internal` (hash computation + compiled-class-hash check) — no `check_declare_permissions`, no `authorized_declarer_accounts` check, no `block_declare` check:

```rust

    -> TransactionConverterResult<(InternalConsensusTransaction, Option<VerifyAndStoreProofTask>)>
{

            let (internal_tx, proof_data) = self.convert_rpc_tx_to_internal(tx).await?;
            ...
            Ok((InternalConsensusTransaction::RpcTransaction(internal_tx), task))
        }
        ...
    }

```

The blockifier's `perform_pre_validation_stage` validates nonce, fee bounds, and proof-facts cryptographic fields, but has no concept of `authorized_declarer_accounts` or `block_declare`. The class is declared and committed to state.

### Impact Explanation

A validator node that has configured `authorized_declarer_accounts` to restrict which accounts may declare contracts, or has set `block_declare = true` to temporarily halt all declares, will still execute and commit declare transactions that arrive via the consensus proposal path. The resulting on-chain state (a new `class_hash` → `compiled_class_hash` mapping) is indistinguishable from a legitimately declared class. This matches **Critical: wrong class hash / state from blockifier execution logic for accepted input**.

### Likelihood Explanation

In a decentralized sequencer deployment, any node whose turn it is to propose a block can include a declare transaction from an unauthorized sender. The proposer node need not share the validator's `authorized_declarer_accounts` config. No cryptographic forgery is required; the transaction is structurally valid. The only prerequisite is that the attacker controls or cooperates with a proposer node, which is a normal participant role in BFT consensus.

### Recommendation

Move the `authorized_declarer_accounts` / `block_declare` enforcement out of the gateway-only `check_declare_permissions` and into the shared `convert_rpc_tx_to_internal` path (or a dedicated pre-execution hook called by both the gateway and the consensus validator). Alternatively, enforce the restriction inside the batcher's `send_txs_for_proposal` handler so that it applies regardless of which entry point delivered the transaction.

### Proof of Concept

1. Operator configures validator node with `authorized_declarer_accounts = ["0xAUTHORIZED"]` and `block_declare = false`.
2. Attacker controls a proposer node with no `authorized_declarer_accounts` restriction.
3. Attacker submits a declare transaction from address `0xUNAUTHORIZED` to the proposer's gateway (which accepts it) or injects it directly into the proposer's mempool.
4. Proposer includes the transaction in a `ProposalPart::Transactions` batch and broadcasts it via consensus P2P.
5. Validator receives the proposal. `handle_proposal_part` calls `convert_consensus_tx_to_internal_consensus_tx` — no `check_declare_permissions` is invoked.
6. The internal transaction is forwarded to the batcher via `send_txs_for_proposal`. The blockifier executes `try_declare`, writing the new class hash to state.
7. The proposal commitment matches; consensus finalizes the block. The unauthorized class is now declared on-chain, bypassing the operator's `authorized_declarer_accounts` restriction entirely. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
