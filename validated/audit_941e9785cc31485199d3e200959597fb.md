Audit Report

## Title
`SwapAllowlistExtension` checks the router's address instead of the actual user's address, allowing any user to bypass the swap allowlist via the router - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the immediate caller of `MetricOmmPool.swap`. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router's address, not the user's address. A pool admin who adds the router to the allowlist (the only way to permit router-mediated swaps) inadvertently opens the pool to every user, completely defeating the allowlist. Any unprivileged user can bypass the swap allowlist by calling the router instead of the pool directly.

## Finding Description

**Root cause — wrong actor is checked:**

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-240
_beforeSwap(
  msg.sender,   // ← immediate caller, not the end user
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to the extension:

```solidity
// ExtensionCalling.sol:160-176
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeSwap,
    (sender, recipient, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol:37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap`. When the user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap`:

```solidity
// MetricOmmSimpleRouter.sol:72-80
IMetricOmmPoolActions(params.pool).swap(
  params.recipient, params.zeroForOne, ..., params.extensionData
);
```

So `sender` = `address(router)`, not `address(user)`. The allowlist lookup becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Exploit path:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a curated set of addresses.
2. Pool admin calls `setAllowedToSwap(pool, address(router), true)` — the only way to allow any router-mediated swap.
3. Any unprivileged user (not on the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the restricted pool.
4. The pool calls `_beforeSwap(address(router), ...)`.
5. The extension checks `allowedSwapper[pool][address(router)]` → `true` → swap proceeds.
6. The unprivileged user successfully swaps on a pool that was supposed to be restricted to specific addresses.

**Existing guards are insufficient:** The only guard is `allowedSwapper[pool][sender]`. There is no mechanism to recover the original `msg.sender` of the router call. The router does not forward the user's identity in `extensionData` by default.

## Impact Explanation
The swap allowlist is a core access-control feature. Its bypass allows any unprivileged user to trade on pools that pool admins intended to restrict to specific counterparties (e.g., KYC'd addresses, whitelisted market makers). This constitutes broken core pool functionality and an admin-boundary break: the pool admin's configured access control is rendered ineffective by any user routing through the public router. Depending on the pool's purpose, this can result in unauthorized parties extracting value from the pool or trading against oracle prices that were only intended for trusted counterparties.

## Likelihood Explanation
The condition is trivially reachable: any user who calls `MetricOmmSimpleRouter` instead of `MetricOmmPool` directly bypasses the check. The router is the standard, documented user-facing entry point. A pool admin who wants to allow any router-mediated swap must add the router to the allowlist, which is the natural and expected configuration. No special privileges, flash loans, or unusual conditions are required. The bypass is repeatable on every swap.

## Recommendation
The `sender` forwarded to extensions should represent the economic actor, not the immediate contract caller. Two complementary fixes:

1. **In the router**: pass the original `msg.sender` (the end user) through `extensionData` or a dedicated field so extensions can recover the true initiator.
2. **In `SwapAllowlistExtension`**: if the `sender` is a known router, decode and check the embedded user address from `extensionData` rather than the router address itself.

Alternatively, define a standard interface for periphery contracts to declare the originating user, and have the extension check that declared address when `sender` is a registered router.

```solidity
// Foundry test sketch
function test_allowlistBypassViaRouter() public {
    // 1. Deploy pool with SwapAllowlistExtension
    // 2. Add liquidity
    // 3. Do NOT add attacker to allowlist
    // 4. Add router address to allowlist (required for any router swap)
    swapExtension.setAllowedToSwap(address(pool), address(router), true);

    // 5. Attacker (not on allowlist) swaps via router
    vm.prank(attacker); // attacker is not in allowedSwapper
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        recipient: attacker,
        tokenIn: address(token0),
        zeroForOne: true,
        amountIn: 1000,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    }));
    // Swap succeeds — allowlist bypassed
}
```
