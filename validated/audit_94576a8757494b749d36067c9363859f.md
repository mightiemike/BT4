### Title
Unrestricted `creditDeposit()` Allows Any Caller to Force All DDA Token Balances Into the Exchange — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` carries no access-control modifier. Any unprivileged external caller can invoke it at any time to sweep every token balance held by the DDA contract into the Nado exchange, regardless of the DDA owner's intent. This is the direct structural analog to the reported `allocate()` vulnerability: a public, permissionless function that moves third-party funds into a protocol destination the caller chooses the timing of, with no cap or owner gate.

---

### Finding Description

`DirectDepositV1` is a per-user "direct deposit address" contract. Users send ERC-20 tokens to its address; the owner later calls `creditDeposit()` (or the sequencer does) to credit those tokens to the fixed `subaccount` on the exchange. The function is declared `external` with no `onlyOwner` or any other guard: [1](#0-0) 

When called, it:
1. Iterates over **every** product ID registered in the spot engine.
2. For each product, reads the DDA's full token balance.
3. Approves the `endpoint` for that full balance.
4. Calls `endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1")`.

Inside `depositCollateralWithReferral`, `handleDepositTransfer` **immediately** moves the tokens from the DDA to the clearinghouse via `safeTransferFrom` / `safeTransferTo`: [2](#0-1) 

The tokens leave the DDA's custody the moment `creditDeposit()` is called. A slow-mode queue entry is created, but the asset transfer is not deferred — it is final at call time. [3](#0-2) 

The `withdraw()` escape hatch that the owner could use to recover tokens is `onlyOwner`: [4](#0-3) 

Once an attacker calls `creditDeposit()` first, `withdraw()` has nothing left to return.

---

### Impact Explanation

An attacker who observes tokens arriving at a DDA (trivially visible on-chain) can front-run the owner's `withdraw()` call by calling `creditDeposit()`. The tokens are immediately transferred to the clearinghouse and are no longer recoverable from the DDA. The owner must now wait for the sequencer to process the queued slow-mode deposit (up to the configured delay), then submit a separate withdrawal transaction, before regaining access to the funds.

More critically — mirroring the external report exactly — if the Nado exchange or a specific product is compromised or paused at the moment tokens arrive at the DDA, an attacker can call `creditDeposit()` to push those tokens into the exchange before the owner can react and call `withdraw()`. The owner loses the ability to keep funds outside the exchange during an incident window.

Because `creditDeposit()` iterates over **all** product IDs, a single call drains every token balance the DDA holds, not just one asset. The corrupted state is: tokens that were safely held outside the exchange are now locked inside the clearinghouse.

---

### Likelihood Explanation

- DDA contract addresses and their token balances are fully visible on-chain.
- No special privilege, signature, or fee is required to call `creditDeposit()`.
- Front-running a `withdraw()` call is a standard mempool attack requiring only gas.
- The scenario is realistic whenever a user sends tokens to a DDA and then wants to cancel (e.g., exchange pause, product delisting, user error), or whenever an attacker wants to maximize funds locked in the exchange during an incident.

---

### Recommendation

Add `onlyOwner` to `creditDeposit()`:

```solidity
function creditDeposit() external onlyOwner {
```

This matches the pattern already applied to `withdraw()` and `withdrawNative()` in the same contract, and mirrors the recommendation in the external report to guard the analogous `allocate()` function with `onlyOwner`. [5](#0-4) 

---

### Proof of Concept

1. A `DirectDepositV1` is deployed with `subaccount = alice_subaccount`.
2. Alice sends 10,000 USDC to the DDA address, intending to deposit later.
3. The Nado exchange is paused due to an incident; Alice decides to recover her USDC by calling `withdraw(USDC)`.
4. Attacker observes Alice's pending `withdraw()` in the mempool and front-runs it by calling `creditDeposit()` with higher gas.
5. `creditDeposit()` executes: `USDC.approve(endpoint, 10000e6)` → `endpoint.depositCollateralWithReferral(alice_subaccount, USDC_PRODUCT_ID, 10000e6, "-1")` → `handleDepositTransfer` moves 10,000 USDC from the DDA to the clearinghouse immediately.
6. Alice's `withdraw()` call executes but finds a zero balance — the DDA holds nothing.
7. Alice's 10,000 USDC is now locked inside the clearinghouse during the incident, inaccessible until the exchange resumes and a withdrawal is processed. [1](#0-0)

    function creditDeposit() external {
        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            address tokenAddr = spotEngine.getToken(productId);
            require(tokenAddr != address(0), "Invalid productId.");
            IIERC20Base token = IIERC20Base(tokenAddr);
            uint256 balance = token.balanceOf(address(this));
            if (balance != 0) {
                token.approve(address(endpoint), balance);
                endpoint.depositCollateralWithReferral(
                    subaccount,
                    productId,
                    uint128(balance),
                    "-1"
                );
            }
        }
    }
