Audit Report

## Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter and gates only on `owner`. Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts a caller-supplied `owner` with no identity check between `owner` and `msg.sender`, any non-allowlisted address can deposit into a curated pool by naming an allowlisted address as `owner`, paying tokens from their own balance while minting shares into the allowlisted address's position key.

## Finding Description
`DepositAllowlistExtension.beforeAddLiquidity` is declared with the first `address` parameter unnamed (discarded):

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
```

Its guard reads:

```solidity
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
}
```

`msg.sender` here is the pool (correct for pool identity), and `owner` is the position recipient — not the entity paying tokens. The `sender` argument (which is `msg.sender` of the pool's `addLiquidity` call, i.e. `MetricOmmPoolLiquidityAdder`) is completely ignored.

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (the overload with an explicit `owner` parameter) performs no identity check between `owner` and `msg.sender`:

```solidity
function _validateOwner(address owner) internal pure {
    if (owner == address(0)) revert InvalidPositionOwner();
}
```

The payer stored in transient context is always `msg.sender` (attacker B), while `positionOwner` passed to the pool is the attacker-supplied allowlisted address A:

```solidity
return _addLiquidity(pool, owner, salt, deltas, msg.sender, maxAmountToken0, maxAmountToken1, extensionData);
```

The pool passes `msg.sender` (the adder contract) as `sender` to the extension hook:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` encodes both `sender` and `owner` in the call:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

But `DepositAllowlistExtension` discards `sender` entirely and only checks `owner`. The existing test `test_exactShares_canAddOnBehalfOfAnotherOwner` and `test_exactShares_usesMsgSenderAsPayerNotOwner` confirm the adder intentionally supports owner ≠ payer, but the extension does not account for this separation.

## Impact Explanation
The deposit allowlist — the primary curation mechanism for private/permissioned pools — is fully bypassed. Any address can deposit into a pool that is supposed to accept only allowlisted depositors by supplying an allowlisted address as `owner`. The attacker's tokens are pulled from their balance and minted into the allowlisted address's position key `(pool, A, salt)`. Since `removeLiquidity` enforces `msg.sender == owner`, only A can withdraw — the attacker's tokens are permanently locked in A's position unless A chooses to remove them. This breaks the pool's curation invariant (non-allowlisted capital enters the pool), enables griefing of allowlisted users by inflating their positions without consent, and eliminates effective control for pool admins relying on the allowlist for regulatory or strategy-isolation purposes.

## Likelihood Explanation
The attack requires only a public call to `addLiquidityExactShares` with a known allowlisted address as `owner`. No privileged access, no special setup, no flash loan. The `allowedDepositor` mapping is public, so any observer can identify allowlisted addresses. The attack is immediately executable by any address that has approved the `MetricOmmPoolLiquidityAdder` for token spending.

## Recommendation
In `DepositAllowlistExtension.beforeAddLiquidity`, check `sender` (the actual depositing entity) rather than `owner` (the position recipient):

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to restrict both who pays and who holds positions, check both `sender` and `owner`. Additionally, `_validateOwner` in `MetricOmmPoolLiquidityAdder` should enforce `owner == msg.sender` unless delegation is explicitly intended and documented.

## Proof of Concept
1. Deploy a pool with `DepositAllowlistExtension`; allowlist only address A: `setAllowedToDeposit(pool, A, true)`.
2. As attacker B (not allowlisted), approve `MetricOmmPoolLiquidityAdder` for token spending.
3. Call `addLiquidityExactShares(pool, A, salt, deltas, max0, max1, "")` from B.
4. `_validateOwner(A)` passes (A ≠ 0). Payer stored as B in transient context.
5. Pool calls `_beforeAddLiquidity(msg.sender=adder, owner=A, ...)`. Extension checks `allowedDepositor[pool][A] == true` → passes.
6. Shares minted under `(pool, A, salt)`; callback pulls tokens from B.
7. Assert: B's token balance decreased; A's position shares increased; B holds no shares and cannot withdraw.
