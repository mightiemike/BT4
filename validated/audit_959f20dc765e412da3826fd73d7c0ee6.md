### Title
Stale `digestToSubaccount` Entry After Isolated Subaccount Closure Allows Old Orders to Execute Against Reused Subaccount — (`core/contracts/OffchainExchange.sol`)

### Summary
`_tryCloseIsolatedSubaccount` clears three of the four tracking structures for an isolated subaccount but never clears `digestToSubaccount[digest]`. Because isolated subaccount `bytes32` identifiers are deterministically derived from `(address, productId, id)`, closing and reopening a subaccount for the same product reuses the same `id` slot and therefore the same `bytes32`. The stale `digestToSubaccount` entry then silently re-binds the old order to the newly created subaccount, allowing the old order to be matched against the new subaccount's margin without the user's intent.

### Finding Description
`_tryCloseIsolatedSubaccount` performs the following cleanup:

```solidity
isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);   // clears bit
isolatedSubaccounts[parent][id] = bytes32(0);                // clears slot
parentSubaccounts[subaccount] = bytes32(0);                  // clears parent link
// digestToSubaccount[digest] is NEVER cleared
// digestToMargin[digest]     is NEVER cleared
``` [1](#0-0) 

When a new isolated order `D2` is later submitted for the same `productId`, `createIsolatedSubaccount` searches `isolatedSubaccounts[parent][id]` for an existing slot. Because bit `id` was cleared, no existing subaccount is found, and a fresh one is allocated at the same `id`:

```solidity
newIsolatedSubaccount = bytes32(
    (uint256(uint160(senderAddress)) << 96) |
    (uint256(txn.productId) << 32) |
    (uint256(id) << 24) |          // same id reused
    6910831
);
``` [2](#0-1) 

The new subaccount `ISO_2` is byte-for-byte identical to the closed `ISO_1`. `digestToSubaccount[D1]` was never cleared, so it now silently points to the live `ISO_2`.

In `matchOrders`, the sender substitution reads this stale entry:

```solidity
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    taker.order.sender = digestToSubaccount[ordersInfo.taker.digest];
}
``` [3](#0-2) 

Old order `D1` (partially filled, still live in the sequencer's book) is now routed to `ISO_2` and its remaining fill is executed against `ISO_2`'s margin — a subaccount the user created for an entirely different order `D2`.

The `createIsolatedSubaccount` early-return guard also fails to protect against this:

```solidity
if (digestToSubaccount[digest] != bytes32(0)) {
    return digestToSubaccount[digest];   // returns stale closed subaccount
}
``` [4](#0-3) 

If `D1` is resubmitted as a `CreateIsolatedSubaccount` transaction, the function returns the stale (closed) `ISO_1` without transferring any margin, yet `matchOrders` will still route fills to it (or to `ISO_2` once it is recreated).

### Impact Explanation
A user who closes an isolated subaccount and opens a new one for the same product will have their new subaccount's margin consumed by fills of the old order. The old order's `filledAmounts` counter continues to advance against `ISO_2`'s balance. If the old order's remaining size is large relative to `ISO_2`'s margin, the health check `require(isHealthy(taker.order.sender))` may still pass (because `ISO_2` was funded for `D2`), resulting in an unintended position being opened in `ISO_2` at `D1`'s price. This constitutes direct, unintended asset movement out of the user's new isolated margin account. [5](#0-4) 

### Likelihood Explanation
The trigger requires: (1) a user creates an isolated order `D1` that is only partially filled, (2) the user closes the isolated subaccount before `D1` is fully filled or cancelled, and (3) the user opens a new isolated order for the same `productId`. All three steps are normal user actions reachable through supported endpoint transaction types (`CreateIsolatedSubaccount`, `CloseIsolatedSubaccount`, `MatchOrders`). The sequencer, acting in good faith, will continue to match `D1` because it remains valid in the order book. No privileged access is required. [6](#0-5) 

### Recommendation
In `_tryCloseIsolatedSubaccount`, iterate over all digests associated with the closing subaccount and delete their `digestToSubaccount` and `digestToMargin` entries. Alternatively, maintain a reverse mapping from `isolatedSubaccount → []digest` so that closure can efficiently clear all stale entries. At minimum, `createIsolatedSubaccount` should verify that the subaccount returned by the early-return guard is still active (i.e., `parentSubaccounts[digestToSubaccount[digest]] != bytes32(0)`) before returning it, and clear the stale entry if it is not. [7](#0-6) 

### Proof of Concept
1. User signs isolated order `D1` (productId `P`, amount 100, nonce 1). Sequencer submits `CreateIsolatedSubaccount` → `ISO_1` created at `id=0`; `digestToSubaccount[D1] = ISO_1`; margin `M` transferred from parent to `ISO_1`.
2. Sequencer partially fills `D1` (50 units). `filledAmounts[D1] = 50`.
3. User closes position in `ISO_1` (amount → 0). Sequencer submits `CloseIsolatedSubaccount` → `_tryCloseIsolatedSubaccount(ISO_1)` executes: mask bit 0 cleared, `isolatedSubaccounts[parent][0] = 0`, `parentSubaccounts[ISO_1] = 0`. **`digestToSubaccount[D1]` remains `= ISO_1`.**
4. User signs new isolated order `D2` (productId `P`, nonce 2). Sequencer submits `CreateIsolatedSubaccount` → loop finds no active subaccount for `P`; allocates `id=0` again; `ISO_2 = (address<<96)|(P<<32)|(0<<24)|6910831` — **identical bytes32 to `ISO_1`**; `digestToSubaccount[D2] = ISO_2`; margin `M2` transferred.
5. `digestToSubaccount[D1]` now resolves to the live `ISO_2`.
6. Sequencer matches `D1` (remaining 50 units) against a maker. `matchOrders` sets `taker.order.sender = digestToSubaccount[D1] = ISO_2`. Fill executes against `ISO_2`'s margin. `ISO_2` now holds an unintended 50-unit position at `D1`'s price, funded by `M2`. [8](#0-7)

    function _tryCloseIsolatedSubaccount(bytes32 subaccount) internal {
        uint32 productId = RiskHelper.getIsolatedProductId(subaccount);
        if (productId == 0) {
            return;
        }
        IPerpEngine.Balance memory balance = perpEngine.getBalance(
            productId,
            subaccount
        );
        if (balance.amount == 0) {
            uint8 id = RiskHelper.getIsolatedId(subaccount);
            address addr = address(uint160(bytes20(subaccount)));
            bytes32 parent = parentSubaccounts[subaccount];
            if (balance.vQuoteBalance != 0) {
                perpEngine.updateBalance(
                    productId,
                    subaccount,
                    0,
                    -balance.vQuoteBalance
                );
                perpEngine.updateBalance(
                    productId,
                    parent,
                    0,
                    balance.vQuoteBalance
                );
            }
            int128 quoteBalance = spotEngine
                .getBalance(QUOTE_PRODUCT_ID, subaccount)
                .amount;
            if (quoteBalance != 0) {
                spotEngine.updateBalance(
                    QUOTE_PRODUCT_ID,
                    subaccount,
                    -quoteBalance
                );
                spotEngine.updateBalance(
                    QUOTE_PRODUCT_ID,
                    parent,
                    quoteBalance
                );
            }
            isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);
            isolatedSubaccounts[parent][id] = bytes32(0);
            parentSubaccounts[subaccount] = bytes32(0);

            emit CloseIsolatedSubaccount(subaccount, parent);
        }
    }
