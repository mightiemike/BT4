### Title
Missing `max_cost` Enforcement in `op_ash` and `op_lsh` Allows Execution Beyond Cost Budget — (`File: src/more_ops.rs`)

### Summary

Both `op_ash` and `op_lsh` accept a `max_cost` parameter but deliberately ignore it (underscore-prefixed `_max_cost`). They perform the full bignum shift and heap allocation before any cost check can stop them. This is the direct analog to the external report's missing deviation check: a threshold that should gate execution is absent, so work proceeds that should have been rejected.

### Finding Description

Every other cost-sensitive operator in `src/more_ops.rs` calls `check_cost(cost, max_cost)?` inside its argument loop to fail early when the budget is exhausted. `op_ash` and `op_lsh` do not. Both signatures mark the parameter unused:

```rust
pub fn op_ash(a: &mut Allocator, input: NodePtr, _max_cost: Cost, _flags: ClvmFlags) -> Response {
``` [1](#0-0) 

```rust
pub fn op_lsh(a: &mut Allocator, input: NodePtr, _max_cost: Cost, _flags: ClvmFlags) -> Response {
``` [2](#0-1) 

Neither function contains a single `check_cost` call. The cost is computed only at the end and returned inside a `Reduction`. The `run_program` loop accumulates that returned cost and checks it at the **top of the next iteration**, not before the operation executes:

```rust
if cost > effective_max_cost {
    return Err(EvalErr::CostExceeded);
}
// ...
cost += match op {
    Operation::Apply => self.apply_op(cost, effective_max_cost - cost)?,
    // ...
};
``` [3](#0-2) 

`apply_op` passes the remaining budget to `self.dialect.op(...)`, but `op_ash`/`op_lsh` ignore it entirely. [4](#0-3) 

Contrast with `op_sha256`, which calls `check_cost` inside its argument loop: [5](#0-4) 

And `op_concat`, which also checks cost per argument: [6](#0-5) 

### Impact Explanation

An attacker-controlled CLVM program can craft a single `ash` or `lsh` call whose true cost exceeds the remaining budget. The full bignum shift (up to 65535 bits) and heap allocation for the result execute to completion before the cost overrun is detected. This violates the strict cost-budget invariant: a program performs more computation than its fee covers. In Chia block validation, where each block has a hard cost ceiling, this allows a transaction to consume more CPU and heap than permitted, undermining the fee/resource accounting model.

The shift bound (±65535 bits) caps the maximum over-budget work per call, but the invariant is still broken: the budget is not enforced at the point of execution.

### Likelihood Explanation

Any caller of `run_program` with a finite `max_cost` is affected. An attacker submits a CLVM program containing `(ash large_atom large_shift)` or `(lsh large_atom large_shift)` where the operation's cost exceeds the remaining budget. No special privileges are required — this is reachable via any standard CLVM execution path (Chia full node, Python wheel, or direct Rust API).

### Recommendation

Add `check_cost(cost, max_cost)?` inside both `op_ash` and `op_lsh` before returning, mirroring the pattern used by every other operator. The parameter names should be changed from `_max_cost` to `max_cost` and the check inserted after the cost is computed:

```rust
let cost = ASHIFT_BASE_COST + ((l0 + l1) as Cost) * ASHIFT_COST_PER_BYTE;
check_cost(cost, max_cost)?;  // add this
Ok(malloc_cost(a, cost, r))
```

### Proof of Concept

```
; Program: (ash (q . <8192-byte atom>) (q . 65535))
; With max_cost = 1000
;
; op_ash executes the full 65535-bit left shift of an 8192-byte atom,
; allocates ~16 KB for the result, computes cost >> 1000,
; returns Reduction(large_cost, result_node).
; run_program adds large_cost to the running total,
; then detects CostExceeded on the NEXT loop iteration —
; after the shift and allocation have already completed.
```

The attacker-controlled entry path is: submit crafted CLVM bytes → `run_program` → `apply_op` → `dialect.op` → `op_ash`/`op_lsh` with `_max_cost` ignored → full shift executes → cost overrun detected one iteration too late. [7](#0-6) [8](#0-7)
