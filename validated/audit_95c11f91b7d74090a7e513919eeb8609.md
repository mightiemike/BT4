### Title
secp256r1 Signature Malleability: `op_secp256r1_verify` Accepts High-S Signatures — (`src/secp_ops.rs`)

### Summary
`op_secp256r1_verify` uses `p256::ecdsa::Signature::from_slice` followed by `VerifyingKey::verify_prehash` with no low-s enforcement. The `p256` crate (unlike `k256`) does not normalize or reject high-s values. Because ECDSA is mathematically symmetric — if `(r, s)` verifies, so does `(r, n − s)` — both forms pass the operator, giving two distinct 64-byte byte strings that each produce `Ok(Reduction(1850000, nil))` for the same `(pubkey, message)` pair.

### Finding Description

In `src/secp_ops.rs`, `op_secp256r1_verify` parses the signature with:

```rust
let sig = P1Signature::from_slice(sig.as_ref()).map_err(|_| { ... })?;
let result = verifier.verify_prehash(msg.as_ref(), &sig);
``` [1](#0-0) 

There is no call to `Signature::normalize_s()`, no `s > n/2` rejection, and no low-s flag anywhere in the production secp code (confirmed: zero matches for `normalize_s`, `low_s`, `lowS` in `src/**/*.rs`).

The `p256` crate at version `0.14.0-rc` (the version pinned in `Cargo.toml`) implements P-256 for general-purpose use (TLS, FIDO2, etc.) and deliberately does **not** enforce low-s in `verify_prehash`. This contrasts with `k256`, which enforces low-s for Bitcoin compatibility. The `p256` crate's `hazmat::verify_prehash` performs the raw ECDSA equation check without any s-normalization step. [2](#0-1) 

The ECDSA math confirms the malleability: for signature `(r, s)` over curve order `n`, the negated point `−R` has the same x-coordinate as `R`, so `(r, n − s)` satisfies the verification equation identically. Both byte strings are structurally valid 64-byte fixed-size signatures that `from_slice` accepts, and both pass `verify_prehash`.

The `op_secp256k1_verify` counterpart in the same file uses `k256`, which **does** enforce low-s in its verifier — so the malleability is asymmetric: k1 is protected, r1 is not. [3](#0-2) 

### Impact Explanation

For any CLVM puzzle that uses `secp256r1_verify` (opcode 65 under `ENABLE_SECP_OPS`, or the softfork-guarded variant), an attacker who observes a valid spend bundle can produce a second, semantically identical spend bundle by replacing the 64-byte signature with its high-s complement. The two bundles have different serialized bytes and therefore different spend-bundle IDs. Concrete effects:

- **Mempool**: Both bundles are independently valid; a node that receives both before either is confirmed may admit both, wasting mempool slots and causing deduplication failures.
- **Wallet / indexer tracking**: Any system that tracks a pending spend by its bundle ID can be confused — the original ID may never confirm while the mutated one does, or vice versa.
- **Puzzle-level replay / uniqueness assumptions**: Puzzles that embed a secp256r1 signature as a unique nonce or use it to derive a coin ID will have their uniqueness invariant broken. [4](#0-3) 

### Likelihood Explanation

The attacker needs only: (1) observe a valid spend bundle containing a `secp256r1_verify` puzzle, (2) extract the 64-byte signature, (3) compute `s' = n − s` (trivial modular arithmetic over the P-256 order), (4) resubmit. No key material, no privileged access, no special CLVM knowledge beyond the opcode layout is required. The operation is O(1) and requires no cryptographic secrets.

### Recommendation

Add an explicit low-s check immediately after `P1Signature::from_slice`:

```rust
let sig = P1Signature::from_slice(sig.as_ref()).map_err(|_| { ... })?;
// Enforce low-s canonical form
if sig.normalize_s().is_some() {
    Err(EvalErr::InvalidOpArg(input,
        "secp256r1_verify: signature is not in low-s form".to_string()))?;
}
```

`Signature::normalize_s()` returns `Some(normalized)` when `s > n/2` and `None` when already canonical. Returning an error (rather than silently normalizing) ensures exactly one byte string is accepted per `(pubkey, message)` pair, matching the behavior of `k256` and eliminating the malleability surface.

### Proof of Concept

```rust
use p256::{
    ecdsa::{Signature, SigningKey, VerifyingKey},
    ecdsa::signature::hazmat::PrehashSigner,
    ecdsa::signature::hazmat::PrehashVerifier,
};

let sk = SigningKey::random(&mut rand::thread_rng());
let vk = VerifyingKey::from(&sk);
let msg = [0u8; 32]; // 32-byte prehash

// Sign and obtain low-s signature
let (sig_low, _): (Signature, _) = sk.sign_prehash(&msg).unwrap();
let (r, s) = (sig_low.r(), sig_low.s());

// Compute high-s variant: s' = n - s
let n = p256::NistP256::ORDER;
let s_high = n - s.as_ref(); // scalar subtraction mod n
let sig_high = Signature::from_scalars(r, s_high).unwrap();

// Both verify successfully — demonstrating the malleability
assert!(vk.verify_prehash(&msg, &sig_low).is_ok());
assert!(vk.verify_prehash(&msg, &sig_high).is_ok()); // passes — no low-s check
assert_ne!(sig_low.to_bytes(), sig_high.to_bytes()); // different bytes, same validity
```

Running both signatures through `op_secp256r1_verify` (via `run_serialized_chia_program` or the test harness) will return `Ok(Reduction(1850000, nil))` for both, confirming that two distinct spend bundles are accepted for the same coin spend.

pub fn op_secp256r1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256R1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256r1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256r1_verify pubkey")?;
    let verifier = P1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256r1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is sha256 hash of message
    let msg = atom(a, msg, "secp256r1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(

        ))?;
    }

        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

        Err(EvalErr::Secp256Failed(input))?

    }
