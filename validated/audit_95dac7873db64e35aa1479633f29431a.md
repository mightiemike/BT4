### Title
Empty `chain_id` Silently Produces `Felt::ZERO` in Transaction Hash Domain — (`crates/starknet_api/src/core.rs`, `crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary
`ChainId::Other("")` is a valid, deserializable value accepted everywhere a `ChainId` is consumed. When an empty string is used, `ChainId::as_hex()` returns `"0x"`, and `Felt::from_hex("0x")` resolves to `Felt::ZERO`. Every transaction hash computed by `TransactionConverter` then embeds `Felt::ZERO` as the chain-id domain separator instead of the intended network identifier, collapsing the cross-chain replay protection that the chain-id field is designed to provide.

### Finding Description

`ChainId` is an enum whose `Other(String)` arm accepts any string without length or content validation: [1](#0-0) 

The hex-encoding path for an empty string produces `"0x"`: [2](#0-1) 

The `TryFrom<&ChainId> for Felt` conversion then calls `Felt::from_hex("0x")`, which resolves to `Felt::ZERO` — no error is returned: [3](#0-2) 

`TransactionConverter::new()` stores the caller-supplied `chain_id` without any non-empty check: [4](#0-3) 

This stored `chain_id` is then passed directly into `calculate_transaction_hash` for every incoming transaction: [5](#0-4) 

The hash functions for all transaction types (invoke v1/v3, declare v0/v1/v2/v3, deploy-account v1/v3, L1-handler) embed the chain-id felt as a domain separator: [6](#0-5) [7](#0-6) 

The production deployment template ships with `"chain_id": ""` as the default placeholder value, meaning any deployment that omits the override will instantiate the converter with an empty chain-id: [8](#0-7) 

Neither the stateless transaction validator nor the stateful validator checks that the node's configured chain-id is non-empty before transactions are hashed and admitted: [9](#0-8) 

### Impact Explanation

When `chain_id` is empty the chain-id component of every transaction hash becomes `Felt::ZERO`. Two concrete consequences follow:

1. **Wrong hash stored and served.** The `InternalRpcTransaction.tx_hash` field, which is the authoritative hash returned by the RPC `add_transaction` endpoint and stored in the MDBX database, is computed with the wrong domain. The RPC returns an authoritative-looking but incorrect value for every transaction.

2. **Cross-chain replay.** A transaction signed on any real network (Mainnet, Sepolia) carries a signature over a hash that includes the real chain-id felt. On a node with empty chain-id the hash is recomputed with `Felt::ZERO`, so the signature check runs against a different preimage. Conversely, a transaction crafted specifically for `chain_id = Felt::ZERO` will pass signature validation on any such node regardless of the intended network.

Both consequences match the allowed impact scope: *"Wrong state, receipt, event … or revert result"* and *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

### Likelihood Explanation

The default deployment template sets `chain_id` to the empty string. Any operator who deploys without explicitly overriding this value — or who passes an empty string through an environment variable or config file — will silently activate the broken domain. No attacker capability beyond submitting a normal transaction is required once the node is in this state; the misconfiguration is the only prerequisite.

### Recommendation

1. **Reject empty `chain_id` at construction.** Add a non-empty check in `TransactionConverter::new()` and in `ChainId::from(String)` (or at the deserialization boundary) that returns an error when the string is empty.

2. **Add config-level validation.** Annotate `chain_id` fields in `GatewayStaticConfig`, `DbConfig`, and related structs with a validator that rejects the empty-string value, consistent with how other fields use `#[validate(range(min = 1))]`.

3. **Change the default placeholder.** Replace `"chain_id": ""` in all template configs with a sentinel that cannot be mistaken for a valid chain-id (e.g., remove the key entirely and require explicit provision, or use a clearly invalid value that fails deserialization).

### Proof of Concept

```
1. Deploy a sequencer node using the unmodified general_config.json
   (chain_id = "").

2. Submit any valid V3 invoke transaction signed for SN_MAIN.

3. The gateway calls TransactionConverter::convert_rpc_tx_to_internal_rpc_tx().

4. Inside convert_rpc_tx_to_internal():
     tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)
   where self.chain_id = ChainId::Other("").

5. calculate_transaction_hash calls Felt::try_from(&ChainId::Other(""))
     → as_hex() = "0x" + hex::encode("") = "0x"
     → Felt::from_hex("0x") = Felt::ZERO

6. The Poseidon/Pedersen hash chain includes Felt::ZERO as the chain-id
   element instead of the SN_MAIN felt (0x534e5f4d41494e).

7. The returned InternalRpcTransaction carries a tx_hash computed over
   the wrong preimage; the RPC endpoint returns this wrong hash to the
   caller, and it is persisted to MDBX under the wrong key.
```
