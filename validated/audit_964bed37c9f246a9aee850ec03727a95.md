### Title
Anyone Can Prune Mainchain Block Headers via Unguarded `run_mainchain_gc` - (File: contract/src/lib.rs)

### Summary
`run_mainchain_gc` is a public, state-destructive function that permanently removes block headers from the contract's canonical chain state. It carries only a `#[pause]` decorator, meaning any unprivileged NEAR account can call it freely when the contract is not paused. This is a direct analog to HAL-06: just as any caller could invoke `burn()` to destroy token balances without authorization, any caller can invoke `run_mainchain_gc` to destroy stored block headers without authorization.

### Finding Description
`submit_blocks` is correctly gated behind `#[trusted_relayer]`, restricting block submission to authorized relayers. However, `run_mainchain_gc` — which permanently deletes entries from `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height`, and advances `mainchain_initial_blockhash` — carries no equivalent restriction:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
``` [1](#0-0) 

The `#[pause]` decorator only blocks calls when the contract is explicitly paused; it imposes no caller identity check during normal operation. Any NEAR account — with no staked deposit, no granted role, and no relationship to the relayer infrastructure — can call this function.

The deletion logic removes entries from all three canonical-chain maps and permanently advances `mainchain_initial_blockhash`:

```rust
self.remove_block_header(blockhash);
self.mainchain_height_to_header.remove(&height);
...
self.mainchain_initial_blockhash = self
    .mainchain_height_to_header
    .get(&end_removal_height)
    ...
``` [2](#0-1) 

`remove_block_header` deletes from both `mainchain_header_to_height` and `headers_pool`: [3](#0-2) 

The amount removed per call is bounded by `amount_of_headers_we_store - gc_threshold`, but passing `batch_size = u64::MAX` causes the maximum eligible pruning to execute in a single transaction: [4](#0-3) 

By contrast, `submit_blocks` — which internally calls `run_mainchain_gc` — is correctly restricted: [5](#0-4) 

### Impact Explanation
Once `run_mainchain_gc` is called by an attacker:

1. Block headers are permanently deleted from `headers_pool` and both mainchain index maps — this state cannot be recovered without a full re-submission of those headers.
2. `mainchain_initial_blockhash` is permanently advanced, so the contract's view of its own history is corrupted.
3. `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` will panic with `"block does not belong to the current main chain"` for any transaction whose containing block was pruned — SPV proofs that were valid before the attack become permanently unverifiable.
4. `get_block_hash_by_height` and `get_height_by_block_hash` return `None` for all pruned heights/hashes, breaking any downstream consumer that relies on historical lookups. [6](#0-5) 

### Likelihood Explanation
The entry path requires no privileges, no staked deposit, and no special knowledge beyond knowing the contract account ID. The attacker only needs to wait until the mainchain grows beyond `gc_threshold` (which happens automatically as the relayer submits blocks), then call `run_mainchain_gc` with a large `batch_size`. This is a single, cheap NEAR function call. The `gc_threshold` is documented as `52704` (approximately one year of Bitcoin blocks), so the window opens after roughly one year of operation — but once open, the attack is trivially executable at any time. [7](#0-6) 

### Recommendation
Restrict `run_mainchain_gc` to authorized callers using the same `#[trusted_relayer]` attribute applied to `submit_blocks`, or add an explicit role check (e.g., `Role::DAO` or a dedicated `GCManager` role) using `acl_is_super_admin` / `has_role`. The internal call from `submit_blocks` is already authorized by the `#[trusted_relayer]` guard on that function; the public entrypoint is the only path that needs to be closed.

### Proof of Concept
```
// Attacker is any unprivileged NEAR account.
// Precondition: mainchain has grown beyond gc_threshold (e.g., 52704 blocks).

// Step 1: Confirm current mainchain size exceeds gc_threshold.
let size = contract.view("get_mainchain_size").await;
assert!(size > gc_threshold);

// Step 2: Attacker calls run_mainchain_gc with maximum batch_size.
attacker_account
    .call(contract.id(), "run_mainchain_gc")
    .args_json(json!({"batch_size": u64::MAX}))
    .transact()
    .await
    .expect("succeeds — no access control");

// Step 3: Mainchain is now pruned to exactly gc_threshold blocks.
// All pruned block headers are permanently gone from headers_pool.
// verify_transaction_inclusion for any pruned block now panics:
//   "block does not belong to the current main chain"
```

The existing integration test at `contract/tests/test_basics.rs:515-521` already demonstrates that `user_account` (an unprivileged account) can successfully call `run_mainchain_gc` — confirming the absence of any caller restriction. [8](#0-7)
