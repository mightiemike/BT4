### Title
`DISABLE_OP` Flag in `MEMPOOL_MODE` Incorrectly Disables `op_modpow` (Opcode 60), Creating Mempool/Consensus Divergence — (File: `src/chia_dialect.rs`)

---

### Summary

`MEMPOOL_MODE` includes the undocumented `DISABLE_OP` flag, which completely disables `op_modpow` (opcode 60) in mempool validation. Because consensus validation does not set this flag, programs using `op_modpow` are **rejected by the mempool** but **accepted by consensus**. This is a flag/operator wiring error that creates a mempool/consensus divergence: a class of valid on-chain programs cannot be submitted through the mempool, and the divergence can be exploited to bypass mempool-level validation.

---

### Finding Description

In `src/chia_dialect.rs`, the `MEMPOOL_MODE` constant bundles `DISABLE_OP` (0x200) with the other mempool restrictions:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)          // ← undocumented flag
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [1](#0-0) 

The `DISABLE_OP` flag has **no documentation comment**, unlike every other flag in the same `bitflags!` block. Its sole concrete effect in the production dispatch path is to hard-block opcode 60 (`op_modpow`):

```rust
60 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;
    }
    op_modpow
}
``` [2](#0-1) 

Opcode 60 is a fully wired, named operator in the `ChiaDialect` dispatch table. It is **not** gated by any hard-fork or softfork flag — it is unconditionally available in consensus mode (i.e., when `DISABLE_OP` is absent). The `DISABLE_OP` flag is the only thing that blocks it, and that flag is only set in `MEMPOOL_MODE`.

The secondary uses of `DISABLE_OP` in `op_div`, `op_divmod`, and `op_mod` (and their malachite variants) add a check `a0_len > 2048`, but this is **dead code**: the immediately following unconditional check `a0_len > 256` would already have rejected the input. So `DISABLE_OP` has no real effect on those operators. [3](#0-2) 

---

### Impact Explanation

The broken invariant is: **the mempool rejects programs that consensus accepts**. Specifically:

- Any CLVM program containing opcode 60 (`modpow`) is rejected by the mempool with `EvalErr::Unimplemented`.
- The same program is accepted and executed normally by consensus validation (which does not set `DISABLE_OP`).

This has two concrete consequences:

1. **Denial of service for legitimate users**: A coin whose puzzle uses `op_modpow` cannot be spent through the mempool. The spend bundle will be rejected with an `unimplemented operator` error even though the spend is perfectly valid on-chain. Users are silently locked out of a valid operator.

2. **Mempool bypass for attackers**: An attacker who controls a farmer (block producer) can include a spend bundle using `op_modpow` directly in a block, bypassing all mempool-level validation (fee checks, rate limits, `NO_UNKNOWN_OPS` enforcement, heap limits). The block will be accepted by all consensus nodes. This is the direct analog to the external report: the "pause" (`DISABLE_OP` in `MEMPOOL_MODE`) blocks a critical operation (`op_modpow`) that should remain reachable, and an adversary can exploit the gap between the paused state and the resumed state (consensus) to act without the normal gating.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: the attacker supplies arbitrary CLVM bytes. Opcode 60 is a single-byte opcode. Any program containing the byte `0x3c` as an operator triggers the divergence. A farmer-controlling attacker can exploit this without any special privileges beyond block production rights, which is a realistic threat model in Chia.

---

### Recommendation

Either:

- **Remove `DISABLE_OP` from `MEMPOOL_MODE`** if `op_modpow` is intended to be a live consensus operator (which the dispatch table implies). The mempool should not reject programs that consensus accepts.
- **Or gate `op_modpow` behind a proper hard-fork flag** (like `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` or `ENABLE_SHA256_TREE`) with a matching documentation comment, so that both mempool and consensus agree on its availability.

Add a documentation comment to `DISABLE_OP` explaining its intended semantics and activation conditions, consistent with all other flags in the block. [4](#0-3)
