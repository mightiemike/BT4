Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of end user, allowing any user to bypass per-pool swap allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the pool receives the router as `msg.sender`, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][end_user]`. Any pool admin who allowlists the router to enable standard-periphery access inadvertently grants every user â€” including those not individually allowlisted â€” the ability to swap on a curated pool.

## Finding Description
The call chain is fully confirmed in production code:

1. `MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

2. `ExtensionCalling._beforeSwap` forwards that value unchanged as the first argument to the extension: [2](#0-1) 

3. `SwapAllowlistExtension.beforeSwap` uses that `sender` value to look up the per-pool allowlist: [3](#0-2) 

4. `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly from the router's address, making the router `msg.sender` at the pool level: [4](#0-3) 

The same wrong-actor binding applies to `exactInput`, `exactOutputSingle`, and `exactOutput`, all of which call `pool.swap()` from the router: [5](#0-4) 

The extension has no mechanism to recover the true end-user identity. The `extensionData` bytes are passed through but the extension ignores them entirely: [6](#0-5) 

No existing guard compensates for this. The `allowAllSwappers` flag is a separate escape hatch and does not fix the per-user check. The pool admin has no way to simultaneously (a) allowlist the router so that router-mediated swaps can proceed and (b) enforce per-user restrictions through this extension.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` as a `beforeSwap` hook intends to restrict live swaps to a curated set of addresses. Once the pool admin allowlists the router â€” the only way to enable standard-periphery access â€” the allowlist is structurally nullified for all router-mediated swaps. Non-allowlisted users can execute swaps on a curated pool, directly violating the pool's curation policy. This constitutes broken core pool functionality (the allowlist hook does not enforce its stated invariant) and an admin-boundary break via an unprivileged path: an unprivileged trader bypasses a pool-admin-configured access control by routing through the standard periphery.

## Likelihood Explanation
The trigger requires only that the pool admin has allowlisted the router, which is the routine operational step for any pool that intends to support the standard periphery. Once that step is taken, any user can call `exactInputSingle` or any other router entry point with no special privilege, flash loan, or multi-step setup. The condition is self-fulfilling: the very act of enabling router access enables the bypass.

## Recommendation
The extension must gate on the actual end user, not the direct pool caller. The cleanest options are:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and verifies it against the allowlist. This requires the router to be trusted to supply honest data and requires coordination between router and extension.
2. **Move the allowlist check into the pool's core `swap()` path**: The check always sees the true `msg.sender` regardless of the extension system, eliminating the indirection problem entirely.
3. **Signed-permit pattern**: The end user's identity is cryptographically bound to the swap call, making it unforgeable regardless of who the direct caller is.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension as beforeSwap hook.
  - Pool admin calls setAllowedToSwap(pool, router, true)   // allowlist the router
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  - attacker calls MetricOmmSimpleRouter.exactInputSingle({
        pool: pool,
        recipient: attacker,
        ...
    })
  - Router calls pool.swap(recipient=attacker, ...)         // router is msg.sender
  - Pool calls _beforeSwap(sender=router, ...)
  - Extension checks allowedSwapper[pool][router] â†’ true   âś“
  - Swap executes; attacker is not individually allowlisted but swaps successfully.

Expected: revert NotAllowedToSwap
Actual:   swap succeeds
```

Foundry test outline:
1. Deploy `SwapAllowlistExtension`, configure a pool with it as `beforeSwap` hook.
2. Call `setAllowedToSwap(pool, router, true)` from pool admin; do not allowlist `attacker`.
3. From `attacker`, call `MetricOmmSimpleRouter.exactInputSingle(...)`.
4. Assert the call succeeds (demonstrating the bypass) and that a direct `pool.swap()` call from `attacker` reverts with `NotAllowedToSwap`.
