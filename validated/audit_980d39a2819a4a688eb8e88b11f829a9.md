Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router's address as `sender` instead of the real end-user, allowing any caller to bypass the swap allowlist on curated pools — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the value forwarded by `MetricOmmPool._beforeSwap(msg.sender, ...)`. Because `MetricOmmPool.swap` binds `sender` to its own `msg.sender` — which is `MetricOmmSimpleRouter` when the router initiates the swap — the extension checks the router's allowlist entry, not the real end-user's. Any pool admin who allowlists the router to enable standard periphery trading inadvertently grants every address unrestricted swap access to the curated pool.

## Finding Description

**Root cause — `MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`:** [1](#0-0) 

When `MetricOmmSimpleRouter` calls `pool.swap(...)`, `msg.sender` inside the pool is the router's address, not the end-user.

**`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router `msg.sender` to the pool:** [2](#0-1) 

The same pattern applies to `exactInput`: [3](#0-2) 

**`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the router:** [4](#0-3) 

`msg.sender` inside the extension is the pool (correct), but `sender` is the router. The check `allowedSwapper[msg.sender][sender]` therefore evaluates to `allowedSwapper[pool][router]`. The real end-user's address is never forwarded to the extension at any point in the call chain.

**Exploit call chain:**
```
User → MetricOmmSimpleRouter.exactInputSingle()
         → IMetricOmmPoolActions(pool).swap(recipient, ...)
              msg.sender to pool = MetricOmmSimpleRouter address
         → MetricOmmPool._beforeSwap(msg.sender=router, ...)
         → SwapAllowlistExtension.beforeSwap(sender=router, ...)
              checks allowedSwapper[pool][router]  ← wrong actor
              passes if router is allowlisted
```

No existing guard prevents this: the extension has no mechanism to distinguish the router acting on behalf of an unauthorized user from the router acting on behalf of an authorized one.

## Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` intends to restrict swaps to a specific set of addresses (e.g., KYC-verified counterparties). To allow those users to trade via the canonical periphery, the admin must call `setAllowedToSwap(pool, address(router), true)`. Once the router is allowlisted, **any address** — including those explicitly excluded — can call `MetricOmmSimpleRouter.exactInputSingle` and execute swaps against the restricted pool. The allowlist guard is completely nullified for all router-mediated swaps. This constitutes a broken core pool functionality (access control) causing direct policy violation and exposure of LP funds to trades the pool admin explicitly intended to block.

## Likelihood Explanation

- `MetricOmmSimpleRouter` is the canonical, documented periphery swap path; any pool admin enabling user-facing trading will allowlist it.
- No special privilege or setup is required from the attacker: any EOA can call `exactInputSingle` on the router.
- The bypass is deterministic, requires no timing, oracle manipulation, or multi-step preparation, and is repeatable indefinitely.

## Recommendation

The extension must gate on the economically relevant actor — the end-user — not the intermediate contract. Two complementary fixes:

1. **Pass the original user through the router.** Store `msg.sender` (the real user) in transient storage at the router entry point (the router already uses transient storage for the callback payer via `_setNextCallbackContext`) and expose it as a separate `originator` argument in the pool's `swap` signature, forwarding it to all extension hooks alongside `sender`.

2. **Check `recipient` or a separately forwarded originator in the extension.** Until the core is updated, the extension can require that `extensionData` carries the real user's address and verify it against the allowlist, though this is fragile and requires router cooperation.

The cleanest fix is option 1: add an `originator` field to the swap call path so `SwapAllowlistExtension.beforeSwap` can check `allowedSwapper[pool][originator]` instead of `allowedSwapper[pool][sender]`.

## Proof of Concept

```solidity
// Setup:
//   pool configured with SwapAllowlistExtension
//   pool admin calls: swapAllowlist.setAllowedToSwap(pool, address(router), true)
//   attacker = address NOT in allowedSwapper[pool]

// Step 1: attacker calls router directly
// extension sees sender=router → allowedSwapper[pool][router]=true → passes
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool:             address(pool),
        tokenIn:          token0,
        recipient:        attacker,
        zeroForOne:       true,
        amountIn:         1_000e18,
        amountOutMinimum: 0,
        priceLimitX64:    type(uint128).max,
        deadline:         block.timestamp + 1,
        extensionData:    ""
    })
);
// ✓ swap succeeds — attacker received token1 despite not being allowlisted

// Step 2: attacker calls pool directly
// extension sees sender=attacker → allowedSwapper[pool][attacker]=false → reverts
pool.swap(attacker, true, int128(1_000e18), type(uint128).max, "", "");
// ✗ reverts with NotAllowedToSwap
```

Step 1 confirms the bypass: the allowlist check passes because `sender` is the router (allowlisted), not the attacker. Step 2 confirms the allowlist exists and is correctly enforced for direct calls, proving the bypass is structural and specific to router-mediated swaps.
