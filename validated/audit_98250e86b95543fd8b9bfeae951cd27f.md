### Title
Storage keys written under `max_length_storage_key=4194304` become permanently inaccessible after protocol version 53 reduces the limit to 2048 — (`runtime/near-vm-runner/src/logic/logic.rs`)

### Summary
Protocol version 53 reduces `max_length_storage_key` from 4,194,304 to 2,048 bytes. Both `storage_write` and `storage_read` enforce this limit against the caller-supplied key length at call time, with no grandfathering for keys already stored in the trie. Any contract that wrote a key of length 2,049–4,194,304 bytes before the upgrade can never read, overwrite, or remove that key after the upgrade, making the associated contract state permanently inaccessible.

### Finding Description
`VMLogic::storage_read` (and `storage_remove`, `storage_has_key`) unconditionally compare the caller-supplied key length against `self.config.limit_config.max_length_storage_key` and return `HostError::KeyLengthExceeded` before touching the trie if the key exceeds the current limit. [4](#0-3) 

The `LimitConfig::max_length_storage_key` field is a plain `u64` that is set per-protocol-version from the parameter store. [5](#0-4) 

Protocol version 53 sets `max_length_storage_key: { old: 4_194_304, new: 2_048 }`. [1](#0-0) 

No migration, no trie cleanup, and no read-path exemption for pre-existing oversized keys is present anywhere in the diff or in the runtime.

### Impact Explanation
Any contract that stored a key of length 2,049–4,194,304 bytes before the protocol 53 upgrade loses the ability to read, update, or delete that key after the upgrade. Funds or state locked behind such a key (e.g., a balance map keyed by a long identifier, or a contract that uses large binary keys) are permanently frozen. The trie entry remains, consuming storage rent, but is unreachable through any host function.

### Likelihood Explanation
The 4 MiB limit was the default from genesis. Any contract deployed before protocol version 53 that used keys longer than 2,048 bytes is affected. The window between genesis and protocol 53 is long enough that real contracts on mainnet or testnet could have written such keys. The scenario requires no special privileges — any unprivileged user can deploy a contract and call `storage_write` with a long key.

### Recommendation
The `storage_read`, `storage_remove`, and `storage_has_key` host functions should not apply the `max_length_storage_key` limit to read/remove operations. The limit is a write-time invariant (preventing new oversized keys from being created), not a read-time invariant. Alternatively, a migration should be run at protocol version 53 activation to enumerate and remove all trie keys exceeding 2,048 bytes, or the limit reduction should be accompanied by a grace-period protocol flag that allows reads of pre-existing oversized keys.

### Proof of Concept
A differential upgrade test would:
1. Instantiate `VMLogic` with `max_length_storage_key = 4_194_304` (pre-v53 config).
2. Call `storage_write` with a 3,000-byte key and any value — this succeeds.
3. Re-instantiate `VMLogic` over the same trie state with `max_length_storage_key = 2_048` (post-v53 config).
4. Call `storage_read` with the same 3,000-byte key.
5. Assert the read returns `1` (value found) — it instead returns `Err(KeyLengthExceeded { length: 3000, limit: 2048 })`.

The existing test `test_key_length_limit` in `runtime/near-vm-runner/src/logic/tests/miscs.rs` already demonstrates that `storage_read` enforces the limit; the missing piece is the cross-version scenario where the key was written under the old limit. [6](#0-5)

#[test]
fn test_key_length_limit() {
    let mut logic_builder = VMLogicBuilder::default();
    let limit = 1024;
    logic_builder.config.limit_config.max_length_storage_key = limit;
    let mut logic = logic_builder.build();

    // Under the limit. Valid calls.
    let key = crate::logic::MemSlice { ptr: 0, len: limit };
    let val = crate::logic::MemSlice { ptr: 0, len: 5 };
    logic
        .storage_has_key(key.len, key.ptr)
        .expect("storage_has_key: key length is under the limit");
    logic
        .storage_write(key.len, key.ptr, val.len, val.ptr, 0)
        .expect("storage_write: key length is under the limit");
    logic.storage_read(key.len, key.ptr, 0).expect("storage_read: key length is under the limit");
    logic
        .storage_remove(key.len, key.ptr, 0)
        .expect("storage_remove: key length is under the limit");

    // Over the limit. Invalid calls.
    let key = crate::logic::MemSlice { ptr: 0, len: limit + 1 };
    assert_eq!(
        logic.storage_has_key(key.len, key.ptr),
        Err(HostError::KeyLengthExceeded { length: key.len, limit }.into())
    );
    assert_eq!(
        logic.storage_write(key.len, key.ptr, val.len, val.ptr, 0),
        Err(HostError::KeyLengthExceeded { length: key.len, limit }.into())
    );
    assert_eq!(
        logic.storage_read(key.len, key.ptr, 0),
        Err(HostError::KeyLengthExceeded { length: key.len, limit }.into())
    );
    assert_eq!(
        logic.storage_remove(key.len, key.ptr, 0),
        Err(HostError::KeyLengthExceeded { length: key.len, limit }.into())
    );
}
