### Title
`compute_meta_tx_v0_hash` Omits Nonce, Enabling Signature Replay of Meta-Transactions — (`crates/blockifier/src/execution/syscalls/syscall_base.rs`, `crates/starknet_api/src/transaction_hash.rs`, `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `meta_tx_v0` syscall computes a transaction hash that does not include a nonce. The resulting hash is identical for any two invocations with the same `(contract_address, entry_point_selector, calldata, chain_id)` tuple. Because the OS also hardcodes `nonce = 0` in the synthetic `TxInfo` and skips `check_and_increment_nonce` for version-0 transactions, there is no on-chain state that prevents the same user-provided signature from being accepted an unlimited number of times across different outer transactions.

---

### Finding Description

The `meta_tx_v0` syscall is designed to allow gasless meta-transactions: a user signs a version-0 invoke hash and a relayer submits it inside an outer transaction. The hash is computed in two places that must agree:

**Cairo OS (`compute_meta_tx_v0_hash`):**

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=0,          // ← no nonce
        additional_data=cast(0, felt*),
    );
    return tx_hash;
}

The combined effect:

| Property | Value |
|---|---|
| Hash inputs | `INVOKE_PREFIX ‖ 0 ‖ contract_address ‖ selector ‖ H(calldata) ‖ 0 ‖ chain_id` |
| Nonce in hash | **absent** |
| `tx_info.nonce` exposed to `__execute__` | hardcoded `0` |
| OS nonce increment | skipped for version 0 |
| `__validate__` | skipped for version 0 |

Because the hash is purely a function of `(contract_address, calldata, chain_id)`, any signature that was valid once is valid forever for the same calldata.

---

### Impact Explanation

An attacker who observes a valid `(calldata, signature)` pair for a meta-tx (e.g., from a prior on-chain outer transaction) can replay it in a new outer transaction. The target account's `__execute__` is invoked again with the same authorized signature and the same `transaction_hash`. If `__execute__` transfers tokens, approves allowances, or performs any state-changing action, the attacker can repeat it an unlimited number of times.

The account contract cannot defend itself using the standard `tx_info.nonce` field because it is always `0`. Custom nonce storage inside calldata is possible but is not part of the standard account interface and is not enforced by the protocol.

This matches the **Critical** impact: *Invalid or unauthorized Starknet transaction accepted through account validation, signature, nonce, chain id, fee/resource bound, paymaster, or account-deployment logic.*

---

### Likelihood Explanation

- The `meta_tx_v0` syscall is callable by any Cairo 1 contract from within an executing transaction — no privilege is required.
- Signatures are visible on-chain once the first outer transaction is included in a block.
- The replay requires only constructing a new outer transaction that calls `meta_tx_v0_syscall` with the same `(address, selector, calldata, signature)` — a trivial operation.
- Any account contract that relies on `tx_info.nonce` or `tx_info.transaction_hash` for replay protection (the standard pattern) is fully vulnerable.

---

### Recommendation

Include a nonce in the meta-tx hash. The nonce can be sourced from:

1. **Account storage nonce**: read the target contract's current nonce from `contract_state_changes` before computing the hash, include it in `additional_data`, and increment it after execution. This mirrors how `compute_l1_handler_transaction_hash` includes the L1 nonce via `additional_data_size=1, additional_data=&nonce`.

2. **Caller-supplied nonce in calldata**: require the calldata to contain a nonce field and include it in the hash, with the OS verifying and consuming it.

The `compute_l1_handler_transaction_hash` function already demonstrates the correct pattern:

```cairo
let (transaction_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
    ...
    additional_data_size=1,
    additional_data=&nonce,   // ← nonce included
);
``` [6](#0-5) 

Apply the same pattern to `compute_meta_tx_v0_hash` and its Rust counterpart in `syscall_base.rs`.

---

### Proof of Concept

1. Deploy an account contract `A` at address `addr_A` whose `__execute__` transfers 1 token to the caller when `tx_info.signature` is valid over `tx_info.transaction_hash`.

2. User signs: `sig = ECDSA.sign(H(INVOKE, 0, addr_A, __execute__, H([transfer_calldata]), 0, chain_id))`.

3. User submits outer transaction `T1` (nonce=5) calling `meta_tx_v0(addr_A, __execute__, [transfer_calldata], sig)`. Token is transferred. `T1` is accepted and nonce advances to 6.

4. Attacker submits outer transaction `T2` (nonce=0, from attacker's account) calling `meta_tx_v0(addr_A, __execute__, [transfer_calldata], sig)`.

5. The blockifier computes:
   ```rust
   InvokeTransactionV0 { max_fee: Fee(0), signature: sig, contract_address: addr_A,
       entry_point_selector: __execute__, calldata: [transfer_calldata] }
   .calculate_transaction_hash(&chain_id, &TransactionVersion::ZERO)
   ```
   This produces the **identical hash** as step 2 because no nonce is included.

6. `__execute__` on `addr_A` receives `tx_info.transaction_hash == original_hash` and `tx_info.signature == sig`. Signature verification passes. Token is transferred again.

7. Repeat step 4 indefinitely to drain `addr_A`. [7](#0-6) [2](#0-1)
