### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing 64-Byte Merkle Proof Forgery Mitigation â€” (`File: contract/src/lib.rs`)

---

### Summary

The contract exposes two SPV proof verification endpoints. The original `verify_transaction_inclusion` was deprecated in v0.5.0 because it is vulnerable to the 64-byte transaction Merkle proof forgery attack. The replacement `verify_transaction_inclusion_v2` adds a coinbase Merkle proof check to close that attack vector. However, the deprecated function was never removed or access-controlled â€” it remains `pub` and callable by any unprivileged NEAR account. An attacker can call it directly, bypassing the coinbase proof check entirely, and obtain a `true` verification result for a transaction that does not exist.

---

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` but retains `pub` visibility and is gated only by `#[pause]` â€” the same gate as the secure v2 endpoint. [1](#0-0) 

The function performs only a raw Merkle root comparison with no coinbase anchor: [2](#0-1) 

The v2 replacement closes the attack by first verifying a coinbase Merkle proof against the same root, which forces the proof depth to match a real transaction tree: [3](#0-2) 

Because `verify_transaction_inclusion` is still `pub`, any caller can skip v2 entirely and call v1 directly. The `#[deprecated]` attribute is a Rust compiler hint â€” it produces a warning at the call site in Rust code, but has zero enforcement at the NEAR runtime level. The on-chain ABI exposes the method name as a plain string; the deprecation annotation is invisible to external callers.

---

### Impact Explanation

The 64-byte transaction Merkle proof forgery (documented at the bitmex link in the contract's own doc comment) allows an attacker to craft a `tx_id` that is actually the hash of an internal Merkle tree node rather than a real transaction. Because Bitcoin's Merkle tree uses the same double-SHA256 for both leaf and internal nodes, a carefully chosen internal node hash can be presented as a `tx_id` with a shortened proof path, and `compute_root_from_merkle_proof` will reconstruct the correct Merkle root, causing the function to return `true`. [4](#0-3) 

Any downstream NEAR contract or dApp that calls `verify_transaction_inclusion` (v1) to gate a cross-chain action â€” token release, bridge unlock, settlement â€” can be deceived into accepting a forged proof as valid. The broken invariant is: **a `true` return value from the verification endpoint must correspond to a real, confirmed Bitcoin transaction**. That invariant does not hold for v1.

---

### Likelihood Explanation

The entry path requires no privileges: any NEAR account can call `verify_transaction_inclusion` with a borsh-serialized `ProofArgs`. The 64-byte forgery technique is publicly documented and has known tooling. The only precondition is that the attacker knows a valid mainchain block hash stored in the contract (trivially obtained via `get_block_hash_by_height` or `get_last_block_header`). Likelihood is high for any deployment where a downstream contract consumes the v1 endpoint.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely. Since `verify_transaction_inclusion_v2` already calls v1 internally via `#[allow(deprecated)]`, the internal logic can be preserved as a private helper. Removing the `pub` keyword (or adding an explicit `env::panic_str` at the top of the function body) closes the bypass with a one-line change and does not affect v2 callers. [5](#0-4) 

---

### Proof of Concept

1. Identify any confirmed mainchain block hash `B` via `get_block_hash_by_height(h)`.
2. From the real Bitcoin block at height `h`, select any internal Merkle tree node hash `N` at depth `d`. Construct a Merkle proof of length `d-1` that reconstructs the block's `merkle_root` starting from `N`.
3. Call `verify_transaction_inclusion` (v1) with:
   - `tx_id = N` (the internal node hash, not a real txid)
   - `tx_block_blockhash = B`
   - `tx_index` = the index corresponding to the chosen internal node
   - `merkle_proof` = the crafted proof path of length `d-1`
   - `confirmations = 1`
4. The function computes `compute_root_from_merkle_proof(N, tx_index, proof)`, which equals `merkle_root`, and returns `true` â€” despite `N` not being a real transaction.
5. The same call to `verify_transaction_inclusion_v2` would fail at the coinbase proof length check (`merkle_proof.len() == coinbase_merkle_proof.len()` with the shortened proof) or at the coinbase root reconstruction, blocking the forgery. [6](#0-5)
