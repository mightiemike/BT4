### Title
Non-Atomic Block-Header Deletion in DB Migration 48→49 Permanently Bricks Node on Partial Failure - (`nearcore/src/migrations.rs`)

### Summary

`migrate_48_to_49` calls `delete_old_block_headers`, which first commits an unconditional `delete_all(DBCol::BlockHeader)` to RocksDB, then re-adds only the needed headers in separate batches. If the process crashes or an I/O error occurs between those two commits, the hot store is left with zero block headers while the DB version remains at 48. On every subsequent restart the migration re-runs, `verify_block_headers` immediately fails because the headers it expects to read from `DBCol::BlockHeader` are gone, and the node enters a permanent boot loop with no built-in recovery path.

### Finding Description

`migrate_48_to_49` executes three sequential, non-atomic steps:

1. `copy_block_headers_to_cold_db` — writes headers to cold store via SST ingestion
2. `update_epoch_sync_proof` — generates and persists the compressed epoch-sync proof
3. `delete_old_block_headers` — **irreversibly** deletes all headers, then re-adds only the tail-to-head window

Inside `delete_old_block_headers`:

```rust
// Step A — committed immediately, no rollback
let mut store_update = store.store_update();
store_update.delete_all(DBCol::BlockHeader);
store_update.commit();                          // ← all headers gone

// Step B — re-add in batches; each commit can fail independently
let mut store_update = chain_store.store_update();
for height in tail_height..(latest_known_height + 1) {
    for block_hash in chain_store.get_all_header_hashes_by_height(height) {
        if let Ok(block) = chain_store.get_block(&block_hash) {
            store_update.set_block_header_only(block.header());
        }
    }
    if height % BATCH_SIZE == 0 {
        store_update.commit()?;                 // ← can return Err here
        store_update = chain_store.store_update();
    }
}
store_update.commit()?;                         // ← or here

The DB version is only bumped **after** `migrate()` returns `Ok`:

```rust
migrator
    .migrate(&hot_store, cold_db.as_ref(), version, is_snapshot)
    .map_err(StoreOpenerError::MigrationError)?;

hot_store.set_db_version(version + 1);   // never reached if migrate() fails
``` [2](#0-1) 

If the process dies or returns `Err` anywhere in Step B, the DB is left at version 48 with an empty `DBCol::BlockHeader`. On the next startup, `ensure_version` sees version 48 and re-runs the migration. `verify_block_headers` — which runs before `delete_old_block_headers` — calls `chain_store.get_block_header(&block_hash)?` for every height in `[tail, head]`:

```rust
let header_from_store = chain_store.get_block_header(&block_hash)?;
assert_eq!(header_from_block, header_from_store.as_ref(), "block header mismatch");
``` [3](#0-2) 

Because `DBCol::BlockHeader` is now empty, this returns `DBNotFoundErr`, the migration returns `Err`, the version stays at 48, and the cycle repeats indefinitely.

The only documented escape hatch is the `SetVersionCommand` tool, whose own comment reads:

```rust
// TODO: remove this cmd once we have a proper way to rollback migration
``` [4](#0-3) 

Manually setting the version to 49 skips the migration entirely, leaving the hot store with no block headers — breaking header-dependent operations (transaction validation, epoch sync proof generation, GC) until the node re-derives them.

The `DB_VERSION` itself is tied to `ProtocolFeature::ContinuousEpochSync`, making this migration mandatory for every node running the 2.13 binary:

```rust
pub const DB_VERSION: DbVersion =
    if ProtocolFeature::ContinuousEpochSync.enabled(PROTOCOL_VERSION) { 49 } else { 48 };
``` [5](#0-4) 

### Impact Explanation

Any node upgrading to the 2.13 binary that experiences an I/O error, disk-full condition, or process kill during the batch re-add phase of `delete_old_block_headers` is permanently bricked at startup. For validator nodes this means inability to participate in consensus (slashing risk). For archival nodes with a cold store, the cold DB also receives an incomplete or empty `BlockHeader` column because `copy_block_headers_to_cold_db` on a retry reads from the already-emptied hot store. There is no automatic recovery; the only path is manual operator intervention with `SetVersionCommand`, which itself leaves the node in a degraded state.

### Likelihood Explanation

The migration is triggered automatically on every node upgrading from DB version 48 to 49. The `delete_all` + batch-re-add pattern is inherently non-atomic across two separate RocksDB transactions. The re-add loop iterates over every block height from `tail` to `latest_known_height` — potentially hundreds of thousands of heights on a long-running mainnet node — committing every `BATCH_SIZE = 100,000` heights. Any transient I/O failure, OOM kill, or operator `SIGKILL` during this window triggers the bug. The CHANGELOG notes this migration "is expected to take 1–2 hours" for archival nodes, substantially widening the failure window. [6](#0-5) 

### Recommendation

Make the deletion and re-addition atomic by staging both operations in a single `StoreUpdate` transaction, or by inverting the order: first write the needed headers (from `DBCol::Block`), then delete the surplus ones. Alternatively, gate the `delete_all` on a successful completion flag stored in the DB, so a retry skips the deletion if the re-add already succeeded. The version bump should only occur after the entire operation — including the re-add — is durably committed.

### Proof of Concept

1. Upgrade a mainnet node binary to 2.13 (DB version 48 → 49 migration triggers).
2. After `delete_all(DBCol::BlockHeader)` commits (Step A) but before all batch re-add commits complete (Step B), send `SIGKILL` to the process.
3. Restart the node. Observe that `verify_block_headers` immediately returns `DBNotFoundErr` for the first block hash it checks, the migration returns `Err`, the version stays at 48, and the node exits without starting.
4. Repeat restart — the node loops indefinitely. The exact divergent DB state is: `DBCol::DbVersion` = `"48"`, `DBCol::BlockHeader` = empty, `DBCol::Block` = intact.

/// This migration does three things:
/// 1. Copy block headers from hot_store to cold_db (if cold_db is present)
/// 2. Generate and save the compressed epoch sync proof
/// 3. Clear the block headers from genesis to tail in hot_store
fn migrate_48_to_49(
    hot_store: &Store,
    cold_db: Option<&ColdDB>,
    transaction_validity_period: BlockHeightDelta,
    home_dir: &Path,
    cold_store_config: Option<&StoreConfig>,
    is_snapshot: bool,
) -> anyhow::Result<()> {
    tracing::info!(target: "migrations", "starting migration from DB version 48 to 49");

    // State snapshot DBs only contain flat storage columns and lack the
    // epoch/chain data that every step of this migration requires. Skip them.
    if is_snapshot {
        tracing::info!(target: "migrations", "state snapshot DB, skipping chain-dependent migration steps");
        return Ok(());
    }

    // Fresh nodes and forknet-initialized nodes have BlockMisc cleared, so
    // HEAD is absent; nodes that only produced blocks in the genesis epoch
    // have HEAD set but head.epoch_id == EpochId::default(). In both cases
    // there are no block headers to copy, no epoch sync proof to derive, and
    // nothing to verify or delete.
    match hot_store.chain_store().head() {
        Ok(head) if head.epoch_id == EpochId::default() => {
            tracing::info!(target: "migrations", "chain is in the genesis epoch, skipping chain-dependent migration steps");
            return Ok(());
        }
        Err(Error::DBNotFoundErr(_)) => {
            tracing::info!(target: "migrations", "chain head not set (fresh/forknet DB), skipping chain-dependent migration steps");
            return Ok(());
        }
        Ok(_) => {}
        Err(err) => return Err(err.into()),
    }

    if let Some(cold_db) = cold_db {
        let cold_store_config =
            cold_store_config.expect("cold_store config must be present when cold_db exists");
        copy_block_headers_to_cold_db(hot_store, cold_db, home_dir, cold_store_config)?;
    }

    update_epoch_sync_proof(hot_store.clone(), transaction_validity_period)?;
    verify_block_headers(hot_store)?;
    delete_old_block_headers(hot_store)?;
    Ok(())

fn delete_old_block_headers(store: &Store) -> anyhow::Result<()> {
    tracing::info!(target: "migrations", "deleting all block headers from hot store");

    let mut store_update = store.store_update();
    store_update.delete_all(DBCol::BlockHeader);
    store_update.commit();
    let chain_store = store.chain_store();
    let tail_height = chain_store.tail();
    let latest_known_height =
        store.get_ser::<LatestKnown>(DBCol::BlockMisc, LATEST_KNOWN_KEY).unwrap().height;

    tracing::info!(target: "migrations", ?tail_height, ?latest_known_height, "adding required block headers to hot store");

    let mut store_update = chain_store.store_update();
    for height in tail_height..(latest_known_height + 1) {
        for block_hash in chain_store.get_all_header_hashes_by_height(height) {
            // We've already checked for errors and missing blocks in the verify_block_headers function
            if let Ok(block) = chain_store.get_block(&block_hash) {
                store_update.set_block_header_only(block.header());
            }
        }
        if height % BATCH_SIZE == 0 {
            tracing::info!(target: "migrations", ?height, ?latest_known_height, "committing addition of required block headers to hot store");
            store_update.commit()?;
            store_update = chain_store.store_update();
        }
    }
    store_update.commit()?;
    tracing::info!(target: "migrations", ?latest_known_height, "completed deletion of old block headers from hot store");

    Ok(())
}
