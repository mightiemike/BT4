### Title
AuxPoW Parent Block PoW Verified Against Dogecoin Target Instead of Parent Block's Own Target - (File: contract/src/dogecoin.rs)

### Summary
In the Dogecoin AuxPoW validation path, the parent Bitcoin block's proof-of-work hash is checked against the **Dogecoin block's difficulty target** (`block_header.bits`) rather than the **parent block's own difficulty target** (`aux_data.parent_block.bits`). These two values are distinct and not interchangeable. This mirrors the original report's class exactly: two separate values (DAI price vs. DAIx price; Dogecoin target vs. parent-block target) are incorrectly assumed equivalent, causing a critical validation step to use the wrong reference value.

### Finding Description
In `contract/src/dogecoin.rs`, the `check_aux` function performs AuxPoW validation. The final PoW check at lines 149–154 reads:

```rust
let pow_hash = aux_data.parent_block.block_hash_pow();
require!(
    self.skip_pow_verification
        || U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
    format!("block should have correct pow")
);
``` [1](#0-0) 

`block_header` is the **Dogecoin** block being submitted. `aux_data.parent_block` is the **Bitcoin** parent block whose hash must satisfy Bitcoin's own PoW. The correct check must use `aux_data.parent_block.bits` (the parent block's encoded difficulty target), not `block_header.bits` (the Dogecoin block's difficulty target). These are two independent fields from two independent block headers on two different chains with different difficulty levels.

The `AuxData` struct carries the parent block as a caller-supplied `Header`:

```rust
pub struct AuxData {
    pub coinbase_tx: Vec<u8>,
    pub merkle_proof: Vec<H256>,
    pub chain_merkle_proof: Vec<H256>,
    pub chain_id: usize,
    pub parent_block: Header,   // <-- attacker-controlled
}
``` [2](#0-1) 

The parent block's `bits` field is never independently validated against any expected Bitcoin network difficulty. The contract only validates the Dogecoin block's `bits` via `check_target` / `get_next_work_required`. [3](#0-2) 

### Impact Explanation
Bitcoin's current difficulty is orders of magnitude higher than Dogecoin's. In target terms, `target_from_bits(block_header.bits)` (Dogecoin target, a larger U256 value = easier) is far larger than `target_from_bits(aux_data.parent_block.bits)` (Bitcoin target, a smaller U256 value = harder). The check `pow_hash <= target_from_bits(block_header.bits)` is therefore much easier to satisfy than the correct check `pow_hash <= target_from_bits(aux_data.parent_block.bits)`.

An attacker can submit a crafted `parent_block` whose hash satisfies Dogecoin's target but not Bitcoin's actual PoW requirement. The contract accepts this as a valid AuxPoW block, inserting a fraudulent header into the light client's canonical chain. Downstream consumers calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` against this fraudulent header will receive incorrect SPV proof results, enabling cross-chain fraud. [4](#0-3) 

### Likelihood Explanation
The `submit_blocks` entry point is gated by `#[trusted_relayer]`, meaning only a registered relayer can call it. However, the threat model explicitly includes "relayer-path user supplying adversarial chain data." A malicious or compromised relayer can supply a crafted `AuxData` with a parent block whose hash satisfies Dogecoin's (easier) target. The computational cost is Dogecoin-level mining work, which is feasible. No privileged key compromise beyond the relayer role is required. [5](#0-4) 

### Recommendation
Replace `block_header.bits` with `aux_data.parent_block.bits` in the PoW check inside `check_aux`:

```rust
// Before (incorrect):
U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits)

// After (correct):
U256::from_le_bytes(&pow_hash.0) <= target_from_bits(aux_data.parent_block.bits)
``` [1](#0-0) 

Additionally, validate that `aux_data.parent_block.bits` encodes a target that is at or below the Bitcoin network's `pow_limit` to prevent a secondary bypass where an attacker sets `parent_block.bits` to an artificially easy value.

### Proof of Concept

1. Deploy the contract in Dogecoin mode (`feature = "dogecoin"`) with `skip_pow_verification = false`.
2. Register a relayer account.
3. Construct a Dogecoin block header with AuxPoW flag set (`version & 0x100 != 0`) and valid `bits` matching the expected Dogecoin difficulty.
4. Craft `AuxData` where:
   - `parent_block` is a Bitcoin-format header with `bits` set to a very easy value (e.g., `0x207fffff`), but the actual hash of this header satisfies only Dogecoin's current target (not Bitcoin's real target).
   - `coinbase_tx` embeds the Dogecoin block hash in its script_sig at the correct position.
   - `merkle_proof` and `chain_merkle_proof` are valid for the crafted coinbase and chain root.
5. Call `submit_blocks` with this header and `AuxData`.
6. The contract computes `pow_hash = aux_data.parent_block.block_hash_pow()` and checks it against `target_from_bits(block_header.bits)` (Dogecoin's target). Since the crafted parent block hash was chosen to satisfy Dogecoin's easier target, the check passes.
7. The fraudulent Dogecoin block is stored in `headers_pool` and promoted to the mainchain if its chainwork exceeds the current tip.
8. `verify_transaction_inclusion` called against this block's hash returns `true` for attacker-chosen transactions. [6](#0-5)

    pub(crate) fn check_aux(&mut self, block_header: &Header, aux_data: &AuxData) {
        // The Dogecoin block must have the AuxPoW flag set (bit 8) when AuxPoW data is present.
        // https://github.com/dogecoin/dogecoin/blob/master/src/auxpow.h
        const BLOCK_VERSION_AUXPOW: i32 = 0x100;

        require!(
            aux_data.chain_merkle_proof.len() <= 30,
            "Aux POW chain merkle branch too long"
        );
        require!(
            block_header.version & BLOCK_VERSION_AUXPOW != 0,
            "Aux POW block does not have AuxPoW flag set in version"
        );

        let chain_id = self.get_config().aux_chain_id;

        require!(
            chain_id == block_header.get_chain_id(),
            format!(
                "block does not have our chain ID (got {}, expected {chain_id})",
                block_header.get_chain_id()
            )
        );

        require!(
            chain_id != aux_data.parent_block.get_chain_id(),
            "Aux POW parent has our chain ID"
        );

        let chain_root = merkle_tools::compute_root_from_merkle_proof(
            block_header.block_hash(),
            aux_data.chain_id,
            &aux_data.chain_merkle_proof,
        );

        let coinbase_tx = aux_data.get_coinbase_tx();
        let coinbase_tx_hash = coinbase_tx.compute_txid();

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                H256::from(coinbase_tx_hash.to_raw_hash().to_byte_array()),
                0,
                &aux_data.merkle_proof,
            ) == aux_data.parent_block.merkle_root
        );

        let script_sig = coinbase_tx
            .input
            .first()
            .unwrap()
            .script_sig
            .to_hex_string();
        let pos_merged_mining_header = script_sig.find(MERGED_MINING_HEADER);
        let mut pos_chain_root = script_sig
            .find(&chain_root.to_string())
            .expect("Aux POW missing chain merkle root in parent coinbase");

        match pos_merged_mining_header {
            Some(pos_merged_mining_header) => {
                if script_sig[pos_merged_mining_header + MERGED_MINING_HEADER.len()..]
                    .contains(MERGED_MINING_HEADER)
                {
                    env::panic_str("Multiple merged mining headers in coinbase");
                }

                require!(
                    pos_merged_mining_header + MERGED_MINING_HEADER.len() == pos_chain_root,
                    "Merged mining header is not just before chain merkle root"
                );
            }
            None => {
                require!(pos_chain_root <= 40, "Aux POW chain merkle root must start in the first 20 bytes of the parent coinbase");
            }
        }

        pos_chain_root += chain_root.to_string().len();
        require!(
            script_sig.len() - pos_chain_root >= 16,
            "Aux POW missing chain merkle tree size and nonce in parent coinbase"
        );

        let bytes = hex::decode(&script_sig[pos_chain_root..pos_chain_root + 8]).unwrap();
        let n_size = u32::from_le_bytes([bytes[0], bytes[1], bytes[2], bytes[3]]);
        require!(
            n_size == (1u32 << aux_data.chain_merkle_proof.len()),
            "Aux POW merkle branch size does not match parent coinbase"
        );

        let bytes = hex::decode(&script_sig[pos_chain_root + 8..pos_chain_root + 16]).unwrap();
        let n_nonce = u32::from_le_bytes([bytes[0], bytes[1], bytes[2], bytes[3]]);

        let chain_id = block_header.get_chain_id();

        let expected_index =
            Self::get_expected_index(n_nonce, chain_id, aux_data.chain_merkle_proof.len());

        require!(
            u32::try_from(aux_data.chain_id).ok() == Some(expected_index),
            "Aux POW wrong index"
        );
        let pow_hash = aux_data.parent_block.block_hash_pow();
        require!(
            self.skip_pow_verification
                || U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
            format!("block should have correct pow")
        );
    }
