### Title
Stale Main-Chain Reference Block Used for Fork Difficulty Validation — (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

### Summary

When a fork block is submitted, the difficulty-target calculation reads the reference block from `mainchain_height_to_header` (the current canonical chain) instead of traversing the fork's own ancestor chain. For Dogecoin post-height 145,000 — where difficulty adjusts every single block — this desynchronization is triggered on every fork block submission. An adversarial relayer can craft fork blocks with timestamps later than the corresponding main-chain blocks, causing the contract to compute a longer apparent timespan and therefore a lower required difficulty than the fork's own chain would mandate.

---

### Finding Description

The vulnerability class is **cross-module desynchronization**: a state value that must reflect the fork's chain is read from the main-chain index before that index has been updated to reflect the fork, causing the security-critical difficulty calculation to use a stale reference.

**Call chain (Dogecoin):**

```
submit_blocks (lib.rs:169)
  └─ submit_block_header (dogecoin.rs:166)
       └─ check_target → check_pow → get_next_work_required (dogecoin.rs:229)
            └─ get_header_by_height (lib.rs:677)   ← reads mainchain_height_to_header
```

In `get_next_work_required` for Dogecoin post-145,000, `difficulty_adjustment_interval = 1`, so the retarget path is taken for every block: [1](#0-0) 

```rust

The TODO comment at line 291 is the developers' own acknowledgment of this uncertainty.

`get_header_by_height` is implemented as: [2](#0-1) 

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

`mainchain_height_to_header` is only updated when a block is stored on the main chain or after a reorg completes. It is **never updated to reflect the fork's ancestor chain** before `check_pow` is called. The fork block's difficulty is therefore validated against the main chain's reference block at `height_first`, not the fork's own ancestor at that height.

The correct `actual_timespan` for a fork block at height H should be:

```
fork_block_at_(H-1).time  −  fork_block_at_(H-2).time
```

But the contract computes:

```
fork_block_at_(H-1).time  −  mainchain_block_at_(H-2).time
``` [3](#0-2) 

---

### Impact Explanation

If an adversarial relayer submits a fork that diverged at height H-2 or earlier, and sets the fork block at H-2 to a timestamp **later** than the main-chain block at H-2, then:

- `mainchain_block_at_(H-2).time` is earlier than `fork_block_at_(H-2).time`
- The apparent timespan `fork_block_at_(H-1).time − mainchain_block_at_(H-2).time` is **longer** than the correct timespan
- A longer timespan produces a **lower required difficulty** (easier target)

The Dogecoin difficulty formula clamps the timespan to `[75%, 150%]` of `retarget_timespan`: [4](#0-3) 

With `retarget_timespan = 60 s`, the attacker can inflate the apparent timespan up to 90 s, reducing the required difficulty by up to ~33%. Fork blocks accepted at reduced difficulty accumulate less real chainwork per block. If the fork eventually surpasses the main chain's `chain_work`, `reorg_chain` is triggered: [5](#0-4) 

This corrupts `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height`, causing `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` to return results based on a fraudulent canonical chain.

The same structural bug exists in `bitcoin.rs` and `litecoin.rs` at their respective difficulty adjustment boundaries (every 2016 blocks), but the Dogecoin case is the most severe because it applies to every fork block after height 145,000. [6](#0-5) [7](#0-6) 

---

### Likelihood Explanation

- `submit_blocks` is reachable by any account holding the `UnrestrictedSubmitBlocks` role or any registered trusted relayer supplying adversarial chain data — both are explicitly in-scope entry points per the audit objective.
- For Dogecoin post-145,000 (the current live chain state), the desynchronization is triggered on **every** fork block, not just at periodic boundaries.
- The attack requires submitting only a small batch of fork blocks (≥ 3 blocks deep) with crafted timestamps, which is a normal relayer operation.
- No cryptographic key compromise or social engineering is required.

---

### Recommendation

Replace the `get_header_by_height` call in `get_next_work_required` with an ancestor traversal that walks back through the fork's own chain (via `get_prev_header`) to find the reference block at `height_first`. This mirrors the fix pattern from the external report: the state used for the calculation must be derived from the same chain context as the block being validated, not from a separate (main-chain) index that has not yet been updated to reflect the fork.

---

### Proof of Concept

1. Main chain is at height H (H > 145,001). Main-chain block at H-2 has timestamp `T_main`.
2. Attacker submits fork block `F1` at height H-2, building on the common ancestor at H-3. `F1.time = T_main + Δ` where `Δ > 0` (e.g., 30 s), satisfying MTP and future-time checks.
3. Attacker submits fork block `F2` at height H-1, building on `F1`. `F2.time = T_main + Δ + 60` (normal spacing).
4. Attacker submits fork block `F3` at height H, building on `F2`.
5. Contract calls `get_next_work_required` for `F3`:
   - `height_first = H - 2`
   - `first_block_time = mainchain_block_at_(H-2).time = T_main` ← stale, should be `F1.time = T_main + Δ`
   - `actual_timespan = F2.time − T_main = Δ + 60` instead of correct `F2.time − F1.time = 60`
   - Inflated timespan → lower `expected_bits` → easier mining target accepted for `F3`.
6. Repeat for subsequent fork blocks. Each block accumulates less real chainwork than required.
7. If the fork's `chain_work` exceeds the main chain's, `reorg_chain` fires, corrupting the canonical chain state and invalidating all subsequent SPV proofs. [8](#0-7) [9](#0-8)

fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
}
