### Title
PRNG Seeded from OS Entropy Injected into Consensus-Critical Arithmetic Operators (`op_add`, `op_subtract`) — (File: src/more_ops.rs)

---

### Summary

The slow-path implementations of `op_add` and `op_subtract` in `src/more_ops.rs` call `rand::rng()` — a PRNG seeded from OS entropy — and use `rng.random_range(0..2)` to distribute BigInt operands between two accumulator buckets. This introduces non-deterministic external state into consensus-critical arithmetic operators. The `rand` crate is a production dependency (`Cargo.toml` line 95). The slow path is reachable by any attacker-controlled CLVM program that passes a large-integer atom to `+` or `-`.

---

### Finding Description

In `src/more_ops.rs`, both `op_add` and `op_subtract` have a two-tier structure:

- **Fast path** (`#[cfg(not(feature = "no-fastpath"))]`): handles `U32` (SmallAtom) operands only, using native integer arithmetic. Fully deterministic.
- **Slow path**: triggered whenever any operand is a `Buffer` (a large integer atom that does not fit in a `U32`). This path calls `rand::rng()` and uses the PRNG to distribute operands between two `Number` (BigInt) buckets.

**`op_add` slow path** (`src/more_ops.rs`, lines 452–481):

```rust
let mut rng = rand::rng();                          // line 453 — OS-entropy PRNG
let mut acc = [Number::from(0), Number::from(0)];
let mut small_acc: Number = 0.into();
while let Some((arg, rest)) = a.next(input) {
    ...
    NodeVisitor::Buffer(buf) => {
        let val = number_from_u8(buf);
        acc[rng.random_range(0..2)] += val;         // line 465 — PRNG decides bucket
    }
    ...
}
let total = a.new_number(&acc[0] + &acc[1] + small_acc)?;  // line 480
```

**`op_subtract` slow path** (`src/more_ops.rs`, lines 532–583):

```rust
let mut rng = rand::rng();                          // line 533 — OS-entropy PRNG
let mut acc = [Number::from(0), Number::from(0)];
...
// first arg:
acc[rng.random_range(0..2)] += number_from_u8(buf); // line 546
// subsequent args:
acc[rng.random_range(0..2)] -= number_from_u8(buf); // line 565
...
let total = a.new_number(&acc[0] + &acc[1] + small_acc)?;  // line 582
```

The PRNG is seeded from OS entropy on every call. Different nodes, different runs, and different threads will have different PRNG states.

**Mathematical analysis of current arithmetic result:** Because BigInt addition is exact and commutative, `acc[0] + acc[1]` always equals the sum of all values distributed to both buckets, regardless of which bucket each value was placed in. The final arithmetic output of `op_add` and `op_subtract` is therefore the same on every node. The PRNG does not currently corrupt the arithmetic result.

**The broken invariant is the determinism guarantee itself.** CLVM is a sandboxed, deterministic VM. Its entire security model rests on the property that the same program with the same environment produces the same result on every node. The injection of `rand::rng()` — seeded from OS entropy — into production arithmetic operators violates this invariant at the code level, even if the current bucket-sum identity happens to preserve the arithmetic output.

**There is no legitimate reason for a PRNG in an arithmetic operator.** The two-bucket pattern adds zero value: `acc[0] + acc[1]` is always equal to a single accumulator. The only effect of the PRNG is to introduce OS-entropy-dependent intermediate state into a consensus-critical code path. This is the defining signature of a backdoor-class insertion: code that is currently "safe" due to a mathematical identity, but which would cause consensus divergence if the final aggregation step were changed from `acc[0] + acc[1]` to any expression that depends on the bucket assignment (e.g., `acc[0] * acc[1]`, `acc[0] - acc[1]`, or a conditional on which bucket is larger).

---

### Impact Explanation

**Corrupted invariant:** CLVM's determinism guarantee — the property that `run_program(program, env)` returns the same `(cost, result)` on every conforming node — is violated at the implementation level by the presence of OS-entropy PRNG in production arithmetic operators.

**Concrete activation path for consensus divergence:** If any future commit changes the final aggregation from `acc[0] + acc[1] + small_acc` to any non-commutative or non-symmetric expression (e.g., `acc[0] - acc[1] + small_acc`, or a branch on `acc[0] > acc[1]`), different nodes will compute different arithmetic results for the same CLVM program. This would cause a hard consensus split: nodes would disagree on whether a coin spend is valid, breaking the blockchain.

**Current state:** The arithmetic output is not currently corrupted. The PRNG affects only intermediate bucket assignment, which cancels out in the final sum.

**Backdoor readiness:** The PRNG infrastructure is fully in place. The `rand` crate is a production dependency. The PRNG is called on every slow-path arithmetic invocation. A one-line change to the aggregation expression would activate consensus divergence.

---

### Likelihood Explanation

**Trigger:** Any CLVM program that calls `+` or `-` with at least one operand that is a large integer atom (any atom whose value does not fit in a `u32`, i.e., value > 4,294,967,295 or negative) bypasses the fast path and enters the slow path. An attacker submitting a coin spend with a puzzle that performs bignum arithmetic triggers this path unconditionally.

**Frequency:** The slow path is the fallback for all non-trivial integer arithmetic in CLVM programs. It is a high-frequency production code path.

**Discovery:** The PRNG calls are inside `use rand::Rng;` scoped imports at lines 417 and 491, making them slightly less visible than top-level imports, but they are present in the production `clvmr` crate with no feature flag gating.

---

### Recommendation

Remove `rand::rng()` and the two-bucket accumulator pattern entirely from `op_add` and `op_subtract`. Replace with a single `Number` accumulator:

```rust
// op_add slow path — correct replacement
let mut total: Number = 0.into();
while let Some((arg, rest)) = a.next(input) {
    ...
    NodeVisitor::Buffer(buf) => {
        cost += ARITH_COST_PER_BYTE * (buf.len() as Cost);
        check_cost(cost, max_cost)?;
        total += number_from_u8(buf);
    }
    NodeVisitor::U32(val) => {
        cost += len_for_value(val) as Cost * ARITH_COST_PER_BYTE;
        check_cost(cost, max_cost)?;
        total += val;
    }
    ...
}
let result = a.new_number(total)?;
```

Remove `rand` from `[dependencies]` in `Cargo.toml` if it is not required by any other production code path (verify `src/run_program.rs` and `src/allocator.rs` usages as well, since `grep` found matches there).

### Proof of Concept

**Trigger program (CLVM assembly):**

```
(+ 0x010000000000 0x010000000000)
```

Both operands are `0x010000000000` = 1,099,511,627,776, which exceeds `u32::MAX`. The fast path returns `None` at line 431 (`let NodeVisitor::U32(val) = a.node(arg) else { return Ok(None); }`), falling through to the slow path. `rand::rng()` is called at line 453, and `rng.random_range(0..2)` is called at line 465 for each operand. The result is `0x020000000000` on every node (correct), but the PRNG has been invoked with OS entropy in a consensus-critical execution path.
