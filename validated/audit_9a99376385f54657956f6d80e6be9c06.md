### Title
`op_secp256k1_verify` Accepts Non-Canonical High-S Signatures, Enabling Signature Malleability — (File: src/secp_ops.rs)

---

### Summary

`op_secp256k1_verify` in `src/secp_ops.rs` does not enforce low-S normalization on the caller-supplied signature. For any valid compact signature `(r, s)`, the malleable form `(r, n − s)` — where `n` is the secp256k1 curve order — is also accepted as valid. This breaks the invariant that a given `(pubkey, message)` pair has exactly one canonical valid signature, directly analogous to the external report's finding that the `s` component of the ECDSA signature was not checked.

---

### Finding Description

In `op_secp256k1_verify`, the signature is parsed and verified as follows:

```rust
let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| { ... })?;
let result = verifier.verify_prehash(msg.as_ref(), &sig);
```

`K1Signature::from_slice()` (from the `k256` crate) parses a compact 64-byte `(r ‖ s)` encoding and does **not** enforce low-S normalization. `verify_prehash()` likewise does not enforce it. No call to `sig.normalize_s()` or any equivalent low-S check is present anywhere in the function.

Because ECDSA on secp256k1 is symmetric in `s`: if `(r, s)` is a valid signature for `(pubkey, msg)`, then `(r, n − s)` is also a valid signature for the same `(pubkey, msg)`. Both forms pass `op_secp256k1_verify` unchanged.

The same structural issue exists in `op_secp256r1_verify` (using the `p256` crate), but secp256k1 is the more widely deployed curve in Chia puzzles and is the direct analog to the external report. [1](#0-0) 

---

### Impact Explanation

Any CLVM puzzle that uses `secp256k1_verify` to gate a coin spend can be satisfied by **two** distinct byte strings for the same `(pubkey, message)` pair: the canonical low-S signature and its high-S malleable twin. An attacker who observes a valid spend broadcast to the mempool can:

1. Extract the signature `(r, s)` from the solution.
2. Compute `s' = n − s` (trivial arithmetic over the curve order).
3. Construct an alternative spend solution containing `(r, s')`.
4. Broadcast the alternative spend, which `op_secp256k1_verify` accepts as valid.

Concrete broken invariant: the `Reduction` returned by `op_secp256k1_verify` is `Ok(Reduction(cost, a.nil()))` for both `(r, s)` and `(r, n−s)`, meaning the CLVM execution engine treats both as indistinguishable successes. Any protocol layer that assumes a unique signature per `(pubkey, message)` pair — for example, to deduplicate spends, to bind a signature to a specific spend bundle, or to prevent front-running — is broken. [2](#0-1) 

---

### Likelihood Explanation

Computing the malleable signature requires only one modular subtraction: `s' = n − s mod n`. This is a single arithmetic operation, requires no secret knowledge, and can be performed by any observer of a valid spend. The attacker-controlled entry path is direct: the `sig` argument to `secp256k1_verify` is a raw atom supplied by the CLVM program's solution, fully controlled by the caller.

The likelihood of exploitation depends on how Chia puzzles use `secp256k1_verify`. Any puzzle that does not bind the signature to a unique spend context (e.g., by hashing the coin ID or spend bundle hash into the message) is immediately vulnerable to front-running via the malleable form. [3](#0-2) 

---

### Recommendation

After parsing the signature, enforce low-S normalization before verification:

```rust
let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| { ... })?;
// Reject high-S signatures to prevent malleability
if let Some(normalized) = sig.normalize_s() {
    // normalized means the original was high-S — reject it
    return Err(EvalErr::InvalidOpArg(input,
        "secp256k1_verify: signature has high-S component".to_string()))?;
}
let result = verifier.verify_prehash(msg.as_ref(), &sig);
```

Alternatively, use `k256`'s `normalize_s()` and only accept the normalized form, or check `s <= n/2` explicitly. Apply the same fix to `op_secp256r1_verify`. [4](#0-3)

pub fn op_secp256k1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256K1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256k1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256k1_verify pubkey")?;
    let verifier = K1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256k1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is message
    let msg = atom(a, msg, "secp256k1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256k1_verify sig")?;
    let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
