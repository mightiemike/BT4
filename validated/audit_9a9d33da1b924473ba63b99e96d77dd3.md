Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks caller-controlled `owner` instead of actual depositor `sender`, fully defeating the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension` is documented to gate `addLiquidity` by depositor address, but its `beforeAddLiquidity` hook silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`) and instead checks `owner` — a free parameter supplied by the caller. Any unprivileged address can bypass the allowlist by calling `addLiquidity` with `owner` set to any allowlisted address, causing unauthorized tokens to enter the pool and forcing an unsolicited LP position onto the allowlisted address.

## Finding Description
In `MetricOmmPool.addLiquidity`, the pool invokes `_beforeAddLiquidity(msg.sender, owner, ...)` where `msg.sender` is the actual depositor and `owner` is the LP position recipient freely chosen by the caller: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both arguments faithfully to the extension via `abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, ...))`: [2](#0-1) 

However, `DepositAllowlistExtension.beforeAddLiquidity` declares the first parameter (`sender`) as unnamed, discarding it entirely, and checks `owner` against the allowlist: [3](#0-2) 

Because `owner` is a free argument in `addLiquidity(address owner, ...)`, any caller can set it to any allowlisted address and the guard passes unconditionally regardless of who the actual depositor (`sender`) is. The `SwapAllowlistExtension` does not share this flaw — it correctly checks `sender`: [4](#0-3) 

The allowlist storage is public and readable by anyone: [5](#0-4) 

## Impact Explanation
The pool admin's deposit access control is completely defeated by any unprivileged caller with knowledge of one allowlisted address (publicly readable from `allowedDepositor`). Unauthorized liquidity enters the pool, breaking the core invariant that only allowlisted depositors may provide liquidity. Additionally, the allowlisted `owner` receives an unsolicited LP position funded by the attacker's tokens; since `removeLiquidity` enforces `msg.sender == owner`, the allowlisted address can drain those tokens — constituting a forced transfer from the attacker into the allowlisted address's control. This is a broken core pool functionality / admin-boundary break with direct fund impact.

## Likelihood Explanation
Exploitation requires only: (a) reading the public `allowedDepositor[pool][addr]` mapping to identify one allowlisted address, and (b) calling `pool.addLiquidity(owner=allowlisted_addr, ...)` with valid `deltas` and a callback that supplies tokens. No special privileges, flash loans, or oracle manipulation are required. The attack is repeatable at will.

## Recommendation
Replace the `owner` check with a `sender` check to match the documented intent and the pattern used by `SwapAllowlistExtension`:

```solidity
// FIXED
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```
Setup:
  Pool P has DepositAllowlistExtension configured.
  allowedDepositor[P][alice] = true
  bob is NOT in the allowlist.

Attack:
  1. bob calls P.addLiquidity(owner=alice, salt=S, deltas=D, callbackData, extensionData)
  2. Pool calls _beforeAddLiquidity(sender=bob, owner=alice, ...)
  3. Extension receives (_, alice, ...) — bob is silently dropped.
  4. Check: allowedDepositor[P][alice] == true → no revert.
  5. Pool credits LP position to alice at salt S.
  6. Pool calls bob.metricOmmSwapCallback(...) — bob's tokens are pulled into the pool.
  7. alice holds the LP position worth bob's deposited tokens.
  8. alice calls P.removeLiquidity(owner=alice, salt=S, ...) and recovers the tokens.

Result:
  - bob's deposit bypasses the allowlist entirely.
  - alice receives an unsolicited LP position (and can drain it).
  - The pool holds unauthorized liquidity, defeating the pool admin's access control.
```
