### Title
USDT `approve()` Non-Boolean Return Causes `creditDeposit()` Reversion — (`core/contracts/DirectDepositV1.sol`)

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` through the `IIERC20Base` interface, which declares `approve` as returning `bool`. USDT's `approve()` returns no value. Solidity ≥0.8 reverts when ABI-decoding an empty return as `bool`, so any DDA holding USDT will be permanently bricked for deposit crediting.

### Finding Description

`DirectDepositV1` defines its own local token interface: [1](#0-0) 

`approve` is declared to return `bool`. Inside `creditDeposit()`, which carries **no access modifier** and is callable by any address: [2](#0-1) 

At line 92, `token.approve(address(endpoint), balance)` is called through this interface. When the token is USDT (which emits no return data from `approve`), the Solidity ABI decoder attempts to decode 0 bytes as `bool` and reverts.

The `ERC20Helper` library does implement low-level safe wrappers for `transfer` and `transferFrom`: [3](#0-2) 

But **no `safeApprove` equivalent exists** in `ERC20Helper`, and `DirectDepositV1` does not use `ERC20Helper` at all — it calls `approve` directly through the interface.

The same raw `approve` pattern appears in `ContractOwner.wrapVaultAsset()` via `IERC20Base`: [4](#0-3) 

`IERC20Base.approve` also declares a `bool` return: [5](#0-4) 

### Impact Explanation

If USDT is registered as a supported spot product token (a stated design goal — USDT is a "first natural candidate" for collateral), every call to `creditDeposit()` on any DDA holding USDT will revert at the `approve` line. The entire deposit-crediting flow for that subaccount is permanently blocked. Funds sent to the DDA are trapped: they cannot be credited to the subaccount via `creditDeposit()`, and `withdraw()` is `onlyOwner`, so the user has no self-service recovery path.

### Likelihood Explanation

USDT is explicitly named as a primary collateral candidate. Any user who sends USDT to their DDA and triggers `creditDeposit()` (directly or via `ContractOwner.creditDepositV1()`) hits this revert unconditionally. No special conditions are required beyond USDT being listed as a product token.

### Recommendation

Replace the direct `token.approve(...)` call in `creditDeposit()` with a low-level safe wrapper analogous to the existing `safeTransfer` in `DirectDepositV1` itself:

```solidity
function safeApprove(IIERC20Base self, address spender, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IIERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        "Approve failed"
    );
}
```

Apply the same fix to `ContractOwner.wrapVaultAsset()` by adding a `safeApprove` to `ERC20Helper` and using it in place of the raw `assetToken.approve(...)` calls.

### Proof of Concept

1. USDT is listed as a supported spot product (productId `X`) in `SpotEngine`.
2. A user sends USDT to their DDA address.
3. Anyone calls `ContractOwner.creditDepositV1(subaccount)` or `DirectDepositV1.creditDeposit()` directly.
4. The loop reaches the USDT product; `token.approve(address(endpoint), balance)` is called.
5. USDT's `approve()` executes successfully on-chain but returns no data.
6. Solidity's ABI decoder tries to decode 0 bytes as `bool` → reverts.
7. The entire `creditDeposit()` call reverts; the USDT balance remains in the DDA, uncredited and inaccessible to the user. [6](#0-5)
