### Title
Linked Signer Can Authorize Its Own Replacement via `LinkSigner`, Enabling Unauthorized Subaccount Mutation - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This means the current linked signer of a subaccount can sign a `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with any address — including an attacker-controlled one — without the subaccount owner's involvement. The authorization check uses the wrong identity class (linked signer instead of subaccount owner) for a privileged account-management operation.

---

### Finding Description

In `processTransactionImpl`, the `LinkSigner` branch is:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
```

`validateSignedTx` calls `validateSignature`, which calls `verifier.validateSignature` with the linked signer address when `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 178–183
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
```

`Verifier.validateSignature` accepts the signature if it recovers to either the subaccount owner address **or** the linked signer:

```solidity
// Verifier.sol lines 298–303
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
```

This means the existing linked signer can produce a valid `LinkSigner` transaction that replaces `linkedSigners[subaccount]` with any arbitrary address.

**Contrast with the slow-mode path**, which correctly restricts `LinkSigner` to the subaccount owner via `validateSender`:

```solidity
// EndpointTx.sol lines 237–239
validateSender(txn.sender, sender);   // requires msg.sender == subaccount owner
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

The fast (sequencer-processed) path and the slow-mode path apply inconsistent authorization for the same operation.

---

### Impact Explanation

A compromised or malicious linked signer (e.g., a hot wallet or API key) can:

1. Sign a `LinkSigner` transaction pointing `signedTx.tx.signer` to a fresh attacker-controlled address.
2. The sequencer processes it; `linkedSigners[victimSubaccount]` is overwritten.
3. The attacker's new address is now the linked signer and can sign `WithdrawCollateral` (V1, `allowLinkedSigner = true`) or `WithdrawCollateralV2` with `sendTo == address(0)` to drain the subaccount's collateral.
4. The legitimate owner's only recourse is a slow-mode `LinkSigner` transaction, which is subject to a 3-day delay (`SLOW_MODE_TX_DELAY`), during which the attacker retains signing authority.

**Corrupted state**: `linkedSigners[subaccount]` is set to an unauthorized address, granting an attacker persistent signing authority over the subaccount.

**Impact: 4** — direct collateral theft path via subsequent withdrawal transactions signed by the attacker-controlled linked signer.

---

### Likelihood Explanation

Linked signers are commonly hot wallets or programmatic API keys, which carry higher compromise risk than the cold-wallet subaccount owner. Any party that obtains the linked signer's private key can execute this attack without any on-chain preconditions beyond the linked signer being set. The sequencer processes the transaction normally.

**Likelihood: 3** — requires linked signer key compromise, which is realistic for API-key-style deployments.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. Only the subaccount owner (the address embedded in the `bytes32 sender`) should be permitted to authorize a change to the linked signer, consistent with the slow-mode path.

```solidity
// EndpointTx.sol — processTransactionImpl, LinkSigner branch
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only subaccount owner may change linked signer
);
```

---

### Proof of Concept

1. Alice owns subaccount `aliceSub` and has set `linkedSigners[aliceSub] = hotWallet`.
2. Attacker compromises `hotWallet`.
3. Attacker uses `hotWallet` to sign a `SignedLinkSigner` transaction: `{ sender: aliceSub, signer: attackerAddress, nonce: currentNonce }`.
4. Attacker submits the signed transaction to the sequencer (via any supported off-chain channel).
5. Sequencer calls `submitTransactionsChecked` → `processTransactionImpl` → `LinkSigner` branch.
6. `validateSignedTx(..., true)` passes because `hotWallet == getLinkedSigner(aliceSub)`.
7. `linkedSigners[aliceSub]` is overwritten with `attackerAddress`.
8. Attacker signs a `WithdrawCollateral` transaction from `aliceSub` using `attackerAddress`, draining Alice's collateral. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
