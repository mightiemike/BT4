### Title
Broken Coinbase-to-Transaction Binding in `verify_transaction_inclusion_v2` Allows 64-Byte Forgery Bypass - (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase proof. However, there is no on-chain assertion that `coinbase_tx_id` is distinct from `tx_id`, nor that it represents a real leaf-level transaction rather than an internal Merkle tree node. An unprivileged caller can supply the same Merkle-internal-node hash as both `coinbase_tx_id` and `tx_id`, satisfying both checks simultaneously and causing the contract to return `true` for a transaction that was never included in any Bitcoin block.

---

### Finding Description

`verify_transaction_inclusion_v2` accepts a `ProofArgsV2` struct containing two logically distinct proof pairs that must refer to different entities:

- `(coinbase_tx_id, coinbase_merkle_proof)` — supposed to prove the real coinbase transaction is at index 0
- `(tx_id, tx_index, merkle_proof)` — supposed to prove the target transaction is at `tx_index`

The only structural constraint linking them is a length equality check:

```rust
require!(
    args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
    "Coinbase merkle proof and transaction merkle proof should have the same length"
);
```

The coinbase proof is then validated independently:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.coinbase_tx_id.clone(),
    0usize,
    &args.coinbase_merkle_proof,
) == header.block_header.merkle_root
```

And the transaction proof is validated independently inside the delegated call to `verify_transaction_inclusion`. There is **no assertion** that `coinbase_tx_id != tx_id`, and no assertion that `coinbase_tx_id` is a real leaf-level transaction hash rather than an internal Merkle node.

**Concrete attack using a real block with 4 transactions:**

Given a real mainchain block with Merkle tree:
```
Leaves:  H(T0), H(T1), H(T2), H(T3)
Level 1: N01 = H(H(T0)||H(T1)),  N23 = H(H(T2)||H(T3))
Root:    R = H(N01 || N23)
```

The attacker submits `ProofArgsV2` with:
- `tx_id`                 = `N01`  (an internal node, not a real tx)
- `tx_index`              = `0`
- `merkle_proof`          = `[N23]`
- `coinbase_tx_id`        = `N01`  (same internal node)
- `coinbase_merkle_proof` = `[N23]` (same proof)
- `tx_block_blockhash`    = real mainchain block hash

Step 1 — length check: `[N23].len() == [N23].len()` ✓

Step 2 — coinbase check:
`compute_root_from_merkle_proof(N01, 0, [N23])` = `H(N01 || N23)` = `R` ✓

Step 3 — transaction check (inside `verify_transaction_inclusion`):
`compute_root_from_merkle_proof(N01, 0, [N23])` = `H(N01 || N23)` = `R` ✓

The function returns `true` for a transaction `N01` that does not exist as a real Bitcoin transaction.

---

### Impact Explanation

`verify_transaction_inclusion_v2` is the production-recommended SPV proof API (v1 is explicitly deprecated). Any downstream NEAR contract or application that calls it to gate asset releases, cross-chain bridges, or settlement logic will accept a forged proof as valid. The coinbase-proof mitigation — the entire reason v2 exists — is rendered completely ineffective. The impact is **proof-verification forgery**: an attacker can convince the contract that an arbitrary 32-byte value was a confirmed Bitcoin transaction in any mainchain block.

---

### Likelihood Explanation

The attack requires no privileges, no staked tokens, no PoW, and no cryptographic preimage computation. The attacker only needs:
1. Any real mainchain block already accepted by the contract (publicly observable on-chain state).
2. Knowledge of that block's Merkle tree structure (derivable from any Bitcoin full node or block explorer).

The call path is fully open: `verify_transaction_inclusion_v2` is a public, non-paused NEAR view/call method reachable by any account.

---

### Recommendation

Add an explicit binding assertion between the two proof subjects before any Merkle computation:

```rust
require!(
    args.coinbase_tx_id != args.tx_id || args.tx_index == 0,
    "coinbase_tx_id must differ from tx_id for non-coinbase transactions"
);
```

More robustly, enforce that `coinbase_tx_id` is structurally a real coinbase transaction (e.g., validate the raw transaction bytes contain the coinbase marker `0xFFFFFFFF` input sequence), or require that `tx_index != 0` whenever `coinbase_tx_id == tx_id` is disallowed. The core invariant to enforce is: **the coinbase proof and the target-transaction proof must not be satisfiable by the same input**.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
