Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks Position Recipient (`owner`) Instead of Token Depositor (`sender`), Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension` is documented as gating `addLiquidity` by depositor address. However, `beforeAddLiquidity` silently discards the `sender` parameter (the actual token payer) and checks `owner` (the LP position recipient) instead. Because `addLiquidity` explicitly supports an operator pattern where `sender != owner`, any unpermissioned address can bypass the allowlist by nominating any allowlisted address as `owner`, paying tokens themselves while the allowlisted address receives the position.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the position recipient to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` correctly forwards both as the first and second arguments to the extension hook: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but leaves it unnamed (discarded), then evaluates only `owner`: [3](#0-2) 

The guard `allowedDepositor[msg.sender][owner]` checks whether the pool has allowlisted the *position recipient*, not the *token payer*. An attacker (`bob`) who is not on the allowlist calls `pool.addLiquidity(alice, ...)` where `alice` is allowlisted. The pool calls `_beforeAddLiquidity(bob, alice, ...)`. The extension checks `allowedDepositor[pool][alice]` → `true` → no revert. `bob` pays tokens via callback; `alice` receives the LP position. `bob`'s address is never evaluated.

`SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual swapper), confirming the asymmetry is a defect in `DepositAllowlistExtension`: [4](#0-3) 

## Impact Explanation

The deposit allowlist is a pool-level access control mechanism intended to restrict which addresses may inject liquidity (e.g., compliance, whitelisted market-maker programs, controlled bootstrapping). Because `sender` is never checked, the allowlist is completely nullified: any address can deposit tokens into a restricted pool. The pool admin's access control boundary is fully bypassed by an unprivileged caller with a single transaction. This constitutes a broken core pool functionality and admin-boundary break causing the pool to accept liquidity from unpermissioned sources.

## Likelihood Explanation

The `addLiquidity` operator pattern (`sender != owner`) is explicitly supported by the pool interface. Any actor who can observe a single prior allowlisted deposit on-chain (the allowlisted `owner` address is visible in transaction history) can exploit this with one transaction. No privileged access, special tokens, or malicious setup is required. The attack is repeatable indefinitely.

## Recommendation

Replace the unnamed first parameter with `sender` and gate on it, consistent with `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

## Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` configured.
2. Admin calls `setAllowedToDeposit(pool, alice, true)` — only `alice` is permitted.
3. Attacker (`bob`, not on allowlist) calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(bob /*sender*/, alice /*owner*/, ...)`.
5. Extension checks `allowedDepositor[pool][alice]` → `true` → no revert.
6. `bob` pays tokens via the modify-liquidity callback; `alice` receives the LP position.
7. `bob` has successfully deposited into a pool that was supposed to block him.

Foundry test plan: deploy `DepositAllowlistExtension`, configure a pool with it, allowlist only `alice`, call `addLiquidity` from `bob` with `owner = alice`, assert no revert and that `bob`'s token balance decreased.
