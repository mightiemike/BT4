### Title
`add_backward_compatibility_space` mishandles named nested tuple `cairo_type` strings, producing wrong hinted class hash for pre-0.10.0 deprecated contracts — (`crates/starknet_os/src/hints/class_hash/hinted_class_hash.rs`)

### Summary
`add_backward_compatibility_space` uses `pair_regex` directly on the flat inner content of a `cairo_type` tuple. For named nested tuples such as `(a: (b: felt, c: felt))`, the regex greedily matches `a: (b:` as a single pair (consuming the opening paren and the inner key as part of the "value" token), leaving the inner `b: felt` untransformed. The Python reference transforms every `key: value` pair at all nesting depths, producing a different byte sequence and therefore a different Keccak256 hash.

### Finding Description
In `add_backward_compatibility_space`: [1](#0-0) 

The `cairo_type` branch applies `pair_regex` once, flat, to the captured inner string. `pair_regex` is: [2](#0-1) 

`([^,\s)]+)` excludes comma, whitespace, and `)` but **not** `(` or `:`. For inner content `a: (b: felt, c: felt)`, the engine matches `a: (b:` as one pair (group 2 = `(b:`), replacing it with `a : (b:` and leaving `b: felt` inside the nested tuple untouched. The Python reference adds a space before every colon in every named-tuple pair at every depth.

Concrete divergence:
- Input `cairo_type`: `(a: (b: felt, c: felt))`
- Rust output: `(a : (b: felt, c : felt))`
- Python output: `(a : (b : felt, c : felt))`

The difference is the missing ` ` before `:` in `b: felt`. This changes the Keccak256 of the serialized JSON, changing the hinted class hash returned by `compute_cairo_hinted_class_hash`: [3](#0-2) 

The hinted class hash is then fed as a field into `compute_deprecated_class_hash`: [4](#0-3) 

and loaded into the VM as `hinted_class_hash` during OS execution: [5](#0-4) 

The Cairo OS then hashes this field as part of `deprecated_compiled_class_hash`: [6](#0-5) 

The test suite does not cover this case: [7](#0-6) 

### Impact Explanation
Any deprecated (Cairo 0) contract with `compiler_version = None` (pre-0.10.0) whose program identifiers contain a `cairo_type` with a named nested tuple will have its hinted class hash — and therefore its full class hash — computed differently by the Rust OS than by the Python reference. Contracts declared on the Python sequencer with such types will have their on-chain class hash H_python; the Rust OS will compute H_rust ≠ H_python, causing the OS to fail class verification and select the wrong (or no) compiled class for execution. This falls under **Critical: wrong class hash computed, wrong compiled class selected for execution**.

### Likelihood Explanation
Low-to-Medium. Named nested tuple types in Cairo 0 `cairo_type` fields are syntactically valid and appear in real contracts (e.g., structs-of-structs passed by value). The `compiler_version = None` gate is satisfied by any contract compiled before Cairo 0.10.0 or by a crafted JSON with the field omitted. An unprivileged user can submit a DECLARE v1 transaction (`BroadcastedDeclareV1Transaction` / `DeclareV1Transaction`) with such a class. The existing test corpus does not include this pattern, so the bug is undetected by CI. [8](#0-7) 

### Recommendation
Replace the flat `pair_regex.replace_all` on the raw inner string with a recursive or iterative approach that descends into nested parentheses. For `cairo_type`, the correct algorithm is: parse the type string as a tree of parenthesized groups, then for each group apply the `key: value` → `key : value` substitution to its direct children only, recursing into sub-groups. Alternatively, port the exact Python regex/substitution logic verbatim and add a test case for `(a: (b: felt, c: felt))` verified against the Python-produced hash.

    // The sorting is necessary because serde_json with `preserve_order` feature enabled
    // maintains insertion order instead of sorting keys.
    // TODO(Meshi): Compute hinted hashes when loading serialized contracts from storage, before
    // deserializing, to avoid back-and-forth serde.
    let contract_value = serde_json::to_value(contract_class)?;
    let sorted_contract_value = sort_json_value(contract_value);
    let contract_definition_vec = serde_json::to_vec(&sorted_contract_value)?;
    let contract_definition: CairoContractDefinition<'_> =
        serde_json::from_slice(&contract_definition_vec)?;

    let mut string_buffer = vec![];

    let mut ser = serde_json::Serializer::with_formatter(&mut string_buffer, PythonJsonFormatter);
    contract_definition.serialize(&mut ser)?;

    let mut raw_json_output = String::from_utf8(string_buffer)?;
    if contract_definition.program.compiler_version.is_none() {
        add_backward_compatibility_space(&mut raw_json_output);
    }

    keccak_writer
        .write_all(raw_json_output.as_bytes())
        .expect("writing to KeccakWriter never fails");

    let KeccakWriter(hash) = keccak_writer;
    Ok(truncated_keccak(<[u8; 32]>::from(hash.finalize())))

    let cairo_regex = Regex::new(r#""cairo_type"\s*:\s*"\(([^"]*)\)""#).unwrap();
