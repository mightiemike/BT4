Audit Report

## Title
Permissionless `register()` Clears ADMIN-Set Pool Blacklist, Bypassing Emergency Oracle Access Revocation — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

## Summary
`OracleBase.register()` is a public, fee-gated function that unconditionally clears `blacklisted[pool]` for any pool that passes the factory check, requiring only 1 wei (the default `registrationFee`). Because `setBlacklist()` is the sole ADMIN mechanism to revoke a pool's oracle-price access, any actor — including the blacklisted pool operator — can pay 1 wei to immediately undo an ADMIN emergency blacklist, restoring the pool's ability to call `price()` and execute swaps against live oracle data.

## Finding Description
`setBlacklist()` (L271–276) is restricted to `ADMIN_ROLE` and is the only path to set `blacklisted[pool] = true`. The `price()` function (L160–172) enforces this at call time via `require(!blacklisted[pool], Blacklisted(pool))` at L167, and also via the `notBlacklisted` modifier on `msg.sender` at L163.

`register()` (L201–214) requires only:
- `msg.value >= registrationFee` (default: 1 wei, set at L53)
- `approvedFactories.contains(factory)` — any previously approved factory
- `IPoolFactory(factory).isPool(pool)` — the pool is still recognized by that factory

Lines 207–210 then unconditionally clear the blacklist:
```solidity
if (blacklisted[pool]) {
    blacklisted[pool] = false;
    emit BlacklistUpdated(pool, false);
}
```

The NatSpec comment at L197–198 even documents this as intentional: *"Paying also clears any blacklist on the pool."* There is no role check, no timelock, and no guard preventing a blacklisted pool operator from calling `register()` immediately after an ADMIN blacklist action. Once `blacklisted[pool]` is cleared, `price()` at L167 passes, and the pool resumes reading live oracle bid/ask.

## Impact Explanation
The blacklist is the emergency mechanism by which ADMIN revokes a pool's oracle access — e.g., when a pool is found to be malicious, compromised, or draining LP funds through bad-price execution. The intended invariant is that once blacklisted, the pool cannot call `price()` and cannot execute swaps. Because `register()` clears the blacklist permissionlessly for 1 wei, this invariant is broken: LP funds remain exposed to the condition that triggered the emergency blacklist. This is a direct admin-boundary break — an ADMIN-level security action is overridden by a permissionless, near-zero-cost call — matching the allowed impact gate for oracle role checks bypassed by an unprivileged path.

## Likelihood Explanation
The blacklisted pool operator has both motive and trivial means. The only preconditions are that the pool is still recognized by an approved factory (`isPool(pool)` returns `true`) and that 1 wei is available. No privileged access, no timelock, and no special role is required. The attack is a single public transaction, immediately repeatable after every ADMIN blacklist action.

## Recommendation
Remove the blacklist-clearing logic from `register()`. Blacklisted pools should be rejected, not silently un-blacklisted. Only `ADMIN_ROLE` via `setBlacklist()` should be able to clear the blacklist:

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
    require(pool != address(0));
    require(approvedFactories.contains(factory), FactoryNotApproved(factory));
    require(IPoolFactory(factory).isPool(pool), NotAPool(pool));
-   if (blacklisted[pool]) {
-       blacklisted[pool] = false;
-       emit BlacklistUpdated(pool, false);
-   }
+   require(!blacklisted[pool], Blacklisted(pool));
    registeredPool[feedId][pool] = true;
    emit PoolRegistered(feedId, pool, msg.sender, msg.value);
}
```

## Proof of Concept
```solidity
// 1. ADMIN blacklists the pool (emergency response)
oracle.setBlacklist(address(pool), true);
// pool.price() now reverts with Blacklisted(pool)

// 2. Anyone (including the pool operator) calls register() with 1 wei
oracle.register{value: 1 wei}(feedId, address(pool), address(approvedFactory));
// Lines 207-210: blacklisted[pool] = false

// 3. Pool reads oracle price again — blacklist fully bypassed
(uint256 mid, uint256 spread,,) = oracle.price(feedId, address(pool));
// Succeeds. Pool executes swaps against live oracle data.
```

Preconditions: `approvedFactory.isPool(pool)` returns `true` (pool not removed from factory) and 1 wei. Both are trivially satisfied for any legitimately deployed pool whose operator has not been removed from the factory.
