### Title
Missing AuxPoW-Flag Enforcement in `submit_block_header` Allows AuxPoW-Flagged Block Acceptance Without AuxPoW Proof — (`contract/src/dogecoin.rs`)

---

### Summary

`submit_block_header` dispatches to `check_aux` only when the caller supplies `aux_data=Some(...)`. There is no guard that rejects a block whose `version & 0x100 != 0` (the `BLOCK_VERSION_AUXPOW` flag) when `aux_data=None`. A trusted relayer can therefore submit a header with the AuxPoW version flag set, omit the AuxPoW proof entirely, and have the block accepted after only the standard scrypt PoW check — bypassing every AuxPoW invariant enforced by `check_aux`.

---

### Finding Description

The branching logic in `submit_block_header` is:

```rust
if let Some(ref aux_data) = aux_data {
    self.check_aux(&block_header, aux_data);   // full AuxPoW validation
} else {
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        "block should have correct pow"
    );
}
``` [1](#0-0) 

The branch is driven entirely by whether the caller passed `Some` or `None` for `aux_data`. The block's `version` field is never consulted in the `None` arm. `check_aux` itself does verify the flag — but only when it is actually called:

```rust
require!(
    block_header.version & BLOCK_VERSION_AUXPOW != 0,
    "Aux POW block does not have AuxPoW flag set in version"
);
``` [2](#0-1) 

That check is unreachable when `aux_data=None`. No code in `submit_block_header` or its callers enforces the inverse: *if the version flag is set, `aux_data` must be present*.

The public entry point is `submit_blocks`, which is gated by `#[trusted_relayer]` / `Role::UnrestrictedSubmitBlocks`: [3](#0-2) 

A trusted relayer is an explicitly supported attacker role per the scope rules ("trusted-relayer submission path with attacker-relevant data").

---

### Impact Explanation

A block accepted through this path:

1. Has `version & 0x100 != 0` — the Dogecoin network treats this as an AuxPoW block and would reject it without a valid AuxPoW proof.
2. Was never validated by `check_aux`, so none of the following invariants were checked: chain-ID correctness, parent-block chain-ID exclusion, coinbase merkle proof, merged-mining header position, chain-merkle-tree size/nonce, or the parent block's scrypt PoW against the Dogecoin target.
3. Is stored in `headers_pool` and can become the canonical tip if its `chain_work` exceeds the current tip.

The result is **false canonical header acceptance**: the light client's canonical chain diverges from the real Dogecoin chain, which invalidates any downstream SPV proof that relies on the contract's `mainchain_tip_blockhash` or `mainchain_height_to_header` index.

---

### Likelihood Explanation

**Testnet / synthetic chains (high likelihood):** The `doge_init_blocks()` fixture uses `DOGE_BITS = 0x1e0fffff` (minimum difficulty). At that difficulty an attacker can brute-force a valid scrypt PoW in milliseconds, set `version |= 0x100`, pass `aux_data=None`, and the block is accepted. This is directly, locally testable with the existing test harness.

**Mainnet (low-to-medium likelihood):** `check_target` enforces that `bits` matches the retarget algorithm, so the attacker cannot lower the difficulty arbitrarily. They must mine a scrypt solution at real Dogecoin mainnet difficulty — the same work as mining a legitimate block. This is expensive but not impossible for a well-resourced actor (e.g., a mining pool that is also a registered relayer). The absence of the AuxPoW check means they do not need to be a merged miner or construct a valid parent-chain coinbase.

---

### Recommendation

Add an explicit rejection of AuxPoW-flagged blocks that arrive without AuxPoW data, inside the `None` arm of `submit_block_header`:

```rust
} else {
    const BLOCK_VERSION_AUXPOW: i32 = 0x100;
    require!(
        block_header.version & BLOCK_VERSION_AUXPOW == 0,
        "Block has AuxPoW version flag set but no AuxPoW proof was provided"
    );
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        "block should have correct pow"
    );
}
```

This closes the gap without changing the happy path for either pure-PoW blocks or legitimate AuxPoW blocks.

fn auxpow_flag_no_proof_accepted() {

    // The last block in the chain is the parent for our attack block.

        bits: DOGE_BITS,              // minimum difficulty — trivially minable
        nonce: 0,
    };

    // Brute-force a valid scrypt PoW (takes < 1 s at min difficulty).
    loop {
        let pow = attack_header.block_hash_pow();
        if U256::from_le_bytes(&pow.0) <= target_from_bits(DOGE_BITS) { break; }
        attack_header.nonce += 1;

    contract.submit_block_header((attack_header, None), false);

    let tip = contract.get_last_block_header();
    assert_eq!(tip.block_header.version & 0x100, 0x100); // AuxPoW flag in canonical tip

    pub(crate) fn submit_block_header(
        &mut self,
        header: (Header, Option<AuxData>),
        skip_pow_verification: bool,
    ) {
        let (block_header, aux_data) = header;

        let prev_block_header = self.get_prev_header(&block_header);
        let current_block_hash = block_header.block_hash();

        if !skip_pow_verification {
            self.check_target(&block_header, &prev_block_header);

            if let Some(ref aux_data) = aux_data {
                self.check_aux(&block_header, aux_data);
            } else {
                let pow_hash = block_header.block_hash_pow();
                // Check if the block hash is less than or equal to the target
                require!(
                    U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
                    format!("block should have correct pow")
                );
            }
        }

        let (current_block_computed_chain_work, overflow) = prev_block_header
            .chain_work
            .overflowing_add(work_from_bits(block_header.bits));
        require!(!overflow, "Addition of U256 values overflowed");

        let current_header = ExtendedHeader {
            block_header: block_header.clone().into_light(),
            block_hash: current_block_hash,
            chain_work: current_block_computed_chain_work,
            block_height: 1 + prev_block_header.block_height,
        };

        self.submit_block_header_inner(current_header, &prev_block_header);
    }
