### Title
`verify_transaction_inclusion` Unconditionally Rejects Valid Empty Merkle Proofs for Single-Transaction Blocks — (`contract/src/lib.rs`)

### Summary

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` contain a hard `require!` that rejects any empty `merkle_proof`. For a single-transaction Bitcoin block (where the only transaction is the coinbase), the Merkle proof is legitimately empty because the transaction hash *is* the Merkle root. The underlying `compute_root_from_merkle_proof` already handles this correctly, but the guard prevents it from ever being reached. Any downstream NEAR contract or user that calls either verification function for a transaction in a single-transaction block will always receive a panic, permanently breaking that verification path.

### Finding Description

In `contract/src/lib.rs`, `verify_transaction_inclusion` contains:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

For a single-transaction block, the Merkle proof is empty by definition: the transaction hash equals the Merkle root, and no sibling hashes are needed. `compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` handles this correctly — with an empty `merkle_proof` slice, the `for` loop does not execute and the function returns `transaction_hash` unchanged:

```rust
for proof_hash in merkle_proof {
    // never entered for empty proof
}
current_hash  // returns transaction_hash as-is
``` [2](#0-1) 

So if `tx_id == merkle_root`, the comparison would correctly return `true` — but the `require!` guard panics before that comparison is ever made.

`verify_transaction_inclusion_v2` is also affected. It delegates to `verify_transaction_inclusion` at the end:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

For a single-transaction block, both `merkle_proof` and `coinbase_merkle_proof` are empty. The length-equality check passes (`0 == 0`), the coinbase check passes (since `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id`, which equals `merkle_root`), and then the call to `verify_transaction_inclusion` panics on the empty-proof guard. Both public verification entry points are broken for this valid block class. [4](#0-3) 

### Impact Explanation

Any NEAR contract or user that submits a valid SPV proof for a transaction in a single-transaction Bitcoin block will always receive a panic from the light client. If a downstream protocol gates fund release on a `true` result from `verify_transaction_inclusion` or `verify_transaction_inclusion_v2`, those funds become permanently unreleasable for this class of blocks. This is a direct analog to M-4: the contract is intended to verify all valid Bitcoin transaction inclusions, but silently fails for a legitimate subset of them due to a missing handler for the empty-proof case.

### Likelihood Explanation

Single-transaction blocks (containing only the coinbase) are valid Bitcoin blocks and appear throughout Bitcoin's history, including in early mainnet blocks. Any relayer that submits such a block header to the contract will store it successfully, but no downstream consumer will ever be able to verify a transaction against it. The entry path is fully unprivileged: any NEAR account can call `verify_transaction_inclusion_v2` with a valid `ProofArgsV2` for a single-transaction block and trigger the panic. [5](#0-4) 

### Recommendation

Remove the unconditional empty-proof guard, or replace it with a check that explicitly allows the empty-proof case when `tx_id == merkle_root`:

```rust
// Allow empty proof only when tx_id is the merkle root (single-tx block)
if args.merkle_proof.is_empty() {
    return args.tx_id == header.block_header.merkle_root;
}
```

This preserves the intent of the guard (rejecting malformed proofs) while correctly handling the single-transaction block case that `compute_root_from_merkle_proof` already supports.

### Proof of Concept

1. Submit a real single-transaction Bitcoin block header via `submit_blocks`. The contract accepts it and stores it.
2. Call `verify_transaction_inclusion_v2` with:
   - `tx_id` = the coinbase transaction hash (= the block's `merkle_root`)
   - `tx_block_blockhash` = the block hash stored in step 1
   - `tx_index` = 0
   - `merkle_proof` = `[]`
   - `coinbase_tx_id` = same coinbase transaction hash
   - `coinbase_merkle_proof` = `[]`
   - `confirmations` = 1
3. The length check passes (`0 == 0`). The coinbase check passes (`coinbase_tx_id == merkle_root`). The call delegates to `verify_transaction_inclusion`, which panics: `"Merkle proof is empty"`.
4. The transaction reverts. Any downstream protocol waiting for a `true` result to release funds is permanently blocked for this block. [6](#0-5) [7](#0-6)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
