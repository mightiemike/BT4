### Title
Duplicate `PositionId` in `MsgClaimTierRewards` Allows Double-Claiming Bonus Rewards from the Rewards Pool - (File: x/tieredrewards/keeper/msg_server.go)

---

### Summary

`ClaimTierRewards` in `x/tieredrewards/keeper/msg_server.go` accepts a caller-supplied slice of `PositionIds` without checking for duplicates. When the same position ID appears twice, the position is loaded once with its pre-call state, appended twice to the in-memory `positions` slice, and then processed twice by `claimRewardsAndUpdatesPositions`. The second pass operates on the stale (pre-first-pass) in-memory copy and recomputes the current-segment bonus from the same `LastBonusAccrual` checkpoint, causing `SendCoinsFromModuleToAccount` to transfer bonus coins from `RewardsPoolName` a second time.

---

### Finding Description

`ClaimTierRewards` builds a `positions` slice by iterating `msg.PositionIds`:

```go
// x/tieredrewards/keeper/msg_server.go  lines 434-446
positions := make([]types.PositionState, 0, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    pos, err := ms.getPositionState(ctx, posId)
    ...
    if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
        return nil, err
    }
    positions = append(positions, pos)
}
```

There is no deduplication guard on `msg.PositionIds`. If the same ID appears twice, two copies of the same `PositionState` — both carrying the identical pre-call `LastBonusAccrual` and `LastEventSeq` — are appended to the slice.

`claimRewardsAndUpdatesPositions` then iterates the slice:

```go
// x/tieredrewards/keeper/claim_rewards.go  lines 110-132
for i := range positions {
    pos := &positions[i]
    ...
    base, err := k.claimBaseRewards(ctx, *pos)
    ...
    bonus, err := k.processEventsAndClaimBonus(ctx, pos)
    ...
    if err := k.setPosition(ctx, pos.Position, nil); err != nil { ... }
}
```

**First pass (positions[0]):** `processEventsAndClaimBonus` walks validator events since `pos.LastEventSeq`, decrements their reference counts, computes the current-segment bonus, calls `SendCoinsFromModuleToAccount` to pay it, and advances `pos.LastBonusAccrual` to `blockTime`. `setPosition` persists the updated state.

**Second pass (positions[1]):** This copy still holds the *old* `LastBonusAccrual` and `LastEventSeq`. Historical events whose reference counts were decremented to zero in the first pass are gone, so `getValidatorEventsSince` returns nothing. However, the current-segment bonus computation at the end of `processEventsAndClaimBonus` is unconditional:

```go
// x/tieredrewards/keeper/claim_rewards.go  lines 206-213
if bonded && val.IsBonded() {
    currentRate, err := k.getTokensPerShare(ctx, valAddr)
    ...
    bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
    totalBonus = totalBonus.Add(bonus)
}
```

`segmentStart` equals the *old* `pos.LastBonusAccrual` (identical to the first pass), `blockTime` is the same block, and `currentRate` is the same. The result is the same bonus amount, and `SendCoinsFromModuleToAccount` transfers it again from `RewardsPoolName`. `setPosition` then overwrites the correctly updated store entry with the stale second-pass state.

---

### Impact Explanation

The `RewardsPoolName` module account is drained by the duplicated `SendCoinsFromModuleToAccount` call. An owner of any active, bonded tier position can extract twice (or N times, for N duplicates) the accrued current-segment bonus in a single transaction. This is a direct, quantifiable loss of funds from the protocol's bonus rewards pool, harming all other position holders who depend on that pool for future rewards.

The corrupted invariant: *each position's bonus rewards are paid at most once per `ClaimTierRewards` invocation.*

---

### Likelihood Explanation

1. The entry point is a standard user-signed `MsgClaimTierRewards` transaction — no privileged role, no leaked key, no social engineering required.
2. The only precondition is owning at least one active, bonded tier position.
3. The exploit is a trivial client-side change: repeat the same position ID in the `PositionIds` field.
4. Nothing in `msg.Validate()` or `validateClaimRewards` prevents duplicate IDs.
5. The bonus pool is a shared resource; repeated exploitation across many positions or blocks compounds the drain.

Likelihood: **High**.

---

### Recommendation

Add a deduplication check on `msg.PositionIds` before building the `positions` slice in `ClaimTierRewards`. A simple seen-set suffices:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    if _, dup := seen[posId]; dup {
        return nil, errorsmod.Wrapf(types.ErrDuplicatePositionID, "position id %d appears more than once", posId)
    }
    seen[posId] = struct{}{}
    // ... existing load and validate logic
}
```

Alternatively, enforce uniqueness in `MsgClaimTierRewards.Validate()` so the check is applied at the message-validation layer before any keeper logic runs.

---

### Proof of Concept

1. Alice owns position ID `42`, which is actively delegated to a bonded validator and has accrued bonus rewards.
2. Alice submits `MsgClaimTierRewards{Owner: alice, PositionIds: [42, 42]}`.
3. `ClaimTierRewards` loads position `42` twice, appending two identical `PositionState` copies to `positions`.
4. `claimRewardsAndUpdatesPositions` processes `positions[0]`: computes and pays bonus `B` from `RewardsPoolName`, advances `LastBonusAccrual` to `blockTime`, saves to store.
5. `claimRewardsAndUpdatesPositions` processes `positions[1]` (stale copy, old `LastBonusAccrual`): historical events are gone (ref counts hit 0), but the current-segment bonus is recomputed identically and paid again — another `B` leaves `RewardsPoolName`.
6. Alice receives `2B` bonus instead of `B`, and the rewards pool is short by `B`.

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}
