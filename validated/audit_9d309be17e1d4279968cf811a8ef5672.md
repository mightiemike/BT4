### Title
Duplicate-Leaf Merkle Padding Allows Phantom `tx_index` to Pass `verify_transaction_inclusion_v2` — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` accepts a caller-supplied `tx_index` and verifies it solely by checking that `compute_root_from_merkle_proof(tx_id, tx_index, proof) == merkle_root`. Because `merkle_proof_calculator` pads odd-length trees by duplicating the last leaf, the same `(tx_id, proof)` pair produces the correct Merkle root for **both** the real last-leaf position **and** the phantom duplicate position. No guard checks that `tx_index < actual_tx_count`, so the phantom position passes verification.

---

### Finding Description

**Duplication in `merkle_proof_calculator`:** [1](#0-0) 

When the current level has an odd number of hashes, the last hash is appended to itself before pairing. This is standard Bitcoin Merkle tree behavior, but it creates a structural ambiguity: the last real leaf (`tx[N-1]`) and the phantom leaf at position `N` share identical subtree paths.

**`compute_root_from_merkle_proof` is position-sensitive only via parity:** [2](#0-1) 

The function uses `current_position % 2` to decide left/right placement at each level. For a 3-tx block, positions 2 and 3 differ only at the first level: position 2 (even) hashes as `H(tx[2], sibling)`, position 3 (odd) hashes as `H(sibling, tx[2])`. Because the sibling at that level **is also `tx[2]`** (the duplicate), both orderings produce `H(tx[2], tx[2])` — identical intermediate nodes — and the root is the same.

**Concrete proof for a 3-transaction block** (`tx[0]=coinbase, tx[1], tx[2]`):

| Level | Nodes |
|---|---|
| 0 | `tx[0], tx[1], tx[2], tx[2]` (padded) |
| 1 | `H(tx[0],tx[1])`, `H(tx[2],tx[2])` |
| root | `H(H(tx[0],tx[1]), H(tx[2],tx[2]))` |

- `compute_root_from_merkle_proof(tx[2], 2, [tx[2], H(tx[0],tx[1])])` → root ✓ (real position)
- `compute_root_from_merkle_proof(tx[2], 3, [tx[2], H(tx[0],tx[1])])` → root ✓ (phantom position)

Both evaluate to `H(H(tx[0],tx[1]), H(tx[2],tx[2]))` because `H(tx[2], tx[2]) == H(tx[2], tx[2])` regardless of which side `tx[2]` is placed.

**`verify_transaction_inclusion_v2` has no `tx_index` upper-bound check:** [3](#0-2) 

The only guards are:
1. `merkle_proof.len() == coinbase_merkle_proof.len()` — satisfied (both length 2).
2. Coinbase proof matches `merkle_root` — satisfied with a valid coinbase proof for `tx[0]`.
3. `compute_root_from_merkle_proof(tx_id, tx_index, proof) == merkle_root` — satisfied at phantom index 3.

There is no check that `tx_index` is within the actual transaction count of the block. [4](#0-3) 

---

### Impact Explanation

An unprivileged caller can invoke `verify_transaction_inclusion_v2` with a `tx_index` pointing to a phantom position that does not correspond to any real transaction in the block, and the function returns `true`. Any on-chain or off-chain consumer that relies on this return value to confirm a transaction's inclusion in a Bitcoin block will accept a false proof. This directly violates the stated invariant that `tx_index` must correspond to an actual transaction.

---

### Likelihood Explanation

The attack requires only:
1. Knowledge of a Bitcoin block with an odd transaction count (extremely common — the majority of Bitcoin blocks have an odd number of transactions).
2. The ability to call `verify_transaction_inclusion_v2` as any unprivileged NEAR account.
3. Construction of a valid coinbase Merkle proof (trivially computable from public block data).

No privileged role, private key, or special access is needed. The call is a public `view`-style function. [5](#0-4) 

---

### Recommendation

Add an upper-bound check on `tx_index` before the Merkle proof computation. The proof depth encodes the tree height: a proof of length `k` implies the tree has at most `2^k` leaves. The valid range for `tx_index` is `[0, 2^k - 1]`, but more precisely, the contract should require that `tx_index < 2^(proof.len())` **and** that the index does not fall in the phantom-duplicate range. The most robust fix is to require that `tx_index` is strictly less than the number of transactions in the block — if the block header stores a transaction count, validate against it. If not, at minimum reject any `tx_index` where `tx_index >= 2^(proof.len() - 1)` when the proof length implies an odd-padded tree, or require the caller to supply the transaction count and validate it against the proof depth.

### Proof of Concept

For a 3-transaction block with `tx[0]` (coinbase), `tx[1]`, `tx[2]`:

```
merkle_root = H(H(tx[0], tx[1]), H(tx[2], tx[2]))

// Coinbase proof (valid, index 0):
coinbase_merkle_proof = [tx[1], H(tx[2], tx[2])]

// Phantom proof (index 3, does not exist):
tx_merkle_proof = [tx[2], H(tx[0], tx[1])]

// Call:
verify_transaction_inclusion_v2(ProofArgsV2 {
    tx_id:                tx[2],
    tx_block_blockhash:   <valid block hash>,
    tx_index:             3,          // phantom — only tx[0..2] exist
    merkle_proof:         [tx[2], H(tx[0], tx[1])],
    coinbase_tx_id:       tx[0],
    coinbase_merkle_proof:[tx[1], H(tx[2], tx[2])],
    confirmations:        0,
})
// Returns: true  ← false positive for non-existent transaction
```

Step-by-step evaluation of `compute_root_from_merkle_proof(tx[2], 3, [tx[2], H(tx[0],tx[1])])`:

- position=3 (odd) → `H(tx[2], tx[2])`, position=1
- position=1 (odd) → `H(H(tx[0],tx[1]), H(tx[2],tx[2]))` = `merkle_root` ✓ [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
