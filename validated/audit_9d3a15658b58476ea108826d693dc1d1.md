Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the actual user, enabling allowlist bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the pool's `msg.sender` — the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is always the router contract, not the end user. Any pool admin who allowlists the router to restore router-mediated access for legitimate users simultaneously grants every non-allowlisted user the ability to bypass the allowlist by calling the router.

## Finding Description
`SwapAllowlistExtension.beforeSwap` enforces access via `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the extension's caller) and `sender` is the first argument forwarded by the pool:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmPool.swap()` sets `sender = msg.sender` (L230–231), which is whoever called `pool.swap()` directly. `ExtensionCalling._beforeSwap` forwards this value unchanged (L160–176).

Every `MetricOmmSimpleRouter` entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap()` directly. The pool's `msg.sender` is therefore the router contract. The original user's address is stored only in transient storage via `_setNextCallbackContext` for payment settlement and is never passed to the pool as the swap initiator.

**Mode A (router not allowlisted):** Individual users are allowlisted but the router is not. Every router-mediated swap reverts with `NotAllowedToSwap`, locking allowlisted users out of the primary swap interface.

**Mode B (router allowlisted to restore access):** The pool admin adds the router to `allowedSwapper[pool]`. Because the extension checks the router's address — which is now approved — every user, including those explicitly excluded, can bypass the gate by calling `router.exactInputSingle()`. The allowlist is silently nullified.

## Impact Explanation
In Mode B, a non-allowlisted user (e.g., a sanctioned address or an explicitly excluded counterparty) can execute swaps against the pool's LP liquidity. The LP's exposure to unwanted counterparties is the direct fund-impacting consequence. This constitutes broken core pool functionality (allowlist bypass) and an admin-boundary break where an unprivileged caller bypasses a pool-admin-configured access control. The bypass requires no privileged capability — any public user with a wallet can call `exactInputSingle`.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any production pool deploying `SwapAllowlistExtension` will encounter Mode A the first time a legitimate user attempts a router-mediated swap. The pool admin's natural remediation — allowlisting the router — immediately triggers Mode B. The trigger is a single public call to `router.exactInputSingle()` with no special setup.

## Recommendation
The router must forward the original user's address so the extension can gate on the correct actor. Two viable approaches:

1. **Pass originator in `extensionData`**: The router encodes `msg.sender` into `extensionData` before forwarding to the pool. `SwapAllowlistExtension.beforeSwap` decodes and checks this address when `sender` is a known trusted router.

2. **Dedicated originator field in `pool.swap()`**: Add an `originator` parameter to `pool.swap()` that the pool passes to extensions as a separate argument. The router sets `originator = msg.sender`; direct callers set `originator = address(0)` (falling back to `sender`).

Either approach must be authenticated — only trusted routers may set a non-zero originator — to prevent a malicious caller from spoofing an allowlisted address.

## Proof of Concept
```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension in beforeSwap order
  admin allowlists Alice:  allowedSwapper[pool][alice] = true
  admin allowlists router: allowedSwapper[pool][router] = true
    (necessary so Alice can use the router)

Attack:
  Bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, ..., recipient: bob})

Execution trace:
  router.exactInputSingle()          msg.sender = bob
    pool.swap(recipient=bob, ...)    msg.sender = router
      _beforeSwap(sender=router, ...)
        SwapAllowlistExtension.beforeSwap(sender=router, ...)
          allowedSwapper[pool][router] == true  → PASSES
      swap executes, bob receives tokens

Result:
  Bob swaps successfully on a pool he was explicitly excluded from.
  The allowlist is bypassed without any privileged action.
```

Foundry test: deploy pool with `SwapAllowlistExtension`, allowlist Alice and the router, call `router.exactInputSingle` from Bob (not allowlisted), assert the swap succeeds and Bob receives output tokens.
