### Title
Slow-Mode Deposit Delay Allows Liquidators to Front-Run Users' Protective Collateral Deposits During Sequencer Downtime — (`Endpoint.sol`)

---

### Summary

When the Nado sequencer is offline, users can only protect their positions by calling `depositCollateral` directly on-chain. However, this function transfers tokens immediately but only queues a slow-mode transaction with a hardcoded **3-day delay** before the balance is credited. When the sequencer resumes, it can process `LiquidateSubaccount` transactions submitted by liquidators before it executes the queued slow-mode deposits. This allows liquidators to unfairly liquidate users who have already committed funds to protect their positions, causing direct loss of collateral.

---

### Finding Description

In `Endpoint.sol`, `depositCollateral` performs two distinct steps:

1. **Immediate token transfer** — tokens are pulled from the user and sent to the clearinghouse.
2. **Deferred balance credit** — a `SlowModeTx` is enqueued with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY`. [1](#0-0) 

`SLOW_MODE_TX_DELAY` is hardcoded to **3 days**. [2](#0-1) 

The balance update in `Clearinghouse.depositCollateral` — the call to `spotEngine.updateBalance` — only executes when the slow-mode transaction is eventually processed. [3](#0-2) 

The sequencer controls when slow-mode transactions are executed. It does so by including a `TransactionType.ExecuteSlowMode` entry in a submitted batch. Critically, there is **no on-chain ordering constraint** that forces the sequencer to drain the slow-mode queue before processing `LiquidateSubaccount` transactions. [4](#0-3) 

Liquidation is processed as a normal sequencer transaction via `processTransactionImpl`, entirely independent of the slow-mode queue. [5](#0-4) 

The health check inside `liquidateSubaccountImpl` reads the live on-chain balance, which does **not** include the pending slow-mode deposit. [6](#0-5) 

---

### Impact Explanation

A user who calls `depositCollateral` to rescue an unhealthy position has their tokens locked in the clearinghouse for up to 3 days without any health benefit. During this window — or immediately when the sequencer resumes after downtime — a liquidator can submit a `LiquidateSubaccount` transaction. The sequencer processes the liquidation first, the user is liquidated at a discount price, and the liquidator captures the liquidation fee fraction (50% of the spread between oracle price and liquidation price). [7](#0-6) 

The user suffers a double loss: their deposited collateral is credited only after the liquidation has already stripped their position, and they paid the liquidation penalty on top.

---

### Likelihood Explanation

Sequencer downtime is a realistic operational scenario in any sequencer-based protocol (network issues, maintenance, upgrades). During downtime, `depositCollateral` is the **only** on-chain path available to users to protect their positions — withdrawals, order matching, and position closure all require the sequencer. When the sequencer resumes, liquidation bots will immediately scan for unhealthy subaccounts and submit `LiquidateSubaccount` transactions. The sequencer has no protocol-enforced obligation to flush the slow-mode queue before processing those liquidations. The 3-day delay makes it virtually certain that any position that became unhealthy during downtime will be liquidated before the protective deposit is credited.

---

### Recommendation

1. **Immediate balance credit on deposit**: Credit the subaccount balance atomically inside `depositCollateral` rather than deferring it through the slow-mode queue. The token transfer already happens immediately, so the accounting should match.

2. **Sequencer-enforced ordering**: If the slow-mode queue design must be preserved, require the sequencer to execute all pending slow-mode deposits before processing any `LiquidateSubaccount` transaction in the same batch.

3. **Grace period after resumption**: Analogous to the M-24 recommendation, implement a configurable grace window after sequencer resumption during which liquidations are blocked, giving users time to have their queued deposits processed.

---

### Proof of Concept

1. User holds a leveraged perp position near the maintenance margin threshold.
2. The sequencer goes offline (network outage, maintenance).
3. Oracle prices move adversely; the user's position crosses below the maintenance health threshold.
4. The user calls `Endpoint.depositCollateral(subaccountName, productId, amount)` on-chain.
   - `handleDepositTransfer` pulls tokens from the user and sends them to the clearinghouse immediately. [8](#0-7) 
   - A `SlowModeTx` is enqueued with `executableAt = block.timestamp + 259200` (3 days). [9](#0-8) 
   - `spotEngine.updateBalance` is **not** called; the user's on-chain balance is unchanged.
5. The sequencer resumes. A liquidator signs and submits a `LiquidateSubaccount` transaction off-chain to the sequencer.
6. The sequencer calls `submitTransactionsChecked` with a batch containing only the `LiquidateSubaccount` transaction — no `ExecuteSlowMode` entry.
7. `liquidateSubaccountImpl` checks `isUnderMaintenance(txn.liquidatee)` — the user's balance still does not include the pending deposit, so the check passes. [10](#0-9) 
8. The liquidation executes: the user's position is closed at a discount, the liquidator receives the spread, and the liquidation fee is added to the insurance fund.
9. The sequencer later executes the slow-mode deposit — but the position is already gone and the user has already absorbed the liquidation loss.

        TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );
        if (txType == TransactionType.ExecuteSlowMode) {
            SlowModeConfig memory _slowModeConfig = slowModeConfig;
            _executeSlowModeTransaction(_slowModeConfig, true);
            slowModeConfig = _slowModeConfig;

            _delegatecallEndpointTx(
                abi.encodeWithSelector(

    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
