Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router as the swapper instead of the originating user, enabling allowlist bypass or blocking allowlisted users from the router path - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` bound to `msg.sender` of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the originating user. If a pool admin allowlists the router address to enable router-mediated swaps on a curated pool, every unpermissioned user can bypass the allowlist entirely. Conversely, if individual users are allowlisted (the intended use case), those users cannot swap through the router because the extension sees the router address and reverts.

## Finding Description
In `MetricOmmPool.swap`, the pool passes `msg.sender` (its direct caller) as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← direct caller, i.e. the router
  recipient,
  ...
);
```

`SwapAllowlistExtension.beforeSwap` then gates on that value:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

When `MetricOmmSimpleRouter.exactInputSingle` is called, it invokes `pool.swap(params.recipient, ...)` with itself as `msg.sender`:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    ...
  );
```

So the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The same misbinding applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

**Exploit path (allowlist bypass):**
1. Pool admin deploys pool with `SwapAllowlistExtension`.
2. Admin calls `setAllowedToSwap(pool, router, true)` — the natural step to enable router-mediated swaps.
3. Attacker (not individually allowlisted) calls `router.exactInputSingle({pool, recipient: attacker, ...})`.
4. Router calls `pool.swap(attacker, ...)` with `msg.sender = router`.
5. Extension checks `allowedSwapper[pool][router] == true` → passes.
6. Attacker executes a swap on the curated pool without being allowlisted.

**Broken path (allowlisted user blocked):**
1. Admin calls `setAllowedToSwap(pool, alice, true)`. Router is not allowlisted.
2. Alice calls `router.exactInputSingle(...)`.
3. Extension sees `sender = router`, `allowedSwapper[pool][router] == false` → reverts `NotAllowedToSwap`.
4. Alice (legitimately allowlisted) cannot use the standard periphery path.

Existing unit tests only simulate the pool calling the extension directly (`vm.prank(address(pool))`), and the integration test uses `setAllowAllSwappers` rather than individual allowlisting, so neither test surface catches this misbinding.

## Impact Explanation
**High.** A pool admin who allowlists the router to support the standard periphery path loses the entire allowlist guarantee: every unpermissioned address can execute swaps on the curated pool. This is a direct bypass of the access-control guard with no additional privilege required beyond calling the public router. It also constitutes broken core pool functionality: allowlisted users cannot use the supported periphery swap path.

## Likelihood Explanation
**High.** The router is the canonical swap entry point deployed alongside the pool. A pool admin who wants to support both direct and router-mediated swaps on a curated pool will allowlist the router. The misbinding is invisible from the admin's perspective because `isAllowedToSwap(pool, router)` returns `true` as intended, yet the actual enforcement is wrong. No special attacker capability is required beyond calling the public router.

## Recommendation
The extension must check the original user, not the intermediary. The most targeted fix without a core interface change is to gate on `recipient` instead of `sender` in `SwapAllowlistExtension.beforeSwap`. The router passes `params.recipient` (the actual user) as the `recipient` argument to `pool.swap`, so the extension already receives the correct address:

```diff
- function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
+ function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
  {
-   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
+   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
      revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
```

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (intending to enable router-mediated swaps)
  - Pool admin does NOT allowlist attacker address

Attack:
  - attacker calls router.exactInputSingle({pool, recipient: attacker, ...})
  - router calls pool.swap(attacker, ...)  [msg.sender = router]
  - pool calls _beforeSwap(router, attacker, ...)
  - SwapAllowlistExtension.beforeSwap(sender=router, recipient=attacker, ...)
  - allowedSwapper[pool][router] == true  → passes
  - attacker executes swap on curated pool without being individually allowlisted

Result:
  - Swap allowlist is completely bypassed
  - Any user can trade on the curated pool via the router
```

Foundry test: deploy pool with `SwapAllowlistExtension`, call `setAllowedToSwap(pool, address(router), true)`, then call `router.exactInputSingle` from an address not in the allowlist and assert the swap succeeds. Separately, call `setAllowedToSwap(pool, alice, true)` without allowlisting the router, then call `router.exactInputSingle` as alice and assert it reverts with `NotAllowedToSwap`.
