### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery — (`File: contract/src/lib.rs`)

### Summary
The deprecated `verify_transaction_inclusion` (v1) function is still reachable by any unprivileged NEAR caller. It accepts a caller-supplied `tx_id` without validating that it is a leaf-level transaction hash. An attacker can supply an internal Merkle tree node as `tx_id` and craft a matching proof, causing the function to return `true` for a transaction that does not exist in the block.

### Finding Description
`verify_transaction_inclusion` is marked `#[deprecated]` but carries no runtime access restriction — no `#[private]`, no role guard, and no `#[pause]` that would block it independently of the contract's global pause state. [1](#0-0) 

The function's own documentation acknowledges the broken invariant: [2](#0-1) 

The verification logic in `compute_root_from_merkle_proof` is a pure hash-chain computation with no leaf-vs-internal-node distinction: [3](#0-2) 

Each level hashes a 64-byte concatenation of two 32-byte child hashes: [4](#0-3) 

Because a 32-byte internal node is structurally identical to a 32-byte leaf hash from the function's perspective, an attacker can supply an internal node as `tx_id` and a proof that walks up the tree from that node, producing the correct `merkle_root`. The final comparison: [5](#0-4) 

…passes, and the function returns `true`.

`verify_transaction_inclusion_v2` was introduced specifically to close this gap by requiring a coinbase proof at index 0, which forces the proof depth to be consistent with a real leaf: [6](#0-5) 

But v1 remains live and callable.

### Impact Explanation
Any recipient contract or off-chain application that calls `verify_transaction_inclusion` and acts on a `true` result — releasing funds, confirming a cross-chain payment, unlocking collateral — can be deceived into accepting a fabricated transaction inclusion proof. The corrupted value is the boolean proof result returned to the caller, which is the sole output of the SPV verification API.

### Likelihood Explanation
The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` with arbitrary `ProofArgs`. The attacker only needs a block already accepted into the mainchain (its `merkle_root` is public state) and the ability to compute SHA-256 offline to craft a valid internal-node proof. The 64-byte Merkle forgery technique is well-documented and tooled.

### Recommendation
Remove `verify_transaction_inclusion` (v1) from the contract's public interface entirely, or add an explicit `#[private]` attribute and a role guard so it cannot be called by unprivileged accounts. All callers should be migrated to `verify_transaction_inclusion_v2`, which enforces coinbase proof consistency and closes the internal-node forgery path.

### Proof of Concept
1. Identify any block `B` accepted in the mainchain; read its `merkle_root` `R` from `get_last_block_header()`.
2. Offline, pick any internal node `N` at depth `d` in `B`'s Merkle tree (e.g., the hash of the two coinbase-side children at level 1). `N` is a valid 32-byte value.
3. Construct `merkle_proof` of length `d` such that `compute_root_from_merkle_proof(N, index, proof) == R`. This is straightforward because the remaining siblings above `N` are known from the block.
4. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_block_blockhash = B.hash`, `tx_index = index`, `merkle_proof = proof`, `confirmations = 1`.
5. The function returns `true`, falsely asserting that the non-existent transaction `N` is included in block `B`. [7](#0-6)

        since = "0.5.0",

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
