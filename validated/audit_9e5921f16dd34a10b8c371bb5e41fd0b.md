### Title
Division-by-Zero in `PerpEngine.socializeSubaccount` When `openInterest == 0` Permanently Blocks Liquidation Finalization — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when residual bad debt remains after insurance is exhausted. There is no zero-guard on the divisor. When `state.openInterest == 0` — a reachable state for any product where all position holders have closed their `amount` — `MathSD21x18.div` reverts with `ERR_DIV_BY_ZERO`, causing the entire `_finalizeSubaccount` call in `ClearinghouseLiq` to revert and permanently blocking finalization of the bankrupt subaccount.

---

### Finding Description

**The division site:**

In `PerpEngine.socializeSubaccount`, after insurance is applied, if `balance.vQuoteBalance` is still negative the code socializes the residual loss across all open-interest holders:

```solidity
// core/contracts/PerpEngine.sol lines 164-168
if (balance.vQuoteBalance < 0) {
    int128 fundingPerShare = -balance.vQuoteBalance.div(
        state.openInterest          // ← divisor, no zero-guard
    );
``` [1](#0-0)

    unchecked {
        require(y != 0, ERR_DIV_BY_ZERO);

**How `openInterest` reaches zero while `vQuoteBalance` stays negative:**

`state.openInterest` is the sum of `|balance.amount|` across all subaccounts for a product. In `_updateBalance`:

```solidity
// core/contracts/PerpEngineState.sol lines 30, 45-50
state.openInterest -= balance.amount.abs();   // remove old contribution
// ...
if (balance.amount > 0) {
    state.openInterest += balance.amount;      // add new long
} else {
    state.openInterest -= balance.amount;      // add new short (abs)
}
``` [3](#0-2) 

When a position is liquidated to `amount = 0`, the subaccount's contribution to OI is removed. `vQuoteBalance` is **not** cleared — it accumulates realized PnL independently. A subaccount can therefore have `amount == 0` (no OI contribution) and `vQuoteBalance < 0` (unrealized loss from the closed trade) simultaneously.

If the liquidatee was the only participant in a product, or all other participants have closed their positions, `state.openInterest == 0` while the liquidatee's `vQuoteBalance < 0` persists.

**The call chain that triggers the revert:**

`_finalizeSubaccount` is entered when `txn.productId == type(uint32).max`:

```solidity
// core/contracts/ClearinghouseLiq.sol lines 284-286
if (txn.productId != type(uint32).max) {
    return false;
}
``` [4](#0-3) 

It requires all perp `balance.amount == 0` before proceeding:

```solidity
// core/contracts/ClearinghouseLiq.sol lines 313-320
for (uint32 i = 0; i < v.perpIds.length; ++i) {
    uint32 perpId = v.perpIds[i];
    IPerpEngine.Balance memory balance = perpEngine.getBalance(perpId, txn.liquidatee);
    require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
}
``` [5](#0-4) 

Then calls `perpEngine.socializeSubaccount`:

```solidity
// core/contracts/ClearinghouseLiq.sol lines 386-389
v.insurance = perpEngine.socializeSubaccount(
    txn.liquidatee,
    v.insurance
);
``` [6](#0-5) 

The `require(balance.amount == 0)` check at line 319 **guarantees** that the liquidatee's OI contribution is zero. If no other subaccount holds an open position in that product, `state.openInterest == 0` at the point of the division.

---

### Impact Explanation

- `_finalizeSubaccount` reverts every time it is called for the affected subaccount.
- The bankrupt subaccount's bad debt is never socialized; the loss is never distributed to other participants.
- The insurance fund balance is never decremented to reflect the absorbed loss.
- The protocol's accounting invariant — that all bad debt is resolved through socialization — is permanently broken for that product/subaccount pair.
- The subaccount remains in a zombie state: it cannot be finalized, its negative `vQuoteBalance` is never cleared, and the protocol is left undercollateralized by the unresolved amount.

---

### Likelihood Explanation

The condition is reachable in any product that has ever had a single dominant participant (or where all other participants have closed positions before the liquidatee is finalized). This is realistic for:
- Newly launched products with low participation
- Products where market makers have withdrawn all positions
- Any scenario where the liquidatee is the last remaining open-interest holder

No privileged access, governance capture, or external oracle manipulation is required. A liquidator calling `liquidateSubaccount` with `productId == type(uint32).max` on a qualifying subaccount is sufficient.

---

### Recommendation

Add a zero-guard before the division in `socializeSubaccount`. If `state.openInterest == 0`, there are no other participants to socialize the loss to; the residual bad debt should be absorbed by the insurance fund or written off without attempting the division:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; absorb or write off
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
``` [7](#0-6)
