Audit Report

## Title
Uninitialized `lastMidPriceX64` in `PriceVelocityGuardExtension` causes velocity guard to fail open on first swap — (File: `metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary
`PriceVelocityGuardExtension` stores per-pool velocity state in `priceVelocityState[pool_]`, a mapping whose entries default to all-zero. The `beforeSwap` hook unconditionally skips velocity enforcement when `prevMid == 0`. Because `lastMidPriceX64` is never seeded during pool creation or extension initialization, every pool's first swap bypasses the velocity cap entirely, regardless of how extreme the oracle-derived mid-price is.

## Finding Description
`PriceVelocityGuardExtension.beforeSwap` reads `s.lastMidPriceX64` into `prevMid` and gates all velocity arithmetic behind `if (prevMid != 0)`:

```solidity
// PriceVelocityGuardExtension.sol lines 53-76
PriceVelocityState storage s = priceVelocityState[pool_];
uint128 prevMid = s.lastMidPriceX64;   // zero for any new pool
uint64  prevBlock = s.lastUpdateBlock;

s.lastMidPriceX64 = midPrice;
s.lastUpdateBlock = uint64(block.number);

if (prevMid != 0) {                    // guard entirely skipped when zero
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        // velocity arithmetic and potential revert
    }
}
```

The mapping entry for a newly-registered pool is zero-initialized by the EVM. `PriceVelocityGuardExtension` does not override `initialize`; the base `BaseMetricExtension.initialize` (lines 41–43) simply returns the selector without writing any velocity state. The factory's `createPool` calls `initialize` on each extension (lines 206–210 of `MetricOmmPoolFactory.sol`), but this call is a no-op for velocity state. The only seeding mechanism is the admin-callable `setLastMidPrice` (lines 29–34), which requires a separate transaction after pool creation. Any swap that executes before that transaction bypasses the guard completely: `prevMid == 0` causes the outer branch to be skipped, `s.lastMidPriceX64` is written to the current (potentially extreme) mid-price, and `IMetricOmmExtensions.beforeSwap.selector` is returned — success.

## Impact Explanation
The velocity guard is the designated on-chain circuit-breaker against rapidly-moved oracle prices reaching pool swap math. When it fails open on the first swap, a swap executes at an oracle price that may have moved by an arbitrary percentage, with no enforcement of the configured `maxChangePerBlockE18` cap. LPs suffer adverse selection: the counterparty trades at a price the guard was configured to block, extracting value from LP reserves. This is a direct, concrete instance of **bad-price execution** (an unclamped oracle quote reaches a pool swap) and **oracle guard path failure** (velocity guard fails open instead of closed on a zero/uninitialized prior observation), both within the allowed impact gate.

## Likelihood Explanation
Every pool using `PriceVelocityGuardExtension` starts with `lastMidPriceX64 == 0`. The admin must perform a separate `setLastMidPrice` transaction after pool creation; there is no atomic initialization path. Any swap landing between pool creation and that admin call bypasses the guard. An attacker monitoring the mempool can backrun the pool-creation transaction with a swap in the same block, exploiting the unguarded window. No special permissions are required beyond the ability to call `pool.swap()`.

## Recommendation
Seed `lastMidPriceX64` atomically at pool registration time. Override `initialize` in `PriceVelocityGuardExtension` to query the pool's price provider for the current mid-price and write it to `priceVelocityState[pool_]`, so the guard is active from the very first swap. Alternatively, change the guard logic to fail closed when `prevMid == 0`: if `maxChangePerBlockE18 != 0` but no baseline has been set, revert, forcing the admin to explicitly seed state before any swap is permitted.

## Proof of Concept
1. Admin deploys a pool with `PriceVelocityGuardExtension` attached and calls `setMaxChangePerBlock(pool, 1e16)` (1% per block cap). `priceVelocityState[pool].lastMidPriceX64 == 0`.
2. Admin has not yet called `setLastMidPrice`.
3. Oracle price moves 40% in one block (legitimate market event or oracle update lag).
4. Attacker backruns the pool-creation transaction with `pool.swap(...)`.
5. `beforeSwap` executes: `prevMid = s.lastMidPriceX64 = 0`. The `if (prevMid != 0)` branch is skipped. The hook returns `IMetricOmmExtensions.beforeSwap.selector` — success.
6. The swap executes at the 40%-moved oracle price. The 1% velocity cap is never enforced.
7. `s.lastMidPriceX64` is written to the current extreme mid-price; subsequent swaps are checked — but LP funds from step 6 are already lost.
