Unhandled Refund Promise Failure Permanently Locks NEAR Tokens in Contract - (File: contract/src/lib.rs)

### Summary
The `submit_blocks` function in `contract/src/lib.rs` issues a fire-and-forget `Promise::transfer(refund)` to return excess deposit to the caller. There is no `.then()` failure callback. In NEAR, when a returned `Promise` fails, the state changes from the originating function are already committed and are **not** rolled back. If the transfer promise fails, the refund NEAR tokens are returned to the contract's own balance with no on-chain mechanism to recover or re-distribute them, permanently locking those funds.

### Finding Description
In `submit_blocks`, after computing the storage cost and the excess deposit, the contract issues:

```rust
let refund = amount.saturating_sub(required_deposit);
if refund > NearToken::from_near(0) {
    Promise::new(env::predecessor_account_id())
        .transfer(refund)
        .into()
}
``` [1](#0-0) 

No `.then(callback)` is chained onto the promise to detect or handle failure. In NEAR's asynchronous execution model, the function's state mutations (header insertions, GC updates) are committed to storage when the function returns. The `Promise::transfer` is dispatched as a separate async receipt in a subsequent block. If that receipt fails — because the predecessor account no longer exists at execution time — the NEAR runtime returns the tokens to the contract's own balance. The contract has no storage slot, no pending-refunds map, and no admin withdrawal function to recover those tokens. They are permanently locked.

The analogous root cause to the original report is identical in class: a native-token transfer is issued with no error-handling path, so a failure silently discards the recipient's funds.

### Impact Explanation
Any NEAR tokens that constitute an overpayment refund and whose transfer promise fails are permanently locked inside the `BtcLightClient` contract. There is no `withdraw`, `claim_refund`, or admin-drain function anywhere in `contract/src/lib.rs`. The locked balance accumulates silently and is irrecoverable without a contract upgrade. [2](#0-1) 

### Likelihood Explanation
The concrete failure condition is: the predecessor account is deleted (via `DeleteAccount` action) between the block in which `submit_blocks` executes and the block in which the transfer receipt executes (typically the very next block). NEAR accounts can be deleted by their owner at any time. A relayer that rotates keys by deleting and recreating an account, or a smart-contract account that self-destructs as part of a multi-step operation, falls into this window. The window is one block (~1 second on NEAR mainnet), making accidental triggering unlikely but not impossible, and deliberate triggering (e.g., a relayer that overpays and then deletes its account) straightforward. Likelihood is **Low**, but the impact is irreversible.

### Recommendation
Chain a `.then()` callback on the refund promise that stores the refund amount in a persistent map keyed by account ID if the transfer fails, and expose a separate `claim_refund()` entry point. Alternatively, use the established NEAR pattern of checking the promise result in a callback and re-crediting a `pending_refunds: LookupMap<AccountId, NearToken>` field:

```rust
Promise::new(env::predecessor_account_id())
    .transfer(refund)
    .then(
        Self::ext(env::current_account_id())
            .on_refund_complete(env::predecessor_account_id(), refund)
    )
    .into()
```

The `on_refund_complete` callback inspects `env::promise_result(0)` and, on failure, inserts the amount into a `pending_refunds` map so the caller can retrieve it later.

### Proof of Concept

1. Relayer account `relayer.near` calls `submit_blocks` attaching `2 NEAR` when only `0.5 NEAR` of storage cost is incurred.
2. `submit_blocks` commits all header state changes and schedules `Promise::transfer(1.5 NEAR)` to `relayer.near`.
3. In the same block (or the immediately following one, before the transfer receipt executes), `relayer.near` issues a `DeleteAccount` action pointing its beneficiary to a different account.
4. When the transfer receipt executes in the next block, `relayer.near` no longer exists; the NEAR runtime cannot credit the account and returns the `1.5 NEAR` to the contract's balance.
5. The contract has no `pending_refunds` map and no withdrawal function. The `1.5 NEAR` is permanently locked. [3](#0-2)
