### Title
Premature Deletion of Slow Mode Transaction Before External Call Execution Causes Irrecoverable User Fund Loss — (`core/contracts/Endpoint.sol`)

---

### Summary

In `Endpoint.sol`, `_executeSlowModeTransaction` deletes the queued `SlowModeTx` entry from storage **before** attempting to execute it via an external `try/catch` call. If the external call fails for any non-OOG reason, the transaction is permanently gone, the catch block does nothing (a removed refund mechanism is confirmed by an inline comment), and any user funds already transferred to the clearinghouse at deposit time are irrecoverably locked with no balance credit.

---

### Finding Description

`_executeSlowModeTransaction` in `Endpoint.sol` follows this exact sequence:

```solidity
// 1. State is deleted FIRST — transaction is now unrecoverable
SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
delete slowModeTxs[_slowModeConfig.txUpTo++];   // line 194

// 2. External call that can fail
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    // OOG check ...
    // try return funds now removed          // line 226 — explicit admission
}
``` [1](#0-0) 

The comment `// try return funds now removed` at line 226 is a direct admission that a refund path was intentionally stripped out, leaving the catch block as a silent no-op for all non-OOG failures.

**Concrete trigger path — `DepositCollateral` slow mode:**

1. A user calls `depositCollateralWithReferral` (or `depositCollateral`). At this point, `handleDepositTransfer` immediately moves the user's ERC-20 tokens from `msg.sender` into the `clearinghouse` contract. [2](#0-1) 

2. A `SlowModeTx` of type `DepositCollateral` is queued with a 3-day delay (`SLOW_MODE_TX_DELAY`). [3](#0-2) 

3. Anyone (no access control) calls `executeSlowModeTransaction()`. [4](#0-3) 

4. `_executeSlowModeTransaction` deletes the queued entry, then calls `this.processSlowModeTransaction`, which delegates to `EndpointTx.processSlowModeTransactionImpl`, which calls `clearinghouse.depositCollateral(txn)`. [5](#0-4) 

5. If `clearinghouse.depositCollateral` reverts (e.g., the product was delisted in the 3-day window, a contract upgrade introduced a new validation, or any other revert condition), the catch block fires. The `slowModeTxs` entry is already deleted. No balance is credited. The user's tokens remain in the clearinghouse with no accounting entry and no replay path.

The same pattern applies to `DepositInsurance` slow mode transactions, where tokens are also transferred at submission time in `submitSlowModeTransactionImpl`: [6](#0-5) 

---

### Impact Explanation

- User ERC-20 tokens are transferred to `clearinghouse` at deposit time but the corresponding balance credit in `SpotEngine` never occurs.
- The `slowModeTxs` entry is deleted before the call, so there is no replay mechanism.
- The catch block explicitly has no refund logic (confirmed by the `// try return funds now removed` comment).
- Funds are permanently locked in the clearinghouse with no on-chain recovery path for the user.
- Severity: **High** — direct, permanent asset loss for any user whose slow mode deposit transaction fails at execution time.

---

### Likelihood Explanation

- `executeSlowModeTransaction()` is permissionless — any address can trigger execution.
- The 3-day delay (`SLOW_MODE_TX_DELAY`) between deposit and execution creates a realistic window for conditions to change: a product can be delisted via `clearinghouse.delistProduct`, a contract upgrade can introduce new validation, or a product's token address can be zeroed.
- The `delistProduct` function is callable by the sequencer/owner and explicitly removes products, which would cause `clearinghouse.depositCollateral` to revert for that product. [7](#0-6) 
- The inline comment `// try return funds now removed` confirms the developers were aware of this gap and chose not to implement a recovery path.

---

### Recommendation

Move the `delete` to **after** a successful execution, mirroring the fix recommended in H-07:

```solidity
function _executeSlowModeTransaction(...) internal {
    SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
    // Do NOT delete here

    require(fromSequencer || (txn.executableAt <= block.timestamp), ...);

    try this.processSlowModeTransaction(txn.sender, txn.tx) {
        delete slowModeTxs[_slowModeConfig.txUpTo++]; // delete only on success
    } catch {
        if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
            assembly { invalid() }
        }
        // On non-OOG failure: leave the entry intact so it can be retried,
        // OR implement a refund path that returns tokens to the original sender.
    }
}
```

Additionally, reinstate a refund mechanism for `DepositCollateral` and `DepositInsurance` slow mode transactions that fail at execution time, so that users can recover their tokens even if the transaction cannot be replayed.

---

### Proof of Concept

1. User calls `depositCollateralWithReferral(subaccount, productId, 1000e6, "")`. Tokens are transferred to `clearinghouse`. A `SlowModeTx` is queued at index `N`.
2. Within the 3-day window, the sequencer calls `delistProduct` for `productId`, removing it from the protocol.
3. After 3 days, anyone calls `executeSlowModeTransaction()`.
4. `_executeSlowModeTransaction` executes: `delete slowModeTxs[N]` fires at line 194.
5. `this.processSlowModeTransaction` is called → `clearinghouse.depositCollateral` reverts because the product no longer exists.
6. The catch block at line 207 fires. `gasleft()` is well above 250,000, so the OOG guard does not trigger.
7. The catch block exits silently (`// try return funds now removed`).
8. `slowModeTxs[N]` is gone. The user's 1000 USDC is in the clearinghouse. No balance is credited. No replay is possible. Funds are permanently lost. [8](#0-7)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
```
