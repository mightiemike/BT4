### Title
`ContractOwner` Lacks `receive()` Function, Permanently Locking Native ETH in `DirectDepositV1` Contracts - (File: `core/contracts/ContractOwner.sol`)

### Summary

`ContractOwner.withdrawFromDirectDepositV1` is the designated recovery path for native ETH stuck in a `DirectDepositV1` (DDA) contract. It calls `DirectDepositV1.withdrawNative()`, which pushes ETH to `msg.sender` (i.e., `ContractOwner`) via a low-level `.call{value: ...}("")`. Because `ContractOwner` declares no `receive()` or `fallback()` function, this ETH transfer always reverts, making the native-ETH recovery path permanently broken and any ETH that accumulates in a DDA permanently unrecoverable.

### Finding Description

`DirectDepositV1.withdrawNative()` transfers the contract's entire ETH balance to its caller:

```solidity
function withdrawNative() external onlyOwner {
    uint256 balance = address(this).balance;
    (bool success, ) = msg.sender.call{value: balance}("");
    require(success, "Failed to transfer native token to owner");
}
``` [1](#0-0) 

The only production caller of this function is `ContractOwner.withdrawFromDirectDepositV1`:

```solidity
if (token == address(0)) {
    uint256 preBalance = address(this).balance;
    DirectDepositV1(directDepositV1).withdrawNative();
    uint256 postBalance = address(this).balance;
    require(postBalance > preBalance, "empty");
    ...
}
``` [2](#0-1) 

When `withdrawNative()` executes, `msg.sender` is `ContractOwner`. The low-level call sends ETH to `ContractOwner`, but `ContractOwner` has **no `receive()` or `fallback()` function**. A grep across all production Solidity files confirms only `DirectDepositV1.sol` and `MockWrappedNative.sol` (a test utility) define `receive()`: [3](#0-2) 

`ContractOwner` inherits only `EIP712Upgradeable` and `OwnableUpgradeable`, neither of which defines a `receive()` function. Therefore, the `.call{value: balance}("")` in `withdrawNative()` always returns `success = false`, causing the `require` to revert with `"Failed to transfer native token to owner"`. [4](#0-3) 

### Impact Explanation

Any native ETH that accumulates inside a `DirectDepositV1` contract becomes permanently unrecoverable through the intended protocol path. ETH can reach a DDA's balance via two concrete mechanisms:

1. **Constructor pre-funding failure**: DDAs are deployed with a fixed `salt: bytes32(uint256(1))`, making their CREATE2 address deterministic and predictable. If ETH is sent to the DDA's address before deployment, the constructor attempts to wrap it but explicitly does **not** revert on failure — it emits `NativeTokenTransferFailed` and continues. The ETH then remains in the DDA with no recovery path. [5](#0-4) 

2. **`selfdestruct` force-send**: Any contract can force-send ETH to the DDA via `selfdestruct`, bypassing the `receive()` function entirely. The resulting ETH balance is then irrecoverable.

In both cases, the owner calls `withdrawFromDirectDepositV1(subaccount, address(0))`, which always reverts. The ETH is permanently locked in the DDA. [6](#0-5) 

### Likelihood Explanation

The DDA's `receive()` function does `require(success, "Failed to wrap native token.")`, so ETH sent via normal transfers is either wrapped or the transaction reverts — ETH does not silently accumulate under normal operation. However, the constructor path (pre-funded CREATE2 address with a failing `wrappedNative` call) is a realistic griefing vector given the deterministic salt, and `selfdestruct` force-sends are always possible. The broken recovery function is a latent, always-present defect that activates the moment any ETH reaches a DDA balance through either path. [3](#0-2) 

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept ETH pushed from `DirectDepositV1.withdrawNative()`:

```solidity
receive() external payable {}
```

This mirrors the fix class from the reference report: ensure the contract that must receive ETH actually has a payable entry point. Alternatively, refactor `withdrawNative()` to accept a `sendTo` address parameter so ETH can be sent directly to the multisig owner rather than routing through `ContractOwner`.

### Proof of Concept

1. Compute the deterministic CREATE2 address for a DDA with `salt = bytes32(uint256(1))` and the known `ContractOwner` deployer.
2. Send 1 ETH to that address before `createDirectDepositV1` is called.
3. Call `ContractOwner.creditDepositV1(subaccount)` (or `createDirectDepositV1` directly). The constructor runs, attempts `wrappedNative.call{value: 1 ether}("")`. If this call fails (e.g., wrappedNative is paused or not yet deployed), `NativeTokenTransferFailed` is emitted and 1 ETH remains in the DDA.
4. Call `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))` as the owner.
5. Internally, `DirectDepositV1.withdrawNative()` executes `ContractOwner.call{value: 1 ether}("")`. `ContractOwner` has no `receive()` → `success = false` → `require` reverts with `"Failed to transfer native token to owner"`.
6. The 1 ETH is permanently locked in the DDA. [1](#0-0) [2](#0-1)

        onlyOwner

        address payable directDepositV1 = directDepositV1Address[subaccount];

        if (token == address(0)) {
