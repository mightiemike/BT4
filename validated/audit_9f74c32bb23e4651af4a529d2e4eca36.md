### Title
Interest Not Accrued Before Config Update in `SpotEngine.addOrUpdateProduct()` — (File: `core/contracts/SpotEngine.sol`)

---

### Summary

`SpotEngine.addOrUpdateProduct()` overwrites the interest-rate `Config` for an existing product without first calling `updateStates()` to settle accrued interest under the old parameters. The next sequencer-submitted `SpotTick` will then apply the new rates across the entire elapsed `dt`, retroactively mis-pricing interest for all depositors and borrowers of that product.

---

### Finding Description

`SpotEngineState._updateState()` computes the borrow rate and deposit rate entirely from `configs[productId]`: [1](#0-0) 

The five fields that drive the rate curve are `interestFloorX18`, `interestSmallCapX18`, `interestLargeCapX18`, `interestInflectionUtilX18`, and `minDepositRateX18`. These are read fresh on every call to `_updateState`, so whatever value is stored at call time is applied to the full `dt` window.

`SpotEngine.addOrUpdateProduct()` unconditionally replaces `configs[productId]` for both new and existing products: [2](#0-1) 

There is no call to `updateStates()` (or the internal `_updateState`) before the overwrite. `updateStates` is `onlyEndpoint` and is driven by the sequencer via a `SpotTick` transaction: [3](#0-2) [4](#0-3) 

Because `addOrUpdateProduct` is called directly on the engine (not through the sequencer pipeline), no `SpotTick` is guaranteed to precede it. The gap between the last `SpotTick` and the config change is therefore silently absorbed into the next `SpotTick`'s `dt`, and the new rates are applied to that entire window.

`ContractOwner.addOrUpdateProducts()` is the standard governance path that calls `spotEngine.addOrUpdateProduct()`: [5](#0-4) 

---

### Impact Explanation

**Scenario A — rate increase:** If `interestFloorX18`, `interestSmallCapX18`, or `interestLargeCapX18` is raised, the next `SpotTick` applies the higher borrow rate to the full `dt` since the last tick. Borrowers are retroactively charged more than they accrued under the old rate; depositors receive a windfall at borrowers' expense.

**Scenario B — rate decrease:** The symmetric case silently under-charges borrowers and under-pays depositors for the pre-change portion of `dt`.

**Scenario C — `minDepositRateX18` increase:** The minimum deposit subsidy is applied to the full `dt`, inflating `cumulativeDepositsMultiplierX18` and `cumulativeBorrowsMultiplierX18` beyond what was earned, distorting every user's real balance.

In all cases the corrupted multipliers (`cumulativeDepositsMultiplierX18`, `cumulativeBorrowsMultiplierX18`) are stored permanently in `states[productId]` and affect every subsequent balance read and health check. [6](#0-5) 

---

### Likelihood Explanation

Updating interest-rate parameters is a routine governance operation (e.g., responding to utilization changes or market conditions). `ContractOwner.addOrUpdateProducts()` is the documented path for doing so. No malicious intent is required; the mis-accounting occurs on every legitimate config update for an existing product that has non-zero deposits.

---

### Recommendation

Before overwriting `configs[productId]` for an existing product in `SpotEngine.addOrUpdateProduct()`, the protocol should ensure interest is settled under the old config. The cleanest fix is for the sequencer to always submit a `SpotTick` transaction immediately before any `addOrUpdateProduct` call in the same submission batch. Alternatively, `addOrUpdateProduct` can be restructured to call `_updateState` internally using a `dt` derived from the oracle time stored in `EndpointStorage.times.spotTime`, then update the config only after settlement.

---

### Proof of Concept

1. At time `T0`, `SpotTick` is processed; `times.spotTime = T0`. `interestFloorX18 = 1e16` (1 %).
2. At time `T1 = T0 + 86400` (one day later), the owner calls `addOrUpdateProducts()` → `spotEngine.addOrUpdateProduct(...)` with `interestFloorX18 = 5e16` (5 %). No `SpotTick` is submitted first.
3. At time `T2 = T1 + 1`, the sequencer submits `SpotTick` with `dt = T2 - T0 = 86401`.
4. `_updateState` runs with `interestFloorX18 = 5e16` and `dt = 86401 s` — the full day is charged at 5 % instead of 1 %, retroactively over-charging every borrower for the 86 400 seconds that elapsed before the config change. [7](#0-6) [8](#0-7)

    function _updateState(
        uint32 productId,
        State memory state,
        uint128 dt
    ) internal {
        int128 borrowRateMultiplierX18;
        int128 totalDeposits = state.totalDepositsNormalized.mul(
            state.cumulativeDepositsMultiplierX18
        );
        int128 totalBorrows = state.totalBorrowsNormalized.mul(
            state.cumulativeBorrowsMultiplierX18
        );
        int128 utilizationRatioX18 = totalBorrows.div(totalDeposits);
        int128 minDepositRateX18;
        {
            Config memory config = configs[productId];

            // annualized borrower rate
            int128 borrowerRateX18 = config.interestFloorX18;
            if (utilizationRatioX18 == 0) {
                // setting borrowerRateX18 to 0 here has the property that
                // adding a product at the beginning of time and not using it until time T
                // results in the same state as adding the product at time T
                borrowerRateX18 = 0;
            } else if (utilizationRatioX18 < config.interestInflectionUtilX18) {
                borrowerRateX18 += config
                    .interestSmallCapX18
                    .mul(utilizationRatioX18)
                    .div(config.interestInflectionUtilX18);
            } else {
                borrowerRateX18 +=
                    config.interestSmallCapX18 +
                    config.interestLargeCapX18.mul(
                        (
                            (utilizationRatioX18 -
                                config.interestInflectionUtilX18).div(
                                    ONE - config.interestInflectionUtilX18
                                )
                        )
                    );
            }

            // convert to per second
            borrowerRateX18 = borrowerRateX18.div(
                MathSD21x18.fromInt(31536000)
            );
            borrowRateMultiplierX18 = (ONE + borrowerRateX18).pow(int128(dt));
            minDepositRateX18 = config.minDepositRateX18;
        }

    function addOrUpdateProduct(
        uint32 productId,
        uint32 quoteId,
        int128 sizeIncrement,
        int128 minSize,
        Config calldata config,
        RiskHelper.RiskStore calldata riskStore
    ) public onlyOwner {
        bool isNewProduct = _addOrUpdateProduct(
            productId,
            quoteId,
            sizeIncrement,
            minSize,
            riskStore
        );
        configs[productId] = config;

        if (isNewProduct) {
            require(productId != QUOTE_PRODUCT_ID);
            _setState(
                productId,
                State({
                    cumulativeDepositsMultiplierX18: ONE,
                    cumulativeBorrowsMultiplierX18: ONE,
                    totalDepositsNormalized: 0,
                    totalBorrowsNormalized: 0
                })
            );
        }
    }
