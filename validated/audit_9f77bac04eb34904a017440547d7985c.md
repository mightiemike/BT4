Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual depositor, i.e., `msg.sender` of `addLiquidity`) and instead validates `owner`, a free parameter supplied by the caller. Because any caller can set `owner` to any allowlisted address, the allowlist is trivially bypassed. The unauthorized caller's tokens are permanently locked under `owner`'s LP position with no recovery path.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the LP-position beneficiary to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` correctly forwards both addresses to the extension hook: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` names the first parameter with a blank identifier (discarding `sender`) and evaluates only `owner`: [3](#0-2) 

Since `owner` is freely chosen by the caller, any address can pass the guard by supplying an allowlisted address as `owner`. The actual depositor (`sender`) is never checked. After the deposit executes, the caller cannot recover their tokens because `removeLiquidity` enforces `msg.sender == owner`: [4](#0-3) 

The analogous `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (not `recipient`), confirming the intended design and the deviation in `DepositAllowlistExtension`: [5](#0-4) 

## Impact Explanation
Any pool configured with `DepositAllowlistExtension` and a non-empty allowlist (e.g., KYC-gated or institutional-only pools) accepts deposits from any address. The unauthorized depositor's tokens are pulled via the modify-liquidity callback, bin balances and `binTotals` are updated, and LP shares are credited to `owner`. The depositor holds no position and cannot call `removeLiquidity` (`NotPositionOwner` reverts). This constitutes a direct, permanent loss of user principal and a complete bypass of a core access-control mechanism — matching the Critical/High allowed impact gate (broken core pool functionality causing loss of funds, admin-boundary break by an unprivileged path).

## Likelihood Explanation
Exploitability requires only knowing one allowlisted address, which is observable on-chain via `AllowedToDepositSet` events or direct `allowedDepositor` mapping reads. No special role, flash loan, or oracle manipulation is needed — a plain `addLiquidity` call suffices. Every pool deploying `DepositAllowlistExtension` with a non-empty allowlist is affected, and the attack is repeatable at will.

## Recommendation
Check `sender` (the first parameter) instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][alice] = true
  bob is NOT allowlisted

Attack:
  bob calls pool.addLiquidity(
      owner  = alice,   // allowlisted — guard passes
      salt   = 0,
      deltas = { binIdxs: [0], shares: [1e18] },
      callbackData = "",
      extensionData = ""
  )

  Inside beforeAddLiquidity:
    sender (bob) is silently discarded
    owner  (alice) passes the allowlist check → no revert

  LiquidityLib.addLiquidity executes:
    - bin balances updated
    - positionBinShares[keccak(alice, 0, 0)] += 1e18
    - callback fires, pulling bob's tokens into the pool

Result:
  - bob's tokens are permanently in the pool
  - alice holds the LP position
  - bob calls removeLiquidity(owner=alice, ...) → NotPositionOwner revert
  - pool's deposit restriction is silently bypassed
```
