### Title
`ValidResourceBounds` Variant Heuristic in Protobuf Converter Causes Hash Domain Collision and P2P Rejection of Valid Invoke V3 Transactions — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf converter for `ValidResourceBounds` uses a value-based heuristic (`l1_data_gas.is_zero() && l2_gas.is_zero()`) to decide between the `L1Gas` and `AllResources` variants. This heuristic conflates two structurally distinct transaction types: old pre-0.13.3 transactions (which genuinely carry only L1 gas bounds) and new `AllResources` transactions that happen to have zero `l2_gas` and `l1_data_gas`. The gateway always accepts the latter as valid, but the P2P layer rejects them because the protobuf round-trip produces `L1Gas`, which then fails the `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` conversion. Additionally, the two variants produce structurally different hash preimages in `get_tip_resource_bounds_hash`, so any node that does accept the transaction computes a different `TransactionHash` than the originating node.

---

### Finding Description

**Step 1 — Gateway accepts the transaction.**

`RpcInvokeTransactionV3` stores `resource_bounds: AllResourceBounds` (not `ValidResourceBounds`). The gateway validator accepts a transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` — this is an explicitly tested valid case (`valid_l1_gas`). [1](#0-0) 

**Step 2 — Hash is computed using `AllResources`.**

`InternalRpcInvokeTransactionV3` also stores `resource_bounds: AllResourceBounds`. Its `InvokeTransactionV3Trait::resource_bounds()` implementation unconditionally wraps the stored value in `ValidResourceBounds::AllResources(...)`: [2](#0-1) 

`get_invoke_transaction_v3_hash` then calls `get_tip_resource_bounds_hash` with `AllResources`, which includes `L1_DATA_GAS` (even when zero) in the Poseidon hash chain: [3](#0-2) 

So the originating node computes:
```
poseidon(tip, concat(L1_GAS, X), concat(L2_GAS, 0), concat(L1_DATA, 0))
```

**Step 3 — Protobuf converter applies a value-based heuristic.**

When the transaction is serialized to protobuf and deserialized by a peer, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` fires: [4](#0-3) 

Because `l1_data_gas.is_zero() && l2_gas.is_zero()`, the converter produces `ValidResourceBounds::L1Gas(l1_gas)` — the old pre-0.13.3 variant — even though the originating transaction was a new-style `AllResources` transaction.

**Step 4 — P2P conversion fails, rejecting the valid transaction.**

The P2P mempool path converts `protobuf::InvokeV3WithProof → InvokeTransactionV3 → RpcInvokeTransactionV3`. The second step uses `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3`, which explicitly rejects any `resource_bounds` that is not `AllResources`: [5](#0-4) 

The protobuf converter documents this failure path with the comment: *"This conversion can fail only if the resource_bounds are not AllResources."* [6](#0-5) 

The transaction is silently dropped at the P2P layer.

**Step 5 — Hash domain collision if the transaction reaches a second node.**

If the transaction were to reach a peer via any other path, `InvokeTransactionV3::resource_bounds()` returns the stored `ValidResourceBounds::L1Gas(X)` directly: [7](#0-6) 

`get_tip_resource_bounds_hash` with `L1Gas` omits `L1_DATA_GAS` entirely: [8](#0-7) 

The peer computes:
```
poseidon(tip, concat(L1_GAS, X), concat(L2_GAS, 0))
```

This is a different Felt from the originating node's hash. The `TransactionHash` stored in the originating node's mempool and the hash recomputed by the peer are structurally divergent — the same transaction bytes bind to two different canonical identifiers depending on which conversion path is taken.

---

### Impact Explanation

A valid `RpcInvokeTransactionV3` with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is accepted by the gateway but permanently dropped at the P2P propagation boundary. No peer sequencer ever receives it. The transaction is never sequenced. This matches the **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

The secondary hash-domain collision means that if the transaction were somehow admitted through a non-P2P path, the `TransactionHash` computed by the receiving node would differ from the one computed by the originating node, potentially causing the transaction to be treated as a duplicate of a non-existent transaction or accepted under a wrong hash — matching the **High** impact: *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

---

### Likelihood Explanation

Any user submitting an Invoke V3 transaction that specifies only L1 gas bounds (setting `l2_gas` and `l1_data_gas` to zero) triggers this path. The gateway validator explicitly tests and accepts this as a valid configuration (`valid_l1_gas` test case). The trigger requires no special privileges — it is a normal, unprivileged RPC submission.

---

### Recommendation

The protobuf converter must not use a value-based heuristic to determine the `ValidResourceBounds` variant. The protobuf schema should carry an explicit discriminant field (e.g., a `bounds_type` enum: `L1_GAS_ONLY` vs `ALL_RESOURCES`) so that the variant is preserved losslessly across serialization. Until the schema is updated, the converter should default to `AllResources` for all transactions received over the new P2P path (post-0.13.3), reserving `L1Gas` only for transactions explicitly tagged as legacy. The `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` conversion should not be used as a gate in the P2P deserialization path for new-style transactions.

---

### Proof of Concept

1. Construct a valid `RpcInvokeTransactionV3` with `resource_bounds = AllResourceBounds { l1_gas: ResourceBounds { max_amount: 100, max_price_per_unit: 1 }, l2_gas: ResourceBounds::default(), l1_data_gas: ResourceBounds::default() }`.
2. Submit it to the gateway — it is accepted (matches the `valid_l1_gas` test case in `stateless_transaction_validator_test.rs`).
3. Serialize the transaction to protobuf bytes (as done in `RpcTransactionBatch`).
4. Deserialize the bytes on a peer node via `RpcTransactionBatch::try_from(bytes)`.
5. Observe that `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` produces `ValidResourceBounds::L1Gas(...)` because `l1_data_gas.is_zero() && l2_gas.is_zero()`.
6. Observe that the subsequent `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` returns `Err(DEPRECATED_RESOURCE_BOUNDS_ERROR)` and the transaction is dropped.
7. Confirm the hash divergence: compute `get_invoke_transaction_v3_hash` on the originating `InternalRpcInvokeTransactionV3` (uses `AllResources`, includes `L1_DATA_GAS = 0` in hash) and on the deserialized `InvokeTransactionV3` (uses `L1Gas`, omits `L1_DATA_GAS`). The two Felt values differ.

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}

impl TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3 {
    type Error = ProtobufConversionError;
    fn try_from(mut value: protobuf::InvokeV3WithProof) -> Result<Self, Self::Error> {
        // Extract proof first, since `starknet_api::transaction::InvokeTransactionV3` does not
        // carry a `proof` field.
        let proof = Proof::from(std::mem::take(&mut value.proof));

        let snapi_invoke: InvokeTransactionV3 = value
            .invoke
            .ok_or(ProtobufConversionError::MissingField {
                field_description: "InvokeV3WithProof::invoke",
            })?
            .try_into()?;

        // This conversion can fail only if the resource_bounds are not AllResources.
        Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
    }
}
