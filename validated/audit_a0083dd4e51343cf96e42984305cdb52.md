### Title
Locked Tier Position Principal Inaccessible When Bonus Pool Is Depleted After Exit Commitment Elapses - (File: `x/tieredrewards/keeper/msg_server.go`)

### Summary
All exit paths from a tier position (`MsgTierUndelegate`, `MsgExitTierWithDelegation`, `MsgClearPosition`) unconditionally settle bonus rewards before executing the exit. If the bonus pool is empty and the user has accrued but unclaimed bonus, every exit path fails atomically, leaving the user's locked principal inaccessible until an external actor replenishes the pool.

### Finding Description
After a position's exit commitment elapses, the user has exactly two exit paths:

**Path A — `MsgExitTierWithDelegation`**: calls `claimRewards` unconditionally before transferring the delegation back to the owner. [1](#0-0) 

**Path B — `MsgTierUndelegate` → `MsgWithdrawFromTier`**: `TierUndelegate` calls `claimRewards` unconditionally before starting the staking unbonding period. [2](#0-1) 

`MsgWithdrawFromTier` does not call `claimRewards`, but it requires the position to be undelegated first — which is only reachable through `TierUndelegate`. [3](#0-2) 

**Escape hatch — `MsgClearPosition`**: would cancel the exit and allow redelegation, but it also calls `claimRewards` whenever an exit has been triggered. [4](#0-3) 

**`MsgTierRedelegate`** is explicitly blocked once the exit commitment has elapsed. [5](#0-4) 

The bonus pool failure mode is documented: when `pool balance < accrued bonus`, `claimRewards` returns an error and the entire transaction is rolled back. [6](#0-5) 

This creates a complete exit blockade: every message that can move the position toward withdrawal first demands a successful bonus settlement. If the pool is empty, none of them can proceed.

### Impact Explanation
**Impact: Medium.** The corrupted value is the user's locked principal — `validator.TokensFromShares(pos.Delegation.Shares)` tokens held at the position's per-position delegator address (`pos.DelegatorAddress`). These tokens cannot be returned to the owner until the bonus pool is replenished by an external actor. If governance never replenishes the pool, the principal is permanently inaccessible. This is a direct loss of user funds, not merely a reward dilution. [7](#0-6) 

### Likelihood Explanation
**Likelihood: Medium.** Two independent conditions must coincide:

1. The user has accrued but unclaimed bonus (normal for long-duration tiers — 1-year or 5-year exit durations mean users may not claim frequently).
2. The bonus pool balance falls below the user's accrued bonus at the moment they attempt to exit.

The pool is funded by external sources. A sustained period without replenishment — or a large simultaneous claim wave by other users — can deplete it. Neither condition requires privileged access or an attacker; both arise from normal protocol operation.

### Recommendation
Decouple principal withdrawal from bonus settlement. `MsgTierUndelegate` should proceed with the staking unbonding even when the bonus pool is insufficient, recording the unclaimed bonus for later collection. Alternatively, introduce a `ForceUndelegate` path that skips bonus settlement entirely and allows the user to claim the bonus separately once the pool is replenished. The invariant to preserve is that bonus is not double-counted, not that it must be paid atomically before the principal can move.

### Proof of Concept
1. User calls `MsgLockTier` with a 1-year exit duration; validator is bonded; bonus accrues over months.
2. User calls `MsgTriggerExitFromTier`; exit clock starts.
3. One year elapses; `block_time >= ExitUnlockAt`.
4. The bonus pool is depleted (e.g., many other users claimed simultaneously).
5. User attempts every available exit path:
   - `MsgExitTierWithDelegation` → `claimRewards` fails → `ErrInsufficientBonusPool` → tx reverted. [1](#0-0) 
   - `MsgTierUndelegate` → `claimRewards` fails → `ErrInsufficientBonusPool` → tx reverted. [2](#0-1) 
   - `MsgTierRedelegate` → rejected: `ErrExitLockDurationElapsed`. [5](#0-4) 
   - `MsgClearPosition` → `claimRewards` fails → `ErrInsufficientBonusPool` → tx reverted. [8](#0-7) 
6. The user's locked principal remains escrowed at `pos.DelegatorAddress` with no available exit path until the pool is externally replenished. [9](#0-8)

```go
func (k Keeper) validateWithdrawFromTier(ctx context.Context, pos types.PositionState, owner string) error {
	if !pos.IsOwner(owner) {
		return types.ErrNotPositionOwner
	}

	if !pos.HasTriggeredExit() {
		return types.ErrExitNotTriggered
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if !pos.CompletedExitLockDuration(sdkCtx.BlockTime()) {
		return types.ErrExitLockDurationNotReached
	}

	if pos.IsDelegated() {
		return types.ErrPositionDelegated
	}

	isUnbonding, err := k.isUnbonding(ctx, pos.DelegatorAddress)
	if err != nil {
		return err
	}
	if isUnbonding {
		return types.ErrPositionUnbonding
	}

	return nil
```
