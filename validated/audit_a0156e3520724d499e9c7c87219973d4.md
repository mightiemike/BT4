### Title
Linked Signer Can Permanently Hijack Subaccount Authority via `LinkSigner` — (`core/contracts/EndpointTx.sol`)

### Summary

`processTransactionImpl` handles `TransactionType.LinkSigner` by calling `validateSignedTx` with `allowLinkedSigner = true`. This permits the currently-linked signer to sign a `LinkSigner` transaction that replaces themselves with any arbitrary address, including an attacker-controlled one. Only the subaccount owner (the address encoded in the `sender` bytes32) should be able to reassign `linkedSigners[sender]`, but the code grants this power to the linked signer as well.

### Finding Description

In `processTransactionImpl`, the `LinkSigner` branch is:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner=true` calls `validateSignature`, which passes `getLinkedSigner(sender)` to the verifier:

```solidity
// EndpointTx.sol lines 172–184
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` accepts a signature from **either** the subaccount owner **or** the linked signer:

```solidity
// Verifier.sol lines 297–303
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

Therefore, if `signerA` is the current linked signer for `subaccountA`, `signerA` can sign `LinkSigner{sender=subaccountA, signer=attackerAddr, nonce=N}` and the transaction will pass all validation, writing `linkedSigners[subaccountA] = attackerAddr`.

**Contrast with the slow-mode path**, which correctly restricts `LinkSigner` to the subaccount owner only:

```solidity
// EndpointTx.sol lines 232–239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // msg.sender must be the owner
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
}
``` [4](#0-3) 

The sequencer path (`processTransactionImpl`) lacks this owner-only guard.

### Impact Explanation

Once `linkedSigners[subaccountA]` is set to `attackerAddr`, the attacker controls all future signed transactions for `subaccountA` that accept a linked signer, including:
- `WithdrawCollateral` (called with `allowLinkedSigner=true`) — direct asset theft
- `TransferQuote` — fund redirection
- `LiquidateSubaccount` — forced liquidation of the victim's positions

The subaccount owner loses the ability to recover control unless they submit a corrective `LinkSigner` through the slow-mode path (which requires a 3-day delay and gas fees), during which the attacker can drain all assets.

### Likelihood Explanation

Any address that has ever been granted linked-signer status can execute this attack unilaterally. No additional privileges, leaked keys, or admin access are required. The attacker only needs to be the current linked signer and have the sequencer include their transaction — a normal operational condition.

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may reassign the linked signer
);
```

This aligns the sequencer path with the slow-mode path, which already enforces owner-only authorization via `validateSender`.

### Proof of Concept

State-machine test on an unmodified Hardhat setup:

1. Deploy the protocol; create `subaccountA` owned by `ownerAddr`.
2. `ownerAddr` submits `LinkSigner{sender=subaccountA, signer=signerA, nonce=0}` — sets `linkedSigners[subaccountA] = signerA`.
3. `signerA` constructs `LinkSigner{sender=subaccountA, signer=attackerAddr, nonce=1}`, signs it with `signerA`'s private key, and submits it through `processTransactionImpl`.
4. `validateSignedTx` passes: nonce is correct; `Verifier.validateSignature` recovers `signerA`, which equals `getLinkedSigner(subaccountA)`.
5. Assert: `linkedSigners[subaccountA] == attackerAddr`. ✓
6. `attackerAddr` signs `WithdrawCollateral{sender=subaccountA, ...}` and submits it — funds are withdrawn to the attacker. ✓
