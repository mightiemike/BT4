### Title
Insufficient Lower-Bound Validation on `confirmations` in `verify_transaction_inclusion` Allows Zero-Confirmation Proof Acceptance — (`File: contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept `confirmations = 0` without any lower-bound check. Because the confirmation-depth guard reduces to `any_u64 >= 0` (always true for `u64`), any unprivileged NEAR caller can obtain a `true` proof result for a transaction that has zero on-chain confirmations, completely bypassing the intended confirmation-depth security model.

### Finding Description
`verify_transaction_inclusion` enforces only an upper bound on `confirmations`:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [1](#0-0) 

There is no lower-bound check. The subsequent depth guard is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [2](#0-1) 

When `args.confirmations = 0`, the right-hand side is `0`, and the left-hand side is a `u64` that is always `>= 0`. The guard is therefore trivially satisfied for **any** block on the main chain, regardless of how recently it was submitted. The function then proceeds to the Merkle proof check and returns `true` if the proof is structurally valid.

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after its own coinbase-proof check, so it inherits the same flaw:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

Neither function carries `#[trusted_relayer]` or any role-based access control — only `#[pause]` — so both are reachable by any unprivileged NEAR account when the contract is not paused. [4](#0-3) [5](#0-4) 

The `ProofArgs` struct exposes `confirmations` as a plain `u64` with no constraints at the type level:

```rust
pub struct ProofArgs {
    pub tx_id: H256,
    pub tx_block_blockhash: H256,
    pub tx_index: u64,
    pub merkle_proof: Vec<H256>,
    pub confirmations: u64,
}
``` [6](#0-5) 

### Impact Explanation
The purpose of the `confirmations` parameter is to let a recipient contract (e.g., a bridge or escrow) require that a Bitcoin transaction be buried under a minimum number of blocks before releasing funds or triggering an action. With `confirmations = 0` accepted, a caller can obtain a `true` verification result the instant a block containing the target transaction is submitted to the light client — before any meaningful finality has accumulated. A bridge contract that forwards the caller-supplied `confirmations` value, or that fails to enforce its own minimum, would release funds against a zero-confirmation Bitcoin transaction, enabling a double-spend attack: the attacker broadcasts a conflicting transaction on the Bitcoin network while simultaneously claiming the bridged asset on NEAR.

The corrupted invariant is the **proof result**: `verify_transaction_inclusion` is supposed to attest that a transaction has at least N confirmations; with `confirmations = 0` the attestation is vacuously true and carries no security guarantee.

### Likelihood Explanation
Medium. Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are public, permissionless view functions. Any NEAR account can call them with an arbitrary `confirmations` value. The attack requires only that a downstream recipient contract does not independently enforce a minimum confirmation count — a realistic assumption for contracts that treat the light client as the authoritative source of confirmation depth.

### Recommendation
Add an explicit lower-bound check at the top of `verify_transaction_inclusion`:

```rust
require!(args.confirmations >= 1, "confirmations must be at least 1");
```

For stronger guarantees, enforce a protocol-meaningful minimum (e.g., 6 for Bitcoin mainnet) and document the expected range in the public API. The same check should be applied before the delegation in `verify_transaction_inclusion_v2`.

### Proof of Concept
1. Relayer submits a single block header containing transaction `T` to the light client. The block is accepted and becomes the chain tip.
2. Attacker immediately calls `verify_transaction_inclusion` with:
   - `tx_id` = hash of `T`
   - `tx_block_blockhash` = hash of the just-submitted block
   - `tx_index` = correct index of `T`
   - `merkle_proof` = valid Merkle path for `T`
   - `confirmations = 0`
3. The `confirmations <= gc_threshold` check passes (`0 <= gc_threshold`).
4. The depth check evaluates `(tip_height - tip_height) + 1 >= 0` → `1 >= 0` → `true`.
5. The Merkle proof check passes (proof is valid).
6. The function returns `true`.
7. A recipient bridge contract that relies on this result releases funds, while the attacker simultaneously broadcasts a double-spend on the Bitcoin network.
