### Title
Per-Tier `nonDefaultFeeTierMask` Conflates Tier-Level Customization with Per-Product Fee Configuration, Causing Zero Fees on Unconfigured Products — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`nonDefaultFeeTierMask` is a per-tier bitmask that gates whether `getTierFeeRateX18` returns the stored custom rate or the protocol default (2 bps taker). However, `feeRates` is stored per `(tier, productId)`. When an admin sets a custom fee rate for a single product in a tier, the entire tier is marked non-default, causing every other product in that tier to silently return zero fees — because their `feeRates[tier][productId]` slots were never written and remain zero-initialized.

---

### Finding Description

`updateTierFeeRates` has two code paths:

1. `txn.productId == QUOTE_PRODUCT_ID` → writes `feeRates[tier][p]` for **every** spot and perp product, then sets the tier bit.
2. `txn.productId != QUOTE_PRODUCT_ID` → writes `feeRates[tier][txn.productId]` for **one** product, then sets the tier bit. [1](#0-0) 

In path 2, `nonDefaultFeeTierMask |= uint128(1) << txn.tier` marks the entire tier as non-default even though only one product's rate was written. [2](#0-1) 

`getTierFeeRateX18` checks only the tier bit. Once set, it returns `feeRates[tier][productId]` for **any** product. For products never explicitly configured in that tier, this mapping slot is zero-initialized, returning `FeeRates({makerRateX18: 0, takerRateX18: 0})` — zero taker fee — instead of the default 2 bps.

The conflation is exact: the "is this tier customized?" predicate uses `tier` as its sole key, but the actual fee rate data uses `(tier, productId)` as a composite key. These two scopes are mismatched. [3](#0-2) 

---

### Impact Explanation

Any trader assigned to a tier that has been partially customized (i.e., one product's rate was set via the single-product path) will pay **zero taker fees** on all other products in that tier. The protocol loses all taker fee revenue from those trades. The `collectedFees` and `sequencerFee` accounting in `applyFee` and `chargeFee` are both driven by the fee rate returned from `getTierFeeRateX18`, so the accounting corruption is direct and permanent for each affected trade. [4](#0-3) 

---

### Likelihood Explanation

The admin legitimately uses the single-product path when they want to give one market a differentiated fee rate (e.g., a new listing, a promotional rate, or a builder-specific override). This is a realistic operational action. Once triggered, every user in that tier immediately benefits from zero fees on all other products — no further attacker action is required beyond normal trading through `matchOrders`. [5](#0-4) 

---

### Recommendation

Decouple the "has this (tier, product) been explicitly configured?" predicate from the tier-level mask. Replace `nonDefaultFeeTierMask` with a per-`(tier, productId)` boolean mapping, or initialize all existing products' fee rates to the protocol default whenever a tier is first marked non-default. Alternatively, require that the single-product path can only be called after the tier has already been fully initialized via the `QUOTE_PRODUCT_ID` bulk path.

---

### Proof of Concept

1. Protocol has products A (id=1) and B (id=2), both in tier 0 (default). All users pay 2 bps taker fee.
2. Admin calls `updateTierFeeRates` with `tier=1, productId=1, takerRateX18=100_000_000_000_000` (1 bps for product A).
3. `feeRates[1][1]` is written; `nonDefaultFeeTierMask |= 1 << 1` is set. `feeRates[1][2]` is never written (zero).
4. Admin assigns trader Alice to tier 1 via `updateFeeTier`.
5. Alice submits a `MatchOrders` transaction for product B.
6. `getUserFeeRateWithBuilder` calls `getTierFeeRateX18(1, 2)`.
7. `nonDefaultFeeTierMask & (1 << 1) != 0` → returns `feeRates[1][2]` = `FeeRates(0, 0)`.
8. Alice pays zero taker fee. `market.collectedFees` receives 0. Protocol loses the 2 bps it would have collected. [6](#0-5)

    function getTierFeeRateX18(uint32 tier, uint32 productId)
        public
        view
        returns (FeeRates memory)
    {
        if (nonDefaultFeeTierMask & (1 << tier) != 0) {
            return feeRates[tier][productId];
        }
        return
            FeeRates({
                makerRateX18: 0,
                takerRateX18: 200_000_000_000_000 // 2 bps
            });
    }

    function updateTierFeeRates(IEndpoint.UpdateTierFeeRates memory txn)
        external
        onlyEndpoint
    {
        if (txn.productId == QUOTE_PRODUCT_ID) {
            uint32[] memory spotProductIds = spotEngine.getProductIds();
            uint32[] memory perpProductIds = perpEngine.getProductIds();
            for (uint32 i = 0; i < spotProductIds.length; i++) {
                if (spotProductIds[i] == QUOTE_PRODUCT_ID) {
                    continue;
                }
                feeRates[txn.tier][spotProductIds[i]] = FeeRates(
                    txn.makerRateX18,
                    txn.takerRateX18
                );
            }
            for (uint32 i = 0; i < perpProductIds.length; i++) {
                feeRates[txn.tier][perpProductIds[i]] = FeeRates(
                    txn.makerRateX18,
                    txn.takerRateX18
                );
            }
        } else {
            feeRates[txn.tier][txn.productId] = FeeRates(
                txn.makerRateX18,
                txn.takerRateX18
            );
        }
        nonDefaultFeeTierMask |= uint128(1) << txn.tier;
    }
