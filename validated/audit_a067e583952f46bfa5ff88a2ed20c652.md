Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the actual swapper, allowing full allowlist bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of the pool's `swap` call — the router address when `MetricOmmSimpleRouter` is used. Because the router is a single shared contract, allowlisting it grants every user unrestricted swap access, completely defeating the per-user gate. `DepositAllowlistExtension` does not share this flaw: it checks `owner` (the actual position owner), not `sender` (the adder contract).

## Finding Description
**Call chain — swap via router:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle(params)`. [1](#0-0) 

2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — so inside `MetricOmmPool.swap`, `msg.sender` = **router**. [2](#0-1) 

3. Pool calls `_beforeSwap(msg.sender, recipient, ...)`, forwarding the **router address** as `sender`. [3](#0-2) 

4. `SwapAllowlistExtension.beforeSwap(sender=router, ...)` evaluates `allowedSwapper[msg.sender/*pool*/][sender/*router*/]`. [4](#0-3) 

**Contrast — deposit via liquidity adder:**

`DepositAllowlistExtension.beforeAddLiquidity` ignores `sender` (the adder contract) and checks `owner` — the actual position owner passed through `pool.addLiquidity(positionOwner, ...)`. [5](#0-4) [6](#0-5) 

**Two concrete failure modes:**

- **Bypass (primary impact):** Pool admin allowlists the router address (`setAllowedToSwap(pool, router, true)`) so that router-mediated swaps work. Because the check is `allowedSwapper[pool][router]`, every user who calls the router passes the gate regardless of their own allowlist status. The per-user restriction is entirely void.
- **Lockout (secondary impact):** Pool admin allowlists individual EOAs but not the router. Those EOAs cannot swap through the router even though they are individually permitted; they must call the pool directly, breaking the standard periphery UX.

## Impact Explanation
The swap allowlist extension is the sole on-chain mechanism for restricting which addresses may trade in a pool. When the router is allowlisted (the natural configuration for any pool that supports periphery swaps), the gate is bypassed for all callers. Any non-allowlisted user can execute swaps in a pool explicitly configured to restrict access, potentially draining LP value from a pool designed for a closed set of counterparties. This is broken core pool functionality causing unauthorized access to restricted swap flows.

## Likelihood Explanation
The bypass requires no special privilege. Any user with token approval to the router can call `exactInputSingle` or `exactInput`. The precondition — that the router is allowlisted — is the expected operational state for any pool that intends to support the standard periphery. The exploit is repeatable every block with no setup beyond a standard ERC-20 approval.

## Recommendation
In `SwapAllowlistExtension.beforeSwap`, replace the `sender` check with a check against the `recipient` parameter (the economic beneficiary), or — preferably — redesign the extension to accept the actual end-user identity via `extensionData` forwarded by the router. Alternatively, mirror the deposit pattern: have the router pass the original `msg.sender` as an explicit parameter and check that identity. The simplest safe fix is to check `recipient` (second argument to `beforeSwap`) when the intent is to gate who receives the output, or to require the router to embed the originating user in `extensionData` and verify it in the extension.

## Proof of Concept
```solidity
// Setup: pool with SwapAllowlistExtension; router is allowlisted, attacker is not
swapExtension.setAllowedToSwap(address(pool), address(router), true);
// attacker is NOT in allowedSwapper

// Attacker bypasses allowlist via router
vm.prank(attacker);
token0.approve(address(router), type(uint256).max);
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool: address(pool),
    tokenIn: address(token0),
    tokenOut: address(token1),
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    recipient: attacker,
    deadline: type(uint256).max,
    priceLimitX64: 0,
    extensionData: ""
}));
// Swap succeeds: allowedSwapper[pool][router] == true, attacker identity never checked
```
