### Title
Duplicate Position ID in `MsgClaimTierRewards` Enables Double-Claim of Bonus Rewards from Pool — (`x/tieredrewards/keeper/msg_server.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`ClaimTierRewards` loads all requested positions into an in-memory slice before processing any of them. If the same `position_id` appears more than once in `msg.PositionIds`, the second (and every subsequent) copy carries stale checkpoint values (`LastBonusAccrual`, `LastEventSeq`). Because the position state is only persisted to the KV store *after* the bonus transfer for each iteration, the stale copy re-computes and re-sends the same bonus window, draining the `RewardsPoolName` module account beyond what is legitimately owed.

---

### Finding Description

**Root cause — stale in-memory state used for duplicate position IDs**

`ClaimTierRewards` first loads every requested position into a local slice: [1](#0-0) 

All positions are fetched from the KV store at this point, before any reward is paid. The slice is then handed to `claimRewardsAndUpdatesPositions`: [2](#0-1) 

Inside `claimRewardsAndUpdatesPositions`, for each element the loop:
1. Calls `processEventsAndClaimBonus` — which reads `pos.LastBonusAccrual` and `pos.LastEventSeq` from the **in-memory** struct, computes the bonus, checks the pool, and **sends coins** to the owner.
2. Only then calls `setPosition` to persist the updated checkpoints. [3](#0-2) 

The bonus computation and transfer happen inside `processEventsAndClaimBonus`: [4](#0-3) 

The pool-sufficiency guard reads the **current** bank balance (correctly reduced after the first send), but the **position checkpoints** used to compute the bonus amount are taken from the stale in-memory copy, not re-read from the KV store: [5](#0-4) 

**CEI analogy:**
- *Check*: `sufficientBonusPoolBalance` (line 230)
- *Interaction*: `SendCoinsFromModuleToAccount` (line 239)
- *Effect*: `setPosition` (line 129 in the outer loop — persists updated checkpoints)

The effect (checkpoint persistence) happens after the interaction (coin transfer), and the positions are all pre-loaded before any effect is applied. A duplicate entry in `msg.PositionIds` therefore presents a stale `LastBonusAccrual`/`LastEventSeq` to the second iteration, causing the same bonus window to be paid out again.

**No deduplication guard exists** in the handler: [6](#0-5) 

---

### Impact Explanation

The `RewardsPoolName` module account balance is the exact value corrupted. An attacker who owns a position with accrued bonus can submit a single `MsgClaimTierRewards` transaction with `position_ids = [X, X, X, …, X]` (the same ID repeated *k* times). Each iteration re-computes the full bonus for the same time window and transfers it from the pool. The pool balance is drained by up to *k × bonus_per_claim*, bounded only by the pool's current balance (the `sufficientBonusPoolBalance` guard prevents the pool from going negative, but does not prevent repeated extraction up to that limit). This constitutes insolvency of the bonus pool: legitimate future claimants receive `ErrInsufficientBonusPool` errors and lose their entitled rewards.

---

### Likelihood Explanation

Any position owner can trigger this with a standard `MsgClaimTierRewards` transaction. No privileged role, leaked key, or social engineering is required. The attacker only needs one delegated position with non-zero accrued bonus. The attack is executable in a single transaction and is repeatable across blocks.

---

### Recommendation

1. **Deduplicate `position_ids` before processing.** Either in `MsgClaimTierRewards.Validate()` or at the start of the handler, reject or deduplicate any repeated IDs.
2. **Re-read position state from the KV store inside the loop** (or after each `setPosition` call) rather than relying on the pre-loaded in-memory slice, so that a duplicate entry would see the already-advanced checkpoints and compute zero bonus.
3. **Fix CEI order**: call `setPosition` to persist updated checkpoints *before* calling `SendCoinsFromModuleToAccount`, so that any re-entry or duplicate processing sees the post-claim state.

---

### Proof of Concept

1. Attacker creates a position (position ID = 7) and waits for bonus to accrue (e.g., 1 000 000 basecro owed).
2. Attacker submits:
   ```
   MsgClaimTierRewards{
     owner:        attacker,
     position_ids: [7, 7, 7, 7, 7],   // same ID five times
   }
   ```
3. Iteration 1 (positions[0], ID=7): `LastBonusAccrual = T0` → computes 1 000 000, sends 1 000 000 from pool, persists `LastBonusAccrual = T_now`.
4. Iteration 2 (positions[1], ID=7, **stale copy**): `LastBonusAccrual = T0` (old value) → computes 1 000 000 again, pool still has ≥ 1 000 000, sends another 1 000 000.
5. Iterations 3–5: same, until pool is exhausted.
6. Attacker receives 5 000 000 basecro instead of 1 000 000. Pool is drained; subsequent legitimate claimants receive `ErrInsufficientBonusPool`.

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}
