### Title
Off-by-One in Supply Cap Enforcement Allows MaxSupply to Be Exceeded Before Halt - (File: x/inflation/abci.go)

### Summary
The `BeginBlocker` in `x/inflation/abci.go` uses a strict greater-than (`GT`) comparison when enforcing the `MaxSupply` hard cap. When the circulating supply equals `MaxSupply` exactly, the check passes without halting. The Mint module then mints additional tokens in the next block, pushing the circulating supply above `MaxSupply` before the halt finally triggers. The hard cap is therefore exceeded by at least one block's worth of newly minted tokens.

### Finding Description
In `x/inflation/abci.go`, the supply cap enforcement reads:

```go
if maxsupply.IsPositive() && totalsupply.GT(maxsupply) {
    panic(...)
}
``` [1](#0-0) 

The execution order is:
1. Mint module `BeginBlocker` runs first and mints new tokens into the bank.
2. Inflation module `BeginBlocker` runs second and checks the resulting supply. [2](#0-1) 

When the circulating supply lands exactly on `MaxSupply` after a block's minting, the condition `totalsupply.GT(maxsupply)` evaluates to `false` (equal is not greater-than), so no halt occurs. In the very next block the Mint module mints another provision, making `totalsupply > maxsupply`, and only then does the halt trigger. The cap has already been breached by one full block's minting amount.

The test suite explicitly encodes this behavior as intentional:

```go
// GT, not GTE — should NOT panic
require.NotPanics(t, func() { ... })
``` [3](#0-2) 

This is the direct analog to the external report: the cap check does not account for the value that will be added in the immediately following operation (the next block's mint), so the maximum is exceeded at least once before enforcement fires.

### Impact Explanation
The `MaxSupply` parameter is documented as a **hard cap** on circulating supply. The off-by-one in the comparison means the cap is not truly hard: the circulating supply will always overshoot by one block's minting provision (at 1% annual inflation on ~100 B CRO, one block's provision is on the order of ~1,600 basecro per second of block time, but the overshoot is a concrete, measurable violation of the stated invariant). The corrupted value is the circulating supply, which exceeds `MaxSupply` by one block's `BlockProvision` before the chain halts. [4](#0-3) 

### Likelihood Explanation
This is not a probabilistic edge case. As the chain approaches `MaxSupply` through normal inflation, there will be exactly one block where the post-mint supply equals `MaxSupply`. The `GT` check passes silently. The following block's minting then breaches the cap. This is a deterministic outcome of normal chain operation — no attacker action is required.

### Recommendation
Change the comparison from strict greater-than to greater-than-or-equal:

```go
// Before (incorrect):
if maxsupply.IsPositive() && totalsupply.GT(maxsupply) {

// After (correct):
if maxsupply.IsPositive() && totalsupply.GTE(maxsupply) {
``` [5](#0-4) 

This ensures the chain halts the moment circulating supply reaches `MaxSupply`, preventing any further minting beyond the cap.

### Proof of Concept
Sequence of events on a live chain with `MaxSupply = M`:

1. **Block N−1**: Post-mint circulating supply = `M − ε` (just below cap). `GT` check: `M − ε > M` → false. Chain continues.
2. **Block N**: Mint module mints provision `P`. Post-mint circulating supply = `M − ε + P = M` (exactly at cap). `GT` check: `M > M` → false. **Chain does not halt.**
3. **Block N+1**: Mint module mints another provision `P`. Post-mint circulating supply = `M + P`. `GT` check: `M + P > M` → true. Chain halts.

Result: circulating supply has exceeded `MaxSupply` by `P` (one full block's minting provision) before enforcement fires. The hard cap invariant is violated for one block. [6](#0-5)

func BeginBlocker(ctx context.Context, k keeper.Keeper) error {
	defer telemetry.ModuleMeasureSince(types.ModuleName, telemetry.Now(), telemetry.MetricKeyBeginBlocker)
	params, err := k.GetParams(ctx)
	if err != nil {
		panic("could not get params: " + err.Error())
	}
	totalsupply, denom, err := k.GetSupplyAndDenom(ctx)
	if err != nil {
		panic("could not get supply and denom: " + err.Error())
	}

	burned := math.NewInt(0)
	for _, ba := range params.BurnedAddresses {
		balance, err := k.GetAddressBalance(ctx, ba, denom)
		if err != nil {
			panic(fmt.Sprintf("could not get %s balance: %s", ba, err.Error()))
		}
		burned = burned.Add(balance)
	}
	totalsupply = totalsupply.Sub(burned)

	maxsupply := params.MaxSupply
	if maxsupply.IsPositive() && totalsupply.GT(maxsupply) {
		panic(fmt.Sprintf("the total supply has exceeded the maximum supply: %s > %s", totalsupply, maxsupply))
	}

	return nil
```
