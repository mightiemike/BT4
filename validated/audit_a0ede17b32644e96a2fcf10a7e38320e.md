### Title
`feeX18` Excluded from EIP-712 Digest Allows Sequencer to Charge Arbitrary Fees on `WithdrawCollateralV2` - (File: core/contracts/Verifier.sol)

---

### Summary

The `SignedWithdrawCollateralV2` struct contains a `feeX18` field that is **not included** in the EIP-712 typed-data digest the user signs. The sequencer can freely set `feeX18` to any value between `0` and `currentFeeX18` when submitting the transaction on-chain, and the user's signature provides no cryptographic commitment to the fee they will actually be charged.

---

### Finding Description

`SignedWithdrawCollateralV2` is defined as:

```solidity
struct SignedWithdrawCollateralV2 {
    WithdrawCollateralV2 tx;
    CompactSignature signature;
    int128 feeX18;          // ← NOT in the signed digest
}
``` [1](#0-0) 

The EIP-712 type string covers only the inner `WithdrawCollateralV2` struct:

```
"WithdrawCollateralV2(bytes32 sender,uint32 productId,uint128 amount,uint64 nonce,address sendTo,uint128 appendix)"
``` [2](#0-1) 

`computeDigest` hashes only the six fields of `WithdrawCollateralV2`; `feeX18` is never encoded:

```solidity
digest = keccak256(
    abi.encode(
        keccak256(bytes(WITHDRAW_COLLATERAL_V2_SIGNATURE)),
        signedTx.tx.sender,
        signedTx.tx.productId,
        signedTx.tx.amount,
        signedTx.tx.nonce,
        signedTx.tx.sendTo,
        signedTx.tx.appendix   // feeX18 absent
    )
);
``` [3](#0-2) 

In `processTransactionImpl`, after signature validation passes, `feeX18` is read directly from the sequencer-supplied calldata and charged to the user:

```solidity
int128 currentFeeX18 = spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18;
require(signedTx.feeX18 >= 0);
require(signedTx.feeX18 <= currentFeeX18);
chargeFee(signedTx.tx.sender, signedTx.feeX18, signedTx.tx.productId);
``` [4](#0-3) 

The only constraint is `0 ≤ feeX18 ≤ currentFeeX18`. The user's signature provides zero commitment to which value within that range will be applied.

---

### Impact Explanation

A user signs a `WithdrawCollateralV2` transaction committing to a specific `sender`, `productId`, `amount`, `nonce`, `sendTo`, and `appendix`. They have no cryptographic guarantee of the fee they will pay. The sequencer can submit the signed transaction with `feeX18 = currentFeeX18` (the protocol maximum) regardless of any off-chain fee quote shown to the user. The user's collateral balance is reduced by the sequencer-chosen fee with no recourse, because the signature is valid for any `feeX18` in range. This is a direct, concrete asset delta: the user loses up to `currentFeeX18` more than they consented to.

---

### Likelihood Explanation

The sequencer is the sole entity that submits `WithdrawCollateralV2` transactions on-chain and constructs the `feeX18` field. No external attacker capability is required beyond normal sequencer operation. Any time the sequencer wishes to maximize fee revenue, it can set `feeX18 = currentFeeX18` on every withdrawal without violating any on-chain check. The entry path is the normal `submitTransactions` → `processTransactionImpl` → `WithdrawCollateralV2` branch, reachable on every user-initiated withdrawal using the V2 path.

---

### Recommendation

Include `feeX18` in the EIP-712 type string and digest so that users explicitly commit to the fee they accept:

```
"WithdrawCollateralV2(bytes32 sender,uint32 productId,uint128 amount,uint64 nonce,address sendTo,uint128 appendix,int128 feeX18)"
```

Update `computeDigest` in `Verifier.sol` to encode `feeX18` alongside the other fields. This mirrors the resolution in the referenced tBTC report: the root cause was that a critical parameter affecting user funds was outside the scope of the verification mechanism. Here, `feeX18` must be inside the signed envelope so the user's signature constitutes explicit consent to the exact fee charged.

---

### Proof of Concept

1. User signs `WithdrawCollateralV2` off-chain with `amount = 1000 USDC`, expecting a fee of `0` (as quoted by the UI).
2. Sequencer constructs `SignedWithdrawCollateralV2` with `feeX18 = currentFeeX18` (e.g., `5e15` = 0.5%).
3. Sequencer calls `submitTransactions` → `processTransactionImpl`.
4. `validateSignedTx` recovers the signer correctly — the digest does not include `feeX18`, so the signature is valid.
5. `chargeFee(signedTx.tx.sender, signedTx.feeX18, signedTx.tx.productId)` deducts the maximum fee from the user's balance.
6. `clearinghouse.withdrawCollateral` executes the withdrawal with the reduced net amount.
7. The user receives less than expected with no on-chain mechanism to dispute it, because their signature was valid for any `feeX18 ∈ [0, currentFeeX18]`. [5](#0-4) [3](#0-2)
