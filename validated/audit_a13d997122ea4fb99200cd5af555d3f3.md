Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual user — allowlist fully bypassable via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to the `_beforeSwap` extension hook. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` inside the pool is the **router address**, not the end user. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` against this router address, so any unprivileged user can bypass the per-address swap allowlist simply by routing through the public router.

## Finding Description

**Root cause — `sender` is the direct pool caller, not the end user:**

In `MetricOmmPool.sol` line 231, `msg.sender` (the direct caller of `pool.swap()`) is forwarded as `sender` to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-240
_beforeSwap(
  msg.sender,   // ← always the direct caller of pool.swap()
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` passes this value unchanged to the extension:

```solidity
// ExtensionCalling.sol:160-176
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks this `sender` against the allowlist:

```solidity
// SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

**Exploit path:**

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap(...)` directly (line 72-80 of `MetricOmmSimpleRouter.sol`). Inside the pool, `msg.sender` is the **router**, so `sender` delivered to the extension is the router address. The extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

Two concrete failure modes:
1. **Allowlist bypass**: Pool admin allowlists the router (necessary for any user to swap via router) → every user, including non-allowlisted ones, can swap freely through the router.
2. **Allowlisted users blocked**: Pool admin does NOT allowlist the router → individually allowlisted users cannot use the standard router path at all, breaking the intended UX.

**Existing guards are insufficient:** There is no mechanism in the pool, router, or extension to propagate the original `msg.sender` (the end user) through the call chain. The `_setNextCallbackContext` in the router stores the payer for the callback but does not pass the originating user to the pool's `swap()` call or to the extension.

## Impact Explanation
The `SwapAllowlistExtension` is a core access-control primitive for pools that require KYC/permissioned trading. Its bypass allows any unprivileged, non-allowlisted address to execute swaps on a pool that was explicitly configured to restrict trading to approved addresses. This constitutes broken core pool functionality (allowlist gate) and an admin-boundary break reachable by any public user via the standard router. Severity: **High** — the allowlist is completely ineffective for router-mediated swaps, which is the primary user-facing swap path.

## Likelihood Explanation
Any user can call `MetricOmmSimpleRouter.exactInputSingle()` with no preconditions beyond having tokens. The router is a public, permissionless contract. The bypass is trivially repeatable on every swap on any allowlisted pool. No special privileges, flash loans, or timing are required.

## Recommendation
Pass the originating user identity through the swap path. Options:
1. Add an `originator` parameter to `IMetricOmmPoolActions.swap()` and have the router pass `msg.sender` explicitly; the pool forwards it as `sender` to extensions.
2. Alternatively, have the extension read the originating caller from a trusted context (e.g., transient storage set by the router before calling the pool), but this requires a coordinated protocol-level convention.
The cleanest fix is option 1: the pool's `swap()` accepts an explicit `sender` address that the router populates with its own `msg.sender`, and the pool validates that `msg.sender` is a trusted router before accepting a non-`msg.sender` value.
