### Title
`limbs_for_int` Undercounts Actual CLVM Atom Length in `op_multiply`, Causing Cost Undercharge and Size-Limit Bypass вҖ” (`File: src/more_ops.rs`)

---

### Summary

`op_multiply` tracks the running byte-size of the intermediate product using `limbs_for_int(&total)`, which computes `bits().div_ceil(8)` вҖ” the magnitude byte count. For positive numbers whose high bit is set (e.g., 128, 255, 32768 вҖҰ), the canonical CLVM signed-integer encoding requires one extra leading zero byte, so the actual atom length is `limbs_for_int + 1`. This one-byte undercount is then used both for the per-step cost formula and for the 1 024-byte size-limit guard, causing systematic cost undercharging and a one-byte size-limit bypass. A second, independent divergence exists between `op_multiply` and the `op_unknown` cost-function-2 path, which models the same multiply cost but updates `l0` by `l0 += l1` instead of `limbs_for_int`, producing different totals for the same input sequence.

---

### Finding Description

**`limbs_for_int` vs. actual CLVM atom length**

`limbs_for_int` is defined as:

```rust
fn limbs_for_int(v: &Number) -> usize {
    v.bits().div_ceil(8) as usize
}
```

`BigInt::bits()` returns the number of bits in the **absolute value**, with no sign bit. For a positive integer whose magnitude fills exactly N bytes (i.e., the most-significant bit of the N-byte representation is 1), `bits()` = 8В·N and `div_ceil(8)` = N. But CLVM's signed-integer encoding (`to_signed_bytes_be`) prepends a `0x00` byte to keep the value positive, so the actual atom is N+1 bytes.

Concrete examples:

| Value | `limbs_for_int` | Actual CLVM atom | О” |
|-------|----------------|-----------------|---|
| 128 (`0x0080`) | 1 | 2 | **вҲ’1** |
| 255 (`0x00ff`) | 1 | 2 | **вҲ’1** |
| 32 768 (`0x008000`) | 2 | 3 | **вҲ’1** |
| 127 (`0x7f`) | 1 | 1 | 0 |
| 256 (`0x0100`) | 2 | 2 | 0 |

**Where the undercount is used in `op_multiply`**

After each multiplication step, `l0` is set to the undercounted value:

```rust
l0 = limbs_for_int(&total);   // line 649 вҖ” can be 1 less than actual atom length
if l0 > 1024 {                 // line 650 вҖ” size-limit check uses wrong value
    return Err(...);
}
```

The next iteration then computes cost with the wrong `l0`:

```rust
cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;          // line 615/623/643
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;  // line 616/624/644
```

**Divergence with `op_unknown` cost-function 2**

`op_unknown` (cost-function 2) is documented as computing cost "like operator mul". It updates `l0` as:

```rust
l0 += l1;   // line 239 вҖ” sum of input byte lengths
```

`op_multiply` updates `l0` as `limbs_for_int(&total)`, which is the magnitude byte count of the actual product. These two formulas diverge whenever the product has fewer significant bytes than the sum of the operand lengths (which is the common case for large multiplications), or whenever the product is a positive number with its high bit set.

---

### Impact Explanation

1. **Cost undercharge in `op_multiply`**: Every multiplication step where the intermediate product is a positive number with its high bit set charges `(l0 + l1) * 6 + (l0 * l1) / 128` with `l0` one less than the actual atom length. The undercharge per step is up to `(1 + l1) * 6 + l1 / 128` cost units. An attacker can chain many such steps (up to the 1 024-byte limit) to accumulate a meaningful undercharge, allowing programs that should exceed the cost limit to pass.

2. **Size-limit bypass**: The guard `l0 > 1024` uses the undercounted `l0`. A product whose actual CLVM atom is 1 025 bytes (e.g., a positive number with exactly 1 024 bytes of magnitude) has `limbs_for_int` = 1 024, passes the check, and is stored in the allocator. The intended limit is 1 024 bytes; the effective limit is 1 025 bytes.

3. **Consensus divergence between `op_multiply` and `op_unknown` cost-function 2**: The two paths that are supposed to model the same multiply cost produce different totals for the same input sequence. A validator that uses `op_unknown` cost-function 2 to pre-screen or estimate cost will disagree with the actual `op_multiply` execution cost, potentially accepting programs that `op_multiply` would reject (or vice versa).

---

### Likelihood Explanation

The trigger condition вҖ” a positive intermediate product with its high bit set вҖ” is common and requires no special attacker knowledge. Any sequence of multiplications that produces an intermediate result in the range [2^(8kвҲ’1), 2^(8k)) for any integer k вүҘ 1 (e.g., 128, 255, 32 768, 65 535, вҖҰ) hits the undercount. An attacker can craft CLVM bytes with a sequence of small multiplications that repeatedly land on such values, accumulating cost undercharge across many steps. The entry path is fully attacker-controlled CLVM bytes passed to `run_program`.

---

### Recommendation

Replace `limbs_for_int` with a function that returns the actual CLVM atom length of the product, consistent with how `new_number` encodes it:

```rust
// Instead of:
l0 = limbs_for_int(&total);

// Use the actual encoded length:
let encoded = total.to_signed_bytes_be();
l0 = if encoded == [0] { 0 } else { encoded.len() };
```

Alternatively, align `op_unknown` cost-function 2 to use the same formula as `op_multiply`, or vice versa, so both paths compute identical costs for the same logical operation.

### Proof of Concept

**Undercount demonstration:**

- Input: `(* 128 1)` вҖ” multiply 128 (atom `[0x00, 0x80]`, 2 bytes) by 1 (atom `[0x01]`, 1 byte).
- After the first argument: `l0 = 2` (from `int_atom`, correct atom length).
- After the multiplication: `total = 128`, `limbs_for_int(128) = 1` (bits = 8, div_ceil(8) = 1).
- `l0` is now **1**, but the actual CLVM atom for 128 is `[0x00, 0x80]` = **2 bytes**.
- If a third argument follows, the cost is computed with `l0 = 1` instead of `l0 = 2`, undercharging by `(1 + l1) * 6 + l1 / 128`.

**Size-limit bypass demonstration:**

- Construct a sequence of multiplications whose product is a positive integer with exactly 1 024 bytes of magnitude (high bit set), so the actual CLVM atom is 1 025 bytes.
- `limbs_for_int` returns 1 024; the check `l0 > 1024` is false; the oversized atom is stored.

**Cost divergence between `op_multiply` and `op_unknown` cost-function 2:**

- Input sequence: A (128 bytes), B (128 bytes), C (1 byte).
- `op_multiply` after AГ—B: `l0 = limbs_for_int(A*B)` вүӨ 256 (could be 255 if high bit set вҶ’ `l0 = 255`). Cost for BГ—C step uses `l0 = 255`.
