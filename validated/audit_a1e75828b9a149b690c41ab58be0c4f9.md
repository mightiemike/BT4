### Title
`block_declare` Gateway Admission Flag Not Re-Enforced at Blockifier Execution — (`crates/apollo_gateway/src/gateway.rs`)

### Summary
The `block_declare` flag in `GatewayConfig.static_config` is checked only at gateway admission time inside `check_declare_permissions`. The blockifier has no knowledge of this flag and never re-checks it at execution time. A declare transaction admitted to the mempool while `block_declare=false` will be executed by the blockifier even after the operator subsequently sets `block_declare=true`, directly mirroring the external bug's "blacklist checked at creation but not at execution" pattern.

### Finding Description
In `check_declare_permissions`, the gateway enforces the `block_declare` flag: [1](#0-0) 

When `block_declare=true`, new declare transactions are rejected with `StarknetErrorCode.BLOCKED_TRANSACTION_TYPE`. However, this flag lives exclusively in `GatewayConfig.static_config` and is never passed to, nor consulted by, the blockifier's `TransactionExecutor`. The blockifier receives a `BlockContext` containing `VersionedConstants`: [2](#0-1) 

`VersionedConstants` contains `block_casm_hash_v1_declares`, `block_direct_execute_call`, and other execution-time flags, but has **no `block_declare` field**. The batcher pulls transactions from the mempool and hands them directly to the blockifier without re-running gateway-level admission checks: [3](#0-2) 

The gateway validator itself only runs at admission time. Once a declare transaction clears the gateway and enters the mempool, no subsequent execution-time gate re-checks `block_declare`.

The `is_authorized_declarer` check in the same function exhibits the identical pattern — an address removed from the authorized-declarers list after its declare transaction enters the mempool will still have that transaction executed: [4](#0-3) 

### Impact Explanation
An operator who sets `block_declare=true` to halt class declarations (e.g., in response to a security incident or a protocol upgrade freeze) cannot prevent already-queued declare transactions from executing. A malicious or unintended contract class can be declared on-chain and become permanently available for deployment, despite the operator's explicit intent to block it. This produces wrong on-chain state (a declared class hash and its compiled class hash in storage) that the operator did not authorize.

Matching impact: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing** (the admission control is rendered ineffective for in-flight transactions), and **Critical — Wrong state … class hash, storage value … from blockifier/syscall/execution logic for accepted input** (the class hash and compiled class hash are written to state against operator policy).

### Likelihood Explanation
The window between transaction admission and blockifier execution spans at least one full block-building cycle and can span many blocks under mempool congestion. An attacker who anticipates an imminent `block_declare=true` config push (e.g., by monitoring governance channels or operator announcements) can front-run the change by submitting a declare transaction immediately before it takes effect. The attacker's action (submitting a declare transaction) is entirely unprivileged.

### Recommendation
Re-check the `block_declare` (and `is_authorized_declarer`) invariant at blockifier execution time, either by:
1. Propagating the flag into `VersionedConstants` or `BlockContext` so the blockifier can enforce it, or
2. Having the batcher/mempool drain or re-validate pending declare transactions whenever `block_declare` transitions to `true`.

This mirrors the external report's recommendation: check the blacklist at the execution stage, not only at the creation/admission stage.

### Proof of Concept
1. Gateway config: `block_declare = false`, `authorized_declarers` contains attacker address `A`.
2. Attacker submits `DeclareV3` transaction for a malicious Sierra class → `check_declare_permissions` passes → transaction enters mempool.
3. Operator sets `block_declare = true` (or removes `A` from `authorized_declarers`).
4. Batcher pulls the declare transaction from the mempool; blockifier executes it via `TransactionExecutor::execute_txs` with a `BlockContext` that has no `block_declare` field.
5. Blockifier writes the class hash and compiled class hash to state; the malicious class is now permanently declared on-chain.
6. Operator's intent to block the declaration is violated; the class is available for deployment by any `deploy_account` or `deploy` syscall.

    fn check_declare_permissions(
        &self,
        declare_tx: &RpcDeclareTransaction,
    ) -> Result<(), StarknetError> {
        // TODO(noamsp): Return same error as in Python gateway.
        if self.config.static_config.block_declare {
            return Err(StarknetError {
                code: StarknetErrorCode::UnknownErrorCode(
                    "StarknetErrorCode.BLOCKED_TRANSACTION_TYPE".to_string(),
                ),
                message: "Transaction type is temporarily blocked.".to_string(),
            });
        }
        let RpcDeclareTransaction::V3(declare_v3_tx) = declare_tx;
        if !self.config.is_authorized_declarer(&declare_v3_tx.sender_address) {
            return Err(StarknetError {
                code: StarknetErrorCode::KnownErrorCode(
                    KnownStarknetErrorCode::UnauthorizedDeclare,
                ),
                message: format!(
                    "Account address {} is not allowed to declare contracts.",
                    &declare_v3_tx.sender_address
                ),
            });
        }
        Ok(())
    }

    #[sequencer_latency_histogram(GATEWAY_VALIDATE_TX_LATENCY, true)]
    async fn run_validate_entry_point(
        &mut self,
        executable_tx: &ExecutableTransaction,
        skip_validate: bool,
    ) -> StatefulTransactionValidatorResult<()> {
        let only_query = false;
        let charge_fee = enforce_fee(executable_tx, only_query);
        let strict_nonce_check = false;
        let execution_flags =
            ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };

        let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };

        // Build block context.
        let mut versioned_constants = VersionedConstants::get_versioned_constants(
            self.config.versioned_constants_overrides.clone(),
        );
        // The validation of a transaction is not affected by the casm hash migration.
        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();
        let block_context = BlockContext::new(
            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),
        );

        // Move state into the blocking task and run CPU-heavy validation.
        let state_reader_and_contract_manager = self.take_state_reader_and_contract_manager();

        let cur_span = Span::current();
        #[allow(clippy::result_large_err)]
        tokio::task::spawn_blocking(move || {
            cur_span.in_scope(|| {
                let state = CachedState::new(state_reader_and_contract_manager);
                let mut blockifier_validator = StatefulValidator::create(state, block_context);
                blockifier_validator.validate(account_tx)
            })
        })
        .await
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::UnknownErrorCode(
                "StarknetErrorCode.InternalError".to_string(),
            ),
            message: format!("Blocking task join error when running the validate entry point: {e}"),
        })?
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::KnownErrorCode(KnownStarknetErrorCode::ValidateFailure),
            message: e.to_string(),
        })?;
        Ok(())
    }
