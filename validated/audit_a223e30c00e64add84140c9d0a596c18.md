### Title
Paused `submit_blocks` Freezes Chain State, Enabling Stale-State SPV Proof Acceptance Immediately After Unpause — (File: `contract/src/lib.rs`)

---

### Summary

Both `submit_blocks` and `verify_transaction_inclusion`/`verify_transaction_inclusion_v2` carry the `#[pause]` decorator. During a pause, no new Bitcoin headers can be submitted, so the contract's canonical chain state freezes. If a Bitcoin reorg occurs during the pause, the contract has no way to learn about it. The moment the contract is unpaused, `verify_transaction_inclusion` becomes callable again — but against a stale, pre-pause chain view that still treats reorganized-away blocks as mainchain blocks. An unprivileged NEAR caller can immediately exploit this window to obtain a `true` SPV proof for a transaction whose containing block no longer exists on the real Bitcoin chain.

---

### Finding Description

`submit_blocks` is gated by `#[pause]`: [1](#0-0) 

This means that while the contract is paused, the three canonical-chain mappings — `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height` — are completely frozen. The real Bitcoin network continues to produce blocks and can undergo reorgs of any depth.

Both verification entry points are also gated by `#[pause]`: [2](#0-1) [3](#0-2) 

The moment the contract is unpaused, these functions become callable with zero delay. The confirmation check inside `verify_transaction_inclusion` computes depth against the stale frozen tip: [4](#0-3) 

And the mainchain membership check consults `mainchain_header_to_height`, which was never updated during the pause: [5](#0-4) 

If block B was in the contract's mainchain before the pause but was reorganized away on the real Bitcoin chain during the pause, the contract still maps B's hash to a height in `mainchain_header_to_height`. A caller supplying a valid Merkle proof for a transaction in B will receive `true` from `verify_transaction_inclusion`, even though B is no longer canonical on Bitcoin.

---

### Impact Explanation

Any downstream NEAR contract (e.g., a BTC bridge, a cross-chain DEX, a custody protocol) that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` and acts on a `true` result will accept a Bitcoin transaction that has been reorganized away. This enables a double-spend: the attacker spends BTC in block B, the block is reorged away (the BTC is returned to the attacker), and the NEAR-side contract still releases the bridged asset because the light client certified the now-invalid inclusion proof. The corrupted state is the `mainchain_header_to_height` mapping, which permanently retains a reorganized block as a mainchain member until a relayer submits the competing chain after unpause.

---

### Likelihood Explanation

Bitcoin reorgs of 1–2 blocks occur several times per year on mainnet. A security-triggered pause lasting hours is a realistic operational scenario. The attacker does not need to cause the pause or the reorg — they only need to observe both events and call `verify_transaction_inclusion` in the first transaction after unpause, before the relayer has submitted enough headers to overwrite the stale state. The relayer must submit the entire competing chain tip before the contract's `reorg_chain` logic can correct `mainchain_header_to_height`: [6](#0-5) 

There is no atomic "catch-up before verification" guarantee, so the window between unpause and full relayer catch-up is exploitable.

---

### Recommendation

**Option 1 (preferred — mirrors the report's first recommendation):** Remove `#[pause]` from `submit_blocks` for accounts holding `Role::UnrestrictedSubmitBlocks`, or add an explicit `except(roles(Role::UnrestrictedSubmitBlocks))` clause so that trusted relayers can continue submitting headers during a pause. This keeps the canonical chain state current and eliminates the stale-state window entirely.

**Option 2 (mirrors the report's second recommendation):** Implement a post-unpause grace period. After the contract is unpaused, block calls to `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` for a configurable number of blocks or wall-clock seconds, giving the relayer time to catch up before proofs are accepted.

**Option 3:** Add a staleness guard inside `verify_transaction_inclusion`: compare `env::block_timestamp()` or the tip header's timestamp against an expected maximum lag, and reject proofs if the contract's chain state appears stale.

---

### Proof of Concept

1. Contract is live; relayer has submitted headers up to Bitcoin height H. Block B at height H−5 is in `mainchain_height_to_header` and `mainchain_header_to_height`. Attacker sends BTC in transaction T included in block B.

2. `PauseManager` pauses the contract (maintenance or security event). `submit_blocks` is now blocked.

3. Bitcoin network produces a reorg: a competing chain of length > 5 replaces B. Block B is no longer canonical. The attacker's BTC is returned to them on the real Bitcoin chain.

4. Contract is unpaused. `mainchain_header_to_height` still maps B's hash to height H−5. `mainchain_tip_blockhash` still points to height H.

5. Attacker immediately calls `verify_transaction_inclusion_v2` with:
   - `tx_block_blockhash` = hash of B
   - `tx_id` = T
   - `confirmations` = 6
   - valid Merkle proof for T in B (attacker has this from step 1)
   - valid coinbase Merkle proof for B

6. `mainchain_header_to_height.get(&B)` returns `H−5` (stale). Confirmation check: `H − (H−5) + 1 = 6 >= 6` passes. Merkle proof verifies against B's stored `merkle_root`. Function returns `true`.

7. Downstream bridge contract releases bridged asset. Attacker has both the BTC (returned by reorg) and the bridged asset. [7](#0-6) [2](#0-1) [3](#0-2) [8](#0-7)
