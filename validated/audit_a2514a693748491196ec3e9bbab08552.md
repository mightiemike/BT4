### Title
`GlobalContractDistribution` Receipt Silently Routed to Only One Child Shard After Resharding — (`File: core/primitives/src/shard_layout/v3.rs`)

### Summary

`ShardLayoutV3::resolve_to_current_shard` always follows `children[0]` when mapping a retired parent shard ID to a live descendant. When a `GlobalContractDistribution` receipt whose `target_shard` is a pre-split parent shard is processed after resharding, `Receipt::receiver_shard_id` calls `resolve_to_current_shard` and routes the receipt exclusively to the left child shard. The right child shard never receives the distribution, so every account on that shard is silently denied access to the global contract.

### Finding Description

`ShardLayoutV3::resolve_to_current_shard` is defined as:

```rust
pub fn resolve_to_current_shard(&self, shard_id: ShardId) -> Option<ShardId> {
    if self.shard_ids.contains(&shard_id) {
        return Some(shard_id);
    }
    let children = self.shards_split_map.get(&shard_id)?;
    self.resolve_to_current_shard(children[0])   // ← always first child only
}
``` [1](#0-0) 

The `ShardLayout::resolve_to_current_shard` wrapper for V0/V1/V2 has the identical defect:

```rust
Self::V0(_) | Self::V1(_) | Self::V2(_) => {

``` [2](#0-1) 

This function is the sole resolution path used by `Receipt::receiver_shard_id` for `GlobalContractDistribution` receipts whose `target_shard` is no longer present in the current layout:

```rust
ReceiptEnum::GlobalContractDistribution(receipt) => {
    let target_shard = receipt.target_shard();
    if shard_layout.shard_ids().contains(&target_shard) {
        target_shard
    } else {
        let Some(current_shard) = shard_layout.resolve_to_current_shard(target_shard)
        ...
        current_shard   // ← always children[0], never children[1]
    }
}
``` [3](#0-2) 

A `GlobalContractDistribution` receipt is created with a specific `target_shard`. When that shard is later split into two children (e.g., `children = [left_child, right_child]`), the receipt must logically reach **both** children so that every account in the post-split layout can use the global contract. Instead, `resolve_to_current_shard` returns only `children[0]` (the left child). The right child shard (`children[1]`) never receives the distribution receipt. Every account whose account ID maps to the right child shard will find the global contract absent from that shard's state.

The `GlobalContracts` protocol feature (version 77) and `DynamicResharding` (version 85) are both active at the current stable protocol version 86, making this scenario reachable on mainnet. [4](#0-3) [5](#0-4) 

### Impact Explanation

Any `GlobalContractDistribution` receipt that was enqueued (delayed) before a resharding event and is processed after the split will be delivered only to the left child shard. The right child shard's state will not contain the global contract. Calls from accounts on the right child shard that depend on the global contract will fail with a missing-contract error. Because the receipt is consumed (not retried), the omission is permanent for that distribution event — the contract can only be re-distributed by a new deployment action.

**Impact: High** — permanent, silent loss of global contract availability for all accounts on the right child shard after any resharding event that coincides with an in-flight `GlobalContractDistribution` receipt.

### Likelihood Explanation

The trigger requires two conditions to coincide: (1) a global contract deployment whose distribution receipts are still in the delayed-receipt queue at the moment a resharding epoch boundary is crossed, and (2) the target shard being split. Both features are live at protocol version 86. Resharding events are operator-scheduled but not rare; delayed receipts can persist across epoch boundaries under congestion. The window is narrow per individual resharding event but grows with network activity and the number of global contracts deployed.

**Likelihood: Medium** — requires a specific timing coincidence, but both preconditions are normal production events.

### Recommendation

`resolve_to_current_shard` must not be used as the routing function for `GlobalContractDistribution` receipts, because a single receipt cannot be delivered to multiple shards. The correct fix is one of:

1. **At distribution time**: When the global contract distribution logic creates receipts, enumerate all current shards and create one receipt per shard, so no single receipt ever needs to be re-routed across a split boundary.
2. **At resharding time**: During the resharding migration step, detect any delayed `GlobalContractDistribution` receipts targeting the parent shard and synthesize a duplicate receipt for each child shard.
3. **In `receiver_shard_id`**: Return an error (rather than silently picking `children[0]`) for `GlobalContractDistribution` receipts targeting a retired shard, so the issue surfaces loudly rather than silently dropping the distribution.

Additionally, `resolve_to_current_shard` should carry a doc comment warning that it is a **lossy** mapping (one-to-one, not one-to-many) and must not be used in contexts where all descendants must be reached.

### Proof of Concept

1. Deploy a global contract at protocol version 86 with dynamic resharding enabled.
2. Arrange for the resulting `GlobalContractDistribution` receipt to enter the delayed-receipt queue (e.g., by saturating the shard's congestion window).
3. Trigger a resharding event that splits the target shard into `left_child` and `right_child`.
4. Allow the delayed receipt to be processed in the post-split epoch.
5. Observe: `resolve_to_current_shard(parent_shard_id)` returns `left_child` only.
6. Query the global contract from an account on `right_child` — the contract is absent; the call fails.

The divergent Borsh/state value is the absence of the global contract entry in `right_child`'s trie state, which should be present but is never written because the distribution receipt was never routed there. [6](#0-5) [7](#0-6)
