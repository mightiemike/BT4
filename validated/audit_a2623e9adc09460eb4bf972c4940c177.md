### Title
Unrestricted Hard-Fork Flag Injection via `run_serialized_chia_program` - (File: `wheel/src/api.rs`)

---

### Summary

The Python-facing `run_serialized_chia_program` function accepts an arbitrary `flags: u32` parameter and converts it with `ClvmFlags::from_bits_truncate(flags)`, which silently accepts every defined `ClvmFlags` bit — including hard-fork flags explicitly annotated as *"Hard-fork; enable only when it activates"* — without any validation or allowlist. Any Python caller can therefore enable `RELAXED_BLS` (0x0008) or `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` (0x0100) before the corresponding network upgrade activates, causing the local node to accept CLVM programs that consensus-correct peers reject.

---

### Finding Description

`run_serialized_chia_program` is the primary Python entry point for CLVM execution:

```python
# Python call
cost, result = run_serialized_chia_program(program_bytes, args_bytes, max_cost, flags)
```

Inside the function the raw integer is converted without any validation:

```rust
let flags = ClvmFlags::from_bits_truncate(flags);   // silently accepts every known bit
``` [1](#0-0) 

`from_bits_truncate` is a `bitflags` method that **drops unknown bits but keeps every known bit**. The `ClvmFlags` type defines two hard-fork bits that carry explicit "enable only when it activates" warnings:

```rust
/// Make bls_g1_negate and bls_g2_negate accept invalid points …
/// Hard-fork; enable only when it activates.
const RELAXED_BLS = 0x0008;

/// Enables the keccak256 op *outside* the softfork guard. Hard-fork;
/// enable only when it activates.
const ENABLE_KECCAK_OPS_OUTSIDE_GUARD = 0x0100;
``` [2](#0-1) 

Neither `RELAXED_BLS` nor `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` is exported as a Python constant in the module initializer:

```rust
m.add("NO_UNKNOWN_OPS", ClvmFlags::NO_UNKNOWN_OPS.bits())?;
m.add("LIMIT_HEAP",     ClvmFlags::LIMIT_HEAP.bits())?;
m.add("MEMPOOL_MODE",   MEMPOOL_MODE.bits())?;
// RELAXED_BLS and ENABLE_KECCAK_OPS_OUTSIDE_GUARD are intentionally absent
``` [3](#0-2) 

Despite not being exported, both bits are accepted without complaint when passed as raw integers, because `from_bits_truncate` has no concept of "approved for external use."

The `ChiaDialect::op` method then propagates the caller-supplied flags directly to every operator, including `op_bls_g1_negate` (opcode 51) and `op_bls_g2_negate` (opcode 55), which branch on `RELAXED_BLS`, and to `op_keccak256` (opcode 62), which is gated on `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`:

```rust
62 if flags.contains(ClvmFlags::ENABLE_KECCAK_OPS_OUTSIDE_GUARD) => op_keccak256,
``` [4](#0-3) 

---

### Impact Explanation

**Consensus divergence.** A node running with `RELAXED_BLS` enabled accepts `bls_g1_negate` / `bls_g2_negate` calls on byte strings that are not valid G1/G2 points. Every other node on the network (without the flag) rejects those same programs. The affected node therefore builds a different view of which coin spends are valid, breaking consensus. The same applies to `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`: programs using opcode 62 outside a softfork guard succeed locally but fail on peers, again diverging the chain view.

The corrupted result is concrete: a `Response` of `Ok(Reduction(cost, result_node))` where every other node returns `Err(EvalErr::Unimplemented)` or a BLS validation error.

---

### Likelihood Explanation

**Low–Medium.** The attacker-controlled entry path is:

1. Any Python process that can `import clvm_rs` and call `run_serialized_chia_program` — this includes third-party tooling, wallet software, or plugins built on top of the library.
2. The caller passes `flags=0x0008` or `flags=0x0100` (raw integer literals; the bit values are visible in the public Rust source).
3. The caller submits a CLVM program that exercises the newly-unlocked opcode path.

No compromise of the core node binary is required; only the ability to invoke the public Python API with an attacker-chosen integer argument. The raw bit values are discoverable from the open-source `chia_dialect.rs`. The risk is highest during the period between a hard-fork being merged into `clvm_rs` and the fork actually activating on-chain, when the bits are live in the library but should not yet be reachable.

---

### Recommendation

Replace `from_bits_truncate` with an explicit allowlist of flags that Python callers are permitted to supply:

```rust
const PYTHON_ALLOWED_FLAGS: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::ENABLE_SHA256_TREE)
    .union(ClvmFlags::ENABLE_SECP_OPS);

pub fn run_serialized_chia_program(..., flags: u32) -> PyResult<...> {
    let raw = ClvmFlags::from_bits_truncate(flags);
    if raw != raw & PYTHON_ALLOWED_FLAGS {
        return Err(PyValueError::new_err("unsupported flags"));
    }
    ...
}
```

Hard-fork flags (`RELAXED_BLS`, `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`, `MALACHITE`, etc.) should only be injectable through a separate, explicitly versioned API that the node software calls after verifying the current block height against the fork activation height.

### Proof of Concept

```python
import clvm_rs

# A CLVM program that calls bls_g1_negate (opcode 51) on 48 bytes of garbage.
# Without RELAXED_BLS this raises an error; with it the call succeeds.
RELAXED_BLS = 0x0008

garbage_g1 = b'\xc0' + b'\x00' * 47          # 48 bytes, not a valid G1 point
# (51 . (garbage_g1))  serialised as CLVM bytes
program = bytes([0xff, 0x33, 0xff]) + bytes([0x30 + 48]) + garbage_g1 + bytes([0x80, 0x80])

cost, result = clvm_rs.run_serialized_chia_program(
    program,
    b'\x80',          # nil environment
    10_000_000,
    RELAXED_BLS,      # raw bit value — not an exported constant
)
# Succeeds locally; every peer without RELAXED_BLS returns an error.
print("accepted with cost", cost)
``` [5](#0-4) [6](#0-5)

pub fn run_serialized_chia_program(
    py: Python,
    program: &[u8],
    args: &[u8],
    max_cost: Cost,
    flags: u32,
) -> PyResult<(u64, LazyNode)> {
    let flags = ClvmFlags::from_bits_truncate(flags);
    let mut allocator = if flags.contains(ClvmFlags::LIMIT_HEAP) {
        Allocator::new_limited(500000000)
    } else {
        Allocator::new()
    };

    let r: Response = (|| -> PyResult<Response> {

        let args = node_from_bytes(&mut allocator, args).map_err(eval_to_py)?;
        let dialect = ChiaDialect::new(flags);

        Ok(py.detach(|| run_program(&mut allocator, &dialect, program, args, max_cost)))
    })()?;
    adapt_response(py, allocator, r)
}
