Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Original User, Enabling Complete Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to its own `msg.sender` — the router, not the originating user. When a pool admin allowlists `MetricOmmSimpleRouter` to enable standard periphery swaps for approved users, every unprivileged caller of the router bypasses the allowlist entirely, because the extension sees only the router's address and that address is allowlisted.

## Finding Description

**Root cause — pool forwards its own `msg.sender` as `sender`:**

In `MetricOmmPool.swap` (lines 230–240), the pool calls:

```solidity
_beforeSwap(
    msg.sender,   // direct caller of pool.swap() — the router, not the original user
    recipient,
    ...
);
```

**Extension checks that forwarded address:**

`SwapAllowlistExtension.beforeSwap` (lines 37–39) evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (correct). `sender` is whatever the pool received as its own `msg.sender` — the router when the user goes through `MetricOmmSimpleRouter`.

**Router never forwards the original user to the pool:**

In `exactInputSingle` (lines 71–80), the original user's address is stored only in transient callback context for payment:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The pool receives `msg.sender = router`. The same pattern holds in `exactInput` (line 104), `exactOutputSingle` (line 136), `exactOutput` (line 165), and the recursive `_exactOutputIterateCallback` path (line 220).

**Allowlist check resolves against the router:**

The extension evaluates `allowedSwapper[pool][router]`. If the pool admin has allowlisted the router — the necessary step to let approved users swap via the standard periphery — the check passes for **every caller of the router**, regardless of whether that caller is individually allowlisted.

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of addresses loses that restriction entirely for any user routing through `MetricOmmSimpleRouter`. The attacker receives real token output from the pool; LP assets are consumed by an actor the pool admin explicitly intended to exclude. This is a direct loss of LP principal and a complete failure of the pool's core access-control invariant. This meets the "Admin-boundary break" criterion: the pool admin's access control is bypassed by an unprivileged path.

## Likelihood Explanation

The bypass requires the pool admin to have allowlisted the router address. This is the expected operational step for any curated pool that also wants to support the standard periphery: without allowlisting the router, even individually allowlisted users cannot use `exactInputSingle` or any multi-hop path. The pool admin faces a forced choice — either allowlist the router (opening the gate to everyone) or block the router (breaking the standard swap UX for legitimate users). There is no configuration that simultaneously allows router-mediated swaps for approved users and blocks them for unapproved users. The condition is therefore met in every realistic deployment of a curated pool with router support.

## Recommendation

The extension must receive and check the **original user's address**, not the direct caller of `pool.swap()`. Two approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated convention between router and extension.
2. **Add a `trustedForwarder` registry to the extension**: The extension checks whether `sender` is a registered trusted forwarder; if so, it decodes the real user from `extensionData` and checks that address instead (ERC-2771 meta-transaction pattern applied to hook guards).

Either way, the extension must never treat the router's address as the identity to gate when the router is a public, permissionless contract.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured in beforeSwap order.
  - Pool admin calls setAllowedToSwap(pool, router, true)   // allowlist the router
  - Pool admin does NOT call setAllowedToSwap(pool, charlie, true)

Attack:
  - Charlie (not allowlisted) calls:
      router.exactInputSingle(ExactInputSingleParams{
          pool: curated_pool,
          tokenIn: token0,
          tokenOut: token1,
          zeroForOne: true,
          amountIn: X,
          ...
      })

Execution trace:
  router.exactInputSingle()
    → pool.swap(recipient, zeroForOne, ...) [msg.sender = router]
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓  (no revert)
      → swap executes, charlie receives token1 output

Result:
  Charlie successfully swaps on a pool that was supposed to block him.
  The allowlist is completely bypassed.
```
