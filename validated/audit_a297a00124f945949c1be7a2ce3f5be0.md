### Title
Empty Bonus Pool Permanently Blocks Delegator Exit from Tier Position — (`x/tieredrewards/keeper/msg_server.go`, `x/tieredrewards/keeper/claim_rewards.go`)

### Summary

`MsgTierUndelegate` and `MsgExitTierWithDelegation` unconditionally call `claimRewards` before executing the exit. `claimRewards` calls `processEventsAndClaimBonus`, which hard-fails with `ErrInsufficientBonusPool` when the rewards pool cannot cover accrued bonus. Because both exit paths share this mandatory settlement step and no bypass exists, a delegator who has accrued any non-zero bonus is completely unable to exit their tier position whenever the pool balance is insufficient — regardless of how long they have waited for the exit commitment to elapse.

### Finding Description

`MsgTierUndelegate` in `x/tieredrewards/keeper/msg_server.go` calls `claimRewards` unconditionally before initiating the staking unbond:

```go
pos, _, _, err = ms.claimRewards(ctx, pos)
if err != nil {
    return nil, err   // exit blocked here
}
```

`claimRewards` delegates to `processEventsAndClaimBonus`, which computes accrued bonus and then checks pool solvency:

```go
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err   // ErrInsufficientBonusPool propagates up
}
```

`sufficientBonusPoolBalance` returns an error whenever `bonus > 0` and the pool balance is below the owed amount:

```go
if !poolBalance.IsAllGTE(bonus) {
    return errorsmod.Wrapf(types.ErrInsufficientBonusPool, ...)
}
```

The same mandatory `claimRewards` call appears in `MsgExitTierWithDelegation` (line 532 of `msg_server.go`), the only other exit path available after the exit commitment elapses. `MsgTierRedelegate` is explicitly blocked once the exit commitment has elapsed (`ErrExitLockDurationElapsed`). `MsgClearPosition` also calls `claimRewards` first and only cancels the exit rather than completing it.

The rewards pool is drained by two independent, unprivileged mechanisms:
1. The `BeginBlocker` automatically transfers pool funds to `x/distribution` every block to cover the `TargetBaseRewardsRate` shortfall.
2. Any other position owner claiming their own bonus rewards reduces the shared pool balance.

Neither mechanism requires any privileged action. A delegator who has waited the full exit commitment duration (e.g., one year) and has accrued any non-zero bonus is completely unable to initiate unbonding if the pool is empty at that moment.

The `BeforeRedelegationSlashed` hook already recognises this problem and explicitly forfeits bonus silently when the pool is insufficient to avoid a chain halt. No equivalent escape hatch exists for the user-driven exit paths.

### Impact Explanation

A delegator who has:
- Locked tokens in a tier position
- Triggered exit and waited the full exit commitment duration
- Accrued any non-zero bonus rewards

…is completely unable to undelegate or exit with delegation when the rewards pool balance is below their accrued bonus. Their principal (staked tokens) remains locked in the staking module with no available exit path. The lock persists for as long as the pool remains underfunded. Because the pool is drained automatically by the `BeginBlocker` and by other users' claims, this condition can arise and persist without any privileged action.

The corrupted invariant is: **a delegator who has satisfied all exit preconditions (exit triggered, exit elapsed, position delegated) must always be able to initiate unbonding of their principal**. The pool-solvency check on bonus settlement breaks this invariant.

### Likelihood Explanation

The pool is a shared resource drained by every block's `BeginBlocker` top-up and by every bonus claim across all positions. In a live deployment with a non-zero `TargetBaseRewardsRate` and many active positions, the pool balance fluctuates continuously. A delegator whose exit commitment elapses during a period of low pool balance — which can occur naturally without any attacker — will find both exit paths blocked. The longer the exit commitment duration (e.g., one year), the more bonus accrues, and the larger the pool balance required to unblock the exit.

### Recommendation

Decouple exit execution from mandatory bonus settlement. Two options:

1. **Skip bonus settlement on exit when pool is insufficient.** Mirror the `BeforeRedelegationSlashed` hook's behaviour: if `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool` inside `TierUndelegate` or `ExitTierWithDelegation`, forfeit the bonus (advance checkpoints without paying) and proceed with the exit. The delegator loses accrued bonus but recovers their principal.

2. **Separate bonus claim from exit.** Allow `MsgTierUndelegate` and `MsgExitTierWithDelegation` to proceed without settling bonus, leaving the bonus claim as a separate step the delegator can retry once the pool is replenished. Advance `LastBonusAccrual` and `LastEventSeq` to the current block to prevent double-counting on a future claim.

### Proof of Concept

1. Alice locks 10,000 CRO in a 1-year tier, triggering exit immediately.
2. One year passes. Alice has accrued, say, 400 CRO in bonus rewards.
3. During this year the `BeginBlocker` has drained the rewards pool to 0 (or another user's claim reduced it below 400 CRO).
4. Alice submits `MsgTierUndelegate`.
5. `TierUndelegate` calls `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` → returns `ErrInsufficientBonusPool`.
6. The transaction reverts. Alice's 10,000 CRO remain locked in the staking module.
7. Alice submits `MsgExitTierWithDelegation`. Same call chain, same failure.
8. `MsgTierRedelegate` is rejected with `ErrExitLockDurationElapsed` (exit already elapsed).
9. Alice has no remaining exit path until the pool is externally replenished.

```go
func (k Keeper) claimRewards(ctx context.Context, pos types.PositionState) (types.PositionState, sdk.Coins, sdk.Coins, error) {
	if !pos.IsDelegated() {
		return pos, sdk.NewCoins(), sdk.NewCoins(), nil
	}

	base, err := k.claimBaseRewards(ctx, pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	bonus, err := k.processEventsAndClaimBonus(ctx, &pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	return pos, base, bonus, nil
}
```

```go
func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
```
