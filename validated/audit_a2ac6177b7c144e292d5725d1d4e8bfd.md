Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of actual swapper, allowing allowlist bypass via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which equals `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, the router becomes `msg.sender` of `pool.swap(...)`, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actualUser]`. Any pool admin who allowlists the router to support router-mediated swaps inadvertently grants every user access to the curated pool, completely defeating the allowlist.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension is called by the pool via `CallExtension.callExtension`) and `sender` is the value forwarded by `ExtensionCalling._beforeSwap`, which receives it as the `sender` parameter — the `msg.sender` of the pool's own `swap` call.

In `MetricOmmSimpleRouter.exactInputSingle`, the router calls:

```solidity
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
);
```

The router is `msg.sender` of this call. The pool therefore passes `router` as `sender` to `_beforeSwap`, which forwards it to the extension. The extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

The same pattern applies to `exactInput` (line 104), `exactOutputSingle` (line 136), `exactOutput` (line 165), and the recursive callback path `_exactOutputIterateCallback` (line 220) — all router entry points call `pool.swap(...)` with the router as `msg.sender`.

**Bypass path:**
1. Pool admin deploys a curated pool with `SwapAllowlistExtension` and allowlists specific user addresses.
2. Pool admin also allowlists `MetricOmmSimpleRouter` so allowlisted users can use the router.
3. Non-allowlisted user `eve` calls `router.exactInputSingle(pool, ...)`.
4. Router calls `pool.swap(...)` — `msg.sender = router`.
5. Pool calls `extension.beforeSwap(sender=router, ...)`.
6. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
7. `eve` trades on the curated pool despite not being allowlisted.

If the admin does not allowlist the router, the inverse problem occurs: even allowlisted users cannot use the router. There is no configuration that simultaneously enforces per-user allowlisting and permits router-mediated swaps.

## Impact Explanation
The swap allowlist — a primary security control for curated pools — silently fails open for all router-mediated swaps. Non-allowlisted users can trade freely on pools intended to be restricted. Curated pools designed to trade only with trusted counterparties (e.g., to avoid toxic flow or MEV) are exposed to unauthorized swaps, causing direct LP fund loss. This constitutes broken core pool functionality causing loss of funds, matching the allowed impact gate.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical periphery entry point for swaps. Any user aware that a pool uses `SwapAllowlistExtension` can trivially route through the router with no privileged access, special tokens, or unusual setup. The pool admin has no way to enforce per-user allowlisting while also permitting router access — the two goals are mutually exclusive under the current design. The attack is repeatable and requires only a public router call.

## Recommendation
The extension must check the economically relevant actor, not the direct caller of `pool.swap`. The cleanest fix is to have `MetricOmmSimpleRouter` pass `msg.sender` (the actual user) as the `sender` argument when calling `pool.swap(...)`. The pool's `swap` interface already accepts a `sender` parameter for exactly this purpose, so the pool can forward the real user identity to the extension regardless of which periphery path is used.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - allowedSwapper[pool][router] = true   (admin allowlists router for convenience)
  - allowedSwapper[pool][alice] = true    (alice is an allowlisted user)
  - allowedSwapper[pool][eve] = false     (eve is NOT allowlisted)

Attack:
  1. eve calls MetricOmmSimpleRouter.exactInputSingle(pool, ...)
  2. Router calls pool.swap(recipient=eve, ...)  ← msg.sender = router
  3. Pool calls extension.beforeSwap(sender=router, ...)
  4. Extension checks allowedSwapper[pool][router] → true
  5. Swap executes — eve trades on the curated pool despite not being allowlisted

Result:
  - Eve bypasses the swap allowlist entirely
  - LPs are exposed to unauthorized flow
  - The allowlist guard silently fails open on the supported periphery path
