### Title
Stale `v.liquidationPayment` Emitted in `Liquidation` Event for Spread Liquidations — (`File: core/contracts/ClearinghouseLiq.sol`)

---

### Summary

In `ClearinghouseLiq._handleLiquidationPayment`, when processing a spread liquidation (`txn.isEncodedSpread == true`), the variable `v.liquidationPayment` is first assigned the spot-leg quote value, used for spot balance updates, then **overwritten** with the perp-leg oracle payment for perp balance updates. The `Liquidation` event is emitted after this overwrite, so `amountQuote` reflects the perp oracle payment rather than the spot liquidation payment that was actually transferred to the liquidatee.

---

### Finding Description

Inside `_handleLiquidationPayment`, the spread branch executes as follows:

1. `v.liquidationPayment` is set to the spot liquidation price times amount (the quote the liquidatee receives for the spot leg): [1](#0-0) 

2. Spot balance updates use this value correctly: [2](#0-1) 

3. `v.liquidationPayment` is then **overwritten** with the perp oracle price times amount: [3](#0-2) 

4. Perp balance updates use this new value: [4](#0-3) 

5. The `Liquidation` event is emitted at the end of the function using the **overwritten** `v.liquidationPayment` (the perp oracle value), not the original spot liquidation payment: [5](#0-4) 

The `Liquidation` event's `amountQuote` parameter is defined as the quote amount of the liquidation: [6](#0-5) 

For spread liquidations, the spot liquidation payment (`liquidationPriceX18 * amount`) and the perp oracle payment (`oraclePriceX18Perp * amount`) are distinct values — the liquidation price is discounted from oracle price by `LIQUIDATION_FEE_FRACTION`, and the perp oracle price is a separate price feed. The emitted `amountQuote` is therefore incorrect for every spread liquidation.

---

### Impact Explanation

Off-chain systems — indexers, liquidation bots, accounting dashboards, and any consumer of the `Liquidation` event — will record an incorrect `amountQuote` for every spread liquidation. The value emitted is `oraclePriceX18Perp * amount` instead of the actual spot quote transferred (`liquidationPriceX18 * amount`). These two values diverge by the liquidation fee fraction applied to the spot leg, plus any difference between the spot and perp oracle prices. Systems relying on this event for accounting, PnL attribution, or liquidation tracking will have systematically incorrect data for all spread liquidation events.

On-chain balances are updated correctly; the corruption is confined to the emitted event data.

---

### Likelihood Explanation

Spread liquidations (`isEncodedSpread == true`) are a supported, documented code path in the protocol. Any liquidator calling `liquidateSubaccountImpl` against a liquidatee holding a spread position triggers this path. No special privileges are required — any unprivileged caller can invoke `liquidateSubaccountImpl` on an undercollateralized subaccount. The bug fires on every spread liquidation without exception. [7](#0-6) 

---

### Recommendation

Preserve the spot liquidation payment in a separate variable before overwriting `v.liquidationPayment` for the perp leg, and emit the spot value in the `Liquidation` event. For example:

```solidity
int128 spotLiquidationPayment = v.liquidationPriceX18.mul(txn.amount);
// ... use spotLiquidationPayment for spot transfers ...
v.liquidationPayment = v.oraclePriceX18Perp.mul(txn.amount);
// ... use v.liquidationPayment for perp transfers ...
emit Liquidation(
    txn.sender,
    txn.liquidatee,
    txn.productId,
    txn.isEncodedSpread,
    txn.amount,
    spotLiquidationPayment  // correct: spot quote transferred to liquidatee
);
```

---

### Proof of Concept

1. Liquidatee holds a spread position (spot product `spotId` + perp product `perpId`), is under maintenance margin.
2. Liquidator calls `liquidateSubaccountImpl` with `txn.isEncodedSpread = true`, `txn.productId = (perpId << 16) | spotId`, and a valid `txn.amount`.
3. `_handleLiquidationPayment` enters the `isEncodedSpread` branch.
4. `v.liquidationPayment` is set to `liquidationPriceX18 * amount` at line 466 — e.g., `950e18` (discounted spot price).
5. Spot transfers execute correctly using `950e18`.
6. `v.liquidationPayment` is overwritten at line 486 to `oraclePriceX18Perp * amount` — e.g., `1000e18`.
7. Perp transfers execute using `1000e18`.
8. `emit Liquidation(..., 1000e18)` fires — but the actual spot quote transferred to the liquidatee was `950e18`.
9. Any off-chain consumer of this event records `amountQuote = 1000e18` instead of the correct `950e18`, a `~5%` error equal to the liquidation fee fraction on the spot leg. [8](#0-7) [5](#0-4)
