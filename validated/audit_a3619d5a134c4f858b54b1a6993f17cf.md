### Title
Residual Allowance in `DirectDepositV1.creditDeposit()` Exposes DDA Token Balance to Endpoint Drain - (`File: core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` approves the `endpoint` contract for the full `uint256` token balance before calling `depositCollateralWithReferral`. When `depositCollateralWithReferral` silently returns early (minimum deposit amount check fails), no tokens are transferred but the full approval remains. Because the `Endpoint` contract is upgradeable via `endpointTx`, this residual allowance exposes the DDA's token balance to theft if the endpoint implementation is ever compromised.

---

### Finding Description

In `DirectDepositV1.creditDeposit()`:

```solidity
function creditDeposit() external {
    uint32[] memory productIds = spotEngine.getProductIds();
    for (uint256 i = 0; i < productIds.length; i++) {
        ...
        uint256 balance = token.balanceOf(address(this));
        if (balance != 0) {
            token.approve(address(endpoint), balance);          // (1) approve full uint256 balance
            endpoint.depositCollateralWithReferral(
                subaccount,
                productId,
                uint128(balance),                               // (2) deposit only uint128(balance)
                "-1"
            );
        }
    }
}
``` [1](#0-0) 

In `Endpoint.depositCollateralWithReferral`, there is an explicit early-return path that skips the token transfer entirely:

```solidity
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked...
    // we can just skip the deposit and continue with the next asset.
    return;
}
handleDepositTransfer(...);
``` [2](#0-1) 

When this early return fires, `handleDepositTransfer` is never called, so no tokens are pulled from the DDA. However, the `token.approve(address(endpoint), balance)` executed just before the call is never revoked. The full `balance` remains as a live allowance on the `endpoint` contract indefinitely.

A second, independent residual allowance path exists due to the `uint256 â†’ uint128` truncation: if `balance > type(uint128).max`, the approval is for `balance` (uint256) but only `uint128(balance)` is deposited, leaving `balance - uint128(balance)` as residual allowance.

The `Endpoint` contract is upgradeable: its `endpointTx` implementation can be replaced via `upgradeEndpointTx`, which is gated only by the `ProxyManagerHelper`. [3](#0-2) 

---

### Impact Explanation

The DDA contract holds real ERC-20 tokens on behalf of a subaccount owner. A residual allowance on the `endpoint` means the endpoint can call `transferFrom` on the DDA at any future time. If the `endpointTx` implementation is upgraded to a malicious version (or the `ProxyManagerHelper` key is compromised), the attacker can drain every DDA that has a non-zero residual allowance â€” stealing tokens that belong to users who deposited below the minimum threshold or whose `uint128` cast truncated their balance.

Because `creditDeposit()` has no access control, any unprivileged caller can trigger the approval-without-transfer sequence on any DDA at any time, continuously refreshing or creating residual allowances.

---

### Likelihood Explanation

The early-return path in `depositCollateralWithReferral` is explicitly designed to be reachable: the comment in the code acknowledges that some assets may be below the minimum deposit amount and must be skipped silently. Any user who sends a small token amount to their DDA (below `MIN_DEPOSIT_AMOUNT` or `MIN_FIRST_DEPOSIT_AMOUNT`) will trigger this path when `creditDeposit()` is called. Since `creditDeposit()` is permissionless, an attacker can call it proactively to ensure the residual allowance is set before exploiting a compromised endpoint.

---

### Recommendation

Reset the token allowance to zero after each `depositCollateralWithReferral` call, regardless of whether the deposit succeeded or was skipped:

```diff
if (balance != 0) {
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(
        subaccount,
        productId,
        uint128(balance),
        "-1"
    );
+   token.approve(address(endpoint), 0);  // reset residual allowance
}
``` [4](#0-3) 

---

### Proof of Concept

1. Alice sends 1 wei of a supported token to her DDA (below `MIN_FIRST_DEPOSIT_AMOUNT`).
2. Attacker calls `DirectDepositV1(aliceDDA).creditDeposit()`.
3. Inside the loop, `token.approve(address(endpoint), 1)` is executed.
4. `endpoint.depositCollateralWithReferral(...)` is called; `isValidDepositAmount` returns `false`; the function returns early without calling `handleDepositTransfer`.
5. The endpoint now holds an allowance of `1` over Alice's DDA token balance.
6. Alice later sends a larger amount to the DDA (e.g., 1000 USDC). The allowance from step 3 is still live (it was never reset).
7. If the `endpointTx` implementation is upgraded to a malicious contract, it calls `token.transferFrom(aliceDDA, attacker, 1000e6)`, draining Alice's DDA using the stale allowance. [1](#0-0) [5](#0-4)

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned

        requireUnsanctioned(sender);

            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
