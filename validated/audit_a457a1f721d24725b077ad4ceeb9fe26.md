Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `swap` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the router contract address, not the end user. As a result, the extension checks whether the router is allowlisted rather than whether the individual user is allowlisted. If the pool admin allowlists the router to enable any router-mediated swap for legitimate users, every unprivileged user can bypass the per-user allowlist by routing through the public router.

## Finding Description

**Root cause — wrong actor binding in `SwapAllowlistExtension.beforeSwap`:**

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (correct for the pool-key lookup). `sender` is the first argument forwarded by the pool.

**Pool passes `msg.sender` as `sender` to the extension:**

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // <-- whoever called pool.swap()
    recipient,
    ...
);
```

**Router calls `pool.swap()` directly, so `msg.sender` to the pool is the router:**

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
);
```

There is no `sender` parameter in `pool.swap`; the pool has no way to receive the original caller's identity from the router. The pool's `swap` signature is `swap(address recipient, bool zeroForOne, int128 amountSpecified, uint128 priceLimitX64, bytes calldata callbackData, bytes calldata extensionData)` — the original user address is never forwarded.

**Exploit flow:**
1. Pool is deployed with `SwapAllowlistExtension` on the `beforeSwap` hook.
2. Pool admin allowlists specific user addresses (e.g., KYC'd traders) and does NOT allowlist arbitrary addresses.
3. To allow those legitimate users to also use the router, the admin must allowlist the router address: `setAllowedToSwap(pool, router, true)`.
4. Any unprivileged user calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting this pool.
5. Router calls `pool.swap(...)` → pool calls `_beforeSwap(msg.sender=router, ...)` → extension checks `allowedSwapper[pool][router]` → **passes**, because the router is allowlisted.
6. The non-allowlisted user's swap executes successfully on the curated pool.

The same bypass applies to `exactInput` (multi-hop), `exactOutputSingle`, and `exactOutput` — all router entry points call `pool.swap` with `msg.sender = router`.

**Existing guards are insufficient:** The `onlyPool` modifier on the extension only verifies the caller is a registered pool, not the identity of the end user. The allowlist lookup `allowedSwapper[msg.sender][sender]` correctly uses the pool as the key but uses the router as the swapper key, which is the wrong actor.

## Impact Explanation
Any unprivileged user can trade on a pool that the admin intended to restrict to a curated set of swappers, by routing through the public `MetricOmmSimpleRouter`. This is a direct policy bypass on curated pools. The pool admin's allowlist is rendered meaningless for router-mediated swaps. Unauthorized users gain full swap access, which can result in direct loss of LP value on pools designed for controlled trading environments (e.g., institutional or KYC-gated pools).

## Likelihood Explanation
The bypass requires only that the pool admin has allowlisted the router address — a necessary step for any legitimate user to use the router on an allowlisted pool. Once the router is allowlisted (which is the only way to make the router usable for legitimate users), the bypass is unconditional and repeatable by any unprivileged caller with no special setup. The attacker needs only to call the public router with the target pool address.

## Recommendation
The `SwapAllowlistExtension` must check the economically relevant actor — the end user — not the intermediary router. Two approaches:

1. **Pass original caller through router:** Add an optional `originator` field to the pool's `swap` interface or `extensionData` convention, and have the router populate it with `msg.sender`. The extension reads the originator from `extensionData` when present.
2. **Check `tx.origin` as a fallback (not recommended for general use):** Only acceptable in tightly controlled environments.
3. **Preferred — router-level allowlist enforcement:** The router should verify the caller is allowlisted before forwarding to the pool, and the extension should trust only direct pool callers (not router-mediated calls) for allowlist enforcement. Alternatively, deploy a dedicated allowlist-aware router that the extension can recognize and unwrap the true caller from.
