### Title
ICS-721 `ValidateBasic` Rejects Packets from Non-Bech32 Counterparty Chains — (`File: x/nft-transfer/types/packet.go`)

### Summary

`NonFungibleTokenPacketData.ValidateBasic()` unconditionally validates both the `Sender` and `Receiver` fields using `sdk.AccAddressFromBech32`, despite the function's own comment explicitly stating that address formats must not be validated because counterparty chains may use different formats. This makes the ICS-721 NFT transfer module permanently unusable with any non-Bech32 counterparty chain (e.g., EVM chains).

### Finding Description

`ValidateBasic()` in `x/nft-transfer/types/packet.go` contains a direct contradiction: the comment at line 40–41 states *"The addresses formats are not validated as the sender and recipient can have different formats defined by their corresponding chains that are not known to IBC"*, yet lines 59–70 call `sdk.AccAddressFromBech32` on both `Sender` and `Receiver` and return an error if either fails. [1](#0-0) 

This `ValidateBasic()` is called unconditionally inside `OnRecvPacket` before any application logic runs: [2](#0-1) 

When a packet arrives from a counterparty chain whose addresses are not Bech32 (e.g., an EVM chain using `0x…` hex addresses), `sdk.AccAddressFromBech32(data.Sender)` returns an error. `OnRecvPacket` propagates this error, causing `ibc_module.go` to write an error acknowledgement: [3](#0-2) 

The result is that every inbound IBC NFT packet from a non-Bech32 chain is permanently rejected with an error acknowledgement. The NFT is refunded on the source chain, but the transfer can never succeed regardless of how many times it is retried.

The analog to the external report is exact: just as `Curve2TokenPoolMixin` hardcodes assumptions about how to retrieve the LP token address (making certain pools unusable), this module hardcodes the assumption that all IBC counterparty addresses are Bech32 (making certain counterparty chains unusable).

### Impact Explanation

Any IBC channel established between this chain and a non-Bech32 counterparty (e.g., an EVM-compatible chain) is rendered non-functional for NFT transfers. All inbound `OnRecvPacket` calls will return an error acknowledgement. The corrupted invariant is the ICS-721 packet lifecycle: packets that should succeed (valid NFT, valid receiver on this chain) are permanently rejected because the sender's address format on the foreign chain is not Bech32.

### Likelihood Explanation

EVM chains are the most common IBC counterparties that use non-Bech32 addresses. Any relayer attempting to relay an IBC NFT packet from such a chain to this chain will trigger the bug deterministically. No special attacker capability is required — a normal user on the counterparty chain initiating a `MsgTransfer` is sufficient.

### Recommendation

Remove the `sdk.AccAddressFromBech32` validation for `Sender` and `Receiver` from `NonFungibleTokenPacketData.ValidateBasic()`, consistent with the comment already present in the function. The receiver address on this chain is validated separately when `processReceivedPacket` calls `sdk.AccAddressFromBech32(data.Receiver)` at the point of actual token delivery, which is the correct place for chain-local address validation. [4](#0-3) 

### Proof of Concept

1. Chain A (this chain, Bech32) establishes an ICS-721 channel with Chain B (EVM chain, addresses like `0xAbCd…`).
2. A user on Chain B calls `MsgTransfer` to send an NFT to a Bech32 address on Chain A. The packet's `Sender` field is `0xAbCd…`.
3. The relayer submits the packet to Chain A via `OnRecvPacket`.
4. `ibc_module.go` calls `im.keeper.OnRecvPacket(...)`, which calls `data.ValidateBasic()`.
5. `sdk.AccAddressFromBech32("0xAbCd…")` fails → `ValidateBasic` returns `ErrInvalidAddress`.
6. `OnRecvPacket` returns the error → `ibc_module.go` sets `ack = types.NewErrorAcknowledgement(err)`.
7. The packet is acknowledged with an error. The NFT is refunded on Chain B. The transfer is permanently blocked. [5](#0-4)

```go
// NOTE: The addresses formats are not validated as the sender and recipient can have different
// formats defined by their corresponding chains that are not known to IBC.
func (nftpd NonFungibleTokenPacketData) ValidateBasic() error {
	if strings.TrimSpace(nftpd.ClassId) == "" {
		return newsdkerrors.Wrap(ErrInvalidClassID, "classId cannot be blank")
	}

	if len(nftpd.TokenIds) == 0 {
		return newsdkerrors.Wrap(ErrInvalidTokenID, "tokenId cannot be blank")
	}

	if len(nftpd.TokenIds) != len(nftpd.TokenUris) {
		return newsdkerrors.Wrap(ErrInvalidPacket, "tokenIds and tokenUris lengths do not match")
	}

	if strings.TrimSpace(nftpd.Sender) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "sender address cannot be blank")
	}

	// decode the sender address
	if _, err := sdk.AccAddressFromBech32(nftpd.Sender); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid sender address")
	}

	if strings.TrimSpace(nftpd.Receiver) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "receiver address cannot be blank")
	}

	// decode the receiver address
	if _, err := sdk.AccAddressFromBech32(nftpd.Receiver); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid receiver address")
	}
```
