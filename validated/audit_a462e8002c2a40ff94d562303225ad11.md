### Title
Unpermissioned `wrapVaultAsset` Allows Any Caller to Force Underlying-to-Vault-Share Conversion on Any Subaccount's DDA, Altering Credited Product and Collateral Weight — (`core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.wrapVaultAsset` carries no access-control modifier. Any EOA can invoke it for any subaccount. When the vault's underlying asset is also a registered spot product, the call silently converts the DDA's underlying balance into vault shares before `creditDepositV1` runs, causing the subaccount to receive vault-share credit (at the vault's collateral weight) instead of underlying credit (at the underlying's collateral weight). Because the two products carry independent `RiskStore` parameters, the forced substitution can reduce the subaccount's health factor and, in the worst case, push it into liquidation.

---

### Finding Description

`wrapVaultAsset` is declared `external` with no `onlyOwner`, `onlyDeployer`, or any other guard: [1](#0-0) 

The function:
1. Resolves the DDA for `subaccount` (creating one if absent).
2. Reads `assetTokenAddr = IERC4626Base(tokenAddr).asset()`.
3. Calls `DirectDepositV1.withdraw(assetTokenAddr)` — which is `onlyOwner` on the DDA, but `ContractOwner` *is* the owner, so this succeeds — pulling the entire underlying balance from the DDA to `ContractOwner`. [2](#0-1) 

4. Approves the vault and calls `vault.deposit(assetBalance, directDepositV1)`, minting vault shares back to the DDA.

`creditDepositV1` / `creditDeposit` are also unrestricted: [3](#0-2) [4](#0-3) 

`creditDeposit` iterates every registered spot product and deposits whatever balance the DDA holds. After `wrapVaultAsset` runs, the DDA holds **zero** underlying and **non-zero** vault shares, so the subaccount is credited under `vaultProductId` only.

---

### Impact Explanation

Both the underlying token (e.g., USDC, `productId=1`) and the vault token (e.g., `vaultUSDC`, `productId=2`) are registered spot products with independent `RiskStore` entries (`longWeightInitial`, `longWeightMaintenance`). Vault tokens typically carry lower collateral weights than their underlying. A forced conversion therefore:

- Reduces the subaccount's initial and maintenance health.
- Can push a subaccount that was healthy under the underlying's weight below the liquidation threshold under the vault's weight.
- Requires no funds from the attacker and no privileged access. [5](#0-4) 

---

### Likelihood Explanation

- `wrapVaultAsset` is `external` with zero guards — reachable by any EOA in a single transaction.
- The precondition (underlying also registered as a spot product) is the standard deployment pattern for vault products on Nado.
- The attacker needs only to know the victim's `subaccount` bytes32 (publicly derivable from their address) and the vault's `productId`.

---

### Recommendation

Add an access-control modifier to `wrapVaultAsset` so only the subaccount owner (or an authorized operator) can trigger the conversion for their own DDA:

```solidity
// Option A: restrict to the address encoded in the subaccount bytes32
function wrapVaultAsset(bytes32 subaccount, uint32 productId) external {
    require(address(bytes20(subaccount)) == msg.sender, "unauthorized");
    ...
}

// Option B: onlyOwner (protocol-level gate, consistent with other admin ops)
function wrapVaultAsset(bytes32 subaccount, uint32 productId)
    external onlyOwner { ... }
```

`creditDepositV1` should similarly be restricted to the subaccount owner or made callable only after an explicit opt-in, to prevent a separate griefing vector where an attacker forces a deposit at an unfavorable time.

---

### Proof of Concept

```solidity
// Setup (owner/deployer actions, not part of the attack):
// 1. Register USDC as productId=1 in SpotEngine.
// 2. Register vaultUSDC (ERC-4626, asset()=USDC) as productId=2 with lower longWeightInitial.

// Attack (unprivileged EOA):
// 3. Victim sends 1000 USDC to their DDA (intending USDC credit at productId=1).
bytes32 victimSubaccount = bytes32(abi.encodePacked(victim, bytes12("default")));

// 4. Attacker forces conversion — no special role required.
contractOwner.wrapVaultAsset(victimSubaccount, 2 /*vaultProductId*/);
// DDA now holds 0 USDC, ~1000 vaultUSDC shares.

// 5. Attacker (or anyone) triggers credit.
contractOwner.creditDepositV1(victimSubaccount);
// Subaccount receives vaultUSDC credit at vault's (lower) collateral weight.

// 6. Assert: victim's health is lower than if USDC had been credited directly.
//    If victim had open positions near the maintenance margin, they are now liquidatable.
```

        address payable directDepositV1 = directDepositV1Address[subaccount];

            assetToken.approve(tokenAddr, 0);
            assetToken.approve(tokenAddr, assetBalance);
