Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the real end-user, allowing any caller to bypass per-user swap restrictions via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on `allowedSwapper[pool][sender]` where `sender` is the pool's `msg.sender`. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual end-user. If the router is allowlisted (a natural admin action to enable curated users to access the router), every unprivileged user can bypass the per-user restriction and trade on the curated pool. The real user address is stored in the router's transient storage but is never forwarded to the extension.

## Finding Description

**Root cause:** `MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When the call originates from `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap(...)` directly: [4](#0-3) 

So the pool's `msg.sender` is the **router address**, and the extension evaluates `allowedSwapper[pool][router]`. The actual end-user's address (`msg.sender` of the router call) is stored in transient storage via `_setNextCallbackContext` at line 71 but is never forwarded to the extension.

**Two concrete failure modes:**

| Scenario | Outcome |
|---|---|
| Admin allowlists the router (to let curated users use it) | Every unprivileged user can swap through the router — per-user allowlist fully bypassed |
| Admin allowlists specific user addresses only | Those users cannot use the router at all; they must call the pool directly |

**Existing guards are insufficient:** The only check in `beforeSwap` is `allowedSwapper[msg.sender][sender]`. There is no mechanism to unwrap the router intermediary and inspect the originating caller. The `extensionData` field passed through the call chain is caller-controlled and unauthenticated, so it cannot serve as a trusted originator signal without additional protocol-level enforcement.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` for KYC or curation purposes is fully open to any caller who routes through `MetricOmmSimpleRouter` once the router is allowlisted. Non-allowlisted users can execute swaps against LP positions at oracle-anchored prices. Because the pool's spread and notional fees are calibrated for a known counterparty set, unrestricted access directly exposes LP principal to adverse selection and value leakage that the allowlist was designed to prevent. This constitutes a direct loss of LP principal — a High-severity allowed impact under the contest rules.

## Likelihood Explanation
The bypass requires the router to be allowlisted. A pool admin who wants curated users to access the router will naturally call `setAllowedToSwap(pool, router, true)`, not realising this grants universal access. The admin-facing API gives no indication of this behavior. The router is a supported, documented periphery contract, making this a realistic production configuration. Likelihood is Medium.

## Recommendation
Pass the real end-user through the swap path so the extension can gate on the correct actor. Two options:

1. **Preferred — forward the originating user as an extra field.** Add an `originator` field to the `beforeSwap` hook signature (or encode it in `extensionData` with router-level authentication). The router sets `originator = msg.sender` before calling the pool; the pool forwards it to the extension. The extension checks `allowedSwapper[pool][originator]`.

2. **Simpler — maintain a registry of trusted routers.** The extension falls back to checking an originator encoded in `extensionData` when `sender` is a registered trusted router. This requires the factory or extension to maintain a router registry and the router to encode `msg.sender` into `extensionData`.

Either way, the extension must never treat a public intermediary contract as the authoritative identity for access control.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension as before-swap hook
  admin calls: setAllowedToSwap(pool, router, true)   // intends to let curated users use the router
  admin calls: setAllowedToSwap(pool, alice, true)    // alice is the only intended swapper

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: bob, ...})

Trace:
  router → pool.swap(recipient=bob, ...)          // pool.msg.sender = router
  pool   → _beforeSwap(sender=router, ...)        // MetricOmmPool.sol L230-240
  pool   → extension.beforeSwap(sender=router, ...)  // ExtensionCalling.sol L160-176
  extension checks: allowedSwapper[pool][router]  // = true ← router is allowlisted
  → no revert; bob's swap executes on the curated pool

Result:
  bob, who is not on the allowlist, successfully swaps against LP positions.
  The per-user allowlist is completely bypassed via the supported router path.
```

A Foundry integration test can confirm this by deploying a pool with `SwapAllowlistExtension`, allowlisting the router, and asserting that an address not in `allowedSwapper` can successfully call `exactInputSingle` without revert.
