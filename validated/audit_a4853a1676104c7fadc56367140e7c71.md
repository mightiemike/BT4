### Title
Negative-PnL Settlement Inflates `availableSettle` Pool and Drains Spot Collateral — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.settlePnl` does not guard against a negative `canSettle` value. When a subaccount holds a perp position with negative unrealized PnL and the sequencer submits a `SettlePnl` transaction for it, `getSettlementState` returns a negative `canSettle`. The subsequent arithmetic in `settlePnl` then **increases** both `state.availableSettle` and `balance.vQuoteBalance` (subtracting a negative), while `Clearinghouse._settlePnl` applies the negative `amountSettled` to the spot engine, **decreasing** the subaccount's spot quote balance. The net result is an inflated `availableSettle` pool with no real backing, artificial improvement of the perp vQuoteBalance, and a corresponding drain of spot collateral.

---

### Finding Description

**Step 1 — `getSettlementState` returns a negative value** [1](#0-0) 

```
positionPnl = price * amount + vQuoteBalance   // < 0 (negative PnL position)
state.availableSettle                           // > 0

canSettle = MathHelper.min(positionPnl, state.availableSettle)
          = MathHelper.min(negative, positive)
          = negative
```

`MathHelper.min` is a plain signed comparison with no clamping to zero: [2](#0-1) 

**Step 2 — `settlePnl` applies the negative `canSettle` without a guard** [3](#0-2) 

```
state.availableSettle -= canSettle   // -= negative  →  INCREASES (pool inflated)
balance.vQuoteBalance -= canSettle   // -= negative  →  INCREASES (perp balance improved)
totalSettled          += canSettle   // += negative  →  NEGATIVE return value
```

There is no `if (canSettle <= 0) continue;` guard anywhere in this loop.

**Step 3 — `Clearinghouse._settlePnl` applies the negative `amountSettled` to spot** [4](#0-3) 

```
amountSettled < 0
→ spotEngine.updateBalance(QUOTE_PRODUCT_ID, subaccount, negative)
→ subaccount's spot quote balance DECREASES
```

**Net accounting delta per settlement of a negative-PnL position:**

| Variable | Direction | Magnitude |
|---|---|---|
| `state.availableSettle` | +\|canSettle\| | inflated, unbacked |
| `balance.vQuoteBalance` | +\|canSettle\| | artificially improved |
| spot quote balance | −\|canSettle\| | drained |

The `availableSettle` pool now contains phantom value: it grew without any real asset being contributed. Future positive-PnL settlements will draw from this inflated pool, creating protocol bad debt when the pool is exhausted.

---

### Impact Explanation

- **`availableSettle` pool inflation**: The pool increases instead of decreasing, meaning it can be over-drawn by subsequent legitimate settlements, leaving the protocol insolvent.
- **Spot collateral drain**: The subaccount's spot quote balance is reduced by `|canSettle|` with no corresponding benefit to the protocol.
- **Perp vQuoteBalance improvement**: The subaccount's perp health is artificially improved, potentially allowing it to open larger positions or avoid liquidation.
- **Protocol bad debt**: The inflated `availableSettle` has no real backing; when legitimate positive-PnL subaccounts settle against it, the protocol cannot cover the obligation.

---

### Likelihood Explanation

The `settlePnl` path is a normal sequencer operation. The sequencer does not need to be malicious — it only needs to submit a `SettlePnl` transaction for a subaccount that happens to have negative unrealized PnL at the time of settlement. No special privileges, key compromise, or governance capture are required. The only precondition is `state.availableSettle > 0` (which is the normal operating state) and a subaccount with a losing perp position.

---

### Recommendation

Add a guard in `PerpEngine.settlePnl` to skip (or clamp to zero) any product where `canSettle <= 0`:

```solidity
if (canSettle <= 0) {
    productIds >>= 32;
    continue;
}
```

This enforces the invariant that settlement only transfers value **from** `availableSettle` **to** `vQuoteBalance`, never the reverse.

---

### Proof of Concept

State setup:
- `state.availableSettle = 1000e18`
- Subaccount holds a short perp position: `balance.amount = -10e18`, `balance.vQuoteBalance = 500e18`, `price = 200e18`
- `positionPnl = 200e18 * (-10e18) + 500e18 = -1500e18` (negative)

Call: `Endpoint → Clearinghouse.settlePnl → PerpEngine.settlePnl`

```
canSettle = min(-1500e18, 1000e18) = -1500e18

state.availableSettle -= (-1500e18)  →  2500e18   (was 1000e18, now 2500e18)
balance.vQuoteBalance -= (-1500e18)  →  2000e18   (was 500e18, now 2000e18)
totalSettled = -1500e18

spotEngine.updateBalance(QUOTE_PRODUCT_ID, subaccount, -1500e18)
→ spot quote balance decreases by 1500e18
```

Assert: `availableSettle_after (2500e18) > availableSettle_before (1000e18)` — invariant broken. [5](#0-4) [1](#0-0) [4](#0-3)

```text
    function settlePnl(bytes32 subaccount, uint256 productIds)
        external
        returns (int128)
    {
        _assertInternal();
        int128 totalSettled = 0;

        while (productIds != 0) {
            uint32 productId = uint32(productIds & ((1 << 32) - 1));
            // otherwise it means the product is a spot.
            if (productId % 2 == 0) {
                (
                    int128 canSettle,
                    State memory state,
                    Balance memory balance
                ) = getSettlementState(productId, subaccount);

                state.availableSettle -= canSettle;
                balance.vQuoteBalance -= canSettle;

                totalSettled += canSettle;

                _setState(productId, state);
                _setBalanceAndUpdateBitmap(productId, subaccount, balance);
            }
            productIds >>= 32;
        }
        return totalSettled;
    }
```
