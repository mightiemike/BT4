Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the End-User, Allowing Full Allowlist Bypass When the Router Is Allowlisted - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the pool's `msg.sender` — the router contract — when a user routes through `MetricOmmSimpleRouter`. A pool admin who enables router-based swaps on an allowlisted pool must add the router to `allowedSwapper`, which grants every caller of the router implicit allowlist approval, completely nullifying the access control for all router-mediated swaps.

## Finding Description
**Root cause:** `MetricOmmPool.swap` passes `msg.sender` as the first argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← router address when called via router
  recipient,
  ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks this value against `allowedSwapper[msg.sender][sender]` (where `msg.sender` is the pool and `sender` is the router):

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly without forwarding the original caller:

```solidity
// MetricOmmSimpleRouter.sol L72-80
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    ...
);
```

The router stores the original `msg.sender` in transient storage via `_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn)` at line 71, but this is used only for the payment callback — it is never forwarded to the pool or the extension hook.

**Exploit flow:**
1. Pool admin deploys pool with `SwapAllowlistExtension` to restrict swaps to KYC-verified addresses.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` — the only way to allow router-based swaps.
3. Attacker (non-allowlisted EOA) calls `router.exactInputSingle({pool: restrictedPool, ...})`.
4. Router calls `pool.swap(...)` — `msg.sender` inside the pool is the router.
5. Pool calls `_beforeSwap(router, ...)` → extension checks `allowedSwapper[pool][router] == true` → passes.
6. Swap executes; attacker receives output tokens from a pool they are not allowlisted for.

**Why existing guards fail:** The extension has no mechanism to distinguish individual callers behind a shared router. The check `allowedSwapper[pool][sender]` is structurally correct only for direct pool callers; it collapses to a single shared identity (the router) for all router users.

## Impact Explanation
The swap allowlist — a core pool access-control feature — is completely bypassed for all router-mediated swaps once the router is allowlisted. LP providers who deposited into a restricted pool (e.g., expecting only KYC-verified counterparties) are exposed to unrestricted public swap flow against their liquidity at oracle-derived prices they did not consent to trade against with arbitrary counterparties. This constitutes broken core pool functionality and a direct loss of LP assets above Sherlock thresholds.

## Likelihood Explanation
Likelihood is high. The `MetricOmmSimpleRouter` is the primary user-facing entry point. Any pool admin who configures a swap allowlist and also wants users to access the pool via the router must allowlist the router — there is no alternative path. The bypass requires no special privileges, no flash loans, and no exotic token behavior: any EOA calls `exactInputSingle` on the router pointing at the restricted pool. The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

## Recommendation
The extension must gate the end-user, not the intermediary. The cleanest fix is to add an explicit `swapper` field to the swap parameters that the router fills with its `msg.sender` before calling the pool, and have the pool forward that field — not its own `msg.sender` — to extension hooks. The router already captures the original caller in `_setNextCallbackContext(..., msg.sender, ...)` for payment purposes; this same value should be passed through to the pool and then to `_beforeSwap`. Alternatively, the pool could expose a `swapper` override parameter in `swap()` that trusted routers populate, with the pool defaulting to `msg.sender` when the override is `address(0)`.

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, router, true)   // only way to enable router usage
  - Pool admin does NOT allowlist attacker address

Attack:
  - Attacker (non-allowlisted EOA) calls:
      router.exactInputSingle({
          pool: restrictedPool,
          recipient: attacker,
          zeroForOne: true,
          amountIn: X,
          ...
      })

Trace:
  router.exactInputSingle()                          // MetricOmmSimpleRouter.sol L67
    → pool.swap(recipient, ...)                      // msg.sender = router
      → _beforeSwap(router, ...)                     // MetricOmmPool.sol L230-231
        → extension.beforeSwap(router, ...)
          → allowedSwapper[pool][router] == true ✓   // SwapAllowlistExtension.sol L37
      → swap executes, attacker receives output tokens

Result: Non-allowlisted attacker successfully swaps on a restricted pool.
```
