### Title
`op_multiply` Undercharges Execution Cost Due to `limbs_for_int` vs `int_atom` Byte-Length Mismatch — (`File: src/more_ops.rs`)

### Summary

`op_multiply` tracks the accumulated product's byte size in `l0` and uses it in the per-step cost formula. After each multiplication, `l0` is updated via `limbs_for_int(&total)`, which counts bytes using the **unsigned** bit width (`bits().div_ceil(8)`). However, the initial `l0` comes from `int_atom`, which returns the **signed** byte length (`atom_len`). For any positive product whose most-significant bit is set (e.g., 255, 32767, 0xFFFFFF), the signed encoding requires a leading zero byte, so `int_atom` returns one more byte than `limbs_for_int`. This causes `l0` to be underestimated by 1 for every subsequent multiplication step, systematically undercharging the cost.

### Finding Description

In `op_multiply` (`src/more_ops.rs`), the cost formula for each multiplication step is:

```rust
cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
```

`l0` is supposed to represent the byte size of the accumulated product. It is initialized from `int_atom` on the first argument:

```rust
(total, l0) = int_atom(a, arg, "*")?;
```

`int_atom` returns `a.atom_len(args)` — the **signed** big-endian byte length as stored in the allocator. [1](#0-0) 

After each multiplication, `l0` is updated:

```rust
l0 = limbs_for_int(&total);
```

`limbs_for_int` is defined as: [2](#0-1) 

```rust
fn limbs_for_int(v: &Number) -> usize {
    v.bits().div_ceil(8) as usize
}
```

`bits()` returns the number of bits in the **absolute value**, ignoring the sign bit. For a positive number whose MSB is set (e.g., 255 = `0xFF`), the signed encoding requires a leading zero byte (`[0x00, 0xFF]` = 2 bytes), but `bits(255)` = 8, so `limbs_for_int(255)` = 1. The `int_atom` test confirms this discrepancy: `0xffffff` has `atom_len` = 4 (signed: `[0x00, 0xFF, 0xFF, 0xFF]`) but `limbs_for_int(0xffffff)` = 3. [3](#0-2) 

The `limbs_for_int` test helper explicitly strips leading zero bytes: [4](#0-3) 

```rust
let expected = if !bytes.is_empty() && bytes[0] == 0 {
    bytes.len() - 1  // strips the leading zero
} else {
    bytes.len()
};
```

So whenever the accumulated product is a positive number with its MSB set, `l0` is 1 less than the actual signed byte length. The next multiplication step uses this underestimated `l0` in the cost formula, undercharging by at least `MUL_LINEAR_COST_PER_BYTE` = 6 cost units per step. [5](#0-4) 

This is the same structural bug as the Allora M-26 report: a value is modified/computed differently in one part of a two-part formula, causing the formula to produce an incorrect result. Here, `l0` is used in the cost formula (calibrated for signed byte lengths) but updated with an unsigned byte count.

### Impact Explanation

The cost model for `op_multiply` is undercharged whenever an intermediate product has its MSB set. An attacker can craft a CLVM program using the `*` operator with operands chosen to produce intermediate products in the ranges [128–255], [32768–65535], [8388608–16777215], etc. Each such step saves 6+ cost units. Over a chain of multiplications (up to the 1024-byte product limit), the total undercharge accumulates. This allows a program to perform more actual computation than the cost limit should permit, enabling a DoS vector against Chia full nodes by submitting transactions whose true computational cost exceeds what was charged. [6](#0-5) 

### Likelihood Explanation

The trigger condition — an intermediate product with its MSB set — is extremely common in any non-trivial multiplication chain. Any product in the range [128, 255] (e.g., `127 * 2 = 254`) triggers it. An attacker can reliably and deterministically craft CLVM bytes to hit this condition. The entry path is direct: attacker-controlled CLVM program bytes are parsed and executed by `run_program`, which dispatches to `op_multiply` via the operator table. No special privileges or configuration are required. [7](#0-6) 

### Recommendation

Replace `limbs_for_int` with a function that returns the **signed** byte length, consistent with what `int_atom` and `atom_len` return. Concretely, after computing the new product, `l0` should be set to the number of bytes needed to represent `total` in signed big-endian encoding — i.e., `limbs_for_int(&total) + (1 if total > 0 and total's MSB is set, else 0)`. Alternatively, allocate the intermediate result and use `a.atom_len(...)` to get the canonical signed length, or define a helper `signed_byte_len(v: &Number) -> usize` that adds 1 when `v > 0 && v.bit(v.bits() - 1)`.

### Proof of Concept

**Concrete numeric example:**

1. Multiply `[0x7F]` (127, `l0 = 1`) × `[0x02]` (2, `l1 = 1`):
   - Product = 254 = `[0x00, 0xFE]` (signed, 2 bytes)
   - Cost charged: `(1+1)*6 + (1*1)/128 = 12`
   - `l0 = limbs_for_int(254)` = `bits(254).div_ceil(8)` = `8/8` = **1** (should be 2)

2. Next multiply by `[0x7F, 0xFF]` (32767, `l1 = 2`):
   - Cost charged with `l0 = 1`: `(1+2)*6 + (1*2)/128 = 18`
   - Correct cost with `l0 = 2`: `(2+2)*6 + (2*2)/128 = 24`
   - **Undercharge: 6 cost units**

A CLVM program `(* 0x7f 0x02 0x7fff 0x7fff ...)` with a chain of multiplications that keep the intermediate product in the MSB-set range will accumulate this undercharge at every step, allowing the program to exceed its true computational cost relative to what the cost meter records.
