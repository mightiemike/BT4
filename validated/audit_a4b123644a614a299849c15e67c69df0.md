Audit Report

## Title
`uint32` Truncation in `_afterTimelock` Allows Pool Admin to Bypass OracleValueStopLoss Timelock Immediately — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`_afterTimelock` casts `block.timestamp + timelock` to `uint32` without overflow protection. When `timelock` is set to `type(uint32).max`, the sum exceeds `uint32` max and wraps to a value already in the past, causing every subsequent `_requireElapsed` check to pass immediately. This lets the pool admin bypass the timelock on all parameter changes — drawdown, decay, and watermarks — with zero delay, defeating the LP-protection guarantee the timelock was designed to enforce.

## Finding Description

`_afterTimelock` at line 297–299 computes the deadline for every pending admin proposal:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

The addition is performed in `uint256` space, but the explicit `uint32(…)` cast silently truncates in Solidity ≥ 0.8 (explicit downcasts are unchecked). With `block.timestamp ≈ 1,753,000,000` (July 2026) and `timelock = type(uint32).max = 4,294,967,295`, the sum is `6,047,967,295`, which wraps to `1,752,999,999` — one second in the past.

`_requireElapsed` at line 301–303 then checks `block.timestamp < executeAfter`. Since `1,753,000,000 < 1,752,999,999` is false, the guard never fires.

There is no validation on `timelock` at initialization (lines 56–58 validate only `drawdownE6` and `decayPerSecondE8`) and no validation on `newTimelock` in `proposeOracleStopLossTimelock` (lines 78–84). No `MAX_TIMELOCK` constant or `_validateTimelock` function exists anywhere in the codebase.

The exploit path:
1. Pool deployed with `timelock = 0` (permitted; no lower or upper bound enforced).
2. Admin calls `proposeOracleStopLossTimelock(pool, type(uint32).max)`. `_afterTimelock` returns `uint32(block.timestamp + 0) = block.timestamp`. `_requireElapsed(block.timestamp)` passes immediately (`block.timestamp < block.timestamp` is false).
3. Admin calls `executeOracleStopLossTimelock` in the same block. `oracleStopLossConfig[pool].timelock` is now `type(uint32).max`.
4. Admin calls any of `proposeOracleStopLossDrawdown`, `proposeOracleStopLossDecay`, or `proposeOracleStopLossHighWatermarks`. `_afterTimelock` wraps to a past timestamp.
5. Admin immediately calls the corresponding `execute*` function. `_requireElapsed` passes. The parameter change takes effect with zero delay.

## Impact Explanation

This is an admin-boundary break: the pool admin bypasses the timelock that is the sole LP-protection mechanism. Concrete consequences:

- **Drawdown → 0**: the check at line 216–217 (`if (drawdown == 0) return`) disables the stop-loss entirely, removing LP protection with no notice period.
- **Drawdown → E6 (100%)**: every swap breaches the floor and reverts via `OracleStopLossTriggered`, permanently bricking the pool's swap path and trapping LP assets.
- **Decay → 0**: freezes watermarks; any future value dip permanently blocks the affected swap direction.
- **Watermarks → inflated values**: immediately triggers `OracleStopLossTriggered` on the next swap, halting all trading.

LPs who deposited relying on the timelock as a reaction window lose that protection entirely. Swap functionality can be made permanently unusable, constituting broken core pool functionality and loss of LP funds.

## Likelihood Explanation

The attack requires the pool admin to act maliciously, but the exploit path is fully on-chain, requires no external conditions, and is executable in two blocks (or even one block if `timelock = 0` at deployment). The admin needs only to wait through the existing timelock once (to install `type(uint32).max`), after which all future proposals bypass the delay permanently. No special privileges beyond pool admin are required.

## Recommendation

Replace the bare `uint32` cast with a checked conversion and add an explicit upper bound on acceptable timelock values:

```solidity
uint256 private constant MAX_TIMELOCK = 365 days * 2;

function _afterTimelock(address pool_) private view returns (uint32) {
    uint256 result = block.timestamp + oracleStopLossConfig[pool_].timelock;
    require(result <= type(uint32).max, TimelockOverflow());
    return uint32(result);
}
```

Add validation in `initialize` and `proposeOracleStopLossTimelock`:

```solidity
require(timelock <= MAX_TIMELOCK, TimelockTooLarge());
```
