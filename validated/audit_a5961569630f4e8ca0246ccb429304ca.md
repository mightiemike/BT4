Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Original Caller, Allowing Allowlist Bypass via Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension::beforeSwap` receives `sender = router_address` (the pool's immediate `msg.sender`) when a swap is routed through `MetricOmmSimpleRouter`. The hook checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][original_user]`. If the router is allowlisted to enable router-mediated swaps for legitimate users, every address on the network can bypass the allowlist by routing through the router, completely nullifying the access control the extension was designed to enforce.

## Finding Description
In `MetricOmmPool::swap`, the pool passes its own `msg.sender` (the router) as `sender` to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← router address, not original user
  recipient,
  ...
  extensionData
);
```

`ExtensionCalling::_beforeSwap` (L149-177) encodes that `sender` value unchanged and calls the extension. `SwapAllowlistExtension::beforeSwap` (L37) then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router. The original user's identity is stored only in transient callback context for payment (`_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn)` at `MetricOmmSimpleRouter.sol` L71) and never forwarded to the pool's `swap` call or the extension. This creates a binary failure: if the router is NOT allowlisted, all allowlisted users are blocked from using the router; if the router IS allowlisted (the only way to enable router-mediated swaps), every user on the network can bypass the allowlist by routing through the router.

## Impact Explanation
The `SwapAllowlistExtension` is designed to restrict swaps to a specific set of addresses per pool (e.g., institutional traders, whitelisted market makers). This access control is completely nullified for router-mediated swaps. Any unprivileged user can swap in a restricted pool by calling `MetricOmmSimpleRouter::exactInputSingle` (or any `exact*` variant), provided the router is allowlisted. This breaks the core functionality the extension was designed to enforce and constitutes a broken core pool functionality / admin-boundary bypass by an unprivileged path.

## Likelihood Explanation
The router is the standard, documented entrypoint for swaps. Pool admins who want their allowlisted users to use the router must allowlist the router, which opens the pool to all users. There is no in-protocol warning or mitigation. This is a near-certain misconfiguration path that is triggered by normal, expected usage.

## Recommendation
The extension must verify the original initiator, not the immediate caller. Options:
- Require the router to forward the original caller's address in `extensionData`, and have the extension decode and verify it — with the pool or factory enforcing that only trusted routers can set this field.
- Alternatively, document clearly that pools using `SwapAllowlistExtension` must allowlist individual users and must NOT allowlist the router, and that router-mediated swaps are incompatible with this extension.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router usage for allowlisted users.
3. Unprivileged attacker (not in the allowlist) calls `MetricOmmSimpleRouter::exactInputSingle` targeting the restricted pool.
4. The pool calls `_beforeSwap(router, ...)`. The extension checks `allowedSwapper[pool][router] == true`. The check passes.
5. The attacker's swap executes successfully despite not being allowlisted.
