### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the 64-Byte Merkle Forgery Protection Added in v2 — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to close the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase Merkle proof. However, the original `verify_transaction_inclusion` (v1) is still a fully public NEAR contract method. Any unprivileged NEAR caller can invoke v1 directly, bypassing the coinbase proof check entirely and obtaining a `true` return value for a forged transaction-inclusion proof. The Rust `#[deprecated]` attribute is a compiler hint only; it has no effect on external NEAR callers.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces the coinbase proof check: [1](#0-0) 

It requires that `coinbase_merkle_proof` reconstructs the block's `merkle_root` from index 0, which constrains the Merkle tree structure and prevents an attacker from passing an internal 64-byte node as a leaf `tx_id`.

The deprecated v1 function has no such check: [2](#0-1) 

Its only guard is `require!(!args.merkle_proof.is_empty(), ...)` at line 315. It then directly computes `compute_root_from_merkle_proof(args.tx_id, ...)` and compares to `merkle_root` with no validation that `tx_id` is a real leaf transaction rather than an internal node. [3](#0-2) 

The `#[deprecated]` attribute at line 283 is a Rust-compiler-level annotation. It produces a warning for Rust callers at compile time, but it does **not** remove the method from the NEAR contract's public ABI. Any NEAR account can call `verify_transaction_inclusion` directly via a NEAR transaction, completely bypassing v2's coinbase proof requirement.

The `compute_root_from_merkle_proof` function in `merkle-tools` is a pure hash-chain computation with no leaf/node distinction: [4](#0-3) 

This is the exact same structural pattern as the reported `FighterFarm` bug: a security restriction was added to one code path (`verify_transaction_inclusion_v2` / `safeTransferFrom` with check), but an alternative entry point to the same underlying operation (`verify_transaction_inclusion` / inherited `safeTransferFrom(data)`) remains reachable and skips the restriction entirely.

---

### Impact Explanation

An attacker can forge a Bitcoin transaction-inclusion proof and have the contract return `true` for a transaction that was never included in any Bitcoin block. Any downstream NEAR contract that calls `verify_transaction_inclusion` and acts on a `true` result (e.g., releasing funds, minting tokens, unlocking a bridge withdrawal) will be deceived. The corrupted value is the **proof verification result** — the contract's core security guarantee.

The contract's own documentation acknowledges the severity of the 64-byte forgery: [5](#0-4) 

---

### Likelihood Explanation

The attack requires no privileged role, no leaked keys, and no social engineering. Any NEAR account can call `verify_transaction_inclusion` directly. The 64-byte forgery technique is publicly documented (https://www.bitmex.com/blog/64-Byte-Transactions) and the crafted inputs are computable offline. The only prerequisite is that a valid confirmed block header exists in the contract's `headers_pool`, which is the normal operating state of the contract.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public contract ABI. The simplest approach is to change its visibility from `pub fn` to `pub(crate) fn` (or `fn`), so it is only callable internally by `verify_transaction_inclusion_v2`. This mirrors the recommended fix in the external report: enforce the restriction at the single shared internal path rather than relying on callers to use the correct public entry point.

```rust
// Change:
pub fn verify_transaction_inclusion(&self, ...) -> bool { ... }
// To:
fn verify_transaction_inclusion(&self, ...) -> bool { ... }
```

This ensures all external callers are forced through `verify_transaction_inclusion_v2`, which enforces the coinbase proof check.

---

### Proof of Concept

**Setup:** A Bitcoin block `B` with `N` transactions has a Merkle tree. An internal node at depth 1 is the hash of two leaf transactions: `node = SHA256d(tx_leaf_A || tx_leaf_B)`. This `node` value is 32 bytes, but it was computed from a 64-byte input — the same size as a minimal Bitcoin transaction.

**Attack steps:**

1. Attacker identifies a confirmed block `B` in the contract's mainchain (any block in `mainchain_header_to_height`).
2. Attacker computes an internal Merkle node `fake_tx_id = SHA256d(tx_A || tx_B)` for some pair of real transactions in `B`.
3. Attacker constructs a `merkle_proof` that walks from `fake_tx_id` up to `merkle_root` (one level shorter than a real leaf proof, since `fake_tx_id` is already one level up).
4. Attacker calls `verify_transaction_inclusion` with `tx_id = fake_tx_id`, the crafted `merkle_proof`, and `tx_block_blockhash = B`.
5. `compute_root_from_merkle_proof(fake_tx_id, tx_index, &merkle_proof)` correctly reconstructs `merkle_root`.
6. The function returns `true` — certifying that `fake_tx_id` is an included transaction, when it is not.

The v2 function would reject this because the coinbase proof at index 0 would constrain the tree depth, making the forged proof inconsistent. But v1 has no such constraint and accepts it. [6](#0-5) [7](#0-6)

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
