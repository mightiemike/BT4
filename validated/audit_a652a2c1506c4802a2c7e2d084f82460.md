### Title
Linked Signer Can Front-Run Its Own Revocation by Self-Issuing a New `LinkSigner` Transaction — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction in Nado's fast-path (`processTransactionImpl`) is validated with `allowLinkedSigner = true`, meaning the **current linked signer** is permitted to sign a new `LinkSigner` transaction. This allows a malicious or compromised linked signer to front-run a revocation by re-linking themselves (or an address they control) before the owner's revocation is processed. The `linkedSigners` mapping is overwritten directly with no atomic revoke-then-set mechanism, mirroring the ERC-20 `approve` front-running class exactly.

---

### Finding Description

The `linkedSigners` mapping stores a single authorized signer per subaccount, analogous to the ERC-20 `allowances` mapping storing a single approved spender per owner. [1](#0-0) 

When a subaccount owner wants to revoke or rotate their linked signer, they submit a `LinkSigner` transaction. In the fast path (`processTransactionImpl`), this transaction is validated with `allowLinkedSigner = true`: [2](#0-1) 

The `validateSignedTx(..., true)` call at line 581–587 passes `allowLinkedSigner = true` into `validateCompactSignature`, which in turn passes the current linked signer address to `verifier.validateCompactSignature`: [3](#0-2) 

The verifier accepts a signature from **either** the subaccount owner address **or** the current linked signer: [4](#0-3) 

This means the current linked signer can craft and submit a valid `LinkSigner` transaction that overwrites `linkedSigners[subaccount]` to any address they control — including themselves — before the owner's revocation is sequenced.

By contrast, the slow-mode path for `LinkSigner` correctly restricts the operation to the subaccount owner via `validateSender`, which checks `msg.sender` against the address embedded in the `sender` bytes32: [5](#0-4) [6](#0-5) 

The fast path has no equivalent owner-only guard for `LinkSigner`.

---

### Impact Explanation

A malicious or compromised linked signer can:

1. Observe a pending `LinkSigner` revocation in the sequencer queue.
2. Submit their own `LinkSigner` transaction (signed with their own key, which is still valid at that moment) pointing `signer` to a new address they control.
3. If the sequencer processes their transaction first, the revocation is overwritten and the attacker retains persistent signing authority.
4. With persistent signing authority, the attacker can sign `WithdrawCollateral` / `WithdrawCollateralV2` / `TransferQuote` transactions, draining the subaccount's collateral.

The corrupted state is `linkedSigners[subaccount]` — the attacker maintains an authorized signer entry that the owner believed was revoked. The downstream asset delta is full collateral loss from the subaccount.

---

### Likelihood Explanation

Any entity that has ever been granted linked signer status can execute this attack the moment the owner attempts to rotate or revoke them. The attacker needs only to monitor the sequencer's pending transaction queue (which is observable) and submit a competing `LinkSigner` transaction before the revocation is finalized. No privileged access, leaked keys, or governance capture is required beyond the linked signer key the attacker already holds.

---

### Recommendation

The `LinkSigner` transaction in `processTransactionImpl` should be validated with `allowLinkedSigner = false`, restricting signing authority to the subaccount owner only — consistent with the slow-mode path's use of `validateSender`. A linked signer should never be permitted to modify its own authorization entry.

```solidity
// Before (vulnerable):
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // ← allows current linked signer to sign
);

// After (fixed):
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // ← only subaccount owner may change linked signer
);
```

---

### Proof of Concept

1. Owner `O` controls subaccount `S`. They previously linked signer `A` via a `LinkSigner` transaction. `linkedSigners[S] = A`.
2. `O` decides to revoke `A` and submits a `LinkSigner` transaction: `{ sender: S, signer: address(0), nonce: N }`, signed by `O`.
3. `A` observes this pending transaction in the sequencer queue.
4. `A` crafts a competing `LinkSigner` transaction: `{ sender: S, signer: C, nonce: N }` where `C` is an address `A` controls. `A` signs it with their own key.
5. Because `allowLinkedSigner = true` at line 586, `verifier.validateCompactSignature` accepts `A`'s signature (line 312–318 of `Verifier.sol`).
6. The sequencer processes `A`'s transaction first (nonce `N` consumed). `linkedSigners[S] = C`.
7. `O`'s revocation transaction now fails with `ERR_WRONG_NONCE` (nonce `N` already used).
8. `A` (via `C`) retains full signing authority over `S` and can submit `WithdrawCollateralV2` transactions to drain `S`'s collateral to an arbitrary `sendTo` address. [7](#0-6) [8](#0-7)

        bytes32 digest,

        address recovered = ECDSA.recover(digest, signature.r, signature.vs);

            (recovered != address(0)) &&
