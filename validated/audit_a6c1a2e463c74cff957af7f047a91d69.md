### Title
Linked Signer Can Reassign Signing Authority to Any Arbitrary Address Without Subaccount Owner Consent — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction handler validates the signature with `allowLinkedSigner = true`. This means the current linked signer — a delegated hot-wallet key, not the subaccount owner — can sign a `LinkSigner` transaction to replace itself with any arbitrary address. This is a direct privilege-escalation path: a linked signer can silently transfer full signing authority over a subaccount to an attacker-controlled address, without the subaccount owner's knowledge or consent. The slow-mode path for the same operation correctly restricts this to the subaccount owner only, confirming the asymmetry is unintentional.

---

### Finding Description

**Root cause — fast-mode `LinkSigner` handler:**

In `processTransactionImpl`, the `LinkSigner` case is:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // allowLinkedSigner
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` delegates to `validateSignature`, which accepts a signature from either the subaccount owner address OR the current `linkedSigners[sender]`:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

The `Verifier` accepts the signature if `recovered == address(uint160(bytes20(sender))) || recovered == linkedSigner`: [3](#0-2) 

This means the current linked signer can produce a valid `LinkSigner` transaction that writes any new address into `linkedSigners[subaccount]`.

**Contrast with slow-mode path — correctly owner-only:**

The slow-mode `LinkSigner` handler uses `validateSender(txn.sender, sender)`, which requires `address(uint160(bytes20(txn.sender))) == msg.sender` — i.e., only the actual subaccount owner address can set the linked signer:

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The asymmetry is the bug: slow mode enforces owner-only, fast mode does not.

**Attacker-controlled entry path:**

1. Victim sets `linkedSigners[subaccount] = hotWalletAddress` (normal usage — API key, trading bot).
2. Attacker compromises `hotWalletAddress` (leaked key, phishing, server breach — hot wallets are the standard linked-signer use case).
3. Attacker uses `hotWalletAddress` to sign a `LinkSigner` transaction setting `linkedSigners[subaccount] = attackerAddress`.
4. The sequencer includes this transaction (it is a valid, properly signed `LinkSigner` transaction).
5. `attackerAddress` is now the linked signer and can sign any transaction with `allowLinkedSigner = true` on behalf of the victim's subaccount.

**What the attacker can do with the new linked signer:**

- Sign `WithdrawCollateral` (fast mode, `allowLinkedSigner = true`) — funds go to `address(uint160(bytes20(sender)))`, i.e., the subaccount owner's address. Direct theft via this path is blocked.
- Sign `WithdrawCollateralV2` with `sendTo = address(0)` — same destination restriction.
- Sign `MintNlp` / `BurnNlp` (`allowLinkedSigner = true`) — can force unfavorable NLP operations, causing value loss to the victim.
- Sign `LiquidateSubaccount` (`allowLinkedSigner = true`) — can use the victim's subaccount as liquidator, draining it via repeated `LIQUIDATION_FEE` charges, or force the victim's subaccount into positions that become liquidatable.
- Sign `TransferQuote` (`allowLinkedSigner = true`) — restricted by `bytes20(sender) == bytes20(recipient)` in `Clearinghouse.transferQuote`, so cross-address theft is blocked, but intra-address subaccount manipulation is possible. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

A compromised linked signer can silently escalate to permanent signing authority over the victim's subaccount by replacing itself with an attacker-controlled address. The subaccount owner has no on-chain mechanism to detect or prevent this before it occurs. While direct withdrawal theft is constrained (withdrawals route to the subaccount owner's address), the attacker can cause material losses through forced NLP burns/mints at unfavorable oracle prices, repeated liquidation fee extraction, and subaccount position manipulation. The corrupted state is `linkedSigners[subaccount]`, which governs all fast-mode transaction authorization for that subaccount.

**Impact: 5 / 10**

---

### Likelihood Explanation

Linked signers are by design hot wallets, API keys, or trading-bot keys — credentials stored on internet-connected servers. These are the most commonly compromised credential class in DeFi. Any user who has set a linked signer (the primary intended usage pattern) is exposed. The attacker needs only a single valid signature from the current linked signer key, which is a realistic post-compromise action.

**Likelihood: 5 / 10**

---

### Recommendation

In `processTransactionImpl`, change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only subaccount owner may change linked signer
);
```

This aligns the fast-mode path with the slow-mode path, which already correctly enforces owner-only authorization for linked signer changes. The linked signer role should carry signing authority for trading operations, not the authority to reassign itself.

---

### Proof of Concept

1. Victim (`0xALICE`) deposits collateral and sets `linkedSigners[aliceSubaccount] = 0xHOTWALLET` via slow mode (correctly owner-only).
2. Attacker obtains `0xHOTWALLET`'s private key (server breach, leaked `.env`, etc.).
3. Attacker constructs a `SignedLinkSigner` transaction:
   - `sender = aliceSubaccount`
   - `signer = bytes32(uint256(uint160(0xATTACKER)))`
   - `nonce = nonces[0xALICE]` (current nonce)
   - Signs with `0xHOTWALLET`'s key
4. Attacker submits this to the sequencer (or waits for it to be included).
5. `processTransactionImpl` validates the signature — `recovered == 0xHOTWALLET == linkedSigners[aliceSubaccount]` — and writes `linkedSigners[aliceSubaccount] = 0xATTACKER`.
6. Attacker now signs `BurnNlp` transactions on behalf of `aliceSubaccount`, burning Alice's NLP holdings at oracle prices the sequencer accepts, extracting value. Alice cannot recover signing authority without submitting a slow-mode `LinkSigner` transaction (3-day delay). [1](#0-0) [4](#0-3)
