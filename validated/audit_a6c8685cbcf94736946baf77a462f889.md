### Title
Missing `transferOwnership` Delegation for `DirectDepositV1` in `ContractOwner` Permanently Orphans DDA Recovery Path — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner` deploys `DirectDepositV1` instances with itself (`address(this)`) as the Ownable owner but exposes no function to forward a `transferOwnership` call to those DDAs. If `ContractOwner` is ever replaced with a new deployment, all previously deployed `DirectDepositV1` instances become permanently orphaned: the emergency `withdraw()` and `withdrawNative()` recovery paths are irrevocably locked, and any tokens held in those DDAs that have not yet been credited become unrecoverable.

---

### Finding Description

In `ContractOwner.createDirectDepositV1()`, a new `DirectDepositV1` contract is deployed via CREATE2:

```solidity
DirectDepositV1 directDepositV1 = new DirectDepositV1{
    salt: bytes32(uint256(1))
}(address(endpoint), address(spotEngine), subaccount, wrappedNative);
``` [1](#0-0) 

Because `ContractOwner` is the deployer, OpenZeppelin's `Ownable` constructor sets `ContractOwner` as the Ownable owner of the DDA. The DDA's `withdraw()` and `withdrawNative()` functions are guarded by `onlyOwner`: [2](#0-1) [3](#0-2) 

`ContractOwner` exposes `withdrawFromDirectDepositV1()` so the multisig can recover tokens from DDAs: [4](#0-3) 

However, `ContractOwner` provides **no function** that calls `DirectDepositV1.transferOwnership()` to migrate DDA ownership to a new address. The full set of DDA-related functions in `ContractOwner` is limited to `createDirectDepositV1`, `creditDepositV1`, `wrapVaultAsset`, and `withdrawFromDirectDepositV1` — none of which delegate ownership transfer. [5](#0-4) 

If `ContractOwner` must be replaced with a fresh deployment (e.g., a critical bug requiring a new proxy), the new `ContractOwner` address differs from the old one. All previously deployed `DirectDepositV1` instances still record the old `ContractOwner` address as their Ownable owner. The new `ContractOwner` cannot call `withdraw()` or `withdrawNative()` on those DDAs, and no on-chain path exists to transfer DDA ownership to the new contract.

This is structurally identical to the Fathom report: an intermediary contract registers itself as the owner of a sub-resource (`address(this)` → DDA Ownable owner, analogous to `address(this)` → proxy wallet owner in `ProxyWalletRegistry`), but lacks the delegation function needed to migrate that sub-resource ownership when the intermediary is replaced.

---

### Impact Explanation

Tokens held in orphaned `DirectDepositV1` instances that have not yet been credited via `creditDeposit()` become permanently unrecoverable. The `creditDeposit()` path is permissionless and still works for valid deposit assets, but any non-deposit tokens or native ETH that arrived at the DDA and require emergency recovery via `withdraw()` / `withdrawNative()` are permanently locked. The `directDepositV1Address` mapping in the new `ContractOwner` would also be empty, so the new contract would have no record of the old DDAs. [6](#0-5) 

---

### Likelihood Explanation

Low. The scenario requires `ContractOwner` to be replaced with a new deployment rather than upgraded in place. Since `ContractOwner` is an upgradeable proxy (`initializer` pattern, `_disableInitializers()` in constructor), most changes are handled via implementation upgrades that preserve the proxy address. However, a critical bug in the proxy storage layout or a governance decision to migrate to a new deployment would trigger this condition, at which point the missing delegation function causes permanent asset loss with no recovery path. [7](#0-6) 

---

### Recommendation

Add a function in `ContractOwner` that forwards a `transferOwnership` call to a specified `DirectDepositV1` instance:

```solidity
function transferDirectDepositV1Ownership(bytes32 subaccount, address newOwner)
    external
    onlyOwner
{
    address payable dda = directDepositV1Address[subaccount];
    require(dda != address(0), "no dda");
    DirectDepositV1(dda).transferOwnership(newOwner);
}
```

This mirrors the Fathom report recommendation: expose a delegation function so the intermediary contract can transfer sub-resource ownership when needed.

---

### Proof of Concept

1. Multisig calls `ContractOwner_OLD.createDirectDepositV1(subaccount)` → `DirectDepositV1` deployed at `DDA_OLD`; `ContractOwner_OLD` is its Ownable owner. [8](#0-7) 
2. A critical bug is found in `ContractOwner_OLD`; protocol deploys `ContractOwner_NEW` at a different address.
3. `ContractOwner_NEW.directDepositV1Address[subaccount]` is `address(0)` (fresh storage).
4. `DDA_OLD.owner()` still returns `ContractOwner_OLD`. [2](#0-1) 
5. Multisig calls `ContractOwner_NEW.withdrawFromDirectDepositV1(subaccount, token)` → reverts at `require(directDepositV1 != address(0), "no dda")`. [9](#0-8) 
6. Even if the multisig manually records `DDA_OLD` in `ContractOwner_NEW`, calling `DDA_OLD.withdraw()` reverts because `msg.sender == ContractOwner_NEW ≠ ContractOwner_OLD == owner`. [10](#0-9) 
7. No function exists in either `ContractOwner_OLD` or `ContractOwner_NEW` to call `DDA_OLD.transferOwnership(ContractOwner_NEW)`.
8. Tokens in `DDA_OLD` requiring emergency recovery are permanently locked.
