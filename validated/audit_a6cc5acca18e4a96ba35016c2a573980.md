### Title
`run_mainchain_gc` Missing Pause-Bypass Roles Breaks `submit_blocks` for Authorized Emergency Relayers - (`contract/src/lib.rs`)

### Summary

`submit_blocks` unconditionally calls `self.run_mainchain_gc()` as part of its execution. `run_mainchain_gc` is decorated with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`, meaning when the contract is paused it only allows callers holding `Role::UnrestrictedRunGC` to proceed. However, the roles that are authorized to bypass the pause on `submit_blocks` — `Role::UnrestrictedSubmitBlocks` and `Role::DAO` — are absent from `run_mainchain_gc`'s bypass list. As a result, any call to `submit_blocks` by an emergency relayer while the contract is paused will always panic inside `run_mainchain_gc`, making the `UnrestrictedSubmitBlocks` emergency-bypass role completely non-functional.

### Finding Description

The `#[trusted_relayer]` macro applied to the `BtcLightClient` impl block is configured with:

```rust
#[trusted_relayer(
    bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks),
    ...
)]
``` [1](#0-0) 

The project wiki and role documentation explicitly state that `Role::UnrestrictedSubmitBlocks` allows calling `submit_blocks` even when the contract is paused — this is the designated emergency-relayer bypass.

`submit_blocks` is decorated with `#[pause]` and `#[trusted_relayer]`, and its body unconditionally calls `self.run_mainchain_gc(num_of_headers)`:

```rust
#[payable]
#[pause]
#[trusted_relayer]
pub fn submit_blocks(...) -> PromiseOrValue<()> {
    ...
    self.run_mainchain_gc(num_of_headers);   // line 181
    ...
}
``` [2](#0-1) 

`run_mainchain_gc` is a separate public `#[near]` method with its own independent pause guard:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
``` [3](#0-2) 

The bypass list for `run_mainchain_gc` contains only `Role::UnrestrictedRunGC`. `Role::UnrestrictedSubmitBlocks` and `Role::DAO` are absent.

In NEAR Protocol, when a contract method calls another method on itself (`self.run_mainchain_gc(...)`), `env::predecessor_account_id()` inside the callee still resolves to the original external caller — not the contract itself. Therefore the `#[pause]` guard inside `run_mainchain_gc` evaluates the roles of the original relayer account. An emergency relayer holding `Role::UnrestrictedSubmitBlocks` (but not `Role::UnrestrictedRunGC`) will pass the `submit_blocks` pause check but immediately panic inside `run_mainchain_gc`, reverting the entire transaction.

### Impact Explanation

The `Role::UnrestrictedSubmitBlocks` role exists precisely to keep the chain-tracking function operational during an emergency pause — for example, when the contract is paused to respond to a security incident but block headers must still be ingested to maintain liveness. Because `run_mainchain_gc` is always called inside `submit_blocks` and its pause bypass does not include `Role::UnrestrictedSubmitBlocks` or `Role::DAO`, the emergency bypass is entirely non-functional. No relayer — regardless of role — can submit block headers while the contract is paused, defeating the purpose of the role and breaking a core protocol function.

### Likelihood Explanation

The contract is designed to be pausable for security incidents. Any operator who grants `Role::UnrestrictedSubmitBlocks` to an emergency relayer and then pauses the contract will immediately discover that `submit_blocks` is broken. The trigger is a normal operational sequence (pause contract → emergency relayer submits blocks), not an exotic attack path.

### Recommendation

Add `Role::UnrestrictedSubmitBlocks` and `Role::DAO` to the pause-bypass list of `run_mainchain_gc` to match the bypass roles already granted on `submit_blocks`:

```rust
// Before
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {

// After
#[pause(except(roles(Role::UnrestrictedRunGC, Role::UnrestrictedSubmitBlocks, Role::DAO)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
``` [3](#0-2) 

### Proof of Concept

1. Deploy the contract and pause it (via an account holding `Role::PauseManager`).
2. Grant `Role::UnrestrictedSubmitBlocks` to a relayer account (but do **not** grant `Role::UnrestrictedRunGC`).
3. Have the relayer call `submit_blocks` with one valid block header and the required deposit.
4. Observe: the call panics with the pause error originating from inside `run_mainchain_gc`, not from `submit_blocks` itself.
5. Grant the same relayer `Role::UnrestrictedRunGC` in addition and repeat — the call now succeeds, confirming the missing role is the root cause.
