Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Originating User, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`, which forwards it verbatim to `SwapAllowlistExtension.beforeSwap`. When a user swaps through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the extension checks whether the **router** is allowlisted rather than the originating user. Any user can bypass a pool's swap allowlist by routing through the public router, fully defeating the admin-configured access control.

## Finding Description
`MetricOmmPool.swap` invokes `_beforeSwap(msg.sender, recipient, ...)` at line 230:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- whoever called pool.swap()
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` encodes `sender` verbatim into the extension call (lines 149–177). `SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) is used, the router directly calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` (lines 72–80 of `MetricOmmSimpleRouter.sol`), making the router the `sender` delivered to the extension.

For the router to be usable at all by allowlisted users, the pool admin must call `setAllowedToSwap(pool, router, true)`. Once that is done, `allowedSwapper[pool][router] == true`, and the gate passes for **every** caller of the router regardless of their individual allowlist status. The extension has no mechanism to recover the originating EOA.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates on `owner` (the LP-share recipient, supplied explicitly by the caller), not on `sender` (the dispatcher), making it robust to intermediary contracts.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, whitelisted market makers) is fully bypassed. Any unprivileged user routes through `MetricOmmSimpleRouter`, the extension sees the allowlisted router address, and the swap executes. This is an **admin-boundary break**: the pool admin's access control is circumvented by an unprivileged path (the public router). Depending on pool configuration, this results in unauthorized price impact, unauthorized extraction of LP value, or violation of regulatory/operational constraints the allowlist was designed to enforce — constituting direct loss of protocol integrity and potentially LP principal.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard, publicly documented entry point for swaps. No special privilege is required. The pool admin has no on-chain mechanism to prevent router-mediated calls while keeping the router usable for allowlisted users. Any user aware of the router can exploit this unconditionally against any pool that deploys `SwapAllowlistExtension` with the router allowlisted. Likelihood is **high**.

## Recommendation
Replace the `sender` check with the originating user identity. Two options:

1. **Gate on `recipient` instead of `sender`**: For exact-input swaps the recipient is the user-controlled address. Mirror `DepositAllowlistExtension`'s approach — check the economically relevant actor (the token recipient), not the intermediate dispatcher.

2. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires router cooperation but is trustless if the extension validates the encoding.

The extension interface already delivers both `sender` and `recipient` to `beforeSwap`, so option 1 requires only changing `sender` to `recipient` in the allowlist lookup.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` — required for any router-mediated swap to function.
3. Pool admin does **not** allowlist `attacker`.
4. `attacker` calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
5. Router calls `pool.swap(recipient=attacker, ...)` — router is `msg.sender` to the pool.
6. Pool calls `_beforeSwap(sender=router, ...)` → extension receives `sender=router`.
7. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
8. Swap executes for `attacker` despite `allowedSwapper[pool][attacker] == false`.
