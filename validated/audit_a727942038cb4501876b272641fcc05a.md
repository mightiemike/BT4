Audit Report

## Title
`OracleValueStopLossExtension` Uses Arithmetic Mean Instead of Geometric Mean for Mid-Price, Causing Systematic Stop-Loss Miscalibration — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`_afterSwapOracleStopLoss` computes `midPriceX64` as the arithmetic mean of bid and ask prices, while every other mid-price derivation in the protocol (`SwapMath.midAndSpreadFeeX64FromBidAsk`, `MetricOmmPoolDataProvider._marginalBestBidAsk`, `getLiquidityDepth`) uses the geometric mean. Because AM ≥ GM always, the stop-loss systematically overestimates `metricToken1` and underestimates `metricToken0`, making the guard less protective for `!zeroForOne` swaps (token1 in, token0 out) — the direction that drains token0 from LPs.

## Finding Description

In `_afterSwapOracleStopLoss`, the mid-price is computed as:

```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
``` [1](#0-0) 

Every other component in the protocol uses the geometric mean. `SwapMath.midAndSpreadFeeX64FromBidAsk` (used for actual swap pricing):

```solidity
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);
``` [2](#0-1) 

`MetricOmmPoolDataProvider` (used for depth/quote views):

```solidity
uint256 midPriceX64 = Math.sqrt(uint256(env.oracleBidX64) * uint256(env.oracleAskX64));
``` [3](#0-2) 

The per-share metrics are: [4](#0-3) 

```
metricToken0 = t0·SCALE/shares + (t1 · 2^64 / mid) · SCALE/shares
metricToken1 = (t0 · mid / 2^64) · SCALE/shares + t1·SCALE/shares
```

Because AM > GM (by AM-GM inequality, strictly when bid ≠ ask):

| Metric | Effect of AM > GM |
|---|---|
| `metricToken0` | **Underestimated** (`t1/mid` term shrinks) |
| `metricToken1` | **Overestimated** (`t0·mid` term grows) |

The stop-loss blocks `!zeroForOne` swaps when `metricToken1 < hwm1 · floorMultiplier / E6`: [5](#0-4) 

Because `metricToken1` is overestimated by the AM, the breach condition `metricT1 < floor` is harder to satisfy. The stop-loss fails to trigger for `!zeroForOne` swaps that should have been blocked.

The watermarks are ratcheted up automatically from the AM-based metric during normal operation: [6](#0-5) 

This means the high watermark is set using AM, and the floor check uses AM — but the actual swap pricing uses GM. When the true (GM-based) value per share has fallen below the configured drawdown floor, the AM-based check may still report no breach, allowing the swap to proceed and draining LP token0 beyond the protection boundary.

## Impact Explanation

The `OracleValueStopLossExtension` is the primary on-chain mechanism protecting LP capital from oracle-driven value leakage. When the stop-loss fails to trigger for `!zeroForOne` swaps, token0 continues to leave the pool at a price the stop-loss was configured to block. LPs suffer direct loss of token0 principal beyond the configured drawdown floor. The miscalibration magnitude scales with oracle spread: a 10% spread causes AM to exceed GM by ~0.125% of mid, which is material relative to tight drawdown floors (e.g., 0.5%). This constitutes a direct loss of LP principal above Sherlock thresholds for pools with non-trivial oracle spreads.

## Likelihood Explanation

The error is present on every swap touching a pool configured with `OracleValueStopLossExtension` and a non-zero drawdown. No special conditions are required — any public `swap` call with `!zeroForOne` direction triggers the miscalibrated check. An attacker needs only to time their swap to a moment when the true GM-based metric has breached the floor while the AM-based metric has not. This window is persistent and proportional to the oracle spread.

## Recommendation

Replace the arithmetic mean with the geometric mean, consistent with `SwapMath.midAndSpreadFeeX64FromBidAsk`:

```diff
- uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
+ uint256 midPriceX64 = Math.sqrt(uint256(bidPriceX64) * uint256(askPriceX64));
```

## Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension`, `drawdownE6 = 5000` (0.5% floor), and a 10% oracle spread (bid = 0.95·P, ask = 1.05·P, so GM = P, AM ≈ 1.00125·P).
2. Execute a `!zeroForOne` swap that drains token0 such that the GM-based `metricToken1` falls to exactly `hwm1 · 0.995` (just below the 0.5% floor).
3. Compute AM-based `metricToken1`: because AM ≈ 1.00125·GM, the AM-based metric is ~0.125% higher, placing it above the floor at `hwm1 · 0.99625`.
4. Observe that `_afterSwapOracleStopLoss` does not revert; the swap executes; LP token0 is drained beyond the configured protection boundary.
5. A Foundry unit test can reproduce this by calling `afterSwap` directly with bid/ask prices set to create a 10% spread, storing bin state at the boundary value, and verifying the revert does not occur when it should.
