### Title
Stateless OS-invariant validation (DA mode, `account_deployment_data`, `paymaster_data`) is enforced on the RPC/gateway path but silently skipped on the consensus-to-internal conversion path — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`, `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`, `crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

The gateway's `StatelessTransactionValidator` explicitly enforces several OS-level invariants on every `RpcTransaction` before it enters the mempool. The code comments state these checks exist specifically *"to prevent transactions from failing the OS."* The consensus proposal-validation path (`validate_proposal` → `handle_proposal_part` → `convert_consensus_tx_to_internal_consensus_tx` → `convert_rpc_tx_to_internal`) performs the same public-to-internal conversion but **never calls the stateless validator**. A malicious proposer can therefore inject transactions that carry `nonce_data_availability_mode = L2`, `fee_data_availability_mode = L2`, non-empty `account_deployment_data`, or non-empty `paymaster_data` directly into a consensus proposal. The validator node accepts and forwards them to the blockifier without any of the gateway's protective gates.

---

### Finding Description

**Protected path — RPC/gateway (`add_tx_inner`):**

```
stateless_tx_validator.validate(&tx)   // ← enforces all OS invariants
  → validate_nonce_data_availability_mode   // must be L1
  → validate_fee_data_availability_mode     // must be L1
  → validate_empty_account_deployment_data  // must be empty
  → validate_empty_paymaster_data           // must be empty
  → validate_resource_bounds
  → validate_tx_size
  → validate_contract_address
convert_rpc_tx_to_internal_rpc_tx(tx)  // hash computed with chain_id
```

**Unprotected path — consensus (`handle_proposal_part`):**

```
convert_consensus_tx_to_internal_consensus_tx(tx)
  → convert_rpc_tx_to_internal(tx)     // hash computed with chain_id
      // NO stateless_tx_validator.validate call
```

The `StatelessTransactionValidator::validate` function is called in `add_tx_inner` at line 236 of `crates/apollo_gateway/src/gateway.rs`, before any conversion. The consensus path in `crates/apollo_consensus_orchestrator/src/validate_proposal.rs` (lines 598–646) calls only `transaction_converter.convert_consensus_tx_to_internal_consensus_tx(tx)` and then `batcher.send_txs_for_proposal`. There is no intervening call to the stateless validator.

The comments in the stateless validator make the intent explicit:

> *"The Starknet OS enforces that the nonce data availability mode is L1. We add this validation here in the gateway to prevent transactions from failing the OS."*

> *"The Starknet OS enforces that the deployer data is empty. We add this validation here in the gateway to prevent transactions from failing the OS."*

These are not optional policy checks — they are pre-conditions the OS requires for every transaction. The gateway adds them as a defense-in-depth gate. The consensus path has no equivalent gate.

**Exact divergent field values:**

| Field | Gateway path | Consensus path |
|---|---|---|
| `nonce_data_availability_mode` | Rejected if `≠ L1` | Any value accepted |
| `fee_data_availability_mode` | Rejected if `≠ L1` | Any value accepted |
| `account_deployment_data` | Rejected if non-empty | Any value accepted |
| `paymaster_data` | Rejected if non-empty | Any value accepted |

---

### Impact Explanation

The blockifier runs the account's `__validate__` entry point, which checks the ECDSA signature over the transaction hash. The transaction hash for a V3 invoke includes `nonce_data_availability_mode` and `fee_data_availability_mode` (via `concat_data_availability_mode` in `get_invoke_transaction_v3_hash`). A malicious proposer who controls an account contract can sign a transaction with `nonce_data_availability_mode = L2`; the blockifier will verify the signature as valid and execute the transaction. The blockifier does **not** independently enforce the L1-only DA mode constraint — that enforcement lives in the OS. The transaction is therefore included in the committed block, but the OS cannot generate a valid proof for it. This produces a committed block state that cannot be proven, matching the **"Wrong state … from blockifier/syscall/execution logic for accepted input"** (Critical) impact category. At minimum it is a **"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing"** (High) impact, since the validator's admission logic accepts what the gateway would reject.

---

### Likelihood Explanation

Any consensus participant can become a proposer in a round-robin or stake-weighted schedule. The attacker needs only to:
1. Deploy a custom account contract that signs transactions with `nonce_data_availability_mode = L2`.
2. Wait for (or arrange) a round in which they are the proposer.
3. Craft a `ConsensusTransaction` protobuf with the invalid field and stream it in a `TransactionBatch`.

No special privilege beyond being a consensus participant is required. The P2P mempool path is not a viable alternative because `MempoolP2pRunner` forwards received transactions through `gateway_client.add_tx`, which does apply the stateless validator. Only the consensus proposal stream bypasses it.

---

### Recommendation

In `handle_proposal_part` (or inside `convert_consensus_tx_to_internal_consensus_tx`), apply the same stateless validation that the gateway applies before forwarding transactions to the batcher. Concretely, before calling `batcher.send_txs_for_proposal`, iterate over the converted `InternalConsensusTransaction` batch and reject any transaction whose underlying `RpcTransaction` fields violate the OS invariants (`nonce_data_availability_mode == L1`, `fee_data_availability_mode == L1`, `account_deployment_data.is_empty()`, `paymaster_data.is_empty()`). Alternatively, extract the invariant checks from `StatelessTransactionValidator` into a shared function callable from both the gateway and the consensus orchestrator.
