Audit Report

## Title
`PriceVelocityGuardExtension.beforeSwap` Silently Bypasses Velocity Check When `lastMidPriceX64` Is Uninitialized — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary

`PriceVelocityGuardExtension.beforeSwap` unconditionally commits the current oracle mid-price to storage before gating the entire velocity check behind `if (prevMid != 0)`. Because `priceVelocityState[pool]` is zero-initialized by default and the contract has no `initialize` hook that seeds `lastMidPriceX64`, the guard is silently skipped on the first swap of every newly configured pool. The current oracle price — regardless of how far it has moved since pool deployment — is committed as the new baseline, and LPs are exposed to unconstrained bad-price execution on that first swap.

## Finding Description

`PriceVelocityGuardExtension` stores per-pool state in `mapping(address pool => PriceVelocityState) public priceVelocityState`, which is zero-initialized on first use. [1](#0-0) 

In `beforeSwap`, the current mid-price is written to storage unconditionally at lines 57–58, before the velocity check is gated at line 60:

```solidity
s.lastMidPriceX64 = midPrice;          // written unconditionally
s.lastUpdateBlock = uint64(block.number);

if (prevMid != 0) {                    // skipped when prevMid == 0 (default)
    ...velocity check...
}
``` [2](#0-1) 

When `lastMidPriceX64` is zero (the default for any pool that has not had `setLastMidPrice` called), `prevMid == 0` and the entire velocity check is skipped. The current oracle mid-price — regardless of magnitude — is committed as the new baseline.

The only seeding mechanism is `setLastMidPrice`, which is `onlyPoolAdmin` and has no enforcement ordering relative to the first swap: [3](#0-2) 

The factory does call `initialize` on all configured extensions at pool creation: [4](#0-3) 

However, `PriceVelocityGuardExtension` does not override `initialize` to seed `lastMidPriceX64`. By contrast, `OracleValueStopLossExtension` correctly overrides `initialize` to seed its state at pool creation: [5](#0-4) 

Additionally, the `beforeSwap` override drops the `onlyPool` modifier that the base class declares, meaning any address — not just a registered pool — can call `beforeSwap` directly and write to `priceVelocityState[msg.sender]`: [6](#0-5) [7](#0-6) 

## Impact Explanation

The velocity guard is the protocol's primary defense against rapid oracle price movement reaching live swaps. When `prevMid == 0`, the guard fails open: the first swap on any pool with this extension executes at whatever oracle price the price provider returns, with no velocity constraint applied. If the oracle price has moved significantly since pool deployment (genuine market move, brief oracle spike, or a Pyth update with a large confidence interval that passes staleness checks), LPs are exposed to a swap at a price the guard was configured to block. After this first swap, the extreme price is committed as the new baseline, potentially allowing a second swap that is within the velocity envelope relative to the manipulated baseline but still far from the true market price. This constitutes bad-price execution reaching a live pool swap, with direct LP principal loss — a qualifying impact under the contest's "Bad-price execution: stale, inverted, unbounded, or unclamped bid/ask quote reaches a pool swap" category.

## Likelihood Explanation

Any pool that deploys `PriceVelocityGuardExtension` without the admin calling `setLastMidPrice` before the first swap is vulnerable. This is the default state: the extension has no `initialize` hook, so the zero state is the starting condition for every pool. The first public swapper — any unprivileged user — triggers the bypass. No special permissions or oracle manipulation are required; the bypass is structural and reachable on every newly configured pool.

## Recommendation

Fail closed when no prior observation exists. Options:

1. **Override `initialize`** — seed `lastMidPriceX64` from the pool's live oracle price at pool creation, analogous to `OracleValueStopLossExtension.initialize`. This is the most robust fix as it eliminates the unseeded window entirely.

2. **Revert if `prevMid == 0`** — require the admin to seed the price before the first swap is permitted:
   ```solidity
   if (prevMid == 0) revert NoPriorObservation();
   ```

3. **Restore `onlyPool`** — add the `onlyPool` modifier to the `beforeSwap` override to match the base class declaration and prevent direct external calls from writing to extension state.

## Proof of Concept

1. Deploy a pool with `PriceVelocityGuardExtension` configured (`maxChangePerBlockE18 = 1e16`, i.e., 1% per block). Do **not** call `setLastMidPrice`.
2. Oracle price at pool deployment: `P0 = 1000 USD`. Oracle price at block N: `P1 = 2000 USD` (100% move, far exceeding the 1% cap).
3. Any user calls `pool.swap(...)` at block N. The pool fetches `bid/ask` from the oracle (P1-based) and calls `PriceVelocityGuardExtension.beforeSwap`.
4. `prevMid = s.lastMidPriceX64 = 0`. The `if (prevMid != 0)` branch is skipped. The swap executes at P1 with no velocity check.
5. `s.lastMidPriceX64` is now set to `sqrt(bid1 * ask1)` ≈ P1.
6. LPs suffer losses from a swap at 2× the price the guard was configured to block.
7. Contrast: if `setLastMidPrice(pool, P0_encoded)` had been called before step 3, the swap at block N would revert with `PriceVelocityExceeded` because `(P1 - P0)^2 / P0^2 = 1.0 >> maxChange^2 * (1 + blockDiff)`.
