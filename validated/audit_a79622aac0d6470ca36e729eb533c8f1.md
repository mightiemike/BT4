### Title
`limbs_for_int` Underestimates Byte Length for Negative Integers, Causing Cost Undercharge and Size-Limit Bypass in `op_multiply` — (`src/more_ops.rs`)

---

### Summary

`limbs_for_int` computes the byte length of a `BigInt` using `bits().div_ceil(8)`. For negative numbers whose absolute value has its MSB set (i.e., `bits(|n|) % 8 == 0` and `|n|` is not a power of two), the actual CLVM signed-big-endian byte length is `bits()/8 + 1`, but `limbs_for_int` returns `bits()/8`. In `op_multiply`, the variable `l0` is updated each iteration with `limbs_for_int(&total)` and is used both for cost calculation and for the 1024-byte size-limit guard. When `total` is a negative number in this class, `l0` is off by one, causing the multiplication cost to be undercharged and allowing a 1025-byte intermediate product to bypass the intended 1024-byte cap.

---

### Finding Description

**Root cause — `limbs_for_int` (`src/more_ops.rs:100-102`):**

```rust
fn limbs_for_int(v: &Number) -> usize {
    v.bits().div_ceil(8) as usize
}
```

`BigInt::bits()` returns the number of bits in the **absolute value** of the integer. For a negative number `n`, the CLVM signed-big-endian encoding requires an extra byte whenever the two's-complement of `|n|` in `bits()/8` bytes does not have its MSB set — i.e., when `|n| > 2^(bits()-1)`.

Concrete example:

| Value | `|n|` | `bits()` | `limbs_for_int` | Actual CLVM bytes |
|---|---|---|---|---|
| `-128` | `0x80` | 8 | **1** | `[0x80]` → 1 ✓ |
| `-129` | `0x81` | 8 | **1** | `[0xff, 0x7f]` → 2 ✗ |
| `-255` | `0xff` | 8 | **1** | `[0xff, 0x01]` → 2 ✗ |
| `-256` | `0x100` | 9 | **2** | `[0xff, 0x00]` → 2 ✓ |

The affected range is every negative integer in `(-2^(8k), -2^(8k-1))` for any `k ≥ 1`.

The existing test `test_limbs_for_int` only exercises values whose first byte is `0x00` (stripped) or whose first byte is non-zero and the value is positive or a power-of-two-magnitude negative. It never tests e.g. `[0xff, 0x7f]` (`-129`), so the defect is invisible to the test suite. [1](#0-0) 

**Propagation into `op_multiply` (`src/more_ops.rs:586-656`):**

After each multiplication step, `l0` is refreshed:

```rust
l0 = limbs_for_int(&total);
if l0 > 1024 {
    return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
}
```

`l0` is then used in the next iteration's cost formula:

```rust
cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
```

When `total` is a negative number in the affected class, `l0` is 1 less than the true byte length. Two consequences follow:

1. **Cost undercharge.** The linear term is short by `MUL_LINEAR_COST_PER_BYTE = 6` and the quadratic term by `l1 / 128`. For `l1 = 256` (the per-argument maximum), the undercharge is 8 cost units per affected multiplication step.

2. **Size-limit bypass.** If `total` is a negative number with `|total| ∈ (2^8191, 2^8192)` (actual byte length 1025), `limbs_for_int` returns 1024. The guard `if l0 > 1024` evaluates to `false` and execution continues, allowing a 1025-byte intermediate product that exceeds the intended cap. [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**Cost undercharge (consensus divergence).** Any full node or light client that re-evaluates a CLVM program containing `*` with negative intermediate products will compute a lower cost than the true cost. If a reference implementation uses the correct byte length, the two implementations will disagree on whether the program fits within `max_cost`, creating a consensus split. Even within a single implementation, an attacker can execute more computation per cost unit than the model intends.

**Size-limit bypass.** The 1024-byte cap on intermediate products is a resource-exhaustion guard. Bypassing it by one byte allows a 1025-byte intermediate, which can then be multiplied by another argument up to 256 bytes, producing a result up to ~1281 bytes before the next guard fires. While the cascade is bounded, it violates the invariant that no intermediate exceeds 1024 bytes.

---

### Likelihood Explanation

The trigger is fully attacker-controlled: any CLVM program submitted to `run_program` that uses the `*` operator with arguments chosen to produce a negative intermediate product in the affected range. No special permissions, keys, or side-channels are required. The affected range of negative integers is dense (e.g., all of `-129` through `-255`, `-32769` through `-65535`, etc.), so it is trivially reachable.

---

### Recommendation

Replace `limbs_for_int` with a function that correctly accounts for the sign-extension byte:

```rust
fn limbs_for_int(v: &Number) -> usize {
    let bits = v.bits();
    if bits == 0 {
        return 0;
    }
    let byte_len = bits.div_ceil(8) as usize;
    // For negative numbers whose magnitude fills all bits of `byte_len` bytes,
    // the two's-complement representation needs an extra byte so the MSB
    // is not misread as a sign bit.
    if v.sign() == Sign::Minus && bits % 8 == 0 {
        // Check whether |v| > 2^(bits-1), i.e. the magnitude is not a pure power of two.
        // Equivalently: the magnitude has any bit set below the MSB.
        let magnitude = v.magnitude(); // BigUint
        if magnitude != &(BigUint::one() << (bits - 1)) {
            return byte_len + 1;
        }
    }
    byte_len
}
```

Alternatively, allocate the number into the `Allocator` and use `a.atom_len(node)` as the authoritative byte length, since `new_number` already serialises correctly.

Add test vectors to `test_limbs_for_int` covering `[0xff, 0x7f]` (`-129`), `[0xff, 0x01]` (`-255`), and their multi-byte analogues.

### Proof of Concept

**Cost undercharge (small program):**

CLVM program `(* -129 1)` produces `-129`.  
`limbs_for_int(-129)` returns **1** instead of **2**.  
In a chain `(* -129 1 1 … 1)` with `N` trailing `1`s, each step charges `(1+1)*6 = 12` instead of `(2+1)*6 = 18`, saving 6 cost units per step.

**Size-limit bypass:**

1. Construct a positive atom `p` of exactly 1024 bytes with its MSB set (e.g., `0x80 || 0x00 * 1023`).
2. Evaluate `(* p -1)`. The result is `-p`, with `|p| ∈ (2^8191, 2^8192)`.
3. `limbs_for_int(-p)` returns **1024**; the guard `if l0 > 1024` passes.
4. The 1025-byte intermediate is now live in `total`, violating the invariant.
5. A subsequent multiplication step uses `l0 = 1024` instead of 1025, undercharging the quadratic cost term by `256 / 128 = 2` units (for a 256-byte next argument). [4](#0-3) [5](#0-4) [6](#0-5)
