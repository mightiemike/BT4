### Title
Hardcoded Quote Token Price of $1.00 Prevents Accurate Health and Collateral Valuation During Stablecoin Depeg - (File: `core/contracts/SpotEngine.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

The Nado protocol permanently hardcodes the price of `QUOTE_PRODUCT_ID` (the quote/USDC token) to `ONE` ($1.00) in two independent locations. There is no oracle update path for this price. During a USDC depeg event, all health calculations, collateral valuations, and minimum deposit checks continue to treat USDC as worth exactly $1.00, enabling under-collateralized borrowing and blocking legitimate liquidations.

---

### Finding Description

In `SpotEngine.initialize()`, the `RiskStore` for `QUOTE_PRODUCT_ID` is set with a hardcoded `priceX18: ONE`:

```solidity
_risk().value[QUOTE_PRODUCT_ID] = RiskHelper.RiskStore({
    longWeightInitial: 1e9,
    shortWeightInitial: 1e9,
    longWeightMaintenance: 1e9,
    shortWeightMaintenance: 1e9,
    priceX18: ONE   // hardcoded $1.00, never updated
});
``` [1](#0-0) 

This `priceX18` value is stored in the shared `RISK_STORAGE` slot and is read directly by `BaseEngine._calculateProductHealth()` for every product including `QUOTE_PRODUCT_ID`:

```solidity
health += amount.mul(weight).mul(risk.priceX18);
``` [2](#0-1) 

`SpotEngineState._getBalance()` returns `(amount, 0)` for all spot products including the quote:

```solidity
function _getBalance(uint32 productId, bytes32 subaccount)
    internal view override returns (int128, int128)
{
    return (getBalance(productId, subaccount).amount, 0);
}
``` [3](#0-2) 

This means the quote balance's health contribution is computed as `amount * weight * priceX18`, where `priceX18` is permanently `ONE`. There is no code path that calls `updatePrice(QUOTE_PRODUCT_ID, ...)` — `BaseEngine.updatePrice` is restricted to `_clearinghouse`, and the clearinghouse only calls it for `NLP_PRODUCT_ID`: [4](#0-3) 

A second independent hardcoding exists in `Clearinghouse.checkMinDeposit()`:

```solidity
int128 priceX18 = ONE;
if (productId != QUOTE_PRODUCT_ID) {
    priceX18 = _getPriceX18(productId);
}
``` [5](#0-4) 

This explicitly skips the oracle for `QUOTE_PRODUCT_ID` and hardcodes $1.00 for minimum deposit threshold checks.

---

### Impact Explanation

If the quote token (USDC) depegs — e.g., to $0.90 — the following concrete state corruptions occur:

1. **Under-collateralized borrowing enabled**: `getHealth()` calls `spotEngine.getHealthContribution()` → `_calculateProductHealth()` → values USDC collateral at $1.00 instead of $0.90. A user with 1000 USDC collateral appears to have $1000 of health when they actually have $900. They can borrow up to the $1000 limit, leaving the protocol with a $100 shortfall per 1000 USDC.

2. **Liquidations blocked**: Positions that should be underwater (maintenance health < 0) appear healthy because USDC collateral is overvalued. `ClearinghouseLiq` liquidation checks depend on `getHealth()` returning a negative value to proceed; with the hardcoded price, this threshold is never crossed. [6](#0-5) 

3. **Minimum deposit check bypassed**: `checkMinDeposit` always treats USDC as $1.00, so the `MIN_DEPOSIT_AMOUNT` ($0.10) and `MIN_FIRST_DEPOSIT_AMOUNT` ($5.00) thresholds are evaluated against an inflated price. [7](#0-6) 

The corrupted state delta is: **subaccount health is overstated by `(1 - depeg_price) * USDC_balance * weight`** for every subaccount holding USDC collateral, directly inflating borrowing capacity and suppressing liquidation triggers.

---

### Likelihood Explanation

USDC has historically depegged (March 2023: ~$0.87 low). The protocol is deployed on-chain with no admin circuit breaker or oracle integration for the quote price. Any depeg event — even a brief one — immediately creates exploitable conditions because the hardcoded price takes effect for every health check in real time. A sophisticated user monitoring the USDC price can front-run the depeg by maximizing their USDC-backed borrow position the moment the peg breaks, extracting value before any manual intervention is possible.

---

### Recommendation

1. Integrate an on-chain oracle (e.g., Chainlink USDC/USD) to fetch the live quote token price and pass it into `_risk().value[QUOTE_PRODUCT_ID].priceX18` via a permissioned `updatePrice` call on each sequencer tick, analogous to how other product prices are updated.
2. Remove the hardcoded `priceX18 = ONE` branch in `Clearinghouse.checkMinDeposit()` and route `QUOTE_PRODUCT_ID` through `_getPriceX18()` like all other products.
3. Add a circuit breaker that halts new borrows if the quote token price deviates more than a configurable threshold (e.g., 2%) from $1.00.

---

### Proof of Concept

1. USDC depegs to $0.90 on-chain.
2. Attacker holds 100,000 USDC deposited as `QUOTE_PRODUCT_ID` collateral.
3. `getHealth(attacker, INITIAL)` calls `_calculateProductHealth(QUOTE_PRODUCT_ID, ...)` which computes `100,000 * 1.0 * 1.0 = $100,000` health contribution — overvalued by $10,000.
4. Attacker borrows $100,000 worth of other assets against this collateral. True collateral value is $90,000, leaving a $10,000 undercollateralized position.
5. Because health still reads positive (USDC still valued at $1.00), `ClearinghouseLiq` cannot liquidate the position.
6. If USDC continues to depeg further, the protocol accumulates bad debt that must be socialized via `socializeSubaccount()`, diluting all depositors. [8](#0-7)
