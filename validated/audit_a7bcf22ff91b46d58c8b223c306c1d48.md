### Title
`ValidResourceBounds::AllResources` with zero `l2_gas`/`l1_data_gas` silently collapses to `L1Gas` variant across protobuf boundary, producing a divergent transaction hash preimage — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` applies a value-based heuristic to decide which enum variant to reconstruct. When a transaction carries `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`, the deserializer silently collapses it to `L1Gas(X)`. Because `get_tip_resource_bounds_hash` includes `L1_DATA_GAS` in the Poseidon preimage only for the `AllResources` variant, the two variants produce **different transaction hashes** for identical numeric values. A transaction accepted and hashed by the originating sequencer node therefore carries a different hash on every syncing peer that reconstructs it from protobuf.

---

### Finding Description

**Step 1 – Originating node (gateway ingestion)**

`RpcInvokeTransactionV3` always carries `resource_bounds: AllResourceBounds`. The gateway converts it to `InternalRpcInvokeTransactionV3` (still `AllResourceBounds`) and computes the hash via `calculate_transaction_hash` → `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash`. [1](#0-0) 

Inside `get_tip_resource_bounds_hash`, the `AllResources` branch appends a third element `concat(L1_DATA_GAS, 0)` to the Poseidon input:

```
H1 = poseidon(tip, concat(L1_GAS, X), concat(L2_GAS, 0), concat(L1_DATA_GAS, 0))
``` [2](#0-1) 

**Step 2 – Serialization to protobuf (block-sync path)**

`InvokeTransactionV3` (blockifier type) carries `resource_bounds: ValidResourceBounds::AllResources(...)`. The `From<ValidResourceBounds> for protobuf::ResourceBounds` serializer faithfully emits all three fields, including `l1_data_gas = 0`: [3](#0-2) 

**Step 3 – Deserialization on the syncing peer (the variant flip)**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies the heuristic:

```rust
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // ← variant collapses here
} else {
    ValidResourceBounds::AllResources(...)
})
``` [4](#0-3) 

Because `l2_gas = 0` and `l1_data_gas = 0`, the peer reconstructs `L1Gas(X)` instead of `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`.

**Step 4 – Hash recomputation on the peer produces H2 ≠ H1**

`get_tip_resource_bounds_hash` for `L1Gas(X)` omits the third element:

```
H2 = poseidon(tip, concat(L1_GAS, X), concat(L2_GAS, 0))
``` [5](#0-4) 

`H1 ≠ H2` because the Poseidon hash over a 3-element sequence differs from the hash over a 2-element sequence even when the extra element is zero.

**Step 5 – The gateway accepts the triggering transaction**

The stateless validator accepts `AllResourceBounds { l1_gas: NON_ZERO, l2_gas: 0, l1_data_gas: 0 }` as a valid transaction: [6](#0-5) 

The `InternalRpcInvokeTransactionV3` → `InvokeTransactionV3` conversion always wraps `AllResourceBounds` as `AllResources`: [7](#0-6) 

---

### Impact Explanation

Any syncing node that receives a block containing such a transaction via the protobuf block-sync path will reconstruct the transaction with the `L1Gas` variant and compute hash H2. The originating node stored the transaction under H1. Consequences:

- **Wrong transaction hash served by RPC**: `starknet_getTransactionByHash(H1)` fails or returns wrong data on syncing nodes (matches *High – RPC returns an authoritative-looking wrong value*).
- **Wrong hash/signature binding**: If the peer re-verifies the transaction hash (e.g., for block validation or signature verification), it uses H2 while the user signed over H1, causing verification failure (matches *High – transaction conversion binds the wrong hash*).
- **State divergence**: The originating sequencer and all syncing peers disagree on the canonical transaction hash for the same on-chain transaction.

---

### Likelihood Explanation

Any unprivileged user can trigger this by submitting a V3 invoke transaction with `l2_gas = 0` and `l1_data_gas = 0` (a common pattern for pre-0.13.3 style transactions). The gateway accepts it. The transaction is included in a block. Every syncing peer that processes the block via protobuf will silently compute the wrong hash. No special permissions or malicious peer are required.

---

### Recommendation

Remove the value-based heuristic from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Instead, add an explicit discriminant field to the protobuf `ResourceBounds` message (e.g., `bool has_all_resources`) so the variant is preserved across the wire boundary regardless of numeric values. Alternatively, always reconstruct `AllResources` when all three fields are present in the protobuf message (even if zero), and only fall back to `L1Gas` when `l1_data_gas` is absent (i.e., the `unwrap_or_default` path for legacy 0.13.2 messages).

The analogous fix in the RPC path already exists correctly: `AllResourceBounds::try_from(protobuf::ResourceBounds)` in `crates/apollo_protobuf/src/converters/rpc_transaction.rs` requires all three fields and never collapses to `L1Gas`: [8](#0-7) 

The block-sync path should apply the same strict reconstruction.

---

### Proof of Concept

1. Submit an invoke V3 transaction with `resource_bounds = AllResourceBounds { l1_gas: { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: { 0, 0 }, l1_data_gas: { 0, 0 } }`. The gateway accepts it.

2. The originating node computes:
   ```
   H1 = poseidon("invoke", version=3, sender, 
                 poseidon(tip, concat(L1_GAS,1000,1), concat(L2_GAS,0,0), concat(L1_DATA_GAS,0,0)),
                 ...)
   ```
   Transaction is included in block B with hash H1.

3. A syncing peer receives block B via protobuf. `TryFrom<protobuf::ResourceBounds>` sees `l1_data_gas = 0, l2_gas = 0` and reconstructs `ValidResourceBounds::L1Gas(...)`.

4. The peer computes:
   ```
   H2 = poseidon("invoke", version=3, sender,
                 poseidon(tip, concat(L1_GAS,1000,1), concat(L2_GAS,0,0)),
                 ...)
   ```
   H1 ≠ H2 (different Poseidon input length).

5. `starknet_getTransactionByHash(H1)` on the syncing peer returns wrong data or fails. Signature verification using H2 fails for a signature computed over H1.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
