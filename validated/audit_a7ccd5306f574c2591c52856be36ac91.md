### Title
Pause Blocks `verify_transaction_inclusion` While GC Remains Callable, Permanently Destroying SPV Proof Availability — (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` carry a plain `#[pause]` with no bypass role, so they are completely inaccessible to every caller when the contract is paused. `run_mainchain_gc`, however, carries `#[pause(except(roles(Role::UnrestrictedRunGC)))]`, meaning a holder of `UnrestrictedRunGC` can still invoke it while the contract is paused. GC permanently deletes block headers from `headers_pool`. If a block that a user needs to verify against is garbage-collected during a pause window, the SPV proof for that Bitcoin transaction becomes permanently unverifiable — even after the contract is unpaused — because the header is gone from storage with no recovery path.

### Finding Description

`BtcLightClient` exposes three relevant public functions with asymmetric pause behavior:

```
#[pause]                                          // no bypass role
pub fn verify_transaction_inclusion(...)

#[pause]                                          // no bypass role
pub fn verify_transaction_inclusion_v2(...)

#[pause(except(roles(Role::UnrestrictedRunGC)))]  // bypass exists
pub fn run_mainchain_gc(batch_size: u64)
``` [1](#0-0) [2](#0-1) [3](#0-2) 

`run_mainchain_gc` permanently removes the oldest mainchain block headers from both `mainchain_height_to_header` and `headers_pool` via `remove_block_header`:

```rust
fn remove_block_header(&mut self, header_block_hash: &H256) {
    self.mainchain_header_to_height.remove(header_block_hash);
    self.headers_pool.remove(header_block_hash);
}
``` [4](#0-3) 

Both verification functions require the target block to be present in `headers_pool`:

```rust
let header = self
    .headers_pool
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));
``` [5](#0-4) [6](#0-5) 

If the block is removed by GC, this lookup panics permanently — there is no mechanism to re-insert a GC'd block.

### Impact Explanation

Any downstream NEAR contract or protocol that uses `verify_transaction_inclusion_v2` to confirm a Bitcoin deposit before releasing NEAR-side assets is exposed. The Bitcoin transaction is irreversible. If the NEAR contract is paused while the user waits to submit their SPV proof, and GC runs during that window (via the `UnrestrictedRunGC` bypass), the block containing the user's transaction is permanently deleted from `headers_pool`. After the contract is unpaused, the call to `verify_transaction_inclusion_v2` panics with `"cannot find requested transaction block"` for that block hash forever. The user's Bitcoin funds are spent; the NEAR-side claim is permanently blocked.

**Impact: High** — permanent, irrecoverable loss of SPV proof availability for affected transactions.

### Likelihood Explanation

Requires: (1) the contract to be paused by a `PauseManager`, (2) an `UnrestrictedRunGC` account to call `run_mainchain_gc` during the pause (or for the chain to advance enough post-unpause that GC catches up past the target block), and (3) the target block to fall within the GC removal range. The `gc_threshold` default of 52,704 blocks (~1 year) provides a large window, but a prolonged pause combined with active GC makes this reachable. The `UnrestrictedRunGC` role is explicitly designed to allow GC during pauses, making step (2) an intended operational action, not an attack.

**Likelihood: Low** — requires a pause event coinciding with GC advancement past the target block.

### Recommendation

Add an `UnrestrictedVerify` bypass role to both verification functions, mirroring the pattern used for `submit_blocks` and `run_mainchain_gc`:

```rust
#[pause(except(roles(Role::UnrestrictedVerify)))]
pub fn verify_transaction_inclusion_v2(...) -> bool { ... }
```

Alternatively, document that `run_mainchain_gc` must not be called while the contract is paused if any pending SPV proofs exist, and enforce this operationally. The stronger fix is the bypass role, which eliminates the asymmetry at the code level.

### Proof of Concept

1. Contract is deployed with `gc_threshold = 52704`. Block B at height H is submitted and confirmed on Bitcoin.
2. `PauseManager` pauses the contract.
3. User calls `verify_transaction_inclusion_v2` for block B → reverts immediately due to `#[pause]` with no bypass.
4. `UnrestrictedRunGC` account calls `run_mainchain_gc(batch_size)` with `batch_size` large enough to remove block B (height H is now below `tip - gc_threshold`). This succeeds because `UnrestrictedRunGC` bypasses the pause check.
   - `remove_block_header(&block_B_hash)` executes: `headers_pool.remove(&block_B_hash)` — permanent deletion.
5. `PauseManager` unpauses the contract.
6. User calls `verify_transaction_inclusion_v2` for block B → `headers_pool.get(&block_B_hash)` returns `None` → `env::panic_str("cannot find requested transaction block")`.
7. No recovery path exists. The Bitcoin transaction is spent; the NEAR-side proof is permanently unverifiable. [7](#0-6) [8](#0-7)
