### Title
`ValidResourceBounds` Type Coercion in P2P Protobuf Deserialization Silently Changes Transaction Hash Domain - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` implementation in the P2P sync path silently coerces `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` into `L1Gas(X)` after a protobuf round-trip. Because `get_tip_resource_bounds_hash` produces structurally different outputs for these two variants (3-element vs 2-element poseidon hash), a syncing node that recomputes or serves the transaction will bind a different hash domain and different resource-bounds semantics than the proposing node.

### Finding Description

**Proposing node path (correct):**

A user submits an `InvokeV3` transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`. The gateway stateless validator accepts this (test case `valid_l1_gas` confirms zero l2/l1_data_gas is valid). [1](#0-0) 

The transaction converter calls `tx_without_hash.calculate_transaction_hash(&self.chain_id)`, which routes to `get_invoke_transaction_v3_hash`, which calls `get_tip_resource_bounds_hash` with `ValidResourceBounds::AllResources(...)`. [2](#0-1) 

`get_tip_resource_bounds_hash` for `AllResources` hashes **4 elements**: `[tip, concat(L1_GAS, X), concat(L2_GAS, 0), concat(L1_DATA_GAS, 0)]`. [3](#0-2) 

**P2P sync path (divergent):**

When the committed transaction is serialized to protobuf for P2P sync, `From<ValidResourceBounds> for protobuf::ResourceBounds` emits `l2_gas: Some(zero)` and `l1_data_gas: Some(zero)` for the `AllResources` arm. [4](#0-3) 

On the receiving node, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies the following gate:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // ← type silently changed
} else {
    ValidResourceBounds::AllResources(...)
})
``` [5](#0-4) 

The transaction that entered the network as `AllResources` is now stored as `L1Gas` on the syncing node.

**Hash domain divergence:**

`get_tip_resource_bounds_hash` for `L1Gas` hashes only **3 elements**: `[tip, concat(L1_GAS, X), concat(L2_GAS, 0)]` — the `L1_DATA_GAS` term is absent. [6](#0-5) 

Poseidon over 3 elements ≠ Poseidon over 4 elements even when the 4th element is zero. The transaction hash recomputed by the syncing node from its stored data is therefore different from the hash committed by the proposing node.

**Execution semantics divergence:**

`get_gas_vector_computation_mode()` returns `NoL2Gas` for `L1Gas` and `All` for `AllResources`. [7](#0-6) 

This changes which resource bounds are enforced during pre-validation and post-execution fee checks on the syncing node. [8](#0-7) 

### Impact Explanation

A syncing node that receives a committed `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` transaction over P2P stores it as `L1Gas(X)`. Consequences:

1. **Wrong hash binding**: Any path that recomputes the transaction hash from stored fields (e.g., hash validation, RPC `starknet_getTransactionByHash` with hash verification) produces a hash that does not match the committed block hash.
2. **Wrong RPC response**: The syncing node serves `resource_bounds` with the wrong type to clients, which is an authoritative-looking wrong value.
3. **Wrong fee estimation / simulation**: `starknet_estimateFee` and `starknet_simulateTransactions` on the syncing node use `NoL2Gas` computation mode instead of `All`, producing different gas vectors and fee estimates than the proposing node.

This matches: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value** and **High — Transaction conversion or signature/hash logic binds the wrong hash or type**.

### Likelihood Explanation

- The gateway's stateless validator explicitly accepts `AllResourceBounds` with zero l2_gas and l1_data_gas (confirmed by test `valid_l1_gas`).
- Any unprivileged user can submit such a transaction.
- The coercion is unconditional in the deserialization path — no version gate or flag prevents it.
- The TODO comment `// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.` confirms this is a known transitional gap, not an intentional permanent gate.

### Recommendation

In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, do not use the zero-value of `l2_gas`/`l1_data_gas` as a discriminant for the `ValidResourceBounds` variant. Instead, either:

1. Transmit the variant tag explicitly in the protobuf message (add a `resource_bounds_type` enum field), or
2. Require that the sender always transmits `l1_data_gas` as `Some(...)` for post-0.13.2 transactions (enforce the TODO), and treat `None` as the only signal for `L1Gas`.

The `From<ValidResourceBounds> for protobuf::ResourceBounds` serializer already emits `l1_data_gas: Some(zero)` for `L1Gas`, so the zero-value test is ambiguous by construction and cannot distinguish the two variants.

### Proof of Concept

1. Submit an `InvokeV3` transaction with `resource_bounds = AllResourceBounds { l1_gas: { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: { 0, 0 }, l1_data_gas: { 0, 0 } }` to the gateway. The gateway accepts it.

2. The proposing node computes `tx_hash_A = poseidon(INVOKE, version, sender, poseidon(tip, pack(L1_GAS,1000,1), pack(L2_GAS,0,0), pack(L1_DATA_GAS,0,0)), ...)` — 4-element resource hash.

3. The transaction is committed to a block. A syncing node fetches the block over P2P.

4. The syncing node deserializes the protobuf `ResourceBounds { l1_gas: {1000,1}, l2_gas: {0,0}, l1_data_gas: {0,0} }`. Because `l1_data_gas.is_zero() && l2_gas.is_zero()`, it produces `ValidResourceBounds::L1Gas({1000,1})`.

5. The syncing node recomputes `tx_hash_B = poseidon(INVOKE, version, sender, poseidon(tip, pack(L1_GAS,1000,1), pack(L2_GAS,0,0)), ...)` — 3-element resource hash.

6. `tx_hash_A ≠ tx_hash_B`. The syncing node's RPC returns wrong resource bounds and wrong fee estimates for this transaction. Any hash-based lookup or validation on the syncing node is broken for this transaction.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
