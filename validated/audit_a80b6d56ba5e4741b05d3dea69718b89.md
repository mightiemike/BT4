### Title
`GetExecutionOutcome` Derives `outcome_root_proof` Shard Index from Caller-Supplied `receiver_id` Instead of Actual Execution Shard, Silently Breaking Light-Client Proofs for `GlobalContractDistributionReceipt` — (`chain/client/src/view_client_actor.rs`)

---

### Summary

The `GetExecutionOutcome` handler in `ViewClientActor` selects which shard's merkle path to return as `outcome_root_proof` by calling `account_id_to_shard_id` on the caller-supplied `receiver_id`. For `GlobalContractDistributionReceipt` receipts, the actual execution shard is determined by an embedded `target_shard` field — not by `account_id_to_shard_id(receiver_id)`. When these two values disagree, the handler silently returns a merkle path for the wrong shard. The returned proof fails verification at the light client, yet the RPC returns no error. The `ShardLayout::account_id_to_shard_id` docstring explicitly flags this misuse: *"Calling function with receipt receiver_id to determine target shard is incorrect, `Receipt::receiver_shard` should be used instead."*

---

### Finding Description

`TransactionOrReceiptId` is a public, unprivileged-user-controlled enum: [1](#0-0) 

The `GetExecutionOutcome` handler extracts the caller-supplied `receiver_id` and immediately maps it to a shard via `account_id_to_shard_id`: [2](#0-1) 

That `target_shard_index` is then used to select the merkle path from the block's per-shard outcome-root array: [3](#0-2) 

The invariant the light-client protocol requires is:

```
shard_outcome_root = compute_root(sha256(borsh(execution_outcome)), outcome_proof.proof)
block_outcome_root = compute_root(sha256(borsh(shard_outcome_root)), outcome_root_proof)
block_outcome_root == block_header_lite.inner_lite.outcome_root
```

`outcome_root_proof` must be the merkle path for the shard that **actually executed** the receipt. The handler instead uses the shard derived from the caller-supplied `receiver_id`.

For `GlobalContractDistributionReceipt`, the actual execution shard is `target_shard` (embedded in the receipt), resolved via `Receipt::receiver_shard_id`: [4](#0-3) 

`account_id_to_shard_id` is explicitly documented as wrong for this use: [5](#0-4) 

When `account_id_to_shard_id(receiver_id) ≠ target_shard`, the handler:
1. Calls `get_next_block_hash_with_new_chunk` for the wrong shard, potentially selecting the wrong block.
2. Returns `merklize(&outcome_roots).1[wrong_index]` — a merkle path for a shard that did not execute the receipt.

No error is surfaced. The caller receives a structurally well-formed `RpcLightClientExecutionProofResponse` whose `outcome_root_proof` is cryptographically invalid.

---

### Impact Explanation

Any bridge or light-client relying on `EXPERIMENTAL_light_client_proof` (or the internal `GetExecutionOutcome` message) to prove a `GlobalContractDistributionReceipt` receives a silently broken proof. Verification fails at the verifier:

```
block_outcome_root ≠ block_header_lite.inner_lite.outcome_root
```

The proof cannot be fixed by retrying with the same inputs; the node will always return the wrong shard's path. This breaks the proof-of-inclusion guarantee for this receipt class, which is the exact invariant the light-client protocol is designed to provide.

---

### Likelihood Explanation

The divergence is triggered whenever:
- A `GlobalContractDistributionReceipt` is queried via `EXPERIMENTAL_light_client_proof`, **and**
- `account_id_to_shard_id(receiver_id)` maps to a different shard than `target_shard` embedded in the receipt.

In a multi-shard deployment with V2/V3 shard layouts (boundary-account partitioning), the `receiver_id` of a `GlobalContractDistributionReceipt` and its `target_shard` can easily disagree. The condition is reachable without any privileged role.

---

### Recommendation

In the `GetExecutionOutcome` handler, after locating the outcome, fetch the actual receipt from the store and call `Receipt::receiver_shard_id` to determine the correct shard:

```rust
// Instead of:
let target_shard_id =
    account_id_to_shard_id(self.epoch_manager.as_ref(), &account_id, &epoch_id)?;

// Do:
let target_shard_id = if let Some(receipt) = self.chain.chain_store().get_receipt(&id) {
    receipt.receiver_shard_id(&shard_layout)?
} else {
    // fallback for transactions: sender_id is correct
    account_id_to_shard_id(self.epoch_manager.as_ref(), &account_id, &epoch_id)?
};
```

This mirrors the correct pattern already used in `Chain::group_receipts_by_shard` and `Chain::build_receipts_hashes`: [6](#0-5) 

---

### Proof of Concept

1. On a multi-shard network (V2/V3 layout), deploy a global contract. The runtime emits a `GlobalContractDistributionReceipt` with `target_shard = S` (determined by the distribution logic, not by `receiver_id`).

2. Let `R` be the `receiver_id` stored in the receipt. Compute `shard_R = account_id_to_shard_id(R)`. If `shard_R ≠ S` (which is possible whenever the boundary-account partition places `R` in a different range than `S`), the divergence is active.

3. Query:
   ```json
   { "type": "receipt", "receipt_id": "<id>", "receiver_id": "<R>", "light_client_head": "<head>" }
   ```

4. The handler computes `target_shard_index = index_of(shard_R)` and returns `outcome_roots[index_of(shard_R)]` as `outcome_root_proof`.

5. At the verifier:
   ```python
   shard_outcome_root = compute_root(sha256(borsh(execution_outcome)), outcome_proof.proof)
   # outcome_proof.proof is correct (from the actual shard S's outcome tree)
   block_outcome_root = compute_root(sha256(borsh(shard_outcome_root)), outcome_root_proof)
   # outcome_root_proof is for shard_R, not shard S → wrong path
   assert block_outcome_root == block_header_lite.inner_lite.outcome_root  # FAILS
   ```

The exact divergent value is `outcome_roots[index_of(shard_R)]` where the correct value is `outcome_roots[index_of(S)]`. [7](#0-6) [4](#0-3) [5](#0-4)

    pub fn group_receipts_by_shard(
        receipts: Vec<Receipt>,
        shard_layout: &ShardLayout,
    ) -> Result<HashMap<ShardId, Vec<Receipt>>, EpochError> {
        let mut result = HashMap::new();
        for receipt in receipts {
            let shard_id = receipt.receiver_shard_id(shard_layout)?;
            let entry = result.entry(shard_id).or_insert_with(Vec::new);
