### Title
Hardcoded OOG Detection Threshold in `_executeSlowModeTransaction` Silently Drops Slow Mode Transactions on Non-Standard Gas Accounting Chains — (File: `core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint._executeSlowModeTransaction` uses two hardcoded constants — an absolute `250000` gas floor and a relative `gasRemaining / 2` heuristic — to distinguish an out-of-gas (OOG) error from a normal revert inside a `try/catch`. Both constants are calibrated for Arbitrum's 63/64 gas-forwarding rule. On a chain where gas accounting does not follow that rule (exactly the class of chain described in the external report), an OOG in the inner call can satisfy neither condition, causing the catch block to silently swallow the failure. Because the slow mode transaction is deleted from the queue **before** the try block executes, the transaction is permanently lost with no recourse for the user.

---

### Finding Description

In `_executeSlowModeTransaction`, the queue entry is deleted unconditionally before the inner call is attempted:

```solidity
delete slowModeTxs[_slowModeConfig.txUpTo++];   // deleted before execution
``` [1](#0-0) 

The inner call is then wrapped in a try/catch. On failure, the code checks two conditions to decide whether the failure was an OOG:

```solidity
if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
    assembly { invalid() }
}
``` [2](#0-1) 

The comment in the code explicitly acknowledges this is a heuristic derived from the 63/64 EVM gas-forwarding rule:

> *"the amount of gas left for execution is (63/64)\*\*len(stack) … having gasleft() <= gasRemaining / 2 buys us 44 nested calls before we miss out of gas errors; 1/2 ~= (63/64)\*\*44"* [3](#0-2) 

**The root cause:** Both thresholds assume the 63/64 forwarding rule holds. The external report demonstrates that on Etherlink, `gasleft()` returned `999,978,817` against a gas limit of `999,999,999` — meaning the chain does not enforce gas limits in the expected way. On such a chain:

- The inner `this.processSlowModeTransaction(...)` call can OOG.
- Because gas is not properly metered, the outer frame retains far more than `gasRemaining / 2` after the inner OOG.
- `gasleft()` also exceeds `250000`.
- Neither condition fires; `invalid()` is never called.
- The catch block silently swallows the OOG.
- The slow mode transaction has already been deleted from `slowModeTxs` and is permanently gone.

The `executeSlowModeTransaction()` entry point is permissionless — any caller can invoke it:

```solidity
function executeSlowModeTransaction() external {
``` [4](#0-3) 

Slow mode transactions include `WithdrawCollateral`, `DepositCollateral`, and `LinkSigner`, all of which carry direct asset or access-control consequences: [5](#0-4) 

---

### Impact Explanation

A slow mode transaction that OOGs in the inner call on a non-standard gas accounting chain is permanently removed from the queue without execution. For a `WithdrawCollateral` slow mode transaction, the user's withdrawal is silently dropped — their collateral remains locked in the protocol with no way to re-queue the same transaction (the nonce has not been consumed, but the queue entry is gone). For a `DepositCollateral` slow mode transaction queued via `depositCollateral`, the ERC-20 transfer has already occurred but the credit is never applied. This is a direct, concrete asset loss for the affected user.

---

### Likelihood Explanation

The external report proves that chains with non-standard gas accounting exist in production (Etherlink). The `250000` and `gasRemaining / 2` constants are explicitly documented as Arbitrum-specific heuristics. Any deployment of Nado on a chain where `gasleft()` does not reflect the 63/64 forwarding rule — exactly the class of chain the external report describes — activates this bug. An attacker can trigger it by calling `executeSlowModeTransaction()` with a gas limit crafted to cause the inner call to OOG, then observing that the outer frame retains more gas than the heuristic expects.

---

### Recommendation

Replace the dual-constant heuristic with a gas stipend approach: before the try block, verify that sufficient gas is available to cover the worst-case execution of `processSlowModeTransaction`, and revert (rather than silently swallow) if the gas budget is insufficient. Alternatively, do not delete the queue entry before the try block; instead, only delete it on success, and on failure propagate the OOG unconditionally (e.g., by re-checking `gasleft()` against a configurable, chain-specific minimum rather than a hardcoded constant).

---

### Proof of Concept

1. Deploy Nado on a chain where gas accounting does not enforce the 63/64 forwarding rule (e.g., Etherlink as described in the external report).
2. User submits a slow mode `WithdrawCollateral` transaction via `submitSlowModeTransaction`. The ERC-20 fee is paid; the entry is queued.
3. Attacker calls `executeSlowModeTransaction()` with a gas limit chosen so that `processSlowModeTransaction` runs out of gas internally, but the outer frame retains `gasleft() > 250000` and `gasleft() > gasRemaining / 2` (possible because the chain does not enforce the 63/64 rule).
4. The catch block fires. Neither OOG condition is satisfied. `invalid()` is not called.
5. The slow mode transaction was already deleted at step `delete slowModeTxs[_slowModeConfig.txUpTo++]`.
6. The user's withdrawal is permanently lost; their collateral remains locked with no queue entry to re-execute.
