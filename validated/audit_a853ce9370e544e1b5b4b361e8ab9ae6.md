### Title
Missing `extcodesize` / zero-address check before `delegatecall` in `liquidateSubaccount` silently swallows liquidations — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.liquidateSubaccount()` delegates all liquidation logic to the `clearinghouseLiq` address stored in contract state. Neither the call site nor the setter (`upgradeClearinghouseLiq`) validates that the target address contains deployed code. If `clearinghouseLiq` is `address(0)` or a non-existent contract, the EVM returns `(success=true, returndata="")` for the `delegatecall`, the `require(success, ...)` guard passes, and every liquidation silently does nothing.

---

### Finding Description

`Clearinghouse.liquidateSubaccount()` constructs a call to `clearinghouseLiq` via `delegatecall`:

```solidity
// core/contracts/Clearinghouse.sol L658-661
(bool success, bytes memory result) = clearinghouseLiq.delegatecall(
    liquidateSubaccountCall
);
require(success, string(result));
```

`clearinghouseLiq` is a plain storage variable declared in `ClearinghouseStorage`:

```solidity
// core/contracts/ClearinghouseStorage.sol L14
address internal clearinghouseLiq;
```

It is written in two places, neither of which validates the supplied address:

1. **`Clearinghouse.initialize()`** (line 36) — accepts `_clearinghouseLiq` directly from the deployer with no check.
2. **`Clearinghouse.upgradeClearinghouseLiq()`** (lines 677–683) — accepts any address from the `ProxyManagerHelper` with no zero-address or `extcodesize` guard.

The EVM's `delegatecall` opcode to `address(0)` (or any address with no code) returns `(1, "")` — i.e., `success = true` with empty return data. The only guard in `liquidateSubaccount` is `require(success, ...)`, which passes unconditionally in this scenario. The function returns normally, and the caller (the sequencer pipeline) observes no error.

---

### Impact Explanation

Liquidation is the primary solvency mechanism of the Nado protocol. If `clearinghouseLiq` is ever `address(0)` or a non-existent contract:

- Every `LiquidateSubaccount` transaction submitted by the sequencer silently succeeds without transferring any collateral, closing any position, or updating any balance.
- Underwater subaccounts are never actually liquidated; their losses accumulate against the insurance fund and ultimately against solvent counterparties.
- The failure is completely invisible on-chain: the transaction does not revert, emits no error, and the sequencer has no signal that liquidation logic was skipped.
- Protocol insolvency can compound undetected until the misconfiguration is discovered and corrected.

---

### Likelihood Explanation

The scenario requires `clearinghouseLiq` to be set to a codeless address. This can occur:

- During a migration where `upgradeClearinghouseLiq` is called with a new implementation address before that implementation is deployed (a common deployment ordering mistake).
- If `initialize()` is called with `address(0)` for `_clearinghouseLiq` due to a deployment script error.
- If a `ClearinghouseLiq` implementation contract is self-destructed and the pointer is not updated.

Each scenario is a realistic operational mistake, not a theoretical one. The external report's analogous scenario (RocketStorage returning `address(0)` for an unset key) was assessed at the same likelihood tier. The absence of any defensive check means the failure mode is entirely silent, making it harder to detect and recover from.

---

### Recommendation

Add a zero-address and `extcodesize` guard at the `delegatecall` call site in `liquidateSubaccount`, and enforce the same invariant inside `upgradeClearinghouseLiq`:

```solidity
// In liquidateSubaccount, before the delegatecall:
require(clearinghouseLiq != address(0), "clearinghouseLiq not set");
uint256 codeSize;
assembly { codeSize := extcodesize(clearinghouseLiq) }
require(codeSize > 0, "clearinghouseLiq has no code");

// In upgradeClearinghouseLiq:
require(_clearinghouseLiq != address(0), "zero address");
uint256 codeSize;
assembly { codeSize := extcodesize(_clearinghouseLiq) }
require(codeSize > 0, "no code at address");
clearinghouseLiq = _clearinghouseLiq;
```

---

### Proof of Concept

1. Deploy `Clearinghouse` with `_clearinghouseLiq = address(0)` (or call `upgradeClearinghouseLiq(address(0))` via the `ProxyManagerHelper`).
2. Submit a `LiquidateSubaccount` transaction through the sequencer pipeline.
3. The sequencer calls `Endpoint.submitTransactionsChecked()` → `processTransaction()` → `_delegatecallEndpointTx(EndpointTx.processTransactionImpl)` → `Clearinghouse.liquidateSubaccount(txn)`.
4. Inside `liquidateSubaccount`, `clearinghouseLiq.delegatecall(liquidateSubaccountCall)` is executed against `address(0)`.
5. The EVM returns `(true, "")`. `require(true, ...)` passes.
6. The function returns. No position is closed, no collateral is transferred, no balance is updated.
7. The sequencer observes a successful transaction. The underwater subaccount remains open. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
