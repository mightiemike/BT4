### Title
Unbounded Loops in `reorg_chain` Can Cause Gas Exhaustion, Permanently Blocking Chain Reorganizations - (File: `contract/src/lib.rs`)

### Summary
The `reorg_chain` function in `BtcLightClient` contains two unbounded loops whose iteration count is proportional to the fork length. There is no on-chain cap on this depth. When a sufficiently long fork is submitted via `submit_blocks`, the transaction exhausts NEAR's gas limit, the reorg fails, and the contract is permanently unable to promote the heavier fork to the main chain — corrupting the canonical chain tracking invariant.

### Finding Description

`submit_block_header_inner` automatically calls `reorg_chain` whenever a submitted fork header has more accumulated `chain_work` than the current main chain tip: [1](#0-0) 

`reorg_chain` then executes two sequential unbounded loops:

**Loop 1** — removes main chain blocks above the fork tip height: [2](#0-1) 

The iteration count equals `last_main_chain_block_height − fork_tip_height`. Each iteration performs two `LookupMap` storage operations (`get` + `remove`).

**Loop 2** — walks the fork chain back to the common ancestor, promoting each fork block to the main chain: [3](#0-2) 

The iteration count equals the number of fork blocks not yet on the main chain. Each iteration performs up to five `LookupMap` operations (`contains_key`, `insert` ×2, `remove` ×2) plus a `headers_pool.get`.

Neither loop has any on-chain iteration cap. The only limit that exists is `max_fork_len = 500` in the relayer's off-chain configuration: [4](#0-3) 

This limit is **not enforced by the contract**. Any caller who can invoke `submit_blocks` — including any registered trusted relayer — can submit a chain of fork headers of arbitrary depth, triggering both loops for the full fork length in a single transaction.

### Impact Explanation

**High.** NEAR Protocol enforces a hard per-transaction gas limit of 300 Tgas. A reorg involving hundreds of blocks, each requiring multiple `LookupMap` storage reads and writes, will exceed this limit. When the transaction runs out of gas it reverts entirely, leaving the contract state unchanged. The fork tip header was already stored in `headers_pool` (via `store_fork_header` before `reorg_chain` is called): [5](#0-4) 

After the failed reorg, the fork header remains in `headers_pool` but the main chain tip pointer (`mainchain_tip_blockhash`) still points to the old, lower-work chain. Subsequent submissions of the same fork tip will again trigger `reorg_chain` and again fail. The contract is permanently stuck tracking a stale chain that has less accumulated proof-of-work than the true Bitcoin chain — a direct violation of the core invariant the light client exists to enforce.

### Likelihood Explanation

**Low.** Exploiting this in production requires either (a) a genuine multi-hundred-block Bitcoin reorganization (extremely rare but not impossible — the relayer config explicitly budgets for up to 500-block forks), or (b) a trusted relayer submitting a crafted sequence of valid PoW headers forming a long fork. Creating valid PoW headers requires real mining work, making deliberate exploitation expensive. However, the scenario where a legitimate long fork occurs and the honest relayer's reorg attempt fails is a realistic operational risk, not a purely theoretical one.

### Recommendation

Add an on-chain maximum reorg depth parameter (e.g., `max_reorg_depth: u64`) stored in contract state. At the start of `reorg_chain`, compute the fork length and `panic` if it exceeds the limit, rather than attempting an unbounded traversal. Alternatively, split the reorg into a multi-step process: cache the pending reorg state and allow it to be completed across multiple transactions, similar to how `run_mainchain_gc` accepts a `batch_size` parameter to bound its work per call. [6](#0-5) 

### Proof of Concept

1. Deploy the contract with `skip_pow_verification = true` (or use a chain where PoW is cheap, e.g., testnet).
2. Submit N main chain blocks (e.g., N = 300) to establish a long main chain.
3. Submit N+1 fork blocks branching from the genesis, each with slightly higher `bits` (more work per block), so the fork tip's `chain_work` exceeds the main chain tip's `chain_work`.
4. On submission of the fork tip, `submit_block_header_inner` detects `current_header.chain_work > total_main_chain_chainwork` and calls `reorg_chain`.
5. Loop 1 iterates 0 times (fork is longer). Loop 2 iterates N+1 times, performing ~5 storage ops each = ~1500 storage operations.
6. At NEAR's storage operation gas cost, this exhausts the 300 Tgas limit well before completion.
7. The transaction reverts. `mainchain_tip_blockhash` still points to the old N-block chain. The fork tip (with more work) is stranded in `headers_pool` and can never be promoted. [7](#0-6)

    /// The most expensive operation which reorganizes the chain, based on fork weight
    fn reorg_chain(&mut self, fork_tip_header: ExtendedHeader, last_main_chain_block_height: u64) {
        let fork_tip_height = fork_tip_header.block_height;
        if last_main_chain_block_height > fork_tip_height {
            // If we see that main chain is longer than fork we first garbage collect
            // outstanding main chain blocks:
            //
            //      [m1] - [m2] - [m3] - [m4] <- We should remove [m4]
            //     /
            // [m0]
            //     \
            //      [f1] - [f2] - [f3]
            for height in (fork_tip_height + 1)..=last_main_chain_block_height {
                let current_main_chain_blockhash = self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str("cannot get a block"));
                self.remove_block_header(&current_main_chain_blockhash);
                self.mainchain_height_to_header.remove(&height);
            }
        }

        // Now we are in a situation where mainchain is equivalent to fork size:
        //
        //      [m1] - [m2] - [m3] - [m4] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip
        //
        //
        // Or in a situation where it is shorter:
        //
        //      [m1] - [m2] - [m3] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip

        let fork_tip_hash = fork_tip_header.block_hash.clone();
        let mut fork_header_cursor = fork_tip_header;

        while !self
            .mainchain_header_to_height
            .contains_key(&fork_header_cursor.block_hash)
        {
            let prev_block_hash = fork_header_cursor.block_header.prev_block_hash;
            let current_block_hash = fork_header_cursor.block_hash;
            let current_height = fork_header_cursor.block_height;

            // Inserting the fork block into the main chain, if some mainchain block is occupying
            // this height let's save its hashcode
            let main_chain_block = self
                .mainchain_height_to_header
                .insert(&current_height, &current_block_hash);
            self.mainchain_header_to_height
                .insert(&current_block_hash, &current_height);

            // If we found a mainchain block at the current height than remove this block from the
            // header pool and from the header -> height map
            if let Some(current_main_chain_blockhash) = main_chain_block {
                self.remove_block_header(&current_main_chain_blockhash);
            }

            // Switch iterator cursor to the previous block in fork
            fork_header_cursor = self
                .headers_pool
                .get(&prev_block_hash)
                .unwrap_or_else(|| env::panic_str("previous fork block should be there"));
        }

        // Updating tip of the new main chain
        self.mainchain_tip_blockhash = fork_tip_hash;
    }
