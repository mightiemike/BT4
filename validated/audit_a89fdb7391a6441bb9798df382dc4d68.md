### Title
`calculate_reward` Returns `Balance::ZERO` as `minted_amount` While Treasury Reward Is Non-Zero When No Validators Exist — (`File: chain/epoch-manager/src/reward_calculator.rs`)

### Summary

`RewardCalculator::calculate_reward` inserts a non-zero `epoch_protocol_treasury` reward into the result map before the `num_validators == 0` early-return, but returns `Balance::ZERO` as the minted amount. The minted amount is stored in `EpochInfo::minted_amount` and used by every node to verify and compute `total_supply` at epoch boundaries. When no validators are present, the treasury receives tokens that are not counted in `minted_amount`, permanently understating the protocol's `total_supply`.

### Finding Description

In `calculate_reward`:

```rust
// line 84
res.insert(self.protocol_treasury_account.clone(), epoch_protocol_treasury);
// line 85-87
if num_validators == 0 {
    return (res, Balance::ZERO);   // ← minted_amount is ZERO, but treasury reward is in res
}
``` [1](#0-0) 

The function's own docstring states the second return value is "amount of newly minted tokens **including to protocol's treasury**." [2](#0-1) 

When `num_validators == 0`, `epoch_protocol_treasury` (computed from `total_supply * max_inflation_rate * protocol_reward_rate`) is placed in `res` — meaning the treasury account will be credited — but `Balance::ZERO` is returned as the minted amount instead of `epoch_protocol_treasury`.

The call site in `finalize_epoch` captures both values:

```rust
let (validator_reward, minted_amount) = {
    ...
    self.reward_calculator.calculate_reward(validator_block_chunk_stats, ...)
};
``` [3](#0-2) 

`minted_amount` is then passed directly to `proposals_to_epoch_info` and stored in `EpochInfo::minted_amount`: [4](#0-3) 

Every node reads `EpochInfo::minted_amount()` at the epoch boundary to verify `total_supply`:

```rust
let minted_amount = if self.epoch_manager.is_next_block_epoch_start(&prev_hash)? {
    Some(self.epoch_manager.get_epoch_info(header.next_epoch_id())?.minted_amount())
} else { None };
block.verify_total_supply_checked(prev.total_supply(), minted_amount)
``` [5](#0-4) 

`verify_total_supply_checked` computes:

```rust
let Some(new_total_supply) = prev_total_supply
    .checked_add(minted_amount.unwrap_or(Balance::ZERO))?
    .checked_sub(balance_burnt)
``` [6](#0-5) 

With `minted_amount = 0` but treasury credited `epoch_protocol_treasury`, the on-chain `total_supply` is understated by exactly `epoch_protocol_treasury` for that epoch, and the error compounds into every subsequent epoch's inflation calculation.

The `num_validators == 0` condition is reached in `finalize_epoch` when all validators in `validator_block_chunk_stats` are removed because they were kicked out for `NotEnoughBlocks`, `NotEnoughChunks`, or `NotEnoughChunkEndorsements`: [7](#0-6) 

The `all_kicked_out` guard in `compute_validators_to_reward_and_kickout` saves one validator only when `max_validator` is `Some`; if all validators were already in `prev_validator_kickout`, `max_validator` stays `None` and no validator is saved, leaving `validator_block_chunk_stats` empty after the removal loop. [8](#0-7) 

### Impact Explanation

`EpochInfo::minted_amount` is Borsh-serialized and stored on-chain. All nodes read the same wrong value (`0` instead of `epoch_protocol_treasury`). Every node computes the same wrong `total_supply` for the epoch boundary block, so consensus is not broken, but the protocol's global token supply invariant is permanently violated: the treasury account holds tokens that are not reflected in `total_supply`. Every subsequent epoch's inflation reward is computed from this understated `total_supply`, compounding the error. The divergent value is exact: `minted_amount = Balance::ZERO` vs. the correct `epoch_protocol_treasury`.

### Likelihood Explanation

Requires an epoch where all validators are removed from `validator_block_chunk_stats` after the kickout filter. The `all_kicked_out` guard normally prevents this, but fails when `max_validator` is `None` (all validators already in `prev_validator_kickout`). This is an edge case but the code path is reachable without any privileged action — it is triggered purely by epoch finalization logic when validator performance and prior kickout state align.

### Recommendation

Change the early return to include the treasury reward in the minted amount:

```rust
if num_validators == 0 {
    return (res, epoch_protocol_treasury);
}
```

This matches the behavior of the normal path, where `epoch_actual_reward` is initialized to `epoch_protocol_treasury` at line 90 and returned at line 145. [9](#0-8) 

### Proof of Concept

1. Configure an epoch where all validators are already in `prev_validator_kickout` (e.g., all slashed in the prior epoch) so `max_validator` stays `None`.
2. Trigger epoch finalization: `finalize_epoch` calls `calculate_reward` with an empty `validator_block_chunk_stats`.
3. `num_validators == 0` → function returns `(res={treasury: epoch_protocol_treasury}, Balance::ZERO)`.
4. `EpochInfo::minted_amount` is stored as `0`.
5. The epoch boundary block is produced with `total_supply = prev_total_supply + 0 - balance_burnt`.
6. All nodes accept this block (they all read `minted_amount = 0` from `EpochInfo`).
7. The treasury account balance increases by `epoch_protocol_treasury`, but `total_supply` does not reflect this mint — the invariant `total_supply == sum(all account balances)` is broken permanently. [9](#0-8) [3](#0-2) [10](#0-9)

```rust
    pub fn verify_total_supply_checked(
        &self,
        prev_total_supply: Balance,
        minted_amount: Option<Balance>,
    ) -> Option<bool> {
        let mut balance_burnt = Balance::ZERO;

        for chunk in self.chunks().iter_new() {
            balance_burnt = balance_burnt.checked_add(chunk.prev_balance_burnt())?;
        }

        let Some(new_total_supply) = prev_total_supply
            .checked_add(minted_amount.unwrap_or(Balance::ZERO))?
            .checked_sub(balance_burnt)
        else {
            // This corresponds to balance_burnt > prev_total_supply + minted_amount
            // which indicates invalid balance burnt, not arithmetic overflow
            return Some(false);
        };
        Some(self.header().total_supply() == new_total_supply)
    }
