### Title
`ENABLE_SECP_OPS` Flag Never Checked Before Dispatching secp Operators — Unconditional Activation Regardless of Flag — (`File: src/chia_dialect.rs`)

---

### Summary

The `ClvmFlags::ENABLE_SECP_OPS` flag is defined, documented, and exposed to Python callers as the gate that enables `secp256k1_verify` and `secp256r1_verify`. However, the `ChiaDialect::op()` dispatch function never checks this flag before routing 4-byte opcodes `0x13d61f00` and `0x1c3a8f00` to their implementations. The secp operators are therefore unconditionally active regardless of the flag value, making the flag a dead wire. This is a direct analog to the external report's "flag/operator wiring error" class: a boolean control is always set to one effective value without checking the condition that determines which value is correct.

---

### Finding Description

`ClvmFlags::ENABLE_SECP_OPS` is declared at line 63 of `src/chia_dialect.rs` with the comment *"Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify)"* and is exported to Python callers at line 321 of `wheel/src/api.rs`. [1](#0-0) 

Inside `ChiaDialect::op()`, the 4-byte opcode branch dispatches secp operators with no flag guard: [2](#0-1) 

The match arms for `0x13d61f00` (`op_secp256k1_verify`) and `0x1c3a8f00` (`op_secp256r1_verify`) are reached unconditionally. There is no `flags.contains(ClvmFlags::ENABLE_SECP_OPS)` guard before the match or on the individual arms. A caller that constructs `ChiaDialect::new(ClvmFlags::empty())` or any flag set that omits `ENABLE_SECP_OPS` will still execute the full secp verification logic.

The correct behavior — falling through to `unknown_operator` when the flag is absent — never occurs for these two opcodes. [3](#0-2) 

Compare with `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`, which is correctly injected into the effective `flags` variable only when the matching `OperatorSet::Keccak` extension is active, and is then checked inside `op_keccak256` before execution: [4](#0-3) 

The secp path has no equivalent guard.

---

### Impact Explanation

**Consensus divergence.** A node running with `ChiaDialect::new(ClvmFlags::empty())` (pre-hardfork consensus mode, no `ENABLE_SECP_OPS`) will execute secp signature verification and return a result. A node that correctly implements the flag would treat opcodes `0x13d61f00`/`0x1c3a8f00` as unknown operators: in consensus mode they return `nil` with a formula-derived cost; in mempool mode (`NO_UNKNOWN_OPS`) they raise `EvalErr::Unimplemented`. Because the flag is never checked, both node types instead run the real secp implementation, producing a different result and cost than the unknown-operator path would. Any CLVM coin that uses a secp opcode before the hardfork activates will be evaluated differently by nodes that believe the flag controls activation versus the actual code.

**Python API misuse.** `ENABLE_SECP_OPS` is exported to Python callers via `wheel/src/api.rs` line 321. A Python caller that deliberately omits the flag to simulate pre-activation consensus will silently get post-activation behavior, producing wrong cost accounting and wrong accept/reject decisions. [5](#0-4) 

---

### Likelihood Explanation

The entry path is fully attacker-controlled: any CLVM byte sequence whose operator atom is the 4-byte encoding `0x13d61f00` or `0x1c3a8f00` triggers the bug. The `run_serialized_chia_program` Python entry point accepts arbitrary serialized bytes, and the Rust `run_program` function accepts any `NodePtr` tree. No special privileges are required. The bug fires on every invocation of those opcodes when the caller omits `ENABLE_SECP_OPS`, which is the expected state before the hardfork activates.

---

### Recommendation

Add a flag guard in the 4-byte opcode branch before dispatching secp operators:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This mirrors the pattern used for `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` and `ENABLE_SHA256_TREE`, where the flag is checked before the operator is considered active.

---

### Proof of Concept

1. Construct a CLVM program whose operator atom is `\x13\xd6\x1f\x00` (secp256k1_verify opcode) with valid arguments.
2. Call `run_program` with `ChiaDialect::new(ClvmFlags::empty())` — no `ENABLE_SECP_OPS`.
3. **Observed:** the program executes `op_secp256k1_verify` and returns a secp verification result with the secp cost (1,850,000).
4. **Expected:** the opcode falls through to `unknown_operator`; in consensus mode it returns `nil` with a formula-derived cost; in mempool mode it raises `EvalErr::Unimplemented`.

The divergence between observed and expected behavior is the consensus split. Any coin spending puzzle that embeds a secp opcode will be accepted/rejected differently depending on whether the evaluating node's code checks the flag (correct) or not (this bug).
