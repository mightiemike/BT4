### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the 64-Byte Merkle Proof Forgery Protection — (File: `contract/src/lib.rs`)

---

### Summary

The contract introduced `verify_transaction_inclusion_v2` to mitigate the well-known 64-byte transaction Merkle proof forgery vulnerability. However, the original `verify_transaction_inclusion` function remains a live, publicly callable `pub` entry point. Any unprivileged NEAR caller can invoke it directly, completely bypassing the coinbase-proof guard that `v2` enforces. The `#[deprecated]` Rust attribute is a compile-time hint only; it imposes no runtime restriction on NEAR RPC calls.

---

### Finding Description

`verify_transaction_inclusion_v2` was added to close the 64-byte Merkle forgery attack. Its protection is a coinbase-proof check: it requires the caller to also supply a valid Merkle proof for the coinbase transaction (always at index 0), which proves the supplied `tx_id` is a real leaf and not an internal 32-byte node pair concatenated to 64 bytes. [1](#0-0) 

After that check passes, `v2` delegates to the original function: [2](#0-1) 

The original function, however, is still `pub` and decorated with `#[pause]` (active by default), making it a first-class NEAR contract method: [3](#0-2) 

Its own doc comment acknowledges the risk explicitly: [4](#0-3) 

An attacker who calls `verify_transaction_inclusion` directly never encounters the coinbase-proof requirement. The function performs only a plain Merkle root comparison: [5](#0-4) 

---

### Impact Explanation

The 64-byte attack (https://www.bitmex.com/blog/64-Byte-Transactions) works because Bitcoin's Merkle tree is computed by hashing pairs of 32-byte child hashes, producing a 32-byte parent. An internal parent node is itself 32 bytes and can be presented as a `tx_id`. A crafted Merkle proof path can make `compute_root_from_merkle_proof` reconstruct the correct Merkle root starting from an internal node rather than a real transaction hash.

Because `verify_transaction_inclusion` performs no coinbase check, it will return `true` for such a forged proof. Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate a payment, bridge withdrawal, or other high-value action will accept a fabricated Bitcoin transaction as confirmed.

---

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` via a standard NEAR RPC `call` transaction. The attacker only needs:
1. A valid mainchain block hash stored in the contract (publicly readable via `get_block_hash_by_height`).
2. Knowledge of that block's Merkle tree structure (available from any Bitcoin node or block explorer).
3. Construction of a 64-byte internal-node proof — a well-documented technique with public tooling.

No whitelisting, staking, or role is required to reach this function.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` or restrict it to `pub(crate)` / `#[private]` so it is no longer callable as a NEAR contract method. All external callers must be forced through `verify_transaction_inclusion_v2`. Alternatively, delete the function entirely and inline its logic inside `v2` only. [3](#0-2) 

---

### Proof of Concept

1. Identify a confirmed mainchain block `B` with at least two transactions. Obtain its Merkle tree from a Bitcoin node.
2. Pick any internal node `N` at depth 1 (the hash of the two leaf txids). `N` is 32 bytes and looks like a valid txid.
3. Construct a Merkle proof path from `N` upward to the Merkle root. Because `N` is already a parent of two leaves, the path is one step shorter than a leaf proof, but `compute_root_from_merkle_proof` will still reconstruct the correct root.
4. Call `verify_transaction_inclusion` on the NEAR contract with `tx_id = N`, `tx_block_blockhash = B`, and the crafted `merkle_proof`.
5. The function returns `true` — confirming a transaction that does not exist.

`verify_transaction_inclusion_v2` would reject this call at step 4 because the coinbase proof check would fail (the coinbase tx at index 0 cannot produce the same root when `N` is used as `tx_id` at a non-zero index with a mismatched proof length). But since `verify_transaction_inclusion` is callable directly, that guard is never reached. [6](#0-5)

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );
