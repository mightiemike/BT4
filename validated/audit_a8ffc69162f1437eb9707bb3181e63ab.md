### Title
Missing Sanction Check on Linked Signer Allows Sanctioned Addresses to Authorize Protocol Transactions - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

Both the sequencer-path and slow-mode-path handlers for `LinkSigner` transactions set a new linked signer without verifying that the new signer is unsanctioned. This is directly analogous to the commented-out blacklist check in `PerpetualBond::_update`: a restriction that should gate who can act on a subaccount is simply absent, allowing a sanctioned address to be installed as an authorized signer and subsequently sign withdrawals and other sensitive transactions on behalf of any subaccount.

---

### Finding Description

The protocol enforces sanctions checks at deposit and slow-mode submission entry points. In `depositCollateralWithReferral`, both the caller and the subaccount owner are checked: [1](#0-0) 

In `submitSlowModeTransactionImpl`, the submitting sender is checked before the transaction is queued: [2](#0-1) 

However, when a `LinkSigner` transaction is processed — both through the sequencer path in `processTransactionImpl` and through the slow-mode execution path in `processSlowModeTransactionImpl` — the new signer address is written to `linkedSigners` with **no** `requireUnsanctioned` check on the incoming signer:

**Sequencer path (`processTransactionImpl`):** [3](#0-2) 

**Slow-mode path (`processSlowModeTransactionImpl`):** [4](#0-3) 

Once a sanctioned address is stored as a linked signer, `validateSignature` and `validateCompactSignature` pass it directly to the verifier as an accepted signer: [5](#0-4) 

The verifier then accepts signatures from the sanctioned address as valid authorization for the subaccount, with no further on-chain sanction gate.

---

### Impact Explanation

A sanctioned address installed as a linked signer can sign `WithdrawCollateral` and `WithdrawCollateralV2` transactions for the subaccount. The sequencer processes these via `submitTransactionsChecked`; `validateSignedTx` with `allowLinkedSigner = true` accepts the sanctioned address's signature, and the withdrawal proceeds: [6](#0-5) 

The concrete asset delta: collateral held in the subaccount can be withdrawn to an arbitrary `sendTo` address controlled by the sanctioned party, bypassing the protocol's entire sanctions enforcement layer.

---

### Likelihood Explanation

The slow-mode path is directly reachable by any non-sanctioned user without sequencer cooperation for the setup step. The user submits a `LinkSigner` slow-mode transaction naming a sanctioned address as signer; after the 3-day `SLOW_MODE_TX_DELAY`, anyone calls `executeSlowModeTransaction` to finalize it. The sanctioned address then signs withdrawal transactions that the sequencer processes on-chain with no further sanction check. The sequencer may have off-chain filters, but the on-chain enforcement gap means any sequencer that does not independently re-validate every linked signer on every transaction will propagate the bypass.

---

### Recommendation

Add `requireUnsanctioned` on the incoming signer address in both `LinkSigner` handlers, mirroring the pattern already used for depositors:

```solidity
// In processTransactionImpl (LinkSigner branch):
address newSigner = address(uint160(bytes20(signedTx.tx.signer)));
requireUnsanctioned(newSigner);
linkedSigners[signedTx.tx.sender] = newSigner;

// In processSlowModeTransactionImpl (LinkSigner branch):
address newSigner = address(uint160(bytes20(txn.signer)));
requireUnsanctioned(newSigner);
linkedSigners[txn.sender] = newSigner;
```

---

### Proof of Concept

1. Sanctioned address `B` exists in the sanctions list.
2. Non-sanctioned user `A` calls `submitSlowModeTransaction` with a `LinkSigner` payload naming `B` as signer for `A`'s subaccount. `requireUnsanctioned(A)` passes at line 375; no check is made on `B`.
3. After `SLOW_MODE_TX_DELAY`, anyone calls `executeSlowModeTransaction`. `processSlowModeTransactionImpl` writes `linkedSigners[A_subaccount] = B` at line 239 with no sanction check on `B`.
4. `B` signs a `WithdrawCollateralV2` transaction for `A`'s subaccount, specifying `sendTo = B_controlled_address`.
5. The sequencer includes this in `submitTransactionsChecked`. `validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which passes `B` to the verifier as the accepted linked signer.
6. The verifier accepts `B`'s signature; `clearinghouse.withdrawCollateral` executes, transferring `A`'s collateral to `B`'s address. [4](#0-3) [7](#0-6) [1](#0-0)
