### Title
Wrong `MAX_FUTURE_BLOCK_TIME_MTP` Constant Causes Valid Zcash Blocks to Be Rejected - (`btc-types/src/network.rs`, `contract/src/zcash.rs`)

### Summary

The constant `MAX_FUTURE_BLOCK_TIME_MTP` is hardcoded to `90 * 60 = 5400` seconds (90 minutes) in `btc-types/src/network.rs`. The Zcash protocol defines this value as `60 * 60 * 36 = 129600` seconds (36 hours). The code's own inline comment in `contract/src/zcash.rs` explicitly acknowledges the correct Zcash value is 129600 seconds, yet the wrong constant is used in the MTP-based timestamp validity check. This is a direct analog to the "wrong init code hash" class: a hardcoded protocol constant does not match the upstream reference implementation, causing the light client to apply a rule that is 24× stricter than the actual Zcash consensus rule.

### Finding Description

`MAX_FUTURE_BLOCK_TIME_MTP` is defined as:

```rust
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 90 * 60;  // 5400 seconds
``` [1](#0-0) 

It is consumed in `contract/src/zcash.rs` inside `check_pow`, which is called for every submitted Zcash block:

```rust
// MAX_FUTURE_BLOCK_TIME_MTP is typically 129600 seconds (36 hours) in Zcash
require!(
    block_header.time
        <= next_work_result.prev_block_median_time_past + MAX_FUTURE_BLOCK_TIME_MTP,
    "time-too-far-ahead-of-mtp: block timestamp is too far ahead of median-time-past"
);
``` [2](#0-1) 

The comment is correct: the Zcash reference implementation (`zcash/zcash v6.2.0 src/main.cpp`) defines `MAX_FUTURE_BLOCK_TIME_MTP` as `60 * 60 * 36 = 129600` seconds. The constant in this repository is `5400` seconds — a 24× undercount. Any valid Zcash block whose timestamp falls between 90 minutes and 36 hours ahead of the previous block's MTP satisfies the real Zcash consensus rule but is unconditionally rejected by this contract.

`check_pow` is called from `submit_block_header` for every non-skipped Zcash block submission: [3](#0-2) 

### Impact Explanation

A valid Zcash block accepted by the real Zcash network (timestamp ≤ MTP + 36 hours) but with a timestamp > MTP + 90 minutes is permanently rejected by the light client with `"time-too-far-ahead-of-mtp"`. The relayer cannot work around this: the block is valid on-chain but the contract's hardcoded constant makes it unacceptable. The light client stalls at the last accepted block, the `mainchain_tip_blockhash` is never advanced past that point, and all subsequent `verify_transaction_inclusion` / SPV proof calls for blocks beyond the stall height return false negatives — legitimate cross-chain transaction proofs are denied.

### Likelihood Explanation

Zcash's 75-second post-Blossom block time means the MTP lags the chain tip by roughly 5–6 blocks (~375–450 seconds). A block timestamped 91 minutes ahead of MTP is within the Zcash protocol's allowed window but outside this contract's window. Mining pools and nodes occasionally set timestamps ahead of wall-clock time; any such block in the 90-minute–36-hour window triggers the stall. The condition is not attacker-manufactured but arises from normal Zcash network operation.

### Recommendation

Replace the constant with the correct Zcash value:

```rust
// Zcash: 36 hours, matching zcash/zcash src/main.cpp MAX_FUTURE_BLOCK_TIME_MTP
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 60 * 60 * 36; // 129600 seconds
``` [1](#0-0) 

Because `MAX_FUTURE_BLOCK_TIME_MTP` is also imported by the Bitcoin and Litecoin modules (neither of which uses it in their `check_pow` today), the safest fix is to introduce a Zcash-specific constant or move the value into `ZcashConfig` alongside the other chain-specific parameters. [4](#0-3) 

### Proof of Concept

1. Deploy the Zcash-feature contract.
2. Obtain a real Zcash mainnet block whose `nTime` satisfies:
   - `nTime > MTP_of_prev + 5400` (rejected by this contract)
   - `nTime ≤ MTP_of_prev + 129600` (accepted by the real Zcash network)
3. Submit it via `submit_blocks`. The contract panics with `"time-too-far-ahead-of-mtp"`.
4. The `mainchain_tip_blockhash` is not updated; all subsequent SPV proof calls for blocks at or above this height return `false`.

The root cause is the single constant at `btc-types/src/network.rs:11` being `5400` instead of `129600`, confirmed by the developer's own comment at `contract/src/zcash.rs:40`. [5](#0-4)
