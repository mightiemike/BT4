### Title
`ValidResourceBounds::AllResources` with zero non-L1 gas silently collapses to `L1Gas` across protobuf round-trip, producing a divergent transaction hash - (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a zero-value heuristic to choose between the `L1Gas` and `AllResources` variants. An `AllResources` transaction whose `l2_gas` and `l1_data_gas` fields are both zero is serialized correctly but deserialized as `L1Gas`. Because `get_tip_resource_bounds_hash` includes the `L1_DATA_GAS` element only for `AllResources`, the transaction hash computed after a consensus P2P round-trip differs from the hash computed at the gateway. The test suite itself acknowledges the problem and works around it by artificially inflating gas values.

### Finding Description

**Lossy deserialization in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`** [1](#0-0) 

When `l1_data_gas` and `l2_gas` are both zero the deserializer returns `ValidResourceBounds::L1Gas(l1_gas)` instead of `ValidResourceBounds::AllResources(...)`. A transaction that was originally `AllResources` with zero non-L1 gas values is therefore silently reinterpreted as the older `L1Gas` variant.

**Hash divergence in `get_tip_resource_bounds_hash`** [2](#0-1) 

The function appends the `L1_DATA_GAS` element to the hash chain only for `AllResources`. For `L1Gas` that element is omitted entirely. Consequently:

- Gateway path: `AllResources` → hash includes `[tip, L1_GAS, L2_GAS, L1_DATA_GAS(0)]`
- After P2P round-trip: `L1Gas` → hash includes `[tip, L1_GAS, L2_GAS]`

The two hashes are distinct even though the underlying numeric values are identical.

**Hard failure when converting back to `RpcInvokeTransactionV3`** [3](#0-2) 

`TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` returns `StarknetApiError::OutOfRange` for any variant other than `AllResources`. A consensus peer that receives the protobuf-encoded transaction and deserializes it as `L1Gas` will therefore fail to reconstruct the `ConsensusTransaction`, causing the entire proposal to be rejected.

**Test-level acknowledgement of the invariant break** [4](#0-3) 

The comment reads: *"If all the fields of `AllResources` are 0 upon serialization, then the deserialized value will be interpreted as the `L1Gas` variant."* The workaround (`resource_bounds.l2_gas.max_amount = GasAmount(1)`) is applied only in tests; production code has no such guard.

### Impact Explanation

A valid `InvokeV3` transaction submitted through the gateway with `AllResourceBounds` where `l2_gas = 0` and `l1_data_gas = 0` is accepted and assigned hash H₁ (computed with the `AllResources` path). When the proposer broadcasts the block over the consensus P2P channel, every validator deserializes the `ResourceBounds` as `L1Gas` and either:

1. Recomputes a different hash H₂ ≠ H₁, breaking block-hash consistency and causing the block to be rejected; or
2. Fails outright during `RpcInvokeTransactionV3` reconstruction, causing the proposal to be dropped.

This matches **High – Transaction conversion or signature/hash logic binds the wrong hash or executable payload** and **High – Mempool/gateway/RPC admission rejects valid transactions before sequencing**.

### Likelihood Explanation

The gateway's stateless validator explicitly accepts transactions with zero `l2_gas` and `l1_data_gas`. [5](#0-4) 

Any user who submits such a transaction (e.g., a legacy-style V3 invoke that sets only `l1_gas`) will trigger the bug without any special privilege. The trigger is a normal RPC call.

### Recommendation

1. **Add an explicit discriminator** to the protobuf `ResourceBounds` message (e.g., a boolean `is_all_resources` flag) so the variant is preserved independently of numeric values.
2. **Alternatively**, always serialize and deserialize `ResourceBounds` as `AllResources` in the consensus/mempool P2P path (since `RpcInvokeTransactionV3` already mandates `AllResourceBounds`), and reserve the `L1Gas` variant only for the storage/blockifier layer.
3. **Remove the workaround** in `add_gas_values_to_transaction` and replace it with a proper fix so the invariant is enforced in production, not just in tests.

### Proof of Concept

1. Submit an `InvokeV3` via RPC with `resource_bounds = { l1_gas: {max_amount: N, max_price: P}, l2_gas: {0, 0}, l1_data_gas: {0, 0} }`.
2. Gateway accepts it; `convert_rpc_tx_to_internal` computes hash H₁ via `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` with `AllResources` → hash chain includes `L1_DATA_GAS(0)`. [6](#0-5) 
3. Proposer serializes the `ConsensusTransaction` to protobuf; `From<ValidResourceBounds> for protobuf::ResourceBounds` emits `l2_gas = 0`, `l1_data_gas = 0`. [7](#0-6) 
4. Validator deserializes: `l2_gas.is_zero() && l1_data_gas.is_zero()` → `ValidResourceBounds::L1Gas`. [8](#0-7) 
5. Conversion to `RpcInvokeTransactionV3` hits the `_ => Err(OutOfRange)` branch → consensus deserialization fails → proposal rejected. [9](#0-8) 
6. Even if the error path were bypassed, `get_tip_resource_bounds_hash` with `L1Gas` produces hash H₂ ≠ H₁ because the `L1_DATA_GAS` element is absent. [10](#0-9)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
