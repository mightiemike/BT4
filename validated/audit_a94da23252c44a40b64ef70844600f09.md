### Title
Duplicate-Last-Leaf Position Aliasing in Odd-Width Merkle Trees Allows Phantom Transaction Proof — (`merkle-tools/src/lib.rs::compute_root_from_merkle_proof`)

---

### Summary

`compute_root_from_merkle_proof` never validates that `transaction_position` is within the actual transaction count of the block. Bitcoin's Merkle tree construction duplicates the last leaf when the transaction count is odd. This means the proof for the real last transaction at position `N-1` is mathematically identical to a proof for a phantom position `N` — a position that does not exist in the block. `verify_transaction_inclusion` returns `true` for both, enabling a downstream bridge, unlock, or mint to accept a nonexistent transaction as proven.

---

### Finding Description

**Root cause — `compute_root_from_merkle_proof`:** [1](#0-0) 

The function uses `transaction_position` only to decide left/right placement at each tree level (`current_position % 2`). It performs no bounds check against the actual number of transactions. The contract stores only block headers, not transaction counts, so no upper bound is ever enforced.

**Bitcoin's odd-width duplication rule:**

When a block has an odd number of transactions `N`, Bitcoin's Merkle construction pads the leaf array by appending a copy of the last transaction: `[T0, T1, …, T_{N-1}, T_{N-1}]`. This is implemented in `merkle_proof_calculator`: [2](#0-1) 

**Concrete proof of aliasing (N=3 example):**

Tree: `[T0, T1, T2]` → padded to `[T0, T1, T2, T2]`
- Level 1: `[H(T0,T1), H(T2,T2)]`
- Root: `H(H(T0,T1), H(T2,T2))`

Legitimate proof for `T2` at position `2`:
- `proof[0] = T2` (sibling at position 3, the duplicate)
- `proof[1] = H(T0,T1)` (sibling at level 1)

Now run `compute_root_from_merkle_proof(T2, 3, [T2, H(T0,T1)])`:
1. `position=3` is odd → `current_hash = H(proof[0]=T2, T2) = H(T2,T2)`, `position=1`
2. `position=1` is odd → `current_hash = H(proof[1]=H(T0,T1), H(T2,T2)) = root` ✓

The function returns the correct root for phantom position `3`, which does not exist in the block. `verify_transaction_inclusion` therefore returns `true` for it: [3](#0-2) 

**No guard prevents this.** The only checks in `verify_transaction_inclusion` are: block is in the main chain, confirmations are sufficient, and the proof is non-empty: [4](#0-3) 

None of these constrain `tx_index` to `[0, actual_tx_count)`.

---

### Impact Explanation

A downstream bridge, unlock, or mint contract that calls `verify_transaction_inclusion` and tracks processed transactions by `(tx_id, block_hash, tx_index)` can be made to accept the same real transaction twice — once at the real position `N-1` and once at the phantom position `N`. Both calls return `true` with the same `tx_id` and the same proof. If the downstream system uses `tx_index` as part of its deduplication key (a natural design when a single address can send multiple outputs in one block), the attacker achieves a double-spend or double-mint.

The deprecated status of `verify_transaction_inclusion` does not eliminate the risk: `verify_transaction_inclusion_v2` calls it internally after the coinbase check: [5](#0-4) 

The coinbase proof check in v2 validates only that the coinbase tx is at position `0`; it does not constrain the target `tx_index`.

---

### Likelihood Explanation

- The function is public and callable by any unprivileged account (`#[pause]` only, no role restriction).
- Any Bitcoin block with an odd transaction count (the majority of real blocks) is a valid target.
- The attacker needs only a real Bitcoin transaction in such a block and the block's header already relayed to the contract — no privileged access, no key compromise, no social engineering.
- The proof construction is trivial: generate the real proof for position `N-1`, then resubmit with `tx_index = N`.

---

### Recommendation

1. **Enforce position bounds in `compute_root_from_merkle_proof`:** The proof length encodes the tree depth. The maximum valid position is `2^depth - 1`, but the actual valid range is `[0, tx_count)`. The contract must either store `tx_count` per block header or require callers to supply it and validate `tx_index < tx_count` before invoking the Merkle verifier.

2. **Alternatively, reject even-position proofs whose sibling equals the transaction itself:** If `proof[0] == tx_id` and `tx_index` is even, the caller may be exploiting the duplicate-leaf padding. Reject such proofs unless the tree is known to have exactly one transaction.

3. **Downstream consumers** must deduplicate verified events by `tx_id` alone (not by `(tx_id, tx_index)`) to prevent the phantom-position replay even before the contract is patched.

### Proof of Concept

```rust
// Pseudocode — runs against an unmodified contract in a NEAR sandbox test.
// Block has 3 transactions: [T0, T1, T2].  Merkle root = H(H(T0,T1), H(T2,T2)).

let real_proof   = vec![T2.clone(), hash(T0, T1)];  // legitimate proof for position 2
let phantom_proof = real_proof.clone();               // identical bytes

// Both calls return true:
assert!(contract.verify_transaction_inclusion(ProofArgs {
    tx_id: T2.clone(), tx_block_blockhash: block_hash,
    tx_index: 2, merkle_proof: real_proof, confirmations: 1,
}));

assert!(contract.verify_transaction_inclusion(ProofArgs {
    tx_id: T2.clone(), tx_block_blockhash: block_hash,
    tx_index: 3,           // phantom — does not exist in the block
    merkle_proof: phantom_proof, confirmations: 1,
}));
// A bridge tracking by (tx_id, block_hash, tx_index) processes T2 twice → double-mint.
```

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

            "The required number of confirmations exceeds the number of blocks stored in memory"
