Audit Report

## Title
SwapAllowlistExtension validates router address as `sender` instead of actual user, enabling allowlist bypass — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap()` passes `msg.sender` (the immediate caller) as the `sender` argument to `_beforeSwap`, which forwards it to `SwapAllowlistExtension.beforeSwap`. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the router contract, not the actual end user. The allowlist therefore checks whether the **router** is permitted, not the individual trader. Any user who routes through an allowlisted router bypasses per-user access control entirely.

## Finding Description

**Call path:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle()` (or `exactInput`, `exactOutputSingle`, `exactOutput`).
2. The router calls `IMetricOmmPoolActions(pool).swap(recipient, zeroForOne, amountSpecified, priceLimitX64, "", extensionData)` — `msg.sender` to the pool is the **router address**.
3. Inside `MetricOmmPool.swap()` (line 230–240), the pool calls `_beforeSwap(msg.sender, ...)` — `msg.sender` here is the router.
4. `ExtensionCalling._beforeSwap` forwards `sender = router_address` to `SwapAllowlistExtension.beforeSwap`.
5. `SwapAllowlistExtension.beforeSwap` (line 37) evaluates:
   ```solidity
   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
       revert IMetricOmmPoolActions.NotAllowedToSwap();
   }
   ```
   Here `msg.sender` = pool, `sender` = router. The check is `allowedSwapper[pool][router]`.

**Root cause:** The allowlist is keyed on the pool's direct caller (`msg.sender` of `pool.swap()`), which is the router — not the originating user (`tx.origin` or the router's own `msg.sender`).

**Exploit flow:**
- Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a whitelist of KYC'd addresses.
- Pool admin allowlists the `MetricOmmSimpleRouter` (a reasonable operational step so that legitimate users can use the standard router).
- Any non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle()` targeting the restricted pool.
- The allowlist check passes because `allowedSwapper[pool][router] == true`.
- The non-allowlisted user successfully swaps in a pool they were explicitly excluded from.

**Existing guards are insufficient:** The only guard is `allowedSwapper[pool][sender]` where `sender` is always the router when the router is used. There is no mechanism in the extension to recover the original user identity.

## Impact Explanation
Broken core pool functionality: the allowlist extension — a production periphery contract — fails to enforce per-user access control when swaps are routed through `MetricOmmSimpleRouter`. Any unprivileged user can swap in a pool that is explicitly restricted to a whitelist by routing through the standard router if the router itself is allowlisted. This constitutes an admin-boundary break (allowlist bypass by an unprivileged path) and broken core swap-gating functionality, meeting Sherlock High/Medium thresholds.

## Likelihood Explanation
The condition requires the router to be allowlisted for the target pool. This is a natural operational step — pool admins who want to support standard router usage must allowlist the router. Once the router is allowlisted, the bypass is trivially reachable by any user with no special privileges, no capital requirements beyond the swap amount, and is fully repeatable.

## Recommendation
The `SwapAllowlistExtension` must validate the actual originating user, not the pool's direct caller. Options:
1. Require callers to pass the real user address in `extensionData` and validate it, combined with a trusted-router check on `sender`.
2. Check `tx.origin` as a fallback (with documented limitations for smart-contract wallets).
3. Document that the allowlist extension is incompatible with router usage and revert if `sender` is a known router, or require direct pool calls only.
