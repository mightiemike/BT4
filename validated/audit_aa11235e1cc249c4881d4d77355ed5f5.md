### Title
Unbounded `NlpLockedBalanceQueue` While-Loop Permanently Locks NLP Liquidity Provider Funds — (`SpotEngineState.sol`)

---

### Summary

`tryUnlockNlpBalance` in `SpotEngineState.sol` iterates over every unlocked entry in a subaccount's `NlpLockedBalanceQueue` with no iteration cap. Because `handleNlpLockedBalance` in `SpotEngine.sol` appends a new queue entry for every NLP deposit that arrives at a distinct oracle timestamp, a user who makes many NLP deposits over time accumulates an unbounded number of entries. Once the queue depth exceeds the block gas limit capacity, every subsequent NLP operation for that subaccount permanently reverts, locking the user's NLP balance with no on-chain recovery path.

---

### Finding Description

**Root cause — unbounded while loop with no batch limit**

`tryUnlockNlpBalance` (`SpotEngineState.sol:285-306`) iterates from `queue.unlockedUpTo` to the first still-locked entry in a single `while` loop:

```solidity
while (
    queue.unlockedUpTo < queue.balanceCount &&
    queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
) {
    queue.unlockedBalanceSum.amount += queue.balances[queue.unlockedUpTo].balance.amount;
    delete queue.balances[queue.unlockedUpTo];
    queue.unlockedUpTo++;
}
```

There is no `maxIterations` guard, no partial-batch mechanism, and no way to resume a partial run — if the transaction runs out of gas mid-loop, all state changes revert and `unlockedUpTo` is unchanged.

**Root cause — unbounded queue growth**

`handleNlpLockedBalance` (`SpotEngine.sol:139-174`) appends a **new** `NlpLockedBalance` entry whenever `amountDelta > 0` and the last entry's `unlockedAt` timestamp differs from `getOracleTime() + NLP_LOCK_PERIOD`:

```solidity
} else {
    queue.balances[queue.balanceCount] = NlpLockedBalance({
        balance: Balance({amount: amountDelta}),
        unlockedAt: getOracleTime() + NLP_LOCK_PERIOD
    });
    queue.balanceCount++;
}
```

`balanceCount` is a `uint64` with no upper-bound check anywhere in the codebase. The `NlpLockedBalanceQueue` struct (`ISpotEngine.sol:53-58`) imposes no storage cap.

**Call chain that triggers the lock**

Every NLP balance update calls `handleNlpLockedBalance` → `tryUnlockNlpBalance` → the unbounded while loop:

```
Endpoint.depositCollateralWithReferral / submitSlowModeTransaction
  → EndpointTx.processSlowModeTransactionImpl
    → Clearinghouse / OffchainExchange
      → SpotEngine.updateBalance (productId == NLP_PRODUCT_ID)
        → handleNlpLockedBalance
          → tryUnlockNlpBalance   ← unbounded loop
```

**Lock period and entry-creation rate**

`NLP_LOCK_PERIOD = 4 days` (`Constants.sol:52`). `getOracleTime()` is updated by the sequencer on each submission. Each NLP deposit at a distinct oracle timestamp creates one new queue entry. Entries only merge when two consecutive deposits share the exact same oracle timestamp — in practice, each deposit at a different block creates a separate entry.

---

### Impact Explanation

Once `queue.balanceCount - queue.unlockedUpTo` exceeds the block gas limit capacity (roughly 3 000–6 000 cold-storage iterations at 30 M gas), every call that touches the affected subaccount's NLP balance permanently reverts:

- **NLP deposits** revert → user cannot add liquidity.
- **NLP withdrawals** revert → user cannot redeem locked NLP tokens.
- **Any sequencer-submitted NLP settlement** for that subaccount reverts → the sequencer must permanently skip the subaccount, leaving its NLP balance frozen on-chain.

The user's NLP balance is permanently inaccessible with no on-chain recovery path short of a contract upgrade. This is a permanent freeze of user assets, not a transient gas-cost increase.

---

### Likelihood Explanation

`NLP_LOCK_PERIOD = 4 days` and `SLOW_MODE_FEE = $1` (`Constants.sol:23, 52`). A user making one NLP deposit per oracle-time tick over 4 days accumulates entries continuously. To reach ~3 000 entries costs ~$3 000 in slow-mode fees — a realistic cost for a large liquidity provider operating over weeks. The condition is also reachable accidentally: a legitimate LP who deposits frequently and never explicitly calls `tryUnlockNlpBalance` between lock periods will silently grow their queue until the threshold is crossed. No privileged access is required; the trigger path is the standard NLP deposit flow available to any user.

---

### Recommendation

1. **Add a per-call iteration cap** to `tryUnlockNlpBalance` (e.g., `maxIterations = 200`) and return a boolean indicating whether the queue was fully drained. Callers that need a fully-drained queue can loop externally.
2. **Cap `balanceCount`** at a safe maximum (e.g., 500) in `handleNlpLockedBalance`, reverting or merging aggressively when the cap is reached.
3. **Expose a standalone `drainNlpQueue(bytes32 subaccount, uint256 maxSteps)`** function so users or keepers can incrementally drain a large queue before it becomes unserviceable.

---

### Proof of Concept

```
Precondition: oracle time advances with each sequencer submission (normal operation).

1. Attacker/user calls depositCollateralWithReferral for NLP (productId = NLP_PRODUCT_ID = 11)
   at oracle timestamp T₀  → queue entry 0: unlockedAt = T₀ + 4 days
2. Repeats at T₁, T₂, … T_N (each at a distinct oracle timestamp, $1 fee each)
   → queue.balanceCount = N, all entries have distinct unlockedAt values.
3. After 4 days, all N entries satisfy unlockedAt ≤ getOracleTime().
4. User (or sequencer) submits any NLP operation for the subaccount.
5. handleNlpLockedBalance → tryUnlockNlpBalance enters the while loop.
6. Loop must process all N entries in one atomic call.
   At ~5 000–10 000 gas/iteration (cold SLOAD + SSTORE), N ≈ 3 000–6 000
   exhausts the 30 M block gas limit.
7. Transaction reverts. unlockedUpTo is unchanged.
8. Every future NLP operation for this subaccount reverts identically.
   The user's NLP balance is permanently locked on-chain.
