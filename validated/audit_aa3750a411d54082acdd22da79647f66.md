### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery — (File: contract/src/lib.rs)

---

### Summary

`verify_transaction_inclusion` is marked `#[deprecated]` specifically because it cannot distinguish a real leaf-level transaction hash from an internal Merkle tree node hash. Rust's `#[deprecated]` attribute is a compiler hint only — it does not restrict on-chain access. Any unprivileged NEAR caller can still invoke the function directly, supply an attacker-chosen `tx_id` that is an internal node of a real block's Merkle tree, and receive `true` in return, forging proof of a Bitcoin transaction that never existed.

---

### Finding Description

`verify_transaction_inclusion` accepts a caller-supplied `tx_id` and a caller-supplied `merkle_proof` and computes whether they reconstruct the stored `merkle_root`: [1](#0-0) 

The function's own documentation acknowledges the broken invariant: [2](#0-1) 

The function is `pub` with only a `#[pause]` guard (no access-control role restriction): [3](#0-2) 

`#[deprecated]` in Rust emits a compiler warning; it does not remove the method from the WASM ABI. Any NEAR account can call it via a cross-contract call or a direct transaction.

Bitcoin's Merkle tree uses `double_sha256(left_child || right_child)` for every internal node — the same hash function used for leaf (transaction) nodes: [4](#0-3) 

Because `compute_root_from_merkle_proof` treats the supplied `tx_id` as an opaque 32-byte value and simply hashes it upward through the proof path, an internal node hash at depth `d` with a proof of length `d` will reconstruct the correct Merkle root just as legitimately as a real leaf would: [5](#0-4) 

`verify_transaction_inclusion_v2` was introduced to close this gap by requiring a coinbase proof of equal length, forcing the attacker's proof depth to match the full tree depth: [6](#0-5) 

However, because v1 is never removed from the public ABI, the fix in v2 is bypassed entirely by calling v1 directly.

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate a security-critical action — releasing bridged BTC, confirming a cross-chain payment, minting wrapped tokens — will receive `true` for a fabricated `tx_id`. The attacker does not need to produce a real Bitcoin transaction; they only need to identify an internal node in any already-submitted block's Merkle tree and supply the corresponding shorter proof. The corrupted proof result propagates directly to the consuming contract's authorization logic.

---

### Likelihood Explanation

- No privileged role is required; any NEAR account can call `verify_transaction_inclusion`.
- All inputs needed for the attack (block Merkle trees, internal node hashes, proof paths) are publicly available from Bitcoin's blockchain.
- The contract is not paused by default, so the `#[pause]` guard provides no protection in normal operation.
- The function is documented as vulnerable, meaning integrators who read the deprecation notice may still call it if they have not yet migrated.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public WASM ABI entirely (change `pub fn` to `fn` or delete it). If backward compatibility must be preserved temporarily, add an explicit `Role`-gated access-control check identical to the pattern used for other privileged operations in the contract, so that unprivileged callers cannot invoke it. All integrators must be migrated to `verify_transaction_inclusion_v2`.

---

### Proof of Concept

1. Attacker selects any block `B` already stored in the contract (its Merkle root `R` is in `headers_pool`).
2. From Bitcoin's public data, attacker reads the Merkle tree of `B` and picks an internal node `N` at depth `d` (where `d < tree_depth`). The hash of `N` is a valid 32-byte value.
3. Attacker constructs a Merkle proof `P` of length `d` for `N` — this proof is valid because `N` is a genuine node in the tree.
4. Attacker calls:
   ```
   verify_transaction_inclusion({
     tx_id: N,
     tx_block_blockhash: B,
     tx_index: <position of N at depth d>,
     merkle_proof: P,
     confirmations: 1
   })
   ```
5. Inside the function, `compute_root_from_merkle_proof(N, position, P)` hashes `N` upward through `P` and produces `R`.
6. `R == header.block_header.merkle_root` → function returns `true`.
7. The downstream contract that called `verify_transaction_inclusion` treats this as proof that a Bitcoin transaction with id `N` was confirmed in block `B`, and releases funds to the attacker. [7](#0-6) [8](#0-7)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
