### Title
Phantom Merkle Index: Duplicate-Leaf Duplication Allows `verify_transaction_inclusion` to Return `true` for a Non-Existent Transaction Position — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

### Summary
Bitcoin's Merkle tree construction duplicates the last leaf when the transaction count is odd. `compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` is purely positional and has no knowledge of the actual transaction count. `verify_transaction_inclusion` in `contract/src/lib.rs` performs no bounds check on `tx_index`. As a result, for any canonical block with an odd number of transactions `n`, an attacker can supply `tx_index = n` (a position that does not exist) with `tx_id = tx[n-1]` and the proof for index `n-1`, and the function returns `true`.

### Finding Description
`merkle_proof_calculator` handles odd-length lists by appending a duplicate of the last element before pairing: [1](#0-0) 

This means positions `n-1` and `n` (the real last leaf and its phantom duplicate) produce **identical proofs** and identical computed roots. `compute_root_from_merkle_proof` is a pure positional computation: [2](#0-1) 

`verify_transaction_inclusion` compares only the computed root against the stored `merkle_root` — there is no check that `tx_index` is within the actual transaction count of the block: [3](#0-2) 

`verify_transaction_inclusion_v2` adds a coinbase proof check but delegates to the same vulnerable function without adding a bounds check: [4](#0-3) 

### Impact Explanation
Any unprivileged NEAR account can call `verify_transaction_inclusion` (or `verify_transaction_inclusion_v2`) and receive `true` for a transaction at a position that does not exist in the committed block. Downstream contracts using this as an SPV oracle (e.g., to gate token minting, bridge unlocks, or settlement) can be deceived. The same real transaction `tx[n-1]` can be "proven" at two distinct indices (`n-1` and `n`), enabling double-processing of a single Bitcoin transaction if the downstream contract keys its replay protection on `(tx_id, tx_index)` rather than `tx_id` alone.

### Likelihood Explanation
The precondition — a canonical block with an odd transaction count — is extremely common in Bitcoin (the majority of blocks have an odd number of transactions). No privileged role is required. The attacker only needs the last transaction hash and the standard Merkle proof for it, both of which are publicly available from any Bitcoin node.

### Recommendation
Add an explicit upper-bound check on `tx_index` in `verify_transaction_inclusion`. Since the contract does not store the transaction count, the proof length encodes the tree depth: the maximum valid index is `2^proof_len - 1`. A tighter fix is to require `tx_index < 2^(proof_len) - 1` (i.e., strictly less than the padded tree size minus the phantom slot). Concretely:

```rust
// After computing proof_len = args.merkle_proof.len()
let max_valid_index = (1usize << args.merkle_proof.len()) - 1;
require!(
    usize::try_from(args.tx_index).unwrap() < max_valid_index,
    "tx_index out of range for the given proof length"
);
```

This rejects the phantom position `n` while accepting all real positions `0..n-1`. Alternatively, require callers to supply the transaction count and validate `tx_index < tx_count`, storing or committing to `tx_count` in the block header metadata.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
