Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of end-user, allowing any caller to bypass per-user swap allowlist - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which `MetricOmmPool.swap` populates with its own `msg.sender`. When `MetricOmmSimpleRouter` intermediates the call, the pool's `msg.sender` is the router contract, not the end user. Any pool admin who allowlists the router to support curated users inadvertently opens the pool to every unprivileged caller who routes through it.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` directly as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every registered extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool received as its own `msg.sender`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` stores the original `msg.sender` in transient storage for callback settlement purposes only, but calls `pool.swap(...)` with no mechanism to forward the original caller identity: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

The result: the pool sees `msg.sender = router`, `_beforeSwap` receives `sender = router`, and the extension evaluates `allowedSwapper[pool][router]` — never `allowedSwapper[pool][user]`. The `DepositAllowlistExtension` avoids this by gating on the explicit `owner` parameter, which represents the actual depositor regardless of who calls the pool: [6](#0-5) 

`SwapAllowlistExtension` has no equivalent mechanism to recover the true end-user identity.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, institutional traders, or protocol-owned bots) is fully bypassed by any unprivileged user who calls any router entry point. Once the router is allowlisted — a necessary step for any allowlisted user to benefit from multi-hop routing or slippage protection — the curation boundary collapses entirely. Unauthorized users can execute swaps against oracle-anchored prices, extract LP value, and front-run or sandwich allowlisted traders, all actions the pool admin explicitly intended to prevent. This constitutes a broken core pool access-control mechanism causing direct loss of LP value and violation of the admin-boundary invariant.

## Likelihood Explanation

The trigger is the standard, publicly documented periphery path. Any user who reads the router interface can route through it. The only precondition is that the pool admin has allowlisted the router, which is a natural and expected action for any pool whose curated users need multi-hop routing or slippage protection. No privileged access, no special token behavior, and no admin collusion is required. The condition is self-inflicted by correct, intended admin behavior.

## Recommendation

Pass the original end-user identity through the swap path so the extension can gate on it:

1. **Thread an `originator` field through the extension interface.** The router already stores the original `msg.sender` in transient storage for callback settlement; expose it as an additional argument to `beforeSwap` so extensions can check `allowedSwapper[pool][originator]` rather than `allowedSwapper[pool][router]`.

2. **Key the allowlist on `recipient` instead of `sender` (partial fix only).** For pools where the recipient is always the economically relevant party this narrows the gap, but it does not cover cases where recipient differs from the intended gated actor.

Option 1 is the correct fix: thread the original caller through the extension interface so `SwapAllowlistExtension.beforeSwap` checks the actual end user.

## Proof of Concept

```
Setup
─────
1. Pool admin deploys a pool with SwapAllowlistExtension as beforeSwap hook.
2. Admin allowlists Alice:  setAllowedToSwap(pool, alice, true)
3. Admin allowlists the router: setAllowedToSwap(pool, router, true)
   (required so Alice can use the router)

Attack
──────
4. Bob (not allowlisted) calls:
     MetricOmmSimpleRouter.exactInputSingle({
       pool: pool,
       recipient: bob,
       ...
     })
5. Router calls pool.swap(...) — pool sees msg.sender = router.
6. _beforeSwap is called with sender = router.
7. Extension checks allowedSwapper[pool][router] → true → swap proceeds.
8. Bob receives tokens from the curated pool with no allowlist check on his address.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds
```

Root cause is at `SwapAllowlistExtension.beforeSwap` line 37, where `sender` resolves to the router address rather than the end user whenever the router intermediates the call. [7](#0-6)

  function exactInput(ExactInputParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    _validatePath(params.tokens, params.pools, params.extensionDatas);

    uint256 last = params.pools.length - 1;
    int128 amount = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn);

    for (uint256 i = 0; i <= last; i++) {
      address pool = params.pools[i];
      bool zeroForOne = MetricOmmSwapPath.resolveZeroForOneBitmap(params.zeroForOneBitMap, i);

      _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
      (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
        .swap(
          i == last ? params.recipient : address(this),
          zeroForOne,
          amount,
          MetricOmmSwapPath.openLimit(zeroForOne),
          "",
          params.extensionDatas[i]
        );

      int128 amountInActual = MetricOmmSwapResults.extractAmountIn(zeroForOne, amount0Delta, amount1Delta);
      if (amountInActual < amount) revert InvalidInputAmountAtHop(uint8(i), amountInActual, amount);

      amount = MetricOmmSwapResults.extractAmountOut(zeroForOne, amount0Delta, amount1Delta);
    }

    if (amount <= 0) revert InvalidSwapDeltas();
    amountOut = MetricOmmSwapInputs.int128ToUint128(amount);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
