### Title
BeginBlocker Drains Rewards Pool Without Accounting for Accrued Bonus Obligations, Locking Position Holders Out of All Exit Operations — (`x/tieredrewards/keeper/abci.go`)

### Summary
`topUpBaseRewards` in `abci.go` uses the raw `RewardsPoolName` module-account balance to determine how much to transfer for base-reward top-ups. It never subtracts the total accrued-but-unclaimed bonus rewards owed to all active positions. When a fee-collector shortfall is large enough, the BeginBlocker drains the entire pool in a single block. Every subsequent position lifecycle operation that requires a bonus claim (`TierUndelegate`, `TierRedelegate`, `AddToTierPosition`, `ClearPosition`, `ExitTierWithDelegation`) then fails atomically, locking user funds in positions. During any concurrent redelegation slash the bonus is silently and permanently forfeited.

### Finding Description

`topUpBaseRewards` reads the pool balance and, if the shortfall exceeds it, transfers the entire pool to the distribution module:

```go
poolBalance := k.bankKeeper.GetBalance(ctx, poolAddr, bondDenom)
topUpAmount := shortFallAmount
if poolBalance.Amount.LT(shortFallAmount) {
    topUpAmount = poolBalance.Amount   // entire pool drained
}
err = k.bankKeeper.SendCoinsFromModuleToModule(ctx, types.RewardsPoolName, distributiontypes.ModuleName, ...)
``` [1](#0-0) 

The pool balance at this point includes every token that is already owed as accrued bonus rewards to delegated positions. No subtraction of pending obligations is performed before the transfer.

Every position lifecycle mutation that modifies delegation state calls `claimRewards` first, which calls `processEventsAndClaimBonus`, which calls `sufficientBonusPoolBalance`:

```go
func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
    poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
    poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
    if !poolBalance.IsAllGTE(bonus) {
        return errorsmod.Wrapf(types.ErrInsufficientBonusPool, ...)
    }
    return nil
}
``` [2](#0-1) 

If the pool is empty after the BeginBlocker transfer, this check fails and the entire message is rejected:

```go
pos, _, _, err = ms.claimRewards(ctx, pos)
if err != nil {
    return nil, err   // TierUndelegate, TierRedelegate, AddToTierPosition, ClearPosition, ExitTierWithDelegation all fail here
}
``` [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

Additionally, the ADR documents that the `BeforeRedelegationSlashed` hook silently forfeits bonus when the pool is insufficient to avoid a chain halt. If the pool is drained by the BeginBlocker and a redelegation slash fires, the position holder permanently loses all accrued bonus rewards with no recourse. [8](#0-7) 

### Impact Explanation

**Impact: High.**

1. **Fund lock-up:** Any delegator with accrued bonus rewards cannot undelegate, redelegate, add to their position, clear an exit, or exit with delegation. Their principal is locked in the tier module until governance replenishes the pool. Since the BeginBlocker runs every block and can drain the pool repeatedly, the lock-up can persist indefinitely.

2. **Permanent bonus forfeiture:** If a redelegation slash fires while the pool is empty, `BeforeRedelegationSlashed` silently forfeits the position's entire accrued bonus. The corrupted value is the `RewardsPoolName` module-account balance, which is treated as fully available for base-reward top-ups when a portion of it is already owed as bonus rewards to active positions.

### Likelihood Explanation

**Likelihood: Medium.**

- `TargetBaseRewardsRate > 0` is the expected production configuration (the feature exists to guarantee a minimum staker APY).
- A fee-collector shortfall occurs whenever block fees are below the target rate — a routine condition during low-activity periods.
- When the shortfall exceeds the pool balance, the entire pool is drained in one block.
- The number of positions and their accrued bonus obligations grow over time, making the gap between "pool balance" and "actually available balance" larger.
- Redelegation slashes are infrequent but realistic (validator misbehaviour during an active redelegation window).

### Recommendation

Introduce a module-level accumulator that tracks the total accrued but unclaimed bonus rewards across all active positions (updated on every `processEventsAndClaimBonus` call and decremented on every `SendCoinsFromModuleToAccount`). In `topUpBaseRewards`, subtract this accumulator from the pool balance before computing `topUpAmount`:

```go
pendingBonus := k.GetTotalPendingBonusObligations(ctx)
availableForTopUp := poolBalance.Amount.Sub(pendingBonus)
if !availableForTopUp.IsPositive() {
    return nil
}
topUpAmount = math.MinInt(shortFallAmount, availableForTopUp)
```

This mirrors the fix recommended in the external report: introduce a variable to track pending obligations and subtract it from all balance calculations before allocating funds.

### Proof of Concept

1. Governance sets `TargetBaseRewardsRate = 0.10` (10% APY target).
2. Alice locks 1,000,000 tokens in Tier-1 for 365 days. Over time she accrues 40,000 tokens of bonus rewards (unclaimed).
3. Block fees drop to near zero; the fee-collector shortfall per block exceeds the pool balance.
4. BeginBlocker fires: `poolBalance = 40,000`, `shortFallAmount = 50,000` → `topUpAmount = 40,000` → entire pool transferred to distribution.
5. Alice calls `MsgTierUndelegate` → `claimRewards` → `sufficientBonusPoolBalance(40,000 bonus, pool=0)` → `ErrInsufficientBonusPool` → transaction rejected. Alice cannot exit her position.
6. Simultaneously, Bob has an active redelegation. A validator slash fires → `BeforeRedelegationSlashed` → `processEventsAndClaimBonus` → pool is 0 → bonus silently forfeited. Bob permanently loses his accrued bonus.
