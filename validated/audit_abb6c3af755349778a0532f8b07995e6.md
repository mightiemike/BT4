### Title
Missing Lower Bound on `confirmations` Parameter Allows Zero-Confirmation Proof Acceptance — (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` enforce an upper bound on `confirmations` (must not exceed `gc_threshold`) but impose **no lower bound**. An unprivileged NEAR caller can supply `confirmations = 0`, causing the confirmation-depth guard to trivially pass for any mainchain transaction, bypassing the only security guarantee the function is designed to provide.

### Finding Description
In `verify_transaction_inclusion`, the only bounds check on `args.confirmations` is:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [1](#0-0) 

The subsequent confirmation-depth guard is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [2](#0-1) 

When `args.confirmations == 0`, the right-hand side is `0`, and the expression `(any_u64) >= 0` is always `true` in Rust's unsigned arithmetic. The guard never fires. The function proceeds directly to Merkle proof verification and returns `true` for any transaction present in any mainchain block, regardless of how recently it was included.

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` via `args.into()` after its own coinbase-proof check, so it inherits the same flaw:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

The `confirmations` field in `ProofArgs` and `ProofArgsV2` is a plain `u64` with no enforced minimum at the type or contract level:

```rust
pub struct ProofArgs {
    ...
    pub confirmations: u64,
}
``` [4](#0-3) 

### Impact Explanation
The sole purpose of the `confirmations` parameter is to ensure a Bitcoin transaction has accumulated enough proof-of-work depth to be considered irreversible before a downstream NEAR contract acts on it (e.g., minting wrapped BTC, releasing escrowed funds). With `confirmations = 0`, that guarantee is completely removed. A downstream contract that passes caller-supplied `confirmations` would accept a transaction the moment it appears in any mainchain block — before any reorganization safety window has elapsed — enabling a double-spend: the attacker receives NEAR-side value, then the Bitcoin block is reorganized and the BTC transaction is reversed.

### Likelihood Explanation
The two verification functions are the primary public API for SPV consumers. Any downstream NEAR contract that forwards a user-supplied `confirmations` value (a natural design when the confirmation requirement is configurable per-call) is directly exploitable. The attacker needs only to craft a valid Merkle proof for a freshly-included transaction and pass `confirmations = 0`. No privileged role, leaked key, or social engineering is required.

### Recommendation
Add an explicit lower-bound check at the top of `verify_transaction_inclusion`:

```rust
require!(args.confirmations >= 1, "Confirmations must be at least 1");
```

Alternatively, enforce a protocol-level minimum (e.g., `>= 6` for Bitcoin mainnet) stored in contract state alongside `gc_threshold`, mirroring how the upper bound is already enforced against `gc_threshold`.

### Proof of Concept

1. Deploy the contract on NEAR testnet (Bitcoin feature flag).
2. Submit a single Bitcoin block header containing a target transaction `tx_id`.
3. Call `verify_transaction_inclusion` with:
   - `tx_id` = the transaction hash
   - `tx_block_blockhash` = the block just submitted
   - `tx_index` = correct index
   - `merkle_proof` = valid Merkle path
   - `confirmations = 0`
4. Observe the function returns `true` immediately, with the block having depth 1 (or even depth 0 if the block is the tip).
5. The confirmation-depth `require!` at line 304–308 is never triggered because `N >= 0` is always satisfied for any unsigned `N`. [5](#0-4)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
