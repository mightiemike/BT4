Audit Report

## Title
Swap Allowlist Bypass via Router: `msg.sender` (Router) Checked Instead of Actual User - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is `msg.sender` from the pool's perspective — i.e., the router address. When a user swaps through `MetricOmmSimpleRouter`, the pool's `swap` function is called with `msg.sender == router`, so the extension checks whether the **router** is allowlisted, not the actual end user. Any user who routes through the allowlisted router can bypass a per-user swap allowlist on a curated pool.

## Finding Description

**Root cause — `SwapAllowlistExtension.beforeSwap`:**

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension is called by the pool), and `sender` is the first argument passed by the pool to the extension.

**How the pool calls the extension — `MetricOmmPool.swap`:**

```solidity
// metric-core/contracts/MetricOmmPool.sol:230-240
_beforeSwap(
    msg.sender,   // <-- this is the router address when called via router
    recipient,
    ...
);
```

The pool passes its own `msg.sender` (the router) as `sender` to `_beforeSwap`.

**How `_beforeSwap` forwards it — `ExtensionCalling._beforeSwap`:**

```solidity
// metric-core/contracts/ExtensionCalling.sol:160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(
        IMetricOmmExtensions.beforeSwap,
        (sender, ...)  // sender == router address
    )
);
```

**Router call path — `MetricOmmSimpleRouter.exactInputSingle`:**

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol:72-80
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    ...
);
```

The router calls `pool.swap(...)` directly, so from the pool's perspective `msg.sender == router`.

**Result:** The extension checks `allowedSwapper[pool][router]`. If the pool admin allowlists the router (which is the natural operational setup for a router-accessible curated pool), **every user who calls through the router is allowed**, regardless of whether they are individually permitted. Conversely, if the router is not allowlisted, no router-mediated swap works at all — the allowlist is effectively broken in both directions.

The existing test in `FullMetricExtensionTest.test_allowedSwapSucceeds` confirms this: it allowlists `address(callers[0])` (the `TestCaller` contract, which is the direct `msg.sender` to the pool), not the end user `users[0]`. This demonstrates the design checks the intermediate caller, not the economic actor.

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict swaps to a specific set of addresses is fully bypassed: any unprivileged user can call `MetricOmmSimpleRouter.exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` and execute swaps on the restricted pool, provided the router address is allowlisted. This constitutes a broken core pool functionality (allowlist policy bypass) and a direct admin-boundary break — an unprivileged trader can trade on a pool they are not permitted to access. Severity: **High**.

## Likelihood Explanation
The router is the standard public entry point for swaps. Any pool that uses `SwapAllowlistExtension` and also permits router access (the expected production configuration) is immediately and repeatably exploitable by any unprivileged address. No special preconditions, flash loans, or timing are required. The attacker simply calls the router with a valid swap path targeting the restricted pool.

## Recommendation
`SwapAllowlistExtension.beforeSwap` should gate on the **economic actor** (the original `msg.sender` of the router call), not the intermediate caller. Two options:

1. **Pass the original caller through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a protocol-level convention.
2. **Check `tx.origin` as a fallback** (not recommended for general use, but noted).
3. **Preferred**: Redesign the pool's `swap` interface to pass an explicit `swapper` parameter (the original user) separately from `msg.sender`, and have the extension check that field. Alternatively, the router should be modified to pass the original caller's address in `extensionData`, and the extension should decode and verify it.

## Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension, router is allowlisted, attacker is not
swapExtension.setAllowedToSwap(address(pool), address(router), true);
// attacker is NOT allowlisted

// Attacker calls router directly — succeeds because extension sees router as sender
vm.prank(attacker);
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool: address(pool),
    tokenIn: address(token0),
    recipient: attacker,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    extensionData: "",
    deadline: block.timestamp + 1
}));
// Swap succeeds — attacker bypassed the per-user allowlist
```
