Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the originating user, allowing allowlist bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is always `msg.sender` of the `pool.swap` call. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` at the pool, so the extension checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actualUser]`. A pool admin who allowlists the router to support curated users inadvertently opens the pool to every caller of the router, completely defeating per-user curation.

## Finding Description
In `MetricOmmPool.swap`, the `sender` forwarded to `_beforeSwap` is unconditionally `msg.sender`: [1](#0-0) 

`ExtensionCalling._beforeSwap` passes this value straight through as the `sender` argument to every registered extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

where `msg.sender` is the pool and `sender` is whoever called `pool.swap`. In `MetricOmmSimpleRouter.exactInputSingle`, the router calls the pool directly, passing no user identity: [4](#0-3) 

The router stores the original `msg.sender` only in transient callback context for payment purposes (`_setNextCallbackContext`), but never encodes it into `extensionData` or any field visible to the extension. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][originalUser]`. No existing guard corrects this: the extension has no knowledge of trusted routers and no mechanism to decode the real initiator.

## Impact Explanation
**Scenario A (bypass):** A pool admin allowlists the router so curated users can swap via the periphery. This single entry makes the pool accessible to every user who calls `exactInputSingle`, `exactInput`, or `exactOutput` on the router, regardless of their own allowlist status. Non-allowlisted users execute swaps against a pool explicitly configured to exclude them, draining LP value or violating compliance constraints the admin intended to enforce. This is a direct admin-boundary break and a broken core pool invariant (the allowlist no longer gates the actual economic actor).

**Scenario B (broken functionality):** If the admin does not allowlist the router, all allowlisted users are unable to use the router; their swaps revert with `NotAllowedToSwap`. The pool is only usable via direct `pool.swap` calls, which require the caller to implement `IMetricOmmSwapCallback`, breaking the documented periphery flow for curated pools.

Both scenarios represent broken core swap functionality and/or a direct policy bypass with fund-impacting consequences, meeting Sherlock High severity thresholds.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap entrypoint. Any pool admin deploying a curated pool and allowlisting the router (the natural step to make the pool usable) immediately enables the bypass. The attacker requires no special privileges: they call a public router function with a valid pool address. The condition is trivially reachable and repeatable by any unprivileged user.

## Recommendation
The extension must verify the **economic actor**, not the intermediate contract. The most robust fix is for the router to encode `msg.sender` into `extensionData` before calling the pool, and for `SwapAllowlistExtension` to maintain a registry of trusted routers; when `sender` is a trusted router, the extension decodes and checks the real initiator from `extensionData`. Alternatively, document and enforce at the factory level that `SwapAllowlistExtension` is incompatible with router-mediated swaps (e.g., reject configurations that pair the extension with a non-zero router allowlist entry).

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - allowedSwapper[pool][router] = true   // admin allowlists router for curated users
  - allowedSwapper[pool][alice] = true    // alice is curated
  - allowedSwapper[pool][bob] = false     // bob is NOT curated

Attack:
  bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(recipient, ...)          // msg.sender at pool = router
    → MetricOmmPool._beforeSwap(msg.sender=router, ...)
    → SwapAllowlistExtension.beforeSwap(sender=router, ...)
    → allowedSwapper[pool][router] == true  → check passes
    → bob's swap executes against the curated pool

Result: bob, a non-allowlisted user, successfully swaps against a pool
        configured to exclude him.
```
