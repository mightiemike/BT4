Audit Report

## Title
`PriceVelocityGuardExtension.beforeSwap` unconditionally resets `lastMidPriceX64` before the velocity check, enabling intra-block oracle-stepping to bypass the per-block price cap Рђћ (`File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary

`PriceVelocityGuardExtension.beforeSwap` writes the current mid-price into `lastMidPriceX64` and `block.number` into `lastUpdateBlock` unconditionally before performing the velocity check. For same-block swaps (`blockDiff = 0`), each swap is checked only against the immediately preceding swap's mid-price, not the block-start price. An attacker who interleaves permissionless Pyth Lazer oracle pushes with pool swaps in a single block can step the reference anchor by at most `maxChangePerBlockE18` per swap, achieving a cumulative intra-block price movement of N ├Ќ `maxChangePerBlockE18` while every individual check passes. LPs suffer direct adverse-selection losses equal to the uncapped total price movement.

## Finding Description

In `PriceVelocityGuardExtension.beforeSwap`, lines 57РђЊ58 unconditionally overwrite `lastMidPriceX64` and `lastUpdateBlock` before the velocity check at lines 60РђЊ75:

```solidity
// Lines 53-75
PriceVelocityState storage s = priceVelocityState[pool_];
uint128 prevMid = s.lastMidPriceX64;
uint64 prevBlock = s.lastUpdateBlock;

s.lastMidPriceX64 = midPrice;          // Рєљ anchor reset BEFORE check
s.lastUpdateBlock = uint64(block.number);

if (prevMid != 0) {
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        uint256 blockDiff = block.number - prevBlock;
        uint256 delta = midPrice > prevMid ? uint256(midPrice - prevMid) : uint256(prevMid - midPrice);
        uint256 changeE18 = (delta * 1e18) / uint256(prevMid);
        uint256 actualSq = changeE18 * changeE18;
        uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
        if (actualSq > allowedSq) revert PriceVelocityExceeded(actualSq, allowedSq);
    }
}
```

For two swaps in the same block, `blockDiff = block.number - prevBlock = 0`, so `allowedSq = maxChange┬▓`. The second swap is checked against the first swap's mid-price (already stored), not the block-start price. This means the guard measures only the incremental step between consecutive swaps, not the cumulative movement from the block's opening price.

The Pyth Lazer oracle used by the pool is a push model: `LazerConsumer._verifyAndStore` accepts any signed update whose per-feed timestamp is strictly newer than the stored one (`if (ts.isAfter(__data[feedId].timestampMs))`). This is permissionless Рђћ anyone holding valid signed price updates (publicly available from Pyth's price service) can push them on-chain. Multiple updates with increasing timestamps can be pushed within a single block.

The pool's `MetricOmmPool.swap` calls `_getBidAndAskPriceX64()` fresh on every invocation, so each swap reads the current oracle state. The `nonReentrant` guard uses transient storage that clears between transactions; sequential (non-nested) swap calls within one block are not blocked.

## Impact Explanation

The velocity guard is the LP's primary protection against oracle-speed manipulation and adverse selection. By stepping through intermediate Pyth price updates in a single block, an attacker executes swaps at a final price that is N ├Ќ `maxChangePerBlockE18` away from the block-start price Рђћ a movement the guard was explicitly configured to block. LPs are forced to trade at prices far outside the configured safety envelope with no opportunity to react or withdraw, resulting in direct principal loss through adverse selection. This matches the "bad-price execution" and "direct loss of LP principal" allowed impact categories.

**Severity: High** Рђћ direct, repeatable LP principal loss through a bypassed core protection hook.

## Likelihood Explanation

- Pyth Lazer's push model is permissionless: any actor can submit valid signed price updates obtained from Pyth's public price service without any privileged role.
- When the real market price moves significantly within a block's time window, multiple valid intermediate Pyth updates with increasing timestamps exist and are publicly available.
- The attacker needs no flash loan, no special token, and no privileged role Рђћ only the ability to call `swap()` and push Pyth updates sequentially within one block, achievable via a single purpose-built contract.
- On L2s (Arbitrum, Base) with ~250 ms blocks, many Pyth updates can accumulate per block, widening the attack window further.
- The attack is repeatable across blocks.

## Recommendation

Track the block-start reference price separately. Only advance `lastMidPriceX64` once per block (when `block.number > lastUpdateBlock`). All velocity checks within the same block must compare against the price recorded at the start of that block:

```solidity
PriceVelocityState storage s = priceVelocityState[pool_];
uint128 prevMid = s.lastMidPriceX64;
uint64 prevBlock = s.lastUpdateBlock;

// Only update the reference when entering a new block
if (block.number > prevBlock) {
    s.lastMidPriceX64 = midPrice;
    s.lastUpdateBlock = uint64(block.number);
}
// Always check against the block-start reference (prevMid)
if (prevMid != 0) { ... }
```

This ensures that no matter how many swaps occur in a single block, the velocity check always measures the total movement from the block-start price.
