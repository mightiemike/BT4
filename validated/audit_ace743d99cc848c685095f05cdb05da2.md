Looking at the tiered rewards module's position lifecycle, specifically the `deletePosition` guard and the `WithdrawFromTier` full-exit path.

### Title
Position Deletion Ignores Pending Unbonding Delegations During Full Exit - (File: `x/tieredrewards/keeper/position.go`)

### Summary
The `deletePosition` function only checks for an active (bonded) delegation before allowing a position to be deleted. It does not check for pending unbonding delegations. This means a user can call `WithdrawFromTier` while their position's funds are still locked in the staking module's unbonding queue, causing the position to be deleted before the unbonding completes. When unbonding finishes, the staking module sends the tokens to the now-orphaned position delegator address, where they are permanently inaccessible.

### Finding Description

The guard inside `deletePosition` reads:

```go
del, err := k.getDelegation(ctx, pos.DelegatorAddress)
if err != nil {
    return err
}
if del != nil {
    return errorsmod.Wrapf(types.ErrPositionDelegated, ...)
}
``` [1](#0-0) 

`getDelegation` returns the active staking `Delegation` object. Once `TierUndelegate` is called, the active delegation is removed and replaced by an `UnbondingDelegation` entry in the staking module. At that point `getDelegation` returns `nil`, so the guard above passes — even though the user's tokens are still locked in the unbonding queue for the standard 21-day period.

The `WithdrawFromTier` handler then:
1. Sweeps only the **spendable bank balance** of the position's delegator address (which is zero while unbonding is in progress).
2. Calls `deletePosition`, which succeeds because there is no active delegation. [2](#0-1) 

After the unbonding period expires, the Cosmos SDK staking module transfers the unbonded tokens to the position's delegator address. But the position record has already been deleted — there is no `WithdrawFromTier` path left to recover those tokens, and the delegator address is a derived account with no private key.

The system is aware that unbonding delegations carry value: `getUndelegatedAmount` explicitly queries `GetUnbondingDelegations` to compute a position's amount when it is not actively delegated. [3](#0-2) 

Yet this awareness is never applied as a guard inside `deletePosition`.

### Impact Explanation

A user who calls `TierUndelegate` and then immediately calls `WithdrawFromTier` (before the unbonding period completes) will have their position deleted while their principal is still locked in the staking module. When unbonding completes the tokens land in the orphaned position delegator address — a derived account with no private key and no associated position record — making them permanently unrecoverable. This is a direct, irreversible loss of user principal.

### Likelihood Explanation

The trigger is a normal two-step user flow (`TierUndelegate` → `WithdrawFromTier`) executed out of the expected temporal order. No privileged access is required. A user who misunderstands the unbonding lifecycle, or who is front-run/griefed into calling `WithdrawFromTier` early, can hit this path. The 21-day unbonding window provides a wide opportunity.

### Recommendation

Add an unbonding-delegation check inside `deletePosition` (or inside `validateWithdrawFromTier`) before allowing position deletion:

```go
ubds, err := k.stakingKeeper.GetUnbondingDelegations(ctx, delAddr, 1)
if err != nil {
    return err
}
if len(ubds) > 0 {
    return errorsmod.Wrapf(types.ErrPositionDelegated,
        "cannot delete position %d: unbonding delegation still pending", pos.Id)
}
```

This mirrors the fix suggested in the reference report: extend the "full exit" guard to cover **all** outstanding obligations (active delegation **and** unbonding delegation), not just the primary one.

### Proof of Concept

1. Alice calls `LockTier` → position created, funds delegated.
2. Alice calls `TierUndelegate` → active delegation removed, `UnbondingDelegation` entry created in staking module; `getDelegation` now returns `nil` for the position's delegator address.
3. Alice (or anyone) calls `WithdrawFromTier` immediately.
   - `validateWithdrawFromTier` runs (no unbonding check in `deletePosition`).
   - `SpendableCoins(delAddr)` returns zero (funds are in the unbonding queue, not the bank).
   - `deletePosition` checks `getDelegation` → `nil` → guard passes → position record deleted.
4. 21 days later the staking module's `AfterUnbondingInitiated` / completion hook sends the unbonded tokens to `delAddr`.
5. `delAddr` is a derived account with no private key and no associated position. Tokens are permanently stuck. [4](#0-3) [5](#0-4)

func (k Keeper) deletePosition(ctx context.Context, pos types.Position, update *ValidatorTransition) error {
	owner, err := sdk.AccAddressFromBech32(pos.Owner)
	if err != nil {
		return errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid owner address")
	}

	delAddr, err := sdk.AccAddressFromBech32(pos.DelegatorAddress)
	if err != nil {
		return errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")
	}

	del, err := k.getDelegation(ctx, pos.DelegatorAddress)
	if err != nil {
		return err
	}
	if del != nil {
		return errorsmod.Wrapf(types.ErrPositionDelegated, "cannot delete position %d: still has active delegation to %s", pos.Id, del.ValidatorAddress)
	}

	if err := k.removeBaseRewardsRouting(ctx, delAddr, owner); err != nil {
		return err
	}

	// Defensive
	if err := k.deletePositionRedelegationMappings(ctx, pos.Id); err != nil {
		return err
	}

	if err := k.Positions.Remove(ctx, pos.Id); err != nil {
		return err
	}
	if err := k.PositionsByOwner.Remove(ctx, collections.Join(owner, pos.Id)); err != nil {
		return err
	}
	if err := k.PositionsByTier.Remove(ctx, collections.Join(pos.TierId, pos.Id)); err != nil {
		return err
	}
	if err := k.decreasePositionCountForTier(ctx, pos.TierId); err != nil {
		return err
	}

	if update == nil {
		return nil
	}
	return k.reindexPositionCountByValidator(ctx, update.PreviousAddress, "")
}

func (ms msgServer) WithdrawFromTier(ctx context.Context, msg *types.MsgWithdrawFromTier) (*types.MsgWithdrawFromTierResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	pos, err := ms.getPositionState(ctx, msg.PositionId)
	if err != nil {
		return nil, err
	}

	if err := ms.validateWithdrawFromTier(ctx, pos, msg.Owner); err != nil {
		return nil, err
	}

	ownerAddr, err := sdk.AccAddressFromBech32(pos.Owner)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid owner address")
	}

	delAddr, err := sdk.AccAddressFromBech32(pos.DelegatorAddress)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")
	}

	balances := ms.bankKeeper.SpendableCoins(ctx, delAddr)
	if !balances.IsZero() {
		if err := ms.bankKeeper.SendCoins(ctx, delAddr, ownerAddr, balances); err != nil {
			return nil, err
		}
	}

	if err := ms.deletePosition(ctx, pos.Position, nil); err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventPositionWithdrawn{
		Position: pos.Position,
		Amount:   balances,
	}); err != nil {
		return nil, err
	}

	return &types.MsgWithdrawFromTierResponse{
		Amount: balances,
	}, nil
}

func (k Keeper) getUndelegatedAmount(ctx context.Context, delegatorAddress string) (math.Int, error) {
	delAddr, err := sdk.AccAddressFromBech32(delegatorAddress)
	if err != nil {
		return math.Int{}, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")
	}
	ubds, err := k.stakingKeeper.GetUnbondingDelegations(ctx, delAddr, 1)
	if err != nil {
		return math.Int{}, err
	}
	if len(ubds) > 0 && len(ubds[0].Entries) > 0 {
		return ubds[0].Entries[0].Balance, nil
	}
	bondDenom, err := k.stakingKeeper.BondDenom(ctx)
	if err != nil {
		return math.Int{}, err
	}
	return k.bankKeeper.SpendableCoins(ctx, delAddr).AmountOf(bondDenom), nil
}
