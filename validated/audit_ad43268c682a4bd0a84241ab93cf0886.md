Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks Position Owner Instead of Payer, Allowing Allowlist Bypass via Operator Pattern — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument and gates on `owner` (the position holder) instead of the actual economic actor paying for the deposit. Because `MetricOmmPool.addLiquidity` explicitly supports an operator pattern where `msg.sender` (the payer) differs from `owner`, and `MetricOmmPoolLiquidityAdder` exposes this as a public entrypoint, any unprivileged user can bypass the allowlist by nominating any allowlisted address as `owner` while paying themselves. The pool admin's curation policy is fully circumvented.

## Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` ignores its first parameter (`sender`) and checks only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

The contract's own NatSpec, mapping name (`allowedDepositor`), and setter (`setAllowedToDeposit`) all indicate the intended gate is the depositing economic actor, not the position holder: [2](#0-1) 

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to the extension hook: [3](#0-2) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, ...)` accepts an arbitrary `owner` while always using `msg.sender` as the payer stored in transient context. `_validateOwner` only rejects `address(0)`: [4](#0-3) 

The full call chain for the exploit:

1. Bob calls `LiquidityAdder.addLiquidityExactShares(pool, alice, salt, deltas, ...)` where Alice is allowlisted and Bob is not.
2. The adder stores Bob as `payer` in transient context and calls `pool.addLiquidity(alice, ...)` with `msg.sender = LiquidityAdder`.
3. The pool calls `_beforeAddLiquidity(sender=LiquidityAdder, owner=alice, ...)`.
4. The extension evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
5. The pool's callback fires; the adder pulls tokens from Bob (the stored payer): [5](#0-4) 

6. Alice receives the LP position; Bob's funds are now in the pool. The allowlist check never evaluated Bob's address.

The wrong value is the extension decision: `allowedDepositor[msg.sender][owner]` passes when it should be checking `allowedDepositor[msg.sender][sender]`, where `sender` is the actual payer/caller.

## Impact Explanation

The pool admin's deposit curation policy — restricting which economic actors can provide liquidity (e.g., for KYC/AML compliance or curated LP pools) — is fully bypassed. Any unprivileged user can inject funds into the pool by nominating any allowlisted address as `owner`. This is an admin-boundary break: the pool admin's access control is circumvented via an unprivileged public path. The `DepositAllowlistExtension` provides no actual restriction on who pays for deposits, rendering it ineffective as a gate.

## Likelihood Explanation

`MetricOmmPoolLiquidityAdder` is the documented production periphery for EOA deposits and explicitly exposes the `owner ≠ msg.sender` overload as a supported public entrypoint. No special privileges, malicious setup, or non-standard tokens are required. Any user who knows a single allowlisted address can execute this in one transaction. Allowlisted addresses are discoverable on-chain via the `AllowedToDepositSet` event or by reading `allowedDepositor` storage.

## Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual caller/payer) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is instead to gate by position owner (not payer), rename the mapping and setter to `allowedOwner`/`setAllowedOwner` and update documentation accordingly so pool admins are not misled about what the allowlist enforces.

## Proof of Concept

```
Setup:
  - Pool configured with DepositAllowlistExtension
  - Alice (0xAlice) is allowlisted: allowedDepositor[pool][alice] = true
  - Bob (0xBob) is NOT allowlisted

Attack:
  1. Bob approves LiquidityAdder for token0/token1
  2. Bob calls:
       LiquidityAdder.addLiquidityExactShares(
           pool,
           alice,   // owner = allowlisted address
           salt,
           deltas,
           maxAmount0,
           maxAmount1,
           ""
       )
  3. Pool calls _beforeAddLiquidity(sender=LiquidityAdder, owner=alice, ...)
  4. Extension checks allowedDepositor[pool][alice] → true → no revert
  5. Callback pulls tokens from Bob (payer in transient context)
  6. Alice receives position shares; Bob's funds are in the pool

Result:
  - Bob (unauthorized) has deposited into a curated pool
  - DepositAllowlistExtension did not block him
  - Pool admin's curation policy is violated
```
