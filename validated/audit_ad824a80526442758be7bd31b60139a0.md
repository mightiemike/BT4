### Title
Deposited Tokens Permanently Locked When Slow Mode `DepositCollateral` Transaction Fails Silently - (File: `core/contracts/Endpoint.sol`)

---

### Summary

In `Endpoint.sol`, the `depositCollateral` / `depositCollateralWithReferral` flow transfers user tokens to the Clearinghouse **before** queuing the corresponding `DepositCollateral` slow mode transaction. If that slow mode transaction later fails during execution, the failure is silently discarded — the code comment `// try return funds now removed` explicitly acknowledges that a previously existing refund mechanism was deleted. The user's tokens remain in the Clearinghouse, their balance is never credited, and no on-chain recovery path exists.

---

### Finding Description

The deposit flow is split into two phases:

**Phase 1 — Token custody (immediate):**
`depositCollateralWithReferral` calls `handleDepositTransfer`, pulling tokens from `msg.sender` into the Clearinghouse, then queues a `DepositCollateral` slow mode transaction with a 3-day delay. [1](#0-0) 

**Phase 2 — Balance credit (deferred, up to 3 days later):**
`_executeSlowModeTransaction` wraps `processSlowModeTransaction` in a `try/catch`. Any non-OOG revert is silently swallowed. The comment at line 226 reads:

```
// try return funds now removed
``` [2](#0-1) 

If `processSlowModeTransaction` reverts — for any reason — the catch block does nothing. The tokens already transferred in Phase 1 are permanently stranded in the Clearinghouse with no credit to the user's subaccount balance and no refund path.

The `processSlowModeTransactionImpl` handler for `DepositCollateral` calls `clearinghouse.depositCollateral(txn)`, which can revert due to product state changes, engine invariant violations, or subaccount registration failures that occur in the 3-day window between deposit and execution. [3](#0-2) 

---

### Impact Explanation

A user who calls `depositCollateral` has their tokens taken immediately. If the corresponding slow mode transaction fails at execution time, the user:

- Receives **zero balance credit** in the Clearinghouse
- Has **no on-chain function** to reclaim their tokens (no emergency withdrawal, no refund path)
- Suffers **total loss** of the deposited amount

The stranded tokens accumulate in the Clearinghouse contract with no accounting entry, making them unrecoverable without privileged admin intervention (`removeLiquidity` on the WithdrawPool, or `withdrawFromDirectDepositV1` on the ContractOwner — neither of which is accessible to the affected user). [4](#0-3) 

---

### Likelihood Explanation

The 3-day slow mode delay creates a meaningful window for state divergence. Concrete failure triggers include:

1. **Product delisting**: A spot product's token address is zeroed or the product is removed between deposit and execution, causing `clearinghouse.depositCollateral` to revert.
2. **Engine upgrade**: The `SpotEngine` implementation is replaced via the proxy upgrade system between deposit and execution, causing a state mismatch.
3. **Subaccount registration edge case**: `_recordSubaccount` reverts due to a bitmap overflow or subaccount limit.
4. **Sanctions race**: The `depositCollateralWithReferral` path (used by `DirectDepositV1.creditDeposit`) does not re-check sanctions at execution time, but other revert conditions in `clearinghouse.depositCollateral` can still trigger.

Any of these conditions — which are realistic over a 3-day window in a live protocol — produces the loss. [5](#0-4) 

---

### Recommendation

Restore a fund-return mechanism in the `catch` block of `_executeSlowModeTransaction` for `DepositCollateral` transactions. Specifically:

- Record the `(token, sender, amount)` tuple before the `try` call.
- In the `catch` block, if the failed transaction was a `DepositCollateral`, transfer the tokens back to the original depositor.

Alternatively, restructure the deposit flow so that token custody is only taken **inside** `processSlowModeTransaction` (i.e., atomically with the balance credit), eliminating the window entirely.

---

### Proof of Concept

```
1. User calls depositCollateral(subaccountName, productId, amount)
   → handleDepositTransfer pulls `amount` tokens from user into Clearinghouse  [Endpoint.sol:144-148]
   → SlowModeTx queued with executableAt = now + 3 days               [Endpoint.sol:152-165]

2. During the 3-day window, a state change occurs that will cause
   clearinghouse.depositCollateral(txn) to revert
   (e.g., product reconfiguration, engine upgrade, subaccount limit hit)

3. Anyone calls executeSlowModeTransaction()
   → _executeSlowModeTransaction() runs
   → try this.processSlowModeTransaction(txn.sender, txn.tx) {}
   → processSlowModeTransactionImpl → clearinghouse.depositCollateral → REVERT
   → catch block reached: "// try return funds now removed"
   → failure silently discarded                                        [Endpoint.sol:207-227]

4. Result:
   - User's tokens are in the Clearinghouse
   - User's subaccount balance = 0 (never credited)
   - No on-chain function exists for the user to recover their tokens
   - Total loss of deposited amount
``` [6](#0-5)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
```
