### Title
Missing Minimum Confirmation Enforcement Allows Zero-Confirmation SPV Proof Acceptance - (File: contract/src/lib.rs)

### Summary
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations` value of `0` without any lower-bound enforcement. This is the direct analog of M-06: just as hardcoding `amountOutMinimum = 0` accepts any swap output, accepting `confirmations = 0` accepts any block depth — including the chain tip — as "confirmed," completely bypassing the protocol's reorganization-safety guarantee.

### Finding Description
Both public verification functions enforce only an upper bound on `confirmations`:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [1](#0-0) 

No lower bound (`>= 1`) is enforced. The subsequent depth check is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [2](#0-1) 

When `args.confirmations == 0`, the right-hand side is `0`, and any `u64` value satisfies `>= 0`. The check always passes regardless of how recently the block was submitted. A transaction in the current chain tip — with zero depth — is returned as verified.

`verify_transaction_inclusion_v2` delegates unconditionally to the deprecated function after its own coinbase-proof check, inheriting the same flaw:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

The `confirmations` field in `ProofArgs` and `ProofArgsV2` is a plain `u64` with no documented minimum: [4](#0-3) 

### Impact Explanation
Any downstream NEAR contract (bridge, DeFi protocol, cross-chain application) that calls `verify_transaction_inclusion{_v2}` with `confirmations = 0` receives `true` for a Bitcoin transaction that exists in any mainchain block, including the most recent one. That block can still be reorganized away. A bridge contract releasing funds on this result is vulnerable to a double-spend: the attacker submits a Bitcoin transaction, immediately calls the NEAR bridge with `confirmations = 0`, receives the bridged asset, and then the Bitcoin transaction is reorganized out of the chain.

**Impact: High** — proof-verification result is corrupted; downstream asset-release decisions are made on unfinalized Bitcoin state.

### Likelihood Explanation
**Likelihood: Low** — requires a caller to deliberately pass `confirmations = 0`. However, no privileged role is needed; any unprivileged NEAR account can call either public function. A malicious downstream contract or a user interacting directly with the light client API is a realistic attacker path.

### Recommendation
Enforce a minimum confirmation count at the contract level. Add a `require!` immediately after the upper-bound check in `verify_transaction_inclusion`:

```rust
require!(args.confirmations >= 1, "Confirmations must be at least 1");
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
```

Alternatively, expose a configurable `min_confirmations` field in contract state (analogous to introducing `amountOutMinimum` as recommended in M-06) so the operator can enforce a chain-appropriate floor (e.g., 6 for Bitcoin mainnet).

### Proof of Concept
1. Deploy the contract with any valid genesis and a few submitted blocks.
2. Construct `ProofArgs` with a valid `tx_id`, `tx_block_blockhash` pointing to the chain tip, a correct `merkle_proof`, and `confirmations = 0`.
3. Call `verify_transaction_inclusion` (or `verify_transaction_inclusion_v2` with matching coinbase proof).
4. The function returns `true` despite the block having zero depth — the confirmation guard is fully bypassed. [5](#0-4)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
```
