### Title
Missing Access Control on `liquidateSubaccountImpl` Allows Unauthorized Direct Liquidation Bypassing Endpoint Signature Verification — (File: `core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`ClearinghouseLiq.liquidateSubaccountImpl` is declared `external` with no `onlyEndpoint` modifier. Any unprivileged caller can invoke it directly, supplying an arbitrary `txn.sender` subaccount without owning it or providing a valid EIP-712 signature. This bypasses the Endpoint's signature-verification and sequencer-ordering gate and allows unauthorized mutation of any subaccount's balances through the liquidation accounting path.

---

### Finding Description

`ClearinghouseLiq` inherits `EndpointGated`, which defines the `onlyEndpoint` modifier that restricts callers to the deployed `Endpoint` contract. [1](#0-0) 

Every other state-mutating function in the settlement layer that is meant to be called only through the Endpoint carries this modifier (e.g., `burnNlp`, `forceRebalanceNlpPool` in `Clearinghouse.sol`). [2](#0-1) 

`liquidateSubaccountImpl`, however, is declared `external` with no modifier at all: [3](#0-2) 

The function accepts a fully caller-controlled `IEndpoint.LiquidateSubaccount` struct. The field `txn.sender` — which identifies the liquidator subaccount whose balances are credited and debited — is never verified against `msg.sender` or any signature inside this function. In the normal Endpoint flow, the sequencer submits an EIP-712-signed transaction and the Endpoint verifies the signature before dispatching to the clearinghouse. Calling `liquidateSubaccountImpl` directly skips that entire path.

The internal checks present (`isUnderMaintenance`, `txn.sender != txn.liquidatee`, `!isIsolatedSubaccount(txn.sender)`) are business-logic guards, not access-control guards. They do not prevent an attacker from impersonating an arbitrary subaccount as the liquidator. [4](#0-3) 

The liquidation payment path then modifies the balances of both `txn.sender` and `txn.liquidatee` through `_handleLiquidationPayment`: [5](#0-4) 

---

### Impact Explanation

An attacker can:

1. **Impersonate any subaccount as liquidator** — set `txn.sender` to a victim's subaccount without owning it or holding its private key. The liquidation accounting credits the liquidation discount to that subaccount and debits `liquidationFees` from it, mutating the victim's balance without authorization.

2. **Bypass sequencer ordering and Endpoint signature verification** — call `liquidateSubaccountImpl` directly to execute liquidations outside the sequencer queue, enabling front-running of liquidation opportunities or circumventing any sequencer-level rate limits.

3. **Force a subaccount to take on an unwanted position** — in the perp liquidation branch, `txn.sender` receives `txn.amount` of the liquidated position. An attacker can force this position onto a victim's subaccount, potentially degrading its health.

The corrupted state is concrete: `spotEngine` and `perpEngine` balance mappings for `txn.sender` and `txn.liquidatee` are written without the owner of `txn.sender` ever authorizing the operation. [6](#0-5) 

---

### Likelihood Explanation

The function is `external` on a deployed contract with no precondition tied to `msg.sender`. The only real precondition is that `txn.liquidatee` must be under maintenance health — a condition that is observable on-chain by any watcher. Any liquidatable position is therefore a trigger. No privileged access, leaked keys, or governance capture is required.

---

### Recommendation

Add the `onlyEndpoint` modifier to `liquidateSubaccountImpl`, consistent with every other state-mutating settlement function in the codebase:

```solidity
function liquidateSubaccountImpl(IEndpoint.LiquidateSubaccount calldata txn)
    external
    onlyEndpoint   // <-- add this
{
```

This ensures the function is only reachable through the Endpoint's signature-verification and sequencer-ordering pipeline, matching the security model of `burnNlp`, `forceRebalanceNlpPool`, and all other `onlyEndpoint`-gated functions. [7](#0-6) 

---

### Proof of Concept

```solidity
// Attacker calls ClearinghouseLiq directly — no signature, no Endpoint
function testUnauthorizedLiquidation() external {
    IEndpoint.LiquidateSubaccount memory txn = IEndpoint.LiquidateSubaccount({
        sender:         victimSubaccount,   // attacker does NOT own this subaccount
        liquidatee:     underwaterSubaccount,
        productId:      targetProductId,
        isEncodedSpread: false,
        amount:         liquidationAmount,
        nonce:          0
    });
    // Direct call — bypasses Endpoint.submitTransactions and all EIP-712 checks
    clearinghouseLiq.liquidateSubaccountImpl(txn);
    // victimSubaccount's balances are now mutated without its owner's consent
}
```

The call succeeds as long as `underwaterSubaccount` is under maintenance health, which is publicly verifiable. No private key for `victimSubaccount` is needed. [8](#0-7) [1](#0-0)

    function liquidateSubaccountImpl(IEndpoint.LiquidateSubaccount calldata txn)
        external
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.sender != txn.liquidatee, ERR_UNAUTHORIZED);
        require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
        require(
            txn.liquidatee != X_ACCOUNT && txn.liquidatee != N_ACCOUNT,
            ERR_NOT_LIQUIDATABLE
        );
        require(
            txn.productId != QUOTE_PRODUCT_ID,
            ERR_INVALID_LIQUIDATION_PARAMS
        );

        ISpotEngine spotEngine = ISpotEngine(
            address(engineByType[IProductEngine.EngineType.SPOT])
        );
        IPerpEngine perpEngine = IPerpEngine(
            address(engineByType[IProductEngine.EngineType.PERP])
        );

        if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
            if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.liquidatee);
            }
            return;
        }

        if (
            (txn.amount < 0) &&
            (txn.isEncodedSpread ||
                address(productToEngine[txn.productId]) == address(spotEngine))
        ) {
            // when it's spread or spot liquidation, we need to make sure the liquidatee has
            // enough quote to buyback the liquidated amount.
            _assertCanLiquidateLiability(txn, spotEngine, perpEngine);
            _settlePositivePerpPnl(txn, spotEngine, perpEngine);
        }

        _assertLiquidationAmount(txn, spotEngine, perpEngine);

        // beyond this point, we can be sure that we can liquidate the entire
        // liquidation amount knowing that the insurance fund will remain solvent
        // subsequently we can just blast the remainder of the liquidation and
        // cover the quote balance from the insurance fund at the end
        _handleLiquidationPayment(txn, spotEngine, perpEngine);
    }
