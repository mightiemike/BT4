Audit Report

## Title
`OracleValueStopLossExtension` Uses Arithmetic Mid Instead of Geometric Mid, Miscalibrating the LP Value Guard - (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid price as the arithmetic mean of bid and ask prices, while the protocol's canonical mid-price function `SwapMath.midAndSpreadFeeX64FromBidAsk` and `PriceVelocityGuardExtension` both use the geometric mean (`sqrt(bid * ask)`). By the AM-GM inequality, the arithmetic mean is always ≥ the geometric mean, causing `metricToken0` to be systematically understated and its watermark to be set lower than intended. This weakens the stop-loss guard for the `zeroForOne = true` direction, allowing token1 to drain from the pool beyond the LP's configured drawdown threshold without triggering a revert.

## Finding Description

In `_afterSwapOracleStopLoss`, the mid price is computed as:

```solidity
// OracleValueStopLossExtension.sol line 218
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
``` [1](#0-0) 

The protocol's canonical function `SwapMath.midAndSpreadFeeX64FromBidAsk` uses the geometric mean:

```solidity
// SwapMath.sol line 70
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);
``` [2](#0-1) 

`PriceVelocityGuardExtension.beforeSwap` calls `SwapMath.midAndSpreadFeeX64FromBidAsk` for its own mid-price computation, confirming the geometric mean is the protocol-wide standard for oracle bid/ask mid-price derivation: [3](#0-2) 

The two per-share value metrics computed in `_metrics` are: [4](#0-3) 

- `metricToken0 = t0*SCALE/shares + (t1 * Q64 / mid) * SCALE / shares`
- `metricToken1 = (t0 * mid / Q64) * SCALE / shares + t1*SCALE/shares`

Because `arithmeticMid ≥ geometricMid` (AM-GM), the inflated mid has opposite effects:

| Metric | Effect of inflated mid | Guard consequence |
|---|---|---|
| `metricToken0` | `t1 * Q64 / mid` is smaller → metric understated | Watermark set too low; drawdown floor too low; guard for `zeroForOne=true` (token1 outflow) is less sensitive |
| `metricToken1` | `t0 * mid / Q64` is larger → metric overstated | Watermark set too high; drawdown floor too high; guard for `zeroForOne=false` (token0 outflow) is more sensitive |

`_checkAndUpdateWatermarks` then compares these miscalculated metrics against miscalculated watermarks: [5](#0-4) 

The breach check `metric < (hwm * floorMultiplier) / E6` fires against a watermark that was set too low (for `metricToken0`), so the floor is lower than intended and swaps that drain token1 past the configured drawdown threshold are not blocked.

## Impact Explanation

The contract's own NatSpec states that `metricToken0` breach blocks `zeroForOne == true` (token1 outflow): [6](#0-5) 

Because the arithmetic mid is always ≥ the geometric mid, `metricToken0` is always understated. The watermark is set lower than it should be, and the drawdown floor is lower than intended. Swaps that drain token1 past the configured drawdown threshold are not blocked, causing LPs to lose principal beyond the protection they configured. The magnitude scales with the oracle spread: for a 10% bid/ask spread the arithmetic mid exceeds the geometric mid by ~0.125%; for a 50% spread the error reaches ~3.2%. This constitutes a direct loss of LP principal above the configured protection threshold — a medium-severity impact under Sherlock standards, affecting every pool deploying this extension with a non-zero `drawdownE6`.

## Likelihood Explanation

Every pool that deploys `OracleValueStopLossExtension` with a non-zero `drawdownE6` is affected on every swap that traverses a bin. No special attacker action is required — the miscalculation is structural and fires on every normal swap. Any unprivileged user who swaps `zeroForOne = true` benefits from the weakened guard without any privileged access.

## Recommendation

Replace the arithmetic mean with the geometric mean, consistent with the rest of the protocol:

```solidity
// Before (incorrect):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (correct):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
```

This matches the formula used by `PriceVelocityGuardExtension` and the protocol's canonical `SwapMath` library, ensuring the stop-loss evaluates LP value at the same oracle mid price the protocol uses everywhere else.

## Proof of Concept

Given `bid = 0.9 * Q64`, `ask = 1.1 * Q64` (≈10% spread), `t0 = t1 = 1000`, `shares = 1000`, `drawdownE6 = 50_000` (5%):

- **Geometric mid** (correct): `sqrt(0.9 * 1.1) * Q64 = sqrt(0.99) * Q64 ≈ 0.99499 * Q64`
- **Arithmetic mid** (used): `(0.9 + 1.1)/2 * Q64 = 1.0 * Q64`

`metricToken0` at geometric mid: `1000 + 1000/0.99499 ≈ 2005`

`metricToken0` at arithmetic mid (what the contract computes): `1000 + 1000/1.0 = 2000`

The watermark is set to 2000 instead of 2005. The drawdown floor is `2000 * 0.95 = 1900` instead of `2005 * 0.95 = 1904.75`. A subsequent swap that reduces `metricToken0` to 1902 would correctly trigger the guard at the geometric watermark (1902 < 1904.75) but passes silently at the arithmetic watermark (1902 > 1900), allowing token1 to drain past the LP's configured protection.

A Foundry unit test can reproduce this by: (1) deploying `OracleValueStopLossExtension` with `drawdownE6 = 50_000`, (2) calling `afterSwap` with `bid = 0.9 * Q64`, `ask = 1.1 * Q64`, `t0 = t1 = 1000`, (3) observing the stored watermark is 2000 rather than 2005, and (4) confirming a subsequent swap reducing `metricToken0` to 1902 does not revert despite being below the correct geometric floor.
