### Title
Incorrect Spread Health Adjustment Due to Aggregated Price-Weight Multiplication â€” (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.getHealth()` computes the spread health adjustment by summing `spotPrice + perpPrice` and multiplying by a single averaged weight delta. This is the same class of arithmetic error as the Vader TwapOracle bug: values that must be multiplied per-component before summing are instead summed first and then multiplied together, producing an incorrect result.

---

### Finding Description

In `Clearinghouse.getHealth()`, after accumulating individual spot and perp health contributions, the code applies a spread bonus for hedged positions (long spot + short perp):

```solidity
// core/contracts/Clearinghouse.sol lines 125â€“135
int128 existingWeight = (spotCoreRisk.longWeight +
    perpCoreRisk.longWeight) / 2;
int128 spreadWeight = RiskHelper._getSpreadWeightX18(
    perpCoreRisk,
    spotCoreRisk,
    healthType
);

health += basisAmount
    .mul(spotCoreRisk.price + perpCoreRisk.price)
    .mul(spreadWeight - existingWeight);
```

This computes:

```
basisAmount Ă— (P_spot + P_perp) Ă— (W_spread âˆ’ (W_spot + W_perp) / 2)
```

The correct formula â€” replacing each product's individual weight with the spread weight â€” is:

```
basisAmount Ă— P_spot Ă— (W_spread âˆ’ W_spot)
  + basisAmount Ă— P_perp Ă— (W_spread âˆ’ W_perp)
```

Expanding the code's formula:

```
(P_spot + P_perp) Ă— (W_spread âˆ’ (W_spot + W_perp)/2)
= (P_spot + P_perp)Ă—W_spread âˆ’ (P_spot + P_perp)Ă—(W_spot + W_perp)/2
```

Expanding the correct formula:

```
P_spotĂ—W_spread âˆ’ P_spotĂ—W_spot + P_perpĂ—W_spread âˆ’ P_perpĂ—W_perp
= (P_spot + P_perp)Ă—W_spread âˆ’ (P_spotĂ—W_spot + P_perpĂ—W_perp)
```

The difference between the two is:

```
Error = (P_spotĂ—W_spot + P_perpĂ—W_perp) âˆ’ (P_spot + P_perp)Ă—(W_spot + W_perp)/2
      = ((P_spot âˆ’ P_perp) Ă— (W_spot âˆ’ W_perp)) / 2
```

This is non-zero whenever spot and perp prices diverge (which occurs during funding rate stress, liquidation cascades, or low-liquidity events) and whenever the two products carry different risk weights (which is the normal case â€” each product has its own `RiskStore`).

A second compounding error: `getCoreRisk` always passes `amount = 1` to `_getWeightX18`, so it always returns `longWeightInitialX18` regardless of whether the actual position is long or short.

```solidity
// core/contracts/BaseEngine.sol lines 186â€“191
return IProductEngine.CoreRisk(
    amount,
    risk.priceX18,
    RiskHelper._getWeightX18(risk, 1, healthType)  // always longWeight
);
```

For a short perp leg of a spread, the actual health calculation in `_calculateProductHealth` uses `shortWeightInitialX18 â‰¥ 1`, but `getCoreRisk` returns `longWeightInitialX18 â‰¤ 1`. The `existingWeight` average therefore uses the wrong weight for the perp side, further distorting the spread adjustment.

---

### Impact Explanation

The spread health adjustment is used in every call to `getHealth()`, which gates:
- `withdrawCollateral` (health check after withdrawal)
- `mintNlp` / `burnNlp`
- `transferQuote`
- `liquidateSubaccountImpl` (both `isAboveInitial` and `isUnderInitial`)

**Overestimated health** (when the error term is positive): a user holding a spread position receives more health credit than the risk model intends. They can withdraw more collateral, take on more leverage, or avoid liquidation when they should be liquidatable. This directly threatens protocol solvency.

**Underestimated health** (when the error term is negative): a user is liquidated when their position is actually healthy, causing unjust asset loss.

The magnitude scales with `basisAmount Ă— |P_spot âˆ’ P_perp| Ă— |W_spot âˆ’ W_perp| / 2`. During market stress â€” exactly when correct health accounting matters most â€” spot-perp price divergence is largest.

---

### Likelihood Explanation

Any user who holds a spread position (long spot + short perp, or short spot + long perp) triggers this path on every health check. No special permissions are required. The `getHealth` function is called on every sensitive action. The error is always present; its magnitude grows with market volatility.

---

### Recommendation

Replace the aggregated formula with per-component multiplication before summing:

```solidity
// Correct spread health adjustment
health += basisAmount
    .mul(spotCoreRisk.price)
    .mul(spreadWeight - spotCoreRisk.longWeight);

// For the perp leg, use the actual short weight, not longWeight
int128 perpActualWeight = RiskHelper._getWeightX18(
    _risk(_perpId), perpCoreRisk.amount, healthType
);
health += basisAmount
    .mul(perpCoreRisk.price)
    .mul(spreadWeight - perpActualWeight);
```

Additionally, fix `getCoreRisk` to pass the actual `amount` (not a hardcoded `1`) to `_getWeightX18` so that the returned weight reflects the actual position direction.

### Proof of Concept

Consider a spread position with:
- `basisAmount = 1e18` (1 unit)
- `P_spot = 1000e18`, `P_perp = 1010e18` (1% funding divergence)
- `W_spot = 0.9e18` (long weight), `W_perp = 0.9e18` (long weight, as returned by `getCoreRisk`)
- `W_spread = 0.99e18`

**Code computes:**
```
basisAmount Ă— (1000 + 1010) Ă— (0.99 âˆ’ 0.90)
= 1 Ă— 2010 Ă— 0.09 = 180.9
```

**Correct formula:**
```
1 Ă— 1000 Ă— (0.99 âˆ’ 0.90) + 1 Ă— 1010 Ă— (0.99 âˆ’ 0.90)
= 90 + 90.9 = 180.9
```

In this symmetric case the error is zero. Now use asymmetric weights:
- `W_spot = 0.85e18`, `W_perp = 0.95e18`

**Code computes:**
```
1 Ă— 2010 Ă— (0.99 âˆ’ (0.85 + 0.95)/2)
= 2010 Ă— (0.99 âˆ’ 0.90) = 2010 Ă— 0.09 = 180.9
```

**Correct formula:**
```
1 Ă— 1000 Ă— (0.99 âˆ’ 0.85) + 1 Ă— 1010 Ă— (0.99 âˆ’ 0.95)
= 1000 Ă— 0.14 + 1010 Ă— 0.04
= 140 + 40.4 = 180.4
```

The code overestimates health by `0.5` units per unit of position. At scale (`basisAmount = 1,000,000` units), this is a `500,000`-unit overstatement of health, allowing the user to withdraw collateral or avoid liquidation incorrectly. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    function _getWeightX18(
        Risk memory risk,
        int128 amount,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }

        int128 weight;
        if (amount >= 0) {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.longWeightInitialX18
                : risk.longWeightMaintenanceX18;
        } else {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.shortWeightInitialX18
                : risk.shortWeightMaintenanceX18;
        }

        return weight;
    }
