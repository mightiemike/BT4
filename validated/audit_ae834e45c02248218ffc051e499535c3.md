### Title
Incorrect Ancestor Block Used in Fork Difficulty Calculation Allows Underpriced PoW Acceptance — (File: `contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

---

### Summary

`get_next_work_required` in all three chain modules fetches the difficulty-interval boundary block via `get_header_by_height`, which unconditionally reads from `mainchain_height_to_header`. When the block being validated belongs to a fork, the correct ancestor at that height is a fork block stored only in `headers_pool`, not in the mainchain map. The contract therefore computes the expected difficulty using the wrong block's timestamp, producing an incorrect `expected_bits` value that the fork block is then validated against.

---

### Finding Description

In `contract/src/dogecoin.rs`, `get_next_work_required` computes the difficulty-adjustment timespan as:

```rust
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

The developer-inserted TODO on line 291 explicitly acknowledges the problem:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
```

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [2](#0-1) 

Fork blocks are stored only in `headers_pool` via `store_fork_header`, never in `mainchain_height_to_header`:

```rust
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
``` [3](#0-2) 

So when a fork block at a difficulty-adjustment boundary is validated, `get_header_by_height(height_first)` silently returns the **mainchain** block at that height rather than the fork's actual ancestor. The resulting `first_block_time` is wrong, `actual_time_taken` (or `modulated_timespan`) is wrong, and `expected_bits` is wrong.

The same structural defect exists in `bitcoin.rs`:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(config, prev_block_header,
    interval_tail_extend_header.block_header.time.into())
``` [4](#0-3) 

And in `litecoin.rs`:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(config, prev_block_header,
    interval_tail_extend_header.block_header.time.into())
``` [5](#0-4) 

---

### Impact Explanation

`check_pow` enforces `expected_bits == block_header.bits`. If `expected_bits` is computed from the wrong ancestor timestamp, the contract accepts a fork block whose `bits` field encodes a **lower difficulty** than the real protocol requires. A lower `bits` value means a higher target, so the attacker's fork blocks need less PoW to satisfy the hash check:

```rust
require!(
    U256::from_le_bytes(&pow_hash.0) <= target_from_bits(header.bits),
    "block should have correct pow"
);
``` [6](#0-5) 

If the attacker's fork accumulates enough (artificially cheapened) chainwork to exceed the mainchain tip, `reorg_chain` promotes it:

```rust
if current_header.chain_work > total_main_chain_chainwork {
    log!("Chain reorg");
    self.reorg_chain(current_header, last_main_chain_block_height);
}
``` [7](#0-6) 

After reorg, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` return `true` for transactions in the attacker's fabricated fork blocks — **proof forgery** against the real Bitcoin chain.

---

### Likelihood Explanation

**Dogecoin** (most severe): `difficulty_adjustment_interval = 1` at height ≥ 145 000, so every fork block beyond the first triggers the wrong-ancestor path. A fork of length ≥ 2 is sufficient. The attacker controls the fork block timestamps, so the timestamp delta between the mainchain block at `height_first` and the fork's actual ancestor can be chosen to maximize the difficulty reduction.

**Bitcoin / Litecoin**: The fork must diverge before the start of the current 2016-block difficulty period. This requires a longer fork but is still within the stated threat model of a relayer supplying adversarial chain data.

The entry point — `submit_blocks` — is gated by `#[trusted_relayer]`, which the prompt explicitly includes as a valid attacker role ("relayer-path user supplying adversarial chain data").

---

### Recommendation

Replace `get_header_by_height` (mainchain-only lookup) with an ancestor walk that follows `prev_block_hash` links through `headers_pool` from `prev_block_header` back `blocks_to_go_back` steps. This correctly resolves the boundary block for both mainchain and fork submissions, matching the reference implementations in Bitcoin Core, Litecoin Core, and Dogecoin Core.

---

### Proof of Concept

1. Deploy the contract (Dogecoin build, height ≥ 145 000, `skip_pow_verification = false`).
2. Submit mainchain blocks at heights `H` and `H+1` with timestamp `T_main` at `H`.
3. Submit a fork block at height `H+1` (diverging at `H`) with timestamp `T_fork` at `H+1`.
4. Submit a fork block at height `H+2`. The contract calls `get_next_work_required`:
   - `height_first = H+1 - 1 = H`
   - `get_header_by_height(H)` returns the **mainchain** block at `H` with time `T_main`
   - `modulated_timespan` is computed from `T_fork_prev - T_main` (wrong; should use fork ancestor at `H`)
5. By choosing `T_fork_prev` (the fork block at `H+1`'s timestamp) such that `T_fork_prev - T_main` is large, the attacker inflates `modulated_timespan`, raising the new target (lowering difficulty).
6. The contract accepts the fork block at `H+2` with the reduced `bits`, requiring less PoW.
7. Repeat until the fork's `chain_work` exceeds the mainchain tip; the contract reorgs.
8. `verify_transaction_inclusion` now returns `true` for attacker-chosen fake transactions in the fork.
