### Title
Wrong Weight Parameter Used in Spread Health Calculation — (`File: core/contracts/libraries/RiskHelper.sol`, `core/contracts/Clearinghouse.sol`)

### Summary
`RiskHelper._getSpreadWeightX18` and `Clearinghouse.getHealth` both use `longWeight` for the **short leg** of a spread position when computing the spread health adjustment. This mirrors the `calculateSpotPrice` bug class exactly: a wrong parameter is substituted into a pricing/health formula, producing a systematically incorrect output that is consumed by downstream financial logic.

### Finding Description

In `Clearinghouse.getHealth`, the spread health adjustment is:

```solidity
int128 existingWeight = (spotCoreRisk.longWeight +
    perpCoreRisk.longWeight) / 2;
int128 spreadWeight = RiskHelper._getSpreadWeightX18(
    perpCoreRisk,
    spotCoreRisk,
    healthType
);
health += basisAmount
    .mul(spotCoreRisk.price + perpCoreRisk.price)
    .mul(spreadWeight - existingWeight);
``` [1](#0-0) 

`existingWeight` is always computed as the average of `spotCoreRisk.longWeight` and `perpCoreRisk.longWeight`, regardless of position direction. For a **long spot + short perp** spread, the perp leg is short, so `existingWeight` should use `perpCoreRisk.shortWeight`. For a **short spot + long perp** spread, the spot leg is short, so `existingWeight` should use `spotCoreRisk.shortWeight`.

The same error propagates into `_getSpreadWeightX18`:

```solidity
if (spotCoreRisk.amount > 0) {
    spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
} else {
    spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;
}
``` [2](#0-1) 

The function explicitly branches on `spotCoreRisk.amount > 0` to identify which leg is short, but then reads `longWeight` from that short leg in both branches. The correct field for the short leg is `shortWeight`.

This is structurally identical to the `calculateSpotPrice` analog: a formula parameter is substituted with the wrong variant (`longWeight` instead of `shortWeight`), producing a systematically biased result.

### Impact Explanation

Using concrete representative values (`longWeightInitial = 0.9`, `shortWeightInitial = 1.1`, `basisAmount = 1`, `spotPrice = perpPrice = 1`):

**Wrong calculation (current code):**
- `existingWeight = (0.9 + 0.9) / 2 = 0.9`
- `spreadWeight = 1 − (1 − 0.9) / 5 = 0.98`
- Health adjustment = `1 × 2 × (0.98 − 0.9) = +0.16`

**Correct calculation:**
- `existingWeight = (0.9 + 1.1) / 2 = 1.0`
- `spreadWeight = 1 − (1 − 1.1) / 5 = 1.02`, capped at `0.99`
- Health adjustment = `1 × 2 × (0.99 − 1.0) = −0.02`

The current code produces a health adjustment of **+0.16** where the correct value is **−0.02** — a delta of **+0.18 per unit of spread notional**. For a $1 M spread position this overstates health by ~$180,000.

`getHealth` is consumed by every critical solvency gate in the protocol:

- `withdrawCollateral` — users can withdraw excess collateral [3](#0-2) 
- `mintNlp` — users can mint NLP against inflated health [4](#0-3) 
- `transferQuote` — users can transfer quote beyond true capacity <cite repo="patrichyt/nado-

    function _getSpreadWeightX18(
        IProductEngine.CoreRisk memory perpCoreRisk,
        IProductEngine.CoreRisk memory spotCoreRisk,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }
        int128 spreadWeight;
        if (spotCoreRisk.amount > 0) {
            spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
        } else {
            spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;
        }
        int128 maxSpreadWeight;
        if (healthType == IProductEngine.HealthType.INITIAL) {
            maxSpreadWeight = ONE - ONE / 100; // 0.99
        } else {
            maxSpreadWeight = ONE - (ONE * 6) / 1000; // 0.994
        }
        if (spreadWeight > maxSpreadWeight) {
            spreadWeight = maxSpreadWeight;
        }
        return spreadWeight;
    }
