### Title
`ENABLE_SECP_OPS` Flag Does Not Gate the 4-Byte Secp Opcode Dispatch Path, Allowing Secp Operators to Execute Without the Flag - (`File: src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` contains two independent dispatch paths for the secp256k1 and secp256r1 verify operators. The 1-byte opcode paths (opcodes 64 and 65) are correctly gated behind `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte opcode paths (`0x13d61f00` and `0x1c3a8f00`) are not gated at all. Any caller that relies on the absence of `ENABLE_SECP_OPS` to suppress secp execution — including mempool-mode validators — can be bypassed by an attacker who encodes the same operators using their 4-byte form.

---

### Finding Description

`ChiaDialect::op()` in `src/chia_dialect.rs` dispatches operators through two separate branches.

**Branch 1 — 4-byte opcodes (lines 157–183):** When the operator atom is exactly 4 bytes long, the function decodes it as a `u32` and matches against two hardcoded secp opcodes. No flag check is performed before calling the operator function:

```rust
// lines 157–183
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,
        0x1c3a8f00 => op_secp256r1_verify,
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);  // no ENABLE_SECP_OPS check
}
``` [1](#0-0) 

**Branch 2 — 1-byte opcodes (lines 248–249):** When the operator atom is 1 byte, opcodes 64 and 65 are gated behind `ENABLE_SECP_OPS`:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The flag itself is documented as "Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify)": [3](#0-2) 

The code comment inside the 4-byte branch explicitly identifies `0x13d61f00` and `0x1c3a8f00` as *the secp operators*, not as generic unknown operators: [4](#0-3) 

The wiring is therefore broken: the same two operator implementations are reachable via two different opcode encodings, but only one encoding is gated by the hard-fork flag.

`MEMPOOL_MODE` does not include `ENABLE_SECP_OPS`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [5](#0-4) 

So in mempool mode, a program using opcode `64` or `65` is rejected (`Unimplemented`), while the identical program using `0x13d61f00` or `0x1c3a8f00` executes successfully and returns `nil` at full secp cost. The flag gate is silently bypassed.

The Python-facing API exports `ENABLE_SECP_OPS` as a meaningful constant, confirming it is intended to be a caller-controlled gate: [6](#0-5) 

---

### Impact Explanation

Any caller that constructs a `ChiaDialect` without `ENABLE_SECP_OPS` — including the Chia mempool validator running `MEMPOOL_MODE` — intends to reject secp signature verification. An attacker who submits a CLVM program using the 4-byte opcode encoding bypasses that intent: the program executes `op_secp256k1_verify` or `op_secp256r1_verify` at full cost, returns `nil`, and is accepted as valid. This creates a concrete operator-dispatch inconsistency between the two encodings of the same operator, which is a flag/operator wiring error with direct consensus and mempool-validation impact.

---

### Likelihood Explanation

The 4-byte opcode values are derivable from the cost formula documented in the code comment at line 172. Any attacker reading the source can compute `0x13d61f00` and `0x1c3a8f00` and craft a program that uses them. No privileged access, social engineering, or dependency compromise is required. The entry path is fully attacker-controlled CLVM bytes passed to `run_serialized_chia_program` or any other `run_program` caller.

---

### Recommendation

Add an `ENABLE_SECP_OPS` flag check inside the 4-byte dispatch branch before calling the secp operator functions, mirroring the guard already present on the 1-byte paths:

```rust
// in the op_len == 4 branch
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures both opcode encodings of the secp operators are uniformly gated by the same flag.

---

### Proof of Concept

1. Construct a `ChiaDialect` with `MEMPOOL_MODE` (which excludes `ENABLE_SECP_OPS`).
2. Build a CLVM program that calls `secp256k1_verify` using the 4-byte opcode `0x13d61f00` instead of the 1-byte opcode `64`.
3. Run the program via `run_program`.
4. Observe: the program executes successfully and returns `nil`, consuming 1,300,000 cost units — despite `ENABLE_SECP_OPS` being absent.
5. Repeat with opcode `64` (1-byte): the program is rejected with `Unimplemented`.

The two encodings of the same operator produce different accept/reject outcomes under identical flags, confirming the broken wiring.
