### Title
`backfillDelegatorAddress` Bypasses `setPositionWithState`, Leaving `PositionCountByValidator` at Zero — Migrated Positions Receive Bonus for Unbonded Validator Segments — (`x/tieredrewards/migrations/v2/migrate.go`)

---

### Summary

`backfillDelegatorAddress` writes directly to the raw `positions` `collections.Map`, bypassing the keeper's `setPositionWithState`. As a result, `PositionCountByValidator` is never populated for any migrated position. Every staking hook that gates on `count == 0` — `AfterValidatorBeginUnbonding`, `AfterValidatorBonded`, `BeforeValidatorSlashed` — silently skips event creation for those validators. When a validator unbonds and later re-bonds, migrated positions claim bonus for the entire elapsed window (including the unbonded gap) because `LastKnownBonded` is never flipped by a missing UNBOND event, and the final `val.IsBonded()` guard in `processEventsAndClaimBonus` only protects the current open segment, not the historical one.

---

### Finding Description

**Step 1 — Migration writes past the keeper**

`backfillDelegatorAddress` iterates all positions and calls `positions.Set(ctx, pos.Id, pos)` directly: [1](#0-0) 

The keeper's `setPositionWithState` is the sole path that calls `reindexPositionCountByValidator`: [2](#0-1) 

Because the migration never calls `setPositionWithState`, `PositionCountByValidator` remains 0 for every validator that holds migrated positions.

**Step 2 — All three staking hooks gate on the zero count**

`AfterValidatorBeginUnbonding`, `AfterValidatorBonded`, and `BeforeValidatorSlashed` all share the same guard: [3](#0-2) 

With count = 0, the hook returns `nil` without appending any `ValidatorEvent`. No UNBOND event is ever stored for a validator whose only positions are migrated ones.

**Step 3 — `processEventsAndClaimBonus` uses `LastKnownBonded`, not live state, for historical segments**

When a position claims rewards, the function initialises the bonded flag from the persisted field: [4](#0-3) 

It then replays events since `LastEventSeq`. Because no UNBOND event was ever written, `events` is empty, `bonded` stays `true`, and `segmentStart` stays at `LastBonusAccrual`.

After the (empty) loop, the only live check is: [5](#0-4) 

This guard protects only the **current open segment**. If the validator has already re-bonded by claim time, `val.IsBonded()` is `true`, and `computeSegmentBonus` is called for `[LastBonusAccrual, blockTime]` — a window that spans the entire unbonded gap.

**Step 4 — `LastKnownBonded` is never corrected**

At the end of `processEventsAndClaimBonus`, `UpdateLastKnownBonded(bonded)` persists the in-memory flag: [6](#0-5) 

Because no UNBOND event set `bonded = false`, the flag remains `true` indefinitely, compounding the error across future unbond/rebond cycles.

**Step 5 — Bonus is paid from the module rewards pool**

The computed bonus is transferred directly from `types.RewardsPoolName` to the position owner: [7](#0-6) 

This is real value leaving the rewards pool for work (bonded delegation) that was never performed.

---

### Impact Explanation

Every position that existed before the v1→v2 upgrade and whose validator undergoes at least one unbond/rebond cycle after the upgrade will receive bonus rewards for the unbonded segment. The overpayment is proportional to `shares × tokensPerShare × tier.BonusApy × unbondedDuration / SecondsPerYear`. On a chain with many long-lived positions and a validator that briefly unbonds (e.g., due to double-sign jailing), the cumulative drain on the rewards pool can be material. The invariant that `ValidatorEvent.ReferenceCount` equals the number of positions that will process it is violated in the degenerate direction: no event is created at all, so `ReferenceCount` is effectively −∞ relative to the positions that should have consumed it.

---

### Likelihood Explanation

The bug is triggered by the upgrade itself — no attacker action is required. Any validator that begins unbonding after the upgrade (routine jailing, governance-forced unbonding, or natural rotation) activates the overpayment path for all migrated positions on that validator. The window is permanent: `PositionCountByValidator` is never backfilled for migrated positions unless they are individually redelegated or deleted through the normal keeper path.

---

### Recommendation

Add a third step to `v2.Migrate` that rebuilds `PositionCountByValidator` from the now-populated `DelegatorAddress` fields. After `backfillDelegatorAddress` completes, iterate all positions, call `getDelegation(ctx, pos.DelegatorAddress)` for each, and for every position with a live delegation call `increasePositionCountForValidator` for the corresponding validator. This mirrors what `InitGenesis` already does via `setPosition(ctx, pos, &ValidatorTransition{PreviousAddress: ""})`. [8](#0-7) 

Alternatively, call `k.setPosition(ctx, pos, &ValidatorTransition{PreviousAddress: ""})` inside `backfillDelegatorAddress` instead of `positions.Set` directly, which would rebuild all secondary indexes atomically.

---

### Proof of Concept

```
1. Upgrade chain from v1 to v2 with N positions delegated to valA.
2. Migrate1to2 runs: DelegatorAddress is backfilled, PositionCountByValidator[valA] = 0.
3. valA gets jailed → AfterValidatorBeginUnbonding fires → count=0 → no UNBOND event stored.
4. valA unjails and re-bonds → AfterValidatorBonded fires → count=0 → no BOND event stored.
5. Any position owner calls MsgClaimTierRewards.
6. processEventsAndClaimBonus: events=[], bonded=true (LastKnownBonded), val.IsBonded()=true.
7. computeSegmentBonus([LastBonusAccrual, blockTime]) is called — the window includes the
   entire unbonded gap.
8. Bonus coins are sent from RewardsPoolName to the owner for the unbonded segment.
9. Assert: totalBonus > correct bonus (bonus for bonded-only segments).
```
