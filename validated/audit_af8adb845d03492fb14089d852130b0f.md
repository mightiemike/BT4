### Title
`ContractOwner` Lacks `receive()` Function, Permanently Locking Native ETH Sent to DDAs — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner` has no `receive()` or `fallback()` function, yet its `withdrawFromDirectDepositV1` function attempts to pull native ETH from a `DirectDepositV1` (DDA) contract into itself before forwarding it to the caller. Because `ContractOwner` cannot accept ETH, the inner call always reverts, permanently locking any native ETH that accumulates in a DDA.

---

### Finding Description

`DirectDepositV1.withdrawNative()` sends ETH to its caller via a low-level call:

```solidity
// DirectDepositV1.sol line 108-112
function withdrawNative() external onlyOwner {
    uint256 balance = address(this).balance;
    (bool success, ) = msg.sender.call{value: balance}("");
    require(success, "Failed to transfer native token to owner");
}
```

`ContractOwner.withdrawFromDirectDepositV1` calls this function when `token == address(0)`, expecting to receive the ETH at `address(this)` and then forward it:

```solidity
// ContractOwner.sol lines 628-636
if (token == address(0)) {
    uint256 preBalance = address(this).balance;
    DirectDepositV1(directDepositV1).withdrawNative();
    uint256 postBalance = address(this).balance;
    require(postBalance > preBalance, "empty");
    (bool success, ) = msg.sender.call{value: postBalance - preBalance}("");
    require(success, "xfer");
}
```

`ContractOwner` is an upgradeable contract that declares no `receive()` or `fallback()` function. The grep across all production Solidity files confirms only `DirectDepositV1.sol` and `MockWrappedNative.sol` define `receive()`. When `withdrawNative()` executes `msg.sender.call{value: balance}("")` with `msg.sender = ContractOwner`, the call fails because `ContractOwner` cannot accept ETH. `withdrawNative()` then reverts with `"Failed to transfer native token to owner"`, and the entire `withdrawFromDirectDepositV1` call reverts.

Native ETH can accumulate in a DDA in two ways:
1. **Constructor path**: if `wrappedNative.call{value: balance}("")` fails at DDA construction time, the ETH is silently retained (the constructor emits `NativeTokenTransferFailed` but does not revert).
2. **`selfdestruct` / pre-deployment funding**: ETH force-sent to the DDA address bypasses the `receive()` hook entirely.

In both cases the ETH is irrecoverable because the only owner-accessible recovery path (`withdrawFromDirectDepositV1`) is permanently broken.

---

### Impact Explanation

Any native ETH that ends up in a DDA and is not successfully wrapped at receipt time is permanently locked. The owner cannot recover it. The asset delta is the full ETH balance of the affected DDA(s). This is a direct, irreversible asset loss, not merely a denial-of-service.

---

### Likelihood Explanation

Medium-low. The most realistic trigger is the constructor's silent failure path: if `wrappedNative` is temporarily unavailable or misconfigured at DDA creation time, the constructor retains the ETH and emits an event rather than reverting. A `selfdestruct`-based force-send is also always possible. Both scenarios are realistic on a live network.

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept ETH forwarded from `DirectDepositV1.withdrawNative()`:

```solidity
receive() external payable {}
```

Alternatively, refactor `withdrawNative()` to accept a `recipient` address parameter so ETH is sent directly to the intended recipient rather than routing through `ContractOwner`.

---

### Proof of Concept

1. Deploy `ContractOwner` (no `receive()` function present).
2. Create a DDA for a subaccount via `creditDepositV1`.
3. Force ETH into the DDA via `selfdestruct` from a helper contract, or trigger the constructor's silent-failure path by having `wrappedNative` reject the call.
4. Call `withdrawFromDirectDepositV1(subaccount, address(0))` as the owner.
5. Observe revert: `DirectDepositV1.withdrawNative()` calls `ContractOwner.call{value: balance}("")`, which fails because `ContractOwner` has no `receive()`, causing the require `"Failed to transfer native token to owner"` to fire.
6. ETH remains permanently locked in the DDA. [1](#0-0) [2](#0-1) [3](#0-2)
