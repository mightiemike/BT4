### Title
Missing Escrow-Address Guard in `processReceivedPacket` Allows NFT to Be Permanently Locked — (`x/nft-transfer/keeper/packet.go`)

---

### Summary

`OnRecvPacket` → `processReceivedPacket` mints IBC voucher NFTs to whatever address `data.Receiver` decodes to. There is no guard preventing `data.Receiver` from being the escrow address itself. A malicious IBC counterparty can craft a packet whose `Receiver` field is the bech32 encoding of `GetEscrowAddress("nft", "channel-X")` on the receiving chain. `ValidateBasic` passes (the escrow address is a valid bech32 address), and the NFT is minted to the escrow address — an address with no private key and no recovery path — permanently locking it.

---

### Finding Description

**Entry point:** `IBCModule.OnRecvPacket` in `x/nft-transfer/ibc_module.go` calls `keeper.OnRecvPacket`, which calls `data.ValidateBasic()` then `processReceivedPacket`.

**`ValidateBasic` guard** (`x/nft-transfer/types/packet.go` lines 63–70):

```go
if strings.TrimSpace(nftpd.Receiver) == "" {
    return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "receiver address cannot be blank")
}
if _, err := sdk.AccAddressFromBech32(nftpd.Receiver); err != nil {
    return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid receiver address")
}
```

The escrow address is a valid bech32 address (it is a 20-byte SHA-256 hash), so this check passes unconditionally. [1](#0-0) 

**`processReceivedPacket` — away-from-origin path** (`x/nft-transfer/keeper/packet.go` lines 143–185):

```go
receiver, err := sdk.AccAddressFromBech32(data.Receiver)   // decodes to escrowAddress bytes
...
escrowAddress := types.GetEscrowAddress(packet.GetDestPort(), packet.GetDestChannel())
...
k.nftKeeper.MintNFT(ctx, voucherClassID, tokenID, "", data.TokenUris[i], "", escrowAddress, receiver)
//                                                                                            ^^^^^^^^
//                                                                              owner = receiver = escrowAddress
```

When `data.Receiver` is the bech32 of the escrow address, `receiver == escrowAddress`, and the NFT is minted with `owner = escrowAddress`. [2](#0-1) 

**`MintNFT` in the NFT keeper** (`x/nft/keeper/keeper.go` lines 72–82): checks only that `sender` is the denom creator (which is `escrowAddress` — it created the denom at line 168 of `packet.go`). It then calls `MintNFTUnverified` with `owner = receiver = escrowAddress`. No blocked-address check exists anywhere in this path. [3](#0-2) 

**Escrow address construction** (`x/nft-transfer/types/keys.go` lines 45–55): The address is deterministic and publicly computable from `portID` and `channelID`, so any counterparty can pre-compute it. [4](#0-3) 

**`AccountKeeper` interface** (`x/nft-transfer/types/expected_keepers.go` lines 47–54): The interface exposes no `BlockedAddr` / `IsBlockedAddr` method, confirming the module never checks whether the receiver is a module/escrow account. [5](#0-4) 

---

### Impact Explanation

The NFT is minted to the escrow address. The escrow address:
- Has no private key (it is a hash-derived address, not a user key).
- Is not a Cosmos SDK module account, so there is no module-level recovery mechanism.
- Can only release NFTs via `TransferOwner(ctx, denomID, tokenID, escrowAddress, dst)` (called in the back-to-origin unescrow path), but that path requires the NFT to have been legitimately escrowed by a user-initiated send — not minted directly to the escrow address.

No user can ever claim the NFT. It is permanently locked.

---

### Likelihood Explanation

Exploitability requires a malicious IBC counterparty chain (one that controls what goes into `data.Receiver`). This is a realistic threat: IBC channels can be opened with any chain, and the receiving chain is responsible for protecting its own invariants regardless of counterparty behavior. The ICS-20 fungible token transfer module in ibc-go addresses exactly this class of attack with a `BlockedAddr` check; this ICS-721 implementation omits it.

---

### Recommendation

Add a blocked-address check in `processReceivedPacket` before minting. Concretely:

1. Extend the `AccountKeeper` interface in `x/nft-transfer/types/expected_keepers.go` to include `BlockedAddr(addr sdk.AccAddress) bool`.
2. In `processReceivedPacket` (`x/nft-transfer/keeper/packet.go`), after decoding `receiver`, reject the packet if `k.authKeeper.BlockedAddr(receiver)` is true, or if `receiver.Equals(escrowAddress)`.

---

### Proof of Concept

```go
// keeper_test.go (pseudocode)
escrowAddr := types.GetEscrowAddress("nft", "channel-0")

packet := channeltypes.Packet{
    SourcePort:      "nft",
    SourceChannel:   "channel-1",
    DestinationPort: "nft",
    DestinationChannel: "channel-0",
    ...
}
data := types.NonFungibleTokenPacketData{
    ClassId:   "someClass",
    ClassUri:  "",
    TokenIds:  []string{"token1"},
    TokenUris: []string{""},
    Sender:    "cosmos1...",          // any valid sender
    Receiver:  escrowAddr.String(),   // ← escrow address as receiver
}

err := keeper.OnRecvPacket(ctx, "ics721-1", packet, data)
// err == nil  (no guard fires)

nft, _ := nftKeeper.GetNFT(ctx, "ibc/...", "token1")
// nft.GetOwner() == escrowAddr  ← NFT permanently locked
```

```go
func GetEscrowAddress(portID, channelID string) sdk.AccAddress {
	// a slash is used to create domain separation between port and channel identifiers to
	// prevent address collisions between escrow addresses created for different channels
	contents := fmt.Sprintf("%s/%s", portID, channelID)

	// ADR 028 AddressHash construction
	preImage := []byte(Version)
	preImage = append(preImage, 0)
	preImage = append(preImage, contents...)
	hash := sha256.Sum256(preImage)
	return hash[:20]
```
