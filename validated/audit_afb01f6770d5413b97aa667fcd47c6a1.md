Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to its own `msg.sender`. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. A pool admin who allowlists the router (the necessary step to support router-mediated swaps for curated users) inadvertently grants every unprivileged user the ability to bypass the per-pool swap allowlist by routing through the public router.

## Finding Description

**Step 1 — Pool passes its own `msg.sender` as `sender` to the extension.**

`MetricOmmPool.swap` calls `_beforeSwap` with `msg.sender` as the first argument:

```solidity
_beforeSwap(
  msg.sender,   // ← whoever called pool.swap(), i.e. the router
  recipient,
  ...
  extensionData
);
``` [1](#0-0) 

**Step 2 — Extension checks `sender`, which is the router address when called via router.**

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the pool's caller:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [2](#0-1) 

**Step 3 — Router always calls the pool as itself.**

In `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly. The pool's `msg.sender` is therefore the router address, not the end user: [3](#0-2) 

The same pattern holds for `exactInput` (all hops): [4](#0-3) 

And for `exactOutputSingle` and `exactOutput` recursive callback hops: [5](#0-4) 

**Step 4 — Irreconcilable configuration trap.**

A pool admin who wants to restrict swaps to specific users (e.g., KYC'd addresses A, B, C) faces two bad options:

- **Do not allowlist the router**: allowlisted users A, B, C cannot use `MetricOmmSimpleRouter` at all.
- **Allowlist the router**: the extension checks `allowedSwapper[pool][router] = true` for every swap routed through the router, so any unprivileged user D can bypass the allowlist by calling `router.exactInputSingle(...)`.

The extension does not use `extensionData` at all — it only checks `sender` — so there is no existing mechanism to pass end-user identity through the router. [6](#0-5) 

## Impact Explanation

Any user can bypass a curated pool's swap allowlist by routing through the public `MetricOmmSimpleRouter`. For pools designed to restrict trading to specific counterparties (e.g., to prevent adverse selection against LPs or enforce regulatory access controls), this allows unauthorized traders to swap at oracle-derived prices. The allowlist guard — the sole mechanism protecting curated pools — silently fails open for all router-mediated swaps once the router is allowlisted. This constitutes an admin-boundary break where an unprivileged path circumvents the pool admin's intended access control, and broken core pool functionality (the allowlist extension) causing potential loss of value to LPs.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool admin who configures a `SwapAllowlistExtension` and also wants their allowlisted users to be able to use the router will naturally allowlist the router — this is the expected, non-malicious configuration. The attacker requires no special privileges, only the ability to call a public router function. The bypass is repeatable and requires no setup beyond the natural pool configuration.

## Recommendation

The `SwapAllowlistExtension` must gate the economically relevant actor, not the immediate pool caller. Two viable approaches:

1. **Forward the original user in `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling the pool; the extension decodes and checks it. This requires a trusted router convention and the extension to verify the caller is a trusted router before trusting the encoded address.
2. **Dedicated router-aware extension**: Implement a variant of `SwapAllowlistExtension` that accepts a signed user identity in `extensionData` and verifies it against the allowlist, making the check router-agnostic.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Admin allowlists address A (legitimate user): allowedSwapper[pool][A] = true
  - Admin allowlists MetricOmmSimpleRouter (to support router-mediated swaps for A):
      allowedSwapper[pool][router] = true

Attack:
  - Address B (not allowlisted) calls:
      router.exactInputSingle(ExactInputSingleParams({
          pool: curated_pool,
          tokenIn: token0,
          tokenOut: token1,
          zeroForOne: true,
          amountIn: X,
          amountOutMinimum: 0,
          recipient: address(B),
          deadline: block.timestamp,
          priceLimitX64: 0,
          extensionData: ""
      }))
  - Router calls pool.swap() → pool's msg.sender = router
  - _beforeSwap passes sender = router to SwapAllowlistExtension
  - Extension checks allowedSwapper[pool][router] = true → passes
  - B successfully swaps on a pool they are not authorized to access

Verification:
  - Foundry test: deploy pool + SwapAllowlistExtension, configure as above,
    call router.exactInputSingle from an address not in the allowlist,
    assert the swap succeeds (no NotAllowedToSwap revert).
```
