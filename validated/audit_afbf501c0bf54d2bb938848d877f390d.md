### Title
No Upper Bound on `ExitDuration` Allows Governance to Permanently Trap Users' Locked Tokens - (`File: x/tieredrewards/types/tier.go`)

### Summary

The `Tier.Validate()` function in `x/tieredrewards/types/tier.go` enforces only a lower bound (`ExitDuration > 0`) on the tier's exit wait period, with no upper bound. Governance can pass `MsgUpdateTier` to set `ExitDuration` to an arbitrarily large value (e.g., 1000 years). Any user who subsequently calls `MsgTriggerExitFromTier` will have their `ExitUnlockAt` stamped as `now + huge_duration`, making both `MsgTierUndelegate` and `MsgExitTierWithDelegation` permanently unreachable. Their locked tokens are trapped in the tier module indefinitely.

### Finding Description

`Tier.Validate()` checks only that `ExitDuration > 0`:

```go
if t.ExitDuration <= 0 {
    return fmt.Errorf("exit duration must be positive")
}
```

There is no maximum. Governance can submit `MsgUpdateTier` with `ExitDuration = math.MaxInt64` nanoseconds (≈292 years) or any arbitrarily large `time.Duration`.

When a user calls `MsgTriggerExitFromTier`, the handler reads the current tier's `ExitDuration` and stamps the position:

```go
tier, err := ms.getTier(ctx, pos.TierId)
...
pos.TriggerExit(sdkCtx.BlockTime(), tier.ExitDuration)
```

`ExitUnlockAt` is stored on the position as `ExitTriggeredAt + tier.ExitDuration`. Both exit paths — `MsgTierUndelegate` and `MsgExitTierWithDelegation` — require `block_time >= pos.ExitUnlockAt` (checked in `validateWithdrawFromTier` and the analogous exit-with-delegation validator). If `ExitUnlockAt` is set to a date centuries in the future, neither path is ever reachable.

`MsgClearPosition` can cancel a triggered exit, but only while the position is still delegated. If governance simultaneously marks the tier `CloseOnly = true`, `ClearPosition` is also blocked, leaving users with no escape path at all.

### Impact Explanation

Users' locked tokens (held in per-position delegator accounts inside the tier module) become permanently inaccessible. The exact corrupted invariant is: `pos.ExitUnlockAt` is set to an unreachable future timestamp, so the position's delegator account balance can never be returned to the owner via any supported exit message. This is a direct, permanent loss of access to user funds.

### Likelihood Explanation

The entry path is a standard governance proposal (`MsgUpdateTier`). Any governance participant can submit such a proposal; it passes if the normal quorum/threshold is met. No leaked keys or social engineering are required — only a governance vote, which is the same privileged-but-reachable mechanism the external report's "approved EOA" represented. The missing safeguard (an upper bound) is the root cause, not the identity of the setter.

### Recommendation

Add a maximum `ExitDuration` constant (e.g., 5 years) and enforce it in `Tier.Validate()`:

```go
const MaxExitDuration = 5 * 365 * 24 * time.Hour

if t.ExitDuration > MaxExitDuration {
    return fmt.Errorf("exit duration must not exceed %s: got %s", MaxExitDuration, t.ExitDuration)
}
```

This mirrors the existing `BonusApy` cap (`<= 1.0`) that was added explicitly to "prevent governance from draining the rewards pool."

### Proof of Concept

1. Governance submits `MsgUpdateTier` with `ExitDuration = 292 * 365 * 24 * time.Hour` (≈292 years, near `math.MaxInt64` nanoseconds). `Tier.Validate()` accepts it because `ExitDuration > 0`.
2. User calls `MsgLockTier` to lock tokens in the tier.
3. User calls `MsgTriggerExitFromTier`. Handler executes `pos.TriggerExit(blockTime, tier.ExitDuration)`, setting `pos.ExitUnlockAt = blockTime + 292 years`.
4. User calls `MsgTierUndelegate` → rejected: `ExitLockDurationNotReached` (block time is centuries before `ExitUnlockAt`).
5. User calls `MsgExitTierWithDelegation` → same rejection.
6. If governance also set `CloseOnly = true`, `MsgClearPosition` is rejected with `ErrTierIsCloseOnly`.
7. User's locked tokens remain permanently trapped in the position's delegator account.

**Root cause:** [1](#0-0)  — only a lower bound is checked; no upper bound exists.

**Governance entry path:** [2](#0-1)  — `UpdateTier` calls `SetTier(ctx, msg.Tier)` after validating only via `Tier.Validate()`.

**ExitUnlockAt stamped at trigger time from live tier state:** [3](#0-2) 

**Withdrawal blocked until ExitUnlockAt:** [4](#0-3) 

**Contrast: BonusApy has an explicit cap; ExitDuration does not:** [5](#0-4)
