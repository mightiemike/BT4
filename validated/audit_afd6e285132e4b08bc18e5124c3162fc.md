### Title
Unchecked u64 Multiplication in `op_unknown` Cost Computation Silently Wraps, Bypassing Cost Validation — (File: src/more_ops.rs)

---

### Summary

In `op_unknown`, the final cost multiplication `cost *= cost_multiplier + 1` is performed without overflow protection. In Rust release builds, integer overflow wraps silently (two's complement). The subsequent guard `if cost > u32::MAX as u64` then evaluates the wrapped (incorrect) value, allowing an attacker-crafted unknown opcode to report a cost of 0 or a small value when the true cost would be enormous. This causes consensus divergence between the Rust implementation and the Python reference implementation (which uses arbitrary-precision integers).

---

### Finding Description

In `op_unknown` (`src/more_ops.rs`), the cost computation proceeds as follows:

1. `cost_multiplier` is extracted from attacker-controlled opcode bytes via `u32_from_u8`, yielding a value in `[0, u32::MAX]`.
2. `cost` is computed from attacker-controlled arguments (cost_function 0–3) and checked against `max_cost` via `check_cost(cost, max_cost)?`.
3. The multiplication `cost *= cost_multiplier + 1` is performed **without overflow protection**.
4. The result is validated by `if cost > u32::MAX as u64`. [1](#0-0) 

`cost_multiplier` is at most `u32::MAX = 4294967295`, so `cost_multiplier + 1` is at most `2^32 = 4294967296`. If `cost` before multiplication is a multiple of `2^32` (e.g., exactly `2^32 = 4294967296`), then:

```
cost * (cost_multiplier + 1) = 2^32 * 2^32 = 2^64 ≡ 0 (mod 2^64)
```

The wrapped result `0` passes the check `0 > u32::MAX` (false), and `Ok(Reduction(0, nil))` is returned — a cost of **zero** for an operation that should cost billions.

### Impact Explanation

**Consensus divergence**: The Python reference implementation uses arbitrary-precision integers and computes the correct (non-overflowed) cost. The Rust implementation wraps, computing a different cost for the same program. Two nodes running different implementations would disagree on whether a block is valid, causing a consensus split.

**Undercharged execution**: An unknown opcode that should cost a large amount (e.g., `2^64`) reports cost 0, bypassing the block cost limit. This allows a program to consume more "theoretical" cost budget than it pays for.

Unknown ops are dispatched from both `ChiaDialect` and `RuntimeDialect`: [6](#0-5) 

---

### Likelihood Explanation

**Trigger conditions**:
- `max_cost >= 2^32 ≈ 4.3e9` — satisfied by Chia's block cost limit (~11e9).
- `cost` before multiplication must be a multiple of `2^32`. For `cost_function=1` (add-like), this requires approximately `(2^32 - 99) / 320 ≈ 13.4 million` zero-byte arguments. This is large but not impossible without `LIMIT_HEAP`.
- `cost_multiplier = u32::MAX` — trivially crafted via opcode bytes `[0xFF, 0xFF, 0xFF, 0xFF, 0x40]`.

**Mode restriction**: In mempool mode (`MEMPOOL_MODE`), `NO_UNKNOWN_OPS` is set and `op_unknown` is never called. The vulnerability only affects consensus mode (block validation), which is the critical path. [7](#0-6) 

The test suite explicitly acknowledges that programs attempting to wrap around the 64-bit cost counter are a concern: [8](#0-7) 

---

### Recommendation

Replace the unchecked multiplication with a checked variant that returns an error on overflow:

```rust
// Before (vulnerable):
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}

// After (safe):
cost = cost
    .checked_mul(cost_multiplier + 1)
    .filter(|&c| c <= u32::MAX as u64)
    .ok_or(EvalErr::Invalid(o))?;
```

This mirrors the pattern already used elsewhere in the codebase (e.g., `checked_add` in `op_add`'s fast path and `checked_mul` in `serialized_length_serde_2026`): [9](#0-8) [10](#0-9)

### Proof of Concept

**Opcode**: `[0xFF, 0xFF, 0xFF, 0xFF, 0x40]`
- Last byte `0x40` → `cost_function = (0x40 >> 6) & 0x3 = 1` (add-like)
- First 4 bytes `[0xFF, 0xFF, 0xFF, 0xFF]` → `cost_multiplier = u32::MAX = 4294967295`
- `cost_multiplier + 1 = 2^32`

**Arguments**: Approximately 13,421,772 zero-byte atoms (nil nodes), making:
```
cost = 99 + 13421772 * 320 + 0 * 3 = 4294967139 ≈ 2^32
```
(adjust arg count to hit exact multiple of `2^32`)

**Result**:
```
cost *= 2^32  →  2^32 * 2^32 = 2^64 ≡ 0 (mod 2^64)
0 > u32::MAX  →  false
→ Ok(Reduction(0, nil))   // cost reported as 0
```

The Python reference implementation would compute `cost = 2^64`, which is `> u32::MAX`, and reject the opcode as `EvalErr::Invalid`. The Rust implementation accepts it with cost 0. This is a concrete consensus divergence. [1](#0-0)
