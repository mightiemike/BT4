### Title
Duplicate-Leaf Merkle Phantom Index Forgery Allows False Inclusion Proof at Non-Existent `tx_index` — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion_v2` accepts a `tx_index` that points to the phantom duplicate position created by Bitcoin's odd-length Merkle tree padding. Because the contract stores only block headers (which carry no transaction count), it cannot bound-check `tx_index`. An unprivileged caller can prove that the last transaction of any odd-length block exists at index `N+1` (a position that holds no real transaction), and the function returns `true`.

### Finding Description

`merkle_proof_calculator` pads odd-length levels by appending a copy of the last element: [1](#0-0) 

`compute_root_from_merkle_proof` is a pure arithmetic function — it takes `(tx_hash, position, siblings)` and walks up the tree using only the parity of the position at each level: [2](#0-1) 

For a 3-tx block with root `R = H(H(tx0,tx1), H(tx2,tx2))`, the proof `[tx2, H(tx0,tx1)]` produces `R` for **both** `tx_index=2` and `tx_index=3`, because the first step in both cases computes `H(tx2, tx2)` — the only difference is whether `tx2` is the left or right operand, but since both operands are identical the result is the same.

`verify_transaction_inclusion_v2` performs three checks before delegating to `verify_transaction_inclusion`:

1. Proof lengths equal — satisfied (both depth-2 for a 3-tx block).
2. Coinbase proof validates against `header.block_header.merkle_root` — satisfied with a legitimate coinbase proof.
3. Tx proof validates against `header.block_header.merkle_root` — satisfied at the phantom index. [3](#0-2) 

`verify_transaction_inclusion` then performs only a Merkle root comparison with no `tx_index` upper-bound check: [4](#0-3) 

The project's own documentation acknowledges the root cause: "Block headers do not contain the transaction count, so proof depth cannot be validated on-chain." [5](#0-4) 

`ProofArgsV2` carries `tx_index` as a plain `u64` with no structural constraint: [6](#0-5) 

### Impact Explanation
Any downstream contract or application that calls `verify_transaction_inclusion_v2` and uses `(tx_block_blockhash, tx_index)` as a unique identifier — for example, to gate a one-time action such as a cross-chain bridge mint, a payment settlement, or a replay-prevention record — can be bypassed. The attacker proves the same real transaction at two distinct indices (the real index `N` and the phantom index `N+1`), obtaining two successful verifications for a single on-chain event. This directly violates the invariant that a verified `(blockhash, tx_index)` pair corresponds to exactly one transaction.

### Likelihood Explanation
The attack requires only knowledge of a real odd-length Bitcoin block (common — any block with an odd transaction count qualifies), the hash of its last transaction, and the ability to call the public NEAR contract method. No privileged role, no key material, and no chain manipulation is needed. The call is a read-only view function, so it costs only gas. The attack is fully constructible offline and submittable by any account.

### Recommendation
Since the block header contains no transaction count, an on-chain bounds check is not directly possible. The recommended mitigations are:

1. **Proof-length parity check**: Reject any proof where `tx_index >= (1 << proof.len())` — this does not fully prevent the duplicate-leaf case but narrows the attack surface.
2. **Odd-index rejection for the last leaf**: Require that if `tx_index` is odd, the supplied sibling at level 0 must differ from `tx_id`. This breaks the phantom proof because the phantom proof requires `proof[0] == tx_id`.
3. **Document and enforce at the application layer**: Callers must independently verify that `tx_index < tx_count` using the raw block data before trusting the inclusion result. This is the only fully reliable mitigation given the header-only storage model.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
