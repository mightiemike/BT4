### Title
`FeeEstimation` RPC Response Omits `l2_gas_consumed` While `overall_fee` Silently Includes L2 Gas Costs, Returning an Authoritative-Looking Wrong Value — (`crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

The `FeeEstimation` struct returned by `starknet_estimateFee` and `starknet_simulateTransactions` exposes `l2_gas_price` but omits `l2_gas_consumed`. Simultaneously, the `overall_fee` field and its OpenAPI description state the fee equals `gas_consumed * gas_price + data_gas_consumed * data_gas_price`, yet the actual value silently includes L2 gas costs. Any caller who trusts the documented formula to verify or decompose the fee receives an authoritative-looking wrong value.

---

### Finding Description

**`FeeEstimation` struct — missing field and wrong description** [1](#0-0) 

The struct carries `l2_gas_price` but no `l2_gas_consumed`. The inline comment at line 104 acknowledges the gap:

```rust
// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
// close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
```

The `overall_fee` doc-comment says:

> "The total amount of fee. This is equal to: gas_consumed \* gas_price + data_gas_consumed \* data_gas_price."

**`tx_execution_output_to_fee_estimation` — l2_gas silently baked into `overall_fee`** [2](#0-1) 

`overall_fee` is taken directly from `receipt.fee`. That fee is computed by `GasVector::cost()`, which sums all three gas dimensions: [3](#0-2) 

So `overall_fee` = `l1_gas * l1_gas_price + l1_data_gas * l1_data_gas_price + l2_gas * (l2_gas_price + tip)`, but the response only exposes the first two terms' consumed amounts.

**OpenAPI spec repeats the wrong formula** [4](#0-3) 

The `FEE_ESTIMATE` schema description says `overall_fee` equals `gas_consumed*gas_price + data_gas_consumed*data_gas_price` and does not list `l2_gas_consumed` as a field at all.

**Concrete divergence confirmed by test fixture**

The transaction prover's own test fixture shows what the response *should* contain: [5](#0-4) 

```json
"l1_gas_consumed": "0x0",
"l2_gas_consumed": "0xb56b6",   ← present in fixture, absent from FeeEstimation struct
"l2_gas_price":    "0x1dcd65000",
"overall_fee":     "0x151eb86f3ed400"
```

Arithmetic check:
- `l2_gas_consumed` = 0xb56b6 = 743,094
- `l2_gas_price` = 0x1dcd65000 = 8,000,000,000
- L2 cost = 5,944,752,000,000,000
- `l1_data_gas_consumed` = 0x80 = 128; `l1_data_gas_price` = 0x3e8 = 1,000 → 128,000
- `overall_fee` = 5,944,752,000,128,000 = 0x151eb86f3ed400 ✓

The documented formula (`gas_consumed * gas_price + data_gas_consumed * data_gas_price`) yields only **128,000**, not **5,944,752,000,128,000** — a factor of ~46 million off.

**RPC API surface** [6](#0-5) 

Both `estimateFee` and `simulateTransactions` return `Vec<FeeEstimation>` / `SimulatedTransaction { fee_estimation: FeeEstimation }` directly to external callers.

---

### Impact Explanation

Any client that calls `starknet_estimateFee` or `starknet_simulateTransactions` for a V3 `AllResources` transaction:

1. **Cannot reconstruct the fee breakdown**: `l2_gas_consumed` is absent, so the caller cannot derive the L2 gas amount needed to set a correct `l2_gas` resource bound in the subsequent real transaction.
2. **Receives a wrong authoritative formula**: The API spec and struct doc-comment state `overall_fee = gas_consumed*gas_price + data_gas_consumed*data_gas_price`. A caller who trusts this formula to verify the returned `overall_fee` will compute a value orders of magnitude smaller than the actual fee, causing silent confusion or incorrect resource-bound sizing.
3. **Downstream transaction failure or over-commitment**: A caller who sets `l2_gas.max_amount = 0` (the only value derivable from the response) will have their real transaction rejected with "Insufficient max L2Gas". A caller who guesses a large value over-commits resources.

This matches the allowed impact: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

Every V3 (`AllResources`) transaction submitted through `estimateFee` or `simulateTransactions` triggers this path. Since Starknet 0.13.3 made `AllResources` the standard transaction format, this affects the majority of fee estimation calls on any node running this sequencer.

---

### Recommendation

1. Add `l2_gas_consumed: Felt` to `FeeEstimation` in `crates/apollo_rpc_execution/src/objects.rs`.
2. Populate it in `tx_execution_output_to_fee_estimation`:
   ```rust
   l2_gas_consumed: gas_vector.l2_gas.0.into(),
   ```
3. Fix the `overall_fee` doc-comment and the OpenAPI `FEE_ESTIMATE` description to reflect the true formula including L2 gas.
4. Add `l2_gas_consumed` to the `required` array in the OpenAPI schema.

---

### Proof of Concept

```
1. Deploy a Cairo 1 contract on a node running this sequencer.
2. Call starknet_estimateFee with a V3 AllResources invoke transaction.
3. Observe the response:
   - "l2_gas_price" is present (e.g. "0x1dcd65000")
   - "l2_gas_consumed" is ABSENT
   - "overall_fee" is e.g. "0x151eb86f3ed400"
4. Apply the documented formula:
     gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price
   = 0 * 0xe8d4a51000 + 128 * 0x3e8
   = 128000
5. Compare to overall_fee = 5,944,752,000,128,000.
   The documented formula is off by a factor of ~46,000,000.
6. Attempt to submit the real transaction with l2_gas.max_amount derived
   from the response (only possible value: 0).
   Result: transaction rejected — "Insufficient max L2Gas".
```

pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}

    /// Computes the cost (in fee token units) of the gas vector (panicking on overflow).
    pub fn cost(&self, gas_prices: &GasPriceVector, tip: Tip) -> Fee {
        let tipped_l2_gas_price =
            gas_prices.l2_gas_price.checked_add(tip.into()).unwrap_or_else(|| {
                panic!(
                    "Tip overflowed: addition of L2 gas price ({}) and tip ({}) resulted in \
                     overflow.",
                    gas_prices.l2_gas_price, tip
                )
            });

        let mut sum = Fee(0);
        for (gas, price, resource) in [
            (self.l1_gas, gas_prices.l1_gas_price, Resource::L1Gas),
            (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
            (self.l2_gas, tipped_l2_gas_price, Resource::L2Gas),
        ] {
            let cost = gas.checked_mul(price.get()).unwrap_or_else(|| {
                panic!(
                    "{resource} cost overflowed: multiplication of gas amount ({gas}) by price \
                     per unit ({price}) resulted in overflow."
                )
            });
            sum = sum.checked_add(cost).unwrap_or_else(|| {
                panic!(
                    "Total cost overflowed: addition of current sum ({sum}) and cost of \
                     {resource} ({cost}) resulted in overflow."
                )
            });
        }
        sum
    }
