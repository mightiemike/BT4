Audit Report

## Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Non-Allowlisted Depositors to Bypass the Gate — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` is intended to gate token deposits by the actual depositor address. However, it silently discards its `sender` argument and checks `owner` (the LP position recipient) instead. Because `MetricOmmPool.addLiquidity` accepts any arbitrary `owner` with no requirement that `owner == msg.sender`, any non-allowlisted address can bypass the gate by supplying an allowlisted address as `owner`.

## Finding Description

`beforeAddLiquidity` ignores its first argument entirely and checks `owner`: [1](#0-0) 

The pool's `addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to `_beforeAddLiquidity`, with no enforcement that they match: [2](#0-1) 

Notably, `removeLiquidity` does enforce `msg.sender != owner` at L206, proving the design intent exists but was not applied to `addLiquidity`: [3](#0-2) 

The periphery's `_validateOwner` only rejects `address(0)` and does not enforce `owner == msg.sender`: [4](#0-3) 

**Exploit path:** Bob (non-allowlisted) calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")` directly, where `alice` is allowlisted. The pool calls `_beforeAddLiquidity(bob /*sender*/, alice /*owner*/, ...)`. The extension checks `allowedDepositor[pool][alice]` = `true` and passes. Bob's tokens enter the pool via the `metricOmmModifyLiquidityCallback`, and Alice receives an LP position she never requested. The allowlist gate is completely nullified.

## Impact Explanation

This is a broken admin-boundary: the pool admin's depositor restriction is fully bypassed by an unprivileged path. Any non-allowlisted address can inject tokens into a restricted pool by borrowing any allowlisted address as `owner`. This corrupts the pool's depositor invariant and allows unauthorized token deposits — a direct violation of the access control the extension is designed to enforce.

## Likelihood Explanation

`pool.addLiquidity` is a public function with no `owner == msg.sender` guard. The attack requires only knowing one allowlisted address (publicly discoverable via `AllowedToDepositSet` events or direct `allowedDepositor` reads) and implementing the `metricOmmModifyLiquidityCallback` interface. This is trivially achievable by any EOA wrapper contract or protocol integration.

## Recommendation

Change `beforeAddLiquidity` to validate `sender` (the actual payer/caller) instead of `owner` (the position recipient):

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

```solidity
function test_nonAllowlistedBypassViaOwner() public {
    // alice is allowlisted, bob is not
    depositExtension.setAllowedToDeposit(address(pool), alice, true);
    assertFalse(depositExtension.isAllowedToDeposit(address(pool), bob));

    // bob calls pool.addLiquidity directly with owner=alice
    vm.startPrank(bob);
    token0.approve(address(pool), type(uint256).max);
    token1.approve(address(pool), type(uint256).max);
    // bob implements metricOmmModifyLiquidityCallback to pay tokens
    pool.addLiquidity(alice, salt, deltas, callbackData, "");
    vm.stopPrank();

    // bob's tokens entered the pool, alice has LP position
    // allowlist gate was bypassed — bob (non-allowlisted) deposited
}
```

The `sender` passed to the extension is `bob`, but the extension checks `owner` = `alice` (allowlisted), so the gate passes and Bob's tokens enter the pool.
