### Title
`get_invoke_transaction_v3_hash` conditionally omits `proof_facts` from the hash preimage while the OS Cairo program always includes it — creating a hash-domain inconsistency for non-empty `proof_facts` - (File: `crates/starknet_api/src/transaction_hash.rs`)

### Summary

The Rust `get_invoke_transaction_v3_hash` function conditionally appends `proof_facts` to the hash preimage only when the field is non-empty. The Starknet OS Cairo program (`transaction_hash.cairo`) always appends `proof_facts` unconditionally when it is present in the transaction. This creates a structural inconsistency: a transaction with non-empty `proof_facts` hashes to a different value depending on which code path computes the hash, and a transaction with empty `proof_facts` produces the same hash as one that never had the field at all — meaning the hash does not commit to the presence or absence of the field.

### Finding Description

In `crates/starknet_api/src/transaction_hash.rs`, `get_invoke_transaction_v3_hash` builds the hash chain and then conditionally appends `proof_facts`:

```rust
// crates/starknet_api/src/transaction_hash.rs lines 399-403
if !transaction.proof_facts().0.is_empty() {
    let proof_facts_hash =
        HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
    hash_chain = hash_chain.chain(&proof_facts_hash);
}
```

The hash is therefore:
- **Without proof_facts**: `Poseidon(INVOKE, version, sender, fee_hash, paymaster_hash, chain_id, nonce, da_mode, account_deployment_data_hash, calldata_hash)`
- **With proof_facts**: `Poseidon(INVOKE, version, sender, fee_hash, paymaster_hash, chain_id, nonce, da_mode, account_deployment_data_hash, calldata_hash, proof_facts_hash)`

The Cairo OS program in `transaction_hash.cairo` (`compute_invoke_transaction_hash`) always hashes `proof_facts` via `poseidon_hash_update_with_nested_hash` when `proof_facts_size != 0`, which mirrors the Rust conditional. However, the backward-compatibility storage deserializer in `crates/apollo_storage/src/serialization/serializers.rs` silently defaults `proof_facts` to empty when the field is absent from the on-disk record:

```rust
// crates/apollo_storage/src/serialization/serializers.rs lines 1402-1408
// Backward compatibility: proof_facts may not exist in old transactions.
// If no data remains, default to empty; otherwise, deserialize normally.
let proof_facts = if data.is_empty() {
    ProofFacts::default()
} else {
    ProofFacts::deserialize_from(data)?
};
```

This means a transaction that was originally submitted with non-empty `proof_facts` and stored in the old format (before the field was added) will be deserialized with `proof_facts = []`. When the hash is recomputed from storage (e.g., for RPC `get_transaction_hash`, re-execution, or block hash verification), the hash will be computed **without** the `proof_facts` component, producing a different hash than the one originally accepted and stored in the block.

The analog to the external bug is exact: `getScore` returns 0 for non-participants (empty `proof_facts` → no hash contribution), but `getResult` calls `getScore` and may return a non-zero value (the OS re-execution path may accept a transaction whose stored hash no longer matches the recomputed hash). Here, the "score" is the transaction hash commitment: the hash function returns one value when `proof_facts` is present and a different value when it is absent, but the storage migration silently converts the former to the latter.

### Impact Explanation

**Wrong state / receipt / revert result from blockifier/syscall/execution logic for accepted input (Critical).**

A node that re-executes a block from storage will recompute transaction hashes from the deserialized `InvokeTransactionV3`. If the original transaction had non-empty `proof_facts` but was stored in the legacy format (or if the migration shim fires incorrectly), the recomputed hash will differ from the hash stored in the block header. This causes:

1. **Block hash verification failure** during re-execution or sync: the recomputed transaction hash fed into the block hash calculation will not match the committed block hash.
2. **Wrong `tx_hash` field** returned by RPC `get_transaction_by_hash` / `get_transaction_receipt` for any such transaction, since the hash is recomputed from storage.
3. **Incorrect executable transaction** constructed by `convert_internal_rpc_tx_to_executable_tx`: the `tx_hash` field of the `AccountTransaction::Invoke` will be wrong, causing the syscall `get_execution_info` to return the wrong transaction hash to the contract, which is observable on-chain.

### Likelihood Explanation

The backward-compatibility shim is explicitly marked as temporary (`TODO(AvivG): Remove once all nodes have re-synced`) and is active in production. Any node that has not yet re-synced and reads a legacy `InvokeTransactionV3` record that originally had non-empty `proof_facts` will silently drop the field. The gateway enforces that `proof_facts` and `proof` are either both present or both absent, and client-side proving is a live feature (`allow_client_side_proving: true` in the production gateway config). Therefore, transactions with non-empty `proof_facts` exist on-chain and in storage.

### Recommendation

1. **Remove the conditional hash inclusion**: `proof_facts` should always be included in the hash preimage, even when empty, using a canonical empty-hash sentinel (e.g., `Poseidon([])`) so that the hash commits to the field's presence or absence unambiguously. This matches how `paymaster_data` and `account_deployment_data` are always hashed regardless of whether they are empty.

2. **Fix the storage migration shim**: The backward-compatibility deserializer must not silently default `proof_facts` to empty for records that were stored after the field was introduced. A version tag or length prefix should distinguish old-format records from new-format records with an intentionally empty `proof_facts`.

3. **Re-verify stored hashes**: After fixing the hash function, audit all stored `InvokeTransactionV3` records with non-empty `proof_facts` to confirm their stored hashes match the canonical computation.

### Proof of Concept

1. Submit an `InvokeTransactionV3` with non-empty `proof_facts` through the gateway. The gateway accepts it and the `TransactionConverter` computes `tx_hash = get_invoke_transaction_v3_hash(tx_with_proof_facts)` which includes the `proof_facts_hash` component. This hash is stored in the block.

2. The node stores the transaction in MDBX using the `StorageSerde for InvokeTransactionV3` serializer, which writes `proof_facts` at the end of the record.

3. Simulate a node that was running the old binary (before `proof_facts` was added) and stored the transaction without the field. On upgrade, the backward-compatibility deserializer fires: `proof_facts = ProofFacts::default()` (empty).

4. The RPC layer calls `get_transaction_by_hash`. The storage reader returns the deserialized `InvokeTransactionV3` with `proof_facts = []`. The hash is recomputed: `get_invoke_transaction_v3_hash` skips the `proof_facts` branch because `proof_facts.is_empty()` is true. The returned `tx_hash` differs from the hash in the block header.

5. The blockifier re-executes the block from storage. The `AccountTransaction::Invoke` is constructed with the wrong `tx_hash`. The `get_execution_info` syscall returns the wrong hash to the contract. The block hash computed from the re-executed transactions does not match the committed block hash, causing sync failure.
