I found a very promising lead in `trie_key.rs` and `congestion_info.rs`. Let me investigate the `ShardId` type and the `allowed_shard` field truncation more carefully.

### Title
Silent Truncation of `ShardId` to `u16` in `CongestionInfoV1.allowed_shard` Breaks Congestion-Control Progress Guarantee — (`core/primitives-core/src/types.rs`, `core/primitives/src/congestion_info.rs`)

---

### Summary

`CongestionInfoV1.allowed_shard` is stored as `u16`, but `ShardId` is a `u64` newtype. The `Into<u16> for ShardId` implementation performs a silent truncating cast (`self.0 as u16`) with no bounds check. `finalize_allowed_shard` calls `self.set_allowed_shard(allowed_shard.into())` using this conversion. If any shard ID exceeds `u16::MAX`, the stored `allowed_shard` is silently wrong, and the congestion-control invariant — that exactly one shard is always permitted to forward receipts to a fully-congested shard — is broken.

---

### Finding Description

`ShardId` is defined as a `u64` newtype: [1](#0-0) 

The `Into<u16>` conversion is a bare truncating cast: [2](#0-1) 

`CongestionInfoV1`, which is Borsh-serialized into every `ChunkHeader`, stores the allowed shard as `u16`: [3](#0-2) 

`finalize_allowed_shard` selects the shard that may forward receipts during full congestion and writes it through the truncating conversion: [4](#0-3) 

The comparison that enforces the progress guarantee reads the stored `u16` back and widens it to `ShardId`: [5](#0-4) 

If the selected shard's ID is, say, `0x10001` (65537), `allowed_shard.into()` stores `1` (the low 16 bits). The comparison `sender_shard == ShardId::from(self.info.allowed_shard())` then matches shard `1` instead of shard `65537`. Shard `65537` is permanently blocked from forwarding receipts to the congested shard; shard `1` gets an unintended forwarding slot.

The same silent `as u16` path exists in `Into<u32> for ShardId`: [6](#0-5) 

---

### Impact Explanation

`CongestionInfoV1` is part of the on-chain `ChunkHeader` and is validated by every node. A silently wrong `allowed_shard` means:

1. **Progress guarantee broken**: The protocol guarantees that at least one shard can always send receipts to a fully-congested shard. With the truncated value, the intended shard is blocked and a different (possibly irrelevant) shard gets the slot.
2. **Receipt queue stall**: If the "allowed" shard happens to have no receipts to send, the congested shard's delayed-receipt queue grows without bound, stalling execution on that shard.
3. **Deterministic divergence**: Because `CongestionInfoV1` is part of the chunk header and is checked by stateless validators, all nodes compute the same wrong `allowed_shard`, so the chain does not fork — it silently operates with the wrong invariant.

---

### Likelihood Explanation

Current mainnet shard IDs are small integers (0–11), well within `u16::MAX`. However:

- `ProtocolFeature::DynamicResharding` is assigned to protocol version 85 and is already in the stable feature set. [7](#0-6) 

- `ShardId` is explicitly documented as an arbitrary `u64` that need not be sequential. [8](#0-7) 

- As dynamic resharding proceeds and shard IDs are assigned by `derive_v3`, there is no protocol-level constraint preventing a shard ID from eventually exceeding 65535. The truncation is latent today but becomes reachable as the shard count grows.

---

### Recommendation

Replace the silent `as u16` cast in `Into<u16> for ShardId` with a checked conversion, and change `CongestionInfoV1.allowed_shard` to `u64` (matching `ShardId`'s underlying type). Since `CongestionInfoV1` is Borsh-serialized into the protocol, this requires introducing `CongestionInfoV2` with a `u64 allowed_shard` field and gating it behind a new protocol version, exactly as the struct's own documentation instructs: [9](#0-8) 

Until the new version is deployed, add a `debug_assert!(shard_id.0 <= u16::MAX as u64)` in `finalize_allowed_shard` and a hard `panic!` / `Err` path in `set_allowed_shard` to prevent silent corruption.

---

### Proof of Concept

```rust
// In core/primitives-core/src/types.rs
impl Into<u16> for ShardId {
    fn into(self) -> u16 {
        self.0 as u16   // ← silent truncation, no bounds check
    }
}

// In core/primitives/src/congestion_info.rs
pub fn finalize_allowed_shard(
    &mut self,
    own_shard: ShardId,
    all_shards: &[ShardId],
    congestion_seed: u64,
) {
    let allowed_shard = Self::get_new_allowed_shard(own_shard, all_shards, congestion_seed);
    // If allowed_shard == ShardId(65537), this stores 1 (0x10001 & 0xFFFF)
    self.set_allowed_shard(allowed_shard.into());
}

// Later, in outgoing_gas_limit:
if sender_shard == ShardId::from(self.info.allowed_shard()) {
    // ShardId::from(1u16) == ShardId(1), not ShardId(65537)
    // → ShardId(65537) is permanently blocked; ShardId(1) gets the slot
    self.config.allowed_shard_outgoing_gas
} else {
    Gas::ZERO
}
```

A concrete trigger: with dynamic resharding active, construct a shard layout where `derive_v3` assigns a shard ID ≥ 65536. When that shard is selected as `allowed_shard` by `get_new_allowed_shard`, the stored `CongestionInfoV1.allowed_shard` is the low 16 bits of the real ID, and the progress guarantee silently fails.
