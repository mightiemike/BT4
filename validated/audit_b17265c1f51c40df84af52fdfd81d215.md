### Title
`topUpBaseRewards` Reads Fee-Collector Balance After Distribution Has Already Emptied It, Causing Inflated Shortfall and Rewards-Pool Over-Drain — (`File: x/tieredrewards/keeper/abci.go`)

---

### Summary

`topUpBaseRewards` in `x/tieredrewards/keeper/abci.go` reads the live fee-collector module-account balance to estimate how much the stakers will receive from normal block fees, then tops up any shortfall from the `RewardsPoolName` module account. If the Cosmos SDK distribution module's `BeginBlocker` runs **before** the tieredrewards `BeginBlocker` (the standard SDK ordering), the fee collector has already been emptied by distribution when line 73 executes. The balance snapshot is therefore `0`, the shortfall is computed as the full `targetStakersRewardPerBlock`, and the entire target amount is transferred from the rewards pool to distribution and re-allocated to validators — even though validators already received their normal fee-based rewards in the same block. The rewards pool is drained at the full target rate every block instead of only the true shortfall, and validators receive double rewards.

---

### Finding Description

`topUpBaseRewards` is called from `BeginBlocker` every block:

```
x/tieredrewards/keeper/abci.go  BeginBlocker → topUpBaseRewards
```

The critical read is:

```go
feeCollectorBalance := k.bankKeeper.GetBalance(ctx, feeCollectorAddr, bondDenom)   // line 73
defaultStakersRewardPerBlock := math.LegacyNewDecFromInt(feeCollectorBalance.Amount).
    MulTruncate(math.LegacyOneDec().Sub(communityTax))                              // line 74-75

shortFallAmount := targetStakersRewardPerBlock.Sub(defaultStakersRewardPerBlock).TruncateInt() // line 77
```

The intent is: if the fee collector already holds enough to meet `TargetBaseRewardsRate`, no top-up is needed; only the gap is funded from the rewards pool.

In Cosmos SDK v0.50 the standard `SetOrderBeginBlockers` places the distribution module before application-specific modules. Distribution's `BeginBlocker` calls `AllocateTokens`, which sweeps the entire fee-collector balance into the distribution module account and allocates it to validators. By the time tieredrewards's `BeginBlocker` runs, `feeCollectorBalance.Amount` is `0`.

Consequence:
- `defaultStakersRewardPerBlock = 0`
- `shortFallAmount = targetStakersRewardPerBlock` (the full target, not just the gap)
- Line 113 transfers the full target from `RewardsPoolName` to `distributiontypes.ModuleName`
- Lines 121-143 allocate that full amount to validators via `AllocateTokensToValidator`

Validators therefore receive:
1. Their normal share of block fees (distributed by the distribution `BeginBlocker`)
2. The full `targetStakersRewardPerBlock` from the rewards pool (distributed by tieredrewards)

The rewards pool is drained at the full target rate every single block, regardless of how much fee revenue was actually collected.

---

### Impact Explanation

- **Rewards pool premature depletion**: The `RewardsPoolName` module account is drained at the full target rate every block instead of only the shortfall. Under normal fee conditions (where fees already cover the target), the pool should not be touched at all, but it is fully consumed every block.
- **Validator double-payment**: Validators receive both their normal fee-based rewards and an additional full-target top-up from the rewards pool, constituting unauthorized asset movement out of the rewards pool.
- **Bonus reward insolvency for position holders**: The same `RewardsPoolName` account funds bonus rewards for tiered-position holders (see `processEventsAndClaimBonus`, line 239: `SendCoinsFromModuleToAccount(ctx, types.RewardsPoolName, ownerAddr, bonusCoins)`). Premature depletion of the pool causes `sufficientBonusPoolBalance` (bonus_rewards.go line 54-58) to return `ErrInsufficientBonusPool`, blocking legitimate bonus reward claims for all position holders.

---

### Likelihood Explanation

This occurs automatically every block in which `targetBaseRewardsRate > 0` and the distribution module's `BeginBlocker` precedes the tieredrewards `BeginBlocker` — the standard Cosmos SDK module ordering. No attacker action is required; normal block production is sufficient. The condition is deterministic and persistent.

---

### Recommendation

Do not read the fee-collector balance as a proxy for "what distribution will pay out." Instead, either:

1. **Run tieredrewards `BeginBlocker` before distribution's `BeginBlocker`** by placing it earlier in `SetOrderBeginBlockers` in `app/app.go`, so the fee-collector balance is still intact when line 73 executes; or
2. **Track the expected distribution amount independently** — e.g., snapshot the fee-collector balance at `EndBlocker` (before distribution runs) and store it for use in the next block's top-up calculation, avoiding reliance on a live balance that may already have been swept.

Option 1 is simpler but fragile to future reordering. Option 2 is more robust.

---

### Proof of Concept

**Normal block execution sequence (standard SDK ordering):**

1. Distribution `BeginBlocker` runs → `AllocateTokens` sweeps fee collector → fee collector balance = 0; validators credited their fee share.
2. Tieredrewards `BeginBlocker` runs → `topUpBaseRewards` reads fee collector balance = 0.
3. `defaultStakersRewardPerBlock = 0 * (1 - communityTax) = 0`.
4. `shortFallAmount = targetStakersRewardPerBlock - 0 = targetStakersRewardPerBlock`.
5. `topUpAmount = min(shortFallAmount, poolBalance)` → entire target transferred from `RewardsPoolName` to distribution.
6. Full target allocated to validators via `AllocateTokensToValidator`.

**Result per block:** validators receive `feeShare + targetStakersRewardPerBlock` instead of `max(feeShare, targetStakersRewardPerBlock)`. The rewards pool loses `targetStakersRewardPerBlock` tokens every block unconditionally. Over N blocks, the pool loses `N × targetStakersRewardPerBlock` tokens regardless of fee revenue, eventually hitting zero and blocking all bonus reward claims for tiered-position holders. 		return nil
	}

	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)

		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
