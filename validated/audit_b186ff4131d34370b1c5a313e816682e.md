Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, enabling allowlist bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool` sets to its own `msg.sender` — the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` intermediates the call, `sender` is the router contract, not the original user. A pool admin who allowlists the router to support router-mediated swaps for curated users inadvertently grants every unprivileged user access to the curated pool, completely defeating the allowlist invariant.

## Finding Description
**Root cause — pool passes its own `msg.sender` as `sender`:**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)` at line 231. When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)`, the pool's `msg.sender` is the router contract address. [1](#0-0) [2](#0-1) 

**Extension checks the router, not the user:**

`SwapAllowlistExtension.beforeSwap` receives `sender` (= router) and evaluates `allowedSwapper[msg.sender][sender]`, i.e., `allowedSwapper[pool][router]`. The original user's address (`tx.origin` or the router's `msg.sender`) is never inspected. [3](#0-2) 

**Bypass path:**

A pool admin who wants allowlisted users to use the router calls `setAllowedToSwap(pool, router, true)`. Once set, `allowedSwapper[pool][router] == true` and the guard passes for every caller of the router — including addresses never added to the allowlist. There is no restriction in `exactInputSingle` on who may call the router. [4](#0-3) 

**Contrast with `DepositAllowlistExtension`:**

`DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` — the LP position owner explicitly passed by the caller — not `sender` (the direct pool caller). `SwapAllowlistExtension` lacks this distinction. [5](#0-4) 

## Impact Explanation
A curated pool using `SwapAllowlistExtension` to restrict trading to a specific set of addresses is fully bypassed by any unprivileged user routing through `MetricOmmSimpleRouter`. The attacker executes swaps on the curated pool with the same output as an allowlisted user. This constitutes a broken core pool functionality (the curation/access-control guarantee) and an admin-boundary break: the pool admin's explicit access restriction is circumvented by an unprivileged path without any admin error or malicious setup. [6](#0-5) 

## Likelihood Explanation
The trigger requires the pool admin to call `setAllowedToSwap(pool, router, true)` — a natural, non-malicious operational step for any operator who wants their allowlisted users to be able to use the protocol's primary user-facing entry point. No privileged abuse is required. Any pool that (a) uses `SwapAllowlistExtension` and (b) enables router support is permanently vulnerable. The exploit is repeatable by any address with no special capability. [7](#0-6) 

## Recommendation
The extension must check the economically relevant actor — the original user — not the intermediary contract. Two viable approaches:

1. **Pass the originator via `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling `pool.swap`. The extension decodes and checks that address. This requires a convention between the router and the extension.
2. **Add an `originator` field to hook arguments at the pool level**: Set to `msg.sender` for direct calls and to the user-supplied address for router calls, so the extension always sees the true swapper identity.

The cleanest fix is option 2, mirroring how `DepositAllowlistExtension` uses `owner` (an explicit, caller-supplied identity) rather than `sender` (the direct pool caller). [5](#0-4) 

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension configured.
2. Pool admin calls:
     setAllowedToSwap(pool, Alice, true)   // Alice is the intended curated user
     setAllowedToSwap(pool, router, true)  // natural step to let Alice use the router
3. Charlie (not allowlisted) calls:
     router.exactInputSingle({pool: pool, recipient: Charlie, ...})
4. Router calls pool.swap(recipient=Charlie, ...) — pool's msg.sender = router.
5. Pool calls _beforeSwap(sender=router, recipient=Charlie, ...).
6. SwapAllowlistExtension checks allowedSwapper[pool][router] == true → passes.
7. Charlie's swap executes on the curated pool despite not being allowlisted.
```

The allowlist is silently defeated. Alice's exclusive access is lost the moment the admin enables router support. [1](#0-0) [8](#0-7)
