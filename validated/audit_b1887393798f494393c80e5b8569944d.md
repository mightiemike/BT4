### Title
Protocol Fee Balance Inflates `totalDeposits` Denominator, Understating Utilization and Suppressing Interest Rates - (File: `core/contracts/SpotEngineState.sol`)

---

### Summary

In `SpotEngineState._updateState`, the utilization ratio is computed as `totalBorrows / totalDeposits`, where `totalDeposits` is derived from `state.totalDepositsNormalized`. Because protocol fee accruals are credited to `FEES_ACCOUNT` via `_updateBalanceNormalized`, the fees balance is included in `totalDepositsNormalized`. These protocol-owned funds are not available for user borrowing, yet they inflate the denominator of the utilization ratio, causing the interest rate to be persistently lower than it should be.

---

### Finding Description

In `_updateState`, the utilization ratio is calculated as:

```solidity
int128 totalDeposits = state.totalDepositsNormalized.mul(
    state.cumulativeDepositsMultiplierX18
);
int128 totalBorrows = state.totalBorrowsNormalized.mul(
    state.cumulativeBorrowsMultiplierX18
);
int128 utilizationRatioX18 = totalBorrows.div(totalDeposits);
``` [1](#0-0) 

Later in the same function, protocol fees are accrued into `FEES_ACCOUNT` by calling `_updateBalanceNormalized(state, feesAccBalance, feesAmt)`:

```solidity
int128 feesAmt = totalDeposits.mul(
    totalDepositRateX18 - realizedDepositRateX18
);
// ...
if (feesAmt != 0) {
    BalanceNormalized memory feesAccBalance = balances[productId][FEES_ACCOUNT];
    _updateBalanceNormalized(state, feesAccBalance, feesAmt);
    _setBalanceAndUpdateBitmap(productId, FEES_ACCOUNT, feesAccBalance);
}
``` [2](#0-1) 

`_updateBalanceNormalized` unconditionally adds a positive `feesAmt` to `state.totalDepositsNormalized`:

```solidity
if (balance.amountNormalized > 0) {
    state.totalDepositsNormalized += balance.amountNormalized;
} else {
    state.totalBorrowsNormalized -= balance.amountNormalized;
}
``` [3](#0-2) 

Because `feesAmt > 0`, the `FEES_ACCOUNT` balance is always a positive (deposit-side) entry in `totalDepositsNormalized`. On every subsequent call to `_updateState`, `totalDeposits` includes the accumulated protocol fee balance. These fees are protocol-owned and not available for user borrowing, yet they inflate the denominator of `utilizationRatioX18`, causing it to be lower than the true utilization. A lower utilization feeds directly into a lower `borrowerRateX18` across all three branches of the interest rate model. [4](#0-3) 

---

### Impact Explanation

The true utilization should be `totalBorrows / (totalDeposits - feesBalance)`, because the fees balance is not lendable capital. By including it, the protocol systematically undercharges borrowers and underpays depositors. The distortion is self-reinforcing: lower interest rates generate less fee income per period, but the accumulated fee balance still grows over time, continuously widening the gap between reported and true utilization. This is a solvency/accounting corruption: the interest rate model is permanently miscalibrated relative to actual capital availability, causing depositors to receive less yield than the protocol's own rate model intends.

---

### Likelihood Explanation

This is triggered automatically on every call to `updateStates`, which is a routine sequencer operation. No special attacker action is required. The effect begins from the first interest accrual and compounds with every subsequent cycle as `FEES_ACCOUNT` grows. All spot products with non-zero `INTEREST_FEE_FRACTION` are affected. [5](#0-4) 

---

### Recommendation

Before computing `utilizationRatioX18`, subtract the `FEES_ACCOUNT` balance from `totalDeposits`:

```solidity
int128 feesBalance = balanceNormalizedToBalance(
    state,
    balances[productId][FEES_ACCOUNT]
).amount;

int128 availableDeposits = totalDeposits - feesBalance;
int128 utilizationRatioX18 = (availableDeposits == 0)
    ? 0
    : totalBorrows.div(availableDeposits);
```

This mirrors the Sentiment fix: exclude protocol-owned funds from the liquidity denominator so that the utilization ratio reflects only capital that is genuinely available to borrowers.

---

### Proof of Concept

**Setup:**
- Product has `totalDeposits = 10,000` (user deposits)
- Accumulated `FEES_ACCOUNT` balance = `1,500` (included in `totalDepositsNormalized`)
- `totalBorrows = 7,000`
- `interestInflectionUtilX18 = 0.8e18`

**Current (buggy) behavior:**
- `utilizationRatioX18 = 7000 / 10000 = 0.70`
- Utilization is below inflection → lower rate branch applies

**Correct behavior:**
- Available deposits = `10,000 - 1,500 = 8,500`
- `utilizationRatioX18 = 7000 / 8500 ≈ 0.824`
- Utilization is above inflection → higher rate branch applies, yielding a materially higher borrow rate

The difference is not rounding noise — it crosses the `interestInflectionUtilX18` threshold, switching the entire rate branch and producing a significantly lower interest rate than the model intends. Depositors are underpaid and borrowers are undercharged by a compounding margin that grows with every accrual cycle. [6](#0-5)
