### Title
Raw `approve` Calls With Non-Standard ERC20 Return Values Permanently Break Vault Asset Wrapping — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.wrapVaultAsset()` calls `assetToken.approve()` directly through the `IERC20Base` interface, which is declared to return `bool`. For tokens like USDT that return no value on `approve`, the Solidity ABI decoder reverts unconditionally. The protocol's own `ERC20Helper` library provides `safeTransfer` and `safeTransferFrom` to handle this exact class of non-standard token, but no `safeApprove` equivalent exists, and `wrapVaultAsset()` bypasses the safe wrappers entirely.

---

### Finding Description

`ContractOwner.wrapVaultAsset()` is the on-chain entry point for wrapping a raw ERC20 asset held in a Direct Deposit Account (DDA) into its corresponding ERC4626 vault token before crediting the deposit to the protocol. The function has no access control — any external caller may invoke it for any `(subaccount, productId)` pair.

The critical path is:

```solidity
// core/contracts/ContractOwner.sol  lines 526–532
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(assetTokenAddr));
IERC20Base assetToken = IERC20Base(assetTokenAddr);
assetToken.approve(tokenAddr, 0);          // ← raw interface call
assetToken.approve(tokenAddr, assetBalance); // ← raw interface call
IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1);
```

`IERC20Base.approve` is declared as:

```solidity
// core/contracts/interfaces/IERC20Base.sol  line 41
function approve(address spender, uint256 value) external returns (bool);
```

When Solidity calls a function declared to return `bool` and the callee returns zero bytes (as USDT and several other widely-deployed tokens do), the ABI decoder reverts. This is identical in mechanism to the M-04 finding, which concerned `transfer`; here the same class of defect applies to `approve`.

The protocol already recognises this problem for `transfer` and `transferFrom`: `ERC20Helper` wraps both with low-level calls that accept either a `bool` return or empty return data:

```solidity
// core/contracts/libraries/ERC20Helper.sol  lines 14–20
(bool success, bytes memory data) = address(self).call(
    abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
);
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
```

No equivalent `safeApprove` exists in `ERC20Helper`, and `wrapVaultAsset()` does not use any low-level call for its `approve` invocations. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

For any ERC4626 vault product whose underlying asset token does not return a boolean on `approve` (USDT being the canonical example):

- Every call to `wrapVaultAsset()` reverts at the first `approve` line.
- The `withdraw()` call that precedes it is part of the same transaction and therefore also reverts, so no tokens are moved — but the wrapping step can **never succeed**.
- Users who send such tokens to a DDA expecting them to be wrapped and credited as vault shares are permanently unable to complete the deposit flow through this path.
- The corrupted state is the vault-share balance that should have been credited to the subaccount: it remains at zero indefinitely while the raw asset sits idle in the DDA. [4](#0-3) 

---

### Likelihood Explanation

- `wrapVaultAsset()` is `external` with no access control; any user or keeper can trigger it.
- ERC4626 vaults backed by USDT or other non-boolean-returning tokens are a realistic and common deployment pattern.
- The failure is deterministic: every invocation for an affected token reverts, not just under edge conditions.
- No special attacker capability is required; a normal user attempting to complete a vault deposit is sufficient to observe the failure. [5](#0-4) 

---

### Recommendation

Add a `safeApprove` function to `ERC20Helper` following the same low-level-call pattern already used for `safeTransfer` and `safeTransferFrom`:

```solidity
function safeApprove(
    IERC20Base self,
    address spender,
    uint256 amount
) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
```

Replace the two raw `approve` calls in `wrapVaultAsset()` with `assetToken.safeApprove(tokenAddr, 0)` and `assetToken.safeApprove(tokenAddr, assetBalance)`. [6](#0-5) [7](#0-6) 

---

### Proof of Concept

1. Deploy a mock ERC20 token whose `approve` function performs the approval but returns no data (mimicking USDT).
2. Register an ERC4626 vault backed by this token as a spot product in the protocol.
3. Send a non-zero balance of the mock token to the DDA for any subaccount.
4. Call `ContractOwner.wrapVaultAsset(subaccount, productId)` from any EOA.
5. Observe the transaction reverts at `assetToken.approve(tokenAddr, 0)` with an ABI decoding error, despite the underlying transfer having been set up correctly.
6. Confirm the DDA balance is unchanged (transaction fully reverted) and the vault share balance for the subaccount remains zero.
7. Repeat indefinitely — the function can never succeed for this token. [8](#0-7)
