### Title
Fork Difficulty Computed Against Stale Mainchain Ancestor Instead of Fork Ancestor — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

### Summary

When a fork block lands on a difficulty-adjustment boundary, the contract fetches the start-of-period block's timestamp from the **mainchain** height map rather than from the fork's own ancestor chain. This is the direct analog of the Lybra rebase bug: a value recorded at one point in time (the mainchain's block at `height_first`) is used to govern a later computation (fork difficulty) even though the relevant state (the fork's actual ancestor at that height) has diverged. The result is that the expected difficulty for fork blocks at every retarget boundary is computed against the wrong reference block, allowing an adversary to submit fork blocks whose `bits` field satisfies an incorrectly derived target.

### Finding Description

`get_header_by_height` is implemented exclusively against `mainchain_height_to_header`:

```rust
// contract/src/lib.rs  lines 677-682
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

All three chain-specific difficulty routines call this function to obtain the timestamp of the first block in the retarget window:

**Bitcoin** (`contract/src/bitcoin.rs` lines 78–86):
```rust
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
```

**Litecoin** (`contract/src/litecoin.rs` lines 86–93) and **Dogecoin** (`contract/src/dogecoin.rs` lines 286–297, which even carries a developer `TODO` acknowledging the problem):
```rust
// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

When the block being validated is a **fork** block, `prev_block_header` is a fork header stored only in `headers_pool` (via `store_fork_header`), not in `mainchain_height_to_header`. If the fork diverged at or before `height_first`, the mainchain block at `height_first` is a completely different block from the fork's ancestor at that height. The contract silently uses the mainchain block's timestamp, producing a `first_block_time` that does not correspond to the fork's actual history.

### Impact Explanation

The difficulty retarget formula is:

```
actual_time_taken = prev_block_time − first_block_time   (clamped to [T/4, 4T])
new_target = prev_target × actual_time_taken / T
```

If the mainchain's `first_block_time` is earlier than the fork's true ancestor's timestamp, `actual_time_taken` is inflated, the computed target is higher (easier), and the contract accepts a fork block whose `bits` encodes a lower difficulty than the fork's actual history warrants. The attacker mines that single boundary block against the artificially easy target, then continues building the fork. If the fork's cumulative chainwork eventually exceeds the mainchain's, `reorg_chain` is triggered and the canonical chain is replaced with the attacker-controlled fork.

The difficulty adjustment is capped at 4× in each direction, so the maximum per-boundary advantage is a 4× reduction in required work. Over multiple retarget periods this compounds, and the attacker can choose the fork divergence point to maximise the discrepancy between the mainchain's `first_block_time` and the fork's.

### Likelihood Explanation

Any unprivileged NEAR caller can invoke `submit_blocks` with adversarial fork headers. The attacker must mine real PoW blocks, but the incorrectly computed (lower) difficulty directly reduces the cost of doing so. The Dogecoin build is most exposed because its retarget interval is 1 block (every block is a retarget), meaning every fork block beyond the divergence point is computed against the wrong mainchain ancestor. The developer `TODO` comment in `dogecoin.rs` confirms the team identified but did not resolve this issue.

### Recommendation

Replace `get_header_by_height` (mainchain-only lookup) with an ancestor-walk that follows `prev_block_hash` links backward from the fork tip until the target height is reached. This mirrors how Bitcoin Core's `GetAncestor` works. The walk is bounded by `difficulty_adjustment_interval` steps and is already performed implicitly for MTP computation in `get_median_time_past`, so the pattern is established in the codebase.

### Proof of Concept

1. Mainchain is at height 4031 (one block before the second Bitcoin retarget at 4032). The mainchain block at height 2016 (`first_block_height` for the next retarget) has `time = T_main`.
2. Attacker submits a fork that diverges at height 2015 (before `first_block_height = 2016`). The fork's ancestor at height 2016 has `time = T_fork`, where `T_fork > T_main` (the fork's block is newer).
3. Attacker builds 2016 fork blocks (heights 2016–4031) with timestamps set as large as the MTP and `MAX_FUTURE_BLOCK_TIME_LOCAL` checks allow, maximising `prev_block_time`.
4. Attacker submits fork block at height 4032 (retarget boundary). The contract calls `get_header_by_height(2016)` and retrieves the **mainchain** block with `time = T_main` instead of the fork's ancestor with `time = T_fork`.
5. Because `T_main < T_fork`, `actual_time_taken = prev_block_time − T_main > prev_block_time − T_fork`, so the computed target is higher (easier) than the fork's history justifies.
6. The attacker mines the boundary block against this easier target, submits it, and continues extending the fork. If cumulative chainwork exceeds the mainchain, `reorg_chain` fires and the contract's canonical chain is replaced.
