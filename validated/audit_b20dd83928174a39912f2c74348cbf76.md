Audit Report

## Title
`AnchoredPriceProvider` L1-Only Staleness Check Breaks All Swaps on Base Due to Missing Future Tolerance - (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

## Summary

`AnchoredPriceProvider` is the standard price provider for public pools and is deployed on Base (L2), but its `_isStale` function unconditionally rejects any oracle `refTime` ahead of `block.timestamp`. On Base, sequencer clock skew routinely causes oracle `refTime` to be slightly ahead of `block.timestamp`, which is precisely why `PriceProviderL2` was written with a `FUTURE_TOLERANCE` immutable. No `AnchoredPriceProviderL2` variant exists, leaving every pool using `AnchoredPriceProvider` on Base unable to execute swaps during any clock skew window.

## Finding Description

The protocol explicitly separates L1 and L2 staleness logic. `PriceProviderL2` defines a four-argument `_isStale` with a `FUTURE_TOLERANCE` immutable that tolerates oracle `refTime` up to `FUTURE_TOLERANCE` seconds ahead of `block.timestamp`:

```solidity
// PriceProviderL2.sol L135-150
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta, uint256 futureTol)
    internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) {
        return (refTime - nowTs) > futureTol;
    }
    return (nowTs - refTime) > maxDelta;
}
```

`AnchoredPriceProvider._isStale` is annotated `(L1)` and hard-rejects any future `refTime` at line 228:

```solidity
// AnchoredPriceProvider.sol L221-230
/// @dev Pure staleness check (L1). Any future refTime is stale.
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta)
    internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return true;   // ← broken on L2
    return (nowTs - refTime) > maxDelta;
}
```

The exploit path is fully confirmed in code:

1. `_readLeg` calls `_isStale(refTime, block.timestamp, MAX_REF_STALENESS)` at line 283. When the oracle publishes a price with `refTime = block.timestamp + N` (normal Base sequencer clock skew), this returns `true`.
2. `_readLeg` returns `ok = false`.
3. `_getBidAndAskPrice` returns `(0, type(uint128).max)` at line 260.
4. `getBidAndAskPrice()` reverts `FeedStalled()` at line 216.
5. `MetricOmmPool._getBidAndAskPriceX64` catches and reverts `PriceProviderFailed(reason)` at line 811, blocking `swap()`.

No `AnchoredPriceProviderL2.sol` exists in the codebase. The audit scope explicitly includes `AnchoredPriceProvider.sol` and `AnchoredProviderFactory.sol`.

## Impact Explanation

Every swap through any pool whose price provider is an `AnchoredPriceProvider` fails during any period of sequencer clock skew on Base. This is broken core pool functionality — the swap flow is completely unusable for the duration of the skew window. This matches the allowed impact gate: "Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows." LPs cannot exit via swap-based routes and traders cannot execute.

## Likelihood Explanation

Base is an explicitly listed supported chain (README line 10: "Ethereum, Base, HyperEVM"). Sequencer clock skew causing `refTime > block.timestamp` is a documented, recurring condition on optimistic rollups — it is precisely why `PriceProviderL2` was written with `FUTURE_TOLERANCE`. No privileged action or malicious setup is required; any ordinary swap call during a skew window triggers the failure. The condition is not "incorrect oracle data" — the oracle is publishing valid prices; the staleness check is simply not L2-aware.

## Recommendation

Create an `AnchoredPriceProviderL2` variant that adds a `FUTURE_TOLERANCE` immutable and replaces the three-argument `_isStale` with the four-argument version from `PriceProviderL2`, replacing the hard `if (refTime > nowTs) return true` guard with:

```solidity
if (refTime > nowTs) {
    return (refTime - nowTs) > FUTURE_TOLERANCE;
}
```

Alternatively, add a `futureTolerance` immutable directly to `AnchoredPriceProvider` (defaulting to 0 for L1 deployments) and pass it through `_isStale`, consistent with how `PriceProviderL2` already handles this.

## Proof of Concept

1. Deploy `AnchoredPriceProvider` on Base with any Pyth Lazer or Chainlink Data Streams feed.
2. The oracle publishes a price update with `refTime = block.timestamp + 2` (two-second sequencer clock skew, within normal Base operating range).
3. Call `pool.swap(...)`.
4. `_getBidAndAskPriceX64` calls `provider.getBidAndAskPrice()`.
5. Inside `_readLeg`, `_isStale(refTime=T+2, nowTs=T, MAX_REF_STALENESS)` hits `if (refTime > nowTs) return true` → stale.
6. `_readLeg` returns `ok = false`; `_getBidAndAskPrice` returns `(0, type(uint128).max)`.
7. `getBidAndAskPrice()` reverts `FeedStalled()`.
