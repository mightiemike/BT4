### Title
Unchecked `transferFrom` Return Value Enables Costless usdcE Drain from Direct Deposit Accounts — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the boolean return value. The project's own `ERC20Helper` library — imported and active via `using ERC20Helper for IERC20Base` in the same contract — provides `safeTransferFrom` that enforces this check, but it is bypassed here. If the USDC token returns `false` instead of reverting, the function proceeds to drain usdcE from the target DDA and deliver it to the caller without any USDC payment.

---

### Finding Description

`ContractOwner` imports `ERC20Helper` and activates it for `IERC20Base` at the top of the contract: [1](#0-0) 

`ERC20Helper` provides `safeTransferFrom`, which wraps the low-level call and requires both `success == true` and a truthy decoded return value: [2](#0-1) 

Despite this, `replaceUsdcEWithUsdc` calls `transferFrom` directly on line 616 without using `safeTransferFrom` and without checking the return value: [3](#0-2) 

The three-step sequence is:
1. **Line 616** — `transferFrom(msg.sender, directDepositV1, balance)`: pull USDC from caller (return value ignored).
2. **Line 617** — `withdraw(usdcE)`: transfer all usdcE from the DDA to `ContractOwner`.
3. **Line 618** — `safeTransfer(msg.sender, balance)`: deliver usdcE to the caller.

If step 1 silently returns `false`, steps 2 and 3 still execute. The caller receives usdcE without having paid USDC.

The same contract correctly uses `safeTransfer` on line 618 and in `withdrawFromDirectDepositV1` (line 642), making the omission on line 616 an inconsistency within the same function body.

---

### Impact Explanation

An unprivileged caller on the Ink chain (chainid 57073) can invoke `replaceUsdcEWithUsdc` for any subaccount whose DDA holds usdcE. If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` returns `false` on a failed `transferFrom` (e.g., insufficient allowance, non-standard implementation), the caller extracts the full usdcE balance of the target DDA at zero cost. The corrupted state delta is: usdcE balance of the DDA goes to zero; USDC balance of the DDA remains zero; caller's usdcE balance increases by `balance`.

---

### Likelihood Explanation

The function is `external` with no access modifier beyond a `block.chainid` check, making it reachable by any unprivileged caller on Ink. The concrete exploitability depends on whether the deployed USDC token returns `false` rather than reverting on failure. Standard Circle USDC implementations revert, which would prevent exploitation in the common case. However, the structural defect is present and the `ERC20Helper.safeTransferFrom` remedy is already available in scope — the omission is not intentional. Any token upgrade, proxy swap, or non-standard USDC variant that returns `false` would make this immediately exploitable.

---

### Recommendation

Replace the bare `transferFrom` call on line 616 with the `safeTransferFrom` method already provided by `ERC20Helper` and already active via `using ERC20Helper for IERC20Base`:

```solidity
// Before (line 616):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After:
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

Additionally audit the other unchecked `approve` calls in the same contract (`depositInsurance` line 254, `wrapVaultAsset` lines 530–531) and in `DirectDepositV1.creditDeposit` line 92, and add a `safeApprove` wrapper to `ERC20Helper` analogous to the existing `safeTransfer`/`safeTransferFrom`. [4](#0-3) [5](#0-4) 

---

### Proof of Concept

1. Attacker identifies a DDA (`directDepositV1Address[subaccount]`) holding a non-zero usdcE balance on Ink (chainid 57073).
2. Attacker does **not** approve `ContractOwner` to spend their USDC (or holds zero USDC).
3. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)`.
4. Line 616 executes `IERC20Base(usdc).transferFrom(attacker, dda, balance)` — if the USDC token returns `false` instead of reverting, execution continues.
5. Line 617 executes `DirectDepositV1(dda).withdraw(usdcE)` — usdcE is transferred from the DDA to `ContractOwner`.
6. Line 618 executes `IERC20Base(usdcE).safeTransfer(attacker, balance)` — usdcE is delivered to the attacker.
7. Result: attacker holds `balance` usdcE; DDA holds zero usdcE; no USDC was transferred. [6](#0-5)

```text
        require(block.chainid == 57073, ERR_UNAUTHORIZED);

        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
