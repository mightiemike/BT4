Audit Report

## Title
SwapAllowlistExtension Gates on Router Address Instead of End User, Allowing Any Caller to Bypass the Swap Allowlist via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which resolves to `msg.sender` of `MetricOmmPool.swap` — the router contract, not the end user. If a pool admin allowlists the router to support router-mediated swaps for their curated users, the check passes for every caller of the router, regardless of whether that caller is on the allowlist. The access-control invariant the extension is meant to enforce is structurally unenforceable for router-mediated swaps.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs the check:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension is called by the pool via `CallExtension.callExtension`), and `sender` is the first argument forwarded from `ExtensionCalling._beforeSwap`. [1](#0-0) 

`_beforeSwap` in `ExtensionCalling` passes `sender` directly into the encoded call to `IMetricOmmExtensions.beforeSwap`: [2](#0-1) 

And `sender` is set to `msg.sender` inside `MetricOmmPool.swap`: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` is called, the router calls `pool.swap(...)` directly: [4](#0-3) 

The pool sees `msg.sender = router`. The extension therefore checks `allowedSwapper[pool][router]`. If the pool admin has allowlisted the router — a natural step when they want to permit router-mediated swaps for their allowlisted users — the check passes for every caller of the router, regardless of whether that caller is on the allowlist. There is no mechanism in the extension or the call path to recover the original EOA caller; `msg.sender` stored by the router in transient storage (the `payer`) is used only for the payment callback and is never surfaced to the extension.

## Impact Explanation
LPs in a pool that uses `SwapAllowlistExtension` to restrict trading to a curated set of counterparties (e.g., a private OTC pool, a KYC-gated pool, or a pool restricted to a single market-maker) lose that protection entirely once the router is allowlisted. Any unprivileged user can execute swaps against the pool's liquidity, draining LP value through adverse selection or simply trading in ways the LP did not consent to. This is a direct loss of LP principal and breaks the core access-control invariant the extension is meant to enforce. This meets the "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" impact criteria.

## Likelihood Explanation
A pool admin who deploys a swap-allowlisted pool and also wants to support the standard periphery router will naturally add the router to the allowlist. The `SwapAllowlistExtension` interface gives no indication that doing so opens the gate to all callers. The mistake requires no special privileges to exploit — any EOA can call `MetricOmmSimpleRouter.exactInputSingle` with the target pool. The bypass is repeatable and requires zero on-chain setup beyond the admin's own natural configuration step.

## Recommendation
The extension must gate on the economic actor (the end user), not the intermediary. Two viable approaches:

1. **Router-aware extension**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a convention between router and extension.
2. **Transient storage read**: The router already stores the real caller (`payer`) in transient storage for the payment callback. Expose a view on the router that reads this value; the extension calls it when `sender` is a known router address.

The simplest safe default is to document clearly that allowlisting any intermediary contract (router, multicall, etc.) is equivalent to `allowAll = true`, and to provide a router-aware extension variant that reads the real caller.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is the intended swapper
  allowedSwapper[pool][router] = true  // admin adds router to support alice's router usage

Attack:
  bob (not on allowlist) calls:
    MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})

Execution trace:
  router.exactInputSingle()
    → pool.swap(recipient, zeroForOne, amount, ..., extensionData)
        msg.sender = router
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
            allowedSwapper[pool][router] == true  ✓  (no revert)
      → swap executes against LP liquidity
      → bob receives tokens

Result:
  bob swaps successfully despite not being on the allowlist.
  The LP's intended restriction is bypassed.
```

A Foundry integration test can confirm this by: (1) deploying a pool with `SwapAllowlistExtension`, (2) setting `allowedSwapper[pool][router] = true` and `allowedSwapper[pool][alice] = true` but not bob, (3) calling `exactInputSingle` as bob and asserting no revert.
