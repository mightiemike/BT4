### Title
Silent Stale-Rate Fallback in `ExchangeRateOracleClient::fetch_rate` Causes RPC Fee Estimation to Return Authoritative-Looking Wrong Values - (File: crates/apollo_l1_gas_price/src/exchange_rate_oracle.rs)

### Summary

`ExchangeRateOracleClient::fetch_rate` silently returns the **previous time-bucket's cached ETH→STRK rate** as `Ok(stale_rate)` whenever the current bucket's background query is still in-flight. The caller `get_l1_prices_in_fri_and_wei_and_conversion_rate` treats any `Ok` result as a valid live rate and uses it to compute FRI-denominated L1 gas prices for block proposals. Because the fallback is silent, the orchestrator's own error-handling fallback chain (previous-block info → configured minimums) is never reached. With `lag_interval_seconds = 900` in production, the stale rate can be up to 1 800 seconds (30 minutes) old. `starknet_estimateFee` and `starknet_simulateTransactions` consume these prices from the pending block and return authoritative-looking wrong fee values to every caller during the transition window.

---

The quantization formula maps the current wall-clock timestamp `T` to bucket `Q = (T − lag) / lag`. The background query for bucket `Q` fetches the rate at adjusted time `Q × lag`. When that query is still in-flight, the code falls back to bucket `Q − 1`, whose rate was fetched at `(Q − 1) × lag`. The fallback rate therefore corresponds to a market snapshot that is at least `lag_interval_seconds` seconds earlier than the current query's target, and up to `2 × lag_interval_seconds` seconds earlier than wall-clock time.

In production `lag_interval_seconds = 900` s, so the fallback rate can be up to **1 800 seconds (30 minutes) stale**. [2](#0-1)

The orchestrator's fallback chain (previous-block prices → configured minimums) is only entered when `eth_to_fri_rate.is_err()`. Because `fetch_rate` returns `Ok(stale_rate)`, the branch is never taken; the stale rate is used directly to compute `l1_gas_price_fri` and `l1_data_gas_price_fri` for the block proposal.

**L1 gas price provider also averages over 300 blocks with a 600 s lag** [4](#0-3) 

`get_price_info` computes the arithmetic mean of `base_fee_per_gas` and `blob_fee` over the last `number_of_blocks_for_mean = 300` L1 blocks, anchored at `timestamp − lag_margin_seconds (600 s)`. This is the sequencer's direct analog of the TWAP: a time-averaged price that structurally lags the spot price. Combined with the stale oracle rate, the FRI-denominated gas prices embedded in every block proposal can diverge substantially from true market values during volatile periods.

---

### Impact Explanation

Every call to `starknet_estimateFee` or `starknet_simulateTransactions` reads gas prices from the pending block, which is built using the stale FRI prices. The returned `FeeEstimation.l1_gas_price`, `l1_data_gas_price`, and `overall_fee` fields are wrong by a factor proportional to the ETH/STRK price movement over the staleness window (up to 30 minutes in production). Users who set `max_price_per_unit` based on these estimates may have transactions rejected at sequencing time if the true price has risen, or overpay if it has fallen. This matches the **High** impact: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

Additionally, the incorrect FRI prices are committed into accepted block proposals. Validators that independently recompute the expected FRI price from a live oracle will see a divergence, which can cause consensus disagreement or force validators to accept a block whose embedded gas prices do not reflect the true ETH→STRK rate at proposal time.

---

### Likelihood Explanation

The silent fallback fires at every `lag_interval_seconds` boundary — once every 900 seconds in production — for the duration of the background HTTP query (up to `query_timeout_sec = 10` s). This is a **predictable, recurring** window, not a rare edge case. Any user who calls `starknet_estimateFee` during this window receives a wrong value. During periods of high ETH/STRK volatility the magnitude of the error is amplified.

---

### Recommendation

1. **Tag the stale fallback explicitly.** Return a distinct result variant (e.g., `Ok(StaleFallback(rate, age_seconds))`) or a dedicated error so callers can decide whether to accept the stale value or escalate to the next fallback tier.

2. **Propagate staleness to the orchestrator.** In `get_l1_prices_in_fri_and_wei_and_conversion_rate`, treat a stale-rate response the same as an error so the previous-block-info fallback is used instead of a potentially 30-minute-old rate.

3. **Bound the acceptable staleness.** Add a configurable `max_rate_age_seconds` check: if the fallback rate's bucket age exceeds the threshold, return `QueryNotReadyError` rather than silently promoting it.

4. **Align with the TWAP recommendation.** For liquidation-equivalent decisions (e.g., fee validation at the gateway), use the most recent available spot rate rather than the averaged/lagged value. Reserve the averaged price for minting-equivalent operations where conservatism is appropriate.

---

### Proof of Concept

```
Production config:
  lag_interval_seconds = 900 s
  query_timeout_sec    = 10 s

Timeline:
  T = 0:      Bucket Q=1 query resolves, rate_1 = 1000 (ETH/STRK = 1000 FRI/WEI)
  T = 900:    New bucket Q=2 starts; background query for Q=2 is spawned.
  T = 900–910: Query for Q=2 is in-flight (not finished).
               fetch_rate(T=900) → Ok(rate_1 = 1000)   ← stale, no error
  T = 905:    ETH/STRK spikes +15% → true rate = 1150
  T = 910:    Query for Q=2 resolves, rate_2 = 1150.

During T=900–910 (10 s window, every 900 s):
  - get_l1_prices_in_fri_and_wei_and_conversion_rate receives Ok(1000)
  - l1_gas_price_fri = wei_price × 1000  (should be × 1150, i.e. 13% too low)
  - starknet_estimateFee returns overall_fee 13% below true cost
  - User sets max_price_per_unit = estimated value
  - Transaction is rejected at sequencing because actual FRI price > max_price_per_unit
```

Any unprivileged user can reproduce this by calling `starknet_estimateFee` immediately after a 900-second bucket boundary while the background oracle query is in-flight.
