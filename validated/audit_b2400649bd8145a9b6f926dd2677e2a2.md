### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas` When L2Gas and L1DataGas Are Zero, Producing a Different Transaction Hash — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a **value-based** heuristic (both `l2_gas` and `l1_data_gas` are zero) to decide between the `L1Gas` and `AllResources` variants. The serializer, however, always emits all three fields for both variants, making them indistinguishable in the wire format when L2Gas and L1DataGas are zero. A valid `AllResources` transaction with zero L2Gas and L1DataGas (but non-zero L1Gas) is accepted by the gateway, serialized to protobuf, and then silently re-typed as `L1Gas` by any P2P peer that deserializes it. Because `get_tip_resource_bounds_hash` hashes a different number of elements for the two variants, the peer computes a different transaction hash than the one the sequencer committed to, causing transaction-commitment mismatches and block rejection.

### Finding Description

**Serializer** (`crates/apollo_protobuf/src/converters/transaction.rs`, lines 471–490):

```rust
impl From<ValidResourceBounds> for protobuf::ResourceBounds {
    fn from(value: ValidResourceBounds) -> Self {
        match value {
            ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
                l1_gas: Some(l1_gas.into()),
                l2_gas: Some(value.get_l2_bounds().into()),   // zero
                l1_data_gas: Some(ResourceBounds::default().into()), // zero
            },
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
                protobuf::ResourceBounds {
                    l1_gas: Some(l1_gas.into()),
                    l2_gas: Some(l2_gas.into()),
                    l1_data_gas: Some(l1_data_gas.into()),
                },
        }
    }
}
```

When `AllResources` has `l2_gas = 0` and `l1_data_gas = 0`, the serialized bytes are **identical** to those of a `L1Gas` transaction.

**Deserializer** (`crates/apollo_protobuf/src/converters/transaction.rs`, lines 417–436):

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)          // ← AllResources silently becomes L1Gas
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}
```

**Hash function** (`crates/starknet_api/src/transaction_hash.rs`, lines 188–211):

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],                          // 2 elements total
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 elements total
        }
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```

For the same L1Gas value `X`, tip `T`, and zero L2Gas/L1DataGas:

| Variant | Hash preimage |
|---|---|
| `L1Gas(X)` | `poseidon([T, concat(X, L1_GAS), concat(0, L2_GAS)])` |
| `AllResources(X, 0, 0)` | `poseidon([T, concat(X, L1_GAS), concat(0, L2_GAS), concat(0, L1_DATA_GAS)])` |

These produce **different** Poseidon digests.

**Trigger path**:

1. User submits an `RpcInvokeTransactionV3` with `AllResourceBounds { l1_gas: NON_ZERO, l2_gas: 0, l1_data_gas: 0 }`. The gateway's stateless validator accepts this (test case `valid_l1_gas` confirms it).
2. The gateway converts it to `InvokeTransactionV3` with `ValidResourceBounds::AllResources(...)` and computes hash **H1** via the 3-element path.
3. The transaction is propagated over P2P as a `protobuf::InvokeV3` message.
4. The receiving node calls `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, which returns `ValidResourceBounds::L1Gas(l1_gas)` because both zero checks pass.
5. The receiving node computes hash **H2** via the 2-element path. **H1 ≠ H2**.
6. The block's `transaction_commitment` (a Merkle tree of transaction hashes) was built with H1. The syncing node recomputes it with H2, finds a mismatch, and rejects the block.

### Impact Explanation

This is a **transaction hash domain boundary violation** at the public-to-internal conversion layer. The wrong hash is bound to the transaction after the protobuf round-trip. Any node that receives the block over P2P and recomputes transaction hashes will derive a different `transaction_commitment` than the one committed by the sequencer, causing the block to be rejected. This matches:

> **High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

and potentially:

> **Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.**

### Likelihood Explanation

Any unprivileged user can submit a valid `AllResources` transaction with only L1Gas set (L2Gas=0, L1DataGas=0). The gateway accepts it. The trigger requires no special privileges, no malformed bytes, and no peer-only behavior — it is a normal RPC submission that exercises a standard code path.

### Recommendation

The serializer must encode the variant discriminant unambiguously. The simplest fix is to **not** emit `l1_data_gas` for `L1Gas` transactions (leave it as `None`), and update the deserializer to treat `None` as the `L1Gas` variant and `Some(zero)` as `AllResources`:

```rust
// Serializer: L1Gas case
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),
    l1_data_gas: None,   // ← absence signals L1Gas variant
},

// Deserializer
Ok(match value.l1_data_gas {
    None => ValidResourceBounds::L1Gas(l1_gas),
    Some(_) => ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }),
})
```

This aligns with the existing TODO comment (`// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2`) and with the JSON deserialization, which already uses presence (key existence) rather than value to distinguish the two variants.

### Proof of Concept

```
1. Craft RpcInvokeTransactionV3 with:
     resource_bounds = AllResourceBounds {
         l1_gas:      ResourceBounds { max_amount: 1000, max_price_per_unit: 1 },
         l2_gas:      ResourceBounds { max_amount: 0,    max_price_per_unit: 0 },
         l1_data_gas: ResourceBounds { max_amount: 0,    max_price_per_unit: 0 },
     }

2. Gateway accepts it (validate_resource_bounds passes because l1_gas is non-zero).
   Gateway computes H1 = get_invoke_transaction_v3_hash(AllResources path)
   → tip_resource_bounds_hash uses 3 resource felts (L1, L2, L1Data).

3. Transaction is serialized to protobuf::InvokeV3.
   From<ValidResourceBounds> for protobuf::ResourceBounds emits:
     l1_gas = Some(1000/1), l2_gas = Some(0/0), l1_data_gas = Some(0/0)

4. Receiving P2P node calls TryFrom<protobuf::ResourceBounds> for ValidResourceBounds:
     l1_data_gas.is_zero() && l2_gas.is_zero() → true
     → returns ValidResourceBounds::L1Gas(l1_gas)

5. Receiving node computes H2 = get_invoke_transaction_v3_hash(L1Gas path)
   → tip_resource_bounds_hash uses 2 resource felts (L1, L2).

6. H1 ≠ H2.
   Block transaction_commitment built with H1 ≠ transaction_commitment recomputed with H2.
   Syncing node rejects the block.
```
