### Title
Single-Step `LinkSigner` Operation Grants Immediate Signing Authority Without New-Signer Confirmation — (`File: core/contracts/EndpointTx.sol`)

---

### Summary
The `LinkSigner` transaction type immediately and irrevocably activates a new linked signer for a subaccount in a single step, with no confirmation required from the designated signer address. If a user accidentally specifies an attacker-controlled address, the attacker gains full signing authority over the subaccount instantly and can drain collateral before the user can issue a corrective transaction. Because both the original owner and the linked signer share the same per-subaccount nonce, the attacker can also front-run the user's recovery `LinkSigner` transaction, making asset loss effectively irrecoverable.

---

### Finding Description

Both the fast-mode and slow-mode paths in `EndpointTx.sol` write the new linked signer directly into `linkedSigners` in a single step with no pending/accept pattern and no confirmation from the incoming signer:

**Fast-mode path** (`processTransactionImpl`):

```solidity
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

**Slow-mode path** (`processSlowModeTransactionImpl`):

```solidity
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [2](#0-1) 

The moment this write lands, `validateSignature` in `Verifier.sol` accepts signatures from either the subaccount owner **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

Critically, `validateSignedTx` for `LinkSigner` passes `allowLinkedSigner = true`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // allowLinkedSigner
);
``` [4](#0-3) 

This means the attacker (now the linked signer) can immediately sign a second `LinkSigner` transaction to rotate the linked signer to a different address they control, consuming nonce N+1 before the victim can. Nonces are keyed per-subaccount address:

```solidity
nonce == nonces[address(uint160(bytes20(sender)))]++
``` [5](#0-4) 

Because the original owner and the linked signer share the same nonce counter for the same `sender` subaccount, the attacker's front-run consumes the nonce the victim's recovery transaction would need, causing the victim's transaction to revert with `ERR_WRONG_NONCE`.

---

### Impact Explanation

An attacker who controls the accidentally-linked address can:

1. Sign and submit a `WithdrawCollateral` transaction to drain all collateral from the subaccount.
2. Simultaneously sign and submit a new `LinkSigner` transaction (consuming the next nonce) to rotate the linked signer to a fresh attacker address, blocking the victim's recovery attempt.

The collateral loss is irrecoverable once the withdrawal is processed. The `linkedSigners` state is corrupted in a way that the victim cannot unilaterally undo if the attacker front-runs the corrective transaction.

---

### Likelihood Explanation

Medium-low. The trigger is a user error — specifying an incorrect `signer` address in a `LinkSigner` transaction (e.g., clipboard poisoning, copy-paste mistake, wrong address from a UI). This is the exact class of mistake the original report identifies as realistic for single-step critical operations. No privileged access, sequencer compromise, or social engineering is required beyond the user's own transaction.

---

### Recommendation

Implement a two-step linked-signer procedure:

1. **Propose**: The subaccount owner submits a `ProposeLinkSigner` transaction that stores a `pendingLinkedSigner[sender]` but does not activate it.
2. **Accept**: The proposed signer must submit a signed `AcceptLinkSigner` transaction (proving they control the key) before `linkedSigners[sender]` is updated.

This ensures that a typo or wrong address can never grant signing authority, because the proposed address must demonstrate key control before activation.

---

### Proof of Concept

1. Alice holds a subaccount with 50,000 USDC collateral.
2. Alice intends to link her hardware wallet (`0xAlice_HW`) but accidentally pastes Bob's address (`0xBob`) into the `signer` field of a `LinkSigner` transaction.
3. Alice signs and submits the transaction. The sequencer processes it; `linkedSigners[alice_subaccount] = 0xBob` is written immediately with no validation.
4. Bob (who controls `0xBob`) observes the on-chain state change and immediately signs two transactions with nonces N and N+1:
   - **Nonce N**: `WithdrawCollateral` for 50,000 USDC to Bob's external address.
   - **Nonce N+1**: `LinkSigner` rotating the linked signer to `0xBob2` (another Bob-controlled address).
5. Bob submits both to the sequencer API. The sequencer processes them as valid signed transactions.
6. Alice attempts to send a corrective `LinkSigner` with nonce N+1, but it reverts with `ERR_WRONG_NONCE` because Bob already consumed it.
7. Alice's 50,000 USDC is gone and the linked signer slot is now permanently under Bob's control.
