### Title
Caller-Controlled Zero-Confirmation Bypass Allows SPV Proof Acceptance Before Reorg Safety — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations: u64` field with no contract-enforced minimum. Any unprivileged NEAR caller can pass `confirmations: 0`, causing the contract to return `true` for a transaction whose block has zero depth on the main chain and is trivially reorgable. A higher-level protocol (bridge, DeFi settlement) that trusts this result can be drained before the attacker submits a competing fork that reorgs the block out.

---

### Finding Description

`ProofArgs` and `ProofArgsV2` (defined in `btc-types/src/contract_args.rs`) both expose a `confirmations: u64` field that is entirely caller-controlled. [1](#0-0) 

Inside `verify_transaction_inclusion`, the contract enforces exactly two constraints on this field:

1. An **upper bound**: `args.confirmations <= self.gc_threshold` — prevents asking for more confirmations than blocks stored.
2. An **availability check**: `(heaviest_block_header.block_height).saturating_sub(target_block_height) + 1 >= args.confirmations` — ensures the chain is deep enough to satisfy the requested count. [2](#0-1) 

There is **no minimum floor**. When a caller passes `confirmations: 0`, the availability check evaluates to `(any_u64_value) >= 0`, which is always `true` for `u64`. The function proceeds to verify the Merkle proof and returns `true` for a transaction whose block sits at the chain tip with zero depth.

`verify_transaction_inclusion_v2` inherits the same flaw — it validates the coinbase Merkle proof and then delegates directly to `verify_transaction_inclusion` with the caller's original `confirmations` value unchanged. [3](#0-2) 

The contract's own test suite confirms `confirmations: 0` is accepted as valid input: [4](#0-3) 

The contract also implements a full chain reorganization mechanism: when a fork accumulates more `chain_work` than the current main chain, `reorg_chain` is triggered, demoting previously main-chain blocks and promoting fork blocks. This means a block that was on the main chain — and for which `verify_transaction_inclusion` returned `true` — can be removed from the main chain after a subsequent `submit_blocks` call. [5](#0-4) 

---

### Impact Explanation

A higher-level NEAR protocol (bridge, cross-chain settlement contract) that calls `verify_transaction_inclusion_v2` and acts on a `true` result will accept a Bitcoin transaction as final when it has zero confirmations. The attacker then submits a competing fork with more accumulated chain work via `submit_blocks`. The contract's `reorg_chain` logic promotes the fork, removing the original block from `mainchain_header_to_height`. The transaction is no longer on the main chain, but the higher-level protocol has already released funds or updated state based on the forged `true` result.

The corrupted value is the **SPV proof result** (`true`) returned for a block that is subsequently evicted from the canonical chain — directly analogous to the Fortuna seed being revealed before the corresponding request is finalized.

---

### Likelihood Explanation

**Medium.** The `confirmations: 0` bypass is reachable by any unprivileged NEAR caller with no special role or stake. The attacker additionally needs to submit a competing fork with more chain work. For Bitcoin mainnet this requires significant hash power; for Dogecoin or Litecoin (also supported via feature flags) the required hash rate is substantially lower. The attack is most realistic against smaller supported chains or during periods of low network hash rate. The `submit_blocks` entry point is open to any caller who can pay the storage deposit. [6](#0-5) 

---

### Recommendation

Enforce a contract-level minimum confirmation floor inside `verify_transaction_inclusion`. The minimum should be a stored constant or a constructor parameter (e.g., `min_confirmations: u64`) that reflects the reorg safety depth for each supported chain. Reject any call where `args.confirmations < self.min_confirmations`:

```rust
require!(
    args.confirmations >= self.min_confirmations,
    "Requested confirmations below the safety minimum"
);
```

For Bitcoin mainnet, 6 is the conventional minimum; for Dogecoin/Litecoin, a higher value (e.g., 20–100) is appropriate given lower hash rates. The minimum should not be overridable by the caller.

---

### Proof of Concept

1. Deploy the contract with `skip_pow_verification: true` and `gc_threshold: 100`.
2. Submit a block header `B` containing transaction `T` via `submit_blocks`. Block `B` is now the chain tip.
3. Call `verify_transaction_inclusion_v2` with:
   - `tx_block_blockhash = B.block_hash()`
   - valid Merkle proof for `T`
   - `confirmations: 0`
   - Result: **`true`** — the higher-level protocol releases funds.
4. Construct a competing fork `B'` at the same height as `B` but with higher `chain_work` (trivially achievable with `skip_pow_verification: true`; on real networks, requires hash power). Submit `B'` via `submit_blocks`.
5. `reorg_chain` fires: `B` is removed from `mainchain_header_to_height`; `B'` becomes the tip.
6. Call `verify_transaction_inclusion_v2` again with the same args — it now **panics** with `"block does not belong to the current main chain"`.
7. Transaction `T` is no longer on the canonical chain; funds were already released in step 3. [7](#0-6) [8](#0-7)

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }

    fn reorg_chain(&mut self, fork_tip_header: ExtendedHeader, last_main_chain_block_height: u64) {
        let fork_tip_height = fork_tip_header.block_height;
        if last_main_chain_block_height > fork_tip_height {
            // If we see that main chain is longer than fork we first garbage collect
            // outstanding main chain blocks:
            //
            //      [m1] - [m2] - [m3] - [m4] <- We should remove [m4]
            //     /
            // [m0]
            //     \
            //      [f1] - [f2] - [f3]
            for height in (fork_tip_height + 1)..=last_main_chain_block_height {
                let current_main_chain_blockhash = self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str("cannot get a block"));
                self.remove_block_header(&current_main_chain_blockhash);
                self.mainchain_height_to_header.remove(&height);
            }
        }

        // Now we are in a situation where mainchain is equivalent to fork size:
        //
        //      [m1] - [m2] - [m3] - [m4] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip
        //
        //
        // Or in a situation where it is shorter:
        //
        //      [m1] - [m2] - [m3] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip

        let fork_tip_hash = fork_tip_header.block_hash.clone();
        let mut fork_header_cursor = fork_tip_header;

        while !self
            .mainchain_header_to_height
            .contains_key(&fork_header_cursor.block_hash)
        {
            let prev_block_hash = fork_header_cursor.block_header.prev_block_hash;
            let current_block_hash = fork_header_cursor.block_hash;
            let current_height = fork_header_cursor.block_height;

            // Inserting the fork block into the main chain, if some mainchain block is occupying
            // this height let's save its hashcode
            let main_chain_block = self
                .mainchain_height_to_header
                .insert(&current_height, &current_block_hash);
            self.mainchain_header_to_height
                .insert(&current_block_hash, &current_height);

            // If we found a mainchain block at the current height than remove this block from the
            // header pool and from the header -> height map
            if let Some(current_main_chain_blockhash) = main_chain_block {
                self.remove_block_header(&current_main_chain_blockhash);
            }

            // Switch iterator cursor to the previous block in fork
            fork_header_cursor = self
                .headers_pool
                .get(&prev_block_hash)
                .unwrap_or_else(|| env::panic_str("previous fork block should be there"));
        }

        // Updating tip of the new main chain
        self.mainchain_tip_blockhash = fork_tip_hash;
    }
