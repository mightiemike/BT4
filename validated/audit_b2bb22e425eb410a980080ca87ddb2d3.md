Audit Report

## Title
Router `pay()` Consumes Unattributed Native ETH Balance for WETH Swaps, Enabling Theft of Stranded ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` reads `address(this).balance` as the available ETH for WETH settlement rather than the current caller's `msg.value`. When a user overpays ETH relative to `amountIn` and omits `refundETH()`, the excess ETH is stranded on the router with no owner record. Any subsequent WETH swap — including one with `msg.value = 0` — will consume that stranded ETH, giving the attacker a free swap while the original depositor loses their ETH. The public `refundETH()` function provides a direct theft path for the same stranded balance.

## Finding Description

In `PeripheryPayments.pay()`, the WETH branch reads the router's total native balance:

```solidity
uint256 nativeBalance = address(this).balance;   // entire router balance, not msg.value
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
```

`nativeBalance` is never scoped to the current transaction's `msg.value`. The `receive()` guard at lines 32–34 only blocks plain ETH transfers from non-WETH addresses; it does not block ETH arriving via `msg.value` in the four payable swap entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`), all of which are `payable`.

**Exploit path:**

1. User A calls `exactInputSingle{value: 1000}(tokenIn=WETH, amountIn=500, ...)` directly (not via `multicall`). Inside `pay()`, `nativeBalance = 1000 >= value = 500`, so 500 ETH is deposited as WETH and sent to the pool. 500 ETH remains on the router with no owner record.
2. User B calls `exactInputSingle{value: 0}(tokenIn=WETH, amountIn=500, ...)`. Inside `pay()`, `nativeBalance = 500 >= value = 500`, so User A's 500 ETH is deposited as WETH and sent to the pool. `safeTransferFrom(User B, pool, 0)` is never reached. User B pays nothing and receives the full swap output.
3. Alternatively, any address calls `refundETH()` after step 1 to sweep the 500 ETH directly to themselves.

The transient callback context (`_setNextCallbackContext`) stores pool, callback mode, payer, and token, but **not** the ETH amount contributed by the current caller. There is no per-transaction ETH cap enforced in `pay()`.

The protocol's own test suite (`test_multicall_ethInput_exactInputSingle_refundsUnusedEth`) demonstrates that excess ETH is expected to be reclaimed via `refundETH()` in a `multicall`, but this is optional and not enforced for direct calls to `exactInputSingle`, `exactOutputSingle`, `exactInput`, or `exactOutput`.

## Impact Explanation

Direct loss of user principal (High). User A loses ETH equal to `msg.value − amountIn` — the overpayment amount — which is transferred to the pool on behalf of User B or swept by any caller of `refundETH()`. The pool's accounting is correct (it received the right WETH amount), so the loss is entirely borne by User A. The attacker receives a free swap or free ETH with no special privileges required.

## Likelihood Explanation

Medium-to-High. All four swap entry points are `payable` and accept arbitrary `msg.value`. For exact-output swaps, the exact input is unknown until the callback fires, so users routinely send a safety-margin ETH amount. The protocol's own tests (`test_mixedNativeAndWeth_exactOutputSingle_wethForToken`) confirm the mixed native+WETH pattern is an intended design. Users who call swap functions directly (not via `multicall`) have no automatic refund mechanism. An attacker can monitor the router's ETH balance on-chain and immediately follow any transaction that leaves ETH stranded, or call `refundETH()` directly.

## Recommendation

1. **Track per-transaction ETH in transient storage.** At the entry of each payable swap function, record `msg.value` in a transient slot. In `pay()`, consume only up to that recorded amount rather than `address(this).balance`.
2. **Auto-refund excess ETH.** After each swap function completes, transfer any remaining `address(this).balance` back to `msg.sender`, or require callers to use `multicall` + `refundETH()` and enforce this at the contract level.
3. **Restrict `refundETH()` to the original depositor.** Record the depositor in transient storage and enforce it in `refundETH()`, or remove the public helper and rely on the auto-refund above.

## Proof of Concept

```
State before:
  router.balance = 0
  User A WETH balance = 0
  User B WETH balance = 1000

Step 1 — User A overpays (no multicall, no refundETH):
  User A calls exactInputSingle{value: 1000}(tokenIn=WETH, amountIn=500, ...)
  → pay() sees nativeBalance=1000 >= value=500
  → deposits 500 ETH as WETH, transfers to pool
  → 500 ETH remains on router

Step 2 — Attacker exploits:
  User B calls exactInputSingle{value: 0}(tokenIn=WETH, amountIn=500, ...)
  → pay() sees nativeBalance=500 >= value=500
  → deposits User A's 500 ETH as WETH, transfers to pool
  → safeTransferFrom(User B, pool, 0) never called
  → User B pays 0, receives full swap output

Alternative Step 2 — Direct theft:
  Any address calls refundETH()
  → sweeps 500 ETH to msg.sender

Final state:
  router.balance = 0
  User A: lost 1000 ETH (500 for own swap + 500 stolen)
  User B / thief: received swap output or 500 ETH, paid nothing
  Pool: received correct WETH — no pool-side anomaly
```

Foundry test plan: fork the router, fund User A, call `exactInputSingle{value: 1000}(amountIn=500)` without `refundETH`, assert `router.balance == 500`, then call `refundETH()` from a fresh address and assert that address received 500 ETH.
