### Title
Fork Retarget Difficulty Computed from Mainchain Ancestor Instead of Fork Ancestor — (`contract/src/bitcoin.rs`, `contract/src/lib.rs`)

---

### Summary

`get_next_work_required` fetches the retarget-interval start block via `blocks_getter.get_header_by_height(first_block_height)`. The sole implementation of that trait method reads from `mainchain_height_to_header`, a map that is **never written to during fork submission**. When a fork diverges before the retarget-interval start height, the function silently uses the mainchain's start block instead of the fork's, producing an incorrect expected difficulty. A trusted relayer can exploit this to submit a fork whose retarget block carries an artificially easy `bits` value, pass `check_pow`, accumulate more chain work, and trigger `reorg_chain` to corrupt `mainchain_tip_blockhash`.

---

### Finding Description

**Step 1 — Retarget path in `get_next_work_required`**

When `(prev_block_header.block_height + 1) % difficulty_adjustment_interval == 0`, the function computes:

```
first_block_height = prev_block_header.block_height - (difficulty_adjustment_interval - 1)
```

and then fetches the interval-start block:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [1](#0-0) 

**Step 2 — `get_header_by_height` reads only the mainchain map**

```rust

        .and_then(|hash| self.headers_pool.get(&hash))

**Step 3 — Fork blocks are never written to `mainchain_height_to_header`**

`store_fork_header` inserts only into `headers_pool`:

```rust

    self.headers_pool.insert(&header.block_hash, header);

``` [3](#0-2) 

Compare with `store_block_header`, which writes to both maps: [4](#0-3) 

**Step 4 — PoW check fires before any fork storage**

`check_target` (and therefore `check_pow` → `get_next_work_required`) is called at line 518, before `submit_block_header_inner` at line 528. At that moment `mainchain_height_to_header` still holds only mainchain blocks. [5](#0-4) 

**Consequence:** For any fork that diverges at height `D < first_block_height`, `get_header_by_height(first_block_height)` returns the mainchain block at that height, not the fork's ancestor. The timestamp used in `calculate_next_work_required` is therefore wrong, producing an incorrect `expected_bits`. [6](#0-5) 

---

### Impact Explanation

A trusted relayer constructs a fork whose block at `first_block_height` carries a timestamp earlier than the mainchain's block at the same height. This makes the mainchain-derived `actual_time_taken` larger, which lowers the computed difficulty (raises the target). The relayer then submits a retarget block with `bits` matching this artificially easy mainchain-derived target. `check_pow` accepts it because `expected_bits == block_header.bits` is satisfied using the wrong reference block. With a lower difficulty, each fork block contributes less work per `work_from_bits`, but the relayer can extend the fork further to accumulate enough total chain work to exceed the mainchain tip. `reorg_chain` then replaces `mainchain_tip_blockhash` and rewrites `mainchain_height_to_header`, corrupting the canonical chain state. [7](#0-6) 

---

### Likelihood Explanation

The attack requires a trusted-relayer key, which is an explicitly listed in-scope entry point. No external dependency, social engineering, or DAO compromise is needed. The relayer only needs to submit well-formed headers with a crafted timestamp at the retarget-interval start position of the fork. The condition `D < first_block_height` (fork diverges before the interval start) is easy to arrange: the relayer simply begins the fork one block before the interval start. The Bitcoin mainnet retarget interval is 2016 blocks, so the window is large and the scenario is straightforward to construct.

---

### Recommendation

Replace the height-based lookup with an ancestor traversal that follows `prev_block_hash` links through `headers_pool`. Starting from `prev_block_header`, walk back `difficulty_adjustment_interval - 1` steps using `get_prev_header`. This mirrors Bitcoin Core's `GetAncestor` call and guarantees the timestamp comes from the block's own ancestor chain regardless of whether it is on the mainchain or a fork. [8](#0-7)

### Proof of Concept

**Setup (Bitcoin mainnet, `difficulty_adjustment_interval = 2016`):**

1. Initialize the contract at height `H = 2016` (a retarget boundary). Mainchain grows to height `4031`.
2. The mainchain block at height `2016` has timestamp `T_main`.

**Fork construction:**

3. Submit fork block at height `2015` (diverge point `D = 2015 < first_block_height = 2016`). This block is stored only in `headers_pool`.
4. Submit fork blocks at heights `2016` through `4031`. The fork's block at height `2016` carries timestamp `T_fork < T_main` (earlier than mainchain's). All fork blocks are stored only in `headers_pool`.

**Retarget block submission (height 4032):**

5. Submit fork block at height `4032`. `check_pow` fires:
   - `first_block_height = 4031 - 2015 = 2016`
   - `get_header_by_height(2016)` → returns **mainchain** block with timestamp `T_main`
   - `actual_time_taken = T_fork_tip - T_main` (uses mainchain's `T_main`, not fork's `T_fork`)
   - Because `T_main < T_fork` (mainchain interval appears longer), `actual_time_taken` is larger → lower difficulty → easier `bits`
   - Fork block is submitted with `bits` matching this easier target; `check_pow` passes
6. Fork's `chain_work` exceeds mainchain tip → `reorg_chain` executes → `mainchain_tip_blockhash` is set to the fork tip.

**Assert:** The accepted `bits` on the fork's retarget block differs from what `calculate_next_work_required` would produce if given the fork's own block at height `2016` (timestamp `T_fork`), proving the invariant is broken. [9](#0-8)

fn get_next_work_required(
    config: &NetworkConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    if (prev_block_header.block_height + 1) % config.difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            if block_header.time
                > prev_block_header.block_header.time + 2 * config.pow_target_spacing
            {
                return config.proof_of_work_limit_bits;
            }

            let mut current_block_header = prev_block_header.clone();
            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            let last_bits = current_block_header.block_header.bits;
            return last_bits;
        }
        return prev_block_header.block_header.bits;
    }

    let first_block_height =
        prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);

    let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
    calculate_next_work_required(
        config,
        prev_block_header,
        interval_tail_extend_header.block_header.time.into(),
    )
}

fn calculate_next_work_required(
    config: &NetworkConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let prev_block_time: i64 = prev_block_header.block_header.time.into();

    let mut actual_time_taken: i64 = prev_block_time - first_block_time;
    if actual_time_taken < config.pow_target_timespan / 4 {
        actual_time_taken = config.pow_target_timespan / 4;
    }
    if actual_time_taken > config.pow_target_timespan * 4 {
        actual_time_taken = config.pow_target_timespan * 4;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(actual_time_taken).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target = new_target
        / U256::from(<i64 as TryInto<u64>>::try_into(config.pow_target_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()
