Audit Report

## Title
`addLiquidity` Missing `whenNotPaused` Guard Allows Principal Deposit Into Compromised Pool — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary
`MetricOmmPool.addLiquidity` and `MetricOmmPool.removeLiquidity` are missing the `whenNotPaused` modifier that is correctly applied to `swap`. When the pool is paused in response to a critical event, any unprivileged caller can still deposit new token0/token1 principal into the compromised pool, and extension hooks (`_beforeAddLiquidity`, `_afterAddLiquidity`, `_beforeRemoveLiquidity`, `_afterRemoveLiquidity`) continue to execute against potentially vulnerable extension logic, defeating the purpose of the pause mechanism and exposing newly deposited principal to the same invariant break that triggered the pause.

## Finding Description
`MetricOmmPool` defines `pauseLevel` (0 = active, 1 = admin-paused, 2 = protocol-paused) and a `whenNotPaused` modifier that calls `_checkNotPaused()`, which reverts with `PoolPaused` when `pauseLevel != 0`:

```solidity
// L174-177
modifier whenNotPaused() {
    _checkNotPaused();
    _;
}
// L643-645
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
```

`swap` correctly applies this guard (L224):
```solidity
) external whenNotPaused nonReentrant(PoolActions.SWAP) returns (int128, int128) {
```

But `addLiquidity` (L188) and `removeLiquidity` (L201) do not:
```solidity
// addLiquidity — L188
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {

// removeLiquidity — L201
  external
  nonReentrant(PoolActions.REMOVE_LIQUIDITY)
  returns (uint256 amount0Removed, uint256 amount1Removed)
```

Both functions invoke extension hooks before and after their core logic (L191, L195, L207, L211), which dispatch into configured extension contracts via `_callExtensionsInOrder`. These hooks execute unconditionally even when the pool is paused.

Exploit path:
1. An exploit or active bug is discovered in the pool or an extension.
2. Admin calls `factory.setPause(poolAddress, 1)` — `pauseLevel` becomes 1.
3. `swap()` now reverts with `PoolPaused` — the swap drain path is blocked.
4. Any address calls `pool.addLiquidity(...)` — succeeds because `whenNotPaused` is absent.
   - `_beforeAddLiquidity` fires extension hooks on the still-live (potentially compromised) extension.
   - `LiquidityLib.addLiquidity` credits shares and updates `binTotals.scaledToken0`/`scaledToken1`.
   - `_afterAddLiquidity` fires extension hooks again.
5. Newly deposited token0/token1 are now tracked in `binTotals` and subject to the broken invariant.
6. When the pool is unpaused, the exploit can be triggered against the enlarged pool, draining the newly deposited principal.

For `removeLiquidity`: an attacker holding a position inflated via the exploit reason for the pause can still exit with inflated shares before the admin can react, and extension hooks (e.g., a stop-loss or allowlist extension) still execute against potentially compromised state.

## Impact Explanation
Direct loss of user principal. Unsuspecting LPs or MEV bots can deposit token0/token1 into a pool that is paused precisely because it is compromised. The deposited amounts are immediately credited to `binTotals.scaledToken0`/`scaledToken1` and become subject to the same broken invariant that triggered the pause. Additionally, extension hooks continue to execute during the paused state, widening the attack surface. This constitutes a broken core pool safety mechanism causing direct loss of deposited principal — a High-severity impact per Sherlock thresholds.

## Likelihood Explanation
- **Trigger condition**: Pool must be paused — an infrequent but explicitly designed scenario.
- **Attacker capability**: No privilege required. Any address can call `addLiquidity`; any position owner can call `removeLiquidity`. Both are standard user operations.
- **Realistic scenario**: A MEV bot monitors the mempool for the `setPause` transaction, front-runs it with `addLiquidity` to seed a position, then back-runs the unpausing to exploit the still-broken invariant against the enlarged pool. Ordinary users unaware of the pause also continue to add liquidity.

Likelihood: **Medium** (requires an active pause event, no privilege escalation). Impact: **High** (direct loss of deposited principal). Overall: **High/Medium**.

## Recommendation
Apply `whenNotPaused` to `addLiquidity`, mirroring the pattern already used on `swap`:

```solidity
function addLiquidity(
  address owner, uint80 salt, LiquidityDelta calldata deltas,
  bytes calldata callbackData, bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
```

For `removeLiquidity`, if the protocol intentionally allows LP exits during a pause (to let users withdraw funds), this must be explicitly documented and extension hooks must be skipped during a paused state to prevent hook-level exploitation. Otherwise, apply `whenNotPaused` symmetrically:

```solidity
function removeLiquidity(address owner, uint80 salt, LiquidityDelta calldata deltas, bytes calldata extensionData)
  external
  whenNotPaused
  nonReentrant(PoolActions.REMOVE_LIQUIDITY)
  returns (...) {
```

## Proof of Concept
```
1. Deploy pool with any extension (e.g., DepositAllowlistExtension).
2. Discover exploit; admin calls factory.setPause(poolAddress, 1).
3. Call pool.swap(...) → reverts with PoolPaused. ✓
4. Call pool.addLiquidity(owner, salt, deltas, callbackData, extensionData):
   - Succeeds (no whenNotPaused guard).
   - _beforeAddLiquidity fires extension hooks.
   - LiquidityLib.addLiquidity credits shares, updates binTotals.scaledToken0/scaledToken1.
   - _afterAddLiquidity fires extension hooks.
5. Assert binTotals.scaledToken0 increased by deposited amount — confirms principal is now at risk.
6. Unpause pool; trigger original exploit against enlarged binTotals — drain includes newly deposited principal.
```

Root cause: asymmetric application of `whenNotPaused` — present on `swap` (L224) but absent on `addLiquidity` (L188) and `removeLiquidity` (L201).
