### Title
Non-bondDenom Tokens Sent to `rewards_pool` Module Account Are Permanently Stuck — (`app/app.go`, `x/tieredrewards/keeper/abci.go`, `x/tieredrewards/keeper/claim_rewards.go`)

### Summary
The `rewards_pool` module account is explicitly whitelisted to receive external `MsgSend` transfers of any denomination. However, every outflow path from the module account is hardcoded to only move the chain's `bondDenom` (basecro). Any other token denomination sent to the `rewards_pool` address is permanently locked with no withdrawal or governance-controlled drain mechanism.

### Finding Description
`app/app.go` defines two maps that together create the vulnerability:

1. `maccPerms` registers `tieredrewardstypes.RewardsPoolName` with `nil` permissions (no minter, burner, or staker), meaning the module account cannot self-manage its balance. [1](#0-0) 

2. `moduleAccsAllowedToReceiveExternalFunds` explicitly whitelists `RewardsPoolName`, bypassing the default Cosmos SDK block on `MsgSend` to module accounts. Any unprivileged user can send any token denomination to the `rewards_pool` address via a standard bank send transaction. [2](#0-1) 

The only two code paths that move coins **out** of the `rewards_pool` are:

- `topUpBaseRewards` in `x/tieredrewards/keeper/abci.go`, which calls `SendCoinsFromModuleToModule` exclusively for `bondDenom`: [3](#0-2) 

- `processEventsAndClaimBonus` in `x/tieredrewards/keeper/claim_rewards.go`, which constructs `bonusCoins` only from `bondDenom` and calls `SendCoinsFromModuleToAccount`: [4](#0-3) 

No governance message, admin message, or upgrade handler exists to drain arbitrary denominations from the `rewards_pool`. The module's message set (`MsgLockTier`, `MsgClaimTierRewards`, `MsgWithdrawFromTier`, `MsgUpdateTier`, etc.) contains no operation that touches non-bondDenom balances of the pool account. [5](#0-4) 

### Impact Explanation
**High.** Any token denomination (IBC vouchers, CRC-20 wrapped assets, etc.) sent to the `rewards_pool` module address is permanently irrecoverable. The module account has `nil` permissions so it cannot burn or transfer on its own authority, and no keeper function iterates or drains non-bondDenom balances. The funds are stuck for the lifetime of the chain unless a chain upgrade explicitly adds a drain path.

### Likelihood Explanation
**Low.** A user must make an error and send a non-bondDenom token to the `rewards_pool` address. The address is a module account address (not a human-readable name), so accidental sends are plausible but not common. The risk is elevated because the address is publicly queryable via `QueryRewardsPoolBalances` and the module is documented as accepting external funding. [6](#0-5) 

### Recommendation
Restrict the `moduleAccsAllowedToReceiveExternalFunds` whitelist for `RewardsPoolName` to only accept the `bondDenom`, or add a governance-controlled message (e.g., `MsgDrainRewardsPool`) that sweeps all non-bondDenom balances from the `rewards_pool` to the community pool or a specified recipient. Alternatively, add an `AnteHandler` check that rejects `MsgSend` to the `rewards_pool` address for any denomination other than `bondDenom`.

### Proof of Concept
1. Query the `rewards_pool` module address via `chain-maind query tieredrewards rewards-pool-balances`.
2. Submit a `MsgSend` from any account sending, e.g., `1000ibc/XXXX` (an IBC voucher) to the `rewards_pool` address. This succeeds because the address is in `moduleAccsAllowedToReceiveExternalFunds`. [7](#0-6) 
3. Observe that `QueryRewardsPoolBalances` now shows the IBC voucher balance.
4. Wait indefinitely: `topUpBaseRewards` never touches the IBC voucher balance (it only reads and sends `bondDenom`), and `processEventsAndClaimBonus` only ever constructs `sdk.NewCoin(bondDenom, totalBonus)`. [8](#0-7) 
5. The IBC voucher balance remains in the module account permanently with no on-chain path to recover it.
