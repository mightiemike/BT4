### Title
AuxPoW Coinbase Script Parsed via Non-Byte-Aligned Hex Substring Search, Allowing Forged Merged-Mining Proofs — (`contract/src/dogecoin.rs`)

---

### Summary

In `contract/src/dogecoin.rs`, the `check_aux` function verifies Dogecoin AuxPoW by converting the coinbase `script_sig` to a hex string and then using Rust's `.find()` to locate the merged-mining header sentinel (`"fabe6d6d"`) and the chain merkle root. Because `.find()` is a plain substring search over characters, it can match at **odd character positions** in the hex string — positions that do not correspond to any real byte boundary. This is the direct analog of the DNSSEC M-05 bug: a flat string containment check is used where a structurally-aligned (byte-by-byte) search is required.

---

### Finding Description

In `check_aux` the script is converted to hex and then searched:

```rust
let script_sig = coinbase_tx.input.first().unwrap()
    .script_sig.to_hex_string();                          // raw bytes → 2-char-per-byte hex

let pos_merged_mining_header = script_sig.find(MERGED_MINING_HEADER);   // "fabe6d6d" (8 chars)
let mut pos_chain_root = script_sig
    .find(&chain_root.to_string())                        // 64-char hex string
    .expect("Aux POW missing chain merkle root in parent coinbase");
``` [1](#0-0) 

`to_hex_string()` encodes every byte as exactly two hex characters, so byte boundaries fall only at **even** character positions (0, 2, 4, …). Rust's `str::find` is a character-level substring search; it will return an **odd** position if the target pattern happens to appear there. For example, if the raw script bytes are `[0x0f, 0xab, 0xe6, 0xd6, 0xd0, …]`, the hex string is `"0fabe6d6d0…"`, and `find("fabe6d6d")` returns position **1** — a non-byte-aligned hit.

All subsequent logic uses the returned positions as offsets into the hex string:

```rust
require!(
    pos_merged_mining_header + MERGED_MINING_HEADER.len() == pos_chain_root,
    "Merged mining header is not just before chain merkle root"
);
``` [2](#0-1) 

```rust
pos_chain_root += chain_root.to_string().len();          // advance past the root (64 chars)
let bytes = hex::decode(&script_sig[pos_chain_root..pos_chain_root + 8]).unwrap();
let n_size = u32::from_le_bytes([bytes[0], bytes[1], bytes[2], bytes[3]]);
``` [3](#0-2) 

```rust
let bytes = hex::decode(&script_sig[pos_chain_root + 8..pos_chain_root + 16]).unwrap();
let n_nonce = u32::from_le_bytes([bytes[0], bytes[1], bytes[2], bytes[3]]);
``` [4](#0-3) 

When `pos_chain_root` is odd, `script_sig[pos_chain_root..pos_chain_root+8]` is an 8-character window that straddles two real bytes at every position. `hex::decode` of this misaligned window produces four bytes that are **not** the actual bytes at that script location — they are a nibble-shifted blend of adjacent bytes. The attacker therefore controls what `n_size` and `n_nonce` decode to, independently of the real script content.

The final checks that depend on these values are:

```rust
require!(
    n_size == (1u32 << aux_data.chain_merkle_proof.len()),
    "Aux POW merkle branch size does not match parent coinbase"
);
``` [5](#0-4) 

```rust
require!(
    u32::try_from(aux_data.chain_id).ok() == Some(expected_index),
    "Aux POW wrong index"
);
``` [6](#0-5) 

Both checks can be satisfied with attacker-chosen misaligned bytes, bypassing the chain-index binding that prevents one chain's AuxPoW from being replayed for another.

---

### Impact Explanation

A successful exploit allows an attacker to submit a Dogecoin block header accompanied by an `AuxData` whose `chain_id` does not match the expected Merkle-tree slot, yet passes `check_aux`. This breaks the chain-ID isolation guarantee of AuxPoW: the contract would accept a merged-mining proof that was not actually committed to the Dogecoin chain slot, corrupting `headers_pool` and the mainchain maps with a fraudulent block. Downstream callers of `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` would then verify SPV proofs against a forged block header. [7](#0-6) 

---

### Likelihood Explanation

The attacker must be a Bitcoin miner (or collude with one) to produce a parent block with a crafted coinbase script and valid PoW. This is a meaningful barrier. However, merged-mining pools already control the coinbase script content by design, and the nibble-shift constraint is a simple arithmetic relationship that can be solved offline before mining begins. The attack surface is reachable via the public `submit_blocks` entry point with no privileged role required. [8](#0-7) 

---

### Recommendation

Replace the hex-string substring search with a byte-aligned search over the **raw script bytes**:

1. Work directly on `coinbase_tx.input[0].script_sig.as_bytes()` (or the equivalent raw-byte slice from the bitcoin library).
2. Search for the 4-byte sentinel `[0xfa, 0xbe, 0x6d, 0x6d]` using a byte-slice search, ensuring the match position is used as a **byte** offset.
3. Read `n_size` and `n_nonce` by indexing the raw byte slice at `byte_pos + 4 + 32` and `byte_pos + 4 + 32 + 4` respectively, then deserialise with `u32::from_le_bytes`.
4. Compare the chain root by reading 32 raw bytes from the script and constructing an `H256` directly, rather than comparing hex strings.

This mirrors the DNSSEC fix recommendation: check structure element-by-element (byte-by-byte) rather than performing a flat string containment test.

---

### Proof of Concept

Suppose `chain_merkle_proof.len() == 1`, so `n_size` must equal `2` (`0x02000000` LE) and `expected_index` must equal `get_expected_index(n_nonce, chain_id, 1)`.

Craft a coinbase script whose raw bytes, when hex-encoded, satisfy:

- Position 1 (odd): `"fabe6d6d"` appears (i.e., raw bytes at positions 0–4 are `[0x_f, 0xab, 0xe6, 0xd6, 0xd_]` with appropriate nibbles).
- Position 9 (odd, = 1 + 8): the 64-char hex of `chain_root.to_string()` appears.
- Position 9 + 64 = 73 (odd): the misaligned 8 hex chars decode via `hex::decode` to `[0x02, 0x00, 0x00, 0x00]` → `n_size = 2` ✓.
- Position 81 (odd): the misaligned 8 hex chars decode to a chosen `n_nonce` such that `get_expected_index(n_nonce, chain_id, 1)` equals the `chain_id` value supplied in `aux_data`. [9](#0-8) 

All four `require!` checks in `check_aux` pass. The parent block's PoW check is independent and uses the real Bitcoin block hash, so it is unaffected by the coinbase content. The contract stores the fraudulent Dogecoin header as a valid mainchain block.

    pub fn submit_blocks(

        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
