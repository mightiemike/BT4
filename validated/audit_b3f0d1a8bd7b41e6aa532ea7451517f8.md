### Title
`AllResources`→`L1Gas` Misclassification in P2P Protobuf Deserialization Produces Wrong Execution Semantics and Hash Divergence — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf-to-`ValidResourceBounds` converter uses a zero-value sentinel check to decide whether a deserialized transaction belongs to the pre-0.13.3 `L1Gas` variant or the post-0.13.3 `AllResources` variant. When an `AllResources` transaction carries zero values for both `l2_gas` and `l1_data_gas`, the converter silently downgrades it to `L1Gas`. This changes the gas-computation mode, the initial Sierra gas limit, the fee-enforcement decision, and the transaction-hash preimage — all without any error or warning.

### Finding Description

In `crates/apollo_protobuf/src/converters/transaction.rs` the P2P state-sync deserializer for `ValidResourceBounds` reads:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

`is_zero()` requires **both** `max_amount == 0` and `max_price_per_unit == 0`:

```rust
pub fn is_zero(&self) -> bool {
    self.max_amount == GasAmount(0) && self.max_price_per_unit == GasPrice(0)
}
``` [2](#0-1) 

A valid `AllResources` transaction whose `l2_gas` and `l1_data_gas` fields are both `{max_amount:0, max_price_per_unit:0}` — but whose `l1_gas` is non-zero — satisfies the condition and is silently reclassified as `L1Gas`. The test file itself acknowledges the hazard:

```
// If all the fields of `AllResources` are 0 upon serialization,
// then the deserialized value will be interpreted as the `L1Gas` variant.
``` [3](#0-2) 

The workaround applied in the test is to inject a non-zero `l2_gas.max_amount = GasAmount(1)` before serialization, confirming the team is aware of the ambiguity but has not fixed the root cause in the converter. [4](#0-3) 

**Divergence 1 — Transaction hash preimage.**  
`get_tip_resource_bounds_hash` includes the `L1DataGas` element only for `AllResources`:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
``` [5](#0-4) 

`AllResources` with zero `l1_data_gas` hashes `[tip, L1Gas, L2Gas(0), L1DataGas(0)]` (4 elements); the misclassified `L1Gas` hashes `[tip, L1Gas, L2Gas(0)]` (3 elements). The Poseidon hash over a different-length input produces a different digest, so the stored hash and the recomputed hash diverge.

**Divergence 2 — Initial Sierra gas.**  
`initial_sierra_gas()` returns `l2_gas.max_amount` for `AllResources` and `initial_gas_no_user_l2_bound()` (a large constant) for `L1Gas`:

```rust
TransactionInfo::Current(CurrentTransactionInfo {
    resource_bounds: ValidResourceBounds::L1Gas(_), ..
}) => self.block_context.versioned_constants.initial_gas_no_user_l2_bound(),
TransactionInfo::Current(CurrentTransactionInfo {
    resource_bounds: ValidResourceBounds::AllResources(AllResourceBounds { l2_gas, .. }), ..
}) => l2_gas.max_amount,
``` [6](#0-5) 

An `AllResources` transaction with `l2_gas.max_amount = 0` receives **zero** initial gas and immediately reverts. After misclassification to `L1Gas` it receives the large default gas limit and may succeed — producing a different receipt, state diff, and event log.

**Divergence 3 — Gas-vector computation mode and fee enforcement.**  
`get_gas_vector_computation_mode()` returns `All` for `AllResources` and `NoL2Gas` for `L1Gas`: [7](#0-6) 

`enforce_fee()` calls `max_possible_fee(tip)`, which for `L1Gas` ignores `l2_gas` and `l1_data_gas` entirely: [8](#0-7) 

### Impact Explanation

A syncing node that receives an `AllResources` transaction with zero `l2_gas`/`l1_data_gas` via P2P state sync will:

1. Store it as `L1Gas` — a different type than the proposing node committed to.
2. Execute it with a large initial gas limit instead of zero, potentially producing a successful receipt where the canonical execution produces a revert.
3. Compute a different transaction hash if hash verification is performed, breaking the commitment chain.

This matches **Critical — Wrong state, receipt, event, or revert result from blockifier/syscall/execution logic for accepted input** and **Critical — Wrong compiled class, class hash, or contract code selected for execution** (gas mode selects the execution path).

### Likelihood Explanation

The gateway stateless validator rejects transactions where `l2_gas.max_price_per_unit < min_gas_price`. When `min_gas_price > 0` this prevents `l2_gas.is_zero()` for gateway-submitted transactions. However:

- The P2P state-sync path has **no equivalent gate**; a peer can deliver any well-formed protobuf.
- Historical transactions from networks where `min_gas_price = 0` can satisfy the condition.
- The test workaround (`GasAmount(1)`) confirms the condition is reachable in practice.

Likelihood is **Low** for mainnet gateway traffic but **Medium** for adversarial P2P peers or cross-network replay.

### Recommendation

Replace the zero-value heuristic with an explicit wire-level discriminator. The simplest fix is to add a boolean or enum tag to the protobuf `ResourceBounds` message that records whether the sender intended `L1Gas` or `AllResources`, and validate it on receipt. Alternatively, mandate that `AllResources` transactions always carry a non-zero `l2_gas.max_amount` (enforced at the gateway and at the P2P ingress), so the `is_zero()` check becomes a reliable sentinel. The test comment at line 26–27 of `consensus_test.rs` should be promoted to a hard invariant enforced in the converter, not silently papered over by injecting `GasAmount(1)`.

### Proof of Concept

1. Construct an `AllResources` transaction: `l1_gas = {max_amount:1000, max_price_per_unit:1}`, `l2_gas = {max_amount:0, max_price_per_unit:0}`, `l1_data_gas = {max_amount:0, max_price_per_unit:0}`.
2. Compute its hash via `get_tip_resource_bounds_hash` as `AllResources` → hash H₁ (4-element Poseidon input).
3. Serialize to `protobuf::ResourceBounds` via `From<ValidResourceBounds>` — all three fields are emitted with zero values for l2/l1_data.
4. Deserialize via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` — `l1_data_gas.is_zero() && l2_gas.is_zero()` is `true` → result is `ValidResourceBounds::L1Gas(l1_gas)`.
5. Compute hash again as `L1Gas` → hash H₂ (3-element Poseidon input). Assert `H₁ ≠ H₂`.
6. Execute the transaction: as `AllResources` with `l2_gas.max_amount=0`, `initial_sierra_gas()` returns 0 → immediate out-of-gas revert. As misclassified `L1Gas`, `initial_sierra_gas()` returns `initial_gas_no_user_l2_bound()` → execution proceeds normally.
