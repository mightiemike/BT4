### Title
Accrued Bonus Rewards Permanently Lost When Bonus Pool Is Empty During Redelegation Slash - (`x/tieredrewards/keeper/slash.go`)

### Summary

In `slashRedelegationPosition`, when `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, the error is silently swallowed and `setPositionWithState` is still called with the in-memory `pos` struct — which `processEventsAndClaimBonus` has already mutated (advancing `LastBonusAccrual`, `LastEventSeq`, `LastKnownBonded`). The advanced checkpoints are persisted, permanently erasing the accrual window. The position owner's earned bonus is irrecoverably lost with no tracking, no deferred-claim record, and no retry path.

### Finding Description

`slashRedelegationPosition` in `x/tieredrewards/keeper/slash.go` is invoked by the `BeforeRedelegationSlashed` staking hook. Its job is to settle accrued bonus rewards against pre-slash shares before the staking module reduces the delegation. [1](#0-0) 

`processEventsAndClaimBonus` takes `&pos` (a pointer) and mutates the position's bonus checkpoints (`LastBonusAccrual`, `LastEventSeq`, `LastKnownBonded`) as it walks validator events and computes the owed bonus. Per the ADR, checkpoints are advanced **before** the pool-balance check:

> Advance LastBonusAccrual, LastEventSeq, LastKnownBonded → If pool balance < bonus: fail atomically [2](#0-1) 

For user-driven message paths the entire Cosmos SDK transaction rolls back on error, so the in-memory mutations are discarded. But `BeforeRedelegationSlashed` is a **hook** called inside the staking module's `SlashRedelegation`; there is no outer transaction rollback. When `ErrInsufficientBonusPool` is caught and swallowed: [3](#0-2) 

execution continues and `setPositionWithState` is called with the already-mutated `pos`: [4](#0-3) 

The advanced checkpoints are written to state. The accrual window that earned the bonus is permanently skipped. For a full slash, `ClearBonusCheckpoints()` additionally zeros the checkpoints: [5](#0-4) 

making recovery doubly impossible. There is no deferred-claim record, no "owed bonus" accumulator, and no governance or admin function to retroactively pay the forfeited amount — an exact structural parallel to the external report's `lossFee` being added to `commonData` with no reimbursement path.

The ADR documents this explicitly as a deliberate trade-off:

> Bonus forfeits silently if the pool is insufficient (chain-halt avoidance). [6](#0-5) 

But the absence of documentation in the protocol's known-issues list and the absence of any recovery mechanism make this a valid medium-level finding by the same reasoning applied in the external report.

### Impact Explanation

A position owner who has an active redelegation slashed at a moment when the bonus pool is empty permanently loses all bonus rewards accrued since their last checkpoint. The lost amount equals `shares × tokensPerShare × bonusApy × durationSeconds / SecondsPerYear` for the elapsed period. The corrupted value is `pos.LastBonusAccrual` (and `LastEventSeq`, `LastKnownBonded`) persisted with advanced values while no coins were transferred, breaking the invariant that checkpoints only advance when the corresponding bonus has been paid. [7](#0-6) 

### Likelihood Explanation

The bonus pool is an externally funded module account. It can be empty or underfunded at any time — especially early in the protocol's life, after a large wave of claims, or if governance has not replenished it. Redelegation slashes are a normal staking event triggered whenever a validator is jailed for downtime or double-signing while a redelegation is in flight. The combination is realistic and requires no privileged access: any delegator who calls `MsgTierRedelegate` and whose source validator is subsequently slashed while the pool is empty hits this path. [8](#0-7) 

### Recommendation

Two complementary fixes:

1. **Do not persist checkpoint advancements when the bonus cannot be paid.** Either (a) snapshot `pos` before calling `processEventsAndClaimBonus` and restore it on `ErrInsufficientBonusPool`, or (b) restructure `processEventsAndClaimBonus` so checkpoints are only advanced after a successful `SendCoinsFromModule`.

2. **Track the forfeited amount.** If chain-halt avoidance requires swallowing the error, record the owed bonus in a per-position "deferred bonus" field and pay it on the next successful claim, analogous to how the external report recommends tracking the `lossFee` for later reimbursement.

### Proof of Concept

1. User calls `MsgLockTier` → position created, delegated to validator A.
2. User calls `MsgTierRedelegate` to validator B → redelegation entry created, `RedelegationMappings[unbondingId] = positionId` stored.
3. Time passes; bonus accrues (e.g., 30 days × `BonusApy`).
4. Bonus pool is empty (governance has not funded it, or it was drained by prior claims).
5. Validator B is jailed for downtime → staking calls `SlashRedelegation` → `BeforeRedelegationSlashed(unbondingId, sharesToUnbond)` fires.
6. `slashRedelegationPosition` calls `processEventsAndClaimBonus(&pos)`.
7. `processEventsAndClaimBonus` advances `pos.LastBonusAccrual` to `blockTime`, computes bonus > 0, checks pool → pool empty → returns `ErrInsufficientBonusPool`.
8. Error is swallowed; `setPositionWithState(ctx, pos, ...)` persists the advanced `LastBonusAccrual`.
9. Pool is later replenished. User calls `MsgClaimTierRewards` — bonus for the 30-day window is zero because `LastBonusAccrual` already points past it. [9](#0-8) [10](#0-9)
