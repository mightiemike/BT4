### Title
Linked Signer Can Overwrite Its Own Authority via `processTransactionImpl` — (`EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`, meaning an existing linked signer can sign and submit a `LinkSigner` transaction to replace the linked signer mapping with an attacker-controlled address. The slow-mode path for the same operation enforces a stricter owner-only check (`validateSender`), but the fast-mode path bypasses it. This is a direct analog to the reported bug class: a safety check enforced in one execution path is absent in a functionally equivalent path.

---

### Finding Description

**Two paths update `linkedSigners[sender]` — only one enforces the owner-only constraint.**

**Fast-mode path** (`processTransactionImpl`, `EndpointTx.sol` lines 576–590):

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // ← allowLinkedSigner = true
);
linkedSigners[signedTx.tx.sender] = address(uint160(bytes20(signedTx.tx.signer)));
```

`validateSignedTx` with `allowLinkedSigner = true` accepts a signature from either the subaccount's own address **or** the currently registered linked signer. This means the linked signer itself can sign a `LinkSigner` transaction and have the sequencer process it, overwriting `linkedSigners[sender]` with any address.

**Slow-mode path** (`processSlowModeTransactionImpl`, `EndpointTx.sol` lines 232–239):

```solidity
validateSender(txn.sender, sender);   // ← requires msg.sender == subaccount address
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

`validateSender` checks that `address(uint160(bytes20(txn.sender))) == msg.sender`, meaning only the actual Ethereum address that owns the subaccount can submit this transaction. A linked signer cannot satisfy this check.

Both paths write to the same `linkedSigners` mapping and produce the same state mutation. The safety invariant — *only the account owner may update the linked signer* — is enforced in the slow-mode path but absent in the fast-mode path.

---

### Impact Explanation

A compromised or malicious linked signer can:

1. Submit a `LinkSigner` transaction (signed by the current linked signer) to the sequencer.
2. The sequencer calls `processTransactionImpl`, which accepts the linked signer's signature and overwrites `linkedSigners[victim]` with an attacker-controlled address.
3. The new linked signer can now sign any transaction that `allowLinkedSigner = true` permits: `WithdrawCollateral`, `BurnNlp`, `MintNlp`, `TransferQuote`, `LiquidateSubaccount`, etc.
4. The victim's only recourse is the slow-mode `LinkSigner` path, which carries a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`). During this window the attacker retains full signing authority.

Concrete asset impact: the attacker-controlled linked signer can place adversarial trades (e.g., crossing the spread at unfavorable prices), mint NLP to lock liquidity, or initiate withdrawals to the subaccount's own address and then drain via further trades — all before the victim's slow-mode revocation executes.

**Impact: Medium** — direct fund loss requires chaining with trading or NLP operations; the attacker cannot send tokens to an arbitrary external address in a single step.

---

### Likelihood Explanation

Linked signers are the standard mechanism for automated trading bots and API keys in Nado. Private key compromise of a hot-wallet bot key is a realistic operational threat. No privileged access, governance capture, or sequencer compromise is required — the attacker only needs the current linked signer's private key and the ability to submit a transaction to the sequencer.

**Likelihood: Medium**

---

### Recommendation

In `processTransactionImpl`, change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may update the linked signer
);
```

This aligns the fast-mode path with the owner-only invariant already enforced by the slow-mode path.

---

### Proof of Concept

1. Alice sets `linkedSigners[aliceSubaccount] = botAddress` via the slow-mode path.
2. Attacker compromises `botAddress` private key.
3. Attacker constructs a `SignedLinkSigner` transaction: `sender = aliceSubaccount`, `signer = attackerAddress`, signed by `botAddress`.
4. Attacker submits to the sequencer. The sequencer calls `processTransactionImpl`.
5. `validateSignedTx(..., true)` recovers `botAddress` from the signature, matches it against `getLinkedSigner(aliceSubaccount) == botAddress` → passes.
6. `linkedSigners[aliceSubaccount]` is now `attackerAddress`.
7. Alice submits a slow-mode `LinkSigner` to revoke, but it executes in 3 days.
8. During those 3 days, `attackerAddress` signs `WithdrawCollateral`, `MintNlp`, and trade transactions against Alice's subaccount.
