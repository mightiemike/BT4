### Title
Freed Storage Deposits Permanently Locked in Contract — No Withdrawal Mechanism (`contract/src/lib.rs`)

### Summary

The `submit_blocks` function is `#[payable]` and collects NEAR token deposits from relayers to cover on-chain storage costs. When `run_mainchain_gc` removes old block headers (freeing storage), the NEAR tokens previously deposited for that storage are released into the contract's own balance. There is no function in the contract to withdraw or recover these accumulated freed deposits. They are permanently locked.

### Finding Description

`submit_blocks` measures net storage growth after both adding new headers and running GC: [1](#0-0) 

The critical sequence is:

1. `initial_storage = env::storage_usage()` is captured before any work.
2. New headers are inserted (storage grows).
3. `run_mainchain_gc(num_of_headers)` removes old headers (storage shrinks).
4. `diff_storage_usage = env::storage_usage().saturating_sub(initial_storage)` — because `saturating_sub` floors at zero, if GC frees more than the new headers consume, the net is reported as 0.
5. `required_deposit = 0`, so the entire `amount` attached by the *current* caller is refunded.

The storage deposit paid by *previous* callers for the now-GC'd blocks is not tracked per-depositor and is never returned. It accumulates silently in the contract's balance.

Additionally, `run_mainchain_gc` is a standalone public function: [2](#0-1) 

Any caller can invoke it directly. It removes entries from `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height`, freeing storage, but there is no refund path of any kind in this code path.

There is no `withdraw`, `recover_storage`, or equivalent function anywhere in the contract. The only transfer out of the contract is the per-call excess refund inside `submit_blocks`: [3](#0-2) 

This only covers the *current* caller's overpayment, not previously deposited funds freed by GC.

### Impact Explanation

NEAR tokens paid as storage deposits by relayers for block headers that are subsequently garbage-collected are permanently locked in the contract. As the chain grows and GC runs continuously (it is called on every `submit_blocks` invocation once the chain exceeds `gc_threshold`), the locked balance grows monotonically. There is no privileged or unprivileged path to recover these funds.

### Likelihood Explanation

GC is triggered automatically on every `submit_blocks` call once `amount_of_headers_we_store > gc_threshold`: [4](#0-3) 

For a live deployment tracking Bitcoin (with `gc_threshold = 52704` per the recommended init comment), GC will run continuously after the first year of operation. Every subsequent `submit_blocks` call will free storage deposits with no recovery path. The condition is deterministic and inevitable in production.

### Recommendation

Implement a privileged withdrawal function (callable by `Role::DAO` or a designated treasury role) that transfers the contract's excess balance — the difference between `env::account_balance()` and the minimum required for storage staking — to a designated recipient. This mirrors the standard NEAR storage-management pattern.

### Proof of Concept

1. Relayer submits 1000 headers via `submit_blocks` with sufficient NEAR deposit. Storage grows; deposit is consumed.
2. Chain tip advances. `gc_threshold` is exceeded.
3. On the next `submit_blocks` call, `run_mainchain_gc` removes 1000 old headers. Storage shrinks back.
4. `diff_storage_usage = saturating_sub(...)` = 0. The current caller's deposit is fully refunded.
5. The original 1000-header deposit from step 1 is now freed into the contract's balance.
6. Repeat indefinitely. Contract balance grows with each GC cycle.
7. No function exists to withdraw the accumulated balance. Funds are permanently locked.

    #[pause(except(roles(Role::UnrestrictedRunGC)))]
    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
