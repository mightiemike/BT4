### Title
Silent Bonus Reward Forfeiture with Permanent Checkpoint Advancement During Redelegation Slash — (`x/tieredrewards/keeper/slash.go`)

---

### Summary

In `slashRedelegationPosition`, when the bonus rewards pool is insufficient to cover accrued bonus at the time of a redelegation slash, the `ErrInsufficientBonusPool` error is deliberately swallowed to prevent a chain halt. However, `processEventsAndClaimBonus` has already mutated the in-memory `pos` object (advancing `LastEventSeq`, `LastBonusAccrual`, and `LastKnownBonded`) and decremented event reference counts in the store before returning the error. The caller then persists this advanced position state. The accrued-but-unpaid bonus is permanently destroyed with no tracking and no user recourse.

---

### Finding Description

The `BeforeRedelegationSlashed` staking hook routes to `slashRedelegationPosition` in `x/tieredrewards/keeper/slash.go`. [1](#0-0) 

Inside `slashRedelegationPosition`, `processEventsAndClaimBonus` is called with a pointer to `pos`: [2](#0-1) 

Inside `processEventsAndClaimBonus`, the function walks all pending validator events and, **for each event**, calls `pos.UpdateLastEventSeq(entry.Seq)` and `k.decrementEventRefCount` — both of which mutate state — **before** the pool balance check: [3](#0-2) 

After the event loop, `applyBonusAccrualCheckpoint` advances `pos.LastBonusAccrual` to the current block time and `pos.UpdateLastKnownBonded` updates the bonded state — again **before** the pool check: [4](#0-3) 

Only then does the pool balance check fire: [5](#0-4) 

When `ErrInsufficientBonusPool` is returned, the caller swallows it and continues with the already-mutated `pos`. For a **partial slash**, the position is persisted with the advanced checkpoints: [6](#0-5) 

For a **full slash**, `ClearBonusCheckpoints()` is called and the position is persisted: [7](#0-6) 

`ClearBonusCheckpoints()` zeros `LastBonusAccrual`, `LastEventSeq`, and `LastKnownBonded`: [8](#0-7) 

In both cases, the accrued bonus period is permanently erased from the position's accounting state. There is no mapping, queue, or deferred-claim mechanism to record the owed amount.

This contrasts sharply with all user-driven paths, which fail atomically so the transaction rolls back and the user can retry: [9](#0-8) 

The ADR explicitly acknowledges the hook path forfeits silently: [10](#0-9) 

---

### Impact Explanation

A position owner whose redelegation is slashed while the bonus pool is empty permanently loses all bonus rewards accrued since their last claim. The loss magnitude is:

```
bonus = shares × tokensPerShare × tier.BonusApy × durationSinceLastClaim / SecondsPerYear
``` [11](#0-10) 

For a position that has been locked for months without claiming, this can be a substantial amount. The user has no on-chain recourse: the checkpoints are advanced, the event reference counts are decremented, and no record of the owed amount exists anywhere in the store.

---

### Likelihood Explanation

The trigger requires two concurrent conditions:

1. A user has an active tier position that was redelegated (via `MsgTierRedelegate`), creating a `RedelegationMappings` entry.
2. The source validator is slashed (downtime or double-sign) while the redelegation is still in the 21-day unbonding window.
3. The bonus rewards pool (`tieredrewards` `RewardsPoolName` module account) is empty or has insufficient balance at that block.

Condition 3 is realistic: the pool is funded by governance and can be drained by normal user claims. A large number of simultaneous `MsgClaimTierRewards` calls in the same block as a slash event can deplete the pool. The pool balance is not reserved per-position, so any user can drain it before the hook fires.

---

### Recommendation

The fix must ensure that when `ErrInsufficientBonusPool` is returned from `processEventsAndClaimBonus` inside `slashRedelegationPosition`, the position's bonus checkpoints are **not** advanced and the owed amount is **tracked** for later claim.

Two concrete approaches:

1. **Snapshot-and-restore**: Before calling `processEventsAndClaimBonus`, snapshot `pos.LastEventSeq`, `pos.LastBonusAccrual`, and `pos.LastKnownBonded`. On `ErrInsufficientBonusPool`, restore the snapshot so the position is persisted with its original checkpoints. The user can then call `MsgClaimTierRewards` once the pool is replenished.

2. **Deferred-claim mapping**: Record `(positionId → owedBonus)` in a new store collection when the pool is insufficient. Add a `MsgClaimDeferredBonus` message that pays out from this mapping when the pool is topped up.

Option 1 is simpler and consistent with the existing "fail atomically, retry later" design for user-driven paths. The key change is that `processEventsAndClaimBonus` must not advance checkpoints before the balance check, or the caller must not persist the mutated `pos` on pool-insufficient failure.

Additionally, `decrementEventRefCount` is called inside the event loop before the balance check. If the context is not a cache context, these store mutations persist even when the error is swallowed. The event loop should either be deferred until after the balance check, or the entire function should operate on a branched cache context that is only committed on success.

---

### Proof of Concept

1. Alice creates a tier position with a large delegation and a high-APY tier. She has not claimed bonus rewards for 6 months.
2. Alice calls `MsgTierRedelegate` to move her position to validator B. A `RedelegationMappings` entry is created.
3. Multiple other users call `MsgClaimTierRewards` in the same block, draining the bonus pool to zero.
4. Validator A (the source) is slashed for downtime. The staking module calls `BeforeRedelegationSlashed`.
5. `slashRedelegationPosition` calls `processEventsAndClaimBonus`. The function computes Alice's 6-month accrued bonus, advances `pos.LastBonusAccrual` to the current block time, advances `pos.LastEventSeq`, decrements event reference counts in the store, then hits `sufficientBonusPoolBalance` — which returns `ErrInsufficientBonusPool`.
6. `slashRedelegationPosition` logs the error and continues. It persists `pos` with the advanced `LastBonusAccrual` and `LastEventSeq`.
7. Alice's 6-month bonus is permanently lost. If she calls `MsgClaimTierRewards` after the pool is replenished, `processEventsAndClaimBonus` starts from the already-advanced `LastBonusAccrual` and computes zero bonus for the forfeited period. [12](#0-11) [13](#0-12)

func (k Keeper) computeSegmentBonus(pos types.PositionState, tier types.Tier, segmentStart, segmentEnd time.Time, tokensPerShare math.LegacyDec) math.Int {
	if !pos.ExitUnlockAt.IsZero() && segmentEnd.After(pos.ExitUnlockAt) {
		segmentEnd = pos.ExitUnlockAt
	}

	if !segmentEnd.After(segmentStart) {
		return math.ZeroInt()
	}

	durationSeconds := int64(segmentEnd.Sub(segmentStart) / time.Second)
	if durationSeconds <= 0 {
		return math.ZeroInt()
	}

	tokens := pos.Delegation.Shares.Mul(tokensPerShare)

	return tokens.
		Mul(tier.BonusApy).
		MulInt64(durationSeconds).
		QuoInt64(types.SecondsPerYear).
		TruncateInt()
}

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
