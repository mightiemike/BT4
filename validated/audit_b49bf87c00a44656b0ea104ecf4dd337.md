### Title
`run_serialized_chia_program` Uses `node_from_bytes` Instead of `node_from_bytes_backrefs`, Causing Failures on Back-Reference-Compressed Programs — (File: `wheel/src/api.rs`)

---

### Summary

`run_serialized_chia_program`, the primary Python-exposed entry point for executing CLVM programs, deserializes both the program and its arguments using `node_from_bytes` (the basic, back-reference-unaware deserializer). A second, strictly more capable variant — `node_from_bytes_backrefs` — exists and handles both plain and back-reference-compressed programs. When a caller passes a back-reference-compressed program blob (the standard on-chain format), `run_serialized_chia_program` will fail to deserialize it correctly, while the same blob would succeed through any other code path that uses `node_from_bytes_backrefs`.

---

### Finding Description

The repository exposes two deserializer variants:

- `node_from_bytes` (`src/serde/de.rs`) — plain legacy format only.
- `node_from_bytes_backrefs` (`src/serde/de_br.rs`) — handles both plain and back-reference-compressed format.

The `deser_auto` docstring in `wheel/src/api.rs` explicitly states: *"the backrefs deserializer is used (which also handles plain classic format)"*, confirming that `node_from_bytes_backrefs` is a strict superset.

`run_serialized_chia_program` at lines 55–56 uses only the basic variant:

```rust
let program = node_from_bytes(&mut allocator, program).map_err(eval_to_py)?;
let args   = node_from_bytes(&mut allocator, args).map_err(eval_to_py)?;
``` [1](#0-0) 

Every other Python-facing deserializer in the same file (`deser_backrefs`, `deser_auto`) correctly uses `node_from_bytes_backrefs`:

```rust
fn deser_backrefs(blob: &[u8]) -> PyResult<LazyNode> {
    let node = node_from_bytes_backrefs(&mut a, blob)...
}
``` [2](#0-1) 

The back-reference format uses `0xfe` as a special prefix byte. The basic deserializer (`node_from_bytes`) has no knowledge of this byte; it interprets `0xfe` as a multi-byte atom-length prefix (since `0xfe = 0b11111110` triggers a 6-byte length field in the CLVM encoding). This causes the deserializer to attempt to read an astronomically large atom, which will fail with a deserialization error rather than producing the correct tree. [3](#0-2) 

---

### Impact Explanation

Any caller that passes a back-reference-compressed program blob to `run_serialized_chia_program` will receive a deserialization error, even though the blob is a valid CLVM program. Because `node_from_bytes_backrefs` is a strict superset (it also accepts plain programs), switching to it would never break existing callers. The corrupted result is a hard deserialization failure (`PyErr`) returned to the Python caller instead of a valid `(cost, LazyNode)` pair.

If the Chia full node routes on-chain generator programs (which are routinely back-reference-compressed for block-space efficiency) through this function, the node would reject valid blocks or transactions, constituting a consensus-divergence or liveness failure.

---

### Likelihood Explanation

Back-reference compression (`node_to_bytes_backrefs`) is the standard serialization path for on-chain CLVM programs in the Chia ecosystem. The `run_serialized_chia_program` function is the primary Python API for executing CLVM programs and is the natural target for any Python-layer caller (e.g., `chia-blockchain`'s generator runner). Any such caller that serializes programs with `node_to_bytes_backrefs` and then passes the result to `run_serialized_chia_program` will trigger the failure. The attacker-controlled entry path is simply supplying a validly back-reference-compressed CLVM blob as the `program` or `args` argument.

---

### Recommendation

Replace both `node_from_bytes` calls in `run_serialized_chia_program` with `node_from_bytes_backrefs`:

```rust
let program = node_from_bytes_backrefs(&mut allocator, program).map_err(eval_to_py)?;
let args    = node_from_bytes_backrefs(&mut allocator, args).map_err(eval_to_py)?;
```

This is a strict improvement: `node_from_bytes_backrefs` accepts all inputs that `node_from_bytes` accepts, plus back-reference-compressed inputs. [4](#0-3) 

---

### Proof of Concept

1. Serialize any CLVM program using `node_to_bytes_backrefs` (the standard compressed path).
2. Pass the resulting bytes to `run_serialized_chia_program`.
3. Observe a deserialization error (`PyValueError`) instead of a successful execution result.
4. Pass the same bytes to `deser_backrefs` followed by manual `run_program` — observe correct execution.

The root cause is the wrong deserializer variant selected at `wheel/src/api.rs:55–56`, directly analogous to the original report's pattern of using the 4-parameter `_depositAndSwap()` where the 5-parameter version (with the extra resource-handling argument) was required. [5](#0-4) [3](#0-2)

pub fn run_serialized_chia_program(
    py: Python,
    program: &[u8],
    args: &[u8],
    max_cost: Cost,
    flags: u32,
) -> PyResult<(u64, LazyNode)> {
    let flags = ClvmFlags::from_bits_truncate(flags);
    let mut allocator = if flags.contains(ClvmFlags::LIMIT_HEAP) {
        Allocator::new_limited(500000000)
    } else {
        Allocator::new()
    };

    let r: Response = (|| -> PyResult<Response> {
        let program = node_from_bytes(&mut allocator, program).map_err(eval_to_py)?;
        let args = node_from_bytes(&mut allocator, args).map_err(eval_to_py)?;
        let dialect = ChiaDialect::new(flags);

        Ok(py.detach(|| run_program(&mut allocator, &dialect, program, args, max_cost)))
    })()?;
    adapt_response(py, allocator, r)
}
