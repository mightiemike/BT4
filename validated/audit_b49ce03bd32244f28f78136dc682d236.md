### Title
IBC NFT Packet Processing Bypasses URI Validation, Allowing Arbitrary URIs to Be Stored On-Chain — (`File: x/nft-transfer/keeper/packet.go`)

---

### Summary

The `x/nft-transfer` module's `processReceivedPacket` and `refundPacketToken` functions pass `data.ClassUri` and `data.TokenUris[i]` from an IBC packet directly into `IssueDenom` and `MintNFT` without applying the URI validation (`ValidateTokenURI`) that is enforced on every direct `MsgMintNFT` / `MsgEditNFT` transaction. A malicious counterparty chain can craft IBC NFT packets carrying URIs of arbitrary length and arbitrary scheme, causing them to be stored permanently in the chain's NFT KV state, violating the chain's own `MaxTokenURILen = 256` invariant and exposing downstream consumers to unvalidated URI content.

---

### Finding Description

`x/nft/types/validation.go` defines `ValidateTokenURI`, which enforces a hard cap of `MaxTokenURILen = 256` bytes on any NFT URI: [1](#0-0) 

This check is called in `MsgMintNFT.ValidateBasic()` and `MsgEditNFT.ValidateBasic()`, so every direct transaction path is gated: [2](#0-1) 

However, the IBC receive path in `processReceivedPacket` calls `IssueDenom` and `MintNFT` with URI values taken verbatim from the packet: [3](#0-2) 

The packet's `ValidateBasic()` validates addresses and token-ID/URI array length parity, but performs **no content or length check** on `ClassUri` or any element of `TokenUris`: [4](#0-3) 

`MintNFTUnverified`, which is the terminal write path, stores `tokenURI` directly into the KV store with no validation: [5](#0-4) 

The same bypass exists in `refundPacketToken`, which also calls `MintNFT` with `data.TokenUris[i]` on the error/timeout path: [6](#0-5) 

---

### Impact Explanation

**Corrupted value:** The `URI` field of `BaseNFT` records and the `Uri` field of `Denom` records stored in the NFT KV store can contain arbitrary byte strings that violate the chain's own `MaxTokenURILen = 256` invariant.

Concrete consequences:
1. **Invariant violation in chain state:** NFT URIs stored via IBC bypass the length and content constraints enforced on all direct-transaction paths, creating an inconsistent state that the chain's own genesis validation (`ValidateGenesis`) would reject if exported and re-imported.
2. **Unbounded storage growth:** A malicious counterparty chain can mint many NFTs with multi-kilobyte URIs, causing disproportionate KV store growth per NFT.
3. **Unvalidated URI content propagated to consumers:** Wallets, explorers, and dApps that query `QueryNFT` / `QueryDenoms` and render the returned URI field without their own validation receive attacker-controlled strings — the direct on-chain analog of the original report's unvalidated image URL rendering.

---

### Likelihood Explanation

**Medium.** The attack requires control of a counterparty chain that has an established IBC channel with the target chain, or the ability to submit a crafted IBC packet through a permissioned relayer. IBC channels are permissionless to open (any chain can open one), so a purpose-built malicious chain can establish a channel and send crafted packets. No privileged key on the target chain is required; the attacker only needs to be the operator of the counterparty chain.

---

### Recommendation

1. **Validate `ClassUri` and each element of `TokenUris` in `NonFungibleTokenPacketData.ValidateBasic()`** using the same `ValidateTokenURI` function already defined in `x/nft/types/validation.go`.

2. **Add URI validation inside `processReceivedPacket` and `refundPacketToken`** before calling `IssueDenom` or `MintNFT`, as a defense-in-depth measure.

3. **Consider adding URI scheme validation** to `ValidateTokenURI` itself (e.g., allow only `https://` and `ipfs://`) so the invariant is meaningful for both direct and IBC paths.

---

### Proof of Concept

**Attack path:**

1. Attacker operates chain B and opens an IBC channel to the target chain A.
2. Attacker crafts an IBC NFT transfer packet with:
   - `ClassUri`: a 10,000-byte arbitrary string (e.g., `"javascript:alert(1)"` repeated)
   - `TokenUris[0]`: a 10,000-byte arbitrary string
3. The packet is relayed to chain A and processed by `OnRecvPacket`: [7](#0-6) 
4. `data.ValidateBasic()` passes — it checks only that `len(TokenIds) == len(TokenUris)` and that addresses are valid; it does not check URI content or length. [8](#0-7) 
5. `processReceivedPacket` calls `IssueDenom(..., data.ClassUri, ...)` and `MintNFT(..., data.TokenUris[0], ...)`, storing the 10,000-byte strings in chain A's KV store. [9](#0-8) 
6. Any wallet or explorer querying `QueryNFT` on chain A receives the attacker-controlled URI string and, if it renders it without its own validation, is exposed to the unvalidated content — exactly the trust boundary violation described in the original report.

```go
func (nftpd NonFungibleTokenPacketData) ValidateBasic() error {
	if strings.TrimSpace(nftpd.ClassId) == "" {
		return newsdkerrors.Wrap(ErrInvalidClassID, "classId cannot be blank")
	}

	if len(nftpd.TokenIds) == 0 {
		return newsdkerrors.Wrap(ErrInvalidTokenID, "tokenId cannot be blank")
	}

	if len(nftpd.TokenIds) != len(nftpd.TokenUris) {
		return newsdkerrors.Wrap(ErrInvalidPacket, "tokenIds and tokenUris lengths do not match")
	}

	if strings.TrimSpace(nftpd.Sender) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "sender address cannot be blank")
	}

	// decode the sender address
	if _, err := sdk.AccAddressFromBech32(nftpd.Sender); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid sender address")
	}

	if strings.TrimSpace(nftpd.Receiver) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "receiver address cannot be blank")
	}

	// decode the receiver address
	if _, err := sdk.AccAddressFromBech32(nftpd.Receiver); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid receiver address")
	}
	return nil
}
```

```go
// MintNFTUnverified mints an NFT without verifying if the owner is the creator of denom
// Needed during genesis initialization
func (k Keeper) MintNFTUnverified(ctx sdk.Context, denomID, tokenID, tokenNm, tokenURI, tokenData string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	if k.HasNFT(ctx, denomID, tokenID) {
		return sdkerrors.Wrapf(types.ErrNFTAlreadyExists, "NFT %s already exists in collection %s", tokenID, denomID)
	}

	k.setNFT(
		ctx, denomID,
		types.NewBaseNFT(
			tokenID,
			tokenNm,
			owner,
			tokenURI,
			tokenData,
		),
	)
	k.setOwner(ctx, denomID, tokenID, owner)
	k.increaseSupply(ctx, denomID)

	return nil
```

```go
func (im IBCModule) OnRecvPacket(
	ctx sdk.Context,
	channelVersion string,
	packet channeltypes.Packet,
	relayer sdk.AccAddress,
) ibcexported.Acknowledgement {
	ack := channeltypes.NewResultAcknowledgement([]byte{byte(1)})

	var data types.NonFungibleTokenPacketData
	if err := types.ModuleCdc.UnmarshalJSON(packet.GetData(), &data); err != nil {
		ack = channeltypes.NewErrorAcknowledgement(fmt.Errorf("cannot unmarshal ICS-721 nft-transfer packet data"))
	}

	// only attempt the application logic if the packet data
	// was successfully decoded
	if ack.Success() {
		if err := im.keeper.OnRecvPacket(ctx, channelVersion, packet, data); err != nil {
			ack = types.NewErrorAcknowledgement(err)
		}
	}
	sdkCtx := sdk.UnwrapSDKContext(ctx)
	sdkCtx.EventManager().EmitEvent(
		sdk.NewEvent(
			types.EventTypePacket,
			sdk.NewAttribute(sdk.AttributeKeyModule, types.ModuleName),
			sdk.NewAttribute(sdk.AttributeKeySender, data.Sender),
			sdk.NewAttribute(types.AttributeKeyReceiver, data.Receiver),
			sdk.NewAttribute(types.AttributeKeyClassID, data.ClassId),
			sdk.NewAttribute(types.AttributeKeyTokenIDs, strings.Join(data.TokenIds, ",")),
			sdk.NewAttribute(types.AttributeKeyAckSuccess, fmt.Sprintf("%t", ack.Success())),
		),
	)

	// NOTE: acknowledgement will be written synchronously during IBC handler execution.
	return ack
}
```
