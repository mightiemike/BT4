### Title
Storage Deposit NEAR Tokens Permanently Trapped After Garbage Collection — (`contract/src/lib.rs`)

### Summary

`submit_blocks` requires callers to attach NEAR tokens to cover on-chain storage costs for new block headers. When the garbage collector (`run_mainchain_gc`) later removes those headers and frees the storage, the NEAR tokens that originally paid for that storage remain permanently locked in the contract's account balance. There is no withdrawal or recovery function anywhere in the contract.

### Finding Description

`submit_blocks` is `#[payable]` and enforces a storage deposit: [1](#0-0) 

The function measures net storage delta **after** GC runs, refunds only the surplus from the **current call**, and retains `required_deposit` in the contract's balance to cover the newly added headers. [2](#0-1) 

Later, `run_mainchain_gc` removes those same headers from all three storage structures: [3](#0-2) 

When GC removes a header, the storage bytes are freed and the NEAR tokens that paid for them are no longer needed for storage staking. They become "free balance" inside the contract's account. However, the contract exposes no `withdraw`, `recover_storage_deposit`, or equivalent function. The complete public API is: [4](#0-3) 

`get_*` and `verify_*` are view-only; `run_mainchain_gc` only removes storage; `migrate` is `#[private]`. None transfers the freed balance out.

The documentation confirms GC is designed to run continuously for the lifetime of the contract (`gc_threshold = 52704` ≈ one year of Bitcoin blocks): [5](#0-4) 

### Impact Explanation

Every `submit_blocks` call that results in net new storage permanently donates NEAR to the contract. Once GC removes those headers, the corresponding NEAR is freed from storage staking but is irrecoverable. Over the intended multi-year operational lifetime of the contract, the cumulative trapped amount grows proportionally to the total number of headers ever submitted and GC'd — on the order of `gc_threshold × storage_byte_cost × header_size_bytes` per GC cycle, repeating indefinitely.

### Likelihood Explanation

This is triggered by normal, intended operation. The relayer is designed to submit blocks continuously; GC is designed to run automatically inside every `submit_blocks` call. No adversarial input is required. Any relayer (trusted or `UnrestrictedSubmitBlocks`) who deposits NEAR to submit headers will have those funds trapped once GC removes the corresponding headers.

### Recommendation

Track the total storage deposit balance separately and refund the freed amount to the original depositor (or a configurable admin/treasury address) when GC removes headers. Concretely:

- Record `env::storage_usage()` before and after `run_mainchain_gc` inside `submit_blocks`.
- Compute the freed bytes and the corresponding NEAR value.
- Transfer that amount to `env::predecessor_account_id()` or a designated admin account, mirroring the pattern already used for the per-call surplus refund at line 192.

### Proof of Concept

1. Relayer calls `submit_blocks([h1, h2, ..., hN])` with deposit `D = N × storage_byte_cost × header_size`. Headers are stored; `D` is retained by the contract.
2. Chain grows past `gc_threshold`. On the next `submit_blocks` call, `run_mainchain_gc` removes `h1..hN` from `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height`.
3. `env::storage_usage()` after GC is now **lower** than `initial_storage`, so `diff_storage_usage = env::storage_usage().saturating_sub(initial_storage) = 0`, `required_deposit = 0`, and the full current-call deposit is refunded — but `D` from step 1 remains in the contract balance with no path out.
4. Repeat for every GC cycle. The trapped balance grows without bound and is unrecoverable.

    /// # Panics
    /// Cannot parse headers len as u64
    #[payable]
    #[pause]
    #[trusted_relayer]
    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
