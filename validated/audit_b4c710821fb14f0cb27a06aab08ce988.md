### Title
`op_unknown` cost check performed before multiplier application allows returned cost to exceed `max_cost` — (File: `src/more_ops.rs`)

---

### Summary

In `op_unknown`, `check_cost(cost, max_cost)` is called on the **pre-multiplication** base cost. Immediately after, the cost is multiplied by `(cost_multiplier + 1)`, where `cost_multiplier` is attacker-controlled and can be up to `u32::MAX`. The post-multiplication cost returned in the `Reduction` is never re-validated against `max_cost`. The only post-multiplication guard is `if cost > u32::MAX as u64`, which caps the returned cost at `u32::MAX` but does not enforce the caller's budget.

---

### Finding Description

In `src/more_ops.rs`, `op_unknown` handles unknown opcodes in lenient mode. The opcode byte layout encodes two attacker-controlled fields:

- **`cost_function`** (top 2 bits of the last byte): selects one of four cost formulas (constant, add-like, mul-like, concat-like).
- **`cost_multiplier`** (all bytes except the last, interpreted as a `u32`): a scaling factor applied to the base cost. [1](#0-0) 

The base cost is computed from `cost_function`, then the budget check is performed: [2](#0-1) 

The sequence is:
1. `check_cost(cost, max_cost)` — validates the **pre-multiplication** cost.
2. `cost *= cost_multiplier + 1` — scales the cost by up to `u32::MAX + 1`.
3. `if cost > u32::MAX as u64` — only rejects if the result exceeds `u32::MAX`; does **not** re-check against `max_cost`.
4. `Ok(Reduction(cost as Cost, allocator.nil()))` — returns the scaled cost to the caller.

This is directly analogous to the external report: just as `steth_amount - steth_amount_adj` may be smaller than the transfer fee (making the net return negative/dust), here the pre-multiplication cost passes the budget check but the post-multiplication cost can far exceed the budget — the check is applied at the wrong point in the arithmetic.

**Concrete trigger**: craft an opcode with `cost_function = 0` (base cost = 1) and `cost_multiplier = max_cost`. The pre-multiplication cost is 1, which passes `check_cost(1, max_cost)` for any `max_cost ≥ 1`. The post-multiplication cost is `1 × (max_cost + 1) = max_cost + 1`, which exceeds `max_cost` and is returned without error. [3](#0-2) 

---

### Impact Explanation

The `Cost` type is `u64`. [4](#0-3) 

The `check_cost` calls inside operators are the primary per-operator budget enforcement mechanism. `run_program` passes the remaining budget as `max_cost` to each operator and expects the operator to reject if it would exceed that budget. Because `op_unknown` returns a `Reduction` with a cost that can exceed `max_cost` — up to `u32::MAX` (4,294,967,295) — the running cost total in `run_program` can be pushed past the original budget without the operator having raised `EvalErr::CostExceeded`. This constitutes **undercharged execution**: a program that should be rejected for exceeding its cost limit is instead accepted, with the reported cost exceeding the agreed budget. In a consensus context this can cause divergence between nodes that enforce the limit strictly at the `run_program` accumulation level and those that rely on per-operator checks.

---

### Likelihood Explanation

Unknown opcodes are reachable whenever the dialect is in lenient mode (e.g., `ClvmFlags` does not set a strict-only flag). An attacker who can submit arbitrary CLVM bytes — the normal attacker model for a blockchain VM — can trivially craft a multi-byte opcode with any desired `cost_multiplier`. No special privileges are required. The opcode just must not start with `0xffff` and must not be empty. [5](#0-4) 

---

### Recommendation

Move `check_cost` to **after** the multiplication so the enforced value matches the value actually returned:

```rust
assert!(cost > 0);
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}
// Now check the post-multiplication cost against the budget
check_cost(cost, max_cost)?;
Ok(Reduction(cost as Cost, allocator.nil()))
```

This mirrors the pattern used by every other operator in the file, where `check_cost` is called on the value that will actually be returned.

### Proof of Concept

#   op[0..3] = [0x00, 0x03, 0xe8]  -> cost_multiplier = 0x0003e8 = 1000
#   op[3]    = 0x00                 -> cost_function = 0 (constant, base cost = 1)
#
# Call: op_unknown(allocator, opcode, nil_args, max_cost=1000)
#
# Step 1: cost = 1  (cost_function == 0)
# Step 2: check_cost(1, 1000) -> Ok  (1 <= 1000, passes)
# Step 3: cost *= 1000 + 1 = 1001
# Step 4: 1001 <= u32::MAX, so no Invalid error
# Step 5: returns Ok(Reduction(1001, nil))
#
# Result: returned cost 1001 > max_cost 1000, budget exceeded without error.
