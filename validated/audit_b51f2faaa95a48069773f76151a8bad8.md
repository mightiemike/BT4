### Title
Hardcoded `Fee(1)` Placeholder in Consensus-Path L1 Handler Conversion Silently Bypasses the `paid_fee_on_l1` Fee Guard — (`File: crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

When an `L1HandlerTransaction` arrives through the consensus path (proposer → validator), `convert_consensus_l1_handler_to_internal_l1_handler` unconditionally injects `paid_fee_on_l1 = Fee(1)` — a hardcoded placeholder — instead of the actual fee paid on L1. The blockifier's `execute_raw` for L1 handler transactions contains a guard that rejects any transaction where `paid_fee_on_l1 == Fee(0)`. Because the placeholder is always `Fee(1)`, this guard is permanently bypassed for every L1 handler transaction that enters the system through the consensus conversion path, regardless of whether any fee was actually paid on L1.

---

### Finding Description

**Root cause — wrong value injected at conversion boundary**

`crates/apollo_transaction_converter/src/transaction_converter.rs` lines 473–483:

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),   // ← hardcoded placeholder, never the real L1 fee
    )?)
}
```

The `ConsensusTransaction::L1Handler` wire type (`crates/starknet_api/src/transaction.rs`) carries only `version`, `nonce`, `contract_address`, `entry_point_selector`, and `calldata`. The `paid_fee_on_l1` field is **not transmitted** in the consensus message, so the converter has no real value to use — and it substitutes `Fee(1)` unconditionally.

**Guard that is bypassed — blockifier execution check**

`crates/blockifier/src/transaction/l1_handler_transaction.rs` lines 101–113:

```rust
// TODO(Arni): Consider removing this check. It is covered by the starknet core contract.
let paid_fee = self.paid_fee_on_l1;
// For now, assert only that any amount of fee was paid.
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee {
            paid_fee,
            actual_fee: receipt.fee,
        },
    )));
}
```

The guard fires only when `paid_fee == Fee(0)`. Because the converter always supplies `Fee(1)`, the guard **never fires** for any consensus-path L1 handler transaction, even one whose actual `paid_fee_on_l1` is zero.

**Wrong value propagated downstream — Cende serialization**

`crates/apollo_consensus_orchestrator/src/cende/central_objects.rs` lines 383–394:

```rust
impl From<L1HandlerTransaction> for CentralL1HandlerTransaction {
    fn from(tx: L1HandlerTransaction) -> CentralL1HandlerTransaction {
        CentralL1HandlerTransaction {
            ...
            paid_fee_on_l1: tx.paid_fee_on_l1,  // always Fee(1) from consensus path
        }
    }
}
```

The placeholder `Fee(1)` is serialised into the Cende blob that is sent to the central system, so the authoritative record of the L1 fee paid is permanently wrong for every consensus-path L1 handler transaction.

**Structural analogy to the seed bug**

The external bug: `pool.repayCreditAccount(debtData.debt, 0, loss)` passes `profit = 0` instead of `debtData.accruedInterest`, so the interest component is silently dropped and the pool's accounting diverges from reality.

The sequencer analog: `L1HandlerTransaction::create(tx, &self.chain_id, Fee(1))` passes `paid_fee_on_l1 = Fee(1)` instead of the actual L1 fee, so the fee guard is silently bypassed and the Cende record diverges from the real L1 payment.

---

### Impact Explanation

1. **Fee guard permanently bypassed.** Every L1 handler transaction that enters through the consensus conversion path passes the `paid_fee == Fee(0)` check unconditionally. A proposer can include an L1 handler transaction for which zero fee was paid on L1 and every honest validator will execute it without error.

2. **Incorrect authoritative record.** The `paid_fee_on_l1` value written into the Cende blob is always `Fee(1)`, not the real L1 fee. This corrupts the off-chain record used for accounting and auditing of L1→L2 messages.

3. **Execution of unverified L1→L2 messages.** Because the `paid_fee_on_l1` field is the only per-transaction signal that an L1 event actually occurred and was paid for, bypassing it means the sequencer will execute L1 handler transactions that have no corresponding verified L1 event, producing wrong state transitions.

This matches the allowed impact: *"Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input"* and *"Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."*

---

### Likelihood Explanation

The conversion is exercised for every L1 handler transaction that arrives through the consensus path. No special attacker capability is required beyond being a block proposer (a normal validator role in the Tendermint-based consensus). The TODO comment confirms the developers are aware the value is wrong but have not yet fixed it.

---

### Recommendation

1. **Transmit `paid_fee_on_l1` in the consensus wire format.** Add the field to the `ConsensusTransaction::L1Handler` variant so validators can verify the actual fee paid on L1.

2. **Until (1) is done, do not bypass the guard.** If the real value cannot yet be transmitted, the guard in `l1_handler_transaction.rs` must be strengthened (e.g., check against the actual L1 event log) rather than relying on a placeholder that always satisfies it.

3. **Do not propagate the placeholder to Cende.** The Cende serialisation path should not emit `paid_fee_on_l1` until a real value is available, or should emit a sentinel that the central system treats as "unknown" rather than `Fee(1)`.

---

### Proof of Concept

```
1. Proposer constructs ConsensusTransaction::L1Handler(tx) where tx has
   any contract_address / entry_point_selector / calldata.
   (paid_fee_on_l1 is not part of the wire type — it is absent.)

2. Validator calls:
     convert_consensus_l1_handler_to_internal_l1_handler(tx)
   which calls:
     L1HandlerTransaction::create(tx, &chain_id, Fee(1))
   producing an executable L1HandlerTransaction with paid_fee_on_l1 = Fee(1).

3. Blockifier executes the transaction.
   In execute_raw():
     let paid_fee = self.paid_fee_on_l1;  // = Fee(1)
     if paid_fee == Fee(0) { ... }        // false → guard skipped

4. Execution succeeds. The L1 handler body runs, mutating L2 state,
   even though no fee was ever paid on L1 and no L1 event exists.

5. CentralL1HandlerTransaction { paid_fee_on_l1: Fee(1), ... } is
   written to the Cende blob with the wrong fee value.
