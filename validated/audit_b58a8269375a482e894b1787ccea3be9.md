### Title
Division by Zero in `PerpEngine.socializeSubaccount()` Blocks Subaccount Finalization When `openInterest` Is Zero — (File: `core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount()` loops over all perp products and, for any product where the subaccount has a negative `vQuoteBalance` that insurance cannot fully cover, divides by `state.openInterest`. If `openInterest` is zero for that product (all other participants have closed their positions), the division reverts, permanently blocking the entire subaccount finalization path.

---

### Finding Description

In `PerpEngine.socializeSubaccount()`, the function iterates over all registered product IDs:

```solidity
for (uint128 i = 0; i < _productIds.length; ++i) {
    uint32 productId = _productIds[i];
    (State memory state, Balance memory balance) = getStateAndBalance(productId, subaccount);
    if (balance.vQuoteBalance < 0) {
        ...
        if (balance.vQuoteBalance < 0) {
            int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest); // ← division by zero
            state.cumulativeFundingLongX18 += fundingPerShare;
            state.cumulativeFundingShortX18 -= fundingPerShare;
            balance.vQuoteBalance = 0;
        }
        ...
    }
}
``` [1](#0-0) 

There is no guard against `state.openInterest == 0` before the division. A subaccount can have a negative `vQuoteBalance` (accumulated funding losses) even when its own `balance.amount` is zero — which is precisely the state required for finalization. If all other participants in that perp market have also closed their positions, `openInterest` becomes zero, and the `div()` call reverts.

This function is called from `ClearinghouseLiq._finalizeSubaccount()`:

```solidity
v.insurance = perpEngine.socializeSubaccount(txn.liquidatee, v.insurance);
``` [2](#0-1) 

`_finalizeSubaccount()` is triggered when a liquidator submits a `LiquidateSubaccount` transaction with `productId == type(uint32).max`. The revert propagates through the `delegatecall` in `Clearinghouse.liquidateSubaccount()`:

```solidity
(bool success, bytes memory result) = clearinghouseLiq.delegatecall(liquidateSubaccountCall);
require(success, string(result));
``` [3](#0-2) 

This causes the entire finalization transaction to revert.

The finalization path is entered via `processTransactionImpl` in `EndpointTx.sol`:

```solidity
if (txType == IEndpoint.TransactionType.LiquidateSubaccount) {
    ...
    clearinghouse.liquidateSubaccount(signedTx.tx);
``` [4](#0-3) 

---

### Impact Explanation

Subaccount finalization is permanently blocked for any underwater subaccount that has a negative `vQuoteBalance` in a perp product where `openInterest` has dropped to zero. The underwater subaccount cannot be resolved: its bad debt cannot be socialized, the insurance fund accounting is left inconsistent, and the subaccount remains in a permanently broken state. This corrupts the protocol's solvency accounting and prevents the liquidation system from completing its critical function.

---

### Likelihood Explanation

The condition requires: (1) a perp product where all other participants have closed their positions (`openInterest == 0`), and (2) the subaccount being finalized still carries a negative `vQuoteBalance` in that product. This is realistic in low-activity or newly-listed perp markets where the liquidatee was one of few participants. The subaccount's `vQuoteBalance` can remain negative even after its `amount` reaches zero (a prerequisite for finalization) due to accumulated funding payments. No privileged access is required — any liquidator can trigger this path.

---

### Recommendation

Add an explicit guard for `state.openInterest == 0` before the division inside `PerpEngine.socializeSubaccount()`. If there are no other open-interest participants to socialize against, the remaining negative `vQuoteBalance` should be zeroed out (absorbed by the protocol) rather than causing a revert:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

---

### Proof of Concept

1. A perp market (e.g., product ID 2) has two participants: Alice and Bob (the subaccount to be finalized).
2. Bob accumulates a negative `vQuoteBalance` (e.g., `-100e18`) through funding payments while holding a position.
3. Bob's position is closed (`balance.amount = 0`), but `vQuoteBalance` remains `-100e18`.
4. Alice also closes her position, making `state.openInterest = 0` for product 2.
5. Bob's account becomes underwater (maintenance health < 0) due to the negative `vQuoteBalance`.
6. A liquidator submits `LiquidateSubaccount` with `productId = type(uint32).max` to finalize Bob.
7. `_finalizeSubaccount()` verifies all perp `balance.amount == 0` ✓, then calls `perpEngine.socializeSubaccount(Bob, insurance)`.
8. The loop reaches product 2: `balance.vQuoteBalance = -100e18 < 0`, insurance is insufficient, so it enters the inner branch and executes `-balance.vQuoteBalance.div(state.openInterest)` = `100e18 / 0` → **division by zero revert**.
9. The revert propagates through `delegatecall`, blocking finalization permanently. Bob's bad debt is never resolved.

    function socializeSubaccount(bytes32 subaccount, int128 insurance)
        external
        returns (int128)
    {
        require(msg.sender == address(_clearinghouse), ERR_UNAUTHORIZED);

        uint32[] memory _productIds = getProductIds();
        for (uint128 i = 0; i < _productIds.length; ++i) {
            uint32 productId = _productIds[i];
            (State memory state, Balance memory balance) = getStateAndBalance(
                productId,
                subaccount
            );
            if (balance.vQuoteBalance < 0) {
                int128 insuranceCover = MathHelper.min(
                    insurance,
                    -balance.vQuoteBalance
                );
                insurance -= insuranceCover;
                balance.vQuoteBalance += insuranceCover;
                state.availableSettle += insuranceCover;

                // actually socialize if still not enough
                if (balance.vQuoteBalance < 0) {
                    // socialize across all other participants
                    int128 fundingPerShare = -balance.vQuoteBalance.div(
                        state.openInterest
                    );
                    state.cumulativeFundingLongX18 += fundingPerShare;
                    state.cumulativeFundingShortX18 -= fundingPerShare;
                    balance.vQuoteBalance = 0;
                }
                _setState(productId, state);
                _setBalanceAndUpdateBitmap(productId, subaccount, balance);
            }
        }
        return insurance;
    }
