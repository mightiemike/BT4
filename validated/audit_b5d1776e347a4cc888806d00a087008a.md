Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the intermediary router address instead of the end user, enabling full allowlist bypass â€” (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` as the direct caller of `pool.swap()`. When swaps are routed through the public, permissionless `MetricOmmSimpleRouter`, `sender` equals the router's address. A pool admin who allowlists the router to enable router-mediated swaps for permitted users inadvertently opens the pool to every unprivileged user, because any caller can invoke the router and have their swap pass the allowlist check under the router's identity. Independently, legitimately allowlisted users who attempt to swap through the router are blocked, rendering the supported periphery path unusable for curated pools.

## Finding Description
In `MetricOmmPool.swap`, `msg.sender` (the direct caller of `pool.swap()`) is passed as `sender` to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // whoever called pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension via `abi.encodeCall`:

```solidity
// metric-core/contracts/ExtensionCalling.sol L162-165
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, zeroForOne, ...)
)
```

`SwapAllowlistExtension.beforeSwap` then uses that `sender` as the allowlist key:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter` is a public, permissionless contract. Its `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` functions call `pool.swap()` directly with no access control:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

When any user calls the router, `msg.sender` inside the pool is the router address. The allowlist check evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Mode A â€” Bypass:** The pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps for permitted users. Because the router is public, every unprivileged user can now route through it and pass the allowlist check. The restriction is completely nullified.

**Mode B â€” Broken flow:** The admin does not allowlist the router. Legitimately allowlisted users who attempt to swap through `MetricOmmSimpleRouter` receive `NotAllowedToSwap`, even though they are individually permitted. They are forced to call the pool directly, breaking the supported periphery flow.

No existing guard prevents either mode. The `allowAllSwappers` escape hatch only bypasses the check entirely; it does not fix the actor-identity mismatch.

## Impact Explanation
Mode A constitutes a direct loss of LP principal: a pool designed to restrict trading to specific counterparties (e.g., whitelisted market makers or KYC'd participants) can be traded against by any unprivileged address. Unauthorized swaps execute at oracle-derived prices without LP consent, draining LP value and breaking the core pool invariant that only permitted swappers may trade. This meets the "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" criteria. Mode B independently breaks the core swap flow for legitimately allowlisted users through the supported periphery path.

## Likelihood Explanation
Allowlisting the router is the natural, expected configuration for any pool that wants to support both direct and router-mediated swaps for its permitted users. The admin has no way to achieve this intent correctly with the current code â€” allowlisting the router is the only mechanism available, and it triggers the bypass. The trigger requires no privileged malicious action; the admin acts in good faith, and any unprivileged user completes the exploit by calling the public, deployed router. The exploit is repeatable on every swap.

## Recommendation
The extension must gate the actual economic actor, not the intermediary. Two options:

1. **Pass the originating user through `extensionData`**: The router encodes the end user's address in `extensionData`; the extension decodes and checks it. This requires the router to be trusted to supply the correct value.
2. **Add an originator field to the hook interface**: Extend `beforeSwap` with an explicit `originator` parameter that the pool populates from a transient context set by the router before calling `pool.swap()`. This is the most robust fix and mirrors how Uniswap v4 handles hook actor binding.

Until fixed, pools relying on `SwapAllowlistExtension` should not allowlist the router and should document that router-mediated swaps are incompatible with the allowlist guard.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension configured.
2. Admin calls: extension.setAllowedToSwap(pool, router, true)
   // Intended: allow router so permitted users can swap via router
3. Disallowed user (alice, NOT in allowlist) calls:
   MetricOmmSimpleRouter.exactInputSingle({pool: pool, ..., extensionData: ""})
4. Router calls: pool.swap(recipient, zeroForOne, amount, priceLimit, "", extensionData)
   // msg.sender inside pool = router address
5. Pool calls: _beforeSwap(msg.sender=router, ...)
6. Extension evaluates: allowedSwapper[pool][router] == true â†’ passes
7. Alice's swap executes successfully despite not being allowlisted.
```
