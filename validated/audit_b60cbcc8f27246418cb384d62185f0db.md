Audit Report

## Title
SwapAllowlistExtension Gates on Router Address Instead of Actual Swapper, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of the `pool.swap()` call. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool level is the router contract, not the end user. Any pool admin who allowlists the router to support router-mediated swaps for their users simultaneously grants every unprivileged address the ability to bypass the allowlist entirely.

## Finding Description

**Root cause — `MetricOmmPool.swap` passes `msg.sender` (the router) as `sender`:**

In `MetricOmmPool.sol` L230–240, `_beforeSwap` is called with `msg.sender` as the first argument. When `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) calls `pool.swap(...)`, `msg.sender` inside the pool is the router address, not the original caller.

**`ExtensionCalling._beforeSwap` forwards `sender` unchanged:**

`ExtensionCalling.sol` L149–177 encodes `sender` directly into the `beforeSwap` call dispatched to every configured extension.

**`SwapAllowlistExtension.beforeSwap` gates on that forwarded `sender`:**

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router. The check passes whenever `allowedSwapper[pool][router] == true`.

**Exploit flow:**

```
attacker → MetricOmmSimpleRouter.exactInputSingle(pool, ...)
             → pool.swap(recipient=attacker, ...) [msg.sender = router]
                  → _beforeSwap(sender=router, ...)
                       → allowedSwapper[pool][router] == true → PASSES
                            → swap executes for attacker
```

For any pool admin who wants allowlisted users to swap via the standard router, they must call `setAllowedToSwap(pool, router, true)`. That single admin action opens the allowlist to every address that can call the router.

**Contrast with `DepositAllowlistExtension`:**

`DepositAllowlistExtension.sol` L32–42 ignores `sender` and gates on `owner` — the address that owns the LP position — which is passed separately and is not overridden by the router intermediary. The swap extension has no analogous field to identify the true initiator.

## Impact Explanation

A pool deploying `SwapAllowlistExtension` intends to restrict swaps to a curated set (KYC-verified users, institutional counterparties, whitelisted market makers). Once the router is allowlisted — the required operational step to support normal user flows — the allowlist guard silently fails open for all router-mediated swaps. Any address can execute swaps on a pool they are explicitly excluded from. This constitutes broken core pool functionality: the curation guarantee is entirely defeated, which can result in direct loss of the pool's intended access control and exposure to counterparties the pool admin explicitly rejected.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap entry point. Any pool admin who wants their allowlisted users to use the standard router must allowlist the router address — this is the expected operational path, not an edge case. The bypass requires no special access, no privileged role, and no unusual setup: any unprivileged address calls a public router function. The condition is met on every curated pool that supports router-mediated swaps.

## Recommendation

Gate on the economically relevant actor — the address initiating and paying for the swap — not the intermediate dispatcher. Two sound approaches:

1. **Pass the original caller through the router.** Have `MetricOmmSimpleRouter` encode the real `msg.sender` inside `extensionData` and have `SwapAllowlistExtension.beforeSwap` read and verify that value (with appropriate authentication to prevent spoofing).

2. **Gate on `recipient` instead of `sender`.** The `recipient` argument is the address receiving swap output and is the closest available proxy for the true initiator in the current interface. This avoids interface changes but requires the extension to treat `recipient` as the identity to check.

The cleanest fix mirrors `DepositAllowlistExtension`: identify the actor the pool's economic action is attributed to and check that identity, not the intermediary.

## Proof of Concept

```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension configured
  allowedSwapper[pool][alice]   = true   // intended grantee
  allowedSwapper[pool][router]  = true   // admin must set to support router swaps
  allowedSwapper[pool][attacker]= false  // attacker is NOT allowlisted

Attack:
  attacker calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: attacker, ...})
    → pool.swap(recipient=attacker, ...) [msg.sender = router]
    → _beforeSwap(sender=router, ...)
    → allowedSwapper[pool][router] == true → PASSES
    → swap executes for attacker

Result:
  attacker swaps on a pool they are explicitly excluded from.
  The allowlist guard is completely bypassed.
```

Foundry test outline: deploy pool with `SwapAllowlistExtension`, configure `allowedSwapper[pool][router] = true` and `allowedSwapper[pool][attacker] = false`, call `router.exactInputSingle` from `attacker`, assert the swap succeeds (no `NotAllowedToSwap` revert).
