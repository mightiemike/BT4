### Title
Fork Retarget Uses Mainchain Lookback Instead of Fork Branch — (`contract/src/litecoin.rs` + `contract/src/lib.rs`)

---

### Summary

`get_next_work_required` resolves the difficulty-period tail block via `blocks_getter.get_header_by_height`, which is implemented exclusively against `mainchain_height_to_header`. Fork blocks are stored only in `headers_pool` and are never inserted into `mainchain_height_to_header`. Therefore, when a fork block lands on a retarget boundary and the fork diverges at or before the lookback height, the contract silently substitutes the mainchain's block at that height — with its own timestamp — for the fork's block. The resulting `expected_bits` value diverges from what the real Litecoin network would compute, and the contract accepts a retarget block that the source chain would reject as having an invalid `bits` field.

---

### Finding Description

**Root cause — `get_header_by_height` reads only the mainchain index:** [1](#0-0) 

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // ← mainchain-only map
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

**Fork blocks are never written to `mainchain_height_to_header`:** [2](#0-1) 

```rust
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);  // headers_pool only
}
```

**Retarget path calls `get_header_by_height` for the period-tail block:** [3](#0-2) 

```rust
let first_block_height = prev_block_header.block_height - blocks_to_go_back;
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),  // ← mainchain timestamp
)
```

**Contrast with the non-retarget (min-difficulty) path**, which correctly walks the fork via `get_prev_header` (hash-linked through `headers_pool`): [4](#0-3) 

The non-retarget path is not affected; only the retarget boundary path is broken.

---

### Impact Explanation

When the fork diverges at or before height `T − 2016` (where `T` is the next retarget boundary), the contract fetches the **mainchain** block at `first_block_height` and uses its timestamp `T_main` instead of the fork's block timestamp `T_fork`. The `actual_time_taken` fed into `calculate_next_work_required` is therefore wrong:

- If `T_main > T_fork` (attacker sets the fork's divergence block to an earlier timestamp than the mainchain block at the same height), the contract computes a **shorter** elapsed period → **higher difficulty** → more `work_from_bits` per block → the fork's retarget block carries inflated chainwork.
- The fork's `bits` field is set to match the contract's (incorrect) expected value. The real Litecoin network, which would compute `actual_time_taken` using `T_fork`, derives a **different** `bits` value and rejects the block outright.
- Once the fork's cumulative chainwork exceeds the mainchain's, `reorg_chain` promotes it: the fraudulent fork becomes canonical in the contract's `mainchain_height_to_header` / `mainchain_tip_blockhash`.
- `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` then confirm transactions inside this fork as final, while the real Litecoin network has never accepted those blocks. [5](#0-4) 

---

### Likelihood Explanation

On **Litecoin testnet** (`pow_allow_min_difficulty_blocks = true`, `pow_target_spacing = 150 s`): [6](#0-5) 

The attacker can build the 2016-block fork cheaply: any block whose timestamp exceeds `prev.time + 300 s` is accepted at minimum difficulty (`0x1e0fffff`). Only the single retarget block requires real scrypt work at the (contract-computed) target. The trusted-relayer gate is the only access-control barrier; a compromised or colluding relayer key is sufficient to submit the crafted batch via `submit_blocks`. [7](#0-6) 

---

### Recommendation

Replace the height-indexed lookup in `get_next_work_required` with a hash-linked walk that follows the fork branch. Pass the fork's `prev_block_header` and walk backwards through `headers_pool` via `get_prev_header` until the correct ancestor at `first_block_height` is reached. This mirrors how the non-retarget min-difficulty path already traverses the chain correctly, and matches the reference Litecoin implementation which walks by `pprev` pointer, not by height index.

---

### Proof of Concept

1. Initialize the contract on Litecoin testnet at a height `H` where `(H+1) % 2016 == 0` (one block before a retarget boundary).
2. Advance the mainchain to height `H + 2016` (the retarget block). Record the mainchain block at height `H` — call its timestamp `T_main`.
3. Build a competing fork starting at height `H`:
   - Fork block at height `H`: set timestamp `T_fork < T_main` (e.g., `T_main − 10000 s`, still above MTP).
   - Fork blocks `H+1 … H+2015`: use min-difficulty (`time > prev.time + 300 s`).
   - Fork block at height `H+2016` (retarget): compute `expected_bits` as the contract will — using `T_main` as `first_block_time` and the fork's `prev.time` as `prev_block_time`. Mine a scrypt-valid header with this `bits` value.
4. Submit the fork batch via `submit_blocks`. The contract calls `get_header_by_height(H)`, retrieves the **mainchain** block (timestamp `T_main`), computes `expected_bits` matching the crafted header, and accepts it.
5. If the fork's cumulative chainwork exceeds the mainchain's, `reorg_chain` fires and the fraudulent fork becomes canonical.
6. Call `verify_transaction_inclusion` with a transaction inside the fork — it returns `true`.
7. Verify that the real Litecoin testnet node rejects the fork's retarget block with `bad-diffbits` (because it computes `expected_bits` using `T_fork`, not `T_main`).
