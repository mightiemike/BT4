### Title
Difficulty Calculation for Fork Blocks Uses Stale Mainchain Ancestor Instead of Actual Fork Ancestor - (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

---

### Summary

When computing the required difficulty for a fork block at a retarget boundary, all three chain modules call `get_header_by_height` to retrieve the boundary block. This function always reads from `mainchain_height_to_header`, returning the **current mainchain block** at that height rather than the fork's actual ancestor. When a fork diverges before the boundary, the two blocks differ, causing the difficulty calculation to use a wrong timestamp. An unprivileged proof submitter can exploit this to have fork blocks accepted with lower difficulty than the real chain requires, ultimately corrupting the canonical-chain mapping.

---

### Finding Description

In `get_next_work_required` for all three chain modules, the first block of the difficulty adjustment interval is fetched by height:

```

**`contract/src/bitcoin.rs` line 81:**

```

**`contract/src/litecoin.rs` line 88:**
```rust

```

The `get_header_by_height` implementation in `contract/src/lib.rs` lines 677‚Äì682 always resolves through `mainchain_height_to_header`:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

`mainchain_height_to_header` is only updated when mainchain blocks are stored (lines 651‚Äì655). It is never updated to reflect fork ancestors. Therefore, when a fork diverges before the difficulty boundary height, the block returned at `height_first` is the **mainchain block** at that height, not the fork's ancestor. The two blocks can have entirely different timestamps, producing a wrong `first_block_time` and thus a wrong expected difficulty.

The TODO comment in `dogecoin.rs` explicitly acknowledges this uncertainty.

For **Dogecoin post-block 145,000**, `difficulty_adjustment_interval = 1`, so every single fork block after the first triggers this path. For **Bitcoin and Litecoin**, the interval is 2016 blocks, so any fork that crosses a retarget boundary is affected.

---

### Impact Explanation

The corrupted `first_block_time` feeds directly into `calculate_next_work_required` in all three modules. For Dogecoin:

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
```

If the attacker's fork block at `height_first` carries a large timestamp (T_fork >> T_mainchain), the correct timespan for the next block would be very small, requiring very high difficulty. But the contract uses T_mainchain instead, producing a normal timespan and normal (easy) difficulty. The fork block therefore passes `check_pow` with lower difficulty than the real Dogecoin network would require.

Lower difficulty means lower `work_from_bits` per block. However, the attacker can submit an unbounded number of such blocks (paying only NEAR storage costs). Once the fork's cumulative `chain_work` exceeds the mainchain's, `submit_block_header_inner` triggers `reorg_chain`, overwriting `mainchain_tip_blockhash` and the `mainchain_height_to_header` / `mainchain_header_to_height` maps with the attacker's fabricated chain. All subsequent `verify_transaction_inclusion` calls then operate against this corrupted canonical chain, returning false confirmations for real transactions and true confirmations for fabricated ones.

---

### Likelihood Explanation

For **Dogecoin**: The retarget fires on every block (post-145,000). Any fork of depth ‚â• 2 triggers the wrong-ancestor lookup on every subsequent fork block. An unprivileged NEAR caller can submit fork headers via the public `submit_blocks` entry point with no special permissions. The attack requires only crafting headers with a manipulated timestamp in the divergence block and submitting enough low-difficulty successors to outweigh the mainchain's chainwork.

For **Bitcoin/Litecoin**: The fork must span a 2016-block retarget boundary, which is less common but not impossible for a long-running fork scenario.

---

### Recommendation

Replace `get_header_by_height` in the retarget path with an ancestor traversal that follows `prev_block_hash` links from `prev_block_header` backward until reaching `height_first`. This mirrors how Bitcoin Core's `GetAncestor` works and ensures the correct fork ancestor is used regardless of what the mainchain mapping contains.

```rust
// Instead of:
let interval_tail = blocks_getter.get_header_by_height(first_block_height);

// Use ancestor traversal:
let mut cursor = prev_block_header.clone();
while cursor.block_height > first_block_height {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
let interval_tail = cursor;
```

### Proof of Concept

**Dogecoin mainnet (post-145,000), fork of depth 2:**

1. Mainchain state: blocks at heights N‚àí2 (timestamp T‚ÇÄ), N‚àí1 (timestamp T‚ÇÅ), N (timestamp T‚ÇÇ). `mainchain_height_to_header[N-2]` ‚Üí mainchain block with timestamp T‚ÇÄ.

2. Attacker calls `submit_blocks` with fork block F_{N-2}: `prev_block_hash` = mainchain block N‚àí3, timestamp T‚ÇÄ' where T‚ÇÄ' >> T‚ÇÄ (e.g., T‚ÇÄ + 3600). Difficulty check for F_{N-2} uses `get_header_by_height(N-3)` = mainchain block N‚àí3, which **is** the correct ancestor (fork diverges here), so F_{N-2} passes normally.

3. Attacker submits fork block F_{N-1}: `prev_block_hash` = F_{N-2}. `get_next_work_required` computes `height_first = N-2` and calls `get_header_by_height(N-2)` ‚Üí returns **mainchain block** with timestamp T‚ÇÄ, not F_{N-2} with timestamp T‚ÇÄ'.

4. Correct timespan = T‚ÇÅ' ‚àí T‚ÇÄ' (small, because T‚ÇÄ' is large) ‚Üí high required difficulty. Computed timespan = T‚ÇÅ' ‚àí T‚ÇÄ (large) ‚Üí normal/low required difficulty. F_{N-1} passes `check_pow` with normal difficulty bits.

5. Attacker repeats for F_{N}, F_{N+1}, ‚Ä¶ Each fork block uses the wrong mainchain ancestor timestamp, accepting lower difficulty than required.

6. After enough blocks, fork's `chain_work` > mainchain's `chain_work`. `reorg_chain` fires, overwriting `mainchain_tip_blockhash` with the attacker's fork tip.
