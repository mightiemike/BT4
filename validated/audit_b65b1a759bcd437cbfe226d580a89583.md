### Title
`ContractOwner` Cannot Receive ETH, Making `withdrawFromDirectDepositV1` Always Revert for Native Token Recovery — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.withdrawFromDirectDepositV1` is the sole mechanism to recover native ETH stuck in a `DirectDepositV1` (DDA) contract. When called with `token == address(0)`, it invokes `DirectDepositV1.withdrawNative()`, which attempts to push ETH to `msg.sender` (i.e., `ContractOwner`) via a low-level `.call{value: balance}("")`. Because `ContractOwner` declares no `receive()` or `fallback()` function, this ETH transfer always fails, the `require(success, ...)` in `withdrawNative()` reverts the entire call, and any native ETH stranded in a DDA is permanently unrecoverable.

---

### Finding Description

`DirectDepositV1.withdrawNative()` sends ETH to its caller using a low-level call: [1](#0-0) 

```solidity
function withdrawNative() external onlyOwner {
    uint256 balance = address(this).balance;
    (bool success, ) = msg.sender.call{value: balance}("");
    require(success, "Failed to transfer native token to owner");
}
```

The only caller of this function in the protocol is `ContractOwner.withdrawFromDirectDepositV1`: [2](#0-1) 

```solidity
if (token == address(0)) {
    uint256 preBalance = address(this).balance;
    DirectDepositV1(directDepositV1).withdrawNative();
    uint256 postBalance = address(this).balance;
    require(postBalance > preBalance, "empty");
    (bool success, ) = msg.sender.call{value: postBalance - preBalance}("");
    require(success, "xfer");
}
```

`ContractOwner` inherits only from `EIP712Upgradeable` and `OwnableUpgradeable` and defines no `receive()` or `fallback()` function. This is confirmed: the only production contracts with a `receive()` function are `DirectDepositV1` itself and a test mock. [3](#0-2) 

When `withdrawNative()` executes `msg.sender.call{value: balance}("")` with `msg.sender == ContractOwner`, the EVM rejects the ETH transfer (no payable receiver), `success` is `false`, and `require(success, "Failed to transfer native token to owner")` reverts the entire transaction.

---

### Impact Explanation

Native ETH can become stranded in a DDA via the constructor's silent-failure path: if `wrappedNative.call{value: balance}("")` fails during construction (e.g., the wrapped-native contract is paused or temporarily non-functional), the constructor emits `NativeTokenTransferFailed` and continues without reverting, leaving ETH in the DDA with no other recovery path. [4](#0-3) 

Once ETH is stranded, `withdrawFromDirectDepositV1(subaccount, address(0))` is the only designed recovery mechanism, but it unconditionally reverts. The ETH is permanently locked with no alternative extraction path.

---

### Likelihood Explanation

The trigger requires two conditions: (1) ETH ends up in a DDA (possible via the constructor's silent-failure branch, or via CREATE2 pre-funding before deployment), and (2) the owner attempts recovery. Both are realistic operational scenarios. The broken function is the protocol's own designated recovery path, so the failure is guaranteed once the conditions are met.

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept ETH forwarded by `DirectDepositV1.withdrawNative()`:

```solidity
receive() external payable {}
```

This mirrors the fix in the referenced M-31 report: the contract that needs to handle ETH must declare itself capable of receiving it.

---

### Proof of Concept

1. A DDA is deployed via `ContractOwner.createDirectDepositV1`. At construction time, `address(this).balance > 0` (e.g., ETH was pre-sent to the CREATE2 address) and `wrappedNative.call{value: balance}("")` fails — `NativeTokenTransferFailed` is emitted, ETH remains in the DDA.
2. The owner calls `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))`.
3. Internally, `DirectDepositV1.withdrawNative()` executes `ContractOwner.call{value: balance}("")`.
4. `ContractOwner` has no `receive()` function → EVM rejects the transfer → `success == false`.
5. `require(success, "Failed to transfer native token to owner")` reverts.
6. The ETH remains permanently locked in the DDA. [5](#0-4) [1](#0-0)

    function withdrawFromDirectDepositV1(bytes32 subaccount, address token)
        external
        onlyOwner
    {
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        if (token == address(0)) {
            uint256 preBalance = address(this).balance;
            DirectDepositV1(directDepositV1).withdrawNative();
            uint256 postBalance = address(this).balance;
            require(postBalance > preBalance, "empty");
            (bool success, ) = msg.sender.call{value: postBalance - preBalance}(
                ""
            );
            require(success, "xfer");
        } else {
            uint256 preBalance = IERC20Base(token).balanceOf(address(this));
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(token));
            uint256 postBalance = IERC20Base(token).balanceOf(address(this));
            require(postBalance > preBalance, "empty");
            IERC20Base(token).safeTransfer(
                msg.sender,
                postBalance - preBalance
            );
        }
    }
