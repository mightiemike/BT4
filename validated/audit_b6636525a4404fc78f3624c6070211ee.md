Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the End User, Allowing Any User to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `sender` is `msg.sender` of the `pool.swap()` call — the immediate caller, not the originating user. When swaps are routed through `MetricOmmSimpleRouter`, `sender` equals the router address. If the pool admin allowlists the router to permit router-mediated swaps for legitimate users, every address — including explicitly blocked ones — can bypass the allowlist by calling the router.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks that `sender` is in the per-pool allowlist: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router is the direct caller of `pool.swap()`: [3](#0-2) 

So `sender` received by the extension is the **router address**, not the actual end user. The extension has no visibility into who called the router. The same bypass applies to multi-hop `exactInput` calls where intermediate hops set `sender` to `address(this)` (the router itself): [4](#0-3) 

This creates an inescapable dilemma: not allowlisting the router blocks legitimate allowlisted users from using it; allowlisting the router grants every user unrestricted access regardless of their individual allowlist status.

## Impact Explanation
The `SwapAllowlistExtension` is the protocol's primary mechanism for restricting which addresses may trade against a pool. A complete bypass of this guard means any user can swap against a pool configured to be permissioned (e.g., KYC-gated, institutional-only, or whitelist-only). LP providers who deposited under the assumption that only vetted counterparties would trade against their positions are exposed to unrestricted counterparty flow. This constitutes a broken core pool functionality (access control) and an admin-boundary break where the pool admin's configured access boundary is silently voided for all router-mediated swaps by an unprivileged path.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing entry point for swaps. Any pool admin who enables `SwapAllowlistExtension` and also allowlists the router — the natural and expected configuration to allow legitimate users to use the router — is immediately vulnerable. The attacker requires no special privileges; a standard `exactInputSingle` call suffices. The condition is trivially reachable by any address.

## Recommendation
The extension must verify the actual end user, not the intermediate router. Recommended approaches:

1. **Forward real user identity via `extensionData`**: Require the router to encode `msg.sender` into `extensionData`. The extension decodes the real user from `extensionData` when `sender` is a known router, and verifies it against the allowlist.
2. **Gate by `recipient` instead of (or in addition to) `sender`**: Since `recipient` is the address that receives swap output, it is harder to spoof without economic loss to the attacker.
3. **Remove router allowlisting entirely**: Require all allowlisted users to call `pool.swap()` directly, accepting that the router cannot be used with this extension.

## Proof of Concept
```solidity
// Setup: pool with SwapAllowlistExtension; only `allowedUser` is allowlisted.
// Pool admin also allowlists the router so allowedUser can use it.
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Attacker (not in allowlist) calls the router directly.
// The extension sees sender = address(router), which IS allowlisted → swap succeeds.
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: attacker,
    tokenIn: token1,
    zeroForOne: false,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: type(uint128).max,
    deadline: block.timestamp,
    extensionData: ""
}));
// Attacker successfully swaps despite being explicitly excluded from the allowlist.
```

  function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
      );
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
