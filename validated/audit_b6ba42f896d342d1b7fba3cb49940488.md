Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the end user, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the router becomes `msg.sender` of that call. A pool admin who allowlists the router to enable router-mediated swaps inadvertently opens the allowlist to every user on the network, because the extension checks `allowedSwapper[pool][router]` — true — rather than the originating user.

## Finding Description

**Step 1:** `MetricOmmPool.swap` passes `msg.sender` (the direct caller of `swap`) as the `sender` argument to `_beforeSwap`: [1](#0-0) 

**Step 2:** `ExtensionCalling._beforeSwap` forwards that `sender` value unchanged to every configured extension: [2](#0-1) 

**Step 3:** `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

**Step 4:** `MetricOmmSimpleRouter.exactInputSingle` (and all other router entry points) calls `pool.swap()` directly, making the router the `msg.sender` of that call — not the end user: [4](#0-3) 

The router stores the originating user only in transient storage as `payer` for the callback payment path, but never exposes it to the pool or extension as the swap identity. There is no mechanism by which the extension can distinguish which end user initiated the router call.

The result is a binary, mutually exclusive admin choice: either do not allowlist the router (breaking UX for all approved users who want to use the standard periphery), or allowlist the router (collapsing the per-user gate to a per-contract gate, allowing any EOA to bypass the restriction).

## Impact Explanation

A pool admin deploying a restricted pool (e.g., for KYC'd or institutional counterparties) configures `SwapAllowlistExtension` and calls `setAllowedToSwap(pool, address(router), true)` to let approved users trade through the standard periphery. From that point, any address on the network can call `router.exactInputSingle(...)` and the `beforeSwap` hook passes because `sender == router` is allowlisted. Non-approved users can execute swaps against LP positions that were priced assuming only trusted counterparties would trade, causing direct loss of LP principal through unauthorized swap execution in a pool whose access control is supposed to prevent it. This meets the "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" impact criteria.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing entry point documented and deployed by the protocol. Any pool admin who wants their allowlisted users to have a normal trading UX will allowlist the router — this is the natural and expected configuration step. The bypass requires no special privilege: any EOA or contract can call the router. The trigger is a routine admin configuration action, not an exotic attack setup, making exploitation highly likely once a restricted pool is deployed with router access enabled.

## Recommendation

Pass the economically relevant actor — the end user — through the hook chain. The cleanest fix is for the router to write the originating `msg.sender` into transient storage before calling the pool (it already stores `payer` there for the callback), and for the pool to read and forward that value as `sender` to extensions when the direct caller is a known router. Alternatively, the pool could accept an explicit `swapper` parameter separate from `msg.sender` that the router populates with `msg.sender` before calling the pool, and the extension gates on that value. A fragile but simpler workaround is for `SwapAllowlistExtension` to check `recipient` when `sender` is a known router, but this is easily circumvented.

## Proof of Concept

```solidity
// Pool admin sets up a restricted pool and allowlists the router
// (natural action to let approved users trade via standard periphery)
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// alice is NOT individually allowlisted
// alice calls the router — beforeSwap sees sender=router, which IS allowlisted → passes
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: token0,
    tokenOut: token1,
    zeroForOne: true,
    amountIn: 1_000e18,
    amountOutMinimum: 0,
    recipient: alice,
    deadline: block.timestamp + 1,
    priceLimitX64: 0,
    extensionData: ""
}));
// alice successfully swaps in a pool she was never allowlisted for
// The check at SwapAllowlistExtension.sol:37 evaluates allowedSwapper[pool][router] == true
```

The existing unit tests in `SwapAllowlistExtensionTest` only test direct pool calls (where `msg.sender` of the pool call is the actual swapper) and do not cover the router-mediated path, confirming the bypass is untested and present in production code. [5](#0-4)
