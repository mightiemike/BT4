### Title
`ValidResourceBounds` Protobuf Round-Trip Collapses `AllResources` to `L1Gas`, Producing Wrong Transaction Hash on Syncing Nodes — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` silently downgrades an `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` variant to `L1Gas(X)`. Because `get_tip_resource_bounds_hash` hashes a different number of field elements for each variant, the transaction hash computed at the originating node (4-element Poseidon input) differs from the hash recomputed at any syncing node (3-element Poseidon input). A valid V3 invoke transaction submitted via RPC with zero `l2_gas` and `l1_data_gas` bounds therefore carries a hash that no peer can reproduce, and the `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` conversion hard-errors on the `L1Gas` variant, making the transaction unservable via RPC on every synced node.

### Finding Description

**Step 1 — Hash domain split in `get_tip_resource_bounds_hash`**

`crates/starknet_api/src/transaction_hash.rs` lines 203–208 conditionally append `l1_data_gas` to the Poseidon input only for the `AllResources` variant:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 3-element hash
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 4-element hash
    }
});
```

Even when `l1_data_gas = 0`, `get_concat_resource` encodes the `L1_DATA_GAS` resource-name bytes into the felt, so the 4-element hash is never equal to the 3-element hash.

**Step 2 — Originating node always uses `AllResources`**

`InternalRpcInvokeTransactionV3::resource_bounds()` (line 637–639 of `crates/starknet_api/src/rpc_transaction.rs`) unconditionally wraps the stored `AllResourceBounds` as `ValidResourceBounds::AllResources`:

```rust
fn resource_bounds(&self) -> ValidResourceBounds {
    ValidResourceBounds::AllResources(self.resource_bounds)
}
```

The `From<InternalRpcInvokeTransactionV3> for InvokeTransactionV3` conversion (lines 679–694) preserves this, so the on-disk `InvokeTransactionV3` also carries `AllResources`.

**Step 3 — Protobuf round-trip silently downgrades to `L1Gas`**

`crates/apollo_protobuf/src/converters/transaction.rs` lines 431–435:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong variant for new V3 txs
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

A transaction with `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` serialises to protobuf with those zero fields and deserialises as `L1Gas(X)`. There is no version marker in the wire format to distinguish a pre-0.13.3 `L1Gas` transaction from a 0.13.3+ `AllResources` transaction whose optional bounds happen to be zero.

**Step 4 — Hash mismatch and hard conversion failure**

| Node | `ValidResourceBounds` variant | Poseidon input length | Hash |
|---|---|---|---|
| Originating (RPC path) | `AllResources` | 4 elements | H1 |
| Syncing (protobuf path) | `L1Gas` | 3 elements | H2 ≠ H1 |

Additionally, `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` (lines 586–611 of `crates/starknet_api/src/rpc_transaction.rs`) returns `Err` for the `L1Gas` variant:

```rust
resource_bounds: match value.resource_bounds {
    ValidResourceBounds::AllResources(bounds) => bounds,
    _ => return Err(StarknetApiError::OutOfRange { string: "resource_bounds".to_string() }),
},
```

Every syncing node that attempts to serve such a transaction via RPC will hit this error.

### Impact Explanation

A syncing node that receives a block containing a V3 invoke transaction with `AllResources { l2_gas: 0, l1_data_gas: 0 }` via P2P block sync will:

1. Store the transaction with the wrong `L1Gas` variant (hash H2 ≠ H1 stored by the originating node).
2. Fail to convert the transaction to `RpcInvokeTransactionV3`, causing `starknet_getTransactionByHash`, `starknet_getTransactionReceipt`, `starknet_simulateTransactions`, and fee-estimation calls to return errors for a transaction that was validly executed and committed.
3. If hash validation is performed during sync, reject the block entirely, stalling the syncing node.

This matches **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value"** and **"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload"**.

### Likelihood Explanation

The trigger is a standard, unprivileged RPC submission: any user can send a V3 invoke transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`. The gateway accepts it (zero bounds are valid). The vulnerability fires on every syncing node that processes the block containing that transaction. No special access or malicious peer is required.

### Recommendation

1. **Preserve the `AllResources` variant unconditionally for 0.13.3+ transactions in the protobuf converter.** Add a version/era marker to the protobuf `ResourceBounds` message so the deserialiser can distinguish old `L1Gas` transactions from new `AllResources` transactions with zero optional bounds.

2. **Alternatively, make `get_tip_resource_bounds_hash` variant-agnostic** by always including all three resource fields in the Poseidon input, using zero for absent fields. This would make the hash identical for `L1Gas` and `AllResources { l2_gas: 0, l1_data_gas: 0 }`, eliminating the split.

3. **Add a round-trip hash invariant test**: serialize an `InvokeTransactionV3` with `AllResources { l2_gas: 0, l1_data_gas: 0 }` to protobuf, deserialize it, recompute the hash, and assert it equals the original hash.

### Proof of Concept

```
1. Submit via RPC:
   InvokeTransactionV3 {
     resource_bounds: AllResourceBounds { l1_gas: {max_amount:100, max_price:1}, l2_gas: {0,0}, l1_data_gas: {0,0} },
     ...
   }

2. Gateway computes hash H1 via get_tip_resource_bounds_hash(AllResources{...}):
   Poseidon([tip, pack(L1_GAS,100,1), pack(L2_GAS,0,0), pack(L1_DATA_GAS,0,0)])
   → H1

3. Transaction included in block N, stored with resource_bounds = AllResources, hash = H1.

4. Block N synced via P2P. InvokeTransactionV3 serialised to protobuf:
   ResourceBounds { l1_gas: {100,1}, l2_gas: {0,0}, l1_data_gas: {0,0} }

5. Receiving node deserialises:
   l1_data_gas.is_zero() && l2_gas.is_zero() == true
   → ValidResourceBounds::L1Gas({100,1})

6. Receiving node recomputes hash H2 via get_tip_resource_bounds_hash(L1Gas{100,1}):
   Poseidon([tip, pack(L1_GAS,100,1), pack(L2_GAS,0,0)])   // no l1_data_gas element
   → H2 ≠ H1

7. starknet_getTransactionByHash(H1) on receiving node:
   TryFrom<InvokeTransactionV3{L1Gas}> for RpcInvokeTransactionV3
   → Err(OutOfRange { "resource_bounds" })   // hard error, transaction unservable
```
