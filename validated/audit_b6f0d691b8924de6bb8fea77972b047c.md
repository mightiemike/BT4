### Title
Equihash Solution Validity Boolean Return Value Silently Discarded — Blocks with Invalid PoW Accepted — (`contract/src/zcash.rs`)

---

### Summary

In the Zcash build of the BTC light client, `equihash::is_valid_solution` returns `Result<bool, Error>`. The contract handles the `Err` variant by panicking, but **silently discards the `bool` inside `Ok`**. If the library returns `Ok(false)` — indicating an invalid Equihash solution — the contract continues without rejecting the block. This is a direct analog to M-03: just as `IERC20.transfer` can return `false` instead of reverting, `equihash::is_valid_solution` can return `Ok(false)` instead of `Err`, and in both cases the caller ignores the failure signal.

---

### Finding Description

In `contract/src/zcash.rs`, the `check_pow` function validates the Equihash proof-of-work solution as follows:

```rust
equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
``` [1](#0-0) 

`equihash::is_valid_solution` has the signature `fn is_valid_solution(...) -> Result<bool, Error>`. The code correctly panics on `Err(e)`, but the `Ok(bool)` value is **never bound and never checked**. Rust's `unwrap_or_else` on a `Result<bool, E>` returns the `bool` on `Ok`; here that `bool` is immediately dropped. If the library returns `Ok(false)` — the canonical way to signal "solution is structurally parseable but cryptographically invalid" — the `require!` gate is never reached and the block passes PoW verification.

The call chain is:

`submit_blocks` → `submit_block_header` → `check_target` → `check_pow` → `equihash::is_valid_solution` [2](#0-1) [3](#0-2) 

---

### Impact Explanation

A block header carrying an invalid Equihash solution (one that parses without error but fails the cryptographic validity check, causing `Ok(false)`) is accepted by `submit_block_header_inner` and stored in `headers_pool`. If its accumulated `chain_work` exceeds the current mainchain tip, `reorg_chain` is triggered and the fraudulent block becomes the canonical tip. Downstream callers of `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` then operate against a mainchain that includes a block whose PoW was never actually verified, breaking the fundamental security guarantee of the light client. [4](#0-3) 

---

### Likelihood Explanation

The `equihash` crate's `is_valid_solution` is documented to return `Ok(false)` for solutions that are well-formed but fail the Equihash constraint check (e.g., duplicate indices, wrong XOR sum). An attacker who can submit headers — either as a trusted relayer or via the `UnrestrictedSubmitBlocks` bypass role — can craft a Zcash header with a nonce and solution that parse without error but fail the cryptographic check, reliably triggering `Ok(false)`. The `skip_pow_verification` flag is `false` in production, so the path through `check_pow` is always taken for live deployments. [5](#0-4) [6](#0-5) 

---

### Recommendation

Bind the returned `bool` and assert it with `require!`:

```rust
let is_valid = equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
require!(is_valid, "Invalid Equihash solution");
```

This mirrors the fix recommended in M-03: explicitly check the return value rather than relying solely on the error path.

---

### Proof of Concept

1. Construct a Zcash block header whose `solution` field contains well-formed bytes (correct length, parseable indices) but whose XOR sum of selected values does not equal zero — the canonical `Ok(false)` case for the `equihash` crate.
2. Set `bits` and `time` to pass the difficulty and timestamp checks.
3. Call `submit_blocks([crafted_header])` on the Zcash-compiled contract.
4. Observe that `check_pow` returns without panicking despite the invalid solution.
5. Confirm via `get_last_block_header` that the fraudulent block is now the mainchain tip. [7](#0-6) [8](#0-7)

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }

    fn submit_block_header_inner(
        &mut self,
        current_header: ExtendedHeader,
        prev_block_header: &ExtendedHeader,
    ) {
        // Main chain submission
        if prev_block_header.block_hash == self.mainchain_tip_blockhash {
            // Probably we should check if it is not in a mainchain?
            // chainwork > highScore
            log!("Block {}: saving to mainchain", current_header.block_hash);
            // Validate chain
            assert_eq!(
                self.mainchain_tip_blockhash,
                current_header.block_header.prev_block_hash
            );

            self.store_block_header(&current_header);
            self.mainchain_tip_blockhash = current_header.block_hash;
        } else {
            log!("Block {}: saving to fork", current_header.block_hash);
            // Fork submission
            let main_chain_tip_header = self
                .headers_pool
                .get(&self.mainchain_tip_blockhash)
                .unwrap_or_else(|| env::panic_str("tip should be in a header pool"));

            let last_main_chain_block_height = main_chain_tip_header.block_height;
            let total_main_chain_chainwork = main_chain_tip_header.chain_work;

            self.store_fork_header(&current_header);

            // Current chainwork is higher than on a current mainchain, let's promote the fork
            if current_header.chain_work > total_main_chain_chainwork {
                log!("Chain reorg");
                self.reorg_chain(current_header, last_main_chain_block_height);
            }
        }
    }
