### Title
`ValidResourceBounds::AllResources` with zero L2/L1DataGas serializes to protobuf and deserializes back as `ValidResourceBounds::L1Gas`, producing a divergent transaction hash - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

A V3 transaction submitted via RPC with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` has its hash computed over **four** poseidon elements (tip + L1_GAS + L2_GAS + L1_DATA_GAS). After the transaction is serialized to protobuf and deserialized by a syncing peer, the resource bounds are reconstructed as `ValidResourceBounds::L1Gas(X)`, which causes any subsequent hash computation to use only **three** poseidon elements (tip + L1_GAS + L2_GAS), producing a different hash for the same transaction.

### Finding Description

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` branches on the `ValidResourceBounds` variant:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 felts total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 felts total
    }
});
``` [1](#0-0) 

`AllResources` always appends the `L1_DATA_GAS` felt, even when it is zero. `L1Gas` never appends it. Because poseidon is length-sensitive, `poseidon(tip, L1, L2, 0)` ≠ `poseidon(tip, L1, L2)`.

The RPC layer forces every incoming V3 transaction into `AllResourceBounds`:

```rust
pub struct RpcInvokeTransactionV3 {
    pub resource_bounds: AllResourceBounds,
    ...
}
``` [2](#0-1) 

`InternalRpcInvokeTransactionV3` also stores `AllResourceBounds`, and its `InvokeTransactionV3Trait` impl wraps it as `ValidResourceBounds::AllResources`:

```rust
fn resource_bounds(&self) -> ValidResourceBounds {
    ValidResourceBounds::AllResources(self.resource_bounds)
}
``` [3](#0-2) 

So the gateway computes hash H₁ using the `AllResources` path (4-element poseidon).

On the P2P block-sync path, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies the following collapse:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← collapses AllResources → L1Gas
} else {
    ValidResourceBounds::AllResources(...)
})
``` [4](#0-3) 

When the serializer encodes `ValidResourceBounds::L1Gas`, it explicitly sets `l1_data_gas: Some(ResourceBounds::default().into())` (zero):

```rust
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),
    l1_data_gas: Some(ResourceBounds::default().into()),   // ← zero
},
``` [5](#0-4) 

A transaction originally stored as `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` round-trips through protobuf as `L1Gas(X)`. Any call to `calculate_transaction_hash` on the deserialized form produces hash H₂ ≠ H₁.

`validate_transaction_hash` in `crates/starknet_api/src/transaction_hash.rs` recomputes the hash from the deserialized `Transaction` struct and checks it against the committed value:

```rust
possible_hashes.push(get_transaction_hash(transaction, chain_id, transaction_options)?);
Ok(possible_hashes.contains(&expected_hash))
``` [6](#0-5) 

A syncing node that calls this function on a protobuf-deserialized transaction will compute H₂ and fail to match the committed H₁, causing the block to be rejected or the stored hash to diverge from the canonical value.

### Impact Explanation

The canonicalization invariant is broken: the same transaction fields produce two distinct hashes depending on whether the transaction was received via RPC (AllResources path) or via P2P block sync (L1Gas path after protobuf round-trip). Any component that recomputes the hash from the deserialized transaction — including `validate_transaction_hash` during block sync — will bind the wrong hash to the transaction. This matches the **High** impact: "Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."

### Likelihood Explanation

The gateway's stateless validator explicitly accepts transactions with only `l1_gas` non-zero and both `l2_gas` and `l1_data_gas` at zero (test case `valid_l1_gas` in `crates/apollo_gateway/src/stateless_transaction_validator_test.rs`). Any user submitting such a transaction triggers the divergence. The TODO comment in the protobuf converter acknowledges the issue is known but deferred. [7](#0-6) 

### Recommendation

In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, do not collapse `AllResources` with zero L2/L1DataGas into `L1Gas`. The `L1Gas` variant should only be produced when the protobuf message was explicitly sent without an `l1_data_gas` field (i.e., `value.l1_data_gas.is_none()`), not when it is present but zero. Alternatively, enforce that `get_tip_resource_bounds_hash` always hashes exactly three resource felts for all V3 transactions regardless of variant, making the hash independent of the `ValidResourceBounds` discriminant.

### Proof of Concept

1. Submit a V3 invoke transaction via RPC with `resource_bounds = { L1_GAS: { max_amount: 1000, max_price_per_unit: 1 }, L2_GAS: { 0, 0 }, L1_DATA_GAS: { 0, 0 } }`.
2. The gateway converts this to `InternalRpcInvokeTransactionV3` with `resource_bounds: AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`.
3. `calculate_transaction_hash` calls `get_tip_resource_bounds_hash` with `ValidResourceBounds::AllResources(...)` → poseidon over `[tip, L1_GAS_felt, L2_GAS_felt(0), L1_DATA_GAS_felt(0)]` → hash H₁.
4. The transaction is included in a block and propagated via P2P.
5. A syncing node deserializes the transaction from protobuf: `l1_data_gas.is_zero() && l2_gas.is_zero()` → `ValidResourceBounds::L1Gas(X)`.
6. `calculate_transaction_hash` on the deserialized form calls `get_tip_resource_bounds_hash` with `ValidResourceBounds::L1Gas(X)` → poseidon over `[tip, L1_GAS_felt, L2_GAS_felt(0)]` → hash H₂.
7. H₁ ≠ H₂. `validate_transaction_hash` returns `false`; the syncing node rejects the block or stores the transaction under the wrong hash.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
