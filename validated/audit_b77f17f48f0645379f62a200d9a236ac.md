Audit Report

## Title
Swap allowlist gates the router address instead of the originating user, enabling full bypass via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` as the first argument and checks `allowedSwapper[msg.sender][sender]`. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, so `sender` passed to the extension is the router address — not the originating user. A pool admin who allowlists the router to permit router-mediated swaps for their curated users inadvertently grants every unprivileged user the ability to bypass the allowlist by calling through the router.

## Finding Description
**Exact code path:**

`MetricOmmSimpleRouter.exactInputSingle` (line 71–80) sets callback context and calls:
```solidity
IMetricOmmPoolActions(params.pool).swap(params.recipient, params.zeroForOne, ...)
```
Here `msg.sender` to the pool is the router contract.

`MetricOmmPool.swap` (line 230–240) calls:
```solidity
_beforeSwap(msg.sender, recipient, ...)
```
So `sender` = router address.

`ExtensionCalling._beforeSwap` (line 160–176) encodes and dispatches:
```solidity
abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
```
with `sender` = router address.

`SwapAllowlistExtension.beforeSwap` (line 37) checks:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` = pool (correct key), `sender` = **router** (wrong actor). The check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][originalUser]`.

**Root cause:** The pool passes its own `msg.sender` (the immediate caller) as `sender` to the extension. When the router intermediates, the original user's address is never forwarded to the extension.

**Exploit flow:**
1. Pool is deployed with `SwapAllowlistExtension` configured.
2. Pool admin allowlists the router address (`setAllowedToSwap(pool, router, true)`) so that allowlisted users can use the standard router UX.
3. Attacker (not individually allowlisted) calls `router.exactInputSingle(...)`.
4. Router calls `pool.swap(...)` with `msg.sender` = router.
5. Extension checks `allowedSwapper[pool][router]` = `true` → passes.
6. Attacker's swap executes on the curated pool, bypassing the allowlist entirely.

**Existing guards are insufficient:** The `onlyPool` modifier on the extension (via `BaseMetricExtension`) only verifies that the caller of `beforeSwap` is a registered pool — it does not verify the identity of the original user. No guard in the router, pool, or extension cross-checks the original `msg.sender` of the router call against the allowlist.

## Impact Explanation
Complete bypass of the swap allowlist on curated pools. Any unprivileged user can trade on a pool that the admin intended to restrict to specific addresses, by routing through `MetricOmmSimpleRouter`. This constitutes a broken core pool functionality (allowlist policy) and an admin-boundary break where an unprivileged path circumvents the configured access control. Direct financial impact: disallowed users can execute swaps against pool liquidity, extracting value from LP positions in pools designed to be curated.

## Likelihood Explanation
Exploitable by any unprivileged user with no special capability beyond calling the public router. The only precondition is that the pool admin has allowlisted the router address — a necessary step if any allowlisted user is expected to use the standard router UX. The attack is repeatable every block with zero cost beyond gas.

## Recommendation
The extension must check the original user, not the intermediate caller. Options:
1. Pass the original `msg.sender` of the router call through `callbackData` or a dedicated field, and have the pool forward it as a separate `originator` argument to the extension.
2. Alternatively, require that allowlisted pools are only accessed directly (no router intermediation), and enforce this at the pool level by checking `msg.sender` is not a known router.
3. The cleanest fix: add an `originator` field to the swap call that the router populates with `msg.sender`, and have `SwapAllowlistExtension` check `originator` instead of `sender`.
