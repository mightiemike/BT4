### Title
Raw `approve` in `DirectDepositV1.creditDeposit()` Permanently Bricks Deposits for Non-Standard ERC20 Tokens — (`File: core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` via the `IIERC20Base` interface, which declares `approve` as returning `bool`. Tokens such as USDT on Ethereum mainnet return no data from `approve`, causing the Solidity ABI decoder to revert. Because `creditDeposit()` is permissionlessly callable and iterates over all registered spot products, any non-standard token listed as a spot product permanently prevents the function from executing, locking all funds already held in the DirectDepositV1 contract.

---

### Finding Description

`DirectDepositV1.creditDeposit()` iterates over every spot product token and, for each non-zero balance, calls:

```solidity
token.approve(address(endpoint), balance);
``` [1](#0-0) 

The `IIERC20Base` interface declares `approve` as returning `bool`:

```solidity
function approve(address spender, uint256 amount) external returns (bool);
``` [2](#0-1) 

When Solidity calls a function declared to return a value, it ABI-decodes the return data. Tokens like USDT return zero bytes from `approve`, causing the decoder to revert. The entire `creditDeposit()` loop reverts, not just the single token iteration.

The rest of the Nado codebase correctly avoids this by routing all transfers through `ERC20Helper.safeTransfer` / `ERC20Helper.safeTransferFrom`, which use low-level `.call()` and explicitly handle the empty-return case:

```solidity
(bool success, bytes memory data) = address(self).call(
    abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
);
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [3](#0-2) 

`DirectDepositV1` even implements its own `safeTransfer` wrapper using the same low-level pattern for the `withdraw()` path, but omits the equivalent protection for `approve`: [4](#0-3) 

A secondary instance of the same class exists in `ContractOwner.wrapVaultAsset()`, which calls raw `assetToken.approve(tokenAddr, 0)` and `assetToken.approve(tokenAddr, assetBalance)` on an `IERC20Base` (also declared to return `bool`) with no access control: [5](#0-4) 

---

### Impact Explanation

If any non-standard ERC20 token (e.g., USDT) is registered as a spot product in the `SpotEngine`, every call to `creditDeposit()` reverts. Funds already transferred into the DirectDepositV1 contract for that token — or for any other token in the same loop — cannot be credited to the target subaccount and are permanently stuck in the DDA contract. The `withdraw()` function (owner-only) can recover them, but the deposit credit path is completely broken for all tokens as long as the non-standard token remains listed.

---

### Likelihood Explanation

Medium. USDT is one of the most commonly used collateral tokens in DeFi protocols. If Nado lists USDT as a spot product on Ethereum mainnet, the condition is immediately triggered. `creditDeposit()` is permissionlessly callable by any address, so no privileged action is required to expose the bug — it manifests on the first call after a non-standard token is listed.

---

### Recommendation

Replace the raw `approve` call in `creditDeposit()` with a low-level safe-approve wrapper analogous to the existing `safeTransfer` in `DirectDepositV1`:

```solidity
// In DirectDepositV1
function safeApprove(IIERC20Base self, address spender, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IIERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        "Approve failed"
    );
}
```

Then replace:
```solidity
- token.approve(address(endpoint), balance);
+ safeApprove(token, address(endpoint), balance);
```

Apply the same fix to `ContractOwner.wrapVaultAsset()` for both `assetToken.approve` calls.

---

### Proof of Concept

1. USDT is listed as a spot product in `SpotEngine` (token address set in config).
2. A user sends USDT to a DirectDepositV1 contract address.
3. Any caller invokes `creditDeposit()`.
4. The loop reaches the USDT product; `token.approve(address(endpoint), balance)` is called.
5. USDT's `approve` returns no data; Solidity's ABI decoder reverts attempting to decode `bool` from empty bytes.
6. The entire transaction reverts; the USDT balance remains stuck in the DDA contract.
7. All other tokens in the loop are also blocked from being credited in the same call. [1](#0-0)
