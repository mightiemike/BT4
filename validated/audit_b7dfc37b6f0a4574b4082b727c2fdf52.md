### Title
Unprivileged Caller Can Aggressively Prune Mainchain Headers via `run_mainchain_gc`, Causing Transaction Inclusion Proofs to Fail Prematurely — (`contract/src/lib.rs`)

### Summary

`BtcLightClient::run_mainchain_gc` is a public, state-mutating function with no caller access control when the contract is not paused. Any NEAR account can call it with an arbitrarily large `batch_size`, immediately removing every GC-eligible mainchain block header in a single transaction. This allows an attacker to cause `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` to panic for blocks that are currently stored but above the GC threshold, invalidating proofs that would otherwise succeed.

### Finding Description

`run_mainchain_gc` is decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`. [1](#0-0) 

The `#[pause]` decorator blocks the call only when the contract is **paused**. When the contract is running normally, the decorator imposes no restriction — any NEAR account can call `run_mainchain_gc` with any `batch_size`. There is no `#[trusted_relayer]`, no role check, and no `#[private]` guard.

The internal logic removes up to `min(amount_of_headers_we_store − gc_threshold, batch_size)` of the oldest mainchain blocks: [2](#0-1) 

By passing `batch_size = u64::MAX`, an attacker removes every GC-eligible block in one call — the same total that normal operation would remove gradually (one batch per `submit_blocks` invocation).

`verify_transaction_inclusion` looks up the target block in `mainchain_header_to_height`: [3](#0-2) 

`remove_block_header` deletes the entry from both `mainchain_header_to_height` and `headers_pool`: [4](#0-3) 

Once a block is GC'd, any proof referencing it panics with `"block does not belong to the current main chain"`.

**Attack path:**

1. The chain has `gc_threshold + N` blocks stored (N > 0). Blocks at heights `[initial, initial + N)` are GC-eligible.
2. A user or downstream DApp is about to call `verify_transaction_inclusion` for a transaction in one of those N blocks.
3. The attacker calls `run_mainchain_gc(u64::MAX)`. All N blocks are deleted immediately.
4. The user's proof call panics. The attacker can repeat this every time new blocks push the chain above the threshold.

The CLAUDE.md documentation also notes a secondary impact: aggressive GC near a fork point causes chain reorgs to panic with `PrevBlockNotFound`: [5](#0-4) 

### Impact Explanation

External NEAR DApps that rely on `verify_transaction_inclusion_v2` for trustless Bitcoin transaction verification receive false-negative panics for blocks that are legitimately within the stored window. An attacker can selectively target specific blocks (e.g., blocks containing a transaction a DApp is about to verify) and remove them just before the proof call, causing the verification to fail. This breaks the core SPV guarantee the contract provides. The attack requires no privileged keys, no stake, and costs only standard NEAR gas.

### Likelihood Explanation

The entry point is a plain public NEAR contract call with a single `u64` argument. Any account can execute it at any time the contract is not paused. An attacker monitoring the NEAR mempool can front-run a `verify_transaction_inclusion` call with `run_mainchain_gc(u64::MAX)` to guarantee the proof fails. The attack is repeatable indefinitely as long as the chain grows beyond `gc_threshold`.

### Recommendation

Add an explicit role check to `run_mainchain_gc` so that only authorized accounts (e.g., holders of `Role::DAO` or a new `GCManager` role) can invoke it directly. The internal call from `submit_blocks` can bypass this check via a private helper. The `#[pause]` decorator alone is insufficient because it only restricts access when the contract is paused, not during normal operation.

### Proof of Concept

```rust
// Any unprivileged NEAR account can execute:
let outcome = attacker_account
    .call(contract.id(), "run_mainchain_gc")
    .args_json(json!({ "batch_size": u64::MAX }))
    .max_gas()
    .transact()
    .await?;
assert!(outcome.is_success()); // succeeds with no role check

// Subsequent proof call for a GC'd block now panics:
let proof_outcome = user_account
    .call(contract.id(), "verify_transaction_inclusion_v2")
    .args_borsh(proof_args_for_gc_eligible_block)
    .transact()
    .await?;
assert!(proof_outcome.is_failure()); // "block does not belong to the current main chain"
```
