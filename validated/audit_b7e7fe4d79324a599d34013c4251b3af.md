### Title
Unsigned `ProposalFin.proposal_commitment` Allows Transaction Substitution in Consensus Proposal Stream — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The `ProposalFin.proposal_commitment` is transmitted over the P2P network without a proposer signature. A man-in-the-middle relay node can substitute transactions in a proposal stream, compute a matching `proposal_commitment` for the substituted payload, and cause all receiving validators to execute a different block than what the proposer built. The codebase itself acknowledges the gap with two TODO comments that have not been resolved.

---

### Finding Description

**Proposal flow (proposer side)**

The proposer builds a block, streams `ProposalPart::Transactions` batches, then closes with `ProposalPart::Fin` carrying a `proposal_commitment`. [1](#0-0) 

The `proposal_commitment` is derived from the batcher's `PartialBlockHash` and the `fee_proposal_fri` from `ProposalInit`. [2](#0-1) 

**Validation side**

The validator converts each received `ConsensusTransaction` to `InternalConsensusTransaction` by recomputing the hash from the transaction body using its own `chain_id`. [3](#0-2) 

The recomputed hash is never compared against any proposer-signed value; it is simply used as the identity of the transaction going forward. [4](#0-3) 

After executing all transactions, the validator compares the batcher's locally-computed `batcher_block_commitment` against the wire value `received_fin.proposal_commitment`:

```rust
// TODO(matan): Switch to signature validation.
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}
``` [5](#0-4) 

**The missing invariant**

Neither `ProposalInit` nor `ProposalFin` carries a cryptographic signature from the proposer. The `ProposalFin` definition itself carries the acknowledgement:

```rust
/// The commitment identifying the proposed block.
/// TODO(Matan): Consider changing the content ID to a signature.
pub proposal_commitment: ProposalCommitment,
``` [6](#0-5) 

The protobuf `ConsensusTransaction` message has a `transaction_hash` field (tag 5), but the Rust serializer always sets it to `None` and the deserializer silently drops it. [7](#0-6) 

This means the entire proposal stream — transactions and the closing commitment — is unauthenticated by the proposer.

**Attack path (direct analog to Connext M-09)**

| Connext M-09 | Sequencer analog |
|---|---|
| Relayer swaps `_args.routers[]` with own routers | Relay node swaps `TransactionBatch` with own transactions |
| Relayer re-signs `routerHash` with own key | Relay node re-executes substituted txs to compute new `partial_block_hash` |
| Bridge checks router signed the hash — passes | Validator checks batcher hash == `proposal_commitment` — passes |
| Sequencer's chosen router loses fee opportunity | Proposer's chosen transactions are replaced |

Concrete steps:

1. Attacker operates a relay node in the libp2p network between proposer and validators.
2. Attacker intercepts `ProposalPart::Transactions` batches and replaces them with attacker-chosen transactions (e.g., front-running, censorship, or fee extraction).
3. Attacker runs a local batcher to execute the substituted transactions and obtain `partial_block_hash'`.
4. Attacker computes `proposal_commitment' = proposal_commitment_from(partial_block_hash', fee_proposal_fri)`. [2](#0-1) 
5. Attacker replaces `ProposalFin.proposal_commitment` with `proposal_commitment'` and forwards the modified stream.
6. Each validator executes the substituted transactions, computes the same `batcher_block_commitment = proposal_commitment'`, and the check at line 244 passes.
7. Validators prevote for `proposal_commitment'`; the proposer prevotes for the original `proposal_commitment`. If the attacker reaches a 2/3 quorum of validators, the substituted block is committed.

---

### Impact Explanation

If the attacker controls the network path to a 2/3 quorum of validators, the substituted block is committed — wrong state, wrong receipts, wrong events, wrong L1 messages. This matches:

> **Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.**

If the attacker reaches fewer than 2/3 of validators, the proposer and the manipulated validators vote for different commitments, causing consensus to stall — matching:

> **High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

---

### Likelihood Explanation

The attack requires a man-in-the-middle position in the libp2p network. libp2p authenticates direct peer connections (noise/TLS) but does not authenticate the *content* of relayed messages. Any legitimate relay node that sits on the path between the proposer and a quorum of validators can perform this substitution without any cryptographic privilege. The two TODO comments in production code confirm the developers have identified the gap but have not yet closed it.

---

### Recommendation

Sign `ProposalFin.proposal_commitment` with the proposer's private key before broadcasting, and verify the signature against the `proposer` address declared in `ProposalInit` before the batcher comparison at line 244. The `SignatureManager` already provides `sign_precommit_vote` / `verify_precommit_vote_signature` over a `BlockHash`; the same pattern applies directly to `ProposalCommitment`. [8](#0-7) 

---

### Proof of Concept

```
Proposer P builds block B with txs [T1, T2, T3]:
  partial_block_hash  = H(execute(T1,T2,T3))
  proposal_commitment = proposal_commitment_from(partial_block_hash, fee_fri)
  Sends: Init | Txs([T1,T2,T3]) | Fin(proposal_commitment)

Relay node R intercepts and substitutes:
  Executes [T1', T2', T3'] locally
  partial_block_hash' = H(execute(T1',T2',T3'))
  proposal_commitment'= proposal_commitment_from(partial_block_hash', fee_fri)
  Forwards: Init | Txs([T1',T2',T3']) | Fin(proposal_commitment')

Validator V receives modified stream:
  Executes [T1',T2',T3'] via batcher
  batcher_block_commitment = proposal_commitment'
  Check: batcher_block_commitment == received_fin.proposal_commitment  ✓ PASS
  V prevotes for proposal_commitment'

P prevotes for original proposal_commitment  ← disagrees with V
If R reaches ≥2/3 validators: substituted block B' is committed
``` [5](#0-4) [1](#0-0)
