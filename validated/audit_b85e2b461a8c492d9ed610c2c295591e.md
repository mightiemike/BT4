### Title
Deterministic "Random" Parent Eviction in `NodeEntry::add_parent` Enables Attacker-Controlled Compression Failure — (File: src/serde/tree_cache.rs)

---

### Summary

`NodeEntry::add_parent` in `TreeCache` caps the parent list at `MAX_PARENTS = 8` and, when full, claims to evict a "random" parent. The eviction index is computed as `(parent as usize + pos as usize) % MAX_PARENTS` — a fully deterministic function of attacker-controlled values. An attacker who controls CLVM bytes can craft a tree that causes specific parent entries to be systematically overwritten, causing `find_path()` to miss valid back-reference paths during compressed serialization, producing non-canonical (larger-than-optimal or size-limit-exceeding) serialized output.

---

### Finding Description

In `src/serde/tree_cache.rs`, `NodeEntry::add_parent` enforces a hard cap of `MAX_PARENTS = 8` on the parent list:

```rust
fn add_parent(&mut self, parent: u32, pos: ChildPos) {
    if self.parents.len() >= MAX_PARENTS {
        // evict a "random" parent
        let idx = parent as usize + pos as usize;
        self.parents[idx % MAX_PARENTS] = (parent, pos);
    } else {
        self.parents.push((parent, pos));
    }
}
```

The comment says "evict a 'random' parent" but the eviction slot is `(parent as usize + pos as usize) % MAX_PARENTS`, where:

- `parent` is the `NodeEntry` index of the new parent — determined by the order of tree traversal, which is fully controlled by the CLVM bytes.
- `pos` is `ChildPos::Left` (0) or `ChildPos::Right` (1) — also determined by the tree structure.

There is no randomness. An attacker who controls the CLVM bytes can craft a tree where a specific atom or subtree appears as a child of many pairs, with parent NodeEntry indices chosen so that `(parent_idx + pos) % 8` always resolves to the same slot. This causes the same slot to be overwritten on every call, leaving only one surviving parent in the list regardless of how many true parents exist.

The same broken invariant appears at the root-parent truncation path in `update()`:

```rust
if root_entry.parents.len() > MAX_PARENTS {
    let num_drop = root_entry.parents.len() - MAX_PARENTS;
    root_entry.parents.drain(0..num_drop);
}
```

This always drops the *oldest* parents (index 0..n), which is also deterministic and attacker-exploitable.

`find_path()` performs a BFS from the target node upward through its `parents` list to reach the top of the parse stack. If a parent is evicted, that branch of the search is never explored. `find_path()` returns `None` even though a valid (and possibly shorter) path exists through the evicted parent.

---

### Impact Explanation

`find_path()` is called in two production serialization paths:

1. `node_to_stream_backrefs()` in `src/serde/ser_br.rs` — the standard compressed serializer.
2. `Serializer::add()` in `src/serde/incremental.rs` — the incremental compressed serializer used for puzzle reveals and solutions.

When `find_path()` returns `None` due to a missing parent, the serializer emits the full subtree instead of a back-reference (`0xfe` + path). This causes:

- **Non-canonical serialization**: Two logically identical programs serialize to different byte sequences depending on tree structure, breaking the invariant that compressed serialization is deterministic for a given logical tree.
- **Size-limit violation**: Chia enforces maximum serialized sizes on puzzle reveals and solutions. A program that would fit within the limit with correct compression may exceed it when parents are evicted, causing the transaction to be rejected by the mempool even though the program is logically valid.
- **Attacker-controlled entry path**: The attacker submits crafted CLVM bytes (as a puzzle or solution) that trigger the deterministic eviction pattern, causing the node operator's serializer to produce bloated output.

---

### Likelihood Explanation

The eviction formula `(parent as usize + pos as usize) % MAX_PARENTS` is simple and reversible. An attacker can compute, for any target node, which parent indices will collide modulo 8 and craft a tree where all 9+ parents hash to the same slot. The `NodeEntry` index is assigned sequentially during `update()` traversal (post-order), so the attacker can predict it by controlling the tree shape. This requires only crafting a CLVM program with a repeated subtree — a common and unrestricted pattern in Chia puzzles.

---

### Recommendation

Replace the deterministic eviction with true randomness. The `TreeCache` already holds a `salt: [u8; 8]` field initialized from `rand::rng()` in `new()`. Use it (or a separate `SmallRng` seeded from it) to pick the eviction slot:

```rust
fn add_parent(&mut self, parent: u32, pos: ChildPos, rng: &mut impl Rng) {
    if self.parents.len() >= MAX_PARENTS {
        let idx = rng.random_range(0..MAX_PARENTS);
        self.parents[idx] = (parent, pos);
    } else {
        self.parents.push((parent, pos));
    }
}
```

Similarly, replace the `drain(0..num_drop)` truncation in `update()` with a random-index removal.

---

### Proof of Concept

Construct a CLVM atom `A` (e.g., `b"foobar"`, 6 bytes, serialized length ≥ 4 so it qualifies for back-references). Place `A` as the right child of 9 pairs `P0..P8`, where the pairs are arranged so that during `update()` traversal, the `NodeEntry` indices assigned to `P0..P8` satisfy:

```
(idx(P_i) + 1) % 8 == k   for all i >= 8
```

(i.e., all new parents after the 8th hash to slot `k`). This is achievable by padding the tree with dummy subtrees to shift the sequential index assignment. After `update()`, `A`'s parent list contains only 8 entries, with slot `k` holding only the last parent `P8`. The true shortest path to the parse stack may go through `P0`, which has been evicted.

When `find_path(A)` is called, it explores only the 8 surviving parents. If the shortest path to the stack top goes through the evicted `P0`, `find_path()` returns `None`, and the serializer emits the full atom `A` again instead of a 1-byte back-reference, producing a larger serialized output.
