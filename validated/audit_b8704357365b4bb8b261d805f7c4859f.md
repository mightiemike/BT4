### Title
4-Byte Secp Opcode Path Bypasses `ENABLE_SECP_OPS` Flag Gate — (`src/chia_dialect.rs`)

### Summary

The `ChiaDialect::op()` function in `src/chia_dialect.rs` contains two separate dispatch paths for secp operations. The 1-byte path (opcodes 64/65) correctly gates execution behind `ENABLE_SECP_OPS`. The 4-byte path (opcodes `0x13d61f00` / `0x1c3a8f00`) dispatches directly to the same underlying functions — `op_secp256k1_verify` and `op_secp256r1_verify` — **without any `ENABLE_SECP_OPS` check**. An attacker-controlled CLVM program using the 4-byte encoding executes secp verification regardless of whether the flag is set.

### Finding Description

In `ChiaDialect::op()`, the 4-byte opcode branch: [1](#0-0) 

dispatches unconditionally to `op_secp256k1_verify` / `op_secp256r1_verify` with no flag check. The 1-byte branch, by contrast: [2](#0-1) 

requires `flags.contains(ClvmFlags::ENABLE_SECP_OPS)` before dispatching to the same functions. Both paths call the identical production implementations in `src/secp_ops.rs`: [3](#0-2) [4](#0-3) 

The `ENABLE_SECP_OPS` flag is defined as the sole gate for secp operations: [5](#0-4) 

### Impact Explanation

Any CLVM program embedding the 4-byte atom `\x13\xd6\x1f\x00` as an operator will invoke `op_secp256k1_verify` and return `nil` (success) on a valid signature, or raise `Secp256Failed` on an invalid one — **identical behavior to when `ENABLE_SECP_OPS` is set**. A node running with `ClvmFlags::empty()` (or any flags set that exclude `ENABLE_SECP_OPS`) will evaluate such a program differently than intended: the secp gate is bypassed, the program result changes (succeeds vs. being treated as an unknown op), and consensus/mempool divergence becomes possible between nodes that do and do not set `ENABLE_SECP_OPS`.

### Likelihood Explanation

The attacker-controlled input is a CLVM program — a standard, fully supported production entrypoint. No special privileges are required. The 4-byte atom is trivially constructable. The bypass is deterministic and locally reproducible.

### Recommendation

Add an `ENABLE_SECP_OPS` guard to the 4-byte dispatch path, mirroring the 1-byte path:

```rust
0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
_ => return unknown_operator(allocator, o, argument_list, flags, max_cost),
```

If the 4-byte opcodes are intentionally ungated (e.g., as a pre-softfork encoding), the `ENABLE_SECP_OPS` flag documentation and all callers must be updated to reflect that it does **not** gate the 4-byte variants, and the invariant must be explicitly documented.

    // (use any known-good test vector)
    let args = /* ... valid secp256k1 pubkey, msg, sig ... */;

    let program = allocator.new_pair(opcode, args).unwrap();

    // Run WITHOUT ENABLE_SECP_OPS
    let dialect_no_flag = ChiaDialect::new(ClvmFlags::empty());
    let result_no_flag = run_program(&mut allocator, &dialect_no_flag, program, allocator.nil(), u64::MAX);

    // Run WITH ENABLE_SECP_OPS
    let dialect_with_flag = ChiaDialect::new(ClvmFlags::ENABLE_SECP_OPS);

    let opcode_1byte = allocator.new_atom(&[64]).unwrap();

    let result_1byte_no_flag = run_program(&mut allocator, &dialect_no_flag, program_1byte, allocator.nil(), u64::MAX);
    assert!(result_1byte_no_flag.is_err(), "1-byte path correctly blocked");
