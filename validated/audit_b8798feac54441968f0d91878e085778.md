### Title
IBC NFT Packet Receiver Not Checked Against Blocked Module Accounts, Causing Permanent NFT Freeze - (`x/nft-transfer/keeper/packet.go`)

### Summary
The `processReceivedPacket` function in the IBC NFT-transfer module mints or unescrows NFTs directly to the `receiver` address decoded from the incoming packet without verifying that the address is not a blocked module account. Any user on a counterparty IBC-connected chain can craft a transfer packet whose `receiver` field is set to a module account address on Cronos POS Chain. The NFT is then assigned to that module account, which has no private key and cannot sign transactions, permanently freezing the NFT.

### Finding Description
In `processReceivedPacket`, the receiver address is decoded from the packet data and used directly as the NFT owner:

```go
receiver, err := sdk.AccAddressFromBech32(data.Receiver)
if err != nil {
    return err
}
// ...
k.nftKeeper.MintNFT(ctx, voucherClassID, tokenID, "", data.TokenUris[i], "", escrowAddress, receiver)
// or
k.nftKeeper.TransferOwner(ctx, voucherClassID, tokenID, escrowAddress, receiver)
```

The only validation performed on `receiver` is that it is a syntactically valid bech32 address (`sdk.AccAddressFromBech32`). There is no check against the application's blocked-address list (`BlockedAddrs()`), which includes module accounts such as `bonded_tokens_pool`, `distribution`, `gov`, `mint`, etc. [1](#0-0) 

The `NonFungibleTokenPacketData.ValidateBasic()` called in `OnRecvPacket` also only validates that the receiver is a non-empty, parseable bech32 string; it does not reject module account addresses. [2](#0-1) 

The `ValidateMsgTransferDecorator` in the ante handler only runs on `CheckTx` (mempool admission), not during `FinalizeBlock`, and in any case applies only to outgoing `MsgTransfer` transactions — not to incoming IBC packets processed via `OnRecvPacket`. [3](#0-2) 

The application defines blocked addresses via `BlockedAddrs()`, but this list is never consulted by the NFT-transfer keeper. [4](#0-3) 

The underlying `TransferOwner` and `MintNFT` keeper functions perform no blocked-address check either. [5](#0-4) 

### Impact Explanation
An NFT transferred via IBC to a module account address becomes permanently inaccessible. The module account has no private key, so no transaction can ever be signed to move the NFT out. The corrupted state is the `nft.Owner` field, which is set to a module account address. In the "away from origin" path the voucher NFT is minted to the module account; in the "back to origin" path the original escrowed NFT is unescrow-transferred to the module account. In both cases the NFT is irrecoverably frozen.

### Likelihood Explanation
Any user who holds an NFT on a counterparty IBC-connected chain can trigger this by submitting a standard `MsgTransfer` with `receiver` set to a known module account bech32 address (e.g., the `bonded_tokens_pool` address, which is deterministically derivable). No privileged access is required. The IBC relayer will relay the packet and `OnRecvPacket` will execute without error, writing a success acknowledgement back to the source chain. [6](#0-5) 

### Recommendation
In `processReceivedPacket`, after decoding the receiver address, verify it is not a blocked module account before proceeding:

```go
receiver, err := sdk.AccAddressFromBech32(data.Receiver)
if err != nil {
    return err
}
if k.authKeeper.GetAccount(ctx, receiver) != nil {
    // check against blocked addresses
    if k.bankKeeper.BlockedAddr(receiver) {
        return sdkerrors.Wrapf(sdkerrors.ErrUnauthorized,
            "receiver %s is a blocked module account", data.Receiver)
    }
}
```

This requires adding a `BlockedAddr(addr sdk.AccAddress) bool` method to the `AccountKeeper` or a separate `BankKeeper` interface in `expected_keepers.go`, consistent with how the fungible token transfer module handles this in ibc-go. [7](#0-6) 

### Proof of Concept
1. Chain A (Cronos POS Chain) and Chain B are connected via an IBC NFT channel.
2. User on Chain B owns NFT `(classB, tokenX)`.
3. User submits `MsgTransfer` on Chain B with:
   - `receiver` = bech32 address of the `bonded_tokens_pool` module account on Chain A (deterministically computed as `sha256("bonded_tokens_pool")[:20]` encoded as bech32 with prefix `cro`).
4. IBC relayer relays the packet to Chain A.
5. `OnRecvPacket` → `processReceivedPacket` is called. `sdk.AccAddressFromBech32` succeeds. `data.ValidateBasic()` passes. `MintNFT(..., receiver)` is called with the module account address.
6. The NFT is now owned by `bonded_tokens_pool`. No private key exists for this address. The NFT is permanently frozen.
7. A success acknowledgement is written back to Chain B, confirming the transfer succeeded. The user's NFT on Chain B is burned. The NFT is lost. [8](#0-7) [9](#0-8)

```go
func (im IBCModule) OnRecvPacket(
	ctx sdk.Context,
	channelVersion string,
	packet channeltypes.Packet,
	relayer sdk.AccAddress,
) ibcexported.Acknowledgement {
	ack := channeltypes.NewResultAcknowledgement([]byte{byte(1)})

	var data types.NonFungibleTokenPacketData
	if err := types.ModuleCdc.UnmarshalJSON(packet.GetData(), &data); err != nil {
		ack = channeltypes.NewErrorAcknowledgement(fmt.Errorf("cannot unmarshal ICS-721 nft-transfer packet data"))
	}

	// only attempt the application logic if the packet data
	// was successfully decoded
	if ack.Success() {
		if err := im.keeper.OnRecvPacket(ctx, channelVersion, packet, data); err != nil {
			ack = types.NewErrorAcknowledgement(err)
		}
	}
	sdkCtx := sdk.UnwrapSDKContext(ctx)
	sdkCtx.EventManager().EmitEvent(
		sdk.NewEvent(
			types.EventTypePacket,
			sdk.NewAttribute(sdk.AttributeKeyModule, types.ModuleName),
			sdk.NewAttribute(sdk.AttributeKeySender, data.Sender),
			sdk.NewAttribute(types.AttributeKeyReceiver, data.Receiver),
			sdk.NewAttribute(types.AttributeKeyClassID, data.ClassId),
			sdk.NewAttribute(types.AttributeKeyTokenIDs, strings.Join(data.TokenIds, ",")),
			sdk.NewAttribute(types.AttributeKeyAckSuccess, fmt.Sprintf("%t", ack.Success())),
		),
	)

	// NOTE: acknowledgement will be written synchronously during IBC handler execution.
	return ack
}
```
