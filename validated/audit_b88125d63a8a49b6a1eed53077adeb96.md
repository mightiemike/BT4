### Title
Authz Grantee Can Redirect Minted NFTs to Themselves Using Denom Creator's Authority — (`x/nft/keeper/keeper.go`, `x/nft/keeper/msg_server.go`)

---

### Summary

An authz grantee holding a `GenericAuthorization` for `MsgMintNFT` from the denom creator can construct a `MsgMintNFT` with `Sender = Alice` (the granter/creator) and `Recipient = Bob` (themselves), submit it via `MsgExec`, and have the chain mint a new NFT directly into Bob's ownership — all while consuming Alice's denom-creator authority. Alice has no per-mint control over the `Recipient` field once the grant is issued.

---

### Finding Description

The exploit path is fully traceable through production code:

**Step 1 — `GetSigners` returns `msg.Sender`.**

`MsgMintNFT.GetSigners()` returns the address in the `Sender` field: [1](#0-0) 

When Bob constructs `MsgMintNFT{Sender: Alice, Recipient: Bob, ...}`, `GetSigners()` returns Alice. The authz module therefore only requires that Bob holds a grant from Alice for `MsgMintNFT` — which he does.

**Step 2 — `msgServer.MintNFT` passes `sender = Alice` and `recipient = Bob` to the keeper.** [2](#0-1) 

`sender` is parsed from `msg.Sender` (Alice) and `recipient` from `msg.Recipient` (Bob). Both are passed independently to the keeper.

**Step 3 — `keeper.MintNFT` only checks `IsDenomCreator` for `sender`, never for `recipient`.** [3](#0-2) 

`IsDenomCreator(ctx, denomID, sender)` passes because `sender` is Alice, the legitimate creator. There is no check that `recipient == sender` or any restriction on who the recipient may be.

**Step 4 — `IsDenomCreator` only compares the `address` argument against the stored creator.** [4](#0-3) 

The check is purely `creator.Equals(address)` where `address = Alice`. It passes unconditionally regardless of who `recipient` is.

**Step 5 — No custom `Authorization` type exists for `MsgMintNFT`.**

The grep across the entire `x/nft` module finds no `Authorization` implementation — only `GenericAuthorization` (from the SDK) is available. `GenericAuthorization` imposes zero restrictions on message fields, so Bob can freely set `Recipient` to any address, including his own. [5](#0-4) 

---

### Impact Explanation

Bob, holding a single `MsgMintNFT` grant from Alice, can:

- Mint **any number of NFTs** (up to grant expiry or spend limit, but `GenericAuthorization` has neither) into his own wallet from Alice's denom.
- Effectively **seize the entire future mint supply** of Alice's collection without Alice's knowledge or per-mint consent.
- Alice's NFT count is unchanged; Bob's grows. The ownership invariant — that the denom creator decides who receives each minted NFT — is broken.

This is a direct, irreversible NFT ownership state change: `nft.Owner = Bob`, `ownerStore[denomID][tokenID] = Bob`.

---

### Likelihood Explanation

- Authz grants for `MsgMintNFT` are a natural use case (e.g., delegating minting to a backend service or co-creator).
- The grant requires no special privilege beyond Alice issuing it once.
- The exploit requires only a standard `MsgExec` transaction — no governance, no operator access, no leaked keys.
- The attack is silent: events show `Sender = Alice`, which may not immediately alert Alice.

---

### Recommendation

Two complementary fixes:

1. **Add a custom `MintNFTAuthorization` type** (analogous to Cosmos SDK's `SendAuthorization`) that restricts the allowed `Recipient` addresses or enforces `Recipient == Granter`. Register it via `RegisterImplementations` in `codec.go`.

2. **Alternatively, enforce `recipient == sender` in `keeper.MintNFT`** if the intended semantic is that the creator always receives the freshly minted NFT before optionally transferring it. This is a breaking change to the message API but closes the hole at the keeper level regardless of authz.

---

### Proof of Concept

```
// 1. Alice creates denom "myCollection"
MsgIssueDenom{Sender: Alice, Id: "myCollection", ...}

// 2. Alice grants Bob GenericAuthorization for MsgMintNFT
MsgGrant{
  Granter: Alice,
  Grantee: Bob,
  Grant: {Authorization: GenericAuthorization{MsgTypeUrl: "/chainmain.nft.v1.MsgMintNFT"}}
}

// 3. Bob submits MsgExec with Recipient = Bob
MsgExec{
  Grantee: Bob,
  Msgs: [MsgMintNFT{
    Sender:    Alice,       // GetSigners() → Alice → authz grant check passes
    DenomId:   "myCollection",
    Id:        "token1",
    Recipient: Bob,        // NFT ownership goes here
    ...
  }]
}

// Result:
// - IsDenomCreator(ctx, "myCollection", Alice) → passes
// - NFT "token1" minted with Owner = Bob
// - Alice's collection supply +1, but Bob owns the token
// - Alice never consented to Bob receiving this specific NFT
```

```go
func (k Keeper) IsDenomCreator(ctx sdk.Context, denomID string, address sdk.AccAddress) (types.Denom, error) {
	denom, err := k.GetDenom(ctx, denomID)
	if err != nil {
		return types.Denom{}, err
	}

	creator, err := sdk.AccAddressFromBech32(denom.Creator)
	if err != nil {
		panic(err)
	}

	if !creator.Equals(address) {
		return types.Denom{}, sdkerrors.Wrapf(types.ErrUnauthorized, "%s is not the creator of %s", address, denomID)
	}

	return denom, nil
```
