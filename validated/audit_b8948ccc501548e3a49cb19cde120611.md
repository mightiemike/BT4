### Title
Denom Creator Role Is Immutable with No Transfer Mechanism, Permanently Locking NFT Collection Privileges After `MsgTransferNFT` or IBC Receive — (`x/nft/keeper/denom.go`, `x/nft/keeper/keeper.go`, `x/nft-transfer/keeper/packet.go`)

---

### Summary

The `Denom.creator` field is set once at `IssueDenom` time and is never updated. There is no `TransferDenom` or `UpdateDenomCreator` instruction in the NFT module. Both `EditNFT` and `BurnNFT` require the caller to be simultaneously the NFT token owner **and** the denom creator. After a `MsgTransferNFT`, the new NFT owner is permanently locked out of `EditNFT` and `BurnNFT` because they are not the denom creator. For IBC-received NFTs, the denom creator is set to the escrow address — a module-controlled account no user controls — permanently locking all IBC NFT receivers out of `EditNFT` and `BurnNFT` on the destination chain.

---

### Finding Description

**1. The `Denom.creator` field is immutable.**

`SetDenom` only creates new denoms and rejects updates to existing ones: [1](#0-0) 

`IssueDenom` calls `SetDenom` with the creator fixed at issuance time: [2](#0-1) 

The `Msg` service in `tx.proto` exposes only `IssueDenom`, `MintNFT`, `EditNFT`, `TransferNFT`, and `BurnNFT` — there is no `TransferDenom` or `UpdateDenomCreator` instruction: [3](#0-2) 

**2. `EditNFT` and `BurnNFT` require the caller to be both NFT owner AND denom creator.**

`EditNFT` calls both `IsOwner` and `IsDenomCreator` on the same address: [4](#0-3) 

`BurnNFT` does the same: [5](#0-4) 

**3. `MsgTransferNFT` transfers NFT ownership but does not update the denom creator.**

`TransferOwner` updates only the NFT's `Owner` field and the owner index — the `Denom.creator` is never touched: [6](#0-5) 

After a transfer, the new NFT owner satisfies `IsOwner` but fails `IsDenomCreator`, permanently blocking `EditNFT` and `BurnNFT`.

**4. IBC receive sets the denom creator to the escrow address — an uncontrollable module account.**

In `processReceivedPacket`, when an NFT arrives on the destination chain, the denom is issued with `escrowAddress` as creator: [7](#0-6) 

NFTs are then minted to the `receiver` with `escrowAddress` as the `sender` (creator check): [8](#0-7) 

The `escrowAddress` is a deterministic hash of the port/channel — no private key exists for it: [9](#0-8) 

Since there is no `TransferDenom` instruction, the receiver on the destination chain can never become the denom creator. They are permanently locked out of `EditNFT` and `BurnNFT` for any IBC-received NFT.

---

### Impact Explanation

**Corrupted invariant:** The `Denom.creator` field in the NFT module is the privileged role controlling minting, editing, and burning. After `MsgTransferNFT`, the new NFT owner holds the asset but cannot exercise `EditNFT` or `BurnNFT` because the creator role is permanently stuck with the original issuer. For IBC-received NFTs, the creator is the escrow address — a keyless module account — so no user on the destination chain can ever `EditNFT` or `BurnNFT` those NFTs. The `MintNFT` operation is also permanently restricted to the original creator, meaning the new owner of a collection cannot mint new tokens.

---

### Likelihood Explanation

**High.** Every `MsgTransferNFT` transaction creates this condition for the recipient. Every IBC NFT receive (`OnRecvPacket`) creates this condition for the destination-chain receiver. These are normal, supported production entry paths. Any user who receives an NFT via transfer or IBC is immediately affected.

---

### Recommendation

Introduce a `MsgTransferDenom` (or `MsgUpdateDenomCreator`) instruction that allows the current denom creator to transfer the creator role to a new address. The keeper should expose an `UpdateDenomCreator` function that updates the `Denom.creator` field in the KV store. This mirrors the fix described in the referenced report (introducing a separate instruction for assigning the privileged role). For IBC-received denoms, consider setting the creator to the first receiver rather than the escrow address, or allowing the receiver to claim the creator role.

---

### Proof of Concept

1. Alice issues a denom: `MsgIssueDenom{id: "mycollection", sender: alice}` → `Denom.creator = alice`.
2. Alice mints an NFT to herself: `MsgMintNFT{denom_id: "mycollection", sender: alice, recipient: alice}`.
3. Alice transfers the NFT to Bob: `MsgTransferNFT{denom_id: "mycollection", id: "token1", sender: alice, recipient: bob}`.
4. Bob attempts `MsgEditNFT{denom_id: "mycollection", id: "token1", sender: bob}` → fails: `IsDenomCreator` returns `"bob is not the creator of mycollection"`.
5. Bob attempts `MsgBurnNFT{denom_id: "mycollection", id: "token1", sender: bob}` → fails: same error.
6. There is no `MsgTransferDenom` instruction for Alice to delegate the creator role to Bob.

**IBC variant:**
1. Chain A sends NFT via `MsgTransfer` to Chain B receiver Carol.
2. On Chain B, `processReceivedPacket` creates `Denom{creator: escrowAddress}` and mints to Carol.
3. Carol attempts `MsgEditNFT` or `MsgBurnNFT` on Chain B → permanently fails because `escrowAddress` is the creator and no private key exists for it.
4. No `MsgTransferDenom` exists to fix this.

```go
func (k Keeper) SetDenom(ctx sdk.Context, denom types.Denom) error {
	if k.HasDenomID(ctx, denom.Id) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denomID %s has already exists", denom.Id)
	}

	if k.HasDenomNm(ctx, denom.Name) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denomName %s has already exists", denom.Name)
	}

	store := ctx.KVStore(k.storeKey)
	bz := k.cdc.MustMarshal(&denom)
	store.Set(types.KeyDenomID(denom.Id), bz)
	store.Set(types.KeyDenomName(denom.Name), []byte(denom.Id))
	return nil
```

```go
func (k Keeper) TransferOwner(

	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)

	nft, err := k.IsOwner(ctx, denomID, tokenID, srcOwner)
```

```go
// BurnNFT deletes a specified NFT
func (k Keeper) BurnNFT(ctx sdk.Context, denomID, tokenID string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	nft, err := k.IsOwner(ctx, denomID, tokenID, owner)
	if err != nil {
		return err
	}

	_, err = k.IsDenomCreator(ctx, denomID, owner)
	if err != nil {
		return err
	}

	k.deleteNFT(ctx, denomID, nft)
	k.deleteOwner(ctx, denomID, tokenID, owner)
	k.decreaseSupply(ctx, denomID)

	return nil
}
```

```go
func GetEscrowAddress(portID, channelID string) sdk.AccAddress {
	// a slash is used to create domain separation between port and channel identifiers to
	// prevent address collisions between escrow addresses created for different channels
	contents := fmt.Sprintf("%s/%s", portID, channelID)

	// ADR 028 AddressHash construction
	preImage := []byte(Version)
	preImage = append(preImage, 0)
	preImage = append(preImage, contents...)
	hash := sha256.Sum256(preImage)
	return hash[:20]
}
```
