### Title
Ghost-Heap Accounting Omission in `new_substr` SmallAtom Branch Allows Heap-Limit Bypass ŌĆö (`File: src/allocator.rs`)

---

### Summary

`Allocator::new_substr()` contains a branch for `SmallAtom` inputs that produces a `SmallAtom` result. This branch increments `ghost_atoms` but never increments `ghost_heap`, unlike every other allocation path that produces a `SmallAtom`. Because the heap-limit guard (`u8_vec.len() + ghost_heap + new_size > heap_limit`) relies on `ghost_heap` being a faithful running total of all virtual heap bytes, the omission lets an attacker craft a CLVM program that silently under-counts `ghost_heap` and then allocates more real heap bytes than the consensus limit permits.

---

### Finding Description

**Root cause ŌĆö the broken invariant**

Every allocation path that produces a `SmallAtom` is supposed to increment `ghost_heap` by the atom's byte length, so that the heap-limit check remains accurate even though the bytes are stored inline in the `NodePtr` rather than in `u8_vec`. `new_atom` enforces this correctly:

```rust
// src/allocator.rs  new_atom(), lines 626-629
if let Some(ret) = fits_in_small_atom(v) {
    self.ghost_atoms += 1;
    self.ghost_heap += v.len();   // ŌåÉ ghost_heap IS updated
    Ok(self.mk_node(ObjectType::SmallAtom, ret as usize))
```

`new_small_number` also increments `ghost_heap` (lines 647-648). But the `SmallAtom` branch of `new_substr` does not:

```rust
// src/allocator.rs  new_substr(), lines 852-854
if let Some(new_val) = fits_in_small_atom(substr) {
    self.ghost_atoms += 1;
    // ghost_heap is never touched here
    Ok(self.mk_node(ObjectType::SmallAtom, new_val as usize))
```

**How the heap-limit guard is defeated**

Every subsequent real allocation checks:

```rust
// src/allocator.rs  new_atom(), lines 621-623
if start as usize + self.ghost_heap + v.len() > self.heap_limit {
    return Err(EvalErr::OutOfMemory);
}
```

`start` is `u8_vec.len()`. If `ghost_heap` is under-counted by *N* bytes (one missed increment per `substr`-on-SmallAtom call), the guard permits *N* extra bytes to be written into `u8_vec`. The total real heap usage can therefore reach `heap_limit + N` bytes before the guard fires.

**Attacker-controlled entry path**

The CLVM `substr` operator is a standard, unprivileged opcode. An attacker submits a crafted CLVM program (e.g., as a coin puzzle) that:

1. Holds a SmallAtom in the environment (e.g., a small integer argument).
2. Calls `substr` on it in a loop, each time producing a SmallAtom result.
3. Each iteration increments `ghost_atoms` but leaves `ghost_heap` unchanged.
4. After exhausting the desired number of iterations, allocates large real atoms; the heap-limit guard is now permissive by *N* bytes.

The maximum under-count is bounded by `MAX_NUM_ATOMS ├Ś max_SmallAtom_byte_length = 62 500 000 ├Ś 3 Ōēł 187 MB`, because the atom limit (`check_atom_limit`) is still enforced at the top of `new_substr`.

---

### Impact Explanation

The `ghost_heap` counter exists specifically to maintain consensus-compatible memory limits: nodes must agree on whether a program exceeds the heap limit. If `ghost_heap` is under-counted, a program that should be rejected as `OutOfMemory` is instead accepted. This produces:

- **Consensus divergence**: a node running the patched reference implementation (or a future version that fixes the omission) rejects the program; a node running the current code accepts it. The two nodes disagree on the validity of a block, which is a critical integrity failure in the Chia consensus model.
- **Memory exhaustion**: validator nodes can be forced to allocate up to ~187 MB more than the intended heap limit per program execution, degrading or crashing full nodes under adversarial load.

---

### Likelihood Explanation

The `substr` opcode is reachable by any coin puzzle submitted to the mempool ŌĆö no privileged keys or special roles are required. The cost of each `substr` call is non-zero, so the maximum discrepancy is bounded by both the atom limit and the block cost limit. Nevertheless, even a modest number of calls (thousands) produces a measurable and exploitable discrepancy in `ghost_heap`, making this a realistic attack vector for a motivated adversary.

---

### Recommendation

Add the missing `ghost_heap` increment in the `SmallAtom` branch of `new_substr`, mirroring the pattern used in `new_atom` and `new_small_number`:

```rust
// src/allocator.rs  new_substr(), SmallAtom branch
if let Some(new_val) = fits_in_small_atom(substr) {
    self.ghost_atoms += 1;
    self.ghost_heap += (end - start) as usize;  // ŌåÉ add this line
    Ok(self.mk_node(ObjectType::SmallAtom, new_val as usize))
}
```

Additionally, add a heap-limit pre-check analogous to the one in `new_atom` before the SmallAtom branch executes, to ensure the guard fires consistently regardless of which allocation path is taken.
