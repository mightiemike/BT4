Audit Report

## Title
Pool Admin Can Reduce Timelock to Zero and Atomically Disable All Stop-Loss Protections Without LP Reaction Window — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`proposeOracleStopLossTimelock` accepts `newTimelock = 0` with no minimum floor validation. Once executed, `_afterTimelock` returns `block.timestamp`, and `_requireElapsed` uses a strict `<` comparison that passes immediately at `block.timestamp < block.timestamp == false`. The pool admin can then propose and execute `drawdownE6 = 0` in the same block, causing `_afterSwapOracleStopLoss` to return early unconditionally, fully disabling the stop-loss guard with zero LP reaction window for all subsequent parameter changes.

## Finding Description

**Root cause — no minimum timelock floor:**

`proposeOracleStopLossTimelock` at line 78 accepts any `uint32 newTimelock` including 0, with no validation: [1](#0-0) 

Compare with `_validateDrawdown` and `_validateDecay` which do enforce upper bounds — there is no analogous `_validateTimelock`. [2](#0-1) 

**Root cause — strict `<` in `_requireElapsed`:**

When `timelock = 0`, `_afterTimelock` returns `uint32(block.timestamp + 0) = block.timestamp`: [3](#0-2) 

`_requireElapsed(block.timestamp)` evaluates `block.timestamp < block.timestamp` → **false** → no revert: [4](#0-3) 

**Root cause — `drawdownE6 = 0` is accepted and disables the guard:**

`_validateDrawdown` only rejects values `> E6`; zero passes: [5](#0-4) 

`_afterSwapOracleStopLoss` short-circuits unconditionally when `drawdownE6 == 0`: [6](#0-5) 

**Full exploit path:**

1. Pool admin calls `proposeOracleStopLossTimelock(pool, 0)` — accepted, `executeAfter = block.timestamp + currentTimelock`.
2. After the current timelock T elapses, admin calls `executeOracleStopLossTimelock(pool)` — writes `oracleStopLossConfig[pool].timelock = 0`.
3. In the same block (or any subsequent block), admin calls `proposeOracleStopLossDrawdown(pool, 0)` — `executeAfter = block.timestamp + 0 = block.timestamp`.
4. In the same transaction or same block, admin calls `executeOracleStopLossDrawdown(pool)` — `_requireElapsed(block.timestamp)` passes, writes `drawdownE6 = 0`.
5. Every subsequent `afterSwap` call hits `if (drawdown == 0) return;` and exits without any watermark check. The stop-loss is permanently disabled until re-initialized.

The same zero-delay pattern applies identically to `proposeOracleStopLossDecay`/`executeOracleStopLossDecay` and `proposeOracleStopLossHighWatermarks`/`executeOracleStopLossHighWatermarks`. [7](#0-6) 

## Impact Explanation

Once `drawdownE6 = 0`, the `afterSwap` hook returns early for every swap. The stop-loss that prevents per-share value from falling below the drawdown floor is completely inoperative. LPs are exposed to unlimited value leakage through swaps with no on-chain protection. The NatDoc at line 16 explicitly states the timelock exists so "LPs can react" — this mechanism is fully bypassed after the one-time timelock reduction. This is a direct admin-boundary break: the pool admin exceeds the constraint imposed by the timelock mechanism that was explicitly designed to limit their power. **Severity: High** — broken core pool functionality (stop-loss guard) causing potential loss of LP principal. [8](#0-7) 

## Likelihood Explanation

The pool admin is a single EOA or contract returned by `IMetricOmmPoolFactory(FACTORY).poolAdmin(pool_)`. The pool admin is explicitly semi-trusted — the timelock exists precisely to limit their power. The attack requires only two phases: wait the current timelock (visible on-chain), then atomically disable protections. No external oracle manipulation, no non-standard tokens, and no factory owner involvement is needed. The on-chain proposal of `newTimelock = 0` is the only observable signal LPs have, and they must act within the current timelock window — which may be short or unmonitored. [1](#0-0) 

## Recommendation

1. **Enforce a minimum timelock floor** in `proposeOracleStopLossTimelock`: add `require(newTimelock >= MIN_TIMELOCK)` where `MIN_TIMELOCK` is a meaningful constant (e.g., 1 hour or 1 day).
2. **Use non-strict inequality** in `_requireElapsed`: change `block.timestamp < executeAfter` to `block.timestamp <= executeAfter` so same-block execution is always rejected regardless of timelock value.
3. **Validate timelock at initialization**: `initialize` calls `_validateDrawdown` and `_validateDecay` but has no `_validateTimelock` — add one to prevent zero-timelock pools from being created. [9](#0-8) 

## Proof of Concept

```solidity
function test_timelockZeroDisablesStopLoss() public {
    // Pool deployed with timelock = 1 days, drawdownE6 = 50_000 (5%)

    // Phase 1: propose timelock = 0 (must wait current timelock)
    vm.prank(poolAdmin);
    ext.proposeOracleStopLossTimelock(pool, 0);

    // Warp past current timelock
    vm.warp(block.timestamp + 1 days);

    // Execute timelock reduction — oracleStopLossConfig[pool].timelock == 0
    vm.prank(poolAdmin);
    ext.executeOracleStopLossTimelock(pool);

    // Phase 2: same block — propose drawdown = 0
    // executeAfter = block.timestamp + 0 = block.timestamp
    vm.prank(poolAdmin);
    ext.proposeOracleStopLossDrawdown(pool, 0);

    // Same block — execute drawdown = 0
    // _requireElapsed: block.timestamp < block.timestamp == false → passes
    vm.prank(poolAdmin);
    ext.executeOracleStopLossDrawdown(pool);

    // Assert drawdown is now 0
    (uint32 drawdown,,,) = ext.oracleStopLossConfig(pool);
    assertEq(drawdown, 0);

    // Phase 3: swap that would have triggered stop-loss now succeeds
    // _afterSwapOracleStopLoss hits `if (drawdown == 0) return;` — no revert
    _executeValueDrainingSwap(pool); // succeeds, stop-loss fully disabled
}
```
