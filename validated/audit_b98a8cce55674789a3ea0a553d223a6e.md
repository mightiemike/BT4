### Title
Stale Mainchain Block Timestamp Used in Fork Difficulty Calculation — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

### Summary

When computing the required difficulty for a fork block at a difficulty-adjustment boundary, `get_next_work_required` fetches the interval-start block via `get_header_by_height`, which unconditionally reads from `mainchain_height_to_header`. If the fork diverged before that interval-start height, the mainchain block at that height is a **different block** from the fork's actual ancestor, and its timestamp is used instead of the fork ancestor's timestamp. This produces an incorrect `expected_bits`, allowing a fork block to pass difficulty validation with a lower (or higher) difficulty than the protocol requires.

The Dogecoin module even contains an explicit developer acknowledgment of this uncertainty:

> `// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor`

### Finding Description

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // always the canonical chain
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

All three chain-specific difficulty modules call this function to obtain the interval-start block when a retarget is due:

The timestamp retrieved (`first_block_time` / `interval_tail_extend_header.block_header.time`) is then fed directly into `calculate_next_work_required` to compute `actual_time_taken` and derive `expected_bits`:

```rust
let mut actual_time_taken: i64 = prev_block_time - first_block_time;
// clamped to [pow_target_timespan/4, pow_target_timespan*4]
let new_target = target_from_bits(prev_block_header.block_header.bits);
new_target * actual_time_taken / pow_target_timespan
``` [5](#0-4) 

When a fork block is submitted whose ancestor chain diverged **before** `height_first`, the mainchain block at `height_first` is a completely different block from the fork's actual ancestor at that height. Its timestamp `T_main` is substituted for the fork ancestor's timestamp `T_fork`, corrupting `actual_time_taken` and therefore `expected_bits`.

### Impact Explanation

**Scenario — attacker lowers the required difficulty for a fork block:**

1. The mainchain has a block at `height_first` with timestamp `T_main`.
2. The attacker submits a fork that diverges before `height_first`. The fork's block at `height_first` is crafted with timestamp `T_fork > T_main` (within the 2-hour future-timestamp window).
3. When the attacker submits the fork block at the retarget boundary, the contract calls `get_header_by_height(height_first)` and retrieves the **mainchain** block with `T_main`.
4. `actual_time_taken = prev_block_time − T_main` is **larger** than the correct value `prev_block_time − T_fork`.
5. The contract computes a lower `expected_bits` (easier target) than the protocol requires.
6. The attacker mines the fork block at this reduced difficulty and submits it; `expected_bits == block_header.bits` passes.
7. Subsequent fork blocks inherit this incorrect difficulty baseline.
8. If the fork accumulates sufficient `chain_work`, `reorg_chain` is triggered, corrupting the canonical chain mapping.

The corrupted state is: `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height` — the three structures that define what the light client considers the canonical Bitcoin/Litecoin/Dogecoin chain. Any downstream SPV proof verified against this corrupted canonical chain will return incorrect results. [6](#0-5) [7](#0-6) 

### Likelihood Explanation

- The entry path is `submit_blocks`, a production public endpoint callable by any trusted relayer. The trusted-relayer staking system allows any NEAR account with sufficient funds to register.
- For Bitcoin/Litecoin, the attack requires building a fork that spans a 2016-block retarget boundary — a significant but not prohibitive amount of work, especially if the difficulty manipulation itself reduces the mining cost.
- For Dogecoin (post-height 145,000, `difficulty_adjustment_interval = 1`), the stale lookup fires on **every single fork block**, making the surface much larger.
- No privileged role, leaked key, or social engineering is required. The vulnerability is purely in the on-chain logic.

### Recommendation

Replace the height-based mainchain lookup with an ancestor traversal that follows the fork's own `prev_block_hash` chain back to `height_first`. Concretely, starting from `prev_block_header`, walk backwards through `headers_pool` via `get_prev_header` until the block at `height_first` is reached, and use that block's timestamp. This mirrors how full-node implementations (Bitcoin Core, Litecoin Core, Dogecoin Core) compute the retarget: they always walk the ancestor chain of the block being validated, never a separate canonical-chain index.

### Proof of Concept

```
Setup (Bitcoin mainnet, difficulty_adjustment_interval = 2016):

height_first = B - 2015  (B = next retarget boundary)

Mainchain:
  [B-2015] time=T_main ... [B-1] time=T_prev

Attacker's fork (diverges at height B-2016, before height_first):
  [B-2015_fork] time=T_fork  (T_fork = T_main + 7200, within 2-hour window)
  ...
  [B-1_fork]   time=T_prev_fork

When attacker submits fork block at height B:
  Contract calls get_header_by_height(B-2015)
  → returns mainchain block with time=T_main   ← STALE VALUE

  actual_time_taken = T_prev_fork - T_main     ← inflated by (T_fork - T_main) = 7200s
  expected_bits = lower difficulty than correct

  Attacker's block at height B has bits matching this lower difficulty → ACCEPTED

  work_from_bits(lower_bits) still accumulates chainwork;
  if fork tip chain_work > mainchain chain_work → reorg_chain() fires
  → canonical chain corrupted
``` [1](#0-0) [4](#0-3) [8](#0-7)
