Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any caller to bypass the deposit allowlist — (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently drops its first parameter (`sender`, the actual token depositor) and gates only on `owner` (the position recipient). Because `MetricOmmPool.addLiquidity` allows `msg.sender` and `owner` to be independent addresses, any unprivileged caller can bypass the allowlist by naming an already-allowlisted address as `owner`, completely defeating the only on-chain mechanism restricting who may supply liquidity to a restricted pool.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the position owner to the extension hook: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` declares its first parameter unnamed (discarded) and checks only `owner` against `allowedDepositor`: [2](#0-1) 

The contract's own NatSpec and mapping name both declare the intent is to gate by **depositor** (the actual token provider), not the position recipient: [3](#0-2) 

By contrast, `removeLiquidity` enforces `msg.sender == owner` before calling the hook, so `sender` and `owner` are always identical there — the bypass surface exists only in `addLiquidity`: [4](#0-3) 

**Exploit path:**
1. Pool admin deploys a pool with `DepositAllowlistExtension`; only Alice (`0xAlice`) is allowlisted.
2. Bob (`0xBob`, not allowlisted) calls `pool.addLiquidity(owner=0xAlice, salt=0, deltas=..., callbackData=..., extensionData=...)`.
3. The pool calls `DepositAllowlistExtension.beforeAddLiquidity(sender=0xBob, owner=0xAlice, ...)`.
4. The hook evaluates `allowedDepositor[pool][0xAlice]` → `true`; the check passes.
5. Bob's callback (`metricOmmSwapCallback`) transfers Bob's tokens into the pool.
6. Alice receives LP shares she never requested; Bob has deposited into a restricted pool without authorization.

The wrong value is the `allowedDepositor` lookup key: it is `owner` (position recipient) when it must be `sender` (actual token provider, i.e., `msg.sender` of `addLiquidity`).

## Impact Explanation
The deposit allowlist — the only on-chain mechanism restricting who may supply liquidity — is rendered completely ineffective. Any unprivileged EOA or contract can add liquidity to a pool the admin intended to be private or KYC-gated. Additionally, the allowlisted `owner` (Alice) receives an involuntary LP position: if the pool is in an adverse price state at the time of the forced deposit, Alice's unintended exposure causes direct loss of LP value when she eventually withdraws. This constitutes a broken core pool functionality causing loss of funds and an admin-boundary break where an unprivileged path bypasses a configured access control.

## Likelihood Explanation
The bypass requires no special role, no flash loan, and no complex setup. Any EOA or contract can call `addLiquidity` with an allowlisted address as `owner`. Allowlisted addresses are publicly readable from `allowedDepositor` mapping events or direct storage reads. The attack is repeatable at any time and at zero cost beyond gas.

## Recommendation
Name and check `sender` (the actual depositor) instead of `owner` in `beforeAddLiquidity`:

```solidity
function beforeAddLiquidity(address sender, address /*owner*/, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Setup: pool has DepositAllowlistExtension; only `alice` is allowlisted. Bob is NOT.
contract BobBypassDeposit {
    IMetricOmmPool pool;
    IERC20 token0;
    IERC20 token1;
    address alice;

    function exploit(LiquidityDelta calldata deltas) external {
        // Bob calls addLiquidity with owner = alice (allowlisted)
        // Hook checks allowedDepositor[pool][alice] == true → passes
        // Bob's tokens are pulled in the callback; alice gets the shares
        pool.addLiquidity(
            alice,                          // owner — passes the allowlist check
            0,                              // salt
            deltas,
            abi.encode(address(this)),      // callbackData
            ""
        );
    }

    function metricOmmSwapCallback(int128 amount0Delta, int128 amount1Delta, bytes calldata) external {
        if (amount0Delta > 0) token0.transfer(address(pool), uint256(int256(amount0Delta)));
        if (amount1Delta > 0) token1.transfer(address(pool), uint256(int256(amount1Delta)));
    }
}
```

A Foundry integration test can confirm this by: (1) deploying a pool with `DepositAllowlistExtension`, (2) allowlisting only `alice`, (3) calling `addLiquidity` from `bob` with `owner=alice`, and (4) asserting the call succeeds and LP shares are minted to `alice` — demonstrating the allowlist is bypassed.
