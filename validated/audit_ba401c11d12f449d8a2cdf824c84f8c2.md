### Title
Receipt Size Limit Bypassed via `promise_return` `output_data_receivers` Injection After `NewReceipt` Validation — (`runtime/runtime/src/lib.rs`, `runtime/runtime/src/verifier.rs`)

---

### Summary

The `max_receipt_size` invariant can be violated by any unprivileged contract caller. A contract crafts a receipt whose Borsh-serialized size equals exactly `max_receipt_size` (4,194,304 bytes). That receipt passes the `NewReceipt` size gate. The runtime then appends `output_data_receivers` entries to the receipt's `output_data_receivers` field via the `promise_return` path — **after** validation — without re-checking the size. The resulting receipt, now larger than `max_receipt_size`, is forwarded or queued with no further size enforcement, breaking the protocol invariant that all receipts must be below `max_receipt_size`.

---

### Finding Description

`validate_receipt` in `runtime/runtime/src/verifier.rs` enforces the size limit only when `mode == ValidateReceiptMode::NewReceipt`: [1](#0-0) 

After a contract executes and returns `ReturnData::ReceiptIndex(receipt_index)` (i.e., calls `promise_return`), the runtime in `runtime/runtime/src/lib.rs` mutates the already-validated receipt by appending the parent receipt's `output_data_receivers` directly into the child receipt's `output_data_receivers` field: [2](#0-1) 

No size re-check is performed after this mutation. The receipt is then forwarded or buffered via `receipt_sink.forward_or_buffer_receipt`.

When the oversized receipt is later dequeued from the delayed queue or arrives as an incoming receipt, `validate_receipt` is called with `ValidateReceiptMode::ExistingReceipt`, which explicitly skips the size check: [3](#0-2) 

The comment in the code explicitly acknowledges this as an open bug (issue #12606): [4](#0-3) 

The `ExistingReceipt` mode is used for both delayed receipt processing and incoming receipt processing: [5](#0-4) [6](#0-5) 

The test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` explicitly confirms that an oversized receipt is produced and enters the chain: [7](#0-6) 

A second vector exists via `test_max_receipt_size_value_return`: returning a value of exactly `max_receipt_size` bytes causes the runtime to wrap it in a `DataReceipt` whose Borsh encoding exceeds `max_receipt_size`, again bypassing the limit. [8](#0-7) 

---

### Impact Explanation

The `ChunkStateWitness` size budget is designed to stay under ~17 MiB. The `max_receipt_size` limit of 4 MiB is a hard component of that budget: [9](#0-8) 

An oversized receipt that bypasses the limit inflates the `ChunkStateWitness` beyond the intended bound. Additionally, congestion control size accounting reads the `congestion_size` field from `StateStoredReceiptMetadata`, which is computed from the pre-mutation Borsh size, so the stored size is wrong for any receipt that was enlarged by `output_data_receivers` injection. This causes the congestion control system to undercount memory pressure for the affected receipt.

---

### Likelihood Explanation

Any unprivileged account that can deploy a contract can trigger this. The attack requires:
1. Deploy a contract that creates a receipt with args sized to `max_receipt_size - base_receipt_overhead`.
2. Set up a promise DAG `[A -then-> B]` where A calls `promise_return(C)`.
3. When A executes, C's `output_data_receivers` is extended with B's data receiver, pushing C above `max_receipt_size`.

The test `test_max_receipt_size_promise_return` demonstrates this is reproducible with a standard test contract. The exact `args_size` needed is `max_receipt_size - base_receipt_size` where `base_receipt_size` is the Borsh length of a minimal action receipt: [10](#0-9) 

---

### Recommendation

After the `output_data_receivers` extension at lines 1029–1035 of `runtime/runtime/src/lib.rs`, re-compute the Borsh size of the mutated receipt and return a `NewReceiptValidationError::ReceiptSizeExceeded` if it exceeds `limit_config.max_receipt_size`. The same check should be applied after the `DataReceipt` wrapping path (lines 1044–1056). This mirrors the existing check in `validate_receipt` for `NewReceipt` mode and closes the window between validation and mutation.

---

### Proof of Concept

The existing test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` is the proof of concept. It:

1. Deploys a contract.
2. Calls `max_receipt_size_promise_return_method1` with `args_size = max_receipt_size - base_receipt_size`.
3. The contract creates receipt C at exactly `max_receipt_size`, then calls `promise_return(C)`.
4. The runtime appends `output_data_receivers` to C, pushing it above `max_receipt_size`.
5. `assert_oversized_receipt_occurred` confirms the oversized receipt is present in the chain. [11](#0-10)

#[test]
fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
