Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Immediate Pool Caller Instead of End-User, Enabling Allowlist Bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded by the pool, which is always `msg.sender` of the `pool.swap()` call. When `MetricOmmSimpleRouter` is used, `sender` resolves to the router's address rather than the originating user. A pool admin who allowlists the router to enable router-mediated swaps for legitimate users simultaneously grants every address on-chain the ability to bypass the curated-pool gate by routing through the router.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender`:**
`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, so `sender` is always the immediate caller of `pool.swap()`. [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards `sender` unchanged:**
The value is encoded directly into the extension call with no transformation. [2](#0-1) 

**Step 3 — Extension checks `allowedSwapper[pool][sender]`:**
`msg.sender` inside the extension is the pool; `sender` is whatever the pool forwarded. When the router calls the pool, `sender = router`. [3](#0-2) 

**Step 4 — Router calls `pool.swap()` directly without forwarding the originating user:**
`exactInputSingle`, `exactInput`, and `exactOutputSingle` all call `IMetricOmmPoolActions(pool).swap(...)` with the router as `msg.sender`. The originating user's address is stored in transient storage only for the payment callback — it is never passed to the pool as `sender` or embedded in `extensionData` for the extension to read. [4](#0-3) [5](#0-4) 

**Result:** On a direct call (`user → pool.swap()`), `sender = user` and the check is correct. On a router call (`user → router → pool.swap()`), `sender = router`. If the admin has allowlisted the router (the only way to let allowlisted users use the standard periphery), `allowedSwapper[pool][router] = true` passes for every caller regardless of whether that individual user is on the allowlist.

## Impact Explanation
Any address not individually allowlisted can trade on a curated pool by routing through `MetricOmmSimpleRouter` when the router is allowlisted. This is a direct bypass of the pool's access-control policy. Curated pools are deployed to restrict trading to known counterparties (e.g., KYC'd users, protocol-owned addresses, specific market makers). Unauthorized swaps drain LP-owned liquidity and generate fees from actors the pool was explicitly designed to exclude. This constitutes broken core pool functionality and direct loss of LP principal through unauthorized execution.

## Likelihood Explanation
The scenario is operationally necessary and therefore highly likely:
1. `MetricOmmSimpleRouter` is the canonical user-facing swap entry point documented in the protocol. [6](#0-5) 
2. A pool admin who wants allowlisted users to use the standard router **must** call `setAllowedToSwap(pool, router, true)` — there is no other mechanism. [7](#0-6) 
3. Once the router is allowlisted, the bypass is immediately available to any address with no additional preconditions, no privileged access, and no special token behavior.

## Recommendation
The extension must resolve the actual end-user address rather than trusting the `sender` argument when the immediate caller is a known periphery contract. Sound approaches:

1. **Pass-through identity**: Require the router to embed the originating user address in `extensionData`, and have `SwapAllowlistExtension` decode and check that address when `sender` is a recognized router.
2. **Transient-storage identity**: Have the router write the originating user into transient storage before calling the pool, and have the extension read it from there (the router already uses transient storage for callback context).
3. **Direct-only policy**: Document and enforce at the factory level that pools using `SwapAllowlistExtension` must not allowlist any router or intermediary contract.

The root fix is that `allowedSwapper` must be keyed on the economically responsible actor (the end user), not the immediate `msg.sender` of the pool call.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension (allowAllSwappers = false).
  - Pool admin: setAllowedToSwap(pool, router, true)   // enable router usage
  - Pool admin: setAllowedToSwap(pool, alice, true)    // alice is the intended allowlisted user
  - bob is NOT allowlisted.

Attack:
  1. bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: bob, ...})
  2. Router calls pool.swap(recipient=bob, ...) — msg.sender in pool = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → true ✓
  5. Swap executes for bob despite bob not being allowlisted.

Direct call (for comparison):
  1. bob calls pool.swap(...) directly
  2. Pool calls _beforeSwap(sender=bob, ...)
  3. SwapAllowlistExtension checks allowedSwapper[pool][bob] → false → revert ✓

Foundry test plan:
  - Deploy pool + SwapAllowlistExtension + MetricOmmSimpleRouter.
  - allowlist router and alice; do NOT allowlist bob.
  - Assert bob's direct pool.swap() reverts with NotAllowedToSwap.
  - Assert bob's router.exactInputSingle() succeeds (bypass confirmed).
```
