Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` validates `owner` instead of `sender`, allowing any caller to bypass the deposit allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and instead validates `owner`, which is a free parameter chosen by the caller. Because `addLiquidity` imposes no constraint that `msg.sender == owner`, any unauthorized caller can name any already-authorized address as `owner` and pass the allowlist check unconditionally, completely nullifying the deposit restriction mechanism.

## Finding Description
`MetricOmmPool.addLiquidity` passes two distinct addresses to the extension hook:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`msg.sender` is the entity that will provide tokens via the swap callback; `owner` is the LP-share recipient and is a free caller-supplied parameter with no enforced relationship to `msg.sender`. [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` leaves the first parameter (sender) unnamed and gates exclusively on `owner`:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly gates on `sender`: [3](#0-2) 

The asymmetry is the root cause. An unauthorized caller Bob sets `owner = alice` (an authorized address). The extension checks `allowedDepositor[pool][alice]` → `true` → no revert. Bob's tokens enter the pool via the callback; alice receives LP shares. No existing guard prevents this.

## Impact Explanation
The `DepositAllowlistExtension` is the sole on-chain mechanism for restricting who may add liquidity to a pool. The bypass is complete: an unauthorized actor can deposit into KYC-gated, institutional, or single-LP pools, mutating `curPosInBin`, `curBinDistFromProvidedPriceE6`, and `binTotals` — values that downstream extensions such as `OracleValueStopLossExtension` rely on for per-bin watermarks. This constitutes broken core pool functionality causing direct pool state corruption by an explicitly excluded actor, meeting the contest's High severity threshold. [4](#0-3) 

## Likelihood Explanation
Exploitation requires no special privilege. Any EOA or contract can call `addLiquidity` with an arbitrary `owner`. The `allowedDepositor` and `allowAllDepositors` mappings are public, so the set of authorized addresses is trivially enumerable on-chain. The attack is repeatable with no cooldown or cost beyond gas. [4](#0-3) 

## Recommendation
Replace the `owner` check with a `sender` check, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`sender` is `msg.sender` of the `addLiquidity` call — the entity that will actually provide tokens — which is the correct subject of the allowlist. [2](#0-1) 

## Proof of Concept
1. Deploy pool with `DepositAllowlistExtension`; call `setAllowedToDeposit(pool, alice, true)`. Bob is not in the allowlist.
2. Bob calls `pool.addLiquidity(owner = alice, salt, deltas, callbackData, extensionData)`.
3. Pool calls `extension.beforeAddLiquidity(bob /*sender*/, alice /*owner*/, ...)`.
4. Extension evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
5. `LiquidityLib.addLiquidity` executes with `owner = alice`; Bob's tokens are pulled via callback; alice receives LP shares.
6. Bob has successfully deposited into a restricted pool, mutating `curPosInBin`, `curBinDistFromProvidedPriceE6`, and `binTotals`. [5](#0-4)
