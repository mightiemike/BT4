### Title
`getCoreRisk` Hardcodes `amount=1`, Causing `getHealth` to Overestimate Spread Health for Short Perp Positions — (`core/contracts/BaseEngine.sol`)

---

### Summary

`BaseEngine.getCoreRisk` always passes the literal `1` (positive) to `RiskHelper._getWeightX18`, so the returned `CoreRisk.longWeight` field is always the **long** weight regardless of the subaccount's actual position direction. `Clearinghouse.getHealth` then uses this field to compute `existingWeight` for the spread adjustment. For a long-spot / short-perp basis position, `existingWeight` is underestimated because it uses `perpLongWeight` instead of `perpShortWeight`, making `spreadWeight - existingWeight` larger than correct and inflating the health contribution of the spread.

---

### Finding Description

**Root cause — `BaseEngine.getCoreRisk` line 190:**

```solidity
RiskHelper._getWeightX18(risk, 1, healthType)   // hardcoded positive 1
```

`_getWeightX18` branches on `amount >= 0`:

```solidity
if (amount >= 0) {
    weight = healthType == INITIAL
        ? risk.longWeightInitialX18       // always taken
        : risk.longWeightMaintenanceX18;
} else {
    weight = healthType == INITIAL
        ? risk.shortWeightInitialX18      // never reached
        : risk.shortWeightMaintenanceX18;
}
```

Because `1 >= 0` is always true, `shortWeightInitialX18` / `shortWeightMaintenanceX18` are never returned. The `CoreRisk` struct field is named `longWeight` and is always populated with the long weight.

**Downstream error — `Clearinghouse.getHealth` lines 125–135:**

```solidity
int128 existingWeight = (spotCoreRisk.longWeight +
    perpCoreRisk.longWeight) / 2;          // perpCoreRisk.longWeight is always longWeight
int128 spreadWeight = RiskHelper._getSpreadWeightX18(...);

health += basisAmount
    .mul(spotCoreRisk.price + perpCoreRisk.price)
    .mul(spreadWeight - existingWeight);   // inflated for short perp
```

`getHealthContribution` (via `_calculateProductHealth`) correctly uses the actual `amount` to select the weight, so for a short perp it applies `shortWeightInitialX18`. The spread adjustment is supposed to subtract out the weight already applied (`existingWeight`) and add back the spread-discounted weight. But because `existingWeight` uses `longWeightInitialX18` for the perp leg instead of `shortWeightInitialX18`, it is too low, and `spreadWeight - existingWeight` is too large.

---

### Impact Explanation

Concrete numbers with typical perp risk parameters:

| Parameter | Value |
|---|---|
| `spotLongWeightInitialX18` | 0.9 |
| `perpLongWeightInitialX18` | 0.9 |
| `perpShortWeightInitialX18` | 1.1 |
| `basisAmount` | 100 units |
| `spotPrice + perpPrice` | 4000 USDC |

**Actual (buggy) computation:**
- `existingWeight = (0.9 + 0.9) / 2 = 0.90`
- `spreadWeight = 1 − (1 − 0.9)/5 = 0.98`
- `spreadWeight − existingWeight = +0.08`
- `health += 100 × 4000 × 0.08 = +32,000 USDC`

**Correct computation** (using `perpShortWeight` for the short leg):
- `existingWeight = (0.9 + 1.1) / 2 = 1.00`
- `spreadWeight = 1 − (1 − 1.1)/5 = 1.02`, capped at `0.99`
- `spreadWeight − existingWeight = −0.01`
- `health += 100 × 4000 × (−0.01) = −4,000 USDC`

**Overestimation: 36,000 USDC** on a 100-unit basis position at these prices. An attacker can exploit this to pass `INITIAL` health checks with substantially less collateral than required, leaving the protocol undercollateralized.

---

### Likelihood Explanation

The long-spot / short-perp basis trade is a standard, publicly accessible position type. Any user can open it via normal order flow through the Endpoint. The spread pairs are configured in the `spreads` storage variable and are always active. No special privileges are required. The overestimation scales linearly with position size, so a large basis position produces a proportionally large health inflation.

---

### Recommendation

In `BaseEngine.getCoreRisk`, pass the actual `amount` (not the hardcoded `1`) to `_getWeightX18`:

```solidity
function getCoreRisk(...) external returns (IProductEngine.CoreRisk memory) {
    RiskHelper.Risk memory risk = _risk(productId);
    (int128 amount, ) = _getBalance(productId, subaccount);
    return IProductEngine.CoreRisk(
        amount,
        risk.priceX18,
-       RiskHelper._getWeightX18(risk, 1, healthType)
+       RiskHelper._getWeightX18(risk, amount, healthType)
    );
}
```

This ensures `CoreRisk.longWeight` reflects the direction-correct weight (long weight for long positions, short weight for short positions), so `existingWeight` in `Clearinghouse.getHealth` accurately represents the weight already applied by `getHealthContribution`.

---

### Proof of Concept

1. Deploy the protocol with a spread pair `(spotId=1, perpId=2)`.
2. Configure perp risk: `longWeightInitial = 0.9e9`, `shortWeightInitial = 1.1e9`.
3. Open subaccount with: long 100 spot, short 100 perp, minimal USDC collateral.
4. Call `clearinghouse.getHealth(subaccount, INITIAL)`.
5. Observe the spread loop adds `+32,000 USDC` to health (buggy path).
6. Compute correct health manually using `shortWeightInitialX18` for the perp leg: spread term is `−4,000 USDC`.
7. Assert `getHealth(INITIAL) > correctHealth` — the subaccount passes the initial health check despite being undercollateralized by ~36,000 USDC.

    function _getWeightX18(
        Risk memory risk,
        int128 amount,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }

        int128 weight;
        if (amount >= 0) {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.longWeightInitialX18
                : risk.longWeightMaintenanceX18;
        } else {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.shortWeightInitialX18
                : risk.shortWeightMaintenanceX18;
        }

        return weight;
    }
```

    function _getSpreadWeightX18(
        IProductEngine.CoreRisk memory perpCoreRisk,
        IProductEngine.CoreRisk memory spotCoreRisk,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }
        int128 spreadWeight;
        if (spotCoreRisk.amount > 0) {
            spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
        } else {
            spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;
        }
        int128 maxSpreadWeight;
        if (healthType == IProductEngine.HealthType.INITIAL) {
            maxSpreadWeight = ONE - ONE / 100; // 0.99
        } else {
            maxSpreadWeight = ONE - (ONE * 6) / 1000; // 0.994
        }
        if (spreadWeight > maxSpreadWeight) {
            spreadWeight = maxSpreadWeight;
        }
        return spreadWeight;
    }
```
