### Title
Missing Slippage Protection in `burnNlp` and `mintNlp` Allows Sequencer-Provided Oracle Price to Silently Reduce User Output - (File: core/contracts/Clearinghouse.sol)

---

### Summary

`burnNlp` and `mintNlp` in `Clearinghouse.sol` lack any minimum-output protection. The user signs only `{sender, nlpAmount, nonce}` (for burn) or `{sender, quoteAmount, nonce}` (for mint), but the `oraclePriceX18` that determines the actual output is **not covered by the user's signature** — it is appended by the sequencer at execution time. If the NLP oracle price moves between when the user signs and when the sequencer processes the transaction, the user silently receives a materially worse outcome with no on-chain recourse.

---

### Finding Description

In `Verifier.sol`, the EIP-712 digest for `BurnNlp` covers only the inner struct fields:

```solidity
digest = keccak256(abi.encode(
    keccak256(bytes(BURN_NLP_SIGNATURE)),
    signedTx.tx.sender,
    signedTx.tx.nlpAmount,
    signedTx.tx.nonce
));
``` [1](#0-0) 

`oraclePriceX18` is **not** included in the digest. It lives only in the outer `SignedBurnNlp` wrapper, which the sequencer constructs:

```solidity
struct SignedBurnNlp {
    BurnNlp tx;       // ← user signs only this
    bytes signature;
    int128 oraclePriceX18;   // ← sequencer-supplied, not signed
    int128[] nlpPoolRebalanceX18;
}
``` [2](#0-1) 

In `Clearinghouse.burnNlp`, the quote returned to the user is computed entirely from this unsigned price:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [3](#0-2) 

There is no `minQuoteAmount` field in `BurnNlp` and no on-chain check that `quoteAmount >= userExpectedMinimum`.

The same structural problem exists for `mintNlp`: the user signs `{sender, quoteAmount, nonce}` but the NLP shares minted are `quoteAmount / oraclePriceX18`, where `oraclePriceX18` is again sequencer-supplied and unsigned. [4](#0-3) [5](#0-4) 

The sequencer submits the transaction via `EndpointTx`, which passes `signedTx.oraclePriceX18` directly to `clearinghouse.burnNlp` / `clearinghouse.mintNlp` without any bound check against a user-specified tolerance. [6](#0-5) 

---

### Impact Explanation

A user who signs a `BurnNlp` for `X` NLP shares expecting to receive `Y` quote (based on the price at signing time) may receive materially less if the NLP oracle price has declined by the time the sequencer processes the transaction. The NLP price is expected to fluctuate because the NLP pool bears risk from the protocol's trading activity. There is no on-chain mechanism for the user to reject execution at an unfavorable price. The corrupted state delta is `spotEngine.balance[QUOTE_PRODUCT_ID][txn.sender]` — it is credited with a lower-than-expected `quoteAmount` with no revert path.

**Impact: Medium**

---

### Likelihood Explanation

The NLP price (`oraclePriceX18`) is a live market price that changes continuously as the NLP pool absorbs PnL from trading. Any non-trivial delay between a user signing a `BurnNlp` and the sequencer processing it (e.g., during high-volatility periods, sequencer queue backlog, or the mandatory 4-day lock period expiry window) creates a window where the price can move adversarially. No special privileges are required — any user interacting with the NLP burn/mint flow is exposed.

**Likelihood: Medium**

---

### Recommendation

1. Add a `minQuoteAmount` field to the `BurnNlp` struct (user-signed and included in the EIP-712 digest), and enforce `require(quoteAmount >= txn.minQuoteAmount)` in `burnNlp`.
2. Add a `minNlpAmount` field to the `MintNlp` struct (user-signed and included in the EIP-712 digest), and enforce `require(nlpAmount >= txn.minNlpAmount)` in `mintNlp`.
3. Optionally, add a `deadline` field to both structs so users can bound the time window in which their signed intent remains valid.

---

### Proof of Concept

1. NLP oracle price is `$1.00` at signing time. User signs `BurnNlp{nlpAmount: 1000e18, nonce: N}` expecting ~`$1000` quote back.
2. Before the sequencer processes the transaction, the NLP pool suffers losses and the oracle price drops to `$0.80`.
3. Sequencer submits `SignedBurnNlp{tx: {nlpAmount: 1000e18, nonce: N}, oraclePriceX18: 0.80e18, ...}`.
4. `validateSignedTx` passes — the user's signature is valid because `oraclePriceX18` is not in the digest.
5. `burnNlp` computes `quoteAmount = 1000e18 * 0.80e18 / 1e18 - burnFee ≈ 799e18`.
6. User receives `~$799` instead of `~$999` with no on-chain protection and no ability to revert. [7](#0-6) [1](#0-0)

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
