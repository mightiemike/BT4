Audit Report

## Title
`SwapAllowlistExtension` Gates the Router Address Instead of the Originating User, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which `MetricOmmPool.swap()` sets to its own `msg.sender`. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the originating user. The allowlist therefore gates the router address rather than the actual swapper. Any user can bypass a curated pool's swap allowlist by routing through the public router once the admin allowlists the router to enable any legitimate router-mediated swap.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender`:**

`MetricOmmPool.swap()` at lines 230–240 calls:
```solidity
_beforeSwap(
    msg.sender,   // direct caller of pool.swap()
    recipient,
    ...
);
```

**Step 2 — `ExtensionCalling._beforeSwap` forwards `sender` unchanged:**

Lines 160–176 of `ExtensionCalling.sol` encode `sender` directly into the extension call with no transformation.

**Step 3 — Extension checks `sender` against the allowlist:**

`SwapAllowlistExtension.beforeSwap` at lines 37–39:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool (correct), and `sender` is whoever called `pool.swap()`.

**Step 4 — Router is the direct caller of `pool.swap()`:**

`MetricOmmSimpleRouter.exactInputSingle()` at lines 72–80:
```solidity
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    ...
    params.extensionData
);
```
The originating user's address (`msg.sender` at the router level) is stored only in transient callback context for payment purposes — it is never forwarded to the pool as the `sender`. At the pool, `msg.sender` is the router.

**Why existing checks fail:**

The extension has no mechanism to distinguish between a direct pool caller and a router-mediated call. The `sender` field is structurally bound to `msg.sender` at the pool, which is always the immediate caller. There is no `swapper` parameter in `pool.swap()` analogous to the `owner` parameter in `pool.addLiquidity()` that `DepositAllowlistExtension` correctly uses.

**Two exploitable outcomes:**

1. **Bypass path**: The pool admin must allowlist the router for any router-mediated swap to succeed. Once the router is allowlisted (`allowedSwapper[pool][router] = true`), every user — including those the allowlist was meant to exclude — can swap by routing through `MetricOmmSimpleRouter`.

2. **Lockout path**: If the admin does not allowlist the router, every allowlisted user is silently blocked from using the router, forcing direct pool calls and losing slippage protection, deadline enforcement, and multi-hop routing.

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to KYC'd counterparties, whitelisted market makers, or protocol-controlled addresses loses that restriction entirely for any user who routes through the public `MetricOmmSimpleRouter`. The allowlist provides no protection on the router path. LP funds in the pool are exposed to swaps from actors the pool admin explicitly intended to exclude. This is a direct admin-boundary break: the pool admin's access control is bypassed by an unprivileged path (calling the public router). This meets the "Admin-boundary break: pool admin exceeds caps, bypasses timelocks, or factory/oracle role checks are bypassed by an unprivileged path" criterion.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard periphery entry point for swaps. Any user aware of the allowlist restriction can trivially bypass it by calling the router instead of the pool directly. No special privileges, flash loans, or multi-step setup are required. The bypass is reachable on every swap through the router once the admin has allowlisted the router (which is a necessary operational step for any allowlisted user to use the router).

## Recommendation

The allowlist must gate the economically relevant actor, not the immediate caller of `pool.swap()`.

**Option A — Pass the originating user explicitly.** Add a `swapper` parameter to `pool.swap()` (analogous to `owner` in `addLiquidity`) that the router populates with `msg.sender` before forwarding to the pool. The extension then checks that field.

**Option B — Allowlist the router and enforce identity inside the router.** Gate the router itself with an on-router allowlist that verifies `msg.sender` before forwarding to the pool. This keeps the pool interface unchanged but requires the router to be trusted and non-upgradeable.

**Option C — Check `recipient` as a proxy.** If the pool's design cannot change, the extension can check `recipient` instead of `sender`. This is imperfect but closer to the economic actor for single-hop swaps.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is the only allowed swapper
  allowedSwapper[pool][router] = true  // admin must allowlist router so alice can use it

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: bob, ...})

  router calls:
    pool.swap(bob, zeroForOne, amount, limit, "", extensionData)
    // msg.sender inside pool = router

  pool calls _beforeSwap(sender=router, ...)

  extension checks:
    allowedSwapper[pool][router] == true → passes

  bob's swap executes despite not being on the allowlist

Lockout alternative (admin does NOT allowlist router):
  alice calls router.exactInputSingle(...)
  extension checks allowedSwapper[pool][router] == false → revert
  alice cannot use the router despite being allowlisted
```

Foundry test plan: deploy pool with `SwapAllowlistExtension`, set `allowedSwapper[pool][alice] = true` and `allowedSwapper[pool][router] = true`, then call `router.exactInputSingle` as `bob` (not allowlisted) and assert the swap succeeds.
