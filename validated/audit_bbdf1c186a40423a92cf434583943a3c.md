## Title
`_validateDrawdown` off-by-one allows `drawdownE6 = E6`, zeroing `floorMultiplier` and silently disabling the stop-loss guard — (`File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`_validateDrawdown` uses a strict `>` comparison, so `drawdownE6 = 1e6` passes validation. This stores a non-zero drawdown that appears active but produces `floorMultiplier = E6 − 1e6 = 0` at line 234, making the breach condition in `_applyWatermark` (`metric < 0`) permanently false for any `uint256` metric. The stop-loss guard is silently disabled while the extension reports a configured, non-zero drawdown.

## Finding Description

**Root cause — `_validateDrawdown` (lines 305–307):**
```solidity
function _validateDrawdown(uint256 drawdownE6) private pure {
    if (drawdownE6 > E6) revert OracleStopLossDrawdownTooLarge(drawdownE6);
}
```
`drawdownE6 = 1e6` satisfies `1e6 > 1e6 == false`, so no revert occurs.

**Floor computation (line 234):**
```solidity
uint256 floorMultiplier = E6 - drawdown;  // 1e6 - 1e6 = 0
```

**Breach condition (lines 333–334):**
```solidity
if (metric >= hwm) return (metric, false);
breached = metric < (hwm * floorMultiplier) / E6;  // metric < 0 → always false
```
For any `uint256 metric`, `metric < 0` is impossible, so `breached` is always `false`. Neither the `zeroForOne` branch (line 271) nor the `!zeroForOne` branch (line 276) ever reverts.

**Existing guard bypassed (line 217):**
```solidity
if (drawdown == 0) return;
```
This explicit disable path is not reached when `drawdown = 1e6`; the function proceeds, computes `floorMultiplier = 0`, and silently disables the guard through arithmetic rather than an early return.

**Pool admin update path (lines 103–120):** `proposeOracleStopLossDrawdown(pool_, 1e6)` passes `_validateDrawdown`, stores the value after the timelock via `executeOracleStopLossDrawdown`, and overwrites `oracleStopLossConfig[pool_].drawdownE6` with `1e6`. This is a fund-impacting cap failure: the validation is the sole cap preventing a zero-floor configuration, and it is broken by the off-by-one.

The same structural defect exists in `_validateDecay` (line 309–311): `decayPerSecondE8 = 1e8` passes, and `_decayed` returns `0` after one second (`factor = 1e8 * 1 >= E8`), collapsing watermarks to zero and making `metric >= 0` trivially true.

## Impact Explanation

The `OracleValueStopLossExtension` is the sole on-chain mechanism blocking value-draining swaps past the configured drawdown floor. With `floorMultiplier = 0`, every swap — including those that reduce per-share bin value to zero — passes `afterSwap` without reverting. LP token0 and token1 balances in affected bins can be fully extracted by a sequence of swaps while the guard silently reports no breach. This is direct, unbounded LP principal loss, meeting the Critical/High threshold for broken core pool functionality and direct loss of user principal.

## Likelihood Explanation

The pool admin sets `drawdownE6` at factory initialization and can update it via the timelocked propose/execute flow. `1e6` (100%) is the natural "maximum" value a developer reaches for when testing permissive configurations or intending a wide tolerance floor. The validation passes it without error or warning. Because `drawdownE6 = 0` is the documented disable path, an admin who sets `1e6` expecting a "100% tolerance" configuration has no on-chain signal that the guard is broken. The timelock on subsequent corrections extends the exposure window by days.

## Recommendation

Change both validation helpers to `>=` so boundary values are rejected:

```solidity
function _validateDrawdown(uint256 drawdownE6) private pure {
    if (drawdownE6 >= E6) revert OracleStopLossDrawdownTooLarge(drawdownE6);
}

function _validateDecay(uint256 decayPerSecondE8) private pure {
    if (decayPerSecondE8 >= E8) revert OracleStopLossDecayTooLarge(decayPerSecondE8);
}
```

Additionally, add an explicit invariant check in `_afterSwapOracleStopLoss` or `_checkAndUpdateWatermarks`: revert if `floorMultiplier == 0` and `drawdown != 0`, to catch any future path that produces a zero floor from a non-zero drawdown.
