### Title
Linked Signer Can Hijack Subaccount Signing Authority via Self-Authorized `LinkSigner` Replacement — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This means the currently registered linked signer — a delegated session key — can unilaterally sign a new `LinkSigner` transaction to replace itself with an attacker-controlled address. A compromised linked signer can permanently escalate to full subaccount signing authority, enabling withdrawal, liquidation, and transfer of all collateral.

---

### Finding Description

When the sequencer submits a `LinkSigner` transaction in the fast-mode path, `processTransactionImpl` calls:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // allowLinkedSigner = true
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` routes through `validateSignature`, which passes the current `linkedSigner` as an accepted signer to `verifier.validateSignature`:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` accepts the signature if it recovers to either the subaccount owner address **or** the `linkedSigner`:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

The result: the existing linked signer has the authority to authorize a `LinkSigner` transaction that replaces itself with any arbitrary address. This is a meta-privilege that should be exclusive to the subaccount owner (the primary key), not the delegated session key.

By contrast, the slow-mode `LinkSigner` path correctly enforces that only the actual `msg.sender` (the subaccount owner) can submit the transaction:

```solidity
validateSender(txn.sender, sender);
``` [4](#0-3) 

The fast-mode path has no equivalent restriction.

---

### Impact Explanation

Once a linked signer is replaced with an attacker-controlled address, the attacker's key is accepted as a valid signer for all subsequent user-signed transaction types: `WithdrawCollateral`, `WithdrawCollateralV2`, `LiquidateSubaccount`, `TransferQuote`, `MintNlp`, `BurnNlp`, and future `LinkSigner` updates. The attacker can drain the subaccount's collateral entirely. The original owner cannot recover control through the fast path because the attacker can continuously re-link to new addresses before the owner's slow-mode revocation executes (3-day delay). [5](#0-4) 

---

### Likelihood Explanation

Linked signers are explicitly designed as session keys for browser or hot-wallet environments, which are materially more exposed to compromise than a user's primary hardware wallet. Any XSS, private key leak, or malicious dApp interaction that exposes the linked signer's key is sufficient to trigger this attack. No admin access, sequencer compromise, or governance capture is required — only a compromised session key that the protocol itself encourages users to create.

---

### Recommendation

`LinkSigner` is a privileged operation that modifies signing authority. It must only be authorized by the subaccount owner (primary key), never by the linked signer itself. Change the `allowLinkedSigner` flag to `false` for `LinkSigner` processing in the fast-mode path:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // must be signed by the primary key only
);
``` [6](#0-5) 

This aligns the fast-mode path with the slow-mode path, which already enforces owner-only authorization for `LinkSigner`.

---

### Proof of Concept

1. Alice sets linked signer A (a browser session key) on her subaccount via a legitimate `LinkSigner` transaction.
2. Attacker compromises signer A (e.g., via XSS or key exfiltration).
3. Attacker uses signer A to sign a new `LinkSigner` transaction: `{ sender: Alice's subaccount, signer: attacker_address, nonce: current_nonce }`.
4. The sequencer submits this transaction. `validateSignedTx` accepts it because `recovered == linkedSigner` (signer A).
5. `linkedSigners[Alice's subaccount]` is now set to `attacker_address`.
6. Attacker signs a `WithdrawCollateral` transaction from `attacker_address`, which passes `validateSignature` as the new linked signer.
7. Alice's collateral is withdrawn to the attacker. Alice's only recourse is a slow-mode `LinkSigner` revocation, which takes 3 days — during which the attacker can drain all funds. [7](#0-6)
