Audit Report

## Title
SwapAllowlistExtension allowlist bypassed via MetricOmmSimpleRouter router identity substitution â€” (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against a per-pool allowlist, where `sender` is `msg.sender` of the pool's `swap()` call. When `MetricOmmSimpleRouter` intermediates a swap, the pool's `msg.sender` is the router contract address, not the end user. If the pool admin allowlists the router to enable legitimate router-mediated swaps, every non-allowlisted user can bypass the restriction by calling the same public router, completely defeating the access control.

## Finding Description
**Root cause â€” wrong identity forwarded to the extension hook:**

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first argument of `IMetricOmmExtensions.beforeSwap`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks that `sender` against the allowlist using `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the address passed in: [3](#0-2) 

**How the router breaks the identity:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, so the pool sees `msg.sender = router`: [4](#0-3) 

The same applies to `exactInput` (multi-hop), where the router is always the direct caller of each pool: [5](#0-4) 

And to `exactOutputSingle` and `exactOutput`, which also call `pool.swap()` directly from the router context. [6](#0-5) 

**The forced dilemma for the pool admin:**

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router at all â€” broken functionality |
| **Allowlist the router** | Every non-allowlisted user can bypass the gate by routing through the router |

There is no configuration that simultaneously allows legitimate users to use the router and blocks non-allowlisted users. The exact wrong value is `allowedSwapper[pool][router] = true`, which the extension evaluates as a pass for any caller who routes through the router.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` (e.g., a KYC-gated or institutional-only pool) is completely open to any public user who calls `MetricOmmSimpleRouter`. The attacker pays no special cost and needs no privileged access. Unauthorized swaps can drain pool liquidity at oracle-derived prices the pool admin did not intend to expose to the public, cause unexpected price impact, and violate regulatory or contractual access controls the pool admin relied on. This constitutes a broken core pool invariant (access-controlled swap) and a direct loss of LP assets â€” a High severity impact under Sherlock thresholds.

## Likelihood Explanation
Likelihood is high. The router is a public, permissionless contract â€” any EOA or contract can call it. The bypass requires zero privileged access; the attacker only needs to know the pool address and use the standard router interface. The only precondition is that the pool admin has allowlisted the router, which is the expected and necessary operational configuration to enable legitimate router-mediated swaps for allowlisted users.

## Recommendation
The extension must gate on the economic actor (the end user), not the intermediary. Two viable approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData` for each hop; the extension decodes and checks it. This requires a coordinated convention between router and extension.
2. **Router-aware identity field**: Add an explicit `swapper` field to the router's swap params that is forwarded in `extensionData`; the extension reads it only when `msg.sender` (the pool) reports a known router as `sender`. The pool admin allowlists individual users, not the router.

Until fixed, pool admins should **not** allowlist the router address in `SwapAllowlistExtension` and should document that allowlisted users must call the pool directly.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` registered as a `beforeSwap` hook.
2. Admin calls `setAllowedToSwap(pool, alice, true)` â€” only Alice may swap.
3. Admin calls `setAllowedToSwap(pool, router, true)` â€” so Alice can use the router.
4. Mallory (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. The router calls `pool.swap(...)` â€” `msg.sender` inside the pool is the router address.
6. `_beforeSwap` passes `sender = router` to `SwapAllowlistExtension.beforeSwap`.
7. The extension evaluates `allowedSwapper[pool][router]` â†’ `true` â†’ **no revert**.
8. Mallory's swap executes on the restricted pool, bypassing the allowlist entirely.

Foundry test outline:
