### Title
Configured `bouncer_config` silently ignored in `BaseBlockInfo::new()` ŌĆö hardcoded `BouncerConfig::default()` used for virtual-block execution - (File: crates/starknet_transaction_prover/src/running/virtual_block_executor.rs)

### Summary

`RpcVirtualBlockExecutorConfig` exposes a configurable `bouncer_config` field that operators can set via CLI or config file. `BaseBlockInfo::new()` accepts `use_latest_versioned_constants` as a parameter and correctly threads it through, but it **never accepts or uses the configured `bouncer_config`**. Instead it unconditionally passes `BouncerConfig::default()` to `BlockContext::new()`. The configured value is silently discarded, exactly mirroring the external bug where a setter updates a field that is never read.

### Finding Description

`RpcVirtualBlockExecutorConfig` declares `bouncer_config` as a first-class, operator-visible field:

```rust
pub(crate) struct RpcVirtualBlockExecutorConfig {
    pub(crate) bouncer_config: BouncerConfig,   // configurable
    pub(crate) use_latest_versioned_constants: bool,
    ...
}
``` [1](#0-0) 

The server config layer loads this value from a JSON file or an embedded resource:

```rust
let bouncer_config: BouncerConfig = if let Some(path) = args.bouncer_config_override {
    serde_json::from_str(&contents)...
} else {
    serde_json::from_str(include_str!("../../resources/bouncer_config.json"))...
};
``` [2](#0-1) 

`BaseBlockInfo::new()` receives `use_latest_versioned_constants` as a parameter and branches on it correctly, but the `bouncer_config` is never passed in. The `BlockContext` is always constructed with the hardcoded default:

```rust
let block_context = BlockContext::new(
    block_info,
    chain_info,
    versioned_constants.clone(),
    BouncerConfig::default(),   // ŌåÉ always default, never the configured value
);
``` [3](#0-2) 

The `BouncerConfig` drives two execution-critical quantities:

1. **Block capacity limits** (`block_max_capacity`) ŌĆö controls whether a transaction is accepted or rejected with `TransactionTooLarge`.
2. **Per-builtin proving-gas costs** via `builtin_gas_costs()`:

```rust
pub fn builtin_gas_costs(&self) -> BuiltinGasCosts {
    self.builtin_instance_limits.induced_gas_costs(self.block_max_capacity.proving_gas)
}
``` [4](#0-3) 

These per-builtin costs feed directly into transaction gas metering during execution. Using `BouncerConfig::default()` (which derives `proving_gas` from zero-initialized weights) instead of the operator-configured value produces wrong builtin gas costs for every transaction executed in the virtual block.

### Impact Explanation

The virtual block executor is the proving path: it re-executes transactions to generate a STARK proof whose `proof_fact` is embedded in the on-chain Invoke V3 transaction. Wrong builtin gas costs mean:

- **Incorrect fee/gas accounting** for every transaction in the virtual block ŌĆö the gas consumed reported to the OS diverges from what the configured bouncer would produce.
- **Wrong resource bounds enforcement** ŌĆö transactions that exceed the configured capacity may be accepted (or vice versa), producing a proof over an execution that the sequencer's real bouncer would have rejected.
- The resulting `proof_fact` hash commits to execution output computed under the wrong gas regime, which can cause the proof to be rejected or, worse, accepted with incorrect state.

This falls squarely within: *Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact.*

### Likelihood Explanation

The `bouncer_config.json` embedded resource is explicitly distinct from `BouncerConfig::default()` (the comment notes "embedded defaults set those caps high" for the virtual-OS path). Any deployment that loads a non-default `bouncer_config.json` ŌĆö which is the normal production path ŌĆö will silently execute every virtual block under the wrong bouncer, with no error or log message indicating the discrepancy.

### Recommendation

Add `bouncer_config: BouncerConfig` as a parameter to `BaseBlockInfo::new()` and pass it to `BlockContext::new()` instead of `BouncerConfig::default()`. The call site in `RpcVirtualBlockExecutor` should forward `self.config.bouncer_config.clone()`.

### Proof of Concept

The existing (ignored) integration test already encodes the expected invariant:

```rust
executor.config.bouncer_config = bouncer_config; // n_txs = 0
let error = match executor.execute(...) {
    Err(error) => error,
    Ok(_) => panic!("Execution should fail when bouncer capacity is zero"),
};
assert_matches!(error, VirtualBlockExecutorError::TransactionExecutionError(msg)
    if msg.contains("Transaction size exceeds the maximum block capacity"));
``` [5](#0-4) 

Because `BaseBlockInfo::new()` ignores `executor.config.bouncer_config` and always uses `BouncerConfig::default()`, the transaction is executed under the default (permissive) bouncer. The test would not produce the expected `TransactionTooLarge` error ŌĆö the transaction succeeds ŌĆö demonstrating that the configured value has no effect on execution. The test is marked `#[ignore]` (requires RPC), so this regression is not caught in CI.

/// Verifies that a transaction is rejected when the bouncer config has tight capacity limits.
///
/// Sets `n_txs: 0` so that any transaction exceeds the block capacity, and asserts that
/// execution returns `TransactionExecutionError` with a "Transaction size exceeds" message.
#[rstest]
#[ignore] // Requires RPC access
fn test_execute_rejected_by_tight_bouncer_limits(
    rpc_virtual_block_executor: RpcVirtualBlockExecutor,
) {
    // Override the bouncer config with zero capacity so any transaction is too large.
    let mut executor = rpc_virtual_block_executor;
    let mut bouncer_config = BouncerConfig::max();
    bouncer_config.block_max_capacity.n_txs = 0;
    executor.config.bouncer_config = bouncer_config;

    let (tx, tx_hash) = construct_balance_of_invoke();
    let contract_class_manager = ContractClassManager::start(ContractClassManagerConfig::default());

    let error = match executor.execute(
        BlockId::Number(BlockNumber(TEST_BLOCK_NUMBER)),
        contract_class_manager,
        vec![(tx, tx_hash)],
    ) {
        Err(error) => error,
        Ok(_) => panic!("Execution should fail when bouncer capacity is zero"),
    };

    assert_matches!(
        error,
        VirtualBlockExecutorError::TransactionExecutionError(msg)
            if msg.contains("Transaction size exceeds the maximum block capacity")
    );
}
