### Title
`ENABLE_SECP_OPS` Flag Guard Bypassed via 4-Byte Opcode Path — (`File: src/chia_dialect.rs`)

### Summary

The `ChiaDialect::op()` function in `src/chia_dialect.rs` dispatches `op_secp256k1_verify` and `op_secp256r1_verify` through two separate code paths: a 1-byte opcode path (opcodes 64 and 65) that correctly checks `ClvmFlags::ENABLE_SECP_OPS`, and a 4-byte opcode path (opcodes `0x13d61f00` and `0x1c3a8f00`) that invokes the same operator functions **without** checking `ENABLE_SECP_OPS`. An attacker-controlled CLVM program using the 4-byte opcode form can execute secp signature verification regardless of whether the flag is set, breaking the invariant that `ENABLE_SECP_OPS` gates these operators.

### Finding Description

In `src/chia_dialect.rs`, the `op()` method of `ChiaDialect` has two distinct dispatch branches.

**Branch 1 — 4-byte opcodes (lines 157–182):** Handles opcodes whose atom length is 4. The secp operators are matched by their full 4-byte encoding and dispatched unconditionally: [1](#0-0) 

```rust
if op_len == 4 {
    ...
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,   // NO flag check
        0x1c3a8f00 => op_secp256r1_verify,   // NO flag check
        _ => { return unknown_operator(...); }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

**Branch 2 — 1-byte opcodes (lines 248–249):** The same operators are also reachable via single-byte opcodes 64 and 65, but here the `ENABLE_SECP_OPS` flag is correctly enforced: [2](#0-1) 

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The flag is defined as a hard-fork gate: [3](#0-2) 

The design intent is unambiguous: `ENABLE_SECP_OPS` must be set for secp operators to be active. The 4-byte path violates this invariant by dispatching to the same operator functions without the flag check.

This is a direct structural analog to the JOJO M-5 bug: `_deposit()` checks `isDepositAllowed`; `_withdraw(isInternal=true)` performs the same deposit state change but skips the check. Here, the 1-byte opcode path checks `ENABLE_SECP_OPS`; the 4-byte opcode path invokes the same operator but skips the check.

### Impact Explanation

**Consensus divergence.** A CLVM program that encodes a secp opcode as a 4-byte atom (`0x13d61f00` or `0x1c3a8f00`) will execute secp signature verification on any node, regardless of whether `ENABLE_SECP_OPS` is set. A program using the 1-byte opcode form (64 or 65) will fail with `Unimplemented` on nodes where the flag is absent. This means two syntactically different but semantically equivalent programs produce different results depending on which opcode encoding is used, breaking consensus determinism during the period before the hard-fork activates.

Concretely: a coin puzzle that uses the 4-byte secp opcode would be evaluated as valid by all nodes (flag set or not), while the same puzzle using the 1-byte opcode would be rejected by nodes without the flag. This is a split-brain consensus condition exploitable by any party who can submit CLVM programs to the mempool.

### Likelihood Explanation

The 4-byte opcode encoding is a valid, reachable CLVM atom. Any attacker who can submit a transaction to the Chia mempool can craft a CLVM program using the 4-byte secp opcode. No special privileges, compromised nodes, or social engineering are required. The trigger is purely attacker-controlled CLVM bytes.

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch branch, mirroring the guard already present for the 1-byte opcodes:

```rust
// in the op_len == 4 branch:
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures both opcode encodings are gated by the same flag, eliminating the bypass path.

### Proof of Concept

1. Construct a CLVM program that invokes secp256k1 verification using the 4-byte opcode atom `\x13\xd6\x1f\x00` as the operator.
2. Run the program with `ChiaDialect::new(ClvmFlags::empty())` (no `ENABLE_SECP_OPS`).
3. Observe: the 4-byte path dispatches to `op_secp256k1_verify` and executes successfully.
4. Run the same logical program using 1-byte opcode `64` as the operator.
5. Observe: the 1-byte path falls through to `unknown_operator` and returns `Unimplemented`.

The two encodings of the same operator produce divergent results under identical flag settings, confirming the bypass. [1](#0-0) [4](#0-3)
