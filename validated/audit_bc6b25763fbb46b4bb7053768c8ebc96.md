### Title
`verify_transaction_inclusion` Accepts Internal Merkle-Tree Nodes as Valid `tx_id`, Enabling SPV Proof Forgery — (`File: contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is a publicly callable NEAR contract method with no caller restriction. It performs no validation that the supplied `tx_id` is a real leaf-level transaction hash. An unprivileged attacker can supply an internal Merkle-tree node as `tx_id`, pair it with a structurally valid Merkle proof, and cause the function to return `true` — falsely certifying transaction inclusion to any downstream consumer.

---

### Finding Description

`verify_transaction_inclusion` is decorated only with `#[pause]` (no `#[trusted_relayer]`, no role guard). Any NEAR account can call it. [1](#0-0)

There is no validation that `args.tx_id` is a leaf-level transaction hash rather than an internal Merkle node. The function immediately passes the caller-supplied `tx_id` into the Merkle root computation: [3](#0-2) 

The contract's own documentation acknowledges this broken invariant: [4](#0-3) 

The 64-byte Merkle forgery attack (https://www.bitmex.com/blog/64-Byte-Transactions) exploits exactly this gap: an attacker concatenates two adjacent real transaction hashes (64 bytes total), hashes the result to obtain an internal node `N`, then submits `tx_id = N` with a Merkle proof that correctly walks from `N` to the stored `merkle_root`. The comparison at line 322 succeeds and the function returns `true`.

The `#[deprecated]` Rust attribute is a compile-time lint only; it imposes no on-chain restriction. The method remains fully callable in production.

---

### Impact Explanation

Any NEAR smart contract or off-chain service that calls `verify_transaction_inclusion` and acts on a `true` result — e.g., releasing bridged funds, minting wrapped tokens, or updating cross-chain state — can be deceived into accepting a fabricated transaction inclusion proof. The attacker needs no privileged role; only a valid on-chain block header (already stored by the relayer) and knowledge of two real transactions in that block.

---

### Likelihood Explanation

The 64-byte Merkle forgery is a published, well-understood attack with concrete tooling. Bitcoin block Merkle trees are public; any attacker can read them from a Bitcoin node. The only precondition is that the target block has been submitted to the light client, which is the normal operating state. Likelihood is **Medium**: the attack requires off-chain computation but no privileged access and no secret material.

---

### Recommendation

1. **Remove or gate the deprecated function.** Add `#[trusted_relayer]` or an explicit role check so only authorized callers can invoke `verify_transaction_inclusion`, or remove it from the public ABI entirely.
2. **Enforce use of `verify_transaction_inclusion_v2`** for all production consumers. The v2 path requires a coinbase Merkle proof at index 0, which structurally prevents the 64-byte forgery.
3. **Validate `tx_id` length/format** before the Merkle computation: reject any input whose byte representation could correspond to a concatenation of two 32-byte hashes (i.e., enforce that `tx_id` is a proper 32-byte leaf, not a synthetic internal node).

---

### Proof of Concept

**Attacker preconditions:** any NEAR account; a block already stored in the light client at height `H` with known transactions `TxA` and `TxB` that are adjacent leaves in the Merkle tree.

**Steps:**

1. Compute `internal_node = SHA256d(TxA_hash || TxB_hash)` — this is a real internal Merkle node.
2. Build a Merkle proof `P` that walks from `internal_node` up to `merkle_root` stored in the block header at height `H`. All intermediate hashes are real and publicly available from the Bitcoin network.
3. Call `verify_transaction_inclusion` with:
   - `tx_id = internal_node`
   - `tx_block_blockhash` = the block hash at height `H`
   - `tx_index` = the index corresponding to `internal_node`'s position in the tree
   - `merkle_proof = P`
   - `confirmations = 1`
4. The function executes `compute_root_from_merkle_proof(internal_node, index, P)`, which correctly reconstructs `merkle_root`.
5. The comparison at line 322 succeeds; the function returns `true`. [5](#0-4) 

Any downstream contract that gates an asset release on this `true` result is now exploited without the attacker ever having submitted a real transaction to Bitcoin.
