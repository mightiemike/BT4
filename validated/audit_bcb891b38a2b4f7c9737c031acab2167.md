### Title
Missing Liquidator Health Check in `_finalizeSubaccount` Enables Bad Debt Creation — (`core/contracts/ClearinghouseLiq.sol`)

### Summary
When `liquidateSubaccountImpl` is called with `productId == type(uint32).max`, `_finalizeSubaccount` settles all positive `vQuoteBalance` from the liquidatee to the liquidator by debiting the liquidator's QUOTE balance. No health check is performed on the liquidator in this code path, allowing an undercapitalized liquidator to accumulate unbounded negative QUOTE balance, creating bad debt that is socialized onto other depositors.

### Finding Description
In `liquidateSubaccountImpl`, when `_finalizeSubaccount` returns `true`, the function returns immediately at line 626 without ever calling `_handleLiquidationPayment`. [1](#0-0) 

The only liquidator solvency guard in the contract lives exclusively inside `_handleLiquidationPayment`: [2](#0-1) 

Inside `_finalizeSubaccount`, the "settle all positive pnl" loop (lines 322–338) calls `_settlePnlAgainstLiquidator` for every perp product where the liquidatee holds a positive `vQuoteBalance`: [3](#0-2) 

`_settlePnlAgainstLiquidator` unconditionally debits the liquidator's QUOTE balance by the full PnL amount: [4](#0-3) 

There is no floor, no pre-check on the liquidator's quote balance, and no post-check on the liquidator's health anywhere in `_finalizeSubaccount`.

### Impact Explanation
An undercapitalized liquidator can have their QUOTE balance driven to an arbitrarily large negative value. This negative balance represents bad debt — real USDC that was credited to the liquidatee's account but was never backed by the liquidator. The protocol's socialization mechanism (`spotEngine.socializeSubaccount`) then distributes this loss across all other depositors, directly reducing their collateral value. This is a theft/loss of depositor funds matching the Critical scope: unbacked creation of quote balances and socialization of losses.

### Likelihood Explanation
The preconditions are achievable in normal protocol operation:
1. A liquidatee accumulates large positive `vQuoteBalance` across multiple perp products (e.g., profitable long positions closed during prior liquidation steps).
2. All perp `amount` fields are zero and all non-USDC spot balances are zero — the standard state after a fully liquidated subaccount.
3. Any liquidator (no minimum capital requirement is enforced) submits `liquidateSubaccountImpl` with `productId = type(uint32).max`.

The attacker can be the liquidator themselves, deliberately using a throwaway subaccount with near-zero capital to absorb the PnL transfer and immediately leave the protocol with bad debt.

### Recommendation
Add a liquidator health check at the end of `_finalizeSubaccount` (or immediately after it returns `true` in `liquidateSubaccountImpl`) mirroring the existing guard in `_handleLiquidationPayment`:

```solidity
if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
    require(
        txn.sender == N_ACCOUNT || !isUnderInitial(txn.sender),
        ERR_SUBACCT_HEALTH
    );
    // ... existing isolated subaccount cleanup ...
    return;
}
```

This ensures the liquidator remains solvent after absorbing the liquidatee's positive PnL, consistent with the invariant enforced in all other liquidation paths.

### Proof of Concept
1. Deploy protocol with two perp products (perpId=1, perpId=2).
2. Create `liquidatee` subaccount: close all perp positions to zero `amount`, but leave `vQuoteBalance = +500_000e18` on perpId=1 and `+500_000e18` on perpId=2 (total +1M USDC of positive PnL). Ensure liquidatee's maintenance health is negative (e.g., via a negative USDC spot balance of -1).
3. Create `liquidator` subaccount with USDC spot balance = `+1e18` (1 USDC).
4. Call `liquidateSubaccountImpl({sender: liquidator, liquidatee: liquidatee, productId: type(uint32).max, ...})`.
5. `_finalizeSubaccount` runs, settles +1M USDC PnL: liquidatee's QUOTE goes from -1 to +999_999, liquidator's QUOTE goes from +1 to -999_999.
6. No revert occurs. Assert `spotEngine.getBalance(QUOTE_PRODUCT_ID, liquidator).amount < 0` — confirmed deeply negative.
7. `spotEngine.socializeSubaccount` distributes the liquidator's bad debt to all depositors. [5](#0-4)
