Audit Report

## Title
SwapAllowlistExtension checks router address instead of end user, allowing any caller to bypass per-user swap allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which is `msg.sender` of `MetricOmmPool.swap`. When `MetricOmmSimpleRouter` intermediates a swap, the pool's `msg.sender` is the router contract, not the end user. Any pool admin who allowlists the router to support curated users inadvertently opens the pool to all unprivileged callers who route through it.

## Finding Description
`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every registered extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool received as its own `msg.sender`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` stores the original `msg.sender` in transient storage for callback settlement but calls `pool.swap(...)` with no originator argument — the pool sees `msg.sender = router`: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. The extension therefore checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates on the explicit `owner` parameter (the actual depositor), not on `sender`: [5](#0-4) 

No equivalent mechanism exists in the swap path to recover the true end-user identity.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set (e.g., KYC'd counterparties) is fully bypassed by any unprivileged user who calls the router once the router is allowlisted. Unauthorized users can execute swaps against oracle-anchored prices, extract LP value, and front-run allowlisted traders — all actions the pool admin explicitly intended to prevent. This constitutes a broken core pool functionality and an admin-boundary break where an unprivileged path defeats an access control the pool admin configured.

## Likelihood Explanation
The trigger is the standard, publicly documented router interface. The only precondition is that the pool admin has allowlisted the router — a natural and expected action for any curated pool whose allowlisted users need multi-hop routing or slippage protection. No privileged access, special token behavior, or admin collusion is required. Any user who reads the router ABI can exploit this.

## Recommendation
Thread the original caller identity through the swap path so the extension can gate on it. The cleanest fix is to expose the originator (already stored in transient storage by the router for callback settlement) as an additional argument to `beforeSwap`, and have `SwapAllowlistExtension` check `allowedSwapper[pool][originator]` instead of `allowedSwapper[pool][sender]`. A partial alternative is to key the allowlist on `recipient` instead of `sender`, but this does not cover cases where the recipient differs from the intended gated actor.

## Proof of Concept
```
Setup
─────
1. Pool admin deploys a pool with SwapAllowlistExtension as beforeSwap hook.
2. Admin allowlists Alice:  setAllowedToSwap(pool, alice, true)
3. Admin allowlists the router: setAllowedToSwap(pool, router, true)
   (required so Alice can use the router)

Attack
──────
4. Bob (not allowlisted) calls:
     MetricOmmSimpleRouter.exactInputSingle({
       pool: pool,
       recipient: bob,
       ...
     })
5. Router calls pool.swap(...) — pool sees msg.sender = router.
6. Extension checks allowedSwapper[pool][router] → true → swap proceeds.
7. Bob receives tokens from the curated pool with no allowlist check on his address.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds
```
