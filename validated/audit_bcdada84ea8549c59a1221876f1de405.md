Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks direct caller (`sender`) instead of economic actor, allowing any user to bypass per-user swap allowlist via allowlisted router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender` — the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` is used, `sender` becomes the router address. If the router is allowlisted (required for any user to use it), every user passes the guard regardless of individual allowlist status. `DepositAllowlistExtension` avoids this by checking `owner` (the economic actor), but `SwapAllowlistExtension` has no equivalent binding.

## Finding Description

**`MetricOmmPool.swap` passes `msg.sender` as `sender` to the extension:** [1](#0-0) 

`_beforeSwap` receives `msg.sender` as its first argument. When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)`, `msg.sender` inside the pool is the router address.

**`SwapAllowlistExtension.beforeSwap` checks `sender` (the router), not the end-user:** [2](#0-1) 

The check `allowedSwapper[msg.sender][sender]` evaluates `allowedSwapper[pool][router]`. If the router is allowlisted, this returns `true` for every caller of the router, regardless of who they are.

**`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` with `params.recipient` as the recipient (the user's address), but the pool receives the router as `msg.sender`:** [3](#0-2) 

**`DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` (second argument, the economic actor), ignoring `sender`:** [4](#0-3) 

`SwapAllowlistExtension` has no equivalent — it only sees the direct caller. The admin faces an impossible choice: not allowlisting the router blocks all router-based swaps for legitimate users; allowlisting the router opens the pool to every user.

## Impact Explanation
A pool admin who deploys a curated pool with `SwapAllowlistExtension` to restrict trading to specific addresses (e.g., KYC'd counterparties, institutional LPs, whitelisted market makers) cannot enforce that restriction when the router is in use. Any unprivileged user can call `router.exactInputSingle()` and trade in the restricted pool, bypassing the intended access control. This breaks the core curation guarantee of the extension and constitutes broken core pool functionality — the access control invariant (`allowedSwapper[pool][user]` must be `true` for a user to swap) is violated for every router-mediated swap.

## Likelihood Explanation
The router is the primary user-facing entry point for swaps. Any pool admin who wants allowlisted users to be able to use the router must allowlist the router address. This is the expected operational configuration, making the bypass reachable in every realistic curated-pool deployment that uses the router. No special attacker capability is required — a standard `exactInputSingle` call suffices.

## Recommendation
Change `SwapAllowlistExtension.beforeSwap` to check `recipient` (the address receiving output tokens, which the router sets to the actual user) rather than `sender`:

```solidity
// Before (wrong actor):
function beforeSwap(address sender, address, ...)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {

// After (economic actor):
function beforeSwap(address, address recipient, ...)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
```

This mirrors the pattern used by `DepositAllowlistExtension`, which correctly gates by `owner` rather than the direct caller. Note that for multi-hop `exactInput`, intermediate hops pass `address(this)` (the router) as `recipient`; the router would need to be allowlisted for intermediate hops, or the extension could decode the originating user from `extensionData` for a fully robust fix.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is allowed to swap.
3. Admin calls `setAllowedToSwap(pool, router, true)` — router is allowlisted so Alice can use it.
4. Bob (not allowlisted) calls `router.exactInputSingle({..., recipient: bob})`.
5. Router calls `pool.swap(recipient=bob, ...)` — pool sees `msg.sender = router`.
6. Pool calls `extension.beforeSwap(sender=router, recipient=bob, ...)`.
7. Extension evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds.
8. Bob successfully swaps in the restricted pool, bypassing the per-user allowlist.

Foundry test: deploy pool with `SwapAllowlistExtension`, allowlist only Alice and the router, call `exactInputSingle` from Bob's address with `recipient=bob`, assert the swap succeeds (demonstrating the bypass) and that a direct `pool.swap` call from Bob reverts.
