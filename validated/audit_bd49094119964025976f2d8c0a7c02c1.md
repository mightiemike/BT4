### Title
`get_sortable_validator_online_ratio` ignores `endorsement_cutoff_threshold` while `get_validator_online_ratio` (used for rewards) respects it, producing divergent uptime values for the same validator — (`chain/epoch-manager/src/validator_stats.rs`)

---

### Summary

The kickout-exemption sort in `compute_validators_to_reward_and_kickout` calls `get_sortable_validator_online_ratio`, which hard-codes `endorsement_cutoff_threshold = None`. The reward calculation in `RewardCalculator::calculate_reward` calls `get_validator_online_ratio` with the epoch-config-supplied `endorsement_cutoff_threshold` (which can be `Some(n)`). When the threshold is active, the two call sites compute a **different uptime ratio for the same validator**, so the validator that is sorted as "highest online ratio" (and therefore exempted from kickout) is not the same validator that receives the highest reward — a direct analog of the Balancer rounding-direction divergence between two consumers of the same invariant.

---

### Finding Description

`get_sortable_validator_online_ratio` is the sole function used to rank validators for the `validator_max_kickout_stake_perc` exemption gate:

```rust
// chain/epoch-manager/src/lib.rs  line 500-502
let mut sorted_validators = validator_block_chunk_stats
    .iter()
    .map(|(account, stats)| (get_sortable_validator_online_ratio(stats), account))
    .collect_vec();
```

Its implementation unconditionally passes `None` for the cutoff:

```rust
// chain/epoch-manager/src/validator_stats.rs  line 110-111
pub(crate) fn get_sortable_validator_online_ratio(stats: &BlockChunkValidatorStats) -> BigRational {
    let ratio = get_validator_online_ratio(stats, None);   // ← always None
```

Meanwhile, `RewardCalculator::calculate_reward` passes the epoch-config value:

```rust
// chain/epoch-manager/src/reward_calculator.rs  line 95-96
let production_ratio =
    get_validator_online_ratio(&stats, online_thresholds.endorsement_cutoff_threshold);
```

`endorsement_cutoff_threshold` is a field of `ValidatorOnlineThresholds` that can be `Some(n)` (e.g. `Some(50)` in tests). When it is set, `get_endorsement_ratio` replaces the raw `produced/expected` ratio with a binary `0/1` signal:

```rust
// chain/epoch-manager/src/validator_stats.rs  line 128-129
} else if let Some(threshold) = cutoff_threshold {
    if stats.less_than(threshold) { (0, 1) } else { (1, 1) }
```

A validator whose endorsement ratio is, say, 46 % (below a 50 % cutoff) gets:
- **Reward path**: endorsement component = 0 → lower composite uptime → lower reward, possibly zero.
- **Kickout-sort path**: endorsement component = 46/100 → higher composite uptime → ranked higher → more likely to be exempted from kickout.

The two paths therefore disagree on which validators are "most online", exactly as Balancer's old `StableMath` (rounding up) and the new Composable Pool (rounding down) disagreed on the invariant value.

---

### Impact Explanation

**High.** The kickout-exemption mechanism is the last safety valve that prevents the network from ejecting too many validators at once. When `endorsement_cutoff_threshold` is active:

1. A validator with a sub-threshold endorsement ratio is sorted *higher* than it deserves (because the sort ignores the cutoff), so it consumes one of the limited exemption slots.
2. A validator with a super-threshold endorsement ratio that should be exempted may instead be kicked out.
3. The wrong set of validators is retained for the next epoch, undermining the economic security guarantee that `validator_max_kickout_stake_perc` is meant to provide.

Because `compute_validators_to_reward_and_kickout` is called at every epoch boundary and its output is written into `EpochInfo` (which is consensus-critical), the divergence affects on-chain state deterministically and reproducibly.

---

### Likelihood Explanation

**Medium.** The divergence is latent until `endorsement_cutoff_threshold` is set to a non-`None` value in the epoch config. The field exists in `ValidatorOnlineThresholds` and is already exercised in tests (`Some(50)`). Once any epoch config activates it (e.g. via a protocol upgrade that sets `endorsement_cutoff_threshold`), the bug fires on every epoch boundary for every validator whose endorsement ratio straddles the cutoff.

---

### Recommendation

Pass the same `endorsement_cutoff_threshold` to `get_sortable_validator_online_ratio` that is used in the reward path, or expose the threshold as a parameter:

```rust
// chain/epoch-manager/src/validator_stats.rs
pub(crate) fn get_sortable_validator_online_ratio(
    stats: &BlockChunkValidatorStats,
    endorsement_cutoff_threshold: Option<u8>,   // ← add parameter
) -> BigRational {
    let ratio = get_validator_online_ratio(stats, endorsement_cutoff_threshold);
    ...
}
```

Then update the call site in `compute_validators_to_reward_and_kickout` to supply the same threshold that `calculate_reward` receives, ensuring both paths compute identical uptime values for every validator.

---

### Proof of Concept

Consider a validator with:
- blocks: 95/100
- chunks: 95/100
- endorsements: 46/100

Epoch config: `endorsement_cutoff_threshold = Some(50)`.

**Reward path** (`get_validator_online_ratio(stats, Some(50))`):
- endorsement component → `(0, 1)` (below cutoff)
- composite = mean(95/100, 95/100, 0/1) = (95·1·1 + 95·100·1 + 0·100·100) / (3·100·100·1) = 9595/30000 ≈ 0.3198
- Below `online_min_threshold = 0.9` → **reward = 0**

**Kickout-sort path** (`get_sortable_validator_online_ratio(stats)` → `get_validator_online_ratio(stats, None)`):
- endorsement component → `(46, 100)` (raw ratio)
- composite = mean(95/100, 95/100, 46/100) = (95·100·100 + 95·100·100 + 46·100·100) / (3·100·100·100) = 236/300 ≈ 0.787
- Ranked **above** a validator with 0 % endorsements → **exempted from kickout**

Result: the validator receives zero reward (correctly penalized) but is exempted from kickout (incorrectly protected), occupying an exemption slot that should go to a genuinely high-uptime validator. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
