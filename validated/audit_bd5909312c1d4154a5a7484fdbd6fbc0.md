### Title
Missing Deleted-Pool Check in `_applyNlpRebalance` Allows Quote to Be Permanently Stuck in Deleted NLP Pool Subaccounts — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`_applyNlpRebalance` in `Clearinghouse.sol` iterates over all NLP pools and applies quote balance updates without checking whether a pool has been deleted (`owner == address(0)`). This is the direct Nado analog of the external report's missing `isAlive[_gauge]` check: a "dead" entity (deleted NLP pool) is processed without a liveness guard, causing quote tokens to be deposited into an inaccessible subaccount and permanently locked.

---

### Finding Description

When an NLP pool is deleted via `deleteNlpPool`, the pool entry is **not removed** from the `nlpPools` array. Instead, `owner` is set to `address(0)` and `balanceWeightX18` to `0`, and all existing balances are swept to `N_ACCOUNT` via `clearNlpPoolPosition`. [1](#0-0) 

The pool slot remains in the array permanently. When a user submits a `MintNlp` or `BurnNlp` signed transaction, the full `nlpPools` storage array (including deleted entries) is passed to `Clearinghouse.mintNlp` / `burnNlp`, alongside the user-supplied `nlpPoolRebalanceX18` array. [2](#0-1) [3](#0-2) 

The on-chain validation function `_validateNlpRebalance` only checks that:
1. Array lengths match, and
2. The sum of rebalance amounts equals the delta quote amount.

It does **not** check whether any pool in the array is deleted. [4](#0-3) 

`_applyNlpRebalance` then unconditionally calls `spotEngine.updateBalance` for every pool index, including deleted ones: [5](#0-4) 

If `nlpPoolRebalanceX18[i]` is non-zero for a deleted pool index `i`, quote is deposited into that pool's subaccount. There is no subsequent mechanism to retrieve it: `nlpProfitShare` explicitly blocks access to deleted pools: [6](#0-5) 

And `clearNlpPoolPosition` is only called once at deletion time — any quote deposited after deletion is never swept. [7](#0-6) 

---

### Impact Explanation

Quote tokens deposited into a deleted NLP pool's subaccount are permanently inaccessible. Because `mintNlp` mints NLP tokens proportional to the full `quoteAmount` paid by the user, but some of that quote is routed to a dead subaccount, the NLP token supply becomes partially unbacked. This corrupts the NLP accounting invariant: the total quote backing NLP tokens is less than what the oracle price implies, harming all NLP holders.

**Impact: Medium** — permanent loss of quote tokens, NLP backing corruption.

---

### Likelihood Explanation

The concrete trigger is a race condition: a user signs a `MintNlp` or `BurnNlp` transaction when pool `i` is alive (assigning it a non-zero rebalance), but a `DeleteNlpPool` slow-mode transaction is processed before the `MintNlp`/`BurnNlp` transaction is executed on-chain. The sequencer cannot modify the user's signed `nlpPoolRebalanceX18` without invalidating the signature, so it must either reject the transaction (off-chain, with no on-chain enforcement) or process it as-is, routing quote to the dead subaccount.

**Likelihood: Medium** — pool deletions are infrequent but the on-chain code provides no guard; any pending signed transaction spanning a deletion event is vulnerable.

---

### Recommendation

In `_validateNlpRebalance` (or `_applyNlpRebalance`), add a check that any pool with `owner == address(0)` must have a corresponding `nlpPoolRebalanceX18[i] == 0`. For example:

```solidity
for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
    if (nlpPools[i].owner == address(0)) {
        require(nlpPoolRebalanceX18[i] == 0, ERR_INVALID_NLP_REBALANCE);
    }
    rebalanceAmount += nlpPoolRebalanceX18[i];
}
```

This mirrors the fix recommended in the external report: skip (or reject) dead entities rather than silently processing them.

---

### Proof of Concept

1. Admin adds NLP pool at index `1` (alive, `owner != address(0)`).
2. User signs a `MintNlp` transaction: `quoteAmount = 1000`, `nlpPoolRebalanceX18 = [0, 1000]` (all quote to pool 1).
3. Before the sequencer processes the `MintNlp`, admin submits `DeleteNlpPool(poolId=1)` as a slow-mode transaction. It executes first: pool 1's `owner` becomes `address(0)`, all balances swept to `N_ACCOUNT`.
4. Sequencer processes the `MintNlp` transaction. `_validateNlpRebalance` passes (sum `1000 == 1000`). `_applyNlpRebalance` calls `spotEngine.updateBalance(QUOTE_PRODUCT_ID, pool1.subaccount, 1000)`.
5. 1000 quote is now in pool 1's subaccount. `nlpProfitShare` reverts for pool 1 (`owner == address(0)`). No other path exists to retrieve the funds.
6. The user received `nlpAmount` of NLP tokens backed by 0 accessible quote — NLP supply is inflated relative to actual backing. [5](#0-4) [4](#0-3) [1](#0-0)

    function _validateNlpRebalance(
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18,
        int128 deltaQuoteAmount
    ) internal pure {
        require(
            nlpPools.length == nlpPoolRebalanceX18.length,
            ERR_INVALID_NLP_REBALANCE
        );
        int128 rebalanceAmount = 0;
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            rebalanceAmount += nlpPoolRebalanceX18[i];
        }
        require(deltaQuoteAmount == rebalanceAmount, ERR_INVALID_NLP_REBALANCE);
    }

        external
        virtual
        onlyEndpoint
    {
        require(subaccount != N_ACCOUNT, ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        uint32[] memory spotProducts = spotEngine.getProductIds();

        IPerpEngine perpEngine = _perpEngine();
        uint32[] memory perpProducts = perpEngine.getProductIds();

        for (uint32 i = 0; i < spotProducts.length; i++) {
            uint32 productId = spotProducts[i];

            ISpotEngine.Balance memory balance = spotEngine.getBalance(
                productId,
                subaccount
            );
            spotEngine.updateBalance(productId, subaccount, -balance.amount);
            spotEngine.updateBalance(productId, N_ACCOUNT, balance.amount);
        }

        for (uint32 i = 0; i < perpProducts.length; i++) {
            uint32 productId = perpProducts[i];

            IPerpEngine.Balance memory balance = perpEngine.getBalance(
                productId,
                subaccount
            );
            perpEngine.updateBalance(
                productId,
                subaccount,

                -balance.vQuoteBalance
            );
            perpEngine.updateBalance(
                productId,
                N_ACCOUNT,

        }
    }
