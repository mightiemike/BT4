### Title
Fee-on-Transfer Token Deposit Inflates Subaccount Collateral Credit Without Pre/Post Balance Verification — (File: `core/contracts/EndpointStorage.sol`)

---

### Summary

`handleDepositTransfer` in `EndpointStorage.sol` performs two sequential token transfers (user → Endpoint → Clearinghouse) using the caller-supplied `amount` with no pre/post balance check at either hop. If a fee-on-transfer token is listed as a collateral product and the Endpoint holds a residual balance of that token, the Clearinghouse receives `amount - fee` but the slow-mode deposit record encodes the full `amount`. The sequencer then credits the depositor with more collateral than the Clearinghouse actually holds, corrupting the solvency invariant.

---

### Finding Description

`EndpointStorage.handleDepositTransfer` (lines 111–119) executes two transfers in sequence:

```solidity
safeTransferFrom(token, from, amount);                    // User → Endpoint
safeTransferTo(token, address(clearinghouse), amount);    // Endpoint → Clearinghouse
```

No balance snapshot is taken before or after either call. The function trusts the caller-supplied `amount` throughout. [1](#0-0) 

In `Endpoint.depositCollateralWithReferral`, after `handleDepositTransfer` returns, the slow-mode transaction is enqueued with `amount: amount` — the original caller-supplied value, not the actual amount received by the Clearinghouse: [2](#0-1) 

When the sequencer later processes this slow-mode tx, `Clearinghouse.depositCollateral` calls `spotEngine.updateBalance(txn.productId, txn.sender, amountRealized)` where `amountRealized` is derived directly from `txn.amount` — the inflated value: [3](#0-2) 

**Concrete trigger path for a fee-on-transfer collateral token:**

1. `safeTransferFrom(token, from, amount)` → Endpoint receives `amount - fee1`. If the Endpoint holds no prior balance of this token, step 2 reverts (deposit fails). However, if the Endpoint holds a residual balance ≥ `fee1` (e.g., from a direct token send, or from slow-mode fee accumulation when the collateral token is also the quote token via `chargeSlowModeFee`), step 2 proceeds. [4](#0-3) 

2. `safeTransferTo(token, address(clearinghouse), amount)` → Clearinghouse receives `amount - fee2`. The Endpoint's residual balance is consumed by `fee1`.

3. The slow-mode tx records `amount` (not `amount - fee2`). The Clearinghouse credits the depositor with `amount` worth of collateral but physically holds only `amount - fee2` tokens.

The discrepancy is `fee2` per deposit. Over repeated deposits, the Clearinghouse's token balance falls progressively below the sum of all credited subaccount balances.

---

### Impact Explanation

The Clearinghouse's on-chain token balance becomes undercollateralized relative to the aggregate of all credited subaccount spot balances. Each deposit via a fee-on-transfer collateral token inflates the depositor's collateral by the transfer fee. Legitimate withdrawals will eventually fail because the Clearinghouse cannot honor all credited balances, constituting a solvency/accounting corruption. The corrupted state variable is the `spotEngine` balance for the affected `productId`, which exceeds the actual token holdings of the Clearinghouse.

**Impact: 5 / 10**

---

### Likelihood Explanation

Requires two conditions: (1) a fee-on-transfer token is listed as a supported collateral product, and (2) the Endpoint holds a residual balance of that token sufficient to cover the first-hop fee. Neither condition is architecturally prevented. The Endpoint can accumulate residual balances via `chargeSlowModeFee` (if the collateral token is also the quote token) or via direct token sends. Fee-on-transfer tokens are uncommon in current deployments but are not blocked by any protocol-level guard.

**Likelihood: 2 / 10**

---

### Recommendation

In `handleDepositTransfer`, snapshot the Clearinghouse's token balance before the transfers and verify the actual received amount after:

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 preBal = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    uint256 postBal = token.balanceOf(address(clearinghouse));
    require(postBal >= preBal + amount, "Insufficient received amount");
}
```

Alternatively, pass the actual delta (`postBal - preBal`) as the deposit amount into the slow-mode transaction instead of the caller-supplied `amount`.

---

### Proof of Concept

1. Token T is a fee-on-transfer token (2% fee per transfer) listed as a collateral product.
2. The Endpoint holds 5 T of residual balance (e.g., from a direct send).
3. Alice calls `depositCollateral(subaccountName, productId, 100e18)`.
4. `handleDepositTransfer` is called with `amount = 100e18`.
5. `safeTransferFrom(T, Alice, 100e18)` → Endpoint receives 98 T. Endpoint now holds 103 T.
6. `safeTransferTo(T, clearinghouse, 100e18)` → Clearinghouse receives 98 T. Endpoint now holds 3 T.
7. Slow-mode tx is enqueued with `amount = 100e18`.
8. Sequencer processes the slow-mode tx: `Clearinghouse.depositCollateral` credits Alice's subaccount with `100e18` (normalized) collateral.
9. Clearinghouse physically holds 98 T but has credited Alice with 100 T — a 2 T discrepancy.
10. Repeated deposits compound the discrepancy; eventually the Clearinghouse cannot honor all withdrawals. [5](#0-4) [6](#0-5) [3](#0-2)

    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
