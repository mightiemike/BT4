### Title
Insufficient Bonus Pool Blocks All Exit Paths, Permanently Locking User Funds - (`x/tieredrewards/keeper/claim_rewards.go`)

### Summary

`MsgTierUndelegate` and `MsgExitTierWithDelegation` both call `claimRewards()` before performing the delegation transfer. `claimRewards()` calls `processEventsAndClaimBonus()`, which fails atomically with `ErrInsufficientBonusPool` if the rewards pool cannot cover the accrued bonus. Because neither exit path has a fallback that forfeits the bonus and proceeds, an empty pool permanently blocks users from exiting their tier positions — even after the full exit commitment has elapsed.

### Finding Description

Every mutation that modifies a delegated position settles rewards first. For the two exit paths:

**`MsgTierUndelegate`** (`msg_server.go` line 166):
```go
pos, _, _, err = ms.claimRewards(ctx, pos)
if err != nil {
    return nil, err
}
```

**`MsgExitTierWithDelegation`** (`msg_server.go` line 532):
```go
pos, _, _, err = ms.claimRewards(ctx, pos)
if err != nil {
    return nil, err
}
```

`claimRewards` calls `processEventsAndClaimBonus`, which at line 230 of `claim_rewards.go` checks:

```go
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err
}
```

`sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool` if `poolBalance < bonus`. This error propagates all the way back to the message handler, causing the entire transaction to revert. No state is changed, and the user's position remains locked.

The `BeforeRedelegationSlashed` hook in `slash.go` (lines 54–64) already demonstrates the correct pattern for this situation — it silently forfeits the bonus and continues when the pool is insufficient:

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
    } else {
        return err
    }
}
```

The user-driven exit paths have no equivalent fallback.

The bonus pool is drained by two mechanisms: (1) the `BeginBlocker` continuously transfers shortfall from the pool to `x/distribution` to top up base rewards, and (2) successful `MsgClaimTierRewards` calls. The pool can reach zero through normal operation, and there is no on-chain guarantee it will be replenished promptly.

### Impact Explanation

A user who has locked tokens, triggered exit, and waited the full `ExitDuration` (e.g. 1 year) cannot recover their tokens via either exit path if the bonus pool is empty at the time they attempt to exit. `MsgTierUndelegate` fails, `MsgExitTierWithDelegation` fails, and `MsgClearPosition` also fails (it calls `claimRewards` too). The user's delegation remains inside the tier module with no reachable code path to retrieve it until an external actor funds the pool. This is a user-fund lockup with no on-chain resolution mechanism.

### Likelihood Explanation

The `BeginBlocker` drains the pool every block when `TargetBaseRewardsRate > 0`. The pool is funded externally (governance/community transfers). Any gap between pool depletion and replenishment creates a window where all exit operations fail. Early in the protocol's life — analogous to the "early market conditions" in the external report — the pool may be small or empty for extended periods. Any user who accrued non-zero bonus and attempts to exit during such a window is blocked.

### Recommendation

Mirror the `BeforeRedelegationSlashed` pattern in both exit paths: if `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, log the forfeiture and allow the exit to proceed without paying the bonus. The user loses accrued bonus but recovers their principal delegation. Alternatively, separate the reward-settlement step from the exit step so that exits can proceed independently of pool state.

### Proof of Concept

1. User calls `MsgLockTier` with `amount = MinLockAmount`, `trigger_exit_immediately = true`.
2. Time advances past `ExitDuration`. Bonus accrues to a non-zero value.
3. The `BeginBlocker` drains the rewards pool to zero (or the pool was never funded).
4. User calls `MsgTierUndelegate`:
   - `validateUndelegatePosition` passes (exit triggered, elapsed, delegated).
   - `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` → returns `ErrInsufficientBonusPool`.
   - Transaction reverts. Position unchanged.
5. User calls `MsgExitTierWithDelegation` with `amount = positionAmount`:
   - `validateExitTierWithDelegation` passes.
   - `claimRewards` → same failure path.
   - Transaction reverts. Position unchanged.
6. User calls `MsgClearPosition` to cancel exit and re-lock:
   - `claimRewards` → same failure path.
   - Transaction reverts.
7. User has no remaining on-chain action that can recover their tokens until the pool is externally replenished.
