### Title
`ValidResourceBounds` Protobuf Round-Trip Collapses `AllResources` to `L1Gas`, Producing a Different Transaction Hash - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion is not a bijection. A `ValidResourceBounds::AllResources` value with zero `l2_gas` and `l1_data_gas` serializes to protobuf and deserializes back as `ValidResourceBounds::L1Gas`. Because `get_tip_resource_bounds_hash` hashes a different number of resource-bound felts for the two variants, the transaction hash computed after a protobuf round-trip differs from the hash computed at ingestion time.

### Finding Description

`ValidResourceBounds` has two variants:

- `L1Gas(ResourceBounds)` — legacy pre-0.13.3, only L1 gas
- `AllResources(AllResourceBounds)` — post-0.13.3, all three gas types

The protobuf deserialization path in `crates/apollo_protobuf/src/converters/transaction.rs` reconstructs the variant by inspecting the decoded field values:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

Any `AllResources` value whose `l2_gas` and `l1_data_gas` are both zero is therefore silently collapsed to `L1Gas` on deserialization. The serialization path emits `Some(zero)` for both fields when the original is `AllResources`, so the information that the variant was `AllResources` is lost:

```rust
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(l2_gas.into()),
        l1_data_gas: Some(l1_data_gas.into()),
    },
``` [2](#0-1) 

The transaction hash for V3 invoke transactions is computed by `get_tip_resource_bounds_hash`, which includes a different number of felts depending on the variant:

- `AllResources` → hashes `[tip, L1_GAS_packed, L2_GAS_packed, L1_DATA_GAS_packed]` (4 elements)
- `L1Gas` → hashes `[tip, L1_GAS_packed, L2_GAS_packed]` (3 elements, no L1_DATA_GAS) [3](#0-2) 

The hash is then embedded in the full V3 invoke hash chain: [4](#0-3) 

### Impact Explanation

A user submits an `InvokeV3` transaction via the JSON-RPC gateway with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`. This is a structurally valid transaction accepted by the gateway. The gateway converts it to `InternalRpcInvokeTransactionV3` (which always stores `AllResourceBounds`) and computes hash **H₁** using the `AllResources` path (4-element poseidon input). [5](#0-4) 

When the block containing this transaction is propagated via P2P block sync, the `InvokeTransactionV3` (which carries `ValidResourceBounds`) is serialized to protobuf and deserialized on the receiving node as `ValidResourceBounds::L1Gas(X)`. The receiving node then computes hash **H₂** using the `L1Gas` path (3-element poseidon input). **H₁ ≠ H₂**.

Additionally, `get_gas_vector_computation_mode()` returns `All` for `AllResources` and `NoL2Gas` for `L1Gas`, so fee validation, balance checks, and resource-bound enforcement all behave differently on the synced node versus the sequencer that originally executed the block. [6](#0-5) 

### Likelihood Explanation

Any `InvokeV3` transaction with non-zero L1 gas but zero L2 gas and zero L1_DATA gas triggers the bug. Such transactions are structurally valid and accepted by the gateway without special privileges. The `RpcInvokeTransactionV3` struct mandates `AllResourceBounds` for all new transactions, so every post-0.13.3 transaction that happens to set only L1 gas is affected. [7](#0-6) 

### Recommendation

The deserialization logic must preserve the original variant. One approach: add a discriminant field to the protobuf `ResourceBounds` message that explicitly encodes whether the transaction was `L1Gas` or `AllResources`. Alternatively, always deserialize to `AllResources` when all three fields are present in the protobuf message (even if they are zero), and only fall back to `L1Gas` when `l1_data_gas` is absent (the legacy 0.13.2 case flagged by the existing TODO comment):

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
``` [8](#0-7) 

When `l1_data_gas` is `Some(zero)` (explicitly present), the result should be `AllResources`; only when it is `None` (absent, legacy) should the `L1Gas` fallback apply.

### Proof of Concept

1. Submit an `InvokeV3` RPC transaction with `resource_bounds = { L1_GAS: { max_amount: 1000, max_price_per_unit: 1 }, L2_GAS: { 0, 0 }, L1_DATA_GAS: { 0, 0 } }`. The gateway accepts it and computes hash H₁ via `get_invoke_transaction_v3_hash` with `AllResources` (4-element resource hash).

2. Serialize the resulting `InvokeTransactionV3` (with `ValidResourceBounds::AllResources(l1_gas=1000@1, l2_gas=0, l1_data_gas=0)`) to protobuf using `From<ValidResourceBounds> for protobuf::ResourceBounds`. The wire message has `l1_gas=Some(...)`, `l2_gas=Some(zero)`, `l1_data_gas=Some(zero)`.

3. Deserialize via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Because `l1_data_gas.is_zero() && l2_gas.is_zero()`, the result is `ValidResourceBounds::L1Gas(l1_gas=1000@1)`.

4. Recompute the transaction hash. `get_tip_resource_bounds_hash` now hashes only `[tip, L1_GAS_packed, L2_GAS_packed]` (3 elements), producing hash H₂ ≠ H₁.

5. The synced node stores or validates the transaction with H₂, diverging from the canonical hash H₁ recorded in the block header and used by the original sequencer.
