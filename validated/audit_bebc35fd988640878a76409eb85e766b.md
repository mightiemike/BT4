### Title
`isHealthy` Stub Always Returns `true`, Permanently Disabling Post-Trade Health Enforcement in `matchOrders` — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.sol` declares an `isHealthy` virtual function that unconditionally returns `true`. It is the sole on-chain post-trade health gate in `matchOrders`, invoked after all balance deltas are applied to both taker and maker. Because the function never evaluates actual subaccount health, the two `require` guards that depend on it are permanently bypassed. Any matched trade — regardless of how deeply it drives a subaccount into negative health — will succeed on-chain. This is structurally identical to the `isPaused` bug in the external report: a protective mechanism is declared but the implementation that would make it enforce protection is absent, so the protection is permanently disabled.

---

### Finding Description

In `OffchainExchange.sol` the function is declared as:

```solidity
function isHealthy(
    bytes32 /* subaccount */
) internal view virtual returns (bool) {
    return true;
}
``` [1](#0-0) 

It is the only on-chain post-trade health check. It is called in `matchOrders` after all balance mutations have been committed:

```solidity
require(isHealthy(taker.order.sender), ERR_INVALID_TAKER);
require(isHealthy(maker.order.sender), ERR_INVALID_MAKER);
``` [2](#0-1) 

The `virtual` keyword signals the function was designed to be overridden with a real health evaluation (e.g., a call to `clearinghouse.getHealth`). No such override exists anywhere in the deployed `OffchainExchange` contract. The result mirrors the `isPaused` bug exactly: the variable/function that is supposed to gate a critical action is declared, but the mechanism that would give it a meaningful value is missing, so the gate is permanently open.

The `matchOrders` function is the on-chain settlement path for every sequencer-submitted trade: [3](#0-2) 

Balance deltas are applied to both sides before `isHealthy` is called: [4](#0-3) 

---

### Impact Explanation

After a trade executes, neither the taker's nor the maker's post-trade health is verified on-chain. A subaccount can be driven to arbitrarily negative initial health through a matched order with no on-chain rejection. Concretely:

- The on-chain backstop against health-violating trades is completely absent.
- Subaccounts can accumulate positions that exceed their collateral, creating insolvency risk for the protocol.
- The solvency guarantee for matched trades relies entirely on off-chain sequencer logic; there is no on-chain enforcement layer.

The corrupted state is the subaccount's health invariant: `getHealth(subaccount, INITIAL) >= 0` can be violated after `matchOrders` returns successfully.

---

### Likelihood Explanation

The sequencer is expected to enforce health off-chain before submitting `MatchOrders` transactions. However, the complete absence of any on-chain check means any sequencer-side bug, race condition between price update and order matching, or off-chain misconfiguration that allows a health-violating trade to be submitted will result in that trade being accepted on-chain with no rejection. Likelihood is **medium**: the sequencer is trusted, but the on-chain safety net is entirely absent, leaving the protocol one sequencer-side error away from accepting health-violating trades.

---

### Recommendation

Override `isHealthy` in `OffchainExchange` to perform a real health evaluation:

```solidity
function isHealthy(bytes32 subaccount)
    internal
    view
    virtual
    override
    returns (bool)
{
    return
        clearinghouse.getHealth(
            subaccount,
            IProductEngine.HealthType.INITIAL
        ) >= 0;
}
```

This mirrors how `withdrawCollateral` in `Clearinghouse.sol` enforces health after balance mutations: [5](#0-4) 

---

### Proof of Concept

1. Trader A signs an order to open a large perp position. At the execution price, the position would push their subaccount below initial health.
2. The sequencer (due to a bug, stale price, or race condition) submits `MatchOrders` with this trade.
3. `matchOrders` executes: `_updateBalances` applies the base and quote deltas to Trader A's subaccount.
4. `require(isHealthy(taker.order.sender), ERR_INVALID_TAKER)` is reached.
5. `isHealthy` returns `true` unconditionally — no call to `clearinghouse.getHealth` is made.
6. The trade succeeds on-chain. Trader A now holds a position that violates initial health with no on-chain rejection, corrupting the health invariant that the rest of the protocol (liquidations, withdrawals) depends on. [1](#0-0) [2](#0-1)
