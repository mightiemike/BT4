### Title
`VersionedConstants::latest_constants()` Used Instead of Block-Version-Specific Constants in Proposal Validation — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`is_proposal_init_valid` fetches consensus margin parameters (`l1_gas_price_margin_percent`, `fee_proposal_margin_ppt`) from `VersionedConstants::latest_constants()` — a global "latest" snapshot — instead of from the constants that correspond to the block's own `starknet_version`. This is the direct Sequencer analog of the `latestMarket`-vs-`marketIndex` bug: a globally-shared identifier is substituted for the locally-scoped one that the call site already holds.

### Finding Description

In `is_proposal_init_valid` (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`), two separate margin checks unconditionally call `VersionedConstants::latest_constants()`:

```rust
// Line ~330
let l1_gas_price_margin_percent =
    VersionedConstants::latest_constants().l1_gas_price_margin_percent.into();
```

```rust
// Line ~403
let (lower_bound, upper_bound) = fee_proposal_bounds(
    fee_actual,
    VersionedConstants::latest_constants().fee_proposal_margin_ppt,
);
``` [1](#0-0) [2](#0-1) 

At the same call site, `init_proposed.starknet_version` is already validated and available:

```rust
if init_proposed.starknet_version != proposal_init_validation.starknet_version {
    return Err(...)
}
``` [3](#0-2) 

The `apollo_versioned_constants::VersionedConstants` crate defines distinct per-version constant files (V0_14_0 through V0_14_4), each potentially carrying different values for `l1_gas_price_margin_percent` and `fee_proposal_margin_ppt`:

```rust
define_versioned_constants!(
    VersionedConstants,
    VersionedConstantsError,
    StarknetVersion::V0_14_0,
    "resources/versioned_constants_diff_regression",
    (V0_14_0, "../resources/orchestrator_versioned_constants_0_14_0.json"),
    ...
    (V0_14_4, "../resources/orchestrator_versioned_constants_0_14_4.json"),
);
``` [4](#0-3) 

The `define_versioned_constants!` macro generates a `get(version)` accessor, so the correct call is `VersionedConstants::get(&init_proposed.starknet_version)`. Instead, both checks bypass this and always read from the latest snapshot.

### Impact Explanation

During any network version upgrade (e.g., V0_14_3 → V0_14_4), a subset of validators will have upgraded their binary while the network's canonical `starknet_version` is still the older one. Those upgraded validators call `latest_constants()` and obtain V0_14_4 margin values; non-upgraded validators obtain V0_14_3 values. The two cohorts then apply different acceptance windows to the same proposal's L1 gas price and fee-proposal fields:

- If V0_14_4 tightens `l1_gas_price_margin_percent`, upgraded validators reject proposals that non-upgraded validators accept → consensus liveness failure.
- If V0_14_4 loosens the margin, upgraded validators accept proposals with gas prices outside the range that the network version's rules permit → a block with an economically incorrect L1 gas price is committed, corrupting fee accounting for every transaction in that block.

The second path maps directly to **"Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."** The first maps to **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing"** (consensus admission of a block proposal).

### Likelihood Explanation

Every protocol version upgrade creates the window. The `fee_proposal` path is additionally gated on `starknet_version >= V0_14_3`, meaning the `fee_proposal_margin_ppt` divergence is reachable on the current mainnet version range. No privileged attacker is required; the divergence is triggered automatically by any validator that upgrades its binary before the network-wide version flip.

### Recommendation

Replace both `VersionedConstants::latest_constants()` calls with a version-specific lookup:

```rust
let vc = VersionedConstants::get(&init_proposed.starknet_version)
    .map_err(|_| ValidateProposalError::InvalidProposalInit(...))?;

let l1_gas_price_margin_percent = vc.l1_gas_price_margin_percent.into();
// ...
let (lower_bound, upper_bound) = fee_proposal_bounds(fee_actual, vc.fee_proposal_margin_ppt);
```

This mirrors the correct pattern already used for the `fee_proposal_required` gate, which reads `init_proposed.starknet_version` directly rather than a global constant.

### Proof of Concept

1. Deploy two validator nodes: node A running software where `latest_constants()` = V0_14_3 constants, node B running software where `latest_constants()` = V0_14_4 constants.
2. Set the network's canonical `starknet_version` to V0_14_3 (pre-upgrade state).
3. Suppose V0_14_4 changes `l1_gas_price_margin_percent` from 10 to 5.
4. A proposer submits a `ProposalInit` with an L1 gas price 8% above the oracle price — valid under V0_14_3's 10% margin, invalid under V0_14_4's 5% margin.
5. Node A (V0_14_3 latest) calls `VersionedConstants::latest_constants().l1_gas_price_margin_percent` → 10 → accepts.
6. Node B (V0_14_4 latest) calls `VersionedConstants::latest_constants().l1_gas_price_margin_percent` → 5 → rejects.
7. Consensus splits: the network cannot reach a quorum on the proposal, halting block production; or, if node B is the minority, the block is committed with an L1 gas price outside the protocol-defined range, corrupting fee calculations for all transactions in that block.
