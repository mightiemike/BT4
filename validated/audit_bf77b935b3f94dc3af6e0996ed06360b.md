Audit Report

## Title
SwapAllowlistExtension gates on the router address instead of the actual user when swaps are routed through MetricOmmSimpleRouter, enabling allowlist bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against `allowedSwapper[pool][sender]`, but `sender` is sourced directly from `msg.sender` at the pool's `swap` entry point. When `MetricOmmSimpleRouter` calls the pool, the pool's `msg.sender` is the router contract, so the extension evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actualUser]`. This wrong-actor binding either allows any user to bypass the allowlist (if the router is allowlisted) or locks allowlisted users out of the standard periphery path (if only end-users are allowlisted).

## Finding Description

In `MetricOmmPool.swap`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // ← this is the router when called via MetricOmmSimpleRouter
    recipient,
    ...
);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value unchanged as the `sender` argument to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (correct key) and `sender` is the router (wrong actor):

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls the pool directly without passing the originating user: [4](#0-3) 

The same wrong-actor binding applies to `exactInput` (intermediate hops use `address(this)` as payer), `exactOutputSingle`, and `exactOutput` recursive callbacks: [5](#0-4) 

No existing guard corrects this: the pool has no mechanism to distinguish the originating user from the immediate caller, and the extension has no fallback identity check.

## Impact Explanation

**Path A — Allowlist bypass (High):** A pool admin who allowlists the router to enable standard periphery access causes `allowedSwapper[pool][router] == true`. Any non-allowlisted user can then call `router.exactInputSingle` and pass the guard, fully defeating the curated pool's access control for all router-mediated volume. This is a direct bypass of a core security invariant (unauthorized trading in a permissioned pool).

**Path B — Broken core functionality (Medium):** A pool admin who allowlists only specific end-users (e.g., KYC'd addresses) will find those users cannot use the router at all, because the extension sees `sender = router` and the router is not allowlisted. Authorized users are locked out of the standard periphery swap path, constituting broken core swap functionality.

## Likelihood Explanation

Path A requires only that the pool admin has allowlisted the router — a natural and expected action for any pool that intends to support the standard periphery. Once that condition holds, any unprivileged user can trigger the bypass with a single `exactInputSingle` call. Path B is triggered by any allowlisted user simply using the router. Neither path requires malicious setup, special roles, non-standard tokens, or any off-chain coordination.

## Recommendation

The extension must gate on the actual end-user, not the intermediary. The cleanest fix is to pass the originating user explicitly through the swap call chain:

1. Add a `swapper` parameter (separate from `recipient`) to the pool's `swap` function, set by the router to `msg.sender` before forwarding to the pool.
2. Have the pool pass that value as `sender` to `_beforeSwap` and the extension.
3. Alternatively, the router can store the originating user in transient storage (analogous to how it already stores the payer via `_setNextCallbackContext`) and the pool reads it to populate `sender` for extensions.

Using `tx.origin` as a fallback is discouraged due to phishing risks and should not be used.

## Proof of Concept

```
Setup:
  1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
  2. Pool admin calls setAllowedToSwap(pool, router, true)
     — natural action to enable router-mediated swaps.
  3. Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

Attack:
  4. attacker (non-allowlisted) calls:
       router.exactInputSingle({pool: pool, recipient: attacker, ...})
  5. Router calls pool.swap(attacker, zeroForOne, amount, limit, "", extensionData)
     → pool's msg.sender = router
  6. Pool calls _beforeSwap(router /*sender*/, attacker /*recipient*/, ...)
  7. Extension evaluates:
       allowedSwapper[pool][router] == true  → guard passes
  8. Swap executes. attacker receives output tokens.

Expected: revert NotAllowedToSwap (attacker is not allowlisted)
Actual:   swap succeeds — allowlist fully bypassed via router indirection.

Symmetric broken-functionality PoC:
  - allowlist alice but not the router
  - alice calls router.exactInputSingle
  - extension sees sender = router, not allowlisted → reverts NotAllowedToSwap
  - alice is locked out of the standard periphery path despite being authorized
```
