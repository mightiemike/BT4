### Title
Compromised Validators Can Forge Withdrawal Submission Indices in `submitFastWithdrawal` Without On-Chain Proof — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.submitFastWithdrawal` validates the caller-supplied `idx` only by checking `idx > minIdx`. There is no on-chain proof that the `idx` was ever actually processed by the Endpoint sequencer (i.e., no check that `idx ≤ nSubmissions`). A fully compromised validator set can sign off on a forged `(idx, transaction)` pair — where the withdrawal transaction was never submitted to the Endpoint — and drain the `WithdrawPool` of real tokens.

---

### Finding Description

`submitFastWithdrawal` is the fast-withdrawal path in `BaseWithdrawPool`. It accepts a `uint64 idx`, a raw `bytes calldata transaction`, and a full set of individual ECDSA validator signatures. The function verifies those signatures via `Verifier.requireValidTxSignatures`, which hashes `keccak256(chainid || idx || txn)` and checks that every registered validator signed it (`nSignatures == nSigner`).

The only on-chain guard on `idx` is:

```solidity
require(!markedIdxs[idx], "Withdrawal already submitted");
require(idx > minIdx, "idx too small");
```

`minIdx` is updated to the most recent `idx` that the sequencer submitted via `submitWithdrawal` (called from `Clearinghouse.handleWithdrawTransfer`). There is no check that `idx ≤ nSubmissions` (the Endpoint's current submission counter), and no cryptographic proof that the supplied `transaction` was ever enqueued in the Endpoint's slow-mode queue or processed by `submitTransactionsChecked`.

In the legitimate flow, the sequencer processes a `WithdrawCollateral` transaction at submission index `nSubmissions`, calls `Clearinghouse.withdrawCollateral(..., nSubmissions)`, which calls `handleWithdrawTransfer(..., idx)` → `BaseWithdrawPool.submitWithdrawal(..., idx)`. The fast-withdrawal path is designed to let a user pre-claim tokens before the sequencer reaches that `idx`, with validators attesting that the withdrawal is genuine.

The missing invariant: **nothing on-chain proves that the `idx` the validators signed over was ever submitted to the Endpoint**. The validators sign an arbitrary `(idx, txn)` tuple; the contract only checks that `idx` is numerically larger than the last sequencer-confirmed withdrawal.

This is structurally identical to the reported `TokenBridge.sol` bug: in that case, the `blockRid` was validated only by a timestamp/height comparison (`blockRid < massExitBlock`) with no hash-chain proof linking it to the mass-exit block. Here, `idx` is validated only by a numeric comparison (`idx > minIdx`) with no proof linking it to any on-chain submission.

---

### Impact Explanation

A fully compromised validator set (all `nSigner` keys) can:

1. Construct a `SignedWithdrawCollateral` or `SignedWithdrawCollateralV2` transaction for an arbitrary `sendTo` address and an arbitrary `amount` — one that was **never submitted** to the Endpoint.
2. Choose any `idx` satisfying `idx > minIdx` (e.g., `minIdx + 1`).
3. Have all validators sign `keccak256(chainid || idx || txn)`.
4. Call `submitFastWithdrawal(idx, txn, signatures)`.

The function marks `markedIdxs[idx] = true`, verifies the signatures, and immediately transfers `amount − fee` tokens from the `WithdrawPool` to `sendTo`. Because the withdrawal was never submitted to the Endpoint, the sequencer will never call `submitWithdrawal` for that `idx`, so the `markedIdxs` flag is never cleared and the stolen tokens are never reconciled. The `WithdrawPool` balance is permanently reduced by the forged amount.

Corrupted state: `WithdrawPool` ERC-20 token balance, `markedIdxs[idx]`, and `fees[productId]` — all without any corresponding off-chain or on-chain withdrawal record.

---

### Likelihood Explanation

The attack requires compromise of **all** `nSigner` validator keys (since `requireValidTxSignatures` enforces `nSignatures == nSigner`, not a quorum). This is a stronger precondition than the original bug (which required only a quorum). However:

- The validator set is small (at most 8 slots, `pubkeys[8]`), making full compromise feasible if the key-management infrastructure is targeted.
- The original reported bug also operates under the premise of compromised validators; the structural flaw is the same — the absence of a binding cryptographic link between the submitted value and the actual on-chain state.
- The `WithdrawPool` holds real user funds and is a high-value target, making the attack economically motivated.

---

### Recommendation

Add a check in `submitFastWithdrawal` that the supplied `idx` does not exceed the Endpoint's current `nSubmissions` counter, ensuring the index refers to a submission that has already been processed on-chain:

```solidity
require(idx <= IEndpoint(endpoint).nSubmissions(), "idx not yet processed");
```

Alternatively, mirror the approach recommended for `TokenBridge.sol`: require a hash-chain proof (e.g., a Merkle inclusion proof) that the supplied `transaction` was included in the Endpoint's slow-mode queue at the claimed `idx`, linking the fast-withdrawal claim to verifiable on-chain state rather than relying solely on validator attestation.

---

### Proof of Concept

**Setup:** `nSigner = 2`, `minIdx = 100`, `WithdrawPool` holds 1,000,000 USDC.

1. Attacker compromises both validator keys.
2. Attacker constructs `txn = abi.encodePacked(uint8(WithdrawCollateral), abi.encode(SignedWithdrawCollateral({sender: attackerSubaccount, productId: USDC_PRODUCT_ID, amount: 1_000_000e6, nonce: 0, signature: ...})))`.
3. Both validators sign `keccak256(abi.encodePacked(uint256(block.chainid), uint256(101), txn))`.
4. Attacker calls `submitFastWithdrawal(101, txn, [sig0, sig1])`.
5. `markedIdxs[101]` is set; `requireValidTxSignatures` passes; `resolveFastWithdrawal` returns `(USDC_PRODUCT_ID, attacker, 1_000_000e6)`; `handleWithdrawTransfer` sends tokens to attacker.
6. The Endpoint never had a withdrawal at `nSubmissions == 101` for this `txn`; the sequencer never calls `submitWithdrawal(101)`. The `WithdrawPool` is drained.

    function requireValidTxSignatures(
        bytes calldata txn,
        uint64 idx,
        bytes[] calldata signatures
    ) public view {
        require(signatures.length <= 256, "too many signatures");
        bytes32 data = keccak256(
            abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
        );
        bytes32 hashedMsg = keccak256(
            abi.encodePacked("\x19Ethereum Signed Message:\n32", data)
        );

        uint256 nSignatures = 0;
        for (uint256 i = 0; i < signatures.length; i++) {
            if (signatures[i].length > 0) {
                nSignatures += 1;
                require(
                    checkIndividualSignature(
                        hashedMsg,
                        signatures[i],
                        uint8(i)
                    ),
                    "invalid signature"
                );
            }
        }
        require(nSignatures == nSigner, "not enough signatures");
    }
