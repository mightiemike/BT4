### Title
Missing `ENABLE_SECP_OPS` Flag Guard on 4-Byte Secp Opcode Dispatch Allows Secp Verification to Execute Regardless of Softfork Activation - (`File: src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` dispatches `op_secp256k1_verify` and `op_secp256r1_verify` through two separate code paths. The 1-byte opcode path (opcodes 64 and 65) correctly gates execution behind `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte opcode path (`0x13d61f00` and `0x1c3a8f00`) dispatches the same functions **without any flag check**, allowing secp signature verification to execute unconditionally regardless of whether the softfork has activated.

---

### Finding Description

In `src/chia_dialect.rs`, `ChiaDialect::op()` contains two dispatch branches for secp operations:

**1-byte path (correctly gated):**
```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [1](#0-0) 

**4-byte path (ungated):**
```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
``` [2](#0-1) 

The 4-byte opcodes `0x13d61f00` and `0x1c3a8f00` were deliberately chosen so their unknown-op cost formula yields exactly 1300000 and 1850000 respectively — matching the fixed costs of `op_secp256k1_verify` and `op_secp256r1_verify`. This means cost is identical whether the opcode is treated as unknown or as secp. However, **behavior differs critically**: an unknown op always returns nil, while secp verify returns an error (`EvalErr::Secp256Failed`) when the signature is invalid. [3](#0-2) 

The `ENABLE_SECP_OPS` flag is a softfork guard — it is supposed to be disabled until the secp softfork activates on the network. [4](#0-3) 

---

### Impact Explanation

**Consensus divergence.** Before `ENABLE_SECP_OPS` activates:

- A program using 1-byte opcode 64/65 with an invalid secp signature → treated as unknown op → returns nil (coin spendable).
- The same program using 4-byte opcode `0x13d61f00`/`0x1c3a8f00` with an invalid secp signature → secp verify executes → `EvalErr::Secp256Failed` (coin rejected).

The two encodings of the same logical operation produce different consensus outcomes. An attacker who crafts a coin puzzle using the 4-byte opcode form can cause it to be rejected by all nodes (even those that have not activated the secp softfork), while the equivalent 1-byte form would be accepted. Conversely, a coin that should be protected by secp verification (once the softfork activates) can be bypassed pre-activation using the 4-byte form only if the node's flag state differs from what the coin author assumed.

The broken invariant: `ENABLE_SECP_OPS` is supposed to be the single gate controlling whether secp verification is active. The 4-byte path breaks this invariant unconditionally.

---

### Likelihood Explanation

The 4-byte opcode values are documented in the codebase comment and are deterministic from the cost formula. Any attacker who reads `src/chia_dialect.rs` can trivially construct a CLVM program using `0x13d61f00` or `0x1c3a8f00` as the operator atom. The entry path is the standard `run_serialized_chia_program` Python API, which is the primary execution surface for all Chia transaction validation. [5](#0-4) 

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch branch, mirroring the 1-byte path:

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
        0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

This ensures that before `ENABLE_SECP_OPS` activates, both the 1-byte and 4-byte forms of the secp opcodes are treated as unknown operators, restoring the intended single-gate invariant.

---

### Proof of Concept

Attacker-controlled CLVM bytes encoding a program that uses the 4-byte opcode `0x13d61f00` with an invalid signature. When run via `run_serialized_chia_program` with flags that do **not** include `ENABLE_SECP_OPS`:

- **Expected** (per flag semantics): unknown op → nil returned, cost = 1300000, program succeeds.
- **Actual**: `op_secp256k1_verify` executes → `EvalErr::Secp256Failed` raised, program fails.

The operator atom is simply the 4-byte sequence `[0x13, 0xd6, 0x1f, 0x00]` serialized as a CLVM atom. No special privileges or configuration are required beyond submitting a transaction to any node running this code. [6](#0-5)
