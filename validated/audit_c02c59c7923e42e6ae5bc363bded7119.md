Audit Report

## Title
`OracleValueStopLossExtension` Uses Arithmetic Mid Price While Pool Uses Geometric Mid Price, Causing Stop-Loss Guard to Misfire on Spread Changes — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid price as the arithmetic mean of bid and ask at line 218, while `MetricOmmPool.swap` and `PriceVelocityGuardExtension.beforeSwap` both use the geometric mean via `SwapMath.midAndSpreadFeeX64FromBidAsk`. Because AM ≥ GM always holds, the per-bin value metrics fed to the watermark guard are systematically overstated in token1 terms. When the oracle spread widens between the watermark-setting swap and a later swap, the `metricT1` breach condition becomes harder to trigger than it should be, allowing token0-draining swaps to pass the guard that should have been reverted.

## Finding Description

**Root cause — arithmetic mean at line 218:**

```solidity
// OracleValueStopLossExtension.sol L218
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
``` [1](#0-0) 

**Correct formula used everywhere else:**

`SwapMath.midAndSpreadFeeX64FromBidAsk` returns `Math.sqrt(bidPriceX64 * askPriceX64)` (geometric mean). [2](#0-1) 

`MetricOmmPool.swap` uses it at lines 242–243: [3](#0-2) 

`PriceVelocityGuardExtension.beforeSwap` uses it at line 48: [4](#0-3) 

**How the wrong mid price corrupts the metrics:**

```solidity
// L254-255
metricT0 = t0ps + (t1 * Q64 / midPriceX64) * SCALE / shares;   // divides by mid → understated
metricT1 = (t0 * midPriceX64 / Q64) * SCALE / shares + t1ps;   // multiplies by mid → overstated
``` [5](#0-4) 

Since AM ≥ GM, `metricT1` is always overstated relative to what the geometric mid would produce.

**How the watermark guard fails:**

The watermark ratchets up on new highs and checks breach below the drawdown floor: [6](#0-5) 

Let spread at watermark-setting time be S1 and at check time be S2 > S1. The AM overstatement error is `ε ≈ S²/8` (second-order Taylor expansion of AM − GM). So:

- `hwm1` was set to `metricT1_true × (1 + ε₁)` where `ε₁ ∝ S1²`
- Current `metricT1` = `metricT1_true_now × (1 + ε₂)` where `ε₂ ∝ S2² > ε₁`

Breach condition: `metricT1_true_now × (1 + ε₂) < metricT1_true_hwm × (1 + ε₁) × floorMultiplier / E6`

When the true value has dropped exactly to the drawdown floor (`metricT1_true_now = metricT1_true_hwm × floorMultiplier / E6`), the AM-based check reduces to `(1 + ε₂) < (1 + ε₁)`, which is **false** since ε₂ > ε₁. The breach is not triggered. The swap proceeds and LPs lose token0 principal.

No existing guard compensates for this: the `_applyWatermark` and `_decayed` helpers operate purely on the already-corrupted `metricT1` value. [7](#0-6) 

## Impact Explanation

Direct loss of LP token0 principal. The `OracleValueStopLossExtension` is deployed specifically to prevent drawdown past a configured floor. When the oracle spread widens — most likely during volatile market conditions, exactly when stop-loss protection is most critical — the `metricT1` watermark guard systematically underestimates the true value loss in token1 terms. Token0-draining swaps (`!zeroForOne`) that exceed the drawdown floor are not blocked. The error magnitude is `O(ΔS²)` and is always in the direction of weakening the guard, making this a systematic rather than edge-case failure.

## Likelihood Explanation

Any pool deploying `OracleValueStopLossExtension` with a price provider whose spread varies over time is affected. Oracle spreads routinely widen during market stress. No privileged action is required: any public swap triggers `afterSwap`, which calls `_afterSwapOracleStopLoss` with the wrong mid price formula. The trigger is fully reachable by any unprivileged trader on any affected pool. [8](#0-7) 

## Recommendation

Replace the arithmetic mean with the geometric mean in `_afterSwapOracleStopLoss`, consistent with the rest of the protocol:

```solidity
// Before (wrong):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (correct):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
```

This requires importing `SwapMath` into `OracleValueStopLossExtension`, which is already available in the `metric-core` package used by the periphery.

## Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension` configured with `drawdownE6 = 50_000` (5% drawdown floor) and a price provider whose spread can be varied.
2. Set oracle spread to 0.1% (bid = 1.000, ask = 1.001). Execute a swap. The watermark `hwm1` is set using `midPriceX64_arith = 1.0005`, while the correct geometric mid is `≈1.00049975`. The watermark is overstated by `~0.00025%`.
3. Widen oracle spread to 5% (bid = 1.000, ask = 1.050). Execute a `!zeroForOne` swap that drains token0 past the 5% drawdown floor. The current `metricT1` is computed with `midPriceX64_arith = 1.025`, while the correct geometric mid is `≈1.02470`. The current metric is overstated by `~0.03%` relative to the watermark's overstatement of `~0.00025%`. The net effect: the breach check sees `metricT1` as `~0.03%` higher than it should be relative to `hwm1`, masking a real drawdown breach. The swap proceeds and LPs lose token0 value that the guard should have blocked.
