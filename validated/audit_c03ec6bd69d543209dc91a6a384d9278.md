### Title
P2P Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Producing a Different Transaction Hash Preimage — (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` converter in the P2P state-sync path uses a zero-value heuristic to distinguish pre-0.13.3 (`L1Gas`) from post-0.13.3 (`AllResources`) transactions. Any V3 transaction submitted via RPC with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is accepted and hashed with a 4-element preimage (tip + L1 + L2(0) + L1_DATA(0)). When the same transaction is later received over P2P protobuf, the converter classifies it as `L1Gas(X)` and the hash is recomputed with a 3-element preimage (tip + L1 + L2(0)), producing a different digest. This is a canonicalization invariant break: the same logical transaction produces two different hashes depending on which ingestion path is used.

### Finding Description

**Step 1 — RPC ingestion path always uses `AllResources`.**

`RpcInvokeTransactionV3` stores resource bounds as `AllResourceBounds` (three fields). The conversion to `InternalRpcInvokeTransactionV3` preserves this:

```rust
impl From<RpcInvokeTransactionV3> for InternalRpcInvokeTransactionV3 {
    fn from(tx: RpcInvokeTransactionV3) -> Self {
        Self {
            resource_bounds: tx.resource_bounds,   // AllResourceBounds
            ...
        }
    }
}
```

The hash is then computed via `get_invoke_transaction_v3_hash`, which calls `get_tip_resource_bounds_hash` with `ValidResourceBounds::AllResources(...)`. [1](#0-0) 

**Step 2 — `get_tip_resource_bounds_hash` produces different-length preimages for the two variants.**

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],                          // 2 elements
        ValidResourceBounds::AllResources(all) => {
            vec![get_concat_resource(&all.l1_data_gas, L1_DATA_GAS)?]    // 3 elements
        }
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```

`AllResources` hashes `poseidon(tip, L1, L2, L1_DATA)` while `L1Gas` hashes `poseidon(tip, L1, L2)`. Even when L2 and L1_DATA are both zero, the two digests differ because the Poseidon hash is length-sensitive. [2](#0-1) 

**Step 3 — P2P protobuf serializer always emits all three fields, even for `L1Gas`.**

```rust
impl From<ValidResourceBounds> for protobuf::ResourceBounds {
    fn from(value: ValidResourceBounds) -> Self {
        match value {
            ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
                l1_gas: Some(l1_gas.into()),
                l2_gas: Some(value.get_l2_bounds().into()),
                l1_data_gas: Some(ResourceBounds::default().into()),  // zero
            },
            ValidResourceBounds::AllResources(...) => { ... }
        }
    }
}
```

So `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` and `L1Gas(X)` produce **identical** protobuf bytes. [3](#0-2) 

**Step 4 — P2P protobuf deserializer uses a zero-check heuristic that is ambiguous.**

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)          // ← wrong for post-0.13.3 txs
        } else {
            ValidResourceBounds::AllResources(...)
        })
    }
}
```

A post-0.13.3 transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is silently downgraded to `L1Gas(X)` after a protobuf round-trip. [4](#0-3) 

**Step 5 — The hash recomputed from the deserialized form differs from the hash stored in the block.**

| Path | `ValidResourceBounds` variant | Hash preimage elements | Hash |
|---|---|---|---|
| RPC submission | `AllResources { X, 0, 0 }` | tip + L1 + L2(0) + L1_DATA(0) | H1 |
| P2P protobuf round-trip | `L1Gas(X)` | tip + L1 + L2(0) | H2 ≠ H1 |

### Impact Explanation

Any node that receives a block via P2P state sync and recomputes transaction hashes (e.g., during block validation in `validate_transaction_hash`) will compute H2 for a transaction whose stored hash is H1. The mismatch causes the block to be rejected as invalid. An unprivileged user can trigger this by submitting any V3 invoke transaction with non-zero L1 gas but zero L2 gas and zero L1 data gas — a configuration the gateway's stateless validator explicitly accepts. This maps to: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing**, and potentially **Critical — Wrong state or receipt from blockifier/execution logic**.

### Likelihood Explanation

The gateway's stateless validator accepts `AllResourceBounds` with any combination of zero/non-zero fields. Transactions with only L1 gas set (L2 and L1_DATA both zero) are a natural pattern for users migrating from pre-0

            return Err(missing("ResourceBounds::l1_gas"));

            return Err(missing("ResourceBounds::l2_gas"));

            ValidResourceBounds::L1Gas(l1_gas)
        } else {
