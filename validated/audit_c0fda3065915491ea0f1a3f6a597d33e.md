### Title
Missing Zero-Address Validation in `Endpoint.initialize` for Critical Protocol Parameters — (File: `core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.initialize` accepts six address parameters — `_sanctions`, `_sequencer`, `_offchainExchange`, `_clearinghouse`, `_verifier`, and `_endpointTx` — and assigns them directly to state variables with no zero-address validation. Since `initialize` is guarded by the `initializer` modifier and can only be called once, and since no post-initialization setters exist for any of these addresses, a zero-address deployment is permanent and unrecoverable without a proxy upgrade. The most severe consequence is that a zero `_clearinghouse` causes `depositCollateral` to silently transfer user tokens to `address(0)`, permanently burning them.

---

### Finding Description

In `Endpoint.initialize`, all six address parameters are stored without any `!= address(0)` guard:

```solidity
// core/contracts/Endpoint.sol L31–L66
function initialize(
    address _sanctions,
    address _sequencer,
    address _offchainExchange,
    IClearinghouse _clearinghouse,
    address _verifier,
    address _endpointTx
) external initializer {
    __Ownable_init();
    __EIP712_init("Nado", "0.0.1");
    sequencer = _sequencer;
    clearinghouse = _clearinghouse;
    offchainExchange = _offchainExchange;
    verifier = IVerifier(_verifier);
    sanctions = ISanctionsList(_sanctions);
    endpointTx = _endpointTx;
    ...
}
``` [1](#0-0) 

A grep across all production contracts confirms there are no post-initialization setters for `sequencer`, `clearinghouse`, `verifier`, or `endpointTx`. The `initializer` modifier from OpenZeppelin makes re-initialization impossible. Any zero address set here is permanent without a proxy upgrade.

The four critical failure modes are:

**1. `_clearinghouse = address(0)` → user funds burned**

`depositCollateralWithReferral` calls `handleDepositTransfer`, which is defined in `EndpointStorage`:

```solidity
// core/contracts/EndpointStorage.sol L111–L119
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);  // ← sends to address(0)
}
``` [2](#0-1) 

If `clearinghouse == address(0)`, `safeTransferTo` sends the deposited tokens to `address(0)`, permanently burning them. The user's ERC-20 tokens are gone with no recourse.

**2. `_sequencer = address(0)` → all sequencer-submitted transactions blocked**

`sequencer` is stored in `EndpointStorage` and used to gate all sequencer-submitted transaction batches. With `sequencer == address(0)`, no valid sequencer can ever satisfy the check (`msg.sender` can never equal `address(0)`), permanently halting all protocol transaction processing. [3](#0-2) 

**3. `_verifier = address(0)` → all signed user operations blocked**

`verifier` is used for all EIP-712 signature verification (withdrawals, liquidations, linked signer operations). A zero verifier causes all such calls to revert. [4](#0-3) 

**4. `_endpointTx = address(0)` → all delegatecall-based operations blocked**

`_delegatecallEndpointTx` has a runtime guard:

```solidity
// core/contracts/Endpoint.sol L72
require(endpointTx != address(0), "Endpoint Tx not set");
``` [5](#0-4) 

This guard exists at runtime but not at initialization time. If `_endpointTx = address(0)`, every call to `submitSlowModeTransaction` and all delegatecall-routed operations revert permanently. [6](#0-5) 

The same class of missing validation exists in `Clearinghouse.initialize` (no check for `_quote`, `_clearinghouseLiq`, `_withdrawPool`) and `BaseEngine._initialize` (no check for `_clearinghouseAddr`, `_offchainExchangeAddr`, `_endpointAddr`, `_admin`). [7](#0-6) [8](#0-7) 

---

### Impact Explanation

The most severe impact is permanent user fund loss. If `_clearinghouse` is set to `address(0)`, every call to `depositCollateral` or `depositCollateralWithReferral` will transfer the user's ERC-20 tokens to `address(0)` via `safeTransferTo(token, address(clearinghouse), amount)`. These tokens are irrecoverably burned. There is no mechanism to recover them — the `initializer` modifier prevents re-initialization, and no `setClearinghouse` setter exists. Secondary impacts include permanent protocol halt (zero sequencer, zero verifier, zero endpointTx), which traps all previously deposited funds inside the Clearinghouse with no withdrawal path.

---

### Likelihood Explanation

Initialization of upgradeable proxy systems involves coordinating multiple contract addresses across a deployment script. A single copy-paste error, an uninitialized variable, or an out-of-order deployment step can result in a zero address being passed. The `initializer` guard makes this a one-shot, irreversible operation. The likelihood is low but non-negligible for complex multi-contract deployments, and the impact is catastrophic (permanent fund loss), making this a high-severity finding.

---

### Recommendation

Add explicit non-zero address checks for all critical address parameters in `Endpoint.initialize`:

```solidity
require(_sequencer != address(0), "zero sequencer");
require(address(_clearinghouse) != address(0), "zero clearinghouse");
require(_verifier != address(0), "zero verifier");
require(_endpointTx != address(0), "zero endpointTx");
require(_offchainExchange != address(0), "zero offchainExchange");
require(_sanctions != address(0), "zero sanctions");
```

Apply the same pattern to:
- `Clearinghouse.initialize`: check `_quote`, `_clearinghouseLiq`, `_withdrawPool`
- `BaseEngine._initialize`: check `_clearinghouseAddr`, `_offchainExchangeAddr`, `_endpointAddr`, `_admin`
- `ContractOwner.initialize`: check `multisig`, `_spotEngine`, `_perpEngine`, `_endpoint`, `_clearinghouse`, `_verifier`

---

### Proof of Concept

1. Deploy `Endpoint` behind a transparent proxy.
2. Call `Endpoint.initialize(validSanctions, validSequencer, validOffchainExchange, address(0), validVerifier, validEndpointTx)` — passing `address(0)` for `_clearinghouse`.
3. The `initializer` modifier marks the contract as initialized. `initialize` cannot be called again.
4. User Alice calls `depositCollateral("default", USDC_PRODUCT_ID, 1000e6)`.
5. `depositCollateralWithReferral` calls `handleDepositTransfer(IERC20Base(usdcToken), alice, 1000e6)`.
6. Inside `handleDepositTransfer`: `safeTransferFrom(usdc, alice, 1000e6)` succeeds — Alice's 1000 USDC leaves her wallet.
7. `safeTransferTo(usdc, address(clearinghouse), 1000e6)` = `safeTransferTo(usdc, address(0), 1000e6)` — 1000 USDC is sent to `address(0)` and permanently burned.
8. Alice has lost 1000 USDC with no recovery path. [9](#0-8) [2](#0-1)
