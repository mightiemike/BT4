### Title
Quote Token Permanently Hardcoded to $1 — Stablecoin Depeg Corrupts Health Calculations and Liquidation Thresholds - (File: `core/contracts/SpotEngine.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

The Nado protocol permanently hardcodes the quote token (stablecoin, `QUOTE_PRODUCT_ID = 0`) price to exactly `ONE` (1e18 = $1.00) with no mechanism to ever update it. This assumption propagates into every subaccount health calculation, every liquidation threshold check, and the minimum deposit gate. During a stablecoin depeg event, all of these values become incorrect, allowing undercollateralized positions to avoid liquidation and enabling users to pass minimum deposit checks with less real value than required.

---

### Finding Description

In `SpotEngine.initialize()`, the quote product's risk store is set with a hardcoded price:

```solidity
_risk().value[QUOTE_PRODUCT_ID] = RiskHelper.RiskStore({
    longWeightInitial: 1e9,
    shortWeightInitial: 1e9,
    longWeightMaintenance: 1e9,
    shortWeightMaintenance: 1e9,
    priceX18: ONE   // <-- permanently $1.00
});
``` [1](#0-0) 

This price is stored in the shared `RISK_STORAGE` slot and is read by `_risk(productId)` in `BaseEngine` for every health computation. [2](#0-1) 

The normal price update path in `Clearinghouse.updatePrice` routes through `productToEngine[txn.productId]`. Because `SpotEngine.initialize()` adds `QUOTE_PRODUCT_ID` directly to `productIds` without calling `_addOrUpdateProduct` (which is the only path that calls `_clearinghouse.registerProduct`), `productToEngine[QUOTE_PRODUCT_ID]` remains `address(0)`. The update silently returns `(0, 0)` without touching the quote price:

```solidity
IProductEngine engine = productToEngine[txn.productId]; // address(0) for QUOTE
if (address(engine) != address(0)) {
    engine.updatePrice(txn.productId, txn.priceX18);
    return (txn.productId, txn.priceX18);
} else {
    return (0, 0);  // quote price is never updated
}
``` [3](#0-2) 

The same assumption is independently hardcoded in `checkMinDeposit`:

```solidity
int128 priceX18 = ONE;
if (productId != QUOTE_PRODUCT_ID) {
    priceX18 = _getPriceX18(productId);
}
return priceX18.mul(amountRealized) >= minDepositAmount;
``` [4](#0-3) 

The hardcoded `priceX18 = ONE` for the quote product flows into `_calculateProductHealth` in `BaseEngine`, which computes:

```solidity
health += amount.mul(weight).mul(risk.priceX18);
``` [5](#0-4) 

For the quote product, `risk.priceX18` is always `ONE`, so 1 unit of the quote token always contributes exactly $1 to health regardless of its real market value.

---

### Impact Explanation

**Health / Liquidation corruption:** All subaccount health values — initial, maintenance, and PnL — are computed using the hardcoded quote price. If the quote stablecoin depegs to $0.95, a subaccount holding 10,000 USDC as collateral has its health overstated by $500. Positions that are genuinely undercollateralized will not be flagged for liquidation, leaving the protocol exposed to bad debt. Conversely, a brief upward depeg could cause healthy accounts to appear undercollateralized. [6](#0-5) 

**Minimum deposit bypass:** `checkMinDeposit` uses `priceX18 = ONE` for the quote product unconditionally. During a depeg, a user depositing quote tokens passes the `MIN_DEPOSIT_AMOUNT` ($0.10) and `MIN_FIRST_DEPOSIT_AMOUNT` ($5) gates with tokens worth less than those thresholds in real USD. [7](#0-6) 

**Insurance fund erosion:** Liquidations that are delayed or skipped due to incorrect health values result in larger losses that must be absorbed by the insurance fund, potentially draining it. [8](#0-7) 

---

### Likelihood Explanation

The quote token is a stablecoin (USDC or equivalent). Historical stablecoin depeg events are well-documented — USDC itself depegged to ~$0.87 during the March 2023 SVB crisis. The Nado protocol is deployed on Ink Chain, a newer network where liquidity is thinner and depeg events may be more pronounced. The bug requires no privileged access — any market event that moves the quote token off its peg triggers the incorrect accounting automatically.

---

### Recommendation

1. Register the quote product in `productToEngine` during `SpotEngine.initialize()` so that the sequencer's `UpdatePrice` transaction can update its price.
2. Remove the hardcoded `priceX18 = ONE` branch in `checkMinDeposit` and route the quote product through `_getPriceX18(productId)` like all other products.
3. Add a Chainlink or equivalent oracle feed for the quote token and include it in the sequencer's regular price update cycle.

---

### Proof of Concept

1. Protocol is deployed with USDC as the quote token. `SpotEngine.initialize()` sets `_risk().value[0].priceX18 = ONE`.
2. A trader deposits 10,000 USDC and opens a leveraged perp position. Their maintenance health is calculated as `10,000 * 1.0 * weight = $10,000`.
3. USDC depegs to $0.90 (as occurred during the SVB crisis). Real collateral value is $9,000.
4. The sequencer submits an `UpdatePrice` transaction for `productId = 0`. `productToEngine[0] == address(0)`, so `Clearinghouse.updatePrice` returns `(0, 0)` without updating the quote price.
5. `getHealth(subaccount, MAINTENANCE)` still returns a value based on `priceX18 = ONE`, so the position appears healthy.
6. `isUnderMaintenance` returns `false`; liquidation is blocked.
7. The position continues to accumulate losses. When USDC eventually re-pegs, the protocol has absorbed bad debt that should have been liquidated. [1](#0-0) [3](#0-2) [9](#0-8)

        IProductEngine.HealthType healthType
    ) internal returns (int128 health) {

        (int128 amount, int128 quoteAmount) = _getBalance(
            productId,
            subaccount

        int128 weight = RiskHelper._getWeightX18(risk, amount, healthType);
        health += quoteAmount;

        if (amount != 0) {
            if (weight == 2 * ONE) {
                return -INF;
            }
            health += amount.mul(weight).mul(risk.priceX18);
            emit PriceQuery(productId);
        }
    }
