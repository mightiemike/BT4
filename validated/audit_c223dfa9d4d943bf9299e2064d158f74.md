Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the end-user, allowing any caller to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool binds to `msg.sender` of `pool.swap` — the direct caller, not the originating user. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router contract address. If the pool admin allowlists the router to enable router-mediated swaps for curated users, every unprivileged address can bypass the allowlist by calling the router, because the extension evaluates `allowedSwapper[pool][router]` (true) instead of `allowedSwapper[pool][actual_user]`.

## Finding Description

**Root cause — wrong actor bound to the allowlist check.**

In `MetricOmmPool.swap`, the pool passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first positional argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks that value against the per-pool allowlist: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router is the direct caller of `pool.swap` and passes empty bytes `""` as `extensionData`, so the originating user address is never forwarded: [4](#0-3) 

The same mis-binding applies to `exactOutputSingle` (L136-137), `exactInput` (L104-112), and `exactOutput` (L165-181), all of which call `pool.swap` directly with the router as `msg.sender` and pass empty or opaque `extensionData` that does not encode the originating user. [5](#0-4) 

The exact corrupted value is `sender = address(router)` where `sender = address(actual_user)` is required. The broken invariant is `allowedSwapper[pool][actual_user]` — the check is applied to the wrong address, so the guard fails open for any user who routes through the public router.

## Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` faces an inescapable dilemma: not allowlisting the router breaks UX for legitimate users, while allowlisting the router opens the gate to every unprivileged address. Any address explicitly excluded by the admin can execute swaps on the curated pool by routing through `MetricOmmSimpleRouter`. This breaks the core curation invariant — the pool's liquidity is exposed to actors the admin intended to gate — which can result in direct loss of LP principal through adversarial or uninformed swaps on a pool designed for a restricted audience. This constitutes broken core pool functionality causing loss of funds and an admin-boundary break reachable by an unprivileged path.

## Likelihood Explanation

The trigger is fully unprivileged: any address can call `MetricOmmSimpleRouter.exactInputSingle`. The only precondition is that the pool admin has allowlisted the router address, which is the expected and natural operational step to make the router usable for the pool's intended users. The admin has no on-chain signal that allowlisting the router opens the gate to everyone. No special tokens, malicious setup, or privileged access is required. The bypass is repeatable on every swap.

## Recommendation

The `SwapAllowlistExtension` must gate the economically relevant actor — the end-user — not the intermediary contract. Two sound approaches:

1. **Router-side forwarding:** Have `MetricOmmSimpleRouter` encode `msg.sender` (the actual user) into `extensionData` for each hop. Have `SwapAllowlistExtension.beforeSwap` decode and check that address when `sender` is a known router, falling back to checking `sender` directly when the caller is not a router.

2. **Extension-side originator check:** Require the router to pass the originating user address as a typed field in `extensionData`; the extension verifies the field is present and checks it against the allowlist instead of `sender`.

Either way, the invariant must hold: `allowedSwapper[pool][actual_user]` is checked regardless of which supported periphery entrypoint reaches the pool.

## Proof of Concept

```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension configured
  admin calls setAllowedToSwap(pool, alice, true)   // alice is the intended curated user
  admin calls setAllowedToSwap(pool, router, true)  // to let alice use the router
  bob is NOT allowlisted

Attack:
  bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(...)          [msg.sender = router]
    → pool calls _beforeSwap(router, ...)
    → SwapAllowlistExtension.beforeSwap(router, ...)
    → checks allowedSwapper[pool][router]  ← TRUE (admin allowlisted router)
    → swap proceeds for bob

Result:
  bob, who was explicitly excluded, executes a swap on the curated pool.
  The allowlist guard is silently bypassed through the supported periphery path.

Foundry test outline:
  1. Deploy pool with SwapAllowlistExtension as a before-swap extension.
  2. Pool admin allowlists alice and the router address.
  3. Call pool.swap() directly from bob → expect revert NotAllowedToSwap.
  4. Call MetricOmmSimpleRouter.exactInputSingle() from bob → expect success (demonstrates bypass).
```
