### Title
Missing Underflow Guard in `remove_ghost_pair` Corrupts Pair-Limit Accounting in Release Builds - (File: src/allocator.rs)

### Summary
`remove_ghost_pair` in `src/allocator.rs` protects against subtracting more ghost pairs than are tracked only with a `debug_assert!`, which is compiled away in release builds. An attacker who can trigger this path with a crafted CLVM byte stream can silently wrap the `ghost_pairs` counter to a near-`usize::MAX` value, permanently disabling the `MAX_NUM_PAIRS` enforcement in `add_ghost_pair` for the lifetime of that `Allocator` instance.

### Finding Description

`remove_ghost_pair` is the counterpart to `add_ghost_pair`. `add_ghost_pair` pre-charges the pair budget for back-reference deserialization; `remove_ghost_pair` refunds that charge when the real pairs are materialised. The invariant is that `ghost_pairs >= amount` at every call site.

```rust
// src/allocator.rs:784-789
pub fn remove_ghost_pair(&mut self, amount: usize) -> Result<()> {
    // currently let this panic with overflow if we go below 0 to debug if/where it happens
    debug_assert!(self.ghost_pairs >= amount);
    self.ghost_pairs -= amount;
    Ok(())
}
```

The comment itself acknowledges the risk ("let this panic with overflow … to debug"). `debug_assert!` is erased by the compiler in release mode (`--release`), so the subtraction executes unconditionally. In Rust, integer overflow in release mode wraps silently for primitive arithmetic. If `self.ghost_pairs < amount`, the result wraps to approximately `usize::MAX`.

`add_ghost_pair` then reads the corrupted counter:

```rust
// src/allocator.rs:775-780
pub fn add_ghost_pair(&mut self, amount: usize) -> Result<()> {
    if MAX_NUM_PAIRS - self.ghost_pairs - self.pair_vec.len() < amount {
        return Err(EvalErr::TooManyPairs);
    }
    self.ghost_pairs += amount;
    Ok(())
}
```

When `ghost_pairs` is near `usize::MAX`, the subtraction `MAX_NUM_PAIRS - self.ghost_pairs` itself underflows, producing a large positive value. The guard condition `large_value - pair_vec.len() < amount` evaluates to `false` for any realistic `amount`, so `Err(TooManyPairs)` is never returned. The pair budget is effectively unlimited for the rest of that execution.

### Impact Explanation

The `MAX_NUM_PAIRS` limit is a consensus-critical resource bound. Nodes that enforce it reject programs that exceed it; nodes that do not enforce it accept them. If a release-mode node processes a crafted CLVM byte stream that triggers the underflow, its `Allocator` silently accepts arbitrarily many pairs for the remainder of that program's execution. A debug-mode node (or a node that has not yet processed the trigger) would reject the same program. This is a **consensus divergence**: two honest nodes running the same CLVM program reach different accept/reject decisions, which is the highest-severity class of bug in a blockchain VM.

### Likelihood Explanation

The trigger requires a CLVM byte stream (attacker-controlled input accepted at every public API entry point) that causes `remove_ghost_pair` to be called with `amount > ghost_pairs`. The back-reference deserializer (`src/serde/de_br.rs`) is the only production caller of both `add_ghost_pair` and `remove_ghost_pair`. A carefully crafted back-reference sequence that materialises fewer pairs than were pre-charged, then calls `remove_ghost_pair` with the full pre-charged amount, would satisfy the trigger. The attacker needs only to submit a transaction containing such bytes to a full node running a release build.

### Recommendation

Replace the `debug_assert!` with a proper checked subtraction that returns an error on underflow, consistent with the `Result<()>` return type already present:

```rust
pub fn remove_ghost_pair(&mut self, amount: usize) -> Result<()> {
    self.ghost_pairs = self.ghost_pairs.checked_sub(amount)
        .ok_or(EvalErr::TooManyPairs)?;
    Ok(())
}
```

This makes the invariant enforcement unconditional in both debug and release builds and surfaces any accounting bug as a recoverable error rather than silent state corruption.

### Proof of Concept

1. Craft a CLVM byte stream using the back-reference (`0xfe`) encoding that pre-charges N ghost pairs via `add_ghost_pair(N)` during the first pass, then during pair materialisation calls `remove_ghost_pair(N+1)` (one more than was charged).
2. Submit the byte stream to a release-mode node via the public deserialization API.
3. `ghost_pairs` wraps to `usize::MAX - 0` (approximately).
4. Any subsequent `add_ghost_pair` call in the same execution passes the guard unconditionally.
5. The node accepts a program that allocates more than `MAX_NUM_PAIRS` pairs; a debug-mode node rejects it — consensus split confirmed. [1](#0-0) [2](#0-1)
