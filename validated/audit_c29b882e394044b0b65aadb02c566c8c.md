Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Allowlist Fully Bypassed via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is `msg.sender` of the `pool.swap()` call — the router's address when `MetricOmmSimpleRouter` is used. A pool admin who allowlists the router to enable router-mediated swaps inadvertently grants every user access to the curated pool, rendering per-user allowlist entries meaningless.

## Finding Description

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value unchanged as the first argument to every registered extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks that argument against the per-pool allowlist: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutputSingle` / `exactOutput`) is used, the router calls `pool.swap()` directly: [4](#0-3) 

At that point `msg.sender` inside the pool is the router, so the extension receives `sender = router`. The check becomes `allowedSwapper[pool][router]` instead of the intended `allowedSwapper[pool][actual_end_user]`. The real caller's identity (`msg.sender` stored in transient storage via `_setNextCallbackContext`) is never consulted by the extension.

## Impact Explanation

**High — allowlist bypass with direct fund impact on curated pools.** A pool admin who calls `extension.setAllowedToSwap(pool, address(router), true)` to enable router-mediated swaps for allowlisted users inadvertently opens the pool to every user. Any non-allowlisted attacker can call `router.exactInputSingle(...)` and the extension check passes because `allowedSwapper[pool][router] == true`. The per-user allowlist entries for excluded users are never evaluated. Unauthorized users can drain liquidity at oracle prices from a pool designed to be restricted.

Secondary impact: if the admin allowlists individual users but not the router, those users cannot swap through the router even though they are individually permitted, breaking expected UX.

## Likelihood Explanation

**High.** `SwapAllowlistExtension` is a production periphery contract explicitly documented as "Gates `swap` by swapper address, per pool." Pool admins deploying it alongside the official `MetricOmmSimpleRouter` will naturally allowlist the router to enable router-mediated swaps. The bypass requires no special privileges, no malicious setup, and no non-standard tokens — any user can call `router.exactInputSingle` with a valid swap path.

## Recommendation

The pool must forward the original end-user identity through the extension interface, or the extension must derive it from a trusted source:

1. **Pass the original caller through the router**: The router already stores the real `msg.sender` in transient storage via `_setNextCallbackContext`. The extension could read this from a trusted router registry, or the router could pass it as a standardized prefix in `extensionData` that the extension decodes and verifies.
2. **Router registry check**: The extension could compare `sender` against a registry of known routers and, when `sender` is a router, require the real user identity to be ABI-encoded in `extensionData` with a transient-storage proof.
3. **Simplest safe fix**: Prohibit the router from being added to the allowlist and require all curated-pool users to call `pool.swap()` directly, with a clear NatSpec warning in `setAllowedToSwap`.

## Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension configured.
2. Pool admin calls:
       extension.setAllowedToSwap(pool, address(router), true);
   (intending to allow router-mediated swaps for allowlisted users)
3. Non-allowlisted attacker calls:
       router.exactInputSingle(ExactInputSingleParams({
           pool: pool,
           recipient: attacker,
           zeroForOne: true,
           amountIn: LARGE_AMOUNT,
           ...
       }));
4. Router calls pool.swap(...) — msg.sender in pool = router.
5. Pool calls _beforeSwap(router, ...) → extension receives sender = router.
6. Extension checks allowedSwapper[pool][router] == true → passes.
7. Attacker swaps on the curated pool without being individually allowlisted.
   The check allowedSwapper[pool][attacker] == false is never evaluated.
```
