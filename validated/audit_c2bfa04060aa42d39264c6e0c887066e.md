### Title
Unbounded Validator Event Log Iteration in `processEventsAndClaimBonus` Permanently Locks User Staked Tokens — (`x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`processEventsAndClaimBonus` walks every pending `ValidatorEvent` for a position's validator since `pos.LastEventSeq` with no upper bound. Because every exit path (undelegate, redelegate, add-to-position, clear-position, exit-with-delegation) calls this function before mutating state, a position whose event backlog grows large enough will exhaust transaction gas on every call, permanently locking the user's staked tokens inside the tier module with no recovery path.

---

### Finding Description

**Root cause — unbounded KV-store walk:**

`getValidatorEventsSince` opens a range iterator from `(valAddr, startSeq+1)` to the end of the validator's prefix and materialises every entry into a slice before returning:

```go
// x/tieredrewards/keeper/validator_events.go:67-81
func (k Keeper) getValidatorEventsSince(..., startSeq uint64) ([]EventEntry, error) {
    rng := collections.NewPrefixedPairRange[sdk.ValAddress, uint64](valAddr).
        StartExclusive(startSeq)
    var entries []EventEntry
    err := k.ValidatorEvents.Walk(ctx, rng, func(...) (bool, error) {
        entries = append(entries, EventEntry{...})
        return false, nil   // never stops early
    })
    ...
}
```

The caller iterates the returned slice and performs a KV read + write per entry (decrement reference count):

```go
// x/tieredrewards/keeper/claim_rewards.go:172-198
for _, entry := range events {
    ...
    if err := k.decrementEventRefCount(ctx, valAddr, entry.Seq); err != nil {
        return nil, err
    }
}
```

**Event accumulation mechanism:**

Staking hooks append one `ValidatorEvent` per validator state transition (slash, unbond, bond) whenever `PositionCountByValidator > 0`:

```go
// x/tieredrewards/keeper/hooks.go:27-49 (AfterValidatorBeginUnbonding)
// x/tieredrewards/keeper/hooks.go:53-75 (AfterValidatorBonded)
// x/tieredrewards/keeper/hooks.go:100-123 (BeforeValidatorSlashed)
```

Each event's `ReferenceCount` is set to the number of delegated positions at that moment. An event is only garbage-collected when every position that was present at event time has processed it. A position that never claims rewards keeps every subsequent event alive indefinitely.

**All exit paths are blocked:**

`processEventsAndClaimBonus` is called unconditionally by every state-mutating path:

```go
// x/tieredrewards/keeper/claim_rewards.go:59-77  (claimRewardsAndUpdateTierPositions)
// x/tieredrewards/keeper/claim_rewards.go:87-103 (claimRewards — used by Undelegate, Redelegate, AddToPosition, ClearPosition, ExitTierWithDelegation)
```

The ADR confirms: "MsgTierUndelegate, MsgTierRedelegate, MsgAddToTierPosition, MsgClearPosition, and MsgExitTierWithDelegation all claim rewards before modifying the position."

`MsgWithdrawFromTier` requires the position to already be undelegated, so it is also unreachable if undelegation is blocked.

---

### Impact Explanation

Once the event backlog for a position exceeds the gas budget of a single transaction, every transaction that touches that position fails. Because undelegation, redelegation, and exit-with-delegation all call `processEventsAndClaimBonus` first, the user's staked tokens — held at the position's per-position delegator address — become permanently inaccessible. The user cannot undelegate, cannot transfer the delegation back, and cannot withdraw. The staked principal is locked inside the tier module with no recovery path.

The exact corrupted invariant: `pos.DelegatorAddress` holds bonded tokens that the position owner can never retrieve.

---

### Likelihood Explanation

A validator operator (a permissionless network participant) can deliberately cycle their validator through jail/unjail states. Each cycle appends two events (UNBOND + BOND). With a `DowntimeJailDuration` of 10 minutes, a malicious operator can accumulate ~144 events per day. At roughly 3 000–5 000 gas per event (one KV read for the Walk entry + one read/write for `decrementEventRefCount`), and a typical block gas limit of 40 M gas, ~8 000–13 000 events suffice to exhaust gas — reachable in under 100 days of sustained cycling.

The scenario also arises naturally: a validator with chronic downtime accumulates UNBOND/BOND events without any malicious intent, and a user who simply does not claim rewards for an extended period on such a validator will eventually find all their exit transactions failing.

---

### Recommendation

Add a per-claim cap on the number of events processed in a single call to `processEventsAndClaimBonus`. If the backlog exceeds the cap, process only the first N events and advance `LastEventSeq` accordingly, allowing the user to drain the backlog across multiple transactions. Alternatively, enforce a maximum event backlog per validator (analogous to the maximum-limit fix applied in the referenced audit) by capping the number of live events per validator in `appendValidatorEvent`.

---

### Proof of Concept

1. User calls `MsgLockTier` on validator V; position created with `LastEventSeq = 0`.
2. Validator V is jailed (downtime or deliberate) and unjails itself repeatedly — each cycle appends one UNBOND event and one BOND event to `ValidatorEvents[(V, seq)]`. After K cycles, `ValidatorEventSeq[V] = 2K` and all 2K events have `ReferenceCount ≥ 1` (the user's position has not processed any of them).
3. User submits `MsgClaimTierRewards` (or any of `MsgTierUndelegate`, `MsgTierRedelegate`, `MsgExitTierWithDelegation`). The handler calls `processEventsAndClaimBonus`, which calls `getValidatorEventsSince(ctx, V, 0)`, walking all 2K entries. Each entry triggers a `decrementEventRefCount` KV write. Total gas ≈ 2K × 4 000. For K ≈ 5 000, gas ≈ 40 M → transaction out-of-gas, reverts.
4. Every subsequent attempt by the user to exit or claim also reverts. The user's staked tokens at `pos.DelegatorAddress` are permanently locked.

func (k Keeper) getValidatorEventsSince(ctx context.Context, valAddr sdk.ValAddress, startSeq uint64) ([]EventEntry, error) {
	// Range from (valAddr, startSeq+1) to end of valAddr prefix.
	rng := collections.NewPrefixedPairRange[sdk.ValAddress, uint64](valAddr).
		StartExclusive(startSeq)

	var entries []EventEntry
	err := k.ValidatorEvents.Walk(ctx, rng, func(key collections.Pair[sdk.ValAddress, uint64], event types.ValidatorEvent) (bool, error) {
		entries = append(entries, EventEntry{Seq: key.K2(), Event: event})
		return false, nil
	})
	if err != nil {
		return nil, err
	}
	return entries, nil
}

// claimRewardsAndUpdateTierPositions claims base and bonus rewards for all
// delegated positions in the given tier.
func (k Keeper) claimRewardsAndUpdateTierPositions(ctx context.Context, tierId uint32) error {
	ids, err := k.getPositionsIdsByTier(ctx, tierId)
	if err != nil {
		return err
	}
	if len(ids) == 0 {
		return nil
	}

	for _, id := range ids {
		pos, err := k.getPositionState(ctx, id)
		if err != nil {
			return err
		}
		if !pos.IsDelegated() {
			continue
		}

		if _, err := k.claimBaseRewards(ctx, pos); err != nil {
			return err
		}
		if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
			return err
		}
		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return err
		}
	}

	return nil
}

func (h Hooks) AfterValidatorBeginUnbonding(ctx context.Context, _ sdk.ConsAddress, valAddr sdk.ValAddress) error {
	count, err := h.k.getPositionCountForValidator(ctx, valAddr)
	if err != nil {
		return err
	}
	if count == 0 {
		return nil
	}

	tokensPerShare, err := h.k.getTokensPerShare(ctx, valAddr)
	if err != nil {
		return err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	_, err = h.k.appendValidatorEvent(ctx, valAddr, types.ValidatorEvent{
		Height:         sdkCtx.BlockHeight(),
		Timestamp:      sdkCtx.BlockTime(),
		EventType:      types.ValidatorEventType_VALIDATOR_EVENT_TYPE_UNBOND,
		TokensPerShare: tokensPerShare,
		ReferenceCount: count,
	})
	return err
}
