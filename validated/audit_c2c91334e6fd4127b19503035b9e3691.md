### Title
Linked Signer Can Self-Replace Its Own Authorization Without Subaccount Owner Consent — (`core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This means the current linked signer — not just the subaccount owner — can sign a `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with any address. A compromised or malicious linked signer can therefore permanently redirect signing authority to an attacker-controlled address, escalating from "delegated API key" to "full account controller," without the subaccount owner's knowledge or consent.

---

### Finding Description

The sequencer path for `LinkSigner` in `processTransactionImpl` is:

```solidity
// EndpointTx.sol L576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
```

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which accepts a signature from **either** the subaccount owner **or** the current linked signer:

```solidity
// Verifier.sol L297–303
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),   // linked signer accepted
    ERR_INVALID_SIGNATURE
);
```

The slow mode path for the same `LinkSigner` transaction type enforces the opposite constraint — only the subaccount owner (`msg.sender`) may set the linked signer:

```solidity
// EndpointTx.sol L232–239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // msg.sender must == subaccount owner
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

This inconsistency reveals the design intent: only the owner should be able to change the linked signer. The sequencer path breaks that invariant by accepting the linked signer's own signature for this mutation.

The root cause is structurally identical to the reference bug: just as `transferPower` lacked a check that `m.agent == msg.sender` before overwriting `m.agent`, the sequencer-path `LinkSigner` handler lacks a check that the signer is the subaccount owner (not merely the current linked signer) before overwriting `linkedSigners[subaccount]`.

---

### Impact Explanation

A linked signer that is compromised (e.g., a leaked API key from a trading bot) can:

1. Sign a `LinkSigner` transaction setting `linkedSigners[victim_subaccount] = attacker_address`.
2. The sequencer processes it; `validateSignedTx` passes because the current linked signer's signature is accepted.
3. `linkedSigners[victim_subaccount]` is now the attacker's address.
4. The attacker can sign `WithdrawCollateral`, `TransferQuote`, and order-matching transactions on behalf of the victim.
5. The victim's only recovery path is a slow mode `LinkSigner` transaction, which is subject to a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`). During that window the attacker can drain the subaccount.

The corrupted state is `linkedSigners[subaccount]` — a mapping that gates signing authority for all subsequent sequencer-path transactions for that subaccount.

**Impact: 3 / 5** — Full subaccount takeover is possible, but requires a prior linked signer key compromise as a precondition.

---

### Likelihood Explanation

Linked signers are explicitly designed as API keys for automated trading bots and market makers. Private key exposure through server compromise, misconfigured secrets management, or supply-chain attacks on trading software is a realistic and historically observed threat. Once a linked signer key is leaked, the attacker can execute this escalation in a single sequencer submission with no on-chain footprint until the state is already changed.

**Likelihood: 3 / 5** — Requires linked signer key compromise, but that is a common real-world scenario for automated trading infrastructure.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. Only the subaccount owner's signature should be accepted when modifying signing authority:

```solidity
// EndpointTx.sol L581–587  — change true → false
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // owner-only: linked signer must not be able to replace itself
);
```

This aligns the sequencer path with the slow mode path, which already enforces owner-only authorization for the same operation.

---

### Proof of Concept

**Setup:**
- Alice owns subaccount `alice_sub` (bytes32 encoding of her address + subaccount name).
- Alice has previously set `linkedSigners[alice_sub] = api_key_addr` (her trading bot's key).

**Attack:**
1. Attacker obtains `api_key_addr`'s private key (e.g., via server breach).
2. Attacker constructs a `SignedLinkSigner` transaction:
   - `tx.sender = alice_sub`
   - `tx.signer = bytes32(uint256(uint160(attacker_addr)))`
   - `tx.nonce = nonces[address(uint160(bytes20(alice_sub)))]`
   - Signs with `api_key_addr`.
3. Attacker submits the transaction to the sequencer.
4. Sequencer calls `processTransactionImpl`; `validateSignedTx(..., true)` passes because `ECDSA.recover(digest, sig) == api_key_addr == getLinkedSigner(alice_sub)`.
5. `linkedSigners[alice_sub] = attacker_addr` is written.
6. Attacker signs a `WithdrawCollateralV2` transaction with `sendTo = address(0)` (which allows linked signer) and drains Alice's collateral.
7. Alice submits a slow mode `LinkSigner` to reset, but the 3-day delay means the attacker has a full window to act.

**Corrupted invariant:** `linkedSigners[alice_sub]` now points to the attacker, not Alice's intended API key. All subsequent `allowLinkedSigner = true` sequencer transactions for `alice_sub` are under attacker control. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
