### Title
Incorrect `verifyingContract` in EIP-712 Domain Separator Enables Cross-Contract Order Replay — (`File: core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.getDigest` constructs the EIP-712 domain separator using `address(uint160(productId))` as the `verifyingContract` field instead of `address(this)`. This is a direct analog to M-12: an incorrect hardcoded value substituted for the correct identifying constant in a critical cryptographic computation. The domain separator no longer binds signed orders to the actual `OffchainExchange` contract address, breaking the cross-contract replay protection that EIP-712 is designed to provide.

---

### Finding Description

In `OffchainExchange.getDigest`, the domain separator is built as:

```solidity
bytes32 domainSeparator = keccak256(
    abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),
        _EIP712VersionHash(),
        block.chainid,
        address(uint160(productId))   // ← wrong: should be address(this)
    )
);
``` [1](#0-0) 

The EIP-712 standard ([EIP-712 §Domain Separator](https://eips.ethereum.org/EIPS/eip-712)) defines `verifyingContract` as *"The Ethereum address of the contract that will verify the signature."* Substituting `address(uint160(productId))` (e.g., `address(1)`, `address(2)`, …) produces a near-zero address that has no relationship to the deployed `OffchainExchange` contract. The domain separator is therefore identical across every `OffchainExchange` deployment that shares the same `name`, `version`, and `chainId`.

This digest is consumed directly in `matchOrders`:

```solidity
ordersInfo.taker.digest = getDigest(callState.productId, taker.order);
...
_validateOrder(..., ordersInfo.taker.digest, true, txn.takerLinkedSigner)
``` [2](#0-1) 

`_validateOrder` calls `_checkSignature` with that digest:

```solidity
_checkSignature(order.sender, orderDigest, linkedSigner, signedOrder.signature)
``` [3](#0-2) 

`_checkSignature` performs `ECDSA.recover(digest, signature)` and accepts the result if it matches the subaccount owner or linked signer — no further binding to the contract address exists. [4](#0-3) 

The `filledAmounts` mapping that tracks partial fills is per-contract storage:

```solidity
mapping(bytes32 => int128) public filledAmounts;
``` [5](#0-4) 

On any second `OffchainExchange` instance, `filledAmounts[digest]` starts at zero, so a fully-filled order on the first instance can be replayed in full on the second.

---

### Impact Explanation

If Nado deploys a second `OffchainExchange` contract — whether for a second clearinghouse, after an upgrade, or after a redeployment — every order signature ever produced for the first instance is cryptographically valid on the second instance. The sequencer (or any party who can submit `matchOrders` transactions) can replay previously executed orders against the new contract, causing:

- Unauthorized position changes for traders whose orders were already fully filled
- Double-execution of trades, corrupting balances in `SpotEngine` or `PerpEngine`
- Potential fund loss if the replayed order executes at a stale price or against a counterparty who did not intend to trade again

---

### Likelihood Explanation

Nado's hardhat config already defines two distinct networks (`test` chainId 763373, `prod` chainId 57073), and the protocol is upgradeable. Redeployment or multi-clearinghouse expansion is a realistic operational event. The incorrect `verifyingContract` value is present in the current production code and requires no attacker privilege — only the ability to submit a `matchOrders` transaction through the sequencer path, which is the normal order-execution flow. [6](#0-5) 

---

### Recommendation

Replace `address(uint160(productId))` with `address(this)` in the domain separator construction inside `getDigest`:

```solidity
bytes32 domainSeparator = keccak256(
    abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),
        _EIP712VersionHash(),
        block.chainid,
        address(this)   // correct: binds to this contract's address
    )
);
```

This ensures each `OffchainExchange` deployment produces a unique domain separator, preventing cross-contract order replay.

---

### Proof of Concept

1. User signs an `Order` for `productId = 1` on `OffchainExchange_A` (deployed at `0xAAAA…`). The domain separator's `verifyingContract` is `address(1)` — not `0xAAAA…`.
2. Nado redeploys `OffchainExchange_B` at `0xBBBB…` (same chain, same name/version). Its domain separator for `productId = 1` is also `address(1)`.
3. `getDigest(1, order)` returns the **identical** bytes32 on both contracts.
4. On `OffchainExchange_B`, `filledAmounts[digest] == 0`.
5. The sequencer submits `matchOrders` to `Endpoint` pointing at `OffchainExchange_B` with the user's original signature. `_checkSignature` recovers the correct signer and returns `true`.
6. The order executes again in full on `OffchainExchange_B`, corrupting the user's position and balances. [7](#0-6)

    function _checkSignature(
        bytes32 subaccount,
        bytes32 digest,
        address linkedSigner,
        bytes memory signature
    ) internal view virtual returns (bool) {
        address signer = ECDSA.recover(digest, signature);
        return
            (signer != address(0)) &&
            (signer == address(uint160(bytes20(subaccount))) ||
                signer == linkedSigner);
    }
