### Title
Strict `ValidateDenomID` in `MsgTransferNFT.ValidateBasic()` Permanently Locks IBC NFT Vouchers from On-Chain Transfer — (File: `x/nft/types/msgs.go`)

---

### Summary

`MsgTransferNFT.ValidateBasic()` calls the strict `ValidateDenomID` instead of the IBC-aware `ValidateDenomIDWithIBC`. IBC NFT vouchers minted by `x/nft-transfer` always carry denom IDs of the form `ibc/<64-hex-hash>` (68 characters, containing `/` and uppercase hex), which the strict validator unconditionally rejects. Any user who receives an IBC NFT voucher on this chain is permanently unable to transfer it to another address on-chain via `MsgTransferNFT`, `MsgEditNFT`, or `MsgBurnNFT`.

---

### Finding Description

The `x/nft` module defines two denom ID validators:

1. `ValidateDenomID` — strict: lowercase alphanumeric only (`^[a-z0-9]+$`), 3–64 characters, must begin with a letter.
2. `ValidateDenomIDWithIBC` — IBC-aware: accepts the strict format **or** `ibc/<64-hex-hash>` (68 characters). [1](#0-0) 

`MsgTransferNFT.ValidateBasic()` uses the **strict** validator:

```go
func (msg MsgTransferNFT) ValidateBasic() error {
    if err := ValidateDenomID(msg.DenomId); err != nil {  // strict, NOT IBC-aware
        return err
    }
    ...
}
``` [2](#0-1) 

The same strict check is applied in `MsgEditNFT.ValidateBasic()` and `MsgBurnNFT.ValidateBasic()`: [3](#0-2) [4](#0-3) 

When `x/nft-transfer` receives an IBC packet, `processReceivedPacket` mints NFT vouchers under a denom ID of the form `ibc/<64-hex-hash>` — for example `ibc/27394FB092D2ECCD56123C74F36E4C1F926001CEADA9CA97EA622B25F41E5EB2`. This ID is 68 characters long, contains `/`, and contains uppercase hex characters — all of which fail `ValidateDenomID`. [5](#0-4) 

The `ValidateDenomIDWithIBC` function is already used in `ValidateGenesis` and CLI query handlers, confirming the codebase knows IBC denom IDs are valid at those call sites — but the runtime message path was not updated. [6](#0-5) 

---

### Impact Explanation

Every IBC NFT voucher received on this chain is permanently non-transferable on-chain. The owner cannot:
- Transfer the voucher to another address (`MsgTransferNFT` rejected at `ValidateBasic`)
- Edit the voucher's metadata (`MsgEditNFT` rejected at `ValidateBasic`)
- Burn the voucher (`MsgBurnNFT` rejected at `ValidateBasic`)

The NFT is minted successfully by the IBC receive path (the keeper's `MintNFT` has no such restriction), but the owner is then permanently locked out of all on-chain operations on it. The corrupted invariant is **NFT ownership**: the owner field is set, but the owner cannot exercise any transfer rights on-chain.

---

### Likelihood Explanation

This affects **every** IBC NFT transfer to this chain. The `x/nft-transfer` module is a core production module. Any counterparty chain sending an NFT to this chain triggers the condition. No special attacker capability is required — a normal user initiating a standard `MsgTransferNFT` transaction with their received IBC voucher denom ID is sufficient to observe the failure.

---

### Recommendation

Replace `ValidateDenomID` with `ValidateDenomIDWithIBC` in the `ValidateBasic` methods of `MsgTransferNFT`, `MsgEditNFT`, and `MsgBurnNFT` in `x/nft/types/msgs.go`. [7](#0-6) 

---

### Proof of Concept

1. Chain A sends an NFT with `classID = "mynft"` to this chain via IBC `x/nft-transfer`.
2. `OnRecvPacket` → `processReceivedPacket` → `k.nftKeeper.MintNFT(ctx, voucherClassID, tokenID, ...)` succeeds, minting the NFT under denom ID `ibc/27394FB092D2ECCD56123C74F36E4C1F926001CEADA9CA97EA622B25F41E5EB2`.
3. The receiver submits: `chain-maind tx nft transfer <recipient> ibc/27394FB092D2ECCD56123C74F36E4C1F926001CEADA9CA97EA622B25F41E5EB2 <tokenID>`
4. `MsgTransferNFT.ValidateBasic()` → `ValidateDenomID("ibc/2739...")`:
   - Length = 68 > `MaxDenomLen` (64) → **rejected**
   - Contains `/` → not alphanumeric → **rejected**
   - Contains uppercase → `^[a-z0-9]+$` fails → **rejected**
5. Transaction is rejected. The NFT is permanently locked to the receiver's address with no on-chain recourse. [8](#0-7) [9](#0-8)

// ValidateDenomID verifies whether the parameters are legal.
func ValidateDenomID(denomID string) error {
	if len(denomID) < MinDenomLen || len(denomID) > MaxDenomLen {
		return sdkerrors.Wrapf(ErrInvalidDenom, "the length of denom(%s) only accepts value [%d, %d]", denomID, MinDenomLen, MaxDenomLen)
	}
	if !IsBeginWithAlpha(denomID) || !IsAlphaNumeric(denomID) {
		return sdkerrors.Wrapf(ErrInvalidDenom, "the denom(%s) only accepts lowercase alphanumeric characters, and begin with an english letter", denomID)
	}
	return nil
}

// ValidateDenomIDWithIBC verifies whether the parameters are legal and considers IBC denom IDs when checking.
func ValidateDenomIDWithIBC(denomID string) error {
	if strings.HasPrefix(denomID, IBCPrefix) {
		if len(denomID) != IBCDenomLen {
			return sdkerrors.Wrapf(ErrInvalidDenom, "the length of ibc denom(%s) only accepts value [%d]", denomID, IBCDenomLen)
		}
		if _, err := hex.DecodeString(denomID[len(IBCPrefix):]); err != nil {
			return sdkerrors.Wrapf(ErrInvalidDenom, "the hash of ibc denom(%s) must be valid hex", denomID)
		}

		return nil
	}

	return ValidateDenomID(denomID)
}
