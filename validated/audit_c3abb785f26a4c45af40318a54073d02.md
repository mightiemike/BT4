### Title
Stale Spot Interest Multipliers Used in Health Checks Enable Liquidation Bypass and Wrongful Liquidation — (`core/contracts/SpotEngineState.sol`)

---

### Summary

`SpotEngine` stores user balances as normalized amounts (`amountNormalized`) that must be multiplied by `cumulativeDepositsMultiplierX18` or `cumulativeBorrowsMultiplierX18` to obtain actual balances. These multipliers are only updated when a `SpotTick` transaction is processed by the sequencer. Because `SpotTick` and `LiquidateSubaccount` are independent transaction types with no on-chain ordering constraint, health checks during liquidation and withdrawal can operate on stale multipliers, producing incorrect balance values and wrong solvency decisions.

---

### Finding Description

`SpotEngineState.getBalance` reads `states[productId]` directly from storage and passes it to `balanceNormalizedToBalance`:

```solidity
// SpotEngineState.sol L246-254
function getBalance(uint32 productId, bytes32 subaccount)
    public view returns (Balance memory)
{
    State memory state = states[productId];
    BalanceNormalized memory balance = balances[productId][subaccount];
    return balanceNormalizedToBalance(state, balance);
}
```

`balanceNormalizedToBalance` computes the actual balance using the stored multipliers:

```solidity
// SpotEngineState.sol L180-192
function balanceNormalizedToBalance(State memory state, BalanceNormalized memory balance)
    internal pure returns (Balance memory)
{
    int128 cumulativeMultiplierX18;
    if (balance.amountNormalized > 0) {
        cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
    } else {
        cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
    }
    return Balance(balance.amountNormalized.mul(cumulativeMultiplierX18));
}
```

These multipliers are only advanced inside `_updateState`, which is called exclusively from `updateStates(dt)`:

```solidity
// SpotEngineState.sol L265-283
function updateStates(uint128 dt) external onlyEndpoint {
    require(dt < 7 * SECONDS_PER_DAY, ERR_INVALID_TIME);
    for (uint32 i = 0; i < productIds.length; i++) {
        ...
        _updateState(productId, state, dt);
        _setState(productId, state);
    }
}
```

`updateStates` is triggered only by a `SpotTick` transaction in `EndpointTx.processTransactionImpl`:


`LiquidateSubaccount` is a completely separate transaction type with no requirement that a `SpotTick` precede it:

```solidity
// EndpointTx.sol L391-412
if (txType == IEndpoint.TransactionType.LiquidateSubaccount) {
    ...
    clearinghouse.liquidateSubaccount(signedTx.tx);
```

The liquidation health check path is:

`liquidateSubaccountImpl` → `isUnderMaintenance` → `getHealthFromClearinghouse` → `Clearinghouse.getHealth` → `spotEngine.getHealthContribution` → `BaseEngine._calculateProductHealth` → `SpotEngineState._getBalance` → `getBalance` → `balanceNormalizedToBalance(states[productId], ...)`.

```solidity
// ClearinghouseLiq.sol L51-58
function isUnderMaintenance(bytes32 subaccount) internal returns (bool) {
    return getHealthFromClearinghouse(subaccount, IProductEngine.HealthType.MAINTENANCE) < 0;
}
```

```solidity
// BaseEngine.sol L162-174
function _calculateProductHealth(...) internal returns (int128 health) {
    RiskHelper.Risk memory risk = _risk(productId);
    (int128 amount, int128 quoteAmount) = _getBalance(productId, subaccount);
    int128 weight = RiskHelper._getWeightX18(risk, amount, healthType);
    health += quoteAmount;
    if (amount != 0) {
        ...
        health += amount.mul(weight).mul(risk.priceX18);
    }
}
```

The `_getBalance` call resolves to `SpotEngineState._getBalance` → `getBalance`, which uses the stale stored multipliers. There is no freshness enforcement anywhere in this call chain.

The protocol permits up to 7 days of multiplier staleness (`require(dt < 7 * SECONDS_PER_DAY, ERR_INVALID_TIME)`), meaning the window of exposure is structurally bounded but large.

---

### Impact Explanation

**Liquidation bypass (borrowers):** A borrower's `amountNormalized` is negative. Their actual liability is `amountNormalized * cumulativeBorrowsMultiplierX18`. If the multiplier is stale (lower than the true current value), the liability is understated. The borrower appears healthier than they are. An insolvent borrower cannot be liquidated, leaving bad debt accumulating in the protocol.

**Wrongful liquidation (depositors):** A depositor's `amountNormalized` is positive. Their actual collateral is `amountNormalized * cumulativeDepositsMultiplierX18`. If the multiplier is stale, collateral is understated. A solvent depositor appears unhealthy and can be liquidated at a penalty price by a liquidator.

**Wrongful withdrawal rejection:** `Clearinghouse.withdrawCollateral` also calls `getHealth` after the balance update. A user with sufficient collateral (once interest is properly accrued) may be denied a withdrawal because the stale multiplier understates their collateral.

```solidity
// Clearinghouse.sol L419
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
```

---

### Likelihood Explanation

The sequencer submits `SpotTick` periodically, but there is no on-chain constraint enforcing that a `SpotTick` must appear in the same batch before a `LiquidateSubaccount` or `WithdrawCollateral`. During any interval between `SpotTick` submissions — which can legally span up to 7 days — all health checks use stale multipliers. In practice, even a gap of hours at high utilization rates produces measurable balance discrepancies. A liquidator who observes that `SpotTick` has not been submitted recently can submit a liquidation transaction targeting a depositor whose collateral is understated by the stale multiplier, profiting from the penalty while the victim is actually solvent.

---

### Recommendation

Before any health check in `liquidateSubaccountImpl`, `withdrawCollateral`, and `transferQuote`, apply the current interest accrual in-memory (without writing to storage) to compute the true current balance. Alternatively, require that `SpotTick` (and `PerpTick`) be included in the same sequencer batch before any transaction that performs a health check, enforced on-chain via a timestamp freshness guard (e.g., `require(block.timestamp - times.spotTime < MAX_STALENESS)`).

---

### Proof of Concept

1. At time T, `SpotTick` is submitted. `cumulativeBorrowsMultiplierX18` for USDC = 1.0000.
2. Alice borrows USDC. Her `amountNormalized` = -1000 (normalized). True liability = 1000 USDC.
3. 24 hours pass. No `SpotTick` is submitted. At 10% APR, the true multiplier is now ≈ 1.000274. Alice's true liability = 1000.274 USDC.
4. Alice's collateral (ETH) drops in price. At the true multiplier, her maintenance health = -5 USDC (insolvent).
5. A liquidator submits `LiquidateSubaccount(alice, ...)`.
6. `isUnderMaintenance` calls `getHealth`, which calls `getBalance` using the stale multiplier 1.0000.
7. Alice's liability is computed as 1000 USDC instead of 1000.274 USDC.
8. If the 0.274 USDC difference is enough to push health above zero (e.g., Alice is borderline), `isUnderMaintenance` returns `false`.
9. The liquidation reverts with `ERR_NOT_LIQUIDATABLE`. Alice's bad debt is not cleared.

Conversely, replace Alice with a depositor: her collateral is understated by the stale deposit multiplier, making her appear unhealthy when she is not, enabling a wrongful liquidation. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8) [10](#0-9)
