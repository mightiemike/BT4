### Title
NEAR Tokens Permanently Locked When GC Frees Previously-Paid Storage Deposits — (`contract/src/lib.rs`)

### Summary

`submit_blocks` is `#[payable]` and requires callers to attach NEAR tokens to cover storage growth. When `run_mainchain_gc` subsequently removes old headers, it frees storage that was paid for by **previous** `submit_blocks` calls. The freed NEAR tokens accumulate in the contract's balance with no withdrawal or recovery function anywhere in the contract. They are permanently locked.

### Finding Description

In `submit_blocks`, storage accounting is performed as follows:

```rust
let amount = env::attached_deposit();
let initial_storage = env::storage_usage();

for header in headers {
    self.submit_block_header(header, self.skip_pow_verification);
}

self.run_mainchain_gc(num_of_headers);   // ← frees storage paid by prior callers
let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());
``` [1](#0-0) 

The refund logic only returns the **current caller's** overpayment:

```rust
let refund = amount.saturating_sub(required_deposit);
if refund > NearToken::from_near(0) {
    Promise::new(env::predecessor_account_id()).transfer(refund).into()
}
``` [2](#0-1) 

`run_mainchain_gc` removes entries from `mainchain_height_to_header`, `mainchain_header_to_height`, and `headers_pool` for every header beyond `gc_threshold`: [3](#0-2) 

Each removed entry frees storage bytes. In NEAR's storage-staking model, the minimum balance the contract must hold decreases proportionally, making those previously-locked tokens available as free contract balance. However, the contract exposes **no function** to withdraw this balance — no `withdraw`, `claim`, `rescue`, or equivalent exists anywhere in `contract/src/lib.rs`. [4](#0-3) 

### Impact Explanation

Every time GC fires and removes headers, the NEAR tokens that were attached by prior relayers to pay for those headers' storage are freed into the contract's balance and become irrecoverable. With the recommended `gc_threshold = 52704` (one year of Bitcoin blocks), and each `ExtendedHeader` consuming on the order of hundreds of bytes of storage, the cumulative locked amount grows continuously and unboundedly over the contract's lifetime. There is no admin escape hatch.

### Likelihood Explanation

GC is triggered automatically inside every `submit_blocks` call once the stored chain exceeds `gc_threshold`. This is the normal steady-state operation of a live deployment — it is not an edge case. Any production relayer running continuously will trigger GC on every batch after the threshold is reached, locking a small but non-zero amount of NEAR with each call.

### Recommendation

Add an owner-only withdrawal function that transfers the contract's free balance (total balance minus the minimum storage-staking reserve) to the admin:

```rust
pub fn withdraw_freed_storage_deposit(&mut self) -> Promise {
    // Only DAO or admin role
    let min_balance = env::storage_byte_cost()
        .saturating_mul(env::storage_usage().into());
    let free_balance = env::account_balance().saturating_sub(min_balance);
    require!(free_balance > NearToken::from_yoctonear(0), "Nothing to withdraw");
    Promise::new(env::predecessor_account_id()).transfer(free_balance)
}
```

Alternatively, track cumulative freed storage per-relayer and return it proportionally.

### Proof of Concept

1. Deploy the contract with `gc_threshold = 3`.
2. Relayer A calls `submit_blocks` with 4 headers, attaching the required storage deposit for 4 headers. GC removes 1 header. Net storage change = +3 headers. Relayer A is refunded for 1 header's worth of deposit. The freed storage deposit for the removed header remains in the contract.
3. Relayer B calls `submit_blocks` with 1 more header. GC removes 1 more header (the one Relayer A originally paid for). Net storage change = 0 (`saturating_sub` clamps to 0). `required_deposit = 0`. Relayer B gets a full refund of their attached deposit. The freed storage deposit from Relayer A's header is now permanently locked in the contract balance.
4. Repeat indefinitely. The contract balance grows monotonically with no recovery path. [5](#0-4)
