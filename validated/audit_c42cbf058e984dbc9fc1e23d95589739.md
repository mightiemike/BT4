### Title
`DelegateAction` / `DelegateActionV2` signing domain (`MessageDiscriminant`) omits any chain-specific identifier, enabling cross-network signature replay — (`core/primitives/src/signable_message.rs`, `core/primitives/src/action/delegate.rs`)

---

### Summary

The NEP-461 signing hash for both `DelegateAction` (NEP-366) and `DelegateActionV2` (NEP-611) is computed over a payload whose only "domain tag" is a fixed integer derived from the NEP number. No chain ID, genesis hash, or any other network-specific value is mixed into the signed bytes. A `SignedDelegateAction` produced on one NEAR network (e.g., mainnet) is cryptographically valid on any other NEAR network (e.g., testnet) whenever the sender's account exists there with the same access key and a matching nonce.

---

### Finding Description

`DelegateAction::get_nep461_hash()` constructs the signed payload as:

```
SHA-256( borsh(MessageDiscriminant) ‖ borsh(DelegateAction) )
```

`MessageDiscriminant` is computed as `MIN_ON_CHAIN_DISCRIMINANT + NEP_366_META_TRANSACTIONS = (1 << 30) + 366 = 1 073 742 190` — a compile-time constant that is identical on every NEAR network. [1](#0-0) 

`SignableMessage::sign` hashes the borsh encoding of `(discriminant, msg)` and signs the result; neither the call site nor the struct itself injects a chain ID. [2](#0-1) 

`DelegateAction::get_nep461_hash()` calls `SignableMessage::new(&self, SignableMessageType::DelegateAction)` and hashes the result. The `DelegateAction` struct contains `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key` — none of which are chain-specific. [3](#0-2) 

The same structural gap exists for `DelegateActionV2` via `VersionedDelegateActionPayload::get_nep461_hash()`, which uses `SignableMessageType::DelegateActionV2` (discriminant `(1 << 30) + 611`). [4](#0-3) 

`SignedDelegateAction::verify()` and `VersionedSignedDelegateAction::verify()` both call `get_nep461_hash()` and check the signature against it — no chain-specific value is verified at any point. [5](#0-4) 

`apply_delegate_action` in the runtime calls `signed_delegate_action.verify()` and then checks only the nonce and `max_block_height`. Neither check is chain-specific. [6](#0-5) 

The nonce check (`validate_delegate_action_key`) compares the action's nonce against the on-chain access-key nonce for `(sender_id, public_key)`. This prevents replay on the *same* network but provides no protection across networks. [7](#0-6) 

By contrast, regular `SignedTransaction` is bound to a specific block hash (`transaction.block_hash`), which is chain-specific and expires within `transaction_validity_period` blocks — a protection that `DelegateAction` entirely lacks. [8](#0-7) 

---

### Impact Explanation

An attacker who observes a `SignedDelegateAction` on one NEAR network — it is embedded in a publicly visible on-chain transaction — can submit it verbatim on any other NEAR network. If the sender's account exists on the target network with the same access key registered and the same nonce value, `apply_delegate_action` accepts the signature and executes the inner actions with the sender's identity. This enables unauthorized token transfers, contract calls, key additions/deletions, and account deletions on the target network without the user's knowledge or consent.

---

### Likelihood Explanation

- NEAR operates multiple live networks (mainnet, testnet, betanet/shardnet). Implicit accounts (64-character hex IDs derived from a public key) have **identical account IDs** across all networks. A user who funds the same implicit account on both mainnet and testnet with the same key pair starts with nonce 0 on both, making their first `SignedDelegateAction` immediately replayable.
- `max_block_height` is a block height, not a block hash. Block heights on different networks are independent integers; a height that is "far in the future" on mainnet is also valid on testnet if testnet is at a lower height, providing no meaningful cross-network expiry.
- `SignedDelegateAction` objects are transmitted off-chain to relayers and are also permanently embedded in on-chain transactions, giving attackers multiple extraction points.
- No privileged access is required: any observer of either network's transaction history can mount the attack.

---

### Recommendation

Include a chain-specific identifier in the signed payload. Two concrete options:

1. **Add `chain_id` to `DelegateAction` / `DelegateActionV2`** as a mandatory field verified during `apply_delegate_action`. The runtime already has access to the chain ID via genesis config.
2. **Encode the chain ID inside `MessageDiscriminant`** or add a second borsh-serialized field to `SignableMessage` so that the discriminant is `(chain_id, nep_number)` rather than just `nep_number`. This requires no struct change to `DelegateAction` itself.

Either approach makes the signed hash chain-specific, so a signature produced on mainnet cannot verify on testnet.

---

### Proof of Concept

1. Alice creates implicit account `0xabcd…ef` on both mainnet and testnet using the same key pair. Both accounts start with access-key nonce 0.
2. Alice signs a `DelegateAction` on mainnet:
   ```
   DelegateAction {
     sender_id:       "0xabcd…ef",
     receiver_id:     "usdt.tether-token.near",
     actions:         [FunctionCall { method: "ft_transfer", args: {receiver_id: "attacker", amount: "1000000"} }],
     nonce:           1,
     max_block_height: 200_000_000,   // far future on both networks
     public_key:      alice_pk,
   }
   ```
   Signing hash = `SHA-256( borsh(1_073_742_190) ‖ borsh(DelegateAction) )` — **identical on mainnet and testnet**.
3. The attacker extracts the `SignedDelegateAction` from the mainnet transaction (publicly visible).
4. The attacker wraps it in a testnet transaction (attacker pays gas) and submits it.
5. On testnet, `apply_delegate_action`:
   - `verify()` → passes (same hash, same signature)
   - `max_block_height` check → passes (testnet height ≪ 200 000 000)
   - nonce check → passes (testnet nonce = 0, action nonce = 1 > 0)
   - Executes `ft_transfer` on testnet's USDT contract with `predecessor_id = "0xabcd…ef"`
6. Alice's testnet USDT balance is drained without her consent.

The exact divergent Borsh bytes are the first four bytes of the signed payload: `[0x6E, 0x01, 0x40, 0x40]` (little-endian `1_073_742_190`), which are identical across all NEAR networks and constitute the entirety of the "domain separator." [9](#0-8) [10](#0-9)

impl<'a, T: BorshSerialize> SignableMessage<'a, T> {
    pub fn new(msg: &'a T, ty: SignableMessageType) -> Self {
        let discriminant = ty.into();
        Self { discriminant, msg }
    }

    pub fn sign(&self, signer: &Signer) -> Signature {
        let bytes = borsh::to_vec(&self).expect("Failed to deserialize");
        let hash = hash(&bytes);
        signer.sign(hash.as_bytes())
    }
}

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)

/// Validate access key which was used for signing DelegateAction:
///
/// - Checks whether the access key is present fo given public_key and sender_id.
/// - Validates nonce and updates it if it's ok.
/// - Validates access key permissions.
fn validate_delegate_action_key(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    delegate_action: VersionedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let sender_id = delegate_action.sender_id();
    let public_key = delegate_action.public_key();
    // 'sender_id' account existence must be checked by a caller
    let mut access_key = match get_access_key(state_update, sender_id, public_key)? {
        Some(access_key) => access_key,
        None => {
            result.result = Err(ActionErrorKind::DelegateActionAccessKeyError(
                InvalidAccessKeyError::AccessKeyNotFound {
                    account_id: sender_id.clone(),
                    public_key: public_key.clone().into(),
                },
            )
            .into());
            return Ok(());
        }
    };
