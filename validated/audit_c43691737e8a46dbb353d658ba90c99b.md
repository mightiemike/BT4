### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Proof Validation — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle proof forgery vulnerability by requiring a valid coinbase proof before verifying the transaction proof. However, the original `verify_transaction_inclusion` function is still a publicly callable NEAR contract method. Any unprivileged caller can invoke it directly, bypassing the coinbase Merkle proof validation entirely, and obtain a `true` return value for a forged transaction inclusion claim.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces a two-step check:

1. Validate that the coinbase transaction proof is consistent with the block's stored `merkle_root`.
2. Only then delegate to `verify_transaction_inclusion` for the target transaction proof. [1](#0-0) 

The coinbase check is the security gate that prevents the 64-byte transaction Merkle proof forgery: an attacker cannot supply an internal Merkle tree node as `tx_id` and produce a matching proof, because the coinbase proof anchors the tree structure.

However, `verify_transaction_inclusion` is not private — it is a public `#[near]` method. The `#[deprecated]` Rust attribute is a compile-time lint only; it imposes no runtime restriction. Any NEAR account can call `verify_transaction_inclusion` directly: [2](#0-1) 

The function itself explicitly documents this risk:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [3](#0-2) 

The structural parallel to M-01 is exact:

| M-01 (Holograph) | This repository |
|---|---|
| Fallback appends `msg.sender`; direct enforcer calls do not | `verify_transaction_inclusion_v2` validates coinbase proof; direct calls to v1 do not |
| `isOwner` reads from calldata — result differs by call path | Merkle proof check runs in both paths, but the coinbase anchor only runs in v2 |
| Attacker passes arbitrary `data` bytes interpreted as owner | Attacker passes internal node hash as `tx_id`, bypassing the anchor |

The `ProofArgsV2 → ProofArgs` conversion used internally by v2 drops `coinbase_tx_id` and `coinbase_merkle_proof` entirely: [4](#0-3) 

This confirms that the coinbase validation lives exclusively in v2's preamble. Calling v1 directly skips it with no compensating check.

---

### Impact Explanation

An attacker who calls `verify_transaction_inclusion` directly can supply an internal Merkle tree node hash as `tx_id` together with a sibling proof that reconstructs the block's `merkle_root`. The function returns `true`. Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate an asset release, a state transition, or a cross-chain action will accept the forged inclusion claim and execute the privileged action for a Bitcoin transaction that was never confirmed in that block.

---

### Likelihood Explanation

The entry point is fully unprivileged — any NEAR account can call `verify_transaction_inclusion`. The Bitcoin Merkle tree structure for any mainchain block is public; internal node hashes are computable from the block's transaction list. The 64-byte transaction forgery technique is well-documented (BitMEX research, CVE-2017-12842). No special capability beyond knowledge of the target block's transaction set is required.

---

### Recommendation

Make `verify_transaction_inclusion` a private helper so it is not reachable as a NEAR contract method. Rename it (e.g., `_verify_transaction_inclusion_inner`) and call it only from `verify_transaction_inclusion_v2`. If backward compatibility must be preserved, replicate the coinbase proof validation inside `verify_transaction_inclusion` itself so both entry points enforce the same security invariant.

---

### Proof of Concept

1. Attacker selects any mainchain block `B` with at least two transactions. The block's `merkle_root` is stored in `headers_pool`.
2. Attacker computes the internal Merkle node `N` that is the parent of transactions at indices 2 and 3 (for example). `N` is a 32-byte hash derivable from public block data.
3. Attacker constructs `merkle_proof` such that `compute_root_from_merkle_proof(N, index_of_N_as_leaf, proof) == merkle_root`. This is valid because `N` is itself a node in the tree.
4. Attacker calls `verify_transaction_inclusion` directly with `tx_id = N`, `tx_block_blockhash = B.hash`, `tx_index = <chosen index>`, `merkle_proof = <constructed proof>`, `confirmations = 0`.
5. `verify_transaction_inclusion` passes all `require!` guards and returns `true`.
6. A downstream contract that gates a fund release on this call's return value executes the transfer for a transaction that does not exist at that position. [5](#0-4)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
