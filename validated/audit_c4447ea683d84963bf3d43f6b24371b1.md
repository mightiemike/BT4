### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Allowing Bypass of Coinbase Merkle Proof Forgery Protection — (`contract/src/lib.rs`)

---

### Summary

The contract exposes two transaction-inclusion proof endpoints. `verify_transaction_inclusion_v2` was introduced specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase merkle proof. However, the original `verify_transaction_inclusion` remains a fully public, callable NEAR method. Any unprivileged caller can invoke it directly, bypassing the coinbase proof check entirely and obtaining a `true` result for a fabricated transaction inclusion claim.

The security invariant — that every accepted proof has been validated against a coinbase anchor — is only enforced on one of the two reachable entry points.

### Impact Explanation

The 64-byte transaction Merkle proof forgery attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions) allows an attacker to craft a 64-byte input that is simultaneously a valid serialized internal Merkle tree node and a plausible `tx_id`. By calling `verify_transaction_inclusion` directly with such a crafted `tx_id` and a matching Merkle path, an attacker can obtain a `true` return value for a Bitcoin transaction that does not exist. Any downstream NEAR contract that consumes this result — e.g., to release funds, mint tokens, or record a cross-chain event — will act on a forged proof.

---

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` as a view or cross-contract call. The 64-byte forgery technique is well-documented and has known tooling. The only prerequisite is knowledge of a valid block hash already stored in the contract's `headers_pool`, which is public state. Likelihood is high for any deployment where downstream contracts rely on this light client for bridge or settlement logic.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` or gate it behind an access-control role so it cannot be called directly by unprivileged accounts. All external callers should be forced through `verify_transaction_inclusion_v2`. Alternatively, delete the v1 method entirely and inline its logic into v2, since v2 already calls v1 internally. [6](#0-5) 

---

### Proof of Concept

1. Attacker identifies a block hash `B` stored in `mainchain_header_to_height` (readable from public state).
2. Attacker constructs a 64-byte value `fake_tx` that, when hashed with a chosen sibling at position `i`, produces the block's known `merkle_root`. This is the standard 64-byte Merkle forgery construction.
3. Attacker calls `verify_transaction_inclusion` directly (not v2) with:
   - `tx_id = fake_tx`
   - `tx_block_blockhash = B`
   - `tx_index = i`
   - `merkle_proof = [sibling]`
   - `confirmations = 1`
4. The function computes `compute_root_from_merkle_proof(fake_tx, i, [sibling])` which equals `merkle_root`, and returns `true`.
5. No coinbase proof was ever checked. A downstream contract receives `true` for a non-existent transaction. [7](#0-6) [8](#0-7)

        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
