### Title
Migration Aborts with `ErrValidatorNotBonded` When Vesting-Owned Position's Validator Is Jailed at Upgrade Height — (`x/tieredrewards/migrations/v2/migrate.go`, `x/tieredrewards/keeper/force_exit.go`, `x/tieredrewards/keeper/transfer_delegation.go`)

### Summary

The v8 `Migrate()` function unconditionally calls `ForceFullExitWithDelegation` for every vesting-owned tier position. That function calls `transferDelegationFromPosition`, which contains a hard guard that returns `types.ErrValidatorNotBonded` if the position's validator is not in `Bonded` status. The error propagates without any catch or fallback through the entire call stack and causes `Migrate()` to return an error. In Cosmos SDK upgrade handlers, a returned error causes a panic and a chain halt. Any vesting account owner whose position is delegated to a jailed or unbonding validator at upgrade height can trigger this path, halting the chain.

### Finding Description

**Error guard in `transferDelegationFromPosition`:** [1](#0-0) 

```go
if !validator.IsBonded() {
    return ..., types.ErrValidatorNotBonded
}
```

This guard exists to protect the normal user-facing `MsgExitTierWithDelegation` flow, where it is correct to reject a non-bonded validator. However, the same function is reused unchanged by the migration path.

**`ForceFullExitWithDelegation` propagates the error without handling the non-bonded case:** [2](#0-1) 

**`exitVestedAccountsPositions` propagates the error, aborting the entire migration:** [3](#0-2) 

**`Migrate` propagates the error to the upgrade handler:** [4](#0-3) 

There is no `continue`-on-error, no fallback unbonding path, and no skip-if-not-bonded logic anywhere in the migration path.

**The existing test `TestTransferDelegationFromPosition_ValidatorNotBonded` explicitly confirms the error is returned:** [5](#0-4) 

### Impact Explanation

In Cosmos SDK, an upgrade handler that returns an error causes the upgrade module to panic during `BeginBlock` at the upgrade height. Every node on the network hits the same panic simultaneously, halting the chain permanently at that block. The chain cannot advance until a coordinated emergency patch and restart is performed. This is a **Critical** impact: chain halt at upgrade height.

### Likelihood Explanation

The preconditions are both realistic and independently likely:
1. **Vesting accounts with tier positions exist.** Vesting accounts are standard in Cronos POS Chain (team, investor, foundation allocations). The `MsgLockTier` and `MsgCommitDelegationToTier` messages are available to any account, including vesting accounts.
2. **Validators get jailed.** Validators are routinely jailed for downtime (missed blocks) or double-signing. The window between position creation and the upgrade block can span months, making it highly probable that at least one validator hosting a vesting-owned position will be jailed or in an unbonding state at upgrade time.

Neither condition requires attacker privilege. A passive scenario (validator jailed for downtime, vesting account holder did nothing wrong) is sufficient to trigger the halt.

### Recommendation

`ForceFullExitWithDelegation` must handle the non-bonded validator case. Options:

1. **Skip the `IsBonded` check in the migration path.** The `Unbond` call in `transferDelegationFromPosition` (line 149) does not itself require the validator to be bonded — only the subsequent `Delegate` back to the owner does. For a jailed validator, the re-delegation to the owner on the same jailed validator should be replaced with a standard unbonding entry (initiating the unbonding period for the owner), or the delegation can be moved to a bonded validator.

2. **Add a non-bonded fallback in `ForceFullExitWithDelegation`.** Before calling `transferDelegationFromPosition`, check `validator.IsBonded()`. If not bonded, call `stakingKeeper.Undelegate` directly from the position's delegator address to the owner, initiating a normal unbonding period rather than an instant transfer.

3. **Catch and continue in `exitVestedAccountsPositions`.** As a minimal mitigation, log and skip positions whose validator is not bonded, then handle them post-upgrade via governance. This avoids the halt but leaves those positions in a stale state.

### Proof of Concept

```go
func (s *KeeperSuite) TestMigrate_VestingOwner_JailedValidator_Halts() {
    // 1. Setup: create a vesting-owned position on a second validator
    s.setupTier(1)
    vals, bondDenom := s.getStakingData()
    val := vals[0]
    valAddr := sdk.MustValAddressFromBech32(val.GetOperator())

    lockedAmount := sdkmath.NewInt(sdk.DefaultPowerReduction.Int64())
    owner := s.newVestingOwnerWithBalance(bondDenom, lockedAmount, lockedAmount.MulRaw(2))
    _ = s.createLockTierPositionV1(owner, valAddr, lockedAmount)

    // 2. Jail the validator (simulates downtime between position creation and upgrade)
    s.jailAndUnbondValidator(valAddr)

    // 3. Run the v8 migration — must NOT return an error, but currently does
    err := v2.Migrate(s.ctx, s.keeper.Positions, s.app.AccountKeeper, s.keeper)
    // This assertion FAILS with the current code, proving the chain halt:
    s.Require().NoError(err, "migration must complete regardless of validator bonding state")
}
```

Running this test against the unmodified codebase will produce `ErrValidatorNotBonded` wrapped in the migration error chain, confirming the chain halt.

func (s *KeeperSuite) TestTransferDelegationFromPosition_ValidatorNotBonded() {
	lockAmount := sdkmath.NewInt(10000)
	pos := s.setupNewTierPosition(lockAmount, true)

	s.advancePastExitDuration()

	valAddr := sdk.MustValAddressFromBech32(pos.Delegation.ValidatorAddress)
	s.jailAndUnbondValidator(valAddr)

	// Compute token value from shares.
	val, err := s.app.StakingKeeper.GetValidator(s.ctx, valAddr)
	s.Require().NoError(err)
	positionAmount := val.TokensFromShares(pos.Delegation.Shares).TruncateInt()

	_, _, _, err = s.keeper.TransferDelegationFromPosition(s.ctx, pos, valAddr, positionAmount)
	s.Require().ErrorIs(err, types.ErrValidatorNotBonded)
}
