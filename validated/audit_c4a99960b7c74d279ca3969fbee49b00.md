Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Gates the Router Address Instead of the End User, Enabling Allowlist Bypass — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter against the per-pool allowlist, but `sender` is sourced from `msg.sender` inside `MetricOmmPool.swap`. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. Any pool admin who allowlists the router to enable router-mediated swaps for their curated users simultaneously opens the pool to every unprivileged address that calls the router, completely defeating the allowlist.

## Finding Description

**Call chain:**

`MetricOmmPool.swap` captures `msg.sender` and passes it as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards `sender` unchanged into every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool (correct) and `sender` is whoever called `pool.swap`. When a user enters through `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap` directly: [4](#0-3) 

So `sender` received by the extension is the **router's address**. The allowlist lookup becomes `allowedSwapper[pool][router]` — a single entry that covers every user who routes through that contract. There is no mechanism in the router to pass the original `msg.sender` as a distinct field to the extension.

By contrast, `DepositAllowlistExtension` gates the `owner` parameter (the position owner explicitly supplied by the caller), which cannot be substituted by an intermediary: [5](#0-4) 

## Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` and wants allowlisted users to be able to use the canonical router must call `setAllowedToSwap(pool, router, true)`. Once that entry exists, **any address** can call `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) targeting that pool and the extension will pass — the allowlist is completely bypassed. Disallowed traders gain full swap access to a pool designed to be curated, directly impacting LP principal and protocol fee assumptions tied to the curated user set. This constitutes an admin-boundary break: an unprivileged path bypasses the pool admin's access control.

## Likelihood Explanation

The router is the canonical, publicly documented swap entry point for the protocol. A pool admin who configures a swap allowlist and also wants their allowlisted users to be able to use the router has no correct option: they must either allowlist the router (triggering the bypass) or block the router entirely (making the pool unusable via the standard periphery). The bypass path requires only a normal `exactInputSingle` call — no special privileges, no flash loans, no multi-step setup. The condition is reachable by any unprivileged address.

## Recommendation

The extension must gate the **economic actor**, not the intermediary. Two complementary fixes:

1. **Pass the original user through the router.** Add a `swapper` field to the router's swap parameters and forward it via `extensionData` so the extension can recover the true initiator.
2. **Adopt the `DepositAllowlistExtension` pattern.** Require that the pool or router passes the original `msg.sender` of the outermost call in a dedicated slot of `extensionData`, and have the extension decode and gate on that value rather than on `sender`.

## Proof of Concept

```
Setup
─────
1. Deploy a pool with SwapAllowlistExtension wired as beforeSwap hook.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   (necessary so that allowlisted users can use the router).
3. Pool admin calls setAllowedToSwap(pool, alice, true)   // intended gated user
   Pool admin does NOT add bob.

Attack
──────
4. bob (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({
           pool:      <curated pool>,
           recipient: bob,
           zeroForOne: true,
           amountIn:  X,
           ...
       })

5. Router calls pool.swap(bob_as_recipient, ...) with msg.sender = router.
6. Pool calls _beforeSwap(sender=router, ...).
7. Extension evaluates allowedSwapper[pool][router] == true → passes.
8. bob's swap executes on the curated pool despite never being allowlisted.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds
```
