### Title
`run_serialized_chia_program` Uses `node_from_bytes` Instead of `node_from_bytes_backrefs`, Silently Rejecting Valid Back-Reference-Compressed Programs — (File: `wheel/src/api.rs`)

---

### Summary

`run_serialized_chia_program`, the primary Python-facing entry point for executing CLVM programs, deserializes both `program` and `args` using `node_from_bytes` (legacy-only). Since the Chia hard fork at block height 5,496,000, valid on-chain programs may be serialized with back-reference compression (`0xfe` opcodes). `node_from_bytes` cannot parse these; `node_from_bytes_backrefs` can. Any caller passing a back-reference-compressed blob to `run_serialized_chia_program` receives a deserialization error instead of a result, while the same blob would succeed through `node_from_bytes_backrefs`. This is a direct structural analog to the `transfer()` vs `call()` pattern: a less-capable method is used where a strictly more-capable superset method is available and required.

---

### Finding Description

In `wheel/src/api.rs` lines 55–56:

```rust
let program = node_from_bytes(&mut allocator, program).map_err(eval_to_py)?;
let args = node_from_bytes(&mut allocator, args).map_err(eval_to_py)?;
```

`node_from_bytes` is the legacy deserializer. It handles only `0xff` (pair) and atom tokens. It does not recognize `0xfe` (back-reference). If the input byte stream contains a `0xfe` byte, `node_from_bytes` will attempt to parse it as an atom length prefix, producing either a wrong parse or a `SerializationError`.

`node_from_bytes_backrefs` (used in `deser_backrefs`, `deser_auto`, and the round-trip tests) handles both the legacy format and back-references. It is a strict superset: every blob accepted by `node_from_bytes` is also accepted by `node_from_bytes_backrefs` with identical results.

The `deser_auto` function at lines 148–156 correctly uses `node_from_bytes_backrefs` as the fallback for non-2026 blobs. `run_serialized_chia_program` does not.

---

### Impact Explanation

`run_serialized_chia_program` is the single Python API function used by downstream Chia tooling (wallets, full nodes, mempool validators) to execute CLVM programs. If a spend bundle's generator or solution is serialized with back-reference compression — which is valid and canonical on-chain since the hard fork — this function returns a Python `ValueError` (deserialization failure) instead of executing the program and returning a cost and result. A node or wallet relying on this function would incorrectly reject the spend bundle. Nodes using a different deserialization path (e.g., a Rust-native path that calls `node_from_bytes_backrefs`) would accept it. This is a consensus-divergence class impact: two implementations of the same validation logic produce different accept/reject outcomes for the same valid input bytes.

---

### Likelihood Explanation

Back-reference compression has been enabled on mainnet since block height 5,496,000. Any program or solution blob produced by a compressing serializer (e.g., `node_to_bytes_backrefs` or `ser_backrefs`) and then passed directly to `run_serialized_chia_program` triggers the failure. An attacker who controls the serialization of a spend bundle (or who simply uses the standard compressed format) can reliably trigger the rejection. No special privileges or side-channel access are required; the attacker-controlled entry path is the `program: &[u8]` and `args: &[u8]` parameters.

---

### Recommendation

Replace both `node_from_bytes` calls in `run_serialized_chia_program` with `node_from_bytes_backrefs` (or `deser_auto` if 2026-format support is also desired). `node_from_bytes_backrefs` is a strict superset and introduces no regression for legacy-format inputs.

```rust
// Before (line 55-56 of wheel/src/api.rs):
let program = node_from_bytes(&mut allocator, program).map_err(eval_to_py)?;
let args = node_from_bytes(&mut allocator, args).map_err(eval_to_py)?;

// After:
let program = node_from_bytes_backrefs(&mut allocator, program).map_err(eval_to_py)?;
let args = node_from_bytes_backrefs(&mut allocator, args).map_err(eval_to_py)?;
```

---

### Proof of Concept

1. Construct a CLVM program with a repeated subtree, e.g. `(foo . foo)`.
2. Serialize it with back-references: `node_to_bytes_backrefs` produces `ff 83 66 6f 6f fe 02` (the second `foo` is a back-reference).
3. Pass this blob as `program` to `run_serialized_chia_program`.
4. `node_from_bytes` encounters `0xfe`, misparses it as an atom-length prefix, and returns `SerializationError` (or a wrong node).
5. The same blob passed to `node_from_bytes_backrefs` correctly reconstructs `(foo . foo)` and execution proceeds normally.

The round-trip test in `src/serde/test.rs` line 72 (`check("ff83666f6ffe02", None)`) confirms that `node_from_bytes_backrefs` correctly handles this exact back-reference encoding. `node_from_bytes` has no equivalent test because it cannot handle it.
