### Title
`DelegateAction` / `DelegateActionV2` signing domain omits chain ID, enabling cross-network replay of meta-transaction signatures — (`File: core/primitives/src/signable_message.rs`, `core/primitives/src/action/delegate.rs`)

---

### Summary

The `SignableMessage` wrapper used to produce the NEP-461 hash for both `DelegateAction` (NEP-366) and `DelegateActionV2` (NEP-611) contains only a NEP-number-derived discriminant and the serialized action payload. Neither the wrapper nor the payload includes the NEAR chain ID (`"mainnet"`, `"testnet"`, etc.). A malicious relayer can take a `SignedDelegateAction` produced and signed on one NEAR network and submit it verbatim on a different NEAR network where the same account, key, and nonce state exist, executing the inner actions under the signer's identity without their consent.

---

### Finding Description

`SignableMessage` is the sole domain-separation layer for meta-transaction signatures:

```
SignableMessage { discriminant: u32, msg: &DelegateAction }
```

The discriminant is computed as `MIN_ON_CHAIN_DISCRIMINANT + NEP_number` — a compile-time constant identical on every NEAR network. [1](#0-0) 

`SignableMessage::sign` hashes `borsh(discriminant || msg)` and signs the result. No chain ID is mixed in at any point. [2](#0-1) 

`DelegateAction.get_nep461_hash()` calls `SignableMessage::new(&self, SignableMessageType::DelegateAction)` and hashes the result. The `DelegateAction` struct itself carries `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key` — no chain ID field. [3](#0-2) [4](#0-3) 

`VersionedDelegateActionPayload.get_nep461_hash()` (used by `DelegateActionV2`) follows the same pattern with `SignableMessageType::DelegateActionV2`. [5](#0-4) 

`apply_delegate_action` — the runtime entry point — verifies the signature, checks `max_block_height`, and validates the nonce, but performs **no chain ID check** at any step. [6](#0-5) 

Because the Borsh-serialized signing payload is byte-for-byte identical across all NEAR networks for the same `DelegateAction` content, `SignedDelegateAction.verify()` returns `true` on any network. [7](#0-6) 

---

### Impact Explanation

A malicious relayer who receives a `SignedDelegateAction` intended for testnet can submit it on mainnet (or any other NEAR network) by wrapping it in a fresh outer `SignedTransaction`. If the signer's account exists on the target network with the same public key and a matching nonce, the runtime will accept the signature and execute the inner actions — token transfers, key additions/deletions, contract deployments, function calls — under the victim's `sender_id`. The relayer needs no special privilege; submitting transactions is an unprivileged operation open to any network participant.

---

### Likelihood Explanation

The preconditions are realistic:

1. **Same account on both networks** — NEAR account IDs are human-readable strings; users routinely create the same named account on mainnet and testnet.
2. **Same key on both networks** — Many wallets and tools derive keys deterministically from a seed phrase, producing the same `ed25519` key pair on every network.
3. **Matching nonce** — A freshly created account starts at nonce 0 on every network. A testnet `DelegateAction` with `nonce = 1` is valid on mainnet if the mainnet key nonce is still 0.
4. **`max_block_height` not expired** — Relayers commonly set this to a large value (e.g., `BlockHeight::MAX`) to avoid expiry races; such a value is valid on any network.

The attack requires no brute-force or cryptographic break — only the ability to intercept or observe a `SignedDelegateAction` in transit to a testnet relayer and re-submit it to a mainnet node.

---

### Recommendation

Include the chain ID in the signed payload. The minimal fix is to add a `chain_id: String` field to `DelegateAction` and `DelegateActionV2`, or to incorporate it into `SignableMessage` as an additional domain component serialized before the discriminant. The runtime's `apply_delegate_action` must then compare the embedded chain ID against `ApplyState`'s chain ID and reject mismatches. Because this changes the Borsh layout of the signed message, it must be gated behind a new `ProtocolFeature` and old signatures must be rejected once the feature activates.

---

### Proof of Concept

1. Alice has account `alice.near` on both testnet and mainnet, with the same ED25519 key pair and nonce `0` on both.
2. Alice signs a testnet `DelegateAction` transferring 10 NEAR to `bob.near`, with `nonce = 1` and `max_block_height = 10_000_000`.
3. The signing payload is:
   ```
   SHA256( borsh(discriminant=0x4000016E) || borsh(DelegateAction{sender="alice.near", receiver="bob.near", nonce=1, max_block_height=10_000_000, ...}) )
   ```
   This value is identical on mainnet because neither the discriminant nor the payload contains the chain ID.
4. A malicious relayer intercepts the `SignedDelegateAction` and wraps it in a mainnet `SignedTransaction` (signed by the relayer's own mainnet key).
5. The mainnet runtime calls `apply_delegate_action`, which calls `signed_delegate_action.verify()` — this returns `true` because the hash is identical.
6. The nonce check passes (mainnet nonce is 0, action nonce is 1).
7. The `max_block_height` check passes (mainnet height < 10,000,000).
8. The inner transfer of 10 mainnet NEAR executes, draining Alice's mainnet balance without her consent.

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }

pub(crate) fn apply_delegate_action(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    action_receipt: &VersionedActionReceipt,
    sender_id: &AccountId,
    signed_delegate_action: VersionedSignedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    if !signed_delegate_action.verify() {
        result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
        return Ok(());
    }
    let delegate_action = signed_delegate_action.delegate_action();
    if apply_state.block_height > delegate_action.max_block_height() {
        result.result = Err(ActionErrorKind::DelegateActionExpired.into());
        return Ok(());
    }
    if delegate_action.sender_id().as_str() != sender_id.as_str() {
        result.result = Err(ActionErrorKind::DelegateActionSenderDoesNotMatchTxReceiver {
            sender_id: delegate_action.sender_id().clone(),
            receiver_id: sender_id.clone(),
        }
        .into());
        return Ok(());
    }

    validate_delegate_action_key(state_update, apply_state, delegate_action, result)?;
