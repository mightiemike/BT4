### Title
Linked Signer Can Self-Escalate by Signing `LinkSigner` Transaction in Fast Mode — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` fast-mode transaction is validated with `allowLinkedSigner: true`. This permits the currently-registered linked signer to sign a `LinkSigner` transaction and overwrite the linked signer mapping for the subaccount. The slow-mode path for the same transaction type explicitly enforces that only the wallet owner (`msg.sender`) may submit it. The missing state guard — "is the signer the wallet owner, not the linked signer?" — is the direct analog to the `enableTradingWithWeights` bug, where a function that should only be callable in one state (swap disabled / owner-only) lacked the check and allowed a lower-privilege actor to perform a privileged mutation.

---

### Finding Description

`processTransactionImpl` in `EndpointTx.sol` handles `TransactionType.LinkSigner` as follows:

```solidity
// EndpointTx.sol lines 576-590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx(..., true)` resolves the acceptable signer as either the wallet owner **or** the currently-registered linked signer via `getLinkedSigner`:

```solidity
// EndpointTx.sol lines 172-184
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

The slow-mode path for the identical transaction type enforces the opposite: only the wallet owner (`msg.sender`) may submit it, via `validateSender`:

```solidity
// EndpointTx.sol lines 232-239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(transaction[1:], ...);
    validateSender(txn.sender, sender);   // sender == msg.sender == wallet owner
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [3](#0-2) 

`validateSender` enforces `address(bytes20(txSender)) == sender`, i.e., the actual wallet owner:

```solidity
// EndpointTx.sol lines 17-23
function validateSender(bytes32 txSender, address sender) internal view {
    require(
        address(uint160(bytes20(txSender))) == sender ||
            sender == address(this),
        ERR_SLOW_MODE_WRONG_SENDER
    );
}
``` [4](#0-3) 

The protocol already demonstrates awareness of this class of risk: `WithdrawCollateralV2` sets `allowLinkedSigner` to `false` when `sendTo != address(0)`, precisely because a linked signer must not be permitted to redirect funds to an arbitrary address:

```solidity
// EndpointTx.sol lines 442-448
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    signedTx.tx.sendTo == address(0)   // false when sendTo is external
);
``` [5](#0-4) 

The same guard is absent for `LinkSigner`.

---

### Impact Explanation

A linked signer that is compromised (e.g., a leaked API key) can craft and submit — through the sequencer's fast-mode path — a `LinkSigner` transaction that replaces the `linkedSigners[subaccount]` entry with an attacker-controlled address. After this mutation:

- The attacker's new address is the recognized linked signer for the subaccount.
- The attacker can continue to sign orders, withdrawals, and other fast-mode transactions on behalf of the subaccount indefinitely.
- The wallet owner's only recourse is a slow-mode `LinkSigner` submission, which is subject to the `SLOW_MODE_TX_DELAY` (hardcoded to three days per `Endpoint.sol` line 153). During that window the attacker retains full signing authority.

The corrupted state is `linkedSigners[subaccount]` in `EndpointStorage`, which governs who may authorize every fast-mode transaction for that subaccount. [6](#0-5) 

---

### Likelihood Explanation

Linked signers are commonly API keys or hot-wallet keys used by trading bots. Key leakage is a realistic operational risk. Once a linked signer key is leaked, the attacker has an immediate, on-chain path to lock in persistent access before the wallet owner can react. No admin compromise, governance capture, or social engineering is required — only possession of the current linked signer's private key.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` case in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the wallet owner may change the linked signer
);
```

This mirrors the slow-mode path's intent and is consistent with the existing `WithdrawCollateralV2` precedent.

---

### Proof of Concept

1. Alice registers subaccount `alice‖default` and sets a linked signer `apiKey` (e.g., a trading bot key).
2. `apiKey` is leaked.
3. Attacker constructs a `SignedLinkSigner` struct: `sender = alice‖default`, `signer = attacker_address`, signed by `apiKey`.
4. Attacker submits this as a fast-mode transaction through the sequencer.
5. `processTransactionImpl` calls `validateSignedTx(..., true)`. `getLinkedSigner(alice‖default)` returns `apiKey`. The signature is valid.
6. `linkedSigners[alice‖default]` is overwritten with `attacker_address`.
7. Alice submits a slow-mode `LinkSigner` to revoke, but it cannot execute for three days.
8. During those three days, `attacker_address` signs withdrawals and orders on Alice's behalf. [1](#0-0) [7](#0-6)
