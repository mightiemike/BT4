### Title
Incomplete `l1_data_gas` Stripping Condition in `deserialize_transaction_json_to_starknet_api_tx` Silently Converts `AllResources` Transaction to `L1Gas` Variant, Producing Wrong Hash - (`File: crates/starknet_api/src/serde_utils.rs`)

### Summary

`deserialize_transaction_json_to_starknet_api_tx` contains an "rpc_v8 fix" that strips the `l1_data_gas` field from a transaction's resource bounds when `l1_data_gas.max_amount == "0x0"` AND `l2_gas.max_amount == "0x0"`. The condition checks only `max_amount` and ignores `max_price_per_unit`. A valid on-chain `AllResources` transaction with `l1_data_gas.max_amount = 0` but `l1_data_gas.max_price_per_unit ≠ 0`, and `l2_gas` fully zero, will have its `l1_data_gas` silently stripped. The resulting deserialization produces a `ValidResourceBounds::L1Gas` variant instead of `AllResources`, causing the hash to be recomputed under the wrong formula — omitting `l1_data_gas` from the Poseidon preimage — and diverging from the on-chain hash the user signed.

### Finding Description

The "rpc_v8 fix" in `deserialize_transaction_json_to_starknet_api_tx` is intended to remove a zeroed `l1_data_gas` entry that RPC v8 redundantly added to pre-0.13.3 (`L1Gas`) transactions. The correct invariant for that case is that both `l1_data_gas` and `l2_gas` are **fully** zero (both `max_amount` and `max_price_per_unit` are zero). However, the implemented condition only checks `max_amount`:

```rust
// rpc_v8 fix (remove redundantly added L1DataGas)
if !raw_resourcebounds.is_null()
    && !raw_resourcebounds["l1_data_gas"].is_null()
    && raw_resourcebounds["l1_data_gas"]["max_amount"] == "0x0"   // ← only checks amount
    && !raw_resourcebounds["l2_gas"].is_null()
    && raw_resourcebounds["l2_gas"]["max_amount"] == "0x0"        // ← only checks amount
{
    raw_transaction["resource_bounds"]
        .as_object_mut()
        .expect("should be map of resource bounds")
        .remove("l1_data_gas");
}
``` [1](#0-0) 

After stripping `l1_data_gas`, the remaining JSON has only `l1_gas` and `l2_gas`. When `l2_gas` is fully zero, `DeprecatedResourceBoundsMapping::try_into()` sees `l2_bounds.is_zero() == true` and converts to `ValidResourceBounds::L1Gas`:

```rust
None => {
    if l2_bounds.is_zero() {
        Ok(Self::L1Gas(*l1_bounds))
    } else {
        Err(...)
    }
}
``` [2](#0-1) 

The hash is then computed via `get_tip_resource_bounds_hash` using the `L1Gas` branch, which emits an **empty** `l1_data_gas` contribution:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],          // l1_data_gas absent from preimage
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
``` [3](#0-2) 

The on-chain transaction was signed over the `AllResources` hash (which includes `l1_data_gas`). The deserialized transaction produces the `L1Gas` hash. These two values are distinct.

### Impact Explanation

Any valid on-chain `AllResources` transaction satisfying:
- `l1_data_gas.max_amount = 0x0`, `l1_data_gas.max_price_per_unit ≠ 0x0`
- `l2_gas.max_amount = 0x0`, `l2_gas.max_price_per_unit = 0x0`

will be deserialized with the wrong resource-bounds variant and a wrong transaction hash during central sync. This maps to **"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload"** (High) and **"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value"** (High): the stored transaction carries wrong resource bounds and a hash that does not match what was signed, so any RPC query returning that transaction's fields returns incorrect data.

### Likelihood Explanation

The trigger requires a specific but valid combination of resource bounds: zero `l1_data_gas.max_amount` with a non-zero `max_price_per_unit`, and a fully-zero `l2_gas`. Any user can craft and submit such a transaction through the gateway (which accepts `AllResources` bounds). Once included in a block, every node running central sync will hit this path.

### Recommendation

Extend the stripping condition to also verify that `max_price_per_unit` is zero for both `l1_data_gas` and `l2_gas`, matching the actual invariant for the RPC v8 redundant-zero case:

```rust
&& raw_resourcebounds["l1_data_gas"]["max_amount"] == "0x0"
&& raw_resourcebounds["l1_data_gas"]["max_price_per_unit"] == "0x0"  // add
&& raw_resourcebounds["l2_gas"]["max_amount"] == "0x0"
&& raw_resourcebounds["l2_gas"]["max_price_per_unit"] == "0x0"       // add
```

### Proof of Concept

1. Construct an `AllResources` Invoke V3 transaction with:
   - `l1_gas = {max_amount: 0x100, max_price_per_unit: 0x100}`
   - `l2_gas = {max_amount: 0x0, max_price_per_unit: 0x0}`
   - `l1_data_gas = {max_amount: 0x0, max_price_per_unit: 0x5}`
2. Sign it with the `AllResources` hash (which includes `l1_data_gas` in the Poseidon preimage via `get_tip_resource_bounds_hash`). [4](#0-3) 
3. Submit through the gateway; it is accepted and included in a block.
4. During central sync, `deserialize_transaction_json_to_starknet_api_tx` is called on the feeder-gateway JSON. The condition at line 191–193 fires (`l1_data_gas.max_amount == "0x0"` and `l2_gas.max_amount == "0x0"`), stripping `l1_data_gas`. [5](#0-4) 
5. The deserialized transaction becomes `ValidResourceBounds::L1Gas`, and its hash is recomputed without `l1_data_gas` — a different value from the on-chain hash.
6. `validate_transaction_hash` fails (hash mismatch), causing sync failure; or, if validation is skipped, the transaction is stored with wrong resource bounds and the wrong hash, causing all subsequent RPC queries for that transaction to return incorrect data. [6](#0-5)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
