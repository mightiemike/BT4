### Title
NLP Pool Subaccount Solvency Not Enforced in `burnNlp` — Unrealized Perp Losses Can Drain Pool Quote — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`burnNlp` deducts quote from NLP pool subaccounts via `_applyNlpRebalance` but performs **no health check on those pool subaccounts** afterward. Because the NLP price (`oraclePriceX18`) is supplied by the sequencer without on-chain verification that it accounts for unrealized perp losses held by the pool, a burn can extract more quote than the pool can actually afford, leaving pool subaccounts insolvent.

---

### Finding Description

When a user burns NLP, `Clearinghouse.burnNlp` computes the quote to return as `nlpAmount × oraclePriceX18` and then calls `_applyNlpRebalance` to deduct that quote from the NLP pool subaccounts: [1](#0-0) 

`_applyNlpRebalance` applies the deductions with no guard: [2](#0-1) 

After the deduction, `burnNlp` only checks the **burner's** health — it never checks the health of the pool subaccounts: [3](#0-2) 

Contrast this with `forceRebalanceNlpPool`, which does check pool subaccount health (for pools ≥ 1) after any rebalance: [4](#0-3) 

And `nlpProfitShare`, which also checks pool health after transferring quote out: [5](#0-4) 

The `oraclePriceX18` for NLP is set by the sequencer and stored without any on-chain validation that it correctly reflects unrealized perp P&L: [6](#0-5) 

NLP pool subaccounts act as market-maker counterparties in `MatchOrders`. They accumulate perp positions whose unrealized losses are tracked in `vQuoteBalance` inside `PerpEngine`. The `getHealth` function **does** include this unrealized P&L via `_calculateProductHealth`: [7](#0-6) 

So a proper health check on the pool subaccount would catch insolvency caused by unrealized perp losses. The missing check in `burnNlp` is the root cause.

---

### Impact Explanation

If traders hold large profitable perp positions against the NLP pool (i.e., the pool has large unrealized losses), and the `oraclePriceX18` used for the burn does not fully discount those unrealized losses, then:

1. Users burn NLP and receive quote that the pool cannot actually afford.
2. Pool subaccount quote balances go negative beyond what their perp positions can cover.
3. When profitable perp positions are later settled via `settlePnl`, there is insufficient quote to pay out.
4. The NLP pool becomes insolvent — a direct solvency/accounting corruption matching the reference vulnerability class.

---

### Likelihood Explanation

NLP pool subaccounts participate as market makers in perp trading. In volatile markets, the pool can accumulate significant unrealized losses. The `oraclePriceX18` is computed off-chain and is not verified on-chain to account for these unrealized losses. Any NLP holder can sign a `BurnNlp` transaction; the sequencer processes it. The structural absence of a pool health check means this path is always open whenever the pool has unrealized perp losses.

---

### Recommendation

After `_applyNlpRebalance` in `burnNlp`, add a health check for each NLP pool subaccount, consistent with the pattern already used in `forceRebalanceNlpPool` and `nlpProfitShare`:

```solidity
// After _applyNlpRebalance in burnNlp:
for (uint128 i = 0; i < nlpPools.length; i++) {
    require(
        getHealth(nlpPools[i].subaccount, IProductEngine.HealthType.INITIAL) >= 0,
        ERR_SUBACCT_HEALTH
    );
}
```

This ensures that unrealized perp losses are accounted for (via `getHealth` → `perpEngine.getHealthContribution` → `vQuoteBalance`) before any quote is released to the burner.

---

### Proof of Concept

1. Traders open large long perp positions against the NLP pool subaccount. The pool accumulates unrealized losses (large negative `vQuoteBalance` in `PerpEngine`).
2. The sequencer computes `oraclePriceX18` for NLP based on realized quote balances only, without fully discounting unrealized perp losses. The price is overstated.
3. An NLP holder signs a `BurnNlp` transaction. The sequencer processes it with the inflated `oraclePriceX18`.
4. `burnNlp` computes `quoteAmount = nlpAmount × oraclePriceX18` (inflated), deducts it from pool subaccounts via `_applyNlpRebalance`.
5. No health check is performed on the pool subaccounts. Their quote balance goes negative.
6. The burner's health check passes (they received quote, their health improved).
7. When the profitable perp positions are settled via `settlePnl`, `state.availableSettle` is insufficient to cover payouts. The pool is insolvent.

    function _calculateProductHealth(
        uint32 productId,
        bytes32 subaccount,
        IProductEngine.HealthType healthType
    ) internal returns (int128 health) {
        RiskHelper.Risk memory risk = _risk(productId);
        (int128 amount, int128 quoteAmount) = _getBalance(
            productId,
            subaccount
        );
        int128 weight = RiskHelper._getWeightX18(risk, amount, healthType);
        health += quoteAmount;

        if (amount != 0) {
            if (weight == 2 * ONE) {
                return -INF;
            }
            health += amount.mul(weight).mul(risk.priceX18);
            emit PriceQuery(productId);
        }
    }
