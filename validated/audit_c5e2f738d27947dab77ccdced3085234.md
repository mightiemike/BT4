### Title
Linked Signer Can Overwrite Its Own Authorization via `LinkSigner` Transaction — (`core/contracts/EndpointTx.sol`)

### Summary

In `EndpointTx.processTransactionImpl()`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This permits the currently-set linked signer to sign a new `LinkSigner` transaction and overwrite the `linkedSigners[subaccount]` mapping — an ownership-level operation that should be restricted to the subaccount owner (the original key holder). This is a direct analog to the reported `walletV4_plugin.fc` bug: a lower-privileged entity retains the ability to mutate authorization state that should be exclusively controlled by the higher-privileged principal after a state transition.

---

### Finding Description

In `EndpointTx.processTransactionImpl()`, the `LinkSigner` transaction branch calls `validateSignedTx` with `allowLinkedSigner = true`:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ← allowLinkedSigner
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` routes through `validateSignature`, which passes `getLinkedSigner(sender)` as the accepted signer:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` accepts either the subaccount owner address OR the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

The result: once a linked signer is set for a subaccount, that linked signer can sign a `LinkSigner` transaction to replace itself with any arbitrary address — including an attacker-controlled one — without the subaccount owner's involvement.

The `linkedSigners` mapping is stored in `EndpointStorage`: [4](#0-3) 

---

### Impact Explanation

The `linkedSigners` mapping is the authorization root for all sequencer-processed transactions. A linked signer that can overwrite this mapping can:

1. Replace itself with a fresh attacker-controlled address, maintaining persistent access even after the original linked signer key is rotated by the subaccount owner.
2. Use the new linked signer to sign `TransferQuote` (also `allowLinkedSigner = true`, line 600–604) to drain the subaccount's quote balance to an attacker-controlled subaccount.
3. Use the new linked signer to sign `WithdrawCollateral` (V1, `allowLinkedSigner = true`, line 418–424) to withdraw collateral.

The concrete corrupted state is: `linkedSigners[victimSubaccount]` is overwritten to an attacker-controlled address, permanently mutating the subaccount's signing authority without the owner's consent. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

Any subaccount that has set a linked signer (a common pattern for API/hot-wallet trading) is exposed. The attack requires only a compromised linked signer key — no admin access, no governance capture, no sequencer compromise. The attacker-controlled entry path is fully unprivileged: submit a sequencer transaction of type `LinkSigner` signed by the compromised linked signer key. This is a realistic threat for any user operating with a hot-wallet linked signer.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This ensures that modifying the `linkedSigners` mapping requires a signature from the subaccount owner (the original key holder), not the delegated linked signer. This mirrors the fix in the reported `walletV4_plugin.fc` bug: the lower-privileged entity must be blocked from performing ownership-level state mutations after the delegation relationship is established.

---

### Proof of Concept

1. Alice sets `linkedSigners[aliceSubaccount] = hotWallet` via a `LinkSigner` slow-mode or sequencer transaction.
2. Attacker compromises `hotWallet`.
3. Attacker uses `hotWallet` to sign a `LinkSigner` transaction: `{ sender: aliceSubaccount, signer: attackerAddress, nonce: currentNonce }`.
4. Sequencer processes the transaction; `validateSignedTx` with `allowLinkedSigner = true` accepts the `hotWallet` signature.
5. `linkedSigners[aliceSubaccount]` is now set to `attackerAddress`.
6. Attacker signs a `TransferQuote` transaction using `attackerAddress` as the linked signer, transferring Alice's entire quote balance to `attackerSubaccount`.
7. Alice's collateral is drained. Even if Alice attempts to rotate the linked signer using her original subaccount key, the attacker can front-run with the already-set `attackerAddress` to drain funds first. [1](#0-0)
