Audit Report

## Title
`DepositAllowlistExtension` gates on `owner` instead of `sender`, allowing allowlist bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently drops the `sender` argument (the real depositor, i.e. `msg.sender` of `addLiquidity`) and checks the caller-supplied `owner` address instead. Because `owner` is a free parameter of `addLiquidity`, any non-allowlisted party can pass an allowlisted address as `owner` and the guard passes, while the actual depositor who pays tokens via the callback is never checked.

## Finding Description
`MetricOmmPool.addLiquidity` accepts `owner` as a free caller-supplied parameter and calls `_beforeAddLiquidity(msg.sender, owner, ...)`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both `sender` and `owner` positionally to the extension via `abi.encodeCall`: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first `address` argument but leaves it unnamed and unused, checking only `owner`: [3](#0-2) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly names and checks `sender`: [4](#0-3) 

No other guard in the `addLiquidity` path validates the actual caller against the allowlist. The `removeLiquidity` path enforces `msg.sender == owner`, but `addLiquidity` has no such constraint, making the `owner` parameter fully attacker-controlled. [5](#0-4) 

## Impact Explanation
A non-allowlisted caller (Bob) calls `pool.addLiquidity(owner = alice, ...)` where Alice is allowlisted. The guard checks `allowedDepositor[pool][alice]` → passes. Bob pays tokens via the modify-liquidity callback; Alice receives the LP shares. The allowlist is completely ineffective: every non-allowlisted address can deposit by nominating any allowlisted address as `owner`. Alice receives an unsolicited LP position she did not authorise and bears pool-price risk on capital she never intended to commit. The pool admin's access-control intent is entirely defeated, constituting broken core pool functionality and a direct loss of funds path for the allowlisted victim. [6](#0-5) 

## Likelihood Explanation
The bypass requires only a single direct call to `addLiquidity` with a known allowlisted address as `owner`. No special permissions, flash loans, or privileged access are needed. The `allowedDepositor` mapping is public, so any actor can observe the allowlist state and execute this immediately and repeatedly. [7](#0-6) 

## Recommendation
Replace the unnamed first parameter with `sender` and check it instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```solidity
// Alice is allowlisted; Bob is not.
allowlistExtension.setAllowedToDeposit(pool, alice, true);

// Bob calls addLiquidity with owner = alice.
// The guard checks allowedDepositor[pool][alice] == true → passes.
// Bob pays tokens via callback; Alice receives LP shares she never requested.
vm.prank(bob);
pool.addLiquidity(
    alice,        // owner — allowlisted, guard passes
    0,            // salt
    deltas,
    callbackData, // Bob's contract pays tokens here
    ""
);

// Bob has bypassed the allowlist entirely.
// Alice now holds an unsolicited LP position.
assertGt(positionShares(pool, alice, 0, binIdx), 0);
```
