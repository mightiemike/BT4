### Title
`check_declare_permissions` enforced on the RPC gateway admission path but absent from the consensus proposal conversion path — (`crates/apollo_gateway/src/gateway.rs`, `crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary

The gateway's `add_tx_inner` enforces `check_declare_permissions` — which gates on both `block_declare` (a kill-switch for all declares) and `is_authorized_declarer` (an allowlist of permitted senders) — for every `RpcTransaction::Declare` submitted through the HTTP/P2P gateway. The parallel consensus path, `convert_consensus_tx_to_internal_consensus_tx`, calls the shared `convert_rpc_tx_to_internal` helper directly and applies neither check. A malicious proposer can therefore include declare transactions from any sender in a block proposal; every validating node will convert and execute them without ever consulting the permission policy.

### Finding Description

**Gateway (single-transaction) path** — `crates/apollo_gateway/src/gateway.rs`:

```
add_tx_inner(RpcTransaction::Declare)
  └─ check_declare_permissions()          ← block_declare + is_authorized_declarer
  └─ stateless_tx_validator.validate()
  └─ convert_rpc_tx_to_internal_and_executable_txs()
       └─ convert_rpc_tx_to_internal()    ← compiled_class_hash check, hash calc
  └─ stateful validation
  └─ mempool.add_tx()
```

**Consensus (batch-proposal) path** — `crates/apollo_transaction_converter/src/transaction_converter.rs`:

```
convert_consensus_tx_to_internal_consensus_tx(ConsensusTransaction::RpcTransaction)
  └─ convert_rpc_tx_to_internal()         ← compiled_class_hash check, hash calc
                                           ← NO check_declare_permissions
```

`check_declare_permissions` reads two fields from `GatewayStaticConfig`:

- `block_declare: bool` — "If true, the gateway will block declare transactions."
- `authorized_declarer_accounts: Option<Vec<ContractAddress>>` — "If set, only these accounts can declare new contracts."

Both are operator-level controls intended to restrict which declare transactions reach the chain. They are enforced at lines 228–233 of `gateway.rs` for the RPC path but have no counterpart in `transaction_converter.rs` for the consensus path.

The consensus path is exercised in `validate_proposal.rs` when a validator node receives a `ProposalPart::Transactions` batch from a proposer:

```rust
futures::future::join_all(txs.into_iter().map(|tx| {
    transaction_converter.convert_consensus_tx_to_internal_consensus_tx(tx)
}))
```

No permission gate is applied before or after this conversion.

### Impact Explanation

When `authorized_declarer_accounts` is set (non-`None`), the operator intends that only the listed addresses may declare new contract classes. A malicious proposer — one that controls a sequencer node and can craft a `ProposalPart::Transactions` message directly, bypassing its own gateway — can include a `ConsensusTransaction::RpcTransaction(RpcTransaction::Declare(...))` from any arbitrary sender address. Every validating node will call `convert_consensus_tx_to_internal_consensus_tx`, which calls `convert_rpc_tx_to_internal`, which compiles the class and computes the hash without checking `is_authorized_declarer`. The transaction is forwarded to the batcher and executed, committing an unauthorized class declaration to the chain state.

Similarly, when `block_declare = true` (a temporary kill-switch), a malicious proposer can still land declare transactions on-chain through the consensus path.

This matches the allowed impact: **"Critical. Invalid or unauthorized Starknet transaction accepted through account validation … logic."** The `is_authorized_declarer` check is an account-level authorization gate; its absence from the consensus path means an unauthorized account's declare is accepted and committed.

### Likelihood Explanation

Any sequencer node that participates in consensus can act as a proposer. A malicious operator of such a node can construct a `ProposalPart::Transactions` payload directly (without routing through its own gateway) and include declare transactions from non-allowlisted senders. Validators have no independent gate and will accept the proposal. The attacker needs only a valid BFT proposer slot — no cryptographic break is required.

### Recommendation

Apply the same permission check in the consensus conversion path. The cleanest fix mirrors the gateway pattern: before (or inside) `convert_consensus_tx_to_internal_consensus_tx`, inspect each `ConsensusTransaction::RpcTransaction` for the `Declare` variant and enforce the same `block_declare` / `is_authorized_declarer` invariants that `check_declare_permissions` enforces in `add_tx_inner`.

```rust
// In convert_consensus_tx_to_internal_consensus_tx, before calling convert_rpc_tx_to_internal:
if let ConsensusTransaction::RpcTransaction(RpcTransaction::Declare(ref declare_tx)) = tx {
    check_declare_permissions(declare_tx, &self.gateway_config)?;
}
```

Alternatively, move `check_declare_permissions` into `convert_rpc_tx_to_internal` so it is enforced on every code path that converts a declare transaction.

### Proof of Concept

1. Operator configures the sequencer with `authorized_declarer_accounts = Some(["0xALLOWED"])` and `block_declare = false`.
2. A malicious proposer node constructs a `ProposalPart::Transactions` batch containing a `ConsensusTransaction::RpcTransaction(RpcTransaction::Declare(V3 { sender_address: 0xUNAUTHORIZED, ... }))`.
3. The proposer sends this batch to all validators via the consensus P2P channel.
4. Each validator calls `convert_consensus_tx_to_internal_consensus_tx` → `convert_rpc_tx_to_internal`. No call to `check_declare_permissions` occurs. The class is compiled, the hash is computed, and the transaction is forwarded to the batcher.
5. The batcher executes the declare transaction; the unauthorized class is committed to the Starknet state.
6. The same attack works with `block_declare = true`: the proposer bypasses the kill-switch entirely.

    fn check_declare_permissions(
        &self,
        declare_tx: &RpcDeclareTransaction,
    ) -> Result<(), StarknetError> {
        // TODO(noamsp): Return same error as in Python gateway.
        if self.config.static_config.block_declare {
            return Err(StarknetError {
                code: StarknetErrorCode::UnknownErrorCode(
                    "StarknetErrorCode.BLOCKED_TRANSACTION_TYPE".to_string(),
                ),
                message: "Transaction type is temporarily blocked.".to_string(),
            });
        }
        let RpcDeclareTransaction::V3(declare_v3_tx) = declare_tx;
        if !self.config.is_authorized_declarer(&declare_v3_tx.sender_address) {
            return Err(StarknetError {
                code: StarknetErrorCode::KnownErrorCode(
                    KnownStarknetErrorCode::UnauthorizedDeclare,
                ),
                message: format!(
                    "Account address {} is not allowed to declare contracts.",
                    &declare_v3_tx.sender_address
                ),
            });
        }
        Ok(())
    }
