Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the ultimate user, allowing any caller to bypass the per-pool swap allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the direct caller of `MetricOmmPool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. A pool admin who allowlists the router to support normal UX for approved users inadvertently grants unrestricted swap access to every address on the network.

## Finding Description

**Root cause — pool passes `msg.sender` verbatim:**

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that `sender` into the `beforeSwap` call without modification: [2](#0-1) 

**Root cause — extension checks the wrong actor:**

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

**Root cause — router does not forward the original user:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly. The router's `msg.sender` (the user) is stored only in transient callback context for payment purposes; it is never forwarded as `extensionData` or any other argument that the extension could read to identify the real swapper: [4](#0-3) 

**Exploit flow:**

1. Pool is deployed with `SwapAllowlistExtension`.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — alice is approved.
3. Admin calls `setAllowedToSwap(pool, router, true)` — router allowlisted so alice can use it.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
5. Router calls `pool.swap(...)` — `msg.sender` inside the pool is the router.
6. Pool calls `SwapAllowlistExtension.beforeSwap(sender=router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router]` → `true`.
8. Bob's swap executes despite bob not being in the allowlist.

**Existing guards are insufficient:** The extension has no mechanism to read the original user from `extensionData`, and the router provides no such data. The `allowedSwapper` mapping is keyed by the direct pool caller, not the economic actor.

## Impact Explanation

Any unprivileged EOA or contract can execute swaps on a pool intended to be restricted to a specific set of approved addresses (e.g., KYC-gated, institutional-only, or compliance-restricted pools) by routing through `MetricOmmSimpleRouter`. The allowlist extension's core invariant — that only approved addresses may swap — is silently broken for all router-mediated swaps. This constitutes a broken core pool functionality causing policy bypass and potential unauthorized extraction of LP value by actors the pool admin explicitly excluded. Severity: **High**.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard documented periphery entry point for swaps. Pool admins who configure `SwapAllowlistExtension` and want their approved users to have normal UX will naturally allowlist the router. The bypass requires no special privileges, no flash loans, and no malicious contracts — any EOA can call the router directly. The precondition (router allowlisted) is the expected production configuration for any allowlist-gated pool that supports router usage.

## Recommendation

The extension must verify the ultimate user, not the direct caller of `pool.swap()`. The preferred fix is to have `MetricOmmSimpleRouter` encode `msg.sender` (the real user) into `extensionData` for each hop, and have `SwapAllowlistExtension.beforeSwap` read and verify that declared swapper from `extensionData` when `sender` is a known trusted router. Alternatively, redesign so the allowlist is enforced at the router level with an `onlyPool` guard ensuring only the trusted router can reach the pool — separating concerns cleanly. Using `tx.origin` is discouraged as it excludes legitimate contract callers.

## Proof of Concept

```solidity
// Setup
pool = deploy pool with SwapAllowlistExtension
admin.setAllowedToSwap(pool, alice, true)   // alice approved
admin.setAllowedToSwap(pool, router, true)  // router allowlisted for alice's UX

// Attack (bob is NOT allowlisted)
// bob calls:
router.exactInputSingle(ExactInputSingleParams({pool: pool, ...}))
// → router calls pool.swap(...) with msg.sender = router
// → pool calls extension.beforeSwap(sender=router, ...)
// → extension checks allowedSwapper[pool][router] → true
// → swap executes for bob despite bob not being allowlisted

// Verify: direct call by bob reverts, router call succeeds
vm.prank(bob);
pool.swap(...) // reverts: NotAllowedToSwap ✓

vm.prank(bob);
router.exactInputSingle(...) // succeeds: bypass confirmed ✗
```
