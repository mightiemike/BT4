### Title
Missing Lower-Bound Validation on `chunk_size` in `starknet_getEvents` Returns Authoritative-Looking Wrong Pagination State — (`File: crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

`starknet_getEvents` validates that `chunk_size` does not exceed `max_events_chunk_size` (upper bound) but performs no lower-bound check. Sending `chunk_size: 0` causes the server to return an empty events array paired with a non-advancing `continuation_token` on every call, producing an authoritative-looking wrong RPC value and trapping any spec-compliant client in an infinite pagination loop.

---

### Finding Description

`EventFilter.chunk_size` is declared as a plain `usize` with no minimum constraint enforced in Rust code. [1](#0-0) 

The OpenAPI schema declares `"minimum": 1` for `chunk_size`, but this is a documentation hint only — it is never validated server-side. [2](#0-1) 

Inside `get_events`, the only guard is an upper-bound check:

```rust
if filter.chunk_size > self.max_events_chunk_size {
    return Err(ErrorObjectOwned::from(PAGE_SIZE_TOO_BIG));
}
``` [3](#0-2) 

There is no corresponding `chunk_size == 0` (or `chunk_size < 1`) rejection. Execution then falls into the event-collection loop:

```rust
if do_event_keys_match_filter(&content, &filter) {
    if filtered_events.len() == filter.chunk_size {   // 0 == 0 → true immediately
        return Ok(EventsChunk {
            events: filtered_events,                  // always empty
            continuation_token: Some(ContinuationToken::new(
                ContinuationTokenAsStruct(event_index), // points to first matching event
            )?),
        });
    }
    filtered_events.push(emitted_event);              // never reached
}
``` [4](#0-3) 

Because `filtered_events` starts empty and `0 == 0` is immediately true on the first matching event, the function returns `EventsChunk { events: [], continuation_token: Some(token_at_first_event) }` without ever pushing a single event. On the next call with that token, the same condition fires again at the same position, producing the identical response indefinitely.

The same zero-check path exists in the pending-block branch: [5](#0-4) 

---

### Impact Explanation

**High — RPC returns an authoritative-looking wrong value.**

A spec-compliant client that receives `EventsChunk { events: [], continuation_token: Some(t) }` correctly interprets this as "no events matched in this page, but more pages exist." It will re-issue the request with `t`, receive the same response, and loop forever. The server performs a full storage iterator scan (`iter_events`) on every such request, opening a storage read transaction and traversing the event index up to the first matching event each time.

The wrong value is the `continuation_token` itself: it claims there is a next page, but the next page is identical and the cursor never advances. This is an authoritative-looking incorrect pagination state delivered by the RPC layer.

---

### Likelihood Explanation

Any unauthenticated caller of `starknet_getEvents` can supply `chunk_size: 0`. The JSON-RPC framework deserializes it as `usize(0)` without error. No privilege is required. The trigger is a single well-formed JSON-RPC request:

```json
{"jsonrpc":"2.0","method":"starknet_getEvents","params":[{"chunk_size":0}],"id":1}
```

---

### Recommendation

Add an explicit lower-bound check immediately after the upper-bound check in `get_events`:

```rust
if filter.chunk_size == 0 {
    return Err(ErrorObjectOwned::from(PAGE_SIZE_TOO_BIG));
}
if filter.chunk_size > self.max_events_chunk_size {
    return Err(ErrorObjectOwned::from(PAGE_SIZE_TOO_BIG));
}
```

Alternatively, change `chunk_size` to a `NonZeroUsize` so the constraint is enforced at the type level during deserialization, matching the `"minimum": 1` already declared in the OpenAPI schema. [6](#0-5) 

---

### Proof of Concept

**Setup:** A running node with at least one block containing events.

**Request:**
```json
{"jsonrpc":"2.0","id":1,"method":"starknet_getEvents",
 "params":[{"chunk_size":0,"from_block":{"block_number":0}}]}
```

**Expected (correct) response:** `PAGE_SIZE_TOO_BIG` error (code 31).

**Actual response:**
```json
{
  "result": {
    "events": [],
    "continuation_token": "<token pointing to first event>"
  }
}
```

**Repeat** the request with `"continuation_token": "<token>"` added — the server returns the identical response with the same token on every call. The cursor never advances, no events are ever delivered, and each call opens a storage read transaction and scans the event index. [7](#0-6) [4](#0-3)
