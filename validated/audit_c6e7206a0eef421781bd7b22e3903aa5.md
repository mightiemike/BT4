Audit Report

## Title
`block.number` Returns L1 Ancestor Block on Arbitrum, Breaking `PriceVelocityGuardExtension` Velocity Guard — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary

`PriceVelocityGuardExtension.beforeSwap` uses `block.number` to compute `blockDiff`, which scales the allowed oracle mid-price movement between swaps. On Arbitrum, `block.number` returns the L1 ancestor block number (~12 s cadence) rather than the L2 block number (~0.25 s cadence). Because ~48 L2 blocks are produced per L1 block, `blockDiff` is `0` for every swap after the first within the same L1 epoch, collapsing the guard's time-scaling factor to `1`. This produces two fund-impacting failure modes: DoS of legitimate swaps (if calibrated for L2 blocks) and complete guard bypass enabling LP value drain (if calibrated for L1 blocks). No chain-aware block-number override exists anywhere in the codebase.

## Finding Description

The velocity guard in `beforeSwap` computes:

```solidity
s.lastUpdateBlock = uint64(block.number);   // line 58
// ...
uint256 blockDiff = block.number - prevBlock; // line 63
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff); // line 70
```

On Arbitrum, `block.number` is the L1 ancestor block (advances ~every 12 s). With ~48 L2 blocks per L1 block, every swap after the first within a single L1 epoch reads `block.number == prevBlock`, so `blockDiff = 0` and `allowedSq = maxChange²` — a constant, regardless of elapsed L2 time.

**Failure Mode A — DoS (calibrated for L2 blocks):** A pool admin deploying on Arbitrum naturally calibrates `maxChangePerBlockE18` for the 0.25 s L2 cadence (e.g., 0.1% per L2 block). Normal market volatility of 0.2% over 0.5 s (2 L2 blocks, still within one L1 block) causes `actualSq = (2e15)² = 4e30 > 1e30 = allowedSq`, reverting with `PriceVelocityExceeded`. Every subsequent swap in that L1 window is blocked. Core swap functionality is unusable for ~12 s per L1 block.

**Failure Mode B — Guard bypass / LP drain (calibrated for L1 blocks):** If the admin calibrates for L1 blocks (e.g., 1% per 12 s), each of the 48 L2-block swaps within one L1 block individually passes the guard (each sees `blockDiff = 0`, each is allowed 1% movement). A sequence of 48 swaps can shift the oracle-derived mid by up to ~48% within a single L1 block — 48× the intended cap — while the guard never fires.

A grep across the entire codebase for `arbBlockNumber`, `ArbSys`, and chain-specific block-number overrides returns zero results in production contracts, confirming no mitigation exists.

## Impact Explanation

- **Failure Mode A**: Core swap functionality is rendered unusable on Arbitrum for any pool using this extension with L2-calibrated parameters. LPs cannot earn fees; traders cannot execute. This is broken core pool functionality causing loss of funds (foregone fees, inability to exit positions at desired prices).
- **Failure Mode B**: The oracle-velocity guard — the sole mechanism protecting LP principal from rapid oracle-price manipulation — is bypassed on Arbitrum. LPs trade against prices up to 48× outside the intended velocity envelope, constituting direct loss of LP principal. This meets the "bad-price execution" and "direct loss of user principal" impact gates.

## Likelihood Explanation

Arbitrum is a confirmed deployment target (Arbitrum config present in the repository). The miscalibration is silent — no revert, no event — and `block.number` on Arbitrum is documented to return the L1 block. Any pool admin who sets `maxChangePerBlockE18` based on Arbitrum's 0.25 s L2 block time (the natural, documented block cadence) triggers Failure Mode A immediately on the first swap pair within an L1 epoch. Failure Mode B requires no special privilege: any actor who can submit 48 swaps across 48 consecutive L2 blocks within one L1 epoch (a standard MEV or arbitrage pattern) can exploit it.

## Recommendation

Replace `block.number` with a chain-aware helper that calls the ArbSys precompile (`address(0x64).call(abi.encodeWithSignature("arbBlockNumber()"))`) on Arbitrum and falls back to `block.number` on other chains. Alternatively, replace the block-based velocity formula entirely with a `block.timestamp`-based one (which is correct and consistent on all EVM chains, including Arbitrum, without chain-specific branching). Both `setLastMidPrice` (line 32) and `beforeSwap` (lines 58, 63) must be updated consistently.

## Proof of Concept

**Setup**: Arbitrum mainnet fork. `PriceVelocityGuardExtension` deployed. Pool configured with `maxChangePerBlockE18 = 1e15` (0.1% per block, calibrated for 0.25 s L2 blocks).

**Failure Mode A — DoS**:
1. Swap S1 at L2 block 1000, L1 block 50. Oracle mid = 1.000. Guard writes `lastMidPriceX64 = 1.000`, `lastUpdateBlock = 50`.
2. Oracle mid moves to 1.002 (0.2% — normal 0.5 s market move, 2 L2 blocks elapsed).
3. Swap S2 at L2 block 1002, L1 block 50 (same L1 block). `blockDiff = 50 − 50 = 0`. `allowedSq = (1e15)² × 1 = 1e30`. `changeE18 = 2e15`. `actualSq = 4e30 > 1e30`. Guard reverts `PriceVelocityExceeded`. All subsequent swaps in this L1 window are blocked.

**Failure Mode B — bypass**:
1. Admin sets `maxChangePerBlockE18 = 1e16` (1% per L1 block).
2. Adversary submits 48 swaps across L2 blocks 1000–1047, all within L1 block 50. Each moves oracle mid by 1%. `blockDiff = 0` for swaps 2–48; each passes individually.
3. Cumulative oracle-mid shift: ~48%. Guard never fires. LPs trade against a price 48× outside the intended velocity cap.
