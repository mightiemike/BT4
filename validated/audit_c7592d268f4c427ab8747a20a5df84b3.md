### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` was deprecated in v0.5.0 specifically because it is vulnerable to the 64-byte transaction Merkle proof forgery attack. Its replacement, `verify_transaction_inclusion_v2`, adds a mandatory coinbase proof check to close that attack vector. However, the deprecated function carries no access control beyond `#[pause]` and remains fully callable by any unprivileged NEAR account. An attacker can bypass the security fix entirely by calling the old entry point directly.

---

### Finding Description

`verify_transaction_inclusion` is still a live, public NEAR method: [1](#0-0) 

Its only guard is `#[pause]`, meaning any account can call it while the contract is unpaused. The function computes a Merkle root from the caller-supplied `tx_id` and `merkle_proof` and compares it to the stored block `merkle_root`: [2](#0-1) 

It performs **no coinbase proof validation**. The 64-byte vulnerability (documented at the bitmex link in the deprecation note) allows an attacker to supply an internal Merkle-tree node hash as `tx_id` together with a valid sibling path; the computed root still matches the real block root, so the function returns `true` for a transaction that was never included.

`verify_transaction_inclusion_v2` closes this by first verifying that the coinbase transaction anchors the proof at leaf position 0: [3](#0-2) 

It then delegates to the deprecated function for the rest of the check: [4](#0-3) 

Because the deprecated function is still a public method, the coinbase guard in `verify_transaction_inclusion_v2` is trivially bypassed by calling the old entry point directly.

---

### Impact Explanation

Any downstream NEAR contract or off-chain system that calls `verify_transaction_inclusion` (rather than `_v2`) receives a `true` result for a fabricated transaction inclusion claim. This corrupts the proof-verification output that the light client is designed to provide, potentially allowing an attacker to prove that an arbitrary Bitcoin transaction was included in a block when it was not. Contracts that gate fund releases or cross-chain actions on this result are directly at risk.

---

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` while the contract is unpaused. The 64-byte Merkle forgery technique is publicly documented and well understood. The only prerequisite is knowledge of a real block's Merkle tree structure, which is public Bitcoin data. Likelihood is **Medium** — the attack requires crafting a valid internal-node proof, but the technique is known and the entry point is wide open.

---

### Recommendation

Remove `verify_transaction_inclusion` as a public NEAR method. If backward compatibility is required, restrict it with an access-control role (e.g., `Role::DAO`) so it cannot be called by unprivileged accounts, or gate it behind a feature flag that is disabled in production builds. The internal call from `verify_transaction_inclusion_v2` can be refactored into a private helper to preserve code reuse without exposing the vulnerable path.

---

### Proof of Concept

1. Identify any mainchain block stored by the contract (e.g., via `get_block_hash_by_height`).
2. Obtain the block's full transaction Merkle tree from a Bitcoin node.
3. Select an internal node `N` at depth ≥ 1 whose 32-byte hash, when treated as a transaction ID, produces a valid sibling path to the Merkle root.
4. Construct `ProofArgs { tx_id: N, tx_block_blockhash: <block>, tx_index: <index of N>, merkle_proof: <sibling path>, confirmations: 1 }`.
5. Call `verify_transaction_inclusion` directly (not `_v2`) with these args.
6. The function computes `compute_root_from_merkle_proof(N, index, siblings)` which equals the real Merkle root, and returns `true` — falsely asserting that the internal node is an included transaction. [5](#0-4)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
