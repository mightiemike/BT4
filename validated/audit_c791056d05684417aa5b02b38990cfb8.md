### Title
`SpotEngineState::_updateState()` applies a stale utilization ratio as constant over the full `dt` window, causing systematic interest undercharge — (File: `core/contracts/SpotEngineState.sol`)

### Summary

`SpotEngineState::_updateState()` derives the borrower rate from the utilization ratio at the **start** of the elapsed period, then compounds it for the full `dt` seconds as if utilization were frozen. Because interest accrual itself grows `totalBorrows`, the true utilization ratio rises monotonically during `dt`, meaning the rate applied is always lower than the time-averaged rate that should have governed the period. Borrowers systematically underpay; lenders and the protocol fee account receive less than the rate model intends.

---

### Finding Description

In `SpotEngineState::_updateState()`, the function loads the state that was persisted at the **previous** `updateStates` call and computes:

```solidity
// SpotEngineState.sol lines 61–64
int128 totalBorrows = state.totalBorrowsNormalized.mul(
    state.cumulativeBorrowsMultiplierX18          // multiplier from last update
);
int128 utilizationRatioX18 = totalBorrows.div(totalDeposits);
``` [1](#0-0) 

This `utilizationRatioX18` reflects the borrow balance **as of the previous update** — the multiplier has not yet been advanced. The function then derives `borrowerRateX18` from this stale utilization and compounds it for the entire window:

```solidity
// SpotEngineState.sol line 98
borrowRateMultiplierX18 = (ONE + borrowerRateX18).pow(int128(dt));
``` [2](#0-1) 

The multiplier is then applied to `cumulativeBorrowsMultiplierX18`:

```solidity
// SpotEngineState.sol lines 129–131
state.cumulativeBorrowsMultiplierX18 = state
    .cumulativeBorrowsMultiplierX18
    .mul(borrowRateMultiplierX18);
``` [3](#0-2) 

The broken invariant: the rate model is a function of utilization, and utilization is a function of `totalBorrows`. As `totalBorrows` grows during `dt` (due to the very interest being accrued), utilization rises, which pushes the rate higher — especially past the inflection point where `interestLargeCapX18` applies. The code treats utilization as constant for the full `dt`, so the rate used is always the **floor** of what should have applied over the interval.

`updateStates` is the entry point, called by the endpoint with a caller-supplied `dt` bounded by `7 * SECONDS_PER_DAY`:

```solidity
// SpotEngine.sol lines 265–283
function updateStates(uint128 dt) external onlyEndpoint {
    ...
    require(dt < 7 * SECONDS_PER_DAY, ERR_INVALID_TIME);
    for (uint32 i = 0; i < productIds.length; i++) {
        ...
        State memory state = states[productId];
        _updateState(productId, state, dt);
        _setState(productId, state);
    }
}
``` [4](#0-3)

### Impact Explanation

Every `updateStates` call where `dt > 0` and `totalBorrows > 0` undercharges borrowers. The error magnitude scales with:

- **`dt`**: larger gaps between updates → larger error (up to 7 days).
- **Utilization**: above the inflection point (`interestInflectionUtilX18 = 0.8` for the quote product), the steep `interestLargeCapX18` slope amplifies the rate sensitivity to utilization changes.
- **Absolute borrow size**: larger pools → larger absolute fee shortfall.

The `FEES_ACCOUNT` accrues fewer protocol fees (20% of interest, `INTEREST_FEE_FRACTION`), and lenders' `cumulativeDepositsMultiplierX18` grows more slowly than it should. The accounting invariant `liquidity = deposits − borrows` is preserved, but the **distribution** of that liquidity between borrower cost and lender yield is skewed in borrowers' favor on every update cycle. [5](#0-4) 

---

### Likelihood Explanation

This is not an edge case. It fires on every periodic `updateStates` call as long as any product has outstanding borrows — which is the normal operating state of the lending pool. No special user action, privileged access, or external condition is required. The sequencer calls `updateStates` routinely; the error accumulates silently on every such call.

---

### Recommendation

Replace the single-snapshot rate compounding with a closed-form solution that accounts for the coupled growth of borrows and utilization over `dt`. Concretely, the ODE governing the system is:

```
dB/dt = r(B/D) * B
```

where `B` is total borrows, `D` is total deposits (approximately constant between updates), and `r(u)` is the piecewise-linear rate function. A closed-form integral exists for each linear segment. Alternatively, if a closed form is impractical, reduce the maximum allowed `dt` significantly (e.g., to minutes rather than days) so the approximation error is negligible.

---

### Proof of Concept

Assume:
- `totalDeposits = 1000`, `totalBorrows = 850` → utilization = 85% (above inflection at 80%)
- `interestFloorX18 = 1%`, `interestSmallCapX18 = 4%`, `interestLargeCapX18 = 100%`
- Annualized rate at 85%: `1% + 4% + 100% * (0.85 − 0.80)/(1 − 0.80) = 30%`
- `dt = 7 days`

**What the code does:** applies 30% annualized for 7 days straight, using the utilization at the start of the window.

**What should happen:** as borrows compound at 30%/yr, after ~3 days `totalBorrows` has grown enough to push utilization to ~86%, raising the rate to ~35%. The second half of the window should use a higher rate. The code ignores this, undercharging borrowers and shortchanging lenders and the fee account for the entire 7-day window. [6](#0-5)
