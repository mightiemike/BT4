Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual user, enabling allowlist bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the `pool.swap()` call — the direct caller of the pool, not the economic actor initiating the trade. When `MetricOmmSimpleRouter` intermediates a swap, `sender` is the router's address. Any pool admin who allowlists the router (a necessary step for any allowlisted user to swap via the router) simultaneously grants every non-allowlisted user the ability to bypass the per-user swap restriction by routing through the router.

## Finding Description
`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, recipient, ...)`, passing its own `msg.sender` — the direct caller — as `sender` to the extension system. `ExtensionCalling._beforeSwap` encodes this `sender` value and dispatches it to `SwapAllowlistExtension.beforeSwap(address sender, ...)`, which checks `allowedSwapper[msg.sender][sender]` (i.e., `allowedSwapper[pool][router]`).

`MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` with `msg.sender = router`. The router forwards `params.extensionData` unmodified and injects no user identity into any field the extension can read. There is no mechanism in the router or extension to recover the actual user's address.

This creates an irresolvable conflict: allowlisting the router (required for any allowlisted user to use the router) simultaneously allowlists every user who calls the router. A non-allowlisted user B can call `router.exactInputSingle({pool: P, ...})`, the router calls `P.swap(B, ...)` with `msg.sender = router`, the extension checks `allowedSwapper[P][router] == true`, and the swap executes — bypassing the per-user restriction entirely. The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

Contrast with `DepositAllowlistExtension.beforeAddLiquidity`, which correctly checks `allowedDepositor[msg.sender][owner]` — the position owner, the economically relevant actor — and is not susceptible to this router-mediated bypass.

## Impact Explanation
Non-allowlisted users can execute swaps against LP liquidity in pools configured with `SwapAllowlistExtension` and a router allowlist entry. LPs in KYC-gated, institutional, or otherwise restricted pools suffer adverse selection from unauthorized counterparties they did not consent to trade with. This constitutes a direct loss of owed LP assets through unintended exposure and adverse price impact from unintended swap counterparties — matching the "Broken core pool functionality causing loss of funds" and "Admin-boundary break: pool admin role checks bypassed by an unprivileged path" allowed impact criteria.

## Likelihood Explanation
Medium. The precondition — router allowlisted in the extension — is a necessary operational step for any pool that uses `SwapAllowlistExtension` and wants to support router-mediated swaps for its allowlisted users. Any such pool is vulnerable by construction. The attacker requires no special privileges, no malicious setup, and no tokens beyond what is needed to execute the swap itself. The attack is repeatable on every block.

## Recommendation
1. **Correct fix**: Modify `MetricOmmSimpleRouter` to prepend `msg.sender` (the actual user) to `extensionData` before forwarding to the pool. Modify `SwapAllowlistExtension.beforeSwap` to decode the actual user's address from `extensionData` when `sender` is a known router, and verify that decoded address against `allowedSwapper`.
2. **Alternative**: Add a separate `allowedRouter` mapping in the extension. When `sender` is an allowed router, require the extension to decode and verify the actual user identity from `extensionData`. This separates "router is trusted to relay" from "specific user is allowed to swap."
3. **Minimum**: Document explicitly that `SwapAllowlistExtension` is incompatible with router-mediated swaps and must only be used with direct `pool.swap()` calls; disable router allowlisting at the extension level if possible.

## Proof of Concept
```
Setup:
  - Pool P has SwapAllowlistExtension E configured
  - Pool admin allowlists user A (KYC'd): allowedSwapper[P][A] = true
  - Pool admin allowlists router R (so A can use the router): allowedSwapper[P][R] = true

Attack:
  1. Non-allowlisted user B calls:
       router.exactInputSingle({pool: P, recipient: B, amountIn: X, ...})

  2. Router (msg.sender = R) calls:
       P.swap(B, zeroForOne, amountIn, priceLimitX64, "", extensionData)

  3. Pool calls:
       _beforeSwap(sender=R, recipient=B, ...)

  4. Extension checks:
       allowedSwapper[P][R] == true  →  no revert

  5. Swap executes. User B, not allowlisted, successfully swaps in the restricted pool.

Foundry test outline:
  - Deploy pool with SwapAllowlistExtension
  - setAllowedToSwap(pool, userA, true)
  - setAllowedToSwap(pool, router, true)
  - vm.prank(userB); router.exactInputSingle(...)
  - Assert swap succeeds (no NotAllowedToSwap revert)
  - Assert userB received output tokens
```
