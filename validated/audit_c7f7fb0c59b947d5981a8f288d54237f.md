### Title
Deprecated `verify_transaction_inclusion` Remains Live Public Endpoint, Enabling 64-Byte Merkle Proof Forgery — (`File: contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` was deprecated in v0.5.0 because it is vulnerable to the 64-byte transaction Merkle proof forgery attack. Despite the deprecation, the function remains a fully callable public NEAR contract method with no access restriction. Any unprivileged caller can supply a crafted `ProofArgs` where `tx_id` is an internal Merkle tree node rather than a real transaction hash, and the function will return `true`. Downstream contracts that gate value-bearing actions on this result can be deceived into accepting a forged proof.

### Finding Description

The function `verify_transaction_inclusion` is annotated `#[deprecated(since = "0.5.0")]` and its own doc-comment warns:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [1](#0-0) 

The Rust `#[deprecated]` attribute is a **compiler lint only**. It does not remove the method from the compiled WASM binary or prevent it from being dispatched on-chain. The function is still decorated with `#[pause]` (making it a live NEAR contract method) and accepts any `ProofArgs` from any caller. [2](#0-1) 

The root cause is in `merkle_tools::compute_root_from_merkle_proof`: it accepts any 32-byte value as `transaction_hash` and walks it up the tree without distinguishing leaf nodes from internal nodes. [3](#0-2) 

Bitcoin's Merkle tree is computed as `double_sha256(left || right)`. An internal node at depth `d` is itself a valid 32-byte hash. An attacker can treat that internal node as the `tx_id`, provide the remaining `d`-step proof to the root, and `compute_root_from_merkle_proof` will reproduce the correct `merkle_root`, causing the function to return `true`.

The v2 function fixes this by first verifying a coinbase proof (the coinbase is always at index 0 and is a real leaf), which anchors the tree depth and prevents an internal node from being passed off as a leaf. [4](#0-3) 

However, the v1 function is never removed or access-gated, so the fix in v2 is bypassed simply by calling v1 directly.

### Impact Explanation

Any NEAR contract that calls `verify_transaction_inclusion` (v1) to authorize a state change — e.g., releasing bridged funds, minting wrapped tokens, or recording a cross-chain event — can be made to accept a proof for a transaction that does not exist. The attacker does not need to forge a block header or break SHA-256; they only need to identify an internal Merkle node in any already-accepted mainchain block and construct the shorter proof from that node to the root. The corrupted value is the boolean proof result returned to the consuming contract, which is the sole authorization signal for the downstream action.

### Likelihood Explanation

The entry path requires no privileges: `verify_transaction_inclusion` is callable by any NEAR account. The required inputs (a real mainchain block hash and its Merkle tree structure) are fully public on-chain and via Bitcoin RPC. The forgery technique is well-documented (https://www.bitmex.com/blog/64-Byte-Transactions). Any attacker who knows a consuming contract relies on v1 can execute this immediately.

### Recommendation

Remove `verify_transaction_inclusion` from the public contract interface entirely, or add an explicit `#[private]` attribute (or a role-gated access check) so it cannot be dispatched by external callers. The `#[deprecated]` Rust attribute provides zero on-chain enforcement. The v2 function should be the only externally callable proof-verification endpoint.

### Proof of Concept

1. Identify any block `B` already stored in the contract's mainchain (e.g., via `get_block_hash_by_height`). Its `merkle_root` is stored in `headers_pool`.
2. Obtain the full transaction list for block `B` from a Bitcoin node. Compute the Merkle tree. Pick any internal node `N` at depth `d` (e.g., the parent of the first two transactions). `N = double_sha256(tx0 || tx1)`.
3. Compute the Merkle proof from `N` upward: a `(tree_depth - d)`-step proof that reconstructs `merkle_root` from `N`.
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (an internal node, not a real transaction)
   - `tx_block_blockhash = B`
   - `tx_index = 0` (position of `N` at depth `d`)
   - `merkle_proof = [sibling_of_N, ..., sibling_at_root_level]`
   - `confirmations = 1`
5. `compute_root_from_merkle_proof(N, 0, proof)` reproduces `merkle_root` exactly.
6. The function returns `true` for a transaction that does not exist in block `B`. [5](#0-4) [6](#0-5)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
