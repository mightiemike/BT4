Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Swapper, Allowing Full Allowlist Bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the pool's `msg.sender`. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract address, not the end user. A pool admin who allowlists the router address — the natural step to enable router-based swaps for permitted users — inadvertently grants every on-chain address the ability to bypass the per-user allowlist, collapsing the curated pool's entire access-control invariant.

## Finding Description

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- pool's msg.sender, not the originating user
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first argument to every configured extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`.

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

Here `msg.sender` is the pool and `sender` is whatever the pool received as its own `msg.sender`. When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `IMetricOmmPoolActions(params.pool).swap(...)` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ...);
```

At that point the pool's `msg.sender` is the **router address**, so the extension evaluates `allowedSwapper[pool][router]` — not `allowedSwapper[pool][actual_user]`. The same substitution occurs for `exactInput` (L104), `exactOutputSingle` (L136), and `exactOutput` (L165). The admin has no on-chain signal that `allowedSwapper[pool][router] = true` is semantically different from any other per-address entry; the storage key looks identical.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` does not share this flaw because `addLiquidity` carries an explicit `owner` parameter that the pool passes through to the extension separately from `msg.sender`, so the check `allowedDepositor[msg.sender][owner]` always reflects the true position owner regardless of who called the pool.

## Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` (i.e., `allowAll = false`) and then calls `setAllowedToSwap(pool, router, true)` — the required step to let their allowlisted users trade through the standard periphery — inadvertently opens the allowlist to every address on-chain. Any unpermitted address can call `MetricOmmSimpleRouter.exactInputSingle` and the extension will approve the swap because `allowedSwapper[pool][router] == true`. The curated pool's access-control invariant collapses: unauthorized traders can execute swaps, drain the pool's preferred-price liquidity, and front-run allowlisted LPs, causing direct loss of LP principal and fee revenue. This meets the "Broken core pool functionality causing loss of funds" and "Admin-boundary break" allowed impact criteria.

## Likelihood Explanation

The bypass requires the pool admin to allowlist the router, which is the natural, expected configuration step for any curated pool that wants to support the standard periphery. Without it, even allowlisted users cannot trade through the router. The admin has no on-chain signal that allowlisting the router grants universal access; the extension's storage key (`allowedSwapper[pool][router]`) looks identical to any other per-address entry. The bypass is therefore reachable through a routine, well-motivated admin action with no special attacker capability required beyond calling the public router.

## Recommendation

The pool's `swap` function must expose the originating user identity to extensions. The cleanest fix is to add a `swapper` parameter to `IMetricOmmPoolActions.swap` and have the router forward `msg.sender` there. The extension then checks `allowedSwapper[pool][swapper]` instead of `allowedSwapper[pool][sender]`, mirroring the pattern already used correctly by `DepositAllowlistExtension` with its explicit `owner` parameter. Until this is applied, `SwapAllowlistExtension` must document that it is incompatible with any intermediary caller and that the pool must only be accessed directly.

## Proof of Concept

```
Setup
─────
1. Deploy pool with SwapAllowlistExtension configured (allowAll = false).
2. Pool admin calls extension.setAllowedToSwap(pool, router, true)
   — intending to let allowlisted users trade through the router.
3. Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

Attack
──────
4. Attacker (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
5. Router calls pool.swap(recipient, ...) — pool's msg.sender = router.
6. Pool calls _beforeSwap(sender=router, ...).
7. SwapAllowlistExtension checks allowedSwapper[pool][router] → true → passes.
8. Swap executes. Attacker receives output tokens.

Expected: revert NotAllowedToSwap.
Actual:   swap succeeds; allowlist is fully bypassed.

Foundry test skeleton:
- Deploy SwapAllowlistExtension, pool, and MetricOmmSimpleRouter.
- Admin: setAllowedToSwap(pool, address(router), true).
- Attacker (EOA not in allowlist): call router.exactInputSingle(...).
- Assert swap succeeds (no revert), confirming bypass.
```
