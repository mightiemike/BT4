### Title
Checkpoint Advancement Without Coin Transfer on Insufficient Bonus Pool During Redelegation Slash — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

When `BeforeRedelegationSlashed` fires and the `RewardsPoolName` module account has insufficient balance to cover the computed bonus, `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool` **after** it has already (a) decremented event reference counts in persistent store and (b) mutated the in-memory `pos` with advanced `LastEventSeq`, `LastBonusAccrual`, and `LastKnownBonded`. The caller `slashRedelegationPosition` deliberately swallows that specific error, then calls `setPositionWithState` with the mutated `pos`, permanently persisting the advanced checkpoints. The position owner's accrued bonus is irrecoverably lost: the events are consumed, the checkpoints are advanced, but `SendCoinsFromModuleToAccount` is never called.

---

### Finding Description

**Step 1 — Entry point.**
`BeforeRedelegationSlashed` is a staking hook that fires during `SlashRedelegation` in the SDK staking module. It is reachable whenever a validator that is the *source* of an active redelegation is slashed. No special privilege is required; any validator slash triggers it. [1](#0-0) 

**Step 2 — Error swallowing in `slashRedelegationPosition`.**
The function calls `processEventsAndClaimBonus` and explicitly swallows `ErrInsufficientBonusPool` to avoid a chain halt: [2](#0-1) 

After the swallow, execution falls through to `setPositionWithState` with the already-mutated `pos`: [3](#0-2) 

**Step 3 — State mutations that occur BEFORE the pool check inside `processEventsAndClaimBonus`.**

Inside the event loop, for every event processed:
- `pos.UpdateLastEventSeq(entry.Seq)` advances the in-memory checkpoint.
- `k.decrementEventRefCount(ctx, valAddr, entry.Seq)` **writes to the persistent store** (either decrements `ReferenceCount` or removes the event entirely). [4](#0-3) 

After the loop, `applyBonusAccrualCheckpoint` and `UpdateLastKnownBonded` advance the remaining in-memory checkpoints: [5](#0-4) 

**Only then** is the pool balance checked: [6](#0-5) 

When `sufficientBonusPoolBalance` fails, the function returns `ErrInsufficientBonusPool`. At this point:
- All `decrementEventRefCount` writes are already committed to the `CacheMultiStore` — they are **not** rolled back by returning an error.
- `pos.LastEventSeq`, `pos.LastBonusAccrual`, and `pos.LastKnownBonded` are all advanced in memory.
- `SendCoinsFromModuleToAccount` has **not** been called. [7](#0-6) 

**Step 4 — Checkpoint persistence.**
Back in `slashRedelegationPosition`, the error is swallowed and `setPositionWithState` is called with the mutated `pos`, writing the advanced `LastEventSeq`, `LastBonusAccrual`, and `LastKnownBonded` to `k.Positions`: [8](#0-7) 

**Step 5 — Irrecoverability.**
On the next call to `processEventsAndClaimBonus` for this position, `getValidatorEventsSince` starts from the now-advanced `pos.LastEventSeq`: [9](#0-8) 

The events that were already reference-count-decremented (and potentially garbage-collected) will not be returned. The bonus for those segments is permanently unclaimable.

The `decrementEventRefCount` implementation confirms that events are deleted from store when `ReferenceCount` reaches 1: [10](#0-9) 

---

### Impact Explanation

**Permanent loss of accrued bonus rewards.** The position owner is owed bonus coins for the bonded segments between `pos.LastBonusAccrual` and the slash block time. After this bug triggers:
- The events encoding those segments are consumed (ref counts decremented, events possibly deleted).
- The position's checkpoints are advanced past those events.
- No coins are ever transferred.
- There is no recovery path: the owner cannot re-claim because the events are gone and the checkpoints have moved forward.

This is a direct, quantifiable fund loss scoped to the position owner. The magnitude equals `shares × tokensPerShare × bonusApy × duration / SecondsPerYear` for all bonded segments since the last accrual.

---

### Likelihood Explanation

The precondition — `RewardsPoolName` balance below the computed bonus at the moment of a redelegation slash — is realistic:
- The pool is funded by governance/admin deposits and can be depleted or unfunded.
- Any validator slash on a source validator of an active tier-position redelegation triggers the path.
- No attacker action is required beyond having a position with an active redelegation; the slash itself is a normal staking event.

The ADR explicitly documents the intended behavior for user-driven paths as "fail atomically, user retries after pool is replenished" — but the `BeforeRedelegationSlashed` path does not implement atomic failure.



---

### Recommendation

The fix must ensure that **no persistent state is mutated** when the pool is insufficient. Two concrete approaches:

1. **Move the pool check before the loop.** Pre-compute `totalBonus` in a dry-run pass (without decrementing ref counts or advancing checkpoints), check the pool, and only then execute the state-mutating pass.

2. **Use a branch/cache context.** Execute `processEventsAndClaimBonus` inside a `ctx.CacheContext()` branch. Only `Write()` the branch if the function succeeds; discard it on `ErrInsufficientBonusPool`. This ensures all store writes (including `decrementEventRefCount`) are atomically rolled back.

Option 2 is simpler and less error-prone. In `slashRedelegationPosition`, replace the direct call with a cache-context wrapper:

```go
cacheCtx, commit := sdk.UnwrapSDKContext(ctx).CacheContext()
_, err := k.processEventsAndClaimBonus(cacheCtx, &pos)
if err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error(...)
        // pos is NOT mutated; do not commit; proceed with original pos
    } else {
        return err
    }
} else {
    commit() // only commit store writes on success
}
```

This preserves the chain-halt-prevention intent while guaranteeing the invariant: **a position's checkpoints must not advance unless the corresponding bonus coins are actually transferred**.

### Proof of Concept

```go
func (s *KeeperSuite) TestSlashRedelegation_InsufficientPool_CheckpointsAdvanceWithoutPayout() {
    lockAmount := sdkmath.NewInt(sdk.DefaultPowerReduction.Int64())
    _, bondDenom := s.getStakingData()
    // Fund pool with ZERO so sufficientBonusPoolBalance always fails
    // (do NOT call s.fundRewardsPool)

    pos, _, unbondingID := s.setupRedelegatingPosition(lockAmount)
    owner := sdk.MustAccAddressFromBech32(pos.Owner)

    // Advance time so bonus accrues
    s.ctx = s.ctx.WithBlockTime(s.ctx.BlockTime().Add(30 * 24 * time.Hour))

    balBefore := s.app.BankKeeper.GetBalance(s.ctx, owner, bondDenom)

    // Trigger redelegation slash — pool is empty, ErrInsufficientBonusPool fires
    err := s.keeper.Hooks().BeforeRedelegationSlashed(s.ctx, unbondingID, pos.Delegation.Shares.Quo(sdkmath.LegacyNewDec(10)))
    s.Require().NoError(err) // hook swallows the error

    balAfter := s.app.BankKeeper.GetBalance(s.ctx, owner, bondDenom)
    s.Require().Equal(balBefore.Amount, balAfter.Amount, "owner received zero coins")

    // Now fund the pool and try to claim again
    s.fundRewardsPool(sdkmath.NewInt(1_000_000_000), bondDenom)
    updated, _ := s.keeper.GetPositionState(s.ctx, pos.Id)
    bonus, err := s.keeper.ProcessEventsAndClaimBonus(s.ctx, &updated)
    s.Require().NoError(err)

    // BUG: bonus is zero because checkpoints already advanced and events consumed
    s.Require().True(bonus.IsZero(), "events consumed, checkpoints advanced, bonus permanently lost")
}
```
