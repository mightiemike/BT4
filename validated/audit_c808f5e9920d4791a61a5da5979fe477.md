Audit Report

## Title
Router-mediated swaps pass router address as `sender` to `SwapAllowlistExtension`, breaking per-user allowlist gating — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` from the pool's perspective. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the user EOA. The hook therefore checks whether the router is allowlisted rather than the actual economic actor, making per-user allowlist gating either fully bypassable or fully broken for router-mediated swaps.

## Finding Description

In `MetricOmmPool.swap()`, `_beforeSwap` is called with `msg.sender` as the `sender` argument:

```solidity
_beforeSwap(
  msg.sender,   // <-- always the immediate caller, i.e. the router
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` passes this value unchanged as the first argument to the extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`.

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router address. When `MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `IMetricOmmPoolActions(params.pool).swap(...)`, the pool's `msg.sender` is the router contract, not the originating user EOA.

There is no mechanism in the router or pool to forward the original user's identity to the extension. The `extensionData` field is user-controlled and not authenticated, so it cannot be used as a trust anchor without additional design.

## Impact Explanation

Two mutually exclusive broken invariants result:

**Bypass path**: If the pool admin allowlists the router address (the natural configuration to enable router-mediated swaps), then every user — including those never individually allowlisted — can bypass the per-user gate by routing through `MetricOmmSimpleRouter`. The allowlist becomes entirely ineffective for router-mediated swaps, allowing unauthorized users to swap on KYC-gated, institutional, or otherwise restricted pools.

**Denial path**: If the pool admin allowlists only individual user addresses (not the router), those allowlisted users cannot use the router at all — the hook sees the router address and reverts `NotAllowedToSwap()`. The two configurations are mutually exclusive.

The `SwapAllowlistExtension` is a production extension explicitly designed to gate access to curated pools. Either outcome breaks the core access-control functionality of the extension, constituting broken core pool functionality.

## Likelihood Explanation

Any pool that deploys `SwapAllowlistExtension` and also wants to support `MetricOmmSimpleRouter` faces this dilemma immediately upon configuration. Allowlisting the router is the obvious and predictable step an admin takes to enable router support, and doing so silently opens the pool to all users. This is a predictable misconfiguration path for any production pool using this extension. The `generate_scanned_questions.py` audit pivot explicitly identifies this exact scenario as a target: "Test direct swaps and router swaps on allowlisted pools and assert the hook cannot be bypassed by routing through an intermediate public contract."

## Recommendation

The `beforeSwap` hook should not rely solely on `sender` (the immediate caller) to identify the economic actor when routers are involved. Options:

1. Have `MetricOmmSimpleRouter` encode `msg.sender` (the originating user) into `extensionData` before calling the pool, and have `SwapAllowlistExtension` verify a pool-authenticated originator from `extensionData` rather than `sender`.
2. Document that `SwapAllowlistExtension` is incompatible with `MetricOmmSimpleRouter` and require direct pool interaction for allowlisted pools.
3. Redesign the extension to check a signed attestation or a separate on-chain registry that maps router → originator.

## Proof of Concept

**Bypass scenario:**
1. Deploy pool P with `SwapAllowlistExtension`; call `setAllowedToSwap(P, router, true)` (admin enables router support).
2. Non-allowlisted attacker calls `MetricOmmSimpleRouter.exactInputSingle({pool: P, ...})`.
3. Router calls `P.swap(recipient, ...)` — `msg.sender` to pool = router address.
4. Pool calls `SwapAllowlistExtension.beforeSwap(sender=router, ...)`.
5. Hook checks `allowedSwapper[P][router]` → `true` → passes.
6. Attacker completes swap on a pool they were never individually allowlisted for.

**Denial scenario:**
1. Deploy pool P with `SwapAllowlistExtension`; call `setAllowedToSwap(P, userA, true)` (router NOT allowlisted).
2. UserA calls `MetricOmmSimpleRouter.exactInputSingle({pool: P, ...})`.
3. Hook checks `allowedSwapper[P][router]` → `false` → reverts `NotAllowedToSwap()`.
4. UserA is blocked from using the router despite being individually allowlisted.

A Foundry integration test can reproduce both scenarios by deploying the full stack (`MetricOmmPool` + `SwapAllowlistExtension` + `MetricOmmSimpleRouter`), configuring the allowlist as described, and asserting the revert/pass behavior.
