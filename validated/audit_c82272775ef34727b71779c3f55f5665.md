Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates the direct `pool.swap()` caller (router address) instead of the originating EOA, enabling full allowlist bypass when the router is allowlisted — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of `pool.swap()`. When users swap through `MetricOmmSimpleRouter`, `sender` resolves to the router contract address, not the originating EOA. If a pool admin allowlists the router address (a natural operational step to enable router-mediated swaps for allowlisted users), every unprivileged user can bypass the per-user gate by routing through the router, nullifying the access control entirely.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to `_beforeSwap`.**

`MetricOmmPool.swap` at lines 230–240 calls:
```solidity
_beforeSwap(
    msg.sender,   // whoever called pool.swap()
    recipient,
    ...
);
```
`ExtensionCalling._beforeSwap` (lines 160–176) forwards this value verbatim as the `sender` argument in `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))` to every configured extension.

**Step 2 — `SwapAllowlistExtension.beforeSwap` checks that `sender` value.**

At line 37 of `SwapAllowlistExtension.sol`:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`.

**Step 3 — The router calls `pool.swap()` as itself, never forwarding the originating EOA.**

`MetricOmmSimpleRouter.exactInputSingle` (lines 71–80) stores the original `msg.sender` only in transient storage for payment purposes via `_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn)`, then calls `IMetricOmmPoolActions(params.pool).swap(...)` directly. The pool receives `msg.sender = router address`. The originating EOA is never passed to the pool or any extension. The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

**Result — identity mismatch and bypass.**

The allowlist is keyed `allowedSwapper[pool][sender]`. When a user goes through the router, `sender` = router address. Two broken outcomes:

| Scenario | Effect |
|---|---|
| Admin allowlists specific EOAs, not the router | Allowlisted users **cannot** swap through the router; they must call `pool.swap()` directly |
| Admin allowlists the router (to enable router UX for allowlisted users) | **Any** unprivileged user bypasses the per-user gate by calling `router.exactInputSingle()` |

The second scenario is the fund-impacting one. No existing guard prevents it: `_requireExpectedCallbackCaller` only validates that the callback caller is a factory-registered pool, which the pool is.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict swaps to a curated set of addresses (KYC'd counterparties, whitelisted market makers, protocol-internal addresses) loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter` once the router is allowlisted. Because the pool's pricing is oracle-driven, a restricted pool may carry liquidity at prices only safe when counterparties are trusted; opening it to arbitrary swappers breaks that assumption and can cause direct LP principal loss. This constitutes broken core pool functionality causing loss of funds, meeting Sherlock contest thresholds.

## Likelihood Explanation
Medium. The bypass is always latent in the code. It activates the moment a pool admin allowlists the router address — a natural operational step for any pool that wants to support the standard periphery UX for allowlisted users. No attacker action is required beyond calling the public router. The router is a deployed, factory-registered contract, so `_requireExpectedCallbackCaller` does not block this path.

## Recommendation
The extension must gate the **economic actor**, not the technical caller. Two viable fixes:

1. **Router-side**: Have `MetricOmmSimpleRouter` encode the original `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and verify it (requires a trusted encoding convention between router and extension).
2. **Extension-side**: Add an explicit `originator` field to the `beforeSwap` interface, populated by the pool from a router-supplied parameter, so the extension can check the true initiator.

Option 3 (documentation only) is insufficient as a security mitigation.

## Proof of Concept
```
Setup:
  pool deployed with SwapAllowlistExtension as beforeSwap hook
  allowedSwapper[pool][alice]  = true   // alice is the intended gated user
  allowedSwapper[pool][router] = true   // admin adds router to support alice's UX

Attack:
  attacker (not alice, not allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: attacker, ...})

Execution trace:
  router.exactInputSingle()
    _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, attacker_eoa, tokenIn)
    → pool.swap(recipient, ...)          // msg.sender = router address
      → _beforeSwap(router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓  (passes)
      → swap executes, attacker receives output tokens

Result:
  Attacker bypasses the per-user allowlist and swaps against restricted liquidity.
  alice's intended exclusive access is nullified.
```

Foundry test plan: deploy pool with `SwapAllowlistExtension`, set `allowedSwapper[pool][alice] = true` and `allowedSwapper[pool][router] = true`, call `router.exactInputSingle` from an unlisted EOA, assert the swap succeeds and the attacker receives output tokens.
