### Title
`CANONICAL_INTS` Flag Defined to Gate Integer Canonicality Is Never Checked in `int_atom`, Leaving Most Integer Operators Unenforced in Mempool Mode ã (`File: src/op_utils.rs`)

### Summary

`ClvmFlags::CANONICAL_INTS` is declared to require that all integers passed to operators use canonical representation (no unnecessary leading zeros). It is included in `MEMPOOL_MODE` and is checked inside `uint_atom`. However, `int_atom` ã the function used by the majority of integer-consuming operators ã never consults this flag. As a result, operators such as `op_gr`, `op_ash`, `op_multiply`, `op_div`, `op_divmod`, `op_mod`, and the BLS scalar operators silently accept non-canonical integers even when running in mempool mode, defeating the stated invariant of the flag.

### Finding Description

`ClvmFlags::CANONICAL_INTS` is defined in `src/chia_dialect.rs` with the explicit purpose of rejecting non-canonical integer encodings:

```rust
/// require integers passed to operators use canonical representation,
/// meaning no unnecessary leading zeros
const CANONICAL_INTS = 0x0001;
``` [1](#0-0) 

It is included unconditionally in `MEMPOOL_MODE`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [2](#0-1) 

The flag **is** enforced in `uint_atom`, which is used only for softfork cost and extension arguments:

```rust
if flags.contains(ClvmFlags::CANONICAL_INTS) {
    // strip potential zero
    if buf[0] == 0 {
        if buf.len() < 2 || (buf[1] & 0x80) == 0 {
            return Err(...);
        }
    }
}
``` [3](#0-2) 

However, `int_atom` ã the function used by the vast majority of integer-consuming operators ã performs no such check:

```rust
pub fn int_atom(a: &Allocator, args: NodePtr, op_name: &str) -> Result<(Number, usize)> {
    match a.sexp(args) {
        SExp::Atom => Ok((a.number(args), a.atom_len(args))),
        _ => Err(...)?,
    }
}
``` [4](#0-3) 

The operators that call `int_atom` and receive `flags: ClvmFlags` but never forward it to any canonicality check include:

- `op_gr` ã uses `_flags` (ignored), calls `int_atom`
- `op_ash` ã uses `_flags` (ignored), calls `int_atom` and `i32_atom`
- `op_multiply` ã uses `_flags` (ignored), calls `int_atom`
- `op_div` / `op_divmod` / `op_mod` ã check `DISABLE_OP` and `MALACHITE` but not `CANONICAL_INTS`, call `int_atom`
- `op_bls_g1_multiply` ã uses `_flags` (ignored), calls `int_atom` for the scalar [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

Additionally, `op_add` and `op_subtract` use `_flags` (ignored entirely) and call `number_from_u8` directly, also bypassing `CANONICAL_INTS`. [9](#0-8) [10](#0-9) 

The wiring error is structural: `CANONICAL_INTS` is connected only to `uint_atom` (softfork argument parsing) and is never wired into `int_atom`, `i32_atom`, or `number_from_u8`, which handle all regular operator integer arguments.

### Impact Explanation

In mempool mode, an attacker can submit CLVM programs that pass non-canonical integers (e.g., `0x0001` instead of `0x01`, or `0x00ff` instead of `0xff`) to operators such as `op_gr`, `op_ash`, `op_multiply`, `op_div`, `op_divmod`, `op_mod`, and BLS scalar operators. These programs are accepted by the mempool despite `CANONICAL_INTS` being set, violating the invariant the flag is meant to enforce.

The concrete broken invariant: the mempool is supposed to be strictly more restrictive than consensus for integer encoding, but for all operators using `int_atom`, it is equally permissive. This enables:

1. **Mempool policy bypass**: Transactions with non-canonical integer encodings pass mempool validation that `CANONICAL_INTS` was designed to block.
2. **Transaction malleability surface**: Multiple distinct byte-level encodings of the same integer value are accepted for the same semantic program, creating a malleability vector at the mempool layer.

The corrupted result is the mempool's acceptance decision: a program that should be rejected (non-canonical integer in `op_gr`, `op_ash`, `op_multiply`, etc.) is accepted and propagated.

### Likelihood Explanation

The entry path is direct and requires only attacker-controlled CLVM bytes. Any caller of `run_program` with `MEMPOOL_MODE` flags (e.g., the Chia full node mempool) is affected. Crafting a CLVM program with a non-canonical integer argument (a leading zero byte before a non-negative integer) is trivial. No special privileges or access are required.

### Recommendation

Add a canonicality check inside `int_atom` (and `i32_atom`) that consults the `flags` parameter, mirroring the check already present in `uint_atom`. Alternatively, thread `flags` through to `int_atom` and reject atoms with unnecessary leading zeros when `CANONICAL_INTS` is set. All operator functions that currently declare `_flags: ClvmFlags` (ignoring the parameter) and call `int_atom` should be updated to pass `flags` through.

    if flags.contains(ClvmFlags::MALACHITE) {
        return op_div_malachite(a, input, max_cost, flags);
    }
    let [v0, v1] = get_args::<2>(a, input, "/")?;
    let (a0, a0_len) = int_atom(a, v0, "/")?;
    let (a1, a1_len) = int_atom(a, v1, "/")?;
    if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 2048 {
        return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
    }
    if a0_len > 256 || a1_len > 1024 {
        return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
    }
    let cost = DIV_BASE_COST + ((a0_len + a1_len) as Cost) * DIV_COST_PER_BYTE;
    check_cost(cost, max_cost)?;
    if a1.sign() == Sign::NoSign {
        return Err(EvalErr::DivisionByZero(input));
    }
    let q = a0.div_floor(&a1);
    let q = a.new_number(q)?;
    Ok(malloc_cost(a, cost, q))
}

            GR_BASE_COST + (len_for_value(lhs) + len_for_value(rhs)) as Cost * GR_COST_PER_BYTE;

    Ok(Reduction(cost, if v0 > v1 { a.one() } else { a.nil() }))
