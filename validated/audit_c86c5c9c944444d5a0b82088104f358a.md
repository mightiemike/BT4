### Title
`submit_blocks` Paused but `verify_transaction_inclusion` Remains Active — Stale Chain State Accepted as Valid Proof - (File: `contract/src/lib.rs`)

### Summary
The `near-plugins` `#[pause]` attribute is applied independently to `submit_blocks`, `verify_transaction_inclusion`, and `verify_transaction_inclusion_v2`. A `PauseManager` can pause `submit_blocks` (freezing the chain tip) while leaving the two verification functions active. Any downstream contract that calls `verify_transaction_inclusion` during this window receives results computed against a frozen, potentially stale chain state — including for transactions that have since been reorganized out of the real Bitcoin chain.

### Finding Description

The contract uses `near-plugins`' per-feature pause mechanism. Each `#[pause]`-decorated function is independently pausable by its feature key (the function name). The three relevant functions are:

- `submit_blocks` — `#[pause]` at line 167: the only path to advance `mainchain_tip_blockhash` and update `mainchain_height_to_header` / `mainchain_header_to_height`.
- `verify_transaction_inclusion` — `#[pause]` at line 287: reads `mainchain_tip_blockhash` to compute the confirmation count and reads `mainchain_header_to_height` to assert the block is on the main chain.
- `verify_transaction_inclusion_v2` — `#[pause]` at line 346: delegates to `verify_transaction_inclusion` after a coinbase proof check. [1](#0-0) [2](#0-1) [3](#0-2) 

When `submit_blocks` is paused, `mainchain_tip_blockhash` is frozen. The confirmation check inside `verify_transaction_inclusion` is:

```rust
(heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
    >= args.confirmations
``` [4](#0-3) 

With a frozen tip, this value never increases. More critically, if the real Bitcoin chain reorganizes during the pause window, the contract's `mainchain_header_to_height` index still maps the pre-reorg block hash to its old height. `verify_transaction_inclusion` will return `true` for a transaction whose block has been orphaned on the real chain, because the contract has no way to learn about the reorg while `submit_blocks` is paused.

There is no coupling enforced between pausing `submit_blocks` and pausing the verification functions. The `Role::PauseManager` can issue `pa_pause_feature("submit_blocks")` without touching `verify_transaction_inclusion` or `verify_transaction_inclusion_v2`. [5](#0-4) 

Additionally, `run_mainchain_gc` carries `#[pause(except(roles(Role::UnrestrictedRunGC)))]`, meaning GC can still prune mainchain blocks while `submit_blocks` is paused. If GC removes a block whose transaction a downstream contract later queries, `verify_transaction_inclusion` panics with `"block does not belong to the current main chain"` — a secondary desynchronization path. [6](#0-5) 

### Impact Explanation

Any downstream NEAR contract that gates a financial action (token release, bridge unlock, cross-chain settlement) on a `true` result from `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` will operate on a frozen, potentially incorrect view of the Bitcoin chain. A transaction that was confirmed before the pause but subsequently reorganized out of the real Bitcoin chain will still pass the SPV check. This breaks the core security guarantee of the light client: that a `true` result means the transaction is currently on the canonical Bitcoin chain with the requested number of confirmations.

### Likelihood Explanation

The `PauseManager` role is expected to be used during incidents or maintenance. It is operationally natural to pause only `submit_blocks` (to stop header ingestion) while leaving verification active (to avoid disrupting downstream consumers). Bitcoin reorgs of 1–3 blocks occur several times per year; a pause lasting even a few hours during a period of elevated reorg activity is sufficient to create the divergence. The combination of a routine admin action and a low-frequency but real chain event makes this a realistic operational failure mode.

### Recommendation

Enforce that pausing `submit_blocks` atomically pauses `verify_transaction_inclusion` and `verify_transaction_inclusion_v2`. Options:

1. Use a single shared pause feature key for all three functions so they cannot be paused independently.
2. Add an explicit check inside both verification functions that aborts if the `submit_blocks` feature is currently paused:
   ```rust
   require!(!self.pa_is_paused("submit_blocks"), "Chain updates are paused; verification disabled");
   ```
3. Document and enforce in the `PauseManager` runbook that pausing `submit_blocks` must always be accompanied by pausing both verification functions.

### Proof of Concept

1. `PauseManager` calls `pa_pause_feature("submit_blocks")` — only `submit_blocks` is paused; both `verify_transaction_inclusion` functions remain callable.
2. The real Bitcoin chain produces a 2-block reorg, orphaning block `B` (which contains transaction `T`). The contract cannot learn of this because `submit_blocks` is paused.
3. A downstream bridge contract calls `verify_transaction_inclusion_v2` with `tx_id = T`, `tx_block_blockhash = B`, `confirmations = 6`.
4. The contract finds `B` in `mainchain_header_to_height` (it was on the main chain before the pause), computes `tip_height - B_height + 1 >= 6` (passes, tip is frozen), and returns `true`.
5. The bridge contract releases funds to the claimant. The transaction `T` does not exist on the canonical Bitcoin chain. [7](#0-6) [8](#0-7)
