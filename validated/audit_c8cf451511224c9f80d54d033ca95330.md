Based on my investigation of `app/app.go` and the `chainmain` module types, this is a real vulnerability.

### Title
Tokens Sent to `authtypes.NewModuleAddress("chainmain")` Are Permanently Locked Due to Missing `maccPerms` Entry — (`app/app.go`)

### Summary

The `chainmain` module name is absent from `maccPerms`, so `authtypes.NewModuleAddress("chainmain")` is never registered as a module account and never appears in `BlockedAddrs()`. The bank module therefore permits `MsgSend` to that address. Because the address has no corresponding private key and no keeper-controlled withdrawal path, any tokens sent there are permanently and irrecoverably locked.

### Finding Description

`maccPerms` in `app/app.go` lists every module account the chain recognises: [1](#0-0) 

`chainmaintypes.ModuleName` (`"chainmain"`) has no entry there. The Cosmos SDK's `BlockedAddrs()` is built exclusively from the keys of `maccPerms`, so `authtypes.NewModuleAddress("chainmain")` is absent from the blocked-address set.

The custom `moduleAccsAllowedToReceiveExternalFunds` map only covers `tieredrewardstypes.RewardsPoolName` and `tieredrewardstypes.ModuleName`; `chainmain` is not listed there either: [2](#0-1) 

`chainmaintypes.ModuleName` is defined as the string `"chainmain"`: [3](#0-2) 

The `chainmain` module exposes no bank-related keeper methods and has no mechanism to move funds out of its derived address: [4](#0-3) 

### Impact Explanation

Any unprivileged account can call `bank.MsgSend` with `ToAddress = authtypes.NewModuleAddress("chainmain").String()`. The bank keeper's `BlockedAddr` check passes (address is not blocked), the send succeeds, and the tokens are credited to an address for which:

- No private key exists (module addresses are hash-derived).
- No keeper holds a reference or can call `BankKeeper.SendFromModuleToAccount`.
- No governance or upgrade path recovers the balance without a hard fork.

The funds are permanently locked. This matches the Critical scope: *unprivileged on-chain action causes permanent lock of user funds*.

### Likelihood Explanation

The attack requires only a standard `MsgSend` transaction — no special permissions, no governance vote, no validator collusion. Any user who mistakenly (or maliciously, e.g. via a phishing UI) targets this address loses their tokens with no recourse.

### Recommendation

Add `chainmaintypes.ModuleName` to `maccPerms` with a `nil` permission set (receive-only, no minting/burning):

```go
// app/app.go  maccPerms
chainmaintypes.ModuleName: nil,
```

This registers the address as a module account in the `AccountKeeper` and causes `BlockedAddrs()` to include it, so the bank module will reject any `MsgSend` targeting it.

### Proof of Concept

```go
// Pseudocode unit test
chainmainAddr := authtypes.NewModuleAddress(chainmaintypes.ModuleName)

// 1. Confirm address is NOT blocked
blockedAddrs := app.BlockedAddrs()
assert.False(t, blockedAddrs[chainmainAddr.String()])  // passes → not blocked

// 2. Send tokens via MsgSend
msg := banktypes.NewMsgSend(userAddr, chainmainAddr, sdk.NewCoins(sdk.NewInt64Coin("basecro", 1_000_000)))
_, err := bankMsgServer.Send(ctx, msg)
assert.NoError(t, err)  // send succeeds

// 3. Confirm tokens are locked — no withdrawal path exists
balance := bankKeeper.GetBalance(ctx, chainmainAddr, "basecro")
assert.Equal(t, int64(1_000_000), balance.Amount.Int64())  // tokens are there, irrecoverable
``` [1](#0-0)
