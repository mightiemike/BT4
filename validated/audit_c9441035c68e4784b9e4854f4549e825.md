### Title
Unbounded Slow Mode Queue Allows Attacker to Delay Censorship-Resistant Transactions - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

`submitSlowModeTransactionImpl` in `EndpointTx.sol` imposes no per-address limit on slow mode queue submissions. An unprivileged attacker can flood the `slowModeTxs` queue with cheap transactions (costing `SLOW_MODE_FEE = $1` each), pushing legitimate users' slow mode transactions arbitrarily far back in the FIFO queue and indefinitely delaying their execution.

---

### Finding Description

The slow mode queue is Nado's censorship-resistance mechanism: when the sequencer refuses to include a user's transaction, the user submits it via `submitSlowModeTransaction`, waits 3 days, and then calls `executeSlowModeTransaction()` to force execution on-chain.

The queue is stored as a mapping indexed by a monotonically increasing counter: [1](#0-0) 

Submissions are appended unconditionally in `submitSlowModeTransactionImpl`: [2](#0-1) 

The only cost imposed on a non-admin caller is the flat `SLOW_MODE_FEE`: [3](#0-2) 

`SLOW_MODE_FEE` is a fixed $1 in quote-token units: [4](#0-3) 

There is **no per-address submission cap** anywhere in `submitSlowModeTransactionImpl`. An attacker with a registered subaccount can call `submitSlowModeTransaction` with valid `WithdrawCollateral` or `LinkSigner` payloads thousands of times, paying $1 per entry, and fill the queue ahead of any victim.

Queue execution is strictly FIFO: `_executeSlowModeTransaction` always pops `slowModeTxs[txUpTo]` and increments `txUpTo`: [5](#0-4) 

The sequencer can drain the queue early via `ExecuteSlowMode` transactions, but the slow mode queue is specifically the fallback for when the **sequencer is the adversary** (censorship scenario). In that scenario the sequencer will not help drain the attacker's spam, so the victim must wait for every attacker entry to be processed one-by-one via `executeSlowModeTransaction()`.

---

### Impact Explanation

A victim whose sequencer-submitted transactions are being censored submits a `WithdrawCollateral` slow mode transaction to reclaim their collateral. If an attacker has pre-filled the queue with N spam entries, the victim's transaction sits at position N. Every one of those N entries must be individually popped and executed before the victim's transaction is reached. With N = 1 000, the victim must either wait for 1 000 sequential `executeSlowModeTransaction()` calls (each costing gas) or remain locked out of their collateral indefinitely. This directly undermines the censorship-resistance guarantee and delays access to user funds.

---

### Likelihood Explanation

The attack requires only a registered subaccount and quote-token capital. At $1 per slot, flooding 1 000 positions costs $1 000 — a realistic budget for a motivated attacker targeting a specific victim. The attacker can also recoup most of the capital by submitting `WithdrawCollateral` spam that succeeds on execution (withdrawing their own funds), making the net cost only gas. No privileged access, no sequencer compromise, and no governance action is required.

---

### Recommendation

1. **Per-address submission cap**: Track `slowModeSubmissions[address]` and enforce a maximum (e.g., 10 pending entries per address) in `submitSlowModeTransactionImpl`.
2. **Dynamic fee**: Scale `SLOW_MODE_FEE` with the current queue depth so that flooding becomes exponentially expensive.
3. **Batch execution**: Allow `executeSlowModeTransaction` to accept a `count` parameter so a victim can clear a backlog in fewer transactions.

---

### Proof of Concept

```
1. Attacker holds a registered subaccount with sufficient quote balance.
2. Attacker calls Endpoint.submitSlowModeTransaction(WithdrawCollateral{...}) 
   1 000 times, paying $1 SLOW_MODE_FEE each time.
   → slowModeConfig.txCount advances by 1 000.
3. Victim (being censored by sequencer) calls 
   Endpoint.submitSlowModeTransaction(WithdrawCollateral{...}).
   → Victim's entry lands at queue index txCount = 1 000.
4. After 3 days, victim calls executeSlowModeTransaction().
   → Only the attacker's entry at txUpTo = 0 is processed.
5. Victim must call executeSlowModeTransaction() 1 000 more times 
   (or pay others to do so) before their own withdrawal executes.
   → Collateral access is delayed by 1 000 sequential on-chain calls.
``` [6](#0-5) [7](#0-6)

    function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
    }

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }

    function executeSlowModeTransaction() external {
        SlowModeConfig memory _slowModeConfig = slowModeConfig;
        _executeSlowModeTransaction(_slowModeConfig, false);
        nSubmissions += 1;
        slowModeConfig = _slowModeConfig;
    }
