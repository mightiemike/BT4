Audit Report

## Title
`DepositAllowlistExtension` Validates Position `owner` Instead of Actual Payer `sender`, Allowing Any Address to Bypass the Deposit Guard - (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards its first parameter (`sender`, the actual token payer) and validates only `owner` (the position beneficiary). Because `MetricOmmPool.addLiquidity` permits `msg.sender ≠ owner` with no constraint, any address not on the allowlist can bypass the guard by supplying an allowed `owner` address, completely nullifying the pool admin's LP access control.

## Finding Description
`MetricOmmPool.addLiquidity` passes two distinct actors to the extension hook:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`msg.sender` is the operator who triggers the call and pays tokens; `owner` is the address whose position is credited. No constraint enforces `msg.sender == owner` in `addLiquidity` (contrast with `removeLiquidity` at line 206, which explicitly requires `msg.sender == owner`).

`ExtensionCalling._beforeAddLiquidity` correctly forwards both actors:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

`DepositAllowlistExtension.beforeAddLiquidity` then receives `sender` as its first argument but discards it (unnamed `address,`), checking only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The contract's NatSpec states it "Gates `addLiquidity` by depositor address", and its admin API (`setAllowedToDeposit`, `allowedDepositor`) names the subject "depositor" — all pointing to the payer as the intended subject. The implementation checks the wrong actor.

The existing test suite confirms the bug: every test passes `address(0)` as the first argument (sender) and `depositor` as the second (owner), and the allowlist check is performed against `owner` only, never against the actual caller.

## Impact Explanation
Any address not on the allowlist can call `pool.addLiquidity(allowedOwner, salt, deltas, ...)`, pay tokens through the modify-liquidity callback, and have the position credited to `allowedOwner`. The `DepositAllowlistExtension` guard passes because it only validates `allowedOwner`, which is on the allowlist. Every pool that relies solely on this extension for LP access control is open to any depositor who can identify one allowed owner address — trivially discoverable from on-chain `LiquidityAdded` events or `AllowedToDepositSet` events. This is an admin-boundary break: an unprivileged actor bypasses a pool-admin-configured access control gate, with the wrong value being `allowedDepositor[pool][owner]` evaluated instead of `allowedDepositor[pool][sender]`.

## Likelihood Explanation
Likelihood is high. The operator pattern (`msg.sender ≠ owner`) is a first-class feature of the pool — `addLiquidity` imposes no equality check between caller and owner. Allowed owner addresses are publicly visible from `AllowedToDepositSet` events emitted by `setAllowedToDeposit`. No special privilege, flash loan, or oracle manipulation is required — a standard `addLiquidity` call suffices, and the attack is repeatable indefinitely.

## Recommendation
Replace the discarded first parameter with a named `sender` and validate it instead of (or in addition to) `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to restrict both the payer and the position owner, both should be checked independently.

## Proof of Concept
1. Pool admin deploys a pool with `DepositAllowlistExtension` configured on `beforeAddLiquidity`.
2. Admin calls `setAllowedToDeposit(pool, alice, true)`. Bob is **not** added.
3. Bob calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(Bob, alice, ...)` → extension receives `sender=Bob, owner=alice`.
5. Extension discards `Bob`, checks `allowedDepositor[pool][alice]` → `true` → no revert.
6. Bob's tokens are pulled via the modify-liquidity callback; Alice's position is credited.
7. Bob has successfully deposited despite being absent from the allowlist.

Foundry test plan: fork the pool setup, call `setAllowedToDeposit(pool, alice, true)`, then call `addLiquidity` from Bob with `owner=alice` and assert no revert occurs and Bob's tokens are transferred.
