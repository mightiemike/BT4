Audit Report

## Title
Pool Admin Can Reduce `OracleValueStopLossExtension` Timelock to Zero and Instantly Disable Drawdown Protection — (File: `metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`OracleValueStopLossExtension` uses a per-pool `timelock` to delay parameter changes so LPs can react. However, once the admin reduces `timelock` to `0` (after waiting out the original timelock), `_afterTimelock` returns exactly `block.timestamp`, and `_requireElapsed`'s strict `<` check passes immediately. This allows the admin to propose and execute `drawdownE6 = 0` in the same block, silently disabling the stop-loss guard with zero LP warning time.

## Finding Description
`_afterTimelock` computes the execution deadline using the **current** stored timelock value:

```solidity
// L297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

When `timelock == 0`, this returns `block.timestamp`. `_requireElapsed` uses a strict `<`:

```solidity
// L301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(executeAfter, block.timestamp);
}
```

`block.timestamp < block.timestamp` is `false`, so the revert is never triggered. Both `proposeOracleStopLossDrawdown` (L106) and `executeOracleStopLossDrawdown` (L115) use these helpers, meaning propose and execute can be called in the same block once `timelock == 0`.

`_validateDrawdown` imposes no minimum floor — `drawdownE6 = 0` passes:

```solidity
// L305-307
function _validateDrawdown(uint256 drawdownE6) private pure {
    if (drawdownE6 > E6) revert OracleStopLossDrawdownTooLarge(drawdownE6);
}
```

Once `drawdownE6` is set to `0`, `_afterSwapOracleStopLoss` short-circuits immediately:

```solidity
// L215-217
PoolStopLossConfig memory cfg = oracleStopLossConfig[pool_];
uint256 drawdown = cfg.drawdownE6;
if (drawdown == 0) return;
```

Attack path:
1. Admin deploys pool with `timelock = T`, `drawdownE6 = D`. LPs deposit trusting the stop-loss.
2. Admin calls `proposeOracleStopLossTimelock(pool, 0)`. `executeAfter = now + T`.
3. After `T` seconds, admin calls `executeOracleStopLossTimelock(pool)`. `timelock` is now `0`.
4. Admin calls `proposeOracleStopLossDrawdown(pool, 0)`. `executeAfter = block.timestamp`.
5. Admin immediately calls `executeOracleStopLossDrawdown(pool)` in the same block/tx. `_requireElapsed(block.timestamp)` passes.
6. `drawdownE6 = 0`. All subsequent swaps bypass the stop-loss entirely.

Steps 4–6 are atomic (single transaction), giving LPs zero blocks to observe and exit.

## Impact Explanation
Once `drawdownE6 == 0`, `_afterSwapOracleStopLoss` returns immediately without checking any watermarks. Swaps that drain LP value in any bin are no longer blocked by `OracleStopLossTriggered`. LPs who deposited relying on the stop-loss invariant ("value per share at oracle marks cannot fall faster than drawdown") lose that protection with no on-chain warning and no time to withdraw. This is a direct loss of LP principal protection — a medium-to-high impact depending on pool TVL. This falls under the allowed impact category: **Admin-boundary break: pool admin bypasses timelocks**.

## Likelihood Explanation
The attack requires the pool admin to be malicious and to wait out the initial timelock `T`. However, `T` is chosen by the pool admin at pool creation and can be set to a minimal value (e.g., 1 hour). After `T` elapses, the remaining steps are atomic. LPs monitoring `OracleStopLossTimelockProposed` events have exactly `T` seconds to react to the timelock reduction — the same window they were already given — but receive **zero** additional warning before the drawdown is zeroed. The NatSpec at L16 explicitly states the timelock exists so "LPs can react," confirming this is a user-protection mechanism the admin can fully circumvent.

## Recommendation
Enforce a minimum floor on the timelock value so it can never be reduced to zero:

```solidity
uint32 public constant MIN_TIMELOCK = 1 days;

function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    require(newTimelock >= MIN_TIMELOCK, "timelock below minimum");
    ...
}
```

Alternatively, compute `executeAfter` for a timelock-reduction proposal using `max(currentTimelock, newTimelock)` so reducing the timelock always takes at least as long as the new value would impose. Additionally, enforce a minimum floor on `drawdownE6` (e.g., `> 0`) in `_validateDrawdown` to prevent the stop-loss from being fully disabled.

## Proof of Concept

```solidity
// 1. Deploy pool with timelock = 1 hours, drawdownE6 = 50_000 (5%)
// 2. LPs deposit

// 3. Admin reduces timelock to 0 (waits 1 hour)
vm.prank(admin);
ext.proposeOracleStopLossTimelock(pool, 0);
vm.warp(block.timestamp + 1 hours);
vm.prank(admin);
ext.executeOracleStopLossTimelock(pool);
// oracleStopLossConfig[pool].timelock == 0

// 4. Admin disables drawdown instantly (same block)
vm.prank(admin);
ext.proposeOracleStopLossDrawdown(pool, 0);
// _afterTimelock returns block.timestamp; pendingDrawdownExecuteAfter = block.timestamp

vm.prank(admin);
ext.executeOracleStopLossDrawdown(pool);
// _requireElapsed(block.timestamp): block.timestamp < block.timestamp == false → no revert

// 5. Verify guard is disabled
assertEq(ext.oracleStopLossConfig(pool).drawdownE6, 0);
// afterSwap now returns immediately at L217 — LP funds unprotected
```
