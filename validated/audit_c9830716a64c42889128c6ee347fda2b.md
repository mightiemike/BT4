### Title
Residual Token Allowance in `DirectDepositV1.creditDeposit()` Causes Permanent DoS for USDT-like Tokens — (File: `core/contracts/DirectDepositV1.sol`)

### Summary
`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without first resetting the allowance to zero. When `Endpoint.depositCollateralWithReferral()` silently returns early (deposit amount below minimum), the allowance is never consumed. On the next invocation of `creditDeposit()`, the `approve` call on a USDT-like token reverts because the existing allowance is non-zero, permanently bricking the deposit path for that DDA and locking user funds.

---

### Finding Description

`DirectDepositV1.creditDeposit()` iterates over all spot product tokens and, for each non-zero balance, issues an approval then calls `endpoint.depositCollateralWithReferral`:

```solidity
// core/contracts/DirectDepositV1.sol lines 91-99
if (balance != 0) {
    token.approve(address(endpoint), balance);          // ← no prior reset to 0
    endpoint.depositCollateralWithReferral(
        subaccount,
        productId,
        uint128(balance),
        "-1"
    );
}
```

`Endpoint.depositCollateralWithReferral()` contains a silent early-return path:

```solidity
// core/contracts/Endpoint.sol lines 137-142
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked when there are
    // multiple assets awaiting credit but one of them is below the minimum deposit amount.
    // we can just skip the deposit and continue with the next asset.
    return;
}
handleDepositTransfer(...);   // ← never reached; allowance never consumed
```

When `isValidDepositAmount` returns `false` (e.g., a small USDT dust amount below `MIN_DEPOSIT_AMOUNT`), `handleDepositTransfer` is never called, so the allowance granted to `endpoint` remains at `balance`. On the next call to `creditDeposit()` — after more USDT has been sent to the DDA — the contract attempts `token.approve(address(endpoint), newBalance)` on top of a non-zero existing allowance. USDT (and any token with approval race-condition protection) reverts this call, permanently blocking `creditDeposit()` for that token.

There is no allowance-reset logic anywhere in `creditDeposit()` after the `depositCollateralWithReferral` call, unlike the pattern correctly applied in `ContractOwner.wrapVaultAsset()` (lines 530–531) which does `approve(tokenAddr, 0)` before `approve(tokenAddr, assetBalance)`.

---

### Impact Explanation

**Medium.** Once the residual allowance is set, every subsequent call to `creditDeposit()` for that token reverts. Funds sent to the DDA in USDT (or any approval-race-protected token) cannot be credited to the subaccount through the normal deposit flow. The funds are locked in the DDA contract until the multisig owner manually calls `ContractOwner.withdrawFromDirectDepositV1()` to recover them — a privileged recovery path that is not part of the normal user flow. This constitutes a limited, temporary fund lock with a broken deposit invariant.

---

### Likelihood Explanation

**Medium.** The trigger requires only that a USDT-like token is supported as a spot product and that a deposit below the minimum threshold is attempted first (e.g., dust sent to the DDA). This is a realistic operational scenario. `creditDeposit()` has no access control and is callable by any external account, meaning the residual-allowance state can be reached without any privileged action.

---

### Recommendation

Reset the allowance to zero before setting a new non-zero allowance, and also reset it to zero after the `depositCollateralWithReferral` call to clear any unconsumed allowance:

```solidity
if (balance != 0) {
    token.approve(address(endpoint), 0);
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
    token.approve(address(endpoint), 0);  // clear residual if deposit was skipped
}
```

---

### Proof of Concept

1. A USDT-like token (non-zero allowance revert protection) is registered as a spot product in `SpotEngine`.
2. A user sends 1 wei of USDT to their DDA address.
3. Anyone calls `DirectDepositV1.creditDeposit()`.
   - Line 92: `token.approve(address(endpoint), 1)` — allowance set to 1.
   - Line 93: `endpoint.depositCollateralWithReferral(...)` is called.
   - Inside `Endpoint.depositCollateralWithReferral`, `isValidDepositAmount` returns `false` (1 wei < `MIN_DEPOSIT_AMOUNT`).
   - The function returns early at line 141; `handleDepositTransfer` is never called.
   - Allowance of 1 remains on `endpoint`.
4. User sends 100 USDT to the DDA.
5. Anyone calls `DirectDepositV1.creditDeposit()` again.
   - Line 92: `token.approve(address(endpoint), 100e6)` — USDT reverts because existing allowance (1) ≠ 0 and ≠ `UINT_MAX`.
   - The entire `creditDeposit()` call reverts.
6. The 100 USDT is permanently stuck in the DDA until the multisig manually recovers it via `ContractOwner.withdrawFromDirectDepositV1()`.
