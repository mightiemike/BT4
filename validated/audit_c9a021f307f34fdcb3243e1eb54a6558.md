Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. Any admin who allowlists the router to enable legitimate router-mediated swaps simultaneously opens the gate to every non-allowlisted user, completely nullifying the allowlist policy for the primary public entrypoint.

## Finding Description

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [2](#0-1) 

When the call originates from `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly without forwarding the original user's address anywhere in the call: [3](#0-2) 

The router stores the original `msg.sender` only in transient storage for the payment callback (`_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn)`), but this value is never passed to the pool or extension. The extension therefore evaluates `allowedSwapper[pool][router]` — the router's identity — rather than the actual trader's identity.

The admin faces an inescapable dilemma: not allowlisting the router blocks all legitimate router users; allowlisting the router grants every non-allowlisted user a bypass path. There is no existing guard that recovers the original caller's identity inside the extension.

## Impact Explanation

Any unprivileged user not on the allowlist can call `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) targeting an allowlist-gated pool. If the router is allowlisted — the only operational way to let legitimate users use the router — the extension approves the call based on the router's identity and the swap executes. The allowlist policy is completely nullified for router-mediated paths. This constitutes a broken core pool functionality (allowlist access control) with direct fund-impacting consequences for regulatory-gated, KYC, or counterparty-restricted pools.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the canonical, documented public entrypoint for swaps. Any operator deploying a pool with `SwapAllowlistExtension` who wants allowlisted users to access the router must allowlist the router address — this is the expected operational pattern. The bypass condition is therefore highly likely to be present in any production deployment that uses both the allowlist extension and the router.

## Recommendation

Pass the original caller's identity through the extension data or as a dedicated parameter so the extension can gate the economic actor rather than the intermediary. Concrete options:

1. Have the router encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check that value when `sender` is a known router address.
2. Add an explicit `originator` parameter to `pool.swap()` that the router populates with its `msg.sender`, and have extensions gate on `originator` instead of `sender`.

## Proof of Concept

```solidity
// Setup
SwapAllowlistExtension ext = new SwapAllowlistExtension(factory);
ext.setAllowedToSwap(pool, user1, true);           // only user1 allowed directly
ext.setAllowedToSwap(pool, address(router), true); // router allowlisted so user1 can use it

// Exploit: attacker (not on allowlist) routes through the router
vm.prank(attacker);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: token0,
    recipient: attacker,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// pool.swap() is called with msg.sender = router
// _beforeSwap(sender=router, ...) is forwarded to extension
// extension checks allowedSwapper[pool][router] == true → swap succeeds
// attacker's identity is never checked
