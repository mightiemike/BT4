### Title
`ValidResourceBounds` Protobuf Deserialization Silently Downgrades `AllResources` to `L1Gas` When `l2_gas` and `l1_data_gas` Are Zero, Producing a Different Transaction Hash - (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion silently reclassifies a V3 transaction's `AllResources` resource-bounds variant as the pre-0.13.3 `L1Gas` variant whenever both `l2_gas` and `l1_data_gas` are zero. Because `get_tip_resource_bounds_hash` includes the L1DataGas element in the Poseidon preimage for `AllResources` but omits it for `L1Gas`, the transaction hash computed at submission time diverges from the hash recomputed after a protobuf round-trip. Any V3 transaction with non-zero `l1_gas` but zero `l2_gas` and `l1_data_gas` triggers this divergence.

### Finding Description

**Step 1 – Submission and hash calculation.**

`RpcDeclareTransactionV3` and `RpcInvokeTransactionV3` carry `resource_bounds: AllResourceBounds` (never `ValidResourceBounds`). When the gateway converts an incoming RPC transaction to `InternalRpcTransactionWithoutTxHash`, the hash is computed via `InternalRpcTransactionWithoutTxHash::calculate_transaction_hash`, which calls `get_declare_transaction_v3_hash` / `get_invoke_transaction_v3_hash`. Both delegate to `get_tip_resource_bounds_hash` with `ValidResourceBounds::AllResources(self.resource_bounds)`.

For `AllResources`, `get_tip_resource_bounds_hash` builds a four-element Poseidon preimage:

```
[tip, l1_packed, l2_packed, l1_data_packed]
```

Even when `l2_gas` and `l1_data_gas` are zero, `get_concat_resource` embeds the 7-byte resource-name tag into the felt, so `l2_packed` and `l1_data_packed` are **non-zero**:

```rust
// crates/starknet_api/src/transaction_hash.rs  lines 203-210
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
```

**Step 2 – Protobuf serialization.**

`From<ValidResourceBounds> for protobuf::ResourceBounds` always emits all three fields (including a zero `l1_data_gas` for the `L1Gas` variant):

```rust
// crates/apollo_protobuf/src/converters/transaction.rs  lines 471-489
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),
    l1_data_gas: Some(ResourceBounds::default().into()),  // zero
},
```

**Step 3 – Protobuf deserialization (the broken fallback).**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies a version-detection heuristic:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs  lines 417-436
let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // zero when absent
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong variant for a V3 tx
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

A V3 transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is serialized with zero `l2_gas` and `l1_data_gas`. On deserialization the condition `l1_data_gas.is_zero() && l2_gas.is_zero()` is true, so the result is `ValidResourceBounds::L1Gas(X)` — the pre-0.13.3 variant.

**Step 4 – Hash divergence.**

For `L1Gas`, `get_tip_resource_bounds_hash` builds only a three-element preimage:

```
[tip, l1_packed, l2_packed]
```

The `l1_data_packed` element is absent. Because `l1_data_packed` is non-zero (resource-name tag bits are always set), the two Poseidon hashes are **different**:

```
H_submit  = poseidon([tip, l1_packed, l2_packed, l1_data_packed])   // AllResources path
H_resync  = poseidon([tip, l1_packed, l2_packed])                   // L1Gas path
H_submit ≠ H_resync
```

**Step 5 – Trigger is unprivileged.**

The gateway's stateless validator accepts a transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` as long as `max_possible_fee(Tip::ZERO) > 0`, which holds whenever `l1_gas.max_amount * l1_gas.max_price_per_unit > 0`. No special privilege is required.

### Impact Explanation

Any node that receives the block via P2P sync and reconstructs the transaction through `DeclareTransactionV3Common` (which uses `ValidResourceBounds`) will compute `H_resync ≠ H_submit`. When `validate_transaction_hash` is called against the stored block hash, the mismatch causes the block to be rejected or the wrong hash to be committed. This matches:

> **High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

In the worst case, if the syncing node accepts the transaction with the recomputed (wrong) hash, the committed state contains an incorrect transaction hash — a Critical wrong-state outcome.

### Likelihood Explanation

The trigger requires only a valid V3 transaction with zero `l2_gas` and `l1_data_gas`. The gateway's resource-bounds check passes as long as `l1_gas` is non-zero. Any user can craft such a transaction. The protobuf round-trip is exercised on every P2P sync of a block containing such a transaction.

### Recommendation

Replace the zero-value heuristic in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` with an explicit version tag carried in the protobuf message, or always deserialize V3 transactions as `AllResources` regardless of whether `l2_gas` and `l1_data_gas` are zero. The TODO comment already acknowledges this:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
```

Until 0.13.2 support is dropped, the deserialization should at minimum preserve `AllResources` when the source transaction version is V3, rather than silently downgrading to `L1Gas`.

### Proof of Concept

1. Craft a V3 declare transaction with `AllResourceBounds { l1_gas: ResourceBounds { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: ResourceBounds::default(), l1_data_gas: ResourceBounds::default() }`.
2. Submit to the gateway. The gateway accepts it (non-zero `max_possible_fee`).
3. The gateway computes `H_submit = get_tip_resource_bounds_hash(AllResources { ... }, tip)` — a four-element Poseidon hash including `l1_data_packed ≠ 0`.
4. The transaction is included in a block with `H_submit` as its transaction hash.
5. A syncing peer receives the block via P2P. The `DeclareTransactionV3Common` deserialization calls `ValidResourceBounds::try_from(protobuf_resource_bounds)`. Since `l2_gas = 0` and `l1_data_gas = 0`, the result is `ValidResourceBounds::L1Gas(l1_gas)`.
6. The peer recomputes `H_resync = get_tip_resource_bounds_hash(L1Gas { ... }, tip)` — a three-element Poseidon hash without `l1_data_packed`.
7. `H_resync ≠ H_submit`. `validate_transaction_hash` returns `false`, and the block is rejected or stored with the wrong hash.

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```
