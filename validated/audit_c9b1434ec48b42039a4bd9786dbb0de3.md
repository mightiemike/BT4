### Title
Deposit to Uncontrollable Contract Subaccount Permanently Locks Collateral - (File: `core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral` is a `public` function that accepts a caller-supplied `bytes32 subaccount` and credits deposited collateral to it. The first 20 bytes of `subaccount` become the owner address that must sign all future operations (withdrawals, linked-signer registration). There is no check that this embedded address is capable of producing EIP-712 signatures. If the embedded address is a smart contract that cannot sign, the deposited funds are permanently locked with no admin recovery path.

---

### Finding Description

`depositCollateralWithReferral` in `Endpoint.sol` is `public` and accepts an arbitrary `bytes32 subaccount`:

```solidity
function depositCollateralWithReferral(
    bytes32 subaccount,
    uint32 productId,
    uint128 amount,
    string memory
) public {
    require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);
    address sender = address(bytes20(subaccount));
    requireUnsanctioned(msg.sender);
    requireUnsanctioned(sender);
    ...
    handleDepositTransfer(IERC20Base(spotEngine.getToken(productId)), msg.sender, uint256(amount));
``` [1](#0-0) 

Tokens are pulled from `msg.sender` and credited to the arbitrary `subaccount`. The only validation is a sanctions check on the embedded address and an isolation check — there is no verification that the embedded address can sign EIP-712 messages.

Every subsequent operation on a subaccount requires a valid EIP-712 signature from the address embedded in its first 20 bytes. `WithdrawCollateral` and `WithdrawCollateralV2` both call `validateSignedTx`, which calls `validateSignature` → `verifier.validateSignature`: [2](#0-1) 

`LinkSigner` — the only mechanism to delegate signing authority to another address — also requires a signature from the subaccount owner: [3](#0-2) 

If the embedded address is a smart contract that cannot produce ECDSA signatures (e.g., a Gnosis Safe, an account-abstraction wallet, a DeFi vault, or any contract without a pre-configured linked signer), neither withdrawal nor linked-signer registration is possible. The funds are permanently locked in the `SpotEngine` balance.

By contrast, the safe sibling function `depositCollateral` always derives the subaccount from `msg.sender`, ensuring the depositor controls the embedded address: [4](#0-3) 

---

### Impact Explanation

Deposited collateral is permanently unrecoverable. The `SpotEngine` balance is incremented for the target subaccount, but no withdrawal, transfer, or linked-signer transaction can ever be authorized because all require an EIP-712 signature from the contract address that owns the subaccount. There is no owner-controlled rescue or recovery function. The asset delta is a full loss of the deposited `amount` for the depositing user.

---

### Likelihood Explanation

Medium. The scenario arises naturally when:

1. A user interacts with Nado through a smart contract wallet (Gnosis Safe, ERC-4337 account) and passes their contract address as the first 20 bytes of `subaccount`.
2. A third-party integration contract calls `depositCollateralWithReferral` on behalf of a user, embedding the integration contract's own address in `subaccount` by mistake.
3. A `DirectDepositV1` contract calls `depositCollateralWithReferral` with a `subaccount` whose embedded address is itself or another contract. [5](#0-4) 

Smart contract wallets are increasingly common and the `depositCollateralWithReferral` interface is explicitly designed for third-party callers (it accepts an arbitrary subaccount rather than deriving it from `msg.sender`), making this a realistic integration footgun.

---

### Recommendation

1. **Add a contract-code check** on the address embedded in `subaccount` before accepting the deposit:
   ```solidity
   address owner = address(bytes20(subaccount));
   require(owner.code.length == 0, "subaccount owner must be an EOA");
   ```
   Or, alternatively, require that a linked signer is pre-registered before the deposit is credited.

2. **Restrict `depositCollateralWithReferral`** so that the embedded address must equal `msg.sender`, mirroring the safe `depositCollateral` path, unless the caller is a whitelisted integration contract (e.g., `DirectDepositV1`).

3. **Add an owner-controlled fund-rescue path** for subaccounts whose owner address is provably unable to sign (e.g., a contract with no signing capability), so that mistakenly locked funds can be recovered.

---

### Proof of Concept

1. Deploy a contract `VaultContract` that holds user funds and calls `depositCollateralWithReferral`. The vault's own address (`address(this)`) is embedded as the first 20 bytes of `subaccount`.
2. `VaultContract` approves the Endpoint and calls:
   ```solidity
   endpoint.depositCollateralWithReferral(
       bytes32(bytes20(address(this))), // subaccount owner = VaultContract
       QUOTE_PRODUCT_ID,
       1_000e6,
       ""
   );
   ```
3. Tokens are transferred from `VaultContract` to the Endpoint; `SpotEngine` credits `1_000e6` (scaled) to the subaccount.
4. `VaultContract` attempts to withdraw by constructing a `WithdrawCollateral` transaction and submitting it via `submitSlowModeTransaction`. The call reverts at `validateSignedTx` because `VaultContract` cannot produce a valid ECDSA signature over the EIP-712 digest.
5. `VaultContract` attempts to register a linked signer via `LinkSigner` slow-mode transaction. This also reverts for the same reason.
6. The `1_000e6` collateral is permanently locked with no recovery path. [6](#0-5) [7](#0-6)

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
        // copy from submitSlowModeTransaction
        SlowModeConfig memory _slowModeConfig = slowModeConfig;

        slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: abi.encodePacked(
                uint8(TransactionType.DepositCollateral),
                abi.encode(
                    DepositCollateral({
                        sender: subaccount,
                        productId: productId,
                        amount: amount
                    })
                )
            )
        });
        slowModeConfig = _slowModeConfig;
    }
