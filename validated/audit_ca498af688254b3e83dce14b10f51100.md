### Title
`ValidResourceBounds` Protobuf Deserialization Silently Downgrades `AllResources` to `L1Gas` When L2/Data Gas Are Zero, Causing Consensus Proposal Rejection - (File: crates/apollo_protobuf/src/converters/transaction.rs)

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a zero-value heuristic to decide between the `L1Gas` and `AllResources` variants. A transaction submitted to the gateway with `AllResourceBounds{l1_gas: X, l2_gas: 0, l1_data_gas: 0}` is accepted and hashed under the `AllResources` path. When that same transaction is later serialized into a `ConsensusTransaction` protobuf and deserialized by a validator peer, the deserializer produces `ValidResourceBounds::L1Gas` instead of `ValidResourceBounds::AllResources`. The downstream conversion to `RpcInvokeTransactionV3` (which requires `AllResourceBounds`) then fails with `DEPRECATED_RESOURCE_BOUNDS_ERROR`, causing the validator to reject the entire block proposal.

### Finding Description

In `crates/apollo_protobuf/src/converters/transaction.rs`, the `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` implementation uses a value-based heuristic to reconstruct the Rust enum variant:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

The protobuf wire format (`ResourceBounds` message) carries no discriminant bit to distinguish the two variants; the deserializer infers the variant purely from whether `l2_gas` and `l1_data_gas` are zero.

On the submission side, `RpcDeclareTransactionV3` and `RpcInvokeTransactionV3` both carry `resource_bounds: AllResourceBounds` (not `ValidResourceBounds`), so the gateway always hashes them under the `AllResources` branch of `get_tip_resource_bounds_hash`:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
``` [2](#0-1) 

The `AllResources` path hashes three resource bounds (L1, L2, L1_DATA); the `L1Gas` path hashes only two. A transaction with `l2_gas = 0` and `l1_data_gas = 0` is accepted by the gateway and hashed with the three-element preimage. When the batcher later serializes it into a `ConsensusTransaction` protobuf and a validator peer deserializes it, the heuristic fires and the variant becomes `L1Gas`. The conversion from `ValidResourceBounds::L1Gas` to `AllResourceBounds` (required by `RpcInvokeTransactionV3` / `RpcDeclareTransactionV3`) then fails:

```rust
resource_bounds: match common.resource_bounds {
    ValidResourceBounds::AllResources(resource_bounds) => resource_bounds,
    _ => {
        return Err(DEPRECATED_RESOURCE_BOUNDS_ERROR);
    }
},
``` [3](#0-2) 

The codebase itself acknowledges this ambiguity in the consensus round-trip test, which explicitly adds a non-zero `l2_gas` value before serialization to prevent the heuristic from firing:

```rust
// If all the fields of `AllResources` are 0 upon serialization,
// then the deserialized value will be interpreted as the `L1Gas` variant.
fn add_gas_values_to_transaction(transactions: &mut [ConsensusTransaction]) {
    ...
    resource_bounds.l2_gas.max_amount = GasAmount(1);
``` [4](#0-3) 

No equivalent guard exists in the production gateway admission path.

### Impact Explanation

A validator that receives a `ConsensusTransaction` protobuf for a transaction with `l2_gas = 0` and `l1_data_gas = 0` will fail to deserialize it and reject the entire block proposal. The proposer must retry without that transaction. An attacker who continuously submits such transactions can keep them cycling through the mempool and into proposals, repeatedly aborting consensus rounds. This matches the **High** impact scope: "Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing" and "Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."

### Likelihood Explanation

The gateway's stateless validator explicitly accepts transactions where only one resource bound is non-zero (test cases `valid_l1_gas`, `valid_l2_gas`, `valid_l1_data_gas` all pass with the other two at zero): [5](#0-4) 

Any user can craft such a transaction with no special privileges. The condition (`l2_gas = 0` and `l1_data_gas = 0`) is trivially reachable.

### Recommendation

1. **Add a discriminant to the protobuf schema.** Add a boolean or enum field to `ResourceBounds` (e.g., `bool is_all_resources`) so the deserializer does not need to infer the variant from values.
2. **Alternatively, reject at the gateway.** If `AllResources` transactions with `l2_gas = 0` and `l1_data_gas = 0` are semantically equivalent to `L1Gas` transactions, reject them at the gateway with a clear error rather than accepting them and letting them fail later in consensus.
3. **Remove the heuristic.** Replace the zero-value heuristic in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` with an explicit discriminant check, and update the serializer to emit that discriminant.

### Proof of Concept

1. Submit an `InvokeV3` transaction to the gateway with:
   - `resource_bounds.l1_gas = { max_amount: 1000, max_price_per_unit: 1 }`
   - `resource_bounds.l2_gas = { max_amount: 0, max_price_per_unit: 0 }`
   - `resource_bounds.l1_data_gas = { max_amount: 0, max_price_per_unit: 0 }`
2. Gateway accepts the transaction; hash is computed via the `AllResources` branch (three-element preimage including `l1_data_gas = 0`).
3. Transaction enters the mempool. Batcher includes it in a block proposal.
4. Proposer serializes the transaction as `ConsensusTransaction` protobuf and broadcasts it.
5. Validator calls `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`: `l1_data_gas.is_zero() && l2_gas.is_zero()` → returns `ValidResourceBounds::L1Gas`.
6. Downstream conversion to `RpcInvokeTransactionV3` hits the `_ => Err(DEPRECATED_RESOURCE_BOUNDS_ERROR)` arm.
7. Validator rejects the proposal. Consensus round fails. [6](#0-5) [7](#0-6)

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;

            ValidResourceBounds::L1Gas(l1_gas)
        } else {

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
