### Title
Bonus Reward Checkpoints Advanced Without Payment on Insufficient Pool During Redelegation Slash - (`x/tieredrewards/keeper/slash.go`)

### Summary

In `slashRedelegationPosition`, when `processEventsAndClaimBonus` fails with `ErrInsufficientBonusPool`, the error is deliberately swallowed to prevent a chain halt. However, by the time the error is returned, `processEventsAndClaimBonus` has already (a) decremented validator event reference counts in the persistent store and (b) advanced the position's `LastBonusAccrual`, `LastEventSeq`, and `LastKnownBonded` checkpoints in memory. `slashRedelegationPosition` then calls `setPositionWithState` with the modified position, persisting those advanced checkpoints. The bonus coins are never sent to the owner, and the accrual window is permanently closed — the rewards are irrecoverably lost.

---

### Finding Description

`processEventsAndClaimBonus` performs two categories of work before it checks pool sufficiency:

**Persisted store writes (inside the event loop):**
```go
// decrementEventRefCount writes to ValidatorEvents store
if err := k.decrementEventRefCount(ctx, valAddr, entry.Seq); err != nil {
    return nil, err
}
``` [1](#0-0) 

**In-memory checkpoint advances (after the loop, before the pool check):**
```go
applyBonusAccrualCheckpoint(&pos.Position, blockTime)
pos.UpdateLastKnownBonded(bonded)
``` [2](#0-1) 

Only then does the pool sufficiency check fire and return an error:
```go
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err
}
``` [3](#0-2) 

Back in `slashRedelegationPosition`, the `ErrInsufficientBonusPool` error is caught and swallowed:
```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
    } else {
        return err
    }
}
``` [4](#0-3) 

Execution then falls through to `setPositionWithState`, which persists the position with the already-advanced checkpoints:
```go
return k.setPositionWithState(ctx, pos, &ValidatorTransition{PreviousAddress: dstValStr})
// or
return k.setPositionWithState(ctx, pos, nil)
``` [5](#0-4) 

The net result is a three-way desynchronization:
1. **Validator event reference counts** are decremented (and events may be garbage-collected) — the position can never replay those events again.
2. **Position checkpoints** (`LastBonusAccrual`, `LastEventSeq`, `LastKnownBonded`) are advanced to the slash block time — the accrual window is permanently closed.
3. **Bonus coins** are never transferred to the owner.

The ADR explicitly documents the intended behavior for user-driven paths as "fail atomically; user retries after pool is replenished": [6](#0-5) 

But the hook path does not retry — it silently forfeits and advances state, making the loss permanent.

---

### Impact Explanation

A position owner permanently loses all bonus rewards that accrued since their last claim checkpoint, up to the moment of the redelegation slash. The exact amount is `shares × tokensPerShare × bonusApy × duration / SecondsPerYear` for the unclaimed segment. Because `LastBonusAccrual` is advanced to the slash block time and the validator events are reference-count-decremented (potentially GC'd), there is no mechanism to recover or replay the lost accrual. The corrupted value is the position's `LastBonusAccrual` timestamp and `LastEventSeq` stored in `Positions` collection, and the `ReferenceCount` fields in `ValidatorEvents`.

---

### Likelihood Explanation

The trigger requires two concurrent conditions:
1. A validator is slashed during an active redelegation period for a tier position — a normal protocol event reachable via governance evidence submission or double-sign evidence.
2. The `RewardsPoolName` module account balance is insufficient to cover the accrued bonus — realistic because the `BeginBlocker` continuously drains the same pool for base-reward top-ups, and the pool may simply not be replenished promptly.

Neither condition requires privileged access or social engineering. The `BeforeRedelegationSlashed` hook is triggered by the standard Cosmos SDK staking module's `SlashRedelegation` path, reachable by any validator double-sign or governance slash proposal.

---

### Recommendation

`processEventsAndClaimBonus` must not advance checkpoints or decrement event reference counts before confirming the pool can cover the bonus. The pool sufficiency check should be moved to before the event-processing loop, or the function should be restructured so that all store mutations (including `decrementEventRefCount`) occur only after a successful `SendCoinsFromModuleToAccount`. Alternatively, `slashRedelegationPosition` should not call `setPositionWithState` when `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, preserving the original checkpoints so the owner can claim later.

---

### Proof of Concept

1. Owner creates a tier position and redelegates to a new validator (creating a redelegation entry and a `RedelegationMappings` entry).
2. Time passes; bonus accrues on the destination validator.
3. The `RewardsPoolName` balance is drained to zero (e.g., by BeginBlocker top-ups or other claimants).
4. The source validator is slashed, triggering `BeforeRedelegationSlashed` → `slashRedelegationPosition`.
5. `processEventsAndClaimBonus` walks all pending events, decrements their reference counts in the store, advances `LastBonusAccrual` to block time, then returns `ErrInsufficientBonusPool`.
6. `slashRedelegationPosition` swallows the error and calls `setPositionWithState` with the modified position.
7. The owner's balance is unchanged; `LastBonusAccrual` is now at the slash block time; the validator events have been reference-count-decremented (and may be deleted).
8. Any subsequent `MsgClaimTierRewards` for this position yields zero bonus for the lost period — the accrual window `[original LastBonusAccrual, slash block time]` is gone.
