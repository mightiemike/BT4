### Title
Missing Upper Bound on Fast Withdrawal Fee Allows Fee to Exceed Withdrawal Amount — (`File: core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.fastWithdrawalFeeAmount` computes a fee as `max(proportionalFee, minFee)` where `minFee = 5 * withdrawFeeX18` is a fixed absolute amount, not proportional to the withdrawal. There is no cap ensuring the fee never exceeds the withdrawal amount. The guard that prevents `fee > amount` in `submitFastWithdrawal` is only applied in the `sendTo == msg.sender` branch; the `sendTo != msg.sender` branch has no such check, allowing a fast withdrawal provider to be charged a fee larger than the withdrawal amount itself.

---

### Finding Description

`fastWithdrawalFeeAmount` computes the fee as:

```solidity
int128 proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18); // 0.1% of amount
int128 minFeeX18 = 5 * spotEngine().getConfig(productId).withdrawFeeX18; // fixed floor
int128 feeX18 = MathHelper.max(proportionalFeeX18, minFeeX18);
``` [1](#0-0) 

`FAST_WITHDRAWAL_FEE_RATE` is `0.1%` and is always bounded relative to the amount. However, `minFeeX18 = 5 * withdrawFeeX18` is a **fixed absolute value** independent of the withdrawal amount. For small withdrawals, `minFeeX18` dominates and can exceed `amountX18`. [2](#0-1) 

In `submitFastWithdrawal`, the guard against `fee > amount` is only applied in the `sendTo == msg.sender` branch:

```solidity
if (sendTo == msg.sender) {
    require(transferAmount > uint128(fee), "Fee larger than balance"); // ✅ protected
    transferAmount -= uint128(fee);
} else {
    safeTransferFrom(token, msg.sender, uint128(fee)); // ❌ no upper bound check
}
``` [3](#0-2) 

When `sendTo != msg.sender` (a third-party fast withdrawal provider fronting the funds), the fee is pulled from `msg.sender` with no check that `fee <= transferAmount`. The provider can be charged more than the withdrawal is worth.

---

### Impact Explanation

A fast withdrawal provider (`msg.sender`) processing a small withdrawal where `sendTo != msg.sender` will have `safeTransferFrom` pull a fee exceeding the withdrawal amount from their balance. For example, if `withdrawFeeX18 = 1e18` ($1 in X18), then `minFeeX18 = 5e18` ($5). For a USDC withdrawal (6 decimals, `multiplier = 1e12`), any withdrawal under 5 USDC results in `fee > amount`. The provider loses more in fees than the user receives, making fast withdrawal provision economically destructive for small amounts and deterring liquidity providers from participating.

---

### Likelihood Explanation

Medium. The `withdrawFeeX18` is a per-product config value set by the protocol admin. With any non-trivial `withdrawFeeX18`, small withdrawals will trigger this condition. Users can submit arbitrarily small withdrawal amounts (no minimum withdrawal is enforced in `submitFastWithdrawal`), and third-party fast withdrawal providers calling the function without pre-checking `fastWithdrawalFeeAmount` will be overcharged. The asymmetric protection (only guarded in the `sendTo == msg.sender` branch) makes this a concrete, reachable code path.

---

### Recommendation

Add an upper bound check in the `sendTo != msg.sender` branch of `submitFastWithdrawal`, mirroring the existing guard in the `sendTo == msg.sender` branch:

```solidity
} else {
    require(uint128(fee) <= transferAmount, "Fee larger than withdrawal amount");
    safeTransferFrom(token, msg.sender, uint128(fee));
}
```

Alternatively, enforce the cap inside `fastWithdrawalFeeAmount` itself by capping `feeX18` at `amountX18`.

---

### Proof of Concept

1. Protocol admin sets `withdrawFeeX18 = 1e18` ($1) for USDC (productId = 1).
2. User submits a `WithdrawCollateral` transaction for `amount = 1_000_000` (1 USDC, 6 decimals), with `sendTo` set to their own EOA address.
3. A third-party fast withdrawal provider calls `submitFastWithdrawal(idx, transaction, signatures)`.
4. `fastWithdrawalFeeAmount` computes:
   - `multiplier = 10^(18-6) = 1e12`
   - `amountX18 = 1_000_000 * 1e12 = 1e18` (= $1 in X18)
   - `proportionalFeeX18 = 0.001 * 1e18 = 1e15` ($0.001)
   - `minFeeX18 = 5 * 1e18 = 5e18` ($5)
   - `feeX18 = max(1e15, 5e18) = 5e18`
   - `fee = 5e18 / 1e12 = 5_000_000` (5 USDC)
5. Since `sendTo` (user's EOA) `!= msg.sender` (provider), the `else` branch executes: `safeTransferFrom(token, msg.sender, 5_000_000)`.
6. The provider is charged 5 USDC in fees to process a 1 USDC withdrawal — a 500% fee with no on-chain revert. [4](#0-3) [5](#0-4)

    function fastWithdrawalFeeAmount(
        IERC20Base token,
        uint32 productId,
        uint128 amount
    ) public view returns (int128) {
        uint8 decimals = token.decimals();
        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - uint8(decimals)));
        int128 amountX18 = int128(amount) * int128(multiplier);

        int128 proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18);
        int128 minFeeX18 = 5 * spotEngine().getConfig(productId).withdrawFeeX18;

        int128 feeX18 = MathHelper.max(proportionalFeeX18, minFeeX18);
        return feeX18 / int128(multiplier);
    }
```
