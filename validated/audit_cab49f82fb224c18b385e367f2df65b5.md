### Title
CEI Violation in `withdrawCollateral` Enables Double-Withdrawal via Reentrant Slow-Mode Execution — (`core/contracts/Clearinghouse.sol`)

### Summary
`Clearinghouse.withdrawCollateral` transfers tokens to the recipient address **before** decrementing the subaccount's spot balance in `SpotEngine`. Because `executeSlowModeTransaction` is permissionlessly callable by any address, a malicious recipient contract can reenter during the token transfer callback and execute a second queued withdrawal against the still-unmodified balance. If the attacker holds a perp position whose health contribution exceeds the over-drawn spot deficit, both health checks pass and the attacker drains more tokens than their spot balance.

### Finding Description

In `Clearinghouse.withdrawCollateral`, the call to `handleWithdrawTransfer` — which ultimately calls `token.safeTransfer(sendTo, amount)` — occurs **before** `spotEngine.updateBalance` decrements the balance:

```
handleWithdrawTransfer(token, sendTo, amount, idx);   // ← external call, tokens leave
...
spotEngine.updateBalance(productId, sender, amountRealized); // ← balance decremented AFTER
...
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
``` [1](#0-0) 

`handleWithdrawTransfer` in `Clearinghouse` forwards tokens to `withdrawPool` and then calls `withdrawPool.submitWithdrawal`, which calls `token.safeTransfer(sendTo, amount)`: [2](#0-1) 

`BaseWithdrawPool.submitWithdrawal` performs the final transfer to `sendTo`: [3](#0-2) 

`BaseWithdrawPool.handleWithdrawTransfer` calls `token.safeTransfer(to, amount)` directly: [4](#0-3) 

`Endpoint.executeSlowModeTransaction` has **no access control and no reentrancy guard**: [5](#0-4) 

This means any contract that receives tokens during step 1 can immediately call `executeSlowModeTransaction` to process a second queued withdrawal while the first call's balance decrement has not yet occurred.

### Impact Explanation

The attacker withdraws `2 × amount` tokens while only holding `amount` in their spot balance. The deficit (`-amount`) in the spot engine is masked by a perp position whose health contribution exceeds the spot shortfall. Both the reentrant call's health check and the outer call's health check pass, leaving the protocol short by `amount` tokens. This is a direct theft of protocol funds from the `WithdrawPool`/`Clearinghouse` token reserves.

### Likelihood Explanation

The attack requires:
1. The collateral token supports a transfer callback (ERC777 `tokensReceived`, or any token with a `transfer` hook). If the protocol lists any such token, the path is open.
2. The attacker controls a contract address as the subaccount owner (trivially achievable — subaccounts are derived from `msg.sender`).
3. Two `WithdrawCollateral` slow-mode transactions queued at least 3 days in advance (the `SLOW_MODE_TX_DELAY`).
4. A perp position with health contribution `H > amount × spotWeightX18` to absorb the negative spot balance after both decrements.

All four preconditions are reachable by an unprivileged user with no special keys.

### Recommendation

Move `spotEngine.updateBalance` and the health check **before** `handleWithdrawTransfer`, following the checks-effects-interactions pattern:

```solidity
// 1. Compute and apply balance delta first
int128 amountRealized = -int128(amount) * int128(multiplier);
spotEngine.updateBalance(productId, sender, amountRealized);
spotEngine.assertUtilization(productId);
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);

// 2. Only then transfer tokens
handleWithdrawTransfer(token, sendTo, amount, idx);
```

Additionally, add a `nonReentrant` modifier (OpenZeppelin `ReentrancyGuard`) to `withdrawCollateral` and to `Endpoint.executeSlowModeTransaction` as a defense-in-depth measure.

### Proof of Concept

1. Attacker deploys `MaliciousContract` (MC). MC's address becomes the first 20 bytes of its subaccount.
2. MC deposits `X` units of an ERC777 token and opens a perp position with health contribution `H > X × spotWeightX18`.
3. MC queues two slow-mode `WithdrawCollateral` transactions of `X` tokens each. Both become executable after 3 days.
4. MC calls `Endpoint.executeSlowModeTransaction()` to process the first withdrawal.
5. Execution reaches `BaseWithdrawPool.handleWithdrawTransfer` → `token.safeTransfer(MC, X)`.
6. The ERC777 `tokensReceived` hook fires on MC. At this point `spotEngine` balance is still `X`.
7. Inside the hook, MC calls `Endpoint.executeSlowModeTransaction()` for the second withdrawal.
   - `spotEngine.updateBalance` decrements balance: `X → 0`.
   - `getHealth`: spot = 0, perp = H ≥ 0. ✓ Second withdrawal completes; MC receives `X` tokens.
8. Control returns to the first withdrawal. `spotEngine.updateBalance` decrements balance: `0 → -X`.
9. `getHealth`: spot contribution = `-X × weight`, perp = H. Since `H > X × weight`, health ≥ 0. ✓ First withdrawal completes; MC receives another `X` tokens.
10. MC has received `2X` tokens. Protocol's token reserve is short by `X` tokens. MC's spot balance is `-X`, covered only by the perp position — a permanent accounting desynchronization exploitable further if the perp position is later closed. [6](#0-5) [5](#0-4)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {

        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
