### Title
`alignVestingDelegationTracking` No-Op on Slashed Commit-Origin Position Leaves DV+DF Stale-High, Enabling Spend of Locked Tokens — (`x/tieredrewards/keeper/force_exit.go`)

---

### Summary

`alignVestingDelegationTracking` is unidirectional: it only tops up `DV+DF` when `actualDelegated > tracked`. For a commit-origin vesting position whose validator was slashed before the v8 upgrade, the returned delegation is `X − slash`, which is **less than** the stale-high `DV+DF = X`. The guard at line 156 fires and the function returns without adjustment. `DV+DF` is permanently left at `X` while actual delegations are `X − slash`. A subsequent normal `MsgUndelegate` reduces `DV` by only `X − slash`, leaving a residual `DV = slash`. This understates `LockedCoins` by `slash`, allowing the owner to spend `slash` tokens that should be permanently locked.

---

### Finding Description

**Root cause — the one-sided guard:** [1](#0-0) 

```go
if !actualDelegated.GT(tracked) {
    // ...
    return nil   // ← no-op when actualDelegated ≤ tracked
}
```

The function's own comment documents the design assumption that breaks under a slash: [2](#0-1) 

> "For CommitDelegationToTier-origin positions DV+DF was already stale-high pre-migration and the **returning delegation closes the gap**."

This is only true when no slash occurred. A slash reduces the position's token value from `X` to `X − slash`, so the returned delegation is `X − slash`, not `X`. The gap is **not** closed.

**Why `transferDelegationFromPosition` does not fix it:**

The re-delegation to the owner uses `subtractAccount=false`: [3](#0-2) 

```go
ownerNewShares, err := k.stakingKeeper.Delegate(ctx, owner, transferredAmount, validator.GetStatus(), validator, false)
```

`subtractAccount=false` skips the bank-side `TrackDelegation` hook, so `DV+DF` is never updated by the transfer itself. `alignVestingDelegationTracking` is the sole correction mechanism, and it silently no-ops.

**State after migration (slash scenario):**

| Field | Value |
|---|---|
| `DV` | `X` (stale-high, unchanged) |
| `DF` | `0` |
| Owner on-chain delegation | `X − slash` |
| `LockedCoins = OriginalVesting − DV` | `X − X = 0` (should be `slash`) |

**State after owner calls `MsgUndelegate(X − slash)`:**

`TrackUndelegation(X − slash)` reduces `DV` by `X − slash`:

- `DV = X − (X − slash) = slash`
- `DF = 0`
- Actual delegations = `0`

After unbonding completes, owner's bank balance = `(X − slash) + Y` (where `Y` is any free tokens received from rewards, gas refunds, etc.):

| Metric | With stale-high DV | Correct (DV = 0) |
|---|---|---|
| `LockedCoins` | `X − slash` | `X` |
| `SpendableCoins` | `Y` | `Y − slash` |

The owner can spend `slash` tokens that should be permanently locked.

---

### Impact Explanation

A `PermanentLockedAccount` owner whose commit-origin tier position was on a slashed validator can, after the v8 migration, spend up to `slash` tokens that should be permanently locked. The `SpendableCoins` invariant (`SpendableCoins ≤ balance − LockedCoins`) is violated by exactly `slash` tokens. Any free balance at the owner address (staking rewards, gas refunds, transfers) is the pool from which the extra `slash` tokens are unlocked.

---

### Likelihood Explanation

- Validator slashing is a normal, non-privileged chain event.
- Commit-origin vesting positions existed on mainnet before v8.
- The attacker needs only to have had a commit-origin position on a validator that was slashed before the upgrade — no special access required.
- The only additional requirement is that the owner holds some free tokens post-migration (rewards, gas refunds), which is routine.

---

### Recommendation

`alignVestingDelegationTracking` must handle both directions. When `actualDelegated < tracked`, call `TrackUndelegation` to reduce `DV+DF` to match `actualDelegated`:

```go
if actualDelegated.GT(tracked) {
    deficit := actualDelegated.Sub(tracked)
    coins := sdk.NewCoins(sdk.NewCoin(bondDenom, deficit))
    sdkCtx := sdk.UnwrapSDKContext(ctx)
    vacc.TrackDelegation(sdkCtx.BlockTime(), coins, coins)
    k.accountKeeper.SetAccount(ctx, vacc)
} else if tracked.GT(actualDelegated) {
    surplus := tracked.Sub(actualDelegated)
    coins := sdk.NewCoins(sdk.NewCoin(bondDenom, surplus))
    vacc.TrackUndelegation(coins)
    k.accountKeeper.SetAccount(ctx, vacc)
}
```

### Proof of Concept

Extend `TestForceFullExitWithDelegation_VestingOwner_CommitOrigin` with a slash before `ForceFullExitWithDelegation`:

```go
// After createCommitPositionV1, slash the validator by 10%
slashFactor := sdkmath.LegacyNewDecWithPrec(10, 2)
s.app.StakingKeeper.Slash(s.ctx, consAddr, s.ctx.BlockHeight(), power, slashFactor, stakingtypes.Infraction_INFRACTION_DOUBLE_SIGN)

// Run migration
s.Require().NoError(s.keeper.ForceFullExitWithDelegation(s.ctx, pos.Id))

// alignVestingDelegationTracking was a no-op: DV+DF still == lockedAmount
s.Require().Equal(lockedAmount, s.trackedTotal(owner, bondDenom),
    "BUG: DV+DF is stale-high after slash; should equal actualDelegated")

// Owner undelegates the returned (slashed) amount
ownerDel := s.totalDelegated(owner)
_, _, err := s.app.StakingKeeper.Undelegate(s.ctx, owner, valAddr, ownerDel /* shares */)
s.Require().NoError(err)
s.completeStakingUnbonding(valAddr, owner)

// Fund owner with a small free amount (simulates rewards received)
freeTokens := sdkmath.NewInt(1000)
s.Require().NoError(banktestutil.FundAccount(s.ctx, s.app.BankKeeper, owner,
    sdk.NewCoins(sdk.NewCoin(bondDenom, freeTokens))))

// SpendableCoins should be 0 (all locked), but is freeTokens due to stale DV
spendable := s.app.BankKeeper.SpendableCoins(s.ctx, owner).AmountOf(bondDenom)
s.Require().True(spendable.GT(sdkmath.ZeroInt()),
    "EXPLOIT: owner can spend locked tokens due to stale-high DV after slash")
```

The test demonstrates that `SpendableCoins > 0` when it should be `≤ 0` (clamped), violating the `SpendableCoins ≤ balance − LockedCoins` invariant.
