### Title
`nlpPools` Array Grows Unboundedly Due to Missing Pop in `deleteNlpPool`, Causing Permanent Fund Freeze for NLP Token Holders - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

`deleteNlpPool()` in `EndpointTx.sol` zeroes out a pool entry but never removes it from the `nlpPools` storage array. Meanwhile, `addNlpPool()` always pushes a new entry. Every add/delete cycle permanently inflates `nlpPools.length`. Because `_validateNlpRebalance()` enforces `nlpPools.length == nlpPoolRebalanceX18.length`, and `_applyNlpRebalance()` iterates the full array, every `MintNlp`, `BurnNlp`, and `ForceRebalanceNlpPool` transaction must process an ever-growing array. Over time this leads to a permanent fund freeze for NLP token holders who cannot burn their tokens to recover quote collateral.

---

### Finding Description

`addNlpPool()` unconditionally pushes a new `NlpPool` struct onto the `nlpPools` storage array: [1](#0-0) 

`deleteNlpPool()` calls `updateNlpPool(poolId, address(0), uint128(0))`, which only zeroes the `owner` and `balanceWeightX18` fields in-place. It never calls `.pop()` or swaps the entry out: [2](#0-1) [3](#0-2) 

The `nlpPools` array is stored in `EndpointStorage` as a dynamic array that only ever grows: [4](#0-3) 

Every `MintNlp` and `BurnNlp` sequencer transaction passes the full `nlpPools` storage array to `Clearinghouse`: [5](#0-4) [6](#0-5) 

`_validateNlpRebalance()` enforces strict length parity between the passed `nlpPools` and the caller-supplied `nlpPoolRebalanceX18` array: [7](#0-6) 

`_applyNlpRebalance()` then iterates the full array length, calling `spotEngine.updateBalance()` for every entry including deleted (zeroed) ones: [8](#0-7) 

`forceRebalanceNlpPool()` additionally calls `getHealth()` for every pool entry in a separate loop: [9](#0-8) 

**Concrete invariant broken:** After `N` add/delete cycles, `nlpPools.length == N + 1` (initial pool + N added), while the number of active pools remains bounded. The array length is a monotonically increasing counter that can never decrease.

---

### Impact Explanation

Each `BurnNlp` transaction — the only mechanism for an NLP token holder to recover quote collateral — must iterate the full `nlpPools` array via `_applyNlpRebalance` and `_validateNlpRebalance`. As `nlpPools` grows:

1. **Temporal fund freeze**: Gas cost per `BurnNlp` increases proportionally to `nlpPools.length`, making the operation economically non-viable before the block gas limit is reached.
2. **Permanent fund freeze**: Once `nlpPools.length` is large enough that a `BurnNlp` transaction exceeds the block gas limit, NLP token holders permanently lose the ability to redeem their tokens for quote collateral. `MintNlp` is similarly blocked, halting new NLP issuance.

The corrupted state is `nlpPools.length` — a queue-length invariant that should equal the number of active pools but instead equals the total number of pools ever created.

---

### Likelihood Explanation

Normal protocol administration involves adding new NLP pools (e.g., for new liquidity providers) and deleting obsolete ones. Each such add/delete cycle permanently increments `nlpPools.length` by one. This is not a malicious scenario — it is the expected operational lifecycle. A protocol running in production for months or years with routine pool management will accumulate this bloat organically.

---

### Recommendation

`deleteNlpPool()` should remove the entry from the array rather than zeroing it in place. The standard swap-and-pop pattern is appropriate:

```solidity
function deleteNlpPool(uint64 poolId) private {
    require(poolId > 0 && poolId < nlpPools.length);
    clearinghouse.clearNlpPoolPosition(nlpPools[poolId].subaccount);
    // Swap with last element and pop
    nlpPools[poolId] = nlpPools[nlpPools.length - 1];
    nlpPools[poolId].poolId = poolId; // update poolId to reflect new position
    nlpPools.pop();
}
```

Alternatively, if pool ordering must be preserved, use a tombstone flag and filter it in `_applyNlpRebalance` / `_validateNlpRebalance` rather than iterating tombstoned entries.

---

### Proof of Concept

1. Admin submits `AddNlpPool` slow-mode transaction → `addNlpPool()` is called → `nlpPools.length` becomes 2.
2. Admin submits `DeleteNlpPool` for `poolId=1` → `deleteNlpPool(1)` is called → `nlpPools[1].owner = address(0)`, `nlpPools[1].balanceWeightX18 = 0`, but `nlpPools.length` remains 2.
3. Repeat steps 1–2 fifty times → `nlpPools.length == 51`, active pools == 1.
4. A user signs and submits a `BurnNlp` transaction. The sequencer must supply `nlpPoolRebalanceX18` of length 51. `_validateNlpRebalance` enforces this at [7](#0-6) . `_applyNlpRebalance` calls `spotEngine.updateBalance()` 51 times at [8](#0-7) .
5. After enough cycles, the gas cost of step 4 exceeds the block gas limit. The user's NLP tokens are permanently unredeemable.
