### Title
Dead-Code `DISABLE_OP` Dividend Size Guard in `op_div` / `op_divmod` / `op_mod` — (`File: src/more_ops.rs`)

---

### Summary

In `op_div`, `op_divmod`, and `op_mod` (and their Malachite variants), the `DISABLE_OP`-gated size check for the dividend (`a0_len > 2048`) is unreachable dead code. The unconditional check (`a0_len > 256`) is stricter and always fires first, making the `DISABLE_OP` guard a no-op. The thresholds are swapped relative to their intended roles, mirroring the external report's inverted-bounds-check class.

---

### Finding Description

Each of the six arithmetic division functions in `src/more_ops.rs` contains this two-check pattern:

```rust
// Check 1 — DISABLE_OP-gated (mempool-mode stricter limit)
if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 2048 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
// Check 2 — unconditional hard limit
if a0_len > 256 || a1_len > 1024 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
```

Because `256 < 2048`, Check 2 always fires before Check 1 for any input where `a0_len > 256`. Check 1 can therefore never be reached. The `DISABLE_OP` flag has zero effect on the dividend size limit for these operators.

The same dead-code pattern appears identically in all six functions:
- `op_div` / `op_div_malachite`
- `op_divmod` / `op_divmod_malachite`
- `op_mod` / `op_mod_malachite` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

The `DISABLE_OP` flag is part of `MEMPOOL_MODE` and is explicitly intended to impose stricter operational limits: [5](#0-4) [6](#0-5) 

For `op_modpow` (opcode 60), `DISABLE_OP` correctly disables the operator entirely. For div/divmod/mod, the intent was clearly to impose a tighter dividend limit in mempool mode — but the threshold values (256 unconditional, 2048 conditional) are swapped, making the conditional check dead code. [7](#0-6) 

---

### Impact Explanation

**Consensus / mempool divergence:** If the intended design was to allow dividends up to 2048 bytes in consensus mode and restrict to 256 bytes in mempool mode, the current code incorrectly rejects dividends of 257–2048 bytes in *all* modes. Any CLVM puzzle that uses `(/ large-dividend divisor)` with a dividend in that range is rejected by consensus nodes when it should succeed, and the `DISABLE_OP` mempool guard provides no additional protection.

**Corrupted result:** The concrete corrupted output is `EvalErr::InvalidOpArg` returned for inputs that should succeed in consensus mode (dividend 257–2048 bytes), and the absence of any stricter rejection in mempool mode for the same range.

---

### Likelihood Explanation

The entry path is direct: any attacker-controlled CLVM program submitted to a full node that uses `(/ ...)`, `(divmod ...)`, or `(% ...)` with a dividend atom between 257 and 2048 bytes triggers the broken invariant. The `run_serialized_chia_program` Python API and the `run_program` Rust entry point both reach this code through `ChiaDialect::op`. [8](#0-7) 

---

### Recommendation

Swap the threshold values so the unconditional check uses the larger limit and the `DISABLE_OP` check imposes the stricter limit:

```rust
// Stricter limit in mempool mode
if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 256 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
// Consensus-mode hard limit
if a0_len > 2048 || a1_len > 1024 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
```

Apply the same fix to all six functions (`op_div`, `op_div_malachite`, `op_divmod`, `op_divmod_malachite`, `op_mod`, `op_mod_malachite`).

---

### Proof of Concept

```
; Construct a CLVM program: (/ <257-byte-atom> 2)
; a0_len = 257, a1_len = 1
;
; Expected (if DISABLE_OP check were correct):
;   consensus mode  → OK  (257 ≤ 2048)
;   mempool mode    → Err (257 > 256)
;
; Actual (both modes):
;   → Err InvalidOpArg  (257 > 256, unconditional check fires)
;   DISABLE_OP check at 2048 is never reached.
```
