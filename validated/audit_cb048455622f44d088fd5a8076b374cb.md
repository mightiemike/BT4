### Title
No Chain-Tip Staleness Validation in SPV Proof Verification — (`contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` return proof results based on the contract's internally tracked chain tip without ever checking whether that tip is temporally fresh. If the relayer stops submitting headers, the contract's view of the Bitcoin chain becomes stale, and the confirmation count it computes is relative to an outdated tip — not the real Bitcoin chain. A transaction that has since been reorganized away on the real chain can still receive a `true` verification result.

### Finding Description

Both SPV verification entry points fetch the chain tip and compute confirmations against it: [1](#0-0) 

The `heaviest_block_header` is taken directly from `mainchain_tip_blockhash` with no check on whether `heaviest_block_header.block_header.time` is recent relative to `env::block_timestamp_ms()`. The same path is followed by `verify_transaction_inclusion_v2`, which delegates to `verify_transaction_inclusion` after its coinbase proof check: [2](#0-1) 

The contract does enforce a forward-time bound when *ingesting* new headers — `block_header.time <= current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL` — but there is no symmetric backward-time bound applied when *consuming* the stored tip during verification: [3](#0-2) 

`MAX_FUTURE_BLOCK_TIME_LOCAL` is defined as 2 hours: [4](#0-3) 

No equivalent "tip must be no older than X seconds" guard exists anywhere in the verification path.

### Impact Explanation

Any NEAR contract or protocol that calls `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` to gate a financial operation (e.g., a cross-chain bridge releasing funds upon confirmed Bitcoin deposit) will receive a `true` result even when the contract's tracked chain is arbitrarily far behind the real Bitcoin chain. If the real chain has reorganized the block containing the target transaction after the relayer stalled, the contract returns a false positive. The corrupted value is the boolean proof result itself.

### Likelihood Explanation

The relayer is a single off-chain process with no on-chain liveness enforcement. A crash, misconfiguration, or network partition stops header submission silently; the contract has no circuit-breaker. Bitcoin reorgs of 1–2 blocks occur regularly; deeper reorgs are rare but not impossible. The combination — relayer lag plus a reorg of a recently-submitted block — is a realistic operational scenario, not a purely theoretical one. An adversary who can delay the relayer (e.g., via resource exhaustion) while simultaneously double-spending on Bitcoin has a concrete exploit path.

### Recommendation

Before computing or returning a confirmation-based proof result, validate that the stored chain tip's timestamp is within an acceptable staleness window:

```rust
let tip_time = heaviest_block_header.block_header.time;
let now = u32::try_from(env::block_timestamp_ms() / 1000).unwrap();
require!(
    now <= tip_time + MAX_ACCEPTABLE_TIP_STALENESS,
    "chain tip is stale: relayer may have stopped"
);
```

`MAX_ACCEPTABLE_TIP_STALENESS` should be chosen conservatively (e.g., 2–4 hours for Bitcoin, accounting for normal block-time variance). This is the direct analog of the Chainlink fix: "pool-level expiration checks with a buffer using `updatedAt`."

### Proof of Concept

1. Deploy the contract and initialize it with a genesis block.
2. Submit a batch of headers up to height H; the relayer then stops.
3. On the real Bitcoin chain, the block at height H−1 is reorganized (a natural 1-block reorg).
4. Call `verify_transaction_inclusion_v2` for a transaction in the now-orphaned block at H−1, with `confirmations = 1`.
5. The contract computes `heaviest_block_header.block_height (H) - target_block_height (H-1) + 1 = 2 >= 1` and proceeds to Merkle verification.
6. The Merkle proof is valid (the block is still in `headers_pool`), so the function returns `true`.
7. The real Bitcoin chain no longer contains that transaction; the NEAR-side consumer has been deceived. [5](#0-4)
