Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Enabling Full Allowlist Bypass ‚Äî (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, that `msg.sender` is the router contract address, not the actual user. A pool admin who allowlists the router to support router-mediated swaps on a curated pool inadvertently grants every user access, completely collapsing the per-user allowlist invariant.

## Finding Description
The call chain is confirmed by production code:

**Step 1 ‚Äî Router calls pool as `msg.sender`:**
`MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` at `MetricOmmSimpleRouter.sol` L72‚Äì80. The router is `msg.sender` to the pool.

**Step 2 ‚Äî Pool passes `msg.sender` (router) as `sender` to `_beforeSwap`:**
`MetricOmmPool.swap` at L230‚Äì231 calls `_beforeSwap(msg.sender, recipient, ...)`. When invoked via the router, `msg.sender` is the router address.

**Step 3 ‚Äî `_beforeSwap` forwards `sender` unchanged to the extension:**
`ExtensionCalling._beforeSwap` at L160‚Äì176 encodes `sender` directly into `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))` with no substitution.

**Step 4 ‚Äî Extension evaluates the wrong actor:**
`SwapAllowlistExtension.beforeSwap` at L37 checks:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool (correct) and `sender` is the **router** (wrong). The check becomes `allowedSwapper[pool][router]`, never evaluating the end user's address.

**Bypass path:** A pool admin who wants allowlisted users to trade via the standard router must call `setAllowedToSwap(pool, router, true)`. Once set, every user who calls any router entry point (`exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`) passes the guard, regardless of whether their own address is allowlisted.

**Denial path (secondary):** If the admin allowlists individual users but not the router, those users cannot swap through the router at all ‚Äî the router's address fails the check ‚Äî forcing direct pool calls, which is not the intended UX.

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of participants loses that restriction entirely the moment the router is allowlisted. Any unprivileged user can call `MetricOmmSimpleRouter.exactInputSingle` and execute swaps against the pool's liquidity. This is a direct, complete bypass of the access-control invariant the pool admin configured, exposing LP funds to trades from actors the pool was explicitly designed to exclude. This constitutes broken core pool functionality causing potential loss of funds and an admin-boundary break where the pool admin's configured restriction is rendered ineffective by an unprivileged path.

## Likelihood Explanation
The router is the standard, documented periphery entry point for swaps. Pool admins who deploy a curated pool with a swap allowlist and also want to support the router (the expected user-facing path) will naturally allowlist the router address. The bypass requires no special knowledge or exploit tooling ‚Äî any user who calls the router reaches the pool. The misconfiguration is a predictable consequence of the wrong-actor check and will occur in normal operational usage.

## Recommendation
The extension must gate on the economic actor (the end user), not the intermediary. The cleanest fix is to have the router store `msg.sender` in transient storage (it already uses transient storage for callback context via `_setNextCallbackContext`) and expose a read function. The extension then reads the original caller from the router's transient slot when `sender` is a known router address. Alternatively, the router can forward the original `msg.sender` as encoded `extensionData`, and the extension decodes and checks that value when present. A simpler but more restrictive option is to document and enforce that `SwapAllowlistExtension` is incompatible with router-mediated swaps and revert if `sender` is a known router address.

## Proof of Concept
```solidity
// Setup: pool with SwapAllowlistExtension, only alice is allowlisted
swapExtension.setAllowedToSwap(address(pool), alice, true);
// Admin also allowlists the router so alice can use it
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// Attack: bob (not allowlisted) swaps through the router
vm.startPrank(bob);
token0.approve(address(router), type(uint256).max);
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool: address(pool),
    tokenIn: address(token0),
    tokenOut: address(token1),
    zeroForOne: true,
    amountIn: 1_000,
    amountOutMinimum: 0,
    recipient: bob,
    deadline: block.timestamp + 1,
    priceLimitX64: 0,
    extensionData: ""
}));
// Succeeds: extension checked allowedSwapper[pool][router] = true
// bob's address was never evaluated
vm.stopPrank();
```

Root cause trace:
1. `MetricOmmSimpleRouter.sol` L72‚Äì80 ‚Äî router calls `pool.swap(...)` as `msg.sender`, losing original user identity
2. `MetricOmmPool.sol` L230‚Äì231 ‚Äî pool passes `msg.sender` (router) as `sender` to `_beforeSwap`
3. `ExtensionCalling.sol` L160‚Äì176 ‚Äî `_beforeSwap` forwards `sender` unchanged to the extension
4. `SwapAllowlistExtension.sol` L37 ‚Äî extension checks `allowedSwapper[pool][sender]` where `sender` is the router, not the user
