### Title
`total_mem_usage` reads whole-trie root `memory_usage` (all columns) while `find_trie_split` and `TrieSplit::total_memory()` only account for 4 account-keyed subtrees — representation mismatch in dynamic-resharding threshold and shard-selection logic (`core/store/src/trie/split.rs`, `chain/chain/src/runtime/mod.rs`, `chain/epoch-manager/src/lib.rs`)

---

### Summary

In `check_dynamic_resharding`, the split-trigger threshold is compared against `total_mem_usage`, which returns the trie **root node's cumulative `memory_usage`** — a value that includes every trie column, including global-state columns that are never split by account-ID boundary (delayed receipts, buffered receipts, promise yields, bandwidth-scheduler state, global contract code, etc.). The split algorithm (`find_trie_split` → `TrieDescent`) and the resulting `TrieSplit::total_memory()` used in `pick_shard_to_split` only account for **4 account-keyed subtrees** (`ACCOUNT=0`, `CONTRACT_CODE=1`, `ACCESS_KEY=2`, `CONTRACT_DATA=9`). The two quantities measure fundamentally different things, yet they are used interchangeably in the same decision pipeline, producing incorrect threshold firings and wrong shard-selection outcomes.

---

### Finding Description

**Representation 1 — `total_mem_usage` (threshold check):**

`total_mem_usage` returns `root_node.memory_usage`, the cumulative cost of every node in the entire trie: [1](#0-0) 

This includes all trie columns. The global-state columns that are **not** keyed by account ID and are **not** split by the boundary account include:

| Column | Byte |
|---|---|
| `DELAYED_RECEIPT_OR_INDICES` | 7 |
| `PROMISE_YIELD_INDICES` | 10 |
| `PROMISE_YIELD_TIMEOUT` | 11 |
| `BUFFERED_RECEIPT_INDICES` | 13 |
| `BUFFERED_RECEIPT` | 14 |
| `BANDWIDTH_SCHEDULER_STATE` | 15 |
| `BUFFERED_RECEIPT_GROUPS_QUEUE_DATA` | 16 |
| `BUFFERED_RECEIPT_GROUPS_QUEUE_ITEM` | 17 |
| `GLOBAL_CONTRACT_CODE` | 18 |
| `GLOBAL_CONTRACT_NONCE` | 19 | [2](#0-1) 

**Representation 2 — `TrieDescent::total_memory()` / `TrieSplit::total_memory()` (split algorithm and shard selection):**

`TrieDescent::new` initialises `middle_memory` by summing only the 4 account-keyed subtrees: [3](#0-2) [4](#0-3) 

`TrieDescent::total_memory()` is therefore `left + right + middle` over those 4 subtrees only: [5](#0-4) 

`TrieSplit::total_memory()` (used in `pick_shard_to_split`) is `left_memory + right_memory`, again only account-keyed: [6](#0-5) 

**The mismatch in `check_dynamic_resharding`:**

```
let mem_usage = total_mem_usage(shard_trie)?;   // ALL columns
...
if mem_usage < config.memory_usage_threshold {   // fires on global state too
    return Ok(None);
}
let trie_split = find_trie_split(shard_trie)?;  // only 4 account-keyed subtrees
if trie_split.left_memory < config.min_child_memory_usage { ... }
if trie_split.right_memory < config.min_child_memory_usage { ... }
``` [7](#0-6) 

**The mismatch in `pick_shard_to_split`:**

`pick_shard_to_split` selects the shard with the highest `split.total_memory()` (account-keyed only), but the threshold that admitted each shard into `proposed_splits` was `total_mem_usage` (all columns): [8](#0-7) 

---

### Impact Explanation

**Wrong shard selected for splitting.** Consider two shards both above `memory_usage_threshold`:

| Shard | `total_mem_usage` | account-keyed | global state | `split.total_memory()` |
|---|---|---|---|---|
| A | 1 000 | 500 | 500 | 500 |
| B | 900 | 850 | 50 | 850 |

`pick_shard_to_split` selects **B** (850 account-keyed) even though **A** has more total memory (1 000). Shard A continues to grow unchecked while B is split unnecessarily.

**Premature threshold firing.** A shard with a large delayed-receipt or buffered-receipt queue (global state) can push `total_mem_usage` above `memory_usage_threshold` even when account-keyed state is small. `find_trie_split` is then called, and `min_child_memory_usage` (also account-keyed only) may reject the split — but the expensive trie traversal has already been performed, and the `proposed_split` field is set in the chunk header, consuming bandwidth.

**Calibration incoherence.** `memory_usage_threshold` is documented as calibrated against the root node's `memory_usage` (all columns), while `min_child_memory_usage` is compared against account-keyed memory only. The two thresholds are therefore expressed in different units, making it impossible to set them consistently.

---

### Likelihood Explanation

Any user can send cross-shard transactions whose receipts are delayed (e.g., by targeting a congested shard), inflating `DELAYED_RECEIPT_OR_INDICES` and `BUFFERED_RECEIPT` columns and thus `total_mem_usage` without affecting `split.total_memory()`. Congestion control bounds the absolute size, but the relative inflation is sufficient to shift the threshold decision. The effect is deterministic and reproducible across all validators, so it does not cause consensus failure — it causes a consistently wrong resharding decision that persists until the next epoch boundary.

---

### Recommendation

Use a consistent representation throughout the resharding decision pipeline:

1. **Preferred:** Replace the `total_mem_usage` threshold check with the sum of the 4 account-keyed subtrees (i.e., `TrieDescent::total_memory()` computed during `find_trie_split`). This makes the threshold, the split quality check, and the shard-selection criterion all operate on the same quantity. The `memory_usage_threshold` and `min_child_memory_usage` config values would then be expressed in the same units.

2. **Alternative:** Keep `total_mem_usage` for the threshold but also expose the account-keyed subtree total from `TrieDescent` and use it for `pick_shard_to_split`, so the shard with the most splittable state is selected.

The documentation note that `total_mem_usage` is "roughly equal to *double* of the actual RAM usage" should be updated to clarify that it includes non-splittable global state.

---

### Proof of Concept

**Divergent values (concrete):**

```
SUBTREES = [ACCOUNT=0, CONTRACT_CODE=1, ACCESS_KEY=2, CONTRACT_DATA=9]

total_mem_usage(trie)
  = root_node.memory_usage
  = Σ memory_usage(all columns 0..23)
  = account_keyed_mem + global_state_mem

TrieSplit::total_memory()
  = left_memory + right_memory
  = Σ memory_usage(SUBTREES only)
  = account_keyed_mem

Divergence = global_state_mem
           = DELAYED_RECEIPT + BUFFERED_RECEIPT + PROMISE_YIELD_* + BANDWIDTH_SCHEDULER + GLOBAL_CONTRACT_*
```

**Scenario triggering wrong shard selection:**

1. Shard A accumulates a large delayed-receipt queue (e.g., 500 units of global state) due to cross-shard traffic. Its `total_mem_usage = 1000`, `split.total_memory() = 500`.
2. Shard B has little global state. Its `total_mem_usage = 900`, `split.total_memory() = 850`.
3. Both pass `memory_usage_threshold = 800` and `min_child_memory_usage = 200`.
4. `pick_shard_to_split` calls `.max_by_key(|(_, split)| (split.total_memory(), *shard_id))` and selects **B** (850).
5. Shard A, with 11% more total memory, is not split. Its global state continues to grow, and its account-keyed state also grows unchecked until the next epoch. [9](#0-8) [10](#0-9) [11](#0-10) [8](#0-7) [6](#0-5)

/// Return the total memory usage of the trie (the root node's `memory_usage` field).
/// Used as a quick check before running the more expensive `find_trie_split`.
pub fn total_mem_usage(trie: &Trie) -> FindSplitResult<u64> {
    match trie.lock_memtries() {
        Some(memtries) => {
            let trie_storage = MemTrieIteratorInner::new(&memtries, trie);
            let root_id = trie_storage.get_root().ok_or(FindSplitError::NoRoot)?;
            let root_node = trie_storage.get_node_with_size(root_id, AccessOptions::DEFAULT)?;
            Ok(root_node.memory_usage)
        }
        None => {
            let trie_storage = DiskTrieIteratorInner::new(trie);
            let root_id = trie_storage.get_root().ok_or(FindSplitError::NoRoot)?;
            let root_node = trie_storage.get_node_with_size(root_id, AccessOptions::DEFAULT)?;
            Ok(root_node.memory_usage)
        }
    }
}
