### Title
Fast Withdrawal Fees Permanently Locked in `BaseWithdrawPool` — No Claiming Mechanism Exists - (File: `core/contracts/BaseWithdrawPool.sol`)

### Summary
`BaseWithdrawPool` collects fast withdrawal fees into a `fees[productId]` mapping on every `submitFastWithdrawal` call, but the contract contains no function to withdraw, redistribute, or reinvest those accumulated fees. The mapping is never decremented. The tokens representing those fees are held by the `WithdrawPool` contract indefinitely with no protocol-accessible claiming path.

### Finding Description
Every call to `submitFastWithdrawal` deducts a fee from the withdrawing user and credits it to `fees[productId]`: [1](#0-0) [2](#0-1) 

The fee is either deducted from `transferAmount` (when `sendTo == msg.sender`) or pulled from the caller via `safeTransferFrom`. Either way, the fee amount in native token decimals is added to `fees[productId]` and the corresponding ERC-20 tokens remain in the `WithdrawPool` contract.

Scanning the entire `BaseWithdrawPool` contract, there is no function that:
- reads `fees[productId]` to transfer tokens out,
- decrements the `fees` mapping, or
- routes accumulated fees to any treasury, engine balance, or protocol account.

The only token-exit function is `removeLiquidity`: [3](#0-2) 

`removeLiquidity` is a generic liquidity-management function. It does not reference `fees`, does not clear the mapping, and does not distinguish fee tokens from liquidity-provider tokens. Calling it does not constitute a proper fee-claiming mechanism and leaves the `fees` mapping permanently out of sync with actual balances.

The analog to the original report is direct: just as the wrapped rebasing token wrapper held Aave/EigenLayer rewards with no `claimAllRewards` call, `WithdrawPool` holds fast-withdrawal fee tokens with no `claimFees` call.

### Impact Explanation
Fast withdrawal fees — real ERC-20 tokens (USDC, WETH, etc.) — accumulate in the `WithdrawPool` contract and cannot be properly claimed, reinvested, or distributed to the protocol treasury. The `fees` mapping grows monotonically and is never decremented, creating a permanent accounting desync between the mapping and the actual contract balance. Protocol revenue from fast withdrawals is effectively lost.

### Likelihood Explanation
`submitFastWithdrawal` is a `public` function callable by any user or liquidity provider. Every fast withdrawal that goes through the pool triggers fee accumulation. On a live exchange with active fast-withdrawal usage, fees will accumulate continuously. No special privilege or attack is required — normal protocol usage is sufficient to cause the issue.

### Recommendation
Add a dedicated `claimFees(uint32 productId, address recipient)` function (owner-gated or endpoint-gated) that:
1. Reads `fees[productId]`.
2. Resets `fees[productId] = 0`.
3. Transfers the corresponding token amount to the protocol treasury or credits it to the `X_ACCOUNT` via the spot engine, consistent with how `claimSequencerFees` and `dumpFees` handle protocol revenue elsewhere. [4](#0-3) [5](#0-4) 

### Proof of Concept
1. Liquidity provider deposits tokens into `WithdrawPool`.
2. User calls `submitFastWithdrawal(idx, transaction, signatures)` with a valid signed withdrawal.
3. `fees[productId] += fee` is executed; ERC-20 tokens remain in the contract.
4. Repeat across many withdrawals — `fees[productId]` grows.
5. No function exists to extract `fees[productId]` tokens. The only exit is `removeLiquidity`, which does not clear `fees` and cannot distinguish fee tokens from LP liquidity, leaving the accounting permanently broken and the fee tokens inaccessible through any proper protocol path. [6](#0-5)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
