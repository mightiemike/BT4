### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables DDA USDC-E Drain Without USDC Provision — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the boolean return value. If the USDC token on Ink Mainnet returns `false` on failure instead of reverting, any unprivileged caller can drain USDC-E from any Direct Deposit Account (DDA) without providing USDC, causing permanent asset loss for the subaccount owner.

---

### Finding Description

`replaceUsdcEWithUsdc` is a permissionless migration function (no `onlyOwner` modifier, only a chain ID gate) that is intended to perform a 1:1 swap: the caller provides USDC and receives USDC-E held in a DDA. [1](#0-0) 

At line 616, the return value of `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` is silently discarded: [2](#0-1) 

If `transferFrom` returns `false` without reverting, execution continues unconditionally to:
1. `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))` — pulls all USDC-E out of the DDA into `ContractOwner`
2. `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` — forwards the USDC-E to the caller

The protocol's own `ERC20Helper` library exists precisely to handle both reverting and non-reverting ERC20 tokens by checking the boolean return value: [3](#0-2) 

`replaceUsdcEWithUsdc` bypasses this wrapper entirely and calls `transferFrom` directly on the raw `IERC20Base` interface, which declares the function as returning `bool`: [4](#0-3) 

This is the exact same vulnerability class as the referenced CToken report: an external token interface method returns a value indicating failure, the return value is ignored, and the protocol continues executing state-changing operations as if the call succeeded.

---

### Impact Explanation

If exploited, the attacker receives the full USDC-E balance of the victim's DDA without depositing any USDC. The DDA is left with zero balance. The subaccount's pending collateral deposit is permanently lost. `ContractOwner` holds no residual USDC-E after the call, so the loss is unrecoverable. The corrupted asset delta is: DDA USDC-E balance → 0, attacker USDC-E balance → +`balance`, USDC never transferred.

---

### Likelihood Explanation

The function is callable by any address on Ink Mainnet (chain 57073) with no access control beyond the chain ID check. [5](#0-4) 

Exploitability depends on whether the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink Chain returns `false` on a failed transfer rather than reverting. Bridged or wrapped USDC deployments on non-mainnet EVM chains frequently exhibit this behavior. The fact that the protocol's own `ERC20Helper.safeTransferFrom` wrapper was written to handle exactly this case — and is used everywhere else in the codebase — makes the omission here a concrete, identifiable deviation from the protocol's own defensive pattern.

---

### Recommendation

Replace the bare `transferFrom` call with the protocol's existing `ERC20Helper.safeTransferFrom` wrapper:

```solidity
// Before (unsafe) — ContractOwner.sol line 616:
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`IERC20Base` already uses `ERC20Helper` via `using ERC20Helper for IERC20Base` in `ContractOwner`, so no import changes are needed. [6](#0-5) 

---

### Proof of Concept

1. Attacker identifies a DDA (`directDepositV1Address[victimSubaccount]`) holding USDC-E on Ink Mainnet (chain 57073).
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(victimSubaccount)` with zero USDC balance or allowance.
3. `IERC20Base(usdc).transferFrom(attacker, dda, balance)` returns `false` — no revert, return value discarded.
4. Execution continues: `DirectDepositV1(dda).withdraw(usdcE)` transfers all USDC-E from the DDA to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, balance)` sends the full USDC-E amount to the attacker.
6. Result: attacker holds USDC-E; victim's DDA is empty; no USDC was ever deposited. [7](#0-6)

```text
    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
