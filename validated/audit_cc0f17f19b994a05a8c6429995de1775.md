Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the actual user, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the router becomes `msg.sender` to the pool, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actual_user]`. If the router is allowlisted — a natural operational step so that allowlisted users can trade via the standard periphery — any unprivileged address can bypass the curated pool's swap allowlist entirely.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim as the first argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the forwarded value: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly, making itself `msg.sender` inside the pool: [4](#0-3) 

The pool therefore passes the router's address as `sender` to `_beforeSwap`, and the extension evaluates `allowedSwapper[pool][router]`. The real user's address is never consulted. No existing guard in the extension, pool, or router corrects this: the router stores the original `msg.sender` only in transient storage for the payment callback (`_setNextCallbackContext`), and that value is never forwarded to the pool's `swap` call or to any extension.

## Impact Explanation
A pool admin deploys a curated pool with `SwapAllowlistExtension` to restrict swaps to a specific set of addresses. If the router is added to the allowlist — the natural operational step so that allowlisted users can trade via the standard periphery — the allowlist is completely voided: any address can call `router.exactInputSingle` and the extension passes because it only sees the router. The wrong value evaluated is `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][attacker]`. The direct consequence is unauthorized swaps executing on a curated pool, constituting a broken core pool access-control mechanism with potential direct loss of funds from the pool.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard, publicly documented swap entry point for the protocol. Pool admins who want their allowlisted users to be able to trade will naturally allowlist the router. The bypass requires no special privileges, no flash loans, and no unusual token behavior — any address can call the router. The precondition (router allowlisted) is the expected operational configuration, making exploitation highly likely once a curated pool is deployed.

## Recommendation
The extension must gate the original user, not the intermediary. Two complementary fixes:

1. **Pass the original user through the router.** The router already stores the original `msg.sender` in transient storage via `_setNextCallbackContext`. It should forward that value as an explicit `sender` argument to `pool.swap`, and the pool should pass that value (rather than its own `msg.sender`) to `_beforeSwap`. This requires a small interface change to `IMetricOmmPoolActions.swap`.

2. **Alternatively, use a trusted-router registry.** The extension checks `sender` (the first argument) only when `msg.sender` (the pool's caller) is a registered trusted router; otherwise it checks `sender` directly. This avoids the interface change but requires a registry of trusted routers in the extension or factory.

The simplest safe fix is option 1: the router passes the original caller as `sender` to the pool, which forwards it to extensions, so `allowedSwapper[pool][original_user]` is evaluated correctly.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` so that allowlisted users can trade via the standard periphery.
3. Attacker (not in the allowlist) calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient, ...)` — pool's `msg.sender` is the router.
5. Pool calls `_beforeSwap(sender=router, ...)`.
6. Extension evaluates `allowedSwapper[pool][router]` → `true` → passes.
7. Attacker's swap executes on the curated pool despite never being allowlisted.

The check that was supposed to block the attacker — `allowedSwapper[pool][attacker]` — is never evaluated.
