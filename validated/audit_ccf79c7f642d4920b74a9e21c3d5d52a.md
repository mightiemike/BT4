### Title
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` Panic When Paused Instead of Returning `false` — (File: `contract/src/lib.rs`)

---

### Summary

Both SPV verification functions are decorated with `#[pause]`, causing them to **panic** (revert) when the contract is paused rather than returning `false`. Consuming NEAR contracts that call these functions to gate fund releases or other actions will receive a failed cross-contract call instead of a clean `false` result, breaking any protocol that integrates with this light client.

---

### Finding Description

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are the primary public API for SPV proof verification. Both are annotated with `#[pause]`: [1](#0-0) [2](#0-1) 

The `#[pause]` macro from `near-plugins` causes the function to **panic** with a "contract is paused" message when the contract is in the paused state (unless the caller holds a bypass role). Neither function has a bypass role configured.

Both functions have the signature `-> bool`, meaning their contract to callers is: return `true` if the transaction is verified, `false` otherwise. When the contract is paused, the correct behavior is to return `false` — verification is disabled — not to panic. This is directly analogous to `PrincipalToken::maxWithdraw` reverting when paused instead of returning `0`.

No other view functions (`get_last_block_header`, `get_block_hash_by_height`, etc.) carry the `#[pause]` decorator; only the two verification functions do, making this an inconsistency specific to the SPV proof path. [3](#0-2) 

---

### Impact Explanation

Any NEAR smart contract (bridge, atomic swap, DApp) that calls `verify_transaction_inclusion_v2` to confirm a BTC transaction before releasing funds will receive a **failed cross-contract call** (panic) whenever the light client is paused. In NEAR's async model, the calling contract's callback receives a failure receipt. If the callback does not explicitly handle the failure case — a reasonable assumption for integrators who expect only `true`/`false` — the calling contract's state machine may be left in an inconsistent state, funds may be locked, or the action may be silently dropped.

The bug corrupts the **proof result** returned to consuming contracts: instead of the expected boolean, they receive an unrecoverable panic, which is a qualitatively different signal. [4](#0-3) 

---

### Likelihood Explanation

The `PauseManager` role exists precisely to pause the contract during security incidents or upgrades. [5](#0-4) 

Pausing is a routine operational action. Any integrating protocol that calls `verify_transaction_inclusion_v2` during a pause window will be broken. The likelihood is **medium**: pauses are infrequent but expected, and the impact on integrators is concrete and immediate.

---

### Recommendation

Remove `#[pause]` from both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2`, and instead add an explicit guard at the top of each function that returns `false` when the contract is paused:

```rust
// pseudo-code
pub fn verify_transaction_inclusion_v2(&self, args: ProofArgsV2) -> bool {
    if self.pa_is_paused(PausedKey::All) {
        return false;
    }
    // ... existing logic
}
```

This mirrors the fix applied to `PrincipalToken::maxWithdraw` and `maxRedeem` in the referenced Spectra audit: replace a panic with a safe sentinel value that consuming contracts can handle gracefully.

---

### Proof of Concept

```rust
// Pseudocode for a NEAR integration test using near-workspaces
#[tokio::test]
async fn test_verify_panics_when_paused() {
    // 1. Deploy and initialize the BtcLightClient contract
    // 2. Submit enough headers so a valid proof exists
    // 3. Grant pause_manager role to an account and call pause()
    // 4. From an unprivileged account, call verify_transaction_inclusion_v2
    //    with a valid proof that would normally return `true`
    let result = contract
        .call("verify_transaction_inclusion_v2")
        .args_borsh(valid_proof_args)
        .view()
        .await;

    // Should return false (paused), but instead panics:
    // "Contract is paused"
    assert!(result.is_err(), "call panicked instead of returning false");
}
```

The entry path is fully unprivileged: any NEAR account can call `verify_transaction_inclusion_v2`. The panic is triggered by the contract's own paused state, set by the `PauseManager`. The bug is in the function's response to that state, not in the pausing action itself. [6](#0-5)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
