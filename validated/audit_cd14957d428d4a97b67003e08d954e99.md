Audit Report

## Title
`SwapAllowlistExtension` checks the intermediary router address instead of the originating user, enabling allowlist bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` — the direct caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the pool passes `sender = router_address` to the extension. If the router is allowlisted (required for any router-mediated swap to work on a curated pool), every unprivileged user can bypass the allowlist by routing through the public router contract, breaking the pool admin's access control invariant.

## Finding Description

In `MetricOmmPool.swap`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
  msg.sender,   // <-- direct caller of pool.swap()
  recipient,
  ...
);
```

`_beforeSwap` in `ExtensionCalling` forwards this value verbatim to every configured extension via `_callExtensionsInOrder`. `SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is the direct pool caller. When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap(...)`, the pool sees `msg.sender = router_address` and passes that as `sender` to the extension. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][originating_user]`.

This creates an irresolvable dilemma for the pool admin:
- **Router NOT allowlisted**: All router-mediated swaps revert, even for allowlisted users.
- **Router allowlisted**: Any unprivileged user bypasses the allowlist by calling `MetricOmmSimpleRouter`.

The router has no access control on who can invoke its swap functions — it is a fully public permissionless contract.

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to gate specific counterparties (e.g., KYC-verified traders, institutional partners) cannot enforce that policy for router-mediated swaps. Any unprivileged user can bypass the allowlist by calling `MetricOmmSimpleRouter`, which is a public contract callable by anyone. This constitutes an **admin-boundary break**: the pool admin's configured access control is bypassed by an unprivileged path. If the pool's risk model depends on counterparty curation (e.g., to prevent adverse selection against LP positions), the bypass directly threatens LP principal. The wrong value is `allowedSwapper[pool][router]` being evaluated instead of `allowedSwapper[pool][originating_user]`.

## Likelihood Explanation

Likelihood is **medium-high**. A pool admin who wants allowlisted users to be able to use the standard periphery entrypoint (`MetricOmmSimpleRouter`) must allowlist the router address. This is the natural and expected configuration. The admin has no mechanism to allowlist the router for specific users only — the check is binary on the router address. The `MetricOmmSimpleRouter` is a public contract callable by anyone with no restrictions on callers.

## Recommendation

The extension must check the **economically relevant actor**, not the intermediary. Two viable approaches:

1. **Pass original user in `extensionData`**: Have the router encode the originating user's address in `extensionData` and have the extension verify it (with a trusted-forwarder pattern or router-signed attestation).
2. **Check `recipient` instead of `sender`**: If the pool's curation intent is to gate who receives output tokens, check the `recipient` argument instead. This changes semantics but avoids the intermediary problem.
3. **Document incompatibility**: Document that pools using `SwapAllowlistExtension` must not allowlist the router, and the router must not be used for such pools. This is a workaround, not a fix.

## Proof of Concept

1. Pool admin deploys pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to allow router-mediated swaps for their allowlisted users.
3. Non-allowlisted attacker calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the curated pool.
4. Router calls `pool.swap(recipient, zeroForOne, amount, priceLimit, "", extensionData)` — pool sees `msg.sender = router`.
5. Pool calls `_beforeSwap(router, recipient, ...)` — extension receives `sender = router`.
6. Extension evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds.
7. Attacker successfully trades on a pool they were never authorized to access.

Foundry test plan: deploy pool with `SwapAllowlistExtension`, allowlist only the router and one legitimate user, call `exactInputSingle` from an unallowlisted EOA, assert the swap succeeds (demonstrating bypass).
