Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of end-user address, allowing allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the extension evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. Any pool admin who allowlists the router to enable router-mediated swaps inadvertently opens the allowlist to every user, voiding the extension's access-control guarantee.

## Finding Description
`MetricOmmPool.swap()` accepts no explicit `sender` parameter and passes `msg.sender` directly to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← always the direct caller of pool.swap()
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension via `abi.encodeCall`. `SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When a user calls any of `MetricOmmSimpleRouter.exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput`, the router calls `IMetricOmmPoolActions(pool).swap(...)` directly, making the router the `msg.sender` inside `pool.swap()`. The extension therefore evaluates `allowedSwapper[pool][router]`, never touching `allowedSwapper[pool][user]`.

Contrast with `DepositAllowlistExtension.beforeAddLiquidity`, which checks `allowedDepositor[msg.sender][owner]` where `owner` is an explicit first-class parameter of `addLiquidity` that the router can set to the actual user. `pool.swap()` has no equivalent parameter, so there is no correct way for the router to forward the original user's identity under the current interface.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` is intended to restrict trading to a curated set of addresses (e.g., KYC-verified counterparties or whitelisted market makers). Allowlisting the router — the only way to enable router-mediated swaps on such a pool — silently voids the allowlist: every user who calls the router can trade regardless of whether their own address is on the list. Non-allowlisted users can execute swaps and drain liquidity from LP positions that were deposited under the assumption that only vetted counterparties could trade. This constitutes direct loss of user principal and broken core pool functionality (the allowlist extension's sole purpose is defeated).

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical periphery swap entrypoint. A pool admin who deploys a `SwapAllowlistExtension` pool and wants users to access it through the router has no choice but to call `setAllowedToSwap(pool, router, true)`. This is the standard operational setup, not an exotic configuration. Any non-allowlisted user who discovers the router path can exploit it immediately without any privileged access or special preconditions.

## Recommendation
Add an explicit `sender` parameter to `MetricOmmPool.swap()` (analogous to `owner` in `addLiquidity`) so that the router can forward the original user's address. The pool would pass this explicit `sender` to `_beforeSwap` instead of `msg.sender`. The router would supply `msg.sender` (the end user) as this parameter. This mirrors the `owner` pattern already used in `addLiquidity` and relied upon correctly by `DepositAllowlistExtension`.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` as a `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps.
3. Pool admin does **not** call `setAllowedToSwap(pool, attacker, true)`.
4. Attacker calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the pool.
5. Router calls `pool.swap(recipient, zeroForOne, amount, priceLimit, "", extensionData)` with `msg.sender = router`.
6. Pool calls `_beforeSwap(msg.sender=router, ...)` → extension receives `sender = router`.
7. Extension evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds.
8. Attacker successfully swaps on a pool they were never authorized to access; `allowedSwapper[pool][attacker]` is never evaluated. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
