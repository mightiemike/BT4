Audit Report

## Title
`SwapAllowlistExtension` Allowlist Bypassed via Router: Any User Can Swap in Allowlisted Pools - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which is the direct caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the router is allowlisted (a necessary precondition for any allowlisted user to use the router), every user — including non-allowlisted ones — can bypass the per-user gate by calling through the router.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly, making itself the `msg.sender` seen by the pool and extension: [4](#0-3) 

The same applies to `exactOutputSingle`: [5](#0-4) 

The call chain is:
```
User → Router.exactInputSingle()
     → Pool.swap(msg.sender = Router)
     → Extension.beforeSwap(sender = Router)
     → allowedSwapper[pool][Router]  ← checked, NOT the end user
```

A pool admin who wants to restrict swaps to specific KYC'd or whitelisted users adds those user addresses to `allowedSwapper`. To also allow those users to trade through the router, the admin must add the router to `allowedSwapper`. Once the router is allowlisted, **any** user — including non-allowlisted ones — can bypass the gate by calling `router.exactInputSingle`, `router.exactInput`, `router.exactOutputSingle`, or `router.exactOutput`. The admin cannot simultaneously allow allowlisted users to use the router and block non-allowlisted users from using it. There are no additional guards in the router or pool that check the originating user identity.

## Impact Explanation
Any user can trade in a pool whose admin intended to restrict access to a specific set of addresses. If the pool is a private liquidity venue (e.g., restricted to institutional counterparties or KYC'd users), non-allowlisted users can extract value from LPs by trading at oracle-derived prices the pool was not intended to offer them. The allowlist guard — the only on-chain mechanism for access control on swaps — is rendered ineffective for all router-mediated flows. This constitutes a broken core pool access-control functionality causing potential loss of funds to LPs and violation of the admin-boundary invariant.

## Likelihood Explanation
The bypass requires only that the router is allowlisted for the pool, which is a necessary precondition for any allowlisted user to use the router at all. Any user who knows the router address can exploit this immediately with no special privileges, capital, or timing. `MetricOmmSimpleRouter` is a public, permissionless contract. The exploit is repeatable on every swap through the router.

## Recommendation
The `SwapAllowlistExtension` should gate on the end user, not the immediate caller of `pool.swap()`. Two options:

1. **Pass the original user through `extensionData`**: the router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires router cooperation and is trust-dependent.
2. **Add a separate `originator` field to the hook interface**: the pool passes both `msg.sender` (the direct caller) and an optional originator so extensions can gate on the true economic actor.

The simplest safe default is to document that `SwapAllowlistExtension` gates on the **direct caller of `pool.swap()`**, not the end user, and that allowlisting the router opens the gate to all users. Pool admins who need per-user access control must not allowlist the router and must require users to call the pool directly.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured as `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is allowed.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` — router is allowed so Alice can use it.
4. Bob (not allowlisted) calls `router.exactInputSingle({pool: pool, tokenIn: ..., ...})`.
5. Router calls `pool.swap(msg.sender = router)`.
6. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
7. Bob successfully swaps in a pool he was supposed to be blocked from.
