Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the End User, Allowing Any User to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` at the pool, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. This makes the allowlist ungatable: allowlisting the router to enable legitimate users simultaneously opens the pool to every user on the network.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the first argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← whoever called pool.swap()
  recipient, ...
);
```

`ExtensionCalling._beforeSwap` forwards that value unchanged as `sender` to every configured extension via `abi.encodeCall`:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter.exactInputSingle` is used, the router calls `pool.swap()` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ...);
```

The end user's address is stored in transient storage via `_setNextCallbackContext` for the payment callback only — it is never passed to `pool.swap()` in any position that reaches the extension. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The `DepositAllowlistExtension` avoids this by keying on `owner` (an explicit position-owner argument independent of the call chain), but no equivalent separation exists for the swap path.

## Impact Explanation
A pool admin deploying `SwapAllowlistExtension` to restrict swaps to a curated set faces an inescapable dilemma. If the router is not allowlisted, allowlisted users cannot use `MetricOmmSimpleRouter` at all, breaking the primary documented swap entry point. If the router is allowlisted to restore usability for legitimate users, every address on the network can call the router and bypass the allowlist entirely, executing swaps at oracle-determined prices against pool liquidity without authorization. There is no configuration that simultaneously preserves router usability for allowlisted users and blocks non-allowlisted users. This constitutes broken core pool functionality and an admin-boundary break where an unprivileged path bypasses a configured access control.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap entry point. Any user who calls `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` through the router triggers the bypass automatically, with no privileged access, special tokens, or setup required. The bypass is immediate and repeatable once the router is allowlisted for any legitimate user's benefit.

## Recommendation
Pass the original end-user address through the swap path so the extension can gate on it. The preferred fix is to add an explicit `swapper` (or `originator`) field to the swap parameters that the router populates with `msg.sender` before calling the pool, and forward it to extension hooks as a separate argument distinct from `sender`. Alternatively, document that `SwapAllowlistExtension` gates only the direct pool caller and require curated pools to be accessed directly rather than through the router, with this restriction made explicit in the extension interface and NatDoc.

## Proof of Concept
```
Setup:
  Pool P configured with SwapAllowlistExtension E
  Pool admin allowlists only Alice: allowedSwapper[P][Alice] = true
  Router R is NOT allowlisted

Step 1 — Alice tries router:
  Alice calls MetricOmmSimpleRouter.exactInputSingle({pool: P, ...})
  Router calls P.swap(recipient=Alice, ...)  ← msg.sender at pool = Router R
  Extension checks allowedSwapper[P][R] → false → reverts
  Alice cannot use the router despite being allowlisted.

Admin workaround — allowlist the router:
  allowedSwapper[P][R] = true  (so Alice can use the router)

Step 2 — Bob (never allowlisted) exploits:
  Bob calls MetricOmmSimpleRouter.exactInputSingle({pool: P, ...})
  Router calls P.swap(recipient=Bob, ...)  ← msg.sender at pool = Router R
  Extension checks allowedSwapper[P][R] → true → bypass succeeds
  Bob swaps freely against the curated pool.

Foundry test outline:
  1. Deploy pool P with SwapAllowlistExtension E.
  2. vm.prank(poolAdmin); E.setAllowedToSwap(P, alice, true);
  3. vm.prank(poolAdmin); E.setAllowedToSwap(P, address(router), true);
  4. vm.prank(bob); router.exactInputSingle({pool: P, ...});
  5. Assert swap succeeds for Bob despite Bob not being in allowedSwapper.
```
