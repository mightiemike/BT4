### Title
`get_protocol_config` Omits `num_chunk_validator_seats`, `num_chunk_producer_seats`, and `chunk_producer_assignment_changes_limit` from Epoch-Config Sync, Causing `EXPERIMENTAL_protocol_config` RPC to Return Stale Genesis Values - (File: chain/chain/src/runtime/mod.rs)

---

### Summary

`NightshadeRuntime::get_protocol_config` manually copies a subset of `EpochConfig` fields into a cloned `GenesisConfig`, but silently omits `num_chunk_producer_seats`, `num_chunk_validator_seats`, and `chunk_producer_assignment_changes_limit`. These three fields are versioned per protocol upgrade (e.g., `num_chunk_validator_seats` was raised from 300 → 500 at mainnet protocol version 80), yet the `EXPERIMENTAL_protocol_config` RPC response permanently reflects the genesis-time values for them. Additionally, `ProtocolConfigView` — the serialized RPC type — does not declare these fields at all, so they are entirely absent from every response regardless of the epoch queried.

---

### Finding Description

`get_protocol_config` in `chain/chain/src/runtime/mod.rs` constructs a `ProtocolConfig` by cloning `self.genesis_config` and then overwriting it field-by-field with live `EpochConfig` values:

```rust
genesis_config.epoch_length = epoch_config.epoch_length;
genesis_config.num_block_producer_seats = epoch_config.num_block_producer_seats;
genesis_config.block_producer_kickout_threshold = epoch_config.block_producer_kickout_threshold;
// ... 10 more fields ...
genesis_config.max_inflation_rate = epoch_config.max_inflation_rate;
``` [1](#0-0) 

Three `EpochConfig` fields that are present in both `EpochConfig` and `GenesisConfig` are never written back:

| `EpochConfig` field | `GenesisConfig` field | Copied? |
|---|---|---|
| `num_chunk_producer_seats` | `num_chunk_producer_seats` | **No** |
| `num_chunk_validator_seats` | `num_chunk_validator_seats` | **No** |
| `chunk_producer_assignment_changes_limit` | `chunk_producer_assignment_changes_limit` | **No** |

Compare with the symmetric `From<&GenesisConfig> for EpochConfig` conversion, which **does** copy all three: [2](#0-1) 

The resulting `ProtocolConfig` is then converted to `ProtocolConfigView` for the RPC response. `ProtocolConfigView` does not declare `num_chunk_producer_seats`, `num_chunk_validator_seats`, or `chunk_producer_assignment_changes_limit` as fields at all: [3](#0-2) 

The `From<ProtocolConfig> for ProtocolConfigView` conversion therefore never serializes these fields: [4](#0-3) 

The `EXPERIMENTAL_protocol_config` RPC handler passes the result directly to the caller: [5](#0-4) 

---

### Impact Explanation

**Stale / absent validator-set parameters in the public RPC.** Any consumer of `EXPERIMENTAL_protocol_config` — indexers, wallets, bridges, monitoring tools — receives a response that:

1. **Omits** `num_chunk_validator_seats`, `num_chunk_producer_seats`, and `chunk_producer_assignment_changes_limit` entirely (they are not fields of `ProtocolConfigView`).
2. Even if the view were extended, the underlying `genesis_config` values would still be wrong because `get_protocol_config` never overwrites them from `epoch_config`.

**Concrete divergence on mainnet.** Protocol version 80 raised `num_chunk_validator_seats` from 300 to 500 on both mainnet and testnet: [6](#0-5) 

Any `EXPERIMENTAL_protocol_config` query for a block at protocol version ≥ 80 silently returns the genesis value (300) instead of the live value (500). The field is not even present in the JSON response, so callers cannot distinguish "not upgraded yet" from "field missing."

`chunk_producer_assignment_changes_limit` and `num_chunk_producer_seats` are similarly affected whenever their epoch-config values diverge from genesis.

---

### Likelihood Explanation

The bug is deterministic and affects every call to `EXPERIMENTAL_protocol_config` on mainnet and testnet for any block at protocol version ≥ 80. No special conditions are required. Any unprivileged user can trigger it by issuing a standard JSON-RPC request.

---

### Recommendation

In `get_protocol_config` (`chain/chain/src/runtime/mod.rs`), add the three missing assignments after the existing block:

```rust
genesis_config.num_chunk_producer_seats = epoch_config.num_chunk_producer_seats;
genesis_config.num_chunk_validator_seats = epoch_config.num_chunk_validator_seats;
genesis_config.chunk_producer_assignment_changes_limit =
    epoch_config.chunk_producer_assignment_changes_limit;
```

Additionally, add `num_chunk_producer_seats`, `num_chunk_validator_seats`, and `chunk_producer_assignment_changes_limit` as fields to `ProtocolConfigView` in `core/chain-configs/src/genesis_config.rs` and populate them in the `From<ProtocolConfig> for ProtocolConfigView` conversion.

To prevent future regressions, a compile-time exhaustiveness check (e.g., a `let _ = EpochConfig { .. }` destructure inside `get_protocol_config`) would force the function to be updated whenever a new field is added to `EpochConfig`.

pub struct ProtocolConfigView {
    /// Current Protocol Version
    pub protocol_version: ProtocolVersion,
    /// Official time of blockchain start.
    pub genesis_time: DateTime<Utc>,
    /// ID of the blockchain. This must be unique for every blockchain.
    /// If your testnet blockchains do not have unique chain IDs, you will have a bad time.
    pub chain_id: String,
    /// Height of genesis block.
    pub genesis_height: BlockHeight,
    /// Number of block producer seats at genesis.
    pub num_block_producer_seats: NumSeats,
    /// Enable dynamic re-sharding.
    pub dynamic_resharding: bool,
    /// Threshold of stake that needs to indicate that they ready for upgrade.
    #[cfg_attr(feature = "schemars", schemars(with = "Rational32SchemarsProvider"))]
    pub protocol_upgrade_stake_threshold: Rational32,
    /// Epoch length counted in block heights.
    pub epoch_length: BlockHeightDelta,
    /// Initial gas limit.
    pub gas_limit: Gas,
    /// Minimum gas price. It is also the initial gas price.
    pub min_gas_price: Balance,
    /// Maximum gas price.
    pub max_gas_price: Balance,
    /// Threshold for kicking out block producers, between 0 and 100.
    pub block_producer_kickout_threshold: u8,
    /// Threshold for kicking out chunk producers, between 0 and 100.
    pub chunk_producer_kickout_threshold: u8,
    /// Threshold for kicking out nodes which are only chunk validators, between 0 and 100.
    pub chunk_validator_only_kickout_threshold: u8,
    /// Number of target chunk validator mandates for each shard.
    pub target_validator_mandates_per_shard: NumSeats,
    /// Online minimum threshold below which validator doesn't receive reward.
    #[cfg_attr(feature = "schemars", schemars(with = "Rational32SchemarsProvider"))]
    pub online_min_threshold: Rational32,
    /// Online maximum threshold above which validator gets full reward.
    #[cfg_attr(feature = "schemars", schemars(with = "Rational32SchemarsProvider"))]
    pub online_max_threshold: Rational32,
    /// Gas price adjustment rate
    #[cfg_attr(feature = "schemars", schemars(with = "Rational32SchemarsProvider"))]
    pub gas_price_adjustment_rate: Rational32,
    /// Runtime configuration (mostly economics constants).
    pub runtime_config: RuntimeConfigView,
    /// Number of blocks for which a given transaction is valid
    pub transaction_validity_period: NumBlocks,
    /// Protocol treasury rate
    #[cfg_attr(feature = "schemars", schemars(with = "Rational32SchemarsProvider"))]
    pub protocol_reward_rate: Rational32,
    /// Maximum inflation on the total supply every epoch.
    #[cfg_attr(feature = "schemars", schemars(with = "Rational32SchemarsProvider"))]
    pub max_inflation_rate: Rational32,
    /// Expected number of blocks per year
    pub num_blocks_per_year: NumBlocks,
    /// Protocol treasury account
    pub protocol_treasury_account: AccountId,
    /// Fishermen stake threshold.
    pub fishermen_threshold: Balance,
    /// The minimum stake required for staking is last seat price divided by this number.
    pub minimum_stake_divisor: u64,
    /// Max stake percentage of the validators we will kick out.
    pub max_kickout_stake_perc: u8,
    /// The lowest ratio s/s_total any block producer can have.
    /// See <https://github.com/near/NEPs/pull/167> for details
    #[cfg_attr(feature = "schemars", schemars(with = "Rational32SchemarsProvider"))]
    pub minimum_stake_ratio: Rational32,
    /// If true, shuffle the chunk producers across shards. In other words, if
    /// the shard assignments were `[S_0, S_1, S_2, S_3]` where `S_i` represents
    /// the set of chunk producers for shard `i`, if this flag were true, the
    /// shard assignments might become, for example, `[S_2, S_0, S_3, S_1]`.
    pub shuffle_shard_assignment_for_chunk_producers: bool,
    /// The minimum number of validators each shard must have
    pub minimum_validators_per_shard: NumSeats,
    /// Layout information regarding how to split accounts to shards
    pub shard_layout: ShardLayout,
}
