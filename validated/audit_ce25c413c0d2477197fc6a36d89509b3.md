Audit Report

## Title
Absolute `executeAfter` timestamps allow pool admin to bypass a newly-increased timelock — (`File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`OracleValueStopLossExtension` timelocks stop-loss parameter changes so LPs can exit before harmful settings take effect. However, every `propose*` function records an absolute `executeAfter = block.timestamp + currentTimelock` at proposal time. If the admin simultaneously proposes a harmful drawdown change and a timelock increase, both proposals inherit the old short deadline. LPs observing the timelock-increase proposal may conclude they have the new longer window to react, but the harmful drawdown change executes after only the original short window — effectively bypassing the protection the new timelock was meant to provide.

## Finding Description

`_afterTimelock` computes the absolute execution deadline using the timelock value at the moment of proposal:

```solidity
// L297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

Every `propose*` function stores this absolute value:

- `proposeOracleStopLossDrawdown` (L106-108): stores `pendingDrawdownExecuteAfter`
- `proposeOracleStopLossTimelock` (L80-82): stores `pendingTimelockExecuteAfter`
- `proposeOracleStopLossDecay` (L133-135): stores `pendingDecayExecuteAfter`
- `proposeOracleStopLossHighWatermarks` (L162-164): stores `PendingHighWatermarks.executeAfter`

Each corresponding `execute*` function checks only the stored absolute value and never re-derives the deadline from the current timelock:

```solidity
// L114-115
if (sched.pendingDrawdownExecuteAfter == 0) revert OracleStopLossNoPendingDrawdown(pool_);
_requireElapsed(sched.pendingDrawdownExecuteAfter);
```

**Exploit path:**

1. Pool is deployed with `timelock = 1 day`.
2. Admin calls `proposeOracleStopLossDrawdown(pool, 1_000_000)` — sets `pendingDrawdownExecuteAfter = T + 1 day`.
3. Admin simultaneously calls `proposeOracleStopLossTimelock(pool, 30 days)` — sets `pendingTimelockExecuteAfter = T + 1 day`.
4. LPs observe the timelock-increase proposal and conclude future changes carry 30-day protection; they do not urgently exit.
5. At `T + 1 day`, admin calls `executeOracleStopLossTimelock` (timelock → 30 days) and `executeOracleStopLossDrawdown` (drawdown → 1,000,000).
6. Stop-loss is now disabled. The drawdown change had only 1 day of real LP warning, not 30.

Existing guards are insufficient: `_requireElapsed` (L301-303) only compares `block.timestamp` against the stored absolute value; there is no mechanism to re-check against the current timelock at execution time.

## Impact Explanation

Setting `drawdownE6 = E6 = 1_000_000` causes `floorMultiplier = E6 - drawdown = 0` (L234). Inside `_applyWatermark`, the breach condition becomes `metric < (hwm * 0) / E6 = 0`, which is always false (L334). The stop-loss is permanently silenced: `OracleStopLossTriggered` is never emitted, adversarial swaps can drain value from every bin without triggering a revert, and LP principal is directly at risk. This is a fund-impacting admin-boundary break — the pool admin exceeds the constraint the timelock was designed to enforce.

## Likelihood Explanation

The attack requires only two simultaneous `propose*` calls followed by two `execute*` calls after the old short window. No external oracle manipulation, no special token behavior, and no role beyond pool admin is needed. Any pool deployed with a short initial timelock is permanently exposed to this race. The pool admin is semi-trusted; the timelock exists precisely to constrain admin power, and this path bypasses it.

## Recommendation

Store the proposal timestamp instead of the absolute deadline, and recompute `executeAfter` from the **current** timelock at execution time:

```solidity
// Replace pendingDrawdownExecuteAfter with pendingDrawdownProposedAt in PoolStopLossSchedule.

// In proposeOracleStopLossDrawdown:
sched.pendingDrawdownProposedAt = uint32(block.timestamp);

// In executeOracleStopLossDrawdown:
uint32 executeAfter = sched.pendingDrawdownProposedAt
    + oracleStopLossConfig[pool_].timelock;  // uses CURRENT timelock
_requireElapsed(executeAfter);
```

Apply the same change to the decay, watermark, and timelock-change schedulers. This ensures any increase to the governing timelock is automatically honoured for all in-flight proposals.

## Proof of Concept

```solidity
function testTimelockBypassViaAbsoluteExecuteAfter() public {
    vm.prank(factory);
    extension.initialize(address(pool),
        abi.encode(uint32(100_000), uint32(58), uint32(1 days)));

    vm.startPrank(admin);
    extension.proposeOracleStopLossDrawdown(address(pool), 1_000_000);
    extension.proposeOracleStopLossTimelock(address(pool), uint32(30 days));
    vm.stopPrank();

    vm.warp(block.timestamp + 1 days);

    vm.startPrank(admin);
    extension.executeOracleStopLossTimelock(address(pool));
    extension.executeOracleStopLossDrawdown(address(pool));
    vm.stopPrank();

    (uint32 drawdown,,,) = extension.oracleStopLossConfig(address(pool));
    assertEq(drawdown, 1_000_000); // stop-loss disabled after only 1 day
}
```
