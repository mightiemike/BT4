Audit Report

## Title
SwapAllowlistExtension gates router address instead of actual swapper, enabling allowlist bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the user. If the pool admin allowlists the router (a natural operational step to support router-mediated swaps), any unprivileged user can bypass the allowlist entirely by calling the public router. The `DepositAllowlistExtension` has a symmetric flaw: it ignores `sender` and checks the user-supplied `owner`, allowing a non-allowlisted caller to pass an allowlisted `owner` address and bypass the deposit gate.

## Finding Description

**Swap allowlist bypass:**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`, passing the immediate caller as `sender`: [1](#0-0) 

`_beforeSwap` encodes `sender` and forwards it to each configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` — where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap(...)` directly, making the router the `msg.sender` at the pool level: [4](#0-3) 

The extension therefore checks whether the **router** is allowlisted, not the actual user. If the admin allowlists the router to support standard UX, the allowlist is completely ineffective for all router-mediated swaps.

**Deposit allowlist bypass (secondary):**

`DepositAllowlistExtension.beforeAddLiquidity` ignores the `sender` argument (unnamed, discarded) and checks only `owner`, which is user-supplied: [5](#0-4) 

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-controlled `owner` parameter separately: [6](#0-5) 

A non-allowlisted caller can supply any allowlisted address as `owner` and pass the gate, while remaining the actual payer via the liquidity callback.

## Impact Explanation

A pool admin deploys a pool with `SwapAllowlistExtension` to restrict trading to a curated set (e.g., KYC-gated or institutional pool). Allowlisting the `MetricOmmSimpleRouter` — a natural operational step to let allowlisted users trade via the standard interface — causes the extension to pass for **any** caller of the public router. This breaks the core access-control invariant: unauthorized parties can execute swaps that drain pool reserves at oracle-quoted prices, causing bad-price execution against LPs who deposited under the assumption of a restricted counterparty set. This matches the **Admin-boundary break** and **Broken core pool functionality causing loss of funds** impact categories.

## Likelihood Explanation

The trigger is fully unprivileged: any address with tokens can call `MetricOmmSimpleRouter.exactInputSingle()` or `exactInput()` targeting the allowlist-protected pool. No special role, malicious setup, or non-standard token is required. The only precondition — that the pool admin has allowlisted the router — is a natural and expected operational step. This is a one-transaction bypass, repeatable indefinitely.

## Recommendation

`SwapAllowlistExtension.beforeSwap` must check the **original user identity**, not the intermediary router. Two viable approaches:

1. **Require routers to forward the original caller via `extensionData`**: the router encodes `msg.sender` into `extensionData` before calling the pool; the extension decodes and verifies that address against the allowlist. This requires a trusted router registry or a signed-caller pattern.
2. **Do not allowlist the router address**: instead, require each allowlisted user to call the pool directly. Document this constraint explicitly so pool admins do not inadvertently allowlist the router.

For `DepositAllowlistExtension`, the hook should check `sender` (the actual caller/payer) rather than `owner` (the position recipient), or check both, to prevent a non-allowlisted caller from depositing by supplying an allowlisted `owner`.

## Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension configured.
2. Pool admin calls setAllowedToSwap(pool, MetricOmmSimpleRouter, true).
3. Pool admin does NOT allowlist attacker's EOA.
4. Attacker calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...}).
5. Router calls pool.swap(...) — msg.sender at pool = router address.
6. _beforeSwap passes sender = router to SwapAllowlistExtension.beforeSwap.
7. Extension checks allowedSwapper[pool][router] == true → passes.
8. Attacker's swap executes on the allowlist-protected pool.
9. Attacker receives tokens; pool LPs suffer unauthorized price impact.

For DepositAllowlistExtension:
1. Deploy pool with DepositAllowlistExtension configured.
2. Pool admin allowlists address A as a depositor.
3. Attacker (not allowlisted) calls pool.addLiquidity(owner=A, ...).
4. beforeAddLiquidity checks allowedDepositor[pool][A] == true → passes.
5. Attacker is the actual payer via the liquidity callback; A receives the position.
```
