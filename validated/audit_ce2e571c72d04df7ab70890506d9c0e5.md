### Title
Division-by-Zero in `PerpEngine.socializeSubaccount` When `openInterest == 0` Permanently Blocks Finalization — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when socializing a residual negative `vQuoteBalance`. If `openInterest` is zero at the time of the call, `MathSD21x18.div` reverts with `"DBZ"`, permanently blocking `_finalizeSubaccount` and trapping the liquidatee in an unliquidatable state while the protocol absorbs unrecoverable bad debt.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, after insurance is exhausted, the code attempts to spread the remaining loss across all open-interest holders:

```solidity
if (balance.vQuoteBalance < 0) {
    // socialize across all other participants
    int128 fundingPerShare = -balance.vQuoteBalance.div(
        state.openInterest   // ← no zero-guard
    );
``` [1](#0-0) 

`MathSD21x18.div` explicitly reverts on a zero denominator:

```solidity
function div(int128 x, int128 y) internal pure returns (int128) {
    unchecked {
        require(y != 0, ERR_DIV_BY_ZERO);
``` [2](#0-1) 

`state.openInterest` is the sum of `abs(balance.amount)` across all subaccounts for a given product. It reaches zero whenever every participant has closed their position (`balance.amount == 0`). Crucially, closing a position zeroes `amount` but **does not zero `vQuoteBalance`** — accumulated PnL (including negative PnL from funding or adverse price moves) persists in `vQuoteBalance` after the position is closed. [3](#0-2) 

`_finalizeSubaccount` is only reachable when `txn.productId == type(uint32).max`. It **requires** `balance.amount == 0` for every perp product before proceeding, then calls `perpEngine.socializeSubaccount`: [4](#0-3) [5](#0-4) 

Because `_finalizeSubaccount` mandates `amount == 0` for all perps, the liquidatee's own contribution to `openInterest` is already zero. If all other participants have also closed their positions, `openInterest == 0` for that product, and the division reverts.

The protocol's own `updateStates` already guards against this exact condition with `if (state.openInterest == 0) { continue; }`, confirming the zero-OI state is a known, reachable condition — but `socializeSubaccount` has no equivalent guard. [6](#0-5) 

---

### Impact Explanation

- The `liquidateSubaccountImpl` call reverts every time it is attempted for this subaccount, permanently blocking finalization.
- The negative `vQuoteBalance` is never cleared; the protocol holds bad debt that can never be socialized.
- The subaccount remains in an unhealthy state indefinitely, leaving the protocol undercollateralized.
- This matches the Critical scope: solvency/accounting failure in the liquidation path that leaves the protocol undercollateralized.

---

### Likelihood Explanation

The condition is reachable in normal operation:
- A low-liquidity or newly-launched perp product where only one or a few subaccounts ever traded.
- A product being wound down where all other participants have exited.
- A subaccount that accumulated negative `vQuoteBalance` via funding payments while holding a position, then closed the position (amount → 0) before becoming unhealthy.

No privileged access, governance capture, or external dependency failure is required.

---

### Recommendation

Add a zero-check for `state.openInterest` before the division in `socializeSubaccount`. When `openInterest == 0`, there are no other participants to socialize the loss against; the loss should be absorbed entirely by the insurance fund (or written off), and `vQuoteBalance` should be zeroed unconditionally:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; write off the loss
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
``` [7](#0-6) 

---

### Proof of Concept

1. Deploy the protocol on a local Hardhat fork.
2. Create a perp product (e.g., `productId = 2`).
3. Have subaccount A open a long position; subaccount B open a short position (to create open interest).
4. Apply funding payments so subaccount A accumulates a negative `vQuoteBalance`.
5. Have subaccount A close its position (`amount → 0`, `vQuoteBalance` remains negative).
6. Have subaccount B close its position (`amount → 0`). Now `state.openInterest == 0`.
7. Drain subaccount A's spot quote balance so its maintenance health < 0.
8. Call `liquidateSubaccountImpl` with `productId = type(uint32).max` targeting subaccount A.
9. Observe the transaction reverts with `"DBZ"`.
10. Repeat step 8 — it reverts every time, permanently blocking finalization.
