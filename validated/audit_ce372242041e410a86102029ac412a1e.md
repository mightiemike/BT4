### Title
Stale Off-Chain `nlpPoolRebalanceX18` Accepted Without Proportional Weight Validation in `mintNlp`/`burnNlp` — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary
`Clearinghouse.mintNlp` and `Clearinghouse.burnNlp` accept `nlpPoolRebalanceX18` as off-chain computed calldata. The sole on-chain guard, `_validateNlpRebalance`, checks only that the array sum equals the quote delta — it never verifies that each element is proportional to the corresponding pool's `balanceWeightX18`. Because the user signs the full `SignedMintNlp`/`SignedBurnNlp` struct (including `nlpPoolRebalanceX18`) before the sequencer executes it, the sequencer cannot alter those values after signing. Any pool-state change that occurs between signing and execution — or any intentional skew chosen by the signer — is applied verbatim, corrupting NLP pool accounting.

---

### Finding Description

`SignedMintNlp` and `SignedBurnNlp` each carry two off-chain computed fields outside the core `tx` body:

```solidity
struct SignedMintNlp {
    MintNlp tx;          // sender, quoteAmount, nonce  ← what the user commits to
    bytes   signature;
    int128  oraclePriceX18;        // off-chain computed
    int128[] nlpPoolRebalanceX18;  // off-chain computed
}
```

The user's EIP-712 signature covers the entire `transaction` byte blob, which includes `oraclePriceX18` and `nlpPoolRebalanceX18`. Once signed, the sequencer cannot modify these values without invalidating the signature. The sequencer therefore executes whatever distribution the signer embedded.

On-chain, `_validateNlpRebalance` enforces only one invariant:

```solidity
require(deltaQuoteAmount == rebalanceAmount, ERR_INVALID_NLP_REBALANCE);
```

There is no check that `nlpPoolRebalanceX18[i]` is proportional to `nlpPools[i].balanceWeightX18`. `_applyNlpRebalance` then blindly credits each pool subaccount with the caller-supplied delta:

```solidity
spotEngine.updateBalance(QUOTE_PRODUCT_ID, nlpPools[i].subaccount, nlpPoolRebalanceX18[i]);
```

Two concrete triggers exist:

**Trigger 1 — Natural staleness (no malice required):** Multiple users submit `MintNlp` transactions concurrently. Each computes `nlpPoolRebalanceX18` based on pool balances at signing time. The sequencer processes them in order. After the first transaction executes, pool balances have shifted; every subsequent transaction carries a distribution that was correct at signing time but is now wrong relative to the current state. The sequencer cannot retroactively fix the signed values.

**Trigger 2 — Intentional skew by a pool owner:** A user who owns NLP pool `P1` (registered via `nlpSigners`) signs a `MintNlp` with `nlpPoolRebalanceX18 = [quoteAmount, 0, 0, …]`, directing the entire incoming quote to `P1`. `_validateNlpRebalance` passes (sum == `quoteAmount`, all ≥ 0). `P1` receives the full deposit; other pools receive nothing. The user later extracts the excess via `nlpProfitShare`.

After `mintNlp` applies the rebalance, it checks only `txn.sender`'s health — not the health of any pool subaccount:

```solidity
require(getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0, ERR_SUBACCT_HEALTH);
```

A pool subaccount that is under-funded by the skewed distribution can silently become unhealthy with no on-chain revert.

---

### Impact Explanation
NLP pool subaccounts receive incorrect quote allocations. Over-funded pools accumulate quote that belongs to other pools; under-funded pools may fall below health thresholds. Because `mintNlp` does not health-check pool subaccounts, the imbalance is not caught at execution time. This corrupts the pool composition invariant, distorts NLP token valuation (which is derived from aggregate pool value), and enables a pool owner to siphon quote from peer pools via `nlpProfitShare` after inflating their own pool's balance.

---

### Likelihood Explanation
Trigger 1 (natural staleness) occurs whenever two or more `MintNlp`/`BurnNlp` transactions are in-flight simultaneously — a routine condition on any active exchange. Trigger 2 requires the attacker to own an NLP pool (`nlpSigners` entry), which is granted by the owner but is a persistent, reusable capability once granted. Both triggers require only a standard user-signed sequencer transaction with no privileged access.

---

### Recommendation
Compute `nlpPoolRebalanceX18` on-chain at execution time using the live `balanceWeightX18` values stored in `nlpPools`, rather than accepting caller-supplied values. This eliminates both staleness and intentional skew. If off-chain computation must be retained for gas reasons, add an on-chain proportionality check:

```solidity
uint128 totalWeight = 0;
for (uint i = 0; i < nlpPools.length; i++) totalWeight += nlpPools[i].balanceWeightX18;
for (uint i = 0; i < nlpPoolRebalanceX18.length; i++) {
    int128 expected = deltaQuoteAmount.mul(int128(nlpPools[i].balanceWeightX18)).div(int128(totalWeight));
    require(abs(nlpPoolRebalanceX18[i] - expected) <= TOLERANCE, ERR_INVALID_NLP_REBALANCE);
}
```

Additionally, add a post-rebalance health check for every pool subaccount inside `mintNlp` and `burnNlp`, mirroring the check already present in `forceRebalanceNlpPool`.

---

### Proof of Concept

**Setup:** Two NLP pools — `P1` (`balanceWeightX18 = 0.5e18`) and `P2` (`balanceWeightX18 = 0.5e18`). Attacker owns `P1`.

1. Attacker constructs `SignedMintNlp` with `quoteAmount = 1000e18` and `nlpPoolRebalanceX18 = [1000e18, 0]`.
2. `_validateNlpRebalance` passes: `sum(1000e18, 0) == 1000e18` ✓; all values `>= 0` ✓.
3. `_applyNlpRebalance` credits `P1` with `+1000e18` quote and `P2` with `0`.
4. `mintNlp` health-checks only `txn.sender` — pool subaccount health is never verified.
5. `P2` is now under-funded by `500e18` quote relative to its weight; `P1` is over-funded by `500e18`.
6. Attacker calls `nlpProfitShare` (via `ContractOwner`) to extract the excess from `P1` to their own subaccount.

        IEndpoint.NlpPool[] calldata nlpPools,

        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {

        require(deltaQuoteAmount == rebalanceAmount, ERR_INVALID_NLP_REBALANCE);
    }
