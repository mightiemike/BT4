Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Actual User, Allowing Allowlist Bypass via Router - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which equals `msg.sender` of the pool's `swap` call — the router's address when users route through `MetricOmmSimpleRouter`. If a pool admin allowlists the router (required for any permitted user to use the standard router), every user including explicitly disallowed ones can bypass the allowlist by routing through the router. This completely defeats the curation guarantee of the extension.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,  // ← router address when called via router
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension. `SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router's address:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly with no mechanism to forward the original user's identity:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
```

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

**Attack path:**
1. Pool admin deploys a curated pool with `SwapAllowlistExtension` and allowlists Alice and Bob.
2. Admin calls `setAllowedToSwap(pool, router, true)` — required for Alice/Bob to use the standard router, since the extension will see `sender = router` for all router-mediated swaps.
3. Disallowed user Eve calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(...)` — pool's `msg.sender` = router.
5. Extension checks `allowedSwapper[pool][router]` = `true` → passes.
6. Eve's swap executes on the curated pool despite being explicitly disallowed.

The extension has no way to distinguish Alice-via-router from Eve-via-router; both present `sender` = router. The wrong value is the identity checked against the allowlist: `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actualUser]`.

## Impact Explanation
Any user can bypass a pool's swap allowlist by routing through `MetricOmmSimpleRouter` whenever the router address is itself allowlisted. This breaks the core invariant of the `SwapAllowlistExtension`: disallowed counterparties gain full swap access to pools designed to be restricted (e.g., for compliance, KYC, or LP-protection reasons). LPs who relied on the allowlist to control pool exposure face unauthorized trading and potential direct loss, constituting broken core pool functionality and unauthorized access to restricted pools above Sherlock thresholds.

## Likelihood Explanation
The trigger condition — the router being allowlisted — is a natural and expected operational step for any pool admin who wants permitted users to be able to use the standard router. There is no alternative mechanism to allow router-mediated swaps for specific users without opening the pool to all users. The bypass is reachable by any unprivileged user with no special setup beyond calling the public router. The existing test suite (`FullMetricExtensionTest`) only tests direct pool swaps, not router-mediated swaps against allowlisted pools, confirming the gap is undetected.

## Recommendation
The `SwapAllowlistExtension` must gate the economically relevant actor, not the intermediary. Two viable approaches:

1. **Router-forwarded identity**: Have `MetricOmmSimpleRouter` encode the original `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check that address when `sender` is a known router. Requires the extension to maintain a trusted router registry.

2. **Separate `swapOnBehalfOf` entrypoint**: Add a pool-level function that accepts an explicit `originator` address (verified via a trusted router registry), and pass `originator` as `sender` to extensions.

Until fixed, pool admins using `SwapAllowlistExtension` must not allowlist the router address, which means permitted users cannot use the standard router — a broken UX that defeats the purpose of the periphery layer.

## Proof of Concept
```solidity
// Setup: curated pool with SwapAllowlistExtension
// Admin allowlists Alice and the router (to let Alice use the router)
extension.setAllowedToSwap(pool, alice, true);
extension.setAllowedToSwap(pool, address(router), true); // required for Alice to use router

// Eve is NOT allowlisted
assertFalse(extension.isAllowedToSwap(pool, eve));

// Eve bypasses the allowlist via the router
vm.prank(eve);
// Router calls pool.swap() with msg.sender = router
// Extension sees sender = router → allowedSwapper[pool][router] = true → passes
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: eve,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    tokenIn: token0,
    extensionData: ""
}));
// Eve's swap succeeds — allowlist bypassed
```
