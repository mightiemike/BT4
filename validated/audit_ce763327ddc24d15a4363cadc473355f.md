### Title
Missing `approve(0)` Before `approve(balance)` in `creditDeposit()` Permanently Bricks USDT Deposits for a Subaccount — (`File: core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without first resetting the allowance to zero. For tokens like USDT that require the allowance to be set to zero before a new non-zero approval, any scenario that leaves a residual allowance after a prior call will cause every subsequent `creditDeposit()` invocation to revert, permanently blocking collateral deposits for the affected subaccount.

---

### Finding Description

In `DirectDepositV1.creditDeposit()`, for each spot product token the contract executes:

```solidity
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
``` [1](#0-0) 

The `approve` is issued for the full `uint256 balance`, but the deposit call passes `uint128(balance)`. If `balance > type(uint128).max`, the cast silently truncates and the endpoint's `transferFrom` only consumes `uint128(balance)` tokens, leaving a residual allowance of `balance - uint128(balance)` on the endpoint spender. On the next call to `creditDeposit()`, the `approve(endpoint, newBalance)` call reverts for USDT-like tokens because their implementation requires the allowance to be zero before setting a new non-zero value.

The same function also iterates over all product IDs in a single transaction:

```solidity
uint32[] memory productIds = spotEngine.getProductIds();
for (uint256 i = 0; i < productIds.length; i++) {
    ...
    token.approve(address(endpoint), balance);
``` [2](#0-1) 

A revert on any single token's `approve` causes the entire loop to revert, blocking deposits for all tokens for that subaccount in one call.

By contrast, `ContractOwner.wrapVaultAsset()` already applies the correct pattern in the same codebase:

```solidity
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
``` [3](#0-2) 

The `depositInsurance()` function in `ContractOwner` has the same missing-`approve(0)` pattern but is `onlyOwner`-gated and therefore out of scope for unprivileged impact. [4](#0-3) 

---

### Impact Explanation

If a residual allowance is left on the endpoint for a USDT-like token (triggered by the `uint128` truncation path or by the endpoint consuming less than the full approved amount), every future call to `creditDeposit()` for that DDA will revert. The subaccount's collateral deposit path via the Direct Deposit Address is permanently bricked for that token. Funds sent to the DDA accumulate but cannot be credited to the subaccount, effectively locking user assets in the DDA contract with no recovery path through the normal deposit flow.

---

### Likelihood Explanation

`creditDeposit()` has no access control — it is a plain `external` function callable by any address. [5](#0-4) 

The trigger requires either (a) a token balance exceeding `type(uint128).max` (unlikely for most tokens at normal decimals) or (b) the endpoint consuming less than the full approved amount. Scenario (b) depends on endpoint internals. The likelihood is low-to-medium, but the consequence once triggered is permanent and unrecoverable through the normal user flow.

---

### Recommendation

Apply the same pattern already used in `ContractOwner.wrapVaultAsset()`: reset the allowance to zero immediately before setting the new value.

```solidity
// In DirectDepositV1.creditDeposit()
if (balance != 0) {
    token.approve(address(endpoint), 0);       // reset first
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(...);
}
``` [1](#0-0) 

---

### Proof of Concept

1. A DDA is created for a subaccount via `ContractOwner.createDirectDepositV1()`.
2. A USDT-like token is a registered spot product in `SpotEngine`.
3. A large token balance (`> type(uint128).max`) accumulates in the DDA, or the endpoint partially consumes the approved amount.
4. `creditDeposit()` is called: `token.approve(endpoint, balance)` succeeds; `depositCollateralWithReferral` consumes only `uint128(balance)`, leaving residual allowance.
5. Any subsequent call to `creditDeposit()` reaches `token.approve(endpoint, balance)` with a non-zero existing allowance — USDT reverts here.
6. All future deposit attempts for this subaccount via the DDA are permanently blocked. [6](#0-5)
