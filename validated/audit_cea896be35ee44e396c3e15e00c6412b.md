Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on `sender`, which resolves to `msg.sender` of the pool's `swap` call. When `MetricOmmSimpleRouter` calls `pool.swap(...)`, the pool's `msg.sender` is the router contract, not the actual user. Any pool admin who allowlists the router to enable router-mediated swaps for legitimate users simultaneously grants unrestricted swap access to every unprivileged user who routes through that same router.

## Finding Description
In `SwapAllowlistExtension.beforeSwap` (L37), the check is:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` is the pool (the extension is called via `ExtensionCalling._callExtensionsInOrder`). `sender` is the first argument forwarded by the pool — specifically `msg.sender` of the pool's own `swap` call, as seen in `MetricOmmPool.swap` (L230-231):

```solidity
_beforeSwap(
    msg.sender,   // direct caller of pool.swap()
    ...
```

`ExtensionCalling._beforeSwap` (L149-177) encodes that `sender` and forwards it verbatim to the extension. When `MetricOmmSimpleRouter.exactInputSingle` (L72-80) calls `pool.swap(...)`, the pool sees `msg.sender = router`. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

For any legitimate user to swap through the router, the pool admin must call `setAllowedToSwap(pool, router, true)`. Once that entry is set, the check `allowedSwapper[pool][router]` passes for every caller of the router — including completely unprivileged users who are not individually allowlisted.

This is structurally asymmetric with `DepositAllowlistExtension.beforeAddLiquidity` (L32-41), which receives `owner` as a distinct parameter representing the economic actor, allowing the check to gate on the beneficiary regardless of which contract calls the pool. The swap path has no equivalent explicit `swapper` parameter — the pool's `swap` signature always uses `msg.sender` as the identity forwarded to extensions.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., KYC'd users, institutional partners, or whitelisted market makers) is fully bypassed by any unprivileged user routing through `MetricOmmSimpleRouter`. LP assets are exposed to unauthorized counterparties, and the pool's core access-control invariant — that only allowlisted addresses may swap — is rendered ineffective for any pool that supports router-mediated swaps. This constitutes broken core pool functionality with direct exposure of LP principal to unauthorized swap counterparties.

## Likelihood Explanation
The bypass requires the pool admin to have allowlisted the router address. This is a necessary and expected configuration for any pool that wants to support router-mediated swaps for legitimate users — without it, no user can swap through the router at all. Once the router is allowlisted (a standard operational requirement), the bypass is reachable by any unprivileged user with no special permissions or capital requirements beyond the swap amount. Likelihood is **Medium**: conditional on the router being allowlisted, but that condition is a standard operational requirement for router-enabled pools.

## Recommendation
Gate on the actual user identity rather than the direct caller. Two viable approaches:

1. **Extension-data forwarding**: The router encodes the original `msg.sender` into `extensionData` for each hop, and `SwapAllowlistExtension` decodes and checks that address instead of `sender`. The extension must verify the forwarded address is authentic (e.g., by trusting only known router contracts via a registry).
2. **Explicit swapper parameter**: Add an explicit `swapper` parameter to the pool's `swap` interface that the router populates with the actual user, and forward it to extensions as the gated identity — mirroring how `addLiquidity` passes `owner` separately from `msg.sender`.

At minimum, document clearly that allowlisting the router grants unrestricted access to all router users, and advise pool admins not to combine per-user allowlisting with router allowlisting.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Admin allowlists user A: `setAllowedToSwap(pool, userA, true)`.
3. Admin allowlists the router to enable router-mediated swaps for user A: `setAllowedToSwap(pool, router, true)`.
4. User B (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(recipient, ...)` — pool sees `msg.sender = router`.
6. Pool calls `_beforeSwap(router, ...)` — extension checks `allowedSwapper[pool][router]` → **passes**.
7. User B successfully swaps on the curated pool, bypassing the allowlist entirely.
