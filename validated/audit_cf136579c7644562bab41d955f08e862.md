Audit Report

## Title
SwapAllowlistExtension checks router identity instead of end-user identity, enabling allowlist bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
When a swap is routed through `MetricOmmSimpleRouter`, the `sender` argument forwarded to `SwapAllowlistExtension.beforeSwap` is the router's address, not the originating user's address. Because the allowlist is keyed on this `sender` value, any pool that allowlists the router contract grants unrestricted swap access to every user who calls through it, completely defeating per-user gating. Conversely, individually allowlisted users are blocked from using the router at all.

## Finding Description

**Call chain:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle` — `msg.sender` = user EOA.
2. Router calls `IMetricOmmPoolActions(params.pool).swap(...)` — inside the pool, `msg.sender` = **router address**.
3. `MetricOmmPool.swap` (line 230–240) calls `_beforeSwap(msg.sender, ...)`, forwarding the router address as `sender`.
4. `ExtensionCalling._beforeSwap` (line 162–176) ABI-encodes `sender` = router address and dispatches to the extension via `CallExtension.callExtension`.
5. `SwapAllowlistExtension.beforeSwap` (line 37) evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` = pool (correct pool key) and `sender` = **router address** (wrong identity — should be the end-user).

**Exploit path A — allowlist bypass:**
- Pool admin allowlists the router: `allowedSwapper[pool][router] = true` (a natural setup to enable router-based swaps).
- Any unprivileged user calls `exactInputSingle` through the router.
- The check passes because `allowedSwapper[pool][router]` is true.
- Non-allowlisted users swap freely in a pool intended to be restricted.

**Exploit path B — broken functionality:**
- Pool admin allowlists individual users: `allowedSwapper[pool][user] = true`.
- User calls the router; the pool sees `sender` = router (not allowlisted) → reverts with `NotAllowedToSwap`.
- Allowlisted users are permanently blocked from using the router.

**Existing guards are insufficient:** `BaseMetricExtension.onlyPool` (line 19–24) only verifies that `msg.sender` is a registered pool — it does not validate the `sender` argument passed into the hook. There is no mechanism in the router or pool to forward the original `tx.origin` or user address as the `sender`.

## Impact Explanation
Exploit path A allows any unprivileged trader to bypass a pool's swap allowlist by routing through `MetricOmmSimpleRouter`, breaking the admin-boundary invariant that restricts swap access to approved addresses. Exploit path B renders the allowlist extension non-functional for all router-mediated swaps, breaking core swap functionality for legitimate allowlisted users. Both outcomes fall within the allowed impact gate: admin-boundary break and broken core pool functionality.

## Likelihood Explanation
Path A requires only that the pool admin has allowlisted the router address — a natural and expected configuration for any pool that intends to support router-based trading. Once that condition holds, any unprivileged user can exploit it with a single `exactInputSingle` call. No special privileges, flash loans, or complex setup are required. Path B is triggered unconditionally whenever an allowlisted user attempts to swap via the router.

## Recommendation
The pool must forward the true initiating address to extensions. One approach: add an optional `originator` field to the swap call or store it in transient storage in the router before calling the pool, then have the pool pass it as `sender` to extensions instead of `msg.sender`. Alternatively, `SwapAllowlistExtension.beforeSwap` should check both `sender` (direct caller) and a separately passed originator, or the router should be redesigned to call the pool directly from the user's context (not feasible without `delegatecall` on the pool side). The simplest fix is for the router to encode the real user address in `extensionData` and for `SwapAllowlistExtension` to decode and check it, with the pool verifying the router's identity separately.

## Proof of Concept

```solidity
// 1. Deploy pool with SwapAllowlistExtension configured.
// 2. Pool admin allowlists the router:
swapAllowlist.setAllowedToSwap(pool, address(router), true);
// 3. Attacker (not individually allowlisted) calls:
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: attacker,
    tokenIn: token0,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// 4. Swap succeeds — allowlist bypass confirmed.
// Verify: allowedSwapper[pool][attacker] == false, yet swap executed.
```
