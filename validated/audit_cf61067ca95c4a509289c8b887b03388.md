### Title
Deposit Accounting Credits User-Supplied Amount Without Verifying Actual Received Tokens, Enabling Over-Crediting with Fee-on-Transfer Collateral — (File: `core/contracts/Endpoint.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

The `depositCollateralWithReferral` flow in Nado transfers tokens from the depositor and then credits the user-supplied `amount` to their on-chain balance without measuring the actual tokens received. For any ERC20 collateral token that deducts a transfer fee, the Clearinghouse credits more than it holds, corrupting the solvency invariant and enabling the depositor to withdraw funds that belong to other users.

---

### Finding Description

**Step 1 — Token transfer in `Endpoint.depositCollateralWithReferral`**

`Endpoint.sol` lines 144–165 call `handleDepositTransfer` with the caller-supplied `amount`, then immediately queue a slow-mode transaction that encodes that same `amount`:

```solidity
handleDepositTransfer(
    IERC20Base(spotEngine.getToken(productId)),
    msg.sender,
    uint256(amount)                          // user-supplied, not balance-delta
);
// ...
slowModeTxs[...] = SlowModeTx({
    tx: abi.encodePacked(
        uint8(TransactionType.DepositCollateral),
        abi.encode(DepositCollateral({
            sender: subaccount,
            productId: productId,
            amount: amount                   // same user-supplied value
        }))
    )
});
```

`handleDepositTransfer` (via `ERC20Helper.safeTransferFrom`) issues a plain `transferFrom` with no before/after balance snapshot. [1](#0-0) 

**Step 2 — Balance update in `Clearinghouse.depositCollateral`**

When the slow-mode transaction is later executed, `Clearinghouse.depositCollateral` normalises and credits the same `txn.amount` that was stored in the queue:

```solidity
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
```

There is no measurement of what the contract actually received. [2](#0-1) 

**Root cause — no balance delta**

The correct pattern (analogous to the external report's recommended fix) would be:

```solidity
uint256 before = token.balanceOf(address(this));
token.safeTransferFrom(from, address(this), amount);
uint256 actualReceived = token.balanceOf(address(this)) - before;
// queue slow-mode tx with actualReceived, not amount
```

Neither `Endpoint.sol` nor `Clearinghouse.sol` performs this check. [3](#0-2) 

---

### Impact Explanation

For any collateral token that charges a transfer fee (fee-on-transfer ERC20), the Clearinghouse receives `amount - fee` tokens but credits the full `amount` to the depositor's SpotEngine balance. The depositor can then withdraw the full credited amount, forcing the shortfall onto subsequent withdrawers. Repeated deposits inflate the credited balances beyond the actual token reserves, making the protocol insolvent. The broken invariant is:

```
sum(spotEngine.balance[productId][*]) <= token.balanceOf(clearinghouse)
```

This invariant is enforced on withdrawal via `assertUtilization`, but the over-credit happens silently at deposit time, before any utilisation check runs. [4](#0-3) 

---

### Likelihood Explanation

The protocol's `addEngine` / product registration path places no restriction on token type; any ERC20 can be registered as a collateral product by the owner. Fee-on-transfer tokens (e.g., tokens with a built-in burn or redistribution mechanism) are a well-established ERC20 variant. If any such token is ever listed as collateral — intentionally or by mistake — the over-crediting is immediately exploitable by any depositor without special privileges. The entry path (`depositCollateral` / `depositCollateralWithReferral`) is fully public. [5](#0-4) 

---

### Recommendation

Replace the fixed-amount transfer with a balance-delta measurement in `Endpoint.depositCollateralWithReferral` (or in `handleDepositTransfer`):

```solidity
uint256 balanceBefore = IERC20Base(token).balanceOf(address(this));
token.safeTransferFrom(msg.sender, address(this), uint256(amount));
uint256 actualReceived = IERC20Base(token).balanceOf(address(this)) - balanceBefore;
require(actualReceived <= type(uint128).max);
// use actualReceived (cast to uint128) when building the slow-mode DepositCollateral struct
```

This mirrors the fix recommended in the external report: measure the actual delta rather than trusting the caller-supplied value.

---

### Proof of Concept

1. Deploy a fee-on-transfer ERC20 (1 % burn on every `transferFrom`).
2. Owner registers it as a collateral product via `addEngine` / product initialisation.
3. Attacker calls `depositCollateralWithReferral(subaccount, productId, 1_000e18)`.
   - `handleDepositTransfer` pulls 1 000 tokens; contract receives 990 (1 % fee burned).
   - Slow-mode tx is queued with `amount = 1_000e18`.
4. Sequencer (or attacker after the delay) executes the slow-mode tx.
   - `Clearinghouse.depositCollateral` credits `1_000e18 * multiplier` to the attacker's SpotEngine balance.
5. Attacker submits a `WithdrawCollateral` for 1 000 tokens.
   - `assertUtilization` passes because the over-credit has not yet been detected.
   - Attacker receives 1 000 tokens; the 10-token shortfall is borne by the next withdrawer.
6. Repeating this drains the reserve proportional to the fee rate. [6](#0-5) [7](#0-6)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
