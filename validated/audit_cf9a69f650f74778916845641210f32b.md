### Title
`ClearinghouseLiq::_assertLiquidationAmount` Can Be DoSed by Front-Running with a Dust Liquidation — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`_assertLiquidationAmount` enforces a strict check that the caller-specified `txn.amount` must not exceed `maxLiquidatable`. A colluding party can submit a dust-sized liquidation for the same position first, reducing `maxLiquidatable` by the minimum increment and causing the original full-position liquidation to revert with `ERR_NOT_LIQUIDATABLE_AMT`, while the liquidator loses their upfront `LIQUIDATION_FEE`.

---

### Finding Description

In `ClearinghouseLiq._assertLiquidationAmount`, the terminal check is:

```solidity
if (txn.amount >= 0) {
    require(maxLiquidatable >= txn.amount, ERR_NOT_LIQUIDATABLE_AMT);
} else {
    require(maxLiquidatable <= txn.amount, ERR_NOT_LIQUIDATABLE_AMT);
}
``` [1](#0-0) 

`maxLiquidatable` is derived from the liquidatee's live on-chain position balance at execution time:

```solidity
perpAmount = perpEngine.getBalance(perpId, txn.liquidatee).amount;
...
maxLiquidatable = perpAmount;
``` [2](#0-1) [3](#0-2) 

A liquidator who signs a transaction for the exact full position amount (`txn.amount = perpAmount`) will have their transaction revert if the liquidatee's position has decreased by even one `perpSizeIncrement` before the transaction is processed.

**Attack path:**

1. Liquidatee is underwater; their perp position is `X` units.
2. Liquidator A signs a `LiquidateSubaccount` transaction for `txn.amount = X` (full position).
3. A colluding party (Liquidator B, potentially controlled by the liquidatee) signs a `LiquidateSubaccount` transaction for `txn.amount = perpSizeIncrement` (the minimum valid amount, enforced at line 183).
4. Liquidator B's transaction is submitted to the sequencer first and processed first.
5. After Liquidator B's execution, `maxLiquidatable = X − perpSizeIncrement`.
6. Liquidator A's transaction is processed: `require(X − perpSizeIncrement >= X)` → reverts with `ERR_NOT_LIQUIDATABLE_AMT`.
7. Liquidator A loses their `LIQUIDATION_FEE`, charged upfront in `EndpointTx.processTransactionImpl`:

```solidity
if (signedTx.tx.productId != type(uint32).max) {
    chargeFee(signedTx.tx.sender, LIQUIDATION_FEE);
}
clearinghouse.liquidateSubaccount(signedTx.tx);
``` [4](#0-3) 

The fee is charged before the call to `liquidateSubaccount`, so a revert inside `liquidateSubaccountImpl` does not refund it. The liquidatee can repeat this pattern indefinitely, front-running each full-position liquidation attempt with a dust liquidation.

Self-liquidation is blocked by:

```solidity
require(txn.sender != txn.liquidatee, ERR_UNAUTHORIZED);
``` [5](#0-4) 

However, the liquidatee can collude with a second account they control to act as Liquidator B.

---

### Impact Explanation

- **Liquidator fee drain**: Every failed full-position liquidation attempt costs the liquidator `LIQUIDATION_FEE` with no recourse.
- **Delayed liquidation**: The underwater account is never fully closed; only dust amounts are liquidated per cycle, keeping the position alive.
- **Protocol bad-debt accumulation**: The longer an underwater position remains open, the greater the risk of the insurance fund being insufficient to cover the shortfall, directly threatening protocol solvency.

---

### Likelihood Explanation

The attack requires the liquidatee to collude with a second account. In a competitive liquidation environment, the liquidatee can monitor the sequencer's pending transaction queue (which is observable in many sequencer implementations) and submit a dust liquidation via a colluding account before the full liquidation is processed. The sequencer processes transactions in submission order, making this a realistic race condition. The cost to the attacker is only the `LIQUIDATION_FEE` for the dust liquidation, while the benefit is avoiding full liquidation and draining competing liquidators' fees.

---

### Recommendation

Replace the strict `require` with a clamped amount, mirroring the M-13 fix (`min(deficit, amount)`). Since `txn` is `calldata`, store the clamped value in a local variable and propagate it through `_handleLiquidationPayment`:

```solidity
int128 effectiveAmount;
if (txn.amount >= 0) {
    effectiveAmount = MathHelper.min(maxLiquidatable, txn.amount);
} else {
    effectiveAmount = MathHelper.max(maxLiquidatable, txn.amount);
}
// use effectiveAmount in place of txn.amount for the remainder of execution
``` [1](#0-0)

### Proof of Concept

```solidity
// Liquidatee has perpAmount = 100e18 and is underwater
// perpSizeIncrement = 1e18

// Step 1: Liquidator A signs for full position
LiquidateSubaccount memory txnA = LiquidateSubaccount({
    sender:          liquidatorA,
    liquidatee:      victim,
    productId:       perpProductId,
    isEncodedSpread: false,
    amount:          100e18,   // full position
    nonce:           nonceA
});

// Step 2: Colluding Liquidator B front-runs with one size increment
LiquidateSubaccount memory txnB = LiquidateSubaccount({
    sender:          liquidatorB,  // controlled by victim
    liquidatee:      victim,
    productId:       perpProductId,
    isEncodedSpread: false,
    amount:          1e18,         // minimum valid perp amount
    nonce:           nonceB
});

// Sequencer processes txnB first:
//   victim.perpAmount = 99e18, maxLiquidatable = 99e18
//   liquidatorB receives dust liquidation reward

// Sequencer processes txnA:
//   _assertLiquidationAmount: require(99e18 >= 100e18) → REVERT ERR_NOT_LIQUIDATABLE_AMT
//   liquidatorA already paid LIQUIDATION_FEE — fee is not refunded

// Victim repeats this for every subsequent liquidation attempt.
```
