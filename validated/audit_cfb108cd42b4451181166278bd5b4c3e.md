### Title
Unrestricted Public Access to `run_mainchain_gc` Allows Any Caller to Aggressively Prune Mainchain Headers, Permanently Invalidating SPV Proofs - (File: contract/src/lib.rs)

---

### Summary

`run_mainchain_gc` is a state-mutating `pub` function with no caller identity check. Any unprivileged NEAR account can call it directly with an attacker-controlled `batch_size`, causing the contract to permanently delete mainchain block headers from `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height`. This makes `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` permanently fail for any pruned block, enabling a front-running attack that invalidates legitimate SPV proofs.

---

### Finding Description

`submit_blocks` — the only intended header-submission path — is gated behind `#[trusted_relayer]` and `#[pause]`: [1](#0-0) 

Inside `submit_blocks`, GC is invoked as an internal call with `batch_size = headers.len()` (typically 1–100): [2](#0-1) 

However, `run_mainchain_gc` is also exposed as a standalone `pub` method with **only** a `#[pause]` guard and no caller identity restriction: [3](#0-2) 

The caller supplies `batch_size` directly. The function computes:

```
selected_amount_to_remove = min(amount_of_headers_we_store - gc_threshold, batch_size)
``` [4](#0-3) 

By passing `batch_size = u64::MAX`, an attacker forces the contract to remove every header beyond `gc_threshold` in a single transaction. Each removed header is permanently deleted from `headers_pool` and both index maps: [5](#0-4) 

`mainchain_initial_blockhash` is then advanced to the new oldest block. The deleted headers are unrecoverable — the relayer cannot re-submit them because `submit_block_header` requires the previous block to already be in `headers_pool`.

---

### Impact Explanation

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` look up the target block in `mainchain_header_to_height` and `headers_pool`: [6](#0-5) [7](#0-6) 

After an attacker calls `run_mainchain_gc(u64::MAX)`, any block that was pruned is permanently absent from both maps. Every subsequent SPV proof call for those blocks panics with "block does not belong to the current main chain" or "cannot find requested transaction block". Any downstream NEAR contract that gates asset release or cross-chain settlement on a `verify_transaction_inclusion` result receives a permanent false-negative, causing locked or lost funds for users whose Bitcoin transactions can no longer be proven on-chain.

---

### Likelihood Explanation

The attack requires no privileged keys, no staked tokens, and no special role. Any NEAR account can call `run_mainchain_gc` directly. The function is callable whenever the contract is not paused. A single transaction with `batch_size = u64::MAX` is sufficient to execute the maximum possible pruning. Front-running a pending `verify_transaction_inclusion` call is straightforward on NEAR because transaction ordering within a block is observable.

---

### Recommendation

Add a caller identity restriction to `run_mainchain_gc` matching the restriction on `submit_blocks`. The simplest fix is to add `#[trusted_relayer]` to the function, since the relayer already calls it indirectly through `submit_blocks`. Alternatively, introduce a dedicated role (e.g., `GCManager`) and gate the public entry point behind an `acl_require` check. The internal call from `submit_blocks` is unaffected because it is a Rust method call, not a NEAR cross-contract call, and does not pass through the access-control macro.

---

### Proof of Concept

1. Deploy the contract with `gc_threshold = 52704` and submit enough blocks so that `amount_of_headers_we_store > gc_threshold` (i.e., the chain has grown beyond the threshold).
2. A user constructs a valid `verify_transaction_inclusion` call for a block at height `H` where `H < mainchain_tip - gc_threshold` (i.e., the block is eligible for GC).
3. The attacker observes the pending transaction in the NEAR mempool and front-runs it by calling:
   ```
   run_mainchain_gc(18446744073709551615)  // u64::MAX
   ```
4. The contract removes all headers from `mainchain_initial_blockhash` up to `mainchain_tip - gc_threshold`, including the block at height `H`. `mainchain_initial_blockhash` is advanced past `H`.
5. The user's `verify_transaction_inclusion` transaction executes next and panics: `mainchain_header_to_height.get(&args.tx_block_blockhash)` returns `None` → `env::panic_str("block does not belong to the current main chain")`.
6. The user's SPV proof is permanently invalidated. Any asset release gated on this proof is permanently blocked.
