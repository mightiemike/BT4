### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables usdcE Drain Without USDC Payment — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(...)` without checking its return value. If the USDC token on chain 57073 (Ink) returns `false` instead of reverting on a failed transfer, any caller can drain usdcE from a victim's Direct Deposit Address (DDA) without providing any USDC.

---

### Finding Description

`replaceUsdcEWithUsdc` is a publicly callable function (no access control beyond a chain-id check) that is intended to swap usdcE held in a DDA for native USDC provided by the caller:

```
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);  // ← unchecked
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
``` [1](#0-0) 

The raw `transferFrom` at line 616 does not check the boolean return value. If it silently returns `false`, execution falls through to `withdraw` (which moves usdcE from the DDA to `ContractOwner`) and then `safeTransfer` (which sends that usdcE to `msg.sender`). The caller receives usdcE without having deposited any USDC.

The `ERC20Helper` library — which provides `safeTransferFrom` with proper return-value checking — is already imported and bound to `IERC20Base` via `using ERC20Helper for IERC20Base` in this same contract, yet is not used here. [2](#0-1) [3](#0-2) 

The inconsistency is stark: the very next line (618) uses `safeTransfer` for the outbound usdcE leg, but the inbound USDC leg is left unchecked. [4](#0-3) 

---

### Impact Explanation

An attacker who calls `replaceUsdcEWithUsdc(subaccount)` for any subaccount whose DDA holds usdcE will:

1. Trigger `withdraw` on the DDA, moving all usdcE to `ContractOwner`.
2. Receive that usdcE via `safeTransfer` — at zero cost if `transferFrom` silently failed.

The corrupted asset delta is the full usdcE balance of the targeted DDA, transferred to the attacker. The DDA's subaccount loses its usdcE collateral with no USDC replacement deposited.

---

### Likelihood Explanation

The function is gated only by `require(block.chainid == 57073)` — any EOA or contract on Ink mainnet can call it. [5](#0-4) 

The USDC address hardcoded at line 613 (`0x2D270e6886d130D724215A266106e6832161EAEd`) is a bridged/wrapped USDC on Ink. Bridged token implementations vary; some return `false` on insufficient allowance or balance rather than reverting. If this token exhibits that behaviour, the exploit is directly reachable by any unprivileged caller with no preconditions beyond the existence of a DDA with a non-zero usdcE balance. [6](#0-5) 

---

### Recommendation

Replace the raw `transferFrom` call with the `safeTransferFrom` wrapper already available in `ERC20Helper`:

```solidity
// Before (unsafe)
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe)
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ERC20Helper.safeTransferFrom` uses a low-level `call` and checks `success && (data.length == 0 || abi.decode(data, (bool)))`, handling both reverting and non-reverting token variants. [2](#0-1) 

---

### Proof of Concept

1. A subaccount `S` has a DDA at `directDepositV1Address[S]` holding 1000 usdcE.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(S)` on chain 57073 with zero USDC allowance granted.
3. `balance = IERC20Base(usdcE).balanceOf(directDepositV1)` → 1000.
4. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, 1000)` → returns `false` (no revert, attacker has no USDC allowance).
5. Return value is not checked; execution continues.
6. `DirectDepositV1(directDepositV1).withdraw(usdcE)` → 1000 usdcE moved to `ContractOwner`.
7. `IERC20Base(usdcE).safeTransfer(attacker, 1000)` → attacker receives 1000 usdcE.
8. Subaccount `S`'s DDA is emptied; no USDC was deposited. [7](#0-6)

        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
