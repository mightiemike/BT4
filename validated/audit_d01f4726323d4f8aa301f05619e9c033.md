### Title
Odd-Leaf Phantom Index Allows Same Transaction to Verify at Two Distinct `tx_index` Values — (`merkle-tools/src/lib.rs::compute_root_from_merkle_proof`, `contract/src/lib.rs::verify_transaction_inclusion`)

---

### Summary

`compute_root_from_merkle_proof` never validates that `transaction_position` is within the actual tree bounds. In any odd-width Merkle tree, Bitcoin's standard duplicate-last-leaf padding means the last real transaction (index `N-1`) and its phantom duplicate (index `N`) produce identical root computations. An unprivileged caller can therefore pass the same `(tx_id, merkle_proof)` with two different `tx_index` values and receive `true` from `verify_transaction_inclusion` both times — breaking the invariant that a proof binds to exactly one position.

---

### Finding Description

**Root cause — `merkle-tools/src/lib.rs`, `compute_root_from_merkle_proof`:**

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }
    current_hash
}
```

The function uses only `current_position % 2` (left/right selector) and `current_position /= 2` (level advance). It never checks that `transaction_position < tree_leaf_count`. [1](#0-0) 

**Why odd trees create a phantom index:**

`merkle_proof_calculator` pads an odd-length level by appending a copy of the last element before hashing:

```rust
if current_hashes.len() % 2 == 1 {
    current_hashes.push(current_hashes[current_hashes.len() - 1].clone());
}
``` [2](#0-1) 

For a 3-transaction tree `[T0, T1, T2]`:

| Level | Nodes | Padding |
|---|---|---|
| 0 | T0, T1, T2 | → T0, T1, T2, **T2** |
| 1 | H(T0,T1), H(T2,T2) | — |
| 2 | root | — |

The valid proof for T2 at index 2 is `[T2, H(T0,T1)]`.

Now trace `compute_root_from_merkle_proof(T2, index=3, [T2, H(T0,T1)])`:

- Step 1: `position=3` (odd) → `H(proof[0]=T2, T2)` = `H(T2,T2)`, position → 1
- Step 2: `position=1` (odd) → `H(proof[1]=H(T0,T1), H(T2,T2))` = **root** ✓

The same proof and `tx_id` produce the correct root for **both** `tx_index=2` (real) and `tx_index=3` (phantom). No bound check exists anywhere in the call chain to reject index 3.

**Call path in the contract:**

`verify_transaction_inclusion` passes `args.tx_index` directly to `compute_root_from_merkle_proof` without any upper-bound validation:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [3](#0-2) 

`verify_transaction_inclusion_v2` does not fix this — it adds a coinbase proof check at hardcoded index 0, then delegates to `verify_transaction_inclusion` with the caller-supplied `tx_index` unchanged:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [4](#0-3) 

The `ProofArgs` / `ProofArgsV2` structs accept `tx_index: u64` with no constraints: [5](#0-4) 

---

### Impact Explanation

Any bridge or withdrawal contract that calls `verify_transaction_inclusion` (or `_v2`) and uses the tuple `(tx_id, tx_index, block_hash)` as a unique replay-prevention key is vulnerable to double-spend:

1. Attacker submits a real Bitcoin withdrawal transaction T2 in an odd-width block (N=3).
2. First call: `tx_id=T2, tx_index=2, block_hash=B` → `true` → withdrawal #1 released.
3. Second call: `tx_id=T2, tx_index=3, block_hash=B` → `true` → withdrawal #2 released.

The light client itself returns `true` for both calls against the same canonical block, so the replay key differs (`tx_index` 2 vs 3) while the underlying Bitcoin transaction is identical. This directly enables double-spend / double-withdrawal against any consumer of this API.

---

### Likelihood Explanation

- The entrypoint is fully public (`pub fn verify_transaction_inclusion`, no `#[private]`, no role gate beyond `#[pause]`). [6](#0-5) 
- The attacker controls all five proof arguments (`tx_id`, `tx_block_blockhash`, `tx_index`, `merkle_proof`, `confirmations`). [5](#0-4) 
- Odd-width blocks are common on Bitcoin mainnet (any block with an odd transaction count triggers this).
- No special chain state is required beyond a confirmed block with an odd transaction count being present in the light client's mainchain map.

---

### Recommendation

In `compute_root_from_merkle_proof`, add an explicit upper-bound check before the loop:

```rust
// The tree has at most 2^proof_len leaves; reject out-of-range indices.
let max_index = (1usize << merkle_proof.len()).saturating_sub(1);
assert!(transaction_position <= max_index, "tx_index out of tree bounds");
```

More precisely, the caller should supply the actual leaf count and the function should reject any `transaction_position >= leaf_count`. Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` should require callers to pass the block's transaction count and enforce `tx_index < tx_count`.

---

### Proof of Concept

```rust
// 3-tx odd tree: [T0, T1, T2]
// Valid proof for T2 at index 2 = [T2, H(T0,T1)]
// Phantom call with index 3 also returns the correct root.

use merkle_tools::{compute_root_from_merkle_proof, merkle_proof_calculator};

let txs = vec![T0, T1, T2];
let root = merkle_root(txs.clone());          // canonical root
let proof = merkle_proof_calculator(txs, 2);  // proof for index 2

// Real index — expected to pass
assert_eq!(compute_root_from_merkle_proof(T2.clone(), 2, &proof), root);

// Phantom index — must NOT pass, but currently does
assert_eq!(compute_root_from_merkle_proof(T2.clone(), 3, &proof), root);
// ^^^ Both assertions pass on the unmodified codebase.
```

The second assertion demonstrates that `verify_transaction_inclusion` returns `true` for `tx_index=3` even though only 3 transactions (indices 0–2) exist in the block, breaking the one-position binding invariant. [1](#0-0)
