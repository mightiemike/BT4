### Title
`op_multiply` Uses Magnitude-Only Byte Count (`limbs_for_int`) Instead of Actual CLVM-Encoded Byte Length for Running-Product Size, Causing Systematic Cost Undercharging — (File: `src/more_ops.rs`)

---

### Summary

In `op_multiply`, the variable `l0` tracks the byte length of the running product for cost calculation. After the first argument, `l0` is correctly initialized from `int_atom` (actual CLVM atom byte length). After every subsequent multiplication, however, `l0` is updated via `limbs_for_int(&total)`, which computes only the magnitude byte count (`total.bits().div_ceil(8)`). For any positive intermediate product whose most-significant bit is set, the canonical CLVM two's-complement encoding requires one extra leading `0x00` byte, so `limbs_for_int` returns a value that is exactly 1 less than the true encoded length. Every subsequent multiplication step therefore uses a stale, underestimated `l0`, causing the quadratic-and-linear cost formula to undercharge the operation.

---

### Finding Description

`op_multiply` (`src/more_ops.rs`, lines 586–655) maintains `l0 : usize` as the byte-length proxy for the running product:

```
(total, l0) = int_atom(a, arg, "*")?;   // first arg: real atom byte length
...
l0 = limbs_for_int(&total);             // every subsequent arg: magnitude bytes only
```

`limbs_for_int` is defined as:

```rust
fn limbs_for_int(v: &Number) -> usize {
    v.bits().div_ceil(8) as usize
}
```

This returns the number of bytes needed to hold the absolute value's bit pattern. CLVM's signed-integer encoding, however, prepends a `0x00` byte whenever the top bit of the magnitude is set (to distinguish positive from negative). Concretely:

| Running product | `limbs_for_int` | Actual CLVM byte length | Delta |
|---|---|---|---|
| 127 (0x7f) | 1 | 1 | 0 |
| 128 (0x80) | 1 | 2 | **−1** |
| 32 767 (0x7fff) | 2 | 2 | 0 |
| 32 768 (0x8000) | 2 | 3 | **−1** |
| 8 388 607 (0x7fffff) | 3 | 3 | 0 |
| 8 388 608 (0x800000) | 3 | 4 | **−1** |

The cost formula applied at each step is:

```
cost += (l0 + l1) * MUL_LINEAR_COST_PER_BYTE
      + (l0 * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
```

When `l0` is 1 less than the true encoded length, both the linear term and the quadratic term are undercharged. The quadratic undercharge grows with `l1` (up to 256 bytes, the per-argument cap enforced at line 600/612/640).

The first argument is immune because `int_atom` returns the real atom byte length. Only the running product's size after the first multiplication is affected.

---

### Impact Explanation

An attacker submits a CLVM program invoking `*` with a sequence of arguments chosen so that every intermediate product has its top bit set (e.g., starting with 128 and multiplying by values that keep the MSB set). At each step the declared cost is lower than the true computational cost. Over a chain of multiplications the cumulative undercharge is:

```
Σ [ MUL_LINEAR_COST_PER_BYTE + l1_i / MUL_SQUARE_COST_PER_BYTE_DIVIDER ]
```

for every step where the running product's MSB is set. Because the running product is capped at 1024 bytes (`l0 > 1024` check at line 650), the maximum total undercharge is bounded but non-zero. The practical consequence is that a program can consume more real CPU than its declared cost budget permits, weakening the DoS-prevention guarantee of the cost model. In a consensus context, nodes that enforce a strict `max_cost` budget may accept programs that a stricter implementation would reject, creating a potential consensus divergence between implementations that compute cost differently.

---

### Likelihood Explanation

The trigger is fully attacker-controlled: any CLVM byte string encoding `(* 128 <arg2> <arg3> …)` where each `<argN>` keeps the running product's MSB set exercises the bug. No special permissions, keys, or node configuration are required. The entry path is the standard `run_program` evaluator dispatching opcode `0x12` (`*`) through `ChiaDialect::op`, which calls `op_multiply` directly. The condition (running product MSB set) is trivially achievable and stable across many input combinations.

---

### Recommendation

Replace `limbs_for_int(&total)` with the actual CLVM-encoded byte length of the running product. The simplest correct approach is to mirror what `int_atom` does for the first argument: after computing `total`, allocate it temporarily and read back `a.atom_len(node)`, or compute the encoded length directly:

```rust
// correct: account for the sign-extension byte
fn encoded_len_for_positive(v: &Number) -> usize {
    let mag_bytes = v.bits().div_ceil(8) as usize;
    // if top bit of magnitude is set, CLVM needs a leading 0x00
    if mag_bytes > 0 && (v >> ((mag_bytes * 8 - 1) as u64)).to_u8() == Some(1) {
        mag_bytes + 1
    } else {
        mag_bytes
    }
}
```

Alternatively, allocate the intermediate result into the `Allocator` and call `a.atom_len(node)` to get the canonical length, then free it before continuing.
