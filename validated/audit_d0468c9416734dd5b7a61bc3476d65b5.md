Audit Report

## Title
`SwapAllowlistExtension` Per-User Allowlist Bypassed When Router Is Allowlisted — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of `MetricOmmPool.swap`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router, not the user. A pool admin who allowlists the router to enable standard UX inadvertently grants every unpermissioned user full swap access, completely defeating the per-user allowlist.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs:

```solidity
// SwapAllowlistExtension.sol:37-38
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is the first argument forwarded by the pool. In `MetricOmmPool.swap`, that argument is `msg.sender` of the swap call:

```solidity
// MetricOmmPool.sol:230-231
_beforeSwap(
    msg.sender,   // ← router address when called via router
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly with no encoding of the original caller:

```solidity
// MetricOmmSimpleRouter.sol:72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData   // ← user-supplied, not router-signed
    );
```

The router does not encode `msg.sender` into `extensionData`; it passes the user-supplied `params.extensionData` verbatim. The extension ignores `extensionData` entirely (the last `bytes calldata` parameter is unnamed and unused). Therefore the extension evaluates `allowedSwapper[pool][router]`, never `allowedSwapper[pool][original_user]`.

For Alice (an allowlisted user) to use the router, the pool admin must set `allowedSwapper[pool][router] = true`. Once set, Bob (not allowlisted) calls `exactInputSingle` → router calls `pool.swap` → pool passes `sender = router` → extension checks `allowedSwapper[pool][router]` → `true` → swap executes. Bob's identity is never consulted. The same path applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

## Impact Explanation
The `SwapAllowlistExtension` is the protocol's sole mechanism for creating permissioned pools. Once the router is allowlisted (the necessary operational step for standard UX), the allowlist is completely ineffective: any address can execute swaps by routing through `MetricOmmSimpleRouter`. Unauthorized swaps alter pool token balances and LP share claims, constituting a direct loss of the access-control invariant with fund-impacting consequences. This matches the allowed impact: an unprivileged path bypasses pool admin–configured access control, and unauthorized swaps can drain or manipulate pool reserves in ways the admin explicitly intended to prevent.

## Likelihood Explanation
The router is the primary periphery contract for end-user interaction. Any pool admin who deploys `SwapAllowlistExtension` and also wants their allowlisted users to use the standard router must allowlist the router. This is the expected operational path, not an edge case. The bypass is therefore reachable in every realistic deployment of `SwapAllowlistExtension` that supports router-mediated trading, and is repeatable by any unpermissioned address at zero cost.

## Recommendation
The extension must gate the **original user**, not the intermediary. Two viable approaches:

1. **Router encodes original caller in `extensionData`**: Modify `MetricOmmSimpleRouter` to ABI-encode `msg.sender` into `extensionData` before calling `pool.swap`. Modify `SwapAllowlistExtension.beforeSwap` to decode and check that address when `sender` is a known/trusted router. This requires a trusted-router registry in the extension.

2. **Require direct pool interaction for allowlisted pools**: Document and enforce that pools using `SwapAllowlistExtension` must not allowlist the router; allowlisted users must call `pool.swap` directly. This breaks standard UX but preserves the invariant.

Option 1 is the recommended fix as it preserves UX while restoring the access-control invariant.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin: allowedSwapper[pool][alice] = true
  - Pool admin: allowedSwapper[pool][router] = true
    (required so Alice can use the router)

Attack (Bob, not allowlisted):
  1. Bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(...) — msg.sender at pool = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. Extension checks allowedSwapper[pool][router] → true → passes
  5. Bob's swap executes; pool balances change; Bob receives output tokens
     despite never being on the allowlist
```

Foundry test outline:
- Deploy `SwapAllowlistExtension`, pool, and `MetricOmmSimpleRouter`
- `setAllowedToSwap(pool, alice, true)` and `setAllowedToSwap(pool, router, true)`
- `vm.prank(bob); router.exactInputSingle(...)` — assert it succeeds
- `vm.prank(bob); pool.swap(...)` — assert it reverts with `NotAllowedToSwap`
