Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the end user, allowing any caller to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap()` gates swaps by checking the `sender` argument, which the pool sets to its own `msg.sender` — the immediate caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the end user. A pool admin who allowlists the router to support router-based swaps for curated users inadvertently opens the pool to every user on the network, completely defeating the allowlist.

## Finding Description

**Root cause — wrong actor bound to the allowlist check.**

`MetricOmmPool.swap()` passes its own `msg.sender` as the `sender` argument to `_beforeSwap()`: [1](#0-0) 

`ExtensionCalling._beforeSwap()` forwards that value unchanged as the first positional argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap()` then checks that forwarded `sender` against the per-pool allowlist: [3](#0-2) 

**The router never forwards the end user's identity.**

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly from the router contract without encoding the originating user into `extensionData`: [4](#0-3) 

From the pool's perspective `msg.sender` is the router, so `sender` delivered to the extension is the router address — not the end user who called `exactInputSingle()`. The same applies to `exactInput()`, `exactOutputSingle()`, and `exactOutput()`. [5](#0-4) 

**The dilemma the pool admin cannot escape.**

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router at all |
| **Allowlist the router** | Every user on the network can bypass the allowlist |

There is no configuration that simultaneously enforces a user-level allowlist and supports the standard router path. The `setAllowedToSwap` setter accepts an arbitrary address with no warning: [6](#0-5) 

## Impact Explanation
**High.** Any unprivileged user can trade on a pool that the admin intended to restrict to a curated set of swappers. The allowlist — the sole access-control mechanism on the swap path — is rendered inoperative for every pool that supports router-based swaps. Depending on the pool's purpose (e.g., institutional-only, KYC-gated, or rate-limited), this constitutes a direct policy bypass: unauthorized users can drain liquidity at oracle prices, extract fees, or move the pool cursor in ways the LP set never consented to. This meets the "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" impact criteria.

## Likelihood Explanation
**High.** `MetricOmmSimpleRouter` is the primary user-facing swap interface shipped with the protocol. Any pool admin who wants their allowlisted users to have a normal UX will allowlist the router. The admin has no on-chain signal that doing so opens the pool to everyone. The attack requires no special privileges — any EOA can call `exactInputSingle()` on the router pointing at the restricted pool.

## Recommendation
The extension must check the **economically relevant actor** — the end user — not the intermediate contract. Two viable approaches:

1. **Pass the originating user through `extensionData`.** The router encodes `msg.sender` into `extensionData` before calling `pool.swap()`. `SwapAllowlistExtension.beforeSwap()` decodes and checks that address instead of (or in addition to) `sender`.

2. **Check `sender` and fall back to `extensionData` for router calls.** If `sender` is a known router, decode the real user from `extensionData`; otherwise use `sender` directly. This preserves backward compatibility for direct pool callers.

Either way, the allowlist key must be the address that controls the economic action, not the address that happens to be the immediate `msg.sender` to the pool.

## Proof of Concept

```
Setup
─────
1. Pool admin deploys a pool with SwapAllowlistExtension configured.
2. Admin calls setAllowedToSwap(pool, alice, true)   // alice is the curated user
3. Admin calls setAllowedToSwap(pool, router, true)  // router allowlisted so alice can use it

Attack
──────
4. bob (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
5. Router calls pool.swap(recipient, zeroForOne, amount, limit, "", extensionData)
   → msg.sender to pool = router address
6. Pool calls _beforeSwap(sender=router, ...)
7. SwapAllowlistExtension.beforeSwap checks:
     allowedSwapper[pool][router] == true  ✓
   → no revert
8. bob's swap executes at oracle price on a pool he was never supposed to access.

Foundry test sketch
───────────────────
function test_allowlistBypass() public {
    // Setup: alice allowlisted, router allowlisted, bob is not
    vm.prank(admin);
    extension.setAllowedToSwap(address(pool), alice, true);
    vm.prank(admin);
    extension.setAllowedToSwap(address(pool), address(router), true);

    // bob bypasses allowlist via router
    vm.prank(bob);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool), ...
    }));
    // succeeds — bob was never supposed to be able to swap
}
```
