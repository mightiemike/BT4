### Title
Unchecked `approve` Return Value in `DirectDepositV1.creditDeposit()` Permanently Blocks Deposits for Non-Standard ERC20 Tokens — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` through the `IIERC20Base` interface, which declares `approve` as returning `bool`. For tokens that do not return a value from `approve` (e.g., USDT on mainnet), Solidity 0.8's ABI decoder will attempt to decode empty return data as `bool`, causing a revert. The return value is also never checked, so tokens that return `false` on failure silently skip the approval. Either way, the deposit flow is broken for such tokens.

---

### Finding Description

`DirectDepositV1` is a per-subaccount deposit address contract. Its `creditDeposit()` function iterates over all registered spot products, checks the DDA's balance of each token, and if non-zero, approves the endpoint and calls `depositCollateralWithReferral`.

The `IIERC20Base` interface used inside `DirectDepositV1` declares:

```solidity
function approve(address spender, uint256 amount) external returns (bool);
```

At line 92, the call is:

```solidity
token.approve(address(endpoint), balance);
``` [1](#0-0) 

The return value is never checked. More critically, for tokens like USDT that do not return any data from `approve`, Solidity 0.8 will attempt to ABI-decode the empty return buffer as `bool` (requiring 32 bytes), which causes a hard revert. This is the same class of issue that OpenZeppelin's `SafeERC20.safeApprove` was designed to address — it uses a low-level call and explicitly accepts both a `true` return and an empty return.

The `ERC20Helper` library in the codebase correctly implements `safeTransfer` and `safeTransferFrom` using low-level calls with return-data checks:

```solidity
(bool success, bytes memory data) = address(self).call(
    abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
);
require(success && (data.length == 0 || abi.decode(data, (bool))), ERR_TRANSFER_FAILED);
``` [2](#0-1) 

But `DirectDepositV1` does not use `ERC20Helper` for its `approve` call, and no equivalent safe-approve wrapper exists in the codebase. [3](#0-2) 

---

### Impact Explanation

If a spot product token does not return a value from `approve` (USDT-style), every call to `creditDeposit()` reverts at the `approve` line for that token. Because the loop has no try/catch, a revert on any token in the iteration prevents all subsequent tokens in the same call from being deposited. User funds sent to the DDA address are effectively stuck: the only recovery path is the multisig calling `withdrawFromDirectDepositV1` in `ContractOwner`, which is `onlyOwner`. [4](#0-3) 

---

### Likelihood Explanation

`creditDeposit()` is `external` with no access control — any caller can trigger it. [5](#0-4) 

The protocol supports multiple spot products via `spotEngine.getProductIds()`. If any listed token uses non-standard `approve` (no return value), the bug is immediately reachable by any user who sends that token to their DDA and calls `creditDeposit()`. USDT is one of the most widely used stablecoins and is a realistic candidate for listing as a spot collateral asset.

---

### Recommendation

Replace the raw `token.approve(...)` call with a safe-approve pattern that uses a low-level call and accepts both a `true` return and an empty return, consistent with how `ERC20Helper.safeTransfer` is already implemented:

```solidity
(bool success, bytes memory data) = address(token).call(
    abi.encodeWithSelector(IIERC20Base.approve.selector, address(endpoint), balance)
);
require(success && (data.length == 0 || abi.decode(data, (bool))), "Approve failed");
```

Additionally, reset the allowance to 0 before setting a new non-zero value to handle tokens (like USDT) that revert when changing a non-zero allowance directly.

---

### Proof of Concept

1. A spot product is listed with a USDT-style token (no return value from `approve`).
2. A user sends that token to their `DirectDepositV1` address.
3. Anyone calls `creditDeposit()`.
4. At line 92, `token.approve(address(endpoint), balance)` is called through the `IIERC20Base` interface.
5. The token returns no data; Solidity 0.8 attempts to ABI-decode empty bytes as `bool`, reverts.
6. The entire `creditDeposit()` call reverts; the user's tokens remain stuck in the DDA.
7. Recovery requires the multisig to call `ContractOwner.withdrawFromDirectDepositV1`, returning funds to the owner rather than crediting the user's subaccount. [6](#0-5)

        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            address tokenAddr = spotEngine.getToken(productId);
            require(tokenAddr != address(0), "Invalid productId.");
            IIERC20Base token = IIERC20Base(tokenAddr);
            uint256 balance = token.balanceOf(address(this));
            if (balance != 0) {
                token.approve(address(endpoint), balance);
                endpoint.depositCollateralWithReferral(
                    subaccount,
                    productId,
                    uint128(balance),
                    "-1"
                );
            }
        }
    }
