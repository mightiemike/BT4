Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Identity Instead of End User, Allowing Any User to Bypass Per-User Swap Allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool always sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the direct caller of `pool.swap()` is the router contract, not the end user. A pool admin who allowlists the router to support standard periphery swaps on a curated pool inadvertently grants every user — including those explicitly not allowlisted — the ability to bypass the per-user swap gate by routing through the router.

## Finding Description
The full call chain is confirmed by the production code:

**Step 1 — Pool always passes `msg.sender` as `sender` to `_beforeSwap`:**

In `MetricOmmPool.swap` (lines 230–231), the pool calls:
```solidity
_beforeSwap(
    msg.sender,   // always the direct caller of pool.swap()
    ...
);
```

**Step 2 — `ExtensionCalling._beforeSwap` forwards `sender` as the first argument to the extension:**

In `ExtensionCalling._beforeSwap` (lines 162–165), the extension is called with:
```solidity
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, ...)  // sender = msg.sender of pool.swap()
)
```

**Step 3 — `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]`:**

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool and `sender` is the direct caller of `pool.swap()`.

**Step 4 — `MetricOmmSimpleRouter` calls `pool.swap()` directly, making itself the `sender`:**

In `MetricOmmSimpleRouter.exactInputSingle` (lines 72–80):
```solidity
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    ...
);
```
At the pool boundary, `msg.sender` is the router address. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`.

**Existing guards are insufficient:** There is no mechanism in the extension, the pool, or the router to forward or verify the original end-user identity. The `extensionData` field is available but the extension does not decode it for identity resolution.

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` intends to restrict swaps to a specific set of addresses. Once the router is allowlisted (a necessary step to support the standard periphery swap path), the allowlist check passes for every user who routes through the router, regardless of whether that user is individually allowlisted. This breaks the core curation invariant of the pool — unauthorized users can trade against the pool, drain liquidity at oracle-anchored prices, and interact with pools designed to be restricted (e.g., KYC-gated, institutional-only, or whitelist-only pools). The direct consequence is unauthorized access to pool liquidity and potential loss of LP principal through trades that should have been blocked.

## Likelihood Explanation
The trigger requires the pool admin to allowlist the router — a routine, non-malicious action that any operator of a curated pool supporting the standard periphery would take. The exploit itself requires no special privilege: any unprivileged user can call `MetricOmmSimpleRouter` with a standard swap. The combination of a common admin action and a trivially reachable public path makes this medium-to-high likelihood on any curated pool that also supports router-based swaps.

## Recommendation
The `SwapAllowlistExtension` must gate the end user, not the direct caller of the pool. The cleanest fix: when `sender` is a recognized periphery contract (e.g., `MetricOmmSimpleRouter`), the extension should decode the actual user from `extensionData` and apply the allowlist check to that address. The router must encode the original `msg.sender` into `extensionData` before calling `pool.swap()`. Alternatively, document and enforce at the factory or extension configuration level that the router must never be allowlisted on a curated pool using `SwapAllowlistExtension`.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured on `beforeSwap`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-based swaps.
3. Pool admin does **not** call `setAllowedToSwap(pool, attacker, true)`.
4. Attacker calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the curated pool.
5. The router calls `pool.swap(...)` — `msg.sender` at the pool is the router address.
6. Pool calls `_beforeSwap(msg.sender=router, ...)` → extension receives `sender=router`.
7. `beforeSwap` evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds.
8. Attacker successfully swaps in a pool they were explicitly not authorized to access.

The wrong value is `allowedSwapper[pool][router]` being evaluated instead of `allowedSwapper[pool][attacker]` — the router's allowlist entry is inherited by every end user who routes through it.
