Audit Report

## Title
Missing Deadline Check in `MetricOmmPoolLiquidityAdder` Allows Stale Oracle-Driven Composition on Liquidity Additions — (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

All four public entry points of `MetricOmmPoolLiquidityAdder` (`addLiquidityExactShares` × 2, `addLiquidityWeighted` × 2) accept no `deadline` parameter and perform no timestamp check before interacting with the pool. A block proposer can withhold a pending `addLiquidityWeighted` transaction until the oracle price has moved adversely, causing the LP to deposit at a bin composition materially different from their intent, resulting in a direct, unrecoverable loss of principal bounded only by the user-supplied `maxAmountToken0`/`maxAmountToken1` caps.

## Finding Description

`MetricOmmSwapRouterBase._checkDeadline` is defined at lines 91–94 and is called at entry by every swap function in `MetricOmmSimpleRouter` (`exactInputSingle` L68, `exactInput` L93, `exactOutputSingle` L131, `exactOutput` L155). No equivalent guard exists anywhere in `MetricOmmPoolLiquidityAdder`.

The critical path is `addLiquidityWeighted` (L88–116 and L123–149). It executes a probe `addLiquidity` call that always reverts via `LiquidityProbe(need0, need1)` to read the current token composition from the pool at execution time. `_scaleWeightsToShares` (L226–243) then computes `scaleWad = min(max0/need0, max1/need1)` and scales the user's weight vector accordingly. Because `need0`/`need1` are entirely determined by the pool cursor position at execution time — which is driven by the oracle price at that moment — any oracle price movement between broadcast and inclusion changes the composition the LP actually deposits.

The existing `_validateBinAndBinPosition` guard (L263–286) checks the cursor's `int8` bin index and `uint104` intra-bin position against user-supplied bounds. This is insufficient for two reasons: (1) oracle price can move significantly within a single bin without changing `curBinIdx`, and (2) users who set wide bounds (or any bounds that encompass the post-movement cursor) receive no protection at all. The `maxAmountToken0`/`maxAmountToken1` caps are absolute ceilings on token pull amounts, not composition guards — they do not prevent the LP from depositing the full cap of a token that has depreciated.

## Impact Explanation

Direct loss of LP principal. A validator delays `addLiquidityWeighted` until token0 depreciates. The probe then returns `need0` >> `need1`, `scaleWad` is set by the token0 leg, and the LP deposits up to `maxAmountToken0` of the now-cheaper token0. The LP receives shares priced at the new oracle mid but paid in tokens at the value they approved at broadcast time. The difference — up to the full `maxAmountToken0` cap — is an unrecoverable loss of principal. This meets the Sherlock "direct loss of user principal" threshold at Medium/High severity depending on pool size and price movement magnitude.

## Likelihood Explanation

No special privilege beyond block-production ordering is required. On Ethereum, Base, and HyperEVM, block proposers routinely observe the mempool and can selectively delay individual transactions. Liquidity additions are typically large transactions, making the expected profit positive even for modest oracle price moves. The attack is repeatable across any pool and any LP who uses `addLiquidityWeighted` without a deadline. The only precondition is that the oracle price moves enough between broadcast and inclusion to shift the probe composition beyond the attacker's gas cost — a condition that is frequently met in volatile markets.

## Recommendation

Add a `uint256 deadline` parameter to all four public entry points and call `_checkDeadline` (or an equivalent inline check) before any pool interaction:

```solidity
function addLiquidityWeighted(
    address pool,
    address owner,
    uint80 salt,
    LiquidityDelta calldata weightDeltas,
    uint256 maxAmountToken0,
    uint256 maxAmountToken1,
    int8 minimalCurBin,
    uint104 minimalPosition,
    int8 maximalCurBin,
    uint104 maximalPosition,
    uint256 deadline,          // ← add
    bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    if (block.timestamp > deadline) revert TransactionExpired(deadline, block.timestamp);
    ...
}
```

Apply the same change to both overloads of `addLiquidityExactShares` and both overloads of `addLiquidityWeighted`.

## Proof of Concept

1. LP calls `addLiquidityWeighted` with `maxAmountToken0 = 10_000e18`, `maxAmountToken1 = 10_000e6` (USDC), targeting a 50/50 deposit at oracle price $1.00/token0. `minimalCurBin`/`maximalCurBin` are set to allow ±1 bin of drift.
2. Block proposer observes the transaction in the mempool and withholds it.
3. Oracle price of token0 drops to $0.50 within the same bin (no bin index change, so `_validateBinAndBinPosition` passes). Probe returns `need0 = 20_000e18`, `need1 = 1e6`.
4. `scaleWad0 = 10_000e18 * WAD / 20_000e18 = 0.5 WAD`; `scaleWad1 = 10_000e6 * WAD / 1e6 = 10_000 WAD`; `scaleWad = 0.5 WAD`.
5. Block proposer includes the transaction. LP deposits `~10_000e18` token0 (worth $5,000 at new price) and `~0.5e6` USDC — total deposit worth ~$5,000.50 instead of the intended ~$10,000.
6. LP has lost ~$5,000 of principal with no recourse. A Foundry fork test can reproduce this by manipulating the oracle feed between the broadcast block and the inclusion block and asserting the actual token amounts pulled against the intended 50/50 split.
