### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc()` Allows usdcE Drain Without Providing USDC - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the return value and without using the `safeTransferFrom` wrapper that is already available via `using ERC20Helper for IERC20Base`. The same function uses `safeTransfer` for the outgoing leg on the very next line, making the omission inconsistent. If the USDC `transferFrom` returns `false` rather than reverting (e.g. due to insufficient allowance or a non-reverting token implementation), execution continues, the usdcE is withdrawn from the `DirectDepositV1` vault and sent to `msg.sender` — with no USDC ever deposited in exchange.

---

### Finding Description

`ContractOwner` declares `using ERC20Helper for IERC20Base` at the top of the contract, giving every `IERC20Base` instance access to both `safeTransfer` and `safeTransferFrom`. [1](#0-0) 

`ERC20Helper.safeTransferFrom` wraps the low-level call and requires `success && (data.length == 0 || abi.decode(data, (bool)))`, so a `false` return value causes a revert. [2](#0-1) 

Inside `replaceUsdcEWithUsdc`, the three-step swap is:

1. Pull USDC from `msg.sender` into `directDepositV1` — **raw, unchecked `transferFrom`** (line 616).
2. Withdraw usdcE from `directDepositV1` to `ContractOwner` (line 617).
3. Push usdcE from `ContractOwner` to `msg.sender` — **`safeTransfer`** (line 618). [3](#0-2) 

Step 1 discards the `bool` returned by `IERC20Base.transferFrom`. [4](#0-3) 

If that call returns `false` (no revert), steps 2 and 3 still execute: usdcE leaves the `DirectDepositV1` vault and arrives in `msg.sender`'s wallet, while `directDepositV1` receives zero USDC.

---

### Impact Explanation

Any `DirectDepositV1` vault on chain 57073 (Ink) that holds a non-zero usdcE balance can be drained of its usdcE by a caller who provides zero USDC. The attacker receives usdcE for free; the vault subaccount loses its bridged collateral with no replacement. This is a direct, concrete asset loss for the subaccount owner whose `directDepositV1Address` mapping entry is targeted.

---

### Likelihood Explanation

The function is `external` with no `onlyOwner` or `onlyDeployer` guard — the only gate is `block.chainid == 57073`. [5](#0-4) 

The exploit fires whenever the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` returns `false` instead of reverting on a failed `transferFrom` (e.g. zero allowance). Ink is a newer chain; the bridged USDC deployed there may be a non-standard or wrapped variant whose failure mode is a `false` return rather than a revert. Even if the current deployment reverts, the inconsistency with the rest of the codebase (which uniformly uses `safeTransfer`/`safeTransferFrom`) represents a latent integration risk that becomes critical if the token address is ever updated or if the same pattern is copied to a chain where the token does not revert.

---

### Recommendation

Replace the raw `transferFrom` call with the safe wrapper already available on the type:

```solidity
// Before (unsafe)
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe)
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ERC20Helper.safeTransferFrom` is already in scope via `using ERC20Helper for IERC20Base` and handles both reverting and non-reverting token implementations uniformly. [2](#0-1) 

---

### Proof of Concept

1. Identify any `subaccount` whose `directDepositV1Address` entry is non-zero and whose `DirectDepositV1` vault holds usdcE (balance > 0).
2. Call `ContractOwner.replaceUsdcEWithUsdc(subaccount)` from an EOA that has **zero** USDC allowance granted to `ContractOwner`.
3. If USDC at `0x2D270e6886d130D724215A266106e6832161EAEd` returns `false` on the `transferFrom` (rather than reverting):
   - Line 616 silently fails; `directDepositV1` receives no USDC.
   - Line 617 executes: `DirectDepositV1.withdraw(usdcE)` transfers the full usdcE balance to `ContractOwner`.
   - Line 618 executes: `safeTransfer` sends the full usdcE balance to `msg.sender`.
4. Result: attacker holds usdcE; `directDepositV1` holds neither usdcE nor USDC. [6](#0-5)

```text
    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
