Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks Caller-Supplied `owner` Instead of Actual `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of the pool call) and validates only `owner`, which is a free caller-supplied argument to `MetricOmmPool.addLiquidity`. Any unprivileged address can bypass the deposit allowlist by naming any allowlisted address as `owner`, while the pool pulls tokens from the actual (unauthorized) caller via the callback mechanism. This fully nullifies the pool admin's deposit access control.

## Finding Description

In `DepositAllowlistExtension.beforeAddLiquidity`, the first parameter (`sender`) is unnamed and discarded; the guard checks only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`owner` is a free caller-supplied argument in `MetricOmmPool.addLiquidity`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L182-191
function addLiquidity(address owner, uint80 salt, LiquidityDelta calldata deltas, ...) external ... {
    ...
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both `sender` (the real `msg.sender`) and `owner` to the extension — the bug is entirely in the extension's choice to ignore `sender`:

```solidity
// metric-core/contracts/ExtensionCalling.sol L88-98
function _beforeAddLiquidity(address sender, address owner, ...) internal {
    _callExtensionsInOrder(
        BEFORE_ADD_LIQUIDITY_ORDER,
        abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
    );
}
```

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` and discards the second positional argument, demonstrating the intended pattern:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-40
function beforeSwap(address sender, address, bool, int128, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
```

The asymmetry is the root cause: the deposit guard checks the position recipient (`owner`), while the swap guard checks the actual caller (`sender`).

## Impact Explanation

The pool pulls tokens from the actual caller (`msg.sender` of `addLiquidity`) via the callback mechanism (`callbackData`); `owner` only receives the resulting LP shares. An unauthorized address `bob` calls `pool.addLiquidity(owner=alice, ...)` where `alice` is allowlisted. The extension evaluates `allowedDepositor[pool][alice]` → passes. The pool executes the liquidity addition and fires the callback on `bob`, pulling `bob`'s tokens. `alice` receives LP shares; `bob` has deposited into a pool that was supposed to reject them. The pool admin's primary mechanism for running a permissioned liquidity pool is fully bypassed — this is a direct admin-boundary break where an unprivileged path circumvents configured access control with no special role or privilege required.

## Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with `owner` set to any address already on the allowlist. The allowlist mappings are public, so any observer can identify a valid `owner` to name. No flash loans, reentrancy, oracle manipulation, or special privileges are needed. The bypass is executable in a single transaction by any address.

## Recommendation

Replace the `owner` check with a `sender` check, consistent with `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the token provider and the position owner, check both `sender` and `owner`.

## Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` configured; only `alice` is allowlisted via `setAllowedToDeposit(pool, alice, true)`.
2. `bob` (not allowlisted) calls `pool.addLiquidity(owner=alice, salt=0, deltas=..., callbackData=..., extensionData=...)`.
3. Pool calls `_beforeAddLiquidity(sender=bob, owner=alice, ...)` → extension receives `(bob, alice, ...)`.
4. `DepositAllowlistExtension` evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
5. Pool executes liquidity addition; callback fires on `bob`, pulling `bob`'s tokens into the pool.
6. `alice` receives LP shares; `bob` has successfully deposited into a pool that was configured to reject them.
