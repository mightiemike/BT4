### Title
`get_proposer()` Uses Total Committee Weight Including Ineligible Proposers, Binding Wrong Signer to Block Proposals — (`crates/apollo_staking/src/staking_manager.rs`)

### Summary

`Committee::get_proposer()` generates a random value in `[0, total_weight)` where `total_weight` is computed from **all** committee members, including those with `can_propose = false`. The `choose_proposer()` function then selects from all members without filtering for eligibility. This means `get_proposer()` can return an ineligible proposer, causing the consensus manager to bind the wrong signer to the expected block proposal and reject valid proposals from eligible proposers.

### Finding Description

In `fetch_and_build_committee()`, `cumulative_weights` and `total_weight` are built by iterating over all `committee_members` regardless of their `can_propose` flag:

```rust
let cumulative_weights: Vec<u128> = committee_members
    .iter()
    .scan(0, |acc, staker| {
        *acc = u128::checked_add(*acc, staker.weight.0)...
        Some(*acc)
    })
    .collect();
let total_weight = *cumulative_weights.last().unwrap_or(&0);
```

`eligible_proposers` is computed separately by filtering on `can_propose`:

```rust
let eligible_proposers =
    self.calculate_eligible_proposers(&committee_members, &active_config.stakers);
```

The `Committee` struct stores both, but `get_proposer()` only uses `total_weight` (from all members) as the random range:

```rust
let random_value = self.random_generator.generate(height, round, self.total_weight);
let proposer = self.choose_proposer(random_value);
```

`choose_proposer()` iterates over `cumulative_weights` and returns from `committee_members` with no eligibility check. The `eligible_proposers` field is never consulted in this path.

This is gated by `use_only_actual_proposer_selection`. The `StakingManagerStaticConfig::default()` sets it to `false`, and the canonical `config_schema.json` also defaults it to `false`. Production deployment configs override it to `true`, but any deployment relying on the schema default is exposed.

The consensus manager calls `get_proposer()` via `get_proposer_for_height()` to validate incoming proposals:

```rust
let Ok(proposer) =
    get_proposer_for_height(&self.committee_provider, init.height, init.round).await
...
if proposer != init.proposer {
    warn!("Invalid proposer for height {} and round {}: expected {:?}, got {:?}", ...);
    return Ok(VecDeque::new());
}
```

When `get_proposer()` returns an ineligible proposer, the consensus manager expects that ineligible node to propose. Valid proposals from eligible proposers are rejected because `proposer != init.proposer`. The round times out, and the next round begins with another potentially ineligible selection.

### Impact Explanation

This is a direct analog of the external bug: the external contract used total liquidity weight (including ineligible positions) as the denominator for reward distribution, diluting eligible positions. Here, `get_proposer()` uses total committee weight (including ineligible proposers) as the random range, diluting the selection probability of eligible proposers and allowing ineligible proposers to be selected.

The probability of selecting an eligible proposer is `eligible_weight / total_weight` instead of 1.0. If ineligible proposers hold significant weight, the consensus can stall for many rounds. The consensus manager binds the wrong signer to the expected block proposal, causing valid proposals to be dropped.

**Impact: High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

### Likelihood Explanation

Triggered when:
1. `use_only_actual_proposer_selection = false` (the default in `StakingManagerStaticConfig::default()` and `config_schema.json`)
2. At least one committee member has `can_propose = false`

The production deployment configs explicitly set the flag to `true`, but the schema default is `false`. Any deployment using the default schema without explicit override is vulnerable. The `can_propose = false` condition is a normal operational state (e.g., during epoch transitions or when a staker is not yet ready to propose).

### Recommendation

In `get_proposer()`, build a separate `eligible_cumulative_weights` and `eligible_total_weight` from only the eligible proposers, and use those for random selection. Alternatively, assert at committee construction time that `total_weight` equals `eligible_proposers_total_weight` when `use_only_actual_proposer_selection = false`, or always use `get_actual_proposer()` (round-robin over eligible proposers only) and remove the weighted path entirely.

### Proof of Concept

Given a committee with two members:
- `STAKER_A`: weight=9000, `can_propose=false`
- `STAKER_B`: weight=1000, `can_propose=true`

`total_weight = 10000`. `get_proposer()` generates a random value in `[0, 10000)`. `STAKER_A` is selected 90% of the time. When `STAKER_A` is selected, the consensus manager expects `STAKER_A` to propose. `STAKER_A` is ineligible and does not propose. The round times out. `STAKER_B` (the only eligible proposer) has its valid proposals rejected 90% of the time because `proposer != init.proposer`.

The existing test `get_actual_proposer_some_eligible` (line 792 of `staking_manager_test.rs`) demonstrates the eligibility filtering for `get_actual_proposer()` but there is no equivalent test for `get_proposer()` with mixed eligibility, confirming the gap.     fn choose_proposer(&self, random: u128) -> ContractAddress {
        let total_weight = self.total_weight;
        assert!(
            random < self.total_weight,
            "Invalid random value {random}: exceeds total weight limit of {total_weight}."
        );

        // Iterates over stakers and selects staker `i` if `random < cumulative_weights[i]`.
        // Each staker occupies a range of values proportional to their weight, defined as:
        //     [cumulative_weights[i - 1], cumulative_weights[i])
        // Since we iterate in order, the first staker whose cumulative weight exceeds `random`
        // is the one whose range contains it.
        for (i, cum_weight) in self.cumulative_weights.iter().enumerate() {
            if random < *cum_weight {
                return self
                    .committee_members
                    .get(i)
                    .expect(
                        "Inconsistent committee data; cumulative_weights and committee_members \
                         are not the same length.",
                    )
                    .address;
            }
        }

        // We should never reach this point.
        panic!("Inconsistent committee data; cumulative_weights inconsistent with total weight.")
    }

/// Fetches the committee for the given height and returns the proposer for that height and round.
async fn get_proposer_for_height(
    committee_provider: &Arc<dyn CommitteeProvider>,
    height: BlockNumber,
    round: Round,
) -> Result<ValidatorId, ConsensusError> {
    let committee = committee_provider
        .get_committee(height)
        .await
        .map_err(|e| ConsensusError::CommitteeError(e.to_string()))?;
    committee.get_proposer(height, round).map_err(|e| ConsensusError::CommitteeError(e.to_string()))
}
