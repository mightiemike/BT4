### Title
`AddToTierPosition` Does Not Update `pos.Delegation.Shares` After Delegating Added Tokens — (`File: x/tieredrewards/keeper/msg_server.go`)

### Summary

`AddToTierPosition` calls `delegate()` to stake additional tokens into a position's delegator sub-account, but the returned `newShares` are never added to `pos.Delegation.Shares` before `setPosition` persists the position. Every subsequent exit path (`TierUndelegate`, `ExitTierWithDelegation`, `ForceFullExitWithDelegation`) reads the stale share count from state and therefore only undelegates/transfers the original shares, leaving the added delegation permanently locked under the position's delegator address.

---

### Finding Description

In `AddToTierPosition` (`x/tieredrewards/keeper/msg_server.go`, lines 286–345):

```go
newShares, err := ms.delegate(ctx, delAddr, valAddr, msg.Amount)
if err != nil {
    return nil, err
}

if err := ms.setPosition(ctx, pos.Position, nil); err != nil {   // pos.Delegation.Shares is STALE
    return nil, err
}
```

`delegate()` creates a real staking delegation from `delAddr` to `valAddr` and returns the newly minted shares. Those shares are captured in `newShares` but are **never added to `pos.Delegation.Shares`**. `setPosition` then persists the position with the pre-`AddToTierPosition` share count.

Compare with `LockTier` (lines 61–65), where the position is created *after* delegation so it naturally captures the correct share count from the staking module at creation time. `AddToTierPosition` loads an existing position, adds to the delegation, but fails to reconcile the stored share count.

The corrupted invariant is:

```
pos.Delegation.Shares  <  actual staking delegation shares held by delAddr
```

Every exit path then reads the stale `pos.Delegation.Shares`:

- **`TierUndelegate`** (line 182): `ms.undelegate(ctx, delAddr, valAddr, pos.Delegation.Shares)` — only undelegates the original shares; added shares remain bonded under `delAddr`.
- **`ExitTierWithDelegation`** (line 543): `ms.reconcileAmountFromShares(ctx, valAddr, pos.Delegation.Shares)` — computes a `positionAmount` that is too low; `transferDelegationFromPosition` transfers only the original amount.
- **`ForceFullExitWithDelegation`** (`x/tieredrewards/keeper/force_exit.go`, line 52): same `reconcileAmountFromShares` call on stale shares.

After any of these exits, `deletePosition` removes the position record. The remaining delegation under `delAddr` (a module-controlled sub-account the user has no key for) is permanently inaccessible.

---

### Impact Explanation

A user who calls `AddToTierPosition` one or more times and then exits via any path loses all tokens added after the initial `LockTier`. Those tokens remain bonded under the position's delegator address, which is a deterministically derived module sub-account. No message exists to recover them once the position is deleted. The loss is proportional to the total amount added via `AddToTierPosition`.

---

### Likelihood Explanation

`AddToTierPosition` is a standard user-facing message reachable by any position owner. The full lifecycle (lock → add → exit) is explicitly tested in the integration suite (`test_entry_add_then_exit_flow`). Any user who adds to a position and then exits will trigger the loss. Likelihood is high for any active user of the tiered rewards system.

---

### Recommendation

After calling `delegate()`, add the returned shares to the position before persisting:

```go
newShares, err := ms.delegate(ctx, delAddr, valAddr, msg.Amount)
if err != nil {
    return nil, err
}

// Fix: accumulate the new shares into the stored position
pos.Delegation.Shares = pos.Delegation.Shares.Add(newShares)

if err := ms.setPosition(ctx, pos.Position, nil); err != nil {
    return nil, err
}
```

This mirrors how `LockTier` implicitly captures the correct share count at position creation and ensures `TierUndelegate`/`ExitTierWithDelegation`/`ForceFullExitWithDelegation` operate on the full delegation.

---

### Proof of Concept

1. User calls `LockTier` with `amount = X` → position created with `Delegation.Shares = S_X`.
2. User calls `AddToTierPosition` with `amount = Y` → staking module now holds `S_X + S_Y` shares under `delAddr`, but position still stores `Delegation.Shares = S_X`.
3. User calls `TriggerExitFromTier`, waits for exit duration.
4. User calls `TierUndelegate` → `undelegate(delAddr, valAddr, S_X)` — only `X` tokens begin unbonding; `Y` tokens remain bonded.
5. After unbonding completes, user calls `WithdrawFromTier` → receives only `X` tokens (spendable balance of `delAddr`); position is deleted.
6. `Y` tokens remain delegated under `delAddr` with no recovery path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
