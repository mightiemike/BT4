Audit Report

## Title
`SwapAllowlistExtension` allowlist bypass via router intermediary тАФ (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which is `msg.sender` of `pool.swap()` тАФ the direct caller, not the originating user. When `MetricOmmSimpleRouter` is used, the pool receives the router address as `sender`. Any pool admin who allowlists the router to enable legitimate users to swap via the standard periphery simultaneously grants unrestricted swap access to every user of the public, permissionless router, completely defeating the allowlist invariant.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` verbatim as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // direct caller of pool.swap()
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that exact address against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly without forwarding the originating `msg.sender`:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    ...,
    params.extensionData
);
```

So when any user calls `exactInputSingle`, the pool sees `msg.sender = router`, and the extension checks `allowedSwapper[pool][router]`. There is no mechanism in `beforeSwap` to inspect `extensionData` for the true caller's identity тАФ the parameter is accepted but ignored entirely. The dilemma is irresolvable under the current design: allowlisting the router to support any legitimate user grants access to all users of the public router.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., KYC-verified addresses, designated market makers, or whitelisted institutions) can be fully bypassed by any user routing through `MetricOmmSimpleRouter`. Once the router is allowlisted тАФ a necessary step for any allowlisted user who wants to use the standard periphery тАФ the restriction is void. Non-allowlisted users can execute swaps against LP positions that were never intended to be exposed to them, exposing LP funds to toxic flow or arbitrage the allowlist was designed to prevent. This constitutes a direct loss of LP principal and a broken core pool functionality (the allowlist extension fails to enforce its stated invariant).

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap entry point. Any pool admin who deploys a `SwapAllowlistExtension` and also wants their allowlisted users to access the standard router will naturally allowlist the router address тАФ this is the expected and foreseeable configuration. Once in place, the bypass requires no special knowledge or privilege: any user simply calls `exactInputSingle` or `exactInput` targeting the restricted pool. The attack is repeatable and requires no elevated access.

## Recommendation
The extension must gate on the **actual end user**, not the intermediary. Viable approaches:

1. **Extension-data identity forwarding**: The router encodes `msg.sender` into `extensionData`; the extension verifies a trusted-forwarder proof or signature over that identity. Requires coordinated changes to both the router and extension.
2. **Router-aware allowlist**: Add a separate `allowedRouter` mapping; when `sender` is a known router, require that `recipient` (or an identity embedded in `extensionData`) is individually allowlisted.
3. **Recipient-based gating**: Gate on `recipient` rather than `sender` when the pool is known to be router-mediated. Weaker but avoids the router-identity problem.

## Proof of Concept
```
1. Admin deploys pool with SwapAllowlistExtension.
2. Admin calls setAllowedToSwap(pool, userA, true)    // allowlist a legitimate user
3. Admin calls setAllowedToSwap(pool, router, true)   // allowlist router so userA can use it
4. Non-allowlisted userB calls:
       MetricOmmSimpleRouter.exactInputSingle({
           pool: restrictedPool,
           recipient: userB,
           ...
       })
5. Router calls pool.swap(userB, ...) тАФ msg.sender = router.
6. Pool calls _beforeSwap(router, ...).
7. SwapAllowlistExtension checks allowedSwapper[pool][router] == true тЖТ passes.
8. userB's swap executes against LP funds never intended to be accessible to them.
