### Title
Missing `max_l2_gas_amount` Admission Check for Declare Transactions Allows Gateway Bypass - (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary
The `StatelessTransactionValidator::validate_resource_bounds` function enforces a configured `max_l2_gas_amount` cap on L2 gas for Invoke and DeployAccount transactions, but explicitly skips this check for Declare transactions. An unprivileged user can submit a Declare transaction with an arbitrarily large `l2_gas.max_amount` (up to `u64::MAX`) and the gateway will accept it, bypassing the admission control limit that exists for all other transaction types.

### Finding Description
In `crates/apollo_gateway/src/stateless_transaction_validator.rs`, the `validate_resource_bounds` function contains the following logic:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

The `if let RpcTransaction::Declare(_) = tx { }` branch is an empty no-op — it intentionally skips the `max_l2_gas_amount` check for Declare transactions. The configured default limit is `1_210_000_000` (from `StatelessTransactionValidatorConfig::default()`). The test `valid_l2_gas_amount_on_declare` in `stateless_transaction_validator_test.rs` explicitly confirms this bypass: a Declare transaction with `max_amount: GasAmount(200)` passes validation when `max_l2_gas_amount: 100`. The negative test `test_invalid_max_l2_gas_amount` only covers `TransactionType::DeployAccount` and `TransactionType::Invoke`, explicitly excluding `Declare`.

The analog to the external bug is direct: just as the flashloan code was missing `require(amountDaiNeeded <= flashLender.maxFlashLoan(...))` before proceeding, the sequencer gateway is missing `require(l2_gas.max_amount <= config.max_l2_gas_amount)` before admitting a Declare transaction.

### Impact Explanation
Any unprivileged user can submit a well-formed Declare transaction (valid Sierra class, valid signature, valid nonce) with `resource_bounds.l2_gas.max_amount` set to `u64::MAX` or any value exceeding `max_l2_gas_amount`. The gateway's stateless validator will accept it. The transaction will be forwarded to the stateful validator and then to the mempool. The mempool and batcher will process it without any further check on this field at the admission layer. This matches the impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation
Trivially triggerable by any external caller with a valid Starknet account. No special privilege is required. The attacker only needs to craft a Declare transaction with an oversized `l2_gas.max_amount`. The code path is exercised on every Declare submission and the bypass is unconditional.

### Recommendation
Remove the empty `if let RpcTransaction::Declare(_) = tx { }` branch and apply the same `max_l2_gas_amount` check uniformly to all transaction types, or introduce a separate `max_l2_gas_amount_declare` config value if Declare transactions legitimately require a different limit. The TODO comment already acknowledges this gap.

```rust
// Apply to all transaction types uniformly:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

### Proof of Concept
The existing test `valid_l2_gas_amount_on_declare` in `crates/apollo_gateway/src/stateless_transaction_validator_test.rs` is itself the proof of concept — it passes today and demonstrates the bypass:

```rust
// config: max_l2_gas_amount = 100
// tx: Declare with l2_gas.max_amount = 200
// result: Ok(()) — bypass confirmed
assert_matches!(tx_validator.validate(&tx), Ok(()));
```

A real attacker would set `max_amount = GasAmount(u64::MAX)` in a Declare transaction and submit it to the gateway endpoint. The gateway accepts it, the mempool admits it, and the batcher schedules it — all without the `max_l2_gas_amount` cap being enforced.
