### Title
`setWithdrawPool()` Replaces Pool Address Without Migrating Stranded ERC20 Collateral — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.setWithdrawPool()` swaps the active `withdrawPool` address in a single storage write with no migration of ERC20 token balances held in the old pool. Any collateral already transferred to the old `WithdrawPool` becomes permanently unreachable through the protocol's standard admin recovery path, because `ContractOwner.removeWithdrawPoolLiquidity()` always resolves the pool address dynamically via `clearinghouse.getWithdrawPool()`, which after the switch returns the new pool.

---

### Finding Description

`Clearinghouse.setWithdrawPool()` performs a bare address replacement:

```solidity
function setWithdrawPool(address _withdrawPool) external onlyOwner {
    require(_withdrawPool != address(0));
    withdrawPool = _withdrawPool;
}
``` [1](#0-0) 

Every user withdrawal routes real ERC20 collateral into the active `withdrawPool` via `handleWithdrawTransfer`:

```solidity
function handleWithdrawTransfer(...) internal virtual {
    token.safeTransfer(withdrawPool, uint256(amount));
    BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx);
}
``` [2](#0-1) 

So the old `WithdrawPool` contract accumulates real token balances over time. The only sanctioned admin path to recover those tokens is `ContractOwner.removeWithdrawPoolLiquidity()`:

```solidity
function removeWithdrawPoolLiquidity(...) external onlyOwner {
    BaseWithdrawPool withdrawPool = BaseWithdrawPool(
        clearinghouse.getWithdrawPool()   // <-- always resolves to the NEW pool
    );
    withdrawPool.removeLiquidity(productId, amount, sendTo);
}
``` [3](#0-2) 

After `setWithdrawPool` is called, `clearinghouse.getWithdrawPool()` returns the new pool address. The old pool's token balances are therefore unreachable through `ContractOwner` — the standard operator interface. The admin would need to know the old pool's address and call `removeLiquidity` on it directly, which is not part of the documented or expected operational flow and is easy to miss.

A secondary desynchronization: `BaseWithdrawPool` tracks processed withdrawal indices in `markedIdxs` and `minIdx`. [4](#0-3) 

The new pool starts with `minIdx = 0` and an empty `markedIdxs` mapping. Any in-flight slow-mode withdrawal that was already dispatched to the old pool (i.e., `submitWithdrawal` was called on the old pool) will not be reflected in the new pool's state, creating a desynchronized view of which withdrawals have been settled.

---

### Impact Explanation

- **Stranded collateral**: ERC20 tokens (user collateral) already transferred to the old `WithdrawPool` are inaccessible through the `ContractOwner` admin interface after the pool switch. The new pool starts with zero balance, so any pending withdrawal that relied on the old pool's liquidity will fail.
- **User withdrawal failures**: `submitWithdrawal` on the old pool requires `msg.sender == clearinghouse`; after the switch the clearinghouse routes to the new pool, so the old pool's tokens cannot be disbursed to users through normal settlement.
- **Broken accounting invariant**: The protocol's total withdrawal liquidity is split across two pools with no mechanism to reconcile them through the standard operator path.

---

### Likelihood Explanation

`setWithdrawPool` is an owner-callable upgrade path that would be exercised during any pool migration or upgrade. The stranding is silent — no revert, no event indicating lost liquidity — making it easy to overlook. The `ContractOwner.removeWithdrawPoolLiquidity` misdirection is a concrete code-level trap that makes recovery non-obvious even for a careful operator.

---

### Recommendation

Before updating `withdrawPool`, drain all per-product token balances from the old pool to the clearinghouse (or directly to the new pool), and migrate `minIdx` to the new pool to preserve replay protection:

```solidity
function setWithdrawPool(address _withdrawPool) external onlyOwner {
    require(_withdrawPool != address(0));

    // Migrate state from old pool
    if (withdrawPool != address(0)) {
        BaseWithdrawPool oldPool = BaseWithdrawPool(withdrawPool);
        // Transfer per-product balances to new pool
        uint32[] memory productIds = _spotEngine().getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {
            IERC20Base token = IERC20Base(
                _spotEngine().getConfig(productIds[i]).token
            );
            uint128 bal = uint128(token.balanceOf(address(oldPool)));
            if (bal > 0) {
                oldPool.removeLiquidity(productIds[i], bal, _withdrawPool);
            }
        }
        // Carry forward minIdx to preserve replay protection
        BaseWithdrawPool(_withdrawPool).setMinIdx(oldPool.minIdx());
    }

    withdrawPool = _withdrawPool;
}
```

At minimum, emit an event with the old pool address and its per-product balances so operators can manually recover stranded funds.

---

### Proof of Concept

1. Protocol is live; users have made withdrawals. The old `WithdrawPool` holds 100,000 USDC in token balance (transferred via `handleWithdrawTransfer`).
2. Owner calls `ContractOwner.setWithdrawPool(newPool)` → `Clearinghouse.setWithdrawPool(newPool)` → `withdrawPool = newPool`.
3. Owner later calls `ContractOwner.removeWithdrawPoolLiquidity(QUOTE_PRODUCT_ID, amount, treasury)`.
4. Inside that function, `clearinghouse.getWithdrawPool()` returns `newPool` (not `oldPool`). The call targets the new pool, which has zero balance. The 100,000 USDC in `oldPool` is never recovered through this path.
5. Any user whose withdrawal was in-flight at the time of the switch has their funds stuck in `oldPool`. The clearinghouse now routes all new `submitWithdrawal` calls to `newPool`, so `oldPool` can never disburse those funds via the normal settlement path (it requires `msg.sender == clearinghouse`).
