### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function is still reachable by any unprivileged NEAR caller. It lacks the coinbase-proof guard introduced in `verify_transaction_inclusion_v2`, allowing an attacker to supply an internal Merkle-tree node hash as `tx_id` and receive a `true` proof-verification result for a transaction that was never included in the block.

---

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` but carries no access-control restriction beyond the shared `#[pause]` flag. [1](#0-0) 

When the contract is not paused, any NEAR account can call it directly. The function's sole verification step is: [2](#0-1) 

It calls `compute_root_from_merkle_proof(tx_id, tx_index, &merkle_proof)` and compares the result to the stored `merkle_root`. It does **not** verify that `tx_id` is a leaf-level transaction hash rather than an internal Merkle-tree node. The contract's own warning acknowledges this: [3](#0-2) 

The 64-byte transaction forgery attack exploits this directly. An attacker crafts a 64-byte blob whose double-SHA256 hash equals an internal node of the Merkle tree, then supplies:
- `tx_id` = hash of the internal node
- `tx_index` = the position of that internal node in the tree
- `merkle_proof` = the sibling hashes from that internal node to the root

`compute_root_from_merkle_proof` in `merkle-tools` traverses the proof without any leaf-vs-internal-node distinction: [4](#0-3) 

The function returns `true`, falsely certifying that the crafted blob was included as a transaction in the block.

`verify_transaction_inclusion_v2` mitigates this by requiring a coinbase proof of equal length, which forces the target to be at the same tree depth as the coinbase (leaf level): [5](#0-4) 

But v1 remains callable and unguarded.

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate a financial action (e.g., releasing bridged funds, minting tokens, settling a cross-chain settlement) can be tricked into accepting a forged proof. The proof-verification result is corrupted: the function returns `true` for a transaction that does not exist. This directly undermines the SPV security guarantee that is the contract's primary purpose.

---

### Likelihood Explanation

The function is publicly callable with no role restriction. The 64-byte forgery technique is well-documented and requires only knowledge of a block's Merkle tree structure, which is fully public Bitcoin data. Any attacker who can identify a downstream contract relying on v1 can execute this attack without any privileged access or leaked keys.

---

### Recommendation

Remove `verify_transaction_inclusion` entirely from the public ABI, or gate it with an access-control role so it cannot be called by unprivileged accounts. All downstream integrators must be migrated to `verify_transaction_inclusion_v2`. The deprecation marker alone is insufficient enforcement — the function must be made unreachable at the protocol level.

---

### Proof of Concept

1. Identify a mainchain block with hash `B` and a known Merkle tree (all data is public on Bitcoin).
2. Compute the hash of an internal Merkle node `N` at depth 1 (one level above leaves).
3. Construct `merkle_proof` = the sibling hashes from `N` to the root (length = `tree_height − 1`).
4. Call `verify_transaction_inclusion` with:
   - `tx_id` = `N`
   - `tx_block_blockhash` = `B`
   - `tx_index` = position of `N` in the tree at that depth
   - `merkle_proof` = constructed proof
   - `confirmations` = 1
5. `compute_root_from_merkle_proof` reconstructs the correct root from `N` upward; the comparison `== header.block_header.merkle_root` succeeds; the function returns `true`, falsely certifying inclusion of a non-existent transaction. [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
