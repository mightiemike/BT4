### Title
Linked Signer Can Overwrite Subaccount Authorization State, Permanently Hijacking Signing Authority — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction type in `processTransactionImpl` accepts signatures from the **existing linked signer** (`allowLinkedSigner = true`). This allows a linked signer — a session key or dapp-controlled address — to replace itself with an attacker-controlled address, permanently mutating the subaccount's authorization state without the subaccount owner's knowledge or consent.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch validates the transaction with `allowLinkedSigner = true`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // ← allowLinkedSigner
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

`validateSignedTx` calls `validateSignature` with the current linked signer as an accepted signer:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`validateSignature` in `Verifier.sol` accepts the recovered address if it equals either the subaccount owner **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

The result: the current linked signer can sign a `LinkSigner` transaction that writes any new address into `linkedSigners[subaccount]`, overwriting the authorization mapping without the subaccount owner's direct signature. [4](#0-3) 

The `LinkSigner` EIP-712 struct commits only to `sender`, `signer`, and `nonce` — there is no field binding the signer to the subaccount owner's identity:

```
"LinkSigner(bytes32 sender,bytes32 signer,uint64 nonce)"
``` [5](#0-4) 

---

### Impact Explanation

The corrupted state is `linkedSigners[subaccount]` — the mapping that determines who may sign all privileged transactions on behalf of a subaccount.

A malicious or compromised linked signer can:

1. Sign a `LinkSigner` transaction setting `signer` to an attacker-controlled address.
2. The attacker's address is now the accepted linked signer for the subaccount.
3. The attacker signs `WithdrawCollateral` / `WithdrawCollateralV2` (both accept `allowLinkedSigner = true`) to drain all collateral.
4. The original owner loses signing authority. Even if they attempt recovery via the slow-mode `LinkSigner` path, the attacker can front-run by continuously re-submitting a new `LinkSigner` through the fast sequencer path, creating a persistent race condition. [6](#0-5) 

The concrete asset delta: full collateral balance of the victim subaccount across all spot products, drained via `Clearinghouse.withdrawCollateral`. [7](#0-6) 

---

### Likelihood Explanation

**Medium.** Linked signers are explicitly designed for dapp-delegated or bot-driven execution. Any user who grants a third-party dapp or trading bot linked signer authority is exposed. The user's mental model is that the linked signer is a scoped session key for trading — not an entity that can permanently reassign authorization. A malicious dapp, a compromised bot key, or a supply-chain attack on a trading frontend can trigger this with a single sequencer-submitted transaction. No admin access, governance capture, or key compromise beyond the already-trusted linked signer is required.

---

### Recommendation

`LinkSigner` transactions should only be signable by the **subaccount owner directly**. Change `allowLinkedSigner` to `false` for the `LinkSigner` branch in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // ← must be owner, not linked signer
);
``` [8](#0-7) 

This mirrors the slow-mode path, which already enforces `validateSender(txn.sender, sender)` — requiring `msg.sender` to be the subaccount owner — before writing to `linkedSigners`. [9](#0-8) 

---

### Proof of Concept

1. Alice calls `submitSlowModeTransaction` with a `LinkSigner` transaction, setting `signer = sessionKey` (a dapp-controlled address). `linkedSigners[Alice_subaccount] = sessionKey`.
2. The dapp constructs a new `SignedLinkSigner` with `sender = Alice_subaccount`, `signer = attacker_address`, `nonce = current_nonce`, signed by `sessionKey`.
3. The sequencer includes this in a `submitTransactionsChecked` batch.
4. `processTransactionImpl` reaches the `LinkSigner` branch. `validateSignedTx(..., true)` recovers `sessionKey`, which equals `getLinkedSigner(Alice_subaccount)` — validation passes.
5. `linkedSigners[Alice_subaccount] = attacker_address`.
6. The attacker signs a `WithdrawCollateralV2` transaction (with `sendTo = address(0)`, so `allowLinkedSigner = true`) and submits it via the sequencer.
7. `Clearinghouse.withdrawCollateral` drains Alice's collateral to `address(uint160(bytes20(Alice_subaccount)))` — or to any address if `sendTo != address(0)` and the attacker signs directly. [10](#0-9)
