### Title
Stale Chain State Race Condition After Unpause Enables False SPV Proof Acceptance — (File: `contract/src/lib.rs`)

---

### Summary

When the BTC light client contract is paused and then unpaused, there is a race condition between the relayer (which must submit all blocks missed during the pause via `submit_blocks`) and any unprivileged NEAR caller (which can immediately invoke `verify_transaction_inclusion` / `verify_transaction_inclusion_v2`). If a Bitcoin chain reorganization occurred during the pause period, the contract's frozen chain state causes the verification functions to return `true` for a transaction that has been reorganized out of the canonical Bitcoin chain.

---

### Finding Description

The contract derives `Pausable` and applies `#[pause]` to both `submit_blocks` and the two verification entry points: [1](#0-0) [2](#0-1) [3](#0-2) 

When the contract is paused, `submit_blocks` is blocked, so the relayer cannot advance the on-chain chain state. The Bitcoin network, however, continues to produce blocks and may produce a reorganization. The contract's canonical chain state — `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height` — is frozen at the height it was at when the pause was applied.

When the contract is unpaused, **both** `submit_blocks` and `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` become callable simultaneously. The relayer must submit every block missed during the pause before the contract's view of the chain is accurate. During this catch-up window, any NEAR caller can invoke the verification functions against the stale state.

`verify_transaction_inclusion` determines main-chain membership by looking up the block hash in `mainchain_header_to_height`: [4](#0-3) 

If a reorg occurred during the pause, the reorganized-out blocks are still present in `mainchain_header_to_height` (the contract never saw the reorg). The function therefore returns `true` for a transaction in a reorganized-out block. Only after the relayer submits the reorg headers does `reorg_chain()` remove the displaced blocks from the index: [5](#0-4) 

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after its coinbase check, so it is equally affected: [6](#0-5) 

---

### Impact Explanation

The corrupted on-chain value is the `mainchain_header_to_height` mapping, which incorrectly maps reorganized-out block hashes to heights. Any downstream NEAR contract that gates an action (e.g., releasing bridged funds) on a `true` result from `verify_transaction_inclusion` can be tricked into accepting a proof for a Bitcoin transaction that no longer exists in the canonical chain. The attacker-controlled entry path is a direct call to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` immediately after the unpause, before the relayer has submitted the reorg headers.

---

### Likelihood Explanation

**Low.** Four conditions must coincide: (1) the contract is paused — a normal operational event by a `PauseManager`-role account; (2) a Bitcoin reorganization occurs during the pause window — natural but infrequent, probability grows with pause duration; (3) the attacker holds or can reference a transaction confirmed in the reorganized-out block; (4) the attacker calls the verification function before the relayer's catch-up transactions are finalized on NEAR. The window is narrow for short pauses but widens for upgrade-related pauses that can last hours or days.

---

### Recommendation

After unpausing, introduce a mandatory relayer catch-up period before `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are re-enabled. Concretely:

- Record the `mainchain_tip_blockhash` (and its height) at the moment of pausing.
- Keep the verification functions in a separate paused state that is only lifted once `submit_blocks` has advanced the tip beyond the recorded height, confirming the relayer has caught up.
- Alternatively, expose a privileged `unpause_verification()` call that the operator invokes only after confirming the relayer is synchronized, mirroring the recommendation in the external report to "make it possible instantly to supply more collateral" while delaying the adversarial action.

---

### Proof of Concept

1. Contract is paused at Bitcoin height H; `mainchain_tip_blockhash` = `B_H`; `mainchain_header_to_height[B_{H-K}]` = `H-K`.
2. Bitcoin network mines blocks `B_{H+1}…B_{H+N}` and a reorg replaces `B_{H-K}…B_H` with `B'_{H-K}…B'_H`. The contract sees none of this.
3. Admin unpauses the contract. Both `submit_blocks` and `verify_transaction_inclusion` are now open.
4. Attacker immediately calls `verify_transaction_inclusion` with `tx_block_blockhash = B_{H-K}.hash` and a valid Merkle proof for transaction `TX` that was in the now-reorganized-out block.
5. Contract executes `mainchain_header_to_height.get(B_{H-K}.hash)` → returns `H-K` (stale entry still present). [7](#0-6) 

6. Confirmation count check passes (stale tip `B_H` is still recorded as the heaviest block).
7. Merkle proof validates against the stored `merkle_root` of `B_{H-K}`.
8. Function returns `true`; downstream contract releases funds to attacker.
9. Relayer subsequently submits the reorg headers; `reorg_chain()` removes `B_{H-K}` from `mainchain_header_to_height` — but the false proof has already been consumed. [8](#0-7)
