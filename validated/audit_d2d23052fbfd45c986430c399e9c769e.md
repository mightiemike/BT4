### Title
Silent `AllResources`→`L1Gas` Downgrade in Protobuf Deserialization Produces Wrong Transaction Hash - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` silently downgrades a V3 transaction carrying `AllResources` bounds (with `l2_gas = 0` and `l1_data_gas = 0`) into the `L1Gas` variant. Because `get_tip_resource_bounds_hash` produces a structurally different hash preimage for the two variants (3 vs 4 resource-bound elements), the transaction hash recomputed after P2P deserialization diverges from the hash the signer committed to, binding the wrong hash to the transaction.

### Finding Description

In `crates/apollo_protobuf/src/converters/transaction.rs` the conversion from a protobuf `ResourceBounds` message to the internal `ValidResourceBounds` enum is:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // silently zero when absent
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)                      // ← wrong variant
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

A user may legitimately submit a Starknet 0.13.3+ (`AllResources`) V3 transaction with `l2_gas = 0` and `l1_data_gas = 0` — for example, a pure L1-gas-only transaction submitted through the new API. The gateway accepts it, computes the hash with `AllResources` semantics, and the user signs that hash. When the block is later propagated over P2P sync, the protobuf message carries `l1_data_gas = 0` (or omits the field entirely for 0.13.2 backward-compat). The deserializer then silently selects `ValidResourceBounds::L1Gas` instead of `ValidResourceBounds::AllResources`.

The hash function `get_tip_resource_bounds_hash` produces a **different output** for the two variants:

```rust
let mut resource_felts = vec![
    get_concat_resource(&l1_resource_bounds, L1_GAS)?,
    get_concat_resource(&l2_resource_bounds, L2_GAS)?,
];
// For new V3 txs, need to also hash the data gas bounds.
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // ← 2 elements
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // ← 3 elements
    }
});
``` [2](#0-1) 

`AllResources` with `l1_data_gas = 0` hashes `poseidon(tip, L1_GAS_packed, L2_GAS_packed, L1_DATA_GAS_packed_zero)` — four elements. `L1Gas` hashes `poseidon(tip, L1_GAS_packed, L2_GAS_packed)` — three elements. These are distinct field elements even when the numeric values of the bounds are identical.

The hash is recomputed from the deserialized transaction in `convert_rpc_tx_to_internal`:

```rust
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
Ok((InternalRpcTransaction { tx: tx_without_hash, tx_hash }, proof_data))
``` [3](#0-2) 

The recomputed hash diverges from the hash the account signed. `validate_transaction_hash` then fails:

```rust
possible_hashes.push(get_transaction_hash(transaction, chain_id, transaction_options)?);
Ok(possible_hashes.contains(&expected_hash))
``` [4](#0-3) 

The Cairo OS also enforces the same 3-element `AllResources` hash:

```rust
assert n_resource_bounds = 3;
``` [5](#0-4) 

### Impact Explanation

**High — Transaction conversion or signature/hash logic binds the wrong hash.**

A valid, correctly-signed V3 `AllResources` transaction (with `l2_gas = 0`, `l1_data_gas = 0`) is silently re-typed to `L1Gas` during P2P sync deserialization. The hash recomputed from the downgraded type does not match the hash the signer committed to. Depending on where the recomputed hash is consumed:

- **P2P sync path**: `validate_transaction_hash` rejects the transaction, causing the syncing node to reject an otherwise valid block — a valid transaction is refused before sequencing.
- **RPC trace/simulation path**: the RPC returns an authoritative-looking wrong transaction hash for the affected transaction.

### Likelihood Explanation

**Low.** The trigger requires a user to submit a 0.13.3+ `AllResources` transaction with both `l2_gas = 0` and `l1_data_gas = 0`. This is an unusual but protocol-valid configuration (e.g., a transaction that intentionally sets no L2 or data-gas budget). No privileged access is required; any user can craft such a transaction.

### Recommendation

The `unwrap_or_default` fallback for `l1_data_gas` must not silently collapse the variant. Two options:

1. **Preserve the variant explicitly**: add a boolean flag or a separate protobuf field that encodes whether the original transaction was `AllResources` or `L1Gas`, and use that flag during deserialization instead of inferring from zero values.
2. **Reject missing `l1_data_gas` for post-0.13.2 messages**: enforce the TODO already present in the code — once 0.13.2 support is dropped, assert that `l1_data_gas` is always present and remove the `unwrap_or_default` path entirely.

### Proof of Concept

```
1. Craft a V3 InvokeTransaction with:
     resource_bounds = AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }
   
2. Compute hash H_all = poseidon(INVOKE, version, sender,
       get_tip_resource_bounds_hash(AllResources{...}),   // 4-element inner hash
       ..., chain_id, nonce, ...)
   
3. Sign H_all with the account key.

4. Submit the transaction; gateway accepts it with hash H_all.

5. Block is produced and propagated via P2P sync.

6. Receiving node deserializes the protobuf ResourceBounds:
     l1_data_gas = unwrap_or_default() → 0
     l2_gas = 0
     → ValidResourceBounds::L1Gas(l1_gas)   ← wrong variant

7. Recompute hash H_l1 = poseidon(INVOKE, version, sender,
       get_tip_resource_bounds_hash(L1Gas{...}),          // 3-element inner hash
       ..., chain_id, nonce, ...)

8. H_l1 ≠ H_all  →  validate_transaction_hash returns false
   → valid block/transaction rejected by the syncing node.
```

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
