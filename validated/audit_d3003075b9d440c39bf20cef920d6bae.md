### Title
Bonus Rewards Permanently Lost When RewardsPoolName Is Empty During Redelegation Slash — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`processEventsAndClaimBonus` mutates the `pos` struct (advancing `LastEventSeq`, `LastBonusAccrual`, `LastKnownBonded`) and decrements on-chain event reference counts **before** the `sufficientBonusPoolBalance` check at line 230 of `claim_rewards.go`. When the pool is empty and the check fails, the function returns `ErrInsufficientBonusPool` — but the mutations have already occurred. `slashRedelegationPosition` in `slash.go` explicitly swallows that error and then calls `setPositionWithState` with the already-mutated `pos`, permanently advancing the position's reward checkpoints without ever sending the bonus coins to the owner.

---

### Finding Description

**Step 1 — Entrypoint**

The `BeforeRedelegationSlashed` staking hook routes to `slashRedelegationPosition`: [1](#0-0) 

This is a production path triggered by the Cosmos SDK staking module whenever a redelegation entry is slashed.

**Step 2 — processEventsAndClaimBonus mutates pos before the pool check**

Inside `processEventsAndClaimBonus`, the event loop at lines 172–198 calls `pos.UpdateLastEventSeq(entry.Seq)` and `k.decrementEventRefCount` for every processed event: [2](#0-1) 

After the loop, `applyBonusAccrualCheckpoint` and `UpdateLastKnownBonded` advance the remaining checkpoints: [3](#0-2) 

Only **after** all these mutations does the pool balance check occur: [4](#0-3) 

If the pool is empty, `ErrInsufficientBonusPool` is returned — but `pos.LastEventSeq`, `pos.LastBonusAccrual`, `pos.LastKnownBonded` are already advanced, and the on-chain `ValidatorEvent.ReferenceCount` values are already decremented (potentially garbage-collecting the events).

**Step 3 — slashRedelegationPosition swallows the error and persists the mutated state** [5](#0-4) 

The `ErrInsufficientBonusPool` branch logs and continues. Both the partial-slash path (line 77) and the full-slash path (line 71) then call `setPositionWithState` with the already-mutated `pos`. The advanced checkpoints are written to the store.

**Step 4 — Future ClaimTierRewards finds nothing to pay**

When the owner later calls `ClaimTierRewards`, `processEventsAndClaimBonus` reads `pos.LastEventSeq` and `pos.LastBonusAccrual` from the store. Because they were already advanced past the slash event, `getValidatorEventsSince` returns no new events for the pre-slash bonded segment, and `segmentStart` equals the already-advanced `LastBonusAccrual`. The bonus for the bonded segment that existed before the slash is never computed again. [6](#0-5) 

**Step 5 — Contradiction with documented invariant**

The ADR documentation explicitly states: *"If pool balance < bonus: fail atomically (user retries later)"*. The slash path violates this: it does not fail atomically — it persists the advanced checkpoints without paying the bonus, making retry impossible. [7](#0-6) 

---

### Impact Explanation

The position owner permanently loses all accrued bonus rewards for every bonded segment between `pos.LastBonusAccrual` and the slash event time. The loss is irreversible: the checkpoints are advanced, the event ref counts are decremented (events may be garbage-collected), and no recovery path exists. This constitutes a direct, permanent loss of user funds from the `RewardsPoolName` module account that the user was entitled to receive.

---

### Likelihood Explanation

The `RewardsPoolName` balance is consumed by normal reward claims and can reach zero during periods of high claim activity or insufficient pool replenishment. Validator slashing of redelegations is a standard Cosmos SDK production event. The two conditions — empty pool and a redelegation slash — can coincide without any attacker action; a malicious actor can also deliberately drain the pool (via their own legitimate claims) before a known slash event to guarantee the loss for targeted positions.

---

### Recommendation

Move all `pos` mutations and `decrementEventRefCount` calls to **after** the `sufficientBonusPoolBalance` check, or restructure `processEventsAndClaimBonus` to compute `totalBonus` in a read-only first pass and only mutate state in a second pass that is gated on the pool check passing. Alternatively, in `slashRedelegationPosition`, when `ErrInsufficientBonusPool` is caught, do **not** call `setPositionWithState` with the mutated `pos`; instead persist the original (pre-mutation) position so the owner can retry the claim once the pool is replenished.

---

### Proof of Concept

```
1. Create a tier position with a bonded validator (MsgLockTier).
2. Redelegate the position to a second validator (MsgTierRedelegate),
   recording the unbondingID from the redelegation mapping.
3. Advance block time by 30 days so a non-zero bonus accrues.
4. Drain the RewardsPoolName balance to zero (e.g., via ClaimTierRewards
   on other positions until the pool is empty).
5. Call BeforeRedelegationSlashed(ctx, unbondingID, sharesToUnbond)
   where sharesToUnbond < pos.Delegation.Shares (partial slash).
   → processEventsAndClaimBonus advances LastEventSeq/LastBonusAccrual,
     decrements event ref counts, then returns ErrInsufficientBonusPool.
   → slashRedelegationPosition swallows the error and calls setPositionWithState
     with the mutated pos.
6. Refill the RewardsPoolName with sufficient funds.
7. Call ClaimTierRewards for the position.
   → processEventsAndClaimBonus starts from the already-advanced LastEventSeq
     and LastBonusAccrual; no pre-slash segment is found; totalBonus == 0.
8. Assert: owner received zero bonus despite 30 days of bonded accrual.
```
