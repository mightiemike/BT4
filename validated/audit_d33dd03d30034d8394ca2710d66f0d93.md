Audit Report

## Title
Deposit Allowlist Checks `owner` Instead of `sender`, Allowing Non-Allowlisted Callers to Bypass the Gate â€” (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument and gates access on the caller-supplied `owner` parameter instead. Because `MetricOmmPool.addLiquidity` accepts any address as `owner`, any non-allowlisted caller can bypass the gate by nominating an allowlisted address as `owner`, causing the attacker's tokens to be pulled via callback while LP shares are credited to the allowlisted address.

## Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first parameter but leaves it unnamed and unused. The allowlist check is performed exclusively on `owner`: [1](#0-0) 

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` and passes `msg.sender` as `sender` to `_beforeAddLiquidity`: [2](#0-1) 

Because `owner` is a free parameter in the public signature, any caller can supply an allowlisted address as `owner`. The extension evaluates `allowedDepositor[pool][allowlisted_address]` which is `true`, and the gate passes unconditionally for the non-allowlisted caller.

After the gate passes, `LiquidityLib.addLiquidity` records the position under `owner` and fires the liquidity callback on `msg.sender` (the attacker) to pull tokens: [3](#0-2) 

The position key is derived from `owner`, so shares are credited to the allowlisted address: [4](#0-3) 

`removeLiquidity` enforces `msg.sender == owner`, so the attacker cannot recover the deposited tokens â€” they are permanently transferred to the allowlisted address's position: [5](#0-4) 

## Impact Explanation

The deposit allowlist is the pool admin's mechanism to restrict which addresses may add liquidity to a curated pool. The invariant â€” only allowlisted addresses may act as the depositing party â€” is broken. Any address can deposit into a restricted pool by nominating an allowlisted address as `owner`. This constitutes an admin-boundary break: an unprivileged path bypasses an admin-configured access control gate. The attacker's tokens are irreversibly transferred into the pool under the allowlisted address's position, constituting a direct loss of the attacker's principal and unauthorized mutation of pool state in a restricted pool.

## Likelihood Explanation

The bypass requires only a direct call to the public `pool.addLiquidity` with an allowlisted address as `owner`. No special privileges, flash loans, or complex setup are needed. Allowlisted addresses are publicly discoverable via the `AllowedToDepositSet` event or the `allowedDepositor` view function. Likelihood is high.

## Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositing party) rather than `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0)
