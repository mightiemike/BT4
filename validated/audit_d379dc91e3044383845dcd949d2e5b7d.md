### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Proof Mitigation — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is marked `#[deprecated]` but remains a fully live, publicly callable NEAR contract method with no role restriction. Any unprivileged caller can invoke it directly, bypassing the coinbase Merkle proof check introduced in `verify_transaction_inclusion_v2`, and obtain a `true` return value for a forged internal Merkle node as `tx_id`. This applies to all chain builds compiled from `lib.rs`, including the Litecoin build.

---

### Finding Description

`verify_transaction_inclusion` is declared `pub` with only a `#[pause]` gate and a Rust `#[deprecated]` attribute: [1](#0-0) 

The `#[deprecated]` attribute is a **Rust compiler-level lint only**. It emits a warning at compile time for Rust callers but has **zero effect on the compiled WASM ABI**. The method is exported as a normal public NEAR contract function and is callable by any NEAR account without any role check.

`verify_transaction_inclusion_v2` was introduced specifically to mitigate the 64-byte internal-node forgery attack by requiring a valid coinbase Merkle proof before delegating to v1: [2](#0-1) 

The coinbase proof check forces the proof depth to be consistent with a real block (since the coinbase is always at index 0 and its proof length must match the claimed tree depth). Without it, an attacker can supply any internal node hash as `tx_id`.

`verify_transaction_inclusion` itself only calls `compute_root_from_merkle_proof` and compares the result to `header.block_header.merkle_root`: [3](#0-2) 

`compute_root_from_merkle_proof` is a pure hash-chain computation with no transaction-count or depth validation: [4](#0-3) 

The Litecoin module (`contract/src/litecoin.rs`) contains only PoW-related logic and does not override or restrict `verify_transaction_inclusion` in any way: [5](#0-4) 

The shared `lib.rs` implementation is compiled into the Litecoin build via the `litecoin` feature flag: [6](#0-5) 

---

### Impact Explanation

An attacker who knows the internal structure of a real Litecoin block's Merkle tree can:

1. Pick any internal node `N` at depth `d` in the tree.
2. Construct a shortened proof of length `(tree_depth - d)` that walks from `N` up to `merkle_root`.
3. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_index` set to the correct parity path, and the shortened proof.
4. The function returns `true`, asserting that `N` is a valid included transaction.

Any downstream contract or off-chain system that calls `verify_transaction_inclusion` (v1) to gate a state change (e.g., releasing bridged funds, minting tokens, confirming a cross-chain payment) can be deceived into accepting a forged inclusion claim. The v2 mitigation is entirely bypassed because the attacker never calls v2.

---

### Likelihood Explanation

- The function is unconditionally reachable by any NEAR account as long as the contract is not paused.
- No privileged role, staking, or relayer key is required.
- The attack requires only knowledge of a real block's Merkle tree (publicly available from any Litecoin node or block explorer).
- The `#[deprecated]` marker provides no on-chain enforcement whatsoever.
- The CLAUDE.md documentation explicitly acknowledges the v1 vulnerability and directs callers to v2, but the v1 endpoint remains live. [7](#0-6) 

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public NEAR ABI entirely. Since `verify_transaction_inclusion_v2` already calls v1 internally via `#[allow(deprecated)]`, v1 can be refactored into a private helper (`fn verify_inclusion_inner`) with no `pub` visibility and no `#[near]` export. This eliminates the bypass surface while preserving the v2 code path.

---

### Proof of Concept

Given a Litecoin mainchain block with transactions `[T0 (coinbase), T1, T2, T3]`:

```
         merkle_root
        /           \
    N_left         N_right
    /    \          /    \
   T0    T1       T2    T3
```

`N_left = double_sha256(T0 || T1)` is an internal node.

**Forge call (v1 direct):**
```
verify_transaction_inclusion({
    tx_id:              N_left,          // internal node, not a real tx
    tx_block_blockhash: <valid block>,
    tx_index:           0,               // left child parity
    merkle_proof:       [N_right],       // one-element proof to root
    confirmations:      1,
})
```

`compute_root_from_merkle_proof(N_left, 0, [N_right])` computes `double_sha256(N_left || N_right) == merkle_root` → returns `true`.

The same call to `verify_transaction_inclusion_v2` would **panic** at the coinbase proof check because `N_left` is not a valid coinbase transaction hash and its proof would not reconstruct `merkle_root` at index 0 with the correct depth. [8](#0-7)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
