The code confirms the vulnerability claim is partially valid, but the impact differs by class type. Let me trace the full path.

## Conversion Layer (No Hash Verification)

`TryFrom<protobuf::Class> for (ApiContractClass, ClassHash)` blindly trusts the wire-supplied `class_hash`: [1](#0-0) 

No hash is computed from the class body. The `class_hash` field is extracted as-is.

## P2P Sync Client: The `is_declared` Guard

In `parse_data_for_block`, the received `class_hash` is checked against the **already-synced state diff**: [2](#0-1) 

This check only verifies the hash exists in the state diff — it does **not** verify the class body hashes to that value. An attacker peer can send `class_hash = H` (a legitimate hash from the state diff) paired with an arbitrary wrong class body, and the check passes.

## Divergence by Class Type

**Cairo1 (Sierra) path** — `write_to_storage` calls `add_class(class.clone())` without passing the hash: [3](#0-2) 

Inside `ClassManager::add_class`, the hash is **recomputed** from the body: [4](#0-3) 

So for Cairo1, the wrong body is stored under `hash(body)` ≠ H. The class at H is never stored — this is a DoS/missing-class outcome, not wrong-class execution. The developer even left a `TODO` acknowledging the missing cross-check: [5](#0-4) 

**Cairo0 (deprecated) path** — `add_deprecated_class` takes the protobuf-supplied `class_hash` directly as the storage key: [6](#0-5) [7](#0-6) 

No hash recomputation occurs. The wrong class body is stored under H.

## Verdict

### Title
Malicious P2P peer stores wrong Cairo0 class body under a valid ClassHash via unauthenticated protobuf class sync — (`crates/apollo_p2p_sync/src/client/class.rs`, `crates/apollo_protobuf/src/converters/class.rs`)

### Summary
A malicious peer responding to a `ClassesRequest` can send a `protobuf::ClassesResponse` where `class_hash = H` (a hash legitimately present in the already-synced state diff) but with a Cairo0 class body whose actual hash ≠ H. The conversion layer trusts the wire hash, the `is_declared` guard only checks set membership, and `add_deprecated_class` stores the body under the supplied key without recomputation. The wrong class body is durably stored under H.

### Finding Description
`TryFrom<protobuf::Class> for (ApiContractClass, ClassHash)` at `class.rs:75-76` extracts `class_hash` from the wire without computing or comparing it against the class body. The downstream guard in `parse_data_for_block` (`client/class.rs:133,137`) only checks that the hash appears in the state diff — it does not verify `hash(body) == class_hash`. For the Cairo0 branch, `add_deprecated_class(class_hash, deprecated_class)` stores the body under the attacker-controlled hash with no further verification.

### Impact Explanation
Wrong compiled class selected for execution. Any contract deployed at an address whose class hash is H will execute the attacker-substituted Cairo0 bytecode instead of the legitimate one. State transitions, storage writes, events, and L1 messages produced by that contract will be wrong. This maps to the **Critical** impact tier: *Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution.*

### Likelihood Explanation
Requires the attacker to be a responding P2P peer (not an operator/admin). Any node that connects to a malicious peer during the class sync phase is vulnerable. The state diff must already be synced (normal sequencing order), but that is the ordinary operating condition. The `TODO` comment at `client/class.rs:39` confirms the missing check is a known gap.

### Recommendation
After `add_class` / `add_deprecated_class`, compare the returned or recomputed class hash against the protobuf-supplied hash and treat a mismatch as a `BadPeerError`, disconnecting the peer. For Cairo0, compute the deprecated class hash from the body before calling `add_deprecated_class` and reject if it does not match the state-diff hash.

### Proof of Concept
```rust
// Construct a protobuf::Class with mismatched class_hash
let real_class = make_cairo0_protobuf_class(); // valid Cairo0 body
let wrong_hash = some_other_declared_class_hash; // H from state diff, hash(body) != H
let protobuf_class = protobuf::Class {
    class: Some(protobuf::class::Class::Cairo0(real_class)),
    class_hash: Some(wrong_hash.into()),
    domain: 0,
};
let (api_class, returned_hash) =
    <(ApiContractClass, ClassHash)>::try_from(protobuf_class).unwrap();
// returned_hash == wrong_hash, not hash(body) — no error raised
assert_eq!(returned_hash, wrong_hash); // passes: wrong hash accepted
```
When this pair passes `parse_data_for_block`'s `is_declared` check (because `wrong_hash` is in the state diff), `add_deprecated_class(wrong_hash, body)` stores the wrong body permanently.
