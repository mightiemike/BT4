Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the originating user, allowing any caller to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` evaluates the `sender` argument, which `MetricOmmPool.swap` sets to its own `msg.sender` — the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the extension checks `allowedSwapper[pool][router]` rather than the actual end-user's entry. Any pool admin who allowlists the router to enable router-mediated swaps inadvertently grants every user on the network the ability to bypass the per-user allowlist.

## Finding Description

**Root cause:** `MetricOmmPool.swap` passes `msg.sender` (the direct caller) as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← router address, not the end-user
    recipient,
    ...
);
``` [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then evaluates that argument against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [2](#0-1) 

Here `msg.sender` is the pool and `sender` is the router, so the check resolves to `allowedSwapper[pool][router]`.

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly with no user identity forwarded in the extension path. The only place `msg.sender` (the real user) is stored is in the transient callback context via `_setNextCallbackContext`, which is used exclusively for the payment callback and is never passed to extensions:

```solidity
// MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData   // ← user-supplied, no enforced user identity
    );
``` [3](#0-2) 

The same pattern applies to `exactInput` (L104-112), `exactOutputSingle` (L136-137), and `exactOutput` (L165-181). [4](#0-3) 

**Existing guards are insufficient:** The only guard is `allowedSwapper[pool][sender]` in the extension. Once the router is allowlisted, this guard passes for every caller regardless of their individual allowlist status. There is no secondary check on `tx.origin`, `extensionData`, or any other user-identifying field.

## Impact Explanation
The `SwapAllowlistExtension` is designed to restrict swaps to a curated set of counterparties (e.g., KYC'd users, institutional traders, whitelisted protocols). Once the router is allowlisted, this restriction is completely nullified for all router-mediated paths. Any unauthorized user can execute swaps on a restricted pool, receiving output tokens at oracle-anchored prices and draining pool liquidity that the pool operator intended to reserve for approved counterparties only. This constitutes a broken core pool access-control mechanism with direct potential for loss of pool assets to unauthorized parties.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary production entry point for end-users. Any pool that deploys `SwapAllowlistExtension` and also expects users to interact via the router must allowlist the router — this is the only way to enable router-mediated swaps on a restricted pool. This is the standard operational configuration. No special privileges, flash loans, or unusual token behavior are required; a plain `exactInputSingle` call from any EOA suffices. The bypass is repeatable on every such deployment. [5](#0-4) 

## Recommendation
Pass the originating user's identity through the extension rather than the direct `pool.swap()` caller. Two options:

1. **Preferred — encode the user address in `extensionData` at the router level.** `MetricOmmSimpleRouter` encodes `msg.sender` into `extensionData` before calling `pool.swap()`; `SwapAllowlistExtension.beforeSwap` decodes and checks that address. This requires a trusted router or a signed payload to prevent spoofing.

2. **Alternative — check `recipient` instead of `sender`.** The recipient receives output tokens and cannot be spoofed without economic loss to the attacker, though it still allows routing output to an allowlisted address.

Additionally, update `SwapAllowlistExtension` NatSpec to document that `sender` is the direct caller of `pool.swap()`, not the originating user, so pool admins understand the implication of allowlisting router contracts. [6](#0-5) 

## Proof of Concept

```
Setup (Foundry integration test):
  pool = deploy MetricOmmPool with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is the only approved user
  allowedSwapper[pool][router] = true  // admin adds router to enable router swaps

Attack:
  bob (not individually allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: bob, ...})
      → pool.swap(recipient=bob, ...) [msg.sender = router]
        → _beforeSwap(sender=router, ...)
          → SwapAllowlistExtension.beforeSwap(sender=router, ...)
            → allowedSwapper[pool][router] == true → PASSES
  bob successfully swaps despite not being individually allowlisted

Verification:
  Assert bob's output token balance increased.
  Assert the swap would revert if bob called pool.swap() directly
  (allowedSwapper[pool][bob] == false).
``` [7](#0-6)
