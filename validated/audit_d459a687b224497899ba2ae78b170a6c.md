Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` validates position recipient (`owner`) instead of the actual token payer, enabling allowlist bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` checks whether the LP-position recipient (`owner`) is allowlisted, but the address that actually provides tokens is `msg.sender` of the `MetricOmmPoolLiquidityAdder` call, stored separately in transient storage. Because `addLiquidityExactShares(pool, owner, ...)` accepts any non-zero `owner` with no requirement that `owner == msg.sender`, an unprivileged caller can name an allowlisted address as `owner`, pass the extension check, and have their own tokens deposited into the restricted pool.

## Finding Description
`DepositAllowlistExtension.beforeAddLiquidity` performs:

```solidity
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension's caller) and `owner` is the position recipient supplied by the original caller. The actual token payer is never exposed to the extension.

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (owner-explicit overload) validates `owner` only for the zero address:

```solidity
function _validateOwner(address owner) internal pure {
    if (owner == address(0)) revert InvalidPositionOwner();
}
``` [2](#0-1) 

The payer is always `msg.sender` of the adder call, stored in transient context separately from `owner`:

```solidity
return _addLiquidity(pool, owner, salt, deltas, msg.sender, maxAmountToken0, maxAmountToken1, extensionData);
``` [3](#0-2) 

The pool's `addLiquidity` passes `msg.sender` (the adder contract) as `sender` and the caller-supplied `owner` as the position owner to the extension:

```solidity
_callExtensionsInOrder(
  BEFORE_ADD_LIQUIDITY_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
);
``` [4](#0-3) 

The pool's `addLiquidity` passes `msg.sender` (the adder contract — identical for every user) as `sender`: [5](#0-4) 

The extension therefore cannot identify the real payer. It can only see `sender` (the adder contract, constant across all users) or `owner` (the position recipient, caller-controlled). It checks `owner`.

**Attack path:**
1. Pool admin deploys a pool with `DepositAllowlistExtension`; only `alice` is allowlisted.
2. Unauthorized `bob` approves the adder for his tokens.
3. `bob` calls `addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, "")`.
4. `_validateOwner(alice)` passes (non-zero).
5. Pool calls `beforeAddLiquidity(adderContract, alice, ...)`.
6. Extension checks `allowedDepositor[pool][alice]` → `true` → no revert.
7. Callback pulls tokens from `bob` (the payer in transient storage).
8. `alice` receives the LP position; `bob`'s tokens enter the restricted pool.

The allowlist is fully bypassed. `bob` is not allowlisted, yet his funds enter the pool.

## Impact Explanation
The deposit allowlist is the pool admin's primary mechanism for restricting who can provide liquidity (e.g., for regulatory compliance, KYC-gated pools, or risk management). The bypass allows any unprivileged address to inject funds into a restricted pool. The pool receives unauthorized liquidity, altering its composition and potentially affecting existing LPs' value. This constitutes a broken core pool functionality causing unauthorized fund entry — a direct violation of the allowlist invariant the extension is designed to enforce.

## Likelihood Explanation
The `MetricOmmPoolLiquidityAdder` is the standard periphery entry point for EOAs. The owner-explicit overload `addLiquidityExactShares(pool, owner, ...)` is a documented, publicly callable function. Any actor who knows a single allowlisted address (observable on-chain via `AllowedToDepositSet` events) can execute the bypass without any special privilege. The only cost is the tokens deposited, which the attacker voluntarily provides. The attack is repeatable and requires no elevated access.

## Recommendation
Two complementary fixes:

1. **In `DepositAllowlistExtension.beforeAddLiquidity`**: check `sender` (the direct pool caller, i.e., the adder contract) rather than `owner`. However, since `sender` is the adder contract (identical for all users), this alone is insufficient — the adder must enforce `owner == msg.sender` so the payer and position recipient are always the same address.

2. **In `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (owner-explicit overload)**: enforce `owner == msg.sender` so the payer and position recipient are always the same address, making the extension's `owner` check equivalent to a payer check.

Alternatively, thread the real payer address through `extensionData` so the extension can validate it directly, though this requires the adder to be trusted to populate it correctly.

## Proof of Concept
```solidity
// Pool configured with DepositAllowlistExtension; only alice is allowlisted.
// bob is NOT allowlisted.

// bob approves the adder for his tokens
token0.approve(address(adder), type(uint256).max);
token1.approve(address(adder), type(uint256).max);

// bob calls with alice as owner — allowlist check passes on alice, bob pays
vm.prank(bob);
adder.addLiquidityExactShares(
    address(pool),
    alice,          // owner: allowlisted → check passes
    uint80(1),
    deltas,
    1_000e18,
    1_000e18,
    ""
);

// Result: bob's tokens are in the pool, alice holds the LP position
// The deposit allowlist did not block bob
assertGt(pool.positionBinShares(alice, 1, binIdx), 0);
```
