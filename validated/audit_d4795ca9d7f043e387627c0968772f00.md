### Title
Protocol Fee Account Earns Compounding Deposit Interest on Accrued Fees, Causing Over-Collection and Potential Withdrawal DOS - (`core/contracts/SpotEngineState.sol`)

---

### Summary

In `SpotEngineState._updateState()`, protocol fees (`feesAmt`) are calculated in actual asset terms but credited to `FEES_ACCOUNT` as a **normalized balance** (shares). Because normalized balances grow with `cumulativeDepositsMultiplierX18` in every subsequent state update, the `FEES_ACCOUNT` continuously earns deposit interest on top of the original fee. This causes the protocol to claim more than the intended `INTEREST_FEE_FRACTION` (20%) of borrower interest. Under high utilization, the compounding `FEES_ACCOUNT` balance can exceed available pool liquidity, DOSing regular depositor withdrawals.

---

### Finding Description

Each time `updateStates(dt)` is called (triggered by the sequencer via `Endpoint`), `_updateState()` computes the fee portion of interest and credits it to `FEES_ACCOUNT`:

```solidity
// core/contracts/SpotEngineState.sol lines 119-145
int128 feesAmt = totalDeposits.mul(
    totalDepositRateX18 - realizedDepositRateX18   // assets
);
// ...
if (feesAmt != 0) {
    BalanceNormalized memory feesAccBalance = balances[productId][FEES_ACCOUNT];
    _updateBalanceNormalized(state, feesAccBalance, feesAmt);   // stored as shares
    _setBalanceAndUpdateBitmap(productId, FEES_ACCOUNT, feesAccBalance);
}
``` [1](#0-0) 

`_updateBalanceNormalized` converts `feesAmt` (assets) into a normalized amount by dividing by the **current** `cumulativeDepositsMultiplierX18`:

```solidity
balance.amountNormalized = newAmount.div(cumulativeMultiplierX18);
``` [2](#0-1) 

This normalized balance is then added to `totalDepositsNormalized`:

```solidity
if (balance.amountNormalized > 0) {
    state.totalDepositsNormalized += balance.amountNormalized;
}
``` [3](#0-2) 

In every future call to `_updateState`, `cumulativeDepositsMultiplierX18` is multiplied by `depositRateMultiplierX18`:

```solidity
state.cumulativeDepositsMultiplierX18 = state
    .cumulativeDepositsMultiplierX18
    .mul(depositRateMultiplierX18);
``` [4](#0-3) 

Because `FEES_ACCOUNT.amountNormalized` is part of `totalDepositsNormalized`, the `FEES_ACCOUNT`'s actual balance = `amountNormalized × cumulativeDepositsMultiplierX18` grows in every subsequent period — earning deposit interest on the fee itself. When the protocol eventually claims these fees, it receives more than the original `feesAmt`.

The intended fee is exactly `INTEREST_FEE_FRACTION × totalDepositRateX18 × totalDeposits` per period. The actual fee claimed is that amount **plus all deposit interest earned on the accumulated normalized fee balance** across all subsequent periods.

The `INTEREST_FEE_FRACTION` constant is 20%:

```solidity
int128 constant INTEREST_FEE_FRACTION = 200_000_000_000_000_000; // 20%
``` [5](#0-4) 

---

### Impact Explanation

**Impact 1 — Over-collection of fees (always present):** In any pool with non-zero utilization, the protocol will claim more than 20% of borrower interest. The excess grows with time and utilization, silently extracting value from depositors.

**Impact 2 — Withdrawal DOS (high utilization):** The `FEES_ACCOUNT` balance is part of `totalDepositsNormalized`. Under high utilization (available liquidity = `totalDeposits − totalBorrows` is small), the `FEES_ACCOUNT`'s compounding balance can grow to exceed the available liquidity. When a regular depositor attempts to withdraw, the actual token balance in the contract is insufficient to cover both the depositor's claim and the `FEES_ACCOUNT`'s claim, causing the withdrawal to revert or the `assertUtilization` invariant to break:

```solidity
function assertUtilization(uint32 productId) external view {
    // ...
    require(totalDeposits >= totalBorrows, ERR_MAX_UTILIZATION);
}
``` [6](#0-5) 

---

### Likelihood Explanation

- **Impact 1** is triggered by normal protocol operation — any pool with borrows and regular `updateStates` calls. No special conditions required.
- **Impact 2** requires sustained high utilization (>80–90%) combined with accumulated unclaimed fees. This is a realistic scenario for popular pools with high borrow demand, especially since `updateStates` is called by the sequencer on every transaction batch.

---

### Recommendation

Three mitigations (in order of preference):

1. **Immediate fee transfer:** Instead of crediting `FEES_ACCOUNT` as a normalized balance, transfer `feesAmt` tokens directly to the fee recipient at the time of accrual. This prevents any compounding.

2. **Store fees in assets, not shares:** Track `FEES_ACCOUNT` accruals in a separate `uint128 accruedFeesAssets` variable (not normalized). When claimed, transfer exactly that amount. This correctly captures the intended `INTEREST_FEE_FRACTION` without compounding.

3. **Dilute the deposit multiplier on fee minting:** When new fee shares are minted, adjust `cumulativeDepositsMultiplierX18` downward to account for the new shares, so the conversion back to assets at claim time yields exactly `feesAmt`.

---

### Proof of Concept

**Setup:** Pool with 1000 USDC deposited, 800 USDC borrowed (80% utilization), `INTEREST_FEE_FRACTION = 20%`, `updateStates` called every second for 12 hours.

**Step 1:** Each `updateStates(1)` call accrues `feesAmt` to `FEES_ACCOUNT` as normalized shares.

**Step 2:** In each subsequent call, `cumulativeDepositsMultiplierX18` grows, increasing the `FEES_ACCOUNT`'s actual balance beyond the original `feesAmt`.

**Step 3:** After 12 hours, compute:
```
feesAccountActualBalance = FEES_ACCOUNT.amountNormalized × cumulativeDepositsMultiplierX18
```
This will exceed `sum(feesAmt_per_period)` — the intended 20% of total interest paid.

**Step 4:** Compute `availableLiquidity = totalDeposits − totalBorrows`. Under high utilization, `feesAccountActualBalance > availableLiquidity`.

**Step 5:** Regular depositor attempts to withdraw their full balance. The withdrawal fails because the contract cannot simultaneously honor the depositor's claim and the `FEES_ACCOUNT`'s claim with the available token balance.

The root cause is identical to M-11: fees stored as normalized shares in `SpotEngineState._updateState()` at lines 139–145 of `core/contracts/SpotEngineState.sol` accrue deposit interest, causing over-collection and potential liquidity exhaustion. [7](#0-6)
