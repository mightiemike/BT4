### Title
`ClampOutgoingGasAdmission` Uses Clamped Gas for Admission but Full Gas for Deduction, Causing Limit Over-Consumption - (`File: runtime/runtime/src/congestion_control.rs`)

### Summary

When `ProtocolFeature::ClampOutgoingGasAdmission` is enabled (protocol version ≥ 85, currently active on mainnet), the `try_forward` function in `ReceiptSinkV2` checks admission against a **clamped** `admission_gas = min(gas, allowed_shard_outgoing_gas)` but then deducts the **full unclamped** `gas` from `forward_limit.gas`. This divergence between the admission threshold and the actual deduction causes the per-shard outgoing gas limit to be over-consumed whenever a buffered receipt carries more gas than `allowed_shard_outgoing_gas`, systematically blocking subsequent receipts that should be forwardable within the same chunk.

### Finding Description

In `ReceiptSinkV2::try_forward`:

```rust
let admission_gas = if ProtocolFeature::ClampOutgoingGasAdmission
    .enabled(apply_state.current_protocol_version)
{
    gas.min(apply_state.config.congestion_control_config.allowed_shard_outgoing_gas)
} else {
    gas
};

if forward_limit.gas >= admission_gas && forward_limit.size >= size {
    outgoing_receipts.push(receipt);
    forward_limit.gas = forward_limit.gas.saturating_sub(gas);  // ← full gas, not admission_gas
    ...
}
``` [1](#0-0) 

The admission check on line 451 uses `admission_gas` (clamped to `allowed_shard_outgoing_gas`, currently 1 PGAS), but the deduction on line 454 uses the full `gas`. When `gas > allowed_shard_outgoing_gas`:

- **Admission**: requires only `allowed_shard_outgoing_gas` from `forward_limit.gas`
- **Deduction**: removes the full `gas` from `forward_limit.gas` (via `saturating_sub`, flooring at 0)

**Concrete scenario** (receiver shard at 50% congestion):
- `forward_limit.gas = mix(300 PGAS, 1 PGAS, 0.5) ≈ 150 PGAS`
- Buffered receipt with `gas = 200 PGAS`
- Without feature: `150 PGAS >= 200 PGAS` → fails, receipt stays buffered (stuck)
- With feature: `admission_gas = min(200, 1) = 1 PGAS`; `150 PGAS >= 1 PGAS` → passes
- Deduction: `forward_limit.gas = 150 PGAS - 200 PGAS = 0` (saturating)
- All subsequent receipts in this chunk are blocked because `forward_limit.gas = 0`

The `allowed_shard_outgoing_gas` is 1 PGAS per the congestion control config: [2](#0-1) 

`ClampOutgoingGasAdmission` is assigned to protocol version 85: [3](#0-2) 

The current stable protocol version is 86, so this feature is active on mainnet: [4](#0-3) 

The `outgoing_gas_limit` for a non-local shard is computed from `CongestionControl::outgoing_gas_limit`: [5](#0-4) 

And is stored in `OutgoingLimit::gas`: [6](#0-5) 

### Impact Explanation

The over-consumption of `forward_limit.gas` means that after forwarding one large receipt (with `gas > allowed_shard_outgoing_gas`), the limit drops to 0 via `saturating_sub`. All subsequent buffered receipts for that shard in the same chunk are blocked, even small ones that would have easily passed the admission check. This:

1. **Reduces throughput** when draining the outgoing buffer: instead of forwarding many receipts up to the gas limit, only one large receipt is forwarded per chunk.
2. **Defeats the purpose of `ClampOutgoingGasAdmission`**: the feature was introduced to prevent large receipts from getting stuck, but the over-deduction causes the limit to be exhausted after the first large receipt, leaving subsequent receipts stuck.
3. **Worsens congestion recovery**: when a shard is recovering from congestion, the buffer drains slower than the protocol intends, prolonging the congestion period.

This is a liveness/throughput issue, not a fund-loss issue. However, it systematically degrades congestion control correctness on mainnet (protocol version 86).

### Likelihood Explanation

The condition is triggered whenever:
1. `ClampOutgoingGasAdmission` is enabled — **always true** on mainnet (protocol version 86 ≥ 85).
2. A buffered receipt has `gas > allowed_shard_outgoing_gas` (> 1 PGAS) — **common**, since function calls routinely attach 10–300 PGAS.
3. The receiver shard has any congestion — **common** under load.

Any user who submits a function call with > 1 PGAS attached gas that gets buffered due to congestion will trigger this divergence.

### Recommendation

Clamp the deduction to match the admission check. Replace:

```rust
forward_limit.gas = forward_limit.gas.saturating_sub(gas);
```

with:

```rust
forward_limit.gas = forward_limit.gas.saturating_sub(admission_gas);
```

This ensures the limit is consumed by exactly the amount that was checked during admission, keeping the two values consistent. The `stats` reporting on line 456 should continue to use the full `gas` for accurate accounting of what was actually forwarded.

### Proof of Concept

Given:
- Protocol version 86 (mainnet, `ClampOutgoingGasAdmission` enabled)
- `allowed_shard_outgoing_gas = 1 PGAS`
- Receiver shard at 80% congestion → `outgoing_gas_limit = mix(300 PGAS, 1 PGAS, 0.8) ≈ 60 PGAS`
- Outgoing buffer contains: receipt A with `gas = 100 PGAS`, receipts B–Z each with `gas = 1 PGAS`

**Current behavior:**
1. Receipt A: `admission_gas = min(100, 1) = 1 PGAS`; `60 >= 1` → forwarded; `forward_limit.gas = 60 - 100 = 0`
2. Receipt B: `admission_gas = min(1, 1) = 1 PGAS`; `0 >= 1` → **blocked**
3. Receipts C–Z: all blocked

Only 1 receipt forwarded out of 26.

**Expected behavior (with fix):**
1. Receipt A: `admission_gas = 1 PGAS`; `60 >= 1` → forwarded; `forward_limit.gas = 60 - 1 = 59 PGAS`
2. Receipt B: `admission_gas = 1 PGAS`; `59 >= 1` → forwarded; `forward_limit.gas = 58 PGAS`
3. … continues until limit exhausted

Up to 60 receipts forwarded.
