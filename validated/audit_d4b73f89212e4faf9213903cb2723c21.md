Audit Report

## Title
Router-Mediated Swaps Substitute Router Identity for Caller Identity in `SwapAllowlistExtension::beforeSwap`, Allowing Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension::beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool — not the original EOA. Any pool that allowlists the router (required for allowlisted users to use the router) is fully open to every user of the router, completely nullifying the access control.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the hook.**

In `MetricOmmPool::swap`, the pool calls `_beforeSwap(msg.sender, ...)`, where `msg.sender` is whoever called `pool.swap()`: [1](#0-0) 

`ExtensionCalling::_beforeSwap` forwards that value unchanged as the first argument to every configured extension: [2](#0-1) 

**Step 2 — The hook checks that `sender` argument, not `tx.origin`.**

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool; `sender` is whoever called `pool.swap()`: [3](#0-2) 

**Step 3 — The router calls `pool.swap()` as itself with no mechanism to forward the original caller's identity.**

`exactInputSingle` calls the pool directly: [4](#0-3) 

For multi-hop `exactInput`, hops after the first use `address(this)` (the router) as payer, so the router's identity is presented to every intermediate pool's hook: [5](#0-4) 

**Step 4 — Identity substitution.**

| Call path | `sender` seen by hook |
|---|---|
| EOA → `pool.swap()` directly | EOA address |
| EOA → `router.exactInputSingle()` → `pool.swap()` | **Router address** |

The hook has no way to distinguish these two cases. There is no trusted-router registry, no `originalSender` forwarding, and no `tx.origin` fallback in the extension.

## Impact Explanation

`SwapAllowlistExtension` is the protocol's mechanism for pools to restrict trading to a defined set of counterparties (e.g., compliance whitelists, institutional-only pools). Router-mediated swaps completely nullify this control: any user who routes through `MetricOmmSimpleRouter` is checked as the router, not as themselves. A pool that allowlists the router to support normal UX for its permitted users is fully open to all users of the router. This is a broken core access-control boundary — an unprivileged path bypasses a live access control gate. The wrong value is the `sender` identity presented to the extension: it is the router address instead of the originating EOA, causing `allowedSwapper[pool][router]` to be evaluated instead of `allowedSwapper[pool][EOA]`.

## Likelihood Explanation

The router is the standard entry point for end users. Any pool that uses `SwapAllowlistExtension` and also wants to support router-mediated swaps for its permitted users must allowlist the router, which immediately opens the bypass to all router users. The attack requires no special privileges, no timing, and no prior state manipulation — a single transaction suffices. The attacker only needs to call `router.exactInputSingle(...)` targeting the pool.

## Recommendation

The hook must verify the original initiator, not the immediate caller. The preferred fix is to redesign the hook to accept an authenticated `originalSender` field passed through `extensionData` by a trusted-router registry: the router encodes `msg.sender` into `extensionData`; the hook decodes and verifies it only when `msg.sender` (the pool's caller) is a registered trusted router. Alternatively, require that pools using `SwapAllowlistExtension` interact with the pool directly and document that the router must revert when targeting such pools, though this is operationally fragile.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin: setAllowedToSwap(pool, routerAddress, true)
    (necessary for permitted users A and B to use the router)
  - Pool admin: setAllowedToSwap(pool, attackerEOA, false)
    (attacker is explicitly NOT allowlisted)

Attack (1 transaction):
  - attacker calls router.exactInputSingle({pool: pool, ...})
  - router calls pool.swap(recipient, ...)  [msg.sender = router]
  - pool calls extension.beforeSwap(sender=router, ...)
  - hook checks allowedSwapper[pool][router] → true
  - swap executes successfully for attacker

Result: attacker bypasses the allowlist entirely.
```

Foundry test plan: deploy `SwapAllowlistExtension`, configure a pool with it, allowlist the router, explicitly block an attacker EOA, call `router.exactInputSingle` from the attacker EOA, and assert the swap succeeds (demonstrating the bypass).
