**Key code flow:**

`_handleLiquidationPayment` (lines 579–586):
```solidity

```

`_finalizeSubaccount` (lines 368–411):
```solidity

// ...

```

The design intent (comment lines 581–585): fees from the last liquidation step are added to `insurance`, but must be excluded from the coverage/socialization check so they don't "block" socialization. They are subtracted at line 369 and restored at line 410.

**The stale-value accounting error:**

Let `I` = insurance before, `feesA` = fees from last step of subaccountA, `feesB` = fees from a subsequent liquidation of subaccountB.

| Step | `insurance` | `lastLiquidationFees` |
|---|---|---|
| liquidate subaccountA (last step) | `I + feesA` | `feesA` |
| liquidate subaccountB | `I + feesA + feesB` | `feesB` ← overwrites |
| finalize subaccountA | reads stale `feesB` | — |

During finalization of subaccountA with stale `lastLiquidationFees = feesB`:
- `v.insurance = (I + feesA + feesB) − feesB = I + feesA`

Correct behavior (if `lastLiquidationFees = feesA`):
- `v.insurance = (I + feesA + feesB) − feesA = I + feesB`

**When `feesA > feesB`:** the stale path yields `v.insurance = I + feesA` > correct `I + feesB`. More insurance is available for coverage. If `I + feesA ≥ |quoteBalance_A|` but `I + feesB < |quoteBalance_A|`, the stale path provides **full coverage and skips socialization**, while the correct path would provide only partial coverage and trigger socialization. The insurance fund is over-consumed by `|quoteBalance_A| − (I + feesB)`.

**When `feesA < feesB`:** the stale path yields `v.insurance = I + feesA` < correct `I + feesB`. Less insurance is available; socialization is triggered prematurely, harming other users.

The `canLiquidateMore` check at line 370 is also affected: a stale larger `v.insurance` makes `canLiquidateMore = true`, imposing a stricter spot-asset-zero requirement that may incorrectly block finalization.

**Is this exploitable without sequencer compromise?**

`liquidateSubaccountImpl` is a public external function gated only by `EndpointGated`. A liquidator submits signed `LiquidateSubaccount` transactions to the sequencer. The sequencer processes them in submission order within a batch. A liquidator can submit all three transactions in a single batch:

1. `liquidate(subaccountA, productId=X)` — last normal step, sets `lastLiquidationFees = feesA` (large)
2. `liquidate(subaccountB, productId=Y)` — any small liquidation, sets `lastLiquidationFees = feesB` (small)
3. `liquidate(subaccountA, productId=MAX)` — finalization, reads stale `feesB`

No sequencer compromise is needed; the sequencer faithfully processes the submitted order. The attacker controls both the liquidator subaccount and (optionally) subaccountA to benefit from the extra coverage.

**Conclusion: this is a real, reachable vulnerability.**

### Title
`lastLiquidationFees` Global State Contamination Causes Insurance Fund Over-Consumption During Finalization — (`core/contracts/ClearinghouseLiq.sol`)

### Summary
`lastLiquidationFees` is a single global `int128` storage slot shared across all concurrent liquidations. Its value is unconditionally overwritten by every call to `_handleLiquidationPayment`. When `_finalizeSubaccount` reads it, it may reflect fees from a completely different subaccount's liquidation, corrupting the insurance coverage and socialization logic for the subaccount being finalized.

### Finding Description
`ClearinghouseStorage.sol` declares `lastLiquidationFees` as a contract-level storage variable with no per-subaccount scoping. [1](#0-0) 

Every call to `_handleLiquidationPayment` unconditionally writes to it: [2](#0-1) 

`_finalizeSubaccount` uses this value in two places: first to compute the effective insurance available for coverage/socialization (line 369), and then to restore it after the coverage check (line 410): [3](#0-2) [4](#0-3) 

If any liquidation of a different subaccount executes between the last normal step for subaccountA and its finalization step, `lastLiquidationFees` no longer corresponds to subaccountA. The subtract-then-restore pattern breaks: the wrong amount is excluded from the coverage window, and the wrong amount is restored afterward.

### Impact Explanation
**Over-consumption path (`feesA > feesB`):** The stale small `feesB` is subtracted at line 369, leaving `v.insurance = I + feesA` — larger than the correct `I + feesB`. This inflated value is passed to `perpEngine.socializeSubaccount` and used to compute `insuranceCover`. If `I + feesA ≥ |quoteBalance_A|` but `I + feesB < |quoteBalance_A|`, the insurance fund provides full coverage and the `v.insurance ≤ 0` socialization gate at line 407 is never reached. Losses that should have been socialized across all LPs are instead absorbed by the insurance fund, depleting it by up to `|quoteBalance_A| − (I + feesB)`. [5](#0-4) 

**Premature socialization path (`feesA < feesB`):** The stale large `feesB` is subtracted, leaving `v.insurance = I + feesA` — smaller than correct. Socialization is triggered when the insurance fund actually had enough to cover the deficit, incorrectly spreading losses to other users.

**`canLiquidateMore` distortion:** The same stale subtraction at line 369–370 affects the spot-asset-zero gate, potentially blocking or permitting finalization incorrectly. [6](#0-5) 

### Likelihood Explanation
Any liquidator can trigger this by submitting, in a single sequencer batch:
1. A normal liquidation step for subaccountA (sets `lastLiquidationFees = feesA`)
2. Any liquidation step for subaccountB (overwrites `lastLiquidationFees = feesB`)
3. The finalization step for subaccountA (`productId = type(uint32).max`)

No sequencer compromise is required; the sequencer processes transactions in submission order. The liquidator need not own subaccountA to cause insurance over-consumption — they only need two liquidatable subaccounts available simultaneously, which is routine during market stress when many accounts are under maintenance margin. [7](#0-6) 

### Recommendation
Replace the global `lastLiquidationFees` with a per-subaccount mapping, or pass the fees as a local variable through the call stack rather than persisting them in storage. The simplest fix is to return `v.liquidationFees` from `_handleLiquidationPayment` and pass it directly into `_finalizeSubaccount` as a parameter, eliminating the shared mutable state entirely.

//   v.insurance <= 0? YES → socializeSubaccount(subaccountA) triggered
