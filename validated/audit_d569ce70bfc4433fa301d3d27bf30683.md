### Title
Protocol Treasury Reward Credited to Account but Excluded from `minted_amount` When All Validators Are Kicked Out — (File: `chain/epoch-manager/src/reward_calculator.rs`)

---

### Summary

In `RewardCalculator::calculate_reward`, when the epoch ends with zero eligible validators (all kicked out for insufficient block/chunk production), the function credits `epoch_protocol_treasury` tokens into the reward map but returns `Balance::ZERO` as the `minted_amount`. This `minted_amount` is what gets stored in `EpochInfo::minted_amount` and propagated into the block header's `total_supply`. The result is a permanent accounting mismatch: the treasury account balance increases, but `total_supply` does not, breaking the invariant `Σ(account balances) == total_supply`.

---

### Finding Description

In `calculate_reward()`, the protocol treasury reward is computed and inserted into the result map unconditionally, before the `num_validators == 0` guard:

```rust
res.insert(self.protocol_treasury_account.clone(), epoch_protocol_treasury);
if num_validators == 0 {
    return (res, Balance::ZERO);   // ← minted_amount is ZERO, not epoch_protocol_treasury
}
``` [1](#0-0) 

The second element of the return tuple is the `minted_amount` — the amount of newly issued tokens that must be added to `total_supply`. When `num_validators == 0`, this is returned as `Balance::ZERO` even though `epoch_protocol_treasury` (a non-zero value whenever `max_inflation_rate > 0`) was already placed in the reward map.

This `minted_amount` flows directly into `EpochInfo`:

```rust
let (validator_reward, minted_amount) = { ... self.reward_calculator.calculate_reward(...) };
// minted_amount is passed to proposals_to_epoch_info → EpochInfo::minted_amount
``` [2](#0-1) 

At the next epoch boundary, `chain.rs` reads `EpochInfo::minted_amount()` and uses it to update `total_supply` in the block header:

```rust
let minted_amount = if self.epoch_manager.is_next_block_epoch_start(&prev_hash)? {
    Some(self.epoch_manager.get_epoch_info(header.next_epoch_id())?.minted_amount())
} else { None };
``` [3](#0-2) 

```rust
let new_total_supply = prev
    .total_supply()
    .checked_add(minted_amount.unwrap_or(Balance::ZERO))
    .unwrap()
    .checked_sub(balance_burnt)
    .unwrap();
``` [4](#0-3) 

Meanwhile, `update_validator_accounts` in the runtime **does** apply the treasury reward to the treasury account's on-chain balance, because the reward map (`validator_rewards`) still contains the non-zero `epoch_protocol_treasury` entry:

```rust
if let Some(account_id) = &validator_accounts_update.protocol_treasury_account_id {
    if !validator_accounts_update.stake_info.contains_key(account_id) {
        ...
        account.set_amount(account.amount().checked_add(treasury_reward)...);
        set_account(state_update, account_id.clone(), &account);
    }
}
``` [5](#0-4) 

The divergence is exact: `total_supply` is understated by exactly `epoch_protocol_treasury` yoctoNEAR for every epoch in which all validators are kicked out.

---

### Impact Explanation

The invariant `Σ(account.amount + account.locked) == block_header.total_supply` is permanently broken after such an epoch. Because all nodes run the same code, they all compute the same wrong `minted_amount = 0` and reach consensus on the wrong `total_supply`. The error is undetectable by `verify_total_supply_checked` (which only checks internal consistency, not absolute correctness). [6](#0-5) 

Downstream effects:
- Future epoch reward calculations use `total_supply` as the inflation base, so rewards are permanently slightly understated.
- Any off-chain or on-chain system that relies on `total_supply` as a canonical token count receives a wrong value.
- The mismatch compounds across multiple such epochs.

**Severity: High** — permanent, consensus-agreed corruption of the canonical token supply accounting.

---

### Likelihood Explanation

The trigger condition is `num_validators == 0` in `validator_block_chunk_stats` after the kickout filter at `finalize_epoch`:

```rust
for (account_id, reason) in &validator_kickout {
    if matches!(reason,
        ValidatorKickoutReason::NotEnoughBlocks { .. }
        | ValidatorKickoutReason::NotEnoughChunks { .. }
        | ValidatorKickoutReason::NotEnoughChunkEndorsements { .. }
    ) {
        validator_block_chunk_stats.remove(account_id);
    }
}
``` [7](#0-6) 

This requires every validator in the epoch to fall below the kickout threshold (80% block production, 80% chunk production). This is an extreme network disruption scenario. On mainnet with ~100 validators, this is very unlikely under normal conditions but is reachable (e.g., coordinated outage, severe network partition lasting a full epoch). **Likelihood: Low.**

---

### Recommendation

Change the early-return branch to return `epoch_protocol_treasury` as the `minted_amount` instead of `Balance::ZERO`:

```rust
res.insert(self.protocol_treasury_account.clone(), epoch_protocol_treasury);
if num_validators == 0 {
    return (res, epoch_protocol_treasury);  // ← include treasury issuance
}
``` [1](#0-0) 

This ensures that whenever tokens are credited to the treasury account, the same amount is reflected in `minted_amount` and therefore in `total_supply`.

---

### Proof of Concept

1. Configure a testnet epoch with `max_inflation_rate = 1/40` (2.5% annual), `protocol_reward_rate = 1/10`, and a `total_supply` of 1 billion NEAR.
2. Arrange for all validators to produce fewer than 80% of their expected blocks/chunks in epoch T (e.g., by taking all validators offline for most of the epoch while still allowing the epoch to end).
3. At epoch T boundary, `finalize_epoch` removes all validators from `validator_block_chunk_stats` via the kickout loop.
4. `calculate_reward` computes `epoch_protocol_treasury > 0` (e.g., ~684,931 NEAR for a ~12-hour epoch), inserts it into `res`, then returns `(res, Balance::ZERO)`.
5. `EpochInfo::minted_amount` for epoch T+2 is stored as `0`.
6. The first block of epoch T+1 sets `total_supply = prev_total_supply + 0 - balance_burnt`.
7. The treasury account's on-chain balance increases by `epoch_protocol_treasury`.
8. Query `total_supply` from the block header and compare to `Σ(account balances)` — the latter exceeds the former by exactly `epoch_protocol_treasury`. [8](#0-7) [9](#0-8)

    pub fn verify_total_supply_checked(
        &self,
        prev_total_supply: Balance,
        minted_amount: Option<Balance>,
    ) -> Option<bool> {
        let mut balance_burnt = Balance::ZERO;

        for chunk in self.chunks().iter_new() {
            balance_burnt = balance_burnt.checked_add(chunk.prev_balance_burnt())?;
        }

        let Some(new_total_supply) = prev_total_supply
            .checked_add(minted_amount.unwrap_or(Balance::ZERO))?
            .checked_sub(balance_burnt)
        else {
            // This corresponds to balance_burnt > prev_total_supply + minted_amount
            // which indicates invalid balance burnt, not arithmetic overflow
            return Some(false);
        };
        Some(self.header().total_supply() == new_total_supply)
    }
