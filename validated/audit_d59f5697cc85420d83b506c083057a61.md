### Title
GC Prunes Difficulty-Period Start Block, Causing Permanent Panic at Every Subsequent Difficulty Boundary — (`contract/src/bitcoin.rs`, `contract/src/lib.rs`)

---

### Summary

`InitArgs.gc_threshold` accepts any `u64` with no lower-bound validation. When `gc_threshold < difficulty_adjustment_interval` (2016 for Bitcoin mainnet), `run_mainchain_gc` — called automatically inside `submit_blocks` — will eventually delete the block at the start of the current difficulty period from `mainchain_height_to_header`. The next call to `submit_block_header` at a difficulty boundary then calls `get_header_by_height(first_block_height)`, which panics with `ERR_KEY_NOT_EXIST` because that entry no longer exists. The contract is permanently bricked for all future difficulty-boundary submissions.

---

### Finding Description

**Missing validation on `gc_threshold` in `init`**

`InitArgs` carries `gc_threshold: u64` with no constraint: [1](#0-0) 

`init` stores it verbatim: [2](#0-1) 

**`run_mainchain_gc` is difficulty-period-blind**

`run_mainchain_gc` removes the oldest `min(total_excess, batch_size)` blocks unconditionally, with no awareness of difficulty-period alignment: [3](#0-2) 

It is called automatically after every `submit_blocks` batch with `batch_size = num_of_headers`: [4](#0-3) 

**`get_next_work_required` requires the period-start block to exist**

At every difficulty boundary (`(prev_height + 1) % 2016 == 0`), the function computes:

```
first_block_height = prev_height - (2016 - 1)   // = prev_height - 2015
```

and immediately calls: [5](#0-4) 

`get_header_by_height` panics with `ERR_KEY_NOT_EXIST` if the entry is absent: [6](#0-5) 

---

### Impact Explanation

Once GC removes the block at `first_block_height`, every future `submit_blocks` call that crosses a difficulty boundary panics unconditionally. Because `mainchain_height_to_header` is a `LookupMap` (persistent NEAR storage), the deletion is permanent. No recovery path exists short of a contract migration. All downstream proof verification (`verify_transaction_inclusion`, `verify_transaction_inclusion_v2`) also becomes permanently unavailable because the chain can no longer advance past a difficulty boundary.

---

### Likelihood Explanation

The `init` docstring recommends `gc_threshold = 52704` (≈1 year of blocks), but there is no enforcement. An operator who sets `gc_threshold` to a small value (e.g., 100, 500, or any value below 2016) for storage-cost reasons will trigger this silently. The bug activates automatically via the `submit_blocks` → `run_mainchain_gc` path — no separate attacker action is required after initialization.

---

### Recommendation

Add a lower-bound check in `init` (or in `run_mainchain_gc`) that ensures `gc_threshold` is at least `difficulty_adjustment_interval`. For Bitcoin this is 2016; the check should be network-aware:

```rust
require!(
    args.gc_threshold >= config.difficulty_adjustment_interval,
    format!(
        "gc_threshold must be >= difficulty_adjustment_interval ({})",
        config.difficulty_adjustment_interval
    )
);
```

Additionally, `run_mainchain_gc` should never remove a block whose height is the start of the current difficulty period (i.e., `height % difficulty_adjustment_interval == 0` for Bitcoin).

---

### Proof of Concept

```
1. Deploy with:
     gc_threshold = 100
     genesis_block_height = 0          // height 0 is a difficulty-period start (0 % 2016 == 0)
     skip_pow_verification = true
     network = Mainnet

2. submit_blocks([block_1 .. block_2015])   // 2015 headers in one batch
   After submission: chain stores heights 0..2015 (2016 entries).
   run_mainchain_gc(2015) fires:
     amount_of_headers_we_store = 2016
     total_amount_to_remove     = 2016 - 100 = 1916
     selected_amount_to_remove  = min(1916, 2015) = 1916
     → removes heights 0..1915 inclusive
     → height 0 (period-start block) is DELETED from mainchain_height_to_header

3. submit_blocks([block_2016])
   prev_block_header.block_height = 2015
   (2015 + 1) % 2016 == 0  → difficulty boundary triggered
   first_block_height = 2015 - 2015 = 0
   get_header_by_height(0)  → ERR_KEY_NOT_EXIST  ← PANIC, contract bricked
```
