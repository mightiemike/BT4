Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of End User, Enabling Full Allowlist Bypass via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` inside the pool is the router contract, not the end user. Allowlisting the router — the only way to permit any router-mediated swap — opens the gate to every user, defeating the per-user access control the extension is designed to enforce.

## Finding Description
`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, passing the direct caller of `pool.swap` as `sender`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- direct caller of pool.swap
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this `sender` verbatim to every registered extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap`. When `MetricOmmSimpleRouter.exactInputSingle` is used, it calls `pool.swap` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ...);
```

This makes the router `msg.sender` inside the pool, so the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`. There are only two outcomes:

| Router allowlisted? | Result |
|---|---|
| **No** | Every allowlisted user is silently broken — they cannot use the router even though they are individually permitted. |
| **Yes** | Every non-allowlisted user can bypass the gate by routing through the router. |

No configuration simultaneously (a) allows allowlisted users to use the router and (b) blocks non-allowlisted users from using the router. The `extensionData` field is passed through verbatim but the extension does not read it, and there is no originator field in the swap path.

## Impact Explanation
A pool deployer who configures `SwapAllowlistExtension` to restrict swaps to a curated set of addresses (KYC'd traders, institutional LPs, whitelist-only launch) loses that guarantee the moment any user routes through `MetricOmmSimpleRouter`. If the router is allowlisted — the only way to support router-mediated swaps — the allowlist is fully bypassed. Any unprivileged user can execute swaps in a pool that was intended to be restricted, accessing liquidity and oracle-anchored prices that the pool admin explicitly did not grant them. This satisfies the "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" impact criterion.

## Likelihood Explanation
High. `MetricOmmSimpleRouter` is the primary user-facing swap interface. Pool admins who want to support normal UX must allowlist the router, which is the exact configuration that enables the bypass. The flaw is structural: the extension API receives the direct caller of `pool.swap`, not the originating user, so no amount of careful allowlist management can close the gap while the router is in use. Any unprivileged user can trigger this by calling `exactInputSingle` on the router targeting a restricted pool.

## Recommendation
The end-user identity must be threaded through the swap path so the extension can gate on it. Two viable approaches:

1. **`extensionData` forwarding**: The router encodes `msg.sender` (the end user) into `extensionData` before calling `pool.swap`. The extension decodes and checks that address. The pool already forwards `extensionData` verbatim to every extension hook. This must be paired with a trust model: the extension only accepts the originator claim from a trusted router address, otherwise any caller can spoof it.
2. **Dedicated `originator` field**: Add an `originator` parameter to `pool.swap` that the router populates with `msg.sender`. The pool passes it to `_beforeSwap` alongside `sender`. Extensions that need user-level gating check `originator`; extensions that need caller-level gating check `sender`.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
2. Pool admin allowlists the router:
       swapExtension.setAllowedToSwap(pool, address(router), true)
   (required so that any router-mediated swap can proceed)
3. Pool admin does NOT allowlist attacker:
       allowedSwapper[pool][attacker] == false
4. Attacker calls:
       router.exactInputSingle(ExactInputSingleParams{pool: pool, ...})
5. Router calls pool.swap(...) — msg.sender inside pool = router.
6. Pool calls _beforeSwap(sender=router, ...).
7. Extension evaluates:
       allowedSwapper[pool][router] == true  →  passes
8. Attacker's swap executes in the restricted pool.
   The per-user allowlist is fully bypassed.
```
