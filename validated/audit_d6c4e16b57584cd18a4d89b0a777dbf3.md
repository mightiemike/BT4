### Title
Linked Signer Can Unilaterally Rotate Itself to an Attacker-Controlled Address Without Subaccount Owner Consent — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction in `EndpointTx.sol` is validated with `allowLinkedSigner = true`, meaning the **current linked signer** can sign a new `LinkSigner` transaction to replace itself with any arbitrary address — without the subaccount owner's knowledge or consent. This is a direct analog to the external report's vulnerability class: a delegated entity performing a sensitive, account-mutating operation silently, bypassing the principal's explicit authorization.

---

### Finding Description

In `processTransactionImpl`, the `LinkSigner` branch decodes a `SignedLinkSigner` and calls `validateSignedTx` with the fifth argument hardcoded to `true`:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` resolves to `validateSignature` → `verifier.validateSignature`, which accepts a signature from **either** the subaccount owner **or** the current `linkedSigners[sender]`:

```solidity
// Verifier.sol lines 291–303
function validateSignature(...) public pure {
    address recovered = ECDSA.recover(digest, signature);
    require(
        (recovered != address(0)) &&
            ((recovered == address(uint160(bytes20(sender)))) ||
                (recovered == linkedSigner)),   // ← linked signer accepted
        ERR_INVALID_SIGNATURE
    );
}
``` [2](#0-1) 

Because `LinkSigner` itself is the operation that **writes** `linkedSigners[sender]`, accepting the current linked signer as a valid authorizer for this write creates a self-escalation path: the linked signer can overwrite its own entry with any address it chooses.

The `LinkSigner` struct carries only `sender`, `signer`, and `nonce` — there is no field that restricts what `signer` value may be written, and no on-chain check that the new signer is the subaccount owner or any trusted party:

```solidity
// IEndpoint.sol lines 176–180
struct LinkSigner {
    bytes32 sender;
    bytes32 signer;
    uint64 nonce;
}
``` [3](#0-2) 

---

### Impact Explanation

Once the linked signer rotates itself to an attacker-controlled address, the attacker inherits all permissions that `allowLinkedSigner = true` grants across the protocol:

1. **`TransferQuote` (lines 593–614)** — validated with `allowLinkedSigner = true`. The attacker-controlled linked signer can sign `TransferQuote` transactions to move the victim's entire quote balance to any attacker-controlled subaccount. [4](#0-3) 

2. **`WithdrawCollateral` V1 (lines 413–436)** — validated with `allowLinkedSigner = true`. Funds are sent to `address(0)` which resolves to the subaccount owner's address, so direct theft via this path is limited; however, the attacker can still trigger forced withdrawals that disrupt positions. [5](#0-4) 

3. **`LiquidateSubaccount` (lines 391–412)** — validated with `allowLinkedSigner = true`. The attacker can submit liquidation transactions against the victim's subaccount once it is made unhealthy (e.g., by draining quote via `TransferQuote` first). [6](#0-5) 

4. **`MintNlp` / `BurnNlp` (lines 534–573)** — both validated with `allowLinkedSigner = true`. The attacker can manipulate the victim's NLP position. [7](#0-6) 

The `TransferQuote` path is the most direct route to full collateral theft: the attacker signs a `TransferQuote` from the victim's subaccount to an attacker-controlled subaccount, draining the quote balance entirely.

---

### Likelihood Explanation

The precondition is that the victim has previously called `LinkSigner` to grant a linked signer — a common, documented use case for automated trading bots and API integrations. Any party that holds the linked signer's private key (including a compromised third-party service, a malicious trading bot operator, or an attacker who exfiltrates the key) can execute this rotation silently. No on-chain signal is emitted that would alert the subaccount owner before the rotation completes. The sequencer processes the transaction as a normal fast-path operation.

---

### Recommendation

Change the `allowLinkedSigner` flag for the `LinkSigner` transaction type from `true` to `false`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
``` [8](#0-7) 

This mirrors the design intent already applied to `WithdrawCollateralV2` when `sendTo != address(0)` — the more sensitive the operation, the stricter the signer requirement. Changing the linked signer is a privileged account-configuration action that must require the subaccount owner's direct signature, not a delegated one.

### Proof of Concept

**Setup:**
- Victim subaccount: `0xAlice000000000000000000000000000000000000` + `"default\x00\x00\x00\x00\x00"`
- Victim has previously linked `0xBot` as their trading bot linked signer
- Attacker controls `0xAttacker`

**Step 1 — Linked signer rotates itself (signed by `0xBot`):**
```
LinkSigner {
    sender: <Alice's subaccount bytes32>,
    signer: bytes32(uint256(uint160(0xAttacker))),
    nonce:  <current nonce>
}
```
Signature produced by `0xBot`. `validateSignedTx(..., true)` accepts it because `recovered == linkedSigners[sender]`.

After sequencer processes this: `linkedSigners[Alice's subaccount] = 0xAttacker`.

**Step 2 — Attacker drains quote (signed by `0xAttacker`):**
```
TransferQuote {
    sender:    <Alice's subaccount bytes32>,
    recipient: <Attacker's subaccount bytes32>,
    amount:    <Alice's full quote balance>,
    nonce:     <current nonce>
}
```
Signature produced by `0xAttacker`. `validateSignedTx(..., true)` accepts it because `recovered == linkedSigners[sender]`.

`clearinghouse.transferQuote` moves Alice's entire quote balance to the attacker's subaccount. Alice's subaccount health collapses; the attacker withdraws from their own subaccount normally. [9](#0-8)
