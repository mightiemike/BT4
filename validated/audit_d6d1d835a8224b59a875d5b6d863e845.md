Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Enabling Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which is `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool level is the router contract, not the actual end user. If a pool admin allowlists the router address (a natural action to enable router-mediated swaps), every unprivileged user can bypass the curated allowlist by routing through the public router, defeating the intended access control and exposing LP funds to unrestricted adverse selection.

## Finding Description

**Root cause — wrong actor checked:**

`SwapAllowlistExtension.beforeSwap` (line 37) checks:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool (correct for namespacing) and `sender` is the first argument forwarded by `ExtensionCalling._beforeSwap`, which is `msg.sender` of the pool's `swap` call.

**Pool passes its own `msg.sender` as `sender`:**

`MetricOmmPool.swap` (line 231) calls:
```solidity
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap
    recipient,
    ...
);
```

**Router is the caller of `pool.swap`:**

`MetricOmmSimpleRouter.exactInputSingle` (line 72) calls:
```solidity
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    ...
);
```
The router is `msg.sender` at the pool boundary, so the extension receives `sender = address(router)`, not the actual end user.

**Exploit path:**
1. Pool admin deploys a curated pool with `SwapAllowlistExtension` to restrict trading to specific addresses (e.g., KYC'd counterparties or designated market makers).
2. Pool admin calls `swapExtension.setAllowedToSwap(pool, address(router), true)` — a natural action to allow allowlisted users to trade via the supported periphery router.
3. Non-allowlisted user Bob calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(...)` with `msg.sender = router`.
5. Extension evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. Bob, who was never individually allowlisted, executes a swap on the curated pool.

The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput` paths, all of which call `pool.swap` with the router as `msg.sender`.

**Existing guards are insufficient:**

`BaseMetricExtension.onlyPool` only verifies that the caller is a registered pool — it does not verify the identity of the economic actor. The `_requireExpectedCallbackCaller` check in the router only validates that the callback comes from the expected pool; it does not restrict which end users may initiate swaps.

## Impact Explanation
A pool admin who configures a swap allowlist to restrict trading to specific counterparties and also allowlists the router (to enable periphery access) inadvertently opens the pool to all users. Non-allowlisted users can trade against LP positions, causing adverse selection losses to LPs. The allowlist — the sole mechanism protecting curated pools — is rendered ineffective. This constitutes broken core pool functionality and potential direct loss of LP principal above Sherlock thresholds.

## Likelihood Explanation
The condition is reachable by any unprivileged user. The only prerequisite is that the pool admin has allowlisted the router, which is a natural and expected administrative action for any pool that intends to support the official periphery. The router is a public, permissionless contract with no user-level access control. The bypass is repeatable on every swap and requires no special privileges or timing.

## Recommendation
The extension must check the economically relevant actor, not the intermediary. Two options:

1. **Pass the original `msg.sender` through the router:** Have the router encode the original caller in `extensionData` and have the extension decode and verify it. This requires a trusted encoding convention.
2. **Check `recipient` or use a dedicated forwarded-sender field:** Redesign the `beforeSwap` hook signature to include a separate `originalCaller` field that the pool populates from a trusted transient context set by the router before calling `pool.swap`.
3. **Document that the router must never be allowlisted and enforce this at the factory level** by rejecting `setAllowedToSwap(pool, router, true)` — a weaker mitigation that relies on off-chain policy.

The cleanest fix is option 1 or 2: the extension must receive and verify the true end-user identity, not the intermediary's address.

## Proof of Concept

```solidity
// Foundry test sketch
function test_allowlistBypassViaRouter() public {
    // Setup: curated pool with SwapAllowlistExtension
    // Admin allowlists the router (natural action for periphery support)
    swapExtension.setAllowedToSwap(address(pool), address(router), true);

    // Bob is NOT individually allowlisted
    address bob = makeAddr("bob");
    // Fund bob and approve router
    token0.mint(bob, 1e18);
    vm.prank(bob);
    token0.approve(address(router), type(uint256).max);

    // Bob bypasses allowlist via router
    vm.prank(bob);
    router.exactInputSingle(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(token0),
            recipient: bob,
            deadline: block.timestamp + 1,
            amountIn: 1000,
            amountOutMinimum: 0,
            zeroForOne: true,
            priceLimitX64: 0,
            extensionData: ""
        })
    );
    // Bob successfully swapped despite not being individually allowlisted
}
```
