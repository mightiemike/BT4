Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of the `addLiquidity` call) and gates only on the caller-supplied `owner` address. Because `MetricOmmPool.addLiquidity` imposes no `msg.sender == owner` constraint, any unprivileged address can pass an allowlisted address as `owner`, satisfy the extension check, and deposit liquidity into a permissioned pool — permanently modifying bin state and breaking the pool's core access-control invariant.

## Finding Description
`MetricOmmPool.addLiquidity` passes both `msg.sender` (as `sender`) and the free-form caller-supplied `owner` to the extension hook:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` discards the first argument and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [2](#0-1) 

Unlike `removeLiquidity`, which enforces `msg.sender == owner` before proceeding: [3](#0-2) 

`addLiquidity` has no such constraint: [4](#0-3) 

**Exploit path:**
1. Attacker observes an allowlisted address `allowlistedLP` via on-chain `AllowedToDepositSet` events.
2. Attacker deploys a contract implementing `metricOmmModifyLiquidityCallback` and calls `pool.addLiquidity(allowlistedLP, salt, deltas, callbackData, extensionData)`.
3. Pool calls `_beforeAddLiquidity(attacker, allowlistedLP, ...)` → extension receives `sender=attacker` (discarded), `owner=allowlistedLP`.
4. Extension evaluates `allowedDepositor[pool][allowlistedLP]` → `true` → no revert.
5. Attacker pays tokens in the callback; position is minted under `allowlistedLP`.
6. Pool bin state (`curBinIdx`, `curPosInBin`, `binTotals`) is permanently modified by an unauthorized actor.

The `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, ...)` path also exposes this: it only validates `owner != address(0)` and calls `pool.addLiquidity(owner, ...)` with `msg.sender = LiquidityAdder`, so the extension again sees only `owner`. [5](#0-4) 

## Impact Explanation
The deposit allowlist is the primary access-control mechanism for permissioned pools. Bypassing it allows an unauthorized actor to modify pool bin state (`curBinIdx`, `curPosInBin`, `binTotals.scaledToken0/1`), which directly affects the marginal price seen by subsequent swaps and the composition read by oracle guard extensions (`OracleValueStopLossExtension`, `PriceVelocityGuardExtension`). Additionally, the attacker can grief allowlisted LPs by forcing positions under their address with arbitrary salts, interfering with position accounting. This constitutes broken core pool functionality and a broken access-control invariant with direct impact on swap price execution and LP position integrity — matching the allowed impact gate for broken core pool functionality and bad-price execution risk.

## Likelihood Explanation
No special privilege is required; any EOA or contract can call `pool.addLiquidity` directly. The only precondition is knowing one allowlisted address (observable on-chain via `AllowedToDepositSet` events) and funding the attacker contract with tokens. The attacker forfeits the deposited tokens (cannot call `removeLiquidity` since `msg.sender != owner`), making the attack economically rational only when bin-state manipulation yields a larger off-pool profit (e.g., arbitrage or front-running). Likelihood is medium.

## Recommendation
Gate on `sender` (the actual caller) rather than `owner`, consistent with the contract's stated purpose of gating `addLiquidity` by depositor:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Alternatively, if the intent is to restrict which addresses may *own* positions, `MetricOmmPool.addLiquidity` must enforce `msg.sender == owner` (mirroring the `removeLiquidity` guard), so that `sender` and `owner` are always the same address.

## Proof of Concept

```solidity
contract BypassDeposit {
    IMetricOmmPool pool;
    address allowlistedLP; // observed from AllowedToDepositSet event

    function attack(LiquidityDelta calldata deltas) external {
        // owner = allowlistedLP → extension check passes
        // sender = address(this) → extension check is never evaluated
        pool.addLiquidity(allowlistedLP, 0, deltas, abi.encode(uint8(1)), "");
    }

    function metricOmmModifyLiquidityCallback(
        uint256 amount0, uint256 amount1, bytes calldata
    ) external {
        IERC20(token0).transfer(msg.sender, amount0);
        IERC20(token1).transfer(msg.sender, amount1);
    }
}
```

Steps:
1. Deploy `BypassDeposit`, fund it with token0/token1.
2. Call `attack(deltas)` targeting any bin.
3. `DepositAllowlistExtension.beforeAddLiquidity` receives `sender = BypassDeposit` (discarded), `owner = allowlistedLP`; checks `allowedDepositor[pool][allowlistedLP]` → `true` → no revert.
4. Liquidity is minted under `allowlistedLP`; pool bin state is modified by the unauthorized actor. [2](#0-1) [4](#0-3)
