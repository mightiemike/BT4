### Title
`gc_state()` deletes child-shard `State` but never calls `delete_shard_uid_mapping()`, leaving stale `StateShardUIdMapping` entries that permanently block parent-shard GC after resharding — (`chain/chain/src/garbage_collection.rs`)

---

### Summary

`gc_state()` carries a comment stating it will "Delete State of `shards_to_cleanup` and **associated ShardUId mapping**", but its implementation only calls `delete_shard_uid_prefixed_state()` and never calls `delete_shard_uid_mapping()`. The parallel cleanup path in `TrieStateResharder::process_batch_and_update_status()` correctly calls `delete_shard_uid_mapping()` when it finishes each child shard. The divergence leaves stale `DBCol::StateShardUIdMapping` entries that cause `gc_parent_shard_after_resharding()` to permanently skip deletion of the parent shard's state, producing unbounded storage growth on any node that stops tracking child shards before resharding completes.

---

### Finding Description

**Write path — mappings are set during resharding:**

`ReshardingManager::process_memtrie_resharding_storage_update()` writes `StateShardUIdMapping` entries for every tracked child shard:

```rust
for &child_shard_uid in tracked_children {
    store_update.set_shard_uid_mapping(child_shard_uid, parent_shard_uid_prefix);
}
``` [1](#0-0) 

**Correct cleanup path — `TrieStateResharder` deletes the mapping when done:**

When `process_batch_and_update_status()` finishes iterating a child shard it calls `delete_shard_uid_mapping()` atomically with the final batch commit:

```rust
store_update.trie_store_update().delete_shard_uid_mapping(child.shard_uid);
*child_ref = None;
``` [2](#0-1) 

**Broken cleanup path — `gc_state()` promises but omits the mapping deletion:**

The comment at line 1162 explicitly states the intent, but the loop body only calls `delete_shard_uid_prefixed_state()` and never `delete_shard_uid_mapping()`:

```rust
// Delete State of `shards_to_cleanup` and associated ShardUId mapping.
let mut trie_store_update = store.trie_store().store_update();
for shard_uid_prefix in shards_to_cleanup {
    trie_store_update.delete_shard_uid_prefixed_state(shard_uid_prefix);
    // ← delete_shard_uid_mapping() is never called here
}
``` [3](#0-2) 

`delete_shard_uid_mapping` has exactly **two** call sites in the entire codebase: its own definition and the single call inside `TrieStateResharder`. `gc_state()` is not one of them. [4](#0-3) 

**Downstream consequence — stale mapping blocks `gc_parent_shard_after_resharding()`:**

`gc_parent_shard_after_resharding()` guards parent-state deletion behind a check for active child mappings:

```rust
let has_active_mapping = children_shards.into_iter().any(|child_shard_uid| {
    let mapped_shard_uid = maybe_get_shard_uid_mapping(&store, child_shard_uid);
    mapped_shard_uid.as_ref() == Some(&parent_shard_uid)
});
if !has_active_mapping {
    trie_store_update.delete_shard_uid_prefixed_state(parent_shard_uid);
} else {
    // skipping parent shard cleanup - active mappings exist
}
``` [5](#0-4) 

A stale `StateShardUIdMapping` entry left by `gc_state()` makes `has_active_mapping` permanently `true`, so the parent shard's state is never deleted.

---

### Impact Explanation

**Storage invariant broken:** After a shard split, the parent shard's trie data should be deleted once all children have their own copies and no child mapping remains. With stale mappings, `gc_parent_shard_after_resharding()` always finds `has_active_mapping == true` and skips the deletion. The parent shard's entire `State` prefix accumulates indefinitely — proportional to the size of the split shard's state, which can be gigabytes on mainnet.

**Refcount corruption:** `decrement_refcount_by()` and `increment_refcount_by()` both consult `maybe_get_shard_uid_mapping()` and double-write to both the child prefix and the mapped parent prefix. With a stale mapping pointing to a parent whose state has been partially or fully deleted, these double-writes produce incorrect refcounts on the parent prefix, corrupting the trie GC invariant. [6](#0-5) 

---

### Likelihood Explanation

The trigger is any node that:
1. Tracks a parent shard at the resharding block (causing `set_shard_uid_mapping` to be written), and
2. Stops tracking the child shards before `TrieStateResharder` completes (e.g., config change, state sync, or restart with a different `tracked_shards_config`).

This is a normal operational scenario for validator nodes that rotate shard assignments across epochs. Once triggered, the stale mapping persists for the lifetime of the node because no other code path calls `delete_shard_uid_mapping()` for these entries.

---

### Recommendation

In `gc_state()`, after calling `delete_shard_uid_prefixed_state(shard_uid_prefix)` for each shard being cleaned up, also delete any `StateShardUIdMapping` entries whose **value** (the mapped-to parent) equals `shard_uid_prefix`. Concretely, iterate `DBCol::StateShardUIdMapping` and call `delete_shard_uid_mapping(child_shard_uid)` for every child whose mapped parent is in `shards_to_cleanup`. This mirrors what `TrieStateResharder` already does per-child and satisfies the comment's stated intent.

---

### Proof of Concept

1. Resharding splits `parent_shard` → `left_child`, `right_child` at epoch T.
2. `process_memtrie_resharding_storage_update()` writes `StateShardUIdMapping[left_child] = parent_shard` and `StateShardUIdMapping[right_child] = parent_shard`.
3. Before `TrieStateResharder` finishes, the node's `tracked_shards_config` is updated to stop tracking `left_child` and `right_child` (e.g., epoch rotation).
4. GC reaches epoch T+1. `gc_state()` finds `left_child` and `right_child` in `shards_to_cleanup` (node no longer tracks them). It calls `delete_shard_uid_prefixed_state(left_child)` and `delete_shard_uid_prefixed_state(right_child)` — both are no-ops because actual state bytes live under `parent_shard`'s prefix. It does **not** call `delete_shard_uid_mapping()`.
5. GC reaches epoch T. `gc_parent_shard_after_resharding()` calls `maybe_get_shard_uid_mapping(left_child)` → returns `parent_shard`. `has_active_mapping = true`. Parent state deletion is skipped.
6. This repeats on every subsequent GC cycle. `parent_shard`'s entire trie state is never reclaimed.
