### Title
Protobuf `ValidResourceBounds` Misclassification Produces Wrong Transaction Hash for V3 Transactions with Zero L2/L1_DATA Gas - (File: crates/apollo_protobuf/src/converters/transaction.rs)

### Summary
The protobuf deserializer for `ValidResourceBounds` classifies a V3 transaction as `ValidResourceBounds::L1Gas` whenever both `l2_gas` and `l1_data_gas` are zero, even though the transaction was originally submitted and hashed as `ValidResourceBounds::AllResources`. Because `get_tip_resource_bounds_hash` produces a structurally different hash for `L1Gas` (two resource entries) versus `AllResources` with zero l2/l1_data (three resource entries), the same transaction data yields two distinct hash values depending on which deserialization path is used. This breaks the canonicalization invariant that a transaction's hash must be stable across all representations.

### Finding Description

**Root cause â€” protobuf path (`crates/apollo_protobuf/src/converters/transaction.rs`):**

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

When `l2_gas = 0` and `l1_data_gas = 0`, the protobuf converter emits `L1Gas`, discarding the fact that the transaction was a V3 `AllResources` transaction.

**Submission path always uses `AllResources`:**

Every new transaction enters through `RpcTransaction` â†’ `InternalRpcTransactionWithoutTxHash`, which stores `resource_bounds: AllResourceBounds`. The hash is computed by wrapping it as `ValidResourceBounds::AllResources(self.resource_bounds)`: [2](#0-1) 

The hash function `get_tip_resource_bounds_hash` then includes **three** resource entries (L1_GAS, L2_GAS, L1_DATA_GAS) for `AllResources`: [3](#0-2) 

For `L1Gas`, only **two** entries are hashed (L1_GAS, L2_GAS â€” no L1_DATA_GAS): [4](#0-3) 

Even when all three values are numerically identical (l2=0, l1_data=0), the Poseidon hash over two elements differs from the hash over three elements. The stored hash `H_all` (3 entries) and the re-derived hash `H_l1` (2 entries) are distinct field elements.

**The same misclassification exists in the RPC layer:** [5](#0-4) 

**JSON deserialization is inconsistent with protobuf:** JSON classifies as `AllResources` whenever the `L1DataGas` key is present (even with zero value), while protobuf classifies by value alone: [6](#0-5) 

### Impact Explanation

A V3 transaction with `l1_gas > 0`, `l2_gas = 0`, `l1_data_gas = 0` passes gateway admission (fee > 0 from l1_gas alone). Its canonical hash `H_all` is stored and propagated. When a peer node deserializes this transaction from a protobuf block sync message, it computes `H_l1 â‰  H_all`. Any component that re-derives or verifies the transaction hash from the protobuf-deserialized `Transaction` object â€” including block hash computation, RPC `starknet_getTransactionByHash`, fee estimation, or re-execution â€” will bind the wrong hash to the transaction. This matches: **High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

Additionally, the misclassification changes `get_gas_vector_computation_mode()` from `All` to `NoL2Gas`, altering fee pre-validation logic during re-execution.

### Likelihood Explanation

Any unprivileged user can submit a V3 invoke transaction with `l2_gas = {max_amount: 0, max_price_per_unit: 0}` and `l1_data_gas = {max_amount: 0, max_price_per_unit: 0}` while setting `l1_gas` to a non-zero value. The gateway's `validate_resource_bounds` only checks that `max_possible_fee > 0`, which is satisfied by non-zero l1_gas. The `min_gas_price` check applies to `l2_gas.max_price_per_unit`, which is zero â€” this check only rejects if `min_gas_price > 0`, a configuration-dependent gate that does not universally prevent the trigger.

### Recommendation

Replace the value-based classification in the protobuf converter with an explicit version-based gate. For transactions with `version = 3`, always produce `ValidResourceBounds::AllResources`, regardless of whether l2_gas and l1_data_gas are zero:

```rust
// In TryFrom<protobuf::ResourceBounds> for ValidResourceBounds:
// Remove the zero-value heuristic; always use AllResources for V3 transactions.
Ok(ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }))
```

Apply the same fix to `From<ResourceBoundsMapping> for ValidResourceBounds` in `crates/apollo_rpc/src/v0_8/transaction.rs`. The `L1Gas` variant should only be produced when deserializing pre-0.13.3 transactions that genuinely lack the `L1DataGas` field, not based on a zero-value check.

### Proof of Concept

1. Submit a V3 invoke transaction with:
   - `l1_gas = {max_amount: 1000, max_price_per_unit: 1}`
   - `l2_gas = {max_amount: 0, max_price_per_unit: 0}`
   - `l1_data_gas = {max_amount: 0, max_price_per_unit: 0}`

2. Gateway accepts it; `InternalRpcTransactionWithoutTxHash::calculate_transaction_hash` calls `get_invoke_transaction_v3_hash` with `ValidResourceBounds::AllResources(...)`, producing hash `H_all` (tip + L1_GAS_concat + L2_GAS_concat(0) + L1_DATA_GAS_concat(0)).

3. A peer node receives the block via P2P protobuf sync. `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` fires: `l1_data_gas.is_zero() && l2_gas.is_zero()` â†’ `ValidResourceBounds::L1Gas(l1_gas)`.

4. The peer calls `get_invoke_transaction_v3_hash` on the deserialized transaction, which calls `get_tip_resource_bounds_hash` with `L1Gas` â†’ only two resource entries â†’ hash `H_l1 â‰  H_all`.

5. Any hash verification, block hash derivation, or RPC response using this re-derived hash returns the wrong value.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
