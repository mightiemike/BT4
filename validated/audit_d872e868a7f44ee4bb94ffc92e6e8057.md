### Title
NLP Lock-Period Bypass via `matchOrders` — Locked NLP Sold Before 4-Day Expiry (`core/contracts/SpotEngine.sol`)

---

### Summary

`handleNlpLockedBalance` contains no floor guard when `amountDelta < 0`: it unconditionally subtracts from `unlockedBalanceSum.amount`, allowing it to go negative. Because `OffchainExchange.matchOrders` has no guard against `NLP_PRODUCT_ID`, a user can submit a signed NLP sell order that the sequencer routes through `matchOrders → updateBalance(NLP_PRODUCT_ID, sub, -X) → handleNlpLockedBalance`. This reduces `unlockedBalanceSum` without checking whether the NLP being sold has actually cleared the 4-day lock, bypassing the invariant that NLP can only leave a subaccount via the `burnNlp` path after `getNlpUnlockedBalance` confirms the lock has expired.

---

### Finding Description

**Step 1 — `handleNlpLockedBalance` has no floor on the negative branch.** [1](#0-0) 

When `amountDelta < 0` the function reads `unlockedBalanceSum`, adds the (negative) delta, and writes it back. There is no `require(balanceSum.amount + amountDelta >= 0)` or equivalent. `unlockedBalanceSum.amount` can become arbitrarily negative.

**Step 2 — Both `updateBalance` overloads invoke `handleNlpLockedBalance` for `NLP_PRODUCT_ID`.** [2](#0-1) [3](#0-2) 

**Step 3 — `matchOrders` has no guard against `NLP_PRODUCT_ID`.**

`OffchainExchange.matchOrders` contains zero references to `NLP_PRODUCT_ID`. [4](#0-3) 

The only order-level checks are signature validity, expiration, crossing price, and isolated-subaccount status. None of these block an NLP sell order.

**Step 4 — `_updateBalances` reaches the single-arg `updateBalance` for NLP when `quoteId != QUOTE_PRODUCT_ID`, or the four-arg variant otherwise; both paths call `handleNlpLockedBalance`.** [5](#0-4) 

**Step 5 — `burnNlp` checks `getNlpUnlockedBalance`; `matchOrders` does not.**

`getNlpUnlockedBalance` calls `tryUnlockNlpBalance` and returns `unlockedBalanceSum`. [6](#0-5) 

`tryUnlockNlpBalance` only moves entries from the locked queue to `unlockedBalanceSum` once `unlockedAt <= getOracleTime()`. [7](#0-6) 

The `burnNlp` path is the only intended exit that enforces the 4-day lock. `matchOrders` is a completely separate exit that skips this check entirely.

**Step 6 — `NLP_PRODUCT_ID` is a registered spot product; interest accrual is explicitly skipped for it, confirming it lives in `productIds`.** [8](#0-7) 

`NLP_PRODUCT_ID = 11` and `NLP_LOCK_PERIOD = 4 days` are protocol constants. [9](#0-8) 

---

### Impact Explanation

A user who has just minted NLP (all of it sitting in the locked queue, `unlockedBalanceSum = 0`) can immediately submit a signed sell order for that NLP. The sequencer routes it through `matchOrders`. The user's actual NLP balance is reduced by `_updateBalanceNormalized` and they receive quote tokens in return. `handleNlpLockedBalance` sets `unlockedBalanceSum` to a negative value (e.g. `-X`). When the locked NLP eventually matures, `tryUnlockNlpBalance` adds `+X` back, netting to zero — but the user has already exited their position and received quote tokens `X` seconds before the lock expired. The 4-day lock period is fully bypassed. Additionally, `unlockedBalanceSum` being negative during the interim period corrupts the accounting used by `burnNlp` for any remaining NLP the user holds.

---

### Likelihood Explanation

Any user can sign an NLP sell order and submit it to the sequencer. The sequencer has no on-chain obligation to reject it; the only protection would be an off-chain filter, which is not part of the audited production contracts. The path is concrete, requires no privileged access, and is locally reproducible on an unmodified Hardhat setup by calling `matchOrders` with `productId = 11` and a negative `amountDelta`.

---

### Recommendation

1. In `handleNlpLockedBalance`, when `amountDelta < 0`, add:
   ```solidity
   require(balanceSum.amount + amountDelta >= 0, ERR_INSUFFICIENT_UNLOCKED_NLP);
   ```
   This enforces that only already-unlocked NLP can be debited through any path.

2. Alternatively, add an explicit guard at the top of `matchOrders` (or in `_updateBalances`) that reverts if `callState.productId == NLP_PRODUCT_ID`, restricting NLP transfers exclusively to the `mintNlp`/`burnNlp` clearinghouse paths.

---

### Proof of Concept

```solidity
// 1. User mints 100e18 NLP → locked for 4 days, unlockedBalanceSum = 0
// 2. User signs a sell order: productId=11, amount=-100e18
// 3. Sequencer calls OffchainExchange.matchOrders(...)
//    → _updateBalances → SpotEngine.updateBalance(11, user, -100e18)
//    → handleNlpLockedBalance(user, -100e18)
//    → unlockedBalanceSum.amount = 0 + (-100e18) = -100e18  ← no revert
//    → user's NLP balance = 0, user's quote balance += price*100e18
// 4. Assert: user received quote tokens before the 4-day lock expired
// 5. Assert: unlockedBalanceSum.amount < 0  ← invariant broken
```

        while (
            queue.unlockedUpTo < queue.balanceCount &&
            queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
