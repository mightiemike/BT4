### Title
FoT Token Deposit Overstates User Balance, Corrupting Clearinghouse Solvency — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.depositCollateral()` credits users with the nominal deposit amount (`txn.amount`) without verifying the actual number of tokens received by the clearinghouse. For fee-on-transfer (FoT) tokens, the clearinghouse receives fewer tokens than `txn.amount`, but the user's internal balance is inflated by the full nominal amount. This desynchronizes the protocol's accounting from its real token holdings, enabling users to collectively withdraw more than the clearinghouse holds and leaving the last withdrawer unable to claim their funds. The same pattern is independently present in `handleWithdrawTransfer`.

---

### Finding Description

**Deposit path — root cause:**

When a user deposits collateral, the Endpoint pulls `txn.amount` tokens from the user via `handleDepositTransfer` and then calls `clearinghouse.depositCollateral(txn)`. Inside `depositCollateral`, the user's balance is updated using `txn.amount` directly, with no balance check before or after the transfer: [1](#0-0) 

```solidity
int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
int128 amountRealized = int128(txn.amount) * int128(multiplier);

spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
```

For a FoT token with a 1% fee, a user depositing 100 tokens causes the clearinghouse to receive only 99 tokens, but the user is credited with 100 tokens of balance. The gap between actual holdings and recorded balances grows with every deposit.

**Withdrawal path — secondary root cause:**

`handleWithdrawTransfer` transfers `amount` tokens to `withdrawPool` and immediately calls `submitWithdrawal` with the same nominal `amount`, without checking what `withdrawPool` actually received: [2](#0-1) 

```solidity
function handleWithdrawTransfer(...) internal virtual {
    token.safeTransfer(withdrawPool, uint256(amount));
    BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx);
}
```

This is the direct structural analog to the external report's `sendToSplitter` / `incrementWindow(splitterShare)` pattern: tokens are transferred to a downstream contract, and that contract is told to account for the full nominal amount rather than what it actually received.

**Why `assertUtilization` does not protect against this:** [3](#0-2) 

`assertUtilization` checks `totalDeposits >= totalBorrows` using the same inflated accounting state. Both sides are overstated equally, so the check passes even when the clearinghouse is insolvent relative to actual token holdings.

---

### Impact Explanation

- The clearinghouse's actual token balance becomes less than the sum of all user balances for any FoT token used as collateral.
- Users can withdraw more than the protocol holds, draining funds from other depositors.
- The last user(s) to withdraw will be unable to claim their full balance — a direct, permanent asset loss.
- The `WithdrawPool` is separately affected: it is instructed to distribute more tokens than it received, causing the final claimant's withdrawal to revert or be underpaid.

---

### Likelihood Explanation

- Any unprivileged user can trigger this by depositing a FoT token through the standard `depositCollateralWithReferral` entry point — no special role or privilege is required.
- The impact is proportional to the FoT fee rate and the number of deposits; it accumulates silently until a withdrawal fails.
- The protocol supports arbitrary ERC20 tokens as collateral products (registered via `addOrUpdateProduct`), making FoT token exposure realistic if any such token is listed.

---

### Recommendation

In `handleDepositTransfer` (Endpoint.sol) or at the start of `depositCollateral`, perform a balance check before and after the token pull and use the actual received delta for accounting:

```solidity
uint256 balanceBefore = IERC20Base(token).balanceOf(address(this));
// ... perform transferFrom ...
uint256 balanceAfter = IERC20Base(token).balanceOf(address(this));
uint128 actualReceived = uint128(balanceAfter - balanceBefore);
// use actualReceived instead of txn.amount
```

Apply the same fix to `handleWithdrawTransfer`: measure `withdrawPool`'s balance before and after `safeTransfer` and pass the delta to `submitWithdrawal`.

Alternatively, explicitly document that FoT and rebasing tokens are unsupported and enforce this with a token allowlist at the `addOrUpdateProduct` level.

---

### Proof of Concept

1. A FoT token with a 1% transfer fee is registered as a supported collateral product via `addOrUpdateProduct`.
2. Alice calls `depositCollateralWithReferral(aliceSubaccount, productId, 100e18, "")`.
3. The Endpoint pulls 100e18 tokens from Alice; the clearinghouse receives 99e18 (1 token taken as fee).
4. `depositCollateral` credits Alice with `amountRealized` derived from `txn.amount = 100e18`. [4](#0-3) 
5. Bob similarly deposits 100e18; clearinghouse receives another 99e18. Total held: 198e18. Total credited: 200e18.
6. Alice withdraws 100e18 — succeeds (clearinghouse sends 100e18 to `withdrawPool`, which receives 99e18, but `submitWithdrawal` records 100e18). [5](#0-4) 
7. Bob attempts to withdraw 100e18 — the clearinghouse no longer holds sufficient tokens; Bob's withdrawal reverts or is underpaid, resulting in a permanent loss of funds.
