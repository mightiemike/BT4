### Title
`verify_transaction_inclusion` Returns Stale `true` Across a Pending Chain Reorganization — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` snapshot mainchain membership at the moment of the call. Because `submit_blocks` can atomically reorg the canonical chain in the same NEAR block or a subsequent one, an unprivileged caller can race their proof call ahead of a pending reorg submission and obtain a `true` result for a transaction whose containing block is about to be evicted from the mainchain. Any downstream contract that acts on this result (e.g., a cross-chain bridge releasing NEAR-side funds) is exposed to a false-finality exploit.

---

### Finding Description

`verify_transaction_inclusion` resolves mainchain membership with a single lookup:

```rust
let target_block_height = self
    .mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));
``` [1](#0-0) 

The only guard on the `confirmations` argument is an upper bound against `gc_threshold`; **zero is a valid value**:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [2](#0-1) 

Meanwhile, `submit_blocks` → `submit_block_header_inner` triggers `reorg_chain` whenever a fork's cumulative chainwork exceeds the current mainchain tip:

```rust
if current_header.chain_work > total_main_chain_chainwork {
    log!("Chain reorg");
    self.reorg_chain(current_header, last_main_chain_block_height);
}
``` [3](#0-2) 

`reorg_chain` removes the displaced mainchain blocks from both `mainchain_header_to_height` and `mainchain_header_to_height` (via `remove_block_header`):

```rust
if let Some(current_main_chain_blockhash) = main_chain_block {
    self.remove_block_header(&current_main_chain_blockhash);
}
``` [4](#0-3) 

The two operations — verification and reorg — share no mutual exclusion and no snapshot isolation. A `true` result from `verify_transaction_inclusion` is therefore only valid for the instant it was computed; it carries no guarantee of permanence.

The `confirmations` field in `ProofArgs` is fully caller-controlled with no enforced floor:

```rust
pub struct ProofArgs {
    pub tx_id: H256,
    pub tx_block_blockhash: H256,
    pub tx_index: u64,
    pub merkle_proof: Vec<H256>,
    pub confirmations: u64,
}
``` [5](#0-4) 

`verify_transaction_inclusion_v2` delegates to the same vulnerable path after its coinbase check:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [6](#0-5) 

---

### Impact Explanation

A downstream contract (cross-chain bridge, payment gateway, escrow) that calls `verify_transaction_inclusion` as a cross-contract call and releases NEAR-side value upon a `true` result can be drained. The attacker obtains a `true` result for a Bitcoin transaction whose block is on the pre-reorg mainchain, then the relayer's `submit_blocks` reorgs that block away. The downstream contract has already acted on the now-invalid proof. This is a direct financial loss to the downstream protocol, not merely an observability issue.

---

### Likelihood Explanation

- Bitcoin shallow reorgs (1–3 blocks) occur several times per year on mainnet; deeper reorgs are rarer but not impossible.
- The relayer is off-chain and its pending `submit_blocks` transactions are visible in the NEAR mempool before they are finalized.
- `verify_transaction_inclusion` is an unrestricted public call — no role, no stake, no fee — so any NEAR account can race it.
- The attacker does not need to control the relayer; they only need to observe a pending reorg submission and submit their proof call in the same or an earlier NEAR block.
- `confirmations = 0` is accepted, making even a 1-block reorg sufficient to invalidate a previously returned `true`.

---

### Recommendation

1. **Enforce a minimum confirmation depth** in `verify_transaction_inclusion`. For Bitcoin, 6 confirmations is the conventional finality threshold; the contract should `require!(args.confirmations >= MIN_CONFIRMATIONS)` where `MIN_CONFIRMATIONS` is a protocol constant, not a caller-supplied value.
2. **Return the chain tip height alongside the boolean** so downstream contracts can detect whether a reorg has occurred between the proof call and their own state update.
3. **Document the race condition explicitly** in the NatSpec/doc comments so integrators know that a `true` result is only valid at the block height at which it was computed.

---

### Proof of Concept

```
State before attack:
  mainchain: ... → [B_n-2] → [B_n-1] → [B_n]  ← mainchain_tip
  fork:      ... → [B_n-2] → [F_n-1] → [F_n] → [F_n+1]
             (fork has higher cumulative chainwork than mainchain)

  Attacker holds a Bitcoin tx T included in block B_n-1.
  A trusted relayer has a pending submit_blocks([F_n-1, F_n, F_n+1]) in the NEAR mempool.

Step 1 — Attacker observes the pending submit_blocks call.

Step 2 — Attacker submits verify_transaction_inclusion({
    tx_id: T,
    tx_block_blockhash: B_n-1,
    confirmations: 1,   // accepted; no minimum enforced
    merkle_proof: <valid proof>,
}) to be processed BEFORE submit_blocks.

Step 3 — verify_transaction_inclusion executes:
  mainchain_header_to_height.get(B_n-1) → Some(n-1)  ✓ (block is on mainchain)
  depth = n - (n-1) + 1 = 2 >= 1                     ✓
  merkle proof valid                                   ✓
  → returns true

Step 4 — Downstream contract releases NEAR-side funds to attacker.

Step 5 — submit_blocks([F_n-1, F_n, F_n+1]) executes:
  fork chainwork > mainchain chainwork → reorg_chain() fires
  B_n-1 is removed from mainchain_header_to_height and headers_pool

Step 6 — verify_transaction_inclusion for the same T now panics:
  "block does not belong to the current main chain"

  But the downstream contract has already paid out. The attacker has
  successfully claimed funds for a Bitcoin transaction that is no longer
  on the canonical chain.
```
