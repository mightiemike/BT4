### Title
`DeclareTransactionV3` (and `InvokeTransactionV3`) with non-empty `account_deployment_data` accepted by gateway but rejected by OS Cairo assertion, causing hash/execution divergence and unprovable blocks — (`crates/starknet_api/src/transaction_hash.rs`, `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The Rust hash function `get_declare_transaction_v3_hash` (and `get_invoke_transaction_v3_hash`) includes `account_deployment_data` in the Poseidon hash preimage. The OS Cairo program `compute_declare_transaction_hash` (and `compute_invoke_transaction_hash`) unconditionally asserts `account_deployment_data_size = 0`. The gateway performs no validation that rejects non-empty `account_deployment_data` for either transaction type. A user can submit a `DeclareTransactionV3` or `InvokeTransactionV3` with non-empty `account_deployment_data`; the gateway accepts it and computes a hash over the non-empty data; the OS Cairo program then fails with an assertion error when attempting to prove any block containing that transaction, making the block unprovable.

---

### Finding Description

**Step 1 — Rust hash includes `account_deployment_data`.**

`get_declare_transaction_v3_hash` hashes `account_deployment_data` unconditionally:

```rust
let account_deployment_data_hash = HashChain::new()
    .chain_iter(transaction.account_deployment_data().0.iter())
    .get_poseidon_hash();
// ...
.chain(&account_deployment_data_hash)
``` [1](#0-0) 

The same pattern exists in `get_invoke_transaction_v3_hash`: [2](#0-1) 

**Step 2 — `DeclareTransactionV3` struct carries the field.** [3](#0-2) 

**Step 3 — Gateway propagates `account_deployment_data` without validation.**

In `convert_rpc_tx_to_internal`, the Declare V3 branch copies `account_deployment_data` verbatim into `InternalRpcDeclareTransactionV3` and then calls `calculate_transaction_hash`, which invokes `get_declare_transaction_v3_hash` over the non-empty data: [4](#0-3) 

No check rejects non-empty `account_deployment_data` before the hash is committed.

**Step 4 — OS Cairo asserts the field must be zero.**

`compute_declare_transaction_hash` in the OS program hard-asserts `account_deployment_data_size = 0` before computing the hash:

```cairo
// TODO(Noa, 01/01/2026): remove the following `assert` once the field is supported.
assert account_deployment_data_size = 0;
``` [5](#0-4) 

The same assertion exists for invoke: [6](#0-5) 

**Step 5 — Divergence.**

| Layer | Behaviour with non-empty `account_deployment_data` |
|---|---|
| Rust `calculate_transaction_hash` | Computes hash H₁ that includes the non-empty data |
| Gateway / mempool | Accepts the transaction with hash H₁ |
| Blockifier `create_tx_info` | Propagates `account_deployment_data` to `CurrentTransactionInfo` (Declare V3 line 230) |
| OS Cairo `compute_declare_transaction_hash` | Assertion `account_deployment_data_size = 0` fires → proof generation fails | [7](#0-6) 

---

### Impact Explanation

A single transaction with non-empty `account_deployment_data` causes the OS to abort proof generation for the entire block that contains it. The sequencer has already committed the state diff; the block cannot be proven and therefore cannot be finalized on L1. This matches:

- **High** — "Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."
- **Critical** — "Wrong state, receipt, event … or revert result from blockifier/syscall/execution logic for accepted input" (the OS cannot produce a valid proof for the accepted state transition).

---

### Likelihood Explanation

Any unprivileged user who can submit transactions to the gateway can trigger this. The `account_deployment_data` field is a standard part of the V3 transaction schema exposed via the public RPC write API. No special privilege or malformed bytes are required — the field simply needs to be non-empty. The gateway performs no length check on it.

---

### Recommendation

Add an explicit validation step in the gateway (or in `convert_rpc_tx_to_internal`) that rejects `DeclareTransactionV3` and `InvokeTransactionV3` transactions whose `account_deployment_data` is non-empty, returning an appropriate `StarknetError` (e.g., `MalformedRequest`), until the OS Cairo program removes the `assert account_deployment_data_size = 0` guard and fully supports the field.

---

### Proof of Concept

1. Construct a valid `RpcDeclareTransactionV3` JSON body with `"account_deployment_data": ["0x1"]` (any non-empty array).
2. POST it to the gateway's `add_transaction` endpoint.
3. The gateway calls `convert_rpc_tx_to_internal` → `get_declare_transaction_v3_hash` → hash H₁ is computed over the non-empty `account_deployment_data`. No error is returned; the transaction is admitted to the mempool.
4. The batcher includes the transaction in a proposed block.
5. The OS program is invoked to prove the block. `compute_declare_transaction_hash` executes `assert account_deployment_data_size = 0` with the hint-provided size of 1 → assertion fails → proof generation aborts.
6. The block is unprovable and cannot be submitted to L1, while the sequencer has already advanced its local state.

func compute_invoke_transaction_hash{range_check_ptr, poseidon_ptr: PoseidonBuiltin*}(
    common_fields: CommonTxFields*,
    execution_context: ExecutionContext*,
    account_deployment_data_size: felt,
    account_deployment_data: felt*,
    proof_facts_size: felt,
    proof_facts: felt*,
) -> felt {
    alloc_locals;

    // TODO(Noa, 01/01/2026): remove the following `assert` once the field is supported.
    assert account_deployment_data_size = 0;
    with_attr error_message("Invalid transaction version: {version}.") {
        assert common_fields.version = 3;
    }

    let hash_state: PoseidonHashState = poseidon_hash_init();
    with hash_state {
        hash_tx_common_fields(common_fields=common_fields);
        poseidon_hash_update_with_nested_hash(
            data_ptr=account_deployment_data, data_length=account_deployment_data_size
        );
        poseidon_hash_update_with_nested_hash(
            data_ptr=execution_context.calldata, data_length=execution_context.calldata_size
        );
        // For backward compatibility, we don't hash proof facts if they are empty.
        if (proof_facts_size != 0) {
            poseidon_hash_update_with_nested_hash(
                data_ptr=proof_facts, data_length=proof_facts_size
            );
        }
    }
    let transaction_hash = poseidon_hash_finalize(hash_state=hash_state);
    return transaction_hash;
}

// See comment above `compute_invoke_transaction_hash()`.
func compute_declare_transaction_hash{range_check_ptr, poseidon_ptr: PoseidonBuiltin*}(
    common_fields: CommonTxFields*,
    class_hash: felt,
    compiled_class_hash: felt,
    account_deployment_data_size: felt,
    account_deployment_data: felt*,
) -> felt {
    alloc_locals;

    // TODO(Noa, 01/01/2026): remove the following `assert` once the field is supported.
    assert account_deployment_data_size = 0;
    with_attr error_message("Invalid transaction version: {version}.") {
        assert common_fields.version = 3;
    }

    let hash_state: PoseidonHashState = poseidon_hash_init();
    with hash_state {
        hash_tx_common_fields(common_fields=common_fields);
        poseidon_hash_update_with_nested_hash(
            data_ptr=account_deployment_data, data_length=account_deployment_data_size
        );
        // Add the class hash to the hash state.
        poseidon_hash_update_single(item=class_hash);
        poseidon_hash_update_single(item=compiled_class_hash);
    }
    let transaction_hash = poseidon_hash_finalize(hash_state=hash_state);

    return transaction_hash;
}
