### Title
Stale `lastLiquidationFees` Cross-Subaccount Contamination Inflates Insurance Storage — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`lastLiquidationFees` is a single shared storage variable that is overwritten by every call to `_handleLiquidationPayment`, regardless of which subaccount is being liquidated. `_finalizeSubaccount` reads this variable without any check that it belongs to the subaccount currently being finalized. When a different subaccount's liquidation occurs between subaccount B's last position-liquidation step and B's finalization call, the stale value is used, producing incorrect `insuranceCover` and leaving `insurance` storage inflated relative to actual token backing.

---

### Finding Description

**Storage layout** — `lastLiquidationFees` is declared as a plain `int128` in `ClearinghouseStorage`, shared across all liquidations: [1](#0-0) 

**Write site** — every call to `_handleLiquidationPayment` unconditionally overwrites it with the fees of the current liquidation, for any subaccount: [2](#0-1) 

**Read site** — `_finalizeSubaccount` reads the variable twice with no ownership check: [3](#0-2) 

**Design intent** (per the inline comment at lines 581–585): when finalizing a subaccount, the fees from *that subaccount's* last liquidation step are subtracted from `v.insurance` before socialization, then added back afterward. The purpose is to prevent those freshly-deposited fees from blocking the socialization path. The subtract/add-back is a no-op on the final `insurance` value, but it changes the intermediate `insuranceCover = min(v.insurance, -quoteBalance.amount)`.

**The bug** — `liquidateSubaccountImpl` calls `_finalizeSubaccount` first; if `productId != type(uint32).max` it returns `false` and then `_handleLiquidationPayment` runs: [4](#0-3) 

This means any non-finalization liquidation of *any* subaccount overwrites `lastLiquidationFees`. If subaccount A's liquidation is processed between B's last position-liquidation and B's finalization call, `lastLiquidationFees` holds A's fees (`X`) instead of B's fees (`Y`).

**Accounting trace** — let:
- `I` = `insurance` storage before B's finalization
- `X` = stale `lastLiquidationFees` (from A)
- `Y` = B's correct last liquidation fees
- `Q` = B's quote balance (negative; `-Q > 0` is the deficit)

With stale `X`:
```
v.insurance = I - X
insuranceCover = min(I - X, -Q)
insurance (final) = I - insuranceCover_stale
```

With correct `Y`:
```
v.insurance = I - Y
insuranceCover = min(I - Y, -Q)
insurance (final) = I - insuranceCover_correct
```

When `X > Y`, `insuranceCover_stale < insuranceCover_correct`, so `insurance_final_stale > insurance_final_correct`. The protocol records more insurance than it actually deployed.

**Worst-case scenario** — `I - X < -Q ≤ I - Y` (i.e., `X > I + Q` and `Y ≤ I + Q`):
- Correct path: full coverage, `insurance = I + Q`, no socialization
- Stale path: partial coverage `I - X`, `insurance = X`, socialization fires incorrectly
- Insurance inflated by `X − (I + Q)` and losses are incorrectly mutualized across other users

**Common scenario** — both `I - X < -Q` and `I - Y < -Q`:
- Correct: `insurance = Y`
- Stale: `insurance = X`
- Insurance inflated by `X − Y`

---

### Impact Explanation

The `insurance` storage variable no longer equals actual quote tokens held by the clearinghouse minus outstanding liabilities. It is inflated by up to `X` (the stale fees). Any subsequent operation that reads `insurance` — including `_assertLiquidationAmount` at line 137/175, spread liability checks, and future finalizations — operates on a phantom balance. Over time, repeated cross-contamination compounds the inflation. An attacker who controls the ordering of liquidation transactions (or who can predict sequencer ordering) can maximize `X` by ensuring a high-fee liquidation of A is interleaved just before B's finalization, maximizing the inflation delta. [5](#0-4) 

---

### Likelihood Explanation

The precondition is two sequential liquidations of different subaccounts, which is routine in any market stress event. No special privileges are required — the sequencer processes `LiquidateSubaccount` transactions in order, and any liquidator can submit them. The stale-read path is triggered automatically whenever the ordering is: `liquidate(A, productId≠max)` → `liquidate(B, productId=max)`.

---

### Recommendation

Bind `lastLiquidationFees` to the subaccount being liquidated. The simplest fix is to replace the single `int128 lastLiquidationFees` storage slot with a `mapping(bytes32 => int128) lastLiquidationFees` keyed by subaccount, and reset the entry to zero after `_finalizeSubaccount` consumes it. Alternatively, pass the fees as a local parameter through the call chain rather than persisting them in storage across calls.

---

### Proof of Concept

```solidity
// Pseudocode — two-liquidation sequence
// Setup: subaccount A and B are both under maintenance health.
// B has already had all positions liquidated except finalization.
// A has one remaining perp position.

// Step 1: liquidate A's perp → _handleLiquidationPayment sets lastLiquidationFees = X (large)
endpoint.submitSlowModeTransaction(liquidateSubaccount(A, perpProductId, amount));

// Step 2: finalize B → _finalizeSubaccount reads lastLiquidationFees = X (stale, should be Y)
endpoint.submitSlowModeTransaction(liquidateSubaccount(B, type(uint32).max, 0));

// Assert: insurance storage != actual USDC balance of clearinghouse minus liabilities
// Expected: insurance inflated by (X - Y) or more
assertEq(clearinghouse.getInsurance(), actualTokenBalance - liabilities); // FAILS
```

The test should set `X >> Y` (e.g., A is a large perp position, B had a tiny last fee) to maximize the observable delta. After both transactions, `clearinghouse.getInsurance()` will exceed the true backing by approximately `X − Y`.
