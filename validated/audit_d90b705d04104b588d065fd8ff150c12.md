Audit Report

## Title
SwapAllowlistExtension checks router address instead of end-user, allowing any trader to bypass the per-swapper allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` as the address that called `MetricOmmPool.swap`, which is the router contract (`MetricOmmSimpleRouter`), not the actual end user. Because the allowlist check validates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][end_user]`, any unprivileged trader can bypass a pool's swap allowlist simply by routing through `MetricOmmSimpleRouter`. If the router is allowlisted (required for any router-mediated swap to succeed), the per-user restriction is entirely nullified.

## Finding Description
**Root cause — `sender` is the router, not the end user:**

In `MetricOmmPool.swap` (line 230–240), `_beforeSwap` is called with `msg.sender` as the first argument:
```solidity
_beforeSwap(
    msg.sender,   // ← this is the router, not the end user
    recipient, zeroForOne, amountSpecified,
    priceLimitX64, packedSlot0Initial,
    bidPriceX64, askPriceX64, extensionData
);
```
`msg.sender` here is `MetricOmmSimpleRouter` because the router called `pool.swap()` at line 72–80 of `MetricOmmSimpleRouter.exactInputSingle`:
```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```
The actual end user's address (`msg.sender` of `exactInputSingle`) is stored only in transient callback context for payment purposes — it is never forwarded to the pool or the extension as the swapper identity.

**The allowlist check:**

`SwapAllowlistExtension.beforeSwap` (lines 31–41):
```solidity
function beforeSwap(address sender, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```
`msg.sender` = pool address, `sender` = router address. The check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`.

**Exploit path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists only specific addresses (e.g., KYC'd users).
2. Pool admin also allowlists `MetricOmmSimpleRouter` so that legitimate users can swap via the router.
3. A non-allowlisted attacker calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the restricted pool.
4. The pool calls `extension.beforeSwap(router_address, ...)`.
5. The extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. The attacker successfully swaps in a pool they were explicitly excluded from.

**Existing guards are insufficient:** The `allowedSwapper` mapping is keyed by the caller of `pool.swap()`, which is always the router for router-mediated swaps. There is no mechanism in the hook path to recover the original end user's address.

## Impact Explanation
The swap allowlist is a core access-control feature allowing pool admins to restrict which addresses may trade in a pool (e.g., for regulatory compliance, private pools, or phased rollouts). The bypass allows any unprivileged trader to execute swaps in pools they are explicitly excluded from. This constitutes a broken core pool functionality and an admin-boundary break: the pool admin's configured restriction is fully circumvented by an unprivileged path (the public router). Depending on pool design, this can result in unauthorized fund flows through restricted pools.

## Likelihood Explanation
Exploitability is trivial and requires no special privileges. Any user with access to `MetricOmmSimpleRouter` (a public contract) can bypass the allowlist. The condition is met whenever: (a) a pool uses `SwapAllowlistExtension` with a non-universal allowlist, and (b) the router is allowlisted (which is necessary for any router-based swap to work). This is a realistic and common deployment configuration. The attack is repeatable with every swap call.

## Recommendation
The `sender` forwarded to extension hooks must represent the true end user, not the intermediary router. Two complementary fixes:

1. **In the router:** Pass the actual end user's address as `recipient` or encode it in `extensionData` so the extension can recover it. Alternatively, use a dedicated field in the swap call for the "originating user."
2. **In the extension:** Do not rely solely on the `sender` parameter from the hook. If the caller is a known router, extract the true user from `extensionData` (which the router can populate with `msg.sender` before calling the pool).
3. **Preferred fix:** Add an `originator` parameter to the pool's `swap` function and the `beforeSwap` hook signature, populated by the router with `msg.sender`, so the extension can check `allowedSwapper[pool][originator]`.
