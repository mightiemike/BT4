### Title
`deser_auto` silently drops `max_atom_len` for legacy/backrefs blobs, creating format-dependent acceptance asymmetry — (`wheel/src/api.rs`)

### Summary
`deser_auto` accepts a `max_atom_len` parameter that is enforced only on the 2026-format path. For legacy/backrefs blobs, the parameter is silently ignored and `node_from_bytes_backrefs` is called with no atom-length cap. The same logical CLVM tree (e.g., a single 2 MB atom) is accepted when encoded in legacy format and rejected when encoded in 2026 format, despite identical `max_atom_len` arguments.

### Finding Description
`PY_DEFAULT_MAX_ATOM_LEN` is `1 << 20` (1 MiB). [1](#0-0) 

`deser_auto` dispatches on the magic prefix. For 2026 blobs it passes `max_atom_len` to `deserialize_2026_body_from_stream`. For all other blobs it calls `node_from_bytes_backrefs` with no length cap. [2](#0-1) 

`node_from_bytes_backrefs` has no `max_atom_len` parameter and imposes no per-atom limit. [3](#0-2) 

The only size guard in the legacy path is `decode_size_with_offset`'s hard rejection of atoms ≥ `0x400000000` (~17 GB). [4](#0-3) 

The 2026 path enforces `max_atom_len` via `checked_bounded_usize` on every atom group. [5](#0-4) 

`checked_bounded_usize` is used exclusively in `src/serde_2026/de.rs` — it is never called from the legacy deserializer. [6](#0-5) 

### Impact Explanation
Any caller that passes `max_atom_len` to `deser_auto` expecting uniform enforcement across all formats receives inconsistent behavior. An attacker who controls the serialization format can choose legacy encoding to bypass the caller-supplied atom-length cap. The same logical CLVM tree is accepted or rejected purely based on wire format, not on the tree's content. This is a concrete API-equivalence failure: `deser_auto(legacy_2mb_blob, max_atom_len=1<<20)` succeeds; `deser_auto(serde2026_2mb_blob, max_atom_len=1<<20)` raises `ValueError`. [7](#0-6) 

### Likelihood Explanation
The attacker only needs to control the serialization format of the blob — a trivially achievable precondition since both formats are produced by the same library. The legacy format is the default for existing tooling. No special privileges or compromised components are required. [8](#0-7) 

### Recommendation
Pass `max_atom_len` through to the legacy path. Either add a `max_atom_len` parameter to `node_from_bytes_backrefs` / `node_from_stream_backrefs`, or add a post-parse atom-length walk in `deser_auto` for the legacy branch. The fix must ensure that `deser_auto(blob, max_atom_len=N)` rejects any atom > N bytes regardless of wire format. [9](#0-8)

pub fn node_from_stream_backrefs(
    allocator: &mut Allocator,
    f: &mut Cursor<&[u8]>,
    mut backref_callback: impl FnMut(NodePtr),
) -> Result<NodePtr> {
    // this contains the actual value and then an optional cached list which represents value stack as a chialisp list
    // we only build this list when we need it - i.e we have a backreference that points at the stack itself rather than a value
    // we reuse the most recent cached value when rebuilding the list
    let mut values = Vec::<(NodePtr, Option<NodePtr>)>::new();
    let mut ops = vec![ParseOp::SExp];

    let mut b = [0; 1];
    while let Some(op) = ops.pop() {
        match op {
            ParseOp::SExp => {
                f.read_exact(&mut b)?;
                if b[0] == CONS_BOX_MARKER {
                    ops.push(ParseOp::Cons);
                    ops.push(ParseOp::SExp);
                    ops.push(ParseOp::SExp);
                } else if b[0] == BACK_REFERENCE {
                    let path = parse_path(f)?;
                    let back_reference = traverse_path_with_vec(allocator, path, &mut values)?;
                    backref_callback(back_reference);
                    allocator.add_ghost_pair(1)?;
                    values.push((back_reference, None));
                } else {
                    let new_atom = parse_atom(allocator, b[0], f)?;
                    allocator.add_ghost_pair(1)?; // return error if we have too many pairs
                    values.push((new_atom, None));
                }
            }
            ParseOp::Cons => {
                // cons
                // pop left and right values off of the "values" stack, then
                // push the new pair onto it
                let right = values.pop().expect("No cons without two vals.");
                let left = values.pop().expect("No cons without two vals.");
                let root_node = allocator.new_pair(left.0, right.0)?;
                allocator.add_ghost_pair(1)?;
                values.push((root_node, None));
            }
        }
    }
    Ok(values.pop().expect("Top of the stack").0)
}
