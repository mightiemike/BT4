### Title
Deprecated `verify_transaction_inclusion` Exposes Merkle Second-Preimage Forgery to Any NEAR Caller — (File: `contract/src/lib.rs`)

---

### Summary

The `verify_transaction_inclusion` function remains a live, callable public NEAR contract method despite being deprecated. Its underlying `compute_root_from_merkle_proof` implementation never validates proof depth against the actual Merkle tree depth, so any unprivileged NEAR caller can supply an internal Merkle-tree node hash as `tx_id` with a shortened proof and receive a `true` return value — forging Bitcoin transaction inclusion.

---

### Finding Description

`verify_transaction_inclusion` delegates all cryptographic work to `merkle_tools::compute_root_from_merkle_proof`:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

`compute_root_from_merkle_proof` iterates over the caller-supplied proof hashes and hashes upward, but it never checks that `merkle_proof.len()` equals the actual tree depth:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;
    for proof_hash in merkle_proof {
        ...
        current_position /= 2;
    }
    current_hash
}
``` [2](#0-1) 

The only guard in `verify_transaction_inclusion` is `require!(!args.merkle_proof.is_empty(), ...)`, which does not constrain depth: [3](#0-2) 

**Concrete attack for a block with 4 transactions T0–T3:**

| Merkle tree level | Nodes |
|---|---|
| Leaves | T0, T1, T2, T3 |
| Level 1 | H01 = hash(T0,T1), H23 = hash(T2,T3) |
| Root | R = hash(H01, H23) |

An attacker submits `tx_id = H01`, `tx_index = 0`, `merkle_proof = [H23]`.  
`compute_root_from_merkle_proof(H01, 0, [H23])` = `hash(H01, H23)` = `R` = `header.block_header.merkle_root`.  
The function returns `true` — but `H01` is an internal node, not a real transaction.

The function is still a live `pub` NEAR method (the `#[deprecated]` attribute is a Rust compiler hint only, not a runtime restriction): [4](#0-3) 

---

### Impact Explanation

Any NEAR caller can invoke `verify_transaction_inclusion` with a crafted `ProofArgs` and receive `true` for a Bitcoin transaction that never existed. Downstream consumers — bridges, atomic-swap contracts, payment-verification logic — that rely on this function's return value would incorrectly treat a non-existent Bitcoin payment as confirmed. The corrupted proof result is the exact value that drives cross-chain authorization decisions.

---

### Likelihood Explanation

- No privilege is required; the function is callable by any NEAR account.
- All information needed to construct the attack (block Merkle tree structure) is publicly available from any Bitcoin node or block explorer.
- The attack is deterministic and requires no brute force.
- The `#[pause]` guard is inactive by default; the function is live on deployment.

---

### Recommendation

1. **Remove** `verify_transaction_inclusion` from the contract entirely, or gate it with an explicit runtime panic rather than a deprecation notice, so it cannot be called.
2. Direct all callers to `verify_transaction_inclusion_v2`, which enforces equal-length coinbase and transaction proofs, bounding the effective tree depth and preventing the second-preimage substitution. [5](#0-4) 

---

### Proof of Concept

1. Identify any confirmed Bitcoin block with ≥ 2 transactions (e.g., block at height 685452 used in the contract's own tests).
2. Compute `H01 = double_sha256(T0 || T1)` from the block's transaction list.
3. Call `verify_transaction_inclusion` on the NEAR contract:
   ```
   tx_id            = H01          // internal node, not a real tx
   tx_block_blockhash = <valid block hash on mainchain>
   tx_index         = 0
   merkle_proof     = [H23]        // one sibling is enough
   confirmations    = 0
   ```
4. The contract returns `true`.
5. Any bridge or payment-verification contract consuming this result now believes a non-existent Bitcoin transaction was confirmed in that block.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
