### Title
Resharding Cooldown Invariant Silently Bypassed at Static-to-Dynamic Resharding Transition — (`chain/epoch-manager/src/lib.rs`)

### Summary

The `can_reshard()` function enforces a minimum number of epochs between shard splits by reading `EpochInfo::last_resharding()`. However, `last_resharding()` returns `None` for all pre-V5 `EpochInfo` variants (V1–V4), which are the variants produced before `ProtocolFeature::DynamicResharding` is enabled. At the static-to-dynamic transition boundary, the first V5 `EpochInfo` inherits `last_resharding = None` from its V4 predecessor regardless of whether a static resharding occurred in the immediately preceding epoch. The `is_none_or` guard then unconditionally permits a dynamic resharding, breaking the cooldown invariant the code itself documents as safety-critical.

### Finding Description

`can_reshard()` in `chain/epoch-manager/src/lib.rs` reads the cooldown state from the next epoch's `EpochInfo`:

```rust
// last_resharding() returns `None` if no resharding happened since dynamic resharding
// has been enabled. It is theoretically possible that a static resharding was scheduled
// right before enabling dynamic resharding, but we assume this didn't happen.
let can_reshard = next_epoch_info.last_resharding().is_none_or(|last_resharding| {
    next_epoch_info.epoch_height() - last_resharding >= min_epochs_between_resharding.get()
});
``` [1](#0-0) 

`EpochInfo::last_resharding()` returns `None` for every pre-V5 variant:

```rust
pub fn last_resharding(&self) -> Option<EpochHeight> {
    match self {
        Self::V1(_) | Self::V2(_) | Self::V3(_) | Self::V4(_) => None,
        Self::V5(v5) => v5.last_resharding,
    }
}
``` [2](#0-1) 

`EpochInfoV5` is only created when `ProtocolFeature::DynamicResharding` is enabled for the epoch's protocol version: [3](#0-2) 

In `finalize_epoch()`, the `last_resharding` value propagated into the first V5 `EpochInfo` is derived from the predecessor's `last_resharding()`:

```rust
let last_resharding = (!has_same_shard_layout)
    .then(|| next_epoch_info.epoch_height() + 1)
    .or_else(|| next_epoch_info.last_resharding());
``` [4](#0-3) 

When the predecessor is V4 (the epoch immediately before dynamic resharding activates), `next_epoch_info.last_resharding()` is `None`. If no dynamic resharding occurred in that transitional epoch (the normal case), `has_same_shard_layout` is `true`, so `last_resharding` is set to `None`. The first V5 `EpochInfo` therefore carries `last_resharding = None` even if a static resharding occurred one or two epochs earlier.

The concrete epoch sequence:

| Epoch | Protocol | EpochInfo | Static resharding? | `last_resharding` stored |
|-------|----------|-----------|-------------------|--------------------------|
| N     | X        | V4        | Yes (e.g. 5→6 shards) | N/A (V4 has no field) |
| N+1   | X+1 (DynamicResharding enabled) | V5 | No | `None` (inherited from V4 predecessor) |
| N+2   | X+1      | V5        | —                 | `None` (no dynamic split yet) |

When `get_upcoming_shard_split()` is called at the last block of epoch N+1 to decide the layout for epoch N+3, `can_reshard()` checks the V5 `EpochInfo` for epoch N+2, finds `last_resharding = None`, and `is_none_or` returns `true`. The cooldown is bypassed entirely. [5](#0-4) 

### Impact Explanation

The documentation explicitly states the consequence:

> `min_epochs_between_resharding` must be `> 0`: allowing back-to-back reshardings is unsafe because a freshly-created child shard would inherit `proposed_split` from the parent's final chunk while its own first chunk freshly computes `proposed_split = None`, triggering `InvalidChunkHeaderShardSplit`. [6](#0-5) 

A dynamic resharding triggered in epoch N+2 (two epochs after a static resharding in epoch N) constitutes a back-to-back resharding. The child shards produced by the static split in epoch N would be the parents for the dynamic split in epoch N+2. Their chunk headers carry a `proposed_split` computed from the parent's final chunk, while the child's own first chunk computes `proposed_split = None`, producing an `InvalidChunkHeaderShardSplit` validation error that halts chunk processing for the affected shard.

The shard-layout compatibility invariant — that `ShardUId` values are unique across epochs and that the ancestor map in `ShardLayoutV3` is consistent — is also at risk: a V3 layout derived from a V2 layout that was itself just derived from a static split may produce incorrect ancestor tracking if the split history reconstruction via `get_shard_layout_history()` does not account for the interleaved static split. [7](#0-6) 

### Likelihood Explanation

The scenario requires a static resharding to occur in the epoch immediately before (or within `min_epochs_between_resharding` epochs of) the activation of `ProtocolFeature::DynamicResharding`. This is precisely the expected mainnet deployment path: static resharding adds shards up to the desired count, then dynamic resharding is enabled for future automatic splits. The `force_split_shards` field in `DynamicReshardingConfig` (set by the NEAR team in the epoch config) can trigger a resharding in the very first dynamic epoch without requiring any memory threshold to be met, making the bypass reachable without unprivileged user action. The code comment at line 844 explicitly acknowledges the assumption is unverified: *"we assume this didn't happen."* [8](#0-7) 

### Recommendation

When constructing the first V5 `EpochInfo` (i.e., when `next_epoch_info` is V4), compare the current and previous shard layouts to detect whether a static resharding occurred within the cooldown window. If one did, initialize `last_resharding` to the epoch height of that static resharding rather than `None`. Concretely, in `finalize_epoch()`, before the `or_else(|| next_epoch_info.last_resharding())` fallback, check whether the shard layout changed between the epoch before `next_epoch_id` and `next_epoch_id` itself, and if so, set `last_resharding` to `next_epoch_info.epoch_height()`.

Alternatively, remove the `is_none_or` shortcut and treat `None` as "resharding happened at epoch 0," which would make the cooldown check conservative (always enforce the cooldown from genesis) rather than permissive.

### Proof of Concept

1. Mainnet is at protocol version 85 with a static 6-shard layout (V2). A static resharding to 7 shards is included in protocol version 86.
2. Protocol version 87 enables `ProtocolFeature::DynamicResharding` with `min_epochs_between_resharding = 1` and `force_split_shards = [shard_6]`.
3. Epoch N finalizes at protocol version 86: static resharding executes, `EpochInfo` for epoch N+2 is V4 with the new 7-shard layout.
4. Epoch N+1 finalizes at protocol version 87: `finalize_epoch()` creates V5 `EpochInfo` for epoch N+2. `next_epoch_info` (epoch N+1) is V4, so `last_resharding()` = `None`. `has_same_shard_layout` = `true` (no dynamic split yet). Result: V5 `EpochInfo` for epoch N+2 has `last_resharding = None`.
5. At the last block of epoch N+1, `get_upcoming_shard_split()` is called. `can_reshard()` reads the V5 `EpochInfo` for epoch N+2, finds `last_resharding = None`, `is_none_or` returns `true`. `force_split_shards` triggers a split of shard 6.
6. `finalize_epoch()` for epoch N+1 derives a new `ShardLayoutV3` for epoch N+3 (7→8 shards), one epoch after the static 6→7 split.
7. The child shards of the static split (epoch N+2) carry `proposed_split` from their parent's final chunk. Their own first chunk computes `proposed_split = None`. `validate_chunk_with_chunk_extra_and_receipts_root` detects the mismatch and returns `InvalidChunkHeaderShardSplit`, stalling chunk production for those shards. ```rust
    pub fn get_upcoming_shard_split(
        &self,
        protocol_version: ProtocolVersion,
        parent_hash: &CryptoHash,
        chunk_headers: &[ShardChunkHeader],
    ) -> Result<Option<(ShardId, AccountId)>, EpochError> {
        // Check if dynamic resharding is enabled
        let epoch_config = self.get_epoch_config(protocol_version);
        let dynamic_resharding_config = match &epoch_config.shard_layout_config {
            ShardLayoutConfig::Static { .. } => return Ok(None),
            ShardLayoutConfig::Dynamic { dynamic_resharding_config } => dynamic_resharding_config,
        };

        // Check if resharding is allowed based on epoch constraints
        let can_reshard = self
            .can_reshard(&parent_hash, dynamic_resharding_config.min_epochs_between_resharding)?;
        if !can_reshard {
            return Ok(None);
        }

        // Collect proposed splits from chunk headers
        let mut proposed_splits = HashMap::new();
        for chunk_header in chunk_headers {
            if let Some(split) = chunk_header.proposed_split() {
                proposed_splits.insert(chunk_header.shard_id(), split.clone());
            }
        }

        // Pick the shard to split
        let Some((shard_id, split)) =
            pick_shard_to_split(&proposed_splits, dynamic_resharding_config)
        else {
            return Ok(None);
        };

        Ok(Some((shard_id, split.boundary_account)))
    }
