### Title
Abrupt Risk Weight Change via `updateRisk` Enables Liquidation Sandwich Attack — (`core/contracts/BaseEngine.sol`)

---

### Summary

`updateRisk` in `BaseEngine.sol` atomically replaces all risk parameters for a product with no continuity guard. When an owner tightens `longWeightMaintenance` or `shortWeightMaintenance`, previously healthy subaccounts can instantly fall below the maintenance threshold. An unprivileged liquidator can observe the pending `updateRisk` transaction in the mempool and back-run it with `liquidateSubaccount` calls, extracting value from users who had no opportunity to reduce their exposure.

---

### Finding Description

`updateRisk` in `BaseEngine.sol` performs a single unconditional write:

```solidity
_risk().value[productId] = riskStore;
``` [1](#0-0) 

There is no check that the health of existing positions remains consistent before and after the write. The only validation is the ordering invariant between initial and maintenance weights:

```solidity
require(
    riskStore.longWeightInitial <= riskStore.longWeightMaintenance &&
        riskStore.shortWeightInitial >= riskStore.shortWeightMaintenance,
    ERR_BAD_PRODUCT_CONFIG
);
``` [2](#0-1) 

These weights feed directly into `RiskHelper._getWeightX18`, which is the sole source of per-product weight used in every health calculation:

```solidity
weight = healthType == IProductEngine.HealthType.INITIAL
    ? risk.longWeightInitialX18
    : risk.longWeightMaintenanceX18;
``` [3](#0-2) 

The `ContractOwner` contract exposes `spotUpdateRisk` and `perpUpdateRisk` as direct `onlyOwner` entry points that call `updateRisk` immediately: [4](#0-3) 

The liquidation path in `ClearinghouseLiq.sol` uses these weights to determine whether a subaccount is liquidatable. Because the weight change is atomic and instantaneous, the transition from "healthy" to "liquidatable" happens in a single block with no grace period. [5](#0-4) 

---

### Impact Explanation

When `longWeightMaintenance` is reduced (or `shortWeightMaintenance` increased) for a product, the maintenance health of every subaccount holding that product drops instantly. Subaccounts that were healthy by a small margin become liquidatable in the same block the `updateRisk` transaction is included. A liquidator who sandwiches the transaction can liquidate those accounts before their owners can react, capturing the liquidation bonus at the expense of users who had no warning.

The `spreads` variable in `Clearinghouse.setSpreads` presents a secondary analog: changing which (spot, perp) pairs are recognized as spread positions alters the liability liquidation path in `_assertCanLiquidateLiability`, which iterates `spreads` to determine valid liquidation preconditions. [6](#0-5) 

---

### Likelihood Explanation

- `updateRisk` is a routine admin operation expected to be called when market conditions change (e.g., tightening collateral requirements for a volatile asset).
- Transactions are publicly visible in the mempool before inclusion.
- The liquidation entry point (`liquidateSubaccount` via `Endpoint`) is permissionless — any address can call it.
- No time-lock or delay mechanism exists between the `updateRisk` call and its effect on health calculations.

---

### Recommendation

Before applying new risk weights, compute the maintenance health of the affected product under the new weights and require that the change does not cause any currently-tracked open interest to become immediately liquidatable, or enforce a time-lock / two-step commit-reveal pattern for `updateRisk` so that users have at least one block to adjust their positions before the new weights take effect. This mirrors the fix recommended in the external report: ensure the value returned by the health function remains the same immediately before and after the parameter change.

---

### Proof of Concept

1. Owner broadcasts `spotUpdateRisk(productId, newRiskStore)` where `newRiskStore.longWeightMaintenance` is reduced from `0.9e9` to `0.7e9`.
2. Liquidator observes the pending transaction in the mempool.
3. Liquidator submits `liquidateSubaccount` targeting subaccounts with large long positions in `productId`, with a higher gas price to ensure inclusion in the same block immediately after `updateRisk`.
4. `updateRisk` executes: `_risk().value[productId] = newRiskStore` — maintenance health of long holders drops instantly.
5. Liquidator's `liquidateSubaccount` executes in the same block: the subaccounts now fail the maintenance health check and are liquidated, with the liquidator capturing the liquidation bonus.
6. Affected users had no opportunity to reduce their positions or add collateral. [1](#0-0) [7](#0-6)

    function _getWeightX18(
        Risk memory risk,
        int128 amount,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }

        int128 weight;
        if (amount >= 0) {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.longWeightInitialX18
                : risk.longWeightMaintenanceX18;
        } else {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.shortWeightInitialX18
                : risk.shortWeightMaintenanceX18;
        }

        return weight;
    }
```
