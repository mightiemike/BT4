### Title
Single Hardcoded `latest_constants()` Cap Applied Uniformly Across All Block Versions in Fee-Proposal Validation — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid()` and `compute_proposer_fee_proposal()` both read `fee_proposal_margin_ppt` from `VersionedConstants::latest_constants()` rather than from the version-specific constants keyed on the block's own `starknet_version`. This is a direct analog of the external report's single hardcoded cap: one margin value is applied uniformly to every block regardless of the Starknet version that block was produced under. When `fee_proposal_margin_ppt` diverges across versions — a plausible future change — the proposer and validator will silently compute different acceptance windows for the same `fee_proposal`, breaking the consensus canonicalization invariant and producing an incorrect committed L2 gas price.

---

### Finding Description

`is_proposal_init_valid()` already performs a version-aware gate:

```rust
// fee_proposal is required iff Starknet version >= V0_14_3.
let fee_proposal_required = init_proposed.starknet_version >= StarknetVersion::V0_14_3;
``` [1](#0-0) 

But immediately after, when computing the acceptance window, it discards the block's version and falls back to the binary-global latest:

```rust
let (lower_bound, upper_bound) = fee_proposal_bounds(
    fee_actual,
    VersionedConstants::latest_constants().fee_proposal_margin_ppt,
);
``` [2](#0-1) 

The proposer side has the identical pattern:

```rust
let proposal = compute_fee_proposal(
    fee_target,
    fee_actual,
    VersionedConstants::latest_constants().fee_proposal_margin_ppt,
);
``` [3](#0-2) 

`VersionedConstants::latest_constants()` is defined to always return the constants for `StarknetVersion::LATEST`:

```rust
fn latest_constants() -> &'static Self {
    Self::get(&starknet_api::block::StarknetVersion::LATEST)
        .expect("Latest version should support VC.")
}
``` [4](#0-3) 

The orchestrator's `VersionedConstants` struct carries `fee_proposal_margin_ppt` as a per-version field:

```rust
/// Maximum `fee_proposal` change per block in parts per thousand (e.g., `2` = 0.2%).
pub fee_proposal_margin_ppt: u128,
``` [5](#0-4) 

All currently deployed versions happen to share the same value of `2`: [6](#0-5) [7](#0-6) 

The version-aware lookup path (`VersionedConstants::get(version)`) exists and is used correctly elsewhere (e.g., blockifier's `build_block_context`):

```rust
let mut versioned_constants =
    VersionedConstants::get(&block_info.starknet_version)?.clone();
``` [8](#0-7) 

The same `latest_constants()` shortcut also appears in `calculate_next_base_gas_price` for `gas_price_max_change_denominator` and `gas_target`:

```rust
let versioned_constants = VersionedConstants::latest_constants();
``` [9](#0-8) 

---

### Impact Explanation

`fee_proposal_fri` is the L2 gas price the proposer commits to for the next block; it is bound into the `ConsensusProposalCommitment` that all validators sign. When `fee_proposal_margin_ppt` diverges between versions:

1. **Wrong committed L2 gas price (Critical – Incorrect fee/gas effect with economic impact):** A proposer running a binary whose `latest_constants()` returns a wider margin can submit a `fee_proposal` outside the narrower margin of the block's actual Starknet version. Validators whose binary has the same wider `latest_constants()` accept it; the wrong L2 gas price is committed to the block and propagated to all subsequent EIP-1559 calculations.

2. **Consensus split (High – admission of invalid / rejection of valid proposals):** During a rolling upgrade where `fee_proposal_margin_ppt` changes, nodes running the old binary and nodes running the new binary compute different `[lower_bound, upper_bound]` windows for the same `fee_proposal`. Some validators accept the proposal; others reject it. The network cannot reach BFT agreement on the block.

---

### Likelihood Explanation

The bug is latent today because all five deployed versions share `fee_proposal_margin_ppt = 2`. It becomes reachable the moment any future version file sets a different value — a routine tuning change. The code path is exercised on every proposal validation round, so there is no additional precondition once the constant diverges. No privileged attacker is required; the divergence is introduced by a normal protocol upgrade.

---

### Recommendation

Replace both `latest_constants()` calls with a version-keyed lookup using the block's own `starknet_version`:

```rust
// In is_proposal_init_valid():
let margin_ppt = VersionedConstants::get(&init_proposed.starknet_version)
    .map(|vc| vc.fee_proposal_margin_ppt)
    .unwrap_or_else(|_| VersionedConstants::latest_constants().fee_proposal_margin_ppt);
let (lower_bound, upper_bound) = fee_proposal_bounds(fee_actual, margin_ppt);

// In compute_proposer_fee_proposal():
let margin_ppt = VersionedConstants::get(&self.starknet_version)
    .map(|vc| vc.fee_proposal_margin_ppt)
    .unwrap_or_else(|_| VersionedConstants::latest_constants().fee_proposal_margin_ppt);
let proposal = compute_fee_proposal(fee_target, fee_actual, margin_ppt);
```

Apply the same fix to `gas_price_max_change_denominator` and `gas_target` in `calculate_next_base_gas_price`, passing the block's Starknet version rather than calling `latest_constants()`.

---

### Proof of Concept

**Current state (no divergence):**

All five version files set `fee_proposal_margin_ppt = 2`. [10](#0-9) 

**Trigger scenario (one-line future change):**

Suppose `orchestrator_versioned_constants_0_14_5.json` sets `"fee_proposal_margin_ppt": 5`.

- A node running the V0_14_5 binary proposes a V0_14_3 block (the network has not fully upgraded). `latest_constants()` returns V0_14_5 constants → margin = 5 ppt. The proposer computes `fee_proposal` at +4 ppt from `fee_actual` and emits it.
- A validator still running the V0_14_4 binary validates the same V0_14_3 block. `latest_constants()` returns V0_14_4 constants → margin = 2 ppt. `fee_proposal_bounds` returns a window that excludes +4 ppt. The validator rejects the proposal with `"Fee proposal out of bounds"`. [11](#0-10) 

- The network splits: upgraded validators accept; non-upgraded validators reject. No BFT quorum is reachable until all nodes converge on the same binary — but the root cause is the version-boundary mismatch, not the upgrade itself.
