Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Validates `owner` Instead of `sender`, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` receives both the actual caller (`sender`, i.e. `msg.sender` of `addLiquidity`) and the caller-supplied position beneficiary (`owner`) but silently discards `sender` and validates only `owner`. Because `owner` is a free parameter with no constraint tying it to `msg.sender`, any unprivileged address can bypass the allowlist by nominating any already-allowlisted address as `owner`, breaking the pool admin's primary mechanism for restricting who may provide liquidity.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as the first argument to the extension hook:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

The extension hook's first parameter is unnamed (discarded), and the guard checks `allowedDepositor[msg.sender][owner]` — where `msg.sender` is the pool address and `owner` is the caller-supplied beneficiary:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

The admin-facing setter names its parameter `depositor`, confirming the intent is to gate the actual depositing actor (the `sender`), not the position beneficiary: [3](#0-2) 

Unlike `removeLiquidity`, which enforces `msg.sender == owner` at line 206, `addLiquidity` imposes no such constraint — `owner` is a completely free parameter: [4](#0-3) 

**Exploit path:**
1. Pool P is deployed with `DepositAllowlistExtension` configured.
2. Admin calls `setAllowedToDeposit(P, Alice, true)`.
3. Bob (not allowlisted) calls `P.addLiquidity(owner=Alice, salt, deltas, callbackData, extensionData)`.
4. Pool calls `E.beforeAddLiquidity(sender=Bob, owner=Alice, ...)`.
5. Extension evaluates `allowedDepositor[P][Alice] == true` → no revert.
6. `LiquidityLib.addLiquidity` executes; Bob's tokens are pulled via callback and credited to Alice's position key.
7. Bob cannot reclaim the tokens because `removeLiquidity` enforces `msg.sender == owner`.

The existing guard is entirely insufficient: it never inspects the actual caller identity.

## Impact Explanation
This is a broken core pool functionality / admin-boundary break. The deposit allowlist is the pool admin's sole mechanism for restricting who may provide liquidity (e.g., KYC-gated or curated market-maker-only pools). The bypass allows any unprivileged address to inject liquidity into restricted pools, disrupting the pool's intended liquidity profile, diluting existing LPs' fee earnings, and potentially shifting `curBinIdx`/`curPosInBin` in ways the pool admin did not sanction. The attacker's tokens are permanently locked in the allowlisted address's position (irrecoverable for the attacker), but the pool's core invariant — a restricted depositor set — is completely broken.

## Likelihood Explanation
No special privilege is required. Any EOA or contract can call `addLiquidity` with `owner` set to any allowlisted address. Allowlisted addresses are publicly readable on-chain via `allowedDepositor(pool, addr)` or `isAllowedToDeposit`. The bypass requires a single transaction with no preconditions and is repeatable indefinitely.

## Recommendation
Replace the `owner` check with a `sender` check to match the stated intent:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external view override returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

## Proof of Concept
```
Setup:
  - Pool P deployed with DepositAllowlistExtension E.
  - Admin: E.setAllowedToDeposit(P, Alice, true).
  - Bob is NOT in the allowlist.

Attack:
  1. Bob calls P.addLiquidity(owner=Alice, salt, deltas, callbackData, extensionData).
  2. Pool calls E.beforeAddLiquidity(sender=Bob, owner=Alice, ...).
  3. Extension: allowedDepositor[P][Alice] == true → passes, no revert.
  4. LiquidityLib.addLiquidity executes; Bob's tokens pulled via callback.
  5. Position minted under Alice's key (owner=Alice, salt).
  6. Bob's tokens locked in Alice's position; Bob cannot remove them
     (removeLiquidity enforces msg.sender == owner).

Result:
  - Bob (not allowlisted) successfully deposited into a restricted pool.
  - The deposit allowlist is completely ineffective against any caller who
    knows at least one allowlisted address.
  - Foundry test: assert that Bob's addLiquidity call succeeds without revert
    and that pool token balances increase by the deposited amounts.
```
