### Title
v8 Migration Bricked by Empty Bonus Pool When Force-Exiting Vesting-Owned Positions — (File: x/tieredrewards/keeper/force_exit.go)

---

### Summary

The v8 upgrade migration (`Migrate1to2`) calls `ForceFullExitWithDelegation` for every vesting-owned tier position. That function calls `claimRewards` → `processEventsAndClaimBonus`, which returns `ErrInsufficientBonusPool` when the rewards pool is empty and the position has accrued non-zero bonus. The error propagates unhandled all the way up through the upgrade handler, causing the chain to panic at the upgrade height. The slash hook in the same codebase already demonstrates the correct pattern — silently forgoing bonus when the pool is insufficient — but `ForceFullExitWithDelegation` omits that guard.

---

### Finding Description

**Root cause — missing `ErrInsufficientBonusPool` guard in `ForceFullExitWithDelegation`:**

`ForceFullExitWithDelegation` calls `claimRewards` unconditionally for every delegated vesting-owned position:

```go
posState, baseRewards, bonusRewards, err := k.claimRewards(ctx, posState)
if err != nil {
    return fmt.Errorf("claim rewards for position %d: %w", posID, err)
}
``` [1](#0-0) 

`claimRewards` calls `processEventsAndClaimBonus`, which calls `sufficientBonusPoolBalance`:

```go
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err
}
``` [2](#0-1) [3](#0-2) 

`sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool` whenever `bonus > 0` and the pool balance is insufficient. That error propagates back through `claimRewards` → `ForceFullExitWithDelegation` → `exitVestedAccountsPositions` → `Migrate` → `Migrate1to2` → the upgrade handler:

```go
if err := pk.ForceFullExitWithDelegation(ctx, posID); err != nil {
    return fmt.Errorf("force-exit position %d: %w", posID, err)
}
``` [4](#0-3) 

```go
m, err := app.ModuleManager.RunMigrations(ctx, app.configurator, fromVM)
if err != nil {
    return map[string]uint64{}, err
}
``` [5](#0-4) 

In Cosmos SDK, an upgrade handler that returns an error causes a panic at the upgrade height, halting the chain.

**Contrast with the slash hook**, which already handles this case correctly to prevent chain halt:

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    // Deliberately forgo bonus rewards if pool is insufficient to prevent chain halt.
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
    } else {
        return err
    }
}
``` [6](#0-5) 

`ForceFullExitWithDelegation` has no equivalent guard.

**Attacker-controlled entry path:**

In v7 (before the v8 upgrade), vesting accounts were permitted to create tier positions. The v8 migration was introduced specifically to force-exit those pre-existing positions. The `validateNonVestingAccount` check that blocks vesting accounts is only enforced in `validateNewPosition`, which is called by `MsgLockTier` and `MsgCommitDelegationToTier` — it was not present in v7. [7](#0-6) 

An unprivileged actor who held a vesting account on v7 and created a tier position (a fully valid v7 transaction) has already placed the triggering state on-chain. The position accrues bonus rewards every block it remains delegated to a bonded validator with a non-zero `BonusApy` tier. The rewards pool is drained every block by the BeginBlocker top-up mechanism; if the pool is empty at upgrade height, the migration fails. [8](#0-7) 

The integration test `_precreate_position_delegator_vesting_acc` confirms that creating a vesting-owned position before the v8 upgrade is a supported and tested scenario: [9](#0-8) 

---

### Impact Explanation

If any vesting-owned tier position has accrued non-zero bonus rewards and the rewards pool is empty at the v8 upgrade height, the upgrade handler panics. The chain halts. Operators must identify the root cause, patch the migration binary to handle `ErrInsufficientBonusPool` gracefully, and coordinate a re-upgrade — a significant operational disruption with potential for FUD and loss of confidence in the upgrade process.

The corrupted invariant is: the upgrade handler assumes `ForceFullExitWithDelegation` always succeeds for delegated positions, but the bonus pool balance is an independent on-chain variable that can be zero at any time.

---

### Likelihood Explanation

- Vesting accounts creating tier positions was valid on v7; the integration test suite explicitly exercises this path.
- The rewards pool is drained every block by the BeginBlocker. If the pool is not actively funded in the blocks immediately before the upgrade, it will be empty.
- An attacker who created a vesting-owned position in v7 and then simply waited for the pool to drain (a passive action) would trigger this failure with no further effort.
- The pool can also be drained faster by any user claiming bonus rewards from their own positions, which is a normal user action.

---

### Recommendation

In `ForceFullExitWithDelegation`, handle `ErrInsufficientBonusPool` the same way the slash hook does — log the event and forgo the bonus rather than propagating the error:

```go
posState, baseRewards, bonusRewards, err := k.claimRewards(ctx, posState)
if err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        logger.Error("force-exit: insufficient bonus pool, forgoing bonus rewards",
            "position_id", posID)
        // re-load posState without bonus checkpoint advancement
        posState, err = k.getPositionState(ctx, posID)
        if err != nil {
            return fmt.Errorf("reload position %d after pool skip: %w", posID, err)
        }
    } else {
        return fmt.Errorf("claim rewards for position %d: %w", posID, err)
    }
}
```

Alternatively, fund the rewards pool as part of the upgrade handler before calling `RunMigrations`, ensuring the pool can cover all pending bonus claims.

---

### Proof of Concept

1. On v7, a vesting account (`PermanentLockedAccount`) calls `MsgLockTier` to create a tier position delegated to a bonded validator on a tier with non-zero `BonusApy`. This succeeds because `validateNonVestingAccount` did not exist in v7.
2. The position accrues bonus rewards over time (proportional to `shares × BonusApy × duration`).
3. The BeginBlocker drains the rewards pool to zero over subsequent blocks (normal operation).
4. The governance upgrade proposal for v8 passes and the upgrade height is reached.
5. `registerV8UpgradeHandler` calls `RunMigrations`, which calls `Migrate1to2`, which calls `exitVestedAccountsPositions`.
6. `exitVestedAccountsPositions` identifies the vesting-owned position and calls `ForceFullExitWithDelegation`.
7. `ForceFullExitWithDelegation` calls `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance`, which returns `ErrInsufficientBonusPool` because `totalBonus > 0` and the pool is empty.
8. The error propagates unhandled through the entire call stack to the upgrade handler, which returns it.
9. The Cosmos SDK upgrade module panics, halting the chain at the upgrade height. [10](#0-9) [11](#0-10) [3](#0-2) [12](#0-11)

func (k Keeper) ForceFullExitWithDelegation(ctx context.Context, posID uint64) error {
	logger := k.logger(ctx)
	logger.Info("force-exit: begin", "position_id", posID)

	posState, err := k.getPositionState(ctx, posID)
	if err != nil {
		return fmt.Errorf("get position %d: %w", posID, err)
	}
	if !posState.IsDelegated() {
		logger.Error("force-exit: position is not delegated; cannot force full exit",
			"position_id", posID,
			"owner", posState.Owner,
		)
		return nil
	}
	logger.Info("force-exit: position state loaded",
		"position_id", posID,
		"owner", posState.Owner,
		"tier_id", posState.TierId,
		"validator", posState.Delegation.ValidatorAddress,
		"shares", posState.Delegation.Shares.String(),
	)

	posState, baseRewards, bonusRewards, err := k.claimRewards(ctx, posState)
	if err != nil {
		return fmt.Errorf("claim rewards for position %d: %w", posID, err)
	}
	logger.Info("force-exit: claimed rewards",
		"position_id", posID,
		"base_rewards", baseRewards.String(),
		"bonus_rewards", bonusRewards.String(),
	)

	valAddr, err := sdk.ValAddressFromBech32(posState.Delegation.ValidatorAddress)
	if err != nil {
		return fmt.Errorf("parse validator address for position %d: %w", posID, err)
	}

	positionAmount, err := k.reconcileAmountFromShares(ctx, valAddr, posState.Delegation.Shares)
	if err != nil {
		return fmt.Errorf("reconcile amount for position %d: %w", posID, err)
	}
	logger.Info("force-exit: reconciled position amount",
		"position_id", posID,
		"amount", positionAmount.String(),
		"validator", valAddr.String(),
	)

	if _, _, _, err := k.transferDelegationFromPosition(ctx, posState, valAddr, positionAmount); err != nil {
		return fmt.Errorf("transfer delegation back to owner for position %d: %w", posID, err)
	}
	logger.Info("force-exit: delegation transferred back to owner",
		"position_id", posID,
		"owner", posState.Owner,
		"amount", positionAmount.String(),
	)

	ownerAddr, err := sdk.AccAddressFromBech32(posState.Owner)
	if err != nil {
		return fmt.Errorf("parse owner address for position %d: %w", posID, err)
	}
	if err := k.alignVestingDelegationTracking(ctx, ownerAddr); err != nil {
		return fmt.Errorf("align vesting delegation tracking for position %d: %w", posID, err)
	}

	if err := k.deletePosition(ctx, posState.Position, &ValidatorTransition{PreviousAddress: valAddr.String()}); err != nil {
		return fmt.Errorf("delete position %d: %w", posID, err)
	}
	logger.Info("force-exit: position deleted",
		"position_id", posID,
		"owner", posState.Owner,
	)

	logger.Info("force-exit: complete",
		"position_id", posID,
		"owner", posState.Owner,
		"amount", positionAmount.String(),
		"validator", valAddr.String(),
	)
	return nil

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil

func exitVestedAccountsPositions(
	ctx context.Context,
	positions collections.Map[uint64, types.Position],
	ak AccountKeeper,
	pk PositionForceExiter,
) error {
	sdkCtx := sdk.UnwrapSDKContext(ctx)

	var toExit []uint64
	if err := positions.Walk(ctx, nil, func(posID uint64, pos types.Position) (bool, error) {
		ownerAddr, err := sdk.AccAddressFromBech32(pos.Owner)
		if err != nil {
			return false, fmt.Errorf("parse owner address for position %d: %w", posID, err)
		}
		acc := ak.GetAccount(ctx, ownerAddr)
		if acc == nil {
			return false, fmt.Errorf("owner account not found for position %d: %s", posID, ownerAddr.String())
		}
		if _, ok := acc.(sdkvesting.VestingAccount); ok {
			toExit = append(toExit, posID)
		}
		return false, nil
	}); err != nil {
		return fmt.Errorf("walk positions: %w", err)
	}

	for _, posID := range toExit {
		sdkCtx.Logger().Info("v8 migration: force-exit vesting-owned position", "position_id", posID)
		if err := pk.ForceFullExitWithDelegation(ctx, posID); err != nil {
			return fmt.Errorf("force-exit position %d: %w", posID, err)
		}
	}

	sdkCtx.Logger().Info("v8 migration: vesting-owned positions exited", "count", len(toExit))
	return nil
}

def _precreate_position_delegator_vesting_acc(cluster):
    """Set up the position-delegator-precreate scenario before the v8
    upgrade fires.

    On v7.2.0, the position delegator address is deterministic from the
    position id — `tieredrewards/position/<id>` module address. An
    attacker can pre-create a `PermanentLockedAccount` at that address
    """
    val_addr = get_node_validator_addr(cluster)
    tiers = query_tiers(cluster).get("tiers", [])
    assert tiers, "expected at least one tier seeded by the v7 upgrade handler"
    tier_id = int(tiers[0]["id"])
    lock_amount = int(tiers[0]["min_lock_amount"])

    # Predict next position id; pre-create a PermanentLockedAccount at
    # the v1 delegator address that LockTier will pick.
    next_pos_id = _next_position_id(cluster)
    predicted_del_addr = position_delegator_address(next_pos_id)
    dust = 1
    cli = cluster.cosmos_cli()
    rsp = cli.tx(
        "vesting",
        "create-permanent-locked-account",
        predicted_del_addr,
        f"{dust}{DENOM}",
        from_=cluster.address("signer1"),
        chain_id=cli.chain_id,
    )
    assert (
        rsp["code"] == 0
    ), f"front-run vesting-account create failed: {rsp.get('raw_log', rsp)}"
    wait_for_new_blocks(cluster, 1)

    # Create a fresh base-account owner and fund it with principal + gas.
    name = "precreate-victim"
    owner_addr = create_new_address(cluster, name)

    rsp = cluster.transfer(
        cluster.address("signer1"),
        owner_addr,
        f"{lock_amount + GAS_ALLOWANCE}{DENOM}",
    )
    assert rsp["code"] == 0, f"owner top-up failed: {rsp.get('raw_log', rsp)}"
    wait_for_new_blocks(cluster, 1)

    # LockTier — picks up `predicted_del_addr` because the v7.2.0
    # derivation is deterministic from the position id alone.
    rsp = lock_tier(cluster, owner_addr, tier_id, lock_amount, val_addr)
    assert (
        rsp["code"] == 0
    ), f"lock-tier on pre-poisoned address failed: {rsp.get('raw_log', rsp)}"

    positions = query_positions_by_owner(cluster, owner_addr).get("positions", [])
    assert len(positions) == 1, f"expected 1 precreate-victim position, got {positions}"
    actual_pos_id = int(positions[0]["id"])
    assert actual_pos_id == next_pos_id, (
        f"expected position id {next_pos_id} (the front-run target), "
        f"got {actual_pos_id} — front-run prediction stale"
    )

    return {
        "precreate_owner_addr": owner_addr,
        "precreate_val_addr": val_addr,
        "precreate_pos_id": next_pos_id,
        "precreate_predicted_del_addr": predicted_del_addr,
        "precreate_lock_amount": lock_amount,
        "precreate_dust": dust,
    }
