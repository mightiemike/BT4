### Title
Unattributable Panic via Unchecked Deserialization of Attacker-Controlled `coinbase_tx` in AuxPoW Validation — (`btc-types/src/aux.rs`)

---

### Summary

In the Dogecoin build, `AuxData::get_coinbase_tx()` calls `deserialize(&self.coinbase_tx).unwrap()` on a raw byte buffer that is entirely supplied by an unprivileged NEAR caller via `submit_blocks`. If the bytes are malformed, the `.unwrap()` produces a generic Rust panic that escapes the contract's structured error-handling path, making the failing submitter unidentifiable from the revert message alone — a direct analog to the `validatePaymasterUserOp` try-catch decoding bypass.

---

### Finding Description

`AuxData` is the attacker-controlled struct submitted as part of the `(Header, Option<AuxData>)` argument to `submit_blocks` on the Dogecoin build. [1](#0-0) 

The `coinbase_tx` field is a raw `Vec<u8>` with no pre-validation. Inside `check_aux`, the contract calls: [2](#0-1) 

This is invoked unconditionally at line 84 of `dogecoin.rs`: [3](#0-2) 

If `coinbase_tx` contains bytes that do not deserialize as a valid Bitcoin `Transaction`, `deserialize(...)` returns `Err(...)` and `.unwrap()` panics with a generic Rust message (e.g., `"called Result::unwrap() on an Err value: ..."`). This panic:

1. **Bypasses every `require!` guard** in `check_aux` that would otherwise emit a structured, attributable error string identifying the invalid field.
2. **Produces no structured error** that a relayer or monitoring system can use to identify which submitter provided the malformed data.

Two additional unguarded `.unwrap()` calls in the same function compound the issue:

- `coinbase_tx.input.first().unwrap()` at line 98 — panics generically if the deserialized transaction has zero inputs.
- `hex::decode(...).unwrap()` at lines 130 and 137 — panics generically if the script_sig hex slice is out of bounds or malformed. [4](#0-3) [5](#0-4) [6](#0-5) 

The entry point is the public, payable `submit_blocks`: [7](#0-6) 

which calls `submit_block_header` and, for Dogecoin, `check_aux`: [8](#0-7) 

---

### Impact Explanation

The contract's structured error-attribution mechanism is bypassed. All legitimate validation failures in `check_aux` use `require!(condition, "descriptive message")` or `env::panic_str("descriptive message")`, which emit a named, attributable error. The `.unwrap()` panics instead emit a generic Rust panic string that carries no contract-level context. A relayer or downstream consumer of `submit_blocks` cannot distinguish a maliciously crafted `coinbase_tx` from a legitimate validation failure, and cannot identify the submitter from the error alone. This is the exact analog of the `FailedOpWithRevert` bypass described in the external report.

---

### Likelihood Explanation

`submit_blocks` is a public, payable NEAR entry point requiring no privileged role (beyond the `trusted_relayer` check, which can be bypassed if the relayer stake requirement is met or if the contract is configured without it). The `coinbase_tx` field is a raw `Vec<u8>` — providing invalid bytes requires zero cryptographic knowledge. Any unprivileged NEAR account that can call `submit_blocks` with a Dogecoin AuxPoW header can trigger this.

---

### Recommendation

Replace `.unwrap()` with `env::panic_str` carrying a structured, attributable message:

```rust
// btc-types/src/aux.rs
pub fn get_coinbase_tx(&self) -> Transaction {
    deserialize(&self.coinbase_tx)
        .unwrap_or_else(|e| near_sdk::env::panic_str(
            &format!("AuxPoW: invalid coinbase_tx bytes: {e}")
        ))
}
```

Apply the same pattern to `coinbase_tx.input.first()` and both `hex::decode(...)` calls in `dogecoin.rs`, replacing `.unwrap()` with `unwrap_or_else(|_| env::panic_str("AuxPoW: <specific field> is invalid"))`. This ensures every failure path emits a structured, attributable error consistent with the rest of the contract's error-handling convention.

---

### Proof of Concept

1. Deploy the contract with the `dogecoin` feature enabled.
2. Call `submit_blocks` with a valid `Header` (AuxPoW flag set, correct chain ID, correct `bits`) and an `AuxData` where `coinbase_tx` is set to any byte sequence that is not a valid Bitcoin transaction serialization, e.g., `[0x00]`.
3. Execution reaches `check_aux` → `get_coinbase_tx()` → `deserialize(&[0x00]).unwrap()` → Rust panic with message `"called Result::unwrap() on an Err value: ..."`.
4. The NEAR transaction receipt shows a generic panic with no structured contract error, no mention of `coinbase_tx`, and no identification of the submitting account as the source of the malformed data.
5. A relayer monitoring the chain cannot distinguish this from any other contract panic (e.g., a storage miss or arithmetic overflow), making the malicious submitter unattributable from the error alone.
