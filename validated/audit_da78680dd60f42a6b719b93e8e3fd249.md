### Title
Difficulty Retarget Check Uses Mainchain Ancestor Instead of Fork Ancestor, Enabling Incorrect Difficulty Acceptance for Fork Blocks - (File: contract/src/dogecoin.rs)

### Summary

The `get_next_work_required` function in all chain implementations resolves the retarget-boundary block via `get_header_by_height`, which reads exclusively from `mainchain_height_to_header`. When a fork block is being validated, the correct retarget-boundary block is the fork's own ancestor at that height, not the mainchain block at the same height. For Dogecoin (where every block is a retarget), this desynchronization is present for every fork block deeper than one block. The developer-acknowledged TODO comment at line 291 of `dogecoin.rs` confirms the uncertainty.

### Finding Description

When `submit_block_header` processes a fork block, `get_prev_header` correctly retrieves the fork's previous block from `headers_pool`. However, `check_pow` → `get_next_work_required` then calls `blocks_getter.get_header_by_height(height_first)` to obtain the retarget-boundary block's timestamp: [1](#0-0) 

`get_header_by_height` is implemented as: [2](#0-1) 

It reads only from `mainchain_height_to_header`, which contains only mainchain blocks. Fork blocks are stored solely in `headers_pool` and are never inserted into `mainchain_height_to_header`: [3](#0-2) 

For Dogecoin mainnet, `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` for any non-genesis block: [4](#0-3) 

This means `height_first = prev_block_header.block_height - 1`. For any fork that diverged at or before that height, the fork's ancestor at `height_first` is a different block than the mainchain block at `height_first`. The difficulty calculation:

```
modulated_timespan = prev_block_header.time - first_block_time
```

uses `first_block_time` from the **mainchain** block at `height_first`, not the fork's actual ancestor. This produces an incorrect `expected_bits`.

The same structural flaw exists in `bitcoin.rs` and `litecoin.rs` for forks that diverge more than `difficulty_adjustment_interval - 1` blocks deep: [5](#0-4) [6](#0-5) 

### Impact Explanation

The difficulty check enforces: [7](#0-6) 

If `expected_bits` is computed from the wrong (mainchain) retarget-boundary timestamp, two outcomes are possible:

1. **Invalid fork blocks accepted**: If the mainchain's retarget block has a later timestamp than the fork's ancestor, the computed difficulty is lower (easier target). An attacker submits a fork block whose `bits` field matches this easier expected value, passes the `expected_bits == block_header.bits` check, and finds a PoW solution against the easier target. Accumulating enough such blocks can produce a fork with chainwork exceeding the mainchain, triggering `reorg_chain` and corrupting the canonical chain mapping in `mainchain_height_to_header` and `mainchain_tip_blockhash`.

2. **Valid fork blocks rejected**: If the mainchain's retarget block has an earlier timestamp, the computed difficulty is higher than the fork's actual required difficulty. Legitimate fork blocks from the real Dogecoin network are rejected, preventing the contract from tracking the correct chain during a reorganization.

### Likelihood Explanation

For Dogecoin with `difficulty_adjustment_interval = 1`, the flaw is triggered for every fork block where the fork diverged more than one block ago — a routine occurrence in any multi-block fork. The attacker entry path is `submit_blocks`, which is gated by the `#[trusted_relayer]` staking mechanism: [8](#0-7) 

Any NEAR account can stake to become a trusted relayer, making this reachable by an unprivileged actor willing to stake. Likelihood: **Medium**.

### Recommendation

Replace `get_header_by_height` (mainchain-only lookup) with a traversal that walks the fork chain backward from `prev_block_header` to find the actual ancestor at `height_first`. Store the ancestor's hash in the `ExtendedHeader` or traverse via `get_prev_header` iteratively. For Bitcoin and Litecoin, the same fix applies to `get_next_work_required` in `bitcoin.rs` and `litecoin.rs`.

### Proof of Concept

1. Contract is initialized with Dogecoin mainnet config (`difficulty_adjustment_interval = 1`).
2. Mainchain has blocks at heights 100, 101, 102 with timestamps T100, T101, T102.
3. Attacker submits a fork block at height 101 (diverging from height 100), with timestamp T101_fork ≠ T101.
4. Attacker submits a fork block at height 102 whose `prev` is the fork block at 101.
5. `get_next_work_required` computes `height_first = 101 - 1 = 100`. `get_header_by_height(100)` returns the **mainchain** block at height 100 with timestamp T100.
6. `modulated_timespan = T101_fork - T100` (uses mainchain T100, not fork ancestor's time).
7. The resulting `expected_bits` differs from what the fork's actual difficulty should be.
8. Attacker crafts a block whose `bits` field matches this incorrect `expected_bits` and finds a PoW solution against the easier target.
9. The block passes `check_pow` and is stored via `store_fork_header`. If chainwork exceeds mainchain, `reorg_chain` is called, corrupting the canonical chain.
