### Title
Stale Mainchain Tip Accepted Without Recency Validation in `verify_transaction_inclusion` — (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` return SPV proof results based on the stored `mainchain_tip_blockhash` without any check that the tip's block timestamp is recent. If the relayer stops submitting headers, the contract's chain view freezes silently, and any unprivileged NEAR caller can still obtain a `true` inclusion proof against a stale, potentially reorganized chain state.

### Finding Description

The `BtcLightClient` struct stores the current chain tip as `mainchain_tip_blockhash` and the associated `ExtendedHeader` (which carries `block_header.time`, a Bitcoin-protocol Unix timestamp). Neither the struct nor any public method records the NEAR block timestamp at which `submit_blocks` was last called. [1](#0-0) 

`verify_transaction_inclusion` reads the tip to compute the confirmation depth, then verifies the Merkle proof. It performs no check that `heaviest_block_header.block_header.time` (or any NEAR-side timestamp) is within an acceptable window of the current wall-clock time: [2](#0-1) 

The confirmation check at lines 304–308 only measures the height difference between the stored tip and the target block — it says nothing about whether the stored tip is current: [3](#0-2) 

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after the coinbase proof check, so it inherits the same gap: [4](#0-3) 

The chain-specific `check_pow` implementations do validate that an *incoming* header's timestamp is not too far in the past or future relative to the NEAR block clock: [5](#0-4) 

But this guard runs only during `submit_blocks`. Once a header is stored, its timestamp is never re-checked against the current time when proof queries are served.

### Impact Explanation

A consumer contract (e.g., a bridge or escrow) that calls `verify_transaction_inclusion` to gate fund release relies on the contract's chain view being current. If the relayer has been offline for hours or days:

1. The stored tip is frozen at some past Bitcoin height `H`.
2. A Bitcoin reorg that occurred after height `H` is invisible to the contract.
3. A transaction that was in the chain at height `H − N` but was later reorganized out will still produce a `true` result with `N` confirmations.
4. The consumer contract releases funds or records a state transition based on a proof that is no longer valid on the live Bitcoin chain.

The `get_last_block_header` view function exposes the stale tip to any caller without any freshness signal: [6](#0-5) 

### Likelihood Explanation

The relayer is a single off-chain process. Its liveness is not enforced by the contract. Any of the following realistic events leave the contract silently stale: relayer crash, network partition between the relayer and the Bitcoin node, operator key rotation failure, or a targeted DoS against the relayer's Bitcoin RPC endpoint. On chains with shorter block times and higher reorg rates (Dogecoin, Litecoin), the window during which a stale proof is exploitable is wider. The attack requires no privileged access — any NEAR account can call `verify_transaction_inclusion`.

### Recommendation

1. **Record the NEAR submission timestamp.** Add a `last_submission_timestamp_ms: u64` field to `BtcLightClient`, updated in `submit_blocks` with `env::block_timestamp_ms()`.
2. **Add a recency guard in `verify_transaction_inclusion`.** Before computing confirmations, assert that `env::block_timestamp_ms() - self.last_submission_timestamp_ms` is below a configurable `max_staleness_ms` threshold (e.g., 2 hours for Bitcoin, shorter for Dogecoin/Litecoin).
3. **Expose a `is_synced()` view.** Let consumer contracts query freshness independently before calling the proof API.

This mirrors the `_assertMinIntervalBetweenUpdatesPassed` pattern referenced in the original report.

### Proof of Concept

```
1. Deploy the contract and submit headers up to Bitcoin height 900 000.
2. Stop the relayer. Bitcoin continues to produce blocks; a reorg occurs at height 899 990,
   removing a transaction T that was previously confirmed at height 899 985.
3. After the relayer has been offline for 24 hours, call on NEAR:
     verify_transaction_inclusion({
       tx_id: T,
       tx_block_blockhash: <hash of block 899985>,
       tx_index: ...,
       merkle_proof: [...],
       confirmations: 6
     })
4. The contract reads mainchain_tip_blockhash → height 900 000, computes
   900 000 − 899 985 + 1 = 16 ≥ 6, verifies the Merkle proof against the
   stored (pre-reorg) merkle_root, and returns `true`.
5. The consumer contract releases funds for a transaction that no longer
   exists on the live Bitcoin chain.
``` [7](#0-6)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]

    }
