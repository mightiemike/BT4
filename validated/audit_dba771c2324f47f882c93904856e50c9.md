Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Originating User, Enabling Allowlist Bypass or Blocking Allowlisted Users — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of the `pool.swap()` call. When swaps are routed through `MetricOmmSimpleRouter`, `sender` resolves to the router's address rather than the originating user. This produces two distinct failures: if the router is allowlisted (a natural admin action to support router-based swaps), every unprivileged user bypasses the per-user allowlist entirely; if the router is not allowlisted, individually allowlisted users cannot use the router at all.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates the allowlist keyed on that `sender`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router is `msg.sender` to the pool, so `sender = router`: [4](#0-3) 

The same substitution occurs in `exactInput` (intermediate hops use `address(this)` as payer): [5](#0-4) 

And in `exactOutput` recursive callback hops, which call `pool.swap` from inside `_exactOutputIterateCallback`: [6](#0-5) 

`DepositAllowlistExtension` does not share this flaw because it keys on `owner` (the position owner explicitly passed by the caller), which correctly identifies the economic actor regardless of who the immediate caller is: [7](#0-6) 

No existing guard in `SwapAllowlistExtension` or `ExtensionCalling` resolves the originating user; the `sender` value is accepted as-is from the pool's `msg.sender`.

## Impact Explanation
**Impact A — Allowlist bypass (High).** A pool admin who allowlists the router (`setAllowedToSwap(pool, router, true)`) to support router-based swaps for their curated pool causes `allowedSwapper[pool][router]` to be `true` for every call arriving through the router, regardless of who the originating user is. Any address — including addresses the admin explicitly excluded — can trade on the curated pool by routing through `MetricOmmSimpleRouter`. The curated pool's access control is silently voided, constituting a broken core pool functionality and admin-boundary break reachable by any unprivileged user.

**Impact B — Broken core functionality (Medium).** If the admin does not allowlist the router, individually allowlisted users cannot use `MetricOmmSimpleRouter` at all: the extension sees the router as `sender`, finds it absent from the allowlist, and reverts with `NotAllowedToSwap`. The primary supported swap entrypoint is unusable for every curated pool with `SwapAllowlistExtension` active, regardless of the user's individual allowlist status.

## Likelihood Explanation
Both paths are reachable by normal, unprivileged user actions with no malicious setup required. `SwapAllowlistExtension` is a production periphery contract in scope. `MetricOmmSimpleRouter` is the primary public swap entrypoint. Any pool admin who deploys a curated pool and wants to support router-based swaps will encounter Impact A the moment they allowlist the router. Any pool admin who does not allowlist the router will encounter Impact B immediately upon the first allowlisted user attempting a router swap. No special privileges, flash loans, or off-chain manipulation are required.

## Recommendation
The extension must resolve the originating user rather than the immediate pool caller. The preferred approach is to have the router encode `msg.sender` into `extensionData` before calling the pool, and have the extension decode and check that address when `msg.sender` (the pool) is a factory-registered pool. Alternatively, the extension can accept a factory-verified trusted-router registry and decode the user identity from `extensionData` only when the immediate `sender` is a registered router. At minimum, `SwapAllowlistExtension` documentation must warn that allowlisting the router opens the pool to all users, and the test suite must cover the router path explicitly.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension as beforeSwap hook
  allowedSwapper[pool][alice] = true   // alice is the only intended swapper
  allowedSwapper[pool][router] = true  // admin allowlists router to support router-based swaps for alice

Attack:
  charlie (not allowlisted) calls:
    router.exactInputSingle({pool: pool, ..., extensionData: ""})

  Execution trace:
    router.exactInputSingle()
      → pool.swap(recipient, ...) [msg.sender = router]
        → _beforeSwap(sender=router, ...)
          → SwapAllowlistExtension.beforeSwap(sender=router, ...)
            → allowedSwapper[pool][router] == true  ✓  (no revert)
        → swap executes, charlie receives tokens

Result:
  charlie swaps successfully on a pool restricted to alice only.
  The allowlist is bypassed without any privileged action by charlie.
```

A Foundry integration test can confirm this by: (1) deploying a pool with `SwapAllowlistExtension`, (2) calling `setAllowedToSwap(pool, router, true)` as pool admin, (3) calling `router.exactInputSingle` from an address not in the allowlist, and asserting the swap succeeds rather than reverting with `NotAllowedToSwap`.
