Audit Report

## Title
SwapAllowlistExtension Bypassed via Router: Any Unprivileged User Can Swap on Allowlisted Pools — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender` — the immediate caller of `pool.swap()`. When `MetricOmmSimpleRouter` mediates a swap, the router is `msg.sender`, so the extension checks whether the **router** is allowlisted, not the end user. Any pool admin who allowlists the router (required for router-mediated swaps to succeed) simultaneously opens the pool to every unprivileged user who calls through the router.

## Finding Description
**Root cause — pool passes `msg.sender` as `sender`:**
In `MetricOmmPool.swap`, `_beforeSwap` is invoked with `msg.sender` as the first argument:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // always the immediate caller of pool.swap()
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this value verbatim to every configured extension via `_callExtensionsInOrder`:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

**Extension checks the wrong identity:**
`SwapAllowlistExtension.beforeSwap` receives `sender` (the router) and checks it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) calls the pool, `sender = router`.

**Router calls pool directly:**
```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
IMetricOmmPoolActions(params.pool).swap(
    params.recipient, params.zeroForOne, ..., params.extensionData
);
```

**Exploit path:**
1. Admin deploys pool with `SwapAllowlistExtension` as `BEFORE_SWAP` extension.
2. Admin calls `setAllowedToSwap(pool, Alice, true)` — Alice is the intended user.
3. Admin calls `setAllowedToSwap(pool, router, true)` — required so Alice can use the router.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(...)` → `msg.sender = router`.
6. Pool calls `_beforeSwap(sender=router, ...)`.
7. Extension checks `allowedSwapper[pool][router]` → `true` → passes.
8. Bob's swap executes. `allowedSwapper[pool][Bob] == false` is violated.

No existing guard recovers the original EOA: `_callExtensionsInOrder` passes `sender` as-is, and the extension has no mechanism to distinguish a router-mediated call from a direct call.

## Impact Explanation
A pool admin who deploys `SwapAllowlistExtension` to restrict swaps to a curated set (e.g., KYC'd counterparties) and also enables router access creates the bypass automatically. Any unprivileged address can trade on the pool by routing through `MetricOmmSimpleRouter`, defeating the per-user access control entirely. LP principal is at risk because the pool's liquidity was sized and priced for a controlled counterparty set; adversarial or retail traders can front-run institutional flow, drain LP inventory at oracle mid, or trade in ways the pool admin explicitly prohibited. This constitutes a broken core pool access-control invariant with direct risk to LP principal — an admin-boundary break reachable by any unprivileged caller.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical periphery entry point; most users interact with pools through it. Any pool admin who enables both a swap allowlist and router access — a natural and expected combination — creates the bypass automatically. No special privilege, flash loan, or oracle manipulation is required: a plain router call suffices. The bypass is silent; no event distinguishes a router-mediated swap from a direct one. The precondition (router allowlisted) is a necessary operational step, not an exotic misconfiguration.

## Recommendation
The extension must gate the **economic actor**, not the immediate caller. Viable approaches:

1. **Propagate the originating user via `extensionData`**: The router encodes `msg.sender` (the end user) into `extensionData` before forwarding to the pool. `SwapAllowlistExtension` decodes and checks that address. Requires a trusted-router convention.
2. **Router registry + user verification inside the extension**: The extension recognises known routers and, for router calls, requires the user address to be supplied and verified (e.g., via a signed payload or transient-storage forwarding pattern).
3. **Operational policy**: Document and enforce that the router must never be allowlisted on per-user-gated pools; use `allowAllSwappers = true` as a separate pool type for open-access pools.
