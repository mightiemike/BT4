### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables Draining DDA usdcE Without Providing USDC ŌĆö (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` is a permissionless function that is supposed to atomically swap USDC-e held in a Direct Deposit Address (DDA) for USDC provided by the caller. The USDC inbound leg uses a raw `.transferFrom()` call whose return value is completely ignored. If the USDC token returns `false` on failure rather than reverting, the function continues and transfers the DDA's entire usdcE balance to the caller for free.

---

### Finding Description

`replaceUsdcEWithUsdc` is declared `external` with no access-control modifier. Its only guard is a chain-id check (`block.chainid == 57073`, Ink mainnet).

```solidity
// ContractOwner.sol L608-L620
function replaceUsdcEWithUsdc(bytes32 subaccount) external {
    require(block.chainid == 57073, ERR_UNAUTHORIZED);
    address payable directDepositV1 = directDepositV1Address[subaccount];
    require(directDepositV1 != address(0), "no dda");
    address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
    address usdc  = 0x2D270e6886d130D724215A266106e6832161EAEd;
    uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
    if (balance > 0) {
        IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // ŌåÉ raw call, return value discarded
        DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
        IERC20Base(usdcE).safeTransfer(msg.sender, balance);                 // ŌåÉ safe call
    }
}
``` [1](#0-0) 

The three-step sequence is:
1. Pull `balance` USDC from `msg.sender` into the DDA ŌĆö **raw `.transferFrom()`, return value silently discarded**.
2. Withdraw all usdcE from the DDA to `ContractOwner` (`DirectDepositV1.withdraw` calls `safeTransfer` internally).
3. Forward that usdcE to `msg.sender` via `safeTransfer`.

If step 1 returns `false` (token signals failure without reverting), steps 2 and 3 still execute. The caller receives the DDA's full usdcE balance while having transferred zero USDC.

The project already has `ERC20Helper.safeTransferFrom`, which performs a low-level call and explicitly checks `success && (data.length == 0 || abi.decode(data, (bool)))`, handling both no-return-value tokens and false-returning tokens. [2](#0-1) 

`ContractOwner` imports and uses this library (`using ERC20Helper for IERC20Base`) and correctly uses `safeTransfer` on line 618 and `safeTransfer` again in `withdrawFromDirectDepositV1` on line 642, but inconsistently falls back to the raw interface call for the inbound USDC leg. [3](#0-2) 

---

### Impact Explanation

An attacker who calls `replaceUsdcEWithUsdc(subaccount)` for any DDA that holds a non-zero usdcE balance, while having zero USDC allowance granted to `ContractOwner`, will:

- Receive the DDA's entire usdcE balance.
- Pay nothing (the USDC `transferFrom` returns `false` and is ignored).

The corrupted asset delta is the full usdcE balance of the targeted DDA. The DDA is left with zero usdcE and zero USDC (the USDC transfer never actually occurred). Any user whose funds were sitting in that DDA pending `creditDeposit` loses their usdcE collateral.

---

### Likelihood Explanation

The function is permissionless on Ink mainnet (chain 57073). The USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink is a bridged/wrapped USDC. If that implementation follows the USDT pattern (no return value) or returns `false` on insufficient allowance rather than reverting, the exploit is directly triggerable by any caller with zero USDC allowance. Even if the current deployment reverts on failure, the unchecked pattern is a latent bug that activates if the token is ever upgraded or replaced with one that returns `false`.

---

### Recommendation

Replace the raw `.transferFrom()` call at line 616 with the project's own `safeTransferFrom` wrapper from `ERC20Helper`:

```solidity
// Before (vulnerable)
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe)
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [4](#0-3) 

This is consistent with how every other outbound transfer in `ContractOwner` and `EndpointStorage` is handled. [5](#0-4) 

---

### Proof of Concept

1. Identify a `subaccount` whose DDA (`directDepositV1Address[subaccount]`) holds a non-zero usdcE balance (e.g., a user who sent usdcE to their DDA but has not yet called `creditDeposit`).
2. From an EOA with **zero** USDC allowance granted to `ContractOwner`, call:
   ```solidity
   contractOwner.replaceUsdcEWithUsdc(subaccount);
   ```
3. If USDC's `transferFrom` returns `false` instead of reverting:
   - Line 616 executes and returns `false` ŌĆö silently ignored.
   - Line 617 withdraws all usdcE from the DDA to `ContractOwner`.
   - Line 618 transfers all usdcE to `msg.sender`.
4. The attacker holds the victim's usdcE; the DDA holds neither usdcE nor USDC. [6](#0-5)

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
