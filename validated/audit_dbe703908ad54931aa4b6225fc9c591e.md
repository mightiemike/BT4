### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable Without Coinbase Proof Guard, Enabling 64-Byte Transaction Forgery — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` adds a coinbase merkle proof check specifically to block the 64-byte transaction forgery attack. The original `verify_transaction_inclusion` (v1) is still a fully public, on-chain-callable function that lacks this guard entirely. Any unprivileged NEAR caller can invoke v1 directly, bypassing the coinbase proof validation and forging SPV inclusion proofs for non-existent Bitcoin transactions.

---

### Finding Description

`verify_transaction_inclusion_v2` (the "safe" path) validates the coinbase merkle proof before proceeding: [1](#0-0) 

Specifically, it requires:
1. `merkle_proof.len() == coinbase_merkle_proof.len()` [2](#0-1) 
2. The coinbase proof must reconstruct the block's `merkle_root` [3](#0-2) 

`verify_transaction_inclusion` (v1, the "unsafe" path) has neither of these checks. It only verifies that the supplied `tx_id` and `merkle_proof` reconstruct the block's `merkle_root`: [4](#0-3) 

The `#[deprecated]` Rust attribute is a **compiler-only hint**. It generates a warning at compile time for Rust callers but does **not** remove the function from the deployed WASM contract. The function remains a public, `#[pause]`-gated method callable by any NEAR account: [5](#0-4) 

The contract's own documentation acknowledges the vulnerability class explicitly in the v1 docstring: [6](#0-5) 

---

### Impact Explanation

The 64-byte transaction forgery attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions) exploits the fact that Bitcoin's merkle tree does not distinguish between leaf nodes (real transactions, 32-byte hashes) and internal nodes (also 32-byte hashes). A 64-byte blob can be crafted so that its double-SHA256 hash equals an internal merkle tree node, allowing an attacker to construct a valid-looking merkle proof for a transaction that does not exist in the block.

Without the coinbase proof guard, `verify_transaction_inclusion` will return `true` for such a forged proof. Any NEAR smart contract that calls v1 to gate a high-value action (e.g., a Bitcoin bridge releasing funds, a payment verification system) will accept the fraudulent proof as valid, enabling unauthorized fund releases or protocol-level failures.

The corrupted invariant is the SPV proof result: `verify_transaction_inclusion` returns `true` for a `tx_id` that is not a real transaction in the claimed block.

---

### Likelihood Explanation

- The function is publicly callable by any unprivileged NEAR account with no role restriction beyond the global pause.
- The 64-byte transaction forgery is a well-documented, known attack with published tooling.
- Any downstream NEAR contract that integrated against v1 before v2 was introduced, or that calls v1 by mistake, is immediately exploitable without any privileged access.
- The attacker needs only to identify a suitable block and craft a `ProofArgs` struct — no key compromise, social engineering, or privileged role is required.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public contract interface entirely, or gate it with an access-control role so it cannot be called by unprivileged accounts. The `#[deprecated]` attribute alone provides no on-chain protection. Alternatively, inline the coinbase proof validation directly into v1 so both paths are equally safe, and remove the delegation from v2 to v1.

---

### Proof of Concept

1. Attacker selects a Bitcoin block `B` whose merkle tree has at least two transactions, so internal nodes exist.
2. Attacker identifies an internal merkle node `N` (32 bytes) such that the 64-byte concatenation `N || N` (or a crafted sibling pair) double-SHA256-hashes to a value that can be walked up to the block's `merkle_root` with a valid proof path.
3. Attacker constructs `ProofArgs { tx_id: N, tx_block_blockhash: B.hash, tx_index: <crafted>, merkle_proof: <crafted path>, confirmations: 1 }`.
4. Attacker calls `verify_transaction_inclusion(args)` directly on the NEAR contract (not v2).
5. The function skips the coinbase guard, computes `compute_root_from_merkle_proof(N, index, proof)`, finds it equals `B.merkle_root`, and returns `true`.
6. The downstream NEAR contract receives `true` and releases funds or records the fraudulent transaction as confirmed. [7](#0-6)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

            "Incorrect coinbase merkle proof"
