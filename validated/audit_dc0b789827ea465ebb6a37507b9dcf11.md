### Title
No Minimum Confirmation Threshold Enforced — Caller-Supplied `confirmations = 0` or `1` Bypasses Bitcoin Finality — (`contract/src/lib.rs`)

---

### Summary

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations` field with no enforced lower bound. Any unprivileged NEAR caller can pass `confirmations = 0` or `confirmations = 1`, causing the contract to certify a Bitcoin transaction as "confirmed" with essentially no finality, making the result trivially invalidated by a Bitcoin chain re-org.

---

### Finding Description

`ProofArgs.confirmations` and `ProofArgsV2.confirmations` are fully caller-controlled fields. [1](#0-0) 

Inside `verify_transaction_inclusion`, the contract enforces only an **upper** bound — that `confirmations` does not exceed `gc_threshold` — and a **sufficiency** check — that the current chain tip is at least `confirmations` blocks ahead of the target block: [2](#0-1) 

There is no lower-bound check. Passing `confirmations = 0` or `confirmations = 1` satisfies both guards trivially. `verify_transaction_inclusion_v2` delegates directly to `verify_transaction_inclusion` after the coinbase proof check, so it inherits the same flaw: [3](#0-2) 

---

### Impact Explanation

A downstream NEAR contract (e.g., a BTC→NEAR bridge, a payment verifier, or any contract that gates asset release on SPV proof) calls `verify_transaction_inclusion_v2` with `confirmations = 1`. The light client returns `true`. The Bitcoin transaction is then re-orged away (Bitcoin re-orgs of 1–2 blocks occur regularly; 6-block re-orgs are historically documented). The downstream contract has already released funds or updated state based on a transaction that no longer exists in the canonical Bitcoin chain.

The corrupted value is the boolean SPV proof result: the contract returns `true` for a transaction that is not yet final, and any state transition gated on that result is irreversible on NEAR.

---

### Likelihood Explanation

Bitcoin re-orgs of depth 1–2 are routine. An attacker does not need to control any Bitcoin mining power — they only need to submit a valid Merkle proof for a transaction in a block that is later re-orged away, and call the NEAR contract with `confirmations = 1` before the re-org is observed. The entry path is fully unprivileged: `verify_transaction_inclusion_v2` is a public, unpaused NEAR view/call with no access control beyond the `#[pause]` attribute. [4](#0-3) 

---

### Recommendation

Enforce a protocol-level minimum confirmation count inside the contract. For Bitcoin mainnet, 6 confirmations is the widely accepted minimum for low-value transactions; higher-value use cases should require more. Add a constant and a `require!` guard:

```rust
const MIN_CONFIRMATIONS: u64 = 6;

require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    format!("Confirmations must be at least {}", MIN_CONFIRMATIONS)
);
```

This mirrors the fix applied in the referenced report (raising `REQUEST_CONFIRMATIONS` from 3 to 30 for Polygon), adapted to Bitcoin's re-org profile.

---

### Proof of Concept

1. A Bitcoin transaction `tx_id` is included in block `B` at height `H`.
2. Attacker calls `verify_transaction_inclusion_v2` on NEAR with `confirmations = 1` and a valid Merkle proof for `tx_id` in block `B`.
3. The contract checks: `gc_threshold >= 1` ✓, `tip_height - H + 1 >= 1` ✓ (block `B` is already in the pool). Returns `true`.
4. The downstream NEAR contract releases bridged assets.
5. Bitcoin miners produce a competing chain that excludes block `B`. The relayer submits the new chain; the contract re-orgs its internal state, removing `B` from `mainchain_header_to_height`.
6. The Bitcoin transaction is gone; the NEAR-side assets have already been released.

The root cause is the absence of a lower-bound check on `args.confirmations` at: [5](#0-4)
