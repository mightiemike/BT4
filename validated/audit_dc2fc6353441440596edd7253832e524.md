### Title
Fast Withdrawal Fees Permanently Locked in WithdrawPool Contract — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary
The `submitFastWithdrawal` function in `BaseWithdrawPool` collects fees on every fast withdrawal and accumulates them in the `fees[productId]` mapping, but no function exists anywhere in `BaseWithdrawPool` or `WithdrawPool` to extract those tokens. The fee tokens are permanently locked in the `WithdrawPool` contract.

---

### Finding Description

In `BaseWithdrawPool.submitFastWithdrawal`, a fee is computed and collected on every fast withdrawal: [1](#0-0) 

Two collection paths exist:

- **`sendTo == msg.sender`**: The fee is silently deducted from `transferAmount` before the transfer. The fee-sized portion of tokens that were already sitting in the pool is never sent out — it stays in the contract.
- **`sendTo != msg.sender`**: The fast-withdrawal provider pays the fee explicitly via `safeTransferFrom(token, msg.sender, uint128(fee))`, depositing real ERC20 tokens into the pool.

In both cases `fees[productId] += fee` records the accumulation: [2](#0-1) 

The `WithdrawPool` contract is a thin wrapper that adds nothing beyond `initialize`: [3](#0-2) 

`BaseWithdrawPool` exposes only `submitFastWithdrawal`, `submitWithdrawal`, `fastWithdrawalFeeAmount`, `handleWithdrawTransfer`, `safeTransferFrom`, `getToken`, and `spotEngine`. None of these reads `fees[productId]` to transfer tokens out. There is no `claimFees`, `withdrawFees`, or equivalent function. The fee tokens are irrecoverable through any supported contract entry point.

---

### Impact Explanation

Every fast withdrawal permanently locks a fee-sized portion of ERC20 tokens inside `WithdrawPool`. Over time, as fast withdrawals accumulate, the locked amount grows. The tokens cannot be routed to the treasury, insurance fund, or any other protocol-controlled address through any on-chain path. Because `WithdrawPool` is upgradeable, an emergency upgrade could recover the funds, but this carries its own operational risk. **Impact: Medium** (funds locked, upgradeable contract mitigates worst case).

---

### Likelihood Explanation

Fast withdrawals are a core, user-facing feature of the protocol. Any user or market maker calling `submitFastWithdrawal` triggers the fee lock. The condition is met on every successful fast withdrawal with a non-zero fee. **Likelihood: High**.

---

### Recommendation

Add a fee-extraction function to `BaseWithdrawPool` that transfers accumulated fees to a designated recipient (e.g., the clearinghouse owner or a treasury address):

```solidity
function claimFees(uint32 productId, address recipient) external {
    require(msg.sender == clearinghouse, "unauthorized");
    int128 amount = fees[productId];
    require(amount > 0, "no fees");
    fees[productId] = 0;
    IERC20Base token = getToken(productId);
    token.safeTransfer(recipient, uint256(uint128(amount)));
}
```

---

### Proof of Concept

1. A user signs a `WithdrawCollateral` transaction for product `P`, amount `A`.
2. A fast-withdrawal provider calls `submitFastWithdrawal(idx, transaction, signatures)`.
3. `fastWithdrawalFeeAmount` returns `fee > 0`.
4. Because `sendTo == msg.sender` (provider self-serves), `transferAmount -= fee`; the fee-sized tokens remain in the pool.
5. `fees[P] += fee` is recorded.
6. No function in `BaseWithdrawPool` or `WithdrawPool` reads `fees[P]` to transfer tokens out.
7. Repeat for every fast withdrawal — fees compound and are permanently locked. [4](#0-3)

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
