### Title
Hardcoded `lastKnownBonded = true` in `TierRedelegate` Enables Bonus Reward Overpayment for Unbonded Validator Periods — (File: `x/tieredrewards/keeper/msg_server.go`)

---

### Summary

In `TierRedelegate`, after a redelegation completes, the position's bonus accounting checkpoint is unconditionally set to `lastKnownBonded = true` regardless of the destination validator's actual bonded state. If the destination validator is unbonded at the time of redelegation, the position will incorrectly accrue bonus rewards for the entire period between the redelegation and the destination validator's next BOND event, draining the `RewardsPoolName` module account with unbacked payments.

---

### Finding Description

In `TierRedelegate`, after the staking redelegation is executed and rewards are claimed, the position's bonus checkpoint is updated at line 263:

```go
pos.UpdateBonusCheckpoints(latestSeq, sdkCtx.BlockTime(), true)
``` [1](#0-0) 

The third argument (`true`) hardcodes `lastKnownBonded = true` for the destination validator. The actual bonded state of the destination validator is never queried here. The Cosmos SDK staking module permits delegation (and therefore redelegation) to validators in any bonded state — bonded, unbonding, or unbonded — so this assumption is not guaranteed.

The `processEventsAndClaimBonus` function initializes its segment-walk using this persisted flag:

```go
bonded := pos.LastKnownBonded
segmentStart := pos.LastBonusAccrual
``` [2](#0-1) 

It then iterates over validator events recorded after `pos.LastEventSeq`. If the destination validator was unbonded at the time of redelegation and later transitions to bonded, a BOND event is appended. When the loop reaches that BOND event, it computes a bonus segment for `[lastBonusAccrual, bondEvent.Timestamp]` using `bonded = true`:

```go
if bonded {
    bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
    totalBonus = totalBonus.Add(bonus)
}
``` [3](#0-2) 

This segment spans the entire period during which the validator was actually unbonded — a period for which no bonus should accrue. The computed bonus is then paid out directly from the `RewardsPoolName` module account:

```go
if err := k.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.RewardsPoolName, ownerAddr, bonusCoins); err != nil {
``` [4](#0-3) 

The `computeSegmentBonus` formula uses the position's current delegation shares and the tier's `BonusApy`, so the overpayment scales with both the locked amount and the duration of the unbonded period:

```go
tokens := pos.Delegation.Shares.Mul(tokensPerShare)
return tokens.Mul(tier.BonusApy).MulInt64(durationSeconds).QuoInt64(types.SecondsPerYear).TruncateInt()
``` [5](#0-4) 

The same hardcoded `true` applies in both the instant-redelegation path (`unbondingID == 0`, source validator unbonded) and the standard path (`unbondingID != 0`, source validator bonded), since `UpdateBonusCheckpoints` is called unconditionally after the branch: [6](#0-5) 

---

### Impact Explanation

The corrupted value is `pos.LastKnownBonded` stored in the `Positions` KVStore. When it is incorrectly set to `true` for an unbonded destination validator, every subsequent call to `processEventsAndClaimBonus` — whether via `ClaimTierRewards`, `TierUndelegate`, `TierRedelegate`, `AddToTierPosition`, `ClearPosition`, or `ExitTierWithDelegation` — will compute and pay out bonus rewards for the unbonded gap. The payment is a real `SendCoinsFromModuleToAccount` transfer from `RewardsPoolName`, so the bonus pool balance is reduced by unbacked amounts. A position with a large locked amount in a high-APY tier, redelegated to a validator that remains unbonded for weeks, can extract a material fraction of the bonus pool.

---

### Likelihood Explanation

Validators enter the unbonded state through jailing (double-sign or downtime) or voluntary unbonding. A delegator whose validator is jailed has a direct incentive to redelegate. If the chosen destination validator is also currently unbonded (e.g., a validator that was recently jailed and is awaiting unjail), the precondition is met with no privileged access required. The trigger is a standard `MsgTierRedelegate` transaction signed by the position owner — a fully supported, unprivileged production entry path.

---

### Recommendation

Replace the hardcoded `true` with the actual bonded state of the destination validator at the time of redelegation:

```go
dstValidator, err := ms.stakingKeeper.GetValidator(ctx, dstValAddr)
if err != nil {
    return nil, err
}
pos.UpdateBonusCheckpoints(latestSeq, sdkCtx.BlockTime(), dstValidator.IsBonded())
``` [1](#0-0) 

This mirrors the intent of the `BeforeValidatorSlashed` and `AfterValidatorBeginUnbonding` hooks, which record validator state transitions so that `processEventsAndClaimBonus` can reconstruct the correct bonded timeline.

---

### Proof of Concept

1. Alice holds a tiered position (e.g., 100,000 CRO, Tier 1, 20% BonusApy) delegated to **Validator A** (currently bonded).
2. Validator A is jailed and transitions to unbonded. A `VALIDATOR_EVENT_TYPE_UNBOND` event is appended.
3. Alice calls `ClaimTierRewards` — `processEventsAndClaimBonus` processes the UNBOND event, sets `lastKnownBonded = false`, and checkpoints `LastBonusAccrual = T0`.
4. Alice calls `MsgTierRedelegate` targeting **Validator B**, which is also currently unbonded (e.g., recently jailed).
   - `claimRewards` runs; no new events since T0, so no bonus is paid.
   - `ms.redelegate(...)` succeeds (SDK permits redelegation to unbonded validators).
   - `pos.UpdateBonusCheckpoints(latestSeq_B, T0, true)` — `lastKnownBonded` is incorrectly set to `true`; `LastBonusAccrual = T0`.
5. Validator B unjails and bonds at time **T1** (e.g., 30 days later). A `VALIDATOR_EVENT_TYPE_BOND` event is appended with seq `S`.
6. Alice calls `ClaimTierRewards`.
   - `processEventsAndClaimBonus` starts with `bonded = true`, `segmentStart = T0`.
   - The BOND event at T1 is fetched (seq > `latestSeq_B`).
   - `bonded = true` → bonus is computed for `[T0, T1]` (30 days) at the snapshot rate.
   - Bonus ≈ `100,000 × 0.20 × (30 × 86400) / 31,536,000` ≈ **1,643 CRO** paid from `RewardsPoolName` for a period when Validator B was entirely unbonded.
7. After the BOND event, `bonded = true` (no change), and the final segment `[T1, now]` is also correctly paid.

The net effect is that Alice receives ~1,643 CRO of unbacked bonus rewards. Repeated across multiple positions or a longer unbonded period, this drains the `RewardsPoolName` module account.
