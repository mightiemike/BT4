### Title
`run_mainchain_gc()` Lacks Caller Authorization, Allowing Anyone to Prematurely Purge Mainchain Headers — (File: `contract/src/lib.rs`)

### Summary
`run_mainchain_gc()` is a public, state-mutating function that permanently removes mainchain block headers. It carries no caller-identity check: any unprivileged NEAR account can invoke it with an arbitrarily large `batch_size`, forcing the contract to immediately discard every header that exceeds `gc_threshold`. This causes SPV proofs for those headers to fail permanently, because the removed headers are irrecoverable.

### Finding Description
`run_mainchain_gc` is decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`. [1](#0-0) 

That attribute restricts callers **only when the contract is paused** — when unpaused, the function is callable by any NEAR account with no role requirement. The `UnrestrictedRunGC` role comment confirms this reading: it exists to allow GC to run *on a paused contract*, not to gate who may call GC in general. [2](#0-1) 

Inside the function, `selected_amount_to_remove` is bounded by `min(total_amount_to_remove, batch_size)`, where `total_amount_to_remove = amount_of_headers_we_store - gc_threshold`. [3](#0-2) 

An attacker passes `batch_size = u64::MAX`. The contract then removes every header currently above `gc_threshold` in a single transaction, advancing `mainchain_initial_blockhash` to the new oldest surviving header. [4](#0-3) 

By contrast, the only other caller of this function is `submit_blocks`, which is gated behind `#[trusted_relayer]` and requires the `UnrestrictedSubmitBlocks` or `DAO` role. [5](#0-4) [6](#0-5) 

The design intent is that GC runs gradually (one batch per `submit_blocks` call). The public exposure without a role check breaks this invariant.

### Impact Explanation
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` look up the target block in `mainchain_header_to_height` and `headers_pool`. [7](#0-6) 

Once `run_mainchain_gc` removes a header, both maps no longer contain it. Any subsequent SPV proof call for that block panics with `"block does not belong to the current main chain"`. The removal is permanent — there is no re-insertion path. Consumers who were waiting for sufficient confirmations before submitting a proof lose the ability to prove inclusion for those blocks forever.

### Likelihood Explanation
The entry path requires no privilege: any NEAR account can call `run_mainchain_gc` with `batch_size = u64::MAX` at any time the contract is unpaused. The call costs only gas. An attacker who wants to invalidate a specific pending SPV proof simply monitors the chain, waits until the target block's height falls within the removable window (i.e., `amount_of_headers_we_store > gc_threshold`), and fires the call before the legitimate proof is submitted.

### Recommendation
Add a role check at the top of `run_mainchain_gc` so that only accounts holding an appropriate role (e.g., `UnrestrictedRunGC`, `DAO`, or a new dedicated role) may call it directly. The `#[pause(except(...))]` attribute is not a substitute for a caller-identity guard when the contract is unpaused.

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
    // Add before any state mutation:
    require!(
        self.acl_has_role(Role::UnrestrictedRunGC, &env::predecessor_account_id())
            || self.acl_has_role(Role::DAO, &env::predecessor_account_id()),
        "Caller is not authorized to run GC"
    );
    // ... rest of function
}
```

### Proof of Concept
1. Deploy the contract and initialize it with `gc_threshold = 10`.
2. Submit 15 headers (5 over threshold) via a trusted relayer account.
3. From any unprivileged account, call:
   ```json
   { "method": "run_mainchain_gc", "args": { "batch_size": 9999999 } }
   ```
4. Observe that `get_mainchain_size()` returns `10` — all 5 excess headers are gone.
5. Attempt `verify_transaction_inclusion` for any of the 5 removed headers — the call panics with `"block does not belong to the current main chain"`.

The existing integration test `test_gc` already demonstrates this call path succeeding from a non-privileged account (`user_account`), confirming the absence of any authorization check. [8](#0-7)
