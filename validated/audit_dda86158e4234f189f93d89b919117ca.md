### Title
`validate_proof_size` Is Never Called in Gateway Stateless Validation, Allowing Oversized Proofs to Bypass Admission — (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The `StatelessTransactionValidator::validate()` function calls `validate_client_side_proving_allowed` and `validate_proof_facts_and_proof_consistency` for invoke transactions, but never calls `validate_proof_size`. The proof-size enforcement function is defined in the same file but is a dead code path — it has exactly two references in the file (its own definition and its body), and zero call sites anywhere in the repository. An unprivileged user can submit an `RpcInvokeTransaction::V3` carrying an arbitrarily large `proof` field that passes all gateway stateless checks and enters the mempool.

### Finding Description

`StatelessTransactionValidator::validate()` dispatches invoke-specific checks as follows:

```rust
// crates/apollo_gateway/src/stateless_transaction_validator.rs
if let RpcTransaction::Invoke(invoke_tx) = tx {
    self.validate_client_side_proving_allowed(invoke_tx)?;
    self.validate_proof_facts_and_proof_consistency(invoke_tx)?;
    // validate_proof_size is NEVER called
}
``` [1](#0-0) 

The function `validate_proof_size` is defined in the same file and enforces `proof.0.len() <= self.config.max_proof_size`:

```rust
fn validate_proof_size(
    &self,
    tx: &RpcInvokeTransaction,
) -> StatelessTransactionValidatorResult<()> {
    let RpcInvokeTransaction::V3(tx) = tx;
    let proof_size = tx.proof.0.len();
    if proof_size > self.config.max_proof_size {
        return Err(StatelessTransactionValidatorError::ProofTooLarge {
            proof_size,
            max_proof_size: self.config.max_proof_size,
        });
    }
    Ok(())
}
``` [2](#0-1) 

`validate_client_side_proving_allowed` returns `Ok(())` immediately when `allow_client_side_proving` is `true`, which is the production setting for nodes that accept client-side proving transactions:

```rust
fn validate_client_side_proving_allowed(...) {
    if self.config.allow_client_side_proving {
        return Ok(());   // early return — proof size never checked
    }
    ...
}
``` [3](#0-2) 

This is the direct sequencer analog of the external bug: just as `verifyAndAccruePremium()` returned `(0, false)` early when `_latestPaymentTimestamp < _startTimestamp` — preventing the expiration branch from ever running — here the early return in `validate_client_side_proving_allowed` combined with the missing call to `validate_proof_size` means the size gate is never reached. The `proof` field on `RpcInvokeTransactionV3` is an `Arc<Vec<u8>>` with no upper bound enforced at admission:

```rust
pub struct RpcInvokeTransactionV3 {
    ...
    pub proof_facts: ProofFacts,
    pub proof: Proof,   // unbounded at gateway admission
}
``` [4](#0-3) 

The proof is later passed to `spawn_proof_verification` inside `convert_rpc_tx_to_internal`, which calls the cryptographic verifier. An arbitrarily large proof blob reaches the verifier without any prior size gate. [5](#0-4) 

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

A single unprivileged user can craft an `RpcInvokeTransaction::V3` with a `proof` field of arbitrary byte length. The transaction passes all stateless gateway checks (`validate_tx_size` checks overall calldata/signature sizes, not the proof blob), enters the mempool, and forces the proof-verification worker to process an unbounded payload. This can exhaust CPU/memory in the proof-verification path and stall or crash the gateway worker, effectively denying service to legitimate client-side-proving transactions.

### Likelihood Explanation

**Medium.** The `allow_client_side_proving` flag must be `true` (the intended production state for nodes supporting client-side proving). Any user who can reach the gateway HTTP endpoint can trigger this with a single malformed transaction. No privileged access, special key material, or prior state is required.

### Recommendation

Add the missing call to `validate_proof_size` inside the invoke-specific validation block in `validate()`:

```rust
if let RpcTransaction::Invoke(invoke_tx) = tx {
    self.validate_client_side_proving_allowed(invoke_tx)?;
    self.validate_proof_facts_and_proof_consistency(invoke_tx)?;
    self.validate_proof_size(invoke_tx)?;   // add this line
}
``` [1](#0-0) 

### Proof of Concept

1. Construct an `RpcInvokeTransaction::V3` with `proof_facts` set to any valid non-empty `ProofFacts` and `proof` set to a `Proof` whose inner `Vec<u8>` has length `max_proof_size + 1` (e.g., 10 MB of zeros).
2. Submit it to the gateway's `add_transaction` endpoint on a node with `allow_client_side_proving: true`.
3. Observe that `StatelessTransactionValidator::validate()` returns `Ok(())` — the `validate_proof_size` check is never reached.
4. The transaction is forwarded to `convert_rpc_tx_to_internal`, which calls `spawn_proof_verification` with the oversized proof blob, triggering unbounded work in the verifier. [3](#0-2) [2](#0-1)
