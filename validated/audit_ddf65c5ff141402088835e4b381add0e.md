### Title
`deser_auto` Silently Drops `max_atom_len` and `strict` Constraints for Backrefs-Format Blobs — (File: `wheel/src/api.rs`)

---

### Summary

The Python-exposed `deser_auto` function accepts `max_atom_len` and `strict` security parameters but silently discards them when the input blob is in legacy/backrefs format. Only the serde_2026 path enforces these constraints. An attacker who supplies a backrefs-format blob to any caller using `deserialize(blob, "auto", max_atom_len=N)` bypasses the caller's atom-size limit entirely.

---

### Finding Description

`deser_auto` in `wheel/src/api.rs` routes between two deserializers based on the magic prefix: [1](#0-0) 

For the 2026 path, `max_atom_len` and `strict` are forwarded to `deserialize_2026_body_from_stream`. For the backrefs path, `node_from_bytes_backrefs` is called with **neither parameter**: [2](#0-1) 

`node_from_bytes_backrefs` has no `max_atom_len` parameter at all: [3](#0-2) 

The Python `deserialize()` wrapper in `serde.py` passes `max_atom_len` to `deser_auto` when `fmt="auto"`, creating a false expectation that the limit is enforced for all formats: [4](#0-3) 

The backrefs atom parser's only internal size bound is the `decode_size` function, which caps at `0x400000000` (16 GiB): [5](#0-4) 

This means a caller passing `max_atom_len=32` (e.g., for consensus-critical validation) receives no enforcement whatsoever when the blob is in backrefs format.

---

### Impact Explanation

Any Python caller using `deserialize(blob, "auto", max_atom_len=N)` or `deser_auto(blob, max_atom_len=N)` with a backrefs-format blob will have the atom-size constraint silently bypassed. Concrete consequences:

1. **Memory exhaustion**: An attacker crafts a backrefs-format blob with a multi-megabyte or multi-gigabyte atom. The caller's `max_atom_len` guard is not applied, and the allocator will attempt to store the full atom, potentially causing OOM.
2. **Consensus divergence**: If downstream consensus code (e.g., `chia_rs`) uses `deser_auto` with a specific `max_atom_len` to enforce a protocol-level atom size limit, nodes that receive a backrefs-format blob with an oversized atom will accept it while nodes using a format-aware path will reject it. This produces a split in accepted/rejected transactions.

The corrupted result is a `LazyNode` containing an atom whose byte length exceeds `max_atom_len`, returned as `Ok(...)` instead of `Err(SerializationError)`.

---

### Likelihood Explanation

**Medium.** The `deser_auto` Rust docstring notes "Consensus-aware callers should sniff the prefix themselves and use their own caps," but the Python `deserialize()` wrapper does not document this limitation. A Python developer calling `deserialize(blob, "auto", max_atom_len=32)` has a reasonable expectation that the limit is enforced regardless of format. The backrefs format is the dominant wire format for existing Chia programs, so the unprotected path is the common case, not the exception.

---

### Recommendation

Forward `max_atom_len` to the backrefs deserializer path in `deser_auto`. This requires either:

1. Adding a `max_atom_len` parameter to `node_from_bytes_backrefs` / `node_from_stream_backrefs` and enforcing it inside `parse_atom`, or
2. Post-parse validation: after `node_from_bytes_backrefs` returns, walk the resulting tree and reject any atom exceeding `max_atom_len`.

Additionally, the Python `deserialize()` docstring should explicitly document that `max_atom_len` is currently only enforced for the `"2026"` format path until the backrefs path is updated.

---

### Proof of Concept

```python
from clvm_rs.serde import deserialize

# Craft a backrefs-format blob containing a 1000-byte atom.
# Atom length encoding: 0xC3 0xE8 = length 1000 (two leading-1 prefix bytes).
blob = bytes([0xC3, 0xE8]) + b'\x41' * 1000

# Caller expects no atom larger than 32 bytes to be accepted.
result = deserialize(blob, "auto", max_atom_len=32)

# Result: a LazyNode containing a 1000-byte atom — max_atom_len=32 was silently ignored.
# The 2026 path would have raised ValueError; the backrefs path does not.
print(result.atom)  # b'A' * 1000
```

The same blob passed through the 2026-aware path raises `ValueError`; through `deser_auto` with a backrefs blob it succeeds, demonstrating the format-dependent enforcement gap. [6](#0-5) [7](#0-6)

```rust
fn deser_auto(blob: &[u8], max_atom_len: usize, strict: bool) -> PyResult<LazyNode> {
    let mut a = Allocator::new();
    let node = if let Some(body) = blob.strip_prefix(SERDE_2026_MAGIC_PREFIX.as_slice()) {
        deserialize_2026_body_from_stream(&mut a, &mut Cursor::new(body), max_atom_len, strict)

    } else {
        node_from_bytes_backrefs(&mut a, blob).map_err(eval_to_py)?
    };
    Ok(LazyNode::new(Rc::new(a), node))
}
```

```python
def deserialize(
    blob: bytes,
    fmt: str = "auto",
    *,
    max_atom_len: int | None = None,
    strict: bool = True,
):
    """Deserialize bytes into a LazyNode.

    Formats: "auto" (default), "legacy", "backrefs", "2026".
    "auto" handles all three formats by inspecting the magic prefix.

    Keyword-only limits (applied to "2026" and "auto" paths):
        max_atom_len:   largest single atom in bytes (default 1 MiB)
        strict:         reject overlong varint encodings

    Total input bytes is bounded by the input slice itself; consensus-aware
    callers (e.g. chia_rs) should pass their own ``max_atom_len``.
    """
    fn = _DESERIALIZERS.get(fmt)
    if fn is None:
        raise ValueError(f"unknown deserialize format {fmt!r}, expected one of {list(_DESERIALIZERS)}")
    if fmt in ("2026", "auto"):
        kwargs: dict = {"strict": strict}
        if max_atom_len is not None:
            kwargs["max_atom_len"] = max_atom_len
        return fn(blob, **kwargs)
    return fn(blob)
```
