Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Allowing Full Allowlist Bypass When Router Is Allowlisted - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which equals `msg.sender` of the pool — the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the pool admin allowlists the router address (a necessary operational step to let legitimate users use the router), every unprivileged user can bypass the allowlist by routing through the same public router contract.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to `_beforeSwap`:**

In `MetricOmmPool.swap`, the pool calls `_beforeSwap(msg.sender, recipient, ...)`: [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards `sender` unchanged to every configured extension:** [2](#0-1) 

**Step 3 — `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`:** [3](#0-2) 

**Step 4 — `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the pool's `msg.sender`:** [4](#0-3) 

At that point, `sender` received by the extension equals the router's address, not the end user's address. The pool admin has no way to allowlist individual users who use the router without also allowlisting the router itself. Once the router is allowlisted, the check `allowedSwapper[pool][router] == true` passes for every caller of the router, regardless of whether that caller is on the allowlist.

No existing guard prevents this: the `onlyPool` modifier on `beforeSwap` only verifies the caller is a registered pool, not that the `sender` argument reflects the true initiating EOA. [5](#0-4) 

## Impact Explanation

A curated pool that restricts swaps to a specific set of addresses (e.g., approved market makers, KYC'd counterparties) is fully open to any user who routes through `MetricOmmSimpleRouter`. The attacker receives the pool's output tokens and pays only the pool's spread/notional fee, exactly as an allowlisted user would. LP principal is exposed to unauthorized trading flow that the pool admin explicitly intended to block. This constitutes a broken core pool functionality (allowlist policy) causing direct loss of LP assets and unauthorized access to restricted pools — matching the "Admin-boundary break" and "broken core pool functionality" impact categories.

## Likelihood Explanation

The trigger requires the pool admin to have allowlisted the router address. This is a natural and expected operational step: any pool admin who wants their allowlisted users to be able to use the standard periphery router must allowlist it. The router is a public, permissionless contract, so once the router is allowlisted, any address can exploit the bypass without any privileged access or special setup. The attack is repeatable and requires no special capability beyond calling the public router.

## Recommendation

The extension must verify the end user's identity, not the immediate caller of `pool.swap()`. Options:

1. **Pass the original caller through the router**: Have `MetricOmmSimpleRouter` include the end user's address in `extensionData` and have `SwapAllowlistExtension` decode and check it. This requires a coordinated convention between router and extension.
2. **Preferred — dedicated `originalSender` field or transient storage**: Add a dedicated `originalSender` field to the extension interface or use a transient-storage slot (analogous to how the router already stores the payer in transient storage via `TransientCallbackPool`) so the extension can always recover the true initiating EOA. [6](#0-5) 

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin allowlists alice (EOA) and MetricOmmSimpleRouter
  (so alice can use the router — a necessary operational step)

Attack:
  charlie (not allowlisted) calls:
    router.exactInputSingle({pool: pool, ..., recipient: charlie, ...})

  router calls:
    pool.swap(charlie, zeroForOne, amount, priceLimit, "", extensionData)
    // msg.sender in pool = router address

  pool calls _beforeSwap(sender=router_address, ...)

  SwapAllowlistExtension.beforeSwap checks:
    allowedSwapper[pool][router_address] == true  ← passes because admin allowlisted the router

  Swap executes; charlie receives output tokens from the restricted pool.

Foundry test plan:
  1. Deploy pool with SwapAllowlistExtension configured as beforeSwap hook.
  2. Pool admin calls setAllowedToSwap(pool, alice, true) and setAllowedToSwap(pool, router, true).
  3. As charlie (not allowlisted), call router.exactInputSingle(...).
  4. Assert swap succeeds and charlie receives output tokens — demonstrating full allowlist bypass.
```

The exact corrupted value is `allowedSwapper[pool][router]` being used as a proxy for per-user authorization when it actually grants blanket access to every caller of the router. [7](#0-6)
