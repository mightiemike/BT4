### Title
Equihash Solution Validity Return Value Ignored — Zcash PoW Verification Bypassed - (File: contract/src/zcash.rs)

### Summary
In the Zcash build of the BTC light client, `equihash::is_valid_solution` returns `Result<bool, Error>` where the `bool` indicates whether the solution is cryptographically valid. The contract code handles only the `Err` branch (malformed input) via `.unwrap_or_else`, but the `Ok(bool)` value is silently discarded. An unprivileged relayer can submit a Zcash block header with a completely invalid Equihash solution and the contract will accept it as a valid PoW block.

### Finding Description
In `contract/src/zcash.rs`, the `check_pow` function calls `equihash::is_valid_solution` and chains `.unwrap_or_else`:

```rust
equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
``` [1](#0-0) 

`unwrap_or_else` on `Result<bool, Error>` returns the inner `bool` when the result is `Ok`. The closure only executes on `Err` (a deserialization/parameter error). The returned `bool` — which is `false` when the Equihash solution is cryptographically invalid — is a discarded expression; it is never bound to a variable, never used in a `require!`, and never checked in any conditional. The Rust compiler does not warn about this because the expression is syntactically valid as a statement.

The `equihash` crate's `is_valid_solution` contract is:
- `Ok(true)` → solution is valid
- `Ok(false)` → solution is **invalid** (wrong answer to the PoW puzzle)
- `Err(e)` → input is malformed

Only the `Err` path causes a panic. `Ok(false)` silently passes through. [2](#0-1) 

### Impact Explanation
Any unprivileged NEAR account acting as a relayer can call `submit_blocks` with Zcash headers whose `solution` field is arbitrary garbage (e.g., all zeros). The difficulty target check (`block_header.bits`) and timestamp checks still run, but the Equihash PoW puzzle itself is never enforced. The contract will store these headers in `headers_pool`, update `mainchain_tip_blockhash`, and update `mainchain_height_to_header` / `mainchain_header_to_height`. Downstream consumers calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` will receive `true` for transactions claimed to be in these fake blocks, because the Merkle proof is checked against the `merkle_root` field of an accepted-but-fake header. This constitutes a complete proof-verification forgery: an attacker can fabricate an entire fake Zcash chain and have SPV proofs for non-existent transactions verified as valid. [3](#0-2) 

### Likelihood Explanation
The `submit_blocks` entry point is gated only by the `#[trusted_relayer]` macro and a `#[pause]` guard. [4](#0-3) 

The `trusted_relayer` macro has bypass roles (`Role::DAO`, `Role::UnrestrictedSubmitBlocks`), but in a permissionless deployment any staked relayer can submit blocks. Even if the trusted-relayer gate is strict, the bug is in the cryptographic verification layer itself, not in access control — any account that can submit a block at all can exploit this. The `skip_pow_verification` flag is documented to be `false` in production, meaning `check_pow` is always called, but the Equihash check inside it is silently a no-op. [5](#0-4) 

### Recommendation
Replace the discarded-expression pattern with an explicit validity assertion:

```rust
let valid = equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
require!(valid, "Invalid Equihash solution");
```

Or equivalently, use a single `unwrap_or_else` that also checks the boolean:

```rust
require!(
    equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
        .unwrap_or_else(|e| env::panic_str(&format!("Invalid Equihash solution: {e}"))),
    "Invalid Equihash solution"
);
```

### Proof of Concept
1. Build the contract with `--features zcash`.
2. Initialize the contract with a valid Zcash genesis block.
3. Construct a `ZcashHeader` whose `bits` field satisfies the difficulty target (copy from a real block), whose `time` is within bounds, whose `version >= 4`, but whose `solution` field is `vec![0u8; 1344]` (the standard Equihash(200,9) solution size, all zeros — cryptographically invalid).
4. Call `submit_blocks` with this header.
5. Observe that the block is accepted: `get_last_block_header()` returns the fake header, and `verify_transaction_inclusion` returns `true` for any Merkle proof constructed against the fake header's `merkle_root`. [6](#0-5)
