Audit Report

## Title
`SwapAllowlistExtension` checks the router address as `sender` instead of the originating user, allowing any caller to bypass the per-address swap allowlist via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the `pool.swap()` call. When users route through `MetricOmmSimpleRouter`, the router is the direct caller of `pool.swap()`, so `sender` equals the router address. Any pool admin who allowlists the router (required for router-mediated swaps to function) simultaneously grants every address on the network the ability to bypass the per-user allowlist by routing through the router.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // direct caller of pool.swap()
    ...
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension caller) and `sender` is the first argument — the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter.exactInputSingle()` is used, the router calls `pool.swap()` directly without forwarding the originating user:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData   // no originator embedded
    );
```

So `sender` inside `beforeSwap` is the router address. The check evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. A pool admin who calls `setAllowedToSwap(pool, router, true)` to enable router usage simultaneously opens the allowlist to every user who routes through the router. The same structural flaw exists in `exactInput` (multi-hop forward) and `exactOutputSingle`/`exactOutput` paths, all of which call `pool.swap()` with the router or a pool as `msg.sender`.

Existing guards are insufficient: `allowAllSwappers` is a separate flag, and `allowedSwapper` is keyed on the direct caller — there is no mechanism in the extension or the pool to recover the originating EOA.

## Impact Explanation
The `SwapAllowlistExtension` is the sole on-chain mechanism for restricting which addresses may trade against a pool. Its bypass means any unpermissioned user can execute swaps against a pool the admin intended to restrict to a whitelist. LP capital, oracle-derived prices, and protocol fees are all accessible to actors the pool admin explicitly excluded. The pool admin has no on-chain recourse short of removing the router from the allowlist, which simultaneously blocks all legitimate router users. This is a direct, complete loss of access-control over swap execution — a broken core pool functionality causing potential loss of funds to LPs and the protocol.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard periphery entry point for swaps. Any pool that wants to support normal user interaction through the router must allowlist the router address via `setAllowedToSwap`. Once that single admin action is taken, the bypass is available to every address on the network with zero additional preconditions — no privileged role, special token, or unusual setup is required. A single public `exactInputSingle` call suffices.

## Recommendation
The extension must gate the originating user, not the direct pool caller. The router should embed `msg.sender` in `extensionData` before calling `pool.swap()`, and `SwapAllowlistExtension.beforeSwap` should decode and check that value when `sender` is a known router, falling back to `sender` when no originator is present. Alternatively, the extension can maintain a registry of trusted routers and, when `sender` is a router, require `extensionData` to carry the originator address. The simplest backward-compatible fix is for the router to ABI-encode `msg.sender` into `extensionData` and for the extension to decode it.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls setAllowedToSwap(pool, router, true)   // enable router usage
  pool admin does NOT call setAllowedToSwap(pool, alice, true)  // alice is blocked

Attack:
  alice calls router.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(recipient, ...)
      → pool calls _beforeSwap(msg.sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  → PASSES
  alice's swap executes successfully despite being individually blocked
```

Foundry test plan: deploy pool with `SwapAllowlistExtension`, call `setAllowedToSwap(pool, router, true)`, confirm `setAllowedToSwap(pool, alice, false)` (or simply never allowlist alice), then call `router.exactInputSingle` from alice's address and assert the swap succeeds — demonstrating the allowlist invariant is broken.
