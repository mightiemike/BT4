### Title
`paid_fee_on_l1` silently dropped and replaced with hardcoded `Fee(1)` during `ConsensusTransaction → InternalConsensusTransaction` L1Handler conversion — (`File: crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary

The `paid_fee_on_l1` field of `executable_transaction::L1HandlerTransaction` is permanently lost when an L1Handler transaction crosses the consensus P2P boundary. The proposer strips it during `convert_internal_consensus_tx_to_consensus_tx`, the protobuf wire format (`L1HandlerV0`) has no slot for it, and the validator reconstructs the executable transaction with a hardcoded `Fee(1)` instead of the real value. This binds the wrong executable payload to every L1Handler transaction executed by a validator node.

### Finding Description

**Type hierarchy mismatch at the conversion boundary**

`ConsensusTransaction::L1Handler` wraps `transaction::L1HandlerTransaction` (the raw, hash-less, fee-less type), while `InternalConsensusTransaction::L1Handler` wraps `executable_transaction::L1HandlerTransaction` (which carries both `tx_hash` and `paid_fee_on_l1`). [1](#0-0) 

**Step 1 — proposer drops the field**

`convert_internal_consensus_tx_to_consensus_tx` extracts only `tx.tx`, silently discarding `tx.tx_hash` and `tx.paid_fee_on_l1`:

```rust
InternalConsensusTransaction::L1Handler(tx) => {
    Ok(ConsensusTransaction::L1Handler(tx.tx))   // paid_fee_on_l1 gone
}
``` [2](#0-1) 

**Step 2 — wire format has no slot**

The protobuf `L1HandlerV0` message carries only `nonce`, `address`, `entry_point_selector`, and `calldata`. There is no `paid_fee_on_l1` field. [3](#0-2) 

**Step 3 — validator injects hardcoded `Fee(1)`**

`convert_consensus_l1_handler_to_internal_l1_handler` reconstructs the executable transaction with a constant:

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
``` [4](#0-3) 

**Step 4 — blockifier uses the field for fee enforcement**

The blockifier's L1Handler execution path checks `paid_fee_on_l1` and rejects the transaction if it is zero:

```rust
let paid_fee = self.paid_fee_on_l1;
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee { paid_fee, actual_fee: receipt.fee },
    )));
}
``` [5](#0-4) 

**Step 5 — central blob records the real fee; execution uses `Fee(1)`**

The `CentralL1HandlerTransaction` blob written by the proposer carries the real `paid_fee_on_l1`: [6](#0-5) 

But every validator node executes the same transaction with `paid_fee_on_l1 = Fee(1)`, creating a permanent discrepancy between the authoritative blob record and the actual executable object used for state transitions.

### Impact Explanation

**Impact: High — transaction conversion binds the wrong executable payload.**

Every L1Handler transaction executed by a validator node carries `paid_fee_on_l1 = Fee(1)` regardless of the actual ETH fee paid on L1. This is a structural wrong-value in the executable object. Two concrete consequences:

1. **Proposer/validator divergence on zero-fee messages.** If an L1 message is sent with `paid_fee = 0` (permitted by the L1 contract interface), the proposer's blockifier rejects it (`paid_fee == Fee(0)` → `InsufficientFee` error). The validator's blockifier accepts it (`Fee(1) != Fee(0)`). This is a reachable consensus divergence: the proposer cannot include the transaction, but a validator receiving it via a different path would accept it.

2. **Blob/execution discrepancy.** The central blob records the real `paid_fee_on_l1` (e.g., `Fee(1000)`), but the validator's blockifier executes with `Fee(1)`. Any system that cross-checks the blob against execution results will observe a mismatch.

### Likelihood Explanation

**Likelihood: High** — the wrong value is injected for *every* L1Handler transaction processed by a validator node. The trigger is the normal consensus validation path, not an adversarial condition. The divergence on zero-fee messages requires a message sent with 0 ETH, which is an edge case but is not blocked at the protocol level.

### Recommendation

1. Add `paid_fee_on_l1` to the `ConsensusTransaction::L1Handler` variant (or a wrapper type) so the field survives the conversion boundary.
2. Add a `paid_fee_on_l1` field to the protobuf `L1HandlerV0` message and update the serializer/deserializer.
3. Remove the hardcoded `Fee(1)` in `convert_consensus_l1_handler_to_internal_l1_handler` and use the value carried in the consensus message.
4. Add a round-trip test asserting that `paid_fee_on_l1` is preserved across `InternalConsensusTransaction → ConsensusTransaction → InternalConsensusTransaction`.

### Proof of Concept

```
Proposer side:
  L1HandlerTransaction { tx, tx_hash, paid_fee_on_l1: Fee(5000) }   // from L1 provider
  → convert_internal_consensus_tx_to_consensus_tx
  → ConsensusTransaction::L1Handler(tx.tx)                          // Fee(5000) dropped
  → protobuf L1HandlerV0 { nonce, address, selector, calldata }     // no paid_fee field
  → sent over P2P

Validator side:
  protobuf L1HandlerV0 { nonce, address, selector, calldata }
  → ConsensusTransaction::L1Handler(transaction::L1HandlerTransaction { ... })
  → convert_consensus_l1_handler_to_internal_l1_handler
  → executable_transaction::L1HandlerTransaction::create(tx, chain_id, Fee(1))
                                                              ^^^^^^
                                                  wrong value; real was Fee(5000)

Blockifier check on validator:
  paid_fee = Fee(1)
  if paid_fee == Fee(0) { reject }   // passes — wrong value silently accepted

Central blob (proposer):
  CentralL1HandlerTransaction { paid_fee_on_l1: Fee(5000), ... }   // real value
  vs. validator execution with Fee(1)                               // divergent
```
