### Title
`ValidResourceBounds` Variant Collapse in Protobuf Deserialization Silently Rejects Valid V3 Invoke Transactions with Zero L2/L1DataGas Bounds During Consensus P2P Propagation — (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a zero-value heuristic to reconstruct the enum variant: if both `l2_gas` and `l1_data_gas` are zero, it produces `ValidResourceBounds::L1Gas` instead of `ValidResourceBounds::AllResources`. The gateway, however, accepts V3 invoke transactions whose `AllResourceBounds` has all-zero `l2_gas` and `l1_data_gas`. When such a transaction is later serialized by the proposer and deserialized by a validator over the consensus P2P channel, the variant collapses to `L1Gas`, causing `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` to return `DEPRECATED_RESOURCE_BOUNDS_ERROR`. The validator then marks the entire proposal as `Failed`, blocking consensus progress.

---

### Finding Description

**Step 1 — Gateway accepts the transaction.**

`RpcInvokeTransactionV3` always uses `AllResourceBounds` (not `ValidResourceBounds`), so the gateway imposes no constraint requiring non-zero `l2_gas` or `l1_data_gas`. The stateless validator test confirms that `AllResourceBounds { l1_gas: NON_EMPTY_RESOURCE_BOUNDS, ..Default::default() }` (all-zero `l2_gas` and `l1_data_gas`) is accepted. [1](#0-0) 

**Step 2 — Proposer serializes the transaction.**

`From<RpcInvokeTransactionV3> for InvokeTransactionV3` always wraps the bounds as `ValidResourceBounds::AllResources(tx.resource_bounds)`. `From<ValidResourceBounds> for protobuf::ResourceBounds` for the `AllResources` arm emits all three fields, including `l1_data_gas: Some(0)` and `l2_gas: Some(0)`. [2](#0-1) [3](#0-2) 

**Step 3 — Validator deserializes: variant collapses.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies the heuristic: if `l1_data_gas.is_zero() && l2_gas.is_zero()` → return `L1Gas`. Because the wire bytes carry `l1_data_gas = 0` and `l2_gas = 0`, the reconstructed value is `ValidResourceBounds::L1Gas(l1_gas)` — not `AllResources`. [4](#0-3) 

**Step 4 — Conversion to `RpcInvokeTransactionV3` fails.**

`TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` only accepts `ValidResourceBounds::AllResources`; any other variant returns `StarknetApiError::OutOfRange`. The caller in `TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` maps this to `DEPRECATED_RESOURCE_BOUNDS_ERROR` and propagates it. [5](#0-4) [6](#0-5) 

**Step 5 — Consensus proposal is rejected.**

`TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction` propagates the error. The `validate_proposal` handler catches it and returns `HandledProposalPart::Failed`, aborting the current proposal. [7](#0-6) [8](#0-7) 

**The hash domain divergence (analog to the `transfer` gas stipend).**

`get_tip_resource_bounds_hash` produces a different hash for `L1Gas` (two resource felts: L1 + L2) versus `AllResources` (three resource felts: L1 + L2 + L1DataGas). A transaction signed and hashed at the gateway under `AllResources` would, if the variant were silently accepted after collapse, bind to a different hash preimage — the exact analog of the 2300-gas stipend silently truncating the call. [9](#0-8) 

---

### Impact Explanation

An unprivileged user submits a syntactically valid V3 invoke transaction with `l2_gas = 0` and `l1_data_gas = 0`. The gateway accepts it, the mempool stores it, and the proposer includes it in a block proposal. Every validator that receives the proposal over the consensus P2P channel fails to deserialize the transaction and marks the proposal `Failed`. The proposer cannot commit the block, stalling consensus for that round. Repeated injection of such transactions constitutes a low-cost, sustained denial-of-service against block production.

Matching impact: **High — Transaction conversion logic binds the wrong type/executable payload; mempool/gateway admission accepts transactions that are rejected before sequencing.**

---

### Likelihood Explanation

The trigger requires only a standard RPC `starknet_addInvokeTransaction` call with `resource_bounds = { l1_gas: <nonzero>, l2_gas: 0x0, l1_data_gas: 0x0 }`. No privileged access, no special account, and no invalid bytes are needed. The gateway's stateless validator explicitly permits this combination. The attack is therefore trivially reachable by any external caller.

---

### Recommendation

1. **Gateway-side gate**: Add a stateless validation rule rejecting V3 invoke transactions where both `l2_gas.max_amount == 0 && l2_gas.max_price_per_unit == 0` and `l1_data_gas.max_amount == 0 && l1_data_gas.max_price_per_unit == 0`. This mirrors the intent of the `L1Gas` / `AllResources` split and prevents the ambiguous wire encoding from entering the mempool.

2. **Protobuf deserializer fix**: Remove the zero-value heuristic in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Since the consensus and mempool P2P channels only carry post-0.13.3 transactions, always produce `AllResources` when `l1_data_gas` is `Some(...)` (even if zero). The existing TODO comment (`// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2`) should be resolved by enforcing `AllResources` unconditionally for the consensus path.

3. **Defensive conversion**: `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` could map `L1Gas` to `AllResources` with zero `l2_gas`/`l1_data_gas` rather than returning an error, preventing a hard failure from a recoverable encoding ambiguity.

---

### Proof of Concept

```
POST /gateway/add_transaction
{
  "type": "INVOKE",
  "version": "0x3",
  "sender_address": "<valid_account>",
  "calldata": [...],
  "signature": [...],          // signed over AllResources hash
  "nonce": "0x0",
  "resource_bounds": {
    "l1_gas":      { "max_amount": "0x1", "max_price_per_unit": "0x1" },
    "l2_gas":      { "max_amount": "0x0", "max_price_per_unit": "0x0" },
    "l1_data_gas": { "max_amount": "0x0", "max_price_per_unit": "0x0" }
  },
  "tip": "0x0",
  "paymaster_data": [],
  "account_deployment_data": [],
  "nonce_data_availability_mode": "L1",
  "fee_data_availability_mode": "L1"
}
```

1. Gateway accepts the transaction (stateless validator passes `AllResourceBounds { l1_gas: nonzero, l2_gas: 0, l1_data_gas: 0 }`).
2. Transaction enters the mempool.
3. Proposer pulls it, executes it via blockifier (succeeds — `AllResources` is used throughout execution), and streams it in a `ProposalPart::Transactions` batch.
4. Validator deserializes the batch: `protobuf::ResourceBounds { l1_gas: 1, l2_gas: 0, l1_data_gas: 0 }` → `ValidResourceBounds::L1Gas(1)` → `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` returns `DEPRECATED_RESOURCE_BOUNDS_ERROR`.
5. `validate_proposal` returns `HandledProposalPart::Failed`; the proposal is rejected; the round times out.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
