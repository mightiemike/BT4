### Title
Unbounded Iteration Over User-Inflatable `NlpLockedBalanceQueue` Permanently Locks NLP Tokens - (File: `core/contracts/SpotEngineState.sol`)

---

### Summary

`tryUnlockNlpBalance` in `SpotEngineState.sol` contains an unbounded `while` loop that iterates over a per-subaccount `NlpLockedBalanceQueue`. Any user can inflate their own queue to an arbitrary size by making NLP deposits at distinct oracle timestamps. Once the `NLP_LOCK_PERIOD` (4 days) elapses, all entries become eligible for processing in a single call, causing an out-of-gas revert that permanently blocks `burnNlp` and locks the attacker's NLP tokens.

---

### Finding Description

`NlpLockedBalanceQueue` is a per-subaccount data structure defined in `ISpotEngine`:

```solidity
struct NlpLockedBalanceQueue {
    mapping(uint64 => NlpLockedBalance) balances;
    uint64 balanceCount;
    uint64 unlockedUpTo;
    Balance unlockedBalanceSum;
}
``` [1](#0-0) 

In `SpotEngine.sol`, `handleNlpLockedBalance` appends a **new** entry to the queue whenever a deposit arrives at a timestamp not matching the last entry's `unlockedAt`:

```solidity
} else {
    queue.balances[queue.balanceCount] = NlpLockedBalance({
        balance: Balance({amount: amountDelta}),
        unlockedAt: getOracleTime() + NLP_LOCK_PERIOD
    });
    queue.balanceCount++;
}
``` [2](#0-1) 

There is no cap on `balanceCount`. A user who deposits NLP at N distinct oracle timestamps accumulates N entries.

`tryUnlockNlpBalance` in `SpotEngineState.sol` then iterates over all unlocked entries in a single unbounded `while` loop:

```solidity
while (
    queue.unlockedUpTo < queue.balanceCount &&
    queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
) {
    queue.unlockedBalanceSum.amount += queue.balances[queue.unlockedUpTo].balance.amount;
    delete queue.balances[queue.unlockedUpTo];
    queue.unlockedUpTo++;
}
``` [3](#0-2) 

`NLP_LOCK_PERIOD` is 4 days: [4](#0-3) 

After 4 days all N entries unlock simultaneously. The next invocation of `tryUnlockNlpBalance` must process all N entries in one transaction. With each iteration requiring cold SLOADs and SSTOREs (~10,000–20,000 gas each), ~1,500–3,000 entries suffice to exhaust the 30M block gas limit.

---

### Impact Explanation

`burnNlp` in `Clearinghouse.sol` calls `spotEngine.getNlpUnlockedBalance(txn.sender)` before every burn:

```solidity
require(
    spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
    ERR_UNLOCKED_NLP_INSUFFICIENT
);
``` [5](#0-4) 

`getNlpUnlockedBalance` unconditionally calls `tryUnlockNlpBalance`: [6](#0-5) 

Once the queue is large enough, every `burnNlp` call for that subaccount reverts with OOG. The user's NLP tokens are **permanently unredeemable** — they cannot be converted back to USDC. This is a direct, irreversible asset loss.

A secondary impact: `clearNlpPoolPosition` in `Clearinghouse.sol` iterates over all spot products and calls `updateBalance(NLP_PRODUCT_ID, subaccount, ...)` for the target subaccount: [7](#0-6) 

This triggers `handleNlpLockedBalance` → `tryUnlockNlpBalance`, which also OOGs, blocking the sequencer from clearing that subaccount's NLP pool position.

---

### Likelihood Explanation

- The attacker only needs to call `mintNlp` once per unique oracle timestamp over 4 days. At one deposit per minute, 5,760 entries are created — well above the OOG threshold.
- `MIN_DEPOSIT_AMOUNT` is $0.1, so the total capital required is negligible (~$576 for 5,760 deposits).
- No privileged access is required. Any user with a valid subaccount can execute this.
- The attack is irreversible once the queue is large enough and the lock period has elapsed.

---

### Recommendation

Replace the unbounded `while` loop in `tryUnlockNlpBalance` with a bounded batch that processes at most `MAX_UNLOCK_BATCH` entries per call, and expose a separate paginated unlock function. Additionally, enforce a hard cap on `balanceCount` in `handleNlpLockedBalance` (e.g., revert or merge aggressively if the queue exceeds a safe limit such as 256 entries).

---

### Proof of Concept

1. Attacker calls `mintNlp` once per oracle-time tick over 4 days, accumulating N ≥ 3,000 entries in `nlpLockedBalanceQueues[attacker]`.
2. After 4 days all entries satisfy `unlockedAt <= getOracleTime()`.
3. Attacker (or anyone) calls `burnNlp` for the attacker's subaccount.
4. `burnNlp` → `getNlpUnlockedBalance` → `tryUnlockNlpBalance` iterates 3,000 entries, each requiring cold SLOAD + SSTORE + delete ≈ 15,000 gas → ~45M gas total → OOG revert.
5. Every subsequent `burnNlp` call for this subaccount reverts identically. The attacker's NLP balance is permanently locked. [8](#0-7) [9](#0-8) [10](#0-9)
