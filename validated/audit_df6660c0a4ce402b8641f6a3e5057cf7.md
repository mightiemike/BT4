Audit Report

## Title
`SwapAllowlistExtension` Allowlist Bypassed via Router — Any User Can Swap on Allowlisted Pools Through `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to its own `msg.sender` at the time `swap` is called. When `MetricOmmSimpleRouter` calls the pool, the pool's `msg.sender` is the router, so the extension evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actualUser]`. Any pool admin who allowlists the router to enable router-mediated swaps for permitted users inadvertently grants swap access to every user of the router, fully voiding the per-address allowlist.

## Finding Description
**Root cause — pool passes its own `msg.sender` as `sender` to the extension:**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`:
```solidity
_beforeSwap(
    msg.sender,   // ← this is the router when called via MetricOmmSimpleRouter
    recipient, ...
);
```
(`metric-core/contracts/MetricOmmPool.sol`, lines 230–240)

`ExtensionCalling._beforeSwap` forwards that value unchanged as the `sender` argument to every registered extension (`metric-core/contracts/ExtensionCalling.sol`, lines 149–177).

`SwapAllowlistExtension.beforeSwap` then checks:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
(`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`, lines 37–39)

Here `msg.sender` is the pool (correct) and `sender` is the router (wrong — should be the end user).

**Router call path — router never passes the original caller:**

`MetricOmmSimpleRouter.exactInputSingle` calls the pool directly with no mechanism to convey the original `msg.sender` to the extension:
```solidity
IMetricOmmPoolActions(params.pool).swap(
    params.recipient, params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64, "", params.extensionData
);
```
(`metric-periphery/contracts/MetricOmmSimpleRouter.sol`, lines 72–80)

The same path exists for `exactOutputSingle` (lines 136–137), `exactInput` (lines 104–112), and `exactOutput` (lines 165–181). The `multicall` wrapper uses `delegatecall`, so `msg.sender` is preserved through multicall but the pool still sees the router as its caller.

**Exploit flow:**
1. Pool admin deploys pool with `SwapAllowlistExtension` as `beforeSwap` hook.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — alice is the only permitted swapper.
3. Admin calls `setAllowedToSwap(pool, router, true)` — to let alice use the router.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
5. Router calls `pool.swap(...)` → pool's `msg.sender = router` → `sender` forwarded to extension = router.
6. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
7. Bob receives output tokens from LP reserves without being allowlisted.

**Why existing guards fail:** The extension has no way to distinguish which end user initiated the router call. The `extensionData` field is caller-supplied and untrusted. There is no transient-storage slot set by the router that the extension could read to recover the true initiator.

**Contrast with `DepositAllowlistExtension`:** That extension gates on `owner` (a caller-supplied parameter distinct from `sender`), which the pool passes from the explicit function argument rather than from `msg.sender`, so it does not share this flaw (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`, lines 32–42).

## Impact Explanation
A pool configured with `SwapAllowlistExtension` is intended to restrict swapping to a curated set of addresses (e.g., KYC'd traders, institutional counterparties). LP providers deposit under the assumption that only those addresses can trade against their liquidity. Once the pool admin allowlists the router — the only way to let permitted users access the router UX — the allowlist is silently voided for every user of the router. Any address can execute swaps at oracle-derived bid/ask prices against the pool's LP reserves. LP principal is directly at risk: unauthorized traders can extract value from the pool at will. This constitutes a broken core swap-access invariant causing direct loss of LP principal, qualifying as a High/Medium finding under the Metric OMM Allowed Impact Gate (broken core pool functionality causing loss of funds; admin-boundary break bypassed by an unprivileged path).

## Likelihood Explanation
The trigger requires the pool admin to allowlist the router address in `SwapAllowlistExtension`. This is not a malicious or unusual action — it is the only way to let allowlisted users access the router. Any pool that deploys `SwapAllowlistExtension` and also wants router support will reach this configuration. The attacker needs no special role: a standard `exactInputSingle` call through the public router is sufficient. The misconfiguration is the natural, expected setup for any allowlisted pool that supports the periphery router, making this Medium likelihood.

## Recommendation
Pass the original caller through the swap path so the extension can gate on the actual user, not the intermediary. Two complementary fixes:

1. **In `MetricOmmSimpleRouter`**: store `msg.sender` in transient storage before calling the pool and expose it via a dedicated getter so the extension can read it during the `beforeSwap` hook.
2. **In `SwapAllowlistExtension`**: decode the actual initiator from `extensionData` (a router-attested payload written by the router into transient storage and read by the extension) rather than trusting the `sender` argument, which is always the immediate pool caller.

Alternatively, document that `SwapAllowlistExtension` gates the **direct pool caller** only, and require pools that use the router to deploy a custom extension that reads the true initiator from a trusted transient-storage slot set by the router.
