### Title
Circuit-Broken `MsgClaimTierRewards` Bypassed via Reward-Settling Sibling Messages — (File: `x/tieredrewards/keeper/msg_server.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

The `x/circuit` module's circuit breaker operates exclusively at the AnteHandler level, blocking transactions whose top-level message type URL is disabled. When an operator disables `MsgClaimTierRewards` to halt reward payouts (e.g., in response to a bonus-calculation bug), the identical reward-settlement code path (`claimRewards`) remains fully reachable through four unprivileged sibling messages — `MsgTierUndelegate`, `MsgTierRedelegate`, `MsgAddToTierPosition`, and `MsgClearPosition` — each of which calls `claimRewards` internally before performing its primary operation. Any position owner can drain the rewards pool and collect base rewards through these messages even while the direct claim path is circuit-broken.

---

### Finding Description

The `x/circuit` circuit breaker is wired into the AnteHandler pipeline via `circuitante.NewCircuitBreakerDecorator`: [1](#0-0) 

It checks the message type URL of every message in a transaction before execution. It does **not** intercept internal keeper-to-keeper calls made during message execution.

`MsgClaimTierRewards` is the direct reward-claiming message. Its handler calls `claimRewardsAndUpdatesPositions`, which in turn calls `claimBaseRewards` (→ `distributionKeeper.WithdrawDelegationRewards`) and `processEventsAndClaimBonus` (→ `bankKeeper.SendCoinsFromModuleToAccount` from `RewardsPoolName`): [2](#0-1) [3](#0-2) 

The following four unprivileged message handlers each call the **same** `claimRewards` function before performing their primary operation:

**`MsgTierUndelegate`** (line 166): [4](#0-3) 

**`MsgTierRedelegate`** (line 229): [5](#0-4) 

**`MsgAddToTierPosition`** (line 314): [6](#0-5) 

**`MsgClearPosition`** (line 406): [7](#0-6) 

None of these message type URLs are `/chainmain.tieredrewards.v1.MsgClaimTierRewards`, so the circuit breaker does not block them. The reward settlement is a side-effect of their execution, not their declared purpose, making it non-obvious to an operator that disabling `MsgClaimTierRewards` alone is insufficient.

The ADR explicitly documents this design:

> "Reward settlement before mutations: MsgTierUndelegate, MsgTierRedelegate, MsgAddToTierPosition, MsgClearPosition, and MsgExitTierWithDelegation all claim rewards before modifying the position." [8](#0-7) 

---

### Impact Explanation

The corrupted value is the `RewardsPoolName` module account balance (bonus rewards) and the per-position distribution rewards (base rewards). If an operator disables `MsgClaimTierRewards` to stop reward payouts during an emergency (e.g., a discovered over-payment bug in `processEventsAndClaimBonus`), every position owner can still:

1. Submit `MsgTierUndelegate` or `MsgTierRedelegate` — both pass the circuit breaker and internally execute the full `claimRewards` path, transferring bonus coins from the rewards pool to the owner and routing base rewards to the owner via the distribution withdraw-address.
2. Submit `MsgAddToTierPosition` with a dust amount — same effect.
3. Submit `MsgClearPosition` on any exiting position — same effect.

The rewards pool can be fully drained and base rewards fully collected despite the circuit break, defeating the operator's emergency response.

---

### Likelihood Explanation

**Medium.** The scenario requires the operator to have disabled `MsgClaimTierRewards` via the circuit breaker. This is a realistic emergency action if a bonus-calculation bug is discovered. The bypass is non-obvious: the sibling messages are documented as "position management" operations; their reward-settlement side-effect is a secondary concern. An operator following the natural response of "disable the claim message" would not know to also disable `MsgTierUndelegate`, `MsgTierRedelegate`, `MsgAddToTierPosition`, and `MsgClearPosition`. Any position owner (unprivileged) can exploit the bypass immediately.

---

### Recommendation

The `x/tieredrewards` message handlers that settle rewards as a side-effect should check whether reward settlement is currently permitted before calling `claimRewards`. One approach: introduce a module-level `RewardsEnabled` flag (governable or circuit-aware) that `claimRewards` checks at entry, returning zero rewards without transferring funds when disabled. Alternatively, the operator runbook must document that disabling reward claims requires disabling all five sibling message types simultaneously.

---

### Proof of Concept

1. User creates a tiered rewards position via `MsgLockTier`.
2. Operator discovers a bug in `processEventsAndClaimBonus` and disables `MsgClaimTierRewards` via `x/circuit`:
   ```
   chain-maind tx circuit disable /chainmain.tieredrewards.v1.MsgClaimTierRewards --from=admin
   ```
3. User submits `MsgTierRedelegate` (a different message type URL, not blocked by the circuit breaker):
   ```
   chain-maind tx tieredrewards tier-redelegate <position_id> <dst_validator> --from=owner
   ```
4. Inside `TierRedelegate`, `ms.claimRewards(ctx, pos)` executes at line 229 of `msg_server.go`, calling `claimBaseRewards` and `processEventsAndClaimBonus`, transferring bonus coins from `RewardsPoolName` to the owner and routing base rewards via distribution — identical to what `MsgClaimTierRewards` would have done.
5. The circuit break on `MsgClaimTierRewards` is fully bypassed; the rewards pool continues to be drained. [9](#0-8) [2](#0-1) [1](#0-0)

func (k Keeper) claimRewards(ctx context.Context, pos types.PositionState) (types.PositionState, sdk.Coins, sdk.Coins, error) {
	if !pos.IsDelegated() {
		return pos, sdk.NewCoins(), sdk.NewCoins(), nil
	}

	base, err := k.claimBaseRewards(ctx, pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	bonus, err := k.processEventsAndClaimBonus(ctx, &pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	return pos, base, bonus, nil
}
