### Title
Fast Withdrawal Fees Permanently Locked in `WithdrawPool` With No Recovery Path - (`File: core/contracts/BaseWithdrawPool.sol`)

### Summary
`BaseWithdrawPool` accumulates fast-withdrawal fees into a `fees` mapping but provides no function to withdraw or claim those tokens. Every call to `submitFastWithdrawal` by an unprivileged fast-withdrawal provider causes fee tokens to accumulate in the `WithdrawPool` contract balance with no protocol-defined path to recover them, permanently locking protocol revenue.

### Finding Description
In `BaseWithdrawPool.submitFastWithdrawal`, a fee is computed and credited to `fees[productId]`:

```solidity
// BaseWithdrawPool.sol L102-L111
int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

if (sendTo == msg.sender) {
    require(transferAmount > uint128(fee), "Fee larger than balance");
    transferAmount -= uint128(fee);   // fee stays in pool balance
} else {
    safeTransferFrom(token, msg.sender, uint128(fee));  // fee pulled into pool
}

fees[productId] += fee;
``` [1](#0-0) 

In both branches, the fee tokens end up held in the `WithdrawPool` contract's ERC-20 balance. The `fees` mapping is declared `public` and is only ever incremented — it is never decremented, and no function in `BaseWithdrawPool` or `WithdrawPool` reads it to release tokens to any recipient. [2](#0-1) 

The only token-exit functions in the contract are:

- `handleWithdrawTransfer` — sends tokens to a withdrawing user, not to a fee recipient. [3](#0-2) 
- `removeLiquidity` — owner-callable, but it operates on raw token balances with no reference to `fees`, meaning it would drain user liquidity alongside fees with no accounting separation. [4](#0-3) 

There is no `claimFees`, `withdrawFees`, or equivalent function anywhere in `BaseWithdrawPool.sol` or `WithdrawPool.sol`. [5](#0-4) 

### Impact Explanation
All fast-withdrawal fee revenue is permanently locked in the `WithdrawPool` contract. The `fees[productId]` accounting variable grows monotonically but the corresponding tokens can never be extracted through any protocol-sanctioned path. This is a direct analog to the RocketPool bug: tokens are sent to a contract that has no mechanism to recover "unaccounted" balances. The locked amount scales with protocol usage — every fast withdrawal executed by any liquidity provider contributes to the locked balance.

### Likelihood Explanation
High. `submitFastWithdrawal` is an unpermissioned external function callable by any fast-withdrawal provider. [6](#0-5) 
Every successful call increments `fees[productId]` and leaves the corresponding tokens stranded. No special conditions, governance actions, or attacker coordination are required — normal protocol operation continuously locks more funds.

### Recommendation
Add a `claimFees(uint32 productId, address recipient)` function (owner-gated) that reads `fees[productId]`, resets it to zero, and transfers the corresponding token amount to `recipient`. This mirrors the pattern used for sequencer fees in `Clearinghouse.claimSequencerFees`, which correctly drains the `FEES_ACCOUNT` balance and transfers it to `X_ACCOUNT`. [7](#0-6) 

### Proof of Concept
1. Fast-withdrawal provider calls `WithdrawPool.submitFastWithdrawal(idx, transaction, signatures)` for a valid withdrawal of 10,000 USDC.
2. `fastWithdrawalFeeAmount` returns, e.g., 5 USDC as the fee.
3. Either 5 USDC is deducted from the user's payout (staying in the pool) or pulled from the provider via `safeTransferFrom`.
4. `fees[productId] += 5e6` is recorded. [8](#0-7) 
5. Repeat for every fast withdrawal. `fees[productId]` grows; the 5 USDC per withdrawal accumulates in the contract.
6. No function exists to transfer these tokens out. The protocol cannot claim its fee revenue. The tokens are permanently locked in `WithdrawPool`.
