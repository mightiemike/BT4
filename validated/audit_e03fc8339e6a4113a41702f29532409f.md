### Title
Base Rewards Top-Up Ignores Validator Commission, Causing Systematic Under-Distribution to Stakers — (File: `x/tieredrewards/keeper/abci.go`)

---

### Summary

`topUpBaseRewards` computes the shortfall between `TargetBaseRewardsRate` and the fee-collector balance using only community tax as a deduction. It then distributes the top-up via `AllocateTokensToValidator`, which internally splits every allocation into validator commission and delegator rewards. Because commission is never subtracted from either the "default" estimate or the top-up itself, stakers receive `targetStakersRewardPerBlock × (1 − commission)` instead of `targetStakersRewardPerBlock`. The difference accrues to validators as commission, silently breaking the minimum-APY guarantee the module is designed to enforce.

---

### Finding Description

`topUpBaseRewards` in `x/tieredrewards/keeper/abci.go` runs every BeginBlocker when `TargetBaseRewardsRate > 0`.

**Step 1 — target is computed correctly** (lines 63–65):

```
targetStakersRewardPerBlock = totalBonded × targetBaseRewardsRate / blocksPerYear
```

**Step 2 — "default" staker reward is over-estimated** (lines 74–75):

```go
defaultStakersRewardPerBlock := math.LegacyNewDecFromInt(feeCollectorBalance.Amount).
    MulTruncate(math.LegacyOneDec().Sub(communityTax))
```

Only community tax is subtracted. The fee-collector balance will later be processed by `x/distribution`'s `AllocateTokens` (EndBlocker), which also deducts validator commission before delegators receive anything. The code therefore over-estimates what stakers will actually receive from the fee collector by a factor of `1 / (1 − commission)`.

**Step 3 — shortfall is under-estimated** (line 77):

```go
shortFallAmount := targetStakersRewardPerBlock.Sub(defaultStakersRewardPerBlock).TruncateInt()
```

Because `defaultStakersRewardPerBlock` is too high, `shortFallAmount` is too low.

**Step 4 — top-up is distributed through `AllocateTokensToValidator`** (line 139):

```go
err = k.distributionKeeper.AllocateTokensToValidator(ctx, validator, reward)
```

The Cosmos SDK's `AllocateTokensToValidator` splits every allocation into `commission = reward × validator.Commission.Rate` (credited to the validator) and `shared = reward − commission` (credited to delegators). So the top-up amount itself is further reduced by commission before reaching stakers.

**Net result:**

```
actual_staker_reward
  = feeCollector × (1 − communityTax) × (1 − commission)   [from fee collector]
  + shortFallAmount × (1 − commission)                       [from top-up]
  = (feeCollector × (1 − communityTax) + shortFallAmount) × (1 − commission)
  = targetStakersRewardPerBlock × (1 − commission)
```

Stakers receive `targetStakersRewardPerBlock × (1 − commission)` instead of `targetStakersRewardPerBlock`. The gap `targetStakersRewardPerBlock × commission` is silently redirected to validators as commission every block.

The test helper at line 172–182 of `keeper_helpers_test.go` explicitly notes that the genesis validator carries **100% commission by default** and tests must override it to zero to observe delegator rewards — confirming commission is a live, non-zero production parameter.

---

### Impact Explanation

The `TargetBaseRewardsRate` parameter is the module's core invariant: it guarantees a minimum staking APY for delegators. With any non-zero validator commission, that invariant is broken every block a shortfall top-up fires. The under-payment is proportional to commission rate and compounds over the entire lifetime of the pool. Rewards pool funds are consumed at the full `shortFallAmount` rate, but delegators receive only `shortFallAmount × (1 − commission)` — the remainder is permanently diverted to validators. This constitutes a concrete, measurable under-distribution of delegator rewards and premature depletion of the rewards pool relative to the intended schedule.

---

### Likelihood Explanation

High. The condition requires only:
1. `TargetBaseRewardsRate > 0` (governance-set parameter, expected to be non-zero when the module is active).
2. A shortfall exists (normal whenever fee revenue is below the target, which is the entire reason the top-up mechanism exists).
3. Validators have non-zero commission (standard in every production deployment; the default testnet genesis sets 10% commission per `x/chainmain/client/cli/testnet.go` line 335).

No privileged action, no special transaction, and no attacker is required. The BeginBlocker fires automatically on every block.

---

### Recommendation

Gross up the shortfall to account for commission, so that after commission is deducted the net delegator reward equals the target. Because commission rates differ per validator, the gross-up must be computed per-validator:

```
grossUpFactor = 1 / (1 − validator.Commission.Rate)
perValidatorTopUp = shortFallAmount × powerFraction × grossUpFactor
```

Alternatively, compute `defaultStakersRewardPerBlock` using the weighted-average post-commission rate across all bonded validators, so the shortfall already reflects what delegators will actually receive from the fee collector.

---

### Proof of Concept

Concrete numbers with a single validator at 10% commission, `communityTax = 2%`, `TargetBaseRewardsRate = 5%`, `totalBonded = 1,000,000`, `blocksPerYear = 6,311,520`, and an empty fee collector:

```
targetStakersRewardPerBlock = 1,000,000 × 0.05 / 6,311,520 ≈ 7.92 tokens

feeCollectorBalance = 0
defaultStakersRewardPerBlock = 0 × (1 − 0.02) = 0

shortFallAmount = 7.92 − 0 = 7.92 tokens   ← transferred from rewards pool

AllocateTokensToValidator(validator, 7.92):
  commission = 7.92 × 0.10 = 0.792 tokens  → validator
  delegators = 7.92 × 0.90 = 7.128 tokens  → stakers

Shortfall vs. target: 7.92 − 7.128 = 0.792 tokens per block
Over 1 year (6,311,520 blocks): 0.792 × 6,311,520 ≈ 4,998,724 tokens
  silently diverted to validators instead of stakers,
  while the rewards pool is drained at the full 7.92/block rate.
```

The entry path is `BeginBlocker → topUpBaseRewards` at [1](#0-0) , with the broken shortfall calculation at [2](#0-1)  and the commission-applying distribution call at [3](#0-2) . The `DistributionKeeper` interface confirms `AllocateTokensToValidator` is the only distribution path used for the top-up at [4](#0-3) . The ADR documents the intent as guaranteeing a minimum staker reward at [5](#0-4) .
