### Title
Duplicate Headers in `submit_blocks` Batch Inflate GC `batch_size`, Causing Mainchain Over-Pruning - (File: `contract/src/lib.rs`)

### Summary

`submit_blocks` accepts a `Vec<BlockHeader>` and counts all submitted headers (including duplicates) as `num_of_headers`, which is then passed directly to `run_mainchain_gc` as `batch_size`. Because there is no check that a header already exists in `headers_pool` before processing, a trusted relayer can pad a batch with repeated copies of a single valid header to inflate `num_of_headers` far beyond the number of blocks actually added. This causes `run_mainchain_gc` to prune far more mainchain blocks than warranted, potentially destroying the common-ancestor history needed for future chain-reorg resolution.

### Finding Description

In `submit_blocks` (lines 169–198 of `contract/src/lib.rs`):

```rust
let num_of_headers = headers.len().try_into().unwrap();   // line 175 — counts ALL entries

for header in headers {
    self.submit_block_header(header, self.skip_pow_verification);  // line 178
}

self.run_mainchain_gc(num_of_headers);   // line 181 — batch_size = inflated count
``` [1](#0-0) 

There is no guard anywhere in `submit_block_header` or `submit_block_header_inner` that checks whether `current_block_hash` is already present in `headers_pool` before proceeding. [2](#0-1) 

**What happens with a duplicate header in the same batch:**

- **First occurrence** of header H (whose `prev_block_hash` is the current tip T): `prev_block_header.block_hash == mainchain_tip_blockhash` → TRUE → H is stored on the mainchain; `mainchain_tip_blockhash` is updated to H.
- **Every subsequent occurrence** of the same H: `prev_block_header` is still T; `T.block_hash != mainchain_tip_blockhash` (tip is now H) → goes to the fork path; `store_fork_header` silently overwrites the `headers_pool` entry with identical data; `chain_work` equals the tip's `chain_work` so no reorg fires. [3](#0-2) 

Each duplicate is accepted without error, so `num_of_headers` equals the full padded length. `run_mainchain_gc` then uses this inflated value as `batch_size`:

```rust
let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);
``` [4](#0-3) 

**Concrete example:** `gc_threshold = 100`, mainchain has 150 blocks (heights 0–149). A relayer submits `[block_150, block_150, …, block_150]` — 1 real block plus 99 duplicates (100 total). After the loop, `amount_of_headers_we_store = 151`, `total_amount_to_remove = 51`. With `batch_size = 100`, `selected_amount_to_remove = min(51, 100) = 51` — 51 blocks pruned. With a legitimate single-block submission the same GC call would remove only `min(51, 1) = 1` block. The attacker causes 50 extra mainchain blocks to be deleted.

### Impact Explanation

`remove_block_header` deletes entries from both `mainchain_header_to_height` and `headers_pool`. [5](#0-4) 

`reorg_chain` walks the fork chain back to the common ancestor by following `prev_block_hash` pointers through `headers_pool`. If those ancestor blocks have been GC-pruned, the walk panics with `PrevBlockNotFound`. [6](#0-5) 

The contract's own documentation acknowledges this: *"If mainchain blocks near the fork point have been garbage collected, reorg will fail — the contract panics with `PrevBlockNotFound`."* Over-pruning caused by duplicate-inflated GC permanently prevents the contract from promoting a heavier fork to the mainchain, locking the contract to a lighter (incorrect) chain tip. This is a critical integrity failure in the canonical-chain tracking invariant.

### Likelihood Explanation

`submit_blocks` is gated by `#[trusted_relayer]`. The test suite confirms that an account without the role receives "Relayer is not active". [7](#0-6) 

However, the `trusted_relayer` macro is configured with a staking mechanism managed by `RelayerManager` ("May manage trusted relayer staking: reject applications and update relayer config"). [8](#0-7) 

If the staking entry is permissionless (anyone can stake to become an active relayer before a rejection is processed), a malicious actor can exploit this window. Additionally, `bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks)` means any account granted `UnrestrictedSubmitBlocks` — including a compromised or malicious relayer granted that role — can trigger the bug unconditionally. [9](#0-8) 

The attack requires only one valid PoW-passing block header (or `skip_pow_verification = true`) and the ability to pad the batch with copies of it. The exploit is deterministic and requires no cryptographic break.

### Recommendation

Before processing each header in the batch loop, check whether its hash is already present in `headers_pool` and skip (or reject) it:

```rust
let current_block_hash = header.block_hash(); // or equivalent
if self.headers_pool.contains_key(&current_block_hash) {
    continue; // or: env::panic_str("DuplicateHeader")
}
```

Alternatively, count only the headers that result in a new `headers_pool` insertion and pass that count to `run_mainchain_gc`, decoupling the GC batch size from the raw input length.

### Proof of Concept

1. Deploy the contract with `gc_threshold = 5` and submit 6 genesis-chain blocks so the mainchain is at the threshold boundary.
2. As a trusted relayer, call `submit_blocks` with a `Vec` of 100 entries all containing the same valid block header (block 7, whose `prev_block_hash` is the current tip).
3. Observe: `num_of_headers = 100`; only 1 block is actually added; `run_mainchain_gc(100)` removes `min(total_to_remove, 100)` blocks — far more than the 1 block added.
4. Attempt to submit a fork block that branches from a now-pruned ancestor; observe the contract panics with `PrevBlockNotFound`, confirming the mainchain history is unrecoverably corrupted.

    #[cfg(not(feature = "dogecoin"))]
    #[allow(clippy::needless_pass_by_value)]
    fn submit_block_header(&mut self, header: Header, skip_pow_verification: bool) {
        // We do not have a previous block in the headers_pool, there is a high probability
        // it means we are starting to receive a new fork,
        // so what we do now is we are returning the error code
        // to ask the relay to deploy the previous block.
        //
        // Offchain relay now, should submit blocks one by one in decreasing height order
        // 80 -> 79 -> 78 -> ...
        // And do it until we can accept the block.
        // It means we found an initial fork position.
        // We are starting to gather new fork from this initial position.
        #[allow(clippy::useless_conversion)]
        let prev_block_header = self.get_prev_header(&header.clone().into());
        let current_block_hash = header.block_hash();

        let (current_block_computed_chain_work, overflow) = prev_block_header
            .chain_work
            .overflowing_add(work_from_bits(header.bits));
        require!(!overflow, "Addition of U256 values overflowed");

        let current_header = ExtendedHeader {
            block_header: header.clone().into_light(),
            block_hash: current_block_hash,
            chain_work: current_block_computed_chain_work,
            block_height: 1 + prev_block_header.block_height,
        };

        if !skip_pow_verification {
            self.check_target(&header, &prev_block_header);

            let pow_hash = header.block_hash_pow();
            // Check if the block hash is less than or equal to the target
            require!(
                U256::from_le_bytes(&pow_hash.0) <= target_from_bits(header.bits),
                format!("block should have correct pow")
            );
        }

        self.submit_block_header_inner(current_header, &prev_block_header);
    }

    fn submit_block_header_inner(
        &mut self,
        current_header: ExtendedHeader,
        prev_block_header: &ExtendedHeader,
    ) {
        // Main chain submission
        if prev_block_header.block_hash == self.mainchain_tip_blockhash {
            // Probably we should check if it is not in a mainchain?
            // chainwork > highScore
            log!("Block {}: saving to mainchain", current_header.block_hash);
            // Validate chain
            assert_eq!(
                self.mainchain_tip_blockhash,
                current_header.block_header.prev_block_hash
            );

            self.store_block_header(&current_header);
            self.mainchain_tip_blockhash = current_header.block_hash;
        } else {
            log!("Block {}: saving to fork", current_header.block_hash);
            // Fork submission
            let main_chain_tip_header = self
                .headers_pool
                .get(&self.mainchain_tip_blockhash)
                .unwrap_or_else(|| env::panic_str("tip should be in a header pool"));

            let last_main_chain_block_height = main_chain_tip_header.block_height;
            let total_main_chain_chainwork = main_chain_tip_header.chain_work;

            self.store_fork_header(&current_header);

            // Current chainwork is higher than on a current mainchain, let's promote the fork
            if current_header.chain_work > total_main_chain_chainwork {
                log!("Chain reorg");
                self.reorg_chain(current_header, last_main_chain_block_height);
            }
        }
    }
