### Title
Unit Mismatch Between Limb Count and Byte Count in `op_multiply` Cost Accounting — (`File: src/more_ops.rs`)

### Summary

In `op_multiply`, the running size variable `l0` is initialized as a **byte count** (from `int_atom`) but is updated after each multiplication step using `limbs_for_int(&total)`, which returns the number of internal BigInt **limbs** (32-bit `u32` digits in `num_bigint`). All subsequent cost calculations and the intermediate-product size guard use `l0` as if it were still a byte count, causing systematic undercharging and an inflated effective size limit for intermediate products.

### Finding Description

`op_multiply` in `src/more_ops.rs` maintains `l0: usize` to track the byte-size of the accumulated product for cost purposes.

**First iteration** — `l0` is set from `int_atom`, which returns `a.atom_len(args)`, i.e., the raw byte length of the atom:

```rust
(total, l0) = int_atom(a, arg, "*")?;   // l0 = byte length
``` [1](#0-0) 

**Every subsequent iteration** — `l0` is overwritten with `limbs_for_int(&total)`:

```rust
l0 = limbs_for_int(&total);
``` [2](#0-1) 

`limbs_for_int` (defined at line 100) returns the number of 32-bit internal digits (`u32` limbs) of the `num_bigint::BigInt`. For a number whose canonical byte representation is N bytes, `limbs_for_int` returns approximately `⌈N/4⌉` — roughly **4× smaller** than the byte count.

The cost formula applied in every subsequent iteration uses `l0` and `l1` (always a byte count from `buf.len()` or `len_for_value`) as if they are both byte counts:

```rust
cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
``` [3](#0-2) 

The same mismatch applies in the `no-fastpath` compilation path: [4](#0-3) 

The intermediate-product size guard also uses the limb-count `l0`:

```rust
if l0 > 1024 {
    return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
}
``` [5](#0-4) 

Because `l0` is in limbs, this guard allows intermediate products up to `1024 × 4 = 4096` bytes, not 1024 bytes as the constant implies.

### Impact Explanation

For a 3-argument multiply `(* A B C)` where A and B are each 256 bytes:

- After `A * B`, the product can be up to 512 bytes. `limbs_for_int` returns ≈128 (limbs).
- The cost for the `B * product` step is computed with `l0 = 128` instead of `l0 = 512`.
- Linear cost undercharge: `(128 + 256) × 6 = 2304` vs correct `(512 + 256) × 6 = 4608` — **50% undercharge**.
- Quadratic cost undercharge: `(128 × 256) / 128 = 256` vs correct `(512 × 256) / 128 = 1024` — **75% undercharge**.

For programs with many arguments and growing intermediate products, the undercharge compounds multiplicatively. An attacker can craft a CLVM program that performs a chain of large-number multiplications and execute it for a fraction of the true computational cost, potentially bypassing the `max_cost` budget enforced by `check_cost`.

The size guard being 4× too permissive compounds the issue: intermediate products up to 4096 bytes are allowed, further amplifying the cost undercharge for later steps.

### Likelihood Explanation

The entry path is direct: any caller of `run_program` (including the Chia full node evaluating spend bundles) passes attacker-controlled CLVM bytes. A CLVM program invoking `(* large_a large_b large_c ...)` with 3 or more large-atom arguments is trivially constructable. No special privileges or configuration are required. The `no-fastpath` feature flag does not eliminate the bug; both compilation paths share the same `l0 = limbs_for_int(&total)` update.

### Recommendation

Replace `limbs_for_int(&total)` with a byte-count measurement of the accumulated product, consistent with how `l0` is initialized and how `l1` is always measured:

```rust
// After: total *= ...
l0 = (total.bits() as usize + 7) / 8;  // byte length, matching atom_len semantics
```

Alternatively, convert `l1` to limbs as well, but byte-count is the established unit throughout the rest of the cost model (all other operators use `atom_len` / `buf.len()`).

### Proof of Concept

Craft a CLVM program:
```
(* 0x<256 bytes of 0xff> 0x<256 bytes of 0xff> 0x<256 bytes of 0xff>)
```

- Step 1 cost (first × second): `l0=256, l1=256` → correct.
- Step 2 cost (product × third): `l0=limbs_for_int(product)≈128, l1=256` → undercharged by ~50% linear, ~75% quadratic.

The actual CPU work for step 2 is proportional to `512 × 256` byte-pairs, but the charged cost uses `128 × 256` limb-byte pairs. Extending to N arguments with growing products, the total undercharge grows with each step, allowing an attacker to exhaust a node's CPU budget while consuming far less than the declared `max_cost`. pub fn op_multiply(
    a: &mut Allocator,
    mut input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let mut cost: Cost = MUL_BASE_COST;
    let mut first_iter: bool = true;
    let mut total: Number = 1.into();
    let mut l0: usize = 0;
    while let Some((arg, rest)) = a.next(input) {
        input = rest;
        if first_iter {
            (total, l0) = int_atom(a, arg, "*")?;
            if l0 > 256 {
                return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
            }
            first_iter = false;
            continue;
        }

        cost += MUL_COST_PER_OP;
        #[cfg(not(feature = "no-fastpath"))]
        match a.node(arg) {
            NodeVisitor::Buffer(buf) => {
                let l1 = buf.len() as u64;
                if l1 > 256 {
                    return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
                }
                cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
                cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
                check_cost(cost, max_cost)?;

                total *= number_from_u8(buf);
            }
            NodeVisitor::U32(val) => {
                let l1 = len_for_value(val) as u64;
                cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
                cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
                check_cost(cost, max_cost)?;

                total *= val;
            }
            NodeVisitor::Pair(_, _) => {
                Err(EvalErr::InvalidOpArg(
                    arg,
                    "Requires Int Argument: *".to_string(),
                ))?;
            }
        }
        #[cfg(feature = "no-fastpath")]
        {
            let (n1, l1) = int_atom(a, arg, "*")?;
            let l1 = l1 as u64;
            if l1 > 256 {
                return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
            }
            cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
            cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
            check_cost(cost, max_cost)?;

            total *= n1;
        }
        l0 = limbs_for_int(&total);
        if l0 > 1024 {
            return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
        }
    }
    let total = a.new_number(total)?;
    Ok(malloc_cost(a, cost, total))
}
