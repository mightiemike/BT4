Audit Report

## Title
SwapAllowlistExtension Bypass via Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which equals `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the pool admin allowlists the router — the necessary step to let approved users trade through it — every unprivileged caller of the public router bypasses the allowlist gate entirely.

## Finding Description
The call chain is fully confirmed in production code:

**Step 1:** `MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

**Step 2:** `ExtensionCalling._beforeSwap` forwards that value unchanged as the first positional argument to every configured extension: [2](#0-1) 

**Step 3:** `SwapAllowlistExtension.beforeSwap` gates on `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the pool's caller: [3](#0-2) 

**Step 4:** `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the pool's `msg.sender` the router address, not the end user: [4](#0-3) 

The router does store the real user as the payer in transient storage (`_setNextCallbackContext(..., msg.sender, ...)`) for payment purposes only — this value is never forwarded to the extension as the `sender` argument and the `SwapAllowlistExtension` does not decode any user identity from `extensionData`. [5](#0-4) 

The same bypass applies to `exactOutputSingle`, `exactInput`, and `exactOutput` — all call `pool.swap()` with the router as `msg.sender`. [6](#0-5) 

## Impact Explanation
Any unprivileged user can swap in a pool restricted to a curated set of addresses by routing through the public `MetricOmmSimpleRouter`. This results in direct loss of LP principal at oracle-anchored prices, defeats KYC/compliance-gated pool designs, and allows explicitly excluded actors to drive pool state (bin position, fee accrual). This meets the contest's "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" impact criteria at High severity.

## Likelihood Explanation
The trigger is fully unprivileged: any EOA or contract can call `MetricOmmSimpleRouter.exactInputSingle`. The required precondition — the router being allowlisted — is the natural and expected admin action. Without allowlisting the router, approved users cannot use the router at all, making the allowlist and the router mutually exclusive. The admin is forced into the vulnerable configuration to make the system usable, making exploitation near-certain once a pool uses both features.

## Recommendation
The extension must verify the original user, not the immediate pool caller. Options:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it when `sender` is a known router address.
2. **Never allowlist the router as a blanket entry**: Require direct pool calls for allowlisted users and document this constraint explicitly in the extension's NatSpec. This is the simplest safe fix without code changes.
3. **Dedicated router with identity forwarding**: Extend the router to forward the real caller in a verifiable way (e.g., signed permit or dedicated router that re-encodes the caller into `extensionData`).

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  alice  → allowedSwapper[pool][alice]  = true   (legitimate user)
  router → allowedSwapper[pool][router] = true   (admin adds this so alice can use the router)
  bob    → allowedSwapper[pool][bob]   = false   (explicitly excluded)

Attack:
  bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(recipient, zeroForOne, amount, ...)
        pool.msg.sender = router
      → pool calls _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          allowedSwapper[pool][router] == true  ✓  (no revert)
      → swap executes for bob
      → pool calls router.metricOmmSwapCallback(...)
        → router pays from bob's tokens (payer stored in transient storage)

Result: bob swaps successfully despite being explicitly excluded.
        LP funds flow to bob at oracle-anchored prices; allowlist invariant is broken.
```

Foundry test plan: deploy a pool with `SwapAllowlistExtension`, allowlist `alice` and the router, confirm `bob` is not allowlisted, call `router.exactInputSingle` as `bob`, assert the swap succeeds without revert. [7](#0-6) [8](#0-7)
