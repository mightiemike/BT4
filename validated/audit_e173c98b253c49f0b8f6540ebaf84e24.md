### Title
v8 Migration Silently Skips Undelegated Vesting-Owned Positions, Leaving Stale `DelegatedVesting` That Enables Vesting Bypass — (File: x/tieredrewards/migrations/v2/migrate.go)

---

### Summary

The v8 upgrade migration's `exitVestedAccountsPositions` function calls `ForceFullExitWithDelegation` for every vesting-owned tier position. `ForceFullExitWithDelegation` silently returns `nil` — a no-op — when the position is not currently delegated. Positions that a vesting account owner undelegated (via `MsgTierUndelegate`) before the upgrade height are therefore silently skipped. For `CommitDelegationToTier`-origin positions this leaves the vesting account's `DelegatedVesting` field stale-high in `x/auth` state, which causes `LockedCoins` to be computed as zero. After unbonding completes the owner can call `MsgWithdrawFromTier` (which has no vesting-account guard) and receive the principal as fully spendable bank balance, bypassing the vesting schedule.

---

### Finding Description

**Step 1 — Migration walk collects undelegated positions.**

`exitVestedAccountsPositions` walks every position and appends vesting-owned ones to `toExit`: [1](#0-0) 

It does not distinguish between delegated and undelegated positions; all vesting-owned positions are enqueued.

**Step 2 — `ForceFullExitWithDelegation` silently no-ops on undelegated positions.**

For each enqueued ID the migration calls `pk.ForceFullExitWithDelegation`. The implementation immediately returns `nil` when `!posState.IsDelegated()`: [2](#0-1) 

No error is returned, no position is deleted, and — critically — `alignVestingDelegationTracking` is never called. The position survives the migration intact.

**Step 3 — Stale-high `DelegatedVesting` is never corrected.**

`alignVestingDelegationTracking` is the only place that reconciles `DelegatedVesting`/`DelegatedFree` after a force-exit: [3](#0-2) 

For a `CommitDelegationToTier`-origin position the owner originally delegated from their own address (setting `DelegatedVesting = lockedAmount`), then transferred that delegation to the position delegator address. After the transfer the owner's on-chain delegation is zero but `DelegatedVesting` remains at `lockedAmount` — stale-high. The migration was designed to fix this via `alignVestingDelegationTracking`, but for undelegated positions it is never reached.

**Step 4 — `MsgWithdrawFromTier` has no vesting-account guard.**

After unbonding completes the owner calls `MsgWithdrawFromTier`. `validateWithdrawFromTier` checks only ownership, exit-triggered, exit-elapsed, not-delegated, and no-pending-unbonding: [4](#0-3) 

There is no `validateNonVestingAccount` call. The handler then sends `SpendableCoins` from the position delegator address to the owner: [5](#0-4) 

The owner's bank balance increases by the returned principal. Because `DelegatedVesting` is still stale-high, `LockedCoins = max(0, VestingCoins − DelegatedVesting) = 0`, making the entire bank balance spendable.

---

### Impact Explanation

**Corrupted value:** `DelegatedVesting` in the `VestingAccount` stored in `x/auth`. It remains stale-high (equal to the original delegation amount) after the migration, causing `LockedCoins` to be computed as zero for a `PermanentLockedAccount` or any vesting account whose `VestingCoins(blockTime) ≤ DelegatedVesting`.

**Concrete outcome:** A vesting account that committed a delegation to a tier position in v7, triggered exit, undelegated before the v8 upgrade, and then calls `MsgWithdrawFromTier` post-upgrade receives the principal as fully spendable bank balance. Tokens that should remain locked by the vesting schedule become freely transferable.

---

### Likelihood Explanation

The preconditions are:
1. A vesting account created a `CommitDelegationToTier` position in v7 (allowed before the fix in the UNRELEASED changelog entry).
2. The account triggered exit and called `MsgTierUndelegate` before the v8 upgrade height.

Both steps are normal user actions reachable via standard Cosmos SDK transactions. The v7 module launched April 25, 2026; any user who started the exit process during the v7 window and whose unbonding period overlaps the upgrade height satisfies the preconditions. The attacker controls the timing entirely and needs no privileged access.

---

### Recommendation

In `exitVestedAccountsPositions`, after `ForceFullExitWithDelegation` returns `nil` for an undelegated position, explicitly handle the residual state. Two options:

1. **Delete the position and align vesting tracking directly** — call `alignVestingDelegationTracking` and `deletePosition` for undelegated vesting-owned positions, mirroring what `ForceFullExitWithDelegation` does for delegated ones.
2. **Extend `ForceFullExitWithDelegation`** — instead of returning `nil` for undelegated positions, perform the alignment and deletion so the caller does not need to know about the delegated/undelegated distinction.

Additionally, add a post-migration invariant check that no vesting-owned positions remain in the store.

func (k Keeper) validateWithdrawFromTier(ctx context.Context, pos types.PositionState, owner string) error {
	if !pos.IsOwner(owner) {
		return types.ErrNotPositionOwner
	}

	if !pos.HasTriggeredExit() {
		return types.ErrExitNotTriggered
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if !pos.CompletedExitLockDuration(sdkCtx.BlockTime()) {
		return types.ErrExitLockDurationNotReached
	}

	if pos.IsDelegated() {
		return types.ErrPositionDelegated
	}

	isUnbonding, err := k.isUnbonding(ctx, pos.DelegatorAddress)
	if err != nil {
		return err
	}
	if isUnbonding {
		return types.ErrPositionUnbonding
	}

	return nil
