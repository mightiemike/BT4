### Title
`DISABLE_OP` Flag Incorrectly Gates `op_modpow` in Mempool Mode, Creating Consensus Divergence — (`File: src/chia_dialect.rs`)

---

### Summary

In `src/chia_dialect.rs`, the `DISABLE_OP` flag (included in `MEMPOOL_MODE`) completely disables opcode 60 (`op_modpow`) in mempool mode by returning `EvalErr::Unimplemented`. However, `op_modpow` is a hardforked, fully valid operator in consensus mode. Any CLVM program using `op_modpow` with attacker-controlled bytes will be rejected by the mempool but accepted by consensus, producing a mempool-consensus divergence.

---

### Finding Description

`MEMPOOL_MODE` is defined as: [1](#0-0) 

It includes `ClvmFlags::DISABLE_OP`. In the operator dispatch inside `ChiaDialect::op()`, opcode 60 is handled as: [2](#0-1) 

When `DISABLE_OP` is set (i.e., in mempool mode), the dispatch immediately returns `Err(EvalErr::Unimplemented(o))` — `op_modpow` is completely unreachable.

In contrast, `op_div`, `op_divmod`, and `op_mod` use `DISABLE_OP` only to add an *additional* operand-size restriction (2048-byte cap), not to disable the operator entirely: [3](#0-2) [4](#0-3) [5](#0-4) 

`op_modpow` itself enforces its own operand-size limit (256 bytes per argument) independently of any flag: [6](#0-5) 

So the `DISABLE_OP` gate on `op_modpow` is not a size guard — it is a total, unconditional block on a hardforked operator. The flag name `DISABLE_OP` is undocumented and its semantics are inconsistent: it restricts size for `div`/`divmod`/`mod` but completely eliminates `modpow`.

---

### Impact Explanation

`op_modpow` (opcode 60) is a hardforked operator present in the main dispatch table without any "enable" flag guard (unlike `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`, `ENABLE_SHA256_TREE`, or `ENABLE_SECP_OPS`): [7](#0-6) 

In consensus mode (`DISABLE_OP` absent), `op_modpow` executes normally. In mempool mode (`DISABLE_OP` present), it returns `Unimplemented`. This is a **mempool-consensus divergence**:

- A coin puzzle using `op_modpow` with valid arguments (all ≤ 256 bytes) is **rejected** by the mempool.
- The same puzzle is **accepted** by consensus when included in a block.

Concretely: an attacker or user can craft a CLVM program using opcode `0x3c` (60) that is valid on-chain but cannot propagate through the mempool. This breaks the invariant that the mempool is a strict subset of consensus, and means any node running mempool validation will diverge from on-chain state for such transactions.

---

### Likelihood Explanation

The trigger is any CLVM program byte sequence that invokes opcode 60 with ≤ 256-byte arguments. This is directly attacker-controlled via the program bytes passed to `run_program`. No special privileges or configuration are required. The divergence is deterministic and reproducible.

---

### Recommendation

Replace the total `DISABLE_OP` block on `op_modpow` with the same pattern used for `op_div`/`op_divmod`/`op_mod`: apply an additional operand-size restriction in mempool mode rather than completely disabling the operator. If `op_modpow` is intentionally excluded from mempool acceptance, this must be documented and the consensus operator set must be adjusted to match, or the flag must be renamed and its semantics clearly specified.

---

### Proof of Concept

1. Construct a CLVM program invoking opcode 60 with three small atoms (e.g., base=3, exponent=10, modulus=7).
2. Run it via `run_program` with `ChiaDialect::new(MEMPOOL_MODE)` → returns `EvalErr::Unimplemented`.
3. Run the identical program with `ChiaDialect::new(ClvmFlags::empty())` → returns `Reduction(cost, atom(4))` (3^10 mod 7 = 4).

The divergence is confirmed by the existing test at: [8](#0-7) 

which passes `ClvmFlags::empty()` and `ClvmFlags::MALACHITE` but never `ClvmFlags::DISABLE_OP` — because with `DISABLE_OP` the call never reaches `op_modpow` at all.
