### Title
`op_unknown` Cost Limit Check Applied to Pre-Multiplied Cost, Allowing Returned Cost to Silently Exceed Per-Operator Budget — (File: `src/more_ops.rs`)

---

### Summary

In `op_unknown`, the `check_cost` guard is called on the **pre-multiplied** base cost, but the value actually returned to the caller is the **post-multiplied** cost (`cost * (cost_multiplier + 1)`). This is a direct arithmetic semantic mismatch: the enforcement check sees a value that is up to `(cost_multiplier + 1)` times smaller than the cost actually charged, allowing the returned cost to silently exceed the per-operator budget passed into the function.

---

### Finding Description

`op_unknown` in `src/more_ops.rs` computes a base cost from the opcode's cost function bits, then multiplies it by `(cost_multiplier + 1)` before returning. The critical ordering error is:

```rust
// line 260: check done on PRE-multiplied cost
check_cost(cost, max_cost)?;
// line 261: THEN multiply — returned value is now potentially >> max_cost
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))  // post-multiplied cost returned
}
``` [1](#0-0) 

The `max_cost` argument represents the **remaining execution budget** (`effective_max_cost - current_cost`) passed down from `run_program`'s main loop through `apply_op`. The `check_cost` call at line 260 validates only the pre-multiplied base cost (e.g., `1` for `cost_function = 0`), which trivially passes. The actual cost returned is `cost * (cost_multiplier + 1)`, which can be up to `u32::MAX` (4,294,967,295) — capped only by the overflow guard at line 262, not by the budget check.

The `cost_multiplier` is derived directly from the leading bytes of the opcode atom:

```rust
let cost_multiplier: u64 = match u32_from_u8(&op[0..op.len() - 1]) {
    Some(v) => v as u64,
    ...
};
``` [2](#0-1) 

This value is fully attacker-controlled via the opcode bytes of the CLVM program.

The outer `run_program` loop does eventually catch the excess cost on the **next iteration**:

```rust
if cost > effective_max_cost {
    return Err(EvalErr::CostExceeded);
}
``` [3](#0-2) 

But the per-operator guard — the `check_cost` inside `op_unknown` — is the intended early enforcement point. It is rendered ineffective because it checks the wrong (pre-multiplied) value.

---

### Impact Explanation

The per-operator cost budget enforcement in `op_unknown` is bypassed. A single unknown opcode invocation can return a cost up to `u32::MAX` (~4.3 billion) regardless of the remaining budget, because the internal guard only sees the pre-multiplied base cost. The outer loop still rejects the program with `CostExceeded`, so execution does not proceed past the limit — but the per-operator cost accounting diverges from what the internal check enforced. In the softfork context, where `effective_max_cost` is set to `guard.expected_cost`, this mismatch can cause the softfork cost validation (`SoftforkCostMismatch`) to fire in unexpected ways, since the accumulated cost jumps discontinuously past the expected boundary.

---

### Likelihood Explanation

Unknown opcodes are valid and reachable in consensus/lenient mode (`allow_unknown_ops() == true`), which is the standard production path for Chia full nodes evaluating puzzle reveals. The opcode byte sequence is entirely attacker-controlled CLVM input. No special privileges are required — any submitted transaction can include an unknown opcode with a crafted multiplier byte prefix. The trigger is deterministic and requires only a few bytes of crafted opcode data.

---

### Recommendation

Move `check_cost` to **after** the multiplication, so it validates the actual cost that will be returned:

```rust
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    return Err(EvalErr::Invalid(o))?;
}
check_cost(cost, max_cost)?;   // now checks post-multiplied cost
Ok(Reduction(cost as Cost, allocator.nil()))
```

---

### Proof of Concept

Craft an unknown opcode atom with:
- Last byte `= 0x00` → `cost_function = 0` (constant base cost = 1), lower 6 bits ignored
- Leading bytes encoding `cost_multiplier = 0xFFFFFFFE` (u32::MAX − 1)

With `cost_function = 0`, the base cost is `1`. `check_cost(1, max_cost)` passes for any `max_cost >= 1`. After multiplication: `cost = 1 * (0xFFFFFFFE + 1) = 0xFFFFFFFF = 4,294,967,295`. This is ≤ `u32::MAX`, so the overflow guard does not fire. The function returns `Reduction(4_294_967_295, nil)` — a cost ~4.3 billion — regardless of the remaining budget that was passed in. The outer loop catches it on the next iteration, but the per-operator guard was completely bypassed. [1](#0-0) [4](#0-3)
