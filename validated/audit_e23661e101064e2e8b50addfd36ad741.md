Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of originating user, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which `MetricOmmPool.swap()` sets to its own `msg.sender`. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the originating user. A pool admin who allowlists the router to enable router-mediated swaps for their approved users inadvertently opens the gate to every caller of the router, including those explicitly excluded from the allowlist.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` verbatim as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // router address when called via MetricOmmSimpleRouter
    recipient,
    ...
    extensionData
);
```

`SwapAllowlistExtension.beforeSwap` then checks that value against its per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` stores the original `msg.sender` only in transient storage for the payment callback and never forwards it to the pool as the swap initiator:

```solidity
// MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
);
```

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. The original user's address is never reachable by the extension. The only address the extension can check is the router's address. If the admin allowlists the router (the only way to permit any router-mediated swap), every caller of the router passes the check regardless of individual allowlist status. ## Impact Explanation

Any user — including those the pool admin explicitly excluded — can execute swaps in a restricted pool by routing through `MetricOmmSimpleRouter`. This directly violates the access-control invariant of `SwapAllowlistExtension`: the pool receives and pays out tokens for trades that should have been blocked. In pools where the allowlist enforces compliance, counterparty restrictions, or rate limits, the bypass enables unrestricted trading against the pool's liquidity, causing direct loss of LP principal through arbitrage or directional flow that the allowlist was designed to prevent. This meets the "broken core pool functionality causing loss of funds" and "admin-boundary break" impact criteria. [4](#0-3) 

## Likelihood Explanation

The trigger requires no special privilege. Any public user can call `MetricOmmSimpleRouter.exactInputSingle`. The only precondition is that the pool admin has allowlisted the router — a natural and expected administrative action when the admin wants their approved users to access the router. Once the router is allowlisted, the bypass is immediately reachable by any caller. The router is a deployed, public contract. [5](#0-4) 

## Recommendation

Pass the original user's address through the swap path so the extension can gate on the economically relevant actor. The preferred fix is pool-side: add an optional `originator` field to the swap call that the pool passes to extensions, set by the router to `msg.sender` before calling the pool. Alternatively, the router can encode the original `msg.sender` into `extensionData` and `SwapAllowlistExtension.beforeSwap` can decode it when `sender` is a known router. A registry of trusted routers in the extension or factory would be required for the extension-side approach to be safe. [6](#0-5) 

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured in `BEFORE_SWAP_ORDER`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to allow router-mediated swaps.
3. Pool admin does **not** call `setAllowedToSwap(pool, attacker, true)`.
4. Attacker calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(...)` — `msg.sender` at the pool is the router address.
6. `_beforeSwap(router, ...)` → `SwapAllowlistExtension.beforeSwap(router, ...)` → `allowedSwapper[pool][router] == true` → passes.
7. Attacker's swap executes against pool liquidity despite never being individually allowlisted.

Foundry test: deploy pool with extension, set `allowedSwapper[pool][router] = true`, call `exactInputSingle` from an address not in the allowlist, assert the swap succeeds and tokens are transferred. [7](#0-6) [8](#0-7)
