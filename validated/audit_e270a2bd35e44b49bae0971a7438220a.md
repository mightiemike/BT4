### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables USDCe Theft from DirectDepositV1 — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(...)` directly without checking its return value, while every other transfer in the codebase routes through `ERC20Helper.safeTransferFrom`. If the USDC token at the hardcoded address returns `false` on failure instead of reverting, any unprivileged caller can drain USDCe from a `DirectDepositV1` account without providing any USDC.

---

### Finding Description

`ContractOwner.replaceUsdcEWithUsdc` is a public migration function (gated only by `block.chainid == 57073`) that is intended to swap USDCe held in a `DirectDepositV1` (DDA) for USDC provided by the caller:

```solidity
// core/contracts/ContractOwner.sol  lines 608-620
function replaceUsdcEWithUsdc(bytes32 subaccount) external {
    require(block.chainid == 57073, ERR_UNAUTHORIZED);
    ...
    uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
    if (balance > 0) {
        IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // ← raw call, return value ignored
        DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
        IERC20Base(usdcE).safeTransfer(msg.sender, balance);
    }
}
```

The raw `.transferFrom(...)` at line 616 does **not** use `ERC20Helper.safeTransferFrom`, which is the wrapper used consistently everywhere else in the protocol:

```solidity
// core/contracts/libraries/ERC20Helper.sol  lines 23-42
function safeTransferFrom(...) internal {
    (bool success, bytes memory data) = address(self).call(...);
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
```

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` (Ink Mainnet) returns `false` on a failed transfer instead of reverting, execution continues past line 616 even though no USDC was received. The subsequent steps — withdrawing USDCe from the DDA to `ContractOwner`, then forwarding it to `msg.sender` — proceed unconditionally.

The three-step sequence becomes:
1. **Step 1 (line 616):** USDC `transferFrom` silently fails → no USDC enters the DDA.
2. **Step 2 (line 617):** `DirectDepositV1.withdraw(usdcE)` sends all USDCe from the DDA to `ContractOwner`.
3. **Step 3 (line 618):** `safeTransfer(msg.sender, balance)` sends that USDCe to the attacker.

The attacker receives USDCe without paying anything.

---

### Impact Explanation

Direct theft of all USDCe tokens held in any `DirectDepositV1` account. The corrupted state delta is: USDCe balance of the DDA goes to zero; the attacker's USDCe balance increases by `balance`; no USDC is deposited in exchange. This is a concrete, irreversible asset loss for the protocol or its users whose funds sit in DDAs awaiting migration.

---

### Likelihood Explanation

The function is callable by any address on Ink Mainnet (chain ID 57073) — no role, signature, or ownership check beyond the chain ID guard. The trigger requires only that the hardcoded USDC token returns `false` on a failed transfer rather than reverting. Standard Circle USDC reverts, but the specific deployment at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink Mainnet is a non-canonical chain where token behavior is not guaranteed to match mainnet USDC. The code inconsistency (raw call vs. `safeTransferFrom` everywhere else) is the root cause regardless of token behavior.

---

### Recommendation

Replace the raw `transferFrom` with the protocol's own safe wrapper:

```solidity
// Before (vulnerable):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ContractOwner` already imports `ERC20Helper` and declares `using ERC20Helper for IERC20Base`, so this is a one-word fix consistent with the rest of the codebase.

---

### Proof of Concept

**Precondition:** A DDA for `subaccount` holds `N` USDCe tokens. The USDC token at the hardcoded address returns `false` on a failed `transferFrom` (e.g., caller has no USDC or has not approved `ContractOwner`).

1. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` with zero USDC balance or zero allowance.
2. `balance = IERC20Base(usdcE).balanceOf(directDepositV1)` → `N`.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, N)` → returns `false`, no revert, no USDC transferred.
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` → DDA sends `N` USDCe to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, N)` → `ContractOwner` sends `N` USDCe to attacker.
6. Attacker holds `N` USDCe; DDA holds 0 USDCe; no USDC was ever deposited. [1](#0-0) [2](#0-1) [3](#0-2)

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
