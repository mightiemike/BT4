### Title
Deprecated `verify_transaction_inclusion` Accepts Internal Merkle Nodes as Valid Transaction IDs, Enabling SPV Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` remains publicly callable by any unprivileged NEAR account despite being deprecated. It performs no check that the caller-supplied `tx_id` is a leaf node (a real transaction) rather than an internal Merkle tree node. An attacker can craft a proof where `tx_id` is an internal node, causing the function to return `true` for a transaction that was never included in any Bitcoin block. This is the on-chain analog of the AES-CTR malleability class: data is accepted and "verified" without an integrity check that would distinguish legitimate inputs from forged ones.

---

### Finding Description

`verify_transaction_inclusion` is a public, `#[pause]`-gated function with no access-control role restriction: [1](#0-0) 

It accepts four fully attacker-controlled fields — `tx_id`, `tx_block_blockhash`, `tx_index`, and `merkle_proof` — and delegates to `compute_root_from_merkle_proof`: [2](#0-1) 

`compute_root_from_merkle_proof` treats whatever hash is passed as `transaction_hash` as the starting leaf and hashes upward through the proof path: [3](#0-2) 

There is no check anywhere in this path that `tx_id` is a 32-byte leaf hash corresponding to a real serialized transaction rather than a 32-byte hash of two concatenated child nodes (an internal node). The code's own inline warning acknowledges this gap explicitly: [4](#0-3) 

Rust's `#[deprecated]` attribute generates only a compiler-time warning; it imposes no runtime restriction. The function remains fully reachable on-chain by any NEAR account.

**Attack path (64-byte internal-node forgery):**

1. Attacker identifies a confirmed Bitcoin block already stored in the contract's `headers_pool` (its `merkle_root` is on-chain and public).
2. Attacker selects any internal node `N` in that block's Merkle tree. `N = double_sha256(left_child || right_child)` is a valid 32-byte value.
3. Attacker constructs a shorter `merkle_proof` (fewer hops) that leads from `N` upward to the known `merkle_root`, and picks a `tx_index` consistent with `N`'s position.
4. Attacker calls `verify_transaction_inclusion` with `tx_id = N`, the real `tx_block_blockhash`, the crafted `tx_index`, and the crafted `merkle_proof`.
5. `compute_root_from_merkle_proof(N, tx_index, proof)` produces the correct `merkle_root`; the function returns `true`.

The only guard present is: [5](#0-4) 

This rejects an empty proof but does not prevent a structurally valid proof rooted at an internal node.

---

### Impact Explanation

Any recipient NEAR contract that calls `verify_transaction_inclusion` to gate a privileged action (e.g., releasing bridged funds, minting tokens, unlocking collateral) will receive `true` for a Bitcoin transaction that does not exist. The attacker needs no Bitcoin private keys, no NEAR privileged role, and no access to the relayer — only knowledge of a stored block's Merkle tree structure, which is derivable from public Bitcoin data.

The corrupted invariant is the SPV proof result: the contract asserts `tx_id ∈ block` when `tx_id` is not a transaction at all.

---

### Likelihood Explanation

The 64-byte transaction Merkle forgery is a well-documented Bitcoin protocol-level attack (referenced in the contract's own deprecation notice pointing to the BitMEX blog post). The required inputs — a stored block hash and its Merkle tree — are entirely public. No privileged access, leaked keys, or social engineering is required. The only barrier is that the attacker must know to call the deprecated endpoint rather than v2, which is trivially discoverable from the on-chain ABI.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely. Because `#[deprecated]` provides no runtime enforcement, the function is indistinguishable from any other public method to an on-chain caller. If backward compatibility is required, add an explicit `env::panic_str("use verify_transaction_inclusion_v2")` body, or gate the function with an access-control role that no external account holds.

All callers should migrate to `verify_transaction_inclusion_v2`, which adds the coinbase-proof constraint: [6](#0-5)

### Proof of Concept

```
# Precondition: block B is stored in the contract with known merkle_root R.
# Attacker derives internal node N from B's public Merkle tree.
# N = double_sha256(left_child_bytes || right_child_bytes)
# Attacker constructs proof P such that compute_root_from_merkle_proof(N, idx, P) == R.

near call <contract> verify_transaction_inclusion '{
  "tx_id":             "<hex of internal node N>",
  "tx_block_blockhash":"<hex of block B hash>",
  "tx_index":          <idx>,
  "merkle_proof":      ["<proof hash 1>", ...],
  "confirmations":     1
}' --accountId attacker.near
# Returns: true
# N is not a real Bitcoin transaction.
```

                args.coinbase_tx_id.clone(),

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
