Audit Report

## Title
Router-Mediated Swaps Replace End-User Identity with Router Address in `SwapAllowlistExtension.beforeSwap`, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmSimpleRouter` calls `MetricOmmPool.swap` directly, making the pool see the router as `msg.sender`. The pool passes this `msg.sender` (the router address) as the `sender` argument to `SwapAllowlistExtension.beforeSwap`. The extension then checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][originating_user]`, meaning any unprivileged user can bypass a per-user swap allowlist by routing through the router if the router is allowlisted.

## Finding Description

**Step 1 — Router calls pool directly as `msg.sender`:**
`MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` at line 72. The pool sees the router contract as `msg.sender`.

**Step 2 — Pool passes `msg.sender` (= router) as `sender` to the hook:**
`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at line 230-231. `msg.sender` here is the router address.

**Step 3 — `ExtensionCalling._beforeSwap` forwards `sender` (= router) to the extension:**
At `ExtensionCalling.sol` lines 149-177, the `sender` argument (router address) is encoded into the `abi.encodeCall` for `IMetricOmmExtensions.beforeSwap`.

**Step 4 — `SwapAllowlistExtension.beforeSwap` checks the router, not the user:**
```solidity
function beforeSwap(address sender, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```
- `msg.sender` = pool address (correct)
- `sender` = **router address** (wrong — should be the originating EOA)

The check evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The `bytes calldata extensionData` parameter is unnamed and entirely ignored by the extension — there is no fallback mechanism to recover the originating user.

**Existing guards are insufficient:** The `onlyPool` modifier in `BaseMetricExtension` only verifies the caller is a registered pool, which is correct and expected. There is no guard that validates the `sender` argument represents the true originating user. The `allowedSwapper` mapping is set only by `onlyPoolAdmin` and is keyed by `[pool][swapper]`, but the swapper identity delivered to the check is structurally wrong for router-mediated calls.

## Impact Explanation
This breaks the core access-control invariant of `SwapAllowlistExtension`. Any pool configured with this extension to gate swaps to KYC'd or authorized users is fully exposed: an unprivileged user can trade in the curated pool by routing through `MetricOmmSimpleRouter`. This constitutes broken core pool functionality — the allowlist gate is rendered ineffective for all router-mediated swaps, which is the standard periphery entrypoint.

## Likelihood Explanation
The router is the standard, documented periphery entrypoint. Any pool that (a) uses `SwapAllowlistExtension` and (b) allowlists the router — a natural and expected configuration for KYC'd pools — is fully exposed. No privileged access, malicious setup, or special conditions are required. Any user can exploit this by calling the public router functions (`exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`).

## Recommendation
The router should encode `msg.sender` (the originating user) into `extensionData` before calling the pool. `SwapAllowlistExtension.beforeSwap` should decode and use this value when the direct `sender` is a known router. Alternatively, the pool's hook interface should carry a separate `origin` field distinct from `sender`, passed explicitly through the call chain, analogous to `tx.origin` but verifiable. A simpler short-term fix: `SwapAllowlistExtension` could fall back to checking `tx.origin` when `sender` is a known router, though this is less robust.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` as a `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` — allowlisting the router so KYC'd users can trade via periphery.
3. Pool admin calls `setAllowedToSwap(pool, alice, true)` — alice is the only intended user.
4. Bob (not allowlisted, `allowedSwapper[pool][bob] == false`) calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the pool.
5. Pool calls `_beforeSwap(router, ...)` → extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. Bob successfully swaps in a pool he was never authorized to access.

Foundry test assertion: `assertEq(allowedSwapper[pool][bob], false)` yet Bob's swap succeeds because `allowedSwapper[pool][router] == true` and the hook never inspects Bob's address.
