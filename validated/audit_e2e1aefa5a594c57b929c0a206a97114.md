### Title
`paid_fee_on_l1` Silently Dropped and Replaced with Hardcoded `Fee(1)` During `ConsensusTransaction::L1Handler` → `InternalConsensusTransaction::L1Handler` Conversion — (`File: crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

The `ConsensusTransaction::L1Handler` variant carries a `transaction::L1HandlerTransaction` that has no `paid_fee_on_l1` field. The protobuf wire format (`L1HandlerV0`) also omits this field. When a validator node converts an incoming consensus L1Handler transaction to its internal executable form, `convert_consensus_l1_handler_to_internal_l1_handler` unconditionally substitutes `Fee(1)` for the actual fee paid on L1. This is the exact structural analog of the external report: a field that must be propagated through a specific conversion path is silently dropped and replaced with a wrong constant, causing the downstream executable object to carry an incorrect value.

---

### Finding Description

**Conversion path:**

`ConsensusTransaction::L1Handler` (protobuf P2P) → `convert_consensus_tx_to_internal_consensus_tx` → `convert_consensus_l1_handler_to_internal_l1_handler` → `executable_transaction::L1HandlerTransaction::create(..., Fee(1))`

The `ConsensusTransaction` enum holds `transaction::L1HandlerTransaction` (the "public" type), which has no `paid_fee_on_l1` field:

```rust
// crates/starknet_api/src/consensus_transaction.rs
pub enum ConsensusTransaction {
    RpcTransaction(RpcTransaction),
    L1Handler(transaction::L1HandlerTransaction),  // no paid_fee_on_l1
}
```

The protobuf serializer for `ConsensusTransaction::L1Handler` converts to `protobuf::L1HandlerV0`, which also has no `paid_fee_on_l1` field — the field is structurally absent from the wire format:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs
impl From<L1HandlerTransaction> for protobuf::L1HandlerV0 {
    fn from(value: L1HandlerTransaction) -> Self {
        Self {
            nonce: Some(value.nonce.0.into()),
            address: Some(value.contract_address.into()),
            entry_point_selector: Some(value.entry_point_selector.0.into()),
            calldata: value.calldata.0.iter().map(|calldata| (*calldata).into()).collect(),
            // paid_fee_on_l1 is not present
        }
    }
}
```

When the validator reconstructs the internal executable transaction, the converter hardcodes `Fee(1)`:

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs:473-483
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
```

The `executable_transaction::L1HandlerTransaction` struct that reaches the blockifier therefore always carries `paid_fee_on_l1 = Fee(1)`, regardless of the actual fee paid on L1:

```rust
// crates/starknet_api/src/executable_transaction.rs:380-384
pub struct L1HandlerTransaction {
    pub tx: crate::transaction::L1HandlerTransaction,
    pub tx_hash: TransactionHash,
    pub paid_fee_on_l1: Fee,   // always Fee(1) on validator path
}
```

The blockifier's fee check in `execute_raw` reads this field directly:

```rust
// crates/blockifier/src/transaction/l1_handler_transaction.rs:103-113
let paid_fee = self.paid_fee_on_l1;
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee {
            paid_fee,
            actual_fee: receipt.fee,
        },
    )));
}
```

The check only gates on `Fee(0)`. Because the validator always injects `Fee(1)`, the check always passes on the validator side, even if the actual `paid_fee_on_l1` carried by the proposer's original L1 event was `Fee(0)`.

The proposer, by contrast, builds the `executable_transaction::L1HandlerTransaction` from the L1 events scraper, which reads the real `paid_fee_on_l1` from the Ethereum event. The `CentralL1HandlerTransaction` blob serialized by the proposer correctly includes the real fee:

```rust
// crates/apollo_consensus_orchestrator/src/cende/central_objects.rs:383-393
impl From<L1HandlerTransaction> for CentralL1HandlerTransaction {
    fn from(tx: L1HandlerTransaction) -> CentralL1HandlerTransaction {
        CentralL1HandlerTransaction {
            ...
            paid_fee_on_l1: tx.paid_fee_on_l1,  // correct on proposer
        }
    }
}
```

---

### Impact Explanation

**Impact: High**

This is a transaction conversion boundary where the executable payload carries the wrong field value. The divergence between proposer and validator is:

| Node | `paid_fee_on_l1` | Fee check result |
|---|---|---|
| Proposer | Actual value from L1 event (e.g., `Fee(0)`) | May fail with `InsufficientFee` |
| Validator | Hardcoded `Fee(1)` | Always passes |

If an L1Handler transaction is included in a proposal with `paid_fee_on_l1 = Fee(0)` (a zero-fee message), the proposer's blockifier returns `TransactionExecutionError::TransactionFeeError(InsufficientFee)` and the transaction is reverted. The validator's blockifier, using `Fee(1)`, passes the check and commits the state change. This is a consensus execution divergence: the proposer and validator produce different state roots for the same block, which breaks the consensus invariant that all honest nodes must agree on the execution result.

Additionally, the `paid_fee_on_l1` field is part of the executable transaction object that is passed to the blockifier and recorded in the execution context. Any downstream component that reads `paid_fee_on_l1` from the executed transaction (e.g., for receipt generation, OS hints, or proof inputs) will receive `Fee(1)` on the validator path instead of the real value.

---

### Likelihood Explanation

**Likelihood: Low**

In practice, the Starknet L1 core contract requires a non-zero ETH value to be sent with `sendMessageToL2`, so `paid_fee_on_l1 = Fee(0)` is prevented at the L1 contract level. The consensus divergence scenario therefore requires either a bug in the L1 contract or a future change to the fee enforcement logic. However, the broken invariant is structural and will silently produce wrong `paid_fee_on_l1` values in all validator-side executions of L1Handler transactions, regardless of the actual fee.

---

### Recommendation

1. Add `paid_fee_on_l1` to the `ConsensusTransaction::L1Handler` variant (or to a wrapper type) so the field is carried through the consensus P2P path.
2. Add `paid_fee_on_l1` to the `protobuf::L1HandlerV0` (or `protobuf::ConsensusTransaction`) wire format so it survives serialization/deserialization.
3. Remove the `Fee(1)` placeholder in `convert_consensus_l1_handler_to_internal_l1_handler` and use the real value from the deserialized transaction.
4. Add a round-trip test asserting that `paid_fee_on_l1` is preserved through the `ConsensusTransaction` → protobuf → `InternalConsensusTransaction` conversion.

---

### Proof of Concept

**Step 1.** An L1Handler transaction with `paid_fee_on_l1 = Fee(0)` is scraped from L1 and injected into the proposer's batcher.

**Step 2.** The proposer includes it in a block proposal and broadcasts it via consensus P2P as a `ConsensusTransaction::L1Handler`. The protobuf encoding uses `L1HandlerV0`, which omits `paid_fee_on_l1`.

**Step 3.** The validator receives the protobuf message and calls `convert_consensus_tx_to_internal_consensus_tx`:

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs:197-199
ConsensusTransaction::L1Handler(tx) => {
    let internal_tx = self.convert_consensus_l1_handler_to_internal_l1_handler(tx)?;
    Ok((InternalConsensusTransaction::L1Handler(internal_tx), None))
}
```

**Step 4.** `convert_consensus_l1_handler_to_internal_l1_handler` injects `Fee(1)`:

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs:477-482
Ok(executable_transaction::L1HandlerTransaction::create(
    tx,
    &self.chain_id,
    Fee(1),   // wrong: actual is Fee(0)
)?)
```

**Step 5.** The proposer's blockifier executes with `paid_fee_on_l1 = Fee(0)` → `InsufficientFee` error → transaction reverts → state unchanged.

**Step 6.** The validator's blockifier executes with `paid_fee_on_l1 = Fee(1)` → fee check passes → state committed → different state root.
