### Title
Reward Payment Embedded in Critical Exit Operations Blocks User Undelegation When Bonus Pool Is Insufficient — (`File: x/tieredrewards/keeper/msg_server.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

In `x/tieredrewards`, the `claimRewards` call is a mandatory prerequisite inside `TierUndelegate`, `TierRedelegate`, `AddToTierPosition`, `ClearPosition`, and `ExitTierWithDelegation`. The reward-payment path inside `claimRewards` calls `SendCoinsFromModuleToAccount` after a `sufficientBonusPoolBalance` guard. If the bonus pool is insufficient to cover accrued rewards, the guard returns an error that propagates all the way up, causing every one of those critical operations to revert. Users with accrued bonus rewards cannot undelegate, redelegate, or exit their positions — their principal remains locked in the position delegator account until the pool is externally refilled.

---

### Finding Description

`processEventsAndClaimBonus` in `claim_rewards.go` computes accrued bonus rewards and, when non-zero, performs two sequential steps:

```go
// x/tieredrewards/keeper/claim_rewards.go  lines 230-241
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err
}
...
if err := k.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.RewardsPoolName, ownerAddr, bonusCoins); err != nil {
    return nil, err
}
``` [1](#0-0) 

Both errors are returned directly to the caller. `processEventsAndClaimBonus` is called by `claimRewards`, which is called as a hard prerequisite in every critical position-mutation message handler:

None of these handlers have a fallback path that skips the reward payment and proceeds with the structural operation (undelegation, redelegation, exit). The reward transfer and the exit are atomically coupled with no separation of concerns.

The analogous pattern to the Solidity `.transfer` issue is exact:

| Solidity `.transfer` | Cosmos analog |
|---|---|
| Fixed 2300 gas forwarded to recipient | `sufficientBonusPoolBalance` hard-fails if pool balance < owed bonus |
| Recipient contract with complex fallback causes revert | Pool depletion (external operational condition) causes revert |
| `buy`/`sell` blocked for all users | `TierUndelegate`/`TierRedelegate`/`ExitTierWithDelegation` blocked for all users with accrued bonus |

---

### Impact Explanation

When the `RewardsPoolName` module account balance falls below the total accrued bonus owed to a position, every attempt by that position's owner to undelegate, redelegate, add funds, clear, or exit with delegation reverts. The user's principal (staked tokens held in the position's dedicated delegator account) is locked. The tokens are not destroyed, but the user has no permissionless path to recover them until governance or an admin refills the pool. The broken invariant is: **a user must always be able to exit a tiered-rewards position and recover their principal regardless of the state of the bonus pool**.

**Impact: Medium** — principal locked, not permanently destroyed; recovery requires external governance action.

---

### Likelihood Explanation

The bonus pool is a module account funded by external deposits (governance or protocol revenue). It is not self-replenishing from staking inflation in a guaranteed way. As the number of positions and accrued rewards grows, the pool can be depleted. Any period where the pool balance is insufficient — even transiently — blocks all affected users from exiting. This is a realistic operational condition, not a theoretical one.

**Likelihood: Medium** — pool depletion is a plausible steady-state condition under normal protocol operation.

---

### Recommendation

Decouple the reward payment from the structural exit operations. Specifically:

1. In `TierUndelegate`, `TierRedelegate`, `AddToTierPosition`, `ClearPosition`, and `ExitTierWithDelegation`, replace the hard `claimRewards` call with a best-effort variant: attempt to pay rewards, and if the pool is insufficient, record the owed amount as a claimable debt and proceed with the structural operation (undelegation/exit) regardless.
2. Alternatively, allow `processEventsAndClaimBonus` to skip the `SendCoinsFromModuleToAccount` step when the pool is insufficient, advance the checkpoints, and store the unpaid bonus as a pending claim that can be settled later when the pool is refilled.
3. This mirrors the recommendation in the original report: use a mechanism that does not allow the payment step to block the principal-recovery step.

---

### Proof of Concept

1. Alice creates a tiered-rewards position via `MsgLockTier` and accrues bonus rewards over time.
2. The `RewardsPoolName` module account balance drops to zero (pool depleted).
3. Alice submits `MsgTierUndelegate` for her position.
4. Execution path: `TierUndelegate` → `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` returns error because `pool balance (0) < bonusCoins`.
5. `TierUndelegate` reverts with the pool-insufficient error.
6. Alice cannot undelegate. She also cannot redelegate (`TierRedelegate`), add funds (`AddToTierPosition`), clear (`ClearPosition`), or exit with delegation (`ExitTierWithDelegation`) — all five handlers hit the same blocking `claimRewards` call.
7. Alice's principal remains locked in the position delegator account with no permissionless recovery path. [7](#0-6) [8](#0-7)

func (k Keeper) claimRewards(ctx context.Context, pos types.PositionState) (types.PositionState, sdk.Coins, sdk.Coins, error) {
	if !pos.IsDelegated() {
		return pos, sdk.NewCoins(), sdk.NewCoins(), nil
	}

	base, err := k.claimBaseRewards(ctx, pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	bonus, err := k.processEventsAndClaimBonus(ctx, &pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	return pos, base, bonus, nil
}

func (k Keeper) processEventsAndClaimBonus(ctx context.Context, pos *types.PositionState) (sdk.Coins, error) {
	// Rewards should have been claimed before undelegation
	if !pos.IsDelegated() {
		return sdk.NewCoins(), nil
	}

	valAddr, err := sdk.ValAddressFromBech32(pos.Delegation.ValidatorAddress)
	if err != nil {
		return nil, err
	}

	events, err := k.getValidatorEventsSince(ctx, valAddr, pos.LastEventSeq)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	blockTime := sdkCtx.BlockTime()

	totalBonus := math.ZeroInt()
	// Use the persisted bonded state from the last replay, not a hardcoded default.
	// This prevents overpaying bonus for unbonded gaps between claims.
	bonded := pos.LastKnownBonded
	segmentStart := pos.LastBonusAccrual

	tier, err := k.getTier(ctx, pos.TierId)
	if err != nil {
		return nil, err
	}

	for _, entry := range events {
		evt := entry.Event

		if bonded {
			// Compute bonus for the bonded segment [segmentStart, eventTime]
			// using the snapshot rate at the event.
			bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
			totalBonus = totalBonus.Add(bonus)
		}

		// Update bonded state based on event type.
		switch evt.EventType {
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_UNBOND:
			bonded = false
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_BOND:
			bonded = true
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_SLASH:
			// Slash doesn't change bonded state.
		}

		segmentStart = evt.Timestamp
		pos.UpdateLastEventSeq(entry.Seq)

		// Decrement reference count.
		if err := k.decrementEventRefCount(ctx, valAddr, entry.Seq); err != nil {
			return nil, err
		}
	}

	val, err := k.stakingKeeper.GetValidator(ctx, valAddr)
	if err != nil {
		return nil, err
	}
	// Defensive: validator bond status check
	if bonded && val.IsBonded() {
		currentRate, err := k.getTokensPerShare(ctx, valAddr)
		if err != nil {
			return nil, err
		}
		bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
		totalBonus = totalBonus.Add(bonus)
	}

	applyBonusAccrualCheckpoint(&pos.Position, blockTime)
	// Persist the bonded state so the next replay starts correctly.
	pos.UpdateLastKnownBonded(bonded)

	if totalBonus.IsZero() {
		return sdk.NewCoins(), nil
	}

	bondDenom, err := k.stakingKeeper.BondDenom(ctx)
	if err != nil {
		return nil, err
	}

	bonusCoins := sdk.NewCoins(sdk.NewCoin(bondDenom, totalBonus))

	if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
		return nil, err
	}

	ownerAddr, err := sdk.AccAddressFromBech32(pos.Owner)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid owner address")
	}

	if err := k.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.RewardsPoolName, ownerAddr, bonusCoins); err != nil {
		return nil, err
	}
```
