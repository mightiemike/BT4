### Title
`check_cost` in `op_unknown` Enforced on Base Cost Before Multiplier Is Applied, Allowing Returned Cost to Exceed `max_cost` - (File: src/more_ops.rs)

### Summary

In `op_unknown`, the `max_cost` guard (`check_cost`) is applied to the intermediate base cost **before** the cost multiplier is applied. The final cost returned to the caller can be orders of magnitude larger than `max_cost`, meaning the protection parameter provides no real enforcement on the actual output cost.

### Finding Description

In `src/more_ops.rs`, `op_unknown` computes a base cost from the operator's arguments (using one of four cost functions), then calls `check_cost(cost, max_cost)` to verify the cost is within budget. Immediately after, it multiplies the cost by `(cost_multiplier + 1)` ŌĆö where `cost_multiplier` can be up to `u32::MAX` ŌĆö and returns the multiplied value:

```rust
// src/more_ops.rs lines 258-266
check_cost(cost, max_cost)?;          // ŌåÉ guard applied to BASE cost only
cost *= cost_multiplier + 1;          // ŌåÉ multiplier applied AFTER the guard
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

The `cost_multiplier` is decoded from the operator bytes as a `u32` (up to `4_294_967_295`), so `cost_multiplier + 1` is up to `4_294_967_296`. With `cost_function = 0` (constant cost), the base cost is `1`. `check_cost(1, max_cost)` passes for any `max_cost Ōēź 1`. The returned cost is then `1 * (cost_multiplier + 1)`, which can be up to `u32::MAX = 4_294_967_295` ŌĆö completely independent of `max_cost`.

This is the direct analog to the external report: `minTokenAmounts_` was checked at the pool-withdrawal step but not at the final user-receipt step (after skimming). Here, `max_cost` is checked at the base-cost step but not at the final returned-cost step (after multiplication).

The `max_cost` value passed to `op_unknown` from `apply_op` is `effective_max_cost - cost` (the remaining budget). An attacker-controlled unknown opcode with a large multiplier causes `op_unknown` to return a cost far exceeding the remaining budget. The main loop in `run_program` catches this only at the **top of the next iteration**:

```rust
// src/run_program.rs lines 514-516
if cost > effective_max_cost {
    return Err(EvalErr::CostExceeded);
}
```

So the cost limit is violated for one full iteration before detection.

### Impact Explanation

The `max_cost` parameter passed to `op_unknown` is supposed to be a hard limit on the cost the operator may return ŌĆö consistent with every other operator (`op_sha256`, `op_add`, `op_multiply`, etc., all call `check_cost` on their **final** cost). `op_unknown` breaks this contract: it can return up to `u32::MAX` regardless of `max_cost`. Any caller that relies on the returned cost being within `max_cost` (e.g., the softfork guard's `expected_cost` accounting, or external cost-estimation tooling) receives a corrupted cost value. The program ultimately fails with `CostExceeded`, but only after the cost has already overflowed the budget by up to ~4.3 billion units.

### Likelihood Explanation

An attacker can trivially craft a CLVM program using an unknown opcode whose first 3 bytes encode a large multiplier (e.g., `0x3FFFFFFF`) and whose last byte selects `cost_function = 0` (constant base cost of 1). This is a single attacker-controlled atom in the program bytes, reachable through any path that invokes `op_unknown` in lenient/consensus mode (i.e., without `NO_UNKNOWN_OPS`).

### Recommendation

Move `check_cost` to **after** the multiplication so the guard is applied to the actual returned cost:

```rust
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    check_cost(cost, max_cost)?;   // ŌåÉ enforce on final cost
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

### Proof of Concept

Craft an unknown opcode atom: bytes `[0x3F, 0xFF, 0xFF, 0x00]`.
- `op.len() = 4`, so it falls into the 4-byte unknown-op path in `ChiaDialect::op` ŌĆö but only if it doesn't match the secp opcodes. Use any 4-byte value not equal to `0x13d61f00` or `0x1c3a8f00`, e.g. `[0x00, 0xFF, 0xFF, 0x00]`.
- Actually, for the general `op_unknown` path (1-byte or >4-byte ops in lenient mode), use a 2-byte opcode: `[0x3F, 0x00]`.
  - `cost_function = (0x00 & 0b11000000) >> 6 = 0` ŌåÆ base cost = 1
  - `cost_multiplier = u32_from_u8(&[0x3F]) = 63` ŌåÆ multiplied cost = `64`
- For maximum effect, use `[0xFF, 0x00]` (not `0xFFFF`-prefixed, so not reserved):
  - `cost_multiplier = 255`, multiplied cost = `256`
- For a 5-byte opcode `[0x3F, 0xFF, 0xFF, 0xFF, 0x00]`:
  - `cost_multiplier = u32_from_u8([0x3F, 0xFF, 0xFF, 0xFF]) = 0x3FFFFFFF = 1_073_741_823`
  - multiplied cost = `1_073_741_824`
  - `check_cost(1, max_cost)` passes for any `max_cost Ōēź 1`
