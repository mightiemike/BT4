### Title
`gas_target` Set Disproportionately Low Relative to `max_block_size` in Versioned Constants V0_14_2–V0_14_4, Causing Excessively Volatile EIP-1559 L2 Gas Price Adjustments — (`crates/apollo_versioned_constants/resources/`)

---

### Summary

The `gas_target` versioned constant in `orchestrator_versioned_constants_0_14_2.json` through `_0_14_4.json` is set to a value that is dramatically lower than the documented 60%-of-`max_block_size` invariant. The code explicitly states this ratio must be maintained when `max_block_size` changes, but it was not. The result is that the EIP-1559 L2 gas price adjustment formula produces price changes per block that are ~6–18× more aggressive than the design intends, causing incorrect fee accounting with direct economic impact on every transaction.

---

### Finding Description

The `VersionedConstants` struct in `crates/apollo_versioned_constants/src/lib.rs` carries two coupled fields:

```rust
// NOTE: When max_block_size is changed, update `gas_target` accordingly to maintain the ratio.
pub max_block_size: GasAmount,
// Target is 60% of max_block_size, making price adjustment more responsive to congestion.
pub gas_target: GasAmount,
```

The code comment establishes two invariants:
1. `gas_target` should be ~60% of `max_block_size`.
2. When `max_block_size` changes, `gas_target` must be updated to maintain the ratio.

The actual values across all shipped versions violate both invariants:

| Version | `gas_target` | `max_block_size` | Ratio |
|---|---|---|---|
| V0_14_0 | 3,200,000,000 | 4,000,000,000 | **80%** |
| V0_14_1 | 4,000,000,000 | 5,000,000,000 | **80%** |
| V0_14_2 | 1,500,000,000 | 5,800,000,000 | **25.9%** |
| V0_14_3 | 1,040,000,000 | 5,800,000,000 | **17.9%** |
| V0_14_4 | 1,040,000,000 | 5,800,000,000 | **17.9%** |

In V0_14_2, `max_block_size` increased from 5 B to 5.8 B (+16%), but `gas_target` was simultaneously *reduced* from 4 B to 1.5 B (−62.5%), a ratio collapse from 80% to 26%. By V0_14_3/V0_14_4 it fell further to 17.9%.

The `gas_target` feeds directly into `calculate_next_base_gas_price` in `crates/apollo_consensus_orchestrator/src/fee_market/mod.rs`:

```rust
let denominator =
    gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
let price_change = (price_u256 * gas_delta) / denominator;
```

where `gas_delta = |gas_used − gas_target|`.

**Maximum price change per block** (full block, `gas_used = max_block_size`):

```
price_change / price = (max_block_size − gas_target) / (gas_target × denominator)
```

| Version | Max Δprice/block |
|---|---|
| V0_14_1 (80%) | (5B − 4B) / (4B × 48) ≈ **0.52%** |
| V0_14_2 (26%) | (5.8B − 1.5B) / (1.5B × 48) ≈ **5.97%** |
| V0_14_3/4 (18%) | (5.8B − 1.04B) / (1.04B × 48) ≈ **9.54%** |

The `gas_price_max_change_denominator = 48` is described as the sensitivity parameter that *limits* the rate of change. But because `gas_target` is the denominator's multiplier, a 4.5× reduction in `gas_target` produces an 18× amplification of the per-block price swing, effectively nullifying the dampening role of `gas_price_max_change_denominator`. This is the direct analog of the external bug: a constant set too low causes a sensitivity parameter to become irrelevant.

The function `calculate_next_l2_gas_price_for_fin` reads `gas_target` from `VersionedConstants::latest_constants()` and passes it to `calculate_next_base_gas_price`. Both the proposer (`build_proposal`) and validator (`validate_proposal`) paths call `calculate_next_l2_gas_price` which calls this function, so every block's L2 gas price is computed with the wrong target.

---

### Impact Explanation

**Critical — Incorrect fee/gas accounting with economic impact.**

With `gas_target` at 17.9% of `max_block_size`:

- The equilibrium point is at 17.9% block utilization. Any block above that threshold (i.e., virtually every real block) causes price increases.
- The L2 gas price can swing up to ~9.5% per block instead of the ~0.52% the denominator was designed to enforce.
- Fee estimation (`starknet_estimateFee`, `starknet_simulateTransactions`) returns authoritative-looking values derived from this price, which are wrong.
- The `next_l2_gas_price` committed into `BlockHeaderWithoutHash` and propagated via `SyncBlock` to state sync carries the miscalculated value, so the wrong price is stored on-chain and served by RPC.
- Users systematically overpay during any period of moderate-to-high block utilization, and the price oscillates far more than the protocol design intends.

---

### Likelihood Explanation

**High.** This affects every block produced under Starknet versions V0_14_2, V0_14_3, and V0_14_4 — the current production versions. No special attacker action is required; normal transaction submission fills blocks above the 17.9% threshold, triggering the amplified price adjustment automatically.

---

### Recommendation

Restore the `gas_target`-to-`max_block_size` ratio to the documented ~60% (or at minimum the ~80% ratio used in V0_14_0/V0_14_1) when `max_block_size = 5,800,000,000`:

```json
// 60% target:
"gas_target": 3480000000

// 80% target (matching V0_14_0/V0_14_1 practice):
"gas_target": 4640000000
```

Additionally, add a compile-time or startup assertion enforcing the ratio invariant stated in the code comment, so future `max_block_size` changes cannot silently violate it:

```rust
assert!(
    versioned_constants.gas_target.0 >= versioned_constants.max_block_size.0 * 50 / 100,
    "gas_target must be at least 50% of max_block_size"
);
```

### Proof of Concept

The discrepancy is directly observable from the shipped JSON files and the price formula. No exploit harness is needed; the math is deterministic.

**Step 1 — Observe the broken ratio** (from the versioned constants files):

```
V0_14_3: gas_target=1_040_000_000, max_block_size=5_800_000_000
ratio = 1_040_000_000 / 5_800_000_000 = 17.93%   ← documented invariant: 60%
```

**Step 2 — Compute maximum per-block price change under V0_14_3:**

```
gas_delta_max = max_block_size − gas_target = 5_800_000_000 − 1_040_000_000 = 4_760_000_000
denominator   = gas_target × gas_price_max_change_denominator
              = 1_040_000_000 × 48 = 49_920_000_000
max_Δprice/price = 4_760_000_000 / 49_920_000_000 ≈ 9.54% per block
```

**Step 3 — Compare with V0_14_1 (80% ratio, same denominator=48):**

```
gas_delta_max = 5_000_000_000 − 4_000_000_000 = 1_000_000_000
denominator   = 4_000_000_000 × 48 = 192_000_000_000
max_Δprice/price = 1_000_000_000 / 192_000_000_000 ≈ 0.52% per block
```

**Step 4 — Confirm the call path:**

`calculate_next_l2_gas_price` (called every block) → `calculate_next_l2_gas_price_for_fin` → reads `VersionedConstants::latest_constants().gas_target` → passes to `calculate_next_base_gas_price` → computes `price_change = (price × gas_delta) / (gas_target × denominator)`.

The wrong `gas_target` is baked into the latest versioned constants and is used unconditionally for every block's L2 gas price computation, producing a price that is up to 18× more volatile than the `gas_price_max_change_denominator` was designed to allow. pub fn calculate_next_l2_gas_price_for_fin(
    current_l2_gas_price: GasPrice,
    height: BlockNumber,
    l2_gas_used: GasAmount,
    override_l2_gas_price_fri: Option<u128>,
    min_l2_gas_price_per_height: &[PricePerHeight],
    fee_actual: Option<GasPrice>,
) -> GasPrice {
    if let Some(override_value) = override_l2_gas_price_fri {
        info!(
            "L2 gas price ({}) is not updated, remains on override value of {override_value} fri",
            current_l2_gas_price.0
        );
        return GasPrice(override_value);
    }
    let gas_target = VersionedConstants::latest_constants().gas_target;
    let config_min = get_min_gas_price_for_height(height, min_l2_gas_price_per_height);
    let effective_min = match fee_actual {
        Some(fa) => GasPrice(max(config_min.0, fa.0)),
        None => config_min,
    };
    calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
}
