### Title
Rewards Pool Module Account Has No Governance-Controlled Withdrawal Path — (`x/tieredrewards/types/keys.go`, `x/tieredrewards/keeper/abci.go`, `x/tieredrewards/keeper/msg_server.go`)

---

### Summary

The `rewards_pool` module account in `x/tieredrewards` accepts funds from any account via a standard bank send, but the module exposes no governance-controlled message to withdraw or recover the pool balance. Funds can only leave the pool via two automatic paths: the `BeginBlocker` top-up (which sends to `x/distribution`, not back to the funder) and bonus reward claims by position owners. If governance sets `TargetBaseRewardsRate` to zero and all positions exit, any remaining pool balance is permanently stranded with no on-chain recovery path.

---

### Finding Description

The `RewardsPoolName = "rewards_pool"` module account is the direct analog to the `ReachFactory` contract described in the external report.

**Funds enter the pool** via an unrestricted bank send to the module account address. The integration test helper `fund_pool` demonstrates this is the intended funding mechanism — any account can call `cluster.transfer(from_addr, pool_addr, amount_coin)` with no access control: [1](#0-0) 

The module account name and its use as a send/receive target are defined here: [2](#0-1) 

**Funds leave the pool** via exactly two code paths:

1. `topUpBaseRewards` in the `BeginBlocker` — sends shortfall from the pool to `x/distribution`, allocated pro-rata to validators: [3](#0-2) 

2. `processEventsAndClaimBonus` — sends bonus rewards from the pool to position owners: [4](#0-3) 

**No governance withdrawal message exists.** The complete `MsgServer` interface implemented in `msg_server.go` contains fourteen messages (LockTier, CommitDelegationToTier, TierUndelegate, TierRedelegate, AddToTierPosition, TriggerExitFromTier, ClearPosition, ClaimTierRewards, WithdrawFromTier, ExitTierWithDelegation, UpdateParams, AddTier, UpdateTier, DeleteTier). None of them withdraws funds from the pool to a governance-controlled address: [5](#0-4) 

The governance messages confirmed in ADR-006 are only `MsgUpdateParams`, `MsgAddTier`, `MsgUpdateTier`, and `MsgDeleteTier` — none touches the pool balance.

---

### Impact Explanation

**Stranded pool balance scenario:**

1. Governance sets `TargetBaseRewardsRate = 0` via `MsgUpdateParams` (a supported, documented governance action).
2. All existing positions complete their lifecycle and exit — no active positions remain, so `processEventsAndClaimBonus` is never called.
3. Any remaining balance in the `rewards_pool` module account is now permanently stranded. The BeginBlocker skips top-up because the rate is zero; bonus claims cannot occur because there are no positions.

Even without step 1, if the pool is overfunded relative to the bonus obligations of all active positions, the excess is irrecoverable by governance. The only exit is through `x/distribution` (validators/delegators receive it as base rewards), not back to the project treasury or the original funder.

The exact corrupted value is the `rewards_pool` module account balance — specifically any amount exceeding what will naturally drain through the two automatic outflow paths.

---

### Likelihood Explanation

**Medium.** The scenario is realistic in at least two production situations:

- **Module wind-down:** Governance marks all tiers `CloseOnly`, waits for positions to exit, then sets `TargetBaseRewardsRate = 0`. Any pool remainder is permanently stranded.
- **Accidental overfunding:** Because `fund_pool` is an unrestricted bank send, any account (including a user who misreads the module address) can send funds to the pool. Those funds cannot be returned.

Both paths are reachable by unprivileged accounts (bank send) or by normal governance transactions (`MsgUpdateParams`).

---

### Recommendation

Add a governance-gated `MsgWithdrawFromRewardsPool` message (authority = `x/gov` module address) that calls `bankKeeper.SendCoinsFromModuleToAccount(ctx, types.RewardsPoolName, recipientAddr, amount)`. This mirrors the pattern already used for bonus payouts and gives governance a recovery path for excess or mistakenly sent funds, aligning with the project's stated goal of governance-driven evolution.

---
