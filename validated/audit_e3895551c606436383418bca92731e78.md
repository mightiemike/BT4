Audit Report

## Title
`SwapAllowlistExtension` gates on router address instead of real swapper, enabling allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `sender` is the `msg.sender` of the `pool.swap` call. When swaps route through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension checks the router's allowlist entry rather than the real user's. This produces two fund-impacting failure modes: allowlisted users are permanently blocked from using the router, and if the admin allowlists the router to restore access, every address on-chain can bypass the allowlist.

## Finding Description
**Call chain:**

`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that same `sender` value and forwards it verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` — where `msg.sender` is the pool and `sender` is whoever called `pool.swap`: [3](#0-2) 

**Router path:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly. The pool therefore sees `msg.sender = router`. The extension receives `sender = router` and checks `allowedSwapper[pool][router]`. The actual user's address (`msg.sender` of `exactInputSingle`) is only stored in transient storage via `_setNextCallbackContext` for the payment callback — it is never forwarded to the pool as the swap initiator: [4](#0-3) 

The same wrong-actor binding applies to `exactOutputSingle`, `exactInput`, and `exactOutput`: [5](#0-4) 

`setAllowedToSwap` accepts any address with no warning that allowlisting the router opens the gate to all callers: [6](#0-5) 

## Impact Explanation
**Mode A — Broken core swap flow for allowlisted users.** A pool admin allowlists specific users (e.g., KYC addresses). Those users call `exactInputSingle` through the router. The extension sees `sender = router`, which is not allowlisted, and reverts with `NotAllowedToSwap`. The curated pool is permanently unusable for its intended participants through the only supported periphery path — broken core pool functionality.

**Mode B — Complete allowlist bypass.** To restore router access, the admin calls `setAllowedToSwap(pool, router, true)`. The extension now passes for any caller who routes through the router, because `allowedSwapper[pool][router] = true`. Any unprivileged user can swap freely on the curated pool. The allowlist is entirely nullified for all router-mediated swaps — a direct admin-boundary break triggered by a natural remediation step.

## Likelihood Explanation
The trigger is a semi-trusted pool admin action that is both natural and well-motivated: allowlisting the router to restore router-mediated swap access after discovering Mode A. The admin has no on-chain signal that doing so opens the gate to all users. The `setAllowedToSwap` setter accepts any address and emits only a generic `AllowedToSwapSet` event with no warning. The router is a factory-registered, protocol-endorsed contract, making it a plausible allowlist entry. Mode A is triggered by any allowlisted user attempting to use the router — no attacker action required.

## Recommendation
The extension must gate the economically relevant actor — the human initiating the swap — not the intermediary contract. Two viable approaches:

1. **Extension-data forwarding:** The router encodes the original `msg.sender` into `extensionData` for each hop. The extension decodes and checks that address instead of `sender`. The router already forwards per-hop `extensionData` to the pool.

2. **Separate payer/swapper fields:** Extend the `beforeSwap` hook signature (or use a dedicated field in `extensionData`) so the router can attest the real initiator, and the extension verifies the attestation came from a trusted router address.

Until fixed, `SwapAllowlistExtension` should be documented as incompatible with `MetricOmmSimpleRouter` for pools that require per-user swap gating.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  setAllowedToSwap(pool, alice, true)   // only Alice is allowed

Step 1 — Mode A (broken flow):
  alice calls router.exactInputSingle({pool: pool, ...})
  router calls pool.swap(params.recipient, ...) → pool sees msg.sender = router
  pool calls _beforeSwap(sender=router, ...)
  extension: allowedSwapper[pool][router] == false → revert NotAllowedToSwap
  Alice cannot use the router at all.

Step 2 — Admin remediation (natural but fatal):
  admin calls setAllowedToSwap(pool, router, true)

Step 3 — Mode B (bypass):
  bob (never allowlisted) calls router.exactInputSingle({pool: pool, ...})
  router calls pool.swap(params.recipient, ...) → pool sees msg.sender = router
  pool calls _beforeSwap(sender=router, ...)
  extension: allowedSwapper[pool][router] == true → passes
  Bob swaps freely on the curated pool.
```
