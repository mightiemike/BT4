### Title
`RpcTransactionError::InternalError::debug_info` Serialized Verbatim into Public JSON-RPC Responses — (File: `chain/jsonrpc-primitives/src/types/transactions.rs`)

### Summary

`RpcTransactionError::InternalError { debug_info: String }` is the catch-all error variant for transaction submission and status endpoints. Unlike the sibling `InvalidTransaction { context }` field — which is correctly suppressed with `#[serde(skip_serializing)]` — the `debug_info` field carries no such suppression. It is populated with raw `format!("{:?}", err)` Debug output of internal chain errors and then serialized verbatim into the public JSON-RPC response body, reachable by any unprivileged user who submits a transaction.

### Finding Description

`RpcTransactionError` is defined in `chain/jsonrpc-primitives/src/types/transactions.rs`:

```rust
#[derive(thiserror::Error, Debug, Clone, serde::Serialize, serde::Deserialize)]
#[serde(tag = "name", content = "info", rename_all = "SCREAMING_SNAKE_CASE")]
pub enum RpcTransactionError {
    #[error("An error happened during transaction execution: {context:?}")]
    InvalidTransaction {
        #[serde(skip_serializing)]   // ← suppressed correctly
        context: near_primitives::errors::InvalidTxError,
    },
    ...
    #[error("The node reached its limits. Try again later. More details: {debug_info}")]
    InternalError { debug_info: String },   // ← NO skip_serializing
    ...
}

``` [3](#0-2) 

The `From<RpcTransactionError> for RpcError` conversion then places `debug_info` into the response in **two** locations:

1. The legacy `data` field — via `error.to_string()` which expands the `{debug_info}` format string directly.
2. The structured `error_struct.cause.info.debug_info` field — via `serde_json::to_value(error)` which serializes the full struct including the unsuppressed `debug_info` field. [4](#0-3) 

The `TxStatusError::ChainError` variant wraps `near_chain_primitives::Error`, whose Debug output can include `StorageError::StorageInconsistentState(String)`, `StorageError::MemTrieLoadingError(String)`, `Error::DBNotFoundErr(String)` (containing raw DB key strings), and `Error::IOErr` (containing OS-level error messages). [5](#0-4) [6](#0-5) 

### Impact Explanation

Any unprivileged user who submits a transaction via `broadcast_tx_commit`, `send_tx`, or `tx` and triggers an internal chain error path receives the raw Debug-formatted internal error in the JSON response. This exposes:

- Internal DB key structures (from `DBNotFoundErr`)
- Storage inconsistency details (from `StorageInconsistentState`)
- In-memory trie loading failure details (from `MemTrieLoadingError`)
- OS-level IO error messages (from `IOErr`)
- Internal `ProcessTxResponse` variant names and associated data for unhandled cases

This is a direct protocol-boundary information disclosure: the field is explicitly named `debug_info`, signaling developer intent that it is for internal debugging only, yet it is serialized into the public API contract with no gate.

### Likelihood Explanation

The path is reachable by any user who can submit a transaction to a public RPC endpoint. No special privileges are required. Triggering `TxStatusError::ChainError` requires only that the node encounters a chain-level error during transaction status lookup, which can happen under normal operating conditions (e.g., storage pressure, shard tracking changes). The `from_network_client_responses` catch-all at line 354 of `lib.rs` is also reachable for any `ProcessTxResponse` variant not explicitly handled.

### Recommendation

Apply `#[serde(skip_serializing)]` to the `debug_info` field in `RpcTransactionError::InternalError`, mirroring the treatment of `context` in `InvalidTransaction`. Log the full `debug_info` server-side only (e.g., via `tracing::warn!`). Return a generic opaque error code to the caller.

```rust
InternalError {
    #[serde(skip_serializing)]
    debug_info: String,
},
```

Additionally audit `RpcQueryError::InternalError { error_message }`, `RpcBlockError::InternalError { error_message }`, and all other `InternalError` variants across `chain/jsonrpc-primitives/src/types/` for the same pattern, as they are populated from `StorageError::to_string()` in `runtime/runtime/src/state_viewer/errors.rs`. [7](#0-6) 

### Proof of Concept

Submit a transaction to a node that is not tracking the relevant shard, or that encounters a storage error during status lookup. The JSON-RPC response will contain:

```json
{
  "error": {
    "name": "HANDLER_ERROR",
    "cause": {
      "name": "INTERNAL_ERROR",
      "info": {
        "debug_info": "ChainError(StorageInconsistentState(\"<raw internal message>\"))"
      }
    }
  },
  "data": "The node reached its limits. Try again later. More details: ChainError(StorageInconsistentState(\"<raw internal message>\"))"
}
```

The `debug_info` value is the verbatim `format!("{:?}", err)` output of the internal `near_chain_primitives::Error`, exposing internal node state to the caller. [8](#0-7) [9](#0-8)

impl From<RpcTransactionError> for crate::errors::RpcError {
    fn from(error: RpcTransactionError) -> Self {
        let error_data = match &error {
            RpcTransactionError::InvalidTransaction { context } => {
                if let Ok(value) =
                    serde_json::to_value(crate::errors::ServerError::TxExecutionError(
                        near_primitives::errors::TxExecutionError::InvalidTxError(context.clone()),
                    ))
                {
                    value
                } else {
                    Value::String(error.to_string())
                }
            }
            _ => Value::String(error.to_string()),
        };

        let error_data_value = match serde_json::to_value(error) {
            Ok(value) => value,
            Err(err) => {
                return Self::new_internal_error(
                    None,
                    format!("Failed to serialize RpcTransactionError: {:?}", err),
                );
            }
        };

        Self::new_internal_or_handler_error(Some(error_data), error_data_value)
    }
