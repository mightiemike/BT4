Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the actual end-user, allowing any user to bypass the swap allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the pool's `msg.sender` — the direct caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `sender` is the router address, not the actual end user. A pool admin who allowlists the router to enable router-mediated swaps inadvertently allows any user to bypass the per-user allowlist by routing through the router, breaking the core access-control invariant of curated pools.

## Finding Description

In `SwapAllowlistExtension.beforeSwap`, the allowlist check is:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` is the pool (correct pool key). `sender` is the first argument forwarded from the pool — the direct caller of `pool.swap()`.

In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as `sender` to the extension dispatcher:

```solidity
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged as the first argument to every configured extension.

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

So `msg.sender` to the pool is the **router**, and the extension receives `sender = router`. The check becomes `allowedSwapper[pool][router]`.

For a pool with a swap allowlist to support router-mediated swaps for curated users, the pool admin must allowlist the router. Once the router is allowlisted, **any** user — including those not individually allowlisted — can call `router.exactInputSingle(pool, ...)` and the extension passes, because it sees `sender = router` (allowlisted) rather than the actual end user (not allowlisted).

The `extensionData` field is available in the `beforeSwap` signature and is forwarded by the router unchanged, but `SwapAllowlistExtension` ignores it entirely — the `bytes calldata` parameter is unnamed and unused at line 31 — so there is no mechanism to recover the actual end user's identity.

The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput` on `MetricOmmSimpleRouter`, since all of them call `pool.swap()` with `msg.sender = router`.

## Impact Explanation

Any user can bypass the swap allowlist on a curated pool by routing through `MetricOmmSimpleRouter`. The allowlist guard — the only mechanism protecting a curated pool from unauthorized swappers — silently fails open for all router-mediated swaps once the router is allowlisted. Unauthorized users can trade against LP positions on pools explicitly configured to restrict access, breaking the core pool protection invariant and exposing LP funds to trading by actors the pool admin intended to exclude. This constitutes a broken core pool functionality causing loss of funds and an admin-boundary break where an unprivileged path bypasses the pool admin's access control.

## Likelihood Explanation

Pool admins who configure a `SwapAllowlistExtension` and also want to support the standard periphery router (the documented user-facing entrypoint) face a forced choice: either allowlist the router (breaking the allowlist for all users) or do not allowlist the router (making the router unusable for all users on that pool). The natural operational path — allowlist the router to support normal UX — directly triggers the bypass. No attacker-controlled setup is required beyond calling the public router. The precondition (router allowlisted) is the expected production configuration for any pool that intends to support both access control and router-mediated swaps.

## Recommendation

The extension must gate the **actual end user**, not the intermediary. Two viable approaches:

1. **`extensionData` attestation:** Require the router to encode the actual caller's address in `extensionData` and have the extension verify it against the allowlist. The router already forwards `params.extensionData` unchanged, so this is additive. The extension would decode the caller address from `extensionData` when `sender` is a known router.

2. **Separate router-aware allowlist:** Add a second mapping `allowedRouter` and require that router-mediated swaps carry a signed or encoded user identity that the extension validates, rather than treating the router address as the gated actor.

The current design where `sender` (pool's `msg.sender`) is used as the gated identity is fundamentally incompatible with any intermediary routing layer.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Pool admin allowlists Alice: `setAllowedToSwap(pool, alice, true)`.
3. Pool admin allowlists the router to support router-mediated swaps: `setAllowedToSwap(pool, router, true)`.
4. Bob (not individually allowlisted) calls `router.exactInputSingle({pool: pool, recipient: bob, ...})`.
5. The router calls `pool.swap(bob, ...)` with `msg.sender = router`.
6. The pool calls `_beforeSwap(router, ...)` which dispatches to `extension.beforeSwap(sender=router, ...)`.
7. The extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
8. Bob has successfully swapped on a pool he was never authorized to access.

The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput` on `MetricOmmSimpleRouter`, since all of them call `pool.swap()` with `msg.sender = router`.
