Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks LP position `owner` instead of actual depositor `sender`, allowing full allowlist bypass — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension` is documented as gating `addLiquidity` by depositor address, but its `beforeAddLiquidity` hook silently discards the `sender` parameter and validates only `owner` — a free caller-controlled parameter. Any address not on the allowlist can bypass the restriction by calling `pool.addLiquidity(allowedOwner, …)` with any address already allowlisted as `owner`, depositing tokens into a restricted pool without authorization.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the LP position recipient: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` correctly forwards both addresses to the extension: [2](#0-1) 

However, `DepositAllowlistExtension.beforeAddLiquidity` leaves the first parameter (`sender`) unnamed and checks only `owner`: [3](#0-2) 

Because `owner` is a free parameter chosen by the caller, any address can pass the guard by supplying any address that the pool admin has already allowlisted. The actual token depositor (`msg.sender`) is never validated.

For contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` and ignores `recipient`: [4](#0-3) 

The asymmetry confirms the deposit extension has the wrong parameter bound. Additionally, `removeLiquidity` enforces `msg.sender != owner` revert, meaning the named `owner` (who did not initiate the deposit) can later call `removeLiquidity` to extract the tokens provided by the unauthorized depositor: [5](#0-4) 

## Impact Explanation

This is an admin-boundary break: an unprivileged path bypasses a configured pool guard. Any address not on the allowlist can deposit tokens into a pool the admin intended to restrict, violating the pool admin's access-control invariant. The unauthorized depositor provides tokens via the swap callback; LP shares are credited to the named `owner`, who can subsequently call `removeLiquidity` to extract them. Unauthorized liquidity enters a restricted pool, corrupting the pool admin's intended liquidity composition.

## Likelihood Explanation

The bypass requires no special privilege. Any EOA or contract can call `pool.addLiquidity` with a known allowed owner address. Allowed owner addresses are discoverable on-chain via `AllowedToDepositSet` events or by reading `allowedDepositor` storage. The attacker only needs to implement `IMetricOmmAddLiquidityCallback` to provide tokens in the callback. The attack is repeatable with zero preconditions beyond knowing one allowlisted address.

## Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositor) instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

```solidity
// Setup: pool has DepositAllowlistExtension configured.
// allowedDepositor[pool][alice] = true; bob is NOT on the allowlist.

// Bob calls addLiquidity with owner = alice (an allowed address).
// Extension checks allowedDepositor[pool][alice] → true → passes.
// Bob provides tokens via IMetricOmmAddLiquidityCallback. LP shares go to alice.
// Bob has bypassed the deposit allowlist entirely.

pool.addLiquidity(
    alice,       // owner = allowed address (bypass key)
    0,           // salt
    deltas,
    callbackData,
    extensionData
);
// Result: bob (unauthorized) deposited tokens; alice received LP shares.
// alice calls removeLiquidity to extract bob's tokens (enforced: msg.sender == owner).
```
