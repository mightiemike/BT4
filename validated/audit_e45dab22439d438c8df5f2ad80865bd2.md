### Title
Fee Rate Applied at Fill Time Is Not Committed in Signed Order Digest — (`core/contracts/OffchainExchange.sol`)

---

### Summary

The `IEndpoint.Order` struct that traders sign does not include any fee rate field. At fill time, `applyFee` reads the taker/maker fee rate from live contract storage (`feeRates[tier][productId]`). Because `updateTierFeeRates` can be submitted through the endpoint pipeline and takes effect immediately, a fee rate change between order signing and order execution causes traders to pay different fees than they implicitly agreed to when signing.

---

### Finding Description

The `IEndpoint.Order` struct that forms the basis of every signed order contains only:

```
sender | priceX18 | amount | expiration | nonce | appendix
``` [1](#0-0) 

No fee rate — taker or maker — is included in this struct, and therefore none is committed in the EIP-712 digest that the trader signs.

At fill time, `applyFee` calls `getUserFeeRateWithBuilder`, which reads the user's current fee tier from `feeTiers[address]` and then calls `getTierFeeRateX18(feeTier, productId)`, which returns `feeRates[tier][productId]` directly from live storage: [2](#0-1) [3](#0-2) 

The fee rate stored in `feeRates[tier][productId]` can be updated at any time via `updateTierFeeRates`, which is `onlyEndpoint` and is dispatched directly from the sequencer's transaction pipeline in `EndpointTx._processTransaction`: [4](#0-3) 

The owner can also submit this update via `ContractOwner.updateTierFeeRates` (slow-mode path): [5](#0-4) 

The result is that `feeRates[tier][productId]` is a mutable global that is read at execution time, not at signing time, and the signed order provides no protection against it changing.

---

### Impact Explanation

A trader signs an order expecting to pay, for example, 2 bps taker fee (the default `takerRateX18 = 200_000_000_000_000`). If `updateTierFeeRates` raises the taker rate for that tier and product before the order is matched, the trader's `quoteDelta` is reduced by the larger fee amount: [6](#0-5) 

The trader's quote balance is debited more than they agreed to. The excess fee flows into `collectedFees` / `sequencerFee`, representing a direct, concrete asset transfer away from the trader with no on-chain consent.

---

### Likelihood Explanation

Triggering this requires either the protocol owner (via `ContractOwner.updateTierFeeRates`, `onlyOwner`) or the sequencer (via the `UpdateTierFeeRates` transaction type in the sequencer batch) to update fee rates while signed orders are pending. This is a privileged action, analogous to the Sofa DAO updating the settlement fee rate. Legitimate fee schedule changes — not necessarily malicious — are sufficient to cause the discrepancy. Orders with long expiration windows (the `expiration` field is a `uint64` and can be set far in the future) are most exposed, as they remain valid across multiple fee schedule updates. [7](#0-6) 

---

### Recommendation

Include the expected taker and maker fee rates in the `IEndpoint.Order` struct (or in the `appendix` encoding) so they are committed in the order digest. At fill time, `applyFee` should verify that the live `feeRates[tier][productId]` does not exceed the fee rate committed in the order. If it does, the fill should either revert or apply only the committed rate, protecting traders from post-signing fee increases.

---

### Proof of Concept

1. Trader signs an `IEndpoint.Order` for product `P` at taker fee tier 0 (default 2 bps). The signed digest covers `sender | priceX18 | amount | expiration | nonce | appendix` — no fee rate.
2. Owner calls `ContractOwner.updateTierFeeRates(tier=0, productId=P, makerRateX18=0, takerRateX18=50_000_000_000_000)` (5 bps), which enqueues an `UpdateTierFeeRates` slow-mode transaction.
3. The slow-mode transaction is processed by the sequencer, updating `feeRates[0][P]` to 5 bps via `OffchainExchange.updateTierFeeRates`.
4. The sequencer then matches the trader's still-valid signed order. `applyFee` reads `feeRates[0][P]` = 5 bps and deducts 5 bps from `quoteDelta`.
5. The trader pays 2.5× the fee they expected when signing, with no on-chain mechanism to have prevented or detected this. [8](#0-7) [9](#0-8)

    function applyFee(
        uint32 productId,
        OrderInfo memory orderInfo,
        MarketInfo memory market,
        int128 alreadyMatched, // in quote
        uint128 appendix,
        bool taker
    ) internal {
        // X account is passthrough for trading and incurs
        // no fees
        if (orderInfo.sender == X_ACCOUNT) {
            return;
        }
        int128 matchQuote = orderInfo.quoteDelta;
        int128 meteredQuote = 0;
        if (taker) {
            // flat minimum fee
            if (alreadyMatched == 0) {
                meteredQuote += market.minSize;
                if (matchQuote < 0) {
                    meteredQuote = -meteredQuote;
                }
            }

            // exclude the portion on [0, self.min_size) for match_quote and
            // add to metered_quote
            // fee is only applied on [minSize, quote_amount)
            int128 feeApplied = MathHelper.abs(alreadyMatched + matchQuote) -
                market.minSize;
            feeApplied = MathHelper.min(feeApplied, matchQuote.abs());
            if (feeApplied > 0) {
                if (matchQuote < 0) {
                    feeApplied = -feeApplied;
                }
                meteredQuote += feeApplied;
            }
        } else {
            // for maker rebates things stay the same
            meteredQuote += matchQuote;
        }
        FeeInfo memory feeInfo = getUserFeeRateWithBuilder(
            orderInfo.sender,
            productId,
            appendix,
            taker
        );

        int128 keepRateX18 = ONE - feeInfo.feeRate;
        int128 newMeteredQuote = (meteredQuote > 0)
            ? meteredQuote.mul(keepRateX18)
            : meteredQuote.div(keepRateX18);
        orderInfo.fee = meteredQuote - newMeteredQuote;
        orderInfo.builderFee = matchQuote.abs().mul(feeInfo.builderFeeRate);
        orderInfo.quoteDelta =
            orderInfo.quoteDelta -
            orderInfo.fee -
            orderInfo.builderFee;
```

        if (txn.productId == QUOTE_PRODUCT_ID) {

    function updateTierFeeRates(
        uint32[] memory tier,
        uint32[] memory productId,
        int128[] memory makerRateX18,
        int128[] memory takerRateX18
    ) external onlyOwner {
        if (
            tier.length != productId.length ||
            tier.length != makerRateX18.length ||
            tier.length != takerRateX18.length
        ) {
            revert InvalidInput();
        }
        for (uint256 i = 0; i < tier.length; i++) {
            IEndpoint.UpdateTierFeeRates memory _txn = IEndpoint
                .UpdateTierFeeRates(
                    tier[i],
                    productId[i],
                    makerRateX18[i],
                    takerRateX18[i]
                );
            _submitSlowModeTransaction(
                IEndpoint.TransactionType.UpdateTierFeeRates,
                abi.encode(_txn)
            );
        }
    }
