### Title
`DelegateAction` Expiry Boundary Off-by-One: Action Executes at Exact `max_block_height` — (`File: runtime/runtime/src/actions.rs`)

---

### Summary

`DelegateAction.max_block_height` is documented as "the maximal height of the block **below which** the given DelegateAction is valid," and the protocol spec states the action expires "if the current block is **equal or greater than** `max_block_height`." The runtime enforcement in `apply_delegate_action` uses a strict `>` comparison instead of `>=`, so the action remains executable at the exact `max_block_height` block — one block beyond the user's intended deadline.

---

### Finding Description

In `runtime/runtime/src/actions.rs`, the expiry gate is:

```rust
if apply_state.block_height > delegate_action.max_block_height() {
    result.result = Err(ActionErrorKind::DelegateActionExpired.into());
    return Ok(());
}
``` [1](#0-0) 

The field definition in `core/primitives/src/action/delegate.rs` reads:

```rust
/// The maximal height of the block in the blockchain below which the given DelegateAction is valid.
pub max_block_height: BlockHeight,
``` [2](#0-1) 

The protocol specification in `docs/RuntimeSpec/Actions.md` states:

> If the current block is **equal or greater than** `max_block_height` → `DelegateActionExpired` [3](#0-2) 

The OpenRPC schema repeats the same "below which" wording for both `DelegateAction` and `DelegateActionV2`. [4](#0-3) 

The only existing test for this path (`test_delegate_action_max_height`) only exercises `max_block_height + 1`, leaving the exact boundary untested:

```rust
// Setup current block as higher than max_block_height. Must fail.
let apply_state =
    create_apply_state(signed_delegate_action.delegate_action.max_block_height + 1);
``` [5](#0-4) 

The same `max_block_height()` accessor is shared by `DelegateActionV2` via `VersionedDelegateActionRef`, so the off-by-one applies to both action variants. [6](#0-5) 

---

### Impact Explanation

A user signs a `DelegateAction` and sets `max_block_height = H` to authorize a relayer to submit it only for blocks strictly below `H`. The user's mental model — and the documented invariant — is that at block `H` the authorization is expired and the relayer can no longer act on their behalf.

Because the runtime check is `block_height > H` rather than `block_height >= H`, the relayer can hold the signed action and submit it at exactly block `H`. The runtime accepts it, executes all inner actions (token transfers, key additions, contract calls, etc.) with the user's identity as predecessor, and charges the relayer for gas. The user's intended deadline is silently violated by one block.

---

### Likelihood Explanation

Every meta-transaction relayer that receives a `DelegateAction` controls the submission timing. Submitting at the exact `max_block_height` requires no special privilege — only the ability to observe the current block height and time a broadcast accordingly. Block heights on NEAR are publicly observable and advance at a predictable ~1-second cadence, making precise targeting straightforward. Any relayer (including a malicious one) can exploit this without any on-chain state manipulation.

---

### Recommendation

Change the expiry check from strict greater-than to greater-than-or-equal, aligning the code with the documented invariant:

```rust
// Before (off-by-one):
if apply_state.block_height > delegate_action.max_block_height() {

// After (matches spec and field doc):
if apply_state.block_height >= delegate_action.max_block_height() {
```

Add a test that asserts `DelegateActionExpired` is returned when `block_height == max_block_height`, complementing the existing `max_block_height + 1` test.

---

### Proof of Concept

1. User signs `DelegateAction { max_block_height: H, actions: [Transfer { deposit: 1 NEAR }], ... }` and hands it to a relayer.
2. User expects the action to be invalid starting at block `H`.
3. Relayer waits until the chain reaches block `H`.
4. Relayer wraps the `DelegateAction` in a transaction and broadcasts it.
5. Runtime evaluates: `H > H` → `false` → no expiry error is raised.
6. The transfer executes at block `H`, one block after the user's intended deadline.
7. The user's 1 NEAR is transferred without their current consent, violating the authorization boundary they signed.
