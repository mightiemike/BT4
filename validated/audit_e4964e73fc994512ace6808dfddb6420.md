### Title
`WithdrawCollateralV2` Accepted by Slow-Mode Submission but Silently Rejected on Execution, Causing Fee Loss and Failed Withdrawal — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

`submitSlowModeTransactionImpl` accepts `WithdrawCollateralV2` transactions (charges the slow-mode fee and queues them), but `processSlowModeTransactionImpl` has no handler for this transaction type and falls through to `revert()`. On mainnet, the revert is silently swallowed by the try/catch in `_executeSlowModeTransaction`. The user permanently loses their slow-mode fee and their withdrawal silently fails — an exact structural analog to M-32's "submission layer accepts what execution layer rejects."

---

### Finding Description

**Submission path — permissive:**

`submitSlowModeTransactionImpl` has three branches:
1. `DepositCollateral` → hard revert.
2. A fixed owner-only list → `require(sender == owner())`.
3. Everything else → charge slow-mode fee and enqueue.

`WithdrawCollateralV2` is absent from both special-case lists, so it falls into branch 3: the fee is pulled from the caller and the transaction is enqueued. [1](#0-0) 

**Execution path — restrictive:**

`processSlowModeTransactionImpl` handles `WithdrawCollateral` (V1) but contains no case for `WithdrawCollateralV2`. The function ends with `else { revert(); }`, which fires for every unrecognised type. [2](#0-1) [3](#0-2) 

**Silent swallow on mainnet:**

`_executeSlowModeTransaction` wraps the call in a try/catch. On any chain that is not Hardhat (`chainid != 31337`), a revert from `processSlowModeTransaction` is silently discarded unless the gas heuristic fires. The transaction slot is consumed, the fee is gone, and the withdrawal never executes. [4](#0-3) 

**The desynchronisation:**

| Layer | Behaviour for `WithdrawCollateralV2` |
|---|---|
| `submitSlowModeTransactionImpl` | Accepts, charges fee, enqueues |
| `processSlowModeTransactionImpl` | No handler → `revert()` |
| `_executeSlowModeTransaction` | Silently swallows the revert |

This is structurally identical to M-32: the gating layer (submission) is more permissive than the execution layer, and the try/catch plays the role of INSTR being the sole caller of `Kernel.executeAction` — the user has no alternative path to recover.

---

### Impact Explanation

1. **Slow-mode fee is permanently lost.** The fee is transferred out of the user's wallet during submission and is never refunded.
2. **Withdrawal silently fails.** The user's collateral remains locked in the protocol. The `WithdrawCollateralV2`-specific feature — specifying a custom `sendTo` address — is entirely unreachable via the slow-mode path.
3. **Sequencer-unavailability scenario is broken.** Slow mode exists precisely as a censorship-resistance fallback. A user who needs to withdraw to a non-default address (the only reason to use V2 over V1) and whose sequencer is unresponsive has no working on-chain path. [5](#0-4) 

---

### Likelihood Explanation

`WithdrawCollateralV2` is a first-class `TransactionType` in the enum and is the only withdrawal variant that supports a custom `sendTo` address. Any user who:
- wants to withdraw to an address other than their subaccount owner, **and**
- cannot rely on the sequencer (downtime, censorship, or the 3-day slow-mode timeout scenario),

will naturally attempt to submit a `WithdrawCollateralV2` slow-mode transaction. The submission succeeds and charges a fee, giving no indication that execution will fail. Likelihood is **medium**. [6](#0-5) 

---

### Recommendation

Choose one of:

1. **Add a handler** for `WithdrawCollateralV2` in `processSlowModeTransactionImpl`, mirroring the existing `WithdrawCollateral` branch but decoding `SignedWithdrawCollateralV2` and forwarding `sendTo` to `clearinghouse.withdrawCollateral`.
2. **Reject at submission time** by adding `WithdrawCollateralV2` to the explicit revert list (alongside `DepositCollateral`) in `submitSlowModeTransactionImpl`, so users receive an immediate on-chain error instead of a silent fee-consuming failure.

---

### Proof of Concept

1. User calls `Endpoint.submitSlowModeTransaction` with a `WithdrawCollateralV2` payload specifying a non-zero `sendTo`.
2. `submitSlowModeTransactionImpl` charges `SLOW_MODE_FEE` from the user and enqueues the transaction — no revert, no warning.
3. After the 3-day timeout, the user (or sequencer) calls `Endpoint.executeSlowModeTransaction`.
4. `_executeSlowModeTransaction` dequeues the transaction and calls `this.processSlowModeTransaction` inside a try/catch.
5. `processSlowModeTransactionImpl` reaches `else { revert(); }` because `WithdrawCollateralV2` has no handler.
6. The revert is caught silently (gas heuristic not triggered for a simple revert).
7. `nSubmissions` increments, the slot is consumed, the fee is gone, and the withdrawal never happens. [7](#0-6) [8](#0-7)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
