### Title
No Minimum Confirmation Quorum in `verify_transaction_inclusion` Allows Zero-Confirmation Transaction Acceptance — (File: `contract/src/lib.rs`)

---

### Summary

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations` value of `0` with no lower-bound enforcement. Any unprivileged NEAR caller can pass `confirmations = 0`, causing the confirmation-depth check to evaluate trivially to `true` for any block already in the contract's state. This is the direct analog of the "no quorum" bug: just as the voting contract required no minimum number of votes to pass a candidate, this contract requires no minimum number of confirmed blocks to declare a transaction verified.

---

### Finding Description

`verify_transaction_inclusion` enforces only an **upper** bound on `confirmations`: [1](#0-0) 

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
```

The actual depth check is: [2](#0-1) 

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
```

When `args.confirmations = 0`, the right-hand side is `0`. Because all values are `u64`, the expression `N >= 0` is **always true**, so the check is a no-op. The function then proceeds to evaluate the Merkle proof and returns `true` for a transaction in a block that may have been submitted seconds ago and could still be reorganized away.

The `confirmations` field is a plain `u64` with no minimum at the type or argument level: [3](#0-2) 

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after its coinbase-proof check, so it inherits the same flaw: [4](#0-3) 

---

### Impact Explanation

A consumer contract (e.g., an atomic-swap bridge or a cross-chain asset release contract) that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` with `confirmations = 0` will receive `true` for any transaction whose block has been submitted to the light client, regardless of how recently that block arrived. Because Bitcoin-family chains can reorganize, a block submitted one second ago may be orphaned. The contract provides no floor to prevent this, so the verification result carries no finality guarantee. Any downstream contract that relies on this result to release funds or update state is exposed to double-spend via reorg.

---

### Likelihood Explanation

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are public, `#[pause]`-gated functions with no role restriction: [5](#0-4) [6](#0-5) 

Any NEAR account can call them. The relayer continuously submits real blocks, so the headers pool is always populated with recent blocks. An attacker needs only to know a transaction hash and its block hash (both public on-chain data) and call the verification function with `confirmations = 0` immediately after the relayer submits the block. No privileged access is required.

---

### Recommendation

Add a contract-enforced minimum confirmation floor. For example:

```rust
const MIN_CONFIRMATIONS: u64 = 6; // or a configurable parameter set at init

require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    format!("Confirmations must be at least {}", MIN_CONFIRMATIONS)
);
```

This mirrors the "quorum" fix recommended in the original report: just as the voting contract should require a minimum number of votes before a candidate can pass, the light client should require a minimum number of confirmed blocks before a transaction is declared verified. The value can be made configurable via `InitArgs` so different deployments (mainnet vs. testnet) can set appropriate thresholds.

---

### Proof of Concept

1. The relayer submits block `B` at height `H` containing transaction `TX` (a double-spend attempt).
2. Attacker immediately calls:
   ```json
   verify_transaction_inclusion({
     "tx_id": "<TX hash>",
     "tx_block_blockhash": "<B hash>",
     "tx_index": 0,
     "merkle_proof": ["<valid proof>"],
     "confirmations": 0
   })
   ```
3. The check `(tip_height - H) + 1 >= 0` evaluates to `1 >= 0` → `true`.
4. The Merkle proof is valid, so the function returns `true`.
5. The consumer contract releases funds.
6. Block `B` is reorganized away on the Bitcoin network; the transaction never confirms.
7. The attacker has successfully claimed funds against a transaction that does not exist in the canonical chain.
