### Title
Unrestricted `depositCollateralWithReferral` Allows Collateral to Be Deposited to Uncontrollable Subaccounts, Permanently Locking Funds — (File: `core/contracts/Endpoint.sol`)

---

### Summary
`Endpoint.depositCollateralWithReferral()` is a `public` function that accepts an arbitrary `bytes32 subaccount` with no validation that the address portion (`bytes20(subaccount)`) is a valid EOA or a contract capable of signing withdrawal transactions. Tokens are transferred to the clearinghouse before the slow-mode credit is queued, so if the subaccount owner is `address(0)` or any address that cannot produce a valid EIP-712 signature, the deposited collateral is permanently locked.

---

### Finding Description
`depositCollateralWithReferral` (Endpoint.sol line 123) is `public` and accepts a caller-supplied `bytes32 subaccount`. The only guards applied are:

- `!RiskHelper.isIsolatedSubaccount(subaccount)` — rejects the isolated-subaccount magic suffix
- `requireUnsanctioned(msg.sender)` / `requireUnsanctioned(sender)` — OFAC sanctions check on the extracted address
- `isValidDepositAmount(...)` — minimum deposit threshold

There is no check that `address(bytes20(subaccount))` is a registered subaccount, an EOA, or any address that can ever sign a withdrawal. The token transfer happens unconditionally at line 144–148, before the slow-mode entry is written at line 152–165. Once tokens reach the clearinghouse, recovery requires a signed withdrawal from the subaccount owner. If that owner is `address(0)` or an uncontrollable contract, no such signature is ever possible.

By contrast, `depositCollateral` (line 103) constructs the subaccount as `bytes32(abi.encodePacked(msg.sender, subaccountName))`, so the owner is always the depositor — the safe path. `depositCollateralWithReferral` deliberately breaks that invariant to support third-party depositors (e.g., `DirectDepositV1`), but does so without any owner-validity guard. ---

### Impact Explanation
Tokens transferred to the clearinghouse via a deposit to an uncontrollable subaccount (e.g., `bytes32(0)`, a subaccount whose first 20 bytes are `address(0)`, or any contract that cannot produce EIP-712 signatures) are permanently locked. The clearinghouse credits the balance to the subaccount via `spotEngine.updateBalance`, but no withdrawal path exists because all withdrawal flows require a valid signature from `address(bytes20(subaccount))`. [3](#0-2) [4](#0-3) 

---

### Likelihood Explanation
Medium. The function is `public` and callable by any unsanctioned address. Realistic triggers include:

1. A user or integrating contract accidentally passes a malformed `bytes32 subaccount` (e.g., a raw `address` cast to `bytes32` without left-packing, producing `address(0)` in the first 20 bytes).
2. A contract integration (similar to `DirectDepositV1`) that calls `depositCollateralWithReferral` with a subaccount whose owner is a contract lacking signing capability.
3. A griefing scenario where a caller intentionally deposits dust to `bytes32(0)` to pollute the slow-mode queue, though the primary loss is the caller's own tokens. [5](#0-4) 

---

### Recommendation
Add a validation that the address portion of the supplied `subaccount` is non-zero and matches `msg.sender`, or restrict `depositCollateralWithReferral` so that `address(bytes20(subaccount)) == msg.sender` (mirroring the invariant enforced by `depositCollateral`). If third-party deposits must be supported, require that the target subaccount is already registered (`subaccountIds[subaccount] != 0`) before accepting the token transfer, analogous to the `requireSubaccount` helper already present in `EndpointStorage`. [6](#0-5)

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
        // copy from submitSlowModeTransaction
        SlowModeConfig memory _slowModeConfig = slowModeConfig;

        slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: abi.encodePacked(
                uint8(TransactionType.DepositCollateral),
                abi.encode(
                    DepositCollateral({
                        sender: subaccount,
                        productId: productId,
                        amount: amount
                    })
                )
            )
        });
        slowModeConfig = _slowModeConfig;
    }

    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
