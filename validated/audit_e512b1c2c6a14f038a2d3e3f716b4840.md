### Title
Accrued Bonus Rewards Permanently Forfeited When Rewards Pool Is Empty During Redelegation Slash — (`File: x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

In `x/tieredrewards`, when a validator slash fires against a redelegating tier position and the rewards pool is empty, `slashRedelegationPosition` silently swallows `ErrInsufficientBonusPool`. However, `processEventsAndClaimBonus` has already performed irreversible state mutations — decrementing event reference counts directly in the KV store and advancing the position's accrual checkpoints in memory — before the pool check is reached. The caller then persists the advanced checkpoints. The result is that the position owner's accrued bonus is permanently erased without payment: the events are garbage-collected and the checkpoints are advanced past the accrual window, making the rewards unrecoverable.

---

### Finding Description

`processEventsAndClaimBonus` (`x/tieredrewards/keeper/claim_rewards.go`) performs the following operations **before** the pool sufficiency check at line 230:

1. **Line 193** — `pos.UpdateLastEventSeq(entry.Seq)`: advances the position's last-processed event sequence in memory.
2. **Line 196** — `k.decrementEventRefCount(ctx, valAddr, entry.Seq)`: writes directly to the KV store (`ValidatorEvents.Set` / `ValidatorEvents.Remove`), decrementing or garbage-collecting the event. This is a committed store mutation regardless of what happens next.
3. **Lines 215–217** — `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded`: advance `LastBonusAccrual` and `LastKnownBonded` in memory.

Only after all of the above does the function check pool balance:

```go
// claim_rewards.go:230
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err
}
```

In `slashRedelegationPosition` (`slash.go:54–64`), the `ErrInsufficientBonusPool` error is explicitly caught and swallowed:

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
    } else {
        return err
    }
}
```

Execution then continues with the modified `pos` (advanced checkpoints). For a **partial slash**, `setPositionWithState(ctx, pos, nil)` is called at line 77, persisting the advanced `LastEventSeq`, `LastBonusAccrual`, and `LastKnownBonded`. For a **full slash**, `pos.ClearBonusCheckpoints()` is called at line 70 before `setPositionWithState`, zeroing the checkpoints — but the event ref counts were already decremented and the bonus was never paid.

In both cases:
- The `ValidatorEvents` reference counts are decremented (store-committed, irreversible within the transaction).
- The position's accrual window is advanced or cleared.
- The bonus is never transferred to the owner.

The user-driven paths (`MsgClaimTierRewards`, `MsgTierRedelegate`, etc.) correctly fail atomically on `ErrInsufficientBonusPool`, allowing the user to retry after the pool is replenished. The `BeforeRedelegationSlashed` hook path does not share this property.

---

### Impact Explanation

A tier position owner who has a redelegating position loses all accrued bonus rewards permanently when:
- The rewards pool balance is zero or insufficient to cover the accrued bonus, **and**
- A slash fires on the destination validator while the redelegation entry is active.

The corrupted values are:
- `pos.LastBonusAccrual` — advanced to block time without payment.
- `pos.LastEventSeq` — advanced past the accrual events.
- `ValidatorEvent.ReferenceCount` — decremented to zero, causing event garbage-collection.

After this, no future claim can recover the forfeited bonus: the events are gone and the checkpoints are past the accrual window. The rewards pool retains the funds, but the position's claim to them is permanently erased.

---

### Likelihood Explanation

The rewards pool drains continuously via the BeginBlocker top-up mechanism and via user bonus claims. An empty pool is a realistic steady-state condition (the pool requires active replenishment by the chain operator). Validator slashes are normal protocol events (downtime, double-sign). The combination of an empty pool and a slash on a validator with at least one redelegating tier position is a realistic, non-adversarial scenario. No privileged access is required; the trigger is a standard staking slash event.

---

### Recommendation

Move the pool sufficiency check **before** any state mutations in `processEventsAndClaimBonus`. Specifically, compute `totalBonus` first (read-only), check `sufficientBonusPoolBalance`, and only then decrement event reference counts and advance position checkpoints. Alternatively, in `slashRedelegationPosition`, when `ErrInsufficientBonusPool` is caught, do **not** persist the modified `pos` (i.e., reload the original position state before calling `setPositionWithState`), so that the accrual window is not advanced and the user can retry after the pool is replenished.

---

### Proof of Concept

1. Alice creates a tier position and redelegates it to validator V2. A `RedelegationMapping` entry is created.
2. The rewards pool drains to zero (naturally, via BeginBlocker or other claims).
3. Alice accrues bonus rewards over time (visible via `EstimatePositionRewards` query at `GET /chainmain/tieredrewards/v1/estimate_rewards/{position_id}`).
4. V2 is slashed. The staking module calls `BeforeRedelegationSlashed` → `slashRedelegationPosition`.
5. `processEventsAndClaimBonus` runs: event ref counts are decremented in the KV store (line 196), `LastBonusAccrual` and `LastEventSeq` are advanced in memory (lines 193, 215–217). The pool check at line 230 fails with `ErrInsufficientBonusPool`.
6. `slashRedelegationPosition` swallows the error (lines 56–63) and calls `setPositionWithState` with the modified `pos` (line 77 for partial slash).
7. Alice's position now has `LastBonusAccrual` = slash block time and `LastEventSeq` = latest event seq. The events are garbage-collected.
8. Alice calls `MsgClaimTierRewards`. `processEventsAndClaimBonus` finds no pending events and computes zero bonus for the already-advanced accrual window. Alice receives zero bonus despite having accrued it.

```go
func (k Keeper) decrementEventRefCount(ctx context.Context, valAddr sdk.ValAddress, seq uint64) error {
	key := collections.Join(valAddr, seq)
	event, err := k.ValidatorEvents.Get(ctx, key)
	if errors.Is(err, collections.ErrNotFound) {
		return nil // already cleaned up
	}
	if err != nil {
		return err
	}

	if event.ReferenceCount <= 1 {
		return k.ValidatorEvents.Remove(ctx, key)
	}

	event.ReferenceCount--
	return k.ValidatorEvents.Set(ctx, key, event)
}
```

```go
		return nil, err
```
