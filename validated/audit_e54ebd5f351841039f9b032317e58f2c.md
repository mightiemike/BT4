Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Gates the Router Address Instead of the Actual Swapper, Allowing Any User to Bypass a Curated Pool's Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is the pool's `msg.sender`. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. A pool admin who allowlists the router (required for any router-mediated swap) inadvertently permits every user of that router to bypass the per-address allowlist entirely.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs the check:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (correct), and `sender` is whatever the pool passed as the first argument to `beforeSwap`. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, binding `sender` to the pool's `msg.sender`: [2](#0-1) 

`ExtensionCalling._beforeSwap` then forwards that value verbatim to every extension: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly without forwarding the originating user's address: [4](#0-3) 

Therefore, the extension sees `sender = router`, not `sender = user`. If the pool admin allowlists the router (the only way to permit router-mediated swaps), `allowedSwapper[pool][router] = true` passes for every caller of the router, regardless of whether that individual user is permitted. The check `allowedSwapper[pool][attacker]` is never evaluated.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates the `owner` parameter — an explicit argument the caller sets to the actual position owner, separate from `msg.sender`: [5](#0-4) 

The swap path has no equivalent "owner" parameter; the only identity available to the extension is `sender`, which collapses to the router.

## Impact Explanation
A pool admin deploying a curated pool with `SwapAllowlistExtension` (e.g., to restrict trading to KYC'd counterparties or protocol-controlled addresses) must allowlist the router to support standard periphery access. Once the router is allowlisted, the allowlist becomes a no-op: any unpermissioned EOA can execute swaps against the curated pool's liquidity by calling `exactInputSingle` on the public router. This constitutes a direct loss of LP principal on pools whose security model depends on the allowlist, meeting the "Admin-boundary break bypassed by an unprivileged path" and "direct loss of user principal" impact criteria.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard, documented periphery entry point. Any pool admin supporting normal user flows will allowlist the router. The bypass requires no special privileges, no flash loans, and no exotic token behavior — any EOA can call `exactInputSingle` pointing at the curated pool. The precondition (router allowlisted) is the expected operational state, not an edge case.

## Recommendation
The extension must gate the economically relevant actor, not the intermediary. The cleanest fix mirrors the deposit path: add a `swapper` or `payer` field to the swap call path (analogous to how `addLiquidity` separates `owner` from `msg.sender`) so the extension can check the real initiator. Alternatively, the router could pass the originating user's address as an explicit parameter through `extensionData`, and the extension could decode and verify it — though this requires trust that the router populates it correctly. Checking `recipient` is fragile since it can be set to any address.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, router, true)   // enable router path
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)
  - Allowlisted LP adds liquidity

Attack:
  - Attacker (not in allowedSwapper) calls:
      router.exactInputSingle({pool: curatedPool, recipient: attacker, ...})
  - Router calls pool.swap(attacker, ...) → pool's msg.sender = router
  - _beforeSwap(router, ...) → extension checks allowedSwapper[pool][router] = true → PASSES
  - Attacker executes swap against curated pool's liquidity
  - allowedSwapper[pool][attacker] is never evaluated
```
