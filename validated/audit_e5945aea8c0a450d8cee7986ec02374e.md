### Title
`gas_refund_penalty` Protocol Upgrade Retroactively Applied to In-Flight Delayed Receipts — (`runtime/runtime/src/lib.rs`)

---

### Summary

When the `gas_refund_penalty` parameter is increased via a protocol upgrade (as explicitly planned per NEP-536 and documented in the codebase), every receipt already sitting in the delayed-receipt queue will have the **new** penalty applied to its gas refund at execution time, even though the gas was purchased under the **old** terms (currently 0%). The receipt stores the `gas_price` at which gas was purchased, but stores no record of the `gas_refund_penalty` that was in effect at purchase time. The penalty is always read from the live `RuntimeConfig` at execution time.

---

### Finding Description

**Root cause — `runtime/runtime/src/lib.rs`**

When a receipt executes, `refund_unspent_gas_and_deposits` computes the gas-refund penalty using the **current** epoch's `RuntimeConfig`:

```rust
// line 1186
let refund_penalty: Gas = config.fees.gas_penalty_for_gas_refund(gross_gas_refund);
``` [1](#0-0) 

`config` is `apply_state.config`, which is the config for the **current** protocol version at execution time, not the version at which the transaction was submitted and gas was purchased.

The `ActionReceipt` stores only `gas_price` (the purchase price):

```rust
// gas_purchase_price is read from the receipt
let gas_purchase_price = action_receipt.gas_price();
``` [2](#0-1) 

There is no field in the receipt for the `gas_refund_penalty` that was in effect when the gas was purchased. The `gas_refund_penalty` parameter lives only in `RuntimeFeesConfig`: [3](#0-2) 

**The planned upgrade that triggers the issue**

The documentation explicitly states the current 0% penalty is a temporary placeholder:

> "As of version 78, `gas_refund_penalty` is 0% and `min_gas_refund_penalty` 0 Tgas, always resulting in a zero-cost fee. This is only a stepping stone to give projects time to adapt before the fee is taking effect. The plan is to increase this to 5% and 1 Tgas." [4](#0-3) 

The base config confirms the current zero values: [5](#0-4) 

**Contrast with the handled `AccountCostIncrease` boundary**

The nearcore team explicitly handled the upgrade-boundary case for `AccountCostIncrease`: a receipt whose gas was purchased before the upgrade at a lower price cannot cover the new `account_creation_charge`, so the runtime charges only what is available:

```rust
// line 1241-1244
// `burned_gas_refund < amount_to_charge` could happen for receipts where the gas was
// purchased in protocol versions before `ProtocolFeature::AccountCostIncrease`, at a lower
// gas price that isn't enough to cover the cost of creating an account.
let amount_actually_charged = std::cmp::min(amount_to_charge, burned_gas_refund);
``` [6](#0-5) 

No equivalent boundary guard exists for `gas_refund_penalty`. The penalty is applied unconditionally from the current config regardless of when the gas was purchased.

**Delayed-receipt queue is the concrete attack surface**

Receipts that exceed the per-chunk gas limit are stored in the delayed-receipt queue in the trie and processed in subsequent chunks: [7](#0-6) 

A receipt queued before the upgrade executes after it, with the new penalty applied.

---

### Impact Explanation

**Severity: High**

Every receipt in the delayed queue at the moment the `gas_refund_penalty` upgrade activates will have its gas refund silently reduced by the new rate. For the planned 5% / 1 Tgas parameters:

- A receipt with 30 Tgas attached (the common `ft_transfer_call` pattern) that burns only 5 Tgas leaves 25 Tgas unspent.
- Under the old terms: full 25 Tgas refunded.
- Under the new terms applied retroactively: `max(25 * 5%, 1 Tgas)` = 1.25 Tgas penalty → only 23.75 Tgas refunded.
- At mainnet gas price (~100 µNEAR/gas), 1.25 Tgas = ~0.125 mNEAR per receipt, multiplied across all queued receipts at upgrade time.

The user cannot prevent this: unlike a single-admin rate change that can be front-run, a protocol upgrade activates at an epoch boundary with no per-user escape hatch.

---

### Likelihood Explanation

**Likelihood: High (certain when the upgrade occurs)**

- The upgrade is explicitly planned and documented in the codebase and NEP-536.
- The delayed-receipt queue is always non-empty on mainnet during normal operation.
- Every receipt in the queue at the upgrade epoch boundary is affected.
- No code path checks whether a receipt's gas was purchased before or after the `gas_refund_penalty` change.

---

### Recommendation

Store the `gas_refund_penalty` and `min_gas_refund_penalty` values in the receipt at creation time (analogous to how `gas_price` is stored), and use those stored values at execution time instead of the current config values. This mirrors the existing pattern for `gas_price` and the explicit boundary handling already implemented for `AccountCostIncrease`.

Alternatively, apply the new `gas_refund_penalty` only to receipts whose `gas_price` is at or above the new `min_gas_purchase_price` (1 Tgas/yN), since receipts purchased below that floor were created under the old economic model.

---

### Proof of Concept

1. User submits `ft_transfer_call` with 30 Tgas attached. Gas is purchased at `gas_price = 100_000_000 yN/gas`. Receipt is created with `gas_price = 100_000_000` stored in the `ActionReceipt`.
2. The chunk is full; the receipt enters the delayed-receipt queue in the trie.
3. A protocol upgrade activates, changing `gas_refund_penalty` from `0/100` to `5/100` and `min_gas_refund_penalty` from `0` to `1_000_000_000_000` (1 Tgas).
4. The receipt is dequeued and executed. It burns 5 Tgas; 25 Tgas remain unspent.
5. `refund_unspent_gas_and_deposits` is called. At line 1186, `config.fees.gas_penalty_for_gas_refund(25 Tgas)` returns `max(25 * 5%, 1 Tgas)` = 1.25 Tgas using the **new** config.
6. The user receives `(25 - 1.25) Tgas * 100_000_000 yN/gas` = 2.375 mNEAR instead of the expected 2.5 mNEAR — a 5% loss on unspent gas that was purchased under a 0% penalty regime.

The exact divergent Borsh-serialized value is the `gas_refund_penalty` rational `{numerator: 0, denominator: 100}` (at purchase time) versus `{numerator: 5, denominator: 100}` (at execution time), with no field in the stored receipt to record which was in effect. [8](#0-7) [9](#0-8)

**File:** core/parameters/src/cost.rs (L701-715)
```rust
    /// Given a left over gas amount to be refunded, returns how much should be
    /// subtracted as a penalty introduced with NEP-536.
    ///
    /// Must return a value smaller or equal to the `gas_refund` parameter.
    pub fn gas_penalty_for_gas_refund(&self, gas_refund: Gas) -> Gas {
        let relative_cost = Gas::from_gas(
            (u128::from(gas_refund.as_gas()) * *self.gas_refund_penalty.numer() as u128
                / *self.gas_refund_penalty.denom() as u128)
                .try_into()
                .unwrap(),
        );

        let penalty = std::cmp::max(relative_cost, self.min_gas_refund_penalty);
        std::cmp::min(penalty, gas_refund)
    }
```
