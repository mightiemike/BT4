### Title
`DirectDepositV1` Deterministic `create2` Address Enables Cross-Chain Fund Theft via Shared Proxy Factory - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.createDirectDepositV1` deploys `DirectDepositV1` via `create2` with a hardcoded salt of `bytes32(uint256(1))`. Because the resulting address is fully deterministic and `createDirectDepositV1` is `public` with no access control, an attacker who can deploy a `ContractOwner` proxy at the same address on a different chain (via a shared CREATE2 proxy factory) can replicate the `DirectDepositV1` at the identical address on that chain — with themselves as the effective owner — and drain any funds accidentally sent there.

---

### Finding Description

`ContractOwner.createDirectDepositV1` is declared `public` with no access restriction: [1](#0-0) 

It deploys `DirectDepositV1` using `create2` with a hardcoded salt of `bytes32(uint256(1))`: [2](#0-1) 

The resulting `create2` address is fully deterministic, computed from:
- The deployer address (`ContractOwner` proxy address)
- The fixed salt `bytes32(uint256(1))`
- The init code hash, which includes constructor arguments: `address(endpoint)`, `address(spotEngine)`, `subaccount`, `wrappedNative`

`ContractOwner` itself is an upgradeable proxy (`_disableInitializers()` in constructor, `initializer` modifier on `initialize`): [3](#0-2) 

If `ContractOwner` is deployed via a shared CREATE2 proxy factory (e.g., OpenZeppelin's deterministic proxy factory), it can be replicated at the same address on any EVM-compatible chain. The `initialize` function requires only that `_deployer == msg.sender` and transfers ownership to an arbitrary `multisig` address supplied by the caller: [4](#0-3) 

An attacker who deploys a `ContractOwner` proxy at the same address on another chain and initializes it with themselves as `multisig` gains full `onlyOwner` control over that instance. They can then:

1. Deploy fake `endpoint`, `spotEngine`, and `wrappedNative` contracts at the same addresses as on Ink Chain (using CREATE2 with matching salts).
2. Call `createDirectDepositV1(victimSubaccount)` — which is `public` — to deploy `DirectDepositV1` at the identical address as on Ink Chain.
3. The deployed `DirectDepositV1` is `Ownable` and its owner is set to `msg.sender` at construction time, which is the attacker's `ContractOwner`: [5](#0-4) 

4. When a user accidentally sends tokens to the `DirectDepositV1` address on the wrong chain, the attacker calls `withdrawFromDirectDepositV1` on their `ContractOwner` to drain the funds: [6](#0-5) 

Additionally, even if the attacker cannot steal funds (e.g., if they cannot replicate the constructor args exactly), they can still **grief** recovery: by deploying any contract at the `DirectDepositV1` address on the other chain before the legitimate protocol does, they permanently block the legitimate `create2` deployment at that address, making funds irrecoverable.

---

### Impact Explanation

Funds accidentally sent to a `DirectDepositV1` address on the wrong chain can be stolen by an attacker who has replicated the `ContractOwner` deployment on that chain and initialized it with themselves as the multisig owner. The attacker calls the `public` `withdrawFromDirectDepositV1` path through their controlled `ContractOwner`, transferring the victim's tokens to themselves. Alternatively, the attacker can permanently grief recovery by front-running the legitimate `create2` deployment at the same address on the other chain.

---

### Likelihood Explanation

The attack requires deploying `ContractOwner` at the same address on another chain, which is feasible if the protocol uses a shared deterministic proxy factory (a common pattern for multi-chain deployments). The fixed salt `bytes32(uint256(1))` and the `public` visibility of `createDirectDepositV1` remove any deployment-time barriers. An attacker can run a monitoring script watching for token transfers to known `DirectDepositV1` addresses on other chains and execute the attack opportunistically. The constructor-arg replication step (matching `endpoint`, `spotEngine`, `wrappedNative` addresses) is achievable via CREATE2 address grinding on the target chain.

---

### Recommendation

1. **Remove the fixed salt**: Replace `salt: bytes32(uint256(1))` with a salt that incorporates `block.chainid` and `subaccount`, e.g., `keccak256(abi.encode(block.chainid, subaccount))`. This makes the `DirectDepositV1` address chain-specific and non-replicable.
2. **Restrict `createDirectDepositV1`**: Add `onlyOwner` or `onlyDeployer` access control to prevent unpermissioned deployment.
3. **Document safe deployment patterns**: Clearly state that the protocol must not be deployed via shared CREATE2 factories if cross-chain fund recovery guarantees are expected.

---

### Proof of Concept

1. Nado deploys `ContractOwner` proxy at address `X` on Ink Chain via a shared CREATE2 proxy factory with salt `S`.
2. Attacker deploys a `ContractOwner` proxy at address `X` on Ethereum using the same factory and salt `S`, calling `initialize(attackerAddress, attackerAddress, ...)` to set themselves as multisig.
3. Attacker uses CREATE2 to deploy stub contracts at the same addresses as Ink Chain's `endpoint`, `spotEngine`, and `wrappedNative` on Ethereum.
4. Attacker calls `createDirectDepositV1(victimSubaccount)` on their `ContractOwner` at `X` on Ethereum. Because all CREATE2 inputs match, `DirectDepositV1` is deployed at the same address `D` as on Ink Chain.
5. Victim accidentally sends 10,000 USDC to address `D` on Ethereum.
6. Attacker calls `withdrawFromDirectDepositV1(victimSubaccount, USDC_address)` on their `ContractOwner` at `X` on Ethereum.
7. `DirectDepositV1.withdraw` transfers the 10,000 USDC to the attacker. [7](#0-6)
