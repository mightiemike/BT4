Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks LP-share recipient (`owner`) instead of the token payer (`sender`), allowing a non-allowlisted caller to bypass the deposit gate — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` receives two address arguments: the actual token payer (`sender`, first param, unnamed and discarded) and the LP-share recipient (`owner`, second param, the only one checked). Because `MetricOmmPool.addLiquidity` freely accepts any caller-supplied `owner` independently of who pays, a non-allowlisted EOA can name any allowlisted address as `owner`, pass the gate, and — if they control that allowlisted address — recover the deposited tokens via `removeLiquidity`. The deposit allowlist invariant of the private pool is structurally broken.

## Finding Description

**Call path:**

`MetricOmmPool.addLiquidity` passes `msg.sender` (the payer) as `sender` and the caller-supplied `owner` (the LP-share recipient) to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes both into the hook call: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as the first `address` parameter but leaves it unnamed and never reads it. Only `owner` is checked: [3](#0-2) 

The allowlist mapping is keyed by pool (= `msg.sender` inside the extension, i.e. the pool) and `owner`: [4](#0-3) 

Because `addLiquidity` imposes no constraint that `owner == msg.sender`, any caller can supply an arbitrary `owner`. The payer is never validated against the allowlist.

**Existing guards are insufficient:**

- `BaseMetricExtension.onlyPool` only verifies the caller of the extension is a registered pool — it does not validate the payer identity. [5](#0-4) 
- `MetricOmmPoolLiquidityAdder._validateOwner` only rejects `address(0)` — it does not require `owner == msg.sender`. [6](#0-5) 
- `removeLiquidity` enforces `msg.sender == owner`, so an attacker who controls the allowlisted `owner` address can fully recover deposited tokens. [7](#0-6) 

## Impact Explanation

A pool configured with `DepositAllowlistExtension` is a private liquidity venue. The bypass breaks the core access-control invariant:

- A non-allowlisted attacker who controls a second, allowlisted wallet calls `pool.addLiquidity(allowlistedWallet, salt, delta, ...)` from their non-allowlisted wallet.
- The extension evaluates `allowedDepositor[pool][allowlistedWallet]` → `true` → no revert.
- LP shares are minted to `allowlistedWallet`; tokens are pulled from the non-allowlisted wallet.
- The attacker calls `removeLiquidity` from `allowlistedWallet` and recovers the tokens.
- Net effect: the non-allowlisted wallet deposited into a restricted pool with zero restriction.

Even without controlling the `owner` address, an attacker can force-inject liquidity to any allowlisted address, diluting existing LPs' fee share and disrupting per-share metrics used by stop-loss or oracle-guard extensions. This is a broken access-control invariant with direct LP-asset impact (unauthorized share minting, fee dilution).

## Likelihood Explanation

- Requires no special privilege — any EOA or contract can call `addLiquidity` directly on the pool.
- Allowlisted addresses are publicly visible on-chain via `AllowedToDepositSet` events.
- The attacker only needs to control one allowlisted address (e.g., a second wallet they own that was allowlisted by the pool admin).
- `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, ...)` further enables the owner/payer split without any additional barrier. [8](#0-7) 

## Recommendation

Change `beforeAddLiquidity` to check the first parameter (the actual token payer) rather than — or in addition to — `owner`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to restrict both who pays and who receives shares, check both `sender` and `owner`.

## Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` attached with `beforeAddLiquidity` order set.
2. Admin calls `setAllowedToDeposit(pool, allowlistedWallet, true)` — only `allowlistedWallet` is permitted.
3. Attacker (non-allowlisted EOA `attackerWallet`) calls:
   ```solidity
   pool.addLiquidity(allowlistedWallet, salt, delta, callbackData, extensionData);
   ```
4. Pool calls `extension.beforeAddLiquidity(attackerWallet, allowlistedWallet, ...)`.
5. Extension evaluates `allowedDepositor[pool][allowlistedWallet]` → `true` → **no revert**.
6. LP shares are minted to `allowlistedWallet`; tokens are transferred from `attackerWallet` via callback.
7. Attacker calls `pool.removeLiquidity(allowlistedWallet, salt, delta, "")` from `allowlistedWallet` → tokens recovered.
8. Result: `attackerWallet` (non-allowlisted) successfully deposited into and withdrew from a restricted pool.
