### Title
`Clearinghouse.initialize` Sets `withdrawPool` Without the Zero-Address Validation Present in `setWithdrawPool` ŌĆö (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.initialize` assigns `withdrawPool` directly from a caller-supplied argument with no zero-address guard, while the post-deployment setter `setWithdrawPool` enforces `require(_withdrawPool != address(0))`. This inconsistency allows the contract to be initialized with `withdrawPool = address(0)`, permanently bricking all collateral, insurance, and rebalance withdrawals and burning user funds in the process.

---

### Finding Description

`setWithdrawPool` enforces a strict invariant before writing to `withdrawPool`:

```solidity
function setWithdrawPool(address _withdrawPool) external onlyOwner {
    require(_withdrawPool != address(0));   // ŌåÉ guard present
    withdrawPool = _withdrawPool;
}
``` [1](#0-0) 

The `initialize` function sets the same storage variable without any equivalent check:

```solidity
function initialize(
    address _endpoint,
    address _quote,
    address _clearinghouseLiq,
    uint256 _spreads,
    address _withdrawPool
) external initializer {
    ...
    withdrawPool = _withdrawPool;   // ŌåÉ no zero-address guard
    ...
}
``` [2](#0-1) 

The invariant "withdrawPool must not be the zero address" is therefore only enforced on the post-deployment update path, not during initialization ŌĆö the exact same class of inconsistency as the referenced external report.

---

### Impact Explanation

Every withdrawal path in the protocol routes through `handleWithdrawTransfer`:

```solidity
function handleWithdrawTransfer(
    IERC20Base token,
    address to,
    uint128 amount,
    uint64 idx
) internal virtual {
    token.safeTransfer(withdrawPool, uint256(amount));
    BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx);
}
``` [3](#0-2) 

If `withdrawPool == address(0)`:

1. `token.safeTransfer(address(0), amount)` ŌĆö transfers user collateral to the zero address, **permanently burning it**.
2. `BaseWithdrawPool(address(0)).submitWithdrawal(...)` ŌĆö external call to the zero address, **reverting the transaction**.

This affects every withdrawal entrypoint: `withdrawCollateral`, `withdrawInsurance`, and `rebalanceXWithdraw`. All user funds deposited into the clearinghouse become permanently unrecoverable. [4](#0-3) 

---

### Likelihood Explanation

Low but non-zero. The deployer could pass `address(0)` as `_withdrawPool` during a multi-step deployment (e.g., deploying `WithdrawPool` after `Clearinghouse`, or using a placeholder zero address in a deployment script). The inconsistency between the initializer and the setter creates a false sense of safety ŌĆö a developer reading `setWithdrawPool` would assume the zero-address invariant is always enforced, but it is not at initialization time.

---

### Recommendation

Add the same zero-address guard to `initialize` that `setWithdrawPool` already enforces:

```solidity
require(_withdrawPool != address(0));
withdrawPool = _withdrawPool;
```

This mirrors the fix applied in the referenced external report: add the corresponding validations to the constructor/initializer.

---

### Proof of Concept

1. Deploy `Clearinghouse` proxy and call `initialize(endpoint, quote, clearinghouseLiq, spreads, address(0))`.
2. `withdrawPool` is now `address(0)` ŌĆö no revert occurs because `initialize` has no guard.
3. A user deposits collateral and later calls `withdrawCollateral` (or the sequencer submits a `WithdrawCollateral` transaction).
4. `handleWithdrawTransfer` executes: `token.safeTransfer(address(0), amount)` burns the user's tokens, then `BaseWithdrawPool(address(0)).submitWithdrawal(...)` reverts.
5. The `setWithdrawPool` setter cannot rescue the situation if the deployer later tries to fix it ŌĆö the tokens transferred in step 4 are already burned.
6. All subsequent withdrawals are permanently bricked for every user. [1](#0-0) [3](#0-2)
