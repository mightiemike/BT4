Audit Report

## Title
`setPoolAdminFeeDestination` Redirects Accrued-But-Uncollected Admin Fees to New Destination Without Prior Settlement — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary
`setPoolAdminFeeDestination` overwrites `poolAdminFeeDestination[pool]` without first calling `collectFees`, unlike the analogous `setPoolAdminFees` and `setPoolProtocolFee` functions which both flush accrued fees before mutating state. All spread-fee surplus and notional fees already accrued in the pool are subsequently paid to the new destination on the next `collectFees` invocation, permanently depriving the old destination of ERC-20 tokens it had already earned.

## Finding Description
Every swap accrues admin fees inside `MetricOmmPool` as spread-fee surplus (real tokens in the contract) and notional fees tracked in `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled`. Both are paid to `poolAdminFeeDestination[pool]` only when `collectFees` is called.

`setPoolAdminFees` (L417–425) and `setPoolProtocolFee` (L327–335) both call `collectFees` with the current config and current destination before mutating any state. `setPoolAdminFeeDestination` (L444–446) performs no such flush — it directly writes the new address. After the call, `collectPoolFees` (L379–389) reads the now-updated `poolAdminFeeDestination[pool]` and transfers all previously accrued fees to the new destination. The old destination receives nothing.

## Impact Explanation
Direct loss of owed ERC-20 fee assets to the old `adminFeeDestination`. The loss equals the total admin-fee share of all swap volume since the last `collectFees` call — unbounded for high-volume pools with long collection intervals. This is a concrete loss of owed LP/fee assets, matching the allowed impact gate for Medium/High severity under Sherlock thresholds.

## Likelihood Explanation
The pool admin is a semi-trusted role with no cap or timelock on `setPoolAdminFeeDestination`. The old fee destination is a separate address from the pool admin (set independently at pool creation via `params.adminFeeDestination`). A pool admin can redirect all accrued fees in a single transaction, either maliciously or inadvertently. The public `collectPoolFees` function means anyone can immediately trigger payout to the new destination after the destination change, requiring no further privileged action.

## Recommendation
Mirror the pattern used by `setPoolAdminFees` and `setPoolProtocolFee`: call `collectFees` with the current config and the **old** destination before overwriting `poolAdminFeeDestination[pool]`.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]  // old destination
    );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

## Proof of Concept
1. Pool deployed with `adminFeeDestination = Alice`, `poolAdmin = Bob`.
2. Many swaps occur; spread-fee surplus accumulates (e.g., 1,000 USDC owed to Alice).
3. Bob calls `setPoolAdminFeeDestination(pool, Carol)` — no `collectFees` triggered (L444–446).
4. Anyone calls `collectPoolFees(pool)` — factory reads `poolAdminFeeDestination[pool] == Carol` (L387) and transfers 1,000 USDC to Carol.
5. Alice receives 0 USDC despite having earned the fees during her tenure as fee destination.

A Foundry integration test can confirm this by: deploying a pool, executing swaps to accrue fees, calling `setPoolAdminFeeDestination` to switch from Alice to Carol, then calling `collectPoolFees` and asserting Alice's balance is 0 and Carol's balance equals the accrued amount.
