### Title
Missing `__GAP` Storage Variables in All Upgradeable Base Contracts Enables Storage Collision on Future Upgrades — (`EndpointGated.sol`, `ClearinghouseStorage.sol`, `BaseEngine.sol`, `SpotEngineState.sol`, `PerpEngineState.sol`, `EndpointStorage.sol`)

---

### Summary

Every abstract base contract that defines storage for Nado's upgradeable proxies (`Clearinghouse`, `SpotEngine`, `PerpEngine`, `Endpoint`) is missing `__GAP` arrays entirely. This is a more severe instance of the same vulnerability class as the external report: rather than having incorrectly-sized gaps, Nado has no gaps at all. Any future upgrade that appends a storage variable to any of these base contracts will silently shift all downstream storage slots, corrupting balances, nonces, ownership state, and linked signers across the protocol.

---

### Finding Description

The Nado protocol deploys `Clearinghouse`, `SpotEngine`, `PerpEngine`, and `Endpoint` as ERC1967 transparent proxies managed by `ProxyManager`. Each of these proxied contracts inherits from one or more abstract storage base contracts. None of those base contracts declare a `__GAP` array:

| Base Contract | Storage Variables Declared | `__GAP` Present |
|---|---|---|
| `EndpointGated.sol` | 1 (`address private endpoint`) | No |
| `ClearinghouseStorage.sol` | 10 (`quote`, `clearinghouse`, `clearinghouseLiq`, `productToEngine`, `engineByType`, `supportedEngines`, `insurance`, `lastLiquidationFees`, `spreads`, `withdrawPool`) | No |
| `BaseEngine.sol` | 4 (`_clearinghouse`, `productIds`, `canApplyDeltas`, `nonZeroBalances`) | No |
| `SpotEngineState.sol` | 4 (`configs`, `states`, `balances`, `nlpLockedBalanceQueues`) | No |
| `PerpEngineState.sol` | 3 (`states`, `balances`, `migrationFlag`) | No |
| `EndpointStorage.sol` | 24 (all endpoint state) | No |

The inheritance chains that make this dangerous are:

- `Clearinghouse` → `EndpointGated` → `OwnableUpgradeable` + `ClearinghouseStorage`
- `SpotEngine` → `SpotEngineState` → `BaseEngine` → `EndpointGated` → `OwnableUpgradeable`
- `PerpEngine` → `PerpEngineState` → `BaseEngine` → `EndpointGated` → `OwnableUpgradeable`
- `Endpoint` → `EIP712Upgradeable` + `OwnableUpgradeable` + `EndpointStorage`

`EndpointGated` is the most critical base: it is inherited by both `Clearinghouse` and `BaseEngine` (and transitively by `SpotEngine` and `PerpEngine`). It holds `address private endpoint` at a fixed slot with no reserved gap. [1](#0-0) 

`ClearinghouseStorage` holds all core accounting state — insurance fund, product-to-engine mappings, withdraw pool address — with no gap. [2](#0-1) 

`BaseEngine` holds `_clearinghouse`, `productIds`, `canApplyDeltas`, and `nonZeroBalances` with no gap, and is the shared base for both engine proxies. [3](#0-2) 

`EndpointStorage` holds 24 storage variables including `nonces`, `subaccountIds`, `linkedSigners`, `nlpSigners`, `sequencer`, and `endpointTx` with no gap. [4](#0-3) 

`SpotEngineState` and `PerpEngineState` hold all per-product balance and state mappings with no gap. [5](#0-4) [6](#0-5) 

The upgrade mechanism is live and actively used: `ProxyManager.migrateAll` upgrades all proxies, and `Clearinghouse.upgradeClearinghouseLiq` / `Endpoint.upgradeEndpointTx` handle logic-extension upgrades via `delegatecall`. [7](#0-6) [8](#0-7) 

---

### Impact Explanation

If a developer adds even a single new storage variable to `EndpointGated`, every storage slot in `ClearinghouseStorage`, `BaseEngine`, `SpotEngineState`, and `PerpEngineState` shifts by one. Concretely:

- `quote` in `ClearinghouseStorage` would read the value previously stored in `clearinghouse`, causing all token transfers to go to the wrong address.
- `insurance` would read from the slot previously holding `supportedEngines`, corrupting the insurance fund accounting.
- `_clearinghouse` in `BaseEngine` would read from the slot previously holding `endpoint`, breaking all engine-to-clearinghouse calls.
- `nonZeroBalances` in `BaseEngine` would overlap with `productIds`, corrupting the bitmap used for health calculations across all subaccounts.
- `nonces` in `EndpointStorage` would overlap with adjacent slots, enabling replay attacks on signed transactions.

The impact is **high**: corrupted storage silently produces wrong balances, broken health checks, and bypassed replay protection — all without any on-chain revert.

---

### Likelihood Explanation

**Medium.** The protocol has an active, owner-operated upgrade pipeline (`ProxyManager`). The codebase is under active development (the wiki references ongoing feature work including NLP pools, isolated subaccounts, and migration flags). Any developer adding a new field to a base contract — a common, routine operation — will trigger this corruption without any compiler or test warning, since Solidity does not validate storage layout continuity across upgrades.

---

### Recommendation

Add `__GAP` arrays to every abstract base storage contract, sized so that the total storage footprint of each contract equals 50 slots (matching the OpenZeppelin convention). Specifically:

- `EndpointGated`: 1 variable used → `uint256[49] private __gap;`
- `ClearinghouseStorage`: count packed slots carefully → size gap to reach 50 total
- `BaseEngine`: 4 variables used → `uint256[46] private __gap;`
- `SpotEngineState`: 4 variables used → `uint256[46] private __gap;`
- `PerpEngineState`: 3 variables used → `uint256[47] private __gap;`
- `EndpointStorage`: 24+ variables used → size gap to reach 50 total (or a larger multiple)

Use OpenZeppelin's `@openzeppelin/upgrades-core` storage layout checker in CI to enforce this invariant on every future upgrade.

---

### Proof of Concept

1. Deploy `Clearinghouse` proxy with current storage layout. `EndpointGated.endpoint` occupies slot N (after OZ slots). `ClearinghouseStorage.quote` occupies slot N+1.
2. Upgrade `EndpointGated` to a new version that adds `address private newField` after `endpoint`.
3. After upgrade, `ClearinghouseStorage.quote` now reads from slot N+2 (previously `clearinghouse`), and `ClearinghouseStorage.clearinghouse` reads from slot N+3 (previously `clearinghouseLiq`).
4. Any call to `Clearinghouse.depositCollateral` will call `_spotEngine()` which reads `engineByType` from a shifted slot, returning `address(0)` or a garbage address, causing all deposits to revert or route to an attacker-controlled address if the slot happens to contain a value the attacker pre-staged.
5. No attacker action is required beyond waiting for the upgrade — the corruption is automatic and irreversible without a re-migration.
