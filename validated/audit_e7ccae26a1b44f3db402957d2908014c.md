### Title
Overlapping Opcode Dispatch: `ENABLE_SECP_OPS` Flag Bypassed via 4-Byte Opcode Path — (`File: src/chia_dialect.rs`)

---

### Summary

`op_secp256k1_verify` and `op_secp256r1_verify` are reachable via **two independent dispatch paths** in `ChiaDialect::op()`: a 4-byte opcode path (always active, no flag check) and a 1-byte opcode path (gated by `ENABLE_SECP_OPS`). The 4-byte path silently bypasses the `ENABLE_SECP_OPS` flag, making the flag an incomplete gate. Any caller or node that relies on `ENABLE_SECP_OPS = false` to disable secp operations is wrong — the operations remain reachable via the 4-byte encoding regardless.

---

### Finding Description

In `src/chia_dialect.rs`, `ChiaDialect::op()` contains two separate match arms that both dispatch to the same secp verification functions:

**Path 1 — 4-byte opcode (always active, no flag check):** [1](#0-0) 

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

**Path 2 — 1-byte opcode (gated by `ENABLE_SECP_OPS`):** [2](#0-1) 

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The `ENABLE_SECP_OPS` flag is defined as: [3](#0-2) 

> "Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify)."

The flag's name and description imply it controls whether secp operations are available. But the 4-byte opcode path at lines 175–182 is unconditional — it dispatches to `op_secp256k1_verify` / `op_secp256r1_verify` regardless of whether `ENABLE_SECP_OPS` is set. An attacker-controlled CLVM program that encodes the operator as the 4-byte atom `0x13d61f00` or `0x1c3a8f00` will always reach the secp verification logic, even when the caller has deliberately omitted `ENABLE_SECP_OPS`.

The underlying functions themselves are identical in both paths: [4](#0-3) 

```rust
const SECP256R1_VERIFY_COST: Cost = 1850000;
const SECP256K1_VERIFY_COST: Cost = 1300000;
```

Both paths call the same `op_secp256k1_verify` / `op_secp256r1_verify` with the same hardcoded cost, so cost is not the divergence point. The divergence is **flag enforcement**: one path enforces `ENABLE_SECP_OPS`, the other ignores it entirely.

---

### Impact Explanation

**Broken invariant:** `ENABLE_SECP_OPS` is supposed to be the activation gate for secp operations (consistent with the soft-fork / hard-fork flag pattern used throughout the codebase). Any node or caller that sets `ENABLE_SECP_OPS = false` to operate in a pre-activation state still executes real secp signature verification when the program uses the 4-byte opcode encoding.

**Concrete corrupted result:** The `ENABLE_SECP_OPS` guard at lines 248–249 is bypassed. `op_secp256k1_verify` / `op_secp256r1_verify` execute and return `Ok(Reduction(cost, nil))` in contexts where the caller expected them to fall through to `unknown_operator` (a no-op in lenient mode, or an error in `NO_UNKNOWN_OPS` mode).

**Consensus divergence:** A node running without `ENABLE_SECP_OPS` that encounters a transaction using the 4-byte opcode encoding will execute secp verification and accept the transaction. A node that has stripped the 4-byte path (or a future version that removes it) would treat the same bytes as an unknown operator. This is a flag/operator wiring error with consensus-level impact.

---

### Likelihood Explanation

The 4-byte opcodes `0x13d61f00` and `0x1c3a8f00` are valid, parseable CLVM atoms. Any attacker who can submit attacker-controlled CLVM bytes (the primary attack surface of this VM) can craft a program using these 4-byte encodings. The entry path is direct: submit a CLVM program with operator atom `\x13\xd6\x1f\x00` or `\x1c\x3a\x8f\x00`. No special privileges, no social engineering, no dependency compromise required.

---

### Recommendation

Enforce the independence of the two dispatch paths. The 4-byte secp opcode path should be subject to the same `ENABLE_SECP_OPS` flag check as the 1-byte path. Concretely, in the 4-byte match arm, add a flag guard before dispatching:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

Alternatively, if the 4-byte encoding is intended to be permanently active (pre-dating the flag), document this explicitly and rename `ENABLE_SECP_OPS` to `ENABLE_SECP_SHORT_OPS` to accurately reflect that it only gates the 1-byte encoding, not the operation itself. The current naming creates a false invariant that misleads callers.

---

### Proof of Concept

A CLVM program using the 4-byte opcode encoding for secp256k1 verification:

```
; operator atom is 0x13d61f00 (4 bytes), no ENABLE_SECP_OPS flag set
; run with ChiaDialect::new(ClvmFlags::empty())  -- ENABLE_SECP_OPS absent

(0x13d61f00 pubkey msg sig)
```

With `ENABLE_SECP_OPS` absent, the 1-byte opcode `64` would fall through to `unknown_operator`. But the 4-byte opcode `0x13d61f00` dispatches directly to `op_secp256k1_verify` at line 176, executes the full ECDSA verification, and returns `Ok(Reduction(1300000, nil))` — bypassing the flag gate entirely.
