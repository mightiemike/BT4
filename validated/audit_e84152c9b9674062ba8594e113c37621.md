### Title
Unverified `voter` field in consensus `Vote` allows a single peer to forge quorum by impersonating all validators — (File: `crates/apollo_consensus/src/single_height_consensus.rs`)

---

### Summary

The `handle_vote` function in `SingleHeightConsensus` accepts the `voter` field from incoming `Vote` messages at face value without verifying the cryptographic signature. A single malicious network peer can send N votes each claiming to be a different committee member, meeting the Byzantine quorum threshold without any legitimate validator participation.

---

### Finding Description

The `Vote` struct carries both a `voter: ContractAddress` (the claimed identity) and a `signature: RawSignature` (the cryptographic proof): [1](#0-0) 

When a vote arrives over the P2P network, `handle_vote` in `SingleHeightConsensus` performs two checks before accepting it:

1. **Committee membership** — checks that `vote.voter` is in the committee: [2](#0-1) 

2. **Deduplication** — calls `received_vote()`, which keys on `(vote.round, vote.voter)`: [3](#0-2) 

**Neither check verifies that the sender actually controls the `voter` address.** The signature field is explicitly left unverified, as acknowledged by the TODO comment: [4](#0-3) 

Even self-votes are produced with a default/empty signature: [5](#0-4) 

The deduplication map uses `(round, voter)` as the key: [6](#0-5) 

Because `voter` is self-reported and unverified, an attacker can send N votes with N distinct `voter` values (each a real committee member address). Each vote:
- Passes the committee membership check (the address is a known validator)
- Passes deduplication (different `voter` → different key → `VoteStatus::New`)
- Is inserted into `prevotes` or `precommits` with the full weight of the impersonated validator

The quorum check then sums the weights of all inserted votes: [7](#0-6) 

This is the direct analog of the external bug: instead of replaying the same signature multiple times, the attacker sends one message per validator identity, each counted once — but all originating from a single peer.

---

### Impact Explanation

**Critical.** A single malicious peer connected to the consensus P2P network can forge votes from every committee member for a chosen `proposal_commitment`, meet the 2/3 Byzantine quorum threshold, and cause `DecisionReached` to fire with an attacker-controlled block. This bypasses the BFT safety guarantee entirely, allowing wrong state, wrong receipts, wrong events, and wrong storage values to be committed to the Starknet chain.

---

### Likelihood Explanation

**High.** The attack requires only network connectivity to the consensus broadcast topic. No validator key material, no privileged access, and no special timing is needed. The `// TODO(Asmaa): verify the signature` comment confirms this is a known missing control, not a deliberate design choice.

---

### Recommendation

Implement signature verification inside `handle_vote` before a vote is accepted as `VoteStatus::New`. The verifier must:

1. Reconstruct the canonical vote preimage (covering `vote_type`, `height`, `round`, `proposal_commitment`) using a domain separator.
2. Recover or look up the public key associated with `vote.voter`.
3. Verify `vote.signature` against that public key and the preimage.
4. Reject the vote if verification fails, regardless of whether `vote.voter` is in the committee.

The existing `verify_precommit_vote_signature` / `verify_identity` infrastructure in `crates/apollo_signature_manager/src/signature_manager.rs` provides a reference pattern: [8](#0-7) 

---

### Proof of Concept

Assume a committee of 4 validators `[A, B, C, D]` each with weight 1 (total weight 4, Byzantine quorum requires weight > 8/3 ≈ 2.67, so weight 3 suffices).

A single malicious peer sends three `Vote` messages over the broadcast channel, all with `proposal_commitment = X`, `height = H`, `round = 0`, `vote_type = Precommit`, and `signature = RawSignature::default()`:

```
Vote { voter: A, proposal_commitment: X, signature: [] }
Vote { voter: B, proposal_commitment: X, signature: [] }
Vote { voter: C, proposal_commitment: X, signature: [] }
```

Processing in `handle_vote`:
- Each `vote.voter` is found in the committee → passes membership check.
- Each `(round=0, voter)` key is absent from `precommits` → `VoteStatus::New` for all three.
- Each is inserted into `precommits` with weight 1.
- After the third insertion, `votes_meet_threshold` computes weight_sum = 3 > 4 × 2/3 ≈ 2.67 → quorum met.
- `upon_decision` fires and returns `SMRequest::DecisionReached(Decision { block: X, ... })`.

The attacker has forced consensus to commit block `X` without a single legitimate validator signing anything.
