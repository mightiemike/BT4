### Title
USDC Quote Token Hardcoded to $1.00 in Health and Deposit Calculations — (`core/contracts/SpotEngine.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

Nado's `SpotEngine` initializes the quote token (USDC, `QUOTE_PRODUCT_ID = 0`) with a hardcoded `priceX18 = ONE` (i.e., exactly $1.00). This price is used directly in all subaccount health calculations. Additionally, `Clearinghouse.checkMinDeposit` explicitly bypasses any oracle price for the quote token and forces `priceX18 = ONE`. If USDC depegs, every health check, liquidation threshold, and minimum deposit gate in the protocol will silently compute incorrect values, because USDC collateral will be overvalued relative to its true USD worth.

---

### Finding Description

**Root cause 1 — Hardcoded initialization in `SpotEngine.initialize`:**

`SpotEngine.sol` initializes the risk store for `QUOTE_PRODUCT_ID` with `priceX18: ONE`:

```solidity
_risk().value[QUOTE_PRODUCT_ID] = RiskHelper.RiskStore({
    longWeightInitial: 1e9,
    shortWeightInitial: 1e9,
    longWeightMaintenance: 1e9,
    shortWeightMaintenance: 1e9,
    priceX18: ONE          // ← hardcoded 1.0 = $1.00
});
``` [1](#0-0) 

There is no oracle feed that updates this price for USDC. The `updatePrice` path in `Clearinghouse` can technically write to this slot, but the protocol design never feeds a USDC/USD oracle price — all other asset prices are denominated in USDC, making a USDC/USD oracle absent by design.

**Root cause 2 — Explicit bypass in `Clearinghouse.checkMinDeposit`:**

```solidity
int128 priceX18 = ONE;
if (productId != QUOTE_PRODUCT_ID) {
    priceX18 = _getPriceX18(productId);
}
return priceX18.mul(amountRealized) >= minDepositAmount;
``` [2](#0-1) 

This explicitly hardcodes the USDC price to 1.0 for minimum deposit checks, regardless of any stored `priceX18` value.

**How health is computed:**

`BaseEngine._calculateProductHealth` uses `risk.priceX18` for every product, including `QUOTE_PRODUCT_ID`:

```solidity
health += amount.mul(weight).mul(risk.priceX18);
``` [3](#0-2) 

For `QUOTE_PRODUCT_ID`, `_getBalance` returns `(usdc_balance, 0)`: [4](#0-3) 

So the USDC health contribution is always `usdc_balance × 1.0 × 1.0 = usdc_balance`, treating every USDC as worth exactly $1.00.

The `getHealth` function in `Clearinghouse` aggregates these contributions and is the single gate for all withdrawal, liquidation, and trade health checks: [5](#0-4) 

---

### Impact Explanation

If USDC depegs (e.g., to $0.88 as occurred during the SVB crisis):

- A subaccount holding 10,000 USDC as collateral is treated as having $10,000 of health contribution, when its real value is $8,800.
- The $1,200 phantom health allows the subaccount to carry open borrows or perp positions that are actually undercollateralized.
- `getHealth` returns a non-negative value, so `withdrawCollateral`, `transferQuote`, `mintNlp`, and `liquidateSubaccount` (the `isAboveInitial` / `isUnderMaintenance` checks) all operate on inflated health figures.
- Liquidations that should trigger at the maintenance threshold do not, because the USDC collateral is overvalued.
- The insurance fund absorbs losses that should have been caught by liquidation.

The corrupted state delta is: **subaccount health is overstated by `usdc_balance × (1 - actual_usdc_price)`** for every subaccount holding USDC collateral.

---

### Likelihood Explanation

USDC has historically depegged. During the March 2023 SVB collapse it reached $0.88. The Nado protocol has no mechanism to feed a USDC/USD oracle price — the entire price system is denominated in USDC, so a USDC/USD feed is structurally absent. Any USDC depeg event, however brief, creates a window where the protocol's health accounting is materially wrong. This is not a theoretical edge case; it is a known, recurring market event.

---

### Recommendation

1. Integrate a USDC/USD oracle (e.g., Chainlink USDC/USD) and feed its price into `QUOTE_PRODUCT_ID`'s `priceX18` via the existing `updatePrice` path on each price update cycle.
2. Remove the explicit `priceX18 = ONE` override in `Clearinghouse.checkMinDeposit` and use `_getPriceX18(QUOTE_PRODUCT_ID)` consistently.
3. Alternatively, compute all TVL and health values in USDC terms (not USD), and accept that the unit of account is USDC — but then ensure all thresholds (`MIN_DEPOSIT_AMOUNT`, `LIQUIDATION_FEE`, etc.) are also expressed in USDC, not USD.

### Proof of Concept

**Setup:**
- USDC depegs to $0.90.
- Alice holds 10,000 USDC collateral (`QUOTE_PRODUCT_ID` balance = `10000e18`).
- Alice has borrowed 8,000 USDC worth of WETH (spot borrow), giving her a WETH balance of `+8000e18 / weth_price` and a quote balance of `-8000e18`.
- Net USDC balance = `10000e18 - 8000e18 = 2000e18`.

**Protocol health calculation (current):**
- USDC health contribution = `2000e18 × 1.0 × 1.0 = 2000e18` (treated as $2,000).
- WETH borrow health contribution = `weth_amount × shortWeight × weth_price` (negative).
- Suppose this nets to `+200e18` — Alice appears healthy.

**Real health (with USDC at $0.90):**
- USDC health contribution should be `2000e18 × 0.90 = 1800e18` ($1,800).
- The same WETH borrow contribution nets to `-200e18` in real terms.
- Real health = `1800e18 - 2000e18 = -200e18` — Alice should be liquidatable.

**Result:** `getHealth` returns `+200e18`, `isUnderMaintenance` returns `false`, and Alice's undercollateralized position is not liquidated. The protocol accumulates bad debt. [1](#0-0) [2](#0-1) [6](#0-5)
