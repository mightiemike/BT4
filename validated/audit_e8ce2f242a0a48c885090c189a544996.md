### Title
Unvalidated `flags` Parameter Accepts Undocumented Hard-Fork Bits via `from_bits_truncate`, Enabling Premature `RELAXED_BLS` Activation and Consensus Divergence - (File: `wheel/src/api.rs`)

---

### Summary

The Python-facing `run_serialized_chia_program` function accepts a raw `u32` flags value from callers and converts it with `ClvmFlags::from_bits_truncate`, which silently accepts any known bit — including internal hard-fork flags that are deliberately **not** exported to the Python module. A caller who passes `flags = 0x0008` (`RELAXED_BLS`) enables a hard-fork behavior (accepting invalid BLS points in `bls_g1_negate`/`bls_g2_negate`) before that fork has activated on-chain, causing the node to accept transactions that consensus-mode peers would reject.

---

### Finding Description

In `wheel/src/api.rs`, `run_serialized_chia_program` receives a `flags: u32` from Python and converts it:

```rust
let flags = ClvmFlags::from_bits_truncate(flags);
```

`from_bits_truncate` (from the `bitflags` crate) silently drops bits that do not correspond to any defined flag, but **accepts every bit that does correspond to a known flag**, whether or not that flag is part of the documented Python surface.

The Python module exports only seven constants:

```rust
m.add("NO_UNKNOWN_OPS",    ClvmFlags::NO_UNKNOWN_OPS.bits())?;   // 0x0002
m.add("LIMIT_HEAP",        ClvmFlags::LIMIT_HEAP.bits())?;        // 0x0004
m.add("MEMPOOL_MODE",      MEMPOOL_MODE.bits())?;                  // composite
m.add("ENABLE_SHA256_TREE",ClvmFlags::ENABLE_SHA256_TREE.bits())?;// 0x0400
m.add("ENABLE_SECP_OPS",   ClvmFlags::ENABLE_SECP_OPS.bits())?;  // 0x0800
m.add("DISABLE_OP",        ClvmFlags::DISABLE_OP.bits())?;        // 0x0200
m.add("CANONICAL_INTS",    ClvmFlags::CANONICAL_INTS.bits())?;    // 0x0001
```

Five flags are **not** exported but remain fully effective if their bit values are passed:

| Flag | Bits | Risk |
|---|---|---|
| `RELAXED_BLS` | `0x0008` | Hard-fork: accepts invalid BLS points — **consensus divergence** |
| `LIMIT_SOFTFORK` | `0x0010` | Mempool-only depth limit — skipped in consensus mode |
| `ENABLE_GC` | `0x0020` | Allocator GC — changes heap behavior |
| `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` | `0x0100` | Activates keccak outside softfork guard |
| `MALACHITE` | `0x1000` | Switches bignum library for `/`, `divmod`, `mod`, `modpow` |

The most dangerous is `RELAXED_BLS`. Its definition states:

> "Make bls_g1_negate and bls_g2_negate accept invalid points, as long as they at least have the right number of bytes in the atoms. Hard-fork; enable only when it activates."

A Python caller who passes `flags = 0x0008` silently activates this hard-fork behavior. The `ChiaDialect` constructed from those flags is then used for the entire program execution, and `bls_g1_negate`/`bls_g2_negate` will accept atoms that are the right byte length but encode invalid curve points — inputs that every other node (running without the flag) would reject.

---

### Impact Explanation

The corrupted result is the **return value of `op_bls_g1_negate` or `op_bls_g2_negate`** when given an invalid-but-correctly-sized point atom. With `RELAXED_BLS` set, the operator succeeds and returns a value; without it, the operator raises an error. Any CLVM puzzle that uses these operators with such inputs will produce a different execution outcome (success vs. error) depending on whether the flag is set. A node that enables `RELAXED_BLS` prematurely will accept coin spends that all other nodes reject, causing a consensus split. The broken invariant is: *only flags that have activated on-chain should influence program execution*.

---

### Likelihood Explanation

The `flags` parameter is a "dialect flag exposed by callers" — explicitly listed as an attacker-controlled entry in the threat model. Any Python script, wallet, or tool that passes a user-supplied integer to `run_serialized_chia_program` is directly exploitable. The Chia ecosystem has multiple Python-layer consumers of this API. An attacker who can influence the `flags` argument (e.g., via a misconfigured integration, a CLI tool that forwards raw flag integers, or a Python library that does not whitelist flag values) can activate `RELAXED_BLS` without any on-chain action. The CLVM program bytes themselves do not need to be crafted specially — the flag alone changes operator semantics.

---

### Recommendation

Replace `from_bits_truncate` with strict validation that rejects any bit not in the documented Python-exported set:

```rust
// In run_serialized_chia_program:
const ALLOWED_PYTHON_FLAGS: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::ENABLE_SHA256_TREE)
    .union(ClvmFlags::ENABLE_SECP_OPS)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS);

let raw = ClvmFlags::from_bits(flags)
    .ok_or_else(|| PyValueError::new_err("unknown flag bits"))?;
if raw.difference(ALLOWED_PYTHON_FLAGS) != ClvmFlags::empty() {
    return Err(PyValueError::new_err("flag not permitted via Python API"));
}
let flags = raw;
```

Hard-fork flags (`RELAXED_BLS`, `MALACHITE`, `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`) should only be settable through the node software (chia_rs) after verifying the corresponding fork height, never through the raw Python integer interface.

### Proof of Concept

```python
from clvm_rs import run_serialized_chia_program

# RELAXED_BLS = 0x0008 — not exported by the module, but accepted silently
RELAXED_BLS = 0x0008

# A program that calls bls_g1_negate with an invalid-but-48-byte atom.
# Without RELAXED_BLS: raises an error (invalid G1 point).
# With    RELAXED_BLS: succeeds, returns a value — consensus divergence.
program = bytes.fromhex("...")   # serialized (bls_g1_negate <48-byte-invalid-atom>)
args    = bytes.fromhex("ff80") # ()

# Node A (correct, no flag):
result_a = run_serialized_chia_program(program, args, 11_000_000_000, 0)
# → raises EvalErr (invalid G1 point)

# Node B (attacker-influenced, RELAXED_BLS set):
result_b = run_serialized_chia_program(program, args, 11_000_000_000, RELAXED_BLS)
# → succeeds, returns a NodePtr — consensus split
```
