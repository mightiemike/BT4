### Title
`LastBonusAccrual` Advanced Before Pool Check in `slashRedelegationPosition` Causes Permanent Bonus Loss — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`processEventsAndClaimBonus` advances `pos.LastBonusAccrual` (and `pos.LastEventSeq`, and decrements on-chain event ref counts) **before** calling `sufficientBonusPoolBalance`. When the pool is empty, `ErrInsufficientBonusPool` is returned — but `slashRedelegationPosition` swallows that error and then calls `setPositionWithState`, persisting the already-advanced checkpoint. The accrued bonus for the entire period is permanently unrecoverable.

---

### Finding Description

**Execution order in `processEventsAndClaimBonus`:**

1. **Lines 172–198** — event loop runs: `pos.UpdateLastEventSeq(entry.Seq)` and `k.decrementEventRefCount(...)` are called for every event. The ref-count decrements are **persistent keeper writes** that happen unconditionally.
2. **Line 215** — `applyBonusAccrualCheckpoint(&pos.Position, blockTime)` advances `pos.LastBonusAccrual` on the pointer (in-memory, but the pointer is the caller's `pos`).
3. **Line 217** — `pos.UpdateLastKnownBonded(bonded)` updates bonded state.
4. **Lines 219–221** — early return if `totalBonus.IsZero()` (no error path).
5. **Line 230** — `sufficientBonusPoolBalance` is checked. If the pool is empty, returns `ErrInsufficientBonusPool` **without paying**. [1](#0-0) 

**Back in `slashRedelegationPosition`:**

- `processEventsAndClaimBonus` is called with `&pos` (pointer), so all in-memory mutations to `pos.LastBonusAccrual` and `pos.LastEventSeq` are visible in the caller's `pos`.
- `ErrInsufficientBonusPool` is caught and **swallowed** (logged only).
- `setPositionWithState(ctx, pos, ...)` is then called unconditionally, persisting the advanced `LastBonusAccrual` and `LastEventSeq`. [2](#0-1) 

**Result:** The position's `LastBonusAccrual` is now at `blockTime`, and `LastEventSeq` has consumed all pending events. Any future call to `processEventsAndClaimBonus` will compute zero bonus for the already-advanced window — the accrued rewards are gone.

---

### Impact Explanation

The position owner permanently loses all bonus rewards that accrued since their last successful claim. The invariant that `LastBonusAccrual` must not advance past the last **successfully paid** bonus segment is violated. Even after the `RewardsPoolName` is refilled, a subsequent `processEventsAndClaimBonus` call starts from the already-advanced `LastBonusAccrual` and finds no new accrual window to pay.

Additionally, the event ref counts are decremented inside the loop (line 196) before the pool check, so those events are consumed from the validator event log regardless of whether the bonus is paid. [3](#0-2) 

---

### Likelihood Explanation

The `RewardsPoolName` balance reaching zero is a realistic operational condition — the pool is funded externally and can be drained by concurrent claims from other positions. The `BeforeRedelegationSlashed` hook fires automatically during any redelegation slash event (a normal staking operation), requiring no special attacker privilege. [4](#0-3) 

The attacker's only requirement is to have an active tiered-rewards position with a pending redelegation and to time the slash when the pool is empty — a condition that can be engineered by draining the pool via legitimate claims from other positions.

---

### Recommendation

Move `applyBonusAccrualCheckpoint` and `UpdateLastKnownBonded` to **after** the `sufficientBonusPoolBalance` check succeeds and the `SendCoinsFromModuleToAccount` call completes. Alternatively, if the intent is to deliberately skip bonus payment when the pool is insufficient, the checkpoint must **not** be advanced and the event ref counts must **not** be decremented in that code path — the position should be left in its pre-call state so the next successful claim can cover the full accrual window. [5](#0-4) 

---

### Proof of Concept

```
1. Create a tiered-rewards position with an active delegation to validator V.
2. Trigger a redelegation from V to V2 (creates a redelegation entry with unbondingId U).
3. Let some blocks pass so bonus accrues (LastBonusAccrual < blockTime).
4. Drain RewardsPoolName to zero (e.g., via legitimate claims from other positions).
5. Trigger a slash on the redelegation entry U (e.g., via evidence submission against V).
   → BeforeRedelegationSlashed fires → slashRedelegationPosition(ctx, U, shares).
   → processEventsAndClaimBonus runs:
       - event loop advances LastEventSeq, decrements ref counts (persistent).
       - applyBonusAccrualCheckpoint advances LastBonusAccrual to blockTime.
       - sufficientBonusPoolBalance returns ErrInsufficientBonusPool.
   → slashRedelegationPosition swallows the error.
   → setPositionWithState persists pos with advanced LastBonusAccrual.
6. Refill RewardsPoolName.
7. Call processEventsAndClaimBonus again for the same position.
   → segmentStart = LastBonusAccrual = blockTime (from step 5).
   → No new accrual window exists; totalBonus = 0.
   → Owner receives zero bonus despite having accrued rewards in step 3.
``` [6](#0-5) [7](#0-6)

func applyBonusAccrualCheckpoint(pos *types.Position, blockTime time.Time) {
	accrualEnd := blockTime
	if pos.CompletedExitLockDuration(blockTime) {
		accrualEnd = pos.ExitUnlockAt
	}
	pos.UpdateLastBonusAccrual(accrualEnd)
}
