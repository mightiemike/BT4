### Title
`ValidResourceBounds` Protobuf Deserialization Silently Downgrades `AllResources` to `L1Gas` on Zero L2/L1-Data Gas, Producing a Divergent Transaction Hash — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a zero-value sentinel check to distinguish old `L1Gas`-only (pre-0.13.3) transactions from new `AllResources` transactions. When a valid `AllResources` V3 transaction carries zero `l2_gas` and zero `l1_data_gas` bounds, the deserializer silently converts it to `ValidResourceBounds::L1Gas`. Because `get_tip_resource_bounds_hash` includes the `L1_DATA_GAS` field in the Poseidon preimage only for `AllResources`, the transaction hash computed after the protobuf round-trip differs from the hash computed at submission time, binding the wrong hash to the transaction across the P2P boundary.

### Finding Description

**Root cause — `crates/apollo_protobuf/src/converters/transaction.rs`, line 431:**

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong for AllResources with zero bounds
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

The intent is to detect legacy 0.13.2 transactions that only carried L1 gas. However, any `AllResources` transaction whose `l2_gas` and `l1_data_gas` are both zero satisfies the same condition and is silently re-typed as `L1Gas`.

**Hash divergence — `crates/starknet_api/src/transaction_hash.rs`, lines 188–210:**

`get_tip_resource_bounds_hash` branches on the enum variant:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // no L1_DATA_GAS element
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // extra element
    }
});
``` [2](#0-1) 

Consequently the Poseidon preimage differs by exactly one field:

| Variant | Preimage elements |
|---|---|
| `AllResources` (zero l2/l1_data) | `[tip, L1_GAS_concat, L2_GAS_concat(0), L1_DATA_GAS_concat(0)]` |
| `L1Gas` (after protobuf round-trip) | `[tip, L1_GAS_concat, L2_GAS_concat(0)]` |

The resulting `tip_resource_bounds_hash` is different, so `get_invoke_transaction_v3_hash` produces a different `TransactionHash`. [3](#0-2) 

**Serialization side is consistent but asymmetric.** The `From<ValidResourceBounds> for protobuf::ResourceBounds` serializer always emits all three fields (setting `l1_data_gas` to `ResourceBounds::default()` for `L1Gas`), so the wire bytes are identical for both variants when l2/l1_data are zero. The asymmetry is entirely in the deserializer's type decision. [4](#0-3) 

**Trigger path.** `RpcInvokeTransactionV3` always uses `AllResourceBounds` (the gateway-facing type), so every V3 transaction submitted through the RPC is `AllResources`. The gateway's resource-bounds validator only rejects transactions where the *total* max fee is zero; a transaction with non-zero `l1_gas` but zero `l2_gas` and `l1_data_gas` passes validation and enters the mempool as `AllResources`. [5](#0-4) 

When this transaction is propagated over P2P, the protobuf round-trip converts it to `L1Gas`, and any node that re-derives the hash from the deserialized fields (e.g., for mempool admission, block-hash commitment, or signature-domain verification) computes a hash that does not match the hash the signer committed to.

### Impact Explanation

The wrong `ValidResourceBounds` variant is bound to the transaction after P2P deserialization. Any component that calls `calculate_transaction_hash` on the deserialized transaction — including hash-based mempool deduplication, block-hash construction, and the account-contract signature-domain check via `get_execution_info` — receives a hash that differs from the one the user signed. This matches the impact category: **"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."** A valid, correctly-signed transaction can be rejected or committed under the wrong hash.

### Likelihood Explanation

The trigger requires only that a user submit a V3 invoke with `l2_gas = 0` and `l1_data_gas = 0` (non-zero `l1_gas`). This is a natural configuration for wallets that have not yet adopted L2-gas pricing or that set conservative zero bounds for data-gas. No privileged access or malformed bytes are needed; the transaction passes all gateway-side validation.

### Recommendation

Remove the zero-value sentinel and instead preserve the variant that was transmitted. The cleanest fix is to always deserialize as `AllResources` when all three fields are present in the protobuf message, and only fall back to `L1Gas` when `l2_gas` is absent (i.e., the field is `None`, not merely zero). The existing `TODO` comment already acknowledges this:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
```

Once 0.13.2 support is dropped, the correct fix is:

```rust
let Some(l1_data_gas) = value.l1_data_gas else {
    return Err(missing("ResourceBounds::l1_data_gas"));
};
// Always AllResources for V3 transactions.
Ok(ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }))
```

Until then, the sentinel should check `l1_data_gas.is_none()` (field absent) rather than `l1_data_gas.is_zero()` (field present but zero).

### Proof of Concept

```
1. Construct AllResources V3 invoke:
     l1_gas = { max_amount: 1000, max_price_per_unit: 1 }
     l2_gas = { max_amount: 0,    max_price_per_unit: 0 }   ← zero
     l1_data_gas = { max_amount: 0, max_price_per_unit: 0 } ← zero

2. Compute hash_A = get_invoke_transaction_v3_hash(tx)
   tip_resource_bounds_hash_A = poseidon(tip, L1_GAS_concat, L2_GAS_concat(0), L1_DATA_GAS_concat(0))

3. Serialize tx → protobuf bytes (all three ResourceLimits fields present, l2/l1_data = 0)

4. Deserialize protobuf bytes:
   l1_data_gas.is_zero() && l2_gas.is_zero() == true
   → ValidResourceBounds::L1Gas(l1_gas)   ← type changed

5. Compute hash_B = get_invoke_transaction_v3_hash(deserialized_tx)
   tip_resource_bounds_hash_B = poseidon(tip, L1_GAS_concat, L2_GAS_concat(0))
                                 ← L1_DATA_GAS_concat(0) absent

6. hash_A ≠ hash_B  ← divergent hash; user's signature is over hash_A
```

```rust
pub(crate) fn get_invoke_transaction_v3_hash<T: InvokeTransactionV3Trait>(
    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();
    let calldata_hash =
        HashChain::new().chain_iter(transaction.calldata().0.iter()).get_poseidon_hash();
    let mut hash_chain = HashChain::new()
        .chain(&INVOKE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address().0.key())
        .chain(&tip_resource_bounds_hash)
        .chain(&paymaster_data_hash)
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce().0)
        .chain(&data_availability_mode)
        .chain(&account_deployment_data_hash)
        .chain(&calldata_hash);
    if !transaction.proof_facts().0.is_empty() {
        let proof_facts_hash =
            HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
        hash_chain = hash_chain.chain(&proof_facts_hash);
    }
    Ok(TransactionHash(hash_chain.get_poseidon_hash()))
}
