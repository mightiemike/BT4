### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables usdcE Drain from DirectDepositV1 Addresses Without Providing USDC — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` is an unpermissioned `external` function that calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the return value. If the USDC token returns `false` on failure rather than reverting, execution continues: the DDA's usdcE balance is withdrawn to `ContractOwner` and then forwarded to `msg.sender` via `safeTransfer`. An unprivileged caller can drain usdcE from any DDA without providing USDC.

---

### Finding Description

`replaceUsdcEWithUsdc` is intended to swap usdcE held in a `DirectDepositV1` (DDA) for USDC by pulling USDC from the caller and returning usdcE. The function has no access control modifier — only a chain-ID guard — making it callable by any address on Ink Chain (chain ID 57073).

The critical flaw is at line 616:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
```

This is a raw `transferFrom` call whose boolean return value is silently discarded. Every other token transfer in the protocol uses `ERC20Helper.safeTransferFrom` / `safeTransfer`, which wraps the call and reverts on `false`. This one call does not.

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` returns `false` on an insufficient-allowance or insufficient-balance condition (rather than reverting), execution falls through to:

```solidity
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));   // pulls usdcE → ContractOwner
IERC20Base(usdcE).safeTransfer(msg.sender, balance);              // forwards usdcE → attacker
```

The attacker receives the full usdcE balance of the DDA without having transferred any USDC. [1](#0-0) 

The contrast with the rest of the codebase is stark: `EndpointStorage.safeTransferFrom`, `BaseWithdrawPool.safeTransferFrom`, and `ERC20Helper.safeTransferFrom` all wrap the call and revert on failure. [2](#0-1) [3](#0-2) 

---

### Impact Explanation

- **Asset delta**: The usdcE balance of any DDA whose `subaccount` is known is fully drained to the attacker. `ContractOwner` is the owner of all `DirectDepositV1` instances (evidenced by `withdrawFromDirectDepositV1` calling `DirectDepositV1.withdraw` which is `onlyOwner`), so the withdrawal step succeeds unconditionally.
- **Corrupted invariant**: The protocol assumes `replaceUsdcEWithUsdc` is a 1-for-1 swap. If the USDC pull silently fails, the swap becomes a free withdrawal of usdcE.
- **Scope**: All DDAs on Ink Mainnet (chain ID 57073) holding usdcE at `0xF1815bd50389c46847f0Bda824eC8da914045D14`. [4](#0-3) 

---

### Likelihood Explanation

The function is `external` with no `onlyOwner` or role guard — only `require(block.chainid == 57073)`. Any EOA or contract on Ink Mainnet can call it. The exploitability depends on whether the specific USDC token at the hardcoded address returns `false` or reverts on failure. Many ERC20 tokens (including older USDC versions and non-OpenZeppelin implementations) return `false` rather than reverting, making this a realistic attack path. The attacker needs only to know a `subaccount` with a non-empty DDA — subaccount IDs are recorded on-chain via `Endpoint` events. [5](#0-4) 

---

### Recommendation

Replace the bare `transferFrom` call with the protocol's own safe wrapper:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`safeTransferFrom` from `ERC20Helper` already handles both reverting and return-value-returning tokens correctly. [2](#0-1) 

---

### Proof of Concept

1. Identify a `subaccount` whose `directDepositV1Address` is non-zero and whose DDA holds usdcE (balance > 0).
2. Call `replaceUsdcEWithUsdc(subaccount)` from an EOA with **zero USDC allowance** to `ContractOwner`.
3. `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` returns `false` (no revert) because allowance is zero.
4. Execution continues: `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))` transfers the full usdcE balance from the DDA to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` transfers the usdcE from `ContractOwner` to the attacker.
6. Attacker holds usdcE; no USDC was ever transferred. [1](#0-0) [6](#0-5)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
