Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the End User, Allowing Any Caller to Bypass the Swap Allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `sender`, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router, not the end user. If the router is allowlisted — the only way to permit legitimate users to use the router — every caller, including non-allowlisted users, bypasses the gate entirely.

## Finding Description

In `MetricOmmPool.swap`, the pool passes `msg.sender` (the router) as the first argument to `_beforeSwap`: [1](#0-0) 

In `SwapAllowlistExtension.beforeSwap`, the check is `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the first argument — the router address: [2](#0-1) 

In `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)` directly, making itself `msg.sender` at the pool level: [3](#0-2) 

The result is that `allowedSwapper[pool][router]` is evaluated, not `allowedSwapper[pool][user]`. The actual end user is never checked.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` (the second argument), which the pool sets to the actual position owner: [4](#0-3) 

The swap path has no equivalent `owner`-style argument — the only identity forwarded is `msg.sender`, which collapses to the router.

## Impact Explanation

A pool deploying `SwapAllowlistExtension` to restrict swaps to KYC'd or otherwise vetted addresses is fully bypassed. Any unprivileged user calls `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) targeting the restricted pool. The extension sees `sender = router`; if the router is allowlisted, the check passes for all callers. Unauthorized users execute swaps in a pool designed to exclude them, accessing oracle-derived prices the pool admin intended only for approved counterparties. This constitutes a broken core access-control invariant with direct fund-impact potential (unauthorized swap execution against restricted liquidity).

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing entry point for the protocol. Any pool admin who wants allowlisted users to use the router must add the router to the allowlist. Once the router is allowlisted, the gate is open to all callers. The attack requires no special privileges, no flash loans, and no oracle manipulation — a single standard router call suffices. The condition (router allowlisted) is the expected operational state, making this reliably exploitable.

## Recommendation

Pass the economically relevant identity through the swap path, mirroring `DepositAllowlistExtension`. The cleanest fix is to add a `swapper` field to the `_beforeSwap` call (analogous to `owner` in `_beforeAddLiquidity`) that the pool sets to `msg.sender` and the extension checks instead of `sender`. Alternatively, require that allowlisted pools be called directly (not through the router) and enforce this via a trusted router registry, so the extension can gate the correct identity.

## Proof of Concept

```
Setup:
  - Pool P configured with SwapAllowlistExtension
  - allowedSwapper[P][alice] = true   (alice is KYC'd)
  - allowedSwapper[P][router] = true  (router added so alice can use it)
  - bob is NOT on the allowlist

Attack:
  1. bob calls router.exactInputSingle({pool: P, ...})
  2. router calls P.swap(recipient=bob, ...)  →  msg.sender = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. Extension checks allowedSwapper[P][router] == true  →  PASSES
  5. bob's swap executes against the restricted pool

Result:
  - bob, an unauthorized user, swaps in a pool the admin intended to restrict
  - The allowlist provides zero protection for router-mediated swaps
```
