### Title
Zero-Threshold Signature Bypass in Fast Withdrawal Authorization — (`File: core/contracts/Verifier.sol`, `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`Verifier.requireValidTxSignatures` uses the storage variable `nSigner` as the required signature count. `nSigner` defaults to `0` and is only incremented when pubkeys are assigned. When `nSigner == 0`, the final check `require(nSignatures == nSigner)` reduces to `require(0 == 0)`, which always passes. Because `BaseWithdrawPool.submitFastWithdrawal` is a public function that calls `requireValidTxSignatures` as its sole authorization gate, any caller can drain the pool by submitting an empty signatures array whenever `nSigner` is zero.

---

### Finding Description

`Verifier.requireValidTxSignatures` counts non-empty entries in the caller-supplied `signatures` array and then asserts that count equals `nSigner`:

```solidity
uint256 nSignatures = 0;
for (uint256 i = 0; i < signatures.length; i++) {
    if (signatures[i].length > 0) {
        nSignatures += 1;
        require(checkIndividualSignature(...), "invalid signature");
    }
}
require(nSignatures == nSigner, "not enough signatures");
``` [1](#0-0) 

`nSigner` is a plain storage `uint256` that starts at `0` and is incremented only inside `_assignPubkey`, which is called from the owner-gated `assignPubKey` and from `initialize`. [2](#0-1) 

When `nSigner == 0`, passing `signatures = []` (empty array) causes the loop body to never execute, leaving `nSignatures = 0`, and `require(0 == 0)` passes unconditionally. There is no guard that rejects the zero-threshold state as invalid.

`BaseWithdrawPool.submitFastWithdrawal` is `public`, callable by any EOA or contract, and delegates its entire authorization to `requireValidTxSignatures`:

```solidity
function submitFastWithdrawal(
    uint64 idx,
    bytes calldata transaction,
    bytes[] calldata signatures
) public {
    require(!markedIdxs[idx], "Withdrawal already submitted");
    require(idx > minIdx, "idx too small");
    markedIdxs[idx] = true;

    Verifier v = Verifier(verifier);
    v.requireValidTxSignatures(transaction, idx, signatures);

    (uint32 productId, address sendTo, uint128 transferAmount) = resolveFastWithdrawal(transaction);
    ...
    handleWithdrawTransfer(token, sendTo, transferAmount);
}
``` [3](#0-2) 

`resolveFastWithdrawal` decodes the caller-supplied `transaction` bytes and returns the `sendTo` address and `transferAmount` directly from the payload — no on-chain balance check, no sequencer-issued nonce, no proof that the subaccount authorized the withdrawal: [4](#0-3) 

The structural parallel to DLT-01 is exact:

| DLT-01 (Nomad) | Nado analog |
|---|---|
| `messages[hash]` defaults to `bytes32(0)` | `nSigner` defaults to `0` |
| `acceptableRoot(bytes32(0)) == true` | `require(0 == nSigner)` passes |
| Unproven message accepted | Unsigned withdrawal accepted |
| Drain bridge | Drain `WithdrawPool` |

---

### Impact Explanation

An attacker who calls `submitFastWithdrawal` while `nSigner == 0` can:

1. Craft a `WithdrawCollateralV2` payload with `sendTo = attacker_address` and `amount = pool_balance`.
2. Pass `signatures = []`.
3. `requireValidTxSignatures` passes (`0 == 0`).
4. `resolveFastWithdrawal` returns the attacker-controlled address and amount.
5. `handleWithdrawTransfer` sends the full pool balance to the attacker.

The `markedIdxs` replay guard only prevents the same `idx` from being reused; it does not prevent a fresh `idx` from being used for a new forged withdrawal. The `idx > minIdx` check is trivially satisfied by choosing any `idx >= 1` on a fresh pool.

**Corrupted state**: complete drainage of all ERC-20 token balances held by `WithdrawPool` / `BaseWithdrawPool`.

---

### Likelihood Explanation

`nSigner == 0` is reachable in two realistic scenarios without requiring a persistent attacker-controlled admin:

**Scenario A — Deployment window**: The standard deployment sequence is: deploy `Verifier` → deploy `WithdrawPool` (pointing at `Verifier`) → fund pool → call `Verifier.initialize(pubkeys)`. Between funding and initialization, `nSigner == 0`. Any mempool watcher can front-run `initialize` to drain the pool.

**Scenario B — Key rotation gap**: The owner calls `deletePubkey` for all active keys before calling `assignPubKey` with new keys. During this window `nSigner` transiently returns to `0`. The `deletePubkey` function decrements `nSigner` with no floor guard: [5](#0-4) 

Both scenarios are operationally plausible and create a window exploitable by any unprivileged caller watching the chain.

---

### Recommendation

1. **Reject the zero-threshold state explicitly** in `requireValidTxSignatures`:
   ```solidity
   require(nSigner > 0, "no signers registered");
   require(nSignatures == nSigner, "not enough signatures");
   ```

2. **Enforce a minimum signer count** in `deletePubkey` to prevent `nSigner` from reaching zero while the pool holds funds.

3. **Add an invariant check** in `submitFastWithdrawal` that `nSigner > 0` before delegating to `requireValidTxSignatures`, so the pool itself is not silently open even if the verifier is misconfigured.

4. **Separate "uninitialized" from "zero required"** — use a sentinel value or an explicit `initialized` flag in `Verifier` so that the zero default is never treated as a valid threshold.

---

    function requireValidTxSignatures(
        bytes calldata txn,
        uint64 idx,
        bytes[] calldata signatures
    ) public view {
        require(signatures.length <= 256, "too many signatures");
        bytes32 data = keccak256(
            abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
        );
        bytes32 hashedMsg = keccak256(
            abi.encodePacked("\x19Ethereum Signed Message:\n32", data)
        );

        uint256 nSignatures = 0;
        for (uint256 i = 0; i < signatures.length; i++) {
            if (signatures[i].length > 0) {
                nSignatures += 1;
                require(
                    checkIndividualSignature(
                        hashedMsg,
                        signatures[i],
                        uint8(i)
                    ),
                    "invalid signature"
                );
            }
        }
        require(nSignatures == nSigner, "not enough signatures");
    }
```

    function resolveFastWithdrawal(bytes calldata transaction)
        internal
        pure
        returns (
            uint32 productId,
            address sendTo,
            uint128 amount
        )
    {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );
        if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
            IEndpoint.SignedWithdrawCollateral memory signedTx = abi.decode(
                transaction[1:],
                (IEndpoint.SignedWithdrawCollateral)
            );
            return (
                signedTx.tx.productId,
                address(uint160(bytes20(signedTx.tx.sender))),
                signedTx.tx.amount
            );
        }
        if (txType == IEndpoint.TransactionType.WithdrawCollateralV2) {
            IEndpoint.SignedWithdrawCollateralV2 memory signedTx = abi.decode(
                transaction[1:],
                (IEndpoint.SignedWithdrawCollateralV2)
            );
            // V2 appendix is intentionally ignored until fast-withdraw features use it.
            address resolvedSendTo = signedTx.tx.sendTo == address(0)
                ? address(uint160(bytes20(signedTx.tx.sender)))
                : signedTx.tx.sendTo;
            return (signedTx.tx.productId, resolvedSendTo, signedTx.tx.amount);
        }
        revert("Invalid withdrawal tx type");
    }

        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
