### Title
Difficulty Retarget Calculation for Fork Blocks Always Reads Ancestor from Main Chain Instead of Fork Chain — (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

### Summary

When a fork block is submitted at a retarget boundary, the difficulty calculation calls `get_header_by_height` to fetch the first block of the retarget interval. This function unconditionally reads from `mainchain_height_to_header`, returning the **main-chain** block at that height rather than the fork's actual ancestor. For Dogecoin after block 145,000 (where every block is a retarget boundary), this affects every fork block that is two or more blocks deep from the divergence point. The developers themselves flagged this with a TODO comment. The result is that the light client enforces an incorrect difficulty for fork blocks, allowing an attacker to submit fork blocks that satisfy a lower PoW threshold than the actual Dogecoin consensus requires.

---

### Finding Description

**Root cause — `get_header_by_height` is main-chain-only:** [1](#0-0) 

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // ← always the canonical chain
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

Fork blocks are stored only in `headers_pool` (keyed by hash), never in `mainchain_height_to_header`. So any call to `get_header_by_height` during fork-block processing silently returns the main-chain block at that height.

**Dogecoin — every block is a retarget after height 145,000:** [2](#0-1) 

```rust
let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
let difficulty_adjustment_interval = if new_difficulty_protocol { 1 } else { ... };
```

With `difficulty_adjustment_interval = 1`, the retarget path is taken for every block. The retarget then calls: [3](#0-2) 

```rust
// TODO: check if it is correct to get block header by height from mainchain
//       without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)   // ← returns main-chain block, not fork ancestor
    .block_header
    .time;
```

The developers themselves flagged this with a TODO. The same uncorrected pattern appears in Bitcoin and Litecoin: [4](#0-3) [5](#0-4) 

**Contrast with `get_prev_header`, which is fork-aware:** [6](#0-5) 

```rust
fn get_prev_header(&self, current_header: &LightHeader) -> ExtendedHeader {
    self.headers_pool.get(&current_header.prev_block_hash)  // ← hash-keyed, works for forks
        .unwrap_or_else(|| env::panic_str("PrevBlockNotFound"))
}
```

`get_prev_header` correctly follows the fork chain via hash links. `get_header_by_height` does not.

**Where fork blocks are stored (height map is never updated for forks):** [7](#0-6) 

```rust
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);  // ← no mainchain_height_to_header entry
}
```

### Impact Explanation

Consider a Dogecoin fork diverging at height N (after 145,000). The fork block at N has attacker-controlled timestamp `T_fork_N`; the main-chain block at N has timestamp `T_main_N`.

When the contract validates fork block at height N+2:
- It calls `get_header_by_height(N)` → returns main-chain block with timestamp `T_main_N`
- Correct behavior: use fork block at N with timestamp `T_fork_N`
- The computed timespan is `T_fork_N1 − T_main_N` instead of `T_fork_N1 − T_fork_N`

If the attacker sets `T_fork_N` significantly later than `T_main_N` (up to `MAX_FUTURE_BLOCK_TIME_LOCAL` = 7200 s ahead of local time), the computed timespan becomes larger than the correct one, and the Dogecoin dampened retarget formula: [8](#0-7) 

```rust
let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;
let max_timespan = retarget_timespan + (retarget_timespan / 2);
```

…caps the difficulty reduction at 50% per block. Over a multi-block fork, the attacker can accumulate a chain where each block requires only ~50% of the PoW the actual Dogecoin network demands. The light client accepts these blocks as valid; the actual Dogecoin network would reject them. Downstream NEAR contracts calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` against this fraudulent fork would receive incorrect `true` results. [9](#0-8) 

---

### Likelihood Explanation

- **Dogecoin (high):** After block 145,000, every block is a retarget boundary. Any fork two or more blocks deep triggers the bug on every subsequent block. An attacker only needs to submit a fork starting at any height above 145,000.
- **Bitcoin / Litecoin (medium):** The retarget interval is 2,016 blocks. The bug fires only when a fork crosses a 2,016-block boundary, which is less frequent but still reachable by any unprivileged `submit_blocks` caller.

---

### Recommendation

Replace `get_header_by_height` in the retarget path with an ancestor walk that follows `prev_block_hash` links through `headers_pool` starting from the current fork tip, until the block at the required height is reached. This is the same approach already used by `get_prev_header` and by `zcash_get_next_work_required` (which walks backward via `get_prev_header` for its averaging window and never calls `get_header_by_height`). [10](#0-9) 

---

### Proof of Concept

1. Deploy the Dogecoin contract with mainnet genesis at height ≥ 145,000.
2. Submit main-chain blocks up to height N. Record `T_main_N` (timestamp of main-chain block N).
3. Submit fork block `fork_N` at height N with timestamp `T_fork_N = T_main_N + 7000` (just under the 7200 s future-time limit). This block diverges from the main chain.
4. Submit fork block `fork_N1` at height N+1 with any valid timestamp `T_fork_N1`.
5. Submit fork block `fork_N2` at height N+2 with `bits` = value computed from timespan `T_fork_N1 − T_main_N` (the incorrect timespan the contract will use).
6. Observe: the contract accepts `fork_N2` with `bits` encoding ~50% lower difficulty than the actual Dogecoin network would require for a block at that height, because `get_header_by_height(N)` returned the main-chain block (timestamp `T_main_N`) instead of `fork_N` (timestamp `T_fork_N`), inflating the computed timespan and reducing the expected difficulty.
