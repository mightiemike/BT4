### Title
`alignVestingDelegationTracking` Misclassifies LockTier-Origin Delegation as DelegatedVesting, Unlocking Locked Vesting Tokens at v8 Migration — (`x/tieredrewards/keeper/force_exit.go`)

---

### Summary

`alignVestingDelegationTracking` calls `vacc.TrackDelegation(blockTime, deficit, deficit)` to reconcile a vesting account's DV+DF after `ForceFullExitWithDelegation` returns a delegation to the owner. For a LockTier-origin position, the delegation was funded by the owner's **spendable** tokens (not vesting tokens), so the correct classification is `DelegatedFree += deficit`. Instead, `TrackDelegation` uses `vestingCoins(blockTime)` to determine the DV/DF split and — when `vestingCoins ≥ deficit` — sets `DelegatedVesting += deficit`. This reduces `lockedCoins = max(0, vestingCoins − DV)` by `deficit`, making that many previously-locked bank tokens spendable. The bug is triggered automatically by the v8 migration for every vesting-owned LockTier position.

---

### Finding Description

**Root cause — `alignVestingDelegationTracking`, line 171:**

```go
vacc.TrackDelegation(sdkCtx.BlockTime(), coins, coins)
```

The comment at lines 165–169 correctly notes that `balance` is only used for the `amount ≤ balance` invariant check, and that the DV/DF split is computed from `vestingCoins(blockTime)` and the existing `DelegatedVesting`. What the comment misses is that this split is **wrong** for LockTier-origin positions.

**SDK `TrackDelegation` split logic (ContinuousVestingAccount):**

```
x = min(amount, vestingCoins(blockTime) − DV)   // → DelegatedVesting
y = amount − x                                   // → DelegatedFree
```

**LockTier-origin state before migration:**

| Field | Value |
|---|---|
| `OriginalVesting` | V |
| `bank balance` | V (locked portion; spendable S was sent to position) |
| `DelegatedVesting` | 0 |
| `DelegatedFree` | 0 |
| `vestingCoins(t)` | ≈ V (early in schedule) |

**After `transferDelegationFromPosition` returns S tokens as delegation:**

- `actualDelegated = S`, `tracked = 0`, `deficit = S`
- `TrackDelegation(t, S, S)` → `x = min(S, V − 0) = S` → **DV = S**
- `lockedCoins = max(0, V − S)` ← reduced from V
- `SpendableCoins = max(0, V − (V − S)) = S` ← **locked tokens now spendable**

The correct result would be `DelegatedFree += S` (delegation came from free tokens), leaving `lockedCoins = V` and `SpendableCoins = 0`.

**The existing test `TestForceFullExitWithDelegation_VestingOwner_LockOrigin` explicitly asserts `DV = lockedAmount` (line 241) and `DF = 0` (line 243), confirming the misclassification occurs. It does not assert `SpendableCoins = 0`, so the invariant break is undetected.**

The `TestForceFullExitWithDelegation_VestingOwner_Mixed` test only checks `spendable ≤ balance` (line 309), which is trivially satisfied and does not catch the exploit.

---

### Impact Explanation

After the v8 migration runs `exitVestedAccountsPositions` → `ForceFullExitWithDelegation` for a LockTier-origin vesting position:

- `DelegatedVesting` is inflated by `S` (the position amount)
- `lockedCoins` is reduced by `S`
- The owner can immediately `bank.Send` or spend `S` tokens that the vesting schedule requires to remain locked

For a `PermanentLockedAccount` with `OriginalVesting = V` and a LockTier position of size `V`, the entire locked principal becomes spendable. For a `ContinuousVestingAccount` early in the schedule, the impact scales with `min(S, vestingCoins)`.

---

### Likelihood Explanation

The exploit is triggered automatically by the v8 upgrade migration — no post-upgrade user action is required. The attacker only needs to:

1. Hold a vesting account (`ContinuousVestingAccount` or `PermanentLockedAccount`) with locked tokens — a common genesis configuration for team/investor allocations.
2. Create a LockTier position funded by spendable tokens before the v8 upgrade.

Both preconditions are achievable through normal production paths (`LockTier` message, genesis vesting account setup). The migration unconditionally calls `ForceFullExitWithDelegation` for every vesting-owned position.

---

### Recommendation

In `alignVestingDelegationTracking`, do not use `TrackDelegation` to classify the deficit. Instead, directly increment `DelegatedFree` by `deficit`, since the returning delegation for a LockTier-origin position came from spendable tokens:

```go
// Instead of vacc.TrackDelegation(sdkCtx.BlockTime(), coins, coins):
// Directly set DF += deficit, preserving the vesting lock.
// DV must not be inflated by tokens that were never vesting-delegated.
```

Alternatively, compute the correct split by comparing `deficit` against `max(0, vestingCoins − DV)` and routing only the truly-vesting portion to DV and the remainder to DF. Add an assertion `SpendableCoins(owner) == 0` (for fully-locked accounts) to the test suite.

### Proof of Concept

```
Setup:
  ContinuousVestingAccount: OriginalVesting=1_000_000, balance=2_000_000
  vestingStart=T0, vestingEnd=T0+4years
  blockTime = T0 (early in schedule, vestingCoins ≈ 1_000_000)

Step 1: LockTier(owner, validator, 1_000_000)
  → bank.Send(owner → posDelAddr, 1_000_000)  [spendable tokens]
  → owner.balance = 1_000_000 (locked portion)
  → DV=0, DF=0

Step 2: v8 upgrade triggers ForceFullExitWithDelegation(posID)
  → transferDelegationFromPosition returns 1_000_000 as delegation to owner
  → actualDelegated=1_000_000, tracked=0, deficit=1_000_000
  → TrackDelegation(T0, 1_000_000, 1_000_000):
       vestingCoins(T0) = 1_000_000
       x = min(1_000_000, 1_000_000 − 0) = 1_000_000 → DV=1_000_000
  → lockedCoins = max(0, 1_000_000 − 1_000_000) = 0
  → SpendableCoins = 1_000_000  ← should be 0

Step 3: attacker calls bank.Send(owner → attacker2, 1_000_000)
  → succeeds; locked principal drained
```
