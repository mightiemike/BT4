### Title
`ClaimTierRewards` Blocks Base Reward Withdrawal When Bonus Pool Is Insufficient — (`x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

In `claimRewardsAndUpdatesPositions`, base staking rewards are withdrawn first via `claimBaseRewards`, but if the subsequent `processEventsAndClaimBonus` call returns `ErrInsufficientBonusPool`, the entire transaction reverts. Because each position's delegator address is a module-internal derived address, `ClaimTierRewards` is the **only** path to claim base rewards. An empty or insufficient bonus pool therefore permanently blocks base reward access until the pool is refunded.

---

### Finding Description

`claimRewardsAndUpdatesPositions` processes both reward types sequentially:

```go
base, err := k.claimBaseRewards(ctx, *pos)   // withdraws from distribution
if err != nil {
    return nil, nil, err
}
totalBase = totalBase.Add(base...)

bonus, err := k.processEventsAndClaimBonus(ctx, pos)  // checks pool balance
if err != nil {
    return nil, nil, err   // rolls back everything, including base
}
``` [1](#0-0) 

Inside `processEventsAndClaimBonus`, after computing `totalBonus`, the pool check fires:

```go
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err
}
``` [2](#0-1) 

`sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool` whenever the `RewardsPoolName` module account cannot cover the accrued bonus: [3](#0-2) 

This error propagates through `ClaimTierRewards` in `msg_server.go`, causing the entire Cosmos SDK message to revert: [4](#0-3) 

The base rewards that `claimBaseRewards` already withdrew via `WithdrawDelegationRewards` are rolled back with the transaction.

The critical constraint is that each position's delegator address (`pos.DelegatorAddress`) is a module-derived address created at lock time: [5](#0-4) 

Users have no direct access to this address and cannot call `distribution.WithdrawDelegationRewards` themselves. `ClaimTierRewards` is the sole path to retrieve base rewards.

The same `claimRewards` helper is called inside `TierUndelegate`, `TierRedelegate`, `AddToTierPosition`, `ClearPosition`, and `ExitTierWithDelegation`: [6](#0-5) [7](#0-6) 

So an empty bonus pool also blocks all position lifecycle operations, including undelegation.

The codebase itself acknowledges this asymmetry: the slash hook (`slashRedelegationPosition`) explicitly catches `ErrInsufficientBonusPool` and forges ahead without bonus to prevent a chain halt, but the user-facing message handlers do not: [8](#0-7) 

---

### Impact Explanation

A position owner with accrued base staking rewards cannot withdraw them when the bonus pool is empty or insufficient. The base rewards remain locked in the distribution module. Additionally, all position lifecycle operations (`TierUndelegate`, `TierRedelegate`, `AddToTierPosition`, `ClearPosition`, `ExitTierWithDelegation`) fail for the same reason, meaning users cannot exit their locked positions. The corrupted invariant is: **earned base staking rewards must be claimable independently of the bonus pool state**.

---

### Likelihood Explanation

The `RewardsPoolName` module account is shared between bonus reward payouts and the `topUpBaseRewards` BeginBlocker mechanism, which drains the pool every block when staking rewards fall below `TargetBaseRewardsRate`: [9](#0-8) 

Under sustained load or if the pool is not topped up by governance, the pool balance can reach zero. Any position that has accrued bonus (i.e., any delegated position that has been active for more than a few seconds) will then fail `ClaimTierRewards`. The test `TestMsgClaimTierRewards_FailsWhenBonusPoolInsufficient` explicitly demonstrates this reachable state: [10](#0-9) 

---

### Recommendation

Decouple base reward withdrawal from bonus reward payment. If `processEventsAndClaimBonus` fails with `ErrInsufficientBonusPool`, `claimRewardsAndUpdatesPositions` should still commit the base rewards and advance the position's bonus checkpoint (recording the debt), rather than reverting the entire operation. Alternatively, expose a separate `ClaimBaseRewards`-only message path that bypasses the bonus pool check entirely, mirroring the pattern already used in `slashRedelegationPosition`.

---

### Proof of Concept

1. Create a tier position via `MsgLockTier` and let time advance so both base and bonus rewards accrue.
2. Ensure the `RewardsPoolName` module account balance is zero (pool drained by `topUpBaseRewards` or never funded).
3. Call `MsgClaimTierRewards` for the position.
4. The call fails with `ErrInsufficientBonusPool` — the base staking rewards that were already earned from the validator's distribution record are inaccessible.
5. Call `MsgTierUndelegate` to attempt to exit — it also fails with `ErrInsufficientBonusPool` because it internally calls `claimRewards`.
6. The user's tokens remain locked in the tier module with no exit path until the pool is externally refunded.

The exact broken invariant: `claimBaseRewards` at line 30 successfully calls `WithdrawDelegationRewards`, but `sufficientBonusPoolBalance` at line 230 reverts the entire state, leaving the user with no base rewards and no exit. [11](#0-10) [12](#0-11)

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil

// TestMsgClaimTierRewards_FailsWhenBonusPoolInsufficient verifies that ClaimTierRewards
// returns ErrInsufficientBonusPool when accrued bonus cannot be paid, so the tx rolls
// back and the user can retry later without losing base rewards to a partial claim.
func (s *KeeperSuite) TestMsgClaimTierRewards_FailsWhenBonusPoolInsufficient() {
	lockAmount := sdkmath.NewInt(sdk.DefaultPowerReduction.Int64())
	pos := s.setupNewTierPosition(lockAmount, false)
	delAddr := sdk.MustAccAddressFromBech32(pos.Owner)
	valAddr := sdk.MustValAddressFromBech32(pos.Delegation.ValidatorAddress)
	_, bondDenom := s.getStakingData()
	msgServer := keeper.NewMsgServerImpl(s.keeper)

	s.setValidatorCommission(valAddr, sdkmath.LegacyZeroDec())

	// Advance time and allocate base rewards, but intentionally leave bonus pool empty.
	s.ctx = s.ctx.WithBlockHeight(s.ctx.BlockHeight() + 1)
	s.ctx = s.ctx.WithBlockTime(s.ctx.BlockTime().Add(time.Hour * 24 * 365))
	s.allocateRewardsToValidator(valAddr, sdkmath.NewInt(100), bondDenom)

	// Bonus pool remains at 0 — bonus accrued but pool cannot cover it.
	balBefore := s.app.BankKeeper.GetBalance(s.ctx, delAddr, bondDenom)

	// Use a branched context so a failed message does not persist state (matches DeliverTx rollback).
	cacheCtx, _ := s.ctx.CacheContext()
	resp, err := msgServer.ClaimTierRewards(cacheCtx, &types.MsgClaimTierRewards{
		Owner:       delAddr.String(),
		PositionIds: []uint64{pos.Id},
	})
	s.Require().Error(err)
	s.Require().True(errors.Is(err, types.ErrInsufficientBonusPool))
	s.Require().Nil(resp)

	balAfter := s.app.BankKeeper.GetBalance(s.ctx, delAddr, bondDenom)
	s.Require().True(balAfter.Amount.Equal(balBefore.Amount), "failed claim must not transfer rewards")
}
