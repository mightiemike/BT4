### Title
Unguarded Class-Level `UNSAFE_MAX_COST` Mutation via `set_run_unsafe_max_cost` Affects All `Program.run()` Callers — (File: `wheel/python/clvm_rs/program.py`)

---

### Summary

`Program.set_run_unsafe_max_cost` is a classmethod that writes to a shared class-level attribute with no caller restriction. Any Python code in the same process can silently overwrite the cost cap used by every other component's `Program.run()` call, either inflating it to an unsafe value (risking OOM / runaway execution) or deflating it to zero (causing all subsequent `run()` calls to fail with cost-exceeded).

---

### Finding Description

`UNSAFE_MAX_COST` is declared as a class-level attribute on `Program`: [1](#0-0) 

`set_run_unsafe_max_cost` writes to it unconditionally on the class object itself: [2](#0-1) 

`run()` reads the value back from the class at call time: [3](#0-2) 

Because Python class attributes are shared across every instance and every import of the same class object, a single call to `Program.set_run_unsafe_max_cost(X)` from *any* code in the process immediately changes the cost cap for *every* caller of `Program.run()`. There is no per-caller isolation, no capability token, and no way for a caller to protect its own invocations from being overwritten by another caller.

This is structurally identical to the reported `takeFee()` pattern: a state-mutating function with no access control that lets one party silently alter the execution parameters of all other parties.

---

### Impact Explanation

**Impact: Medium**

Two concrete outcomes:

1. **Cost-cap inflation** — an attacker (or buggy plugin) calls `Program.set_run_unsafe_max_cost(2**64)`. Every subsequent `Program.run()` call in the process now executes with an unbounded cost limit. A CLVM program that would normally be rejected for exceeding the network's cost budget runs to completion, potentially exhausting memory or CPU.

2. **Cost-cap deflation / denial** — an attacker calls `Program.set_run_unsafe_max_cost(0)`. Every subsequent `Program.run()` call raises `CostExceeded` immediately, breaking wallet signing, puzzle evaluation, and any other component that relies on `run()`.

The corrupted value is the `max_cost` argument forwarded to `run_serialized_chia_program`: [4](#0-3) 

which maps directly to the `max_cost` parameter of the Rust evaluator: [5](#0-4) 

---

### Likelihood Explanation

**Likelihood: Medium**

The Chia Python stack is a multi-component ecosystem. Wallets, full nodes, and third-party plugins all import `clvm_rs.program.Program` from the same package. Any component — including a third-party Chia plugin, a test harness left active in production, or a library that calls `set_run_unsafe_max_cost` for its own purposes — silently overwrites the shared class attribute. No privilege escalation or network access is required; only the ability to import and call a public classmethod.

---

### Recommendation

1. Remove `set_run_unsafe_max_cost` as a classmethod that mutates shared class state. Replace it with an instance-level or call-site parameter so each caller controls its own cost cap independently.
2. If a class-level default is still desired, make it read-only after initial configuration (e.g., raise if already set, or use a module-level constant).
3. Alternatively, require callers to pass `max_cost` explicitly to `run()` and remove the shared default entirely, matching the explicit `max_cost` parameter already present in `run_with_cost`. [6](#0-5) 

---

### Proof of Concept

```python
from clvm_rs.program import Program

# Component A (legitimate wallet code) sets its intended limit
Program.set_run_unsafe_max_cost(11_000_000_000)

# Component B (malicious plugin, loaded in the same process) overwrites it
Program.set_run_unsafe_max_cost(0)

# Component A's subsequent run() now uses max_cost=0 — raises CostExceeded
# for every program, regardless of what A intended.
puzzle = Program.fromhex("01")  # identity program
try:
    result = puzzle.run(Program.fromhex("80"))
except Exception as e:
    print(e)  # cost exceeded (0 < any positive cost)
```

No attacker-controlled CLVM bytes are required. The entry path is a public Python classmethod with no guard, callable by any code sharing the process.
