Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates on router address instead of end user, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the end user. A pool admin who allowlists the router to enable router-based swaps for their approved users inadvertently opens the pool to all users, completely bypassing the per-user swap allowlist.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that `sender` value directly to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly, making itself `msg.sender` to the pool: [4](#0-3) 

The same applies to `exactInput` (line 104), `exactOutputSingle` (line 136), and `exactOutput` (line 165). The extension therefore evaluates `allowedSwapper[pool][router]` — not `allowedSwapper[pool][end_user]`. Once the router is allowlisted, the check passes for any caller of the router.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly ignores `sender` and gates on `owner` (the economic actor): [5](#0-4) 

No equivalent economic-actor parameter exists in the `beforeSwap` interface — only `sender` (immediate caller) and `recipient` (output receiver) — so the swap allowlist has no way to recover the true initiator without an out-of-band mechanism such as `extensionData`.

## Impact Explanation

Any address can call `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) against a pool that has `SwapAllowlistExtension` configured, provided the router is allowlisted. The per-user allowlist is completely bypassed. Unauthorized users can execute swaps against LP liquidity at oracle prices in a pool that was intended to be restricted, constituting a direct loss of LP principal. This meets the Critical/High threshold: broken core pool access control causing loss of funds.

## Likelihood Explanation

The scenario is the natural operational pattern: a pool admin configures a curated pool with `SwapAllowlistExtension`, allowlists specific users, then allowlists the router so those users can benefit from slippage protection and multi-hop routing. The admin has no on-chain signal that allowlisting the router is equivalent to `setAllowAllSwappers(pool, true)`. The bypass requires no privileged access — any unprivileged address can call the public router entry points. It is repeatable every block.

## Recommendation

Gate on the economic actor, not the immediate caller. The preferred fix mirrors `DepositAllowlistExtension`: require the router to attest the real initiator via `extensionData` (e.g., `abi.encode(msg.sender)`) and verify it in `beforeSwap`. The extension would decode `extensionData` to extract the real user and check `allowedSwapper[pool][realUser]`. The router already has access to `msg.sender` at entry and already forwards `params.extensionData` to the pool unmodified, so the router can prepend or encode the initiator address before forwarding. Alternatively, document explicitly that allowlisting the router is equivalent to opening the pool to all users, so admins can make an informed decision — though this does not fix the invariant break.

## Proof of Concept

1. Pool admin deploys pool with `SwapAllowlistExtension` as `EXTENSION_1`, `beforeSwap` order set to extension 1.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — intending only Alice to swap.
3. Admin calls `setAllowedToSwap(pool, router, true)` — intending Alice to use the router.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(params.recipient, ...)` with `msg.sender = router`.
6. Pool calls `_beforeSwap(router, ...)` → extension receives `sender = router`.
7. Extension checks `allowedSwapper[pool][router]` → `true` → no revert.
8. Bob's swap executes against LP funds in a pool he was never authorized to access.

Foundry test skeleton:
```solidity
function test_routerBypassesSwapAllowlist() public {
    // Setup: pool with SwapAllowlistExtension, only alice and router allowlisted
    extension.setAllowedToSwap(pool, alice, true);
    extension.setAllowedToSwap(pool, address(router), true);

    // Bob (not allowlisted) swaps via router — should revert but does not
    vm.prank(bob);
    router.exactInputSingle(ExactInputSingleParams({pool: pool, ...}));
    // Bob's swap succeeds — allowlist bypassed
}
```
