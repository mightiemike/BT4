### Title
Stale `linkedSigners` entry persists after subaccount finalization via liquidation — (`EndpointStorage.sol` / `ClearinghouseLiq.sol`)

---

### Summary

When a subaccount is finalized through liquidation, `_finalizeSubaccount` in `ClearinghouseLiq.sol` zeros all balances and socializes the subaccount. However, the `linkedSigners[subaccount]` entry in `EndpointStorage` is never cleared. If the owner re-deposits to the finalized subaccount, the stale linked signer can immediately authorize transactions (trades, withdrawals) without the owner's explicit re-authorization — breaking the invariant that a finalized subaccount's authorization state should be reset.

---

### Finding Description

`EndpointStorage` maintains:

```solidity
mapping(bytes32 => address) internal linkedSigners;
``` [1](#0-0) 

This mapping is set via `LinkSigner` transactions (both slow-mode and fast-mode paths) and is consumed in `validateSignature` / `validateCompactSignature` to allow a third-party address to sign on behalf of a subaccount: [2](#0-1) 

When a subaccount is finalized through liquidation, `liquidateSubaccountImpl` in `ClearinghouseLiq.sol` calls `_finalizeSubaccount`, which zeros all balances, settles PnL, and socializes the subaccount. For isolated subaccounts it explicitly calls `tryCloseIsolatedSubaccount` to clean up `OffchainExchange` state: [3](#0-2) 

However, **no analogous cleanup is performed for `linkedSigners[txn.liquidatee]`**. After `_finalizeSubaccount` returns `true`:

1. All balances are zero.
2. `subaccountIds[subaccount]` remains non-zero — the subaccount is still "registered" and passes `requireSubaccount`.
3. `linkedSigners[subaccount]` still contains the old linked signer address. [4](#0-3) 

If the owner re-deposits to the finalized subaccount via `DepositCollateral`, `requireSubaccount` passes (since `subaccountIds` is never cleared), and the stale linked signer can immediately sign `WithdrawCollateral`, `TransferQuote`, or trading transactions for the re-activated subaccount — without the owner having explicitly re-authorized it.

The `LinkSigner` fast-mode path that sets the mapping: [5](#0-4) 

There is no corresponding delete anywhere in the liquidation/finalization flow.

---

### Impact Explanation

**Severity: Medium**

The stale linked signer can authorize trades and withdrawals on a re-activated subaccount without the owner's explicit re-authorization. The most dangerous scenario is when the linked signer key was compromised (e.g., the reason the subaccount was liquidated in the first place). After re-deposit, the compromised key can drain the new deposit immediately. This breaks the invariant that a finalized subaccount's authorization state is reset — exactly analogous to how `tryCloseIsolatedSubaccount` is called to clean up `OffchainExchange` state after finalization.

---

### Likelihood Explanation

**Likelihood: High**

Every subaccount that had a linked signer set and is subsequently finalized through liquidation retains the stale entry. The issue occurs unconditionally every time `_finalizeSubaccount` returns `true` for a subaccount with a non-zero `linkedSigners` entry. Re-deposit to a previously liquidated subaccount is a realistic user action.

---

### Recommendation

After `_finalizeSubaccount` returns `true` in `liquidateSubaccountImpl`, add a call to clear the linked signer for the finalized subaccount. Since `ClearinghouseLiq` operates via `delegatecall` from `Clearinghouse` and does not have direct access to `EndpointStorage`, a new function should be added to the `Endpoint` contract (callable only by the clearinghouse) to clear the entry:

```solidity
// In Endpoint / EndpointStorage:
function clearLinkedSigner(bytes32 subaccount) external {
    require(msg.sender == address(clearinghouse), ERR_UNAUTHORIZED);
    delete linkedSigners[subaccount];
}
```

Then in `liquidateSubaccountImpl`, after `_finalizeSubaccount` returns `true`:

```solidity
if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
    if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
        IOffchainExchange(...).tryCloseIsolatedSubaccount(txn.liquidatee);
    }
    IEndpoint(getEndpoint()).clearLinkedSigner(txn.liquidatee); // add this
    return;
}
```

---

### Proof of Concept

1. User A creates subaccount `S` and links signer `L` via a `LinkSigner` transaction → `linkedSigners[S] = L`.
2. Subaccount `S` becomes undercollateralized and is liquidated step by step.
3. Liquidator submits `LiquidateSubaccount` with `productId = type(uint32).max` to finalize `S`.
4. `_finalizeSubaccount` zeros all balances and returns `true`. `linkedSigners[S]` is **not** cleared.
5. User A re-deposits to subaccount `S` via `DepositCollateral`. `requireSubaccount(S)` passes because `subaccountIds[S] != 0`.
6. `L` (potentially a compromised key) signs a `WithdrawCollateral` transaction for `S`. `validateSignature` passes because `getLinkedSigner(S)` still returns `L`.
7. The new deposit is drained without User A having explicitly re-authorized `L` for the re-activated subaccount. [6](#0-5) [7](#0-6)

    function _finalizeSubaccount(
        IEndpoint.LiquidateSubaccount calldata txn,
        ISpotEngine spotEngine,
        IPerpEngine perpEngine
    ) internal returns (bool) {
        if (txn.productId != type(uint32).max) {
            return false;
        }
        // check whether the subaccount can be finalized:
        // - all perps positions have closed
        // - all spread positions have closed
        // - all spot assets have closed
        // - all positive pnls have been settled

        FinalizeVars memory v;

        v.spotIds = spotEngine.getProductIds();
        v.perpIds = perpEngine.getProductIds();

        require(v.spotIds[0] == QUOTE_PRODUCT_ID);

        // all spot assets (except USDC) must be closed out
        for (uint32 i = 1; i < v.spotIds.length; ++i) {
            uint32 spotId = v.spotIds[i];
            if (spotEngine.getRisk(spotId).longWeightInitialX18 == 0) {
                continue;
            }
            ISpotEngine.Balance memory balance = spotEngine.getBalance(
                spotId,
                txn.liquidatee
            );
            require(balance.amount <= 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
        }

        for (uint32 i = 0; i < v.perpIds.length; ++i) {
            uint32 perpId = v.perpIds[i];
            IPerpEngine.Balance memory balance = perpEngine.getBalance(
                perpId,
                txn.liquidatee
            );
            require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
        }

        // settle all positive pnl
        for (uint32 i = 0; i < v.perpIds.length; ++i) {
            uint32 perpId = v.perpIds[i];
            IPerpEngine.Balance memory balance = perpEngine.getBalance(
                perpId,
                txn.liquidatee
            );
            if (balance.vQuoteBalance > 0) {
                _settlePnlAgainstLiquidator(
                    txn,
                    perpId,
                    balance.vQuoteBalance,
                    spotEngine,
                    perpEngine
                );
            }
        }

        ISpotEngine.Balance memory quoteBalance = spotEngine.getBalance(
            QUOTE_PRODUCT_ID,
            txn.liquidatee
        );

        // settle all negative pnl until quote balance becomes 0
        for (uint32 i = 0; i < v.perpIds.length; ++i) {
            uint32 perpId = v.perpIds[i];
            IPerpEngine.Balance memory balance = perpEngine.getBalance(
                perpId,
                txn.liquidatee
            );
            if (balance.vQuoteBalance < 0 && quoteBalance.amount > 0) {
                int128 canSettle = MathHelper.max(
                    balance.vQuoteBalance,
                    -quoteBalance.amount
                );
                _settlePnlAgainstLiquidator(
                    txn,
                    perpId,
                    canSettle,
                    spotEngine,
                    perpEngine
                );
                quoteBalance.amount += canSettle;
            }
        }

        v.insurance = insurance;
        v.insurance -= lastLiquidationFees;
        v.canLiquidateMore = (quoteBalance.amount + v.insurance) > 0;

        if (v.canLiquidateMore) {
            for (uint32 i = 1; i < v.spotIds.length; ++i) {
                uint32 spotId = v.spotIds[i];
                ISpotEngine.Balance memory balance = spotEngine.getBalance(
                    spotId,
                    txn.liquidatee
                );
                if (spotEngine.getRisk(spotId).longWeightInitialX18 == 0) {
                    continue;
                }
                require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
            }
        }

        v.insurance = perpEngine.socializeSubaccount(
            txn.liquidatee,
            v.insurance
        );

        // we can assure that quoteBalance must be non positive, because if quoteBalance.amount > 0,
        // there must be 1) no negative pnl in perps, and 2) no liabilities in spot after above actions.
        // however, in this case the liquidatee must be healthy and cannot pass the health check at
        // the beginning.
        int128 insuranceCover = MathHelper.min(
            v.insurance,
            -quoteBalance.amount
        );
        if (insuranceCover > 0) {
            v.insurance -= insuranceCover;
            spotEngine.updateBalance(
                QUOTE_PRODUCT_ID,
                txn.liquidatee,
                insuranceCover
            );
        }
        if (v.insurance <= 0) {
            spotEngine.socializeSubaccount(txn.liquidatee);
        }
        v.insurance += lastLiquidationFees;
        insurance = v.insurance;
        return true;
    }
