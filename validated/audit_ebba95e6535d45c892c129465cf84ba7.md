### Title
Unauthenticated `ContractCodeResponse::V1` Accepted Without Signature at Protocol Version < 85 Allows Any Network Peer to Inject Arbitrary Contract Code into Chunk Validators — (File: `chain/client/src/stateless_validation/validate.rs`)

---

### Summary

`validate_contract_code_response` gates signature enforcement on `ProtocolFeature::SignedContractCodeResponse` (protocol version ≥ 85). At protocol version 84 — the minimum supported version — any network peer can send a `ContractCodeResponse::V1` (Borsh discriminant `0`) carrying arbitrary contract code for any relevant chunk, and the validator will accept and store it with zero authentication. This is the direct nearcore analog of the external bug: just as the mobile wallet parsed unencrypted responses when no `nonce` field was present, nearcore processes unsigned contract code responses when the `SignedContractCodeResponse` feature is not yet active.

---

### Finding Description

`ContractCodeResponse` has two Borsh-discriminated variants:

- `V1(ContractCodeResponseV1)` — tag `0`, no signature, no responder field
- `V2(ContractCodeResponseV2)` — tag `1`, signed inner payload, responder account ID [1](#0-0) 

The receive-side gate in `validate_contract_code_response` is:

```rust
if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
    validate_witness_contract_code_response_signature(epoch_manager, response)?;
}
``` [2](#0-1) 

When `protocol_version < 85` the entire `if` block is skipped. The function returns `ChunkRelevance::Relevant` unconditionally, and the caller immediately decompresses and stores the attacker-supplied contracts:

```rust
let contracts = response.decompress_contracts()?;
self.partial_witness_tracker.store_accessed_contract_codes(key, contracts)
``` [3](#0-2) 

`ContractCodeResponse::verify_signature` returns `false` for V1 by design, but that method is never called on the receive path at v84: [4](#0-3) 

`validate_witness_contract_code_response_signature` — the only function that would reject an unsigned response — is only reached when the feature is enabled: [5](#0-4) 

`SignedContractCodeResponse` activates at protocol version 85, and `MIN_SUPPORTED_PROTOCOL_VERSION` is 84, so the vulnerable window is live for any node that has not yet crossed the upgrade boundary: [6](#0-5) [7](#0-6) 

**Attack path:**

1. Attacker observes a relevant `ChunkProductionKey` from public chain data (epoch ID, shard ID, height are all in block headers).
2. Attacker constructs `ContractCodeResponse::V1` with the correct key but garbage `compressed_contracts`.
3. Attacker sends this over the p2p network to a chunk validator running at protocol version 84.
4. `validate_contract_code_response` passes the relevance check, skips the signature block, returns `Relevant`.
5. `store_accessed_contract_codes` stores the garbage code under the chunk's key.
6. When the validator attempts to validate the chunk state witness it uses the stored code; the code hash does not match the expected hash in the witness; validation fails; no endorsement is produced.
7. Repeating across multiple validators prevents the chunk from accumulating sufficient endorsements.

---

### Impact Explanation

An unprivileged network peer — no validator keys required — can prevent chunk validators from endorsing any relevant chunk by injecting garbage contract code. If enough validators are targeted simultaneously, chunks fail to accumulate the required endorsement threshold, causing chunks to be skipped or the chain to stall. This is a network-wide DoS with no cryptographic barrier to entry.

---

### Likelihood Explanation

Prerequisites are minimal: network reachability to chunk validators, and a `ChunkProductionKey` for a relevant chunk (publicly derivable from block headers). The vulnerable window is any node at protocol version 84, which is the minimum supported version for the current binary. The window closes only after the epoch in which 80 % of stake votes for v85 and the protocol version advances.

---

### Recommendation

**Short term:** In `validate_contract_code_response`, unconditionally reject `ContractCodeResponse::V1` responses — do not condition the rejection on the feature flag. Since `MIN_SUPPORTED_PROTOCOL_VERSION` is already 84 and the feature activates at 85, the only nodes that legitimately emit V1 are those still at v84; once the network crosses v85 the V1 path is dead code on the send side and should be dead on the receive side too.

**Long term:** Retire the V1 variant from the Borsh schema once the upgrade window has closed (analogous to how other deprecated variants are marked `_Deprecated*`). Audit all other versioned network messages that follow the same pattern — feature-gated signature enforcement with an unsigned legacy variant still parseable — to ensure no similar window exists elsewhere.

---

pub fn validate_contract_code_response(
    epoch_manager: &dyn EpochManagerAdapter,
    response: &ContractCodeResponse,
    store: &Store,
) -> Result<ChunkRelevance, Error> {
    let key = response.chunk_production_key();
    require_relevant!(validate_chunk_relevant(epoch_manager, key, store)?);
    let protocol_version = epoch_manager.get_epoch_protocol_version(&key.epoch_id)?;
    if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
        validate_witness_contract_code_response_signature(epoch_manager, response)?;
    }

    Ok(ChunkRelevance::Relevant)
}

fn validate_witness_contract_code_response_signature(
    epoch_manager: &dyn EpochManagerAdapter,
    response: &ContractCodeResponse,
) -> Result<(), Error> {
    let Some(responder) = response.responder() else {
        return Err(Error::Other(
            "Unsigned contract code response in epoch where signature is required".to_owned(),
        ));
    };
    let key = response.chunk_production_key();
    let chunk_producers =
        epoch_manager.get_epoch_chunk_producers_for_shard(&key.epoch_id, key.shard_id)?;
    if !chunk_producers.contains(responder) {
        return Err(Error::Other(format!(
            "Contract code response responder {responder} is not a chunk producer for shard {} in epoch {:?}",
            key.shard_id, key.epoch_id,
        )));
    }
    let validator = epoch_manager.get_validator_by_account_id(&key.epoch_id, responder)?;
    if !response.verify_signature(validator.public_key()) {
        return Err(Error::Other("Invalid witness contract code response signature".to_owned()));
    }
    Ok(())
