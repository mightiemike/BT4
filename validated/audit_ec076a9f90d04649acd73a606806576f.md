Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Allowing Complete Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` from the pool's perspective — the immediate caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, that caller is the router contract, not the end user. A pool admin who allowlists the router to enable normal UX for permitted users inadvertently opens the gate to every address on-chain, completely defeating per-user access control.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**

`MetricOmmPool.swap()` calls `_beforeSwap` with `msg.sender` as the `sender` argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` ABI-encodes that value and forwards it to every configured extension: [2](#0-1) 

**Step 2 — SwapAllowlistExtension checks that `sender` against its allowlist.**

`beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

**Step 3 — MetricOmmSimpleRouter is the immediate caller of `pool.swap()`.**

In `exactInputSingle`, the router calls the pool directly, making the router address the `msg.sender` the pool sees: [4](#0-3) 

**The mismatch:** The pool admin intends to allowlist specific end-users (e.g., KYC'd traders). To let those users use the router, the admin also calls `setAllowedToSwap(pool, router, true)`. At that point `allowedSwapper[pool][router] = true`, and the check in `beforeSwap` passes for **every** user who calls through the router — the actual end-user identity is never inspected. [5](#0-4) 

No existing guard resolves the true end-user identity. The `extensionData` field is passed through but the extension does not parse it for any identity attestation. The router stores the real `msg.sender` only in transient callback context for payment purposes, not in any field the extension can read.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` for private/gated access (institutional-only, KYC-gated, or partner-only liquidity) loses all access control the moment the router is allowlisted. Any unpermissioned address can call `MetricOmmSimpleRouter.exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` and execute swaps against the gated pool. LPs in such a pool suffer adverse selection from counterparties they explicitly excluded, resulting in direct loss of LP principal through unfavorable trades against unauthorized counterparties. This constitutes a broken core pool functionality causing loss of funds and a direct loss of LP assets above Sherlock thresholds.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the canonical user-facing entry point for swaps. A pool admin who deploys a gated pool and wants permitted users to have normal UX will naturally allowlist the router. The `SwapAllowlistExtension` NatSpec says it "Gates `swap` by swapper address" without clarifying that the swapper is the immediate pool caller, not the end user. [6](#0-5) 

The misconfiguration is a predictable, non-adversarial mistake by a legitimate pool admin. No special attacker capability is required — any unpermissioned address can exploit this by simply calling the public router.

## Recommendation

`SwapAllowlistExtension` must resolve the true end-user identity rather than the immediate pool caller. Two viable approaches:

1. **`extensionData` attestation**: Require the end-user to sign a permit passed through `extensionData`; the extension verifies the signature and checks the recovered address against the allowlist.
2. **Router-level enforcement**: The router encodes the real `msg.sender` into `extensionData`, and the extension trusts only calls from a known, trusted router that attests the real sender.

Additionally, `DepositAllowlistExtension` has a symmetric issue: it ignores the `sender` parameter (the actual token provider) and checks `owner` (the position beneficiary supplied by the caller), allowing any address to deposit tokens on behalf of an allowlisted owner and bypass the depositor gate: [7](#0-6) 

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls: setAllowedToSwap(pool, userA, true)    // permit userA
  pool admin calls: setAllowedToSwap(pool, router, true)   // permit router so userA can use it

Attack:
  userB (not in allowlist) calls:
    MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})

Execution trace:
  router.exactInputSingle()                          // msg.sender = userB
    → pool.swap(recipient, ...)                      // msg.sender = router
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓  // check passes
      → swap executes, tokens transferred to userB

Result: userB successfully swaps against the gated pool.
        Per-user allowlist is completely bypassed.
        LP funds exposed to unauthorized counterparty.
```

Foundry test plan: deploy a pool with `SwapAllowlistExtension`, allowlist only `userA` and the router, then `vm.prank(userB)` call `router.exactInputSingle(...)` and assert the swap succeeds despite `userB` not being in the allowlist.
