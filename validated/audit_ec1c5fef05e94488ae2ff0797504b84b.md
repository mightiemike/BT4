Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, breaking per-user allowlist enforcement — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` on the pool, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. This produces two fund-impacting failure modes: allowlisted users are blocked from the standard periphery path, and if the admin allowlists the router to work around this, every address on-chain bypasses the per-user curation gate.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks that value against the per-pool allowlist, where `msg.sender` is the pool and `sender` is the router: [2](#0-1) 

When a user calls `exactInputSingle`, the router calls `pool.swap()` directly, making itself `msg.sender` on the pool: [3](#0-2) 

For multi-hop `exactInput`, the same pattern holds for every hop: [4](#0-3) 

The router passes `extensionData` through verbatim without embedding the originating user's address, and the extension does not read `extensionData` at all. There is no existing guard that corrects the identity mismatch. The check `allowedSwapper[msg.sender][sender]` therefore evaluates to `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Failure mode A:** Pool admin allowlists `user1`. `user1` calls the router. Extension sees the router address, finds it not allowlisted, reverts `NotAllowedToSwap`. The standard periphery path is permanently broken for the pool's intended participants.

**Failure mode B:** Admin allowlists the router to unblock access. Now `allowedSwapper[pool][router] = true`, so every address on-chain can swap through the router regardless of individual allowlist status. The per-user curation gate is silently nullified.

## Impact Explanation
Failure mode A renders `MetricOmmSimpleRouter` completely unusable for any pool deploying `SwapAllowlistExtension` with individual user entries — broken core swap functionality. Failure mode B is a direct access-control bypass: any unprivileged user can trade on a pool configured to restrict access to a curated set of counterparties. For pools operating under compliance constraints or holding concentrated LP positions, this exposes LP funds to trades from disallowed counterparties, constituting loss of the intended access-control invariant and potential loss of LP principal.

## Likelihood Explanation
The router is the primary and documented periphery entry point. Any pool deploying `SwapAllowlistExtension` and expecting users to interact through the router immediately encounters failure mode A. Allowlisting the router is the natural admin response and directly triggers failure mode B. No special attacker capability is required — any wallet can exploit failure mode B once the router is allowlisted. The conditions are self-reinforcing and require no off-chain coordination.

## Recommendation
The extension must check the actual originating user, not the intermediary caller. The most robust fix is to have the router embed the originating user's address in `extensionData` before calling the pool, and have `SwapAllowlistExtension.beforeSwap` decode and check that value when `extensionData` is non-empty. The pool's existing guard that only the pool can invoke the extension ensures the override cannot be spoofed by an arbitrary external caller. Alternatively, the router can enforce the allowlist check itself before forwarding to the pool, removing the dependency on the extension seeing the correct actor.

## Proof of Concept
```solidity
// 1. Deploy pool with SwapAllowlistExtension as beforeSwap hook
// 2. Admin allowlists user1 (KYC'd user)
swapExtension.setAllowedToSwap(address(pool), user1, true);

// 3. user1 tries to swap through the router — REVERTS (router not allowlisted)
vm.prank(user1);
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool), zeroForOne: false, amountIn: 1000,
    amountOutMinimum: 0, recipient: user1,
    priceLimitX64: type(uint128).max, deadline: block.timestamp + 1,
    extensionData: ""
}));
// → reverts NotAllowedToSwap (router address checked, not user1)

// 4. Admin allowlists the router to "fix" the issue
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// 5. user2 (NOT allowlisted) swaps through the router — SUCCEEDS (bypass)
vm.prank(user2);
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool), zeroForOne: false, amountIn: 1000,
    amountOutMinimum: 0, recipient: user2,
    priceLimitX64: type(uint128).max, deadline: block.timestamp + 1,
    extensionData: ""
}));
// → succeeds: allowlist fully bypassed
```
