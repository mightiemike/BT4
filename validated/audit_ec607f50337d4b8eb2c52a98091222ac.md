### Title
Deprecated `verify_transaction_inclusion` Remains Callable, Enabling 64-Byte Transaction Merkle Proof Forgery - (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` (v1) is still a live, unprivileged public endpoint on the contract despite being superseded by `verify_transaction_inclusion_v2`. The v2 function was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability, but v1 remains directly callable by any NEAR account, making the security fix bypassable. Any recipient contract or caller that invokes v1 — intentionally or by using a stale integration — receives a proof result that may be forged.

### Finding Description
`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle proof forgery attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions). It does so by requiring a coinbase Merkle proof that must match the block's Merkle root before delegating to v1:

```rust
// contract/src/lib.rs lines 347-368
pub fn verify_transaction_inclusion_v2(&self, ...) -> bool {
    require!(
        merkle_tools::compute_root_from_merkle_proof(
            args.coinbase_tx_id.clone(), 0usize, &args.coinbase_merkle_proof,
        ) == header.block_header.merkle_root,
        "Incorrect coinbase merkle proof"
    );
    #[allow(deprecated)]
    self.verify_transaction_inclusion(args.into())   // delegates to v1 after the guard
}
```

However, v1 is still decorated only with `#[deprecated]` and `#[pause]` — neither of which restricts who may call it:

```rust
// contract/src/lib.rs lines 283-323
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
    // ... no coinbase proof check ...
    merkle_tools::compute_root_from_merkle_proof(
        args.tx_id,
        usize::try_from(args.tx_index).unwrap(),
        &args.merkle_proof,
    ) == header.block_header.merkle_root
}
```

`#[deprecated]` is a Rust compiler hint; it has no effect on the compiled WASM or on who may call the method at runtime. Any NEAR account can call `verify_transaction_inclusion` directly, completely skipping the coinbase guard that v2 enforces.

The 64-byte attack works as follows: Bitcoin's Merkle tree does not distinguish between leaf nodes (transactions) and internal nodes. A 64-byte blob can be crafted to be interpreted as an internal node whose hash equals a real internal node in the tree. Without the coinbase anchor, `compute_root_from_merkle_proof` will return the correct Merkle root for this forged input, causing v1 to return `true` for a transaction that does not exist in the block.

### Impact Explanation
Any recipient contract or off-chain consumer that calls `verify_transaction_inclusion` (v1) — whether because it was integrated before v2 existed, or because it simply calls the wrong method — will accept a forged proof as valid. The corrupted value is the **proof result**: the function returns `true` for a non-existent transaction. Downstream contracts that gate asset releases, bridge operations, or state transitions on this result will be deceived into executing based on a fabricated Bitcoin transaction.

### Likelihood Explanation
The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` with attacker-controlled `tx_id`, `tx_index`, and `merkle_proof`. The 64-byte forgery technique is publicly documented and well-understood. Integrators who deployed against v1 before v2 was released, or who copy example code referencing the old method name, will silently remain on the vulnerable path with no on-chain enforcement to migrate them.

### Recommendation
Remove `verify_transaction_inclusion` from the public ABI entirely, or gate it with an access-control role that prevents unprivileged callers from reaching it. A `#[deprecated]` attribute alone is insufficient — it produces a compiler warning for Rust callers but has no effect on NEAR RPC callers or cross-contract calls. The safest fix is to make the function `pub(crate)` or to add a hard `env::panic_str` at the top of the function body that unconditionally rejects external calls, forcing all callers to migrate to v2.

### Proof of Concept
1. Deploy the contract (Bitcoin feature flag) with a known set of block headers.
2. Identify a block `B` whose Merkle tree has at least two transactions.
3. Craft a 64-byte blob `F` whose double-SHA256 equals an internal node `N` in `B`'s Merkle tree.
4. Construct a Merkle proof `P` such that `compute_root_from_merkle_proof(F, idx, P) == B.merkle_root`.
5. Call `verify_transaction_inclusion` with `tx_id = F`, `tx_block_blockhash = B`, `tx_index = idx`, `merkle_proof = P`, `confirmations = 1`.
6. The call returns `true` despite `F` not being a real transaction in block `B`.
7. Calling `verify_transaction_inclusion_v2` with the same arguments (and no valid coinbase proof) would panic, demonstrating that v2's guard is bypassed by calling v1 directly. [1](#0-0) [2](#0-1) [3](#0-2)

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
