### Title
`ValidResourceBounds` Protobuf Deserialization Silently Collapses `AllResources` to `L1Gas`, Producing a Different Transaction Hash - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` classifies any transaction whose `l1_data_gas` and `l2_gas` are both zero as `ValidResourceBounds::L1Gas`, even when the originating sequencer computed and stored the hash under `ValidResourceBounds::AllResources`. Because `get_tip_resource_bounds_hash` produces structurally different Poseidon preimages for the two variants, the same on-chain transaction receives two distinct hash values depending on which conversion path is taken. A syncing node that deserializes the transaction via protobuf will recompute a hash that does not match the hash committed in the block, causing hash validation to fail and the block to be rejected.

### Finding Description

**Step 1 – Originating sequencer path (AllResources hash)**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds`. The gateway converts it to `InternalRpcInvokeTransactionV3` (same `AllResourceBounds` field) and calls `calculate_transaction_hash`, which delegates to `get_invoke_transaction_v3_hash`, which calls `get_tip_resource_bounds_hash`. [1](#0-0) 

For `ValidResourceBounds::AllResources`, the function builds a three-element resource vector `[L1_GAS_concat, L2_GAS_concat, L1_DATA_GAS_concat]` and hashes it with Poseidon. Even when `l2_gas = 0` and `l1_data_gas = 0`, the `L1_DATA_GAS_concat` felt is **non-zero** (the resource-name bytes `"L1_DATA"` are embedded in the upper bits), so the resulting hash is distinct from the two-element case. [2](#0-1) 

When the transaction is promoted to the blockifier type `InvokeTransactionV3`, the conversion explicitly wraps the bounds as `AllResources`: [3](#0-2) 

So the stored hash `H_all` is computed over three resource entries.

**Step 2 – Protobuf serialization preserves the zero values**

`From<ValidResourceBounds> for protobuf::ResourceBounds` always emits `l1_data_gas: Some(...)`, so the wire representation carries an explicit zero for both `l2_gas` and `l1_data_gas`. [4](#0-3) 

**Step 3 – Syncing node deserializes as `L1Gas` (lossy conversion)**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies the following classification: [5](#0-4) 

When `l1_data_gas = 0` and `l2_gas = 0`, the result is `ValidResourceBounds::L1Gas(l1_gas)`, discarding the `AllResources` variant. The TODO comment on line 426 acknowledges this is a temporary compatibility measure for 0.13.2, but the classification boundary is value-based, not version-based.

**Step 4 – Hash recomputed as `L1Gas` → different value**

On the syncing node, `get_tip_resource_bounds_hash` now receives `L1Gas` and builds only a two-element vector `[L1_GAS_concat, L2_GAS_concat]`: [6](#0-5) 

The resulting hash `H_l1 ≠ H_all`. `validate_transaction_hash` checks `possible_hashes.contains(&expected_hash)` and returns `false`, causing the block to be rejected. [7](#0-6) 

**The two `ValidResourceBounds` variants for the same data:** [8](#0-7) 

### Impact Explanation

A user submits an `InvokeV3` transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`. The sequencer accepts it, computes `H_all`, and includes it in a block. Any node that syncs this block via P2P deserializes the transaction as `L1Gas`, recomputes `H_l1 ≠ H_all`, and rejects the block. This causes a chain split between the sequencer and all syncing nodes for any block containing such a transaction.

This matches: **High – Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

### Likelihood Explanation

The trigger is an unprivileged user submitting a valid `InvokeV3` transaction with non-zero `l1_gas` but zero `l2_gas` and `l1_data_gas`. The gateway's stateless validator does not reject this (it only requires at least one non-zero bound). The condition is reachable by any user without special permissions.

### Recommendation

The protobuf deserializer must not use value-based heuristics to select the `ValidResourceBounds` variant. Options:

1. **Add an explicit version/type tag to the protobuf message** so the deserializer knows whether the original transaction was pre-0.13.3 (`L1Gas`) or post-0.13.3 (`AllResources`), independent of whether the gas values happen to be zero.
2. **Remove the `L1Gas` classification branch from the protobuf converter** once 0.13.2 support is dropped (as the TODO comment already notes), and always deserialize as `AllResources`.
3. **Enforce the invariant** that `AllResources` with zero `l2_gas` and `l1_data_gas` is never produced by the gateway for new transactions, and that the protobuf converter only produces `L1Gas` for transactions whose block number predates 0.13.3.

### Proof of Concept

1. Submit an `InvokeV3` transaction with `resource_bounds = { l1_gas: { max_amount: 1, max_price_per_unit: 1 }, l2_gas: { 0, 0 }, l1_data_gas: { 0, 0 } }` to the gateway.
2. The gateway calls `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` with `AllResources` → hash includes `L1_DATA_GAS_concat` → produces `H_all`.
3. The transaction is included in a block with hash `H_all`.
4. A syncing node receives the block via P2P. The protobuf deserializer calls `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`: `l1_data_gas = 0`, `l2_gas = 0` → returns `L1Gas(l1_gas)`.
5. The syncing node calls `get_tip_resource_bounds_hash` with `L1Gas` → hash omits `L1_DATA_GAS_concat` → produces `H_l1 ≠ H_all`.
6. `validate_transaction_hash` returns `false`; the block is rejected.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
