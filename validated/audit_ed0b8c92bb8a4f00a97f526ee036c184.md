### Title
`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` Conflates `AllResources` (zero L2/L1_DATA_GAS) with `L1Gas`, Producing a Different Transaction Hash Preimage - (File: crates/apollo_protobuf/src/converters/transaction.rs)

### Summary

The protobuf deserialization of `ValidResourceBounds` uses a value-based heuristic (checking whether `l2_gas` and `l1_data_gas` are zero) to decide between the `L1Gas` and `AllResources` variants. A V3 transaction submitted with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is hashed with a 4-element Poseidon preimage at ingestion time, but after a protobuf sync round-trip it is reconstructed as `ValidResourceBounds::L1Gas(X)` and would be hashed with a 3-element preimage — a different hash — breaking the canonicalization invariant that a transaction's hash must be stable across serialization boundaries.

### Finding Description

**Hash preimage divergence by variant:**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` produces different-length Poseidon inputs depending on the `ValidResourceBounds` variant:

```
L1Gas(X)          → poseidon(tip, L1_GAS_packed, L2_GAS_packed)           // 3 elements
AllResources(X,0,0) → poseidon(tip, L1_GAS_packed, L2_GAS_packed, L1_DATA_GAS_packed) // 4 elements
``` [1](#0-0) 

**The broken deserialization gate:**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `transaction.rs` (used for sync/storage transactions) decides the variant purely by inspecting whether the wire values are zero:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // ← wrong for new AllResources txs with zero bounds
} else {
    ValidResourceBounds::AllResources(...)
})
``` [2](#0-1) 

There is no version flag or transaction-type tag in the protobuf `ResourceBounds` message to distinguish a pre-0.13.3 `L1Gas` transaction from a post-0.13.3 `AllResources` transaction whose L2 and L1_DATA_GAS bounds happen to be zero. [3](#0-2) 

**Attacker-controlled path:**

`RpcInvokeTransactionV3` and `InternalRpcInvokeTransactionV3` both carry `AllResourceBounds` (not `ValidResourceBounds`). A user can submit a V3 transaction with `AllResourceBounds { l1_gas: X, l2_gas: {0,0}, l1_data_gas: {0,0} }`. The gateway accepts it (stateless validator does not reject zero L2/L1_DATA_GAS bounds). [4](#0-3) 

At ingestion, `calculate_transaction_hash` is called on `InternalRpcInvokeTransactionV3`, which calls `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` with `ValidResourceBounds::AllResources(...)` → **4-element hash H₄**. [5](#0-4) 

When this transaction is later stored as `InvokeTransactionV3` (via `From<RpcInvokeTransactionV3> for InvokeTransactionV3`, which wraps in `ValidResourceBounds::AllResources`) and then serialized to protobuf for block sync, the `From<ValidResourceBounds> for protobuf::ResourceBounds` serializer emits `l2_gas = 0` and `l1_data_gas = 0` explicitly. [6](#0-5) 

The receiving sync node deserializes via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, sees both fields are zero, and reconstructs `ValidResourceBounds::L1Gas(X)`. Any subsequent hash recomputation (e.g., in `validate_transaction_hash`) produces **3-element hash H₃ ≠ H₄**, causing the block to be rejected as invalid. [7](#0-6) 

### Impact Explanation

The canonicalization invariant — that a transaction's hash is stable across the public-API → internal → protobuf → sync boundary — is broken for any V3 transaction with zero L2 and L1_DATA_GAS resource bounds. A syncing node that recomputes or validates transaction hashes from deserialized protobuf data will compute a different hash than the one committed by the sequencer, causing it to reject valid blocks. This maps to: **High — Transaction conversion or signature/hash logic binds the wrong hash type or executable payload.**

### Likelihood Explanation

Any unprivileged user can craft and submit a valid V3 invoke transaction with `AllResourceBounds { l1_gas: X, l2_gas: {0,0}, l1_data_gas: {0,0} }`. The gateway's stateless validator does not reject zero L2/L1_DATA_GAS bounds. Once such a transaction is included in a block and that block is synced, the deserialization mismatch is deterministically triggered.

### Recommendation

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion must not use field values alone to select the variant. Options:

1. Add an explicit version/type tag to the protobuf `ResourceBounds` message (or to the enclosing transaction message) that encodes whether the transaction was signed under the 2-resource or 3-resource hash scheme.
2. Alternatively, always deserialize to `AllResources` when all three fields are present in the wire message (even if zero), and only fall back to `L1Gas` when `l1_data_gas` is absent (`None`) — matching the original 0.13.2 wire format where the field did not exist.

The second option aligns with the existing TODO comment:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
``` [8](#0-7) 

The fix: treat `l1_data_gas = Some(zero)` as `AllResources` and only treat `l1_data_gas = None` as the legacy `L1Gas` path.

### Proof of Concept

1. Submit a V3 invoke transaction with `AllResourceBounds { l1_gas: {amount:1, price:1}, l2_gas: {0,0}, l1_data_gas: {0,0} }` to the gateway.
2. The gateway computes hash H₄ = poseidon(INVOKE, version=3, sender, poseidon(tip, L1_GAS_packed, L2_GAS_packed(0), L1_DATA_GAS_packed(0)), ...) and stores it.
3. The transaction is included in a block. The block hash commits to H₄.
4. A syncing node receives the block via protobuf sync. It deserializes the transaction's `ResourceBounds` protobuf (which has `l1_data_gas = Some({0,0})`). Since `l1_data_gas.is_zero() && l2_gas.is_zero()`, it produces `ValidResourceBounds::L1Gas({amount:1, price:1})`.
5. The syncing node recomputes the hash: H₃ = poseidon(INVOKE, version=3, sender, poseidon(tip, L1_GAS_packed, L2_GAS_packed(0)), ...).
6. H₃ ≠ H₄. Hash validation fails. The syncing node rejects the block.
