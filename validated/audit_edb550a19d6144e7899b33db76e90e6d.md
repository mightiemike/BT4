### Title
Cost Guard Applied Before Final Multiplier Enables u64 Overflow and Cost Undercharge in Unknown Opcode Handler — (`File: src/more_ops.rs`)

### Summary

`op_unknown` in `src/more_ops.rs` applies `check_cost` to the **base** cost before multiplying by `cost_multiplier + 1`. Because `cost` and `cost_multiplier` are both `u64`, their product can silently overflow (wrap) in Rust release builds, producing a final cost far smaller than the true value — including zero. The wrapped cost then passes the `u32::MAX` guard and is returned as the canonical cost of the operation, undercharging the program and corrupting the consensus cost model.

### Finding Description

`op_unknown` computes cost in two stages:

**Stage 1** — compute a base cost from the opcode's `cost_function` field and its arguments (lines 209–256).

**Stage 2** — multiply by `(cost_multiplier + 1)` and return (lines 260–265):

```rust
check_cost(cost, max_cost)?;          // line 260 — guards the PRE-multiplied value
cost *= cost_multiplier + 1;          // line 261 — multiplication; can overflow u64
if cost > u32::MAX as u64 {           // line 262 — guards against u32::MAX, not max_cost
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))   // line 265
}
``` [1](#0-0) 

`cost_multiplier` is decoded from the opcode atom bytes as a `u32` (up to `0xFFFF_FFFF`) and widened to `u64`: [2](#0-1) 

The guard at line 260 only validates the base cost. After the multiplication at line 261, the product `cost * (cost_multiplier + 1)` can exceed `u64::MAX`. In Rust release builds, integer overflow wraps (two's complement). The wrapped value can be ≤ `u32::MAX`, passing the guard at line 262, and the function returns `Reduction(wrapped_cost, nil)` — a cost far below the true value.

**Concrete overflow path:**

| Variable | Value |
|---|---|
| `cost_function` | `1` (add-like, bits 7:6 of last opcode byte = `01`) |
| Base cost target | `2^32 = 4,294,967,296` (achievable with ~13.3 M one-byte args at `ARITH_COST_PER_ARG=320`, `ARITH_COST_PER_BYTE=3`; within Chia's `max_cost ≈ 11 × 10^9`) |
| `cost_multiplier` | `2^32 − 1 = 0xFFFF_FFFF` (encoded in the 4 leading opcode bytes) |
| `cost_multiplier + 1` | `2^32` |
| Product | `2^32 × 2^32 = 2^64 ≡ 0 (mod 2^64)` |
| Returned cost | **0** |

`check_cost(2^32, 11_000_000_000)` passes (2^32 ≈ 4.3 × 10^9 < 11 × 10^9). The multiplication wraps to 0. The `u32::MAX` guard passes. `op_unknown` returns `Reduction(0, nil)`.

The attacker-controlled entry path is direct: `ChiaDialect::op` dispatches any unrecognised opcode to `unknown_operator` → `op_unknown` when `NO_UNKNOWN_OPS` is not set (consensus/non-mempool mode): [3](#0-2) 

`RuntimeDialect` has the same dispatch: [4](#0-3) 

`MEMPOOL_MODE` sets `NO_UNKNOWN_OPS`, so the vulnerable path is the **consensus (block validation) mode**, not mempool mode: [5](#0-4) 

### Impact Explanation

The CLVM cost model is the sole mechanism preventing resource exhaustion in block validation. A program whose true cost is `2^64` (or any value that wraps to ≤ `u32::MAX`) is accepted with a reported cost of 0 (or a small value). This allows:

1. **Cost undercharge / fee bypass**: A block containing such a program appears cheap, allowing it to be included without paying the appropriate fee.
2. **Consensus divergence**: Rust debug builds panic on overflow; release builds wrap. A program accepted by a release-mode full node would be rejected (panic/error) by a debug-mode node, splitting consensus.
3. **Repeated inclusion**: Because the cost is reported as 0, many such programs can be packed into a single block, each bypassing the cost limit.

### Likelihood Explanation

The opcode bytes are fully attacker-controlled CLVM input. The `cost_function` and `cost_multiplier` fields are decoded directly from the opcode atom with no sanitisation beyond the `u32_from_u8` range check. Constructing an opcode that triggers the overflow requires choosing specific byte values — a straightforward offline calculation. The vulnerable path (consensus mode without `NO_UNKNOWN_OPS`) is the default for block validation.

### Recommendation

Apply `check_cost` **after** the multiplication, not before. Use `u64::checked_mul` to detect overflow explicitly and return an error:

```rust
// Replace lines 260-265 with:
let Some(final_cost) = cost.checked_mul(cost_multiplier + 1) else {
    return Err(EvalErr::Invalid(o))?;
};
check_cost(final_cost, max_cost)?;
if final_cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(final_cost as Cost, allocator.nil()))
}
```

This mirrors the correct pattern used in the Solidity fix for the referenced report: perform all multiplications first, then apply the guard/division.

### Proof of Concept

Craft a 5-byte unknown opcode atom:
- Bytes 0–3 (multiplier field): `[0xFF, 0xFF, 0xFF, 0xFF]` → `cost_multiplier = 0xFFFF_FFFF = 2^32 − 1`
- Byte 4 (last byte): `0x40` → `cost_function = (0x40 >> 6) & 0x3 = 1` (add-like)

Pass approximately `(2^32 − 99) / 323 ≈ 13,297,115` one-byte atom arguments.

Execution trace:
1. `cost_function = 1`: base cost accumulates to `2^32 = 4,294,967,296`.
2. `check_cost(4_294_967_296, 11_000_000_000)` → **passes** (4.3 × 10^9 < 11 × 10^9).
3. `cost *= 0xFFFF_FFFF + 1` → `4_294_967_296 * 4_294_967_296 = 2^64 ≡ 0 (mod 2^64)` in release mode.
4. `0 > u32::MAX` → **false**, guard passes.
5. Returns `Reduction(0, allocator.nil())` — **zero cost** for an operation whose true cost is `2^64`. [6](#0-5)
