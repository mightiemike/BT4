### Title
`addOrUpdateProduct` does not call `updateStates` before updating `configs`, causing retroactive application of new interest rate parameters — (File: `core/contracts/SpotEngine.sol`)

---

### Summary

`SpotEngine.addOrUpdateProduct` overwrites `configs[productId]` — which contains all interest rate curve parameters — without first accruing interest using the old parameters. The next sequencer-driven `updateStates` call will apply the new parameters retroactively over the entire elapsed period since the last accrual, corrupting `cumulativeDepositsMultiplierX18`, `cumulativeBorrowsMultiplierX18`, and the fees balance for every depositor and borrower in that product.

---

### Finding Description

`SpotEngineState._updateState` computes the borrow rate for a product using `configs[productId]` fields: `interestFloorX18`, `interestSmallCapX18`, `interestLargeCapX18`, `interestInflectionUtilX18`, and `minDepositRateX18`. [1](#0-0) 

The result is multiplied by the time delta `dt` supplied by the sequencer to produce `borrowRateMultiplierX18`, which is then compounded into the cumulative multipliers and used to compute the fee amount credited to `FEES_ACCOUNT`. [2](#0-1) 

Interest is accrued only when the sequencer calls `updateStates(uint128 dt)`, which is `onlyEndpoint`. [3](#0-2) 

`SpotEngine.addOrUpdateProduct` is the sole path to change `configs[productId]`. It writes the new config directly, with no preceding interest accrual:

```solidity
// SpotEngine.sol line 83
configs[productId] = config;
``` [4](#0-3) 

Because `updateStates` receives `dt` from the sequencer (not from an on-chain timestamp), the contract has no record of when the last accrual occurred. After `addOrUpdateProduct` executes, the next `updateStates(dt)` call will compute interest for the full `dt` window — which spans time both before and after the config change — entirely using the new parameters. The old parameters are never applied to the pre-change portion of that window.

---

### Impact Explanation

**Corrupted state variables:**

- `state.cumulativeBorrowsMultiplierX18` — used to denormalize every borrower's balance
- `state.cumulativeDepositsMultiplierX18` — used to denormalize every depositor's balance
- `balances[productId][FEES_ACCOUNT]` — protocol fee accumulation

If the new config raises rates (larger `interestSmallCapX18` / `interestLargeCapX18`), borrowers are overcharged and depositors receive excess yield for the pre-change period. If rates are lowered, the opposite occurs. In both cases the fee balance is misstated. Because all user balances are stored normalized against the cumulative multipliers, every subsequent balance read, health check, liquidation, and settlement for that product operates on incorrect values.

---

### Likelihood Explanation

Any routine protocol maintenance that adjusts interest rate parameters for an existing product triggers the bug. No adversarial action is required; the owner legitimately calling `addOrUpdateProduct` is sufficient. The window of exposure equals the time elapsed since the last sequencer `updateStates` call, which can be hours in practice.

---

### Recommendation

Before overwriting `configs[productId]`, accrue interest for the current period using the old parameters. Since `_updateState` is internal and `dt` is sequencer-supplied, the recommended fix is to require the sequencer to submit an `updateStates` transaction in the same sequencer batch immediately before any `addOrUpdateProduct` call, or to refactor `addOrUpdateProduct` to accept and apply a `dt` argument that triggers `_updateState` internally prior to the config write.

---

### Proof of Concept

1. At time T₀, sequencer calls `updateStates(dt₀)`. Interest is accrued using old config `C_old` (e.g., `interestSmallCapX18 = 4e16`).
2. At time T₁ (T₀ + 3600 seconds), owner calls `addOrUpdateProduct(productId, ..., C_new)` where `C_new.interestSmallCapX18 = 2e17` (5× higher).
3. At time T₂ (T₁ + 3600 seconds), sequencer calls `updateStates(dt = 7200)` (covering T₀→T₂).
4. `_updateState` computes `borrowerRateX18` using `C_new` for the full 7200-second window.
5. The 3600 seconds between T₀ and T₁ — which should have used `C_old` — are instead charged at the higher `C_new` rate, inflating `cumulativeBorrowsMultiplierX18` and the fee balance beyond what was contractually owed. [5](#0-4) [6](#0-5)

    function _updateState(
        uint32 productId,
        State memory state,
        uint128 dt
    ) internal {
        int128 borrowRateMultiplierX18;
        int128 totalDeposits = state.totalDepositsNormalized.mul(
            state.cumulativeDepositsMultiplierX18
        );
        int128 totalBorrows = state.totalBorrowsNormalized.mul(
            state.cumulativeBorrowsMultiplierX18
        );
        int128 utilizationRatioX18 = totalBorrows.div(totalDeposits);
        int128 minDepositRateX18;
        {
            Config memory config = configs[productId];

            // annualized borrower rate
            int128 borrowerRateX18 = config.interestFloorX18;
            if (utilizationRatioX18 == 0) {
                // setting borrowerRateX18 to 0 here has the property that
                // adding a product at the beginning of time and not using it until time T
                // results in the same state as adding the product at time T
                borrowerRateX18 = 0;
            } else if (utilizationRatioX18 < config.interestInflectionUtilX18) {
                borrowerRateX18 += config
                    .interestSmallCapX18
                    .mul(utilizationRatioX18)
                    .div(config.interestInflectionUtilX18);
            } else {
                borrowerRateX18 +=
                    config.interestSmallCapX18 +
                    config.interestLargeCapX18.mul(
                        (
                            (utilizationRatioX18 -
                                config.interestInflectionUtilX18).div(
                                    ONE - config.interestInflectionUtilX18
                                )
                        )
                    );
            }

            // convert to per second
            borrowerRateX18 = borrowerRateX18.div(
                MathSD21x18.fromInt(31536000)
            );
            borrowRateMultiplierX18 = (ONE + borrowerRateX18).pow(int128(dt));
            minDepositRateX18 = config.minDepositRateX18;
        }

        // if we don't take fees into account, the liquidity, which is
        // (deposits - borrows) should remain the same after updating state.

        // For simplicity, we use `tb`, `cbm`, `td`, and `cdm` for
        // `totalBorrowsNormalized`, `cumulativeBorrowsMultiplier`,
        // `totalDepositsNormalized`, and `cumulativeDepositsMultiplier`

        // before the updating, the liquidity is (td * cdm - tb * cbm)
        // after the updating, the liquidity is
        // (td * cdm * depositRateMultiplier - tb * cbm * borrowRateMultiplier)
        // so we can get
        // depositRateMultiplier = utilization * (borrowRateMultiplier - 1) + 1

        int128 totalDepositRateX18 = utilizationRatioX18.mul(
            borrowRateMultiplierX18 - ONE
        );

        // deduct protocol fees
        int128 realizedDepositRateX18 = totalDepositRateX18.mul(
            ONE - INTEREST_FEE_FRACTION
        );

        // pass fees balance change
        int128 feesAmt = totalDeposits.mul(
            totalDepositRateX18 - realizedDepositRateX18
        );

        state.cumulativeBorrowsMultiplierX18 = state
            .cumulativeBorrowsMultiplierX18
            .mul(borrowRateMultiplierX18);
```

    function updateStates(uint128 dt) external onlyEndpoint {
        State memory quoteState;
        require(dt < 7 * SECONDS_PER_DAY, ERR_INVALID_TIME);
        for (uint32 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            if (productId == NLP_PRODUCT_ID) {
                continue;
            }
            State memory state = states[productId];
            if (productId == QUOTE_PRODUCT_ID) {
                quoteState = state;
            }
            if (state.totalDepositsNormalized == 0) {
                continue;
            }
            _updateState(productId, state, dt);
            _setState(productId, state);
        }
    }

    function addOrUpdateProduct(
        uint32 productId,
        uint32 quoteId,
        int128 sizeIncrement,
        int128 minSize,

    ) public onlyOwner {
        bool isNewProduct = _addOrUpdateProduct(
            productId,
            quoteId,
            sizeIncrement,
            minSize,
            riskStore
        );
        configs[productId] = config;

        if (isNewProduct) {
            require(productId != QUOTE_PRODUCT_ID);
            _setState(
                productId,
                State({
                    cumulativeDepositsMultiplierX18: ONE,
                    cumulativeBorrowsMultiplierX18: ONE,
                    totalDepositsNormalized: 0,
                    totalBorrowsNormalized: 0
                })

        }
    }
