### Title
Broken Merkle Proof Verification in `verify_transaction_inclusion` Allows Proof Forgery — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) remains publicly callable by any unprivileged NEAR account despite being deprecated. It uses a broken Merkle proof algorithm in `compute_root_from_merkle_proof` that never validates whether the supplied `tx_id` is a leaf-level transaction hash or an internal Merkle tree node. An attacker can supply an internal node hash as `tx_id` with a shortened proof path, causing the function to return `true` for a transaction that does not exist. This is a direct analog to the `elliptic` package's "Use of a Broken or Risky Cryptographic Algorithm" / proof-malleability class: the cryptographic primitive (Merkle inclusion proof) is structurally broken and produces forged verification results.

---

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` in Rust, which is a **compiler-only warning** — it imposes no runtime restriction. The function remains a live, publicly callable NEAR contract method protected only by `#[pause]`, with no access control. [1](#0-0) 

The function delegates proof verification to `compute_root_from_merkle_proof`: [2](#0-1) 

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` iterates over the proof hashes and combines them with the running hash, but **never checks whether the initial `transaction_hash` is a leaf node**: [3](#0-2) 

Bitcoin's Merkle tree is built by double-SHA256-hashing pairs of 32-byte values. Every internal node at depth D is itself a valid 32-byte hash that satisfies the same structural equation as a leaf. An attacker who knows the Merkle tree of any confirmed block can:

1. Pick any internal node `N` at depth D (e.g., `H(tx0, tx1)`).
2. Construct a proof of length `(tree_depth − D)` that walks from `N` up to the root — this proof is mathematically correct.
3. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_index` matching `N`'s position at depth D, and the shortened `merkle_proof`.
4. `compute_root_from_merkle_proof` recomputes the root correctly and the equality check at line 322 passes, returning `true`.

The only guard in v1 against an empty proof is: [4](#0-3) 

This does not prevent the attack — the attacker supplies a non-empty (but shortened) proof.

The v2 function (`verify_transaction_inclusion_v2`) was introduced to mitigate this via a coinbase proof check, and it calls v1 internally via `args.into()`: [5](#0-4) 

However, v1 is still directly reachable by any caller, bypassing the coinbase guard entirely.

---

### Impact Explanation

Any consumer contract that calls `verify_transaction_inclusion` to confirm a Bitcoin payment receives a cryptographically valid-looking `true` result for a transaction that was never broadcast or mined. This enables:

- **Cross-chain fraud**: An attacker claims a BTC payment occurred (e.g., for an atomic swap or bridged asset release) by forging a Merkle inclusion proof against a real, confirmed block. No actual BTC is spent.
- **Corrupted proof result**: The canonical mapping `tx_id → included in block` is falsified. The `headers_pool` and `mainchain_header_to_height` state are untouched, but the verification output — the only security guarantee this contract provides to consumers — is broken.

**Impact: 4**

---

### Likelihood Explanation

- The function is callable by any NEAR account with no staking, role, or deposit requirement beyond gas.
- The Merkle tree structure of every Bitcoin block is publicly derivable from block data available via any full node or block explorer.
- The 64-byte Merkle forgery attack is well-documented (referenced in the contract's own comments at line 268) and has known tooling.
- The attacker does not need to control any privileged role, private key, or relayer.

**Likelihood: 3**

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the contract entirely, or add an explicit `env::panic_str` at its entry point to make it unconditionally revert. Deprecation attributes do not prevent on-chain invocation. All callers must migrate to `verify_transaction_inclusion_v2`, which enforces the coinbase proof guard.

Optionally, add a minimum proof-length check in `compute_root_from_merkle_proof` equal to the full tree depth derived from the block's transaction count, though this requires passing additional context.

---

### Proof of Concept

**Setup**: A confirmed Bitcoin block with 4 transactions: `[tx0, tx1, tx2, tx3]`. The Merkle tree is:

```
Root = H(H(tx0,tx1), H(tx2,tx3))
```

Internal node `I = H(tx0, tx1)` is at depth 1, position 0.

**Attack call** (Borsh-encoded `ProofArgs`):
```
tx_id             = I  (= double_sha256(tx0 || tx1))
tx_block_blockhash = <any confirmed block hash in mainchain>
tx_index          = 0
merkle_proof      = [ H(tx2, tx3) ]   // length 1, not 2
confirmations     = 1
```

**Execution trace**:

1. `compute_root_from_merkle_proof(I, 0, [H(tx2,tx3)])`:
   - `current_position = 0` (even) → `current_hash = H(I, H(tx2,tx3)) = Root` ✓
2. `Root == header.block_header.merkle_root` → `true`

The function returns `true`. No real transaction with hash `I` exists; `I` is an internal node. The consumer contract is deceived into believing a payment occurred. [6](#0-5) [7](#0-6)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
