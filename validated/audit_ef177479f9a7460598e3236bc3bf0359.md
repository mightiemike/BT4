### Title
Consensus-path `convert_consensus_tx_to_internal_consensus_tx` bypasses DA-mode and field canonicalization enforced by the gateway, admitting OS-invalid transactions into committed blocks — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary

The `StatelessTransactionValidator` enforces several constraints that are explicitly documented as "preventing transactions from failing the OS": `nonce_data_availability_mode` and `fee_data_availability_mode` must equal `DataAvailabilityMode::L1`, `paymaster_data` must be empty, and `account_deployment_data` must be empty. These checks are applied to every `RpcTransaction` entering through the HTTP gateway. The consensus path — `convert_consensus_tx_to_internal_consensus_tx` — calls `convert_rpc_tx_to_internal` directly and applies none of these checks. The blockifier's `perform_pre_validation_stage` also does not check DA mode. A malicious proposer can therefore craft a `ConsensusTransaction` with `nonce_data_availability_mode = DataAvailabilityMode::L2`, have it accepted by every validator's blockifier, committed to the block, and then cause the Starknet OS to fail the block proof.

### Finding Description

`StatelessTransactionValidator::validate()` applies the following checks to every inbound `RpcTransaction`: [1](#0-0) 

Two of those checks carry explicit OS-enforcement comments: [2](#0-1) [3](#0-2) 

These checks are **only** applied to the RPC gateway path. The consensus path converts a `ConsensusTransaction` by calling `convert_rpc_tx_to_internal` directly: [4](#0-3) 

`convert_rpc_tx_to_internal` performs no DA-mode, paymaster, or account-deployment-data checks: [5](#0-4) 

The blockifier's `perform_pre_validation_stage` checks nonce, fee bounds, and proof facts — but not DA mode: [6](#0-5) 

The Starknet OS Cairo code enforces `nonce_data_availability_mode = L1` at the block level using hard assertions. A transaction with `L2` DA mode passes the blockifier, is committed to the block, and then causes the OS to fail the block proof.

The same gap applies to `paymaster_data` and `account_deployment_data`: the OS enforces these are empty (as the gateway comments state), but the consensus path never checks them.

### Impact Explanation

A malicious proposer sends a `ConsensusTransaction` with `nonce_data_availability_mode = DataAvailabilityMode::L2`. Every honest validator calls `convert_consensus_tx_to_internal_consensus_tx`, which applies no DA-mode check. The blockifier executes the transaction successfully and produces a state diff. The block is committed by consensus. When the OS attempts to prove the block, it hard-asserts `nonce_data_availability_mode = L1` and fails, making the committed block unprovable. This is a **Critical** impact: wrong execution result / wrong state committed for accepted input, with the block's proof permanently broken.

### Likelihood Explanation

Any validator that is selected as a proposer by the Tendermint round-robin can trigger this. Proposer selection is unprivileged — every validator rotates into the proposer role. No special key or out-of-band access is required; the attacker only needs to craft a valid-looking `ConsensusTransaction` protobuf with a non-`L1` DA mode field and broadcast it as part of a normal proposal stream.

### Recommendation

Apply the same canonicalization checks to `ConsensusTransaction` objects inside `convert_consensus_tx_to_internal_consensus_tx` (or in a dedicated consensus-path validator called before the batcher receives the transactions). At minimum, enforce:

- `nonce_data_availability_mode == DataAvailabilityMode::L1`
- `fee_data_availability_mode == DataAvailabilityMode::L1`
- `paymaster_data.is_empty()`
- `account_deployment_data.is_empty()` (for Invoke and Declare)

These are the same invariants the gateway enforces via `StatelessTransactionValidator`, and the OS enforces them unconditionally. The consensus path must not be a bypass route for them.

### Proof of Concept

1. A malicious proposer constructs:
   ```rust
   ConsensusTransaction::RpcTransaction(RpcTransaction::Invoke(
       RpcInvokeTransaction::V3(RpcInvokeTransactionV3 {
           nonce_data_availability_mode: DataAvailabilityMode::L2, // ← invalid
           fee_data_availability_mode:   DataAvailabilityMode::L1,
           // ... otherwise valid fields ...
       })
   ))
   ```
2. The proposer streams this in a `ProposalPart::Transactions` message to all validators.
3. Each validator calls `convert_consensus_tx_to_internal_consensus_tx(tx)` → `convert_rpc_tx_to_internal(tx)`. No DA-mode check is applied. [7](#0-6) 
4. The transaction is forwarded to the batcher via `send_txs_for_proposal`. The blockifier's `perform_pre_validation_stage` does not check DA mode; the transaction executes and produces a state diff.
5. All validators agree on the block (they all ran the same blockifier path), and the block is committed by consensus.
6. The Starknet OS attempts to prove the block. It hard-asserts `nonce_data_availability_mode = L1` for every transaction. The assertion fails, the block proof cannot be generated, and the chain's proof pipeline is halted.

                Ok((InternalConsensusTransaction::L1Handler(internal_tx), None))

    async fn convert_rpc_tx_to_internal(
        &self,
        tx: RpcTransaction,
    ) -> TransactionConverterResult<(InternalRpcTransaction, Option<(ProofFacts, Proof)>)> {
        let (tx_without_hash, proof_data) = match tx {
            RpcTransaction::Invoke(RpcInvokeTransaction::V3(tx)) => {
                let proof_data = if tx.proof_facts.is_empty() {
                    None
                } else {
                    Some((tx.proof_facts.clone(), tx.proof.clone()))
                };
                (InternalRpcTransactionWithoutTxHash::Invoke(tx.into()), proof_data)
            }
            RpcTransaction::Declare(RpcDeclareTransaction::V3(tx)) => {
                let ClassHashes { class_hash, executable_class_hash_v2 } =
                // TODO(Dori): Make this async and spawn a task to compile and add it to the class manager.
                    self.class_manager_client.add_class(tx.contract_class).await?;
                // TODO(Aviv): Ensure that we do not want to
                // allow declare with compiled class hash v1.
                if tx.compiled_class_hash != executable_class_hash_v2 {
                    return Err(TransactionConverterError::ValidateCompiledClassHashError(
                        ValidateCompiledClassHashError::CompiledClassHashMismatch {
                            computed_class_hash: executable_class_hash_v2,
                            supplied_class_hash: tx.compiled_class_hash,
                        },
                    ));
                }
                (
                    InternalRpcTransactionWithoutTxHash::Declare(InternalRpcDeclareTransactionV3 {
                        sender_address: tx.sender_address,
                        compiled_class_hash: tx.compiled_class_hash,
                        signature: tx.signature,
                        nonce: tx.nonce,
                        class_hash,
                        resource_bounds: tx.resource_bounds,
                        tip: tx.tip,
                        paymaster_data: tx.paymaster_data,
                        account_deployment_data: tx.account_deployment_data,
                        nonce_data_availability_mode: tx.nonce_data_availability_mode,
                        fee_data_availability_mode: tx.fee_data_availability_mode,
                    }),
                    None,
                )
            }
            RpcTransaction::DeployAccount(RpcDeployAccountTransaction::V3(tx)) => {
                let contract_address = tx.calculate_contract_address()?;
                (
                    InternalRpcTransactionWithoutTxHash::DeployAccount(
                        InternalRpcDeployAccountTransaction {
                            tx: RpcDeployAccountTransaction::V3(tx),
                            contract_address,
                        },
                    ),
                    None,
                )
            }
        };
        let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
        Ok((InternalRpcTransaction { tx: tx_without_hash, tx_hash }, proof_data))
    }
