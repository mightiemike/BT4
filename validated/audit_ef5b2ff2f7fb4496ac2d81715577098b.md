### Title
Testing-Only `skip_pow_verification` Flag Persisted in Contract State Enables Complete PoW Bypass on Mainnet - (`contract/src/lib.rs`)

---

### Summary

The `init` function accepts `skip_pow_verification: bool` as a caller-supplied initialization argument and stores it permanently in contract state. The inline documentation explicitly marks `true` as "only for testing purposes," yet there is no on-chain guard preventing a mainnet deployment from setting it to `true`. Once set, every subsequent `submit_blocks` call skips all PoW validation for the lifetime of the contract, allowing any relayer to inject fabricated block headers into the canonical chain.

---

### Finding Description

`InitArgs` exposes `skip_pow_verification` as a plain boolean field with no default and no runtime enforcement: [1](#0-0) 

The `init` function copies this value directly into persistent contract state: [2](#0-1) 

The documentation comment on `init` acknowledges the flag is testing-only but enforces nothing: [3](#0-2) 

Every call to `submit_blocks` passes `self.skip_pow_verification` directly into `submit_block_header`: [4](#0-3) 

Inside `submit_block_header`, when the flag is `true`, the entire PoW validation block is skipped — both the difficulty target check (`check_target`) and the hash-vs-target comparison: [5](#0-4) 

The relayer's `InitConfig` struct also carries `skip_pow_verification: bool` with no default value, meaning it must be explicitly set in configuration files or environment variables — a plain misconfiguration risk: [6](#0-5) 

---

### Impact Explanation

If the contract is initialized with `skip_pow_verification = true` on mainnet:

1. Any relayer (or any account satisfying the `#[trusted_relayer]` gate) can submit block headers with arbitrary `bits`, arbitrary timestamps, and no valid PoW hash.
2. These headers are accepted into `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height`, and can become the canonical tip via the normal chain-work comparison path.
3. `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` will return `true` for Merkle proofs anchored to these fabricated blocks, because they only check that the block hash is in `mainchain_header_to_height` and that the Merkle proof matches the stored `merkle_root` — both of which are attacker-controlled when PoW is skipped.

The corrupted canonical state is: `mainchain_tip_blockhash`, `mainchain_height_to_header`, `mainchain_header_to_height`, and `headers_pool` all populated with fabricated headers, causing `verify_transaction_inclusion` to certify non-existent Bitcoin transactions as confirmed. [7](#0-6) 

---

### Likelihood Explanation

The risk is realistic for the following reasons:

- The flag is a plain `bool` in a configuration struct with no compile-time or runtime guard distinguishing testnet from mainnet deployments.
- The relayer's `InitConfig` requires the operator to explicitly set `skip_pow_verification` in a config file or environment variable; there is no safe default (`false`) enforced by the type system.
- The only protection is a documentation comment, which is identical in nature to the pSTAKE finding: a comment saying "this is for testnet only" with no code-level enforcement.
- Testnet deployments routinely use `skip_pow_verification = true` (confirmed by test helpers), making it easy to accidentally carry that configuration into a mainnet deployment. [8](#0-7) 

---

### Recommendation

1. **Enforce `skip_pow_verification = false` at the contract level for mainnet deployments.** Add a runtime `require!` in `init` that panics if `skip_pow_verification == true && network == Network::Mainnet`.
2. **Remove `skip_pow_verification` from `InitArgs` entirely** for production builds, gating it behind a Rust feature flag (e.g., `#[cfg(feature = "testing")]`) so it cannot appear in a mainnet binary.
3. **Set a safe default of `false`** in `InitConfig` in the relayer (`#[serde(default)]` pointing to a `fn default_false() -> bool { false }`), so omitting the field in config does not silently enable bypass.

---

### Proof of Concept

**Setup**: Deploy the contract on NEAR mainnet with:
```json
{
  "genesis_block_hash": "<valid_hash>",
  "genesis_block_height": 0,
  "skip_pow_verification": true,
  "gc_threshold": 52704,
  "network": "Mainnet",
  "submit_blocks": [<valid_genesis_header>]
}
```

**Attack**: Call `submit_blocks` with a fabricated header batch:
```json
[{
  "version": 4,
  "prev_block_hash": "<current_tip_hash>",
  "merkle_root": "<attacker_chosen_merkle_root>",
  "time": <current_time>,
  "bits": 1,
  "nonce": 0
}]
```

Because `self.skip_pow_verification == true`, the branch at line 517 is not entered. The header passes `submit_block_header_inner` and is stored as the new canonical tip.

**Result**: Call `verify_transaction_inclusion_v2` with a Merkle proof constructed against `attacker_chosen_merkle_root`. The function returns `true`, certifying a fabricated Bitcoin transaction as confirmed on-chain. [9](#0-8)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
