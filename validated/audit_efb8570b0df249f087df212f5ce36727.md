### Title
ERC-4626 Share Inflation Causes Silent Skip in `wrapVaultAsset`, Permanently Locking User Assets in DDA — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.wrapVaultAsset` gates the entire wrapping step on `previewDeposit(assetBalance) != 0`. An attacker can perform the classic ERC-4626 first-deposit share inflation attack on the external vault registered as a Nado product token, forcing `previewDeposit` to return `0`. The function then silently exits without wrapping the user's raw assets, leaving them permanently stuck in the `DirectDepositV1` (DDA) contract and uncreditable to the user's subaccount.

---

### Finding Description

`wrapVaultAsset` in `ContractOwner.sol` is the bridge that converts raw underlying assets held in a user's DDA into ERC-4626 vault shares, which are then credited to the user's Nado subaccount via `creditDeposit`. [1](#0-0) 

The critical gate is at line 525:

```solidity
if (IERC4626Base(tokenAddr).previewDeposit(assetBalance) != 0) {
    DirectDepositV1(directDepositV1).withdraw(IIERC20Base(assetTokenAddr));
    ...
    IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1);
}
```

If `previewDeposit` returns `0`, the entire body is skipped silently — no revert, no event, no retry signal. The raw assets remain in the DDA. [2](#0-1) 

The function has **no access control** (`external` with no modifier), so any caller — including the attacker — can invoke it at the moment of their choosing.

The attack path exploits the standard ERC-4626 share inflation pattern on the external vault (`tokenAddr`):

1. **Attacker mints 1 share** in the external vault (first depositor), paying 1 wei of the underlying asset. Vault state: `totalAssets = 1`, `totalShares = 1`.
2. **Attacker donates** `N` underlying tokens directly to the vault address (bypassing `deposit`), inflating `totalAssets` to `N + 1` while `totalShares` stays at `1`.
3. **User's assets arrive** in their DDA (e.g., `100e6` USDC).
4. **Attacker calls `wrapVaultAsset`**. `previewDeposit(100e6)` computes `100e6 * 1 / (N+1)`. If `N >= 100e6`, this rounds down to `0`.
5. The `if` body is skipped. The DDA still holds raw USDC, not vault shares.
6. **`creditDeposit` is called** (by the sequencer bot). It iterates over supported product IDs and deposits matching token balances. If the raw underlying asset (USDC) is not a registered Nado product — only the vault token (e.g., `aUSDC`) is — the USDC balance in the DDA is never deposited. [3](#0-2) 

7. **Attacker burns their 1 share**, recovering `N + 1` underlying tokens. Net cost to attacker: gas only (the donated `N` tokens are fully recovered from the vault).

The `isWrapVaultAssetReady` view function suffers the same flaw — it returns `false` when `previewDeposit` returns `0`, so the off-chain bot will not retry the wrap. [4](#0-3) 

---

### Impact Explanation

The user's raw underlying assets are stuck in the DDA contract. The DDA is owned by `ContractOwner`, not the user. The only recovery path is `withdrawFromDirectDepositV1`, which is `onlyOwner`. [5](#0-4) 

Until the owner intervenes, the user cannot access their funds, cannot trade, and cannot participate in the protocol. The broken invariant is: *assets deposited to a DDA for a vault-backed product must always be wrappable into vault shares and credited to the subaccount*. The silent skip violates this invariant without any on-chain signal.

**Impact: High** — user funds are inaccessible without privileged owner intervention.

---

### Likelihood Explanation

**Likelihood: Medium.** The attack requires:
- The external ERC-4626 vault registered as a Nado product token to lack dead-share protection (common in newly deployed or minimally-audited vaults).
- The attacker to front-run the first deposit into that vault (standard mempool observation).
- The attacker to spend capital equal to the user's deposit amount for the donation (fully recovered on burn).

The attacker bears no net financial loss beyond gas, making the attack economically rational as a griefing vector against targeted users.

---

### Recommendation

1. **Replace the silent skip with a revert or minimum-shares check**: Instead of `if (previewDeposit(...) != 0)`, use `require(previewDeposit(...) >= minShares, "insufficient shares")` so the off-chain bot receives a clear failure signal and can retry.
2. **Pass a `minSharesOut` parameter to `deposit`**: Call `IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1)` with a slippage guard, reverting if the vault returns fewer shares than expected.
3. **Require vault-side dead-share protection**: When onboarding a new ERC-4626 vault as a product token, verify it implements dead shares or virtual offset (as in OpenZeppelin ERC4626) to prevent share price manipulation.

### Proof of Concept

```
Setup:
  - Nado product `productId` has token = aUSDC (ERC-4626 vault over USDC)
  - USDC is NOT a standalone Nado product
  - User sends 100e6 USDC to their DDA address

Attack:
  1. Attacker calls aUSDC.deposit(1, attacker)
     → vault: totalAssets=1, totalShares=1

  2. Attacker calls USDC.transfer(aUSDC_vault, 100e6 + 1)
     → vault: totalAssets=100e6+2, totalShares=1

  3. Attacker calls ContractOwner.wrapVaultAsset(userSubaccount, productId)
     → assetBalance = 100e6
     → previewDeposit(100e6) = 100e6 * 1 / (100e6+2) = 0
     → if-body skipped, DDA still holds 100e6 USDC

  4. Sequencer calls creditDeposit() on DDA
     → iterates productIds, finds aUSDC balance = 0, USDC not a product
     → nothing deposited

  5. Attacker calls aUSDC.redeem(1, attacker, attacker)
     → receives 100e6+2 USDC back (net: +1 USDC profit from vault inflation)

Result:
  - User's 100e6 USDC is stuck in DDA
  - User's subaccount has no credit
  - Recovery requires ContractOwner owner to call withdrawFromDirectDepositV1
```

    function withdrawFromDirectDepositV1(bytes32 subaccount, address token)
        external
        onlyOwner
    {
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        if (token == address(0)) {
            uint256 preBalance = address(this).balance;
            DirectDepositV1(directDepositV1).withdrawNative();
            uint256 postBalance = address(this).balance;
            require(postBalance > preBalance, "empty");
            (bool success, ) = msg.sender.call{value: postBalance - preBalance}(
                ""
            );
            require(success, "xfer");
        } else {
            uint256 preBalance = IERC20Base(token).balanceOf(address(this));
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(token));
            uint256 postBalance = IERC20Base(token).balanceOf(address(this));
            require(postBalance > preBalance, "empty");
            IERC20Base(token).safeTransfer(
                msg.sender,
                postBalance - preBalance
            );
        }
    }
