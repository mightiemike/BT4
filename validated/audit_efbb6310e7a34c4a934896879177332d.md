### Title
Bonus Reward Computation Uses Current Delegation Shares for All Historical Segments, Allowing Inflated Bonus Claims - (File: x/tieredrewards/keeper/bonus_rewards.go)

### Summary

`computeSegmentBonus` always uses the position's **current** live delegation shares (`pos.Delegation.Shares`) when computing bonus for every historical time segment. Because delegation shares can be increased by operations such as `AddToPosition` without the historical share count being snapshotted, a user who adds shares after validator events have accumulated will receive bonus as if they held the larger share count throughout the entire historical period.

### Finding Description

In `processEventsAndClaimBonus`, the keeper walks through all validator events since `pos.LastEventSeq` and calls `computeSegmentBonus` for each bonded segment:

```go
bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
``` [1](#0-0) 

Inside `computeSegmentBonus`, the token amount for the segment is computed as:

```go
tokens := pos.Delegation.Shares.Mul(tokensPerShare)
``` [2](#0-1) 

`pos.Delegation` is loaded from the **live staking state** at the time of the claim call, not snapshotted at each event boundary. The `tokensPerShare` argument is correctly snapshotted at event time (stored in `evt.TokensPerShare`), but the **shares** component is always the current live value. [3](#0-2) 

This asymmetry means: if a user increases their delegation shares (via `MsgAddToPosition`) after historical validator events have been recorded but before claiming rewards, every historical segment is retroactively computed with the inflated share count.

The `processEventsAndClaimBonus` function's own comment acknowledges the intent to use "snapshot rates," but only `tokensPerShare` is snapshotted — shares are not: [4](#0-3) 

### Impact Explanation

The attacker receives bonus rewards from `types.RewardsPoolName` computed as if they held `currentShares` throughout all historical bonded segments, when they only held `originalShares < currentShares` during those segments. The excess bonus paid is:

```
excess = (currentShares - originalShares) * tokensPerShare_at_event * BonusApy * historicalDuration / SecondsPerYear
```

This is a direct, unbacked withdrawal from the `RewardsPoolName` module account, draining funds that belong to other legitimate position holders. [5](#0-4) 

### Likelihood Explanation

Any position owner can trigger this by:
1. Opening a position with a small delegation.
2. Waiting for validator events (slash, unbond, bond) to accumulate historical segments.
3. Calling `MsgAddToPosition` to increase their delegation shares.
4. Claiming rewards — all historical segments are now computed with the larger share count.

The `MsgAddToPosition` message server path exists (evidenced by `msg_server_add_to_position_test.go`).



No privileged role is required. The attack is repeatable and scales with the size of the rewards pool and the length of the historical window.

### Recommendation

Snapshot the delegation share count at each validator event boundary, analogous to how `tokensPerShare` is already snapshotted. Store a `SharesAtEvent` field in `ValidatorEvent` (or in the position's event-replay state), and pass the historical share count to `computeSegmentBonus` instead of `pos.Delegation.Shares`. Alternatively, force a full reward claim (settling all pending events) before any operation that modifies the position's delegation shares, so that no historical segments remain open when shares change. [6](#0-5) 

### Proof of Concept

**Setup:**
- Validator `V` is bonded; `tokensPerShare = 1.0`.
- User opens position `P` with `1000` delegation shares at time `T0`.

**Step 1 — Validator event accumulates:**
- At time `T1`, validator `V` is briefly unbonded then re-bonded. Two events are appended with `tokensPerShare = 1.0`. The bonded segment `[T0, T1]` (duration = 1 year) is pending.

**Step 2 — User inflates shares before claiming:**
- At time `T1 + ε`, user calls `MsgAddToPosition`, adding `9000` more tokens. Delegation shares become `10000`.

**Step 3 — User claims rewards:**
- `processEventsAndClaimBonus` loads `pos.Delegation.Shares = 10000`.
- For segment `[T0, T1]` it computes: `10000 * 1.0 * BonusApy * 1yr / 1yr = 10000 * BonusApy`.
- Correct computation should be: `1000 * 1.0 * BonusApy * 1yr / 1yr = 1000 * BonusApy`.
- **Attacker receives 10× the legitimate bonus** for the historical segment, with the excess `9000 * BonusApy` tokens drained from `RewardsPoolName`. [7](#0-6)

func (k Keeper) computeSegmentBonus(pos types.PositionState, tier types.Tier, segmentStart, segmentEnd time.Time, tokensPerShare math.LegacyDec) math.Int {
	if !pos.ExitUnlockAt.IsZero() && segmentEnd.After(pos.ExitUnlockAt) {
		segmentEnd = pos.ExitUnlockAt
	}

	if !segmentEnd.After(segmentStart) {
		return math.ZeroInt()
	}

	durationSeconds := int64(segmentEnd.Sub(segmentStart) / time.Second)
	if durationSeconds <= 0 {
		return math.ZeroInt()
	}

	tokens := pos.Delegation.Shares.Mul(tokensPerShare)

	return tokens.
		Mul(tier.BonusApy).
		MulInt64(durationSeconds).
		QuoInt64(types.SecondsPerYear).
		TruncateInt()
}

func (k Keeper) getDelegation(ctx context.Context, delegatorAddress string) (*stakingtypes.Delegation, error) {
	delAddr, err := sdk.AccAddressFromBech32(delegatorAddress)

		return nil, err

		return nil, nil
