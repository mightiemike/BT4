Audit Report

## Title
Router-mediated swaps pass the router address as `sender` to `SwapAllowlistExtension.beforeSwap`, enabling allowlist bypass by any unprivileged user - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the originating user EOA. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the router address. Any pool admin who allowlists the router (required for router-mediated swaps to function at all) inadvertently grants every user the ability to bypass the per-user allowlist by routing through the public router.

## Finding Description

**Call path:**

1. User (non-allowlisted EOA) calls `MetricOmmSimpleRouter.exactInputSingle(params)`.
2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — `msg.sender` at the pool is the **router address**.
3. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` — passes the router address as `sender`.
4. `ExtensionCalling._beforeSwap` encodes `sender = router_address` and dispatches to `SwapAllowlistExtension.beforeSwap(router_address, ...)`.
5. `SwapAllowlistExtension.beforeSwap` evaluates:
   ```solidity
   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender])
   ```
   where `msg.sender` = pool and `sender` = router. The check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user_EOA]`.

**Root cause:** `MetricOmmPool.swap` unconditionally uses `msg.sender` as the `sender` forwarded to extensions. There is no mechanism to propagate the original user identity through the router.

**Existing guards are insufficient:** The `onlyPool` guard in `BaseMetricExtension` only verifies the caller is a registered pool — it does not validate the `sender` argument. The allowlist mapping is keyed by `(pool, sender)` where `sender` is always the immediate caller of `pool.swap`, which is the router for all router-mediated flows.

**Dilemma for pool admin:**
- If the router is **not** allowlisted: allowlisted users cannot use the router (broken core swap functionality for the intended audience).
- If the router **is** allowlisted: every user, including non-allowlisted ones, can bypass the per-user gate by calling `exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` through the public router.

The `exactInput` multi-hop path compounds this: for hop `i > 0`, the payer stored in transient context is `address(this)` (the router itself), so the router is both the `msg.sender` to the pool and the economic actor for intermediate hops.

## Impact Explanation
A non-allowlisted user can swap on a curated pool that the pool admin intended to restrict to specific counterparties. This is a direct policy bypass on pools using `SwapAllowlistExtension`, allowing unauthorized token swaps and undermining the curation model. The impact is loss of access-control integrity on pools whose entire value proposition depends on the allowlist (e.g., institutional or KYC-gated pools). This matches the "admin-boundary break: pool admin exceeds caps, bypasses timelocks, or factory/oracle role checks are bypassed by an unprivileged path" allowed impact and the "allowlist bypass" invariant in the contest scope.

## Likelihood Explanation
The `MetricOmmSimpleRouter` is the canonical public swap entrypoint documented and deployed alongside the core pool. Any user aware of the router can call `exactInputSingle` with a pool that has `SwapAllowlistExtension` active. No special privileges, flash loans, or multi-block setup are required. The bypass is repeatable on every block and requires only a standard ERC-20 approval to the router.

## Recommendation
The pool must forward the original initiator, not `msg.sender`, to extensions. One approach: add an optional `address initiator` parameter to `pool.swap` that the router populates with `msg.sender` (the user EOA), and pass that as `sender` to `_beforeSwap`. Alternatively, `SwapAllowlistExtension.beforeSwap` could decode the true initiator from `extensionData`, but this requires router cooperation and is fragile. The cleanest fix is to have the router pass the user EOA explicitly and have the pool forward it as `sender` to all extension hooks.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Setup:
// 1. Deploy pool with SwapAllowlistExtension.
// 2. Pool admin allowlists the router: swapExtension.setAllowedToSwap(pool, address(router), true)
//    (required for any router-mediated swap to work).
// 3. Non-allowlisted attacker EOA calls:

router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool: address(pool),
    tokenIn: token0,
    tokenOut: token1,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    recipient: attacker,
    deadline: block.timestamp + 1,
    priceLimitX64: 0,
    extensionData: ""
}));

// Result: swap succeeds because beforeSwap checks allowedSwapper[pool][router] == true,
// never checking whether the attacker EOA is allowlisted.
// The attacker EOA is not in allowedSwapper[pool][attacker_EOA], but that mapping is never consulted.
```

Foundry test: deploy pool with `SwapAllowlistExtension`, allowlist only the router address, assert that a non-allowlisted EOA calling `exactInputSingle` succeeds (demonstrating bypass), then assert the same EOA calling `pool.swap` directly reverts with `NotAllowedToSwap`.
