### Title
Truncating Integer Division in Quadratic Multiply Cost Undercharges Execution Cost — (`File: src/more_ops.rs`)

---

### Summary

The `op_multiply` operator and the unknown-op cost-function-2 path both compute the quadratic component of the multiplication cost using plain integer division (`/`), which truncates toward zero (rounds down). Because cost accounting must be conservative — charged cost must be ≥ true cost — this division should round **up**. The truncation silently underestimates the cost of every `*` step, allowing an attacker to craft CLVM programs whose true cost exceeds `max_cost` but whose computed cost does not, bypassing the consensus cost limit.

---

### Finding Description

In `src/more_ops.rs`, the cost of a single multiplication step is computed as:

```rust
// op_multiply fast path (Buffer), line 616
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;

// op_multiply fast path (U32), line 624
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;

// op_multiply no-fastpath, line 644
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;

// unknown-op cost_function == 2, line 238
cost += (l0 * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
```

`MUL_SQUARE_COST_PER_BYTE_DIVIDER` is `128`. Rust's `/` on unsigned integers truncates toward zero. For any `(l0 * l1)` that is not an exact multiple of 128, the charged cost is strictly less than the true quadratic cost. The per-step undercharge is in the range `[1, 127]` cost units.

The invariant that must hold is: **the cost returned by any operator must be ≥ the actual computational work performed**, so that `check_cost(cost, max_cost)` reliably rejects programs that exceed the block cost limit. Rounding down breaks this invariant in the same way that rounding down `supplyAssets` in MetaMorpho breaks the `supplyCap` invariant.

Note that `limbs_for_int` (line 101) correctly uses `div_ceil(8)`, and `atom_length_bits` in `src/serde/serialized_length.rs` (line 30) also correctly uses `div_ceil(8)`. The multiply cost path is the sole site that uses truncating division where ceiling division is required.

---

### Impact Explanation

The cost model is consensus-critical: every full node must agree on whether a program's cost exceeds the block limit. If the cost is underestimated, a program whose true cost exceeds `max_cost` is accepted by all nodes (since all nodes run the same deterministic underestimating code). This is not a divergence between nodes; it is a systematic bypass of the intended cost ceiling.

An attacker can craft a `*`-heavy CLVM program with operand sizes chosen to maximize the per-step truncation error (e.g., `l0 = 127, l1 = 1` gives `(127 * 1) / 128 = 0` instead of `1`, an undercharge of 1 per step; larger products give up to 127 per step). Across many steps the undercharge accumulates. The practical ceiling is bounded by the result-size limit (1024 bytes) and the cost-per-step floor (`MUL_COST_PER_OP = 885`), but the aggregate undercharge can reach hundreds of thousands of cost units for a single program, allowing computationally heavier programs than the block limit intends to pass validation.

---

### Likelihood Explanation

The entry path is direct: an attacker submits attacker-controlled CLVM bytes containing `*` operations. No special privilege, social engineering, or configuration is required. The operand sizes that maximize truncation error are trivially constructable. The bug is present in all build configurations (both the fast path and the `no-fastpath` feature path).

---

### Recommendation

Replace all four truncating divisions with ceiling division:

```rust
// Before (truncates, underestimates cost):
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;

// After (ceiling division, conservative):
cost += (l0 as Cost * l1).div_ceil(MUL_SQUARE_COST_PER_BYTE_DIVIDER);
```

Apply the same fix to the unknown-op cost-function-2 path at line 238. This matches the pattern already used correctly in `limbs_for_int` (`v.bits().div_ceil(8)`) and `atom_length_bits` (`num_bits.div_ceil(8)`).

---

### Proof of Concept

**Trigger**: Submit a CLVM program that applies `*` repeatedly with a 127-byte first operand and a 1-byte second operand.

For each such step:
- `l0 = 127`, `l1 = 1`
- Charged quadratic term: `(127 * 1) / 128 = 0` (truncated)
- Correct quadratic term: `(127 * 1).div_ceil(128) = 1`
- Per-step undercharge: **1 cost unit**

For a step with `l0 = 255, l1 = 127`:
- `(255 * 127) / 128 = 32385 / 128 = 252` (truncated from 252.99…)
- Correct: `32385.div_ceil(128) = 253`
- Per-step undercharge: **1 cost unit** (but the product `l0 * l1` can be tuned to give up to 127 undercharge per step)

For `l0 = 1, l1 = 127`:
- `(1 * 127) / 128 = 0` (truncated)
- Correct: `1`
- Per-step undercharge: **1 cost unit**

For `l0 = 127, l1 = 127`:
- `(127 * 127) / 128 = 16129 / 128 = 126` (truncated from 126.007…)
- Correct: `127`
- Per-step undercharge: **1 cost unit**

The maximum single-step undercharge of 127 occurs when `l0 * l1 ≡ 127 (mod 128)`, e.g., `l0 = 127, l1 = 129` — but operand sizes are capped at 256 bytes, so `l0 * l1` can reach `256 * 256 = 65536 = 512 * 128` (exact, no truncation at the maximum). The worst-case undercharge per step is 127 cost units, achievable with many intermediate operand-size combinations. Across the maximum feasible number of steps before the result exceeds 1024 bytes, the total undercharge is bounded but non-trivial, allowing programs to consume more computation than the cost limit intends to permit.
