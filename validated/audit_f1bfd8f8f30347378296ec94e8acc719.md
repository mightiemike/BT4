Audit Report

## Title
SwapAllowlistExtension Gates Router Address Instead of Actual Swapper, Allowing Allowlist Bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` from the pool's perspective — the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` is used, `sender` is the router address, not the originating user. Any pool admin who allowlists the router to enable router-mediated access for curated users simultaneously grants unrestricted swap access to every user who routes through that router.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this `sender` value directly to each configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then gates on `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` from the pool's perspective. The originating user (`msg.sender` at the router level) is stored only in transient callback context for payment purposes and is never forwarded to the extension: [4](#0-3) 

The effective allowlist lookup becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`. There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users from using the same router. Allowlisting the router grants access to all users; not allowlisting it denies access to all users including legitimate ones.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC-verified counterparties or whitelisted protocols) is fully bypassed by any unpermissioned user who routes through `MetricOmmSimpleRouter`. The attacker receives identical execution quality as an allowlisted user. This breaks the core access-control invariant of the pool: non-allowlisted users can drain liquidity from a pool designed to serve only a restricted set of counterparties. This constitutes a direct, unconditional policy bypass with fund-impacting consequences — **High severity**.

## Likelihood Explanation
The scenario requires the pool admin to allowlist the router address, which is the natural and expected configuration step for any pool that wants its curated users to access the router. The router is a first-class supported periphery entry point. Any production pool that enables router access for its allowlisted users is immediately and unconditionally vulnerable. No privileged access or special conditions are required by the attacker beyond calling the router — **Medium likelihood**.

## Recommendation
The extension must resolve the true economic actor, not the intermediary. The router should encode `msg.sender` (the originating user) into `extensionData` for each hop:

```solidity
// In MetricOmmSimpleRouter.exactInputSingle (and other entry points):
bytes memory enrichedExtensionData = abi.encode(msg.sender, params.extensionData);
// pass enrichedExtensionData instead of params.extensionData to pool.swap()
```

`SwapAllowlistExtension.beforeSwap` should then decode the originating user from `extensionData` when `sender` is a registered router, and check that address instead of `sender`. Alternatively, maintain a registry of approved routers in `BaseMetricExtension`; if `sender` is a registered router, fall back to the decoded payer address from `extensionData`.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, router, true)   // enable router for allowlisted users
  - Pool admin calls setAllowedToSwap(pool, alice, true)    // alice is the intended curated user
  - bob is NOT allowlisted

Attack:
  1. bob calls router.exactInputSingle({pool: pool, tokenIn: token0, zeroForOne: true, ...})
  2. Router calls pool.swap(recipient=bob, zeroForOne=true, ..., extensionData="")
  3. Pool calls _beforeSwap(sender=router, ...)
  4. Extension checks: allowedSwapper[pool][router] == true → passes
  5. Swap executes; bob receives token1 output

Result: bob, a non-allowlisted address, successfully swaps on a curated pool.
Direct pool call by bob (without router) would revert with NotAllowedToSwap.
```
