### Title
Difficulty Retarget Reads Mainchain Ancestor Instead of Fork Ancestor, Corrupting PoW Verification for Fork Blocks - (File: `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

When computing the expected difficulty for a submitted block that extends a fork, `get_next_work_required` calls `get_header_by_height(first_block_height)` to retrieve the first block of the difficulty adjustment period. However, `get_header_by_height` always reads from `mainchain_height_to_header` — the mainchain index — rather than walking the fork's actual ancestor chain. If the fork diverges before `first_block_height`, the mainchain block at that height is a completely different block from the fork's true ancestor at that height. The resulting difficulty target is computed from the wrong block's timestamp, breaking the PoW enforcement invariant for fork submissions. The code itself acknowledges this in a TODO comment.

---

### Finding Description

`get_header_by_height` is implemented in `contract/src/lib.rs`:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // ← always the mainchain index
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}

// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [4](#0-3) 

The correct behavior is to walk backwards through `prev_block_hash` pointers from the fork tip until reaching `first_block_height`, which is what Bitcoin Core does. Instead, the contract substitutes the mainchain block at that height — a completely different block whenever the fork diverges before the retarget boundary.

---

### Impact Explanation

The difficulty target for a fork block at a retarget boundary is computed as:

```
new_target = prev_target * (prev_block.time - first_block.time) / target_timespan
```

If the fork diverges before `first_block_height`, `first_block.time` is taken from the **mainchain** block at that height, not the fork's ancestor. The timestamp difference `(prev_block.time - first_block.time)` is therefore wrong. Depending on the direction of the error:

- If the mainchain's `first_block.time` is **later** than the fork's true ancestor's time, the computed `actual_time_taken` is smaller → the retargeted difficulty is **harder** than it should be, causing legitimate fork blocks to be rejected.
- If the mainchain's `first_block.time` is **earlier** than the fork's true ancestor's time, `actual_time_taken` is larger → the retargeted difficulty is **easier** than it should be, allowing fork blocks with **insufficient real PoW** to pass `check_pow` and be accepted.

The second case is the security-critical one: an attacker who controls a fork chain can craft a scenario where the wrong (easier) difficulty target is computed, allowing them to submit fork blocks that would not satisfy the correct difficulty, and potentially trigger a chain reorganization with less cumulative work than the honest chain requires.

---

### Likelihood Explanation

The bug is triggered whenever a fork block falls on a difficulty adjustment boundary (every 2016 blocks for Bitcoin/Litecoin, every block after height 145,000 for Dogecoin) **and** the fork diverges at or before `first_block_height`. This is a realistic scenario for any relayer submitting a competing fork. The entry path requires no privileges: `submit_blocks` is callable by any account that meets the trusted-relayer staking requirement, and the staking mechanism is designed to be open. The TODO comment in `dogecoin.rs` confirms the developers themselves identified this as an unresolved correctness question.

---

### Recommendation

Replace `get_header_by_height` calls inside `get_next_work_required` with a backward walk through `get_prev_header` starting from `prev_block_header`, stopping when `block_height == first_block_height`. This mirrors Bitcoin Core's `GetAncestor` logic and correctly resolves the ancestor on the fork chain rather than the mainchain.

```rust
// Instead of:
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);

// Use:
let mut cursor = prev_block_header.clone();
while cursor.block_height > first_block_height {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
let interval_tail_extend_header = cursor;
```

### Proof of Concept

1. The contract is initialized with a Bitcoin mainchain starting at height 0.
2. The mainchain advances to height 2016 (the first retarget boundary). The mainchain block at height 0 has timestamp `T_main_0`.
3. An attacker submits a fork starting at height 1 (diverging before height 0 is impossible, but diverging at height 1 means the fork's block at height 0 is the shared genesis). For a more concrete case: the fork diverges at height 1, so the fork's block at height 0 is the same genesis block — timestamps match. To exploit, the fork must diverge **before** the retarget window's start.
4. More concretely: initialize at height 2016, advance mainchain to height 4032 (second retarget). The fork diverges at height 2017. The fork's ancestor at height 2016 is the shared block, but the fork's ancestor at height 2015 (if `blocks_to_go_back = 2016`) is a fork-specific block with timestamp `T_fork_2015 ≠ T_main_2015`.
5. When the fork block at height 4032 is submitted, `get_header_by_height(2016)` returns the **mainchain** block at 2016, not the fork's ancestor. The timestamp difference is computed incorrectly.
6. If `T_main_2016 > T_fork_2016_ancestor`, the computed difficulty is lower than correct, and the fork block passes `check_pow` with less PoW than required.
7. If the fork accumulates enough (artificially easy) chain work, `reorg_chain` is triggered, corrupting `mainchain_tip_blockhash` and all downstream `verify_transaction_inclusion` results. [5](#0-4) [6](#0-5)

fn get_next_work_required(
    config: &NetworkConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    if (prev_block_header.block_height + 1) % config.difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            if block_header.time
                > prev_block_header.block_header.time + 2 * config.pow_target_spacing
            {
                return config.proof_of_work_limit_bits;
            }

            let mut current_block_header = prev_block_header.clone();
            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            let last_bits = current_block_header.block_header.bits;
            return last_bits;
        }
        return prev_block_header.block_header.bits;
    }

    let first_block_height =
        prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);

    let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
    calculate_next_work_required(
        config,
        prev_block_header,
        interval_tail_extend_header.block_header.time.into(),
    )
}
