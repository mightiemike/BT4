Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User — Allowlist Bypassed or Broken for All Router-Mediated Swaps - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap()` always sets to its own `msg.sender` — the immediate caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the router is that immediate caller, so the extension evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][end_user]`. This produces two fund-impacting failure modes: allowlist bypass (if the router is allowlisted) or complete breakage of the supported periphery for legitimate users (if it is not).

## Finding Description

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist using `msg.sender` (the pool) as the mapping key:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

`MetricOmmPool.swap()` populates `sender` with its own `msg.sender` — always the immediate caller of `pool.swap()`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← always the immediate caller of pool.swap()
    ...
);
``` [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` at the pool for every swap: [3](#0-2) 

The same pattern holds for `exactInput` (all hops), `exactOutputSingle`, and `exactOutput`: [4](#0-3) [5](#0-4) [6](#0-5) 

The `_beforeSwap` internal in `ExtensionCalling` faithfully forwards `sender` to the extension without any transformation: [7](#0-6) 

There is no mechanism in the router, pool, or extension to recover or forward the original end-user address. The extension therefore always evaluates `allowedSwapper[pool][router]`, never `allowedSwapper[pool][end_user]`.

## Impact Explanation

**Bypass path (High):** A pool admin configures `SwapAllowlistExtension` to restrict swaps to a curated set of addresses. To let those users access the router, the admin calls `setAllowedToSwap(pool, router, true)`. Because the extension checks `sender = router`, every public user who calls any `exact*` function on the router now passes the gate. The allowlist is entirely ineffective. In a pool with oracle-anchored pricing or subsidized rates, unauthorized traders can extract LP value at will — a direct loss of LP principal.

**Broken-functionality path (Medium):** If the admin does not allowlist the router, every legitimately allowlisted user who calls the router is blocked with `NotAllowedToSwap`. The only working path is a direct `pool.swap()` call requiring the user to implement their own callback — the supported periphery is unusable for the pool's intended participants, constituting broken core swap functionality.

Both outcomes are reachable by any public user without any privileged action beyond the normal pool configuration the admin already performed.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool that deploys `SwapAllowlistExtension` and expects users to interact through the router will immediately hit one of the two failure modes. The admin's natural remediation attempt (allowlisting the router) triggers the bypass. No special timing, flash loan, or oracle manipulation is required — the condition is triggered on every router-mediated swap.

## Recommendation

The extension must resolve the actual end user, not the proximate caller. Two options:

1. **Pass the original initiator through the router via transient storage.** Store `msg.sender` of the router call in a transient storage slot before calling the pool (analogous to how the router already stores the payer via `_setNextCallbackContext`), expose it via a view function, and have the extension read it when `sender` is a known router address.

2. **Accept an `extensionData` payload containing the verified end-user address.** The extension could decode an originator address from `extensionData` when present, falling back to `sender` for direct calls. The router would populate this field with `msg.sender` before calling the pool.

## Proof of Concept

```
Setup:
  pool = deploy MetricOmmPool with SwapAllowlistExtension as beforeSwap hook
  admin calls setAllowedToSwap(pool, router, true)   // intending to enable router access for allowlisted users

Attack:
  attacker (not in allowlist) calls:
    router.exactInputSingle({pool: pool, recipient: attacker, ...})

Trace:
  router.exactInputSingle()
    → pool.swap(recipient=attacker, ...)          // msg.sender = router
      → _beforeSwap(sender=router, ...)           // MetricOmmPool.sol L230-231
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  // ✓ passes — attacker bypasses allowlist
      → swap executes, attacker receives tokens

Result:
  Non-allowlisted attacker completes swap in a curated pool.
  LP principal is exposed to unauthorized traders at oracle-anchored or subsidized rates.
```
