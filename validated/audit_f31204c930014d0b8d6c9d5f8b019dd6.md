### Title
`ENABLE_SECP_OPS` Flag Bypassed via 4-Byte Opcode Form, Allowing Ungated Secp Signature Verification — (`File: src/chia_dialect.rs`)

---

### Summary

The `ChiaDialect::op()` function in `src/chia_dialect.rs` dispatches `op_secp256k1_verify` and `op_secp256r1_verify` through two separate code paths: a 1-byte opcode path (opcodes 64 and 65) that is correctly gated behind `ClvmFlags::ENABLE_SECP_OPS`, and a 4-byte opcode path (opcodes `0x13d61f00` and `0x1c3a8f00`) that is **not gated at all**. An attacker supplying attacker-controlled CLVM bytes using the 4-byte opcode form can invoke secp signature verification in any execution context, including `MEMPOOL_MODE`, regardless of whether `ENABLE_SECP_OPS` is set.

---

### Finding Description

In `src/chia_dialect.rs`, the `op()` method of `ChiaDialect` handles operator dispatch. For 1-byte opcodes, the secp operators are correctly gated:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [1](#0-0) 

However, for 4-byte opcodes, the same operators are dispatched **without any flag check**:

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
``` [2](#0-1) 

The `MEMPOOL_MODE` constant explicitly omits `ENABLE_SECP_OPS`, meaning secp ops are intended to be unavailable in that mode:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [3](#0-2) 

The `ENABLE_SECP_OPS` flag is defined as a distinct, opt-in capability: [4](#0-3) 

The 4-byte opcode path is entered whenever `allocator.atom_len(o) == 4`, which is fully attacker-controlled via the CLVM program bytes. The attacker simply encodes the opcode as the 4-byte atom `\x13\xd6\x1f\x00` (secp256k1) or `\x1c\x3a\x8f\x00` (secp256r1) instead of the 1-byte atom `\x40` or `\x41`. [5](#0-4) 

---

### Impact Explanation

**Flag/operator wiring error.** The `ENABLE_SECP_OPS` flag is the designated access control gate for secp signature verification operators. The 4-byte opcode path completely bypasses this gate. Concretely:

- In **mempool mode** (`MEMPOOL_MODE`, which sets `NO_UNKNOWN_OPS` but not `ENABLE_SECP_OPS`): a program using 1-byte opcode 64 falls through to `unknown_operator` and is rejected with `EvalErr::Unimplemented`. A program using 4-byte opcode `0x13d61f00` executes `op_secp256k1_verify` successfully, consuming 1,300,000 cost units and performing real ECDSA verification. [6](#0-5) 

- In **consensus mode** (without `ENABLE_SECP_OPS`): the 1-byte form is treated as an unknown no-op, while the 4-byte form executes real signature verification. This creates a **consensus divergence** between the two opcode forms for the same logical operation.

- The corrupted invariant is: `ENABLE_SECP_OPS` is supposed to be the single authoritative gate controlling whether secp verification executes. That invariant is broken for the 4-byte opcode form.

---

### Likelihood Explanation

The 4-byte opcode values (`0x13d61f00`, `0x1c3a8f00`) are documented in the codebase itself (in comments and in `tools/src/bin/benchmark-clvm-cost.rs` and `clvm-fuzzing/src/make_tree.rs`), making them discoverable. Any attacker who can submit CLVM programs — the primary attack surface of the Chia VM — can trivially encode the 4-byte form. No special privileges are required; the only requirement is the ability to submit a CLVM program for evaluation, which is the normal use case for the VM. [7](#0-6) 

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch path, mirroring the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures that `ENABLE_SECP_OPS` is the single, consistent gate for both opcode forms.

---

### Proof of Concept

A CLVM program that invokes `op_secp256k1_verify` via the 4-byte opcode form in a context where `ENABLE_SECP_OPS` is not set:

```
; Opcode atom: 4 bytes = 0x13d61f00 (secp256k1_verify, 4-byte form)
; Arguments: (pubkey msg sig) — any valid secp256k1 triple
(0x13d61f00 pubkey msg sig)
```

When evaluated with `ChiaDialect::new(MEMPOOL_MODE)` (which does not include `ENABLE_SECP_OPS`):
- The 1-byte form `(64 pubkey msg sig)` → `EvalErr::Unimplemented` (rejected by `NO_UNKNOWN_OPS`)
- The 4-byte form `(0x13d61f00 pubkey msg sig)` → enters the `op_len == 4` branch at line 157, matches `0x13d61f00`, calls `op_secp256k1_verify` directly, and returns `Reduction(1_300_000, nil)` on a valid signature — **no flag check performed**. [8](#0-7) [9](#0-8)

pub fn op_secp256k1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256K1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256k1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256k1_verify pubkey")?;
    let verifier = K1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256k1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is message
    let msg = atom(a, msg, "secp256k1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256k1_verify sig")?;
    let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
