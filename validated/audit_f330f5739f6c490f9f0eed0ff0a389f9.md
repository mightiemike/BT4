Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Gates on Router Address Instead of End User, Enabling Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the pool's `msg.sender` — the router contract — not the originating user. When a pool admin allowlists `MetricOmmSimpleRouter` so that approved users can perform router-based swaps, any unprivileged user can bypass the curated-pool gate by routing through the public router, because the extension sees the allowlisted router address and passes the check unconditionally.

## Finding Description

**Root cause:** `MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called the pool: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly. The original user's address is stored in transient storage as the payer via `_setNextCallbackContext`, but is never forwarded to the pool's `swap` call as a parameter: [3](#0-2) 

The result: the extension receives `sender = address(router)` and checks `allowedSwapper[pool][router]`. If the router is allowlisted, the check passes for every caller regardless of their identity.

**Contrast with `DepositAllowlistExtension`:** The deposit allowlist correctly ignores `sender` (the intermediary) and gates on `owner` (the economic beneficiary): [4](#0-3) 

No equivalent separation exists for swaps — `beforeSwap` receives no `owner`-equivalent parameter, and the router does not inject the originating user into `extensionData`.

## Impact Explanation

**Allowlist bypass (High):** A pool admin who allowlists `MetricOmmSimpleRouter` so that approved users can perform multi-hop or exact-output swaps inadvertently opens the gate to every user. Any non-allowlisted address calls `router.exactInputSingle()` targeting the curated pool; the extension sees `sender = router` (allowlisted), passes the check, and the unauthorized swap executes against LP funds. This directly violates the core invariant that only approved addresses may trade on a curated pool, constituting a broken core pool functionality causing potential loss of funds to LPs who deployed capital under the assumption of a restricted swapper set.

**Allowlisted users blocked (Medium):** Conversely, if the router is not allowlisted, individually approved users cannot use the router at all on this pool, breaking expected multi-hop and exact-output swap flows.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary public swap entrypoint for the protocol. Pool admins who configure `SwapAllowlistExtension` and also want to support router-based swaps for their approved users will naturally allowlist the router — this is the only mechanism available to them. The bypass is then reachable by any unprivileged user with no special setup, no privileged access, and no non-standard token behavior required.

## Recommendation

The extension must gate on the **original user**, not the intermediary. Two complementary fixes:

1. **Expose the originating user through the router.** The router already stores the original `msg.sender` in transient storage as the payer via `_setNextCallbackContext`. Encode this address into `extensionData` before calling `pool.swap`, and have `SwapAllowlistExtension.beforeSwap` decode and check it when `extensionData` is non-empty and the caller is a known router.

2. **Check `extensionData` for a router-attested user identity** in `SwapAllowlistExtension.beforeSwap`, mirroring how `DepositAllowlistExtension` separates `sender` from `owner`. The swap allowlist should check the attested end-user address rather than the intermediary `sender`. [5](#0-4) 

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured
  - Pool admin calls setAllowedToSwap(pool, address(router), true)
    (to allow approved users to use multi-hop routing)
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  - attacker calls router.exactInputSingle({pool: curatedPool, ...})
  - Router stores msg.sender (attacker) in transient storage, then calls:
      pool.swap(recipient, zeroForOne, amountIn, priceLimit, "", extensionData)
    with msg.sender = router
  - MetricOmmPool.swap calls _beforeSwap(msg.sender=router, ...)
  - SwapAllowlistExtension.beforeSwap receives sender=router
  - Checks allowedSwapper[pool][router] → true
  - Swap executes; attacker receives output tokens from LP funds

Result:
  - Non-allowlisted attacker successfully swaps on a curated pool
  - SwapAllowlistExtension invariant broken: unauthorized user traded
  - LP funds exposed to unrestricted trading against a pool intended to be curated
```

  function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
      );
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
