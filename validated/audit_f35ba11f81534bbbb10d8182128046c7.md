Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, allowing allowlist bypass via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which equals `msg.sender` inside `MetricOmmPool.swap`. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the end user. A pool admin who allowlists the router so that permitted users can reach the pool through it inadvertently opens the pool to every caller, because the router carries no per-user access control and the extension evaluates `allowedSwapper[pool][router]` for all callers.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` directly as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool received as its own `msg.sender`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly, making the router the pool's `msg.sender`. The original user's address is stored in transient storage via `_setNextCallbackContext` only for payment purposes and is never forwarded to the extension: [4](#0-3) 

The result is that the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. There is no configuration that simultaneously permits router-mediated swaps for allowlisted users while blocking non-allowlisted users. If the router is not allowlisted, no user can use the router with this pool. If the router is allowlisted, every user on the network can swap through the router regardless of their individual allowlist status.

## Impact Explanation
This is an admin-boundary break via an unprivileged path. Any user explicitly excluded from a curated pool's swap allowlist can bypass the restriction by routing through `MetricOmmSimpleRouter`. Once the router is allowlisted — a natural and expected admin action to enable UX for permitted users — the allowlist provides zero protection against router-mediated swaps. Unauthorized users can trade in pools designed to be restricted, breaking the admin-set access-control boundary.

## Likelihood Explanation
Pool admins deploying `SwapAllowlistExtension` almost certainly want their permitted users to use the protocol's own router for slippage protection and multi-hop support. Allowlisting the router is the obvious step to enable that. The flaw is silent — the admin receives no warning that doing so opens the pool to all callers — making accidental exploitation highly likely in any production deployment combining the allowlist extension with the router.

## Recommendation
The extension should gate the economically relevant actor — the originating user — rather than the immediate pool caller. Two complementary approaches:

1. **Forward the originating user through `extensionData`.** `MetricOmmSimpleRouter` already stores `msg.sender` in transient storage; it could additionally encode the originating caller inside `extensionData` so the extension can decode and verify it.
2. **Router registry in the extension.** The extension could maintain a set of trusted routers and, when `sender` is a known router, require the original user's address to be encoded in `extensionData` and check that address against the allowlist instead.

Either approach ensures the allowlist gates the user who initiated the trade, not the intermediary contract.

## Proof of Concept
```
Setup
─────
1. Factory deploys Pool with SwapAllowlistExtension as a before-swap hook.
2. Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is KYC'd
3. Pool admin calls setAllowedToSwap(pool, router, true)  // so alice can use the router
   (bob is NOT allowlisted)

Attack
──────
4. bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
5. Router calls pool.swap(recipient, zeroForOne, amount, limit, "", extensionData)
   → inside pool.swap: msg.sender == router
6. Pool calls _beforeSwap(router, recipient, ...)
7. SwapAllowlistExtension.beforeSwap receives sender = router
8. Check: allowedSwapper[pool][router] == true  → passes
9. bob's swap executes successfully in the restricted pool.

Direct call (control)
─────────────────────
10. bob calls pool.swap(...) directly
    → sender = bob → allowedSwapper[pool][bob] == false → NotAllowedToSwap ✓
```
