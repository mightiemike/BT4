Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the actual user, allowing any caller to bypass the swap allowlist via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps on `sender`, which is bound to `msg.sender` of `pool.swap()`. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the original user. A pool admin who allowlists the router to enable router-mediated swaps on a curated pool inadvertently opens the pool to every unpermissioned user, completely defeating the allowlist.

## Finding Description
In `MetricOmmPool.swap`, the pool passes `msg.sender` (its direct caller) as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← direct caller, i.e. the router
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards `sender` unchanged to the extension via `abi.encodeCall`. `SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

where `msg.sender` = pool and `sender` = the router. When `MetricOmmSimpleRouter.exactInputSingle` is called, it calls `pool.swap(params.recipient, ...)` with itself as `msg.sender`, so the extension evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`.

**Exploit path (allowlist bypass):**
1. Pool admin deploys pool with `SwapAllowlistExtension` and calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps.
2. Attacker (not individually allowlisted) calls `router.exactInputSingle({pool, recipient: attacker, ...})`.
3. Router calls `pool.swap(attacker, ...)` with `msg.sender = router`.
4. Pool calls `_beforeSwap(router, attacker, ...)`.
5. Extension checks `allowedSwapper[pool][router] == true` → passes.
6. Attacker executes swap on curated pool without being allowlisted.

**Broken path (secondary impact):**
If the admin allowlists individual users instead of the router, those users cannot swap through the router because the extension sees `sender = router` (not allowlisted) and reverts `NotAllowedToSwap`.

The unit tests in `SwapAllowlistSubExtension.t.sol` only simulate the pool calling the extension directly (`vm.prank(address(pool))`), passing the user address as `sender` — they never simulate the router intermediary, so the misbinding is invisible in the test suite.

## Impact Explanation
This is a direct bypass of the pool's access-control guard with no additional privilege required beyond calling the public router. A pool admin who allowlists the router — the natural and necessary step to enable the canonical periphery swap path — loses the entire allowlist guarantee: every unpermissioned address can execute swaps on the curated pool. This constitutes broken core pool functionality (allowlist enforcement) and an admin-boundary break (the pool admin's intended access restriction is silently nullified). Allowlisted users also cannot use the standard periphery path, making the intended use case non-functional.

## Likelihood Explanation
The router is the canonical, documented swap entry point deployed alongside the pool. Any pool admin who wants to support both direct and router-mediated swaps on a curated pool will allowlist the router. No documentation warns that doing so opens the pool to all users. The misbinding is invisible from the admin's perspective: `isAllowedToSwap(pool, router)` returns `true` as intended, yet the actual enforcement is wrong. The attack requires no special privilege — any address can call the public router.

## Recommendation
The extension must check the original user, not the intermediary. The most targeted fix is to gate on `recipient` instead of `sender` in `SwapAllowlistExtension.beforeSwap`. The router passes `params.recipient` (the actual user) as the `recipient` argument to `pool.swap`, so the extension already receives the correct address:

```diff
- function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
+ function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
  {
-   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
+   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
      revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
```

Alternatively, the router can store the original `msg.sender` in transient storage and the pool can expose it as a separate `originator` argument to `beforeSwap`, but this requires a core interface change.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, router, true)
  - Pool admin does NOT allowlist attacker address

Attack:
  - attacker calls router.exactInputSingle({pool, recipient: attacker, ...})
  - router calls pool.swap(attacker, ...)  [msg.sender = router]
  - pool calls _beforeSwap(sender=router, recipient=attacker, ...)
  - SwapAllowlistExtension.beforeSwap(sender=router, ...)
  - allowedSwapper[pool][router] == true → passes
  - attacker executes swap on curated pool without being individually allowlisted

Foundry test outline:
  1. Deploy SwapAllowlistExtension and a pool configured with it.
  2. Deploy MetricOmmSimpleRouter.
  3. Admin calls setAllowedToSwap(pool, address(router), true).
  4. Call router.exactInputSingle from an attacker address not in the allowlist.
  5. Assert the swap succeeds (allowlist bypassed).
  6. Separately: admin calls setAllowedToSwap(pool, alice, true) without allowlisting router;
     assert alice's router call reverts NotAllowedToSwap (broken path).
```
