Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual swapper, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the `pool.swap()` call. When users route through `MetricOmmSimpleRouter`, the router is `msg.sender` at the pool level, so the allowlist checks the router's address rather than the actual user's address. Any non-allowlisted user can bypass the per-user swap allowlist by routing through the public, permissionless `MetricOmmSimpleRouter`.

## Finding Description

**Root cause — pool passes `msg.sender` as `sender` to extensions:**

In `MetricOmmPool.swap`, the pool calls `_beforeSwap(msg.sender, ...)`: [1](#0-0) 

`_beforeSwap` in `ExtensionCalling` forwards this value as the `sender` argument to every registered extension: [2](#0-1) 

**Router substitutes itself as `msg.sender`:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly — the router is the caller, so `msg.sender` inside `MetricOmmPool.swap` is the router address, not the original user: [3](#0-2) 

The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

**Extension checks the wrong identity:**

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router address for all router-mediated swaps: [4](#0-3) 

This creates two failure modes:
1. **Bypass**: If the pool admin allowlists the router (required for any router-based swap to succeed), every user — including non-allowlisted ones — can swap freely through the router.
2. **Lockout**: If the router is not allowlisted, all router-mediated swaps revert, even for individually allowlisted users.

No existing guard corrects this: the `sender` value is set once at `msg.sender` in `MetricOmmPool.swap` and passed through unchanged. There is no mechanism to recover the original user's address from `extensionData` or any other field.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` is intended to restrict swaps to a specific set of addresses (e.g., KYC'd users, whitelisted participants, or protocol-controlled addresses). Any non-allowlisted user can bypass this restriction entirely by routing through `MetricOmmSimpleRouter`, executing swaps on a pool designed to be access-controlled. This breaks the core allowlist invariant and constitutes an admin-boundary break: the pool admin's access control is rendered ineffective by an unprivileged path, with direct fund-impacting consequences (unauthorized parties can swap on restricted pools).

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing entry point for swaps and is a public, permissionless contract. Any user aware of the router can trivially bypass the allowlist with a standard `exactInputSingle` call. No special privileges, flash loans, or complex setup are required. The bypass is repeatable and deterministic.

## Recommendation

The `SwapAllowlistExtension` must check the actual end-user identity, not the intermediary caller. Two approaches:

1. **Pass the real user via `extensionData`**: The router encodes the original `msg.sender` into `extensionData`; the extension decodes and verifies it, trusting only known routers (verified against the factory registry).
2. **Check `recipient` instead of `sender`**: If the pool's design guarantees `recipient` is always the actual user (not the router), the allowlist can gate on `recipient`. This must be verified against the full call path, including multi-hop `exactInput` where intermediate recipients are the router itself.

The root fix must ensure the identity checked by the allowlist is the economically relevant actor — the user initiating the trade — not the routing intermediary.

## Proof of Concept

```
1. Pool P is deployed with SwapAllowlistExtension E registered in beforeSwap order.
2. Pool admin calls E.setAllowedToSwap(P, router, true)
   (required for any router-based swap to pass the allowlist check).
3. Non-allowlisted user U calls MetricOmmSimpleRouter.exactInputSingle({pool: P, recipient: U, ...}).
4. Router calls P.swap(recipient=U, ...) — router is msg.sender at the pool level.
5. Pool calls _beforeSwap(sender=router, recipient=U, ...).
6. SwapAllowlistExtension.beforeSwap checks allowedSwapper[P][router] → true.
7. Swap proceeds. U successfully swaps on a pool they were never authorized to access.

Alternatively, if the router is NOT allowlisted:
3. Allowlisted user V calls MetricOmmSimpleRouter.exactInputSingle({pool: P, ...}).
4. Router calls P.swap(...) — router is msg.sender.
5. SwapAllowlistExtension.beforeSwap checks allowedSwapper[P][router] → false → revert NotAllowedToSwap().
6. V cannot swap despite being individually allowlisted.
```

A Foundry integration test can confirm both scenarios by deploying a pool with `SwapAllowlistExtension`, allowlisting only the router (or only a specific user), and verifying that non-allowlisted users succeed (bypass) or allowlisted users fail (lockout) when routing through `MetricOmmSimpleRouter`.
