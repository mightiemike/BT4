Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the actual end-user, allowing any unprivileged trader to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension` is documented as gating `swap` by swapper address per pool. However, the `sender` argument it receives is the pool's `msg.sender` — which is the router contract when a user swaps through `MetricOmmSimpleRouter` — not the actual end-user. If the pool admin allowlists the router (required for any router-based swap to succeed), every unprivileged trader can bypass the per-user allowlist by routing through `MetricOmmSimpleRouter`, rendering the access control inoperative.

## Finding Description

**Call chain:**

1. User (`bob`, not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(params)`.
2. The router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — so `msg.sender` seen by the pool is the **router address**.
3. `MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)` where `msg.sender` is the router:

```
// MetricOmmPool.sol line 230-240
_beforeSwap(
    msg.sender,   // ← router address, NOT the end-user
    recipient,
    ...
);
```

4. `ExtensionCalling._beforeSwap()` encodes `sender = router` and calls the extension.
5. `SwapAllowlistExtension.beforeSwap()` receives `sender = router` and checks:

```
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` = pool (correct key), `sender` = **router** (wrong actor). The check resolves to `allowedSwapper[pool][router]`.

**Root cause:** The pool passes its own `msg.sender` (the immediate caller) as `sender` to the extension. When the immediate caller is a router, the extension sees the router address, not the originating user.

**Exploit flow:**
- Pool admin deploys pool with `SwapAllowlistExtension`, allowlists `alice` (KYC'd), and also allowlists `MetricOmmSimpleRouter` so that `alice` can use the router.
- `bob` (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
- The extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds for `bob`.
- The per-user allowlist is fully bypassed.

**Existing guards are insufficient:** `BaseMetricExtension` enforces `onlyPool` on the base `beforeSwap`, but `SwapAllowlistExtension` overrides `beforeSwap` without `onlyPool` and without any mechanism to recover the true originating user. There is no `tx.origin` check or forwarded-user field in the hook calldata.

## Impact Explanation
The `SwapAllowlistExtension` is a production access-control extension. Its bypass allows any unprivileged trader to swap in a pool that the pool admin intended to restrict to specific addresses. This breaks the core allowlist functionality and constitutes an admin-boundary break: an unprivileged path (router) circumvents the pool admin's configured access control. Pools using this extension for compliance or whitelist-gating are rendered fully open to any trader who routes through `MetricOmmSimpleRouter`. Severity: **Medium** (broken core pool access-control functionality; no direct principal loss but complete bypass of the intended restriction gate).

## Likelihood Explanation
Exploitation requires only that the pool admin has allowlisted the router — a natural and expected configuration for any pool that wants to support standard router-based swaps. Any unprivileged trader can then call `exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` on the router. No special privileges, flash loans, or complex setup are needed. The attack is repeatable on every swap.

## Recommendation
The extension must check the true originating user, not the immediate pool caller. Two options:

1. **Pass the originating user through `extensionData`**: The router encodes `msg.sender` (the end-user) into `extensionData` before calling the pool, and the extension decodes and checks it. This requires a convention between router and extension.
2. **Require direct pool interaction**: Document that `SwapAllowlistExtension` only works correctly when users call the pool directly (not through a router), and enforce this by checking `sender == tx.origin` or by rejecting contract callers in `beforeSwap`.
3. **Allowlist routers separately and track originating user via a trusted router interface**: The router exposes the originating user, and the extension queries the router for the real sender before checking the allowlist.

The simplest safe fix is option 2 with an explicit `require(sender == tx.origin || allowedSwapper[msg.sender][sender])` guard, or option 1 with a standardized `extensionData` encoding.

## Proof of Concept

```solidity
// Foundry test sketch
function test_allowlistBypassViaRouter() public {
    // Setup: pool with SwapAllowlistExtension
    // alice is allowlisted, router is allowlisted (required for alice to use router)
    allowlistExt.setAllowedToSwap(pool, alice, true);
    allowlistExt.setAllowedToSwap(pool, address(router), true);

    // bob is NOT allowlisted
    // bob calls the router directly
    vm.prank(bob);
    // This should revert but does NOT — router is allowlisted, so bob passes
    router.exactInputSingle(ExactInputSingleParams({
        pool: pool,
        recipient: bob,
        zeroForOne: true,
        amountIn: 1e18,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        tokenIn: token0,
        extensionData: "",
        deadline: block.timestamp
    }));
    // bob successfully swapped despite not being on the allowlist
}
```
