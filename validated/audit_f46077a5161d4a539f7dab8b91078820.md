### Title
`calculate_next_l2_gas_price_for_fin` Reads `gas_target` from `latest_constants()` Without Version Gating, Causing Wrong EIP-1559 Price Direction at Protocol Upgrades - (File: `crates/apollo_consensus_orchestrator/src/fee_market/mod.rs`)

### Summary

The EIP-1559 gas price adjustment function `calculate_next_l2_gas_price_for_fin` always reads `gas_target` (and `gas_price_max_change_denominator`, `max_block_size`) from `VersionedConstants::latest_constants()` rather than from the version-appropriate constants for the block being evaluated. Because `gas_target` changes significantly between Starknet versions, the price adjustment computed for the transition block uses the new version's target to evaluate the old version's gas usage, potentially reversing the direction of the price change and producing a wrong `next_l2_gas_price` that is committed to state and propagated to the cende blob.

### Finding Description

In `calculate_next_l2_gas_price_for_fin`:

```rust
// crates/apollo_consensus_orchestrator/src/fee_market/mod.rs, line 70

``` [1](#0-0) 

This `gas_target` is passed directly to `calculate_next_base_gas_price`, which also re-reads `latest_constants()` for `max_block_size` and `gas_price_max_change_denominator`:

```rust

The `gas_target` changes substantially across versions:

| Version | `gas_target` | Change |
|---------|-------------|--------|
| V0_14_0 | 3,200,000,000 | — |
| V0_14_1 | 4,000,000,000 | +25% |
| V0_14_2 | 1,500,000,000 | −62.5% |
| V0_14_3 | 1,040,000,000 | −30.7% | [3](#0-2) [4](#0-3) 

The call chain is: `decision_reached` → `update_l2_gas_price` → `calculate_next_l2_gas_price` → `calculate_next_l2_gas_price_for_fin`. None of these functions accept a `StarknetVersion` parameter; they all unconditionally call `latest_constants()`. [5](#0-4) 

The computed `next_l2_gas_price` is then stored in `BlockHeaderWithoutHash` (line 405) and forwarded to the cende blob via `FeeMarketInfo` (line 603–606): [6](#0-5) [7](#0-6) 

### Impact Explanation

When the sequencer binary is upgraded from V0_14_2 to V0_14_3, the first call to `update_l2_gas_price` after the upgrade evaluates the last V0_14_2 block's `l2_gas_used` against the V0_14_3 `gas_target` (1,040,000,000) instead of the V0_14_2 `gas_target` (1,500,000,000).

**Concrete divergence** — suppose the last V0_14_2 block consumed 1,200,000,000 gas:

- **Correct** (V0_14_2 target = 1.5 B): 1.2 B < 1.5 B → price **decreases** by ≈ 0.42 %
- **Actual** (V0_14_3 target = 1.04 B): 1.2 B > 1.04 B → price **increases** by ≈ 0.32 %

The direction of the EIP-1559 adjustment is reversed. The wrong `next_l2_gas_price` is committed to `BlockHeaderWithoutHash` storage and sent to the centralized cende pipeline, producing an incorrect fee state that persists into subsequent blocks (since each block's price is derived from the previous one).

This matches the allowed impact: **Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact (Critical)**.

### Likelihood Explanation

This distortion occurs automatically at every protocol upgrade where `gas_target` changes. The V0_14_1→V0_14_2 transition (−62.5%) and V0_14_2→V0_14_3 transition (−30.7%) both trigger it. No attacker action beyond the upgrade itself is required; the wrong value is computed deterministically.

### Recommendation

Pass the decided block's `StarknetVersion` into `calculate_next_l2_gas_price_for_fin` and select the version-appropriate constants via `VersionedConstants::get(&version)` instead of `latest_constants()`. The `gas_target` used to evaluate block N's gas usage should be the target that was in effect when block N was built, not the target of the version the binary happens to be running at decision time.

```rust
// Proposed fix sketch
pub fn calculate_next_l2_gas_price_for_fin(
    current_l2_gas_price: GasPrice,
    height: BlockNumber,
    l2_gas_used: GasAmount,
    block_version: &StarknetVersion,   // <-- add this
    ...
) -> GasPrice {
    let vc = VersionedConstants::get(block_version)
        .unwrap_or_else(|_| VersionedConstants::latest_constants());
    let gas_target = vc.gas_target;
    ...
}
```

### Proof of Concept

1. Network runs at V0_14_2; `gas_target = 1,500,000,000`.
2. Block N (V0_14_2) is built and uses `l2_gas_used = 1,200,000,000` (below target → price should fall).
3. Sequencer binary is upgraded to V0_14_3; `latest_constants().gas_target = 1,040,000,000`.
4. `decision_reached(height=N, ...)` is called on the upgraded binary.
5. `update_l2_gas_price(N, 1_200_000_000)` → `calculate_next_l2_gas_price_for_fin(...)`.
6. Line 70: `gas_target = VersionedConstants::latest_constants().gas_target = 1_040_000_000`.
7. `calculate_next_base_gas_price`: `gas_used (1.2B) > gas_target (1.04B)` → price **increases**.
8. `self.l2_gas_price` is set to the inflated value; this is written to `BlockHeaderWithoutHash.next_l2_gas_price` and to `FeeMarketInfo.next_l2_gas_price` in the cende blob.
9. All subsequent blocks inherit the wrong base price, compounding the distortion until natural EIP-1559 mean-reversion corrects it over many blocks.
