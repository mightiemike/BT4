### Title
Freed Storage Deposits Are Permanently Locked in the Contract With No Withdrawal Mechanism - (File: `contract/src/lib.rs`)

### Summary

`submit_blocks` requires relayers to attach NEAR deposits proportional to the storage consumed by new block headers. When the garbage-collector (`run_mainchain_gc`) later removes old mainchain headers, the storage bytes are freed and the corresponding NEAR tokens become available in the contract's balance — but no function exists to transfer those tokens out. They are permanently locked in the contract.

### Finding Description

`submit_blocks` measures storage before and after processing, then refunds only the current caller's excess deposit:

```rust
let amount = env::attached_deposit();
let initial_storage = env::storage_usage();

for header in headers { self.submit_block_header(...); }
self.run_mainchain_gc(num_of_headers);          // may DELETE old headers

let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

require!(amount >= required_deposit, ...);

let refund = amount.saturating_sub(required_deposit);
if refund > NearToken::from_near(0) {
    Promise::new(env::predecessor_account_id()).transfer(refund).into()
}
``` [1](#0-0) 

The `saturating_sub` on line 182 means that when GC removes more bytes than the new headers add, `diff_storage_usage` collapses to `0`, `required_deposit` becomes `0`, and the current caller receives a full refund of their own deposit. [2](#0-1) 

However, the NEAR tokens that **previous callers** deposited for the now-deleted headers are not returned to anyone. In NEAR Protocol, deleting storage entries reduces the contract's minimum required balance, making those tokens "available" in the contract account — but they remain there indefinitely because no function in the contract transfers them out.

`run_mainchain_gc` removes mainchain headers from both `mainchain_height_to_header` and `headers_pool` without any corresponding token redistribution: [3](#0-2) 

A second accumulation path exists through fork headers. `store_fork_header` inserts into `headers_pool` but fork headers are only removed during a reorg. Forks that never win the chain are never GC'd, so the storage deposits paid for them are also permanently locked: [4](#0-3) 

Searching the entire `contract/src/` directory confirms there is no `withdraw`, `collect`, or admin-transfer function anywhere in the contract. The only `Promise::transfer` call is the per-call excess refund in `submit_blocks`. [5](#0-4) 

### Impact Explanation

Over the operational lifetime of the contract (recommended `gc_threshold = 52704` blocks, approximately one year of Bitcoin blocks), every GC cycle frees storage that was paid for by earlier relayer calls. At NEAR's storage cost of ~1 NEAR per 10 KB, and with each `ExtendedHeader` occupying on the order of hundreds of bytes, tens of thousands of GC'd headers translate to a material and growing amount of NEAR permanently locked in the contract with no recovery path. The same applies to any fork headers that are submitted but never promoted to the main chain.

### Likelihood Explanation

GC is triggered automatically on every `submit_blocks` call once the mainchain exceeds `gc_threshold`. This is the normal steady-state operation of the contract — not an edge case. Every production deployment will continuously accumulate freed deposits that cannot be retrieved.

### Recommendation

Add an admin-only (e.g., `Role::DAO`) function that transfers the contract's surplus balance (current balance minus the minimum required for current storage) to a designated recipient:

```rust
#[access_control_any(roles(Role::DAO))]
pub fn withdraw_surplus(&mut self, recipient: AccountId) -> Promise {
    let min_balance = env::storage_byte_cost()
        .saturating_mul(env::storage_usage().into());
    let surplus = env::account_balance().saturating_sub(min_balance);
    require!(surplus > NearToken::from_near(0), "No surplus to withdraw");
    Promise::new(recipient).transfer(surplus)
}
```

### Proof of Concept

1. Relayer A calls `submit_blocks` with 100 headers, attaches 50 NEAR (storage deposit). Contract stores headers; 50 NEAR is locked in the contract.
2. Relayer B calls `submit_blocks` with 100 more headers, attaches 50 NEAR. GC fires and removes the 100 headers from step 1. `diff_storage_usage = saturating_sub(...)` = 0 (GC freed ≥ what was added). `required_deposit = 0`. Relayer B receives a full refund of their 50 NEAR.
3. The 50 NEAR deposited by Relayer A for the now-deleted headers remains in the contract balance. No function exists to retrieve it.
4. This repeats every GC cycle. The locked surplus grows monotonically with no bound and no recovery mechanism. [6](#0-5)

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
