### Title
Linked Signer Can Front-Run Its Own Revocation via `allowLinkedSigner = true` on `LinkSigner` Fast-Path — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The fast-path `LinkSigner` transaction in `EndpointTx.sol` passes `allowLinkedSigner = true` to `validateSignedTx`, meaning the **current linked signer** can itself sign a `LinkSigner` transaction to replace the linked signer with a new address. Because `linkedSigners[sender]` is unconditionally overwritten and both the subaccount owner and the current linked signer compete for the same per-subaccount nonce, a compromised linked signer can front-run every revocation attempt by the owner, maintaining persistent unauthorized signing authority over the subaccount.

---

### Finding Description

In the fast-path `processTransactionImpl`, the `LinkSigner` branch is:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:], (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` resolves the valid signer as either the subaccount owner **or** the current `linkedSigners[sender]`:

```solidity
// EndpointTx.sol lines 86–106
function validateSignedTx(..., bool allowLinkedSigner) internal {
    validateNonce(sender, nonce);
    validateSignature(
        sender,
        ...,
        allowLinkedSigner
    );
    requireSubaccount(sender);
}
``` [2](#0-1) 

The nonce is per-subaccount address and is shared between the owner and the linked signer:

```solidity
// EndpointTx.sol lines 72–77
function validateNonce(bytes32 sender, uint64 nonce) internal virtual {
    require(
        nonce == nonces[address(uint160(bytes20(sender)))]++,
        ERR_WRONG_NONCE
    );
}
``` [3](#0-2) 

By contrast, the **slow-mode** `LinkSigner` path uses `validateSender`, which enforces that only the subaccount owner's address can submit the transaction — the linked signer is excluded:

```solidity
// EndpointTx.sol lines 232–239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // owner-only
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The asymmetry is the root cause: the slow-mode path correctly restricts `LinkSigner` to the owner, but the fast path does not.

---

### Impact Explanation

A compromised linked signer (`signerA`) can:

1. Observe the owner's pending `LinkSigner` revocation (nonce `N`, setting signer to `address(0)` or a new address).
2. Submit its own `LinkSigner` transaction (also nonce `N`) signed by `signerA`, pointing to a new attacker-controlled address `signerC`.
3. If the sequencer processes `signerA`'s transaction first, nonce `N` is consumed; the owner's transaction fails with `ERR_WRONG_NONCE`.
4. The owner must resubmit with nonce `N+1`. The attacker repeats step 2–3 indefinitely.

The compromised signer retains the ability to sign `WithdrawCollateral`, `TransferQuote`, `MintNlp`, `BurnNlp`, and other fast-path transactions on behalf of the subaccount, enabling ongoing collateral theft. The `linkedSigners` mapping is the direct state corrupted; the downstream asset delta is the subaccount's collateral balance. [5](#0-4) 

---

### Likelihood Explanation

**Medium.** The attack requires a compromised linked signer (a realistic threat model — linked signers are often hot-wallet API keys). Once compromised, the attacker only needs to submit a competing `LinkSigner` transaction to the sequencer before the owner's revocation is included. The sequencer processes transactions in submission order; a well-timed submission by the attacker is sufficient. No privileged access beyond the already-compromised linked signer key is required.

---

### Recommendation

Remove `allowLinkedSigner = true` from the fast-path `LinkSigner` branch. Only the subaccount owner (the address embedded in the subaccount `bytes32`) should be permitted to change the linked signer, consistent with the slow-mode path's use of `validateSender`. Change:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // ← only owner may change linked signer
);
```

This mirrors the invariant already enforced in the slow-mode path and eliminates the race condition.

---

### Proof of Concept

1. Owner subaccount `S` has linked signer `signerA` (compromised).
2. Owner signs `LinkSigner{sender: S, signer: address(0), nonce: N}` and submits to sequencer.
3. `signerA` signs `LinkSigner{sender: S, signer: signerC, nonce: N}` (valid because `allowLinkedSigner = true`) and submits to sequencer.
4. Sequencer includes `signerA`'s transaction first → `linkedSigners[S] = signerC`, nonce becomes `N+1`.
5. Owner's transaction arrives → `validateNonce` fails (`N != N+1`), reverts silently.
6. `signerC` (attacker-controlled) immediately signs `WithdrawCollateral{sender: S, ...}` and drains collateral.
7. Owner resubmits revocation with nonce `N+1`; attacker repeats from step 3 with nonce `N+1`. [1](#0-0) [3](#0-2)
