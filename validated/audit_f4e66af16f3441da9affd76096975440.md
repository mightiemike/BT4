### Title
Unbounded Bonus Re-Accrual via `ClearPosition` + `TriggerExitFromTier` Loop — (`x/tieredrewards/types/position_state.go`, `x/tieredrewards/keeper/msg_server.go`)

### Summary
`ClearExit` resets `LastBonusAccrual` to the current block time and zeroes `ExitTriggeredAt`/`ExitUnlockAt`. Because `validateTriggerExit` only guards against an already-active exit (`HasTriggeredExit()`), a position owner can immediately re-trigger exit after clearing, earning a full second exit-duration's worth of bonus rewards. This loop is repeatable indefinitely, draining the `RewardsPoolName` module account.

### Finding Description
After a position completes its exit lock duration:

1. `ClaimTierRewards` caps `LastBonusAccrual` at `ExitUnlockAt` (correct).
2. `ClearPosition` calls `claimRewards` (zero incremental bonus), then `ClearExit(blockTime)` which sets `LastBonusAccrual = blockTime` and zeroes both exit timestamps.
3. `TriggerExitFromTier` passes `validateTriggerExit` because `HasTriggeredExit()` is now `false`. It sets a new `ExitUnlockAt = blockTime + tier.ExitDuration`.
4. After waiting another `ExitDuration`, `ClaimTierRewards` accrues bonus from `LastBonusAccrual` (= step-2 blockTime) to the new `ExitUnlockAt` — a full `D`-duration bonus.

The comment in `ClearExit` ("required so that positions who clear exit after exit lock duration won't have extra bonus accrued") only prevents accrual *between* `ExitUnlockAt` and the `ClearPosition` call; it does not prevent a fresh full-duration accrual on the next exit cycle.

### Impact Explanation
Each cycle extracts `shares × tokensPerShare × BonusApy × D / SecondsPerYear` tokens from the rewards pool. With a large position and a long `ExitDuration`, a single attacker can drain the pool in O(pool_balance / per_cycle_bonus) cycles. The delegation remains intact throughout, so there is no capital cost beyond gas.

### Likelihood Explanation
The path requires only standard user-level transactions (`TriggerExitFromTier`, `ClaimTierRewards`, `ClearPosition`) with no governance, operator, or privileged access. Any position owner can execute it.

### Recommendation
In `validateTriggerExit`, reject re-entry if the position has previously completed an exit lock duration without a full withdrawal. One concrete approach: introduce a `HasCompletedExit bool` flag set by `ClearExit` and checked in `validateTriggerExit`. Alternatively, require a full `WithdrawFromTier` / `ExitTierWithDelegation` before a new exit can be triggered.

### Proof of Concept
```
T=0:   TriggerExitFromTier  → ExitTriggeredAt=T0, ExitUnlockAt=T0+D, LastBonusAccrual=T0
T=T0+D+ε: ClaimTierRewards  → bonus=[T0, T0+D], LastBonusAccrual=T0+D
T=T0+D+ε: ClearPosition     → ClearExit sets ExitTriggeredAt=0, ExitUnlockAt=0, LastBonusAccrual=T0+D+ε
T=T0+D+ε: TriggerExitFromTier → ExitTriggeredAt=T0+D+ε, ExitUnlockAt=T0+2D+ε
T=T0+2D+2ε: ClaimTierRewards → bonus=[T0+D+ε, T0+2D+ε] ← second full D-duration bonus
... repeat indefinitely
```

Total bonus paid = N × D-duration accrual instead of 1 × D-duration accrual.

	if !pos.ExitUnlockAt.IsZero() && segmentEnd.After(pos.ExitUnlockAt) {
		segmentEnd = pos.ExitUnlockAt
	}

		return math.ZeroInt()
