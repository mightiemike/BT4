### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Allowing 64-Byte Merkle Proof Forgery to Bypass V2 Security Fix - (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (V1) was deprecated in favour of `verify_transaction_inclusion_v2`, which adds a coinbase Merkle proof check to block the 64-byte transaction Merkle proof forgery attack. However, the V1 method carries only a Rust `#[deprecated]` attribute — a compile-time hint that has no effect on the deployed WASM binary. The method remains `pub` with no access control beyond `#[pause]`, so any unprivileged NEAR caller can invoke it directly, completely bypassing the coinbase proof guard introduced in V2 and obtaining a forged `true` SPV result for a transaction that was never included in a Bitcoin block.

### Finding Description

`verify_transaction_inclusion_v2` was introduced to mitigate the well-known 64-byte transaction Merkle proof forgery vulnerability (https://www.bitmex.com/blog/64-Byte-Transactions). It does so by first verifying a coinbase Merkle proof against the block's `merkle_root`, which constrains the tree depth and prevents an internal node from being passed as a leaf `tx_id`. [1](#0-0) 

The V1 method performs no such check. It accepts a caller-supplied `tx_id` and a caller-supplied `merkle_proof`, computes the Merkle root, and returns `true` if it matches the stored block header's `merkle_root`. [2](#0-1) 

The `#[deprecated]` attribute on V1 is a Rust compiler annotation only. It emits a warning to Rust callers at compile time but has zero effect on the compiled WASM export. The method is still exported under the name `verify_transaction_inclusion` and is callable by any NEAR account. [3](#0-2) 

`ProofArgs` is fully attacker-controlled: `tx_id`, `tx_block_blockhash`, `tx_index`, `merkle_proof`, and `confirmations` are all supplied by the caller with no server-side filtering. [4](#0-3) 

The contract's own developer documentation acknowledges the vulnerability in V1 explicitly:

> "This function is vulnerable to the standard Bitcoin merkle tree second-preimage attack — it may return `true` for an internal node hash rather than a real transaction hash." [5](#0-4) 

### Impact Explanation

An attacker who knows the `merkle_root` of any confirmed Bitcoin block stored in the contract can craft a `(tx_id, merkle_proof)` pair where `tx_id` is an internal 64-byte Merkle tree node. By calling `verify_transaction_inclusion` directly (not V2), the contract computes the same root and returns `true`. Any downstream NEAR contract or bridge protocol that relies on this return value to authorise a cross-chain action (e.g., releasing funds, minting tokens, settling a swap) will be deceived into believing a Bitcoin transaction occurred when it did not. This is a **proof-verification forgery** — the core security invariant of the light client is broken.

### Likelihood Explanation

The attack requires no privileged role, no leaked key, and no social engineering. The method is callable by any NEAR account as a view call (zero gas cost to the attacker). The 64-byte Merkle forgery technique is publicly documented and has known tooling. The only precondition is that the target block hash is present in the contract's `headers_pool` and `mainchain_header_to_height` maps, which is the normal operating state of the contract.

### Recommendation

Remove the `verify_transaction_inclusion` (V1) export from the deployed contract entirely, or gate it behind a privileged role (e.g., `Role::DAO`) so it cannot be called by unprivileged accounts. The safest fix is removal: `verify_transaction_inclusion_v2` already delegates to V1 internally after performing the coinbase check, so no functionality is lost. If backward compatibility is required, the method body should be replaced with a hard `env::panic_str("use verify_transaction_inclusion_v2")`.

### Proof of Concept

1. Identify any confirmed Bitcoin block stored in the contract (e.g., via `get_last_block_header`). Let its hash be `B` and its `merkle_root` be `R`.
2. From the public Bitcoin blockchain, obtain the raw Merkle tree for block `B`. Pick any internal node `N` at depth `d` whose subtree hashes to `R` when combined with a sibling path `P` of length `d`.
3. Call `verify_transaction_inclusion` directly (not V2) with:
   - `tx_id = N` (the internal node, not a real transaction hash)
   - `tx_block_blockhash = B`
   - `tx_index` = the index corresponding to the left-most leaf under `N`
   - `merkle_proof = P`
   - `confirmations = 0`
4. The contract executes `compute_root_from_merkle_proof(N, tx_index, P)`, which reconstructs `R`, and returns `true`.
5. A downstream contract that called `verify_transaction_inclusion` and received `true` now believes a non-existent transaction was confirmed in block `B`. [6](#0-5)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
