### Title
`check_declare_permissions` enforced only on RPC admission path, bypassed via consensus proposal path — (`crates/apollo_gateway/src/gateway.rs`)

### Summary

The gateway enforces `check_declare_permissions` — which checks both the `block_declare` emergency flag and the `authorized_declarer_accounts` allowlist — exclusively when a transaction arrives through the user-facing RPC path (`add_tx_inner`). When declare transactions arrive through the consensus proposal validation path (`validate_proposal` → `handle_proposal_part` → `convert_consensus_tx_to_internal_consensus_tx`), neither check is performed. A malicious proposer can include declare transactions from unauthorized accounts, or include declares while the `block_declare` emergency flag is active, and every validating node will accept and execute them without objection.

### Finding Description

**Path A — RPC admission (guarded):**

In `add_tx_inner`, before any conversion or stateless validation, the gateway calls `check_declare_permissions`:

```rust
if let RpcTransaction::Declare(ref declare_tx) = tx {
    if let Err(e) = self.check_declare_permissions(declare_tx) {
        ...
        return Err(e);
    }
}
```

`check_declare_permissions` enforces two invariants:
1. If `block_declare == true`, all declare transactions are rejected.
2. If `authorized_declarer_accounts` is `Some(list)`, only accounts in `list` may declare.

**Path B — Consensus proposal validation (unguarded):**

In `validate_proposal` → `handle_proposal_part`, incoming `ConsensusTransaction` objects are converted via:

```rust
transaction_converter.convert_consensus_tx_to_internal_consensus_tx(tx)
```

`convert_consensus_tx_to_internal_consensus_tx` delegates to the shared `convert_rpc_tx_to_internal` helper, which performs Sierra compilation, compiled-class-hash validation, and hash calculation — but contains no call to `check_declare_permissions` and has no access to `GatewayConfig`. The `TransactionConverter` struct carries only `class_manager_client`, `proof_manager_client`, and `chain_id`; it has no reference to `block_declare` or `authorized_declarer_accounts`.

The result is that a declare transaction that would be rejected at the gateway is accepted without error when it arrives through consensus, and is forwarded to the batcher for execution.

**Structural analog to the seed report:**

| PaprController | Apollo Sequencer |
|---|---|
| `_addCollateralToVault` checks `isAllowed` | `add_tx_inner` calls `check_declare_permissions` |
| `_increaseDebt` skips `isAllowed` | `handle_proposal_part` skips `check_declare_permissions` |
| Disabled collateral still mints debt | Blocked/unauthorized declares still get declared |

### Impact Explanation

When `authorized_declarer_accounts` is set to a restricted list, the invariant "only these accounts can declare new contracts" is broken: a malicious proposer can declare arbitrary contract classes from any account address, and those class hashes are committed to the global state trie. When `block_declare` is set as an emergency stop (e.g., in response to a discovered vulnerability in the declare mechanism), the emergency measure is ineffective against a malicious proposer.

The concrete wrong output is a new entry in `class_hash_to_compiled_class_hash` state — an unauthorized class declaration committed to Starknet state — matching the impact category: **Critical. Wrong state … class hash … from blockifier/syscall/execution logic for accepted input.**

### Likelihood Explanation

The trigger requires a malicious consensus proposer. Proposer rotation means any validator in the set can eventually propose. The `authorized_declarer_accounts` feature is explicitly deployed in production configuration (the `replacer_gateway_config.json` uses a placeholder for it, and the `gateway_config.json` sets `#is_none: true` by default but the field exists for operator use). When an operator activates the allowlist or the emergency `block_declare` flag, the bypass is immediately exploitable by any validator whose turn it is to propose.

### Recommendation

Move the declare-permission check into the shared `convert_rpc_tx_to_internal` helper inside `TransactionConverter`, or introduce a separate validation step in `handle_proposal_part` that mirrors the gateway's `check_declare_permissions` logic. The `TransactionConverter` should either receive the relevant config fields or expose a separate `validate_declare_permissions` hook that both the gateway and the consensus orchestrator call before forwarding a declare transaction to the batcher.

### Proof of Concept

1. Configure the node with `authorized_declarer_accounts = Some(["0xAUTHORIZED"])` and `block_declare = false`.
2. Craft a `ConsensusTransaction::RpcTransaction(RpcTransaction::Declare(...))` with `sender_address = 0xUNAUTHORIZED`.
3. Inject it into the consensus stream as a proposer (or simulate via `send_proposal_to_validator_context` in the test harness).
4. Observe that `handle_proposal_part` calls `convert_consensus_tx_to_internal_consensus_tx` successfully — no `UnauthorizedDeclare` error is returned.
5. The transaction is forwarded to the batcher via `send_txs_for_proposal`, executed, and the unauthorized class hash is committed to state.

The same steps apply with `block_declare = true`: the gateway rejects the declare via `add_tx`, but the consensus path accepts it without checking the flag.

    fn check_declare_permissions(
        &self,
        declare_tx: &RpcDeclareTransaction,
    ) -> Result<(), StarknetError> {
        // TODO(noamsp): Return same error as in Python gateway.
        if self.config.static_config.block_declare {
            return Err(StarknetError {
                code: StarknetErrorCode::UnknownErrorCode(
                    "StarknetErrorCode.BLOCKED_TRANSACTION_TYPE".to_string(),
                ),
                message: "Transaction type is temporarily blocked.".to_string(),
            });
        }
        let RpcDeclareTransaction::V3(declare_v3_tx) = declare_tx;
        if !self.config.is_authorized_declarer(&declare_v3_tx.sender_address) {
            return Err(StarknetError {
                code: StarknetErrorCode::KnownErrorCode(
                    KnownStarknetErrorCode::UnauthorizedDeclare,
                ),
                message: format!(
                    "Account address {} is not allowed to declare contracts.",
                    &declare_v3_tx.sender_address
                ),
            });
        }
        Ok(())
    }

    async fn convert_consensus_tx_to_internal_consensus_tx(
        &self,
        tx: ConsensusTransaction,
    ) -> TransactionConverterResult<(InternalConsensusTransaction, Option<VerifyAndStoreProofTask>)>
    {
        match tx {
            ConsensusTransaction::RpcTransaction(tx) => {
                let (internal_tx, proof_data) = self.convert_rpc_tx_to_internal(tx).await?;
                let task = proof_data.map(|(proof_facts, proof)| {
                    self.spawn_verify_and_store_proof(proof_facts, proof)
                });
                Ok((InternalConsensusTransaction::RpcTransaction(internal_tx), task))
            }
            ConsensusTransaction::L1Handler(tx) => {
                let internal_tx = self.convert_consensus_l1_handler_to_internal_l1_handler(tx)?;
                Ok((InternalConsensusTransaction::L1Handler(internal_tx), None))
            }
        }
    }
