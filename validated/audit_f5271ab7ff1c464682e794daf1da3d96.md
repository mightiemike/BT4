### Title
`verify_transaction_inclusion` Rejects Valid Single-Transaction Block SPV Proofs Due to Overly Strict Empty-Proof Guard — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (and `verify_transaction_inclusion_v2`, which delegates to it) unconditionally panics when `merkle_proof` is empty. However, an empty Merkle proof is the **correct and only valid proof** for a transaction that is the sole transaction in a Bitcoin block. The guard is therefore an overly strict boundary check that breaks a legitimate, protocol-defined input — a direct structural analog to the 0%-fee DoS in the external report.

---

### Finding Description

In `contract/src/lib.rs`, `verify_transaction_inclusion` contains:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

This fires **before** the Merkle root computation. In Bitcoin, when a block contains exactly one transaction (the coinbase), the Merkle root equals the coinbase `txid` directly, and the Merkle proof path has zero siblings. Passing an empty `merkle_proof` is therefore the only mathematically correct proof for that transaction. The guard rejects it unconditionally.

`verify_transaction_inclusion_v2` — the **non-deprecated, recommended** entry point — also hits this path because it calls `verify_transaction_inclusion` internally after its own coinbase-proof check:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [2](#0-1) 

So both public entry points are broken for single-transaction blocks.

The Merkle computation that follows the guard:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [3](#0-2) 

…would return `tx_id` unchanged when the proof slice is empty, which correctly equals `merkle_root` for a single-transaction block. The guard prevents this correct path from ever executing.

---

### Impact Explanation

Any NEAR caller (downstream contract or user) that submits a valid SPV proof for a transaction in a single-transaction Bitcoin block will always receive a panic (`"Merkle proof is empty"`), even though the proof is cryptographically correct. The contract's core SPV-verification functionality is broken for this valid Bitcoin state. Downstream contracts that rely on `verify_transaction_inclusion_v2` returning `true` for such blocks will be permanently denied service.

---

### Likelihood Explanation

Single-transaction blocks (coinbase-only) occur in Bitcoin mainnet during periods of very low mempool activity and are common in early chain history and test environments. A relayer faithfully submitting such a block header creates a state where no caller can ever prove inclusion of the coinbase transaction, regardless of how correct their proof is. The trigger requires no privilege — any unprivileged NEAR caller supplying a valid `ProofArgs` or `ProofArgsV2` with an empty `merkle_proof` hits the bug.

---

### Recommendation

Remove the blanket empty-proof guard, or replace it with a check that allows the empty-proof case when `tx_index == 0` and the block's `merkle_root` equals `tx_id` directly:

```rust
// Allow empty proof only for single-transaction blocks (merkle root == tx_id)
if args.merkle_proof.is_empty() {
    return args.tx_id == header.block_header.merkle_root;
}
```

This mirrors the fix in the external report (changing `>` to `>=`): relax the strict boundary check to accommodate the valid edge case.

---

### Proof of Concept

1. Relayer submits a Bitcoin block header whose `merkle_root` equals a single coinbase `txid` (a valid single-transaction block).
2. Caller invokes `verify_transaction_inclusion_v2` with:
   - `tx_id` = coinbase txid
   - `tx_index` = 0
   - `merkle_proof` = `[]` (empty — the only correct proof)
   - `coinbase_tx_id` = same txid
   - `coinbase_merkle_proof` = `[]`
3. The length-equality check passes (`0 == 0`). [4](#0-3) 
4. The coinbase Merkle check passes (`compute_root_from_merkle_proof(txid, 0, &[]) == merkle_root`). [5](#0-4) 
5. Execution falls into `verify_transaction_inclusion`, which immediately panics: `"Merkle proof is empty"`. [1](#0-0) 
6. The call reverts. The valid proof is permanently unverifiable through this contract.
