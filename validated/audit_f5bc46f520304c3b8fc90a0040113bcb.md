### Title
`limbs_for_int` Returns Unsigned Magnitude Byte Count While Cost Model Expects Signed Byte Length in `op_multiply` — (`src/more_ops.rs`)

---

### Summary

`op_multiply` tracks the running product's byte size in `l0` to compute per-step multiplication cost. For the **first** operand, `l0` is set via `int_atom`, which returns the **signed** (two's-complement) byte length. For every subsequent step, `l0` is updated via `limbs_for_int(&total)`, which returns the **unsigned magnitude** byte count. These two units diverge whenever the running product has its high bit set (i.e., is a positive integer ≥ 0x80, 0x8000, …), causing the cost charged for subsequent multiplications to be systematically underestimated.

---

### Finding Description

`op_multiply` in `src/more_ops.rs` maintains `l0: usize` as the byte-length of the running product, used in the per-step cost formula:

```rust
cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
```

**First operand** — `l0` is initialised from `int_atom`:

```rust
(total, l0) = int_atom(a, arg, "*")?;
```

`int_atom` is defined as:

```rust
pub fn int_atom(a: &Allocator, args: NodePtr, op_name: &str) -> Result<(Number, usize)> {
    match a.sexp(args) {
        SExp::Atom => Ok((a.number(args), a.atom_len(args))),
        ...
    }
}
```

`a.atom_len(args)` returns the **actual serialised byte length** of the atom — the signed big-endian two's-complement representation used by CLVM. For a positive number whose high bit is set (e.g., 0xfe = 254), this is 2 bytes (`0x00 0xfe`).

**Every subsequent step** — `l0` is updated via:

```rust
l0 = limbs_for_int(&total);
```

where:

```rust
fn limbs_for_int(v: &Number) -> usize {
    v.bits().div_ceil(8) as usize
}
```

`BigInt::bits()` returns the number of bits in the **unsigned magnitude**, ignoring the sign bit. For 0xfe (254), `bits()` = 8, so `limbs_for_int` = 1. But the signed serialisation of 254 is `0x00 0xfe` — **2 bytes** — because the high bit of 0xfe would otherwise be interpreted as negative.

The two "units" diverge for every positive product whose high bit is set:

| Running product | `limbs_for_int` (unsigned) | Actual signed byte length |
|---|---|---|
| 0x7e (126) | 1 | 1 ✓ |
| 0x7f (127) | 1 | 1 ✓ |
| 0x80 (128) | **1** | **2** ✗ |
| 0xfe (254) | **1** | **2** ✗ |
| 0xff (255) | **1** | **2** ✗ |
| 0x7fff (32767) | 2 | 2 ✓ |
| 0x8000 (32768) | **2** | **3** ✗ |

This is the same class of bug as the external report: a value computed in one "space" (unsigned magnitude byte count) is used where a value in a different "space" (signed serialised byte length) is expected, and the two diverge as soon as the product crosses a power-of-two boundary.

---

### Impact Explanation

The cost formula for each multiplication step uses `l0` as the byte-size of the left operand. When `l0` is underestimated by 1, the charged cost is:

- **Linear term**: `6` cost units less than correct (`MUL_LINEAR_COST_PER_BYTE = 6`)
- **Square term**: `l1 / 128` cost units less than correct (`MUL_SQUARE_COST_PER_BYTE_DIVIDER = 128`)

An attacker can craft a chain of multiplications where every intermediate product has its high bit set, accumulating undercharges across all steps. This allows programs to perform more actual computation than the declared cost limit permits — a **cost undercharge** that undermines the DoS-prevention guarantee of the cost model. If any alternative CLVM implementation uses the correct signed byte length, this also constitutes a **consensus divergence** vector: the two implementations will disagree on whether a given program exceeds the cost limit.

---

### Likelihood Explanation

The trigger requires only attacker-controlled CLVM bytes submitted as a normal program. No special permissions, flags, or configuration are needed. The condition (running product with high bit set) is trivially achievable: e.g., `(* 0x7f 0x02 ...)` produces 0xfe after the first step, immediately triggering the mismatch for all subsequent steps. The entry path is the standard `run_program` → `op_multiply` call chain, reachable from any CLVM execution context including mempool validation.

---

### Recommendation

Replace `limbs_for_int` with a function that returns the **signed** byte length — i.e., the number of bytes required to represent the value in CLVM's big-endian two's-complement encoding. Concretely, after each multiplication:

```rust
// Instead of:
l0 = limbs_for_int(&total);

// Use the signed byte length, consistent with int_atom:
l0 = signed_byte_len(&total); // ceil(bits/8) + 1 if high bit of magnitude is set
```

Alternatively, serialise `total` to bytes and take the length, matching what `int_atom` / `a.atom_len` would return for the same value.

---

### Proof of Concept

Consider `(* 0x7f 0x02 0x02)`:

1. **First arg** `0x7f`: `int_atom` → `l0 = 1` (correct: 1 signed byte)
2. **Second arg** `0x02`: `l1 = len_for_value(2) = 1`; cost charged uses `l0=1`; `total = 0xfe`; `l0 = limbs_for_int(0xfe) = 1` ← **should be 2**
3. **Third arg** `0x02`: `l1 = 1`; cost charged uses `l0=1` instead of `l0=2`; undercharge = `(2-1)*6 + (2-1)*1/128 = 6` cost units

The undercharge compounds across longer chains. A program `(* 0x7f 0x02 0x02 0x02 ... 0x02)` with N multiplications undercharges by `6*(N-1)` cost units, allowing proportionally more computation than the cost limit intends.

pub fn len_for_value(val: u32) -> usize {
    if val == 0 {
        0
    } else if val < 0x80 {
        1
    } else if val < 0x8000 {
        2
    } else if val < 0x800000 {
        3
    } else if val < 0x80000000 {
        4
    } else {
        5
    }
}
