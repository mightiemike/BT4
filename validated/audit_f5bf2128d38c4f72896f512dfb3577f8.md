### Title
Missing `ENABLE_SECP_OPS` Flag Check on 4-Byte Secp Opcode Path — (`File: src/chia_dialect.rs`)

### Summary
The `ChiaDialect::op()` function in `src/chia_dialect.rs` dispatches secp256k1/secp256r1 verification through two separate opcode encodings. The 1-byte opcode path (opcodes 64 and 65) correctly gates execution behind `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte opcode path (`0x13d61f00` and `0x1c3a8f00`) dispatches the same secp functions **unconditionally**, with no flag check. An attacker-controlled CLVM program using the 4-byte encoding can invoke secp signature verification regardless of whether `ENABLE_SECP_OPS` is set, bypassing the intended soft-fork activation gate.

### Finding Description

In `src/chia_dialect.rs`, the `op()` method of `ChiaDialect` handles two opcode-length branches:

**4-byte branch (lines 157–182) — no flag check:** [1](#0-0) 

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

**1-byte branch (lines 248–249) — correctly flag-gated:** [2](#0-1) 

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The `ENABLE_SECP_OPS` flag is defined as the soft-fork activation gate for secp operations: [3](#0-2) 

The 4-byte path dispatches `op_secp256k1_verify` and `op_secp256r1_verify` directly without consulting this flag. Any caller — including `run_serialized_chia_program` in `wheel/src/api.rs` — that constructs a `ChiaDialect` without `ENABLE_SECP_OPS` is still exposed to secp execution via the 4-byte encoding. [4](#0-3) 

### Impact Explanation

The `ENABLE_SECP_OPS` flag is the mechanism by which the Chia node controls whether secp operations are active (i.e., whether the corresponding soft-fork has activated). When a node runs without this flag — representing pre-activation consensus state — it must reject secp opcodes. The 1-byte opcodes (64, 65) are correctly rejected. The 4-byte opcodes (`0x13d61f00`, `0x1c3a8f00`) are not rejected; they silently execute full secp signature verification.

This produces a **flag/operator wiring error** with two concrete consequences:

1. **Consensus divergence**: A program using 4-byte secp opcodes executes and returns `nil` (success) on a node without `ENABLE_SECP_OPS`, while the same logical operation expressed via 1-byte opcodes raises `EvalErr::Unimplemented`. Two nodes evaluating the same spend bundle can reach different conclusions about its validity depending on which opcode encoding is used.

2. **Soft-fork gate bypass**: The entire purpose of `ENABLE_SECP_OPS` is to prevent secp execution before the fork activates. The 4-byte path renders this gate ineffective for any attacker who knows the 4-byte encoding.

### Likelihood Explanation

The 4-byte opcode values `0x13d61f00` and `0x1c3a8f00` are documented in the source code comment itself as the secp cost-formula encodings. They are not secret. Any party submitting CLVM to a node can craft a program using these 4-byte atoms as the operator. The entry path is fully attacker-controlled: CLVM bytes → `node_from_bytes` → `run_program` → `ChiaDialect::op()` → 4-byte branch → unconditional secp dispatch. [5](#0-4) 

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode branch, mirroring the 1-byte branch:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures that both opcode encodings are subject to the same activation gate, eliminating the inconsistency.

### Proof of Concept

```rust
// Construct a CLVM program: (0x13d61f00 pubkey msg sig)
// using the 4-byte opcode encoding for secp256k1_verify.
// Run with a ChiaDialect that does NOT have ENABLE_SECP_OPS set.
// The secp operation executes and returns nil (success).
// Run the same program with 1-byte opcode 64 — it returns EvalErr::Unimplemented.
// Same logical operation, different opcode encoding, different result: consensus split.

let flags = ClvmFlags::empty(); // ENABLE_SECP_OPS deliberately absent
let dialect = ChiaDialect::new(flags);
// program using 4-byte opcode 0x13d61f00 → executes secp256k1_verify
// program using 1-byte opcode 64          → EvalErr::Unimplemented
``` [6](#0-5) [1](#0-0)
