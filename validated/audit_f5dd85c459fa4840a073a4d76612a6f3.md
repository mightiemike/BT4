Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Real Swapper, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool populates with its own `msg.sender`. When `MetricOmmSimpleRouter` is the caller, the pool's `msg.sender` is the router contract, not the end user. If a pool admin allowlists the router to enable router-mediated swaps for permitted users, every unpermitted user can bypass the allowlist by calling the router, causing direct loss of LP principal.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` enforces the allowlist against that `sender` parameter, where `msg.sender` inside the extension is the calling pool: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router the pool's `msg.sender`: [3](#0-2) 

The pool therefore receives `msg.sender = router`. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. The same wrong-actor binding applies to `exactInput`, `exactOutputSingle`, `exactOutput`, and the recursive `_exactOutputIterateCallback` path: [4](#0-3) 

No existing guard corrects this: the `extensionData` field passed by the router is `params.extensionData` (user-supplied, line 79), and the extension does not decode it to recover the real caller. There is no transient-slot or other mechanism that propagates the real user identity to the extension.

## Impact Explanation

**Mode 1 — Full allowlist bypass (High severity).** A pool admin who allowlists the router so that their approved users can swap via the periphery inadvertently grants every address on-chain the same permission. Any unpermitted user calls `exactInputSingle` targeting the curated pool; the extension sees `allowedSwapper[pool][router] = true` and passes unconditionally. The disallowed user receives output tokens from LP-owned liquidity, causing direct loss of LP principal. This meets the Sherlock high-severity threshold for direct loss of user funds.

**Mode 2 — Broken router access for permitted users (Medium severity).** If the pool admin does not allowlist the router, `allowedSwapper[pool][router]` is `false` and every router-mediated swap reverts with `NotAllowedToSwap`, even for individually allowlisted users. This breaks core swap functionality for the supported periphery path.

## Likelihood Explanation

`MetricOmmSimpleRouter` is a public, permissionless contract. Any address can call `exactInputSingle` with an arbitrary pool address. No special role, token balance beyond the input amount, or setup is required. The bypass is reachable in a single transaction by any unpermitted user once the router is allowlisted. The precondition (router allowlisted) is the natural and expected admin action to enable router usage for a curated pool.

## Recommendation

The extension must gate the actual end user, not the intermediary router. Two approaches:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` (the real user) into `extensionData` for each hop, and `SwapAllowlistExtension.beforeSwap` decodes and checks that address instead of the `sender` parameter. This requires a trusted encoding convention between router and extension.
2. **Transient-slot real sender**: The router writes the real user's address into a transient storage slot before calling `pool.swap`, and the extension reads that slot. The pool or extension must verify the writer is a trusted router.

The `DepositAllowlistExtension` avoids this problem by gating `owner` (the position owner explicitly passed to `addLiquidity`) rather than `sender` (the adder contract): [5](#0-4) 

`SwapAllowlistExtension` should adopt the same pattern of checking the economically relevant actor.

## Proof of Concept

```
Setup:
  1. Deploy pool with SwapAllowlistExtension (BEFORE_SWAP_ORDER = extension 1).
  2. Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is permitted
  3. Pool admin calls setAllowedToSwap(pool, router, true)  // router allowlisted so alice can use it

Attack (bob is NOT permitted):
  4. bob calls MetricOmmSimpleRouter.exactInputSingle({
         pool: curated_pool,
         tokenIn: token0,
         amountIn: X,
         recipient: bob,
         ...
     })
  5. Router calls pool.swap(bob_recipient, ...) — pool.msg.sender = router
  6. _beforeSwap(router, ...) → extension checks allowedSwapper[pool][router] → true → passes
  7. Swap executes; bob receives output tokens from LP-owned liquidity.

Result: bob, a disallowed user, successfully swaps on a curated pool.
        The allowlist is completely bypassed via the public router.
```

Foundry test plan: deploy `SwapAllowlistExtension`, configure a pool with it, allowlist `alice` and the router, then call `exactInputSingle` from `bob` (not allowlisted) and assert the swap succeeds — demonstrating the bypass.
