### Title
Unbounded `tracked_shards` Vector in `PeerChainInfoV2` Handshake Enables Memory-Exhaustion DoS — (File: chain/network/src/network_protocol/peer.rs)

### Summary

The `tracked_shards` field of `PeerChainInfoV2`, embedded in every network `Handshake` message, is an unbounded `Vec<ShardId>` with no length limit and no duplicate check enforced anywhere in the parsing or handshake-processing path. Any unprivileged peer that can open a TCP connection can send a handshake advertising millions of shard IDs, forcing the receiving node to allocate arbitrary memory before any connection-level rate limiting applies.

### Finding Description

`PeerChainInfoV2` is the network capability advertisement struct carried inside every `Handshake` (Tier1, Tier2, and Tier3). Its `tracked_shards` field is declared as `Vec<ShardId>` with no bound: [1](#0-0) 

The protobuf definition mirrors this with an unconstrained `repeated` field: [2](#0-1) 

The proto-to-Rust conversion allocates the full vector without any length check: [3](#0-2) 

`process_handshake` in `peer_actor.rs` validates `protocol_version`, `genesis_id`, `sender_peer_id`, tier, nonce, and `owned_account` signature, but never inspects `tracked_shards.len()` or checks for duplicates. After all checks pass, the vector is cloned verbatim into the live `Connection` object: [4](#0-3) 

The same vector is then stored in `ConnectedPeerState`: [5](#0-4) 

The codebase already recognises this class of risk for the analogous `SnapshotHostInfo.shards` field, which carries an explicit `MAX_SHARDS_PER_SNAPSHOT_HOST_INFO = 512` limit with the comment "a malicious attack could fill the snapshot host cache with millions of shards": [6](#0-5) 

No equivalent limit exists for `tracked_shards`. Similarly, `BlockHeadersRequest` enforces `MAX_BLOCK_HEADER_HASHES = 20` and bans peers that exceed it: [7](#0-6) 

`tracked_shards` receives no such treatment.

### Impact Explanation

An attacker opens a TCP connection to any NEAR node and sends a `Tier2Handshake` (or Tier1/Tier3) with `sender_chain_info.tracked_shards` containing millions of `u64` entries. The node parses the protobuf (allocating the full vector on the heap), passes all handshake validation checks, and clones the vector into the `Connection` struct — all before any business-logic rate limiting applies. By repeating this with many concurrent connections (up to `max_num_peers`), the attacker exhausts the node's heap, causing an OOM crash or severe performance degradation. This is a network-level DoS requiring no stake, no account, and no prior authentication.

### Likelihood Explanation

NEAR nodes accept inbound TCP connections from the public internet. The handshake is the very first message exchanged, so no prior authentication or stake is required. The attacker only needs the node's IP address and port. The attack is trivially scriptable.

### Recommendation

**Short term:** In `process_handshake` (or in the `TryFrom<&proto::PeerChainInfo> for PeerChainInfoV2` conversion), reject handshakes where `tracked_shards.len()` exceeds a reasonable bound (e.g., `MAX_SHARDS_PER_SNAPSHOT_HOST_INFO = 512`, or the number of shards in the current epoch). Also deduplicate or reject duplicate shard IDs. Treat violations as `ReasonForBan::Abusive`, consistent with the `BlockHeadersRequest` handling.

**Long term:** Centralise all per-field size invariants for `PeerChainInfoV2` in the proto conversion layer so that any future field additions are automatically subject to the same scrutiny.

### Proof of Concept

1. Open a raw TCP connection to a NEAR node's listening port.
2. Construct a `PeerMessage::Tier2Handshake` with a valid `protocol_version`, `genesis_id`, `sender_peer_id`, `target_peer_id`, `partial_edge_info`, and `sender_chain_info.tracked_shards` containing 10,000,000 entries (each a `u64` = 8 bytes, totalling ~80 MB of heap on the receiver).
3. Send the message. The node will parse the protobuf, allocate the vector, pass all handshake checks, and clone the vector into the `Connection` struct — no length check is performed anywhere in the path from `TryFrom<&proto::PeerChainInfo>` through `process_handshake` to `connection::Connection`.
4. Repeat with many concurrent connections. Each connection forces an ~80 MB allocation before the connection is either accepted or rejected by `validate_new_connection`.

The exact divergent value is `handshake.sender_chain_info.tracked_shards.len()` being unbounded, in contrast to the `MAX_SHARDS_PER_SNAPSHOT_HOST_INFO = 512` limit already enforced for the structurally identical `SnapshotHostInfo.shards` field.

/// Limit on the number of shard ids in a single [`SnapshotHostInfo`](state_sync::SnapshotHostInfo) message.
/// The number of shards has to be limited, otherwise a malicious attack could fill the snapshot host cache
/// with millions of shards.
/// The assumption is that no single host is going to track state for more than 512 shards. Keeping state for
/// a shard requires significant resources, so a single peer shouldn't be able to handle too many of them.
/// If this assumption changes in the future, this limit will have to be revisited.
///
/// Warning: adjusting this constant directly will break upgradeability. A new versioned-node would not interop
/// correctly with an old-versioned node; it could send an excessively large message to an old node.
/// If we ever want to change it we will need to introduce separate send and receive limits,
/// increase the receive limit in one release then increase the send limit in the next.
pub const MAX_SHARDS_PER_SNAPSHOT_HOST_INFO: usize = 512;
