Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of end-user, enabling full allowlist bypass for router-mediated swaps — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument forwarded by the pool, which is `msg.sender` of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the router is `msg.sender` of `pool.swap()`, so the extension checks the router address rather than the actual end-user. Any pool admin who allowlists the router to enable router-based swaps for their approved users simultaneously opens the pool to all users, rendering the per-user allowlist completely ineffective for router-mediated swaps.

## Finding Description

**Root cause — allowlist checks router identity, not end-user:**

In `SwapAllowlistExtension.beforeSwap`, the gated identity is `sender`:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol:37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool, enforced by the `onlyPool` modifier in `BaseMetricExtension` which checks `IMetricOmmPoolFactory(FACTORY).isPool(msg.sender)`. `sender` is the first argument forwarded by `ExtensionCalling._beforeSwap`.

**Pool passes its own `msg.sender` as `sender`:**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at line 231, so `sender` = whoever called `pool.swap()`.

**Router always calls `pool.swap()` as itself:**

In `exactInputSingle` (lines 72–80) and `exactInput` (lines 104–112), the router calls `IMetricOmmPoolActions(pool).swap(...)` directly with no mechanism to forward the real caller. For every hop, `msg.sender` of `pool.swap()` is the router contract, not the end-user.

**Identity mismatch table:**

| Entry path | `sender` seen by extension | Allowlist check |
|---|---|---|
| User calls `pool.swap()` directly | User address | Correct |
| User calls `router.exactInputSingle()` | Router address | Wrong — checks router, not user |
| User calls `router.exactInput()` (any hop) | Router address | Wrong — checks router, not user |

**Bypass scenario:**

1. Pool admin deploys a restricted pool with `SwapAllowlistExtension` and adds Alice and Bob to `allowedSwapper[pool]`.
2. Alice and Bob try to swap through the router — blocked because the router is not in the allowlist.
3. Pool admin adds the router to `allowedSwapper[pool]` to fix this.
4. Mallory (not in the allowlist) calls `router.exactInputSingle()` → extension sees `sender` = router → router is allowlisted → Mallory's swap succeeds.

No existing guard prevents this: `extensionData` is passed through but `SwapAllowlistExtension` does not decode it to recover the real user. There is no trusted-router registry or secondary identity check anywhere in the extension.

## Impact Explanation

`SwapAllowlistExtension` is a core access-control guard for restricted pools. Its bypass allows any unprivileged user to swap in a pool explicitly configured to exclude them. LP assets deposited under the assumption that only vetted counterparties would trade against them are exposed to swaps from arbitrary actors. This constitutes broken core pool functionality with direct fund-impact potential: unauthorized swaps extract value from LPs who cannot withdraw mid-swap, and the pool's intended trading model (e.g., KYC-gated, single market-maker, institutional-only) is violated.

## Likelihood Explanation

The bypass requires the pool admin to allowlist the router — a realistic and predictable operational step. Any admin who wants their allowlisted users to use the standard router will add the router to the allowlist, not realizing this opens the pool to all router users. The design offers no safe middle ground: either allowlisted users cannot use the router, or all users can bypass the allowlist through the router. The condition is self-inflicted by normal, expected admin behavior.

## Recommendation

The extension must gate on the actual end-user, not the direct caller of `pool.swap()`. Options:

1. **Trusted-router + `extensionData` forwarding**: Maintain a `trustedRouter` mapping in the extension. When `sender` is a trusted router, decode the real user from `extensionData` (the router must encode `msg.sender` before calling `pool.swap()`). Check the decoded user against `allowedSwapper`.
2. **Reject router-mediated calls unless `allowAllSwappers` is set**: Add a check that reverts if `sender` is a known router and `allowAllSwappers[pool]` is false, forcing admins to explicitly opt into router access only when they intend open access.
3. **Document and enforce direct-call-only**: If router support is out of scope for the allowlist, revert whenever `sender` is not an EOA or is a known router contract.

## Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension, only Alice is allowlisted
extension.setAllowedToSwap(address(pool), alice, true);

// Alice tries to swap through the router — REVERTS (router not allowlisted)
vm.prank(alice);
router.exactInputSingle(ExactInputSingleParams({pool: address(pool), ...}));
// → NotAllowedToSwap

// Admin adds router to fix Alice's access
vm.prank(admin);
extension.setAllowedToSwap(address(pool), address(router), true);

// Alice can now swap through router ✓
vm.prank(alice);
router.exactInputSingle(...); // succeeds

// Mallory (NOT allowlisted) also swaps through router — SUCCEEDS (bypass)
vm.prank(mallory);
router.exactInputSingle(ExactInputSingleParams({pool: address(pool), ...}));
// → succeeds: extension sees sender=router, router is allowlisted
```
