### Title
Deprecated `verify_transaction_inclusion` Remains a Live Public Entry-Point and Enables Proof-Verification Forgery — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is still exported as a callable NEAR method. Its only cryptographic guard is that the caller-supplied Merkle proof hashes up to the stored `merkle_root`; it never checks that `tx_id` is a leaf-level transaction rather than an internal Merkle tree node. Any unprivileged NEAR account can supply an internal node as `tx_id` with a shortened proof and receive `true`, forging a transaction-inclusion result that any consumer contract relying on this entry-point will treat as genuine.

---

### Finding Description

`verify_transaction_inclusion` carries `#[deprecated]` and `#[pause]` attributes. [1](#0-0) 

`#[deprecated]` is a Rust compiler hint only; it does not prevent the method from being called on-chain. `#[pause]` only blocks the call when the contract is administratively paused — it imposes no caller-identity restriction when the contract is live. The sole cryptographic check is:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

`compute_root_from_merkle_proof` iterates over the supplied proof elements without validating the proof length against the tree depth:

```rust
for proof_hash in merkle_proof {
    if current_position % 2 == 0 {
        current_hash = compute_hash(&current_hash, proof_hash);
    } else {
        current_hash = compute_hash(proof_hash, &current_hash);
    }
    current_position /= 2;
}
``` [3](#0-2) 

An internal node at depth `d` from the root can be "proven" with exactly `d` attacker-chosen proof elements. The function accepts any `tx_id` and any `merkle_proof` of length ≥ 1 (the only length guard is `!args.merkle_proof.is_empty()`). [4](#0-3) 

The code itself acknowledges the broken invariant:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [5](#0-4) 

The secure replacement `verify_transaction_inclusion_v2` enforces equal proof lengths and validates a coinbase Merkle proof before delegating to the deprecated path, closing the forgery window. [6](#0-5) 

However, `verify_transaction_inclusion` was never removed — it remains a live, unrestricted public method.

---

### Impact Explanation

Any consumer contract that calls `verify_transaction_inclusion` to gate a privileged action — releasing bridged assets, crediting a cross-chain deposit, unlocking collateral — will receive `true` for a fabricated transaction. The corrupted value is the **proof result**: the function returns `true` for an `(internal_node, shortened_proof)` pair that was never a real Bitcoin transaction. The attacker does not need to mine a Bitcoin block, control any privileged NEAR account, or possess any tokens; they only need knowledge of a real block's Merkle tree structure, which is publicly available from any Bitcoin full node or block explorer.

---

### Likelihood Explanation

The function is exported as a standard NEAR view/call method. Integrations written before `verify_transaction_inclusion_v2` was introduced (before v0.5.0) may still call the deprecated path. The attack requires only a single cross-contract call with crafted Borsh-encoded arguments, executable by any NEAR account with no stake or role requirement.

---

### Recommendation

Remove `verify_transaction_inclusion` from the contract's public API entirely. All consumer contracts must migrate to `verify_transaction_inclusion_v2`. Keeping a deprecated but live method that the codebase itself documents as forgeable is the direct analog of leaving the burn instruction in the withdrawal path: the minimal check (`merkle_proof.is_empty()`) is insufficient to enforce the invariant that `tx_id` must be a real leaf-level transaction.

---

### Proof of Concept

Given a real Bitcoin block B stored in the contract whose Merkle tree has depth ≥ 2 (i.e., ≥ 3 transactions):

1. Obtain the block's full transaction list from any public Bitcoin node.
2. Compute the left subtree root `N = SHA256d(SHA256d(tx0 || tx1))` — this is an internal node at depth 1.
3. Compute its sibling `S = SHA256d(SHA256d(tx2 || tx3))`.
4. Verify: `SHA256d(N || S) == merkle_root(B)`.
5. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (an internal node, not a real transaction)
   - `tx_block_blockhash = B`
   - `tx_index = 0`
   - `merkle_proof = [S]` (length 1, satisfies `!is_empty()`)
   - `confirmations = 1`
6. `compute_root_from_merkle_proof(N, 0, [S])` returns `SHA256d(N || S) == merkle_root(B)` → function returns `true`.
7. The consumer contract accepts this as proof that transaction `N` was included in block `B`, even though `N` is not a transaction.

        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,

            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
