### Title
`ContractOwner` Cannot Receive ETH, Permanently Locking Native Tokens in `DirectDepositV1` - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner` has no `receive()` or `payable fallback()` function. When `withdrawFromDirectDepositV1` is called with `token == address(0)` to rescue native ETH from a `DirectDepositV1` (DDA) contract, `DirectDepositV1.withdrawNative()` attempts to push ETH to `ContractOwner` via a low-level call. Because `ContractOwner` cannot accept ETH, the transfer reverts, the rescue fails, and any native ETH stranded in the DDA is permanently locked.

---

### Finding Description

`ContractOwner.withdrawFromDirectDepositV1` is the designated rescue path for native ETH stuck in a DDA:

```solidity
// ContractOwner.sol L622-L647
function withdrawFromDirectDepositV1(bytes32 subaccount, address token)
    external
    onlyOwner
{
    address payable directDepositV1 = directDepositV1Address[subaccount];
    require(directDepositV1 != address(0), "no dda");
    if (token == address(0)) {
        uint256 preBalance = address(this).balance;
        DirectDepositV1(directDepositV1).withdrawNative();   // <-- sends ETH to address(this)
        uint256 postBalance = address(this).balance;
        require(postBalance > preBalance, "empty");          // <-- always fails
        (bool success, ) = msg.sender.call{value: postBalance - preBalance}("");
        require(success, "xfer");
    }
```

`DirectDepositV1.withdrawNative()` pushes ETH to `msg.sender` — which is `ContractOwner`:

```solidity
// DirectDepositV1.sol L108-L112
function withdrawNative() external onlyOwner {
    uint256 balance = address(this).balance;
    (bool success, ) = msg.sender.call{value: balance}("");
    require(success, "Failed to transfer native token to owner");
}
```

`ContractOwner` declares no `receive()` or `fallback()` function. Its full inheritance chain (`EIP712Upgradeable`, `OwnableUpgradeable`) also defines none. The low-level `.call{value: balance}("")` to `ContractOwner` therefore reverts unconditionally, propagating the revert up through `withdrawNative()` and back into `withdrawFromDirectDepositV1`. The `require(postBalance > preBalance, "empty")` line is never reached; the entire call reverts at the inner `require(success, "Failed to transfer native token to owner")`.

The root cause is structurally identical to the reported proxy bug: a contract that is supposed to forward or receive ETH is missing the `payable` surface (`receive()` / `payable fallback()`), making the ETH-handling path permanently inaccessible.

---

### Impact Explanation

Any native ETH that accumulates in a `DirectDepositV1` contract — whether from a failed wrap in the constructor, a failed `receive()` wrap, or any future ETH sent directly — cannot be recovered. The only designated recovery function (`withdrawFromDirectDepositV1` with `token == address(0)`) always reverts. The ETH is permanently locked in the DDA with no alternative extraction path.

**Corrupted asset delta**: native ETH balance of the `DirectDepositV1` contract is irrecoverably stranded.

---

### Likelihood Explanation

The DDA's `receive()` function wraps incoming ETH by calling the wrapped-native contract. If that call fails (e.g., the wrapped-native contract is paused, upgraded, or otherwise broken), the `require(success, "Failed to wrap native token.")` reverts and ETH is not deposited — but ETH can still accumulate in the DDA via the constructor path (which emits `NativeTokenTransferFailed` and continues rather than reverting). Any ETH that lands in the DDA through that silent-failure path is immediately unrecoverable. The scenario is realistic on any chain where the wrapped-native contract can transiently reject deposits.

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept ETH forwarded from `DirectDepositV1.withdrawNative()`:

```solidity
receive() external payable {}
```

Alternatively, restructure `withdrawFromDirectDepositV1` to avoid routing ETH through `ContractOwner` by having `DirectDepositV1` send ETH directly to the final recipient rather than to the intermediary.

---

### Proof of Concept

1. A `DirectDepositV1` DDA is deployed for `subaccount` via `ContractOwner.createDirectDepositV1`.
2. ETH is sent to the DDA during construction with a non-zero `address(this).balance`; the wrapped-native call fails silently (`NativeTokenTransferFailed` is emitted), leaving ETH in the DDA.
3. The owner calls `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))`.
4. Internally, `DirectDepositV1.withdrawNative()` executes `msg.sender.call{value: balance}("")` where `msg.sender == address(ContractOwner)`.
5. `ContractOwner` has no `receive()` — the call returns `success = false`.
6. `require(success, "Failed to transfer native token to owner")` reverts.
7. The entire transaction reverts; ETH remains locked in the DDA forever. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
