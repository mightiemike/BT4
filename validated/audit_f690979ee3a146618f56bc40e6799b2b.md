### Title
Bonus Rewards Permanently Lost When `ErrInsufficientBonusPool` Is Swallowed in `slashRedelegationPosition` After Partial State Mutation — (`x/tieredrewards/keeper/claim_rewards.go`, `slash.go`)

---

### Summary

`processEventsAndClaimBonus` mutates both the store (event ref counts) and the in-memory `pos` struct before it reaches the `sufficientBonusPoolBalance` guard. When that guard returns `ErrInsufficientBonusPool`, the caller `slashRedelegationPosition` deliberately swallows the error and then persists the already-mutated `pos` via `setPositionWithState`. This leaves the chain in a state where event ref counts are permanently decremented (events deleted when count reaches zero) and the position's `LastEventSeq` is advanced past those events — but no bonus was ever paid to the owner.

---

### Finding Description

**Step 1 — Store writes happen inside the loop, before the pool check.**

In `processEventsAndClaimBonus`, for every event entry the loop:
- calls `pos.UpdateLastEventSeq(entry.Seq)` — mutates the in-memory `pos` pointer [1](#0-0) 
- calls `k.decrementEventRefCount(ctx, valAddr, entry.Seq)` — writes to the store immediately [2](#0-1) 

`decrementEventRefCount` deletes the event from the store when `ReferenceCount <= 1`: [3](#0-2) 

**Step 2 — Checkpoint fields are also updated before the pool check.**

After the loop, `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` update the in-memory `pos` at lines 215–217, still before `sufficientBonusPoolBalance` is called at line 230: [4](#0-3) 

**Step 3 — `sufficientBonusPoolBalance` failure returns an error without rolling back.**

If the pool is drained, the function returns `ErrInsufficientBonusPool` at line 231 — after all store writes and all in-memory mutations have already occurred. No rollback of ref counts or `LastEventSeq` is attempted. [5](#0-4) 

**Step 4 — `slashRedelegationPosition` swallows the error and persists the mutated position.**

`slashRedelegationPosition` passes `&pos` to `processEventsAndClaimBonus`, so all in-memory mutations are visible after the call returns. It then explicitly swallows `ErrInsufficientBonusPool` and falls through to `setPositionWithState`, which persists the advanced `LastEventSeq`, `LastBonusAccrual`, and `LastKnownBonded`: [6](#0-5) 

---

### Impact Explanation

After this sequence:

- The position's `LastEventSeq` is persisted past the events that were consumed — the position will never re-process them.
- The event ref counts for those events are permanently decremented in the store. If this position held the last reference, the events are deleted.
- No bonus coins were transferred to the owner.

**Concrete impact:** The position owner suffers a permanent, unrecoverable loss of all bonus rewards that accrued up to the slash point. If a second position shared the same validator events, those events' ref counts are also decremented by one prematurely; if the second position later processes them, the count reaches zero and the events are deleted correctly for that position — but the first position's share of the reward is gone forever.

---

### Likelihood Explanation

The trigger condition is:
1. A redelegation slash fires (a normal staking event, not privileged).
2. The `RewardsPoolName` module account balance is below the computed bonus for the slashed position at that moment.

Pool exhaustion is a realistic operational state — the pool is funded by governance deposits and can be drained by prior claims. No special privileges are required; a normal validator slash on a redelegation is sufficient.

---

### Recommendation

Move all store-mutating side effects (ref count decrements and `pos` checkpoint updates) to occur **only after** `sufficientBonusPoolBalance` succeeds and `SendCoinsFromModuleToAccount` completes. One approach:

1. Collect the list of `(valAddr, seq)` pairs to decrement during the loop without writing to the store.
2. Compute `totalBonus` and all checkpoint values into local variables.
3. Only after the pool check and bank send succeed, apply the ref count decrements and write the checkpoint fields back to `pos`.

Alternatively, if the intent is to skip bonus payment when the pool is insufficient, the function must also **not** advance `LastEventSeq` or decrement ref counts in that code path — the position must be left in its pre-call state so the events remain claimable once the pool is refilled.

---

### Proof of Concept

```
Setup:
  - Validator V, two positions P1 and P2 delegating to V.
  - Each event for V has ReferenceCount = 2.
  - Drain the RewardsPool to below P1's accrued bonus but above P2's.

Trigger:
  - Cause a redelegation slash on P1's unbonding entry.
  - slashRedelegationPosition fires → processEventsAndClaimBonus(&pos_P1):
      * Loop: decrements ref counts for all events (store write).
      * Loop: advances pos_P1.LastEventSeq (in-memory).
      * applyBonusAccrualCheckpoint / UpdateLastKnownBonded (in-memory).
      * sufficientBonusPoolBalance → ErrInsufficientBonusPool (no payment).
  - slashRedelegationPosition swallows the error.
  - setPositionWithState persists pos_P1 with advanced LastEventSeq.

Assert:
  - P1 can no longer claim any bonus for those events (LastEventSeq advanced past them).
  - Event ref counts are now 1 (P1's share consumed without payment).
  - P2 can still claim (ref count 1 → 0 when P2 processes), but P1's bonus is gone.
  - Refilling the pool does not help P1 — the events are already consumed from P1's perspective.
```
