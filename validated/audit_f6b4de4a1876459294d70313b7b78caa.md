### Title
Zero-Confirmation SPV Proof Acceptance Enables Reorg-Based Front-Running - (File: `contract/src/lib.rs`)

### Summary
Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations: u64` with no enforced minimum. Any unprivileged NEAR caller may pass `confirmations: 0`, causing the contract to attest that a transaction is canonically included in a block that has zero subsequent confirmations. A concurrent or immediately following `submit_blocks` call that triggers a chain reorganization removes that block from the canonical chain, silently invalidating the `true` attestation already returned to the consumer.

### Finding Description
`ProofArgs.confirmations` and `ProofArgsV2.confirmations` are `u64` fields with no lower-bound check in the contract. [1](#0-0) 

The confirmation guard inside `verify_transaction_inclusion` is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [2](#0-1) 

When `confirmations = 0` the right-hand side is `0`, so the inequality `N >= 0` is trivially satisfied for any `u64 N`. The function then proceeds to verify the Merkle proof and returns `true` for a transaction sitting at the current chain tip with no subsequent blocks.

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after the coinbase proof check, inheriting the same zero-confirmation path. [3](#0-2) 

The canonical chain state that backs the attestation — `mainchain_header_to_height` and `mainchain_tip_blockhash` — is mutable and is updated atomically during `submit_block_header_inner` whenever a fork accumulates more chainwork. [4](#0-3) 

After a reorg, `reorg_chain` removes the previously canonical block from `mainchain_height_to_header` and `mainchain_header_to_height`. [5](#0-4) 

The `true` result already returned to the consumer is now stale: the block it referenced is no longer in the canonical chain, yet no on-chain mechanism invalidates or revokes the prior attestation.

### Impact Explanation
A downstream NEAR contract that calls `verify_transaction_inclusion` (or v2) with `confirmations: 0` and acts on the `true` result — for example, releasing bridged funds, minting tokens, or updating state — can be front-run. An attacker who controls a competing fork (pre-mined with more cumulative chainwork) submits it via `submit_blocks` immediately after the victim's verification call. The reorg promotes the fork, evicting the verified block from the canonical chain. The action taken by the downstream contract is now based on a verification result that no longer reflects the canonical state. The corrupted invariant is the `mainchain_header_to_height` mapping: the block hash that `verify_transaction_inclusion` confirmed as canonical is no longer present in that map after the reorg.

### Likelihood Explanation
The attack requires two conditions: (1) the victim calls `verify_transaction_inclusion` with `confirmations: 0`, which is explicitly supported and demonstrated in the test suite; (2) a trusted relayer submits a competing fork with more chainwork before the downstream consumer finalizes its action. On Bitcoin mainnet this is computationally expensive, but on testnet (`pow_allow_min_difficulty_blocks = true`) minimum-difficulty blocks are permitted when block timestamps are spaced more than `2 * pow_target_spacing` apart, making fork construction cheap. [6](#0-5) 

The test suite itself passes `confirmations: 0` in multiple integration tests, confirming this is an expected and reachable call pattern. [7](#0-6) 

### Recommendation
Enforce a protocol-level minimum confirmation count inside `verify_transaction_inclusion`. Define a constant (e.g., `MIN_CONFIRMATIONS: u64 = 6`) and add a `require!` guard before the existing confirmation check:

```rust
const MIN_CONFIRMATIONS: u64 = 6;
require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    format!("Confirmations must be at least {MIN_CONFIRMATIONS}")
);
```

Alternatively, expose a `get_min_confirmations()` view function so downstream consumers can query the enforced minimum, mirroring the `getMinIntervalBetweenUpdates` pattern. Document clearly that passing `confirmations: 0` is unsafe and should never be used in production consumers that take irreversible on-chain actions.

### Proof of Concept
1. Deploy the contract on testnet (Bitcoin testnet config, `pow_allow_min_difficulty_blocks = true`).
2. Trusted relayer A submits the canonical chain up to height H; block B is at the tip.
3. Victim contract calls `verify_transaction_inclusion` with `tx_block_blockhash = B.hash`, `confirmations = 0` → returns `true`; victim contract releases bridged funds.
4. Attacker (trusted relayer B) has pre-built a fork starting at height H-1 with minimum-difficulty blocks (cheap on testnet), accumulating more chainwork than the canonical tip.
5. Attacker calls `submit_blocks` with the fork headers → `submit_block_header_inner` detects `current_header.chain_work > total_main_chain_chainwork` and calls `reorg_chain`, removing block B from `mainchain_header_to_height`.
6. Block B is no longer canonical. The `true` attestation used to release funds in step 3 is now invalid, but the funds are already gone. [8](#0-7) [9](#0-8)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    fn submit_block_header_inner(
        &mut self,
        current_header: ExtendedHeader,
        prev_block_header: &ExtendedHeader,
    ) {
        // Main chain submission
        if prev_block_header.block_hash == self.mainchain_tip_blockhash {
            // Probably we should check if it is not in a mainchain?
            // chainwork > highScore
            log!("Block {}: saving to mainchain", current_header.block_hash);
            // Validate chain
            assert_eq!(
                self.mainchain_tip_blockhash,
                current_header.block_header.prev_block_hash
            );

            self.store_block_header(&current_header);
            self.mainchain_tip_blockhash = current_header.block_hash;
        } else {
            log!("Block {}: saving to fork", current_header.block_hash);
            // Fork submission
            let main_chain_tip_header = self
                .headers_pool
                .get(&self.mainchain_tip_blockhash)
                .unwrap_or_else(|| env::panic_str("tip should be in a header pool"));

            let last_main_chain_block_height = main_chain_tip_header.block_height;
            let total_main_chain_chainwork = main_chain_tip_header.chain_work;

            self.store_fork_header(&current_header);

            // Current chainwork is higher than on a current mainchain, let's promote the fork
            if current_header.chain_work > total_main_chain_chainwork {
                log!("Chain reorg");
                self.reorg_chain(current_header, last_main_chain_block_height);
            }
        }
    }
