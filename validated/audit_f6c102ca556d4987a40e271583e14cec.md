### Title
`op_modpow` Cost Model Undercharges When Exponent and Modulus Are Both Large — (File: `src/more_ops.rs`)

---

### Summary

The `op_modpow` operator charges cost as the **sum** of two independent quadratic terms (`esize² + msize²`), but the actual computational cost of modular exponentiation is proportional to the **product** `esize × msize²`. When both the exponent and modulus are near the 256-byte limit, the actual CPU work is approximately **8× higher** than the charged cost, allowing an attacker to submit CLVM programs that consume far more node CPU time than the cost budget implies.

---

### Finding Description

In `src/more_ops.rs`, `op_modpow` computes cost as:

```rust
cost += (esize * esize) as Cost * MODPOW_COST_PER_BYTE_EXPONENT;  // esize² × 3
// ...
cost += (msize * msize) as Cost * MODPOW_COST_PER_BYTE_MOD;       // msize² × 21
``` [1](#0-0) 

The cost model is the **sum** of two independent quadratic terms. However, the actual computational complexity of modular exponentiation (square-and-multiply) is **O(esize × msize²)**: for each bit of the exponent (~`esize × 8` bits), one modular multiplication of `msize`-byte numbers is performed, each costing O(msize²).

The benchmark methodology in `tools/src/bin/benchmark-clvm-cost.rs` measures each dimension independently, with the other two fixed at a 32-byte `number`:

- `"modpow (modulus cost)"`: fixed base=32B, exponent=32B, varying modulus
- `"modpow (exponent cost)"`: fixed base=32B, modulus=32B, varying exponent
- `"modpow (value cost)"`: fixed exponent=32B, modulus=32B, varying base [2](#0-1) 

This calibrates `MODPOW_COST_PER_BYTE_MOD = 21` against actual timing with `esize = 32`. The constant correctly models the time for `esize=32, msize=256`. But because actual cost is `esize × msize²`, when `esize` grows from 32 to 256 (8×), the actual time grows by 8× while the charged cost (`msize² × 21`) stays the same. The cross-term `esize × msize²` is never captured. [3](#0-2)

### Impact Explanation

An attacker crafts a CLVM program `(modpow base exponent modulus)` with 256-byte exponent and 256-byte modulus (both at the enforced limit): [5](#0-4) 

- **Charged cost**: `17000 + 256×38 + 256²×3 + 256²×21 ≈ 1,599,592` units
- **Actual work**: ~2048 modular multiplications of 256-byte numbers, each O(256²) — approximately 8× more CPU time than the model predicts (based on the calibration at `esize=32`)
- With an 11-billion cost limit, ~6,875 such calls fit in one block; actual node CPU time is ~8× the budget, enabling a sustained DoS against block validation

**Impact: Medium** — no consensus failure or fund loss, but a reliable CPU exhaustion vector against full nodes.

---

### Likelihood Explanation

The attacker only needs to submit a valid transaction containing a CLVM program with 256-byte exponent and 256-byte modulus. No special privileges, compromised keys, or social engineering are required. The 256-byte input limit is enforced, so the undercharge is bounded but still significant (~8×).

**Likelihood: Medium** — straightforward to craft; requires paying a transaction fee, but the amplification factor makes it economically viable for a motivated attacker.

---

### Recommendation

Replace the separable cost model with one that captures the dominant cross-term. The actual cost of modpow is `O(esize × msize²)`, so the cost formula should include a term like:

```rust
cost += (esize as Cost) * (msize as Cost) * (msize as Cost) * MODPOW_COST_PER_BYTE_CROSS;
```

Alternatively, re-benchmark `modpow` with both `esize` and `msize` simultaneously at their maximum values (256 bytes each) and adjust the constants so the model matches the actual worst-case timing.

---

### Proof of Concept

1. Construct a CLVM atom `E` of 256 bytes (positive, so leading byte `0x00`) and a 256-byte odd modulus `M`.
2. Submit the CLVM program `(modpow (q . base) (q . E) (q . M))`.
3. The charged cost is `≈ 1,599,592` units — well within any reasonable per-call budget.
4. The actual computation performs `256 × 8 = 2048` modular multiplications of 256-byte numbers.
5. The benchmark calibrated `MODPOW_COST_PER_BYTE_MOD = 21` at `esize = 32`; at `esize = 256` the same modulus cost is charged but actual time is 8× higher.
6. Repeating this call ~6,875 times within the 11B cost limit causes the node to spend ~8× the expected CPU budget on a single block.
