### Title
Unchecked u64 Multiplication Overflow in `op_unknown` Cost Computation Produces Silently Incorrect Cost - (File: src/more_ops.rs)

### Summary

`op_unknown` in `src/more_ops.rs` computes a final cost by multiplying a base cost (bounded by `max_cost`) by `cost_multiplier + 1` using plain Rust integer arithmetic. In release mode, Rust wraps on overflow rather than panicking. The post-multiplication guard `if cost > u32::MAX as u64` is applied to the already-wrapped value, not the true product, so when the product wraps to a value ≤ `u32::MAX` the function silently returns a drastically undercharged (or zero) cost instead of an error.

### Finding Description

In `src/more_ops.rs`, `op_unknown` encodes the cost of an unknown opcode as:

```
final_cost = base_cost × (cost_multiplier + 1)
```

`cost_multiplier` is extracted from the opcode bytes via `u32_from_u8`, so it is at most `u32::MAX = 4 294 967 295`, making `cost_multiplier + 1` at most `2^32 = 4 294 967 296`. `base_cost` is bounded by `max_cost` (Chia consensus uses `11 000 000 000`).

The critical sequence is:

```rust
check_cost(cost, max_cost)?;          // line 260 – ensures cost ≤ max_cost
cost *= cost_multiplier + 1;          // line 261 – UNCHECKED u64 multiply
if cost > u32::MAX as u64 {           // line 262 – guard applied to wrapped value
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`Cost` is `u64` (`src/cost.rs` line 3). When `cost > 2^32` and `cost_multiplier = u32::MAX`, the product `cost × 2^32` can exceed `u64::MAX`. In Rust release mode this wraps modulo `2^64`. The wrapped value is `(cost mod 2^32) × 2^32`. If `cost` is a multiple of `2^32`, the wrapped value is exactly `0`, which passes the `> u32::MAX` guard and causes the function to return `Ok(Reduction(0, nil))` — a zero-cost unknown op.

**Concrete trigger values** (with `max_cost = 11 000 000 000`):

| Target `cost` | Multiple of 2^32? | Within `max_cost`? |
|---|---|---|
| `4 294 967 296` (= 2^32) | Yes | Yes |
| `8 589 934 592` (= 2×2^32) | Yes | Yes |

For cost function 1 (ARITH-like, `cost = 99 + n×320 + total_bytes×3`), reaching `cost = 4 294 967 296` requires `n = 13 421 771` arguments with `total_bytes = 159`. The interpreter's `STACK_SIZE_LIMIT` is `20 000 000`, so this argument count is within the allowed range.

The opcode byte layout that sets `cost_multiplier = u32::MAX` is a 5-byte opcode whose first 4 bytes are `0xFF 0xFF 0xFF 0xFF` and whose last byte encodes `cost_function` in bits 7-6 (e.g., `0x40` for cost function 1).

### Impact Explanation

When the multiplication wraps to `0`:

1. `op_unknown` returns `Ok(Reduction(0, nil))` — a zero-cost operation.
2. Inside a `softfork` guard, the attacker sets `expected_cost = 0`; the guard's exact-match check (`current_cost == guard.expected_cost`) passes, allowing a softfork that declares zero cost to be accepted by consensus.
3. The true intended cost (`base_cost × (cost_multiplier + 1)`) is never charged, breaking the cost-metering invariant that prevents resource exhaustion.
4. In debug builds (overflow-checks enabled), the same input causes a panic, creating a consensus divergence between debug and release nodes.

### Likelihood Explanation

The attacker fully controls both inputs to the multiplication:
- The opcode bytes (setting `cost_multiplier = u32::MAX`).
- The argument list (tuning `base_cost` to a multiple of `2^32`).

The required argument count (~13.4 million) is below the interpreter's stack limit (20 million). The arithmetic alignment is deterministic and computable offline. No privileged access, governance action, or social engineering is required — only the ability to submit a CLVM program, which is the normal attacker surface for Chia coin puzzles.

### Recommendation

Replace the unchecked multiplication with a checked variant and treat overflow as an invalid opcode:

```rust
// line 261 replacement
let Some(new_cost) = cost.checked_mul(cost_multiplier + 1) else {
    return Err(EvalErr::Invalid(o))?;
};
cost = new_cost;
```

This mirrors the pattern already used in `op_add`'s fast path (`checked_add`, line 435) and eliminates the silent wrap-around.

### Proof of Concept

Opcode bytes: `[0xFF, 0xFF, 0xFF, 0xFF, 0x40]`
- Last byte `0x40` → `cost_function = (0x40 >> 6) = 1` (ARITH-like)
- First 4 bytes `0xFF 0xFF 0xFF 0xFF` → `cost_multiplier = u32::MAX = 4 294 967 295`
- `cost_multiplier + 1 = 2^32`

Argument list: 13 421 771 zero-byte atoms plus enough 1-byte atoms to total 159 bytes.
- `base_cost = 99 + 13 421 771 × 320 + 159 × 3 = 4 294 967 296 = 2^32`

Multiplication: `2^32 × 2^32 = 2^64 ≡ 0 (mod 2^64)` → wrapped cost = `0`.

Guard check: `0 > u32::MAX` → false → returns `Ok(Reduction(0, nil))`.
