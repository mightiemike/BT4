### Title
Fee-on-Transfer Token Inflates Credited Collateral Balance — (`File: core/contracts/EndpointStorage.sol` + `core/contracts/Clearinghouse.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral` transfers tokens via a two-hop path (user → Endpoint → Clearinghouse) and then queues a slow-mode transaction that credits the user with the caller-supplied `amount`. If the deposited token charges a fee on transfer, the Clearinghouse receives strictly less than `amount`, but the user's on-chain balance is inflated by the full `amount`. This creates a permanent solvency gap: the sum of all credited spot balances exceeds the actual token holdings of the Clearinghouse.

---

### Finding Description

The deposit flow is:

1. `Endpoint.depositCollateralWithReferral` receives `amount` from the caller and calls `handleDepositTransfer`.
2. `handleDepositTransfer` performs **two sequential transfers** of the same nominal `amount`:
   - `safeTransferFrom(token, from, amount)` — user → Endpoint
   - `safeTransferTo(token, address(clearinghouse), amount)` — Endpoint → Clearinghouse [1](#0-0) 

3. Immediately after, a `SlowModeTx` is enqueued encoding the original caller-supplied `amount` as the deposit amount. [2](#0-1) 

4. When the sequencer processes the slow-mode queue, `Clearinghouse.depositCollateral` is called, which scales `txn.amount` (the original `amount`) and credits it to the user's spot balance via `spotEngine.updateBalance`. [3](#0-2) 

At no point is the actual post-transfer balance of the Clearinghouse measured. The credited amount is always the caller-supplied `amount`, not `balanceAfter - balanceBefore`.

For a fee-on-transfer token with fee rate `f`:
- User sends `amount`, Endpoint receives `amount * (1 - f)`.
- Endpoint sends `amount * (1 - f)`, Clearinghouse receives `amount * (1 - f)^2`.
- User is credited `amount` (full, no deduction).

The two-hop design doubles the fee loss relative to a single-hop deposit, making the discrepancy `amount - amount*(1-f)^2 ≈ 2*f*amount` per deposit.

---

### Impact Explanation

Every deposit with a fee-on-transfer token inflates the depositor's credited balance beyond what the Clearinghouse actually holds. Over time, the cumulative shortfall grows with deposit volume. When users withdraw, the Clearinghouse will be unable to satisfy all withdrawal requests — later withdrawers will find the vault insolvent. The `spotEngine` state (`totalDepositsNormalized`) is permanently overcounted, corrupting health checks, liquidation thresholds, and interest rate calculations for all users of that product. [4](#0-3) 

---

### Likelihood Explanation

USDT on Ethereum already has the fee-on-transfer logic deployed (fee currently set to 0). If the fee is ever activated, or if any new spot product is listed with a token that charges transfer fees, every deposit immediately triggers the accounting corruption. The entry path is fully permissionless: any user calling `depositCollateral` or `depositCollateralWithReferral` is sufficient. [5](#0-4) 

---

### Recommendation

Replace the two-hop transfer with a balance-check pattern. Measure the Clearinghouse's token balance before and after the transfer and use the delta as the credited amount:

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    token.safeTransferFrom(from, address(clearinghouse), amount); // single hop
    uint256 received = token.balanceOf(address(clearinghouse)) - before;
    // pass `received` (not `amount`) into the slow-mode tx
}
```

This also eliminates the double-fee loss by collapsing the two-hop transfer into a single direct transfer from the user to the Clearinghouse.

---

### Proof of Concept

1. Deploy a mock ERC20 with a 1% fee on every transfer.
2. Register it as a spot product in the Clearinghouse.
3. Call `Endpoint.depositCollateral(subaccountName, productId, 1000e6)`.
   - `handleDepositTransfer` transfers 1000e6 from user → Endpoint: Endpoint receives 990e6.
   - `handleDepositTransfer` transfers 1000e6 from Endpoint → Clearinghouse: **reverts** (Endpoint only holds 990e6) — or if the token allows it, Clearinghouse receives 980.1e6.
4. The slow-mode tx encodes `amount = 1000e6`.
5. Sequencer processes the slow-mode tx; `Clearinghouse.depositCollateral` credits the user with `1000e6` worth of normalized balance.
6. Clearinghouse holds only 980.1e6 but has credited 1000e6 — a 19.9e6 shortfall per deposit.
7. Repeat across many depositors; the vault becomes insolvent for the last withdrawers. [1](#0-0) [6](#0-5)

        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

    function _updateBalanceNormalized(
        State memory state,
        BalanceNormalized memory balance,
        int128 balanceDelta
    ) internal pure {
        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized -= balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized += balance.amountNormalized;
        }

        int128 cumulativeMultiplierX18;
        if (balance.amountNormalized > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        int128 newAmount = balance.amountNormalized.mul(
            cumulativeMultiplierX18
        ) + balanceDelta;

        if (newAmount > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        balance.amountNormalized = newAmount.div(cumulativeMultiplierX18);

        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized += balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized -= balance.amountNormalized;
        }
    }
