### Title
Checks-Effects-Interactions Violation in `withdrawCollateral` Enables Re-Entrancy Balance Restoration Loop — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs the token transfer (interaction) before updating the spot engine balance (effect) and running the health check (check). With no re-entrancy guard anywhere in the codebase, a recipient smart contract using an ERC-777 token (or any token with `_afterTokenTransfer` hooks) can re-enter `Endpoint.depositCollateralWithReferral` during the transfer callback, queue a slow-mode deposit for the same amount, and have the spot engine balance restored by the sequencer — allowing the same collateral to be withdrawn again indefinitely.

---

### Finding Description

`Clearinghouse.withdrawCollateral` executes in the following order:

1. **Interaction** — `handleWithdrawTransfer(token, sendTo, amount, idx)` at line 408, which ultimately calls `token.safeTransfer(sendTo, amount)` inside `BaseWithdrawPool.handleWithdrawTransfer` at line 189.
2. **Effect** — `spotEngine.updateBalance(productId, sender, amountRealized)` at line 412.
3. **Check** — `require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH)` at line 419. [1](#0-0) 

The correct CEI order is Check → Effect → Interaction. Here the interaction fires first, giving the recipient contract a live callback window in which the spot engine still reflects the pre-withdrawal balance.

`Endpoint.depositCollateralWithReferral` is `public` with no re-entrancy guard: [2](#0-1) 

It calls `handleDepositTransfer` (pulling tokens from `msg.sender` back into the clearinghouse) and then enqueues a `DepositCollateral` slow-mode transaction. The sequencer later processes that transaction by calling `clearinghouse.depositCollateral`, which calls `spotEngine.updateBalance(+X)`, restoring the balance. [3](#0-2) 

No `nonReentrant` modifier or equivalent guard exists anywhere in the codebase — confirmed by a full-repo search returning zero matches for `nonReentrant`, `ReentrancyGuard`, `_ENTERED`, or `_status`.

---

### Impact Explanation

Each attack cycle:

| Step | Token balance (clearinghouse) | Spot engine balance (attacker) |
|------|-------------------------------|-------------------------------|
| Before withdrawal | X | X |
| After `handleWithdrawTransfer` (callback fires here) | 0 | X (not yet decremented) |
| Attacker re-deposits X via `depositCollateralWithReferral` | X | X (slow-mode queued) |
| `spotEngine.updateBalance(-X)` executes | X | 0 |
| Health check passes (balance = 0, no positions) | X | 0 |
| Sequencer processes slow-mode deposit | X | X (restored) |

The attacker ends each cycle with their spot engine balance fully restored while having extracted X tokens from the clearinghouse in the prior cycle. Repeating this loop drains the entire clearinghouse token reserve for the affected product.

---

### Likelihood Explanation

- ERC-777 tokens are deployed on mainnet and are valid collateral candidates for any exchange that does not explicitly block them. The protocol does not restrict token standards.
- The `sendTo` address for slow-mode withdrawals resolves to `address(uint160(bytes20(sender)))` — the first 20 bytes of the subaccount — which is the user's own address. Smart contract wallets (Gnosis Safe, ERC-4337 accounts) are common and can implement `tokensReceived`. [4](#0-3) 

- `depositCollateralWithReferral` is unconditionally `public` with no guard, making the re-entry target trivially reachable.
- The sequencer processes slow-mode deposits as part of normal operation and has no mechanism to detect or reject deposits that originated from a re-entrancy callback.

---

### Recommendation

Apply the CEI pattern in `withdrawCollateral`: perform the balance update and health check **before** calling `handleWithdrawTransfer`:

```solidity
function withdrawCollateral(...) public virtual onlyEndpoint {
    // CHECKS
    require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
    require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
    ...
    // EFFECTS
    int128 amountRealized = -int128(amount) * int128(multiplier);
    spotEngine.updateBalance(productId, sender, amountRealized);
    spotEngine.assertUtilization(productId);
    require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
    // INTERACTIONS
    handleWithdrawTransfer(token, sendTo, amount, idx);
    emit ModifyCollateral(amountRealized, sender, productId);
}
```

Alternatively, add a `nonReentrant` modifier (OpenZeppelin `ReentrancyGuard`) to both `withdrawCollateral` and `depositCollateralWithReferral`.

### Proof of Concept

```
Attacker deploys MaliciousWallet at address A.
A registers subaccount A<name> on Nado with ERC-777 token T (balance X).

1. A submits WithdrawCollateral for X of token T.
2. Sequencer calls Endpoint → EndpointTx.processTransactionImpl
   → clearinghouse.withdrawCollateral(A<name>, productId, X, address(0), idx)
3. withdrawCollateral resolves sendTo = A (line 405).
4. handleWithdrawTransfer → token.safeTransfer(withdrawPool, X)
   → BaseWithdrawPool.submitWithdrawal → token.safeTransfer(A, X)   ← ERC-777 hook fires
5. MaliciousWallet.tokensReceived():
     token.approve(endpoint, X);
     endpoint.depositCollateralWithReferral(A<name>, productId, X, "");
     // handleDepositTransfer pulls X back to clearinghouse
     // slow-mode DepositCollateral queued for A<name>
6. Hook returns. Execution resumes in withdrawCollateral:
     spotEngine.updateBalance(-X)  → balance[A<name>] = 0
     getHealth(A<name>) >= 0       → passes (no open positions)
7. Sequencer processes slow-mode deposit:
     clearinghouse.depositCollateral → spotEngine.updateBalance(+X)
     → balance[A<name>] = X again
8. Goto step 1. Each iteration extracts X tokens from the clearinghouse.

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)

        SlowModeConfig memory _slowModeConfig = slowModeConfig;

                        sender: subaccount,
                        productId: productId,
                        amount: amount
                    })
