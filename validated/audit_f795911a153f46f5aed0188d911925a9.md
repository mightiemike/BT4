### Title
Protobuf `ValidResourceBounds` Deserialization Silently Collapses `AllResources(X, 0, 0)` to `L1Gas(X)`, Producing a Different Transaction Hash Domain — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` converter in the P2P block-sync path uses the condition `l1_data_gas.is_zero() && l2_gas.is_zero()` to select the `L1Gas` variant. This condition is also true for a post-0.13.3 `AllResources(X, 0, 0)` transaction (non-zero L1 gas only), causing it to be silently reconstructed as `L1Gas(X)`. Because `get_tip_resource_bounds_hash` hashes **two** resource-bound felts for `L1Gas` and **three** for `AllResources`, the deserialized transaction carries a structurally different hash preimage than the one committed by the proposer, breaking hash-domain canonicalization across the serialization boundary.

### Finding Description

`ValidResourceBounds` has two variants with distinct hash semantics:

- `L1Gas(ResourceBounds)` — pre-0.13.3; `get_tip_resource_bounds_hash` hashes `[tip, L1_GAS_packed, L2_GAS_packed_zero]` (3 Poseidon inputs).
- `AllResources(AllResourceBounds)` — post-0.13.3; hashes `[tip, L1_GAS_packed, L2_GAS_packed, L1_DATA_GAS_packed]` (4 Poseidon inputs).

The gateway always accepts `RpcTransaction` carrying `AllResourceBounds` and computes the hash via `InternalRpcTransactionWithoutTxHash::calculate_transaction_hash` → `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` with `ValidResourceBounds::AllResources(...)`. A transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` (non-zero L1 gas, zero L2 and data gas) passes the gateway's `validate_resource_bounds` check (the `max_possible_fee` is non-zero) and is assigned **hash_A** (4-input Poseidon).

When this block is later synced to a peer via P2P, the `Transaction::Invoke(InvokeTransactionV3 { resource_bounds: ValidResourceBounds::AllResources(X, 0, 0), … })` is serialized through `From<ValidResourceBounds> for protobuf::ResourceBounds`, which emits explicit zero `ResourceLimits` for both `l2_gas` and `l1_data_gas`. On the receiving side, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` executes:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs  lines 426-435
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // explicit zero present
let l1_gas: ResourceBounds = l1_gas.try_into()?;
let l2_gas: ResourceBounds = l2_gas.try_into()?;
let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong variant selected
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

The condition fires even though `l1_data_gas` was explicitly present (not absent), reconstructing `L1Gas(X)`. Any subsequent call to `get_tip_resource_bounds_hash` on this deserialized transaction produces **hash_B** (3-input Poseidon) ≠ **hash_A**.

The `get_tip_resource_bounds_hash` function that drives this divergence:

```rust
// crates/starknet_api/src/transaction_hash.rs  lines 188-211
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 felts total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 felts total
    }
});
Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
```

The Starknet OS Cairo code independently enforces `assert n_resource_bounds = 3` for all V3 transactions, so a transaction stored with the `L1Gas` variant (2 bounds) will cause the OS to abort when the block is proved.

### Impact Explanation

A transaction with `AllResources(X, 0, 0)` is committed to a block with **hash_A** (4-input Poseidon). After P2P block sync, the receiving node stores the same transaction with the `L1Gas(X)` variant. Any downstream operation that recomputes the hash — including `validate_transaction_hash`, re-execution for proving, or the Starknet OS — will derive **hash_B** ≠ **hash_A**. The OS Cairo assertion `assert n_resource_bounds = 3` will abort proving for the affected block. This matches: **Wrong state, receipt, or revert result from blockifier/syscall/execution logic for accepted input** and **Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution** (via OS abort).

### Likelihood Explanation

The trigger requires a V3 transaction with non-zero L1 gas and zero L2 and data gas. The gateway's `validate_resource_bounds` accepts this when `min_gas_price = 0` (the test suite's `DEFAULT_VALIDATOR_CONFIG_FOR_TESTING` demonstrates this path is valid). The protobuf round-trip is exercised on every P2P-synced block. No special privilege is required; any user can craft such a transaction.

### Recommendation

The protobuf serialization must preserve the `ValidResourceBounds` variant across the boundary. Two options:

1. **Add a discriminator field** to `protobuf::ResourceBounds` (e.g., `bool all_resources`) so the deserializer can reconstruct the correct variant without relying on value-based inference.
2. **Remove the value-based inference entirely** for post-0.13.3 transactions: once the 0.13.2 compatibility window is closed (as the `TODO(Shahak)` comment anticipates), always deserialize as `AllResources` when `l1_data_gas` is present, regardless of whether it is zero.
