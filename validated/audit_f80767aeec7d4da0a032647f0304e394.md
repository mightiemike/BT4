### Title
Incorrect `existingWeight` Formula in Spread Health Calculation Overestimates Subaccount Health — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.getHealth()` computes a spread health adjustment using `existingWeight = (spotCoreRisk.longWeight + perpCoreRisk.longWeight) / 2`. Because `getCoreRisk` always populates `CoreRisk.longWeight` with the **long** weight (by passing a hardcoded `amount=1`), the short leg of every spread position is evaluated with the wrong weight. For a long-spot / short-perp spread, the perp's `shortWeight` (≥ 1) is replaced by its `longWeight` (≤ 1), causing the spread health bonus to be materially overestimated. The same wrong weight is fed into `_getSpreadWeightX18`, compounding the error.

---

### Finding Description

**Root cause 1 — `getCoreRisk` always returns `longWeight`**

`BaseEngine.getCoreRisk` calls `_getWeightX18(risk, 1, healthType)`, hardcoding `amount = 1`. Because `1 >= 0`, `_getWeightX18` always returns `longWeightInitialX18` (or `longWeightMaintenanceX18`), regardless of the subaccount's actual position direction. [1](#0-0) 

**Root cause 2 — `existingWeight` uses `longWeight` for both legs**

In `getHealth`, the spread adjustment is:

```solidity
int128 existingWeight = (spotCoreRisk.longWeight + perpCoreRisk.longWeight) / 2;
int128 spreadWeight   = RiskHelper._getSpreadWeightX18(perpCoreRisk, spotCoreRisk, healthType);
health += basisAmount.mul(spotCoreRisk.price + perpCoreRisk.price)
                     .mul(spreadWeight - existingWeight);
``` [2](#0-1) 

`existingWeight` is supposed to represent the average weight **already applied** to the matched portion in `_calculateProductHealth`. For a long-spot / short-perp spread, `_calculateProductHealth` applies `perpShortWeight` (≥ 1) to the perp. But `existingWeight` uses `perpLongWeight` (≤ 1) instead.

**Root cause 3 — `_getSpreadWeightX18` also uses `longWeight` for the short leg**

```solidity
if (spotCoreRisk.amount > 0) {
    spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;  // should use perpShortWeight
} else {
    spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;  // should use spotShortWeight
}
``` [3](#0-2) 

**Protocol invariant confirming `shortWeight ≥ 1 > longWeight`:**

```solidity
riskStore.longWeightMaintenance  <= 10**9 &&   // longWeight  ≤ 1
riskStore.shortWeightMaintenance >= 10**9,      // shortWeight ≥ 1
``` [4](#0-3) 

**Numerical example** (typical values: `spotLongWeight = 0.9`, `perpLongWeight = 0.9`, `perpShortWeight = 1.1`):

| | Correct | Actual (buggy) |
|---|---|---|
| `existingWeight` | `(0.9 + 1.1) / 2 = 1.0` | `(0.9 + 0.9) / 2 = 0.9` |
| `spreadWeight` | `1 − (1 − 1.1)/5 = 1.02` | `1 − (1 − 0.9)/5 = 0.98` |
| `spreadWeight − existingWeight` | `0.02` | `0.08` |

The spread health bonus is **4× larger** than it should be.

---

### Impact Explanation

Every call to `getHealth` for a subaccount holding a spread position (long spot + short perp, or vice versa) returns an inflated health value. This directly affects:

- **`withdrawCollateral`** — health check at line 419 passes when it should fail, allowing collateral withdrawal that leaves the account undercollateralized.
- **`transferQuote`** — health check at line 249 is bypassed similarly.
- **`mintNlp`** — health check at line 480 is bypassed.
- **Liquidation avoidance** — `isUnderMaintenance` in `ClearinghouseLiq` calls `getHealth(MAINTENANCE)`, so accounts that should be liquidatable appear healthy.

The result is that users can hold positions with actual negative health while the protocol believes they are solvent, creating bad debt risk. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

Spread positions (long spot + short perp) are a first-class feature of the protocol — the `spreads` registry is set at initialization and used in every `getHealth` call. Any unprivileged trader who opens a spread position through `OffchainExchange` immediately benefits from the inflated health. No special access or admin action is required. The overestimation scales with position size, so a large spread position can create a substantial undercollateralization gap. [7](#0-6) 

---

### Recommendation

`getCoreRisk` must return the weight that `_calculateProductHealth` actually applies — i.e., the weight conditioned on the actual position direction. Add a `shortWeight` field to `CoreRisk` and populate it correctly:

```solidity
// BaseEngine.getCoreRisk
return IProductEngine.CoreRisk(
    amount,
    risk.priceX18,
    RiskHelper._getWeightX18(risk, 1, healthType),       // longWeight
    RiskHelper._getWeightX18(risk, -1, healthType)       // shortWeight (new field)
);
```

Then in `Clearinghouse.getHealth`, use the direction-aware weight for `existingWeight`:

```solidity
// For long-spot / short-perp (spotCoreRisk.amount > 0):
int128 existingWeight = (spotCoreRisk.longWeight + perpCoreRisk.shortWeight) / 2;
// spreadWeight should also use perpCoreRisk.shortWeight
```

And update `_getSpreadWeightX18` to accept and use the short weight for the short leg.

---

### Proof of Concept

1. Alice opens a spread: long 100 BTC-SPOT, short 100 BTC-PERP. Assume `spotLongWeight = 0.9`, `perpShortWeight = 1.1`, price = $30,000.
2. `getHealth` is called. `basisAmount = 100`. `spotPrice = perpPrice = 30,000e18`.
3. **Correct** spread adjustment: `100 * 60,000 * 0.02 = $120,000` health bonus.
4. **Actual** spread adjustment: `100 * 60,000 * 0.08 = $480,000` health bonus — **$360,000 phantom health**.
5. Alice calls `withdrawCollateral` for $360,000 worth of collateral. The health check passes (phantom health covers it). Her account is now actually undercollateralized by $360,000.
6. If BTC price drops, Alice cannot be liquidated in time because `isUnderMaintenance` also uses the inflated `getHealth`, and the protocol absorbs the bad debt. [2](#0-1) [1](#0-0) [8](#0-7)

    function _getSpreadWeightX18(
        IProductEngine.CoreRisk memory perpCoreRisk,
        IProductEngine.CoreRisk memory spotCoreRisk,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }
        int128 spreadWeight;
        if (spotCoreRisk.amount > 0) {
            spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
        } else {
            spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;
        }
        int128 maxSpreadWeight;
        if (healthType == IProductEngine.HealthType.INITIAL) {
            maxSpreadWeight = ONE - ONE / 100; // 0.99
        } else {
            maxSpreadWeight = ONE - (ONE * 6) / 1000; // 0.994
        }
        if (spreadWeight > maxSpreadWeight) {
            spreadWeight = maxSpreadWeight;
        }
        return spreadWeight;
    }
```
