### Title
Silent Swallow of Failed Slow-Mode Deposit Leaves User Collateral Permanently Locked — (`core/contracts/Endpoint.sol`)

---

### Summary

In `Endpoint.sol`, the `_executeSlowModeTransaction` function wraps `processSlowModeTransaction` in a `try/catch`. If the inner call reverts, the catch block silently discards the failure **without refunding the user's already-transferred collateral**. A comment in the code — `// try return funds now removed` — explicitly confirms that a refund mechanism was intentionally removed. Because `depositCollateralWithReferral` transfers tokens from the user **before** enqueuing the slow-mode transaction, any failure during slow-mode execution results in the user's tokens being permanently locked in the Clearinghouse with no recovery path.

---

### Finding Description

The deposit flow in `Endpoint.sol` is a two-phase operation:

**Phase 1 — Token transfer (immediate):** `depositCollateralWithReferral` calls `handleDepositTransfer`, which pulls the user's ERC-20 tokens into the Clearinghouse. This is irreversible from the user's perspective.

**Phase 2 — Balance credit (deferred):** A `SlowModeTx` of type `DepositCollateral` is enqueued with an `executableAt` timestamp set to `block.timestamp + SLOW_MODE_TX_DELAY` (hardcoded to three days). This transaction, when executed, calls `clearinghouse.depositCollateral()` to credit the user's subaccount balance in the SpotEngine. [1](#0-0) 

When the slow-mode transaction is eventually executed via `_executeSlowModeTransaction`, the call to `processSlowModeTransaction` is wrapped in a `try/catch`: [2](#0-1) 

If `processSlowModeTransaction` reverts for any reason, the catch block fires. The comment `// try return funds now removed` at line 226 explicitly documents that a refund path was present and was deliberately deleted. No refund, re-credit, or recovery mechanism exists in the catch block. The user's tokens remain in the Clearinghouse, but their subaccount balance is never updated.

---

### Impact Explanation

A user who calls `depositCollateralWithReferral` and whose slow-mode `DepositCollateral` transaction subsequently fails will:

1. Have their ERC-20 tokens permanently transferred out of their wallet into the Clearinghouse.
2. Receive zero credit to their subaccount balance in the SpotEngine.
3. Have no on-chain mechanism to recover the funds, since there is no admin mint, no re-queue path, and no refund logic in the catch block.

The asset delta is: user loses `amount` tokens of `productId` collateral with no corresponding balance increase anywhere in the protocol. This is a direct loss of user funds. [3](#0-2) 

---

### Likelihood Explanation

The slow-mode deposit transaction can fail in realistic, non-malicious scenarios:

- **Product delisting:** `clearinghouse.depositCollateral()` calls `_spotEngine().getConfig(productId).token`, which returns `address(0)` for a delisted product, causing a revert. A product can be delisted via `delistProduct` between the time the user deposits (Phase 1) and the time the slow-mode tx is executed (up to 3 days later).
- **Utilization breach:** `spotEngine.assertUtilization(productId)` is called during deposit processing and can revert if the pool is over-utilized at execution time.
- **Out-of-gas heuristic false negative:** The gas heuristic (`gasleft() <= 250000 || gasleft() <= gasRemaining / 2`) is explicitly acknowledged as imperfect. A revert that consumes gas near the threshold boundary could be misclassified as an OOG event, triggering `invalid()` and consuming all gas — but the silent-swallow path is the primary concern.

The 3-day window between deposit and execution materially increases the probability that protocol state changes (delistings, utilization spikes) cause the slow-mode tx to fail. [4](#0-3) 

---

### Recommendation

Restore a refund path in the catch block of `_executeSlowModeTransaction` for `DepositCollateral` transactions. Specifically:

1. In the catch block, decode the transaction type. If it is `TransactionType.DepositCollateral`, decode the `DepositCollateral` struct and transfer `amount` of the corresponding token back to `address(bytes20(txn.sender))`.
2. Alternatively, before executing the slow-mode transaction, record the `(sender, productId, amount)` tuple in a mapping. If execution fails, expose an admin or user-callable function to claim the refund from that mapping.
3. At minimum, emit an event in the catch block so that off-chain monitoring can detect and manually remediate stuck deposits.

---

### Proof of Concept

1. User calls `depositCollateralWithReferral(subaccount, productId, amount, referral)`.
2. `handleDepositTransfer` pulls `amount` tokens from `msg.sender` into the Clearinghouse. Tokens are gone from the user's wallet.
3. A `SlowModeTx` is enqueued with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY`.
4. Before the slow-mode tx is executed, the owner calls `delistProduct` for `productId`, removing it from the engine registry.
5. After the delay, anyone calls `executeSlowModeTransaction()`. Inside `_executeSlowModeTransaction`, `processSlowModeTransaction` is called via `try`. The call reaches `clearinghouse.depositCollateral()`, which calls `_spotEngine().getConfig(productId).token` — this returns `address(0)` for the delisted product, causing a revert.
6. The catch block fires. `gasleft()` is well above the threshold, so `invalid()` is not triggered. Execution falls through to `// try return funds now removed` — no refund is issued.
7. The slow-mode tx slot is deleted (`delete slowModeTxs[_slowModeConfig.txUpTo++]`), permanently discarding the record.
8. The user's `amount` tokens remain in the Clearinghouse. Their subaccount balance is `0`. There is no on-chain path to recover the funds. [5](#0-4)

```text
    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
```
