### Title
NEAR Storage Deposit Permanently Locked After GC Frees Blocks — No Sweep Function (`contract/src/lib.rs`)

---

### Summary

The `BtcLightClient` contract accepts NEAR token deposits from relayers to cover on-chain storage costs for submitted block headers. When the garbage collector (`run_mainchain_gc`) later frees those headers, the storage bytes are released but the NEAR tokens that were originally paid for them remain permanently locked in the contract. No `sweep()`, `withdraw()`, or admin-recovery function exists anywhere in the contract. Over the contract's operational lifetime this causes a monotonically growing pool of irrecoverable NEAR.

---

### Finding Description

`submit_blocks` is marked `#[payable]` and uses a pay-for-storage pattern:

```
amount = env::attached_deposit()
initial_storage = env::storage_usage()
// ... insert new headers, run GC ...
diff_storage_usage = env::storage_usage().saturating_sub(initial_storage)
required_deposit   = storage_byte_cost * diff_storage_usage
refund             = amount.saturating_sub(required_deposit)
if refund > 0 { transfer(refund, predecessor) }
``` [1](#0-0) 

The refund logic is correct for the **current** call: the caller gets back whatever they over-paid. However, it says nothing about the NEAR tokens that were retained by the contract in **previous** calls to cover storage that GC has now freed.

`run_mainchain_gc` removes the oldest mainchain headers from all three storage structures (`headers_pool`, `mainchain_height_to_header`, `mainchain_header_to_height`): [2](#0-1) 

When those entries are deleted, NEAR's storage accounting reduces the contract's required stake, but the tokens that were deposited to cover that stake remain in the contract's balance. There is no bookkeeping of who paid for which block, and no function to return the freed stake to anyone.

A grep across the entire `contract/src/` tree finds zero occurrences of `sweep`, `withdraw`, `recover`, or any admin transfer primitive. The only outbound `Promise::transfer` in the contract is the per-call refund inside `submit_blocks`.

Additionally, `store_fork_header` inserts headers into `headers_pool` without any corresponding removal path (fork blocks are only removed if they are later promoted via reorg): [3](#0-2) 

`run_mainchain_gc` explicitly skips fork blocks: [4](#0-3) 

Storage paid for fork headers that are never promoted is also permanently locked.

---

### Impact Explanation

NEAR tokens deposited by relayers for storage that is subsequently freed by GC are permanently irrecoverable. With the recommended `gc_threshold = 52704` (≈ one year of Bitcoin blocks) and a storage cost of roughly 10 yoctoNEAR per byte, each pruned `ExtendedHeader` entry (≈ 200 bytes across the three maps) releases ~2 000 yoctoNEAR of storage obligation but leaves the corresponding deposit locked. Across a year of continuous operation this accumulates to a non-trivial sum. There is no admin path to recover it.

---

### Likelihood Explanation

This occurs during **normal, intended operation**. GC is called automatically on every `submit_blocks` invocation once the chain exceeds `gc_threshold`. Any production deployment running for more than a few months will exhibit the accumulation. No adversarial action is required; the relayer's own routine submissions trigger it.

---

### Recommendation

Add a privileged `sweep()` function (callable only by `Role::DAO` or a designated treasury role) that transfers the contract's excess balance — i.e., `env::account_balance() - env::storage_byte_cost() * env::storage_usage()` — to a designated recipient. This mirrors the `sweep()` fix applied in the referenced PayrollManager PR 39. Alternatively, track per-relayer deposits and issue refunds at GC time, though this is significantly more complex.

---

### Proof of Concept

1. Relayer (with `UnrestrictedSubmitBlocks`) calls `submit_blocks([h1..h100])` with a deposit of `D` NEAR covering 100 new headers. The contract retains `required = storage_byte_cost * Δstorage`; the rest is refunded. `required` is now locked in the contract balance.

2. After `gc_threshold` more blocks are submitted, `run_mainchain_gc` removes `h1..h100` from `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height`. [5](#0-4) 

3. The contract's `env::storage_usage()` drops by the bytes freed, reducing the minimum required balance. The `required` NEAR from step 1 is now surplus in the contract's balance.

4. No code path exists to transfer this surplus out. Searching `contract/src/lib.rs` for any outbound `Promise::transfer` yields only the single refund line inside `submit_blocks` (line 192), which only covers the current call's overpayment. [6](#0-5) 

5. Repeat for every GC cycle. The locked surplus grows monotonically with no recovery path.
