### Title
Fork Headers Permanently Lock NEAR Storage Deposits With No Recovery Mechanism - (`contract/src/lib.rs`)

---

### Summary

The `BtcLightClient` contract stores fork (orphan) block headers in `headers_pool` via `store_fork_header`, but provides no mechanism to ever remove them. The garbage-collection function `run_mainchain_gc` exclusively prunes mainchain blocks. NEAR storage deposits paid by relayers to cover the on-chain cost of fork headers are therefore permanently locked in the contract account with no withdrawal, sweep, or recovery path.

---

### Finding Description

When `submit_blocks` is called with a header that does not extend the current mainchain tip, `submit_block_header_inner` routes it to `store_fork_header`:

```rust
// contract/src/lib.rs:549-566
} else {
    log!("Block {}: saving to fork", current_header.block_hash);
    ...
    self.store_fork_header(&current_header);

    if current_header.chain_work > total_main_chain_chainwork {
        log!("Chain reorg");
        self.reorg_chain(current_header, last_main_chain_block_height);
    }
}
```

`store_fork_header` writes only to `headers_pool`:

```rust
// contract/src/lib.rs:665-667
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
```

The only removal function is `remove_block_header`, which is called exclusively from `run_mainchain_gc` (iterating over `mainchain_height_to_header`) and from `reorg_chain` (removing old mainchain blocks displaced by a winning fork). Neither path ever targets a fork-only entry — a header that was stored as a fork but whose fork never accumulated enough chainwork to trigger a reorg.

`run_mainchain_gc` is explicit about this scope:

```rust
// contract/src/lib.rs:401-408
for height in start_removal_height..end_removal_height {
    let blockhash = &self
        .mainchain_height_to_header
        .get(&height)
        ...;
    self.remove_block_header(blockhash);
    self.mainchain_height_to_header.remove(&height);
}
```

It iterates only over `mainchain_height_to_header`. Fork headers are never inserted into that map, so they are invisible to GC. The project's own documentation confirms this: *"Only mainchain blocks are deleted; fork/sidechain blocks are not affected."*

`submit_blocks` charges the relayer for the net storage increase after headers are added and GC runs:

```rust
// contract/src/lib.rs:182-188
let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());
require!(amount >= required_deposit, ...);
```

Because fork headers are never removed, the storage they occupy never decreases, and the NEAR tokens staked for that storage are permanently locked in the contract account. There is no `withdraw`, `sweep`, or any other function in the contract to recover them — confirmed by the complete absence of any such function across all source files in `contract/src/`.

---

### Impact Explanation

Every NEAR token paid as a storage deposit for a losing fork header is permanently locked in the contract account. The contract has no function to release this storage or return the deposited NEAR to any party. Over the operational lifetime of the contract, fork header accumulation is unbounded: the contract's own documentation acknowledges that fork blocks are intentionally retained. The locked NEAR is inaccessible to the relayer who paid it, to the DAO, and to any other account, unless the contract is upgraded to add a removal path.

---

### Likelihood Explanation

Bitcoin, Litecoin, Dogecoin, and Zcash all produce orphan blocks in normal operation. Any production deployment of this contract will receive fork headers from relayers as a routine matter. The `submit_blocks` entry point is reachable by any account holding the `Relayer` or `UnrestrictedSubmitBlocks` role, which is the normal operational role granted to relayer services. No adversarial behavior is required; the lockup occurs through ordinary, expected usage.

---

### Recommendation

Add a `run_fork_gc` (or equivalent) function that accepts a list of fork block hashes and removes them from `headers_pool`, provided they are not present in `mainchain_header_to_height`. This mirrors the pattern already used for mainchain GC. The freed storage will automatically restore the corresponding NEAR to the contract account, from which it can be returned to depositors or managed by the DAO. Alternatively, track fork headers in a separate indexed structure so they can be enumerated and pruned on a schedule.

---

### Proof of Concept

1. Authorized relayer calls `submit_blocks` with a valid fork header `F` (prev hash points to a block already in `headers_pool` but not the current mainchain tip). Deposit covers the storage cost of `F`.
2. `submit_block_header_inner` calls `store_fork_header(&F)`, inserting `F` into `headers_pool` only.
3. `F`'s `chain_work` does not exceed the mainchain tip's `chain_work`, so `reorg_chain` is not called.
4. `run_mainchain_gc` runs but iterates only over `mainchain_height_to_header`; `F` is not in that map and is skipped.
5. `F` remains in `headers_pool` indefinitely. The NEAR storage deposit paid in step 1 is permanently locked.
6. Calling `run_mainchain_gc` with any `batch_size` does not remove `F`. No other public function exists that can remove `F`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
