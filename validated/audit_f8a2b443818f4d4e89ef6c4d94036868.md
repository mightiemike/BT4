### Title
Missing Lower-Bound Validation on `confirmations` Parameter Allows Zero-Confirmation Proof Acceptance — (`contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a `confirmations: u64` parameter with no lower-bound check. Passing `confirmations = 0` satisfies every guard in the function, causing the contract to return `true` for a transaction with zero confirmations — completely bypassing the finality guarantee the parameter is meant to enforce.

### Finding Description
Both public proof-verification entry points delegate confirmation enforcement to two `require!` guards:

```rust
// Guard 1 — upper bound only
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [1](#0-0) 

```rust
// Guard 2 — the actual confirmation depth check
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [2](#0-1) 

When `confirmations = 0`:

- Guard 1: `0 <= gc_threshold` — always `true` for any `u64`.
- Guard 2: `(any u64 value) >= 0` — always `true` because `u64` is unsigned and can never be negative.

Both guards pass unconditionally. The function then proceeds to evaluate only the Merkle proof, returning `true` for any transaction that is present in any main-chain block, regardless of how recently that block was submitted.

The same path is reachable through `verify_transaction_inclusion_v2`, which validates the coinbase proof and then calls `verify_transaction_inclusion` via `args.into()`: [3](#0-2) 

The `confirmations` field is part of the public `ProofArgs` / `ProofArgsV2` structs with no constraints: [4](#0-3) 

### Impact Explanation
Any recipient contract or off-chain application that calls `verify_transaction_inclusion{_v2}` and trusts the returned `bool` to mean "this transaction has at least N confirmations" can be deceived. By supplying `confirmations = 0`, an attacker obtains a `true` result for a transaction that sits in a block with zero depth — a block that could still be reorganized away. This breaks the SPV finality guarantee that is the primary security property of the light client, enabling double-spend or premature asset-release attacks against any downstream consumer.

### Likelihood Explanation
The entry point is fully public and requires no privileged role. Any NEAR account can craft a `ProofArgs` with `confirmations = 0` and a valid Merkle proof for a freshly submitted block. The attack requires only knowledge of a valid transaction and its Merkle proof, both of which are public Bitcoin data. Likelihood is high for any deployment where a downstream contract relies on the returned boolean without independently enforcing a minimum confirmation count.

### Recommendation
Add an explicit lower-bound check at the start of `verify_transaction_inclusion`:

```rust
require!(args.confirmations >= 1, "Confirmations must be at least 1");
```

Alternatively, enforce this in the `ProofArgs` / `ProofArgsV2` deserialization or document clearly that callers must never pass `0`, and add the guard in `verify_transaction_inclusion_v2` as well since it is the recommended replacement for the deprecated v1 function.

### Proof of Concept
1. Deploy the contract (Bitcoin feature) with any valid genesis and a few submitted blocks.
2. Call `verify_transaction_inclusion` with:
   - `tx_block_blockhash` = hash of the most recently submitted block (0 confirmations from tip).
   - `tx_id` = any transaction hash.
   - `merkle_proof` = a valid Merkle proof for that transaction.
   - `confirmations = 0`.
3. Guard 1: `0 <= gc_threshold` → passes.
4. Guard 2: `(tip_height - tip_height) + 1 = 1 >= 0` → passes.
5. The function returns `true`, certifying a zero-confirmation transaction as "verified."
