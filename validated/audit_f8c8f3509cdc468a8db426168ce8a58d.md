### Title
Unprivileged Caller Can Trigger Unbounded Mainchain GC, Permanently Deleting Block Headers — (`contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` is a public, state-mutating NEAR contract method that carries no caller authorization check. Any unprivileged NEAR account can invoke it with an arbitrarily large `batch_size`, causing the contract to permanently delete all excess mainchain block headers in a single transaction. Deleted headers cannot be recovered, and any subsequent SPV proof for an affected block will panic, breaking the light client's core verification guarantee.

---

### Finding Description

The function is declared as a plain `pub fn` with only a `#[pause]` attribute:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
``` [1](#0-0) 

`#[pause]` only gates calls when the contract is in a paused state. It does **not** restrict who may call the function when the contract is running normally. There is no `#[trusted_relayer]`, no `#[private]`, and no explicit `env::predecessor_account_id()` check — unlike `submit_blocks`, which carries all three layers of protection:

```rust
#[payable]
#[pause]
#[trusted_relayer]
pub fn submit_blocks(...)
``` [2](#0-1) 

Inside `run_mainchain_gc`, the deletion logic is:

```rust
let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);
``` [3](#0-2) 

With `batch_size = u64::MAX`, `selected_amount_to_remove` equals `total_amount_to_remove`, so every header beyond `gc_threshold` is deleted in one call. Each deleted header is permanently removed from `mainchain_height_to_header`, `mainchain_header_to_height`, and `headers_pool`:

```rust
self.remove_block_header(blockhash);
self.mainchain_height_to_header.remove(&height);
``` [4](#0-3) 

`remove_block_header` also removes the entry from `mainchain_header_to_height` and `headers_pool`:

```rust
fn remove_block_header(&mut self, header_block_hash: &H256) {
    self.mainchain_header_to_height.remove(header_block_hash);
    self.headers_pool.remove(header_block_hash);
}
``` [5](#0-4) 

---

### Impact Explanation

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` look up the target block in `mainchain_header_to_height` and `headers_pool`:

```rust
let target_block_height = self
    .mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));
``` [6](#0-5) 

Once a header is GC'd by the attacker, any SPV proof referencing that block will **panic unconditionally**. The deleted state cannot be restored: the relayer's `submit_blocks` path rejects headers whose `prev_block_hash` is not already in `headers_pool`, so re-submission of a pruned ancestor chain is not possible without a full re-initialization. This permanently breaks the canonical chain mapping for all affected heights, corrupting the light client's core trust guarantee for any downstream contract or user relying on those proofs.

---

### Likelihood Explanation

The entry point is a standard NEAR contract call requiring no stake, no role, and no special setup. Any NEAR account can call `run_mainchain_gc(batch_size: u64::MAX)` at any time the contract is not paused. The attack is a single transaction, costs only gas, and is irreversible. The contract is explicitly designed to be used on mainnet (`btc-client.bridge.near`), where the chain will routinely exceed `gc_threshold`, making the excess-block window continuously available for exploitation.

---

### Recommendation

Add the same `#[trusted_relayer]` guard that protects `submit_blocks`, or introduce an explicit role check (e.g., `Role::DAO` or a dedicated `GCManager` role) at the top of `run_mainchain_gc`. If the function is intended to be callable by anyone as a maintenance helper, enforce a strict per-call `batch_size` cap (e.g., equal to the relayer's normal batch size) to limit the blast radius of any single call.

---

### Proof of Concept

1. Deploy the contract on NEAR testnet with `gc_threshold = 100` and submit 200 blocks via a trusted relayer account. The chain now stores 200 headers; 100 are excess.
2. From a **fresh, unprivileged** NEAR account (no roles granted), call:
   ```
   run_mainchain_gc(batch_size: 18446744073709551615)
   ```
3. The contract deletes all 100 excess headers in one transaction. `mainchain_initial_blockhash` advances by 100 heights.
4. Call `verify_transaction_inclusion` with a `tx_block_blockhash` that was in the deleted range. The call panics: `"block does not belong to the current main chain"`.
5. The relayer cannot restore the deleted headers without a full contract re-initialization, confirming permanent state corruption.
