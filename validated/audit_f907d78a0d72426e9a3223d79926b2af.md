### Title
NFT `tokenData` silently dropped during ICS-721 cross-chain transfer — (`x/nft-transfer/keeper/packet.go`)

### Summary

The `x/nft-transfer` module's ICS-721 implementation fetches each NFT's on-chain `Data` field during outgoing packet construction but never includes it in the `NonFungibleTokenPacketData` packet payload. On the receiving chain, `MintNFT` is called with a hardcoded empty string for `tokenData`. Any NFT with non-empty on-chain metadata (`Data`) that is transferred cross-chain via IBC permanently loses that metadata on the destination chain.

### Finding Description

The `x/nft` module stores a `Data` (tokenData) field on every NFT, settable via `MsgMintNFT` and `MsgEditNFT`. The `MintNFT` keeper function signature explicitly accepts and stores this field:

```go
func (k Keeper) MintNFT(
    ctx sdk.Context, denomID, tokenID, tokenNm,
    tokenURI, tokenData string, sender, owner sdk.AccAddress,
) error {
``` [1](#0-0) 

During outgoing IBC packet construction in `createOutgoingPacket`, the code fetches each NFT but only reads `nft.GetURI()` — `nft.GetData()` is never called:

```go
nft, err := k.nftKeeper.GetNFT(ctx, classID, tokenID)
...
tokenURIs = append(tokenURIs, nft.GetURI())
``` [2](#0-1) 

The packet is then constructed without any `tokenData`:

```go
packetData := types.NewNonFungibleTokenPacketData(
    fullClassPath, denom.Uri, tokenIDs, tokenURIs, sender.String(), receiver,
)
``` [3](#0-2) 

This is structurally enforced: the `NonFungibleTokenPacketData` proto definition and generated Go struct have no `tokenData` field at all — only `token_uris`:

```protobuf
message NonFungibleTokenPacketData {
  string class_id = 1;
  string class_uri = 2;
  repeated string token_ids = 3;
  repeated string token_uris = 4;
  string sender = 5;
  string receiver = 6;
}
``` [4](#0-3) [5](#0-4) 

On the receiving chain, `processReceivedPacket` mints the NFT with a hardcoded empty string `""` for `tokenData`:

```go
k.nftKeeper.MintNFT(ctx, voucherClassID, tokenID, "", data.TokenUris[i], "", escrowAddress, receiver)
``` [6](#0-5) 

The same omission applies to the denom `Schema` field: `IssueDenom` is called with `""` for schema on the receiving chain, even though the sending chain's denom may carry a non-empty schema. [7](#0-6) 

### Impact Explanation

Any NFT with a non-empty `Data` field that is transferred cross-chain via IBC will have its `Data` permanently set to `""` on the destination chain. This is an irreversible state corruption: the destination chain mints a structurally different NFT than the one that was escrowed on the source chain. If the NFT is later returned to the source chain via a reverse transfer, the source chain unescrows the original (with `Data` intact), but the round-trip has produced a permanently degraded representation on the intermediate chain. Applications or users relying on on-chain `tokenData` for provenance, licensing, game attributes, or other metadata will find it silently erased.

### Likelihood Explanation

Any unprivileged user who owns an NFT with a non-empty `Data` field and submits a `MsgTransfer` transaction triggers this. The entry path is a standard, supported production transaction. No special privileges or configuration are required.

### Recommendation

1. Add `token_data` (repeated string, one per token) and `class_data` (string) fields to the `NonFungibleTokenPacketData` proto definition in `proto/chainmain/nft_transfer/v1/packet.proto`.
2. In `createOutgoingPacket` (`x/nft-transfer/keeper/packet.go`), collect `nft.GetData()` alongside `nft.GetURI()` and include it in the packet.
3. In `processReceivedPacket`, pass `data.TokenData[i]` (and `data.ClassData`) to `MintNFT` (and `IssueDenom`) instead of the hardcoded `""`.
4. Update `NewNonFungibleTokenPacketData` and `ValidateBasic` accordingly.

### Proof of Concept

1. On chain A, mint an NFT with non-empty `Data`:
   ```
   chain-maind tx nft mint <denomID> <tokenID> --data='{"trait":"rare"}' --from=<owner>
   ```
2. Transfer it cross-chain via IBC:
   ```
   chain-maind tx nft-transfer transfer nft-transfer <channel> <denomID> <tokenID> <receiver-on-B> --from=<owner>
   ```
3. On chain B, query the received NFT:
   ```
   chain-maind query nft token <ibc/...classID> <tokenID>
   ```
   The `data` field will be `""` — the on-chain metadata is permanently lost.

The root cause is confirmed at:
- `x/nft-transfer/keeper/packet.go` line 99: `nft.GetData()` is never read
- `x/nft-transfer/keeper/packet.go` line 120–122: packet built without tokenData
- `x/nft-transfer/keeper/packet.go` line 182: `MintNFT` called with `""` for tokenData
- `proto/chainmain/nft_transfer/v1/packet.proto` lines 9–22: struct has no `token_data` field
