### Title
Missing `ENABLE_SECP_OPS` Guard on 4-Byte Secp Opcode Dispatch Path — (`File: src/chia_dialect.rs`)

### Summary

The `ChiaDialect::op()` function in `src/chia_dialect.rs` dispatches `op_secp256k1_verify` and `op_secp256r1_verify` via their 4-byte opcodes (`0x13d61f00`, `0x1c3a8f00`) **without** checking the `ENABLE_SECP_OPS` flag. The same operators dispatched via their 1-byte opcodes (64, 65) are correctly gated behind `flags.contains(ClvmFlags::ENABLE_SECP_OPS)`. This asymmetry means an attacker-controlled CLVM program can invoke secp signature verification using the 4-byte encoding even when the dialect is configured to disallow it — including in `MEMPOOL_MODE`.

---

### Finding Description

`ChiaDialect::op()` has two separate dispatch paths for operators:

**4-byte opcode path (lines 157–183) — no `ENABLE_SECP_OPS` check:**

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,   // no flag check
        0x1c3a8f00 => op_secp256r1_verify,   // no flag check
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
``` [1](#0-0) 

**1-byte opcode path (lines 248–249) — correctly gated:**

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The `ENABLE_SECP_OPS` flag is documented as: *"Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify)."* [3](#0-2) 

The guard is applied to the 1-byte path but is entirely absent from the 4-byte path, which dispatches the same underlying functions unconditionally.

---

### Impact Explanation

`MEMPOOL_MODE` — the stricter validation mode used by nodes before accepting transactions — includes `NO_UNKNOWN_OPS` but **does not** include `ENABLE_SECP_OPS`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [4](#0-3) 

In `MEMPOOL_MODE`:
- A CLVM program using 1-byte opcode `64` or `65` → falls through to `unknown_operator` → rejected (because `NO_UNKNOWN_OPS` is set).
- The **same** program using 4-byte opcode `0x13d61f00` or `0x1c3a8f00` → dispatches directly to `op_secp256k1_verify` / `op_secp256r1_verify` → **executes successfully**.

This is a **consensus/mempool divergence**: a transaction that should be rejected by the mempool (secp ops not enabled) is accepted when the attacker uses the 4-byte encoding. Nodes running different software versions or configurations that differ in whether they treat the 4-byte path as guarded will disagree on transaction validity, which is a consensus-critical bug.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: any CLVM program submitted to the mempool can embed a 4-byte atom `0x13d61f00` or `0x1c3a8f00` as an operator. No special privileges, social engineering, or compromised infrastructure are required. The 4-byte encoding is a valid, parseable CLVM atom. The divergence is deterministic and reproducible.

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch path, mirroring the guard already present on the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures both encodings of the secp operators are subject to the same flag guard, eliminating the asymmetry.

---

### Proof of Concept

1. Construct a CLVM program that uses the 4-byte atom `0x13d61f00` as an operator with valid secp256k1 arguments.
2. Execute it via `run_program` with a `ChiaDialect` configured as `MEMPOOL_MODE` (which does not include `ENABLE_SECP_OPS`).
3. Observe that `op_secp256k1_verify` executes and returns a result — no `Unimplemented` error is raised.
4. Repeat with 1-byte opcode `64` under the same `MEMPOOL_MODE` dialect.
5. Observe that step 4 falls through to `unknown_operator` and returns `EvalErr::Unimplemented` due to `NO_UNKNOWN_OPS`.

The two encodings of the same operation produce different outcomes under the same dialect configuration, confirming the missing guard. [5](#0-4) [2](#0-1)
