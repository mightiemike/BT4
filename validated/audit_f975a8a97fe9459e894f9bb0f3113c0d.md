### Title
`VersionedConstants::latest_constants()` Used for Fee-Proposal Bounds and Window Pruning Instead of Block-Version-Bound Constants — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`, `crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs`)

### Summary

The consensus orchestrator reads `fee_proposal_margin_ppt` and `fee_proposal_window_size` from `VersionedConstants::latest_constants()` at the moment of validation and window management, rather than binding them to the block's Starknet version. This is the sequencer-native analog of the external report's "global variable mutation during active draw": a versioned-constant parameter that governs fee-proposal acceptance bounds and the sliding-window size is resolved at runtime from a mutable global (the "latest" binary version), not from the version that was in effect when the proposal was created or when the block's Starknet version was fixed. When a future protocol version changes either constant, nodes on different binary versions will compute divergent `fee_actual` values and divergent acceptance bounds for the same block height, causing valid proposals to be rejected or invalid proposals to be accepted.

### Finding Description

**Root cause — `fee_proposal_margin_ppt` read from `latest_constants()` at validation time**

Inside `is_proposal_init_valid()`, the acceptance band for a proposer's `fee_proposal_fri` is computed as:

```rust
let (lower_bound, upper_bound) = fee_proposal_bounds(
    fee_actual,
    VersionedConstants::latest_constants().fee_proposal_margin_ppt,
);
``` [1](#0-0) 

The proposer computes its own `fee_proposal` using the same call:

```rust
let proposal = compute_fee_proposal(
    fee_target,
    fee_actual,
    VersionedConstants::latest_constants().fee_proposal_margin_ppt,
);
``` [2](#0-1) 

Neither call is anchored to the block's `starknet_version`; both resolve the margin from the binary's compile-time "latest" constant.

**Root cause — `fee_proposal_window_size` read from `latest_constants()` in three independent call sites**

`prune_fee_proposals_window()` uses `latest_constants()` to decide which historical entries to discard:

```rust

self.fee_proposals_window = self.fee_proposals_window.split_off(&cutoff);
``` [3](#0-2) 

`initialize_fee_proposals_window()` uses `latest_constants()` to decide how many historical blocks to load from state-sync:

```rust

let window_start_height = window_end_height.saturating_sub(window_size);

If the binary is upgraded between queuing and dequeuing, the constants used for validation differ from those used by the proposer.

### Impact Explanation

When a future protocol version changes `fee_proposal_window_size` (e.g., from 10 to 20):

- A node on the old binary prunes the window to 10 entries and computes `fee_actual` as the median of 10 values.
- A node on the new binary prunes to 20 entries and computes `fee_actual` as the median of 20 values.
- The two medians diverge.
- The proposer (new binary) clamps `fee_proposal` to ±`margin_ppt` of its `fee_actual`.
- The validator (old binary) checks the same `fee_proposal` against ±`margin_ppt` of its different `fee_actual`.
- The proposal is rejected, causing a consensus liveness failure.

When `fee_proposal_margin_ppt` changes (e.g., from 2 to 5):

- The proposer (new binary) publishes a `fee_proposal` within ±5 ppt of `fee_actual`.
- The validator (old binary) enforces ±2 ppt.
- A proposal that is valid under the new margin is rejected by the old validator.

Both scenarios produce a wrong `next_l2_gas_price` in the committed block header (either through liveness failure or through acceptance of an out-of-band value), matching the "Wrong state … or revert result" and "Incorrect fee … resource accounting … balance" impact categories.

### Likelihood Explanation

The trigger is unprivileged: any rolling upgrade of sequencer nodes to a binary that introduces a new versioned-constants file with a changed `fee_proposal_window_size` or `fee_proposal_margin_ppt` activates the divergence. No special permissions or malicious input are required. The current constants are identical across all five deployed versions, so the issue is latent but structurally present and will activate on the next protocol upgrade that modifies either field.

### Recommendation

Bind `fee_proposal_window_size` and `fee_proposal_margin_ppt` to the block's Starknet version rather than resolving them from `latest_constants()` at call time. Concretely:

1. In `prune_fee_proposals_window()`, `initialize_fee_proposals_window()`, and every `compute_fee_actual()` call site, pass the `window_size` derived from `VersionedConstants::get(block_starknet_version).fee_proposal_window_size`.
2. In `is_proposal_init_valid()` and `compute_proposer_fee_proposal()`, derive `fee_proposal_margin_ppt` from the same version-specific constants object, not from `latest_constants()`.
3. Store the resolved `window_size` and `margin_ppt` in `ProposalInitValidation` so that the spawned validation task uses the same values that were in effect when the proposal was created.

### Proof of Concept

The divergence is mechanically demonstrable by constructing two `SequencerConsensusContext` instances that differ only in their binary's `latest_constants()` value (achievable in tests by temporarily patching the JSON resource files):

1. Bootstrap both contexts with 10 identical `fee_proposal` entries (heights 0–9).
2. On context A (old binary, `window_size = 10`): call `compute_fee_actual` at height 10 → returns median of entries 0–9.
3. On context B (new binary, `window_size = 20`): call `compute_fee_actual` at height 10 → returns `None` (window incomplete), triggering the `l2_gas_price` fallback.
4. Context A proposes `fee_proposal = clamp(fee_target, fee_actual_A ± 2ppt)`.
5. Context B validates the proposal against `l2_gas_price` (the fallback), which differs from `fee_actual_A`.
6. `is_proposal_init_valid()` rejects the proposal with "Fee proposal out of bounds". [13](#0-12) [14](#0-13)

```rust
pub fn compute_fee_actual(
    fee_proposals_window: &BTreeMap<BlockNumber, Option<GasPrice>>,
    height: BlockNumber,
    window_size: u64,
) -> Option<GasPrice> {
    let Some(start) = height.0.checked_sub(window_size) else {
        warn!(
            "Cannot compute fee_actual for height {height}: height is below window_size \
             ({window_size})"
        );
        return None;
    };
    let window_size_usize = usize::try_from(window_size).expect("window_size fits in usize");
    let mut window = Vec::with_capacity(window_size_usize);
    for source_height in (start..height.0).map(BlockNumber) {
        match fee_proposals_window.get(&source_height) {
            Some(Some(price)) => window.push(*price),
            Some(None) | None => {
                warn!(
                    "Cannot compute fee_actual for height {height}: fee_proposals_window has no \
                     recorded fee_proposal for height {source_height}"
                );
                return None;
            }
        }
    }
    window.sort();
    let mid = window_size_usize / 2;
    let median = if window_size_usize.is_multiple_of(2) {
        // Even: average of the two middle values, rounded down.
        // Overflow-safe averaging: a + (b - a) / 2 (safe because sorted, so b >= a).
        GasPrice(window[mid - 1].0 + (window[mid].0 - window[mid - 1].0) / 2)
    } else {
        window[mid]
    };
    Some(median)
}
