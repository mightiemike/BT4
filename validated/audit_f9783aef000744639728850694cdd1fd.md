Audit Report

## Title
`executeOracleStopLossDecay` Retroactively Applies New Decay Rate to Elapsed Period Without Settling Per-Bin Watermarks — (`File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`executeOracleStopLossDecay` writes a new `decayPerSecondE8` into `oracleStopLossConfig` without advancing `lastDecayTs` or pre-settling per-bin `highWatermarks` under the old rate. Because `_checkAndUpdateWatermarks` lazily computes `dt = block.timestamp - hwmS.lastDecayTs` and multiplies the full interval by the current (new) rate, the new rate retroactively covers the entire gap since the last swap. This causes the effective stop-loss floor to be either too low (rate increased) — silently permitting value extraction past the drawdown threshold — or too high (rate decreased) — reverting legitimate swaps and breaking pool usability.

## Finding Description

The lazy-decay model stores a `lastDecayTs` per bin and applies decay on-demand at swap time: [1](#0-0) 

The `decayRate` used is read from `cfg.decayPerSecondE8` at the moment of the swap: [2](#0-1) 

`_decayed` applies a single rate to the full `dt`: [3](#0-2) 

When the admin executes a decay update, only the config field is written — no bin watermarks are touched and `lastDecayTs` is not advanced: [4](#0-3) 

At the next swap, `dt` spans the entire gap `T_swap − T_last`, but the full interval is multiplied by `rate_new`. The correct piecewise computation would be:

```
decayed_correct = hwm − hwm × (rate_old × (T_update − T_last) + rate_new × (T_swap − T_update)) / E8
decayed_actual  = hwm − hwm × (rate_new × (T_swap − T_last)) / E8
```

No existing guard compensates for this: `_validateDecay` only enforces `decayPerSecondE8 <= E8`, and the timelock only delays when the new rate takes effect, not whether retroactive application occurs. [5](#0-4) 

## Impact Explanation

**LP principal at risk (rate increased):** Over-decay drives the effective HWM toward zero, collapsing the stop-loss floor (`hwm × floorMultiplier / E6`) below the intended `drawdownE6` threshold. Swaps that drain bin value past the intended drawdown limit are not reverted. In the extreme PoC case (`rate_new = 1_000`, `dt = 86_401`), `factor = 86_401_000 ≥ 1e8`, so `_decayed` returns `0` and the floor becomes `0` — the guard is completely disabled for the bin, allowing unlimited value extraction from LPs. [6](#0-5) 

**Pool functionality broken (rate decreased):** Under-decay leaves the effective HWM higher than it should be, raising the floor above the intended threshold. Legitimate swaps revert with `OracleStopLossTriggered` even though no actual value loss occurred, rendering the pool unusable until a new high watermark is naturally set. [7](#0-6) 

Both outcomes are direct consequences of a legitimate, timelocked administrative operation, constituting a fund-impacting cap failure of the stop-loss drawdown protection.

## Likelihood Explanation

Every decay-rate update triggers the bug. The timelock only delays execution; it does not mitigate retroactive application. The discrepancy is maximized during quiet periods (no swaps between the last watermark update and the rate change execution), which is a realistic scenario for pools with infrequent trading or admins who update parameters during low-activity windows. No attacker capability beyond being a pool admin executing a routine parameter update is required.

## Recommendation

Before writing the new `decayPerSecondE8`, the accumulated decay under the old rate must be settled. The cleanest fix is to store a rate-change checkpoint `(rate, effectiveFrom)` and compute piecewise decay in `_decayed`:

```solidity
// Two-segment decay: old rate × (T_checkpoint − lastDecayTs) + new rate × (T_now − T_checkpoint)
```

A simpler alternative: store a global `rateChangeEpoch` and `previousDecayPerSecondE8` alongside the new rate in `oracleStopLossConfig`, and split the interval at that epoch inside `_decayed`. This mirrors the correct fix for the analogous Predy `updateIRMParams` bug: settle the current period under the old parameters before switching to the new ones.

## Proof of Concept

1. Pool initialized: `drawdownE6 = 50_000` (5%), `decayPerSecondE8 = 10`.
2. At `T=0`, a swap occurs. `highWatermarks[pool][0] = {token0: 1_000_000, token1: 1_000_000, lastDecayTs: 0}`.
3. Admin proposes and executes `decayPerSecondE8 = 1_000` at `T = 86_400`. No swap occurs in between.
4. At `T = 86_401`, a swap occurs:
   - `dt = 86_401`, `factor = 1_000 × 86_401 = 86_401_000 ≥ 1e8`
   - `_decayed` returns `0` for both token0 and token1 watermarks
   - Floor = `0 × 950_000 / 1_000_000 = 0`
   - `_applyWatermark` never sets `breached = true` (since `metric >= 0` always)
   - Stop-loss guard is completely disabled; any value-draining swap passes through [8](#0-7)

```text
  function _decayed(uint256 hwm, uint256 ratePerSecondE8, uint256 dt) private pure returns (uint256) {
    if (ratePerSecondE8 == 0 || dt == 0 || hwm == 0) return hwm;
    uint256 factor = ratePerSecondE8 * dt;
    if (factor >= E8) return 0;
    return hwm - (hwm * factor) / E8;
  }
```
