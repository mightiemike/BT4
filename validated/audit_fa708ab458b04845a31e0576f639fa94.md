Audit Report

## Title
Stop-Loss Guard Evaluates Per-Bin Metrics at Arithmetic Mid-Price Instead of Geometric Mid-Price, Misaligning Protection Boundary with Actual Swap Settlement - (File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol)

## Summary
`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes mid-price as the arithmetic mean `(bid + ask) / 2`, while the pool's swap settlement path and `PriceVelocityGuardExtension` both derive mid-price via `SwapMath.midAndSpreadFeeX64FromBidAsk`, which computes the geometric mean `sqrt(bid * ask)`. By AM-GM inequality, the arithmetic mean strictly exceeds the geometric mean whenever `bid ŌēĀ ask`, which is the normal case after asymmetric `stepBidFactor`/`stepAskFactor` adjustments. This causes the stop-loss to evaluate per-bin metrics at a price that does not match the price the swap actually settled at, producing false negatives for `!zeroForOne` swaps: the guard fails to block token0-in/token1-out swaps that have genuinely breached the drawdown floor, allowing LP value to leak past the configured protection boundary.

## Finding Description

In `_afterSwapOracleStopLoss`, the mid-price is computed as:

```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
``` [1](#0-0) 

This arithmetic mean is then fed into `_metrics` to compute per-bin value-per-share:

```solidity
metricT0 = _clampMetric(t0ps + Math.mulDiv(Math.mulDiv(uint256(t1), Q64, midPriceX64), METRIC_SCALE, shares));
metricT1 = _clampMetric(Math.mulDiv(Math.mulDiv(uint256(t0), midPriceX64, Q64), METRIC_SCALE, shares) + t1ps);
``` [2](#0-1) 

In contrast, `SwapMath.midAndSpreadFeeX64FromBidAsk` ŌĆö used by both the pool's swap settlement and the velocity guard ŌĆö computes the **geometric** mean:

```solidity
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);
``` [3](#0-2) 

`MetricOmmPool.swap` uses this geometric mid for all swap settlement: [4](#0-3) 

`PriceVelocityGuardExtension.beforeSwap` also uses the geometric mid: [5](#0-4) 

The oracle price providers apply asymmetric step factors: [6](#0-5) 

After these adjustments, `bid_out` and `ask_out` are not symmetric around the oracle mid. By AM-GM inequality, `(bid + ask) / 2 > sqrt(bid * ask)` strictly whenever `bid ŌēĀ ask`. Therefore the stop-loss evaluates bin metrics at a mid-price that is strictly higher than the price the swap settled at.

The breach check for `!zeroForOne` swaps: [7](#0-6) 

Because `metricToken1 ŌłØ t0 * mid / Q64 + t1` increases with mid, using the inflated arithmetic mean produces an overstated `metricT1`. The breach condition `metricT1 < hwm * floorMultiplier / E6` is therefore harder to satisfy than it should be, causing the stop-loss to miss genuine drawdown breaches in the `!zeroForOne` direction.

## Impact Explanation

This is a broken oracle guard path with direct fund impact. When a pool is configured with `OracleValueStopLossExtension` and a price provider with `marginStep > 0`, the stop-loss systematically understates the severity of `!zeroForOne` swaps (token0 in, token1 out). Swaps that have genuinely pushed per-bin `metricToken1` below the drawdown floor are not reverted, allowing LP value to leak past the configured protection boundary. The magnitude of the discrepancy grows with spread width and `|marginStep|`, so pools with wider configured spreads suffer greater exposure. This meets the "broken core pool functionality causing loss of funds" impact gate.

## Likelihood Explanation

Any pool deploying `OracleValueStopLossExtension` with a non-zero `marginStep` price provider is affected. The trigger is a normal public swap in the `!zeroForOne` direction when per-bin metrics are near the drawdown floor. No privileged action is required; any unprivileged swapper can execute the swap that should have been blocked. The condition is repeatable and worsens as spread width increases.

## Recommendation

Replace the arithmetic average in `_afterSwapOracleStopLoss` with the same `SwapMath.midAndSpreadFeeX64FromBidAsk` call used by the pool and the velocity guard:

```solidity
// Before (wrong):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (correct):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
```

This ensures the stop-loss evaluates per-bin metrics at exactly the same geometric mid-price the swap settlement used, keeping the protection boundary aligned with the actual economic impact.

## Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension` and a price provider configured with `marginStep > 0` (e.g., `marginStep = 50e15`, 5% step), producing `bid_out < ask_out` with `(bid_out + ask_out) / 2 > sqrt(bid_out * ask_out)`.
2. Set `drawdownE6 = 50_000` (5% drawdown floor) and seed a bin with known `t0`, `t1`, `totalShares`.
3. Set the high watermark to the current `metricToken1` value computed at the geometric SwapMath mid (the correct price).
4. Execute a `!zeroForOne` swap. The pool settles at `SwapMath.midAndSpreadFeeX64FromBidAsk(bid, ask)` (geometric mid).
5. Observe that the stop-loss computes `metricToken1` at `(bid + ask) / 2 > SwapMath mid`, producing an overstated metric that does not breach the floor, so the swap is not reverted.
6. Verify that if the same metric were computed at the SwapMath geometric mid, it would fall below `hwm * (E6 - drawdownE6) / E6` and the swap would correctly revert with `OracleStopLossTriggered`.
