### Title
`ResourceBoundsMapping`→`ValidResourceBounds` zero-check silently selects `L1Gas` hash domain for V3 transactions, causing `simulateTransactions`/`estimateFee` to compute a divergent transaction hash — (`File: crates/apollo_rpc/src/v0_8/transaction.rs`)

---

### Summary

The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion in the RPC layer silently downgrades a V3 transaction to the `L1Gas` hash domain whenever `l1_data_gas = 0` and `l2_gas = 0`. The gateway write path never passes through this conversion — it always produces `ValidResourceBounds::AllResources`. The result is that `starknet_simulateTransactions` and `starknet_estimateFee` compute a structurally different transaction hash (missing the `L1_DATA_GAS` field in the Poseidon preimage) than the hash that will be committed when the same transaction is actually sequenced. Any contract that reads its own hash via `get_execution_info` will observe a different value during simulation than during execution.

---

### Finding Description

**Asymmetric hash-domain selection (the invariant break)**

`get_tip_resource_bounds_hash` branches on the `ValidResourceBounds` variant:

```
ValidResourceBounds::L1Gas(_)          → hash = Poseidon(tip, L1_GAS_concat, L2_GAS_concat)
ValidResourceBounds::AllResources(...) → hash = Poseidon(tip, L1_GAS_concat, L2_GAS_concat, L1_DATA_GAS_concat)
``` [1](#0-0) 

The two outputs are cryptographically distinct even when `l1_data_gas = 0`.

**Gateway write path — always `AllResources`**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds` (not `ValidResourceBounds`). The converter wraps it unconditionally:

```rust
ValidResourceBounds::AllResources(self.resource_bounds)
``` [2](#0-1) 

So every transaction admitted through the gateway is hashed with the `AllResources` branch, including `L1_DATA_GAS`.

**RPC execution path — zero-check silently picks `L1Gas`**

`BroadcastedTransaction` for `estimateFee`/`simulateTransactions` carries `ResourceBoundsMapping`. The conversion to `ValidResourceBounds` is:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)          // ← L1_DATA_GAS excluded from hash
        } else {
            Self::AllResources(...)
        }
    }
}
``` [3](#0-2) 

The same zero-check exists in the protobuf deserializer: [4](#0-3) 

When a V3 transaction with `l1_data_gas = 0` and `l2_gas = 0` is passed to `simulate_transactions`, the `tx.try_into()` call produces a `ValidResourceBounds::L1Gas`, and `calc_tx_hash` → `get_transaction_hash` → `get_invoke_transaction_v3_hash` hashes it **without** `L1_DATA_GAS`. [5](#0-4) [6](#0-5) 

**Concrete divergence**

| Path | `resource_bounds` type | `ValidResourceBounds` variant | L1_DATA_GAS in hash? |
|---|---|---|---|
| `addInvokeTransaction` (gateway) | `AllResourceBounds` | `AllResources` | **Yes** (zero) |
| `simulateTransactions` / `estimateFee` | `ResourceBoundsMapping` (l1_data_gas=0, l2_gas=0) | `L1Gas` | **No** |

The two hashes differ. The `get_execution_info` syscall inside the simulated contract will return the `L1Gas`-domain hash; the same contract executed on-chain will see the `AllResources`-domain hash.

---

### Impact Explanation

`starknet_simulateTransactions` and `starknet_estimateFee` return an authoritative-looking but wrong transaction hash inside the execution trace for any V3 transaction whose `l1_data_gas` and `l2_gas` bounds are both zero. A contract that reads `get_execution_info().tx_info.transaction_hash` and uses it for any purpose (replay protection, commitment, event emission) will observe a different value during simulation than during actual sequencing. This breaks the simulation guarantee and can cause contracts to behave differently in simulation vs. execution — a silent, hard-to-diagnose correctness failure.

---

### Likelihood Explanation

V3 transactions with zero `l1_data_gas` and `l2_gas` are common: client-side proving transactions explicitly set all prices to zero (as validated by `validate_zero_fee_resource_bounds`), and any pre-0.13.3-style V3 transaction has the same shape. The trigger requires only a normal unprivileged RPC call to `starknet_simulateTransactions` or `starknet_estimateFee`.

---

### Recommendation

Remove the zero-check from `From<ResourceBoundsMapping> for ValidResourceBounds` and always produce `AllResources` for V3 transactions, matching the gateway path:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        Self::AllResources(AllResourceBounds {
            l1_gas: value.l1_gas,
            l1_data_gas: value.l1_data_gas,
            l2_gas: value.l2_gas,
        })
    }
}
```

Apply the same fix to `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. The `L1Gas` variant should only be constructed when deserializing transactions that were originally committed under the pre-0.13.3 hash scheme (i.e., when reading historical blocks from storage), not when accepting user-supplied broadcast transactions.

---

### Proof of Concept

1. Construct a V3 invoke transaction `T` with `l1_data_gas = {max_amount: 0, max_price_per_unit: 0}` and `l2_gas = {max_amount: 0, max_price_per_unit: 0}`.

2. Call `starknet_simulateTransactions` with `T`. Inside the execution trace, the contract's `get_execution_info` syscall returns hash `H_sim` computed via `get_invoke_transaction_v3_hash` over `ValidResourceBounds::L1Gas` — **without** `L1_DATA_GAS` in the Poseidon preimage.

3. Submit the same transaction `T` via `starknet_addInvokeTransaction`. The gateway converts `AllResourceBounds` → `ValidResourceBounds::AllResources` and computes hash `H_exec` — **with** `L1_DATA_GAS = 0` in the Poseidon preimage.

4. `H_sim ≠ H_exec`. Any contract logic that branches on `get_execution_info().tx_info.transaction_hash` will diverge between simulation and execution.

The divergence is rooted in `get_tip_resource_bounds_hash` producing different-length Poseidon inputs for the two variants, confirmed by the branch at line 203–208 of `crates/starknet_api/src/transaction_hash.rs`. [7](#0-6) [3](#0-2) [8](#0-7)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
