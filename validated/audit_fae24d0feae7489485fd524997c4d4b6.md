Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Gates on Router Address Instead of End-User, Enabling Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of the `pool.swap()` call. When a user swaps via `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the hook checks the router's allowlist status rather than the end-user's. If the router is allowlisted (the only way to let allowlisted users use the router), any unprivileged user can bypass the swap allowlist entirely by routing through the public router contract.

## Finding Description

`SwapAllowlistExtension.beforeSwap` enforces the allowlist at line 37:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool and `sender` is the address forwarded by `ExtensionCalling._beforeSwap`:

```solidity
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, ...)
  )
);
``` [2](#0-1) 

That `sender` is whatever address called `pool.swap()`. In `MetricOmmSimpleRouter.exactInputSingle`, the router calls:

```solidity
IMetricOmmPoolActions(params.pool).swap(
    params.recipient, params.zeroForOne, ...
);
``` [3](#0-2) 

So `msg.sender` to the pool is the router contract. The hook receives `sender = router`, not the originating user. The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [4](#0-3) 

Two broken outcomes follow:

1. **Allowlisted users blocked via router**: The router is not on the allowlist, so every router-mediated swap reverts even for allowlisted users — core swap functionality is broken for the intended user path.
2. **Allowlist bypass**: To fix (1), the pool admin adds the router to the allowlist. Now `allowedSwapper[pool][router] = true`, and **any** unprivileged user can bypass the allowlist by calling the router. The configured security boundary is silently inoperative.

`_callExtensionsInOrder` has no mechanism to recover the original user — it only forwards the `sender` it receives: [5](#0-4) 

## Impact Explanation

The pool admin configures `SwapAllowlistExtension` to restrict swaps to a specific set of addresses (e.g., KYC-gated or institutional-only pools). The allowlist is bypassed by any unprivileged user via the public router — an admin-boundary break via an unprivileged path. Additionally, if the router is not allowlisted, allowlisted users cannot use the standard swap entry point, breaking core pool swap functionality. Both outcomes fall within the allowed impact gate.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard, documented user-facing swap entry point — any user can call it without any special privilege. The bypass requires only that the router be allowlisted, which is the natural operational step any pool admin must take to allow their allowlisted users to use the router. No malicious tokens, special setup, or privileged access is required. The trigger is a normal `exactInputSingle` or `exactInput` call through the public router.

## Recommendation

`SwapAllowlistExtension.beforeSwap` must gate on the **economically relevant actor**, not the intermediary dispatcher. Options:

1. **Trusted-router pattern**: Maintain a registry of trusted routers in the extension. When `sender` is a trusted router, decode the real user from `extensionData`; otherwise use `sender` directly.
2. **Router encodes originator**: Require the router to encode `msg.sender` into `extensionData`, and have the extension decode and verify it when `sender` is a known router.
3. **Direct-only enforcement**: Document that allowlisted pools must not be used via the router, and enforce this by checking `sender` is an EOA or known non-router address — though this is fragile.

The trusted-router pattern (option 1) is the most robust: the extension checks `allowedSwapper[pool][realUser]` where `realUser` is extracted from `extensionData` when `sender` is a trusted router, and `sender` otherwise.

## Proof of Concept

```
1. Pool P is deployed with SwapAllowlistExtension on beforeSwap.
2. Admin allowlists userA: allowedSwapper[P][userA] = true.
3. Admin allowlists the router so userA can use it:
       allowedSwapper[P][router] = true.
4. userB (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle(
       pool=P, recipient=userB, ...
   ).
5. Router calls P.swap(recipient=userB, ...) with msg.sender = router.
6. ExtensionCalling._beforeSwap passes sender=router to SwapAllowlistExtension.
7. SwapAllowlistExtension checks allowedSwapper[P][router] == true → PASSES.
8. userB's swap executes successfully despite not being on the allowlist.
9. The configured security boundary is silently bypassed.
```

A Foundry integration test can confirm this by:
- Deploying a pool with `SwapAllowlistExtension`
- Allowlisting `userA` and the router
- Calling `exactInputSingle` from `userB` (not allowlisted)
- Asserting the swap succeeds (demonstrating the bypass)
