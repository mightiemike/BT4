### Title
Deprecated `verify_transaction_inclusion` Remains Callable Without Coinbase Proof Validation, Enabling Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function is still a live, publicly callable NEAR entry point. It lacks the coinbase merkle proof validation that `verify_transaction_inclusion_v2` introduced to close the 64-byte transaction forgery vulnerability. Any unprivileged NEAR caller can supply an internal Merkle tree node hash as `tx_id` and a shortened proof to make the function return `true` for a Bitcoin transaction that does not exist.

---

### Finding Description

`verify_transaction_inclusion` verifies transaction inclusion by computing a Merkle root from the caller-supplied proof and comparing it to the stored block's `merkle_root`: [1](#0-0) 

The only structural guard on the proof is: [2](#0-1) 

There is no check that `tx_id` is a leaf node (a real transaction hash) rather than an internal node of the Merkle tree. `compute_root_from_merkle_proof` is purely positional: [3](#0-2) 

Because the function only checks that the computed root equals the stored `merkle_root`, an attacker can supply an internal node hash as `tx_id` together with a proof that starts one level above that node. The computed root will still match, and the function returns `true`.

`verify_transaction_inclusion_v2` was created specifically to close this gap by requiring a coinbase proof that anchors the tree at its true leaf level: [4](#0-3) 

However, v1 is still decorated with `#[pause]` (not disabled) and remains a public method, so any caller can bypass v2's coinbase check entirely by calling v1 directly. [5](#0-4) 

---

### Impact Explanation

Any NEAR smart contract (e.g., a Bitcoin bridge) that calls `verify_transaction_inclusion` to gate fund releases or other critical operations can be tricked into accepting a forged proof. The attacker receives a `true` result for a Bitcoin transaction that was never broadcast or confirmed, enabling them to claim funds or trigger actions that depend on that verification. This is the direct analog of the external report's missing slippage protection: just as setting `sqrtPriceLimitX96` to MAX/MIN removes all price protection and lets a sandwich attacker drain funds, the absence of coinbase proof validation removes all leaf-node protection and lets a proof forger trigger false positive verifications.

---

### Likelihood Explanation

The attack requires only:
1. Knowledge of any mainchain block's Merkle tree structure — entirely public information derivable from Bitcoin RPC.
2. The ability to call a NEAR contract function — available to any unprivileged NEAR account.

No privileged role, leaked key, or social engineering is needed. The deprecated function is active by default.

---

### Recommendation

Disable `verify_transaction_inclusion` (v1) entirely by replacing its body with `env::panic_str("use verify_transaction_inclusion_v2")`, or remove it from the public ABI. Do not rely on the `#[deprecated]` Rust attribute alone — it is a compiler hint, not an access control mechanism. All callers must be migrated to `verify_transaction_inclusion_v2`.

---

### Proof of Concept

Given a mainchain block with four transactions `[A, B, C, D]` and Merkle tree:

```
root = hash(hash(A,B), hash(C,D))
AB   = hash(A, B)          ← internal node, not a real transaction
CD   = hash(C, D)
```

Attacker calls `verify_transaction_inclusion` with:
- `tx_id`              = `AB`  (internal node hash, no real transaction)
- `tx_index`           = `0`
- `merkle_proof`       = `[CD]`
- `tx_block_blockhash` = any valid mainchain block hash with this tree
- `confirmations`      = `1`

`compute_root_from_merkle_proof(AB, 0, [CD])`:
- position 0 is even → `hash(AB, CD)` = `root` ✓

The function returns `true` for a transaction that does not exist on Bitcoin.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
