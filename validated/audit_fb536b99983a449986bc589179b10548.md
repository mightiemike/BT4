### Title
Silent Permanent Drop of Queued Transactions on Full Network Buffer in `MempoolP2pPropagator::broadcast_queued_transactions` — (`crates/apollo_mempool_p2p/src/propagator/mod.rs`)

---

### Summary

`MempoolP2pPropagator::broadcast_queued_transactions` unconditionally drains the entire `transaction_queue` before attempting the network send. When the underlying channel is full (`err.is_full()`), the code logs a warning and returns `Ok(())` — permanently discarding every transaction that was in the queue with no retry and no re-queuing. This is the sequencer-native analog of the VUSD `processWithdrawals` pattern: a failed operation is silently swallowed, and the affected items are irrecoverably lost.

---

### Finding Description

In `broadcast_queued_transactions`:

```rust
async fn broadcast_queued_transactions(&mut self) -> MempoolP2pPropagatorResult<()> {
    let queued_transactions: Vec<RpcTransaction> = self.transaction_queue.drain(..).collect(); // ← drains unconditionally
    ...
    let result = self
        .broadcast_topic_client
        .broadcast_message(RpcTransactionBatch(queued_transactions))
        .await
        .or_else(|err| {
            if !err.is_full() {
                warn!("Error broadcasting transaction batch: {:?}", err);
                return Err(MempoolP2pPropagatorError::NetworkSendError);
            }
            warn!("... buffer is full. Dropping the transaction batch."); // ← silent drop
            Ok(())  // ← returns success, no re-queue
        });
    ...
    result
}
``` [1](#0-0) 

The invariant broken is: **a transaction removed from the propagation queue must either be successfully broadcast or be restored to the queue for retry**. The code violates this by calling `.drain(..)` before the send attempt, so when `is_full()` triggers the `Ok(())` path, the transactions are gone from `transaction_queue` and will never be re-attempted.

The caller in `MempoolCommunicationWrapper::add_tx` compounds this by also silently ignoring the `P2pPropagatorClientError` that `send_tx_to_p2p` can return:

```rust
if let Err(p2p_client_err) =
    self.send_tx_to_p2p(args_wrapper.p2p_message_metadata, args_wrapper.args.tx).await
{
    warn!("Failed to send transaction to P2P: {:?}", p2p_client_err);
}
Ok(())
``` [2](#0-1) 

The `broadcast_queued_transactions` path is triggered both on `max_transaction_batch_size` fill and on the periodic timer tick in `MempoolP2pRunner`: [3](#0-2) 

---

### Impact Explanation

Valid transactions that have been accepted into the local mempool and enqueued for P2P propagation are permanently dropped from the propagation queue whenever the network send buffer is full. Those transactions will never be broadcast to peer nodes. Peers will not include them in their mempools, so the transactions will not be sequenced on any node other than the originating sequencer. Under sustained load (the exact condition that causes `is_full()`), this silently degrades to a state where the P2P mempool layer stops propagating transactions entirely while reporting no error.

**Matching impact**: *High — Mempool/gateway/RPC admission rejects valid transactions before sequencing* (transactions are effectively rejected from the network-wide sequencing pipeline).

---

### Likelihood Explanation

The `is_full()` branch is a normal operating condition under load — the network send buffer fills when the sequencer is processing a high volume of transactions. The `MempoolP2pRunner` fires `broadcast_queued_transactions` on every timer tick regardless of buffer state. No backpressure signal is propagated back to the queue, so the drop is guaranteed to occur silently during any sustained traffic spike.

---

### Recommendation

Move the `drain` to occur only after a confirmed successful send, or restore the drained transactions to the front of the queue on failure:

```rust
async fn broadcast_queued_transactions(&mut self) -> MempoolP2pPropagatorResult<()> {
    if self.transaction_queue.is_empty() {
        return Ok(());
    }
    // Clone for the attempt; only drain on success.
    let batch: Vec<RpcTransaction> = self.transaction_queue.clone();
    let n = batch.len().into_f64();
    match self.broadcast_topic_client.broadcast_message(RpcTransactionBatch(batch)).await {
        Ok(()) => {
            self.transaction_queue.clear();
            MEMPOOL_P2P_BROADCASTED_BATCH_SIZE.record(n);
            Ok(())
        }
        Err(err) if err.is_full() => {
            warn!("Network buffer full; retaining batch for next tick.");
            Ok(()) // queue is untouched — will retry on next tick
        }
        Err(err) => {
            warn!("Error broadcasting transaction batch: {:?}", err);
            Err(MempoolP2pPropagatorError::NetworkSendError)
        }
    }
}
```

---

### Proof of Concept

1. Configure `max_transaction_batch_size = N` and set the network send channel to capacity `< N`.
2. Submit `N` transactions through the gateway; each is accepted into the mempool and enqueued via `add_transaction` → `transaction_queue.push(...)`.
3. On the `N`-th transaction, `broadcast_queued_transactions` is called automatically.
4. `self.transaction_queue.drain(..)` removes all `N` transactions from the queue.
5. `broadcast_message` returns `Err(is_full = true)`.
6. The `or_else` closure returns `Ok(())`.
7. `transaction_queue` is now empty; the `N` transactions are permanently gone from the propagation layer.
8. Peer nodes never receive these transactions; they are not sequenced network-wide.

The existing test `add_tx_tolerates_p2p_propagation_error` in `crates/apollo_mempool/src/fee_mempool_test.rs` explicitly asserts that `add_tx` succeeds even when P2P propagation fails, confirming the silent-drop behavior is the current intended (but incorrect) contract. [4](#0-3)
