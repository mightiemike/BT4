### Title
Active Redelegation on Vesting-Owned Position Aborts v8 Upgrade Handler — (`x/tieredrewards/migrations/v2/migrate.go`)

### Summary
A vesting account owner with a pre-existing tier position can call `MsgTierRedelegate` (which has no vesting-account guard) before the v8 upgrade, creating an active staking redelegation entry for `pos.DelegatorAddress`. When the upgrade fires, `exitVestedAccountsPositions` calls `ForceFullExitWithDelegation` → `transferDelegationFromPosition`, which checks `isRedelegating(ctx, pos.DelegatorAddress)` and returns a hard `ErrActiveRedelegation`. This error propagates through `Migrate` → `Migrate1to2` → `RunMigrations` → the upgrade handler, halting the chain.

### Finding Description

**Entrypoint:** `MsgTierRedelegate` — no vesting-account check in `validateRedelegatePosition`. [1](#0-0) 

**Redelegation creation:** `TierRedelegate` calls `ms.redelegate(ctx, delAddr, srcValAddr, dstValAddr, pos.Delegation.Shares)` where `delAddr = pos.DelegatorAddress`. When the source validator is bonded, the SDK creates a real redelegation entry with a 21-day completion time. [2](#0-1) 

**Migration hard-error:** `transferDelegationFromPosition` checks `isRedelegating(ctx, pos.DelegatorAddress)` and returns `ErrActiveRedelegation` — not a skip, a hard error. [3](#0-2) 

**Error propagation — no recovery:** `exitVestedAccountsPositions` wraps and returns the error immediately; there is no `continue` or skip path. [4](#0-3) 

**Upgrade handler halts:** `RunMigrations` returns the error; the upgrade handler returns it, causing a chain halt. [6](#0-5) 

The `validateNonVestingAccount` guard is only wired into `validateNewPosition`, which is called by `MsgLockTier` and `MsgCommitDelegationToTier`. It is absent from `validateRedelegatePosition`, leaving the redelegate path open to vesting account owners who already hold positions. [7](#0-6) 

### Impact Explanation
The v8 upgrade handler halts. All funds held in the tieredrewards module (delegations under `pos.DelegatorAddress` for every position) remain locked in the module account. The chain cannot advance past the upgrade block until an emergency patch is deployed. This matches the "permanent lock of user funds" criterion for the duration of the outage, and requires out-of-band governance/validator coordination to recover.

### Likelihood Explanation
The attack requires only a single `MsgTierRedelegate` transaction from a vesting account owner with a pre-existing position, submitted within the 21-day unbonding window before the upgrade block. No special privileges, no governance access, no key compromise. The window is wide (21 days) and the transaction is cheap. Any single vesting account owner with a position can trigger this unilaterally.

### Recommendation
In `transferDelegationFromPosition`, replace the hard-error `isRedelegating` guard with a skip (return success with zero amounts) when called from the migration context, **or** add a dedicated migration-time path in `ForceFullExitWithDelegation` that waits for / ignores active redelegations. The simplest fix is to remove the `isRedelegating` check from `transferDelegationFromPosition` entirely for the force-exit path (the check is labeled "Defensive" and is not needed for correctness when the caller is the migration), and instead guard it only in `validateExitTierWithDelegation` and `validateRedelegatePosition` where it is semantically required.

func (s *KeeperSuite) TestMigrate1to2_VestingOwnerActiveRedelegation_HaltsUpgrade() {
    s.setupTier(1)
    vals, bondDenom := s.getStakingData()
    val := vals[0]
    valAddr := sdk.MustValAddressFromBech32(val.GetOperator())
    s.setValidatorCommission(valAddr, sdkmath.LegacyZeroDec())

    amount := sdkmath.NewInt(sdk.DefaultPowerReduction.Int64())
    vestingOwner := s.newVestingOwnerWithBalance(bondDenom, amount, amount.MulRaw(2))

    // Create a vesting-owned position (simulates pre-v8 state).
    pos := s.createLockTierPositionV1(vestingOwner, valAddr, amount)

    // Vesting owner calls MsgTierRedelegate — no vesting check blocks this.
    dstValAddr, _ := s.createSecondValidator()
    msgServer := keeper.NewMsgServerImpl(s.keeper)
    _, err := msgServer.TierRedelegate(s.ctx, &types.MsgTierRedelegate{
        Owner:        vestingOwner.String(),
        PositionId:   pos.Id,
        DstValidator: dstValAddr.String(),
    })
    s.Require().NoError(err) // succeeds — no vesting guard

    // Upgrade fires while redelegation is still active (within 21-day window).
    migrator := keeper.NewMigrator(s.keeper)
    err = migrator.Migrate1to2(s.ctx)

    // BUG: migration returns ErrActiveRedelegation, aborting the upgrade.
    s.Require().ErrorIs(err, types.ErrActiveRedelegation,
        "migration must NOT fail on active redelegation — this is the bug")

    // Position is NOT deleted; funds remain locked.
    _, getErr := s.keeper.Positions.Get(s.ctx, pos.Id)
    s.Require().NoError(getErr, "position must still exist (not exited)")
}

func (k Keeper) validateRedelegatePosition(ctx context.Context, pos types.PositionState, owner, dstValidator string) error {
	if !pos.IsOwner(owner) {
		return types.ErrNotPositionOwner
	}

	if !pos.IsDelegated() {
		return types.ErrPositionNotDelegated
	}

	if pos.Delegation.ValidatorAddress == dstValidator {
		return types.ErrRedelegationToSameValidator
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if pos.HasTriggeredExit() && pos.CompletedExitLockDuration(sdkCtx.BlockTime()) {
		return types.ErrExitLockDurationElapsed
	}

	tier, err := k.getTier(ctx, pos.TierId)
	if err != nil {
		return err
	}

	if tier.IsCloseOnly() {
		return types.ErrTierIsCloseOnly
	}

	isRedelegating, err := k.isRedelegating(ctx, pos.DelegatorAddress)
	if err != nil {
		return err
	}
	if isRedelegating {
		return errorsmod.Wrapf(types.ErrActiveRedelegation, "position %d has an active redelegation", pos.Id)
	}

	return nil
