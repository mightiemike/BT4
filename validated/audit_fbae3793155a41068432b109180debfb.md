Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of end-user, allowing allowlist bypass via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which `MetricOmmPool.swap()` sets to `msg.sender` — the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, that immediate caller is the router contract, not the end user. Any pool admin who allowlists the router (required for approved users to use the periphery) inadvertently grants every user — including non-allowlisted ones — the ability to swap in the restricted pool.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` verbatim as the first argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // immediate caller of pool.swap()
  recipient, ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that argument against the allowlist:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly:

```solidity
// MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

At this point `msg.sender` inside `pool.swap()` is the router, so the extension evaluates `allowedSwapper[pool][router]`. If the router is allowlisted (the only way for any approved user to use the periphery), the check passes for every caller of the router regardless of their individual allowlist status. The same flaw applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. `DepositAllowlistExtension` does not share this flaw because it checks the explicitly passed `owner` parameter, not `sender`.

## Impact Explanation
A pool admin who deploys a compliance-restricted pool (KYC-gated, institutional-only) and configures `SwapAllowlistExtension` must allowlist the router for any approved user to use the standard periphery. Once the router is allowlisted, the access control degenerates to "is the router allowed?" — always true for every caller. Non-allowlisted users can freely swap in the restricted pool, draining LP funds at oracle-anchored prices that LPs deposited under the assumption of a restricted counterparty set. This is a direct, repeatable loss of LP principal and constitutes an admin-boundary break by an unprivileged path.

## Likelihood Explanation
The exploit requires no privileged action, no malicious setup, and no non-standard tokens. The only precondition — the router being allowlisted — is the natural and expected configuration for any restricted pool where approved users are meant to use the standard periphery. The router is a public, permissionless contract with no access control of its own. Any unprivileged user can trigger this by calling `exactInputSingle` (or any other router swap entry point) on the deployed `MetricOmmSimpleRouter`.

## Recommendation
Gate on the economically relevant actor, not the immediate caller of `pool.swap()`. Two complementary approaches:

1. **Forward the real user through `extensionData`**: The router should encode `msg.sender` (the end user) into `extensionData` before calling `pool.swap()`, and `SwapAllowlistExtension` should decode and check that identity when `sender` is a known router.
2. **Dedicated `payer`/`originator` field in the swap interface**: Add an explicit originator field to the pool's swap interface so extensions can always gate on the true economic actor regardless of routing depth.

The simplest safe fix is option 1: the router encodes the real user in `extensionData`, and the extension decodes it when `sender` is a trusted router address.

## Proof of Concept
1. Admin deploys pool with `SwapAllowlistExtension` configured.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is approved.
3. Admin calls `setAllowedToSwap(pool, router, true)` — router allowlisted so Alice can use the periphery.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `IMetricOmmPoolActions(pool).swap(...)` — `msg.sender` inside `pool.swap()` is the router.
6. Pool calls `extension.beforeSwap(sender=router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds.
8. Bob successfully swaps in a pool he is not authorized to access.
