Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Validates Caller-Supplied `owner` Instead of Actual `sender`, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and instead validates the caller-supplied `owner` argument (the position recipient). Because `owner` is freely chosen by the caller with no on-chain constraint tying it to `msg.sender`, any address not on the allowlist can bypass the guard entirely by nominating an allowlisted address as `owner`. The token payment callback is dispatched to the real caller (`msg.sender`), not to `owner`, so the unprivileged address funds the deposit while the allowlisted address receives the LP position.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the position recipient into the extension hook: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both addresses to the extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` declares the first parameter (`sender`) as unnamed/discarded and checks only `owner`: [3](#0-2) 

Since `owner` is a free caller argument, any address can pass the guard by supplying an allowlisted address as `owner`. The token payment callback is dispatched to `msg.sender` (the real caller), not to `owner`: [4](#0-3) 

The LP position is credited to `owner`. `removeLiquidity` enforces `msg.sender == owner`, so the allowlisted address can later withdraw the tokens: [5](#0-4) 

The correct pattern is already used in `SwapAllowlistExtension.beforeSwap`, which checks `sender` (the actual swap caller), not the recipient: [6](#0-5) 

## Impact Explanation

The `DepositAllowlistExtension` is the sole mechanism a pool admin has to restrict who may provide liquidity. The guard is structurally inert against any caller who knows one allowlisted address. This constitutes an admin-boundary break: a pool admin-configured access-control boundary is bypassed by an unprivileged path. Any pool deploying `DepositAllowlistExtension` without `allowAllDepositors == true` is fully exposed. The allowlisted address can launder the deposit by calling `removeLiquidity` and withdrawing the tokens, completing a full round-trip through a restricted pool without authorization.

## Likelihood Explanation

Requires no special privilege, no flash loan, and no non-standard token behavior. The attacker only needs to know one allowlisted address, which is publicly readable from `allowedDepositor`. The trigger is a single `addLiquidity` call with `owner` set to an allowlisted address. Any pool using this extension in restricted mode is affected.

## Recommendation

Replace the unnamed first parameter with `sender` and validate it instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
// Before (broken):
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {

// After (fixed):
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
``` [3](#0-2) 

## Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension`; Alice (`0xAlice`) is allowlisted via `setAllowedToDeposit`, Bob (`0xBob`) is not.
2. Bob calls:
   ```solidity
   pool.addLiquidity(
       0xAlice,       // owner — allowlisted, passes the guard
       salt,
       deltas,
       callbackData,  // Bob's callback pays the tokens
       extensionData
   );
   ```
3. `beforeAddLiquidity` checks `allowedDepositor[pool][0xAlice]` → `true` → no revert.
4. `LiquidityLib.addLiquidity` calls `IMetricOmmModifyLiquidityCallback(Bob).metricOmmModifyLiquidityCallback(...)` — Bob's contract transfers the required tokens.
5. The LP position (shares in the specified bins) is credited to `0xAlice`.
6. Alice calls `removeLiquidity(alice, salt, deltas, ...)` and withdraws the tokens.
7. Bob has deposited into a restricted pool without being on the allowlist; the guard never triggered.
