Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Allowlist via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end-user. `SwapAllowlistExtension.beforeSwap` therefore checks whether the **router** is allowlisted rather than the actual swapper. If the pool admin adds the router to the allowlist to enable router-mediated swaps, every unprivileged user can bypass the restriction by calling the router, defeating the entire purpose of the allowlist on curated pools.

## Finding Description
`MetricOmmPool.swap` at line 230–231 passes `msg.sender` as the first argument to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // pool's msg.sender = router when called via router
    ...
);
```

`SwapAllowlistExtension.beforeSwap` (lines 31–41) gates on that value as `sender`:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap`. `MetricOmmSimpleRouter.exactInputSingle` (lines 71–80) calls `pool.swap` directly without forwarding the original end-user:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

The real user (`msg.sender` of the router call) is stored only in transient callback context for payment purposes and is never forwarded to the pool or the extension. Consequently, `sender` arriving at the extension is always `address(router)`, not the end-user.

**Two broken scenarios:**
1. **Router allowlisted (allowlist bypass):** Admin adds `address(router)` to the allowlist so router-mediated swaps work. Any unprivileged user calls `router.exactInputSingle(...)` and the extension passes because `allowedSwapper[pool][router] == true`.
2. **Router not allowlisted (allowlisted users locked out):** Admin allowlists individual user addresses. Those users cannot swap through the router because the extension sees `sender = router` and reverts.

No existing guard prevents either scenario. The `extensionData` field is passed through but the extension does not decode it to recover the real user identity.

## Impact Explanation
On a curated pool using `SwapAllowlistExtension` to restrict trading to KYC'd addresses or institutional partners, any unprivileged user can bypass the restriction entirely by routing through `MetricOmmSimpleRouter`. The pool's LP assets are exposed to unrestricted trading, defeating the purpose of the allowlist. This is a direct admin-boundary break: an unprivileged path (`router.exactInputSingle`) circumvents the pool admin's access control, with fund-impacting consequences (LP assets exposed to unintended counterparties).

## Likelihood Explanation
The router is the canonical, documented swap entry point for end-users. A pool admin who deploys `SwapAllowlistExtension` and wants router users to be able to swap will naturally add the router address to the allowlist—this is the only way to make router swaps work at all. Once the router is allowlisted, the bypass is trivially reachable by any unprivileged caller with zero preconditions, no special tokens, no malicious setup, and no privileged access required.

## Recommendation
The pool should forward the **original initiator** of the swap, not its own `msg.sender`, to the extension. The correct fix is to thread the real end-user address through the call stack. One approach: have the router encode the real user address into `extensionData` and have the extension decode it when `sender` is a known router. A more robust approach: add an `initiator` field to the swap call signature so the pool can pass it explicitly to extensions. Until fixed, pool admins must be warned that `sender` in `beforeSwap` is the direct caller of `pool.swap` (which may be a router), and per-user granularity is impossible when the router is allowlisted.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, address(router), true)` — the natural action to permit router-mediated swaps.
3. A completely non-allowlisted `attacker` calls:
   ```solidity
   router.exactInputSingle(ExactInputSingleParams({ pool: address(pool), ... }));
   ```
4. Inside `pool.swap`, `msg.sender == address(router)`.
5. `_beforeSwap(address(router), ...)` is called.
6. `SwapAllowlistExtension.beforeSwap(address(router), ...)` checks `allowedSwapper[pool][router] == true` → passes.
7. `attacker` successfully swaps on a pool restricted to allowlisted addresses only.

Foundry test: deploy pool + extension, call `setAllowedToSwap(pool, router, true)`, call `router.exactInputSingle` from an address never added to the allowlist, assert the swap succeeds. [1](#0-0) [2](#0-1) [3](#0-2)
