Audit Report

## Title
Router-mediated swaps substitute the router's address for the original user in `SwapAllowlistExtension`, enabling allowlist bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension::beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the direct caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the direct caller is the router contract, not the user. If the router is allowlisted (the only way for legitimate users to use the router), any unprivileged user can bypass the allowlist by routing through the router, since the check resolves to `allowedSwapper[pool][router] == true` regardless of the original caller.

## Finding Description

`MetricOmmPool::swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension::beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [2](#0-1) 

`MetricOmmSimpleRouter::exactInputSingle` is an unrestricted `external payable` function that calls `pool.swap()` directly, making the router the `msg.sender` inside the pool: [3](#0-2) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. None of these functions have any caller access control. When the pool admin allowlists the router (the only way for legitimate allowlisted users to use the router), the check `allowedSwapper[pool][router]` returns `true` for every caller of the router, regardless of their identity.

## Impact Explanation

This breaks the core access-control invariant of `SwapAllowlistExtension`: the extension is designed to restrict which addresses may swap on a given pool, but any unprivileged user can bypass this restriction by routing through `MetricOmmSimpleRouter`. This constitutes an admin-boundary break — the pool admin's access control mechanism is rendered ineffective by an unprivileged path. The allowlist, which pool designers rely on to restrict participation, provides no protection once the router is allowlisted.

## Likelihood Explanation

The precondition — the router being allowlisted — is a natural and expected configuration. Without it, no allowlisted user can use the router at all. Any pool that uses `SwapAllowlistExtension` and also wants to support router-based swaps is vulnerable by construction. The attack requires no special privileges: any EOA or contract can call `exactInputSingle` on the router.

## Recommendation

Pass the original user's address through the call chain rather than the direct caller. One approach: have the router encode the original `msg.sender` in `extensionData`, and have the extension decode and verify it (with the pool authenticating the router as a trusted source). Alternatively, maintain a separate allowlist for trusted routers and require routers to attest the original user on-chain. A simpler mitigation is to add a `trustedRouter` mapping in the extension so that when `sender` is a trusted router, the extension decodes the real user from `extensionData` and checks that address instead.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` active; `allowAllSwappers[pool] = false`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` so that allowlisted users can use `MetricOmmSimpleRouter`.
3. Unprivileged attacker (not in the allowlist) calls `MetricOmmSimpleRouter::exactInputSingle` targeting the pool.
4. Router calls `pool.swap(...)` — `msg.sender` inside the pool is the router address.
5. `_beforeSwap(router, ...)` → extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. Attacker successfully swaps on a pool they were never allowlisted for.
