Audit Report

## Title
Deposit Allowlist Gates on `owner` (LP Recipient) Instead of `sender` (Token Provider), Allowing Any Caller to Bypass Pool Curation — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument and checks `owner` instead. Because `owner` is a free caller-supplied parameter in `MetricOmmPool.addLiquidity`, any unlisted address can bypass the allowlist by passing an allowlisted address as `owner`, injecting tokens into a curated pool while LP shares are minted to that allowlisted address. The allowlist — the primary pool curation mechanism — is rendered completely ineffective.

## Finding Description
`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` parameter and passes `msg.sender` (the actual token provider) and `owner` (the LP recipient) to the extension hook: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` faithfully encodes both `sender` and `owner` into the call: [2](#0-1) 

However, `DepositAllowlistExtension.beforeAddLiquidity` declares the first parameter (`sender`) as unnamed and ignores it entirely, then gates only on `owner`: [3](#0-2) 

The token pull happens via callback on `msg.sender` (the actual caller), not on `owner`: [4](#0-3) 

LP shares are recorded under `owner`'s position key, not the caller's: [5](#0-4) 

**Attack path:**
1. Pool admin configures `DepositAllowlistExtension` with `allowedDepositor[pool][alice] = true` and `allowedDepositor[pool][attacker] = false`.
2. Attacker calls `pool.addLiquidity(owner=alice, salt, deltas, callbackData, "")`.
3. Extension receives `(sender=attacker, owner=alice)`, checks `allowedDepositor[pool][alice]` → `true` → passes.
4. Attacker's callback contract provides the tokens (callback fires on `msg.sender` = attacker).
5. LP shares are minted to `alice`'s position. Attacker's tokens are consumed. Pool curation is violated.

No existing guard prevents this: `removeLiquidity` enforces `msg.sender == owner` so alice can withdraw the injected liquidity, but the allowlist boundary is already broken — the attacker has successfully injected tokens into a curated pool.

## Impact Explanation
The deposit allowlist is the primary mechanism for pool curation. Bypassing it allows any unprivileged address to inject liquidity into a pool that the admin intended to restrict. This constitutes a broken core pool functionality and an admin-boundary break: the pool admin's allowlist configuration is bypassed by an unprivileged public call path. The exact corrupted value is `allowedDepositor[pool][attacker]` being effectively ignored — the extension decision is wrong, permitting a deposit that must be rejected.

## Likelihood Explanation
The bypass requires only a single public call to `pool.addLiquidity` with any known allowlisted address as `owner`. No privileged access, no special token behavior, and no off-chain data are needed. Any observer of on-chain allowlist state (via `isAllowedToDeposit` or emitted `AllowedToDepositSet` events) can execute this immediately and repeatedly.

## Recommendation
Change the allowlist check to gate on `sender` (the actual token provider) rather than `owner` (the LP recipient):

```solidity
// Before (incorrect):
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    ...
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {

// After (correct):
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    ...
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
```

## Proof of Concept

```solidity
function test_allowlistBypass_ownerVsSender() public {
    address allowlisted = makeAddr("allowlisted");
    address attacker    = makeAddr("attacker");

    // Pool admin allowlists `allowlisted`, not `attacker`
    vm.prank(admin);
    depositExtension.setAllowedToDeposit(address(pool), allowlisted, true);

    // Fund attacker and approve pool
    deal(address(token0), attacker, 1_000 ether);
    deal(address(token1), attacker, 1_000 ether);
    vm.startPrank(attacker);
    token0.approve(address(pool), type(uint256).max);
    token1.approve(address(pool), type(uint256).max);

    // Attacker calls addLiquidity with owner=allowlisted
    LiquidityDelta memory d = _deltaAbovePrice(4, 10_000);
    pool.addLiquidity(allowlisted, 1, d, callbackData, "");
    vm.stopPrank();

    // LP shares minted to allowlisted — allowlist bypassed
    uint256 shares = stateView.positionBinShares(address(pool), allowlisted, 1, int8(4));
    assertGt(shares, 0, "allowlisted received LP shares");
    assertLt(token0.balanceOf(attacker) + token1.balanceOf(attacker), 2_000 ether);
}
```
