### Title
`feeX18` Excluded from EIP-712 Digest Allows Sequencer to Charge Maximum Withdrawal Fee Without User Consent — (`core/contracts/Verifier.sol`, `core/contracts/EndpointTx.sol`)

---

### Summary

`SignedWithdrawCollateralV2.feeX18` is not included in the EIP-712 type string or digest. The sequencer can freely set `feeX18 = currentFeeX18` when submitting any valid `WithdrawCollateralV2` transaction, charging the victim the maximum protocol fee even if the user intended to pay zero or a reduced fee. The user's signature provides no protection over the fee field.

---

### Finding Description

`SignedWithdrawCollateralV2` is defined with `feeX18` as a top-level field **outside** the signed `tx` struct: [1](#0-0) 

The EIP-712 type string for this transaction type is: [2](#0-1) 

`feeX18` does not appear in the type string. Accordingly, `computeDigest` for `WithdrawCollateralV2` hashes only the six fields of the inner `tx` struct — `sender`, `productId`, `amount`, `nonce`, `sendTo`, `appendix` — and never touches `signedTx.feeX18`: [3](#0-2) 

In `processTransactionImpl`, after the signature is validated against that digest, the sequencer-supplied `feeX18` is used directly to debit the user's balance, subject only to a range check: [4](#0-3) 

Because `feeX18` is not covered by the signature, the sequencer can set it to any value in `[0, currentFeeX18]` when constructing the transaction bytes submitted via `submitTransactionsChecked`, and the `validateSignedTx` call will still pass.

---

### Impact Explanation

A user signs a `WithdrawCollateralV2` struct expecting a specific fee (e.g., zero or a negotiated discount). The sequencer submits the transaction with `feeX18 = currentFeeX18`. The user's collateral balance is debited by `amount + currentFeeX18` instead of `amount + intended_fee`. The delta — up to `currentFeeX18` — is credited to `sequencerFee[productId]`: [5](#0-4) 

This is a direct, unrecoverable asset loss for the user with no on-chain recourse.

---

### Likelihood Explanation

The sequencer is the sole caller of `submitTransactionsChecked`: [6](#0-5) 

Any sequencer operator — whether acting maliciously, under economic pressure, or due to a compromised key — can silently maximize fees on every `WithdrawCollateralV2` submission. Users have no on-chain mechanism to verify or enforce the fee they intended to pay.

---

### Recommendation

Include `feeX18` in the EIP-712 type string and digest computation:

```
"WithdrawCollateralV2(bytes32 sender,uint32 productId,uint128 amount,uint64 nonce,address sendTo,uint128 appendix,int128 feeX18)"
```

And in `computeDigest`:
```solidity
digest = keccak256(abi.encode(
    keccak256(bytes(WITHDRAW_COLLATERAL_V2_SIGNATURE)),
    signedTx.tx.sender,
    signedTx.tx.productId,
    signedTx.tx.amount,
    signedTx.tx.nonce,
    signedTx.tx.sendTo,
    signedTx.tx.appendix,
    signedTx.feeX18   // <-- add this
));
```

This ensures the user's signature cryptographically commits to the exact fee that will be deducted.

---

### Proof of Concept

1. Deploy the protocol with `currentFeeX18 = F` (e.g., `1e18`).
2. Victim signs a `WithdrawCollateralV2` struct `{sender, productId, amount, nonce, sendTo, appendix}` — no `feeX18` field is signed.
3. Sequencer constructs `SignedWithdrawCollateralV2{tx: victim_tx, signature: victim_sig, feeX18: F}`.
4. Sequencer calls `submitTransactionsChecked` with this transaction.
5. `validateSignedTx` passes — the digest does not include `feeX18`, so the victim's signature is valid.
6. `chargeFee(victim, F, productId)` executes, debiting `F` from the victim's spot balance.
7. Assert: victim's spot balance decreased by `amount + F`, not `amount + 0`.

The test requires no special privileges beyond the sequencer's normal operational role and is reproducible on an unmodified Hardhat fork.
