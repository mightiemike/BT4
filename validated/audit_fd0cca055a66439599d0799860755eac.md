### Title
Linked Signer Can Replace Itself With an Attacker-Controlled Address Without Subaccount Owner Consent — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction in `processTransactionImpl` is validated with `allowLinkedSigner = true`, meaning the **currently active linked signer** can sign a `LinkSigner` transaction to replace itself with any arbitrary address — including an attacker-controlled one — without the subaccount owner's knowledge or consent. Once the linked signer is replaced, the attacker controls the new linked signer and can sign `WithdrawCollateral` transactions (also `allowLinkedSigner = true`) to drain the subaccount's entire collateral balance.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is processed as follows: [1](#0-0) 

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true   // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
```

The fifth argument `true` is `allowLinkedSigner`. This flows into `validateSignature`: [2](#0-1) 

```solidity
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
```

And in `Verifier.validateSignature`, the recovered signer is accepted if it equals **either** the subaccount owner **or** the linked signer: [3](#0-2) 

```solidity
function validateSignature(bytes32 sender, address linkedSigner, bytes32 digest, bytes memory signature) public pure {
    address recovered = ECDSA.recover(digest, signature);
    require(
        (recovered != address(0)) &&
            ((recovered == address(uint160(bytes20(sender)))) ||
                (recovered == linkedSigner)),   // ← linked signer accepted
        ERR_INVALID_SIGNATURE
    );
}
```

This means the **current linked signer** can sign a `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with any address it chooses — including an attacker-controlled address — without the subaccount owner ever signing anything.

The `linkedSigners` mapping is stored in `EndpointStorage`: [4](#0-3) 

Once overwritten, the attacker's address becomes the new linked signer. Since `WithdrawCollateral` also uses `allowLinkedSigner = true`: [5](#0-4) 

the attacker can immediately sign a withdrawal to drain the subaccount.

---

### Impact Explanation

A compromised or malicious linked signer can:
1. Silently replace itself with an attacker-controlled address via a single `LinkSigner` transaction.
2. The attacker's new linked signer then signs a `WithdrawCollateral` or `WithdrawCollateralV2` (with `sendTo == address(0)`) transaction.
3. The subaccount's entire collateral balance is transferred to the subaccount owner's address — but the attacker now controls the signing key that authorized it, and can redirect via `WithdrawCollateralV2` with `sendTo != address(0)` only if they also own the subaccount owner key.

More precisely: after step 1, the attacker signs `WithdrawCollateral` (legacy path, `allowLinkedSigner = true` unconditionally) to drain funds to `address(bytes20(sender))`. If the attacker also controls that address (e.g., the subaccount was created by the attacker's own address), the theft is complete. Even if not, the attacker has permanently locked out the legitimate linked signer and can grief the subaccount owner.

The corrupted state is `linkedSigners[subaccount]` — a single storage slot that gates all signed transaction authority for the subaccount.

---

### Likelihood Explanation

Linked signers are the standard mechanism for session keys and automated trading bots in Nado. These keys are typically stored on hot infrastructure (servers, bots). A key compromise — through server breach, leaked environment variable, or supply-chain attack — gives the attacker the ability to execute this escalation silently. The nonce required to sign the `LinkSigner` transaction is publicly readable on-chain via `getNonce()`: [6](#0-5) 

No privileged access, governance capture, or social engineering is required. Any entity that obtains the linked signer's private key can execute this attack atomically in a single sequencer batch.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction in `processTransactionImpl`. Only the subaccount owner (the address embedded in `bytes20(sender)`) should be permitted to change the linked signer:

```solidity
// EndpointTx.sol, processTransactionImpl
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
-   true
+   false   // Only subaccount owner may change linked signer
);
```

This mirrors the pattern already used in `WithdrawCollateralV2`, where `allowLinkedSigner` is conditionally set to `false` when a custom `sendTo` is specified, precisely to prevent linked signers from redirecting funds: [7](#0-6) 

---

### Proof of Concept

1. User Alice sets linked signer to address `B` (a trading bot) via a `LinkSigner` slow-mode transaction signed by Alice.
2. Attacker obtains `B`'s private key (e.g., via server compromise).
3. Attacker reads Alice's current nonce: `Endpoint.getNonce(address(bytes20(alice_subaccount)))` → `N`.
4. Attacker uses key `B` to sign a `SignedLinkSigner` struct: `{sender: alice_subaccount, signer: attacker_address_C, nonce: N}`.
5. Attacker submits this as a sequencer transaction (or via slow mode). `processTransactionImpl` validates it — `B` is the current linked signer, `allowLinkedSigner = true`, so it passes.
6. `linkedSigners[alice_subaccount]` is now `attacker_address_C`.
7. Attacker uses key `C` to sign a `SignedWithdrawCollateral` struct: `{sender: alice_subaccount, productId: QUOTE_PRODUCT_ID, amount: full_balance, nonce: N+1}`.
8. `processTransactionImpl` validates it — `C` is now the linked signer, `allowLinkedSigner = true`, passes.
9. `Clearinghouse.withdrawCollateral` transfers Alice's full collateral balance to `address(bytes20(alice_subaccount))` — but since Alice's address is now under attacker control (or the attacker can use `WithdrawCollateralV2` with `sendTo = attacker_wallet` signed by `C` if `sendTo != address(0)` requires owner key, the attacker at minimum permanently locks Alice out of her linked signer).
