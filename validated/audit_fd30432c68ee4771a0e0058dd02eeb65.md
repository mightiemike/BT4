Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks Caller-Controlled `owner` Instead of `sender`, Completely Bypassing the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual depositing address) and gates on `owner` (a free caller-supplied field). Because `owner` is not authenticated, any unprivileged address can bypass the allowlist by nominating any allowlisted address as `owner`. The pool admin's intent to restrict liquidity provision to a curated set is completely defeated.

## Finding Description

`MetricOmmPool.addLiquidity` fires the hook with `msg.sender` as `sender` and the caller-supplied `owner` as `owner`:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`DepositAllowlistExtension.beforeAddLiquidity` ignores the first argument entirely and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol lines 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts an arbitrary `owner` address, validates only that it is non-zero via `_validateOwner`, and passes it straight through to the pool:

```solidity
// MetricOmmPoolLiquidityAdder.sol lines 56-68
_validateOwner(owner);   // only checks owner != address(0)
return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
```

The contrast with `SwapAllowlistExtension` confirms the mismatch is a defect: the swap guard correctly checks `sender`:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) { ... }
```

Exploit path:
1. Pool `P` is deployed with `DepositAllowlistExtension`; admin allowlists only `alice`.
2. Attacker `bob` calls `liquidityAdder.addLiquidityExactShares(P, alice, salt, deltas, max0, max1, "")`.
3. Pool calls `_beforeAddLiquidity(liquidityAdder, alice, ...)`.
4. Extension evaluates `allowedDepositor[P][alice]` → `true` → no revert.
5. `bob`'s tokens are pulled from `bob` and deposited into pool `P`; the LP position is minted to `alice`.
6. The allowlist is bypassed; `bob` has added liquidity to a pool that was supposed to exclude him.

## Impact Explanation

The deposit allowlist invariant — "only permitted depositors may add liquidity" — is completely broken. Any external actor can inject liquidity into a restricted pool by supplying any allowlisted address as `owner`. This constitutes: (1) a complete defeat of the pool admin's access-control intent (admin-boundary break by an unprivileged path), and (2) a griefing vector enabling forced liquidity injection into `alice`'s position without `alice`'s consent, manipulating pool liquidity composition in pools intended to be restricted to specific LPs.

## Likelihood Explanation

Exploitation requires only knowledge of one allowlisted address (publicly readable from `allowedDepositor`) and a call through the publicly available `MetricOmmPoolLiquidityAdder`. No privileged role, special token, or malicious setup is needed. Any external actor can execute this against any pool configured with `DepositAllowlistExtension`.

## Recommendation

Replace the `owner` check with a `sender` check, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Pool admins using `MetricOmmPoolLiquidityAdder` should then allowlist the adder contract address as a trusted intermediary, consistent with how the swap allowlist handles the router.

## Proof of Concept

1. Deploy pool `P` with `DepositAllowlistExtension` as a `beforeAddLiquidity` extension.
2. Pool admin calls `setAllowedToDeposit(P, alice, true)`. Only `alice` is supposed to deposit.
3. Attacker `bob` (not allowlisted) calls:
   ```solidity
   liquidityAdder.addLiquidityExactShares(P, alice, salt, deltas, max0, max1, "");
   ```
4. Pool calls `_beforeAddLiquidity(address(liquidityAdder), alice, ...)`.
5. Extension evaluates `allowedDepositor[P][alice]` → `true` → no revert.
6. `bob`'s tokens are transferred into pool `P`; the LP position is minted to `alice`.
7. Deposit allowlist bypassed; `bob` has added liquidity to a restricted pool.

Foundry test: assert that `bob` (not allowlisted) can successfully call `addLiquidityExactShares` with `owner = alice` (allowlisted) and that the transaction does not revert, confirming the bypass.
