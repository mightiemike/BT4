### Title
Corruptible Upgradability Pattern — Missing Storage Gaps in Parent Contracts of All Upgradeable Engines — (`core/contracts/BaseEngine.sol`, `core/contracts/EndpointGated.sol`, `core/contracts/ClearinghouseStorage.sol`, `core/contracts/EndpointStorage.sol`, `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

No `__gap` storage slots exist anywhere in the Nado codebase. Every upgradeable contract (`SpotEngine`, `PerpEngine`, `Clearinghouse`, `Endpoint`, `WithdrawPool`) inherits from intermediate abstract contracts that declare storage variables but provide no gap reservation. Adding any new state variable to any of these parent contracts during an upgrade will silently shift the storage layout of all child contracts, corrupting every balance, nonce, and configuration mapping they hold.

---

### Finding Description

The full inheritance chains for the proxied contracts are:

```
SpotEngine  → SpotEngineState → BaseEngine → EndpointGated → OwnableUpgradeable
PerpEngine  → PerpEngineState → BaseEngine → EndpointGated → OwnableUpgradeable
Clearinghouse → EndpointGated + ClearinghouseStorage
Endpoint    → EIP712Upgradeable + OwnableUpgradeable + EndpointStorage
WithdrawPool → BaseWithdrawPool → EIP712Upgradeable + OwnableUpgradeable
```

A `grep` for `__gap` across all `.sol` files returns **zero matches**, confirming no gap slots exist anywhere.

The parent contracts that carry storage variables but no gaps are:

| Contract | Storage Variables | Used By |
|---|---|---|
| `EndpointGated` | `address private endpoint` | `SpotEngine`, `PerpEngine`, `Clearinghouse` |
| `BaseEngine` | `_clearinghouse`, `productIds`, `canApplyDeltas`, `nonZeroBalances` | `SpotEngine`, `PerpEngine` |
| `SpotEngineState` | `configs`, `states`, `balances`, `nlpLockedBalanceQueues` | `SpotEngine` |
| `PerpEngineState` | `states`, `balances`, `migrationFlag` | `PerpEngine` |
| `ClearinghouseStorage` | `quote`, `clearinghouse`, `clearinghouseLiq`, `productToEngine`, `engineByType`, `insurance`, etc. | `Clearinghouse` |
| `EndpointStorage` | `clearinghouse`, `sequencer`, `subaccountIds`, `nonces`, `nSubmissions`, `linkedSigners`, etc. | `Endpoint`, `EndpointTx` |
| `BaseWithdrawPool` | `clearinghouse`, `verifier`, `markedIdxs`, `fees`, `minIdx` | `WithdrawPool` |

The `ProxyManager` wiki confirms `SpotEngine` and `PerpEngine` are deployed as standard Transparent Upgradeable Proxies. `Clearinghouse` and `Endpoint` use `initializer` + `__Ownable_init()` patterns, confirming proxy deployment.

**Concrete corruption scenario for `SpotEngine`/`PerpEngine`:**

If a new variable is added to `EndpointGated` (e.g., a second `address`), the storage slot previously occupied by `BaseEngine._clearinghouse` shifts by one slot. Every subsequent read of `_clearinghouse`, `productIds`, `canApplyDeltas`, `nonZeroBalances`, and all of `SpotEngineState`'s `configs`/`states`/`balances` mappings will read from the wrong slot, returning garbage values or zero.

---

### Impact Explanation

- **`SpotEngine`/`PerpEngine`**: All user spot and perp balances (`balances` mappings in `SpotEngineState`/`PerpEngineState`) would be corrupted. Every trade, deposit, withdrawal, and health check reads these mappings. Corruption here means users lose funds or gain phantom balances.
- **`Clearinghouse`**: `insurance`, `productToEngine`, `engineByType` corruption breaks liquidation logic and engine dispatch.
- **`Endpoint`**: `subaccountIds`, `nonces`, `linkedSigners` corruption enables replay attacks or locks users out of their subaccounts.
- **`WithdrawPool`**: `markedIdxs` corruption allows double-spending of fast withdrawals.

---

### Likelihood Explanation

The protocol explicitly uses a `ProxyManager` with a two-step upgrade flow (`submitImpl` → `migrateAll`), confirming that upgrades are a planned, recurring operation. Any future protocol upgrade that adds a variable to any of the listed parent contracts — a routine and expected action — triggers the corruption. The likelihood is **medium**: it requires a future upgrade, but the upgrade infrastructure is already in place and actively used.

---

### Recommendation

Add `uint256[50] private __gap;` (or an appropriate size) to every non-interface, non-library parent contract that declares storage variables:

- `EndpointGated`
- `BaseEngine`
- `SpotEngineState`
- `PerpEngineState`
- `ClearinghouseStorage`
- `EndpointStorage`
- `BaseWithdrawPool`

The gap size should be chosen so that `(existing variables) + gap = 50` slots, following the OpenZeppelin convention.
