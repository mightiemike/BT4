### Title
`StatelessTransactionValidator.validate_resource_bounds` Missing `max_l2_gas_amount` Enforcement for `Declare` Transactions — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

`StatelessTransactionValidator.validate_resource_bounds()` enforces a `max_l2_gas_amount` ceiling for `Invoke` and `DeployAccount` transactions but explicitly skips this check for `Declare` transactions via an empty match arm. Any user can submit a `Declare` transaction with `max_l2_gas_amount = u64::MAX`, bypass the gateway's resource-bound admission gate, and have the transaction admitted to the mempool — the same protection that guards the other two transaction types is absent here.

---

### Finding Description

In `crates/apollo_gateway/src/stateless_transaction_validator.rs`, `validate_resource_bounds()` contains:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

The empty `if let RpcTransaction::Declare(_) = tx {}` arm is a deliberate no-op that causes the `max_l2_gas_amount` ceiling to be skipped for every `Declare` transaction. The TODO comment confirms the developers are aware of the gap.

The asymmetry is exact:

| Transaction type | `max_l2_gas_amount` checked? |
|---|---|
| `RpcTransaction::Invoke` | ✅ Yes |
| `RpcTransaction::DeployAccount` | ✅ Yes |
| `RpcTransaction::Declare` | ❌ No |

The `validate()` entry point calls `validate_resource_bounds()` unconditionally for all three types, so the missing check is not compensated elsewhere in the stateless validator. The stateful validator (`StatefulTransactionValidator`) and the blockifier enforce actual gas *consumed*, not the user-supplied *maximum*, so they do not close this gap at admission time.

The structural analog to the external report is direct: `LiquidityPool` deposit/withdrawal functions had `whenNotPaused`; `KangarooVault` deposit/withdrawal functions did not. Here, `Invoke` and `DeployAccount` have the `max_l2_gas_amount` admission guard; `Declare` does not.

---

### Impact Explanation

**Impact: High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

A `Declare` transaction carrying `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)` passes every check in `StatelessTransactionValidator.validate()` and is forwarded to the mempool. The gateway's configured `max_l2_gas_amount` limit — the operator-controlled knob that bounds how large a single transaction's L2-gas claim can be — is entirely ineffective for `Declare` transactions.

Consequences:
1. **Admission of out-of-policy transactions.** The gateway is the first admission gate. Transactions that exceed `config.max_l2_gas_amount` are supposed to be rejected here for all types; `Declare` bypasses this.
2. **Mempool and batcher resource exhaustion.** A flood of `Declare` transactions with `max_l2_gas_amount = u64::MAX` passes the gateway and fills the mempool. The batcher must attempt to schedule them; the bouncer then rejects each one from the block because the declared gas claim exceeds the block's L2-gas budget, causing repeated wasted scheduling cycles.
3. **Fee-estimation divergence.** RPC `estimate_fee` and `simulate_transaction` paths that rely on the gateway-validated resource bounds may produce misleading results for `Declare` transactions whose declared gas ceiling is orders of magnitude above any realistic execution cost.

---

### Likelihood Explanation

**Likelihood: High.**

The trigger requires no privilege. Any external caller can craft a valid `Declare` transaction (correct Sierra class, valid signature, correct nonce) and set `resource_bounds.l2_gas.max_amount` to any value above `config.max_l2_gas_amount`. The gateway's HTTP endpoint is public. No special knowledge of internal state is required.

---

### Recommendation

Remove the empty `Declare` arm and apply the same ceiling check uniformly:

```rust
// Before (in validate_resource_bounds):
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(...MaxGasAmountTooHigh...);
}

// After:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(...MaxGasAmountTooHigh...);
}
```

If `Declare` transactions legitimately require a higher ceiling (e.g., because Sierra compilation is gas-intensive), introduce a separate `max_l2_gas_amount_declare` config field rather than removing the check entirely.

---

### Proof of Concept

1. Construct a well-formed `RpcDeclareTransaction::V3` with a valid Sierra class, valid signature, and `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)`.
2. Submit it to the gateway's `add_tx` endpoint.
3. Observe that `StatelessTransactionValidator.validate()` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never raised.
4. Submit an otherwise identical `RpcInvokeTransaction::V3` with the same `max_amount = u64::MAX`.
5. Observe that the invoke transaction is rejected with `MaxGasAmountTooHigh`.

The divergent behavior confirms that the `Declare` path lacks the guard that the `Invoke` and `DeployAccount` paths enforce. [1](#0-0) [2](#0-1)

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }
