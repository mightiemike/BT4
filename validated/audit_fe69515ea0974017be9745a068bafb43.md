### Title
Underwater Subaccount Cannot Use `burnNlp` to Incrementally Improve Health Due to Absolute Post-Execution Maintenance Health Check — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.burnNlp()` applies an absolute post-execution maintenance health check (`>= 0`) after converting NLP tokens to quote. Because burning NLP typically improves a subaccount's health (quote has higher weight than NLP), an underwater user should be able to use `burnNlp` to incrementally recover. Instead, the absolute check causes every such attempt to revert unless a single burn restores the subaccount to full maintenance health, blocking all incremental self-rescue via NLP redemption.

---

### Finding Description

`Clearinghouse.burnNlp()` redeems NLP tokens for quote at oracle price minus a burn fee. The protocol's own comment acknowledges that this operation normally **improves** health:

> "Burning NLP can decrease health if the burn fee exceeds the health improvement from the withdrawal."

After executing the balance updates, the function enforces:

```solidity
// Burning NLP can decrease health if the burn fee exceeds the health improvement
// from the withdrawal. This check prevents malicious actors from deliberately
// creating unhealthy subaccounts through NLP burns.
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [1](#0-0) 

This is an **absolute** check: the subaccount must be above maintenance health after the burn. There is no comparison to the pre-burn health value.

A user who is underwater (maintenance health < 0) and holds NLP tokens cannot use `burnNlp` to improve their position unless a single burn is large enough to bring them fully above maintenance health. If their NLP holdings are insufficient to close the entire health gap in one call — or if a portion of their NLP is still subject to the lock period enforced by `getNlpUnlockedBalance` — every `burnNlp` attempt reverts, even though each burn would have strictly improved their health. [2](#0-1) 

The NLP lock mechanism compounds the problem: newly minted NLP is subject to `NLP_LOCK_PERIOD` and only the `unlockedBalanceSum` portion is eligible for burning. [3](#0-2) 

This means a user with a large NLP position may only be able to burn a fraction of it per call, making it structurally impossible to recover in a single transaction even if their total NLP value exceeds the health deficit.

The entry path is fully user-controlled: the user signs a `BurnNlp` transaction, the sequencer routes it through `EndpointTx.processTransactionImpl`, which calls `clearinghouse.burnNlp(...)`. [4](#0-3) 

---

### Impact Explanation

An underwater subaccount holding NLP tokens is blocked from using `burnNlp` to self-rescue. The user cannot incrementally convert NLP → quote to improve their maintenance health. They are forced to either deposit fresh collateral externally or accept liquidation, even though their existing NLP holdings represent sufficient value to cover the health deficit across multiple burns. This directly mirrors the Size M-09 pattern: a health-improving operation is gated by an absolute post-execution health requirement, preventing incremental recovery.

**Corrupted invariant**: A subaccount with NLP value exceeding its health deficit should be able to avoid liquidation by redeeming NLP. The absolute check breaks this invariant.

---

### Likelihood Explanation

Any user who:
1. Holds NLP tokens (a normal protocol activity — minting NLP is a supported user action),
2. Experiences a health decline due to perp losses, adverse price moves, or accrued borrow interest,
3. Cannot cover the full health gap in a single `burnNlp` call (due to partial NLP lock or insufficient unlocked balance),

will be unable to self-rescue via `burnNlp`. This is a realistic scenario in volatile market conditions, which are precisely the conditions that cause health to deteriorate.

---

### Recommendation

Replace the absolute post-execution maintenance health check with a **relative improvement check**: capture health before the burn and require that health after the burn is strictly greater than health before. Revert only if the burn made health worse.

```solidity
int128 healthBefore = getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE);

// ... execute balance updates ...

int128 healthAfter = getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE);
require(healthAfter >= healthBefore, ERR_SUBACCT_HEALTH);
```

This preserves the original intent (preventing malicious actors from deliberately worsening health via the burn fee) while allowing underwater users to use `burnNlp` as an incremental self-rescue mechanism.

---

### Proof of Concept

**Setup:**
- Alice holds 1000 NLP (all unlocked), NLP oracle price = 1e18 (1:1 with quote)
- NLP `longWeightMaintenanceX18` = 0.8e18 (80%)
- Quote weight = 1.0 (full weight)
- Alice has open perp losses of 300 quote units
- Alice's maintenance health = `1000 * 0.8 - 300 = 500` → healthy

**Scenario (market drop):**
1. Perp losses increase to 1100 quote units
2. Alice's maintenance health = `1000 * 0.8 - 1100 = -300` → underwater
3. Alice submits `BurnNlp` for 500 NLP to recover
4. `burnNlp` executes: Alice receives `500 * 1 - burnFee ≈ 499.5` quote; NLP balance drops to 500
5. Health after burn = `500 * 0.8 + 499.5 - 1100 = 400 + 499.5 - 1100 = -200.5` → still negative
6. `require(getHealth(txn.sender, MAINTENANCE) >= 0)` → **reverts** [5](#0-4) 

7. Alice cannot burn any NLP at all, even though each burn strictly improves her health
8. Alice is liquidated despite holding 1000 NLP worth 1000 quote — more than enough to cover the 300-unit deficit across two burns
