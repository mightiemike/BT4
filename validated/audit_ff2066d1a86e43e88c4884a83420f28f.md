### Title
Missing AuxPoW Flag Guard Allows AuxPoW Bypass via `AuxData=None` Submission — (`contract/src/dogecoin.rs`)

---

### Summary

In the Dogecoin build of the light client, `submit_block_header` dispatches to `check_aux` only when `aux_data` is `Some(...)`. There is no guard that enforces: *if `version & 0x100 != 0` (AuxPoW flag set), then `aux_data` must be `Some`*. A malicious trusted relayer can therefore submit `(Header { version: version | 0x100, … }, None)` and have the block accepted after only a scrypt PoW check, with every AuxPoW invariant (chain ID, coinbase commitment, chain merkle branch, parent-block identity) silently skipped.

---

### Finding Description

`BlockHeader` for the Dogecoin feature is the tuple type `(Header, Option<AuxData>)`: [1](#0-0) 

The public entry point `submit_blocks` accepts `Vec<BlockHeader>` (i.e. `Vec<(Header, Option<AuxData>)>`) from any registered trusted relayer: [2](#0-1) 

Inside `submit_block_header`, the PoW branch is chosen purely on whether `aux_data` is `Some` or `None`: [3](#0-2) 

There is **no check** of the form `require!(version & 0x100 == 0 || aux_data.is_some(), …)` before this branch. When `aux_data` is `None`, the entire `check_aux` call is skipped and the block is accepted if its own scrypt hash satisfies the target. The checks that are bypassed inside `check_aux` include:

- Chain ID equality (`chain_id == block_header.get_chain_id()`) [4](#0-3) 
- Parent-block chain ID exclusion (`chain_id != aux_data.parent_block.get_chain_id()`) [5](#0-4) 
- Coinbase commitment / merkle root verification [6](#0-5) 
- Chain merkle branch and expected-index validation [7](#0-6) 

---

### Impact Explanation

A block accepted through the `else` branch with `version | 0x100` set is stored in `headers_pool` and can become the canonical tip via `submit_block_header_inner`. Any downstream consumer of `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` that trusts the canonical chain will operate against a header that was never validated as a legitimate Dogecoin AuxPoW block. This is a direct false canonical header acceptance. [8](#0-7) 

---

### Likelihood Explanation

The entry point is `submit_blocks`, gated by `#[trusted_relayer]`. The scope rules explicitly include "trusted-relayer submission path with attacker-relevant data" as a valid attack surface. A malicious or compromised trusted relayer can craft and submit the payload directly. No DAO role, no key leak, and no social engineering beyond holding a trusted-relayer registration is required.

---

### Recommendation

Add an explicit guard at the top of the `if !skip_pow_verification` block in `submit_block_header` (Dogecoin build):

```rust
const BLOCK_VERSION_AUXPOW: i32 = 0x100;
if block_header.version & BLOCK_VERSION_AUXPOW != 0 {
    require!(aux_data.is_some(), "AuxPoW flag set but no AuxData provided");
}
```

This mirrors the symmetric check already present inside `check_aux` (which requires the flag to be set when `aux_data` *is* provided): [9](#0-8) 

---

### Proof of Concept

1. Register as a trusted relayer on the deployed Dogecoin light-client contract.
2. Construct a `Header` with `version = <valid_doge_version> | 0x100` (AuxPoW flag set), valid `prev_block_hash`, `bits` matching the expected target, and a `nonce` such that `block_hash_pow()` (scrypt) ≤ `target_from_bits(bits)`.
3. Submit `vec![(header, None)]` via `submit_blocks`.
4. Observe that the call succeeds: `check_aux` is never invoked, chain ID is never verified, and the block is stored as a valid canonical header — despite carrying no AuxPoW proof whatsoever.
