### Title
Missing `ENABLE_SECP_OPS` Flag Check for 4-Byte Secp Opcodes Allows Premature Activation Before Softfork — (`File: src/chia_dialect.rs`)

### Summary

The `ChiaDialect::op()` dispatcher in `src/chia_dialect.rs` has two separate dispatch paths for the secp256k1/secp256r1 verify operators. The 1-byte opcode path (opcodes 64 and 65) correctly gates execution behind `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte opcode path (opcodes `0x13d61f00` and `0x1c3a8f00`) dispatches to the same operator implementations **without checking `ENABLE_SECP_OPS`**, unconditionally executing the secp verification operators regardless of whether the softfork flag has been activated. This is a flag/operator wiring error: the activation guard is present for one encoding of the operator but absent for the other, creating a consensus divergence between nodes that have and have not activated the secp softfork.

### Finding Description

In `src/chia_dialect.rs`, the `op()` method of `ChiaDialect` has two distinct dispatch branches:

**1-byte opcode path (lines 248–249) — correctly flag-gated:**
```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```
Without `ENABLE_SECP_OPS`, these fall through to `unknown_operator`.

**4-byte opcode path (lines 175–182) — no flag check:**
```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

The 4-byte path dispatches directly to `op_secp256k1_verify` / `op_secp256r1_verify` with no check for `ClvmFlags::ENABLE_SECP_OPS`. An attacker-controlled CLVM program that encodes the secp operator as a 4-byte opcode atom (`\x13\xd6\x1f\x00` or `\x1c\x3a\x8f\x00`) will have the secp verification operator executed on any node, regardless of whether the softfork has been activated via the flag.

The root cause is that the 4-byte opcode branch was added to support the "unknown operators with assigned cost" mechanism (where the opcode bytes encode the cost formula), and the secp operators were wired in here as a special case — but the activation flag guard that exists for the 1-byte aliases was not replicated.

### Impact Explanation

**Consensus divergence / flag/operator wiring error.** Nodes that have not set `ENABLE_SECP_OPS` (i.e., pre-softfork nodes) will execute `op_secp256k1_verify` / `op_secp256r1_verify` when the 4-byte opcode form is used, while the design intent is that these operators should be treated as unknown (no-ops with cost) until the softfork activates. A coin puzzle that uses the 4-byte opcode form of secp verify will:
- Be accepted and execute the secp check on pre-softfork nodes (via the 4-byte path)
- Behave identically on post-softfork nodes

This means the softfork activation boundary is bypassed: a coin can enforce secp signature verification before the network has agreed to activate it. If the secp verification fails, the coin raises an error on all nodes (pre- and post-fork), which is consistent but wrong — the pre-fork behavior should be to treat it as an unknown no-op. If the secp verification succeeds, the coin passes on all nodes, again bypassing the intended activation gate.

The concrete broken invariant: before `ENABLE_SECP_OPS` is set, `op_secp256k1_verify` and `op_secp256r1_verify` must never execute. The 4-byte opcode path violates this invariant.

### Likelihood Explanation

Any attacker who can submit a CLVM program to the mempool or embed one in a coin puzzle can trigger this. No privileged access is required — CLVM bytes are fully attacker-controlled. The 4-byte opcode encoding is valid CLVM serialization. The attacker simply encodes the secp operator as a 4-byte atom instead of the 1-byte alias. This is a realistic, low-effort exploit path.

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch branch in `src/chia_dialect.rs`, mirroring the guard on the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

Without the flag, both 4-byte opcodes should fall through to `unknown_operator`, which will either charge the formula-based cost (in lenient mode) or reject with `Unimplemented` (in `NO_UNKNOWN_OPS` mode), consistent with pre-softfork behavior.

### Proof of Concept

1. Construct a CLVM program that uses the 4-byte atom `\x13\xd6\x1f\x00` as the operator (secp256k1_verify in 4-byte form) with valid pubkey, message hash, and signature arguments.
2. Run it via `run_program` with `ChiaDialect::new(ClvmFlags::empty())` — i.e., without `ENABLE_SECP_OPS`.
3. **Expected (correct) behavior:** The operator is unknown; in lenient mode it is a no-op returning nil with formula-based cost.
4. **Actual behavior:** `op_secp256k1_verify` executes, performing real secp256k1 signature verification and returning nil on success or raising `Secp256Failed` on failure.
5. Contrast with the 1-byte opcode 64 form under the same flags: that path correctly falls through to `unknown_operator` because the `if flags.contains(ClvmFlags::ENABLE_SECP_OPS)` guard is present.

The divergence is directly observable by comparing the return value and cost of opcode `\x13\xd6\x1f\x00` vs opcode `\x40` (64 decimal) under `ClvmFlags::empty()` with a valid secp argument triple. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```rust
pub fn op_secp256r1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256R1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256r1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256r1_verify pubkey")?;
    let verifier = P1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256r1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is sha256 hash of message
    let msg = atom(a, msg, "secp256r1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256r1_verify sig")?;
    let sig = P1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
```
