### Title
`burnNlp` Applies Only `MAINTENANCE` Health Check, Enabling Collateral Extraction via Weight Arbitrage — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

The `burnNlp` function in `Clearinghouse.sol` enforces only a `MAINTENANCE` health check after burning NLP tokens, while every other collateral-modifying operation in the protocol requires `INITIAL` health. Because NLP tokens carry a lower initial risk weight than the quote tokens received in exchange, a user whose account is below `INITIAL` health can burn NLP to convert lower-weight collateral into higher-weight quote, push their `INITIAL` health above zero, and then withdraw — extracting collateral that the protocol's risk model should have blocked.

---

### Finding Description

In `Clearinghouse.burnNlp`, the post-burn health check uses `HealthType.MAINTENANCE`:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [1](#0-0) 

By contrast, `mintNlp`, `withdrawCollateral`, `transferQuote`, and `nlpProfitShare` all gate on `INITIAL` health: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

`MAINTENANCE` is the liquidation trigger — a weaker boundary than the collateral-action gate. The protocol's own developer comment acknowledges the health sensitivity of this path:

```solidity
// Burning NLP can decrease health if the burn fee exceeds the health improvement
// from the withdrawal. This check prevents malicious actors from deliberately
// creating unhealthy subaccounts through NLP burns.
``` [6](#0-5) 

The comment addresses only the downside case (health decrease). It misses the upside case: NLP tokens carry a `longWeightInitialX18 < ONE` (diversified pool, non-zero risk), while the quote token is initialized with `longWeightInitial = 1e9` — full weight. [7](#0-6) 

Converting NLP to quote via `burnNlp` therefore increases the INITIAL health contribution of the converted amount by the weight differential. The burn fee (`max(ONE, quoteAmount / 1000)`) is at most 0.1% of the quote amount — far smaller than the health gain from the weight arbitrage. A user between MAINTENANCE and INITIAL health can exploit this to bootstrap their INITIAL health above zero and then call `withdrawCollateral`.

The root cause is the desynchronization between the health threshold applied in `burnNlp` and the threshold applied everywhere else in the collateral lifecycle. This is the on-chain analog of the external report's vulnerability class: a stale/weaker risk gate is applied to a critical state-changing operation, while the rest of the system has moved to a stricter standard.

---

### Impact Explanation

A user can extract quote collateral from the protocol when the risk model says they should not be able to. The concrete state delta: the user's quote balance increases by `nlpAmount × oraclePriceX18 × (1 − fee)`. If this pushes INITIAL health ≥ 0, the user immediately calls `withdrawCollateral` — which checks INITIAL health and passes — to remove those tokens from the protocol. [8](#0-7) 

The under-collateralized position that caused the negative INITIAL health remains open. The insurance fund and other depositors absorb the residual risk.

---

### Likelihood Explanation

Medium. The condition — MAINTENANCE health ≥ 0 but INITIAL health < 0 — is the normal state for any leveraged user during moderate market volatility. NLP holders who also run leveraged perp positions (a common profile for active traders) are naturally in this zone. The exploit requires no special privileges: only a signed `BurnNlp` transaction submitted through the standard sequencer flow. [9](#0-8) 

---

### Recommendation

Change the health check in `burnNlp` from `MAINTENANCE` to `INITIAL`:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
```

This aligns `burnNlp` with every other collateral-modifying operation in the protocol and closes the weight-arbitrage extraction path.

---

### Proof of Concept

1. Alice deposits 1 000 USDC and mints 100 NLP tokens (NLP oracle price = 10 USDC; NLP `longWeightInitialX18` = 0.8e18, `longWeightMaintenanceX18` = 0.9e18).
2. Alice opens a large perp position that pushes her INITIAL health to **−15** and MAINTENANCE health to **+200**.
3. Alice submits a signed `BurnNlp` transaction for 10 NLP tokens.
4. `burnNlp` executes:
   - `quoteAmount = 10 × 10 = 100 USDC`
   - `burnFee = max(1e18, 100e18 / 1000) = 0.1 USDC`
   - Alice receives **99.9 USDC**; NLP balance decreases by 10.
5. Post-burn health check: `getHealth(Alice, MAINTENANCE) ≥ 0` → **passes** (MAINTENANCE health improved).
6. INITIAL health delta: `+99.9 × 1.0 − 10 × 10 × 0.8 = 99.9 − 80 = +19.9`.
7. Alice's INITIAL health is now `−15 + 19.9 = +4.9 ≥ 0`.
8. Alice calls `withdrawCollateral` for 99.9 USDC — the INITIAL health check passes, and the tokens leave the protocol.

Alice has extracted collateral that the protocol's risk model should have blocked, leaving her under-collateralized perp position as residual risk for the system.

        bytes32 sender,

        uint128 amount,

        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

            sendTo = address(uint160(bytes20(sender)));
        }

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
